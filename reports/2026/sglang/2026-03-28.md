### [NVIDIA] Enable automatic NUMA configuration (#19452)
**SHA**: `7160b6c` | 🔗 [查看提交](https://github.com/sgl-project/sglang/commit/7160b6cb76d3619468b219ea066fcb6358a8000e)

**🎯 变更类型**：功能增强 / 重构  
**⚡ 重要程度**：🔴高  
**📋 变更摘要**：  
- 通过新增 `sglang.srt.utils.numa_utils` 模块，实现对 NVIDIA GPU 的 NUMA 节点自动检测与绑定，并在 `scheduler` 启动时统一使用该逻辑。  
- 移除旧的 `bind_to_closest_numa_node_cuda` 与 `disable_hicache_numa_detect` 参数，简化层级缓存（HiCache）初始化路径。  
- 增加大量单元测试覆盖 NUMA 检测、权限检查、以及在 GB200 / B200 硬件上的拓扑验证，提升可靠性。

**🎯 影响范围**：  
- `python/sglang/srt/managers/scheduler.py`（进程启动时的 NUMA 绑定入口）  
- `python/sglang/srt/mem_cache/*`（HiCache 初始化中去除自动 NUMA 绑定）  
- `python/sglang/srt/server_args.py`（CLI 参数 `--disable-hicache-numa-detect` 与 `--numa-node` 文档更新）  
- `python/sglang/srt/utils/common.py`（删除旧的 NUMA 相关实现）  
- 新增 `python/sglang/srt/utils/numa_utils.py`（统一的 NUMA 检测与绑定实现）  
- 新增 `test/registered/utils/test_numa_utils.py`（完整的单元/硬件测试）  

**🔍 技术洞察**：

- **架构影响**  
  - **模块化**：将 NUMA 相关逻辑抽离为独立的 `numa_utils`，使得调度器、缓存等模块只需要通过 `get_numa_node_if_available` 获取节点，降低耦合。  
  - **配置路径统一**：`--numa-node` 仍可手工指定；若未指定且系统支持，则自动查询 GPU‑>NUMA 拓扑（依赖 `pynvml`）。  
  - **向后兼容**：原有 `disable_hicache_numa_detect` 已删除，默认行为保持原来的“在可用时自动绑定”。旧 CLI 参数不再解析，需在文档中提醒用户迁移。  

- **性能影响**  
  - **CPU‑GPU 亲和性提升**：在 NUMA 系统上将调度子进程绑定到最靠近相应 GPU 的 NUMA 节点，可显著降低跨‑NUMA 访问延迟，尤其在大模型推理、显存分配频繁的场景下提升吞吐与响应时间。  
  - **开销**：检测过程涉及一次 `pynvml` 调用与一次 `glob` 文件遍历，属于一次性启动开销，几乎不影响运行时性能。  
  - **回退机制**：若检测失败或权限受限，代码会安全返回 `None`，继续运行而不进行绑定，避免因绑定错误导致的性能回退。  

- **安全考虑**  
  - **权限校验**：`_can_set_mempolicy` 检查进程是否具备 `SYS_NICE` 能力，若不具备仅打印警告，避免因非法系统调用产生 Crash。  
  - **外部依赖**：使用 `ctypes` 调用 `libnuma`，并通过 `numactl` 检查可执行文件是否在系统中，防止因缺失库导致未捕获的异常。  
  - **日志信息**：所有异常与权限不足均记录为 `warning`，不泄露系统内部细节。  

**⚠️ 潜在风险**：

1. **环境依赖不齐**  
   - 需要 `libnuma.so`、`numactl`、`pynvml`（GPU 环境）完整可用。缺失时仅会跳过绑定，可能导致性能下降；但不会导致功能错误。  
2. **容器/沙箱限制**  
   - 在 Docker、K8s 中若未显式加入 `--cap-add SYS_NICE`，`_can_set_mempolicy` 会返回 `False`，用户可能误以为自动绑定失效。  
3. **旧参数移除**  
   - 仍有用户在脚本中使用 `--disable-hicache-numa-detect`，启动时会报未知参数错误，需在升级指南中提醒。  
4. **多节点 GPU 拓扑异常**  
   - 当 `pynvml` 返回多个 NUMA 节点（极少见硬件）时，仅使用第一个节点，可能不是最优选择。  
5. **CPU 亲和性已被外部工具限制**  
   - 若进程已被 `taskset` 或类似工具限制 CPU 集合，`_is_numa_available` 会直接返回 `False`，导致自动绑定失效。  

**💡 关注建议**：

- **发布说明**：在 Release Note 中明确说明：
  - 新增自动 NUMA 检测与绑定（通过 `SGLANG_AUTO_NUMA_BIND` 与 `SGLANG_NUMA_BIND_V2` 环境变量控制）。  
  - `--disable-hicache-numa-detect` 参数已删除，建议直接使用 `--numa-node` 手动指定或依赖自动检测。  
  - 依赖 `libnuma`、`numactl`、`pynvml`，以及容器需要 `SYS_NICE` 权限。  

- **文档更新**：在部署手册、Dockerfile 示例中加入：
  ```bash
  apt-get install -y libnuma-dev numactl
  # Docker run example
  docker run --gpus all --cap-add SYS_NICE my-sglang-image
  ```
  并说明在非 NUMA 系统或无 GPU 环境下，自动绑定会被自动禁用。  

- **监控/可观测性**：考虑在运行时通过 `metrics_collector` 暴露 `numa_binding_success`、`numa_binding_failed` 等指标，帮助运维快速定位因权限或缺失库导致的性能下降。  

- **回滚兼容**：如果有用户强制关闭自动绑定（如在某些 HPC 环境需要手动调度），提供环境变量 `SGLANG_AUTO_NUMA_BIND=0`（默认已关闭）或直接在 `ServerArgs` 中传入 `numa_node=[]` 以跳过检测。  

- **测试覆盖**：现有单元测试已经覆盖大部分分支，建议在 CI 中加入对 `libnuma` 与 `numactl` 的可用性检查，防止在缺失依赖的机器上产生误报。  

通过以上改动，SGLang 在多 NUMA 节点的 GPU 服务器上将获得更佳的 CPU‑GPU 亲和性，显著提升大模型推理的性能，同时保持在非 NUMA 或受限环境下的安全回退。持续关注依赖库的兼容性与容器权限配置，即可充分发挥该功能的价值。

### feat: add gc_threshold arg (#21481)
**SHA**: `38ad251` | 🔗 [查看提交](https://github.com/sgl-project/sglang/commit/38ad2517384cf59d01be1f42ca9ce2f5a94d1b2a)

**🎯 变更类型**：功能增强  
**⚡ 重要程度**：🔴高  
**📋 变更摘要**：  
在 `ServerArgs` 中新增 `gc_threshold` 参数，并在引擎启动时调用 `_set_gc` 根据该参数通过 `gc.set_threshold` 动态调节 Python 垃圾回收阈值。目标是让部署者能够根据业务特性手动控制 GC 频率，以降低 GC 暂停或提升内存回收效率。

**🎯 影响范围**：  
- `python/sglang/srt/entrypoints/engine.py`（进程启动入口）  
- `python/sglang/srt/server_args.py`（CLI 参数定义与校验）  
- 所有使用 **sglang** 作为服务的子进程（scheduler、workers 等）

**🔍 技术洞察**  

- **架构影响**：  
  - 仅在进程启动阶段加入一次性调用，对整体分布式架构无侵入性改变。  
  - 每个子进程都会独立执行 `_set_gc`，因此可以在多进程环境下实现细粒度的 GC 配置，保持先前的 “spawn” 启动方式不受影响。  

- **性能影响**：  
  - **正面**：适当提升阈值（提升 `threshold0/1/2`）可降低 GC 触发频率，减少因全量回收导致的短暂 CPU 峰值和响应延迟，特别是在高并发推理或大 batch 场景下。  
  - **负面**：阈值设得过高会导致对象堆积，峰值内存占用显著上升，甚至触发 OOM；阈值设得过低会导致 GC 频繁运行，CPU 占用升高，进而降低吞吐。  
  - 需要在实际负载下进行基准测试，找到业务最优的阈值组合（1‑3 个整数）。  

- **安全考虑**：  
  - 该改动不涉及网络、权限或数据加密，仅影响内部运行时行为。  
  - 唯一的安全关注点是 **资源耗尽攻击面**：恶意用户若能够控制 `--gc-threshold`（例如在托管平台提供自助参数），可能把阈值设为极低值，引起 CPU 飙升，形成拒绝服务。建议在受信任的部署环境中使用，或在 CI/CD 中对阈值做上限校验。  

**⚠️ 潜在风险**  

1. **内存泄漏/爆炸**：阈值过大导致对象长时间不被回收，进程内存曲线持续上升。  
2. **CPU 占用激增**：阈值过小会让 GC 频繁触发，尤其在推理阶段生成大量临时张量时可能导致 CPU/GPU 空闲率下降。  
3. **兼容性**：`gc.set_threshold` 在不同 Python 版本的实现略有差异（仅在 CPython 有效），如果项目在 PyPy/其它解释器上运行会抛出 `AttributeError`。当前代码未检测解释器类型，可能在非 CPython 环境直接崩溃。  
4. **配置误用**：CLI 参数仅做了长度校验，未校验数值是否为正整数或是否在合理范围内，可能导致运行时异常（如负数阈值）。  

**💡 关注建议**  

- **参数校验加强**：在 `check_server_args` 中加入 `all(v > 0 for v in self.gc_threshold)` 检查，并提供合理的上限提示（如 `<= 10_000`），防止极端值。  
- **文档与示例**：在官方文档或 `README` 中补充 `--gc-threshold` 的使用场景、推荐值范围、以及对不同硬件（CPU vs GPU）推理负载的调优建议。  
- **安全防护**：如果项目提供公开的参数入口（如 SaaS），在入口层面限制 `--gc-threshold` 的可编辑范围，或在容器化部署时通过环境变量限制。  
- **运行时监控**：建议在监控系统中加入 GC 相关指标（`gc.get_count()`、`gc.get_stats()`），以便运维人员观察阈值调节后的内存/CPU 行为。  
- **跨解释器兼容**：在 `_set_gc` 中加入 `if hasattr(gc, "set_threshold"):` 检查，若不支持则记录警告而非抛异常，提升对非 CPython 环境的兼容性。  
- **回归测试**：添加单元测试验证：① 参数为空时不影响默认行为；② 参数长度为 1/2/3 时 `gc.set_threshold` 调用正确；③ 非法参数触发 `ValueError`。  

通过上述措施，可最大化利用 GC 调参带来的性能潜力，同时将内存/CPU 风险降至可接受水平。

### [VLM] Optimize ShmPointerMMData for multi-pickle safety and deferred unwrap (#21465)
**SHA**: `57cf479` | 🔗 [查看提交](https://github.com/sgl-project/sglang/commit/57cf4790ca2f86543bbd773cee15913a38a026d1)

**🎯 变更类型**：功能增强 / Bug 修复  
**⚡ 重要程度**：🟡 中  

**📋 变更摘要**  
1. 重构 `ShmPointerMMData`：在构造时直接在共享内存中写入数据，去除 `cpu_tensor` 缓存，新增 `materialize()`、`__del__()` 以延迟复制并统一在调用方统一释放 `shm`。  
2. `unwrap_shm_features` 支持批量请求并改为调用 `materialize()`，保证只在需要时真正拷贝张量。  
3. `scheduler.recv_requests` 延后对 `ShmPointerMMData` 的解包，避免在广播阶段把完整张量序列化，提高多进程 pickle 安全性。

**🎯 影响范围**  
- `python/sglang/srt/managers/mm_utils.py`（共享内存封装逻辑）  
- `python/sglang/srt/managers/scheduler.py`（请求接收与广播流程）  
- 可能涉及所有使用 `ShmPointerMMData` 的模型推理、VLM 多模态输入路径。

**💡 关注建议**  
1. **资源释放**：`materialize()` 已负责 `unlink`，请确认所有业务路径在使用完 `ShmPointerMMData` 后一定调用它或触发 `__del__`，防止共享内存泄漏。  
2. **并发安全**：多进程/多 rank 场景下若已有进程先 `unlink`，后续 `unlink` 会抛 `FileNotFoundError`，代码已捕获，但仍建议在单元测试中覆盖 `rank>1` 的竞争情况。  
3. **向后兼容**：旧代码仍可使用 `unwrap_shm_features`，但若直接访问 `ShmPointerMMData.tensor` 将得到 zero‑copy 视图，务必在需要独立副本时调用 `materialize()`。  
4. **性能验证**：对比改前后在大量多模态请求下的序列化时间与内存占用，确保延迟复制真的带来预期提升。  
5. **文档更新**：标记 `ShmPointerMMData` 为“只能 pickle 名称，不含实际数据”，并在 API 文档中注明 `batch` 场景的处理方式。  

以上改动对共享内存的管理更安全、对多 pickling 环境更友好，但需在实际部署前完成资源清理与并发压力测试。

### Split workflow for releasing runtime docker (#21563)
**SHA**: `ec29bbb` | 🔗 [查看提交](https://github.com/sgl-project/sglang/commit/ec29bbb286b3c2e5afaab4c44ff1cbeddb708f4a)

**变更类型**：功能拆分（将 Runtime 镜像的构建/发布从 `release-docker.yml` 中抽离为独立工作流）  

**重要程度**：🟡 中  

**变更摘要**  
1. 新增 `release-docker-runtime.yml`，专门负责 **runtime**（瘦身）Docker 镜像的多架构构建、推送以及多‑arch Manifest 的合并。  
2. 在原来的 `release-docker.yml` 中删掉所有与 runtime 相关的构建、推送以及 Manifest 合并步骤，只保留 **framework**（完整开发环境）镜像的流水线。  
3. 两个工作流均在 tag（`vX.Y.Z`）或手动触发时运行，并共享相同的版本解析、Docker 登录、Artifact 上传/下载逻辑。  

**影响范围**  
- Docker 镜像发布流程（`docker/Dockerfile` 的 `runtime` 目标）。  
- CI/CD 触发规则：同一 tag 现在会触发两个独立的工作流。  
- `release-docker.yml` 中的 manifest 合并逻辑被精简，只处理 framework 镜像。  
- 新的 `release-docker-runtime.yml` 添加了磁盘清理步骤（`rm -rf /opt/hostedtoolcache`）以及 `free-disk-space` Action（仅在 x86），防止磁盘不足导致构建失败。  

**核心风险 & 建议**  

| 风险点 | 说明 | 建议 |
|--------|------|------|
| **并行构建冲突** | 同一 tag 会触发两个工作流，若其中一个失败（如 runtime），framework 仍会成功发布，可能导致版本不完整。 | 在 `release-docker.yml` 与 `release-docker-runtime.yml` 中加入 `needs`/`if` 检查，仅在对应工作流全部成功后再创建 `latest`/`latest‑runtime` tag；或在 `release-docker.yml` 的 **environment** 中使用相同 `prod` 环境，使 GitHub 自动阻塞并行部署。 |
| **Artifact 保留时间** | 两个工作流都使用 `retention-days: 1`，若 `create-manifests` 任务在 runtime 工作流完成后稍有延迟，可能找不到相应的 digest 文件。 | 将 `retention-days` 调整为 `2` 或在 `create-manifests` 前加入 `needs: [publish-x86, publish-arm64]`（已实现），但仍可适当延长保存时间以防偶发延迟。 |
| **磁盘清理不一致** | 只在 runtime x86‑64 步骤中使用了 `free-disk-space` Action，arm64 runtime 仍可能因磁盘占用而失败。 | 在 `publish-arm64`（runtime）也加入 `free-disk-space`（或统一抽取为复用的 step），确保两类节点都有足够空间。 |
| **CUDA 版本硬编码** | `cuda_version` 为 `"12.9.1"`（可能是笔误，实际应为 `12.9.0`/`12.9.1`），而 CUDA‑13 镜像使用固定 `13.0.1`。如果上游 CUDA 更新，需要同步修改两处。 | 将 CUDA 版本抽成 repository‑level secret 或变量，统一管理。 |
| **重复代码** | 版本提取、登录、构建参数在两个 workflow 中重复。 | 抽取为 reusable workflow（`workflow_call`）或共享的 action，以降低维护成本。 |
| **缺失 `free-disk-space` 在 framework workflow** | 框架镜像体积更大，构建也会占用大量磁盘。若原 workflow在资源紧张的 runner 上运行，可能出现 OOM/磁盘不足。 | 考虑在 `release-docker.yml` 中同样加入磁盘清理步骤，或使用统一的 pre‑step。 |

**对开发者的建议**  

1. **测试完整链路**：在测试 tag（如 `v0.0.0-test`）上完整跑一遍两套 workflow，确认 runtime 与 framework 镜像都能成功发布、manifest 正确指向对应 digest。  
2. **监控并行执行**：观察 GitHub Actions “queued”/“in_progress” 状态，确保两套 workflow 不相互抢占 runner 导致排队过长。  
3. **文档更新**：在项目的 Release 说明或 CI 文档中明确指出现在需要两套 workflow，且两者都必须成功才能获得完整的镜像集合。  
4. **后续维护**：若以后要再拆分如 “benchmark” 镜像，建议直接采用当前的 workflow‑splitting 模式，保持每类镜像独立、职责单一。  

总体而言，此次拆分提升了 CI 的可读性和职责分离，风险主要在并行触发和磁盘清理不对称上，只要按上面建议做少量改进即可安全上线。
