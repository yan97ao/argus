### [UnifiedRadixTree]: Support HiCache Framework for UnifiedRadixTree  (#23316)
**SHA**: `c0f5950` | 🔗 [查看提交](https://github.com/sgl-project/sglang/commit/c0f59506364f8f16527b574c06a9d4c5f58ce99a)

**🎯 变更类型**：功能增强 / 架构变更  
**⚡ 重要程度**：🔴 高  

**📋 变更摘要**  
本次提交为 UnifiedRadixTree 引入 HiCache 框架，实现了 **Device ↔ Host** 双层缓存（HiCache）以及对 UnifiedRadixCache 的全链路支持。主要改动包括：  
1. 在 `scheduler.py` 中根据环境变量 `SGLANG_ENABLE_UNIFIED_RADIX_TREE` 实例化 `UnifiedRadixCache` 并在层级缓存启用时调用 `init_hicache`。  
2. 为每个组件（Full、Mamba、SWA）加入 **host 层** 数据结构、锁管理、驱动函数，新增 `EvictLayer`（DEVICE/HOST/ALL）标识和 `CacheTransferPhase`（BACKUP_HOST/LOAD_BACK）枚举，实现 **备份（write‑through / write‑back）**、**加载回 Device** 的完整流水线。  
3. 重构 LRU 实现，新增 **host LRU**（通过指针偏移避免冲突）和 **leaf‑set**（`evictable_device_leaves`、`evictable_host_leaves`）用于快速定位可淘汰的叶子节点。  
4. 在 `UnifiedRadixCache` 中实现 **HiCache 初始化、注册共享索引池、写回、加载回、异步事件轮询** 等核心逻辑。  
5. 大量单元测试扩展，覆盖了淘汰链、叶子集合一致性、写回/加载回、混合备份的正确性。  

**🎯 影响范围**  
- `python/sglang/srt/managers/scheduler.py`（调度器）  
- `python/sglang/srt/mem_cache/*`（UnifiedRadixCache、各组件实现、LRU、HybridPoolAssembler）  
- `python/sglang/srt/mem_cache/unified_cache_components/*`（新增枚举、接口、实现细节）  
- `python/sglang/srt/mem_cache/hicache_storage.py`（PoolTransfer 扩展）  
- 单元/集成测试 `test/registered/*`（大量 HiCache 场景）

## 🔍 技术洞察

| 维度 | 影响描述 |
|------|----------|
| **架构影响** | • `UnifiedRadixCache` 由单层 Device‑Only 结构演进为 **双层**（Device + Host）结构，增加了 **CacheController** 与 **HybridPoolGroup** 的协同。<br>• 新增 **host LRU** 与 **leaf‑set**，把原来仅依赖 `lru_lists` 的遍历淘汰改为 **O(log n) / O(1)** 的集合操作，提升了大模型长期推理的可扩展性。<br>• 引入 **HiCache anchor‑KV 共享索引池**，实现 KV 与 Mamba/SWA 索引的统一备份，降低 IO 次数。 |
| **性能影响** | • **写回/写穿**：在 `write_backup` 中会先检查 Host KV 空间，如不足自动触发 `evict_host`，保证写回不阻塞推理路径。<br>• **加载回**：使用 `load_back` 通过 `CacheController.load` 把 Host KV 批量拉回 Device，支持 **阈值（load_back_threshold）** 与 **内存配额（mem_quota）** 控制，避免一次性拉回导致 OOM。<br>• **LRU 结构改造**：原来遍历链表寻找可淘汰节点的 `O(N)` 被 **堆 + leaf‑set** 替换为 `O(log M)`（M 为可淘汰叶子数量），显著提升高并发淘汰时的吞吐。<br>• **锁管理**：在 `acquire_component_lock`/`release_component_lock` 中同步更新 leaf‑set，确保锁定状态不干扰淘汰判断。<br>• **异步事件轮询**：`writing_check`、`loading_check` 在 Scheduler 主循环中被调度，利用 CUDA Event 进行跨 GPU/CPU 同步，几乎无额外 CPU 开销。 |
| **安全/可靠性** | • 所有 host‑side资源释放由 `evict_host`/`_evict_host_leaf` 统一管理，防止 **内存泄漏**。<br>• `CacheTransferPhase` 明确区分 **备份** 与 **加载**，避免同一 `PoolTransfer` 被误用导致数据错位。<br>• 对 **锁计数**（`lock_ref`、`host_lock_ref`）进行严格断言，确保在并发锁定期间不触发错误的淘汰。<br>• 通过 **TP（Tensor Parallel）同步**（`all_reduce`）保证多卡写回 ACK 的逐最小值，使得所有卡的树结构保持一致，防止 **树不一致** 的错误。 |
| **可维护性** | • 抽象了 `EvictLayer` 与 `node_has_component_data`，组件实现统一了 **DEVICE / HOST** 双层接口，后续若再加入其他层（如 SSD）只需实现同一套接口。<br>• 大量注释与 `TODO` 标记帮助后续开发者定位未完成功能（如 SWA host 层尚未实现）。<br>• 单元测试覆盖 **全部组件、写回、加载回、叶子集合**，对回归提供强保障。<br>• 代码组织上保持 **组件化**（Full、Mamba、SWA），改动不影响原有 **RadixCache**（C++ 实现）路径。 |

## ⚠️ 潜在风险

1. **SWA Host 层未实现**  
   - `swa_component.py` 中 `evict_component` 对 `HOST` 直接返回 `(0,0)`，如果未来开启 HiCache 对 SWA 的写回，会导致**数据不完整备份**。  
   - 建议在正式发布前补全 SWA 的 host memory 管理或在文档中明确“不支持 HiCache 的 SWA”。  

2. **写穿阈值过低导致频繁写回**  
   - `write_through_threshold` 默认 `1`（写穿策略），在高并发请求下可能产生大量小粒度 `write_backup`，导致 **IO 放大**。可通过配置 `hicache_write_policy=write_back` 或调高阈值来缓解。  

3. **TP（多卡）同步瓶颈**  
   - `writing_check`、`loading_check` 中的 `all_reduce` 在大规模机器上会同步所有卡的 ACK，若 ack 流量骤增可能出现 **网络拥塞**。可考虑 **局部同步**（只在同一组）或 **异步聚合**。  

4. **异常路径（异常退出、进程崩溃）下的资源泄漏**  
   - `CacheController` 的 `ack_write_queue` / `ack_load_queue` 依赖 CUDA Event 完成后才释放锁。如果进程异常退出，这些锁仍会保留在 `ongoing_write_through` / `ongoing_load_back` 中，导致后续实例化时出现 **锁不匹配**。建议在 `UnifiedRadixCache.__del__` 中强制 `dec_lock_ref` 且清空队列。  

5. **兼容性**  
   - `envs.SGLANG_ENABLE_UNIFIED_RADIX_TREE` 与原 C++ radix tree 实现共存，代码中保留了旧路径的 `elif`，但在极端配置下（如 `enable_hierarchical_cache=True` 且 `SGLANG_ENABLE_UNIFIED_RADIX_TREE=1`）可能产生 **双初始化** 问题。建议在 `init_cache_with_memory_pool` 中加入互斥检查或统一入口。

6. **回退路径**：若官方 GT 生成出现问题，确保仍可通过原有 `diffusion-ci` 方案产生 SGLang GT，避免因单一流程错误导致全部 CI 失效。  

总体来说，此次改动为 CI 引入了官方对比基准，提升了模型质量评估的可靠性，但需要在文档、权限、成本以及异常处理上做好配套工作。
