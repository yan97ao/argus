### [Feature] Integrate Elastic NIXL-EP into SGLang (#19248)
**SHA**: `9991deb` | 🔗 [查看提交](https://github.com/sgl-project/sglang/commit/9991debde37b0ea1ecca8e13f0aef24c6b498d17)

**🎯 变更类型**：功能增强  
**⚡ 重要程度**：🔴高  
**📋 变更摘要**：本次提交为 SGLang 引入全新的 Elastic NIXL‑EP 后端，实现基于 NVIDIA NIXL 框架的 RDMA 与 NVLink 加速的专家并行（MoE）通信。新增 CLI 参数、环境变量、文档以及完整的调度器实现，并在分布式初始化阶段创建全局 TCPStore 供 NIXL‑EP 使用，同时扩展了 EPLB 负载均衡算法。配套单元/集成测试验证了在 TP/DP/Elastic 环境下的正确性和准确率。

**🎯 影响范围**：  
- `sglang/srt/distributed/*`（分布式初始化、全局 TCPStore）  
- `sglang/srt/layers/moe/*`（token dispatcher、后端选择、MoE 实现）  
- `sglang/srt/environ.py`（新增环境变量）  
- `sglang/srt/server_args.py`（CLI 参数、Elastic EP 后端）  
- 文档 `docs/advanced_features/*`（EP 后端说明）  
- 测试 `test/manual/ep/test_nixl_ep.py`  

**🔍 技术洞察**  

- **架构影响**  
  - **新增后端类型**：在 `MoeA2ABackend` 枚举中加入 `NIXL`，并在所有后端判定逻辑 (`is_nixl`) 中进行适配，使 NIXL‑EP 成为与 DeepEP、Mooncake、MORI 并列的官方通信实现。  
  - **全局协调机制**：为跨节点的 NIXL 缓冲区创建了统一的 `TCPStore`（`set_global_tcp_store/get_global_tcp_store`），在 `init_distributed_environment` 中按需实例化，保证所有进程共享同一个键值服务。  
  - **调度器抽象**：实现 `NixlEPDispatcher` 以及底层 `_NixlEPDispatcherImpl`，复用了 DeepEP 的分层调度框架（`dispatch_a/b`、`combine_a/b`），并在低延迟模式下使用 NIXL 的 `Buffer.dispatch/combine` 接口。  
  - **Elastic EP 兼容**：在 `elastic_ep` 初始化路径中加入 `nixl` 选项，并在 `server_args` 中暴露 `--elastic-ep-backend nixl`，实现弹性专家备份与故障恢复的统一调度。  
  - **EPLB 算法扩展**：新增 `elasticity_aware_hierarchical`，提升在层次化集群中的负载均衡能力。  

- **性能影响**  
  - **低延迟 RDMA**：NIXL‑EP 采用原生 RDMA（或 NVLink）进行 token shuffle，避免了 CPU 侧的 All‑Reduce/All‑Gather，理论上在大规模 EP（>8）时可将通信开销降低 30%‑60%（依赖网络硬件）。  
  - **显存占用**：`NixlEPBuffer` 会为每个 rank 分配 `num_rdma_bytes`（受 `SGLANG_NIXL_EP_NUM_MAX_DISPATCH_TOKENS_PER_RANK` 限制），因此在高并发场景下需关注显存预留。  
  - **CPU 负载**：全局 `TCPStore` 的创建与广播仅在进程启动阶段进行，运行时几乎不产生额外 CPU 开销。  

- **安全考虑**  
  - **外部依赖**：引入 `nixl_ep` 包，需要在部署环境中额外安装该闭源（或 GPL）库，若库未签名或来源不可靠，可能带来供应链风险。  
  - **网络暴露**：`TCPStore` 使用 `MASTER_ADDR` 与 `SGLANG_TCP_STORE_PORT`（默认 29600）进行监听；若端口未受限，外部主机可能尝试连接并读取键值，对集群安全构成潜在威胁。建议在防火墙或安全组中限制该端口的入站访问。  
  - **错误容忍**：当 `TCPStore` 创建失败时会记录警告并继续启动，可能导致 NIXL‑EP 无法工作而回退到 `none` 后端，用户需监控日志以防隐性降级。  

**⚠️ 潜在风险**  

1. **依赖缺失**：未安装 `nixl_ep` 会导致 ImportError，阻塞整个 SGLang 启动。  
2. **端口冲突**：默认 `29600` 与集群中其他服务冲突会导致 `TCPStore` 初始化失败，进而影响 NIXL‑EP 及 Elastic EP（备份）功能。  
3. **显存超额**：`SGLANG_NIXL_EP_NUM_MAX_DISPATCH_TOKENS_PER_RANK` 超过显存容量时，会触发 `Buffer.clean_buffer` 错误或 OOM。  
4. **兼容性**：当前仅实现低延迟模式，若用户误将 `--deepep-mode normal` 与 NIXL‑EP 配合，会抛出 `NotImplementedError`，导致启动失败。  
5. **故障恢复**：Elastic EP 在使用 NIXL‑EP 时依赖 `TCPStore` 进行专家状态同步，若 `TCPStore` 中断，故障转移逻辑可能失效。  

**💡 关注建议**  

- **部署前检查**  
  - 确认已在所有节点安装 `nixl_ep`（`pip install nixl_ep` 或源码编译）。  
  - 在防火墙/安全组中仅开放 `SGLANG_TCP_STORE_PORT` 本机访问。  
  - 如有多集群共用同一机器，显式指定 `SGLANG_TCP_STORE_PORT` 避免冲突。  

- **启动脚本**  
  - 在启动脚本中加入 `--moe-a2a-backend nixl --deepep-mode low_latency`，并确保 `--tp` 与 `--ep` 相同（目前仅支持 `ep_size = tp_size`）。  
  - 若使用 Elastic EP，添加 `--elastic-ep-backend nixl` 并根据实际 RDMA 网络配置 `--mooncake-ib-device`（Mooncake）或相应 NIXL 参数。  

- **监控与日志**  
  - 关注 `init_distributed_environment` 中的 `Created global TCPStore` 日志，验证端口、IP 正确。  
  - 检查 `NixlEPBuffer` 的 `update_memory_buffers` 与 `connect_ranks` 是否成功；异常会在 `logger.warning` 中体现。  

- **性能调优**  
  - 根据实际 GPU/网络带宽，适当调节 `SGLANG_NIXL_EP_NUM_MAX_DISPATCH_TOKENS_PER_RANK`（默认 128），上限为 1024。  
  - 在大模型（> 70B）或高 TP/EP 场景下对比 `deepep`、`mooncake` 与 `nixl` 的吞吐量，选取最优后端。  

- **测试与 CI**  
  - 将 `test/manual/ep/test_nixl_ep.py` 加入 nightly/ci 流程，保证在不同硬件（NVIDIA A100/A800、NVLink、RDMA）上均能通过。  
  - 为缺少 `nixl_ep` 的环境提供 “skip” 机制，以免 CI 失败。  

- **文档更新**  
  - 在部署手册中补充 NIXL‑EP 的硬件与软件依赖（RDMA、NVLink、库安装），并给出常见错误（端口冲突、显存不足）的排查步骤。

### feat: fix update last_receive_tstamp logic for health-check in multi-token-worker mode (#20256)
**SHA**: `456934f` | 🔗 [查看提交](https://github.com/sgl-project/sglang/commit/456934fed59eb828e06c116f88c73fc09060d002)

**🎯 变更类型**：功能增强 / Bug修复  
**⚡ 重要程度**：🔴 高  
**📋 变更摘要**：  
1. 将原先的 `return_health_check_ct` 整型计数改为 `return_health_check_ipcs` 队列，记录每个健康检查请求对应的 `http_worker_ipc` 标识。  
2. 在接收健康检查请求时把对应的 `http_worker_ipc` 入队，随后在 `maybe_send_health_check_signal` 中弹出并随 `HealthCheckOutput` 一同返回。  
3. 解决了多 token‑worker 模式下健康检查信号可能被阻塞或丢失的问题，确保每个请求都能得到对应的回执。

**🎯 影响范围**：  
- `python/sglang/srt/managers/scheduler.py`（调度器）  
- 与调度器交互的 **Tokenizers**、**Detokenizer Manager**、以及 **HealthCheckOutput** 的消费者（如 HTTP worker）  
- 相关单元测试及健康检查监控脚本

**🔍 技术洞察**  

- **架构影响**  
  - 引入 `Deque[Optional[str]]` 作为状态容器，使调度器能够在多 token‑worker 场景下保持一对一的 IPC 映射，提升了健康检查信号的可追溯性。  
  - 需要 `HealthCheckOutput` 结构体支持 `http_worker_ipc` 字段，否则会出现序列化或解码错误。此改动对整体架构的侵入度低，仅限调度层与输出层的接口适配。  

- **性能影响**  
  - `deque.append` / `popleft` 均为 O(1) 操作，额外内存开销仅为每条健康检查请求保存一个字符串（或 `None`），对吞吐量几乎无影响。  
  - 通过消除简单计数器的并发争用，避免了在高并发健康检查场景下的竞争瓶颈，整体延迟预计略有下降。  

- **安全考虑**  
  - `http_worker_ipc` 可能包含进程间通信的标识（如 Unix 域套接字路径），若被恶意篡改可能导致错误的 IPC 目标。建议在 `HealthCheckOutput` 生成前对该字段做白名单或格式校验。  
  - 其他模块在解析 `HealthCheckOutput` 时必须验证字段类型，防止注入攻击。  

**⚠️ 潜在风险**  

1. **队列泄漏**：如果 `maybe_send_health_check_signal` 未被触发（例如 token‑worker 持续繁忙），`return_health_check_ipcs` 可能无限增长，导致内存占用异常。  
2. **向后兼容**：未同步更新 `HealthCheckOutput` 定义或下游解析逻辑，可能引起 `TypeError` 或字段缺失错误。  
3. **并发安全**：虽然 `deque` 本身不是线程安全的，但调度器的调用链仍在单线程事件循环中执行；若未来改为多线程，需要加锁或使用 `collections.deque` 的线程安全包装。  

**💡 关注建议**  

- **代码审查**：确认 `HealthCheckOutput` 已加入 `http_worker_ipc` 字段且序列化/反序列化路径完整。  
- **监控与告警**：在调度器中加入队列长度阈值监控（如 >1000），超过时触发告警，防止泄漏。  
- **单元/集成测试**：补充多 token‑worker 场景的健康检查测试，验证每个请求的 `ipc` 能被正确回传。  
- **安全加固**：对 `http_worker_ipc` 做格式校验（例如仅允许字母数字、下划线、短横线），并在下游使用前进行白名单过滤。  
- **文档更新**：在项目的健康检查实现说明中标注新增的 `http_worker_ipc` 传递机制，帮助运维人员理解日志中的 IPC 信息来源。

### [Utils] Add `NetworkAddress` abstraction for IPv6-safe address handling (#20306)
**SHA**: `61cad15` | 🔗 [查看提交](https://github.com/sgl-project/sglang/commit/61cad15d28874fb6cdfc544a29673e3ee64ed176)

**🎯 变更类型**：功能增强 / 重构  
**⚡ 重要程度**：🔴高  

**📋 变更摘要**  
本次提交在 `sglang/srt/utils/network.py` 中新增 `NetworkAddress` 数据类，统一管理 *host:port* 的解析、格式化以及 IPv6 包装逻辑。随后在核心通信层（conn、device_communicators、server_args、utils 等）大量替换旧的 `format_tcp_address / maybe_wrap_ipv6_address / is_valid_ipv6_address / configure_ipv6` 辅助函数，改用 `NetworkAddress` 的方法 (`to_tcp()、to_url()、to_host_port_str()、parse()` 等）。这消除了散落在代码库中的 IPv6 处理碎片，提高了可读性和可维护性，同时也让 IPv6 地址的使用更加安全、统一。

**🎯 影响范围**  
- `sglang/srt/utils`：新增 `network.py`，移除旧 IPv6 工具函数。  
- 所有 **网络/通信** 相关模块：`conn.py`（disaggregation/*, distributed/*），`device_communicators/*`，`server_args.py`，`utils.py`（端口预绑定），以及相应的单元测试。  
- 公开 API：对外使用 `sglang.srt.utils.format_tcp_address`、`maybe_wrap_ipv6_address`、`configure_ipv6`、`is_valid_ipv6_address` 的第三方代码将失效（已在本仓库内部全部迁移）。

---

### 🔍 技术洞察

| 维度 | 影响分析 |
|------|----------|
| **架构影响** | - **抽象层提升**：`NetworkAddress` 将地址“主机+端口+IPv6标识”封装为不可变对象，所有 ZMQ、torch distributed、HTTP、gRPC 等层统一使用同一套 API，降低了跨模块之间的协议耦合。<br>- **去中心化**：旧的 `maybe_wrap_ipv6_address`、`format_tcp_address` 逻辑分散在多个文件，现在全部集中在 `network.py`，减少了重复实现和潜在不一致。<br>- **向后兼容**：内部已全部迁移，但外部直接依赖旧函数的用户需要手动迁移。若项目官方提供兼容层（如保留旧函数的 thin wrapper），升级阻力会更小。 |
| **性能影响** | - **几乎零开销**：`NetworkAddress` 为 `@dataclass(frozen=True)`，仅在创建时进行一次 `ipaddress.IPv6Address` 验证，后续调用均为字符串拼接或属性读取，成本与原来手写的 `format_tcp_address` 相当。<br>- **轻微改善**：统一的 IPv6 检测避免了重复调用 `socket.setsockopt(zmq.IPV6, 1)` 时的分支判断，微小提升在大规模节点启动时会稍有累积。 |
| **安全考虑** | - **更严格的解析**：`NetworkAddress.parse()` 对 IPv6 必须使用方括号、端口必须在 0‑65535 之间、并在出现 “:`:`” 多次时抛异常，防止恶意构造的模糊地址（如 `::1:8080`）导致解析错误或意外绑定到错误接口。<br>- **避免注入**：所有对外形成的 URL/端口字符串均使用受控拼接，降低了路径遍历或 HTTP Header 注入的风险。 |

| **兼容性** | - **内部迁移已完成**，但外部文档、示例或第三方插件仍可能引用旧函数。若不提供兼容 shim，升级时会出现 `ImportError`。建议在 `sglang/srt/utils/__init__.py` 中保留 deprecated wrappers（带 `DeprecationWarning`），或在 `changelog` 明确升级指南。 |

### [Qwen-MOE] Fix memory duplication issues in case layers weights are re-assigned during weight loading (#18255)
**SHA**: `031d0a2` | 🔗 [查看提交](https://github.com/sgl-project/sglang/commit/031d0a2aad444b61c3bbe04e67dda84677b2289d)

**变更类型**：其他（Bug 修复）  
**重要程度**：🟡 中  

**变更摘要**  
本次提交修复了在模型权重加载过程中因层权重被重复分配导致的显存泄漏问题。之前通过 `self._cached_params_dict` 缓存 `named_parameters()` 的结果以避免遍历开销，但当同一个模型实例多次调用 `load_weights` 并重新分配权重时，缓存的张量对象会与新加载的张量共存，从而产生显存复制。现在改为在每次 `load_weights` 调用时重新生成 `params_dict`，彻底避免了旧张量的残留。  

**影响范围**  
- `python/sglang/srt/models/qwen3_moe.py`  
- `python/sglang/srt/models/qwen3_omni_moe.py`  
- `python/sglang/srt/models/qwen3_vl_moe.py`  

这三个 MOE（Mixture‑of‑Experts）模型的权重加载路径均受影响。  

**关注建议**  
1. **显存与性能**：重新生成 `params_dict` 会略微增加一次遍历模型参数的开销，但相对于显存泄漏的风险是可接受的。建议在大模型或高频率加载场景下监控加载时的 CPU 利用率，确认不会出现明显的性能回退。  
2. **缓存清理**：若项目中还有其它地方依赖 `self._cached_params_dict`（如自定义的增量加载逻辑），请同步移除或更新相应代码，防止出现未使用的缓存残留。  
3. **回归测试**：运行完整的模型加载与推理测试，尤其是“重加载权重后继续推理”的场景，确保显存回收正常且推理结果一致。  
4. **文档与注释**：考虑在 `load_weights` 方法的 docstring 中说明“每次调用都会重新遍历参数”，帮助后续维护者了解此 design 决策背后的显存考量。  

总体来说，此改动稳妥地解决了显存重复分配的根本原因，风险主要在于轻微的遍历开销，建议在持续集成中加入显存监控，以便快速捕获异常。
