### Move deferred mamba cow and clear (#29945)
**SHA**: `372a893` | 🔗 [查看提交](https://github.com/sgl-project/sglang/commit/372a893744dc056282b11b0e754ee0ce4eb30154)

**🎯 变更类型**：功能增强 / Bug 修复（将延迟的 Mamba COW & clear 逻辑迁入 `ModelRunner`，避免调度流与前向流竞争）  
**⚡ 重要程度**：🟡 中  

**📋 变更摘要**  
1. 将原本散落在 `HybridLinearAttnBackend` 中的 `_execute_deferred_mamba_cow_and_clear` 移除，统一由 `ModelRunner` 在真实前向执行前调用。  
2. 新增 `_maybe_execute_deferred_mamba_cow_and_clear`，在 `ModelRunner._forward_raw` 中的前向调度前执行，只有在 **extend‑forward 且目标 worker 为 Mamba 模型** 时才生效。  
3. `ModelRunner` 现在使用 `HybridReqToTokenPool`（而非普通 `ReqToTokenPool`）来完成 **物理‑虚拟槽位翻译**，并直接调用 `pool.translate_mamba_indices`。  
4. 相应地更新了导入路径，删除了 `HybridLinearAttnBackend` 中的实现，保持两个模块的职责清晰。

**🎯 影响范围**  
- `python/sglang/srt/layers/attention/hybrid_linear_attn_backend.py`（删除实现）  
- `python/sglang/srt/model_executor/model_runner.py`（新增执行入口、引入 `HybridReqToTokenPool`）  
- 相关的内存池 (`memory_pool.py`) 以及 Mamba‑相关的 checkpoint、copy‑from/clear‑slots 调用路径。

**💡 关注建议**  
- 确认所有使用 Mamba 的模型在启动时已切换到 `HybridReqToTokenPool`，否则会因缺少 `translate_mamba_indices` 报错。  
- 重点测试 `extend` 场景下的并发调度，确保 COW/clear 在前向流上完成后不再出现竞争导致的错误状态（如未清零的槽位或错误的 checkpoint 复制）。  
- 检查其它分支（如 `prefill`、`draft_extend_v2`）是否仍需要调用旧实现，防止遗漏导致潜在内存泄漏或错误的 mamba 池状态。  
- 参考提交的日志或单元测试，验证在开启 int8 checkpoint 时 `load_to_active` 能正确解量化。  

总体来说，此次改动把“延迟清理/拷贝”从注意力层抽离到调度层，使逻辑更集中、竞争更少，提升了 Mamba 模型在高并发推理中的稳定性。

### perf(triton): avoid per-step D2H .item() sync in cuda-graph loc translate (#29921)
**SHA**: `bc25abb` | 🔗 [查看提交](https://github.com/sgl-project/sglang/commit/bc25abb7864bd0ac0e9446fe5656dde875abe4b2)

**🎯 变更类型**：性能优化（perf）  
**⚡ 重要程度**：🟡 中  
**📋 变更摘要**：在 `triton_backend.py` 中的 `_translate_cuda_graph_shared_pool_locs` 里，原先每一步都通过 `self.kv_indptr[bs].item()` 或 `self.window_kv_indptr[bs].item()` 从 GPU 同步回 CPU（D2H）获取 KV 长度信息，导致大量隐式阻塞。改为优先使用已经在 CPU 上维护的 `forward_batch.seq_lens_sum` / `forward_batch.seq_lens_cpu`（若存在），只有在没有 CPU 镜像时才退回原来的 D2H `.item()`。并对滑动窗口（SWA）路径的长度计算也做了相同的镜像优化。

**🎯 影响范围**：

- 关联的 `forward_batch` 结构（新增 `seq_lens_sum`、`seq_lens_cpu` 使用场景）  

**💡 关注建议**：  
1. **兼容性**：确认 `forward_batch` 在所有调用路径里都会在 GPU‑only 场景下提供 `seq_lens_sum`（或 `seq_lens_cpu`），否则仍会触发 D2H，功能保持不变。  
2. **测试**：加入针对 GPU‑only 批次的性能基准，比较 `seq_lens_sum` 有无时的运行时差异，确保优化没有引入数值偏差。  
3. **文档**：在 API 文档或注释中说明 `seq_lens_sum` 为 “mirror present” 信号，并解释何种情况会回退到 `.item()`。  
4. **监控**：如果后续对 `forward_batch` 的实现进行改动，注意保持该 CPU 镜像字段的一致性，以免再次出现隐式同步瓶颈。  

整体来看，此改动通过避免频繁的 GPU‑to‑CPU 同步，能显著降低 CUDA‑Graph 迁移期间的延迟，属于安全且收益明确的性能提升。
