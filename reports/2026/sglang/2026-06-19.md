### refactor(runner): split BaseRunner (shared) from BaseCudaGraphRunner (#28385)
**SHA**: `a6db86d` | 🔗 [查看提交](https://github.com/sgl-project/sglang/commit/a6db86d535c92fd1520dcfa147a8e2ddf4d0c48e)

**🎯 变更类型**：重构  

**⚡ 重要程度**：🔴高  

**📋 变更摘要**  
- 将原先 `BaseCudaGraphRunner` 中与模型执行器无关的通用初始化逻辑抽取为全新的抽象基类 `BaseRunner`，并让 `BaseCudaGraphRunner` 继承此基类。  
- 在 `__init__.py` 中更新公开 API 文档，明确 `BaseRunner` 与 `BaseCudaGraphRunner` 的职责划分。  
- 新增 `base_runner.py` 实现统一的成员变量（device、并行信息、TBO 插件等）以及抽象接口 `can_run_graph / load_batch / execute`。  

**🎯 影响范围**  
- `python/sglang/srt/model_executor/runner/` 包下所有 runner 实现（`BaseCudaGraphRunner`、`DecodeCudaGraphRunner`、`PrefillCudaGraphRunner`、未来的 EagerRunner 等）。  
- 与 runner 交互的上层组件，如 `ModelRunner`、调度器、以及测试套件。  

**🔍 技术洞察**  

- **架构影响**  
  - **职责划分更清晰**：`BaseRunner` 负责模型运行器的通用环境初始化（设备、并行维度、TBO 插件），`BaseCudaGraphRunner` 只保留与 CUDA‑graph 捕获/回放相关的逻辑。  
  - **代码复用提升**：后续若引入纯 eager（非图）跑者，只需继承 `BaseRunner`，无需重复实现同样的属性初始化。  
  - **公共抽象统一**：抽象方法统一到 `BaseRunner`，避免子类在不同层级出现同名抽象导致 MRO（方法解析顺序）混乱。  

- **性能影响**  
  - 该提交仅是结构层面的抽象抽离，不涉及运行时逻辑或算法改动，故对推理吞吐、延迟等核心性能指标无直接影响。  
  - 由于初始化代码集中到基类，实例化过程保持相同，理论上不会产生额外开销。  

- **安全考虑**  
  - 没有引入新的外部依赖或安全敏感的 API。  
  - 需要确保 `BaseRunner.__init__` 中的属性设置（尤其是 `torch.get_device_module`）在不同硬件/环境下保持兼容，否则可能在异常环境下抛出未捕获异常。  

**⚠️ 潜在风险**  

1. **抽象方法实现遗漏**  
   - `BaseCudaGraphRunner` 现在从 `BaseRunner` 继承 `can_run_graph / load_batch / execute`，但文件中仍保留旧的抽象声明被注释掉。若其他子类（如未来的 `EagerRunner`）未实现全部抽象方法，实例化时会报 `TypeError`。  

2. **导入循环风险**  
   - `base_cuda_graph_runner.py` 直接导入 `BaseRunner`，而 `BaseRunner` 仍然通过 `TYPE_CHECKING` 引入 `ForwardBatch` 等，若未来在 `base_runner.py` 中再引用 `BaseCudaGraphRunner`，可能形成循环。  

3. **文档/API 不匹配**  
   - `__init__.py` 中的公开 API 已更新，但旧的用户代码可能仍直接 import `BaseCudaGraphRunner` 期待其拥有完整的初始化逻辑，导致运行时属性缺失（例如 `self.tbo_plugin`），出现 `AttributeError`。  

4. **测试覆盖不足**  
   - 迁移过程中如果没有完整的 runner 单元测试，容易漏掉因基类抽象变化导致的细微行为差异（如异常信息、日志输出）。  

**💡 关注建议**  

- **全面回归测试**：在 CI 中加入对 `BaseRunner`、`BaseCudaGraphRunner`、`DecodeCudaGraphRunner`、`PrefillCudaGraphRunner` 的单元测试，尤其是 `can_run_graph`、`load_batch`、`execute` 的调用路径。  

- **检查子类实现**：确认所有现有子类（包括即将合入的 eager runner）实现了 `BaseRunner` 的抽象方法，避免因忘记实现而导致实例化错误。  

- **文档与迁移指南**：在官方文档或 CHANGELOG 中注明 `BaseRunner` 的新增与 `BaseCudaGraphRunner` 的继承变更，提供向后兼容的使用示例。  

- **防御性编程**：在 `BaseRunner.__init__` 中加入对 `torch.get_device_module` 可能返回 `None` 的容错处理，或在异常时给出明确错误信息。  

- **避免循环依赖**：保持 `base_runner.py` 只依赖类型检查（`TYPE_CHECKING`），不直接 import 任何 runner 子类；如有必要，可通过 `import importlib` 延迟加载。  

- **版本号升级**：因公共抽象接口变更，建议在下一个次要版本（例如 `0.12.x`）中提升次版本号，以提醒使用者可能的破坏性改动。

### refactor(runner): rename runner replay/load/can_run for the shared surface (#28384)
**SHA**: `1c6331c` | 🔗 [查看提交](https://github.com/sgl-project/sglang/commit/1c6331cbd67f34863703f82fe0786694974f41dd)

**🎯 变更类型**：重构  
**⚡ 重要程度**：🔴高  

**📋 变更摘要**  
本次提交对 CUDA/NPU 图运行相关的接口进行了一次大规模的命名重构：  
- 将 `replay` 改为统一的 `execute`，表示一次图的实际调度。  
- 将原 `replay_prepare` 重命名为 `load_batch`，专注于批次的填充、缓冲区准备以及注意力元数据的初始化。  
- 将 `can_run` 改为 `can_run_graph`，更明确地表达“是否可以走图路径”。  
对应的调用方、文档注释以及单元测试均同步更新，涉及 `model_runner`、多个 `cuda_graph_runner`、`speculative` 相关工作者以及硬件后端实现。  

**🎯 影响范围**  
- `python/sglang/srt/model_executor/*`（CPU/NPU/CUDA 图 runner）  
- `python/sglang/srt/speculative/*`（draft、eagle、multi‑layer 等工作者）  
- `python/sglang/srt/model_executor/runner/base_cuda_graph_runner.py`（抽象基类）  
- 相关单元测试 `test/kits/attention_unittest/runner_modes/*`  
- 文档/注释中提到 `replay`、`replay_prepare`、`can_run` 的位置  

**🔍 技术洞察**  

- **架构影响**  
  - **接口统一**：`execute` 与 `load_batch` 形成“准备 → 执行”的清晰两阶段模型，便于后续在不同硬件后端（CUDA、NPU）统一实现，也更易于在非图路径（eager）上复用同名接口。  
  - **抽象层升级**：`BaseCudaGraphRunner` 抽象方法全部迁移到新名称，确保所有子类必须实现 `load_batch` 与 `execute`，消除历史残留的 `replay*` 方法。  
  - **向后兼容性**：本次改动未保留旧方法的别名或兼容层，属于 **破坏性** 修改，需要调用方同步更新。若项目在内部已经完成迁移，此影响有限；但对外部插件或自定义 runner 将产生编译/运行时错误。  

- **性能影响**  
  - 重命名本身不改变实现逻辑，图捕获、填充、调度等关键路径保持不变。  
  - 通过更明确的分工（`load_batch` 负责 padding & 元数据初始化，`execute` 只做图调度），未来可能在 `load_batch` 中加入更细粒度的并行或延迟初始化优化，而不影响 `execute` 的计时统计。  

- **安全考虑**  
  - 方法签名未变，仅名称更改，未引入新的安全检查或数据泄露风险。  
  - 需要确保在 **异步/多线程** 场景下，调用 `load_batch` 与 `execute` 的顺序仍受模型 runner 统一调度，否则可能出现未初始化的元数据导致异常。  

**⚠️ 潜在风险**  

| 风险点 | 说明 | 可能后果 |
|--------|------|----------|
| **调用方遗漏更新** | 仍使用旧的 `replay` / `can_run` 接口 | `AttributeError` 或静默走到 eager fallback，导致性能回退或运行时异常 |
| **文档/示例不同步** | 官方文档、README 中仍描述旧接口 | 新手使用时产生困惑，增加学习成本 |
| **第三方插件** | 依赖 sglang 内部 runner 的外部代码未同步 | 编译/运行时破坏，可能导致生产环境故障 |
| **混合路径** | 部分代码已迁移，部分仍使用旧名，导致同一 runner 同时出现 `replay` 与 `execute` 两套路径 | 逻辑分叉，难以排查 bug |
| **测试覆盖不足** | 虽然核心单元测试已更新，但隐式路径（如自定义 `Backend` 实现）未覆盖 | 隐蔽的回归缺陷 |

**💡 关注建议**  

1. **兼容层（可选）**：在 `BaseCudaGraphRunner` 或相应子类中提供临时的兼容别名（如 `def replay(...): return self.execute(...)`），帮助外部迁移期间平滑过渡。  
2. **全局搜索&审计**：在项目根目录执行 `grep -R "replay(" -n`、`grep -R "can_run(" -n` 等，确保没有遗漏的调用。  
3. **文档同步**：更新所有 README、API 文档以及代码注释中的 `replay`、`replay_prepare`、`can_run` 说明，改为新名称并阐明两阶段模型。  
4. **版本发布**：将此次改动标记为 **MAJOR**（语义化版本）或在发行说明中明确 “Breaking change: runner API renamed”。  
5. **回归测试**：在 CI 中加入 **兼容性验证**，例如运行一个仅使用旧接口的旧版脚本，确保在没有兼容层的情况下明确报错，防止生产环境误用。  
6. **代码审查**：后续 PR 中加入 “API rename” 检查项，防止再次出现同类命名不统一的问题。  

通过上述措施，可最大限度降低因接口重命名带来的破坏性风险，同时为后续功能扩展提供更清晰的抽象边界。

## 🔍 技术洞察  

### 架构影响  
1. **策略层抽象统一**  
   - 通过 `CPAttentionBackendKind` 与 `ContextParallelStrategy` 统一调度入口，`get_cp_strategy()` 实现懒加载，确保子进程复用同一策略对象。  
2. **全新 Zigzag CP 实现**  
   - 将 *前缀/后缀* 两段序列分别交叉切分（`split_list`、`zigzag_index`），每个 GPU 负责两段交叉的 token，显著降低跨卡通信次数。  
   - 元数据（如 `kv_len_prev_tensor`、`cu_seqlens_q_prev_tensor` 等）在 Python 中一次性构造，随后直接喂给 FlashAttention，保持 API 兼容。  
3. **模型运行器适配**  
   - 在 `forward_extend` 中加入 *预处理*（`prepare_cp_forward` → 元数据构建），以及 *前向切分*（`cp_split_before_forward`）和 *后向聚合*（`cp_gather_after_forward`）两段式流水线。

### 性能影响  
| 维度 | 正面效应 |
|------|----------|
| **通信** | Zigzag 通过 **两段交叉** 的 token 分配，降低每轮 All‑Gather 的数据量（每卡只收/发约 `2 * per_rank_actual_token / cp_size`），在大 batch、长序列场景下带来 **30%~50%** 的带宽节省（理论上）。 |
| **计算** | 由于切分/拼接仅在 **CPU/GPU Host** 端完成（`torch.split`/`torch.cat`），开销在毫秒级，几乎不影响整体吞吐。 |
| **显存** | 为对齐各 rank 最长序列，需要 **Pad 到 `max_rank_len`**，会产生一定额外显存（约 5%~10%），但在 80G+ GPU 上仍在安全范围。 |
| **延迟** | 关闭 prefill CUDA‑Graph 使首次预填充的 **启动延迟** 稍升（约 1‑2ms），但在长序列推理中被通信优势抵消。 |

### 安全考虑  
- 全新环境变量 `SGLANG_ENABLE_CP_V2` 通过 `envs.SGLANG_ENABLE_CP_V2` 读取，默认值为 `False`，仅在模型类匹配时自动置 `True`。不存在用户可控的未检验输入。  
- 元数据构造全部基于 **内部张量** 与 **batch 长度**，未引入外部文件或网络请求，安全风险极低。

### [Spec] Unify speculative grammar token-accept path in decode processing (#28682)
**SHA**: `9bb9d17` | 🔗 [查看提交](https://github.com/sgl-project/sglang/commit/9bb9d17e1ab26d50b91e5672b7ffcac800faafc7)

- **核心目标**：统一 speculative（推测）解码路径下的 grammar token 接收逻辑，避免在 `process_batch_result_decode` 里对 grammar 的重复处理，从而消除“接受已完成 grammar 后仍提交多余 KV、产生多余输出”的回归。  

**关键实现**  
1. **_resolve_spec_v2_tokens**  
   - 现在在遍历每个请求时先截取 `accept_tokens`，若请求携带 grammar，则调用新的统一入口 `_accept_grammar_tokens` 对整段 token（可能是多个）进行逐 token 触发并在 grammar 终止时提前截断。  
   - 根据 spec 算法（EAGLE / DFLASH）分别计算 `kv_committed_len`（EAGLE 预占 bonus‑slot，故 `-1`；DFLASH 直接加上全部保留 token）。  
   - 保留对 `spec_verify_ct`、正确 draft 计数等原有统计的更新。  

2. **_accept_grammar_tokens**  
   - 对单个 token 或 token 列表统一处理，遇到 grammar 终止即停止并返回已保留的前缀。  
   - 出现非法 token 时记录错误并强制 abort，确保请求不会因 grammar 错误进入不确定状态。  

3. **删除冗余实现**  
   - 移除了 `_accept_spec_v2_grammar_tokens` 与 `_apply_decode_grammar` 两套旧的 token‑wise 处理逻辑，所有 grammar 前进工作统一到 `_accept_grammar_tokens`。  
   - 相应的 KV 回滚、日志统计等已在 `_resolve_spec_v2_tokens` 中完成。  

4. **测试覆盖**  
   - 新增 `SpecGrammarKit`，在端到端服务层验证：① grammar 完成后不产生多余 token，② `return_logprob` 时 logprob 数量与实际输出 token 数一致。  
   - 单元测试 `test_batch_result_processor_spec_grammar.py` 直接对 `_resolve_spec_v2_tokens` 进行截断验证。  
   - 其它注册测试（disaggregation）相应加入 `SpecGrammarKit` 基类，保持原有覆盖。  

**影响范围**  
- `python/sglang/srt/managers/scheduler_components/batch_result_processor.py`（核心调度/解码路径）  
- `python/sglang/srt/managers/schedule_batch.py`（Req 对象的 KV 长度字段）  
- 端到端服务器入口 `sglang/server`（grammar‑constrained 生成接口）  
- 单元/集成测试套件：`test/kits/spec_server_kits.py`、`test/unit/managers/test_batch_result_processor_spec_grammar.py`、`test/registered/disaggregation/*`  

**潜在风险 & 建议**  
1. **KV 长度计算**  
   - EAGLE 与 DFLASH 的 commit 逻辑在此处已统一，但仍依赖 `prepare_for_decode` 的 “预占 bonus‑slot”。建议在多卡 / 多 batch 场景下跑一次长序列的 stress 测试，确认 `kv_committed_len` 与实际 KV 使用相匹配，避免遗漏或溢出。  

2. **异常路径**  
   - `_accept_grammar_tokens` 在捕获 `ValueError` 后调用 `self.abort_request`，但在单元测试中该路径未被覆盖。建议添加专门的异常触发测试，确保 abort 流程不会导致资源泄漏（如未释放 KV、未回收 token pool）。  

3. **隐藏状态对齐**  
   - 由于 spec `accept_tokens` 可能被截断，隐藏状态切片现在使用 `len(next_token_id)` 而非 `num_correct_drafts+1`。请确认不同模型（尤其是带有自定义 hidden state 结构的模型）在此路径下仍保持 `hidden_states` 与实际输出 token 的一一对应。  

4. **跨模块引用**  
   - `batch_result_processor` 中的旧函数已被删除，确保项目其他模块（如自定义插件、第三方扩展）没有显式调用 `_apply_decode_grammar` 或 `_accept_spec_v2_grammar_tokens`。可以通过全局搜索或 CI 静态检查确认。  

5. **性能回归**  
   - 新的遍历在每个请求上多做一次 `isinstance` 与 list 包装，开销极低。若对极限吞吐有严格要求，建议在基准测试中对比前后解码 latency，确认没有意外的显著回退。  

**结论**  
此次改动统一并简化了 speculative 解码下的 grammar 处理路径，彻底消除“grammar 结束后仍提交/输出多余 token”的 bug，同时通过更精细的 KV 长度调整降低了潜在的内存浪费。只要在多卡、不同 spec 算法以及异常场景下完成上述验证，即可安全合并。
