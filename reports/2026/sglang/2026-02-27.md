### [AMD] [MiniMax-M2.5 Day 0] Add MiniMax-M2.5 nightly accuracy test (#19443)
**SHA**: `403195d` | 🔗 [查看提交](https://github.com/sgl-project/sglang/commit/403195d59de0db4fe37d0b730e9cf6c2a9d9749c)

**🎯 变更类型**：功能增强 / 架构变更 / 性能验证  
**⚡ 重要程度**：🔴高  
**📋 变更摘要**：  
- 在 AMD 系列 CI 工作流中新增 MiniMax‑M2.5（TP=8, EP=8）模型的 nightly 精度测试，分别针对 ROCm 7.2（mi300x/mi325x）和 MI35x 平台。  
- 更新文档、模型支持清单以及部署示例，加入 AITer 后端和相关运行参数。  
- 添加两套完整的 GSM8K few‑shot 完成基准测试代码（`test_minimax_m25_eval_amd.py`、`test_minimax_m25_eval_mi35x.py`），并在 CI 中注册为 nightly‑amd‑accuracy 套件。

**🎯 影响范围**：  
- CI 工作流：`.github/workflows/nightly-test-amd.yml`、`.github/workflows/nightly-test-amd-rocm720.yml`  
- 文档：`docs/basic_usage/minimax_m2.md`、`docs/supported_models/text_generation/generative_models.md`  
- 测试套件：`test/registered/amd/accuracy/mi30x/`、`test/registered/amd/accuracy/mi35x/`  
- 依赖：新增 `tabulate` 安装步骤（仅在 MI35x 测试中）。

**🔍 技术洞察**  
- **架构影响**：  
  - 引入了 MiniMax‑M2.5 的专属 CI 任务，扩展了 AMD 多卡（8‑GPU）测试矩阵。  
  - 使用 `SGLANG_USE_AITER=1` 与 `--attention-backend aiter`，在服务端开启多线程加载 (`"enable_multithread_load": true`) 与 watchdog 超时配置，提升模型启动和推理的并行度。  
  - 注册机制 `register_amd_ci` 让新测试自动纳入 nightly‑amd suite，保持与现有 CI 框架一致。  

- **性能影响**：  
  - 通过 GSM8K 少样本基准（200 题、5‑shot）评估模型准确率，阈值设定为 0.93，提供可量化的性能回归检测。  
  - 并行度 `num_threads=64` 与批量推理 `run_batch` 能在 8‑GPU 环境下显著压缩评估时间（预计 < 2 h），但整体 CI 运行时长会因新增任务上升约 30‑45 min。  
  - 额外的模型加载多线程 (`enable_multithread_load`) 可能降低启动时间，但在资源受限的 CI 实例上需关注内存占用（`--mem-fraction-static 0.85`）。  

- **安全考虑**：  
  - 新增的 `--trust-remote-code` 参数在文档示例中出现，意味着在 CI 环境将执行模型仓库提供的自定义代码；若模型仓库被篡改，可能引入安全风险。CI 环境应确保使用受信任的镜像与只读工作目录。  
  - 环境变量 `SGLANG_USE_AITER` 暴露底层并行实现，暂无已知安全漏洞，但需关注后端库的更新兼容性。  

**⚠️ 潜在风险**  
1. **CI 资源消耗**：8‑GPU 任务对 CI 机器（mi325/mi35x）资源需求高，若并发运行多套测试，可能导致排队或超时。  
2. **模型可达性**：MiniMax‑M2.5 通过 HuggingFace 拉取，若网络或访问权限受限，CI 会因模型下载失败而报错。  
3. **依赖冲突**：新加的 `tabulate` 与现有依赖版本不匹配，可能导致其他测试在同一容器中出现 import 错误。  
4. **结果波动**：GSM8K 基准对随机性敏感，尽管 `temperature=0`，仍受模型加载顺序、并行调度影响，可能出现轻微的准确率波动导致 CI 误报。  
5. **文档同步**：新增 MiniMax‑M2.5 部署示例使用 `--attention-backend aiter`，若后续默认后端改为 `flashinfer`，文档可能出现不一致。  

**💡 关注建议**  
- **资源调度**：在 CI 中为新增任务分配专用标签（如 `linux-mi325-gpu-8`）并限制并发数，避免与已有 GPU 测试争抢。  
- **模型缓存**：在 CI 镜像或持久化存储中预缓存 MiniMax‑M2.5，降低网络依赖导致的失败率。  
- **依赖管理**：将 `tabulate` 的版本锁定在 `requirements-ci.txt` 中，统一管理，防止冲突。  
- **阈值监控**：将 0.93 的准确率阈值写入单独的配置文件，便于后续根据模型升级微调。可在 CI 中加入历史趋势图表（利用 `tabulate`）供审阅。  
- **安全审计**：对 `--trust-remote-code` 的使用范围进行审计，仅在受信任模型（如 MiniMax 官方）开启；对其他社区模型保持默认关闭。  
- **文档维护**：在文档页面添加 “注意：该示例依赖 AITer 后端，若使用 FlashInfer 请相应调整参数” 的提示，以免用户复制示例后出现不兼容。

### [4/N] (Elastic EP) Back up Expert Weights in DRAM (#17374)
**SHA**: `43fade5` | 🔗 [查看提交](https://github.com/sgl-project/sglang/commit/43fade5f690e42ff8e1e78dfbe29ddc8fa8006fb)

**🎯 变更类型**：功能增强  
**⚡ 重要程度**：🔴高  
**📋 变更摘要**：为 Elastic Expert Parallel（Elastic EP）引入了 “专家权重备份至 DRAM” 功能。新增 `--enable-elastic-expert-backup` 参数，启动时会在每个节点启动 *ExpertBackupManager* 子进程，通过 ZeroMQ 与对应的 *ExpertBackupClient* 进行同步、权重注册与跨节点 DRAM‑GPU 传输；权重首次从磁盘加载后存入连续的 CPU‑DRAM 缓冲区，并利用 Mooncake 传输引擎实现高速跨机读取。  

**🎯 影响范围**：  
- `sglang/srt/elastic_ep/`（新增 backup 客户端/管理器）  
- `sglang/srt/entrypoints/engine.py`（启动备份进程）  
- `sglang/srt/model_executor/model_runner.py`（在模型初始化时实例化 client、在专家位置更新时触发权重同步）  
- `sglang/srt/server_args.py`、`docs/advanced_features/server_arguments.md`（新增 CLI 参数）  
- `sglang/srt/environ.py`（新增端口基准环境变量）  
- `sglang/srt/managers/io_struct.py`（新增 ZMQ 消息结构）  
- `sglang/srt/managers/scheduler.py`（对 `elastic_ep_backend` 判空的细微修改）  
- 测试脚本 `test/manual/ep/test_mooncake_expert_backup.py`  

---

### 🔍 技术洞察  

| 维度 | 影响分析 |
|------|----------|
| **架构影响** | - 在 Elastic EP 体系中增加了 **独立的备份子进程**（`ExpertBackupManager`），负责从磁盘一次性加载本节点负责的专家权重到连续的 CPU‑DRAM 缓冲区并向其它节点广播。<br>- 客户端（`ExpertBackupClient`）在 `ModelRunner` 中创建，负责接收各节点的权重映射、注册 GPU 显存指针到 Mooncake 引擎、并在专家位置变更时拉取最新权重。<br>- 通过 **ZeroMQ PUB/SUB + PUSH/PULL** 实现节点间的“准备就绪”同步与权重指针传递，避免了传统的 MPI/torch.distributed 慢加入问题。<br>- 依赖 **Mooncake** 传输引擎的 `register_memory` 与 `batch_transfer_sync_read` 接口，实现跨节点的 DRAM→GPU 直接拷贝，保持了现有 Elastic EP 的分布式调度逻辑。 |
| **性能影响** | - **一次性磁盘加载**：权重首次只从磁盘读取一次，后续切换专家时直接从 DRAM 读取，显著降低专家切换的 I/O 延迟。<br>- **内存占用**：为每个节点额外分配一块 **连续的 CPU‑DRAM 缓冲区**（总大小等于该节点负责的专家权重字节数），在大模型 / 高专家数下可能占用数十 GB，需评估服务器的可用内存。<br>- **网络开销**：使用 Mooncake RDMA 进行 DRAM→GPU 的大块传输；一次性注册后多次读取开销极低，但首次传输仍受网络带宽限制。<br>- **启动时延**：子进程创建、ZeroMQ 连接、权重注册过程会在服务启动阶段额外耗时，尤其在节点数多时。 |
| **安全考虑** | - 新增的 ZeroMQ 端口（基于 `SGLANG_BACKUP_PORT_BASE`）默认开放在 **所有网卡** 上，若未在防火墙或容器网络层进行限制，外部主机可能尝试连接并注入恶意 `UpdateExpertBackupReq`/`BackupDramReq`。<br>- 传输的内容仅为指针信息和权重元数据（不包含实际模型参数），但如果攻击者获得了这些指针，可能导致 **内存泄露** 或 **非法读取**（取决于 Mooncake 引擎的权限检查）。<br>- 代码未对 ZMQ 消息进行身份校验或加密，建议在生产环境下将端口绑定到 **内网** 或使用 **TLS over ZMQ**。 |
| **可维护性** | - 新增的两大模块代码量较大（≈350 行），但结构清晰：`ExpertBackupManager` 负责数据准备与发布，`ExpertBackupClient` 负责消费和权重写回。<br>- 通过 `ServerArgs.enable_elastic_expert_backup` 开关控制，默认关闭，避免对现有路径产生副作用。<br>- 对 `elastic_ep_backend` 判空的微调提升了兼容性，但需要在文档和 CI 中同步更新对应的类型注解。 |

---

### ⚠️ 潜在风险  

1. **内存超限**：DRAM 缓冲区按专家全量大小分配，若节点显存/CPU‑DRAM 比例不匹配可能导致 OOM。  
2. **启动死锁**：`ExpertBackupManager` 等待 `tp_size`（即 total GPU‑world‑size）个 client 报告准备就绪；若任意节点的 client 因异常未能发送 `UpdateExpertBackupReq`，整个集群将卡在 `signal.pause()`。  
3. **端口冲突**：`SGLANG_BACKUP_PORT_BASE` 默认 10000，如与其他服务冲突会导致 ZMQ bind 失败，启动报错难以定位。  
4. **跨版本不兼容**：该特性强依赖 Mooncake 传输引擎的注册/读取接口，若 Mooncake 升级或在不支持的环境下使用（如仅有 NCCL），会触发运行时异常。  
5. **异常恢复**：备份子进程崩溃后没有自动重启机制，权重将回退到磁盘加载路径，可能出现 **权重不一致** 的隐蔽错误。  
6. **安全泄露**：ZeroMQ 未加密/鉴权，内部网络若被渗透，攻击者可伪造请求读取或覆盖权重指针。

### 💡 关注建议  

| 对象 | 建议 |
|------|------|

- **scheduler_output_processor_mixin.py**：mamba ping‑pong 索引改为通过 `ReqToTokenPool.get_mamba_ping_pong_other_idx` 取得，实现双缓冲切换。  
- **eagle_*（v2）**：在验证后调用 `update_mamba_state_after_mtp_verify`，利用 `batch.mamba_track_indices` 维护跨 interval 的 mamba 状态。  
- **测试**：新增 4‑GPU Qwen‑3‑Next MTP V2 测试，用 `envs.SGLANG_ENABLE_SPEC_V2` 控制开关。  

**💡 关注建议**  
1. **兼容性**：`--mamba-scheduler-strategy extra_buffer` 成为 SpecV2 的必选，建议在文档/CLI 帮助中提前提示，否则启动时报错。  
2. **性能**：`enable_overlap_schedule` 默认 true，若用户使用单卡或显存紧张场景，可能导致额外 buffer 布局开销，提供 `--disable-overlap-schedule` 选项并在 README 中说明何时关闭。  
3. **测试覆盖**：当前仅在 4‑GPU 环境下验证，建议补充单卡/多卡混合（tp ≠ 4）以及仅 mamba‑2（无 hybrid GDN）模型的回归测试，防止出现未覆盖的 `ping_pong` 索引越界。  
4. **异常处理**：`SpecV2` 仍假设 `topk==1`，若后续支持更高 top‑k，相关断言需同步更新。  
5. **代码结构**：`envs.SGLANG_ENABLE_SPEC_V2` 与 `self.server_args.enable_mamba_extra_buffer()` 的相互依赖稍显耦合，建议抽象为统一的 “spec_v2_enabled” 配置对象，以降低后续扩展成本。  

总体来说，此次改动为 SpecV2 在混合注意力模型上的关键落地，涉及调度、缓存、前向元数据等多层实现，风险主要在新增的双缓冲索引和验证后 mamba 状态更新上。通过加强文档提示、扩展单卡测试以及审视异常路径，可进一步提升兼容性与可维护性。
