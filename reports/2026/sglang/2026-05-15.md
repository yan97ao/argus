#### 🔴 高重要度变更 (3)

### [UnifiedTree]: Support HiCache For DeepSeek_V4 (#24691)
**SHA**: `d9fa84b` | 🔗 [查看提交](https://github.com/sgl-project/sglang/commit/d9fa84b25b79203b08be54ef6f5e4ccebef8cd5f)

**🎯 变更类型**：功能增强 / 架构变更  
**⚡ 重要程度**：🔴 高  

**📋 变更摘要**  
1. 为 DeepSeek V4（压缩 KV）模型引入 HiCache 支持，实现了统一的层级缓存（UnifiedTree）与压缩 KV 的协同工作。  
2. 在 HiCache 框架中新增 **SidecarPool** 概念：能够复用 KV（或 SWA）层的索引，在 HiCache 备份/加载时自动携带关联的压缩状态/索引池。  
3. 实现了 **LogicalHostPool** 与 **DeepSeekV4PagedHostPool / DeepSeekV4StateHostPool**，在 Host 端为 V4 压缩子池提供独立的内存管理与 I/O。  
4. 调整了 HybridCacheController、HybridPoolAssembler、HybridCacheController 中的 pool transfer 合并、分配以及 side‑car 解析逻辑，以适配新的 sidecar pool。  
5. 在 UnifiedRadixCache 中加入 sidecar‑pool 注册与读写路径的统一处理。  
6. 测试层面扩展了 KL‑divergence 多轮对齐测试，加入了 temperature 参数的显式传递，确保在 HiCache + DeepSeek V4 场景下概率一致性。  

**🎯 影响范围**  
- `sglang/srt/mem_cache/`（deepseek_v4_memory_pool, hicache_storage, hybrid_cache 相关模块）  
- `sglang/srt/mem_cache/unified_radix_cache.py`（sidecar pool 注册、写/读备份路径）  
- `sglang/test/`（KL 多轮对齐工具、统一 radix cache 测试、全新 HiCache‑V4 测试套件）  
- 相关的 `ServerArgs`、`CacheInitParams`、`HybridCacheController`、`HostPoolGroup` 数据结构  

**🔍 技术洞察**  

| 维度 | 影响分析 |
|------|----------|
| **架构影响** | • 引入 **SidecarPoolSpec**，把“共享 KV 索引”从原来的 `share_indices_with_anchor` 标记改成显式的 sidecar 描述，提升可扩展性（后续可支持非 KV 的 sidecar）。<br>• 新增 **LogicalHostPool** 与 **DeepSeekV4PagedHostPool / DeepSeekV4StateHostPool**，形成三层 Host ↔ Device 映射：① Logical Full Pool (仅管理 slot) ② 压缩 KV 子池 ③ 压缩 State 子池。<br>• `HybridCacheController.merge_pool_transfers` 及 `_resolve_sidecar_derived_pool_transfers` 逻辑改写，使得 sidecar 池的索引复用在统一的 `PoolTransfer` 流程中完成，避免了旧的 “share_indices_with_anchor” 隐式耦合。<br>• `UnifiedRadixCache._build_sidecar_transfers` 对 sidecar pool 自动生成 Transfer，统一了备份/加载阶段的侧链处理。 |
| **性能影响** | • 通过 **HiCache‑ratio**（默认 4）在 Host 端为压缩 KV 预分配更大页数，能够显著降低磁盘 I/O 与网络传输次数，对 DeepSeek V4 大模型的多轮推理延迟有正向提升。<br>• 新增的 `wait_layer_transfer` 在每次访层时阻塞至对应层的迁移完成，防止层间竞争，但在高并发场景下可能成为轻微的同步瓶颈。<br>• `DeepSeekV4PagedHostPool` 进行 **host‑memory 预检查**（`psutil.virtual_memory`），避免 OOM，保证缓存分配的可预测性。<br>• 侧链 pool 采用 **direct I/O**（仅支持 `io_backend="direct"`），省去中间拷贝，提升备份/加载吞吐。 |
| **安全考虑** | • 代码中未引入网络或文件系统的新路径，仅在内存层面增加分配与拷贝，未发现安全风险。<br>• 输入校验（如页面对齐、层号合法性）已通过 `ValueError` 抛出，降低了异常导致的内存泄露或越界写风险。 |
| **可维护性** | • 将原有的 “share_indices_with_anchor” 标记废弃，改为**显式 sidecar 描述** (`SidecarPoolSpec`)，代码可读性与可配置性提升。<br>• 新增的 Host Pool 类分离了逻辑与物理职责，便于后续独立优化（如换成 ZSTD 压缩或 GPU‑direct‑storage）。<br>• 大量 `if TYPE_CHECKING`、`__future__` 导入提升了类型检查完整度。<br>• 测试覆盖面扩大到 HiCache‑V4 场景，降低回归风险。 |

**⚠️ 潜在风险**  

1. **内存占用激增**：DeepSeek V4 Paged Host Pool 会预分配 `num_host_pages * page_size` 的 CPU 内存，若 `hicache_ratio` 设置过大或多模型并发，可能触发 OOM。  
2. **同步阻塞**：`wait_layer_transfer` 依赖 `layer_transfer_counter.wait_until`，若层迁移出现异常或迟滞，整个请求链路会被阻塞。  
3. **Sidecar Pool 兼容性**：当前实现仅支持 `indices_from_pool == PoolName.KV` 与 `indices_from_pool == PoolName.SWA`，未来若需要其它 sidecar（如 MAMBA）会抛 `AssertionError`。  
4. **后端限制**：`DeepSeekV4PagedHostPool` 只接受 `io_backend="direct"`，若用户在配置文件中误设其他后端，将导致 `NotImplementedError`。  
5. **测试不确定性**：新增的 `test_unified_radix_hicache_kl.py` 中部分测试被标记为 `skip("no stable.")`，意味着当前 CI 仍可能缺乏完整回归覆盖。  

**💡 关注建议**  

- **监控与告警**：在生产部署时开启 Host 端内存监控（如利用 `psutil` 或 Prometheus），并在 `hicache_ratio` 超过安全阈值时预警。  
- **配置最佳实践**：建议在单卡/单机环境下 `hicache_ratio ≤ 4`，并结合实际显存占用调优。  
- **并发测试**：针对多请求并发（TP=4、PP>1）跑一次大规模压力测试，重点观察 `wait_layer_transfer` 的排队时长。  
- **逐步迁移**：如果已有业务仍依赖旧的 `share_indices_with_anchor` 标记，先在单机环境验证 sidecar pool 行为后，再逐步切换。  
- **代码可视化**：考虑为 `SidecarPoolSpec` 添加 `__repr__`，便于调试缓存层级时快速定位 sidecar 来源。  
- **文档补全**：在项目 README 或部署手册中注明 DeepSeek V4 HiCache 的内存需求、仅支持 `direct` I/O、以及 sidecar pool 的使用方式。  

---  
通过此次改动，sglang 在 **DeepSeek V4 Flash** 场景下实现了高效的二层缓存（GPU → Host → SSD），为大模型多轮对话提供了更低的时延和更高的可靠性。但也引入了对 Host 内存的更大依赖以及同步阻塞点，建议在生产环境中结合监控与压测进行细致调优。

### fix: strip "[asctime]" prefix when parsing JSON log lines in nightly tests (#25340)
**SHA**: `974948a` | 🔗 [查看提交](https://github.com/sgl-project/sglang/commit/974948a7012791b476b1a3ca08204c8c4a573d54)

**🎯 变更类型**：Bug修复  
**⚡ 重要程度**：🔴高  
**📋 变更摘要**：在 nightly 测试中，日志行前会带有 `"[asctime]"` 前缀，导致原有的 `line.startswith("{")` 判断失效，JSON 解析抛异常。此次提交改为在每行中查找首个 `{`，截取其后内容再执行 `json.loads`，并容错处理非 JSON 行，确保日志解析正确。  

**🎯 影响范围**：  
- `test/registered/bench_fn/test_bench_serving_functionality.py`（多轮对话日志解析）  
- `test/registered/utils/test_request_logger.py`（请求日志与 OpenAI 日志解析）  
- `test/registered/utils/test_scheduler_status_logger.py`（调度器状态日志解析）  

**🔍 技术洞察**：  
- **架构影响**：仅限于测试代码层面，无生产代码或服务架构变动，不影响系统整体结构。  
- **性能影响**：新增 `str.find` 与异常捕获，开销极小（线性遍历每行字符），对测试执行时间影响可以忽略不计。  
- **安全考虑**：解析过程仍保持严格的 JSON 校验，未引入外部输入或执行路径，安全风险保持不变。  

**⚠️ 潜在风险**：  
1. **误判非 JSON 行**：如果日志中出现 `{` 前的非 JSON 内容（如异常信息）但实际不是完整 JSON，仍会尝试解析并可能产生 `JSONDecodeError`，当前已捕获并忽略，风险可接受。  
2. **日志格式变动**：若未来日志前缀不再使用 `{` 开头而改为其他符号，仍需相应调整；此实现依赖 “首个 `{` 为有效 JSON 起点”。  

**💡 关注建议**：  
- **测试可靠性**：建议在日志生成端统一使用 JSON 行前缀（如统一加上时间戳）或提供专用的结构化日志写入方式，避免在解析端做过多容错。  
- **代码复用**：可将 “从行中提取 JSON” 的逻辑抽象为测试工具函数，供所有日志相关测试统一调用，降低重复实现和未来维护成本。  
- **监控变化**：CI 中继续监控该解析逻辑的覆盖率和异常统计，确保新日志格式不会导致遗漏关键信息。

### perf(mla): TMA bulk-store set_mla_kv_buffer (up to 12× over baseline) (#25311)
**SHA**: `dca9ba6` | 🔗 [查看提交](https://github.com/sgl-project/sglang/commit/dca9ba63215d09a59762cff6ee1cdfaf7b367354)

**🎯 变更类型**：性能优化 / 功能增强  
**⚡ 重要程度**：🔴高  
**📋 变更摘要**：  
- 新增 JIT 编译的 CUDA TMA **bulk‑store** kernel 用于 `set_mla_kv_buffer`，在 SM90+（Hopper 及以上）GPU 上实现一次 warp‑级共享内存载入后通过 `cp.async.bulk` 完成散列写入。  
- 在 `sglang.srt.mem_cache.utils.set_mla_kv_buffer_triton` 中加入基于 **batch size** 与 **硬件特性** 的自动调度：大批量（≥768）走 JIT TMA 路径，小批量保留改进后的 Triton 单 CTA 实现。  
- 同时提供基准脚本、单元测试以及 JIT 编译缓存/参数检查工具，整体加速可达 **12×**（在 bs=16384 时）。  

**🎯 影响范围**：  
- `python/sglang/jit_kernel/`（新建 `set_mla_kv_buffer.py`、`csrc/elementwise/set_mla_kv_buffer.cuh`、benchmark、tests）  
- `python/sglang/srt/mem_cache/utils.py`（调度逻辑改动）  
- 依赖此接口的上层模块（如模型推理、KV 缓存写入）均会受到加速影响。  

**🔍 技术洞察**  

- **架构影响**  
  - 引入了基于 **TMA（Tensor Memory Access）** 的 bulk‑store 机制，仅在 **SM90+**（Hopper）GPU 上可用；对旧架构保持向后兼容，仍然走 Triton fallback。  
  - JIT 编译路径通过模板化的 `SetMlaKVBufferKernel<nope_bytes, rope_bytes, use_pdl>` 动态生成特化 kernel，减少二进制体积并在运行时根据 **nope/rope** 字节大小加载合适实现。  
  - 新增的 **PDL（Preemptive Distributed Launch）** 支持，使 kernel 能在被抢占后安全恢复，保持与现有 `launch_pdl` 接口一致。  

- **性能影响**  
  - **大批量写入**（`batch_size >= 768`）每个 CTA 处理 4–8 条记录，显著降低 CTA 数量与调度开销；相比原先每条记录一个 CTA 的 Triton 实现，提升 **3–12×**（取决于 batch size 与行宽）。  
  - **小批量**仍使用改进的单 CTA Triton 实现，`BLOCK = next_pow2(total_dim)` 可在 2–512 范围内提供 **1.05–1.5×** 的提升，避免了在 TMA 路径上因批量过小导致的额外同步开销。  
  - 引入的 `fence.proxy.async.shared::cta` 与 `wait_group` 确保数据可见性，避免了在大 batch 下出现“零行”错误。  

- **安全考虑**  
  - 代码对输入张量的 **形状、stride、dtype** 进行严格的 `TensorMatcher` 校验，防止非法内存访问。  
  - `loc` 索引支持 `int32` 与 `int64`，在 kernel 参数构造时已做类型分支，避免了类型不匹配导致的越界。  
  - 仍使用原始指针 (`void*`) 与 `cp.async.bulk`，若开发者错误地绕过 `set_mla_kv_buffer_triton` 的调度逻辑，可能触发未对齐或越界写入；因此文档中必须强调 **仅在 SM90+ 且满足 16‑byte 对齐** 时调用 JIT 路径。  

**⚠️ 潜在风险**  

1. **硬件兼容性**：在非 SM90 GPU（如 Ada、Ampere）上若误触发 JIT 路径会因不支持 `cp.async.bulk` 而抛异常或产生未定义行为。  
2. **对齐要求**：`nope_bytes + rope_bytes` 必须是 16 的倍数，且各自需是 4 的倍数；不满足时 `can_use_set_mla_kv_buffer` 会返回 `False`，但若调用方自行绕过检查仍可能导致 kernel 编译失败。  
3. **JIT 编译开销**：首次调用时会触发 CUDA 源码编译，可能导致启动延迟；在多进程或频繁创建新 CUDA 上下文的场景下需确认编译缓存的共享行为。  
4. **PDL 依赖**：若运行环境未启用 PDL（如旧版 driver），`launch_pdl` 参数会被忽略，仍能正常运行，但在抢占/迁移时可能出现性能回退。  
5. **阈值硬编码**：`_TMA_BULK_STORE_MIN_LOCS = 768` 是在 GB300（A800）上调优的经验值，其他 GPU（SM90+ 的不同 SM 数或不同缓存层）上可能出现次优选择。  

**💡 关注建议**  

- **CI/测试**：在 CI 中加入 SM90+（Hopper）实际硬件的性能基准，以验证阈值 `768` 在不同平台的有效性；若发现偏差，可考虑动态基准或配置化阈值。  
- **文档与使用指南**：在 `set_mla_kv_buffer` 与 `set_mla_kv_buffer_triton` 的 docstring 中明确说明：*仅在 SM90+ 且行宽满足 16‑byte 对齐时才会使用 JIT TMA 路径*，并提供 `can_use_set_mla_kv_buffer` 示例。  
- **错误处理**：在 `set_mla_kv_buffer_triton` 的 fallback 分支前加入日志/警告，记录因对齐或硬件不满足而退回 Triton 的原因，帮助排查性能异常。  
- **缓存管理**：确保 `cache_once` 装饰器的实现能够在多进程/多 GPU 环境下安全共享编译产物，防止重复编译导致的资源浪费。  
- **回滚机制**：如果在生产环境出现异常（如 kernel 启动失败），可以捕获异常并强制回退到 Triton 实现，以保证功能不受影响。  
- **Future 扩展**：考虑将阈值改为基于 `batch_size * row_bytes / SM_mul` 的动态计算，以自适应不同 GPU 的吞吐能力；同时可以在 `set_mla_kv_buffer` 中暴露 `force_triton` 或 `force_jit` 参数，便于调试与 A/B 测试。
