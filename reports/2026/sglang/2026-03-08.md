### fix: add ModelScope cache lookup and speculative path support (#20098)
**SHA**: `72f6dfc` | 🔗 [查看提交](https://github.com/sgl-project/sglang/commit/72f6dfcc31cbc721d6190a23b1f7270a085f1ca0)

**🎯 变更类型**：Bug修复 / 功能增强  
**⚡ 重要程度**：🔴高  
**📋 变更摘要**：  
- 在 `server_args.py` 中抽象出 `_handle_modelscope_paths`，实现对模型、分词器以及 speculative‑draft 模型路径的本地 ModelScope 缓存查找，若本地不存在则回退至 `snapshot_download` 自动下载。  
- 在 `speculative/spec_utils.py` 中完善 token‑map 加载逻辑：先尝试在 ModelScope 本地缓存目录中查找，若未命中再根据环境变量决定使用 ModelScope‑或原生 `snapshot_download` 进行下载。  
- 新增对 `speculative_draft_model_path` 的解析支持，提升了推理时使用草稿模型的灵活性。

**🎯 影响范围**：  
- `python/sglang/srt/server_args.py`（启动参数解析、默认值填充）  
- `python/sglang/srt/speculative/spec_utils.py`（Speculative sampling 的 token map 加载）  
- 依赖于 ModelScope SDK 的模型加载路径解析（所有使用 `ModelScope` 下载模型的入口）  

**🔍 技术洞察**  

- **架构影响**  
  - 将 ModelScope 路径解析抽象为独立方法，降低了 `server_args` 中的耦合度，便于后续在其它组件复用。  
  - 引入对 `speculative_draft_model_path` 的统一处理，使得草稿模型的缓存策略与主模型保持一致，提升了整体架构的对称性。  
  - 通过环境变量 `SGLANG_USE_MODELSCOPE` 控制是否走 ModelScope 路径，保持向后兼容，未影响已有的 HuggingFace 下载路径。

- **性能影响**  
  - **缓存命中**：本地缓存直接返回路径，省去网络请求及磁盘写入，显著加速启动及首次推理时间。  
  - **缓存未命中**：仍然会调用 `snapshot_download`，但在模型已在本地 ModelScope 缓存目录时可避免重复下载，降低磁盘 I/O 与带宽消耗。  
  - **额外检查**：在每次启动时会额外检查 `os.path.exists`、`os.path.join`，开销极小（毫秒级），对整体性能影响可忽略不计。

- **安全考虑**  
  - 依赖 ModelScope SDK 的 `snapshot_download`，仍然遵循其安全校验（文件哈希校验），未改变安全模型。  
  - 通过 `ignore_patterns` 过滤二进制模型文件，防止意外下载不必要的大文件。  
  - 对 `speculative_token_map` 保持旧有逻辑，避免因路径解析错误导致加载未受信任文件。

**⚠️ 潜在风险**  

1. **环境变量冲突**：如果用户同时设置 `SGLANG_USE_MODELSCOPE=1` 且自定义 `download_dir` 与 ModelScope 默认缓存目录不同，可能导致出现两个相同模型的副本，占用额外磁盘空间。  
2. **路径解析误判**：`_resolve_or_download` 把任意非本地路径当作 ModelScope repo_id 处理，若用户误将本地相对路径写成不存在的字符串，仍会触发网络下载并抛出未知错误。  
3. **ModelScope SDK 版本兼容**：该实现依赖 `modelscope.hub.snapshot_download` 与 `modelscope.utils.file_utils.get_model_cache_root`，若 ModelScope SDK 升级删除或改名这些接口，会导致运行时 `ImportError`。  
4. **并发下载**：在多进程/多实例启动时可能出现同一模型的并发下载，导致临时冲突或锁文件异常（虽然 ModelScope 自身有锁机制，但仍需注意）。

**💡 关注建议**  

- **开发者**  
  - 为关键路径添加单元测试：包括本地缓存命中、缓存未命中、`download_dir` 覆盖以及 `speculative_draft_model_path` 的三种情况。  
  - 在 `requirements` 中明确 `modelscope>=X.Y` 的最低版本，以防 API 变更。  
  - 考虑在 `_handle_modelscope_paths` 中加入日志（`logger.debug/info`）记录是“缓存命中”还是“下载”，便于排查问题。  
  - 如有必要，可在后续版本提供 `--model-cache-dir` 参数，让用户显式指定 ModelScope 缓存根目录，避免与默认路径冲突。

- **用户**  
  - 在使用 ModelScope 模型前，确保环境变量 `SGLANG_USE_MODELSCOPE=1` 已正确设置。  
  - 若磁盘空间紧张，可通过 `--download-dir` 指定专门的下载目录，并定期清理不再使用的模型缓存。  
  - 对于大规模部署（多节点），建议统一 `download_dir` 或提前在每台机器上预热模型缓存，避免启动时的并发下载。  

通过上述改动，sglang 在 ModelScope 环境下的启动速度与资源利用率得到明显提升，同时保持了向后兼容和安全性。仅需留意版本兼容与缓存目录管理，即可安全上线。

### Support HiCache for MambaRadixCache (#19663)
**SHA**: `5867c3f` | 🔗 [查看提交](https://github.com/sgl-project/sglang/commit/5867c3fa80be9e2ba5725fcf59eda2f8d31217d1)

**🎯 变更类型**：功能增强 / 架构变更  
**⚡ 重要程度**：🔴 高  

**📋 变更摘要**  
1. 为 **MambaRadixCache** 引入了 HiCache（分层缓存）实现 `HiMambaRadixCache`，实现了 **Full KV**（注意力）缓存的 L2/L3（主机/存储）备份与写回，同时保留 **Mamba 状态** 只在 GPU 上 via LRU。  
2. 在 `CacheController`、`ScheduleBatch`、`Scheduler` 等调度层加入对 **HybridLinearKVPool** 的兼容以及对 **backup‑node**（最深已备份节点） 的新字段 `last_host_backup_node`，以支持从存储预取。  
3. 扩展 `MatchResult`、`BasePrefixCache`，加入 `last_host_backup_node` 与相应的返回逻辑。  
4. 为 `TreeNode` 增加 **host 引用计数**、**hash 值**、**前缀 hash** 计算及 LRU 保护/释放接口。  
5. 对 `MambaRadixCache` 做了细粒度的 **写/读锁**、**命中计数**、**分割节点** 等兼容性改造。  

**🎯 影响范围**  
- `python/sglang/srt/managers/cache_controller.py`  
- `python/sglang/srt/managers/schedule_batch.py`  
- `python/sglang/srt/managers/scheduler.py`  
- `python/sglang/srt/mem_cache/base_prefix_cache.py`  
- `python/sglang/srt/mem_cache/mamba_radix_cache.py`（新增属性 & 方法）  
- `python/sglang/srt/mem_cache/hi_mamba_radix_cache.py`（全新实现）  
- 相关的 import 路径、类型声明与 RPC 参数/返回结构。

---

### 🔍 技术洞察  

| 维度 | 影响说明 |
|------|----------|
| **架构影响** | - 新增 **HiMambaRadixCache** 继承自 `MambaRadixCache`，在原有 GPU‑KV 结构之上加入 **层级缓存（Device → Host → Storage）**。 <br>- `Scheduler.init_cache_with_memory_pool` 根据 `self.is_hybrid_ssm` 动态选择 `HiMambaRadixCache`（或回退 `HiRadixCache`）。 <br>- `CacheController` 在初始化时识别 `HybridLinearKVPool` 并切换到其完整 KV 子池，保证上层逻辑统一使用 `full_kv_pool`。 <br>- `MatchResult` 增加 `last_host_backup_node`，用于 **prefetch** 时定位最近已备份的节点。 |
| **性能影响** | - **正面**：可以把完整 KV（注意力）分页备份到主机或磁盘，显著降低 GPU 显存压力，尤其在 **Hybrid‑SSM**（Mamba + Attention）模型下可容纳更长序列。 <br>- **负面**：<br>  • 计算并维护 **hash_value**、**prefix hash**（递归 `lru_cache`）带来额外 CPU/内存开销。 <br>  • 引入 **host_ref_counter**、**锁引用计数**，在高并发请求下若锁管理不严密可能导致泄漏或阻塞。 <br>  • 额外的 **写‑回 / 预取** IO（取决于后端 storage）会产生延迟，需依赖 `prefetch_threshold`、`prefetch_timeout` 调优。 |
| **安全考虑** | - 目前没有直接的安全敏感路径（如外部输入直接写入磁盘），但： <br>  • **存储后端** 参数（路径、权限）在 `attach_storage_backend` 中直接传入，若管理员误配置可能导致数据泄露或写入非预期位置。 <br>  • `host_ref_counter` 的递减未加锁，若同一节点被多线程并发访问，可能出现负值，从而触发 `RuntimeError`。建议在多进程/多线程环境下加锁或使用原子计数。 |
| **可维护性** | - 代码体量大幅增加（约 1.7k 行新文件），类层次更深，需要 **完整的单元测试** 与 **集成测试**。 <br> - 新增的 `last_host_backup_node` 使 `MatchResult` 接口不向后兼容；调用方必须更新或提供默认值以防止 `AttributeError`。 <br> - 大量 `if isinstance(..., HybridLinearKVPool)` 的分支散布在多个文件，易产生 **分支漂移**，建议抽象为统一的适配层。 |

### ⚠️ 潜在风险  

1. **并发计数错误**  
   - `host_ref_counter`、`full_lock_ref`、`mamba_lock_ref` 的增减没有显式线程/进程同步。若多请求并发同一节点，可能出现计数不匹配导致永久锁定或提前释放。  

2. **回退兼容性**  
   - `MatchResult` 新增字段默认值为 `None`，但已有代码（第三方插件或旧版本）可能直接解构 `MatchResult`，导致 `TypeError`。  

3. **存储后端异常**  
   - `attach_storage_backend`、`detach_storage_backend` 中的大量 I/O 操作若抛异常，可能导致缓存状态不一致（比如已写入一半的 KV 没有回滚）。  

4. **内存泄漏**  
   - `prefetch_loaded_tokens_by_reqid`、`ongoing_*` 字典在异常路径未必被清理（尤其在 `shutdown` 期间），可能残留引用导致显存/主机内存无法释放。  

5. **递归哈希计算**  
   - `TreeNode.get_prefix_hash_values` 使用递归 + `lru_cache(maxsize=1)`，在极深树（大量 token）时可能触发 Python 递归深度限制或导致栈溢出。  

6. **性能回退**  
   - 对不使用 HiCache 的普通模型仍会加载 `HiMambaRadixCache` 相关代码，增加模块加载时间和整体二进制大小。

**🎯 影响范围**  
- `python/sglang/multimodal_gen/benchmarks/bench_serving.py`（核心逻辑、异步请求、指标统计）  
- `python/sglang/multimodal_gen/benchmarks/datasets.py`（数据结构扩展）  
- 相关依赖：`dataclasses.replace`、`numpy`（已隐式使用）
