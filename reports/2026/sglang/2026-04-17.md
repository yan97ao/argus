### feat(observability): add OpenTelemetry tracing for speculative decoding (#19545)
**SHA**: `7390edd` | 🔗 [查看提交](https://github.com/sgl-project/sglang/commit/7390eddf289997059702d941a4dcb2c96bdf18b6)

**🎯 变更类型**：功能增强  
**⚡ 重要程度**：🔴 高  

**📋 变更摘要**  
本次提交为 SGLang 的推理调度层新增 OpenTelemetry 观测能力，专注于 **speculative decoding（投机式解码）** 的关键阶段。新增了 `spec_draft`、`spec_verify`、`spec_draft_extend` 三个请求阶段及对应的时间统计字段，并在 Eagle‑Worker 与 N‑gram‑Worker 中注入启动/结束时间标记，支持通过全局开关仅在需要时进行追踪。

**🎯 影响范围**  
- `python/sglang/srt/observability/req_time_stats.py`（请求时间统计与追踪核心）  
- `python/sglang/srt/speculative/eagle_worker.py`（Eagle 方案的投机式解码）  
- `python/sglang/srt/speculative/ngram_worker.py`（N‑gram 方案的投机式解码）  
- 相关的调度批处理工具函数 `set_time_batch`  

**🔍 技术洞察**  

- **架构影响**  
  - 在观察性子系统中新增 **RequestStageConfig** 实例，形成投机式解码的独立层级（Level 2/3），与已有的 Prefill、Decode 等阶段保持一致的抽象。  
  - `SchedulerReqTimeStats` 中加入三个专用时间点，使每个请求对象能够独立记录投机式解码的开始/结束时间，进而通过 `trace_slice` 把数据写入 OpenTelemetry。  
  - `set_time_batch` 参数扩展为 `trace_only`，配合全局开关 `get_global_tracing_enabled()`，实现 **“仅在打开追踪时才产生额外开销”** 的设计，保持原有调度路径的轻量化。  

- **性能影响**  
  - **关闭追踪**：几乎无额外开销，只会多一次布尔检查。  
  - **打开追踪**：在每个批次的投机式步骤前后调用 `time.perf_counter()` 并执行 `trace_slice`，会产生微量的 CPU 与内存开销（主要是时间戳采集与 OTEL Span 创建），对高吞吐场景的影响可接受。  
  - 新增的字段（`spec_*_start_time`）为标量，内存占用增长极小（每请求 ~24 bytes）。  

- **安全考虑**  
  - 仅记录 **时间戳** 与 **accepted_tokens**（整数计数），不涉及用户内容或模型输出，不会泄露敏感信息。  
  - 需要确保所部署的 OpenTelemetry 后端具备适当的访问控制，防止外部探测系统获取内部时序数据用于侧信道攻击。  

**⚠️ 潜在风险**  
1. **全局追踪开关误用**：如果 `get_global_tracing_enabled()` 返回 `True` 而 OpenTelemetry 未正确初始化，`trace_slice` 可能抛异常导致请求崩溃。  
2. **并发计时冲突**：在同一请求对象的不同阶段并发执行（如多线程/多进程调度）时，写入 `spec_*_start_time` 与 `trace_slice` 可能出现竞争，需确认调度器对单请求的时序是串行的。  
3. **向后兼容性**：新增的 `RequestStage` 常量在其他模块的 `stage` 列表或可视化工具中未及时更新，可能导致 UI/报表显示缺失或报错。  

**💡 关注建议**  
- **测试覆盖**：为 `set_spec_*_start_time` / `set_spec_*_end_time` 增加单元测试，尤其在 **Tracing‑Enabled** 与 **Tracing‑Disabled** 两种环境下的行为验证。  
- **异常防护**：在 `trace_slice` 前加入 try/except，记录错误但不影响主流程，防止追踪后端异常导致服务不可用。  
- **文档与监控**：更新 OpenTelemetry 规范文档，说明新增的 `spec_draft`、`spec_verify`、`spec_draft_extend` span 名称、属性以及推荐的采样策略。  
- **性能基准**：在开启追踪的情况下跑一次吞吐量基准（如 1000 QPS），确认额外的时间开销在可接受范围（≤ 5 ms/req）。  
- **兼容检查**：确保所有旧的监控脚本、仪表盘在加入新阶段后仍能正常运行，必要时加入后向兼容的默认值（如 `0`）。  

通过以上措施，可在保持系统高性能的前提下，获得投机式解码细粒度的可观测性，为后续调优与故障定位提供重要数据支撑。

### Revert "feat: Support MXFP4 quantized dense models on AMD CDNA2/CDNA3 GPUs (#19143)" (#23031)
**SHA**: `f0f0148` | 🔗 [查看提交](https://github.com/sgl-project/sglang/commit/f0f014816719bd3844de5cfe4a110770c7cca137)

**🎯 变更类型**：重构（功能回退）  
**⚡ 重要程度**：🔴高  
**📋 变更摘要**：本次提交撤销了之前对 MXFP4 量化模型在 AMD CDNA2/CDNA3 GPU 上的支持。具体操作包括：删除 `petit_mxfp4` 实现文件及相关导入、在配置校验中去除 `petit_mxfp4` 条目、将 `petit_kernel` 依赖回退到 0.0.2 版本、并在公开的量化选项列表中去除 `petit_mxfp4`。此举恢复到仅支持 `petit_nvfp4` 的状态，避免了潜在的未成熟代码路径。  

**🎯 影响范围**：  
- `sglang/srt/layers/quantization` 目录下的 MXFP4 相关实现（`petit_mxfp4.py`、`petit_nvfp4.py` 旧版）全部被删除。  
- `model_config.py`、`quantization/__init__.py`、`server_args.py` 中的量化选项列表和校验逻辑。  
- `pyproject_other.toml` 中的 `petit_kernel` 依赖版本。  
- 任何使用 `petit_mxfp4` 或在配置文件中声明 `mxfp4`、`quark`（伴随 MXFP4） 的模型加载路径。  

**🔍 技术洞察**：  
- **架构影响**：  
  - 去除了 MXFP4 量化的完整实现链路（配置 → 线性层 → kernel 调用），系统量化子系统结构变得更简洁，仅保留 NVFP4 路径。  
  - `QuantizationConfig` 的映射表从 13 条削减至 12 条，降低了 `override_quantization_method` 判断分支数。  
  - 依赖层面，`petit_kernel` 版本回退至 0.0.2，避免了在未正式发布的 0.0.3 版本上出现的兼容性问题。  

- **性能影响**：  
  - 移除 MXFP4 支持意味着失去其在特定 AMD GPU（gfx90a/gfx942）上潜在的吞吐量提升（更高的位宽压缩 & 专用 kernel）。  
  - 同时，删除了未充分测试的代码路径，可防止因错误实现导致的运行时崩溃或性能回退。整体性能对多数用户（使用 FP16/BF16/其他已支持量化）不受影响。  

- **安全考虑**：  
  - 代码量减少，攻击面随之收窄；没有新增的外部调用或内存操作。  
  - 仍保留对 `petit_kernel` 的动态导入检查，若用户自行安装 0.0.3 版本，仍会在运行时抛出明确错误信息。  

**⚠️ 潜在风险**：  
1. **兼容性回退**：已有使用 `petit_mxfp4`（或基于 MXFP4 的 Quark）模型的用户在更新后将无法加载模型，出现 `ValueError: Unsupported quantization method`。  
2. **文档同步风险**：如果项目文档、示例代码或公开的模型卡仍标记支持 MXFP4，用户可能产生误解。  
3. **依赖冲突**：降级 `petit_kernel` 可能与用户自行安装的更高版本产生冲突（pip 解析时会强制降级），需要确认兼容性。  
4. **遗漏的引用**：虽已删除主要实现文件，但仍可能残留在旧的序列化 checkpoint（`hf_quant_config.json`）中未清理的 `mxfp4` 标记，导致模型加载时报错。  

**💡 关注建议**：  
- **迁移指南**：在 release notes 中明确说明 MXFP4 支持已被移除，并提供替代方案（使用 `petit_nvfp4` 或其他已验证量化）。  
- **回滚兼容层**：若业务上仍有少量 MXFP4 需求，可考虑提供一个 “兼容模式” 警告后直接使用 `petit_nvfp4`，或在 `model_config` 中保留占位键，以免现有 checkpoint 直接报错。  
- **文档与 CI**：更新所有 README、API 文档、模型卡以及 CI 测例，确保不再列出 `petit_mxfp4`。  
- **依赖管理**：在 `pyproject_other.toml` 中声明 `petit_kernel>=0.0.2,<0.0.3`，防止意外升级引入不兼容的代码。  
- **回归测试**：增加针对 `model_config._verify_quantization` 的单元测试，验证 `petit_mxfp4` 关键字被正确拒绝，并确保其他量化路径仍可正常通过。  

通过以上措施，可将本次功能回退的影响降至最小，并保持项目在 AMD 环境下的稳定性与可维护性。

### feat: add coordinated checkpoint prefetch for network filesystem loading (#20843)
**SHA**: `04a5395` | 🔗 [查看提交](https://github.com/sgl-project/sglang/commit/04a53955b98ec0d50f90fb5ad216cf3df4a45ee9)

**🎯 变更类型**：功能增强  
**⚡ 重要程度**：🔴高  
**📋 变更摘要**：  
- 为 safetensors 权重加载器新增“检查点预取”机制，在后台对 checkpoint 文件进行顺序读取以提前填充 OS 页缓存，尤其针对 NFS、Lustre 等共享文件系统。  
- 引入对应的命令行参数 `--weight-loader-prefetch-checkpoints` 与 `--weight-loader-prefetch-num-threads`，以及环境变量 `SGLANG_PREFETCH_BLOCK_SIZE_MB` 用于调节预取块大小。  
- 实现分布式协同预取（每个 Rank 只负责自己的一小部分 shard），并加入单元及多 GPU 集成测试，确保功能正确且权重保持一致。

**🎯 影响范围**：  
- `python/sglang/srt/model_loader/loader.py`、`weight_utils.py`（权重加载流程）  
- `python/sglang/srt/server_args.py`（CLI 参数）  
- `python/sglang/srt/environ.py`（新增 env 变量）  
- 文档 `server_arguments.md`、`environment_variables.md`  
- 测试目录 `test/registered/...`（新增单元与多 GPU 测试）

**🔍 技术洞察**  

- **架构影响**  
  - 在分布式环境中引入了 **节点本地 rank** 的概念 (`get_world_group().local_rank`) 用于划分预取任务，使得预取在同一节点内部均匀分摊。  
  - 通过 **异步 + 多线程**（`asyncio` + `threading.Thread`）实现后台预取，保持原有权重加载路径不变，对现有模型加载架构兼容。  
  - 新增 `prefetch` 与 `prefetch_num_threads` 参数，向 `safetensors_weights_iterator`、`buffered_multi_thread_safetensors_weights_iterator` 传递，保持向后兼容（默认关闭）。

- **性能影响**  
  - **网络 I/O**：原先每个 Rank 都会独立 mmap 同一文件，导致 `N * checkpoint_size` 的读取；现在每个 Rank 只读取 1/N 的 shard，理论上把网络流量降低到 **1 × checkpoint**。  
  - **页缓存命中率**：预读取顺序块（默认 16 MiB）让内核提前把文件页放入页面缓存，后续 mmap 能快速命中，降低磁盘/网络延迟。  
  - **CPU/内存占用**：后台预取使用 `num_threads`（默认 4）并行读文件，会占用一定 CPU 与临时内存，且页面缓存的占用会随 checkpoint 大小线性增长。  
  - **可配置性**：通过 `SGLANG_PREFETCH_BLOCK_SIZE_MB` 调整块大小，可在内存受限的节点上降低一次性读取量。

- **安全考虑**  
  - 读取文件本身不涉及外部网络或权限提升，仅在本地磁盘上执行顺序 `open/read`。  
  - 新增的线程与 asyncio 运行在进程内部，不引入额外的网络通信或跨进程共享内存风险。  
  - 仍需确保文件路径来源可信，防止恶意 checkpoint 被预读取导致 DoS（大量 I/O）——这在原始加载流程中已经存在。

**⚠️ 潜在风险**  

1. **页缓存争用**：在同一节点上多任务同时预取大模型 checkpoint，可能导致页面缓存被大量占用，挤压其他进程的内存。  
2. **I/O 饱和**：若磁盘或网络文件系统本身带宽受限，预取线程的并发读可能触发 I/O 峰值，引起短暂的延迟或超时。  
3. **线程/asyncio 资源泄露**：后台线程以 daemon 方式启动，进程退出时会被强制终止；如果进程异常重启，可能留下未完成的 I/O，需确认不会产生锁文件等副作用。  
4. **与 `weight_loader_disable_mmap` 冲突**：预取仅在 `disable_mmap=False` 时生效，若用户关闭 mmap，预取仍会执行但无实际收益，且会产生不必要的 I/O。  
5. **分布式环境变量不一致**：`get_world_group()` 依赖内部的分布式组实现，若用户自定义了不同的组划分，预取划分可能不符合预期，导致某些 shard 被重复或遗漏预取。

**💡 关注建议**  

- **默认保持关闭**：生产环境建议仅在确认模型文件位于共享网络文件系统（NFS、Lustre 等）且节点具有足够的页缓存空间时打开 `--weight-loader-prefetch-checkpoints`。  
- **调参**：根据节点磁盘带宽与 CPU 核心数，适当增减 `--weight-loader-prefetch-num-threads`（不宜超过磁盘并发读能力），并通过 `SGLANG_PREFETCH_BLOCK_SIZE_MB` 调整块大小以适配内存限制。  
- **监控**：开启预取后，监控 `vmstat`、`iostat`、`nfsstat` 中的页面缓存占用、读取速率以及 `dmesg` 中的 I/O 错误。  
- **兼容性检查**：在使用 `weight_loader_disable_mmap=True` 时，可显式关闭预取以避免无效 I/O；在单机单卡场景下可直接关闭，以免产生额外线程开销。  
- **测试覆盖**：持续运行新增的单元及多 GPU 集成测试，以捕获未来对 `safetensors`、分布式组实现或文件锁逻辑的回归。  
- **文档与 CI**：建议在 CI 中添加针对不同文件系统（本地 SSD vs NFS）模拟的基准，帮助用户判断是否开启预取能带来实质收益。
