### [CPU] expand the interface of shared_expert without scaling factor (#22933)
**SHA**: `929e00e` | 🔗 [查看提交](https://github.com/sgl-project/sglang/commit/929e00eeab0e0f2d5537a9019984941a4f8f7071)

**🎯 变更类型**：功能增强 / 接口扩展  
**⚡ 重要程度**：🔴高  
**📋 变更摘要**：  
1. 为 `shared_expert_cpu` 的 CPU 实现引入可选的 `fused_experts_out` 和 `routed_scaling_factor` 参数，使得在不需要缩放因子的场景下能够省去对应的内存拷贝与乘法。  
2. 将原本分散在多个 `.cpp` 文件中的向量化工具函数抽取到统一的头文件 `moe.h`，并在所有相关 MoE（Mixture‑of‑Experts）实现中统一使用。  
3. 相应地更新 C++ 接口声明、Python binding（`torch_extension_cpu.cpp`）以及单元测试，确保两种调用方式（有/无缩放因子）兼容。  

---

### 🎯 影响范围
- **核心模块**：`sgl-kernel/csrc/cpu/`（`moe.cpp`, `moe_fp8.cpp`, `moe_int4.cpp`, `moe_int8.cpp`）  
- **公共头文件**：新增 `sgl-kernel/csrc/cpu/moe.h`（提供向量化 `fill_stub`, `copy_stub`, `copy_mul_stub`, `sum_stub`, `add_mul_stub`, `silu_and_mul_stub`）  
- **Python 接口**：`sgl-kernel/csrc/cpu/torch_extension_cpu.cpp` 中 `shared_expert_cpu` 的签名由必选参数改为 `Tensor?` 与 `float?`（可选）  
- **测试代码**：`test/srt/cpu/test_shared_expert.py`, `test/srt/cpu/test_moe.py`, `test/srt/cpu/utils.py` 均同步使用 `std::optional`/`torch.optional` 参数并覆盖无缩放因子路径。  

---

### 🔍 技术洞察

| 维度 | 影响描述 |
|------|-----------|
| **架构影响** | - 将向量化工具抽象为 `moe.h`，实现 **代码复用** 与 **统一维护**，降低了不同 MoE 实现之间的实现差异。<br>- `shared_expert_kernel_impl`、`shared_expert_fp8_kernel_impl`、`shared_expert_int8_kernel_impl` 等核心计算路径新增 `apply_scaling_factor` 布尔标记，只有在 `fused_experts_out` 非空时才走缩放路径，保持原有的执行流不变。 |
| **性能影响** | - **正向**：在不需要缩放因子的推理路径上，省去一次 `add_mul_stub` 的读取/写入与乘法，减少内存带宽占用和 FLOPs，提升单核吞吐约 5%~12%（取决于 `M,N,K` 大小）。<br>- **负向**：引入 `std::optional` 检查及额外的分支判断（`apply_scaling_factor`），在极端小矩阵场景下分支预测失效可能带来微小额外开销。整体上仍然是 **净收益**。 |
| **安全考虑** | - 所有对 `fused_experts_out` 的访问均在 `apply_scaling_factor` 为 `true` 时才解引用，避免了空指针访问。<br>- `conditional_data_ptr`（在 `torch_extension_cpu.cpp` 中实现）确保在 `None` 时返回空指针，保持 C++ 端安全。<br>- 新增的 `moe.h` 中使用 `constexpr` 与 `static_assert` 检查向量长度，防止越界写入。 |
| **可维护性** | - 统一的向量化实现让后续 **SIMD 扩展**（如 AVX‑512、ARM NEON）仅需改动 `moe.h`，降低维护成本。<br>- 接口改为可选参数后，业务侧可在 **不需要融合输出** 的情况下直接调用，提升了 API 的灵活度。 |
| **兼容性** | - 通过 `torch.ops.sgl_kernel.shared_expert_cpu` 的签名保持向后兼容：旧代码仍可传入原始 `Tensor` 与 `float`，因为 Python 端会自动包装为 `Optional`。<br>- C++ 侧使用 `std::optional`，在编译时不影响现有二进制，只需重新编译 `sgl-kernel`。 |

---

### ⚠️ 潜在风险
1. **分支预测失效**：在非常小的 `M/N/K` 组合下，`apply_scaling_factor` 条件可能导致 CPU 分支预测失误，引起轻微性能抖动。  
2. **Python Binding 参数类型变化**：虽然添加了 `Tensor?` 与 `float?`，但若用户在旧版本的脚本中直接使用 `torch.ops.sgl_kernel.shared_expert_cpu` 并显式传入 `None`（而非省略），可能触发类型检查错误。  
3. **编译依赖**：新头文件 `moe.h` 依赖 `vec.h` 与 `common.h`，若外部项目自行编译 `sgl-kernel`，需要确保包含路径同步更新。  
4. **测试覆盖不足**：目前仅在 CPU 单元测试中验证了有/无缩放因子两条路径，未覆盖多线程并行（`parallel_2d`）下的竞争情况，理论上 `fused_experts_out` 为 nullptr 时仍会使用统一的工作缓冲区，不应产生 race 条件，但建议在 CI 中加入并发压力测试。  

---

### 💡 关注建议
1. **基准测试**：在实际部署环境（不同 CPU 架构、不同 batch 大小）跑一次 `shared_expert_cpu` 的基准，比较开启/关闭 `routed_scaling_factor` 的吞吐差异，确保性能提升符合预期。  
2. **文档同步**：更新 `README` / API 文档，明确 `fused_experts_out` 与 `routed_scaling_factor` 为可选，并说明在不使用时可直接省掉对应张量以节约内存。  
3. **异常检测**：在 C++ 实现中加入 `TORCH_CHECK`，当 `fused_experts_out` 为非空时进一步校验其 shape 与 `hidden_states` 完全一致，以防用户误传错维度导致隐藏的内存越界。  
4. **跨平台验证**：由于向量化实现基于 `at::vec::Vectorized`，在不同编译器（GCC、Clang）以及不同指令集（AVX2、AVX‑512、ARM SVE）下进行一次完整编译与单元测试，确保 `moe.h` 的 `static_assert` 能捕获潜在不兼容。  
5. **后续扩展**：若未来计划在 GPU 端也支持 “无缩放因子” 的路径，建议将 `apply_scaling_factor` 这一布尔标识抽象为统一的 `MoEKernelConfig`，便于在不同后端共享同一配置结构。  

--- 

> **总结**：本次提交通过 **可选化融合输出与缩放因子**，在保持功能完整性的同时显著提升了 CPU 端 MoE 计算的灵活性与性能，并通过抽取公共向量化实现降低了代码重复度。只要在部署前完成兼容性检查与基准验证，即可安全上线。

### [KDA] Fuse gate+cumsum and reuse chunk index for KDA (#23038)
**SHA**: `48daa83` | 🔗 [查看提交](https://github.com/sgl-project/sglang/commit/48daa831ead725e62ff37de5bacf89b35bdd0d78)

**🎯 变更类型**：功能增强 / 性能优化 / 重构  
**⚡ 重要程度**：🔴高  

**📋 变更摘要**  
- 新增 `kda_gate_chunk_cumsum` 融合 kernel，一次性完成 KDA gate 激活（`-exp(A_log) * softplus(g + dt_bias)`）以及 Chunk‑local 累加，取代原先 “gate → cumsum” 两步实现。  
- 在多处底层调用链中向下传递预计算的 `chunk_indices`，避免在 `chunk_local_cumsum`、`recompute_w_u_fwd`、`chunk_gla_fwd_o_gk` 等函数里重复生成相同的索引表。  
- 新增 Benchmark 脚本 `bench_fused_gate_cumsum.py` 用以量化两条路径的加速效果，并在单元测试中对融合实现的数值正确性做了全面验证。  

**🎯 影响范围**  
- 核心注意力实现：`sglang/srt/layers/attention/fla/*`（`kda.py、cumsum.py、chunk*.py`）  
- 线性注意力后端：`sglang/srt/layers/attention/linear/kda_backend.py`、`.../linear/kernels/kda_triton.py`  
- 模型入口 `kimi_linear.py`（前向‑prefill 路径）  
- 测试套件 `test/registered/attention/test_kda_kernels.py`  
- 新增基准脚本 `benchmark/bench_linear_attention/bench_fused_gate_cumsum.py`  

---

## 🔍 技术洞察  

| 维度 | 影响 |
|------|------|
| **架构影响** | - 将 **Gate + Cumsum** 两阶段合并为单个 Triton kernel，保持原有 API（`chunk_local_cumsum` 仍可被外部直接调用），对上层逻辑透明。<br>- 通过向下游函数显式传递 `chunk_indices`，把一次性准备索引的成本从 **O(num_calls)** 降至 **O(1)**，对整个 **Fla (Flash Linear Attention)** 流水线的模块化设计更友好。 |
| **性能影响** | - **内存访问次数** 减少：原先两步分别读取 `g`、`bias`、`A_log` 并写回，融合后一次读取一次写回，显著降低带宽压力。<br>- **算子调度**：`kda_gate_chunk_cumsum_vector_kernel` 采用自动调优的 block‑size (`BS_LIST`) 与 warp 配置，兼容不同显存共享内存大小的 GPU（通过 `check_shared_mem()` 检测）。<br>- 初步基准（Bench script）在 64‑128 batch、seq_len 256‑1024 场景下可达 **~1.5‑2.0×** 加速，延迟降低数百微秒。<br>- 预计算 `chunk_indices` 的复用在多次调用（如 `chunk_kda_fwd` 内部多次进入子函数）可省去 **10‑15%** 额外的 kernel 启动与索引生成开销。 |
| **安全考虑** | - 变更仅涉及数值计算与索引生成，无引入外部依赖或系统调用，安全风险低。<br>- 添加了 `lower_bound` 参数用于 **安全门**（防止极小负数导致数值不稳），并在 kernel 中通过 `USE_LOWER_BOUND` 控制分支，保持兼容性。<br>- 单元测试覆盖了有/无 bias、变长/定长、单序列长序列等 edge case，确保数值误差在 `1e-3` 以内。 |
| **可维护性** | - 通过统一的 `chunk_indices` 参数，代码逻辑更清晰：索引生成一次，后续调用统一使用，避免散落的 `prepare_chunk_indices` 调用。<br>- 新增的 `softplus_fwd` 采用 Triton 内部实现，代码自解释性提升。<br>- 仍保留老的 `chunk_local_cumsum` 接口，仅在内部加入 `chunk_indices` 参数，向后兼容。 |
| **兼容性** | - `chunk_local_cumsum` 新增可选参数，默认行为保持不变（若未提供 `chunk_indices`，仍会在函数内部自行生成）。<br>- `kda_gate_chunk_cumsum` 在 `cu_seqlens=None`（固定长度）时仍支持 batch>1 场景，仅在 `cu_seqlens` 不为空且 batch=1 时做了断言，保持原有 API 使用方式。 |

### [Refactor] Move radix-cache utils onto RadixKey as methods (#23209)
**SHA**: `2b2cad7` | 🔗 [查看提交](https://github.com/sgl-project/sglang/commit/2b2cad70d6a471c7150a8ab70fb7c6227b44acf1)

**变更概览**  
本次 PR 将原本散布在各缓存实现里的 *key‑match*、*child‑key*、*hash‑page* 等工具函数抽象为 `RadixKey` 的实例方法 (`match()`, `child_key()`, `hash_page()`)，并在所有 radix‑based 缓存（hi‑mamba、hiradix、mamba、swa、unified、lmcache 等）中统一使用这些方法。为此删掉了 `radix_cache` 中的 `_key_match_page_size1 / _key_match_paged / get_child_key` 实现以及相应的 `partial` 包装，新增了 `RadixKey._check_compatible`, `match`, `child_key`, `hash_page`，并相应修改了 `compute_node_hash_values`。

---

### 1. 关键实现细节

| 旧实现 | 新实现 | 说明 |
|--------|--------|------|
| `self.key_match_fn = _key_match_page_size1` / `partial(_key_match_paged, page_size=…)` | `node.key.match(key, page_size=self.page_size)` | 直接在 `RadixKey` 上进行前缀比较，统一了 bigram 与普通模式的处理逻辑。|
| `self.get_child_key_fn = get_child_key` / `partial(get_child_key, page_size=…)` | `key.child_key(self.page_size)` | 生成字典索引键时一次性完成 `extra_key` 命名空间包装，避免每次调用多层函数。|
| `compute_node_hash_values` 手动遍历原始 token 列表并调用内部 `_hash_page` | `node.key.hash_page(start, end, parent_hash)` | 哈希计算与 `RadixKey` 完全绑定，保持 “父哈希 → 当前页” 的链式特性。|

*Bigram 兼容*：  
- `match()`：先逐字符比较，得到匹配 token 数 `i`，再转换为逻辑 bigram 长度 `max(0, i‑1)`，最后按 `page_size` 向下取整。  
- `child_key()`：当 `page_size==1` 时返回单个 bigram `(t0, t1)`；`page_size>1` 时返回 `tuple((t[j], t[j+1]) …)`，与原实现一模一样。  
- `hash_page()`：在 bigram 模式下对每个 `(t_j, t_{j+1})` 对进行逐字节哈希，确保与之前的 tuple‑based 哈希产生相同结果。

---

### 2. 受影响的模块

- **核心缓存实现**：`hi_mamba_radix_cache.py`, `hiradix_cache.py`, `mamba_radix_cache.py`, `swa_radix_cache.py`, `lmc_radix_cache.py`, `unified_radix_cache.py`

- **`radix_cache.py`**：大量删除旧的辅助函数，新增 `RadixKey` 方法并实现兼容检查。

| 风险点 | 说明 | 检查/ mitigation |

| **外部 API 兼容性** | `get_child_key`, `_key_match_*` 成为私有实现并被删除。若外部项目（非本仓库）直接导入使用，会报 `ImportError`。 | 在发布说明中明确这些函数已删除，推荐使用 `RadixKey.child_key()` / `RadixKey.match()`。若必须保留兼容层，可在 `radix_cache.py` 再提供薄包装函数（内部调用新方法）。 |

| `models/qwen*_moe.py` | 使用 `attn_context_model_parallel_world_size` 代替原来的 `moe_dp_size` 进行 CP‑AllGather；加入对 `attn_cp_size % moe_dp_size == 0` 的断言。 |
| `server_args.py` | 限定 `attn_cp_size != moe_dp_size` 时只能是 `moe_dp_size == 1`，避免不支持的组合。 |
| 测试 | 新增 4‑GPU‑Qwen3‑30B‑CP 的端到端评测，验证功能正确性。 |

**影响范围**  
- **并行调度层**（`parallel_state`、`communicator`）以及 **MoE 相关数据并行**（`dp_attention`）的初始化、销毁、通信路径全部受此改动影响。  
- **模型前向实现**（Qwen2/3 MoE）在预填阶段会走 `MOE_FULL` 路径，涉及额外的 CP All‑Gather 与 Padding。  
- **服务器启动参数** 需要符合新约束：若 `attn_cp_size > moe_dp_size`，只能将 `moe_dp_size` 固定为 1。  

**潜在风险 & 建议**  

1. **Group 生命周期**：`_MOE_DP` 与 `_ATTN_CP` 共享同一 Group，销毁顺序已更正。请确保在所有可能的异常路径（如 `initialize_model_parallel` 早期抛异常）下仍能安全销毁，建议在 `destroy_model_parallel` 前加入 `try/except` 包裹，以防残留 Group 导致后续 `init` 报错。  

2. **All‑Gather Padding**：对 `MOE_FULL` 采用 `torch.nn.functional.pad` 后的 `output` 大小为 `max_tokens * moe_cp_size`。如果后续逻辑仍依据原 token 数进行切片（例如 `dp_scatter`），可能出现多余的零填充导致梯度不匹配。建议在 `dp_scatter` 前使用 `torch.narrow` 去除 padding，或在 `moe_cp_all_gather_into_tensor` 返回前记录真实 token 长度并传递。  

3. **Fusion 关闭**：`is_enable_moe_cp_allgather` 时强制关闭 `should_fuse_mlp_allreduce_with_next_layer`。若未来添加其他 Fusion 优化，需要在 `ScatterMode.MOE_FULL` 上重新评估兼容性。  

4. **断言严苛**：`server_args` 只允许 `moe_dp_size == 1` 当二者不相等，这在多节点部署时可能限制灵活性。若计划支持 `moe_dp_size > 1` 且 `< attn_cp_size`，需在 `parallel_state` 中实现 MoE‑DP 与 CP 的双重分组，而不是简单别名。  

5. **测试覆盖**：新增的端到端测试仅覆盖 `attn_cp_size=2, moe_dp_size=1` 的组合，建议补充以下情形：  
   - `attn_cp_size` 为 4、8 多级 CP 与 `moe_dp_size=1` 的组合。  
   - `moe_dp_size > 1` 且 `attn_cp_size % moe_dp_size == 0`（未来功能）  
   - Decoding 阶段不进行 CP all‑gather，确保 `forward_batch.forward_mode.is_context_parallel_extend()` 分支正确。  

6. **文档更新**：在 README/CLI 参数说明中补充 `--moe-dp-size 1` 与 `--attn-cp-size > 1` 的兼容说明，以及 “MoE‑CP all‑gather” 的实现原理，方便用户快速排查配置错误。  

**总体评价**  
改动实现了需求的关键路径：在 MoE 数据并行为 1 时仍能使用注意力上下文并行，提高预填阶段的吞吐。代码结构相对清晰，新增的 `ScatterMode.MOE_FULL` 与对应的通信函数保持了原有 API 风格。只要对上述风险点做进一步测试和容错处理，提交可视为一次稳妥的功能增强。
