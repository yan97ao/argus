### [diffusion] feat: configure encoder as layerwise-offload by default (#25517)
**SHA**: `eccfd6d` | 🔗 [查看提交](https://github.com/sgl-project/sglang/commit/eccfd6dea79c7d24425fe6a19a4b008cf84ad665)

**🎯 变更类型**：功能增强 / 架构变更  
**⚡ 重要程度**：🔴 高  
**📋 变更摘要**：  
- 引入 **layerwise‑offload 组件分组**（`dit`、`text_encoder`、`image_encoder`、`vae`、`default`），并把 **文本/图像编码器** 在默认情况下设为 **layerwise‑offload**（而不是传统的 CPU offload）。  
- 新增/重构了层级选择、规范化、匹配等工具函数，统一了 **组件选择语义**，并在多处改写了 `ServerArgs`、`LayerwiseOffloadManager` 以及各模型基类的标记字段，以支持新的分组逻辑。  
- 自动调优逻辑同步更新：在 `auto_tuner` 中依据显存、平台等自动启用非 DiT（文本/图像/ VAE）默认的 layerwise‑offload 策略。

### 🎯 影响范围
| 模块/组件 | 影响说明 |
|-----------|----------|
| `sglang/multimodal_gen/runtime/managers/memory_managers/layerwise_offload_*` | 新增分组常量、匹配函数、`normalize` 与 `expand` 逻辑；原有 `DEFAULT` 迁移到 `default` 分组。 |

### ⚠️ 潜在风险
1. **兼容性回退**  
   - 旧版 CLI 使用 `--dit-layerwise-offload`（默认选择 DiT 组件）现在对应 `dit` 分组。如果用户仅依赖 `--dit-layerwise-offload` 而未更新脚本，仍能正常工作，但 **`default` 分组已不再自动包含 DiT**，可能导致意外的层级 offload 行为变化。  
   - 需要在文档和发布说明中明确两者区别，避免用户误解。

2. **显式参数追踪误判**  
   - `_explicit_arg_names` 通过 `from_dict` 与 `from_kwargs` 填充，但若用户在程序运行时自行修改 `ServerArgs` 对象的属性而不更新该集合，自动调优可能错误地覆盖用户意图。建议在公共 API 中提供 `set_arg_explicit(name)` 方法或在属性 setter 中自动同步。

3. **分组冲突**  
   - 当用户同时指定 `dit` 与 `default`（或 `text_encoder`、`image_encoder`、`vae`）时，代码会 **合并** 所有匹配的组件；若其中某些组件不实现 `LayerwiseOffloadableModuleMixin`，会在 `configure_layerwise_offload_modules` 中抛出警告。应确保 UI 给出明确错误信息，防止隐藏的配置冲突。

4. **性能回退**  
   - 对 `prefetch_layer` 的新调用在 **reverse‑order** 场景（如 LTX audio VAE）加入了 `if i not in self._gpu_layers` 防护，若出现 **层次不在 GPU 列表** 的异常分配，仍会触发预取；但若 `self._gpu_layers` 未正确维护，可能导致不必要的 H2D 操作，产生额外开销。需要做额外的单元/集成测试覆盖所有 `LayerwiseOffloadableModuleMixin` 的 `layer_names` 组合。

### [MoE] Unify DeepEPMoE+MoriEPMoE through AITER MoeRunner pre/post-permute (#23760)
**SHA**: `be3c425` | 🔗 [查看提交](https://github.com/sgl-project/sglang/commit/be3c425788db359a8b47f832469e870078254944)

**🎯 变更类型**：架构变更 / 功能增强 / 重构  
**⚡ 重要程度**：🔴高  
**📋 变更摘要**：此次 PR 将 DeepEPMoE 与 MoriEPMoE 统一到 AITER‑MoeRunner 的前置 / 后置 permute 流程中，删除了单独的 `MoriEPMoE` 实现，新增 `AiterRunnerCore`、前置/后置 permute 注册以及专家 mask 统一逻辑。量化层的创建代码同步改为在后端支持 AITER 时自动选择 AITER，新增 `supports_aiter` 判定与若干工具函数，以实现「统一调度‑融合‑后处理」的完整闭环。  

---

### 🎯 影响范围
- **核心 MoE 层**：`sglang/srt/layers/moe/ep_moe/layer.py`、`sglang/srt/layers/moe/token_dispatcher/*.py`、`sglang/srt/layers/moe/moe_runner/*`。  
- **量化配置**：所有涉及 `Fp8`, `W4AFp8`, `QuarkW4A4MxFp4` 等 scheme 的 `create_moe_runner` 入口。  
- **运行时工具**：`sglang/srt/utils.py`（新增 `supports_aiter`），以及 `sglang/srt/layers/moe/utils.py` 中的 AITER 判定。  
- **外部依赖**：`aiter`（AMD GPU Fusion 库）在 `SGLANG_USE_AITER` 环境变量打开且硬件为 HIP 时被激活。  

---

### 🔍 技术洞察

| 维度 | 影响 |
|------|------|
| **架构影响** | <ul><li>引入 **Runner Core → Pre‑Permute → AITER Fusion → Post‑Permute** 三层抽象，解耦了 MoE 调度、激活融合、输出组合。</li><li>删除 `MoriEPMoE` 子类，统一在 `DeepEPMoE.run_moe_core` 中通过 `self.deprecate_flag` 选择执行路径，降低代码维护成本。</li><li>新增 `expert_mask_gpu` 在 `DeepEPP` 与 `MoriEP` 调度器中统一生成，统一了对无效 expert ID 的处理。</li></ul> |
| **性能影响** | <ul><li>**AITER Fusion**：一次 CUDA（或 HIP）kernel 完成 `hidden_states → w13 → w2 → activation → combine`，显著减少 kernel 启动和显存搬迁，特别在 **AMD GPU（HIP）** 上可提升 1.3‑2.0×吞吐。</li><li>在 FP8 / FP4 / W4A4 混合量化场景下，仍保留 **upscale** 路径，仅在必要时执行额外的 BF16 ⇢ FP8 转换，避免不必要的 double‑quant。</li><li>`expert_mask` 让 AITER 内部直接屏蔽非法 token，省去原来在 Python 层的 `topk_ids == -1` 过滤逻辑。</li></ul> |
| **安全考虑** | <ul><li>没有直接的安全风险——改动均为内部计算路径。</li><li>唯一需关注的是 **外部依赖 `aiter`**，若该库出现未受控的输入校验或内存越界，可能导致进程崩溃。建议在生产环境中开启 `SGLANG_USE_AITER` 前做版本锁定与安全审计。</li></ul> |
| **可维护性** | <ul><li>通过 **注册式（register_pre_permute / register_post_permute）** 机制，新增后端只需实现对应的 hook，即可接入现有框架，扩展性提升。</li><li>`supports_aiter` 把后端判断抽象为枚举，避免散布多处 `is_none()` 判定。</li></ul> |
| **兼容性** | <ul><li>默认仍支持原有 `TRITON`、`DEEP_GEMM`、`MARLIN` 等路径，仅在 **`SGLANG_USE_AITER=1 && is_hip`** 且后端为 `NONE/DEEPEP/MOONCAKE/NIXL/MORI` 时切换。</li><li>对未开启 AITER 的环境（CPU、CUDA）行为保持不变。</li></ul> |

### ⚠️ 潜在风险
1. **Mask 维度错误**  
   - `expert_mask_gpu` 在 `DeepEPP` 与 `MoriEP` 中分别构造 `(num_local_experts+1)` 与 `(num_experts)`，若后端实现对 mask 长度要求不一致可能导致 OOB。  
2. **量化路径不匹配**  
   - `AiterRunnerInput.quant_type` 通过 `_resolve_mori_quant_type` 推导，若新加入的量化 scheme（比如新 FP8 变体）未在该函数覆盖，可能回退到 `NONE` 导致性能回退或数值误差。  
3. **环境变量失效**  
   - 依赖 `SGLANG_USE_AITER` 与 `is_hip`，若用户在 CUDA 环境误设 `SGLANG_USE_AITER=1`，会尝试导入 `aiter` 并在 import 失败时报错。  
4. **后置 permute 迁移**  
   - `post_permute_aiter_to_deepep_*` 直接把 `runner_output.hidden_states` 塞入 `CombineInput`，若后续 `CombineInput` 添加新字段（如 `token_offset`）未同步更新，可能出现运行时断言。  
5. **内存峰值**  
   - AITER 融合后 **一次性** 处理全部 `hidden_states`，在极大 batch（如 16k token）时可能出现显存峰值超过原先分段处理的上限。

### fix(unified radix cache w/ hicache): backup ancestor nodes before leaf in write_back eviction (#24585)
**SHA**: `e547f3f` | 🔗 [查看提交](https://github.com/sgl-project/sglang/commit/e547f3f80463686e91d8a63b728707a82c480a7e)

**🎯 变更类型**：Bug修复  
**⚡ 重要程度**：🔴 高  

**📋 变更摘要**  
- 在统一基数缓存（Unified Radix Cache）实现中，针对 `write_back` 写回策略加入了在驱逐叶子节点前进行祖先节点备份的校验逻辑，防止在写回模式下错误地将父节点也标记为已备份。  
- 同时在单元测试中新增 `test_hicache_write_back_leaf_backup`，验证写回策略下叶子节点被正确驱逐并备份，而其父节点不被错误备份。  
- 通过 `sanity_check` 中的 `write_back` 标记，避免在写回模式下触发原本针对 `write_through` 的备份完整性检查。

**🎯 影响范围**  
- `python/sglang/srt/mem_cache/unified_radix_cache.py`：核心缓存驱逐与一致性检查逻辑。  
- `test/registered/unit/mem_cache/test_unified_radix_cache_unittest.py`：新增/调整单元测试。  
- 受影响的模块：统一基数缓存（UnifiedRadixCache）、HybridCache（hi‑cache）写回路径、缓存一致性校验 (`sanity_check`)。

**🔍 技术洞察**  

- **架构影响**  
  - 在缓存层加入 `write_back` 写回策略的专属分支，使得统一基数缓存可以在 **write‑through** 与 **write‑back** 两种模式下共享同一套驱逐路径。  
  - `writing_check(write_back=True)` 在 `evict` 前触发，确保在写回模式下进行必要的写入校验（例如把脏数据写回 host）。  
  - `sanity_check` 中通过 `write_back` 标记绕过对父节点备份状态的校验，防止出现 “父节点已备份但子节点未备份” 的错误报告，保持层级一致性。

- **性能影响**  
  - 对 `write_back` 场景只在 **驱逐** 时进行一次额外的写回检查，开销与 `write_through` 基本相同，且仅在 `write_back` 启用时生效。  
  - 通过避免不必要的父节点备份检查，**降低**了 `sanity_check` 的计算量，尤其在大规模树结构下可提升检查速度。  
  - 追加的单元测试对性能影响可忽略（仅在 CI 环境运行）。

- **安全考虑**  
  - 该改动不涉及网络、加密或权限检查，仅是缓存一致性与数据持久化的内部实现。  
  - 正确的写回流程防止了潜在的数据丢失风险——在 `write_back` 模式下，如果叶子节点被驱逐而未写回，会导致后续查询得到 stale 数据。此修复确保数据在写回前已安全落盘（或落到 host 端）。

**⚠️ 潜在风险**  
1. **回归风险**：`writing_check` 在 `evict` 前新增调用，若该方法内部实现出现异常（如 IO 失败），可能导致驱逐路径阻塞。需要确保该函数在异常情况下能够安全回退或记录错误。  
2. **配置兼容性**：如果在运行时 `cache_controller` 为 `None` 或 `write_policy` 非法值，代码仍会正常跳过检查，但外部调用者可能误以为写回已生效。建议在 `CacheController` 初始化阶段添加 stricter 参数校验。  
3. **测试覆盖不足**：当前只覆盖了叶子节点的写回场景，未覆盖多层嵌套父节点同时被驱逐的情况，可能隐藏层级备份错误。

**💡 关注建议**  
- **代码审查**：确认 `writing_check` 的实现对异常处理足够稳健，最好在调用前后加上日志/计时，以便排查写回延迟。  
- **监控与告警**：在生产环境中为 `write_back` 模式添加 “写回失败” 的监控指标，及时捕获 IO/磁盘异常。  
- **扩展测试**：补充以下情景的单元测试  
  - 父节点在 `write_back` 模式下被驱逐后是否正确写回。  
  - 多并发驱逐操作下的写回一致性（锁/并发控制）。  
- **文档更新**：在项目文档中标明 `write_back` 与 `write_through` 两种策略在一致性检查、驱逐行为上的差异，帮助用户正确配置。

| 维度 | 影响分析 |
|------|----------|
