### VLM: add Conv2dLayer/Conv3dLayer to fix PyTorch 2.9.1 CuDNN Conv3d (#20282)
**SHA**: `1c456a0` | 🔗 [查看提交](https://github.com/sgl-project/sglang/commit/1c456a0af540b5fd092a90d3af7627be973416a5)

**🎯 变更类型**：功能增强 / 重构 / 性能优化 / 架构变更  
**⚡ 重要程度**：🔴 高  
**📋 变更摘要**：  
1. 新增 `Conv2dLayer` 与 `Conv3dLayer` 实现，在满足特定条件时把 `nn.Conv*` 替换为 **unfold + linear**（矩阵乘法）实现，以规避 PyTorch 2.9.1 + CuDNN < 9.15 中 `Conv3d` 的已知性能 bug。  
2. 项目中所有视觉模型的 patch‑embedding、投影层统一改用上述自研卷积层，去除原有的 CuDNN 兼容性检测代码。  
3. 补充单元测试覆盖函数路径、参数校验及与原生 `nn.Conv*` 的数值等价性。  

---

### 🎯 影响范围
- **核心层**：`sglang/srt/layers/conv.py`（新实现）  
- **视觉模型**：`clip.py、dots_vlm_vit.py、glm4v.py、idefics2.py、internvl.py、kimi_*、midashenglm.py、paddleocr_vl.py、pixtral.py、qwen2_vl.py、qwen3_vl.py、siglip.py、step3_vl*.py`（全部改为使用 `Conv2dLayer` / `Conv3dLayer`）  
- **服务器启动参数**：`sglang/srt/server_args.py` 中移除对 PyTorch 2.9.1 与 CuDNN 兼容性的强制检查。  
- **单元测试**：新增 `test/unit/test_conv_layer.py`（覆盖 300+ 行），确保数值一致性和异常校验。  

---

## 🔍 技术洞察

| 维度 | 影响分析 |
|------|----------|
| **架构影响** | - 引入 **`MultiPlatformOp`** 基类，统一 `forward_native` 与 `forward_cuda` 两条路径，使得在不同设备上可自由切换实现。<br>- 通过 **参数 `disable_linear`** 可在运行时关闭线性优化，兼容已有代码路径，保持向后兼容性。<br>- 所有视觉模型的卷积层被统一抽象，后续若需再做跨平台优化（例如 Triton、FlashAttention）只需在 `Conv*Layer` 中适配即可，提升可维护性。 |
| **性能影响** | - 当满足 **`kernel_size == stride && padding==0 && dilation==1 && groups==1`** 时，卷积被拆解为 **`unfold → linear`**，在 CUDA 上可避免 CuDNN 的低效路径，理论上 **2–5×** 加速（已在 PR 中声明）。<br>- 对 **`padding_mode != 'zeros'`** 的情况仍回退到原生 `F.conv*`，保证功能完整性。<br>- 通过 `enable_linear` 默认 **关闭**（`disable_linear=True`），避免在不满足条件时误用线性路径，防止性能倒退。 |
| **安全考虑** | - 变更仅涉及算子实现与参数检查，无网络、文件或系统调用，安全风险极低。<br>- 参数校验（通道可整除、padding 字符串、padding_mode）已在 `_validate_conv_args` 中强化，防止异常输入导致未定义行为或内存泄漏。 |
| **可维护性** | - 将卷积的参数处理（tuplify、padding 计算）抽离为私有工具函数，代码结构更清晰。<br>- 单元测试覆盖 **参数合法性、数值等价、各分支路径**，为后续改动提供强保障。<br>- 移除 `server_args` 中的硬性检查，避免因环境差异导致启动失败；如果用户仍需要提醒，可自行通过环境变量 `SGLANG_DISABLE_CUDNN_CHECK` 控制。 |
| **兼容性** | - 对外保持 **Drop‑in** 替代 `nn.Conv2d/3d`，模型配置文件无需改动。<br>- `Conv2dLayer` 默认 **不启用** 线性优化，兼容之前的行为；只有在显式传入 `disable_linear=False` 且满足条件时才启用。<br>- 对 **padding='same'** 仍支持（在非 stride 场景下），但会强制禁用线性路径，保持行为一致。 |
| **资源占用** | - `unfold` 操作会产生 **临时展开张量**，在极大特征图上可能临时占用额外显存（`N·C·K·K`），但在典型的 patch‑embedding 场景（kernel 与 stride 相等，特征图尺寸相对较小）影响可接受。 |

---

## ⚠️ 潜在风险

1. **显存峰值**：`unfold` 会在内存中产生展平的补丁矩阵，若模型使用极大分辨率或大 batch，可能触发 OOM。  
2. **线性路径数值差异**：在极端的浮点精度（`torch.float16`/`bfloat16`）下，`unfold + linear` 与原生卷积可能出现细微数值差别，需在量化/混合精度场景中仔细验证。  
3. **非标准 padding_mode**：`padding_mode` 为 `'reflect'、'replicate'、'circular'` 时仍走原生卷积路径，性能提升不可用，可能让用户误以为已优化。  
4. **多平台一致性**：`forward_native` 与 `forward_cuda` 两条实现保持完全一致性是关键，若后续在 CPU 上实现了不同的优化，需要同步测试。  
5. **旧代码依赖**：移除 `check_torch_2_9_1_cudnn_compatibility` 可能导致老用户在未升级 CuDNN 时仍遇到性能问题，只是没有显式警告。

## 💡 关注建议

| 目标 | 建议 |
|------|------|
| **生产环境部署** | - 对 **大批量**、**高分辨率** 的推理任务监控显存使用，必要时手动关闭 `enable_linear`（`disable_linear=False`）或改为 `nn.Conv*`。 |
| **模型微调 / 量化** | - 在使用 **fp16 / bf16** 时，对比 `forward_native` 与原生卷积的输出误差，确保误差在可接受范围；如有显著差异，可通过 `disable_linear=True` 回退。 |
| **代码维护** | - 将 `Conv*Layer` 的 **参数校验** 与 **padding 计算** 继续抽象成工具函数，以免在后续支持更多特性（如可变卷积、动态 padding）时重复实现。 |
| **单元测试** | - 增加针对 **大 batch / 大特征图** 的压力测试，捕获潜在 OOM 场景。<br>- 添加 `torch.cuda.amp.autocast` 环境下的数值对齐测试，保证混合精度兼容。 |
| **文档/使用指南** | - 在模型文档中标注 **何时会自动启用 linear 优化**（kernel==stride 等），并提供示例说明如何手动关闭。<br>- 说明 `SGLANG_DISABLE_CUDNN_CHECK` 已被移除，若用户仍需兼容老环境，可自行在启动脚本中加入检查逻辑。 |
| **回滚机制** | - 若在特定硬件（如某些旧版 GPU）上出现异常，可通过 `disable_linear=True` 快速回退到原生卷积，实现零代码回滚。 |
| **性能基准** | - 在 CI 中加入 **CUDA** 与 **CPU** 两套基准（使用 `torch.utils.benchmark`），对比 `Conv*Layer` 与 `nn.Conv*` 的吞吐率，确保优化实际生效。 |

### diffusion: fix helios accuracy issue (#20036)
**SHA**: `a6ecf05` | 🔗 [查看提交](https://github.com/sgl-project/sglang/commit/a6ecf050bed7eafec7005a007951472e9f12a956)

**🎯 变更类型**：Bug修复 / 功能增强 / 性能优化  
**⚡ 重要程度**：🔴高  

**📋 变更摘要**  
1. 将 Helios 模型的默认最大序列长度从 226 调整至 512，使其与 Diffusers 实现保持一致，根除因截断导致的生成精度下降。  
2. 为 Helios pipelines 新增序列并行（Sequence‑Parallel）支持、Chunk‑by‑Chunk VAE 解码以及细粒度的性能日志（StageProfiler），并在采样参数中自动开启 `enable_sequence_shard`。  
3. 对 denoising 主循环进行结构化重写，统一 CFG 处理、零初始化以及全局步号计数，以提升多卡分布式推理的正确性和可观测性。  

**🎯 影响范围**  
- `helios.py`（模型前向、注意力、历史尺度、SP 相关逻辑）  
- `helios_pipeline.py`、`helios_decoding.py`（新增解码 stage）  
- `sampling_params.py`（采样配置自动开启 SP）  
- `scheduler` 别名配置  
- 性能基准 JSON 与测试用例（新增 Helios T2V 基准）  

**🔍 技术洞察**  

- **架构影响**  
  - 引入 **Sequence Parallelism**：在 `forward` 入口通过 `get_sp_world_size`、`get_sp_group`、`sequence_model_parallel_all_gather` 实现 token‑level 分片与跨卡聚合，改变了原有单卡序列全局视图。  
  - 新增 **HeliosDecodingStage**：在 VAE 解码阶段按 latent chunk 逐块解码，避免 Wan VAE 因因果卷积产生的时间缝隙，兼容 Diffusers 行为。  
  - `skip_sequence_parallel=True` 参数在注意力层中关闭跨序列并行，保证历史尺度计算仍基于完整上下文。  
  - 通过 `get_forward_context().forward_batch.enable_sequence_shard` 控制 SP 开关，保持向后兼容性。

- **性能影响**  
  - **正面**：SP 能将序列长度均摊到多个 GPU 上，理论上在 512 长度的 Prompt 上可降低显存占用 1/`sp_size`，并提升并行吞吐。  
  - **负面**：每个 SP 步骤后执行 `sequence_model_parallel_all_gather`，在 2‑4 卡配置下额外带来一次跨卡通信；对帧数不整除 `sp_size` 的情况会产生 padding，导致少量无效计算。  
  - `StageProfiler` 在每个 denoise 步加入计时与指标写入，略微增加 CPU 开销，但对定位瓶颈价值明显。  
  - 按 Chunk 解码会把 VAE 的一次性全序列解码拆成 `num_chunks` 次，解码时间累计略增，但能显著提升视觉一致性（避免时间段接缝）。

- **安全考虑**  
  - 变更仅涉及模型推理路径和分布式通信，无新增网络 I/O、文件写入或外部依赖，未引入明显安全风险。  
  - 使用 `torch.distributed` 的 All‑Gather，若部署环境未正确初始化 `sp_group`，可能导致进程挂起或错误泄露内部状态，但不会触发攻击面。

**⚠️ 潜在风险**  

| 风险点 | 可能后果 | 缓解建议 |
|--------|----------|----------|
| 序列长度 512 + SP 分片导致显存 OOM（尤其在 `sp_size` > 1 时仍需保留完整历史） | 推理进程崩溃 | 在部署前通过 `nvidia-smi` 评估每卡显存需求；提供 `max_sequence_length` 手动调节或 `enable_sequence_shard=False` 选项。 |
| `seq_len % sp_size != 0` 时的 padding 可能在极端长序列下产生多余的 token，影响生成质量 | 生成输出轻微偏移或额外噪声 | 在 `forward` 中加入 `assert seq_len % sp_size == 0` 或在 padding 后使用 mask 剔除无效 token。 |
| `get_sp_group().rank_in_group` 获取错误的 rank（未正确初始化） | All‑Gather 死锁或维度不匹配错误 | 在启动脚本中确保 `torch.distributed.init_process_group` 包含 `group_name="sp"`；添加启动检查。 |
| 新增 `HeliosDecodingStage` 对 VAE 状态缓存不兼容（如使用混合精度或自定义 VAE） | 解码阶段报错或显存泄漏 | 在 VAE 初始化时检查 `self.cache` 是否被清理；提供 `skip_chunk_decoding` 回退选项。 |
| `StageProfiler` 写入日志路径/权限错误导致异常 | 推理异常中止 | 捕获文件 I/O 异常，默认回退为无日志模式。 |
| 采样参数 `enable_sequence_shard` 自动对 “helios” 生效，可能误打开其他不支持 SP 的 pipeline | 非 Helios pipeline 产生错误 | 在 `sampling_params.py` 中加入 `pipeline_name_lower.startswith("helios")` 的更严格判断。 |

**💡 关注建议**  

1. **部署前验证**：在多卡环境下跑一次 `python -m sglang.test.run_helios_sp`（自建脚本），检查 `sp_size`、`seq_shard_pad`、All‑Gather 的张量形状。  
2. **显存监控**：开启 `SGLANG_SHOW_MEMORY` 环境变量，确保分片后每卡显存留有余量；若仍 OOM，考虑调小 `max_sequence_length` 或关闭 SP。  
3. **性能调优**：  
   - 对低带宽互连（如 10GbE）集群，可先关闭 SP，观察单卡时延；在高速 NVLink 环境下开启以获得收益。  
   - 通过 `SGLANG_PROFILER=1` 查看 `StageProfiler` 输出，定位最耗时的 denoise 步。  
4. **兼容性回退**：保留 `skip_sequence_parallel=True`（默认）并在 `HeliosConfig` 中提供 `use_sequence_parallel` 开关，便于在出现异常时快速回退到原实现。  
5. **测试覆盖**：在 CI 中加入 **Helios SP** 与 **Chunk Decoding** 两套对比基准，确保新旧路径在相同随机种子下输出差异在可接受范围（< 0.5% BLEU/CLIPScore）。  
6. **文档更新**：在 `README` 与 `docs/pipelines/helios.md` 中注明：  
   - `max_sequence_length` 已提升至 512，显存需求约增加 2×。  
   - `enable_sequence_shard` 默认在 Helios 启用，可通过 `--no-seq-shard` 关闭。  
   - 解释 `HeliosDecodingStage` 的必要性及其对时间一致性的提升。  

通过上述措施即可在保证生成精度的前提下安全、高效地使用 Helios 的新特性。

- CI 注册入口 `sglang.test.ci.ci_register.register_cuda_ci`（新增两条注册记录）  

**💡 关注建议**  

1. **资源依赖**：两套脚本默认启动多进程服务器并要求 4/8 张 GPU、数十个 CPU 线程以及对应的模型权重。确保 CI 环境具备这些硬件及网络带宽，防止因资源不足导致的 timeout。  
2. **超时配置**：`register_cuda_ci(est_time=300, …)` 与 `DEFAULT_TIMEOUT_FOR_SERVER_LAUNCH` 设为 300 s，实际模型加载可能更慢（尤其在多线程加载时）。建议在 CI 机器上跑一次基准，必要时提升 `est_time` 与 `timeout`。  
3. **进程清理**：`tearDownClass` 通过 `kill_process_tree` 结束服务器进程，已覆盖。但若测试中途异常（如 `run_eval` 抛异常），进程可能残留。可在 `tearDownClass` 前加入 `if cls.process.poll() is None:` 检查。

5. **代码复用**：两个文件的结构几乎相同，仅模型名称、TP‑size、loader 线程数不同。考虑抽象出公共测试基类或函数，以降低维护成本，未来新增模型只需更改常量。  
6. **评测阈值**：断言 `score >= 0.96` 较为严格，若模型在不同硬件/驱动上出现轻微波动，CI 可能频繁失效。可考虑使用 `assertGreaterEqual(..., 0.95)` 并在注释中说明容忍范围。  

总体而言，此次提交为新模型增加了完整的端到端 CI 验证，提升了对最新 NVIDIA 大模型的支持。只要确保 CI 环境满足硬件需求并适当放宽超时/阈值，即可平稳运行。
