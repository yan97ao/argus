### Add a sliding-window-attention divergence reporter for the KV-canary (#26820)
**SHA**: `7dd19ae` | 🔗 [查看提交](https://github.com/sgl-project/sglang/commit/7dd19ae3d8ba2bd555cc12eb68ece24bf0332318)

**🎯 变更类型**：功能增强 / 监控/可观测性  
**⚡ 重要程度**：🔴高  
**📋 变更摘要**：  
- 新增环境变量 `SGLANG_KV_CANARY_SWA_DIVERGENCE_STATS_INTERVAL`，用于控制滑动窗口注意力（SWA）池的分歧统计上报频率。  
- 在 KV‑Canary 框架中引入 `SwaDivergenceReporter`，负责在指定间隔收集并在日志中输出 **Full** 与 **SWA** KV‑pool 的验证计数、窗口外 token 计数以及 index 迁移（divergence）计数。  
- 将该 reporter 串联进 `CanaryManager`、`SingleForwardManager` 以及数据缓冲区，实现“观察‑记录‑回报”全链路。  
- 补充大量单元/端到端测试，验证日志格式、统计准确性以及在不同环境下的可选性。  

**🎯 影响范围**：  
- `python/sglang/srt/kv_canary` 子模块（包括 `environ.py`, `api.py`, `runner/canary_manager.py`, `single_forward_manager/*`）  
- `python/sglang/srt/mem_cache/swa_memory_pool.py`（通过类型检查）  
- 测试目录 `sglang/test/kv_canary/*` 与注册脚本  
- 运行时日志系统（新增 `kv_canary_swa_divergence=` 前缀）  

**🔍 技术洞察**  

| 维度 | 影响分析 |
|------|----------|
| **架构影响** | - **可观测性层**：新增 `SwaDivergenceReporter` 属于监控/诊断组件，采用**可选**（`env` 控制）方式接入，避免对既有运行路径产生硬依赖。<br>- **模块耦合**：`CanaryManager` 现在持有 `SwaDivergenceReporter` 实例；`SingleForwardManager` 通过 `post_ops_inside_graph` 将 **verify** 结果写入 reporter。耦合点清晰且均通过构造函数注入，保持 DI（Dependency Injection）原则。<br>- **数据流**：新增 `swa_verify_total_count` 张量在 `PostOpsInsideGraphOutputBuffer` 中传递，确保在图外回收阶段统计数据。 |
| **性能影响** | - **统计开销**：每次 `observe_after_invoke_plan` 将 `verify_num_valid` 原位累加到 device Tensor，开销极小（一次整数加法）。<br>- **间歇性 Host‑Device 同步**：`SwaDivergenceReporter.step` 在 `interval` 步时会在专用 `d2h_stream` 执行一次设备‑到‑主机拷贝并触发日志写入。对于生产环境默认 `interval=0`（关闭），对延迟和吞吐几乎无影响；开启后若间隔较小（如 10）会产生每 10 次前向的额外同步，预计 <1 ms（受 GPU‑CPU 带宽限制）。<br>- **内存占用**：额外的 `verify_total_count_device`（2 · int32）+ 可能的 `swa_verify_total_count`（同 shape）在 GPU 上占用可忽略的几百字节。 |
| **安全考虑** | - 仅收集内部计数并写入日志，不涉及外部输入或网络交互，未引入新的安全漏洞。<br>- 环境变量可被外部控制，若在多租户环境中误开启，可能泄露内部 KV‑pool 状态（对侧信道攻击极低风险）。建议在安全敏感部署中保持 `interval=0`。 |
| **可维护性** | - 代码遵循现有风格：数据类 `SwaDivergenceLog` 使用 `@dataclass(frozen=True, slots=True)`，易于序列化/解析。<br>- 日志格式统一为 `kv_canary_swa_divergence=` 前缀 + JSON，便于机器解析和 CI 检测。<br>- 新增的测试覆盖了日志解析、统计逻辑、环境开关以及与 `CanaryManager` 的 wiring，降低回归风险。<br>- 通过类型注解和 `TYPE_CHECKING` 的懒加载，保持对未使用 SWA allocator 场景的兼容。 |

**⚠️ 潜在风险**  
1. **异常同步导致阻塞**：`DelayedDeviceHostHandler` 在极端 GPU 负载下可能出现 `d2h` 拷贝排队，导致 `step` 被阻塞。若 `interval` 设得过小，可能累计显著的同步延迟。  
2. **错误的 `swa_allocator` 类型**：`install_canary` 中的 `isinstance(..., SWATokenToKVPoolAllocator)` 只在 `model_runner.token_to_kv_pool_allocator` 为该类时才传递，若后续实现改为子类或包装器，可能导致报告失效。  
3. **日志解析的向后兼容**：日志前缀硬编码为 `kv_canary_swa_divergence=`；若后续改动此前缀，现有 CI 与监控脚本将失效。  
4. **统计计数溢出**：`verify_total_count_device` 使用 `int32`，在极端超长会话（>2 B token）可能溢出。虽然目前不现实，但在长时间运行的服务上应留意。  
5. **资源泄漏**：`SwaDivergenceReporter` 创建的 `DelayedDeviceHostHandler` 没有显式释放路径；在进程结束前仍有未完成的 async 复制可能触发 `RuntimeError`。  

**💡 关注建议**  
- **配置建议**：在生产环境默认保持 `SGLANG_KV_CANARY_SWA_DIVERGENCE_STATS_INTERVAL=0`；仅在调试或性能基准时开启，且间隔不宜低于 20，以降低同步开销。  
- **监控与告警**：将 `kv_canary_swa_divergence=` 日志接入统一日志平台，实时监控 `verify_swa < verify_full` 与 `swa_full_idx_divergence` 的趋势，异常下降可能指示 SWA pool未产生预期复用。  
- **测试扩展**：增加对 **大规模 token 序列**（>2^31）下的计数溢出测试；以及在高并发多 GPU 场景下 `d2h_stream` 的排队行为。  
- **代码审计**：如果未来 `SWATokenToKVPoolAllocator` 被抽象为接口或多实现，建议在 `install_canary` 中改为 **鸭子类型** 检测（`hasattr(..., "full_to_swa_index_mapping")`），确保 reporter 仍能接收。  
- **日志向后兼容**：在 `SwaDivergenceLog.format` 中加入版本字段或可变前缀，以防后续格式演进破坏现有解析。  
- **资源清理**：在 `CanaryManager` 的 `mark_init_finished` 或进程退出钩子中显式调用 `self._swa_divergence_report._handler.flush()`（若实现了）以确保所有 pending copy 完成。  

---  

> 总结：本次改动在不破坏原有功能的前提下，为 KV‑Canary 引入了 **SWA 分歧可观测性**，帮助开发者验证 Sliding‑Window‑Attention 的复用行为是否按预期工作。对系统架构、性能与安全的影响均在可接受范围，唯一需要关注的是开启统计后偶尔的 host‑device 同步开销以及对 `SWATokenToKVPoolAllocator` 接口的耦合度。建议在生产环境保守使用环境变量控制开启，并通过日志平台监控关键指标。

### Add the KV-canary perturb modes and PD-disaggregation e2e tests (#26819)
**SHA**: `ae9db7f` | 🔗 [查看提交](https://github.com/sgl-project/sglang/commit/ae9db7ff4bdc4c308d7a5dc70d9363fad2844312)

**🎯 变更类型**：功能增强 / 重构  
**⚡ 重要程度**：🔴高  

**📋 变更摘要**  
本次提交为 SGLang KV‑Canary 框架引入四类 *perturb*（故障注入）模式：  
1. **req‑to‑token** – 随机改写活跃请求的 `req_to_token` 映射。  
2. **real_kv_used** – 在正在使用的 KV 槽位上翻转真实 KV 数据的第一个字节。  
3. **real_kv_unused_cache** – 在 radix‑cache 中未被前缀使用的“孤儿”槽位上翻转真实 KV。  
4. **real_kv_post_forward** – 在 TAIL 核心完成 Canary 哈希写入后，对对应真实 KV 槽位做同样翻转。  

为支撑这些模式，新增环境变量、配置结构、`PerturbManager` 接口以及若干实现文件。与此同时，配套了 PD‑Disaggregation（Prefill‑Decode 分离）场景的端到端（e2e）测试，覆盖无扰动基线、各种 target‑group（FULL / SWA）以及 **raise** / **log** 模式下的异常检验。

---

### 🎯 影响范围
- **核心运行时**：`sglang/srt/kv_canary/*`（配置、管理器、各 perturb 实现、slot_picker）  
- **服务器启动入口**：`python/sglang/srt/environ.py` 新增 4 条环境变量。  
- **Canary 管理器**：`sglang/srt/kv_canary/runner/canary_manager.py` 需注入 `req_to_token_pool`，并在 `post_ops_outside_graph` 里调用 `perturb_post_forward`。  
- **测试套件**：`test/registered/kv_canary/*`、`test/registered/mock_model/*` 新增大量 e2e 与单元测试。  
- **公共 API**：`PerturbConfig` 结构体字段从 2 → 6，外部调用 `PerturbConfig.from_env()` 的代码需兼容新字段（默认 0.0，向后安全）。

---

### 🔍 技术洞察

| 维度 | 影响分析 |
|------|----------|
| **架构影响** | - 将 *perturb* 功能抽象为独立模块，保持原有 Canary 验证路径不变。<br>- `PerturbManager` 现在持有 `req_to_token_pool`，实现 **req‑to‑token** 与 **real_kv_used** 的关联；对已有 `CanaryManager` 的构造函数做了向后兼容的微调。<br>- 通过 `WarmupGate` 与概率阈值实现 “warm‑up 阶段不触发”，保证生产环境默认 (概率 0) 时零开销。 |
| **性能影响** | - 仅在 env 中显式打开相应概率时才执行，且每次调用的工作量仅限于：<br>  • 随机数生成 (`torch.rand` / `torch.randint`) <br>  • 简单张量切片与一次写入 <br>  • 日志记录 <br>在概率为 0 的默认配置下，对前向路径几乎没有 CPU/GPU 开销。<br>- 新增的 `slot_picker` 与 `collect_active_slots` 只在测试/扰动时运行，数据拷贝到 host 的成本在 8‑16KB 量级，微乎其微。 |
| **安全考虑** | - 所有扰动均在内部生成的随机数驱动下执行，未引入外部输入或文件系统交互。<br>- 仅在 **CanaryMode.RAISE** 下会抛出异常，行为已在已有 Canary 错误处理框架内封装，未影响服务的安全边界。 |
| **可维护性** | - 新增 5 组文件（`real_kv_*`, `req_to_token`, `slot_picker`, `config` 更新），代码结构清晰，遵循“单一职责”。<br>- 单元测试覆盖率显著提升（包括 warm‑up、概率为 0 的无扰动分支），降低回归风险。<br>- 通过 `EnvFloat/EnvInt/EnvBool` 统一环境变量入口，后续若需添加更多扰动点，只需在 `Envs` 中声明即可。 |
| **可靠性** | - 通过 `WarmupGate` 防止在系统启动阶段触发扰动，避免因为缓存未就绪导致误报。<br>- `pick_out_cache_loc_slot` 与 `collect_active_slots` 对空指针、异常维度有完整防御，日志会提示“skipped because …”。<br>- `real_kv_unused_cache` 在 sweep‑interval 为 0 时直接返回，避免在不启用 sweep 时产生额外路径。 |

### ⚠️ 潜在风险
1. **构造函数签名不兼容**：`CanaryManager`（已在 `canary_manager.py`）新增 `req_to_token_pool` 参数，若外部仍使用旧签名（如自定义测试或二次封装），会抛出 `TypeError`。  
2. **环境变量误配置**：在生产部署中误将 `SGLANG_KV_CANARY_PERTURB_*_PROB` 设为非 0（如 0.1），会引入随机写入真实 KV，导致不可预期的生成错误。建议在 CI/生产模板中显式置 0。  
3. **日志噪声**：每次扰动都会写 INFO 级别日志，开启高概率时日志量可能激增，影响日志存储与分析系统。可通过调低日志等级或在 CI 中收敛。  
4. **多 GPU/分布式环境**：`real_kv_post_forward` 假设 TAIL 与 Canary 写在同一 CUDA 流上；在自定义流或跨 GPU 场景（如显式设置 `torch.cuda.Stream`）可能破坏“先写后翻转”的顺序，导致验证不一致。  
5. **SWA‑Pool 索引映射**：`real_kv_unused_cache` 与 `real_kv_used` 对 SWA pool 依赖 `swa_index_lut`，若外部在创建 SWA pool 时未提供正确的 LUT，可能导致 *slot translation* 失效，产生 “slot not mapped” 的噪声日志。

| **代码兼容** | - 为 `CanaryManager.__init__` 增加向后兼容包装（如 `*args, **kwargs` 并检测是否缺少 `req_to_token_pool`），避免第三方插件因签名变化报错。 |

### Add token-id verification to the KV-canary (#26818)
**SHA**: `6be4b32` | 🔗 [查看提交](https://github.com/sgl-project/sglang/commit/6be4b32d8de67c1070c6f9ef460c5ddfe5c82c71)

**🎯 变更类型**：功能增强、架构变更、性能优化（可选）  
**⚡ 重要程度**：🔴高  

**📋 变更摘要**

### ⚠️ 潜在风险

1. **显存冲突 / OOM**  
   - Sweep 在极大 radix‑tree（如 1M KV）时会一次性生成 `VerifyPlan`，显存需求可能与常规 per‑forward KV‑canary 同时出现，导致显存不足。  
2. **CPU‑GPU 同步延迟**  
   - `maybe_run_sweep` 在 `post_ops_outside_graph` 执行，若 sweep 触发的 kernel 启动时间较长，可能阻塞下一轮 forward batch 的准备，导致吞吐下降。  
3. **Radix‑Cache 变更竞争**  
   - Sweep 读取 radix‑tree **非锁定**（默认 `unlocked_only=False`），在多线程/多 GPU 环境中若 radix‑cache 正在被其他线程修改（如 eviction），可能出现不一致的 verify 数据。  
4. **错误的 Tag 过滤**  
   - `launch_endpoints_per_forward` 通过 `and not _is_sweep_tag` 过滤，若忘记在未来的新 kernel‑launch 场景加入相同过滤，可能导致 sweep kernel 被误当作 per‑forward 路径执行，引发 `NotImplementedError`。  
5. **配置误用**  
   - `sweep_interval > 0` 且 `kv_canary == none` 会在 `check_server_args` 抛异常；但如果用户在运行时动态修改 `CanaryConfig`（不经 argparse），可能绕过校验，引起运行时无效 sweep。  
6. **SWA 转换错误**  
   - `build_verify_plan_radix_sweep` 使用 `_swa_translate`，若 LUT 包含负数或错误映射，可能把合法 slot 错误翻译为 `0`（padding），导致遗漏检测。

总体来看，此次重构提升了代码组织，避免了在生成路径中无意义的字段处理，但需注意跨路径的状态清理和兼容性，适当补充测试与错误处理后即可合入。
