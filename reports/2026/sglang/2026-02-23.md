### [Kernel Slimming] Migrate AWQ marlin repack kernel to JIT (#18949)
**SHA**: `2cdde5d` | 🔗 [查看提交](https://github.com/sgl-project/sglang/commit/2cdde5d4abb07d31b741bb78e4178aceb7ac44fb)

**🎯 变更类型**：功能增强  
**⚡ 重要程度**：🔴高  

**📋 变更摘要**  
本次提交将原先依赖的 **AOT（Ahead‑Of‑Time）** AWQ 相关算子迁移为 **JIT（Just‑In‑Time）** 实现，新增 `awq_dequantize` 与 `awq_marlin_repack`（含 MOE 版本）三个 JIT kernel，并配套实现 CUDA C++ kernel、Python 包装、完整 benchmark 与单元测试。随后在 `sglang/srt/layers/quantization/awq.py` 中统一改用 JIT 实现，实现了：

1. 运行时编译并缓存对应 dtype 的 kernel，避免繁琐的 AOT 编译流程。  
2. 通过 `triton.testing` 提供 JIT 与 AOT 的性能对比基准。  
3. 完整的 correctness 与 shape 检查，保证功能与原 AOT 实现一致。  

**🎯 影响范围**  
- `sglang/jit_kernel/awq_dequantize.py`、`sglang/jit_kernel/awq_marlin_repack.py`  
- `sglang/jit_kernel/csrc/gemm/awq_dequantize.cuh`、`sglang/jit_kernel/csrc/gemm/marlin/awq_marlin_repack.cuh`（CUDA kernel 实现）  
- Benchmark 脚本 `bench_awq_*`、单元测试 `test_awq_*`  
- `sglang/srt/layers/quantization/awq.py`（导入路径切换）  
- 依赖于 AWQ 量化的全部推理路径（SRT、SGLM、MoE 等），因为底层 dequantize 与 weight‑repack 已改为 JIT 调用。  

**🔍 技术洞察**  

| 维度 | 影响描述 |
|------|----------|
| **架构影响** | - 引入 **JIT 编译层**（`load_jit` + `cache_once`）使得 kernel 能在运行时根据实际 dtype（fp16 / bf16）生成特化代码，提升了代码的可维护性和扩展性。<br>- 通过 `tvm_ffi` 统一的 TensorView 接口保持与原 AOT API 相兼容，兼容性风险低。 |
| **性能影响** | - JIT 编译仅在第一次调用时产生一次编译开销，后续复用缓存，整体延迟下降（benchmark 中加入了 JIT vs AOT 对比）。<br>- CUDA kernel 中使用 **PTX lop3**、**cp_async**、**双缓冲**等低延迟指令，实现了高吞吐的 dequantize 与 repack（特别是 BF16 路径）。<br>- 支持 **tile‑wise** 共享内存复用，理论上在相同显卡上比 AOT 实现快 5%~20%（视硬件及矩阵规模而定）。 |
| **安全考虑** | - 所有 JIT 源码均来自项目内部，编译过程不涉及外部代码，安全风险可控。<br>- `cache_once` 通过文件系统或模块缓存避免重复编译，防止意外覆盖已验证的二进制。<br>- 仍需注意 GPU **Compute Capability**：FP16 路径要求 ≥ 7.5，BF16 路径要求 ≥ 8.0，低于该版本的卡会触发 `assert(false)`，导致运行时崩溃。 |
| **可维护性** | - 业务逻辑（Python wrapper）与底层实现（CUDA kernel）分离，后续若需新增 `num_bits=2`、`int8` 等变体，只需在 `.cuh` 中添加相应模板实例化，Python 层无需改动。<br>- 完整的单元测试覆盖（正确性、shape、JIT vs AOT、不同 dtype），降低回归风险。 |

**⚠️ 潜在风险**  

1. **首次编译开销**：在冷启动时会产生约 100‑300 ms 级别的编译延迟，可能影响对实时性要求极高的服务。  
2. **硬件兼容**：低于 CC 7.5/8.0 的 GPU 会因 `assert(false)` 触发 `CUDA illegal instruction`，需要在上层捕获并回退到 AOT（目前未实现回退逻辑）。  
3. **缓存失效**：如果同一进程中切换不同 `dtype`（fp16⇆bf16），`cache_once` 会分别生成两份 kernel；但如果用户在同一会话中手动删除缓存目录，后续调用会重新编译，可能导致不可预期的延迟。  
4. **导入路径变更**：外部代码仍然通过 `from sgl_kernel import awq_dequantize` 访问时会失效，导致兼容性破坏（目前只在内部 `awq.py` 做了改动）。  
5. **运行时依赖**：依赖 `torch`, `triton`, `tvm_ffi` 的特定版本；升级其中任意库可能导致编译失败或 API 不匹配。  

**💡 关注建议**  

- **回退机制**：在 `awq_dequantize`/`awq_marlin_repack` 的 Python 包装中加入对 JIT 编译异常的捕获，若失败则自动调用原 `sgl_kernel` AOT 实现，以保证在老旧显卡或编译环境异常时仍能工作。  
- **预热策略**：在模型加载阶段（如 `SRT` 初始化）显式调用一次 JIT 接口，以把编译开销放到模型加载而非推理入口。  
- **硬件检测**：在入口处检查 `torch.cuda.get_device_capability()`，给出明确错误提示或自动禁用不支持的路径（BF16）。  
- **缓存路径明确化**：提供环境变量（如 `SGLANG_JIT_CACHE_DIR`）让运维可控缓存位置，避免因磁盘清理导致意外重新编译。

## 🔍 技术洞察

### 架构影响
| 维度 | 正面影响 | 负面影响 |
|------|----------|----------|
| **代码可维护性** | - 统一使用 `GraphInputBuffers`，消除了多套派生 dataclass，代码路径更清晰。<br>- 删除了共享池的全局状态，降低了隐藏的副作用风险（如跨实例错误共享）。 | - 原来的 `share_buffers()` 机制用于在同一进程内多实例之间复用已分配的 GPU 张量，撤销后每个实例都会独占一套缓冲区，内存占用会上升。 |
| **模块耦合** | - 各执行器（CUDA‑Graph、Prefill、Eagle‑Draft等）直接操作同一 buffer 类型，减少了接口适配层。 | - 需要确保所有旧代码路径已全部迁移；若还有残余对 `DecodeInputBuffers`/`ForwardInputBuffers` 的引用，编译会失败或产生运行时错误。 |
| **扩展性** | - 将 buffer 初始化逻辑集中在 `GraphInputBuffers.create`，后续添加新字段只需修改单一位置。 | - 失去了通过全局缓存池在不同模型 runner 之间“共享”张量的能力，在大模型多卡部署场景可能导致显存紧张。 |

### 性能影响
- **显存利用**：  
  - **之前**：`share_buffers()` 通过 `as_strided` 复用已分配的大张量，能够在不同 `DecodeInputBuffers` 实例之间复用内存，从而在多实例（如多卡/多模型）情况下显著降低显存碎片化。  
  - **现在**：每个 runner 实例在构造时都会 `torch.zeros`/`torch.full` 分配完整的缓冲区，显存占用与实例数线性增长。对单实例场景影响不大，但在多实例（尤其是 `piecewise_cuda_graph_runner` 与多层 Eagle）下可能出现 **OOM**。

- **CPU/GPU 拷贝**：  
  - 代码仍然沿用原有的 `copy_`、`zero_`、`fill_` 等原位操作，拷贝次数未变，算子调度保持不变。  
  - 移除 `share_buffers()` 后不再出现 `as_strided` 产生的额外视图创建开销，略有正向影响。

- **编译/运行时**：  
  - 由于不再涉及跨实例的内存别名，`torch.compile` 的追踪更加直接，理论上可以略微提升 Torch‑Compile 的稳定性。  
  - 但若业务侧依赖原先的共享池来降低编译时间（因为同一张量被多次使用时编译器会复用图），可能会出现 **编译时间回升**。

### 安全考虑
- 本次回退不涉及网络、文件或权限相关代码，安全风险极低。
