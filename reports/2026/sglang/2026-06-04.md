### [mem_cache][2/N] refactor: move SWATokenToKVPoolAllocator to allocator/swa.py (#26676)
**SHA**: `1af53f6` | 🔗 [查看提交](https://github.com/sgl-project/sglang/commit/1af53f67b24aeb8e8e39e9326304c452a76c35e7)

**🎯 变更类型**：重构  
**⚡ 重要程度**：🔴高  
**📋 变更摘要**：将 `SWATokenToKVPoolAllocator` 从 `sglang/srt/mem_cache/swa_memory_pool.py` 中抽离，迁移到新的模块 `sglang/srt/mem_cache/allocator/swa.py`，并统一全项目的导入路径。此次改动仅涉及代码组织，功能保持不变，旨在提升代码可维护性与模块划分的清晰度。  

**🎯 影响范围**：  
- `sglang/srt/mem_cache/allocator/swa.py`（新文件）  
- 所有原先直接引用 `sglang.srt.mem_cache.swa_memory_pool.SWATokenToKVPoolAllocator` 的模块，包括：  
  - `kv_canary`（api、runner）  
  - `layers/attention/flashinfer_backend.py`  
  - `managers/schedule_policy.py`  
  - `mem_cache/chunk_cache.py`、`common.py`、`swa_radix_cache.py`、`unified_cache_components/swa_component.py`  
  - `model_executor/model_runner_kv_cache_mixin.py`（仅保留 `SWAKVPool` 导入）  
  - 单元测试相关文件（全部更新 import 路径）

### ⚠️ 潜在风险  

1. **向后兼容破坏**  
   - 外部使用者（例如第三方库或自定义插件）可能仍依赖旧路径 `sglang.srt.mem_cache.swa_memory_pool.SWATokenToKVPoolAllocator`，导致 `ImportError`。  
2. **循环依赖误判**  
   - `swa_memory_pool.py` 中仍引用 `SWAKVPool`，若后续在该文件中再次引入 `allocator.swa`（例如在实现中直接使用分配器），可能产生循环导入错误。  
3. **打包/发布遗漏**  
   - 新增文件 `allocator/swa.py` 必须被包含在 `setup.cfg`/`MANIFEST.in` 或对应的 `pyproject.toml` 中，否则在 pip 包发布后会缺失该模块。

### [diffusion] chore: disagg server args, launch helpers, and warmup utils (#26119)
**SHA**: `858e5a5` | 🔗 [查看提交](https://github.com/sgl-project/sglang/commit/858e5a5109659a82213baab8a8c0ebf071d38d66)

**🎯 变更类型**：功能增强 / 重构 / 架构变更  
**⚡ 重要程度**：🔴高  

**📋 变更摘要**  
1. 将原本散落在 `disagg_args.py` 中的所有 **disaggregated diffusion** 参数、帮助函数以及端点派生逻辑抽离到全新的 `server_args_disagg.py` 并以 `DisaggServerArgsMixin` 形式混入 `ServerArgs`，实现**职责分离**、统一的 CLI 注册入口。  
2. 新增环境变量 `SGLANG_DIFFUSION_PLATFORM_OVERRIDE`，允许在 **分布式/分离角色** 场景下强制指定运行平台（cpu、cuda、rocm、mps、npu、musa），并在平台解析时优先使用该覆盖。  
3. 调整各 `platforms/*.py` 中 `get_available_gpu_memory`、`get_local_torch_device` 等方法的签名，从必填 `device_id:int` 改为可选 `int|None`，并在未提供时自动使用当前设备，实现**更灵活的显存查询**。  
4. 在 `utils/common.py` 中加入 **GPU ID 归一化**、**TCP 地址解析/格式化**、以及更严谨的输入校验函数。  
5. `ServerArgs` 新增 `base_gpu_id`、`gpu_ids`、一批分离模式专用参数（如 `disagg_timeout`、`disagg_downstream_wait_timeout`、`disagg_role_device`、`disagg_transfer_backend` 等），并实现 **decoder_tp → decoder_sp** 的兼容/去除逻辑。  
6. 完善单元测试，确保新路径兼容、参数解析、异常检测等行为。

  - **模块耦合度**：`load_snapshot` 仍保持读写分离，新增的 `fileno`/`poll` 对外暴露最小接口，保持了原有 `LoadSnapshotReader` 的抽象不变，影响范围可控。

- **性能影响**  
  - **降低 CPU 占用**：原先使用固定周期的轮询 (`_poll` 每 10 ms 等) 在空闲时仍会占用 CPU；改为基于 ZMQ FD 的事件驱动后，仅在有新快照消息时才触发，显著减小空转开销。  
  - **延迟提升**：ZMQ 消息到达后立即触发 `poll`，SHM 写入即时可被 tokenizer worker 读取，整体快照同步延迟从毫秒级降至 **子毫秒级**（取决于系统调度）。  
  - **内存占用**：引入 `load_snapshot_reader` 在 Router 进程的生命周期中保持打开的 ZMQ socket，除非显式 `close()`，与原实现差异 negligible。

- **安全考虑**  
  - **绑定冲突防护**：通过单一所有者机制根除 “duplicate bind” 导致的异常退出或端口冲突，提升系统可用性。  
  - **ZMQ FD 暴露**：仅在内部事件循环注册，未对外暴露 socket 对象，仍然受 ZMQ 本身的安全模型（无明文网络，仅在本机 IPC）约束。  
  - **资源泄露风险**：新增 `fileno`/`poll` 需要在进程退出时确保 `close()`，否则可能残留未关闭的 ZMQ socket，潜在的资源泄露应在 `MultiTokenizerRouter` 的关闭流程中处理。

**⚠️ 潜在风险**  
1. **事件循环注册错误**：如果 `load_snapshot_reader` 在非 `asyncio` 循环或已经关闭的 loop 中注册，可能抛出异常或导致 FD 失效。  
2. **多进程竞态**：虽然 `zmq_reader_owner` 已保证唯一所有者，但在极端的启动顺序（例如，一个进程还未初始化 `server_args`）下可能出现短暂的双绑定窗口。  
3. **测试覆盖不足**：当前单元测试覆盖了所有 Caller 组合的所有权判定，但未验证 **异常关闭**、**异常网络**（如 ZMQ 消息丢失）场景。  
4. **兼容性**：旧版使用 `caller="tokenizer"` 的外部插件或自定义脚本如果未同步更新会因找不到对应的 Caller 而失效。

**💡 关注建议**  
- **代码审查**：确认所有自定义或第三方插件在创建 LoadSnapshotReader 时使用最新的 Caller 常量（`TokenizerManager`、`DataParallelController`、`MultiTokenizerRouter`）。  
- **回滚方案**：保留旧的 `caller="tokenizer"` / `"dp_controller"` 兼容路径（如通过内部映射），以防在未升级的子进程中出现 `ValueError`。  
- **资源回收**：在 `MultiTokenizerRouter` 的关闭路径（`__del__` 或显式 `shutdown`）中调用 `self.load_snapshot_reader.close()`，并从 event loop 中 `remove_reader`，防止文件描述符泄漏。  
- **监控**：在生产环境加入对 ZMQ FD 注册成功/失败的日志埋点，便于快速定位因异常启动顺序导致的绑定冲突。  
- **持续测试**：补充 **压力测试**（高并发请求、频繁快照更新）以及 **异常注入**（强制关闭 ZMQ socket、模拟网络错误），确保事件驱动路径在异常情况下仍能安全降级到轮询或安全退出。  

通过上述改动，系统在多 tokenizer 场景下的可靠性得到显著提升，同时也通过事件驱动实现了更好的资源利用和响应速度。>

### Unify full→SWA index translation in init_forward_metadata; drop pool caches (#27091)
**SHA**: `c9ca56d` | 🔗 [查看提交](https://github.com/sgl-project/sglang/commit/c9ca56da8c5e79ed731570859c06204bfb8a6964)

**⚡ 重要程度**：🔴 高  

**📋 变更摘要**

- 移除了 `SGLANG_OPT_CACHE_SWA_TRANSLATION` 环境变量及相关的缓存/失效逻辑，简化了 `DeepSeekV4TokenToKVPool`、`SWAKVPool` 等实现。  
- 明确所有 **FlashMLA、FlashAttention、HiSparse、TRT‑LLM** 等后端在使用页面索引时强制转为 `int32`，消除隐式类型不匹配的风险。

**🎯 影响范围**  
- `python/sglang/srt/layers/attention/*_backend*.py`（DeepSeekV4、FlashAttention、HiSparse、TRT‑LLM）  
- `python/sglang/srt/mem_cache/*`（DeepSeekV4TokenToKVPool、SWAKVPool、BaseSWAKVPool）

---

### 🔍 技术洞察  

#### 架构影响  
1. **统一的翻译入口**  
   - 之前每层在 `store_cache` 时都会调用 `translate_loc_from_full_to_swa`，并通过可选的缓存 (`cached_loc`) 进行复用。现在翻译工作搬到 **一次性** 的前向元数据初始化 (`init_forward_metadata_in_graph`) 中完成，所有层共享同一份 `swa_out_cache_loc`。  
   - 这使得 **SWA KV‑store** 的写入路径更加明确：只依赖 `forward_metadata.core_attn_metadata.swa_out_cache_loc`，不再依赖层级之间的隐式缓存一致性。  

2. **去除环境开关**  
   - `SGLANG_OPT_CACHE_SWA_TRANSLATION`（默认 `True`）在本次 PR 中被删除。此前开启后会在 `DeepSeekV4TokenToKVPool` 中进行 translation‑cache，导致在 HiCache 回写后 **register_mapping** 未清空缓存而出现 **stale‑loc** 错误。  
   - 现在 **所有翻译均即时完成**，不再需要显式失效，架构上减小了状态管理的复杂度。  

3. **统一的 int32 页面索引**  
   - 多个后端（FlashMLA、FlashAttention、HiSparse、TRT‑LLM）在调用 `translate_loc_*` 后都显式 `.to(torch.int32)`，并在 `get_swa_out_cache_loc` 中也返回 `int32`。  
   - 统一数据类型避免了 CUDA kernel 对 **int64** 参数的隐式截断或报错，提升了跨平台（CUDA、HIP）的一致性。  

#### 性能影响  
| 影响维度 | 正面 | 负面 | 说明 |
|---|---|---|---|
| **翻译开销** | 通过一次性翻译降低了 **层级循环** 中的 `translate_loc` 调用次数（原本每层一次）。 | 在 **非图内**（eager idle、只跑 out‑graph）路径仍需每次 translate，略微增加单次写入成本。 | 对大多数推理场景（使用 CUDA 图）收益显著，对极少数仅走 out‑graph 的路径影响可接受。 |
| **缓存失效成本** | 删除 `invalidate_loc_cache` 调用，省去不必要的同步与 CPU‑GPU 调用。 | 失去原有的 **缓存命中**（在极端长序列、极高 batch）可能导致 **额外一次** 翻译。 | 通过 `swa_out_cache_loc` 缓存每次 forward 的结果，仍然保留了高效的 fast‑path。 |
| **内存占用** | 移除 `cached_loc`、`_cached_loc_key` 等临时字段，略微降低内存足迹。 | 无显著负面影响。 |

#### 安全考虑  
- **类型安全**：强制 `int32` 页面索引消除了因 **int64 → int32** 隐式截断导致的内存越界或未定义行为。  
- **并发安全**：`get_swa_out_cache_loc` 在检查缓存有效性时验证了 **shape 相等且非 idle**，防止在 DP‑padding、draft‑extend 等改变 `out_cache_loc` 长度的情况下误用旧缓存。  
- **回退路径**：即使 `forward_metadata` 为 `None`（eager idle）或缓存失效，仍会调用 `translate_loc_from_full_to_swa`，保证功能不因缓存缺失而中断。  

---

### ⚠️ 潜在风险  

1. **性能回退风险**  
   - 对于 **不走 CUDA‑graph**、仅执行 `init_forward_metadata_out_graph` 的工作流（例如某些实验性 `eager idle` 或自定义 runner），每次写入都会重新翻译，可能在 **极高 QPS** 场景下产生可感知的额外延迟。  
   - 建议在性能敏感的自研后端上进行基准对比，评估是否需要再次引入可选的局部缓存。  

2. **迁移兼容性**  
   - 删除 `SGLANG_OPT_CACHE_SWA_TRANSLATION` 以及 `invalidate_loc_cache` 接口后，外部用户若在自定义代码中手动调用这些已不存在的 API，将导致 **AttributeError**。需要在发布说明中明确该破坏性变更。  

3. **潜在的 stale‑metadata 场景**  
   - `get_swa_out_cache_loc` 判断 “cached.shape[0] == out_cache_loc.shape[0]” 仅针对 **shape 长度**，但若 `out_cache_loc` 内容被 **截断后重新填充**（同长但不同内容），仍可能错误使用缓存。理论上这情况极少，但在 DP‑padding 后 `out_cache_loc` 会被 **pad** 扩长而不等长，已被覆盖；若未来出现 **同长但不同 token‑offset** 的重绑定，需要额外的 hash（如 data_ptr）来更严格判断。  

4. **类型转换一致性**  
   - 所有新加的 `.to(torch.int32)` 必须与 C++/CUDA kernel 的签名保持一致。若后端内部改为支持 `int64`，可能出现 **不匹配** 报错，需要同步更新。

### 💡 关注建议  

| 对象 | 建议 |
|---|---|
| **开发者** | - 在自定义模型/后端中检查是否仍有对 `SGLANG_OPT_CACHE_SWA_TRANSLATION` 或 `invalidate_loc_cache` 的引用，及时删除。<br>- 对仅使用 **eager idle** 或 **out‑graph only** 的路径进行性能基准，确认额外的翻译开销在可接受范围内。 |
| **测试/CI** | - 为 `get_swa_out_cache_loc` 新增 **shape‑不匹配** 与 **idle** 场景的回归测试，确保 fallback 始终返回正确 `int32` 张量。<br>- 保留或新增 **long‑sequence**（> 64k）下的吞吐率对比，防止因失去缓存导致显著下降。 |

### [Mamba] extra buffer lazy support (#27118)
**SHA**: `e0b6926` | 🔗 [查看提交](https://github.com/sgl-project/sglang/commit/e0b692600f1072af46846aa83a14520125974aa2)

**🎯 变更类型**：功能增强 / 性能优化  
**⚡ 重要程度**：🔴高  
**📋 变更摘要**：

**🎯 影响范围**：  
- `python/sglang/srt/environ.py`（新增测试环境变量）  
- 调度批处理 `schedule_batch.py`（新增 lazy 预分配、边界处理逻辑）  
- 批处理结果处理 `batch_result_processor.py`（调整 mamba 前缀缓存更新及 KV 释放路径）

**🔍 技术洞察**  

- **架构影响**  
  - 引入 **惰性 extra buffer** 概念，使每个请求在普通状态下仅持有 1 个 ping‑pong 槽（而非 2），在 `mamba_track_interval` 边界才动态分配第二槽。  
  - `ReqToTokenPool` 新增 `enable_mamba_extra_buffer_lazy` 标记、`get_mamba_ping_pong_keep_idx`、`_alloc_ping_pong_buffer`、`set_mamba_ping_pong_slot`、`donate_mamba_ping_pong_slot` 等 API，统一管理懒分配与正常分配路径。  
  - `ScheduleBatch` 在 `prepare_for_decode` 中调用 `mamba_lazy_prealloc_at_boundary`，并在 `BatchResultProcessor` 里统一走 `mamba_lazy_post_decode_at_boundary` 完成资源清理。  
  - 通过 `CacheInitParams.enable_mamba_extra_buffer_lazy` 将配置向下传递至内存池、统一缓存组件、模型运行器，实现全链路的惰性行为。  

- **性能影响**  
  - **内存**：在重叠调度（default）下每个请求的 Mamba 状态从 3/slot（prefix‑cache+extra buffer）降至 **2/slot**，在非重叠调度下保持 1/slot。大幅降低显存占用，特别是在多卡、多请求场景下可提升可容纳的并发数。  
  - **计算**：惰性分配在边界时产生一次额外的 `alloc` + `set` 操作，但该路径只在每 `mamba_track_interval`（通常为 128）才触发，对整体吞吐影响可忽略。  
  - **调度延迟**：若懒分配在边界时失败（预设的测试路径），会触发 cache‑evict + 再次分配的回退机制，导致单次 token 处理略有抖动，但不会导致系统崩溃。  

- **安全考虑**  
  - 新增的环境变量 `SGLANG_TEST_MAMBA_LAZY_ALLOC_FAIL` 与 `SGLANG_TEST_SKIP_CACHE_HIT_ASSERT` 仅用于 **内部测试**，不会影响生产部署。  
  - 所有新路径均在 `assert` 中检查 ping‑pong 缓冲区的合法性（不应出现 `-1`），防止因未分配成功导致的状态覆盖与缓存不一致。  
  - 通过 `enable_mamba_extra_buffer_lazy()` 与 `disable_overlap_schedule` 的相互断言，避免在不兼容的配置下启动懒分配，防止潜在的未定义行为。  

**⚠️ 潜在风险**  

1. **边界同步错误**  
   - 惰性分配依赖 `seq_lens_cpu % interval == 0` 判断边界，若前端对 `seq_lens_cpu` 的增量出现异常（如自定义调度修改 seq_len），可能导致漏分配或多余分配，进而出现状态覆盖。  
2. **并发释放竞争**  
   - 在高并发下，多个请求恰好在同一边界触发 `lazy_prealloc_at_boundary`，若 `alloc` 返回 `None` 并触发 `evict`，可能导致其他请求的 slot 被提前释放，需确保 `evict` 的粒度（仅 `mamba_num=1`）足够细。  
3. **回退路径未充分测试**  
   - 强制分配失败的测试覆盖了单卡环境，分布式（TP/PP）场景下的 `evict` 与 slot 迁移仍需要验证。  
4. **兼容性**  
   - 旧版 `extra_buffer` 与 `extra_buffer_lazy` 在同一进程中共存时，`alloc_req_slots` 会依据 `enable_mamba_extra_buffer_lazy` 选择不同 `factor`，若误配置导致两种策略混用，可能出现 `slot` 数不足的断言。  

**💡 关注建议**  

- **监控**：在生产环境开启 `extra_buffer_lazy` 时，建议监控 `mamba_extra_buffer_lazy_alloc_fail`（可通过新增指标计数）以及 `mamba_lazy_post_decode_at_boundary` 的执行频率，及时发现异常回退次数激增。  
- **配置校验**：在服务器启动阶段加入更明确的冲突检查，如 `if enable_mamba_extra_buffer_lazy and not overlap_schedule: raise ConfigError`（已有断言），并在文档中明确 `extra_buffer_lazy` 仅在启用 overlap schedule 时可用。

## ⚠️ 潜在风险

| 风险点 | 可能影响 | 缓解措施 |
|--------|----------|----------|
| **共享池的注册顺序导致错误复用** | 若不同 runner 在同一进程中使用不同 `max_num_tokens` 或 `max_bs`，可能出现 **键冲突**（同名、同 dtype、同 device 但尺寸不同）导致错误共享同一块内存。 | 实现已在 `share_input_buffer` 中使用 `(name, numel, dtype, device)` 作为键，确保尺寸相同才共享。新增单元测试验证不同尺寸不共享。 |

| **`source_fn` 与 `post_fill` 的副作用** | `source_fn` 可能返回 **不符合切片大小** 的张量，或 `post_fill` 在错误的上下文中修改了全局状态，导致图不可重入。 | `fill_from` 中对 `src.shape[0]` 执行切片，确保只拷贝头部。`post_fill` 仅接受本 slot 的 buffer 与 FB，保持局部性。单元测试覆盖异常返回 `None` 场景。 |
| **多平台（NPU、CPU、GPU）兼容性** | `share_input_buffer` 在 NPU 上被禁用，若误用导致共享池仍尝试在 NPU 创建，可能触发 **设备不支持的 stride** 错误。 | 已在 `share_input_buffer` 中添加平台检测 `if device.type == "npu":` 直接返回原 buffer；CI 中包含 NPU‑skip Path 测试。 |
| **图捕获期间的地址稳定性** | 若在 capture 前后通过 `share_pool=False` 重新分配，可能导致捕获图的 `data_ptr` 与 replay 时不一致。 | 迁移步骤强制在 `CudaGraphBufferRegistry` 中 `share_pool=True`（默认），且在 `Decode`/`Prefill` runner 的 `__init__` 中立即创建并 **adopt** 旧 buffers，确保 capture 与 replay 使用同一块物理内存。 |
