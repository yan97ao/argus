### Support singleton dimension squeezing in dump comparator (#19566)
**SHA**: `5705e02` | 🔗 [查看提交](https://github.com/sgl-project/sglang/commit/5705e02d286e59863b54935a415735f173e29361)

**🎯 变更类型**：功能增强  
**⚡ 重要程度**：🔴高  
**📋 变更摘要**：在调试对比工具中新增 **AxisAligner**，支持在跨框架/跨进程对齐张量时自动 **压缩（squeeze）单例维度**（维度名为 `1`），并在必要时完成维度顺序重排。相应地对 `dims` 解析、名称解析、入口点计划与执行逻辑进行重构，废弃旧的 `AxisSwapper` 实现并更新所有相关调用和测试。  

**🎯 影响范围**：  
- `sglang/srt/debug_utils/comparator/aligner/axis_aligner.py`（新模块）  
- `sglang/srt/debug_utils/comparator/aligner/entrypoint/*`（planner、executor、types）  
- `sglang/srt/debug_utils/comparator/dims.py`（单例维度工具、`resolve_dim_names`）  
- `sglang/srt/debug_utils/comparator/bundle_comparator.py`、`token_aligner/aux_loader.py`（使用 `resolve_dim_names`）  
- 相关的 warning、输出格式及记录序列化代码  
- 单元测试与 CI 注册文件  

---

### 🔍 技术洞察  

| 维度 | 影响说明 |
|------|----------|
| **架构影响** | <ul><li>新增 **AxisAligner** 取代原 `AxisSwapper`，负责 **两步**：① 过滤单例维（`_SingletonDimUtil.filter_out`），② 若必要生成 `einops` 重排模式。</li><li>`AlignerPlan` 结构从 `axis_swapper_plan`→`axis_aligner_plan`，所有上层入口点（`execute_aligner_plan`、`compute_aligner_plan`）同步改造。</li><li>`dims` 模块引入 `SQUEEZE_DIM_NAME = "1"` 与 `_SingletonDimUtil`，统一对单例维的检测、命名替换（`singleton0`, `singleton1`, …），保持 **named‑tensor** 的合法性。</li><li>对外 API (`parse_dim_names` → `resolve_dim_names`) 兼容旧行为，同时提供更安全的名称映射。</li></ul> |
| **性能影响** | <ul><li>过滤单例维和生成 `einops` pattern 的额外 O(N) 计算（N 为维度数），在典型 2~5 维张量中可忽略。</li><li>实际执行仍依赖 `einops.rearrange`，与原 `axis_swapper` 相同的时间复杂度。</li><li>新增 `sanitize_names` 在 `resolve_dim_names` 中执行一次字符串拼接，同样微不足道。</li></ul> |
| **安全考虑** | <ul><li>无外部 I/O、网络或系统调用，仅在本地对张量元数据做转换，安全风险极低。</li><li>对非法 `1(tp)` 这类带修饰符的单例维抛出 `ValueError`，避免异常的隐式静默。</li><li>维度名冲突（如用户自行使用前缀 `singleton`）会被误判为已映射的单例维，可能导致意外重命名；已在实现中通过 `is_singleton_name` 检查，仅在生成新名字时使用，冲突概率低但需文档提醒。</li></ul> |
| **可维护性** | <ul><li>通过 `AxisAlignerPlan` 把 **压缩** 与 **轴重排** 合并为单一计划，代码路径更清晰。</li><li>`_SingletonDimUtil` 把单例维相关逻辑集中，后续若需支持其他 “虚维” 可在此扩展。</li><li>废弃的 `axis_swapper` 仍保留在历史分支中，但主线已完全替换，避免残余引用。</li></ul> |

---

**⚠️ 潜在风险**  

1. **向后兼容性**  
   - 直接引用旧属性 `axis_swapper_plan` 的第三方代码会在升级后出现 `AttributeError`。  
   - 旧的 `parse_dim_names` 已被删除，外部仍调用会报 `ImportError`。  

2. **名称冲突**  
   - 若模型或用户自行使用维度名 `singleton0`、`singleton1`，`sanitize_names` 可能产生冲突，使得实际维度顺序被误解释。  

3. **单例维过滤误用**  
   - 过滤过程只基于名称 `"1"`，若用户误将普通维命名为 `"1"`（而非真实 singleton），会被错误压缩。  

4. **测试覆盖不足的极端情形**  
   - 多层嵌套 tensor（如 `b t 1 h d` 与 `b 1 t d h`）的组合排列尚未在单元测试中全覆盖。  

5. **性能极端场景**  
   - 在极大维度数（>100）且每次都进行过滤/重排的情况下，累计的 Python 循环开销可能可观，需要在高维度模型上做基准。  

---

**💡 关注建议**  

1. **文档与迁移指南**  
   - 在 `debug_utils/comparator/README.md` 中说明 `axis_aligner` 替代 `axis_swapper`，并提供兼容层（如 `AxisSwapperPlan = AxisAlignerPlan`）的示例。  
   - 明确 `SQUEEZE_DIM_NAME = "1"` 的语义及不应在用户自定义维度中使用的约束。  

2. **向后兼容包装**  
   - 在 `aligner/axis_swapper.py`（保留）实现一个轻量包装，将内部调用转发到 `axis_aligner`，仅为旧代码提供临时兼容，后续可以在主分支中删除。  

3. **冲突检测**  
   - 在 `resolve_dim_names` 增加可选参数 `allow_user_singleton_prefix=False`，若检测到用户已使用 `singleton\d+` 抛出警告，防止无意冲突。  

4. **更全面的测试**  
   - 添加组合测试：`"b 1 t d"` ↔ `"t b d"`、多层并行（TP+CP）+单例维混合等，以验证 **压缩+重排** 的双重路径。  
   - 对 `execute_axis_aligner_plan` 在 **GPU** 环境下运行的性能基准，确认不存在显著回退。  

5. **CI/发布**  
   - 将新增的 `AxisAligner` 测试标记为 **required**，确保所有平台（Linux‑CPU, Linux‑GPU, macOS）均通过。  
   - 在发布说明（Release Notes）中标记 **破坏性改动**：`axis_swapper` 已删除，外部调用需迁移。  

--- 

**总体结论**：此次改动在 **功能** 与 **可维护性** 上提升显著，解决了跨框架对比时出现的 singleton 维度不匹配问题。只要做好向后兼容指引、强化文档、并在极端高维度场景做一次基准验证，即可安全上线。

## ⚠️ 潜在风险

1. **磁盘空间/IO 高峰**  
   - 大量张量（尤其是模型大的 transformer）开启可视化会一次性生成多张 PNG，可能在短时间内占满磁盘或导致 I/O 阻塞。  
2. **CPU/内存峰值**  
   - 绘图会把张量拷贝到 CPU (`tensor.detach().cpu().float()`) 并可能临时创建大 `numpy` 数组；在极端 4000x4000+ 的张量上仍可能接近阈值。  
3. **Matplotlib 后端**  
   - 在无显示服务器的环境（如某些 CI 运行器）如果默认后端不是 `Agg` 可能报 `RuntimeError: Invalid DISPLAY variable`。虽然测试已用 `importorskip`，但实际用户运行时仍需确保使用无头后端。  
4. **路径注入**  
   - 虽然已对文件名做正则清理，但如果用户自行传入恶意的 `viz_output_dir`（如指向系统根目录）仍可能写入不期望的位置。  
5. **依赖体积**  
   - `matplotlib` 带来约 50 MB 的二进制依赖，增加镜像体积，对资源受限的部署（如边缘设备）可能不友好。

**SHA**: `4097eb5` | 🔗 [查看提交](https://github.com/sgl-project/sglang/commit/4097eb5ce964b2f3e70b08fd5edb66beade58967)

**🎯 变更类型**：功能增强（调试/代码热修补）  
**⚡ 重要程度**：🔴高  
**📋 变更摘要**：在 `sglang/srt/debug_utils` 中新增 **source_patcher** 子包，提供基于 YAML 配置的源码编辑与函数热补丁能力。实现了 `CodePatcher` 上下文管理器、`patch_function`、源码编辑器 `apply_edits`、以及强类型的 Patch/Edit 描述模型，并配套完整单元测试。目标是让开发者能够在不重启进程的情况下，对函数实现进行快速修改、调试或实验。  

**🎯 影响范围**：  
- `sglang/srt/debug_utils/source_patcher/*`（新模块）  
- 与该包直接交互的调试/实验脚本（如 `sglang` 的 REPL、示例 notebooks）  
- 依赖 `pydantic`、`yaml` 的运行时环境  

**🔍 技术洞察**：

- **架构影响**  
  - 新增 **独立的调试子系统**，不侵入核心推理路径，保持向后兼容。  
  - 通过 `inspect.getsource` → `apply_edits` → `compile` → `exec` → 替换 `__code__`，实现热补丁。  
  - `CodePatcher` 采用 **上下文管理器**，在 `__exit__` 中逆向恢复，符合“临时调试”场景的设计哲学。  
  - 目标解析 `_resolve_target` 支持任意 `pkg.mod.Class.method`，对类方法、普通函数均可工作，增强可扩展性。  

- **性能影响**  
  - 仅在 **调试/测试阶段** 使用，热补丁本身的成本主要是一次性的源码读取、字符串处理、编译与执行。  
  - 对运行时性能几乎无影响——恢复后函数继续使用原始 `code` 对象。  
  - 若在高并发环境错误地使用，可能产生 **额外的编译开销** 与 **临时对象创建**，但仍在可接受范围。  

- **安全考虑**  
  - `apply_patches_from_config` 直接执行 **用户提供的源码**（通过 `exec`），若 YAML 配置来源不可信，会导致 **代码注入**。  
  - 当前实现没有任何安全沙箱或白名单检查，推荐仅在受信任的本地调试环境使用。  
  - 通过 `PatchApplicationError` 对匹配/唯一性进行校验，防止误操作导致意外代码覆盖。  

**⚠️ 潜在风险**  

| 序号 | 风险点 | 可能后果 | 缓解建议 |
|------|--------|----------|----------|
| 1 | **代码注入**（执行任意 `exec`） | 运行恶意代码，破坏系统、泄露数据 | 明确文档约束仅在本地/受信任环境使用；可在 `apply_patches_from_config` 加入可选的沙箱或 `ast.parse` 语法检查。 |
| 2 | **多线程竞态** | 同一函数在多个线程同时被补丁，可能出现恢复不完整或代码对象被并发修改 | 在 `CodePatcher` 中加入线程锁，或在文档中声明 **非线程安全**，仅在单线程调试时使用。 |
| 3 | **闭包/局部变量失效** | 若被补丁的函数引用了外部局部变量（闭包），重新编译后可能找不到同名对象，导致 `NameError` | 在 `patch_function` 中保留 `target.__globals__` 已经做得较好，仍需在测试中覆盖闭包场景并在文档中提示限制。 |
| 4 | **属性解析失败**（`_resolve_target`） | 目标函数路径写错导致 `ImportError/AttributeError`，调试脚本崩溃 | 已抛出异常，建议在 CLI/脚本层捕获并给出清晰错误提示。 |
| 5 | **资源泄漏**（大量临时代码对象） | 长时间运行的调试会占用额外内存 | `CodePatcher` 退出时会 `state.restore()` 并清空列表，使用后应确保上下文退出或手动 `restore()`。 |

**💡 关注建议**

   - 将该模块标记为 **“仅供调试/实验使用”**，在 `README`、`CHANGELOG` 中明确告知生产环境禁用。  
   - 在 CI 流水线中可加入检测，防止误把 `source_patcher` 代码路径加入正式发布的依赖。

   - 若社区有需求在多线程调试时使用，考虑在 `CodePatcher` 内部使用 `threading.Lock` 包装 `__enter__` 与 `__exit__`，或提供 `ThreadSafeCodePatcher` 实现。

   - 若后续计划将其作为公开 API，建议在 `__all__` 中显式导出相关类，保持语义清晰。  

---  

**总体结论**：此次提交为 `sglang` 增加了强大的源码热补丁调试能力，技术实现简洁且易于集成。只要在生产环境严格限制其使用范围，并在文档中对安全风险做出明确提示，即可为开发者提供极大的调试便利而不影响核心性能或稳定性。

**🔍 技术洞察**  

- **架构影响**  
  - **数据流**：新增了从 gRPC → protobuf → `torch.Tensor` → `MultimodalDataItem` → 调度批次的完整路径，形成了视觉模态在 SRT（Scheduling Runtime）中的第一条处理链路。

  - **接口兼容**：对已有的纯文本请求保持向后兼容；若请求中不包含 `mm_inputs`，`mm_inputs` 仍为 `None`，走原有路径。  

- **性能影响**  
  - **解码开销**：`np.frombuffer` + `reshape` + `torch.from_numpy` 将在每个多模态请求中额外分配一次 NumPy 数组（内存视图）并生成 Torch 张量。对单张 224×224×3 的 `float32` 图像约 0.6 MiB，解码耗时约 0.1‑0.3 ms（CPU），对整体推理 latency 影响可接受。  
  - **内存占用**：由于 `torch.from_numpy` 共享底层缓冲区，不会产生额外复制，但后续对张量的 inplace 操作会触发 copy‑on‑write，需在调度器层避免不必要的修改。  
  - **吞吐量**：若并发大量视觉请求，解码 CPU 可能成为瓶颈，建议在高负载场景下开启线程池或使用 `torch.frombuffer`（如果可用）直接构造张量。  

- **安全考虑**  
  - **数据完整性**：当前实现仅根据 protobuf 声明的 `shape` 与 `dtype` 直接创建 NumPy 缓冲区，未校验 `len(tensor_data.data)` 与 `shape` 对应的字节数是否匹配，恶意构造的 protobuf 可能导致 **缓冲区溢出** 或 **异常内存分配**。

1. **缓冲区大小不匹配**导致 `np.frombuffer` 抛异常，进而中断整个 gRPC 调用，影响服务可用性。  
2. **未限制张量尺寸**，在多用户环境下可能被利用进行内存耗尽攻击。  
3. **dtype 映射不完整**，对新模型（如 `bfloat16`、`int32`）的支持会出现运行时错误或误判。  
4. **新增依赖**（NumPy、PyTorch）若在生产镜像中缺失，部署会失败；版本不兼容也可能导致运行时错误。  
5. **占位符偏移计算**使用闭区间 `(start, end_inclusive)`，若调度器内部期待半开区间，可能导致 token 对齐错误。  

**💡 关注建议**  

- **输入校验**：在 `_decode_tensor_data` 前后加入以下检查  
  - `expected_bytes = np.prod(shape) * np.dtype(np_dtype).itemsize` 与 `len(tensor_data.data)` 必须相等。  
  - 对 `shape` 中任意维度设定上限（如 `<= 1024`），防止异常大张量。

- **单元/集成测试**：  
  - 正向测试：不同形状、dtype、模型特定张量的完整解析。  
  - 负向测试：shape 与 data 长度不匹配、非法 dtype、空占位符列表。  
  - 性能基准：对比无多模态请求的延迟与加入多模态解码后的延迟。  
- **文档与示例**：在项目的 gRPC API 文档中补充 `MultimodalInputs` 的字段说明、支持的 dtype 与尺寸限制，提供 Python 客户端示例代码。
