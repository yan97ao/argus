### [Feature] WebSocket streaming audio input for ASR (#22848)
**SHA**: `a95b4e2` | 🔗 [查看提交](https://github.com/sgl-project/sglang/commit/a95b4e2e09eb02631b8ebce5c9d769cd2690b16c)

**🎯 变更类型**：功能增强（WebSocket 实时音频输入与转写）  
**⚡ 重要程度**：🔴 高  

**📋 变更摘要**  
- 为 OpenAI‑兼容的 `/v1/realtime` 添加 WebSocket 入口，实现仅转写子集的实时音频流式处理。  
- 引入会话管理（`RealtimeConnection`），支持并发会话限制、音频缓冲、分块推理、动态采样率重采样以及增量转写事件（delta / completed / failed）。  
- 新增服务器配置项 `asr_max_buffer_seconds`、`asr_max_concurrent_sessions` 并在启动时做合法性校验。  
- 对原有 `streaming_asr` 逻辑进行抽象复用，使 HTTP SSE 与 WebSocket 共享 `process_asr_chunk`、`needs_space`、`normalize_whitespace` 等实现。  
- 完整测试覆盖（单元 + 手动），包括多语言、不同采样率、并发、错误路径等。

### 🎯 影响范围

- **Realtime 包**：`openai/realtime/handler.py`、`protocol.py`、`session.py`（会话协议、状态机、业务逻辑）。

| 维度 | 影响描述 |
|------|-----------|

### ⚠️ 潜在风险

1. **依赖体积**：`torch` 与 `torchaudio` 现在在路径中直接 import，用于重采样。若部署环境未装 GPU 版本或缺少 `torchaudio`，会在运行时抛 `ImportError`。  
2. **CPU 重采样开销**：在高并发场景下，每个音频块都要走 `torch` 计算，可能导致 CPU 饱和。  
3. **缓冲限制误判**：`asr_max_buffer_seconds` 默认 60 s，对长篇连续流（如会议）会提前关闭，需在生产环境根据业务调优。  
4. **并发信号量泄漏**：若 `handle_realtime_transcription` 在异常路径未释放 `session_semaphore`（如 `await ws.accept()` 失败后直接返回），可能导致槽位被永久占用。当前实现将 `async with session_semaphore` 放在最外层，已基本安全。  
5. **跨会话状态混用**：`RealtimeConnection` 对 `self.item.emitted_deltas`、`self.audio.pcm_buffer` 进行就地清空，若未来在同一对象上出现并发使用（例如复用对象），会产生泄漏。确保每个 WS 连接单独实例化。  
6. **语言模型兼容性**：`model_sample_rate` 默认 16000，若后端模型实际期望 24000（如部分商业模型），需要在对应 `TranscriptionAdapter` 子类覆盖该属性，否则会产生二次重采样导致质量下降。

### [diffusion] fix: fix diffusion serve warmup defaults (#26247)
**SHA**: `f70e604` | 🔗 [查看提交](https://github.com/sgl-project/sglang/commit/f70e6041011bca5fedd7989e8a1acec1d3559620)

**🎯 变更类型**：功能增强 / Bug修复 / 性能优化  
**⚡ 重要程度**：🔴高  

**📋 变更摘要**  
1. 引入 **`server_warmup`** 标记，默认在 `sglang serve` 中开启，改为在 HTTP 服务就绪后主动发送轻量级暖机请求，而不再依赖首条真实请求的隐式暖机。  
2. 将暖机相关逻辑抽象到新模块 **`server_warmup.py`**，统一生成暖机请求、解析模型默认采样参数、决定是否需要图片输入、处理临时暖机图片等。  
3. 为 **Flux2Finetuned** 配置新增 `preprocess_vae_encode`/`postprocess_vae_encode`，解决 5‑D latent（单帧）在无 BN VAE 时的维度不匹配问题。  
4. 在 `ImageVAEEncodingStage` 中抽取 `scale_and_shift_encode_latents` 并支持 `shift_factor` 为 `None`，统一 latent 的缩放/平移逻辑。  
5. 调整 `ServerArgs` 解析、默认值注入、日志层级以及层级离线(offload)警告，确保在分布式/单卡场景下 `server_warmup` 与 `warmup` 的互斥与一致性。  
6. 迁移暖机计时、日志、异常处理到异步 HTTP 启动阶段，新增 FastAPI 中间件阻塞非暖机请求直至暖机完成。  
7. 更新多处测试、基准数据以及性能容忍阈值，以适配新的暖机流程。  

**🎯 影响范围**  
- `python/sglang/multimodal_gen/runtime/server_args.py`、`runtime/entrypoints/cli/serve.py`、`runtime/entrypoints/http_server.py`、`runtime/managers/scheduler.py`（核心调度器）  
- `configs/pipeline_configs/base.py`、`configs/pipeline_configs/flux_finetuned.py`（pipeline 配置）  
- `pipelines_core/stages/image_encoding.py`、`pipelines_core/schedule_batch.py`（stage 与请求模型）  
- 新增 `runtime/server_warmup.py`（暖机工具）  
- 多个单元/集成测试文件以及性能基准文件  

**🔍 技术洞察**  

- **架构影响**  
  - **新增暖机子系统**：`server_warmup` 把暖机从 “首请求触发”（隐式）迁移到 “服务启动后主动发送” 的显式流程，解耦了 Scheduler 与 HTTP Server 启动顺序。  
  - **统一暖机生成**：`build_warmup_reqs`、`should_include_warmup_image` 等函数在 `server_warmup.py` 中集中，实现了对 **模型默认采样参数**、**分辨率回退**、**图片输入需求** 的统一判定。  
  - **FastAPI 中间件**：在 `create_app` 中加入 `wait_for_server_warmup`，确保除健康检查外的所有请求在暖机完成前被挂起，提升了对外可观测性（“Server ready”日志仅在暖机成功后打印）。  
  - **Scheduler 调整**：原来 `prepare_server_warmup_reqs` 与 `process_received_reqs_with_req_based_warmup` 的暖机路径被精简，统一使用 `is_warmup_req`、`should_return_warmup_result` 判断，且在 warmup 完成后可选择 **不回传 payload**（`drop_payload_for_warmup`），降低网络/序列化开销。  
  - **Flux2Finetuned VAE 兼容**：针对 5‑D latent（`B, C, 1, H, W`）的 VAE 编码前处理（squeeze）以及后处理（restore/patchify）实现，使得不含 BN 的 VAE 在单帧/多帧场景下能够平滑运行。  

- **性能影响**  
  - **启动时延**：显式暖机在 HTTP 端点可达后立即执行，避免了首次真实请求因为模型懒加载/缓存未命中导致的长尾延迟。  
  - **暖机资源占用**：通过 `return_warmup_result` 与 `drop_payload_for_warmup`，暖机请求的响应仅保留最小元信息，显著降低 ZMQ 传输和序列化成本。  
  - **基准容忍阈值**：`test_server_utils.py` 中对第一步 denoise 的容差放宽、对解码阶段的最小绝对容差提升，以适配暖机期间的初始化开销波动。  
  - **VAE 编码**：`scale_and_shift_encode_latents` 移植至 `ImageVAEEncodingStage`，避免在每个 batch 中重复执行 Python 条件分支，引入了更轻量的 Tensor 运算路径。  

- **安全考虑**  
  - 暖机图片基于固定的 **最小 Base64** 数据生成，写入临时目录，仅在内部使用，不接受外部用户输入，未引入新增的攻击面。  
  - 若暖机图片写入或 `save_image_to_path` 失败，`_run_server_warmup_after_http_ready` 会记录错误并 **SIGTERM** 进程，防止服务在不完整状态下继续提供服务。  
  - `server_warmup` 参数默认在 `serve` 子命令中打开，且可以通过 CLI/配置显式关闭，符合最小权限原则。  

**⚠️ 潜在风险**  

| 风险点 | 描述 | 可能影响 |
|-------|------|----------|
| **默认开启 `server_warmup`** | 在资源受限的环境（如低显存 GPU）启动时会额外消耗一次前向推理。 | 启动时间略增、瞬时显存峰值可能触发 OOM。 |

1. **兼容性**：`torch.ops.aten.alias` 返回的视图不保留任何内部元数据，只是浅拷贝。若后续代码依赖原生 `tensor.names`（如第三方库、调试日志），可能出现属性缺失。建议在项目文档中明确说明仅在 SGLang 调试工具链内部使用，外部代码仍应使用原生 `named_tensor`（若需要）。  
2. **性能**：`apply_dim_names` 每次都会创建一个新 view 并写入 `_dim_names`，这种轻量包装几乎没有额外开销，但大量拼接/求和后频繁调用可能导致大量对象创建。可以在热点路径（如 `execute_unsharder_plan`）加入一次性批量恢复的优化（一次性 `apply_dim_names` 而非在每个子张量上重复）。  
3. **正确性**：所有 `refine_names` 替换为 `apply_dim_names` 时，需要确保维度顺序与 `dim_names` 完全对应。当前实现仅在维度数量不匹配时报错，建议在 `apply_dim_names` 前加入断言：`assert tensor.shape == tuple(... )`，防止因 reshape 导致的维度错位。  
4. **测试覆盖**：提交已更新大量单元测试，但仍建议在 CI 中加入 **交叉验证**：对同一张量分别使用原生 `named_tensor` 与新实现，验证 `torch.equal`、`tensor.shape`、`tensor._dim_names` 是否保持一致。  
5. **文档与迁移指南**：新增函数已在 `dims_spec/__init__.py` 导出，建议在 `README` 或 `docs/debug_utils.md` 中添加迁移示例（`refine_names` → `apply_dim_names`、`tensor.names` → `get_dim_names`），并标记旧 API 为已废弃。  

总体而言，此改动统一了维度名称的管理方式，摆脱了 PyTorch 对 named tensors 的依赖，为跨平台（CPU/GPU）和未来 PyTorch 版本的兼容性提供了更可控的实现。但需注意上述兼容性、性能和文档细节，确保新工具在生产环境中使用安全、透明。
