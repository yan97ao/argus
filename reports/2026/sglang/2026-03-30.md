### fix: Mistral Small 4 fails to start due to config/weight format mismatch (#21620)
**SHA**: `1d6424d` | 🔗 [查看提交](https://github.com/sgl-project/sglang/commit/1d6424d5ad2dd1edb5ce9bb477be81dcf2b06eef)

**🎯 变更类型**：Bug修复  

**⚡ 重要程度**：🔴高  

**📋 变更摘要**：  
- 修复了 Mistral‑Small‑4 模型在启动时因配置/权重格式不匹配导致的加载失败问题。  
- 通过在 `server_args.py` 中加入对 “mistral‑large‑3、mistral‑small‑4、leanstral” 三类模型的特例判定，使其在同时存在 `params.json` 与 `config.json` 时仍强制使用原生（Mistral）权重命名。  
- 新增对应的 CI 测试用例，确保该模型在多 GPU（TP=2）并开启 `--trust-remote-code` 时能够成功启动并通过基本的 GSM8K 与 MMMU 评估阈值。  

**🎯 影响范围**：  
- `python/sglang/srt/server_args.py`（模型加载路径判定逻辑）  
- CI/单元测试目录 `test/registered/models/`（新增 Mistral‑Small‑4 测试）  
- 受影响的模型加载子系统：本地模型目录、通过 HuggingFace Hub 拉取的模型。  

**🔍 技术洞察**：  
- **架构影响**：  
  - 该修改仅在模型加载阶段添加了一个“白名单”判定函数 `_check_format`，不影响整体服务架构或调用链。  
  - 与 `hf_transformers_utils.py` 中的配置获取保持同步，避免因命名不一致产生的权重映射错误。  
- **性能影响**：  
  - 判定逻辑只涉及文件存在性检查和字符串匹配，开销极低（微秒级），对启动时延或运行时性能基本无影响。  
- **安全考虑**：  
  - 引入 `--trust-remote-code` 参数的模型本身已经是安全风险点，此次改动未改变该风险范围。  
  - 仅在白名单模型上强制使用原生权重，未对其他模型放宽安全检查，安全姿态保持不变。  

**⚠️ 潜在风险**：  
1. **误判风险**：如果未来出现新的 Mistral 系列模型但未加入 `_MISTRAL_NATIVE_CONFIG_PATTERNS`，仍会走默认 HF 路径，可能再次出现权重名称冲突。  
2. **兼容性回退**：对已有的、同时包含 `params.json` 与 `config.json` 且不属于白名单的模型，行为保持不变，但如果用户自行在模型目录下添加了 `config.json`（例如自定义改动），可能导致意外使用 HF 格式而出现不兼容。  
3. **硬编码风险**：白名单使用硬编码字符串，若模型命名规则改变（如加入前缀/后缀），需要手动同步更新，否则会出现加载错误。  

**💡 关注建议**：  
- **代码层面**：将白名单抽取到配置文件或统一的常量模块，并提供文档说明，便于后续新增模型时维护。  
- **测试层面**：继续在 CI 中覆盖更多组合（如仅有 `params.json`、仅有 `config.json`、两者均有且不匹配的模型），确保判定函数的分支完整性。  
- **运营层面**：在发布说明中明确指出 Mistral‑Small‑4 及相关系列模型需要使用 `--trust-remote-code` 并保持原生权重格式，以防用户手动覆盖 `config.json`。  
- **安全层面**：提醒使用 `--trust-remote-code` 时的潜在风险，并在文档或 CLI 帮助中给出安全最佳实践（如仅从可信来源拉取模型）。

### [CPU] Implement MXFP4 Gemm kernels for intel AMX to support GPT OSS series. (#14385)
**SHA**: `af62bd9` | 🔗 [查看提交](https://github.com/sgl-project/sglang/commit/af62bd94868b60c4254c1c0b8b4fc3137900820b)

**🎯 变更类型**：功能增强 / 架构变更 / 性能优化  

**⚡ 重要程度**：🔴高  

**📋 变更摘要**  
本次提交为 Intel AMX 系列 CPU 添加了 **MXFP4（4‑bit 量化）** 的 GEMM 核实现，主体工作包括：  
1. 在公共调度宏中加入 `uint8_t`（对应 MXFP4 / INT4）类型的分支。  
2. 为 `uint8_t` 实现 VNNI‑2 打包、行大小计算、BRGEMM 可用性检测。  
3. 新增 **scale‑prepack**（`convert_scale_packed`）用于 MXFP4 的 per‑group 量化比例。  
4. 实现 MXFP4 → BF16 的向量化转换（LUT + 右移 + 加偏），并在 AVX‑512/AMX 路径中使用。  
5. 扩展 `tinygemm_kernel`、`brgemm`、`fp*_scaled_mm_cpu` 等模板，以统一支持 FP8 与 MXFP4 两类低精度。  
6. 在 Python‑C++ 接口层暴露 `convert_scale_packed` 与 `mxfp4_scaled_mm_cpu`。  

**🎯 影响范围**  
- **核心计算模块**：`sgl-kernel/csrc/cpu/*`（`gemm.cpp/h`, `gemm_fp8.cpp`, `torch_extension_cpu.cpp`, `vec.h`）。  
- **公共 API**：`sgl_kernel::convert_scale_packed`、`sgl_kernel::mxfp4_scaled_mm_cpu`。  
- **依赖的上层库**：只要使用 `sgl_kernel` 的项目在支持 AVX‑512/AMX 的 CPU 上即可受益。

| 维度 | 影响分析 |
|------|-----------|
| **架构影响** | - **新数据类型**：在调度宏 `CPU_DISPATCH_PACKED_TYPES` 中加入 `uint8_t`，实现了统一的类型分发路径。<br>- **行大小**：`get_row_size<uint8_t>` 返回 `K/2`（4‑bit → 8‑bit 包装），保持与 VNNI‑2 格式对齐。<br>- **BRGEMM 支持**：`can_use_brgemm<uint8_t>` 复用 FP8 条件，使得在块大小足够时自动走 `brgemm`（Intel AMX）路径。<br>- **Scale 预打包**：`convert_scale_packed` 将每组 32 个 K‑维度的比例从 `[N, K/G]` 重新排列为 `[E, NB, G, BLOCK_N]`，配合 `unpack_B` 中的向量化读取。 |

### [NPU] GLM-5 optimize with fused kernels (#18617)
**SHA**: `ba6d54d` | 🔗 [查看提交](https://github.com/sgl-project/sglang/commit/ba6d54d0f08f82f42b8224908ae2459a496b31b3)

**🎯 变更类型**：功能增强（NPU‑专用融合算子）+ 性能优化  
**⚡ 重要程度**：🟡 中  

**📋 变更摘要**  
1. 在 `model_config.py` 中加入对 **draft‑model** 的判定，避免在草稿模型目录读取 `quant_model_description.json`。  
2. 引入 NPU‑专用融合算子（`fused_split_qk_norm`、`fused_rope_qk_mqa`），在 `deepseek_v2_attention_mla_npu.py`、`nsa_indexer.py`、`rotary_embedding/base.py` 中使用，以一次性完成 Q、K、Norm、RoPE 等步骤，降低显存占用和 kernel 启动开销。  
3. 为 Deepseek‑NextN 添加 `rot.safetensors` 加载路径，并在 `forward` 前后通过环境变量切换 BF16/INT8 调度，以兼容没有量化配置的 NPU 场景。  

**🎯 影响范围**  
- **模型配置**：`ModelConfig._find_quant_modelslim_config`（草稿模型）  
- **注意力实现**：`deepseek_v2_attention_mla_npu.py`、`nsa_indexer.py`（NPU 多流、融合算子）  
- **旋转嵌入**：`rotary_embedding/base.py`（融合 RoPE）  
- **Deepseek‑NextN**：`deepseek_nextn.py`（rot 权重加载、环境变量）  
- 依赖 `sgl_kernel_npu`、`safetensors`  

**💡 关注建议**  

1. **兼容性**  
   - `is_draft_model` 只在配置阶段返回 `None`，请确认上层代码在 `None` 情况下能够安全回退（如 `quant_config` 为 `None`）。  
   - 融合路径依赖 `torch_npu` 与 `sgl_kernel_npu`，建议在 CI 中加入 **CPU/GPU** 以及 **NPU**（若可用）三套测试，防止未装库导致 ImportError。  

2. **阈值硬编码**  
   - `if fused_qkv_a_proj_out.shape[0] < 65535` 直接决定是否走融合算子，这一阈值在不同硬件/BatchSize 下可能不适用，建议迁移到可配置的 `env` 或在运行时根据 `torch_npu.get_device_capability()` 动态决定。  

3. **多流同步**  
   - 多流场景中使用 `wait_event`、`record_event`，但在 `nsa_indexer.py` 仍有 `self.alt_stream` 与主流的同步细节（如 `q_rope_event`）未在所有分支统一处理，需检查是否会出现 “event 未定义” 的路径。  

4. **缓存变量**  
   - `self.rotary_emb.sin_cos_cache` 在第一次使用时被创建，后续直接复用。若同一模型在多线程/多进程环境中并发调用，可能产生共享写冲突；建议在读取前加锁或使用 `torch.nn.Module` 的 `register_buffer` 固化。  

5. **环境变量副作用**  
   - 在 `DeepseekNextN.forward` 前后通过 `os.environ` 修改 `SGLANG_DEEPEP_BF16_DISPATCH`、`DEEP_NORMAL_MODE_USE_INT8_QUANT`，这会影响同一进程中其它模型的调度。若服务器同时运行多个模型，建议改为局部上下文（如 `torch.autocast`）或在 `ModelConfig` 中统一管理。  

6. **新增依赖**  
   - `safetensors` 的 `load_file` 被引入，确保 `requirements.txt` 已加入对应版本，并在没有 `rot.safetensors` 的情况下不会因路径不存在抛异常（已通过 `os.path.isfile` 检查，但仍需考虑权限问题）。  

7. **测试建议**  
   - 对比 **融合前后** 的数值误差（尤其是 RoPE 计算），确保在 BF16/FP16/INT8 三种精度下输出一致。  
   - 验证 `model_path` 为草稿模型时，量化配置确实被跳过且不影响后续推理。  
   - 对 `layer_id==0` 的特殊缓存更新路径进行回归，防止在 `prefill` 与 `decode` 混用时出现位置误差。  

总体来看，此次提交通过引入 NPU 融合算子显著降低了显存峰值与 kernel 启动次数，提升了 Deepseek‑V2 在华为 NPU 上的吞吐。但融合路径、环境变量的全局修改以及多流同步细节仍有潜在风险，建议在发布前完成上述兼容性和回归测试。
