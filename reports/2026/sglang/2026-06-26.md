### fix: batch BlockRemoved events per radix node (#29265)
**SHA**: `c0d4d18` | 🔗 [查看提交](https://github.com/sgl-project/sglang/commit/c0d4d1897c8228f220ac7342cde068b571d92a7a)

**🎯 变更类型**：Bug修复  
**⚡ 重要程度**：🔴高  
**📋 变更摘要**：将 KV 缓存的 `BlockRemoved` 事件从“每个块拆分为单独的事件”改为“同一 radix 节点下的所有块一次性批量上报”。同步更新了单元测试以校验 `block_hashes` 列表不再固定为单元素，且在一次 eviction 中仅产生一条 `BlockRemoved`。  
**🎯 影响范围**：  
- `python/sglang/srt/mem_cache/events.py`（核心事件生成逻辑）  
- KV‑Cache 相关的所有单元测试（`test_kv_events.py`, `test_mamba_unittest.py`, `test_radix_cache_unit.py`, `test_swa_unittest.py`, `test_unified_radix_cache_unittest.py`）  
- 依赖 `BlockRemoved` 事件的外部监控/日志收集模块（如果存在）  

**🔍 技术洞察**  
- **架构影响**：  
  - `BlockRemoved` 事件从“每页一个事件”改为“每个 radix 节点一个事件”，使事件模型更加贴合底层数据结构（radix node = 多页块的集合）。  
  - 事件队列中 `BlockRemoved` 的负载从 `block_hash: int64` → `block_hashes: List[int64]`，对消费方的解析逻辑产生了结构性变化。  
- **性能影响**：  
  - 生成事件的循环次数从 `页面数` 降至 `节点数`，在大规模 KV‑Cache 场景下可显著降低 Python 循环和对象创建开销。  
  - 事件队列长度大幅压缩，减少了后续 `take_events`、日志写入及网络传输的带宽占用。  
- **安全考虑**：  
  - 事件内容未涉及敏感信息，仅为块哈希值，安全风险保持不变。  
  - 改进后避免了因多次上报同一节点导致的“事件丢失/重复”潜在误判，提升了监控准确性。  

**⚠️ 潜在风险**  
1. **下游消费者兼容性**：若外部系统仅实现了“单 hash”处理（`event.block_hashes[0]`），会因列表长度扩大而出现索引错误或逻辑误判。  
2. **事件顺序假设**：原实现一次 `evict` 可能产生多条 `BlockRemoved`，顺序可对应各页面。现在合并为单条，若有业务依赖“逐页面处理”，需相应调整。  
3. **测试/文档同步**：未同步更新的文档或示例代码仍可能展示旧行为，导致使用者困惑。  

**💡 关注建议**  
- **兼容层**：在发布新版本时，可考虑保留一个向后兼容的包装（如在 `BlockRemoved` 中提供 `first_hash` 属性）或在发行说明中明确“事件结构已变更”。  
- **更新消费方**：审查项目内外所有对 `BlockRemoved` 的解析代码，确保改为遍历 `block_hashes` 列表。  
- **性能回归监控**：在 CI 中加入对事件队列大小的基准测试，确保合并后实际降低了内存/CPU 开销。  
- **文档补全**：在 `mem_cache` 模块的 README 或 API 文档中注明 `BlockRemoved.block_hashes` 为“该 radix 节点下所有被驱逐块的哈希”。  
- **回滚机制**：若出现不兼容报错，可临时通过 `enable_kv_cache_events` 开关禁用事件或在代码中加入 `if legacy_mode:` 逻辑，以便平滑迁移。

### [KDA-Pilot] Add diffusion causal Conv3D cat-pad CUDA fast path for Cosmos3 (#29281)
**SHA**: `5996b54` | 🔗 [查看提交](https://github.com/sgl-project/sglang/commit/5996b54bd3e8aca60bb417ba05f6266d1a9851a1)

**🎯 变更类型**：功能增强 / 性能优化 / 重构  
**⚡ 重要程度**：🔴高  

**📋 变更摘要**  
1. 新增 CUDA 本地 kernel `causal_conv3d_cat_pad.cuh`，实现 5‑维 NCTHW Tensor 的 cat + pad 复制/填零操作的向量化、内存绑定快速路径。  
2. 在 Python 层提供 `fused_causal_conv3d_cat_pad_cuda`，注册为自定义 op，支持 FP16/BF16/FP32，加入输入合法性检查并在不满足条件时回退至已有 Triton 实现。  
3. `parallel_conv.py` 通过统一接口 `fused_causal_conv3d_cat_pad` 自动选择 CUDA 快速路径或 Triton，实现一次性回退并记录错误日志。  
4. 将 Cosmos3 pipeline 中 `torch.compile` 的 `dynamic` 参数改为 `False`，避免在动态模式下触发不兼容的 JIT kernel。  
5. 添加针对新 kernel 的基准测试与单元测试，覆盖多种通道/尺寸/缓存组合，确保数值精度 0 误差。

**🎯 影响范围**  
- `sglang/jit_kernel/csrc/diffusion/causal_conv3d_cat_pad.cuh`（CUDA kernel）  
- `sglang/jit_kernel/diffusion/causal_conv3d_cat_pad.py`（Python 包装、注册）  
- `sglang/multimodal_gen/runtime/layers/parallel_conv.py`（调用入口、回退逻辑、日志）  
- `sglang/multimodal_gen/runtime/pipelines_core/stages/model_specific_stages/cosmos3.py`（torch.compile 参数）  
- 新增 benchmark 与 unit test 文件（不影响运行时，仅供 CI/性能评估）

**🔍 技术洞察**  

- **架构影响**  
  - 引入 **CUDA‑only** 的高效路径，避免 Triton JIT 编译开销，保持与现有 JIT‑kernel 框架一致的模块化设计。  
  - 通过 `register_custom_op` 将 kernel 暴露为统一的算子，兼容 TorchScript / torch.compile，提升整体可组合性。  
  - 在 `parallel_conv` 中加入全局失败标记 `_causal_conv3d_cat_pad_cuda_failed`，实现一次性降级，防止在同一进程中多次触发相同异常，提升稳健性。  

- **性能影响**  
  - **向量化加载**：`kVec = 16 / sizeof(T)`（FP16/BF16 → 8 元素；FP32 → 4 元素）配合 `uint4` 写回，提升内存带宽利用率。  
  - **块大小 256** 与 `grid = ceil(total_vecs / 256)` 设计，使得每个线程处理多个向量，减小调度开销。  
  - 仅在 **contiguous、对称 padding、深度左侧≥cache_t** 的常见场景激活，预计在大模型（如 Cosmos‑3 VAE）上的 **30%‑50%** 加速（依据 benchmark）。  
  - 保持 **内存对齐检查**（16‑字节）和 **元素计数对齐**，防止因未对齐导致的额外 memcpy。  

- **安全考虑**  
  - 所有入口均通过 `TensorMatcher` 与 `RuntimeCheck` 进行形状、dtype、设备、对齐、padding 合法性校验，防止越界或未对齐访问。  
  - kernel 本身仅执行 **只读** `src` 与 **写入** `out`，无原子操作、无共享写冲突，安全风险极低。  
  - 仍需注意 **CUDA 驱动/硬件异常**（如 OOM），但这属于通用 CUDA 使用风险。  

**⚠️ 潜在风险**  

| 风险点 | 说明 | 可能后果 |
|--------|------|----------|
| 输入不满足对称 padding 或 `pad_d_left < cache_t` 时自动回退 | 回退逻辑依赖 `can_use_fused_causal_conv3d_cat_pad_cuda` 的判断，若判断误差（如特殊 stride）可能导致不必要的 Triton 调用，性能下降。 | 预期加速未达成；在极端情况下，若 Triton 实现也不支持，抛出异常。 |
| 全局失败标记 `_causal_conv3d_cat_pad_cuda_failed` 未复位 | 若一次失败后标记永久置位，后续即使环境恢复（驱动升级、显存释放）仍只能使用 Triton。 | 长期性能回退，需手动重启进程或在 CI 中重启。 |
| 对齐要求 `out.data_ptr() % 16 == 0` | 某些自定义 Tensor 可能未满足 16‑字节对齐，导致 RuntimeCheck 直接 abort。 | 运行时异常，影响用户体验。 |
| `torch.compile` 动态模式关闭 | 将 `dynamic=False` 可能限制在 Torch‑compile 场景下对输入形状的灵活性，若后续模型需要动态大小，需重新评估。 | 编译后模型只能在固定形状下运行，增加部署限制。 |
| 编译器/驱动兼容性 | 使用 `__launch_bounds__(kBlockSize)` 与 `uint4` 需要 CUDA 11+，旧版驱动可能编译失败。 | CI/用户在旧环境中构建失败。 |

**💡 关注建议**  

1. **回退监控**：在 CI / 生产监控中加入 `_causal_conv3d_cat_pad_cuda_failed` 统计，若出现频繁回退，及时排查硬件/驱动或输入数据异常。  
2. **兼容性检查**：在项目的 `setup.py` 或 CI 脚本中明确 `CUDA >= 11.2`（或实际最低要求）并提示用户。  
3. **对齐容错**：可以在 Python 层对 `out` 进行自动 16‑字节对齐（如 `torch.empty(..., dtype=..., pin_memory=False).align_to(16)`），在不满足时抛出更友好的错误信息。  
4. **失败标记恢复**：提供一个公开的函数 `reset_causal_conv3d_cat_pad_cuda_status()`，供运维在显存清理后手动恢复 CUDA 路径。  
5. **文档与示例**：在 `README` 或 `docs/optimization.md` 中说明何种 padding、形状、contiguity 条件下会走 CUDA fast‑path，帮助用户自行验证。  
6. **进一步优化**：可在后续版本探索 **双向 fused kernel**（即一次完成 cat、pad、卷积），进一步削减内存拷贝次数。  
7. **持续基准**：保留 `benchmark` 脚本并在 nightly CI 中跑对比，确保新 kernel 在不同显卡（A100、H100、RTX 4090）上的相对提升不回退。

#### 架构影响  
1. **持久化环缓存层**：在每个 Mamba 槽（`num_slots = size + 1`）上为 **每层** 分配三个环缓冲（d、k、g），长度 `L = linear_replayssm_cache_len`。  
   - 这是一层 **状态层**（Stateful）缓存，位于 **MambaPool** 与 **HybridLinearAttnBackend** 之间，保持在 CUDA‑graph 捕获期间 **指针不变（静态张量）**，从而实现图安全的重复使用。  
2. **写指针统一管理**：`replayssm_write_pos` 为 **全局 per‑slot** 写指针，所有 Linear‑Attention 层共享同一指针，避免每层维护独立计数导致图捕获不一致。
