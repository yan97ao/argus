### fix(security): replace unsafe pickle.loads with SafeUnpickler for CVE-2026-3989 (#20904)
**SHA**: `5fc5c18` | 🔗 [查看提交](https://github.com/sgl-project/sglang/commit/5fc5c18bed4a4109082d9b14197ed33de1499231)

**🎯 变更类型**：安全修复 / Bug修复  

**⚡ 重要程度**：🔴高  

**📋 变更摘要**  
- 将项目中所有对 `pickle.loads()/pickle.load()` 的调用替换为安全的 `SafeUnpickler`，并在 `common.py` 中提供 `safe_pickle_load` 便利函数。  
- 更新文档提醒贡献者避免使用不安全的 pickle 反序列化，防止 CVE‑2026‑3989 相关的任意代码执行风险。  
- 仅在 `scripts/playground/replay_request_dump.py` 中调整为使用新函数，确保历史的 replay 数据仍可被安全读取。  

**🎯 影响范围**  
- `python/sglang/srt/utils/common.py`（新增 `safe_pickle_load`）  
- `scripts/playground/replay_request_dump.py`（读取 pickle 文件）  
- `docs/developer_guide/contribution_guide.md`（安全编码指南）  
- 任何未来使用 `pickle.load`/`pickle.loads` 或 `recv_pyobj` 的内部或外部代码（潜在受影响）  

**🔍 技术洞察**  
- **架构影响**：  
  - 引入 `SafeUnpickler` 作为反序列化的统一入口，形成“白名单”式的安全层，符合防御式编程原则。  
  - `safe_pickle_load` 为 drop‑in 替代，使迁移成本极低，保持原有函数签名，兼容现有调用方式。  
- **性能影响**：  
  - `SafeUnpickler` 在每次读取时会检查对象类型，較原生 `pickle` 有轻微的 CPU 开销（通常 <5%），对整体服务吞吐影响可忽略。  
  - 只在少数脚本（replay）中使用，且该路径本身 I/O 为瓶颈，性能劣化不明显。  
- **安全考虑**：  
  - 彻底消除通过 pickle 触发的任意代码执行风险，修复了 CVE‑2026‑3989。  
  - 文档层面的提醒进一步降低未来误用风险，符合 OWASP “不信任的数据序列化”最佳实践。  

**⚠️ 潜在风险**  
1. **兼容性**：如果项目中仍有业务代码依赖 pickle 反序列化自定义类（例如模型权重的临时缓存），`SafeUnpickler` 可能拒绝加载导致运行时异常。  
2. **导入路径**：`SafeUnpickler` 必须在 `common.py` 可见；若该类实现位置变更或未在运行时加载，会引发 `ImportError`。  
3. **误用**：开发者仍可手动使用 `pickle.load` 绕过安全层，文档提醒虽已加入，但缺乏代码层面的强制检查（如 lint rule）。  

**💡 关注建议**  
- **代码审计**：全局搜索 `pickle.load`、`pickle.loads`、`recv_pyobj`，确保全部改为 `safe_pickle_load` 或更安全的序列化方案（msgpack、JSON）。  
- **单元测试**：为 `safe_pickle_load` 增加正负用例，验证对合法对象的正常加载、对恶意构造的安全拦截。  
- **CI 检查**：在 CI 中加入 lint/安全扫描规则，禁止出现未受控的 `pickle` 反序列化调用。  
- **回滚方案**：如果出现兼容性异常，可提供临时开关（环境变量）恢复原始 `pickle.load`，但务必在生产环境中保持关闭。  
- **文档与培训**：将安全序列化改动写入开发者 onboarding 文档，举办内部安全编码培训，提升团队安全意识。

### [Diffusion] Add qknorm rope fuse kernel (#21440)
**SHA**: `d633ab7` | 🔗 [查看提交](https://github.com/sgl-project/sglang/commit/d633ab7349a8cdfe939d25745c3f647c508b8be5)

**🎯 变更类型**：功能增强、性能优化、重构、架构变更、测试与基准新增  

**⚡ 重要程度**：🔴高  

**📋 变更摘要**  
本次提交为 **sgl‑project/sglang** 引入了 Q‑K 归一化（QKNorm）与 Rotary Positional Embedding（RoPE）的 **融合 CUDA kernel**（`fused_inplace_qknorm_rope`），包括：  
1. 全新 Triton/JIT‑CUDA 实现的 `qknorm_rope.cuh` 与 Python 包装层 `qknorm_rope.py`。  
2. 自动检测硬件/参数是否支持的 `can_use_fused_inplace_qknorm_rope` 与注册的自定义 OP。  
3. 基准代码 `bench_qknorm_rope.py` 与单元/集成测试 `test_qknorm_rope.py`。  
4. 在核心层归一化模块 `layernorm.py` 中加入 **可选融合路径** (`apply_qk_norm_with_optional_rope`)；相应地，Flux、Flux‑2、Qwen‑Image、ZImage 等模型的前向实现改为调用新接口，以在满足条件时直接走融合 kernel。  

**🎯 影响范围**  
- **核心计算**：`sglang/jit_kernel/diffusion/qknorm_rope.*`、`sglang/jit_kernel/diffusion/triton/norm*`、`sglang/multimodal_gen/runtime/layers/layernorm.py`。  
- **模型实现**：`sglang/multimodal_gen/runtime/models/dits/flux.py`、`flux_2.py`、`qwen_image.py`、`zimage.py`。  
- **基准/测试**：`sglang/jit_kernel/benchmark/diffusion/bench_qknorm_rope.py`、`sglang/jit_kernel/tests/diffusion/test_qknorm_rope.py`。  
- **环境变量**：`SGLANG_ENABLE_FUSED_QKNORM_ROPE`（用于在运行时强制关闭融合）。

## 🔍 技术洞察

### 架构影响
| 维度 | 影响描述 |
|------|----------|
| **模块化** | 新增的 JIT‑CUDA 模块保持与原有 FlashInfer RoPE、独立 QKNorm 实现解耦，仅在 `apply_qk_norm_with_optional_rope` 中根据运行时条件切换。 |
| **接口统一** | 通过 `register_custom_op` 将 fused kernel 暴露为 PyTorch 自定义 OP，保持与原 `apply_qk_norm` 调用兼容（参数 `allow_inplace` 与 `is_neox` 等）。 |
| **依赖** | 引入 `sgl_kernel`（内部 CUDA 工具库）以及 `dlpack`，但这些已经是项目中既有的依赖，无新增外部库。 |
| **可选性** | 通过环境变量和 `can_use_fused_inplace_qknorm_rope` 检查，保证在不满足约束（e.g., head_dim∉{64,128,256}, 非 CUDA、dtype 非 fp16/bf16）时自动回退到原实现，兼容性好。 |
| **代码组织** | 新增的 benchmark 与测试文件位于 `jit_kernel/benchmark` 与 `jit_kernel/tests`，遵循现有目录结构；对模型代码的改动集中在几处 `apply_qk_norm` 替换，改动量可控。 |

### 性能影响
| 场景 | 预期收益 | 关键实现 |
|------|----------|----------|
| **大 batch / 长序列（Flux、Z‑Image 等）** | 通过一次 kernel 完成 Q‑K 归一化 + RoPE 计算，避免两次全局内存读写与一次额外 BF16 rounding，理论上可节省 **≈30%‑50%** 的 kernel 时间（实验基准中 `bench_qknorm_rope.py` 已给出对比曲线）。 |
| **SM‑占用率** | 采用 warp‑level并行（256 线程 / 4 warp）并使用 PDL（Persistent Data Loading）可在 H100 / A100 上达到接近 100% occupancy，提升吞吐。 |
| **内存带宽** | 只对 Q、K 各读取一次、写回一次，且在同一 warp 中完成归一化 + 旋转，显著降低 L2 / DRAM 带宽需求。 |
| **数值精度** | 采用 BF16/FP16 计算，融合路径多了一次乘法后立即进入 RoPE 计算，导致 **最多一次额外的 BF16 舍入**；单元测试已在多种 head_dim / rope_dim / NeoX 配置下验证误差在 `ATOL=8e‑2、RTOL=1e‑2` 范围内，符合实际推理容忍度。 |

### 安全考虑
- **越界风险**：`fused_qknorm_rope_warp` 中对 `positions` 进行索引时使用 `int64_t` 位置乘以 `kCosSinStrideBytes`；若 `positions` 超出 `MAX_SEQ_LEN`（131072），会读取越界。模型层已在 `apply_qk_norm_with_optional_rope` 中对 `positions` 长度做了校验，降低风险。  
- **类型检查**：`can_use_fused_inplace_qknorm_rope` 对 `head_dim、rope_dim、dtype` 做了严格检查，防止在不受支持的配置下加载 kernel，避免运行时崩溃。  
- **异常捕获**：加载 JIT 模块时捕获并记录异常，若失败则回退到原实现，不会导致进程直接退出。  
- **环境变量安全**：`SGLANG_ENABLE_FUSED_QKNORM_ROPE` 只能关闭功能，未引入额外的特权操作。

## ⚠️ 潜在风险

| 风险点 | 说明 | 可能的后果 | 缓解措施 |
|--------|------|------------|----------|
| **硬件兼容性** | 只在 CUDA GPU（支持 Warp‑level）且 `head_dim ≤256` 的情况下可用。旧显卡（如 T4）可能不满足 `kUsePDL` 或 `kWarpThreads` 对齐要求，导致 kernel launch 失败。 | 启动时抛出 `RuntimeError`，推理停机。 | `can_use_fused_inplace_qknorm_rope` 已做检测；在不满足时自动回退。可在 CI/CI‑CI 中加入显卡兼容性测试。 |
| **数值差异** | 融合路径相比分离实现少一次 BF16 round，导致微小数值偏差；在极端低‑precision（如对数梯度累积）场景可能放大误差。 | 可能影响模型微调或极端数值敏感的任务。 | 已在单元测试中使用 `ATOL/RTOL` 验证；如果业务对数值一致性要求极高，可通过环境变量关闭融合。 |
| **代码维护复杂度** | 新增的 C++/CUDA 文件与 Triton 代码混用，增加了维护门槛；若后续更改 `QKNorm` 或 `RoPE` 实现，需要同步更新 fused kernel。 | 未来出现 bug 难以定位，或性能回退。 | 建议在项目文档中记录 **融合约束**（head_dim、rope_dim、NeoX 条件），并编写对应的 **更新指南**；保持 `test_qknorm_rope` 覆盖所有分支。 |
| **PDL（Persistent Data Loading）依赖** | `PDLWaitPrimary`/`PDLTriggerSecondary` 只在特定平台（NVIDIA）有效，若在 ROCm 环境下编译但未适配，可能导致 deadlock。 | 运行时卡死或错误。 | 当前代码在 `#ifdef USE_ROCM` 中使用普通加载；仍需在 CI 中加入 ROCm 测试或在 CMake 中禁用 PDL。 |
| **环境变量误用** | 用户误将 `SGLANG_ENABLE_FUSED_QKNORM_ROPE=0` 与其它优化标记混用，可能导致不必要的性能下降。 | 性能不达预期，排查困难。 | 在日志中加入 `if not fused_enabled: logger.info("Fused QKNorm+RoPE disabled")`，帮助用户定位。 |

- **依赖库**：`sgl‑kernel`（提供通用工具、`tensor.h`、`utils.cuh` 等）

- **原有 AOT 实现**：`sgl‑kernel/csrc/elementwise/cast.cu`（常量改为宏）  
- **跨平台**：CUDA 与 ROCm（通过宏条件编译处理 FP8 最大值）

## ⚠️ 潜在风险

1. **平台兼容性**  
   - ROCm 环境下 `fp8_e4m3_t` 通过宏 `SGL_REGISTER_DTYPE_TRAIT` 条件编译，若 upstream `sgl_kernel` 未导出对应类型，可能导致链接错误。  
   - 部分老旧 CUDA GPU（< Ampere）不支持 FP8，JIT 加载会失败，需要在 Python 层捕获异常并回退。

2. **JIT 编译成本**  
   - 第一次调用会触发 nvcc 编译（或 TVM‑JIT 编译），在高并发启动时可能出现竞争或延迟。建议在服务启动阶段预热一次。

3. **数值一致性**  
   - JIT 使用 `static_cast<fp8_e4m3_t>(T)`（即 `cast<fp8_e4m3_t>`），而 AOT 曾使用手写的 `ConvertToFP8` 结构体。如果两者在极端值（如 NaN、Inf）处理上略有不同，可能导致微小差异。已通过单元测试覆盖大多数边界，但仍建议在实际推理数据上做一次端到端对比。

4. **线程配置假设**  
   - 代码假设 `head * dim` 能整除 `kVecSize`（=16/sizeof(T)），否则会出现未写入的残余元素。当前的模型配置（head≤32，dim≤256）满足，但自定义模型可能触发错误。可在 kernel 前添加断言或在 Python 层对 `head*dim % kVecSize` 做检查。

5. **多流并发**  
   - `LaunchKernel` 使用默认 stream 参数，若上层代码在不同 CUDA stream 中并行调用，可能出现同步问题。建议在包装层提供 `stream` 参数或在内部使用 `torch.cuda.current_stream()`。

4. **精度回归**：`rms` 参数改为 `const`，内部实现保持不变。请对比新旧实现的数值误差（尤其在 fp16 场景），确保 RMSNorm 结果在 1e‑5 以内保持一致。  

5. **测试覆盖**：  
   - 单元测试：对不同 batch、head、hidden‑size 组合跑两次（旧 kernel vs 新 kernel）比对输出。

总体来看，此次改动通过降低共享内存占用和合并归约路径，可提升显存利用率和调度效率，但需重点验证归约正确性和对极端 launch 参数的鲁棒性。通过上述回归与基准测试即可确保改动安全上线。

### [CI] Skip stage health check for release branch cut (#21488)
**SHA**: `17f43d1` | 🔗 [查看提交](https://github.com/sgl-project/sglang/commit/17f43d15187be710828a1ff6a4843fdddb0b1eb7)

**变更概述**  
本次 PR 为 CI 流程新增 `skip_stage_health_check` 开关，使在发布分支切割等场景下可以跳过 “stage health check” 的快速失败逻辑。实现方式是：①在 `check-stage-health` Action 中读取 `SKIP_STAGE_HEALTH_CHECK` 环境变量并提前返回；②在多个可复用 workflow（`pr-test.yml`、`pr-test‑jit‑kernel.yml`、`pr-test‑multimodal‑gen.yml`、`pr-test‑sgl‑kernel.yml`）中新增对应 input 并将其转化为同名 env；③在 `release-branch-cut.yml` 调用时默认打开该开关。

**影响范围**  
- CI 相关的 GitHub Actions：所有使用 `check-stage-health` 的工作流都会受到此开关的控制。  
- 仅在 `release-branch-cut` 流程中默认开启，其他 PR、定时任务仍保持原行为。  

**关注点 & 建议**  
1. **类型一致性**：`skip_stage_health_check` 在不同 workflow 中的声明既有 `boolean` 也有 `string`（如 `pr-test‑multimodal‑gen.yml`），建议统一为 `boolean`，避免因 CI 手动触发时出现类型不匹配。  
2. **环境变量取值**：Action 里判断 `process.env.SKIP_STAGE_HEALTH_CHECK === 'true'`，确保所有调用方都严格传入字符 `'true'`，否则会误判。可以考虑同时接受布尔 `true`（如 `process.env.SKIP_STAGE_HEALTH_CHECK == true`）提升容错。  
3. **文档补全**：在仓库的 CI/Workflow 文档里加入 `skip_stage_health_check` 的说明，明确仅在发布分支切割时使用，防止误用导致潜在缺陷被掩盖。  
4. **回归测试**：新增一个针对该开关的 CI 测例，验证在 `skip_stage_health_check=true` 时 `check-stage-health` 步骤被真正跳过；在 false 时仍正常执行。  
5. **安全审计**：确保该 env 只在受控流程中传递，避免在公共 PR 中被恶意设置跳过健康检查。  

总体来看，改动简洁且对现有功能影响可控，只需注意参数类型统一、文档说明以及相应的回归验证即可。
