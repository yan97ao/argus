### [Spec][Ngram] 6/N: Load an external corpus and construct a Suffix Automaton (#21425)
**SHA**: `12272b6` | 🔗 [查看提交](https://github.com/sgl-project/sglang/commit/12272b6791a3e5bd43cbd7c2787824d362ff6440)

**🎯 变更类型**：功能增强 / 架构变更 / 性能优化  

**⚡ 重要程度**：🔴 高  

**📋 变更摘要**  
- 为 N‑gram 预测（Speculative Decoding）加入 *外部语料库* 加载能力，基于 *后缀自动机（Suffix Automaton，SAM）* 进行只读查询。  
- 新增 C++ 实现的 SAM（`suffix_automaton.{h,cpp}`），以及在 `Ngram` 中的创建、追加、合并、清理接口。  
- 在 Python层提供 `load_external_corpus`、`iter_external_corpus_chunks` 等便利函数，支持 JSONL 文本文件或内存文档的流式 Token 化并分块加载。  
- 扩展 CLI 与配置 (`ServerArgs`)：`--speculative-ngram-external-corpus-path`、`--speculative-ngram-external-sam-budget`、`--speculative-ngram-external-corpus-max-tokens`，并在文档中说明默认行为。  
- 调整 `batch_match` 逻辑，使得在每次匹配时按 *SAM 预算* 与 *Trie 预算* 动态划分 draft token 预算，并在结果层面实现 `combineRootResults_`（根节点结果合并）。  
- 完善单元与集成测试，验证外部 SAM 的加载、预算限制、文档边界、分支合并等行为。

## 🔍 技术洞察

| 维度 | 影响分析 |
|------|----------|
| **架构影响** | 1. **模块化**：引入 **SuffixAutomaton** 作为独立只读子结构，`Ngram` 仍保留原有 **Trie**（可写）实现，实现了 *读写分离*。<br>2. **资源管理**：外部 SAM 通过 `std::unique_ptr` 持有，生命周期受 `Ngram` 控制，`clearExternalCorpus` 可在运行时释放内存，符合服务器长期运行的内存回收需求。<br>3. **参数扩展**：`Param` 中新增 `external_sam_budget` 与 `external_corpus_max_tokens`，向上兼容旧配置（默认 0 表示不使用外部 SAM）。<br>4. **接口兼容**：FFI 层保持原有方法签名，仅在 `__init__` 增加两个 `int64_t` 参数；对外 Python API 仍保持 `NgramCorpus` 同名方法，只是内部实现更丰富。 |
| **性能影响** | 1. **查询路径分流**：在 `batch_match` 中，根据 `sam_budget` 将 draft token 预算划分为 **SAM 部分** 与 **Trie 部分**，两者并行构建后合并。若 `sam_budget` 为 0，则保持原有 Trie 单一路径，几乎无额外开销。<br>2. **SAM 构建成本**：外部语料一次性 **append → finalize**，复杂度为 O(‖tokens‖)（线性），且在 `finalize` 时一次性 **propagateOccurrencesAndRecency_**，相当于一次性后缀自动机构建，成本与传统 Trie 相当或略低（不需要频繁的分支扩展）。<br>3. **运行时查询**：SAM 匹配采用状态跳转与后缀最大匹配，时间复杂度 O(depth)（depth ≤ `max_trie_depth`），与 Trie 的 BFS/DFS 相比在 **高频/最近** 分支上更快。<br>4. **预算分配**：外部 SAM 只占用 `external_sam_budget` 份 draft token，若预算较小（如 2~4），对整体搜索宽度影响有限；若预算接近 `draft_token_num-1`，则可能削弱 Trie 的探索深度，需要调参。 |

| **可维护性** | - **代码划分清晰**：SAM 相关文件独立，易于后续单独测试或迁移。<br>- **文档同步**：CLI 文档已更新，说明默认值随 n‑gram 参数变化，降低使用者困惑。<br>- **测试覆盖**：新增多种场景（文件流、内存文档、预算溢出、跨文档分割等）以及 Smoke 测试，保证回归安全。<br>- **接口向后兼容**：原有不使用 `external_*` 参数的调用不受影响，默认行为保持原有 Trie‑only 实现。 |

## ⚠️ 潜在风险

| 风险点 | 说明 | 缓解措施 |
|--------|------|----------|
| **SAM 预算配置不当** | 若 `external_sam_budget` 设得接近 `draft_token_num-1`，Trie 部分的预算可能为 0，导致失去 Trie 的多样性探索，生成质量可能下降。 | 推荐在文档中加入“预算分配最佳实践”，并在启动时打印 `sam_budget / total_draft_tokens` 比例供运维参考。 |
| **外部语料质量** | 外部 SAM 使用 **只读** 结构，若语料噪声多或与目标任务分布差异大，可能引入错误的候选路径。 | 提供 `external_corpus_max_tokens` 限制，鼓励在预处理阶段进行过滤；可在未来引入 **权重衰减**（如对老文档降低频率）机制。 |
| **内存峰值** | `appendTokens` 期间会临时存放完整的状态向量；大规模外部语料（接近 `external_corpus_max_tokens`) 仍会占用显著 CPU RAM。 | 参数 `external_corpus_max_tokens` 默认 10M tokens（≈ 40 MiB），可根据机器内存调低；在 `finalize` 前可监控 `states_.size()`，如超出阈值提前 abort。 |
| **并发加载** | 当前实现假设 **单线程** 调用 `startExternalCorpusLoad` → `append_external_corpus_tokens` → `finishExternalCorpusLoad`。若在多线程环境误用，可能导致竞争。 | 在 `Ngram` 内部已对所有新接口加 `std::unique_lock<std::mutex>`，确保线程安全；仍建议在文档中明确指出 *外部语料加载必须在服务器启动阶段一次性完成*。 |
| **FFI 参数顺序错误** | 新增 2 参数导致旧的 C++/Python 编译层面签名不匹配，可能在未重新编译的二进制上出现运行时错误。 | CI 已加入对对应编译文件的完整构建；对外发布的二进制包需同步更新 ABI 版本号。 |
| **文档/CLI 参数不一致** | 文档中对默认值`min(--speculative-ngram-max-trie-depth, 12)`的说明可能与实际代码的 `default` 不匹配。 | 已在 `docs/advanced_features/speculative_decoding.md` 中同步说明；建议在 CI 中加入文档与实际默认值的对比检查。 |

### [Spec][Ngram] 5/N: Store and advance anchor match state across decode steps (#21243)
**SHA**: `b2008bf` | 🔗 [查看提交](https://github.com/sgl-project/sglang/commit/b2008bf9e0821cc5d466bbce1b4ab87417944823)

**⚡ 重要程度**：🔴 高  
**📋 变更摘要**：  
1. 在 Ngram Trie 中引入 `MatchState`、`NodeRef` 与 `trie_epoch`，实现跨解码步的锚点（suffix）匹配状态保存与增量推进。

3. 调整 speculative decoding工作流，使其在请求结束后及时清理匹配状态，新增大量单元测试验证增量匹配的正确性。  

**🎯 影响范围**：

### 🔍 技术洞察

| 维度 | 影响分析 |
|------|----------|
| **架构影响** | - 新增 `MatchState` 结构体，作为每个请求的缓存锚点集合，存放在 `Ngram` 的 `std::unordered_map<int64_t, MatchState> match_state_` 中。<br>- 为防止节点被回收后仍被缓存引用，引入 `NodeRef`（指针 + `version`）以及 `Trie::trie_epoch_`，在 `Trie::reset()` 与 `Trie::squeeze()` 时递增，确保旧状态失效后能够自动重建。<br>- 通过 `Trie::match` 的重载实现 **增量推进**（`advanceMatchState_`）和 **全量重建**（`rebuildMatchState_`），两者在 `buildRecency/Probability` 中统一调用。<br>- FFI 层同步更新，保持 C++ 与 Python 之间的状态统一。 |
| **性能影响** | - 之前的 `batchMatch` 每次都对完整的 suffix 重新遍历 Trie（O(depth) × batchSize)。<br>- 新实现若 `state` 仍对应上一次请求的 `total_len`，仅对新追加的 token 逐个前向推进（O(newTokens)），显著降低解码阶段的 CPU 开销，尤其在长上下文或高并发批处理时。<br>- 额外的 `match_state_` 哈希查找成本极低（O(1)），且持有的 `MatchState` 体积仅与 `max_trie_depth` 成线性关系。 |
| **安全考虑** | - 公开的 `state_ids` 通过整数（int64）传递，若上层未及时回收会导致 **状态泄漏**（内存占用增长）。<br>- `MatchState` 只在单一 `mutex_` 保护下操作，未出现跨线程读写冲突，但若业务自行在多线程外部持有 `state_ids` 并并发调用 `batch_match_stateful`，仍依赖内部锁确保安全。<br>- `NodeRef.version` 为 64 位递增，理论上不可能回环；即使出现异常（如节点被大量回收后版本溢出），系统仍会在 `resolve` 阶段检测到不匹配并重新构建，保持安全性。 |
| **可维护性** | - 代码结构更加模块化：<br>  * `Trie::match`、`rebuildMatchState_`、`advanceMatchState_`、`validateMatchState_`、`getExpandableAnchors_` 分离职责。<br>  * `NodeRef` 与 `MatchState` 的抽象避免直接在业务层使用裸指针。<br>- 通过新增单元测试覆盖 **增量 vs. 全量**、**状态失效**、**节点淘汰** 场景，提升回归安全性。<br>- 仍保留原有 `batchMatch(const vector<vector<int32_t>>&)` 接口，向后兼容，降低迁移风险。 |

### ⚠️ 潜在风险

1. **状态泄漏**：如果调用方忘记在请求结束后调用 `erase_match_state`，`match_state_` 将永久占用内存。  
2. **并发竞争**：虽然内部已加 `std::mutex`，但 `match_state_` 的键值对在高并发创建/删除时可能导致锁竞争，潜在的吞吐下降。  
3. **错误的 `total_len`**：`batch_match_stateful` 要求 `total_lens[i]` 与实际已生成 token 长度一致；若传入错误值，增量推进会被误判为不可前进，导致频繁全量重建，性能回退。  
4. **节点版本溢出**：`node->version` 为 `uint64_t`，极端情况下（节点回收频率极高）可能在 2^64 次后回环，虽然实际不大可能，但理论上会导致旧 `NodeRef` 被错误复用。  
5. **API 兼容性**：新接口改变了 FFI 参数顺序和类型，若第三方直接使用旧的 C ABI，需重新编译或更新绑定。  

---

### 💡 关注建议

1. **生命周期管理**：在 `NgramWorker`（或等价调度器）中确保每个 `req_id` 在请求结束、撤回或超时后必调用 `erase_match_state`，可考虑在 `Request` 的析构或统一的 “请求完成” 回调里统一清理。  
2. **监控 & 限额**：加入运行时监控 `match_state_` 的大小（可以在 `Ngram::reset`、`eraseMatchState` 中记录），若超过阈值触发告警或自动清理最久未使用的状态。  
3. **并发优化**：如果批次规模极大，可评估把 `match_state_` 改为分片锁（sharded mutex）或使用 `absl::flat_hash_map` 等更高并发友好的容器。  
4. **参数校验**：在 `batch_match_stateful` 入口添加 `total_len >= tokens[i].size()` 的断言，防止错误调用导致隐蔽的性能回退。  
5. **回退方案**：保持旧的 `batchMatch` 接口不变，外部若不想使用状态化匹配，可继续使用原接口，防止突发性回归。  
6. **文档更新**：在项目 README / API 文档中说明 `state_id` 的生成、回收机制以及推荐的使用模式（如每个请求对应唯一 `state_id`，并在结束时调用 `erase_match_state`）。  
7. **持续集成**：把新增的增量匹配测试加入 CI，确保在不同 `match_type`、不同 `max_trie_depth`、不同 `capacity` 下均能通过。  

> 综合来看，此次改动通过在 Trie 层引入状态缓存与增量推进，大幅降低了逐步解码时的匹配开销，提升了推理吞吐与延迟；同时也引入了状态管理
