### [CPU] Add FP8 Bmm support (#9744)
**SHA**: `cd22aa2` | 🔗 [查看提交](https://github.com/sgl-project/sglang/commit/cd22aa27a941d10ecd3df1e2df69faa84454b443)

**🎯 变更类型**：功能增强 / 架构变更 / 性能优化  
**⚡ 重要程度**：🔴高  

**📋 变更摘要**  
本次 PR 为 CPU 端引入原生 FP8（float8_e4m3fn）权重量化的 BMM（batch‑matrix‑multiply）支持。  
- 在 `sgl-kernel` 中实现了专门的 FP8‑W8A16 kernel（`bmm_cpu` + `tinygemm_kernel` 系列），支持 `scale` 参数进行后量化。  
- 上层模型代码（`bailing_moe_linear.py、deepseek_common/*、longcat_*`）去掉了之前的 “TODO: remove after adding FP8 support” hack，改为在 CPU 路径下直接使用 `torch.bmm`（BF16）或新的 FP8 kernel。  
- 新增 `w_scale` 参数向下传递，以统一 FP8 权重的标定逻辑。  
- 扩充了单元测试 `test_bmm.py`，覆盖 BF16 与 FP8 两种精度、不同 batch/shape/分块情况，确保数值一致性。  

**🎯 影响范围**  
- **核心计算库**：`sgl-kernel/csrc/cpu/*`（bmm、gemm、qkv_proj）  
- **模型层实现**：`sglang/srt/models/*`（DeepSeek、LongChat、Bailing MoE）  
- **权重加载与后处理**：`deepseek_weight_loader.py`、`bailing_moe_linear.py` 等  
- **测试套件**：CPU 测试目录新增 `test_bmm.py` 并更新运行列表  

---

### 🔍 技术洞察

| 维度 | 影响分析 |
|------|----------|
| **架构影响** | • 新增了 FP8‑W8A16 专用的 C++ kernel（`bmm_kernel_impl`、`tinygemm_kernel` 以及 `brgemm2`），并在 Python 层通过 `torch.ops.sgl_kernel.bmm_cpu` 统一调用。<br>• 通过 `scale.has_value()` 判断是否走 FP8 路径，保持了对原 BF16 路径的兼容。<br>• `w_scale` 参数在模型前向传播中显式传递，消除了之前在 `post_load_weights` 中的隐式强制转换。 |
| **性能影响** | • 在支持 AVX‑512/AMX 的 CPU 上，FP8‑W8A16 能显著降低带宽占用（8 bit → 16 bit）并利用 `brgemm`/`tinygemm` 加速矩阵乘。<br>• 通过 `copy_mul_stub` 实现了批量乘以标定因子，避免了逐元素回到 Python 的开销。<br>• 对于不支持 AVX‑512 的机器仍会回退到 BF16 `torch.bmm`，不会降级性能。 |
| **安全考虑** | • 新增的 `scale` 参数必须是 0‑维标量 Tensor；若传入错误形状会触发 `TORCH_CHECK`，防止内存越界。<br>• 所有新的 C++ 分支均使用 `TORCH_CHECK` 检查形状匹配，降低内存泄漏/越界风险。<br>• 没有引入网络 I/O 或系统调用，安全风险基本可控。 |
| **可维护性** | • 将原先散落在多个模型文件中的 “TODO” 代码统一迁移到底层 kernel，代码重复度下降。<br>• 新增 `w_scale` 参数后，模型前向签名略有变化，需同步更新调用方（已在 `test_qkv_proj_with_rope.py` 中完成）。<br>• `tinygemm_kernel` 重载了两套模板（普通 BF16 与 FP8），代码量增加但结构化良好，便于后续扩展。 |

---

### ⚠️ 潜在风险

1. **CPU 指令集限制**  
   - 新的 FP8 kernel 依赖 AVX‑512 (`CVT_FP8_TO_BF16`、`_mm512_*`)。在没有 AVX‑512/AMX 支持的机器上会触发 `TORCH_CHECK(false, "unpack_B: scalar path not implemented!")`。  
   - 需要在发布前确认 `can_use_brgemm` 与 `CPU_CAPABILITY_AVX512` 能正确检测，并提供回退路径（目前回退到 BF16 `torch.bmm`，但 `bmm_cpu` 仍会尝试读取 `scale`，需确保 `scale` 为 `None`）。  

2. **标定因子 (`scale`) 精度**  
   - `scale` 以 `float` 存储并在 C++ 中乘以 `float32` 再写回 BF16。极端情况下（scale 过大/过小）可能导致溢出或下溢，导致数值误差明显。建议在模型转换阶段对 `scale` 做范围校验。  

3. **包装层兼容性**  
   - 许多旧模型或第三方插件可能仍使用旧的 `post_load_weights` 逻辑（隐式 `to(bfloat16)`），导致 double‑scale（一次在 `post_load_weights`，一次在 `bmm_cpu`）的数值偏差。需要在文档或迁移指南中提醒升级。  

4. **测试覆盖不足**  
   - 目前单元测试覆盖了常规 shape 与 chunk 两种情况，但未覆盖 **非 AVX‑512** 环境、**大 batch**（>16）以及 **不同块大小**（如自定义 `BLOCK_M/N/K`）的路径。CI 需要在多种硬件上跑一次。  

---

### 💡 关注建议

| 对象 | 建议 |
|------|------|
| **开发者** | 1. 在 `setup.py`/`CMakeLists.txt` 中明确声明 AVX‑512/AMX 为可选特性，编译时自动检测并在 `sgl_kernel` 中导出 `has_fp8_cpu` 标志。<br>2. 在模型权重转换脚本（`deepseek_weight_loader.py` 等）加入 `assert w.dtype != torch.float8 || w_scale.is_finite()` 之类的校验。 |
| **运维 / CI** | 1. 在 CI 中加入两套机器配置：<ul><li>支持 AVX‑512 的机器（执行 FP8 路径）</li><li>不支持的机器（确保回退到 BF16 且不 crash）</li></ul> <br>2. 将 `test_bmm.py` 加入性能基准测试，记录 BF16 ↔ FP8 的吞吐提升。 |
| **用户** | 1. 在使用 FP8 权重时，务必提供对应的 `w_scale`（可通过 `input_to_float8` 获得），否则会回退到不做 scaling 的错误结果。<br>2. 对于推理部署，建议在硬件支持的前提下打开 `use_fp8_w8a16` 开关，以获取显存与算力双重优势。 |
| **文档** | 更新 API 文档：`bmm_cpu(tensor_out, mat1, mat2, is_vnni, scale=None)` 中 `scale` 参数的含义、类型、取值范围，以及 `w_scale` 在模型层中的作用。 |

---

**结论**  
此次 PR 为 CPU 引入了原生 FP8‑W8A16 矩阵乘实现，消除了之前的临时转换 hack，提升了显存利用率和算子效率。只要注意指令集兼容、scale 取值以及相应的回退逻辑，整体风险可控，建议尽快合入主线并在多平台 CI 中验证回退路径的稳定性。

### feat(mm)(grpc): compute M-RoPE positions for preprocessed VL inputs (#19973)
**SHA**: `0d23a46` | 🔗 [查看提交](https://github.com/sgl-project/sglang/commit/0d23a461a04a8c5e94d3d8a6250a0275d9678921)

**🎯 变更类型**：功能增强  
**⚡ 重要程度**：🔴高  
**📋 变更摘要**：  
在调度器 `scheduler.py` 中引入多模态处理器的加载与 M‑RoPE 位置计算的后备逻辑；为所有多模态处理器基类与具体实现（Ernie45‑VL、GLM4V、Qwen‑VL）添加 `compute_mrope_positions` 接口，使得在 gRPC 预处理路径缺失 RoPE 位置信息时能够自动补全，从而保证多模态模型的推理正确性。  

**🎯 影响范围**：  
- `sglang/srt/managers/scheduler.py`（调度器核心）  
- `sglang/srt/multimodal/processors/base_processor.py`（基类）  
- `sglang/srt/multimodal/processors/ernie45_vl.py`、`glm4v.py`、`qwen_vl.py`（具体多模态处理器实现）  
- 受影响的模型：所有使用 M‑RoPE 的多模态模型（Ernie45‑VL、GLM4V、Qwen‑VL 等）  

**🔍 技术洞察**：  
- **架构影响**：  
  - 调度器在初始化阶段新增对多模态处理器的可选加载（`import_processors`、`get_mm_processor`），并在每条请求的图像/视频输入扩展后调用 `_maybe_compute_mrope_positions`。  
  - 通过基类统一的 `compute_mrope_positions` 接口，实现了“后备计算”与“按需实现”两层解耦，保持原有处理器实现不受侵入性影响。  
  - 新增的 `_mm_processor` 成员为 `None` 时仍可正常工作，兼容不需要 M‑RoPE 的模型。  

- **性能影响**：  
  - **启动时**：首次调用 `import_processors` 与 `get_mm_processor` 会导致额外的模块导入与配置解析，略增启动延迟（毫秒级），对大多数部署影响可忽略。  
  - **运行时**：仅在请求缺少 `mrope_positions` 时触发一次 Tensor 转换与 `MRotaryEmbedding` 的计算，计算成本与原有 Tokenizer/Embedding 过程相当，整体延迟增长预计 < 5 ms（具体取决于模型大小与 GPU/CPU 环境）。  
  - **内存**：额外创建的 `torch.tensor`（`input_ids_tensor`）在函数返回后会被释放，对长期内存占用影响甚微。  

- **安全考虑**：  
  - 新增的代码仅涉及模型内部的张量运算和配置读取，无外部 I/O、网络或文件系统访问，不引入新的攻击面。  
  - `try/except` 包裹的处理器加载逻辑防止因加载失败导致服务崩溃，安全风险极低。  

**⚠️ 潜在风险**：  
1. **处理器实现遗漏**：如果未来新增的多模态模型未实现 `compute_mrope_positions`，调度器仍会调用基类默认返回 `None`，导致 M‑RoPE 位置信息缺失，可能影响模型输出的准确性。  
2. **异常遮蔽**：加载处理器异常仅记录警告，若因路径错误或依赖缺失而未成功加载，用户可能不易察觉 M‑RoPE 回退不可用。  
3. **线程安全**：`_mm_processor` 在调度器的多线程环境中被共享使用，当前实现未显式加锁；若处理器内部持有可变状态（如缓存），可能出现竞态。  
4. **依赖版本**：`torch` 被延迟导入，仅在具体实现中使用，如果部署环境的 `torch` 版本与模型要求不匹配，可能在运行时抛出异常。  

**💡 关注建议**：  
- **单元/集成测试**：为每个已实现的处理器编写包含缺失 `mrope_positions` 场景的测试，确保回退路径返回正确的坐标与 delta。  
- **监控与告警**：在服务器启动日志中加入 `M‑RoPE fallback enabled/disabled` 标记；若出现 `Failed to load multimodal processor` 警告，可触发运维告警。  
- **文档更新**：在多模态模型使用说明中明确指出：在 gRPC 预处理路径下，若模型依赖 M‑RoPE，需确保对应处理器实现了 `compute_mrope_positions`。  
- **线程安全审查**：确认各处理器的 `MRotaryEmbedding` 方法为纯函数或在内部做了适当的同步；如有状态缓存，考虑在调度器层面使用 `threading.Lock` 包装调用。  
- **依赖管理**：在发布包的 `requirements.txt` 中锁定兼容的 `torch` 版本，并在 CI 中加入兼容性检查。  

通过以上检查与实践，可将新功能的价值最大化，同时将潜在的可靠性风险控制在可接受范围内。

### Upgrade transformers==5.3.0 (#17784)
**SHA**: `d1e95af` | 🔗 [查看提交](https://github.com/sgl-project/sglang/commit/d1e95af282068a751ee6a8b60a0ae2a90c464819)

**⚡ 重要程度**：🔴 高  

**📋 变更摘要**  
- 将项目的 `transformers` 依赖从 **4.57.1** 升级到 **5.3.0**，并引入 `mistral_common>=1.9.0`。  
- 为适配 v5 中 **rope 参数结构、tokenizer/processor 重构、hf_transfer 删除、加速库 (accelerate) 挂钩、gguf 兼容性等**，在 `sglang` 的模型实现、加载器、工具函数以及测试用例中大幅补丁：  
  - 统一使用 `config.rope_parameters`（或回退至旧字段）获取 `rope_theta`、`rope_scaling`。  
  - 重写 `get_rope_config`、`compute_mla_mscale_scaling`、`_get_rope_param` 等统一处理 **RoPE**。  
  - 补丁 `AutoTokenizer`、`AutoProcessor` 的默认行为：恢复 `clean_up_tokenization`、`is_torch_fx_available`、`special_tokens_pattern`、BOS/EOS 标记、`added_tokens` 的单 token 编码等。

| **测试覆盖** | 当前测试只跑在单卡、`block_size=16` 场景。建议补充：<br>• 多卡（`torch.distributed`）下的 `fully_sharded=True` 分支；<br>• `split_k>1`（原子加）路径；<br>• 非 SM90 机器的 fallback 行为。 |
| **代码可维护性** | 文件长度 690 行，逻辑交叉较多。建议把 kernel 参数解析、grid 计算、shrink/expand 包装拆分到独立模块或加入详细 docstring，方便后续重构。 |
| **CI 费用** | `register_cuda_ci(est_time=120)` 预估 2 分钟。若 CI 频繁触发，考虑将此测试设为可选标记（e.g., `@pytest.mark.slow_cuda`）并在 nightly 或 perf CI 中运行。 |

**结论**  
本次合并为 LoRA‑MOE 场景提供了高效的 fused Triton 实现，理论上能显著降低两次 GEMM 的内存搬移和调度开销。风险主要来自硬件兼容、指针缓存以及未充分测试的分布式路径。按上述建议完善检测、文档和测试后，即可在生产环境安全开启。

- 新增下载 digest、拼装多架构 manifest（`docker buildx imagetools create …@${DIGEST}`）的步骤，统一了 `release-docker*`、`release-docker-dev.yml`、`release-docker-cu13-framework.yml` 三个工作流。

**🎯 影响范围**：  
- `.github/workflows/release-docker.yml`（主发布 CI）  
- `.github/workflows/release-docker-dev.yml`（dev 镜像 CI）  
- `.github/workflows/release-docker-cu13-framework.yml`（CUDA‑13 框架 CI）  
- 关联的 Docker 镜像命名、标签策略以及 downstream 使用这些镜像的文档/脚本。

**💡 关注建议**  
1. **兼容性检查**：外部用户仍可能依据旧的 `*-amd64`、`*-arm64` tag 拉取镜像。发布后需在 README 或文档中明确这类 tag 已不再维护，推荐使用统一标签（如 `vX.YZ-cu130`）或 `@digest`。  
2. **CI 稳定性**：`metadata.json` 的路径写死为 `/tmp/metadata*.json`，确保所有 runner 都有写权限并且不被并发任务冲突。可考虑使用 `${{ runner.temp }}` 生成唯一目录。  
3. **安全审计**：`push-by-digest=true` 会自动推送未标记的临时镜像，确保仓库的 Docker Hub 凭证只在受信任的 runner 上使用，避免泄漏。  
4. **回滚方案**：若新的 manifest 生成出现错误（如 digest 读取失败），建议保留旧的 per‑arch tag 推送路径（注释形式），方便快速回滚。  
5. **测试覆盖**：在本地或 CI 中加入一步 `docker pull lmsysorg/sglang@${DIGEST}` 验证镜像可被成功拉取，防止因 `metadata.json` 结构变化导致 digest 提取失败。  

总体而言，此次改动显著简化了镜像管理，降低了 tag 碎片化风险，但需同步更新文档并确保 CI 对 digest 的读取与上传流程可靠。

### Fix streaming token ids data loss under load (#19977)
**SHA**: `b9dba85` | 🔗 [查看提交](https://github.com/sgl-project/sglang/commit/b9dba851a0853e1ec8ea51afd4270c61f51364dc)

**🎯 变更类型**：Bug 修复  
**⚡ 重要程度**：🟡 中  
**📋 变更摘要**：在高并发下，流式返回时会出现 token‑id 数据丢失。此次改动把对 `state.out_list` 的处理改为一次性取出全部 pending 输出（流式则全部切片，非流式只取最新），并在循环中逐块处理、记录时间戳、上报指标、以及 abort/错误判断，确保每个 chunk 都被正确 yield，防止漏发。  

**🎯 影响范围**  
- `python/sglang/srt/managers/tokenizer_manager.py`（核心流式响应调度）  
- 关联的请求日志、指标导出以及 LoRA 资源释放逻辑  
- 与 `state.time_stats`、`request_logger`、`request_metrics_exporter_manager` 的交互  

**💡 关注建议**  
1. **并发测试**：在负载测试中确认所有 token‑id 均能被完整返回，尤其是长文本的多块流。  
2. **内存与延迟**：在 backlog 过大时会记录 warning，关注是否导致 P99 TBT 异常升高；如有必要在 upstream 加入背压或限流。  
3. **时间戳逻辑**：`response_sent_to_client_time` 现在在每个 chunk（流式）或最终 chunk（非流式）首次发送前写入，确保监控数据准确。  
4. **异常路径**：保持原有 abort 处理不变，验证 BAD_REQUEST、SERVICE_UNAVAILABLE、INTERNAL_SERVER_ERROR 场景仍能抛出对应异常或正确 yield。  
5. **代码可读性**：循环内部逻辑稍显嵌套，建议后续抽成小函数（如 `_handle_finish`, `_maybe_warn_backlog`）以提升可维护性。  

总体来说，此次改动有效修复了流式 token 丢失问题，对现有功能影响有限，但需通过高并发场景的回归测试确保新逻辑的稳健性。
