### Piecewise Cuda Graph set default (#16331)
**SHA**: `c64274c` | 🔗 [查看提交](https://github.com/sgl-project/sglang/commit/c64274c746f210497c45bd1600d2046faf7f4b10)

**🎯 变更类型**：功能增强 / 架构变更 / 重构  
**⚡ 重要程度**：🔴高  

**📋 变更摘要**  
1. 将 *Piecewise CUDA Graph*（PCG）功能由显式 **开启**（`--enable-piecewise-cuda-graph`）改为默认 **开启**，并新增 `--disable-piecewise-cuda-graph`（关闭） 与 `--enforce-piecewise-cuda-graph`（强制开启） 两个 CLI 选项。  
2. 在 `ServerArgs` 中实现一系列 **自动禁用** 条件（模型架构、torch‑compile、PP、DP‑Attention、MoE‑A2A、LoRA、VLM、GGUF、CPU‑offload、确定性推理、对称内存等），统一由 `_handle_piecewise_cuda_graph` 处理。  
3. 大幅改造 PCG 相关代码：删除原先的 `contextvars` 控制，改用 `piecewise_context_manager.is_in_piecewise_cuda_graph`；统一在 `model_runner`、`piecewise_cuda_graph_runner`、层实现等处使用 `disable_piecewise_cuda_graph` 判断。  
4. 为外部库（flashinfer、sgl‑kernel）中不兼容 `torch.compile` 的 kernel 添加 **custom op** 包装器 (`register_custom_op`、`register_custom_op_from_extern`) 并提供 fake 实现，以避免 Dynamo 追踪进入 JIT 编译路径。  
5. 大量单元测试/手动测试脚本同步更新为新 flag（删去 `--enable-piecewise-cuda-graph`，改为 `--enforce-piecewise-cuda-graph` 或默认）。  

---

### 🎯 影响范围  
- **核心运行时**：`sglang/srt/compilation/*`, `sglang/srt/model_executor/*`, `sglang/srt/layers/*`  
- **配置/CLI**：`sglang/srt/server_args.py`, `sglang/srt/utils/custom_op.py`  
- **模型特定实现**：DeepSeek V2/V3, MiniCPM3, Qwen2‑MOE, Qwen3, Minimax‑M2, etc.  
- **测试套件**：全部 `test/*` 中涉及 PCG 的脚本与基准测试。  

---

### 🔍 技术洞察  

| 维度 | 分析 |
|------|------|
| **架构影响** | - **统一开关**：将 PCG 设为默认开启后，运行时几乎所有路径都假设在 CUDA Graph 环境中，需要在每个层/模块查询 `disable_piecewise_cuda_graph`，导致代码路径稍微复杂化。<br>- **自动禁用逻辑**：通过 `ServerArgs._handle_piecewise_cuda_graph` 将多种不兼容场景一次性屏蔽，提升可维护性，但也把业务决策（如是否支持 LoRA）硬编码在配置层，可能对特殊需求的用户构成使用障碍。<br>- **自定义 Op 框架**：引入 `custom_op` 包装，使外部 kernel 成为黑盒节点，避免 `torch.compile` 追踪，这在整体架构上是对 **torch.compile** 与 **CUDA Graph** 兼容性的抽象层。|
| **性能影响** | - **默认开启**：在支持的模型/硬件上，PCG 能显著降低 kernel launch 开销、提升预填充/扩展阶段的吞吐率，尤其在多卡 TP（`tp>1`）或大 batch 场景。<br>- **内存预留**：`ServerArgs._handle_gpu_memory_settings` 对 PCG 捕获的 token 列表计入 `reserved_mem`，并在 Llama‑2 系列等模型上对 token 上限做了硬上限限制，防止 OOM。<br>- **自定义 Op Fake 实现**：提供轻量的 shape‑only 假实现，避免在编译阶段执行真实 kernel，从而加速 `torch.compile` 的 tracing与 S‑graph 生成。<br>- **潜在回退**：在自动禁用条件触发时会回退到普通执行，性能会下降到原始基线。|
| **安全考虑** | - 代码中对异常捕获（`enable_piecewise_cuda_graph` context）加入日志记录，防止捕获失败导致服务崩溃。<br>- 新增 CLI 参数 `--disable-piecewise-cuda-graph` 与 `--enforce-piecewise-cuda-graph` 均为 **bool**，不存在注入风险。<br>- 对外部 Op 的包装没有改变底层算子本身的安全性，仅是 **调度层面** 的封装。 |
| **可维护性** | - 大量 `if not get_global_server_args().disable_piecewise_cuda_graph` 替换原来的 `enable_*` 判定，统一了语义，代码阅读更直观。<br>- `custom_op` 实现抽象化后，后续若有新外部 kernel 需要兼容 `torch.compile`，只需在 `custom_op.py` 中注册即可，降低重复代码。<br>- 自动禁用规则集中在 `_handle_piecewise_cuda_graph`，便于未来增删条件；但该函数长度较大，建议后期拆分为若干子检查函数。 |

---

### ⚠️ 潜在风险  

1. **误禁用**：自动禁用逻辑覆盖面广，若用户在特定场景（例如 LoRA + PCG）真的需要 PCG，必须显式使用 `--enforce-piecewise-cuda-graph`，否则会被静默关闭，导致性能下降。  
2. **兼容性回归**：对大量模型（尤其是 MoE、DeepSeek 系列）加入了 “不支持 PCG” 判定，若这些模型的实现更新后能够兼容 PCG，仍会被错误禁用。  
3. **自定义 Op 假实现**：Fake 实现仅返回空张量，若后续新增的外部 kernel 在编译阶段需要真实计算（如产生副作用），可能导致编译后行为不一致。  
4. **内存预留误差**：`reserved_mem += len(self.piecewise_cuda_graph_tokens) * 8` 等硬编码的内存预留未经精确测算，极端配置下可能仍触发 OOM。  
5. **日志噪声**：在捕获失败时会打印长堆栈，可能淹没真实错误信息；同时 `enable_piecewise_cuda_graph_compile` 中捕获异常后重新抛出，可能导致上层捕获不到特定错误类型。  

---

### 💡 关注建议  

1. **文档与示例**：在官方文档中明确说明 PCG 默认为开启，列出 **全部自动禁用条件**，并提供 `--enforce-piecewise-cuda-graph` 的使用场景示例。  
2. **配置可视化**：在服务器启动日志中输出 `disable_piecewise_cuda_graph` 的最终取值及禁用原因（哪条规则触发），方便用户排查性能下降原因。  
3. **拆分自动禁用函数**：将 `_handle_piecewise_cuda_graph` 按业务维度（模型、硬件、功能）拆分为子函数，加入单元测试，防止未来 regression。  
4. **自定义 Op 测试**：为 `register_custom_op_from_extern` 添加 **shape‑only** 与 **runtime** 双模测试，确保 fake 实现与真实实现的输入/输出 dtype/shape 保持一致。  
5. **性能基准**：在 CI 中保留一套对比基准（PCG 开启 vs 关闭）针对主流模型（Llama‑2, DeepSeek‑V2, Qwen‑2）进行跑分，监控性能回退或异常增长。  
6. **回退路径监控**：在 `model_runner.init_piecewise_cuda_graphs` 里加入计数器或日志标记，记录每次因何原因回退到普通执行，帮助后续调优。  
7. **安全审计**：复核 `custom_op` 包装的外部库函数是否有潜在的未捕获异常或资源泄漏（如文件句柄、CUDA context），确保在 `torch.compile` 环境下仍能安全回收。

### Enhance error resilience in dump comparator (#19685)
**SHA**: `7579ab3` | 🔗 [查看提交](https://github.com/sgl-project/sglang/commit/7579ab3f3351444f65bb4758554a73fecf110cfb)

**🎯 变更类型**：功能增强 / 错误修复 / 架构变更  
**⚡ 重要程度**：🔴高  

**📋 变更摘要**  
1. 将比较相关的记录类型统一前缀为 `Comparison*`（`TensorComparisonRecord ➜ ComparisonTensorRecord`、`SkipComparisonRecord ➜ ComparisonSkipRecord`、`NonTensorComparisonRecord ➜ ComparisonNonTensorRecord`），并新增 `ComparisonErrorRecord` 用于捕获和报告在 bundle 比较过程中的异常。  
2. `SummaryRecord` 增加 `errored` 计数字段，`compute_exit_code` 现在会在出现错误记录时直接返回非零退出码。  
3. `entrypoint.run` 与 `_compare_bundle_pairs` 通过 `try/except` 捕获单个 bundle 的异常，转化为 `ComparisonErrorRecord`，保证其它 bundle 仍会继续比较。  
4. 调整显示层（`display.py`、`output_formatter.py`、`output_types.py`）以适配新的记录名称并简化 Rich 表格渲染，实现更轻量的文本渲染。  
5. 大量测试同步升级，覆盖错误记录的序列化/反序列化、退出码逻辑以及对异常的容错行为。

---

**🎯 影响范围**  
- `sglang/srt/debug_utils/comparator/*`（核心比较、记录定义、显示、入口、格式化工具）  
- 所有使用比较记录的上层模块（如 `sglang/srt/debug_utils/dumper.py`、CI 测试脚本）  
- 公开 API `sglang.srt.debug_utils.comparator.output_types`（类名变更、字段新增）  

---

**🔍 技术洞察**  

| 维度 | 影响说明 |
|------|----------|
| **架构影响** | - 引入 `ComparisonErrorRecord` 使比较流水线由 **单点失败** 转为 **容错模式**，提升整体系统鲁棒性。<br>- 统一记录命名 (`Comparison*`) 简化模型层级判别，降低误用风险。<br>- `SummaryRecord` 计数字段扩展，使上层调度器 (`compute_exit_code`) 能直观判断错误状态。 |
| **性能影响** | - 仅在异常捕获路径额外创建 `ComparisonErrorRecord`，对常规成功路径几乎无开销。<br>- 移除 Rich 表格构建函数（`_build_rich_table`）与重复渲染逻辑，略微降低 CPU 与内存占用，尤其在大量对齐信息输出时收益明显。 |
| **安全考虑** | - `ComparisonErrorRecord` 暴露异常类型与完整 traceback，可能泄露内部实现细节。若对外提供 JSON 报告，建议在生产环境对 `traceback_str` 做脱敏或可选隐藏。<br>- 其它改动均为内部抽象，不涉及外部输入校验，安全风险极低。 |
| **可维护性** | - 通过统一前缀提升代码可读性，所有记录的 `category` 统一由基类决定，避免散落的硬编码。<br>- 新增错误记录后，`_consume_comparison_records` 与 `compute_exit_code` 的分支更清晰，后续扩展（如警告、信息）更容易加入。 |
| **兼容性** | - 类名改动是 **破坏性** 的 API 变化；外部项目如果直接导入旧类（如 `TensorComparisonRecord`）将报 `ImportError`。<br>- 项目内部已全部迁移，官方文档若有示例需同步更新。若需要平滑迁移，可考虑在 `__init__.py` 中保留旧别名（已删除），目前已彻底移除。 |

---

**⚠️ 潜在风险**  

1. **破坏向后兼容**：外部使用者仍然依赖旧的记录类名或字段（如 `type: "comparison"`）会导致运行时错误。  
2. **异常信息泄露**：`ComparisonErrorRecord.traceback_str` 直接写入报告文件或标准输出，可能在公开 CI 环境中泄漏路径或内部实现细节。  
3. **错误路径未完整覆盖**：虽然大多数入口已包裹 `try/except`，但若未来在新模块中直接调用 `compare_bundle_pair` 而未使用 `entrypoint`，仍可能抛出未捕获异常。  
4. **测试遗漏**：对 `display` 中已删除的 Rich 表格功能未进行回归测试，若有人依赖 `--output-format rich` 可能出现渲染缺失。  

---

**💡 关注建议**

**⚠️ 潜在风险**  

1. **Pin 预算误配置**  
   - 设定过高的 `MAX_PINNED_RATIO` 会占满 Host KV，导致后续普通 KV 无法写入或 Evict 失败，进而阻塞 Prefetch/Load‑Back。  

2. **TTL 失效边界**  
   - `pin_expiry` 使用 `time.monotonic()`，若系统时间被频繁调节（例如容器重启后 monotonic 基准变化），可能导致 TTL 立即失效或异常延长。  

3. **并发竞争**  
   - `pin_prefix` 在遍历树、修改 `host_ref_counter`、`pinned_size_` 时没有显式锁，依赖 GIL（Python）和内部锁实现。若在多进程/多线程模式下（如 DGX 的 DP‑size > 1）并发 Pin 同一前缀，可能出现计数不一致或双重计数。  

4. **旧版节点分裂导致 Pin 丢失**  
   - 在 `_split_node` 中将 Pin 状态复制到新父节点，但若在极端分裂路径（多层级一次性 split）未完整迁移，可能出现已经 Pin 的子节点失去持有计数，导致提前被回收。  

5. **指标与监控缺失**  
   - 新增的 Prometheus 指标只有在 `enable_hierarchical_cache` 为 `True` 时创建，若管理员开启了 Pin 但未打开层次缓存，Pin 仍会被拒绝但对应监控缺失，难以定位问题。  

**💡 关注建议**  

| 建议 | 说明 |
|------|------|
| **配置审计** | 在生产部署前，显式检查 `SGLANG_HICACHE_MAX_PINNED_RATIO`，推荐值 ≤ 0.2（即最多占 20% Host KV）。 |
| **监控告警** | 为 `hicache_host_used_tokens / hicache_host_total_tokens` 添加告警阈值（如 > 80%）并结合 `pin_prefix` 调用频率监控，以防 Pin 预算被耗尽。 |
| **并发安全** | 在多进程环境下，考虑在 `pin_prefix`、`_clear_pin`、`_is_pinned` 等关键路径添加细粒度锁（如 `threading.RLock`）或使用原子操作，以防计数错位。 |
| **TTL 失效测试** | 编写单元/集成测试，覆盖：<br>• Pin 超过预算的回退路径<br>• TTL 到期后自动释放<br>• 命中后 TTL 正确刷新（refresh‑on‑hit）<br>• 多客户端并发 Pin 同一前缀 | 
| **日志可观测** | 在 `pin_prefix` 成功/失败路径加入 `logger.info`（已有），但建议统一使用结构化日志 (`extra={'action':'pin_prefix',...}`) 便于聚合搜索。 |

### feat: add FA4 SM90 paged KV decode support & update attention docs (#18442)
**SHA**: `f51ddba` | 🔗 [查看提交](https://github.com/sgl-project/sglang/commit/f51ddba131c65962cadc770b57eb7115a14df5c0)

**🎯 变更类型**：功能增强  
**⚡ 重要程度**：🔴高  
**📋 变更摘要**：  
- 为 FlashAttention 4 (FA4) 在 Hopper (SM90) 与 Blackwell (SM100) GPU 上加入 **paged KV decode** 支持，限制 page‑size 为 128，并在解码路径中引入 page‑table 逻辑。  
- 同步更新文档说明 FA4 在 Hopper 的 decode 性能退化情况以及使用方式；在服务器启动参数中加入兼容性检查与默认 page‑size 调整。  

**🎯 影响范围**：  
- `python/sglang/jit_kernel/flash_attention/cute/flash_fwd.py`（核心 FA4 前向 kernel）  
- `python/sglang/jit_kernel/flash_attention/cute/interface.py`（FA4 接口层）  
- `python/sglang/srt/server_args.py`（启动参数校验）  
- 文档 `docs/advanced_features/attention_backend.md`  

**🔍 技术洞察**：  
- **架构影响**：  
  - 在 `FlashAttentionForwardSm90/Sm100` 构造函数中新增 `page_size` 参数，并在 kernel 内部创建 `tiles_per_page`、`mPageTable` 等辅助张量，实现 **paged KV**（即将 KV 存储切分为固定大小页）在 decode 阶段的索引映射。  
  - 通过 `get_n_block` JIT 函数把逻辑块号映射到实际页号，实现对 SM90/SM100 的分页访问，保持与原有连续 KV（无 page_table）路径兼容。  
  - `server_args` 中加入对 `prefill_attention_backend=="fa4"` 且非 MLA 模型时强制 page‑size=128 的检查，确保硬件约束得到满足。  

- **性能影响**：  
  - **正向**：在 SM100 (Blackwell) 上分页 KV 与连续 KV 性能基本持平（FA4 已原生支持 split‑KV）。  
  - **负向**：在 SM90 (Hopper) 上缺少 SplitKV 支持，分页 KV 会导致 decode 随序列长度显著下降（2K‑16K token 甚至出现 10%‑49% 的慢速），文档已给出详细警示。  
  - 额外的 page‑table 读取与块号映射在极端小 batch 场景下会产生一定开销，但相对于整体算子时间影响有限。  

- **安全考虑**：  
  - 新增 `assert page_size is None or page_size % n_block_size == 0`，防止不合法的 page‑size 触发越界访问。  
  - `mPageTable` 的索引在 `get_n_block` 中严格受 `tiles_per_page` 限制，理论上不会产生 OOB，但仍需在不同硬件/驱动版本上做充分验证。  
  - 未涉及网络或文件系统的安全改动。  

**⚠️ 潜在风险**：  
1. **兼容性回滚**：旧版模型/脚本未显式传递 `page_size`，在 SM90 上默认 128 可能导致与预期不符的性能退化或意外的 assert。  
2. **页表尺寸错误**：`mPageTable` 必须满足 `[batch, num_pages]` 形状，若构造错误会在 JIT 编译时抛出难以定位的错误。  
3. **内存占用**：分页 KV 需要额外的 page‑table 张量（int32），在大 batch、长序列情况下略增显存使用。  
4. **调度逻辑变更**：`load` 函数对 KV 加载路径分支改动（依据 `mPageTable` 是否为空）可能在极端并发下暴露同步 bug。  

**💡 关注建议**：  
- **测试覆盖**：在 Hopper (SM90) 与 Blackwell (SM100) 上分别跑至少三组序列长度（2K、8K、16K）和 batch 大小（1、8）进行对比基准，验证页面映射与性能预期。  
- **参数校验**：在用户端 CLI/SDK 中加入提示：`--page-size` 必须为 128（除非未来支持其他值），并在文档中突出说明 SM90 的性能警告。  
- **回滚路径**：保留 `prefill_attention_backend="fa4"` 且 `page_size=None` 时使用原有连续 KV 实现，以免意外影响已有部署。  
- **监控与日志**：在运行时记录 `page_table.shape`、`tiles_per_page` 与实际 `n_block` 映射次数，帮助定位潜在 OOB 或不匹配情况。  
- **CI 更新**：在 CI 中加入针对 `torch.cuda.get_device_capability()` 为 (9, 0) 与 (10, 0) 的硬件仿真测试，确保断言逻辑在两代 GPU 上均通过。

### [HiCache] Re-land spec v2 + decode KV cache offloading compatibility (#19615)
**SHA**: `dc02e5b` | 🔗 [查看提交](https://github.com/sgl-project/sglang/commit/dc02e5bea76a9c0e91229e333e9895a54657dc62)

**⚡ 变更类型**：功能增强 / 兼容性修复  
**🟡 重要程度**：中  

**📋 变更摘要**  
1. 在 `DecodeKVCacheOffloadManager._release_finished_req` 中加入对 **spec‑v2 产生的 KV‑cache 超额分配** 的释放逻辑；若 page‑size>1，还会对起始位置做 `ceil_align` 对齐。  
2. 移除 `ServerArgs._handle_cache_compatibility` 中 “spec v2 与 decode‑offload KV‑cache 不兼容” 的限制检查，使二者可以共存。  
3. 新增针对该路径的单元测试，覆盖普通、超额、页对齐、无效释放以及 prefix‑size 更新等情形。

**🎯 影响范围**  
- `python/sglang/srt/disaggregation/decode_kvcache_offload_manager.py`（KV‑cache 释放）  
- `python/sglang/srt/server_args.py`（启动参数兼容检查）  
- `test/registered/disaggregation/test_specv2_kvcache_offloading.py`（CI 用例）  

**💡 关注建议**  

1. **正确性**：  
   - 确认 `req.pop_overallocated_kv_cache` 返回的 `(start, end)` 始终是 **已分配** 区间的整数下标；若实现者在其他位置修改了该返回值语义，当前代码会误释放。建议在 `Req` 类加上文档或类型注解说明。  
   - `ceil_align` 只在 `page_size > 1` 时使用，若 `page_size` 为 0（理论上不应出现）会导致除零错误。可以在 `DecodeKVCacheOffloadManager.__init__` 加入断言 `page_size >= 1`。  

2. **性能**：  
   - 释放过程仍是一次 `tensor` 索引 + `allocator.free` 调用，开销与原来相同；对齐后可能产生 **额外的空洞**（如 `committed=10, alloc=12` 时不释放），这在极端高并发下可能导致碎片。建议在 `allocator.free` 前对 `overalloc_indices` 进行 `torch.unique`（若有重复）或在文档中提醒用户合理设置 `page_size` 与 spec‑v2 参数。  

3. **测试与 CI**：  
   - 当前测试覆盖了大多数路径，但缺少 **空请求**（`kv_committed_len == prefill_offloaded_len`）以及 **全局 page‑size 对齐后仍有零长度区间** 的情况。可再补充一两个极端输入，确保 `free` 不被意外调用。  
   - 新增的测试文件放在 `test/registered/disaggregation/`，已经通过 `register_cuda_ci` 注册。请确认 CI 环境的 GPU 驱动版本与项目要求一致，防止因 `torch.cuda` 初始化失败导致 CI 暂停。  

4. **文档**：  
   - 由于原来的不兼容检查被移除，建议在 `README` 或 `doc/args.md` 中补充说明：“开启 `disaggregation-decode-enable-offload-kvcache` 与 `spec v2` 均可使用，超额分配的 KV‑cache 将在请求结束时自动回收”。  
   - 在 `decode_kvcache_offload_manager.py` 顶部添加简要注释，解释 **为何需要对齐**（page‑size 约束）以及 **何时会产生超额分配**（spec‑v2 的 speculative decode）。  

5. **代码质量**：  
   - 引入 `ceil_align` 后未在文件顶部显式列出 `__all__`，若项目使用 `__all__`，请同步更新。  
   - `pop_overallocated_kv_cache` 的返回值在本文件只解构为 `start_p, end_p`，若以后改为返回 `Tuple[int, int]`，仍兼容；保持返回值顺序不变即可。  

**总体评估**：此改动解决了在 spec‑v2 场景下 KV‑cache 可能的 GPU 内存泄漏问题，并放宽了参数组合限制，兼容性和功能性均得到提升。只需在上述细节处做好防御性检查、文档同步与边界测试，即可安全上线。
