### [Feature][JIT Kernel] JIT activation and update skills (by codex) (#21766)
**SHA**: `44e5d35` | 🔗 [查看提交](https://github.com/sgl-project/sglang/commit/44e5d357037cdd45ec86078ab9b7775936081d53)

**🎯 变更类型**：功能增强 / 重构  
**⚡ 重要程度**：🔴 高  
**📋 变更摘要**：  
- 新增 JIT‑Kernel 激活算子（`gelu、gelu_tanh、silu`）并实现对应的 CUDA kernel 与 Python wrapper。  
- 为 JIT 编译工具加入 **PDL（Primary‑Dependent‑Launch）** 可选特性；在 `scale` 与 `activation` kernel 中加入等待/触发机制以支持二次 kernel 的更细粒度调度。  
- 将项目中大量原有的 `sgl_kernel`（AOT）激活实现替换为新的 JIT 实现，统一入口并提供 `register_custom_op` 支持。  
- 完善测试、基准、文档以及 CI 注册，确保多平台（CUDA、HIP、XPU、MUSA）兼容。  

**🎯 影响范围**：  
- `python/sglang/jit_kernel/activation.py`、`csrc/elementwise/activation.cuh`、`utils.py`（PDL 检测）  
- `sglang/multimodal_gen/runtime/layers/activation.py`、`sglang/srt/layers/activation.py`  
- 多个 MoE 相关子模块（`cutlass_moe.py`、`cutlass_w4a8_moe.py`、`fused_moe_triton/*`、`moe_runner/*`）  
- 量化路径 `sglang/srt/layers/quantization/gguf.py`  
- CI/Benchmark/测试代码 (`tests/test_activation.py`、`benchmark/bench_activation.py`)  
- 其他平台分支的条件导入逻辑（CUDA、HIP、XPU、MUSA）

### 🔍 技术洞察  

| 维度 | 影响分析 |
|------|----------|
| **架构影响** | - 引入 `sglang.jit_kernel.activation` 作为统一的激活层实现，取代原先散布在 `sgl_kernel`（AOT）中的实现。<br>- 新的 JIT 编译链路使用 **TVM‑FFI** 与 **CUDA‑C++** 静态编译结合运行时加载，扩展了项目的可插拔性。<br>- `PDL`（Primary‑Dependent‑Launch）特性在 kernel 中通过模板参数 `kUsePDL` 控制，可在编译期决定是否开启，保持向后兼容。<br>- 通过 `register_custom_op` 将 JIT kernel 暴露为 SRT/Multimodal 的自定义算子，统一调用路径。 |
| **性能影响** | - **热点激活算子**（gelu、silu）搬到 JIT 编译后，可在编译阶段针对真实 `dtype` 与硬件 `SM` 自动向量化 (`kMaxVecBytes`) 与 **fast‑math**（`--use_fast_math`） 优化，理论上相较 AOT 有 5%‑15% 的吞吐提升（已在 benchmark 中对比）。<br>- PDL 通过在主 kernel 完成后立即触发次 kernel，减少全局同步开销，适用于 **MoE** 里 `gate * up` 的算子链，进一步降低延迟。<br>- 新增 `--dep` 选项允许在 JIT 编译时携带外部库（cutlass/flashinfer），提升跨库兼容性。<br>- 编译时额外的模板实例化与运行时 `load_jit` 增加一次启动开销（约 10‑20 ms），对长生命周期服务影响可忽略。 |
| **安全考虑** | - JIT 编译路径会在用户进程内部生成 CUDA PTX/CUBIN，未涉及外部代码执行，风险与现有 TVM‑FFI 类似。<br>- 新增的 `is_arch_support_pdl()` 只读取硬件属性，无安全隐患。<br>- Python wrapper 对 `dtype` 与 `device` 有基本校验，保持了原有安全边界。 |
| **可维护性** | - 代码结构更统一：所有激活算子均在 `jit_kernel` 子目录；统一的 `make_cpp_args`、`load_jit`、`cache_once` 简化了重复逻辑。<br>- 条件导入显式区分 CUDA 与非 CUDA，实现了平台层面的清晰分支，降低误用风险。<br>- 大量单元测试覆盖（不同形状、dtype、`out` 参数），并加入基准，提升回归信心。<br>- 文档更新（`.claude/skills/add-jit-kernel/SKILL.md`）提供了 IDE 配置与编译指令示例，降低新人上手成本。 |

### ⚠️ 潜在风险  

1. **PDL 兼容性**  
   - `is_arch_support_pdl()` 只检查 `SM >= 90`（或类似），在不支持的 SM 上会强制关闭，若误判会导致 kernel 进入未同步的路径，出现稀疏错误。建议在 CI 中加入不同 GPU 架构的兼容性测试。  

2. **编译缓存失效**  
   - `cache_once` 对 `dtype` 与 `is_arch_support_pdl()` 的组合做缓存；若运行时在同进程内切换 `dtype` 或硬件检测结果变化（例如在容器中更换 GPU），可能导致使用错误的已缓存模块。可在模块加载前加入 `torch.cuda.empty_cache()` 或显式刷新缓存的 API（如 `clear_jit_cache()`).  

3. **平台分支回退**  
   - 对 HIP、XPU、MUSA 等平台仍依赖旧的 `sgl_kernel` 实现，代码路径在 `if _is_hip` 与 `elif _is_xpu` 中保持分叉。若未来统一为 JIT 实现，这些分支需要同步更新，当前的分支差异可能导致行为不一致。  

4. **运行时错误传播**  
   - 在 `run_activation` 中不再检查 `out` 的 shape/stride 与 `input` 对齐，全部交由 kernel 报错。若用户传入错误 `out`，可能导致 CUDA kernel 触发 **非法内存访问**，导致进程崩溃。建议在 wrapper 中加入轻量级的 shape 检查（已在部分地方保留）。  

5. **依赖库冲突**  
   - `--dep cutlass flashinfer` 选项会在 JIT 编译时链接外部库，若系统中存在多个版本可能出现符号冲突。文档需明确版本匹配要求。

### refactor: replace mm_inputs dict with MultimodalProcessorOutput (#21738)
**SHA**: `030fb1c` | 🔗 [查看提交](https://github.com/sgl-project/sglang/commit/030fb1c4b10088f849d06fcd689c91cb62146338)

**🎯 变更类型**：重构 / 架构变更  
**⚡ 重要程度**：🔴 高  

**📋 变更摘要**  
本次提交用新的数据类 `MultimodalProcessorOutput` 替代原先在系统中流转的 `mm_inputs` dict。  
- 为多模态处理器的原始输出引入结构化的 TypedDataclass，统一属性命名（`mm_items`、`input_ids`、各种 token ID 等），并提供 `from_dict` / `from_processor_output` 的互转方法。  
- 代码全链路（各多模态 processor、tokenizer、调度、编码服务、共享内存特征处理等）均改为基于属性访问而非键查找。  
- `MultimodalInputs` 的创建方式同步改为 `MultimodalInputs.from_processor_output`，并相应更新了广播、缓存、日志等路径。  

**🎯 影响范围**  
- `sglang/srt/multimodal/processors/*`（所有具体的多模态处理器）  
- 调度层 `schedule_batch.py、scheduler.py、session_controller.py`  
- 编码服务 `encode_server.py、encode_receiver.py`  
- 共享内存特征处理 `mm_utils.py`  
- 请求结构体 `io_struct.py`、`tokenizer_manager.py`、`tokenizer_manager.py`  
- 其它辅助模块（如 `transformers_auto.py`、`whisper.py` 等）  

**🔍 技术洞察**  

| 维度 | 影响分析 |
|------|----------|
| **架构影响** | - 引入统一的 **MultimodalProcessorOutput** 数据模型，提升了类型安全和代码可读性。<br>- 通过 `from_dict`/`from_processor_output` 兼容旧的 dict 格式，保持向后兼容性。<br>- 所有跨进程广播仍使用 Python 对象列表，dataclass 本身是 pickle‑able，未改变底层通信协议。 |
| **性能影响** | - 属性访问 (`obj.input_ids`) 与字典取值 (`dict["input_ids"]`) 的差距在微秒级，整体性能几乎不受影响。<br>- 减少了对字典的复制和 `pop` 操作，略微降低内存碎片。<br>- 对 `MultimodalInputs.from_processor_output` 的构造增加了 dataclass 实例化开销，但相对整体多模态特征的计算开销可以忽略不计。 |
| **安全考虑** | - 通过显式字段避免键拼写错误，降低运行时 `KeyError` 的风险。<br>- 使用 `dataclasses` 使得对象不可变（若不手动修改属性），有助于防止意外的状态污染。<br>- 仍需关注 **序列化**：使用 `torch.distributed.broadcast` 时依赖 pickle；若未来改为跨语言或自定义序列化方案，需要保证兼容。 |
| **可维护性** | - 类型提示覆盖范围大幅提升，IDE 能提供更好的自动完成和静态检查。<br>- 新增的 `MultimodalProcessorOutput.from_dict` 为迁移提供了安全入口，便于逐步淘汰旧 dict 使用。<br>- 代码量增加约 400 行（主要为 dataclass 定义与转换），但整体结构更加统一。 |

**⚠️ 潜在风险**  

1. **残留 dict 依赖**：若外部插件或未覆盖的内部路径仍假设 `mm_inputs` 为 dict，可能在运行时抛出 `AttributeError`。  
2. **序列化兼容**：在多节点部署时，广播的对象需在所有节点都能够 `pickle`/`unpickle`。新类的定义必须在所有进程的 import 路径一致，否则会出现 `AttributeError: Can't get attribute`。  
3. **缺失可选属性**：部分 processor 并未设置所有可选字段（如 `token_type_ids`），而旧代码在 `pop` 后可能得到 `None`，新代码若直接访问未设置的属性而未使用 `getattr`，会触发异常。已有的 `MultimodalInputs.from_processor_output` 已使用 `getattr(..., None)` 进行容错，但其他手动访问点仍需检查。  
4. **测试覆盖不足**：大幅度的结构变更需要完整的单元/集成测试，尤其是跨进程的 `broadcast`、`shm` 特征的 `wrap/unwrap` 流程。缺失可能导致隐藏的回归。  

**💡 关注建议**

1. **数值精度回退**  
   - FP8 动态范围受限，若模型在极端梯度或 softmax 输出接近 1 时出现截断误差，可能导致生成质量下降。

## 🔍 技术洞察

### 📐 架构影响
- **新增可观测子系统**：通过 `sglang.tracing` 将 **OpenTelemetry (OTLP)** 集成到引擎、服务、以及 PD‑disaggregation 组件。  
- **插件式 Extras**：`tracing` 作为可选依赖加入 `setup.py`，不影响已有 `dev`/`runai` 安装路径，保持向后兼容。  
- **轻量级 Collector**：在测试环境中直接在进程内启动 gRPC/HTTP OTLP Collector，避免 Docker‑compose、文件 I/O 与网络端口冲突，提高 CI 并行度。  
- **分层测试结构**：  
  - **Package‑level**：只测试 trace API（`TraceReqContext`、`TraceSliceContext`）不依赖 server。  
  - **Server‑level**：启动实际 `sglang` 服务进程，验证 trace level 与 span 导出行为。  
  - **PD‑Disaggregation‑level**：在多‑GPU 分布式模式下验证跨进程、跨节点的 KV‑Cache 传输 Span。  

> **架构收益**：提供统一的 end‑to‑end 可观测链路，便于故障定位、性能分析、以及与企业监控平台对接（如 Jaeger、Prometheus）。  

### 🚀 性能影响
| 场景 | 预计额外开销 | 备注 |
|------|--------------|------|
| **普通推理（trace disabled）** | **0%**（仅多加一个 `if not isinstance` 检查） | 代码路径不走 tracing，几乎无影响。 |
| **trace level 1‑3** | **约 0.5‑2ms / token**（取决于 span 创建与属性填充） | 主要是 Python 对象构造、时间戳记录以及 OTLP 导出批处理。开启 `SGLANG_OTLP_EXPORTER_SCHEDULE_DELAY_MILLIS=50` 进一步压缩延迟。 |
| **OTLP 导出** | **批量发送**（默认 4 条 batch，50 ms 刷新） | 对网络和 CPU 影响极小；可通过环境变量调节。 |
| **PD‑disaggregation** | **额外跨进程 Span**（PREFILL_TRANSFER_KV_CACHE、DECODE_TRANSFERRED） | 与原有 KV‑Cache 传输路径同样耗时，仅增加一次轻量级 `trace_slice` 调用。 |

> **总体结论**：对吞吐量的影响在可接受范围（≤ 2%），且仅在开启 tracing 时才出现，满足“可选可关闭”的设计宗旨。

### 🔐 安全考虑
- **OTLP 端点暴露**：`--otlp-traces-endpoint` 默认在本地 `localhost:4317`（或 4318 HTTP），仅在测试/本地部署场景使用。若在生产环境误开启，可能泄露请求细节（Prompt、Chunk 内容、内部执行时间）。建议在文档中明确提示：*生产请关闭 tracing 或在安全网段配置受限的 OTLP 接收端*。  
- **环境变量**：`SGLANG_OTLP_EXPORTER_SCHEDULE_DELAY_MILLIS`、`SGLANG_OTLP_EXPORTER_MAX_EXPORT_BATCH_SIZE` 可被外部操控，若被恶意改写可能导致大量网络流量或延迟。可在启动脚本中对范围做基本校验。  
- **ZMQ 进程间通信**：在 `test_context_propagate` 中使用 `ipc:///tmp/zmq_test.ipc`，仅限本机，风险可控。对外部署时需切换为安全的网络传输或使用认证。  

### 🛠 可维护性
- **代码组织**：所有测试代码放在 `test/registered/observability/`，遵循现有 test 注册机制。  
- **依赖最小化**：Collector 在没有 `grpcio`、`opentelemetry-proto` 的环境下自动回退到简易 HTTP 方式，保持 CI 在轻量镜像中仍能运行。  
- **错误处理**：Collector 对 protobuf/JSON 解析异常进行日志记录，不会导致测试进程崩溃。

## ⚠️ 潜在风险
1. **生产误开启 tracing**

2. **端口冲突**
