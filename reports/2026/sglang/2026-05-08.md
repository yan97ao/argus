### [sgl-kernel/cpu] support w8a8 int8 model for arm cpu (#16045)
**SHA**: `55d8223` | 🔗 [查看提交](https://github.com/sgl-project/sglang/commit/55d8223c2b59384fe10f9cd92e14fb8a520fb55f)

**🎯 变更类型**：功能增强 / 重构  
**⚡ 重要程度**：🔴高  

**📋 变更摘要**  
1. 为 `sgl-kernel` 引入了原生 Arm 64 (aarch64) 的 int8 量化算子，实现了 `int8_scaled_mm`、MoE 计算以及相关的量化/反量化函数，支持在 Arm 64 CPU（包括支持 i8mm / sdot 的硬件）上运行 W8A8‑Int8 模型。  
2. 在 Python 层 `w8a8_int8.py` 中加入对 Arm 64 的检测与权重处理分支，使得模型在缺乏 Intel AMX 的 Arm 环境下也可正常加载并执行。  
3. CMake 构建脚本改造为按处理器架构挑选源码，避免在 x86‑only 环境中编译 Arm‑specific 文件。  

**🎯 影响范围**  
- `python/sglang/srt/layers/quantization/w8a8_int8.py`（权重加载与前向分支）  
- `sgl-kernel/csrc/cpu/`：`gemm_int8.cpp、moe.cpp、op.h`（新增 ARM 64 实现）  
- `sgl-kernel/csrc/cpu/CMakeLists.txt`（源码过滤与平台目录）  
- `sgl-kernel/csrc/cpu/torch_extension_cpu.cpp`（条件编译宏）  
- 相关头文件和 Python 接口（`is_host_cpu_arm64`、`_is_cpu_arm64`）

### 🔍 技术洞察  

| 维度 | 影响描述 |
|------|----------|
| **架构影响** | • 新增 `is_host_cpu_arm64()` 检测，Python 层在 CPU 上分三路：<br> - Intel AMX（原路径）<br> - Arm 64（新路径）<br> - 其它 CPU（强制报错）。<br>• CMake 通过正则过滤 `x86_64|aarch64|ppc64` 子目录，只在匹配的处理器上编译对应的实现，防止在交叉编译时出现不兼容文件。<br>• `sgl-kernel` 编译入口加入了 `#ifndef __aarch64__` 包裹，以避免在 ARM 编译环境下重复定义 `int8_scaled_mm_with_quant`。 |

### ⚠️ 潜在风险
1. **硬件依赖不稳定**  
   - CI Runner 必须具备 MUSA 设备；若机器缺失或驱动版本不匹配，作业会直接失败。  
   - `torch.musa.is_available()` 仅在 `torch` 正确编译了 MUSA 才返回 True，可能导致误判。

2. **测试 flaky**  
   - 多模态 Diffusion 性能测试受模型下载、网络 I/O、GPU 共享资源影响，出现 Timeout、SafetensorError、OOM 等错误会导致 CI 重试上限被触发。  
   - 非连续 tensor、残差路径等边界测试在不同驱动/torch 版本上可能产生微小数值差异，导致 `assert_close` 误报。

3. **依赖冲突**  
   - 脚本在 `whl` 目录发现任何 `*.whl` 时都会 **卸载** 旧包并重新安装，若仓库中意外留下不兼容的 wheel（如针对 CUDA 编译的）会导致 MUSA 环境报错。  
   - `--user` 安装路径写入 `$HOME/.local/bin`，若 CI 环境的 `$HOME` 与工作区不一致，后续步骤可能找不到可执行文件。

4. **基准阈值误用**  
   - `perf_baselines_musa.json` 与现有基准合并在 `testcase_configs.py` 中，若文件结构不一致或缺少关键字段，`BaselineConfig.update()` 可能抛异常，导致所有测试失败。

5. **代码路径冲突**  
   - `pr-test-musa.yml` 中 `concurrency` 组名使用 `${{ inputs.ref || github.ref }}`，在 `workflow_call` 场景下如果多个 PR 同时触发相同 ref，可能出现意外的 **cancel‑in‑progress** 行为。

**SHA**: `d8f9d32` | 🔗 [查看提交](https://github.com/sgl-project/sglang/commit/d8f9d32a05a7c8be8fdc43bff7125c559dc50dc3)

**🎯 变更类型**：功能增强 / 重构 / 架构变更  
**⚡ 重要程度**：🔴高  

**📋 变更摘要**  
本次提交为 SGLang 引入了 **聊天模板自动检测** 能力，能够基于模型的 chat template 与 tokenizer 词表自动推断并填充 `--reasoning-parser` 与 `--tool-call-parser` 参数（支持 `auto` 选项）。为此新增 `template_detection` 模块，实现规则‑驱动的检测逻辑；在 `engine` 初始化、`template_manager`、`serving_chat` 与 `ReasoningParser` 中加入相应的调用与兼容处理；同时扩展 CLI 参数、日志、异常容错并补充大量单元测试覆盖。

---

### 🎯 影响范围
- **核心启动流程**：`engine.py`、`template_manager.py`、`template_detection.py`  
- **推理与工具调用解析**：`reasoning_parser.py`、`serving_chat.py`  
- **CLI 参数**：`server_args.py`（新增 `auto` 选项）  
- **测试套件**：新增 `test_template_manager.py`、对 `test_serving_chat.py` 大幅扩展  
- **代码库全局**：涉及 tokenizer 加载、日志输出、默认行为的细微改动。

## 🔍 技术洞察

### 1. 架构影响
- **模块化解耦**：检测逻辑统一迁移至 `sglang.srt.managers.template_detection`，形成 **规则‑上下文**（`TemplateDetectionContext`、`DetectionRule`）抽象，降低 `template_manager` 与 `engine` 对细节的耦合，提升可维护性与可扩展性（新规则只需在文件中添加即可）。
- **启动时机提前**：`resolve_auto_parsers` 在 `engine._launch_subprocesses` **scheduler 进程启动前**执行，确保子进程能够直接使用已解析好的 parser 名称，避免在运行时再做重复检测。
- **向后兼容**：`ServerArgs` 将 `--reasoning-parser` 与 `--tool-call-parser` 的取值范围扩展为 `["auto"] + 原有选项`，原有显式配置保持不变；若检测失败则回退为 `None`（关闭对应功能），不会引起异常崩溃。
- **统一配置入口**：`TemplateManager` 现在负责将检测结果暴露为属性 (`_suggested_reasoning_parser`、`_suggested_tool_call_parser`、`_reasoning_config`)，其它组件统一通过 `server_args` 或 `template_manager` 读取，减少散乱的硬编码判断。

### 2. 性能影响
- **启动开销**：`resolve_auto_parsers` 需要 **一次轻量级 tokenizer 加载**（`get_tokenizer`），并在内存中遍历词表构建 `vocab` 集合。对大模型的 tokenizer（数十万词）来说仍在毫秒级；相较于整体服务启动时间影响可忽略。  
- **运行时零开销**：检测全部在进程启动阶段完成，推理路径 (`_apply_conversation_template`、`_process_tool_call_stream`) 仅在已知 `thinks_internally` 时省去 `<think>` 前缀插入，**实际推理吞吐量略有提升**（少一次 token 输出）。
- **日志与异常捕获**：在检测过程中加入 `logger.info/warning`，对高并发生产环境影响极小，日志级别可通过配置控制。

### 3. 安全考虑
- **远程代码执行**：检测阶段会调用 `get_tokenizer(..., trust_remote_code=…)`，仍然受 `--trust-remote-code` 参数控制，未引入新风险。  
- **异常安全**：所有外部调用均已捕获异常并回退为 `None`，避免因模型或模板缺失导致服务启动失败。  
- **输入可信度**：检测规则基于 **模板文本** 与 **词表**，不涉及用户请求数据，不会产生注入或信息泄露风险。

### 4. 可维护性与可扩展性
- **规则驱动**：通过 `DetectionRule`（名称、返回值、谓词）实现 **可声明式** 检测，新模型或新模板只需新增规则，无需改动检测核心代码。  
- **可测试性**：新增 `test_template_manager.py` 采用 **表驱动**（matrix）方式覆盖所有规则，确保规则改动后可快速定位回归。  
- **文档与日志**：在检测成功/失败时均输出明确日志，便于运维定位自动检测行为。

### fix(router): configure HTTP client connection settings (#24330)
**SHA**: `be088f8` | 🔗 [查看提交](https://github.com/sgl-project/sglang/commit/be088f80766f6162f18165972ded158f8283ea61)

**🎯 变更类型**：功能增强  
**⚡ 重要程度**：🔴高  
**📋 变更摘要**：为 SGL‑Model‑Gateway 的上游 HTTP 客户端新增可配置项，包括连接超时、每主机最大空闲连接数以及 TCP keepalive 时长，并在文档、CLI 参数、内部配置结构及验证逻辑中同步实现，提升可靠性与可调度性。  

**🎯 影响范围**：  
- `sgl-model-gateway/src/config/*`（新增字段、默认值、构造器、验证）  
- `sgl-model-gateway/src/app_context.rs`（HTTP 客户端构造时使用新参数）  
- `sgl-model-gateway/src/main.rs`（CLI 参数解析及映射）  
- 文档 `docs/advanced_features/sgl_model_gateway.md`、MDX 版以及 README 中的说明更新  

**🔍 技术洞察**：  
- **架构影响**：在路由器配置层引入 `connect_timeout_secs`、`pool_max_idle_per_host`、`tcp_keepalive_secs` 三个属性；对应的 Builder 方法、默认值与验证逻辑均已补全。此改动保持向后兼容（默认值保持原有行为），但为上层部署提供了更细粒度的网络层调优入口。  
- **性能影响**：  
  - `connect_timeout_secs` 能够限制建立新连接的最长等待时间，防止异常网络导致的请求阻塞。  
  - `pool_max_idle_per_host` 限制每个目标主机的空闲连接数，防止因过多空闲 socket 消耗系统资源（文件描述符、内存）。  
  - `tcp_keepalive_secs` 启用 TCP keepalive 并可自定义空闲检测间隔，有助于及时回收已失效的长连接，提升整体吞吐与连接复用率。  
  - 以上调参均可能在高并发场景下显著降低连接建立延迟或资源占用，具体收益取决于实际网络环境和后端服务特性。  
- **安全考虑**：虽然不直接涉及身份验证或加密，但合理的超时与 keepalive 设置可以减轻 **DoS**（通过大量慢连接占满资源） 的风险；同时在配置验证中加入 “值必须 > 0” 的检查，避免误配置导致连接永不超时或 keepalive 失效。  

**⚠️ 潜在风险**：  
1. **误配置**：若用户将 `connect_timeout_secs` 设得过低，可能导致正常的慢网络连接被频繁切断，产生请求失败。  
2. **资源限制**：`pool_max_idle_per_host` 设得过高可能导致文件描述符耗尽，特别是在多租户或高并发部署下。  
3. **兼容性**：虽然默认值与旧行为一致，但旧版本的启动脚本或容器镜像若硬编码了 `--pool-idle-timeout-secs` 以外的参数，升级后可能出现未预期的行为。  
4. **监控缺失**：新增参数的运行时指标未同步加入默认监控面板，运维人员可能缺乏调参依据。  

**💡 关注建议**：  
- 在部署文档或 Helm chart 中同步添加新参数的示例与推荐阈值，避免运维人员忽视。  
- 通过监控（如 Prometheus `http_client_connect_seconds_histogram`、`http_client_idle_connections`）观察新参数对连接建立时延与空闲连接数的实际影响，逐步调优。  
- 对关键环境（生产、预检）进行 **回归测试**，确保在不同网络条件下新超时设置不会导致异常重试或超时错误。  
- 若使用容器化部署，确保对应的 `SMG_*` 环境变量已经在容器入口脚本或 ConfigMap 中暴露。  
- 考虑在未来的版本中加入对这些参数的 **运行时热更新**（如 SIGHUP 重载），进一步提升运维灵活性。

| **可维护性** |- 通过抽象 `QuantizationConfig` 与 `LinearMethodBase`，新增量化方式遵循统一接口，后续扩展（如 MXFP4）成本低。<br>- `ModelSlimMXFP8Scheme` 复用了 `ModelSlimLinearScheme` 的大部分实现，代码量控制在 120 行，易于审阅。<br>- 文档同步完整，CLI 示例明确，降低使用门槛。<br>- 变更 `fsdp_load` 只添加了 `weight_scale` 到 `unused_keys`，对已有加载流程影响最小。 |

## ⚠️ 潜在风险

1. **平台误用**  
   - 若用户在非 NPU 环境或 CANN 版本低于 8.0.RC3 时显式指定 `--quantization mxfp8`，会触发 `torch_npu` 导入失败或 kernel 不存在的异常。  
   - 建议在 `MXFP8Config.get_min_capability()` 中返回明确的硬件标识，并在 CLI 启动前进行检测，抛出友好错误。

2. **权重/scale 维度不匹配**  
   - `ModelSlimMXFP8Scheme` 假设 `weight_scale` 维度为 `[out, in/32]` 并在 `process_weights_after_loading` 进行 `reshape(..., 2)`，若源码或 `msmodelslim` 产生的 checkpoint 结构变更，可能导致运行时的 shape 错误。  

3. **FSVD/参数分片冲突**  
   - 在 `fsdp_load` 中加入 `weight_scale`，但对应的 `shard`/`meta_sd` 逻辑未同步更新，极端情况下可能导致参数/元数据不一致（尤其在 ZeRO‑3 场景）。  

4. **内存碎片与复制**  
   - 在线量化阶段会把权重整体搬到 NPU (`to(f"npu:{torch.npu.current_device()}")`) 并生成两个张量（`qw、w_scale`），对超大模型可能导致临时显存峰值高于原有加载方式。  

5. **工具链兼容**  
   - 改写的 `wan_repack.py` 现在依赖 `shutil.copytree` 的 `ignore_patterns`，如果原始 Diffusers 目录里出现额外子目录（如 `tokenizer`）可能被误复制，需在脚本中明确排除仅 `transformer*`。

### [fix] /pause_generation and /continue_generation wrong for --tokenizer-worker-num > 1 (#24462)
**SHA**: `7deed98` | 🔗 [查看提交](https://github.com/sgl-project/sglang/commit/7deed98e1bc7f4d2016c5136a47a78b4f81879cc)

**变更概览**  
本次提交在 `io_struct.py` 中新增 `TokenizerWorkerRegistration`、`PauseContinueBroadcast` 两个 dataclass；在 `multi_tokenizer_mixin.py` 中为 **MultiTokenizerRouter** 与 **TokenizerWorker** 添加了注册、广播暂停/继续的机制，并在 `pause_generation / continue_generation` 中加入了 `Future` 等待逻辑，使得 `--tokenizer-worker-num > 1` 时能够保持所有 tokenizer worker 的 `is_pause` 状态一致。

**核心改动**  
1. **注册阶段**：每个 tokenizer worker 启动后向 router 发送 `TokenizerWorkerRegistration`，router 用 `self.all_worker_ipcs` 记录所有 IPC 名称。  
2. **广播路径**：router 收到 `PauseGenerationReqInput` 或 `ContinueGenerationReqInput` 时，会先构造 `PauseContinueBroadcast(is_pause=…)`，遍历 `self.all_worker_ipcs` 通过 `SocketMapping` 发送给所有 worker，随后（除 abort‑mode 外）再转发原请求到 scheduler。  
3. **worker 端**：在 `TokenizerWorker.__init__` 中立即向 router 注册；在 `pause_generation/continue_generation` 中创建 `self._pause_continue_future`，发送请求后 await 该 future；新增 `_handle_pause_continue_broadcast` 与 `_apply_pause_continue_broadcast`，在收到广播后更新 `self.is_pause`、通知条件变量并完成 pending future。  
4. **Result Dispatcher**：在 worker 初始化时把 `PauseContinueBroadcast` 加入 `self._result_dispatcher`，确保广播能够被 `handle_loop` 正确路由。

**影响范围**  
- `sglang/srt/managers/io_struct.py`（新结构体）  
- `sglang/srt/managers/multi_tokenizer_mixin.py`（router 与 tokenizer worker 逻辑）  
- 可能波及 `sglang/srt/managers/tokenizer_manager.py`、调度层（scheduler）以及任何依赖 `PauseGenerationReqInput`/`ContinueGenerationReqInput` 的调用路径。

**潜在风险 & 注意点**  
1. **注册时机**：worker 在 `__init__` 里即向 router 发送注册信息，但 `router_worker_obj` 仍在循环中接收消息，若 router 尚未完成 `self.socket_mapping = SocketMapping()` 的初始化就收到注册，可能出现未初始化的 `socket_mapping` 使用。建议在 router 启动后先完成 `socket_mapping` 再监听注册，或在发送前确认 router 已就绪。  
2. **广播顺序**：router 先广播再转发（除 abort），但 `PauseGenerationReqInput` 里可能包含 `mode="abort"` 的特殊处理。若其它模式需要同步等待所有 worker 完成 pause，当前实现只等待发起者的 future 完成，其他 worker 的 future 并未暴露给外部，可能导致调度层在收到 scheduler 的响应前就继续生成。可考虑在 router 收到广播后收集所有 worker 的确认（例如使用 `Barrier`）再返回。  
3. **Future 复用**：`self._pause_continue_future` 在同一 worker 上只能一次挂起；若在 pause 期间又收到另一 pause/continue（如用户快速切换），future 可能被覆盖或导致未决状态。需要在 `pause_generation`/`continue_generation` 前检查并防止并发调用（如加锁或直接复用已有 future）。  
4. **异常路径**：如果广播发送失败或某 worker 因异常退出，发起者的 future 将永远卡住。建议在 `router_worker_obj` 捕获发送异常并在对应 worker 超时后主动 set_exception。  
5. **性能**：广播采用逐个 `socket_mapping.send_output`，在 `tokenizer-worker-num` 很大时会产生 O(N) 消息，影响延迟。可考虑改为一次多播（ZMQ PUB/SUB）或合并多条 IPC 为批量发送。  
6. **兼容性**：新添加的 dataclass 不会破坏旧的序列化协议，但若已有持久化日志或外部工具对 `io_struct` 进行解析，需同步更新文档/版本号。

**建议**  
- 为 router 加入 **初始化屏障**（如 `await self.router_ready`）或在 worker 中重试注册，确保两端同步。  
- 在 `pause_generation` / `continue_generation` 前加入互斥锁（`asyncio.Lock`）防止并发请求导致 `_pause_continue_future` 被覆盖。  
- 为广播添加超时检测并在超时后强制完成 future，以避免死锁。  
- 在文档中注明 “当使用 `--tokenizer-worker-num > 1` 时，pause/continue 将自动在所有 tokenizer worker 之间同步”。  
- 若后续需要更高并发，可考虑改用 ZMQ PUB/SUB 进行一次性广播。  

整体来看，此次改动修复了多 tokenizer worker 环境下 pause/continue 状态不一致的 bug，思路清晰，代码结构合理，只需在鲁棒性、并发安全和异常处理上做少量补强即可投入使用。
