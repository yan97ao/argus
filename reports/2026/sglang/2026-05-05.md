### [AMD] Add Kimi-K2.6 in nightly tests for MI30x and MI35x (#23848)
**SHA**: `244531b` | 🔗 [查看提交](https://github.com/sgl-project/sglang/commit/244531bc4f3bc24a5ca2c0ce77b4df05d515cc14)

**🎯 变更类型**：功能增强  
**⚡ 重要程度**：🔴高  
**📋 变更摘要**：在 AMD（MI30x、MI35x）Nightly CI 中新增对 Kimi‑K2.6 模型的准确性与性能测试，替换原有 Kimi‑K2.5 条目，并在 `test/registered/amd/*` 目录下新增对应的评估与基准脚本，实现模型覆盖与报告的自动化。  

**🎯 影响范围**：  
- CI 工作流文件 `.github/workflows/nightly-test-amd*.yml`（模型调度、套件名称）  
- AMD 测试注册与执行模块 `sglang/test/ci/ci_register.py`（间接影响）  
- 新增的测试脚本：  
  - `test_kimi_k26_eval_amd.py`、`test_kimi_k26_eval_mi35x.py`（准确性）  
  - `test_kimi_k26_perf_amd.py`、`test_kimi_k26_perf_mi35x.py`（性能）  

**🔍 技术洞察**：  

- **架构影响**  
  - 仅在 CI 配置层面新增模型入口，未改动 SGLang 核心运行时或调度逻辑。  
  - 注册逻辑依赖 `register_amd_ci`，保持与已有 Kimi‑K2.5 测试一致，确保模型路径、TP、注意力后端（`aiter` + `triton`）的兼容性。  

- **性能影响**  
  - 对生产代码无直接影响；新增的性能基准只在 Nightly 环境执行，耗时约 60‑90 分钟。  
  - 通过 `NightlyBenchmarkRunner` 收集的报告将帮助团队评估 Kimi‑K2.6 在 MI30x/MI35x 上的吞吐和 ITL，间接提升后续模型部署的性能调优依据。  

- **安全考虑**  
  - 测试脚本使用 `--trust-remote-code` 拉取并执行第三方模型（moonshotai/Kimi‑K2.6），在 CI 环境中已默认信任。若模型仓库被恶意篡改，可能导致执行任意代码。  
  - 依赖的 `requests` 调用仅对本地服务器进行 `flush_cache`，无外部网络交互。  
  - 环境变量 `SGLANG_USE_AITER=1`、`SGLANG_ROCM_FUSED_DECODE_MLA=0` 为内部性能开关，未引入新的权限或资源泄露风险。  

**⚠️ 潜在风险**：  
1. **CI 失效**：新增的套件名称、调度规则若与已有矩阵不匹配，可能导致 Nightly 流水线卡死或报错。  
2. **模型获取失败**：`moonshotai/Kimi-K2.6` 若在 HuggingFace 上不可用或访问受限，会导致所有相关测试全部失败，影响整体 CI 通过率。  
3. **安全隐患**：`--trust-remote-code` 在生产环境不应使用，CI 环境若被未经审计的代码污染，可能引入后门。  
4. **资源占用**：新增的性能基准耗时较长（最高 5400 s），在共享 CI 资源池中可能与其他作业竞争，引发调度冲突。  

**💡 关注建议**：  
- **CI 稳定性**：在合并前在 fork 或临时分支跑一次完整的 AMD Nightly，确认新 job 名称和 suite 参数均能被 `run_suite.py` 正确解析。  
- **模型锁定**：考虑在 CI 中使用模型的特定 commit/tag（如 `moonshotai/Kimi-K2.6@v1.0`），防止上游意外更新破坏回归基准。  
- **安全审计**：对 `--trust-remote-code` 加入安全提示或在 CI 环境做一次 hash 校验，确保拉取的模型文件未被篡改。  
- **资源调度**：若 Nightly 时段 CI 节点较紧张，可将性能基准拆分为可选子任务，通过 `inputs.job_filter` 控制执行；或者在跑完准确性后自动跳过性能测试（基于 `is_in_ci` 判断）。  
- **文档同步**：在项目的模型支持列表或 CI 文档中补充 Kimi‑K2.6 的说明，标明对应的 GPU（MI30x/MI35x）与推荐 TP/注意力后端配置。  

通过上述措施，可在保证新模型覆盖的同时，将潜在的 CI 中断与安全风险降到最低。

### [SMG][CI] Add K8s integration tests + wire into pr-test-rust (#24278)
**SHA**: `c545a5b` | 🔗 [查看提交](https://github.com/sgl-project/sglang/commit/c545a5b1c01243a22b9c5ebf2e94da540e1c5888)

**🎯 变更类型**：功能增强  

**⚡ 重要程度**：🔴高  

**📋 变更摘要**  
- 在 CI 中新增 **K8s 集成测试** 工作流 (`k8s-integration` job)，在本地 Kind 集群上通过假 worker pod 验证 SMG 的服务发现与对齐逻辑，且不依赖 GPU。  
- 为该工作流提供专用的 Dockerfile（仅编译 Rust 二进制）以及完整的 **pytest** 套件、fixture、K8s Manifest、setup/teardown 脚本和假 worker 实现。  
- 修正 `sgl-model-gateway/e2e_test/conftest.py` 对 `find_spec` 的异常处理，防止在未安装轮子时抛出 `ModuleNotFoundError`。  

**🎯 影响范围**  
- `sgl-model-gateway`：service discovery 与 reconciliation 代码的测试覆盖大幅提升。  
- CI 配置（`.github/workflows/pr-test-rust.yml`）：新增 `k8s-integration` job，增加整体 PR 检查时长。  
- 新增的 e2e 测试文件、Dockerfile、K8s manifest、脚本等全部位于 `sgl-model-gateway/e2e_test/k8s_integration/`。  

**🔍 技术洞察**  
- **架构影响**：  
  - 仅在 CI 环境引入 **Kind 集群**、 `kubectl`、 `docker buildx` 等组件，不会影响生产部署。  
  - 通过 **ConfigMap** 注入 `fake_worker.py`，模拟真实 worker 行为，帮助验证网关在 **PD（prefill‑decode）模式**、普通模式下的 **UID、标签变更**、以及 **service discovery** 的完整路径。  
  - 新增的 `conftest.py` 通过捕获 `ModuleNotFoundError`，提升本地开发与 CI 的兼容性，避免因轮子缺失导致测试加载失败。  

- **性能影响**：  
  - `k8s-integration` job 需要约 30 分钟（创建 Kind 集群、构建 Docker 镜像、部署多份 Manifest），显著增加 PR 检查时长。  
  - 使用 **CI cargo profile (`ci`)** 与缓存（`type=gha,scope=k8s-integration`）已尽可能压缩构建时间，但仍对 CI 资源产生较大负载。  
  - 运行时对网关的 **端口转发**、轮询等待（最长 RECONCILIATION_WAIT_SECS ≈ 90 s）可能导致测试间隙的“离线”时间。  

- **安全考虑**：  
  - 所有新增流程均在受控的 GitHub Runner（`ubuntu‑22.04`）内部执行，网络仅限于 Docker Hub、Kind 下载地址以及 K8s API，未向外暴露服务。  
  - 使用的 **python:3.12‑slim** 镜像与自研的 `smg-gateway:test` 均为只读容器，未授予特权。  
  - 没有向生产代码引入新依赖，仅在测试环境新增 `httpx`、`pytest`，安全风险可忽略。  

**⚠️ 潜在风险**  
1. **CI 易失**：  
   - Kind 二进制下载、kubectl 版本获取依赖外部 CDN，网络波动会导致工作流提前失败。  
   - 60 秒的服务发现间隔与 `RECONCILIATION_WAIT_SECS` 设定稍显紧凑，若集群启动慢（尤其在共享 runner 上），可能出现超时导致测试失败。  

2. **资源竞争**：  
   - 同时执行的 PR 可能争夺 Docker 构建缓存与 runner 的 CPU/内存，导致构建时间不确定、甚至 OOM。  

3. **清理不彻底**：  
   - 虽然 `setup.sh teardown` 在 `always()` 步骤中调用，但若在测试阶段进程异常终止（如 `port‑forward` 卡死），残余的 Kind 集群或 Pod 可能残留。  

4. **误报的 import 处理**：  
   - `conftest.py` 捕获的 `ModuleNotFoundError` 会把所有 import 失败都当作轮子未安装，可能掩盖其它模块缺失导致的真实错误。  

**💡 关注建议**  
- **提升 CI 稳定性**：  
  - 为 Kind 与 kubectl 下载加入 **重试逻辑**（或使用官方 apt 包），并在 `setup.sh` 中检测版本是否已缓存。  
  - 考虑将 `k8s-integration` 拆分为独立的 workflow，仅在 `run-ci` 标记或定时任务触发，避免每个 PR 都强制跑长时间测试。  

- **优化超时与等待**：  
  - 在 `conftest.py` 中提供可配置的 `RECONCILIATION_WAIT_SECS`（环境变量），在 CI 中可以调大，在本地调小，加速调试。  
  - 对于端口转发的启动与关闭，使用 `subprocess.Popen(..., start_new_session=True)` 防止信号泄漏，确保 `finally` 中一定能终止。  

- **缓存与镜像管理**：  
  - 将 `smg-gateway:test` 镜像的构建结果缓存至 GitHub Packages，或使用 `docker save/ load` 与 `kind load` 组合，进一步压缩首次构建时间。  
  - 明确锁定 `rust:1.90-bookworm`、`python:3.12-slim` 版本标签，防止上游镜像突变导致不可预期的 CI 失败。

  - 在每个测试 fixture（如 `gateway_port_forward`）的 `finally` 中加入 `kind delete cluster --name smg-test || true`，双保险。

### Gemma4-mtp cookbook (#24433)

**⚡ 重要程度**：🟡 中  

**📋 变更摘要**  
1. 在 Gemma 4 的 cookbook 文档中加入了 MTP（NEXTN）推理的使用说明和完整的服务启动命令示例。

**🎯 影响范围**  
- **文档**：`docs_new/cookbook/autoregressive/Google/Gemma4.mdx`。

- **部署脚本**：间接影响用户的 `sglang serve` 调用方式。  

**💡 关注建议**

- **测试**：新增 UI 选项后，建议补充单元测试（如 `gemma4-deployment` 的 commandBuilder 在不同硬件/模型组合下输出的完整命令），并在 CI 中验证文档构建不因新参数报错。  
- **文档同步**：确保 `README`、示例脚本等其它位置也同步更新为 `google/gemma-4-...` 前缀，避免用户混淆。  

总体来看本次变更对核心运行时不产生影响，属于文档与前端配置的功能扩展；只要做好兼容性提示和命令校验，即可安全发布。

### [diffusion] chore: clean CUDA cache only at explicit release points (#24397)

**🎯 变更类型**：性能优化 / bug 修复  
**⚡ 重要程度**：🟡 中  

**📋 变更摘要**  
1. 在 `ComponentResidencyManager` 中增加对大模型/层级卸载后显式调用 `torch.empty_cache()`，仅在曾经在 CUDA 上且已释放或使用 `LayerwiseOffloadStrategy` 时清理缓存。  
2. `GPUWorker` 捕获执行异常后若已初始化 CUDA 立即释放显存，以降低 OOM 复现概率。  
3. `PerfLogger` 的内存打印改为 `empty_cache=False`，避免每个阶段都触发额外清理，降低 CI 开销。

5. **代码风格**：`gemma_dual_rmsnorm_residual_scalar` 参数顺序与 `gemma_rmsnorm_residual_scalar` 不完全一致（后者省略 `weight2`），保持一致可以降低使用时的混淆。
