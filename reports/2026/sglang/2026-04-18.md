### [diffusion] feat: introduce ltx-2-two-stage device manager (#22869)
**SHA**: `0d94c33` | 🔗 [查看提交](https://github.com/sgl-project/sglang/commit/0d94c3366ad684b376f6ebfaa0d8d3657f53302c)

**🎯 变更类型**：功能增强  
**⚡ 重要程度**：🔴 高  
**📋 变更摘要**：  
- 为 LTX‑2.3 两阶段 Diffusion 引入 **LTX2TwoStageDeviceManager**，实现三种设备驻留模式（original、snapshot、resident），在 GPU/CPU 之间智能切换、预加载、预取以及低显存优化。  
- 新增 `--ltx2-two-stage-device-mode` CLI 参数和对应的环境变量，支持自动根据显卡容量选择默认模式。  
- 代码层面在 pipeline、stage、server_args、测试等多个模块加入对新管理器的集成，完善文档说明。  

**🎯 影响范围**：  
- `python/sglang/multimodal_gen/runtime/pipelines/ltx_2_pipeline.py`（核心逻辑）  
- `python/sglang/multimodal_gen/runtime/pipelines_core/*`（stage、LoRA、pipeline 基类）  
- `python/sglang/multimodal_gen/runtime/server_args.py`（CLI 参数、默认值解析）  
- `docs/diffusion/compatibility_matrix.md`（使用说明）  
- 单元/集成测试 `gpu_cases.py`（覆盖新模式）  

**🔍 技术洞察**  

| 维度 | 影响分析 |
|------|----------|
| **架构影响** | - 新增 **LTX2TwoStageDeviceManager** 作为独立的职责层，负责 DiT（Diffusion Transformer）在两阶段间的显存驻留与切换；<br>- 通过 `pipeline._device_manager` 把设备管理职责从 pipeline 主逻辑中抽离，符合单一职责原则，提升可维护性。<br>- 引入 `mode`（original / snapshot / resident）以及自动模式检测逻辑，对 `ServerArgs` 做统一化处理，保持参数来源一致（CLI ↔ 环境变量 ↔ 默认推断）。 |
| **性能影响** | - **resident**：两段 DiT 常驻 GPU，切换仅改引用，几乎零延迟，适合显存充裕（≥130 GiB）机器；<br>- **snapshot**：利用 CPU 快照 + 异步 H2D 预取，显存占用约 ½‑⅔，且在非低显存模式下可在 Stage‑1 运行期间提前把 Stage‑2 拉到 GPU，实现计算‑传输重叠，显著降低切换延迟；<br>- **snapshot‑low‑vram**（自动在 H100 类 <130 GiB GPU 上启用）：在切换前主动把 Stage‑1 释放回 CPU，进一步压显存峰值，但会增加一次额外的 D2H/H2D，导致相对更高的切换时延。<br>- 通过 `pin_cpu_memory` 与 `torch.Stream` 的非阻塞拷贝，实现了 **异步预取** 与 **CPU‑GPU 双向共享**，在典型 8‑GPU 工作负载中预期可提升 10‑30% 吞吐量（取决于显存容量）。 |
| **安全考虑** | - 所有模式均通过 **白名单** (`VALID_MODES`) 与环境变量校验，避免任意字符串导致异常路径。<br>- 新增的环境变量 `SGLANG_LTX2_SNAPSHOT_LOW_VRAM_MODE`、`SGLANG_LTX2_SNAPSHOT_RELEASE_EMPTY_CACHE`、`SGLANG_LTX2_TWO_STAGE_DEVICE_MODE` 均在读取时使用 `get_bool_env_var` / 统一转换，防止注入攻击。<br>- 代码未引入外部 I/O，仅在显存管理层面操作 `torch` 张量，未发现新的安全漏洞。 |
| **可维护性** | - 使用 `defaultdict`, `@property` 等 Python idiom，保持向后兼容；<br>- 将 LoRA 合并、阶段切换逻辑封装在 `DeviceManager`，未来可在不改动 pipeline 代码的情况下增删模式。<br>- 增加了大量日志（`logger.info`），有助于问题排查。 |
| **兼容性** | - 默认模式保持 **snapshot**（与原实现最接近），因此对现有用户行为影响最小。<br>- 通过 `original` 模式保留完全向后兼容路径，用户可显式回退以进行调试。 |
| **测试覆盖** | - 在 `gpu_cases.py` 中加入对 `original` 模式的显式测试，确保在多 GPU 环境下仍可正常启动。<br>- 其他模式未在 CI 中强制跑，但代码路径已通过单元测试覆盖关键函数（如 `_capture_module_cpu_snapshot`、`_schedule_phase_prefetch`）。 |

**⚠️ 潜在风险**  

1. **显存泄漏 / 未释放**  
   - 在 `release_premerged_transformers` 中仅释放 `transformer_2`，若用户在 `resident` 模式手动切换 `stage1`/`stage2`，可能出现未释放的显存残留。  
   - 建议在 `release_premerged_transformers` 中加入对 `transformer`（stage‑1）在不再需要时的显式 `to("cpu")` 与 `empty_cache`。  

2. **异步拷贝竞争**  
   - 使用自建 `torch.Stream` 进行预取，若在同一请求中多次调用 `prefetch_stage2_after_stage1`（例如异常重入），可能产生未同步的拷贝，引发数据不一致或 CUDA 错误。  
   - 可以在 `prefetch_stage2_after_stage1` 前检查 `self._phase_ready_events` 是否已经存在对应的 `event`，若已有则跳过调度。  

3. **低显存模式误判**  
   - 自动判断 H100 类显卡的阈值硬编码为 130 GiB，未来新卡（如 H800）可能突破此阈值，导致误选 `resident` 并触发 OOM。  
   - 建议提供覆盖阈值的环境变量（如 `SGLANG_LTX2_RESIDENT_AUTO_ENABLE_MEM_GB`），或在日志中明确提示实际检测值。  

4. **CPU‑GPU 迁移期间的并发请求冲突**  
   - 目前 `DeviceManager` 只维护单一全局状态 `_active_phase`，在高并发请求下可能出现跨请求相互干扰（如一个请求在 `stage1` 期间释放 `stage2` 导致另一请求在 `stage2` 找不到 GPU 参数）。  
   - 需要在多请求调度层（如 `RequestScheduler`) 中确保同一模型实例的 `pipeline` 只被单个请求占用，或在 `DeviceManager` 引入引用计数/锁机制。  

5. **日志噪声**  
   - `logger.info` 在 `resident`、`snapshot` 启动时会频繁输出，可能导致日志膨胀。建议在生产环境默认改为 `debug`，仅在 `--verbose` 时提升。  

**💡 关注建议**  

| 对象 | 建议 |
|------|------|
| **开发者** | - 为 `DeviceManager.release_premerged_transformers` 增加对 `stage1` 的显存回收路径；<br>- 在 `prefetch_stage2_after_stage1` 加入幂等检查，防止重复调度；<br>- 考虑将显存阈值抽象为可配置项，以兼容未来新显卡。 |
| **运维/用户** | - 在显存紧张（<130 GiB）机器上保持默认 `snapshot`，或显式使用 `--ltx2-two-stage-device-mode snapshot`；<br>- 如显存充裕且追求极致吞吐，可尝试 `resident`；<br>- 若遇到奇怪的阶段切换延迟，可临时切换到 `original` 进行排查。 |
| **CI/测试** | - 增加针对 `snapshot` 与 `resident` 两种模式的端到端 GPU 测试，覆盖显存占用、切换时延以及并发请求的正确性；<br>- 在非 CUDA 环境（CPU‑only）验证自动退回到 `snapshot`（不会报错）。 |
| **文档** | - 在兼容矩阵与 README 中补充 “显存需求 vs. 设备模式” 表格，帮助用户快速决定使用哪种模式；<br>- 明确列出环境变量的含义与默认值。 |

### refactor(moe): de-duplicate triton MoE runner path into shared helpers (#23019)
**SHA**: `5f7aee7` | 🔗 [查看提交](https://github.com/sgl-project/sglang/commit/5f7aee726a10056dd9e5e2fbd72b3bdf3aeed7bf)

**🎯 变更类型**：重构  

**⚡ 重要程度**：🔴高  

**📋 变更摘要**  
此次提交把原先散落在 `sglang.srt.layers.moe.fused_moe_triton` 与 `sglang.srt.layers.moe.moe_runner` 下的 MoE（Mixture‑of‑Experts）运行时实现抽取出来，统一到新的 **`sglang.srt.layers.moe.moe_runner.triton_utils`** 包中。所有 benchmark、内部调用以及 TritonRunner 的实现均改为通过该共享层导入 `fused_moe`、配置、对齐等辅助函数，实现了路径去重并简化了配置流程。  

**🎯 影响范围**  
- `sglang/srt/layers/moe/fused_moe_triton`（旧实现）  
- `sglang/srt/layers/moe/moe_runner/triton.py`（核心运行时）  
- 多个 benchmark 脚本（`benchmark/kernels/...`）  
- 新增 `triton_utils` 包及其内部配置子目录  
- 相关的 `__init__`、`moe_align_block_size`、`override_config` 逻辑  

**🔍 技术洞察**  

- **架构影响**  
  - **模块化提升**：`triton_utils` 成为 MoE‑Triton 相关功能的单一入口，消除了跨目录的循环导入风险。  
  - **公共 API 变更**：原有 `sglang.srt.layers.moe.fused_moe_triton.fused_moe` 的 import 路径被废除，对外依赖该路径的代码（包括第三方插件或自定义脚本）将在升级后失效，需要迁移到 `moe_runner.triton_utils.fused_moe`。  
  - **配置统一**：`override_config/get_config` 从旧的 `__init__` 移至 `triton_utils/__init__`，运行时状态的保存方式保持不变，但更易于在不同 MoE runner（如 CUDA、HIP、CPU）之间共享。  

- **性能影响**  
  - **运行时逻辑未改动**：核心计算仍由原来的 `invoke_fused_moe_kernel` 与 `moe_sum_reduce_*` 系列实现，只是调用路径改为 `_fused_moe_kernel_sequence`（在 `triton_utils.fused_moe` 中统一封装）。  
  - **潜在微小开销**：额外的函数包装层（如 `_prepare_fused_moe_run`）会产生一次 Python 调用，但在大多数 GPU‑bound 场景下对整体吞吐几乎不可感知。  
  - **分支精简**：原先在 `triton.py` 中针对 CUDA、HIP、CPU、XPU 的大量条件分支已被抽离到 `triton_utils`，有助于 compiler 在内部实现中进行更激进的 JIT 优化。  

- **安全考虑**  
  - 代码仅涉及内部计算与文件路径（`get_config_file_name`），未引入网络、文件写入或外部输入解析，安全风险保持不变。  
  - 通过统一的 `override_config` 上下文管理器，降低了因全局变量误用导致的状态泄露风险。  

**⚠️ 潜在风险**  

1. **向后兼容性**  
   - 直接使用旧 import 路径的用户会在运行时抛出 `ImportError`。如果这些路径被文档或示例所引用，升级过程需要提供迁移指引或兼容 shim。  

2. **环境变量依赖**  
   - 原实现中 `_MOE_PADDING_SIZE`、`SGLANG_USE_AITER`、`SGLANG_MOE_PADDING` 等环境变量的读取被移除或迁移到 `triton_utils`。若用户在特定硬件（如 HIP）上依赖这些变量，可能出现行为差异。  

3. **硬件分支丢失**  
   - 旧代码在 `triton.py` 内部对 `is_hip`、`is_xpu`、`cpu_has_amx_support` 等做了显式分支并加载对应 kernel（`sgl_kernel`、`vllm_ops`）。新实现假设这些细节已在 `triton_utils.fused_moe` 中统一处理；如果该层没有完整覆盖某些平台（尤其是 ROCm/AMD），可能导致运行时崩溃或性能退化。  

4. **测试覆盖不足**  
   - 只修改了 benchmark 及内部调用，若项目的 CI 没有针对所有硬件平台的回归测试，隐藏的兼容性问题可能在用户现场显现。  

**💡 关注建议**  

- **迁移文档**：在官方 README 与 API 文档中添加 **“从 5f7aee7 起，`fused_moe` 的 import 路径已迁移到 `sglang.srt.layers.moe.moe_runner.triton_utils`”** 的说明，并提供兼容 shim（例如在旧路径下添加轻量转发 import）以降低升级阻力。  

- **回归测试**：  
  1. 在 CI 中加入针对 **CUDA、HIP、CPU (AMX)** 与 **XPU** 四个平台的完整单元/集成测试，确保 `triton_utils` 能正确加载对应 kernel。  
  2. 对 `SGLANG_USE_AITER`、`SGLANG_MOE_PADDING` 等环境变量进行回归验证，确保行为保持一致。  

- **性能基准**：保留现有 benchmark，新增对比报告，确保“去重”改动未对吞吐、延迟产生可感知的回退。  

- **异常捕获**：在 `triton_utils.fused_moe` 的入口函数（如 `_fused_moe_kernel_sequence`）加入 **明确的异常信息**，帮助用户快速定位因缺失平台实现导致的运行时错误。  

- **逐步废弃**：如果计划在未来彻底移除旧路径，建议采用 **两阶段 deprecation**：先在 1‑2 个 minor 版本中提供警告，随后在下一个 major 版本中删除。  

---  

**结论**：本次重构显著提升了代码组织度，降低了维护成本，对功能本身没有直接性能或安全影响。但由于涉及全局 import 路径与平台分支的迁移，需要格外注意向后兼容、硬件覆盖以及 CI 测试的完整性，以防止在多平台部署中的潜在回归。

### [MLX] Support radix cache (#21509)
**SHA**: `26ae7b8` | 🔗 [查看提交](https://github.com/sgl-project/sglang/commit/26ae7b8bd730044d5344f6ca8fc48d0140e26063)

**🎯 变更类型**：功能增强 / 架构变更  
**⚡ 重要程度**：🔴 高  

**📋 变更摘要**  
本次提交为 MLX（Apple Silicon）后端引入 **Radix Cache**（基于前缀共享的 KV 缓存）实现。核心工作包括：  
1. 新增 `sglang.srt.hardware_backend.mlx.kv_cache` 包，提供 **ContiguousKVCache、PoolBackedCache、OffsetCache、MLXAttentionWrapper、BatchedDecodeContext** 等 KV 缓存抽象以及模型层的注意力包装器。  
2. `MlxModelRunner` 改写为基于 **MlxKVPool**（统一的 KV 池）进行 KV 复用，支持 **prefill、extend、decode** 三种调用模式，并实现自动池大小估算、内存固定（wired）限制、批量解码上下文。  
3. 调整 `bench_one_batch.py` 使其在基准模式下禁用 radix cache 并使用新的 `prefill` 接口。  
4. 为 MLX 后端的 **模型 Runner Stub** 与 **TpModelWorker** 添加与 KV 池大小同步的逻辑，保证调度器仍可在无模型权重加载的情况下正常工作。  

**🎯 影响范围**  
- `python/sglang/srt/hardware_backend/mlx/kv_cache/*`（全新 KV 缓存层）  
- `python/sglang/srt/hardware_backend/mlx/model_runner.py`（核心运行时）  
- `python/sglang/srt/hardware_backend/mlx/model_runner_stub.py`（轻量 stub）  
- `python/sglang/srt/hardware_backend/mlx/tp_worker.py`（分布式 worker）  
- `python/sglang/bench_one_batch.py`（基准脚本）  
- 相关调度器代码（`TokenToKVPoolAllocator`、`RadixCache`）在运行时将开始使用 `MlxKVPool`。  

**🔍 技术洞察**  

| 维度 | 影响描述 |
|------|----------|
| **架构影响** | 1. 引入 **KV 池 + 前缀共享（RadixCache）**，把所有请求的 KV 按槽（slot）统一存放，降低每个请求独立 KV 缓存的内存碎片化。<br>2. `MLXAttentionWrapper` 在 `BatchedDecodeContext` 存在时替换原始注意力实现，实现 **多请求** 同步解码（batch‑size>1）。<br>3. `MlxModelRunner` 从“每个请求维护独立 KV 列表”转为“统一 KV 池 + 按请求的`ContiguousKVCache`/`PoolBackedCache`”。<br>4. 新增 `init_kv_pool`、`flush_all_decode_kv`、`_sync_new_kv_to_pool` 等接口，使调度器（scheduler）可以在**prefill/extend**阶段随时写回或读取共享 KV。 |

| **安全考虑** | - 代码不涉及网络 I/O、文件写入，只在内存中操作 KV，未引入明显安全漏洞。<br>- 使用 `threading.local()` 保存 `BatchedDecodeContext`，若同一进程中多线程共享同一 `MlxModelRunner`（如在 `TpModelWorker` 中），需保证**上下文的 set/clear 成对出现**，否则可能泄漏前一次请求的序列长度信息；但当前实现均在 `decode_batch` 中 use‑finally 包裹，风险可控。<br>- 自动池大小计算依赖系统可用内存值，若系统报告异常可能导致 **pool_size 过小**，进而触发运行时 `MlxKVPool.set_kv` 越界异常。建议在生产环境加上上限校验。 |
