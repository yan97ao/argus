### Helios: Real Real-Time Long Video Generation Model (#19782)
**SHA**: `115f879` | 🔗 [查看提交](https://github.com/sgl-project/sglang/commit/115f8799582a4f90f36209416a859a742ed24702)

**🎯 变更类型**：功能增强 / 架构变更  
**⚡ 重要程度**：🔴高  

**📋 变更摘要**  
本次提交在 `sglang` 项目中加入了全新的 **Helios** 长时视频生成模型及其完整的推理流水线。主要工作包括：  
1. 新增 Helios 的模型配置、调度器、Transformer 实现以及对应的采样参数。  
2. 在 `registry.py` 中注册了 Helios 的模型、采样参数和 pipeline 配置，使其可以通过统一入口加载。  
3. 实现了基于 **Chunked Denoising**、**多尺度历史记忆**、**CFG‑Zero‑Star** 等技术的专属 pipeline（`HeliosPipeline`、`HeliosPyramidPipeline`）以及对应的 **Stage‑2/3 超分辨率** 支撑。  
4. 扩展了多模态配置层（`configs/models/dits/__init__.py`、`pipeline_configs/helios.py`）并提供了默认采样参数（`sample/helios.py`）。  

**🎯 影响范围**  
- `sglang/multimodal_gen/models/dits/helios.py`（核心 3D Transformer）  
- `sglang/multimodal_gen/runtime/schedulers/scheduling_helios.py`（专属调度器）  
- `sglang/multimodal_gen/runtime/pipelines/helios_pipeline.py` 与 `pipelines_core/stages/model_specific_stages/helios_denoising.py`（完整推理流水线）  
- `sglang/multimodal_gen/configs/models/dits/helios.py`、`pipeline_configs/helios.py`、`sample/helios.py`（配置、采样参数）  
- `sglang/multimodal_gen/registry.py`（模型/采样参数注册）  
- 相关 `__init__` 与导入路径的更新  

**🔍 技术洞察**  

| 维度 | 影响分析 |
|------|----------|
| **架构影响** | - 引入了 **HeliosTransformer3DModel**，在原有 DiT 系列模型之上增加了 3D 卷积、3 D RoPE、multi‑term memory patch、scale‑shift 调制等子模块，是一次较大的模型层级扩展。<br>- 新增 **HeliosScheduler**，支持 Euler、UniPC、DMD 三种求解器，且实现了 **动态时间位移 (dynamic shifting)** 与 **阶段式 (stage) 调度**，对已有调度抽象做了兼容扩展。<br>- pipeline 采用 **ComposedPipelineBase + LoRAPipeline** 的组合方式，保持现有 LoRA、FSDP、CPU‑offload 机制，同时在 `HeliosChunkedDenoisingStage` 内部自行管理 chunk、历史缓存、cfg‑zero‑star 逻辑，实现了 **Stage‑1、Stage‑2、Stage‑3** 三阶段的可插拔。<br>- `registry.py` 通过 `model_detectors` 自动匹配模型名称，确保 Helios 与已有模型（Flux、Hunyuan 等）共存。 |
| **性能影响** | - **显存**：Helios 使用 40 层、40 heads、每 head 128 dim，参数量约 1 B（取决于配置），并在每个块中并行加载 **short/mid/long** 历史 patches（3 个 3D Conv），会显著增加显存占用。<br>- **计算**：Chunked‑denoising 将长视频拆分为若干 9 帧块（默认），每块独立推理，整体 **推理步数 = chunk × num_inference_steps**，在长视频（>10 s）下计算成本呈线性增长。<br>- **并行**：利用现有 **FSDP** 与 **TP**（列/行并行 Linear）进行参数切分，减轻单卡负担；但历史 patch 的卷积、RoPE 生成在每个 chunk 中仍需在同一设备上完成，跨卡同步成本未显式优化。<br>- **优化点**：`HeliosChunkedDenoisingStage` 中对 `torch.cuda.empty_cache()` 的显式调用可帮助 CPU‑offload 场景释放显存；`optimized_scale`（CFG‑Zero‑Star）使用单次向量运算，开销可接受。 |
| **安全考虑** | - 代码中未出现外部输入的直接系统调用或文件写入，安全风险主要来自 **模型加载**（潜在的恶意权重）和 **FP16/BF16** 转换导致的数值不稳定。<br>- 新增的 `HeliosScheduler.set_shift` 允许在运行时修改 `shift` 参数，若未对外部接口做访问控制，可能被恶意请求利用来制造 **Denial‑of‑Service**（过大 shift 产生异常 sigma）。<br>- `HeliosRotaryPosEmbed` 在 `forward` 中使用 `torch.device(str)` 的方式创建网格，若输入 `device` 为非合法字符串，可能触发异常抛出。建议在公共 API 前添加 **参数校验**。 |
| **可维护性** | - 新增文件数量大（≈ 3 k 行），涉及多个层级（模型、调度器、pipeline、stage、config），代码结构保持与现有 DiT 系列一致，便于后期统一管理。<br>- 参数映射 (`param_names_mapping`) 与 `reverse_param_names_mapping` 已在 `HeliosArchConfig` 中声明，方便 **hf‑to‑sglang** 权重转换。<br>- 代码中大量硬编码（如 `patch_size=(1,2,2)`、`rope_dim=(44,42,42)`、`num_attention_heads=40`），若后续需要变体模型，需手动编辑配置文件。建议抽离为可通过 JSON/YAML 调整的 **model config**。 |

**⚠️ 潜在风险**  
1. **显存泄漏 / OOM**  
   - Helios 的历史 patch、Chunked denoising 以及 Pyramid SR 都会在同一进程中累计大量中间特征。若 `num_latent_chunks` 较大（长视频），极易触发 OOM，尤其在不使用 FSDP 或 CPU‑offload 的单卡部署。  
2. **数值不稳定**  
   - `HeliosScheduler` 中的 `sigma`、`alpha`、`lambda` 计算涉及对数、指数，极端 sigma（接近 0）可能导致 **NaN**。`step_unipc` 中对 `sigma` 的 `clamp(min=1e-8)` 已有防护，但在动态 shifting 场景仍需监控。  
3. **API 兼容**  
   - 新增的 `HeliosChunkedDenoisingStage` 依赖 `batch.guidance_scale`、`batch.num_inference_steps` 等字段，若旧版 `Req` 对象缺失这些属性，调用会抛异常。  
4. **模型安全**  
   - 通过 `model_detectors` 自动匹配 “helios” 关键字，如果 HF 里出现恶意的同名模型，可能误加载不受信任权重。建议在公开模型列表中加白名单校验。  
5. **并行/分布式**  
   - `HeliosScheduler._fsdp_shard_conditions` 仅在 `HeliosConfig` 中定义了 `is_blocks`，但历史 patch 的卷积层未在该列表中出现，可能导致 **FSDP** 进程间不均衡的 shard，有性能下降或梯度同步错误的风险。  

**💡 关注建议**  

| 对象 | 建议 |
|------|------|
| **开发者** | - 为长视频（> 30 s）提供 **显存预估** 与 **自动分块** 参数（如 `max_chunk_frames`），防止 OOM。<br>- 在 `HeliosScheduler.set_shift`、`HeliosRotaryPosEmbed.forward` 等入口加入 **输入校验**（类型、范围），提升鲁棒性。<br>- 将模型结构中的硬编码抽象为可配置文件，便于后续模型家族（如 Helios‑Lite）复用。<br>-

### fix: Properly return abort error for streaming requests if the abort is triggered by scheduler (#19357)
**SHA**: `d22c6a3` | 🔗 [查看提交](https://github.com/sgl-project/sglang/commit/d22c6a3847e92a9abb5d9c7209db190ad88b01c4)

**🎯 变更类型**：Bug修复  
**⚡ 重要程度**：🔴高  
**📋 变更摘要**：  
1. 在 OpenAI 兼容的 Chat 与 Completion 流式接口中，完善了对调度器主动 **abort**（中止）场景的处理。  
2. 当返回的 `finish_reason.type` 为 `"abort"` 时，依据 `status_code` 与 `message` 生成符合 OpenAI 错误格式的流式错误块并立即终止后续流；同时在正常结束时仍会输出 `[DONE]`。  
3. 新增单元测试覆盖此行为，确保错误块正确生成、流被提前停止且返回的错误码与信息一致。

**🎯 影响范围**：  
- `sglang/srt/entrypoints/openai/serving_chat.py`  
- `sglang/srt/entrypoints/openai/serving_completions.py`  
- 相关单元测试 `test_serving_chat.py`、`test_serving_completions.py`  

**🔍 技术洞察**  

| 维度 | 影响描述 |
|------|----------|
| **架构影响** | - 仅在 **流式生成** 的异步生成函数内部加入异常路径处理，未改动核心调度或模型执行逻辑。 <br>- 引入 `http.HTTPStatus` 用于统一错误码，保持与 OpenAI API 规范一致。 <br>- 通过 `self.create_streaming_error_response` 复用已有错误包装方式，保持代码复用率。 |
| **性能影响** | - 仅在异常分支执行额外的字典访问与对象创建，开销极低（毫秒级）。 <br>- `break` 提前终止流式迭代，反而可节省后续不必要的 token 生成计算。 |
| **安全考虑** | - 正确返回 HTTP 状态码与错误信息，防止客户端因“异常隐藏”而误判服务可用性。 <br>- 错误信息来源于调度器提供的 `message`，未在代码中自行构造，降低信息泄露风险。 <br>- 使用 `HTTPStatus` 防止手写错误码导致的误导或冲突。 |
| **可维护性** | - 将 `finish_reason` 的取值改为 `dict.get`，避免 `KeyError`。 <br>- 将 abort 处理逻辑集中在少数几行，易于后续扩展（如添加自定义错误字段）。 <br>- 新增覆盖 abort 场景的单元测试，提升回归安全性。 |

**⚠️ 潜在风险**  

1. **异常提前退出的语义**  
   - `break` 只会退出当前 `async for` 循环（即生成器内部），随后代码在 `finally` 或外层循环中是否仍会输出 `[DONE]` 取决于实现细节。若后续没有统一的 `[DONE]` 输出，客户端可能收到不完整的 SSE 流。当前测试期望 `[DONE]` 在 `break` 之后出现，需要确认 `yield "data: [DONE]\n\n"` 仍会被执行。  
2. **`self.create_streaming_error_response` 接口兼容性**  
   - 若后续对该方法签名或返回结构做改动，当前实现可能产生不兼容的错误块。建议在调用前加上类型检查或保持该方法的稳定性。  
3. **并发流的多索引场景**  
   - 当一次请求返回多个 `index`（如多模态或并行生成），当前逻辑在检测到 `abort` 后 `break` 会终止整个生成器，可能导致其它正常完成的索引也被截断。若业务需要保留非‑abort 索引的输出，需要改为只停止对应索引的流，而不是整体 `break`。  
4. **错误码的默认值**  
   - 当 `status_code` 缺失时使用 `HTTPStatus.INTERNAL_SERVER_ERROR`，这会返回 500。若业务希望返回更细粒度的错误（如 429、503），需要调度器提前填充。  

**💡 关注建议**  

1. **确认 DONE 包装**  
   - 检查 `OpenAIServingChat/_generate_chat_stream` 与 `OpenAIServingCompletion/_generate_completion_stream` 在异常 `break` 之后的控制流，确保统一在函数结束前 `yield "data: [DONE]\n\n"`，或在 `finally` 中显式发送 DONE。  
2. **细化 abort 场景**  
   - 若未来需要区分不同类型的 abort（如 超时、资源不足），建议在 `finish_reason` 中加入 `subtype` 字段，并在错误响应中映射到对应的 OpenAI 错误码。  
3. **多索引兼容**  
   - 对于返回 `finish_reason` 为 abort 但仅针对单个 `index` 的情况，可考虑在 `finish_reasons` 结构中记录该索引的错误并继续迭代其它索引，避免整条流被提前终止。  
4. **保持错误响应统一**  
   - 建议将 `create_streaming_error_response` 的返回格式（包括 `error.code` 与 `error.message`）在项目文档中明确，防止不同入口产生不一致的错误结构。  
5. **回归测试**  
   - 在 CI 中加入对 **非 abort** 正常结束以及 **异常异常**（如 `finish_reason` 缺失 `type`）的覆盖，确保改动不会意外影响已有流式逻辑。

### Feature:Reserve HTTP server port before model loading to immediately detect port  conflicts instead of failing after several minutes of model loading. (#17754)
**SHA**: `cedb86a` | 🔗 [查看提交](https://github.com/sgl-project/sglang/commit/cedb86a950e32af1b22b670213776ccfe15e54b6)

**🎯 变更类型**：功能增强  
**⚡ 重要程度**：🟡 中  
**📋 变更摘要**：在 `launch_server` 入口中提前调用新工具函数 `_prebind_listening_socket` 绑定 HTTP 监听端口，以在模型加载阶段即捕获端口冲突。实现通过 `uvicorn.run(..., fd=socket.fileno())` 复用该已绑定的 socket，并在 finally 里统一关闭，避免长时间模型加载后才报错。  
**🎯 影响范围**：  
- `python/sglang/srt/entrypoints/http_server.py`（服务器启动流程）  
- `python/sglang/utils.py`（新增 `_prebind_listening_socket`）  
- 相关的日志、错误提示以及多 tokenizer‑worker 分支的参数传递逻辑  

**💡 关注建议**  

1. **异常处理**：当前在 `launch_server` 的 `try` 块外只捕获 `finally`，若 `_prebind_listening_socket` 抛出异常会导致后续资源未清理。建议在预绑定前加入 `try/except`，将异常包装为用户可读的提示并确保 `reserved_socket` 只在成功后进入 `finally`。  
2. **跨平台兼容**：`socket.SO_REUSEPORT` 在 Windows 上可能不存在或行为不同，虽然已有 `hasattr` 检查，但后续 `uvicorn` 使用 `fd` 仍需确认 Windows 环境下是否支持。建议在 CI 中加入 Windows 测试或在文档中注明仅在类 Unix 系统上保证。  
3. **资源释放**：`reserved_socket.close()` 放在 `finally` 已正确，但在多 worker（`tokenizer_worker_num > 1`）模式下，子进程会继承该 socket。确认子进程不会因提前关闭导致 “Bad file descriptor” 错误，或在子进程启动前使用 `socket.detach()` 交给 uvicorn。  
4. **日志与错误信息**：新增函数已在绑定成功后打印 INFO，失败时抛出详细 RuntimeError。建议统一使用项目的 `logger`（已引用）而非直接 `print`，并在文档补充 “端口被占用的排查命令”。  
5. **测试覆盖**：添加单元测试：① 端口已被占用时函数抛出预期错误；② 正常绑定后 `uvicorn` 能通过 `fd` 正确启动（可使用 `uvicorn.Server` 的 `run_in_thread` 模式模拟）；③ 多 tokenizer‑worker 场景下 socket 仍可被子进程使用。  

**对用户**：启动时若指定的 `--host/--port` 已被占用，将立即报错并提供排查命令，避免耗时的模型加载后才发现冲突。使用低于 1024 的端口仍需管理员权限。  

总体而言，此改动显著提升了启动失败的可响应性，只要注意跨平台行为和异常路径的完整清理即可安全发布。
