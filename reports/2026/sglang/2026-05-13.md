### Port KV Compression V2 from deepseek_v4_dev (#24890)
**SHA**: `e2290b1` | 🔗 [查看提交](https://github.com/sgl-project/sglang/commit/e2290b155aa03189fedf2b507f84f04648d68906)

**🎯 变更类型**：功能增强 / 重构  
**⚡ 重要程度**：🔴 高  

**📋 变更摘要**  
- 为 DeepSeek‑V4 KV 压缩实现全新 “V2” 方案（c4 / c128 双倍率、在线 c128），包括：
  - 新的 C++/CUDA 压缩、解压、计划生成、在线状态‑Pool 和 FlashMLA 存储 kernel（约 5 k 行代码），并加入 PDL‑wait/trigger 以实现跨 kernel 同步。  
  - 新的 `fused_norm_rope`、`fused_q_norm_rope`、`fused_q_indexer_rope_hadamard_quant` 等 fused kernel，实现 **RMSNorm + RoPE + FP8 量化** 的一次性算子，极大降低内存带宽与 kernel 启动次数。  
  - Python JIT 接口 (`sglang.jit_kernel.dsv4`) 完全封装上述 kernel，提供 `compress_forward`、`compress_norm_rope_store`、`fused_q_norm_rope`、`fused_k_norm_rope_flashmla`、`fused_q_indexer_rope_hadamard_quant` 等统一入口。  
  - 新的计划（prefill / decode）生成逻辑：CPU + GPU 双阶段（`plan_compress_prefill_kernel0` + `plan_compress_prefill_kernel_1`），支持 **GPU‑side 直接生成（GPU‑input）** 以及 **传统 CPU‑side 生成** 两种路径；兼容旧的 “legacy” 环形 Ring Buffer 方案。  
  - 将后端 `DeepseekV4AttnBackend` 与新 compressor‑V2 通过 `CompressorBackendMixin` 关联，统一 `forward_unified` 接口；在 `deepseek_v4` 模型中切换为新的 fused‑norm‑rope‑inplace 实现。  
  - 环境变量 `SGLANG_OPT_USE_COMPRESSOR_V2`（默认 `True`）用于全局开关；相应代码路径在 `deepseek_v4_backend.py` 与 `compress.py` 中做了分支。  
  - 其它小改动：RMSNorm 支持非‑contiguous stride、去除过时 `SGLANG_OPT_USE_OVERLAP_STORE_CACHE`、将 KV‑cache 写入完全交给 K‑kernel（冗余 flag 删除）。

### fix(nixl): close file descriptors after each FILE transfer (#24671)
**SHA**: `c32f2dc` | 🔗 [查看提交](https://github.com/sgl-project/sglang/commit/c32f2dc1ac0c454e8701ca8775ac48e1114dc19c)

**🎯 变更类型**：Bug修复  
**⚡ 重要程度**：🔴高  

**📋 变更摘要**  
- 在 NIXL 文件传输路径中加入 `file_fds` 收集与 `finally` 统一关闭，实现对每次 FILE 类型传输后打开的文件描述符的显式释放。  
- 为 `HiCacheNixl` 的单元测试加入文件描述符计数检查，确保在 `set/get`、`batch_set/batch_get` 等操作后不存在 fd 泄漏。  
- 调整测试配置以覆盖更多初始化参数（如 `pp_rank`、`attn_cp_rank`、`extra_config`），并简化文件注册流程。

**🎯 影响范围**  
- `python/sglang/srt/mem_cache/storage/nixl/hicache_nixl.py`（核心文件传输实现）  
- `python/sglang/srt/mem_cache/storage/nixl/test_hicache_nixl_storage.py`（对应单元测试）  
- 依赖 NIXL 后端的任何上层缓存/模型推理逻辑间接受影响（尤其在高并发/长时间运行场景下的文件描述符使用）。

**🔍 技术洞察**  

- **架构影响**  
  - 新增 `file_fds` 采集与统一关闭（`finally` 块），让文件描述符的生命周期与单次传输强绑定，避免跨调用残留。  
  - 通过 `self.is_zero_copy = False` 标记，显式声明当前实现未采用 zero‑copy，防止上层误判优化路径。  
  - 测试层面增加了对 `pp`/`attn_cp` 参数的显式配置，为未来多阶段（pipeline parallel）或注意力并行扩展预留兼容性。

- **性能影响**  
  - 关闭文件描述符的开销几乎可以忽略，与之前的泄漏导致的系统级资源枯竭相比是巨大提升。  
  - 重新组织异常处理和 descriptor 初始化的代码块，使得成功路径的分支更少，略微提升了传输准备阶段的执行效率。  
  - 额外的 `os.listdir("/proc/self/fd")` 用于测试，仅在 CI/开发机器上运行，不影响生产性能。

- **安全考虑**  
  - 明确在异常路径下释放文件句柄，降低因资源未释放导致的资源耗尽（DoS）风险。  
  - 关闭文件描述符前未检查是否已经被 `file_manager` 其它逻辑关闭，若出现双关闭可能触发 `OSError`，但在当前实现中 `close_file` 只做 `os.close`，双关闭会抛异常并被捕获在 `finally` 之前的异常处理流程，风险可接受。  
  - 添加对 `/proc/self/fd` 的依赖，只有 Linux 环境支持；在非 Linux 环境运行测试时会报错，需要在 CI 中限定平台或提供 fallback。

**⚠️ 潜在风险**  

1. **双关闭风险**：如果 `file_manager.close_file` 已在其他位置被调用，`finally` 中再次关闭同一 fd 可能抛出异常。  
2. **平台兼容性**：`_open_fds` 读取 `/proc/self/fd` 只在 Linux 可用，可能导致测试在 macOS/Windows 环境失效。  
3. **异常掩盖**：`finally` 的 `for fd in file_fds` 循环在关闭时未捕获异常，若某个 fd 已无效会导致整个传输流程报错并返回 `False`，可能隐藏原始传输错误。  
4. **业务回退**：在极端情况下，`self.is_zero_copy = False` 可能导致上层逻辑选择非最佳路径，需确认未引入意外性能回退。

**💡 关注建议**  

- **代码审计**：确认 `file_manager.close_file` 对已关闭 fd 的行为（是否安全幂等），必要时在 `finally` 中加入 `try/except OSError` 保护。  
- **跨平台适配**：为测试中的 `_open_fds` 添加平台检测，或改为使用 `resource.getrlimit(resource.RLIMIT_NOFILE)` 之类的跨平台方式。  
- **回归验证**：在 CI 中加入长时间/高并发的 NIXL 传输场景，验证 fd 计数始终维持在预期范围，防止新代码在特定负载下仍有泄漏。  
- **文档更新**：在 HiCache NIXL 使用指南中注明 “每次 FILE 传输后文件句柄会被自动关闭”，并提醒用户在自定义 `file_manager` 实现时保持相同语义。  
- **监控预警**：若产品环境中启用了 NIXL，建议加入 fd 使用量的监控报警，确保即使出现意外泄漏也能及时捕获。

### Fix Eagle draft decode positions (#25015)
**SHA**: `48fc26a` | 🔗 [查看提交](https://github.com/sgl-project/sglang/commit/48fc26a814b8a71ffcb1e6a66b2a57393fd75e04)

**🎯 变更类型**：Bug 修复  
**⚡ 重要程度**：🟡 中  

**📋 变更摘要**  
此次提交修正了 Eagle draft 解码时 `forward_batch.positions` 的更新逻辑。原实现把位置自增放在 `draft_forward` 循环内部，导致在异常回退或缓存恢复后位置会累计多次，从而出现 decode‑position 偏移错误。现在在 `run_once` 中先备份 `out_cache_loc` 与 `hidden_states`，调用 `draft_forward` 完成后恢复这些字段，并在返回前统一将 `positions` 回退 `speculative_num_steps‑1`，而 `draft_forward` 只在真正消耗 token 时一次性自增。

**🎯 影响范围**  
- `python/sglang/srt/speculative/eagle_draft_cuda_graph_runner.py`  
- `python/sglang/srt/speculative/eagle_worker.py`  
- `python/sglang/srt/speculative/eagle_worker_v2.py`  

**💡 关注建议**  
1. **功能验证**：新增或扩展单元测试，确保在多步 speculative 推理、回滚及缓存恢复场景下 `forward_batch.positions` 与实际 token 序号保持一致。  
2. **回归风险**：检查其它使用 `forward_batch.positions`（如 KV 缓存对齐、日志打印）的路径，确认不会因位置提前回退导致越界或错位。  
3. **并发/多流**：如果在 CUDA graph 多 stream 环境下复用 `forward_batch`，确认位置的统一回退不会产生竞争条件。  
4. **文档/注释**：在 `draft_forward` 中添加说明 “仅在实际 token 消耗后自增”，以及在 `run_once` 中解释 “恢复后统一回退”，便于后续维护。  

整体来看，此次修改定位准确，修复了潜在的 off‑by‑one 错误，影响范围局限于 Eagle speculative 路径，风险可通过上述测试和代码审查控制。
