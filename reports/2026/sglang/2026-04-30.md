### [4/N] Quantization Refactor: AWQ schemes and Kernel call and weight init split (#21126)
**SHA**: `577dbc4` | 🔗 [查看提交](https://github.com/sgl-project/sglang/commit/577dbc4ab9761ba1f0548e97fa83ca4b70eb134b)

**🎯 变更类型**：重构 / 功能增强  
**⚡ 重要程度**：🔴高  
**📋 变更摘要**：本次提交对 AWQ（4‑bit 量化）实现进行深度重构，拆分出 GPU、NPU 两套专用 kernel，实现量化 scheme 抽象层并迁移原有 `awq.py` 的全部逻辑到 `awq/` 包下的多个模块。新增 Marlin‑加速路径、Ascend（NPU）路径以及对应的权重初始化、后处理流程；同时更新量化配置注册、线性层/MoE 层的创建逻辑以及多处 import 路径。  

**🎯 影响范围**  
- `sglang/srt/layers/quantization/awq/`（核心实现、scheme、config）  
- `sglang/srt/hardware_backend/gpu/quantization/awq_kernels.py`  
- `sglang/srt/hardware_backend/npu/quantization/awq_kernels.py`  
- `sglang/srt/layers/linear.py`（权重加载列表）  
- `sglang/srt/layers/quantization/__init__.py`（注册导出）  
- 相关模型文件的 import 路径（deepseek、longcat 等）  
- 测试用例 `test/registered/quant/test_awq_dequant.py`  

---

### 🔍 技术洞察  

| 维度 | 分析 |
|------|------|
| **架构影响** | 1. 引入 **Scheme 抽象层**（`AWQLinearSchemeBase` / `AWQMoESchemeBase`），把权重创建、后处理、前向调用统一交给 scheme 实例，使得不同硬件（CUDA/ROCm/XPU/NPU）只需实现对应 kernel 即可。<br>2. 原先单文件的 `awq.py` 被拆分为 **config、kernel、scheme、init** 四层，代码职责更清晰，降低循环依赖风险。<br>3. `QuantizationConfig` 通过 `get_quant_method` 返回具体 `LinearMethod`，但现在返回的 `LinearMethod` 只负责 delegating 给 scheme，进一步解耦。 |

| **可维护性** | - **模块化**：每个硬件后端拥有独立 `*_kernels.py`，便于针对特定平台做性能调优或 bug 修复。<br>- **统一 Scheme API**：后续新增如 `QLoRA、GPTQ` 等量化方式，只需实现对应 Scheme，即可插拔。<br>- **代码体积**：删除了 966 行单体实现，新增约 1.4k 行分模块代码，整体可读性提升但新增抽象层也略增学习曲线。 |

### ⚠️ 潜在风险  

1. **破坏向后兼容**  
   - `AWQMarlinLinearMethod` 名称已从 `WEIGHT_LOADER_V2_SUPPORTED` 中移除，使用老配置文件或脚本可能找不到对应方法。  
   - `CPUAWQConfig` 替换为 `AWQCPUConfig`，外部项目直接导入旧类会导致 `ImportError`。  
2. **平台检测错误**  
   - `is_xpu / is_hip / is_npu` 判定错误（例如在混合环境中误判），会导致加载到不兼容的 kernel 并触发 runtime error。  
3. **自定义 Op 注册**  
   - `register_custom_op_from_extern` 在 CUDA 路径仍然依赖 `sgl_kernel` 动态库，一旦库缺失或版本不匹配会在模型加载时抛异常。  
4. **权重后处理不一致**  
   - `process_weights_after_loading` 在不同 Scheme 中实现方式不同，若某个模型在权重文件中使用了与当前平台不匹配的 packing（比如使用了 Marlin weight 但在 NPU 上运行），会导致数值错误。  
5. **测试覆盖不足**  
   - 目前仅新增了对 Triton 实现的 import 调整的单元测试，缺少对 Marlin、Ascend（NPU）路径的端到端推理验证。

### fix(qwen3_5): broadcast per-tensor scale in _make_packed_weight_loader for FP8 models (#23062)
**SHA**: `936c9c2` | 🔗 [查看提交](https://github.com/sgl-project/sglang/commit/936c9c2355961d6b46e6b30fa3f94edf3810e393)

**🎯 变更类型**：Bug修复  
**⚡ 重要程度**：🔴高  

**📋 变更摘要**  
- 修正了 Qwen‑3.5 FP8 模型在 `_make_packed_weight_loader` 中对 **per‑tensor scale**（标量或单元素张量）的处理逻辑。原实现仅对完全 0‑维张量进行特殊处理，并在出现单元素张量时抛出断言；新实现统一使用 `numel() == 1` 判定，并将该张量通过 `.view(-1)` 广播至每个逻辑 shard。  
- 新增 216 行单元测试，覆盖标量、单元素张量、嵌套单元素张量的广播以及普通权重切分、非 tuple shard_id 直通等场景，确保行为回归安全。

**🎯 影响范围**  
- `sglang/srt/models/qwen3_5.py` 中的 **_make_packed_weight_loader**（仅 Qwen‑3.5 系列模型）。  
- 依赖该加载器的 FP8 量化模型加载路径。  
- 关联的单元测试目录 `test/registered/unit/models/`（用于 CI 运行）。

**🔍 技术洞察**  

| 维度 | 影响说明 |
|------|----------|
| **架构影响** | *局部* 改动，仅影响权重加载器的内部分支逻辑。未改变模型整体架构、接口或调用链。对其它模型（如 Qwen‑2、Llama 等）无直接影响。 |
| **性能影响** | - **正向**：通过 `view(-1)` 创建对同一底层存储的视图，避免不必要的拷贝，保持原有的 **O(1)** 开销。<br>- **负向**：在广播多个 shard 时会生成 `len(loaded_shard_id)` 个引用列表，极端情况下（上百 shards）可能导致轻微的列表创建开销，但相较于权重拷贝仍然可忽略。 |
| **安全考虑** | - 该改动不涉及网络、权限或加密，仅是内部张量处理。<br>- 唯一潜在安全点是 **共享同一底层张量**：如果后续 `original_loader` 对张量进行原位修改（极不常见），所有 shard 会同时受到影响。当前代码路径下 `original_loader` 只做拷贝或读取，风险可接受。 |
| **可维护性** | - 使用 `numel() == 1` 统一判定，更直观且覆盖 0‑维、1‑维以及任意形状的 **单元素张量**，降低了未来出现类似 edge‑case 时的维护成本。<br>- 新增的 **单元测试** 完整覆盖了所有分支，提升了回归安全性。 |

**⚠️ 潜在风险**  
1. **共享视图风险**：`chunks = [loaded_weight.view(-1)] * len(loaded_shard_id)` 生成的列表中每个元素指向同一张量视图。如果后续加载器在处理过程中对张量做原位修改（例如 `chunk.mul_(scale)`），会意外影响所有 shard。  
2. **非 tuple shard_id 的兼容性**：代码仍假设 `loaded_shard_id` 为可迭代对象（tuple/list）才能进行广播；如果未来出现自定义 iterable（如 generator）可能导致 `len()` 报错。  
3. **异常路径未覆盖**：若 `loaded_weight` 为空张量（`numel()==0`）或出现负数维度的异常情况，仍会进入普通切分路径，可能触发后续的 shape mismatch。  

**💡 关注建议**  
- **代码审查**：确认 `original_loader`（传入的 `weight_loader`）在处理 **chunk** 时不执行原位修改；如有必要，可在此处加入 `chunks = [loaded_weight.view(-1).clone() for _ in range(...)]` 以彻底避免共享。  
- **文档更新**：在模型文档或注释中说明 “per‑tensor scale 必须是单元素张量，加载器会自动广播”，帮助使用者理解期望的权重量化格式。  
- **CI 保障**：保持已新增的单元测试在每次 CI 运行中执行，尤其在引入新模型或修改 `PerTensorScaleParameter` 逻辑时。若有性能基准测试（benchmark）需要，监测广播列表创建对大规模 shard 场景的微幅影响。  
- **回滚策略**：若在极端多‑shard 环境（>256）出现意外行为，可临时回退至原 `assert` 逻辑并在 CI 中加入对应的压力测试。  

---  
*总体而言，此次修复消除了 FP8 模型在加载 per‑tensor scale 时的断言错误，提升了模型兼容性与稳健性，风险可控且已通过充分的单元测试验证。*

### [Apple Silicon] [MLX] MLX decode partial overlap scheduling for generation (async eval) (#22416)
**SHA**: `3272af2` | 🔗 [查看提交](https://github.com/sgl-project/sglang/commit/3272af2f001b5fa9e24db65e45087d0865f08ae3)

**🎯 变更类型**：功能增强 / 性能优化  
**⚡ 重要程度**：🔴高  

**📋 变更摘要**  
本次提交为 Apple Silicon 引入基于 MLX (Metal‑MLX) 的异步推理调度实现。通过在 `Scheduler` 中混入 `SchedulerMlxOverlapMixin`，在 GPU 上保持两条惰性计算图（lazy‑eval）并行执行，实现 **prefill/extend/decode 的流水线重叠**。为此新增了 `MlxPending*` 数据结构、模型跑步器的 lazy‑API（`*_start` / `*_finalize`），以及 TP worker 的异步入口 (`async_forward_batch_generation_mlx`、`async_chained_decode_mlx`），并对文档、服务器参数及调度器输出处理做了相应适配。

**🎯 影响范围**  
- `docs/platforms/apple_metal.md`、`docs_new/docs/hardware-platforms/apple_metal.mdx`（文档更新）  
- `python/sglang/srt/hardware_backend/mlx/`：`model_runner.py`、`scheduler_mixin.py`、`tp_worker.py`（核心实现）  
- `python/sglang/srt/managers/scheduler.py`（调度器主流程）  
- `python/sglang/srt/managers/scheduler_output_processor_mixin.py`（结果处理）  
- `python/sglang/srt/server_args.py`（启动参数兼容）  
- 相关上下文模块：`sglang.srt.hardware_backend.mps`、`sglang.srt.utils.tensor_bridge`（MLX 开关）  

**🔍 技术洞察**  

- **架构影响**  
  - 引入 **`SchedulerMlxOverlapMixin`** 把 MLX 专用的 overlap 调度逻辑抽离为 mixin，保持原有 `Scheduler` 多继承结构不变。  
  - `MlxPending*` 系列结构把 **惰性图** 与请求状态解耦，使调度器能够在 CPU 完成前一批的 bookkeeping 时 **提前构建并提交** 下一批图，实现 CPU‑GPU 双向流水线。  
  - `TpModelWorker` 通过 `async_forward_batch_generation_mlx` / `async_chained_decode_mlx` 提供 **非阻塞** 前向入口；最终在 `finalize_mlx_result` 中统一收敛为 `GenerationBatchResult`，保持与已有 torch 路径的接口兼容。  
  - 为避免 MPS/CUDA 流的依赖，MLX 路径在 `Scheduler.init_overlap` 与 `run_event_loop` 中直接跳过 stream 初始化，改用 `mx.async_eval`/`mx.eval` 同步。

- **性能影响**  
  - **消除 decode 之间的空闲 Gap**：在 steady‑state 时 GPU 上始终保持两条 lazy‑graph，CPU 只在前一批的 token 完成后进行调度，理论上可以把 **GPU 利用率提升至接近 100%**，尤其在 Apple Silicon 上的 Metal/MLX 后端。  
  - 新的 **`_decode_step_ct`** 与周期性 `mx.clear_cache()` 能在长时间运行后控制显存碎片，避免因 lazy‑graph 累积导致显存泄漏。  
  - 通过 **`async_chained_decode_mlx`** 的依赖追踪，前后两步的 KV cache 写入/读取在同一次 GPU 调度中完成，无需额外的 memcpy，显著降低 latency。  
  - 需要注意 **mx.async_eval** 的调度开销（当前实现一次性把所有缓存 buffer 传入），在极大 batch（> 256）时可能出现额外的 CPU‑GPU 同步成本。

- **安全考虑**  
  - 代码路径主要涉及 **环境变量 `SGLANG_USE_MLX=1`** 的打开与关闭，没有引入网络、文件或系统调用的新增风险。  
  - 采用 **`DynamicGradMode`** 包装调度循环，保持与 PyTorch 自动求导模式的一致性，未改变权限模型。  
  - 唯一潜在安全点是 **MLX 的 lazy‑eval API** 可能在异常路径（如 `mx.async_eval` 抛异常）下泄露未释放的 GPU 句柄；建议在异常捕获层面加入 `mx.clear_cache()` 兜底。

| 风险点 | 说明 | 严重度 |
|--------|------|--------|
| **线程安全** | `MlxPendingPrefill/Extend/Decode` 仍使用 **`threading.local()`** 的全局上下文 (`BatchedDecodeContext`)；如果 scheduler 在多线程模式下并发调用，可能出现上下文冲突。 | 中 |
| **缓存一致性** | `decode_batch_start_chained` 假设前一步的 `offset` 已在 Python 端同步；若 `mx.async_eval` 在 GPU 端出现异常或提前返回，可能导致 `offset` 与实际 KV 状态不匹配。 | 中 |
| **内存泄漏** | `mx.async_eval` 接收的大量 `mx.array`（包括所有 KV cache state）若未及时 `mx.clear_cache()`，在长时间服务中可能导致显存增长。当前在 `decode_batch_finalize` 中每 256 步清理一次，但 prefill/extend 路径未统一计数。 | 中 |

**💡 关注建议**  

1. **功能验证**  
   - 在开启与关闭 `use_cuda_graph` 两种模式下跑完整的 LoRA 推理/微调基准，确保数值结果保持一致。  
   - 对比 CUDA‑graph 之前和修复后的运行日志，确认不再出现 “CUDA graph breaking” 警告或异常。

2. **兼容性检查**  
   - `zero_()` 只能在张量上执行，确保 `sgemm.seg_indptr` 的 dtype 与 `0` 的类型匹配（当前行为与原标量赋值等价）。若以后改为非整型张量，需显式使用 `fill_(0)` 或 `torch.tensor(0, dtype=…)`。

3. **防止类似问题再现**  
   - 将项目中所有对张量的标量直接写（如 `tensor[idx] = value`）审查为使用切片或 `fill_`/`zero_` 系列原位操作，以保持 CUDA‑graph 的连贯性。可在代码审查模板或 lint 规则中加入 “avoid scalar assignment on CUDA‑graph tensors” 的检查。  

4. **性能影响**  
   - `zero_()` 对单元素切片的开销与标量赋值几乎等同，几乎不影响整体推理吞吐。若有极端的循环写入场景，仍建议一次性使用 `torch.zeros_like` 或 `fill_` 进行批量写入。
