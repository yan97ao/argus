### [NPU] update best practice docs from testcase (#28621)
**SHA**: `4e1d251` | 🔗 [查看提交](https://github.com/sgl-project/sglang/commit/4e1d25117bf3134a3a43d55f461ea8708853f0b3)

**🎯 变更类型**：功能增强（文档层面的最佳实践更新）  
**⚡ 重要程度**：🔴 高  

**📋 变更摘要**  
- 为 Ascend NPU 平台的使用手册加入了大量最佳实践示例，包括分布式启动脚本、`trust-remote-code`、`attention-backend ascend`、`device npu` 等新参数的说明。  
- 移除了旧的 *proxy* 警告块并在相关章节统一给出 **unset proxy** 的操作提示，降低因代理拦截导致 EvalScope 重试超时的概率。  
- 更新了章节锚点（`id`）以统一命名（如 `single-node-pd-mixed`），并在多处示例中加入 `--disaggregation-transfer-backend ascend`、`--trust-remote-code`、`--attention-backend ascend`、`--device npu` 等必备参数。  
- 对大量模型（DeepSeek‑R1、GLM‑5.1、Kimi‑K2.6、MiniMax‑M2.5、Qwen 系列等）对应的部署脚本进行细化，新增 `NODE_IPS`、`dist-init-addr`、`node-rank`、`cuda-graph` 等调优参数。  

**🎯 影响范围**  
- `docs_new/docs/hardware-platforms/ascend-npus/` 下所有 MDX 文档。  
- 使用 Ascend NPU 部署 SGLang 的研发、运维及用户。  
- 依赖文档生成（Sphinx/MDX）和交叉引用的 CI 流程。  
- 与文档对应的示例脚本（`sglang.launch_server`、`sglang_router.launch_router`）的调用方式。  

**🔍 技术洞察**  

| 维度 | 影响 | 说明 |
|------|------|------|
| **架构影响** | **分布式部署示例强化** | 增加了 `NODE_IPS`、`dist-init-addr`、`node-rank` 等参数，帮助用户快速搭建 **PD Disaggregation**（预填 & 解码分离）模式，提升在多节点 Ascend 集群上的可扩展性。 |
| **性能影响** | **调优指南显著提升** | 明确了 `--attention-backend ascend`、`--device npu`、`--disaggregation-transfer-backend ascend`、`--cuda-graph-bs`、`--prefill-max-requests`、`--max-mamba-cache-size` 等关键调参，可帮助用户在不同卡数、数据集规模下获得 **更低 TPOT**（如 GLM‑5.1 从 50 ms 降至 20 ms）。 |
| **安全考虑** | **`trust-remote-code` 引入潜在风险** | 文档鼓励使用 `--trust-remote-code` 来加载第三方模型代码，若未加足够提示，用户误启用可能导致远程代码执行。<br>**Proxy 解除**：统一的 `unset http(s)_proxy` 可能干扰已配置的安全代理，需提醒仅在本地调试时使用。 |
| **可维护性** | **大量重复模板** | 相同的环境变量与启动参数在多个模型文档中复制，后续若 SGLang 参数变更，需要同步更新大量 MDX，维护成本上升。 |
| **文档完整性** | **锚点统一** | 将原先的 `single-node`、`pd-disaggregation` 等 id 替换为统一 `single-node-pd-mixed`、`pd-disaggregation` 等，防止外部引用失效。 |

**⚠️ 潜在风险**  
1. **安全风险** – `--trust-remote-code` 默认开启可能导致未受信任模型代码执行；若用户在生产环境误用，可能被植入恶意代码。  
2. **文档/代码同步** – 参数名称或默认值若在后续代码中更改（如 `--attention-backend`、`--cuda-graph-bs`），文档不会自动更新，易产生误导。  
3. **代理环境冲突** – 统一的 `unset https_proxy` 等指令在企业网络中可能导致服务不可达；应在文档中明确 “仅在本机调试” 使用。  
4. **交叉引用失效** – 大幅改动锚点 ID，若其他文档或外部链接仍使用旧 ID，会导致跳转失效。  
5. **环境变量硬编码** – 示例中硬编码 `HCCL_BUFFSIZE=...`、`STREAMS_PER_DEVICE=32` 等，若硬件规格不同需手动修改，增加用户出错概率。  

**💡 关注建议**  
- **安全提示**：在每个使用 `--trust-remote-code` 的示例前加入明显警示（如 “仅在可信模型来源或本地调试时使用”。）  
- **文档与代码同步**：在 CI 中加入文档校验脚本，检测 MDX 中出现的 SGLang 参数是否仍在 `sglang/launch_server.py` 的 `argparse` 中。  
- **模板抽象**：将常用的环境变量、启动参数抽离为可复用的 **MDX 片段**（如 `{% include "ascend_common_flags.mdx" %}`），降低维护成本。  
- **代理使用说明**：在文档中加入 “如果需要通过代理访问外部仓库，请在完成部署后自行恢复代理变量”。  
- **锚点兼容**：保留旧锚点的重定向（在 MDX 中使用 HTML `<a id="single-node">` 指向新的 `single-node-pd-mixed`），避免已有外链失效。  
- **示例验证**：在 PR CI 中加入一个小型脚本，使用 `docker run`

### [UnifiedTree]: Rollback mamba hicache test to direct io backend (#28904)
**SHA**: `70883cb` | 🔗 [查看提交](https://github.com/sgl-project/sglang/commit/70883cb1b04f80965414c86932b3cc8f267b605e)

**🎯 变更类型**：功能限制 / Bug 修复  
**⚡ 重要程度**：🟡 中  
**📋 变更摘要**：在 `MambaPoolHost` 中强制仅支持 `layout='page_first_direct'` 与 `io_backend='direct'`，其余组合抛 `ValueError`；相应测试改为使用直接 I/O 并将原先依赖写回 staging kernel 的测试标记为 `skip`，并给出待修复的 TODO。  

**🎯 影响范围**  
- `python/sglang/srt/mem_cache/memory_pool_host.py`（核心缓存层）  
- `test/registered/hicache/test_qwen35_hicache.py`、`test/registered/radix_cache/unified_radix_tree/test_unified_radix_cache_kl_mamba.py`、`test/registered/unit/mem_cache/test_hicache_staged_write_back_dispatch.py`（测试套件）  

**💡 关注建议**  
1. **配置同步**：所有使用 MambaPool 的调用必须显式设定 `--hicache-mem-layout=page_first_direct` 且 `--hicache-io-backend=direct`，否则在运行时会直接报错。  
2. **兼容性检查**：若项目中仍依赖旧的 `page_first` 布局或 kernel I/O，需要提前改写配置或降级至非‑Mamba 缓存实现，防止启动失败。  
3. **文档/警告**：建议在配置文档中加入此限制说明，并在代码中使用 `DeprecationWarning` 而非直接 `ValueError`，以便平滑迁移。  
4. **后续恢复**：当前跳过的写回 staging 测例表明还有未解决的 kernel bug，待修复后应恢复并在 CI 中验证。  
5. **性能监控**：直接 I/O 可能改变吞吐和延迟，建议在关键工作负载下做基准比较，确保新路径满足性能预期。

### Migrate all ServerArgs fields to Annotated style, reduce add_cli_args by ~2400 lines (#28919)
**SHA**: `b28e990` | 🔗 [查看提交](https://github.com/sgl-project/sglang/commit/b28e990161a389fad5aaadd16cebd5b0b80c5f26)

**🎯 变更类型**：功能增强 / 重构  
**⚡ 重要程度**：🟡 中  

**📋 变更摘要**  
本次提交将 `ServerArgs` 的所有字段迁移到 `Annotated`（`typing.Annotated`+`Arg(...)`）声明方式，核心实现集中在 `arg_utils.add_cli_args_from_dataclass`，新增 `const` 参数并在生成 `argparse` 参数时统一使用 `dest` 绑定字段名，从而把原本约 2400 行的手写 CLI 注册代码压缩至几百行。相应的单元测试亦同步更新，验证别名、`cli_name`、`nargs='?'`+`const`、布尔可选、JSON 解析等边界行为。

**🎯 影响范围**  
- **核心模块**：`python/sglang/srt/arg_groups/arg_utils.py`、`python/sglang/srt/server_args.py`（字段迁移、别名、默认值处理）  
- **测试**：`test/registered/unit/test_server_args_cli_metadata.py`、新增 `test_server_args_migration.py`  
- **使用方**：所有直接访问 `ServerArgs` 实例属性或通过 CLI 传参的脚本/服务。

**💡 关注建议**  
1. **兼容性检查**：迁移后字段名统一为短别名（如 `dp_size`），旧代码若仍使用 `data_parallel_size` 需适配或通过已保留的别名 `--data‑parallel-size` 访问。  
2. **文档同步**：更新用户手册中的 CLI 参数列表，说明新别名、`cli_name` 与 `dest` 的映射规则，尤其是 `--tensor‑parallel-size` → `tp_size`。  
3. **代码审查**：确认项目内部其它模块（如调度、监控）仍通过属性名读取 `ServerArgs`，避免因字段名变化产生 `AttributeError`。  
4. **测试覆盖**：在 CI 中保留旧别名的回归测试（如 `--data-parallel-size`），防止未来不经意删除导致破坏兼容。  
5. **性能验证**：虽然 CLI 生成逻辑更简洁，但仍建议在大规模启动场景下跑一次启动基准，确保解析耗时未出现异常增长。  

总体而言，此次迁移大幅提升可维护性和可扩展性，只要留意属性名与别名的双向兼容，即可平滑升级。

   - 极限 batch = 1、`length == group_topk` 与 `length < group_topk` 的边界；  
   - 大批量（> 32 K）batch 以及 `out_cols` 接近 64 K（核最大 grid 维度）时的异常捕获。  

**总体结论**：本次 PR 为 SGLang 引入了高效、可 JIT 编译的 pool‑level top‑k 核心，代码结构清晰、接口易用，并通过细粒度的单元测试验证功能。重点关注共享内存配置、模板限制以及异常路径的防护，按上述建议补充检查与文档，可进一步提升鲁棒性与可维护性。

**SHA**: `106d293` | 🔗 [查看提交](https://github.com/sgl-project/sglang/commit/106d2930a6c57fb2ad8960c149a4f256f822f3f9)

总体来看，此改动通过细粒度事件同步显著降低了写后读的阻塞时间，预计在高并发 decode 场景下可提升吞吐约 5%‑10%。后续观察实际 GPU 利用率和延迟指标，以验证效果并进一步优化。

### [Feature] Add graceful scheduler shutdown; free hisparse host buffer on exit (#28779)
**SHA**: `e6722c7` | 🔗 [查看提交](https://github.com/sgl-project/sglang/commit/e6722c751b0f5d846c779a1eb2dea5ba6133b411)

**🎯 变更类型**：功能增强（Graceful Scheduler Shutdown）  
**⚡ 重要程度**：🟡 中  

**📋 变更摘要**  
1. 新增 `ShutdownReq` 与调度器的 `gracefully_exit` 标志，使调度器可在收到统一的关闭请求后安全退出循环。  
2. 为 HiSparse 主机缓冲区实现 `HiSparseCoordinator.destroy()` 与 `HostPoolBase.destroy()`，在退出前显式 `cudaHostUnregister`，避免因内核在 SIGKILL 时长时间阻塞导致的进程卡死。  
3. `TokenizerManager` 在 SIGTERM 看门狗结束时发送 `ShutdownReq` 并等待调度器完成资源释放。  
4. 关联测试 `test_dsa_glm5_hisparse.py` 改为在 `tearDownClass` 中主动执行 graceful shutdown，防止外部 SIGKILL 卡住。  

**🎯 影响范围**  
- `python/sglang/srt/managers/scheduler.py`（事件循环、请求分发、资源释放）  
- `python/sglang/srt/managers/hisparse_coordinator.py`（新增 `destroy`）  
- `python/sglang/srt/mem_cache/pool_host/base.py`（新增 `destroy`、host‑unregister 实现）  
- `python/sglang/srt/mem_cache/pool_host/common.py`（新增 `_cuda_host_unregister`）  
- `python/sglang/srt/managers/io_struct.py`（新增 `ShutdownReq`）  
- `python/sglang/srt/managers/tokenizer_manager.py`（SIGTERM 处理）  
- 测试 `test/registered/models_e2e/test_dsa_glm5_hisparse.py`  

**💡 关注建议**  

1. **资源释放完整性**：`Scheduler.release_host_resources()` 只在 `gracefully_exit` 为真且未异常时调用。建议在 `run_scheduler_process` 的 `finally` 块里再做一次防御性调用，以防异常路径下因未设置 `gracefully_exit` 而泄漏。  

2. **线程安全**：`HiSparseCoordinator.destroy()` 在同步两个流后调用 `mem_pool_host.destroy()`，但 `mem_pool_host` 可能仍被其他线程使用（例如正在进行的 `write`）。请确认在调度器主循环已停止所有生产者后再调用，或在 `destroy` 内加锁防止并发访问。  

3. **跨平台兼容**：`_cuda_host_unregister` 依赖 `torch.cuda.cudart()`，在 HIP 环境（ROCm）同样使用 `torch.cuda` 接口，这在 CI 中已加 `_is_cuda or _is_hip` 检查。若未来引入其他后端（如 NPU），需相应补充对应的 unregister 实现。  

4. **异常恢复**：`handle_shutdown` 仅设置 `gracefully_exit=True`，但如果在同一次循环中已有待处理的请求，循环会在下一轮才退出。若有长时间阻塞的 I/O（如大批量 `recv_requests`），可考虑在 `handle_shutdown` 中直接清空请求队列或中断阻塞的 `recv_requests`，避免延迟关闭。  

5. **测试可靠性**：新增的 `tearDownClass` 通过 `process.terminate()` 再 `kill_process_tree` 作为兜底。建议在 CI 环境加入对 `ShutdownReq` 正常路径的断言（如确认 `gracefully_exit` 已被置位），防止因实现细节回退导致测试超时。  

6. **文档与日志**：在 `handle_shutdown` 与 `destroy` 中加入 INFO 级别日志，便于运维在岗位上观察是否真的走到了 graceful shutdown 路径。  

总体而言，本次改动解决了 HiSparse 大尺寸 pinned host 缓冲在强制杀进程时的卡死问题，提升了服务的可控退出能力。只要注意上述并发/异常边界，即可安全上线。
