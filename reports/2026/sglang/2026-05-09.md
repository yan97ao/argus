### [DSV4] Cherry pick missing commits from deepseek_v4 branch and enhance tests (#24793)
**SHA**: `ef5e9f8` | 🔗 [查看提交](https://github.com/sgl-project/sglang/commit/ef5e9f8abab18609c42af4b1a1844ab21f76be0e)

**🎯 变更类型**：功能增强（兼容 DeepSeek‑V4、扩展 OpenAI 协议、优化权重加载、CI 流程升级）  
**⚡ 重要程度**：🔴高  

**📋 变更摘要**  
- 将 DeepSeek‑V4（DSV4）相关缺失提交 cherry‑pick 并在 CI、文档、测试中加入完整支撑，新增 `ci_install_dsv4_dep.sh` 脚本以安装 FlashMLA + DeepEP（DeepEP All‑to‑All + DP‑attention）依赖。  
- 扩展 OpenAI `ChatCompletionRequest` schema，加入 `reasoning_effort="max"`，并在单元测试中校验其行为。  
- 完善 V4/V32 工具调用解析：支持自闭合 `<｜DSML｜invoke name="x"/>` 以及跨块流式解析，提升零参工具调用的正确性。  
- 为模型权重加载器加入 `weight_loader_drop_cache_after_load` 参数，使用 `posix_fadvise(DONTNEED)` 在加载完 safetensors 后释放页面缓存，以防 RL 场景下的 CPU OOM。  
- 在 CI 配置、slash‑command 处理以及测试套件中加入针对 DSV4 的 GPU‑B200/H200 场景，确保新特性在 CI 中得到验证。  

**🎯 影响范围**  
- **核心运行时**：`sglang/srt/model_loader/*`、`sglang/srt/function_call/*`、`sglang/srt/entrypoints/openai/protocol.py`。  
- **服务器配置/CLI**：`sglang/srt/server_args.py`（新增 CLI 参数）以及相应的 CI 脚本。  
- **测试/CI**：新增/修改 `test/registered/dsv4/*`、CI workflow、slash‑command 处理。  
- **文档**：`docs_new/src/snippets/autoregressive/deepseek-v4-deployment.jsx`。  

**🔍 技术洞察**  

- **架构影响**  
  - **模块化扩展**：通过新增 `DeepSeekV4Detector` 与改进的正则实现，工具调用解析层保持向后兼容（V32 与 V4 共享代码），不影响已有模型路径。  
  - **配置统一**：`ServerArgs` 增加 `weight_loader_drop_cache_after_load`，CLI 与配置文件统一入口，保持配置一致性。  
  - **依赖层**：`ci_install_dsv4_dep.sh` 将 FlashMLA 与 DeepEP 合并到同一脚本，避免原 `ci_install_flash_mla.sh` 的冗余，简化 CI 镜像构建路径。  

- **性能影响**  
  - **权重加载**：`posix_fadvise(DONTNEED)` 在加载完每个 safetensors shard 后主动释放页面缓存，可显著降低 RL/多模型加载场景下的内存峰值，提升整体吞吐。  
  - **解析效率**：正则从一次 `re.findall` 改为 `re.finditer` + 结构化拆解，避免一次性捕获大量子组导致回溯，提升流式 token 解析的时间复杂度，尤其在长文本或高并发调用时更稳健。  
  - **CI 性能**：合并 FlashMLA 与 DeepEP 安装步骤，且在 CUDA 13 环境下提供兼容性 symlink，减少 CI 失败率与构建时间。  

- **安全考虑**  
  - **新增 CLI 参数** 仅是调度层面的缓存释放，不涉及特权操作，安全风险极低。  
  - **CI 脚本** 中使用 `apt-get install` 时加入了检测已安装包的回退逻辑，防止因意外的系统包冲突导致 CI 泄露或中断。  
  - **OpenAI schema 扩展**：新增 `reasoning_effort="max"` 为内部约定，仍在 `ChatCompletionRequest` 验证中受限于 Pydantic，未向外开放未知值，保持输入安全性。  

**⚠️ 潜在风险**  
1. **文件缓存释放**：`posix_fadvise(DONTNEED)` 仅在支持的 Linux 内核上有效，若在不支持的系统上调用，可能被静默忽略，导致预期的 OOM 缓解失效。  
2. **正则兼容性**：新 `invoke_regex` 引入命名捕获组，旧版 Python (<3.8) 不支持；虽然项目已定位在 Python 3.9+，但若有用户自行在旧环境使用，可能报错。  
3. **CI 脚本依赖**：DeepEP 构建依赖 `gdrcopy`、`libfabric-dev` 等系统库，若底层镜像缺失相应的内核模块或驱动，CI 将在安装阶段失败。  
4. **模型配置冲突**：`weight_loader_drop_cache_after_load` 默认 `False`，若用户在多进程/多 GPU 环境误开启，可能导致已加载的权重再次被回收，引起后续推理崩溃。  

**💡 关注建议**  
- **文档完善**：在 README 与部署指南中明确 `--weight-loader-drop-cache-after-load` 的使用场景（推荐 RL/大模型并行），并提醒仅在 Linux 系统有效。  
- **回归测试**：在 *nightly* 与 *per‑commit* CI 中保持两套路径（B200 与 H200），确保新解析器在不同 GPU、不同 `tp`/`dp` 组合下均通过全部 12 项 ServerSanity 与 GSM8K 准确度门限。  
- **监控与日志**：为 `_drop_file_cache_after_load` 添加 `info` 级别日志（当前仅 `debug`），便于生产环境排查是否真的触发了缓存释放。  
- **向后兼容验证**：在发布前跑一次完整的 **DeepSeek‑V3/V32** 流式调用测试，确认自闭合解析不会误伤旧模型的 `<｜DSML｜invoke>` 长格式。  
- **安全审计**：尽管风险低，仍建议在 CI 中加入 `shellcheck` 对新增脚本进行静态检查，以捕获潜在的路径注入或未捕获的错误返回码。

### feat(scheduler): add adaptive queue-based prefill delayer trigger (#23189)
**SHA**: `560829a` | 🔗 [查看提交](https://github.com/sgl-project/sglang/commit/560829a171c033b6bc1c89fb6ab31535bf638b03)

**🎯 变更类型**：功能增强  

**⚡ 重要程度**：🔴高  

**📋 变更摘要**  
在 `PrefillDelayer` 中引入了 **自适应队列长度触发** 的延迟机制，并通过 `--prefill-delayer-queue-min-ratio` 与 `--prefill-delayer-max-delay-ms` 两个新服务器参数对其进行可选控制。调度器在计算是否允许 prefill 时，会额外考虑 **等待队列长度** 与 **运行请求数** 的比例，避免因大量碎片化的 prefill 导致的 GPU 利用率低下；同时提供 **墙钟时间上限** 防止 TTFT（首次完整时间）无限增长。  

**🎯 影响范围**  
- `sglang/srt/managers/prefill_delayer.py`（核心调度逻辑）  
- `sglang/srt/managers/schedule_policy.py`、`sglang/srt/managers/scheduler.py`（调度器向 delayer 传递 `waiting_queue_len`）  
- `sglang/srt/server_args.py`（CLI 参数新增）  
- 文档 `docs/advanced_features/server_arguments.md`（参数说明）  
- 单元测试 `test/registered/scheduler/test_prefill_delayer.py`（覆盖新特性）  

**🔍 技术洞察**  

- **架构影响**  
  - 在 All‑Gather 信息缓冲中由 **4 → 5** 个 `int64`，新增 `waiting_queue_len` 项。该改动对已有跨节点通信协议是向后兼容的，只增加了极小的带宽/内存开销。  
  - `PrefillDelayer` 现在支持 **双触发**（槽位触发 + 队列触发），并在 `self._queue_trigger_enabled` 为 `True` 时激活。实现了 **可选** 功能，默认保持原有行为，降低对现有部署的侵入性。  
  - 调度器在创建 `SchedulePolicy` 时捕获当前 `waiting_queue` 长度，保证每一次 prefill 判决都能看到最新的排队情况。  

- **性能影响**  
  - **正向提升**：在高并发、碎片化请求的场景下，等待队列触发可以将多个小的 prefill 合并为更大的 batch，提升 GPU 利用率与总体吞吐。  
  - **潜在延迟**：若 `queue_min_ratio` 设置不当或 `max_delay_ms` 过大，可能导致单个请求的 TTFT 被人为延长。实现已加入 **时间安全阈值**（默认 5 s），在超过阈值时强制放行，防止极端延迟。  
  - **开销**：每次调度多额外一次整数值的 All‑Gather 与几行条件判断，开销在毫秒级以下，可忽略。  

- **安全考虑**  
  - 参数均为数值型，不涉及文件、网络或执行路径的变更。唯一需关注的是 **参数校验**：如果用户传入异常的负数或非数值，可能导致除零或逻辑错误。目前实现依赖 `argparse` 的类型检查，未额外限制范围。  

**⚠️ 潜在风险**  

1. **误配置导致迟滞**  
   - `prefill_delayer_queue_min_ratio` 超过 1 或设置为极小值（如 0.001），会使队列触发几乎永不满足，导致大量请求一直被延迟。  
   - `prefill_delayer_max_delay_ms` 设为极大值或 `None`（内部默认 5000 ms）可能在极端负载下产生不可接受的 TTFT。  

2. **跨节点状态不一致**  
   - `waiting_queue_len` 由调度器本地 `len(self.waiting_queue)` 提供，若多节点间的排队策略不一致，可能出现 **全局视图不对称**，导致部分节点提前放行而其他节点仍在等待。  

3. **通信负载微增**  
   - All‑Gather 维度从 4 → 5，极端大规模部署（数千 GPU）下会产生额外的网络流量，虽小但在带宽极限场景需留意。  

4. **回滚兼容性**  
   - 旧版二进制如果尝试读取新增字段会报维度不匹配；因此在混合部署（新旧二进制共存）时需确保全部节点升级到同一版本。  

**💡 关注建议**  

- **参数校验**：在 `ServerArgs.add_cli_args` 中加入上下界检查，建议 `0 < queue_min_ratio <= 1`、`max_delay_ms > 0`，并在日志中给出警告。  
- **监控与观察**：在生产环境开启 `prefill_delayer_wait_seconds_buckets` 与 `prefill_delayer_forward_passes_buckets`，收集 **等待时长分布** 与 **延迟触发次数**，评估新触发带来的实际吞吐提升与 TTFT 变化。  
- **渐进式启用**：建议先在测试或低流量环境把 `queue_min_ratio` 设为 `0.1~0.2`，观察排队长度与批次大小的变化，再根据业务特性逐步调大。  
- **文档完善**：在官方文档中明确说明 **“opt‑in”** 的含义、典型取值以及对 **高并发 vs 低延迟** 场景的建议，以防用户误以为默认开启。  
- **回滚方案**：若出现异常延迟，可通过 CLI 参数将 `--prefill-delayer-queue-min-ratio` 设置为 `None`（或不传递）快速回退至原有仅基于 slot 的触发逻辑。

此次提交在保持原有行为不变的前提下，为 **高并发碎片化请求** 场景提供了更细粒度的调度优化手段，潜在收益显著，但需通过合理的参数调优与监控来规避可能的延迟回退风险。

**🔍 技术洞察**  

- **架构影响**  
  - **观察者模式引入**：通过 Mixin 将事件生产与核心缓存逻辑解耦，后端可自由订阅 `kv_event_queue`，为分布式 KV‑aware 调度器提供统一的状态流。

  - **继承结构调整**：`MambaRadixCache`、`RadixCache` 等类现在多继承 `KVCacheEventMixin`，确保 MRO（Method Resolution Order）不受已有方法覆盖的影响（Mixin 放在最前或第二位），保持原有行为不变。  

- **性能影响**  
  - **额外计算**：在每次 `store`/`remove` 时会计算或复制 `hash_value`（基于 SHA256），并生成事件对象。对高并发推理场景而言，此开销在 **page_size=1** 时最明显（每 token 生成一次事件），但整体仍是 O(1) 的轻量级操作。  
  - **内存压力**：`kv_event_queue` 按请求生命周期累积事件；如果消费者（如外部 router）未及时消费，可能导致事件堆积，占用额外内存。建议在生产者侧使用 `take_events` 做批量转发，或设置队列上限。  
  - **I/O 开销**：事件最终会经 ZeroMQ、Kafka 等外部发布器发送，网络带宽与序列化成本会随事件频率线性增长。对吞吐量极限的影响应通过压测评估。  

- **安全考虑**  
  - **信息泄露风险**：事件 payload 包含 **token_ids** 与 **block_hashes**（基于 token 内容的哈希），在未加密的发布渠道中可能泄露用户请求的文本片段。生产环境应确保事件通道使用 TLS/加密或在内部受信网络中。  
  - **访问控制**：新增 `--kv-events-config` 参数默认关闭，防止误开启导致敏感信息外泄。若开启，需要在部署策略中加入 RBAC 或防火墙限制订阅者。  
  - **异常处理**：当前实现假设 `kv_event_queue.append` 不会抛异常，若外部发布器因序列化错误导致进程崩溃，需要在未来加入捕获与回退机制。  

**⚠️ 潜在风险**  
1. **事件丢失或顺序错乱**  
   - `kv_event_queue` 为普通列表，单线程使用安全；若后续改为多线程或异步模型，需加锁或改用线程安全队列。  
2. **性能回退**  
   - 大模型（数十万 token）且 `page_size=1` 时，事件数量等同于 token 数，可能导致显著 CPU 与网络开销。  
3. **配置不一致**  
   - `enable_kv_cache_events` 需要在 `CacheInitParams` 里显式传递；遗忘传递会导致业务侧无法收到事件，却仍产生计算开销。  
4. **兼容性冲突**  
   - 迁移 hash 计算函数到 `utils.py`，若已有项目自行实现同名函数或直接 import 旧路径，可能出现 ImportError。  

**💡 关注建议**  

- **上线前**

  2. **安全加固**：在生产配置中强制使用加密的 ZeroMQ/TCP (`tls://`) 或内部 Kafka，禁止将事件推送至公网。  
  3. **日志审计**：记录 `enable_kv_cache_events` 的开启/关闭状态，以及事件发布成功/失败计数，便于运维排查。
