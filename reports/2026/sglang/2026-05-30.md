### feat: add SGLANG_RAY_BUNDLE_INDICES for fine-grained Ray bundle index control (#24667)
**SHA**: `acd689b` | 🔗 [查看提交](https://github.com/sgl-project/sglang/commit/acd689b407a9fd42fbce4b40b04c304d9ba6adb7)

**🎯 变更类型**：功能增强  

**⚡ 重要程度**：🔴高  

**📋 变更摘要**  
- 引入 `SGLANG_RAY_BUNDLE_INDICES` 环境变量，实现对 Ray 自定义 PlacementGroup 中 GPU Bundle 的精细化控制，支持跳过不健康 GPU、拓扑感知（如 NVLink）以及非顺序的 Bundle 分配。  
- 为该特性补齐文档、环境变量说明以及示例代码。  
- 在核心 Ray 调度层面添加 Bundle 解析、校验、节点 IP 查询以及统一的 SchedulerActor 创建函数，兼容原有自动 PlacementGroup 场景。  
- 新增单元/手动测试覆盖自定义 PlacementGroup、Bundle 索引错误以及 DP‑Attention 场景。  

**🎯 影响范围**  
- `python/sglang/srt/environ.py` – 增加 `SGLANG_RAY_BUNDLE_INDICES` 环境变量。  
- Ray 相关核心模块：`engine.py`、`data_parallel_controller.py`、`scheduler_actor.py`（间接调用）。  
- 文档：`offline_engine_api.ipynb`、`environment_variables.mdx`。  
- 测试套件：`test/manual/test_ray_engine.py`（新增多项用例）。  
- `http_server.py`（在启动时强制置 `placement_group=None`）。  

**🔍 技术洞察**  

- **架构影响**  
  1. **自定义 PlacementGroup 支持**：在 `RayEngine.__init__` 中接受 `placement_group` 参数，若提供则走 “Custom PG” 分支；否则维持原有 “Auto PG” 行为。  
  2. **Bundle 索引解析与校验**：新增 `_resolve_bundle_indices`、`_validate_custom_placement_group`、`_compute_world_size`、`_get_bundle_node_ip` 等工具函数，向上层抽象统一 SchedulerActor 创建 (`_create_scheduler_actor`)。  
  3. **分布式初始化地址**：根据每个 DP 组本地的 Rank‑0 Bundle 所在节点 IP 生成 `dist_init_addr`，避免跨节点 NCCL Rendezvous 引入不必要的网络跳转。  
  4. **向后兼容**：当未设置环境变量或未提供自定义 PlacementGroup 时，仍使用原有的 `bundle_for_node` 计算方式，功能保持兼容。  

- **性能影响**  
  - **CPU/网络开销**：额外的远程函数 `_get_bundle_node_ip` 在启动时执行一次，用于获取 Bundle 所在节点 IP，开销极低（一次 RPC）。  
  - **调度路径**：创建 SchedulerActor 统一走 `_create_scheduler_actor`，逻辑略有增多，但与原始 `options(...).remote()` 相当，整体启动时延提升在毫秒级，可忽略。  
  - **运行时**：对执行过程不产生额外开销，除非用户频繁更改 `SGLANG_RAY_BUNDLE_INDICES`（需要重新重启 Engine）。  

- **安全考虑**  
  - 环境变量是外部可控输入，若被恶意设置可能导致 **资源误分配**（如将所有 Bundle 定位到单节点），但不会导致代码执行漏洞。  
  - 已对变量值做严格的类型、范围、重复性检查，抛出明确的 `ValueError`，防止因非法输入导致不可预期的调度错误。  
  - 代码中未出现对外部网络或文件系统的直接访问，安全风险有限。  

**⚠️ 潜在风险**  

| 风险点 | 说明 | 严重程度 |
|--------|------|----------|
| 环境变量配置错误 | `SGLANG_RAY_BUNDLE_INDICES` 长度不等于 `world_size`、出现重复或越界索引时抛异常，可能导致 Engine 启动失败。 | 中 |
| 自定义 PlacementGroup GPU >1 | 若用户提供的 Bundle 包含多于 1 GPU，会在验证阶段报错，使用者可能误解错误来源。 | 低 |
| `http_server.py` 强制 `placement_group=None` | 该改动在启动 HTTP 服务时会忽略用户可能已经在 `RayEngine` 中设定的自定义 PlacementGroup，导致与文档示例不一致。 | 中 |
| 资源碎片化 | 手动指定非连续 Bundle 可能导致节点间 GPU 利用率不均衡，进而影响整体吞吐。 | 低 |
| 兼容性回归 | 部分内部调用仍假设 `bundle_for_node` 为 `List[int]`（如旧的 `DataParallelController`），已通过 `Optional[List[int]]` 调整，但若其他未改动的模块仍使用旧签名，可能出现运行时 `NoneType` 错误。 | 中 |

**💡 关注建议**  

1. **完善错误提示**  
   - 在 `_resolve_bundle_indices` 中对 “长度不匹配” 与 “索引重复” 分别给出更具指导性的错误信息（包括期望的 `world_size` 与实际提供的列表）。  
   - 在 `DataParallelController.__init__` 对 `bundle_for_node` 为 `None` 时加入断言或日志，防止误用。  

2. **文档同步**  
   - 更新 `RayEngine` 使用说明，明确 `placement_group` 与 `SGLANG_RAY_BUNDLE_INDICES` 的配套关系。  
   - 在 `http_server` 章节补充说明：若需要自定义 PlacementGroup，请在创建 `RayEngine` 前完成配置，`launch_server` 会自动清除。  

3. **CI/测试强化**  
   - 为“自定义 PlacementGroup + DP‑Attention” 场景添加压测基准，确保 NCCL 初始化在跨节点 IP 时不出现时延异常。  
   - 增加负载均衡测试：在多节点、不同 GPU 拓扑（如 NVLink、PCIe）下使用非连续 `SGLANG_RAY_BUNDLE_INDICES`，验证性能不受意外下降。  

4. **安全审计**  
   - 考虑在解析环境变量时加入白名单或最小化权限检查，防止意外泄露内部节点 IP（虽然仅用于内部通信）。  
   - 若未来计划在多租户环境中使用 Ray，建议将该变量的读取限制在可信的启动脚本中，防止恶意用户通过环境变量抢占特定 GPU。  

5. **向后兼容性**  
   - 为 `RayEngine.__init__` 添加 `placement_group` 参数的默认值 `None`，并在文档中标记为 “可选”。  
   - 保持 `SGLANG_RAY_BUNDLE_INDICES` 的默认空值行为不变，确保现有不使用该特性的部署不受影响。  

---  
**结论**：本次提交为 SGLang 在 Ray 集群上的部署提供了重要的可配置能力，使用户能够在自定义 PlacementGroup 中精细指定 GPU 使用顺序，提升了在多 GPU、跨节点或拓扑受限环境下的灵活性。实现上通过新增环境变量、校验逻辑以及统一的 SchedulerActor 创建流程，对原有架构影响可控，性能开销极小。主要风险集中在配置错误和对 `http_server` 的行为修改上，建议在发布前完善错误信息、文档说明并强化 CI 覆盖，以确保平滑迁移与安全可靠的生产使用。

### [MLX] Support Qwen3.5 (dense) Model (#25754)
**SHA**: `a952e91` | 🔗 [查看提交](https://github.com/sgl-project/sglang/commit/a952e9174f544816325631835b46132c4b17a880)

**🎯 变更类型**：功能增强 / 架构变更 / 性能优化  
**⚡ 重要程度**：🔴高  

**📋 变更摘要**  
本次提交为 MLX 后端引入对 Qwen 3.5（Dense）模型的完整支持，核心实现包括：  
1. **Attention 缓存抽象重构** – 将原有的 `ContiguousKVCache / PoolBackedCache` 替换为统一的 *Attention KV Cache*（`ContiguousAttentionKVCache`、`PoolBackedAttentionKVCache`、`MlxAttentionKVPool`），并在 `kv_cache/__init__.py` 导出新的合同（`attention_contract`）以及布局帮助类 `MlxModelCacheLayout`。  
2. **统一 Radix Cache 扩展** – 为 MLX 原生的非‑softmax（Mamba、DeltaNet 等）层新增 **Auxiliary‑State** 快照组件 (`MlxAuxiliaryStateComponent/Pool`)，并通过 `CacheInitParams.component_registry_override` 让统一 radix 树在 MLX 环境下自动使用该组件。  
3. **模型运行时改造** – `MlxModelRunner` 重新设计缓存获取、同步、恢复逻辑，支持：  
   - 统一的注意力 KV 池（强制所有注意力层 KV 形状一致）。  
   - 在前缀命中时可回滚到原生 `model.make_cache()` 的 auxiliary 状态。  
   - 按需在 decode 阶段使用 **批量注意力**（`_decode_with_batched_attention`）或 **混合批处理**（`_decode_with_hybrid_batching`）来兼容带有 auxiliary‑state 的模型。  
4. **调度器与 TP‑Worker 适配** – 为 MLX 重叠调度添加 `SchedulerMlxOverlapMixin`，在 `tp_worker.py` 中改为 `init_cache_pools`、`flush_all_decode_kv`，并在请求完成前调用 `prepare_for_kv_cache_release` 快照 auxiliary‑state。  
5. 其他细节：`.isort.cfg` 增加第三方库 `mlx, mlx_lm`，`pyproject_other.toml` 加入 `apache-tvm-ffi` 依赖，若干文件路径、命名、注释更新。

### Optimize get load calls (/v1/loads) using shared-memory load snapshots (#26348)

## 🔍 技术洞察  

### 架构影响  
| 维度 | 影响描述 |
|------|----------|
| **模块解耦** | 通过 `LoadSnapshot` 将调度器的负载信息统一序列化为结构体，TokenizerManager 与 DataParallelController 不再依赖 `WatchLoadUpdateReq` 的多路复用 IPC，实现 **单向、一致** 的数据流。|
| **双后端** | - **SHM**：单节点内存映射文件，速度极快、CPU 负载几乎为零。<br>- **ZMQ**：在多节点 DP Attention 场景下使用 PUSH/PULL + CONFLATE，实现跨机器共享快照。<br>自动在两者间切换（`should_use_zmq`），兼容现有部署。|
| **实例 ID + ZMQ 绑定规则** | 引入 `instance_id` 与 `zmq_reader_owner`，确保仅 **一个进程**（Tokenizer 或 DPController）绑定 ZMQ PULL，防止 “address already in use” 死锁。|
| **统一序列化** | 使用 `msgspec.msgpack` 编码，结构体定义一次，所有读写端共享，同步字段、元信息，降低版本漂移风险。|
| **调度器发布** | `Scheduler.publish_load_snapshot` 在每次 decode/extend 前、以及 idle 时强制写入快照，保证 **DP 控制器** 读取的是最新的调度器视图，避免因频繁 `update_budget` 导致的 “全局突发只落到单个 DP rank” 问题。|

### 性能影响  
- **IPC 带宽下降**：原先 `WatchLoadUpdateReq` 在每个 Generation/Embedding request 中携带完整负载结构，导致大量冗余序列化/网络流量。新方案仅在 **decode/extend**（每 20 ms）或 **idle/prefill** 时写入，极大压缩流量。  
- **CPU 占用**：SHM 读取是 `mmap`+`fcntl` 锁，几乎不产生 CPU 开销；ZMQ 使用 `CONFLATE`，只保留最新消息，写端 `NOBLOCK`，避免阻塞。  
- **延迟**：查询 `/v1/loads` 只需一次 **SHM 读取**（纳秒级）或 **ZMQ 拉取后立即读取 SHM**（几毫秒），比原先的 RPC+序列化快 5‑10 倍。  
- **内存占用**：每个 DP rank 占用固定 **16 KB** 槽（`SLOT_SIZE`），对常规 1‑8 DP 场景几乎可以忽略。  

### 安全考虑  
- **共享内存权限**：创建文件时权限 `0o600`，仅当前用户可读写，基本防止跨用户泄露。  
- **ZMQ 通信**：默认使用 **IPC**（Unix domain socket）或 **TCP**（在多节点模式下），未做加密；若在不可信网络中部署，需要自行在外层加入 TLS 或 VPN。  
- **环境变量** `SGLANG_LOAD_SNAPSHOT_USE_ZMQ`：误开启在单节点环境下会产生额外的 ZMQ 套接字，稍微增加攻击面；建议在生产文档中明确提示。

总体而言，此次改动为运维提供了便利的调试手段，但涉及全局日志状态的修改，需谨慎处理异常与并发场景，并通过测试确保稳定性。

### [perf] Fuse NVFP4 gate_up_gemm + swiglu + output FP4 quant  (#26626)
**SHA**: `3cecc77` | 🔗 [查看提交](https://github.com/sgl-project/sglang/commit/3cecc77ccb3f2be6c82c3cccd5ee52bc74686dcb)

**变更概览**  
本次提交在 FP4 量化路径上实现了 “gate_up_gemm + swiglu + output‑FP4” 的一次性融合（NVFP4 GEMM‑Swiglu‑FP4），并通过环境变量 `SGLANG_ENABLE_NVFP4_GEMM_SWIGLU_FUSION` 默认开启。核心实现位于 `nvfp4_gemm_swiglu_nvfp4_quant.py`，主要修改点包括：

1. **环境变量**：在 `environ.py` 新增开关，统一控制该特性。  
2. **权重后处理**：`ModelOptFp4LinearMethod.process_weights_after_loading` 增加对 `_interleave_for_swiglu_fusion` 标记的检测，完成 **Up/Gate** 权重与对应 scale 的 128‑group 交叉重排并存放在 `weight_swiglu_interleaved / weight_scale_swiglu_interleaved`。  
3. **前向计算**：`ModelOptFp4LinearMethod.apply` 改写为先检测层是否接受预量化的 FP4（`_accepts_prequantized_fp4`），从而在上层模型 `DeepseekV2` 中可以直接传入 `(fp4, scale)` 元组，避免重复量化。  
4. **模型层级**：`deepseek_v2.py` 中对共享专家层加入标记：  
   - `gate_up_proj._interleave_for_swiglu_fusion = True`（触发权重交叉重排）  
   - `shared_experts._enable_nvfp4_gemm_swiglu_fusion = True`（在 forward 中走 fused kernel）  
   - `down_proj._accepts_prequantized_fp4 = True`（接受前一步产生的 FP4 输入）

**影响范围**  
- 量化子模块 `modelopt_quant`、`nvfp4_gemm_swiglu_nvfp4_quant`  
- DeepSeek‑V2 共享专家（Mixture‑of‑Experts）实现  
- 运行时环境配置 `sglang.srt.environ`  

**潜在风险 & 建议**  
- **兼容性**：新特性依赖 SM100 及 `flashinfer.fp4_quantize`，在旧显卡或未编译 flashinfer 的环境会自动回退，但建议在 CI 中加入显卡特征检测，防止因 `is_sm100_supported()` 误判导致 crash。  
- **内存管理**：权重交叉重排后将原 `weight`、`weight_scale_interleaved` 置为空张量，若后续有权重重新加载（如 LoRA、checkpoint reload），需确保 `process_weights_after_loading` 能再次填充，否则会出现空参数报错。可以在 `ModelOptFp4LinearMethod.apply` 前添加断言检测。  
- **数值准确性**：fusion 改写后 `down_proj` 直接接受 `(fp4, scale)`，但 `down_proj` 的 `apply` 仍会走常规 fp4 量化路径；若后续加入自定义量化层，需检查 `output_dtype` 与 `layer.params_dtype` 是否一致，防止精度回退。  
- **调试/日志**：建议在关键分支（环境变量开启、硬件支持判定、权重交叉重排成功）加入 `log_debug`，方便定位在不支持的机器上为何未走 fused 路径。  
- **测试覆盖**：加入针对 `gate_up_proj._interleave_for_swiglu_fusion=True`、`down_proj._accepts_prequantized_fp4=True` 的单元测试，验证：
  1. 权重、scale 正确交叉重排且尺寸满足 128 对齐。  
  2. 前向输出与原非 fused 实现误差在可接受范围（FP4 误差 < 1e‑3）。  
  3. 在不满足条件（如 K‑pad、N‑pad、非 SM100）时仍能回退到普通路径。  

**结论**  
此次改动在支持的 GPU 上可显著削减两次量化/反量化的开销，提升 DeepSeek‑V2 MoE 前向吞吐，属于 **性能优化**（perf）类。只要在低端硬件或未启用 flashinfer 时保持向后兼容，且做好上述安全检查与测试，即可平稳上线。
