### [KDA] Add CuteDSL Prefill Kernel on SM100 (#27488)
**SHA**: `518e35f` | 🔗 [查看提交](https://github.com/sgl-project/sglang/commit/518e35fae7ae881c4a2f2e249a737bb48a6caeb8)

**🎯 变更类型**：功能增强 / 架构变更 / 性能优化  
**⚡ 重要程度**：🔴高  

**📋 变更摘要**  
- 为 **KDA（Kimi Delta Attention）** 添加了基于 **CuteDSL + SM100（Blackwell）** 的 prefill（Chunk‑wise）实现，包含完整的 Triton‑Prologue → KKT‑inverse → h → o 四阶段管线。  
- 在 `kda_backend.py` 中引入 `cutedsl` 后端并实现自动回退到 Triton（当 GPU 不满足 SM100 或维度不匹配时）。  
- 新增 **kda_blackwell** 包，提供三个高性能 Cutlass/CuTeDSL kernel（`kernel_h`, `kernel_kkt_inv_uw`, `kernel_o`）以及元数据准备、工作空间复用、Fused‑prologue（Triton）实现。  
- 新增 **benchmark/bench_kda_prefill_cutedsl.py** 用于正确性验证与性能对比。  
- 增加 **test_kda_prefill_cutedsl.py** 完整单元测试，覆盖多序列、内部门激活、极端门值等情况。  

**🎯 影响范围**  
- `python/sglang/srt/layers/attention/linear/kda_backend.py`（KDA 后端选择）  
- `python/sglang/srt/layers/attention/linear/kernels/kda_blackwell/`（全套 SM100 CuteDSL kernel）  
- `benchmark/bench_kda_prefill_cutedsl.py`（基准/正确性脚本）  
- `test/registered/attention/test_kda_prefill_cutedsl.py`（CI 单元测试）  
- 相关的 `__init__` 与 `setup.cfg`（模块导入）

### 🔍 技术洞察

| 维度 | 影响说明 |
|------|----------|
| **架构影响** | 1. **新后端入口**：`kda_backend` 现在支持两种 prefill 实现（Triton 与 CuteDSL），通过 `prefill_backend.is_cutedsl()` 判断。<br>2. **全新管线**：采用 *Fused‑prologue（Triton）* → *kkt_inv_uw* → *h* → *o* 四段，每段均为独立 Cutlass / CuTeDSL 内核，显著改变了原本单一 Triton chunk‑kda 的执行流。<br>3. **元数据缓存**：`_KDA_WS` 全局字典按 `(Hv, K, V, device, dtype, stream)` 键复用显存工作区，避免每次调用时的 200 MB 级别的张量分配与 zero‑fill。<br>4. **强制硬件约束**：仅在 **SM100+（Blackwell）** GPU、且 `head_k_dim == head_v_dim == 128` 时激活；否则自动回退到 Triton 实现。<br>5. **状态布局**：保持与现有 GDN/原 Triton 实现相同的 `[N, Hv, V, K]`（V‑major）布局，确保 `ssm_states` 可直接 `index_copy_`。 |
| **性能影响** | - **理论加速**：在 Blackwell GPU 上，预先计算的 “pre‑scaled” key/query（`kg`, `qg`, `qg2`）消除了每个 Chunk 中的 per‑token 门 `exp(g_cu‑g_last)` 计算，配合 Cutlass 高效的 MMA、TMA+cp.async 以及分阶段流水线，可在 **~2‑3×**（视序列长度与 head 数）相较于原 Triton `chunk_kda` 获得显著的 **TFLOPS / TB/s** 提升（benchmark 已实现直接对比）。<br>- **启动开销**：首次调用会触发一次 JIT 编译（Cutlass+CuTeDSL），后续复用工作区后，单次前向仅剩几微秒级的 kernel 调度开销。<br>- **内存占用**：工作区约 `~200 MB`（`kL`, `qg2`, `eye`, `U`, `W`, `V_new`, `h_chunks`, `ht`），但在复用后保持恒定，不随 batch/seq 长度线性增长。 |
| **安全/稳定性** | - **GPU 兼容性**：只在 SM100+ 上加载；若误在旧 GPU 上入口仍被选中，会在 `kda_backend` 初始化阶段抛异常，防止非法内存访问。<br>- **边界检查**：内部 kernel 对 `T`, `chunk_id`, `seq_id` 均使用 `mask_row`、`mask`，并在 `prepare_metadata` 中对 `total_chunks` 做 `int` 转换，防止 OOB。<br>- **同步**：大量使用 `mbarrier`、`cp.async`、`warp.barrier`，已在代码中显式 `fence_before_tma_store()`、`fence_after_thread_sync()`，确保 SM‑级别的可见性，降低数据竞争风险。<br>- **异常路径**：若 `prepare_metadata` 与实际 FLA chunker 不匹配，CI 中已有对应 `assert`，在 CI 通过后仍建议在生产环境加入运行时 `torch.testing.assert_close` 防御。 |
| **可维护性** | - 代码量大（≈ 3000 行），核心逻辑散布于三个 kernel 文件，阅读门槛提升。<br>- 使用 `@cache` 静态编译，确保 JIT 只在首次调用时触发，后续调用更易调试。<br>- 通过 `kda_backend` 的 `supports_prefill` 标记，外部调用无需感知内部差异，仅通过 `extend_kernel` 自动切换。<br>- 现有测试覆盖 **多序列数、内部门激活、极端门值**，但仍缺少对 **非 128‑dim**、**多 GPU（跨 stream）**、**混合精度** 的回归。 |

### ⚠️ 潜在风险

1. **硬件锁定导致回退失败**  
   - 若用户在 SM100 GPU 上使用非 128 维度（`head_k_dim != 128`），当前实现会回退到 Triton *但仍保持 `supports_prefill=True`*，可能在运行时产生 **shape mismatch** 错误。  
2. **全局工作区跨 Stream 共享**  
   - `*_workspace` 依据当前 CUDA stream 作为键。如果同一进程在同一 GPU 上**并行**多个不同 stream 的 KDA 前向（例如多模型并发），工作区会被错误共享，导致 **状态污染**。  
3. **编译缓存失效**  
   - 代码改动或环境变量（如 `CUDA_ARCHITECTURES`）改变时，`cute.compile` 产生的缓存可能失效或残留旧二进制，导致不可预期的 **性能回退** 或 **非法指令**。  
4. **数值稳定性**  
   - `exp(g_cu - g_last)` 在极端门值（如真实模型的 `exp(A_log) ~ 200`) 可能产生 **FP32 overflow**，虽然 kernel 内部使用 sub‑chunk归一化方案，但仍有潜在 **NaN** 产生的风险，尤其在 `g_last` 计算时。  
5. **二进制体积**  
   - 引入 Cutlass + CuTeDSL 的大块代码会显著增加 **wheel** 包体积

### [Feature] [Ngram spec] Support ngram spec v2 (#17260)
**SHA**: `111009e` | 🔗 [查看提交](https://github.com/sgl-project/sglang/commit/111009ea542e3430d8eed6e8e99dd8558982699f)

**🎯 变更类型**：功能增强 / 重构 / 架构变更  
**⚡ 重要程度**：🔴高  

**📋 变更摘要**  
1. 在 `ngram` 规格上实现 **Speculative Decoding V2**（即 *ngram spec v2*），引入统一的 `SpecInput` 接口，新增 `NgramVerifyInput` 继承 `EagleDraftInputV2Mixin`/`EagleVerifyInputV2Mixin`，并在模型运行时使用 `custom_mask` 替代旧的 `tree_mask`。  
2. 大幅简化 `speculative` 相关的调度与缓存管理：在 `OverlapUtils` 中直接跳过 ngram 路径（TODO 待删除），在 `ScheduleBatch`、`SpecInfo`、`NgramWorker` 中加入对 **V2** 流程的支持。  
3. 引入 **draft‑token‑num → spec_steps** 自动推导、对 `page_size`、`spec_topk` 的统一处理，以及对 KV‑cache 分配/释放的改写。  
4. 测试层面加入 **non‑overlap**（同步 V2）路径的覆盖，更新 CI 超时时间并添加 `SpecLogprobKit` 以验证 LogProb 计算。

---

**🎯 影响范围**  
- **核心模块**：`sglang/srt/speculative/*`（ngram_info、ngram_worker、spec_info、speculative_hook、overlap_utils、schedule_batch）  
- **模型执行层**：`model_executor/model_runner.py`、`decode_cuda_graph_runner.py`（参数名更改）  
- **调度/缓存层**：`mem_cache/common.py`（分配长度公式）  
- **服务器入口**：`speculative_hook.py`（新增 `speculative_num_steps` 参数）  
- **测试套件**：`test/kits/spec_server_kits.py`、`test/spec/*`（新增 V2 路径、非 Overlap 测试）

**🔍 技术洞察**  

| 维度 | 影响说明 |
|------|----------|
| **架构影响** | - **统一 SpecInput 接口**：`NgramVerifyInput` 现在兼容 Eagle V2 的混合实现，降低了不同 speculative 算法之间的代码重复度。<br>- **去除 Overlap 对 Ngram 的依赖**：在 `OverlapUtils` 中直接返回，表明目前 V2 只在同步路径下工作；后续如果实现 pre‑computed draft，则可以恢复 Overlap。<br>- **Scheduler 与 Worker 解耦**：`ScheduleBatch.copy` 新增 `spec_info`，使得调度层能够透明传递 V2 状态（`future_indices、new_seq_lens、accept_*`），为未来的异步/流水线优化奠定基础。 |

### 🔍 技术洞察  

#### 架构影响  
- **统一翻译层**：在 `ForwardMetadata` 中集中保存 `swa_out_cache_loc`，避免每个后端自行进行全到 SWA 的翻译，形成统一的“翻译‑写入”抽象。  
- **后端解耦**：后端只负责将 `swa_out_cache_loc` 传递给 KV‑Pool，KV‑Pool 本身不再感知全局索引，降低了后端对 `SWAKVPool` 实现细节的耦合。  
- **CUDA‑Graph 支持**：为捕获的图维护专用的 int64 buffer，保证图内读取的写入目标始终是最新的翻译结果，兼容多 batch、变长序列的 replay。  

#### 性能影响  
- **一次性翻译**：原来的实现在每个 token 写入时都要执行 `translate_loc_from_full_to_swa`，导致额外的 memory kernel 调用和同步开销。本次改为 **forward 入口一次完成翻译**，显著降低了 KV‑Cache 写入的算子调用次数。  
- **复制成本**：在 CUDA‑Graph 重放前，需要将 `out_cache_loc` 拷贝到 `swa_out_cache_loc_buf`（`torch.copy_`）。该复制是线性 O(N) 的一次性操作，远小于先前在每次 `set_kv_buffer` 中重复翻译的成本。  
- **后台兼容性**：对非‑SWA 模型路径不产生额外开销，因为 `swa_out_cache_loc` 为 `None`，后端直接走原有路径。  

#### 安全考虑  
- **断言保护**：`SWAKVPool.set_kv_buffer` 在检测到是 SWA 层时强制要求 `swa_loc` 非空 (`assert swa_loc is not None`)。这避免了因忘记传递翻译结果而导致错误写入潜在的内存越界。  
- **输入可信**：`swa_out_cache_loc` 只来源于内部计算（`translate_loc_from_full_to_swa`），没有外部用户可控的输入，故不存在注入或信息泄露风险。  

#### 兼容性  
- **向后兼容**：所有 `set_kv_buffer` 调用已同步更新 `swa_loc` 参数，默认在非 SWA 场景传入 `None`，保证旧代码仍可编译运行。  
- **多后端一致**：统一的元数据结构消除了各后端之间“是否需要翻译”的分歧，降低了未来新增硬件后端的实现成本。

### 2.1 统一的 **CudaGraphConfig**

3. **异常路径的副作用**：在触发边界错误后，代码会将对应位置的 `output_ids` 重写为 `eos_token_id`（或 `stop_token_id`），这可能导致 **计数/日志** 中的 “finished_len” 与实际生成的 token 数不匹配，需要确保上层统计逻辑已设计为接受这种“提前终止”。  

**💡 关注建议**：  
- **回归验证**：在 CI 中加入对自定义词表（如 vocab_size=0、极小 vocab）以及多 GPU/多节点环境的回归测试，确保改动不会在分布式调度器中产生意外行为。  
- **文档更新**：在 `Req._check_vocab_boundary_finish` 方法的 docstring 中注明合法 token 范围为 `[0, vocab_size)`，并解释触发 NaN guard 时的处理方式（会写入 EOS/stop token）。
