### fix(ci): update est_time for 57 tests based on runtime analysis (#21896)
**SHA**: `875a615` | 🔗 [查看提交](https://github.com/sgl-project/sglang/commit/875a61599370ea9a7b53ce11ebe230c89d1eabd5)

**🎯 变更类型**：Bug修复 / 性能优化（CI 估算时间校准）  
**⚡ 重要程度**：🔴 高  
**📋 变更摘要**：此次提交对 *sglang* 项目中 57 条单元/集成测试的 `register_*_ci` 调用的 `est_time` 参数进行了统一的重新评估。基于实际运行时长的统计，部分测试的预计执行时间被显著上调（如某些大模型评估、GPU 大型测试），而少数快速测试的预计时间被下调。此举旨在让 CI 调度器更准确地分配 GPU/CPU 资源，避免因时间估算不准确导致的跑批超时、资源浪费或 flaky 测试被误判。

**🎯 影响范围**：  
- `test/registered/**` 目录下所有 CI 注册文件（约 57 处）  
- CI 调度系统（GitHub Actions / 自建 CI）对 `est_time` 的解析与资源分配逻辑  
- 与 CI 相关的报告、超时阈值及自动重试机制  

**🔍 技术洞察**  

- **架构影响**：  
  - 仅修改了 **CI 注册层**（`sglang.test.ci.ci_register`），不涉及业务代码、模型实现或运行时框架。对项目的运行时架构、服务部署或模型推理路径没有直接影响。  
  - 通过更准确的时间预估，CI 调度器在分配 GPU（H100 / H200 / B200 等）和 CPU 资源时能够更好地进行 “抢占式” 调度，提升整体流水线的吞吐率。  

- **性能影响**：  
  - **正面**：避免了因 `est_time` 低估导致的测试超时被强制终止，从而减少不必要的重跑和资源浪费。  
  - **负面**：对部分测试（如大模型加载、混合注意力后端等）将占用更长的 CI 时段，可能导致整体 CI 队列排队时间增加，进而间接影响开发者提交的反馈时长。  
  - 通过在 `test/registered/disaggregation/test_disaggregation_decode_offload.py` 中加入 `disabled` 标记并暂时禁用不稳定测试，降低因 flaky 测试导致的 CI 失败率。  

- **安全考虑**：  
  - 该改动不涉及代码执行路径、输入校验或权限控制，安全风险为 **None**。唯一需要关注的是 CI 环境的 **资源配额**（GPU 时长）是否足够，否则可能触发配额耗尽导致 CI 失效。  

**⚠️ 潜在风险**  

1. **CI 时长膨胀**：将多个测试的 `est_time` 提高后，整个工作流的预计时长会显著增长，若 CI 平台对单次运行时间有硬性上限（如 GitHub Actions 的 6 小时限制），可能出现超时。  
2. **资源竞争**：高估时间会导致调度器保留 GPU 资源更久，其他并行工作流可能被迫排队，影响团队整体 CI 交付速度。  
3. **误判 flaky 测试**：虽然已对 `test_disaggregation_decode_offload` 加入 `disabled` 标记，仍有不少测试仅通过调整 `est_time` 来“掩盖”不稳定性，长期可能导致 flaky 测试被忽视。  
4. **文档同步**：CI 配置文档或内部手册中若仍保留旧的预计时间，需要同步更新，否则新成员可能误以为旧时间仍有效。  

**💡 关注建议**  

- **监控 CI 实际时长**：在 CI 运行后收集实际耗时数据，与 `est_time` 对比，建立周期性回顾（如每两周一次）以防止时间估算再次偏离。  
- **分层调度**：对超大模型测试（如 `test_nvidia_nemotron_3_nano`）设置专门的 “长时任务” 队列，避免阻塞常规短时任务。  
- **资源配额预警**：在 CI 配置中加入 GPU 时长配额监控，一旦接近上限，自动发送告警给维护者。  
- **flaky 测试治理**：对仅因时间估算而 “通过”的 flaky 测试，安排专人定位根因，考虑使用更可靠的重试或隔离机制，而非单纯调高 `est_time`。  
- **文档与 CI 可视化**：更新仓库 `README` / CI 运行手册，加入 “CI 估算时间表” 链接，帮助贡献者了解不同测试的资源需求。  
- **回退机制**：保留旧的 `est_time` 配置（如通过注释或独立的 JSON 配置），以便在 CI 环境或硬件升级后快速回滚。  

> **总结**：此次提交在不影响业务逻辑的前提下，显著提升了 CI 资源调度的精准度，降低了因时间估算失误导致的 flaky 报错。但也带来了 CI 运行时长和资源竞争的潜在增长，需要配合监控、调度策略以及 flaky 测试的长期治理，才能确保 CI 效率与可靠性同步提升。

### Direct model loading from object storage with Runai Model Streamer (#17948)
**SHA**: `8d9145d` | 🔗 [查看提交](https://github.com/sgl-project/sglang/commit/8d9145d97e71fe1022a7c449bfb912e99b08e7d3)

**🎯 变更类型**：功能增强  
**⚡ 重要程度**：🔴高  
**📋 变更摘要**：  
- 新增 **runai_streamer** 加载格式，支持直接从对象存储（S3、GCS、Azure Blob）流式加载模型权重，避免完整下载并显著缩短启动时间。  
- 实现 **RunaiModelStreamerLoader**、相关工具类 `runai_utils.py`，并在命令行参数、配置枚举、依赖声明等处完成集成。  
- 同步更新文档、示例以及 CI 测试，确保在 CUDA 环境下可用。

**🎯 影响范围**：  
- `sglang/srt/configs/load_config.py`（LoadFormat 枚举）  
- `sglang/srt/configs/model_config.py`（模型路径处理）  
- `sglang/srt/model_loader/loader.py`（新 loader 实现）  
- `sglang/srt/model_loader/weight_utils.py`（权重迭代器适配）  
- `sglang/srt/server_args.py`（参数解析、自动识别）  
- `sglang/srt/utils/runai_utils.py`（缓存、路径解析、对象存储下载）  
- `python/pyproject.toml`（新增 `runai-model-streamer` 可选依赖）  
- 文档 `docs/advanced_features/object_storage.md`、`docs/advanced_features/server_arguments.md`、`docs/index.rst`  
- CI 测试 `test/registered/model_loading/test_runai_model_loader.py`、`test/registered/unit/test_runai_utils.py`

---  

### 🔍 技术洞察

| 维度 | 影响 |
|------|------|
| **架构影响** | - **加载路径分支**：在 `get_model_loader` 中根据 `load_format` 选择 `RunaiModelStreamerLoader`，保持原有 `DefaultModelLoader` 不变，兼容性良好。<br>- **单次元数据下载**：`ObjectStorageModel.download_and_get_path` 在进程启动阶段完成配置/Tokenizer 元数据的本地缓存，避免多进程竞争文件锁。<br>- **全局配置**：新增 `--load-format runai_streamer`（可省略）以及 `--model-loader-extra-config`，通过 `LoadConfig` 传递至 loader。 |
| **性能影响** | - **启动加速**：仅下载少量元数据（json、tokenizer），权重在模型实例化时按需流式读取，显著降低磁盘 I/O 与网络带宽占用。<br>- **运行时开销**：首次访问每个权重文件会触发网络请求，取决于对象存储的吞吐与 latency；`concurrency` 与 `memory_limit` 参数可调节带宽利用率。<br>- **分布式加速**：当 `distributed=True` 并使用 CUDA，权重会在多个 GPU 之间并行流式，进一步压缩加载时间。 |
| **安全考虑** | - **凭证管理**：对象存储的访问凭证通过环境变量（如 `RUNAI_STREAMER_GCS_USE_ANONYMOUS_CREDENTIALS`）或常规云 SDK 提供；若凭证泄露，可能导致未授权模型获取。<br>- **模型代码安全**：仍然受 `--trust-remote-code` 控制；`runai_streamer` 本身仅处理权重文件（`.safetensors`），不执行自定义代码，安全面相对有限。<br>- **输入校验**：`is_runai_obj_uri` 对 URI 前缀做严格校验，防止误将本地路径当作对象存储。 |
| **可维护性** | - **代码分层**：新增的 `runai_utils.py` 将对象存储相关逻辑抽离，便于后续扩展（如新增协议）。<br>- **依赖隔离**：`runai-model-streamer` 设为可选 extras，未使用时不强制安装。<br>- **测试覆盖**：加入单元与集成测试，覆盖 URI 判定、缓存路径、端到端生成，提升回归安全性。 |

---  

### ⚠️ 潜在风险

1. **网络/凭证异常**：对象存储不可达或凭证失效会导致模型加载失败；目前缺少自动重试或降级到本地下载的机制。  
2. **兼容性**：仅支持 `.safetensors` 格式；若模型只提供 `pytorch_bin`，会抛出错误。  
3. **分布式环境限制**：分布式流式仅在 CUDA 设备上生效，CPU 或非 CUDA 设备会回退为单机串行加载，可能出现意料外的性能下降。  
4. **依赖版本冲突**：`runai-model-streamer` 版本提升（>=0.15.7）可能引入 API 变化，需要同步更新。  
5. **缓存目录竞争**：虽然实现了单次元数据下载，但多个不同模型并发首次下载时仍会在同一 `~/.cache/sglang/model_streamer/` 目录下创建子目录，若磁盘配额受限可能出现 `OSError`。  

---  

### 💡 关注建议

| 对象 | 建议 |
|------|------|
| **开发者** | - 在 CI 中保留 `runai` extra，以捕获依赖冲突；<br>- 为关键生产部署显式设置 `RUNAI_STREAMER_XXX` 环境变量，避免凭证泄露；<br>- 若模型仅有 `.bin`，考虑提供 fallback 或提示用户切换至 `safetensors`。 |
| **运维/使用者** | - 确认对象存储访问权限（IAM、ACL）已正确配置；<br>- 根据网络情况调优 `--model-loader-extra-config` 中的 `concurrency` 与 `memory_limit`，以平衡带宽和内存占用；<br>- 监控首次加载的延迟与后续推理性能，确保流式加载未引入不可接受的抖动。 |
| **项目维护** | - 在 `docs/advanced_features/object_storage.md` 中补充 “常见错误 & 调试指南”（如网络超时、凭证错误）；<br>- 考虑在未来加入 `runai_streamer` 对 `.pt` 的自动回退（目前已在 `Source` 中保留 `fall_back_to_pt` 标记但未实现）；<br>- 为 `ObjectStorageModel.download_and_get_path` 增加文件锁或 atomic 替换，以防极端并发下的目录竞争。 |

---  

**结论**：此次合并为 SGLang 引入了直接从对象存储流式加载模型的关键能力，显著提升大模型部署的启动效率与存储需求。架构上保持向后兼容，性能改进明显，但对网络、凭证及仅支持 safetensors 格式的限制需在实际生产环境中加以监控与配置。适当的运维措施与文档补充将进一步降低风险，确保该特性稳健落地。
