### fix(openai): validate assistant tool call arguments before chat template (#28035)
**SHA**: `265202c` | 🔗 [查看提交](https://github.com/sgl-project/sglang/commit/265202cda251bd466930f132ce7ca3b45cf33697)

**🎯 变更类型**：Bug修复  
**⚡ 重要程度**：🔴高  

**📋 变更摘要**  
- 为 OpenAI 接口的工具调用（tool call）参数增加了严格的 JSON 对象校验，统一在 `encoding_dsv32.py`、`encoding_dsv4.py` 与 `serving_chat.py` 中完成。  
- 当工具调用的 `arguments` 不是 JSON 对象（如字符串、数组或其它非 dict）时，将抛出 `ValueError`，而之前会被错误地当作普通字符串继续处理，导致后续模板渲染或编码出现异常。  
- 相应单元测试同步补齐，验证了错误路径的异常行为以及正确路径的参数转换。

**🎯 影响范围**  
- `python/sglang/srt/entrypoints/openai/encoding_dsv32.py`  
- `python/sglang/srt/entrypoints/openai/encoding_dsv4.py`  
- `python/sglang/srt/entrypoints/openai/serving_chat.py`（消息预处理、Jinja 模板路径）  
- 测试套件 `test/registered/unit/entrypoints/openai/test_serving_chat.py`  

**🔍 技术洞察**  

- **架构影响**  
  - **输入校验统一化**：新增的 `parse_tool_call_arguments` 与 `normalize_assistant_tool_call_arguments` 将工具调用参数的解析职责抽离到专门函数，提升了代码可读性并降低了在不同路径（DSV 编码、Jinja 编码）重复实现的风险。  
  - **深拷贝引入**：在 `_apply_jinja_template` 中对 `messages` 进行 `copy.deepcopy`，防止在原始请求对象上进行就地修改，降低了副作用传播的可能性，符合“不可变请求对象”设计原则。  

- **性能影响**  
  - **微小开销**：`json.loads`（或 `orjson.loads`）本已在常规工具调用处理中使用，新增的类型检查和异常抛出几乎不增加运行时成本。  
  - **深拷贝成本**：对消息列表进行 `deepcopy` 会在包含大量历史消息或大体积多模态内容时产生一定的内存与 CPU 开销。鉴于该路径主要用于 **Jinja** 模板渲染（相对少见且通常消息数目有限），整体影响可接受。若在高并发场景下出现性能瓶颈，可考虑在确认不需要修改原对象时改为浅拷贝或就地修改并在后续恢复。  

- **安全考虑**  
  - **避免 JSON 注入**：严格要求 `arguments` 为 JSON **对象**（`dict`），防止恶意用户通过传入 scalar（如 `"1"`）或数组触发模板渲染时的类型错配或代码注入风险。  
  - **错误可控**：统一抛出 `ValueError`，上层可以捕获并返回标准化的 `BadRequest` 响应，提升 API 的错误可预测性。  

**⚠️ 潜在风险**  
1. **向后兼容性**：旧版客户端如果习惯于直接传入非对象 JSON（如 `"true"`、`42`），现在会收到 400 错误，可能导致兼容性破坏。  
2. **深拷贝副作用**：在极端大批量历史消息场景下，`deepcopy` 可能导致显著的内存峰值，进而触发 OOM。  
3. **异常传播**：新增的 `ValueError` 未在所有调用链上统一捕获，若后续某些入口未处理，将导致服务异常退出。  

**💡 关注建议**  
- **文档更新**：在 OpenAI 接口文档中明确说明 `tool_calls[].function.arguments` 必须是 JSON **对象**，并给出示例（`{"key": "value"}`）。  
- **兼容性迁移**：对已有客户端提供迁移指南或在错误信息中加入 “请使用 JSON 对象而非标量” 的提示。  
- **性能监控**：在生产环境监控 `serving_chat._apply_jinja_template` 的内存使用峰值，必要时为大批量历史消息启用分块处理或限制最大历史长度。  
- **统一异常处理**：在入口层（如 `ChatCompletion` 的请求入口）捕获 `ValueError` 并转化为统一的 `BadRequest` 响应，防止未经处理的异常泄漏。  
- **回退方案**：若业务方确实需要支持标量参数，可考虑在配置开关中保留宽松模式，并在该模式下自动包装为单键对象（例如 `{"value": <scalar>}`），但务必记录该行为以防模板误解。

### [CAR] Let custom allreduce support VMM based allocation (#27593)
**SHA**: `fcca461` | 🔗 [查看提交](https://github.com/sgl-project/sglang/commit/fcca4611fa516e2955b787e417ded5a6ff8fcad9)

**🎯 变更类型**：功能增强 / 架构变更  
**⚡ 重要程度**：🔴高  
**📋 变更摘要**：  
- 为 `custom_all_reduce` 引入对 CUDA Virtual Memory Management（VMM）可扩展段（expandable_segments）分配的完整支持，新增图捕获期间的指针基址抽象与跨进程句柄交换机制。  
- 在 C++ `CustomAllReduceBase` 中增加 `get_graph_capture_bases` 与 `register_peer_mapped_inputs`，并在 Python 层实现 `VmmGraphInputManager` 负责 FABRIC/Posix‑FD 句柄的导出/导入、映射以及资源回收。  
- `CustomAllReducePull` 现在在图捕获路径验证输入尺寸，避免溢出。测试用例开启 `PYTORCH_CUDA_ALLOC_CONF=expandable_segments:True` 进行验证。

---

### 🎯 影响范围
- **核心模块**：`python/sglang/jit_kernel/*.cuh`、`python/sglang/jit_kernel/all_reduce.py`、`python/sglang/srt/distributed/device_communicators/custom_all_reduce_v2.py`、新增 `custom_all_reduce_vmm_utils.py`。  
- **分布式运行时**：所有使用 `CustomAllReduce`（包括模型并行、张量并行）的任务。  
- **测试套件**：`test/registered/spec/eagle/test_deepseek_v3_fp4_mtp_small.py`（开启 VMM 环境变量）。

---

### 🔍 技术洞察

| 维度 | 影响描述 |
|------|----------|
| **架构影响** | - 在 `CustomAllReduceBase` 增加了 **VMM 基址抽象层**，将图捕获的输入指针拆分为唯一的基址集合 + 用于每个输入的基址索引/偏移。<br>- Python 端 `VmmGraphInputManager` 负责 **跨进程句柄交换**（FABRIC → POSIX‑FD 备选）以及 **cuMemMap/Reserve/SetAccess** 映射，形成 **Peer‑VA 映射表**，随后通过 `register_peer_mapped_inputs` 注入到 C++ 结构体。<br>- 现有 IPC‑handle 路径保持不变，形成 **双路分支**：VMM ⇢ FABRIC/FD ⇢ 映射 vs CUDA‑Ipc ⇢ 直接注册。 |
| **性能影响** | - **正向**：对使用 `torch.cuda.alloc_conf=expandable_segments:True` 的大张量（尤其是多 GPU‑NVLink 环境）可以 **零拷贝共享** 同一 VMM 分配，显著降低 CUDA graph 捕获时的内存复制与 IPC 开销。<br>- **负向**：首次捕获时需进行 **全局句柄交换、基址去重、跨进程映射**，这会带来 O(`num_bases`) 的一次性成本（数十毫秒至百毫秒，受 `MAX_VMM_BASES`、网络与文件系统影响）。<br>- 运行时额外的 **映射表维护** 与 **cudaMemUnmap/Free** 在 `close` 时回收，开销可忽略。 |
| **安全考虑** | - 使用 **POSIX 文件描述符** 通过 Unix domain socket 传递，需要 **正确关闭** FD 防止泄漏；实现已在异常路径 `finally` 中清理。<br>- FABRIC 句柄在导出/导入时依赖 **CUDA driver** 权限，若进程运行在受限容器可能被阻止。<br>- `cuMemMap` 与 `cuMemSetAccess` 采用 **READWRITE** 权限，若非法进程获得句柄可能访问其他 GPU 进程的显存；因此应在生产环境确保只有受信任的进程组成同一 `ProcessGroup`。 |
| **可维护性** | - 新增约 600 行 Python 实现和约 100 行 C++/CUDA 代码，逻辑相对独立（VMM ↔ IPC 分支），易于单元测试。<br>- 使用 `tvm::ffi` 进行跨语言绑定，保持与现有 JIT 接口一致。<br>- 通过 `VmmGraphInputManager._all_ranks_ok` 保证所有节点一致成功，避免半成功状态导致倾析。 |

---

### ⚠️ 潜在风险
1. **兼容性风险**  
   - 仅在 **CUDA ≥ 11.2** 且驱动支持 `cuMemCreate/cuMemMap`、`CU_MEM_HANDLE_TYPE_FABRIC` 时可用。旧驱动会在 `cuMemRetainAllocationHandle` 失败，导致回退到 IPC 路径或直接报错。  
   - 环境变量 `PYTORCH_CUDA_ALLOC_CONF=expandable_segments:True` 必须在所有节点保持一致，否则 `is_vmm_pointer` 检测会出现混合路径，可能导致死锁或资源泄漏。

2. **句柄交换失败**  
   - FABRIC 导出失败会自动回退 POSIX‑FD，但若 **任意节点** 同时不支持两者（如容器内没有 `memfd` 支持），会抛出 `RuntimeError`，导致图捕获失败。  
   - Unix domain socket 超时或文件系统权限不足会触发 `POSIX fd exchange timeout`。

3. **资源泄漏**  
   - `VmmGraphInputManager.close()` 必须在进程退出前被调用；若异常提前退出，`_release_peer_mappings` 可能未触发，导致显存保留。  
   - 句柄、FD、映射的错误路径虽已在 `finally` 中清理，但在多线程收尾阶段仍有微小竞争窗口。

4. **性能回退**  
   - 当输入跨多个 VMM 基址（`MAX_CHUNKS_PER_INPUT` 限制）或基址数量超过 `MAX_VMM_BASES`（4096），会抛异常并回退到 IPC，可能出现 **意外的图捕获失败**。

5. **调试复杂度**  
   - 新增的 `get_graph_capture_bases` 返回三层嵌套结构，若上层调用者误解顺序或维度，将导致错误的 `peer_ptrs` 注册，进而产生错位的 All‑Reduce 结果。

---

### 💡 关注建议
- **版本检查 & 回退**：在启动时检测 CUDA 驱动版本与 `cuMemCreate` 能力；若不满足，提前禁用 VMM 路径并记录日志。
- **统一环境**：在集群部署脚本中强制统一 `PYTORCH_CUDA_ALLOC_CONF`，防止出现混合分配类型导致的捕获错误。  
- **监控资源**：在 `close` 前加入显存/句柄计数日志，确保所有 `cuMemAddressFree` 与 `cuMemRelease` 成功执行。  
- **限流与参数调优**：根据实际模型大小调整 `MAX_VMM_BASES` 与 `MAX_CHUNKS_PER_INPUT`（默认 4096 / 16），必要时通过配置文件暴露给用户。  
- **容错日志**：在 FABRIC 与 POSIX‑FD 两条路径的异常捕获中记录完整的错误栈与进程 rank，便于在大规模集群中定位哪一台机器不支持某种句柄。  
- **单元/集成测试**：新增 VMM‑only、IPC‑only以及混合环境的 CI 测例

### [diffusion] fix: add precision consistency layer (#27088)

**📋 变更摘要**  
本次提交在 `sglang/multimodal_gen/runtime/utils/precision.py` 中新增一套统一的 **precision 管理工具**，包括：  
1. `precision_to_dtype / resolve_precision` 用于统一从 `pipeline_config` 中解析用户配置的精度并映射到 `torch.dtype`。  
2. `resolve_component_precision` 根据组件名称自动选取对应的精度属性（如 `vae_precision`, `dit_precision`, `audio_vae_precision` 等），并支持多文本编码器的索引解析。  
3. `autocast_enabled、get_module_dtype、align_tensor_to_module_dtype、temporary_module_dtype` 为模型加载、前向推理、VAE 编码/解码等场景提供 ** dtype 对齐与临时 cast** 的上下文管理。

### ⚠️ 潜在风险
1. **向后兼容性**  
   - 某些外部插件或自定义加载器若仍直接引用 `sglang.multimodal_gen.utils.PRECISION_TO_TYPE` 而不走新接口，可能在升级后出现 `KeyError` 或使用了旧的默认 dtype。需在发行说明中提醒用户更新对应代码。  
2. **上下文管理器异常泄漏**  
   - `temporary_module_dtype` 在异常路径会在 `finally` 中恢复 dtype，但如果模块在 `__exit__` 前被销毁（如在 `torch.jit.script` 中封装），可能导致恢复失败。建议在关键路径外层加 `try/except` 保证恢复。  
3. **多精度混合的硬件兼容性**

4. **`align_tensor_to_module_dtype` 对非浮点张量的行为**  
   - 该函数对整数/长整型张量只执行 `to(device)`，不改变 dtype，这在某些需要保持 `torch.long`（如 token IDs）时是安全的，但如果误将 `torch.float32` 的 mask 视为整数张量，可能导致意外的 **dtype 不匹配**。建议在调用处明确注释或在函数中加入类型校验。  
5. **组件名称映射遗漏**  
   - `resolve_component_precision` 目前覆盖了大多数核心组件，但若以后新增如 `depth_vae`、`super_resolution` 等，未同步映射会返回 `None`，导致加载时回退到旧的硬编码或报错。需要在新增组件时同步更新映射逻辑。

### ⚠️ 潜在风险  

1. **驱动/CUDA 版本不匹配**  
   - 若用户的驱动低于 12.8，`cudaMemcpyBatchAsync` 不可用，代码会回退到逐页 `cudaMemcpyAsync`。在极端的大写入场景下仍可能出现性能倒退或显存碎片化。  
2. **写回 staging 缓冲容量不足**  
   - 当写回页数 > 64，`staging_page_capacity` 为 64（`_WRITE_BACK_STAGING_PAGE_CHUNK`），需要分多次循环拷贝。若 `host_pool.move_hybrid_indices` 等路径未正确保留 `resolved_pool_transfers`（已在修改中硬编码），可能导致遗漏未写回的页。

4. **CPU 端索引类型**  
   - 对于 64 页写回使用 `int64_t` 索引，而 64 页以内使用 `int32_t`（批处理优化）。若上层代码误用不同 dtype（如在 `page_first` layout 时仍使用 `int64`），可能触发 `RuntimeCheck` 抛异常。  
5. **JIT 编译缓存**

| 维度 | 影响分析 |

- **文档更新**：在 SGLang 的注意力后端文档中标注 `fa_skip_kv_cache` 在 piecewise CUDA graph 环境下的特殊处理逻辑，并说明其适用硬件范围及可能的配置要求。
