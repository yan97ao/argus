### [Refactor] major JIT kernel clean up for dsv4 (#25884)
**SHA**: `19f55c0` | 🔗 [查看提交](https://github.com/sgl-project/sglang/commit/19f55c0e6d6fe5cf9a0f9ab6fc148cc004facb7b)

**🎯 变更类型**：重构 / 架构变更  
**⚡ 重要程度**：🔴 高  

**📋 变更摘要**  
- 将 *deepseek_v4* JIT‑kernel 代码从 `sglang/jit_kernel/deepseek_v4` 重组至全新的子包 `sglang/jit_kernel/dsv4`，并拆分为若干细粒度模块（`attn.py、compress_old.py、elementwise.py、gemm.py、hisparse.py、moe.py、topk.py`）。  
- 删除无用的 `topk_1024.cuh`，在 `topk_v1.cuh` 中通过宏 `SGL_TOPK` 统一 `kTopK` 与 `kTopKBlockSize` 的定义，实现 512/1024 两种 Top‑K 的统一编译路径。  
- 更新所有业务层（attention、moe、memory‑pool、模型）对 JIT 接口的导入路径，统一使用 `sglang.jit_kernel.dsv4`。  
- 引入轻量级包装函数（例如 `linear_bf16_fp32`、`fused_store_cache`、`triton_create_paged_compress_data`）以及新的 **TopK v1/v2** 调用入口，增强可配置性并去除冗余代码。  
- 通过 `make_cpp_args` 与 `load_jit` 机制仍保持原有的 “一次缓存‑多次加载” 行为，仅是路径和宏定义的改动。

## 🔍 技术洞察

| 维度 | 影响描述 |
|------|----------|
| **架构影响** | 1. **模块化提升**：将原本单块的 `deepseek_v4` 迁移为 `dsv4` 子包并细分功能模块，形成清晰的层次（`attn`、`compress`、`elementwise`、`moe` 等），便于单独测试、独立编译与未来功能逐步迭代。<br>2. **公共接口统一**：`__all__` 在 `dsv4/__init__.py` 中重新导出，所有上层代码只需一次 `from sglang.jit_kernel.dsv4 import …` 即可获取完整功能，减小了散落的导入路径。<br>3. **可配置 Top‑K**：在 `topk_v1.cuh` 中添加 `#ifndef SGL_TOPK … #endif`，外部通过编译选项 `-DSGL_TOPK=512|1024` 控制 Top‑K 大小，避免维护两套几乎相同的文件。 |
| **性能影响** | - **编译层面**：删除 `topk_1024.cuh` 并使用宏统一后，编译器只需处理一个文件，缩短 JIT 编译时间。<br>- **运行时**：核心算法（radix‑topk、压缩、RoPE、Flash‑MLA）未改动，功能保持一致；仅因为 `topk_v1` 与 `topk_v2` 的包装层增加了 Python 调用一次 `module.topk_transform`，开销极小（< 1 µs）。<br>- **共享内存设置**：`setup_kernel_smem_once` 仍然在 `topk_v1` 中使用，未受影响。<br>- **GPU‑Kernel 参数**：宏 `SGL_TOPK` 加入编译期常量，可让 CUDA 编译器在寄存器/共享内存布局上进行更 aggressive 的常量折叠，潜在提升 0‑2% 的吞吐。 |
| **安全考虑** | - 仅移除/重命名文件，无新增外部依赖或 I/O，攻击面未扩大。<br>- 删除的 C++ 文件 (`topk_1024.cuh`) 与 Python 包装一致，不留下未使用的符号。<br>- JIT 编译仍然受 `torch.utils.cpp_extension` 的安全审计约束，未引入不安全的 `exec`。 |
| **可维护性** | - 代码分布更直观，**每个子模块职责单一**（例如 `gemm.py` 只负责 `linear_bf16_fp32`），降低了因修改其他功能导致的意外回归风险。<br>- `__all__` 明确列出公开 API，IDE 自动补全更友好。<br>- 新增的 `utils.make_name` 仍保持统一命名规则，便于后续 JIT 组件统一管理。 |
| **兼容性** | - 所有业务代码已改为导入 `sglang.jit_kernel.dsv4`，但仍保留旧 `deepseek_v4` 包的 **空目录**（未删除），所以外部用户若直接 `import sglang.jit_kernel.deepseek_v4` 将得到 `ModuleNotFoundError`。<br>- 需要在 CI/发布流程中 **确保 JIT 缓存清理**（`torch.cuda.empty_cache()` / `rm -rf ~/.cache/sglang_jit`），否则旧编译的 `.so` 可能因路径变更而加载失败。 |
| **部署/交付** | - 由于 JIT 编译在首次调用时进行，首次启动会重新编译所有 kernel（约 5‑10 秒），与之前行为相同，只是文件路径改变。<br>- 对于 **Docker 镜像** 或 **离线部署**，建议在镜像构建阶段执行一次 “warm‑up” 以预编译所有 kernels，避免运行时突发编译延迟。 |

## ⚠️ 潜在风险

| 风险点 | 说明 | 可能的后果 | 缓解措施 |
|--------|------|------------|----------|
| **导入路径遗漏** | 项目中仍可能残留对 `sglang.jit_kernel.deepseek_v4` 的硬编码（例如第三方插件、旧脚本）。 | 运行时 `ModuleNotFoundError`，导致服务启动失败。 | - 在 CI 中添加搜索 `deepseek_v4` 的脚本，确保仅出现 `dsv4`。<br>- 为向后兼容提供一个轻量的 shim 包（在 `deepseek_v4/__init__.py` 中重新导入 `dsv4`），或在文档中说明迁移步骤。 |
| **宏定义冲突** | 多处 `-DSGL_TOPK=` 编译选项若不统一，可能出现不同 kernel 使用不同 Top‑K 常量导致输入尺寸不匹配。 | 运行时 CUDA 错误（非法内存访问、维度不匹配）。 | - 在 `sglang/jit_kernel/utils.py` 的 `make_cpp_args` 中统一加入 `-DSGL_TOPK=$TOPK`，或在 `topk_v1` 包装层强制传入统一值。<br>- 在单元测试中加入 `assert torch.allclose(output.shape[-1], expected_topk)`. |

**🎯 影响范围**：  
- `sglang/srt/layers/mhc.py`（分割计算与代表 token 选取）  
- `sglang/srt/model_executor/model_runner.py`（统一 warmup 调用）  
- `sglang/srt/models/deepseek_v4.py`、`sglang/srt/models/deepseek_v4_nextn.py`（模型层实现）  
- 依赖的 `tilelang` JIT 编译路径（保持兼容）  

**🔍 技术洞察**：

- **架构影响**：  
  - 引入了 **模型层可选的 warmup 接口**（`kernel_warmup`），在 `ModelRunner` 中统一调度，保持了原有的 FlashInfer 自动调优不受侵入。  
  - 通过 `get_mhc_pre_token_count_representatives` 将 **预分割桶与实际 token 长度解耦**，模型只需关心一组代表性 token 数，降低了对底层实现细节的依赖。  
  - 该机制在 **Hybrid‑Chunk（MHC）** 路径上形成了 **一次性一次性预热**，不再在每次推理时触发 JIT 编译或 kernel 重排。

- **性能影响**：  
  - 预热阶段会执行一次 `hc_pre` 前向路径（包括注意力、FFN 两条路径）对应的 tensor 运算，消耗的时间与 token 长度呈线性关系。  
  - 由于预热只在 **模型启动或第一次运行** 时进行，后续推理可直接使用已调优的 kernel，显著降低了首次 Chunk‑Prefill 的延迟（尤其在启用 `SGLANG_OPT_DEEPGEMM_HC_PRENORM` 与 `SGLANG_OPT_USE_TILELANG_MHC_PRE` 时）。  
  - 通过 `model_runner.tp_group.barrier()` 确保多卡同步，避免了因不同卡上预热状态不一致导致的性能抖动。

- **安全考虑**：  
  - 代码仅涉及计算和日志，不会执行外部输入的动态代码，未引入新的安全风险。  
  - 唯一需要注意的是 **日志信息泄露**：`logger.info` 会把 token 长度打印到日志，若在高度保密环境下使用，建议通过环境变量关闭预热日志。

**⚠️ 潜在风险**：

1. **启动时间增加**：首次加载模型时会额外执行多次 `hc_pre`，在极端 token 长度（如 `max_num_tokens=8192`）下可能导致数秒级的 warmup，影响用户体验。  
2. **显存峰值**：`torch.empty` 为每个 token‑count 创建 `(num_tokens, hc_mult, hidden_size)` 的张量，可能在多卡并行时出现显存峰值，尤其在显存紧张的 GPU 上。  
3. **多进程/多卡同步**：`model_runner.tp_group.barrier()` 依赖 NCCL 正常运行，若集群网络不稳定，可能导致启动阻塞或超时。  
4. **环境变量依赖**：预热仅在 `SGLANG_OPT_DEEPGEMM_HC_PRENORM` 与 `SGLANG_OPT_USE_TILELANG_MHC_PRE` 同时开启时生效，若用户仅打开其中一个，可能产生不一致的行为（部分路径已预热而其他未预热）。  

**💡 关注建议**：

- **可配置性**：提供一个显式的 `--disable-mhc-prewarm` 或对应的环境变量，以让运维人员在显存受限或需要最小化启动时间的场景下关闭该功能。  
- **显存监控**：在 warmup 前后记录显存使用 (`torch.cuda.memory_allocated`) 并在日志中给出提示，帮助用户根据硬件自行调节 `max_num_tokens`。  
- **预热阈值**：可以在内部加入 `if token_counts and token_counts[-1] > device_memory_limit // (hc_mult*hidden_size*2):` 之类的安全检查，防止一次性分配过大张量。  
- **测试覆盖**：在 CI 中加入多卡、不同 `max_num_tokens` 场景的单元/集成测试，确保 `tp_group.barrier()` 不会导致死锁。  
- **文档更新**：在模型部署指南中说明 MHC 预热的作用、何时需要开启以及可能的资源开销，帮助使用者做出合理配置。

### [Spec] Route seq_lens through FutureMap; drop verify_done.wait (#25879)
**SHA**: `baeac17` | 🔗 [查看提交](https://github.com/sgl-project/sglang/commit/baeac179f781f464d19f258c0aaed78e83caaa0e)

**🔍 变更概览**  
1. **FutureMap** 由原来的 `store_to_map_for_new_batch`/`store_to_map` 改为 **publish‑stash** 两步骤：  
   - `publish` 只写入 schedule‑side 的 `new_seq_lens_buf` 并在 GPU 上抛出 `Event`（`publish_ready`），用于后续 **跨流同步**。  
   - `stash` 负责保存 forward‑only 的 `EagleDraftInput`（或 token IDs），延迟到下一个 forward‑batch 使用。  
2. 引入 **lazy‑init** 与 **forward‑only buffer** (`_forward_buf_initialized`) ，避免在 DP 空闲时提前访问未定义的张量形状。

4. `EagleDraftInput` 删除 `verify_done` 事件字段，改为由 `FutureMap.publish` 在 `Scheduler` 里统一触发。  
5. `Scheduler.run_batch`、`EagleWorkerV2`、`MultiLayerEagleWorkerV2` 的 `forward_batch_generation` 接口新增 `on_verify_complete` 回调，以在 **target‑forward**、**verify** 完成后立即发布 fence。  
6. 相关调用处同步改为 `future_map.stash` / `future_map.publish`，删除了大量 `cpu().synchronize()` 逻辑，降低 CPU 阻塞。

**🧭 影响范围**

- `python/sglang/srt/managers/schedule_batch.py`（seq_lens 处理）  
- `python/sglang/srt/managers/scheduler.py`（调度主循环）  
- `python/sglang/srt/speculative/eagle_info*`、`eagle_worker_v2.py`、`multi_layer_eagle_worker_v2.py`（Spec‑v2 工作流）  
- 以及所有依赖 `FutureMap` 的解耦模块（如 `decode_schedule_batch_mixin.py`）。

**💡 关注建议**  
1. **事件正确性**：`publish_ready` 必须在所有涉及 `seq_lens` 的 GPU 操作后记录；若出现跨流 race，检查 `on_verify_complete` 是否在目标/verify 前后调用。
