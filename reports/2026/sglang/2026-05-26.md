### [UnifiedRadixTree]: Support L3 HiStorage framework (#26062)
**SHA**: `38f32c3` | 🔗 [查看提交](https://github.com/sgl-project/sglang/commit/38f32c38abd41a296bd21186b5d8c5e6af1f8997)

**🎯 变更类型**：功能增强 / 架构变更 / 性能优化 / 可观测性扩展  

**⚡ 重要程度**：🔴高  

**📋 变更摘要**  
本次提交为 UnifiedRadixTree 引入 L3 HiStorage 框架的完整支持，主要工作包括：  
1. 在前缀缓存结构中加入 **host‑side 锁引用**（`inserted_host_node`、`swa_uuid_for_host_lock` 等），并在所有组件（Full、SWA、Mamba、SWA）实现 `lock_host` 分支，使得对主机侧状态（host‑value、host‑lock）可独立加解锁。  
2. 扩展 **HybridCacheController**：解析存储后端配置、维护额外的 host‑memory 释放队列、实现 `append_host_mem_release`、`_init_extra_host_mem_release_queues` 等逻辑。  
3. 在 `UnifiedRadixCache` 中加入 **存储备份、预取、锁管理、运行时配置、指标收集** 等功能；实现 `write_backup_storage`、`prefetch_from_storage`、`check_prefetch_progress`、`drain_storage_control_queues` 等一系列调度与资源回收路径。  
4. 为 **Mamba 组件**、**SWA 组件**、**Full 组件** 增加对 host‑value 的写入、预取、备份等处理。  
5. 新增与修改单元/集成测试，加入 GSM8K、MMLU 两轮评估验证缓存刷新前后模型准确率差异，确保功能正确。  

**🎯 影响范围**  
- `sglang/srt/mem_cache/*`（前缀缓存、HybridCache、UnifiedRadixCache、各组件实现）  
- `sglang/srt/observability/metrics_collector.py`（存储指标）  
- 相关测试文件 `test/registered/...`  
- 依赖的运行时配置入口（`ServerArgs`、`CacheInitParams`）  

**🔍 技术洞察**  

| 维度 | 影响描述 |
|------|----------|
| **架构影响** | • 在统一 Radix 树上引入 **两层锁机制**（device lock / host lock），实现对 host‑value（磁盘/文件系统备份）独立保护，提升 HiCache L3（文件后端）对主机侧状态的安全性。<br>• 新增 **HybridCacheController 与 UnifiedRadixCache** 的耦合点，实现统一的 prefetch / backup / release 流程，形成 “存储‑缓存‑计算” 三层闭环。<br>• 通过 `extra_host_mem_release_queues` 支持对非主 pool（如 Mamba pool）进行独立的 host‑memory 回收，增强模块化。 |
| **性能影响** | • **预取阈值、超时、策略**（`prefetch_threshold`、`prefetch_stop_policy`）可在启动时调节，避免小请求频繁 IO，提升吞吐。<br>• `prefetch_from_storage` 在本地 host pool 不足时会自动 **evict_host**，保持高利用率。<br>• 通过 `inc_host_lock_ref / dec_host_lock_ref` 避免因 host‑value 被意外回收导致的重复 IO。<br>• `drain_storage_control_queues` 在多卡同步下统一清理，防止长队列阻塞 IO 线程。<br>• 新增 **StorageMetricsCollector**，可监控预取、写回、备份等关键指标，为后续性能调优提供数据。 |
| **安全考虑** | • 新增 `lock_host` 参数后，所有组件对 host‑value 的加解锁均通过显式路径完成，避免 **并发写冲突**。<br>• `parse_storage_backend_extra_config` 对外部配置文件做 **后缀校验** 与 **类型校验**，防止不受信任的配置导致任意代码执行。<br>• 所有对文件后端的写入均走 **HybridCacheController**，统一使用 `write_storage` 接口，易于审计。 |
| **可维护性** | • 将 host‑side锁相关字段统一到 `IncLockRefResult` / `DecLockRefParams`，保持 API 向后兼容。<br>• 大量业务逻辑被抽象为 `*_host` 方法与 `CacheTransferPhase` 分支，代码结构更清晰。<br>• 通过 `@lru_cache` 缓存 `UnifiedTreeNode.get_prefix_hash_values`，避免重复计算。<br>• 测试用例覆盖了 **两轮评估 + cache flush** 场景，验证功能回归。 |

**⚠️ 潜在风险**  
1. **锁泄漏**：如果在异常路径（如预取分配失败）未必调用 `dec_host_lock_ref`，可能导致 host‑lock 永久持有，导致后续预取被阻塞。  
2. **资源竞争**：`extra_host_mem_release_queues` 采用 `queue.Queue`，在高度并发的多卡环境下可能出现 **生产者‑消费者不平衡**，导致某些 pool 的 host‑memory 迟迟不被释放。  
3. **配置不一致**：运行时 `enable_storage_metrics`、`extra_metric_labels` 的标签集合变化时，仅会打印警告，可能导致监控系统出现 **指标重复或丢失**。  
4. **IO 延迟放大**：在 `prefetch_stop_policy=wait_complete` 下，若后端文件 IO 较慢，整个请求会被阻塞，影响吞吐。  
5. **兼容性**：旧版部署若未打开 `SGLANG_ENABLE_UNIFIED_RADIX_TREE` 环境变量，仍会走旧路径，可能出现 **配置冲突**（如仍使用 `cache_controller` 的老字段）。  

**💡 关注建议**  
- **复核异常路径**：确保在所有 `try/except` 分支中均能匹配 `inc_host_lock_ref` → `dec_host_lock_ref` 成对出现，必要时在 `finally` 中统一释放。  
- **监控 Prefetch 超时**：在监控平台上添加 `prefetch_timeout`、`prefetch_queued` 等指标，及时发现 `wait_complete` 策略导致的阻塞。  
- **调优阈值**：根据实际工作负载（token 长度、页大小）对 `prefetch_threshold`、`prefetch_timeout_*` 进行压测，避免过小导致频繁 IO、过大导致内存占用突增。  
- **资源配额**：对 `extra_host_mem_release_queues` 进行限流或周期性 `drain`，防止单个 pool 的释放请求堆积。  
- **统一标签**：如果需要在运行时更改 `extra_metric_labels`，建议在 `StorageMetricsCollector` 中实现 **标签重建** 并 **注销旧指标**，避免 Prometheus 报错。  
- **回滚方案**：保留旧的 `HybridCacheController._start_storage_threads` 调用方式，确保在不支持 L3 HiStorage 的环境中仍可正常启动。  

通过上述审视，可确认本次改动在功能上实现了 HiCache L3（文件后端）的完整集成，并对性能、可观测性及安全性均作了相应强化。后续应重点关注锁的成对释放及预取策略对整体吞吐的影响。

**📋 变更摘要**

## 🔍 技术洞察  

| 维度 | 影响描述 |
|------|----------|
| **架构影响** | - 在 `ServerArgs` 中加入 `enable_mamba_extra_buffer`、`mamba_cache_chunk_size`、`fallback_attention_backend` 等新字段，形成 **Mamba2‑extra‑buffer** 的全局开关。<br>- `HybridLinearAttnBackend` 与 `Mamba2AttnBackend` 之间的调用链被重新组织：前向返回 **intermediate_states**，随后在 `forward` 中统一完成状态复制与缓存追踪。<br>- `MambaRadixCache` 现在依据 `mamba_cache_chunk_size`（取 `FLA_CHUNK_SIZE` 或模型配置的 `mamba_chunk_size`）进行页面对齐校验，支持 **branching point caching**。<br>- 新增 `track_*` 索引字段在 `Mamba2Metadata` 中通过 `prepare_*` 统一计算，避免在运行时临时分配。 |
| **性能影响** | - **非 KV‑cache‑受限** 场景：`extra_buffer` 能在 prefill 期间提前缓存 Mamba 状态，实现 **prefill‑decode 重叠**，显著提升吞吐（官方文档中提到 1.16×‑2× 之间的提升）。<br>- **KV‑cache‑受限** 场景：每个请求会消耗约 2× 的 Mamba 状态内存，导致最大并发请求数下降；因此对 `--max-mamba-cache-size` 与 `--max-running-requests` 的配合使用尤为关键。<br>- 通过 `mamba_track_interval`（默认 256）与 `page_size` 对齐，降低 ``track`` 过程的 kernel launch 开销。 |
| **安全考虑** | - 变更主要为内部状态管理与调度策略，无新增外部依赖或网络交互，**安全风险极低**。<br>- 新增 **assert** 检查（如 `conv_states_shape[-1] < mamba_chunk_size`、`mamba_track_interval >= mamba_chunk_size`）在不满足前置条件时直接抛异常，防止产生未定义行为，但如果生产环境中误配置会导致服务启动失败。 |

## ⚠️ 潜在风险  

1. **内存激增**  
   - `extra_buffer` 会把每个请求的 Mamba 状态翻倍（非 Spec）或 1.1‑1.2×（Spec）。若 `--max-mamba-cache-size` 与 `page_size` 配置不合理，可能导致 OOM，尤其在多租户服务中。  
2. **参数对齐假设**  
   - 多处代码假设 `mamba_cache_chunk_size` 能被 `page_size` 整除（或相反），并使用 `assert` 检查。若模型的 `mamba_chunk_size` 与 `page_size` 之间不满足整除关系，启动会失败。  
3. **分支点缓存依赖特定后端**  
   - `extra_buffer` 的分支点缓存仅在 **FLA**（FlashInfer）后端实现，若用户强制使用 `triton` 或自定义后端，可能失去该特性或触发兼容性警告。  
4. **状态追踪 Bug**  
   - `track_conv_indices`、`track_ssm_*` 的索引计算逻辑复杂，跨模型（FLA vs Mamba2）共用同一实现，若 chunk 大小不一致或页面尺寸变化，可能出现越界或错误的缓存状态，导致生成质量下降。  
5. **回归影响**  
   - 由于 `forward` 现在返回 `intermediate_states` 并在之后进行状态追踪，若外部调用（例如自定义层或测试 mock）未适配可能报 `NoneType` 错误。

### [GDN] Support SM100 CuTeDSL GDN Prefill Kernel (#26200)
**SHA**: `d34d4d9` | 🔗 [查看提交](https://github.com/sgl-project/sglang/commit/d34d4d9f5f366af3291433650546b8b5f9ed58b4)

**🎯 变更类型**：功能增强 / 架构变更 / 性能优化  
**⚡ 重要程度**：🔴 高

5. **日志可观测性**：异常路径已加入 `logger.error`，建议在文档或 README 中补充对应的日志提示，让运维能够快速定位 “scheduler hit an exception” 并判断是否已触发进程组杀死。  

整体来看，此次改动提升了调度器异常时的快速失效能力，防止残留子进程占用资源；但在生产部署前请评估进程组划分及平台兼容性，避免意外全局杀进程。
