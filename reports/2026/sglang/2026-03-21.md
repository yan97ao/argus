### [diffusion] CI: make auxiliary coverage explicit and simplify testcases (#20983)
**SHA**: `6dfa8a4` | 🔗 [查看提交](https://github.com/sgl-project/sglang/commit/6dfa8a40bc6a18512677b4cc60a42bf76fadea68)

**变更类型**：功能增强 / CI 改进  
**重要程度**：🟡 中  

### 核心改动  
1. **perf_baselines.json**  
   - 多处 stage 时延、`expected_e2e_ms` 等基准值被上调（如 `TextEncodingStage` 从 693 → 800 ms、整体 E2E 由 25 k ms 直接跃至 329 k ms）。这些数据用于 CI 绩效检查，若基准偏高会导致实际性能不达标的告警被掩盖，需要确认是否为真实预期或误写。  

2. **服务器启动参数**  
   - `DiffusionServerArgs` 新增 `cfg_parallel`，在 `diffusion_server` 中自动拼接 `--enable-cfg-parallel`。若底层服务器未实现该开关，CI 将因未知参数报错。  

3. **测试用例可控性**  
   - `DiffusionTestCase` 增加一系列 `run_*` 布尔字段（模型 API、LoRA 基础/动态加载、动态切换、多 LoRA、T2V 输入校验），并在 `__post_init__` 中做依赖检查。对应的测试逻辑在 `test_server_common.py` 中改为基于这些标记有条件执行，从而实现 **“显式 auxiliary coverage”**。  
   - 删除了 Helios 系列的 flaky 测例，降低 CI 不确定性。  

4. **错误信息提取 & 视频尺寸校验**  
   - 新增 `_extract_async_job_error_message`，在异步视频生成任务失败时提供更细致的报错信息。  
   - 在 `test_server_utils._create_and_download_video` 中加入对 `enable_upscaling` 参数的尺寸放大校验，防止因上采样导致宽高不匹配的误判。  

### 影响范围  
- **CI 测试套件**：所有 diffusion 相关的多模态生成测试将执行更多检查，CI 时长会略增。  
- **服务器启动脚本**：依赖 `--enable-cfg-parallel` 参数的环境需要更新或回退兼容。  
- **基准文件**：perf_baselines.json 被多个测试读取，基准值的改动直接影响 “performance check” 是否通过。  

### 建议  
1. **核实基准数值**：确认大幅上调是否为预期，否则可能掩盖实际性能回退。  
2. **兼容性检查**：在本地或镜像环境中跑一遍 CI，确保服务器接受 `--enable-cfg-parallel`，并检查新版 `DiffusionTestCase` 的默认布尔值（多数为 `True`），防止意外打开不必要的测试。  
3. **文档同步**：更新 README/CI 文档，说明新增 `cfg_parallel` 开关及 `run_*` 标记的意义和默认行为。  
4. **监控 CI 时长**：若测试耗时明显增长，可考虑在 CI 中对高成本的 `run_*` 项做分组或并行化。  

总体而言，此次提交提升了 CI 对功能的覆盖度与错误诊断能力，但需要对性能基准和服务器兼容性进行一次确认，避免因基准误差或参数不匹配导致的误报。

### Fix streaming logprobs corruption caused by shared mutable list reference (#21030)
**SHA**: `dba6fb3` | 🔗 [查看提交](https://github.com/sgl-project/sglang/commit/dba6fb3d308d19e8068380fe34ce3292cd7a67ce)

**🎯 变更类型**：Bug 修复（流式返回 `logprobs` 时出现数据错位）  
**⚡ 重要程度**：🟡 中  

**📋 变更摘要**  
- 在 `TokenizerManager.add_logprob_to_meta_info` 中为 `meta_info` 增加了 `output_token_logprobs_length`，记录 `output_token_logprobs` 的真实长度，避免在后续流式处理时共享同一列表导致的切片错位。  
- `serving_chat.py`、`serving_completions.py` 读取该长度字段，并在计算增量 `logprobs` 时使用 `n_prev_token:total_output_logprobs` 而不是直接依据列表长度。  
- 相应的 `_process_streaming_logprobs` 接口签名也加入 `total_output_logprobs` 参数。  
- 单元测试更新，加入对 `logprobs.tokens` 为字符串的显式断言，防止空值或类型异常。  

**🎯 影响范围**  
- **核心模块**：`python/sglang/srt/entrypoints/openai/serving_chat.py`、`serving_completions.py`（流式生成路径）  
- **状态管理**：`python/sglang/srt/managers/tokenizer_manager.py`（meta_info 结构）  
- **测试**：`test/registered/openai_server/basic/test_openai_server.py`  

**💡 关注建议**  
1. **向后兼容**：新增的 `output_token_logprobs_length` 为可选字段，建议在旧版本模型或插件中添加容错（若不存在则回退到 `len(output_token_logprobs)`）。  
2. **并发安全**：确认 `meta_info` 在多请求并行时不会被意外共享，尤其是 `output_token_logprobs` 列表的深拷贝是否仍必要。  
3. **文档更新**：在 OpenAI 接口文档中说明 `logprobs` 现在可能在流式结束时返回 `None`，并解释 `finish_reason` 与日志概率的对应关系。  
4. **测试覆盖**：新增对 `finish_reason` 为 `null`、`stop`、`length` 等多种场景的流式 `logprobs` 断言，防止未来改动再次引入切片错误。  
5. **性能评估**：虽然只增加了一个长度字段，影响微乎其微，但建议在高并发基准中确认不会出现额外的内存拷贝开销。  

总体来看，此次改动修复了流式 `logprobs` 的数据错位问题，影响集中在 OpenAI 兼容层的流式实现，使用方无需改动调用方式，只要确保服务端部署了最新代码即可。

### [Test] Fix flaky multi-instance memory occupation test (#21074)
**SHA**: `9614271` | 🔗 [查看提交](https://github.com/sgl-project/sglang/commit/9614271ae48a43df5d358c450855c838c136b7ce)

**🎯 变更类型**：功能增强 / Bug 修复（针对 flaky 的多实例内存占用测试）  
**⚡ 重要程度**：🟡 中  

**📋 变更摘要**  
- 将原先基于 `torch.cuda.device_memory_used()` 的 GB 统计改为 `torch.cuda.mem_get_info` 的 MB 统计，并抽取为 `get_gpu_memory_mb`。  
- 新增 `MIN_DELTA_MB = 200` 与 `assert_memory_increased/decreased` 辅助函数，用统一阈值校验内存增减，避免因微小波动导致 flaky。  
- 仅在 TP master（`rank % tp_size == 0`）上进行内存测量与断言，防止同 GPU 上的非主进程共享内存导致不确定结果。  
- 移除 `device_map` 参数的 HF 模型加载，改为先 `from_pretrained` 后显式 `.to(f"cuda:{rank}")`，规避 transformers v5 中 accelerate hook 带来的残留引用。  
- 在关键点加入 `dist.barrier`，确保跨进程/跨 TP 同步后再进行后续操作。  
- 删除了原来的 `time.sleep` 与冗余打印，简化流程。

**🎯 影响范围**  
- 仅 `test/registered/rl/test_multi_instance_release_memory_occupation.py`，不影响生产代码。但该测试涉及的 `EngineWrapper`、`init_device_mesh` 等内部接口在 CI 中会被调用，间接影响多卡/多实例的测试环境。

**💡 关注建议**  
1. **阈值可调**：`MIN_DELTA_MB` 取 200 MB 对 1 GB TP‑rank 来说较为安全，若后续加入更大模型或不同硬件，建议把阈值做成可配置或基于模型大小动态计算。  
2. **CUDA 可用性**：CI 环境可能不存在 GPU，考虑在测试入口加入 `@unittest.skipIf(not torch.cuda.is_available(), ...)`，防止因 `mem_get_info` 报错导致全套测试挂掉。  
3. **兼容性**：移除 `device_map` 可能影响其它测试或示例代码，确认项目中不再有依赖该参数的场景，或在文档中注明已改为显式 `.to(device)`。  
4. **同步粒度**：目前在加载/释放 HF 模型前后使用 `dist.barrier`，确保所有 TP rank 已完成相同阶段。若后续加入更多非 TP‑master 操作，需相应补充屏障，以免出现隐式竞争。  
5. **清理策略**：在释放 HF 模型后仅调用 `gc.collect()` 与 `torch.cuda.empty_cache()`，已足够；若未来添加额外的缓存（如 `torch.compile`），请同步更新清理逻辑。  

总体而言，此次改动通过更细粒度的内存监控和进程同步，大幅提升了测试的确定性，风险主要在阈值设置与 GPU 环境依赖上，建议在 CI 中持续观察通过率并根据实际硬件调优阈值。
