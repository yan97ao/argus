### [HiCache][AMD] Add UMBP tiered DRAM + SSD L3 storage backend with hugepage host allocator   (#25377)
**SHA**: `13dc5f2` | 🔗 [查看提交](https://github.com/sgl-project/sglang/commit/13dc5f2dc789e7012a7fd4266e368940a93b7a52)

**🎯 变更类型**：功能增强  
**⚡ 重要程度**：🔴高  

**📋 变更摘要**  
- 为 HiCache 引入全新 “mori” 存储后端，实现基于 AMD UMBP（本地 DRAM + SSD） 的 L3 分层存储。  
- 新增 `UMBPHostTensorAllocator` 以支持大页（hugepage）以及 RDMA 零拷贝注册，统一进程内存分配路径。  
- 扩展 `CacheController`、`BackendFactory`、`ServerArgs` 支持 `mori` 类型，并在配置层面加入完整的可选参数列表（DRAM/SSD 容量、SPDK、DP/TP/PP 隔离、kv events 等）。  
- 同时提交对应单元测试，验证在未装载真实 `mori` 时的回退路径以及完整的存取逻辑。

---

**🎯 影响范围**  
- `python/sglang/srt/managers/cache_controller.py`（缓存控制流程）  
- `python/sglang/srt/mem_cache/pool_host/common.py`（主机张量分配器）  
- `python/sglang/srt/mem_cache/storage/backend_factory.py`（后端工厂注册）  
- `python/sglang/srt/server_args.py`（CLI 参数）  
- 新增模块 `sglang/srt/mem_cache/storage/umbp/`（`umbp_host_allocator.py`、`umbp_store.py`、`__init__.py`）  
- 新增单元测试 `test/registered/unit/mem_cache/test_umbp_host_allocator.py`、`test_umbp_store.py`  

---

**🔍 技术洞察**  

- **架构影响**  
  1. **存储层扩展**：在 HiCache 原有的 L1 （GPU SRAM）→ L2 （CPU DRAM）→ L3 （SSD） 三级层次中，L3 现在可由 AMD UMBP 实现的 “tiered DRAM + SSD” 取代原有的 `hf3fs/mooncake/eic/nixl/simm`。  
  2. **后端工厂统一**：`StorageBackendFactory.register_backend` 新增 `mori` 条目，后端实例化与其他内置后端保持同一代码路径，降低维护成本。  
  3. **分配器抽象**：`get_allocator_from_storage` 增加 `"mori"` 分支，返回 `UMBPHostTensorAllocator`，在缺失 `mori` 时自动回退到 `HostTensorAllocator`，实现 **向后兼容**。  
  4. **Zero‑Copy RDMA**：在分布式模式下，`UMBPStore.register_mem_pool_host` 可一次性注册整个 KV 缓冲区，实现跨节点的零拷贝读写；通过 env `SGLANG_HICACHE_HOST_HUGEPAGE` 及相关变量控制大页和 NUMA。  
  5. **配置可组合**：支持 **DP/TP/PP** 多维度分区、SPDK 代理、租户隔离、自动心跳等高级特性，确保在大规模模型并行时仍能保持 SSD 资源独立与可预测。  

- **性能影响**  
  - **正面**：  
    - **大页分配**（2 MiB）显著降低 TLB miss，提升 DRAM → GPU KV 拷贝吞吐。  
    - **RDMA 零拷贝**（在分布式模式下）避免 staging buffer 的 memcpy，降低 CPU → NIC 带宽占用与延迟。  
    - **SSD Tier**：本地 SSD 高吞吐、低延迟，作为 L3 缓存可显著减少跨机器数据回溯。  
    - **分布式租户配额**的自动计算避免 “NO_SPACE” 错误，提升系统可用性。  
  - **潜在开销**：  
    - 初始化时 `UMBPClient` 会创建内部 IO 引擎、页面分配器等对象，启动时间略增（尤其在 SPDK proxy 模式下）。  
    - 大页分配失败会回退到 4 KiB 页面，带来额外的 **分页与内存碎片** 开销，且会产生警告。  
    - 零拷贝注册失败（如 NIC 不支持）将自动降级为 staging‑buffer 路径，可能导致每次批次的 **额外 memcpy**。  

- **安全考虑**  
  - **依赖安全**：新增对 `mori.umbp`（AMD ROCm 专属） 的动态导入；若库被恶意篡改，可能导致任意内存映射或 RDMA 注册错误。建议在发行版中对 `mori` 包进行签名校验或通过受限的运行时环境（如容器）进行隔离。  
  - **环境变量**：大量通过 `os.getenv` 控制行为（例如 `SGLANG_HICACHE_HOST_HUGEPAGE`、`UMBP_DISABLE_ZERO_COPY_REGISTER`），不当的值可能导致内存泄漏或不可预期的注册行为。建议在文档中明确安全边界，并在启动阶段进行参数校验。  
  - **网络交互**：分布式模式下会向 `master_address`、`io_engine_host:port` 发起 RDMA/Spdk 连接。若未做认证，可能成为 **未授权远程访问** 的入口。建议在生产环境开启 TLS 或使用内部可信网络。  

---

**⚠️ 潜在风险**  

| 风险点 | 说明 | 严重程度 | 缓解措施 |
|--------|------|----------|----------|
| **缺失 `mori` 库** | 在非 AMD 环境或未正确编译 `mori` 时，`import mori.umbp` 抛异常。虽然已有回退，但部分功能（zero‑copy、hugepage）失效，性能下降。 | 中 | 自动回退日志已实现；在 CI 中加入 `mori`‑不可用路径的覆盖测试；在发行说明中明确 “AMD/ROCm 环境必需”。 |
| **大页不可用** | 若系统未配置足够的 HugePages，分配器会降级为 4 KiB 页面，导致性能回退并出现警告。 | 低 | 检查 `/proc/meminfo`，在启动脚本中检测 `vm.nr_hugepages`。 |
| **RDMA 注册失败** | NIC 不支持或 MR 大小受限时，`register_memory` 返回 `False`，系统会回退到 staging‑buffer，增加延迟。 | 中 | `SGLANG_HICACHE_HOST_DISABLE_ZERO_COPY_REGISTER` 环境变量可手动关闭；提供监控指标 `mapped_size`、`_zero_copy_registered`。 |
| **配置冲突/误用** | 某些 `extra_config` 键只在分布式或单机模式有效，错误使用会导致 silent‑no‑op，用户难以定位。 | 低 | 在 `StorageBackendFactory` 初始化阶段加入 **配置校验**（已通过 `_warn_extra_config_scope`），但仍建议在文档中列出“仅在分布式/单机生效”。 |
| **租户 ID 计算错误** | 自动递增租户 ID（`spdk_proxy_tenant_id_base + dp_rank`）如果 `dp_rank` 超出范围，可能产生 ID 冲突。 | 中 | 在 `UMBPStore.__init__` 中加入显式的范围检查并在冲突时抛异常。 |
| **并发安全** | `UMBPHostTensorAllocator.__del__` 在解释器退出时可能被多线程并发调用，导致 `free` 重复或竞争。 | 低 | 当前实现捕获异常并使用 idempotent `free_calls`，仍建议在 `__del__` 前加锁

### [diffusion] CI: add 5090 job (#29791)

**🎯 变更类型**：功能增强  
**⚡ 重要程度**：🔴高  
**📋 变更摘要**：  
- 为 RTX 5090 GPU 新增 CI 作业 `multimodal-gen-test-1-5090`，在 CI 中跑 Diffusion canary 测试并上传执行报告和失败产物。  
- 引入平台化的性能基准（perf_baselines）和一致性阈值（consistency_threshold）配置文件，支持 `h100`、`b200`、`5090` 三类硬件，并通过环境变量 `SGLANG_DIFFUSION_PERF_BASELINE_PLATFORM`、`SGLANG_DIFFUSION_CONSENSUS_PLATFORM` 可手动覆盖。  
- 扩展测试框架以发现、加载并合并平台特定的基线与阈值，同时新增 5090 专用测试用例、基线数据与阈值文件。  

**🎯 影响范围**：  
- `.github/workflows/pr-test-multimodal-gen.yml`（CI 工作流）  
- `python/sglang/multimodal_gen/test/`（diffusion 测试框架、基线与阈值加载）  
- `scripts/ci/utils/diffusion/`（CI 基线解析脚本）  
- 相关内部工具 `test_utils.py`、`testcase_configs.py`、`gpu_cases.py`、`test_server_common.py`  

**🔍 技术洞察**：

- **架构影响**  
  - **平台抽象层**：新增 `get_perf_baseline_platform() / get_consistency_platform()` 与对应的路径解析函数，实现“同一套代码在不同硬件上自动使用对应基线”。这在代码层面引入了平台感知的配置分支，提升了可扩展性。  
  - **CI 扩展**：新的 5090 作业与现有 `multimodal-gen-test-1-gpu` 并行，保持已有流水线不变，仅在 `inputs.multimodal_gen == true` 时激活。  
  - **配置组织**：把原来单一的 `perf_baselines.json` 拆分为 `perf_baselines/h100.json`、`b200.json`、`5090.json`，并在 `testcase_configs.py` 中通过 `PERF_BASELINE_FILE_BY_PLATFORM` 动态定位。  

- **性能影响**  
  - **基线精度提升**：5090 具备更高算力，专属基线使 CI 中的性能回归判定更精准，减少误报/漏报。  
  - **启动开销**：加载多份基线文件并做合并（`load_baselines` 中的 `_iter_baseline_paths`），在 CI 环境中几乎可忽略（I/O < 10 ms）。  
  - **新测试用例**：`ONE_GPU_5090_CASES` 包含层级 CPU offload 场景，可能在 CI 中消耗更长时间，但已在基线中预估（约 94 s‑329 s），超出 `timeout-minutes: 120` 的作业仍受保护。  

- **安全考虑**  
  - 仅是内部配置文件的路径读取和环境变量控制，不涉及网络请求或外部依赖。  
  - 新增的 CI 作业使用官方 Action（`checkout`、`download-artifact`、`upload-artifact`），保持原有安全模型。  
  - 环境变量的可控性：若恶意用户在 PR 中自行注入 `SGLANG_DIFFUSION_PERF_BASELINE_PLATFORM`，仍只能切换基线文件，未暴露系统路径或执行代码的风险。  

**⚠️ 潜在风险**：  
1. **基线文件缺失或命名错误**：若平台对应的 JSON 未同步到仓库，`load_baselines` 会返回空基线，导致 CI 误判或 `pytest` 因找不到基线而失败。  
2. **平台检测失效**：`current_platform.is_sm120()`/`is_blackwell()` 依赖底层硬件检测库，若在 CI 镜像中误判，会加载错误基线，引发性能回归误报。  
3. **环境变量冲突**：用户在 PR 中自行设定 `SGLANG_DIFFUSION_PERF_BASELINE_PLATFORM` 可能导致基线不匹配，产生不一致的报告。  
4. **作业超时**：新增 5090 测试用例的预计执行时间最高达 **329 s**（约 5.5 min），但作业超时限制为 120 min；若多个 5090 用例并行执行或出现性能回退，可能触及 `timeout-minutes: 180` 的上限。  
5. **向后兼容**：旧的 `perf_baselines.json` 仍被引用（如部分脚本或第三方工具），若未同步更新，可能出现“基线文件已迁移但仍被旧路径读取”的情况。  

**💡 关注建议**：  
- **CI 监控**：在 CI 运行后自动检查 `perf_baselines/` 目录是否完整（包括 `h100.json`、`b200.json`、`5090.json`），并在缺失时抛出明确错误。  
- **文档同步**：在项目 README 或 CONTRIBUTING 中补充平台基线的使用说明、环境变量含义以及如何在本地调试不同平台的基线。  
- **回退策略**：若平台检测失败，可在 `get_perf_baseline_platform()` 中加入默认回退到 `h100`，并在日志中给出警告。  
- **测试覆盖**：确保新增的单元测试（`test_consistency_metrics.py`、`test_turbo_wan_backend.py`）覆盖平台合并逻辑、阈值覆盖以及 `load_baselines` 的多路径遍历。  
- **基线维护流程**：为每次硬件或模型更新准备 CI 脚本自动生成平台基线（已支持 `SGLANG_GEN_BASELINE=1 pytest`），并在 PR 模板中加入“更新对应平台基线文件”检查项。  

通过上述改动，项目在多硬件环境下的 Diffusion 测试更加可靠，性能回归检测更具针对性，同时保持了良好的可维护性和安全边界。

### feat(hicache): Use NIXL path-mode (#27060)
**SHA**: `7d9de81` | 🔗 [查看提交](https://github.com/sgl-project/sglang/commit/7d9de81cda08d7fb63cf5f56266cd0ed82a96541)

**🎯 变更类型**：功能增强  
**⚡ 重要程度**：🔴高  
**📋 变更摘要**：在 HiCache 的 NIXL 存储实现中新增对 **path‑mode** FILE 注册的探测与使用，默认在 NIXL ≥ 1.3.0 时启用；若不支持则回退到传统的文件描述符注册。为保证 path‑mode 写入不因目录缺失而失败，文件管理器在初始化时预先创建所有 bucket 目录。相应测试也加入了对键值混淆的校验，以防止因路径冲突导致数据混乱。  

**🎯 影响范围**：  
- `python/sglang/srt/mem_cache/storage/nixl/nixl_registry.py`（注册/注销逻辑）  
- `python/sglang/srt/mem_cache/storage/nixl/nixl_utils.py`（文件管理器）  
- `test/registered/unit/mem_cache/test_hicache_nixl_storage.py`（单元测试）  

**🔍 技术洞察**：  
- **架构影响**：  
  - 引入 **path_mode** 标志，使 `NixlRegistry` 在运行时决定走 “路径+metaInfo” 方式还是传统 `fd` 方式，保持向后兼容。  
  - `NixlFileManager.ensure_all_bucket_dirs` 在启动时一次性创建 `0x0…0xF`（取决于 `_BUCKET_MASK` 与 `BUCKET_HEX_CHARS`）的子目录，防止 NIXL O_CREAT 写入时因父目录缺失抛错。  
  - 通过 `self._probe_path_mode()` 的异常驱动探测实现，无需额外依赖版本号检测，降低硬编码风险。  

- **性能影响**：  
  - **正面**：path‑mode 直接使用文件系统路径注册，省去打开文件描述符的系统调用和 FD 表占用，尤其在高并发/大缓存场景下可以显著降低 `fd` 消耗与 `select/poll` 开销。  
  - **负面**：首次启动时会创建所有 bucket 目录，I/O 开销与磁盘占用略增（但仅一次性操作且目录本身极小）。若目录结构异常（权限不足）会导致注册失败，回退到 FD 方式仍可正常工作。  

- **安全考虑**：  
  - Path‑mode 使缓存文件的真实路径暴露给 NIXL，若底层 NIXL 对路径做不充分的权限检查，可能导致跨用户文件访问风险。当前实现仅在内部 `base_dirs`（由 SGLang 受控）下创建目录，风险相对可控。  
  - 预先创建目录的过程使用 `os.makedirs(..., exist_ok=True)`，默认会遵守系统 umask，未显式设置权限，建议在生产环境审计目录权限。  

**⚠️ 潜在风险**：  
1. **误判 Path‑mode 支持**：`_probe_path_mode` 依赖 NIXL 在不可创建路径时抛异常。如果未来 NIXL 更改错误码或返回成功而不检查路径，可能误判为不支持，导致继续使用 FD 注册，失去性能收益。  
2. **目录创建失败**：在只读文件系统或权限受限的环境下，`ensure_all_bucket_dirs` 可能抛异常，使整个缓存初始化失败。  
3. **兼容性回退路径**：回退到 FD 方式时仍保留 `self.path_mode` 为 `False`，但已有的 bucket 目录仍会被创建，产生不必要的 I/O。  
4. **测试依赖随机路径**：单元测试使用固定的 `"/nonexistent-nixl-probe"`，若系统上存在同名文件或目录（极小概率），探测结果会异常。  

**💡 关注建议**：  
- **版本检测**：在未来 NIXL 稳定发布后，可考虑加入基于 `nixl.__version__` 的显式判断，减少探测依赖。  
- **异常细化**：将 `_probe_path_mode` 捕获的异常限定为 NIXL 相关错误（如 `NIXL_ERR_BACKEND`），避免把其他意外异常误判为 “支持”。  
- **目录权限**：在 `ensure_all_bucket_dirs` 中显式使用 `os.makedirs(path, mode=0o750, exist_ok=True)`，并在文档中说明所需的最小权限。  
- **回退日志**：当前已记录信息级日志，建议在生产环境提升为 **warning**，便于运维快速发现未激活 path‑mode 的节点。  
- **监控指标**：加入 `hicache_nixl_path_mode_enabled` 计数指标，帮助观察实际使用比例及后续性能分析。  
- **测试覆盖**：继续保持对非零拷贝路径的键值唯一性校验，防止未来对 `spec` 拼接的改动导致不同键映射到同一文件。  

总体来看，此次改动在保持向后兼容的前提下，为 HiCache 引入了更高效的 NIXL 文件注册路径，能够在支持的环境中显著降低文件描述符消耗和提升并发度，只要注意上述兼容性与权限细节，风险可控。

### fix: make write_token dynamic (#29271)
**SHA**: `41779d5` | 🔗 [查看提交](https://github.com/sgl-project/sglang/commit/41779d56fdb36dc51a698a5804829b9643a348f8)

**🎯 变更类型**：Bug修复  

**⚡ 重要程度**：🔴高  

**📋 变更摘要**：  
1. `ContiguousAttentionKVCache.write_token` 现在在写入新 token 时会检查是否超出当前 `max_seq_len`，若超出则调用内部 `_grow` 动态扩大缓存。  
2. 移除了 `model_runner` 中因缓存不支持动态扩张而留下的 TODO 注释，表明调用方已兼容此行为。  
3. 新增单元测试验证缓存在超过原始上限后能够正确扩容、保持已写入数据顺序不变，并且 `offset` 与 `max_seq_len` 正确更新。  

**🎯 影响范围**：  
- `python/sglang/srt/hardware_backend/mlx/kv_cache/attention_kv_cache.py`（KV 缓存实现）  
- `python/sglang/srt/hardware_backend/mlx/model_runner.py`（模型执行路径）  
- `test/registered/unit/hardware_backend/mlx/test_attention_patching.py`（新增测试）  

**🔍 技术洞察**：  
- **架构影响**：  
  - KV 缓存从固定大小改为支持 **动态增长**，使得 `MLX` 后端在处理长序列或递增生成时不再受限于初始化时的 `max_seq_len`。  
  - 通过在 `write_token` 中局部判断并调用 `_grow`，保持了原有 API 不变，对上层调用者（如 `ModelRunner`）是透明的。  
  - 该改动消除了 `model_runner` 中的 “TODO” 注释，提升了代码完整性与可维护性。  

- **性能影响**：  
  - 动态扩容会在触发时进行一次额外的内存复制（由 `_grow` 实现），带来 **一次性** 的线性时间开销。  
  - 对于大多数推理场景（一次性预分配足够容量），影响可忽略；但在极端超长生成场景下，扩容次数增加可能导致 **内存碎片** 与 **暂时性 latency spikes**。  
  - 读取/写入路径仍保持原有的切片赋值逻辑，后续 token 写入在已扩容的 buffer 中仍保持 **O(1)** 访问。  

- **安全考虑**：  
  - 该修改仅涉及内存管理与数值写入，无外部 I/O、权限或加密操作，**不引入新的安全风险**。  
  - 唯一需关注的是 **内存泄漏或 OOM**：如果生成序列异常长，动态增长可能导致进程占用过多显存，进而触发 OOM 错误。  

**⚠️ 潜在风险**：  
1. **显存占用失控**：没有上限限制的动态增长在极端输入（如恶意构造的超长序列）下可能耗尽显存。  
2. **复制成本**：`_grow` 实现若使用 `mx.concatenate` 或类似全量复制方式，频繁触发会产生较大复制开销。  
3. **线程安全**：若同一个 `KVCache` 实例被多线程并发写入（目前实现假设单线程），`offset` 与扩容检查可能产生竞争条件。  
4. **兼容性**：老的自定义 `KVCache` 子类若覆写 `write_token` 而未实现动态扩容，仍可能在调用方假设可扩容的情形下崩溃。  

**💡 关注建议**：  
- **显存上限**：在生产环境可考虑在 `_grow` 前加入显存上限检查或抛出自定义异常，以防 OOM。  
- **增长策略**：如果 `_grow` 仍采用固定倍增（如 2×），建议记录增长次数并在文档中说明最坏情况显存需求。  
- **性能监控**：在长序列生成任务中监控 `cache.max_seq_len` 与显存使用，确保增长次数在可接受范围。  
- **线程模型**：明确 `KVCache` 为 **单线程/单图构建** 使用，若将来计划并发推理，需要在 `write_token` 上加锁或使用原子操作。  
- **文档更新**：在 `MLX` 后端的 KV 缓存说明中补充 “动态扩容” 章节，告知用户可以不必预先估算最大序列长度。  
- **回归测试**：持续运行新增的单元测试，并在未来加入更大规模的随机写入测试，以捕捉潜在的复制或对齐错误。

总体来看，此次改动消除了 “Full KV 被错误丢弃” 的 bug，代码路径更简洁，风险主要在资源回收和潜在的旧插件兼容性上，按上述检查进行即可安全合并。
