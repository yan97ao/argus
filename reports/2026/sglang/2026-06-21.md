#### 🔴 高重要度变更 (3)

### fix(anthropic): handle mid-conversation system messages (#26773)
**SHA**: `b4dda8b` | 🔗 [查看提交](https://github.com/sgl-project/sglang/commit/b4dda8b3ce8dcec52f96516da9593d51333782cf)

**🎯 变更类型**：Bug修复  
**⚡ 重要程度**：🔴高  
**📋 变更摘要**：  
在 Anthropic 接口的请求模型 `AnthropicMessagesRequest` 中加入了对 **mid‑conversation `role:"system"`** 消息的兼容处理。通过 `model_validator(mode="before")` 将此类消息折叠到顶层 `system` 字段并从 `messages` 列表中剔除，实现了与官方 Anthropic API 的行为一致。相应单元/集成测试也同步补全，验证了合并、仅顶层、以及对象构造路径的正确性。

**🎯 影响范围**：  
- `python/sglang/srt/entrypoints/anthropic/protocol.py`（请求模型与验证器）  
- `python/sglang/srt/entrypoints/anthropic/serving.py`（`_convert_to_chat_completion_request` 依赖 `system` 字段）  
- 与 Anthropic SDK 兼容的所有调用路径（如 `handle_count_tokens`）  
- 相关单元测试与 OpenAI‑style 代理服务器的测试套件  

**🔍 技术洞察**  

- **架构影响**  
  - **模型层**：在 Pydantic V2 中使用 `mode="before"` 的类方法验证器，使得请求在实例化前即完成系统消息的归并，保持了模型的 **单一入口**（`system`）与 **对话顺序**（`messages`）的清晰分离。  
  - **服务层**：`AnthropicServing._convert_to_chat_completion_request` 直接依赖折叠后的 `system` 字段，避免了在转换阶段再次检查或过滤 `system` 消息，简化了转换逻辑。  
  - **兼容性**：对既有 “只使用顶层 `system`” 与 “在 `messages` 中混入 `system`” 两种用法均兼容，且对程序化构造（使用 `AnthropicMessage` 对象）也保持了原有行为。

- **性能影响**  
  - 新增的验证器在 **请求解析阶段** 进行一次线性遍历 (`O(n)`，n 为消息数) 的过滤与文本抽取，开销极低（几百毫秒以内的微秒级），对整体吞吐量影响可以忽略不计。  
  - 由于折叠后 `messages` 数量可能略有减少，后续推理、token 计数等环节的处理量也会随之略微降低，甚至略有正向收益。

- **安全考虑**  
  - 仅对 **输入验证** 与 **字段合并** 进行操作，不涉及外部资源访问或代码执行，未引入新增的安全风险。  
  - 仍然遵循 Pydantic 的严格类型校验，确保 `system` 内容在最终转为字符串前已去除空白或不合法块，防止潜在的 **注入**（如恶意 `system` 内容被直接拼接进提示词）风险。  

**⚠️ 潜在风险**  

1. **Pydantic 版本依赖**：`mode="before"` 只在 Pydantic v2 及以上可用，若项目在未来降级或在环境中误用了旧版，验证器会失效导致 `system` 消息未被折叠。  
2. **类型兼容**：`system` 字段接受 `str | list[AnthropicContentBlock]`；在合并多个系统消息时强制转为 `str`（`\n` 拼接），若用户原本期望以块列表形式提供结构化系统内容，可能导致 **语义损失**。  
3. **异常路径**：当前实现默认忽略空字符串或空块，若业务侧依赖空系统消息触发特定逻辑（不太常见），会产生行为差异。  
4. **日志/监控**：折叠前的 `system` 消息不再出现在 `messages` 列表，若监控/审计依赖原始对话流，需要相应更新日志记录方式。  

**💡 关注建议**  

- **回归测试**：在 CI 中继续保留现有的集成测试（尤其是 `test_in_messages_system_role` 系列），并对 **块列表形式的系统消息** 增加覆盖，以防止未来的 validator 改动破坏此兼容。  
- **文档更新**：在 Anthropic 接口使用说明中明确指出：“在 `messages` 中出现 `role: "system"` 时会被自动折叠到顶层 `system` 字段”，并给出推荐的写法（优先使用顶层 `system`）。  
- **兼容性开关（可选）**：如果有客户希望保留原始 `system` 消息顺序，可考虑在模型中加入可选的 `preserve_mid_system` 标记，避免强制折叠。  
- **监控/日志**：在 `AnthropicServing` 接收请求后，记录折叠前后的 `system` 与 `messages` 长度，以便快速定位因输入异常导致的请求被意外修改的情况。  
- **依赖检查**：在项目的 `requirements.txt` 或 CI 环境中锁定 Pydantic 版本（>=2.0），并在发布说明里提示 “本次更新使用了 Pydantic 2 的 `model_validator`”。  

---  
此更改显著提升了 SGLang 对 Anthropic 客户端的兼容性，降低了因 “系统消息放错位置” 而产生的 400 错误概率，对整体功能可用性和用户体验有明显正面作用。只要注意上述兼容性与日志层面的细节，即可安全上线。

### [mtp] add rejection sampling for speculative decoding (#26312)
**SHA**: `f42ec35` | 🔗 [查看提交](https://github.com/sgl-project/sglang/commit/f42ec350b431d0305d34d6c70ea45fdfcd29dcad)

**🎯 变更类型**：功能增强 / 架构变更 / 性能优化（引入 Rejection Sampling 用于 Speculative Decoding）  

**⚡ 重要程度**：🔴 高  

**📋 变更摘要**  
- 为 Eagle 系列（NEXTN/EAGLE/EAGLE3）投机解码新增 `--speculative-use-rejection-sampling` 标志，开启基于拒绝抽样的 **draft‑to‑target** 验证路径。  
- 在整个 spec 流程中加入 `draft_probs`（每步草稿分布）缓冲、数据搬运、以及新的 Triton 采样 kernel `chain_speculative_sampling_triton`。  
- 通过一系列参数检查（topk 必须为 1、accept‑threshold 必须为 1.0、不可与确定性推理、multi‑layer Eagle 同时使用），确保仅在受支持的模型/算法上生效。  

**🎯 影响范围**  
- `sglang/srt/speculative/*`（EagleInfo、EagleWorkerV2、EagleDraftInputBuffers、相关 util）  
- `sglang/srt/arg_groups/speculative_hook.py`（启动时参数校验）  
- `sglang/srt/server_args.py`（CLI 参数、默认值）  
- `sglang/srt/overlap_utils.py`、`forward_batch_info.py`（draft_probs 缓存、pad）  
- 新增 `sglang/srt/speculative/reject_sampling.py`（Triton kernel）  
- 采样路径 `sglang/srt/speculative/eagle_utils.py`、`spec_utils.py`（温度归一化、fast_sample）  
- 对外测试 `test_eagle_reject_sampling.py`（Qwen3.5‑9B + NEXTN）  

---

### 🔍 技术洞察  

| 维度 | 影响描述 |
|------|----------|
| **架构影响** | - **数据流扩展**：在 `EagleDraftInput` 与 `EagleVerifyInput` 中加入 `draft_probs`，需要在所有转发、缓存、pad、merge、filter 流程同步更新。<br>- **调度层**：`OverlapUtils` 现在管理 `draft_probs_buf`，并在 `stash` 与 `_resolve_spec_extras` 中完成复制。<br>- **执行层**：`EagleDraftInputBuffers` 根据 flag 动态分配 `draft_probs` 缓冲；`EagleWorkerV2` 在 `draft_forward`、`_draft_extend_*` 中收集该分布并传递至 verify 阶段。<br>- **采样层**：全新 Triton kernel 实现 **链式拒绝抽样**（`chain_speculative_sampling_triton`），取代原先的仅针对 target‑only 的 `tree_speculative_sampling_target_only`。<br>- **配置层**：CLI 与 `ServerArgs` 加入 `speculative_use_rejection_sampling`，并在 `speculative_hook` 中做兼容性检查，防止无效组合。 |
| **性能影响** | - **正向额外成本**：每一步 draft 需要 `renorm_draft_probs`（softmax + 温度缩放）以及一次 `draft_probs` 的拷贝，约 **O(B·Vocab)** 的内存访问（Vocab≈32k），对大 batch 可能增加显存占用 1~2 GB。<br>- **采样阶段**：Triton kernel在每个 batch 上遍历 **VOCAB_SIZE** 两遍（求和 + CDF），在显卡上实现块级并行（BLOCK_V=4096），理论上只比原 `tree_speculative_sampling_target_only` 多 2×的算子调用，但由于 GPU 计算密集，整体 latency 预计提升 **5‑15 %**（具体取决于 vocab 大小与 batch）。<br>- **内存利用**：新增 `draft_probs_buf` 与 `temperatures`（在 rejection 路径下独立保存）会占用额外显存；在多卡 TP（TP=2/4）下需确保每张卡的显存充足。<br>- **吞吐**：在开关打开且 draft 与 target vocab 完全相同的情况下，接受率提升可显著降低 target 前向次数，整体吞吐往往仍优于标准 top‑k/采样方案（测试中 avg_spec_accept_length 从 ≈2 提升至 ≈2.8）。 |
| **安全考虑** | - **非确定性**：强制禁止 `--enable-deterministic-inference`，因为采样随机数（`torch.rand_like`）在不同 batch 间不保持一致，可能导致可重现性问题。<br>- **输入校验**：在 `speculative_hook` 中加入多项 `raise` 检查，提前捕获不受支持的算法、topk、accept‑threshold、multi‑layer 配置，降低运行时未定义行为的风险。<br>- **数值安全**：在 Triton kernel 中对 `p - q` 进行 `max(0, ...)`，防止负数导致 CDF 错误；若 **norm_sum == 0**（极端 p==q 场景），返回 `VOCAB_SIZE‑1`，避免除零。 |
| **可维护性** | - **代码分散**：`draft_probs` 相关逻辑跨多个文件（buffers、info、utils、worker），增加认知负担。建议抽象为统一的 `SpeculativeDraftProbBuffer` 类或统一的 `SpeculativeContext`，统一管理生命周期。<br>- **文档缺失**：新增 CLI 参数未同步到 README/帮助文档，用户可能不知道限制条件。<br>- **单元测试**：仅提供了针对 Qwen3.5‑9B 的集成测试，缺少针对不同算法（EAGLE3、STANDALONE）及错误路径的非功能测试。 |

---

### ⚠️ 潜在风险  

1. **显存超限**  
   - `draft_probs_buf` 大小 = `req_pool_size × vocab_size × float32`。在高并发（>256 requests）且 vocab≈32k 时，显存需求可达数十 GB，容易触发 OOM。  
2. **不兼容的组合**  
   - 若用户自行修改 `speculative_algorithm` 为不支持的值，或在脚本中硬编码 `speculative_eagle_topk > 1`，仍会抛 `NotImplementedError`/`ValueError`，影响 CI 自动化。  
3. **多卡 TP/DP 环境下 **`draft_probs`** 同步错误**  
   - 目前 `draft_probs` 只在单卡（或单 TP）下分配，跨卡的 `draft_probs` 复制路径（`stash` → `overlap_utils`）依赖 `req_pool_indices` 对齐，若出现不一致会导致 **概率错位**。  
4. **随机数产生差异**  
   - 采样随机数使用 `torch.rand_like`（默认在当前设备上），若某些算子在 CPU 上执行（fallback），可能导致 **核间 RNG 不同步**，进而出现不可预期的接受率波动。  
5. **兼容性回退**  
   - 在不满足 `speculative_use_rejection_sampling` 条件的情况下，仍会走旧路径；但如果 `draft_probs` 为 `None` 而 `chain_speculative_sampling_triton` 被错误调用，Triton kernel会报空指针异常。  
6. **性能回退**  
   - 对于 **小 vocab**（如 8k）或 **batch=1** 的场景，额外的两遍遍历可能比原 `tree_speculative_sampling_target_only` 更慢，导致 **延迟反而上升**。  

---

### 💡 关注建议  

| 建议类别 | 具体措施 |
|----------|----------|
| **文档 & 使用指南** | - 在 `README.md`、CLI `--help` 中补充 `--speculative-use-rejection-sampling` 的前置条件（EAGLE/EAGLE3、topk=1、相同 vocab）<br>- 给出显存预估公式：`req_pool_size * vocab_size * 4bytes`，帮助运维规划。 |
| **参数校验** | - 在 `ServerArgs` 解析阶段直接检查 `speculative_algorithm` 与 `speculative_use_rejection_sampling` 的兼容性

| 维度 | 影响 | 说明 |

1. **回归检测缺失**

## 潜在风险点

1. **向后兼容**  
   - 只有带有 `Arg`（或裸字符串）注解的字段会生成 CLI 参数。若已有字段在迁移过程中仍缺 `Arg`，对应的命令行选项将**不再可用**，导致运行时错误或功能缺失。需要确保所有已暴露的参数在迁移后仍保留 `A[…]` 注解，或在 `add_cli_args` 中手动补充。  
   - `required` 推断逻辑：当字段没有默认值且 `required` 未显式设 `False` 时，会自动加 `required=True`（仅限以 `-` 开头的名字）。这会把原本 “必填” 参数（如 `model_path`）自动标记为必填，行为应与旧实现保持一致。

2. **类型解析细节**  
   - `Literal`、`List`、`Optional`、`bool` 的处理已覆盖，但自定义复杂嵌套（如 `Dict[str, Any]`、`Union[int, str]`）仍会回落到 `str`，可能产生隐蔽的类型错误。  
   - `type_parser` 只在标量、列表、Literal 场景下生效；如果用户在 `Arg` 中提供了 `type_parser` 而字段为 `bool`，会被忽略（因为 `bool` 直接映射为 `store_true`）。

3. **默认值与 `default_factory`**  
   - `_field_default` 会在解析阶段直接调用 `default_factory()`。若默认工厂具有副作用或依赖运行时上下文，可能在解析阶段提前触发。当前 `ServerArgs` 中未使用 `default_factory`，但后续新增时需注意。

4. **Help 文本和别名**  
   - `cli_name` 默认使用字段名转换为 `--kebab-case`，而 `aliases` 仍需手动提供（如 `--model` 对应 `model_path`）。若忘记同步别名，用户可能找不到熟悉的旧参数名。

5. **顺序与分组**  
   - `argparse` 会按 **字段遍历顺序** 添加参数，顺序可能与旧手写代码略有不同，影响 `--help` 输出的组织。功能上不受影响，但文档体验需要验证。

6. **Python 版本要求**  
   - `get_type_hints(..., include_extras=True)` 需要 Python ≥3.9（项目已满足），但在旧环境中会报错。若要保留对更早版本的兼容，需要提供回退实现。
