### [Diffusion] [NPU] Wan2.2-T2V-A14B-Diffusers modelslim quantization support (#17996)
**SHA**: `5297b02` | 🔗 [查看提交](https://github.com/sgl-project/sglang/commit/5297b02c882b70e985ce7ed251355399ce68bc49)

**🎯 变更类型**：功能增强 / 架构变更 / 性能优化  
**⚡ 重要程度**：🔴 高  

**📋 变更摘要**  
本次 PR 为 Diffusion 系列模型在 Ascend NPU 上引入 **modelslim 量化** 支持（W4A4、W8A8 动态/静态），实现了模型权重的专属压缩方案，并在 CI、文档、测试以及运行时代码中完成了全链路集成。新增的 `ModelSlimConfig` 与对应的 `LinearMethod` 将量化策略下沉到线性层，实现了 NPU‑专属的权重存储格式（NZ）以及缓存管理。

**🎯 影响范围**  
- `python/sglang/multimodal_gen/`（注册表、loader、runtime layers、utils）  
- `python/sglang/srt/`（NPU backend quantization）  
- CI 工作流 `.github/workflows/pr-test-npu.yml`  
- 文档 `docs/platforms/ascend_npu_quantization.md`  
- 测试套件（8‑NPU Diffusion）  

---

### 🔍 技术洞察  

| 维度 | 影响分析 |
|------|----------|
| **架构影响** | 1. **新增量化子系统**：`ModelSlimConfig` 继承自统一的 `QuantizationConfig`，并通过 `ModelSlimLinearMethod` 将 scheme 注入 `LinearBase`。<br>2. **统一前缀机制**：`add_prefix` 与 `prefix` 参数在大量层初始化处加入，以保留模型分片/融合层名称，兼容 `modelslim` 的权重映射。<br>3. **加载流程改造**：`transformer_loader._resolve_quant_config` 现在接受 `component_model_path`，调用新实现的 `quantization_utils.get_quant_config`，实现对 `quant_model_description.json` 的自动发现与解析。<br>4. **NPU 特化**：在 `fsdp_load.py` 与 `linear_method_npu.py` 中加入 NPU‑专属配置（`torch.npu.config.allow_internal_format`、`torch.npu.empty_cache`）以及对 `pertoken_scale` 的 flatten 处理。 |
| **性能影响** | - **模型体积**：W4A4、W8A8 量化可将权重压缩至 4‑8 bit，显著降低显存占用，特别在 8‑NPU 场景下提升可装载模型规模。<br>- **执行效率**：NPU‑专属 `apply` 实现使用内部 `pertoken_scale.flatten()`，配合 `NZ` 格式可提升算子调度效率；CI 基准中 8‑NPU 的 `wan2_2_t2v_14b_w8a8_8npu` 预期 denoise 速度约 2 k ms/step。<br>- **启动开销**：首次加载时会执行 `process_weights_after_loading` 与 `create_weights` 两步，略微增加模型初始化时延，但后续推理受益明显。 |
| **安全考虑** | - 新增对外部 JSON（`quant_model_description.json`）的读取，若模型来源不可信，恶意配置可能导致异常的 `quant_type` 路径或非法 `packed_modules_mapping`。推荐在模型下载前校验来源或签名。<br>- 使用 `safetensors.safe_open` 读取权重元数据，已具备安全的零拷贝加载机制。<br>- CI 中对镜像、缓存的 DNS 替换未影响安全边界。 |
| **可维护性** | - 通过统一的 `QuantizationMethods` 枚举扩展 (`"modelslim"`)，保持与已有 `fp8` 路径的一致性。<br>- `add_prefix` 与 `prefix` 参数的广泛传播提升了层级追踪的可读性，但也增加了函数签名的复杂度，需在后续代码中坚持使用统一前缀生成工具。<br>- 对 `_get_config_info` 的匹配逻辑改为 `in`，降低误匹配风险，但可能引入意外的模糊匹配，建议添加单元测试验证。 |

**⚠️ 潜在风险**  

1. **兼容性回退**  
   - 非 NPU 环境仍会加载 `modelslim` 相关模块；若 `is_npu()` 判定错误（如 NPU 驱动缺失），可能触发 `ImportError`（`torch_npu`）或运行时属性错误。  
   - `prefix` 参数的默认值为 `""`，但部分旧代码未传递前缀，可能导致层名冲突或 `packed_modules_mapping` 找不到对应条目。  

2. **量化配置解析**  
   - `quant_model_description.json` 必须包含 `"quant_method": "modelslim"`，否则 `find_quant_modelslim_config` 返回 `None`，导致后续 `get_quant_config` 继续走旧路径，可能产生不一致的量化行为。  
   - `replace_prefix` 仍保留在旧实现中，如未同步更新可能产生冗余或错误的前缀映射。  

3. **性能回退**  
   - 在 NPU 上开启 `torch.npu.config.allow_internal_format` 可能导致显存碎片化，若后续模型切换至 CUDA，仍保留该配置会产生不必要的开销。  

4. **测试覆盖不足**  
   - 仅在 CI 中新增了 8‑NPU 基准，缺少跨平台（CPU、GPU）对 `modelslim` 代码路径的验证；潜在的 `AttributeError`（如 `layer.scheme` 未被设置）在非 NPU 环境未被捕获。

### [JIT Kernel] Reland NVFP4 kernels to JIT (#20012)
**SHA**: `f88acf8` | 🔗 [查看提交](https://github.com/sgl-project/sglang/commit/f88acf87805b1e27182c4927a47bc60818f768d8)

**🎯 变更类型**：功能增强 / 性能优化 / 重构  
**⚡ 重要程度**：🔴高  

**📋 变更摘要**  
- 将原本在 **sgl‑kernel** 中的 NVFP4 量化、FP4‑scaled GEMM 与 MoE 组块 GEMM 实现从 AOT（编译期）迁移到 **JIT（运行时即时编译）**，并在 Python 层提供统一的 `sglang.jit_kernel.nvfp4` 接口。  
- 新增 **benchmark** 脚本用于对 JIT、AOT（如可用）以及 Torch 基准的性能对比。  
- 重构 C++/CUDA 实现：改用 `sgl_kernel` 公共库（`tensor.h`、`runtime.cuh`、`utils.cuh`）统一硬件查询、错误检查；去掉了 `torch::` 相关宏与检查；统一使用 `RuntimeCheck`、`RuntimeDeviceCheck`。  
- 为 JIT 编译准备了 **CUTLASS**、**flashinfer**、**deep_gemm** 头文件的自动搜索逻辑，动态设置 `TVM_FFI_CUDA_ARCH_LIST`。  
- 在 SRT MoE、量化、路由等层面统一改用 JIT 实现的 `scaled_fp4_quant`、`cutlass_scaled_fp4_mm`、`cutlass_fp4_group_mm`。  
- 所有对外的 **sgl_kernel** CUDA API（如 `cutlass_fp4_group_mm`、`scaled_fp4_experts_quant` 等）在 Python 包中被移除/替换为 JIT 包装函数。  

**🎯 影响范围**  

| 模块 / 组件 | 受影响文件 / 目录 | 影响类型 |
|------------|------------------|----------|
| **sglang/jit_kernel** | `nvfp4.py`、`benchmark/` 目录、`csrc/gemm/nvfp4/*` | 新增 JIT 编译、API 封装 |
| **sglang/srt/layers/moe** | `cutlass_moe.py`、`cutlass_moe_params.py`、`moe_runner/flashinfer_trtllm.py`、`token_dispatcher/standard.py` | 使用 JIT NVFP4 接口替代旧 `sgl_kernel` |

**🔍 技术洞察**  

| 维度 | 分析 |
|------|------|
| **架构影响** | 1. **运行时编译**：从 AOT 静态编译转向使用 TVM‑FFI JIT 编译，编译过程在首次调用时完成并缓存 (`@cache_once`)；对部署环境有更高的灵活性（只要有 CUDA ≥12.8、SM≥10.0 即可）。<br>2. **统一硬件抽象**：C++ 代码改用 `sgl_kernel::runtime` 与 `host` 命名空间，消除对 `torch::` 的依赖，使 NVFP4 核心可在非‑PyTorch 环境中复用（例如 TVM、FlashInfer）。<br>3. **CUTLASS 版本统一**：所有 NVFP4 GEMM 使用 CUTLASS 3.x 统一接口，针对 SM100a/SM120a 进行特化，避免了在 `sgl_kernel` 中散落的宏定义。 |

| **可维护性** | - **代码复用**：所有 NVFP4 相关逻辑集中在 `sglang/jit_kernel/nvfp4.py` 与对应 C++ 包装文件，API 同步更新更容易。<br>- **接口统一**：通过 `register_custom_op` 注入自定义算子，保持与原 `torch.ops.sgl_kernel` 接口相同的调用签名，便于迁移。<br>- **去除宏**：原 `utils.h` 中的 `CHECK_…` 宏被 `RuntimeCheck` 替代，错误信息更明确，且不再依赖 PyTorch 的异常宏，提升跨项目兼容性。 |

### fix(rope): restore K writeback in fused rope + kv store kernel (#19636)

**⚡ 重要程度**：🟡 中  
**📋 变更摘要**：恢复在 fused‑rope + KV‑cache 内核中对 **K** 的写回。原实现只在 `load_q` 为 false 时直接把旋转后的 K 写入 `k_cache`，但在 `load_q` 为 true（即仅对 Q 做 RoPE）时遗漏了 K 的写回，导致 KV‑cache 内容不一致。代码改为：先把旋转结果写回原始 `input`（Q/K），随后在 `!load_q` 情况下再写入 `k_cache`（包括 X/Y 两段），并同步更新 `register_custom_op` 的 `mutates_args` 列表以声明 K 也会被原位修改。

- `python/sglang/jit_kernel/csrc/elementwise/rope.cuh`：核心 CUDA kernel。  
- `python/sglang/jit_kernel/rope.py`：自定义 OP 注册和文档。  
- 任何使用 `apply_rope_inplace_with_kvcache` 的模型推理路径（尤其是多‑GPU/多‑step KV‑cache 场景）。

**💡 关注建议**  
1. **功能确认**：在含 KV‑cache 的 decoder 流程（如 LLM 推理）上跑完整的回归测试，验证 K 在每一步都正确写回并可被后续注意力读取。  
2. **性能评估**：新增的两次 `store_as` 只在 `!load_q` 时触发，理论上影响不大，但建议对长序列（数千 token）进行基准对比，确保吞吐未受明显回退。  
3. **兼容性检查**：`mutates_args` 现在包括 `"k"`，确保上层调用方（如 TorchScript / JIT）对参数的 inplace 行为有相应的声明，避免潜在的副作用警告。  
4. **文档同步**：更新 README/使用手册中对 `apply_rope_inplace_with_kvcache` 行为的描述，避免用户误以为仅 Q 被修改。  

整体来看，此修复恢复了 KV‑cache 的完整性，对模型正确性至关重要，影响范围限于 RoPE‑KV‑cache 相关路径，风险主要在于新增写回路径的同步与性能，需要通过回归与基准测试加以验证。
