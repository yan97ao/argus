### feat: sync npu nightly test improvements from Ascend testcases (#29403)
**SHA**: `1b481de` | 🔗 [查看提交](https://github.com/sgl-project/sglang/commit/1b481deade53fd23ed787c4dbd5f389b1455a7b2)

**🎯 变更类型**：功能增强 / 性能优化 / 重构 / 架构变更  

**⚡ 重要程度**：🔴 高  

**📋 变更摘要**  
- 将 NPU（Ascend）每日 CI 流程与 Ascend 官方 testcase 同步：新增工作流输入、延长单节点测试超时、在 PR 流程中加入 `pr-single-node-tests`，并在 Nightly 流程中修正 1p/2p 性能测试名称。  
- 大幅改进 NPU 精度与性能测试代码：加入数据集波动容忍、自动重试、跨多次运行取平均、对 ConfigMap 实现严格写入/读取以实现多节点序列化控制；针对多个模型/场景添加/更新推理/工具调用解析器参数、环境变量和性能阈值。  
- 新增 `test_npu_qwen3_6_27b_w8a8_1p_in64k_out1k_50ms` 性能基准，用以评估大输入、单卡吞吐。  

**🎯 影响范围**  
- CI/CD 工作流：`.github/workflows/nightly-test-npu-e2e-single-node.yml`、`.github/workflows/nightly-test-npu.yml`、`.github/workflows/pr-test-npu.yml`。  
- 测试框架代码：`python/sglang/test/ascend/e2e/*`（accuracy、performance、multi_node utils）。  
- 所有基于 Ascend NPU 的模型注册测试文件（qwen3、glm、kimi、minimax、deepseek、qwen‑next 等），共计约 30+ 测例。  

**🔍 技术洞察**  

- **架构影响**  
  - **CI 流程**：通过新增 `install_sglang_deps`、`device_type_for_deps` 参数，CI 可在源码方式下自动装配 PyTorch、CANN 等依赖，进一步解耦镜像与源码两种运行模式。  
  - **多节点协同**：在 `test_npu_multi_node_utils.py` 中实现 `upsert_configmap_field_strict` 与 `wait_for_prefill_decode_exit`，利用 K8s ConfigMap 实现“当前执行的测试类”全局可见，防止多节点同时进行前置/解码导致资源竞争或误判。  
  - **测试框架**：`test_npu_accuracy_utils.py` 抽象出 `get_accuracy_threshold`、`get_max_retries`，将硬编码的容忍比例统一管理；加入 `run_accuracy_multiple` 支持多次运行取平均，提升结果的统计可靠性。  

- **性能影响**  
  - **更宽松的超时**：单节点 E2E 任务超时从 120 分钟提升至 300 分钟，防止长链路/大模型跑完被提前终止。  
  - **调参优化**：大量模型的 `--max-prefill-tokens`、`--max-mamba-cache-size`、`--mem-fraction-static`、`--prefill-delayer` 等参数被细致调优（如将 `max-mamba-cache-size` 从 120 调至 160，`prefill-delayer-max-delay-passes` 从 30 改 100），在实际 CI 中可显著提升吞吐与稳定性。  
  - **思考/工具调用 parser**：统一为多数模型加入 `--reasoning-parser` 与 `--tool-call-parser`，在推理阶段开启“思考”能力，可能导致额外的算子调用，但已在基准中相应调高 `output_token_throughput` 目标值，保持对性能的可接受预期。  
  - **多次运行取平均**：对高波动数据集（aime、gpqa）启用 3 次重试/取最大或取平均，可缓解单次噪声对精度阈值的影响。  

- **安全考虑**  
  - **ConfigMap 写入**：使用 K8s `patch_namespaced_config_map` 且在 404 时抛异常，确保不在不存在的 ConfigMap 上进行写入，避免潜在的权限提升或信息泄露。  
  - **缓存/代理地址**：CI 中对内部镜像源的替换（`sed` 替换 apt 源、`pip config set`）均指向内部受控服务，未暴露外部网络。  
  - **环境变量清理**：在特定推理场景（`generated-shared-prefix`）中主动删除 `SGLANG_IS_IN_CI` 环境变量，防止 CI 标识泄漏到模型服务进程。  

**⚠️ 潜在风险**  

| 风险点 | 说明 | 严重度 |
|--------|------|--------|
| **ConfigMap 竞争** | 多个节点同时尝试写入 `ACTIVE_TEST_CLASS` 可能出现“写覆盖”导致后续节点误判为已有执行。若节点 crash，ConfigMap 可能残留旧值，导致后续测试被阻塞。 | 中 |
| **依赖安装不一致** | `install_sglang_deps` 依赖内部缓存服务；若缓存服务不可达，CI 会因缺失 PyTorch/CANN 而失败。 | 中 |
| **新增 parser 参数兼容性** | 部分老模型/旧版本的二进制可能不支持 `--reasoning-parser`，导致启动失败。 | 低‑中 |
| **性能阈值过紧** | 对部分模型（如 qwen3‑32b、glm‑5.1）调高 `output_token_throughput`，若硬件资源波动或调度争用，CI 可能误报性能回归。 | 中 |
| **长时超时导致资源占用** | 将任务超时提升至 300 分钟，若出现无限循环或卡死，会占用 CI runner 资源更久。 | 低‑中 |
| **环境变量覆盖** | `pip config set` 直接修改全局 pip 配置，若后续步骤依赖官方 PyPI 可能受影响。 | 低 |

**💡 关注建议**  

1. **ConfigMap 生命周期管理**  
   - 在每个测试结束（`tearDownClass`）后显式清除 `ACTIVE_TEST_CLASS` 键，防止残留。  
   - 若出现异常退出，使用 `finally` 块或 GitHub Action `post` 步骤确保清理。  

2. **依赖缓存回退机制**  
   - 在安装脚本 `npu_ci_install_dependency.sh` 中加入对内部缓存不可达的 fallback（直接使用官方 PyPI），并在日志中标记。  

3. **参数兼容性检测**  
   - 在 CI 启动脚本中加入 `--help` 检查，若二进制不识别新 flag，则自动剔除相关参数，避免启动失败。  

4. **性能阈值可配置**  
   - 将 `output_token_throughput` 的基准值提取至 CI 输入或环境变量，方便不同硬件/集群自行调优，降低误报风险。  

5. **监控长时任务**  
   - 在工作流中加入定时 `ps`/`top` 监控脚本，若进程超出设定最长运行时间（如 250 分钟）自动 kill，以防资源泄漏。  

6. **文档同步**  
   - 更新 README / CI 文档，说明新增的 `install_sglang_deps`、`device_type_for_deps` 参数用途、使用场景以及默认值，以免使用者误配。  

7. **回归测试**  
   - 在一次完整的 Nightly CI 中，确保所有新增/修改的性能基准均通过（尤其是 `output_token_throughput` 调整后），并通过历史数据对比，确认阈值设置合理。  

8. **安全审计**  
   - 对 `scripts/ci/npu/npu_ci_install_dependency.sh` 进行一次依赖来源审计，确保所有外部二进制均来自可信渠道。  

总体来看，此次 PR 极大提升了 Ascend NPU 在 CI 中的可维护性、可靠性和覆盖度，尤其在多节点协同、精度容忍及性能基准方面的改进，为后续模型快速迭代提供了坚实的基础。只要注意上述风险点并做好清理/回退措施，即可安全上线。

### Remove retired DSA env paths (#29912)
**SHA**: `5eb1b6a` | 🔗 [查看提交](https://github.com/sgl-project/sglang/commit/5eb1b6a7ba8e67952039b927b8bfc3297c37ef20)

**🎯 变更类型**：重构 / 退役功能清理  
**⚡ 重要程度**：🔴高  
**📋 变更摘要**  
- 移除一批已废弃的 DSA（Data‑Sparse‑Attention）相关环境变量及其别名，实现文档、`environ.py`、后端实现的同步清理。  
- 删除旧的元数据拷贝/预计算验证模块，改为在 CUDA 环境下直接使用融合 kernel，Hip 环境保持兼容。  
- 通过代码简化，统一在 CUDA（非 Hip）下始终启用融合元数据生成与拷贝，去除用户通过环境变量手动关闭的能力。

**🎯 影响范围**  
- `python/sglang/srt/environ.py`（环境变量定义）  
- DSA 后端实现 `dsa_backend.py`、`dsa_backend_mtp_precompute.py`  
- 已删除的验证模块 `dsa_mtp_verification.py`、`nsa_mtp_verification.py`（旧的测试/调试代码）  
- 文档 `environment_variables.mdx`  
- 相关单元测试 `test/manual/test_dsa_alias_cli_registry_env.py`

**🔍 技术洞察**  

- **架构影响**  
  - 环境变量层面被大幅精简：不再通过 `SGLANG_USE_FUSED_METADATA_COPY`、`SGLANG_DSA_USE_FUSED_METADATA_GENERATION`、`SGLANG_DSA_ENABLE_MTP_PRECOMPUTE_METADATA` 等开关控制行为，代码路径统一为 “CUDA && !HIP” 时使用融合 kernel。  
  - 移除的验证模块是仅在开发/调试时使用的功能，对运行时体系结构没有影响，但删除后可减小包体积并避免误用。  
  - `init_forward_metadata_out_graph` 逻辑被简化：不再判断 `envs.SGLANG_DSA_ENABLE_MTP_PRECOMPUTE_METADATA`，而是始终调用一次预计算，再依据 `speculative_num_steps` 决定是否走多后端融合拷贝。  

- **性能影响**  
  - **正面**：去除对 `SGLANG_USE_FUSED_METADATA_COPY` 的分支检查，默认开启融合拷贝，省去大量单独 `copy_` 调用，提升 CUDA 图 replay 时的元数据拷贝速度（官方宣称 3× 加速）。  
  - **潜在负面**：在 AMD HIP 环境或极端显存/带宽受限的场景下，融合 kernel 可能不如逐条拷贝稳定，且之前可以通过环境变量关闭融合的灵活性被除去。  
  - 预计算元数据的开关被移除，始终启用 `precompute`，这会在一些使用 `MTP`（多步并行）但不需要预计算的工作负载上额外消耗显存/计算时间。  

- **安全考虑**  
  - 该改动不涉及网络、权限或文件系统交互，安全风险极低。唯一需关注的是 **向后兼容性**：如果用户在生产环境通过旧环境变量关闭融合以规避已知的 kernel bug，升级后将失去该控制手段，可能触发未预料的运行时错误。  

- **可维护性**  
  - 代码路径更统一，维护成本下降。  
  - 删除了 10+ 行文档和 5+ 行环境变量定义，减少了文档与实现不一致的风险。  
  - 移除的验证文件虽是调试工具，但其存在也可能导致误导性的 “未使用” 警告；现在彻底清理，整体代码基线更干净。  

**⚠️ 潜在风险**  
1. **向后兼容性破坏**：使用 `SGLANG_USE_FUSED_METADATA_COPY=0`、`SGLANG_DSA_ENABLE_MTP_PRECOMPUTE_METADATA=0`、`SGLANG_DSA_USE_FUSED_METADATA_GENERATION=0` 的用户会在升级后得到默认开启的行为，可能导致性能回退或显存超限。  
2. **AMD HIP/非 CUDA 环境**：在非 CUDA 环境（HIP）下仍走单独拷贝路径，但相关代码路径已被简化，若未来出现 `hip` 特有的融合 kernel 实现，当前代码不易扩展。  
3. **潜在的融合 kernel 回归**：融合拷贝和生成 kernel 已经是默认路径，一旦出现 bug（例如在特定张量形状或稀疏布局下），所有使用者都会受到影响，缺少快速关闭的手段。  

**💡 关注建议**  

- **兼容性提示**：在首次启动时检测上述已删环境变量是否被设置，并打印一次 **deprecation 警告**，告知用户将失效并提供临时回滚方案（如 `SGLANG_DSA_DISABLE_FUSED_METADATA=1` 之类的临时开关）。  
- **测试覆盖**：加强在 AMD HIP 与 CUDA 双平台的 CI，尤其验证在显存紧张的模型（DeepSeek V3.2、FlashInfer）下的融合 kernel 稳定性。  
- **监控显存使用**：发布说明中提醒使用 `MTP` 模式的用户关注显存占用，因为预计算元数据始终开启。  
- **文档更新**：在环境变量文档页面明确标记已删除的变量为 “已废弃且不再生效”，并说明默认行为。  
- **回退机制**：若社区报告融合 kernel 在某硬件/驱动组合下不可靠，考虑在后续版本中再引入一个 `SGLANG_FORCE_SINGLE_COPY` 环境变量，以便快速回退。  

---  

> **总体结论**：本次提交通过删除已废弃的 DSA 环境变量、简化元数据预计算与拷贝路径，显著提升了代码可维护性并默认使用高效的融合 kernel，属于正向的架构清理。唯一需要注意的是对依赖旧开关的用户提供明确的升级指引和回退方案，以防止因默认开启融合而产生的性能或显存异常。

### [AMD ]Feat/dsv4 ep tbo prefill (#29362)
**SHA**: `81735ec` | 🔗 [查看提交](https://github.com/sgl-project/sglang/commit/81735ecf8099aab0e38dbe04dff719ea19c43fb9)

**🎯 变更类型**：功能增强 / 性能优化 / Bug 修复  
**⚡ 重要程度**：🔴高  

**📋 变更摘要**  
- 为 DeepSeek‑V4（V2＋V4）实现 **Two‑Batch‑Overlap（TBO）** 预填充路径，支持在 **DP‑Attention + non‑EP MoE** 场景下跨 micro‑batch 重叠 DP `all_gatherv` / `reduce_scatterv` 与另一批次的 Attention+MoE 计算，实现约 7‑8% 的预填充加速。  
- 新增 **异步 gather/combine 流、持久化 CUDA 事件/缓冲区池**，解决之前的 **HSA_STATUS_ERROR_OUT_OF_RESOURCES**（资源泄漏、reserved‑memory OOM）问题。  
- 引入 `tbo_supports_cuda_graph` 标记并在 `TboBackend` 中屏蔽子后端的 CUDA‑graph 捕获，防止在图重放时重复构建压缩/索引器元数据导致 ROCm 资源耗尽。  
- 扩充 `ServerArgs` 校验逻辑，明确 **非 EP a2a**（`moe_a2a_backend='none'`）下仅在开启 `--enable-dp-attention` 时才允许 `--enable-two-batch-overlap`。  
- 添加针对 DeepSeek‑V4‑Flash‑FP8 与 DeepSeek‑V4‑Pro‑FP4 的端到端 **GSM8K** 精度回归测试，验证 TBO 不会影响模型正确性。  

**🎯 影响范围**  
- `sglang/srt/batch_overlap/operations_strategy.py`、`two_batch_overlap.py`（新增 DeepSeekV4 TBO 策略）  
- `sglang/srt/layers/attention/tbo_backend.py`、`deepseek_v4_backend_hip_radix.py`（CUDA‑graph 控制）  
- `sglang/srt/layers/dp_attention.py`（共享通信流、持久化事件/缓冲区实现）

**总体结论**：本次改动显著提升了 NUMA 绑定的鲁棒性，防止因 `numactl` 被 seccomp 或 cpuset 拒绝而导致工作进程直接死亡。实现思路清晰、单元覆盖充分，仅需关注参数解析安全、跨平台限制以及可能的启动时延即可平稳上线。

**SHA**: `cc7d7ba` | 🔗 [查看提交](https://github.com/sgl-project/sglang/commit/cc7d7ba3ddd8ec59824afb9a3c7a289d300d3b46)

**🎯 变更类型**：功能增强 / CI 稳定性提升  
**⚡ 重要程度**：🟡 中  
**📋 变更摘要**  
1. 将 Intel XPU nightly 测试的 per‑file 超时时间从 30 min/40 min提升至 120 min，防止大模型/长序列任务因单文件超时提前失败。  
2. 为 XPU CI 监控工作流 `xpu-ci-job-monitor.yml` 引入了多工作流快照、动态矩阵解析以及 Nightly 报表生成，使 PR 与 Nightly 的作业状态统一可视化。  
3. 在启动 XPU 容器的脚本中加大 `--shm-size` 至 8 GiB，缓解大模型训练/推理时的共享内存不足问题。

**🎯 影响范围**  
- CI/CD：`.github/workflows/nightly-test-intel.yml`、`.github/workflows/xpu-ci-job-monitor.yml`。  
- 容器启动：`scripts/ci/xpu/xpu_ci_start_container.sh`。  
- 监控脚本：`scripts/ci/utils/xpu_job_monitor.py`（间接受影响）。

**💡 关注建议**  

| 关注点 | 说明 | 建议 |
|--------|------|------|
| **超时阈值** | 将 per‑file 超时时间提升至 7200 s，可能导致因某些卡死的测试占用大量 CI 资源。 | 监控实际耗时分布，若多数文件仍在 20 min 左右，考虑在后续周期回退或引入自适应超时。 |
| **Nightly 报表** | 新增 `nightly-reports` job，依赖 `yq`、`jq` 解析 workflow 文件。 | 确保 CI 环境已预装对应二进制，或在 workflow 开头显式 `apt-get install -y yq jq`，防止因缺失导致 job 直接失败。 |
| **容器共享内存** | `--shm-size 8g` 可能在资源受限的自托管 runner 上导致启动失败。 | 在自托管 runner 文档中注明最小可用内存，或在脚本中加入检测并回退到默认大小。 |
| **工作流选择逻辑** | `select‑workflows` 步骤通过 `inputs.job_filter` 判断，仅在 PR 场景采集 `pr-test-xpu.yml`，其余情况采集两份 workflow。 | 若以后新增 XPU 相关 workflow，需要同步更新此逻辑，避免遗漏监控。 |
| **向后兼容** | 旧的 nightly CI 仍会使用新的超时设置，可能导致跑更久的作业占用后续调度窗口。 | 在 CI 调度策略（如并行数、资源配额）上评估是否需要相应调整。 |

总体来看，此次提交主要是提升 XPU CI 的可观测性与鲁棒性，改动局限于 CI 配置和容器启动脚本，对业务代码没有直接影响。建议在下一轮 CI 运行后检查 Nightly 报表的完整性以及容器启动的成功率，确保新超时和共享内存设置未引入新的瓶颈。

### [refactor] Reorder ServerArgs sections common-first; inline LLAMA4/MIMO_V2 arch tuples (#30151)

**⚡ 变更概览**  
- 将 `ServerArgs` 中的字段重新排序，**device‑info** 先于 **HTTP/SSL**，其后统一放置 **API、约束解码、内核后端、CUDA‑graph、通信、torch‑compile、LoRA、Multi‑modal** 等新模块。  
- 内联了 `MIMO_V2_MODEL_ARCHS / LLAMA4_MODEL_ARCHS` 常量，改为直接在后续代码中使用。  
- 新增大量 CLI 参数（如 `api_key、admin_api_key、attention_backend、cuda_graph_*、enable_layerwise_nvtx_marker、enable_lora、mm_process_config` 等），并对部分旧参数做了位置迁移。  

**🧩 受影响的核心模块**  
- `python/sglang/srt/server_args.py`（参数定义与解析）  
- 依赖 `MIMO_V2_MODEL_ARCHS`、`LLAMA4_MODEL_ARCHS` 的模型加载逻辑  
- 所有基于 `ServerArgs` 实例创建服务器/调度器的入口（`sglang/serve_*`、`sglang/cli` 等）  
- 文档、示例脚本以及 CI 测试中硬编码的默认参数顺序或常量名称  

**🔍 关键影响**  
1. **向后兼容**：删除的两个 arch 常量会导致 `import` 失效，使用它们的模块需改为直接引用对应字符串或新统一的模型列表。  
2. **新增参数**：默认值已填充，大多数场景不受影响，但若业务依赖 `--api-key`、`--enable_lora` 等新字段，需要在部署脚本或配置文件中显式设置。  
3. **参数顺序**变更不会影响 argparse 本身，但部分手写 `--help` 文档或自定义解析（如通过 `ServerArgs.__fields__` 遍历）可能出现排序差异。  
4. **CUDA‑graph 与 Torch‑compile** 选项的加入，会影响启动时的图捕获行为，使用者需根据硬件/驱动情况确认 `--disable_cuda_graph`、`--enable_torch_compile` 等开关的合理取值。  

**✅ 建议**  
- **审查并更新**：查找项目中 `MIMO_V2_MODEL_ARCHS`、`LLAMA4_MODEL_ARCHS` 的引用，改为新的硬编码或统一的模型列表。  
- **文档同步**：更新 README、CLI 手册以及示例脚本，加入新参数的说明（尤其是 API key、LoRA、CUDA‑graph、Multi‑modal）。  
- **回归测试**：重点跑包含 HTTP/HTTPS、LoRA、CUDA‑graph、Speculative decoding、Multimodal 的端到端脚本，验证默认参数下服务器能正常启动并响应。  
- **兼容层**：若担心老版本脚本仍会传入已删除常量，可在 `server_args.py` 中添加兼容别名（如 `MIMO_V2_MODEL_ARCHS = ("MiMoV2ForCausalLM", ...)`），降低升级阻力。  
- **监控与日志**：新加的 `enable_layerwise_nvtx_marker`、`enable_nccl_nvls` 等调优开关会产生额外的 NVTX/ NCCL 日志，建议在生产环境先保持默认关闭，调优时再打开。  

总体而言，此次重构提升了参数组织度和功能可扩展性，但需要同步代码、文档与部署配置，以确保平滑升级。
