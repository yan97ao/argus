### [mem_cache][5/N] refactor: extract host KV cache base layer into pool_host package (#27273)
**SHA**: `ff1fc1f` | 🔗 [查看提交](https://github.com/sgl-project/sglang/commit/ff1fc1fbdff315fe44b9431ca5aae00d7bd7f733)

**🎯 变更类型**：重构 / 架构变更  
**⚡ 重要程度**：🔴 高  
**📋 变更摘要**：本次提交将原先散落在 `sglang.srt.mem_cache.memory_pool_host` 中的 Host KV‑Cache 基础层抽取并重新组织为独立的 `pool_host` 包，形成 `base、common、hisparse` 三个子模块，并相应更新了所有使用该层的导入路径、测试以及部分存储实现。目标在于提升代码可维护性、降低耦合度，同时为后续功能扩展（如不同后端的 Host 缓存实现）提供更清晰的模块边界。  

**🎯 影响范围**：  
- `sglang/srt/mem_cache/` 相关子目录：`memory_pool_host`（被废弃） → `pool_host`（新包）  
- 所有基于 Host KV‑Cache 的实现：`MHATokenToKVPoolHost、MLATokenToKVPoolHost、DSATokenToKVPoolHost` 等  
- 多个存储后端实现（AIBrix、EIC、HF3FS、Mooncake、NixL、SIMM 等）  
- 测试文件涉及 `memory_pool_host` 的导入路径更改  
- 可能影响外部用户直接 `import sglang.srt.mem_cache.memory_pool_host` 的代码  

---

### 🔍 技术洞察  

| 维度 | 影响分析 |
|------|----------|
| **架构影响** | - **模块化提升**：`pool_host` 把 Host KV‑Cache 抽象、公共工具、HiSparse 实现分别放在 `base、common、hisparse` 中，职责更明确。<br>- **解耦**：上层 `mem_cache` 与具体实现的耦合度降低，后续可以在 `pool_host` 里平行实现多种 Host 缓存策略（如基于文件、GPU‑direct 等），而无需改动上层业务代码。<br>- **导入路径统一**：所有内部引用统一走 `sglang.srt.mem_cache.pool_host`，便于 future 包重构或迁移。 |
| **性能影响** | - **运行时**：代码逻辑基本保持不变，仅改变了 import 路径，理论上 **无额外运行时开销**。<br>- **初始化**：`HostKVCache.__init__` 中的内存大小检查与分配逻辑被迁入 `base`，行为保持一致。<br>- **潜在轻微影响**：如果用户在运行时仍使用旧路径 `memory_pool_host`（未同步），会触发 `ImportError`，导致程序提前退出，而不是继续执行导致性能异常。 |
| **安全考虑** | - 本次改动不涉及密码、网络、权限等安全敏感功能。<br>- 唯一值得关注的是 **引入新模块的 import 机制**：若在 `pool_host/__init__.py` 中未正确限定 `__all__`，可能意外暴露内部实现细节；当前已显式导出 `HostKVCache、HostTensorAllocator`，风险可控。 |
| **可维护性** | - **代码可读性提升**：抽象层与实现分离，后续在 `pool_host` 中添加新实现（如基于 mmap、nvme、RDMA）时不需要在 `memory_pool` 中混杂。<br>- **测试覆盖**：相应更新了所有单元测试和注册测试的导入路径，确保迁移后的行为得到验证。 |
| **兼容性** | - 对外 **API 迁移**：原始路径 `sglang.srt.mem_cache.memory_pool_host` 将失效，若有第三方项目直接依赖该路径，需要提供向后兼容的别名或文档迁移指南。<br>- **内部依赖**：已在项目内部统一修改，未出现循环依赖。 |

---

### ⚠️ 潜在风险  

1. **旧路径破坏兼容性**  
   - 外部用户仍使用 `sglang.srt.mem_cache.memory_pool_host` 将立即报 `ImportError`。  
2. **导入循环或遗漏**  
   - `pool_host/__init__.py` 只导出 `HostKVCache` 与 `HostTensorAllocator`；如果其他子模块（如 `hisparse`）在外部直接导入 `pool_host` 而未通过 `__all__` 暴露，可能导致运行时 `AttributeError`。  
3. **打包/发布遗漏**  
   - `setup.cfg` 或 `MANIFEST.in` 需要同步添加 `sglang/srt/mem_cache/pool_host/**`，否则发布的 wheel 可能缺失新文件。  
4. **测试遗漏**  
   - 虽然已更新多数测试，但仍可能存在隐藏的动态 import（如通过字符串拼接）未覆盖，导致在特定配置下加载旧模块失败。

### [router] Tokenize prompt once at ingress; forward input_ids to the engine (all policies) (#28744)
**SHA**: `364bf97` | 🔗 [查看提交](https://github.com/sgl-project/sglang/commit/364bf976be6f2fce2e291c0721fca492a4e91ca0)

**🎯 变更类型**：功能增强 / 性能优化 / 架构变更 / 安全修复  
**⚡ 重要程度**：🔴高  

**📋 变更摘要**  
- 在请求入口（Ingress）一次性完成 Prompt/Chat‑messages 的分词，并把得到的 `input_ids` 通过 `SelectionContext` 传递至所有路由策略。  
- Cache‑aware、Round‑Robin、Sticky 等策略均可直接使用预先生成的 token 序列，避免在路由阶段再次分词。  
- 当分词得到的 token 与模型的 Chat‑encoder 完全等价（`engine_equivalent = true`）时，自动在转发给后端推理引擎的 HTTP 请求体中注入 `input_ids`，让引擎跳过自身的 tokenization。  
- 新增 `RequestTokens`、`request_tokens_for`、`tokenize_text`、`extract_prompt_text_from_value` 等通用实现，统一抽象分词逻辑。  
- 为未能成功产生 engine‑equivalent token 的情况新增 Prometheus 计数器 `sgl_router_ingress_tokenize_errors_total` 并在 `MetricsRegistry` 中实现。  
- 通过大量单元/集成测试验证：① 正常聊天请求成功转发 `input_ids`，② 带工具、multimodal、thinking 等字段的请求安全地 **不** 转发，③ 各策略在有/无 token 场景下仍保持原有路由行为。  

---

## 🔍 技术洞察  

### 架构影响
| 维度 | 影响描述 |
|------|----------|
| **请求流‑分层** | 原来分词逻辑散布在 `CacheAwareZmqPolicy::tokens_for_request` 中，仅在 cache‑aware 策略使用；现在分离为 **Ingress**（`chat_completions` handler）统一完成，并通过 `SelectionContext::with_request_tokens` 向下传递。策略层不再关心分词，只在需要 token（如 cache‑aware 前缀匹配）时读取即可。 |
| **组件解耦** | `request_tokens_for` 是 **模型‑层**（TokenizerRegistry）与 **路由层**（Policy）之间的公共服务，所有策略共享同一套分词实现，显著降低代码重复和维护成本。 |
| **新增数据结构** | `RequestTokens { ids, engine_equivalent }` 把分词结果和可否直接转发的语义绑定，避免策略自行判断模型是否有 chat encoder。 |
| **Metrics** | 新增错误计数 `sgl_router_ingress_tokenize_errors_total{model_id}`，帮助运维快速定位 “tokenization offload 失效” 场景。 |
| **扩展性** | 通过 `Policy::needs_request_tokens()`（默认 `false`）为未来可能需要原始 prompt token（例如自定义缓存策略）提供开关。现有策略仅在 `CacheAwareZmqPolicy` 中返回 `true`，保持向后兼容。 |
| **PD‑Disagg 注入** | `build_outgoing_body` 合并了 `input_ids` 与 bootstrap 字段的注入，统一一次序列化，避免多次 `serde_json` 编/解码，提升 PD 场景下的吞吐。 |

### 性能影响
| 场景 | 关键改动 | 预期收益 | 潜在开销 |
|------|----------|----------|----------|
| **普通 Chat（有 chat encoder）** | 入口一次分词 → `input_ids` 注入 | 1️⃣ 减少后端推理服务的重复分词（约 30‑50 µs / 请求），降低 CPU 消耗；2️⃣ 路由阶段不再进行分词，变为 O(1) 读取 token slice，提升路由延迟。 | 解析一次 `serde_json::Value`（仅在需要 token 时）以及一次分词；相较之前的两次分词，整体仍净收益。 |
| **Load‑only 策略（RoundRobin/Sticky）** | 仍然在入口做一次分词（仅当模型拥有 chat encoder） | 对这些策略同样获得 `input_ids` 转发的 offload 效果，后端化简。 | 对于不需要 token 的请求（如无 chat encoder、或仅 `prompt`），通过 `has_chat_encoder` 检查后 **不** 解析 JSON，保持原有低开销。 |
| **Cache‑Aware 前缀匹配** | 直接使用入口生成的 `tokens`，无需再次调用 `TokenizerAdapter::encode` | 省去一次 `encode`（尤其是大模型的多 token 编码），加速前缀哈希计算。 | 若入口解析失败（JSON 错误或 tokenizer 失效），会立即回退到最小负载分配，避免因解析错误导致路由卡顿。 |
| **PD‑Disagg** | 合并 `input_ids` 与 bootstrap 字段一次序列化 | 减少一次 `serde_json::to_vec`，降低内存拷贝，提升大并发时的网络准备阶段吞吐。 | 若两者均需要注入，仍需要一次 `serde_json::from_slice`（已在入口完成），整体仍优于之前两次解析/序列化。 |
| **内存** | 为每个请求在 `SelectionContext` 中持有 `Vec<u32>`（token 序列） | 典型聊天请求 token 数 < 4K → < 16 KB，短暂生命周期，影响可接受。 | 极端长文本（> 1 MiB）可能导致较大内存占用；建议可在入口对 token 长度做上限裁剪或返回错误。 |

### 安全考虑
| 风险点 | mitigations |
|--------|---------------|
| **错误的 `input_ids` 注入**（模型未使用等价的 chat encoder） | `request_tokens_for` 仅在模型声明拥有 chat encoder 并成功 `encode_chat` 时标记 `engine_equivalent = true`；否则即使得到 token 也会被标记 `false`，不会注入。 |
| **工具 / 多模态 / thinking 模式导致 token 与真实 Prompt 不匹配** | `input_ids_safe_to_forward` 在转发前检查 `tools`、`functions`、`multimodal content`、`chat_template`、`chat_template_kwargs`、`reasoning*`、`task`、`continue_final_message`、以及最后一条 `assistant` 消息等字段，若出现任意不被 encoder 复现的信号则 **不** 注入。 |
| **异常分词导致隐式回退** | 当 `tokenize_text` 报错会记录 **WARN** 并返回 `None`。随后 `ingress_tokenize_offload_failed` 触发计数器 `sgl_router_ingress_tokenize_errors_total`，提供可观测的告警入口。 |
| **DoS via超大请求体** | 仍受制于 HTTP 服务器的 body 大小限制（默认为 1 MiB），并在 `parse_probe` 之前做基本 JSON 结构校验；如有需求可在入口对 token 数量进行硬性上限。 |
| **Metric 泄露/泛洪** | 错误计数采用 **WARN** 级别日志一次 per‑request，并在 `MetricsRegistry` 中使用 `AtomicU64` 按 `model_id` 聚合，避免瞬时高频写入导致锁竞争。 |
| **生命周期泄露** | `SelectionContext` 持有对 `request_tokens` slice的引用，生命周期绑定到原始请求
