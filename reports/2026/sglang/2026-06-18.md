### fix: speculative draft worker clobbering target attention backend (#28559)
**SHA**: `fa71064` | 🔗 [查看提交](https://github.com/sgl-project/sglang/commit/fa71064147a16b3710755a520f1257323c99b459)

**🎯 变更类型**：Bug修复  
**⚡ 重要程度**：🔴高  
**📋 变更摘要**：本次提交解决了在 Speculative Draft Worker（推测草稿工作者）中因共享同一 `AttentionBackend` 实例导致的注意力后端被错误覆盖的问题。通过在 `AttentionBackend` 中新增 per‑mode（prefill / decode）后端标识字段，并在 `ModelRunner` 与模型实现（DeepSeek‑V2）中使用这些字段来决策实际使用的后端，避免了全局状态的相互污染。

**🎯 影响范围**：  
- `python/sglang/srt/layers/attention/base_attn_backend.py`（新增属性）  
- `python/sglang/srt/model_executor/model_runner.py`（记录并传播 per‑mode 后端标识）  
- `python/sglang/srt/models/deepseek_v2.py`（后端选择逻辑改为读取实例属性或全局默认）  

**🔍 技术洞察**  
- **架构影响**：  
  - 将原先依赖全局 `server_args` 的后端选择机制，迁移到 **后端实例本身** 持有的模式化标识，提升了 **模块化和职责分离**。  
  - `ModelRunner.init_attention_backend` 现在负责把 `prefill` / `decode` 后端字符串写入 `AttentionBackend` 实例，模型层仅通过 `get_attn_backend()` 读取，降低了跨层耦合。  
  - 对于 **Speculative Draft** 场景，所有草稿模式共用同一后端（`draft_attn_backend`），不再在不同 forward‑mode（prefill/ decode）之间产生冲突。  

- **性能影响**：  
  - 代码改动仅涉及几次字符串读取与属性赋值，时间复杂度不变， **运行时性能基本不受影响**。  
  - 消除了因错误后端切换导致的潜在重复图构建或 CUDA Graph 重建，间接提升推理吞吐量与延迟稳定性。  

- **安全考虑**：  
  - 未引入外部输入或权限检查，**安全风险为零**。  
  - 通过降低全局状态写入频度，间接降低了因错误配置导致的服务异常风险。  

**⚠️ 潜在风险**  
1. **未覆盖的模型实现**：目前只有 DeepSeek‑V2 更新了后端选择逻辑，若项目中还有其他模型仍使用原先的全局 `server_args`，在 Draft Worker 场景下仍可能出现后端被覆盖的情况。  
2. **属性初始化假设**：`prefill_attention_backend_str` 与 `decode_attention_backend_str` 默认 `None`，如果在某些路径（如自定义后端创建）未走 `ModelRunner.init_attention_backend`，则可能出现 `None` 回退到全局默认，行为仍然正确但依赖全局配置。  
3. **并发共享实例**：如果同一个 `AttentionBackend` 实例在多个进程/线程间被共享（极少见），属性写入可能产生竞争。当前设计假设每个 Runner 持有独立实例。  

**💡 关注建议**  
- **回归测试**：在包含 Speculative Draft Worker 的全链路基准测试中，验证 prefill 与 decode 两阶段的注意力后端始终符合预期，尤其在多模型并发加载时。  
- **模型适配**：审查项目中其他模型（如 Llama、ChatGLM 等）是否仍依赖全局 `server_args` 取后端，统一改为使用 `backend.prefill_attention_backend_str / decode_attention_backend_str`。  
- **文档更新**：在服务器配置文档中说明 `prefill_attention_backend`、`decode_attention_backend` 仍是全局默认，仅在未显式在 `ModelRunner` 中覆盖时生效。  
- **代码审查提醒**：后续新增模型或自定义 attention backend 时，务必在对应的 `dispatch_attn_forward_method` 中遵循 “后端实例属性 → 全局默认” 的查找顺序，以防再次出现状态污染。  

通过上述改动，系统在推测草稿场景下的注意力后端管理更加可靠，提升了代码可维护性与运行时的稳定性。祝调试顺利！

### [NPU] Add Ascend NPU support for DeepSeek-V4 (#25144)
**SHA**: `9b10821` | 🔗 [查看提交](https://github.com/sgl-project/sglang/commit/9b10821c8e6eb9f05c357641d28276a2387fa148)

**🎯 变更类型**：功能增强 / 架构变更 / 性能优化  
**⚡ 重要程度**：🔴 高  
**📋 变更摘要**：本次提交为 DeepSeek‑V4 在华为 Ascend NPU（“Ascend”）上提供完整的推理支持。核心改动包括：

1. **新增 Ascend‑NPU 注意力后端（`DeepseekV4AscendAttnBackend`）**，实现 V4‑Flash‑Attention、压缩（compressor）与索引（indexer）在 NPU 上的 fused kernel 调用。  
2. **实现 NPU‑专用的 KV/State 分配器、内存池及 per‑req 表**（`DSV4NPUTokenToKVPoolAllocator`、`DSV4NPUReqToTokenPool`、`dsv4_memory_pool`、`dsv4_allocator`），从环形缓冲区改为 **paged（分页）** 方案以满足 NPU kernel 对 `cache_mode=1` 的要求。  
3. **在调度、图执行、内存回收等路径注入 DSV4‑NPU 钩子**（`maybe_write_dsv4_extend`、`maybe_write_dsv4_decode`、`maybe_evict_dsv4_state` 等），保证分页状态池的正确写入、回收。  
4. **在模型实现层面加入 NPU‑专用的 RoPE、RMSNorm、MHC（Hybrid‑Mixture‑of‑Heads）实现**，使用 `torch.ops.custom.*` 与 `torch_npu` 原语，避免 CUDA/Triton 依赖。  
5. **对其它组件做兼容性适配**：attention‑backend 注册、graph‑runner 判定、MoE‑topk、量化、模型加载等，以在不同平台（CUDA、HIP、NPU）间统一代码路径。  

**🎯 影响范围**  

| 模块/组件 | 影响内容 |
|---|---|
| `sglang/srt/hardware_backend/npu/attention/` | 新增 Ascend V4 注意力后端及图缓存初始化 |
| `sglang/srt/hardware_backend/npu/dsv4/` | 全套 allocator、memory_pool、req‑to‑token、hook 实现 |
| `sglang/srt/layers/` | RoPE、Compressor、MHC、MoE‑topk、Linear、Norm 的 NPU 分支 |
| `sglang/srt/model_executor/` | KV‑pool 初始化、调度、图重放、Runner 适配 |
| `sglang/srt/managers/schedule_batch.py` | 触发 DSV4‑NPU 状态池驱逐 |
| `sglang/srt/utils/` | NPU‑特有的可选 import 与环境判断 |
| 其它（speculative、moe、quantization、model loading） | 小幅改动以兼容 NPU 环境 |

## 🔍 技术洞察

### 架构影响
| 维度 | 说明 |
|---|---|
| **后端抽象层** | 引入 `DeepseekV4AscendAttnBackend`，继承 `AscendAttnBackend` 并混入 `C4IndexerAscendBackendMixin`、`CompressorAscendBackendMixin`。后端注册 (`attention_registry.py`) 增加 NPU 判定，使模型在 `attention_backend="dsv4"` 时自动走 Ascend 实现。 |
| **KV/State 分配** | 原 CUDA 实现使用 **环形缓冲区 + `translate_kv_loc_to_compress_state_loc`**（基于 ring‑hash）。NPU 不支持 ring‑hash（`cache_mode=2` 被 Atlas A3 拒绝），改为 **分页（paged）** 分配器 `NPUPagedTokenToKVPoolAllocator`，并在 `DSV4NPUTokenToKVPoolAllocator` 中统一返回 **`DSV4OutCacheLoc`** 包含 full、SWA、c4、c128 以及各自的 state slot。 |
| **图捕获 / 重放** | `AscendAttnBackend._init_dsv4_graph_buffers` 在捕获阶段预分配 `c4_page_table`、`c128_page_table`、`c4_state_page_table` 等元数据；在 replay 时通过 `_apply_dsv4_graph_metadata` 把实时 seq‑len 填入。 |
| **Per‑request 表** | 新增 `DSV4NPUReqToTokenPool`，在每次 `alloc_extend/alloc_decode` 后由 `maybe_write_dsv4_*` 将 allocator 返回的 slot 写入对应的 per‑req 表（SWA、c4、c128、state）。这与原 `ReqToTokenPool` 完全兼容，仅在 NPU + DSV4 场景实例化。 |
| **状态池驱逐** | `maybe_evict_dsv4_state` 与 `maybe_evict_dsv4_state_on_swa` 在每个 decode 步骤和 SWA eviction 中主动释放压缩状态页，防止小页面池提前耗尽。 |
| **算子划分** | 通过 `torch.ops.custom.compressor`（fused compress）与 `torch.ops.custom.inplace_partial_rotary_mul`（RoPE）提供 NPU 原子算子；对未实现 fused 路径的 fallback 保持原有 PyTorch 实现（如 `forward_native`）。 |
| **模型层次** | `DeepSeekV4ForCausalLM` 在加载权重时检测 `compress_ratio`，对 NPU 自动使用 `DSV4NPUTokenToKVPool`（分页）与 `DeepseekV4AscendAttnBackend`。 |

### 性能影响
| 方面 | 预计效果 |

### 2. 性能维度  
- **运行时开销**：每次访问 `get_parallel().tp_size` 等会走一次属性查找、一次 dict 检查（若无覆盖则调用底层 getter）。这比直接调用底层函数多一次 Python 属性调度和一次 dict 查找，**开销在 ns 量级**（典型函数调用 30‑50 ns，额外 5‑10 ns）。对推理路径（每个 token）影响可以忽略不计。

- **缓存/跨进程**：`ParallelContext` 是 **进程单例**（模块级对象），不涉及跨进程共享，和原始分布式全局状态保持一致。  

### 3. 可维护性  
- **统一入口**：新增 `runtime_context.get_parallel()` 成为所有新代码的并行信息获取标准，后续若需新增并行维度（如 **expert‑parallel‑group**）只在此处添加属性即可，无需在每个文件散布新 getter。

  - `override()` 只在 `runtime_context` 模块内部实现，外部调用需显式 `with …:`，使用后会自动恢复原值。

**SHA**: `4b817f5` | 🔗 [查看提交](https://github.com/sgl-project/sglang/commit/4b817f5d7f6602fcfafe18d0333f0f108f25c4a7)

本次提交在 Flash‑Attention 相关代码中引入了 `only_qv` 开关（用于在 rope‑dim 为 0 时跳过 K 的乘法），并在底层 kernel、Python 包装层以及 CMake 配置中同步了该标志。与此同时，新增了 `sparse_mask_fine` 参数占位，默认关闭 FA3 的自动开启逻辑，改为在 CUDA ≥ 12.4 且非 ARM 架构时通过 `SGL_KERNEL_ENABLE_FA3` 选项显式控制。

**核心影响模块**  

| 模块 | 变更点 | 影响 |
|------|--------|------|
| `python/sglang/jit_kernel/flash_attention*.py` | 新增 `only_qv` 参数；对 `k_cache`、`q` 为 `None` 的情况做容错并生成占位张量 | 兼容仅使用 QV（rope dim = 0）的模型路径，避免原先的 stride 断言报错 |

| `sgl-kernel/python/sgl_kernel/flash_attn.py` | 参数透传、占位张量创建、软阈值计算路径调整 | Python API 向后兼容（默认 `only_qv=False`），但需要确保占位张量的 shape 与后端期望一致 |

**潜在风险**  

1. **占位张量大小**：当 `k_cache` 为 `None` 时，代码使用 `k_head_size = q.shape[-1]`（或 `64` 作为兜底）来创建空 `k` 张量。若后端 kernel 实际读取 `k` 的数值（即使被标记为 “skip”），错误的 shape 可能导致非法内存访问。建议在文档中明确 **仅在 rope‑dim = 0 且后端实现已确认不读取 K** 时使用，或在 kernel 中加入 `if (only_qv) return;` 的安全检查。

2. **`only_qv` 与 `qv` 的耦合**：当前实现要求 `qv` 必须提供，否则会报错。若上层调用者忘记传 `qv`（例如在多模态模型中仅使用 Q），会触发异常。建议在 Python 层加入更友好的错误信息或自动从 `q`、`k` 推断。

3. **兼容性回归**：`sparse_mask_fine` 参数在 C++ 接口中加入，但 Python 调用默认传 `None`。若后续实现了稀疏掩码功能而忘记在 Python 层传递实际张量，可能导致性能下降或功能缺失。文档和 CI 用例需同步更新。

4. **CMake 默认行为**：`DEFAULT_SGL_KERNEL_ENABLE_FA3` 在 ARM 上被强制 OFF，仍然保留 `option(SGL_KERNEL_ENABLE_FA3 ...)`，但后续手动打开仍会生效。若用户在 ARM 上显式 `-DSGL_KERNEL_ENABLE_FA3=ON`，会尝试编译不支持的代码，导致构建失败。可在 `CMakeLists.txt` 中加入警告或限制。

**建议**  

- 为 `only_qv` 分支在底层 kernel 添加断言或 `if (!only_qv) { … }` 的早退出，防止意外读取 `k`。  
- 在 Python 包装层对 `k_cache`、`q` 为 `None` 的情况加入 `warnings.warn`，提醒使用者该路径为 “synthetic”。  
- 更新 README / API 文档，说明 `only_qv` 的使用前提（rope‑dim = 0、必须提供 `qv`），并给出示例。  
- 在 CI 中加入针对 `only_qv=True` 的单元测试，包括不同 batch/seq 长度、不同 dtype（bf16/fp8）以及 ARM 架构交叉编译的情况。  
- 对 `sparse_mask_fine` 参数在后端实现前保持 `optional`，并在 C++ 注释中注明目前未使用，防止误解。  

总体而言，此次改动为后续 “FA3‑only‑QV” 优化奠定了接口基础，但需在实现层面保证安全退出并提供完整的测试与文档，以免在生产环境触发隐藏的 shape/类型错误。

     - 自定义 `model_specific_data` 键不在 `_PATCH_ALIGNED_KEYS` 中的场景，验证仍能正确传递。  
   - 回归测试对比前后特征输出（相同随机种子），确保功能等价且数值误差在可接受范围。

5. **文档 & 示例**：  
   - 在 `mm_utils` 与 `step3_vl` 的 docstring 中补充 “patch‑aligned” 与 “item‑aligned” 的概念说明，帮助用户在自行扩展模型时快速定位需要修改的键。  
   - 示例代码中展示如何为自定义模型添加自己的 patch‑aligned 键（如 `patch_masks`）以及相应的 `_PATCH_ALIGNED_KEYS` 扩展方式。

**结论**  
本次修改修正了 Step3‑VL 多图‑多 patch 场景下的特征拆分错误，并通过批量前向提升了效率，影响主要集中在图像多模态路径。建议在升级后对自定义模型的 `model_specific_data` 结构进行一次检查，并加入相应的回归测试，以防止因键遗漏导致的潜在隐式错误。

**SHA**: `873196f` | 🔗 [查看提交](https://github.com/sgl-project/sglang/commit/873196f7fabe5965320f51bb7a7af21b24b5e122)

- 相关单元测试与 CI（需在非 NPU 环境下仍能成功导入）  

**💡 关注建议**  
1. **运行时依赖**：确保在使用 NPU 时已正确加载 `torch_npu`（通常由运行时自动完成），否则 `torch.ops.npu.*` 会报错。  
2. **平台兼容**：因为模块不再在导入时硬绑定 `torch_npu`，在 CUDA/CPU CI 上仍能 import，建议加入 `import torch_npu` 的 guard，仅在实际跑 NPU 时显式 import，以防意外缺失。  
3. **weight 布局**：两条分支均生成 `weight [in, out]` 以及 `weight_scale_inv [in//64, out, 2]` 的转置视图，切勿调用 `.contiguous()`，否则会破坏 stride，导致 HBM 带宽大幅下降。可在注释中再次强调。  
4. **测试覆盖**：增加离线（ModelSlim 导出的 float8_e4m3fn）和在线（FP16/BF16）两种路径的单元测试，验证 `layer.weight_scale` 被正确删除、`bias_fp32` 是否缓存。  
5. **文档与使用提示**：在量化文档中说明 `--quantization mxfp8` 与 ModelSlim‑MXFP8 两种入口的区别，以及在 CPU 环境下该功能会回退为普通 FP16/BF16。  

总体而言，此次改动将 MXFP8 的核心算子统一到 NPU kernel，提升了代码复用性与运行时灵活性，只要在实际部署前验证 NPU 环境可用、并保持转置视图不被 `contiguous` 打断，即可安全使用。
