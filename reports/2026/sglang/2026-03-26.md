### [MLX] Add native MLX execution backend for Apple Silicon Mac (#20342)
**SHA**: `a305964` | 🔗 [查看提交](https://github.com/sgl-project/sglang/commit/a3059641593b985387a6abbb70c19c4ddc53cf1e)

**🎯 变更类型**：功能增强 / 架构变更 / 重构  
**⚡ 重要程度**：🔴 高  

**📋 变更摘要**  
- 为 Apple Silicon（M1/M2/M3）加入原生 **MLX** 推理后端，实现全模型在 MLX 中运行，避免了 PyTorch‑MPS 与 MLX 之间的张量拷贝。  
- 新增 `tensor_bridge` 工具，统一提供 **torch ↔ mlx** 零拷贝转换、设备选择、同步等 API，并通过环境变量 `SGLANG_USE_MLX=1` 实现可选开启。  
- 引入 `MlxModelRunner`（完整 MLX 推理实现）与 `MlxModelRunnerStub`（仅创建调度所需的轻量结构），以及对应的 `MlxTpModelWorker`，在调度器中根据 `use_mlx()` 动态切换。  
- 更新 benchmark、diffusion 代码以及依赖列表，使得在 MLX 模式下仍能运行原有基准与测试脚本。  

**🎯 影响范围**  
- **核心运行时**：`sglang/srt/model_executor/*`、`sglang/srt/managers/*`、`sglang/srt/hardware_backend/mlx/*`。  
- **Benchmark 与测试**：`bench_one_batch.py`、`diffusion` 相关 Triton/MPS 回退实现。  
- **依赖管理**：`pyproject_other.toml` 新增 `scikit-image`、`trimesh`、`xatlas`（与 MLX 示例模型相关）。  
- **配置入口**：`sglang/srt/utils/tensor_bridge.py`、`sglang/srt/server_args.py`（MPS 后端默认关闭重叠调度）。

| 维度 | 影响分析 |
|------|----------|
| **架构影响** | - **硬件抽象层**：新增 `sglang.srt.hardware_backend.mlx` 包，将 MLX 完全抽离为独立后端。<br>- **调度层**：`Scheduler.init_tp_model_worker` 通过 `use_mlx()` 动态实例化 `MlxTpModelWorker`，保持原有 `TpModelWorker` 不变，实现向后兼容。<br>- **模型加载**：`MlxModelRunner` 直接使用 `mlx_lm.load`，跳过 PyTorch 权重加载；`MlxModelRunnerStub` 只创建 `ReqToTokenPool` 与 `TokenToKVPoolAllocator`（使用零内存 `_DummyKVCache`），保证调度器仍能正常分配 KV 缓存但实际缓存由 MLX 管理。<br>- **张量桥接层**：`tensor_bridge` 统一完备 `torch ↔ mlx` 零拷贝实现，内部考虑 MPS 4 GB 限制，防止因大张量导致的 `MPSTemporaryNDArray` 分配失败。 |

### fix(sgl-kernel): align wheel METADATA/WHEEL with +cu filename (#21437)
**SHA**: `662635e` | 🔗 [查看提交](https://github.com/sgl-project/sglang/commit/662635e7a7674b7f83d1a64573c9378455d033e2)

**🎯 变更类型**：Bug修复/功能增强  
**⚡ 重要程度**：🔴 高  
**📋 变更摘要**：  
- 通过 `rename_wheels.sh` 脚本在构建完成后统一修改 `.whl` 文件名、`METADATA` 中的 `Version` 字段以及 `WHEEL` 中的平台标签，使文件名后缀（+cu124/+cu128/+cu130）与内部元数据保持一致，解决了 Pip 在仅文件名变更时出现的 “inconsistent version” 警告。  
- 新脚本在检测系统 CUDA 版本后，解包 wheel、更新元数据再重新打包，避免手工编辑 `RECORD`，提升了自动化可靠性。

**🎯 影响范围**：  
- `sgl-kernel` 包构建与发布流水线  
- `dist/` 目录下的所有生成的 wheel 包  
- CI/CD 自动化脚本（尤其是涉及 `pip install` 验证的阶段）  
- 依赖于该 wheel 的下游项目（如 `sglang` Python 包）  

**🔍 技术洞察**：  
- **架构影响**：  
  - 只涉及打包/发布层面，不改动运行时代码或库的内部实现，对整体系统架构影响极小。  
  - 引入了对 CUDA 环境的检测 logic（基于 `/usr/local/` 目录），使构建过程与目标 CUDA 版本更紧耦合，提升了发布的一致性。  
- **性能影响**：  
  - 增加了解包、修改、重新打包的步骤，构建时间会略增（通常在数秒到十几秒之间），对 CI 运行时长有轻微影响。  
  - 由于仅在构建阶段执行，对运行时性能无影响。  
- **安全考虑**：  
  - 脚本在解包后直接修改 `METADATA` 与 `WHEEL`，若出现异常（如未正确清理临时目录），可能残留未签名的 wheel，导致下游在进行 wheel 签名或校验时出现不一致。  
  - 使用 `sed` 进行文本替换，若 `Version` 行出现多次或非标准格式，可能误改其他字段，潜在的供应链完整性风险。  
  - 脚本依赖 `python -m wheel`，若系统中 `wheel` 包被篡改，可能导致恶意代码注入。  

**⚠️ 潜在风险**：  
1. **CUDA 检测误判**：只检查 `/usr/local/` 目录下的文件名，若在自定义路径或容器镜像中安装 CUDA，可能检测不到正确后缀，导致版本不匹配。  
2. **多次运行幂等性**：`sed` 替换平台标签时使用全局替换，若脚本意外执行第二次，可能把已经是 `manylinux2014` 的标签再次修改，引发文件名混乱。  
3. **临时目录清理**：`trap 'rm -rf -- "$TMPDIR"' ERR` 只在错误时触发，正常结束时未显式清理，可能留下临时目录占用磁盘空间。  
4. **版本冲突**：如果原始 `METADATA` 已经含有后缀（例如手动编辑过），脚本会跳过，可能导致生成的 wheel 文件名没有后缀，产生不一致的发布产物。  
5. **权限与可执行环境**：脚本假设有写权限到 `dist/` 目录并能执行 `mv`、`rm`，在受限 CI 环境或只读文件系统中会失败。  

**💡 关注建议**：  
- **测试覆盖**：在 CI 中分别模拟 CUDA 12.4、12.8、13.0 以及无 CUDA 三种环境，确保生成的 wheel 文件名、`METADATA.Version` 与 `WHEEL` 中的标签全部一致。  
- **幂等设计**：在脚本结束前加入检查，如果 wheel 已经符合目标命名（含后缀且平台标签为 manylinux2014），直接跳过后续解包/打包步骤。  
- **完善清理**：在脚本结束（无论成功或失败）都执行 `rm -rf "$TMPDIR"`，或使用 `trap 'rm -rf "$TMPDIR"' EXIT` 统一处理。  
- **路径检测增强**：改为通过 `nvcc --version` 或读取 `CUDA_HOME` 环境变量来判断 CUDA 版本，兼容自定义安装路径。  
- **安全审计**：在生成的 wheel 重新打包后，建议在 CI 中签名或执行 `pip install --no-deps --force-reinstall` 进行完整性校验，以防止因元数据修改导致的供应链风险。  
- **文档与发布说明**：在项目的 Release Note 中明确说明新生成的 wheel 将包含 `+cuXXX` 后缀，并更新对应的安装指南（如 `pip install sgl-kernel+cu124`），避免用户因命名变化产生困惑。

### perf(sgl-kernel): expose get_scheduler_metadata for FA3 decode optimization (#21103)
**SHA**: `a12fea2` | 🔗 [查看提交](https://github.com/sgl-project/sglang/commit/a12fea21ed08f1a5a808de2ed21bf8e43165e159)

**🎯 变更类型**：性能优化  

**⚡ 重要程度**：🔴高  

**📋 变更摘要**  
1. 在 `sgl-kernel` 中新增 `get_scheduler_metadata` 接口，将 FlashAttention‑3（FA3）在解码阶段的 tile 调度信息提前预计算。  
2. 通过 C++/CUDA 实现并在 Python 包装层暴露，使得该元数据只需在 **每个 batch** 调用一次，而不必在每层 Transformer 中重复执行 `prepare_varlen_num_blocks`。  
3. 预期显著降低长序列生成（尤其是解码）时的 GPU kernel 调用开销，提升整体吞吐与延迟。  

**🎯 影响范围**  
- `sgl-kernel/csrc/flash_extension.cc` – 注册新的 CUDA 实现。  
- `sgl-kernel/include/sgl_flash_kernel_ops.h` – 声明对应的 C++ 接口。  
- `sgl-kernel/python/sgl_kernel/flash_attn.py` – 添加 Python 调用封装。  
- 依赖 `flash_attn_with_kvcache` / `flash_attn_varlen_func` 的所有上层模型（Decoder、Chat 模型等）。  

**🔍 技术洞察**  

- **架构影响**  
  - **模块解耦**：调度元数据的生成被抽离为独立算子，解耦了 “调度准备” 与 “实际前向” 两个阶段，使得上层代码能够在更灵活的时机（如 batch 创建后、首层前）完成一次性计算。  
  - **接口向后兼容**：新增函数为 **可选** 调用，不影响已有 `flash_attn_*` 接口的使用，保持向后兼容性。  
  - **依赖链**：新算子依赖 CUDA 实现 `mha_fwd_get_scheduler_metadata`，因此编译链需包含对应的 .cu 文件；若用户自行编译扩展，需要确保此文件已加入构建目录。
