### feat: enable SGLANG_PATCH_TOKENIZER by default (#23409)
**SHA**: `c090f71` | 🔗 [查看提交](https://github.com/sgl-project/sglang/commit/c090f71bf207180f34deee9109dad5720e089f7a)

**🎯 变更类型**：功能增强  
**⚡ 重要程度**：🔴高  
**📋 变更摘要**：将 `SGLANG_PATCH_TOKENIZER` 环境开关默认值由 `False` 改为 `True`，在启动时自动启用对 Kimi tiktoken 的补丁，使其在高批量场景下缓存 `all_special_tokens` 与 `all_special_ids`（原本在高并发时会产生约 10 倍的额外开销）。此改动旨在提升大批量推理的吞吐与响应速度。  

**🎯 影响范围**：  
- `python/sglang/srt/environ.py`（环境变量定义）  
- `TokenizerManager` 与所有调用 `Env.SGLANG_PATCH_TOKENIZER` 的模块（包括 SRT 主入口、模型推理流水线、API 层）  
- 可能波及使用自定义 tokenizer 配置的用户代码  

**🔍 技术洞察**  

- **架构影响**：  
  - 环境变量默认值的改变属于全局配置，属于“横向”影响。所有使用 `Env.SGLANG_PATCH_TOKENIZER` 判断是否走补丁路径的代码将在运行时自动进入补丁路径，无需额外显式开启。  
  - 补丁实现依赖 `tiktoken` 的内部属性 (`all_special_tokens`/`all_special_ids`) 并在首次调用时将其缓存至模块级别的全局变量，属于 **轻量级装饰**，对整体系统架构无侵入式改动，但引入了 **全局状态**（缓存对象），需要确保在多进程/多线程环境下的共享一致性。  

- **性能影响**：  
  - **正向**：缓存特殊 token 列表可避免在每次 `encode`/`decode` 时反复遍历 tokenizer 内部结构，特别是当 batch size 大（> 128）时，可把 Tokenizer 的内部循环时间降低约 5‑10 倍，显著提升吞吐。  
  - **负向**：为每个进程/线程维护一次缓存，会消耗额外的内存（几 KB 量级），在极端并发（数千进程）下可能放大总内存占用。  
  - **可观测性**：若项目已开启 `SGLANG_ENABLE_METRICS_*` 相关指标，建议对比开启前后的 `tokenizer_time_ms` 统计，以量化收益。  

- **安全考虑**：  
  - 该补丁不涉及网络、文件系统或权限修改，表面上没有安全风险。  
  - 唯一潜在安全/正确性风险在于 **缓存失效**：如果在运行时动态替换 tokenizer（如热加载新模型、切换语言），缓存的 `all_special_ids` 仍然对应旧 tokenizer，可能导致 token‑id 映射错误，从而产生错误的输入序列或安全审计日志。  

**⚠️ 潜在风险**：  
1. **缓存不一致**：在同一进程中多次切换 tokenizer（不同模型、不同版本）时，缓存不会自动刷新，可能导致 token 化错误。  
2. **全局状态竞争**：在多线程环境下，首次写入缓存的竞态未加锁，极端情况下可能出现写入冲突导致异常。  
3. **兼容性回归**：旧版依赖 `SGLANG_PATCH_TOKENIZER=False` 的行为（如手动补丁或特定测试场景）现在会被覆盖，若用户未注意到可导致意外行为变化。  
4. **内存轻微上升**：大量进程/容器部署时累计的缓存占用可能略微提升总体内存使用。  

**💡 关注建议**：  
- **测试**：在 CI 中加入高批量（≥256）推理基准，分别在 `True/False` 两种配置下对比 `tokenizer` 耗时与整体吞吐，确保性能提升符合预期且结果正确。  
- **缓存失效机制**：考虑在 `TokenizerManager` 中检测 tokenizer 实例变化，一旦发现不一致，主动清除并重新创建缓存。  
- **并发安全**：在首次写入缓存时加入轻量级的 `threading.Lock`，防止稀有的竞态写入异常。  
- **文档与可配置性**：在项目 README 与环境变量说明中明确该变量默认已开启，并提供 `SGLANG_PATCH_TOKENIZER=False` 的回退方式，方便已有部署快速回滚。  
- **监控**：若开启度量功能，新增一项 `tokenizer_patch_enabled` 指标，帮助运维快速判断当前实例是否走补丁路径。  

通过上述措施，可在最大化性能收益的同时，将潜在的正确性与运维风险控制在可接受范围。

3. **性能影响**：新增的字符串拼接与点检测在大模型加载阶段仍属于 O(N) 规模，影响可忽略；若 `ignored_layers` 列表非常长，可考虑将其预处理为集合或前缀树以进一步提升匹配效率。  
4. **向后兼容**：旧代码仍可运行，因默认映射仅在 `proj_name` 未在 `fused_mapping` 中出现时使用，不会影响已有配置。发布前请确保 CI 中的所有量化/FP8 测例全部通过。  

总体来看，此次修复消除了 FP8 配置中模块名冲突导致的误转换风险，兼容性提升明显，风险点主要在回退映射的完整性和匹配函数的边界测试，建议对应补充测试并在文档中给出扩展指南。

### PD streaming: batch notify + SSE fast path (#22658)
**SHA**: `1e34cd0` | 🔗 [查看提交](https://github.com/sgl-project/sglang/commit/1e34cd0ba5fb071b935faf348eb9d7dfc63b6ccc)

**🎯 变更类型**：功能增强 / 性能优化  
**⚡ 重要程度**：🟡 中  

**📋 变更摘要**  
1. 为 OpenAI‑compatible ChatCompletion SSE 流新增 “fast‑path” 编码，使用 `msgspec` 将结构体直接序列化为 JSON，省去 `pydantic` 的 `model_dump_json` 与 `ORJSONResponse` 的中间层。  
2. 在 `TokenizerManager` 中引入 `batch_notify_size`，将多个 token 输出合并后一次性 `event.set()`，降低高并发下的 asyncio 唤醒次数。  
3. 为该参数新增 CLI 选项并在 `ServerArgs` 中提供默认值 16。  

**🎯 影响范围**  
- `python/sglang/srt/entrypoints/openai/serving_chat.py`（SSE 序列化、流生成逻辑）  
- `python/sglang/srt/managers/tokenizer_manager.py`（批量通知实现）  
- `python/sglang/srt/server_args.py`（新增配置项）  

**💡 关注建议**  
1. **SSE 兼容性**：`_StreamDelta` 使用 `omit_defaults=True`，确保 `reasoning_content` 为 `null` 时仍被序列化；建议在单元测试中验证客户端对 `null` 与缺失字段的行为。  
2. **并发安全**：`_handle_batch_output` 中的 `pending_notify` 在同一协程内累积后一次性 `event.set()`，但在异常或提前返回时需确保未遗漏通知。可在 `finally` 块中统一释放。  
3. **配置调优**：`batch_notify_size` 默认 16，实际效果取决于请求并发与 token 输出速率，建议在性能基准中测算最优阈值并在文档中说明。  
4. **回归检查**：旧的 `model_dump_json` 代码已移除，确认所有 OpenAI SDK 客户端仍能解析 `data: {...}` 格式，尤其是 `matched_stop` 与 `logprobs` 字段。  
5. **依赖兼容**：`msgspec` 需要 Python≥3.8 并在部分平台可能缺少二进制轮子，确保在 CI 中加入安装验证。  

总体来看，此次改动通过零拷贝序列化和批量事件通知显著降低了流式响应的 CPU 与调度开销，但需补充对应的兼容性与并发可靠性测试。

### [Diffusion] Add mixed-resolution benchmark support (for #20762) (#20863)

**⚡ 重要程度**：🟡 中  

**📋 变更摘要**  
1. 为离线吞吐基准 `bench_offline_throughput.py` 与在线基准 `bench_serving.py` 增加 `--random-request-config` / `--random-request-seed` 参数，实现对同一数据集（仅 `random`）按不同分辨率、帧数、步数等随机抽样。  
2. `generate_batch` 改为接受每条 prompt 对应的采样字典列表，并在统计像素数时考虑各请求实际的 `width/height/num_frames`。  
3. `datasets.py` 为 `RandomDataset` 引入配置解析、加权抽样以及 `get_sampling_params` 接口，返回每条请求的专属配置。  

**🎯 影响范围**  
- `python/sglang/multimodal_gen/benchmarks/*`（离线、在线 benchmark、数据集）  
- `DiffGenerator.generate` 调用签名间接受影响（需传入完整字典）。  

**💡 关注建议**  
1. **类型安全**：`BenchArgs.random_request_config` 由 `str` -> `Optional[str]`，但在 `BenchArgs` 中未标注 `Optional`，建议统一 `Optional[str]`。  
2. **向后兼容**：`generate_batch` 新增断言检查长度匹配，若外部仍传单一 dict 会抛异常；确保所有调用已改为列表（已在本文件更新，但若有其他入口需同步）。  
3. **随机抽样实现**：使用 `random.Random(seed).choices`，权重列表若缺失 `weight` 会报错；可加容错提示。  
4. **统计准确性**：`calculate_metrics` 现在根据 `all_sampling_params` 重新计算像素数，确保 `num_success` 与参数列表长度对应；若某请求失败，仍使用前 `num_success` 条参数，可能导致统计偏差，建议在统计前过滤成功请求对应的参数。  
5. **文档/帮助**：CLI `--random-request-config` 帮助已补全示例，建议在 README/benchmark 文档中加入 “混分辨率基准使用说明”。  
6. **测试覆盖**：添加单元/集成测试，验证 `random` 数据集在不同权重、seed 下生成的 `all_sampling_params` 长度、分布以及吞吐统计是否符合预期。  

总体改动合理，提升了基准的真实性和灵活性，只需注意兼容性检查与统计细节即可安全上线。
