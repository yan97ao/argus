### [model] Apertus Tool/Function and Reasoning parser (#25100)
**SHA**: `e9dbbd1` | 🔗 [查看提交](https://github.com/sgl-project/sglang/commit/e9dbbd19e925008f2e8c975f03ffa72cf4dfedea)

**🎯 变更类型**：功能增强  
**⚡ 重要程度**：🔴高  
**📋 变更摘要**：本次提交为 SGLang 新增 **Apertus 2509** 系列模型的推理分块与工具调用解析能力。新增了专属的 `Apertus2509Detector`（推理块与工具调用两套检测器），并在 HTTP 接口、OpenAI 兼容层、模板检测、文档等多处完成集成，使得模型可通过 `<|inner_prefix|>…<|inner_suffix|>` 标记分离推理内容，并在工具调用中使用 `<|tools_prefix|>`/`<|tools_suffix|>` 包裹的 JSON 列表格式。  

**🎯 影响范围**：  
- `sglang/srt/parser/reasoning_parser.py`（推理解析）  
- `sglang/srt/function_call/apertus2509_detector.py`（工具调用解析）  
- `sglang/srt/function_call/function_call_parser.py`（注册新 parser）  
- `sglang/srt/entrypoints/http_server.py`、`sglang/srt/entrypoints/openai/serving_chat.py`（API 参数、skip‑token 处理）  
- `sglang/srt/managers/io_struct.py`（新增 `return_blocks`）  
- `sglang/srt/managers/template_detection.py`（模板检测规则）  
- 文档 `docs_new/...`（说明与示例）  
- 单元测试 `test_template_manager.py`（新增检测用例）  

---

### 🔍 技术洞察  

| 维度 | 影响分析 |
|------|----------|
| **架构影响** | • 通过 **插件化的 Detector**（`BaseReasoningFormatDetector` / `BaseFormatDetector`）实现模型专属解析，遵循现有扩展机制，保持架构连贯性。<br>• `ReasoningParser` 与 `FunctionCallParser` 均映射新的键 `"apertus2509"`，无需改动调用方，只是新增映射表项。<br>• `http_server` 新增 `return_blocks` 参数，提升业务侧对块级别输出的可控性。 |
| **性能影响** | • `Apertus2509Detector` 在 **非流式** 场景通过一次正则+ `json.JSONDecoder().raw_decode` 完成解析，时间复杂度 O(n)。<br>• 流式解析维护内部缓冲区与状态机，额外的字符串切片与 token 查找开销极小（与已有 DeepSeek、Mistral 实现相当）。<br>• 对于大量工具调用的长文本，JSON 解析仍是主要成本，使用标准库 `json`，性能可接受。 |
| **安全考虑** | • 解析过程直接使用 `json.loads`（`JSONDecoder().raw_decode`），若模型生成的 JSON 结构不符合规范会抛异常，已在代码中捕获并回退为普通文本，避免服务崩溃。<br>• 通过 `envs.SGLANG_FORWARD_UNKNOWN_TOOLS` 控制未知工具调用的转发策略，默认会 **warning** 并丢弃，降低恶意工具调用的风险。<br>• `skip_special_tokens` 逻辑在 `serving_chat.py` 中对 Apertus 设为 `False`，防止关键分隔符被意外去除。 |
| **可维护性** | • Detector 实现遵循统一基类，代码结构清晰，易于后续扩展其他模型。<br>• 文档同步更新，示例已覆盖新模型，有助于使用者快速上手。<br>• 新增的 `return_blocks` 字段在接口层统一返回，保持向后兼容（默认 `False`）。 |

### ⚠️ 潜在风险  

1. **兼容性冲突**  
   - 若已有模型的 vocab 中恰好出现 `<|inner_prefix|>` / `<|tools_prefix|>`，可能误判为 Apertus。当前检测仅依据 vocab 的 **存在性**，建议在模型发布说明中明确该 token 的唯一性，或在检测规则中加入模型名称白名单。  

2. **JSON 解析异常**  
   - 当模型输出不完整的 JSON（例如缺失闭合 `]`），`raw_decode` 会返回 `None` 并导致后续块被当作普通文本返回，可能导致工具调用漏报。需要在生产环境监控此类异常率。  

3. **状态机边界条件**  
   - 流式解析中对 **部分 token**（如 `<|tools_prefix`）的截取采用 `_ends_with_partial_token`，若出现极端的字符切分（比如跨 token 的 UTF‑8 多字节），仍可能出现误判。建议加入更严格的 token 完整性校验或使用 tokenizer 的 token‑level信息。  

4. **新增 API 参数**  
   - `return_blocks` 参数默认 `False`，但未在 OpenAPI schema 中声明。若客户端误用可能导致返回结构变化，需在文档及 schema 中明确标记为 **可选**。

- **监控**：部署后监控 `Apertus2509Detector` 的 `warning` 日志频率，及时发现未知工具或异常 JSON。  
- **文档**：在模型注册表或 README 中明确标记 **Apertus 2509** 所需的 `--tool-call-parser apertus2509` 与 `--reasoning-parser apertus2509` 参数，避免用户因默认 parser 未匹配而得到错误输出。  
- **安全审计**：确认 `envs.SGLANG_FORWARD_UNKNOWN_TOOLS` 的默认值符合业务安全策略；若业务需要严格限制工具调用，建议将其默认设为 `False` 并在配置中心统一管控。  
- **后续扩展**：如果后续出现其他使用类似 `<|inner_prefix|>` 的模型，可直接复用 `Apertus2509Detector`，只需在 `template_detection.py` 中添加相应的检测规则即可。

### [Doc][CPU]Update Cookbook with Xeon support info (#27248)

**🎯 变更类型**：功能增强（为 Intel Xeon CPU 添加部署文档与交互式指令生成）  
**⚡ 重要程度**：🔴 高  
**📋 变更摘要**：  
- 在《Cookbook》中的所有自回归模型章节加入对 Xeon CPU 的说明，指向专用的 CPU 版安装指南。  
- 前端部署交互组件（`*.jsx`）新增 “Xeon” 选项，随之对硬件‑特定的量化、并行策略、特性可用性做了细粒度约束（如：TP = 6、仅支持 Tensor‑Parallel、禁用 DP/EP/MTP、FP4、FP8‑KV‑Cache 等）。

**🎯 影响范围**：

### 🔍 技术洞察  

| 维度 | 影响分析 |
|------|----------|
| **架构影响** | - 前端 UI 增加了硬件感知的 **条件渲染**（`condition`、`disabledWhen`）以及 **状态同步**（硬件切换时自动清理不兼容选项）。<br>- 新增的 `--device cpu` 与 `--disable-overlap-schedule` 参数依赖后端 **CPU 服务** 实现；如果服务器代码仍然假设 GPU 为默认，则可能出现运行时错误。<br>- 通过在 `DeepSeekR1BasicDeployment` 等处强制把量化切换为 `int8`（Xeon 时），对后端的 **量化渠道**（`w8a8_int8`）提出了新需求。 |
| **性能影响** | - Xeon 采用 **TP = 6**（相较于 GPU 的 8）并关闭 **overlap schedule**，这会导致 **吞吐量下降**（尤其是多路并行的模型）。<br>- 禁用 `--enable-symm-mem`、`--kv-cache-dtype fp8_e4m3` 等针对于 GPU 的优化，进一步降低 CPU 场景下的潜在性能。<br>- 引入 `int8` 量化在 Xeon 上可提升 **推理速度 & 内存占用**（相较于 fp8/fp4），但只有在后端实现了对应的 W8A8 算子时才有效。 |
| **安全考虑** | - 文档中大量出现 `--trust-remote-code`（尤其在 Qwen 系列），这在 CPU 环境下并未做额外限制，仍然可能执行未审计的第三方模型代码。<br>- 新增 `--disable-overlap-schedule` 只影响调度层，不涉及安全。<br>- 前端 UI 采用 `disabledWhen` 动态禁用不支持的选项，降低用户误配的风险。 |
| **可维护性** | - 前端代码逻辑变得更复杂：动态获取选项 (`getDynamicItems`)、条件渲染 (`condition`)、选项互斥（在硬件切换时自动过滤）。若后续再加入新的硬件或特性，需要统一维护 **硬件‑特性映射表**（如 `xeonOnly`、`disabledWhen` 等）。<br>- 文档中大量复制 “For configuring CPU service …” 文字，若 CPU 文档路径变动，需要同步更新。 |
| **兼容性** | - 对已有 GPU‑Only 使用者影响极小，所有新增逻辑均围绕 `hardware === 'xeon'` 分支。<br>- 部分 **默认值**（如 `quantization` 在 Xeon 场景下强制为 `int8`）可能与老版本 UI 不兼容，导致 UI 初始化时出现 “未找到默认项” 的报错，需要确保 `default` 与 `disabled` 组合始终能产生合法默认选项。 |

### ⚠️ 潜在风险  

| 风险点 | 说明 | 可能的后果 |
|--------|------|-------------|
| **后端 CPU 参数支持缺失** | `--device cpu`、`--disable-overlap-schedule`、`--quantization w8a8_int8` 需要后端实现。若 SGLang 主库尚未实现这些 flag，启动会报错。 | 用户在 CPU 场景下无法使用 UI 生成的命令，导致部署失败。 |
| **不一致的默认选项** | 某些 `getDynamicItems` 在 Xeon 场景下将全部选项标记 `disabled:true`（如 FP4、FP8），但仍保留 `default:true`，可能导致 UI 初始化时选中一个被禁用的项。 | UI 渲染异常或误导用户选择不可用的配置。 |
| **并行度硬编码风险** | TP 在 Xeon 场景硬编码为 6（部分模型为 3），未根据实际 CPU 核心数动态适配。 | 在核数更高或更低的 Xeon 机器上性能不佳或产生资源超额错误。 |
| **文档同步** | 多处硬编码的文档链接 `../../../docs/hardware-platforms/cpu_server#...` 若路径变更，需要同步更新。 | 文档失效，用户找不到 CPU 安装指南。 |
| **安全风险** | 仍然默认开启 `--trust-remote-code`，在 CPU 环境下执行未知模型代码的风险没有额外控制。 | 可能导致供应链攻击或恶意代码执行。 |
| **前端状态同步错误** | 在硬件切换时，手动过滤 `strategy`、`quantization`、`modelname` 等字段的代码较为繁琐，若未来添加新选项，遗漏过滤可能导致非法组合被保留下来。 | 启动命令包含不支持的参数，产生运行时错误。 |

**🔍 技术洞察**  

- **架构影响**  
  - 新增 **BitsAndBytes4BitAdapter** 与 **BitsAndBytesLinearMethod**，在模型层级把 `nn.Linear` 替换为只存储 4‑bit 量化权重的实现，并在加载时恢复 `bitsandbytes` 量化状态 (`QuantState`)。  
  - `Ideogram4Transformer2DModel` 通过可选 `quant_config` 参数向所有子层（`input_proj`、`llm_cond_proj`、MLP、Attention、AdaLN 等）注入量化线性算子，实现 **统一量化管线**。  
  - 为 NVFP4 引入 **per‑component transformer 权重覆盖**（`component_transformer_weights_paths`），解决同一 checkpoint 内部 DiT 与 Unconditional DiT 使用不同量化文件的问题。

- **性能影响**  
  - **显存占用**：BitsAndBytes 4‑bit (NF4) 权重仅占原始 FP16/FP32 的 1/8，显著降低部署显存需求，尤其在单 GPU 环境（`num_gpus=1`）下可直接加载 Ideogram 4 NF4。  
  - **计算开销**：`apply_bnb_4bit` 调用 `bitsandbytes.matmul_4bit`，利用 CUDA kernel 高效完成 4‑bit 矩阵乘，本身比 FP8 略慢但相对 FP16/FP32 有明显加速。对大 batch / 长序列仍受限于 Transformer 计算瓶颈。  
  - **IO 与加载时间**：在 `fsdp_load.py` 中拆分量化状态与普通权重，仅一次遍历 safetensors，可在不解压量化元信息的情况下完成权重加载，略提升启动速度。  
  - **NVFP4 支持**：通过检测 `packed_qkv` 与 `comfy_quant` 标记，自动选择 “swizzled” 或 “linear” scale 布局，兼容两种 checkpoint 格式，避免因布局不匹配导致的额外转换步骤。  

- **安全考虑**  
  - 新增 `HF_TOKEN` 环境变量的使用仍保持原有安全模型（通过 HuggingFace 私有仓库身份验证），未改变 token 处理方式。

**⚠️ 潜在风险**  

1. **GPU‑resident 需求**：BitsAndBytes 4‑bit 需要权重在 GPU 上保持完整状态（`bnb_quant_state`、`bnb_shard_offsets`），若用户开启 `fsdp_cpu_offload`、`dit_cpu_offload` 或其它 offload 选项，可能导致运行时崩溃或精度失效。代码已加入警告并自动禁用 offload，但仍需在生产环境验证。  
2. **兼容性**：`component_transformer_weights_paths` 只在新版服务器启动参数中使用，旧版脚本或自定义启动方式可能忽略此字段，导致 NVFP4 的 Unconditional DiT 权重未被加载。  
3. **量化配置冲突**：当 `quantization` 参数与模型内部的 `quantization_config`（bitsandbytes）不一致时，`post_diffusers_config_update` 可能产生 `ideogram_bnb_4bit_weight_only` 与 `ideogram_fp8_weight_only` 双真值的冲突，需要确保仅有一种量化方式被激活。  
4. **测试覆盖不足**：当前单元测试覆盖了大部分路径，但对 **多 GPU（TP/PP）** 环境下的 `bitsandbytes` 量化状态切分未做验证，潜在的分片同步错误可能在大规模部署时显现。  
5. **依赖升级风险**：BitsAndBytes 近期 API（如 `bnb_quant_state`）的内部实现可能变化，若用户自行升级到 >0.48 版本，当前适配层可能失效。  

**💡 关注建议**  

- **部署前检查**：在启动服务前确保 `bitsandbytes>=0.46.1` 已安装，且 `HF_TOKEN` 正确指向 Ideogram 4 私有仓库。  
- **显卡选择**：NF4 建议使用常规 A100 / H100；NVFP4 推荐 Blackwell B200 或更高的 CUDA 12.x 兼容卡，以获得 `matmul_4bit` 最佳性能。  
- **Offload 策略**：对使用 BitsAndBytes 4‑bit 的模型，显式关闭 `dit_cpu_offload` 与 `use_fsdp_inference`（或在 `ServerArgs` 中保留默认），防止自动 offload 破坏 GPU‑resident 权重。  
- **CI 更新**：将 CI 环境变量 `CUDA_VISIBLE_DEVICES` 与 `COMFY_ORG` checkpoint 路径同步，确保多节点测试覆盖 `component_transformer_weights_paths` 场景。  
- **监控量化状态**：在日志中加入 `bnb_quant_state` 是否成功 attach 的调试信息（可利用 `logger.debug`），便于快速定位权重加载异常。  
- **文档同步**：更新 README 中的 “Supported Quantization” 表格，标明 NF4 需要 `bitsandbytes`，NVFP4 需要 “comfy‑org” 权重文件以及对应的 `transformer_weights_path` 配置方式。

### [SPEC][5/N] feat: batchsize-aware support for adaptive speculative_num_steps (#24055)
**SHA**: `6b18095` | 🔗 [查看提交](https://github.com/sgl-project/sglang/commit/6b180959a88ff50c887094e756a36f5557dd57e8)

**🎯 变更类型**：功能增强 (batchsize‑aware 支持的自适应 speculative decoding)  
**⚡ 重要程度**：🔴 高  
**📋 变更摘要**  
本次提交在 SGLang 的 speculative decoding 机制中引入 **批大小感知** 的自适应控制。核心改动包括：  
1. 配置文件扩展为 **按 batch‑size 区间** 定义独立的 EMA 参数、候选 step 列表和上/下阈值。  
2. `AdaptiveSpeculativeParams` 拆分为 **全局路由层** 与 **每个 batch‑size 槽的 `AdaptiveStepSlot`**，实现 batch‑size → slot → step 的两层映射。  
3. `AdaptiveController` 新增 **`activate_step_by_batch`** 与 **`on_verify_complete(..., batch_size)`**，在每轮 draft 前根据当前 batch‑size 查询并切换最佳 step。

**🎯 影响范围**  
- **核心模块**：`sglang/srt/speculative/*`（adaptation 参数、控制器、runtime state）

**🔍 技术洞察**  

| 维度 | 影响 |
|------|------|
| **架构影响** | - **分层路由**：在全局 `AdaptiveSpeculativeParams` 中维护 batch‑size 列表与对应的 `AdaptiveStepSlot`，实现 **按 batch‑size 隔离的 EMA**，避免小 batch 的噪声污染大 batch 的信号。<br>- **运行时状态管理**：`AdaptiveController.init_states` 现在接受 `cuda_graph_bs`，为每个 `step` 只创建 **有效的 CUDA‑graph batch‑size 子集**，降低图捕获冗余。<br>- **Worker 接口扩展**：`BaseSpecWorker` 增加 `activate_step_by_batch` 与 `on_verify_complete_cpu(..., batch_size)`，保持后端实现对自适应层的透明。 |

| **可维护性** | - 配置结构变得更复杂（全局键 + 整数 batch‑size 键），但代码已对非整数键做过滤，且提供统一的 `DEFAULT_ADAPTIVE_CONFIG` 与验证函数，降低误配置概率。<br>- 通过 `cached_property` 实现 `candidate_steps` 的一次性合并，避免重复计算。<br>- 文档已同步更新，示例配置明确分割 “全局参数” 与 “每‑BS 参数”。 |

**⚠️ 潜在风险**  

1. **配置错误导致不可达的 step**  
   - 若某 batch‑size 槽的 `candidate_steps` 与全局 `candidate_steps` 不匹配，`AdaptiveController.init_states` 可能未创建对应 `SpecRuntimeState`，导致激活失败。  
   - 已通过 `validate_adaptive_initial_steps` 检查启动时 `--speculative-num-steps` 是否在 **任意** slot 中，但运行时仍可能出现 **未覆盖的 batch‑size**（例如未在配置中声明的 batch‑size 范围），会回退到最近的低层 slot，行为仍可预测但可能不符合预期。

2. **CUDA‑graph 捕获不匹配**  
   - `cuda_graph_bs_for_step` 会在某 step 没有任何可用的图 batch‑size 时返回空列表，随后在 `_override_worker_state` 中会临时 **禁用 CUDA‑graph**。若用户期望始终使用图而未意识到此自动降级，可能导致**吞吐波动**。

3. **显存占用增长**  
   - 每个（slot, step）组合都要保持一套 `SpecRuntimeState`（包括 CUDA‑graph 缓存），在多 slot + 多 step 场景（如 10‑12 个 slots、每个 5 步）可能显著提高显存需求。  
   - 若显存不足，进程会在捕获阶段 OOM，导致服务启动失败。

4. **回退路径兼容性**  
   - 代码在 `AdaptiveSpeculativeParams` 中对 `initial_steps` 不在候选集合时默认取 **中位数**，这与旧实现“直接插入缺失的 step”行为不同，可能导致 **旧配置迁移时 step 变化**，需在发布说明中提醒。

5. **单元测试覆盖不足**  
   - 现有测试只覆盖基本路径和部分异常情况，未测试 **CUDA‑graph 禁用后** 的完整 forward 路径、以及 **高并发、多 batch‑size 切换** 场景，可能隐藏竞争或状态同步 bug。

### feat(agentic router, 1/N): Add LoadBasedPolicy (#26480)
**SHA**: `e4a7388` | 🔗 [查看提交](https://github.com/sgl-project/sglang/commit/e4a7388187857994d2194a3bf003f383c267431b)

**🎯 变更类型**：功能增强  
**⚡ 重要程度**：🔴高  

**📋 变更摘要**  
本次提交为实验性路由器（sgl‑router）引入 **LoadBasedPolicy**，实现基于工作节点实时 “active_load” 的最小负载路由。CLI 现在接受 `--policy load_based` 参数，`PolicyKind` 枚举、策略工厂以及选择上下文 (`SelectionContext`) 被相应扩展；并随之补充单元测试与一个真实 GPU 环境的端到端（e2e）测试，验证在已有忙碌节点的情况下短请求会自动转向空闲节点。

**🎯 影响范围**  
- `experimental/sgl-router/src/config/*`（CLI 解析、`PolicyKind`）  
- `experimental/sgl-router/src/policies/*`（新增 `load_based.rs`，工厂、`mod.rs` 更新）  
- `experimental/sgl-router/src/policies/factory.rs`（策略构造路径）  
- `experimental/sgl-router/src/policies/mod.rs`（`SelectionContext` 新增 `routing_key` 字段）  
- 新增 e2e 测试 `tests/e2e/chat_completions/test_load_based_policy.py`  

**🔍 技术洞察**  

- **架构影响**  
  - **模块化**：LoadBasedPolicy 作为独立模块加入 `policies`，遵循现有 `Policy` trait，保持工厂化创建方式，未破坏现有抽象。  
  - **配置层**：CLI 与 `PolicyKind` 扩展为向后兼容的增量变更，默认行为（RoundRobin）保持不变。  
  - **上下文扩展**：`SelectionContext` 新增 `routing_key`（可选）字段并提供 `with_routing_key` 构造函数，为后续基于请求内容的路由策略预留空间，对现有调用不产生影响（`new` 仍然可用）。  

- **性能影响**  
  - **负载均衡**：在多节点情况下，通过 `Worker::active_load()` 选取当前最轻的节点，能够显著降低高并发时的单点拥塞，提升整体吞吐与平均响应时间。  
  - **选取成本**：每次路由遍历所有可用 `Worker`（O(N)），其中 N 通常为数个到十几个，开销可忽略；唯一潜在开销在于 `active_load` 的原子读取（如果实现为 `AtomicUsize`），仍在微秒级别。  
  - **资源竞争**：依赖 `Worker::load_guard()` 正确更新 `active_load`，若 guard 实现出现漏释放或异常路径未恢复计数，可能导致误判负载，出现路由倾斜。  

- **安全考虑**  
  - 本次改动仅涉及路由决策逻辑和配置解析，无新增网络、权限或加密机制，不直接影响安全边界。  
  - 需要确认 `active_load` 的采集不泄露敏感内部状态；当前仅通过 Prometheus 指标 `sgl_router_active_load` 暴露，已在项目的监控规范内。  

**⚠️ 潜在风险**  

1. **负载计数不一致**  
   - `Worker::active_load()` 若未在异常路径（如 panic、超时）及时归零，会导致伪负载，进而误导 LoadBasedPolicy。  
2. **Metric 暴露延迟**  
   - 端到端测试依赖 `/metrics` 中的 `sgl_router_active_load` 指标，如果指标更新频率不足，短请求可能仍被分配到忙碌节点。

   - `SelectionContext` 结构体新增字段对所有手动构造的位置无影响，但若外部库直接使用结构体字面量（而非 `new`/`with_routing_key`），将因字段缺失编译错误。  
4. **并发竞争**  
   - 在极端高并发下，所有请求几乎同时读取相同 “最小负载” 值，可能导致瞬时热点（即使是最小负载节点也会瞬间收到大量请求）。目前未实现“随机化冲突”或 “二次抽样” 机制。

### [PD] Fix KV cache corruption on abort by notifying ongoing prefill (#27372)
**SHA**: `57909f7` | 🔗 [查看提交](https://github.com/sgl-project/sglang/commit/57909f731b0fb325f0313946cf358a23fbe0fd00)

**🎯 变更类型**：Bug 修复 / 功能增强  
**⚡ 重要程度**：🟡 中  
**📋 变更摘要**：在 KV 缓存预取（prefill）阶段，如果请求被中止，之前的实现会导致 KV 缓存残留，进而产生错位或重复计算。此次改动在 `CommonKVBootstrapClient` 中加入 `abort_notified` 标记，并在超时或显式 abort 时主动向所有 bootstrap‑side 发送 `"ABORT"` 消息；在 `Mooncake` 端相应地对 `"ABORT"` 进行拦截、标记房间状态为 `Failed`，并回送 `"ABORT_ACK"`。同时在 `transfer_worker` 中跳过已经标记为失败的房间，防止后续 chunk 被错误写入缓存。

**🎯 影响范围**  
- `python/sglang/srt/disaggregation/common/conn.py`（client 报错通知逻辑）  
- `python/sglang/srt/disaggregation/mooncake/conn.py`（worker、bootstrap 与 decode 线程的状态判断与消息交互）  
- 相关 KV 缓存管理与状态追踪模块（`KVPoll`、`kv_mgr`）。

**💡 关注建议**  
1. **幂等性检查**：`abort_notified` 只能在第一次发送后置为 `True`，但在异常路径（如连接失败）未发送成功时仍会标记，后续可能遗漏真正的 abort。建议在发送成功后再置位，或在捕获异常时保留重试机会。  
2. **线程安全**：`abort_notified` 与 `bootstrap_infos` 同时被多个线程读取/写入，若对象在多线程环境下共享，需要加锁或使用 `threading.Event` 替代普通布尔变量。  
3. **资源清理**：`_connect_to_bootstrap_server` 打开的 socket 未在异常后显式关闭，可能导致文件描述符泄漏。加入 `finally` 关闭或使用上下文管理器更安全。  
4. **ACK 处理**：当前仅记录 `ABORT_ACK`，但未利用它实现“延迟释放”或确认成功。若后续计划使用，建议在状态表中记录 ACK 收到的时间，以免重复发送 abort。  
5. **测试覆盖**：增加单元/集成测试，覆盖以下场景：① 超时触发 abort 并成功通知；② 网络异常导致通知失败的回退；③ 已成功完成的房间收到 abort，保持 `Success` 状态；④ 多并发 abort 场景下的幂等性。  
6. **日志级别**：大量 `debug` 日志在生产环境可能影响性能，考虑在关键路径使用 `info` 或 `warning`，并提供开关控制详细度。  

总体而言，此次改动能够显著降低因请求中止导致的 KV 缓存污染风险，但需注意上述并发安全和资源管理细节，并通过自动化测试验证其鲁棒性。
