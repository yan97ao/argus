**⚡ 重要程度**：🔴 中  

**📋 变更摘要**：  
在 `sgl-model-gateway` 的 Python 启动脚本 `launch_router.py` 与 Rust 主程序 `main.rs` 中分别加入了多行警示信息，提示当前仓库已不再维护，推荐用户迁移至独立的 `https://github.com/lightseekorg/smg` 仓库。该横幅会在标准错误流中打印，提醒用户尽快迁移以获取最新特性、Bug 修复和安全更新。

**🔍 技术洞察**：

- **架构影响**：  
  - 仅是输出层的改动，不涉及业务逻辑、路由实现或依赖关系。

- **性能影响**：  
  - 在启动阶段多打印几行文字，CPU 与内存开销可忽略不计。  
  - 由于使用 `eprintln!`/`print(..., file=sys.stderr)`，对标准输出/错误的写入可能对极端高并发日志收集系统产生极小的 I/O 负担，但实际影响可认为是 **0**。

  - 未直接修改安全相关代码或权限检查。  
  - 通过提醒用户迁移到官方维护的仓库，间接提升后续安全性（用户更有可能获得安全补丁）。  
  - 唯一潜在安全风险是：如果有脚本依赖原有标准输出格式进行解析，新增的警告信息可能导致解析错误，进而触发异常或错误路径。  

**⚠️ 潜在风险**：  

1. **兼容性冲突**：  
   - 自动化部署、CI/CD 或监控系统可能对 `sgl-model-gateway` 的 stdout/stderr 做严格匹配，新增的警告信息可能导致构建或监控误报。

   - 警告文字写在 `stderr`，但如果用户仅捕获 `stdout`，可能看不到提示，导致迁移计划延后。  

3. **维护负担**：

**💡 关注建议**：

### Fix chunked prefill and KV cache leaks for streaming sessions (#20476)
**SHA**: `b1246c5` | 🔗 [查看提交](https://github.com/sgl-project/sglang/commit/b1246c50f8873bc1bd5fd4ad57dcabd707692795)

**🎯 变更类型**：Bug 修复（解决 Streaming 会话在 Chunked‑prefill 场景下的 KV 缓存泄漏）  
**⚡ 重要程度**：🟡 中  

**📋 变更摘要**  
1. 在 `memory_pool.alloc` 中放宽了 “同一批次只能有一个 chunked 请求复用 `req_pool_idx`” 的断言，防止因该检查导致合法的 chunk‑prefill 被 abort。  
2. `session_aware_cache.restore_to_req` 不再清除 `req_pool_idx` 与 `mamba_pool_idx`，保证在调度因预算不足等被重新调度时，槽位状态保持幂等。  
3. `cache_unfinished_req` 对 streaming‑session 的 chunked‑prefill 做了专门分支：跳过 stash（radix）路径，仅在最后一块进行完整插入，防止 KV 索引残留。  
4. 新增 `test_streaming_session_leak.py`，通过并发多轮、混合 streaming 与非‑streaming 请求，验证关闭会话后 `/health` 返回 200，确保没有内存泄漏。  

**🎯 影响范围**  
- `python/sglang/srt/mem_cache/memory_pool.py`  
- `python/sglang/srt/mem_cache/session_aware_cache.py`  
- 流式会话调度与 KV 缓存管理（`srt` 模块）  
- 自动化测试套件（`test/registered/sessions/`）  

**💡 关注建议**  
1. **断言放宽**：虽然暂时注释掉的检查可避免异常，但可能在极端并发下出现多个 chunked 请求复用同一槽位的竞争，建议在后续 PR 中加入更细粒度的同步或显式错误提示。  
2. **不清除索引**：保留 `req_pool_idx`/`mamba_pool_idx` 能提高重试幂等性，但若会话异常终止（如客户端断连）而未显式释放，可能导致槽位“僵尸”。请确保在会话关闭或超时路径中强制清理。  
3. **缓存路径分支**：新增的 `chunked` 参数路径对 KV 索引的拷贝操作 (`kv_indices.to(dtype=torch.int64, copy=True)`) 需确认不会产生不必要的内存拷贝，建议在性能基准中评估其开销。  
4. **测试覆盖**：当前测试只检查健康接口，建议加入内存使用量（如 `torch.cuda.memory_allocated()`）的断言，进一步捕获潜在的细微泄漏。  
5. **文档更新**：在 `README` 或 API 文档中注明 `--enable-streaming-session` 与 `--chunked-prefill-size` 的使用注意事项，以及在高并发场景下的潜在限制。  

总体来看，此次修复针对关键的内存泄漏问题，改动集中且有针对性的单元测试验证，风险可控。后续可通过上述建议进一步强化可靠性。

### [diffusion] UX: suppress excessive logging from httpx and httpcore (#20452)
**SHA**: `b638b25` | 🔗 [查看提交](https://github.com/sgl-project/sglang/commit/b638b25b22c7b420aca22c627bcb2c11db737938)

**🎯 变更类型**：功能增强 / 重构  
**⚡ 重要程度**：🟡 中  
**📋 变更摘要**  
- 将原本散落在 `server_args.py` 中的日志脱敏实现迁移至 `logging_utils.py`，统一为 `_sanitize_for_logging`。  
- 在 `schedule_batch.log` 中改为 **INFO** 级别输出并仅在 DEBUG 时展示完整 Prompt / Negative‑Prompt，否则使用脱敏后的摘要。  
- 新增 `globally_suppress_loggers()`，把 `httpx`、`httpcore` 等噪声库的日志等级统一调至 **WARNING**，从而抑制其大量 DEBUG 信息。  

**🎯 影响范围**  
- `python/sglang/multimodal_gen/runtime/pipelines_core/schedule_batch.py`（日志行为）  
- `python/sglang/multimodal_gen/runtime/server_args.py`（导入路径、日志配置）  
- `python/sglang/multimodal_gen/runtime/utils/logging_utils.py`（新增脱敏、tensor 检测、全局抑制实现）  

**💡 关注建议**  
1. **导入与变量**：`schedule_batch.py` 新增 `import logging` 并直接使用 `logger`，但文件顶部未显式创建 `logger = init_logger(__name__)`。若已有同名全局 `logger`，保持一致；否则应在本文件同样调用 `init_logger`，避免 `NameError`。  
2. **日志级别兼容**：将原本的 `debug` 信息提升为 `info`，在非调试模式下仍会输出批次摘要。若用户希望保持安静，建议在 CLI/启动参数中提供 `--quiet` 开关，或在 `log` 方法中加入 `if server_args.quiet: return`。  
3. **脱敏实现迁移**：`_sanitize_for_logging` 现在位于 `logging_utils.py` 并被 `server_args.py`、`schedule_batch.py`、`logging_utils.py` 自己调用。确认没有循环导入（目前 `logging_utils` 不再依赖 `server_args`），否则可能导致导入错误。  
4. **全局抑制的调用时机**：`globally_suppress_loggers()` 只在模块加载时定义，实际执行需在应用入口（如 `sglang/__main__.py`）或 `init_logger` 初始化后调用，否则 `httpx`/`httpcore` 的噪声仍会出现。建议在 `init_logger` 中直接调用，或在项目入口显式 `from ...logging_utils import globally_suppress_loggers; globally_suppress_loggers()`。  
5. **单元/集成测试**：   
   - 验证在 `--log-level=INFO` 时 Prompt 只出现 `<redacted, len=…>`，在 `DEBUG` 时完整显示。  
   - 确认 `httpx`、`httpcore` 的 logger 实际被降级为 WARNING（捕获其日志并断言）。  
   - 检查 `tensor` 脱敏不会因未安装 torch 而抛异常（已通过懒加载处理）。  

总体来说，此次改动提升了用户体验，避免了不必要的网络库日志泄露，同时保护了 Prompt 隐私。只要确保 `logger` 正确实例化、全局抑制及时执行并补充相应测试，即可安全合入。

3. **小批量‑expert 优化**：仅在 `topk_ids.numel()<1024 && num_experts<=64` 时启用，建议在 CI 中覆盖这两条分支，确保 `fill_threads=256` 与共享内存大小匹配。  
4. **多 LoRA 边界**：`adapter_enabled` 与 `lora_ids` 长度必须匹配 `max_loras+1`，否则会出现越界写。可在 Python 接口加入尺寸检查。  
5. **测试可靠性**：当前测试使用随机数据，建议添加固定种子或确定性数据的对照基准，以便在 CI 失效时快速定位。  
6. **兼容性**：新增 `torch.int32` 映射后，确保所有涉及该 dtype 的调用（如 `cumsum_buffer`、`token_mask`）在 CPU 与 CUDA 侧均使用 `int32_t`，否则会出现类型不匹配。  

总体来说，此次改动为 LoRA 多适配器的 MoE 路由提供了高效的 JIT 实现，风险主要集中在显存/共享内存使用和边界检查上，建议在正式发布前完成上述验证。
