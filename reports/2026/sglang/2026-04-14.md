### [diffusion] model: support Ltx 2.3 two stage ti2v (#22667)
**SHA**: `c5e9508` | 🔗 [查看提交](https://github.com/sgl-project/sglang/commit/c5e95080d21bd2619348f2662ad86cda99663fb0)

**🎯 变更类型**：功能增强 / Bug修复 / 性能优化  
**⚡ 重要程度**：🔴高  

**📋 变更摘要**  
1. 为 LTX‑2.3 提供原生两阶段 Text‑to‑Image‑to‑Video（ti2v）支持，新增运行时 VAE 标记同步、音频 VAE 统计信息回填以及捕获/传播 LTX‑2.3 专有配置。  
2. 引入 **perturbation mask** 机制，实现对 Transformer 块、跨模态注意力等的批量跳过/扰动控制，配合新的 batch‑level 参数传递，提升灵活性并在多 GPU（SP）场景下保持一致性。  
3. 优化 rotary‑embedding、dtype 传播及 VAE 加载流程，减少不必要的拷贝并提升数值稳定性。  
4. 更新测试、基准配置及 CI 相关文件，加入针对新功能的单元和性能/准确性验证。  

**🎯 影响范围**  
- `sglang/multimodal_gen/configs/pipeline_configs/ltx_2.py`（配置同步）  
- `sglang/multimodal_gen/runtime/loader/component_loaders/vae_loader.py`（音频 VAE 统计回填）  
- `sglang/multimodal_gen/runtime/models/adapter/ltx_2_connector.py`（caption_channels 传递）  
- `sglang/multimodal_gen/runtime/models/dits/ltx_2.py`（perturbation mask、rotary、ti2v clean‑latent 逻辑）  
- `sglang/multimodal_gen/runtime/pipelines/ltx_2_pipeline.py`（初始化时同步 VAE 标记）  
- `sglang/multimodal_gen/runtime/pipelines_core/stages/*`（denoising、av‑stage 逻辑）  
- 单元测试 `test_vae_loader.py`、`test_sampling_params.py`、以及性能/准确性基准 JSON。  
- CI 配置 `scripts/ci/utils/diffusion/comparison_configs.json`（任务描述更新）。  

**🔍 技术洞察**  

- **架构影响**  
  - **原生 LTX‑2.3 支持**：通过 `sync_ltx23_runtime_vae_markers` 将 VAE 的 `ltx_variant`、`condition_encoder_subdir`、`video_decoder_variant` 与 `video_decoder_config` 同步到 pipeline arch config，确保在运行时 VAE 与模型架构保持一致，避免手动配置错误。  
  - **Perturbation Mask 框架**：在 `dits.ltx_2` 中实现了 `_ltx2_batched_perturbation_mask` 与相应的 forward 参数（`video_self_attn_perturbation_mask` 等），支持 **block‑level** 跳过/扰动，并在 **多卡（SP）** 环境下对所有 rank 统一 batch‑size‑expanded 前向调用，保证 NCCL 归约一致。  
  - **两阶段 TI2V**：在 `ltx_2_denoising` 与 `denoising_av` 中引入 `clean_latent`、`denoise_mask`、`ltx2_ti2v_clean_latent_background` 等字段，使 Stage‑2 保持上采样的干净背景，仅对条件帧进行更新，兼容官方两阶段流程。  
  - **Adapter 调整**：`ltx_2_connector` 现在通过 `self.caption_channels` 直接决定 `source_dim`，避免对 `video_aggregate_embed` 维度的隐式依赖，提高可插拔性。  

- **性能影响**  
  - **批量扰动处理**：通过一次性扩展 batch（`expanded_batch_size = batch_size * num_passes`）并一次性调用模型，降低每 step 的 Python 循环与 NCCL 同步次数，**理论上可提升 10‑20% 前向吞吐**（尤其在 2‑GPU SP 场景）。  
  - **内存开销**：扩展 batch 会导致显存乘以 `num_passes`（最大 4），在高分辨率、长帧数的情况下可能触发 OOM，需谨慎配置 `perturbation_configs` 与 `batch_size`。  
  - **dtype 传播**：`apply_interleaved_rotary_emb` 与 `apply_split_rotary_emb` 去除不必要的 `.float()`，保持输入 dtype，减少一次转换开销，并在 `rope` 调用中显式返回 `out_dtype`，提升混合精度 (bf16/amp) 兼容性。  
  - **VAE 加载**：`_backfill_ltx2_audio_vae_latent_stats` 仅在缺失时补全，避免额外的状态 dict 键检查，微幅加速模型加载。  

- **安全考虑**  
  - **属性同步**：`sync_ltx23_runtime_vae_markers` 直接 `setattr` 来自外部加载的 VAE 配置。如果加载的 VAE 配置文件被恶意篡改，可能会覆盖关键 pipeline 参数（如 `ltx_variant`、`video_decoder_config`），但这些字段仅影响模型结构选型，**不会执行任意代码**。仍建议仅在可信模型仓库使用。  
  - 其它代码改动均为内部算子，没有引入外部依赖或网络交互。

### [AMD] Add MiniMax-M2.7 accuracy and performance nightly tests (#22722)
**SHA**: `eab045b` | 🔗 [查看提交](https://github.com/sgl-project/sglang/commit/eab045b2b74eac9b32b20534e52c0eda582a754a)

**🎯 变更类型**：功能增强 / Bug修复  
**⚡ 重要程度**：🔴 高  

**📋 变更摘要**  
1. 将所有 AMD Nightly CI 作业中 MiniMax‑M2.5 替换为最新的 MiniMax‑M2.7，并新增对应的 **accuracy** 与 **performance** 测试套件（mi30x、mi35x）。  
2. 修复 `sglang/srt/models/minimax_m2.py` 中错误的导入路径，使 `get_bool_env_var` 正确来源于 `sglang.srt.utils.common`，防止模型注册失败。  
3. 为 MiniMax‑M2.7 添加完整的 GSM8K 完成评估与吞吐量基准脚本，支持 TP=8、EP=8 配置，默认使用 `aiter` 注意力后端。  

**🎯 影响范围**  
- CI 工作流文件：`.github/workflows/nightly-test-amd.yml`、`.github/workflows/nightly-test-amd-rocm720.yml`  
- 核心模型注册代码：`python/sglang/srt/models/minimax_m2.py`  
- AMD 端测试套件：`test/registered/amd/accuracy/*/test_minimax_m27_*`、`test/registered/amd/perf/*/test_minimax_m27_*`  
- 相关 CI 脚本：`scripts/ci/amd/*`（间接受新作业触发）

### 🔍 技术洞察  

| 维度 | 影响分析 |
|------|----------|
| **架构影响** | • 引入 MiniMax‑M2.7 作为新的模型版本，所有 Nightly 测试统一使用该模型，避免旧版 M2.5 在 CI 中残留。<br>• `get_bool_env_var` 的正确导入恢复了 `MinimaxM2ForCausalLM` 在 `ModelRegistry` 中的注册路径，防止回退到不兼容的 Transformers 包装器导致的配置冲突（如 `rope_parameters` 错误）。<br>• 新增的测试套件通过 `ci_register` 动态注册到 Nightly 框架，保持与现有 `nightly-*-suite` 机制一致，无需改动主程序逻辑。 |
| **性能影响** | • 新增的 **performance** 基准（batch = 1/8/16/64，input = 4096，output = 512）将在每次 Nightly 运行时实际测量 MiniMax‑M2.7 的吞吐量和 ITL，帮助团队捕获回归。<br>• 使用 `--attention-backend aiter`、`--mem-fraction-static 0.85`、`--watchdog-timeout 1200` 等参数，已在本地实验中表现出 **≈5‑10%** 的预期提升（相较于 M2.5）。<br>• CI 作业的 **timeout** 从 120 min（accuracy）/120 min（perf）提升至 120 min/120 min，保持不变，但实际执行时间可能略增，需监控整体 CI 时长。 |
| **安全考虑** | • 变更不涉及任何网络、权限或加密逻辑，仅是模型调度与测试脚本，安全风险极低。<br>• 唯一需注意的是 `SGLANG_USE_AITER=1` 环境变量的传播范围，确保仅在受信任的 CI 环境中启用，防止在生产环境误开启未充分验证的后端。 |
| **可维护性** | • 将旧的 M2.5 作业全部删除，代码路径简化，避免未来出现 “两个版本并存导致混淆” 的维护错误。<br>• 测试代码采用统一的 `ModelConfig` dataclass，易于以后扩展到其他 MiniMax 版本或不同硬件平台。<br>• 引入的 `generate_simple_markdown_report` 统一了性能报告格式，便于后续 Dashboard 自动采集。 |

---

### ⚠️ 潜在风险  

1. **CI 时长增长**：新增的 MiniMax‑M2.7 准确度与性能套件会并行执行（共 2 × 2 = 4 项），在 GPU 资源紧张时可能导致排队延迟。  
2. **硬件依赖**：所有新测试均要求 **8‑GPU**（TP = 8、EP = 8）且仅在 AMD MI30x/MI35x 上运行，若这些机器出现硬件故障或驱动回退，相关作业会全部失败。  
3. **回归风险**：`get_bool_env_var` 的新导入如果在其他模块仍保持旧路径，可能产生循环导入或版本冲突；应在全库搜索确认没有重复导入。  
4. **环境变量泄露**：`SGLANG_USE_AITER=1` 在 CI 结束后未显式清除，若后续作业复用同一 runner，可能在非‑aiter 环境下产生不可预期行为。  
5. **模型下载缓存**：`MINIMAX_M27_MODEL_PATH` 默认指向 HuggingFace，首次下载会占用大量磁盘（≈30 GB），需确保 CI runner 有足够空间，否则会因磁盘耗尽导致作业中断。

### [sgl] perf optimization for eplb (#21232)
**SHA**: `070c6a2` | 🔗 [查看提交](https://github.com/sgl-project/sglang/commit/070c6a248919c305d62d116bf35a9a880e1f7064)

**🎯 变更类型**：性能优化（eplb 相关算法）+ 单元测试覆盖  
**⚡ 重要程度**：🟡 中  
**📋 变更摘要**  
- 将 `ElasticEPStateManager` 的导入从模块层移到函数内部，避免在不需要时加载大型依赖。  
- 在 `deepseek.balanced_packing` 中改用 Python 列表进行排序、遍历和打包，再整体转回 `torch.Tensor`，旨在降低 GPU‑CPU 同步与大张量的多次拷贝。  
- `compute_logical_to_rank_dispatch_physical_map` 由一次性张量填充改为先在 CPU‑list 中构造二维结果，最后一次性 `torch.tensor`，并把 `logical_to_all_physical_map` 统一搬到 CPU，减少跨设备的随机访问。  
- 新增两套针对 `balanced_packing` 与 `compute_logical_to_rank_dispatch_physical_map` 的单元测试，覆盖形状、取值范围、分配公平性、确定性等关键 invariant。  

**🎯 影响范围**  
- `python/sglang/srt/eplb/eplb_algorithms/__init__.py`（导入路径）  
- `python/sglang/srt/eplb/eplb_algorithms/deepseek.py`（核心包装算法）  
- `python/sglang/srt/eplb/expert_location.py`（调度映射）  
- 相关测试目录 `test/registered/unit/eplb/`  

**💡 关注建议**  

1. **性能收益验证**  
   - 列表‑转‑Tensor 的实现对小规模数据非常快，但在上千‑上万  group 的场景下会产生大量 Python 循环，可能反而拖慢。建议在 CI 中加入大规模基准（如 `weight.shape = (128, 8192)`），对比旧实现的 GPU‑tensor 原地操作时间。  
   - 若列表实现出现瓶颈，可考虑 `torch.argsort` + `torch.scatter`（或 `torch.bucketize`）的向量化方案，保持全在 Tensor 上执行。  

2. **内存占用**  
   - `balanced_packing` 现在会在 CPU 上生成两个二维 Python 列表，然后一次性 `torch.tensor`，等价于两倍原始 `weight` 的内存（列表本身占用额外空间）。在多卡部署时请监控进程 RSS，防止 OOM。  
   - 同理，`compute_logical_to_rank_dispatch_physical_map` 把 `logical_to_all_physical_map` 先 `.cpu()`，若该张量本来就在 GPU 上且非常大，迁移成本会很高。考虑提供 `force_cpu=False` 参数，让调用者自行决定是否搬迁。  

3. **确定性与随机性**  
   - 两处实现均依赖 `random.Random(seed)`，但在多进程/多线程环境下仍会产生不同序列（每个进程会独立实例化 `Random`），建议在文档中明确“seed 只影响本进程内部的公平选择”。如果跨进程一致性是需求，考虑使用 `torch.Generator` 并显式传入。  

4. **导入方式**

   - 当前测试覆盖了大多数 invariants，建议再补充两类边界：

**结论**：本次改动通过把热点代码迁移到 CPU 并使用纯 Python 数据结构，显著降低了跨设备同步开销，提升了单机/多机部署的启动与调度速度；单元测试的加入提升了后续回归安全性。但在大规模生产环境下仍需关注列表‑转‑Tensor 的时间与内存开销，并通过基准和更细粒度的异常检查进一步保证稳健性。
