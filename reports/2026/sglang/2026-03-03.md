### feat(gdn): add FlashInfer K-last SSM layout support for GDN prefill and decode for Hopper (#18361)
**SHA**: `c6377bb` | 🔗 [查看提交](https://github.com/sgl-project/sglang/commit/c6377bbbca276c0e8f63d0d978c69ceb095aa10a)

**🎯 变更类型**：功能增强 / 性能优化  
**⚡ 重要程度**：🔴高  
**📋 变更摘要**：在 GDN（Gated Delta Network）线性注意力实现中新增 FlashInfer 后端，支持 K‑last SSM 布局的预填（prefill）和解码（decode），并在服务器参数和枚举中加入对应选项。此实现基于 SM90+ GPU 的 CUTLASS‑FlashInfer 内核，可显著提升解码吞吐并降低预填时的 CPU 开销。  

**🎯 影响范围**：  
- `sglang.srt.layers.attention.linear.gdn_backend.GDNBackend`（调度器）  
- 新增 `sglang.srt.layers.attention.linear.kernels.gdn_flashinfer.FlashInferGDNKernel`  
- 枚举 `LinearAttnKernelBackend` 与服务器参数 `LINEAR_ATTN_KERNEL_BACKEND_CHOICES`  
- 相关的调度调用路径（prefill、decode、target_verify）  

**🔍 技术洞察**  

- **架构影响**  
  - 在原有的 Triton / CuteDSL 双后端架构上新增第三套实现，保持统一的 `LinearAttnKernelBase` 接口。  
  - 通过惰性导入 `_get_flashinfer_gdn_kernels()`，在运行时判断硬件（CUDA、SM 9.0+）与库是否可用，避免在不支持的环境中加载失败。  
  - 采用 K‑last SSM 布局（[pool, HV, V, K]）统一了 decode 与 prefill 的内存访问模式，降低了 gather/scatter 开销。  
  - `verify_kernel` 复用同一实例，避免额外的资源分配。  

- **性能影响**  
  - 预填阶段：FlashInfer 0.6.4 通过缓存 `cudaGetDeviceProperties`，可省去约 80 ms 的 CPU 开销，提升约 50% 的吞吐。  
  - 解码阶段：K‑last 布局配合 FlashInfer 的 pretranspose kernel，预计相较于原 CutedSL 实现降低 7‑9% 的解码回归（即提升 7‑9% 的吞吐）。  
  - 采用 `torch.float32` 进行状态复制与计算，避免了 Triton 中的跨 dtype 转换成本。  
  - 仅在 SM90+（A100 40GB、H100等）上激活，低于该算力的机器仍使用原有后端，保持兼容。  

- **安全考虑**  
  - 新增对外部依赖 `flashinfer` 的动态加载，若库被篡改或使用了不受信任的二进制，可能引入供应链风险。建议在发行包中明确声明对应的 PyPI/源码来源，并在 CI 中校验 its hash。  
  - 参数校验已在 `gdn_backend.py` 中加入 CUDA 环境检查，避免在 CPU 环境误调用导致崩溃。  

**⚠️ 潜在风险**  

1. **硬件兼容性**：只在 SM90 及以上 GPU 可用，若用户误选 `flashinfer` 后端在不支持的硬件上，会在初始化时报错，需提供友好提示。  
2. **依赖管理**：`flashinfer` 不是项目默认依赖，缺失或版本不匹配会导致后端不可用，可能出现 “Unsupported GDN decode backend” 错误。  
3. **状态布局变更**：K‑last 布局与已有 V‑last 布局不兼容，若在同一进程中混用不同后端（例如 prefill 用 Triton，decode 用 FlashInfer），可能导致状态错位。当前代码通过统一 `self.verify_kernel` 规避，但仍需在用户文档中强调“一致后端”使用。  
4. **数值差异**：FlashInfer 使用 `float32` 进行状态更新，可能与 Triton 中的 `float16`/`bfloat16` 浮点精度产生细微差异，影响极端数值的稳定性。  
5. **内存占用**：K‑last 布局将 V 放在前面，可能导致在某些形状下额外的转置或拷贝，需监控显存峰值。  

**💡 关注建议**  

- **兼容性检测**：在 CLI/配置层加入提前检测函数，提示用户当前硬件是否满足 FlashInfer（SM≥9）以及是否已正确安装 `flashinfer`。  
- **回退机制**：若加载 FlashInfer 失败，自动回退至 Triton 或 CutedSL，并在日志中给出明确原因。  
- **文档更新**：说明 FlashInfer 后端的硬件要求、安装步骤（`pip install -e ".[cutlass]"`）、以及在 `--linear-attn-decode-backend` 与 `--linear-attn-prefill-backend` 中保持一致的使用方式。  
- **CI/测试**：在 CI 中加入针对 SM90+ 的模拟测试（如使用 NVIDIA 的 container）以及针对缺失 flashinfer 库的负面测试，确保报错信息友好。  
- **安全审计**：对 `flashinfer` 的导入路径进行 hash 验证或签名检查，防止恶意篡改。  
- **性能基准**：提供官方基准报告（prefill/decode 吞吐、显存占用）对比 Triton 与 FlashInfer，以帮助用户做决策。  

---  

此次改动为 GDN 线性注意力引入了高效的硬件加速路径，若配合正确的硬件与依赖，可显著提升模型推理速度；同时，需要在发布时做好兼容性、依赖管理与文档指引，以降低因硬件或库缺失导致的部署风险。

### [model gateway][0/N] router EPD support: add encoder grpc server backend support (#16552)

**🎯 变更类型**：功能增强  
**⚡ 重要程度**：🔴高  

**📋 变更摘要**  
1. 为 SGLang 的 EPD（Encode‑Prefill‑Decode）模式新增 **gRPC Encoder** 后端，支持 `--encoder‑only --grpc‑mode` 启动编码器、通过 `SGLANG_ENCODER_MM_RECEIVER_MODE=grpc` 在 Prefill 端使用 gRPC 接收器。  
2. 新增 `encode_grpc_server.py` 实现 gRPC 编码服务（健康检查、反射、Encode / Send / SchedulerReceiveUrl RPC），并在 `launch_server.py` 中根据 `--encoder‑only` 与 `--grpc‑mode` 自动切换。  
3. 重构 `encode_receiver.py`：抽象化接收器基类，分别实现 **HTTP** 与 **gRPC** 两种实现；提供工厂 `create_mm_receiver` 供调度器/Tokenizer 管理器创建。  
4. 通过环境变量 `SGLANG_ENCODER_MM_RECEIVER_MODE`（默认 http）和 `SGLANG_ENCODER_GRPC_TIMEOUT_SECS` 控制接收方式与超时。  
5. 文档、依赖（`smg-grpc-proto` 升级）以及新增/扩展单元测试覆盖 gRPC 编码器的完整链路。

**🎯 影响范围**

- `sglang/srt/disaggregation/encode_grpc_server.py`（全新 gRPC 编码服务）

### ⚠️ 潜在风险

1. **网络安全**：使用明文 gRPC，跨机房或云环境下可能泄露多模态图像路径或嵌入。  
2. **资源泄露**：`serve_grpc_encoder` 启动多个子进程（`launch_encoder`）及 ZMQ sockets；若异常退出未及时关闭，可能导致 **文件描述符泄漏**。  
3. **超时/阻塞**：`Encode` RPC 中调用 `self.encoder.encode`（可能是阻塞的 CUDA 推理），若单次推理时间超过 `SGLANG_ENCODER_GRPC_TIMEOUT_SECS` 会触发 gRPC 超时，返回 INTERNAL 错误。需确保超时配置合理。  
4. **版本兼容**：`smg-grpc-proto` 升级可能破坏已有的 protobuf 消息定义，导致旧客户端（如第三方调用）兼容性问题。  
5. **并发竞争**：`EmbeddingData` 在 gRPC 版仍使用 ZMQ 进行后续 `Send`，可能出现 **双写竞争**（Encoder Send 与 Scheduler ReceiveUrl 并行），需确认 `self.embeddings_buffer` 及 `self.embeddings_engine` 的线程安全性。  
6. **配置误差**：用户若误把 `encoder-urls` 配置为 `http://` 而 `SGLANG_ENCODER_MM_RECEIVER_MODE=grpc`，会在服务器启动时报 `ValueError`，但错误信息会在子进程日志中，可能不易发现。  
7. **负载不均衡**：`encode_idx` 随机分配图像到多个 Encoder 实例；在 gRPC 场景若某 Encoder 机器负载突增，可能成为 **瓶颈**。

### feat: Add FP8 KV cache support for Triton attention backend (#18882)

**⚡ 重要程度**：🔴 高  

**📋 变更摘要**  
本次提交为 Triton 注意力后端新增对 FP8（`fp8_e4m3`）KV 缓存的支持，核心改动包括：  
1. 在 `triton_backend.py` 中引入 `AttentionArch` 判断，区分 MLA 与普通注意力，新增 KV 缓存保存时的 scale 参数传递与反量化逻辑。  
2. 大幅改写 Triton kernel（`decode_attention.py`、`extend_attention.py`），在 `sm_scale` 基础上乘以 `k_scale`（键的反量化系数），并在 Softmax‑V 归约阶段乘以 `v_scale`（值的反量化系数）。  
3. 完善单元测试，新增 FP8 KV 缓存端到端评估 `test_fp8kv_triton.py`，验证在 GSM8K 上的推理准确率。  

**🎯 影响范围**  
- `python/sglang/srt/layers/attention/triton_backend.py`（后端调度、缓存读写）  
- `python/sglang/srt/layers/attention/triton_ops/*`（所有 Triton 注意力 kernel）  
- 相关配置 `sglang/srt/configs/model_config.py`（新增 `AttentionArch.MLA`）  
- 测试套件 `test/registered/attention/*`、`test/registered/quant/test_fp8kv_triton.py`  

**🔍 技术洞察**  

| 维度 | 影响分析 |
|------|----------|
| **架构影响** | - 在 `AttentionBackend` 初始化阶段加入 `use_mla` 标识，决定是否走量化路径；<br>- KV 缓存的写入接口从 `set_kv_buffer(layer, loc, k, v)` 扩展为可接受 `k_scale / v_scale`，使量化缓存与普通缓存统一管理；<br>- Triton kernel 参数签名统一加入 `k_scale`、`v_scale`，对外层调用形成向后兼容的可选参数。 |
| **性能影响** | - 量化 KV（FP8）显著降低显存占用与带宽需求，理论上可提升大模型推理吞吐；<br>- 额外的反量化乘法在 kernel 中仅为标量乘法，几乎无额外算力开销；<br>- 复制 `k.clone()`、`v.clone()`（仅在非量化路径）可能轻微增加内存搬迁，影响极限批次下的延迟。 |
| **安全考虑** | - 反量化系数来源于模型配置，若配置被篡改或未正确加载可能导致数值溢出/NaN，需在 `set_kv_buffer` 前加入 sanity check；<br>- 新增的 `AttentionArch.MLA` 判断在非‑MLA 模式下仍然走量化路径，确保不泄露内部实现细节。 |
| **可维护性** | - Triton kernel 参数列表显著增多（`sm_scale_withk`、`k_scale`、`v_scale`），代码可读性下降；建议在 `triton_ops` 中抽象出统一的 `ScaleContext` 结构体，减小签名膨胀。 |
| **兼容性** | - 对已有的非‑FP8、非‑MLA 场景保持向后兼容（默认 `k_scale = v_scale = 1.0`，`use_mla=False`），但 API 变更可能导致第三方插件未更新而出现 `TypeError`。 |

**⚠️ 潜在风险**  
1. **数值不稳定**：FP8 量化本身误差较大，反量化后仍可能在 Softmax 前产生溢出或 underflow，尤其在长序列或极端 logits 时；<br>2. **Kernel 兼容性**：部分旧版 Triton（< 2.2）在 `tl.dot` 与标量乘法混用可能触发编译错误；<br>3. **MLA 与量化冲突**：当前实现中 `use_mla` 与量化互斥，若未来需要 MLA+FP8 组合，需要额外路径；<br>4. **测试覆盖不足**：仅在单机 1 GPU 环境下跑了 GSM8K，缺少多卡/分布式、不同模型尺寸的回归验证。  

**💡 关注建议**  
- **数值监控**：在 KV 写入前后记录 `k_scale`、`v_scale` 与实际张量最大/最小值，加入断言防止异常放大；  
- **文档与示例**：在 `README` 与配置说明中明确 `--kv-cache-dtype fp8_e4m3` 仅在 Triton 后端且非‑MLA 场景可用；提供 `sglang.srt.configs.model_config.AttentionArch` 的使用示例；  
- **API 兼容层**：为 `set_kv_buffer` 与 kernel 调用提供包装函数，使旧代码仍可调用两参数签名，避免因参数缺失导致崩溃；  
- **性能基准**：补充显存占用、吞吐量与延迟对比（FP8 vs FP16 vs BF16），并在 CI 中加入基准阈值检查；  
- **回归测试**：在多卡/分布式模式、不同序列长度、不同模型（7B、13B、70B）上运行自动化静态检查，确保量化路径不引入隐藏 bug；  
- **安全审计**：对模型配置加载路径加入完整性校验，防止恶意修改 `k_scale`/`v_scale` 施加数值攻击。  

---  
**总体结论**：此提交为 SGLang 引入 FP8 KV 缓存的关键功能，提升了大模型推理的显存效率并保持相对准确率（>70% GSM8K）。改动涉及后端调度、底层 Triton kernel 以及测试体系，风险主要集中在数值稳定性与跨版本 Triton 兼容性。通过加强文档、增加安全校验以及完善多环境回归测试，可将风险降至可接受水平。
