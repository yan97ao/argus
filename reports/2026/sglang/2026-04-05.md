### feat: CI auto-bisect workflow for automated regression analysis (#22119)
**SHA**: `8cbeacd` | 🔗 [查看提交](https://github.com/sgl-project/sglang/commit/8cbeacd783dae2191a43c3d74871adb7ae890874)

**🎯 变更类型**：功能增强（feat）  
**⚡ 重要程度**：🔴高  

**📋 变更摘要**  
- 新增 GitHub Actions 工作流 `ci-auto-bisect.yml`，在 PR‑Test（Nvidia 定时）完成后自动运行回归分析脚本。  
- 添加 `scripts/ci_monitor/ci_auto_bisect.py`：通过 GitHub API 拉取最近的定时 CI 运行、定位持续失效的作业/测试、抽取错误片段并调用 Anthropic Claude‑sonnet‑4‑5 进行根因分类，最终生成 `bisect_results.json`。  
- 添加 `scripts/ci_monitor/post_bisect_to_slack.py`：把分析结果以结构化的摘要+线程详情发送到公司 Slack 频道。  

**🎯 影响范围**  
- CI 基础设施（`.github/workflows/ci-auto-bisect.yml`）  
- `scripts/ci_monitor/` 目录下的全新监控/分析脚本（Python）  
- 依赖：`requests`, `anthropic`, `slack_sdk`（在 CI Runner 中安装）  
- 与 GitHub Actions API、Anthropic Claude API、Slack API 的交互  

**🔍 技术洞察**  

| 维度 | 影响分析 |
|------|----------|
| **架构影响** | - 采用 *独立的* 并发组 `ci-auto-bisect`，与主 PR‑Test 完全隔离，避免对已有 CI 结果产生阻塞。<br>- 脚本自行完成 GitHub API 分页、日志抓取、Git 本地 `git log` 分析，属于 **外部监控层**，对业务代码 (`python/sglang/…`) 没有侵入式改动。<br>- 通过加载仓库根路径下的 `.claude/skills/sglang-bisect-ci-regression/SKILL.md` 让分类规则与业务保持同步，提升可维护性。 |
| **性能影响** | - 工作流最长 30 min，实际运行时间取决于失败作业数（约 1–5 min/作业）以及 Claude 调用（每次 ~1‑2 s）。<br>- 额外的 GitHub API 调用约 20‑30 次（列表、分页、日志），在 GitHub Rate‑Limit（5000 次/小时）内安全，但在高频 CI 场景下需监控。<br>- CI Runner 需要 `fetch-depth: 0` 完整历史，导致克隆时间略增（≈30 s‑1 min），对 CI 成本有轻微提升。 |
| **安全考虑** | - **凭证泄露风险**：工作流以 secret `GH_PAT_FOR_NIGHTLY_CI_DATA`、`ANTHROPIC_API_KEY`、`SGLANG_DIFFUSION_SLACK_TOKEN` 注入环境变量，随后直接作为脚本参数传递。若日志误输出变量值，可能泄露。<br>- **日志数据外泄**：脚本把 CI 作业日志片段（最多 2 MB）发送给 Claude 进行分析，日志中可能包含临时凭证、调试信息或敏感测试数据，需要在抽取 `error_signature` 前脱敏（例如移除 `aws_`, `secret_` 等关键词）。<br>- **供应链风险**：新加入的第三方库未锁定版本（`pip install requests anthropic slack_sdk`），潜在依赖漏洞。建议在 `requirements.txt` 中 pin 版本并使用 Dependabot 进行监控。 |
| **可维护性** | - 脚本采用 `dataclass`、分层函数（API、日志、错误抽取、Claude prompt）结构清晰，易于单元测试与后期扩展。<br>- `load_bisect_skill()` 对 SKILL.md 必要章节做校验，若文档改动会抛出 `SkillLoadError`，确保 CI 失效时能及时告警。<br>- 结果统一写入 `bisect_results.json` 并通过 GitHub Step‑Summary 与 Slack 双渠道报告，提升可观测性。 |

**⚠️ 潜在风险**  
1. **凭证误泄**：`anthropic_api_key`、`github_token`、`slack_token` 若在脚本异常退出或 `set -x` 开启的情形下被写入日志，会暴露给外部。  
2. **日志隐私**：未对 CI 日志做脱敏直接发送给 Claude，可能把内部测试数据、临时 secret 暴露给第三方模型。  
3. **运行时错误导致 CI 反馈阻塞**：如果 `ci-auto-bisect` 步骤因网络/Rate‑Limit 频繁重试未能结束，会占用 `ci-auto-bisect` 并发组，导致后续自动分析被取消。  
4. **依赖未锁定**：`pip install anthropic` 等使用默认最新版本，未来升级可能出现不兼容或安全漏洞。  
5. **Python 3.14 兼容性**：GitHub Actions runner 默认提供 3.12，3.14 尚未正式发布，可能导致工作流在某些 runner 上失败。  
6. **GitHub API Rate‑Limit**：在高并发 CI（多分支、频繁 schedule）下，累计 API 调用可能接近上限，导致后续步骤被 throttled。

### Unify think_end_id to model_config as single source of truth (#22148)
**SHA**: `df9c831` | 🔗 [查看提交](https://github.com/sgl-project/sglang/commit/df9c831ab80c78495142b4dbcb1dfa537c9e1c73)

**变更概览**  
本次提交将 `think_end_id` 从 `Tokenizer` 与 `Scheduler._think_end_id` 两处分散存储，统一迁移到 `ModelConfig`，形成唯一可信源。为此修改了 `create_grammar_backend`、`GrammarManager`、`Scheduler.init_tokenizer` 与 `SchedulerOutputProcessorMixin` 四个核心入口的参数与读取路径。

**影响范围**  
- `sglang/srt/constrained/base_grammar_backend.py`：新增 `think_end_id` 参数并在推理解析器分支传入。  
- `sglang/srt/constrained/grammar_manager.py`：在实例化时从 `scheduler.model_config.think_end_id` 传递。  
- `sglang/srt/managers/scheduler.py`：初始化时不再给 `tokenizer` 打补丁，而是直接写入 `model_config.think_end_id`。  
- `sglang/srt/managers/scheduler_output_processor_mixin.py`：使用 `self.model_config.think_end_id` 替代原来的 `_think_end_id`。

**核心意义**  
- **单一真相**：所有模块统一读取 `model_config.think_end_id`，消除属性不一致风险。  
- **简化维护**：移除对 `Tokenizer.think_end_id` 的临时挂载，降低后期改动成本。

**潜在风险**  
1. 其他未被捕获的代码（如外部插件或旧测试）仍可能访问 `tokenizer.think_end_id`，会导致属性错误。  
2. `ModelConfig` 必须保持可写状态；若在后续实现中改为不可变，需要确保赋值逻辑仍可执行。  
3. `Scheduler.init_tokenizer` 现在在编码完 `think_end_token` 后直接写入 `model_config`，若 `model_config` 在其它地方被提前冻结，可能抛出异常。

**建议**  
- 在项目文档和代码注释中明确 `think_end_id` 的唯一来源。  
- 检查全仓库是否还有对 `tokenizer.think_end_id` 的引用并同步改为 `model_config.think_end_id`。  
- 为 `ModelConfig` 添加类型注解 `think_end_id: Optional[int] = None`，并在配置加载阶段提供默认值。  
- 增加单元测试，验证在开启/关闭 reasoning_parser 时 `model_config.think_end_id` 正确填充且 `GrammarBackend` 能收到该值。  

通过上述检查与测试，可确保此“单一事实源”改动在功能上平滑迁移且不引入回归。

### Add dump_metric to MMMU, lm-eval, and NeMo Skills eval paths (#22147)
**SHA**: `aeff9fb` | 🔗 [查看提交](https://github.com/sgl-project/sglang/commit/aeff9fb7c1cf5204e47d48707ca4a024d8cf015b)

**🎯 变更类型**：功能增强  
**⚡ 重要程度**：🟡 中  

**📋 变更摘要**  
在 MMMU、lm‑eval、NeMo‑Skills 三条评测路径中新增 `dump_metric` 调用，统一把每轮评测的得分写入度量日志，便于 CI / 报表统计。  

**🎯 影响范围**  
- `python/sglang/test/accuracy_test_runner.py`（NeMo‑Skills）  
- `python/sglang/test/kits/lm_eval_kit.py`（lm‑eval）  
- `python/sglang/test/kits/mmmu_vlm_kit.py`（MMMU）  
- 相关 import：`sglang.test.test_utils.dump_metric`  

**💡 关注建议**  

1️⃣ **函数可用性**：确认 `dump_metric` 已在 `sglang/test/test_utils.py` 中实现且在所有测试环境（本地、GitHub Action）可导入；若有新依赖（如 `prometheus_client`、文件写入路径），需在 `requirements.txt` 或 CI 脚本中同步。  

2️⃣ **异常容错**：目前直接调用 `dump_metric`，若日志目录不可写或函数抛异常，将导致评测失败。建议在调用前做 `try/except` 包裹，记录警告而不中断主流程。  

3️⃣ **并发安全**：CI 中可能并行执行多个评测实例，确保 `dump_metric` 对同一文件的写入是原子操作（如使用文件锁或追加模式），防止竞争造成文件损坏。  

4️⃣ **标签一致性**：`labels` 中的键名在三处稍有差异（`api`、`eval`、`model`），建议统一命名规范，例如统一使用 `task` 替代 `eval`，或在文档中明确约定，以利后续聚合查询。  

5️⃣ **文档与 CI**：更新 README/CHANGELOG，说明新增度量输出位置（如 `metrics.jsonl`），并在 CI 步骤中添加对该文件的归档或上传，以便 CI 可视化。  

6️⃣ **回归测试**：新增的 I/O 不应显著影响运行时间，但建议在本地跑一遍完整评估，确认不会出现 “Permission denied” 或 “File not found” 等副作用。  

总体来看，此次改动提升了评测可观测性，对功能无侵入性影响，关注上述细节即可平稳上线。

### Isolate spec V1 path in decode post-processing (#22146)
**SHA**: `cd2d45e` | 🔗 [查看提交](https://github.com/sgl-project/sglang/commit/cd2d45e22085045a5e9cd14666be7b0e96af601d)

- `python/sglang/srt/managers/scheduler_output_processor_mixin.py`（核心后处理逻辑）

**💡 关注建议**

2. **单元/集成测试**：新增或扩展针对 Spec‑V1、Spec‑V2、普通解码三条路径的回归测试，特别是 KV‑cache 回收、异步 offload、日志统计是否保持一致。

4. **文档更新**：在配置说明和 Spec‑V1 使用指南中标明 `think_end_id` 的来源及其作用，便于外部用户了解新增的 reasoning_parser 依赖。  
5. **代码审阅**：关注 `_handle_finished_req` 中的资源释放路径是否与原实现完全等价，尤其是 `decode_offload_manager.offload_kv_cache` 的返回值处理，以防出现缓存泄漏或提前释放的潜在 bug。  

总体而言，此次抽离提升了 Spec‑V1 与其它解码路径的独立性，降低了后续功能叠加导致的耦合风险，只要完成上述兼容性与回归测试，即可安全上线。

### [diffusion] Add is_float64_supported to Platform (#22112)
**SHA**: `10b18b8` | 🔗 [查看提交](https://github.com/sgl-project/sglang/commit/10b18b8b2915c96f79aac6c60802b20f4417dcfa)

**变更概述**  
本次 PR 引入 `Platform.is_float64_supported()` 接口，用以统一判断当前硬件是否宜使用 `float64` 精度；在 `MpsPlatform`、`MusaPlatform` 上显式返回 `False`（其余平台默认 `True`）。随后在 DIT/Flux 系列模型、VAE 相关层以及 `linear` 层的注释和 dtype‑cast 逻辑中，将原先的 `is_mps()/is_musa()` 检查全部替换为 `is_float64_supported()`（或 `is_amp_supported()`）判断。

**影响范围**  
- `python/sglang/multimodal_gen/runtime/platforms/*`：平台基类与子类实现。  
- 多个模型文件：`causal_wanvideo.py、flux.py、flux_2.py、wanvideo.py`，以及 VAE 并行实现 `wan_common_utils.py、wan_dist_utils.py`。  
- `layers/linear.py` 中对 bias 的 dtype 注释更新。  

**关键风险**  
1. **平台默认行为**：基类默认返回 `True`，若后续出现不支持 `float64` 的新平台（如部分老旧 CUDA/ROCm），容易误用高精度导致显存激增或性能下降。  
2. **AMP 与 float64 并存**：在 `linear` 和 VAE 代码中改为 `if current_platform.is_amp_supported() else x.to(self.weight.dtype)`，需要确认在 MPS（AMP 不支持且 float64 也不支持）路径下仍能得到正确的 `float32` 类型，避免出现 `x.dtype == torch.float64` 的意外。  
3. **向后兼容**：旧代码仍可能直接调用 `is_mps()`/`is_musa()`，但大多数已改为新接口，需确保没有遗漏导致未覆盖的分支。  

**建议**
