### [CP] 1/N: Support MLA Prefill Context Parallel (#23292)
**SHA**: `b0ce16d` | 🔗 [查看提交](https://github.com/sgl-project/sglang/commit/b0ce16d0c577e127b3f7b82ada1739277d988687)

**🎯 变更类型**：功能增强 / 架构变更  
**⚡ 重要程度**：🔴高  

**📋 变更摘要**  
- 为 MLA（Multi‑Linear‑Attention）后端实现了 **Prefill Context‑Parallel（CP）** 支持，新增 `--enable-prefill-context-parallel` 旗标以及对应的 `mla_use_prefill_cp`、`is_mla_prefill_cp_enabled` 等工具函数。  
- 在 FlashAttention、LayerCommunicator、模型层、模型 Runner、ServerArgs 等核心模块加入对 MLA‑CP 的判定、元数据准备、KV‑cache 重建、token‑count 调整等逻辑，保持与已有 DSA‑CP（DeepSeek‑DSA）代码路径兼容。  
- 修正了 CP 场景下 KV‑cache 边界的 `page_table` 扩展、`metadata.max_seq_len_k` 增宽、前缀长度计算等细节，防止因 padding 导致的越界读取。  
- 新增一系列单元/集成测试（DeepSeek‑V3‑MLA‑CP、DeepSeek‑V4‑DSA‑CP、FA3‑parity 等）以验证数值一致性和推理正确性。  

**🎯 影响范围**  
- `sglang/srt/layers/attention/flashattention_backend.py`（FA3 后端）  
- `sglang/srt/layers/communicator*.py`（DSA 与普通 Communicator）  
- `sglang/srt/layers/utils/cp_utils.py`（CP 判定、元数据准备、padding 计算）  
- `sglang/srt/model_executor/*`（CudaGraphRunner、ModelRunner、各模型实现）  
- `sglang/srt/server_args.py`（命令行旗标、默认值、互斥检查）  
- 多个模型实现（DeepSeek‑V3/V4、Mistral‑Large‑3‑Eagle、Qwen3‑MoE）  
- 新增/更新的 CI/UT（`test/registered/cp/*`、`test/registered/kernels/*`）

### 🔍 技术洞察  

| 维度 | 影响分析 |
|------|----------|
| **架构影响** | 1. **新增 CP 路径**：在 `FlashAttentionBackend._fa_cp_attn` 中加入 MLA‑CP 分支 (`_mla_cp_attn` + `cp_attn_forward_extend`)；在 `LayerCommunicator` 中加入 `mla_use_prefill_cp` 判定，使得 **DSA‑CP 与 MLA‑CP 两套实现共存**。<br>2. **元数据统一**：`prepare_context_parallel_metadata` 现在接受 `extend_lens` 并统一计算 `prefix_len`，从而为 MLA‑CP 与 DSA‑CP 共享同一套 `ContextParallelMetadata` 结构。<br>3. **全局 Flag**：`ServerArgs.enable_prefill_cp` 被细分为 **DSA** 与 **MLA** 两个子功能，互斥检查放在 `ServerArgs._handle_context_parallelism` 中。 |
| **性能影响** | - **Prefill 加速**：MLA‑CP 将 prefill 的 **extend** token 划分至多个 attention‑TP rank**（`attn_cp_size = tp / dp`），可实现近似线性放大 prefill throughput，尤其在大模型（DeepSeek‑V3‑R1、Kimi‑K2.5）上配合 `fa3` 后端可获得 2–3× 加速。<br>- **内存占用**：为防止 FA3 越界读取，需要在 `metadata.max_seq_len_k` 上做 `pad_delta` 扩展，导致 KV‑cache 额外分配约 `pad_delta * head_dim * tp` 的缓存。<br>- **额外 All‑gather**：MLA‑CP 仍会在 `rebuild_cp_kv_cache` 前进行一次全局 All‑gather（rank‑local KV → latent KV pool），与 DSA‑CP 的 All‑gather 行为相同，开销与 CP size 成正比。 |
| **安全考虑** | - **越界风险**：若 `metadata.max_seq_len_k` 未正确扩宽，FA3 的因果读可能访问未初始化的 KV slot，导致非法内存访问或数值异常。已通过代码注释和单元测试覆盖此路径。<br>- **配置冲突**：新增 `--enable-prefill-context-parallel` 与原 `--enable-dsa-prefill-context-parallel` 互斥，若用户误同时开启可能触发不一致的 CP 布局；在 `ServerArgs._handle_context_parallelism` 增加显式校验并抛异常。<br>- **多批次 CP**：目前仅支持 batch=1（`forward_batch.batch_size == 1`）; 若后续放宽，需要严谨的边界检查，防止跨 batch KV 混用导致信息泄露。 |
| **可维护性** | - **代码重复**：多处出现 `if dsa_use_prefill_cp(... ) or mla_use_prefill_cp(... )`，建议封装为统一函数 `is_any_prefill_cp_enabled(forward_batch)`，降低维护成本。<br>- **文档/注释**：新增的 CP 逻辑在多个文件中分散，建议在 `docs/design/prefill-cp-mla.md` 中完整说明 **MLA‑CP** 与 **DSA‑CP** 的区别、使用前提、限制（batch‑size、ep/DP/TP 关系等）。<br>- **测试覆盖**：已加入数值 parity（FA3 vs full‑seq）以及端到端 GSM8K 精度回归，覆盖率良好；后期可考虑加入多‑batch、不同 `extend_seq_lens` 场景的自动化测试。 |

### feat(kv-events): expose structured KV-event publisher block on /server_info (#25844)
**SHA**: `0857772` | 🔗 [查看提交](https://github.com/sgl-project/sglang/commit/085777210c2f65aeb1d55dda75a50b1fbd6ed32a)

**🎯 变更类型**：功能增强  
**⚡ 重要程度**：🔴高  
**📋 变更摘要**：在 `/server_info` 接口中新增 `kv_events` 结构化字段，向外部 KV‑aware 路由器公开 KV‑event 发布者的配置信息；同时在 `kv_events.py` 的 socket 初始化逻辑中加入对 `0.0.0.0` 通配符的处理，并在 `ServerArgs` 中实现 `describe_kv_events_publisher` 方法用于生成该描述；配套单元测试全面覆盖 happy path 与异常路径。  

**🎯 影响范围**：  
- `python/sglang/srt/disaggregation/kv_events.py`（ZeroMQ 事件发布器初始化）  
- `python/sglang/srt/entrypoints/http_server.py`（`/server_info` 响应）  
- `python/sglang/srt/server_args.py`（新增 `describe_kv_events_publisher`）  
- 新增 `test/registered/unit/entrypoints/test_server_info.py`（单元测试）  

**🔍 技术洞察**  

- **架构影响**  
  - 新增 **“结构化 KV‑event publisher 描述块”**，实现了 **服务自描述(self‑description)**，外部路由器无需手动配置端口或主机即可发现并订阅 KV‑event。  
  - `describe_kv_events_publisher` 采用 **懒加载**，避免在 `ServerArgs` 实例化阶段强制引入 ZeroMQ、msgspec 等重量依赖，保持模块解耦。  
  - `kv_events.py` 中对 `0.0.0.0` 的绑定判断统一了 **“stable → bind，volatile → connect”** 策略，提升了跨平台（IPv4/IPv6）的一致性。  

- **性能影响**  
  - `describe_kv_events_publisher` 仅在 `/server_info` 被调用时执行一次，涉及 JSON 解析与简单整数校验，开销极低（毫秒级）。  
  - `kv_events.py` 绑定判断新增几个字符串比较，对启动时的 socket 创建几乎没有可感知的性能影响。  

- **安全考虑**  
  - `/server_info` 现在会泄露 **端口号、主机通配符** 等信息。若该接口对外公开，攻击者可获取内部 ZeroMQ 端口，用于尝试未授权的订阅或 DoS。建议：  
    - 仅在受信网络或通过身份验证的管理路径暴露 `/server_info`。  
    - 在生产部署中使用防火墙或容器网络策略阻止外部访问 ZeroMQ 端口。  
  - 方法内部对异常（JSON 解析错误、端口非法）统一返回 `None`，避免因未捕获异常导致服务崩溃或返回堆栈信息。  

**⚠️ 潜在风险**  

1. **错误的绑定行为**  
   - 以前使用 `connect` 连接到 `*` 或 `0.0.0.0` 的场景（极少见）现在会改为 `bind`，可能导致端口冲突或多实例竞争。  
2. **信息泄露**  
   - `kv_events` 字段暴露的 `endpoint_port_base` 与 `dp_size` 可能被用于推断集群规模。  
3. **兼容性**  
   - 外部工具仍可能依赖原始 `kv_events_config` 字段的原始 JSON 字符串；若未来文档误导仅使用结构化字段，可能导致误用。  
4. **懒加载依赖**  
   - 虽然使用了局部导入，但在极端环境（如缺失 `zmq`）调用 `/server_info` 时仍会抛出 `ImportError`，导致接口返回 500。  

**💡 关注建议**  

- **发布与治理**  
  - 在文档中明确 **`kv_events` 为可选字段**，并说明其依赖 `--kv-events-config`、`page_size`、合法 TCP endpoint。  
  - 建议在 CI 中加入 **端口冲突检查**（如多实例同时启动）以及 **网络隔离测试**。  

- **安全加固**  
  - 为 `/server_info` 加入 **鉴权 Middleware**（如 API‑Key、OAuth）或只在内部网络开放。  
  - 在生产部署时将 ZeroMQ 端口绑定在内部 IP（例如 `127.0.0.1`）或通过容器网络进行隔离。  

- **代码与测试**  
  - 保持 `describe_kv_events_publisher` 的异常捕获细粒度（可记录日志而非吞掉异常），方便运维排查。  
  - 新增 **集成测试**：启动完整 SGLang 服务器并通过真实 HTTP 客户端请求 `/server_info`，验证端口可被实际订阅。  

- **回滚策略**  
  - 如出现 “bind → 端口被占用” 的生产问题，可通过环境变量或 CLI 参数覆盖 `ServerArgs.kv_events_config` 为 `"null"`，使 `kv_events` 返回 `None` 并回退到原有行为。

此变更显著提升了系统的 **可观测性与可组合性**，为 KV‑aware 路由器提供了自动发现能力，整体风险可通过适当的安全防护与兼容性测试进行控制。

### perf: migrate Req token-id storage to array.array('q') in Scheduler (#25098)
**SHA**: `06c23d5` | 🔗 [查看提交](https://github.com/sgl-project/sglang/commit/06c23d55b58e520a92944f7b6f788cdd01543d03)

**🎯 变更类型**：性能优化  
**⚡ 重要程度**：🔴高  

**📋 变更摘要**  
本次提交将请求（`Req`）在调度路径中对 token‑id 的存储从 `list[int]` 迁移到 `array.array('q')`（64‑bit 整型数组），并在相关的前缀匹配、张量构建、模型输入填充等关键环节使用零拷贝或批量 memcpy 的方式实现。新增专门的 benchmark 用于量化改动收益，同时在 `sglang/srt/utils/common.py` 中提供 `flatten_arrays_to_int64_tensor` 辅助函数，以便高效地将 `array` 列表转化为 CUDA‑pinned `torch.int64` 张量。

**🎯 影响范围**  
- `sglang/srt` 目录下的核心调度实现：`decode_schedule_batch_mixin.py、encode_receiver.py、req.py、scheduler.py、scheduler_pp_mixin.py`  
- KV 缓存层：`mem_cache/*`（RadixCache、MambaRadixCache、UnifiedRadixCache、hiradix_cache）  
- 多模态模型实现：`models/llava.py、llavavid.py、mllama.py、moss_vl.py、whisper.py`  
- 统一工具函数：`utils/common.py`（新增 `flatten_arrays_to_int64_tensor`）  
- 相关数据结构定义：`managers/io_struct.py`（`TokenizedGenerateReqInput`、`BatchTokenIDOutput` 等）  
- 大量单元测试与基准测试文件均已同步改写为使用 `array('q')`。
