### [diffusion] feat: enable spatial-shard vae decode across GPUs (#28071)
**SHA**: `582bd23` | 🔗 [查看提交](https://github.com/sgl-project/sglang/commit/582bd23f71be92337776fb99e4c9c6784a146f60)

**🎯 变更类型**：功能增强 / 架构变更 / 性能优化  

**⚡ 重要程度**：🔴高  

**📋 变更摘要**  
本次 PR 为 VAE（Variational Auto‑Encoder）解码引入 **空间分片（spatial‑shard）并行** 能力，使得在多 GPU 环境下可以把解码特征图在 **高度（H）维度** 按 GPU 划分。核心改动包括  
1. 在 `VAEConfig` 中把 `use_parallel_decode` 默认打开，新增 `parallel_decode_mode`（auto / tiled / patch / spatial_shard / spatial）以及 `auto_parallel_decode_min_latent_elements_per_rank` 参数，用于自动判断是否启用 spatial‑shard。  
2. 在 `parallel_state` 中新增专门的 **VAE‑decode 并行组**（`_VAE_DECODE`），并把相关查询函数迁移到该组。  
3. 实现 **SpatialParallel** 卷积层（`SpatialParallelConv2d`、`SpatialParallelConv3d`、`SpatialParallelCausalConv3d`、`SpatialParallelZeroPad2d`），以及 **halo exchange**、**分块/汇聚**、**高度切片** 等低层通信原语。  
4. 为 Diffusers、Qwen‑Image、Hunyuan、Wan、LTX‑2 等多种 VAE 实现自动替换/patch：  
   * 替换所有 `nn.Conv2d` 为 `SpatialParallelConv2d`（仅在解码时）。  
   * 替换 `GroupNorm`、`Attention` 前向为支持分片的实现。  
   * 通过 `enable_diffusers_decoder_spatial_parallel` 统计并记录上采样层数，用于 `split_for_parallel_decode / gather_and_trim_height`。  
5. 在所有 VAE 解码路径（`AutoencoderKL`, `AutoencoderKLFlux2`, `QwenImageAutoencoderKL`, `HunyuanVideoAutoencoderKL`, `LTX2VideoAutoencoderKL`, `WanVAE` 等）加入 **spatial‑parallel 开关**，在满足阈值时自动走分片路径。  
6. Pipelines 中的 **CFG 并行** 逻辑更新，避免非主 Rank 在需要跨 Rank 结果时盲目执行冗余解码。  
7. 大量单元测试和 CI 适配完成。

### fix(io_struct): index extra_key per sub-request in batched GenerateReqInput (#26971)
**SHA**: `8c334e2` | 🔗 [查看提交](https://github.com/sgl-project/sglang/commit/8c334e22243989ebf6cc9ace41fb598d18624ad4)

**🎯 变更类型**：Bug修复 / 功能增强  

**⚡ 重要程度**：🔴高  

**📋 变更摘要**  
- 为 `GenerateReqInput` 在批量（尤其是并行采样）场景下增加 `extra_key` 的归一化逻辑。  
- 新增 `_normalize_extra_key` 方法，实现字符串广播、列表长度校验以及并行采样时的扩展。  
- 更新 `__getitem__`，在子请求层返回对应的 `extra_key`。  
- 完整的单元测试覆盖普通批量、广播、并行采样、异常情况等路径，提升可靠性。

**🎯 影响范围**  
- `python/sglang/srt/managers/io_struct.py`（核心 IO 结构）  
- 相关单元测试 `test/registered/unit/managers/test_io_struct.py`  
- 任何依赖 `GenerateReqInput.extra_key` 进行租户/业务标识的上层业务逻辑（如多租户调度、日志分流等）。

**🔍 技术洞察**  

- **架构影响**  
  - `extra_key` 现在被视作 **每子请求**（sub‑request）的属性，统一在 `GenerateReqInput` 层完成批处理展开，避免在后续调度/执行阶段额外处理。  
  - 归一化过程在 `_normalize_batch_inputs` 中调用，保持与其它参数（sampling、logprob、bootstrap）统一的批处理流程，架构上更加一致。  

- **性能影响**  
  - 归一化仅在请求创建阶段执行，涉及列表复制与长度检查，时间复杂度 O(batch_size * parallel_sample_num)。在常规请求规模（≤ 64）下对整体吞吐几乎没有可感知影响。  
  - 通过在 `__getitem__` 按需索引 `extra_key[i]`，避免了在每个子请求对象中持有冗余完整列表，略微降低内存占用。  

- **安全考虑**  
  - 新增的输入校验（列表长度必须等于 batch size、类型必须为 `str` 或 `list`）防止恶意或误写导致的 **索引越界** 或 **异常请求泄露**。  
  - 未出现直接的安全漏洞，但强校验提升了系统对未预期输入的鲁棒性，间接降低 DoS 风险。  

**⚠️ 潜在风险**  

1. **误用列表乘法导致意外复制**  
   - 当 `parallel_sample_num` 很大（理论上可配置为 > 100），`extra_key = extra_key * parallel_sample_num` 会生成巨大的列表，可能导致内存激增。  
2. **原地修改属性**  
   - 归一化直接覆盖 `self.extra_key`，若同一 `GenerateReqInput` 实例在同一次请求生命周期外被复用（如对象池），可能残留已扩展的列表。  
3. **向后兼容性**  
   - 之前 `extra_key` 允许任意类型（未校验），升级后若上层仍传入非 `str/list` 将抛异常，需确保所有调用方已适配。  
4. **并行采样与长度校验顺序**  
   - 目前先检查列表长度是否等于 `batch_size`，再乘以 `parallel_sample_num`。如果用户在并行采样场景下提供的列表已经是 `batch_size * parallel_sample_num` 长度，会误报错误。  

**💡 关注建议**  

- **文档升级**：在 API 文档中明确 `extra_key` 支持的类型、广播规则以及并行采样下的扩展行为。  
- **防护大规模并行**：若业务可能使用非常大的 `parallel_sample_num`，在 `_normalize_extra_key` 前加入上限检查或惰性生成（如不复制而通过索引计算）。  
- **对象不可变化**：考虑在归一化后返回一个 **深拷贝** 或使用 `tuple` 存储 `extra_key`，避免意外的后续修改。  
- **兼容迁移**：在发布日志中提醒用户 `extra_key` 现在必须是 `str` 或 `list`，并提供迁移示例。  
- **更健全的错误信息**：将异常信息细化为 “extra_key 长度应等于 batch_size (= X)，实际为 Y”，帮助用户快速定位问题。  

总体来说，此次改动提升了 `extra_key` 在批处理与并行采样场景下的正确性和可预测性，风险可控，建议在下一次发布中同步更新文档并做好回归测试。

通过上述措施，可在提升 MLX 后端长序列解码的稳定性的同时，避免因配置不当带来的性能或资源风险。

**🎯 变更类型**：功能增强  
**⚡ 重要程度**：🔴 高  
**📋 变更摘要**：在 Playground 前端新增 `flagSelects` 轴，支持通过配置声明任意数量的单选框，每个选项可自定义标题、拆除前缀以及对应的 flag 列表，实现模型特性（如 KV‑Cache Dtype、mamba‑scheduler 等）完全基于配置而无需修改引擎代码。同步更新多篇内部文档，明确“模型特性是配置数据而非引擎代码”的开发准则。  

**🎯 影响范围**：  
- `docs_new/src/snippets/_playground.jsx`（前端渲染、状态管理、flag 生成）  
- `.claude/skills/*` 相关文档（cookbook、migration 指南）  
- 所有使用 Playground 的模型页面（只要在配置中声明 `flagSelects`）  

**🔍 技术洞察**：

- **架构影响**  
  - **前端**：在 `AXIS_HANDLERS` 中加入全新 `flagSelects` 处理器，保持与现有轴（`attention`、`speculative` 等）相同的 API（`initState / deriveFromBase / revertHidden / apply / render`），因此整体架构未被破坏，仍然是 “轴‑处理器” 的插件式设计。  
  - **配置驱动**：通过 `config.playgroundFeatures.flagSelects` 完全声明 UI 与 flag 逻辑，符合 “配置即代码” 的原则，降低了后端依赖，提升了模型特性迁移的可维护性。  
  - **向后兼容**：运行时在 `apply/render` 循环里通过 `if (!fc) continue;` 跳过未声明的轴，旧版 Playground 配置不受影响。  

- **性能影响**  
  - **渲染成本**：新增的 `render` 会遍历配置中所有 `flagSelects` 项目并为每项生成卡片，复杂度 O(N·M)（N 为轴数量，M 为每个轴的选项数），一般在 10‑20 项以内，对 UI 渲染毫无压力。  
  - **状态同步**：`deriveFromBase` 与 `apply` 在每次 UI 更新时都会遍历同一集合，产生的 CPU 开销极低（几百微秒），不会影响 Playground 的交互流畅度。  
  - **网络/后端**：本次改动不涉及后端请求或数据传输，网络负载保持不变。  

- **安全考虑**  
  - **Flag 注入**：`apply` 直接把选项中声明的 `flags` 拼接到最终命令行，若配置作者误写恶意 flag（如 `; rm -rf /`）会在后端执行时产生风险。虽然配置一般由内部作者维护，但为了防御外部贡献，建议在后端对最终生成的 flag 列表进行白名单校验。  
  - **XSS/DOM 注入**：所有 UI 文本（`title`、`label`）均通过 React 渲染，未使用 `dangerouslySetInnerHTML`，因此不存在 DOM 注入风险。

1. **配置误用**：若在 `config.playgroundFeatures.flagSelects` 中遗漏 `stripPrefixes` 或写错 flag，可能导致生成的命令缺失必要标志或产生冲突，影响模型推理准确性。  
2. **状态恢复错误**：`revertHidden` 逻辑依赖 `h.isHidden` 判断，若隐藏判定规则改变，可能导致已选项被错误清除，用户体验受损。  
3. **兼容性**：旧版浏览器或不支持最新 React 特性的环境可能在渲染大量卡片时出现性能回退，需要确保兼容性至少到 Chrome 80+（项目本身已有此要求）。  
4. **文档同步**：新增轴的使用规范必须在所有 Cookbook、migration 文档中同步，否则作者可能仍写成“模型‑专属 handler”，导致重复工作或错误 PR。  

**💡 关注建议**：

- **配置校验**：在 CI 或 PR 检查阶段加入 schema 验证（JSON Schema），强制 `title`、`stripPrefixes`、`options[].flags` 为合法字符串数组，防止误写。  
- **后端安全**：在生成最终 CLI 命令前，对所有 flag 做白名单过滤或正则校验，只允许已在 SGLang CLI 注册的 flag 前缀通过。  
- **单元测试**：为 `flagSelects` 轴编写前端单元测试，包括 `initState`、`deriveFromBase`、`apply`、`render` 四个入口的关键路径，确保选项切换、隐藏恢复、以及与其他轴的协同工作无 regress。  
- **性能基准**：在本地和 CI 中添加渲染基准（例如 100 条 `flagSelects`），确保在极端情况下仍保持 60 fps 以上。  
- **文档培训**：组织一次内部培训，明确“模型特性是配置数据”与“仅在真正新形状时才创建 generic primitive”两条原则，防止后续出现模型‑命名的专属轴实现。

**结论**：此次改动通过一次性引入通用、配置驱动的 `flagSelects` 轴，显著提升了 Playground 对模型特性的可扩展性与维护效率，架构保持清晰，性能影响可忽略，安全风险主要在配置层面，可通过 CI 校验与后端白名单 mitigated。建议尽快合并，并在后续 PR 中加入上述校验与测试，以确保稳健推广。
