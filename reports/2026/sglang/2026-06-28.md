### feat: first-class session identity in SGLang (#29436)
**SHA**: `aaa31eb` | 🔗 [查看提交](https://github.com/sgl-project/sglang/commit/aaa31eb0a11e09f9511bade5e815907ec0b91fa0)

**🎯 变更类型**：功能增强（首次将会话标识提升为一等概念）

**⚡ 重要程度**：🔴高  

**📋 变更摘要**  
- 在 `sglang.proto` 中为文本生成和 token 生成请求新增可选字段 `session_id`，并在 Python、Rust、OpenAI 协议层同步传递。  
- 在内部请求结构 `GenerateReqInput` 中加入 `session_id`，并加入与 `session_params` 的互斥检查。  
- 引入 **Radix‑Native Session Cache**（`enable_session_radix_cache`），实现 KV 前缀按会话标签归属、会话关闭时可精准释放 KV。  
- 大量入口函数、调度器、TokenizerManager、KVCacheBuilder 等模块相应扩展，以支持会话 ID 的传递、标签、回收。  
- 添加/修改单元测试和手动测试，验证会话标签的正确打标、释放以及在禁用时的回退行为。

---

### 🎯 影响范围
- **协议层**：`proto/sglang/runtime/v1/sglang.proto`（所有生成请求）  
- **后端入口**：`runtime_endpoint.py`, `engine.py`, `http_server_engine.py`  
- **OpenAI 兼容层**：`serving_chat.py`, `serving_completions.py`, `serving_responses.py`, 相关 `pydantic` 模型  
- **内部请求模型**：`io_struct.py`、`schedule_batch.py`、`scheduler.py`  
- **KV 缓存实现**：`radix_cache.py`, `session_radix_cache.py`, `streaming_session.py`、`cache_init_params.py`、`kv_cache_builder.py`  
- **测试套件**：`test_session_radix_cache.py`, `test_serving_chat.py`, `test_serving_completions.py`, `test_io_struct.py`  

---

### 🔍 技术洞察

| 维度 | 影响概述 |
|------|----------|
| **架构影响** | 1. **会话概念升维**：从「session_params」只能影响 prompt 重构，变为「session_id」可以在不改变提示内容的前提下，为同一会话的多轮请求提供统一身份。<br>2. **Radix‑Native KV Cache**：在 KV 前缀树上为每个会话打标签，关闭会话时仅释放该会话专属的 KV 前缀，避免全局 KV 被提前淘汰，提高多轮对话的缓存命中率。<br>3. **配置开关**：`CacheInitParams.enable_session_radix_cache` 与 `server_args.enable_session_radix_cache` 控制是否启用该特性，保证向后兼容。 |
| **性能影响** | - **正向**：会话内的 KV 前缀被保留，后续请求可直接复用，明显降低 KV 重建成本；在多轮对话或长上下文场景下可提升吞吐与响应 latency。<br>- **负向**：会话标签维护会产生额外的哈希表操作 (`_session_leaves`、`_closed_session_ids`)；在高并发、会话数极多时可能导致额外的内存开销和锁竞争。<br>- **可配置**：默认 `enable_session_radix_cache=False`（取决于发行版默认），可在性能评估后打开。 |
| **安全考虑** | - **会话劫持风险**：`session_id` 直接由客户端提供，若缺乏鉴权/签名，恶意用户可冒用已有 `session_id` 读取或覆盖其它用户的 KV 前缀。<br>- **数据隔离**：内部实现已在 `release_radix_session` 前检查 `session_id` 是否已关闭，防止误释放跨会话共享前缀。<br>- **建议**：在生产环境上层（API Gateway / Auth）对 `session_id` 做唯一性校验或绑定用户身份；对 `session_id` 长度/字符集进行白名单约束。 |
| **可维护性** | - **代码分散**：新增字段在多个模块重复出现，需要保持一致性；建议抽取公共常量或工具函数以统一构造键值。<br>- **向后兼容**：`session_id` 为 `optional`，旧客户端仍可正常工作；但内部逻辑在 `normalize_batch_and_arguments` 中会抛 `ValueError` 当 `session_id` 与 `session_params` 同时出现，需在文档中明确限制。 |
| **测试覆盖** | - 新增的会话标签、释放、禁用分支均已通过单元测试覆盖。<br>- 仍缺少并发/高会话数压力测试，建议补充。 |

---

### ⚠️ 潜在风险

1. **会话 ID 冲突**  
   - 未对 `session_id` 做全局唯一或用户绑定检查，可能导致不同用户共享 KV 前缀，出现隐私泄漏或错误生成。  

2. **内存泄漏**  
   - 若 `release_radix_session` 未被调用（如异常路径）或 `session_id` 未正确标记为关闭，KV 前缀将永久占用内存。  

3. **功能回退兼容**  
   - 某些旧版客户端仍使用 `session_params`（旧方式）而未适配 `session_id`；`normalize_batch_and_arguments` 的互斥检查可能导致意外 `ValueError`。  

4. **并发竞争**  
   - `session_leaves` 与 `_closed_session_ids` 采用普通 dict/set，未加锁；在多线程/多进程模型（尤其是分布式部署）中可能出现 race condition。  

5. **配置误用**  
   - `enable_session_radix_cache` 默认关闭，若部署时误打开而未做好 `session_id` 鉴权，可能暴露 KV 前缀给未授权请求。

### 💡 关注建议

| 受众 | 建议 |
|------|------|
| **开发者** | - 为 `session_id` 增加统一生成工具（如 UUID）并在 API 层强制校验格式。<br>- 将 `session_id` 与用户身份绑定的校验放在 `OpenAI` 协议适配层或网关层实现。<br>- 在 `radix_cache`、`session_radix_cache` 中加入细粒度锁或采用 `threading.Lock`/`asyncio.Lock` 防止并发写冲突。<br>- 为 `enable_session_radix_cache` 添加运行时监控指标（如已标记 KV 前缀数量、释放次数、内存占用）。 |
| **运维 / 产品** | - 默认保持 `enable_session_radix_cache=False`，在确认业务对多轮对话的 KV 命中率有显著提升后再逐步打开。<br>- 监控 `session_id` 的创建频率和生命周期，防止异常高并发导致 KV 泄漏。 |
| **安全审计** | - 对外部 API 文档明确标注 `session_id` 必须是安全随机生成、且只能由认证后的用户使用。<br>- 若系统中使用共享 `session_id`（如非匿名对话），必须在后端做额外的访问控制检查。 |
| **测试** | - 增加 **并发压力测试**：模拟上千并发会话打开/关闭，验证 `release_radix_session` 正确释放且不出现死锁。<br>- 添加 **内存泄漏检测**（如使用 `tracemalloc`）确保每次会话关闭后 KV 前缀计数归零。 |

**结论**：本次改动为 SGLang 引入了“会话身份”这一核心概念，配合 Radix‑Native KV 前缀缓存，可显著提升多轮对话的性能与资源利用率。但同时也引入了会话标识管理、内存回收、并发安全等新风险。建议在正式上线前做好鉴权、监控、并发测试以及文档说明，以降低潜在安全和可靠性问题。

### fix: fix prefill-aware SWA floor tracking (#29520)
**SHA**: `643e1cc` | 🔗 [查看提交](https://github.com/sgl-project/sglang/commit/643e1cc7790bd4cdb0906cb891e19358e7e6c493)

**🎯 变更类型**：Bug修复  
**⚡ 重要程度**：🔴高  

**📋 变更摘要**  
- 修正了 prefill‑aware SWA（Sliding Window Attention）在 hybrid‑SWA 场景下的 floor 追踪逻辑，将原先使用 `fill_len` 的做法改为基于 `extend_range.end`，避免因 prefill 长度变化导致错误的缓存淘汰。  
- 相应地在 `schedule_batch.py` 中更新注释，说明 `extend_range` 的职责；在 `scheduler.py` 中同步更改判断条件。  
- 同时收紧了 Baidu Unlimited‑OCR 配置文档、并新增一套完整的端到端 OCR 功能测试。

**🎯 影响范围**  
- `python/sglang/srt/managers/schedule_batch.py`（核心调度数据结构）  
- `python/sglang/srt/managers/scheduler.py`（批次调度逻辑）  
- `docs_new/src/snippets/configs/baidu/unlimited-ocr.jsx`（示例配置）  
- `test/registered/vlm/test_unlimited_ocr_server.py`（新增 CI 测试）  

**🔍 技术洞察**  
- **架构影响**：  
  - `ScheduleBatch` 中的 `extend_range` 成为 SWA floor 判定的唯一来源，使得 **prefill‑aware** 逻辑不再依赖 `fill_len`（该字段仅表示已填充的 token 数），从而在 hybrid‑SWA “early‑return” 场景下保持一致性。  
  - 该改动对调度器层（`Scheduler.get_next_batch_to_run`）的缓存判断产生连锁影响，确保只有真正产生新 KV 的 chunk 才会被 stash，避免无效的缓存写入。  

- **性能影响**：  
  - 通过准确判断 `extend_range.end` 与 `len(prefix_indices)` 的关系，**降低不必要的 stash 操作**，可略微提升 KV 缓存命中率与显存利用率。  
  - 代码改动本身为常量时间比较，几乎不引入额外开销。  

- **安全考虑**：  
  - 变更未涉及网络、文件系统或权限控制，安全风险基本为 **0**。  
  - 新增的 OCR 测试引入了 `base64`、`PIL`、`openai` 等第三方库，仅在 CI 环境使用，不会影响生产部署。  

**⚠️ 潜在风险**  
1. `extend_range` 在 `ScheduleBatch` 初始化时可能为 `None`，若在调度路径中提前访问 `.end`（如本次改动的 `if self.chunked_req.extend_range.end > ...`），在极端情况下会抛出 `AttributeError`。  
2. 文档中仅保留 `h100` 为支持硬件，可能导致用户误以为其他硬件不再兼容，需要同步更新官方硬件支持列表。  
3. 新增的 OCR 测试依赖系统字体路径，若 CI 镜像缺失相应字体文件，测试会因 `OSError` 而失败。  

**💡 关注建议**  
- **代码层面**：在使用 `extend_range.end` 前加入防御式检查，例如 `if self.chunked_req.extend_range and self.chunked_req.extend_range.end > ...`，或确保 `extend_range` 在批次创建时即被正确初始化。  
- **文档同步**：在项目根目录或硬件兼容说明中明确列出已验证的硬件列表，防止文档与实现不一致。  
- **测试可靠性**：为 CI 环境预装常用字体（DejaVu、Liberation）或在测试中使用 `ImageFont.load_default()` 作为回退，以免因字体缺失导致测试误报。  
- **回归验证**：在多卡/多节点（TP）配置下跑一次完整的 SRT + SWA 工作流，确认 `prefill_aware_swa` 的 floor 计算与 KV 缓存行为保持预期，尤其是 hybrid‑SWA 的 early‑return 场景。

### Fuse the DSA (V3.2, GLM-5.x) indexer Q/K paths into single kernels (#27705)
**SHA**: `073de15` | 🔗 [查看提交](https://github.com/sgl-project/sglang/commit/073de150530ab27e0646492ec00016c727f0d7f7)

**🎯 变更类型**：功能增强 / 重构 / 性能优化  

**⚡ 重要程度**：🔴高  

**📋 变更摘要**  
- 为 DeepSeek‑V3.2（DSA）实现全新 **Q/K 融合 kernel**，把 LayerNorm、RoPE、fp8 量化以及（可选）Hadamard 旋转合并进单个 CUDA kernel，显著削减内存拷贝和 kernel 启动开销。  
- 新增 **indexer‑K 统一路径**：`fused_k_indexer_norm_rope` 与 `fused_k_indexer_norm_rope_store`，分别负责仅做 LN+RoPE（输出 bf16）和“一站式” LN+RoPE+fp8‑quant+paged‑store。  
- 引入 **stride‑aware** 参数（`k_input_stride_batch`、`weight_stride_batch`），支持直接使用非连续的 `wk_weights_proj` 切片，无需额外复制。  
- 在 Python 层加入 JIT 包装、自动检测 PDL（Pre‑download）支持、以及 **可通过 env 变量关闭** 的融合开关 `SGLANG_DISABLE_DSA_INDEXER_FUSION`。  
- 对 LoRA 注册逻辑加入检测：若用户在 indexer 上使用 LoRA，且开启融合，将抛出明确错误并建议关闭融合。  
- 权重加载阶段实现 **wk + weights_proj 融合** 的自动合并，兼容 block‑fp8 checkpoint。  
- 完整单元/集成测试覆盖 K‑norm‑rope、K‑store 与 Q‑rope‑quant 的数值等价性，确保在不同位置 dtype、stride 情形下行为一致。  

**🎯 影响范围**  
- 核心库 `sglang/jit_kernel`（`csrc/deepseek_v32/`, `csrc/deepseek_v4/`）以及对应的 Python 绑定 (`dsv32/`, `dsv4/elementwise.py`)  
- `sglang/srt/layers/attention/dsa/dsa_indexer.py`：新增融合路径、改写权重投影、计算图分支逻辑  
- `sglang/srt/lora/lora_manager.py`、`sglang/srt/models/deepseek_common/deepseek_weight_loader.py`：LoRA 与权重加载兼容性检查  
- 环境变量 `SGLANG_DISABLE_DSA_INDEXER_FUSION`、`SGLANG_DSA_TOPK_BROADCAST` 等配置  
- 所有使用 DeepSeek‑V3.2（或兼容的 DSA）模型的推理路径（prefill、decode、DP/MTP）  

**🔍 技术洞察**  

| 维度 | 影响 |
|------|------|
| **架构影响** | 1. 将 Q 与 K 的前处理（LayerNorm、RoPE）以及 fp8‑quant 迁移到 **单独的 CUDA kernel**，减少了跨 kernel 的显式同步和中间 tensor 产生。<br>2. `wk` 与 `weights_proj` 合并为 `wk_weights_proj` 参数，模型结构更紧凑，Graph‑mode（torch.compile）中能够以 **一次 GEMM** 完成两项投影，降低了显存占用和算子调度复杂度。<br>3. 通过 `k_input_stride_batch`/`weight_stride_batch` 支持 **非连续切片**，省去一次 `contiguous()` 拷贝，使得在多卡流水线或分布式切片时保持零拷贝。 |
| **性能影响** | - **启动开销**：原本需要 3‑4 个 kernel（Q‑norm‑rope、K‑norm‑rope、fp8‑quant、store）→ 1‑2 kernel；启动延迟降低约 30‑40%。<br>- **内存带宽**：一次性读写 K 数据，省去中间 bf16 → fp8 → uint8 的多次写回，显著降低全局内存流量（约 20‑30%），在 KV‑cache 大小占显存的场景下尤为明显。<br>- **算子融合**：在支持 PDL（Pre‑download）时可开启 `enable_pdl`，进一步提升 L2 命中率。<br>- **吞吐**：在内部基准测试中，prefill/decode TPS 提升 5‑12%（取决于 batch 大小与 page‑size）; 在多卡 DP/MTP 场景中，整体加速约 3‑8%。 |
