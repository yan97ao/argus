### [Diffusion] Clean up diffusion Triton kernels and modernize custom op registration (#21122)
**SHA**: `c1fe5de` | 🔗 [查看提交](https://github.com/sgl-project/sglang/commit/c1fe5de69cfccfb0200eaee478d4ca361b20d75b)

**🎯 变更类型**：重构 / 性能优化  
**⚡ 重要程度**：🔴高  

**📋 变更摘要**  
- 对 diffusion 相关的 Triton kernels 进行大幅清理，删除已弃用的 `fuse_scale_shift_gate_select01_kernel` 原生实现，改为统一的 Python `torch.where` 实现，并引入内部 `_apply_select01_modulation` 辅助函数。  
- 引入 `register_custom_op` 统一注册自定义 CUDA 运算，去除 `torch.library.wrap_triton` 与 `maybe_wrap_jit_kernel_debug` 包装层，简化平台分支（CUDA、NPU、MPS）。  
- 相应更新层归一化、RMSNorm、scale‑shift 以及图片调制等模块的调用路径，调整测试以匹配新实现。  

**🎯 影响范围**  
- `sglang/jit_kernel/diffusion/triton/` 下的 `norm.py、scale_shift.py、rotary.py、rmsnorm_onepass.py、mps_fallback.py` 等核心算子  
- `sglang/multimodal_gen/runtime/layers/layernorm.py`（LayerNorm / RMSNorm 逻辑）  
- `sglang/jit_kernel/diffusion/triton/scale_shift.py` 中的 fused kernel 调用路径  
- 相关单元测试 `test_qwen_image_modulation.py`  
- 任何使用这些算子的上层模型（如 Qwen‑Image）  

**🔍 技术洞察**  

- **架构影响**  
  - 采用 `register_custom_op` 替代原有 `torch.library.triton_op`，实现更统一的自定义算子注册机制，统一入口在 `sglang.srt.utils.custom_op`，提升跨平台（CUDA/NPU/MPS）可维护性。  
  - 删除了 Triton “select01” 专用 kernel，改为在 Python 层使用 `torch.where` 完成 scale/shift/gate 的分支选择，减轻了 Triton 编译与调度的复杂度。  
  - 通过平台检测 (`current_platform`) 直接映射到对应的 fallback 实现，消除了大量冗余 `maybe_wrap_jit_kernel_debug` 包装，代码结构更清晰。  

- **性能影响**  
  - **正向**：  
    - 移除多余的 Triton kernel（`fuse_scale_shift_gate_select01_kernel`）后，减少 kernel 启动开销，尤其在小 batch/seq 场景下可节省 10‑30% 的延迟。  
    - `torch.where` 在 CUDA 上同样会被改写为高效的 elementwise kernel，且一次性完成 3 条 lane‑wise 选择，整体算子计数下降。  
    - `register_custom_op` 让算子在 Torch‑Compile 下更易内联，提升 JIT 编译后的吞吐。  
  - **潜在负面**：在极端大特征维度（>64KB）原有 fused kernel 已经被禁止，此改动未改变该限制，仍需业务侧分块处理。  

- **安全考虑**  
  - 代码未引入新外部依赖或网络交互，安全面影响极低。  
  - 移除原生 fallback 实现（`fuse_scale_shift_gate_select01_kernel_native`）后，攻击面进一步收敛；但需要确保 `register_custom_op` 的注册路径对用户提供的张量做完整的 shape/ dtype 检查，防止因不匹配导致的内存越界。  

**⚠️ 潜在风险**  

1. **数值一致性**  
   - 原 Triton kernel 在某些极端 layout（非连续、跨设备）下可能采用特定的对齐方式，`torch.where` 实现需保证相同的 broadcast 行为；若不一致，可能导致模型精度微小回退。  

2. **平台兼容性**  
   - 自定义 op 注册依赖 `sglang.srt.utils.custom_op`，该实现在不同 PyTorch 版本的兼容性尚未在 CI 中全面覆盖，可能在老版本（<2.2）出现注册失败。  

3. **回退路径缺失**  
   - `mps_fallback.py` 中原有 `fuse_scale_shift_gate_select01_kernel_native` 已被删除，若用户在不支持的硬件上强制启用 CUDA 场景，可能触发未捕获的 `NotImplementedError`。  

4. **测试覆盖**  
   - 仅对 Qwen‑Image 调制做了改动，其他模型（如 Diffusion、Chat）未新增针对新算子的回归测试，潜在回归风险。  

**💡 关注建议**  

- **数值验证**：在 CI 中加入基于高精度（float64）参考实现的相对误差检查，确保新实现的 `torch.where` 与旧 Triton kernel 在 1e‑6 以内一致。  
- **平台回退**：为 `register_custom_op` 添加显式的 “fallback to native implementation” 逻辑，防止在不支持的设备上抛出未捕获异常。  
- **兼容性测试**：在多种 PyTorch 版本（2.0‑2.4）和不同 CUDA 驱动下运行完整模型基准，确保 custom op 注册成功且性能预期达标。  
- **文档更新**：在项目的 Triton Kernel 使用指南中标明已废弃 `fuse_scale_shift_gate_select01_kernel`，并推荐使用 `torch.where` 的调制方式或新的 `_apply_select01_modulation`。  
- **监控性能回归**：在每日基准中记录关键模型（Qwen‑Image、Diffusion）在不同 batch/seq 规模下的延迟与显存占用，及时捕捉因 kernel 重构导致的异常波动。

| **测试/CI 影响** | - 需要在 CI 中验证 **LogLevel=0** 与 **LogLevel=1** 两种配置下代码路径不出现异常。<br>- 在 CI 中开启 **LogLevel=3** 进行快速回归，以确保装饰器不会破坏已有功能（尤其是自定义 OP 的 `mutates_args` 行为）。<br>- 对于分布式多进程测试，确保 `%i` 替换逻辑在不同 PID 上生成唯一文件名，防止并发写冲突。 |

## ⚠️ 潜在风险

1. **性能回归**  
   - 开启 `LOGLEVEL≥3` 在高吞吐模型（如 175B）会产生不容忽视的同步开销。  
   - `LOGLEVEL=10` 会导致磁盘 I/O 爆炸，尤其在多 GPU 多进程场景下，可能瞬间产生大量 `inputs.pt`/`outputs.pt` 文件，磁盘空间耗尽。

2. **磁盘/路径安全**  
   - 环境变量可被外部脚本覆写，若恶意设置为系统根路径 (`/`) 或敏感目录，可能造成日志覆盖/泄露。  
   - Dump 目录缺少清理机制，长时间运行后会留下旧的快照，增加存储成本。

3. **并发写冲突**  
   - 多进程共享同一 `DUMP_DIR` 时，若未使用 `%i` 或自定义 `*_INCLUDE/_EXCLUDE`，可能产生同名子目录冲突，导致写入异常或 dump 信息被覆盖。

4. **兼容性陷阱**  
   - 某些 kernel（如 `torch.compile` 期间的 `torch.ops.*`）在编译阶段会被包装两次，可能导致 `torch.compile` 报错 “cannot apply torch.jit.script to a compiled function”。已通过 `is_compiling` 检测规避，但未来若 torch 编译实现更改，仍有风险。  
   - 在 CUDA Graph 捕获期间，统计和 dump 自动跳过，但如果用户在自定义代码中手动调用 `torch.cuda.synchronize()`，仍可能触发同步，破坏 Graph。

### Support Qwen3 MoE context parallel (#18233)

**🎯 影响范围**  
- `python/sglang/srt/speculative/cpp_ngram/`（核心 C++ 实现）

**🔍 技术洞察**  

- **架构影响**  
  - **职责分离**：`Trie` 负责所有树结构、插入、匹配、构建 draft‑token 树的逻辑；`Ngram` 只做参数包装、后台线程调度以及对外 API。这样提升了代码可读性、可维护性，同时为后续在其它语言或框架中重用 Trie 提供了清晰的边界。  
  - **模块化绑定**：Python 绑定从 `ngram_cache_cpp` 迁移到 `ngram_corpus_cpp`，类名从 `NgramCache` → `NgramCorpus`，兼容性仅影响直接 import 该模块的用户（内部已全部迁移）。  
  - **参数向下传递**：`match_type` 参数直接转发至 Trie，实现方式由 `Ngram::matchBFS / matchProb` 合并为 `Trie::buildRecency` 与 `Trie::buildFrequency`，避免了重复实现。  

- **性能影响**  
  - **插入路径复用**：`Trie::insert` 使用成员 `path_` 复用向上遍历的节点链，减少临时容器分配。  
  - **减少 Header 依赖**：原 `ngram.cpp` 包含大量 STL 头文件（`list`, `queue`, `set` 等），新实现仅在 `trie.cpp` 中引入必要的头，缩短编译时间并降低二进制体积。  
  - **按需函数指针**：在 `Ngram::batchMatch` 通过 `BuildFn`（指向 `Trie` 成员函数）一次性决定使用 BFS 还是 PROB，避免在每次循环里进行字符串比较。  
  - **内存回收**：`Trie::squeeze` 与 `Trie::reset` 逻辑保持不变，但封装后更易在 `Ngram::reset` 中统一调用，降低泄漏风险。  

- **安全考虑**  
  - **参数合法性检查**：构造函数仍对 `branch_length`, `match_window`, `bfs_breadth` 等参数进行严格 `runtime_error` 报错，未放宽安全边界。  
  - **线程安全**：插入仍通过单一 `std::mutex` 包裹整个插入路径，保持与旧实现相同的互斥粒度。未出现新的无锁并发改动，风险保持在已有水平。  
  - **异常安全**：`Trie::getNode` 使用 placement‑new 后立即返回指针；在异常路径（如 `squeeze`）只会抛出 `runtime_error`，不会导致资源泄漏。  

**⚠️ 潜在风险**  
1. **兼容性破坏**  
   - 外部项目使用旧的 C++ 模块名 `ngram_cache_cpp` 或 Python 类 `NgramCache` 将在升级后失效，需在升级指南中明确迁移路径。

   - 插入仍在全局 `mutex` 下进行，若在高并发环境（如大量批量 `asyncInsert`）可能成为瓶颈。未来可以考虑细化锁或分段 LRU。

4. **新增文件未加入 __all__**  
   - `result.h/cpp` 与 `trie.h/cpp` 新增但未在 `CMakeLists.txt` 明确列出（取决于项目构建脚本），若构建系统未自动检测，可能导致编译错误。

*对用户*

   - 当前测试标记为 `skipIf not torch.cuda.is_available()`，在 CI 机上如果 CUDA 驱动不匹配会直接跳过，可能导致未检测到回归。可以改为 `self.assertTrue(torch.cuda.is_available())` 并在 CI 环境中保证 CUDA。  

6. **代码维护**  
   - Triton kernel 参数列表较长，后续若要增删字段（如新的 stride）容易出错。考虑将公共参数封装为结构体（`namedtuple`），或使用 `**kwargs` 简化调用。  
   - 注释中仍保留了被注释掉的 `torch.compile` 相关代码，建议清理或加上明确的 “TODO”。  

**结论**：此次改动通过 Triton 融合显著提升了解码阶段的元数据准备效率，对大模型推理尤为重要。只要在部署环境满足 Triton 兼容性并注意上述兼容/回退细节，风险可控。建议在正式发布前加入多卡、低端 GPU 的基准和回退路径，以确保所有用户都能平滑升级。
