### [LoRA] BF16 support + EP cuda-graph crash fix for experimental_sgl_trtllm MoE-LoRA (#28953)
**SHA**: `9fd6d0e` | 🔗 [查看提交](https://github.com/sgl-project/sglang/commit/9fd6d0ec990f5cfcca45898ec67d246a845af820)

**🎯 变更类型**：功能增强 / 重构 / 性能优化 / 安全修复  

**⚡ 重要程度**：🔴高  

**📋 变更摘要**  
1. 在 `sglang/jit_kernel/trtllm_lora_temp` 中实现 BF16 路径的 MoE‑LoRA，提供端到端 BF16（无量化）执行流 `trtllm_bf16_routed_moe_lora`，并在 Python 包装层完成调用、参数校验与注册。  
2. 为 BF16 LoRA 增加两流（side‑stream）并行实现，复用已有 FP8/FP4 两流代码，实现 gate‑up LoRA 的 shrink/expand 与主流计算并行。  
3. 修复 `permuteKernel` 在 expert‑parallel 场景下出现的负索引 OOB 错误，防止 pre‑fill 大批量时的非法内存访问。  
4. 新增环境变量 `SGLANG_OPT_LORA_PREFILL_ROUTING_REUSE`，在 pre‑fill（≥512 token）时统一 routing‑stage 的 `BLOCK_SIZE_M`，避免重复的 align/sort 计算。  
5. 扩展 LoRA 注册/覆盖逻辑，保存原始 BF16 LoRA 实现并在两流模式下切换。  

**🎯 影响范围**  
- **核心算子层**：`python/sglang/jit_kernel/trtllm_lora_temp/core.py`、CUDA kernel `trtllm_fused_moe_dev_kernel.cu`、`trtllm_fused_moe_kernel_launcher.cu`。  
- **Python 调度层**：`sglang/srt/lora/trtllm_lora_temp/*`（`__init__`、`environ.py`、`lora_dispatch.py`、`moe_overlap.py`、`lora_layer.py`、`triton_ops/virtual_experts.py`）。  
- **后端插件**：BF16‑MoE‑LoRA 在所有使用 `experimental_sgl_trtllm` 的模型（如 Qwen‑BF16）可直接受益。  
- **运行时配置**：新增环境变量对 pre‑fill 过程的 routing 计算产生影响。  

**🔍 技术洞察**  

| 维度 | 影响描述 |
|------|----------|
| **架构影响** | - 在 MoE‑LoRA 结构中加入 BF16 专用的 “decomposed” 流：routing → permute → gate_up GEMM → LoRA‑aware activation → down GEMM → finalize。<br>- 采用与 FP8/FP4 相同的 `BlockMajorK` 权重布局，确保跨量化路径的代码复用和权重加载统一。<br>- 两流并行的实现把 LoRA‑shrink/expand 放到 side‑stream，保持主流计算（routing、permute、GEMM）不被阻塞。<br>- `permuteKernel` 的负索引检查提升了 EP（expert parallel）在多卡环境下的鲁棒性。 |
| **性能影响** | - **BF16 端到端**：省去 FP8/FP4 量化/解量化阶段，理论上可提升 5%‑15%（取决于显存带宽与算子占比），尤其在大模型推理时更为明显。<br>- **两流并行**：Gate‑up LoRA 的 shrink/expand 与主流 GEMM 并行，降低整体 latency，部分工作负载在 decode‑batch（<512 token）上可提升约 2%‑4%。<br>- **路由复用**：`SGLANG_OPT_LORA_PREFILL_ROUTING_REUSE` 通过统一 `BLOCK_SIZE_M`，在 pre‑fill（≥512 token）阶段将原本 4 次 align/sort 合并为 1 次，预计减少 15%‑30% 的 routing 计算时间与显存占用。<br>- **内存峰值**：与 FP4/FP8 路径保持相同的 “permute‑buffer‑reuse” 机制，未引入额外显存压力。 |
| **安全/可靠性** | - 修复的 `permuteKernel` OOB 写防止了在 expert parallel 场景下的非法访问，提升了 GPU 设备的安全性和程序的稳定性。<br>- 两流实现中对 CUDA event 的生命周期管理（将事件加入全局列表防止捕获期间被提前销毁）避免了 cuda‑graph 重放时的崩溃。<br>- 新增的环境变量默认开启，且仅在 pre‑fill 大批量时生效，对已有小批量推理不产生副作用。 |
| **可维护性** | - BF16 代码沿用了现有的 `Bf16LoraLauncher` 与 `sgl_trtllm_*` 注册模式，保持了代码风格一致性。<br>- 在 `lora_dispatch.py` 中通过 `isinstance(quant_info, FlashInferTrtllmBf16MoeQuantInfo)` 自动分流，降低了后续添加新量化路径的改动成本。<br>- 环境变量集中在 `environ.py`，日志可追溯。 |

**⚠️ 潜在风险**  
1. **兼容性**：BF16 LoRA 依赖权重已被预处理为 `BlockMajorK` 布局；若用户使用自定义的 BF16 checkpoint（未经过 `process_weights_after_loading`），可能导致 shape 不匹配或性能下降。  
2. **两流调度**：侧流的 `lora_event` 必须在 cuda‑graph 捕获期间保持存活。若外部代码手动销毁或提前回收该事件（如自行调用 `torch.cuda.Event().record()`），可能导致图重放时出现未定义行为。  
3. **路由复用**：`SGLANG_OPT_LORA_PREFILL_ROUTING_REUSE` 将 A 阶段的 `BLOCK_SIZE_M` 强制与 B 阶段相同，若底层 `flashinfer` 在某些硬件（如旧版 Ampere）上对特定 `BLOCK_SIZE_M` 有性能回退，可能导致预期的加速失效。  
4. **调试难度**：新增的 BF16 路径与 FP8/FP4 共用同一套 C++/CUDA 注册入口，调试时需要确保 `act_type` 与激活函数对应，否则会触发 `TVM_FFI_ICHECK` 断言。  
5. **显存碎片**：在两流路径中，side‑stream 需要提前分配 routing/shrink 中间 buffer。如果显存已经高度占用，可能导致 `cudaMalloc` 失败，进而触发回退到单流路径（功能仍可用，但性能下降）。  

**💡 关注建议**  
- **验证权重布局**：在使用 BF16 LoRA 前，请确认模型 checkpoint 已通过 `sgl_process_weights` 或等价脚本进行 `BlockMajorK` 处理。若未处理，可在加载后手动 `reshape` 为 `[E, N, K]`。  
- **监控 CUDA 事件**：建议在开启两流模式 (`SGLANG_LORA_TWO_STREAM=1`) 时，使用 `torch.cuda.synchronize()` 或 `torch.cuda.current_stream().query()` 检查 `lora_event` 是否成功记录，防止图重放时遗漏。  
- **性能基准**：在实际部署前，对比以下三种配置的 latency：<br>① 单流 BF16（baseline）<br>② 两流 BF16（开启）<br>③ 两流 BF16 + `SGLANG_OPT_LORA_PREFILL_ROUTING_REUSE=0`（关闭复用），以确认环境变量对目标硬件的实际收益。  
- **异常捕获**：在生产环境中捕获 `TVM_FFI_ICHECK` 报错，给出清晰提示（如 “gate_up_lora_delta 必须为 bf16 且连续”），避免因数据类型不匹配导致的进程崩溃。  
- **显存预留**：若使用高 rank（>64) 的 LoRA，建议在模型启动时预留额外 10%‑15% 的显存供 side‑stream 中间 buffer 使用，防止 OOM

### [Cleanup] IPC struct renames, better typing, and SenderWrapper removal (#29214)
**SHA**: `6c83936` | 🔗 [查看提交](https://github.com/sgl-project/sglang/commit/6c839368e04e5b5fa0d444d223c3ec80c5be650d)

**🎯 变更类型**：功能增强 / 重构 / 架构变更  
**⚡ 重要程度**：🔴高  

**📋 变更摘要**  
本次提交对 SGLang 核心模块的 IPC（进程间通信）结构进行全局清理与重命名，统一使用 `http_worker_ipc` / `http_worker_ipcs` 进行路由信息标记；引入 `Annotated[..., Body()]` 以提升 FastAPI 参数的类型安全和自动文档生成；去除历史的 `SenderWrapper`，改为轻量级的 `stamp_http_worker_ipc` 辅助函数；相应地更新了多个管理类（`TokenizerManager`、`MultiTokenizerMixin`、`Scheduler` 等）的发送路径、结构体字段以及类型注解。整体提升了代码可维护性、类型检查力度以及运行时的可观测性。

**🎯 影响范围**  
- `python/sglang/srt/io_struct.py`（所有 IPC 请求/响应结构体）  
- `python/sglang/srt/managers/communicator.py`（FanOutCommunicator）  
- `python/sglang/srt/managers/tokenizer_manager.py`、`multi_tokenizer_mixin.py`、`scheduler.py`、`scheduler_components/output_streamer.py`、`scheduler_input_blocker.py`（发送/接收路径）  
- 所有 FastAPI 路由入口（`http_server.py`、`encode_server.py`）  
- 依赖这些结构体的测试、示例及第三方集成代码  

**🔍 技术洞察**  

- **架构影响**  
  1. **统一路由标记**：原先的 `SenderWrapper` 在发送对象时自动写入 `http_worker_ipc`，现在改为显式在 `TokenizerManager._dispatch_to_scheduler` 中调用 `stamp_http_worker_ipc`。这使得路由信息的注入点更明确，降低了隐式副作用，提升了模块解耦度。  
  2. **结构体层级简化**：`BaseReq`、`BaseBatchReq` 取消了对 `ABC` 的继承，去除了冗余的抽象方法；`rid` 与 `rids` 的唯一性检查被迁移到具体请求类（如 `GenerateReqInput`），避免在批处理结构体中重复实现。  
  3. **命名统一**：`PauseContinueBroadcast` → `PauseContinueBroadcastReq`、`TokenizerWorkerRegistration` → `TokenizerWorkerRegistrationReq` 等，所有 “Req/Output” 后缀统一，便于在类型检查和文档生成时快速辨识。  
  4. **类型强化**：大量使用 `Annotated[..., Body()]`、`Union`、`Literal`、`Optional` 等标准 `typing`，配合 Pydantic 的 `field` 机制，提升了运行时校验和编辑器提示。  

- **性能影响**  
  - **微小正向提升**：去掉 `SenderWrapper` 的包装层后，发送方直接调用 `sock_send`，减少一次函数调用和属性检查，理论上对高并发场景的 IPC 带宽影响可忽略。  
  - **潜在负面**：如果在新代码路径中遗漏对 `stamp_http_worker_ipc` 的调用，调度器将无法依据 `http_worker_ipc` 正确路由响应，导致请求卡死或被错误分配；这属于逻辑错误而非性能回退。

  - **输入校验更严格**：FastAPI 现在通过 Pydantic 自动验证请求体字段类型，降低了因手动解析导致的注入风险。  
  - **路由信息暴露**：`http_worker_ipc` 字段仍是明文传递的内部标识符，若攻击者能够构造恶意请求并填入任意 IPC 名称，理论上可以尝试把响应路由到未授权的 worker。项目已有的权限校验（`@auth_level` 装饰器）依旧生效，未因结构体改动而削弱。建议在接收端再次校验 `http_worker_ipc` 是否在白名单范围。

5. **单元/性能回归**：新增共享池后，捕获/回放阶段的内存峰值应该保持不变或下降。请在 CI 中加入对 `prefill` 与 `decode` 连续运行的基准，验证“两个阶段永不并发”的假设。  
6. **文档/示例更新**：在使用 `prefill` 与 `decode` 时说明不需要手动创建池，且在多 GPU 环境下需自行管理。  

综上，此次重构提升了资源利用率，但需注意并发安全和多 GPU 场景，建议补充相应防护并在 CI 加入回归测试。
