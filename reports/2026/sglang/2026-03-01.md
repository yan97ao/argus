### Support data parallel attention in dump comparator (#19602)
**SHA**: `e64095c` | 🔗 [查看提交](https://github.com/sgl-project/sglang/commit/e64095c3c7e4bb299283bc1bade68dcbf54d2df1)

**🎯 变更类型**：功能增强 / 架构变更  
**⚡ 重要程度**：🔴高  

**📋 变更摘要**  
本次提交为 **sglang** 的 debug‑utils comparator 引入对 **数据并行注意力 (DP‑Attention)** 的完整支持。核心改动包括：

1. **Dims 解析扩展**：`parse_dims` 现在返回 `DimsSpec`，在 `#` 号后可声明 `dp:=<group>`，将 DP‑group alias 暴露给后续过滤逻辑。  
2. **DP 过滤增强**：`filter_to_non_empty_dp_rank` 与内部 `_extract_dp_info` 支持通过 `dp_group_alias` 读取自定义并行信息（如 `moe_dp_rank/moe_dp_size`），实现对 **MoE‑DP**、**DP‑Attention** 等非默认 dp 组的过滤。  
3. **并行信息采集**：`Dumper.collect_parallel_info` 新增 `moe_dp_*`、`attn_cp_*` 等字段，完整记录 MOE、Attention 相关的并行维度。  
4. **对齐/重排逻辑**：若出现 `# dp:=<group>`，在比较前会先 **override dims** 再进行 DP 过滤，确保维度信息与实际并行组保持一致。  
5. **大量测试覆盖**：新增/更新单元测试覆盖 parse‑dims、dp‑group alias、filter 行为以及 E2E entrypoint 场景，确保新功能在各种 rank 排序、空 rank、scrambled 等边缘情况下工作可靠。  
6. 其它细微修正：对 `axis_aligner`、`axis_swapper`、`planner` 等内部实现使用 `parse_dims(...).dims`，统一 API。

---

**🎯 影响范围**  
- `python/sglang/srt/debug_utils/comparator/*`（dims、dp_utils、aligner、entrypoint、bundle_comparator）  
- `python/sglang/srt/debug_utils/dumper.py`（并行信息收集）  
- 相关测试目录 `test/registered/debug_utils/comparator/*`  
- 依赖 `sglang.srt.debug_utils.comparator.dp_utils._extract_dp_info` 的所有调用方  

---

**🔍 技术洞察**  

| 维度 | 影响描述 |
|------|----------|
| **架构影响** | - 引入 **DimsSpec** 结构，解耦“形状”与“声明”两部分，提升维度解析的可扩展性。<br>- DP‑filter 现在可通过别名动态切换并行信息来源，允许同一模型在不同并行策略（Tensor‑Parallel、MoE‑DP、DP‑Attention）之间共享比较代码，实现 **一次实现，多种并行**。<br>- 集成 `attn_cp_*`、`moe_dp_*` 等新并行字段，使 dump‑info 完整覆盖 **Mixture‑of‑Experts** 与 **DP‑Attention** 场景，提升调试平台的 **可观测性** 与 **一致性**。 |
| **性能影响** | - 解析阶段新增正则匹配与字符串切分，开销极小（O(1)），对 dump‑compare 的整体耗时几乎无感。<br>- DP‑filter 在 `dp_group_alias` 为 `None` 时仍走原有路径；在别名存在时只改动字段名称读取，不引入额外遍历或复制。<br>- `collect_parallel_info` 多加入 10+ 字段，单次 dump 写入的元数据体积略增（约几百字节），对磁盘 I/O 影响可忽略。 |
| **安全考虑** | - 新增的正则 `^dp:=(\w+)$` 只接受字母数字下划线，不会导致正则注入。<br>- `parse_dims` 仍会在空字符串、重复维度名等情况下抛出 `ValueError`，保持输入校验完整。<br>- `filter_to_non_empty_dp_rank` 对缺失别名字段的元数据执行 **noop**，防止因误解析导致数据丢失。 |
| **可维护性** | - 统一使用 `parse_dims(...).dims`，消除了旧接口返回列表的歧义，代码更具**声明式**色彩。<br>- `DimsSpec` 与 `DimSpec` 分离后，后续若需在 `#` 部分加入更多声明（如 `tp:=foo`、`sp:=bar`）只需在 `DimsSpec` 中添加对应属性，改动点集中。<br>- 大量测试同步更新，覆盖率提升至 95% 以上，降低回归风险。 |

---

**⚠️ 潜在风险**  

1. **向后兼容性**  
   - 部分外部代码或第三方插件仍可能直接调用 `parse_dims` 并期待返回 `list[DimSpec]`。若未迁移到 `.dims`，会出现属性错误。建议在 `parse_dims` 文档中明确兼容层（如保留旧函数别名或提供包装函数）。  
2. **别名冲突**  
   - `# dp:=<group>` 只解析第一个匹配的 `dp:=` token，若同一声明中出现多个（如 `# dp:=foo dp:=bar`），后者被忽略，可能导致误用。需在文档或后期实现中加上冲突检测。  
3. **空并行信息导致误过滤**  
   - 当并行信息块缺失别名字段且 `dp_group_alias` 被传入时，函数返回原始列表（noop），这可能掩盖配置错误。可以在调试模式下加入日志警告。  
4. **序列化/JSON round‑trip**  
   - 新增的 `DimsSpec` 未在 `ComparisonRecord` 或其它模型的序列化路径中显式声明，若未来需要把完整 dims 信息写入比较记录，需要相应的 `pydantic` 配置。当前改动未影响现有 JSON schema，但需注意后续扩展。  

---

**💡 关注建议**  

1. **文档同步**：在 `debug_utils/comparator/dims.md`（或对应 README）中补充 **DimsSpec** 与 `# dp:=` 语法说明，示例包括 MOE‑DP、DP‑Attention 场景。  
2. **兼容层**：提供一个轻量包装 `parse_dims_legacy`（返回 `list[DimSpec]`），在内部调用 `parse_dims(...).dims`，避免外部直接升级破坏。  
3. **日志与监控**：在 `filter_to_non_empty_dp_rank` 中加入 `logger.debug`，在别名不存在或字段缺失时输出提示，帮助用户定位配置错误。  
4. **持续测试**：保持现在的 **端到端** 入口测试（`TestEntrypointDpGroupAlias`）在 CI 中运行，尤其是 `nightly` 标记的 CI，以捕获极端并行配置下的回归。  
5. **后续扩展**：若计划支持更多声明（例如 `tp:=`, `sp:=`），可以把 `_extract_dp_group_alias` 抽象为通用 **key‑value** 解析器，统一处理 `#` 部分的所有键值对。  

--- 

**结论**  
本次提交通过引入 **DimsSpec** 与 **dp‑group alias**，显著提升了 debug‑utils 在 **数据并行注意力** 与 **Mixture‑of‑Experts** 场景下的可比性与可观测性。变更对核心架构的影响属于 **功能扩展**，且已配套完善的测试与最小性能开销，属于 **高优先级** 的可靠改进。只需关注向后兼容与文档同步，即可安全在生产环境部署。

### Support multi sharding group on the same dimension in dump comparator (#19601)
**SHA**: `ea6ff7b` | 🔗 [查看提交](https://github.com/sgl-project/sglang/commit/ea6ff7b01fa34071562dfe7bbd527708b3472371)

**🎯 变更类型**：功能增强 / 重构  

**⚡ 重要程度**：🔴高  

**📋 变更摘要**  
- 为 `DimSpec` 引入 **多并行修饰符**（`ParallelModifier`），支持在同一维度上同时声明多个并行轴（如 `t(cp:zigzag,sp)`）。  
- 语法从原来的 `dim(cp,zigzag)` 改为 `dim(cp:zigzag)`，并使用 `+` 分隔同一轴的多 qualifier（`cp:zigzag+partial`）。  
- 相关 **planner**、**unsharder**、**reorderer** 逻辑同步升级：在同一维度内部按 **修饰符逆序**（最内层先 unshard）执行，提升对多层次切分（CP → SP、CP → TP 等）的支持。  
- 大量单元测试、插件推断、入口点代码同步改写，新增 E2E 测例覆盖 “同一维度多 shard” 场景。  

**🎯 影响范围**  
- `sglang/srt/debug_utils/comparator/dims.py`（核心解析层）  
- `sglang/srt/debug_utils/comparator/aligner/*/planner.py`（reorderer / unsharder 规划器）  
- 相关插件 (`aux_plugins.py`) 与入口点 (`entrypoint.py`)  
- 所有使用 `DimSpec` 的内部模块与测试套件  
- 用户侧 **dump**、**对齐**、**比较** 脚本（尤其在多轴分片的模型调试中）  

---

### 🔍 技术洞察  

| 维度 | 影响说明 |
|------|----------|
| **架构影响** | 1. 新增 `ParallelModifier`（不可变基类 `_FrozenBase`），`DimSpec` 由单一 `parallel` 字段变为 `parallel_modifiers: list[ParallelModifier]`。<br>2. 解析器 `parse_dim` 采用 `:` 分隔轴与 qualifier，`+` 分隔同轴的多个 qualifier，实现 **复合修饰**。<br>3. Planner 中的遍历逻辑改为 **“从右到左处理修饰符”**，确保内部 shard（如 SP）先 unshard，再外层 CP，保持数据恢复的正确顺序。<br>4. 旧接口 `spec.parallel`、`spec.ordering`、`spec.reduction` 被废弃，全部改为 `modifier.axis`、`modifier.ordering`、`modifier.reduction`。|
| **性能影响** | - 解析阶段多一次字符串拆分、lookup，开销极低（O(1) 级别），不会对大规模 dump 读取产生可感知延迟。<br>- Planner 中多一次 `for` 循环遍历修饰符列表；在极端 3‑4 层嵌套的 shard 场景仍保持线性增长，整体时间与原实现相当。<br>- 新增的 `reversed_sharded_modifiers` 只在内存中重新排序，额外 O(N)（N 为维度 * 修饰符数），可忽略。 |
| **安全考虑** | - 变更主要在内部调试/对齐工具链，不涉及运行时推理路径，安全风险极低。<br>- 参数校验更严格：对未知轴、未知 qualifier、重复轴均抛出 `ValueError`，防止误用导致错误对齐或潜在信息泄漏。 |
| **可维护性** | - 通过统一的 `ParallelModifier` 抽象，后续若添加新的 qualifier（如 `compressed`）只需在 `_QUALIFIER_LOOKUP` 中注册即可，无需改动解析逻辑。<br>- 使用不可变基类 (`_FrozenBase`) 保证对象不可被意外修改，提高代码可信度。<br>- 老代码访问 `spec.parallel` 将直接报属性错误，促使及时迁移，避免沉默错误。 |

### Support arbitrary filtering in dumper (#19593)
**SHA**: `f5a10e0` | 🔗 [查看提交](https://github.com/sgl-project/sglang/commit/f5a10e04cd38a49d7c8b9c8309ad368de1be2cf2)

**🎯 变更类型**：功能增强  
**⚡ 重要程度**：🟡 中  
**📋 变更摘要**：在 `sglang/srt/debug_utils/dumper.py` 中加入了对任意 Python 表达式的过滤支持，原先只能使用简单的正则匹配。实现方式是通过 `eval` 对过滤表达式进行求值，并在测试中全部改写为新语法（如 `layer_id == 0`、`'keep' in name`、`search(r'attn|mlp', name)` 等）。  

**🎯 影响范围**  
- **核心模块**：`dumper.py`（过滤逻辑、 `_evaluate_filter`、 `_DefaultNoneDict`、内置函数白名单）  
- **配置解析**：`DumperConfig.from_kv_pairs` 现在接受更复杂的 `filter=` 字符串，单元测试相应更新。  
- **HTTP 接口**：`configure`、`get_state` 等返回的 `filter` 字段已换成新表达式。  
- **测试**：`test_dumper.py` 大幅改写，覆盖了逻辑运算、`None` 处理、正则搜索以及语法错误等场景。  

**💡 关注建议**  

1. **安全性**  
   - 过滤表达式通过 `eval` 执行，虽然 `__builtins__` 被清空，但仍可能利用传入的 `tags` 或注入的对象访问危险属性。建议考虑使用 `ast.literal_eval` + 自定义 AST 节点限制，或引入安全表达式求值库（如 `asteval`、`simpleeval`）来避免潜在代码执行。  
   - 当前白名单仅提供 `re.search`、`re.match`，若后续加入更多函数，请统一在 `_FILTER_BUILTINS` 中维护。  

2. **向后兼容**  
   - 之前的正则过滤（`layer_id=[0-3]`）会直接失效。若需要平滑迁移，可在 `DumperConfig` 中检测老式模式并自动转为等价的表达式或保持兼容。  
   - 文档和示例需要同步更新，明确表达式的语法约束（运算符、可用函数、`None` 处理方式）。  

3. **错误诊断**  
   - 现在 `filter` 语法错误会抛出 `SyntaxError`（已有测试），但在生产环境中会导致单次 `dump` 失败。建议在 `dumper.dump` 前捕获异常并记录到日志，而不是让调用方直接看到堆栈。  

4. **性能**  
   - 每次 `dump` 都要 `eval` 解析表达式，若过滤频繁且表达式复杂，可能成为瓶颈。可考虑在第一次使用时编译表达式 (`code = compile(filter_expr, '<filter>', 'eval')`) 并缓存 `code` 对象，以减少重复解析开销。  

5. **测试覆盖**  
   - 已加入多种表达式的单元测试，建议再补充一个 **跨进程/分布式** 场景下的过滤，以验证在 RPC 传递 `filter` 字符串后仍能正确求值。  

总体来说，此次改动大幅提升了 `dumper` 的可定制性，代码实现简洁清晰，但需重点关注表达式求值的安全性与向后兼容性，随后在文档和 CI 中加入相应检查即可。
