### feat: [2/2][DeepEP] Add waterfill load balancing for shared expert dispatch (#19290)
**SHA**: `c701a08` | 🔗 [查看提交](https://github.com/sgl-project/sglang/commit/c701a08765ba8a4496c0e392cba55db332b2ad95)

**🎯 变更类型**：功能增强  

**⚡ 重要程度**：🔴 高  

**📋 变更摘要**：  
本次提交为 DeepEP（DeepSeek‑EP）引入 *Waterfill* 负载均衡机制，使共享专家（shared expert）能够作为第 9 个路由专家被调度到当前最空闲的 EP（Expert Parallel）节点。实现方式包括：  
1. 新增 `deepep_waterfill.py`，实现基于 Triton 的 fused kernel，用于一次性完成 rank 负载统计、Waterfill 目标 rank 选取以及 Top‑K 扩展。  
2. 在 `TopK`、`ModelRunner`、`ServerArgs` 等核心模块中加入开启开关、自动后端切换、默认开启共享专家融合等逻辑。  
3. 新增环境变量 `SGLANG_DISABLE_STATIC_WATERFILL`，可强制走动态 Waterfill（基于全局 all‑reduce 的运行时负载）。  

预期效果是 **在多 EP 场景（EP ≥ 2）下均衡共享专家的计算负载，提升解码吞吐与延迟，尤其在 DeepSeek‑V3/R1 大模型的高并发推理中发挥作用**。

---

### 🎯 影响范围
- **核心调度层**：`sglang/srt/layers/moe/topk.py`、`sglang/srt/layers/moe/deepep_waterfill.py`  
- **模型执行器**：`sglang/srt/model_executor/model_runner.py`（准备 Waterfill balancer）  
- **服务器配置**：`sglang/srt/server_args.py`（新增 `--enable-deepep-waterfill` 旗标、自动后端切换、强制共享专家融合）  
- **环境变量**：`sglang/srt/environ.py`（`SGLANG_DISABLE_STATIC_WATERFILL`）  
- **文档**：`docs/advanced_features/server_arguments.md`（说明新旗标）  
- **单元测试**：`test/registered/unit/server_args/test_server_args.py`（验证新参数交互）

### ⚠️ 潜在风险
1. **小批次性能回退**：在 `num_tokens < 64` 时仍会执行 `expand_topk_with_shared_expert`（本地扩展）或直接走静态 waterfill。若 Triton kernel 启动成本高于直接本地扩展，可能出现 **轻微延迟增加**。  
2. **全局 all‑reduce 争用**：在大规模多机 EP 部署（> 8 节点）且网络带宽受限时，动态 waterfill 的两次 all‑reduce 可能成为 **瓶颈**，导致整体吞吐下降。  
3. **显存拼接错误**：`expanded_topk_ids`/`expanded_topk_weights` 维度由 `[N, 8]` 直接扩展到 `[N, 9]`，若下游模块仍假设固定的 `topk=8`（例如自定义 schedule、日志或分析脚本），会触发 shape mismatch。  
4. **错误的 `rank_load` 计数**：`_count_routed_per_rank_kernel` 在极端情况下（expert_id 为负值且被错误计入）可能导致统计偏差，从而把共享专家分配到不恰当的 EP。  
5. **环境变量冲突**：`SGLANG_DISABLE_STATIC_WATERFILL=1` 强制走动态路径，若用户未准备好网络环境或未同步 **all‑reduce** 实现，可能出现 **死锁**。

   - `batch_info.lora_ranks` 允许每个 slot 使用不同 rank。内核在每个 segment 通过 `tl.minimum(N, cur_rank)` 动态裁剪，但 **`cur_rank` 为 0** 时直接返回；若出现 `rank` 大于块大小导致 **未对齐的 tail**，可能出现残余未累加的列。需要通过单元测试验证所有 `rank` 范围（16‑128）均无精度泄漏。  

3. **数值精度**  
   - `tl.dot` 采用 **fp32 累加**，但最终结果转换回输入 dtype（bf16/fp16）。在极端 **长序列**、**大 batch** 情况下仍可能出现 **累加误差**，尤其是 `scaling` 为非 1 的 LoRA 系数。建议在关键路径加入 **相对误差检查**（如 unit‑test 对比不使用 LoRA 的基准）。  

4. **显存峰值**  
   - 对于 **多 LoRA slot**（如 8‑16 adapters 同时激活）以及 **大 batch**，`A_buffer`、`B_buffer`、`out` 的临时分配会显著提升显存占用。若显存不足会导致 OOM。  
   - `step_b_v_fwd` 直接在 `base_output` 上 `+=`，在 **torch.autocast** 或 **梯度检查点** 场景下可能产生 **in‑place 与梯度冲突**（虽然前向不需要梯度），需要确认在训练模式下不会触发 `RuntimeError: a leaf Variable that requires grad has been used...`。  

5. **分段（segment）路由错误**  
   - `batch_info.seg_indptr`、`weight_indices`、`sorted_token_ids` 必须保持 **严格的一致性**。若上层代码在切换 LoRA 适配器后未同步这些信息，可能导致 **错误的 token‑slot 对齐**，进而产生 **结果错位**。

| 风险 | 说明 | 缓解/建议 |
|------|------|----------|

### [UnifiedRadixCache] Fix HiCache load back start node (#25088)
**SHA**: `e4378ff` | 🔗 [查看提交](https://github.com/sgl-project/sglang/commit/e4378ff37f61891cfc70fefceb7336fc8acfe021)

**变更概览**  
本次提交为 HiCache（统一前缀缓存）引入了 *best_match_node* 概念，用来统一 “匹配后最深的合法节点” 作为后续 **load_back**、锁定等操作的锚点。为此在 `MatchResult`、各缓存实现（radix、hi‑radix、mamba、swa、chunk、storage）以及调度层（`schedule_batch.py`、`schedule_policy.py`）中大量替换了 `last_node / last_host_node` 的传递方式，并在 `UnifiedRadixCache` 的匹配、后处理、加载回滚等核心路径中同步使用。测试用例也相应补全了对 `best_match_node` 的断言。

**核心影响**  
- **统一锚点语义**：`best_match_node` 取代了原来在不同缓存层级分别使用的 `last_node`、`last_host_node`，避免了因组件验证不一致导致的锚点错误。

- **回退路径**：对 evicted‑anchor 的回滚逻辑做了显式处理（例如 `while best_match_node.evicted`），但仍有 `TODO` 标记提醒后续完善。  

**潜在风险**  
1. **遗漏更新**：仍有少量路径（如某些自定义插件或外部调用）可能仍依赖 `last_node`/`last_host_node`，在未同步的情况下会产生不一致。  
2. **锁的偏移**：`skip_lock_node_ids` 的生成和使用较为细致，若验证器或组件新增导致 `value` 为 `None` 的情况未被覆盖，可能出现锁计数不匹配。  
3. **性能影响**：在 `match_prefix` 中多次复制 `best_match_node`，但影响微乎其微；仍建议在高并发测试中监控锁争用与迁移延迟。  
4. **TODO 未完成**：如 `init_load_back` 中的“fallback path is not correct”提示，未来需要更细致的 device‑host 多组件兼容实现。

**建议**  
- **开发者**：在所有自定义缓存或外部插件中搜索 `last_node`、`last_host_node`，确认已改为 `best_match_node`；对 `acquire_component_lock` 的 `skip_lock_node_ids` 逻辑补充单元测试（包括多组件交叉验证的场景）。
