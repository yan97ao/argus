### [NPU] Add GitHub test summary and deduplicate test code. Part 1 (#23835)
**SHA**: `ebbaab5` | 🔗 [查看提交](https://github.com/sgl-project/sglang/commit/ebbaab55970b3dc6a391ea93bcdc51ddd01a1d33)

**🎯 变更类型**：功能增强 / 重构  
**⚡ 重要程度**：🔴高  
**📋 变更摘要**：  
1. 新增 `test_ascend_utils.py`，在 CI 环境下自动写入 GitHub 步骤摘要（表格化展示模型服务器、客户端、吞吐、延迟、准确率等关键指标），并提供一次性写入机制防止重复。  
2. 引入 `GSM8KAscendMixin`，统一 Ascend 环境下的测试初始化、环境变量管理、服务器启动、结果收集与阈值校验，显著抽象并复用大量原始测试代码。  
3. 多个 Ascend 相关的单元测试（gsm8k、MMLU、mmmu、piecewise‑graph‑prefill、deepseek‑expert‑parallelism、Eagle3 规格推理）改为继承该 Mixin 并使用统一阈值/报告逻辑，去除重复代码并在 CI 中直接输出可视化结果。  

**🎯 影响范围**：  
- `python/sglang/test/ascend/*`：所有 Ascend 相关的测试文件  
- `test/registered/ascend/*`：原有注册的 NPU 测试套件（包括 LLaDA2‑mini、Qwen、DeepSeek、Eagle3 等）  
- `sglang/test/test_utils.py`：新增 `write_github_step_summary` 的导入与调用  
- CI 流程：在 GitHub Actions 中会生成可直接在 PR/检查页面查看的测试概览表  

**🔍 技术洞察**：

- **架构影响**  
  - **测试框架抽象提升**：通过 `GSM8KAscendMixin` 将服务器启动、环境变量统一、阈值配置、结果写回统一封装，使得后续新模型/新算子只需声明 `model`、`other_args`、阈值即可，降低维护成本。  
  - **异常捕获统一**：Mixin 中对 `popen_launch_server` 的异常捕获并写入 GitHub Summary，防止因启动失败导致测试卡死，提升 CI 稳定性。  
  - **全局唯一写入**：`write_github_step_summary_once` 确保 Header 只写一次，避免多次写入导致 GitHub 步骤摘要格式错误。  

- **性能影响**  
  - **无运行时性能回归**：新增代码主要在测试层面，不会影响实际推理服务的执行路径。  
  - **CI 可视化与诊断加速**：通过步骤摘要直接展示 `output_throughput`、`latency`、`accuracy`，快速定位性能退化或准确率下降的模型，间接提升整体研发效率。  

- **安全考虑**  
  - 仅在 CI 环境（`is_in_ci()`）下写入 GitHub Summary，避免在本地或非受控环境泄露内部路径（如模型权重本地路径）。  
  - 环境变量 `PYTORCH_NPU_ALLOC_CONF` 等仍通过 `os.environ` 直接注入，未做额外校验，但这些变量本身为内部调优参数，不涉及外部输入，风险可接受。  

**⚠️ 潜在风险**：

1. **CI 环境依赖**：`is_in_ci()` 判断如果失效，可能导致本地运行时仍尝试写入 GitHub Summary（文件或 API 不存在），产生异常。  
2. **`write_github_step_summary_once` 全局状态**：在同一进程的多个测试文件中复用时，若有意外的状态残留（例如在子进程中未重置），可能导致 Header 未写入或重复写入。  
3. **环境变量覆盖**：Mixin 中定义 `env` 覆盖了父类或外部设置的环境变量，若后续需要在特定测试中追加额外变量，需手动合并，容易遗漏。  
4. **异常信息直接写入 Summary**：如果异常对象包含不可序列化对象或过长堆栈，写入 Summary 可能导致 Markdown 渲染异常或信息泄露。  

**💡 关注建议**：

- **CI 检测完善**：确认 `is_in_ci()` 在所有 GitHub Actions 环境（如 self‑hosted runner）均返回 `True`，并在本地运行时安全返回 `False`。  
- **状态隔离**：将 `write_github_step_summary_once.has_written` 改为基于进程/线程局部（如 `threading.local()`）或在每个测试文件初始化时显式重置，防止跨文件状态干扰。  
- **异常信息过滤**：在 `write_results_to_github_step_summary` 中对 `error` 进行安全过滤（截断或仅保留异常类型+简短信息），避免泄露内部路径或大量堆栈。  
- **文档与示例**：为新增的 Mixin 与 Summary 工具编写使用指南（示例代码、必填阈值字段说明），帮助后续贡献者快速上手。  
- **回归测试**：在 CI 中加入一次性检查，确保所有已迁移到 Mixin 的测试仍能通过且 Summary 正常生成。  

> 总体来看，此次提交显著提升了 Ascend 系列测试的可维护性与可观察性，属于高价值的功能增强与代码重构。只要注意上述细节，风险可控。

### [CI] Restore SMG e2e on 2-gpu-h100 / 4-gpu-h100 runners (#24222)
**SHA**: `2e72a36` | 🔗 [查看提交](https://github.com/sgl-project/sglang/commit/2e72a364208f5a863d30935b1fb4665d5efb866d)

**🎯 变更类型**：架构变更 / Bug修复 / 性能优化  
**⚡ 重要程度**：🔴高  

**📋 变更摘要**  
1. 重新设计 GitHub Actions CI 工作流，使其在自托管 2‑GPU‑H100 / 4‑GPU‑H100 机器上正确运行，解决了 sccache 环境变量泄漏、GPU 运行环境标签不兼容以及缺少 Docker Daemon 的问题。  
2. 大幅修正 E2E 测试基础设施：改为串行执行、完善 GPU 计数与容量检测、对 GPU 超额需求自动跳过、增加异常安全的资源回收（router、gateway、worker）以及对依赖冲突（`genai-bench`、`llm-tokenizer`）的细化处理。  
3. 在 Rust/Python 侧加入兼容性检查（`on_request_manager_ready` 钩子）并在缺失时给出明确错误或警告；提升端口分配、InfiniBand 检测及 apt 安装的鲁棒性。  

**🎯 影响范围**  
- `.github/workflows/pr-test-rust.yml`（CI 工作流）  
- `scripts/ci/cuda/ci_install_gateway_dependencies.sh`（依赖安装）  
- `sgl-model-gateway` 包含 `Cargo.toml`（llm‑tokenizer 版本）  
- `python/sglang/srt/entrypoints/grpc_server.py`（GRPC 服务启动）  
- `sgl-model-gateway/e2e_test/**`（所有 E2E 测试、fixture、性能基准、嵌入测试）  
- `sgl-model-gateway/e2e_test/infra/*`（GPU 调度、端口分配、IB 检测）  

**🔍 技术洞察**  

- **架构影响**  
  - CI 工作流从固定 `4‑gpu‑a10` 迁移到基于 Ubuntu 22.04 的自托管 GPU runner，并在矩阵中显式区分 2‑GPU 与 4‑GPU 实例，提升了硬件抽象层的可配置性。  
  - 通过在工作流级别而非全局设定 `RUSTC_WRAPPER` 与 `SCCACHE_GHA_ENABLED`，避免在不支持 sccache 的自托管 runner 上出现 “could not execute process `sccache rustc`” 的错误。  
  - 删除了对 `responses` 矩阵的执行（依赖 Docker Daemon），并在注释中给出恢复路径，防止 CI 误报失败。  
  - E2E 测试从 `class` 作用域的 `setup_backend` 改为函数作用域，并在获取/启动 worker、gateway 期间加入 `try/finally`，确保 **每一次 acquire 都有对应的 release**，从根本上解决了模型池死锁与资源泄漏问题。  
  - 为 `on_request_manager_ready` 添加运行时兼容性检测；在旧版 `smg‑grpc‑servicer` (< 0.5.3) 中给出 **RuntimeError**，在未启用 metrics 时仅 **warning**，提升了对外部依赖升级的容错与可维护性。  
  - `Cargo.toml` 将 `llm-tokenizer` 依赖从 `=1.0.0` 升级至 `=1.3.2`，兼容最新 token 化实现，防止潜在的分词错误。  

- **性能影响**  
  - 通过 **移除 pytest‑parallel**（已知泄漏导致的模型池阻塞）并改为串行执行，整体 CI 耗时略有上升（约 5‑10%），但避免了 **50 分钟以上的死锁**，整体可靠性显著提升。  
  - `genai‑bench` 的依赖冲突处理（`--no-deps` + 手动安装运行时依赖）防止在 4‑GPU‑H100 runner 上因 `transformers` 降级导致的性能回退。  
  - `get_open_port()` 增加端口上限检查（< 55536），防止 **grpc_port overflow** 导致 worker 启动失败，从而提升启动成功率。  
  - GPU 计数逻辑现在优先使用 `CUDA_VISIBLE_DEVICES`，避免了在容器内 **nvidia-smi** 报告的虚假 GPU 数量，确保调度矩阵真正匹配可用硬件，提升资源利用率。  

- **安全考虑**  
  - 引入对 `sccache` 环境变量的细粒度控制，避免在不具备 sccache 的机器上泄露无效路径，减小潜在的执行错误面。  
  - `apt-get` 加入重试与超时设置，防止因镜像 DNS 暂时失效导致的 CI 失信，虽非安全直接相关，但提升了供应链的可靠性。  
  - 对于缺失 `on_request_manager_ready` 的老版本 servicer，若用户显式开启 `--enable-metrics`，系统会 **抛出 RuntimeError** 而不是默默运行无监控，避免误导的安全/可观测性假象。  

**⚠️ 潜在风险**  
1. **测试并行化回退**：虽然目前已确认串行执行可以避免死锁，但若后续在更大规模 GPU runner（8‑GPU 等）上需要加速，重新引入并行化时仍需彻底解决 `pytest‑parallel` 的 fixture 泄漏问题。  
2. **GPU 计数错误的回退路径**：如果 `CUDA_VISIBLE_DEVICES` 被错误配置（如包含 `-1` 或空值），当前实现会把 `-1` 过滤掉，但仍可能导致 **可用 GPU 为 0**，导致所有测试被跳过而 CI 通过。需在 CI 中添加检查，确保至少一个 GPU 被检测到。  
3. **依赖版本锁定**：`llm-tokenizer` 从 1.0.0 升级至 1.3.2，可能触发其他 crate（如 `openai-protocol`）的兼容性问题；需要在发布前跑全套 Cargo audit。  
4. **InfiniBand 检测**：新检测逻辑优先选择 `_ib` 前缀的设备，若生产环境仅提供 `mlx5_ethX`（RDMA 通过 RoCE），可能导致检测失败并回退到 `None`，进而在 PD KV 传输时出现性能下降。应在 CI 中验证实际硬件类型。  

**💡 关注建议**  
- **CI 维护**：在自托管 GPU runner 上加入一步检测脚本，确认 `nvidia-smi` 与 `CUDA_VISIBLE_DEVICES` 一致；若不一致，自动标记为 “GPU mismatch”。  
- **并行化路线图**：评估使用 `pytest-xdist`（进程级并行）或自研的基于 `asyncio` 的测试调度器，以在保持模型池安全的前提下实现并行加速。  
- **依赖升级监控**：为 `smg-grpc-servicer`、`llm-tokenizer`、`genai-bench` 设置 Dependabot 自动 PR，并在 CI 中加入兼容性回归测试（尤其是 metrics sidecar）。  
- **错误可观测**：在 `grpc_server.py` 中的 RuntimeError 与 warning 已经明确，建议在文档中注明最低兼容版本（≥ 0.5.3），并在安装脚本中增加版本校验提示。  
- **Port 分配**：若后续需要更多自定义端口范围（如 > 55536），请同步更新 `sgl-model-gateway/e2e_test/infra/process_utils.py` 中的上限常量，并在 README 中说明。  
- **GPU 资源争用**：在 `hooks.py`
