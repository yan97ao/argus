### refactor(jit_kernel): reduce duplication and separate test code (#19323)
**SHA**: `74c8e7b` | 🔗 [查看提交](https://github.com/sgl-project/sglang/commit/74c8e7b2152471048c2275d3cb58d07e9281f715)

**🎯 变更类型**：重构  
**⚡ 重要程度**：🔴高  

**📋 变更摘要**  
- 删除了 `cuda_wait_value.cuh` 与对应的 Python 包装层 `cuda_wait_value.py`，并将其内部的测试代码抽离出库主体。  
- 在 `hadamard.py` 中引入统一的内部实现 `_hadamard_transform_impl`，通过函数指针统一处理不同倍率（8、12n、20n、28n、40n）的 Hadamard 变换，显著降低代码重复度。  

**🎯 影响范围**  
- `python/sglang/jit_kernel/csrc/`（CUDA kernel 相关）  
- `python/sglang/jit_kernel/`（Python JIT 接口）  
- `python/sglang/jit_kernel/hadamard.py`（Hadamard 变换）  

**🔍 技术洞察**  

- **架构影响**  
  - **模块解耦**：原本 `cuda_wait_value` 既提供 JIT 加载又携带测试逻辑，现在被完全移除，库的核心层不再包含任何 “demo‑style” 代码，保持了 cleaner 的 package 结构。  
  - **统一实现**：Hadamard 相关函数现在统一走 `_hadamard_transform_impl`，仅在入口层决定 pad‑multiple 与具体 kernel。这样如果以后需要添加新倍率，只需在入口函数中声明即可，减少了维护成本。  
  - **API 稳定性**：`hadamard_transform` 仍保持原有签名，兼容现有调用者；新增 `hadamard_transform_12n/20n/...` 也保持不变。唯一可能受影响的是外部直接 import `stream_wait_value` 或 `Event`（自定义实现），这些符号已被删除。  

- **性能影响**  
  - **Hadamard**：抽象层仅是一层 Python 调用，实际计算仍由原始 JIT kernel 完成，性能几乎不变。统一实现对 `torch.empty_like` 与 `contiguous` 的调用次数未变，额外的函数调用开销可忽略。  
  - **删除 `cuda_wait_value`**：若项目中已有基于该实现的同步场景（例如自定义 Event），删除后会导致运行时错误，而不是性能提升。若该代码本来未被使用，则对整体运行时无影响。  

- **安全考虑**  
  - 没有涉及密码学、权限检查或网络交互的改动。唯一的安全相关点是 **同步语义**：删除自定义 `Event.wait` 可能会让用户误以为仍可使用，若误用会导致死锁或未同步的错误。  

**⚠️ 潜在风险**  

1. **向后兼容性**  
   - 任何外部代码（用户脚本、插件、旧测试）仍然 `import sglang.jit_kernel.cuda_wait_value` 或使用 `Event` 类将会报 `ModuleNotFoundError`。  
2. **隐式依赖**  
   - 项目内部若有未在 CI 中覆盖的路径调用 `stream_wait_value`（例如在某些实验性模型里），删除后会导致运行时异常。  
3. **测试覆盖**  
   - 原来的自检脚本被删除，若没有其他单元测试覆盖 `cuda_wait_value` 的行为，未来可能出现未被捕获的同步错误。  

**💡 关注建议**  

- **API 调整通知**：在项目的发布说明或 `CHANGELOG.md` 中明确标记 `stream_wait_value` 与自定义 `Event` 已被移除，建议使用标准的 `torch.cuda.Event` 与 `torch.cuda.Stream` 替代。  
- **回归测试**：添加或强化针对 Hadamard 各倍率的单元测试，确保统一实现不引入维度或填充错误。  
- **文档更新**：在 `jit_kernel` 的文档页中删除 `cuda_wait_value` 章节，并在 Hadamard 章节加入统一实现的说明，帮助新贡献者快速定位入口函数。  
- **监控 CI**：在 CI 中加入对未使用的 JIT 模块的检测（如 `grep -R stream_wait_value`），防止未来出现遗漏的引用。  

通过上述措施，可最大化降低重构带来的兼容性风险，保持库的高可维护性与易用性。

### Support multi-step alignment and pipeline integration in dump comparator (#19378)
**SHA**: `265eb56` | 🔗 [查看提交](https://github.com/sgl-project/sglang/commit/265eb56d44b2a50dd201f38f08d6223701f63893)

**🎯 变更类型**：功能增强  
**⚡ 重要程度**：🔴高  
**📋 变更摘要**：在 `sglang` 的 dump comparator 中加入对多步（multi‑step）Token 对齐的完整支持，并在入口流水线中自动检测、生成并使用对应的 `TokenAlignerPlan`。核心改动包括为 `TokenLocator` 引入 `steps` 字段、改写对齐执行器以接受 “step → tensor” 映射、完善序列信息构建与规划逻辑、以及在比较入口层面集成该特性并更新跳过键集合。新增大量单元测试覆盖单步、跨步以及跨框架（SGLang ↔ Megatron）对齐场景。

---

### 🎯 影响范围
- **核心模块**：`sglang/srt/debug_utils/comparator/aligner/*`（token_aligner 相关全套实现）  
- **入口层**：`sglang/srt/debug_utils/comparator/entrypoint.py`（命令行入口、skip‑key 计算、aux 过滤）  
- **比较调度**：`sglang/srt/debug_utils/comparator/bundle_comparator.py`（向下游传递 `TokenAlignerPlan`）  
- **类型定义**：`sglang/srt/debug_utils/comparator/aligner/token_aligner/types.py`（`TokenLocator`、`TokenAlignerPlan` 验证）  
- **测试套件**：新增/修改大量对齐、规划、入口行为的单元测试  

---

### 🔍 技术洞察

| 维度 | 影响描述 |
|------|----------|
| **架构影响** | - 为对齐引入 **step 维度**，在 `TokenLocator` 中新增 `steps: List[int]`，使对齐从“单张张量”转变为“多步张量库”。<br>- `execute_token_aligner` 从接受 `Pair[torch.Tensor]` 改为接受 `Pair[Dict[int, torch.Tensor]]`，实现跨步抽取并 `torch.stack`。<br>- `compute_maybe_token_aligner_plan` 在入口阶段自动检测 aux tensor（如 `input_ids`、`positions`）是否存在，若存在即生成 `TokenAlignerPlan` 并注入后续比较流程。<br>- `bundle_comparator` 现在可以接受可选的 `TokenAlignerPlan`，保持向后兼容（旧的单步路径仍然可用）。 |
| **性能影响** | - 额外的 step‑to‑tensor 查找、列表拼接以及 `torch.stack` 带来 **O(N)** 的轻微 CPU/显存开销（N 为对齐 token 数），但仅在对齐阶段执行，整体比较吞吐基本不受影响。<br>- 通过一次性加载所有 step 的 aux 数据（原 `aux_loader` 已去除单步限制），避免了之前在每步重新读取的潜在重复 IO。 |
| **安全考虑** | - 所有新结构均经过 **pydantic** 校验 (`model_validator`)：长度一致性、steps 与 token_index 的对应性得到强制，防止恶意或损坏的 dump 触发数组越界。<br>- 新增的 `Optional[TokenAlignerPlan]` 参数在公共 API 中保持向后兼容，未传入时保持原有行为，无安全回退风险。 |
| **可维护性** | - 将 step‑related 逻辑集中在 `TokenLocator`、`_extract_and_stack_tokens` 中，职责清晰，后续如果引入更多维度（如 pipeline stage）可在同一位置扩展。<br>- 测试覆盖率大幅提升（新增 400+ 行测试），包括单步、跨步、跨框架、异常路径，降低回归风险。 |
| **兼容性** | - 对单步 dump 仍保持兼容：`steps=[]` 与 `token_index_in_step=[]` 会触发空张量返回，旧有 pipeline 不需要改动。<br>- `skip_keys` 在逻辑分组下加入 `step`，但仅在有对齐计划时才跳过，避免误删。 |

---

### ⚠️ 潜在风险
1. **步号不对齐**：如果两侧的 `step` 集合不匹配（例如一侧缺失某 step），`_extract_and_stack_tokens` 会在 `tensor_of_step[s]` 中抛 KeyError。当前实现假设对齐计划只在共有 step 上生成，若前置检测失误可能导致运行时异常。  
2. **大型序列的显存峰值**：跨步对齐会在一次 `torch.stack` 中聚合所有对齐 token，对极长序列（数千 token）可能导致临时显存激增。  
3. **插件/自定义布局兼容**：新增的 `steps` 字段在自定义 `Plugin` 实现中可能未被正确填充，导致验证错误。需要在自定义插件中同步更新 `core_fields` 与 `aux` 读取逻辑。  
4. **API 变更传播**：`execute_token_aligner` 参数签名变化可能影响到内部调用者或第三方工具，虽已在项目内部统一更新，但外部依赖需要注意升级指南。  

---

### 💡 关注建议
- **回归测试**：在 CI 中加入对 **缺失 step**、**step 重复**、**跨框架不同布局** 的负向测试，确保异常路径被捕获并给出友好错误。  
- **文档更新**：在项目的 **debug-utils/comparator** 使用手册中明确说明 `--grouping logical` 在出现 aux tensors 时会自动触发多步对齐，并列出支撑的 `aux` 名称 (`AUX_NAMES`)。  
- **显存预警**：对极大规模 dump（>10k token）提供 `--max-alignment-tokens` 或分批对齐的可选开关，以防单次 `torch.stack` 导致 OOM。  
- **插件兼容**：在自定义插件的基类或文档里提醒实现者在 `TokenAlignerSeqInfo` 中填充 `steps`，并在 `TokenAlignerGlobalAux` 的 `step_auxs` 中保持连续 step 编号。  
- **版本发布说明**：在下一个 Release Notes 中标记 “新增 multi‑step token alignment”，并注明该功能在 `logical` 分组下默认启用，兼容旧版本的单步比较。

### Support token aligner planning and execution in dump comparator (#19377)
**SHA**: `f9a2f03` | 🔗 [查看提交](https://github.com/sgl-project/sglang/commit/f9a2f0398f8d2fa07f29d4cf17b12e1e6e87e7a9)

**🎯 变更类型**：功能增强（新增 Token 对齐规划与执行）  

**⚡ 重要程度**：🔴高  

**📋 变更摘要**  
- 在调试比较器的对齐流程中新增 **token_aligner** 子模块，提供从 auxiliary tensors 构建序列信息、匹配序列、生成对齐计划以及执行令牌级对齐的完整能力。  
- `execute_aligner_plan` 现统一完成 **unshard → reorder → token‑align**，并在 `compute_aligner_plan` 中加入可选的 `token_aligner_plan` 参数。  
- 新增入口、类型定义、规划器、执行器及对应单元测试，完善了未使用 token‑align 时的向后兼容（`token_aligner_plan=None`）。  

**🎯 影响范围**  
- `sglang/srt/debug_utils/comparator/aligner/*`（核心对齐实现）  
- `bundle_comparator.py`（调用入口）  
- 相关单元测试 (`aligner/entrypoint`, `aligner/token_aligner`)  
- 对外接口 `compute_aligner_plan`、`execute_aligner_plan` 的签名与行为  

**🔍 技术洞察**  

| 维度 | 影响描述 |
|------|----------|
| **架构影响** | 1. 引入 **TokenAligner** 子系统，使对齐流程从 “unshard+reorder” 扩展为 “unshard+reorder+token‑align”。<br>2. 通过 `Pair` 与 `TokenAlignerPlan` 在现有 `AlignerPlan` 中进行可选组合，保持向后兼容，同时提升了模块化程度。<br>3. `entrypoint`、`planner`、`executor` 三层划分遵循原有的 **pipeline** 设计，易于后续扩展（如支持更多布局）。 |
| **性能影响** | - **规划阶段**：遍历所有 auxiliary step tensors，累加序列信息 O(总 token 数)。匹配过程使用哈希表 + 前缀搜索，最坏 O(|X|·|Y|) 但实际规模受序列数量与长度限制，已在单元测试覆盖。<br>- **执行阶段**：仅基于 `plan.locators` 对原始张量进行切片 (`tensor[index]`)，开销接近 O(匹配 token 数)。<br>- **内存**：需要额外保存 `TokenAlignerGlobalAux`、`TokenAlignerSeqsInfo`，在大规模 dump 场景下会略增峰值内存。 |
| **安全考虑** | - 所有操作均在本地完成，无网络交互，安全风险极低。<br>- 读取 auxiliary tensors 可能包含异常形状或恶意数据，代码在 `aux_loader` 已做基本校验；仍需防止 **异常触发的断言**（`assert x_ids == y_ids`）导致进程被杀。<br>- 对外暴露的 `execute_token_aligner` 不执行任何写操作，属于只读安全路径。 |

**⚠️ 潜在风险**  
1. **断言导致异常**：在 `compute_token_aligner_plan` 中对匹配 token 的 `input_ids` 使用 `assert` 检查相等，若两 side 的 token 序列在某 step 只部分匹配或出现数据错误，会抛出 AssertionError，可能在生产调试脚本中中断。  
2. **内存峰值**：大型 dump（数十 GB）时，加载全部 auxiliary tensors 与生成完整的 `SeqInfo` 可能导致 OOM。  
3. **签名兼容**：`compute_aligner_plan` 现强制传入 `token_aligner_plan` 参数；若未来有其他内部调用未同步更新，可能出现 `TypeError`。  
4. **前缀匹配的歧义**：当多个 x 序列都是 y 的前缀时，仅返回最长的一个，可能导致部分可匹配序列被遗漏（业务层若依赖完整匹配集合需注意）。  

**💡 关注建议**  
- **容错处理**：将 `assert x_ids == y_ids` 替换为明确的错误抛出并记录警告，避免异常中断整个比较流程。  
- **内存优化**：考虑在 `build_seqs_info` 中使用惰性加载或分块处理，或者提供 `max_token_limit` 参数供用户自行限制。  
- **接口文档**：在 `compute_aligner_plan` 与 `execute_aligner_plan` 的 docstring 中明确说明 `token_aligner_plan` 为可选且何时需要提供（仅在 `grouping == "logical"` 且 auxiliary 存在时）。  
- **回归测试**：在 CI 中加入大规模随机 auxiliary 数据的性能基准，监控内存峰值与规划时长。  
- **代码审计**：确保所有对 `TokenAlignerPlan` 的访问均做 `None` 检查，防止未来新增调用忘记传参导致 `AttributeError`。  

总体来看，此次改动为 `sglang` 调试比较工具引入了 **token‑级对齐** 的关键能力，提升了跨模型、跨框架对比的准确性和可用性。只要在异常处理与资源控制上做好防护，风险可控，价值显著。

  - **几乎无运行时开销**：大部分改动是抽象层次的重构，底层算子（`torch.cat`、`tensor.chunk`、`torch.allclose`）未变更。  
  - **轻微额外开销**：`warning_sink` 使用上下文对象收集警告，涉及列表的 append 操作；但仅在复制检查（`_verify_replicated_group`）阶段触发，整体对大模型 dump 对比的耗时影响可忽略（< 1%）。  
  - **缓存/对象构造成本**：`Pair` 与 `ReordererPlan`/`UnsharderPlan` 均为轻量的 pydantic `_FrozenBase`，实例化成本极低，不会显著影响大规模 tensor 对比。  

- **安全考虑**  
  - **输入校验**：`AxisInfo` 的 validator 防止非法的 `axis_size`（≤0）或 `axis_rank` 超界导致数组越界或错误的并行映射。  
  - **警告泄露**：通过统一的 `warning_sink`，警告信息不再随函数返回，避免误用者忽视返回值导致信息丢失。  
  - **类型安全**：`compute_smaller_dtype` 现在强制使用 `Pair[torch.dtype]`，避免因参数顺序错误产生不易发现的 dtype 错误。  

**⚠️ 潜在风险**  
1. **向后兼容性**：API 已全面改名，外部直接 import 原路径（如 `sglang.srt.debug_utils.comparator.aligner.unshard.executor`）的代码将失效。需要更新所有依赖方（包括第三方插件）。  
2. **warning_sink 线程安全**：当前实现基于普通列表，若在多线程/多进程环境下并发使用 `process_tensor_group`，可能出现警告收集交叉。  
3. **pydantic 验证异常**：在极端情况下（如手动构造 `AxisInfo` 的 dict 并直接通过 `normalize_parallel_info` 传入），异常会在运行时抛出，可能导致 pipeline 在未捕获的情况下中断。  
4. **测试覆盖**：虽然已迁移大量测试，但尚未看到对 `warning_sink` 的并发或异常路径（如 `compare_tensor_pair` 中的异常）进行专门测试，潜在的错误路径可能未完全覆盖。  

**💡 关注建议**

- **兼容层**：考虑在旧路径下添加轻量的兼容 shim（例如 `unalshard/__init__.py` 中导入新实现），降低突发升级的破坏性。  
- **线程安全**：如有并行 dump 对比需求，建议在 `warning_sink` 中使用 `threading.local()` 或显式的 `with warning_sink.context()` 在每个线程独立实例化。

整体来看，此次重构显著提升了调试工具的可维护性、可扩展性和安全性，除非直接依赖旧 API，否则对现有功能无负面影响。建议在后续版本中保持对新命名的严格使用，并逐步淘汰旧入口。
