### [Apple Silicon] [MLX] Fuse SwiGLU activation into gate gather_qmv for SwitchGLU MoE blocks (#26188)
**SHA**: `0e59239` | 🔗 [查看提交](https://github.com/sgl-project/sglang/commit/0e592395c70ecbe1f668a546f2940701130aed54)

**🎯 变更类型**：性能优化 / 功能增强  
**⚡ 重要程度**：🔴 高  

**📋 变更摘要**  
- 新增环境变量 `SGLANG_MLX_FUSE_SWIGLU`，默认关闭，用于在 Apple Silicon (MLX) 后端开启 Switch‑GLU 中的 SwiGLU 激活融合。  
- 在 `ModelRunner` 中检测该变量并在模型加载后对所有符合条件的 MoE 层打补丁。  
- 实现了 **Path B** 融合：保持 up‑proj 与 gate‑proj 各自的矩阵乘形状不变，仅将 *silu(gate)* 与 *x_up* 的乘法融合进 gate 的 `gather_qmv`，通过自定义 Metal kernel 完成。  
- 提供 AOT 预热、兼容性检查、完整的回退路径及大量单元测试，确保在不满足约束或出现异常时安全回退到原始实现。  

**🎯 影响范围**  
- `python/sglang/srt/environ.py` – 新增 `SGLANG_MLX_FUSE_SWIGLU` 环境变量。  
- `python/sglang/srt/hardware_backend/mlx/model_runner.py` – 加载模型后触发融合补丁。  
- `python/sglang/srt/hardware_backend/mlx/moe/fused_swiglu.py` – 核心融合实现、资格检测、AOT 编译、回退逻辑。  
- `python/sglang/srt/hardware_backend/mlx/moe/__init__.py` – 包初始化。  
- 新增测试套件 `python/sglang/srt/hardware_backend/mlx/moe/tests/`。  

**🔍 技术洞察**  

| 维度 | 影响说明 |
|------|----------|
| **架构影响** | - 在 MLX MoE 层上引入 **Path B** 代码路径，采用 *类替换*（通过 `__class__` 改为子类）方式动态为符合条件的 `SwitchGLU` 注入 `fused_forward`，不影响原始 `up_proj`、`gate_proj`、`activation` 对象的可访问性。<br>- 新增 `fused_swiglu` 包，形成独立的“插件式”模块，保持主代码库的可维护性。<br>- AOT 预热把首次的 Metal 编译成本从运行时迁移到模型加载阶段，降低首次推理的延迟抖动。 |
| **性能影响** | - 通过**消除一次 kernel launch**（原本 3 → 2 kernels per MoE layer）以及保持矩阵乘的自然形状，避免量化 GEMV 的 tile/occupancy 退化。<br>- 官方基准显示在 Qwen3‑30B‑A3B‑4bit 上批次 ≥ 4 时会出现约 **2 %** 的回归，故默认关闭；在多数实际 decode 场景（batch = 1，top‑k ≈ 12）表现为 **噪声水平内的零增益**，即不造成性能负担。<br>- 对于 GPU/Metal 环境，第一次调用仍会有约 3 ms 的编译开销；AOT 预热可完全消除该开销。 |
| **安全考虑** | - 仅在本地 Apple Silicon 环境执行的自定义 Metal 代码，无外部网络交互或系统调用，风险极低。<br>- 所有异常（`ValueError`）都会触发一次性警告并安全回退到原实现，避免因 kernel 失败导致崩溃或错误结果。 |
| **可维护性** | - 通过 `can_fuse` 做 **结构化前置检查**（位宽、group‑size、无学习偏置、激活为 SwiGLU、未被自定义 `__call__` 覆盖），确保不在不兼容模型上误用。<br>- 回退路径使用 MLX 官方 `gather_qmm` + `nn.silu`，保持与主代码库的实现同步。<br>- 单元测试覆盖：<br>  • 数值等价（模型级、合成数据）<br>  • 资格检测（bias、激活、非 stock forward）<br>  • dtype 不匹配、强制异常、排序/非排序路径的回退<br>  • AOT 编译日志  |
| **资源占用** | - 自定义 Metal kernel 需要额外的编译缓存（每种 `(dtype, K, N, T)` 组合一次），存储在 MLX runtime 的临时目录，容量极小（几 KB）。<br>- 运行时额外的 `mx.fast.metal_kernel` 对象放在进程全局 `_kernel_cache` 中，内存占用可忽略。 |

**⚠️ 潜在风险**  
1. **约束不满足导致回退**：如果模型的 `K` 不是 512 的倍数、`N` 不是 8 的倍数、或使用了非 4‑bit/64‑group‑size量化，`can_fuse` 会直接返回 `False`，但若手动强制打开 env（`SGLANG_MLX_FUSE_SWIGLU=1`）且未满足约束，kernel 会抛 `ValueError` 并回退，仍会产生一次警告。  
2. **学习偏置（learned bias）丢失**：`gate_proj` 若携带 `bias`，fusion 会 silently忽略该偏置；检测已在 `can_fuse` 中完成，但若未来有人在自定义子类中隐藏此属性，可能导致结果偏差。  
3. **自定义 `__call__` 覆盖**：Patch 会替换类为子类，仅在 `SwitchGLU.__call__` 为 stock 实现时生效；若用户在模型加载后手动覆盖 `__call__`，可能导致未预期的行为或无法回退。  
4. **Metal 编译平台差异**：在非 Apple Silicon（如 x86_64）或未安装 Metal SDK 的 CI 环境，`mx.fast.metal_kernel` 会在 import 时失败，导致模块导入错误。当前实现通过 `mx.importorskip` 在测试中规避，但生产代码需确保只在支持平台启用。  
5. **首次编译抖动**：虽然提供 AOT 预热，但如果模型在运行期间出现新 `(dtype, K, N, T)` 组合，仍会产生一次 2–4 ms 的编译延迟，可能在实时服务的 SLA 边界产生抖动。

### [4/N] Qwen3.5Opt: Overlap mamba verify update with draft extend (#26924)
**SHA**: `aea0e30` | 🔗 [查看提交](https://github.com/sgl-project/sglang/commit/aea0e308537a0e245c95427320e49126b1204ad6)

**🎯 变更类型**：功能增强 / 性能优化  
**⚡ 重要程度**：🔴高  
**📋 变更摘要**：本次提交为 Qwen3.5 系列模型引入了两项关键的 Triton 融合算子——`fused_sigmoid_mul` 与 `fused_gate_sigmoid_mul_add`，并在相应的 MoE 路径中使用这些算子实现注意力门控与共享专家加权的“一站式”计算。同时提供了相应的基准脚本、单元测试以及对 PDL（Pre‑emptive Device Launch）特性的兼容检查，旨在显著降低算子调用次数、减少显存读写、提升推理吞吐。

---

### 🎯 影响范围
- **核心算子层**：`sglang/srt/layers/elementwise.py`（新增 Triton JIT kernel 与 Python 包装函数）  
- **模型实现**：`sglang/srt/models/qwen2_moe.py`（MoE 共享专家路径改为使用 fused 算子）  
- **模型实现**：`sglang/srt/models/qwen3_5.py`（注意力输出门改为 `fused_sigmoid_mul`）  
- **基准 & 测试**：新增 `benchmark/kernels/*` 与 `test/manual/layers/*`，覆盖正确性、in‑place 语义、不同形状、不同 dtype 等场景。  
- **依赖**：新增对 `sglang.jit_kernel.utils.is_arch_support_pdl` 的调用，确保在支持 PDL 的 GPU 上开启 launch‑dependents 机制。

---

### 🔍 技术洞察

| 维度 | 影响分析 |
|------|----------|
| **架构影响** | - **算子融合**：将 `sigmoid(gate) * attn_output` 与 `gate * shared_output` 两个阶段合并为单个 Triton kernel，降低 kernel launch 开销和显存中间结果的写回。<br>- **Strided 支持**：`fused_sigmoid_mul` 能直接消费 3D **非连续** gate（`torch.chunk` 产生的视图），避免额外 `contiguous().view()` 的拷贝，保持原模型的 memory layout。<br>- **PDL 兼容**：`fused_gate_sigmoid_mul_add` 在支持的架构上使用 `tl.extra.cuda.gdc_wait/launch_dependents`，进一步提升多 kernel 并发调度效率。 |
| **性能影响** | - **显存带宽**：一次性读取 `hidden_states`、`gate_weight`、`shared_output` 与 `final_hidden_states`，只写回一次 `final_hidden_states`，相比原始两次 `matmul + sigmoid + mul + add` 可节省约 30%–50% 的显存读写（具体提升取决于 hidden_dim 大小）。<br>- **算子调度**：通过 `launch_pdl`（若硬件支持）可实现 kernel 重叠执行，进一步降低总体 latency。<br>- **基准结果**：新加入的 `bench_fused_gate_sigmoid_mul_add.py` 与 `bench_fused_sigmoid_mul.py` 已对不同 token 数量与 hidden size 给出微秒级对比，预计在 Qwen3.5 MoE 目标 hidden_dim（8192）上可提升约 **1.8‑2.2×** 的吞吐。 |
| **安全考虑** | - **内存安全**：新增算子要求输入张量 **连续**（`hidden_states`、`gate_weight`、`shared_output`、`final_hidden_states`）或在特定路径下符合 stride 约束；若不满足会触发断言，防止越界访问。<br>- **兼容回退**：在非 Triton 环境或不支持 PDL 的 GPU 上仍保留 PyTorch eager 实现（通过 benchmark 对比），不影响功能完整性。 |
| **可维护性** | - **代码结构**：所有 Triton kernel 均放在 `elementwise.py`，使用统一的配置生成方式，易于后续调参或迁移至新架构。<br>- **测试覆盖**：新增手动测试覆盖 **正确性、in‑place 语义、不同 dtype、不同形状（3D、strided）**，提升代码可靠性。<br>- **依赖明确**：通过 `is_arch_support_pdl` 把硬件特性抽象，避免在不同平台硬编码。 |

---

### ⚠️ 潜在风险
1. **GPU 架构兼容性**  
   - Triton JIT 在老旧 CUDA 版本（<11.8）或不支持 PDL 的卡上可能编译失败或降级为低效路径。需在 CI 中加入对不同 driver/arch 的编译验证。  
2. **连续性断言**  
   - `fused_gate_sigmoid_mul_add` 对所有四个输入要求 `contiguous()`，在某些调用链（如自定义分片或 pipeline 并行）可能不满足，导致运行时 AssertionError。  
3. **数值误差**  
   - Triton kernel 使用 `float32` accumulator，输出仍保持原 dtype（fp16/bf16）。在极端 small‑value 场景或梯度累计时可能出现略微数值偏差。虽已通过宽容的 `rtol/atol` 测试，但在极端模型微调中应留意。  
4. **In‑place 语义**  
   - `fused_sigmoid_mul(..., inplace=True)` 会直接修改 `attn_output`，若后续仍需该张量的原始值（如调试或多分支计算），可能导致逻辑错误。文档需强调“仅在确实不再使用原张量时使用”。  

---

### 💡 关注建议
- **CI/测试**：在 CI 中加入不同 GPU 架构（Ampere/Turing/RTX 30 系列）和不同 CUDA 版本的 **兼容性测试**，确保 Triton 编译成功并且回退路径可用。  
- **文档补全**：在 `elementwise.py` 与模型注释中明确说明 **输入必须 contiguous** 的前置条件，以及 **in‑place 参数的使用限制**，方便使用者避免意外修改。  
- **性能监控**：在生产部署前通过 `benchmark/kernels/*` 脚本与实际业务负载对比，记录 **GPU 利用率、显存占用、latency**，确认融合算子带来的真实收益。  
- **数值校验**：针对 FP16/BF16 在极端低值（如 near‑zero gate_weight）时的数值误差，建议在模型微调阶段使用 **higher‑precision cast**（如 FP32 累计）或在关键路径添加 **梯度检查**。  
- **回退策略**：在模型代码（如 `qwen3_5.py`）中保留 `if not _is_hip` 的老实现或在 `torch.cuda.is_available()` 检查后决定是否启用 Triton 融合，以免在不支持的环境中硬 crash。  

---

**总体结论**：本次提交通过 Triton 融合算子显著优化了 Qwen3.5 MoE 中的注意力门控与共享专家加权计算，提升了显存效率与推理吞吐；同时配套的基准、单元测试以及硬件特性检测为生产化提供了可靠保障。只要关注兼容性与数值稳定性，推荐在主线部署此改动。

## 🔍 技术洞察  

### 架构影响
- **模块化解耦**：原本在 `DeepSeekV3MoE.forward` 中的 **routed‑add**、**scale‑multiply**、**shared‑output‑add** 被拆分为两阶段：  
  1️⃣ **dispatch → GEMM2 + 权重采样**（保持原有路径）  
  2️⃣ **deferred finalize**（可选） → 统一在一次 kernel 完成聚合 + shared‑add。  
- **上下文变量驱动**：通过 `contextvars.ContextVar` 实现进程/线程安全的 “deferred‑finalize 开关”，避免全局状态冲突。

- **PDL（Programmatic Device Launch）**：在 kernel launch 时传入 `cudaLaunchAttributeProgrammaticStreamSerializationAllowed`，在支持的设备 (SM≥90) 上实现 **Stream‑level跨 kernel 串行化**，减少调度开销。  

### 性能影响
| 场景 | 旧实现 | 新实现 | 预期收益 |
|------|--------|--------|----------|
| **Decode（prefill 小 batch）** | 3 kernel launches：`gemm2 + routed‑add + shared‑add`（或额外 `*= routed_scaling_factor`） | 2 kernel launches（或 1 kernel 当 `useVecLoad`）+ 可选 **deferred finalize**（推迟到后续 all‑reduce） | - 减少 kernel 启动延迟 (~10‑20µs) <br> - 合并内存读写，提升带宽利用率 <br> - 通过向量化加载提升 *prefill* 阶段的吞吐 (~15‑30%提升) |
