#### 🔴 高重要度变更 (1)

### [Spec V1] Split draft-extend phase from `EagleDraftInput` into new `EagleDraftExtendInput` (#24859)
**SHA**: `d087442` | 🔗 [查看提交](https://github.com/sgl-project/sglang/commit/d08744238a64209cfe5ac64db039eaa44466aa85)

**🎯 变更类型**：功能增强、架构变更、代码重构  

**⚡ 重要程度**：🔴高  

**📋 变更摘要**  
本次提交将原有的 `EagleDraftInput` 中同时承担 **draft** 与 **draft‑extend** 两个阶段的职责，拆分出全新的 `EagleDraftExtendInput`。新结构只保留 `draft‑extend` 所需的字段（如 `hidden_states、num_accepted_*、input_ids、seq_lens` 等），并在 `EagleVerifyOutput` 中改为返回 `draft_extend_input`。随后在 `EagleWorker`、`FrozenKVMTPWorker`、多层 Eagle 实现以及 CUDA‑graph runner 等关键路径中同步完成了调用方的适配。整体目标是：

1. **消除 `isinstance` / “FIXME” 检查**，提升代码可读性与可维护性。  
2. **明确阶段边界**，防止在错误阶段误用字段（如在 draft‑extend 时访问 `topk_p`）。  
3. 为后续 V1 中对 `draft‑extend` 的独立优化（包括 kernel、缓存布局）提供更干净的接口。

---

### 🎯 影响范围
- **核心模型执行层**：`forward_batch_info.py`（padding 逻辑）  
- **Speculative 相关实现**：`eagle_info.py、eagle_worker.py、eagle_draft_extend_cuda_graph_runner.py、multi_layer_eagle_*`  
- **Frozen‑KV‑MTP 实现**：`frozen_kv_mtp_info.py、frozen_kv_mtp_utils.py、frozen_kv_mtp_worker.py`  
- **SpecInfo 类型枚举**：`spec_info.py`（新增 `EAGLE_DRAFT_EXTEND`、`FROZEN_KV_MTP_DRAFT_EXTEND`）  
- **所有使用 `EagleVerifyOutput` 的调用方**（包括服务器调度、Tracing、日志统计等）。  

---

## 🔍 技术洞察

### 架构影响
| 维度 | 正面影响 | 说明 |
|------|----------|------|
| **职责单一** | `EagleDraftInput` 只用于 **draft**（生成候选草稿），`EagleDraftExtendInput` 只用于 **draft‑extend**（接受验证后继续生成）。 | 消除了跨阶段字段混用的隐式约束，后续可以对每个阶段独立演进（如不同的缓存策略、不同的 CUDA‑graph）。 |
| **类型安全** | 引入 `SpecInputType.EAGLE_DRAFT_EXTEND` 与 `FROZEN_KV_MTP_DRAFT_EXTEND`，`SpecInput.is_draft_input()` 自动识别两类。 | 编译期/运行时更易捕获错误，减少 `isinstance` 检查。 |
| **代码组织** | `EagleVerifyOutput` 从 `next_draft_input` 改为 `draft_extend_input`，明确“验证 → 延伸”数据流。 | 调度层 (`EagleWorker.forward_batch_generation`) 的逻辑更加线性：`draft → verify → (optional) draft‑extend → next draft`. |
| **兼容性** | 通过 `create_idle_input` 在两种 `SpecInput` 上保持相同的空态接口，旧的 “idle” 路径仍可使用。 | 老版本的批处理仍能在空 batch 场景运行，只是内部类型切换。 |
| **外部影响** | 任何直接访问 `batch.spec_info` 并假设其拥有 `topk_*`、`bonus_tokens` 等字段的第三方插件或自定义后端需要同步更新。 | 文档/API 说明需同步。 |

### 性能影响
| 维度 | 潜在提升 | 潜在下降 |
|------|----------|----------|
| **内存热点** | `draft‑extend` 阶段不再保留 `topk_*`、`bonus_tokens` 等不再使用的张量，降低了每个 batch 的峰值显存占用（尤其在大 batch、长序列情况下）。 | 若旧代码仍在某些路径误用 `EagleDraftExtendInput`（如 `merge_batch`），可能导致额外的 `torch.cat` 开销。 |
| **CUDA‑graph 重用** | `EagleDraftExtendInput` 为 `*_cuda_graph_runner` 提供更精简的结构，图捕获时无需考虑 `topk_*`，可提升捕获/复用效率。 | 初次捕获阶段仍需要一次 “结构变更” 检查，带来极小的启动开销。 |
| **分支预测** | 通过 `getattr(..., None)` 替代 `isinstance`，避免 Python 类型判断的成本（微乎其微），但代码更清晰。 | 无显著负面影响。 |
| **调度路径** | `forward_draft_extend_after_decode` 现在返回 `EagleDraftInput`，而不是在函数内部直接修改 `batch.spec_info`，有助于 JIT 编译器（如 TorchScript）更好地进行函数内联。 | 若调用方忘记把返回值写回 `batch.spec_info`，会导致错误的状态流转。 |

### 安全考虑
- **数据泄露**：拆分后没有出现跨阶段的意外共享，反而降低了 **意外泄露**（例如 `topk_p` 在 `draft‑extend` 时不应被外部读取）。  
- **异常路径**：`prepare_extend_after_decode` 中大量 `assert` 与 `strict_check`（环境变量）仍保留，确保在批次过滤不一致时抛出明确错误。  
- **并发安全**：所有新字段均通过显式的 `torch.cat` / slice 生成，没有引入全局可变状态。  
- **回滚兼容**：如果用户强制降级到旧版本（未包含 `EagleDraftExtendInput`），需要保持 `EagleDraftInput` 仍能兼容 `spec_info.is_draft_input()` 为 `True`，不影响已有部署。

## ⚠️ 潜在风险
1. **遗漏更新**  
   - 任何自定义的 `SpecInput` 子类或外部插件（如自研的 Speculative 后端）仍可能通过硬编码访问 `topk_*`、`bonus_tokens` 等字段。若未同步到 `EagleDraftExtendInput`，会在运行时触发 `AttributeError`。  
2. **序列化/Checkpoint**  
   - 检查模型 checkpoint / state dict 的序列化代码（如果有）是否已经考虑新类的 `__dict__`。未兼容可能导致 `torch.save` / `torch.load` 出错。  
3. **旧的 `spec_info.is_draft_input()` 语义**  
   - 通过 `SpecInputType` 直接判断为 `draft`，但部分代码仍使用 `spec_info.is_draft_input()` 来区分 “draft vs verify”。若其他阶段（如 `DFlash`）引入类似拆分，需要同步更新该判断。  
4. **函数签名变更**  
   - `forward_draft_extend_after_decode` 现在返回 `EagleDraftInput`，但调用方 `EagleWorker`、`FrozenKVMTPWorker` 等已经更新；若未来再新增调用方（如测试脚本）仍使用旧签名，将导致 `TypeError`。  
5. **混合使用 V1 与 V2 代码路径**  
   - V2 重用 `EagleDraftInput` 仍保留 `num_accepted_*` 等字段；在 V2 与 V1 代码混用时，需要确保两者之间的转换（`_to_frozen_kv_mtp_draft_extend_input`）完整覆盖所有字段。  
6. **日志/Tracing**  
   - 旧日志中仍可能出现 “next_draft_input” 字段名称，需同步监控和监测仪表板的字段映射，以免误报。

4. **回退方案**：若后续发现 dsv4 在特定 TP/PP 配置下仍需独立路径，保留 `_send_state_pages_flat` 的实现或提供特性开关，以免大幅回滚。  

总体而言，此次改动旨在简化代码路径、降低维护成本，但在没有充分回归测试的情况下，可能会对使用 DeepSeek‑V4 的推理分布式场景产生细粒度的兼容性问题，请重点关注上述测试与文档更新。

### Optimize MHC pipeline: DeepGemm, fused norm, fused hc_head (#24775)
**SHA**: `2f06867` | 🔗 [查看提交](https://github.com/sgl-project/sglang/commit/2f06867128e8a7f31517dae395cc01f326be1077)

**🎯 变更类型**：功能增强 + 性能优化  
**⚡ 重要程度**：🟡 中  
**📋 变更摘要**：  
1. 在 `mhc_pre` 流程中引入 **DeepGemm** 加速以及 **TileLang**‑fused RMSNorm，新增 `gemm_last_dim`、`norm_weight/eps` 参数，自动根据 token 数量和隐藏维度计算 split‑K 并行度。  
2. 新增 Triton 实现的 **fused_hc_head**（RMSNorm + Linear + Sigmoid‑gate + 加权求和），专为 DeepSeek‑V4 的 `hc_head` 设计。  
3. `DeepseekV4Model` 调用链改动：当环境变量 `SGLANG_OPT_DEEPGEMM_HC_PRENORM`、`SGLANG_OPT_USE_TILELANG_MHC_PRE` 打开时，层内归一化可被融合进 kernel；`hc_head` 直接走 fused Triton 路径。  
4. CI 配置加入 DSV4 专用测试标签。

**🎯 影响范围**  
- `sglang/srt/layers/mhc.py`（核心 MHC 前置计算）  
- 新增 `sglang/srt/layers/mhc_head.py`（triton kernel）  
- `sglang/srt/models/deepseek_v4.py`（模型前向路径）  
- 环境变量定义 `sglang/srt/environ.py`、CI 脚本

**💡 关注建议**  

1. **兼容性**：TileLang 路径默认要求 `torch.cuda.get_device_properties(0)` 可获取显卡信息，且 `is_arch_support_pdl()` 为真时才启用 PDL，建议在不支持的 GPU（如旧显卡）回退到原实现并添加单元测试。  
2. **内存 & 连续性**：`fused_hc_head` 强制 `contiguous`，模型在 `torch.compile`、`DDP` 等场景下可能产生非连续张量，请在调用前确认 `contiguous()`，或在 fallback 中自动处理。  
3. **环境变量**：新增 `SGLANG_OPT_DEEPGEMM_HC_PRENORM`、`SGLANG_OPT_USE_TILELANG_MHC_PRE`、`SGLANG_OPT_DEEPGEMM_HC_PRENORM` 等开关，需要在文档和 CI 中明确默认值与推荐配置。  
4. **数值准确性**：融合 RMSNorm 时使用 `rsqrt` 与 `norm_eps`，以及 `hc_pre_eps`、`hc_sinkhorn_eps` 的顺序与原实现保持一致；建议加入对比测试（FP16/BF16 精度）以防引入偏差。  
5. **性能基准**：新增 `gemm_last_dim` 动态决定 GEMM 输出维度，确保在 `num_tokens` 较大时分割数不超过 SM 数量；可通过 `torch.cuda.get_device_properties` 采集 `multi_processor_count` 并在实验中记录加速比。

### Support Gemma3/4 + Eagle3 (#23976)
**SHA**: `12f42f2` | 🔗 [查看提交](https://github.com/sgl-project/sglang/commit/12f42f2e7e7520bef573230cadbdf6c10241aa61)

**🎯 变更类型**：功能增强（新增 Gemma 3/4 与 Eagle 3 的捕获与共享机制）  
**⚡ 重要程度**：🟡 中  
**📋 变更摘要**：在 Gemma 3、Gemma 4（因果与多模态）模型中加入 `layers_to_capture`、`capture_aux_hidden_states`、`set_eagle3_layers_to_capture` 等属性，实现对指定层的中间 hidden‑state 捕获；为 TP 场景提供 `get_embed`、`get_embed_and_head` 以及 weight‑sharding 辅助函数；在 Eagle3 入口 `llama_eagle3` 中新增可选的 aux‑norm 处理，以统一不同层的尺度。  

**🎯 影响范围**：  
- `python/sglang/srt/models/gemma3_causal.py`、`gemma3_mm.py`  
- `python/sglang/srt/models/gemma4_causal.py`、`gemma4_mm.py`  
- `python/sglang/srt/models/llama_eagle3.py`  
- `sglang/srt/distributed`（新增 `get_tensor_model_parallel_rank`）  

**💡 关注建议**  

1. **API 兼容**：`forward` 现在可能返回 `(hidden, aux_hidden_states)`，上层调用必须检查 `capture_aux_hidden_states` 标志；若已有自定义 logits‑processor，需兼容新增 `aux_hidden_states` 参数。  
2. **默认行为**：确保默认 `capture_aux_hidden_states=False`，避免无意的额外显存占用和计算开销。  
3. **TP 正确性**：`_shard_weight` 使用 `get_tensor_model_parallel_rank/size`，建议在多卡环境下跑一次完整的 load‑weights + inference 测试，核对分片后 embedding 与 lm‑head 是否对齐。  
4. **层索引约定**：捕获层的索引在内部加 1（捕获的是前一层输出），文档需明确说明，否则使用者传入错误的层号会导致空返回或重复捕获。  
5. **aux‑norm 开关**：`llama_eagle3` 新增 `use_aux_norm`，建议在配置里默认关闭并在 README 中给出开启场景（如不同层尺度差异显著时）。  
6. **性能评估**：捕获多个 hidden‑state 会显著增加显存，建议在 `set_eagle3_layers_to_capture` 中提供可选的 `max_layers` 限制，并在 CI 中加入显存基准测试。  
7. **单元/集成测试**：添加针对 `set_eagle3_layers_to_capture`、`get_embed_and_head`、以及多卡 sharding 的测试用例，确保在单卡、2/4 卡 TP 环境下均能通过。  

总体来看，此次改动为 Eagle 3 方案提供了必要的中间特征共享能力，但涉及多处返回值和分片逻辑，务必通过完整的功能与性能回归验证，并在文档中补全使用说明。
