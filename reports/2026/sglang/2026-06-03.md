### feat(api): add require_reasoning field for engine's generate api (#27019)
**SHA**: `6ba31e3` | 🔗 [查看提交](https://github.com/sgl-project/sglang/commit/6ba31e33e690a13982057f7b35cd505ad03d4d0b)

**🎯 变更类型**：功能增强  
**⚡ 重要程度**：🔴高  
**📋 变更摘要**：在 `engine.py` 中为同步 `generate` 与异步 `async_generate` 接口新增布尔参数 `require_reasoning`（默认 `False`），并向下传递至底层实现。此参数旨在让调用方能够显式指示模型在生成时是否必须给出推理过程，提升 API 可控性与可调度性。  

**🎯 影响范围**：  
- `python/sglang/srt/entrypoints/engine.py`（同步/异步生成入口）  
- 可能波及到内部调用 `generate` 的业务层代码、单元测试以及自定义客户端 SDK。  

**🔍 技术洞察**  
- **架构影响**：  
  - 仅在入口层新增参数并向下转发，保持了现有层级的职责划分。  
  - 若底层实现（如实际的推理引擎）未立即使用该字段，需在后续提交中完成对应逻辑，否则该字段会成为“悬空”参数。  
  - 对外 API 向后兼容（默认 `False`），对向前兼容（需要在文档和 SDK 中同步更新）影响较小。  

- **性能影响**：  
  - 参数本身是轻量布尔值，传递开销可忽略。  
  - 真正的性能变化取决于底层实现对 `require_reasoning` 的处理（如强制生成思路、增加额外的解释 token），若实现后导致生成步骤额外计算，则可能出现一定的 latency 增加。  

- **安全考虑**：  
  - 该字段不涉及敏感数据或权限校验，属于功能开关，对系统安全性无直接影响。  
  - 唯一需要注意的是，若生成的“推理过程”被外部存储或展示，可能泄露模型内部的推理策略或业务机密，建议在使用方做好信息披露控制。  

**⚠️ 潜在风险**  
1. **未被下游使用**：如果底层实现仍未消费 `require_reasoning`，调用方会误以为已经启用了推理约束，导致功能期待落空。  
2. **文档/SDK 同步遗漏**：公开 API 变更后，若文档、类型提示文件（`.pyi`）、以及自动生成的 SDK 未同步更新，可能导致用户调用时报错或出现参数错误。  
3. **兼容性冲突**：第三方库或已有脚本在调用 `generate` 时使用关键字参数顺序或 **`**kwargs** 方式，新增参数可能触发 “unexpected keyword argument” 错误（尽管默认值存在，但在使用 `*args` 且未提供 `require_reasoning` 时可能导致位置参数错位）。  
4. **推理资源占用**：若底层实现将 `require_reasoning=True` 转化为生成额外解释文本，可能导致显存/算力消耗上升，进而影响并发请求的吞吐。  

**💡 关注建议**  
- **实现同步**：检查并确保底层推理引擎（如 `engine_core`）已经对 `require_reasoning` 做出相应处理，或者在本次提交的注释中明确该字段为占位符，计划在后续迭代实现。  
- **文档更新**：立即在项目官方文档、API 参考以及 `README` 中加入 `require_reasoning` 参数说明、取值含义以及默认行为。  
- **SDK 与类型提示**：若项目提供 Python 客户端或生成的 `.pyi` 文件，务必同步更新，以免 IDE 报错。  
- **回归测试**：在 CI 中加入包含 `require_reasoning=True/False` 的正负测试，用以验证参数传递链路完整性及潜在性能回退。  
- **监控与告警**：若后续实现导致推理时长显著增长，建议在服务监控中加入 “reasoning‑mode latency” 指标，提前发现资源瓶颈。  
- **兼容性提示**：在发布说明中提醒用户如果使用 `*args` 调用 `generate`，请检查参数顺序或改为关键字参数，以避免因新增参数导致的调用错误。

3. **线程安全**：`self.prep_handles` 在多线程 `transfer_worker` 中读取，写入仅在 `__init__`/`_add_remote_peer` 中完成，已安全；若未来支持动态添加 peer，需加锁或使用 `ConcurrentDict`。  
4. **索引展开**：`expand_page_indices_for_slice`、`repeat_indices_over_layers` 采用 `np.ravel` 与 `astype(np.int32)`，在极大 slot / layer 数时会产生临时 O(N) 内存，建议在极端模型（数千层）上进行压力测试，确认不会触发 OOM。  
5. **错误处理**：`make_prepped_xfer` 返回 `None` 时抛异常；若底层 NIXL 因资源不足返回错误，当前实现会中止整个 transfer 线程。考虑在异常捕获层加入重试或回退到非预构建路径，以提升鲁棒性。  
6. **单元测试**：新增的预构建路径应覆盖等价‑TP、MLA、以及 TP‑heterogeneous 三类场景，验证 `dst_num_slots` 大于 `src_num_slots` 时的正确索引映射，并对 `transfer_worker` 中的 `notif` 字符串格式保持不变。  

总体来看，此次改动在保持功能一致性的前提下，大幅削减 KV‑slice 传输的 CPU 计算与 NIXL descriptor 创建开销，预计在多‑GPU、TP‑heterogeneous 环境中可带来 30%‑50% 的时延下降。后续关注资源释放、异常回退以及极端规模的内存占用即可。
