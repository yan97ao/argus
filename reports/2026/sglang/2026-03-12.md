### [Test] Refactor KL divergence and prefix cache branching to kits (#19715)
**SHA**: `067353f` | 🔗 [查看提交](https://github.com/sgl-project/sglang/commit/067353f67b56ea6143d9cd77b87b32edde74b6ac)

**🎯 变更类型**：重构 / 架构变更  
**⚡ 重要程度**：🔴高  

**📋 变更摘要**  
本次提交将 KL 散度、前缀缓存分支等测试逻辑抽象为可复用的 *kit*（Mixin），并在 `test/server_fixtures/default_fixture.py` 中提供统一的 `DefaultServerBase` 与 `flush_cache` 辅助方法。所有受影响的测试文件均改为继承这些 Mixins 与 Base Fixture，显著减少重复代码，提升可维护性与可扩展性。

**🎯 影响范围**  
- `python/sglang/test/kits/*`（新增 `kl_divergence_kit.py`、`prefix_cache_branching_kit.py`）  
- `python/sglang/test/server_fixtures/default_fixture.py`（新增 `flush_cache`）  
- 多个 `test/registered/*` 模块（4‑GPU model、MTP、FP4、PCG、radix cache 等）  
- 相关 CI 注册与测试执行脚本（间接受影响）

**🔍 技术洞察**  

- **架构影响**  
  - 引入 **Mixin 结构**：测试用例通过多重继承组合功能（GSM8K、KL 散度、前缀缓存分支），实现“功能即插即用”。  
  - **统一 Fixture**：`DefaultServerBase` 负责启动/关闭服务器、提供 `flush_cache`，统一了测试生命周期管理，避免各测试自行实现相同逻辑。  
  - **模块解耦**：原本散落在每个测试文件中的请求构造、阈值定义、缓存刷新等被抽取到独立文件，提升代码组织度。

- **性能影响**  
  - **测试执行时间**：Mixin 本身不增加运行时开销；唯一新增的网络请求是 `flush_cache`（POST `/flush_cache`），次数与原来的 `requests.get(.../flush_cache)` 相当，整体时间变化可以忽略。  
  - **资源占用**：因统一了服务器启动/关闭逻辑，可能略微降低因重复 `popen_launch_server` 带来的资源波动，间接提升 CI 稳定性。

- **安全考虑**  
  - `flush_cache` 通过 HTTP POST 暴露了清理服务器内部缓存的能力。虽然此接口仅在测试环境内部使用，但如果误将测试服务器暴露到公网或 CI 环境共享网络，可能被恶意调用导致缓存失效、服务性能波动。建议：  
    - 仅在本地/受控 CI 环境启用该端点。  
    - 在生产/开放环境禁用或做身份校验。

**⚠️ 潜在风险**  

1. **属性缺失导致运行时错误**  
   - Mixins 依赖类属性（`model`、`base_url`、`cache_chunk_size`、阈值等），若子类未正确定义，会在测试阶段抛出 `AttributeError`。  
2. **服务器端未实现 `/flush_cache`**  
   - `DefaultServerBase.flush_cache` 假设服务器提供此接口；若底层实现变更或在特定模型/配置下不支持，会导致请求失败、测试中断。  
3. **Mixin 冲突**  
   - 多重继承时同名方法（如 `setUpClass`、`tearDownClass`）的解析顺序需保持一致，误用顺序可能导致资源未正确释放。  
4. **CI 环境的网络波动**  
   - 新增的 HTTP POST 可能受网络不稳定影响，出现偶发超时，需要在 CI 中设置足够的重试或超时容忍。

**💡 关注建议**  

- **文档补全**：在项目的测试指南中说明各 Kit 的使用方式、必填属性及默认阈值，避免新贡献者忘记提供必要字段。  
- **安全加固**：在服务器实现 `/flush_cache` 前加入简单的 token 校验或仅在 `DEBUG` 模式下开放，以防误用。  
- **回退兼容**：保持现有的 `requests.get(.../flush_cache)` 兼容路径，或在 `DefaultServerBase.flush_cache` 中捕获 404 并安全忽略，防止因底层实现变化导致全套测试失败。  
- **CI 稳定性**：为 `flush_cache` 请求添加重试机制（如 `requests` 的 `Session` + `Retry`），并在测试超时阈值上适当放宽，以提升 CI 的鲁棒性。  
- **属性校验**：在 Mixins 中加入 `@classmethod` 的 `validate_config` 方法，在 `setUpClass` 早期检查必需属性是否存在并给出明确错误信息，帮助定位配置遗漏。  

通过上述改动，项目的测试代码结构更加模块化、可维护，同时也需注意上述风险并做好相应防护，以确保长期的测试可靠性。

**📋 变更摘要**：在 `benchmark/bench_linear_attention/bench_gdn_prefill.py` 中加入了一个完整的基准与 correctness 检测脚本，比较 sglang 自研的 Triton 实现与 FlashInfer 的 GDN prefill 实现。脚本包括输入构造、两套实现的调用包装、输出/stride 正确性检查以及基于 Triton `do_bench_cudagraph` 的性能统计，支持多种批次/序列/头数配置并提供 Qwen‑3‑Next 预置。

**🎯 影响范围**  
- **sglang.srt.layers.attention.fla.chunk**（Triton 实现）  
- **flashinfer.gdn_prefill**（外部依赖）  
- 代码路径：`benchmark/bench_linear_attention/` 以及 `python/sglang` 包的 `l2norm_fwd`。  

**💡 关注建议**  
1. **兼容性**：FlashInfer 仅在 `K==V` 且 `head_size∈{128}` 时可用，脚本已提前 skip，但生产代码若误用会导致 CUDA 上下文崩溃，建议在库层面做显式检查并给出友好错误。  
2. **内存与性能**：benchmark 会在 GPU 上复制 pool、进行多次 `contiguous` 与 `transpose`，对显存占用较大（尤其大 batch‑size、长序列时）。建议在文档中注明最低显存需求或提供 `--dry-run` 参数。  
3. **数值容差**：正确性比较使用 `atol=5e‑2、rtol=1e‑2`，对 bf16 可能偏宽，若后续精度提升可适当收紧阈值。  
4. **异常处理**：FlashInfer 调用外层已捕获异常并同步 CUDA，确保后续测试不受污染；若在 CI 中使用，建议在 `requirements.txt` 中明确 `flashinfer>=x.x`。  
5. **可维护性**：文件长度 639 行，逻辑拆分较多，可考虑把输入生成、运行包装、统计函数分别放入内部模块，以免单文件难以阅读。  

总体来看，此提交为 GDN prefill 提供了首个系统化的性能与正确性基准，对后续优化和对比非常有价值，只需注意上述兼容性与资源使用说明即可安全上线。

**⚡ 重要程度**：🟡 中

**🎯 影响范围**  
- **核心模块**：`sglang/srt/layers/attention/aiter_backend.py`（注意力后端）  
- **测试目录**：`test/registered/amd/accuracy/mi35x/`（两套 GSM8K 准确率评估）  
- **PR‑level 测试**：`test/registered/amd/test_kimi_k25_mxfp4.py`（端到端推理速度与准确率）

3. **性能验证**：回退到 `self.k_scale` 可能导致 KV‑cache 与查询尺度不一致，建议在 CI 中加入对 FP8 KV‑cache 计算正确性的单元验证，确保数值误差在可接受范围。

5. **文档更新**：说明在 AMD MLA 后端使用 FP8 KV‑cache 时 `layer.k_scale` 可能为 `None`，并指明 `SGLANG_AITER_MLA_PERSIST=1` 为必需环境变量。  

总体来说，此次修复消除了在特定 FP8 场景下的致命断言，配套的回归与性能测试能够在 AMD MI35x 节点快速验证模型可用性，风险可控。建议在后续合并前跑一遍全部 AMD CI，确认所有新增测试均通过后再发布。
