### [MoE] Deprecate act_and_mul_triton; fold filter_expert into JIT silu/gelu_and_mul (#23707)
**SHA**: `c7878db` | 🔗 [查看提交](https://github.com/sgl-project/sglang/commit/c7878dbb6ddfc9c6721b9db20a876f2718b0e955)

**🎯 变更类型**：功能增强 / 重构（将 `act_and_mul_triton` 废弃并把 `filter_expert` 功能合入 JIT 激活 kernel）  
**⚡ 重要程度**：🟡 中  
**📋 变更摘要**：  
- 为激活算子（silu、gelu、gelu_tanh）新增可选的专家过滤参数 `expert_ids` / `expert_step`，在 CUDA kernel 中实现“‑1‑skip” 行的直接跳过。  
- 在 Python 层包装 `run_activation`、`silu_and_mul`、`gelu_and_mul` 等函数，统一调用新 kernel，并在 `fused_moe` 中改为使用 JIT 版而不是已废弃的 `act_and_mul_triton`。  
- 新增 benchmark 与单元测试，覆盖过滤路径及全过滤/全不过滤的边界情况。  
- 对 C++/CUDA 实现做了模板化改造：`act_and_mul_kernel` 现在接受 `kFilterExpert` 编译标志并在运行时根据 `expert_ids` 是否为空选择对应 kernel。  

**🎯 影响范围**：  
- `python/sglang/jit_kernel/activation.py`、`benchmark/bench_activation.py`、`tests/test_activation.py`  
- `python/sglang/srt/layers/moe/moe_runner/triton_utils/fused_moe.py`（MoE 下游算子）  
- `csrc/elementwise/activation.cuh` 与对应的 `ActivationKernel` 实现  
- 任何直接调用 `act_and_mul_triton` 的外部代码（已标记为废弃）  

**💡 关注建议**  
1. **兼容性**：`act_and_mul_triton` 已被删除，若项目或第三方插件仍引用该符号会出现 ImportError，请尽快迁移到 `silu_and_mul` / `gelu_and_mul` 并传入 `expert_ids`。  
2. **HIP/XPU 路径**：当前实现仅在 CUDA 上启用过滤；HIP/XPU 会回退到无过滤的路径并在 kernel 里自行将被过滤的行置零，确认这种行为符合业务需求。  
3. **参数校验**：`expert_ids` 必须是 `int32`、`expert_step ≥ 1`，且形状应为 `ceil(num_tokens / expert_step)`。运行时检查已加入，但在高频调用场景下仍建议在上层做一次性校验以避免不必要的 `RuntimeCheck`。  
4. **性能监控**：过滤逻辑会导致分支失效和潜在的 warp divergence，建议通过 `benchmark_filter` 对不同 `skip_ratio`、`expert_step`、`dim`、`batch` 组合评估实际加速或损失，并在生产配置中选取最优 `BLOCK_SIZE_M`。  
5. **测试覆盖**：新加入的 `test_activation_filter_expert*` 已覆盖全 skip、全不 skip、随机 skip 三种情况，确保在未来改动时不会意外破坏过滤路径。若添加自定义激活类型，请同步更新 C++ `select_kernel` 分支。  

总体来看，此次合并把 MoE‑expert 过滤从独立 Triton 实现迁移到统一 JIT kernel，提升了代码可维护性并统一了 API；只要注意上面几点即可平稳过渡。

### [diffusion] refactor: make timestep scheduler request-local (#23716)
**SHA**: `d49a037` | 🔗 [查看提交](https://github.com/sgl-project/sglang/commit/d49a0377de42ace38ec6c082a0bc3a2010265fae)

**🎯 变更类型**：功能增强（将 Diffusion 调度器改为请求‑Local）  
**⚡ 重要程度**：🟡 中  
**📋 变更摘要**：  
- 新增 `diffusion_scheduler_utils`，提供 `clone_scheduler_runtime` 与 `get_or_create_request_scheduler` 两个工具函数。  
- `Req` 增加可选字段 `scheduler`，用于保存每个请求独立的调度器实例。  
- 在调度器相关的所有 stage（`causal_denoising、denoising、denoising_av、denoising_dmd、hunyuan3d_*、ltx_2_denoising、helios、mova、qwen_image_layered、timestep_preparation` 等）大量改写，统一使用 `batch.scheduler`（或通过 `get_or_create_request_scheduler`）而不再直接访问 pipeline‑level调度器。  
- 通过 `_init_disagg_request_scheduler` 在 disaggregation（分片）路径上为每个请求克隆调度器并初始化 timesteps。  
- 为保证多次循环的状态干净，新增 `_reset_scheduler_loop_state`，在每轮 denoising 前显式重置内部计数器等。  

**🎯 影响范围**：  
- `python/sglang/multimodal_gen/runtime/pipelines_core/*`（调度器工具、`Req` 定义、`timestep_preparation` stage）  
- 所有具体模型的 **denoising** 相关 stage（`causal_denoising、denoising、denoising_av、denoising_dmd、hunyuan3d_*、ltx_2_denoising、helios、mova、glm_image、qwen_image_layered`）  
- **Disaggregation / 分布式执行**（`scheduler_mixin.py`）  

**💡 关注建议**  

1. **性能与内存**  
   - `clone_scheduler_runtime` 采用 `deepcopy`，在高并发场景下可能导致显著的拷贝开销。建议在文档或配置中提供 `isolate=False`（即复用同一调度器）选项，或实现基于 `torch.clone` 的轻量拷贝（如果调度器内部仅持有张量）。  
   - 对于常规单请求路径，`Req.scheduler` 仍可能指向全局调度器，保持现有 warm‑up 缓存，这一点在 `get_or_create_request_scheduler` 已有说明，但实际代码中多数分支仍强制 `clone`，需核实是否有不必要的复制。

2. **线程安全**  
   - `Scheduler._init_request_scheduler_from_template` 在主线程完成 `set_timesteps`，但 `scheduler` 本身仍可能在后续 GPU 计算线程中被并发读写。确保所有调度器成员在复制后均已转移至目标 `device`，并且不共享任何 `torch.nn.Parameter` 或 `torch.Tensor` 引用。可以在复制后做一次 `assert not any(t.is_shared())` 检查。

3. **状态重置**  
   - `_reset_scheduler_loop_state` 只处理了常见的内部属性，若未来引入新调度器实现（如 `solver_order`、`model_outputs`），需要同步更新此函数。建议将此函数改为 “调度器协议”——调用调度器的 `reset_state()`（如存在），否则回退到属性清理。

4. **兼容性测试**  
   - 受影响的 stage 超过 20 处，务必在 CI 中加入 **跨模型**（LTX‑2、Helios、MOVA、Hunyuan3D、GLM‑Image、Qwen‑Image）以及 **disaggregation** 场景的回归测试。  
   - 特别关注 `batch.scheduler` 为 `None` 的老旧请求路径，确保在 `timestep_preparation` 前会自动创建调度器，不会出现 `AttributeError`.  

5. **文档与使用提示**  
   - 在 `README` / `docs` 中明确说明：**请求本地调度器**仅在并发或分片执行时必需；普通单请求仍可使用共享调度器以获得更快启动。  
   - 暴露环境变量或配置项（如 `sglang.diffusion.isolate_scheduler=True/False`）让用户可自行控制复制策略。  

总体而言，此次改动为分布式/并发推理奠定了必要的状态隔离基础，但需要在性能、状态清理以及测试覆盖上再做细化，以防在高负载或新模型引入时出现意外的内存/并发问题。

### Apply should_use_dp_reduce_scatterv guard to remaining MoE models (follow-up to #23731) (#23732)
**SHA**: `ba4e9d2` | 🔗 [查看提交](https://github.com/sgl-project/sglang/commit/ba4e9d2ac20bcbe800aa95e2664ec18efda0a25f)

**🎯 变更类型**：功能增强（为所有 MoE 模型加入 `should_use_dp_reduce_scatterv` 判定）  
**⚡ 重要程度**：🟡 中  

**📋 变更摘要**  
本次 PR 在 SGLang 中的多处 MoE（Mixture‑of‑Experts）实现里，统一加入 `should_use_dp_reduce_scatterv()` 条件，以在特定 DP‑reduce‑scatter‑v 路径下跳过原有的 `tensor_model_parallel_all_reduce`（或 `moe_expert_parallel_all_reduce`）操作。该 guard 已在 `sglang/srt/layers/moe/__init__.py` 中实现，用于在全局配置要求时禁用显式的 all‑reduce，避免重复通信或冲突。

**🎯 影响范围**  
- **模型层**：`bailing_moe`, `bailing_moe_linear`, `deepseek_v2`, `exaone_moe`, `glm4_moe`, `hunyuan_v3`, `llada2`, `llama4`, `mimo_v2_flash`, `minimax_m2`, `sarvam_moe`, `sdar_moe`, `step3p5` 等全部 MoE 相关模型。  
- **公共层**：`sglang/srt/layers/moe/__init__.py`（新增 `should_use_dp_reduce_scatterv` 导入）和相关后端选路函数。  

**💡 关注建议**  
1. **功能验证**：在多卡 DP 环境下跑一次完整的推理/训练，确认在 `should_use_dp_reduce_scatterv` 为 `True` 时，通信路径已切换至 DP‑reduce‑scatter‑v，实现不再进行显式的 all‑reduce。  
2. **回退兼容**：若用户依赖原来的 all‑reduce 行为，需要通过配置关闭 `should_use_dp_reduce_scatterv`，确保不影响已有部署。建议在文档中明确该开关的默认值及使用场景。  
3. **性能检测**：对比开启/关闭该 guard 前后的通信开销和整体吞吐，避免在某些硬件/网络拓扑下产生额外同步。  
4. **代码维护**：所有新增的 guard 条件保持统一风格（多行括号写法），便于后续审阅和补丁添加。可以考虑抽取公共的 “tp all‑reduce guard” 函数，减少重复代码。  

总体来看，改动是一次一致性的开关引入，提升了 MoE 系统在 DP‑reduce‑scatter‑v 场景下的可配置性，对功能安全影响有限，只要通过上述验证即可安全合入。
