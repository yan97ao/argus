### [diffusion] feat: progressive resolution growing for Ideogram 4 via GPU DCT upsampling with up to 1.56× speedup (#27736)
**SHA**: `7f57b34` | 🔗 [查看提交](https://github.com/sgl-project/sglang/commit/7f57b344c9e1caa65505e72f6f2029591caf5ad7)

**🎯 变更类型**：功能增强 / 架构变更 / 性能优化  
**⚡ 重要程度**：🔴高  

**📋 变更摘要**  
- 为 Ideogram‑4（双 Transformer）新增 **渐进式分辨率（Progressive Resolution Growing）** 支持，实现了早期在低分辨率进行去噪、随后光谱 DCT 上采样的流水线。  
- 在文档、Pipeline 声明、通用 progressive‑resolution 基类以及模型专属实现（`Ideogram4ProgressiveDenoisingStage`）中加入了约 **1.56×**（Ideogram‑4） 的推理加速。  
- 同时为 FLUX/QL‑Image 等模型添加了 **模块别名**，提升单元测试可发现性。  

---

### 🎯 影响范围
- **核心模块**：`sglang/multimodal_gen/runtime/pipelines_core/stages/progressive_resolution/*`（通用基类、Flux、Qwen‑Image、Ideogram‑4）  
- **模型专属 Pipeline**：`python/sglang/multimodal_gen/runtime/pipelines/ideogram.py`（创建 `ProgressiveDenoisingStageRouter`）  
- **文档**：`docs_new/docs/sglang-diffusion/progressive_resolution.mdx`（新增 Ideogram‑4 使用指南与基准）  
- **单元测试**：`test/unit/progressive_resolution/test_progressive.py`（覆盖 Ideogram‑4 PACK/UNPACK 与分辨率切换逻辑）  

---

### 🔍 技术洞察

| 维度 | 影响说明 |
|------|----------|
| **架构影响** | 1. 引入 **`ProgressiveDenoisingStageRouter`**，在运行时根据 `--progressive-mode` 动态切换 **标准去噪** 与 **渐进式去噪** 实例，兼容已有 LoRA、Ulysses 等特性。 <br>2. 为 **双 Transformer**（条件 + 无条件）模型实现专属 `Ideogram4ProgressiveDenoisingStage`，通过多重继承（`ProgressiveDenoisingStage` + `Ideogram4DenoisingStage`）实现 **MRO** 调度，确保两套前向逻辑共享同一缓存上下文。 <br>3. 新增 **`_spectrum_latent_dims`** 重写，使光谱阈值计算基于真实空间‑latent 大小（Patch‑packed 模型的实际分辨率），避免因 “grid ≠ spatial” 造成的错误阈值。 |
| **性能影响** | - 通过在 **coarse‑resolution**（如 32²、64²）执行前 N 步去噪，显著降低 DiT 的二次方注意力开销。<br>- Ideogram‑4 采用 **1 层** progressive（`levels=1`），在 **δ=0.05~0.10** 的配置下可达到 **1.42×–1.56×** 加速（20‑step、48‑step 预设均已验证）。<br>- `refresh_context` 只在 stage 转换时触发，缓存‑DiT 重建次数与传统实现保持一致，未引入额外显式计算。 |
| **安全考虑** | - 变更全部在 **推理路径**，未涉及网络、文件 I/O 或权限提升。<br>- 新增的 `refresh_context_on_transformer` 调用已在原有代码中使用，风险主要为 **错误的 step index** 导致 **IndexError** 或 **意外的梯度泄露**（若误用于训练）。<br>- `ideogram4_progressive` 对 **seed 生成**、**noise dtype** 均保持原有安全性。 |
| **可维护性** | - 为每个模型提供 **模块别名**（`_flux_unpack/_flux_pack`、`_ideogram4_*` 等），提升测试发现度，降低因导入路径变化导致的回归。<br>- 通过 **单元测试**（约 270 行）覆盖 pack/unpack、分辨率切换、latent‑scale‑factor 等关键函数，确保未来改动不会破坏核心逻辑。<br>- 文档同步更新，示例命令完整，可直接在 CI 中跑基准。 |

### ⚠️ 潜在风险

1. **Stage‑Transition 错误**  
   - `ProgressiveDenoisingStage._refresh_cache_dit_context` 现在会在 `Ideogram4ProgressiveDenoisingStage` 中两次刷新（conditional + unconditional）。若 **`n_remaining`** 计算错误或 `scm_preset` 不匹配，可能出现 **形状不一致** 或 **缓存失效**，导致推理崩溃。  
2. **光谱阈值计算不匹配**  
   - `self._spectrum_latent_dims` 假设 `patch_size * ae_scale_factor = 16`。若未来模型更改 patch 或 VAE 参数，需同步更新此函数，否则 **stage transition点** 可能偏移，导致 **speedup 下降或画质退化**。  
3. **LLM 特征适配**  
   - `_adapt_llm_features` 采用 **zero‑填充** 方式对图像 token 重新布局，依赖**无条件 transformer** 在图像 token 位置始终为零。若未来模型改动（如使用非零初始化），这段逻辑需重新评估。  
4. **兼容性冲突**  
   - 已有提示 **“Sequence parallelism incompatible”**；在启用 `--ulysses-degree` 或 `--ring-degree` 时仍会抛 `RuntimeError`。若用户在实验脚本里默认打开 SP，可能产生 **意外的启动失败**。  
5. **测试覆盖局限**  
   - 当前单元测试仅在 **CPU** 环境下跑 **pack/unpack** 与 **_on_resolution_change**，未覆盖 **GPU DCT 上采样**、**cache‑dit 刷新**、**完整推理路径**。在大量 GPU 环境中仍可能出现 **数值误差** 或 **显存泄漏**。

### [codex] Centralize more inline Triton kernels (#27429)
**SHA**: `d571e07` | 🔗 [查看提交](https://github.com/sgl-project/sglang/commit/d571e076fa6b36ecdb63878062611b8385f792bb)

**🎯 变更类型**：功能增强 / 性能优化 / 重构  
**⚡ 重要程度**：🔴高  

**📋 变更摘要**  
- 将多个分散在不同文件中的 Triton 内联 kernel 统一抽取到 `sglang/srt/layers/attention/triton_ops/` 目录，实现 **代码复用、结构清晰**。  
- 新增 *track mamba state*、*chunked prefix‑cache kv indices*、*pad / unpad draft‑extend query*、*rms_normalize* 等 Triton kernel，并在相应后端（HybridLinearAttn、TRT‑LLM‑MLA、DeepSeek‑V4）中调用。  
- 移除旧的内联实现，改为统一入口函数，降低维护成本并提升 **运行时性能**（并行拷贝、批量填充/拆填）  

**🎯 影响范围**  
- `sglang/srt/layers/attention/hybrid_linear_attn_backend.py`  
- `sglang/srt/layers/attention/mamba/mamba_state_scatter_triton.py`  
- `sglang/srt/layers/attention/triton_ops/kv_indices.py`  
- `sglang/srt/layers/attention/triton_ops/pad.py`  
- `sglang/srt/layers/attention/trtllm_mla_backend.py`  
- `sglang/srt/model_executor/forward_batch_deepseek_mha_mixin.py`  
- `sglang/srt/models/deepseek_v4.py` 及新增 `sglang/srt/models/triton_ops/deepseek_v4.py`  

**🔍 技术洞察**  

- **架构影响**  
  - **模块化**：新增 `triton_ops` 包统一管理 Triton kernel，后端实现只需导入统一函数，降低耦合。  
  - **依赖统一**：所有 Triton 相关实现集中在同一层，便于后续升级 Triton 版本或迁移到其他后端（如 CUDA‑C++）。  
  - **API 稳定**：对外保持原有后端类接口不变，仅在内部实现换成统一函数，保证向后兼容。

- **性能影响**  
  - **并行拷贝/填充**：`track_mamba_state_if_needed_kernel`、`pad_draft_extend_query_kernel`、`unpad_draft_extend_output_kernel` 使用 3‑D grid + 块式并行，显著减少 Python 循环和 `torch.index_copy` 的开销。  
  - **块大小自适应**：`BLOCK_SIZE` 通过 `triton.cdiv` 动态计算，兼容不同维度大小，避免线程空闲。  
  - **内存访问局部性**：复制时使用 stride 参数，确保连续读写，提升缓存命中率。  
  - **CUDA Graph 兼容**：`unpad_draft_extend_output` 在图模式下复用预分配缓冲区，避免图内分配导致的性能波动。  
  - **整体加速预估**：在 DeepSeek‑V4、Mamba Attention、TRT‑LLM‑MLA 场景下，预计比原始 Python 实现提升 **30%–70%**（取决于 batch/seq 长度）。

- **安全考虑**  
  - Triton kernel 本身不涉及外部 I/O，安全风险极低。  
  - 需要注意 **索引越界** 与 **mask** 的正确性，防止潜在的未定义行为或内存泄露。  
  - 迁移后删除的 `import triton` 语句已在多数文件中移除，确保不会因未使用而产生 lint/安全警告。

**⚠️ 潜在风险**

### [HiCache] Add opt-in LRU eviction to file storage backend (CP-aware) (#26670)

**变更概述**  
本次提交为 HiCache 文件存储后端新增了可选的 LRU‑Eviction 机制，并在多‑CP（Attention‑CP）环境下为每个 CP 排添加唯一的键后缀，防止跨 rank 写冲突。通过环境变量 `SGLANG_HICACHE_FILE_BACKEND_MAX_SIZE`、`SGLANG_HICACHE_FILE_BACKEND_MIN_FREE_SPACE`、`SGLANG_HICACHE_FILE_BACKEND_EVICTION_RATIO` 可以打开/配置该功能，默认关闭，保持向后兼容。

1. `environ.py` 暴露三项新 env。

4. CP‑aware 键后缀：当 `attn_cp_size>1` 时在 `config_suffix` 中加入 `_cp{rank}_{size}`，确保不同 CP 排的写入文件互相独立。

- 环境变量默认行为、禁用/启用路径  
- 上限驱逐、eviction‑ratio、最小 free‑space 水位  
- MLA 所有权控制、CP 后缀唯一性

   - `modelName` 只区分 `nvfp4` 与 `fp8` 两种前缀，未覆盖可能的 `bf16`（AMD）或未来新增量化。若后续再加入新模型，需要同步此处，否则会产生 **404**。建议抽象为映射表 `const modelMap = {fp8: 'zai-org/GLM-5.1-FP8', bf16: 'zai-org/GLM-5.1', nvfp4: 'nvidia/GLM-5.1-NVFP4'}`。  
4. **`--trust-remote-code` 重复**  
   - 当 `effectiveQuant === 'nvfp4'` 已显式添加，随后 `if (isAMD)` 分支也会在 AMD 场景下再次追加。虽然对 NVIDIA 场景无害，但代码可读性下降，建议统一在统一的 `extraFlags` 数组中收集后统一拼接。  
5. **表格数据同步**  
   - 文档中已将 B200 → B300、FP8 列改为 NVFP4 列，但 `modelConfigs` 中仍保留 `fp8` 选项在 B300（`fp8: {tp:8, mem:0.9}`），这与表格的 “NVFP4 为推荐路径” 表述不完全对应，可能让用户困惑。若 FP8 在 B300 已不再支持，需在 `modelConfigs` 中删除相应条目或在 UI 中禁用。  

**✅ 建议**
