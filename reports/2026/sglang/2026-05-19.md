### [Diffusion][NPU]Add attention backends for diffusion models for Ascend NPU (#23482)
**SHA**: `45a85ef` | 🔗 [查看提交](https://github.com/sgl-project/sglang/commit/45a85efc3aee65c57a374bcd274a6d1243b461c0)

**🎯 变更类型**：功能增强 / 新增后端（针对 Ascend NPU 的三种稀疏注意力实现）  
**⚡ 重要程度**：🔴高  

**📋 变更摘要**  
1. 为 Diffusion 模型在华为 Ascend NPU 上引入 **Laser Attention、Block Sparse Attention、Rain Fusion Attention** 三个新注意力后端，实现长序列（>2048）下的稀疏计算。  
2. 完善文档、兼容性矩阵以及模型表格，新增配置参数 (`skip_first_steps`、`sparsity`) 并在 `denoising` 流程中根据后端类型构造对应的 `AttentionMetadata`。  
3. 在平台抽象层 `AttentionBackendEnum` 中注册新枚举，NPU 平台代码 `npu.py` 完成动态导入与错误提示。  

**🎯 影响范围**  
- 核心模块：`sglang.multimodal_gen.runtime.layers.attention.backends.*`（Laser、BlockSparse、RainFusion 实现）  
- 配置体系：`sglang.multimodal_gen.configs.models.adapter.base`, `sglang.multimodal_gen.configs.models.dits.base`（加入新后端枚举）  
- 运行时管线：`denoising.py`（稀疏元数据构建）  
- 文档与兼容矩阵：`docs/diffusion/*`, `docs_new/*`（新增表列 & 参数说明）  
- 平台抽象层：`interface.py`, `npu.py`（枚举、后端加载）

### 🔍 技术洞察  

| 维度 | 影响说明 |
|------|----------|
| **架构影响** | - **模块化扩展**：通过 `AttentionBackend` 抽象实现了插件式后端，新增后端仅需实现 `get_impl_cls`、`MetadataBuilder` 等接口，保持了原有架构的低耦合。<br>- **平台感知**：`AttentionBackendEnum` 与 `npu.get_attn_backend_cls_str` 的映射，使得后端选择完全在配置层即可，且在未安装依赖时给出明确报错。<br>- **元数据统一**：`BlockSparseAttentionMetadata`、`RainFusionAttentionMetadata` 与已有的 `AttentionMetadata` 结构保持一致，仅在字段上扩展，便于后续统一调度。 |
| **性能影响** | - **Laser Attention**：在序列长度 < 2048 时回退到标准 SDPA，避免因小序列的额外 padding 开销。序列 ≥ 2048 时利用自研 `torch.ops.attentions.la` 实现，预期可将显存占用降低 ~30%（稀疏比例可调）并提升吞吐 1.5‑2×（取决于稀疏度）。<br>- **Block Sparse Attention**：基于块稀疏估计 `sparse_block_estimate` + `block_sparse_attention`，在 `skip_first_steps` 前使用 Laser 保证生成质量；后期通过 `row_sparse` 参数控制块稀疏度，适用于视频帧数多的长序列。<br>- **Rain Fusion Attention**：在块稀疏的基础上引入 **AvgPool + Top‑K** 采样，再通过 `rainfusionattention` 完成稀疏乘法，兼具局部细节保留与全局感受野。理论上相比纯块稀疏可多收获 ~5‑10% 的质量提升。<br>- **总体**：对 CPU / GPU 仍保持兼容（默认回退到 SDPA），对 NPU 只在安装 `sgl_kernel_npu` 后生效。对已有模型的兼容性不受影响。 |
| **安全考虑** | - 新增的三套后端均依赖外部二进制算子 `torch.ops.attentions.*`，如果 `sgl_kernel_npu` 包含安全漏洞（如内存越界），会直接影响运行时安全。建议在发布前通过官方安全审计并提供最低版本约束。<br>- 参数校验已在 `MetadataBuilder.build` 中加入 `skip_first_steps >=0` 与 `0 ≤ sparsity < 1` 检查，避免非法负数或 NaN 引发异常。<br>- `LaserAttentionBackend._pad` 使用 `torch.nn.functional.pad`，不涉及自定义 CUDA/NPU kernel，安全风险低。 |
| **可维护性** | - 后端实现代码保持 **单一职责**：Laser 只实现原始稀疏注意力，BlockSparse 与 RainFusion 复用 Laser 实例进行前期计算，减少重复实现。<br>- 代码风格与现有后端保持一致，使用 dataclasses、typing，易于阅读与单元测试。<br>- 文档同步更新，兼容性矩阵明确标注 NPU‑only，降低使用者误操作的概率。 |

### ⚠️ 潜在风险  

1. **依赖链风险**：`attentions` 包是由 `sgl_kernel_npu` 提供的专属算子，未安装或版本不匹配会导致 ImportError（已在代码中捕获并给出明确提示），但在生产环境中需要提前验证安装脚本的可靠性。  
2. **稀疏度调参**：`sparsity` 设为较高值（接近 1）会导致大量 token 被丢弃，潜在生成质量回退；默认 `0.2` 仍需在不同模型/分辨率下进行经验验证。  
3. **首次几步的稀疏切换**：`skip_first_steps` 默认 10，若实际扩散步数远大于此，早期稀疏可能对纹理细节产生不利影响。用户若自行修改需谨慎。  
4. **块大小硬编码**：`BSA_BLOCK_SIZE = 128` 与 `self.stride = 8`（BlockSparse）固定，若 NPU 未来推出更优块大小，需要改动多处常量。  
5. **显存泄漏风险**：`_rearrange_with_remaining` / `_inv_rearrange_with_remaining` 在处理非 8‑倍尺寸时会产生额外临时张量，可能在大分辨率下导致瞬时显存峰值升高。  
6. **兼容性回退**：在 NPU 上，当序列长度 < 2048 时会自动回退到 SDPA，若用户强制开启稀疏后端但不满足条件，可能导致隐藏的性能波动。

### 💡 关注建议  

1. **依赖检查与 CI**  
   - 在 CI 中加入 `sgl_kernel_npu` 安装步骤并执行一次完整的 NPU 单元测试（包括 `laser_attn`, `block_sparse_attn`, `rain_fusion_attn`），确保算子可被成功加载。  
   - 为每个后端提供最小兼容版本的 `requirements.txt`，并在发布说明中标明对应的 NPU 驱动/固件要求。  

2. **参数调优指南**  
   - 编写一份官方实验报告，展示不同 `skip_first_steps`、`sparsity` 在主流模型（如 FastWan2.1、Wan2.2）上的质量‑性能曲线，帮助用户快速选取。  
   - 在 `docs` 中加入 “稀疏度调参最佳实践” 小节，推荐在 `skip_first_steps ≤ 0.1 * total_steps`、`sparsity ≤ 0.4` 时使用。  

3. **显存峰值监控**  
   - 在 `denoising.py` 中加入可选的 `--log-attn-memory` 开关，记录每一步的显存占用，帮助定位因块重排引起的突发峰值。  
   - 对非 8‑倍分辨率提供警告日志，提示可能的显存额外开销。  

4. **错误信息统一**  
   - 将 `ImportError` 的提示信息抽象为一个统一函数（如 `raise_missing_backend_error(backend_name)`) 以免重复代码。

  - **线程安全**：`torchcodec.VideoDecoder` 在多实例并行使用时已被官方标记为线程安全，但项目内部未对异常捕获做额外包装，极端错误（如非法视频流）仍可能导致未捕获的异常导致进程崩溃。

4. **文档更新**：在模型说明或 API 文档中标明 “当首次层号为 1 时会自动偏移 +1”，帮助用户避免误用。  
5. **回归检查**：确认在已有的 Eagle‑3 相关基准测试中，捕获的层次与预期一致，避免因层号不匹配导致生成质量回退。  

通过上述改进，可确保 Eagle‑3 辅助层的捕获行为更加直观且不易出错，同时提升代码的健壮性和可维护性。
