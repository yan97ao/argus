### Support swa HiCache for unified radix cache (#23391)
**SHA**: `eb5f0fb` | 🔗 [查看提交](https://github.com/sgl-project/sglang/commit/eb5f0fbeef5b90aa522abf0103f6e0eaf00c29d9)

**🎯 变更类型**：功能增强 / 重构  
**⚡ 重要程度**：🔴 高  

**📋 变更摘要**  
本次提交为 HiCache（统一径向缓存）新增对 **Sliding‑Window Attention（SWA）** 的支持。核心改动包括：  
1. 为 `PoolName` 增加 `SWA`，并在 `HybridCacheController` 中引入可选的 **全量分配器** 与 **回滚机制**。  
2. 扩展 `PoolEntry`、`HostPoolGroup` 与 `HybridPoolAssembler`，加入对 **设备端 alloc/free** 的覆盖（针对 SWA 的内部 `swa_attn_allocator`）。  
3. 实现 `SWAKVPool` 与 `SWATokenToKVPoolAllocator` 的层级映射、层间同步计数器以及 **host‑side tombstone** 的等待逻辑。  
4. 在 `UnifiedRadixCache` 与各组件（`TreeComponent`, `SWAComponent`, `MambaComponent` 等）中加入 **HiCache 迁移、备份、恢复** 的全链路实现，包括锁引用参数的解耦 (`IncLockRefResult → DecLockRefParams`) 。  
5. 大量单元测试新增，覆盖 SWA‑HiCache 的备份连续性、加载后缀、host 与 device LRU 交互、匹配结果归一化等场景。  

**🎯 影响范围**  
- `sglang/srt/mem_cache/*`（核心缓存实现）  
- `sglang/srt/mem_cache/hybrid_cache/*`（HybridCache 控制器、池组装器）  
- `sglang/srt/mem_cache/swa_memory_pool.py`（SWA 专属 KV 池）  
- `sglang/srt/mem_cache/unified_cache_components/*`（Tree / SWA / Mamba 等组件）  
- `sglang/srt/mem_cache/unified_radix_cache.py`（统一径向缓存调度）  
- 相关单元测试文件 `test_unified_radix_cache_unittest.py`

| **可维护性** | - 通过 **统一的 Transfer Phase**（`CacheTransferPhase`）抽象，增强了组件之间的解耦，后续加入新的组件（如新型压缩层）将更易集成。<br>- 代码量显著增加，尤其是 `SWAComponent` 中的恢复、匹配、驱逐分支，建议补充完整的文档与注释，防止未来维护时遗漏关键状态（`host_value` / `value` / `lock_ref`）。<br>- 单元测试覆盖面广，已捕获多数边界情况，但仍需关注 **混合堆栈（FULL+MAMBA+SWA）** 的组合路径（目前仅支持单一 aux）。 |

### ⚠️ 潜在风险

1. **锁引用不匹配**  
   - `inc_lock_ref` 返回的 `IncLockRefResult` 通过 `to_dec_params` 传递给 `dec_lock_ref`。若在任何异常分支（如分配回滚、提前返回）忘记调用 `dec_lock_ref`，将导致锁计数永久不降为 0，进而阻塞后续写回/加载。  
2. **回滚逻辑错误**  
   - `_resolve_pool_transfers_allocation` 采用 `newly_allocated` 保存 `(pool, free_fn, indices)`。若 `free_fn` 与实际分配对象不匹配（例如 `device_alloc_fn` 为 `None` 时仍使用默认 `free`），可能出现 **双重释放** 或 **泄漏**。  
3. **Host‑Device 同步错误**  
   - `SWAKVPool._wait_for_layer` 依赖 `layer_transfer_counter`；如果外层 `HybridCacheController` 未正确设置该计数器，SWA 读取可能在对应层的 KV 尚未分配完成前触发错误。  
4. **混合堆栈不兼容**  
   - 当前实现只支持 **FULL + SWA** 或 **FULL + MAMBA** 两种组合。若用户自行构造 **FULL + MAMBA + SWA**，代码会在 `_attach_hybrid_pool_to_unified_cache` 中触发断言，导致启动失败。  
5. **性能回退**  
   - 在极端低显存情况下，大量 **host 备份** 可能导致系统内存被占满，尤其在多请求并发时。需要对 `load_back_threshold`、`prefetch_threshold` 进行合理调参。

   - `Qwen/Qwen3-0.6B` 在首次下载时会占用网络与磁盘，建议在 CI 脚本中使用 `HF_HUB_CACHE` 环境变量指向共享缓存目录，或在 repo 根目录加入 `.github/workflows` 中的 `actions/cache` 步骤。

   - 单元测试里使用 `torch.manual_seed(0)`，但端到端测试直接生成 `torch.full(..., device="cuda")`，这本身是确定性的。确保在未来添加随机权重（如 `torch.randn`）时仍保持固定种子，以免偶发失败。

   - `tearDownClass` 中调用 `kill_process_tree` 已经很好，但如果服务器在启动阶段异常退出，`process.pid` 可能已不存在。可以在 `tearDownClass` 加入 `if cls.process.poll() is None: kill_process_tree(...)` 以防僵尸进程残留。

   - 端到端测试里对错误信息做了硬编码（`"model.layers.5.mlp.gate_up_proj.weight"`），如果后端实现改动导致错误消息结构变化，测试会失效。考虑改为正则匹配或仅检查关键字段（如 `max_abs_err`）来降低耦合。  

6. **依赖升级风险**  
   - `weight_checker` 中使用的 FP8 量化工具 (`block_quant_dequant`, `quant_weight_ue8m0`, `transform_scale_ue8m0`) 依赖 `torch` 与自研量化实现。若将来升级 `torch`（如 2.4 → 2.5）导致这些函数签名或行为变化，单元测试将第一时间捕获异常，建议在 CI 中固定 `torch` 版本或在 `requirements.txt` 中注明兼容范围。  

**总体结论**：  
此次提交通过完整的单元测试与真实服务的端到端验证，大幅提升了 **WeightChecker** 的可靠性和可维护性。唯一需要关注的是 **GPU 资源、模型缓存** 与 **测试对硬编码错误信息的依赖**，在 CI 配置和代码容错上适当加固即可平滑过渡。对现有用户影响极小，主要是提升后续开发与回归测试的信心。
