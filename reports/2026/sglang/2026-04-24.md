### [Experimental] Breakable Piecewise Cuda Graph (#22218)
**SHA**: `60bbb80` | 🔗 [查看提交](https://github.com/sgl-project/sglang/commit/60bbb800db812e1a8a9069abad4b2e391b894578)

**🎯 变更类型**：功能增强 / 实验性特性（Breakable Piecewise CUDA Graph）  
**⚡ 重要程度**：🔴 高  

**📋 变更摘要**  
- 引入 **Breakable CUDA Graph (BCG)** 机制，用一系列可“断点”的 `torch.cuda.CUDAGraph` 段替代原先基于 `torch.compile` 的 Piecewise CUDA Graph。  
- 新增 `breakable_cuda_graph` 包实现捕获、分段、重放逻辑，并在注意力层（RadixAttention、Nemotron‑Mamba）以及 `ModelRunner` 中自动选择调用。  
- 向服务器 CLI 与配置 (`ServerArgs`) 添加 `--enable-breakable-cuda-graph` 开关，默认关闭。  
- 完整的单元测试与集成测试（Qwen3‑8B + mgsm_en）保证功能可用，且在 CI 中跑通。  

**🎯 影响范围**  
- **核心运行时**：`sglang/srt/model_executor/*`（BCG 实现、Runner、上下文管理）  
- **注意力层实现**：`sglang/srt/layers/radix_attention.py`、`sglang/srt/models/nemotron_h.py`（在 BCG 环境下调用 `bcg_*` 版本）  
- **服务器配置**：`sglang/srt/server_args.py`（新增 CLI 参数）  
- **模型初始化**：`sglang/srt/model_executor/model_runner.py`（根据 flag 切换 Runner）  
- **测试套件**：`test/registered/breakable_cuda_graph/*`（单元+集成）

### 🔍 技术洞察  

| 维度 | 影响描述 |
|------|----------|
| **架构影响** | - 通过 `BreakableCUDAGraphCapture` 将原本一次性捕获的 `CUDAGraph` 拆分为 **多段**，每段之间执行 **eager** 函数，实现 **图中断**。<br>- 使用 `ContextVar` 保存当前捕获上下文，实现 **线程局部** 捕获，兼容多线程/并行 TP。<br>- 引入 `weak_ref_tensor` 机制，使段间共享的 `Mempool` 能在任意段结束后安全释放，避免 Python 持有的引用导致内存泄漏。<br>- 通过 `enable_breakable_cuda_graph` 开关保持向后兼容：默认仍走原有 `PiecewiseCudaGraphRunner`。 |
| **性能影响** | - **捕获时间**：分段捕获比一次性捕获略慢（每段需 `capture_begin`/`capture_end`），但 **首次编译成本** 大幅降低，因为每段仅包含少量算子。<br>- **运行时**：段之间的 **eager break** 只涉及少量张量拷贝与函数调用，理论上对吞吐量影响可忽略；相比 `torch.compile` 的大图，能在 **更大 token 长度**（受单段图大小限制）下保持低延迟。<br>- **内存**：共享 `Mempool` 与 `use_count` 机制避免了跨段的额外显式缓存，整体显存占用与原实现相近，甚至在大模型、长序列场景下更节省。 |
| **安全考虑** | - **CUDA 流同步**：在捕获期间自动 `wait_stream` 以 join 侧流，防止 `cudaStreamEndCapture` 报错；但若用户自行在外部创建/销毁流，仍可能产生未捕获的 side‑stream，需在文档中声明限制。<br>- **异常恢复**：捕获期间若函数抛异常，`BreakableCUDAGraphCapture.__exit__` 会仍尝试结束当前段并恢复上下文，风险相对可控。<br>- **平台限制**：代码在非 CUDA/NPU 环境会在 `weak_ref_tensor` import 时硬失败，属于预期的 **功能限制**，不影响安全。 |
| **可维护性** | - 新增约 **400 行**实现与 **50 行**测试，代码结构清晰（Capture、Runner、Context 三层），符合现有项目的模块化风格。<br>- 使用装饰器 `eager_on_graph` 替换原有 `torch.compile` 装饰，使得在 `is_in_breakable_cuda_graph()` 为 `True` 时自动切换。<br>- 依赖 `torch.cuda.CUDAGraph` 原生 API，未来 CUDA 版本升级时保持兼容性。<br>- 通过 `enable_breakable_cuda_graph` 开关，便于逐步回滚或灰度发布。 |

### ⚠️ 潜在风险  

1. **分段边界选择不当**  
   - 目前仅在注意力层插入 break；若后续加入自定义算子或跨层的优化（如 fused‑kernel），可能出现 **跨段张量地址失效** 或 **重复计算**。  
2. **侧流（fork/join）处理错误**  
   - 自动 `wait_stream` 只能捕获在当前捕获上下文中产生的 fork；若外部代码在捕获期间自行创建 CUDA 流并未 join，`capture_end` 仍会报错。  
3. **弱引用张量生命周期**  
   - `weak_ref_tensor` 依赖共享 mempool 的 `use_count`，若用户手动 `torch.cuda.empty_cache()` 可能提前销毁 pool，导致后续段访问已释放内存。  
4. **多请求（batch_size>1）兼容性**  
   - `BreakableCudaGraphRunner.can_run` 直接拒绝多批次输入；在混合使用（prefill + decode）时需要显式回退到 eager 或 PCG，若调用者未检查返回的 `False` 可能导致 **性能回退**。  
5. **调试复杂度**  
   - 由于捕获分段后每段都是独立的 CUDAGraph，异常定位需要在每个 segment 内部逐步排查，现有日志仅打印 break 点名称，建议加入 **segment 编号** 与 **捕获时的 CUDA 错误检查**。

### [diffusion] model: support LTX2.3 high quality pipeline (#23366)
**SHA**: `cd1fa75` | 🔗 [查看提交](https://github.com/sgl-project/sglang/commit/cd1fa7506ac2608b7b2f7c3c71808b85e0ceeeda)

**🎯 变更类型**：功能增强 / 性能优化 / 架构变更（新增 LTX‑2.3 HQ pipeline、改进 LoRA 合并、扩展采样参数、CLI 与文档同步、引入 Res2S 采样器等）  

**⚡ 重要程度**：🔴高  

**📋 变更摘要**  
- 为 Lightricks LTX‑2.3 模型加入全新 **LTX2TwoStageHQPipeline**，默认使用 1920×1088 HQ 分辨率、专用采样参数以及 **Res2S** 采样器，使生成质量与官方基准几乎持平。

- 更新兼容矩阵、文档、测试用例及性能基准，验证在 2‑GPU 及 1‑GPU 环境下的吞吐与延时。

**⚠️ 潜在风险**

1. **分布式噪声一致性**  
   - HQ pipeline 依赖 **固定种子** 与 `renoise` 生成器。如果用户自行修改 `seed` 或使用 `generator` 列表，可能导致两阶段之间的噪声序列不匹配，出现隐蔽的随机性差异。  
   - 建议在文档中明确说明：在 HQ pipeline 中请**不要自行传入自定义 `generator`**，否则结果不保证与官方一致。

2. **LoRA 签名冲突**  
   - `switch_lora_phase` 使用 `(phase, distilled_lora_strength)` 作为签名。如果同一请求在同一次推理中动态修改 `ltx2_distilled_lora_strength_stage_{1,2}`（目前仅通过 `extra` 传入），可能导致 LoRA 权重不在预期阶段切换，出现显存泄漏或错误的 LoRA 应用。

3. **显存占用波动**  
   - `resident` 模式会一次性加载两阶段的全部 transformer，显存需求约 **2×** `snapshot`。在显存边缘机器上若误选 `resident`，可能导致 OOM。  
   - CLI 参数应默认提示显存需求，兼容矩阵中加入显存推荐。

4. **兼容性回退**  
   - 旧的 `legacy` 设备模式已被重命名为 `original`，但仍有外部脚本可能使用旧名称。未捕获的参数会导致 `argparse` 报错。建议在 `ServerArgs` 中加入别名兼容或给出友好错误提示。

5. **Res2S 与非 HQ 模式混用**  
   - `sampler_name` 参数现在可以手动指定，但若在非 HQ pipeline（如普通 `LTX2TwoStagePipeline`）使用 `res2s`，内部噪声生成器不会被初始化，导致 `NoneType` 错误。需要在 `LTX2DenoisingStage.__init__` 检查并回退到默认采样器或抛出清晰异常。

**📋 变更摘要**：`has_fp8_weights_in_checkpoint` 现在既支持本地目录，又支持直接传入 HuggingFace repo ID。对远程仓库只通过 `HfFileSystem` 读取 safetensors 索引或头部（字节范围），不再强制下载完整权重文件。  

**🎯 影响范围**：`python/sglang/srt/utils/common.py` 中的 `has_fp8_weights_in_checkpoint`，间接影响所有在启动时检查模型是否包含 FP8 权重的代码路径（模型加载、自动量化选项等）。  

**💡 关注建议**  

1. **依赖与环境**：新增 `huggingface_hub` 导入，使用者的运行环境必须安装该库且能够访问互联网。建议在 `setup.cfg/requirements.txt` 中声明。  
2. **异常处理**：目前只捕获 `Exception` 并返回 `False`，但网络超时、认证错误会被吞掉。建议记录日志或抛出可辨识的异常，以免误判模型类型。  
3. **缓存与性能**：`HfFileSystem.open` 会进行一次 HTTP Range 请求，频繁调用可能导致网络抖动。若函数会在短时间多次检查同模型，可考虑加层本地缓存（如 `lru_cache`).  
4. **兼容性**：对本地路径保持原有行为；但在本地目录不存在 `model.safetensors.index.json` 时会回退到 `model.safetensors`，这与原实现一致。请确保测试场景覆盖：① 仅有单文件的本地模型；② 多 shard 本地模型；③ HF repo ID（公开/私有）。  
5. **安全性**：远程读取时使用的是只读的 `rb`，但仍要确保不在未受信任的仓库中执行任何代码。  

**后续动作**：  
- 增加单元测试，分别验证本地与 HF repo 两种路径的输出。  
- 在文档中注明函数现在接受 HF repo ID，并说明可能的网络依赖。  

总体来看，此次改动提升了对云端模型的可用性，但需关注依赖、异常可视化及可能的网络开销。
