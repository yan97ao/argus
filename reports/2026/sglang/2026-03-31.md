### Fix remote weight info nnode>1 and dp>1 (#17389)
**SHA**: `20d07c4` | 🔗 [查看提交](https://github.com/sgl-project/sglang/commit/20d07c43842fea39cd94877aaabee6410d937721)

**🎯 变更类型**：Bug修复 / 功能增强  
**⚡ 重要程度**：🔴高  
**📋 变更摘要**：  
1. 修复了在多节点（`nnode>1`）且多数据并行（`dp>1`）场景下，远程权重加载所需的 Transfer Engine 信息未正确同步的问题。  
2. 引入 **EngineInfoBootstrapServer**，通过轻量化 HTTP 服务在 `node_rank==0` 上收集每个 TP Rank 的 Transfer Engine 信息，取代原先依赖 Scheduler 直接返回信息的实现。  
3. 相应地对 `engine.py、http_server.py、scheduler.py、tp_worker.py、model_runner.py、server_args.py` 进行代码重构，删除旧的 `parse_remote_instance_transfer_engine_info_from_scheduler_infos` 逻辑，统一使用新的 Bootstrap 服务。

---

### 🎯 影响范围
- **核心启动入口**：`sglang/srt/entrypoints/engine.py`  
- **HTTP 控制层**：`sglang/srt/entrypoints/http_server.py`  
- **调度器与工作进程**：`sglang/srt/managers/scheduler.py`、`sglang/srt/managers/tp_worker.py`  
- **模型运行器**：`sglang/srt/model_executor/model_runner.py`  
- **服务器参数解析**：`sglang/srt/server_args.py`（新增 `engine_info_bootstrap_port` 与 URL 生成）  
- **网络工具**：`sglang/srt/utils/network.py`（新增 `is_port_available`）  
- **新增服务实现**：`sglang/srt/entrypoints/engine_info_bootstrap_server.py`  
- **测试**：`test/manual/test_cross_node_scheduler_info_sync.py` 与 `test/registered/distributed/test_load_weights_from_remote_instance.py`

### 🔍 技术洞察

| 维度 | 影响分析 |
|------|----------|
| **架构影响** | - 引入 **EngineInfoBootstrapServer** 作为 “bootstrap” 层，完成 **per‑rank Transfer Engine 信息的注册 / 查询**。<br>- 原来 Scheduler 向 Engine 直接注入 `remote_instance_transfer_engine_info` 的路径被拆分为：<br> 1️⃣ ModelRunner 启动后向 Bootstrap Server 注册；<br> 2️⃣ Engine 从 `scheduler_init_result` 中获取已创建的 Server 实例并直接读取内存数据（避免 HTTP 循环）;<br> 3️⃣ HTTP 接口仍保留供外部监控/调试使用。<br>- 通过 `engine_info_bootstrap_port` 参数实现多实例端口隔离，提升在同一节点启动多个实例的可行性。 |
| **性能影响** | - **启动时** 需要一次 HTTP `PUT` 请求，每个 TP Rank 完成一次注册，开销极低（< ms 级）。<br>- **运行时** Engine 通过直接对象引用读取信息，**没有额外网络往返**，性能基本不受影响。<br>- 引入 `uvicorn` + `FastAPI` 常驻线程，内存占用约 10–20 MiB，属于可接受范围。<br>- 移除 Scheduler 端的遍历合并逻辑，略微降低调度器的 CPU 负载。 |
| **安全考虑** | - 新增的 HTTP 接口（`/register_transfer_engine_info`、`/get_transfer_engine_info`）默认 **绑定在 `host` 参数上**（由 `ServerArgs.host` 决定），若用户自行将其暴露到公网，未进行身份校验可能导致信息泄露或恶意篡改。<br>- 代码未实现认证或 IP 白名单，仅依赖 **本地/内部网络** 使用情境。建议文档明确要求 **仅在 127.0.0.1 或受信任内部网** 使用，或在生产环境自行加层 Nginx/防火墙。 |
| **可维护性** | - 将信息收集职责抽离到独立模块，代码职责更单一，后续若需扩展（例如注册其他类型的 per‑rank 元数据）可以在同一 Server 中统一管理。<br>- 删除了 `parse_remote_instance_transfer_engine_info_from_scheduler_infos`，减少冗余实现。<br>- `ServerArgs.url` 方法签名变更（新增 `port` 参数）可能影响少数自定义调用，需要搜索项目内部或第三方扩展的兼容性。 |
| **兼容性** | - 老的 `/get_remote_instance_transfer_engine_info` 仍保留（已标记为 **deprecated**），但内部实现改为调用新 Bootstrap 服务，保证向后兼容。<br>- 新增 `--engine-info-bootstrap-port` 参数默认 6789，若用户已有服务占用此端口，需要自行覆盖。<br>- 需要确保 `requests`、`uvicorn`、`fastapi` 已在运行环境中可用（已在项目依赖中），否则启动会报 ImportError。 |

### ⚠️ 潜在风险
1. **端口冲突**：在同一机器上启动多个 SGLang 实例时，如果未显式指定不同的 `--engine-info-bootstrap-port`，会导致 `RuntimeError`（已在代码中检测）。  
2. **网络暴露**：若 `host` 参数设为 `0.0.0.0`，Bootstrap Server 将对外可达，可能被未授权访问。  
3. **服务启动顺序**：只有 **node_rank==0** 会启动 Bootstrap Server；其它节点在 `Engine.__init__` 时期若提前访问 `scheduler_init_result.engine_info_bootstrap_server`，会得到 `None`，导致 `remote_instance_transfer_engine_info` 为 `None`。若用户在非 `node_rank==0` 上错误开启 `remote_instance_weight_loader_start_seed_via_transfer_engine`，会触发异常。  
4. **异常注册**：ModelRunner 在注册信息失败时仅记录错误，不抛出异常，可能导致后续查询返回 404，业务侧需要做好容错。  
5. **`ServerArgs.url` 参数变更**：外部脚本如果直接调用 `ServerArgs.url()`（不传 `port`），仍兼容；但如果有自定义实现依赖原签名的 `url`，需检查。  

---

### 💡 关注建议
- **部署规范**：在多实例或多节点部署时，**显式指定不同的 `--engine-info-bootstrap-port`**，并确保每个实例的 `host` 绑定在内部网络或 `127.0.0.1`。  
- **安全加固**：如果必须对外暴露端口，建议在前置代理层（Nginx、Envoy）添加 **IP 白名单 + TLS**，或在代码中实现 token‑based 鉴权。  
- **监控 & 健康检查**：利用已提供的 `/health` 接口监控 Bootstrap Server 的存活状态；在部署脚本中加入端口占用检查（`is_port_available`）以提前捕获冲突。  
- **错误处理**：在业务层调用 `/remote_instance_transfer_engine_info` 时，做好 **404 / 5xx** 的回退逻辑，以免因单个 Rank 注册失败导致整体服务不可用。  
- **文档更新**：在官方文档中加入 **EngineInfoBootstrapServer** 的使用说明、端口冲突防护以及安全建议。  
- **回归测试**：已提供的 `test_cross_node_scheduler_info_sync.py` 覆盖了多节点、DP+TP 组合，建议在 CI 中加入该脚本，确保未来更改不破坏信息同步路径。  
- **代码审计**：关注 `EngineInfoBootstrapServer.close()` 的调用时机，确保进程退出前服务器能够优

### [HiMambaTree]: Optimize mamba host lock mechanism (#21750)
**SHA**: `91048b2` | 🔗 [查看提交](https://github.com/sgl-project/sglang/commit/91048b2a8e5b47afc6ffd5f52dd4cb54ecfa44f6)

**🎯 变更类型**：功能增强  
**⚡ 重要程度**：🟡 中  

**📋 变更摘要**  
- 为 `TreeNode` 增加 `host_mamba_ref_counter`，并实现 `protect_host_mamba / release_host_mamba` 接口，区分 KV 与 Mamba 两类 host 资源的保护/释放。  
- 调整 `hi_mamba_radix_cache` 中的节点保护、释放、驱逐以及 LRU 列表维护逻辑，使得在备份、预取以及 Mamba‑host 分配时能够单独控制 Mamba 资源的生命周期。  
- 更新若干断言、状态检查、备份/预取路径的调用签名，以确保在两类资源均为 0 时才进入可驱逐集合。

**🎯 影响范围**  
- `python/sglang/srt/mem_cache/hi_mamba_radix_cache.py`（主逻辑）  
- `python/sglang/srt/mem_cache/mamba_radix_cache.py`（节点数据结构）  
- 相关的备份、预取、驱逐、LRU 管理流程。

**💡 关注建议**  
1. **测试覆盖**：重点加入以下场景的单元/集成测试：  
   - 仅保护 KV 不保护 Mamba、仅保护 Mamba 不保护 KV、两者同时保护以及全部释放的完整生命周期。  
   - 备份失败、预取中途异常、LRU 列表被错误删除的回滚路径，确保计数器始终保持非负。  
2. **并发安全**：当前计数器在多线程/协程环境下仍是普通整数，若缓存层被并行调用，请确认外层已加锁或改为原子操作。  
3. **日志/断言**：新加入的断言在生产环境可能触发，建议在异常路径加入更友好的错误信息或降级方案，以免因计数不匹配导致服务崩溃。  
4. **向后兼容**：`.ongoing_backup` 现在保存 `(node, mamba_host_protected)` 元组，确保旧的持久化或外部工具不直接假设其值为节点本身。  
5. **文档更新**：在 `TreeNode` 与缓存模块的 API 文档中标明 `host_mamba_ref_counter` 的语义、何时会自动置零，以及 `protect_host_mamba` / `release_host_mamba` 的使用约束。  

通过上述检查与补充测试，可降低计数器不同步导致的内存泄漏或提前驱逐风险，确保在 Mamba 加速路径下的内存管理更稳健。

5. **性能监控**：`InputValidationStage` 的执行时间提升明显（约 50%），需关注在大并发场景下是否成为瓶颈，并考虑在 `max_area` 计算前做快速路径判断。

| **资源占用**：预编译会在单机上并行启动 7 × 2 = 14 个子进程，即使 GPU 数不足 8 也会尝试编译所有 world‑size。若机器内存或 CUDA 编译缓存受限，可能出现 OOM 或长时间卡住。建议在异常捕获中添加超时或限制并发数（如 `concurrent.futures.ProcessPoolExecutor(max_workers=4)`）。 |
| **`result` 张量移动**：原实现在 GPU 上创建再 `all_reduce`，现在改为 CPU 张量并使用 `cpu_group`，行为保持，但若以后改动 `cpu_group` 为 NCCL，需恢复 `.to(device)`。确保后续改动不破坏这一假设。 |
| **测试覆盖**：只有 `nproc==1` 时触发预编译，其余 `nproc>1` 仍走旧的 `torchrun` 流程。若 CI 并行运行多个测试文件，可能出现编译竞争。可以在 `torchrun` 前统一调用一次预编译（如 `pytest_addoption` 钩子），避免重复编译。 |
| **进度条移除**：使用 `tqdm` 的代码被删掉，若未来需要可视化进度，建议重新封装为可选的 `--show-pbar` 参数。 |

**总体评估**  
此提交主要提升 CI/单元测试的编译速度，对功能本身没有影响，风险集中在多进程启动和向后兼容。只要在上述风险点加以防护，代码改动是安全且有价值的。建议在合并前加入对 `mp.set_start_method` 的防护以及在文档中注明 `custom_all_reduce_pull/push` 的新名称。

### [NPU][diffusion]: support parallel decoding of qwen-image (#20757)
**SHA**: `752d260` | 🔗 [查看提交](https://github.com/sgl-project/sglang/commit/752d260c771187453076fc714ad504f37d7863d9)

**变更类型**：功能增强  
**重要程度**：🟡 中  

**变更概述**  
1. 在 `qwenimage.py` 中为 VAE 配置新增 `use_parallel_decode` 开关，默认关闭。  
2. `autoencoder_kl_qwenimage.py` 大幅改造：  
   - 引入 `torch.distributed`，实现跨 GPU 并行解码。  
   - 新增 `_decode_with_parallel_dispatch`，在多卡环境下调用 `parallel_tiled_decode` 并裁剪多余帧。  
   - `_decode` 仍保留原实现；所有解码入口统一走 `decode` → `_decode_with_parallel_dispatch`。  
   - 将原来的逐像素 blend 改为向量化实现，提升 CPU/GPU 效率。  
   - 对 `blend_v/h` 增加 `blend_extent<=0` 的快速返回。  
3. `common.py` 中抽取出并优化了并行 tiled 结果的合并逻辑：  
   - `_merge_parallel_tiled_results` 完成所有 rank 的结果拼接、时空融合与混叠处理。  
   - `_process_parallel_tiled_outputs` 统一收集、pad、all‑gather 并调用前者。  
   - `parallel_tiled_decode` 现在只负责本 rank 的切片解码，余下工作交给 `_process_parallel_tiled_outputs`，代码更简洁、容错性更好。  

**影响范围**  
- VAE 配置体系 (`sglang.multimodal_gen.configs.models.vaes.qwenimage.QwenImageVAEConfig`)  
- Qwen‑image VAE 实现 (`autoencoder_kl_qwenimage.AutoencoderKLForQwenImage`)  
- VAE 通用 tiled 处理基类 (`common.ParallelTiledVAE`)  

**关注建议**  
1. **兼容性**：`use_parallel_decode` 默认为 `False`，确保单机/单卡仍走原路径；建议在文档中标明该字段的作用与开启条件。  
2. **分布式同步**：`_process_parallel_tiled_outputs` 中多次使用 `dist.gather`、`dist.all_gather_object`、`dist.broadcast`，需确保 `torch.distributed` 已初始化；若未初始化会抛异常，建议加 guard。  
3. **空 batch 处理**：在 `parallel_tiled_decode` 若本 rank 没有分配到 tile，`local_results` 为空，已返回空 tensor，但后续 `all_gather_into_tensor` 仍会产生 zeros，需确认不会导致 shape 不匹配。  
4. **性能验证**：向量化 blend 与分布式 gather 可能带来显著加速，但也会引入额外内存拷贝。建议在多卡（2、4、8）环境下跑一次完整的 encode‑decode 基准，比较 DecodeLatency 与显存占用。  
5. **错误检查**：`_merge_parallel_tiled_results` 依赖 `self.tile_sample_stride_*` 与 `self.blend_*` 参数，若配置不一致会出现维度不匹配，建议在函数入口加入断言。  
6. **测试覆盖**：新增单元测试应覆盖：① 单卡 `use_parallel_decode=False` 的行为保持不变；② 多卡开启 `use_parallel_decode=True` 时，输出与单卡全局解码结果一致（对比 L2‑norm）。  

总体而言，本次改动为 Qwen‑image VAE 增加了跨卡并行解码能力，代码结构更清晰，向量化实现提升效率。重点关注分布式初始化、空 tile 场景以及与原实现的数值一致性。
