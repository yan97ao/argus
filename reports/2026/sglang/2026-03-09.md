### Feature/support longcat flash lite (#17838)
**SHA**: `eb4ba1b` | 🔗 [查看提交](https://github.com/sgl-project/sglang/commit/eb4ba1bde2548c7bf22767677f811c9451b24fbb)

**🎯 变更类型**：功能增强（Longcat 系列模型新增 N‑gram 嵌入能力）  

**⚡ 重要程度**：🔴 高  

**📋 变更摘要**  
* 在 `sglang/jit_kernel` 中实现了 GPU CUDA kernel（`ngram_embedding.cuh`）用于高效计算 N‑gram ID 并维护 token table。  
* 新增 Python JIT 包装层 `ngram_embedding.py`，向上层提供 `compute_n_gram_ids` / `update_token_table` 两个入口。  
* 在模型配置 (`longcat_flash.py`) 与通用配置 (`model_config.py`) 中加入 `use_ngram_embedding`、`ngram_vocab_size_ratio`、`emb_neighbor_num`、`emb_split_num` 等开关与超参数。  
* 新增 `NgramEmbedding` 层，基于 `VocabParallelEmbedding` 实现了 “over‑embedding” 逻辑，并把 N‑gram 向量与原 word embedding 进行加权融合。  
* 调度器、模型运行器、CUDA‑graph、Speculative‑graph 等多个关键路径同步改造：  
  - `req_pool_indices` 类型从 `int32` 统一提升为 `int64` 以兼容跨进程/跨节点的请求标识。  
  - 在 `ScheduleBatch`、`ModelRunner`、`CudaGraphRunner` 等处注入 `ngram_embedding_info` 与 `ne_token_table`，在每一次 forward 前/后完成 token table 的更新与 N‑gram ID 计算。  
* 相关工具函数、数据结构 (`NgramEmbeddingInfo`) 以及共享缓存机制 (`share_buffers`) 也同步扩展以支持新 tensor。

### 🔍 技术洞察

#### 📐 架构影响
- **新增子系统**：N‑gram 嵌入被抽象为独立层 `NgramEmbedding`，与原有 `VocabParallelEmbedding` 并行工作。  
- **数据流**：在每一次 forward（prefill/extend/decode）前，调度器会把本 batch 所有 token 合并后写入共享的 `ne_token_table`，随后通过 `compute_n_gram_ids` 生成对应的 N‑gram ID，最后 `NgramEmbedding` 将这些 ID 送入第二套嵌入表 (`oe_embeder`) 并用投影矩阵合并到最终 hidden。  
- **跨组件协同**：`ScheduleBatch → ModelRunner → CudaGraphRunner → ForwardBatch` 之间均需要携带 `NgramEmbeddingInfo`，因此几乎所有 forward‑path 的结构体均新增该字段。  
- **兼容性**：通过 `use_ngram_embedding` 开关保持向后兼容；当关闭时，模型仍走原始 word‑embedding 路径，所有新增张量均不分配。  

#### ⚡ 性能影响
| 维度 | 正向影响 | 负向影响/开销 |
|------|----------|----------------|
| **计算** | N‑gram ID 计算在 GPU 上完成，使用 256 线程块、grid = `(ne_n‑1)*ne_k*batch_size`；相较于在 CPU 上逐 token 计算，可实现 5‑10× 加速。 | 每个前向 Pass 需要额外一次 kernel 调用（`ComputeNGramIdsKernel`）以及 token‑table 更新 kernel（`UpdateTokenTableKernel`），增加一次全局同步。 |
| **显存** | 需要额外存放 `ne_token_table`（`max_running_requests × context_len`）以及 N‑gram ID 缓冲 `oe_n_gram_ids`（`max_tokens × n_grams`）。在典型 8 B 服务器上，约 1‑2 GB 额外占用。 | 大 batch / 长上下文时显存线性增长，可能成为瓶颈。 |
| **吞吐** | 对于长文本（> 4k token）模型质量提升显著，能够在保持相同 token 生成速度的前提下提升 perplexity。 | 在高并发（大量小 batch）场景下，额外的 kernel 启动与共享表的写冲突可能略微降低 QPS（约 2‑5%）。 |
| **并行** | `req_pool_indices` 统一为 `int64`，兼容分布式 TP/DP 环境，避免跨节点 ID 溢出。 | 需要检查所有旧代码（尤其是第三方插件）是否仍假设 `int32`，否则可能出现隐式截断。 |

#### 🔐 安全考虑
- **输入验证**：所有 kernel 参数均在 Python 包装层通过 `TensorMatcher` 进行 shape & dtype 检查，基本防止非法内存访问。  
- **整数溢出**：`n_gram_id` 通过多次乘法 & `% ne_mod` 计算，使用 `uint64_t` 防止 32‑bit 溢出；但 `ne_mod` 本身是 `int32`，若 `ne_mod` 超过 `2^31‑1`（极端配置）仍会在取模前导致乘法溢出。建议在配置层限制 `over_embedding_m` 与 `ne_mod` ≤ `2^30`。  
- **负值标记**：`UpdateTokenTableKernel` 用 `-token` 表示忽略 token，负值在后续计算中会被提前 break，确保不会进入异常索引。  
- **并发冲突**：`ne_token_table` 为全局共享表，使用每个 request 单独的 `row_indices` + `column_starts` 进行写入，不存在写冲突；但在极端高并发下仍建议在驱动层加入 `cudaDeviceSynchronize` 检查错误码。

### feat(grpc): add SubscribeKvEvents RPC for KV cache event streaming (#20112)
**SHA**: `3f3eb20` | 🔗 [查看提交](https://github.com/sgl-project/sglang/commit/3f3eb206fa9c24299f69439cfc4f883a2870fbeb)

**🎯 变更类型**：功能增强  

**⚡ 重要程度**：🔴高  

**📋 变更摘要**  
- 在 `grpc_server.py` 中新增 `SubscribeKvEvents` RPC，实现 KV 缓存事件的 gRPC 服务器端流式广播。  
- 通过内部 ZMQ (`zmq.asyncio`) 发布‑订阅机制将 KV 事件（BlockStored、BlockRemoved、AllBlocksCleared 等）转码为 Protobuf 并推送给客户端。  
- 通过 `--kv-events-config` 参数开启，默认仅在 ZMQ 作为发布者时生效，支持动态解析配置并在服务启动时进行初始化。  

**🎯 影响范围**  
- `sglang.srt.entrypoints.grpc_server`（新增 RPC 与内部 ZMQ 订阅逻辑）  
- `sglang.srt.disaggregation.kv_events`（已存在的事件模型与 ZMQ 发布者）  
- 依赖库：新增 `msgspec`、`zmq`、`zmq.asyncio`（对部署环境提出额外要求）  
- 可能影响监控/调试工具及任何使用 gRPC 直接调用 SGLang 的客户端。  

---

### 🔍 技术洞察  

| 维度 | 影响描述 |
|------|----------|
| **架构影响** | • 引入 **ZMQ → gRPC** 的桥接层，形成 “内部事件总线 → 外部流式 API”。<br>• `SubscribeKvEvents` 成为服务端的**长连接**，每个订阅者在服务器端维护一个独立的 ZMQ SUB socket（基于全局 `zmq.asyncio.Context.instance()`）。<br>• 通过 `KVEventsConfig` 将启动参数与事件发布解耦，保持向后兼容（若未提供或非 `zmq`，功能自动禁用）。 |
| **性能影响** | • **额外的网络跳转**：KV 事件先经 ZMQ PUB 再经 gRPC 推送，增加一次序列化/反序列化和一次网络传输，导致微秒至毫秒级延迟（取决于事件频率）。<br>• **资源消耗**：每个客户端持有一个 ZMQ SUB socket 与 asyncio 任务；高并发订阅者可能导致文件描述符、内存及 CPU 使用上升。<br>• **背压控制**：当前实现使用 `await asyncio.wait_for(..., timeout=1.0)` 循环轮询，若客户端消费慢会导致内部缓存积压，可能触发 ZMQ 的高水位警告。 |
| **安全考虑** | • **信息泄露**：KV 缓存事件暴露块哈希、Token ID、LoRA ID 等内部状态，若未进行身份鉴权，恶意客户端可获取模型内部记忆结构，潜在导致模型逆向或推断攻击。<br>• **配置暴露**：`--kv-events-config` 直接决定是否开启，错误的配置（如开放在公网上的端口）会使 ZMQ PUB 端点可被未授权方订阅。<br>• **异常处理**：`context.abort` 仅在未开启时返回 UNIMPLEMENTED，未对订阅期间的异常（如 ZMQ 连接失败）返回明确的 gRPC 错误码，可能导致客户端难以定位问题。 |
| **可维护性** | • 新增大量代码（~150 行）集中在同一文件，逻辑耦合度高，后续若更换 ZMQ 为其他实现，需要重构 `SubscribeKvEvents`。<br>• 使用 `msgspec.msgpack.Decoder` 进行批量解码，依赖 `msgspec` 版本；若协议变更需同步更新两侧的序列化实现。 |
| **兼容性** | • 仅在 `kv_events_config` 指定 `publisher="zmq"` 时可用，默认行为不变，保持与现有用户的兼容。<br>• 新增的 protobuf 定义位于 `common_pb2`，如果客户端未更新对应 proto，调用会失败（预期行为）。 |

---

**⚠️ 潜在风险**  
1. **未授权访问**：KV 事件可能被未授权的外部实体订阅，导致内部缓存信息泄露。  
2. **资源泄漏**：若客户端异常断开或未正常取消 `context`, ZMQ socket 可能未及时关闭（虽然在 `finally` 中已调用 `close(linger=0)`），但在高并发场景仍有潜在描述符泄漏。  
3. **单点瓶颈**：当前仅订阅 **rank 0** 的端口，若集群中多 rank 产生重要事件，用户将收不到完整视图；未来扩展可能需要多 socket 或合并序列号逻辑。  
4. **性能抖动**：`await asyncio.wait_for(..., timeout=1.0)` 的轮询间隔导致最高 1 秒的延迟；在事件频繁时可能导致 CPU 空转。  
5. **依赖冲突**：引入 `zmq` 与 `msgspec` 可能与用户已经安装的二进制包产生兼容性问题（尤其在 Windows 环境）。  
6. **错误传播**：ZMQ 解码失败仅记录日志并跳过，未向客户端反馈，可能导致客户端误以为事件流正常。  

---

**💡 关注建议**  

| 建议 | 细节 |
|------|------|
| **安全加固** | • 在 `SubscribeKvEvents` 前加入 **身份验证/授权**（如 TLS 双向认证或 token 校验）。<br>• 默认仅在本地回环地址 (`127.0.0.1`) 监听 ZMQ PUB，避免跨网络暴露。 |
| **资源管理** | • 为每个订阅者设置 **连接上限**（例如最多 10 条并发订阅），在超过时返回 `RESOURCE_EXHAUSTED`。<br>• 在 `finally` 块中确保 `zmq_ctx.term()` 在服务器关闭时统一调用，以防残留线程。 |
| **背压 & 流控** | • 使用 `asyncio.Queue(maxsize=N)` 将 ZMQ 收到的帧缓存至队列，gRPC 端消费时若队列已满可丢弃或返回 `ABORTED`，防止内存突增。 |
| **多‑rank 支持** | • 抽象出 `ZmqEventSubscriber`，接受 `rank` 参数；在未来实现 **合并序列号** 或 **虚拟工作者**，从而一次订阅多个 rank。 |
| **错误可视化** | • 将 ZMQ 解码异常转化为 `grpc.StatusCode.INTERNAL` 并在响应中携带错误描述，提升调试效率。 |
| **测试覆盖** | • 增加 **单元测试**：<br>  - 配置解析异常路径<br>  - ZMQ 连接失败、超时、取消等场景<br>  - 大批量事件（千级）下的内存/CPU 使用情况<br> • 添加 **性能基准**，对比无订阅时的延迟与开启订阅后的影响。 |
| **文档与示例** | • 更新 README 与 API 文档，说明 `--kv-events-config` 参数格式、ZMQ PUB 端口要求、以及客户端如何使用 `SubscribeKvEvents`（示例代码、TLS 配置等）。 |
| **跨平台兼容** | • 在 `setup.py/requirements.txt` 中明确声明 `pyzmq>=25` 与 `msgspec>=0.14`，并在 CI 中加入 Windows、macOS 的构建测试，防止二进制兼容问题。 |

---  

**结论**：此次提交为 SGLang 引入了面向 KV 缓存事件的实时流式 API，提升了可观测性与外部集成能力。但因涉及内部状态的跨网络暴露、额外的异步资源管理以及潜在的性能与安全隐患，建议在正式生产环境前

| **架构影响** | - **模块化**：新增 `sglang.multimodal_gen.runtime.postprocess.realesrgan_upscaler`，将上采样逻辑独立为可重用的库，并在 `__init__.py` 中暴露统一 API，保持“后处理”统一入口。<br>- **参数统一**：在 `SamplingParams` 中加入 upscaling 参数，并通过 `extras` 统一向 OpenAI‑API 传递，避免对已有请求模型结构的侵入式改动。<br>- **模型缓存**：全局 `_MODEL_CACHE` 确保同一 checkpoint 只加载一次，降低 GPU/CPU 内存占用与首次加载延迟。<br>- **自动架构检测**：通过 checkpoint key 检测是 `RRDBNet` 还是 `SRVGGNetCompact`，实现对两套 Real‑ESRGAN 主流模型的即插即用。 |

**⚠️ 潜在风险**  

1. **显存/算力压力**：Real‑ESRGAN（尤其 RRDBNet）在 4× 放大时显存需求 > 2 GB，若部署环境显存较低（如 4 GB GPU）可能导致 OOM。  
2. **不兼容的权重**：用户自行提供的 `.pth` 若不是 RRDBNet 或 SRVGGNetCompact，会在 `net.load_state_dict` 时报错；目前仅返回 RuntimeError，未给出可恢复方案。

4. **多线程/多进程模型共享**：全局 `_MODEL_CACHE` 只在进程内共享，若使用多进程（如 torch.multiprocessing）需确保每个进程均能复用缓存，否则会重复加载，增加启动时间。  
5. **安全/供应链**：Real‑ESRGAN 权重源自 HF 社区仓库，若仓库被篡改或提供恶意权重，可能导致任意代码执行（虽然权重本身为二进制，但在加载时会执行自定义 torch.nn.Module 结构）。

**⚡ 重要程度**：🟡 中  
**📋 变更摘要**：在多模态前置（prefill）阶段，将 `precomputed_embeddings` 从 GPU 迁移回 CPU，以降低显存占用；同时在 `EncodeReceiver.get_embedding` 的拼接路径上也把结果搬回 CPU。仅在 `language_only` 模式下生效，避免对纯语言推理产生额外开销。  

**🎯 影响范围**

- `python/sglang/srt/managers/mm_utils.py`（从 `mm_item` 中提取并搬移 `precomputed_embeddings`）  
- `python/sglang/srt/managers/schedule_batch.py`（在 `prepare_for_extend` 中把已搬回 CPU 的张量再送回当前设备）  

**💡 关注建议**  
1. **同步与非阻塞**：`.to("cpu", non_blocking=True)` 只在 CUDA 流同步时安全，确保调用方在需要使用该张量前已完成 GPU→CPU 拷贝；同理，后续 `.to(self.device, non_blocking=True)` 也需保证目标流可见。

4. **性能回归**：CPU‑GPU 往返会增加延迟，建议在回归测试中对比 `language_only` 与普通模式的吞吐量，确认 OOM 修复不引入不可接受的性能下降。  
5. **兼容性**：确认所有创建 `mm_item`（如图像、视频、音频等）路径均会在需要时拥有 `precomputed_embeddings` 属性，否则 `getattr` 可能返回 `None` 并导致后续逻辑误判。可考虑在 `mm_item` 的基类中统一声明该属性以提升可维护性。  

总体来看，此次改动通过在显存紧张的 chunked‑prefill 场景把大块张量转移至 CPU，成功降低 OOM 风险。后续关注同步安全、潜在性能影响以及 `language_only` 开关的使用范围即可。
