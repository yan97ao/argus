### [NPU] Support GGUF quantization for Ascend NPU (dense + MoE) (#17883)
**SHA**: `046c14a` | 🔗 [查看提交](https://github.com/sgl-project/sglang/commit/046c14a3edd286ee6ab501224078ca5c7f1d7f0a)

**🎯 变更类型**：功能增强 / 性能优化 / 重构 / 架构变更  
**⚡ 重要程度**：🔴高  

**📋 变更摘要**  
本次 PR 为 Ascend NPU（A2/A3 系列）引入了 GGUF 量化模型的完整支持，覆盖 dense 与 MoE 两大算子。实现方式为在模型加载阶段使用 `gguf` 库在 CPU 上完成全部 dequant 化（预先转为 FP16/BF16），随后将去量化后的权重搬迁到 NPU，配合全新的 `GGUFLinearAscendMethod / GGUFMoEAscendMethod / GGUFEmbeddingAscendMethod` 以及相应的权重加载、材料化逻辑，确保在 NPU 上实现零运行时 dequant 开销。文档、模型加载映射、权重迭代器以及测试用例均同步更新。

**🎯 影响范围**  
- `sglang/srt/layers/linear.py`、`sglang/srt/layers/moe/fused_moe_triton/layer.py`（权重加载与材料化）  
- `sglang/srt/layers/quantization/gguf.py`（新增 NPU 检测、实现 NPU‑专属量化方法）  
- `sglang/srt/model_loader/loader.py`、`sglang/srt/model_loader/weight_utils.py`（模型/权重映射）  
- `sglang/srt/models/qwen2_moe.py`、`sglang/srt/models/qwen3_moe.py`（模型构造接受 `quant_config`）  
- 文档 `docs/advanced_features/quantization.md`、`docs/platforms/ascend/ascend_npu_quantization.md`  
- 新增 Ascend NPU 测试 `test/registered/ascend/basic_function/quant/test_npu_gguf*.py`  

**🔍 技术洞察**  

- **架构影响**  
  - 在 `gguf` 量化路径中加入 *NPU 分支*，通过 `is_npu` 判断动态切换实现，保持原有 CUDA/MUSA 路径不受影响。  
  - 引入 `UninitializedParameter` + `materialize_gguf_weights` 机制，解耦权重加载、sharding 与实际张量创建，提升了 MoE 权重在 TP 场景下的可组合性。  
  - MoE 权重的 `expert_data_map`、`data_container` 结构使得后续的 “按专家合并” 与 “预去量化” 能在统一入口完成，降低了不同算子对权重格式的耦合度。  

- **性能影响**  
  - **加载时**：CPU 端完成全部 dequant，导致模型启动时间相对纯量化模型会略增（尤其是大模型），但仅在一次性加载阶段发生。  
  - **推理时**：去量化权重已转为 FP16/BF16，NPU 只执行普通矩阵乘，避免了每次前向的 dequant 开销，显著提升吞吐与 latency（尤其在 FP16/BF16 优化的 Ascend A2/A3）。  
  - **内存占用**：预去量化会把原始压缩权重（几百 MB）扩展为 FP16（约 2‑4 倍），在显存受限场景需提前评估 `--mem-fraction-static` 参数。  
  - **MoE 方面**：使用 `npu_grouped_matmul`、`npu_moe_*` 系列原生算子，实现专家级并行与路由，理论上可以接近或超过 CUDA 上的实现，尤其在 2‑way Tensor‑Parallel 环境下。  

- **安全考虑**  
  - 仅在 CPU 上调用 `gguf` 库的 `dequantize`，不涉及外部 I/O，安全风险极低。  
  - 新增的 `is_npu` 检测依赖 `sglang.srt.utils`，若检测误判（例如在非 Ascend 环境误报为 NPU），会尝试加载不存在的 NPU 原生算子，导致 `RuntimeError`；这属于功能失效而非安全漏洞。  

**⚠️ 潜在风险**  

1. **加载延时 & 显存压力**  
   - 大模型（≥30B）在预去量化后显存需求可能超出单卡可用空间，导致 OOM。  
2. **Tensor‑Parallel 兼容性**  
   - 权重切分逻辑 (`output_dim`、`shard_id`）在多卡 TP 场景下仍有细碎 edge case，尤其是 MoE 的 `w13` 融合权重。  
3. **平台兼容性**  
   - 依赖 `torch.ops.npu.*` 系列算子，要求运行时安装对应 Ascend 驱动/torch-npu 版本。不同版本的算子签名变更可能导致调用失败。  
4. **权重一致性**  
   - `materialize_gguf_weights` 过程中对专家权重的拼接顺序依赖 `expert_data_map` 的遍历顺序，若出现非确定性遍历（Python <3.7）可能导致权重错位。  
5. **测试覆盖不足**  
   - 目前只验证了 Qwen3‑4B 与 Qwen3‑30B‑A3B 两个 GGUF 量化模型，其他模型（如 LLaMA、Mixtral）尚未覆盖，存在回归风险。  

**💡 关注建议**  

- **资源预估**：在部署前使用 `--mem-fraction-static` 并监控显存占用；对超大模型建议开启分段加载或使用 `offload` 方案。  
- **平台检测**：在 `is_npu` 判定后加入显式的版本校验（`torch_npu.__version__`），若不匹配则回退到 CPU 预去量化 + CUDA/MUSA 路径，避免误报导致的运行时崩溃。  
- **TP 分片校验**：加入单元测试，对 `output_dim` 为 0/1、`tp_size>1`、MoE `moe_tp_size>1` 的所有组合进行权重形状对齐检查。  
- **容错机制**：在 `materialize_gguf_weights` 中捕获可能的 shape/ dtype 不匹配异常，提供清晰的错误信息（如 “expert X weight shape mismatch, expected …”。）  
- **文档与示例**：补充显存需求估算表、典型启动参数示例，以及在多卡 TP 场景下的启动脚本（`--tensor-parallel-size` 与 `--attention-backend ascend` 的配合）。  
- **持续集成**：在 CI 中加入不同 Ascend 驱动版本的矩阵测试，确保算子签名兼容；对非 NPU 环境跑一次 “fallback to CPU” 的快速检查，防止误提交导致全

### [JIT Kernel] Reland JIT activation (#22094)
**SHA**: `82254bd` | 🔗 [查看提交](https://github.com/sgl-project/sglang/commit/82254bd9c5bba9cba42f5469e49cbe438adb5aa0)

**🎯 变更类型**：功能增强 / 性能优化 / 重构  
**⚡ 重要程度**：🔴高  

**📋 变更摘要**  
本次 PR 为 `sglang` 引入了 JIT‑编译的激活算子（`silu`、`gelu`、`gelu_tanh`），实现了统一的 **jit_kernel.activation** 模块，并在多处 CUDA 路径（SRT、MoE、Multimodal 等）切换为该实现。新增 PDL（Primary‑Dependent‑Launch）可选特性、Fast‑Math 编译标志以及完整的单元测试、基准测试，提升了运行时性能并统一了代码风格。

**🎯 影响范围**  
- `python/sglang/jit_kernel/activation.py`、`csrc/elementwise/activation.cuh`（核心 JIT 激活实现）  
- `python/sglang/jit_kernel/utils.py`（新增 `is_arch_support_pdl`）  
- 多数 CUDA 相关层：`sglang/multimodal_gen/runtime/layers/activation.py`、`sglang/srt/layers/activation.py`、`sglang/srt/layers/moe/*`、`sglang/srt/layers/quantization/gguf.py`、`sglang/srt/layers/moe/fused_moe_triton/*` 等  
- 测试与 benchmark 目录 (`tests/test_activation.py`, `benchmark/bench_activation.py`)  
- CI 注册脚本和 CI 配置（新增对 JIT 单元和基准的 CI 入口）  

**🔍 技术洞察**  

- **架构影响**  
  - 引入 **JIT 编译层**：在运行时根据 CUDA 架构动态生成最优 kernel（向量宽度、`kMaxVecBytes`、Fast‑Math）。  
  - **PDL（Primary‑Dependent‑Launch）** 机制：在激活 kernel 中加入 `PDLWaitPrimary` 与 `PDLTriggerSecondary`，为后续二次 kernel（如后处理）提供非阻塞的依赖同步，可在 SM90+ 上进一步降低延迟。  
  - 通过 `is_arch_support_pdl()` 在 Python 层检测硬件是否支持 PDL，保持向后兼容。  
  - 统一了激活算子的入口：`run_activation`/`*_and_mul` 统一返回 `torch.Tensor`，并在 CUDA 路径使用 JIT，而在 HIP、CPU、XPU 等保持原有 AOT 实现。  

- **性能影响**  
  - **向量化**：使用 `kMaxVecBytes` 自动适配 128‑bit 或 256‑bit 向量加载，提升内存带宽利用率。  
  - **Fast‑Math**：在 SM90（或更低的 CUDA 架构）自动开启 `--use_fast_math`，显著加速 `expf/sinf` 等数学函数（牺牲极小的数值精度）。  
  - **PDL**：在多 kernel 流水线中避免全局同步，理论上可降低激活后续算子的排队时间。  
  - 基准 (`bench_activation.py`) 对比了 AOT、JIT 与 `torch.compile`，为后续性能回归提供了量化依据。  

- **安全考虑**  
  - 代码仅在受信任的运行环境中进行 JIT 编译，无外部输入直接参与源码生成，风险极低。  
  - 需要关注 **编译缓存**：`cache_once` 对 JIT 模块进行单例缓存，若缓存目录被篡改可能导致加载恶意二进制，建议在生产环境限制写权限或使用签名校验。  
  - `PDL` 机制不是同步点，错误的 `kUsePDL` 设定可能导致未初始化内存访问，但在实现中已通过 `RuntimeCheck` 进行边界校验。  

**⚠️ 潜在风险**  

1. **硬件检测错误**  
   - `is_arch_support_pdl()` 判断失误会导致在不支持 PDL 的 GPU 上仍尝试开启，可能出现未定义行为或性能下降。  

2. **Python Wrapper 过于薄弱**  
   - 移除对 `out` 张量形状等的显式检查，依赖 C++ 端 `TensorMatcher` 抛错；若 C++ 检查未覆盖的异常（如 stride 不匹配）可能导致 kernel 直接崩溃。  

3. **编译缓存污染**  
   - JIT 编译产物会被缓存至 `$HOME/.cache/sglang/...`（假设），不同 CUDA 版本或驱动切换后可能复用旧的二进制，产生兼容性错误。  

4. **CI 兼容性**  
   - 新增的 Fast‑Math 标志在 HIP 环境下被禁用，但若 CI 机器使用旧版 CUDA（<9.0），Fast‑Math 仍会被关闭，基准对比可能产生误导。  

5. **PDL 语义误用**  
   - `PDLWaitPrimary` 与 `PDLTriggerSecondary` 不是同步点，错误的 early‑exit 逻辑可能导致 secondary kernel 在数据未准备好时启动，产生错误结果。  

**💡 关注建议**  

- **完善硬件检测**：在 `is_arch_support_pdl()` 中加入 fallback 机制，并在日志层级 `INFO` 给出检测结果，便于调试。  
- **缓存失效策略**：在每次进程启动或 CUDA driver 版本变更时，自动清除对应的 JIT 缓存，以防旧二进制运行。  
- **增强 Python 参数校验**：在 `run_activation` 与 `*_and_mul` 中加入轻量级的 shape 检查（如 `out.shape == input.shape[:-1] + (hidden,)`），在不影响性能的前提下提升易用性。  
- **PDL 开关显式化**：提供环境变量 `SGLANG_ENABLE_PDL=0/1`，让用户可以强制关闭 PDL，以便快速定位潜在的 PDL 相关 bug。  
- **基准 CI**：在 CI 中分别跑 `SM90+` 与 `SM80` 两类 GPU，确保 Fast‑Math 与 PDL 路径均被覆盖；同时添加对 HIP、CPU 的回归测试。  
- **文档更新**：在项目 README 与 JIT 章节补充

### perf: eliminate attention DtoD copy by passing pre-allocated output to FA (#21985)
**SHA**: `587fd15` | 🔗 [查看提交](https://github.com/sgl-project/sglang/commit/587fd15bd272d03f78dcc58a3e771a88d0c40f64)

**🎯 变更类型**：性能优化  
**⚡ 重要程度**：🔴高  

**📋 变更摘要**  
本次提交在 FlashAttention 相关的 Python 接口、JIT Kernel 实现以及底层 CUDA C++ 扩展中，引入了 `out` 参数，允许上层直接传入预分配好的输出张量。通过让 FlashAttention kernel 将结果写入该张量，消除了原先的 `Device‑to‑Device`（DtoD）拷贝，从而降低了显存带宽消耗并提升了推理吞吐。  

**🎯 影响范围**  
- `python/sglang/jit_kernel/flash_attention.py`、`flash_attention_v3.py`、`flash_attention_v4.py`（新增 `out` 参数并向下传递）  
- `python/sglang/srt/layers/attention/flashattention_backend.py`（在 `forward_extend`、`forward_decode` 中创建并传递 `_fa_out`）  
- `python/sglang/srt/layers/radix_attention.py`（在统一注意力路径中保存预分配 output 并在必要时拷贝）  
- `sgl-kernel/csrc/flash_extension.cc`（C++ 接口签名改为 `Tensor(a!) out`，返回张量别名）  
- `sgl-kernel/python/sgl_kernel/flash_attn.py`（Python 包装层接受 `out` 并传递到底层）  

**🔍 技术洞察**  

- **架构影响**  
  - 增加了跨层 “预分配输出” 的隐式依赖：上层 `forward_*` 必须在调用 FlashAttention 前准备好合适尺寸的张量。  
  - 底层 C++ Kernel 通过 `Tensor(a!)` 标记实现**写入原位**，不再创建临时输出，符合 PyTorch “in‑place” 约定。  
  - 代码路径保持向后兼容：若 `out` 为 `None`，仍会走原来的分配‑返回路径，故对已有调用无强制破坏。

- **性能影响**  
  - 消除一次显存 DtoD 复制（约 `O(batch·seq·head·dim)` 的内存拷贝），在大模型/长序列情况下可降低 10%‑30% 的推理延迟。  
  - 减少显存峰值：原先的临时输出张量在复制后会被释放，预分配输出复用已有缓冲，提升显存利用率。  
  - 由于 `out` 在 CUDA kernel 中直接写入，CPU‑GPU 同步次数不变，仍保持原有异步执行特性。

- **安全考虑**  
  - 仅涉及显存读写，无新增网络或文件 I/O，安全风险极低。  
  - 需要确保 `out` 的 dtype、device、形状与预期严格匹配，否则会触发 CUDA runtime 错误或导致未定义行为。  

**⚠️ 潜在风险**  

1. **形状/类型不匹配**：若上层提供的 `out` 大小与实际 `q` 的批次/序列长度不一致，Kernel 会在入口处报错，可能导致服务崩溃。  
2. **梯度计算**：`out` 为 in‑place 写入，若后续 autograd 需要保留原始张量进行梯度回传，需要确认 PyTorch 对 `Tensor(a!)` 的梯度传播行为是安全的（目前已在 `torch.autograd` 中支持）。  
3. **并发复用**：在多线程/多流环境下，同一 `out` 张量被重复使用可能导致竞争写入，需在上层确保每个请求拥有独立缓冲。  
4. **向后兼容**：旧版自定义插件或第三方代码如果硬编码了返回张量而未提供 `out`，仍会正常工作，但可能错失性能收益。  

**💡 关注建议**  

- **测试覆盖**：新增单元测试/集成测试，尤其验证：
  - `out` 为 `None` 与提供有效 `out` 两种路径的数值一致性。  
  - 在不同 `real_num_tokens`（实际 token 数）下，`forward_batch._attn_output` 切片正确。  
  - 反向传播梯度与原实现保持一致。  

- **参数校验**：在 Python 包装层增加显式检查（shape、dtype、device），在不匹配时提前抛出友好异常，防止低层 CUDA crash。  

- **文档更新**：在 API 文档中说明 `out` 参数的用途、尺寸要求以及 “如果不提供，将使用内部临时张量” 的默认行为。  

- **性能基准**：提供基准报告（不同 batch/seq 长度、不同头数、不同显存配置），帮助用户评估在自家硬件上的实际收益。  

- **兼容性开关**：可考虑在配置中保留 `use_prealloc_out=False` 的开关，以便在调试或特殊环境（如旧版 CUDA）下回退到原实现。  

通过上述改动，FlashAttention 在推理阶段的显存流量和延迟有望显著降低，但需要在上层确保正确的输出缓冲管理，以避免潜在的运行时错误。继续关注后续的回归测试与跨平台表现。

---

**⚠️ 潜在风险**  

1. **显存激增**：多 LoRA（每个 LoRA 包含 `rank × dim` 参数）会在合并前保留所有适配器的 A/B 矩阵，低‑VRAM 环境仍可能 OOM。<br>2. **FP32 合并误用**：若环境变量误设为 `1`，且 LoRA 包含 **distilled‑LoRA** 权重，合并会在 FP32 中产生额外拷贝，消耗显存并降低性能。<br>3. **硬件模式不匹配**：`prepare_ltx2_upsample_after_stage1` 只在 `snapshot` + `dit_cpu_offload` + `low_vram` 条件下生效，若用户自行改动 `--ltx2-two-stage-device-mode` 但未同步此逻辑，可能导致 Stage‑2 未能预取，出现意外的延迟峰值。<br>4. **条件图像数量**：当前只支持 ≤ 2 张图像；若用户误传 3 张以上，将抛异常并中止请求，需在前端或 CLI 参数层做更友好的校验。<br>5. **兼容性**：旧版 LoRA 只含 transformer‑block 权重会触发 “missing layer” 警告，若用户忽视该日志可能误以为全部层已生效。
