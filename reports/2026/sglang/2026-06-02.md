### [diffusion] feat: add realtime webui super resolution controls (#27026)
**SHA**: `64a1dec` | 🔗 [查看提交](https://github.com/sgl-project/sglang/commit/64a1dec8b6a949c4c4165108ea98e1865bbc7676)

**🎯 变更类型**：功能增强 / 重构  
**⚡ 重要程度**：🔴 高  

**📋 变更摘要**  
- 为实时 WebUI 增加了“超分辨率（Super‑Resolution）”开关及放大倍率选择，并实现了预览缩放（preview scale）控制。  
- 后端新增 Real‑ESRGAN 超分辨率模型的多尺度加载逻辑，支持 2×、4×、8× 权重自动匹配并加入模型路径缓存。  
- 前端 UI、样式与交互细节重构：新增工作区容器、预览帧包装、加载指示、输出尺寸显示等；新增 `websockets` 依赖。  

预期效果：用户可在实时视频流中打开 SR，动态放大输出分辨率，同时保持低延迟模式下的帧率控制；UI 更直观，预览缩放提供更好的视觉调试体验。

---

### 🎯 影响范围
- **核心模块**：`sglang/multimodal_gen/apps/realtime_webui/`（`app.js`, `index.html`, `styles.css`）  
- **后端运行时**：`sglang/multimodal_gen/runtime/postprocess/realesrgan_upscaler.py`  
- **依赖**：`pyproject.toml` 新增 `websockets`（WebSocket 通信层）  
- **测试**：新增单元测试覆盖 SR 参数传递与后端 upscaler 调用。  

---

### 🔍 技术洞察

| 维度 | 影响分析 |
|------|----------|
| **架构影响** | - 前端通过 `readSuperResolutionParams()` 将 SR 参数（`enable_upscaling`, `upscaling_scale`）注入到 `init` 消息中，后端解码器原样转发，无需额外协议变更。<br>- 后端 `RealesrganUpscaler` 现在在实例化时依据 `scale` 选取对应权重文件，使用统一的 `repo_id:filename` 语法，提高可配置性。<br>- 引入全局缓存 (`_MODEL_CACHE`, `_RESOLVED_MODEL_PATH_CACHE`) 减少重复下载/解析，提升多实例启动速度。 |
| **性能影响** | - **前端**：在开启 SR 时，`renderLoop` 会强制使用 `catchupFps = targetFps`（不再降至低延迟阈值），避免因帧率波动导致卡顿，从而保持更平稳的放大视觉。<br>- **后端**：Real‑ESRGAN 的 tiled upscaling仍受内存约束（`_LOW_MEMORY_TILED_UPSCALE_FREE_BYTES`），但多尺度模型直接决定 `scale`，避免二次内部插值。缓存机制大幅减轻首次加载成本。<br>- **整体**：SR 引入额外的 CPU/GPU 计算，建议在资源受限的环境下关闭 SR；低延迟模式 (`!$("superResolution").checked`) 仍保留旧的 FPS floor。 |
| **安全考虑** | - 新增 `websockets` 依赖本身无安全风险，但需要确保服务器端使用安全的 `wss://`（TLS）并进行身份校验，防止中间人攻击。<br>- `readSuperResolutionParams()` 只读取 checkbox 与数值，无字符串注入风险。<br>- `_resolve_model_path` 仍支持本地路径，若允许用户自行指定本地文件，需要防止路径遍历；当前已做文件存在检查，风险可控。 |
| **可维护性** | - 前端代码拆分出 `setPreviewState`、`setPreviewScale` 等函数，职责单一，易于单元测试。<br>- CSS 采用 BEM‑like 结构（`.workspace`, `.preview-frame`），并通过 CSS 变量 `--preview-scale` 实现动态缩放，易于主题或响应式改动。<br>- 后端增加映射表 `_DEFAULT_REALESRGAN_FILENAMES_BY_SCALE`，对新增尺度扩展友好。<br>- 缓存键使用完整 `model_path` 字符串，可避免重复下载，但需注意缓存失效策略（如模型文件更新）。 |

### ⚠️ 潜在风险

1. **资源消耗激增**  
   - 开启 SR（尤其 4×、8×）对 GPU/CPU 需求显著提升，可能导致服务器 OOM 或响应延时。  
   - 建议在文档中标明最低硬件要求，并在 `connect()` 前检测显存/内存阈值，给出警告或自动禁用 SR。  

2. **模型缓存失效**  
   - `_MODEL_CACHE` 与 `_RESOLVED_MODEL_PATH_CACHE` 基于 `model_path` 键，若同一模型文件被更新但路径相同，缓存不会刷新，可能继续使用旧权重。需要提供手动 `clear_cache()` 或在文件修改时间变化时重新加载。  

3. **前端尺寸计算错误**  
   - `updateOutputSizeText()` 默认使用 `size` 输入框的分辨率乘以 `scale`，若用户手动编辑 `size` 与 `upscalingScale` 不匹配，可能出现显示的 `outputSizeText` 与实际渲染尺寸不一致。建议在 UI 中同步更新 `size` 或在参数校验阶段统一。  

4. **兼容性回退**  
   - 新增的 `websockets` 依赖在老版 Python 环境（如 3.7）可能不兼容；保证 `pyproject.toml` 中的最低 Python 版本与 `websockets` 要求一致。  

5. **前端状态同步**  
   - 当连接异常或关闭时，部分路径仅在 `renderedPreviewFrames == 0` 时调用 `setPreviewState("idle")`，如果已有帧但后续错误未触发 `setPreviewState`，UI 可能残留 `waiting/live` 状态。建议在所有异常退出分支统一恢复为 `idle`。

### 💡 关注建议

| 对象 | 建议 |
|------|------|
| **开发者** | - 为 SR 加入显存/算力检测（`torch.cuda.get_device_properties`），在 `connect()` 前给出 “SR 可能导致 OOM” 警告。<br>- 提供命令行/配置文件方式的 SR 默认关闭，以免在 CI 环境意外加载大模型。<br>- 在 `_resolve_model_path` 中加入缓存失效时间或基于文件哈希的二级校验。 |
| **用户** | - 在资源受限的机器上使用 `superResolution` 前请确认 GPU/CPU 能力，推荐 2× 为安全选项。<br>- 若出现 “Preview idle” 长时间停留，检查网络 latency 与服务器负载，或尝试关闭 SR 再重连。 |
| **测试** | - 增加集成测试：在开启 SR 时验证返回的 `header.width/height` 已被放大。<br>- 添加压力测试，模拟多并发连接开启 SR，观察内存占用与模型加载次数是否符合缓存预期。 |
| **文档** | - 更新 README/CHANGELOG，说明新增 `websockets` 依赖、SR 功能、推荐硬件规格及可能的内存开销。<br>- 在 UI 帮助提示（tooltip）中解释 “Super resolution” 与 “Preview scale” 区别：前者是输出分辨率，后者仅 UI 缩放。 |

### [codex] Centralize Triton utility kernels (#26000)
**SHA**: `559581b` | 🔗 [查看提交](https://github.com/sgl-project/sglang/commit/559581b383d31ec10a76d1df72534768ccebce01)

**🎯 变更类型**：功能增强 / 重构 / 性能优化  
**⚡ 重要程度**：🔴 高  
**📋 变更摘要**  
本次提交对 SGLang 代码库进行大规模的 Triton 核心算子抽象与集中化：

1. 将散落在多个层（`flashattention_backend.py、triton_backend.py、wave_backend.py、elementwise.py、logits_processor.py、paged.py、common.py、allocator.py、mem_cache/...`）的 Triton kernel 迁移、合并或重写，统一放置在 `sglang/srt/layers/triton_ops/` 与 `sglang/srt/mem_cache/triton_ops/` 子目录下。  
2. 新增 **软上限 (softcap)**、**缓存移动/复制**、**MLA KV‑buffer**、**位置计算**、**多层 Eagle**、**Cache‑loc 分配** 等一系列高效 Triton kernel，并为关键路径提供 **自动调优**（`triton.autotune`）配置。  
3. 移除旧实现（包括大量手写的 `@triton.jit` kernels），并在调用方统一切换到新的统一 API。  
4. 引入对 **HIP / MUSA / NPU** 等非 CUDA 平台的兼容层，解决了之前在 HIP 上出现的 `int32→int64` 存储 bug。  
5. 对 **内存分配**、**KV‑cache 写入**、**位置计算**、**软上限/激活函数**、**多层 Eagle（草稿/验证）** 等关键推理路径进行 **融合** 与 **一次性调度**，显著减少 kernel 启动次数和全局同步。

### perf(gemma4): single-launch fused router (topk + softmax + scale) (#26502)

**🔍 技术洞察**  

- **架构影响**  
  - 引入了 Triton 依赖，要求运行环境装有 Triton、CUDA 以及对应的 compute capability（>= sm80）才能使用该路径。代码在 `routing_function` 中做了 dtype 与 device 检查，若不满足条件仍回退到原有 PyTorch 实现，保持向后兼容。  
  - `gemma4_fused_routing` 对 `E`（专家数）做了上限 1024 的硬性限制，这在 Gemma‑4 规格内安全，但若以后模型扩展会触发断言，需要上层业务捕获。  
  - 通过 `triton.next_power_of_2(E)` 计算块大小，充分利用了 Triton 的 warp‑level排序与位操作，实现了 **“键-值”打包排序**，从而在单个 kernel 内完成 top‑k 选取，降低了 CPU‑GPU 同步开销和 kernel 启动延迟。  

- **性能影响**  
  - **启动次数**：原实现需要两次 kernel（`torch.topk` + `softmax`）以及一次 CPU‑side mask/scale 处理；新实现一次 kernel 完成全部工作，减少了 kernel 启动开销（数十 µs）以及跨 kernel 的全局内存写入/读取。  
  - **内存访存**：在 kernel 内部仅读取一次 `gating_output` 与一次 `per_expert_scale`，所有中间结果均在寄存器/共享内存中完成，显著降低了全局内存带宽压力。  
  - **算子融合带来的算子调度优势**：Triton 能够在同一 program 中进行 `sort`、`exp`、`sum`、`div`、`store`，避免了 PyTorch 自动调度产生的额外同步点。  
  - **数值精度**：所有 softmax 计算在 fp32 中完成，避免了 fp16/bfloat16 下的数值不稳定；测试显示对不同 dtype 仍保持在合理误差范围（bfloat16≈5e‑3）。

  - 只在 CUDA 环境下执行，未触及网络、文件系统或外部输入的安全门槛。

| 风险点 | 说明 | 严重程度 | 缓解措施 |

| **测试环境依赖** | 新增的文件后端测试依赖临时目录和环境变量；CI 环境若未正确清理目录，可能残留大文件导致磁盘耗尽。 | 低 | CI 脚本中加入 `tmpdir` 自动清理或限制文件大小。 |

3. `TokenizerManager._tokenize_one_request` 在构造 `MultimodalDataItem` 时若检测到 `mm_hashes`，会直接把哈希写入 `item.hash`，跳过内部 `hash_feature()` 的重新计算，从而让外部 KV 路由器的哈希与 sglang 前缀缓存键保持一致。  
4. 新增单元测试验证字段暴露、默认行为、`set_pad_value` 在预设哈希时不调用 `hash_feature`、以及哈希到 `pad_value` 的确定性。  

**🎯 影响范围**：  
- `python/sglang/srt/entrypoints/engine.py`（同步/异步生成接口）  
- `python/sglang/srt/managers/io_struct.py`（请求结构）  
- `python/sglang/srt/managers/tokenizer_manager.py`（tokenize 流程）  
- 新增 `test/registered/unit/managers/test_mm_hashes.py`（单元测试）  

**🔍 技术洞察**  

- **架构影响**：  
  - 在多模态数据流中插入了 “外部哈希” 入口，属于业务层面的扩展，不影响核心推理引擎的调用路径。  
  - 通过 `GenerateReqInput.mm_hashes` 将哈希信息从用户层直达 `MultimodalDataItem`，保持了已有模块的低耦合，新增的逻辑仅在 `TokenizerManager._tokenize_one_request` 中实现，未改动调度、执行或模型层代码。  
  - 为外部 KV 路由器提供了对齐前缀缓存键的能力，提升了系统在分布式缓存或路由场景下的可组合性。  

- **性能影响**：  
  - **正向**：当调用方提供有效的 `mm_hashes` 时，`hash_feature()`（涉及图像特征张量的哈希计算）被跳过，能够显著降低多模态请求的预处理开销，尤其在大批量图片或高分辨率图像场景下。  
  - **负向**：在 `mm_hashes` 长度不匹配或格式错误时会触发警告并回退到内部计算，导致一次额外的检查成本（O(N)）以及潜在的双重计算（一次失败的跳过检查后仍会执行内部哈希）。该成本极小，几乎可以忽略。  

- **安全考虑**：  
  - 允许外部直接注入哈希值，若攻击者提供恶意构造的哈希，可能导致前缀缓存键与实际图像特征不匹配，进而出现缓存冲突或错误命中。系统通过以下方式降低风险：  
    1. 长度不匹配或解析失败时自动回退到安全的内部哈希计算。  
    2. 只在 `MultimodalDataItem` 中保存整数哈希，未对外暴露任何执行路径。  
  - 仍建议在生产环境对 `mm_hashes` 的来源进行可信验证（如签名或内部生成），避免非法路由决策。  

**⚠️ 潜在风险**  

1. **长度或格式不匹配**：如果 `mm_hashes` 列表长度与实际图片数量不一致，当前实现仅记录警告并使用内部哈希，可能导致用户期望的缓存对齐失效。  
2. **异常传播**：`int(hex_hash, 16)` 的异常已被捕获并记录警告，但如果在未来代码路径中遗漏捕获，可能导致请求异常终止。  
3. **兼容性**：旧版调用方未传递 `mm_hashes`，行为保持不变；但若已有代码基于 `mm_hashes` 长度判断（如自行构造），可能需要更新以防误判。  
4. **测试覆盖**：新增的单元测试覆盖了核心逻辑，但未覆盖 `engine.generate` 的参数向下传递以及在实际推理服务中的端到端路径，建议在集成测试中加入相应场景。  

**💡 关注建议**  

- **对调用方**：  
  - 仅在能够可靠计算图像哈希且需要与外部 KV 路由器保持一致时传递 `mm_hashes`。  
  - 确保提供的哈希列表与 `image_data` 的顺序、数量严格对应。  
  - 若使用外部 KV 路由，请在生成哈希后校验其十六进制字符串（长度≥1、合法字符），以免触发回退。  

- **对内部开发者**：  
  - 将 `mm_hashes` 的校验逻辑抽象为独立函数，便于复用和单元测试。  
  - 在日志中加入请求 ID，以便在出现长度不匹配或解析错误时快速定位问题。  
  - 考虑在 `TokenizerManager._tokenize_one_request` 的入口处对 `mm_hashes` 做一次完整的类型与长度验证，提前抛出友好错误。  

- **对运维/安全团队**：  
  - 将 `mm_hashes` 的来源列入可信列表审计，防止恶意构造导致缓存冲突。  
  - 监控 `logger.warning` 中关于 `mm_hashes` 的信息频率，异常升高时检查调用方实现。  

通过上述改动，s​glang 在多模态路由场景下获得了更灵活的缓存键对齐能力，同时保持向后兼容和安全防护。后续可进一步完善验证与监控，以确保在大规模分布式部署时的可靠性。
