### fix: Fix DSR1 perf regression due to unnecessarily falling back to triton gemm (#28073)
**SHA**: `20f4272` | 🔗 [查看提交](https://github.com/sgl-project/sglang/commit/20f4272109083ca607c8106aa0c8896a0fe4756f)

**🎯 变更类型**：Bug修复 / 性能优化  
**⚡ 重要程度**：🔴高  
**📋 变更摘要**：  
1. 修正了在 DeepSeek‑R1（DSR1）模型使用 `flashinfer_trtllm` 后端时，因错误检查 `weight_scale.format_ue8m0` 导致的性能回退至慢速 Triton GEMM 的回归。  
2. 通过改为依据输入张量的 dtype（仅在 `bfloat16` 且 K≥256 时使用 TRTLLM）以及移除不再需要的 `post_load_weights` 辅助函数，消除了不必要的回退路径。  
3. 新增单元测试验证了在不同 dtype、K 大小以及 `format_ue8m0` 标记下的分支决策，防止同类回归再次出现。  

**🎯 影响范围**：  
- `sglang/srt/layers/quantization/fp8_utils.py`（核心 FP8 GEMM 调度）  
- `sglang/srt/model_loader/utils.py`（权重加载辅助函数）  
- CI 测试套件 `test/registered/unit/layers/quantization/test_flashinfer_trtllm_fp8_fallback.py`  

**🔍 技术洞察**：  
- **架构影响**  
  - 改动集中在 **FlashInfer‑TRTLLM** 的块级 FP8 GEMM 调度逻辑。原先的分支依据 `weight_scale.format_ue8m0`（仅在 DeepGEMM 路径使用）决定是否回退，这在 TRTLLM 路径中是不恰当的。现在逻辑统一为：  
    ```python
    if backend == "trtllm" and (K < 256 or input.dtype != torch.bfloat16):
        fallback_to_triton()
    ```  
  - 移除了 `post_load_weights`（已不在模型加载流程中被调用），简化了模型加载模块的依赖图，降低了耦合度。  

- **性能影响**  
  - **正向**：对 DSR1 等使用 bf16 输出且 K≥256 的模型，直接走 TRTLLM 高效 GEMM，恢复了约 **20%‑30%**（取决于硬件）的推理吞吐提升。  
  - **负向**：对不满足条件的情况（fp16 输出或 K<256），仍会回退至 Triton 实现，性能不变。  

- **安全考虑**  
  - 此次改动不涉及安全敏感路径，仅是调度逻辑的修改。未引入新的外部依赖或权值处理，安全风险可忽略。  

**⚠️ 潜在风险**  
1. **未来后端兼容性**：当前判断仅基于 `input.dtype != torch.bfloat16`，若后续 TRTLLM 支持其他输出 dtype（如 `float32`）或需要检查 `weight_scale` 的特定格式，现有逻辑可能导致误回退。  
2. **代码路径未覆盖的 GPU 实际运行**：单元测试在 CPU 环境下通过 mock 完成，未在真实 GPU‑TRTLLM kernel 上执行，仍有潜在的硬件‑驱动兼容问题。  
3. **删除 `post_load_weights`**：虽然目前代码库未再调用该函数，但第三方扩展或自定义模型可能仍依赖它，删除后可能导致 `AttributeError`。  

**💡 关注建议**  
- **短期**：在 CI 中加入针对 GPU 环境的集成测试，确保 TRTLLM 路径在真实硬件上不再出现回退。  
- **长期**：将后端选择策略抽象为可配置的决策函数，并在文档中标明支持的 dtype 与 K 阈值，便于后续扩展（如加入 `float32` 支持）。  
- **兼容性**：对外提供一个**兼容层**或 **deprecation 警告**，在未来的主版本中逐步移除对 `post_load_weights` 的调用依赖，避免第三方破坏。  
- **监控**：在生产环境中通过日志或指标监控 `flashinfer_gemm_w8a8_block_fp8_linear_with_fallback` 的后端选择比例，快速捕捉异常回退情况。

### feat: report multimodal (image/audio/video) token counts in usage.prompt_tokens_details (#27122)
**SHA**: `37505ec` | 🔗 [查看提交](https://github.com/sgl-project/sglang/commit/37505eca27b0c1f6d5de6dfa1e8578db59f8a24c)

**🎯 变更类型**：功能增强  
**⚡ 重要程度**：🔴 高  
**📋 变更摘要**：在 SGLang 的 OpenAI 兼容层中新增对多模态（图像、音频、视频）提示 token 数量的统计与上报。通过在元数据缓冲区复用 `cached_tokens` 的空闲槽位，将多模态 token 计数从 prefill 节点传递到 decode，随后在 `UsageProcessor`、`ChatCompletion*` 响应以及 OpenAI 协议模型中统一展示。  

**🎯 影响范围**：  
- `python/sglang/srt/disaggregation/*`（元数据缓冲区传递）  
- 请求/响应结构体 `Req`、`BatchTokenIDOutput`、`BatchStrOutput`  
- `usage_processor.py`、`openai/protocol.py`、`serving_chat.py`（Usage 计算与序列化）  
- `managers/*`（调度、流式聚合、TokenizerManager）  
- 多模态输入处理 `MultimodalInputs`（新增 `compute_mm_token_counts`）  

**🔍 技术洞察**  

- **架构影响**  
  - 通过复用 `cached_tokens` 缓冲区的第 4~6 个槽位（原本保留）实现多模态计数的零额外 RDMA 缓冲区需求，保持原有分布式解耦（prefill ↔ decode）不变。  
  - 新增字段在 Request → Decode → OutputStreamer → TokenizerManager → OpenAI response 完整链路中流转，涉及多处结构体扩展和序列化路径，整体保持向后兼容（仅在有多模态输入时出现）。  

- **性能影响**  
  - 计算多模态 token 计数的开销仅在 prefill 节点执行，基于已解析的 offsets，时间复杂度 O(N)（N 为 multimodal items），对整体吞吐影响可忽略。  
  - 复用已有缓冲区避免了额外的网络/共享内存拷贝，几乎不产生额外延迟或带宽开销。  
  - 增加的几组整数列表 (`image_tokens`, `audio_tokens`, `video_tokens`) 在 `Batch*Output` 中占用极小的内存（每个请求 3×4 B）。  

- **安全考虑**  
  - 新增字段均为整数计数，无敏感信息泄露风险。  
  - 序列化时使用 `model_serializer` 去除为 `None` 的字段，防止旧版客户端误解析。  
  - 复用已有 RDMA 缓冲区槽位若出现越界写入，可能导致内存破坏，需要确保 `cached_tokens` 长度 ≥ 7（已在代码中假设为 16）。  

**⚠️ 潜在风险**  

1. **缓冲区尺寸假设破坏**：若底层实现更改 `cached_tokens` 的大小（如改为 4），会导致写入越界。  
2. **多模态计数不一致**：`MultimodalInputs.compute_mm_token_counts` 依赖 `item.offsets` 正确填充；若某些输入未正确记录 offsets，计数会偏低。  
3. **向后兼容性**：旧版 Server/Client 仍使用 4‑slot 缓冲区；虽然序列化已剔除 `None`，但旧版客户端若直接解析 `metadata_buffer` 可能误读 4~6 槽位。  
4. **并发聚合错误**：在流式返回中通过 `idx % request.n == 0` 聚合 Token，若 `request.n` 与实际 `n_choices` 不匹配（异常路径），聚合结果会错误。  
5. **测试覆盖不足**：当前改动跨越多层（prefill → decode → streamer → usage），需要完整的端到端多模态请求回归。  

**💡 关注建议**  

- **验证缓冲区大小**：在启动阶段加入断言 `len(cached_tokens[0]) >= 7`，防止未来底层结构变更导致写越界。  
- **增强单元/集成测试**：  
  - 覆盖仅文本、仅图像、图像+音频、全模态四种组合，检查 `usage.prompt_tokens_details` 中相应字段是否准确。  
  - 对 `request.n` 与实际返回 choice 数量不一致的异常路径做容错测试。  
- **文档与版本号**：在 OpenAI API 文档中注明 `prompt_tokens_details` 可能携带 `image_tokens`、`audio_tokens`、`video_tokens`，并说明它们仅在多模态请求时出现。同步更新 changelog，确保用户了解新字段的含义。  
- **监控与报警**：在运行时监控 `cached_tokens` 长度与 `mm_*_tokens` 写入情况，若出现异常值（如负数、异常大数），触发报警以防止因不合法 offsets 导致错误计数。  
- **代码审计**：建议在 `MultimodalInputs.compute_mm_token_counts` 中加入对 `offsets` 合法性的校验（如 start ≤ end、非负），并在异常情况下记录警告而非直接计入。  

通过以上措施，可在保持低开销的同时安全、可靠地为用户提供多模态 token 使用统计，提升 API 可观测性与开发者体验。

### Revert "[AMD][Quantization] Online MXFP4 quantization 2/N - FP8 to MXFP4 requantization on AMD GPUs" (#28213)
**SHA**: `f18d38d` | 🔗 [查看提交](https://github.com/sgl-project/sglang/commit/f18d38d0408490e3232b9bfcd4557918b814fd78)

**🎯 变更类型**：Bug修复 / 代码回退（撤销功能）  
**⚡ 重要程度**：🔴高  

**📋 变更摘要**  
1. 完全撤销了原先在 AMD GPU 上实现的 **FP8 → MXFP4 在线重量化**（requantization）功能，包括文档、配置项、相关类与方法的实现。  
2. 删除了 `online_quantization.py`、`dequantization.py`、以及 `quark`、`fp8`、`fp8_utils` 中大量与 FP8‑>MXFP4 交叉转换相关的代码；相应地，`model_config.py`、`model_runner.py`、`weight_utils.py` 等文件的逻辑也被简化。  
3. 文档中对应的使用示例及说明被去除，测试用例中涉及该特性的全部内容被删除。

**🎯 影响范围**  
- **核心量化子系统**：`sglang/srt/layers/quantization/*`（fp8、quark、online_quantization）  
- **模型配置**：`sglang/srt/configs/model_config.py`（`REQUANTIZATION_METHODS` 常量、日志字段）  
- **权重加载与调度**：`sglang/srt/model_loader/*`（weight_utils、model_loader）  
- **模型运行时**：`sglang/srt/model_executor/model_runner.py`（日志输出改动）  
- **文档**：`docs_new/docs/advanced_features/quantization.mdx`（示例与说明）  
- **单元/集成测试**：`test/registered/quant/test_quark_mxfp4.py`（全部 FP8→MXFP4 场景用例被删除）  

**🔍 技术洞察**  

- **架构影响**  
  - 移除的代码负责 **跨格式重量化**（FP8 checkpoint → MXFP4）以及 **在线 MXFP4 量化**（在加载 BF16/FP16 checkpoint 时即时压缩为 MXFP4）。  
  - 这使得 `Quark`、`Fp8` 两大量化路径的 **依赖耦合** 彻底消失，量化子系统恢复到 “仅支持原生 MXFP4 直接加载” 的单一路径。  
  - `ModelConfig` 中的 `REQUANTIZATION_METHODS` 被删除，简化了配置解析逻辑，降低了模型配置解析时的分支复杂度。  
  - `online_quantization.CopyNumelCounter` 被删除，表明不再需要在多线程权重加载时手动追踪 copy_ 的元素数，权重加载流程回到最初的 **一次性加载 → 参数注册** 模式。  

- **性能影响**  
  - **正面**：去除 FP8‑>MXFP4 的两阶段转换（先 dequantize 到 BF16，再 requantize），显著降低了 **权重加载时的 CPU/GPU 计算和显存占用**，尤其在大模型（30B+）和 TP>1 场景下可避免原先的 “内存峰值激增” 报错。  
  - **负面**：失去 **FP8 → MXFP4** 的压缩路径后，用户只能使用 **原生 MXFP4 checkpoint** 或 **BF16/FP16 → MXFP4 在线量化**（后者也被撤回），因此在 **FP8 checkpoint** 场景下将不再受益于 MXFP4 的显存节省，可能导致 **显存需求回升**。  
  - 代码体积缩减约 **130 行**（删除）+ **1042 行**（修改），编译/解释时间略有提升。  

- **安全考虑**  
  - 该回退没有引入新的外部依赖或网络交互，安全风险几乎为 **0**。

   - 对未能加载的命中项、以及原始 miss 项，统一走 ViT 再计算后写入缓存。  
   - 在缓存读取成功后，显式调用 `mm_global_cache.release_embeddings`，避免缓存引用泄漏。

5. **日志与统计**  
   - 新增对部分回退、部分失败的警告日志，帮助定位缓存池容量问题。

### 影响范围
- **核心模块**：`python/sglang/srt/disaggregation/encode_server.py`、`encode_receiver.py`。  
- **全局缓存层**：`mm_global_cache`（批量查询、广播、释放、写入）。  
- **多节点/多卡并行**：涉及 `torch.distributed.broadcast` 与 TP（tensor‑parallel）同步。  

### 可能的副作用 / 风险
| 风险点 | 说明 | 建议 |
|--------|------|------|
| **Hash 类型统一** | 直接把所有 hash 转为 `str`，若上游代码仍持有 `int` 类型的键，可能导致缓存不命中。 | 检查 `mm_global_cache` 实现是否已全部接受 `str`，或在 API 文档中明确要求统一使用字符串。 |
| **`array("q")` 转换** | 仅在 `encode_receiver` 做了转换，若后续代码仍假设 `input_ids` 为普通 list，会触发 `TypeError`。 | 添加兼容层或在 `EncodeReceiver` 接口处标注返回类型为 `array('q')`，并在单元测试覆盖此路径。 |
| **广播的 `fallback_mask`** | 当 TP>1 时，仅在 rank‑0 计算并广播，若某些 rank 在收到 mask 前已经提前使用了 `hit_indices`，可能出现不一致。 | 确保所有 rank 在使用 `hit_indices` 前已完成 `torch.distributed.broadcast`，或在代码结构上把后续逻辑放在同步之后。 |
| **内存占用** | 回退路径会额外生成 `fallback_slices` 与 `cached_slices`，在大批次时可能临时峰值增高。 | 在关键路径加入 `torch.cuda.empty_cache()`（或显式释放不再使用的 CPU 张量），并在压力测试中验证显存上限。 |
| **异常处理** | 当 `prefetch` 超时或出错时，会把所有 `hit_indices` 标记为回退，但仍保留 `fallback_mask` 为全 1。 | 确认 `fallback_mask` 在异常路径下被正确定义，否则可能产生未广播的空张量导致 deadlock。 |

### 推荐行动
1. **单元/集成测试**：新增覆盖以下场景的测试  
   - 缓存命中且成功加载  
   - 缓存命中但因池满回退  
   - 完全 miss → 正常 ViT 前向  
   - 多 rank 环境下 `fallback_mask` 的广播一致性  

2. **文档更新**：在 `mm_global_cache` 接口说明里明确键必须为字符串；在 `EncodeReceiver` 接口说明返回的 `input_ids` 为 `array('q')` 类型。

3. **性能基准**：在多卡 TP 场景下对比之前和现在的吞吐、延迟，特别关注回退比例提升对整体 QPS 的影响。

4. **容错监控**：在生产监控中加入 `fallback_mask.sum()` 统计，及时发现缓存池容量异常导致的回退激增。

通过上述检查与改进，可确保该 bug 修复在不引入新隐患的前提下提升全局缓存的鲁棒性和多卡协同的正确性。

**变更概览**

4. 大量新增 **单元/集成测试**，验证：

   - 同一 worker 在多次 Reconcile 中保持单飞（`pending` 防止并发）。

- 仅在测试或自行调用 `run_with_introspector` 时需注意默认间隔较长（30 s），如需加速请使用 `run_with_introspector_and_reconcile` 并传入自定义 `Duration`。

| 风险 | 说明 | 建议 |
|------|------|------|
| 定时任务误触 | `RECONCILE_INTERVAL` 设为 30 s，若 `WorkerIntrospector` 重试超时也在同量级，可能导致短时间内多次重扫。 | 根据实际部署规模评估间隔；如有大量 worker 可适当调大，或在 `WorkerIntrospector` 中降低重试次数。 |

| 测试依赖时间敏感 | 多个 `timeout(..)` 依赖实际调度延迟，CI 环境波动可能导致偶发失效。 | 使用更宽松的 `timeout`，或在 CI 中加入 `RUST_LOG=debug` 观察延迟。 |

- 在正式组件（非 `experimental`）中同步 `reconcile_unresolved_workers` 逻辑，避免生产泄漏。  
- 监控 `reconcile` 的执行时长与 `pending` 大小，确保在高并发场景下不会出现显著堆积。  
- 考虑把 `RECONCILE_INTERVAL` 设为可配置项，以便在不同集群规模下灵活调优。  

总体而言，此次改动提升了 router 对 **“注册即不可见”** 场景的恢复能力，代码结构更清晰，新增的测试覆盖关键竞态路径，建议在 CI 通过后合并到主分支。
