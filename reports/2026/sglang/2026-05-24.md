### [AMD] Dsv4/pr1 fix run time issue (#25898)
**SHA**: `af8f669` | 🔗 [查看提交](https://github.com/sgl-project/sglang/commit/af8f66940e9b62947e60edc43fe6f7ac71b4dce4)

**🎯 变更类型**：架构变更 / 性能优化 / 重构  
**⚡ 重要程度**：🔴高  
**📋 变更摘要**：本次提交为 DeepSeek‑V4（dsv4）在 AMD/ROCm 平台上的完整适配与大量融合算子重构。关键工作包括：
- 将所有 `kDLCUDA` 设备标记统一替换为 `kDLGPU`，实现统一的 CUDA/ROCM 抽象；
- 在 JIT kernel 中加入 `USE_ROCM` 条件编译，使用 `__shfl_xor`、`__shfl_up` 等 ROCm 原语，兼容 AMD GPU；
- 为 FP8/E4M3 在 ROCm 上实现软件浮点到 FP8 的转换逻辑；
- 新增 Triton 实现的 `hash_topk`、`fused_store_cache` 以及相关 fallback，解决 CUDA‑Only 代码在 ROCm 上的运行时错误；
- 大幅融合 Q/K 计算链路：  
  - **Q**：RMSNorm（无权重）+ RoPE（BF16）+ 可选结果写回；  
  - **K**：带权重 RMSNorm + RoPE + FP8‑NoPE 量化 + 线程块级分页写回（FlashMLA 场景）；  
  - **Indexer**：RoPE + Hadamard 变换 + FP8 量化。  
- 新增环境变量 `SGLANG_OPT_USE_FUSED_QK_NORM_ROPE`、`SGLANG_OPT_USE_FUSED_COMPRESS`、`SGLANG_OPT_USE_FUSED_CLAMP_ACT_MUL`，让用户可在运行时开启/关闭新融合路径；
- 代码层面补齐了 `torch.Tensor` 零维度保护、AIter 在 ROCm 上的 RMSNorm‑FP8 融合、`is_hip_runtime` 判路分支，提升鲁棒性。

## 🔍 技术洞察

| 维度 | 影响 | 说明 |
|------|------|------|
| **架构** | **统一 GPU 抽象** (`kDLGPU`) | 使得上层 Python 代码不再硬编码 `kDLCUDA`，简化以后添加其他加速器（如 Intel、Vulkan）时的改动。 |
| | **ROCM 条件编译** | 通过 `#ifndef USE_ROCM` 将 CUDA 原语映射到对应的 HIP，实现单文件双平台编译。|
| | **Fusion Kernel 设计** | 将 Q‑norm、K‑norm、RoPE、FP8 量化等多步操作压缩至 1‑2 个 kernel，显著降低内核调度开销和全局内存往返。|
| | **Page‑size 参数化** | `kFusedK` kernel 根据 `page_size`（2^n）自动选取模板特化，避免运行时分支，提高指令缓存命中率。|
| **性能** | **启动次数下降** | 原先 Q/K/Cache 三段 kernel → 现在最多两段（Q‑norm+RoPE、K‑norm+RoPE+FP8），理论上可降低 30%‑50% 的启动延迟。|
| | **内存带宽利用率提升** | 归约、RoPE、量化均在寄存器/共享内存完成，只有最终的 FP8/BF16 结果写回，减少了 2‑3 次全局读写。|
| | **Triton fallback** | 对 `hash_topk` 与 `store_cache` 使用 Triton 实现，避免在 ROCm 上因 CUDA‑Only 原语导致的崩溃，保持跨平台功能完整。|
| | **FP8 量化实现差异** | CUDA 通过硬件 `__float2fp8`，ROCM 必须走软件路径，额外的位运算会带来约 5%‑10% 的额外开销，但仍显著快于两段 float→bf16→fp8 方案。|
| **安全** | **无新增外部依赖** | 仅使用标准 CUDA/HIP、ATen、Triton，未引入网络或文件 I/O，安全面影响微乎其微。|
| | **输入校验** | 增加了对空张量、维度不匹配、页大小非 2 的显式断言，防止非法内存访问导致的 GPU Crash（如原先的 `invalid configuration argument`）。|
| **可维护性** | **代码分层明确** | C++ kernel 与 Python 调用层分离，`sgl_kernel_ops.h` 暴露统一 API，后续可在同一文件中添加新平台实现。 |
| | **宏/常量统一** | `kFullMask`、`kDLGPU`、`USE_ROCM` 统一定义，降低了散落的硬编码风险。|
| | **环境开关** | 通过 `EnvBool` 控制融合路径，可在实验阶段快速回滚，降低因新融合带来的回归风险。|
| **兼容性** | **向下兼容** | 当 `USE_ROCM` 未定义或 GPU 为 CUDA，原有代码路径保持不变；所有新增 kernel 均提供 `#ifdef USE_ROCM` 包裹的备选实现。|

## ⚠️ 潜在风险

1. **FP8 软件实现误差**  
   - ROCm 上的 E4M3/FNUZ 两种 FP8 类型的位宽细节差异可能导致量化误差与 CUDA 略有差别，尤其在极端值（NaN、Inf）处理上。需在跨平台精度基准测试中确认误差在可接受范围 (<1e‑3 相对误差)。  

2. **Warp‑级同步差异**  
   - `__shfl_xor_sync` 与 `__shfl_xor` 的行为在波长（wavefront）为 64 的 AMD GPU 上仍可工作，但 mask 长度必须匹配 32‑lane warp。若在 `kDLGPU` 上运行在 64‑lane波宽的 GPU（如 MI6000 系列），当前 `kFullMask` 仍为 32‑bit 可能出现未定义行为。需在 `device::kFullMask` 按设备属性动态选择 32/64 位。  

3. **共享内存大小限制**  
   - `deepseek_v4_topk_transform.cu` 通过 `cudaFuncSetAttribute` 动态分配 48 KB SMEM；在某些 AMD GPU（如 CDNA）可能不支持如此大共享内存，导致 kernel launch 失败。建议在运行时检测 `cudaOccupancyAvailableDynamicSMemPerBlock`（已提供 ROCm stub）并降级为分块实现。  

4. **环境变量默认开启风险**  
   - `SGLANG_OPT_USE_FUSED_QK_NORM_ROPE` 默认 `True`，在尚未充分验证的旧型号 GPU 上可能导致性能倒退或错误。建议在发行版中将默认设为 `False`，或在首次检测到 ROCm 时自动跑一次轻量基准再决定。  

5. **Triton 与 HIP 兼容性**  
   - Triton 官方仅在 CUDA 上完全支持，ROCM 支持仍在实验阶段。若用户使用 ROCm 但未安装适配的

### [observability] add ServerArgs.stat_loggers for pluggable metrics backend (#24610)
**SHA**: `9d50cd9` | 🔗 [查看提交](https://github.com/sgl-project/sglang/commit/9d50cd9742f8501fce4bc624792d63df86e10eec)

**🎯 变更类型**：功能增强（为所有 MetricsCollector 引入可插拔的后端）  
**⚡ 重要程度**：🟡 中  

**📋 变更摘要**  
- 在 `ServerArgs` 中新增 `stat_loggers: Optional[Dict[str, type]]`，允许调用方在创建引擎时提交自定义的 MetricsCollector 子类。  
- 新增 `resolve_collector_class` 与 `_StatLoggerDIMixin`，实现基于 `stat_loggers` 的依赖注入（DI），并在五个 Collector（scheduler、tokenizer、storage、radix_cache、expert_dispatch）构造点统一调用。  
- 为每个 Collector 添加类属性 `_counter_cls / _gauge_cls / _histogram_cls / _summary_cls`，默认 `None`，在实例化时回退到原生 `prometheus_client`。  
- 相应模块（expert_distribution、tokenizer_manager、radix_cache、storage）改为通过 `resolve_collector_class` 取得实际类。  
- 单元测试覆盖：DI 属性默认值、`resolve_collector_class` 行为、子类替换实际 Prometheus 实例以及跨进程（Scheduler）注入验证。  

**🎯 影响范围**  
- `sglang/srt/server_args.py`（新增字段、CLI 解析调整）  
- `sglang/srt/observability/metrics_collector.py`（核心 DI 实现）  
- 多个缓存、调度与 tokenizer 代码路径的 Collector 实例化点  
- 新增/修改的测试文件（`test_metrics.py`、`test_stat_loggers_di.py`）  

**💡 关注建议**  

1. **兼容性**：`stat_loggers` 仅在代码内部使用，未加入 CLI。若后续需要通过命令行配置，请在 `from_cli_args` 中加入显式的过滤或文档说明，避免意外覆盖默认行为。  
2. **导入顺序**：`resolve_collector_class` 依赖 `ServerArgs`，而某些模块在 import 时已经引用 `ServerArgs`。当前实现没有循环依赖，但建议在模块顶部使用局部导入（如 `from .server_args import ServerArgs`）保持一致，防止潜在的 import‑time副作用。  
3. **全局实例**：`hi_mamba_radix_cache.py` 与 `hiradix_cache.py` 通过 `get_global_server_args()` 获取全局对象。确认在多进程环境下该全局对象在子进程中已被正确复制（已有实现），否则自定义 Collector 仍会落回默认。  
4. **线程安全**：DI 通过类属性修改（`_counter_cls = ...`）是“类级别”全局，若在同一进程中同时运行两个不同的 Engine 并注入不同后端，可能出现冲突。建议在文档中提醒：每个进程只能使用单一后端，或在自定义子类内部自行隔离。  
5. **测试清理**：`TestStatLoggersDI` 使用文件 `/tmp/sglang_di_test_marker` 作为跨进程信号，已在 `tearDown` 中删除，但如果测试异常退出可能残留文件。可以改用 `tempfile.NamedTemporaryFile(delete=True)` 以获得更安全的临时路径。  
6. **文档**：为 `ServerArgs.stat_loggers`、`STAT_LOGGER_ROLE_*` 常量以及 `resolve_collector_class` 添加简明的 docstring，帮助嵌入式使用者（如 Ray Serve）快速定位。  

总体来看，改动实现了灵活的后端替换，代码组织清晰，单元测试覆盖充分。只需留意上述兼容性与多进程/多实例的使用边界，即可平稳发布。

**⚡ 重要程度**：🟡 中  

**📋 变更摘要**  
1. 为 `MultimodalDataItem` 增加 `has_cuda_ipc_proxy` 与可指定目标设备的 `reconstruct`，在调度阶段只在必要时 materialize CUDA‑IPC 代理，避免无意义的拷贝。  
2. `MultimodalInputs.from_processor_output` 改为先过滤无效 `mm_item`，统一在本地完成 proxy 重建后再执行 buffer‑hash 与 pad‑value 处理，返回的新 `MultimodalInputs` 对象保持原有可选属性。  
3. `BaseMultimodalProcessor._wrap_tensor_for_cuda_ipc` 抽取 CUDA‑IPC 包装逻辑，简化 `process_and_combine_mm_data` 中的分支并提前填充 `pad_value`，去掉了原来冗长的手工拷贝代码。

**🎯 影响范围**

- `python/sglang/srt/multimodal/processors/base_processor.py`（CUDA‑IPC 包装）  
- 相关测试文件 `test/manual/vlm/test_mm_utils.py`  

**💡 关注建议**  

*开发者*  
- **兼容性检查**：`reconstruct` 现在需要显式的 `target_device` 参数，确保所有调用点（例如跨进程的调度器、TP‑rank 合并逻辑）传入正确的设备索引。  
- **属性覆盖**：`has_cuda_ipc_proxy` 只检查 `feature`、`precomputed_embeddings` 与 `model_specific_data`，如果以后在 `MultimodalDataItem` 引入新的 tensor 字段，请同步更新该检查。  
- **异常安全**：在 `from_processor_output` 中，如果 `reconstruct_device` 为 `None`（即所有字段均非 proxy），后续 `item.reconstruct` 不会被调用，这已是安全的，但建议在调试模式下加一条日志，帮助定位意外的 “proxy 未被 materialize” 场景。  
- **性能基准**：建议在 GPU‑IPC 开启且 `SGLANG_MM_BUFFER_SIZE_MB > 0` 的环境下跑一次端到端基准，对比本次改动前后内存占用与拷贝时延，确保“避免额外 staging” 的预期效果真实体现。  

*用户*  
- **环境变量**：保持 `SGL_USE_CUDA_IPC=1`（默认）以及 `SGLANG_MM_BUFFER_SIZE_MB` 合理配置即可享受改进；若在不支持 CUDA‑IPC 的机器上运行，仍会回退到普通 `cpu()` 拷贝，行为不变。  
- **调试信息**：若遇到 “CUDA‑IPC proxy 未 materialize” 报错，可通过 `torch.cuda.current_device()` 检查当前进程绑定的 GPU，确保与服务器端的 IPC 池在同一设备上。  

总体来看，此次提交通过统一的 proxy 检测与懒加载策略，显著削减了调度阶段不必要的显存拷贝，对多模态 VLM 的吞吐与显存利用都有正面影响。若后续继续扩展 `MultimodalDataItem` 的字段，请同步更新上述检测与重建路径。

总体而言，此次改动通过合并 H2D 操作、去除不必要的 CPU‑list 转换并使用非阻塞拷贝，可显著降低多模态前处理的显存带宽开销，对 Qwen‑VL 系列推理性能有实质提升，风险主要集中在新模型适配和跨设备同步上。请在多平台回归后再正式发布。
