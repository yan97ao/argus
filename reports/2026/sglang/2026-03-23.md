#### 🔴 高重要度变更 (1)

### HiSparse for Sparse Attention (#20343)
**SHA**: `13f4f01` | 🔗 [查看提交](https://github.com/sgl-project/sglang/commit/13f4f010d8ea7fbf628cd3b5313a73cac6c0285e)

**🎯 变更类型**：功能增强 / 架构变更（引入 HiSparse 稀疏注意力）  
**⚡ 重要程度**：🔴高  

**📋 变更摘要**：  
本次提交为 sglang 项目加入了 **HiSparse**（Hierarchical‑Sparse）稀疏注意力实现，主要包含：  
1. 新增 CUDA‑kernel (`hisparse.cuh`) 实现 **Top‑K 令牌在设备缓存中的高效加载、LRU 替换与稀疏拷贝**。  
2. 对 Python 层提供 JIT‑wrapper (`jit_kernel/hisparse.py`) 并在模型执行流中加入 **HiSparse 协调器** (`hisparse_coordinator.py`) 负责请求的 staging、备份、设备缓冲区分配、增/删、以及 **swap‑in** 逻辑。  
3. 重新设计内存池结构：`HiSparseNSATokenToKVPool`、`HiSparseTokenToKVPoolAllocator`，实现全局 **full→hisparse** 索引映射与双层分配（逻辑层 + hisparse 层）。  
4. 在调度器、前向执行、模型跑步器等关键路径中接入 HiSparse（如 `schedule_batch`, `scheduler`, `model_runner`, `forward_batch`, `cuda_graph_runner` 等），并提供 **decode‑stage的 staging‑to‑decode 转换**。  
5. 添加服务器参数 `--enable-hisparse` 与 `--hisparse-config`，以及相应的配置解析与校验。  

**🎯 影响范围**：  
- **核心计算**：`python/sglang/jit_kernel/csrc/hisparse.cuh`、`jit_kernel/hisparse.py`。  
- **内存管理**：`sglang/srt/mem_cache/hisparse_memory_pool.py`、`memory_pool.py`、`memory_pool_host.py`。  
- **调度与执行**：`sglang/srt/managers/*`（scheduler、schedule_batch、tp_worker、model_runner、cuda_graph_runner、runtime_checker_mixins）。  
- **配置/CLI**：`sglang/srt/server_args.py`、`srt/mem_cache/sparsity/factory.py`。  
- **注意力后端**：`srt/layers/attention/nsa_backend.py`（强制不走 fused‑topk、支持 HiSparse）

## 🔍 技术洞察

### 架构影响
| 维度 | 影响 | 说明 |
|------|------|------|
| **内存池层次** | 新增 **双层索引映射**（全局 logical → hisparse device）以及 **host↔device 备份** 机制。 | 通过 `full_to_hisparse_device_index_mapping` 实现一次映射即可在多个层之间共享同一 KV 坐标，兼容原有 `NSATokenToKVPool`，但对上层（调度器、模型跑步器）引入了 `hisparse_coordinator` 统一管理。 |
| **调度器** | 引入 **staging → decode** 两阶段流水线，支持请求在 prefill 完成后进入 HiSparse “decode‑stage”。 | 调度器在 `get_next_batch_to_run` 中优先处理已完成 staging 的请求，避免与传统 prefilling 混合，降低调度冲突。 |
| **前向路径** | `ForwardBatch` 现在携带 `hisparse_coordinator`，`ModelRunner._forward_raw` 在 decode 前填充 `hisparse_coordinator.num_real_reqs`。 | 使 CUDA‑graph 捕获期间能够捕获 HiSparse 的 `load_cache_to_device_buffer_mla` 调用，从而在图内完成稀疏加载。 |
| **上下文切换** | **LRU** 替换逻辑在 CUDA kernel 中完成，结合 `hot_buffer_size` 与 `top_k`，避免 CPU 端大量数据搬运。 | 通过 warp‑level 并行与共享内存实现 **hit / miss / evict** 分类，显著降低 top‑k 令牌查询的延迟。 |
| **配置与兼容性** | 新增 `--enable-hisparse` 开关，默认关闭；若开启必须关闭 radix cache (`--disable-radix-cache`)。 | 对现有功能保持向后兼容，未开启时代码路径不会走入 HiSparse。 |
| **分布式** | 所有 HiSparse 相关的状态（如 `req_to_device_buffer`, `req_to_host_pool`）在 `tp_group` 中保持同步（通过 `all_reduce MIN` 判断 ready 数量），兼容现有 TP/DP 组合。 | 只在 `tp_worker` 中注册 coordinator，确保跨进程共享同一对象引用。 |

### 性能影响
- **显存利用率**：引入 **device buffer**（默认 `2 * top_k`）只保留热点 token，长序列的其余 token 仍保存在 host side，显著降低显存占用（特别是 16K+ 长上下文）。  
- **数据搬迁**：`load_cache_to_device_buffer_kernel` 通过 **warp‑level gather‑scatter** 与 **非缓存全局加载 (`ld.global.nc`)** 实现一次性拷贝，避免多次 `memcpy` 与 CPU 干预。理论上可把 Top‑K 读取时间从 O(top_k * page_size) 降到 O(top_k) 并行。  
- **CPU 负载**：备份（prefill → host）仍在 CPU 端完成，但通过 `write_staging_stream` 与事件 (`start_event`, `finish_event`) 实现 **异步化**，在 decode 阶段可以并行进行。  
- **调度吞吐**：在 `scheduler.get_next_batch_to_run` 中 HiSparse 请求会被 **提前批次化**（只等 staging 完成），可提升 decode 阶段的 batch 合并度，减少单次调度的碎片化。  
- **潜在瓶颈**：  
  - **Host ↔ Device 备份**：如果 `host_to_device_ratio` 设置过低，主机内存可能频繁租用/释放，导致 PCIe 带宽成为限制。  
  - **单次 swap‑in**：内核在处理 miss 时仍是 **单 warp 复制**，大规模 miss（如 top_k 大多数不在 hot buffer）会导致线程利用率下降。可通过拆分为多 kernel 或增加 warp 数提升并行度。  

### 安全考虑
- **异常处理**：新增代码在多处使用 `assert` 或 `raise RuntimeError`，但在 CUDA kernel 内部缺少错误返回机制；若 kernel 中出现非法访问（如越界 `item_size_bytes`），会触发 GPU 异常导致进程崩溃。建议在 host wrapper 中捕获 CUDA 错误并把异常信息上报。  
- **内存泄漏**：引入的 `req_to_host_pool`、`req_to_device_buffer` 等张量在请求撤销（abort）路径中已加入释放逻辑 (`abort_staging_request`、`request_finished`)；但在 `scheduler.abort_request` 中仅对普通 kv cache 调用 `request_finished`，未处理 hiSparse → 需要确认在所有 abort 场景都走到 `retract_req` 或 `abort_staging_request`。  
- **并发安全**：`HiSparseCoordinator` 使用 CUDA events 与 stream 同步，已在 `collect_ready_reqs` 中通过 `torch.distributed.all_reduce(MIN)` 保证多卡一致。但 `self.ack_staging_queue` 仍是普通 Python list，若多个线程/进程同时访问（如 TP worker 与 scheduler 并发）可能产生竞争。建议加锁或改为 thread‑safe 队列。

## ⚠️ 潜在风险

1. **兼容性回归**  
   - `nsa_backend` 在 `force_unfused_topk` 中强制走非 fused‑topk，可能影响已有模型的性能基准。  
   - 开启 HiSparse 仍然需要 `--disable-radix-cache`，这会影响已经依赖 radix cache 的用户。  

2. **资源配置不合理**  
   - `device_buffer_size` 必须 ≥ `top_k`，若用户自行配置导致冲突，会在 `HiSparseCoordinator` 初始化时报错，但错误信息仅在日志中出现，建议在 CLI 检查提前捕获。  
   - `host_to_device_ratio` 过小会导致频繁页面分配失败（`alloc_extend` 可能返回 `None`），从而触发运行时异常。  

3. **负载不均**

### Fix(jit): support rmsnorm for hidden_size in {64, 128, 256} (#20661)
**SHA**: `777edb6` | 🔗 [查看提交](https://github.com/sgl-project/sglang/commit/777edb6ef7a589f4a00ebc766f4867568cae0c71)

**🎯 变更类型**：功能增强（为 RMSNorm 新增 warp‑level 实现）  
**⚡ 重要程度**：🟡 中  
**📋 变更摘要**：  
1. 在 `rmsnorm.cuh` 中实现了 `rmsnorm_warp` 以及对应的 `RMSNormWarpKernel`，在隐藏维度为 64、128、256 时采用单 warp 计算路径，以降低 CTA 启动开销。  
2. 在 Python 包装层 `norm.py` 添加了对隐藏维度的合法性检查、调度逻辑（`_rmsnorm_kernel_class`），并根据维度选择 `RMSNormWarpKernel` 或原 `RMSNormKernel`。  
3. 测试用例同步更新，覆盖了尺寸支持判断、内核分发以及不支持尺寸的异常路径。

**🎯 影响范围**  
- 核心 JIT 编译层：`python/sglang/jit_kernel/csrc/elementwise/rmsnorm.cuh`、`norm.py`。  
- 运行时调用：所有使用 `sglang.jit_kernel.rmsnorm` 的模型推理路径。  
- CI 测试：`tests/test_norm_jit.py`。

**💡 关注建议**  
1. **兼容性**：新增 warp‑kernel 依赖 `device::kWarpThreads` 与 PDL（Primary‑Dependent‑Launch）机制，确保在不支持 PDL 或 compute capability < 7.0 的 GPU 上仍回退到 CTA 实现。可以在 `RMSNormWarpKernel::run` 中加入显式的 `if (!device::supports_pdl())` 检查。  
2. **性能回归**：虽目标是提升小 hidden_size 的吞吐，但在某些 GPU（如占用率高的多用户场景）可能出现调度冲突。建议在 CI 中加入基准对比（CTA vs warp）并记录 `runtime::get_blocks_per_sm` 的实际取值。  
3. **文档与示例**：更新 README / API 文档，明确 “hidden_size ∈ {64,128,256} 将走 warp 路径” 的说明，并在示例代码中展示这些尺寸的使用场景。  
4. **错误信息**：当前异常信息仅包含 `unsupported hidden_size=`，建议补全 “Supported sizes: …” 以便用户快速定位问题。  
5. **后向兼容**：如果已有模型依赖 `hidden_size` 为 256 的 CTA 实现，确保在旧模型迁移时仍能得到相同数值精度（warp 与 CTA 在数值上应一致），可在单元测试中加入对比验证。  

总体来看，此次改动为 RMSNorm 在小 hidden_size 场景提供了更轻量的实现，提升了推理效率。若按上面建议完善兼容性检查和文档，将进一步降低上线风险。

### [Qwen3.5] Fuse split/reshape/cat ops in GDN projection with Triton kernel (#21019)
**SHA**: `5bdc07d` | 🔗 [查看提交](https://github.com/sgl-project/sglang/commit/5bdc07d974f6cf236fa765a685453ea5e587a838)

**变更概览**  
- 新增 Triton fused kernel (`gdn_fused_proj.py`) 实现 Qwen3‑Next / Qwen3.5 中 **split‑reshape‑cat** 的一次性算子，分别支持 *interleaved*（旧 checkpoint）和 *contiguous*（Qwen3.5）两种输入布局。  
- `Qwen3_5` 模型改为使用 `MergedColumnParallelLinear` 合并的 `in_proj_qkvz`、`in_proj_ba`，并在 forward 中根据硬件（CUDA、CPU‑AMX）调度 Triton kernel、CPU fallback 或旧拆分路径。  
- 为 FP8/Block‑Quant 权重加入 **packed‑checkpoint** 加载包装，覆盖 `weight_loader`，保证合并权重能够正确拆分。  
- `Qwen3_Next` 删除原有内联 Triton 实现，改为统一导入 `gdn_fused_proj.py` 的函数。  
- 添加 `triton`、`cpu_has_amx_support`、`is_cpu` 等实用工具的引用，调整权重映射表以适配新 fused 参数。

**影响范围**  
- `sglang/srt/models/qwen3_5.py`、`sglang/srt/models/qwen3_next.py`（模型构造、前向、权重加载）。  
- `sglang/jit_kernel/triton/gdn_fused_proj.py`（新 Triton 内核）。  
- 相关工具模块（`utils`、`parameter`）的导入与辅助函数。  

**关键风险 & 建议**  
1. **环境依赖**：Triton 需要在目标机器上编译通过；CI 必须加入 Triton 编译检查。若缺失应回退到 CPU 实现。  
2. **形状假设**：kernel 暂假设 `seq_len=1`（batch‑first），若后续支持多序列需修改 `grid` 与指针计算。  
3. **权重加载**：`_bind_packed_weight_loaders` 对 `BlockQuantScaleParameter`、`PerTensorScaleParameter` 的 split 逻辑需与 checkpoint 保存方式保持一致，建议加入单元测试验证不同 shard‑id 场景。  
4. **流与图捕获**：`_forward_input_proj` 中的双流切换与 `disable_piecewise_cuda_graph` 标志耦合，确保在 capture 模式下 `alt_stream` 已初始化。  
5. **CPU fallback**：调用 `torch.ops.sgl_kernel.fused_qkvzba_split_reshape_cat_cpu` 前需确认对应 C++ kernel 已编译进 wheel。  
6. **兼容性**：保留原 `fused_qkvzba_split_reshape_cat`（interleaved）供 Qwen3‑Next 使用，避免因 checkpoint 变更导致旧模型失效。  

**行动项**  
- 为新 Triton kernel 添加性能基准（GPU、不同 head‑group 配比）。  
- 编写 `test_gdn_fused_proj.py`，对比 kernel 输出与原 Python 拆分在随机张量上的一致性（float16/FP8）。  
- 在 CI 中加入 `torch.ops.sgl_kernel` 检查，若缺失自动跳过 CPU fallback测试。  
- 更新文档说明 Triton/AMX 必要条件及 `disable_piecewise_cuda_graph` 对性能的影响。  

总体来说，此次改动显著提升 Qwen3 系列模型在 GPU/CPU‑AMX 上的投影拆分效率，同时兼容多种 checkpoint 格式，但需要注意环境兼容与权重拆分的正确性。

**📋 变更摘要**

**🎯 影响范围**  
- **核心推理**：`sglang/srt/hardware_backend/npu/moe/topk.py` 的 gating 逻辑，可能影响所有使用 NPU MoE 的模型。  
- **测试/CI**：`sglang/test/ascend/test_ascend_utils.py`、新加入的 `test/registered/ascend/llm_models/test_ascend_minimax_m2.py` 以及 CI 注册入口 `sglang/test/ci/ci_register.py`。
