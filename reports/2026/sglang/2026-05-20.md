### [Test] Stage-a sanity kits; consolidate core/ + models_e2e/ tests (#25831)
**SHA**: `614672f` | 🔗 [查看提交](https://github.com/sgl-project/sglang/commit/614672fea5e600ed3d2fa0a0ae89fd2b67216c7c)

**🎯 变更类型**：重构 / 功能增强（测试套件模块化、覆盖面提升）  
**⚡ 重要程度**：🔵 中  
**📋 变更摘要**：  
- 新增三个轻量级 **sanity kit**：`BasicAPIContractMixin`、`BasicDecodeCorrectnessMixin`、`BasicSchedulerStressMixin`，分别覆盖 HTTP / SSE 接口协议、解码正确性、调度器并发/流式压力路径。  
- 通过 `TestBasicSanity` 将这三个 kit 与统一的服务器启动逻辑组合，形成 **stage‑a 基础 Smoke 测试**，并加入 Hellaswag 准确率底线检查。  
- 删除旧的 `server_sanity_kit.py`，并将原有手动/模型测试改为使用新的 decode‑correctness kit。若干旧的 SRT/Backend、debug、CUDA‑coredump 等测试被移除或重命名，以精简 CI。  
- 重构 `test_engine_child_pids`、`test_hidden_states` 等测试，使其在类级别共享 `Engine`，提升资源复用并避免重复启动。  

**🎯 影响范围**：  
- `python/sglang/test/kits/*`（新增/删除）  
- `test/registered/core/*`、`test/registered/models_e2e/*`、`test/manual/*`（测试类改动）  
- CI 配置（stage‑a “base‑a” 目标）  
- 任何外部脚本或文档引用 `server_sanity_kit` 的地方  

**🔍 技术洞察**：  
- **架构影响**：  
  - 测试层实现了 **Mixin‑based 组合模型**，解耦了 “服务器启动 + 测试逻辑”。  
  - 通过统一的 `CustomTestCase` 基类，所有 kit 只依赖 `self.base_url` 与 `self.process`，便于在不同启动方式（单 GPU、分布式）下复用。  
  - 删除 `server_sanity_kit.py` 消除了单一大文件的耦合，促进了代码可维护性。  

- **性能影响**：  
  - 各 kit 的测试均控制在 **≤ 2 分钟**（多数 < 1 分钟），整体 **stage‑a** 运行时间约 **1‑2 分钟**，比原来的 `server_sanity_kit`（包含 12 项）更快。  
  - 复用同一进程的 `Engine`（`test_engine_child_pids`、`test_hidden_states`）显著降低启动开销，提升 CI 资源利用率。  
  - 并发请求与流式 SSE 测试仍使用多线程/流式读取，可能在低端 CI 节点出现轻微波动，需要关注超时阈值。  

- **安全考虑**：  
  - 新增的 **API contract** 检查（`/v1/chat/completions`、`/v1/completions`、`/health*`）帮助及时捕获协议层面的 5xx、返回结构不符等潜在安全风险。  
  - 其本身不修改业务代码，也不引入新的依赖，安全风险极低。  

**⚠️ 潜在风险**：  
1. **兼容性**：外部项目或内部脚本仍 `import sglang.test.kits.server_sanity_kit` 将立即失效。  
2. **测试不确定性**：  
   - 并发/长文本（≈ 8 k token）测试在资源受限（CPU / GPU / 内存）环境下可能出现 **偶发超时**。  
   - `BasicDecodeCorrectnessMixin.test_determinism_temp_zero` 对 `temp=0` 的严格相等检查在不同硬件（FP16 / BF16）上可能出现微小数值漂移。  
3. **覆盖率回退**：删除 `test_srt_backend`、`test_cuda_coredump` 等套件后，部分功能（如 `RuntimeEndpoint` 的端到端流式/非流式组合）在 CI 中不再直接验证。  

**💡 关注建议**：  
- **文档与迁移**：在项目 README / CONTRIBUTING 中补充“从 `ServerSanityMixin` 迁移到新 Kit”的说明，并提供兼容包装（如保留一个薄层 `server_sanity_kit.py` 只做导入转发）。  
- **CI 配置**：确保 `stage-a` CI 任务引用新测试路径（`test/registered/core/test_basic_sanity.py`），并将超时阈值（`DEFAULT_TIMEOUT_FOR_SERVER_LAUNCH`）适当调高，以防并发/长提示导致偶发失败。  
- **监控 flaky**：对 `test_concurrent_requests`、`test_long_prompt` 等加入 **retry** 或 **flaky** 标记，或在 CI 中使用 `--reruns` 参数。  
- **覆盖率补齐**：考虑保留一小套 `RuntimeEndpoint`（如当前 `test_srt_backend` 中的关键路径）作为 **回归监控**，防止因测试删减导致隐藏缺陷。  
- **性能基准**：在主线 CI 中记录 `stage-a` 的总执行时长，若出现显著回退（> 30 %）应检查硬件或网络波动。  

通过上述模块化测试套件的引入，项目的 **可维护性、可组合性** 大幅提升，同时对 **API 合规性** 与 **解码正确性** 提供了更细粒度的安全保障。只需注意迁移兼容与潜在的 flaky 场景，即可顺利投入生产使用。

### feat: Support flashinfer_cutedsl MoE runner with flashinfer alltoall backend (#22669)
**SHA**: `044649c` | 🔗 [查看提交](https://github.com/sgl-project/sglang/commit/044649c23abe32d4596a6b0b6c5aa4a16b4d2d76)

**🎯 变更类型**：功能增强  
**⚡ 重要程度**：🔴高  
**📋 变更摘要**：为 `flashinfer_cutedsl` MoE runner 新增对 FlashInfer All‑to‑All 后端的支持，并在调度器、量化层、模型实现及服务器参数中加入相应的适配逻辑。实现了在 DP=TP 场景下使用 CuteDSL FP4 量化的高效 MoE，新增了专门的单元测试验证 B200（Blackwell）GPU 上的端到端正确性。  

**🎯 影响范围**：  
- `python/sglang/srt/layers/moe/moe_runner/flashinfer_cutedsl.py`（MoE runner 核心）  
- `python/sglang/srt/layers/moe/token_dispatcher/flashinfer.py`（FlashInfer 调度器）  
- `python/sglang/srt/layers/quantization/modelopt_quant.py`（量化路径）  
- `python/sglang/srt/models/qwen2_moe.py`（Qwen2 MoE 模型实现）  
- `python/sglang/srt/server_args.py`（服务器启动参数 & 配置校验）  
- 新增测试 `test/registered/moe/test_flashinfer_a2a_cutedsl_v2.py`  

**🔍 技术洞察**  

- **架构影响**  
  - **后端解耦**：在 `ensure_cutedsl_wrapper` 中加入 `num_tokens` 参数，使其能够在首次前向时依据实际 token 数确定 CUDA‑graph 缓冲区大小，避免原先硬编码的 `cuda_graph_max_bs / chunked_prefill_size`。  
  - **调度器改造**：`flashinfer.py` 现在在无 token 情形下插入 **dummy token**（零张量）并使用 sentinel expert ID (`self.num_experts`) 来触发 FlashInfer 的 padding‑sanitize 逻辑，保证 all‑to‑all 路径在 `x.shape[0]==0` 时仍能安全进入。  
  - **全链路兼容**：`modelopt_quant.py` 调用 `ensure_cutedsl_wrapper(layer, dispatch_output.hidden_states.shape[0])`，显式把实际 batch 大小传递给 wrapper，防止在多 token‑prefill 场景下出现 “num_tokens exceeds max_num_tokens” 错误。  
  - **模型层面**：`qwen2_moe.py` 增加对 `flashinfer` A2A 后端的判定，零 batch 时直接走共享 expert & gate 的空路径，避免 FP4 GEMM 在空张量上崩溃；同时在 TP>1 时排除 `flashinfer` 后端的 all‑reduce（因为 FlashInfer 已完成跨卡归约）。  
  - **配置校验**：`server_args.py` 强化了对 `moe_a2a_backend=flashinfer` 的约束：必须开启 DP‑Attention、`dp_size==tp_size`，并限定 `moe_runner_backend` 必须是 `flashinfer_cutlass` 或 `flashinfer_cutedsl`。还新增 `SGLANG_FLASHINFER_NUM_MAX_DISPATCH_TOKENS_PER_RANK` 环境变量，用于在 `--max-prefill-tokens` 超出单卡调度上限时提前报错。  

- **性能影响**  
  - **正向**：利用 FlashInfer 的高效 all‑to‑all 与 CuteDSL 的 FP4 块矩阵乘加，可在 Blackwell（sm_100a）上实现更低的延迟和更高的吞吐，尤其在 DP=TP=4、EP=4 场景下的预填充（prefill）阶段。  
  - **负向**：为兼容最坏情况，`ensure_cutedsl_wrapper` 现在可能分配更大的 CUDA‑graph 缓冲区（`max_num_tokens` 基于 `dispatcher.max_num_tokens * ep_size`），导致显存占用上升。  
  - **潜在**：若 `SGLANG_FLASHINFER_NUM_MAX_DISPATCH_TOKENS_PER_RANK` 配置不足，会在运行时触发 `ValueError`，这是一种 “安全阈值” 检查，能提前避免运行时崩溃，但也可能导致用户误以为配置有效而被阻断。  

- **安全考虑**  
  - 变更仅涉及内部张量流和环境变量读取，没有引入网络 I/O、文件系统或权限相关的代码，安全风险极低。  
  - 唯一需要注意的是环境变量的可变性：如果用户在多租户环境下错误设置 `SGLANG_FLASHINFER_NUM_MAX_DISPATCH_TOKENS_PER_RANK`，可能导致服务异常（内存 OOM），但这属于配置错误而非安全漏洞。  

**⚠️ 潜在风险**  

| 风险点 | 可能后果 | 缓解措施 |
|--------|----------|----------|
| **配置约束不满足**（`dp_size != tp_size`、未开启 DP‑Attention） | 启动时 AssertionError，服务不可用 | 在文档中突出此约束；在 `server_args.py` 报错信息中给出完整的推荐启动命令示例。 |
| **缓冲区过大**（显存不足） | OOM，进程被杀掉 | 引导用户根据 GPU 容量调低 `max-prefill-tokens` 或增大 `SGLANG_FLASHINFER_NUM_MAX_DISPATCH_TOKENS_PER_RANK`；提供 `--disable-cuda-graph` 备选方案。 |
| **空 batch 处理不完整**（旧代码仍假设 `x.shape[0] > 0`） | 某些自定义 MoE 可能在 zero‑token 场景下触发异常 | 已在 `flashinfer.py` 与 `qwen2_moe.py` 加入 `has_dummy_token` 标志并裁剪 dummy token，建议覆盖测试所有自定义 MoE 子类。 |
| **依赖版本不匹配**（FlashInfer / CuteDSL API 变动） | 导入错误或运行时 `AttributeError` | 在 `requirements.txt` 中锁定兼容版本；CI 中已加入 B200 环境的端到端测试。 |
| **混合后端误用**（选择 `flashinfer_cutedsl` runner 但后台仍为 `none`） | 可能进入不支持的代码路径，导致错误的量化尺度使用 | `server_args.py` 已对 `moe_a2a_backend` 与 `moe_runner_backend` 的组合做断言，确保一致性。 |

**💡 关注建议**  

1. **部署前检查**  
   - 确认 GPU 为 Blackwell（sm_100a）或以上；`torch.cuda.get_device_capability()` 必须返回 ≥ (10,0)。  
   - 必须开启 `--enable-dp-attention` 且 `--dp` 与 `--tp` 相同。  

2. **显存规划**  
   - 计算 `max_num_tokens = dispatcher.max_num_tokens * ep_size`，并与 GPU 可用显存对比。必要时通过 `SGLANG_FLASHINFER_NUM_MAX_DISPATCH_TOKENS_PER_RANK` 调整或降低 `--max-prefill-tokens`。
