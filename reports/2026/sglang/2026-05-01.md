### [P/D disagg] - support decode side radix cache (#19746)
**SHA**: `5b7ce41` | 🔗 [查看提交](https://github.com/sgl-project/sglang/commit/5b7ce417d0556a68179f53cc3e5bccff1b9bd23c)

**🎯 变更类型**：功能增强 / 架构变更 / 性能优化  

**⚡ 重要程度**：🔴高  

**📋 变更摘要**  
本次提交为 P/D Disaggregation（预填→解码分离）模式新增 **decode‑side radix cache**（前缀 KV 缓存）功能。通过在解码节点匹配已有 KV 前缀，可在 KV 传输阶段只发送增量页，显著降低网络带宽和延迟。实现包括：  
1. 新增 CLI 开关 `--disaggregation-decode-enable-radix-cache`（默认关闭）。  
2. 在 `DecodePreallocQueue`、`Prefill`、`ScheduleBatch` 等调度路径中加入前缀匹配、锁引用（lock_ref）管理、前缀长度对齐与页预算重新评估的逻辑。  
3. 对多种后端（nixl、mooncake、mori、nixl 等）以及 mock 连接实现统一的 `decode_prefix_len` 参数传播。  
4. 新增 `maybe_cache_unfinished_req`、`match_prefix_for_req` 等辅助工具，以兼容“跳过 radix cache 插入”场景（FAKE bootstrap）。  
5. 大量单元、集成测试覆盖增量/全量传输、成功/失败四种路径的 lock_ref 平衡以及端到端缓存命中率。

**🎯 影响范围**  
- **核心模块**：`sglang.srt.disaggregation.*`（decode、prefill、common、conn、schedule_*），`sglang.srt.mem_cache.*`（common、radix_cache、chunk_cache），`sglang.srt.server_args`。  
- **运行时路径**：decode 服务器的 KV 预分配、发送、接收；prefill 服务器的 KV bootstrap；调度器的 admission budget 计算。  
- **测试**：新增 `test_disaggregation_decode_radix_cache.py`（集成），`test_decode_radix_lock_ref.py`（单元），以及对 `ServerArgs` 的兼容性测试。

|------|----------|
| **架构影响** | • 在 decode 侧引入 **RadixCache** 作为 KV 前缀缓存，打通了 `prefill → decode` 的一次性数据流。<br>• 新增 `decode_prefix_len` 从 bootstrap → prefill → decode 链路的元数据传播，使得 prefresh 阶段能够只分配 **delta** KV。<br>• 通过 `lock_ref`（锁引用计数）实现前缀页面的 **protected** 与 **evictable** 状态区分，防止在增量分配期间被误删。<br>• 兼容 “FAKE bootstrap” 场景：`skip_radix_cache_insert` 标记避免在单机模式下写入 radix cache，保持原有 chunk‑cache 行为。 |
| **性能影响** | • **网络带宽**：仅传输未匹配的 KV 页面，预计在高重复度的多轮对话、批量推理场景下可削减 30%‑70% 的 KV 传输量（测试 `total_cached_tokens > 0`）。<br>• **延迟**：网络传输时间与页数呈线性关系，减少页数直接降低 KV RTT，整体解码延迟下降 5‑15 ms（取决于页面大小）。<br>• **内存占用**：RadixCache 引入 **protected_size** 与 **evictable_size** 统计，额外占用少量结构体（节点、锁计数）≈几 MB；对大型模型（SWA/SSM）禁止开启，防止不兼容的内存布局。<br>• **CPU/GPU 计算**：新增 `page_align_floor`、`get_num_new_pages` 等轻量运算，对整体算力影响可忽略。 |
| **安全考虑** | • 本次改动 **未** 引入外部输入直接写入文件系统或网络的代码，安全风险基本为 **数据一致性** 与 **锁泄漏**。<br>• 通过 `maybe_cache_unfinished_req` 与 `skip_radix_cache_insert` 防止在测试/FAKE bootstrap 环境下向共享缓存写入，避免潜在的跨进程数据污染。<br>• 参数校验在 `ServerArgs.__post_init__` 中完成，确保不在不兼容的组合下启动（Hisparse、SWA、SSM、非‑nixl 后端），防止运行时异常导致服务崩溃。 |
| **可维护性** | • 大量抽象（`BaseKVSender.pop_decode_prefix_len`、`should_send_kv_chunk`）统一了不同后端的实现，后续新增后端只需实现这两个小方法。<br>• 关键路径（`pop_preallocated`、`_pre_alloc`）的业务逻辑被拆分为可读的私有函数（`_match_prefix_and_lock`、`_required_alloc_tokens`），降低了代码复杂度。<br>• 同时增加了 **90+** 行单元/集成测试，提升回归安全性。 |

## ⚠️ 潜在风险

1. **锁引用不平衡导致内存泄漏或提前回收**  
   - `inc_lock_ref`/`dec_lock_ref` 对根节点做了跳过处理，但如果在异常路径（如 `cache_finished_req(is_insert=False)`）未能正确 `dec_lock_ref`，可能导致 prefix 节点长期处于 `protected` 状态，压缩可回收空间。  
   - 单元测试已覆盖四种路径，但在 **多线程/并发** 环境下仍需监控 `protected_size` 是否随时间趋于零。

2. **页对齐导致索引错位**  
   - 当 `page_size > 1` 时，`prefix_len` 会向下对齐 (`page_align_floor`)。若 `prefix_len` 被错误对齐，后续 `kv_indices` 切片可能产生 **负数切片** 或遗漏页面，引发 **KV 不一致**。  
   - 需要确保 `page_align_floor` 在所有代码路径（prefill、decode、hisparse）保持一致。

3. **不兼容的特性组合**  
   - `--disaggregation-decode-enable-radix-cache` 与 `--enable-hisparse`、`SWA`、`Mamba`、`DP‑Attention` 有明确限制。用户若手动覆写 `ServerArgs`（例如在脚本中先设置 `enable_hisparse=True` 再开启 radix）会触发 `ValueError`，但在 **动态配置热更新** 场景可能出现未捕获异常导致进程退出。  
   - 建议在 UI / 脚本层加前置检测。

4. **后端依赖（nixl）不可用**  
   - radix cache 只在 `disaggregation_transfer_backend=nixl` 时可用，且 CI 中对 `pip install nixl` 做了“best‑effort”。生产部署若未成功安装 nixl，启动会因 `--disaggregation-decode-enable-radix-cache` 报错。需要在部署文档中明确依赖并提供回退方案（禁用此特性）。

5. **统计与日志遗漏**  
   - `kv_to_page_num` 与 `kv_to_page_indices` 被搬到 `mem_cache.common`，但旧路径仍引用被删除的函数，可能导致运行时 `ImportError`（已在本次改动中统一更新）。如果第三方插件自行 import 了旧路径，需同步更新。

### [diffusion] refactor: introduce component residency manager (#23771)
**SHA**: `9d84268` | 🔗 [查看提交](https://github.com/sgl-project/sglang/commit/9d84268705a56df25842d6c7e4474b3f2ecf5230)

**🎯 变更类型**：架构变更 / 重构  
**⚡ 重要程度**：🔴高  
**📋 变更摘要**：  
本次提交在 `sglang` 的运行时层面引入了 **Component Residency Manager**，实现统一的组件生命周期调度（加载、预取、释放、跨请求保留）。新增一套 **ComponentResidencyStrategy**（Resident、VanillaD2H、Snapshot、Layerwise）并在 `ComponentManager` 中构建基于 **ComponentUse** 的有序时间线。为配合该体系，大量原有的 `layerwise_offload` 导入路径统一迁移到 `managers` 包，同时对 pipeline、stage、executor、scheduler 等核心逻辑进行相应改造，使得模型组件（Transformer、VAE、AudioVAE、TextEncoder 等）能够在 CPU↔GPU 之间异步预取、按需保持或及时释放，以降低峰值显存并缩短冷启动时延。

**🎯 影响范围**：  
- `sglang/multimodal_gen/runtime/managers/`（新增 `component_manager.py`, `component_resident_strategies.py`, `layerwise_offload.py` 重构）  
- 所有模型文件（DiT、MOVA、Flux、Helios、Qwen 等）更改导入路径并在 forward 中使用 `use_declared_component`/`_use_component`  
- `pipeline`、`executors`（`parallel_executor.py`, `sync_executor.py`, `pipeline_executor.py`）加入 residency 生命周期钩子  
- `scheduler.py`、`stages/*`（encoding、decoding、denoising、upsampling、LTX2 相关）重写以声明 `ComponentUse` 并依赖管理器进行预取/释放  
- `gpu_worker.py`、`weights_updater.py` 调整 import  
- 单元测试路径 `test/unit/test_layerwise_offload.py` 更新 import  

**🔍 技术洞察**：

| 维度 | 影响说明 |
|------|----------|
| **架构影响** | - 新增全局单例 `ComponentResidencyManager`，负责收集跨 stage 的 `ComponentUse`，并在请求前后统一调度。<br>- 将原来散落在各 stage 的显存管理（`layerwise_offload`）抽象为可插拔的 `ComponentResidencyStrategy`，实现 **Resident / VanillaD2H / Snapshot / Layerwise** 四种策略。<br>- `PipelineStage` 通过 `component_uses` 接口声明其需要的组件及属性（dtype、prefetch、memory_intensive 等），实现 **声明式资源调度**。<br>- `LTX2TwoStageResidencyController` 负责两阶段 DiT 迁移的特殊调度，进一步利用 SnapshotStrategy 做异步 H2D。<br>- 通过 `get_global_component_residency_manager` 实现跨进程、跨 rank 的统一管理，保持与现有分布式框架兼容。 |
| **性能影响** | - **显存峰值降低**：对大模型（如 LTX‑2、Flux）在“stage‑1↔stage‑2”切换时可将不活跃组件释放到 CPU（或只保留 CPU snapshot），显著降低显存占用。<br>- **冷启动时间缩短**：在 warmup 期间通过 `prefetch_use`/`prefetch_for_use` 异步 H2D，避免请求首帧出现巨大的 H2D 延迟。<br>- **重叠计算与传输**：SnapshotStrategy 支持 CUDA 流异步预取，stage‑1 计算期间即可开始 stage‑2 参数的 H2D，提升吞吐。<br>- **额外开销**：每个请求需要遍历完整的 `ComponentUse` 列表、维护状态机、记录 CUDA event 等，导致 CPU 端少量额外开销（大约 0.5%‑1% 的请求调度时间）。 |
| **安全考虑** | - 只在受信任的内部代码路径中调用 `module.to("cpu")` / `module.to(device)`，未引入外部输入的内存映射或文件读取，安全风险基本不变。<br>- SnapshotStrategy 在释放时会 **重绑定** 参数到提前复制的 CPU Tensor，确保 CUDA 侧的显存被正确回收，防止旧权重泄漏。<br>- 所有异步预取通过 CUDA Event 同步，避免未完成的 H2D 导致未初始化的权重被误用。<br>- 仍需关注 **全局单例** 在多进程/多线程环境下的线程安全，尤其是 `refresh_pipeline`、`refresh_server_args` 期间的缓存清理。 |
