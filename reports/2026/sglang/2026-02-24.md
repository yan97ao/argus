### fix: fix the bug blocking completion template application (#17010)
**SHA**: `1a83b2c` | 🔗 [查看提交](https://github.com/sgl-project/sglang/commit/1a83b2c15d6a890754e663063b675b7e99270929)

**🎯 变更类型**：Bug 修复  
**⚡ 重要程度**：🔴 高  
**📋 变更摘要**：  
1. 修复了在使用 `--completion-template` 参数时，模板名称未被正确写入全局状态导致 FIM（Fill‑in‑the‑Middle）功能失效的问题。  
2. 新增 `set_completion_template` 接口并在 `TemplateManager.load_completion_template` 中调用，以确保模板名称在首次加载时写入 `completion_template_name`。  
3. 增加了 `FimPosition` 的 `Enum` 实现以及类型注解，提升代码可读性与安全性。  
4. 补充了针对 Ascend 后端的功能性测试，验证模板加载与 FIM 完成功能在不同调用方式（内置模板、JSON 文件）下均能正常工作。

**🎯 影响范围**：  
- `python/sglang/srt/managers/template_manager.py`（模板加载逻辑）  
- `python/sglang/srt/parser/code_completion_parser.py`（全局模板名称管理、`FimPosition` 定义）  
- 新增测试 `test/registered/ascend/basic_function/test_ascend_fim_completion.py` 与模板文件 `deepseek_coder.json`（CI / Nightly）  

**🔍 技术洞察**  

- **架构影响**  
  - 引入 `set_completion_template` 作为统一入口，避免在多个位置直接修改全局变量 `completion_template_name`，提升模块间解耦度。  
  - `TemplateManager` 在加载模板后立即调用该入口，使得模板信息在整个服务启动期间保持一致，符合“一次写入、全局可读”的设计原则。  
  - `FimPosition` 由普通类改为 `Enum`，强化了类型安全，防止意外创建非法位置值。

- **性能影响**  
  - 仅在服务启动阶段执行一次全局变量写入，几乎不产生运行时开销。  
  - 新增加的测试只在 CI 环境执行，对生产性能无影响。  

- **安全考虑**  
  - 代码仅涉及内部状态设置，没有网络或文件写入风险。  
  - 使用 `Enum` 防止非法的 `fim_position` 值被注入，略微提升了输入校验的安全性。  

**⚠️ 潜在风险**  

1. **全局可变状态**：`completion_template_name` 仍为模块级全局变量，若在多线程/多进程环境下后续再次调用 `set_completion_template`（或直接修改全局）可能导致状态不一致。  
2. **单次写入限制**：`set_completion_template` 只在 `completion_template_name` 为 `None` 时写入，若用户在运行时尝试切换模板（例如通过 API 动态改参数）将被静默忽略，可能产生误解。  
3. **JSON 模板路径**：在 CI 中使用相对路径 `"./deepskip_coder.json"`，若在实际部署环境路径不一致，可能导致模板加载失败。  

**💡 关注建议**  

- **对开发者**  
  1. 如需支持运行时切换模板，考虑把全局变量改为线程安全的上下文对象（例如 `contextvars`）或提供显式的 “reset” 接口。  
  2. 在文档中明确说明 `--completion-template` 只能在服务启动时指定一次，后续修改无效。  
  3. 为防止路径错误，建议在 CLI 参数解析阶段对本地 JSON 文件做 existence 与 schema 校验后再调用 `set_completion_template`。  

- **对用户**  
  1. 确认在启动服务时使用的 `--completion-template` 参数指向已注册的模板名称或合法的 JSON 文件。  
  2. 在使用 Ascend 后端或其他非 CUDA 环境时，务必运行对应的 CI 测试（如本次提交的 `test_ascend_fim_completion.py`）以验证 FIM 能力是否正常。  

- **对 CI/运维**  
  1. 将新增的 Ascend FIM 测试纳入每日或每次模型发布的回归套件，确保模板加载逻辑在所有平台上保持一致。  
  2. 关注全局变量的并发访问日志，如出现 “template already set” 的警告，可进一步评估是否需要更细粒度的状态管理。

### Refactor CUDA graph input buffers with shared buffer pool (#19180)
**SHA**: `6e54361` | 🔗 [查看提交](https://github.com/sgl-project/sglang/commit/6e54361608cf82b7954ce73aed909a37ed13ee27)

**🎯 变更类型**：功能增强 / 重构  
**⚡ 重要程度**：🔴 高  
**📋 变更摘要**：  
- 将原有的 `GraphInputBuffers` 迁移为统一的抽象基类 `ForwardInputBuffers`，并在其之上实现多个专用子类（`DecodeInputBuffers`、`PrefillInputBuffers`、`EagleDraftInputBuffers`、`EagleDraftExtendInputBuffers`、`MultiLayerEagleDraftExtendInputBuffers`）。  
- 引入 **共享缓冲区池**（全局字典 `_forward_input_buffer_pool`）以及 `share_buffers` / `_share_one_buffer` 实现，确保同名张量在不同 runner 实例之间复用同一块显存/内存。  
- 所有 CUDA‑graph 相关的执行器（`CudaGraphRunner`、`PiecewiseCudaGraphRunner`、`EAGLEDraftCudaGraphRunner`、`EAGLEDraftExtendCudaGraphRunner`、`MultiLayerEagleDraftExtendCudaGraphRunner`）改用新的缓冲区子类并在创建后调用 `share_buffers()`。  
- 相关导入、类型标注、实例属性名同步更新，去除了旧的 `GraphInputBuffers` 实现。

- **推理/采样模块**：`sglang/srt/speculative/*`（EAGLE 草稿、扩展、Multi‑layer 版本）

- **测试/脚本**：所有依赖旧 `GraphInputBuffers` 的单元/集成测试将受到影响，需要相应更新。

## 🔍 技术洞察

| 维度 | 影响分析 |
|------|----------|
| **架构影响** | - **统一抽象**：`ForwardInputBuffers` 作为所有缓冲区的共同基类，消除了不同 runner 之间的实现分散，提升代码可维护性与可扩展性。<br>- **共享池**：通过全局 `_forward_input_buffer_pool`，同名张量在不同对象之间复用内存，降低显存占用，尤其在多 GPU/多进程部署时能显著减少显存碎片。<br>- **向后兼容**：旧的 `GraphInputBuffers` 被彻底移除，所有调用已迁移。若外部插件仍引用旧类，需要同步更新。 |
| **性能影响** | - **显存占用下降**：共享同一块张量（尤其是大尺寸的 `input_embeds`、`next_token_logits_buffer`、`hidden_states`）可节省数 GB 显存，提升大模型/大 batch 场景的可运行 batch size。<br>- **内存重用开销**：`share_buffers()` 在创建后立即执行一次 `as_strided` 包装，几乎无运行时成本。<br>- **潜在缓存命中提升**：同一物理缓冲区在不同 CUDA‑graph 捕获阶段持续驻留，减少内存分配/回收带来的碎片与驱动开销。<br>- **线程安全**：共享池是 **全局** 的，当前实现未加锁。若在多线程/多进程（fork‑spawn）环境下并行创建 buffers，可能出现竞争写入导致 **dtype/device** 不一致的断言。建议在进程层面保证单实例创建或在 `share_buffers` 前加轻量锁。 |
| **安全考虑** | - **类型/设备一致性断言**：`_share_one_buffer` 会在冲突时抛出 `assert`，防止错误的 dtype/设备混用。<br>- **未捕获异常**：断言在生产环境可能导致进程崩溃。更稳妥的做法是抛出自定义异常并记录错误日志。<br>- **内存共享风险**：共享张量若被意外 **in‑place** 修改（例如 `.zero_()`、`.fill_()`），所有使用该张量的 runner 实例都会受到影响。代码已在 `populate_from_forward_batch` 中针对 `bs != raw_bs` 的情况 **重新填充**，但仍需保证不会在不需共享的场景中误写。 |
| **可维护性** | - **代码量下降**：原 `GraphInputBuffers.create` 与 `populate_from_forward_batch` 被抽取到基类子类中，避免重复实现。<br>- **类型明确**：每个子类声明了自身需要的张量字段，使得 IDE 自动补全更友好。<br>- **文档/注释**：新文件缺少详细注释，建议在 `share_buffers`、`_share_one_buffer` 添加使用示例与注意事项。 |
| **兼容性** | - **API 变更**：外部直接引用 `GraphInputBuffers` 的代码将报错，需要改为对应子类或基类。<br>- **序列化/持久化**：若模型检查点或 RPC 中序列化了 `GraphInputBuffers` 实例，需要同步迁移序列化逻辑。 |

## ⚠️ 潜在风险

1. **线程/进程竞争**  
   - 共享池在多线程创建 `ForwardInputBuffers` 时没有同步机制，可能导致断言失效或产生不一致的张量。  
   - ** mitigation**：在多进程/多线程环境下确保 `share_buffers()` 只在单线程/单进程中调用，或在实现中加入 `threading.Lock` 包装。

2. **意外原地修改**  
   - 代码中多处使用 `.fill_()`、`.zero_()`，这些操作会影响所有共享实例。若后续加入 **调试/日志** 阶段并在不应共享的 runner 中调用，可能导致数据泄露或错误计算。  
   - ** mitigation**：在 `share_buffers` 时提供 **copy‑on‑write** 标记（如 `clone_if_shared`）或在文档中明确说明只能在 **捕获阶段**（不再更改）安全使用。

3. **异常处理不足**  
   - 当前使用 `assert` 检查 dtype / device，生产环境可能触发 `AssertionError` 导致服务异常退出。  
   - ** mitigation**：改为抛出自定义异常并记录错误，或在 `assert` 前加入 `if not __debug__: return` 仅在调试模式下启用。

4. **兼容性破坏**  
   - 第三方插件、旧的自定义 `CudaGraphRunner` 仍然依赖 `GraphInputBuffers`。升级后可能出现 `ImportError`。  
   - ** mitigation**：提供向后兼容的 shim（如 `class GraphInputBuffers(DecodeInputBuffers): pass`）或在发行说明中明确迁移步骤。

5. **显存泄漏风险**  
   - `as_strided` 返回的视图不持有原始张量的引用，若原始张量被垃圾回收，视图会失效。全局池持有对 **最新** 张量的引用，但如果在 `share_buffers` 之后手动 `del` 旧对象，可能导致视图失效。  
   - ** mitigation**：在 `share_buffers` 完成后，确保全局池对象不被删除，或使用 `weakref` 进行安全管理。

### [diffusion] quant: support quant for all dits (#19156)
**SHA**: `2e053d6` | 🔗 [查看提交](https://github.com/sgl-project/sglang/commit/2e053d6eb64a5a8a0aea12ec868bf6864abdb318)

**🎯 变更类型**：功能增强 / 性能优化  
**⚡ 重要程度**：🔴 高  

**📋 变更摘要**  
本次提交为 *sglang* 的所有 DiT（Diffusion Transformer）系列模型加入统一的量化（Quantization）支持。核心改动包括：  
1. 为模型层（Linear、MLP、LayerNorm 等）新增 `quant_config` 参数，并在构造函数链路中向下传递。  
2. 在模型加载器中自动注入量化配置，完成权重加载后调用 `quant_method.process_weights_after_loading` 完成后处理。  
3. 增加对未匹配 checkpoint 键的警告、完善日志与错误提示。  
4. 文档、测试、转换工具同步更新，提供 FP8 量化转换脚本以及新的基准测试用例。  

**🎯 影响范围**  
- **模型实现**：`flux_2`, `wanvideo`, `glm_image`, `hunyuanvideo`, `ltx_2`, `mova_audio_dit`, `mova_video_dit`, `qwen_image`, `zimage` 等全部 DiT 系列模型。  
- **加载机制**：`transformer_loader.py`, `vae_loader.py`, `fsdp_load.py` 等组件。  
- **配置体系**：`base.py` 中模型配置更新、量化配置结构 (`QuantizationConfig`).  
- **工具链**：`convert_hf_to_fp8.py`（FP8 量化转换脚本）以及对应的 CLI 参数。  
- **测试/基准**：新增 `zimage_image_t2i_fp8` 基准，用于验证 FP8 量化模型的端到端性能。  

**🔍 技术洞察**  

- **架构影响**  
  - **统一量化入口**：通过 `QuantizationConfig` 将量化策略统一注入到层级（Linear/MLP/ColumnParallelLinear/RowParallelLinear 等），实现“全模型量化”。  
  - **后处理钩子**：在 `maybe_load_fsdp_model` 里遍历所有子模块，若其 `quant_method` 实现了 `process_weights_after_loading`，则在模型权重完整加载后执行。这使得如 SVDQuant、FP8 等需要后置处理的算法得以无侵入式集成。  
  - **加载路径简化**：去除了 `transformer_loader` 对 `_class_name` 必须存在的强制检查，拓宽了对非 Diffusers 格式模型的兼容性（但依赖 `ModelRegistry` 解析）。  
  - **配置向上传递**：所有 DiT 子类的 `__init__` 现在接受 `quant_config`（默认 `None`），并在各层创建时显式传递，确保量化配置能够覆盖每一层的权重。  

- **性能影响**  
  - **内存占用**：依据文档，SVDQuant 相比 BF16 可降低约 3.6× 内存。  
  - **推理加速**：在 RTX 4090 上对比 NF4 (W4A16) 基准提升 3.0×，对比 16-bit 基准提升 8.7×；在 RTX 5090（NVFP4）上提升 3.1×。  
  - **FP8 支持**：通过 `convert_hf_to_fp8.py` 可将权重转换为 FP8，进一步提升带宽利用率，测试基准显示每步推理时间约 85–96 ms，整体端到端约 1.37 s。

### ⚠️ 潜在风险

1. **显存与并行度冲突**  
   Whisper 大模型（如 `whisper-large-v3`) 在多卡 TP 环境下仍可能超出显存，尤其在开启高 `concurrency` 时。当前代码未对 `max_batch_size` 进行音频模型专门的限制，可能导致 OOM。

2. **KV Cache 与跨请求音频对齐**  
   在解码阶段，`_encoder_cache` 按请求索引缓存 encoder 输出。若同一请求的 `request_id` 被重复使用（例如在流式转录后再次发送同 `request_id`），缓存可能被误用，导致跨请求音频泄露或错误的上下文。

3. **语言 token 兼容性**

4. **流式结束标记**  
   SSE 流在 `yield "data: [DONE]\n\n"` 后未显式关闭连接，虽然 `StreamingResponse` 会处理，但若客户端不遵循 SSE 规范，可能出现悬挂连接。

5. **CUDA Graph 禁用的副作用**  
   对所有 encoder‑decoder 模型一刀切禁用 CUDA Graph，可能错失在特定模型（如小型 encoder‑decoder）上仍可使用图的性能提升机会。

6. **依赖冲突**  
   Benchmark 脚本引入了 `torchcodec`、`openai` 等额外依赖，若在生产镜像中未同步这些包，部署时会报错。

**结论**
