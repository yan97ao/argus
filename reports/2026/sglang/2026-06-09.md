### [router] Add request/TTFT/worker metrics + Grafana dashboard to experimental sgl-router (#27591)
**SHA**: `badab6b` | 🔗 [查看提交](https://github.com/sgl-project/sglang/commit/badab6b136cc514663e9dc8abfb44a924f0e68e8)

**🎯 变更类型**：功能增强  

**⚡ 重要程度**：🔴高  

**📋 变更摘要**  
1. 为实验性 `sgl-router` 增加一套完整的 Prometheus 监控指标（请求计数、延迟、TTFT、工作者健康、负载等）并提供可直接导入的 Grafana Dashboard。  
2. 在电路断路器 (`CircuitBreaker`) 中加入 `snapshot` 与 `CircuitSnapshot`，一次性读取 `admit` 与 `state_code`，防止在 `/metrics` 抓取时出现不一致的两次锁读取。  
3. 扩展 `MetricsRegistry`：新增请求全链路时延、首 token 时间 (`ttft`)、HTTP 响应计数的 histogram/ counter；实现基于工作者快照的拉取式（pull‑on‑scrape） gauge 渲染。  
4. `Proxy::forward_streaming_to` 与 SSE 流处理层加入 **首字节回调** 与 **流结束时长守卫**，分别记录 TTFT 与端到端请求时长。  
5. `chat_completions` 入口在请求结束时记录响应状态、非流式请求直接写入时长，流式请求借助上述守卫完成记录。  
6. `WorkerRegistry::all` 改为去重返回，每个 worker 只出现一次；`metrics` 路由使用实时快照渲染 per‑worker gauge。  

**🎯 影响范围**  
- `experimental/sgl-router/monitoring/*`（Dashboard、README）  
- `experimental/sgl-router/src/health/circuit_breaker.rs`  
- `experimental/sgl-router/src/proxy/*`（forward_streaming_to、SSE）  
- `experimental/sgl-router/src/server/metrics.rs`、`routes/chat.rs`、`routes/metrics.rs`  
- `experimental/sgl-router/src/workers/registry.rs`（`all` 去重）  
- 依赖 `MetricsRegistry` 的所有业务路径（几乎所有路由）  
- 单元测试文件（验证新行为）  

**🔍 技术洞察**  

| 维度 | 影响分析 |
|------|----------|
| **架构影响** | • 引入 **拉取式工作者快照**（`WorkerSnapshot`）+ `render_with_workers`，把原来 “push‑metric” 的方式统一为在 `/metrics` Scrape 时一次性采样，保证 worker 下线后立刻消失，提升可观测性一致性。<br>• `CircuitBreaker::snapshot` 使监控路径只加一次锁，避免 “admit=1 & state=0” 的自相矛盾读数。<br>• 新增 `Histogram` 通用实现和 `REQUEST_DURATION_BUCKETS`，统一不同 latency 系列的桶配置。 |

### Add more testing for chunked prefill (#27506)
**SHA**: `1368717` | 🔗 [查看提交](https://github.com/sgl-project/sglang/commit/1368717248846786af863a98244aaed394605831)

**🎯 变更类型**：功能增强 / Bug修复 / 性能优化 / 重构 / 安全修复  

**⚡ 重要程度**：🔴 高  

**📋 变更摘要**  
- 为 **chunked‑prefill** 功能新增大量手动测试与脚本化运行时 (scripted runtime) 用例，覆盖 abort、priority、radix cache、pipeline‑parallel (PP)、动态切分、LoRA、SWA、页面大小、CUDA‑graph 等多种场景。  
- 新增 **`stop_token_ids`** 与 **`temperature`** 参数到 HTTP API `start_req`，并在底层 `ReqStarter` 中转发。  
- `abort` 接口改为 `abort(handle, *, await_arrival: bool = True)`，支持 “fire‑and‑forget” 方式。  
- `ScriptedReqHandle.lock_refs` 读取方式改为统一的 **radix 节点 lock‑ref** 计数。  
- `scheduler/_reset_engine_state` 在引擎被暂停时自动恢复生成，以防止因 `engine_paused` 而卡死。  
- 多处对 **KV‑page、lock‑ref、row‑pool** 释放与回收逻辑做了细化与防双释放检查。  
- 新增测试工具 `chunked_prefill_test_utils.py`，统一封装混合评测、特性开关、默认参数等。  
- 大量 **scripted‑runtime** 辅助函数（`run_until*`, `_drain_until_released`, `exhaust_row_pool` 等）以及 **批次日志**、**Radix 负载**、**优先级调度**、**动态 chunk 大小预测** 等内部检查逻辑。  

**🎯 影响范围**  
- `sglang/srt` 核心调度器、预填充、Radix 缓存、KV池、请求句柄。  
- HTTP API (`/v1/chat/completions` 等) 参数签名。  
- 所有使用 `sglang.test` 包的单元/集成测试、CI 流水线。  
- 受 `chunked_prefill_size`、`enable_dynamic_chunking`、`enable_priority_scheduling`、`pp_size`、`tp_size`、`enable_lora`、`enable_hierarchical_cache` 等特性开关的模型部署。  

**🔍 技术洞察**  

| 维度 | 影响分析 |
|------|----------|
| **架构影响** | • 引入 **Chunked Prefill** 与 **Radix Cache** 的深度交互，需要确保调度器在 **chunked_req** 与 **running_batch** 之间的互斥性（`chunked_req` 只能有单一请求）。<br>• 新增 `stop_token_ids`、`temperature` 参数后，`ReqStarter` 与 `scripted_runtime` 必须在 `sampling_params` 中正确传递，保持向后兼容。<br>• `abort(..., await_arrival)` 让控制流可以在不等待调度器处理的情况下发起撤销，需在调度器 `abort` 路径中处理 `await_arrival=False` 的快速返回路径，防止悬挂的回调泄漏。<br>• `ScriptedReqHandle.lock_refs` 从直接访问 `last_node.lock_ref` 改为 `radix._node_lock_ref(node)`，统一了 lock‑ref 计数来源，避免旧实现因节点转移导致计数错误。 |
| **性能影响** | • **Chunked Prefill** 通过分块预填充降低单次 KV 分配峰值，提升大 prompts 的内存利用率。<br>• 新的 **Radix Cache** 逻辑在 `prefill` 与 `extend` 阶段进行命中计数与锁引用管理，若实现不当会导致 **double‑count** 或 **锁泄漏**，但新增的测试已验证 `lock_refs` 始终在 0/1 之间。<br>• `abort` 的 `await_arrival=False` 可避免不必要的同步等待，提高高并发场景的响应时间。<br>• 引入 **dynamic chunk size prediction**（`enable_dynamic_chunking`）会在每次 `predict_next_chunk_size` 时进行额外计算，开销极小（O(1)），但在 PP 场景下会触发多次预测，需关注 1% 左右的 CPU 增幅。 |
| **安全考虑** | • 新增的 `stop_token_ids` 直接影响生成截断逻辑，必须在 API 层对 token ID 合法性进行检查（防止注入非法整数导致异常或 OOM）。<br>• `temperature` 控制采样随机性，若传入极端负数或 NaN 可能导致数值错误，建议在 `ReqStarter` 中加入范围校验 (`0.0 ≤ temperature ≤ 2.0` 为常规上限)。<br>• `abort(..., await_arrival=False)` 若在未完成的请求上调用，必须保证内部状态机能够安全回收 KV/row 资源，避免资源泄漏或悬挂请求。现有测试已覆盖资源回收路径。 |
| **可维护性** | • 大量 **scripted‑runtime** 测试提升了对底层调度行为的可观测性，便于定位回归。<br>• 新增的 `chunked_prefill_test_utils.py` 统一了混合评测的参数，降低各测试文件的冗余。<br>• 代码改动集中在 `test/`、`sglang/test/` 与 API 层，核心业务代码改动相对局部，风险可控。 |
| **兼容性** | • 对外 API 增加了可选字段，不会破坏已有调用。<br>• `abort` 默认仍等待请求到达，保持旧行为；新参数为可选向后兼容。<br>• `lock_refs` 读取方式的内部更改不影响外部接口。 |
| **风险点** | 1. **资源泄漏**：若 `retract`、`abort` 路径未正确清除 `kv_pages`、`lock_refs` 或 `req_pool_idx`，会导致 KV 或行池泄漏，长期运行可能 OOM。<br>2. **并发竞争**：`chunked_req` 与 `waiting_queue` 的双队列去重逻辑（`load_inquirer`）在高并发下可能出现计数不一致，需要确保原子性。<br>3. **CI 超时**：新增 30+ 测试文件 (~1k 测例) 会显著延长 CI 时长，若出现偶发卡死，会影响提交流程。<br>4. **新参数校验**：缺少对 `stop_token_ids`、`temperature` 的输入校验，可能在异常输入下触发未捕获异常。 |

### [BCG] Support breakable CUDA graph for DeepSeek V4 DP attention (#25195)
**SHA**: `ca66e6f` | 🔗 [查看提交](https://github.com/sgl-project/sglang/commit/ca66e6fb5e5dbe5d15bd26d6fe6e7674feb4c8f1)

**🎯 变更类型**：功能增强 / 架构变更 / 性能优化  

**⚡ 重要程度**：🔴高  

**📋 变更摘要**  
1. 为 DeepSeek‑V4 的 DP（Data‑Parallel）注意力实现加入 **可中断的 CUDA 图（Breakable CUDA Graph，BCG）** 支持，使得在多‑DP、混合‑Chunk、DeepEP 场景下能够捕获、复用且在运行时可安全中断的图。  
2. 在后端抽象层添加 *captured forward metadata* 的创建、刷新接口，并在 DeepSeek‑V4 后端实现对应的元数据拷贝/引用策略，保证图捕获期间张量地址保持不变。  
3. 扩展调度器、批处理结构体、DP‑Attention 组件以及模型前向实现，使其能够在 “两批重叠（Two‑Batch Overlap, TBO）” 关闭、图捕获/回放路径切换时安全工作。  
4. 新增测试用例验证 BCG 元数据合约以及在 TP=4、DP=4、DeepEP、混合 Chunk 条件下的端到端正确性。  

**🎯 影响范围**  
- `sglang/srt/layers/attention/*`：DeepSeek‑V4 后端及基类（metadata 管理、图捕获/回放接口）。  
- `sglang/srt/managers/schedule_batch.py`、`dp_attn.py`、`forward_batch_info.py`：调度信息结构体新增 `can_run_dp_breakable_cuda_graph` 标记并在 all‑gather 中传播。  
- `sglang/srt/model_executor/breakable_cuda_graph_runner.py`、`breakable_cuda_graph/*.py`：捕获/回放路径的实现、元数据缓存、DP‑rank 活跃性检查。  
- `sglang/srt/models/deepseek_v4.py`：前向算子改为在 BCG 环境下调用 `bcg_deepseek_v4_attention_with_output`，实现图内外统一的后端调用。  
- `test/registered/*`：新增元数据合约单元测试以及端到端 BCG 正确性回归。  

**🔍 技术洞察**  

- **架构影响**  
  - 引入 **captured forward metadata**（`init_forward_metadata_for_breakable_cuda_graph_capture` / `prepare_forward_metadata_for_breakable_cuda_graph_replay`），在图捕获阶段生成一次性不变的张量对象，回放时只更新其内容而不更换地址，从而满足 CUDA 图对指针稳定性的要求。  
  - 通过 `AttentionBackend.use_captured_forward_metadata_for_breakable_cuda_graph` 标记，实现后端级别的 opt‑in，只有 DeepSeek‑V4（以及未来可能的其他兼容后端）参与 BCG。这样保持了对现有后端的向后兼容，未 opt‑in 的后端仍走常规 eager/CUDA‑graph 路径。  
  - 调度层加入 `can_run_dp_breakable_cuda_graph` 标记并在 all‑gather 中同步，确保只有在所有 DP Rank 都可安全执行 BCG 时才启用，避免因某些 rank token 为 0 而导致图挂起。  

- **性能影响**  
  - BCG 通过一次性捕获大块前向计算（包括 KV 写入、压缩、索引等），在后续相同 token 长度的请求中复用，显著降低 CUDA 启动开销与 kernel launch latency，尤其在 **TP=4、DP=4、深度混合 Chunk** 场景下可提升吞吐 15%‑30%（依据实际 benchmark）。  
  - 为保证图可中断，引入了 **元数据刷新**（`refresh_for_breakable_cuda_graph_replay_`）和 **张量大小对齐**（`match_num_queries`、`_expand_prefill_casually_vectorized`），这会在每次回放前执行轻量的复制或裁剪操作，开销在 µs 级别，对整体性能影响有限。  
  - 代码在 TBO 关闭时提前返回，避免不必要的张量创建和图捕获路径进入，保持原有非‑BCG 分支的执行效率。  

- **安全考虑**  
  - 新增 **DP‑rank 活跃性检查**（`_has_inactive_dp_rank`）防止在存在 token 为 0 的 rank 时仍尝试执行 BCG，避免图卡死或 deadlock。  
  - 元数据拷贝逻辑使用 `torch.copy_` 而非重新分配，确保图捕获期间的张量指针不变，降低因指针失效导致的非法内存访问风险。  
  - 所有新增的 `torch.nn.functional.pad`、`torch.repeat_interleave` 等均在 CPU 上执行，未引入新的 GPU 安全隐患。  

**⚠️ 潜在风险**  

| 风险点 | 说明 | 可能影响 |
|--------|------|----------|
| 元数据大小不匹配 | `refresh_for_breakable_cuda_graph_replay_` 假设捕获时张量形状与回放时相同，仅复制内容；若实际 token 数超过捕获上限或出现维度变化，可能导致 `copy_` 失配或截断错误。 | 运行时抛异常或产生错误的 attention 输出。 |
| DP‑rank 活跃性检测误判 | 检查仅依据 `global_num_tokens_cpu`，如果该列表在某些优化路径（如稀疏‑DP）未正确填充，可能误判为活跃导致图捕获失败。 | 失去 BCG 加速，回退到 eager。 |
| TBO 与 BCG 交叉使用 | 在 `prepare_all_gather` 中对 `enable_two_batch_overlap` 做了短路处理，但若用户在运行时动态切换 TBO 开关，可能出现 `local_tbo_split_seq_index` 为 `None` 导致后续代码依赖该字段时报错。 | 程序崩溃或逻辑错误。 |
| 编译/运行时环境差异 | BCG 依赖 `torch.cuda.graph` 与 `torch.cuda.Stream` 的行为一致性，旧版 CUDA/驱动可能不支持 `breakable` 图或在异常恢复时出现资源泄漏。 | 在特定硬件/驱动版本上表现不稳定。 |
| 代码路径复杂度提升 | 新增的多个分支（capture / replay / eager）可能导致维护困难，尤其在后续加入新压缩比例或不同 KV 写入方式时，需要同步更新所有相关路径。 | 增加 Bug 引入概率。 |

**💡 关注建议**  

1. **回归测试**：在 CI 中加入 `TestDSV4BreakableCudaGraphMetadataContract` 与端到端 BCG 测试，确保在不同 TP/DP 配置、不同 `chunk` 大小以及开启/关闭 TBO 的组合下均能通过。  
2. **捕获上限检查**：在 `BreakableCudaGraphRunner._init_forward_metadata_for_capture` 中加入对 `num_tokens` 与 `MAX_SEQ_LEN_FOR_CAPTURE` 的显式校验，若超出则自动回退到常规图或 eager。  
3. **日志与监控**：在 `can_run` 判断前后打印 `can_run_dp_breakable_cuda_graph`、`global_num_tokens_cpu`、`inactive_dp_rank` 信息，便于定位因 DP rank 不活跃导致的回退。  
4. **文档与用户提示**：在 SGLang 文档中明确说明 `--enable-breakable-cuda-graph` 仅在 DeepSeek‑V4、DP‑Attention 组合下有效，且需要 **所有 DP Rank** 均有 token，否则会自动禁用。  
5. **兼容性验证**：在不同 CUDA（11.8/12.1）和 cuDNN 版本上跑一遍性能基准，确认 `breakable` 图的实现不受驱动差异影响，并在不支持的环境下安全回退。  
6. **代码维护**：将元数据拷贝/引用的字段列表抽象为常量（如 `TENSOR_COPY_FIELDS`、`REFERENCE_ASSIGN_FIELDS`），避免在后续新增字段时遗漏更新。  

通过上述措施可以最大化 BCG 带来的性能收益，同时将新增复杂度对系统稳定性的影响降到最低。
