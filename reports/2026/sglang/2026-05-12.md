### Revert "Migrate Intel CPU cases to the test/registered." (#25044)
**SHA**: `7128533` | 🔗 [查看提交](https://github.com/sgl-project/sglang/commit/71285335f7cfe17e1f3b7967a4ef9cbea08d5163)

**🎯 变更类型**：功能增强 → **实际是一次回滚（Revert）**，撤销了之前把 Intel CPU 相关测试迁移到 `test/registered/` 的工作  

**⚡ 重要程度**：🔴 高  

**📋 变更摘要**  
- 将 GitHub CI 工作流 `pr-test-xeon.yml` 中的测试指令从 `stage‑b‑test‑cpu`（完整的 CPU 基准/回归套件）改为 `per‑commit‑cpu`，并放宽每个文件的超时阈值。  
- 在 `.pre‑commit-config.yaml` 中删除了 `check_registered_tests.py` 对 `registered/**/utils.py` 的排除规则。  
- 大量 **删除** `test/registered/cpu/` 下的 30+ 个单元测试文件，这些文件覆盖了 Intel CPU（`intel_amx`）后端的激活、绑定、BMM、CAUSAL_CONV1D、Moe、Top‑K、ROPE、Mamba、MLA、Flash‑Attention、GEMM、FP8、INT8、INT4、Shared‑Expert 等核心算子。  
- 文档链接、发布日期等微调（无功能影响）。  
- `test/run_suite.py` 中的 **per‑commit** CPU 套件列表从 `["stage‑a‑test‑cpu", "stage‑b‑test‑cpu"]` 仅保留 `stage‑a‑test‑cpu`，即 **不再在每次 PR 自动跑完整的 CPU 测试**。  

---

### 🔍 技术洞察

| 维度 | 影响分析 |
|------|----------|
| **架构影响** | - 这次回滚**不改变运行时代码**，仅影响 **CI/测试层面**。<br>- 删除了大量针对 `intel_amx`（CPU‑AMX）后端的注册测试，使得这些算子不再被 `check_registered_tests.py` 检查，也不在 CI 中执行。<br>- 若后续对这些算子进行改动（性能优化、Bug 修复），缺少自动化回归，可能导致隐藏的架构破坏。 |
| **性能影响** | - CI 运行时间会 **下降**（少跑大量 CPU 基准），但这仅是**开发者体验**提升。<br>- 对生产环境性能没有直接影响。 |
| **安全考虑** | - 与安全无关。<br>- 唯一间接影响：若算子出现 **数值不稳定/溢出** 的安全敏感 bug（如导致模型输出异常），缺少测试可能让此类问题更容易进入主线。 |
| **可维护性** | - **测试覆盖率大幅下降**（从数百个 CPU 核心算子测试降到仅 `stage‑a‑test‑cpu` 的少量基准）。<br>- 代码库的 **CI 质量门槛降低**，后续贡献者可能误以为改动已被充分验证。<br>- 删除 `check_registered_tests.py` 中对 `utils.py` 的 `exclude`，可能导致 **不必要的 CI 失败**（如果 `utils.py` 中的辅助函数不满足注册检查规则）。 |
| **CI/工作流** | - `pr-test-xeon.yml` 现在使用 `per‑commit‑cpu`（单文件 1500 s 超时），相当于 **只跑轻量化的 CPU 测试**，而不是完整的 `stage‑b‑test‑cpu` 基准。<br>- 这会让 PR 合并更快，但也削弱了对 CPU‑AMX 关键路径的回归检验。 |
| **文档** | - 链接、图片、日期更新，仅是版本信息的同步，对代码功能无影响。 |

### Migrate Intel CPU cases to the test/registered. (#22670)
**SHA**: `ecf5d84` | 🔗 [查看提交](https://github.com/sgl-project/sglang/commit/ecf5d844f5bfd8d243043da366fb8b58f6911ef3)

**⚡ 重要程度**：🔴 高  

**📋 变更摘要**  
- 将 **Intel‑CPU**（xeon）测试迁入统一的 `test/registered` 目录，新增 **stage‑b‑test‑cpu** 测试套件。  
- 新增 **50+** CPU 端单元测试，覆盖激活函数、绑定、BMM、卷积、图优化、解码/扩展注意力、MLP、MoE、量化、RoPE、LayerNorm、Top‑K、以及完整的端到端 server‑args 与 Intel‑AMX 后端基准。

| **CI 资源消耗** | - 新增约 **5k+** 行测试代码，单次跑完整 `stage-b-test-cpu` 预估耗时约 **30‑45 分钟**（取决于硬件），会显著增加 CPU runner 的占用时间。<br>- 若使用并行 jobs，可通过 `timeout‑minutes: 36`（已在 workflow 中配置）防止超时。建议监控 runner 的 CPU/内存使用，防止因资源竞用导致不稳定。 |

## ⚠️ 潜在风险

| 风险点 | 描述 | 缓解建议 |
|--------|------|----------|
| **CI 时长显著增长** | `stage-b-test-cpu` 包含 20+ 大型算子验证（如 MoE、FP8、Mamba、Flash‑Attn），在普通 GitHub Actions CPU 实例上可能接近 30 min。 | - 将部分耗时测试标记为 `skip`（使用 `@unittest.skipIf`）并在本地或更强大的自托管 runner 上跑完整套。<br>- 在 workflow 中使用矩阵并行或分块 `max-parallel`。 |
| **测试 flakiness** | 部分算子依赖非确定性浮点运算（例如 `torch.ops.sgl_kernel.*` 的内部实现），在不同 CPU（ARM vs x86）或不同 Torch 版本上可能出现 1e‑2 级别的数值差异。 | - 统一 CI 镜像（指定 `torch==2.5.0`、`numpy` 版本），并在测试里使用 `precision` 表映射的容差。<br>- 对已知会产生轻微差异的 kernel（如 FP8 QKV）适当放宽 `atol/rtol`（已在测试中实现）。 |

  - 该实现保持了与现有 `MHATokenToKVPool` 接口的兼容，故不需要改动上层业务逻辑，只在内部调度时替换实现。  

- **性能影响**  
  - **显著降低显存占用**：在 embedding‑mode 大批量 prefill 时，传统 KV 池会按 `page_size * layer_num * head_num * head_dim` 分配 GB 级显存；新池仅分配 KB 级占位张量，显存使用率下降数百倍。

3. **Hash 失效**：在极端拆分/合并路径中 `hash_value` 可能被置为 `None`，若后续逻辑未检查可能导致空指针异常。

**💡 关注建议**：  
- 在生产环境打开事件功能前，评估 `kv_event_queue` 的消费频率，确保每次推理循环后都调用 `take_events()` 清空。  
- 为 `kv_event_queue` 加入容量上限或自动截断策略，以防止意外泄漏。
