#### 🔴 高重要度变更 (6)

### feat: [1/2] [DeepEP] Fuse shared expert into MoE dispatch under EP (#20089)
**SHA**: `57ffc55` | 🔗 [查看提交](https://github.com/sgl-project/sglang/commit/57ffc55fb647bfc241d8c4766b846f4243b9c81d)

**🎯 变更类型**：功能增强  

**⚡ 重要程度**：🔴高  

**📋 变更摘要**  
此次提交为 DeepEP 系列 MoE 后端（DeepEP / Mooncake / Mori）引入了“共享专家融合”特性。当后端属于 DeepEP 系列且模型配置了共享专家时，shared expert 将被 **直接融合进 MoE kernel**，每个 EP（Expert Parallel） Rank 拥有独立的共享专家槽位。为此，代码在专家布局、Top‑K 计算、权重加载以及调度路径上做了大量改动，使得共享专家不再走独立的 MLP 分支，而是与路由专家交叉排列并在 Top‑K 输出阶段进行映射。

**🎯 影响范围**  
- `sglang/srt/layers/moe/fused_moe_triton/layer.py`（MoE 核心调度逻辑）  
- `sglang/srt/layers/moe/topk.py`（Top‑K 计算、GPU FlashInfer 路径）  
- `sglang/srt/layers/moe/utils.py`（后端类型检测）  
- `sglang/srt/models/deepseek_v2.py`（DeepSeek V2 MoE 层初始化与权重加载）  
- `sglang/srt/server_args.py`（新增 CLI 参数 `--enforce-shared-experts-fusion`）  
- 相关分发/捕获/分布式工具的导入调整  

**🔍 技术洞察**  

- **架构影响**  
  - **专家布局变化**：对 DeepEP 后端，`num_fused_shared_experts` 不再是全局共享计数，而是 **`num_fused_shared_experts * EP_SIZE`** 的本地槽位数量。全局专家总数 = `routed_experts + EP_SIZE`（每个 Rank 多一个本地共享专家）。  
  - **Top‑K 结果映射**：在 `biased_grouped_topk_gpu/cpu` 中加入 `_remap_topk_for_deepep`，将路由专家 ID 按 `e + e // L`（其中 `L` 为每 Rank 本地路由专家数）插入间隙，并显式为每个 Rank 生成唯一的共享专家 ID。  
  - **权重加载路径**：`_map_global_expert_id_to_local_expert_id` 逻辑改为基于 `_num_global_routed`、`_num_local_routed` 计算，确保共享专家能够映射到本地 slot。  
  - **配置入口**：新增 `--enforce-shared-experts-fusion` CLI，用于在 DeepEP 环境下强制开启融合（默认仍关闭以保持向后兼容）。  

- **性能影响**  
  - **正向提升**：共享专家不再走独立的 GEMM ‑→ All‑Reduce 流程，省去一次 All‑Reduce 与额外的显存拷贝，尤其在 EP>1、GPU 多卡场景下可显著降低通信开销。  
  - **FlashInfer 路径**：Top‑K 只针对 **路由专家**（`topk_routed = topk - num_fused_shared_experts`）使用 FlashInfer，避免对共享专家做不必要的 CUDA 核心调用，保持 FlashInfer 约束 (`topk <= 8`)。  
  - **额外计算**：在 `_remap_topk_for_deepep` 中插入共享专家 ID 与权重的向量化操作，开销极低（O(num_tokens × num_shared_experts)），对整体吞吐影响可忽略。  
  - **潜在回退**：如果后端不满足 DeepEP 条件（如非 AMD、显卡 Capability < 9.4），仍会走原有共享专家融合路径或完全禁用（受 `disable_shared_experts_fusion` 控制），保证兼容性。  

- **安全考虑**  
  - 变更主要涉及 **内部调度和张量索引**，未引入外部 I/O、文件系统或网络交互，安全风险极低。  
  - 唯一需要关注的是 **索引越界**：`_remap_topk_for_deepep` 中的 ID 计算依赖 `ep_size` 与 `num_local_routed`，若 EP size 与模型配置不匹配（如手动强制 Fusion 但 EP 与 `n_routed_experts` 不可整除），可能导致负索引或重复 ID。已有 `assert` 检查，但在极端配置下仍需测试。  

**⚠️ 潜在风险**  

| 风险点 | 可能后果 | 缓解措施 |
|--------|----------|----------|
|  EP size 与 `n_routed_experts` 不可整除 | 断言失败或错误映射，导致推理崩溃或错误的路由结果 | 在模型加载前添加显式验证，并在 `--enforce-shared-experts-fusion` 前给用户提示。 |
|  `--enforce-shared-experts-fusion` 与 `--disable-shared-experts-fusion` 同时使用 | 参数冲突导致不可预期的行为 | CLI 已声明互斥；在解析阶段检测冲突并抛出明确错误。 |
|  共享专家权重未正确加载（`_load_g_idx` 逻辑） | 共享专家输出全 0 或异常数值，影响模型质量 | 在 `weight_loader` 中加入日志或校验（如检查 `expert_id` 是否在期望范围），并在调试模式下打印映射表。 |
|  FlashInfer 约束未满足（topk_routed > 8） | 退回到慢速 CUDA 实现，性能下降 | 添加警告日志并自动回退；已在条件中限制 `topk_routed <= 8`。 |
|  DeepEP 后端未来升级（新增接口或不同布局） | 现有映射逻辑失效 | 将 `is_deepep_class_backend` 与后端的版本/特性检测耦合，必要时在后端发布新版本时更新映射规则。 |

**💡 关注建议**  

1. **完善配置校验**  
   - 在模型初始化阶段（`DeepseekV2MoE.__init__`）加入 `assert (config.n_routed_experts % moe_ep_size == 0)` 的显式检查，并在 `--enforce-shared-experts-fusion` 打开时给出可读错误信息。  

2. **添加单元/集成测试**  
   - 对 **DeepEP + Fusion**、**DeepEP + No Fusion**、**非 DeepEP** 三种组合分别跑一次完整的前向+后向，验证 Top‑K ID、权重、最终输出的一致性。  
   - 测试 `num_fused_shared_experts = 0`、`=1`、`>1`（虽然目前仅 1）以及不同 EP size（1、2、4、8、16）。  

3. **监控运行时日志**  
   - 在 `_remap_topk_for_deepep` 与 `weight_loader` 中加入 `logger.debug`，在 `--log-level=debug` 时打印每个 Rank 的 `shared_expert_id` 与 `scaled_weight`，帮助定位潜在映射错误。  

4. **文档与使用指南**  
   - 更新 README/CLI 文档，说明何时可以使用 `--enforce-shared-experts-fusion`（仅在 DeepEP 环境且模型满足 256 routed + 1 shared 的前提），以及禁用该选项的后果。  
   - 为 `--disable-shared-experts-fusion` 与 `--enforce-shared-experts-fusion` 添加互斥提示，防止用户误配。  

5. **兼容性回退**  
   - 当检测到不满足 DeepEP 融合条件（如 `is_deepep_class_backend()` 为 False）时，自动将 `self.num_fused_shared_experts` 设为 0，确保不触发错误路径。  

6. **性能基准**  
   - 建议在公开的 DeepSeek V3/R1 基准上跑 **Fusion On/Off** 两套实验，记录 **吞吐（tokens/s）**、**显存占用** 与 **All‑Reduce 时间**，以量化改动收益并在 release note 中给出数值参考。  

---  

**结论**：此次改动在 DeepEP 系列后端下通过把共享专家融合进 MoE kernel，实现了显著的通信和显存优化，并保持了对非 DeepEP 环境的向后兼容。只要在模型配置、EP size 与路由专家数之间保持一致，上述实现应当是安全且高效的。开发者需要关注配置校验与测试覆盖

- `TokenizerManager`、`ExternalCorpusManager`、`NgramWorker` 逐层检查 `speculative_algorithm=="NGRAM"`，并在成功加载后调用 `commit_external_corpus_load` 进行预算记账。

**🎯 影响范围**

- `python/sglang/jit_kernel/ngram_corpus.py`（FFI 包装）  
- `python/sglang/srt/managers/tokenizer_communicator_mixin.py`（RPC 接口）

**💡 关注建议**  

1. **线程安全**  
   - `resetStagingSam()` 在加载线程外被调用（`cancel_external_corpus_load`），但 `_load_result` 的可见性依赖 `thread.join()`，已满足 happens‑before。仍建议在 `resetStagingSam` 前加一次 `assert(!staging_sam_`‑has‑pending‑ops)`，防止异常并发导致资源泄漏。  

2. **错误路径**  
   - `load_external_corpus_named` 现在在预算不足时抛 `ValueError`，随后 `cancel_external_corpus_load` 会清除 `staging_sam_`。确认在 C++ 侧异常（如 `runtime_error`）同样会触发 `cancel_external_corpus_load`，否则可能留下未提交的 SAM。  

3. **预算计数准确性**  
   - `_total_loaded_tokens` 只在 `commit_external_corpus_load` 与 `remove_external_corpus` 时更新，符合“加载成功才计入”。注意在多节点（DP）环境下，`list_external_corpora` 合并后返回的 `corpus_ids` 仍是全局唯一，预算不应因重复计数而被错误扣除。  

4. **API 向后兼容**  
   - 新增 `cancel_external_corpus_load`、`resetStagingSam` 不破坏旧接口，但在用户代码中若仍调用 `clear_external_corpus()` 来中止加载，需要改为 `cancel_external_corpus_load()`，否则会把已加载的 SAM 误删。建议在文档中明确说明。  

5. **性能**  
   - `sams_.find` + `emplace` 替代原来的直接覆盖，开销极小；但在大量小 SAM（数千）时仍建议使用 `std::unordered_map` 替代 `std::map` 以降低查找成本。  

6. **测试覆盖**  
   - 已加入去重、预算回收、异常恢复等单元测试，建议再补充并发加载的集成测试，验证在同一 `corpus_id` 上并行 `load_external_corpus_named` 时是否会出现 race。  

**对开发者**：在添加新外部语料前务必检查 `remaining_token_budget`，并在出现异常时依赖 `cancel_external_corpus_load` 进行清理；如果需要在运行时动态切换 N‑gram 与其他 speculative 算法，确保相应的 RPC 检查已更新。  

**对用户**：  
- 同一 `corpus_id` 只能保存一次，重复添加会报错，请先调用 `remove_external_corpus`。  
- 每个实例有全局 token 上限（`external_corpus_max_tokens`），超过后会在加载阶段直接报错，已加载的语料仍会保留。  
- 若加载中途出现错误，调用 `cancel_external_corpus_load`（或让异常传播）即可安全回滚，不会影响已存在的语料。  

整体来看，变更为多 SAM 管理提供了完整的生命周期、去重与预算控制，提升了可靠性和可预测性。只需注意异常路径的完整清理以及多线程并发的潜在 race，即可在生产环境安全使用。
