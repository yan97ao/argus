### refactor(multimodal/test): centralize model names and shared utilities in test_utils (#19354)
**SHA**: `be9a9e4` | 🔗 [查看提交](https://github.com/sgl-project/sglang/commit/be9a9e4819ac90102470035ee0e55869209285c7)

**🎯 变更类型**：重构  
**⚡ 重要程度**：🔴中  

**📋 变更摘要**  
1. 在 `sglang/multimodal_gen/test/test_utils.py` 中集中定义所有扩散模型的名称常量，取代各测试文件里的硬编码字符串。  
2. 新增一组通用测试工具：动态端口获取、服务器健康轮询、JSON POST、GPU 内存查询与断言等，统一供多套 CLI 与 Server 测试使用。  
3. 相应地更新了多个测试文件的导入与 `model_path` 参数，全部改为引用上述常量，实现“一处修改，多处生效”。

**🎯 影响范围**  
- `python/sglang/multimodal_gen/test/cli/*`、`test/server/*`、`testcase_configs.py` 等所有多模态相关测试代码。  
- CI/CD 流程中运行的全部单元/集成测试。  
- 依赖 `httpx`、`subprocess`（`nvidia-smi`）的新工具函数。

**🔍 技术洞察**  

| 维度 | 影响分析 |
|------|----------|
| **架构影响** | - **测试层抽象提升**：将模型标识与公共工具抽离为 `test_utils`，实现了测试代码的去耦合，降低了后续新增模型或工具时的改动范围。<br>- **模块依赖**：新增 `httpx` 与 `subprocess` 的导入，提升了 `test_utils` 的外部依赖，需要在 `requirements-dev.txt` 中确保同步。 |
| **性能影响** | - **运行时开销**：`wait_for_server_health` 会轮询 HTTP 接口，默认 180 s 超时；在正常情况下耗时几秒，对整体测试时长影响可忽略。<br>- **GPU 内存查询**：`query_gpu_mem_used_mib` 通过 `nvidia-smi` 子进程调用，若频繁使用会有一定的系统调用开销。建议仅在需要断言显存变化的少量测试中使用。 |
| **安全考虑** | - 新增的 HTTP 请求仅针对本地启动的服务器（默认 `127.0.0.1`），不涉及外部网络，风险极低。<br>- `subprocess.check_output` 受限于固定的 `nvidia-smi` 命令，未传入外部输入，安全风险同样可忽略。 |
| **可维护性** | - 常量统一后，模型升级或镜像地址变更只需在单文件修改，避免遗漏。<br>- 公共工具函数的加入提升了代码复用度，后续新增测试可直接复用健康检查、端口分配等逻辑。 |

**⚠️ 潜在风险**  
1. **GPU 环境依赖**：`query_gpu_mem_used_mib` 依赖 `nvidia-smi`，在没有 NVIDIA GPU、驱动或 `nvidia-smi` 可执行文件的 CI 环境会返回 `None` 或触发断言，导致测试异常。  
2. **网络波动**：`wait_for_server_health` 采用轮询实现，若服务器启动异常或端口被占用会导致长时间阻塞直至超时，可能使 CI 耗时增加。  
3. **硬编码路径**：新常量虽然统一，但任何忘记更新的硬编码字符串仍会导致不一致，需确保所有旧引用均已迁移。  
4. **依赖版本**：项目原本可能未直接依赖 `httpx`，若 `requirements-dev.txt` 未同步，部分测试环境会因缺少库而报错。  
5. **跨平台兼容**：`subprocess` 调用 `nvidia-smi` 只在 Linux/macOS 有效，Windows 环境可能报错，需要提前判断平台。

**💡 关注建议**  
- **CI 适配**：在 CI 脚本中检测 `nvidia-smi` 是否可用，若不可用则跳过或关闭显存相关断言；可以通过 `required=False` 的默认行为让函数返回 `None` 并在测试中自行判断。  
- **超时与重试**：为 `wait_for_server_health` 添加可配置的 `max_retries` 或在 CI 中提供更宽松的 `timeout`，以降低因临时资源争用导致的假阴性。  
- **文档同步**：在项目的 `CONTRIBUTING.md` 或测试章节更新说明，列出新增的常量与工具函数的使用方式，提醒贡献者使用统一入口。  
- **依赖声明**：确保 `httpx` 已被加入 `requirements-dev.txt`（或相应的 `extras_require`）并在发布前进行依赖锁定。  
- **回归验证**：在多个平台（Linux GPU、Linux CPU、macOS）运行完整测试套件，确认新工具在无 GPU 环境下仍能正常通过（或被合理跳过）。

## 🔍 技术洞察

| 维度 | 影响分析 |
|------|----------|
| **架构** | 1. **从 JIT → 编译库**：原本在运行时使用 `torch.compile`/`load_jit` 动态生成的 PTX 代码改为在构建阶段直接编译进 `sgl_kernel`。<br>2. **统一调用入口**：所有 NVFP4 相关算子统一为 `torch.ops.sgl_kernel.*`，消除了 `sglang.jit_kernel` 包的存在；代码路径更清晰。<br>3. **构建依赖**：`sgl_kernel` 现在必须在 CMake 配置阶段打开 `ENABLE_NVFP4`（默认宏）并且需要 CUDA 12.8+（因为 NVFP4 只在 Compute 10.x 及以上可用），这把硬件/编译环境的要求前移。 |
| **性能** | - 编译进库的 CUDA 内核能够在构建时使用 `-O3`、`-lineinfo`、`-Xptxas -dlcm=cg` 等优化，通常 **比 JIT‑generated PTX 稍快**（JIT 必须在首次调用时编译），且启动延迟几乎为零。<br>- 通过显式的 `cutlass` 配置与 `get_cached_workspace`（已改为 `torch.empty`）实现了更稳健的工作空间复用，避免了每次调用 `cudaMallocAsync` 带来的开销。<br>- 移除 benchmark 代码本身不影响性能，但团队失去了一套快速本地性能回归检查。 |
| **安全** | - 没有引入新的外部依赖或代码执行路径，安全面影响极小。<br>- 新增的宏检查 (`#if defined ENABLE_NVFP4 && ENABLE_NVFP4`) 在未编译 NVFP4 支持的情况下会直接抛出 `TORCH_CHECK_NOT_IMPLEMENTED`，防止在不兼容的环境下 silently 执行错误的 kernel。 |
| **可维护性** | - **优点**：代码集中在 `sgl_kernel`，统一的 C++/CUDA 实现更易审计、调试；Python 包层仅作 thin wrapper，减少了 JIT 相关的复杂性（如 `cache_once`, `load_jit`）。<br>- **缺点**：失去了 JIT 自动适配不同 CUDA 版本的灵活性；每当 CUDA 或 CUTLASS 版本升级时，需要重新编译 `sgl_kernel` 并确保 `ENABLE_NVFP4` 宏正确设置。<br>- 测试套件被全部删除（benchmark 与单元测试），导致上游对 NVFP4 功能的回归验证依赖手动实验。 |
| **兼容性** | - **硬件**：仍然要求 Compute 10.0+（Ada/Lovelace/Blackwell），但编译阶段会直接报错；低于该 Compute Capability 的 GPU 现在会在库加载时触发 `RuntimeError`，而不是 JIT 时的 “unsupported”。<br>- **Python 环境**：原本可在不具备 NVFP4 编译环境的机器上运行（因为 JIT 会在运行时尝试加载），现在 **必须** 在安装 `sgl_kernel` 时包含相应的 CUDA 编译工具链。<br>- **API 兼容**：函数签名保持不变（`scaled_fp4_quant`, `cutlass_fp4_group_mm`），但内部实现与异常类型不同（使用 `TORCH_CHECK_NOT_IMPLEMENTED`），调用者若捕获特定异常需适配。 |
| **构建/部署** | - `sgl-kernel/CMakeLists.txt` 新增 5 个 NVFP4 源文件，导致编译时间略增。<br>- 需要在构建 `sgl_kernel` 时打开 `-DENABLE_NVFP4=ON`（通过 `VLLM_CUTLASS_SRC_DIR` 或直接在 CMake 配置中设定），否则对应 ops 会在 runtime 报错。<br>- 对 CI 环境的要求提升：必须拥有 CUDA 12.8+ 与 `cutlass` 3.8 兼容源码。 |

### fix(grpc): use context.abort() with proper status codes instead of in-band errors (#19972)
**SHA**: `dd2bbe6` | 🔗 [查看提交](https://github.com/sgl-project/sglang/commit/dd2bbe6d627d2d51c65e43e11e911ed5df4fc1dc)

**🎯 变更类型**：Bug修复  
**⚡ 重要程度**：🔴高  
**📋 变更摘要**：  
- 将 SGLang gRPC 服务在检测到调度器返回的错误时，从“在响应体内返回 error 字段”改为直接使用 `context.abort()` 抛出 gRPC 状态码，实现了标准化的错误传播。  
- 新增 `sglang/srt/grpc/utils.py`，提供 `abort_code_from_output` 将调度器的 HTTP‑style 错误映射为对应的 `grpc.StatusCode`。  
- 对 `Generate` 与 `Embed` 两个 RPC 的异常处理统一改为 `abort`，并对非法请求返回 `INVALID_ARGUMENT`，内部异常返回 `INTERNAL`。  

**🎯 影响范围**：  
- `python/sglang/srt/entrypoints/grpc_server.py`（核心 gRPC Server 实现）  
- 新增 `python/sglang/srt/grpc/utils.py`（错误码映射工具）  
- 任何使用 SGLang gRPC 接口的客户端（Python、C++, Go 等）  

**🔍 技术洞察**  

- **架构影响**  
  - **错误处理模型统一**：从“业务层在 protobuf 中自行编码错误”转为 “利用 gRPC 本身的错误机制”。这简化了服务器端业务逻辑，降低了在不同 RPC/stream 场景下自行拼装错误响应的复杂度。  
  - **向后兼容性**：原先的 `GenerateResponse.error`/`EmbedResponse.error` 已被废弃，旧客户端若仍解析这些字段将收到 `UNAVAILABLE`（或其他）状态码的异常，而不是成功返回的消息，可能导致运行时错误。需要在 SDK/文档层面同步更新。  
  - **异常捕获语义明确**：新增对 `grpc.aio.AbortError` 的捕获与重新抛出，防止在异常链中误将 `abort` 当作普通异常处理，保证 gRPC 框架自行完成流关闭与资源回收。

- **性能影响**  
  - **降低网络带宽**：错误不再包装在完整的 protobuf 消息中，而是直接通过 gRPC 状态码返回，仅携带简短的错误描述，减小了响应体大小。  
  - **CPU 开销微增**：每次抛出 `abort` 会触发 gRPC 框架内部的状态码映射与流终止逻辑，开销可忽略不计，且相较于序列化大量 `GenerateError` 消息更高效。  
  - **异常路径可控**：统一使用 `abort` 有助于快速退出异常路径，避免在后续的业务循环中继续处理无效 batch，提升异常情况下的响应时间。

- **安全考虑**  
  - **信息泄露控制**：通过 `abort` 直接返回错误，避免了在 protobuf 中携带潜在的堆栈信息或内部实现细节（原实现曾把 `details` 直接写入响应），更符合最小暴露原则。  
  - **状态码映射可靠性**：`abort_code_from_output` 只映射了 `400、503、500` 三种 HTTP 状态，其他自定义错误会默认为 `INTERNAL`，可能导致客户端无法区分具体错误来源。若业务上有细粒度错误分类需求，需要扩展映射表并确保不会把敏感信息误映射为更宽松的状态码。  

**⚠️ 潜在风险**  

1. **客户端兼容性**：依赖 `GenerateResponse.error`/`EmbedResponse.error` 字段的旧版客户端将在收到 gRPC `ABORT` 时抛异常，若未捕获会导致未处理的异常。  
2. **映射表不完整**：仅覆盖三个 HTTP 状态码，其他如 `401、403、404` 等将被统一为 `INTERNAL`，可能误导调用方。  
3. **异常链遗漏**：在 `except Exception as e` 中直接 `await context.abort(...)`，但未记录原始异常的堆栈到 gRPC `details`，调试时可能失去原始信息。  
4. **流式 RPC 中的中止行为**：`Generate` 支持流式返回，`abort` 会立即终止整个流。若业务期望在流式场景下仅标记单个 batch 为错误（而后继续发送其它 batch），则此改动会导致全部后续数据被截断。  

**💡 关注建议**  

- **客户端升级**：在所有使用 SGLang gRPC 的 SDK 中捕获 `grpc.RpcError`，根据 `code()` 判断错误类型，而不是检查 protobuf `error` 字段。文档应同步更新错误处理示例。  
- **扩展错误映射**：评估现有调度器可能返回的所有 HTTP 状态码，完善 `_HTTP_TO_GRPC_CODE`，并在 `abort_code_from_output` 中加入对自定义 `status_code` 的直接映射（如 `grpc.StatusCode.PERMISSION_DENIED`）。  
- **保留错误细节**：可考虑在 `context.abort` 的描述文字中加入 `details`（如调用 `str(e)`），或通过 `grpc.experimental.set_trailing_metadata` 发送调试信息，以便运维排查。  
- **回退兼容**：若业务方仍需兼容旧协议，可在 `grpc_server.py` 中保留一个可选开关（环境变量或配置），在开启时仍返回 protobuf 错误字段，以平滑迁移。  
- **测试覆盖**：新增单元测试与集成测试，验证：  
  1. 各类调度器错误能够映射到预期的 `grpc.StatusCode`；  
  2. 流式 `Generate` 在错误出现后正确终止流并返回对应状态码；  
  3. 非法请求触发 `INVALID_ARGUMENT`。  

通过上述措施，可确保此次错误处理改进在提升标准化、降低泄露风险的同时，保持对现有生态的平稳过渡。

**🎯 影响范围**  
- CI 脚本（`scripts/ci/cuda/*`）

总体来看，这次改动显著降低了 CI 的网络与磁盘开销，避免了因 GPU 异常导致的卡死。建议在几轮 CI 完整跑通后，观察 runner 的日志与缓存命中率，确保上述 Edge‑case（多版本并发、网络抖动）不会引入新问题。
