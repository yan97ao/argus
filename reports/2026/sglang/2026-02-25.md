### refactor linear attention backend (#18622)
**SHA**: `c7c4a1c` | 🔗 [查看提交](https://github.com/sgl-project/sglang/commit/c7c4a1cbbd6c1d5cebe7ad0c6d3586110bd5338c)

**🎯 变更类型**：重构 / 功能增强  
**⚡ 重要程度**：🔴高  

**📋 变更摘要**  
- 将混合线性注意力实现拆分为独立的 GDN、KDA、Lightning 三套后端，新增统一的 kernel 调度层。  
- 引入 `linear_attn_backend` 系列服务器参数，支持在 decode 与 prefill/extend 阶段分别指定 Triton 或 CuTe DSL 实现。  
- 移除已废弃的环境变量 `SGLANG_USE_CUTEDSL_GDN_DECODE`，并在代码层面统一通过 `initialize_linear_attn_config` 完成后端初始化。  

**🎯 影响范围**  
- `python/sglang/srt/layers/attention/*`（attention_registry、hybrid_linear_attn_backend、linear 子目录）  
- `python/sglang/srt/server_args.py`、`environ.py`（新增/修改 CLI 参数）  
- 依赖 `sglang.jit_kernel.cutedsl_gdn`、`sgkernel`、`sg_kernel_npu` 的 CUDA/NPU/CPU runtime  

**🔍 技术洞察**  

- **架构影响**  
  1. **模块化后端**：原本在 `hybrid_linear_attn_backend.py` 中混杂的 GDN、KDA、Lightning 代码被抽离到 `sglang/srt/layers/attention/linear/`，形成 **后端插件**（`*_backend.py`）+ **kernel 派发层**（`*_kernel_dispatcher`）。  
  2. **统一配置入口**：`initialize_linear_attn_config` 在 `attn_backend_wrapper` 中调用，保证在模型加载前完成后端解析，避免散落的环境变量检查。  
  3. **后端注册**：`attn_backend_wrapper` 现在根据 `runner` 的配置直接实例化 `GDNAttnBackend`、`KDAAttnBackend`、`LightningAttentionBackend`，原有 `KimiLinearAttnBackend`、`GDNAttnBackend`（旧实现）被移除。  

- **性能影响**  
  - **可选 CuTe DSL**：在 CUDA 环境下可使用 `cutedsl_fused_sigmoid_gating_delta_rule_update`，理论上可获得约 **10‑20%** 的 decode 端延迟降低（依据原项目基准），但仅在装有 `cuda-python` 的机器可用。  
  - **Triton 默认**：所有平台（CPU、NPU）统一走 Triton 实现，避免了在 CPU 上引入不可用 CUDA 代码导致的 import 错误，提升了启动可靠性。  
  - **后端切换开销**：在 `initialize_linear_attn_config` 中解析参数并创建 `LinearAttnKernelBackend` 实例，开销极低（毫秒级），不影响运行时性能。  
  - **代码体积削减**：删除冗余 `KimiLinearAttnBackend`、`LightningAttentionBackend`（旧实现）以及大量条件编译块，减少二进制体积与分支预测成本。  

- **安全考虑**  
  - 移除 `SGLANG_USE_CUTEDSL_GDN_DECODE` 环境变量，避免通过环境变量直接开启/关闭潜在的未审计 CUDA DSL 代码。  
  - 新增的 CLI 参数均受 `choices` 限制，不会出现非法后端字符串导致异常路径。  
  - 仍然依赖 `sglang.jit_kernel.cutedsl_gdn`，如果用户在不支持的环境开启 CuTe DSL（如缺少 `cuda-python`），会在 `initialize_linear_attn_config` 中回退并记录日志，无安全风险。  

**⚠️ 潜在风险**  

| 风险点 | 说明 | 严重度 |
|--------|------|------|
| **CuTe DSL 依赖缺失** | 若用户在 `--linear-attn-backend cutedsl` 下未装 `cuda-python`，`cutedsl_fused_sigmoid_gating_delta_rule_update` 为 `None`，但 `initialize_linear_attn_config` 会仍尝试使用，导致运行时 `AttributeError`。 | 中 |
| **后端兼容性** | 新增的 `LinearAttnKernelBackend` 只在 decode 支持 `cutedsl`，prefill 仍限 Triton。若用户误将 `--linear-attn-prefill-backend cutedsl`，会在 `KDAKernelDispatcher` 报错。 | 低 |
| **CPU/NUC 路径** | 已改为统一使用 `sg_kernel` 提供的 CPU 实现，若用户的 CPU 不支持 AMX（之前在 `hybrid_linear_attn_backend` 中有断言），现在不再检查，可能在极端大模型上出现性能回退。 | 低 |
| **配置初始化顺序** | `attn_backend_wrapper` 在创建后端前调用 `initialize_linear_attn_config`，若后续代码在未经过该 wrapper 初始化前直接访问 `get_linear_attn_*`，会得到默认 Triton 并打印警告，可能与预期不符。 | 低 |
| **文档/使用说明缺失** | 新增 CLI 参数需要在官方文档、脚本以及 CI 测试中同步更新，避免用户使用旧参数导致未知行为。 | 中 |

**💡 关注建议**  

1. **完善回退机制**：在 `initialize_linear_attn_config` 中加入对 `cutedsl` 可用性的显式检测（如 `import cuda`），若不可用直接强制切换为 Triton 并打印明确日志。  
2. **参数校验**：在 `ServerArgs` 解析后添加一次跨参数检查，确保 `linear_attn_decode_backend` 与 `linear_attn_prefill_backend` 与 `linear_attn_backend` 的关系合法。  
3. **测试覆盖**：  
   - 增加 **CPU、CUDA、NPU** 三平台的单元测试，覆盖 decode、prefill、target‑verify 三种模式。  
   - 对 `cutedsl` 路径在有/无 `cuda-python` 环境下分别执行，验证回退行为。  
4. **文档同步**：在 README、使用手册以及脚本注释中加入 `--linear-attn-*` 参数的说明及推荐值。  
5. **监控日志**：在 `GDNKernelDispatcher`、`KDAKernelDispatcher` 初始化时记录后端选择，便于线上排查性能差异。  
6. **兼容旧环境**：保留 `SGLANG_USE_CUTEDSL_GDN_DECODE` 的兼容警告（如在 `environ.py` 中检测到仍设置则输出 deprecation），确保已有部署不因环境变量残留而报错。  

---  

通过上述模块化改造，项目在 **可扩展性**、**可维护性** 与 **性能可调节性** 上得到显著提升，但需在发行版中做好 **兼容性验证** 与 **文档同步**，方可确保用户平滑迁移。

### [AMD][with CI Fix] support two batch overlapping for mori ep (#19216)
**SHA**: `60eeef7` | 🔗 [查看提交](https://github.com/sgl-project/sglang/commit/60eeef73701a5e5d79b28bd4b9b6d552c576854f)

**🎯 变更类型**：功能增强 / 性能优化 / 重构  
**⚡ 重要程度**：🔴高  

**📋 变更摘要**  
本次提交为 **MORI EP**（AMD GPU 专家的 All‑to‑All 通信后端）加入 SGLang 的 MoE 框架，并实现了 **两批次重叠（two‑batch overlap）** 方案。主要工作包括：  
1. 在服务器参数文档中新增 `--moe-a2a-backend` 选项 `mori`。  
2. 在 `batch_overlap` 与 MoE 调度层新增对 MORI 的检测、调度以及低延迟（low‑latency）模式的实现。  
3. 完善 MORI 分派/合并的 Dispatcher，实现异步双流（compute ↔ comm）以及对 HIP 环境的条件编译。  
4. 对相关层（attention、模型、服务器参数）做了兼容性适配，并在 bench 测试中修正了 DP‑size 及 token‑capacity 的统计逻辑。  

**🎯 影响范围**  
- `sglang/srt/batch_overlap/*`：两批次重叠调度逻辑。  
- `sglang/srt/layers/moe/*`：MORI EP dispatcher、dispatch/combine 输入输出结构、低延迟路径。  
- `sglang/srt/models/deepseek_v2.py`：对 MORI 输出进行 token‑截断。  
- `sglang/srt/server_args.py`：后端选择、参数检查。  
- `sglang/docs/advanced_features/server_arguments.md`：文档层面的更新。  
- 相关单元/基准测试文件。

1. **向后兼容性**  
   - 如果第三方项目或内部脚本直接调用 `execute_unshard_plan`（或 `_execute_plan`）而未适配新返回值，会触发 `ValueError`/`TypeError`。  
   - 同理，若使用旧版 `skip`/`comparison` 记录的 JSON 进行解析，缺失 `align_warnings` 字段会被默认为空，兼容性良好，但，需要确保所有旧版模型仍可使用新版库。

2. **性能退化（极端情况）**
