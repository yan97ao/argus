### [diffusion] model: support LTX2.3 two stage (#22182)
**SHA**: `495ef8e` | 🔗 [查看提交](https://github.com/sgl-project/sglang/commit/495ef8ec64b6b937e59cd530ad3150172061a008)

**🎯 变更类型**：功能增强 / 架构变更 / 性能优化  
**⚡ 重要程度**：🔴高  

**📋 变更摘要**  
本次提交为 **LTX‑2.3 两阶段（one‑stage 与 two‑stage）** 的完整支持，实现了：  
1. 文档与兼容性矩阵更新，明确 LTX‑2/2.3 的两阶段使用方式。  
2. 在 `pipeline_configs` 与核心运行时中加入 **audio‑latents 的 Sequence‑Parallel (SP) 分片**、**RoPE 坐标准备**、**mask 传递** 等新接口。  
3. 为 LTX‑2.3 引入 **原生 overlay**（不再手动 materialize），并在模型加载、路径解析、LoRA 合并策略等处做兼容。  
4. 扩展 `USPAttention` 以支持外部 `attn_mask`，并在多种并行模式（SP、Ulysses、Ring）下安全使用。  
5. 大幅重构 **DenoisingStage** 与 **AVDenoisingStage**：实现 LTX‑2.3 特有的 sigma‑space Euler 步、跨模态注意力、TI2V 清洁状态、音频/视频坐标准备以及 SP‑padding mask。  
6. 统一了测试用例、基准配置以及服务器参数的自动调优（如一机一卡时默认 `attention_backend=fa` 以避免精度损失）。  
7. 删除旧的 `model_overlays/ltx_2_3/_overlay/materialize.py`，改用内置 overlay 注册表。

**🎯 影响范围**  
- **核心模块**：`sglang/multimodal_gen/configs/pipeline_configs/*`、`runtime/models/dits/ltx_2.py`、`runtime/pipelines/ltx_2_pipeline.py`、`runtime/pipelines_core/*`（尤其是 `denoising_av.py`、`latent_preparation_av.py`）。  
- **文档**：`docs/diffusion/compatibility_matrix.md`。  
- **测试**：`test/server/*`、`test/unit/*`（全部基于 LTX‑2.3 的新行为）。  
- **运行时依赖**：新增对 `torch.autocast`、`scaled_dot_product_attention`、SP/ULYSES/Ring 并行的 mask 处理；对 `attention_backend` 自动选择的逻辑。  
- **模型部署**：`model_overlays` 注册表新增 LTX‑2.3 overlay，原有手动 materialize 路径已被删除。

**🔍 技术洞察**  

| 维度 | 影响描述 |
|------|----------|
| **架构影响** | 1. **新增 SP 分片接口**：`can_shard_audio_latents_for_sp`、`shard_audio_latents_for_sp`、`gather_audio_latents_for_sp` 等，使音频 token 在多 GPU 场景下可水平切分，保持与视频 token 对齐。<br>2. **两阶段 Pipeline**：`LTX2TwoStagePipeline` 通过 `--pipeline-class-name LTX2TwoStagePipeline` 自动解析 spatial‑upsampler 与 distilled‑LoRA，兼容 legacy 与 native 两种布局。<br>3. **Attention 扩展**：`USPAttention` 现在接受外部 `attn_mask`，并在需要时执行 all‑to‑all、mask gather，保持在 SP、Ulysses、Ring 环境下的正确性。<br>4. **RoPE 坐标抽象**：`prepare_video_rope_coords_for_sp` / `prepare_audio_rope_coords_for_sp` 按需在局部 SP shard 上生成局部坐标，避免全局坐标复制。<br>5. **模型加载统一化**：通过 `BUILTIN_MODEL_OVERLAY_REGISTRY` 直接指向 HF overlay，删除原有复杂的 materialize 脚本，简化部署流程。 |
| **性能影响** | - **音频/视频并行**：在多 GPU（SP>1）时，audio latents 经过分片后只处理本地帧，可减少显存峰值约 30%（视帧数而定），并提升跨模态注意力的吞吐。<br>- **Mask 计算**：在 `USPAttention` 中对 mask 做一次 gather，后续计算全局统一，避免在每个注意力层重复 host‑device 同步。<br>- **LoRA 合并策略**：两阶段模型默认在 stage2 融合 distilled‑LoRA，减小推理时的额外算子调度开销。<br>- **后端自动选择**：在单卡 CUDA 环境自动切换到 `fa`（FlashAttention）后端，降低显存占用并提升算子速度 (~10‑15% 对比默认 `torch` 实现)。 |
| **安全考虑** | - **Mask 处理**：新增的 mask gather 在 Ring 并行下尚未实现，代码中已抛 `NotImplementedError`，防止在未支持的环境下产生错误的注意力权重。<br>- **路径解析**：`_resolve_ltx2_two_stage_component_paths` 通过安全的 `os.path.join` 拼接，未引入路径遍历风险。<br>- **模型权重转换**：旧的 `materialize.py` 已删除，改用官方 overlay，避免因自行重打包导致的隐藏后门或权重篡改。 |
| **可维护性** | - 将 SP 相关逻辑集中在 `pipeline_configs/base.py` 与 `ltx_2.py`，统一 API，后续新模型只需实现对应的 `can_shard_*`/`shard_*`/`gather_*` 方法即可。<br>- 文档同步，兼容矩阵一目了然，降低使用者误配风险。<br>- 测试用例已根据新模型重新筛选，只保留通过的 case，提升 CI 稳定性。 |

**⚠️ 潜在风险**  
1. **Ring Parallelism 与 Mask**：`USPAttention` 在 Ring 环境下仍不支持 mask，若用户手动开启 `ring_degree>1` 并使用带 mask 的模型（如 LTX‑2.3），将触发 `NotImplementedError`，导致运行时崩溃。  
2. **音频 SP 分片边界**：在极端帧数不被 SP world size 整除时，需要正确的 `sp_audio_valid_token_count` 与 `sp_video_valid_token_count`；若后续代码误用 `sp_*_latent_num_frames` 而非 `*_valid_token_count`，可能产生 padding 影响生成质量。  
3. **LoRA 合并策略误判**：`_should_merge_stage2_distilled_lora` 依赖 `is_ltx23_native_variant` 判断，若未来出现新的变体未更新该判断，可能导致 stage2 LoRA 未正确合并或错误合并，影响图像质量。  
4. **自动切换 FA 后端**：在某些低版本 CUDA/驱动环境下，FA 可能不支持全部算子，自动切换可能产生隐蔽的数值误差或崩溃。  
5. **移除 materialize 脚本**：旧的手动 materialize 逻辑已废弃，若用户仍使用自定义 overlay 路径而未更新配置，可能找不到模型文件导致启动失败。  

**💡 关注建议**  
- **文档与警示**：在官方文档中明确标注 “Ring Parallelism 暂不支持带 mask 的 LTX‑2.3”，并在 CLI 参数校验阶段给出友好错误提示。  
- **单元测试**：补充针对 SP‑padding mask 边界（如 `seq_len % sp_world_size != 0`）的回归测试，确保 `valid_local_frames` 计算正确。  
- **兼容层**：为 Ring 并行提供占位实现或回退到非 Ring 路径，以免用户在默认配置下意外触发 `NotImplementedError`。  
- **LoRA 策略配置**：提供 `--force-merge-stage2-lora` 开关，让用户在特殊变体或调试时手动控制 LoRA 合并行为。

### [diffusion] CI: dynamic load-balanced partitioning for diffusion CI (#15528)
**SHA**: `45472d7` | 🔗 [查看提交](https://github.com/sgl-project/sglang/commit/45472d70cc18c46f5c6f383bcc39d128b95a5480)

**🎯 变更类型**：功能增强 / 架构变更 / 性能优化  
**⚡ 重要程度**：🔴 高  

**📋 变更摘要**  
本次 PR 为 **Diffusion CI** 引入了 **动态负载均衡分区**，通过在 GitHub Actions 中新增 “compute‑diffusion‑partitions” 工作，实现：

1. **自动计算分片数 & 分配**（基于每个 test case 与 standalone 脚本的预估运行时）。  
2. **将分片信息以 matrix、partition‑count、plan‑json 形式输出**，供后续 1‑GPU / 2‑GPU 测试作业使用。  
3. **run_suite.py** 重新设计：支持 **partition‑plan‑json**、记录执行报告（JSON），并在 CI 末端加入 **diffusion‑coverage‑check** 作业校验 100% 覆盖率。  
4. **perf_baselines.json** 添加 `estimated_full_test_time_s` 字段，支持更精确的分片估算；同时在测试基准生成路径中自动写入该字段。  
5. 辅助脚本 **compute_diffusion_partitions.py** 与 **diffusion_case_parser.py** 实现了轻量级的 AST 解析与 LPT（Longest Processing Time）分片算法。  
6. 其它细节：CLI 测试基于统一的 `run_command`、增加执行报告上传、对缺失估算的提示与自动化校验。  

整体目标是 **让 CI 时长在 20‑40 分钟区间均衡分配**，避免单个 shard 超时，同时保证所有 Diffusion 测例都被执行。

## 🔍 技术洞察

### 架构影响
1. **CI 流程分层**  
   - **计算层**：单独的 `compute-diffusion-partitions` 作业在 `ubuntu-latest`（轻量）上完成所有分片计算，输出 GitHub **step outputs**（matrix、partition‑count、plan‑json）。  
   - **执行层**：原先的固定 `matrix: [0,1]` 被替换为 **动态矩阵** (`fromJson(needs.compute-diffusion-partitions.outputs.matrix-1gpu)`)，每个分片仅运行自己分配的 case/standalone。  
   - **验证层**：新增 `diffusion-coverage-check` 作业下载所有执行报告并统一校验覆盖率，确保没有遗漏。  
   - **模块解耦**：`run_suite.py` 不再硬编码分片规则，而是接收 **完整的 partition plan JSON**，实现了 **业务层（测试）与调度层（CI）** 的解耦。

2. **数据流**  
   - `compute_diffusion_partitions.py` 读取 **testcase_configs.py**、**run_suite.py**、**perf_baselines.json**（全部通过 AST 解析），生成 **PartitionItem**（case / standalone） → LPT 分片 → 输出 **GitHub matrix** 与 **plan‑json**。  
   - `run_suite.py` 根据 `--partition-plan-json` 从 plan 中解析出本分片应运行的 `case_ids` 与 `standalone_files`，并通过统一的 `run_pytest` 调用执行。  
   - 每个分片结束后 **upload‑artifact** `execution_report_*.json`，供后置作业统一收集。

3. **可维护性**  
   - 所有与 **分片** 相关的逻辑被抽离到独立脚本，业务代码 (`run_suite.py`) 只负责 **执行**，降低了耦合度。  
   - Baseline JSON 与 `estimated_full_test_time_s` 成为 **唯一的时间来源**，后续新增模型只需要更新基线即可自动参与分片。

### 性能影响
| 维度 | 预期收益 | 可能的副作用 |
|------|----------|--------------|
| **CI 总时长** | 通过 LPT 使每个 shard 运行时间在 20‑40 min 区间，极大降低单GPU作业超时的概率，提升资源利用率。 | 计算分片作业本身会耗费 ~30 s（轻量），对整体 CI 影响可忽略。 |
| **并行度** | 动态决定分片数（最多 10），在模型多、耗时长的 PR 中自动扩展并行度。 | 当所有 case 总时长远小于 `min_time` 时会产生 **单分片**，但这正是预期行为。 |
| **启动开销** | 对每个 case 仍会有 **模型加载 + server 启动**（约 2 min）; 通过分片均衡，这部分开销在整个 CI 中分摊更均匀。 | 若 `estimated_full_test_time_s` 丢失，将使用默认 300 s，导致 **分片误判**，可能出现某 shard 超时。 |
| **磁盘/网络** | 新增 `execution_report_*.json`（每个分片约几 KB）以及 JUnit XML，产生轻量的 artifact，影响不大。 | 大量分片（接近上限 10）时，报告文件数量上升，GitHub Artifact **大小/数量** 必须在配额内。 |

### 安全考虑
- **代码执行安全**：`compute_diffusion_partitions.py` 与 `diffusion_case_parser.py` 通过 **AST 解析** 读取项目源码，未执行任何业务代码，降低了代码注入风险。  
- **环境变量**：CI 新增 `PARTITION_PLAN_JSON` 传递给测试作业，内容来源于内部计算脚本，未涉及外部输入，风险可控。  
- **敏感信息泄露**：上传的 `execution_report_*` 包含 **case IDs** 与 **运行状态**，但不包含模型权重或用户数据，符合公开 CI 的信息安全要求。  
- **权限**：所有新作业仍在原有 CI 角色（repo‑write）下运行

### [lora][moe] Decoupled LoRA MoE backend with Marlin support (#21858)
**SHA**: `8da1cfb` | 🔗 [查看提交](https://github.com/sgl-project/sglang/commit/8da1cfb30d12675c5a45787b391cfec1af48476e)

**🎯 变更类型**：功能增强 / 架构变更 / 性能优化  
**⚡ 重要程度**：🔴 高  

**📋 变更摘要**  
1. 为 MoE（Mixture‑of‑Experts）层引入 **LoRA（Low‑Rank Adaptation）钩子抽象**，把 LoRA 注入点统一为 “after_gate_up” 与 “after_down”。  
2. 在所有 MoE runner（Triton、Triton‑Kernels、Deep‑GEMM、Marlin）上加入 `hooks` 参数并在关键阶段调用，完成 **LoRA‑in‑place** 融合而不是后置相加。  
3. 新增 **Marlin‑LoRA runner**（`lora_moe_runner_marlin.py`），利用 Marlin int4/int8 kernels完成基线 MoE 计算，并通过钩子注入 LoRA。  
4. 扩展压缩张量（CompressedTensors）和量化层，提供 `get_marlin_quant_info`，让 LoRA 在 Marlin 路径下能够获取权重量化元信息。  
5. `LoraLinearLayer` 现在根据全局或量化方法的 `MoeRunnerBackend` 自动选取 Triton 或 Marlin 后端，并提前准备对应的量化信息。  
6. 新增大量单元/集成测试（`test_lora_moe_runner.py`、`test_marlin_lora_correctness.py`），覆盖 Triton 与 Marlin 两条路径、不同专家数、不同 LoRA Rank、以及基线‑LoRA 差值校验。  

**🎯 影响范围**  
- `sglang/srt/layers/moe/moe_runner/*`（所有 MoE runner 实现）  
- `sglang/srt/layers/moe/moe_runner/base.py`（抽象接口新增 `hooks`）  
- `sglang/srt/layers/quantization/compressed_tensors/*`（新增 Marlin 量化信息）

### [tokenizer] eliminate O(n²) copy in non-incremental streaming (#22567)
**SHA**: `c6fd9a0` | 🔗 [查看提交](https://github.com/sgl-project/sglang/commit/c6fd9a00c7ccaecb0211676139ca98f43798410b)

**🔧 变更类型**：功能增强 / 性能优化  
**⚡ 重要程度**：🟡 中  
**📋 变更摘要**：在 `tokenizer_manager.py` 中，对非增量（non‑incremental）流式输出的处理做了改造，避免在每一次增量生成时拷贝完整的 `output_ids` 与拼接 `text`，从而把原先的 **O(n²)** 字符串/列表拷贝降低为 **O(n)**。实现方式是：  
1. 对中间块只返回 `{"text": None, "output_ids": state.output_ids, …}`，不立即生成文本；  
2. 在 `_wait_one_response` 中检测到 `text is None` 时，使用 `state.get_text()` 进行一次性拼接；  
3. 完成后（`state.finished`）仍然拷贝 `output_ids` 并返回完整文本，保持对外行为不变。

**🎯 影响范围**  
- `sglang/srt/managers/tokenizer_manager.py`（流式 Tokenizer 管理）  
- 与流式 API 交互的服务器路径、客户端 SDK（依赖 `text` 字段及时可得的逻辑）  
- 可能影响日志、监控以及用户自定义回调中对 `output_ids` 的即时修改假设。

**💡 关注建议**  

| 关注点 | 建议 |
|--------|------|
| **数据完整性** | `output_ids` 现在在中间块直接引用 `state.output_ids`，如果后续还有并发操作（如用户自行修改返回的列表）会导致已发送的块被意外改变。建议在文档中说明不可在回调中修改 `output_ids`，或在发送前做一次浅拷贝以防止误改。 |
| **延迟 text 生成** | `text` 只在 `_wait_one_response` 中补齐，若有代码在 `_handle_batch_output` 之前尝试读取 `out["text"]` 会得到 `None`。确保所有使用方在收到响应前已完成 `await`，或在新增的文档中明确说明该行为。 |
| **增量 vs 非增量流** | 新增的 `incremental_stream` 条件分支保持原有路径不变。建议补充单元测试覆盖：① 增量流‑正常；② 非增量流‑中间块‑text 为 None 且最终块返回完整文本；③ 完成后 `output_ids` 是否被复制。 |
| **线程/协程安全** | `state.get_text()` 读取 `state.output_ids`，若在补齐 `text` 前 `state` 已被其它协程修改（如提前调用 `finish`），可能出现不一致。考虑在 `_wait_one_response` 中对 `state` 加锁，或在 `state.finish` 前把 `output_ids` 固定。 |
| **回归测试** | 由于改动涉及返回对象的可变性，建议在现有性能基准（特别是长文本、千级 token）上对比前后内存拷贝次数与 CPU 使用率，确保 O(n²) 的热点已消除且没有引入额外的 GC 开销。 |
| **文档更新** | 在官方 API 文档的“流式输出”章节补充说明：**非增量流**的中间块 `text` 可能为 `null`，最终块才会包含完整文本；并提示用户在自定义回调中不要修改返回的 `output_ids`。 |

**总体评价**  
此改动显著降低了长文本流式生成时的内存拷贝成本，对大模型（千‑万 token）尤为关键。只要注意上述可变引用的使用规范并补足对应测试，风险可控，推荐合并。
