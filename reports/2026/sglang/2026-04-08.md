### [Score API] Add SequenceClassification Model support (#22118)
**SHA**: `712c8c5` | 🔗 [查看提交](https://github.com/sgl-project/sglang/commit/712c8c50512e0cbbcf9a073232d23ce7ef44ec5d)

**🎯 变更类型**：功能增强 / 重构  
**⚡ 重要程度**：🔴高  

**📋 变更摘要**  
- 为 SGLang 的 `Engine.score` 接口新增对 **SequenceClassification** 模型的直接评分支持，模型可以返回分类头的 pooled logits（可选 softmax）。  
- 在多项（MIS）评分场景下，引入 **delimiter token** 的概念，实现对同一序列中多个 item 的逐项打分，兼容生成模型与分类模型。  
- 新增 `score_and_pool` 辅助函数，将分类头与原有 `Pooler` 统一抽象，避免在各模型实现里重复逻辑。  
- `TokenizerManagerScoreMixin` 重新设计：支持 **EmbeddingReqInput**（用于分类模型），统一处理 `input_token_ids_logprobs` 与 `embedding` 两种返回格式。  
- API 文档、HTTP 路由以及相关单元/集成测试同步更新，覆盖单条、批量、MIS、Tokenized 输入等多种用例。  

**🎯 影响范围**  
- **核心层**：`engine_score_mixin.py`, `http_server.py`, `layers/pooler.py`, `managers/tokenizer_manager_score_mixin.py`。  
- **模型实现**：`llama_classification.py`, `qwen2_classification.py`, `qwen3_classification.py`（统一使用 `score_and_pool`）。  
- **服务器配置**：`server_args` 中新增 `multi_item_scoring_delimiter` 参数。  
- **API**：`/v1/score` 端点现同时支持 CausalLM 与 SequenceClassification。  
- **测试**：新增 `test_score_classification.py`, `test_pooler_score_and_pool.py`, `test_multi_item_scoring.py`，对新特性进行端到端验证。  

**🔍 技术洞察**  

- **架构影响**  
  - **模型抽象统一**：以前 `score` 只能基于 `token_ids_logprob`（生成模型）计算 log‑prob，现通过 `EmbeddingReqInput` + `score_and_pool` 支持 **classification head** 的直接输出，保持统一的 `ScoreResult` 返回结构。  
  - **请求路由分支**：`TokenizerManagerScoreMixin.score_request` 根据 `self.is_generation`（在 `Engine` 中由模型类型决定）选择 `GenerateReqInput` 或 `EmbeddingReqInput`，新增的分支对后端调度器（scheduler）稍有改动，但对其它路径影响最小。  
  - **MIS 处理**：在 `pooler.score_and_pool` 中根据全局 `multi_item_scoring_delimiter` 与 `forward_batch.is_prefill_only` 判断是否走 MIS 路径，使用 GPU 上的 token 位置直接切片，随后在 CPU 端依据 `extend_seq_lens_cpu` 拆分成 per‑request 张量，保持调度器一次性返回的接口不变。  

- **性能影响**  
  - **Classification 路径**：不再进行 token 生成，仅一次前向 pass + 分类头 → **显著降低** 计算量（尤其在 `max_new_tokens=0` 场景），对吞吐量有正向提升。  
  - **MIS 额外开销**：在 `score_and_pool` 中进行 `nonzero`、位置过滤及 per‑request 切分，时间复杂度与序列长度线性，实际开销在 0.1–0.5 ms（CPU）左右，对大 batch 仍可接受。  
  - **CPU‑GPU 同步**：`extend_seq_lens_cpu` 直接使用 CPU 上的整数列表，避免在循环内调用 `.item()`，已在实现中显式优化。  
  - **整体**：新增功能对已有生成模型几乎不产生额外负担，只在启用 MIS 或分类模型时才触发额外路径。  

- **安全考虑**  
  - **信息泄露**：分类模型返回的是 **logits**（或 softmax 后的概率），与生成模型返回的 token‑level log‑prob 同等级，未暴露模型权重或内部细节。  
  - **输入校验**：新增了对 `label_token_ids` 必要性的分支检查，仅在生成模型时强制要求；对 `items` 为空、类型错误等异常已在 HTTP 层统一返回 4xx。  
  - **配置风险**：如果用户误将常见文本 token 设为 `multi_item_scoring_delimiter`，会导致误切分，产生错误的评分结果；建议在文档中提醒使用 `tokenizer.eos_token_id` 或专用特殊 token。  

**⚠️ 潜在风险**  

1. **向后兼容性**  
   - 旧版调用仍能工作（`label_token_ids` 可省略），但如果用户的模型是 **生成模型** 而误传 `label_token_ids=None**，会触发 `ValueError`。  
   - `TokenizerManagerScoreMixin` 中的 `self.is_generation` 依赖于 `Engine` 初始化时的模型检查，若自定义模型未正确设置 `is_generation`，可能导致分支走错。  

2. **Delimiter 冲突**  
   - delimiter token 与输入文本出现冲突会导致 **额外、错误的分割点**，从而产生多余或错位的分数。  
   - 当 `delimiter_token` 为 0 或出现于文本开头，`score_and_pool` 会自动过滤首位 delimiter，但仍可能产生 **空 item** 评分。  

3. **多 GPU/多进程环境**  
   - `get_global_server_args()` 为全局单例，若在多进程（fork）模式下未进行复制，可能出现 **配置不一致** 的情况。  

4. **测试与 CI 负载**  
   - 新增的端到端服务器测试启动真实进程，CI 时长显著提升（≈2‑3 min），在资源受限的环境下可能导致超时。  

**💡 关注建议**  

| 建议 | 说明 |
|------|------|
| **文档强化** | 在用户手册中明确说明：<br>1️⃣ `label_token_ids` 只在 **CausalLM** 必填；<br>2️⃣ 推荐使用 `tokenizer.eos_token_id` 或自定义 **特殊 token** 作为 `multi_item_scoring_delimiter`，并提供示例检查函数。 |
| **参数校验** | 在 `Engine.__init__` 中校验

### [Feature] Add DFLASH speculative decoding support (#22077)
**SHA**: `f08726f` | 🔗 [查看提交](https://github.com/sgl-project/sglang/commit/f08726fd56c7ff6d8bd258f1545f98148fa4ef58)

**🎯 变更类型**：功能增强（新增 DFLASH 规格推理算法）

**⚡ 重要程度**：🔴 高  

**📋 变更摘要**  
- 在 SGLang 中实现全新的 **DFLASH** 规格推理（Speculative Decoding）算法，提供基于目标模型隐藏层特征的草稿模型加速路径。  
- 新增 `DFlashWorker`、`DFlashDraftModel`、配套的 `dflash_info / dflash_utils` 数据结构与工具函数；在调度器、模型执行器、模型运行器、服务器参数、层实现等多个核心模块中加入对应分支与校验逻辑。  
- 实现 **融合 KV 写入**（triton‑kernel）与 **自适应 mask token** 解析，实现无自定义 mask、非因果注意力的高效前向；同时对 KV‑cell 大小、分页缓存、CUDA‑graph、重叠调度等进行专门适配。

### 🔍 技术洞察  

| 维度 | 影响描述 |
|------|----------|
| **架构影响** | - **新增 SpeculativeAlgorithm.DFLASH**，在 `SpeculativeAlgorithm`、`SpecInputType` 中加入对应枚举，所有调度/执行路径需检查 `is_dflash()`。 <br>- **独立 Draft Worker**：`DFlashWorker` 复用目标模型的 KV‑pool（或在窗口模式下保持紧凑的局部映射），实现目标‑草稿双模型协同。 <br>- **模型层级捕获**：在目标模型 (`LlamaForCausalLM`, `Phi3ForCausalLM`) 中加入 `set_dflash_layers_to_capture`，用于捕获特定层的隐藏状态供 DFLASH 草稿投射。 <br>- **新 Draft Model**：`DFlashDraftModel` 只包含 attention+MLP，去掉嵌入与 LM‑head，使用目标模型的 embedding/lm_head。 <br>- **部署约束**：目前仅支持 **pp_size=1、dp=0、不开启 overlap‑schedule**，这在调度器初始化时通过 `ServerArgs` 强制校验。 |
| **性能影响** | - **加速潜力**：DFLASH 通过一次目标模型前向返回整个 block 的 logit 与隐藏特征，以块（默认 16 token）进行验证，理论可实现 **N‑fold（N≈block_size）** 加速，尤其在长上下文场景下收益显著。 <br>- **融合 KV 写入**：使用 Triton 实现的 `FusedKVMaterializeHelper` 将 K/V 投影、RMSNorm、RoPE 合并为一次 kernel，减少显存/算力占用；在支持的硬件上可比逐层 `kv_proj_only` 快 **30‑50%**。 <br>- **内存占用**：需要为 **草稿模型 KV‑cache** 额外分配与目标模型同等层数的 KV，内存需求约为 `target_cell_size * (1 + draft_num_layers/target_num_layers)`，通过 `scale_kv_cell_size_per_token_for_dflash` 自动放大 `cell_size`，并在 `model_runner_kv_cache_mixin` 中进行扩容。 <br>- **CPU‑GPU 同步**：为避免自定义 mask 的 graph‑capture 开销，默认在 `FlashInfer` 等后端 **不构建 custom_mask**，仅在后端不支持时回退。 <br>- **启用/禁用路径**：若硬件不支持 fused KV 或采样 verify，代码会回退到逐层投影或 greedy‑argmax，保证功能可用但可能牺牲加速。 |
| **安全/鲁棒性** | - **请求校验**：在 `scheduler.validate_dflash_request` 中显式拒绝 `return_logprob`、grammar‑constrained、regex、ebnf、structural_tag 等功能，防止未实现的特性导致未定义行为。 <br>- **Mask Token ID 校验**：`_resolve_mask_token_id` 对 `mask_token` 与 `mask_token_id` 做严格范围检查，确保不超出目标 vocab，避免 OOB 访问或 embedding‑size 不匹配的隐式错误。 <br>- **异常检测**：在多个入口（例如 `parse_dflash_draft_config`、`_init_fused_kv_helper`、`_append_target_hidden_fused`）加入 **显式异常抛出** 与 **日志警告**，有助于在错误配置时快速定位。 <br>- **内存安全**：KV‑allocation 针对分页模式使用 `_compute_paged_keep_slots`，确保释放的页对齐，防止碎片导致 OOM。 <br>- **覆盖旧特性**：`disable_overlap_schedule`、`enable_mixed_chunk=False` 等自动配置，防止旧的 overlap‑schedule 与 DFLASH 的不兼容导致竞争条件。 |
| **可维护性** | - **代码分层**：所有 DFLASH 相关逻辑集中在 `speculative/` 子目录，易于定位与隔离。 <br>- **特性开关**：通过 `ServerArgs.speculative_algorithm == "DFLASH"` 完全控制分支，保持对已有算法（EAGLE、STANDALONE 等）的向后兼容。 <br>- **单元/集成测试**：新增 `test/dflash/test_dflash.py` 包含端到端、不同 page_size、chunked‑prefill、CUDA‑graph 关闭等场景，覆盖主要路径。 <br>- **文档/CLI**：CLI 参数 `--speculative-dflash-block-size`、`--speculative-dflash-draft-window-size` 已加入帮助信息，便于用户自定义。 |

**🔍 技术洞察**  

- **架构影响**  
  - **抽象层**：引入 `flash_attn_varlen_func(..., ver=3|4)` 与 `flash_attn_with_kvcache(..., ver=3|4)`，实现了 “实现可插拔” 的架构，降低了对特定 kernel 实现的耦合。  
  - **统一缓存**：通过 `SGLANG_CACHE_DIR` 将模型权重、kernel lockfile 等统一放置，简化了路径管理并在多进程/多容器环境中避免冲突。  
  - **后端统一**：所有后端均改为直接调用 `sglang.jit_kernel.flash_attention`，删除了之前对 `sgl_kernel` 与 `flash_attention_v4` 的分支导入，提升代码可维护性。  

- **性能影响**  
  - **潜在加速**：社区提供的 `sgl-flash-attn3` 实现据称在 SM80/SM86/SM89/SM90 上更高效，默认启用可在这些硬件上获得约 5%~15% 的吞吐提升（具体取决于 head_dim、seq_len 等）。  
  - **回退路径**：若加载社区 kernel 失败，仍会使用原 `sgl‑kernel` 实现，保证功能可用但可能失去性能收益。  
  - **缓存命中**：在 Docker 镜像构建阶段预先下载并 lock，运行时直接读取本地 lockfile，消除首次启动时的网络依赖和编译开销。  

- **安全考虑**  
  - **供应链风险**：`kernels download` 会从 HuggingFace Hub 拉取代码，若 community repo 被篡改可能导致恶意代码执行。当前使用 lockfile（`kernels.lock`）锁定具体 commit，可在一定程度上降低风险。  
  - **环境变量泄露**：`SGLANG_CACHE_DIR` 默认指向用户主目录，若在多用户容器中使用需确认目录权限，以防缓存文件被未授权用户篡改。  
  - **异常处理**：加载失败时捕获异常并回退，防止因外部依赖不可用导致整个服务崩溃。  

**⚠️ 潜在风险**  

| 风险点 | 可能后果 | 触发条件 | 缓解措施 |
|--------|----------|----------|----------|
| lockfile 与实际 repo 不同步 | 运行时出现 API 不匹配或运行错误 | 社区 repo 更新但未同步 lockfile | CI 中强制在每次发布前重新生成 lockfile；在启动时校验 lockfile 哈希。 |

## 中等深度分析

| 维度 | 说明 | 风险 / 注意点 |
|------|------|---------------|
| **实现思路** | 通过 `get_world_group()` 获取当前进程所在的通信组，选取 `ranks[0]` 为源 rank，仅在该 rank 上写入 warm‑up 图片（使用 `self.server_args.input_save_path` 或临时目录），随后用 `broadcast_pyobj` 把 `{"input_path": ..., "error": ...}` 同步给其它 rank。 | 该方案依赖所有 rank 能够访问 **同一文件系统路径**。如果 `input_save_path` 指向本地磁盘而不是共享存储，非源 rank 会收到一个在本地不存在的路径，后续读取会失败。 |
| **文件系统兼容性** | ① 若用户未显式配置 `input_save_path`，代码会在每个节点创建独立的临时目录 (`tempfile.mkdtemp`)；随后只有源 rank 会写文件，其他 rank 收到的路径指向 **源节点的临时目录**，在本地不可见。 ② 老代码直接在每个节点的 `outputs/uploads` 写入，虽然产生冗余文件，但至少每个节点都有本地可读的副本。 | 在没有共享文件系统的环境（单机多卡、或者多机 NFS 未挂载）下，warm‑up 将报 “文件不存在” 或 “路径为空”。建议在文档或代码注释中强调：*必须使用共享文件目录*（如挂载的 `/shared/sglang_input`），或者在同步时把图片内容也广播，让每个 rank 自己写本地副本。 |
| **依赖导入** | 新增的 `broadcast_pyobj`、`get_world_group` 需要相应的导入语句；当前 diff 中只加入 `from sglang.multimodal_gen.runtime.distributed import get_world_group`，但没有 `broadcast_pyobj` 的导入。若该函数位于 `sglang.utils` 或其他模块，缺少导入会导致运行时 `NameError`。 | 检查项目中 `broadcast_pyobj` 的实际定义位置并补全导入。 |
| **异常处理** | 将源 rank 的异常通过 `broadcast_pyobj` 传播，所有 rank 都会收到相同的错误信息并抛出统一的 `RuntimeError`。这在调试时很有帮助，但会导致 **所有 rank** 同时退出，即使只有源 rank 产生错误。 | 如有需求，可在非源 rank 上仅记录日志而不直接退出，防止因单节点磁盘错误导致全局服务不可用。 |
| **异步调用** | 仍使用 `asyncio.run(save_image_to_path(...))` 在同步上下文中执行 I/O，保持与旧实现一致。 | 运行时仍会阻塞当前线程，若 `save_image_to_path` 实现较慢（解码、写磁盘），会延迟整个 warm‑up 流程。可考虑改为异步 `await`，但改动成本较大。 |
| **代码可读性** | 将 warm‑up 图片路径的获取抽离为私有方法 `_prepare_shared_warmup_image_path`，逻辑划分更清晰。日志信息略作简化。 | 方法名已足够直观，建议在函数内部添加简要 docstring，说明“仅在 src rank 生成图片并广播路径”。 |
| **向后兼容** | 当任务不需要图像输入（如纯文本任务），仍走原来的 `else` 分支，保持兼容。 | 无影响。 |
