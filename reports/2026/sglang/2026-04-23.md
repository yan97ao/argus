### [CPU] [Quantization] Add GPTQ/AWQ 4bits quantization support for CPU  (#22685)
**SHA**: `ad0fc88` | 🔗 [查看提交](https://github.com/sgl-project/sglang/commit/ad0fc888103869858a6e02cbc47d8443e6b2196f)

**🎯 变更类型**：功能增强  

**⚡ 重要程度**：🔴高  

**📋 变更摘要**：  
- 为 CPU 端新增 4bit GPTQ 与 AWQ 量化支持，扩展了 `sglang` 的量化配置、线性层实现及 MoE（Mixture‑of‑Experts）路径。  
- 引入 `CPUQuantAlgo`、`CPUQuantMethod` 枚举，重新设计权重预处理 (`_amx_process_weight_after_loading`) 以兼容普通打包与基于 AMX 的 packed‑weight 格式。  
- 在 C++ kernel 中实现了 AWQ 与 GPTQ 的 4bit 解压、打包逻辑，并同步更新 Python‑C++ 接口。  

**🎯 影响范围**：  
- `python/sglang/srt/configs/update_config.py`、`sglang/srt/layers/*`（amx_utils、linear、quantization）  
- 新增 `awq_cpu.py`、`gptq_cpu.py` 两个量化配置实现  
- C++ 侧 `sgl-kernel/csrc/cpu/*`（枚举、`int4pack`、`convert_weight_packed_scale_zp`）  
- 单元测试 `test/srt/cpu/*`（gemm、moe、utils）  

**🔍 技术洞察**：

- **架构影响**  
  - **量化抽象层扩展**：在 `quantization/__init__.py` 中新增 `CPU_QUANTIZATION_METHODS`，并在 `get_quantization_config` 中根据 `is_cpu()` 与 `cpu_has_amx_support()` 动态切换，形成了 CPU‑Only 量化分支。  
  - **统一权重预处理**：`_amx_process_weight_after_loading` 现在接受 `qweight_packed_method` 参数，分别走普通卷积/线性预打包或专门的 AWQ/GPTQ 打包路径，降低了代码分支散落在各层实现的风险。  
  - **新增枚举 `CPUQuantAlgo`**：在 C++ 与 Python 之间统一量化算法标识，避免硬编码字符串。  
  - **MoE 支持**：为 AWQ/GPTQ 在 MoE 场景下提供了 `*_MoEIntelAMXMethod`，在 `fused_experts_cpu` kernel 中使用 `CPUQuantMethod.INT4_W4A8` 统一调度。  

- **性能影响**  
  - **内存/带宽**：4bit 量化将权重大小压缩至原始的 1/8，显著降低显存/内存带宽需求。  
  - **计算加速**：在支持 Intel AMX 的 CPU 上，`int4_scaled_mm_cpu` 与 `fused_experts_cpu` 利用 VNNI/AMX 微指令实现矩阵乘法，理论上可提升 2‑3×吞吐。  
  - **转换开销**：`convert_weight_packed_scale_zp` 现在会在运行时根据 `quant_method_4bit` 完成解包/重新打包，额外的 CPU 循环（尤其是 `unpack_4bit_to_32bit_signed`）在模型首次加载时可能产生一次性 CPU 负载。  
  - **回退路径**：若 `cpu_has_amx_support()` 为 False，仍会走原有 `amx_process_weight_after_loading`（仅做转置），不会出现功能回退错误。  

- **安全考虑**  
  - 代码仅在本地 CPU 上执行，未引入外部 I/O，安全风险极低。  
  - 需要注意 **整数溢出** 与 **位移操作** 的安全性：所有位移均使用 `int64_t` 索引，且在 C++ 中已有 `TORCH_CHECK` 断言，基本稳妥。  

**⚠️ 潜在风险**：

| 类别 | 风险点 | 可能影响 |
|------|--------|----------|
| 兼容性 | `may_get_weight_block_size` 现在返回 `list`（`[group_size]`）而旧实现返回单值。若下游仍期望 `int`，可能导致类型错误。 | 启动阶段报错或错误的 MoE padding 计算。 |
| 运行时 | `cpu_has_amx_support()` 检测不严（仅在运行时查询一次），在部分虚拟化/容器环境中可能误判为支持，导致走 AMX 路径而实际硬件不支持，触发非法指令异常。 | 程序崩溃。 |
| 性能 | GPTQ 解包循环采用逐元素 Python‑style C++ 循环（`for`），在大模型（数 GB 权重）加载时可能耗时数秒甚至更久。 | 启动延迟明显。 |
| 正确性 | GPTQ 解包后对 zero 的处理为 `+1`（`unpacked_qzeros + 1`），若模型使用不同的 zero 编码（如 `zero_point = 0`），会产生偏差。 | 推理数值误差。 |
| API 兼容 | `get_quantization_config` 在 CPU 上若 `cpu_has_amx_support()` 为 True，但请求的量化方法不在 `CPU_QUANTIZATION_METHODS`，会抛 `ValueError`，破坏原有 “fallback to CUDA” 行为。 | 用户在非‑AMX CPU 上仍想使用 AWQ/GPTQ 时无法使用。 |
| 测试覆盖 | 新增的 `awq_cpu.py`、`gptq_cpu.py` 均未提供完整的异常路径测试（如 group_size=-1、非 4bit、缺失 `group_size`），可能隐藏隐藏错误。 | 维护期出现回归。 |

### [AMD] Use bpreshuffle FP8 blockscale GEMM to replace ABScale GEMM (#23319)
**SHA**: `cd459af` | 🔗 [查看提交](https://github.com/sgl-project/sglang/commit/cd459af4e272ea7449e1a61e7999a0a0aa55e3d1)

**🎯 变更类型**：性能优化 / 功能增强  
**⚡ 重要程度**：🟡 中  
**📋 变更摘要**：在 FP8 量化路径上引入 *bpreshuffle*‑based FP8 Block‑Scale GEMM，以取代原先的 ABScale GEMM，实现对 AMD GFX95 系列显卡的更佳算子调度。代码在 `fp8.py` 与 `fp8_utils.py` 中加入了对新算子的条件导入、调度选择以及权重‑shuffle 的补救逻辑。  

**🎯 影响范围**  
- `sglang/srt/layers/quantization/fp8.py`（权重加载后可能触发权重 shuffle）  
- `sglang/srt/layers/quantization/fp8_utils.py`（实现 `aiter_w8a8_block_fp8_linear`，新增 `gemm_a8w8_blockscale_bpreshuffle` 的调用路径）  
- 依赖 `aiter` 包的 AMD‑GPU（尤其是 GFX950/95）运行时。

**💡 关注建议**  

1. **标记与文档**  
   - 新增的全局变量 `_use_aiter_gfx95` 只在本文件中使用，建议在 `fp8_utils.py` 明确注释其默认值与生效条件（例如 `os.getenv` 或硬件检测），防止误用。  
   - 在 README 或 release notes 中补充“在 AMD GFX95 系列上自动切换为 bpreshuffle‑GEMM”，并说明在不满足条件时仍会回退到原实现。

2. **分支覆盖**  
   - 现在有四条关键分支：`_use_aiter_gfx95` + `use_triton` / `!use_triton`，以及 `input_scale` 有无。务必在 CI 中加入对应的矩阵尺寸（如 `n,k`）组合测试，以捕获 `transpose_scale` 与 `x_scale` 转置的潜在维度不匹配。  
   - `shuffle_weight` 仅在 `use_triton` 为 `False` 且 `use_aiter_triton_gemm_w8a8_tuned_gfx950` 返回 `False` 时执行，建议在单元测试里强制触发该路径，验证权重值在 shuffle 前后保持数值等价。

3. **资源管理**  
   - `layer.weight.copy_(t)` 后立即 `del t`，但 `t` 仍可能被计算图捕获。若在 torchscript / tracing 环境下使用，建议使用 `with torch.no_grad():` 包裹，以避免不必要的梯度记录和显存泄漏。

4. **后向兼容**  
   - 旧的 `gemm_a8w8_blockscale` 已被保留，仅在 `use_triton=False` 时切换到 `gemm_a8w8_blockscale_bpreshuffle`。如果未来删除 `gemm_a8w8_blockscale`，请同步更新 `use_triton` 的默认行为，以免出现 “找不到符号” 的运行时错误。

5. **性能基准**  
   - 变更声明为“Replace ABScale GEMM”。建议在 PR 中提供对比基准（如吞吐量、延迟）以及在不同显卡（AMD vs NVIDIA）上的回退表现，帮助用户判断是否开启该特性。

总体而言，改动紧密围绕 AMD‑GPU 的 FP8 优化，逻辑清晰，但需完善文档、测试覆盖以及显式的硬件/环境检测，以防在非目标平台或未满足条件的环境下产生不可预期的行为。

### [bug fix] fix: detect FP8 weights from safetensors header instead of ass… (#23414)
**SHA**: `2ef1a21` | 🔗 [查看提交](https://github.com/sgl-project/sglang/commit/2ef1a21d5ebbe794d05a0defc06bfa999a4da536)

**变更类型**：Bug 修复  
**核心改动**  
1. **`server_args.py`**：在 DeepSeek‑V3/R1 模型的默认量化判断中，由原来仅凭模型架构名称直接设为 `fp8`，改为先调用新工具函数 `has_fp8_weights_in_checkpoint` 检测 checkpoint 中是否真的包含 FP8 expert 权重。若检测不到则保持 BF16，日志给出明确提示。  
2. **`utils/common.py`**：新增 `has_fp8_weights_in_checkpoint`，通过读取 `model.safetensors`（或其 index）头部的 JSON 元数据，仅检查 dtype 为 `F8_E4M3` 的 expert weight 张量，从而实现“仅凭 header 判断”。异常统一捕获返回 `False`，不影响启动流程。

**影响范围**  
- **模型启动路径**（`sglang/srt/server_args.py`）的量化默认推断逻辑。  
- **公共工具**（`sglang/srt/utils/common.py`）提供新函数，供其它模块潜在复用。  

**风险 & 注意点**  
- 只检查首个 shard（`next(iter(...))`），多 shard 场景若 FP8 权重分布在其他文件可能误判。

- 若模型目录仅有远程或压缩文件（未解压），函数会返回 `False`，可能导致 BF16 被误用。  

**建议**  
- 为该函数添加单元测试，覆盖单文件、分片、缺失文件、异常路径等情况。  
- 考虑在检测成功后缓存结果，避免重复 I/O。  
- 更新文档，说明 DeepSeek‑V3/R1 的量化默认行为已改为 “根据 checkpoint 实际 dtype 判定”。  
- 如有其它模型在同一架构下采用 BF16，建议在模型配置里显式声明 `quantization_config`，防止依赖检测逻辑。
