### [weight checker] refactor: add precision branch; allow ULP quant err; used chunked compare (#28974)
**SHA**: `e1ca92a` | 🔗 [查看提交](https://github.com/sgl-project/sglang/commit/e1ca92a7fd6bbbd5e20d88100784844dc7d4b5d8)

**🎯 变更类型**：功能增强 / 重构 / 性能优化  
**⚡ 重要程度**：🔴高  
**📋 变更摘要**：  
- 为权重校验（WeightChecker）引入 *precision 分支*，支持在 FP8、FP4 等量化模型下进行 **chunked**（分块）对比，并可通过 `allow_quant_error` 标记容忍量化 ULP 误差。  
- 重构 `check_weights` 接口，向下游调度器、模型执行器、IPC 请求等传播该标记。  
- 新增 `weight_checker_comparator` 模块，实现 **可比对象**（ComparableWeight）抽象、块量化解压、误差统计以及量化方法 → 可比较类映射。  
- 相应地更新 IO 请求结构、调度器组件、模型执行器以及单元测试，提升代码可维护性并降低内存峰值。

**🎯 影响范围**：  
- `sglang/srt/managers/io_struct.py`（新增 `allow_quant_error` 字段）  
- `sglang/srt/managers/scheduler_components/weight_updater.py`（传递新标记）  
- `sglang/srt/model_executor/model_runner.py`（新增参数）  
- `sglang/srt/utils/weight_checker.py`（核心逻辑改写）  
- `sglang/srt/utils/weight_checker_comparator.py`（全新实现）  
- 单元测试文件 `test/registered/unit/utils/*`（覆盖新行为）  

**🔍 技术洞察**：

| 维度 | 影响分析 |
|------|----------|
| **架构影响** | - 在权重校验层加入 **可比对象抽象**，将量化细节从 `WeightChecker` 中抽离，使得后续可以轻松扩展至其他量化方案（如 ModelOpt‑Fp4）。<br>- `check_weights` 接口向上传递 `allow_quant_error`，保持向后兼容（默认 `False`），不影响现有调用者。<br>- 新增 `QuantizedWeight` 数据结构，使调度器只需要一次遍历即可构建“量化计划”，减少重复逻辑。 |
| **性能影响** | - **分块比较（Chunked Compare）**：通过 `CHUNK_NUMEL`（默认 64M 元素）分批加载至 GPU，显著降低一次性对全模型进行解量化比较时的显存占用，尤其在大模型（>30B）场景下可避免 OOM。<br>- 误差统计在 **GPU 上** 完成，仅在每块结束时聚合统计数据，整体耗时与全模型一次性比较相当，甚至因降低显存压力而提升整体吞吐。<br>- 新增 `allow_quant_error` 选项在误差容忍范围内直接把错误计入 **info** 而非 **error**，避免不必要的异常抛出。 |
| **安全考虑** | - 变更仅涉及模型权重的内部校验与统计，不涉及外部 I/O、网络或权限控制，安全风险极低。<br>- 唯一潜在安全点是 **误报**：若 `allow_quant_error=True` 被误用，会掩盖真实的权重不一致（可能导致模型损坏或未检测到的攻击），需在文档和接口使用约束中明确警告。 |
| **可维护性** | - 通过 **抽象层**（ComparableWeight、RawComparable、Fp8BlockComparable）把量化耦合点集中，后续增添新量化方式只需实现对应 `ComparableWeight` 子类并在 `select_comparable_weight` 中注册。<br>- `weight_checker` 逻辑被拆分为 *构建检查条目*、*比较* 两大步骤，单元测试覆盖度提升，代码可读性大幅改善。 |
| **兼容性** | - 保持向后兼容：默认 `allow_quant_error=False`，旧调用不受影响。<br>- 新增字段 `allow_quant_error` 在 `CheckWeightsReqInput` 中默认为 `False`，旧的 IPC 请求仍能正常解析。 |

**⚠️ 潜在风险**：

1. **误用 `allow_quant_error`**：开启后即使权重实际不相等，只要误差在 ULP 容忍范围内也不抛异常，可能导致隐蔽的模型损坏未被检测。  
2. **新比较实现的边界条件**：  
   - 对非块量化（如稀疏、混合精度）模型仍走 `RawComparable`，若未来引入新量化方式但未在 `select_comparable_weight` 中注册，会默认进行位级比较，可能产生误报。  
   - `CHUNK_NUMEL` 过小会导致大量 kernel 启动开销；过大则失去显存保护，需要在部署环境中调优。  
3. **跨进程/跨机器的权重快照**：`_snapshot` 仍使用原始张量保存，若模型使用 **参数共享** 或 **梯度累加**，快照与实际权重不一致的风险未改变。  
4. **测试覆盖局限**：当前单元测试主要覆盖 FP8 路径，未覆盖 ModelOpt‑Fp4、MxFP8、3‑dim expert 张量等新量化方案，后续需要补全。  

**💡 关注建议**：

- **上线前灰度验证**：在正式集群开启 `allow_quant_error` 前，可在小流量/测试集群先运行一次完整的 `check_weights`，确认误差统计与预期一致。  
- **监控显存峰值**：部署时记录 `WeightChecker` 的显存占用，若仍出现 OOM，适当调低 `CHUNK_NUMEL`（例如每块 < 32 M 元素）。  
- **文档与使用指引**：在 API 文档中明确 `allow_quant_error` 的语义、使用场景及风险提示，建议仅在量化模型的**容忍误差校验**时打开。  
- **扩展量化支持**：后续若引入新量化方法（如 INT8、FP4），实现对应 `ComparableWeight` 子类并在 `select_comparable_weight` 中注册，避免回退至位级比较导致误报。  
- **CI 增加全量化路径**：在 CI 中加入 ModelOpt‑Fp4、MxFP8 等量化方案的权重校验测试，保证抽象层的完整性。  
- **审计日志**：若业务层对权重校验异常敏感，可在 `WeightChecker` 中加入 **审计日志**（包括 `allow_quant_error` 标记、误差统计）供运维审计。  

---  

总体来看，此次改动显著提升了 **权重校验的可扩展性、显存友好性与误差容忍能力**，对大模型量化部署尤为重要。但需要在生产环境中严格控制 `allow_quant_error` 的使用范围，并补全对其它量化方案的测试与实现，以确保功能的稳健性。

### [misc] Use --cuda-graph-max-bs-decode in tests, examples, and docs (#29591)
**SHA**: `909123d` | 🔗 [查看提交](https://github.com/sgl-project/sglang/commit/909123ddb854f64e1a96ecbd55a29e7f6e6fe148)

**🎯 变更类型**：功能增强（CLI 参数重命名 & 兼容性升级）  
**⚡ 重要程度**：🔴高  

**📋 变更摘要**  
- 将原有的 `--cuda-graph-max-bs` 参数统一更名为 `--cuda-graph-max-bs-decode`，并在 **server_arguments** 中声明 `--cuda-graph-max-bs` 为 **已废弃的别名**。  
- 文档、示例、测试脚本以及部分代码（如参数映射）全部同步更新，以统一“decode 阶段”图捕获的概念。  
- 目的在于消除 **prefill** 与 **decode** 两阶段对同一标志的不明确表达，提升参数可读性并降低误用风险。

**🎯 影响范围**  
- **核心模块**：`sglang/launch_server` 参数解析、`server_args.py`、`sglang_server` 启动脚本。  
- **文档体系**：`benchmark/`、`docs_new/`、`README.md`、MDX/MD 文档、Jupyter notebooks、代码片段（React/JSX）等全部示例。  
- **测试套件**：所有 `python/sglang/test/*`、`test/manual/*`、`test/registered/*` 中引用的旧参数均已替换。  
- **部署脚本 & 示例**：Docker、K8s、Multi‑node、Ascend/NPU 平台的示例配置文件。  

**🔍 技术洞察**  

- **架构影响**  
  - **参数层面的抽象统一**：原本 `--cuda-graph-max-bs` 同时用于 **prefill** 与 **decode**，在多阶段调优时容易产生冲突。改名后，仅指代 **decode** 阶段的 CUDA Graph 捕获大小，业务方可在文档中明确区分 `--chunked-prefill-size` 与 `--cuda-graph-max-bs-decode`，提升整体配置模型的可组合性。  
  - **向后兼容**：`--cuda-graph-max-bs` 仍保留为 *deprecated alias*，解析层已实现自动映射，保证已有 CI 与生产脚本不至于瞬间失效。  

- **性能影响**  
  - 参数本身不改变图捕获逻辑，性能保持不变。  
  - 由于文档和示例明确了 “decode‑only” 的意义，调优人员更易在高并发或大 TP 场景下提升 **max‑running‑requests** 与 **cuda‑graph‑max‑bs-decode** 的匹配度，间接提升吞吐量与显存利用率。  

- **安全考虑**  
  - 参数更名不涉及底层计算或内存安全。  
  - 需要确保在 **参数解析** 中正确处理废弃别名，防止出现 “未识别参数” 错误导致服务启动失败，从而形成可用性（DoS）风险。当前已在 `server_arguments.mdx` 中标注别名并在代码中保留映射，风险可控。  

**⚠️ 潜在风险**  
1. **脚本/CI 失效**：仍使用旧参数且未启用别名映射的自定义启动脚本会报错，导致 CI/生产部署中断。  
2. **文档同步遗漏**：部分第三方工具或社区博客可能基于旧参数编写示例，使用时会出现 “unknown flag”。  
3. **混淆误用**：新老参数共存期间，若同时出现在同一命令行中，解析优先级需明确（已实现 “旧别名覆盖新值”，但仍可能导致配置冲突）。  

**💡 关注建议**  

- **立即检查 CI/CD**：搜索仓库（包括子模块）中出现 `"--cuda-graph-max-bs"` 的地方，确认是否已被替换或已依赖别名。对自定义脚本加入 `--cuda-graph-max-bs-decode` 明确写法。  
- **完善迁移指南**：在 **CHANGELOG** 或 **升级文档** 中加入 “从 `--cuda-graph-max-bs` 到 `--cuda-graph-max-bs-decode` 的迁移步骤”，并注明别名将在 **v0.6.x** 之后移除。  
- **添加单元测试**：确保启动脚本在只传递旧别名、只传递新标志、以及两者同时出现时，行为符合预期（新标志优先，旧别名发出警告）。  
- **监控启动日志**：在服务器启动时输出 “`--cuda-graph-max-bs` is deprecated, please use `--cuda-graph-max-bs-decode`” 警告，帮助用户及时迁移。  
- **评估废弃时机**：建议保持别名至少 **两个主版本周期**（约 6‑12 个月）后再彻底删除，以降低生态破坏。  

---  

**结论**：此次改动主要是 **可维护性与易用性提升**，对系统功能与性能没有直接影响，但在大规模 CI 与生产部署环境中，需要做好 **别名兼容** 与 **文档同步**，以避免因参数误用导致的服务不可用。及时完成迁移检查与警告机制，可确保平滑过渡并提升后续调优效率。

### [AMD] Sgl-data mount opt-in (#29661)

**⚡ 重要程度**：🟡 中  

**📋 变更摘要**  
- 在 AMD 系列 CI workflow（nightly‑test‑amd、nightly‑test‑amd‑rocm720、pr‑test‑amd‑rocm720）中统一注入环境变量 `ENABLE_CACHE_HOST=1`。  
- `scripts/ci/amd/amd_ci_start_container.sh` 新增对 `ENABLE_CACHE_HOST` 的解析：开启时将宿主目录 `/home/runner/sglang-data` 挂载至容器的 `/sgl-data`，并预创建常用缓存子目录；关闭时保持容器本地临时目录。  

**🎯 影响范围**  
- CI 相关脚本：`scripts/ci/amd/amd_ci_start_container.sh`。  
- 所有 AMD GPU CI 工作流（nightly‑test‑amd、nightly‑test‑amd‑rocm720、pr‑test‑amd‑rocm720）。  

**💡 关注建议**  
1. **目录存在性**：开启挂载依赖宿主目录 `/home/runner/sglang-data` 必须预先创建，否则 CI 将因 `exit 1` 直接失败。建议在 repo‑level README 或 CI 文档注明该目录的准备方式（如使用 GitHub Actions cache）。  
2. **默认安全**：目前默认值 `0`（不挂载），保持向后兼容；若以后改为默认开启，需要评估磁盘配额和并发写入冲突。  
3. **清理策略**：挂载的缓存会在 runner 生命周期结束后自动清除，但长期累计可能占用 GitHub Actions 磁盘。可考虑在 workflow 结束时加入 `actions/cache@v3` 的 “save‑cache” 步骤，以持久化或清理。  
4. **日志可观测**：脚本已在挂载前打印提示信息，建议在 CI 结果页面加入 `echo` 输出，以便快速定位是否启用了持久缓存。  
5. **兼容性测试**：在本地或自托管 runner 上手动跑一次 AMD CI，确保 `docker exec` 创建子目录的权限和路径在容器内部可写。  

总体而言，此次改动仅影响 CI 环境，不会影响用户运行时行为。若按照上述检查点完成验证，风险可控，建议合并。

### Add Qwen3 MoE tests for PP compatibility with CP and DP (#29640)
