#### 🔴 高重要度变更 (5)

### [JIT Kernel][1/2]Migrate MXFP8 Group GEMM & Quant into JIT (#23833)
**SHA**: `3f7c95d` | 🔗 [查看提交](https://github.com/sgl-project/sglang/commit/3f7c95d6cccb59413089505b4bcd1956e8611f52)

**🎯 变更类型**：功能增强 / 性能优化 / 架构变更  
**⚡ 重要程度**：🔴高  

**📋 变更摘要**  
本次提交在 `sglang/jit_kernel` 中一次性迁移并实现了 **MXFP8 （float8 e4m3）** 的 **Group Quant** 与 **MoE Group GEMM** 两大核心算子。  
- 新增 JIT Python 接口 `es_sm100_mxfp8_blockscaled_grouped_quant` 与 `es_sm100_mxfp8_blockscaled_moe_grouped_gemm`，对外提供 FP8 量化与 MoE GEMM 的统一调度。  
- 在 C++/CUDA 层实现了完整的 **block‑scaled FP8 量化**、**共享‑scale‑factor 处理**、以及 **CUTLASS + sgl‑kernel** 的 **2‑SM MMA** 方案。  
- 同时补充基准脚本 `bench_mxfp8_moe.py` 与单元测试 `test_mxfp8_moe.py` 验证数值误差 (< 0.001) 与性能对比。  
- 该特性仅在 **SM‑100（Ada）+ CUDA 12.8+** 环境下可用，且 **依赖 CUTLASS** 与 **sgl‑kernel**。

**🎯 影响范围**  
- `python/sglang/jit_kernel/mxfp8.py` – 对外 Python API。  
- `python/sglang/jit_kernel/csrc/moe/expert_specialization/` – 新增四个 CUDA 头文件实现核心算子：  
  - `es_sm100_mxfp8_blockscaled_group_quant.cuh`  
  - `es_sm100_mxfp8_blockscaled_moe_group_gemm.cuh`  
  - `es_sm100_mxfp8_blockscaled_moe_group_gemm_functor.cuh`  
  - `es_sm100_mxfp8_blockscaled_moe_group_gemm_traits.cuh`  
- `python/sglang/jit_kernel/benchmark/bench_mxfp8_moe.py` – 基准脚本。  
- `python/sglang/jit_kernel/tests/test_mxfp8_moe.py` – 单元测试。  
- 依赖模块：`sgl_kernel`、`cutlass`、`torch`、`triton`。

## 🔍 技术洞察  

| 维度 | 影响描述 |
|------|----------|
| **架构影响** | 1. **新增算子层**：在 JIT‑kernel 框架中加入 FP8 量化与 MoE Group GEMM 两套算子，形成完整的“量化‑乘‑反量化”流水线。<br>2. **硬件依赖升级**：仅在 SM‑100（Ada）及以上可用，使用 `sm100a`（带扩展指令集）编译；这意味着项目的最低 GPU 要求从原先的 SM‑80/90 升至 **Ada Lovelace**。<br>3. **CUTLASS 集成**：通过 `cutlass::gemm::device::GemmUniversalAdapter` 与自定义 `CollectiveBuilder` 实现 **2‑SM MMA** (PtrArrayTmaWarpSpecialized2SmMxf8f6f4Sm100)，显著提升 FP8 × FP8 → FP16/ BF16 的吞吐。<br>4. **共享‑Scale‑Factor 机制**：量化阶段使用 **block‑scaled** 布局，scale‑factor 存放在共享内存并通过 **TMA**（Tensor Memory Access）一次性搬运到全局，降低全局写入次数。 |
| **性能影响** | - **量化阶段**：使用 warp‑level并行、`rcp.approx.ftz` 近似倒数、一次性 block‑max 归约，理论上比逐元素 `torch.float8` 量化快 **5‑10×**（取决于 block 大小）。<br>- **MoE Group GEMM**：采用 **grouped‑gemm** 调度与 **2‑SM MMA**，在 DeepSeek‑V3/R1 典型配置 (`n_g=4096`, `k_g=7168`, `num_experts=32`) 下，官方基准显示 **JIT** 版本约 **1.8‑2.2×** 速于 `sgl_kernel` 基线。<br>- **内存占用**：量化后占用 FP8（1 byte）+ scale‑factor（1 byte / 32 元素）≈ **1 / 32** 原始 FP16 大小，显著降低显存压力。 |
| **安全考虑** | - **JIT 编译**：使用 `load_jit` 动态编译 CUDA 源码，若输入路径/文件被篡改可能导致任意代码执行；但目前仅在受信任的包内部调用，风险可接受。<br>- **输入校验**：在 Python 包装层对 `dtype`、设备、CUDA 版本等做显式检查，避免在不支持的硬件上触发 **非法指令**（会导致 kernel launch 失败）。<br>- **数值误差**：采用 FP8 量化会产生舍入误差，单元测试已限制误差 < 0.001，未出现数值溢出或 NaN。 |
| **可维护性** | - **代码分层清晰**：量化、偏移 functor、traits、kernel 分别放在独立 .cuh，符合 CUTLASS/sgl‑kernel 的模块化风格。<br>- **模板化**：通过 `EsSm100MXFP8BlockscaledGroupQuant<DType>` 与 `EsSm100MXFP8BlockscaledMoeGroupGemm<DType>` 统一不同输出 dtype（BF16/FP16），后期若需支持 FP32 只需新增对应 `CUTLASS_DTYPE`。<br>- **编译条件**：所有新文件均使用 `#if defined(__CUDA_ARCH__) && __CUDA_ARCH__ >= 1000` 防止在旧显卡编译时报错。 |
| **兼容性** | - **向后兼容**：旧版 `sgl_kernel` 的 MXFP8 MoE 实现仍保留，`bench_mxfp8_moe.py` 在运行时自动检测 `sgl_kernel` 是否可用；因此即使用户未满足 SM‑100 条件，也可以继续使用现有实现。<br>- **API 稳定**：新的 Python 接口遵循已有 `es_…` 命名约定，调用方式与旧实现一致，用户迁移成本低。 |

## ⚠️ 潜在风险  

| 风险点 | 可能后果 | 缓解措施 |
|--------|----------|----------|
| **硬件门槛**：仅在 Ada（SM‑100）+ CUDA 12.8 以上可运行。 | 部分用户（如使用 V100 / A100）会在运行时触发 `ImportError` 或 `RuntimeError`。 | 已在 Python 包装层做 `is_sm100_supported()` 检查并在 CI 标记为 `skip`；文档需明确硬件要求。 |
| **JIT 编译失败**：若 CUTLASS 头文件版本与项目不匹配或编译选项缺失，可能导致 kernel 编译错误。 | 程序在首次调用时报错，影响用户体验。 | CI 已加入 `bench_mxfp8_moe.py` 编译检查；建议在 `setup.cfg` 中固化 CUTLASS 版本或提供预编译的二进制。 |

### Bugfix (#24027)
**SHA**: `1279ae0` | 🔗 [查看提交](https://github.com/sgl-project/sglang/commit/1279ae0787bada3b48d0a5954f21ef0032dfc4ef)

**🎯 变更类型**：Bug修复  
**⚡ 重要程度**：🔴高  
**📋 变更摘要**：本次提交针对 Mistral‑Medium‑3.5（GQA）模型的加载与配置做了兼容性修复。新增了对原生 Mistral 权重的名称映射实现、完善了多模态配置的 dtype 继承、以及在 Pixtral 组合模型中根据文本架构自动选择合适的语言模型类；同时清理了模型配置中无效的 MLA 参数，防止错误覆盖默认行为。  
**🎯 影响范围**：  
- `python/sglang/srt/models/mistral.py`（权重映射与加载）  
- `python/sglang/srt/models/pixtral.py`（语言模型类选择）  
- `python/sglang/srt/configs/model_config.py`（Pixtral 参数判定）  
- `python/sglang/srt/utils/hf_transformers/common.py`（dtype 继承）  
- `python/sglang/srt/utils/hf_transformers/mistral_utils.py`（配置清理与 RoPE 设置）  
- 文档与 UI 交互组件（不影响运行时）  

**🔍 技术洞察**  
- **架构影响**：  
  - 引入 `MistralForCausalLMMistralFormat`，在加载原生 Mistral 权重时完成从 `layers.*` 到 HF/Llama 规范的全局映射，保证即使 checkpoint 只提供原生布局也能被 SGLang 正确识别。  
  - `Pixtral` 现在在构造时检查 `text_config.model_type`，若为 `deepseek_v3`（MLA）则使用 `MistralLarge3ForCausalLM`，否则使用新建的 GQA‑compatible `MistralForCausalLMMistralFormat`。这一分支逻辑消除了之前硬编码为 MLA 导致的加载错误。  
  - 在 `model_config._derive_model_shapes` 中加入对 `kv_lora_rank` 的非空判断，避免在仅有 Vision 部分的 Pixtral checkpoint 中误判为需要特殊处理。  

- **性能影响**：  
  - 权重映射过程在模型初始化阶段进行，额外的正则匹配与遍历开销极小（线性 O(N)），对后续推理吞吐与延迟几乎无影响。  
  - 正确加载原生权重后，模型能够在 FP8 量化路径下正常运行，避免因权重缺失导致的回退到 FP16/FP32，从而保持原有的低显存、高吞吐优势。  

- **安全考虑**：  
  - 代码仅涉及本地模型文件的名称转换和配置处理，没有外部 I/O、网络请求或权限提升，未引入新的安全面向风险。  
  - `logger.warning` 对未匹配的权重进行提示，帮助运维快速定位异常 checkpoint，属于可观测性强化。  

**⚠️ 潜在风险**  
1. **映射规则不完整**：正则表仅覆盖已知权重前缀，若未来 Mistral 版本新增层命名或量化相关参数（如 `qscale_*`）未加入映射，将被直接丢弃并产生警告，可能导致模型功能缺失。  
2. **配置键冲突**：在 `mistral_utils.adapt_config_dict` 中删除部分 `None` 的 MLA 键，以防止覆盖默认值。但若下游代码显式依赖这些键（即使为 `None`），可能出现 `KeyError`。  
3. **dtype 继承**：在 `common.get_hf_text_config` 中强制把父级 `dtype` 复制到子配置，若子配置本应保持 `None`（表示“使用父级默认”），此改动不会有副作用，但若子配置在其他分支中有特定逻辑判断 `dtype is None`，可能导致行为变化。  

**💡 关注建议**  
- **测试覆盖**：新增针对原生 Mistral 权重（包括 FP8 量化文件）和 HF 兼容布局的单元测试，确保所有权重都被成功映射并加载。  
- **日志监控**：在生产环境开启 `logger` 的 `WARNING` 级别，监控是否出现 “Unrecognized weight” 警告，以便及时捕获新版本 checkpoint 的不兼容。  
- **向后兼容**：保持 `MistralForCausalLMMistralFormat` 与原有 `MistralForCausalLM` 共存，避免因旧模型仍使用旧类而产生冲突。建议在文档中明确说明何种文件结构对应哪个加载类。  
- **文档同步**：确保新增的 cookbook 文档与代码实现保持一致，尤其是 `--tp`、`--tool-call-parser`、`--reasoning-parser` 参数的默认值，以防用户误配。

### [Spec] Split `accept_length` into `num_accepted_drafts` and `num_accepted_tokens` (#23962)
**SHA**: `bd448e5` | 🔗 [查看提交](https://github.com/sgl-project/sglang/commit/bd448e51bd8bb2863253fce4c8c675ddb13cd3e8)

**🎯 变更类型**：重构 / 功能增强  
**⚡ 重要程度**：🔴高  

**📋 变更摘要**  
本次提交将原先的 `accept_length`（表示每个请求在 DRAFT_EXTEND 模式下接受的草稿 token 数）拆分为 `num_accepted_drafts`（仅草稿数量）和 `num_accepted_tokens`（草稿数量 + 1，含验证成功后产生的 “bonus” token）。相应地，所有与 `accept_length` 相关的代码、结构体、CUDA kernel、日志统计、适配器以及测试均完成了迁移和同步。该改动消除了“accept_length 含不含 bonus token” 的歧义，提高了代码可读性和后续扩展的安全性。

**🎯 影响范围**  
- `sglang/srt/speculative/*`（Eagle、NGram、Multi‑layer Eagle、Adaptive runtime）

- 调度与结果处理 (`scheduler_output_processor_mixin`, `tokenizer_manager`, `utils`)  
- 前向批次封装 (`forward_batch_info`)  
- 相关 CUDA/Trition kernel（`create_extend_after_decode_spec_info`, `filter_finished_cache_loc_kernel`, 等）  
- 单元测试与基准套件（`test/attention`, `test/kits/spec_decoding_kit`）  

---

### 🔍 技术洞察

| 维度 | 影响分析 |
|------|----------|
| **架构影响** | - **数据模型统一**：`EagleDraftInput`、`SpecInput`、`GenerationBatchResult` 等结构体现在同时保存 `num_accepted_drafts` 与 `num_accepted_tokens`，两者在内部保持 `num_accepted_tokens = num_accepted_drafts + 1`。<br>- **后端统一接口**：所有 attention 后端在 `init_forward_metadata*` 中改为读取 `num_accepted_tokens`（用于序列长度）和 `num_accepted_drafts`（用于 KV‑cache 释放等），统一了不同 backend 的实现路径。<br>- **适配层减少歧义**：原来 `accept_length` 语义在不同模块间不一致（有时包含 bonus token，有时不），导致注释和 bug（如 TODO/FIXME 注释中提到的 “invalid Eagle tree”）。拆分后，语义清晰，后续新增功能（如动态 draft‑step）更易实现。 |

4. **多进程/分布式广播**
