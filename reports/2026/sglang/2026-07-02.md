### [XPU] Enable XPU graph support (decode full-graph + prefill tc_piecewise) (#29053)
**SHA**: `926140d` | 🔗 [查看提交](https://github.com/sgl-project/sglang/commit/926140d789c8c7ec6622a47630b5894b37fbdc1b)

**🎯 变更类型**：功能增强 / 架构变更  
**⚡ 重要程度**：🔴高  

**📋 变更摘要**  
本次提交为 Intel XPU（torch.xpu）平台引入了完整的 CUDA‑graph（XPUGraph）支持。实现了 **decode‑full‑graph** 与 **prefill‑tc_piecewise** 两种图捕获机制，提供了对应的后端实现（`XPUPiecewiseBackend`、`FullXPUGraphBackend`、`XPUGraphRunner`），并在服务器启动参数、后端选择、分布式通信、注意力层以及 profiling 等多个层面做了适配。默认开启 decode full‑graph，prefill 图为实验性 **opt‑in**，并提供了完整的 JSON 配置接口。

---

### 🎯 影响范围
- **核心编译框架**：`sglang/srt/compilation/*`（新增 XPU 后端、弱引用张量适配）  
- **硬件后端实现**：`sglang/srt/hardware_backend/xpu/*`（full‑graph、piecewise 捕获、runner）  
- **模型执行层**：`xpu_backend.py`（元数据预分配、SWA、Encoder‑Decoder 支持）  
- **分布式通信**：`parallel_state.py`（all‑reduce 包装）  
- **服务器参数解析**：`server_args.py`（XPU‑specific flags）  
- **通用工具**：`utils/common.py`（注册 XPU 设备属性至 Dynamo）  
- **模型 runner**：`model_executor/model_runner.py`（XPU 图后端注入）  
- **文档**：`docs_new/docs/hardware-platforms/xpu.mdx`（使用指南）  
- **测试套件**：新增 XPU 图测试 `test_xpu_graph.py`，以及多处 XPU 相关测试强制关闭 decode‑graph 以兼容 CI。

### 🔍 技术洞察  

#### 架构影响
- **后端抽象统一**：`XPUPiecewiseBackend` 继承自 `CUDAPiecewiseBackend`，复用了 CUDA‑piecewise 逻辑，仅在捕获阶段切换为 `torch.xpu.XPUGraph`。  
- **全图 vs. 分段图**：`FullXPUGraphBackend` 实现 **单图 per‑shape**（decode），而 `XPUPiecewiseBackend` 负责 **分段捕获**（prefill tc_piecewise），二者通过统一的 `--cuda-graph-config` JSON 配置进行组合。  
- **Dynamo 兼容**：通过 `register_xpu_device_properties_for_dynamo` 将 XPU 设备属性加入 Dynamo 常量集合，防止在 `torch.compile` 时出现常量折叠错误。  
- **分布式算子封装**：`parallel_state.py` 将 all‑reduce 重写为自定义 `inplace_all_reduce`，避免 Dynamo 将其拆解成 `_c10d_functional` 调用，从而不触发 `sycl_event.wait()` 导致的图捕获中断。  
- **注意力元数据预分配**：在 `xpu_backend.py` 中新增 `init_cuda_graph_state` 与 `init_forward_metadata_out_graph`，在图捕获前预创建固定地址的缓冲区（`cache_seqlens`、`page_table`、SWA‑table、Encoder‑Decoder‑specific 表），确保捕获图的地址不变。  
- **兼容层**：对 PyTorch ≤ 2.12 的 `device_context` 做了 `nullcontext` 包装，以免 Dynamo 在编译时因 `torch.xpu.device` 产生不可包装对象而报错。  
- **特性禁用**：明确标记当前 XPU 图不支持 memory‑saver、两批次 overlap、speculative decoding、breakable CUDA‑graph 等，防止意外启用导致崩溃。

#### 性能影响
- **Kernel 启动开销**：decode 阶段捕获 `XPUGraph` 后，后续推理只需 **一次图启动**，显著降低每步 kernel launch latency，尤其在高 batch‑size 场景。  
- **Prefill 分段图**：`tc_piecewise` 把不同 token‑length bucket 编译成独立子图，可在 **首次预热** 时一次性编译，后续复用，大幅提升预填阶段吞吐。  
- **torch.compile 加持**：`--enable-torch-compile` 允许在 decode graph 之上再做一次 `torch.compile`，可进一步优化算子（如 fusion），但会显著增加启动时间（compile warm‑up + graph capture）。  
- **内存占用**：图池（global graph memory pool）在 XPU 上默认 **共享** decode 与 prefill，最大占用为 `max(decode, prefill)`，但仍需要预留 `max_bs * max_seq_len` 的 page‑table、SWA 表等，可能导致 **KV‑cache 可用内存下降**。  
- **启动时延**：首次捕获和 piecewise 编译会触发两遍前向（warm‑up + capture）以及可能的 `torch.compile` 编译器运行，启动时间比纯 eager 约提升 1–3 秒（视模型大小而定）。

#### 安全考虑
- **配置注入**：`--cuda-graph-config` 直接解析 JSON，若未严格校验可能导致未知字段被忽略或产生异常。当前实现仅在内部使用 `json.loads`，未做 schema 校验，建议在未来加入 schema 验证或 whitelist。  
- **图捕获错误**：捕获期间若内存不足或图池分配失败，会触发 `RuntimeError`，但代码未捕获并回退到 eager。潜在的 **Denial‑of‑Service**（服务启动失败）风险。  
- **Dynamo 兼容**：在老版本 PyTorch 环境下使用 `nullcontext` 可避免编译错误，但若未来 PyTorch 更新去掉该兼容层，可能导致图捕获失败。建议限制最低 PyTorch 版本至 2.13+，或在 CI 中明确检测版本。

### ⚠️ 潜在风险
1. **启动失败回退缺失**：若图捕获或 `torch.compile` 过程抛异常，当前进程会直接退出，缺少自动 fallback 到 eager。  
2. **内存使用不透明**：全图与分段图共用同一图池，缺少对 **graph memory usage** 的实时监控，可能导致 OOM，尤其在大模型或高并发部署。  
3. **不兼容特性**：XPU 图不支持 memory‑saver、两批次 overlap、speculative decoding、breakable graph 等，若用户误开启对应 flag 将触发断言或错误。

### Speculative decoding support on XPU (#23180)
**SHA**: `4fffc64` | 🔗 [查看提交](https://github.com/sgl-project/sglang/commit/4fffc6448b58e65ba9bdc5b57f0acd677ac2f9b7)

**🎯 变更类型**：功能增强 / 架构变更 / 性能优化  
**⚡ 重要程度**：🔴高  
**📋 变更摘要**：  
- 为 Intel XPU（torch.xpu）平台实现完整的 Speculative Decoding（EAGLE）支持，包括 XPU CUDA‑Graph（XPUGraph）后端、Triton‑based树构建与验证 kernel、以及相应的调度与测试改动。  
- 新增 `XPUCudaGraphBackend`，在 `resolve_decode_backend` 中根据 `device=="xpu"` 自动选择该后端。  
- 在 Eagle 相关代码中引入 `is_xpu` 判定，扩展了多设备（CUDA、HIP、MUSA、NPU、XPU）统一分支，并在 Triton 代码路径上加入 XPU 支持。  
- 更新单元测试，以在 CI 环境中同时跑 CUDA 与 XPU 两套基准，确保功能等价性。  

**🎯 影响范围**：

- **Eagle 规格实现**：`sglang/srt/speculative/*`（eagle_utils、eagle_worker_v2、spec_utils、triton_ops）  
- **测试套件**：`test/registered/spec/*`（新增 XPU CI 注册、Device‑agnostic测试）

| **架构影响** | - **后端抽象统一**：通过 `BaseCudaGraphBackend` 的子类 `XPUCudaGraphBackend` 实现了 XPU 与 CUDA/NPU 类似的图捕获/重放机制，保持了 `runner_backend` 的统一接口。<br>- **设备判定扩展**：在多个模块中加入 `is_xpu`，实现了 “device‑first” 的分支，几乎不影响原有代码路径，仅在 XPU 环境下走新实现。<br>- **Triton kernel 迁移**：原先仅在 CUDA/HIP/MUSA 上使用 Triton，现在通过 `is_xpu` 将同一 Triton 实现用于 XPU，代码复用率提升。 |

| **安全考虑** | - 变更仅涉及计算路径和内部调度，无外部 I/O，安全风险极低。<br>- 新增的 `is_xpu` 判定来自 `sglang/srt/utils.py`，已在 CI 环境下通过设备检测；若检测误判（如在非 XPU 环境误报），仍会走 CUDA 路径，不会导致安全漏洞。 |
| **可维护性** | - 代码分支通过统一的 `is_*` 判定实现，新增的 XPU 分支与其他平台保持高度对称，后续维护成本与新增其他平台（如 ROCm）相同。<br>- Triton kernel 文件 `spec_tree.py` 与原有 CUDA‑only kernel 分离，便于单独调优或废弃。<br>- 测试文件按 CI 设备注册分别跑 CUDA 与 XPU，确保在不同硬件上功能一致，降低回归风险。 |

### ⚠️ 潜在风险  

1. **XPU Graph API 稳定性**：`torch.xpu.XPUGraph` 仍处于相对新鲜的阶段，API 兼容性或内部错误（如图捕获失败、stream 同步问题）可能在特定驱动版本上出现。  
2. **Triton 对 XPU 支持的局限**：虽然 Triton 已可生成 XPU 内核，但某些高级特性（如 `tl.dot` 的高维矩阵乘）在 XPU 上可能会退化为低效实现，需要监控实际运行时的 FLOPs 与内存带宽利用率。  
3. **图池资源泄漏**：`XPUCudaGraphBackend.cleanup` 只清空本地字典并置 `self._pool = None`，若 `graph_pool_handle()` 返回的底层句柄未被显式释放，可能导致显存泄漏。  
4. **CI 环境兼容**：XPU CI 需要特定的硬件/驱动镜像，若镜像升级导致 `torch.xpu` 包版本变更，可能破坏当前捕获逻辑。  
5. **异构多卡（TP）同步**：后端在 `capture_one` 中使用 `self._tp_group.barrier()`，但 XPU 对 NCCL/torch.distributed 的支持可能不同，需要验证跨卡的同步是否可靠。
