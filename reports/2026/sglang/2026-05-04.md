### feat: use structural tags to enable strict tool calling and reasoning for more models (#21722)
**SHA**: `952b3ca` | 🔗 [查看提交](https://github.com/sgl-project/sglang/commit/952b3caf18317b91dc072ebfa1c416a5cfd5eedb)

**🎯 变更类型**：功能增强 / 架构变更 / 性能优化  
**⚡ 重要程度**：🔴 高  
**📋 变更摘要**：本次提交将 **xgrammar** 升级至 0.2.0，并在 SGLang 的函数调用解析体系中引入 **模型原生 structural tag**（结构标签）支持。新增 `get_structural_tag*` 接口、`thinking_mode` 参数以及针对 DeepSeek‑V4、Qwen‑3‑Coder、GPT‑OSS 等模型的专属 detector，实现 **strict tool‑calling** 与 **reasoning‑prefix** 的统一约束；同时修正 speculative 代码中对 torch tensor 传递的类型错误，提升兼容性与性能。  

---

## 🎯 影响范围
- **核心模块**：`sglang/srt/function_call/*`（detect → parser → constraint 逻辑）  
- **入口层**：`sglang/srt/entrypoints/openai/serving_chat.py`（思考模式标记）  
- **依赖**：`python/pyproject*.toml` 系列全部升级 `xgrammar==0.2.0`  
- **CI/测试**：新增/扩展单元测试覆盖结构标签、DeepSeek‑V4、Qwen‑3‑Coder、GPT‑OSS、遍历树（spec_utils）  
- **构建脚本**：`.github/workflows/nightly-test-*.yml` 中同步升级依赖版本  

---

## 🔍 技术洞察

| 维度 | 影响说明 |
|------|----------|
| **架构影响** | 1. **模型原生结构标签路径**：新增 `BaseFormatDetector.get_structural_tag*` 系列抽象，子类实现 `get_structural_tag_name` 返回模型标识， `FunctionCallParser` 优先调用 `detector.get_structural_tag`（模型原生）<br>2. **思考/推理前缀**：在 `serving_chat.py` 中通过 `thinking_mode` 标记，防止在已配置 `--reasoning-parser` 时出现双重 `<think>` 前缀。<br>3. **兼容层**：若 `xgrammar` 未安装或模型不支持，仍可回退到 **legacy structural tag**（OpenAI‑compatible），保证向后兼容。<br>4. **新增 Detector**：`DeepSeekV4Detector` 完整实现 XML/JSON 两种 DSML 语法，统一到基类 `DeepSeekV32Detector`。 |
| **性能影响** | - **原生 structural tag** 直接交给 xgrammar 生成语法树，省去 SGLang 自己的正则/JSON 解析，显著降低 streaming‑parsing 时的 CPU/内存开销。<br>- `speculative/spec_utils.dfs` 中显式 `int()` 转换避免了 0‑dim tensor 传递导致的异常，提升了 **GPU/NPU** 上的推理速度（少一次 Python‑C++ 边界检查）。 |
| **安全考虑** | - 新增的 `xgrammar` 依赖属于纯 Python/C++ 解析库，未引入网络请求或文件写入，仅在 **模型推理** 环节使用，安全风险极低。<br>- 代码对 `ImportError` 设置了安全回退（把 `StructuralTag` 设为 `Any`），防止因缺少库导致服务崩溃。 |
| **可维护性** | - 通过统一的抽象（`get_structural_tag_name` / `get_structural_tag`）简化后续模型适配，只需在对应 detector 中返回模型标识即可；<br>- 测试覆盖从 **detect → parse → streaming → structural‑tag → grammar** 完整链路，降低回归风险。 |
| **兼容性** | - 依赖升级到 `xgrammar==0.2.0`，若用户自行安装旧版可能导致功能失效；建议在发行说明中锁定版本或提供兼容 shim。<br>- `thinking_mode` 参数默认 `False`，保持旧行为不变；开启 `--reasoning-parser` 时在 `serving_chat` 中自动切换。<br>- 对不支持 native structural tag 的模型（如 Kimi‑K2）保持 **legacy** 路径，已有行为不变。 |

## ⚠️ 潜在风险
1. **依赖冲突**：部分用户的环境可能仍锁定 `xgrammar==0.1.*`，升级后 `pip install -r requirements.txt` 可能出现版本冲突。  
2. **模型不兼容**：若模型标识返回错误或 xgrammar 未实现对应 tag（新模型），`detector.get_structural_tag` 会返回 `None` 并回退；但在 **strict** 场景下可能导致 **tool‑call约束缺失**，影响调用可靠性。  
3. **思考模式标记失误**：`thinking_mode` 参数的传递链路较长（`serving_chat` → `parser.get_structure_constraint`），若未来改动忘记转发，可能出现 **双重 `<think>`** 或 **缺失 `<think>`**，导致推理不符合预期。  
4. **性能回退**：在极端高并发或资源受限的部署场景下，若 xgrammar 的 C++ 实现出现锁争用，可能比纯 Python 的 legacy 结构标签稍慢。  
5. **测试依赖**：新增的大量单元测试对 `xgrammar` 的实际实现做了深度校验，CI 环境若缺少 GPU/CPU 兼容的 xgrammar 发行版，可能导致 CI 失效。

### feat(lora): enable csgmv backend with virtual experts for MoE LoRA (#24007)
**SHA**: `b7fefc0` | 🔗 [查看提交](https://github.com/sgl-project/sglang/commit/b7fefc0e85d0f24a4906e7854e026a954f48e6a8)

**🎯 变更类型**：功能增强  
**⚡ 重要程度**：🔴高  
**📋 变更摘要**：  
- 为 LoRA 的 CSGMV 后端引入“virtual experts”支持，使其能够像 Triton 后端一样在 MoE 场景下按请求维度进行适配器映射。  
- 新增 `req_seg_indptr` 与 `req_weight_indices` 两个张量字段，并在批处理构造、后端初始化及 LoRA 层读取逻辑中使用，以实现请求级别的分段与适配器索引。  
- 同时放宽 `--lora-use-virtual-experts` 参数的后端限制，允许在 CSGMV 后端下使用该特性。  

**🎯 影响范围**：  
- `python/sglang/srt/lora/backend/chunked_backend.py`（CSGMV 后端的批信息初始化与准备逻辑）  
- `python/sglang/srt/lora/layers.py`（LoRA 前向计算中对适配器激活与分段索引的读取）  
- `python/sglang/srt/lora/utils.py`（`LoRABatchInfo` 数据结构扩展）  
- `python/sglang/srt/server_args.py`（启动参数校验放宽）  

**🔍 技术洞察**  

- **架构影响**  
  - **后端抽象统一**：通过在 `LoRABatchInfo` 中添加请求级别的 `req_*` 字段，CSGMV 与 Triton 两个后端在“virtual experts”特性上的数据路径趋于一致，提升了后端抽象层的可组合性。  
  - **批处理信息扩展**：在 `ChunkedBackend.init_cuda_graph_batch_info` 和 `prepare_lora_batch` 中为每个请求额外维护两套索引（`req_seg_indptr`、`req_weight_indices`），使得 MoE 场景下的 token‑to‑request 映射不再依赖后端内部的 chunk 分段，实现了“虚拟专家”概念的后端无关实现。  
  - **参数校验松绑**：`server_args` 中去掉了 `--lora-backend triton` 的强制要求，意味着部署时可自行选择更适合硬件的后端（如 CSGMV），提升了部署灵活性。  

- **性能影响**  
  - **正向**：  
    - **Pinned‑memory CPU 张量**（`pin_memory=True`）在构造 `req_seg_indptr` 与 `req_weight_indices` 时避免了即时的 GPU 同步，能够在 GPU‑CUDA‑graph 前置阶段完成拷贝，降低了 CPU‑GPU 交互延迟。  
    - **Virtual Experts** 通过请求级索引直接映射适配器，避免了在 dense‑LoRA 后端内部的跨请求 chunk 合并导致的额外算子分支，预计在高并发 MoE 场景下会提升吞吐量与 latency 稳定性。  
  - **潜在负面**：  
    - 为每个请求额外分配两块 `int32` 张量，会在极端大 batch（如几千请求）时增加显存/显卡内存占用。  
    - `torch.arange` 与 `torch.cumsum` 在 CPU 上执行，若请求数非常大可能成为准备阶段的热点，但整体仍在可接受范围。  

- **安全考虑**  
  - 代码未涉及网络、文件 I/O、密码学等安全敏感操作，安全风险基本为 **无**。  
  - 唯一需要注意的是 **数据完整性**：请求级索引若出现越界或未对齐的情况，可能触发 GPU 越界访问，导致进程崩溃或错误结果，属于可靠性而非安全问题。  

**⚠️ 潜在风险**  

| 风险点 | 说明 | 严重程度 |
|--------|------|----------|
| 索引不匹配 | `req_seg_indptr` 与实际 token 分段不一致（例如 `generate_sequence_lengths` 与 `forward_mode.is_decode()` 分支出现偏差）会导致错误的 LoRA 适配器映射。 | 中 |
| 内存泄漏/显存占用 | 对每个请求分配 `req_weight_indices` 与 `req_seg_indptr`，在极大 batch 下可能触发 OOM。 | 中 |
| 跨平台 Pin‑Memory 兼容性 | 在某些非 Linux 环境（如 Windows）`pin_memory=True` 可能降级为普通复制，影响性能。 | 低 |
| 向后兼容性 | 旧代码仍可能访问 `batch_info.weight_indices`/`seg_indptr`，但新路径已优先使用 `req_*`，若某些分支忘记更新会出现不一致。 | 中 |
| CUDA‑graph 重用错误 | `init_cuda_graph_batch_info` 仍保留老的 `seg_lens`、`max_len` 字段，若后续修改忘记同步新的 `req_*` 可能导致图捕获期间使用旧指针。 | 低 |

**💡 关注建议**  

1. **单元/集成测试**  
   - 添加覆盖 `forward_mode.is_decode()` 与 `!decode` 两条分支的 `ChunkedBackend._build_req_seg_indptr` 单元测试，验证 `indptr` 与实际 token 边界的一致性。  
   - 在 MoE 场景下，使用不同 batch size（小/中/大）跑一次 `sglang` 推理，确保 `req_weight_indices` 与 `weight_indices` 的映射一致。  

2. **显存监控**  
   - 在服务器启动脚本中加入显存预占检查（如 `torch.cuda.memory_reserved()`），在 batch_size 超过阈值时给出警告或自动降级为 Triton 后端。  

3. **文档与使用指引**  
   - 更新 README / 参数帮助，说明 `--lora-use-virtual-experts` 现在可与 `--lora-backend csgmv` 配合使用，并列出推荐的 `max-lora-chunk-size` 与 `batch-size` 区间。  

4. **回滚兼容**  
   - 在 `layers.py` 中保留对 `batch_info.seg_indptr` 与 `batch_info.weight_indices` 的回退路径已经实现，建议在后续发布里加上 `DeprecationWarning`，提醒用户逐步迁移到 `req_*` 字段。  

5. **性能基准**  
   - 对比同等配置下 Triton 与 CSGMV 两个后端的吞吐量、请求延迟（尤其在 4‑8 张 GPU 以及 256‑1024 batch 场景），记录数据以便在后续优化中量化 “virtual experts” 的收益。  

---  

**总结**：此次提交为 SGLang 引入了对 CSGMV 后端的 MoE‑LoRA 虚拟专家支持，提升了后端选择灵活性并在高并发 MoE 场景中预期带来显著性能提升。主要风险集中在索引正确性与显存消耗，建议通过完善测试、监控和文档来降低风险并确保平滑迁移。

### [diffusion] feat: initial support for dynamic batching (#18764)
**SHA**: `62265ca` | 🔗 [查看提交](https://github.com/sgl-project/sglang/commit/62265ca7fc9555bc2cf2e64b435318de95fec318)

**🎯 变更类型**：功能增强  

**⚡ 重要程度**：🔴高  

**📋 变更摘要**  
- 为 SGLang‑Diffusion 引入 **动态推理批处理（dynamic batching）**，在服务端可将多个兼容的 T2I / T2V 请求合并为同一个原生 diffusion 批次执行，从而大幅提升吞吐量。  
- 新增 **BatchAdmissionController** 实现基于模型、分辨率、显存、用户配置的批次容量与成本上限；并在调度器中实现请求合并、批次拆分、统计与日志。  
- 扩展 **PipelineConfig / Req** 以携带文本序列长度、对齐掩码（`TextConditioningOutput`），并在各模型（Flux、Flux2、Qwen‑Image、Z‑Image、SANA 等）中实现对这些信息的消费与校验。  
- 增加 CLI 参数 `--batching-mode / --batching-max-size / --batching-delay-ms / --batching-config / --enable-batching-metrics`，并在文档中加入使用说明。  

**🎯 影响范围**  
- `python/sglang/multimodal_gen/runtime/*`（调度器、GPU/CPU worker、动态批次管理）  
- `python/sglang/multimodal_gen/configs/pipeline_configs/*`（所有 diffusion pipeline 实现）  
- `python/sglang/multimodal_gen/runtime/pipelines_core/*`（`Req`、`BatchMetricsWindow`、采样参数）  
- `docs_new/docs/sglang-diffusion/*`（文档新增 “Inference Batching” 页面）

| **模型实现** | 1. `qwen_image`、`zimage`、`flux`、`flux2` 等注意力层改造，以接受 `encoder_hidden_states_mask` 或 `prompt_embeds_mask`，并在 RoPE、位置缓存构建时使用真实文本长度。<br>2. `zimage` 现在在 `patchify_and_embed` 中统一 **批量 pad & truncate**，并提供 `_replace_padding_with_token` 用于恢复真实 token。 |
| **GPU Worker** | 在 `GPUWorker._execute_forward_common` 中记录 **peak reserved GPU memory**，并在保存输出时支持 **动态输出路径**（`dynamic_batch_output_paths`）。 |
| **采样参数** | 在 `SamplingParams` 中为所有 **不参与批次签名** 的字段标记 `metadata={"batch_sig_exclude": True}`，调度器通过字段签名判断请求兼容性。 |
| **监控/日志** | 新增 `BatchMetricsWindow` 用于统计 `dispatches、merged率、full率、queue_wait、reject_reason`，并在 `Scheduler` 中每 **5 次派发** 打印一次批次统计报告。 |

### 性能影响  
- **吞吐提升**：在多请求并发、相同模型/分辨率的场景下，可将 1‑8 条请求合并为单个原生 diffusion 批次，显著降低算子调度与显存搬运开销（尤其在大模型如 FLUX、Qwen‑Image）。  
- **延时波动**：为等候兼容请求，调度器会在 `batching_delay_ms`（默认 0）内阻塞，这会导致单条请求的 **最大等待时延** 增加（日志中 `queue_wait_ms`），但对整体吞吐是正向的。  
- **显存占用**：合并批次会一次性占用 **max_batch_size × 单条请求显存**。如果成本估算不准确，可能导致 OOM。为此引入 `estimate_request_cost` 与 `max_cost` 限制。  
- **CPU 与 GPU 额外拷贝**：合并/拆分过程涉及 `torch.stack`、`torch.clone` 以及 `deepcopy`，在极高并发下会产生额外的 CPU‑GPU 拷贝开销，需关注 CPU 端的 `copy` 瓶颈。  
- **统计开销**：每 5 次派发计算一次百分位、计数器等，开销在毫秒级，可接受。  

### 安全考虑  
| 维度 | 风险点 | 措施 / 现状 |
|---|---|---|
| **配置文件** | `batching_config.json` 为用户可编辑的 JSON，若缺乏校验可能导致 **异常字段**、路径注入等。 | `BatchingRule.from_dict` 对键进行白名单校验，数值类型强制转换，已基本防护。 |
| **输入合法性** | 动态合并后若 **序列长度不匹配** 或 **mask 错误**，可能触发 **非法内存访问**（torch 报错）或生成 **错误图像**。 | 新增对 `prompt_seq_lens`、`prompt_embeds_mask` 的完整性检查；在模型层面若 mask 与长度不匹配抛异常。 |
| **错误传播** | 合并批次执行失败会一次性返回 **统一错误信息** 给所有子请求，可能隐藏单独请求的根因。 | 目前实现为批次整体失败时返回统一错误，已记录日志；建议在未来提供 **子请求错误归因**。 |
| **信息泄露** | `log_batch_completion` 与 `log_batch_metrics_summary` 打印 **batch size、wait time、reject reason**，在公开服务中可能泄露系统负载与模型配置。 | 仅在 **INFO** 级别输出，默认日志可在生产环境调

**SHA**: `9a5450a` | 🔗 [查看提交](https://github.com/sgl-project/sglang/commit/9a5450ad7386a12fc1ea0082dbb28ae21355ddda)

**🎯 变更类型**：功能增强  
**⚡ 重要程度**：🟡 中  

**📋 变更摘要**  
为 mooncake 迁移引擎加入增量 KV 前缀缓存（decode_prefix_len），在 `TransferInfo` 中新增可选字段并在消息解码/编码、bootstrap、发送逻辑中同步该值；同时放宽 `--disaggregation-transfer-backend` 的校验，允许 `mooncake` 使用 radix 缓存；增加对应单元/集成测试覆盖两种后端并检查准确性不回退。

**🎯 影响范围**  
- `python/sglang/srt/disaggregation/mooncake/conn.py`（核心传输、线程调度）  
- `python/sglang/srt/server_args.py`（CLI 参数校验、帮助信息）  
- 测试目录：`test/registered/distributed/`、`test/registered/unit/server_args/`  

**💡 关注建议**  
1. **兼容性**：新增 `decode_prefix_len` 采用 `Optional[int]`，默认 `None` → 发送端会写 `0`，已有 nixl 框架不受影响。请确认所有旧版 `TransferInfo` 创建点已适配（参数顺序未变）。  
2. **并发安全**：`req_to_decode_prefix_len` 在多线程 `bootstrap_thread` 与 `transfer_worker` 中读写，建议使用 `threading.Lock` 或改为 `collections.defaultdict(int)`，防止稀疏键竞争。  
3. **消息格式**：在 `from_zmq` 中读取第 9 位字段，若后端未传递仍兼容；但若其他组件自行构造 ZMQ 消息，需同步更新序列化顺序。  
4. **测试**：CI 已加入 mooncake 后端的跑分验证，建议本地跑两遍（nixl、mooncake）确认 KV 前缀缓存的命中率提升；同时检查 `should_send_kv_chunk` 的返回逻辑不影响跨轮 flush。  
5. **文档/帮助**：更新 `--disaggregation-decode-enable-radix-cache` 的说明，明确 “支持 nixl 与 mooncake”。若后续再加入其他后端，请相同方式放宽校验。  

以上改动主要聚焦在增量传输与参数校验，整体风险有限，但请重点关注线程安全和消息序列化的一致性。

- `python/sglang/multimodal_gen/runtime/models/dits/qwen_image.py`（模型实现）  
- `python/sglang/multimodal_gen/tools/build_modelopt_fp8_transformer.py`（转换工具）  
- CI 测试 `gpu_cases.py`、`testcase_configs.py`（新增 Qwen 测试）

3. **正则匹配**：`DEFAULT_QWEN_IMAGE_KEEP_BF16_PATTERNS` 采用 `^transformer_blocks\.\d+\.img_mlp\.net\.2$` 等。建议在 CI 中加入对这些模块的量化/非量化检查，防止正则遗漏导致误量化。  
4. **转换顺序**：在 `build_modelopt_fp8_transformer` 中对 `fallback_tensors` 的覆盖在 `if name in fallback_scale_names` 之后进行，逻辑上已正确，但保留注释说明此顺序的意图，防止后续维护者误删。  
5. **CI 资源**：新增 Qwen‑Image FP8 用例会在 B200 CI 中占用显存，建议确认 H100/RTX 测试机器具备足够显存（≈24 GB）并在 CI 配置中加入相应的 GPU 环境变量。  
6. **文档同步**：文档表格已更新为 8 条 checkpoint，请确保发布前的 Release Notes 中也同步列出 Qwen‑Image 系列的支持。  

总体来看，变更在实现层面实现了对 Qwen‑Image 系列的 FP8 量化支持，影响范围限定在模型实现、转换脚本和 CI，未对已有模型产生破坏性改动。只需关注上述细节即可确保平滑上线。
