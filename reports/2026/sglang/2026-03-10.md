### [Feature] Add SANA diffusion model (#19234)
**SHA**: `9610944` | 🔗 [查看提交](https://github.com/sgl-project/sglang/commit/9610944ae60ab10e27e86dcd405cfe9801748fcf)

**🎯 变更类型**：功能增强（新增 SANA Diffusion 模型全链路支持）  
**⚡ 重要程度**：🔴高  

**📋 变更摘要**  
- 在 `sglang` 中完整加入 SANA（Diffusion Transformer）模型，包括模型、文本编码器（Gemma2）、VAE（DC‑AE）、调度器（DPMSolver++）、管线配置、运行时实现以及注册机制。  
- 新增/修改的文件覆盖模型配置、权重加载、FS DP 分片、参数映射、pipeline stage、采样默认参数以及单元测试。  
- 目标是让用户能够直接在 sglang 上使用 SANA‑1.5/4.8B 系列模型进行 1024px/512px 文本‑to‑image 生成，保持与现有 Flux/Qwen 等管线的统一体验。

### 🔍 技术洞察

| 维度 | 分析要点 |
|------|----------|
| **架构影响** | - **全新 DiT‑2D 模型**：`SanaTransformer2DModel` 采用线性注意力（ReLU‑based）+跨注意力（标准 SDPA），替代传统 Quadratic Self‑Attention，显著降低显存 O(N² → O(N·D))。 <br> - **文本编码器**：Gemma2 2B 作为 *decoder‑only* 编码器，仅输出最终 hidden‑state（不做 logits 投射），通过 `caption_projection` 投射到 DiT 维度。 <br> - **VAE**：使用 Diffusers 的 `AutoencoderDC`（Deep Compression AutoEncoder），空间压缩 32×（1024→32），配合 DiT 直接输出 4D latent。 <br> - **调度器**：包装 Diffusers `DPMSolverMultistepScheduler`，配置 order=2、20 步推理，保持与官方 SANA 实现一致。 <br> - **Pipeline**：基于 `SpatialImagePipelineConfig`，简化 latent 形状（4D），去掉旋转位置嵌入，`should_use_guidance=False`（仅使用标准 CFG）。 |
| **性能影响** | - **显存**：线性注意力 + 32× VAE 压缩大幅降低显存需求（同等分辨率下可在单卡 24GB 以上运行 4.8B）。 <br> - **算力**：线性注意力实现使用 `F.relu` &矩阵乘法，CPU/GPU 上算子成熟度不如原生 `scaled_dot_product_attention`，在极长序列（> 1024）上可能略慢，但总体因层数较少（20‑36）而仍快于传统 DiT。 <br> - **加载**：`vae_loader` 改为合并多个 safetensors（防止单文件拆分），以及 `fsdp_load` 放宽对 `norm_q` / `norm_k` 参数的容忍，提升对 SANA 特有层的兼容性。 <br> - **精度**：默认 VAE‑bf16、模型‑bf16，`decoder` 阶段自动开启 `autocast`（除 decoding 已显式处理），提升吞吐。 |
| **安全考虑** | - **权重加载**：仍然依赖从 HuggingFace Hub 拉取的 safetensors，未出现新的可执行代码路径。<br> - **软上限（softcapping）**：Gemma2 注意力中声明的 `attn_logit_softcapping` 与 `final_logit_softcapping` 在实现里被忽略（Diffusers 不支持），对安全无直接影响，仅可能导致输出质量略降。<br> - **输入验证**：pipeline 复用了已有 `InputValidationStage`，未新增风险。 |
| **可维护性** | - **代码组织**：所有 SANA 相关实现均放在独立子目录（`models/dits/sana.py`、`models/encoders/gemma2.py`、`models/vaes/autoencoder_dc.py`），与其它模型保持一致的命名约定，易于扩展。<br> - **参数映射**：`SanaArchConfig.param_names_mapping` 统一了 HF 权重与本地层的映射，配合 `fsdp_load` 中的 `norm_q/k` 兼容；但映射表仅覆盖 `"transformer\.(.*)" → "\1"`，若未来 checkpoint 改动需手动更新。<br> - **测试覆盖**：仅在 `testcase_configs.py` 增加了一个简易的 SANA 测试条目，未加入功能或质量断言，后续应补充 **生成质量** 与 **显存/时延** 基准。 |
| **兼容性** | - **注册系统**：`registry._register_configs` 新增 SANA detection (`lambda hf_id: "sana" in hf_id.lower()`) 与采样/管线映射，保证在 `sgserve` 启动时自动识别。<br> - **调度器接口**：包装的 `DPMSolverMultistepScheduler` 兼容 `BaseScheduler`，但未实现 `set_begin_index` 等少数高级特性（仅保留占位），在大多数使用场景下足够。 |

### ⚠️ 潜在风险

| 风险点 | 可能后果 | 缓解措施 |
|--------|----------|-----------|
| **软上限未实现**（Gemma2） | 对长文本可能出现注意力数值溢出，导致图像质量下降。 | 在后续实现自定义 `scaled_dot_product_attention` kernel，或在 `Gemma2Attention` 中手动对 logits 做 `tanh(logits / cap) * cap` 处理。 |
| **权重映射错误** | 若 HF checkpoint 的层命名改变，加载会出现未匹配的参数，导致模型推理错误或性能下降。 | 加入更细粒度的映射表与自动检测（例如在 `fsdp_load` 中打印未匹配的 key）。 |
| **VAE 多文件合并** | 合并多个 safetensors 时若出现冲突（同名 key），后加载的会覆盖前者，可能导致权重不完整。 | 在 `vae_loader` 合并前加入冲突检测并抛异常；或在 HF 官方发布统一单文件。 |
| **线性注意力数值不稳定** | 由于 ReLU 替代传统缩放，极端输入可能导致梯度爆炸（虽推理阶段对梯度无影响，但可能产生 NaN 输出）。 | 增加 `torch.clamp(..., min=1e-6)` 检查或在 `SanaLinearAttention` 中加入层归一化后再 ReLU（已实现 `norm_q/k`）。 |

### [4/n jit_kernel restruct] speed up CI tests and add benchmark workflow (#20268)

- CI 环境下的测试覆盖率已被适度压缩，若对边缘配置有疑虑，可在本地自行开启全量测试以验证。  

总体而言，此次改动显著提升 CI 稳定性与运行时长可控性，同时为后续 benchmark 引入硬件感知检查奠定基础。建议在发布说明中补充环境变量使用说明，并在 README 中注明对 sm100+ 的最低需求。
