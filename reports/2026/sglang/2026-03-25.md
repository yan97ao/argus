### [diffusion] CI: add performance tracking job to nightly (#21091)
**SHA**: `04eb728` | 🔗 [查看提交](https://github.com/sgl-project/sglang/commit/04eb72801f6c712f6c8a9428727d163f68acf43d)

**🎯 变更类型**：性能优化 / 功能增强 / CI/测试扩展 / 安全修复  
**⚡ 重要程度**：🔴高  

**📋 变更摘要**  
本次提交在 Nightly CI 中新增 **Diffusion 性能追踪** 工作流，包括：  
1. 新增 `nightly-test-diffusion-comparison` 作业，对 SGLang、vLLM‑Omni、LightX2V 等多框架进行端到端的扩散模型（文本‑图、图‑编、文本‑视频 等）跑分并生成对比报告。  
2. 新增一套完整的脚本体系 `scripts/ci/utils/diffusion/`：  
   - `run_comparison.py`：自动启动服务器、执行 warm‑up、收集服务器端（PerfLogger）与客户端延迟。  
   - `generate_diffusion_dashboard.py`：将本轮和历史数据合并，生成 Markdown 报告并绘制 Matplotlib 趋势图。  
   - `publish_comparison_results.py` 与 `publish_diffusion_gt.py`：将报告、图表、原始 JSON 推送至 `sglang-bot/sglang-ci-data`，供公开查询。  
3. API 层加入 `perf_dump_path` 参数，支持在 **服务器模式** 下把每次请求的性能日志写入指定文件。  
4. `gpu_worker.py` 中增加对 `perf_dump_path` 的检测并调用 `PerformanceLogger.dump_benchmark_report`，实现 **服务端性能采集**。  
5. 细微调优：`torch._dynamo.config.recompile_limit` 从 16 提升至 64，以适配更复杂的 Diffusion 编译需求。  
6. CI 元数据合并脚本 `merge_metrics.py` 支持 `comparison-metrics-*.json`。  

---

**🎯 影响范围**  
- **CI/CD**：`.github/workflows/nightly-test-nvidia.yml`、`diffusion-ci-gt-gen.yml` 及相关 Action。  
- **服务器运行时**：`python/sglang/multimodal_gen/runtime/entrypoints/openai/*_api.py`、`gpu_worker.py`。  
- **工具链**：新增 `scripts/ci/utils/diffusion/` 目录下的 5+ 脚本（`run_comparison.py`、`generate_diffusion_dashboard.py`、`publish_comparison_results.py`、`publish_diffusion_gt.py`、`save_diffusion_metrics.py`）。  
- **模型管理**：`torch._dynamo` 编译配置。  
- **Metric 合并**：`merge_metrics.py`。

**🔍 技术洞察**  

| 维度 | 影响说明 |
|------|----------|
| **架构影响** | - 引入 **跨框架对比子系统**，在同一 CI 机器上顺序启动多个服务（SGLang、vLLM‑Omni、LightX2V），每个框架使用独立的启动参数和环境变量。<br>- 新增 `perf_dump_path` 参数，使得 **服务端性能日志** 能够脱离请求返回直接落盘，兼容现有 OpenAI‑兼容入口，不破坏原有业务路径。<br>- `gpu_worker.py` 中的日志写入是 **可选的**（仅在非 warm‑up、且 `perf_dump_path` 有值时触发），对整体架构的耦合度极低。 |
| **性能影响** | - **测量准确性提升**：原本只能通过客户端时间估算，现在直接读取服务器内部 `PerformanceLogger` 的 `total_duration_ms`，消除了网络 I/O 与客户端序列化开销。<br>- **编译调优**：将 `torch._dynamo.config.recompile_limit` 提升至 64，减少在大模型（如 FLUX‑2‑dev）上因频繁重新编译导致的启动慢点。<br>- **CI 时长**：新增的对比作业会占用约 2‑3 小时（取决模型大小），但通过 **GPU‑clear‑wait** 与 **并行分区**（`matrix.part`）已做合理拆分。<br>- **GPU 资源回收**：`kill_server` + `killall_sglang.sh` 确保多 GPU 进程彻底退出，防止残留占用导致后续作业 OOM。 |
| **安全考虑** | - 所有推送至 `sglang-ci-data` 的操作均使用 **GitHub PAT**（`GH_PAT_FOR_NIGHTLY_CI_DATA`），并在脚本中通过 `verify_token_permissions` 进行 **写权限检查**，避免凭证泄漏导致仓库被污染。<br>- 上传文件前做 **路径规范化**（`_sanitize_filename`），防止目录遍历。<br>- 对 GitHub API 的错误做了 **速率限制 / 权限错误** 检测，避免因异常导致 CI 失控。 |

**⚠️ 潜在风险**  

1. **CI 时长显著增长**  
   - 部分模型（如 FLUX‑2‑dev）首次下载并编译需数分钟甚至十几分钟，整个对比作业在 4‑GPU H100 节点上约 3‑4 小时。若节点排队或网络波动，可能导致 Nightly 超时。  
2. **磁盘空间 & 产出文件管理**  
   - 每轮对比会产生数十 MB 的 PNG 图表及 JSON，长期累计在 `sglang-ci-data` 仓库可能触发仓库大小限制。需定期清理或转到对象存储。  
3. **GPU 资源泄漏**  
   - 若 `kill_server` 未能成功结束所有子进程，尤其在多 GPU 环境下，后续作业可能出现 OOM。建议在 CI 中增加 `nvidia-smi` 检查步骤。  
4. **第三方框架兼容性**  
   - `vllm-omni`、`lightx2v` 的安装脚本依赖特定 CUDA / Python 环境，安装失败会导致对应框架的所有对比用例报错并记录为 “installation failed”。需确保 CI 镜像保持同步更新。  
5. **安全凭证泄漏风险**  
   - `GH_PAT_FOR_NIGHTLY_CI_DATA` 必须在 GitHub Secrets 中严格受限，仅拥有 `repo`（写）权限。若泄露，攻击者可往 CI 数据仓库写任意文件。  
6. **性能日志竞争**  
   - 当同一 GPU 上并发跑多个 `perf_dump_path` 时，文件名冲突可能导致覆盖。当前实现使用 case ID + 框架唯一，仍需确保同一作业内不出现重复 case‑framework 组合。

### 🔍 技术洞察  

| 维度 | 影响 |
|------|------|
| **架构影响** | - 在 DiT Transformer 中加入 ** fused QKV / fused Added‑QKV ** 线性层 (`MergedColumnParallelLinear`)；通过 `prefix` 参数实现层级名前缀统一，便于权重映射。<br>- `BaseDiT.__post_init__` 增加 `post_load_weights` 钩子，统一在权重全部 materialize 后执行 `AdaLN` scale/shift 位置交换等模型特有修正。<br>- 配置注册表 (`_NON_DIFFUSERS_MULTIMODAL_PATTERNS`) 增加 `flux.2-dev-nvfp4`，自动识别 HF 缓存快照路径。<br>- 环境变量 `SGLANG_DIFFUSION_FLASHINFER_FP4_GEMM_BACKEND` 可手动指定后端。 |
| **性能影响** | - **内存占用**：FP4 权重量化将原始 `torch.float16/float32` 权重压缩至 4bit，显著降低显存需求，尤其在大模型 (Flux.2) 上可节省约 50%~60% 显存。<br>- **计算加速**：利用 FlashInfer/sgl‑kernel 的 `fp4_quantize` 与 `mm_fp4`（CUTLASS）实现 **块级 SIMD** 计算；在 Blackwell GPU（已装 `comfy_kitchen`）可进一步使用 cuBLAS‑optimzed 路径。<br>- **融合 QKV**：一次线性运算产生 Q/K/V，避免三次矩阵乘和多余的转置，降低带宽压力。<br>- **回退机制**：若缺少加速库，仍会使用通用实现，保持功能可用，只会失去性能提升。 |
| **安全考虑** | - 所有元数据读取均通过 `safetensors.safe_open`（只读、无代码执行）并使用 `json.loads` 解析，未引入可执行代码风险。<br>- 新增的 `build_nvfp4_config_from_safetensors*` 仅在文件存在且元数据合法时返回配置；若 JSON 格式异常会捕获并记录警告，避免异常泄露。<br>- 环境变量解析保持原有安全模型，无新增注入点。 |
| **可维护性** | - 量化实现抽象为 `LinearMethodBase` 子类，保持与已有 `fp8`、`modelslim` 等实现一致的接口，便于后续新增或替换后端。<br>- 参数映射表在 `FluxArchConfig` 中集中管理，新增 NVFP4 规则后保持向后兼容（仍保留 HF Diffusers、Nunchaku 规则）。<br>- `post_load_weights` 为可选钩子，未来若出现其他模型特有的权重纠正，仅需在对应模型实现中覆盖即可。 |

### ⚠️ 潜在风险

1. **依赖缺失或不匹配**  
   - 未安装 `flashinfer`、`sgl_kernel` 或 `comfy_kitchen` 时，量化路径会回退到慢速实现，可能导致显存仍低但吞吐下降。  
   - `comfy_kitchen` 仅在 **Blackwell** GPU 上经验证可用，其他硬件会自动回退，但若误判 `can_use_modelopt_fp4_best_performance_kit`，可能触发运行时异常。

2. **权重映射错误**  
   - NVFP4 检查点的层命名与 Diffusers/Multi‑Modal 代码不同，映射规则若未覆盖全部层（尤其自定义 `packed_modules_mapping`），会出现未加载或错位的权重。  
   - `build_nvfp4_config_from_safetensors_list` 依赖 `param_names_mapping`，若该映射在未来模型中变更，需要同步

**SHA**: `a32e0d5` | 🔗 [查看提交](https://github.com/sgl-project/sglang/commit/a32e0d57e7f04db5fa9a728b4e4ef1a2e537fe00)

**📋 变更摘要**  
本次提交在 SGLang 的 MoE（Mixture‑of‑Experts）实现中加入了 LoRA（Low‑Rank Adaptation）支持，并在 Triton 运行时实现了 LoRA‑aware 的 fused‑MoE kernel。核心改动包括：  
1. `MoeRunner` 增加 `lora_enabled` 标志并在 Triton 后端选择 `TritonRunnerCoreWithLoRA`；  
2. 新增 `FusedMoEWithLoRA` 包装层，实现 LoRA 在 **gate_up** 与 **down** 两个关键位置的融合计算；  
3. 完整的 LoRA‑MoE 运行时实现 `lora_moe_runners.py`，以及对应的 Triton kernel `fused_moe_lora_kernel.py`；  
4. `MemPool`、`utils`、`lora_manager` 等模块同步扩展以管理 4‑维 MoE LoRA 权重（`[num_loras, num_experts, rank, hidden]`）；  
5. 大量测试用例加入，验证在单/双 GPU TP 环境下的正确性与数值一致性。

**🎯 影响范围**  
- **核心层**：`sglang/srt/layers/moe/*`（MoE runner、FusedMoE、TopK、相关 kernel）  
- **LoRA 支持**：`sglang/srt/lora/*`（层包装、内存池、管理器、后端 runner）  
- **配置/工具**：`sglang/srt/utils.py`（隐藏维度查询、stacked rank）  
- **测试**：新增 MoE‑LoRA TP 对齐与 vLLM 对比的回归测试  

**🔍 技术洞察**  

| 维度 | 影响 | 说明 |
|------|------|------|
| **架构影响** | **MoE + LoRA** 形成新的 **4‑维张量** 管线，原有 `BaseLayerWithLoRA` 仍保持兼容；`MoeRunner` 通过 `lora_enabled` 动态切换路径，保证不启用 LoRA 时不增加额外开销。<br>**Tensor‑Parallel (TP)** 支持在 LoRA 计算中正确切分 `gate_up` 与 `down` 权重，涉及 `slice_moe_lora_a_weights` / `slice_moe_lora_b_weights`。 | 1. 代码在 `runner` 端做了 “skip fused func when LoRA enabled” 以避免 fused‑kernel 与 LoRA 路径冲突。<br>2. `FusedMoEWithLoRA` 复用了原 `FusedMoE` 的调度/合并逻辑，仅在内部插入 LoRA delta。 |
| **性能影响** | - **正向加速**：在 LoRA 开启时仍使用 Triton 高效 kernel，LoRA delta 与原计算合并为一次 kernel 调用，避免额外的矩阵乘法或显式的加法。<br>- **额外开销**：LoRA 需要额外的 `moe_lora_align_block_size` 整理和两次 `fused_moe_lora` 调用（gate_up、down），在极端低‑rank 场景下可能占据 5‑10% 计算时间。<br>- **TP 影响**：权重切分后每个 GPU 只计算本地 expert 子集，理论上保持线性加速；对齐阶段的 `moe_lora_align_block_size` 仍是全局操作，可能成为小瓶颈。 | 实测（提交的测试）在 TP=1 与 TP=2 场景下 **log‑prob 差异 ≤ 5e‑4**，输出字符串完全一致，说明数值误差在可接受范围且性能未出现退化。 |
| **安全考虑** | 代码路径仅涉及推理过程，没有新增网络或系统调用；但 LoRA 权重来源于外部适配器（可能是用户自定义），需确保 LoRA 权重文件来源可信。| 建议在生产环境中结合模型签名或 hash 校验，防止恶意 LoRA 权重注入导致模型行为异常。 |
| **可维护性** | - **代码分层清晰**：LoRA‑MoE 逻辑被封装在 `FusedMoEWithLoRA` 与 `TritonRunnerCoreWithLoRA`，不影响原 `FusedMoE`。<br>- **兼容性**：`lora_enabled=False` 时保持原有路径；`lora_enabled=True` 时禁用 fused 函数，防止冲突。<br>- **新增大量测试**，提供回归保障。| 需要维护人员注意 `MemPool` 中对 MoE 权重的 4‑维 shape 与切分逻辑，防止遗漏 `down_proj_moe`、`gate_up_proj_moe` 的特殊处理。 |

**⚠️ 潜在风险**  

1. **内存占用激增**  
   - MoE LoRA 需要为每个 expert 分配独立的 LoRA A/B 张量，形状从 `[max_loras, rank, hidden]` 变为 `[max_loras, num_experts, rank, hidden]`。在大模型（数百个 expert）和高 `max_loras_per_batch` 时，显存需求约提升 *num_experts* 倍。  
2. **对齐 kernel 的全局同步**  
   - `moe_lora_align_block_size` 在每个 batch 都会对所有 LoRA adapters做一次全量排序/重排，TP 场景下仍在单卡上执行，可能导致 CPU‑GPU 同步延迟。  
3. **不一致的切片逻辑**  
   - `slice_moe_lora_a_weights` / `slice_moe_lora_b_weights` 在 TP 环境下对 `gate_up` 与 `down` 使用不同的切片策略，若模型配置（如 `shared_expert_intermediate_size`）变化，现有切片代码可能产生 shape 错误。  
4. **兼容性风险**  
   - 某些自定义 LoRA 后端（未实现 `batch_info` 完整结构）在运行 `FusedMoEWithLoRA` 时会触发 `AttributeError`。  
5. **数值误差累积**  
   - LoRA delta 融合在两次 kernel 中进行，若使用低精度（int4/int8）后端，量化误差可能放大，特别是在极低 rank (r=1) 时。  

**💡 关注建议**  

| 建议 | 说明 |
|------|------|
| **显存评估** | 在部署前使用 `max_loras_per_batch` 与 `num_experts` 计算预估显存占用；必要时通过 `max_loras_per_batch=1` 或 **梯度切分**（分多次加载 LoRA）降低峰值。 |
| **配置校验** | 在启动时加入检查：`if lora_enabled and num_experts * max_loras_per_batch > 256` 给出警告或自动降采样。 |
| **对齐 kernel 优化** | 可考虑把 `moe_lora_align_block_size` 改写为支持 **分布式**（TP）版本，或在 `tp_size>1` 时使用 `torch.distributed.all_gather` 预先完成对齐，以降低单卡瓶颈。 |
| **LoRA 后端兼容** | 为自定义 LoRA 后端实现 `batch_info`（包括 `lora_ranks`、`seg_indptr`、`req_to_lora`、`adapter_enabled`）的统一抽象，避免运行时 `AttributeError`。 |
| **量化误差监控** | 在 LoRA‑MoE + 低位量化（int4/int8）环境下加入 `logprob` 差异阈值检查（比如 1e‑3），并在发现异常时回退到 FP16。 |
| **单元测试持续化** | 将 `test_lora_moe_tp_logprob_diff` 与 `test_lora_moe_vllm_sgl_logprob_diff` 加入 CI 的 nightly/large‑gpu 流程，确保未来任何 kernel 改动仍满足数值一致性。 |
| **安全审计** | 对用户提供的 LoRA 权重文件完成 SHA256 校验或模型签名验证，防止恶意 LoRA 注入模型后门。 |
