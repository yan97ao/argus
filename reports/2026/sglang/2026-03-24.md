### [HiCache][HybridModel]: Support mamba state offloading & HybridCacheController (#20457)
**SHA**: `0986bed` | 🔗 [查看提交](https://github.com/sgl-project/sglang/commit/0986bed8e279ce67659e4e473300fc3c5c717d6a)

**🎯 变更类型**：功能增强 / 架构变更 / 性能优化  
**⚡ 重要程度**：🔴高  

**📋 变更摘要**  
本次提交在 HiCache 中引入了 **HybridCacheController** 与对 **Mamba**（SSM）状态的层级缓存与离线存储支持，实现了 **Mamba 状态从 GPU → Host → 存储** 的完整 off‑loading 流程。为此新建了统一的 **PoolTransfer** 抽象、Host‑Mamba 缓存池、LRU 列表以及相应的迁移、回写、预取、撤销等逻辑，同时对多处调度、匹配、层级缓存等代码做了大量重构以兼容混合模型。测试用例加入了预取后再次评估的验证，benchmark 增加了心跳请求防止服务端因空闲被回收。

## 🔍 技术洞察  

### 架构影响  
1. **层级缓存统一抽象**  
   - 引入 `PoolTransfer`（含 `host_indices` / `device_indices` / `keys` / `hit_policy`）后，HiCache 不再只关注 KV 缓存，所有需要落盘或迁移的状态（如 Mamba）均通过同一通道描述。  
   - `HostPoolGroup` 将多种 Host‑Pool（KV、Mamba）统一管理，保证 **一次 allocate / free** 可同时为 KV 与 Mamba 分配对应 slot，降低手动同步出错概率。  

2. **HybridCacheController 替代 HiCacheController**  
   - 通过组合 `CacheOperation` 与 `StorageOperation`（支持 `pool_transfers`）实现写/读/预取的 **跨池协同**。  
   - 将写/读的 **分层归约（merge）** 与 **调度同步** 刷新到基类，实现了层级计数器 `layer_transfer_counter`（用于 Pipeline/PP 场景的层同步）。  

3. **LRU 结构细分**  
   - 原先只有 `full_lru_list` 与 `mamba_lru_list`，现在额外引入 `HostLRUList` 与 `mamba_host_lru_list`，分别管理 **GPU‑Mamba**、**Host‑Mamba** 的淘汰顺序。  
   - 对 **内部节点（tombstone）** 的处理更加细致：`_evict_device_leaf`、`_evict_to_host`、`_evict_host_leaf`、`_evict_regular` 等四类路径，保证 **KV 与 Mamba 状态的对应关系** 在任何阶段都保持一致。  

4. **去除 `last_host_backup_node`**  
   - 过去的实现需要额外的 “备份节点” 来记录到 L2 (Host) 的 KV，导致树结构冗余。新版直接用 `backuped` 与 `evicted` 两个布尔标记即可表达，同步到 Host 的 `host_value` 与 Mamba 的 `mamba_host_value`。  
   - 这简化了 **树结构的维护**，在层级回滚、删除枝桠时不必再遍历 `last_host_backup_node`。  

5. **存储层多池支持**  
   - `HiCacheFile`（本地磁盘实现）现在支持 **KV + Mamba** 双池的 `batch_exists_v2` / `batch_get_v2` / `batch_set_v2`。  
   - 对 `PoolHitPolicy`（ALL_PAGES vs TRAILING_PAGES）提供灵活的前缀匹配，Mamba 只需要 **尾部页面**（通常只有 1‑2 页）即可命中，降低整体命中阈值计算复杂度。  

### 性能影响  
| 场景 | 正向影响 | 潜在负面 |
|------|----------|----------|
| **写回 / 写穿** | 通过 `write_queue` 合并多池写操作，一次 CUDA memcpy 能同时搬迁 KV 与 Mamba，降低 kernel 启动次数。 | 若 `pool_transfers` 里有大量额外池，合并后单次 memcpy 规模变大，可能触发 **GPU memory bandwidth** 峰值，导致短暂延迟。 |
| **预取** | 在 `prefetch` 时已经可以一次性分配 Host‑Mamba slot，避免后续回填阶段再进行额外分配。 | 预取阶段若 Host‑Mamba 可用槽不足，会触发 **evict_mamba_host**，该路径的实现相对新，可能出现 **死锁**（在 `evict_host` 中调用 `evict_mamba_host` 同时锁住 LRU），需要压测。 |

### [Diffusion][NPU] Add support for Hunyuan3D (#20352)
**SHA**: `9f4d8ac` | 🔗 [查看提交](https://github.com/sgl-project/sglang/commit/9f4d8ac99f81f77097f365d3ebfa8f6c5e326d57)

**🎯 变更类型**：功能增强（为 Hunyuan3D 渲染器添加 NPU 支持）  
**⚡ 重要程度**：🟡 中  
**📋 变更摘要**：  
1. 通过 `is_cuda` 参数让 `_load_custom_rasterizer` 能在 CUDA 与非 CUDA（包括 NPU）环境下选择性编译/加载 GPU 代码。  
2. 在 C++ 层加入 `#ifdef CUDA_ENABLED` 条件编译，默认走 CPU 实现，避免在没有 CUDA 编译宏时链接 GPU 文件。  
3. 在 Python 渲染流水线中把 `device` 传递给渲染器，并在 NPU 场景下强制使用 CPU 实现，同时统一 tensor dtype 为 `float`。  

**🎯 影响范围**：  
- `python/sglang/multimodal_gen/csrc/render/hunyuan3d_rasterizer`（核心渲染 C++/CUDA 扩展）  
- `python/sglang/multimodal_gen/runtime/pipelines_core/stages/hunyuan3d_paint.py`（渲染管线入口）  

**💡 关注建议**  
1. **宏定义一致性**：当前在 `rasterizer.h` 中通过 `#ifdef CUDA_ENABLED` 定义 `__host__/__device__`，但 `custom_rasterizer_kernel` 编译时仍会传入 `-DCUDA_ENABLED`（可选）。请确保所有依赖 CUDA 的文件（如 `.cu`）在 NPU 环境下不被编译，否则会产生未使用的符号警告。  
2. **设备判断**：`device = "cpu" if pos.device.type == "npu" else pos.device.type` 只把 NPU 当作 CPU 处理，若后续加入原生 NPU 加速，需改为更通用的 device 抽象。  
3. **Tensor 迁移**：渲染后 `findices`、`barycentric` 通过 `.to(pos.device)` 回写，已兼容 NPU。建议在调用方添加显式 `contiguous()`，防止非连续张量在某些后端出现性能或错误。  
4. **性能基准**：CPU 分支在 NPU 环境下会退化为纯 CPU 计算，请在 CI 中加入 NPU‑CPU 对比基准，避免意外的慢速回退。  
5. **异常恢复**：`load_extension_with_recovery` 仍假定 CUDA 可用时加载 `.cu`，若在 NPU 机器上缺少 CUDA runtime，可能触发恢复路径失败。建议在 `is_cuda=False` 时显式禁用 `extra_cuda_cflags`（目前已通过 `cuda_enabled_flag`）并捕获编译异常以给出友好提示。  

总体而言，这次改动为 Hunyuan3D 在缺少 CUDA 的硬件（如华为 NPU）上提供了可用的 CPU 回退，实现了功能可达性。后续若要真正利用 NPU 加速，可在 C++ 层加入对应的算子实现或使用 `torch_npu` 库进行适配。

### [AMD] Tilelang sparse fwd for dsv32 mi355/mi300 (#19945)
**SHA**: `855d15a` | 🔗 [查看提交](https://github.com/sgl-project/sglang/commit/855d15adf6571e90b07bb851bbab512def1d3413)

**🎯 变更类型**：功能增强 / 性能优化  
**⚡ 重要程度**：🟡 中  

**📋 变更摘要**  
1. 为 `sparse_mla_fwd_decode_partial` 增加 `inner_iter` 与 `num_stages` 参数，实现一次 GPU Block 处理多个 KV‑tile 的流水线 (Pipelined) 方案，降低 kernel 启动次数并提升 LDS 利用率。  
2. 重新组织共享/fragment 缓存布局，使 Q 可选在 shared 或 fragment 中保存，配合增量 softmax（`alpha`、`m_i_prev`）实现在线归一化。  
3. 在 `tilelang_sparse_fwd` 中根据 AMD GPU 型号自动推算 `inner_iter`，并相应调整 `block_I`、线程数及 combine‑kernel 的 topk 参数。  

**🎯 影响范围**  
- `python/sglang/srt/layers/attention/nsa/tilelang_kernel.py`（核心稀疏注意力前向实现）。  
- 相关调用路径 `tilelang_sparse_fwd`，影响 AMD（HIP）平台的 decode 与 prefill 场景。  

**💡 关注建议**  
- **兼容性**：非 AMD 环境保持原实现，但建议在 CI 中加入对 `inner_iter>1` 的模拟测试，防止参数非法导致 `topk` 不可整除错误。  
- **性能验证**：在不同 GPU（gfx950、gfx942）上对比不同 `inner_iter`、`num_stages` 的执行时间与 LDS 使用率，确保调度逻辑（grid / CU 计数）真正提升吞吐。  
- **数值安全**：增量 softmax 中的 `alpha`、`sumexp` 初始值已在每轮清零，仍需检查极端 mask 全为负时的 `sumexp == 0` 分支是否导致 NaN。  
- **文档与 API**：为新参数添加注释及使用示例，防止上层调用忘记显式传递 `inner_iter`，导致意外回退到单 tile 版本。  
- **单元测试**：覆盖 `topk % (block_I*inner_iter)==0`、`inner_iter=1` 与 `>1` 两种路径，以及 `block_I` 不同取值的 edge case。  

通过上述检查可确保功能提升的同时不引入回归或数值异常。

### 关键风险与注意点

1. **线程池资源泄漏**  
   `BaseGrammarBackend` 在 `setUp` 中创建 `ThreadPoolExecutor`，测试在 `tearDown` 手动 `shutdown`。如果后续新增测试忘记调用 `shutdown`，可能导致 CI 在多进程模式下卡死。建议在 `BaseGrammarBackend.__init__` 中加入 `daemon=True` 或在 `__del__` 中安全关闭，或在测试基类统一实现 `addCleanup(self.backend.executor.shutdown, wait=True)`。

2. **全局注册表的副作用**  
   `GRAMMAR_BACKEND_REGISTRY` 在多个测试里被修改（覆盖、恢复）。`setUp/tearDown` 已保存并恢复原始字典，但若有并行测试（`pytest -n`），不同进程间仍会互相污染。可以考虑使用 `pytest.fixture(scope="function")` 的 `monkeypatch` 替代全局保存，或在 `register_grammar_backend` 中加入冲突检测。

3. **Mock 的 `spec` 兼容性**  
   多处 `MagicMock(spec=BaseGrammarBackend)`、`spec=BaseGrammarObject`。如果底层实现新增成员而未在测试中使用，`spec` 仍会通过；但若实现删除或改名，现有 mock 会在属性访问时报错，反而帮助捕捉 API 变化。保持 `spec` 用法是好事，但要留意 `spec_set` 与 `spec` 的差别，以免误把属性写成只读。

4. **多卡同步路径**  
   `test_multi_rank_sync_*` 直接 mock `torch.distributed.all_gather_object`，并手动设置 `grammar_sync_size=2`。若项目后续改动同步实现（例如改为 `all_gather` 而非 `all_gather_object`），这些测试将失效。建议在测试中使用 `patch.object(GrammarManager, "grammar_sync_size", 2)` 以保持对内部变量的依赖最小化。

5. **异常传播**  
   `GrammarManager.get_ready_grammar_requests` 中对 `Future.result()` 的异常直接抛出。单元测试 `test_future_exception_propagates` 验证了这一行为，确保运行时异常不会被吞掉。若后续在生产代码里改为捕获并转为 `InvalidGrammarObject`，需要同步更新测试逻辑。

6. **CPU‑only CI 注册**  
   每个测试文件都调用 `register_cpu_ci`，这会把测试标记为 “stage‑a‑cpu‑only”。如果项目的 CI 流程改为默认 GPU，可能导致这些测试被意外跳过。建议在项目根目录的 CI 配置中显式声明 `--ignore=...` 或使用 `pytest -m "cpu"`。

### 给开发者的建议

- **确保线程池安全**：在 `BaseGrammarBackend` 加入 `self.executor = ThreadPoolExecutor(max_workers=1, daemon=True)`，并在 `__del__` 中安全关闭，以免在非测试环境产生僵尸线程。  
- **避免全局注册表竞争**：使用 `pytest.fixture` + `monkeypatch` 替代手动 `dict` 复制恢复，或在 `register_grammar_backend` 加入 “warn if overwriting existing key” 提示。  
- **保持测试与实现同步**：新增/删除后端、改动 `GrammarManager` 的属性名时，及时检查对应的 mock 与断言。  
- **文档同步**：将 `is_legacy_structural_tag` 的行为说明（legacy 优先、必需字段）写入 `README` 或 API doc，防止用户误解。  
- **CI 透明度**：在 `pyproject.toml` 或 `setup.cfg` 中声明 `markers = cpu: run on CPU-only runners`，并在 CI 脚本里引用，防止迁移时遗漏。
