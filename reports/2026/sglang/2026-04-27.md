### [Whisper] Automatic language detection via structured generation (#22997)
**SHA**: `a3fc982` | 🔗 [查看提交](https://github.com/sgl-project/sglang/commit/a3fc982ba70bd1d27797c6db4e1f82518f79c106)

**🎯 变更类型**：功能增强 / 架构变更  
**⚡ 重要程度**：🔴高  

**📋 变更摘要**  
本次 PR 为 Whisper 语音转写引入**自动语言检测**能力，采用 SGLang 的结构化生成（regex）在一次前向传播中完成 “检测 + 转写”。核心实现包括：  
1. 在 `TranscriptionAdapter` 中新增 `supports_language_detection`、`build_fused_autodetect_params`、`parse_fused_output`、`strip_special_tokens` 接口。  
2. `WhisperAdapter` 实现上述接口，构造并正则匹配强制前缀，实现语言代码提取与前缀剥离。  
3. `OpenAIServingTranscription` 在请求构建、非流式与流式处理路径中注入 `_fused_autodetect` 标记，实现单请求路径与错误回退。  
4. `WhisperProcessor` 在融合模式下只发送 `<|startoftranscript|>`，其余前缀由 FSM 正则约束。  
5. 在服务器启动时通过 `warmup("whisper_autodetect")` 预编译两套 regex FSM，避免首请求延迟。  
6. 全面单元、集成测试覆盖非流式、流式、时间戳等四种组合及异常分支。  

**🎯 影响范围**  
- `python/sglang/srt/entrypoints/openai/serving_transcription.py`（请求构建、流式/非流式处理）  
- `python/sglang/srt/entrypoints/openai/transcription_adapters/base.py`（抽象基类）  
- `python/sglang/srt/entrypoints/openai/transcription_adapters/whisper.py`（Whisper 具体实现）  
- `python/sglang/srt/multimodal/processors/whisper.py`（解码前缀构造）  
- `python/sglang/srt/entrypoints/warmup.py`（预热 FSM）  
- 单元/集成测试文件  
- 相关协议、工具类的轻微调整  

**🔍 技术洞察**  

- **架构影响**  
  - **请求路径统一**：原本检测‑转写需要两轮 HTTP（先检测语言再转写），现在在 `TranscriptionAdapter` + `OpenAIServingTranscription` 中引入 **fused** 标记，实现“一次请求”。这在服务层保持向后兼容（仅在 `language=None` 且模型支持时激活）。  
  - **抽象层扩展**：在基类中加入 `supports_language_detection` 与一组新方法，保证未来若其他模型（如自研多语言模型）亦能复用同一机制。  
  - **FSM 预编译**：在 warmup 阶段提前触发 xgrammar 正则编译，防止首请求出现 15‑20 s 卡顿，提升可用性。  
  - **状态标记**：通过动态属性 `_fused_autodetect`、`_fused_ts_variant` 将检测模式信息从 `TranscriptionRequest` 传递至后续流式处理，保持业务逻辑分离。  

- **性能影响**  
  - **请求数下降 1 次**：合并检测与转写，降低网络往返、调度开销，对高并发场景提升明显。  
  - **额外 CPU 开销**：正则匹配与前缀剥离在每个响应（非流式和流式）中进行，成本极低（字符串截取 + 正则一次），几乎可以忽略。  
  - **启动延迟**：warmup 会在服务启动时多消耗约 30 s（两套 regex 各 15 s），但这只在首次实例化时发生，可通过配置关闭或在容器化环境预热镜像层来规避。  
  - **流式内存占用**：流式模式需维护 `visible_buffer`、`cumulative_text`，每条请求额外 O(L)（L 为累计文本长度）字符串缓冲，仍在可接受范围。  

- **安全考虑**  
  - **正则安全**：使用的语言代码集合是有限且来源于模型的词表，正则仅匹配固定前缀，未出现回溯攻击风险。  
  - **输入/输出消毒**：`strip_special_tokens` 与 `parse_fused_output` 对模型可能泄漏的 `<|...|>` 特殊标记进行严格过滤，防止把内部控制令牌误当作用户文本返回。  
  - **异常处理**：当强制前缀未能解析（FSM 中止、流提前结束）时会返回 SSE 错误帧，避免客户端误以为“空音频”。日志记录在 `warning`/`info` 级别，便于审计。  
  - **语言码校验**：`normalize_language_to_code` 扩展至全量 Whisper 语言码，防止非法代码被写入请求导致未定义行为。  

**⚠️ 潜在风险**  

1. **旧模型词表不兼容**  
   - 若使用不包含某些三字母语言 token（如 `yue`）的旧 Whisper checkpoint，`build_fused_autodetect_params` 仍会发送对应正则，FSM 能匹配但 `WhisperProcessor._get_language_token_id` 会在显式语言路径抛 `ValueError`。已在代码中加异常提示，但可能导致用户在显式指定不支持的语言时收到 500 错误。  
2. **流式前缀缓冲逻辑**  
   - 在 `incremental_streaming_output=True` 模式下，需要本地累积 `cumulative_text`。若后续修改 `TokenizerManager` 产生非累积文本（例如改为 token‑level分块），当前实现可能产生重复或缺失的 delta。  
3. **warmup 阶段阻塞**  
   - 在容器/无状态服务中若 warmup 被频繁触发（例如 autoscaling 场景），会导致实例快速重启时产生显著启动延迟。  
4. **日志噪声**  
   - 自动检测失败时会产生 `warning` 日志，若在噪声较大的生产环境中频繁出现（例如音频质量极差导致 FSM 中止），可能掩盖真实异常。  
5. **兼容性**  
   - 新增的 `_fused_autodetect` 动态属性未在 `TranscriptionRequest` 的 `dataclass` 定义中声明，理论上可能在类型检查或序列化（如 OpenAPI 文档生成）时缺失描述。  

**💡 关注建议**  

1. **文档与 SDK 更新**  
   - 在 OpenAI 接口文档中明确 `language=null`（或不传）将触发自动语言检测，说明可能的错误码 `language auto-detect failed`。  
   - SDK（Python/JS）提供 `detect_language` 参数或封装该路径，帮助用户避免手动构造 `language=None`。  

2. **启动预热可选化**  
   - 将 `warmup("whisper_autodetect")` 改为可配置开关，或在容器镜像构建阶段提前执行一次以把 FSM 编译结果写入持久化缓存，避免运行时阻塞。  

3. **异常统一返回**  
   - 将流式检测失败的 SSE 错误帧结构化为 OpenAI 标准 `error` 对象

### refactor(moe): centralize post-experts all-reduce skip predicate (#23748)
**SHA**: `85376a6` | 🔗 [查看提交](https://github.com/sgl-project/sglang/commit/85376a61190a58140f39bf11aee42dab283b64d0)

**🎯 变更类型**：重构  

**⚡ 重要程度**：🔴高  

**📋 变更摘要**  
1. 新增 `should_skip_post_experts_all_reduce` 工具函数，统一所有 MoE 模型中 “后专家（post‑experts）全规约（all‑reduce）” 的跳过判定逻辑。  
2. 在 `__init__.py` 中导出该函数并更新导出列表。  
3. 所有涉及 TP/EP all‑reduce 的模型（bailing_moe、deepseek_v2、glm4_moe、hunyuan_v3、llama4 等）改为调用统一函数，移除原先散布的多维度判断。  

**🎯 影响范围**  
- `sglang/srt/layers/moe` 包（`__init__.py`、`utils.py`）  
- 多数 MoE 相关模型文件：`bailing_moe.py`, `bailing_moe_linear.py`, `deepseek_v2.py`, `exaone_moe.py`, `glm4_moe.py`, `hunyuan_v3.py`, `llada2.py`, `llama4.py`, `mimo_v2_flash.py`, `minimax_m2.py`, `qwen2_moe.py`, `qwen3_moe.py`, `sarvam_moe.py`, `sdar_moe.py`, `step3p5.py` 等  

**🔍 技术洞察**  

- **架构影响**  
  - 将分散在多个模型文件中的 all‑reduce 跳过条件抽取到 `utils.py`，形成单一真值源，降低了代码重复度。  
  - 通过显式的函数签名（`is_tp_path`, `use_reduce_scatter`, `should_allreduce_fusion`）使得调用方必须明确上下文，提升了可读性与可维护性。  
  - 对层级通信（`LayerCommunicator`）的行为与调度器（dispatcher）路径的耦合关系被显式化，便于未来新增路径（如新的压缩 All‑Gather）时只需在此函数中补充判定。  

- **性能影响**  
  - **正向**：统一判定避免了因散布实现不一致导致的冗余 all‑reduce，保持原有的最优路径（如 DP reduce‑scatterv、flashinfer‑cutlass FP4 all‑gather、All‑reduce 融合）。  
  - **负向**：函数调用本身的开销极小（仅几条布尔判断），对总体推理吞吐几乎没有影响。  
  - 通过消除潜在的误判（例如遗漏 `should_use_flashinfer_cutlass_moe_fp4_allgather`），可以防止不必要的额外通信，从而在大规模 TP 场景下提升带宽利用率。  

- **安全考虑**  
  - 该变更纯属调度与通信逻辑，不涉及外部输入或加密路径，未引入新的安全风险。  
  - 需要确保函数内部不泄露内部状态或调用栈信息，以免在高度受限的部署环境中产生信息泄露。当前实现仅返回布尔值，符合安全要求。  

**⚠️ 潜在风险**  

1. **语义回归**：旧的散布条件在每个模型中可能存在细微差别（如默认 `use_reduce_scatter=False`），若某个模型在调用时遗漏必需参数，可能导致原本应被跳过的 all‑reduce 被执行，进而引起额外的通信开销或数值差异。  
2. **参数传递错误**：`is_tp_path` 必须在 TP 场景为 `True`，在 EP 场景为 `False`。误传会导致不恰当的跳过或执行。  
3. **兼容性**：如果外部用户直接 import `should_use_dp_reduce_scatterv`（已被废弃）且未更新依赖，可能出现 `ImportError`。需要在发布说明中明确迁移路径。  
4. **测试覆盖不足**：统一函数的行为需要在各种组合（TP/EP、reduce‑scatter、fusion、flashinfer）下验证，缺失覆盖会导致隐藏的逻辑错误。  

**💡 关注建议**  

- **单元/集成测试**：为 `should_skip_post_experts_all_reduce` 增加参数化测试，覆盖所有四个判定分支以及 TP/EP 两种路径，确保与旧实现结果完全一致。  
- **文档与迁移指南**：在 `README` 或代码注释中说明该函数是旧判定的统一入口，建议所有自定义插件改用新接口。  
- **审计调用方**：全局搜索是否还有残留的 `should_use_dp_reduce_scatterv` 或 `should_use_flashinfer_cutlass_moe_fp4_allgather` 直接判断的代码，确保全部迁移。  
- **性能基准**：在多节点、多卡（TP=8、EP=4）环境下跑一次全链路基准，对比 all‑reduce 开销前后差异，确认无意外回退。  
- **回滚策略**：保留旧函数的实现（可标记为 `@deprecated`) 并在必要时通过配置开关临时回退，降低发布风险。  

---  

此次重构通过“中心化判定”提升了代码可维护性并降低了潜在的通信冗余风险，属于底层调度层面的重要改进，建议在下一次发布中同步更新相关文档与测试套件。

### [NPU] Support MTP for Qwen3.5 (#20918)
**SHA**: `32c3513` | 🔗 [查看提交](https://github.com/sgl-project/sglang/commit/32c3513816b0e4ecb4d0fd4e8d630ea30e0d8ffc)

**🎯 变更类型**：功能增强 / 架构变更  
**⚡ 重要程度**：🔴高  

**📋 变更摘要**  
- 为 NPU（Ascend）平台加入 Qwen3.5/ Qwen3‑Next 的 Multi‑Token Prediction（MTP）支持，新增 Ascend‑GDN 与 Ascend‑Hybrid‑Linear Attention 后端实现。  
- 引入环境变量 `DEEP_NORMAL_MODE_USE_INT8_QUANT` 与内部 `SGLANG_DEEPEP_BF16_DISPATCH`，在 NPU 上可自行控制 INT8 量化与 BF16 分派。  
- 在内存池、LayerNorm、模型前向等关键路径加入 NPU 专属的状态初始化、卷积缓存布局与 GEMM‑RMSNorm 加速实现，并通过 `ExitStack` 动态关闭量化。  

**🎯 影响范围**  
- `sglang/srt/hardware_backend/npu/attention/*`：全新 Ascend GDN 与 Hybrid‑Linear Attention 后端。  
- `sglang/srt/layers/attention/attention_registry.py`：后端注册逻辑分支，挂载 NPU 实现。  
- `sglang/srt/layers/layernorm.py`、`sglang/srt/models/qwen3_5_mtp.py`、`sglang/srt/models/qwen3_next_mtp.py`：NPU‑专属前向路径与量化开关。  
- `sglang/srt/mem_cache/memory_pool.py`、`sglang/srt/hardware_backend/npu/memory_pool_npu.py`：卷积状态在 NPU 上的内存布局。  
- 环境配置 `python/sglang/srt/environ.py`：新增 `DEEP_NORMAL_MODE_USE_INT8_QUANT`。  

**🔍 技术洞察**  

- **架构影响**  
  - 新增 **AscendMambaAttnBackendBase**、**AscendHybridLinearAttnBackend**、**AscendGDNAttnBackend**，形成 “全模型 + 线性‑Mamba” 双路注意力的硬件抽象层。  
  - 在 `attention_registry` 中通过 `is_npu()` 动态切换实现，保持对 CPU / CUDA 仍然兼容，保持代码路径清晰。  
  - 内存池为 NPU 调整卷积缓存维度顺序（`dim` 迁至最后），并在 `memory_pool_npu.py` 中统一创建带 **draft‑step** 扩展的 conv_state，兼容 MTP 的多步预测。  

- **性能影响**  
  - **卷积‑Mamba** 相关算子使用 NPU 原生 `torch.ops.npu.*`（如 `causal_conv1d_update`、`recurrent_gated_delta_rule`），避免 CPU‑GPU 多次拷贝，理论上可提升 1.5‑2× 推理吞吐，尤其在 **MTP** 场景（一次生成多 token）下收益更明显。  
  - `LayerNorm` 改为调用 `add_gemma_rms_norm`（NPU 专属实现），在 BF16 环境下减少显存占用并提升计算密度。  
  - 通过 `envs.SGLANG_DEEPEP_BF16_DISPATCH.override(True)` 与 `DEEP_NORMAL_MODE_USE_INT8_QUANT` 控制量化路径，可在模型未量化时强制使用 BF16，避免精度回退。  

- **安全考虑**  
  - 变更均为算子调度与内部状态管理，未引入网络、文件 I/O 或外部依赖，安全风险极低。
