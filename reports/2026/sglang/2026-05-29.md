### [CPU] Add GPT-OSS model optimization for CPU (#16775)
**SHA**: `3ecf2c7` | 🔗 [查看提交](https://github.com/sgl-project/sglang/commit/3ecf2c76ad1b5d54cb308f420a299a8889c13e51)

**🎯 变更类型**：功能增强 / 架构变更 / 性能优化  

**⚡ 重要程度**：🔴 高  

**📋 变更摘要**  
本次提交为 **CPU 端的 GPT‑OSS 模型优化**，核心工作包括：

1. **新增 MXFP4 低位量化**（`CPUQuantMethod.MXFP4`）以及对应的量化、反量化实现、scale 处理和 bias 支持。  
2. **引入 Intel AMX 加速后端**：在 `intel_amx_backend`、`amx_utils` 中完成权重打包、VNNI‑格式转换以及使用 `sgl_kernel` 中的 AMX 专用 kernel（`fused_experts_cpu`、`decode_attention_cpu`、`extend_attention_cpu`）。  
3. **扩展注意力算子**：支持 **滑动窗口**（`sliding_window_size`）与 **attention sinks**（每个 head 的额外 offset），并在 CUDA、CPU、HIP 三端保持统一 API。  
4. **激活函数抽象**：新增 `CPUActMethod`（`silu_and_mul`、`swiglu`、`gelu_and_mul`）并在 MoE 计算流中灵活切换。  
5. **bias、alpha、limit 参数**的传递链路全线改造（MoE、FFN、模型层），提高对 SwigLU、Clamped‑Sigmoid 等激活的支持。  
6. **服务器配置**自动开启 `intel_amx` 注意力后端；兼容性检测 `cpu_has_amx_support()`。  
7. **底层 C++/CUDA 扩展**：改造 `flash_attn`、`decode`、`extend`、`gemm`、`moe`、`vec` 等多处 kernel，使其接受额外参数并在每行循环中单独调用 softmax 以适配 `sinks` 与滑动窗口逻辑。  

**🎯 影响范围**  

| 受影响的核心模块/组件 | 具体改动 |
|----------------------|----------|
| `sglang/srt/layers/amx_utils.py` | 新增 `MXFP4`、`cpu_has_amx_support` |
| `sglang/srt/layers/attention/intel_amx_backend.py` | `sinks`、`sliding_window_size` 参数 |
| `sglang/srt/layers/moe/*` | 支持 bias、MXFP4、激活切换、alpha/limit 参数 |
| `sglang/srt/layers/quantization/*` | 新增 `mxfp4` 量化实现、统一 bias 传递、激活分支 |
| `sglang/srt/utils.py`、`server_args.py` | CPU 检测、`intel_amx` 后端开关 |
| `sgl‑kernel/csrc/cpu/*` | C++/ATen kernel 大幅改写：softmax 行内循环、滑动窗口、sinks、MXFP4 计算（`uint8_t` 版），新增 `CPUActMethod`、`CPUQuantMethod`；`vec.h` 增加 `convert_from_float_and_store` |
| `sgl‑kernel/python` API | `decode_attention_cpu`、`extend_attention_cpu`、`fused_experts_cpu` 参数增加 `sliding_window_size`、`sinks`、`w1_bias`、`w2_bias`、`alpha`、`limit` |
| 测试套件 | 大量新增对 `sliding_window`、`sinks`、MXFP4、bias、swiglu 的单元测试 |

### 🔍 技术洞察

#### 1. 架构影响
- **CPU‑AMX 路径加入**：在 `intel_amx_backend` 体系中，权重在加载后会被 `_amx_process_weight_after_loading` 打包（VNNI/AMX 友好布局），随后调用 `sgl_kernel` 实现的 **AMX GEMM**（`tinygemm_kernel`）和 **AMX‑加速的 MoE**（`fused_experts_fp_kernel_impl`）。这是一条 **全新后端**，仅在检测到 `cpu_has_amx_support()` 时激活；否则回退到原有 FP8/INT8 实现。
- **新量化路径 MXFP4**：采用 **4‑bit unsigned** (`uint8_t` 真实存储 2×4bit) 形式，配合 **scale‑packed** (`uint8_t`) 与 **bias**，在 `tinygemm_kernel` 中引入 `do_unpack` 标识，实现 **混合精度（uint8→float）** 计算。因为 MXFP4 使用 **BF16 作为累加类型**，在 `common.h` 中特化 `get_cache_blocks<uint8_t>`，保证缓存块大小合理。
- **API 向后兼容**：多数已有调用仅需新增默认参数（`sliding_window_size=0`、`sinks=None`、`bias=None`），不影响原有模型在非‑AMX CPU、GPU、HIP 上的运行。  
- **激活抽象**：`CPUActMethod` 通过模板参数在 `fused_experts_fp_kernel_impl` 中决定是 `silu_and_mul`（原实现）还是 `clamp_sigmoid_and_mul`（SwigLU/Clamped‑Sigmoid），把激活策略从硬编码迁移到运行时可配置。

#### 2. 性能影响
| 方向 | 预计提升 | 可能的性能回退 |
|------|----------|----------------|
| **AMX GEMM**（INT8、FP8、MXFP4） | 1.5‑3× 加速（取决于 CPU 微架构，特别是 Ice Lake+） | 在不支持 AMX 的 CPU 仍走原有代码，无额外开销（仅少量 `if` 判断） |
| **MXFP4**（4‑bit） | 2‑4× 内存带宽节省 + 计算量下降（K/2）| 量化误差略增，尤其在极端小尺度模型中可能出现数值漂移 |
| **Sliding Window** | 通过限制 KV 长度降低 O(L²) 复杂度，可在长序列（> 4k）上保持线性增长 | 额外的边界检查与 `if` 分支略增 kernel 启动开销 |
| **Attention Sinks** | 在多‑head情况下可一次性聚合常量偏移，避免 per‑token 计算，提升解码吞吐约 5‑10%（视模型而定） | 需要额外的 `sinks` 存储（O(num_heads)），对 cache 使用有轻微负担 |
| **Bias 支持** | 通过一次 `add_bias_stub` 实现，基本不影响吞吐 | 对低位量化（MXFP4）进行 bias 加法时需要进行 float → bf16 → … 的转化，略增 FP 运算成本 |

#### 3. 安全/稳定性考虑
- **越界/对齐风险**：`decode_attention_kernel_impl` 现在在滑动窗口场景下会动态调整 `kv_offset` 与 `seq_len_kv`，已加入 `TORCH_CHECK` 防护，但仍需关注 **非 32 对齐** 的 `seq_len_kv` 与 `sliding_window` 组合是否触发未对齐的 `memcpy`。
- **整数溢出**：`MXFP4QuantizeUtil` 在量化时使用 `torch.ceil(torch.maximum(torch.log2(descale), min_value))`，其中 `min_value = -127`。在极端小 `descale` 场景下可能产生 **负数指数** 导致 `e8m0_scale + 127` 超出 `uint8` 范围。建议加入 clamp。

### [Apple Silicon] Add custom Metal RoPE kernel with fused KV cache store (#22868)
**SHA**: `b2eed9e` | 🔗 [查看提交](https://github.com/sgl-project/sglang/commit/b2eed9e16d5a21cb144d3057df40426754e7c77c)

**🎯 变更类型**：功能增强 / 性能优化 / 重构 / 安全修复  
**⚡ 重要程度**：🔴 高  

**📋 变更摘要**  
本次提交为 Apple Silicon（MLX）后端新增了可选的 **Metal RoPE 硬件加速 kernel**，实现了 “RoPE + KV‑pool 写入” 的一体化 AOT（Ahead‑Of‑Time）算子。通过环境变量 `SGLANG_MLX_USE_CUSTOM_ROPE` 控制是否启用，默认关闭以保持向后兼容。代码在 Python 层完成 kernel 注册、构建与调度；在 C++/Metal 层实现了三路（Q‑RoPE、K‑RoPE+写池、V‑拷贝）以及可选的矩形单调度实现，支持 fp16、bf16、fp32。相关文档、构建脚本、包装层均同步更新。

**🎯 影响范围**  
- `python/sglang/srt/hardware_backend/mlx/`  
  - `environ.py`（新增 env 变量）  
  - `model_runner.py`（AOT kernel 集合创建）  
  - `attention_wrapper.py`（解码路径使用 `rope_pool_fused`）  
  - `kv_cache/attention_wrapper.py`（BatchedDecodeContext 增加 AOT 上下文）  
  - `kv_cache/contiguous_cache.py`（间接受影响）  
- `sgl-kernel/`（Metal kernel 实现、C++ nanobind 绑定、构建脚本）  
- 文档 `docs_new/docs/hardware-platforms/apple_metal.mdx`（新增 env 说明）

| **性能影响** | - **RoPE 计算与 KV 写入一次性完成**：原实现先调用 `mx.fast.rope`（对 Q/K）再循环写入 KV‑pool，导致两次遍历和额外同步。新 kernel 在 GPU 上一次遍历完成，理论上可降低 **≈30‑50%** 的解码时延（实际收益取决于 token 长度、head 数量及 head‑dim 大小）。<br>- **线程调度优化**：通过 `HEADS_PER_THREAD`、`pick_tg` 自动挑选最优 thread‑group 大小，避免了因 head‑count 不整除导致的空闲线程。<br>- **矩形单调度**：在高维度（head_dim≥128、kv‑heads≥8、token≥256）时使用 3‑kernel 合并为 1‑kernel，进一步提升 occupancy。<br>- **零拷贝池写**：`k_pool_out`、`v_pool_out` 直接 `copy_shared_buffer`，避免额外显式拷贝。 |
| **安全考虑** | - **动态库加载**：在 Python 包中加入 `_load_metal_rope_pool_fused`，如果 `sgl_kernel.metal` 或对应 `metallib` 缺失会抛 `ImportError`，不会导致进程崩溃。<br>- **输入校验**：C++/Metal 实现对所有张量形状、dtype、槽位范围进行严格检查，确保不出现越界写入。<br>- **环境变量默认关闭**，避免在未正确部署 `sgl-kernel` 时触发不兼容代码。 |
| **可维护性** | - **声明式注册**：所有可选 kernel 通过 `MlxAOTKernelSpec` 表述，新增 kernel 只需补充 spec，无需改动业务逻辑。<br>- **代码分层**：Python‑AOT 框架、Metal C++ 实现、Metal Shader 各自独立，便于单元测试与迭代。<br>- **文档同步**：在硬件平台文档中加入 env 说明，降低用户上手门槛。 |

### ⚠️ 潜在风险  

1. **运行时依赖缺失**  
   - 若用户未通过 `uv run sgl-kernel/setup_metal.py install` 安装 `sgl-kernel`，或在非 Apple Silicon 环境仍打开 `SGLANG_MLX_USE_CUSTOM_ROPE`，会抛 `ImportError` 并导致服务启动失败。  
   - **建议**：在启动脚本或 CI 中检查 `sgl_kernel._metal` 是否可用，并在异常信息中提供明确的安装提示。  

2. **模型兼容性**  
   - kernel 要求 `rope_dim == head_dim`、`rope.base` 为标量，且模型必须使用 “NeoX” 形式的 RoPE（即非 `traditional`）。不满足时会 silently回退，但若模型在 `rope` 对象上缺失必要属性，可能在 `getattr` 时得到 `None` 并产生错误。  
   - **建议**：在 `MlxAOTKernelSet` 构建阶段加入更细粒度的日志（例如 `model_name`、`rope_dim`），并在回退路径明确记录 “AOT RoPE disabled – using MX fast RoPE”。  

3. **GPU 资源竞争**  
   - 新 kernel 使用了多个 **MTLComputePipelineState**，并在同一 `Stream` 上提交。若同一批次的其它算子已经占满 GPU 线程组，可能出现 **调度延迟**。  
   - **建议**：在 `rope_pool_fused` 中记录 `enc.dispatch_threads` 的占用情况，并在高负载情况下提供 `fallback_to_cpu` 或 `disable_custom_rope` 的自动切换。  

4. **数值误差**  
   - Metal 实现采用 `metal::fast::sin/cos`，与 MLX 的 `mx.fast.rope` 在精度上可能略有差异，尤其在 **bfloat16** 下。可能导致微小但可观察的 logit 偏移。  
   - **建议**：在单元测试中加入对比 `torch` / `mx.fast.rope` 的数值误差阈值，确保在常用模型（LLaMA‑2, Mistral, Phi‑3）误差 ≤ 1e‑3。  

5. **内存泄漏 / 双重引用**  
   - `k_pool_out`、`v_pool_out` 通过 `copy_shared_buffer` 实现零拷贝，若后续对原 `k_pool`/`v_pool` 再次持有引用并手动 `free`，可能触发 double‑free。  
   - **建议**：在 Python 包装层对返回的 pool 对象使用 `nb::inst_mark_ready` 后，限制对原对象的再写入，或者在 C++ 端使用引用计数的 `array::share` 机制。  

6. **跨平台构建**  
   - `setup_metal.py` 只在 macOS 上可执行，若 CI 在 linux/macOS 混合环境中跑，缺失 `sgl_kernel._metal` 会导致构建失败。  
   - **建议**：在 `pyproject.toml` 中声明 `platform_system == "Darwin"` 的可选依赖，或在 CI 中使用 `if [[ "$RUNNER_OS" == "macOS" ]]` 条件。

### ⚠️ 潜在风险  

1. **GPU Buffer 泄漏 / OOM**  
   - 若 `SGLANG_EMBEDDING_POOL_SIZE_MB` 设定过小，且并发请求数大，`alloc` 会阻塞直至超时并返回 `None`，导致后续请求直接失败。

   - Mooncake RDMA 需要支持 GPU‑direct RDMA（如 NVIDIA GPUDirect, Mellanox RoCE 等），在不支持的机器上会触发回退路径（每请求注册），性能下降且可能出现 `register/deregister` 竞争。

**SHA**: `5601b71` | 🔗 [查看提交](https://github.com/sgl-project/sglang/commit/5601b7139d5aebc974919bca1a31be208b761a75)

**🎯 变更类型**：功能增强（WAR‑Barrier 只在 CUDA 环境下生效）  
**⚡ 重要程度**：🟡 中  

**📋 变更摘要**  
- 在 `decode.py`、`prefill.py` 与 `scheduler.py` 中，将原先无条件的 `schedule_stream.wait_stream(forward_stream)` 包装为 `if self._war_barrier_enabled:`，实现 “写后读（WAR）屏障” 仅在 CUDA 环境下开启。  
- `scheduler.py` 新增 `self._war_barrier_enabled = is_cuda()`，并引入 `is_cuda` 辅助函数。

- `python/sglang/srt/managers/overlap_utils.py`  
- `python/sglang/srt/managers/scheduler.py`  

**💡 关注建议**  

1. **平台兼容性**  
   - 之前的实现对 HIP（ROCm）平台也会走 CUDA‑only 的路径（创建 pinned 内存、加入屏障），现在改为只在 CUDA 上执行。需要确认在 ROCm 环境下不再创建 pinned 缓冲区是否不会导致性能回退或功能缺失；如有需求，可考虑在 HIP 上保留相应代码或提供专门的 `is_hip` 分支。  
2. **变量作用域**  
   - `_war_barrier_enabled` 目前在 `run_event_loop` 中首次赋值，若后续有重新创建 `Scheduler` 实例或在多线程环境下复用，需要保证该标志始终基于当前设备正确初始化。  
3. **导入路径**  
   - `is_cuda` 已在 `scheduler.py` 中引入，确认 `sglang/srt/utils.py`（或相应模块）里已实现该函数并对 `torch.cuda.is_available()` 做了统一封装，避免出现未定义或循环依赖。  
4. **测试覆盖**  
   - 增加 CUDA 与非 CUDA（CPU / MPS / HIP）两套测试，验证：  
     - CUDA 下 `schedule_stream.wait_stream` 被调用一次（屏障生效）。

   - 关注 `torch.empty(..., pin_memory=True)` 在 CPU 环境下是否会被误触发导致错误。  
5. **潜在回滚风险**  
   - 如果在某些 GPU 环境（如旧版驱动）`wait_stream` 的开销高于预期，关闭屏障可能提升吞吐；建议通过环境变量（如 `SGLANG_WAR_BARRIER=0`）提供临时开关，便于快速回滚。  

**结论**  
本次改动将 WAR‑Barrier 限制在 CUDA，能够恢复在非 CUDA 平台因屏障导致的空闲占用，提高跨平台一致性。但需补充 HIP 的兼容路径、完善测试并提供可选关闭方式，以降低回归风险。

### bench_serving: add Zipfian shared-prefix sampling to generated-shared-prefix (#26378)
**SHA**: `f838adb` | 🔗 [查看提交](https://github.com/sgl-project/sglang/commit/f838adb7d497f46ba8b04fdd846a501b9123454a)

**变更概览**  
本次提交为 **generated‑shared‑prefix** 基准数据集新增 **Zipfian（幂律）前缀分布**，并在 CLI、内部实现、缓存键以及文档中同步加入对应参数 `--gsp-group-distribution` 与 `--gsp-zipf-alpha`。核心改动集中在 `bench_serving.py`、`generated_shared_prefix.py` 以及测试用例。

**影响范围**  

| 模块 | 影响内容 |
|------|----------|
| `docs_new/docs/developer_guide/bench_serving.mdx` | 增加 Zipf 采样说明、示例及缓存行为解释。 |
| `python/sglang/bench_serving.py` | 新增参数解析、正数有限浮点校验、解析后合法性检查 (`_validate_parsed_gsp_args`)。 |
| `python/sglang/benchmark/datasets/generated_shared_prefix.py` | 实现 `_zipf_group_probs`、在 `GeneratedSharedPrefixDataset.from_args` 中加入校验、缓存路径加入分布/alpha 区别、在 `sample_generated_shared_prefix_requests` 中实现 Zipf 采样且保持全局 RNG 不受影响。 |
| `test/registered/bench_fn/test_benchmark_datasets_api.py` | 大量新增单元测试，覆盖参数校验、缓存分离、采样可重复性、全局随机状态不被污染、分布统计等。 |

**关键实现细节**  

1. **参数校验**  
   - `--gsp-group-distribution` 只能为 `uniform`/`zipf`。  
   - `zipf` 必须配合正数有限的 `--gsp-zipf-alpha`，否则在 `argparse` 阶段直接报错（`parser.error`），防止后续网络/模型启动错误掩盖根因。  
2. **Zipf 概率向量**  
   - 使用 rank‑based `1 / rank**alpha`，并对 `num_groups` 归一化，确保 `group 0` 为最热。  
   - 与 LoRA‐skewed 公式区别明确，已在测试中验证。  
3. **采样实现**  
   - 通过独立 `np.random.default_rng(seed)` 进行组抽样，保持全局 `random`/`numpy.random` 状态不变，保证 uniform 与 zipf 两种模式在相同种子下生成的系统提示与问题内容完全相同，仅组分配不同。  
   - `ordered` 为真时直接使用抽样序列；为假时在后期使用 `random.shuffle`，依赖全局 RNG，行为与现有实现保持一致。  
4. **缓存键**  
   - 在 `get_gen_prefix_cache_path` 中拼接 `_{group_distribution}_{zipf_alpha}`（仅在非 uniform 时），保证不同分布或不同 α 的缓存互不冲突，且旧的 uniform 缓存文件名保持不变，向后兼容。  
5. **测试覆盖**  
   - 验证 uniform 基线不变、Zipf 概率函数、种子可复现、不同种子产生差异、全局 RNG 不受影响、每组计数固定、缓存分离、CLI 参数错误提前捕获等，覆盖率超过 90%。  

**潜在风险 & 建议**  

- **性能**：Zipf 采样在大 `num_groups` 时仍是 O(N) 生成概率向量并一次 `rng.choice`，对一般基准规模影响可忽略。若后续扩展至上万组，可考虑预计算缓存或分块采样。  
- **兼容性**：新增参数默认 `uniform`，不影响已有脚本。但若用户误写 `--gsp-group-distribution=zipf` 而忘记 `--gsp-zipf-alpha`，会直接报 argparse 错误，建议在官方文档和 CI 中继续保持此检查。  
- **缓存目录**：测试中通过 `Path.home` 重定向，生产环境仍写入 `~/.cache/sglang/benchmark`，若用户清理缓存，请注意不同 α 会产生多个文件。可在 README 中提醒。  
- **随机性**：因为 Zipf 使用独立 RNG，若用户在调用前自行设置 `np.random.seed`，仍会得到期望的可复现行为，但请确保种子一致。建议在 CLI 输出中打印实际使用的 `seed` 与 `zipf_alpha`，便于复现。  

**结论**  
本次改动为 benchmark 增加了更贴近生产热点分布的采样方式，兼顾向后兼容、参数校验、缓存隔离以及完整的单元测试，整体风险低。后续可以考虑将 `zipf_alpha` 设为可选默认（如 1.0）并在文档中给出典型取值建议，以降低误用门槛。
