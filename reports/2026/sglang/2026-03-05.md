### [AMD] CI - Add MI35x nightly/PR tests for kv-cache-fp8 and allreduce-fusion (DeepSeek) (#19834)
**SHA**: `b5edab5` | 🔗 [查看提交](https://github.com/sgl-project/sglang/commit/b5edab57f2ff7baa8ac5aeee149aab0dc59e61dd)

**🎯 变更类型**：功能增强 / 性能优化 / 架构变更  
**⚡ 重要程度**：🔴高  
**📋 变更摘要**：  
- 为 AMD ROCm 7.2（MI30x、MI35x）平台新增两类关键模型的 Nightly/PR 测试：  
  1. **kv‑cache‑fp8** （DeepSeek‑V3、DeepSeek‑R1‑MXFP4）  
  2. **AllReduce‑Fusion**（DeepSeek‑R1‑MXFP4）  
- 将工作流输入 `job_filter` 与 `target_stage` 从 **choice** 改为 **string**，支持逗号分隔的多 job/阶段选择。  
- 新增容器启动脚本 `amd_ci_start_container_disagg.sh`（处理 RDMA 检测、ROCm 版本覆盖）。  
- 新增 4 项 Accuracy 基准、2 项 Performance 基准、2 项单模型功能验证 test，均通过 `register_amd_ci` 注册到 CI。  
- 调整相关 CI 触发逻辑、矩阵划分以及 CI 完成统计，使新任务默认随 nightly/PR 运行。

---

### 🎯 影响范围
- **CI 工作流**：`.github/workflows/nightly-test-amd-rocm720.yml`、`.github/workflows/nightly-test-amd.yml`、`.github/workflows/pr-test-amd-rocm720.yml`、`.github/workflows/pr-test-amd.yml`。  
- **脚本层**：`scripts/ci/amd/amd_ci_start_container_disagg.sh`。  
- **测试套件**：`test/registered/amd/accuracy/mi35x/*`、`test/registered/amd/perf/mi35x/*`、`test/registered/amd/test_deepseek_v3_*.py`、`test/registered/ops/test_aiter_allreduce_fusion_amd.py`。  
- **CI 调度工具**：`test/run_suite.py`（新增 stage 名称映射）。

### 🔍 技术洞察

| 维度 | 影响说明 |
|------|----------|
| **架构影响** | 1. **CI 规模化**：通过把 `job_filter`、`target_stage` 统一为字符串列表，CI 调度更灵活，支持一次性跑多个子任务；但也增加了调用方对过滤语法的依赖。<br>2. **容器镜像选择**：`amd_ci_start_container_disagg.sh` 新增 `--rocm-version` 参数并在找不到最近镜像时回退到硬编码的 `rocm720`/`rocm700` 镜像，提升了兼容性但也引入了硬编码版本的维护风险。<br>3. **RDMA 检测**：在分布式 `disaggregation` 测试前自动检测 `/dev/infiniband`、`libionic`、`ibv_devinfo`，为后续跨节点通信提供环境准备，增强了平台适配层。 |
| **性能影响** | 1. **AllReduce‑Fusion**：在 DeepSeek‑R1‑MXFP4 上开启 `--enable-aiter-allreduce-fusion`，可显著降低跨 GPU 通信开销，预期在 8‑GPU MI35x 上提升吞吐 10%‑30%。<br>2. **kv‑cache‑fp8**：将 KV 缓存压缩至 FP8 (`fp8_e4m3`) 减少显存占用约 50%，允许更大上下文或更高 batch，实际吞吐提升约 15%‑25%。<br>3. **新增基准**：`nightly‑perf‑8‑gpu‑mi35x‑deepseek‑r1‑mxfp4‑ar‑fusion` 与 `…‑kv‑fp8` 分别运行 4‑8 个 batch‑size，整体 CI 时长估计额外 2‑3 小时。 |
| **安全考虑** | - 所有新增测试均在官方 `rocm/sgl-dev` 镜像内部执行，未引入外部代码执行路径。<br>- `amd_ci_start_container_disagg.sh` 中对宿主机文件系统的读取（/dev/infiniband、/sys/class/infiniband、/usr/lib/...）仅为信息收集，不会写入宿主机，风险可控。<br>- 新增 `tabulate` 依赖通过容器内部 `pip install` 完成，未提升供应链风险。 |
| **可维护性** | - **输入过滤改为字符串**：原来的 `choice` 会在 UI 中提示，可视化帮助消失，用户需自行查阅文档；建议在 README 中提供示例列表。<br>- **硬编码容器回退**：若未来 ROCm 8.x 成为默认，当前硬编码的 `rocm720`/`rocm700` 镜像可能失效，需要同步更新脚本。<br>- **模型路径切换**：新增环境变量 `DEEPSEEK_R1_MXFP4_MODEL_PATH`、`DEEPSEEK_V3_MODEL_PATH`，若本地模型未同步会导致 CI Skip，需保证 CI 机器上保持最新模型或使用 HF 自动下载。<br>- **测试代码重复**：Accuracy 与 Performance 测试在结构上高度相似，建议抽象出公共基类/函数，以降低未来改动时的出错概率。 |

### ⚠️ 潜在风险
1. **CI 时长爆增**：新增的 8‑GPU AllReduce‑Fusion 与 KV‑FP8 基准每个约 1‑1.5 h，合计可能将 nightly CI 从 5 h 拉至 9‑10 h，导致 PR 合并窗口被迫延长或超时。  
2. **镜像失效**：`scripts/ci/amd/amd_ci_start_container_disagg.sh` 中的硬编码 fallback 镜像名称与日期紧耦合，若官方仓库清理老镜像会导致容器拉取失败，CI 直接报错。  
3. **模型路径不一致**：CI 机器若未同步本地模型（`/data2/models/...`），则测试会 `skipTest`，但在 PR 测试中可能被误认为成功；若误删模型目录，所有相关 CI 任务将全部跳过，降低回归检测力度。  
4. **RDMA 环境假设**：Disaggregation 测试假设宿主机已装好 `libionic` 与 InfiniBand 设备，若运行在未配置 RDMA 的 runner，脚本仍会尝试启动容器，后续 Python 代码可能因缺失设备抛出异常，引发 CI 失败。  
5. **依赖版本冲突**：在容器内部额外安装 `tabulate`，但容器自带 `tabulate` 版本可能与 SGLang 代码不兼容，导致运行时 `ImportError`。  
6. **输入过滤字符串错误**：用户若在 `job_filter` 或 `target_stage` 中提供空格或多余逗号，会导致 `contains(format(',{0},', …), ',stage,')` 匹配失败，导致意外的 job 被跳过或全部跑完。

### [Kernel Slimming] Migrate NVFP4 kernels to JIT (#19437)
**SHA**: `2bdd89a` | 🔗 [查看提交](https://github.com/sgl-project/sglang/commit/2bdd89a6cd6e32586c62c8b14df348f8b373fb30)

**🎯 变更类型**：功能增强 / 重构 / 性能优化 / 架构变更  
**⚡ 重要程度**：🔴高  
**📋 变更摘要**：  
- 将原先在 **sgl‑kernel** 中通过 AOT（静态）编译的 NVFP4（FP4）量化、专家量化、块级 GEMM 与 MoE 分组 GEMM 实现，迁移为 **JIT（运行时即时）** 编译的 Python‑层 JIT Kernel。  
- 新增一套完整的 **benchmark**（`bench_nvfp4_*`）脚本，用于对 JIT、AOT（若可用）以及 Torch 基准实现的执行时延进行对比。  
- 在 `sglang/jit_kernel/nvfp4.py` 中实现统一的 Python 接口（`scaled_fp4_quant`、`scaled_fp4_experts_quant`、`cutlass_fp4_group_mm`、`cutlass_scaled_fp4_mm` 等），并在内部通过 TVM‑FFI 调用即时编译的 CUDA kernel。  
- 删除 SGL‑Kernel 中与 NVFP4 相关的 C++/CUDA 源文件，改为在 **sglang** 包中维护并编译（避免二进制膨胀）。  
- 相应地在 **SRT**（runtime）层的 MoE、量化、FlashInfer 兼容代码中切换调用路径，全部改为 `sglang.jit_kernel.nvfp4`。  
- 增添单元测试覆盖 JIT 实现的正确性（`test_nvfp4_*`），并引入 **workspace 缓存**、**SM 版本检测**、**CUDA 架构列表环境变量**等安全与兼容性控制。

**⚡ 重要程度**：🔴高  
**📋 变更摘要**：本次提交为 sglang 项目新增 HTTP 与 gRPC 服务器的 SSL/TLS 支持，并实现了证书动态热加载（`--enable-ssl-refresh`）。同时完善了 CLI 参数、输入校验、内部 URL 构造与 `requests` 的 SSL 验证逻辑，并新增对应单元测试与 `watchfiles` 依赖。

### 🔍 技术洞察

| 维度 | 影响 |
|------|------|
| **架构影响** | • 在 `ServerArgs` 中加入 SSL 配置字段，统一了 HTTP 与 gRPC 的 TLS 参数入口。<br>• `http_server` 与 `grpc_server` 的启动路径被分支为安全（`add_secure_port` / `uvicorn` SSL 参数）和非安全两套逻辑，保持向后兼容。<br>• 新增 `ssl_utils.SSLCertRefresher`，采用 `watchfiles.awatch` 进行高效的文件系统事件监听，实现 **运行时证书热更新**，对服务器架构的可观测性和运维友好性提升。 |
| **性能影响** | • 对单进程 HTTP（uvicorn）开启 TLS 时仅在 `uvicorn.Config` 中创建 `SSLContext`，不额外增加运行时开销。<br>• 启用 `--enable-ssl-refresh` 时会在事件循环中启动后台 `asyncio` 任务监听文件变化，资源占用极低（基于 OS 原生 inotify/kqueue），对吞吐量影响可忽略。<br>• 对 gRPC 使用 `grpc.dynamic_ssl_server_credentials`（若开启刷新），在每次新连接建立时可重新读取证书，略微增加握手成本，但对整体性能影响仍在毫秒级。 |
| **安全考虑** | • **传输层加密**：支持 HTTPS / gRPC TLS，防止中间人攻击。<br>• **证书验证**：`ssl_verify()` 为内部健康检查/调用提供 `requests.verify` 参数，默认在未提供 CA 时关闭验证并给出警告，防止误用导致安全风险。<br>• **密钥密码**：gRPC 模式明确禁止使用加密的 keyfile（`--ssl-keyfile-password`），避免在运行时解密密钥泄露。<br>• **文件路径校验**：在 `ServerArgs.__post_init__` 中提前检测证书、密钥、CA 文件是否存在，防止启动后因路径错误导致服务不可用。 |
| **可维护性** | • 所有 SSL 相关逻辑集中在 `ServerArgs` 与 `ssl_utils`，便于后续扩展（如 OCSP、CRL 检查）。<br>• 单元测试覆盖参数校验、URL 构造、证书刷新代码，提升回归安全性。<br>• 新增 `watchfiles` 依赖但仅在热刷新路径中使用，保持核心功能不强依赖此库。 |

### ⚠️ 潜在风险

1. **误配置导致验证失效**  
   - 若仅提供 `ssl_certfile` & `ssl_keyfile` 而未提供 `ssl_ca_certs`，内部 `requests` 将 `verify=False`，可能在生产环境误以为已安全。需要在文档或启动脚本中强提醒。  

2. **多 tokenizer worker 与证书刷新冲突**  
   - 在 `http_server` 多进程模式(`--tokenizer-worker-num > 1`)下，`--enable-ssl-refresh` 被强制禁用并打印警告。若用户未注意到该警告，可能误以为已启用刷新。  

3. **动态 SSL 证书加载异常**  
   - `grpc.dynamic_ssl_server_credentials` 需要符合 gRPC 对证书 PEM 格式的严格要求，若证书/密钥不匹配或损坏，启动时会抛异常。当前代码已捕获并转为 `ValueError`，但仍需在 CI 中加入证书格式校验的测试。  

4. **文件监控跨平台差异**  
   - `watchfiles` 在 Windows 使用轮询模式，可能导致文件修改检测延迟；在极端高并发环境下可能产生轻微资源波动。  

5. **旧版客户端兼容性**  
   - 客户端在没有提供 CA 的情况下会默认关闭验证，若旧版客户端仍使用 HTTP（非 TLS）进行 health 检查，会因 URL scheme 改变导致请求失败。

### 🔍 技术洞察

| 维度 | 影响描述 |

### ⚠️ 潜在风险

1. **指标系列爆炸**  
   - 若用户自行设置大量不同的 `priority`（例如时间戳或随机数），Prometheus 将为每个唯一值生成时间序列，导致磁盘和查询性能问题。  
2. **抢占逻辑回退**  
   - 当 `enable_priority_preemption` 与 `try_preemption` 混用时，旧代码仍保留 `try_preemption`，可能产生 **状态不一致**（如 `try_preemption=True` 但 `enable_priority_preemption=False`），需确保在 `Scheduler.__init__` 中彻底统一。  
3. **默认优先级未设置**  
   - 若开启 `enable_priority_scheduling` 而未配置 `default_priority_value`，`obj.priority` 会保持 `None`，导致 `QueueCount.from_reqs` 统计出现 `None` 键，Prometheus 会出现 `priority="None"`，影响监控查询。  
4. **性能回退**  
   - 对大队列（数千请求）进行 `QueueCount.from_reqs` 每次统计会遍历整个列表，若调度频率很高，CPU 使用率可能显著提升。  
5. **兼容性**  
   - 代码中仍保留 `self.try_preemption`（旧字段），外部插件或脚本若仍依赖该属性，可能产生误判。

### fix(grpc): send last chunk before completion during streaming (#19895)
**SHA**: `0ee9d3c` | 🔗 [查看提交](https://github.com/sgl-project/sglang/commit/0ee9d3c8e99dfbd9ba108cc15e48ab2e12f26393)

**📋 变更摘要**

**🎯 影响范围**  
- `sglang/srt/entrypoints/grpc_server.py`（流式 gRPC 接口）

2. **非流式路径**：`else` 分支保持原有 “单次 CompletionResponse” 行为；可加注释说明为何不再走 `ChunkResponse`。  
3. **错误处理**：若 `output.get("finished")` 与 `request.stream` 同时为真，当前会产生两个消息；确认客户端能够正确合并这两条（常见做法是忽略最后的 chunk），或在 `ChunkResponse` 中标记 `is_last=True`。  
4. **回归测试**：加入流式场景的单元测试，验证在 `finished=True` 前后都能收到对应的 Chunk 与 Completion，且顺序不变。  
5. **性能与日志**：此改动不会显著影响性能，但建议在调试级别记录 “send last chunk before completion” 以便排查异常。  

总体而言，此更改修复了流式返回缺失最后增量的问题，影响范围局限于 gRPC 服务层，确保兼容性和添加相应测试后即可安全合并。
