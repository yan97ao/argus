### feat: tiny improve fp8_gemm tune usage (#23912)
**SHA**: `69a7121` | 🔗 [查看提交](https://github.com/sgl-project/sglang/commit/69a71219cb480f176876c2ebf00bae19379fea7d)

**🎯 变更类型**：功能增强  

**⚡ 重要程度**：🔴高  

**📋 变更摘要**：  
1. 为 FP8 GEMM 调优脚本引入 `extra_kernel_args`，仅在需要时传递 `needs_masking` 参数，避免无意义的 kernel 参数传递。  
2. 将单一 `--batch-size` 参数改为 `--batch-sizes`（支持多批次并行调优），并在 GPU 分配前随机 shuffle `batch_sizes` 以实现更均匀的负载分布。  
3. 通过对调优结果的 JSON 文件进行键排序、去除未使用的临时变量以及微调循环变量，提升可读性与调试体验。

**🎯 影响范围**：  
- `benchmark/kernels/quantization/tuning_block_wise_kernel.py`（核心调优逻辑）  
- 依赖该脚本的量化配置生成流程（`python/sglang/srt/layers/quantization/configs`）  
- 命令行接口及其调用脚本/CI 流程  

**🔍 技术洞察**  

- **架构影响**：  
  - 引入 `extra_kernel_args` 将 kernel 参数的构造从硬编码转为条件化，使调优框架对不同数据类型的适配更具弹性。  
  - `--batch-sizes` 取代原来的单一 batch size，允许在同一次调优运行中并行遍历多个 batch 大小，提升调度层的并行度。  
  - 对 `existing_configs` 进行键排序后写入 JSON，保持配置文件的确定性顺序，便于版本对比与 CI 检查。

- **性能影响**：  
  - **负载均衡**：在多 GPU 环境下随机打乱 `batch_sizes` 再切分，可显著降低某一 GPU 因连续处理大批次而出现的“瓶颈”现象，整体调优耗时预计下降 5%‑15%（取决于批次数量与 GPU 数目）。  
  - **Kernel 调用**：仅在 FP8 场景且 `needs_masking=True` 时才向 kernel 传递 `needs_masking`，避免不必要的参数检查与内部分支，轻微提升单次 GEMM 调用的启动开销。  
  - **代码简化**：`for _ in range(num_iters)` 取代未使用的循环变量，微幅降低 Python 帧栈创建成本。

- **安全考虑**：  
  - 新增 `random` 模块用于 shuffling，无外部输入参与随机数种子生成，不引入可预测性或信息泄露风险。  
  - 参数名称变化可能导致旧脚本误用，若未做好参数校验，可能触发异常导致调优中断，但不会导致安全漏洞。

**⚠️ 潜在风险**  

1. **可复现性**：随机 shuffling 使同一批次大小列表在不同运行间顺序不同，调优时间和最终配置可能出现细微差异。  
2. **向后兼容**：CLI 参数从 `--batch-size` 改为 `--batch-sizes`，旧版调用脚本或 CI 配置若未同步更新会报错或默认使用单 GPU。  
3. **Kernel 参数兼容**：若在其他非 FP8 路径仍需要 `needs_masking`（例如未来扩展），当前实现会丢失该参数，可能导致 kernel 调用失败。  
4. **JSON 排序副作用**：对配置键进行排序后写回文件，若下游工具依赖原始插入顺序（极少见），可能产生意外行为。

**💡 关注建议**  

- **文档与 CI**：立即在项目文档、`README` 以及 CI 流程中更新参数说明，标明 `--batch-sizes` 支持多值并提供示例。  
- **可复现选项**：为调优脚本新增 `--seed` 参数，允许用户显式设定随机种子（`random.seed(args.seed)`），在需要对比实验时保证一致的 batch 分配顺序。  
- **兼容层**：在 argparse 中保持对旧参数 `--batch-size` 的兼容（如 `parser.add_argument('--batch-size', type=int, help=... )` 并在解析后统一转为列表），降低升级阻力。  
- **参数验证**：在构造 kernel 调用前，显式检查 `extra_kernel_args` 中的键是否被 kernel 接口接受，避免因参数缺失导致的运行时异常。  
- **回归测试**：补充针对多 GPU、不同 batch size 组合的调优基准测试，确保 shuffle 逻辑不会导致显著的性能回退。  
- **监控配置文件变更**：由于配置文件排序后写入，建议在 CI 中加入对 `git diff` 的检查，防止因排序导致不必要的提交噪声。

### feat(observability): add OpenTelemetry tracing for pipeline parallelism (#23169)
**SHA**: `71160e4` | 🔗 [查看提交](https://github.com/sgl-project/sglang/commit/71160e4ddb435f44ed90076a3629b26181f42e9a)

**🎯 变更类型**：功能增强  
**⚡ 重要程度**：🔴高  
**📋 变更摘要**：为支持 Pipeline Parallel（PP）调度的可观测性，新增 OpenTelemetry 线程级追踪维度（`pp_rank`），并在调度器执行路径中记录 CPU 侧批处理的开始/结束时间。相关接口、数据结构和测试均同步更新，以实现端到端的跨层级时序追踪。  

**🎯 影响范围**：  
- `python/sglang/srt/managers/scheduler.py`（线程信息设置）  
- `python/sglang/srt/managers/scheduler_pp_mixin.py`（在 PP 批处理前后记录时间）  
- `python/sglang/srt/observability/req_time_stats.py`（新增 `RUN_BATCH_CPU` 阶段 & 对应 setter）  
- `python/sglang/srt/observability/trace.py`（`TraceThreadInfo` 扩展、`trace_set_thread_info` 参数、线程名与属性生成）  
- `scripts/convert_otel_2_perfetto.py`（Perfetto 报表中展示 PP/DP 信息）  
- 单元测试 `test/registered/unit/observability/test_trace.py`（对应数据结构更新）  

**🔍 技术洞察**  

- **架构影响**：  
  - 仅在观察/监控层面做结构性扩展，未触及核心推理或调度逻辑。  
  - 新增 `pp_rank` 字段，使线程上下文能够完整反映 **TP / PP / DP** 三维并行拓扑，便于在分布式追踪系统中定位跨进程/跨机器的时序瓶颈。  
  - `ReqTimeStats` 中加入 `RUN_BATCH_CPU` 阶段，统一了 CPU 端前置处理（如 `set_run_batch_cpu_start_time/end_time`）的度量，提升了整体可观测链路的完整性。  

- **性能影响**：  
  - 记录时间戳的开销极低（`time.perf_counter()` + 一次属性写入），在高吞吐的调度路径中可视为 **微秒级** 额外成本。  
  - OpenTelemetry 的 `trace_set_thread_info` 与 `trace_slice` 只在全局开启 tracing (`opentelemetry_initialized && get_global_tracing_enabled()`) 时执行，默认关闭时几乎无运行时影响。  
  - 额外的线程属性（`pp_rank`）仅在创建 span 时写入一次，对 CPU/内存占用影响可忽略。  

- **安全考虑**：  
  - 变更仅涉及监控数据的收集与标签化，没有对外暴露敏感信息的路径。  
  - 若用户在未初始化 OpenTelemetry 环境下调用 `trace_set_thread_info`，函数会安全返回，不会抛异常。  

**⚠️ 潜在风险**  

1. **兼容性**：`trace_set_thread_info` 签名变更为四参数，若外部插件或自定义脚本仍使用旧签名（仅 3 参数），会出现 `TypeError`。需在项目内部或文档中明确迁移指南。  
2. **Tracing 开启异常**：在极高并发场景下，大量 span 创建可能导致 OpenTelemetry 后端（如 OTLP 导出器）出现背压或内存激增。建议在生产环境使用采样率或限流机制。  
3. **属性冲突**：`pp_rank`/`dp_rank`/`tp_rank` 均以整数形式写入属性，若已有自定义属性使用相同键名可能被覆盖。  
4. **测试覆盖不足**：当前仅更新了 `test_trace.py`，但未对 `scheduler_pp_mixin` 中的 `set_time_batch` 调用进行端到端集成测试，可能在某些异常路径（如批次为空）下未触发对应时间记录。  

**💡 关注建议**  

- **迁移检查**：在 CI 中加入对 `trace_set_thread_info` 调用签名的 lint 检查，防止遗漏老代码。  
- **监控配置**：提供默认的 `OTLP` 导出器配置示例，建议在高并发部署时开启 **采样**（`Sampler.parent_based(root=AlwaysOnSampler(), ...)`）或 **批量导出**（`BatchSpanProcessor`），降低链路开销。  
- **文档更新**：在 observability 文档中添加 “PP Rank 追踪” 小节，说明 `RUN_BATCH_CPU` 阶段的意义及在 Perfetto/Jaeger 中查看的方法。  
- **性能基准**：建议在多节点、不同并行度（TP/PP/DP 组合）下跑一次短时的基准，对比打开/关闭 tracing 的吞吐与延迟，确保额外开销在可接受范围。  
- **错误容忍**：在 `set_time_batch` 中已做好 `trace_only` 与 `attrs` 的安全检查，进一步可以在 `ReqTimeStats` 的各 `set_*` 方法里捕获异常并记录日志，防止监控异常导致调度主流程异常退出。  

---  

通过上述改动，sglang 现在能够在 OpenTelemetry 链路中完整展示 **Tensor Parallel、Pipeline Parallel 与 Data Parallel** 三层并行的线程上下文及 CPU 端批处理时间，为定位跨层级性能瓶颈提供了强有力的工具。只要注意兼容性与导出端的资源管理，这一特性将显著提升大型部署的可观测性与运维效率。

### [diffusion] chore: change default seed to 42 (#23836)
**SHA**: `144038f` | 🔗 [查看提交](https://github.com/sgl-project/sglang/commit/144038fbae146e48d3a3c873898a76418348b8f8)

**核心变更概览**  
1. **种子默认值与传播**：删除 `DEFAULT_SEED = 1024`，改为在请求对象中保存 `seed`（支持单值或列表），在 `extract_transfer_fields` 与 `_build_disagg_req` 中进行序列化/反序列化，确保 disagg 场景下能够重建 `torch.Generator`。  
2. **OpenAI/Vertex 接口**：把所有 `seed` 参数的默认值改为 `None`，并在服务入口处去掉硬编码默认种子，使调用方必须显式提供或使用后端默认（当前为 42）。  
3. **CI 分区逻辑**：新增 `compute_diffusion_partitions.py` 参数 `--parametrized-only`，并把分区算法抽离为 `partition_items_by_lpt`（Longest‑Processing‑Time greedy）。工作流读取该脚本输出的矩阵并把 `partition-plan-json` 传给生成脚本。  
4. **测试与工具**：新增 `partitioning.py`、相应的单元测试（seed 传播、列表种子）、以及对 `run_suite.py`、`gen_diffusion_ci_outputs.py` 的分区实现改造，使其使用统一的 LPT 分区并支持 JSON 计划。  

**影响范围**  
- **运行时**：所有 multimodal‑gen 入口（Vertex、OpenAI Image/Video）现在不再自动使用 1024，而是使用调用方提供的种子或内部默认（42）。  
- **分布式 disaggregation**：生成器的种子信息通过 RPC 传递，支持批量种子列表。  
- **CI/CD**：CI 分区变为参数化‑only（只针对 `DiffusionTestCase`），矩阵结构更稳健，可能影响已有手动分区的工作流。  
- **代码库**：`python/sglang/multimodal_gen/runtime/*`、`.github/workflows/*`、`scripts/ci/utils/diffusion/*`、以及测试套件都有改动。

**可操作建议**  
1. **向后兼容**：若要兼容旧用户，考虑在入口层继续保留 `seed=1024` 的默认值或在文档中明确说明种子已改为 “未指定 → 使用内部默认 42”。  
2. **验证列表种子**：`SchedulerDisaggMixin._build_disagg_req` 里 `torch.Generator(...).manual_seed(int(item))` 返回 `Generator`，但列表推导式的结果被直接赋给 `req.generator`（列表），后续使用时需确保所有调用方都接受列表（目前仅测试覆盖）。建议在 `Req` 类加入明确的类型注解或封装方法。  
3. **CI 脚本路径**：`compute_diffusion_partitions.py` 通过 `importlib` 动态加载 `partitioning.py`，在非 repo‑root 执行时可能找不到路径。可以改为相对导入或在 `PYTHONPATH` 中添加项目根目录，以防 CI 环境变化导致 `FileNotFoundError`。  
4. **文档同步**：更新 README / API 文档，说明 `seed` 参数已不再默认 1024、支持列表形式，以及 `--parametrized-only` 选项的含义。  
5. **回归测试**：在 CI 中加入对 `partition-plan-json` 解析的完整链路测试，确保在分区数变化或方案文件缺失时能给出友好的错误信息。  

总体而言，改动提升了随机性控制的可追踪性与 CI 分区的负载均衡，但涉及 API 默认值的变更需做好兼容提示，防止外部调用意外改变生成结果。

4. **向后兼容**：对未使用 CP（默认 `attn_cp_group=None`）的旧部署，行为保持不变；但 `CacheInitParams` 新增字段会在序列化/日志中出现，需检查任何手动构造该对象的代码是否已更新。  
5. **测试覆盖**：CI 已加入 CP 2 × TP 2 的 Mooncake 测例，建议再补充单元测试覆盖 `HiCacheController._all_reduce_attn_groups` 与 `_barrier_attn_groups` 的空/单节点路径，防止在单机单卡上出现 “group size 1” 的不必要通信。  

上述改动为 HiCache 在上下文并行下的状态一致性提供了可靠机制，但也引入了额外的进程组管理与 Gloo 通信开销，请在生产环境中根据实际 CP/TP 拓扑评估其成本。

| **多节点提示** | `prependMultiNodeNote` 生成的注释行缺少换行符，最终命令会被 `#` 注释整行，导致复制粘贴时需要手动去除。 | 在返回的字符串前后加 `\n`，或在文档中单独展示说明。 |
| **文档构建体积** | MDX 文件约 600 行，含大量表格与示例，渲染时可能触发 Docusaurus 的 Markdown‑to‑React 转换性能问题。 | 保持图片/表格使用外部资源，或拆分为子章节，避免单文件过大。 |

**总体评估**  
本次提交的核心是文档扩充，对核心库功能没有直接影响。只要修复上述 JSX 编译错误、保证字符串转义正确，文档站点即可顺利构建并为 MiMo‑V2.5 系列用户提供完整的部署指南。后续可在 CI 中加入对 `docs_new/**/*.jsx` 的 lint 检查，避免类似语法问题再次出现。
