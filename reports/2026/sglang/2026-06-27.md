### model: support baidu unlimited-ocr (#29186)

**🎯 变更类型**：功能增强（新增 Baidu Unlimited‑OCR 模型支持及完整的预填充感知滑动窗口（Prefill‑Aware SWA）执行路径）

- 在 SGLang 中加入 **Unlimited‑OCR**（Baidu）模型的完整实现：模型类、配置、HF 处理器、图像预处理、对话模板以及文档/示例。

### 2. 性能影响  
- **KV 复用提升**：PA‑SWA 通过 `page_size=1` 的页表直接映射 prefill token，避免在每一次 decode 复制整段 prefill KV（尤其是高分辨率图像的 KV 大小可达数千 tokens）。对长文档 OCR（> 10k tokens）可节省 **30‑50%** 的显存与时间开销（实际取决于窗口大小）。  
- **Triton Kernel 开销**：构建 page‑table 的 kernel 采用单指令流、每块 256 threads。对 Batch Size ≤ 32 的情况延迟几百微秒，可忽略不计；但在极端高并发（> 128）时可能成为瓶颈，需要监控 `triton` GPU SM 使用率。  
- **内存占用**：Pure‑SWA 分配器把 **全层** 视为 SWA，`full_max_total_num_tokens` 设为 0，导致 `max_token_pool_size` 直接取 `swa_max_total_num_tokens`。这减少了 *full* KV pool 的预留，但要求 **page_size 必须为 1**，否则会触发断言。  
- **调度开销**：`swa_evict_floor` 的计算在 `Scheduler._get_new_batch_prefill_raw` 中加入 O(1) 操作，几乎不影响整体调度时延。

### ⚠️ 潜在风险

1. **长度不匹配导致运行时异常**  
   *`kv_data_ptrs`、`kv_data_lens`、`kv_data_mem_kinds`、`dst_kv_mem_kinds`、`dst_kv_item_lens` 必须保持一一对应；若用户自行构造不完整的 JSON 配置或在代码路径中出现遗漏，将触发 `ValueError` 并导致服务启动失败。  

2. **混合段传输的死锁/超时**  
   *在 `TransferWorker` 中对每个 segment 调用 `send_kvcache_mixed`，若某段的预备句柄创建失败或 NIXL 返回错误状态，当前实现仅抛异常并停止整个进程，缺少 **回滚/重试** 机制。  

3. **性能回退**  
   *在显存充裕的场景下误把 KV 目的地设为 DRAM，可能导致不必要的 PCIe 拷贝，显著增加解码时延。需要在默认配置上保持 **VRAM** 为首选，只有在 HiSparse 明确开启时才切换为 DRAM。  

4. **兼容性**  
   *旧版 NIXL 可能不支持 `register_memory(..., mem_kind)` 参数，或者不识别 `DRAM` 标记。该代码假设 NIXL 1.x 已实现该特性；在未满足的环境下会抛 `Exception("NIXL memory registration failed for DRAM kv tensors")`。  

5. **测试覆盖不足**  
   *虽然新增了 `test_disaggregation_hisparse.py` 与单元测试，但仍缺少 **压力/性能基准**（大 batch、长序列）以及 **多机多卡** 场景下的 **混合 KV** 端到端验证。

### ⚠️ 潜在风险

1. **对齐/向量化假设失效**  
   - 若输入张量在某些路径（如 `view/as_strided`）导致 16‑byte 对齐失效，kernel 会退回到 scalar路径，仍能工作但性能下降。需要在上层确保 `contiguous()`，或在模型前置 `torch.nn.functional` 调用前显式 `contiguous()`（已在代码中检查）。  

2. **广播形状限制**  
   - 只支持 `[1, ..., 1, D]` 行广播；更通用的广播（如 `[B, 1, D]`）会被拒绝并回退。若后续模型改为此类形状，需要扩展 `GateMode` 与对应 kernel。  

3. **CUDA 设备兼容性**  
   - 依赖 `kMaxGrid = 65535` 的硬件限制；在超大 batch（> 65535 × 256 元素）时会分块执行，但仍受 `grid_for` 限制。极端情况下可能触发不完整计算，需要在上层做分块或检查 total ≤ `kMaxGrid * kBlockSize * kVec`。  

4. **异常捕获导致一次性禁用**

### Convert IPC dataclasses to msgspec.Struct with opt-in msgpack transport (#28688)

**🔍 技术洞察**  

- **架构影响**

2. `sgl_router_responses_total{route,method,status_code}` – 出口计数，记录返回的 HTTP 状态（包括 400/413/503 等早期退出）。  
3. `sgl_router_worker_requests_total{worker_url,model_id,mode,outcome}` – 仍保留的每个 worker 的调度统计，名称从 `sgl_router_requests_total` 改为 `sgl_router_worker_requests_total`。

为实现上述功能，新增了 **middleware `count_requests`**（在 `app.rs`），负责在请求进入路由时记录 `record_ingress`，在响应返回前记录 `record_response`。`metrics.rs` 也相应重构：新增 `EdgeKey`、`EdgeResponseKey`，分别管理入口和出口计数；原来的 `record_request` 改为 `record_worker_request`；`record_response` 接收 route、method、status_code 参数。

文档、Grafana 仪表盘、以及所有相关测试均同步更新，以使用新的 metric 名称和标签。

---

### 影响范围
- **监控链路**：Prometheus 抓取的 metric 名称与 label 结构全部变更，原有查询、报警规则、Grafana 看板需同步迁移。  
- **日志/链路追踪**：`MatchedPath` 现在被用于标签，确保路由模板固定，避免标签爆炸。  
- **业务代码**：业务侧若直接引用旧的 `sgl_router_requests_total`（如自定义监控脚本）将失效。  
- **单元/集成测试**：所有 tests 已更新，但外部项目的兼容性仍需验证。

---

### 关键关注点 & 建议
1. **兼容性**  
   - 该改动属于 **破坏性**（metric 重命名）。建议在正式发布前保留旧的 `sgl_router_requests_total`（可标记为 deprecated）一段时间，或提供迁移指南。  
   - 确认 **Prometheus 警报**、**Grafana 看板** 已全部切换至 `sgl_router_worker_requests_total` 与新的 edge 指标，否则会出现 “无数据” 警报。

2. **标签基数控制**  
   - `route` 使用 `MatchedPath`（模板）而非原始 URI，避免因高基数导致的内存 / 读取压力。仍需审视是否有非模板路径（如动态路径）泄露到 label，必要时在 `count_requests` 中加入白名单或截断。

3. **中间件顺序**  
   - `count_requests` 被放在 `.layer(middleware::from_fn_with_state(...))` **之后** `Router::new()` 的所有路由注册，确保 `MatchedPath` 已被 Axum 填充。若未来在 router 前插入其他全局中间件，需检查它们是否会提前消费 `Request`，导致 `MatchedPath` 缺失。

4. **性能影响**  
   - 计数使用 `Mutex<HashMap<...>>` + `Arc<AtomicU64>`，和旧实现相同。额外的 `responses_total` 计数和 `EdgeKey`/`EdgeResponseKey` 的构造成本极低，几乎不影响吞吐。可在高并发环境下通过基准测试验证（尤其是 `record_ingress` 与 `record_response` 于同一请求的并发）。

5. **监控数据完整性**  
   - 新增的 `requests_total - responses_total` 可直接反映 “接收但未响应” 的请求数，这是之前无法观测的关键指标。建议在监控大盘加入该差值的可视化，并设置阈值告警。

6. **测试覆盖**  
   - 已加入 `edge_counters_recorded_through_middleware` 测试，验证中间件实际生效。继续保持对 **流式** 与 **非流式**、**错误码 4xx/5xx** 场景的覆盖，防止遗漏早期退出的计数。

---

**结论**：此改动显著提升了路由层面的可观测性，能够区分入口流量、worker 调度和最终响应的完整路径。唯一需要重点关注的是外部监控系统的迁移以及确保标签基数受控。若能在正式发布前提供短期兼容层或迁移文档，将极大降低用户升级风险。
