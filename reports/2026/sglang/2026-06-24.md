### [HiCache] Add NIXL FILE cache cleaner (#28258)
**SHA**: `03773ae` | 🔗 [查看提交](https://github.com/sgl-project/sglang/commit/03773ae35b6271f11a2e512d96e57d7de91b1fce)

**🎯 变更类型**：功能增强  
**⚡ 重要程度**：🔴高  
**📋 变更摘要**：在 HiCache 的 NIXL FILE 后端加入了后台磁盘清理器 `HiCacheL3Cleaner`，实现基于磁盘使用率的自动淘汰；同时在配置、文档、代码中加入相应的开关及水位阈值，并对路由常量进行命名修正。该功能可防止 L3 文件缓存因磁盘占满而导致服务失效或 I/O 报错。  

**🎯 影响范围**：  
- `python/sglang/srt/mem_cache/storage/nixl/`（新增 `nixl_cleaner.py`，修改 `hicache_nixl.py`, `nixl_utils.py`, `nixl_routing.py`, `README.md`、示例配置）  
- 相关单元测试 `test/registered/unit/mem_cache/`  
- 配置文件 `nixl.config.toml.sample`  

**🔍 技术洞察**  

- **架构影响**  
  - 在 HiCache 初始化阶段为每个 TP rank 0 实例创建并启动 `HiCacheL3Cleaner` 线程，实现 L3（FILE‑backend）层的 **主动磁盘回收**，保持原有“被动”清理（缓存失效）机制不变。  
  - 通过 `NixlBackendConfig.get_l3_cleaner_config()` 将清理器的开关与水位阈值从插件层抽离到 SGLang 顶层配置，避免插件对全局资源管理产生耦合。  
  - 引入 `BUCKET_HEX_CHARS` 常量统一命名，提升路由模块的可读性与可维护性。  

- **性能影响**  
  - **正面**：在磁盘使用率超过 `high_watermark`（默认 80%）时，清理器会并发删除最旧的 **逻辑键组**（按文件 mtime 排序），降低磁盘占满导致的 I/O 错误或写入阻塞。  
  - **并发**：使用 `ThreadPoolExecutor`（默认 `8 * storage_dir_num`）并行 `unlink`，对 I/O 产生的阻塞影响极小；删除操作是 **best‑effort**，不会阻塞主业务线程。  
  - **开销**：清理线程每 `interval_sec`（默认 30 s）轮询一次磁盘使用率，扫描所有 bucket 目录（仅目录层级），对磁盘的读取开销在大多数生产环境下可接受。可通过 `interval_sec` 与 `recheck_groups` 调整频率与批次大小。  

- **安全考虑**  
  - 删除文件前仅依据文件名的 **逻辑键组** 进行分组，不会误删非 HiCache 产生的文件。  
  - `_safe_unlink` 捕获 `FileNotFoundError` 与通用 `OSError`，防止因并发删除或权限问题导致清理线程异常退出。  
  - 配置 `use_direct_io` 与清理器互不影响，保持对磁盘 I/O 行为的一致性。  
  - 高水位阈值必须大于低水位阈值，构造函数会抛出 `ValueError`，避免配置错误导致无限循环或不触发清理。  

**⚠️ 潜在风险**  

1. **误删风险**（极小）  
   - 如果外部进程在同一目录下创建与 HiCache 相同命名规则的文件，清理器会把这些文件视为缓存并删除。解决方案：在生产环境中确保 FILE‑backend 目录仅被 HiCache 使用，或禁用内置清理器改用外部专用清理。  

2. **并发竞争**  
   - 多个 TP rank 可能共享同一磁盘目录；目前仅在 `tp_rank == 0` 启动清理线程，避免重复删除，但若用户自行在其他 rank 上启动，会产生竞争。文档已经强调“TP rank 0”。  

3. **资源占用**  
   - 扫描大量 bucket 目录（尤其在高并发模型产生上万文件时）可能导致 CPU/IO 临时波峰。`recheck_groups` 与 `unlink_workers` 可调节该波峰。  

4. **配置不一致**  
   - 旧版配置文件缺少新键时，默认启用清理器并使用默认水位；如果原有部署对磁盘使用率极为敏感，可能出现意外清理。建议在升级后检查 `nixl.config.toml.sample` 并根据需求关闭 `l3_cleaner_enabled`。  

**💡 关注建议**  

- **上线前**：在测试环境验证 `HiCacheL3Cleaner` 对磁盘使用率的响应曲线，尤其关注 `high_watermark` 与 `low_watermark` 的差值（推荐 ≥ 5%）。  
- **监控**：在 Prometheus/Grafana 等监控平台添加指标：`sglang_nixl_l3_cleaner_deleted_groups`, `sglang_nixl_l3_cleaner_deleted_bytes`, `sglang_nixl_l3_cleaner_disk_usage`，便于观察清理频率与磁盘回收效果。  
- **日志**：默认使用 `logger.info` 输出清理概要，建议在生产环境开启 `DEBUG` 级别，捕获 `NIXL L3 cleaner tick failed` 以定位异常。  
- **兼容性**：若已有外部清理脚本，请在 `nixl.config.toml` 中将 `l3_cleaner_enabled = false`，以防止双重删除。  
- **参数调优**：在磁盘 I/O 较慢的环境（如 HDD）可以增大 `interval_sec` 或减小 `unlink_workers`，降低峰值 I/O；在 NVMe 高速磁盘则保持默认或适当降低 `interval_sec` 以加快回收速度。  

---  
此次改动为 HiCache 增加了 **自助磁盘回收** 能力，提升了生产环境的鲁棒性和可运维性，只要按照上述建议配置与监控，即可安全平滑地将功能投入使用。

### [Diffusion] Add Krea 2 support (#29052)
**SHA**: `5beddc8` | 🔗 [查看提交](https://github.com/sgl-project/sglang/commit/5beddc8afce90ddf1662e7a8cff6f89214add9a9)

**🎯 变更类型**：功能增强 / 架构变更  
**⚡ 重要程度**：🔴高  
**📋 变更摘要**：  
- 为 SGLang 添加对 Krea‑2（K2）单流 MMDiT 文本‑图像扩散模型的完整支持，涵盖模型配置、模型实现、管线、采样默认值以及注册表。  
- 改进 `webui/main.py` 中模型任务名获取逻辑，兼容本地 checkpoint 与 ModelScope/HuggingFace Hub。  
- 新增专用的 `Krea2Transformer2DModel`、`Krea2Pipeline`、前置处理阶段 `Krea2BeforeDenoisingStage`，以及对应的配置、采样参数与注册入口。  

**🎯 影响范围**：  
- `python/sglang/multimodal_gen/configs/models/dits/krea2.py`  
- `python/sglang/multimodal_gen/configs/pipeline_configs/krea2.py`  
- `python/sglang/multimodal_gen/configs/sample/krea2.py`  
- `python/sglang/multimodal_gen/registry.py`  
- `python/sglang/multimodal_gen/runtime/models/dits/krea2.py`  
- `python/sglang/multimodal_gen/runtime/pipelines/krea2.py`  
- `python/sglang/multimodal_gen/runtime/pipelines_core/stages/model_specific_stages/krea2.py`  
- `python/sglang/multimodal_gen/multimodal_gen/apps/webui/main.py`（任务名解析）  

**🔍 技术洞察**  

- **架构影响**  
  - **全新模型子系统**：实现了 Krea‑2 单流 MMDiT，采用 28 层 Transformer、GQA 注意力、3‑轴 RoPE、文本‑图像联合流以及 12‑层文本‑encoder 融合层。模型结构与原始 checkpoint 完全对齐（`param_names_mapping` 为 identity），实现了零映射加载。  
  - **模块化管线**：在 `registry` 中注册 `Krea2PipelineConfig` 与 `Krea2SamplingParams`，并通过 `LoRAPipeline`/`ComposedPipelineBase` 组装前置处理 → 去噪 → 解码三个阶段，保持与已有 Diffusers‑style 管线一致的接口。  
  - **任务检测容错**：`webui/main.py` 先尝试从模型仓库查询任务标签，若失败回退至已加载 pipeline 的 `task_type`，提升对本地 checkpoint 的兼容性。  
  - **内存管理**：在模型实现中加入 `LayerwiseOffloadableModuleMixin` 与 `--dit-layerwise-offload` 支持，支持 28 层大模型的逐层加载，显著降低峰值显存需求。  

- **性能影响**  
  - **融合算子**：实现了可选的 **fused QKNorm+RoPE** 核心路径（`_fused_qknorm_rope_enabled`），在支持的 GPU、FP16/BF16 环境下可省去一次 RMSNorm 与 RoPE 的显式计算，提升单步推理吞吐。  
  - **BF16 文本编码**：pipeline 默认使用 `bf16` 的 Qwen‑VL 文本编码器，避免了从 FP32 到 BF16 的精度损失，提升显存利用率并略微加速。  
  - **层级 Offload**：通过 `LayerwiseOffloadableModuleMixin`，在显存受限的机器上仍可运行 K2（约 24 GB 参数），但会因层间切换产生 CPU↔GPU 拷贝开销。  
  - **采样默认**：`Krea2SamplingParams` 设定 8 步、CFG 关闭的 “oss_turbo” 快速模式；`Krea2RawSamplingParams` 提供完整 52 步、CFG 开启的 “oss_raw”。用户可自行权衡速度与质量。  

- **安全考虑**  
  - **外部依赖**：新增对 `ModelScope` 与 `huggingface_hub` 的运行时查询，仅用于获取模型 `task_name`，不存在写入或代码执行风险。  
  - **环境变量**：`SGLANG_ENABLE_FUSED_QKNORM_ROPE` 受用户可控，默认开启；若误设为非布尔值可能导致异常回退但不影响安全。  
  - **模型加载**：`param_names_mapping` 与 `lora_param_names_mapping` 为静态字典，未引入动态路径解析，避免路径注入。  

**⚠️ 潜在风险**  

1. **显存峰值**：即使支持层级 offload，首次加载所有层（约 24 GB）仍会在 GPU 上暂存一次，低显存机器（≤ 12 GB）可能 OOM。

4. **LoRA 前缀剥离**：`lora_param_names_mapping` 只剥离 `transformer.` 前缀，若未来 checkpoint 使用不同前缀会导致 LoRA 权重加载失败。  
5. **文本‑encoder CPU‑offload**：`Krea2BeforeDenoisingStage` 声明了文本编码器组件使用，以便在去噪阶段 offload。若用户未开启 `--text-encoder-cpu-offload`，仍会在 GPU 上驻留，导致显存压力增加。

## 🔍 技术洞察

| 维度 | 影响说明 |
|------|----------|
| **架构影响** | - 在每个 **Tensor‑Parallel (TP)** 组内部再划分 **Decode‑Context‑Parallel (DCP)** 组，实现 KV 缓存的 **横向切分**（每张卡只保存本 rank 对应的 KV 片段）。<br>- 新增 `GroupCoordinator` `_DCP`，与现有 TP、PP、MOE、ATTN‑CP/TP 组保持相同的生命周期管理。<br>- 通过 `all_gather`/`all_reduce` 在 decode 时跨 DCP 组聚合注意力输出（log‑sum‑exp 合并 LSE），保持数值等价。<br>- 只有在 **HIP** 平台且 `decode_context_parallel_size > 1** 时才会创建 DCP 组，保持对非‑HIP 环境的向后兼容。 |
| **性能影响** | - **显存占用**：KV 缓存被等分到 DCP‑rank，单卡显存需求降低约 `1 / dcp_size`，对超长上下文（> 1M）尤为关键。<br>- **吞吐**：在 decode 场景，前缀 KV 的注意力在各 DCP rank 并行完成，再通过 LSE 合并，只需一次 **all_gather**（收集 Q）和一次 **all_reduce**（合并输出），通信量为 `O(batch * heads * dmodel)`，相对全局 TP 方案显著下降。<br>- **Triton kernel**：`extend_attention_fwd` 通过 `skip_prefix`/`skip_extend` 分两次调用，避免在同一 kernel 中处理全量 KV，降低缓存冲突并提升 CUDA‑graph 启动速度。<br>- **JIT 开销**：对 `free_page_ptr` 加入 `do_not_specialize`、对 HIP‑unique 进行预热，显著削减二次 JIT (≈200 ms) 的延迟。<br>- **实验结果**（CI 用例）：TP=8、DCP=2 在 AMD MI35x 上跑 GSM8K，准确率 ≥ 0.90，表明功能完整且性能未出现回退。 |
| **安全考虑** | - 仅在 **AMD HIP** 环境下开启，且在 `ServerArgs.__post_init__` 中强制校验 `dcp_size >= 1` 与平台匹配，防止误用导致未初始化的 DCP 组被访问（会触发 Assertion）。<br>- 所有新 API（`get_dcp_group`、写掩码 `dcp_kv_mask`）均通过 `assert` / `RuntimeError` 进行边界检查，未发现对外暴露的安全漏洞。 |
| **可维护性** | - 代码路径分叉（`if self.dcp_size > 1`）在多个文件中出现，已通过集中在 `utils.py`、`parallel_state.py`、`model_runner.py` 中的统一属性 (`dcp_size`, `dcp_rank`) 进行统一控制，降低散布的硬编码风险。<br>- Triton kernel改动保留了原有参数顺序，仅在 `extend_attention_fwd` 中新增可选 `lse_extend`, `skip_prefix`, `skip_extend`，不影响已有调用。<br>- 文档/注释已明确标注 “HIP‑only”，便于后续迁移或扩展到其他平台。 |

| 风险点 | 说明 | 可能的后果 |
|--------|------|------------|
| **平台限定** | DCP 只在 HIP 上实现，若用户在 CUDA 环境误设 `dcp_size>1`，会在初始化时报错。 | 运行时直接失败，影响用户体验。 |

### [Perf] Overlap result D2H copy with the next forward step (#29075)
**SHA**: `8e1988b` | 🔗 [查看提交](https://github.com/sgl-project/sglang/commit/8e1988b746f7ee3072ea2b0199efb01d8273c2eb)

**🎯 变更类型**：性能优化  
**⚡ 重要程度**：🟡 中  

**📋 变更摘要**  
- 新增 `_async_d2h` 实现基于 pinned‑memory 的异步 D2H（Device→Host）拷贝，并在拷贝完成后通过 `record_stream` 保证源张量生命周期。  
- `GenerationBatchResult.copy_to_cpu` 与 `EmbeddingBatchResult.copy_to_cpu` 全部改用 `_async_d2h`，并在 `Scheduler.run_batch` 中把拷贝搬到专用的 `copy_stream`，通过 `wait_stream` 与前向 `forward_stream` 解耦，实现拷贝与下一轮前向的并行。  
- 引入 `map_device_tensors(fn)` 抽象，统一子对象（`RoutedExpertsOutput`、`IndexerTopkOutput`、`ExpertDistributionMetrics`、`TopkCaptureOutput`）的设备张量拷贝，实现“一处复制、一次安全”。  

**🎯 影响范围**  
- **core/srt/managers/scheduler.py** – 调度器流水线的流同步逻辑。  
- **core/srt/managers/utils.py** – 复制实现与 `GenerationBatchResult`、`EmbeddingBatchResult`。  
- **core/srt/eplb/expert_distribution.py** – `ExpertDistributionMetrics` 改为 `map_device_tensors`。  
- **core/srt/state_capturer/base.py** – `TopkCaptureOutput` 同步改为 `map_device_tensors`。  

**💡 关注建议**  

1. **复制流的完整性**  
   - 确认 `Scheduler.__init__` 已创建 `self.copy_stream` 以及对应的 `self.copy_stream_ctx`（`torch.cuda.stream`），避免在未初始化的情况下调用 `wait_stream`。  
   - 在多 GPU 环境下，`copy_stream` 必须与每块卡对应的 `forward_stream` 匹配，否则会出现跨卡同步错误。

2. **Pinned‑memory 资源**  
   - `_async_d2h` 为每个张量分配 `pin_memory=True` 的临时缓冲，长期运行可能导致 pinned‑memory 耗尽。建议在启动脚本中适当调大 `torch.cuda.set_per_process_memory_fraction` 或监控 `torch.cuda.memory_stats()['pinned_memory_allocated']`。  
   - 若 CPU 端内存受限，可提供环境变量或配置开关让拷贝回退到普通 `to('cpu')`（已在函数里实现）。

3. **兼容性回退**  
   - 非 CUDA 环境仍走同步 `to('cpu')`，但 `self.copy_stream` 在 CPU‑only 条件下不会被使用，确保 `Scheduler.run_batch` 在 `torch.cuda.is_available()` 为 False 时不触发 `self.copy_stream.wait_stream`。  
   - 单元测试应覆盖 CPU、单 GPU、双 GPU 场景，验证 `copy_done` 正常记录且后续 `future_map` 能正确读取。

4. **子对象实现**  
   - 所有加入 `map_device_tensors` 的类（如 `RoutedExpertsOutput`、`IndexerTopkOutput`）必须仅包含设备张量字段，否则会遗漏未拷贝的内容。建议在这些类上添加 `__repr__`/`__post_init__` 检查，或在 `copy_to_cpu` 前添加断言 `assert all(t.is_cuda for t in ...)`。

5. **性能验证**  
   - 使用 `torch.profiler`（已在 `copy_to_cpu` 加入 `record_function`）对比前后吞吐率，关注 `copy_result_to_cpu` 与 `copy_embedding_to_cpu` 的时间占比是否下降。  
   - 注意在高并发请求下 `copy_stream` 可能成为瓶颈，必要时可考虑为每个 batch 创建独立流或使用流池。

6. **文档/发布说明**  
   - 在 README / release note 中标注 “异步 D2H + copy_stream 仍需显存和 pinned‑memory 支持”，并提供 `export SGLANG_DISABLE_ASYNC_D2H=1`（示例）等回退开关，以防极端环境下出现 OOM 或冻结。

**总结**：本次改动通过专用拷贝流与 pinned‑memory 异步拷贝把结果搬迁与下一轮计算解耦，理论上可降低前向‑拷贝的串行阻塞，提升吞吐。开发者应重点检查流初始化、Pinned‑memory 使用上限以及子对象的统一拷贝实现，确保在不同硬件配置下均保持功能正确且不会引入资源泄漏。
