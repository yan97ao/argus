### [diffusion] refactor: LTX2.3 code cleanup (#23207)
**SHA**: `0be6ab0` | 🔗 [查看提交](https://github.com/sgl-project/sglang/commit/0be6ab04dd4209e81f4afac10eb5832dd3b5e62b)

**🎯 变更类型**：重构 / 功能增强  
**⚡ 重要程度**：🔴高  

**📋 变更摘要**  
1. 对 LTX‑2.3 两阶段流水线进行大幅代码清理与结构化重构，新增 `is_ltx2_two_stage_pipeline_name` 判定工具，并将 LoRA 切换逻辑拆分为可复用的 *spec* 与 *builder* 方法。  
2. 将 LTX‑2 denoising 阶段抽象为若干 `@dataclass`（`LTX2ModelInputs`、`LTX2GuidancePassSpec`），统一构建模型输入、attention mask 与 batch‑repeat／split 逻辑，显著降低代码重复度。  
3. 新增 `release_ltx2_phase_state`、`_can_short_circuit_lora_switch` 等细粒度资源管理函数，改善显存释放与阶段切换的性能与可维护性。  
4. 在 `ServerArgs` 中补充对两阶段 pipeline 名称的统一判断以及对应的内部标识函数，确保后续分支判断一致。  
5. 同步更新性能基线 JSON，反映代码重构后得到的微调时间统计。

### [Refactor] Deduplicate NSA utils.py into cp_utils.py for context parallel (#22914)

**⚡ 重要程度**：🔴高  
**📋 变更摘要**：将原先散落在 `sglang/srt/layers/attention/nsa/utils.py` 中的上下文并行（Context‑Parallel，简称 CP）实现抽取并统一到 `sglang/srt/layers/utils/cp_utils.py`，并以 `attn_cp_*` 为统一前缀取代原有 `nsa_*` 命名。涉及的元数据结构从 `NSAContextParallelMetadata` 重命名为 `ContextParallelMetadata`，并在多个模型、attention 层以及调度器中统一使用。此举消除了代码冗余、提升可维护性，同时为后续在非‑NSA 场景（如 Qwen‑3‑MoE）复用同一套 CP 逻辑奠定基础。

### ⚠️ 潜在风险
1. **未覆盖的旧引用**：搜索全项目的 `nsa_cp_metadata`、`nsa_` 前缀函数可能遗漏（尤其在用户自定义脚本或外部插件中），导致运行时 `AttributeError`。  
2. **并发/异步流**：`cp_all_gather_rerange_output` 仍然依赖 `use_symmetric_memory` 与异步 `cp_all_gather_into_tensor_async`，若同步/异步流的调度在新文件中未正确继承上下文（如 `stream` 参数传递），可能出现数据竞争或未同步的情况。  
3. **元数据字段变更**：`ContextParallelMetadata` 结构保持与旧 `NSAContextParallelMetadata` 相同，但若未来添加新字段而未在所有创建路径（`prepare_context_parallel_metadata`、`prepare_input_dp_with_cp_dsa`）同步，可能出现 `AttributeError`。  
4. **测试覆盖不足**：原 `utils.py` 中的大块实现被删除，若对应的单元测试仍指向旧路径，CI 将失效。  
5. **Round‑Robin 分支**：在 `cp_split_and_rebuild_*` 中新增对 `is_nsa_prefill_cp_round_robin_split` 的检查，若该分支逻辑在某些硬件/配置下未经过充分验证，可能导致切分错误或性能回退。

### 💡 关注建议
- **统一迁移检查**：在 CI 中加入一次性脚本，搜索 `nsa_` 前缀的未更新引用，确保所有外部插件也已迁移。

- **回归测试**：重点跑包含上下文并行预填 (`prefill`) 场景的性能基准，比较 `cp_utils` 前后的吞吐/延迟，确保没有意外回退。  
- **异常监控**：在关键路径（如 `forward_npu`, `forward_cuda`）捕获 `AttributeError`，在日志中提示“请使用 attn_cp_metadata”。  
- **功能分支保护**：保持 `is_nsa_prefill_cp_round_robin_split` 的实现不变，若未来计划废弃 round‑robin 方案，确保对应的分支在 `cp_utils` 中有明确标记 `Deprecated`。  
- **Future‑Proof**：考虑在 `cp_utils` 中提供抽象层 `CPMetadata`，让后续如果需要再引入其他上下文并行策略（如 MoE‑CP）时，只需实现子类，而无需再次重构大量调用方。

### fix(ci): repair path filters regressed by #21482 (#23201)
**SHA**: `1d25280` | 🔗 [查看提交](https://github.com/sgl-project/sglang/commit/1d252803f590774f087a4620814da6e8e6b3097c)

**🎯 变更类型**：Bug修复  
**⚡ 重要程度**：🔴 高  
**📋 变更摘要**：本文提交修复了 CI 工作流中路径过滤（paths-filter）规则的回归问题。原先使用 `**/*.!(md|txt)` 语法在部分 GitHub Actions 版本下失效，导致文档或许可证文件的改动错误触发或遗漏工作流。新版改为显式的否定模式 `**/!(*.md|THIRDPARTYNOTICES.txt|LICENSE)`，并统一对 `multimodal_gen` 子目录的过滤规则，确保仅在代码相关改动时触发对应的 CI 阶段。  

**🎯 影响范围**：  
- `.github/workflows/*.yml` 中所有 PR‑test、pr‑test‑*、pr‑test‑sgl‑kernel 等工作流  
- 受影响的 CI 阶段包括 AMD、Xeon、XPU、NPU 以及通用 PR‑test 流程  
- 间接影响的模块：`sgl-kernel`、`python/sglang/multimodal_gen`、`python/sglang/jit_kernel`、`test` 目录  

**🔍 技术洞察**  

- **架构影响**：  
  - 只涉及 CI 配置层面，不会改动运行时代码或服务架构。  
  - 改进了 *dorny/paths-filter* 的匹配逻辑，保证工作流触发条件与实际代码改动保持一致，提升 CI 触发的准确性与可预测性。  

- **性能影响**：  
  - 过滤规则更精确，避免了因无关文件（如 `README.md`、`LICENSE`）导致的冗余工作流执行，从而节约 CI 计算资源、缩短 PR 检测时间。  
  - 由于匹配模式仍然是基于 glob，耗时几乎不变，甚至可能略有下降。  

- **安全考虑**：  
  - 此次改动不涉及业务代码或依赖库，安全风险极低。  
  - 更精确的路径过滤可以防止因文档或许可证文件的误改触发安全相关检查（如 GPL 合规）而产生误报。  

**⚠️ 潜在风险**：  
1. **过滤误判**：若未来新增的非代码文件（比如 `.json`、`.yml` 配置）未在过滤规则中列出，可能导致这些文件的改动不触发相应的 CI，漏掉潜在的测试或 lint。  
2. **GitHub Actions 语法兼容性**：`!(*.md|THIRDPARTYNOTICES.txt|LICENSE)` 依赖于 `extglob` 语法，若 GitHub 更新解析器导致不兼容，可能再次出现过滤失效。  
3. **文档/许可证文件变动未触发 CI**：如果团队希望在修改 `LICENSE` 或 `THIRDPARTYNOTICES.txt` 时运行特定检查（如合规审计），现在这些改动将不再触发 CI，需另行添加专门的工作流。  

**💡 关注建议**：
