### feat: session radix cache (#27058)
**SHA**: `e67b228` | 🔗 [查看提交](https://github.com/sgl-project/sglang/commit/e67b228d4c03fb92014fba6bd7d20b6392ae2db4)

**🎯 变更类型**：功能增强  

**⚡ 重要程度**：🔴高  

**📋 变更摘要**：  
本次提交在 SGLang 的 KV 缓存体系中引入 “Session Radix Cache” 功能（`--enable-session-radix-cache`）。通过在 Radix 树节点上打标签的方式，将 KV 按会话 ID 归属，并在会话关闭时一次性批量回收该会话的 KV。为实现该特性，新增 `SessionRadixCacheMixin`，对 `RadixCache`、调度器 `Scheduler`、`ServerArgs`、以及会话打开/关闭逻辑作了相应改造，并补充单元/手动测试。

**🎯 影响范围**：  
- `python/sglang/srt/mem_cache/radix_cache.py`（核心缓存实现）  
- `python/sglang/srt/mem_cache/session_radix_cache.py`（新 mixin）  
- `python/sglang/srt/managers/scheduler.py`（请求处理及会话管理）  
- `python/sglang/srt/server_args.py`（新增命令行开关及校验）  
- 相关测试文件 `test/manual/core/test_session_radix_cache.py`、`test/registered/unit/server_args/test_server_args.py`  

---

### 🔍 技术洞察

| 维度 | 影响描述 |
|------|----------|
| **架构影响** | 1. **会话感知 KV 缓存**：通过 `SessionRadixCacheMixin` 在 Radix 树节点上维护 `session_ids` 集合，实现 KV 与会话的多对多映射。<br>2. **会话生命周期**：`open_session` 变为**隐式**（仅在显式打开时登记 ID，防止 ID 重复），`close_session` 仅触发 **批量释放**（`release_session`），不再走原来的 `SessionController`。<br>3. **请求路径**：`handle_generate_request` 增加对 “radix‑native 会话” 的判断，禁止使用 `session_params.rid/offset/...`，并在请求对象上写入 `session_id`。<br>4. **依赖限制**：该特性只能在 `--radix-eviction-policy priority` 下使用，防止与现有 LRU/其它策略冲突。 |
| **性能影响** | - **正向**：会话关闭时一次性释放所有该会话的 KV，避免逐条 `evict`，在长会话大量 KV 需要回收时显著降低 CPU 调度和锁竞争。<br>- **负向**：每个树节点需要维护 `session_ids` 集合，引入额外的 Python 对象、集合操作以及在插入/删除时的额外遍历，可能在高并发、短请求场景下产生微小的 CPU 与内存开销。<br>- **内存占用**：额外的 `session_ids`（集合）和 `_session_leaves`（`defaultdict[set]`）会消耗少量额外内存，规模与活跃会话数呈线性关系。<br>- **垃圾回收**：`release_session` 在节点可安全删除后才回收，仍遵循原有 LRU/priority 机制，整体缓存行为保持一致。 |
| **安全考虑** | - 变更不涉及网络协议、权限校验或加密，仅在内部数据结构上做标签，未出现明显的安全漏洞。<br>- 唯一风险是 **会话 ID 重用**：如果用户显式打开同一 ID 两次，旧会话的 KV 可能被错误释放。实现通过 `register_session` 清除 tombstone，已在代码中做防护。<br>- `radix_native_session` 限制了 `session_params` 中的增量特性，防止出现未预期的 KV 合并错误。 |
| **可维护性** | - 引入了 mixin，代码分层清晰；但对 `RadixCache` 的多继承顺序依赖（`SessionRadixCacheMixin, KVCacheEventMixin, BasePrefixCache`），若后续再加入其他 mixin 需注意 MRO。<br>- 新增的内部变量以 `_` 前缀封装，降低意外访问风险。<br>- 单元/手动测试覆盖核心路径，降低回归风险。 |

---

### ⚠️ 潜在风险

1. **会话 ID 重复/滥用**  
   - 若用户在 `enable_session_radix_cache` 场景下显式 `open_session` 后再次使用相同 `session_id`，旧 KV 仍可能被标记为已关闭（tombstone），导致后续请求被忽略或错误释放。需要在业务层确保不重复打开同一 ID，或在 SDK 中加入更严格的 ID 唯一检查。  

2. **并发竞态**  
   - `release_session` 在遍历节点并可能删除节点时，其他线程可能正持有该节点的锁 (`lock_ref != 0`)。虽然代码已检查 `lock_ref` 再决定删除，但在高并发下仍可能出现 **“close 之后仍有请求持有节点”** 的情况，导致 KV 迟迟不被回收，增加内存占用。  

3. **集合对象开销**  
   - 对每个叶子节点维护 `session_ids` 集合，在极端情况下（大量短会话且共享大量前缀）可能导致集合频繁创建/销毁，引发 GC 抖动。  

4. **兼容性**  
   - 必须配合 `--radix-eviction-policy priority` 使用；若用户误配或在已有配置文件中未同步更新，启动时会抛 `ValueError`，但手动修改脚本或 CI 可能遗漏该校验。  

5. **调试与可观测性**  
   - 新增日志主要在 `debug` 级别，默认可能看不到。运维在排查 KV 泄漏或异常释放时，需要打开相应日志级别。  

---

### 💡 关注建议

| 对象 | 建议 |
|------|------|
| **开发者（SDK/业务方）** | - 在使用 `--enable-session-radix-cache` 时，确保 **每个会话 ID 只打开一次**，关闭后若需重新打开必须先调用 `register_session`（如 SDK 中的 `open_session` 隐式调用）。<br>- 禁止在同一会话中使用增量 `session_params`（`rid/offset/replace/drop_previous_output`），否则会被强制 abort。 |
| **运维/部署** | - 强制在启动脚本中添加 `--radix-eviction-policy priority`，或在部署文档中明确依赖关系。<br>- 若需排查 KV 回收异常，开启 `logger.setLevel(logging.DEBUG)` 并关注 `tag session` 与 `release_session` 日志。 |
| **代码维护者** | - 为防止 MRO 混乱，后续若需继续混入功能，建议使用组合而非多继承，或在 `RadixCache` 中显式写明父类顺序。<br>- 考虑在 `SessionRadixCacheMixin` 中对 `register_session`、`release_session` 加入 **幂等** 检查，避免因重复调用导致集合泄漏。 |
| **性能监控** | - 在开启会话缓存后，监控 `protected_size`、`evictable_size` 与 `session_leaves` 的增长趋势，确保不会出现异常累计。<br>- 对比开启/关闭该特性下的 **吞吐** 与 **峰值内存**，验证是否达到预期的“关闭时批量回收”。 |
| **测试** | - 当前已提供手动测试，建议将其迁入 CI（自动化）以防止回归。<br>- 增加 **并发关闭/完成** 场景的压力测试，验证 `lock_ref` 检查的可靠性。 |

### feat: add Mooncake group semantics (#26574)
**SHA**: `62f7ffc` | 🔗 [查看提交](https://github.com/sgl-project/sglang/commit/62f7ffc4929a5c1cdec9ab8f10e5b9eae20fb6ed)

**🎯 变更类型**：功能增强  
**⚡ 重要程度**：🔴高  
**📋 变更摘要**：在 HiCache 的 Mooncake 后端实现了 *group semantics*（`enable_group_semantics`），使 SGLang 能够在写入 Mooncake 时为同一逻辑 HiCache 页的物理 KV 对象批量提供 `group_ids`。新增了对 `ReplicateConfig.group_ids` 的检测、配置开关、group‑id 生成与过滤逻辑，并在所有写路径（单缓冲、Multi‑buffer、V2 接口）中传递该配置。同步加入了完整的单元测试覆盖不同 Mooncake 版本、模型类型和分片场景。

### 🎯 影响范围

- **文档**：`mooncake_store/README.md`（新增使用说明）  
- **测试**：`test/registered/unit/mem_cache/test_mooncake_group_semantics.py`（覆盖新特性）  
- **HiCache 相关组件**：`HiCacheStorageConfig`、`PoolTransfer`、各 `register_mem_*` 接口

### 🔍 技术洞察

| 维度 | 影响分析 |
|------|----------|
| **架构影响** | - **可选特性**：通过 `enable_group_semantics` 开关以及对 `ReplicateConfig.group_ids` 的运行时检测，实现向后兼容；旧版 Mooncake 仍可使用原有三参数 `batch_put_*` 接口。<br>- **职责划分**：所有 group‑id 相关的生成、过滤逻辑封装在 `MooncakeStore` 私有方法 (`_can_use_group_semantics`, `_make_group_id`, `_expand_group_ids`, `_filter_group_ids`) ，对外保持原有 API 不变。<br>- **模块耦合**：仅在 MooncakeStore 层引入 `mooncake.store.ReplicateConfig`，其余 SGLang 代码不依赖 Mooncake 的内部实现，保持模块化。 |
| **性能影响** | - **CPU 开销**：生成 `group_ids`（字符串拼接）和过滤索引在 Python 层执行，时间复杂度 O(N) 与已有键处理相同，实际额外开销极小（毫秒级）。<br>- **网络/存储开销**：`group_ids` 作为额外字段随 `batch_put_*` RPC 发送，字节数约为键长度的 1‑2 倍，整体流量影响可以忽略不计。<br>- **潜在收益**：Mooncake 端可以基于 `group_ids` 实现更高效的元数据路由、租约刷新和批量淘汰，间接提升 HiCache 的命中率与整体吞吐。 |
| **安全考虑** | - **输入来源受控**：`group_ids` 完全由 SGLang 内部拼接得到，不涉及外部用户输入，风险极低。<br>- **向后兼容**：当 Mooncake 版本不支持 `group_ids` 时会自动回退并打印警告，不会导致服务崩溃或数据不一致。<br>- **异常处理**：在长度不匹配的极端情况下抛出 `ValueError`，防止错误的元数据写入 Mooncake。 |
| **可维护性** | - 代码分支清晰（`_use_group_semantics` 标记），后续若 Mooncake 完全移除或改造 `group_ids`，只需关闭开关或删除对应私有方法。<br>- 单元测试覆盖 **检测类属性、实例属性、不同 KV 布局、分片、MLA、索引器、Multi‑buffer** 场景，降低回归风险。 |

---

### ⚠️ 潜在风险
1. **版本兼容性**：若用户自行升级 Mooncake，却未同步更新 `enable_group_semantics` 的默认值或文档，可能出现隐藏的警告或误以为已生效。  
2. **配置误用**：`enable_group_semantics` 开启但 Mooncake 包缺少 `group_ids` 时，仅回退到旧路径并打印警告，若用户忽视日志，可能误判特性已生效。  
3. **长度不匹配错误**：在极端的 `key_multiplier` 计算错误或自定义 `group_ids` 生成逻辑修改后，可能触发 `ValueError`，导致写入失败。
