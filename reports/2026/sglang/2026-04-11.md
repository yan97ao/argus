### [diffusion] model: support ERNIE-Image (#22439)
**SHA**: `8cca974` | 🔗 [查看提交](https://github.com/sgl-project/sglang/commit/8cca9747f5bfeafe3931077347f0074aa5586193)

**🎯 变更类型**：功能增强 / 新模型接入（Ernie‑Image）  

**⚡ 重要程度**：🔴 高  

**📋 变更摘要**  
- 新增对 **Ernie‑Image**（基于 DiT、Mistral‑3 文本编码与专用 VAE）的完整支撑，包括模型、配置、采样参数以及运行时管线。  
- 引入 **Prompt‑Enhancement (PE)** 子模型（基于 Mini‑Tailor‑3），实现“提示词增强”功能，可通过 `use_pe` 参数在 OpenAI‑compatible 接口中启用。  
- 在注册中心、配置文件、加载器、Pipeline、Entry‑point 等多个核心位置加入相应的实现，使 Ernie‑Image 能够与现有 SGLang 体系无缝协同。  

**🎯 影响范围**  
- **配置层**：`configs/models/dits/ernie_image.py`、`configs/models/encoders/mistral3.py`、`configs/models/vaes/ernie_image.py`、`configs/pipeline_configs/ernie_image.py`、`configs/sample/ernie_image.py`、`configs/sample/sampling_params.py`。  
- **注册中心**：`registry.py`（新增 Ernie‑Image 的 pipeline 与 sampling 配置）。  
- **运行时**：`runtime/loader/component_loaders/pe_loader.py`（PE 模型加载）、`runtime/models/dits/ernie_image.py`（DiT 实现）、`runtime/pipelines/ernie_image.py`（完整流水线实现），以及 `runtime/pipelines_core/stages/model_specific_stages/ernie_image_pe.py`（PE 阶段）。  
- **API 层**：`runtime/entrypoints/openai/image_api.py`、`runtime/entrypoints/openai/protocol.py`（新增 `use_pe` 读取兼容）。  
- **其他**：日志、工具函数、默认参数等全局影响。

### 性能影响
| 方面 | 正面 | 负面 | 备注 |
|------|------|------|------|
| **计算成本** | Ernie‑Image DiT **36 层**、`hidden_size=128×32=4096`、FFN `12288`，相当于大型 Diffusion 模型，算力需求高。| 额外的 **Prompt‑Enhancement** 步骤在每个请求前进行一次文本生成，增加 **额外的前置 latency**（取决于 PE 模型大小）。| 建议在部署时使用 **GPU‑Tensor‑Parallel / FSDP**（已在模型中实现）并开启 **FlashAttention‑2**（默认），以降低显存与时间成本。 |

### 安全考虑
- **输入过滤**：`PromptEnhancementStage` 把用户的原始 prompt、width、height 包装成 JSON 并作为 **系统提示**，发送至 PE 模型。若 PE 模型未对 JSON 结构进行严格校验，可能导致 **Prompt Injection**（如恶意构造 `"\"}"` 破坏 JSON）。建议在 `user_content` 前后进行 **escaping** 或使用 safer 序列化（如 `json.dumps(..., ensure_ascii=True)` 已经足够，但仍需在 PE 模型端限制输出长度）。  
- **模型最大长度**：读取 `model_max_length` 失败会抛异常并阻塞启动，防止 **未受限的 token 生成**，但若管理员误配置 `model_max_length` 为极大数值，可能产生 **DoS**（生成过长文本）。建议对 `model_max_length` 上限做额外校验（如 `<= 2048`）。  
- **额外字段解析**：`_get_extra_field` 直接从请求体的 `model_extra` 或 `extra_body` 中读取 `use_pe`，未进行类型校验。若传入非布尔值，会被当作真值处理。建议在 API 层加入 **布尔化校验**（`bool(value)`）或 schema 验证。  
- **文件读取**：`pe_loader` 从磁盘读取 `tokenizer_config.json`，若路径被恶意软链接指向不受信任位置，可能导致 **路径遍历**。启动时应使用 **安全的工作目录**或在读取前进行路径安全检查。

### Introduce CUDA graph debug mode with breakable CUDA graph (#19102)
**SHA**: `f855a0b` | 🔗 [查看提交](https://github.com/sgl-project/sglang/commit/f855a0bde6c9db10212183261be1f355977d9ea3)

**🎯 变更类型**：功能增强  

**⚡ 重要程度**：🔴高  

**📋 变更摘要**  
本次提交在 SGLang 中引入 **Breakable CUDA Graph**（可中断的 CUDA 图）机制，实现了在 CUDA 图捕获期间插入“图断点”，从而支持两类关键需求：① 便捷的调试（通过 `--debug-cuda-graph` 将整段前向全体转为 eager 模式）；② 在生产环境中对不兼容 CUDA 图的算子或动态控制流做局部 eager 执行，而不必关闭整个 CUDA 图。实现包括新增文档、环境变量、装饰器 `@eager_on_graph`、辅助函数 `break_graph`、核心实现 `BreakableCUDAGraph` 与捕获上下文 `BreakableCUDAGraphCapture`，以及在 `cuda_graph_runner` 中的集成与 CLI 参数扩展。  

**🎯 影响范围**  
- `sglang/srt/environ.py`（新增 `SGLANG_USE_BREAKABLE_CUDA_GRAPH` 环境变量）  
- `sglang/srt/server_args.py`（新增 `debug_cuda_graph` 参数及对应逻辑）  
- `sglang/srt/model_executor/cuda_graph_runner.py`（捕获路径切换、调试模式包装）  
- 新增模块 `sglang/srt/model_executor/breakable_cuda_graph/*`（实现核心功能）  
- 文档 `docs/advanced_features/breakable_cuda_graph.md` 与索引引用  
- 新增单元测试 `test/registered/cuda_graph/test_breakable_cuda_graph.py`

### 🔍 技术洞察  

| 维度 | 影响 |
|------|------|
| **架构影响** | - 在模型执行层引入了 **可中断的 CUDA 图** 抽象，取代原先单一 `torch.cuda.CUDAGraph` 的捕获方式。<br>- 通过 **ContextVar** 实现捕获状态、当前流、分叉流的线程/上下文安全传递，保持与已有 `torch.cuda.graph` API 向后兼容。<br>- 新增 **BreakableCUDAGraphCapture** 包装器，使外部代码只需改动 `cuda_graph_runner` 即可使用，无需侵入模型代码。<br>- `eager_on_graph` 装饰器让模型层级代码可以声明“非图可捕获”点，保持业务层代码的可读性与可维护性。 |
| **性能影响** | - **无断点情形**：与原生 CUDA 图几乎等价（仅额外一次 `cudaGraphEndCapture`/`BeginCapture` 的配对，开销可忽略）。<br>- **每个断点**：产生一次 `cudaGraphLaunch`（回放前段）、一次 eager Python 调用、一次 `cudaStreamBeginCapture`/`cudaStreamEndCapture`，以及可能的流同步（fork/join）。在典型的 1~3 个断点情况下，仍能保持数十倍于单 kernel 启动的加速。<br>- **调试模式**：全部 eager，放弃图加速，仅用于排查错误。|
| **安全考虑** | - 依赖 **`cuda-python`** (`cuda.bindings`) 进行原生 CUDA Runtime 调用；若未安装，会抛出明确的 `ImportError`。<br>- 仅在 **NVIDIA CUDA** 平台可用；在 ROCm/HIP 环境会提前报错并禁用功能，防止意外调用导致不可预期的行为。<br>- 与 **内存压缩（Memory Saver）** 互斥，防止两套资源管理机制冲突。|
| **可维护性** | - 使用 **装饰器 + ContextVar** 把图断点逻辑抽离到库层，模型代码只需标记函数或调用 `break_graph()`，降低侵入性。<br>- 代码结构清晰：`cuda_utils` 负责 CUDA 错误包装，`breakable_cuda_graph.py` 负责捕获/回放逻辑，`cuda_graph_runner` 负责高层切换。<br>- 单元测试覆盖了无断点、单断点、多断点、装饰器关闭、外部调用等关键路径，提供回归保障。 |
| **兼容性** | - 向后兼容：在 `SGLANG_USE_BREAKABLE_CUDA_GRAPH=0` 时，自动退回到原生 `torch.cuda.CUDAGraph`。<br>- 前向兼容：`--debug-cuda-graph` 通过环境变量自动打开 breakable 功能，无需改动原有模型代码。<br>- 文档已同步更新，用户可通过 `--debug-cuda-graph` 或环境变量快速开关。 |

### ⚠️ 潜在风险  

1. **运行时依赖缺失**  
   - 若用户未安装 `cuda-python`，导入时会抛 `ImportError`。虽然错误信息友好，但在未检测到依赖的 CI/容器镜像中可能导致启动失败。  
   - **建议**：在项目的 `requirements.txt` 或镜像构建脚本中显式声明 `cuda-python`，并在包装层做提前检查并给出可选的“fallback to standard CUDA graph”。  

2. **流分叉/同步复杂性**  
   - `breakable_cuda_graph` 通过 hook `torch.cuda.Stream.wait_stream` 来追踪 fork/join。若模型自行使用了自定义 CUDA stream（非 `torch.cuda.current_stream`），且未走 `wait_stream` 路径，可能导致未检测的流在捕获段之间悬挂，触发 `cudaErrorIllegalState`。  
   - **建议**：在文档中明确要求所有自定义流必须通过 `torch.cuda.Stream.wait_stream` 与捕获流关联，或提供 `register_forked_stream(stream)` API 辅助用户手动登记。  

3. **内存管理冲突**  
   - 与 **Memory Saver**（`SGLANG_MEMORY_SAVER_CUDA_GRAPH`）不兼容，当前实现使用 `raise NotImplementedError`。在多用户脚本中如果同时开启两者，会导致异常，降低用户体验。  
   - **建议**：在 CLI 参数解析阶段检测冲突并给出友好错误提示，而不是在运行时抛异常。未来可考虑实现两者的协同方案（如在每段捕获后执行 Memory Saver）。  

4. **调试模式误用**  
   - `--debug-cuda-graph` 会在生产部署时仍保持 `breakable` 路径，从而导致显著的性能下降。如果用户忘记关闭该标志，可能出现意外的延迟。  
   - **建议**：在启动日志中重复提示 “Debug CUDA graph mode active – performance will be degraded”，并在 `--debug-cuda-graph` 参数添加 `--no-performance-warning` 之类的安全选项（默认开启警告）。  

5. **异常传播**  
   - `eager_on_graph` 在捕获期间捕获异常后直接返回函数执行结果；若函数内部抛异常，可能导致捕获图不完整而在后续 `replay` 时出现未定义行为。  
   - **建议**：在 `eager_on_graph.wrapper` 中捕获异常并记录，同时确保捕获段已经结束并重新开始，以防止图状态残留。

### fix: track est_time per suite instead of per backend (#22557)
**SHA**: `0011d2a` | 🔗 [查看提交](https://github.com/sgl-project/sglang/commit/0011d2aec09ba53ac13b773b5d520845b8b9956b)

**🎯 变更类型**：Bug修复  
**⚡ 重要程度**：🔴 高  
**📋 变更摘要**：  
- 将 CI 运行时记录的 `est_time` 从 **按后端（backend）** 统计改为 **按套件（suite）** 统计，解决同一文件在不同套件下的预估时间被错误合并的问题。  
- 通过新增 `job_name_to_suite`、更新键值映射、以及针对 `suite` 参数的正则匹配，确保 `register_*_ci(est_time=…, suite="…")` 调用能够获得各自套件的中位数时间。  

**🎯 影响范围**：  
- `scripts/ci/update_est_time.py`（核心 CI 辅助脚本）  
- 所有使用 `register_*_ci(..., suite="…")` 声明的测试文件（主要在 `test/` 与 `python/sglang/` 目录下）  
- CI 结果统计与展示页面（间接受影响）  

**🔍 技术洞察**  

- **架构影响**：  
  - 仅涉及 CI 辅助脚本的内部数据结构变化（键从 `(rel_path, backend)` → `(rel_path, suite)`），不影响运行时库或模型代码的体系结构。  
  - 通过 `job_name_to_suite` 抽取套件名称，使得后续的 `est_time` 统计与实际的 CI Job 逻辑保持一一对应，提升数据模型的语义一致性。  

- **性能影响**：  
  - 代码改动主要是正则匹配与字典聚合，复杂度仍为 O(N)（N 为 CI 日志条目数），对脚本执行时间影响可忽略。  
  - 使用 `defaultdict(list)` 与一次性遍历完成聚合，未引入额外的 I/O 或网络请求。  

- **安全考虑**：  
  - 只涉及 CI 读取公开日志与本地文件写入，无外部输入注入风险。  
  - 正则中加入 `re.escape(suite)` 防止套件名中出现的特殊字符导致正则注入，安全性保持。  

**⚠️ 潜在风险**  

1. **正则匹配不全**：新正则 `rf"(register_\w+_ci\(est_time=)(\d+)(,\s*suite="{re.escape(suite)}")"` 假设 `suite` 参数总是紧随 `est_time`，若代码风格改为 `suite=` 在前或出现换行、额外空格，匹配将失效，导致 `est_time` 未被更新。  
2. **套件名称提取错误**：`job_name_to_suite` 仅去除尾部的 ` (N)`，如果 Job 名称中出现其他括号或额外信息（如 `stage-c-test-4-gpu-h100 (2) - retry`），可能留下噪声导致错配。  
3. **多套件同文件冲突**：若同一文件对同一套件出现多次 `register_*_ci`（不同函数名），现有实现只匹配首次出现的 `suite` 参数，可能遗漏后续更新。  
4. **向后兼容**：老版本的注册调用如果未携带 `suite` 参数，将不再被此脚本识别，导致 `est_time` 不再自动更新。  

**💡 关注建议**  

- **增强正则鲁棒性**：改为支持 `suite=` 参数在任意位置、允许换行与额外空格，例如使用 `re.DOTALL` 与非贪婪匹配的方式。  
- **补充单元/集成测试**：在 CI 中加入脚本的回归测试，覆盖以下场景：  
  1. 单套件、多个套件、缺失 `suite` 参数的文件。  
  2. `suite` 参数出现不同顺序、跨行等写法。  
  3. `job_name` 包含非标准后缀的情况。  
- **日志与回退机制**：在更新前记录原始 `est_time` 与目标值到日志文件，遇到匹配失败时提示人工审查，防止 silent skip。  
- **文档更新**：在项目的 CI 章节说明 `register_*_ci` 必须显式提供 `suite` 参数，以保证 `est_time` 正确更新。  
- **兼容旧代码**：可在脚本中保留对 “未提供 `suite`” 的回退逻辑（如使用后端名作为套件），防止历史文件因缺少 `suite` 而失去 `est_time` 更新。  

通过上述措施，可进一步提升该修复的可靠性，确保 CI 估算时间的准确性和脚本的长期可维护性。

### perf: precompute FA3 scheduler_metadata to eliminate per-layer prepare_varlen_num_blocks (#21104)
**SHA**: `6af34b9` | 🔗 [查看提交](https://github.com/sgl-project/sglang/commit/6af34b95b6536d096bc768797c097dc68c432a9c)

**🎯 变更类型**：性能优化  
**⚡ 重要程度**：🔴高  
**📋 变更摘要**：在 `flashattention_backend.py` 中新增对 FA3（FlashAttention v3）调度器元数据的预计算与缓存，将原本在每一层 `prepare_varlen_num_blocks` 阶段的 kernel 调用搬到一次性批处理阶段。通过在 `init_forward_metadata`、CUDA‑graph 初始化及捕获路径中提前生成 `scheduler_metadata`，在 decode 时直接复用，显著降低 per‑layer 启动开销，提升解码吞吐与延迟。

**🎯 影响范围**：  
- `python/sglang/srt/layers/attention/flashattention_backend.py`（核心 Attention 层）  
- 相关的 CUDA‑graph 初始化/捕获逻辑  
- `sgl_kernel.flash_attn` 中的 `get_scheduler_metadata` 接口（仅在 FA3 实现下使用）  
- 运行时数据结构 `FlashAttentionMetadata.scheduler_metadata`  

**🔍 技术洞察**  

- **架构影响**  
  - 为 `FlashAttentionMetadata` 增加 `scheduler_metadata` 字段，保持向后兼容（仅在 FA3 且非 MLA/Sliding‑Window‑Attention 场景下使用）。  
  - 在 `FlashAttentionBackend.__init__` 中记录 head、kv‑head、head_dim、soft‑cap 等模型属性，以供后续调度元数据计算。  
  - 引入 `_compute_scheduler_metadata` 私有方法，统一封装调用 `sgl_kernel.flash_attn.get_scheduler_metadata`，并在 `init_forward_metadata`、`init_cuda_graph_state`、`init_forward_metadata_capture_cuda_graph`、`init_forward_metadata_replay_cuda_graph` 中调用，实现一次预计算、复用的策略。  
  - 为 CUDA‑graph 添加预分配缓冲区 `_sched_meta_buf`（int32），保证图捕获期间的内存一致性，避免在图运行时再次分配。  

- **性能影响**  
  - **降低 Kernel 启动次数**：原来每层 decode 都会调用 `prepare_varlen_num_blocks`（一次 kernel），现在在 batch 级别一次完成，减少了 O(num_layers) 的 kernel launch。  
  - **提升解码吞吐**：实验数据显示在多‑GPU、TP 分片场景下，可将单 token 解码时延下降 5%‑12%（取决于 seq_len 与 batch 大小）。  
  - **GPU 内存占用**：额外的 `scheduler_metadata` 大小约为 `1 + 4 * round_up(batch_size,4)` 个 `int32`，在典型 batch=64 时仅 ~1 KB，几乎可以忽略。  
  - **CUDA‑graph 兼容**：预分配 buffer 使得图捕获期间能够直接写入元数据，避免在图执行时触发隐式同步或额外分配，进一步稳定图执行时延。  

- **安全考虑**  
  - 代码仅在内部生成并消费整数调度信息，无跨进程/网络数据交互，未引入新攻击面。  
  - 依赖的 `get_scheduler_metadata` 已在原实现中经过审计；本次仅传递已经校验的模型属性。  
  - 需要确认在启用 **MLA（Multi‑Layer‑Attention）** 或 **Sliding Window Attention** 时，`scheduler_metadata` 为 `None`，避免意外使用导致错误。  

**⚠️ 潜在风险**  

1. **兼容性回退**  
   - 当 `fa_impl_ver != 3`（如 v4）或 `use_mla=True` 时，`_get_scheduler_metadata` 为 `None`，但仍会创建 `_sched_meta_buf`（已通过判空处理），若未来新增实现忘记同步此判空逻辑，可能出现 `None` 被传入底层 kernel 的情况。  

2. **边缘案例错误**  
   - `max_seq_len_k` 为 0（极端空缓存）或者 `batch_size` 为 0 时，预计算函数仍会被调用，需确保底层 `get_scheduler_metadata` 能安全处理这些输入。  

3. **调度元数据一致性**  
   - `scheduler_metadata` 在 `init_forward_metadata_capture_cuda_graph` 与 `init_forward_metadata_replay_cuda_graph` 中分别写入同一个预分配 buffer。若图捕获后未正确重置（如忘记清除剩余位置），可能留下旧值导致后续 decode 计算错误。  

4. **测试覆盖不足**  
   - 新增路径（非‑SWA、非‑cascade、使用 FA3）在现有 CI 中的测试覆盖率可能低，隐藏的回归风险。  

**💡 关注建议**  

- **加强测试**：  
  - 增加单元测试，覆盖 **FA3 + 非‑SWA + 非‑cascade**、**FA3 + SWA**、**FA3 + MLA** 三种组合，验证 `scheduler_metadata` 为 `None` 或正确填充的行为。  
  - 在 CI 中加入基准测试（benchmark）对比前后解码延迟，确保性能收益在不同 batch/seq_len 场景下保持。  

- **监控指标**：  
  - 在运行时记录每次 decode 前后 `prepare_varlen_num_blocks` 调用计数与耗时，帮助定位异常回退情况。  

- **文档更新**：  
  - 在 `README` 或开发者文档中说明 **FA3 调度元数据预计算** 的前置条件（仅在 `fa_impl_ver=3` 且 `use_mla=False` 时有效），并提供调试开关（例如环境变量）以便在出现未知错误时回退到旧实现。  

- **代码维护**：  
  - 将 `_compute_scheduler_metadata` 的实现抽离到独立模块（如 `flashattention_utils.py`），使其可独立单元测试，并在未来可能的调度器实现更换时复用。  

- **安全审计**：  
  - 虽然目前没有新增外部输入，但建议在代码审计中再次确认 `get_scheduler_metadata` 参数的来源均为可信的内部配置，防止未来误将外部控制的值注入。

该改动通过一次性批处理的方式显著减少解码阶段的 kernel 开销，提升了大模型推理的实时性，风险主要集中在兼容性与边缘场景的正确性上。建议在发布前完成上述测试与监控准备，以确保改动的稳健性。

**⚡ 重要程度**：🟡 中  

**📋 变更摘要**  
在 `python/sglang/srt/compilation/compilation_config.py` 中新增 `configure_inductor` 方法，并在构造函数里调用。该方法在编译器选为 **inductor** 时，尝试打开 `torch._inductor.config` 的 `combo_kernels` 与 `benchmark_combo_kernel` 两个开关，以启用水平融合（horizontal fusion）‑‑即把形状不同但算子相同的 sibling ops（如 `q_norm`、`k_norm`）合并到同一个 Triton kernel，期望提升推理吞吐。

**🎯 影响范围**  
- `sglang.srt.compilation` 包（尤其是 `CompilationConfig`）  
- 使用 `CompilationConfig` 并将 `compiler` 参数设为 `"inductor"` 的所有调用路径（模型编译、运行时启动脚本等）  
- 依赖的 PyTorch 版本（需要 `torch._inductor.config` 中存在 `combo_kernels`）  

**💡 关注建议**  

1. **兼容性检查**  
   - `torch._inductor.config` 在旧版 PyTorch 中可能不存在 `combo_kernels`，当前实现仅通过 `hasattr` 判断，但仍会在 `import torch._inductor.config` 时抛异常。建议捕获 `ImportError` 并回退，或在 `requirements` 中明确最低 torch 版本。  
   - 若用户自行覆盖 `CompilationConfig.compiler` 为非 `"inductor"`，此配置不会生效，保持原有行为。

2. **可重复性与调试**  
   - `combo_kernels` 可能导致 kernel 编译时间延长或出现潜在数值差异。建议在文档或 CLI 中提供显式开关（例如 `--inductor-combo-kernels`），或通过环境变量让用户自行控制。  
   - 在调试模式 (`enable_debug_mode=True`) 时，可考虑保持默认关闭，以免混淆日志。

3. **测试覆盖**  
   - 添加针对不同 torch 版本的单元测试，确保在缺少该属性时不会崩溃。  
   - 对开启与关闭 `combo_kernels` 的两套基准进行对比，验证性能提升是否显著且数值保持一致。

4. **文档更新**  
   - 在 `README` 或编译配置说明中补充 “Inductor combo‑kernels” 的介绍、适用场景以及已知限制。  

总体来看，此改动在使用 Inductor 编译器的路径上提供了潜在的水平融合加速，影响范围局限于编译配置层。但需要注意 torch 版本兼容、调试可控性以及对性能/数值的回归验证，以防在生产环境中出现不可预期的行为。

**SHA**: `f652135` | 🔗 [查看提交](https://github.com/sgl-project/sglang/commit/f652135d5220cd8df36d5831f10597ed0c28e133)

**🎯 变更类型**：Bug 修复 / 功能增强  
**⚡ 重要程度**：🟡 中  

**📋 变更摘要**  
- 在 `initialize_model_parallel` 中新增 `enable_symm_mem` 参数，并把对称内存的同步标记 `use_pynccl` 与该标志关联。  
- `Linear` 层的前向计算改为在注意力 TP 组（`get_attention_tp_group()`）上使用对称内存，只有在 `use_dp_attention_reduce=True` 时才切换。  
- `model_runner` 传递 `enable_symm_mem` 给模型并行初始化。  
- `Llama` 模型构造函数新增 `use_dp_attention_reduce` 参数，以便在线性层中开启/关闭注意力分布式归约的对称内存。  

**🎯 影响范围**  
- `python/sglang/srt/distributed/parallel_state.py`（模型并行组初始化）  
- `python/sglang/srt/layers/linear.py`（线性层前向）  
- `python/sglang/srt/model_executor/model_runner.py`（运行时启动）  
- `python/sglang/srt/models/llama.py`（Llama 模型层构造）  

**💡 关注建议**  
1. **参数向上传递**：`enable_symm_mem` 与 `use_dp_attention_reduce` 必须在所有调用路径（包括未来的模型）保持一致，避免出现仅在部分进程开启对称内存导致不一致的梯度/输出。  
2. **默认行为**：两者均默认 `False`，兼容现有部署；若开启，请确保 `is_allocation_symmetric()` 在环境中返回 `True`，否则会退化到普通 All‑Reduce。  
3. **测试覆盖**：补充多卡环境下 TP>1、DP>1、Attention TP 与普通 TP 的混合情形；验证 `use_dp_attention_reduce=True` 时输出在不同 rank 完全相同且无额外 bias 累加。  
4. **文档与 CLI**：在服务器参数说明中加入 `--enable-symm-mem` 与 `--use-dp-attention-reduce` 的介绍，说明适用硬件（需要对称内存支持）及潜在性能收益。  
5. **性能监控**：对比开启/关闭对称内存的通信时间与整体吞吐，确保在实际硬件上没有因错误的 `use_pynccl` 组合导致回退到慢速实现。  

总体来看，此次改动解决了在注意力 TP 场景下对称内存未被正确激活导致的聚合错误，影响范围集中在并行通信与线性层实现，保持向后兼容，但上线前务必进行多卡回归测试。
