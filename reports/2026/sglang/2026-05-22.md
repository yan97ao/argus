### [diffusion] feat: support role-based component loading and stage affinity (#25168)
**SHA**: `8c916a7` | 🔗 [查看提交](https://github.com/sgl-project/sglang/commit/8c916a715cf6933abf63156dfef206b823abf29d)

**🎯 变更类型**：功能增强（引入基于角色的组件加载、阶段亲和性以及相应的过滤/校验机制）

**⚡ 重要程度**：🔴 高  

**📋 变更摘要**  
1. 新增 `RoleType` 枚举及其辅助函数，实现 **Encoder / Denoiser / Decoder / Monolithic** 四类角色的统一标识与模块过滤。  
2. `ComposedPipelineBase` 重构：在初始化时校验角色、依据角色过滤必须的配置模块、引入 `extra_allowed_modules` 机制、提供 **stage factory** 与 **_should_add_stage_for_role** 辅助方法，实现 **按角色动态组装阶段**。  
3. 多条现有 Pipeline 与 Stage（Hunyuan3D、QwenImage、MOVA、Flux2、LTX2 等）逐步适配：新增 `role_affinity` 属性、在创建阶段时使用 `add_stage_factory`、修正 VAE/Transformer 等跨角色依赖的显式声明。  
4. `ImageVAEEncodingStage` 支持自定义组件名称，配合角色过滤使用。  
5. 大量单元测试 `test_disagg_roles.py` 覆盖角色解析、模块过滤、额外模块映射、阶段构造及 dtype 推断逻辑，确保新特性行为正确。  

**🎯 影响范围**  
- **核心框架**：`sglang/multimodal_gen/runtime/disaggregation/roles.py`、`sglang/multimodal_gen/runtime/pipelines_core/composed_pipeline_base.py`、`sglang/multimodal_gen/runtime/pipelines_core/stages/*`。  
- **具体 Pipeline 实现**：`hunyuan3d_pipeline.py`, `qwen_image.py`, `mova_pipeline.py`, `flux_2.py`, `ltx_2_pipeline.py`, `wan_i2v_pipeline.py` 等。  
- **组件调度/管理**：`ComponentResidencyManager` 与 `ComponentUse` 记录方式有所变化。  
- **测试套件**：新增 `test/unit/test_disagg_roles.py`（约 640 行），覆盖几乎所有新逻辑。

## 🔍 技术洞察

### 架构影响
- **角色抽象层**：引入 `RoleType` 为模型拆分（Disaggregation）提供统一语义，使得 **Encoder / Denoiser / Decoder** 能够在同一进程/机器上独立加载所需子模块。  
- **模块过滤**：通过 `filter_modules_for_role` 与 `extra_allowed_modules`，在 `ComposedPipelineBase.__init__` 中提前裁剪 `required_config_modules`，显著降低 **GPU/CPU 内存占用**（仅加载当前角色所需的权重）。  
- **阶段亲和性**：所有 Stage 现在可以声明 `role_affinity`（属性或默认实现），`add_stage_factory` 根据当前角色决定是否实例化对应 Stage，从而 **避免无用 Stage 的构造开销**。  
- **灵活的跨角色依赖**：通过 `extra_allowed_modules` 机制，现有 Pipeline 能在 Encoder/Decoder 上显式保留跨角色必需的模块（如 Encoder 仍需 VAE 进行 latent 编码），实现 **细粒度控制** 而不破坏已有 Pipeline 配置。  
- **统一校验入口**：`validate_disagg_role` 为每个 Pipeline 提供角色合法性的统一检查点，防止不支持的组合（如 Hunyuan3D 的 paint 模式）被错误启动。  

### 性能影响
| 维度 | 正面效应 | 可能的负面效应 |
|------|----------|----------------|
| **显存占用** | 仅加载角色对应的子模型（如 Encoder 只加载 TextEncoder、Tokenizer、Scheduler），显著降低跨节点显存需求。 | 额外的过滤逻辑在初始化阶段略有开销（几毫秒），对整体推理时延影响可忽略。 |
| **加载时间** | 因模块数量减少，`load_modules` 的 I/O 与模型反序列化时间缩短。 | 若 `extra_allowed_modules` 配置不完整，可能导致缺失关键模块而在运行时触发错误，需要额外的调试成本。 |
| **推理吞吐** | 角色拆分后可在 **FSDP / ZeRO** 等分布式策略中更细粒度地进行梯度/参数分配，提升多卡并行度。 | Stage 创建的延迟通过 `stage_factory` 仍会在首次调用时触发一次实例化，首次推理可能稍慢。 |
| **代码路径复杂度** | 通过统一 `add_stage_factory`，新老代码统一，后续扩展更易维护。 | 需要在所有自定义 Stage 中实现或继承 `role_affinity`，遗漏时可能导致意外被过滤。 |

### 安全考虑
- **配置安全**：`validate_disagg_role` 增加了对不兼容配置的显式检查（如 `paint_enable=True` 与非 `MONOLITHIC`），防止误用导致崩溃或资源泄露。  
- **类型安全**：`RoleType.from_string` 现在兼容旧别名（`denoising`），避免因用户输入错误导致 `ValueError`。  
- **平台特性**：在 MOVA 的 torch.compile 路径加入 ROCm（HIP）检测，防止在不支持的硬件上触发非法 kernel，提升 **平台安全性**。

## ⚠️ 潜在风险

| 风险点 | 说明 | 可能影响 |
|--------|------|----------|
| **角色过滤不完整** | `extra_allowed_modules` 仅在几条 Pipeline 中显式声明，若后续新增 Pipeline 没有维护该映射，可能导致 Encoder/Decoder 启动时缺失跨角色必需的模块（如 VAE、Audio VAE）。 | 运行时报错 `KeyError` 或模型功能缺失。 |
| **Stage 亲和性遗漏** | 如果自定义 Stage 未实现 `role_affinity`（或误设为错误角色），在对应角色运行时会被错误过滤，导致 Pipeline 不完整。 | 功能缺失、推理异常。 |
| **破坏向后兼容** | `filter_modules_for_role` 新增 `extra_allowed_modules` 参数，已有外部调用若未使用关键词参数，可能因签名不匹配（***已通过关键字参数***，但仍需注意新参数默认行为）。 | 可能导致旧代码在 Python 3.7‑以下环境报错（虽项目已 3.10+）。 |
| **CPU/GPU dtype 推断** | `Hunyuan3DShapeBeforeDenoisingStage._resolve_runtime_dtype` 依赖 `conditioner` 参数遍历，若 `conditioner` 实现异常（非迭代）会触发 warning 并回退。 | 可能出现意外的 dtype（如使用 float16 而期望 float32），导致数值误差。 |
| **日志噪声** | `add_stage` 与 `add_stage_factory` 在跳过阶段时会频繁打印 `info`，在大规模分布式部署中可能产生大量日志。 | 日志审计成本上升。 |

## 💡 关注建议

1. **完整维护 `extra_allowed_modules`**  
   - 为每个 Pipeline（尤其是新加入的）在 `ComposedPipelineBase._get_extra_allowed_modules_for_role` 中补全对应映射。  
   - 编写 CI 检查：验证 `filter_modules_for_role` 的输出必包含所有在原 `required_config_modules` 中的 **跨角色必需** 模块。

   - 在 `PipelineStage` 基类中提供默认实现返回 `RoleType.MONOLITHIC`，并在文档中强调所有自定义 Stage 必须显式声明（或继承基类）。

**⚠️ 潜在风险**  

1. **Wheel 重命名脚本兼容性**  
   - 脚本依赖 `python -m wheel` 与 `sed`，在不同的构建镜像（尤其是非 Linux）上可能缺失对应工具或行为差异。  
   - 如果 `METADATA` 中已有 `+musa` 标记，脚本会跳过但不更新平台标签，可能导致 **manylinux2014** 与实际标签不匹配。  

2. **模型缓存路径硬编码**  
   - `hf_cached_model` 直接使用 `snapshot_download(..., local_files_only=True)`，在 CI 机器上若缓存不完整会导致测试报错，需保证缓存同步策略可靠。  

3. **CI 并行矩阵**

**SHA**: `4374789` | 🔗 [查看提交](https://github.com/sgl-project/sglang/commit/4374789abf17f6df581100b367f5d8eed89c2604)

**🎯 变更类型**：功能增强 / 性能优化 / 重构  
**⚡ 重要程度**：🔴高  
**📋 变更摘要**

- 所有调用这些接口的模型 processor（如 `clip.py、deepseek_vl_v2.py、internvl.py` 等）同步调用改为 `await`，实现非阻塞加载。  
- 目标是让多模态数据的 I/O（图片、视频、音频解码、网络下载等）在 SGLang Server 的健康检查阶段不再阻塞，从而避免 “health check stalls”。  

**🎯 影响范围**  
- `python/sglang/srt/multimodal/processors/` 目录下的全部 30+ processor 实现。  
- `BaseProcessor`（`python/sglang/srt/multimodal/processors/base_processor.py`）的核心加载逻辑。  
- 与 multimodal 数据加载相关的异步调度路径、事件循环的使用方式。  

**🔍 技术洞察**  

- **架构影响**  
  - 引入 `asyncio` 使 multimodal 加载成为协程，符合 SGLang 其余基于 async 的推理路径（如 `process_mm_data_async`）。  
  - 通过 `asyncio.wrap_future` 把 `ThreadPoolExecutor`（或其他 `concurrent.futures`）的阻塞任务桥接到事件循环，保持原有线程池并行解码能力，同时让上层协程可调度。  
  - 改动后，健康检查线程/协程在等待 multimodal 数据时不会阻塞整个服务进程，提升系统的容错和可观测性。  

- **性能影响**  
  - **正面**：I/O 密集的图片/音频/视频读取现在可以在同一事件循环中与其他请求共享 CPU，提升并发度，减小健康检查的延迟。  
  - **潜在**：大量 `await asyncio.wrap_future(...)` 会产生额外的上下文切换，若线程池规模不足，仍可能出现排队延迟。建议根据部署机器的 CPU/IO 特性调优 `ThreadPoolExecutor`（如 `max_workers`）。  
  - **内存**：保持原有 `Future` 对象的引用，瞬时内存占用基本不变。  

- **安全考虑**  
  - 代码路径从同步改为异步，未引入新的外部依赖或权限变更。  
  - `asyncio.wrap_future` 在异常传播上保持原有 `Future` 的异常信息，无额外信息泄露风险。  
  - 需要确认未在非协程上下文（如启动脚本的同步入口）直接调用 `await load_mm_data`，否则会抛 `RuntimeError: no running event loop`。  

**⚠️ 潜在风险**  
1. **事件循环缺失**：某些老旧调用方（如单元测试、脚本化工具）可能仍以同步方式调用 `load_mm_data`，导致运行时错误。  
2. **线程池并发饱和**：`fast_load_mm_data`、`legacy_load_mm_data` 仍依赖 `concurrent.futures.ThreadPoolExecutor`，如果并发请求激增，仍会产生阻塞，进而影响整体响应时间。  
3. **异常处理差异**：原本 `future.result()` 抛出的异常在 `await asyncio.wrap_future` 中会被包装为 `asyncio.CancelledError` / `Exception`，需要确认上层异常捕获逻辑仍然兼容。  
4. **兼容性**：Python 3.7+ 才完全支持 `asyncio.wrap_future`，在极低版本环境下可能报错（不过项目已声明最低 3.9）。  

**💡 关注建议**  
- **回归测试**：新增针对健康检查的压力测试，确保在高并发 multimodal 输入下，`/health` 接口不再超时。  
- **文档更新**：在开发者文档中标明 `load_mm_data` 已改为 `async`，所有同步调用必须改为 `await` 或使用 `asyncio.run` 包裹。  
- **线程池调优**：根据实际部署机器的 I/O 特性，提供配置项 (`max_workers`) 供运维调节，以防止线程池成为瓶颈。  
- **异常统一**：在 `BaseProcessor` 中统一捕获 `asyncio` 异常并转换为项目自定义错误类型，避免上层代码因异常类型变化而失效。  
- **CI 检查**：加入 lint/typing 检查，确保所有 `load_mm_data` 调用点均已改为 `await`，防止遗漏导致运行时错误。

| **降低耦合** | 通过显式的依赖注入（`ForwardContext`）取代隐式属性引用，提升代码可读性、可维护性，后续可以在不修改 `ForwardBatch` 的前提下替换/扩展后端实现。 |

### 2. 性能影响
| 维度 | 正面效应 | 负面 / 注意点 |
|------|----------|---------------|
| **前向调用开销** | `forward_context` 只是一次对象创建与 `set_forward_context`（赋值）以及 `contextmanager` 的 `__enter__/__exit__`，开销极小（< 1µs），对整体推理吞吐几乎没有影响。 |
| **函数调用路径** | 许多后端代码改为 `get_attn_backend()` 再访问成员，增加一次函数调用，但 Python 层面成本微乎其微，主计算仍在 CUDA/Trition/FA 内核中。 |
| **缓存一致性** | 通过后端自身保存的 pool 绑定，避免了 `ForwardBatch` 被删后仍残留的 stale 引用，减少了错误导致的额外调试/重试成本。 |
| **子批次并行** | 现在可以在同一图捕获阶段为每个子批次提供独立 `attn_backend`（如 TBO 子批次），提升了 **Two‑Batch‑Overlap**、**Speculative Draft** 场景下的并行度与内存复用效率。 |
| **图捕获/重放** | 在 `CudaGraphRunner`、`CPUGraphRunner` 中加入 `forward_context` 包装，确保图捕获期间使用正确的后端实例，防止因后端切换导致的图失效或错误的 kv 索引。 |

### 3. 安全考虑
- **断言**：`get_forward_context()` 在未设置时会 `assert`，可以在生产环境捕获未包装的调用，避免使用未初始化的后端导致隐式崩溃。  
- **全局可变**：`_current` 为全局变量，若未来在同一进程中出现多线程/异步调用，需改为 `contextvars.ContextVar` 防止竞争。当前单进程单线程模型下安全。  
- **后端属性不变**：后端在 `__init__` 时捕获的 pool 引用在模型生命周期内保持不变，未出现资源泄露风险。  
- **异常传播**：`forward_context` 使用 `try/finally` 确保即使内部异常也会恢复原上下文，保持后续请求的上下文一致性。
