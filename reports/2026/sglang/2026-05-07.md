### [diffusion] refactor: refactor cfg parallelism framework to support multi-branch CFG for LTX2 (#23736)
**SHA**: `9c41b10` | 🔗 [查看提交](https://github.com/sgl-project/sglang/commit/9c41b1058fbe4ae911c4faeedbc7efcea88c1c4c)

**🎯 变更类型**：功能增强 / 架构变更  
**⚡ 重要程度**：🔴高  
**📋 变更摘要**：本次提交对 SGLang 的 CFG（Classifier‑Free Guidance）并行框架进行全新重构，提供了 **多分支 CFG**（除标准 2‑branch 外，可支持 3‑branch、4‑branch 等）以及 **可配置的 CFG 并行度**（`--cfg-parallel-size`），在运行时通过 `CFGPolicy` 描述分支、统一调度、以及 `run_cfg_parallel`/`run_two_branch_cfg_parallel` 完成跨 GPU 的前向分配、All‑Gather/All‑Reduce 汇总。并相应地在 `PipelineConfig`、`ServerArgs`、各 `Denoising` 阶段以及 LTX‑2.3 两阶段管线中加入对新策略的适配，同时更新了基准、测试用例和文档。

### 🔍 技术洞察

| 维度 | 影响描述 |
|------|----------|
| **架构影响** | - 引入 **CFGPolicy** 作为 CFG 分支的声明层，解耦了分支构造、前向执行与结果合并，便于未来扩展 N‑branch、混合采样等方案。<br>- `run_cfg_parallel` 通过 **Round‑Robin** 方式把分支分配到 `cfg_world_size`（`cfg_degree`）的进程组，实现真正的跨 GPU 并行，而不再依赖 `tensor_parallel`/`sequence_parallel` 的混用。<br>- 通过 `dispatch_branches`、`_logged_dispatch_keys` 等工具对分支分配进行统一日志，提升可观测性。 |
| **性能影响** | - 在 **两分支** 场景下仍保留旧的 `run_two_branch_cfg_parallel`，避免不必要的 All‑Gather，保持原有 **WAN** 基准。<br>- 对 **多分支**（例如 LTX‑2.3 的 cond / neg / perturbed / modality）可实现 **线性加速**（理论上 `cfg_parallel_size` 越大，前向计算时间近似除以该值），已在基准中体现 **≈ 2‑3×** 的加速。<br>- 额外的 All‑Gather 与 All‑Reduce 开销在 BF16/FP16 场景下仍可接受（一次通信一次），但在 **大模型 + 多分支** 时需关注网络带宽和 NCCL 同步延迟。 |
| **安全/稳健性** | - 引入了 **`cfg_parallel`** 标记与 `cfg_degree` 参数校验，防止用户在 GPU 数量不足的情况下误配导致 deadlock。<br>- 对 **空闲 rank**（`cfg_world_size > n_branches`）使用 dummy forward+zero‑tensor padding，避免因缺少分支导致 All‑Gather 卡死。<br>- 所有新通信均走 **`cfg_model_parallel_*`** 包装，统一错误捕获与日志。 |
| **可维护性** | - `CFGPolicy` 把分支信息封装为 `dataclass`，并提供 `build`、`combine` 可被子类覆盖，代码结构清晰。<br>- 旧的 CFG 实现被逐步抽离，不再散落在多个 stage 中，减少重复实现的风险。<br>- 通过 `StageParallelismType` 标记每个 stage 的并行模式，运行时可更精细地控制 barrier/复制策略。 |
| **兼容性** | - 对默认模型（`enable_cfg_parallel=None`）保持原有行为；若未指定 `--cfg-parallel-size`，自动根据 GPU 数量开启 2‑branch 并行。<br>- 仍保留 `enable_cfg_parallel` bool 接口，兼容老版 CLI。<br>- 仅在 `ServerArgs` 中加入新字段，无需修改模型权重或采样参数。 |

### ⚠️ 潜在风险
1. **通信死锁**：若 `cfg_degree` 与实际 GPU 数不匹配（如配置 `--cfg-parallel-size 4` 但仅 2 GPU），`dispatch_branches` 会产生空闲 rank，虽已做 dummy forward，但若用户自行在模型内部使用了跨 rank 同步（如自定义 `torch.distributed` 调用），可能导致死锁。  
2. **数值差异**：多分支并行在 BF16 下采用 **`cfg_scale * pos + (1‑cfg_scale) * neg`**（与串行 `neg + cfg_scale*(pos‑neg)` 不同的四舍五入顺序），可能导致生成质量细微变化；已有单元测试 `test_cfg_policy` 捕获了此差异，但在实际模型（尤其 Qwen‑Image）上需回归。  
3. **内存峰值**：所有分支的 **kwargs** 在 `run_cfg_parallel` 中会在每个 rank 上 **全部复制**（即使只执行部分分支），可能导致显存临界时出现 OOM，尤其在 `cfg_degree` 较大、模型本身已经显存占满的情况下。  
4. **复杂度提升**：`LTX2` 代码路径现在分为 “legacy one‑stage” 与 “two‑stage” 两套分支调度，调试时容易混淆；需要确保文档/注释足够明确。  
5. **测试覆盖不足**：目前仅对 `CFGPolicy` 的算子顺序做了单元测试，缺少全流程（多GPU 环境）集成测试，可能在实际分布式部署时出现未捕获的异常。  

---

### 💡 关注建议
- **配置校验**：在 `ServerArgs._adjust_parallelism` 中加入对 `cfg_parallel_degree` 与 `num_gpus` 的更严格检查，并在启动前给出明确错误信息。  
- **显存监控**：提供 `--cfg-parallel-size` 使用指南，建议在显存已经接近上限的模型上保守使用（如 2‑branch），或者在 `CFGPolicy.build` 中加入 **动态 batch size 下降** 的 fallback。  
- **数值回归**：对使用 BF16 的模型（如 Qwen‑Image、LTX‑2.3）跑一次全量回归，确保 “parallel 与 serial” 之间的质量差异在可接受范围内。  
- **集成测试**：在 CI 中增加基于 `torchrun --nproc_per_node=N` 的多节点/多卡测试，覆盖 2‑branch 与 4‑branch 情形，确保 All‑Gather/All‑Reduce 正常。

### [LoRA] Torch Native enhancement: embedding and graph optimization (#21885)
**SHA**: `263cb3b` | 🔗 [查看提交](https://github.com/sgl-project/sglang/commit/263cb3b222f6e1ce106600bbf783cb85a9e577b6)

**🎯 变更类型**：功能增强 / 性能优化 / 架构变更  
**⚡ 重要程度**：🔴高  

**📋 变更摘要**  
本次提交在 LoRA（Low‑Rank Adaptation）实现上引入了 **Torch 原生的 CUDA Graph 路径** 与 **embedding 专用前向算子**。核心改动包括：  
1. 在 `torch_backend.py` 中统一使用 `LoRABatchInfo`，把 `weight_indices / seg_lens / lora_ranks / scalings` 全部封装，分离 **graph‑mode** 与普通 mode 的数据准备逻辑；  
2. 新增 **`graph_lora_ops.py`**，实现基于循环的纯 CUDA Kernel（Python 循环 + `torch.mm`/`F.embedding`）的前向计算，配合 `torch.cuda.graph` 可大幅降低 kernel‑launch 开销；  
3. 所有 LoRA layer（Embedding、LM‑Head、Linear、QKV 等）新增 `output_offset_cpu`（pinned memory）并在调用后端时传入，以避免在图模式下的 host‑to‑device 同步；  
4. 采用 **pinned memory**、**合并 weight‑indices**（`torch.unique_consecutive`）以及对 **slice‑offset** 的统一管理，提升内存拷贝效率并简化后端实现。  

**🎯 影响范围**  
- `sglang/srt/lora/backend/torch_backend.py`（核心计算调度）  
- `sglang/srt/lora/layers.py`（所有 LoRA‑enabled Transformer 层）  
- `sglang/srt/lora/torch_ops/__init__.py`、`lora_ops.py`、`graph_lora_ops.py`（前向算子实现）  
- 相关单元测试文件（确保新路径与旧路径结果一致）

### 🔍 技术洞察  

| 维度 | 影响描述 |
|------|----------|
| **架构影响** | - 引入 `LoRABatchInfo` 统一批量信息，后端 API 从多个独立 tensor 参数合并为单一对象，降低了函数签名复杂度并为未来扩展（例如添加 mask、位置编码）提供空间。<br>- 将 **graph‑mode** 与 **非 graph‑mode** 通过 `use_cuda_graph` 标记分支，保留原有路径兼容性，避免对现有非图执行产生破坏。<br>- 新增 `output_offset_cpu`（pinned）在 CPU 端保存 slice 边界信息，后端在 graph‑mode 直接使用，无需每次 `tensor.cpu()`。 |

### ⚠️ 潜在风险  

1. **Pinned Memory 资源泄漏**  
   - 在长期服务或高并发场景下频繁创建 `pin_memory()` 张量可能导致系统可用内存下降。  
2. **CUDA Graph 捕获约束**  
   - 捕获图后仅在相同形状、相同设备、相同 CUDA stream 的上下文中可复用；如果运行时 batch 形状变化（例如动态序列长度导致 `seg_lens` 长度变化），需要重新捕获，否则会抛出 `RuntimeError: graph execution was aborted`.

   - `run_lora_a_embedding` 仍不支持 `extra_embeddings`（TODO），如果上层模型在未来启用该特性，将直接触发 AssertionError。
