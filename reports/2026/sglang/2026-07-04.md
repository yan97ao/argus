### [diffusion] perf: add unified SP shard helpers and zero-copy tail-pad attention (#30107)
**SHA**: `763c6bf` | 🔗 [查看提交](https://github.com/sgl-project/sglang/commit/763c6bf372bdb46e62f2840c39d7cf773121d6ba)

**🎯 变更类型**：功能增强 / 重构 / 性能优化  
**⚡ 重要程度**：🔴 高  

**📋 变更摘要**  
本次提交在 `sgl‑lang` 中引入统一的 **Sequence‑Parallel (SP) shard / pad / gather** 辅助库 `sp_shard_utils.py`，并在核心模型、注意力层、流水线配置等多处代码中统一使用该库完成 **文本‑图像/视频跨模态序列的分片、尾部 Pad 与元数据 (tail‑attn‑meta)** 处理。实现了：

1. **统一的分片计划**（`SpShard`）与 **零拷贝尾部 Pad**，避免在不整除 SP 度时强行 Zero‑Pad 产生无效自注意力。  
2. **`join_seqs / split_seqs`** 在 Joint‑Attention 中把 SP 文本的 Pad 行搬到全局序列尾部，使后端 **Flash‑Attention‑Varlen** 能直接跳过 Pad。  
3. **`tail_attn_meta`** 为每个请求生成 `cu_seqlens_tail` 等元数据，实现 **FA 零拷贝变长** 路径。  
4. 删除旧的手写分片逻辑，所有模型（Flux、Flux‑2、Qwen‑Image、Ernie‑Image、Mova‑Video‑Dit、Mova‑Video‑Dit‑Model‑A等）统一调用新工具。  
5. 为不满足分片条件的序列（如 `time_dim % sp_degree != 0`）改为 **跳过 SP** 并给出一次性警告，防止错误注意力计算。  
6. 新增 **单元测试 `test_sp_shard.py`** 验证分片、Pad、Meta、join/split 等关键路径的纯逻辑正确性。

**🎯 变更类型**：性能优化 / 功能增强  
**⚡ 重要程度**：🔴高  
**📋 变更摘要**：  
- 在 TRT‑LLM MHA 后端新增 **triton 融合 kernel** `update_trtllm_mha_graph_metadata`，一次 launch 完成 cache seq‑len、cu_seqlens、page‑table、SWA‑page‑table 以及 swa out‑cache‑loc 的全部重建，取代原先在 CUDA‑graph replay 时散落的 ~25 条 host‑dispatch aten‑ops。  
- 为此在 `HybridAttnBackend`、`HybridLinearAttnBackend` 与多层 Eagle‑draft、Frozen‑KV‑MTP runner 中新增 `init_forward_metadata_in_graph` 接口，确保在 **capture‑phase** 记录该 kernel，并在 **replay‑phase** 调用。  
- 同时补充了大量单元测试，验证不同 batch、q‑mode、SWA、静态/动态 page‑table 宽度等场景的语义等价性。

### 🔍 技术洞察

| 维度 | 影响说明 |
|------|----------|
| **架构影响** | - 将原本散落在 `TRTLLMHAAttnBackend._apply_cuda_graph_metadata` 中的 **多次 host‑dispatch** 合并为单一 **triton kernel**，在 CUDA‑graph 中形成 **pure‑device** 的元数据重建阶段。<br>- 引入 **两阶段 init**（`init_forward_metadata_out_graph` 负责捕获，`init_forward_metadata_in_graph` 负责在图内部执行），使得 **metadata** 与 **业务计算** 完全解耦，提升图的可复用性。<br>- Hybrid 包装层需显式转发 `init_forward_metadata_in_graph`，否则会导致 page‑table stale，已在代码中实现。 |
| **性能影响** | - 之前每次 replay 需要 **≈25 次 host‑dispatch**（每次 ~0.03‑0.04 ms），在 **4×（2 draft‑decode + target‑verify + draft‑extend）** 的 decode 步骤上累计 **0.7‑1.0 ms** **CPU jitter**，进而在 all‑reduce 前形成自旋等待。<br>- 融合后 **一次 launch** 完成全部更新，CPU 端调度成本降至 **<0.1 ms**（仅 kernel launch），可将单次 decode 端到端延迟削减约 **10‑15%**（实际提升视 batch‑size 与页面宽度而定）。<br>- 对 **SWA** 场景同样受益，防止因 -1 sentinel 处理产生额外分支。 |
| **安全考虑** | - 纯设备计算，无额外外部 I/O，未引入新依赖或网络交互。<br>- 唯一潜在安全点是 **越界访问**：kernel 使用 `tl.cdiv(..., PAGE_BLOCK)` 与 mask 控制，已在代码中通过 mask 防止读取未分配区域。<br>- 仍需保证 `page_size`、`max_num_pages` 与实际 KV‑pool 配置一致，否则会触发非法地址访问。 |
| **可维护性** | - 新增的 Triton 文件 `trtllm_mha_graph_metadata.py` 与后端调用分离，逻辑清晰，易于单独调试。<br>- `init_forward_metadata_in_graph` 的抽象层次使得 **Hybrid** 与 **Speculative** runner 只需转发即可，避免在多层包装中重复实现。<br>- 单元测试覆盖 **7×7** 组合（batch、seqlen_offset、q_mode、SWA、static_width），提供回归保障。 |

### ⚠️ 潜在风险
1. **硬件兼容性**：`triton` kernel 依赖 **SM100**（Ada/Blackwell）指令集；在旧 GPU（如 A100、RTX 30 系列）上会报编译错误或运行时 fallback 未实现，可能导致跑不通。  
2. **数值一致性**：融合后的整数除法 `token // PAGE_SIZE` 与原 aten‑ops 的 `torch.div(..., rounding_mode='floor')` 在负数（-1 sentinel）场景必须保持 **完全相同**，否则 SWA‑sentinel 逻辑会错位。  
3. **批次大小极端**：`bs == 0` 已提前返回，但若 `max_seq_pages` 为 0（异常配置）仍会进入 kernel，可能触发 **launch 参数非法**。  
4. **共享缓冲区竞争**：在 Multi‑GPU 分布式环境中，`swa_out_cache_loc` 仍是 **全局** 缓冲区；若不同 rank 同时写入同一位置（理论上不应），可能出现数据竞争。  
5. **调度错误**：`grid = (bs + 1 + grid_extra,)` 默认用 **单维 grid**，若 `bs` 超过 2^31‑1（不大可能）会溢出；但在极端 **bs≈65535** 时，`BS_BLOCK = triton.next_power_of_2(bs)` 可能导致 **超过 512** 的 blockSize，需确认 Triton 对此的支持。

- **性能基准**：在 CI 中加入 **micro‑benchmark**（如 decode 1000 tokens × batch 8）对比旧旧实现，记录 **latency、CPU jitter、All‑reduce 争用** 三项指标，确保改动带来的提升持续存在。  
- **边界测试**：加一组 **负数 sentinel**、**极小 page‑size (≤32)**、**max_seq_pages > max_num_pages**（非法） 的异常测试，确保 kernel 在 mask 下安全返回。

- **缓存刷新**：如果 CI 采用长期运行的 worker（如 `pytest-watch`），考虑在每次 CI 启动时强制清除 `_official_consistency_gt_outputs_cache`，或加入基于 `SGL_TEST_FILES_CI_DATA_REVISION` 的版本校验。  
- **回滚方案**：保留旧的 `_remote_consistency_gt_candidate_sets` 实现（可通过特性标记或环境变量切换），以便在官方 GT 服务出现异常时快速回退到仅使用 sglang GT 的路径。  

通过上述措施，可在提升 CI 稳定性与可维护性的同时，控制潜在的网络、数据同步和阈值回退风险。
