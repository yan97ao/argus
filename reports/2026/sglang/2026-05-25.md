### [diffusion] Add CFG gating for denoising (#25848)
**SHA**: `121cc09` | 🔗 [查看提交](https://github.com/sgl-project/sglang/commit/121cc094054f2fae39ca8c22e267c2edb63904d6)

**🎯 变更类型**：功能增强  
**⚡ 重要程度**：🟡 中  
**📋 变更摘要**：在扩散推理中新增 **CFG gating** 参数 `SGLANG_DIFFUSION_CFG_GATE_STEP`，实现对未条件化分支残差的复用，以减少后半段的两路前向计算。对应在 `denoising` 阶段完成状态初始化、日志统计以及预测逻辑分支。新增单元测试覆盖复用、模型切换失效、并行模式禁用以及异常输入。  

**🎯 影响范围**：  
- `python/sglang/multimodal_gen/envs.py`（新增环境变量）  
- `python/sglang/multimodal_gen/runtime/pipelines_core/stages/denoising.py`（初始化/日志/预测改造）  
- `python/sglang/multimodal_gen/test/unit/test_cfg_gating.py`（测试）  

**💡 关注建议**  
1. **文档同步**：在 README/Env 说明中补齐 `SGLANG_DIFFUSION_CFG_GATE_STEP` 的意义、取值区间及默认行为，防止用户误设 >1.0。  
2. **分布式安全**：`ctx.extra["cfg_gate_state"]` 仅在 rank‑0 打印，其他进程仍会维护状态；确保在多机多卡恢复 checkpoint 时能正确恢复 `gate_step` 与 `delta`（目前只保存在内存）。  
3. **与现有 CFG 并行兼容**：代码已在 `enable_cfg_parallel` 为 True 时禁用 gating，建议在 `cfg_policy` 构造前就做一次 early‑exit，以免产生不必要的 `cfg_gate_state` 初始化开销。  
4. **性能基准**：新增日志会在 warmup 与非主节点跳过，仍建议在 CI 中添加 benchmark，验证在长序列（>100 步）下的加速比例与生成质量的差异。  
5. **异常处理**：`_init_cfg_gate_state` 对非法 fraction 抛 `ValueError`，但在生产环境中可能导致服务崩溃，考虑改为 `logger.error` 并回退到默认 1.0。  
6. **测试覆盖**：当前单元测试覆盖主要路径，建议再加入 **scheduler‑order>1**（多阶 solver）以及 **different dtype** 场景，确保 `delta` 在混合精度下保持一致。  

整体来看，改动实现了一个可选的计算节省机制，对不使用 CFG‑parallel 的推理可显著降低后半段算力消耗，风险主要在分布式状态恢复和异常配置的容错上。适度补充文档与更严格的容错即可上线。

### [diffusion] Cache fp32 layernorm params (#25847)
**SHA**: `85f9522` | 🔗 [查看提交](https://github.com/sgl-project/sglang/commit/85f9522e364bc543fa29c9b81cb006a476c98144)

**🎯 变更类型**：功能增强  
**⚡ 重要程度**：🟡 中  
**📋 变更摘要**：新增 `FP32LayerNorm`，在推理阶段对 `weight`/`bias` 进行一次性 fp32 转换并缓存，避免每次前向重复 cast；在训练（梯度开启）时仍保持原有路径，保证 autograd 正常。配套单元测试验证缓存复用、失效以及 autograd 行为。  

**🎯 影响范围**：  
- `python/sglang/multimodal_gen/runtime/layers/layernorm.py`（实现层）  
- `python/sglang/multimodal_gen/test/unit/test_fp32_layernorm.py`（新增测试）  

**💡 关注建议**：  
1. **缓存键的完整性**：当前使用 `data_ptr、_version、device、目标device、dtype` 生成 key，已能捕获参数改动和设备迁移，建议在注释中说明 `_version` 仅在 PyTorch >=1.9 可用，以免旧版兼容性误判。  
2. **内存泄漏风险**：缓存持有 `fp32` Tensor 的引用，若模型频繁切换设备或参数维度会产生旧张量未释放的短暂峰值。可在 `__del__` 或 `reset_parameters` 中显式清理 `self.__dict__` 中的缓存键。  
3. **状态字典**：已确保缓存不进入 `state_dict`，但 `torch.save` 时仍会序列化 `__dict__` 中的自定义属性。若用户自行 `torch.save(norm)`，缓存会随模型一起保存，增加 checkpoint 大小。可考虑在 `__getstate__`/`__setstate__` 中剔除这些键。  
4. **测试覆盖**：当前只在 CUDA + bf16 环境跑，若在 CPU 或 FP16 环境使用仍会走相同路径。建议补充 CPU‐only的快速测试，以捕获 `device=None` 情形。  
5. **文档与兼容**：在 `README` 或代码注释中明确该层仅用于 diffusion 推理场景，训练仍推荐使用标准 `nn.LayerNorm`，防止误用导致不必要的显存占用。  

整体来看，此改动通过一次性 fp32 转换显著降低了 diffusion 推理时的算子开销，兼顾了 autograd 正确性，风险点主要在缓存管理和序列化上，按上述建议进行细化即可安全合并。

### Zero `req_pool_indices` padding in cuda-graph populate (#26292)
**SHA**: `3e67398` | 🔗 [查看提交](https://github.com/sgl-project/sglang/commit/3e67398a96b341402eea7845de9a6221b6af5ca7)

**🎯 变更类型**：功能增强  
**⚡ 重要程度**：🟡 中  

**📋 变更摘要**  
本次提交在 CPU 与 CUDA 图执行路径以及若干 Speculative 图 runner 中，新增对 `req_pool_indices` 的 zero‐padding。原有在 batch 大小不等于 `raw_bs` 时仅对 `seq_lens`、`out_cache_loc`（以及 `positions`）做零填充，导致 padding 行仍引用上一次 replay 中残留的 `req_pool` 条目，可能触发错误的注意力读取。现在在同一条件下同步将 `req_pool_indices` 置零，使填充行始终指向保留的 “slot 0”（对应 `req_to_token[0, :]` 全零），保证图 replay 的状态一致性。

**🎯 影响范围**  
- `python/sglang/srt/model_executor/cpu_graph_runner.py`  
- `python/sglang/srt/model_executor/cuda_graph_runner.py`  
- `python/sglang/srt/speculative/eagle_draft_extend_cuda_graph_runner.py`  
- `python/sglang/srt/speculative/frozen_kv_mtp_cuda_graph_runner.py`  

这些模块负责在 CUDA/CPU 图中复制、填充以及重放 `ForwardBatch`，因此改动直接影响所有使用图模式的推理路径，尤其是在批次大小变化或使用 padding 的场景。

**💡 关注建议**  
- **测试覆盖**：加入多 batch‑size、包含大量 padding 的单元/集成测试，验证 `req_pool_indices` 为零时注意力计算不再出现越界或错误输出。  
- **性能影响**：zero‑fill 操作成本极低，可忽略，但建议在高吞吐量场景下观察是否出现额外的同步点。  
- **向后兼容**：若用户自行构造 `ForwardBatch` 并在填充值上有特殊需求，需确认新行为（padding 行统一指向 slot 0）仍符合其逻辑。  
- **文档更新**：在图 runner 使用说明中注明 padding 行的 `req_pool_indices` 将被强制置零，以帮助调试和自定义实现。  

总体来说，此改动提升了图 replay 的健壮性，风险有限，只需确保相应测试通过即可安全部署。
