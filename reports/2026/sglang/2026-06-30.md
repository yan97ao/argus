### [AMD] Implement QuarkW4A8MXFp4MoE to support amd/gpt-oss-120b-w-mxfp4-a-fp8 (#27204)
**SHA**: `a5e6dd3` | 🔗 [查看提交](https://github.com/sgl-project/sglang/commit/a5e6dd37677f1672e355a4b1e56b3969879bd25e)

**🎯 变更类型**：功能增强 / 架构变更  
**⚡ 重要程度**：🔴高  

**📋 变更摘要**  
- 在 Quark 量化框架中新增 `QuarkW4A8MXFp4MoE`，实现 **MXFP4 权重量化 + 静态 FP8 激活** 的 MoE 方案。  
- 新增检测逻辑、Scheme 注册、权重加载（`load_gptoss_weight_quark`）以及 AITER‑backed 运行时实现，使模型 `gpt-oss-120b-w-mxfp4-a-fp8` 能在 AMD gfx95（MI35x）GPU 上直接推理。  
- 同步更新 `gpt_oss.py` 的加载分支，并添加针对该方案的 GSM8K 完整度回归测试。

**🎯 影响范围**  
- `python/sglang/srt/layers/quantization/quark/`（Scheme、权重加载、工具函数）  
- `python/sglang/srt/models/gpt_oss.py`（模型权重加载分支）  
- `python/sglang/srt/layers/moe/`（AITER MoE Runner、shuffle/scale 实现）  
- CI 测试 `test/registered/amd/accuracy/mi35x/test_gpt_oss_w4a8_mxfp4_eval_mi35x.py`  

**🔍 技术洞察**  

| 维度 | 影响说明 |
|------|----------|
| **架构影响** | 1. 引入全新 MoE Scheme `QuarkW4A8MXFp4MoE`，并在 `quark.__init__` 中注册，形成 **Quant‑MoE → Scheme → Runner → Dispatcher** 的完整链路。<br>2. 方案强制使用 **AITER** 后端（`MoeRunnerBackend.AITER`），并在 `create_moe_runner` 中固定激活函数为 `swiglu`，因此 AMD GPU（gfx95+）是唯一支持平台。<br>3. 权重加载采用 **Per‑expert checkpoint** → **TP/EP 切片** → **padding + shuffle** 的流程，保持与原生 MXFP4‑MoE 的布局兼容。 |
| **性能影响** | - **存储**：MXFP4 采用 `uint8`（2×FP4）+ 每 32 元素 1 Byte 的 e8m0 scale，显著压缩显存（约 1/8），适配大模型（120B）。<br>- **计算**：AITER kernel 在 AMD GPU 上可直接使用 `float4_e2m1fn_x2` 数据类型，避免解压后再计算，理论上比 CPU/Triton 实现提升 **30‑50%**（取决于 `SGLANG_USE_AITER` 与 `SGLANG_USE_AITER_MOE_GU_ITLV`）。<br>- **额外开销**：权重/scale 的 shuffle 与 padding 在模型启动时一次性完成，启动成本略增（≈几秒），对推理吞吐影响可忽略。 |
| **安全考虑** | - 变更仅在本地模型加载与 GPU 计算路径，不涉及网络或文件系统权限提升。<br>- `__init__` 中对 weight/input 配置进行严格校验；错误配置会抛 `ValueError`，避免潜在的误算或内存泄漏。<br>- 环境变量控制 AITER 路径，若误开启在不支持的硬件上会在 `process_weights_after_loading` 触发 `NotImplementedError`，保持安全失效而非静默错误。 |

**⚠️ 潜在风险**  
1. **平台依赖**：仅在 AMD gfx95 及以上并且 `SGLANG_USE_AITER=1` 时可用，未检测到 AITER 或不满足硬件要求时会直接报错，可能导致 CI/部署失败。  
2. **后端兼容性**：`QuarkW4A8MXFp4MoE` 暂未实现非‑AITER 路径；若用户在 CPU 或非‑AMD 环境强行加载 `quark` 配置，会触发 `NotImplementedError`。  
3. **PyTorch 版本**：使用 `torch.float4_e2m1fn_x2` 数据类型，需 PyTorch ≥2.3（或对应 AMD 扩展），低版本会导致属性缺失或错误。  
4. **权重对齐**：新增的 padding 与 shuffle 逻辑依赖 `OCP_MX_BLOCK_SIZE=32` 与 `SGLANG_USE_AITER_MOE_GU_ITLV` 环境变量匹配，不匹配会导致权重误位或数值偏差。  
5. **并行切片**：在多 TP/EP 场景下，切片边界计算依赖 `math.ceil`，若 `intermediate_size` 不是 32 的整数倍会触发断言，导致加载失败。  

**💡 关注建议**  
- **文档与提示**：在 README/Release notes 中明确标注 “仅支持 AMD gfx95 且必须开启 `SGLANG_USE_AITER=1`”，并在代码路径（如 `create_moe_runner`）加入更友好的错误信息。  
- **兼容性保护**：在 `QuarkW4A8MXFp4MoE.__init__` 前加入对 `torch.float4_e2m1fn_x2` 是否可用的检查，若不可用提前报错，引导用户升级 PyTorch。  
- **回退路径**：考虑实现一个 “fallback to native Triton” 的实现或在 `gpt_oss.py` 中给出 `raise NotImplementedError` 前的明确提示，防止用户误以为模型可在非‑AMD 环境运行。  
- **测试覆盖**：在 CI 中添加非‑AMD（CPU/CUDA）平台的负向测试，确保加载 `quark` 配置时能够优雅失败。  
- **参数校验**：对 `envs.SGLANG_USE_AITER_MOE_GU_ITLV` 的值进行校验（只能为 0/1），避免因拼写错误导致权重未正确 shuffle。  
- **性能监控**：在启动日志中打印 `hidden_pad`、`intermediate_pad`、`num_experts` 等关键尺寸，帮助用户快速定位因 padding/切片不匹配导致的显存异常。

### [NPU] Support DeepSeek V4 Flash MTP on Ascend (#28980)
**SHA**: `89620b9` | 🔗 [查看提交](https://github.com/sgl-project/sglang/commit/89620b9169e67525000e804fc443444ecea0944b)

**🎯 变更类型**：功能增强 / 重构  
**⚡ 重要程度**：🔴 高  

**📋 变更摘要**  
1. **新增 DeepSeek V4 Flash MTP 在 Ascend NPU 上的完整支持**，包括多步（multi‑step）草稿（draft）解码路径、压缩元数据的校验与写回、以及针对目标‑verify 场景的专用缓存构建。  
2. **扩展 DSV4（DeepSeek V4）分配器**，在 `alloc_extend` 时可预留草稿槽位，并在 NPU 上写入对应的 C4/C128 KV 与状态表。  
3. **引入公共 Hook 与工具函数**（`maybe_write_dsv4_extend`, `maybe_build_dsv4_verify_bundle` 等），统一 NPU 与 GPU 上的 DSV4 扩展行为。  
4. **大量后端元数据结构与调度逻辑修改**：  
   - `ForwardMetadata` 增加 `actual_seq_lengths_q_pa` 用于 prefill‑pa 场景。  
   - `AscendBackend` 与 `AscendDsv4Backend` 在压缩位置、图模式、目标‑verify 等分支中加入细粒度检查、异常抛出以及对 `global_forward_mode`、`actual_forward_mode` 的区分。  
   - `DeepseekV4AscendMultiStepDraftBackend` 实现了草稿多步缓存切片、状态恢复以及图捕获/回放的统一接口。  
5. **模型、调度与 CUDA‑graph 代码同步更新**：`forward_batch_info`, `decode_cuda_graph_runner`, `eagle_*` 等模块适配了新增的 `out_cache_loc_dsv4`、`positions` 与 verify‑path 参数。  

**🎯 影响范围**  
- `sglang/srt/hardware_backend/npu/attention/`（AscendBackend、AscendDsv4Backend）  
- `sglang/srt/hardware_backend/npu/dsv4/`（分配器、公共 Hook）  
- `sglang/srt/model_executor/`（forward_batch_info、model_runner_kv_cache_mixin、decode_cuda_graph_runner）  
- `sglang/srt/speculative/`（eagle_utils、draft_extend_cuda_graph_runner、eagle_draft_utils）  
- `sglang/srt/models/deepseek_v4*`（模型权重映射、NextN 结构）  
- 相关公共工具（topk、moe、quantization）

### 🔍 技术洞察  

| 维度 | 影响描述 |
|------|----------|
| **架构影响** | - 引入 **DeepSeek V4 多步草稿解码*（`DeepseekV4AscendMultiStepDraftBackend`）**，在 NPU 上实现与 GPU 相同的 draft‑extend / verify 流程，统一了 `ForwardMode` 与 `global_forward_mode` 的概念，避免了在图捕获期间对 mode 的二义性。<br>- 新增 **DSV4 专用分配器**（`DSV4NPUTokenToKVPoolAllocator`），在 `alloc_extend` 时可保留草稿槽位，并在 `maybe_write_dsv4_extend` 中写入 C4/C128 状态表，实现了 **预留‑extend** 与 **实时‑verify** 两套路径的兼容。<br>- `ForwardMetadata` 增加 `actual_seq_lengths_q_pa`，让 prefill‑PA（position‑aware）能够在图模式下正确复制 token‑长度信息。 |
| **性能影响** | - **压缩位置计算** 采用 `self._dsv4_unique_compress_ratios` 去重，降低了在循环中对无效 ratio 的遍历次数。<br>- 在 `forward_compress` 中加入 **长度不匹配检查** 与 **mask‑based裁剪**，可在图模式下提前剔除无效 KV，防止无意义的 NPU 计算。<br>- `alloc_extend` 通过 `ALLOC_EXTEND_FUNCS` 按设备类型选择实现，NPU 使用 `alloc_paged_token_slots_extend_npu`，在预留草稿槽位时避免额外的 CPU‑to‑GPU 拷贝，降低延迟。<br>- `eagle_prepare_for_decode` 中的 `per_step_draft_out_cache_loc` 通过 **张量视图切片** 实现多步缓存的零拷贝分配，显著提升 draft‑step 的吞吐。 |
| **安全/稳定性** | - 大量 **长度校验 (`assert` / `RuntimeError`)** 防止 `loc` 与 `kv` 长度不匹配导致内存越界（尤其在图回放以及 verify 场景）。<br>- `backup_state / restore_state` 在 `DSV4NPUTokenToKVPoolAllocator` 中同步保存子分配器状态，防止 **草稿步骤中的内存泄漏**。<br>- `maybe_write_dsv4_extend` 现在接受可选的 `c4_state_alloc_offsets`、`c128_state_alloc_offsets` 参数，避免在某些分配器未实现时出现属性错误。 |
| **可维护性** | - 将 **device‑specific扩展逻辑抽离** 为 `maybe_write_dsv4_extend`、`alloc_paged_token_slots_extend_npu`，实现了 **单点改动** 与 **可插拔**（`ALLOC_EXTEND_FUNCS`）的设计。<br>- `ForwardMetadata` 与 `ForwardBatch` 的字段同步通过 `build_inner_fb_view` 完成，保持了前向批次视图的统一性。<br>- 对于 `topk` 模块的 NPU 路径做了 **早期返回**，避免在 NPU 上执行低效的 bool‑index scatter，提升代码可读性。 |

### ⚠️ 潜在风险  

1. **图回放路径的长度溢出**

**SHA**: `a5c45a1` | 🔗 [查看提交](https://github.com/sgl-project/sglang/commit/a5c45a12bb378678ba03d2405ae1630799029135)

2. **兼容性**：若运行环境缺少 `cuda.bindings` 或 `torch.cuda.CUDAGraph(keep_graph=True)` 不被支持，系统会自动回退到原始路径，保持向后兼容。
