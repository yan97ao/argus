### [5/N] Quantization Refactor: GPTQ schemes and kernel split (#26402)
**SHA**: `e33bbbb` | 🔗 [查看提交](https://github.com/sgl-project/sglang/commit/e33bbbb467e45680f44869801ff2d087e6e354fa)

**🎯 变更类型**：功能增强 / 重构 / 架构变更  
**⚡ 重要程度**：🔴 高  
**📋 变更摘要**：  
- 将 GPTQ 量化相关的代码从单一 `gptq.py` 拆分为 **GPU/Marlin、GPU/普通、NPU(Ascend)** 三套实现，并统一使用 **kernel‑scheme** 设计模式。  
- 新增 `GPTQAscendConfig`、`GPTQMarlinConfig` 等配置类，支持在 Ascend NPU 上直接使用 `npu_weight_quant_batchmatmul` 与 `npu_fused_experts`。  
- 引入 `GPTQLinearKernel / GPTQMarlinLinearKernel / GPTQMarlinMoEKernel / GPTQLinearAscendKernel / GPTQMoEAscendKernel` 等底层 kernel 封装，同时提供 **fallback** (`_unsupported_kernel`) 防止在不支持的平台上崩溃。  
- 更新量化注册表，使 `quantization=gptq` 在 NPU 环境自动映射到 `GPTQAscendConfig`，并在 `auto_round`、`quantization.__init__` 中同步新实现。  
- 完全重写 **scheme** 层 (`gptq_linear.py、gptq_marlin.py、gptq_moe.py`) 以统一权重创建、后处理、前向调用的流程，提升可维护性与可扩展性。

## 🔍 技术洞察

### 1. 架构影响
| 影响维度 | 说明 |
|----------|------|
| **模块化** | 将 GPTQ 代码拆分为 **kernel → scheme → config → registration** 四层，明确职责：<br>• *Kernel* 负责硬件函数调用与权重后处理<br>• *Scheme* 负责权重创建、参数包装、调用 Kernel<br>• *Config* 只保存超参数与兼容检测<br>• *Registration* 将配置映射到 `quantization=` CLI 参数。<br>这显著提升了代码可读性、可测试性与后续硬件适配的扩展性。 |
| **硬件抽象层** | 新增 `hardware_backend/gpu` 与 `hardware_backend/npu` 两个子目录，明确硬件边界，避免在通用层混入平台特有的 API（如 `sgl_kernel`、`torch_npu`）。 |
| **向后兼容** | 通过 `_unsupported_kernel` 提供运行时安全降级；如果平台不支持相应 CUDA/NPU kernel，抛出清晰异常而非在 import 阶段直接导致整个库不可用。 |
| **配置统一** | `GPTQAscendConfig` 继承自 `GPTQConfig`，仅在 **`get_supported_act_dtypes`** 与 **`get_quant_method`** 上做平台特化，保持了已有配置文件 (`quantize_config.json`) 的兼容性。 |
| **方案抽象** | `GPTQLinearSchemeBase` 与 `GPTQMoESchemeBase` 抽象出所有量化方案必须实现的接口，使得后续加入（如 **AMD GPU**、**Intel GPU**）只需实现对应 Kernel 与 Scheme，即可无缝集成。 |
| **依赖管理** | 新文件显式 `try/except` 导入 `sgl_kernel`、`torch_npu`，并在 `register_fake_if_exists` 中提供 Torch‑Compile 的占位实现，降低对非目标平台的硬依赖。 |

### 2. 性能影响
| 场景 | 预期提升 | 说明 |
|------|----------|------|
| **GPU‑Marlin** | **~20‑30%** 速度提升（取决于模型大小） | Marlin 通过 **2‑step压缩 + 行列分块** 进行高效 INT4/INT8 GEMM，原实现已在 `apply_gptq_marlin_linear` 中使用。新 Scheme 只负责调用而不再重复布局转换，降低 CPU‑GPU 同步开销。 |
| **GPU‑普通 (shuffle)** | 稍微降低 **CPU‑to‑GPU** 预处理开销 | `GPTQLinearKernel` 在首次 forward 时一次性完成 `gptq_shuffle`，避免每次 forward 重复排序。 |
| **Ascend NPU** | **1.5‑2x** 推理加速（相较于原 CPU‑fallback） | 使用 `torch_npu.npu_weight_quant_batchmatmul` 与 `npu_fused_experts`，一次性完成量化乘法与 MoE fused 计算，显著减少显存复制与算子调度。 |
| **内存占用** | **显著降低**（特别是 4‑bit） | 4‑bit 权重在 NPU 上会被 `int4pack` 成 `int32` 再存储，GPU‑Marlin 采用 **packed column** 形式，同样降低了显存消耗。 |
| **编译兼容性** | **torch.compile** 能直接跟踪 `gptq_*` 自定义 op；若缺失，会使用 `register_fake_if_exists` 提供的占位实现，保证编译不报错。（性能仍取决于实际 kernel 是否可用） |

### 3. 安全考虑
| 风险点 | 说明 | 缓解措施 |
|--------|------|----------|
| **动态加载外部共享库** (`sgl_kernel`, `torch_npu`) | 若库文件被篡改或版本不匹配，可能导致崩溃或执行错误的算子。 | - 使用 `try/except` 包装导入，若加载失败立即 fallback 为 `_unsupported_kernel` 并抛出明确异常。<br>- 在 CI 中加入 **二进制兼容性检查**（`ldd`、`nm`）确保库版本。 |
| **低位数整数运算** (`int4`、`int8`) | 位操作错误会导致数值偏移、溢出或负数误解

### [diffusion][jit_kernel] perf: varlen FA fast path for USPAttention masked branch (#26318)
**SHA**: `d616b8e` | 🔗 [查看提交](https://github.com/sgl-project/sglang/commit/d616b8edad9aa2f8adfe7438b600cf46e9739a01)

**🎯 变更类型**：功能增强 / 性能优化 / 重构  
**⚡ 重要程度**：🔴高  
**📋 变更摘要**：  
- 在 USPAttention 的 mask 分支中新增基于 Triton 的变长（varlen）FlashAttention 快速路径，实现对有效 token 的打包、使用 FlashAttention 进行注意力计算、再散布回稠密张量。  
- 提供配套的 Triton 打包/散布 kernel、逆索引构造工具以及 `build_varlen_mask_meta` 接口。  
- 新增完整单元测试，覆盖数值正确性、端到端等价性以及不同形状/数据类型下的行为。  

**🎯 影响范围**：  
- `sglang/jit_kernel/diffusion/triton/varlen_pack_pad.py`（核心 Triton kernel 实现）  
- `sglang/jit_kernel/tests/diffusion/*`（单元/集成测试）  
- `sglang/multimodal_gen/runtime/layers/attention/__init__.py`、`layer.py`（USPAttention 调用路径）  
- `sglang/multimodal_gen/runtime/models/dits/qwen_image.py`（QwenImageTransformer 使用新接口）  

**🔍 技术洞察**  

- **架构影响**  
  - 在注意力层中引入 **varlen FA fast path**，实现了 “mask‑branch → pack → FlashAttention（varlen）→ scatter”。  
  - 通过 `build_varlen_mask_meta` 将 mask 元信息（`indices`, `inv_indices`, `cu_seqlens`, `max_seqlen`）前置，避免在每个 block 中重复构造，提高了模块化和复用度。  
  - 新增的 Triton kernel 只在 `torch.float16` / `torch.bfloat16`、CUDA 环境下启用，保持原有 SDPA 后备路径不受影响。  

- **性能影响**  
  - **显著提升**：在大批量、长序列（尤其是图文混合）场景下，原本的 SDPA 必须对全部 token（包括大量 masked token）进行计算，导致不必要的 FLOPs。新路径仅对有效 token 进行 FlashAttention，计算量与有效 token 数成线性关系。  
  - **内存占用**：打包后临时张量大小为 `[valid, H, D]`，相较于 `[B, S, H, D]` 可省去大量稀疏区域的存储；Scatter 时再写回稠密张量，整体峰值内存基本持平或略降低。  
  - **启动开销**：首次构造 `indices` / `inv_indices` 需要一次 `torch.nonzero` 与 `scatter`，成本与序列长度成 O(B·S)；但在长序列模型推理中此开销相对计算提升可忽略。  
  - **条件开关**：通过环境变量 `SGLANG_VARLEN_FA` 可关闭该路径，确保在不适用的硬件或调试场景下回退至安全的 SDPA 实现。  

- **安全考虑**  
  - 新增的 Triton kernel 只读取/写入在合法范围内的指针，不涉及跨设备或越界访问；但 Triton 编译器在不同显卡架构上可能产生隐藏的内存对齐问题，需在 CI（CUDA 大模型）上持续验证。  
  - `build_varlen_mask_meta` 对输入 mask 做了维度与 dtype 检查，防止因错误 mask 形状导致的索引越界。  
  - 环境变量关闭机制防止在不支持的显卡（如不支持 Triton 的老旧 GPU）上触发未捕获异常。  

**⚠️ 潜在风险**  

| 风险点 | 说明 | 缓解措施 |
|--------|------|----------|
| **Triton 兼容性** | 不同 CUDA 版本或显卡架构（如低于 compute capability 7.0）可能不支持当前 kernel 配置。 | CI 中已加入多套 GPU（A100、V100、RTX3090）测试；建议在发行说明中标明最低支持 Compute Capability。 |
| **全零 mask** | 当所有 token 被 mask 时 `indices` 为空，FlashAttention varlen 会报错。当前实现会回退到 SDPA，但若环境变量强制开启且调用路径未检查，可能触发运行时异常。 | 已在 `forward` 中加入 `if indices.shape[0] > 0` 判定并回退；保持此逻辑不变。 |
| **float16 / bf16 精度差异** | 打包/散布是纯拷贝，理论上无误差；但 FlashAttention 本身在 bf16 上可能产生数值略差，导致与 SDPA 的对比出现微小误差。 | 在测试中使用 `rtol/atol` 放宽（bfloat16 采用 1e‑2 级），并在生产环境记录误差分布。 |

### feat: convert mm_hashes to str in encode_server for Mooncake key compat (#26487)
**SHA**: `8ff66b7` | 🔗 [查看提交](https://github.com/sgl-project/sglang/commit/8ff66b707f3b7aafc9bd81041b508d97b1388727)

**🎯 变更类型**：功能增强  

**⚡ 重要程度**：🔴高  

**📋 变更摘要**：  
- 将 `encode_server` 中对多模态特征哈希（mm_hashes）的处理改为先保持整数形式计算，然后在与全局缓存交互前统一转换为字符串，以兼容 Mooncake L2 缓存只能使用字符串键的限制。  
- 相应代码在缓存查询、命中处理、以及插入新缓存时均使用 `str_mm_hashes`，其余业务逻辑仍使用原始整数哈希。  

**🎯 影响范围**：  
- `python/sglang/srt/disaggregation/encode_server.py`（主逻辑）  
- `mm_global_cache` 接口的键类型（从 `int` → `str`）  
- 可能间接影响使用 `mm_hashes` 进行日志、调试或统计的其它模块（如果有直接引用整数列表的假设）。  

**🔍 技术洞察**  

- **架构影响**：  
  - **缓存层兼容性**：Mooncake 的 L2 缓存仅接受字符串键，此变更消除了原来 `int` 键导致的缓存失效或异常，从而提升跨平台/跨服务的一致性。  
  - **类型边界**：在 `encode_server` 内部仍保留整数哈希用于业务计算，只在进入缓存层前做一次显式转换，保持了业务层与缓存层的解耦，符合 “Adapter” 模式的轻量实现。  

- **性能影响**：  
  - **额外字符串转换**：对每个哈希调用 `str()`，在每轮编码任务中产生 `O(N)` 次转换（`N` 为待缓存的 slice 数），开销极小（几微秒级），相较于特征抽取、Transformer 前向等算子可忽略不计。  
  - **缓存键查找**：字符串键在哈希表中的碰撞概率与整数相当，查询复杂度保持 `O(1)`，但若底层缓存实现对整数键做了特殊优化（如直接使用原始数值做索引），则可能出现轻度性能回退。建议在 ``mm_global_cache`` 实现层确认字符串键是否仍利用高效哈希函数。  

- **安全考虑**：  
  - 仅改变键的表示形式，无新增外部输入或可注入向量，安全风险微乎其微。  
  - 确保字符串化后不出现意外前导零或负号导致的键冲突（目前 `hash` 为非负整数，安全性可接受）。  

**⚠️ 潜在风险**  

1. **类型泄漏**：若其他模块（如监控、日志、或自定义缓存插件）直接引用 `mm_hashes` 并假设为整数，可能在运行时出现类型不匹配错误。  
2. **缓存键一致性**：历史缓存中仍可能残留整数键，若未统一迁移，查询时会出现 “未命中” 并导致不必要的重复计算。

## 🔍 技术洞察

### 架构影响
1. **Metadata 扁平化 → 列表化**  
   - 原来 `ContextParallelMetadata` 只保存单值（`kv_len_prev`、`actual_seq_q_prev` 等），假设 `batch_size=1`。  
   - 现在改为 **列表 + Tensor**，每个序列独立记录块大小、前缀长度、累计序列长度等。  
   - 这让 CP 的 **ZigZag** 拆分在每个请求上独立进行，`cp_reverse_index`、`split_list` 与 `bs` 强关联，兼容多请求并行。

2. **FlashAttention 与 DSA 双路径统一**  
   - `cu_seqlens_q_prev_tensor` / `cu_seqlens_q_next_tensor` 为 FlashAttention **可变长** API 必要输入。  
   - 当 DSA 开启时，这两个张量仍然提供（只含累计长度），但实际的前缀偏移在 `kv_len_prev_list/kv_len_next_list` 中体现，保持两条路径功能一致。

4. **调度层改动**

| **吞吐** | ✅ 支持 batch>1 CP，理论上在相同 GPU 上可并行处理更多前缀填充请求，提升总体吞吐。 | ❓ 拆分块后需要 **AllGather** 与 **Reorder** 两次（prev/next），对大 batch 可能增加通信开销。 | 大规模实验需要对比 `batch_size=1` 与 `batch_size>1` 在相同 cp‑size 时的 latency/throughput。 |
| **显存** | ✅ 只在每个 rank 上分配 `max_rank_len`（均衡后）而非整批最大序列长度，稍微降低峰值显存。 | ⚠️ 额外的 per‑sequence Tensor（`kv_len_*_list`、`cu_seqlens_q_*`）在每个 rank 上占用额外显存。 | 对于极大 batch（如 64+）可能出现显存碎片，需要监控。 |
| **计算负载** | ✅ 通过 `cp_segment_num = 2*cp_size` 实现更均匀的算子调度，防止 “首 rank 计算少、尾 rank 计算多” 的不平衡。 | ❓ 每条序列的前后块在同一 rank 上仍需两次 Attention 调用（`result_prev` / `result_next`），相当于 **两倍** 计算，对比单块处理略有额外开销。 | 实际加速取决于 GPU 利用率提升是否抵消两次调用的额外时间。 |

### 安全考虑
- **无新增网络/系统调用**，仅在内部 Tensor 操作和通信层面更改。  
- 只影响 **上下文并行** 的内部调度，不涉及外部输入校验或权限，安全风险极低。  
- 唯一需要关注的是 **边界检查**：`extend_seqs_len`、`seqs_len` 与 `cp_size` 的不匹配可能导致 `assert` 触发或产生错误的 `cp_reverse_index`，从而出现数据错位。建议在公共 API 中加入更严格的参数校验。
