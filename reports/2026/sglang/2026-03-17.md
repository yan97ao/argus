### feat: support human-readable suffixes (25.6k, 1M, 1Mi) for token CLI (#20577)
**SHA**: `515b3a3` | 🔗 [查看提交](https://github.com/sgl-project/sglang/commit/515b3a323d65b73f578cfac042d91d3334fb769f)

**🎯 变更类型**：功能增强  

**⚡ 重要程度**：🔴高  

**📋 变更摘要**  
- 新增 `human_readable_int` 工具函数，能够解析带有 SI（k、M、G、T）和 IEC（Ki、Mi、Gi、Ti）后缀的整数表达式，如 `25.6k`、`1Mi`。  
- 在 `server_args.py` 中，将 `--context-length`、`--max-total-tokens`、`--max-prefill-tokens` 等与 token 数量相关的 CLI 参数的类型由原生 `int` 替换为 `human_readable_int`，并在帮助信息中自动拼接函数的 docstring，以提示用户使用方式。  

**🎯 影响范围**  
- `python/sglang/srt/utils/common.py`：新增 `human_readable_int` 实现。  
- `python/sglang/srt/server_args.py`：5 处 CLI 参数的类型改为 `human_readable_int`，帮助信息同步更新。  
- 依赖这些 CLI 参数的启动脚本、CI/测试脚本以及第三方工具（如果直接调用 `argparse` 解析）将受影响。  

**🔍 技术洞察**  

- **架构影响**  
  - 引入的函数位于公共工具模块，属于 **横向关注点**（交叉模块）而非垂直业务逻辑，符合“工具类”抽象，暂无破坏性架构改动。  
  - 只在命令行解析层面做了类型替换，不影响后端调度、模型加载等核心路径。  

- **性能影响**  
  - `human_readable_int` 只在程序启动时解析命令行参数，调用次数极少，额外的正则匹配与 `Decimal` 乘法产生的性能开销可以忽略不计。  
  - 对运行时的 token 处理、调度等关键路径没有任何影响。  

- **安全考虑**  
  - 参数解析阶段已经加入严格的异常处理：  
    - 对 IEC 后缀禁止小数，提供明确错误信息。  
    - 对非法格式统一抛出 `argparse.ArgumentTypeError`。  
  - 由于仅限于数值解析，未引入代码执行或注入风险。  
  - 使用标准库 `re`、`Decimal`，不依赖外部库，安全风险极低。  

**⚠️ 潜在风险**  

1. **兼容性回归**  
   - 现有自动化脚本或 CI 可能直接传入整数 **字符串**（例如 `"1024"`），仍能正常解析，但若传入整数 **对象**（如 `1024`）给 `argparse.parse_args`（极少见），会触发 `TypeError`，因为 `human_readable_int` 期待 `str`。  
   - 某些第三方工具可能依赖 `type=int` 的默认行为（如 `argparse` 自动提供 `choices` 检查），改为自定义函数后，这类检查需要重新评估。  

2. **用户使用误解**  
   - 后缀区分大小写、SI 允许小数、IEC 不允许小数，若文档或提示不够醒目，用户可能因 `25.6Mi` 报错而产生困惑。  

3. **异常信息传播**  
   - `argparse.ArgumentTypeError` 会导致 CLI 直接退出并打印错误信息；若有外部进程捕获异常并做自定义处理，可能需要适配新的异常来源。  

**💡 关注建议**  

- **文档与帮助**  
  - 在项目的 README / CLI 使用文档中专门增列 “人类可读整数（Human‑readable integers）” 示例与限制，尤其强调 IEC 不支持小数。  
  - 考虑在 `--help` 输出中使用 **颜色或加粗**（如果已有 `rich` 支持），提升提示可见度。  

- **向后兼容**  
  - 在 `human_readable_int` 开头加入类型检测，若输入已经是 `int`，直接返回，可避免极端的脚本兼容问题：  
    ```python
    if isinstance(value, int):
        return value
    ```  

- **测试覆盖**  
  - 增加单元测试，覆盖所有合法和非法后缀组合（包括大小写错误、带小数的 IEC、空格、前缀/后缀混淆等），确保异常信息与预期一致。  
  - 在 CI 中加入对旧版脚本的回归测试，验证不使用后缀的纯整数仍能顺利解析。  

- **错误信息本地化**  
  - 现有错误信息为英文，若项目面向多语言社区，可考虑在 `human_readable_int` 中使用统一的 i18n 接口，以保持错误提示风格一致。  

- **监控与日志**  
  - 如项目在生产环境中通过脚本自动启动服务，建议在启动日志中记录解析后的实际数值（如 `context-length=25600`），帮助运维快速确认后缀是否被正确解释。  

---

通过上述改动，用户在指定大规模 token 参数时将获得更直观的表达方式，提升可用性；同时保持了对现有代码路径的低侵入性。只要做好文档、测试与兼容性检查，即可安全上线。

### Enable embedding lookup/lora_a logic for chunked backend (#17692)
**SHA**: `70a6fb5` | 🔗 [查看提交](https://github.com/sgl-project/sglang/commit/70a6fb53af1ec80dd5cc57d485dbbb664db0ea19)

**🎯 变更类型**：功能增强 / 重构  
**⚡ 重要程度**：🔴 高  

**📋 变更摘要**  
本次提交为 **chunked LoRA 后端** 引入了完整的 **embedding‑lookup 与 lm_head LoRA A** 支持，并在 `LogitsProcessor` 中实现了多 Pass（Chunk）日志概率计算的批信息切换。核心改动包括  
1. 新增 `LoRABackendLmHeadMixing` Mixin，使所有 LoRA 后端统一管理 *lm_head* 的 batch‑info（完整与分块两套）。  
2. `ChunkedSgmvLoRABackend` 与 `TritonBackend` 实现 `run_lora_a_embedding` 并使用全新 **chunked_embedding_lora_a_forward** Triton kernel。  
3. `LogitsProcessor.process_input_logprobs_by_chunk` 在每个 chunk 前后调用 `set_lm_head_pass` / `reset_lm_head_pass`，实现 lm_head LoRA 的分块切换。  
4. `LoraLayer.apply_lora` 通过 `_get_lm_head_batch_info` 读取对应 batch‑info 并在 `run_lora_a_sgemm` / `run_lora_b_sgemm` 中传递。  
5. 进阶的 batch‑info 构造函数（`_prepare_lm_head_batch_info`、`_build_lm_head_batch_info`）以及工具函数 `merge_and_chunk_segments`、`build_lm_head_pass_segments`、`get_lm_head_pruned_lens`。  
6. 新增 Triton kernel `chunked_embedding_lora_a_forward` 与对应的导出入口。  
7. 相关单元测试与引用实现同步更新，覆盖 embedding‑shrink、Chunked‑SGMV 与 lm_head‑pruning 一致性。

**🎯 影响范围**  
- `python/sglang/srt/lora/*`（backend、layers、utils、triton_ops）  
- `python/sglang/srt/layers/logits_processor.py`  
- `python/sglang/srt/server_args.py`（去除对 `csgmv` 不支持 embed / lm_head 的警告）  
- 测试目录 `test/registered/lora/*` 与 `test/lora_utils.py`  

**🔍 技术洞察**  

| 维度 | 影响说明 |
|------|----------|
| **架构影响** | 1. 引入 **Mixin**（`LoRABackendLmHeadMixing`），所有 LoRA 后端（Chunked、Triton）统一拥有 `lm_head_batch_info`、`lm_head_pass_batch_infos` 与 `_lm_head_pass_idx`。<br>2. `BaseLoRABackend` 现在在 `__init__` 中调用 `init_lm_head_config()`，确保即使使用旧的后端也能兼容新属性。<br>3. `LogitsProcessor` 通过 `set_lm_head_pass(i)` 通知后端当前正在处理第 i 块 logprobs，从而在 `LoraLayer.apply_lora` 中选取对应的 batch‑info。<br>4. 通过 `run_lora_a_embedding` 为 **Embedding LoRA A** 提供了独立的前向路径，解决之前只支持 `sgmv` 的限制。 |
| **性能影响** | - **正向**：Chunked kernel 只在 **非零 rank**、**有效 token chunk** 上执行，避免了对全部词表的遍历，显著降低 embedding LoRA A 的算子开销（尤其在大 vocab、低 rank 场景）。<br>- **正向**：`merge_and_chunk_segments` 将同适配器的 token 合并后再切分，减少 kernel 调度次数。<br>- **负向**：每一次 `process_input_logprobs_by_chunk` 都会产生一次 `set_lm_head_pass`/`reset_lm_head_pass` 调用，若用户开启极小的 `logits_processor_chunk_size` 可能导致大量 kernel 启动开销。<br>- **CUDA‑Graph**：明确在 `LoraLayer._get_lm_head_batch_info` 中阻止在 CUDA‑Graph 场景使用 pruned batch‑info，防止图‑构建不一致。 |
| **安全考虑** | 变更均为内部算子与内部状态管理，没有外部输入直接影响路径，未引入新的安全风险。唯一需注意的是 **环境变量** (`SGLANG_ENABLE_LOGITS_PROCESSER_CHUNK`、`SGLANG_LOGITS_PROCESSER_CHUNK_SIZE`) 的错误配置可能导致 batch‑info 不匹配，从而抛出运行时异常，属于可控的 **可用性** 风险。 |
| **可维护性** | - 新增的 **Mixin** 与工具函数把 batch‑info 的生成逻辑抽离，代码复用度提升。<br>- `LoraLayer.apply_lora` 中增加的校验（`expected_tokens` 与 `use_cuda_graph`）帮助快速定位 batch‑info 不匹配的错误。<br>- 通过 `test_lora_hf_sgl_logprob_diff` 中的 **chunked** 场景测试，覆盖了多 Pass 的完整路径，提升回归安全性。<br>- 代码注释已同步更新，说明了两侧 (`_get_pruned_states` 与 `get_lm_head_pruned_lens`) 必须保持一致的约束。 |

**⚠️ 潜在风险**  
1. **Batch‑info 不匹配**：如果用户自行修改 `LogitsProcessor` 的 chunk 大小或自行构造 `ForwardBatch`，`lm_head_batch_info.expected_tokens` 与实际输入 token 数不符会触发 `RuntimeError`（已在 `_get_lm_head_batch_info` 中检查），但这会导致服务直接崩溃。  
2. **Triton Kernel 边界**：`chunked_embedding_lora_a_forward` 在 `rank_val == 0` 或 `chunk_idx >= num_segments` 时直接 return，若 `weight_indices` 与 `lora_ranks` 对齐错误，可能出现 *silent zero* (返回全 0 而非报错)。需要在上层确认 `weight_indices` 与 `lora_ranks` 正确对应。  
3. **内存泄漏/溢出**：`_build_lm_head_batch_info` 使用 `dataclasses.replace` 复制 `LoRABatchInfo`，若后续代码再对原对象进行修改可能产生意外的 **共享张量**。当前实现看似安全，但后续若在其他地方不经意修改 `batch_info` 的张量，会影响所有引用。  
4. **CUDA‑Graph 限制**：在开启 CUDA‑Graph 场景下，`lm_head` LoRA **不再支持** pruned‑batch，若用户误认为两者兼容会产生功能缺失（logits 仍会被 chunk 但 lm_head 不被 pruning）。  
5. **后向兼容**：`server_args.check_lora_server_args` 移除对 `csgmv` 不支持 embed/lm_head 的警告，实际 **csgmv** 仍未实现 embedding/lm_head（字段已在 `ChunkedSgmvLoRABackend` 中实现），若用户仍使用 `csgmv` 并期望 embed/lm_head 工作会得到 **silent zero**（因为 `load_lora_weight_tensor` 对缺失的 buffer 进行了 zero‑fill）。

### fix: adding teacache.params back to sampling params as intended (#20665)

**🎯 变更类型**：Bug修复  
**⚡ 重要程度**：🔴高  

**📋 变更摘要**  
1. 在 `SamplingParams` 中恢复 `teacache_params` 字段，使其成为采样配置的正式成员（默认 `None`），以便模型子类能够注入 `TeaCacheParams` 或 `WanTeaCacheParams`。  
2. 为 Wan2.2 系列的采样参数添加注释，说明当前 `teacache_params` 仍未校准，开启 `enable_teacache` 将变为无实际效果的 no‑op。  
3. 在测试套件中新增对 Wan2.2 T2V A14B 场景的 “TeaCache smoke test”，验证开启 `enable_teacache` 不会导致崩溃（但不做性能检查）。  

**🎯 影响范围**  
- `python/sglang/multimodal_gen/configs/sample/sampling_params.py`（采样参数基类）  
- `python/sglang/multimodal_gen/configs/sample/wan.py`（Wan2.2 采样参数实现）  
- `python/sglang/multimodal_gen/test/server/testcase_configs.py`（新增测试）  

**🔍 技术洞察**  

- **架构影响**  
  - `SamplingParams` 由数据类扩展为包含 `teacache_params`，这是一条向后兼容的结构性改动。  
  - 任何通过 **`**kwargs`** 或 `dict` → `SamplingParams` 的动态构造路径需同步接受 `teacache_params`（默认 `None`），否则可能触发 `TypeError`。

### fix: support PP2+CP8+TP8 (PP with context parallelism) (#19548)
**SHA**: `289cbcf` | 🔗 [查看提交](https://github.com/sgl-project/sglang/commit/289cbcf482f3704bc48975e8627646da566cc423)

**🎯 变更类型**：Bug 修复 / 功能增强  
**⚡ 重要程度**：🟡 中  
**📋 变更摘要**：本次提交在 `Scheduler` 的 PP（Pipeline Parallel）混入实现中加入了对 **Context Parallel（CP）** 的支持，使得在 **PP = 2、TP = 8、CP = 8** 的组合下能够正确发送/接收跨层数据并进行广播；同时在模型参数解析 (`server_args.py`) 中为 CP 引入默认大小，并放宽了对 PP 与 CP 同时启用的限制。  

**🎯 影响范围**  
- `python/sglang/srt/managers/scheduler_pp_mixin.py`：新增 `attn_cp_rank` 条件、在接收阶段加入 `broadcast_pyobj`，确保只有 CP 主节点负责点对点通信，其余 CP 通过广播同步。  
- `python/sglang/srt/server_args.py`：在模型特定调整中把 `attn_cp_size` 设为 `tp_size`，并删除了 “PP 与 CP 不能共存” 的断言，使得上下文并行可以在多机环境下使用（仍要求 TP = 8）。  

**💡 关注建议**  
1. **并行配置校验**：`Scheduler` 现在依赖 `attn_cp_rank` 与 `attn_cp_size`，建议在启动脚本或配置检查阶段显式校验 `tp_size % (dp_size * attn_cp_size) == 0`，防止非法组合导致运行时错误。  
2. **性能与正确性**：`broadcast_pyobj` 会在每个 CP 组内部进行一次同步，若数据量较大可能成为瓶颈。可考虑在后续实现中使用更高效的张量广播或仅在必要时广播。  
3. **测试覆盖**：新增的分支（`attn_tp_rank == 0 and attn_cp_rank == 0`）需要在单机、双机以及多机环境下覆盖，确保 PP、CP、TP 三维并行的所有路径都能正确走通。  
4. **文档与示例**：更新部署文档，明确在 `tp_size = 8` 前提下 `attn_cp_size` 必须等于 `tp_size`，并提供 PP2+CP8+TP8 的启动示例，帮助用户快速验证配置。  

总体来说，此次改动为 SGLang 引入了更灵活的并行组合，提升了大模型部署的可扩展性，需关注配置一致性和广播开销的潜在影响。
