### [diffusion] feat: disaggregated diffusion (#21701)
**SHA**: `9da998a` | 🔗 [查看提交](https://github.com/sgl-project/sglang/commit/9da998a882c810cad5bb739e691a457fa61eb1f1)

**🎯 变更类型**  
功能增强 / 架构变更 / 性能优化 / 重构 / 安全修复  

**⚡ 重要程度**  
🔴高  

**📋 变更摘要**  
本次提交在 **sglang** 项目中引入了 *Disaggregated Diffusion*（扩散模型的分离执行）功能，实现了将完整的 Text‑to‑Video/Image 流水线拆分为 **Encoder → Denoiser → Decoder** 三个独立角色，各自运行在独立的 GPU（或机器）上。核心改动包括：

1. **文档 & CLI** – 新增 `docs/diffusion/disaggregation.md` 以及大量 `--disagg-*-*` 参数，提供统一的启动方式。  
2. **调度层** – `SchedulerDisaggMixin` 为 Scheduler 注入了分布式调度、请求路由、超时处理、RDMA 数据搬运等逻辑。  
3. **角色定义** – `RoleType` 枚举以及 `filter_modules_for_role`、`role_affinity` 在各 `PipelineStage` 中决定模块加载/执行范围。  
4. **中心调度器** – `DiffusionServer`（orchestrator）负责前端 HTTP → ZMQ → 各角色的工作分配、容量感知调度（RoundRobin / MaxFreeSlotsFirst）以及错误/超时统一处理。  
5. **传输层** – 完整的 **Transfer Engine**（基于 Mooncake RDMA）、**Buddy Allocator**、**TransferTensorBuffer**、**协议/编解码**（`protocol.py`, `codec.py`）实现了跨进程/跨机器的零拷贝张量传输。  
6. **度量与监控** – `DisaggMetrics`、`DisaggStats`、FastAPI `/stats` 接口提供运行时吞吐、延迟、队列深度等可观测数据。  
7. **调度/管线改动** – Scheduler、GPUWorker、Pipeline、Stage 基类加入对 **disagg_role** 的感知，支持多实例并行、跨 Rank 广播（NCCL）以及批处理调度。  
8. **启动入口** – `dispatch_launch` 根据 `--disagg-role` 选择普通单机 launch、pool‑mode launch、服务器 head、单角色实例等四条路径。  
9. **测试** – `test_disagg_server.py` 添加端到端 CI，覆盖单 Rank 与 SP‑2（多 Rank）两种分布式场景。  

**🎯 影响范围**  
- **核心运行时**：`scheduler.py`, `gpu_worker.py`, `launch_server.py`, `server_args.py`, `pipeline_*`、`stage_*`。  
- **新模块**：`disaggregation/`（包括 `orchestrator`, `dispatch_policy`, `request_state`, `roles`, `scheduler_mixin`, `transport/*`）。  
- **CLI / 文档**：`sglang serve` 命令行、FastAPI HTTP 接口、监控端点。  
- **依赖**：新增对 `mooncake-transfer-engine`（RDMA）库的运行时依赖；新增对 ZMQ bind 重试逻辑的参数 `same_port`。  
- **测试套件**：CI 中增加 `test_disagg_server.py`（需要至少 2 块 GPU）。

### fix: normalize tool message content for GLM5.1 chat template (#22595)
**SHA**: `fbd6dc3` | 🔗 [查看提交](https://github.com/sgl-project/sglang/commit/fbd6dc35657b8fd86a1b4b6996ea8c95a7e240d4)

**🎯 变更类型**：Bug修复  
**⚡ 重要程度**：🔴高  
**📋 变更摘要**：新增 `normalize_tool_content` 用于将 OpenAI 客户端在 tool 消息中以数组形式返回的纯文本片段展平为单一字符串，防止 GLM5.1 chat 模板因收到列表而报错。随后在 `serving_chat` 流程中对生成的消息内容调用该函数并加入相应单元测试，确保兼容性。  

**🎯 影响范围**：  
- `python/sglang/srt/entrypoints/openai/serving_chat.py`（消息处理核心入口）  
- `test/registered/openai_server/basic/test_serving_chat.py`（新增单元测试）  
- 依赖 OpenAI “tool” 消息格式的所有上层 chat 模板及推理服务  

**🔍 技术洞察**  
- **架构影响**：  
  - 仅在 `OpenAIServingChat` 的消息后处理阶段插入一次纯函数调用，保持原有模块化结构不变。  
  - 通过函数抽离，增强了代码可复用性和可测试性，可在其他 entrypoint（如 completions）中复用。  
- **性能影响**：  
  - 该函数执行 O(N) 的列表遍历，N 为 tool 内容片段数，通常极小（几条），对整体请求延迟影响可以忽略不计。  
  - 仅在 tool 消息出现时触发，整体吞吐量基本不受影响。  
- **安全考虑**：  
  - 不涉及任何外部输入的解析或执行，仅做结构检查与字符串拼接。  
  - 防止在模板渲染阶段因异常列表导致的未捕获异常，从而提升服务稳健性。  

**⚠️ 潜在风险**：  
1. **误判非文本列表**：若未来出现混合了非 `text` 类型且仍需展平的自定义 payload，当前实现会保留原列表，可能导致下游模板仍无法处理。  
2. **内容拼接语义差异**：将多个文本片段通过空格连接，可能与原始 JSON 中的顺序或分隔符（如换行）不完全等价，极端情况下影响 tool 的语义解释。  
3. **空列表返回空字符串**：部分业务可能期望保留列表形式（即 `[]`），现在会得到 `""`，需确认此行为在所有使用场景下均可接受。  

**💡 关注建议**：  
- **回归测试**：在完整的 integration test（包括实际 GLM5.1 模板）中验证 `normalize_tool_content` 对所有已知 tool 消息格式的兼容性。  
- **文档更新**：在 OpenAI 接口说明或内部开发手册中加入 “tool 消息 content 格式会被自动展平为字符串” 的说明，避免误解。  
- **可配置化**：若后续出现需要自定义展平规则的场景，可考虑把该函数抽象为可注入的策略，而不是硬编码在 `serving_chat`。  
- **监控报警**：在生产环境中添加日志或度量（如展平次数、列表保留次数），便于快速定位因新 tool 类型导致的异常。

**⚡ 重要程度**：🔴高  

**📋 变更摘要**

**🎯 影响范围**  
- `sglang/srt/entrypoints/*`（Engine Score MixIn、OpenAI 协议、HTTP 服务）

---

### 🔍 技术洞察  

#### 架构影响  
- **数据流扩展**：在 `ForwardBatch`、`ModelWorkerBatch`、`ScheduleBatch` 中新增 `return_pooled_hidden_states` 标记，保证从前向传播到调度器再到外层 API 的全链路传递。  
- **模型层**：`EmbeddingPoolerOutput` 扩展为同时携带 `embeddings` 与 `pooled_hidden_states`；`score_and_pool`、各 reward/cls 模型在 `forward` 时根据 `return_pooled_hidden_states` 决定是否填充该字段。  
- **调度层**：`EmbeddingBatchResult` 新增 `pooled_hidden_states`，并在 `copy_to_cpu` 中同步拷贝到 CPU，保持与 `embeddings` 的重叠调度路径一致。  
- **CUDA 图兼容**：`PiecewiseCudaGraphRunner` 引入 `capture_return_pooled_hidden_states`，在捕获阶段锁定标记，防止图 replay 时状态不一致。  
- **API 层**：OpenAI 协议模型 (`ScoringRequest/Response`) 增加布尔字段和 `pooled_hidden_states` 列表；HTTP 服务使用 `ORJSONResponse` 直接返回原始列表，避免额外的 `json.dumps`。  
- **兼容性检查**：在 `model_config.is_cross_encoding_pooler_model` 与 `TokenizerManagerScoreMixin.score_request` 中加入对不支持 PHS 的模型（如使用 `CrossEncodingPooler`）的显式报错。  

#### 性能影响  
| 影响点 | 正向 | 负向 |
| ------ | ---- | ---- |
| **额外计算** | 只增加一次 `pool_hidden_states` 的切片/索引操作，几乎不影响 FLOPs。 | - |
| **GPU→CPU 拷贝** | `EmbeddingBatchResult.copy_to_cpu` 增加对 PHS 的 `to("cpu")` 拷贝，带来额外 PCIe 带宽消耗。 | 对大 batch、长 hidden_dim（如 4k）可能导致显存/IO 瓶颈。 |
| **序列化成本** | HTTP 层将张量转为 Python 列表 → JSON，增加序列化时间和网络流量。 | 在高并发场景或长 hidden_dim 时可能成为吞吐瓶颈。 |
| **内存占用** | 每个请求多出一份 hidden_state（size = batch * hidden_dim * 4B），在多请求并发时会显著提升显存/CPU 内存需求。 | 若用户未显式关闭，默认 `return_pooled_hidden_states=False`，对现有性能无影响。 |
| **CUDA 图** | 为保证图 replay 与标记一致，若开启 `multi_item_scoring_delimiter` 会自动关闭 CUDA 图，避免错误但失去图加速。 | 失去图优化的性能提升。 |

#### 安全考虑  
- **信息泄露**：PHS 直接暴露模型内部特征向量，若模型用于机密任务，攻击者可能通过这些向量逆向推断输入内容或模型权重。建议仅在受信任环境或明确授权的客户端中开启。  
- **异常处理**：对不支持 PHS 的模型抛出 `ValueError`，避免返回错误或空值导致后端异常。  
- **输入验证**：`return_pooled_hidden_states` 被加入 OpenAI Pydantic schema，确保类型安全；HTTP 层使用 `ORJSONResponse` 防止因手动序列化产生的注入风险。

### ⚠️ 潜在风险  

1. **显存/CPU 内存激增**  
   - 大 batch、长 hidden size（如 4096）时，PHS 的拷贝与缓存可能导致 OOM，尤其在多用户共享 GPU 场景。  

2. **带宽与延迟**  
   - GPU→CPU 拷贝 + JSON 序列化在高并发或网络受限的部署环境会显著提升请求尾程延迟。  

3. **CUDA Graph 失效**  
   - 当 `--multi-item-scoring-delimiter` 被设置时，系统会自动禁用 CUDA 图，导致性能下降；用户若不知情可能误以为图加速仍在生效。  

4. **模型不兼容**  
   - `CrossEncodingPooler` 类模型（如 `BertForSequenceClassification`）仍不支持 PHS，如果用户仅通过 `json_model_override_args` 修改 `architectures`，可能在运行时触发异常。  

5. **序列化大小**  
   - 对于 1024 维隐藏向量，每个 float 4 B，单条返回约 4 KB，10 条请求即 40 KB，若 batch 规模放大到 100+，响应体可能超过常规 HTTP 超时或网关限制。

**SHA**: `32d9fe5` | 🔗 [查看提交](https://github.com/sgl-project/sglang/commit/32d9fe5a3272b369b6ca30cb43902cfee2464ff5)

**变更概述**

1. **`TritonBackend._sgemm_info`**：统一返回用于 `sgemm` 的 batch‑info（普通 `batch_info` 或合并后的 `sgemm_batch_info`），避免在每个调用处重复判断。

3. **Kernel 修改**：在 `sgemm_lora_a/b`, `qkv_lora_b`, `gate_up_lora_b` 中新增 `sorted_token_ids` 与 `SORTED_BY_ADAPTER` 参数，并通过新工具函数 `_resolve_token_positions` 将逻辑段内的偏移映射到物理 token 行。对空 segment、越界等情况提前 `return`，防止非法内存访问。  
4. **测试**：新增 `test_sgemm_sorted_by_adapter.py`，覆盖普通/合并 batch、不同 rank、单适配器、混合 rank 等多种场景，确保排序前后数值结果一致。

1. **兼容性**：  
   - `batch_info.permutation` 仅在 decode 阶段非空，确保在 translate‑only 场景（如嵌入的批量推理）仍能走原路径。建议在 `prepare_lora_batch` 前对 `max_loras_per_batch` 做下界检查，防止在没有 LoRA 的 batch 中创建空 `sgemm_batch_info`。

   - `_resolve_token_positions` 使用 `tl.load(...).to(tl.int64)`，建议明确返回类型为 `tl.int64`，并在调用处确保 `x_stride_0` 与 `output_stride_0` 的类型匹配。

   - kernel 参数顺序保持统一：`sorted_token_ids` 与 `SORTED_BY_ADAPTER` 紧随 `lora_ranks`，易于阅读。  

**结论**  
此次改动在解码路径上提供了显著的 `sgemm` 加速潜力，结构清晰且已通过完整的数值对齐测试。重点关注 `permutation` 的生命周期、`argsort` 开销以及空 segment 的 grid 计算，做适当的 guard 与文档说明后即可合入主线。

| **性能回归** | 删除 `torch.inference_mode()` 可能导致额外的 autograd tracing 开销，建议对比 GPU‑TP 环境下的吞吐量，若出现显著下降，可考虑在 `vision_tower` 前后手动加上 `torch.no_grad()` 或 `torch.inference_mode()`。 |
| **缓存粒度** | 新缓存键 `image_grid_thw` 已在处理器中对齐，确保所有涉及缓存切分的代码（如 `srt/engine/cache_manager.py`）已改为读取该键，避免因旧键名导致缓存失效。 |
| **兼容性** | 仍保留 `use_data_parallel` 条件分支，确保在多机多卡（DP）模式下不受此改动影响；若项目使用 NPU 或其他后端，确认 `is_npu` 判断仍然正确。 |
| **回滚准备** | 代码中仅删除了约 60 行旧批处理逻辑，若出现显存或性能问题，可快速恢复 `vision_tower_forward_auto` 并在配置中恢复 `KIMIV_VT_INFER_MAX_PATCH_NUM`。 |

总体来看，此次改动提升了 Kimi‑K2.5 在多图场景下的缓存细粒度和 CUDA 环境的稳定性，但需要关注显存峰值和潜在的性能回退。建议在 CI 中加入多图批推理的对比测试，以防止回归。
