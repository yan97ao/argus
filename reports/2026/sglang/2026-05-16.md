### [Spec] Clean up draft-window-size handling; extract spec arg setup to arg_groups (#25424)
**SHA**: `aec4022` | 🔗 [查看提交](https://github.com/sgl-project/sglang/commit/aec4022e58c6f74c72f9b7be449ad3e81238c14e)

**🎯 变更类型**：功能增强 / 重构 / 架构变更  
**⚡ 重要程度**：🔴高  

**📋 变更摘要**  
1. 将原本散落在 `ServerArgs.__post_init__` 中的 **speculative decoding** 参数解析与校验逻辑抽离到 `sglang/srt/arg_groups/speculative_hook.py`，形成独立的 “arg‑group” 模块。  
2. 新增 `arg_groups/argparse_actions.py`，提供 **LoRA 路径解析**、**已废弃参数的统一警告** 等通用 `argparse.Action` 实现，并在 `server_args.py` 中统一引用。  
3. 修正 **draft‑window‑size** 的命名与默认处理：`--speculative-draft-window-size` 为唯一入口，旧的 `--speculative-dflash-draft-window-size` 仍保留为别名并标记为废弃。  
4. `LlamaForCausalLMEagle3` 改为在模型初始化时一次性读取 `speculative_draft_window_size`，并在所有层上统一设置 `sliding_window_size`。  
5. 大量冗余代码删除、单元测试迁移到新 Hook。  

**🎯 影响范围**  
- `python/sglang/srt/server_args.py`（参数定义、后置处理）  
- `python/sglang/srt/arg_groups/*`（新抽象层）  
- `python/sglang/srt/models/llama_eagle3.py`（模型初始化）  
- `python/sglang/srt/speculative/*`（各 speculative worker）  
- `test/registered/unit/server_args/test_server_args.py`（单元测试）

### ⚠️ 潜在风险  

1. **导入循环风险**  
   - `speculative_hook.py` 在文件顶部 `from sglang.srt.server_args import ServerArgs`，而 `server_args.py` 在 `__post_init__` 中导入 `speculative_hook.handle_speculative_decoding`。虽然两侧都是局部导入，理论上不形成循环，但在某些工具链（如 `pydoc`、`Sphinx autodoc`）或运行时的 `pickle` 场景可能出现模块未完全初始化的异常。建议在 `speculative_hook.py` 中将 `ServerArgs` 的类型注解改为 `from __future__ import annotations` 或使用 `typing.TYPE_CHECKING` 来避免真正的导入。  

2. **废弃 flag 的误用**  
   - `DeprecatedAliasStoreAction` 通过 `argparse.SUPPRESS` 隐藏帮助信息，用户若通过脚本硬编码旧 flag，仍会产生警告且被映射。若未来决定彻底删除别名，需要确保所有文档、示例、CI 脚本已同步更新，否则会出现 “deprecated flag not recognized” 的错误。  

3. **模型层级窗口大小同步遗漏**  
   - `LlamaForCausalLMEagle3` 现在在 `__init__` 完成后统一设置 `sliding_window_size`，但如果后续还有其他模型（例如 `LlamaForCausalLMEagle3` 的子类或其他实现）在 `forward` 期间动态切换 `draft_window_size` ，新代码不会自动更新已创建的层。该场景目前不存在，但若未来引入动态窗口大小的特性，需要额外的刷新机制。  

4. **测试覆盖不足**  
   - 单元测试仅覆盖了 `ngram` 参数的异常路径，未覆盖 `DFLASH`、`FROZEN_KV_MTP`、`EAGLE` 等分支。若后续对这些路径的细节做进一步改动（如新增参数），可能缺少回归保障。  

5. **JSON 解析的错误信息**  
   - `LoRAPathAction` 抛出 `assert` 语句而非自定义异常，若在生产环境开启 `-O`（优化）模式，`assert` 将被移除，导致错误信息消失并产生不可预期的行为。建议改为显式抛出 `ValueError` 或自定义异常。

### ⚠️ 潜在风险  

1. **层区间错误**：`prefill_start_layer` 与 `prefill_end_layer` 的计算若与实际 PP 切分不一致，会导致 KV 切片偏移错误，进而产生 **attention 错位**。

| **性能影响** | - 单次 `cat + quantize` 从原先的两步（Cat → FP8）合并为一次内存读写，显著降低 **全局内存带宽** 消耗。<br>- 基准报告预计在 **seq_len ≥ 1024**、**batch_size ≥ 4k** 场景下相对 naive Triton 实现提升 **30%‑50%**（具体数值需 CI 实测）。<br>- 对 **小序列 (<16)** 使用 v0，避免 block 过大导致的空闲 warp，确保在低延迟场景不出现性能倒退。 |
| **安全考虑** | - 代码仅在内部 GPU 计算路径中使用，无网络或文件 I/O，风险极低。<br>- 唯一潜在安全点是 **PDL 开关**：如果在不支持的硬件上强制开启 `launch_pdl=True`，可能触发非法指令或未定义行为。实现已通过 `is_arch_support_pdl` 自动检测，风险可控。 |
| **可维护性** | - Triton kernel 相对复杂（大量 constexpr 参数），但代码已按功能拆分为 `_v0_kernel` 与 `_v1_flat_kernel`，并提供统一的 Python 包装 `mla_kv_pack_quantize_fp8`，对外 API 稳定。<br>- 新增的测试覆盖 **不同 dtype、stride、2‑D/3‑D k_pe**，提升回归安全性。<br>- 文档注释简洁，仍建议在 README 中补充“何时开启 PDL”以及 “如何在旧设备上回退”。 |

### ⚠️ 潜在风险  

1. **硬件兼容性**：PDL 仅在 **Compute Capability ≥ 9.0**（Hopper/Ada）上受支持；在旧卡（如 A100）上会回退到普通 launch，仍可运行，但若 `enable_pdl` 被手动强制为 `True` 可能导致 kernel 启动失败。  
2. **数值精度**：FP8 量化误差在极端小/大 scale 时可能放大；当前仅在测试中使用 `rtol=1e-2, atol=0.5`，生产环境需评估量化误差对最终模型输出（特别是 DeepSeek‑V3）是否可接受。
