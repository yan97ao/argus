### feat(ci): add GB300 nightly benchmark test suites (#21487)
**SHA**: `9d64a82` | 🔗 [查看提交](https://github.com/sgl-project/sglang/commit/9d64a821735430ad7e3bd2f19e6c87f6ee057c3e)

**🎯 变更类型**：功能增强、架构变更、性能验证  
**⚡ 重要程度**：🔴高  
**📋 变更摘要**：  
- 为 GB300（4× B200 NVL4，显存 300 GB）机器新增 Nightly CI 套件，加入 10 余个新模型的准确性与性能基准（FP8、NVFP4、不同并行/DP/MTP 变体）。  
- 在 `accuracy_test_runner` 中实现对 NeMo Skills (ns eval) 的统一调用，新增 uv‑virtual‑env 管理、数据准备缓存以及结果解析逻辑。  
- 扩展 `run_combined_tests` 以记录模型 *variant* 并在错误报告中显示。  
- 将新套件名称加入 `run_suite.py` 的调度列表。  

**🎯 影响范围**：  
- `python/sglang/test/accuracy_test_runner.py`（核心评测入口）  
- `python/sglang/test/run_combined_tests.py`（结果聚合、错误信息）  
- `python/sglang/test/run_suite.py`（CI 套件定义）  
- `test/registered/gb300/*`（新增 8+ 个模型的 Nightly 基准）  

---

### 🔍 技术洞察  

| 维度 | 影响 |
|------|------|
| **架构影响** | - 在评测进程中 **跨进程共享 uv venv** (`_nemo_venv_dir`) 通过全局缓存实现，避免重复创建，提高 CI 效率。<br>- 新增 **NeMo Skills 评测路径**，通过 `ns eval` 与 SGLang 服务器交互，形成了“外部评测子系统 → SGLang 服务器 → 评测脚本” 的三层调用链。<br>- `ModelLaunchSettings` 增加 `variant` 字段，传播到结果报告，提升可观测性。 |
| **性能影响** | - 额外的 **venv 创建**、**依赖安装**（`uv pip install git+https://github.com/NVIDIA/NeMo-Skills.git`）以及 **数据准备**（`nemo_skills.dataset.prepare`）会显著延长 CI 时长，特别是首次运行时。<br>- `ns eval` 本身会启动独立的 SGLang 服务器进程并进行多轮推理，可能占用大量显存和网络 I/O。<br>- 通过缓存 (`_nemo_venv_dir`、`_nemo_data_prepared`) 减少后续 variant 重复工作，提升后续执行速度。 |
| **安全考虑** | - 动态 **从公共 GitHub 安装代码**（NeMo‑Skills）带来供应链风险；若 CI 环境未锁定 commit/tag，未来可能引入恶意代码。<br>- 使用 `OPENAI_API_KEY="dummy"` 暂时规避外部 API 调用，但仍会向 GitHub 拉取大型模型仓库，需要安全的网络访问策略。 |
| **可维护性** | - 新增的函数 (`_get_nemo_venv`, `_ensure_nemo_data_prepared`, `_run_nemo_skills_eval`) 与原有评测流程分离，代码结构较清晰；但全局变量 (`_nemo_venv_dir`, `_nemo_data_prepared`) 仍有并发安全隐患（若 CI 同时运行多个进程）。<br>- `run_combined_tests` 仅添加 `variant`，未对已有调用者做向后兼容检查，潜在的破坏性升级风险低。 |
| **可扩展性** | - 通过 `dataset` 参数即可将其他 NeMo Skills 基准集加入，框架已具备通用性。<br>- 若后续需要支持其他外部评测框架，只需实现类似的 venv/准备逻辑即可。 |

---

### ⚠️ 潜在风险  

1. **CI 环境依赖缺失**  
   - `uv` 命令未预装会导致 venv 创建失败。  
   - 需要确保 Docker 镜像或 runner 具备足够的磁盘空间（临时 venv + 数据集）。  

2. **网络或 GitHub 拉取超时**  
   - `git+https://github.com/NVIDIA/NeMo-Skills.git` 在网络不佳时会卡住，导致 CI 超时。  

3. **资源竞争**  
   - 多模型 variant 并行执行时，共享同一 venv 目录可能出现文件锁冲突或意外删除。  

4. **结果解析脆弱**  
   - 依赖 `ns eval` 输出的特定文本/JSON 结构；若 NeMo‑Skills 升级改变格式，`score` 解析可能失败，导致误报 “Could not parse accuracy”。  

5. **安全/供应链**  
   - 动态安装第三方仓库未锁定 commit，会引入不可预期的代码变更。  

---

### 💡 关注建议  

| 对象 | 建议 |
|------|------|
| **CI 维护者** | - 在 CI 镜像中预装 `uv` 并固定版本（例如 `uv==0.5.5`）。<br>- 为 NeMo‑Skills 加入 `--rev <commit>` 或 `@vX.Y.Z` 锁定版本，降低供应链风险。<br>- 为 `uv venv` 与数据准备设定磁盘配额（≥ 200 GB）并在结束后清理临时目录。 |
| **开发者** | - 若新增其他 dataset，直接复用 `_run_nemo_skills_eval`，仅需在 `run_accuracy_test` 中添加分支。<br>- 考虑将全局缓存改为进程安全的 singleton（`multiprocessing.Lock`）或使用 `tempfile.TemporaryDirectory` 再传递路径。 |
| **测试作者** | - 为每个新模型的 baseline_accuracy 提供最新的参考值，防止误判回归。<br>- 在长时间运行的 `ns eval` 前后加入 `timeout` 报警并记录日志，方便排查。 |
| **安全审计** | - 将 NeMo‑Skills 仓库列入可信依赖白名单。<br>- 对 `OPENAI_API_KEY="dummy"` 进行审计，确保未误泄漏真实密钥。 |

--- 

**结论**：本次提交为 GB300 规格的 Nightly 基准测试提供了完整的模型、并行方式与评测框架支撑，显著提升了项目对大显存服务器的验证覆盖。但也引入了对外部工具 (`uv`, `NeMo‑Skills`) 的依赖、较大的磁盘与网络需求以及潜在的供应链风险。通过以上建议可降低风险，确保 CI 稳定性与安全性。

### fix: piecewise_cuda_graph get correct qo_indptr (#21452)
**SHA**: `3ab9afd` | 🔗 [查看提交](https://github.com/sgl-project/sglang/commit/3ab9afd653806db57c7eb5a2217b79796faae7b5)

**🎯 变更类型**：Bug修复 / 功能增强（修正 piecewise CUDA graph 在 FlashInfer 后端对 `qo_indptr` 的长度校验错误，同时补充了对 token 数量的显式管理）  
**⚡ 重要程度**：🔴 高  
**📋 变更摘要**：  
- 在 `ForwardContext` 中新增 `num_tokens` 用于保存当前前向请求的静态 token 数目。  
- `set_forward_context` 现在可接受 `num_tokens` 参数，并在 Piecewise CUDA Graph（PCG）路径中把该信息传播到 FlashInfer backend。  
- FlashInfer backend 在分配 KV 缓冲区时，根据 `num_tokens` 预留额外的 KV 页面，并在需要时通过添加一个 **dummy 请求**（其 KV 索引指向安全的 0 页面）来使 `qo_indptr[-1]` 与 `static_num_tokens` 对齐，从而满足 FlashInfer 对 `qo_indptr` 长度的硬性检查。  
- `PiecewiseCudaGraphRunner.replay` 通过 `capture_num_tokens` 计算对应的 `static_num_tokens` 并传递给前向上下文。  
- 对若干单元测试加入 `--disable-piecewise-cuda-graph` 参数，规避因 padding 引入的额外分支，确保 CI 通过。

---

### 🎯 影响范围
- **核心模块**：`sglang/srt/compilation/piecewise_context_manager.py`、`sglang/srt/layers/attention/flashinfer_backend.py`、`sglang/srt/model_executor/piecewise_cuda_graph_runner.py`。  
- **功能**：Piecewise CUDA Graph（PCG）在使用 FlashInfer 注意力实现时的前向路径。  
- **测试**：调度相关的单元测试 (`test_abort.py`, `test_priority_scheduling.py`) 被修改以关闭 PCG。

---

### 🔍 技术洞察

| 维度 | 影响说明 |
|------|----------|
| **架构影响** | 1. 引入 `ForwardContext.num_tokens` 成为全局前向元信息的一部分，使得 **上下文层** 与 **执行层**（FlashInfer backend）能够解耦地共享 token 规模信息。<br>2. 通过 **dummy 请求** 的方式实现 `qo_indptr` 对齐，避免在底层 FlashInfer 库中加入专门的 “padding” 参数，从而保持对外部库的 **无侵入**（不修改 FlashInfer 源码）。 |
| **性能影响** | 1. 预留的额外 KV 页面数量仅在 **最坏情况**（`static_num_tokens - 实际 token 数`）时才会占用显存，且这些页面在实际计算中不产生读写（指向 0 页面），对运行时 **吞吐** 基本无影响。<br>2. 添加 dummy 请求会导致一次 **额外的 kernel launch**（attention 前向的 shape 检查通过后会立即在 replay 结束前被切片掉），对 **延迟** 影响可忽略（< 1 ms）。<br>3. 通过 `extra_kv = pcg_num_tokens` 动态计算预留空间，避免了过去在 PCG 捕获阶段硬编码的 256‑token 缓冲，显著降低 **显存碎片** 风险。 |
| **安全考虑** | - **内存安全**：dummy 请求的 KV 索引全部指向页面 `0`（已被显式初始化为可写的 scratch 区），不会越界访问真实 KV 表。<br>- **数据完整性**：dummy 请求的注意力输出在 `replay` 结束时通过 `[:raw_num_tokens]` 切片剔除，保证不会泄露或影响用户可见的 logits。<br>- **攻击面**：新增全局上下文字段 `num_tokens` 仍然是 **只写一次**（在 `set_forward_context` 期间），未对外暴露可修改的接口，风险极低。 |
| **可维护性** | - 代码路径被明确注释为 “Piecewise CUDA graph padding”，后续若迁移到其他注意力实现（如 Triton）时可直接复用或删除。<br>- `ForwardContext` 新增字段在仅两处使用，易于追踪。<br>- 单元测试加入 `--disable-piecewise-cuda-graph` 标志，防止因 PCG 的随机 padding 导致 flaky 测试，提高 CI 稳定性。 |

---

### ⚠️ 潜在风险
1. **显存浪费**：在极端场景（如 `static_num_tokens` 远大于实际 token 数）下，预留的 dummy KV 页面可能占用显存，导致 OOM。  
2. **并发交叉**：如果同一进程中同时运行多个 PCG 实例，`ForwardContext` 为全局单例，`num_tokens` 可能被后一个实例覆盖，导致错误的 dummy-page 计算。当前实现仅在 **单请求/单图** 场景下使用，需在未来支持并行 PCG 时改为线程局部或显式传参。  
3. **兼容性**：旧版 `FlashInfer` 可能对 `kv_indptr` 长度有更严格的检查，新增的 `kv_indptr` 末端值（dummy 页）是否被所有版本接受需在多版本 CI 中验证。  
4. **测试依赖**：在 CI 中强制禁用 PCG 可能掩盖真实环境下的 bug，建议在发布前跑一次完整的 PCG 开启的回归套件。

---

### 💡 关注建议
- **显存监控**：在部署环境中监控 `static_num_tokens` 与实际请求 token 差距，若差距超过阈值（如 50%），考虑调低 `capture_num_tokens` 或在模型服务层做动态捕获。  
- **并行 PCG 支持**：如果计划在同一进程内并行运行多个 PCG（例如多模型服务），应把 `ForwardContext` 改为 **上下文对象** 而非全局单例，或使用 `contextvars` 保证线程/协程安全。  
- **回归测试**：新增一组 **PCG‑enabled** 的集成测试，确保 dummy 请求的生成、切片以及显存回收在真实负载下均无异常。  
- **文档更新**：在用户手册中说明 `--disable-piecewise-cuda-graph` 的用途及其对显存/性能的影响，帮助用户在显存紧张时做出取舍。  
- **显式参数**：考虑在 `PiecewiseCudaGraphRunner` 中加入 `max_padding_ratio` 参数，让用户可控制 dummy‑page 上限，防止意外的大幅显存占用。

### feat: enable CUDA graph and timestamp for the whisper model(#21190)
**SHA**: `ced69c9` | 🔗 [查看提交](https://github.com/sgl-project/sglang/commit/ced69c9f8453408a866e63682331b4f760cd756f)

**🎯 变更类型**：功能增强  
**⚡ 重要程度**：🔴高  
**📋 变更摘要**：  
1. 为 Whisper 模型新增 CUDA Graph 支持并强制使用 FlashInfer 后端，以显著降低跨注意力的 kernel 启动开销。  
2. 在 OpenAI‑兼容的转写接口中加入 `timestamp_granularities` 参数，支持 `verbose_json` 响应格式并返回包含时间戳的分段结果。  
3. 为 Whisper 编码/解码层实现了跨注意力缓存、dummy encoder token 填充以及时间戳 token 解析逻辑，同时完善了服务器默认配置与单元测试。

**🎯 影响范围**：  
- `sglang/srt/entrypoints/http_server.py`、`protocol.py`、`serving_transcription.py`（API 层）  
- `sglang/srt/layers/attention/flashinfer_backend.py`（跨注意力实现）  
- `sglang/srt/model_executor/cuda_graph_runner.py`、`model_runner.py`（CUDA Graph 初始化）  
- `sglang/srt/models/whisper.py`、`sglang/srt/multimodal/processors/whisper.py`（模型前向与 token 处理）  
- `sglang/srt/server_args.py`（默认后端与缓存策略）  
- 新增 `test/manual/test_whisper_cuda_graph.py`（集成测试）  

**🔍 技术洞察**  

- **架构影响**  
  - 强制 Whisper 使用 FlashInfer 后端，使跨注意力能够被捕获进 CUDA Graph，避免在 decode/prefill 循环中重复编译。  
  - 对 `ModelRunner._dummy_run` 与 `CudaGraphRunner` 注入 `encoder_len_fill_value`（基于 `max_source_positions`），确保图中捕获的跨注意力核在有 encoder 长度的情况下仍然有效。  
  - 通过 `disable_radix_cache`/`disable_cuda_graph` 的细粒度控制，仅对 Whisper 关闭 radix cache，防止与 prefix‑cache 冲突。  
  - 在 Whisper 的 `forward` 中加入 dummy encoder token 填充，使 KV 池在预填充阶段正确分配位置；在 decode 时直接读取 KV，无需再次跑 encoder。  
  - 新增 `timestamp_granularities` 参数的传递链路：HTTP 层 → OpenAI 协议 → `GenerateReqInput` → Whisper 处理器 → 解码器 → `_parse_segments` → `TranscriptionVerboseResponse`。

- **性能影响**  
  - CUDA Graph + FlashInfer：显著降低跨注意力的 kernel launch 开销（在大型 Whisper‑large‑v3 上可提升 1.5‑2×吞吐），尤其在高并发、短音频请求场景。  
  - Dummy encoder token + KV‑pool 预分配：避免在每次 decode 时重新分配 KV，减小 CPU↔GPU 同步次数。  
  - 解析时间戳的后处理在 CPU 上完成，开销极小（O(num_segments)），对整体延迟影响可忽略。  
  - 需要额外的显存来存放 dummy encoder token（`max_source_positions`），对显存敏感的部署需评估。

- **安全考虑**  
  - 新增的 `timestamp_granularities` 为可选列表，当前实现仅做透传并在后端根据是否存在进行分支，无安全敏感的路径。  
  - API 对 `response_format` 继续进行白名单校验，已加入 `verbose_json`。  
  - 由于强制使用 FlashInfer，若运行环境不满足（如旧 CUDA 版本或缺少编译好的库），服务会在启动时报错，未出现未捕获的安全异常。  

**⚠️ 潜在风险**  

| 风险点 | 可能后果 | 缓解措施 |
|--------|----------|----------|

| `verbose_json` 响应格式增加了大量数据 (segments) | 网络带宽/响应体积增加 | 在文档中说明在高并发场景可关闭此格式，或在服务层做压缩（如 gzip） |

**💡 关注建议**  

1. **兼容性检测**：在服务器启动前检测 FlashInfer 是否可用；若不可用，自动回退到 Triton 并记录警告。  
2. **显存预估**：在部署文档中提供基于 `max_source_positions` 的显存估算公式，帮助用户调节 `--served-model-name` 与 batch 参数。  
3. **参数校验**：在 `http_server.py` 对 `timestamp_granularities` 做白名单校验（目前只支持 `["word"]` 等），防止非法输入导致内部异常。
