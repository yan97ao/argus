## ⚠️ 潜在风险

1. **多重继承冲突**  
   - 目前所有 NPU 测试类同时继承 `GSM8KAscendMixin`, `TestMMLU` 与 `CustomTestCase`。若将来 `CustomTestCase` 引入新的 `setUpClass` 实现，可能与混入提供的属性产生覆盖或顺序问题。  

2. **阈值过严导致 CI 失效**  
   - 通过 `assertGreaterEqual` 检查 `latency`、`throughput`，在硬件资源波动（尤其是共享 CI 节点）时可能导致误报。建议在 CI 中保留 **容错 margin** 或将阈值配置为环境变量。  

3. **环境变量硬编码**  
   - 部分 NPU 环境变量（如 `PYTORCH_NPU_ALLOC_CONF`, `HCCL_BUFFSIZE` 等）仍在每个测试文件中重复声明。若平台升级或需要统一管理，仍可能出现不一致。  

4. **GitHub Summary 写入**  
   - `write_github_step_summary` 在非 GitHub Actions 环境仍会尝试写文件，若没有相应的 `GITHUB_STEP_SUMMARY` 环境变量，可能抛出异常。当前已用 `is_in_ci()` 包装，但仍建议在异常捕获层面冗余防御。  

5. **模型路径硬编码**  
   - 新增的权重路径（如 `Eco-Tech/Qwen3-32B-w4a4-LAOS`）依赖 ModelScope 本地缓存。如果 CI 环境未预先缓存，首次拉取可能耗时显著或因网络不可达导致测试卡死。

### [diffusion] feat: progressive resolution growing for image and video models (#27524)
**SHA**: `b0cd533` | 🔗 [查看提交](https://github.com/sgl-project/sglang/commit/b0cd533a963a53dafc5b904763cf838e3249d095)

**🎯 变更类型**：功能增强 / 性能优化 / 架构变更  
**⚡ 重要程度**：🔴高  
**📋 变更摘要**：  
本次提交在 SGLang Diffusion 子系统中实现了 *Progressive Resolution Growing*（逐步分辨率提升）实验特性。通过在前 N 步使用低分辨率 latent，随后在频域（DCT）上进行光谱上采样并可选地对噪声尺度进行 rewind，显著降低 DiT Transformer 的二次注意力成本，实现 **1.3–2.8×** 的推理加速（不同模型略有差异）。实现涉及：

1. 在 `SamplingParams` 中加入 `progressive_mode / levels / delta` 三个新字段，并完成 CLI 参数、合法性校验。  
2. 为 FLUX.1、FLUX.2、Qwen‑Image、Z‑Image、Wan‑T2V 等模型分别实现 **ProgressiveDenoisingStage** 子类（包装/展开 latent、更新 RoPE/latent_ids 等模型特有状态）。  
3. 新增 **ProgressiveDenoisingStageRouter**，在运行时根据 `progressive_mode` 动态路由至标准 `DenoisingStage` 或对应的渐进阶段实现。  
4. 完整实现 DCT/IDCT 光谱上采样（GPU‑native、无 CPU↔GPU 拷贝）以及 rewind‑mode 的噪声尺度校正。  
5. 在 `ComposedPipelineBase` 中加入 `add_progressive_denoising_stage` 与扩展的 `add_standard_t2i_stages`/`add_standard_ti2i_stages`，实现可选的渐进降噪阶段插入。  
6. 文档、示例 CLI、Python API 完整说明；并在 `docs_new` 中补充使用指南与性能基准。  
7. 实现了对 **Sequence Parallelism、torch.compile、Cache‑DiT** 的兼容性检查与警告。

| 维度 | 影响描述 |
|------|----------|
| **架构影响** | - 引入 `ProgressiveDenoisingStageRouter` 实现 **运行时动态路由**，保持对现有 `DenoisingStage` 的完全兼容。<br>- 通过 `add_progressive_denoising_stage` 把渐进阶段抽象为可插拔组件，所有 diffusion pipelines（Flux、Flux‑2、Qwen‑Image、Z‑Image、Wan‑T2V）只需提供对应子类即可。<br>- `ComposedPipelineBase` 新增 `add_standard_t2i_stages` / `add_standard_ti2i_stages` 参数化，从而在不修改原有 pipeline 代码的情况下平滑迁移。 |
| **性能影响** | - 通过在低分辨率（1/2ⁿ）latent 上执行大量早期 denoising 步骤，降低 **序列长度 O(n²)** 的注意力计算，从而在实验基准上实现 **1.3–2.8×** 的整体推理加速。<br>- DCT/IDCT 实现在 GPU 上全程执行，避免了 CPU‑GPU 数据搬运以及 `torch.compile` 的重新编译。<br>- `rewind` 通过对噪声尺度进行 **γ‑scaling**，在保持生成质量的前提下进一步缩短了高分辨率步骤的噪声水平。 |

| **可维护性** | - **代码组织**：将公共的 sweeper抽象为 `active_load::spawn_sweeper`，降低重复，实现统一日志与错误处理。<br>- **默认值集中**：`StickyConfig::default()` 与 `default_sticky_*` 函数统一管理默认 TTL 与扫荡间隔，易于调参。<br>- **测试覆盖**：单元、属性、端到端测试超过 30 条，覆盖配置校验、淘汰、并发首次触达、fallback 行为等，提升后续改动的安全网。 |
| **兼容性** | - 仅在 `policy=sticky` 时激活新字段，默认 `sticky: None`，对已使用 `round_robin`、`random`、`load_based` 等策略的现有部署完全透明。<br>- CLI 参数若未指定 `--policy sticky`，会报错阻止误用。 |

**⚠️ 潜在风险**  
1. **键空间失控**：若业务方使用高基数（如每请求唯一的 UUID）作为路由键，且 `idle_secs` 设置过大，`DashMap` 可能在长时间运行后占满内存。  
2. **后台扫荡未启动**：在单元测试或某些非 Tokio 运行时环境（如 sync 构造）`StickyPolicy::new` 会检测不到 runtime 并不启动 sweeper，导致键永不淘汰。如果生产环境误在未运行 Tokio 的线程中创建策略，黏性会失效并可能出现内存泄漏。  
3. **错误的 fallback 策略**：虽然 CLI 已限制，但如果 `PolicyKind::Sticky` 被硬编码在代码（跳过 `Cli::into_config`）并手动构造 `StickyConfig`，仍可能传入 `CacheAwareZmq` 或 `Sticky` 本身，引发 `unreachable!` panic。  
4. **指标冲突**：新增的 `sgl_router_sticky_total` 若在旧版 Prometheus 规则或 Grafana 仪表盘中未声明，可能导致监控报警缺失。  
5. **跨实例黏性不一致**：用户若误以为黏性在集群层面生效，可能导致请求被错误路由到错误的后端，产生业务错误。  

**💡 关注建议**  
- **运营建议**

  - 为防止键空间暴涨，建议在业务层保证路由键的 **可复用性**（如会话 ID、用户 ID），并根据业务特性调小 `idle_secs`。  
- **代码/测试**

     - **Reference 实现**：在 `attention_unittest` 中实现 `_dual_chunk_sparse_fallback_indices_reference`、`_dual_chunk_sparse_selection_reference` 等函数，完整复制生产路径的 vertical/slash top‑k、stage 划分以及空‑stage fallback 逻辑，用于对比。  
     - **捕获实现**：通过 monkey‑patch `_vertical_slash_sparse_attention` 捕获实际 `vertical_indices` / `slash_indices`，并转为可比对的结构体 `_DualChunkSparseStageSelection`。  
     - **torch‑sparse 参考**：实现纯 PyTorch 稀疏注意力 (`_torch_sparse_attn_func`) 读取同样的元数据并计算期望输出，确保后端算子在数值上与参考实现一致。  
     - **新测试用例** `run_dual_chunk_sparse_sub_window_case`：构造 `vertical_size=4, slash_size=4, seq_len=128` 的子窗口场景，检查：  
       * 生成的 stage selection 与 reference 完全一致。  
       * 最终输出与 torch‑sparse 参考 `torch.testing.assert_close`（rtol/atol）相符。  
     - 更新 `dual_chunk/README.md`、`KNOWN_FAILURES.md`、测试注册文件，加入该子窗口案例并标记已经可运行。  

**🔍 影响范围**  
| 模块 | 影响 | 备注 |
|------|------|------|
| `python/sglang/srt/layers/attention/dual_chunk_flashattention_backend.py` | 直接修改稀疏 fallback 路径，所有使用 Dual‑Chunk 结构且开启 `sparse_attention_enabled` 的模型都会受益。 |

**⚙️ 行为与性能**  
- **行为**：当没有任何 vertical/slash 索引被 top‑k 选中时，后端现在会返回 **`min(max_count, seq_len)`** 个均匀分布的索引，而不是固定的 “最多 5 条” 的 arange。这仍然满足 fallback 的要求（只要返回合法且不超过 buffer 大小），且不会导致运行时异常。  
- **数值差异**：在极端 `seq_len < max_count` 时，返回的索引集合会更小（等于 `seq_len`），这与之前的实现一致。对实际稀疏注意力计算影响极小，因为 fallback 只在 **完全没有被选中的 stage** 使用，且随后会被 `torch.nn.functional.softmax` 的 mask 填充 `-inf`，只要索引合法即可。  
- **性能**：`_sparse_fallback_indices` 使用一次 `torch.arange` + 切片，和原实现的 `torch.arange` 差别可忽略；增加的 Python 检查 (`min`, `ceil`) 只在极少数空‑stage 场景触发，对整体吞吐量影响几乎为 0。  

**✅ 建议**  
1. **CI 验证**：在 CI 中确保 `run_dual_chunk_sparse_sub_window_case` 被执行；该测试比原来的 “smoke” 更严格，需要 `torch`、`math` 以及 CUDA（若可用）支持。  
2. **文档同步**：将 `dual_chunk/README.md` 中关于 “vertical_size + slash_size >= chunk_len_blocks” 的隐式前提更新为 “fallback 已不再依赖该不变式”。  
3. **回归监控**：由于 fallback 生成方式略有变化，建议在高层模型（例如 OpenAI‑style LLM）运行一次长序列推理，观察 `sparse_attention_enabled=True` 时的显存和速度是否保持不变。  
4. **代码风格**：`_sparse_fallback_indices` 目前返回 `int64`（与后端其他索引保持一致），若未来改为 `int32`，记得同步所有调用处的 dtype。  

**总结**  
此 PR 通过统一的 fallback 索引生成函数解决了 Dual‑Chunk 稀疏注意力在小 `vertical_size/slash_size` 配置下的越界 crash，并配套完整的参考实现与捕获测试，使原本被标记为 “blocked” 的子窗口稀疏路径正式进入 CI 覆盖。对功能安全性提升显著，性能影响可忽略，建议尽快合并并在后续模型验证中继续观察。
