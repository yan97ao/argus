### [kernel slimming] Clean many useless sgl-kernel deprecated kernels (#20277)
**SHA**: `25e3821` | 🔗 [查看提交](https://github.com/sgl-project/sglang/commit/25e38216b6af2472949f17b1ec0f4e476768d334)

**🎯 变更类型**：功能增强 / 重构 / 架构变更  
**⚡ 重要程度**：🔴高  
**📋 变更摘要**：此次提交对 *sgl‑kernel* 中大量已废弃的 CUDA 实现进行删除，并将对应功能统一迁移到 *sglang.jit_kernel*（基于 Triton/JIT 的实现）。包括 `rope.cu`、`per_tensor_quant_fp8.cu`、`store.cu` 等核心算子及其 Python 包装层被移除，相关 benchmark、测试和导入路径同步更新。目标是精简代码体积、降低维护成本，并统一使用更易调试的 JIT 实现。

**🎯 影响范围**  
- `sgl-kernel` 子模块（C++/CUDA 代码、CMake 构建列表）  
- Python 包入口 `sgl_kernel/__init__.py`、`elementwise.py`、`gemm.py`、`memory.py`、`testing/rotary_embedding.py`  
- 相关 benchmark 脚本 `bench_rope.py`, `bench_store_cache.py`, `bench_fp4_gemm.py` 等  
- 单元测试文件（FP4、FP8、Rotary、TensorQuant 等）  
- 依赖 `sgl_kernel` 的第三方项目或内部脚本（若仍使用已删除 API）  

**🔍 技术洞察**  

| 维度 | 影响分析 |
|------|----------|
| **架构影响** | - 将原本散布在 `sgl_kernel`（编译型 C++/CUDA）与 `sglang.jit_kernel`（Triton JIT）两套实现的算子统一到后者，提升了代码一致性。<br>- 删除 `apply_rope_pos_ids_cos_sin_cache`、`sgl_per_tensor_quant_fp8`、`store_kv_cache` 等旧接口，简化了库的公共 API 表面，使得库的使用者只能通过 JIT 接口访问同等功能。<br>- CMake 列表中剔除 400+ 行无用源码，缩短编译时间、减小二进制体积。 |
| **性能影响** | - **正向**：JIT kernel 在多数案例下已经实现了与或优于原 CUDA 实现的吞吐（尤其在 Hopper/Blackwell GPU 上的 FP4/FP8 量化）。移除冗余实现可避免误用旧实现导致潜在的次优路径。<br>- **风险**：某些旧实现可能在特定硬件或低算力 GPU（如未开启 PDL、或不支持 Triton JIT）上仍有优势；移除后若没有提供兼容层，用户会遇到性能回退或运行错误。<br>- `store_kv_cache` 被删除，原本的 AOT（Ahead‑Of‑Time）路径在 HIP 环境下会失效，现仅保留 JIT 版本。 |
| **安全考虑** | - 删除的算子均为数值计算，不涉及外部资源或授权检查，安全风险极低。<br>- 代码移植到 Triton 后，需要确保 Triton 编译器的安全性（已在项目 CI 中通过），不引入新的攻击面。 |
| **可维护性** | - 大幅减少 C++/CUDA 代码量（约 1500 行被删除），降低了维护成本和潜在的内存泄漏、同步错误。<br>- Python 接口统一后，文档和 IDE 自动完成更易管理。<br>- 迁移后，所有算子均在统一的 `sglang.jit_kernel` 包中，便于统一测试、CI 与功能迭代。 |
| **兼容性** | - 直接删除了多个公开函数（例如 `apply_rope_pos_ids_cos_sin_cache`、`sgl_per_tensor_quant_fp8`、`set_kv_buffer_kernel`），会导致依赖旧 API 的项目 **立即编译失败**。<br>- 迁移的函数在新路径下仍保持相同的行为（已在 benchmark 与单元测试中验证），但参数签名略有变化（例如 `scaled_fp4_quant` 现在返回 `Tuple[torch.Tensor, torch.Tensor]`），需要使用者迁移代码。 |

**⚠️ 潜在风险**  
1. **向后兼容破坏**：直接删除 API 没有提供过渡期警告，现有用户（尤其是内部工具）若未同步代码会出现 ImportError。  
2. **平台适配缺失**：原 `store_kv_cache` 在 HIP（AMD）平台上有特殊实现，删除后该平台将只能走 JIT 路径；在不支持 Triton JIT 的环境（如某些旧版 PyTorch）会报错。  
3. **性能回退风险**：部分极端配置（例如极小 batch、低算力 GPU）可能仍受益于原始手写 CUDA 内核，失去回退路径可能导致吞吐下降。  
4. **测试覆盖不足**：虽然多数 benchmark 已更新，但仍有可能遗漏某些边缘 case（如自定义 `FusedSetKVBufferArg` 使用场景），导致运行时错误。  
5. **文档同步**：`sgl_kernel` 官方文档仍列出已删除的函数，若未及时更新，会误导用户。  

**💡 关注建议**  
- **提供迁移指南**：在项目 README 或 Release Notes 中列出所有被删除的接口，并给出对应的 `sglang.jit_kernel` 替代使用示例。  
- **保留兼容层（可选）**：在 `sgl_kernel/__init__.py` 中加入薄包装的 **警告**（`DeprecationWarning`），在内部调用新 JIT 实现，以免突发报错。可在后续 major 版本中彻底移除。  
- **平台检测**：在 `bench_store_cache.py` 中恢复对 HIP 环境的检测逻辑，或在 JIT 实现中加入 fallback 到纯 Python 实现，以保持跨平台可用性。  
- **性能回归基准**：在 CI 中加入对比旧 CUDA 实现与新 JIT 实现的基准（仅在支持的机器上），确保关键路径的吞吐不出现显著下降。  
- **文档清理**：同步更新 `sgl_kernel` API 文档及自动生成的 `sphinx` 页面，删除已废弃函数的条目。  
- **回滚策略**：保留被删除的源码分支（如 `kernel-slimming/backup`），在出现严重兼容或性能问题时可以快速恢复。  

---  

**总结**：本次提交是一项大规模的“内核瘦身”工作，旨在通过统一到 Triton JIT 实现来提升代码可维护性和统一性。若配套做好迁移文档、兼容警告以及平台/性能回归验证，风险可控且长期收益（更小的二进制、更快的编译、更易迭代的实现）将明显超过短期的兼容破坏。开发者在升级后需要检查自己的代码是否仍引用已删除的 `sgl_kernel` 接口，并及时切换到对应的 `sglang.jit_kernel` 实现。

### [Diffusion] Clean upstream fa3 in hopper (#20576)
**SHA**: `5ab2cfe` | 🔗 [查看提交](https://github.com/sgl-project/sglang/commit/5ab2cfe9a899d1d2dffae932fb0f6371014d0d71)

**🎯 变更概览**  
1. 两个 benchmark 脚本的使用说明将硬编码路径 `cd /data/bbuf/sglang` 替换为通用的 `cd /path/to/sglang`，仅是文档层面的清理。  
2. `flash_attn.py` 大幅删减：去掉了 `lru_cache`、日志初始化、`current_platform` 检查以及一整套“上游 Flash‑Attention（hopper）”的检测、缓存函数、`_should_use_upstream_flash_attention` 判定和对应的代码路径；只保留内部实现（fa_ver 3/4）的调用。

**⚡ 影响范围**  
- `sglang/multimodal_gen/runtime/layers/attention/backends/flash_attn.py`（注意力后端）  
- 依赖该后端的所有模型/脚本（尤其是在 Hopper 平台上运行时）  
- 两个 benchmark 脚本的帮助信息（不影响功能）

**🔍 重点关注**  
- **性能回退**：在 Hopper 上原先会自动切换到 upstream `flash_attn3`（CUDA‑optimized）实现，现在全部走内部实现，可能导致显著的吞吐下降。若项目仍在 Hopper 上部署，需在外部自行安装并调用 upstream 包，或恢复相应的切换逻辑。  
- **GQA（Group‑Query‑Attention）支持**：原实现会根据 `num_kv_heads` 判断是否可以使用 upstream，实现中对 `self._upstream_heads_ok` 的设置被删掉，但 `forward` 已不再引用该属性，属性也未再创建。请确认代码其它位置不再访问 `self._upstream_heads_ok`，否则会触发 `AttributeError`。  
- **日志与警告**：删除了 `init_logger` 与未安装 upstream 包的警告信息，用户若缺少依赖将不再得到提示，建议在文档或 README 中明确说明依赖要求。  
- **兼容性**：如果外部代码仍期望 `flash_attn` 包含 `flash_attn_varlen_func_upstream` 等符号，会因为导入失败而报错。请检查公共 API 是否被其他模块使用。  

**💡 建议**  
1. 在 `flash_attn.py` 注释说明已放弃自动切换 upstream，实现仅适用于内部实现。  
2. 如仍希望保留最佳性能，可提供可选的开关（例如环境变量 `SGLANG_USE_UPSTREAM_FA=1`）让用户自行决定。  
3. 确认 `self._upstream_heads_ok` 不再被外部引用，或在 `__init__` 中显式删除/置为 `None` 防止潜在属性错误。  
4. 在项目文档或 benchmark 脚本的 README 中加入对依赖（如 `flash_attn3`）的说明，避免使用者因缺失依赖而误判性能问题。  

总体而言，此次提交主要是简化注意力后端代码并去除平台特定的分支，但会牺牲 Hopper 上的最优性能和部分兼容检查，需评估在目标部署环境中的实际影响后再决定是否合并。

### [Omni] Optimize AudioEncoder for Qwen3_Omni_Thinker (#18185)
**SHA**: `22e6787` | 🔗 [查看提交](https://github.com/sgl-project/sglang/commit/22e67876d6aa6992bff89d5d91aef2c2c6b3d1bb)

**🎯 变更类型**：功能增强 / 性能优化  
**⚡ 重要程度**：🟡 中  

**📋 变更摘要**  
1. 将 Qwen3OmniMoeAudioEncoder 中的前馈层从普通 `nn.Linear` 替换为列/行并行实现（`ColumnParallelLinear`、`RowParallelLinear`），以利用模型并行加速。  
2. 重构卷积前向：使用向量化 mask 生成、合并 `conv_chunksize` 的 fast‑path 与分块路径，避免大量小张量和 OOM。  
3. 修正 `conv_out` 维度计算、`feature_attention_mask` 的设备对齐、以及对 `aftercnn_lens` 的 Python 列表转换，提高速度和显存利用。  

**🎯 影响范围**  
- `python/sglang/srt/models/qwen3_omni_moe.py` 中的 **AudioEncoder** 与 **FeedForward** 逻辑。  
- 依赖 `ColumnParallelLinear` / `RowParallelLinear` 的并行库（如 Megatron‑LM）以及相关注册表。  
- 任何调用 `Qwen3OmniMoeAudioEncoder.get_audio_feature`、`forward` 的上层模型（如 Omni‑Thinker）均受影响。  

**💡 关注建议**  
1. **并行层兼容性**：确认 `ColumnParallelLinear`、`RowParallelLinear` 在单卡或多卡环境下的 fallback 行为，防止在仅有 1 GPU 时出现未初始化的通信张量。  
2. **梯度与权重检查**：并行线性层默认会拆分权重，确保加载原始 checkpoint 时能够正确切分（或提供转换脚本）。  
3. **mask 生成**：向量化 mask 依赖 `feature_lens_after_cnn.max()`，在全零或空 batch 场景下 `max_len_after_cnn` 为 0，需确保后续 `idx` 创建不会报错。  
4. **conv_chunksize 逻辑**：fast‑path 分支仅在 batch ≤ `conv_chunksize` 时走，确认该阈值在实际部署中的取值与显存预算匹配。  
5. **dtype/设备一致性**：`feature_attention_mask` 现在强制转到模型参数所在设备，建议在 `forward` 开头统一 `device = hidden_states.device`，避免潜在的跨设备复制。  
6. **单元测试**：新增对并行前馈层与向量化 mask 的功能/性能基准测试，确保数值误差在容忍范围内（尤其是 fp16 场景）。  
7. **文档/兼容性说明**：在模型配置或 README 中标注 “需要 Megatron‑LM 并行实现”，并说明在非并行环境的回退方式或兼容层实现。  

总体而言，此次改动可显著降低音频特征抽取的显存占用并提升并行吞吐，但需在多环境下做充分验证，防止因并行层初始化或 mask 边界条件导致的训练/推理异常。

### [Model] Support Nemotron 3 Super NVFP4 (#20407)
**SHA**: `75a7879` | 🔗 [查看提交](https://github.com/sgl-project/sglang/commit/75a7879fd4c149e675b50f9b21964e557577cc4f)

**🎯 变更类型**：功能增强（新增对 Nemotron‑3 Mixed‑Precision ModelOpt 检查点的支持）  
**⚡ 重要程度**：🟡 中  

**📋 变更摘要**  
1. 在 `ModelConfig._parse_modelopt_quant_config` 中识别 `MIXED_PRECISION` 量化且模型为 Nemotron 系列，返回 `quant_method="modelopt_mixed"`。  
2. 为 `modelopt_mixed` 新增量化实现：`ModelOptMixedPrecisionConfig`，封装 FP8 与 NVFP4 子配置并实现层级量化映射、权重名映射、量化方法分发等。  
3. 扩展量化方法注册表、加载器分支、服务器参数校验以及 MoE‑kernel 兼容检查，使 `modelopt_mixed` 能在现有流水线中被识别和使用。  
4. 新增单元测试验证：① Nemotron 检测路径；② 不影响 `w4afp8` 的覆盖；③ 最低硬件能力与 `ModelOptFp4Config` 对齐；④ 权重映射后的层前缀解析正确。  

**🎯 影响范围**  
- `python/sglang/srt/configs/model_config.py`（模型配置解析）  
- `python/sglang/srt/layers/quantization/*`（量化方法注册、实现）  
- `python/sglang/srt/model_loader/loader.py`（ModelOpt 加载路径）  
- `python/sglang/srt/server_args.py`（CLI 参数校验、MoE 兼容）  
- 测试套件 `test/registered/model_loading/test_modelopt_loader.py`  

**💡 关注建议**  

1. **兼容性验证**：`ModelOptMixedPrecisionConfig` 只在已经预‑量化的 checkpoint 上生效（`_get_modelopt_quant_type` 抛异常），建议在文档中明确说明，避免用户误把普通 checkpoint 当作 mixed‑precision 使用。  
2. **硬件能力检查**：`get_min_capability` 直接复用 `ModelOptFp4Config`（返回 100），若未来加入新算子需同步更新，否则可能在低端 GPU 上导致意外运行错误。  
3. **KV‑Cache 量化**：当前仅在 `kv_cache_quant_algo` 为 FP8 时返回 `ModelOptFp8KVCacheMethod`，若 mixed‑precision 包含 NVFP4 KV‑cache，需在后续补齐对应实现。  
4. **MOE‑kernel 支持**：`_handle_moe_kernel_config` 已将 `modelopt_mixed` 加入白名单，但实际 MoE 量化实现仍仅覆盖 FP8、NVFP4 两种子配置。请在 Release Note 中提醒用户：混合精度 MoE 仍受限于子层的量化类型。  
5. **日志与错误信息**：在 `ModelConfig._get_modelopt_quant_type` 抛出的 `"modelopt_mixed is only supported for pre‑quantized checkpoints."` 建议换成更友好的提示，指明检查点必须包含 `quantized_layers`。  
6. **测试覆盖**：现有单元测试覆盖了解析与映射逻辑，建议在 CI 中加入一次端到端加载（`ModelOptModelLoader`）以及推理路径的验证，防止新量化分支在实际运行时出现未捕获的异常。  

总体来看，改动实现了 Nemotron‑3 Mixed‑Precision 的全链路支持，结构清晰且与已有 ModelOpt 体系保持一致。只需在文档、硬件检查以及 MoE‑KV‑Cache 的细节上做好提示，即可平滑交付。

### [PP] Fix recv tensor dict potential race condition (#20341)
**SHA**: `99a3b25` | 🔗 [查看提交](https://github.com/sgl-project/sglang/commit/99a3b25c9bbe5367a0ac51aeddd2af04f21bf155)

**🎯 变更类型**：Bug 修复（针对 PP (pipeline parallel) 中 `recv` TensorDict 可能出现的竞争条件）  
**⚡ 重要程度**：🟡 中  

**📋 变更摘要**  
1. 为 PP 通信加入显式的消息类型标识 `__msg_type__`，在发送时通过 `msg_type` 参数区分 “proxy” 与 “output”。

3. 所有发送路径统一改为 `_pp_send_dict_to_next_stage(..., msg_type=…)`，并在首次使用默认类型时给出一次性警告。  

**🎯 影响范围**  
- `python/sglang/srt/managers/scheduler_pp_mixin.py`（核心调度器的 PP 通信逻辑）。  
- 通过 `self.pp_group.send_tensor_dict/recv_tensor_dict` 的所有 pipeline‑parallel 代码路径（prefill、decode、disagg 等）。  

**💡 关注建议**  
- **兼容性**：已有的运行实例若仍使用默认 `msg_type`，会触发一次性 warning，但功能不受影响。建议在自定义插件或外部调用 `send_tensor_dict` 时显式指定 `msg_type`，防止以后出现遗漏。  
- **性能**：增加了 deque 的入队/出队操作以及一次性日志打印，对整体吞吐影响微乎其微，但在极高并发的多节点实验中请关注 `self._pp_tensor_dict_inbox` 的增长情况，必要时在空闲时清理旧键。  
- **调试**：`logger.debug` 会记录“stashing” 信息，开启 DEBUG 可帮助定位仍然被错误分类的消息。  
- **测试**：建议在 CI 中加入多 GPU / 多 TP/PP 场景的并发发送接收测试，验证在高负载下不会出现 “recv_tensor_dict missing” 的错误。  

总体而言，此次改动通过消息类型划分与收件箱缓冲，消除了 pipeline 并行阶段不同消息相互遮盖的 race 条件，提升了系统的鲁棒性。
