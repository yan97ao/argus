### [refactor] Config resolution pipeline: full-stack review (10-PR series, review only) (#30137)
**SHA**: `8fb99bb` | 🔗 [查看提交](https://github.com/sgl-project/sglang/commit/8fb99bbaf8dc3421cdc8188fddcb0082c3c0c5e7)

**🎯 变更类型**：重构 / 架构变更（Config resolution pipeline 全栈重构）  
**⚡ 重要程度**：🔴高  

**📋 变更摘要**  
本次 PR 对 SGLang 的配置解析实现进行了一次全局性重构，核心目标是 **把所有模型‑、硬件‑、运行时默认值的写入从 `ServerArgs` 的命令行/imperative 代码中抽离**，统一走 **“Override + Post‑process Pass”** 流程：

1. 新增 `runtime_context` 与 `Flags`（静态标记层）单例，实现 **分层（ServerArgs → Flags → Capture）**、**懒加载/即时解析**、以及 **冻结机制**（`freeze_flags()`）。  
2. `arg_groups/overrides.py` 成为 **统一的声明/后处理注册中心**，通过 `run_post_process_pass` 按顺序执行一系列 `@register_post_process` 的函数，实现 **模型特定默认值、互斥检查、自动开启/关闭功能**（如 FlashInfer AllReduce Fusion、CuteDSL MLA、DSA KV‑cache dtype、Mamba radix cache 等）。  
3. 大量原来的 `ServerArgs._set_default_*`、`*_hook.py`、`*_model.py` 中的默认写入与校验被迁移到上述 Override‑Pass 中，并保持 **legacy slot**（仍在旧位置调用 `run_post_process_pass`，保证兼容）。  
4. 计数、日志与警告仍保持原有行为，只是实现位置改变。  
5. 为了兼容现有单测与 CI，新增 `STACK_REVIEW_PLACEHOLDER.md` 以及大量单元测试的适配（改为读取 `get_flags()`、使用 `reset_context()` 等）。  

**🎯 影响范围**  

| 受影响模块/组件 | 主要改动点 |
|----------------|-----------|
| `sglang.srt.runtime_context` | 新增 `RuntimeContext`、`Flags`、`CaptureFlags`、全局单例 `_CONTEXT`、`freeze_flags()`、`record_runtime_overrides()`、`reset_context()` |
| `sglang.srt.arg_groups.overrides.py` | 注册所有模型、硬件、特性相关的后处理函数（DeepSeek‑V4、Nemotron‑H、MOSS‑VL、Mamba‑radix、DSA、FlashInfer Fusion、CuteDSL、Deterministic、Pipeline‑parallel 等） |
| 每个模型实现（`deepseek_v2.py`、`nemotron_h.py`、`qwen3_5.py`、`glm4_moe.py` 等） | 大量 `get_global_server_args().xxx` 替换为 `get_flags().xxx`，模型层构造时使用 `get_flags()` 读取解析后的值 |
| `arg_groups/*_hook.py`、`model_runner.py`、`cpu_graph_runner.py`、`*_cuda_graph_runner.py` | 删除/简化旧的默认填充逻辑，改为调用 `run_post_process_pass`，并使用 `get_flags().capture.enable_torch_compile` 替代 `server_args.enable_torch_compile` |
| 单元/集成测试 | 通过 `reset_context()`、`get_flags().override()` 进行 flag 注入，更新了大量测试中对 `ServerArgs` 的直接访问路径 |
| `srt/scheduler.py` | 在 Scheduler 初始化完成后调用 `get_context().freeze_flags()`，标记配置阶段结束 |  

**🔍 技术洞察**  

| 维度 | 分析 |
|------|------|
| **架构影响** | - 引入 **三层配置模型**（`ServerArgs` → `Flags` → `CaptureFlags`）实现 **“发布‑解析‑运行时”** 的职责分离，能让 **CLI 参数、模型加载时的权重决定、运行时动态调优** 在不同阶段独立完成。<br>- `Overrides` 注册机制相当于 **插件式后处理**，新模型或新特性只需要在该文件中注册函数即可，无需改动核心解析代码。<br>- 通过 `RuntimeContext._runtime_overrides` 保存 **双写（dual‑apply）声明**，确保 **load‑time** 与 **run‑time** 两阶段的声明语义一致。 |
| **性能影响** | - **Torch‑compile 开关** 现已在 `flags.capture.enable_torch_compile`（capture 阶段）保存，避免在每个模型实例化时重复读取 `ServerArgs`，略微提升启动速度。<br>- 统一的后处理 Pass 采用 **一次性遍历**（`run_post_process_pass`）并把声明直接写入 `flags`，相较于散落在多个分支的 `if/else`，在 **模型加载阶段的分支判断次数几乎不变**，但代码路径更清晰，利于 JIT 编译器的常量折叠。<br>- 新增的 `freeze_flags()` 防止在 **服务运行期间** 再次写入静态标记，保证 **GPU‑side 代码生成**（如 Triton、FlashInfer）在首次捕获后保持不变，避免因后期改写导致的 **graph invalidation**。<br>- 对 **DSA、Mamba、FlashInfer Fusion** 的默认填充仍在 **模型加载前** 完成，运行时不产生额外开销。 |
| **安全/可靠性** | - 没有新增网络或文件系统交互，安全面影响基本为 **0**。<br>- **风险点** 在于 `record_runtime_overrides()`、`freeze_flags()` 的错误使用可能导致 **配置冻结后仍尝试修改**，抛出 `RuntimeError`，从而导致服务启动失败（可通过 CI 捕获）。<br>- 新的 `declare_load_time_override` 采用 **dual‑apply**，如果实现不一致（服务器端写入和 flag 写入不匹配）会触发 `assert_flag_parity`，在 CI 中已经加入覆盖，生产环境需要确保所有新模型都走 **Overrides** 而非旧的 imperative 写入路径。 |
| **可维护性** | - **统一入口**（`overrides.py`）大幅降低了散落在各模型/hook 中的默认值分散度，后续新特性只需在同一文件新增 `@register_post_process` 即可。<br>- `Flags` 使用 **dataclass** + **_StaticFlags**，字段声明统一带有 `resolvable=True` 标记，自动生成 **whitelist**，防止误写未声明的字段。<br>- 通过 `reset_context()` 在单元测试中可以 **彻底恢复单例**，保证测试之间的隔离性。<br>- 迁移工作量大，涉及 **2000+ 行新增 / 600‑ 行删除**。若遗漏某些旧的 imperative 写入（如在自定义模型/外部插件中直接修改 `ServerArgs`），可能导致 **配置不一致**，需要在 CI 中加入对 `ServerArgs._resolved_overrides` 是否为空的检查。 |
| **兼容性** | - 仍保留了 `get_global_server_args()` 旧接口（很多 legacy 代码仍在调用），但内部大多数读取改为 `get_flags()`。因此 **向后兼容**，但 **外部插件** 若直接访问 `ServerArgs` 并期望其字段在运行时实时生效，需改为 **读取 `get_flags()`** 或使用 `declare_load_time_override`。<br>- `ServerArgs` 仍保留旧的 CLI 参数，仅在 **解析阶段**一次性填充到 `Flags`，因此 **命令行**、**Docker 环境变量** 仍可使用。 |
| **测试覆盖** | - 新增 `test_legacy_global_ratchet.py` 检查 `get_global_server_args` 调用量下降；<br>- `test_runtime_context.py` 对 **publish‑resolve‑record‑freeze** 全流程做了完整单元测试；<br>- 多数模型的 `determine_num_fused_shared_experts`、`_deepseek_v4_*` 等分支已迁移测试，覆盖率提升。 |

### [diffusion] feat: add LingBot realtime prompt, KV window, and lazy VAE controls (#30040)
**SHA**: `5e6f49c` | 🔗 [查看提交](https://github.com/sgl-project/sglang/commit/5e6f49c986b5e629a272caeb6963b088fdcf69b0)

**🎯 变更类型**：功能增强、性能优化、重构、架构变更、可配置化  

**⚡ 重要程度**：🔴高  

**📋 变更摘要**  
1. 为 LingBot‑World 引入实时交互能力：支持**prompt 动态更新**、**复合输入事件**（prompt + camera_actions）以及**相机动作的状态/脚本两种模式**。  
2. 新增 **交互式 KV 窗口**（interactive KV window）与 **最近窗口采样** 参数，使得在实时推理时可以仅保留关键帧的 KV，显著降低显存与计算开销。  
3. 引入 **懒惰 VAE 编码**（lazy VAE encode）可跳过黑帧的 VAE 编码，进一步加速前置图像处理。  
4. 在配置、环境变量、常量、缓存键等层面统一抽象，完成大量模块化重构（`constants.py`、懒加载 `__getattr__`、统一 `envs`）。  

**🎯 影响范围**  
- `sglang/multimodal_gen/configs/pipeline_configs/lingbot_world.py`（模型配置）  
- `sglang/multimodal_gen/envs.py`（新增环境变量）  
- `sglang/multimodal_gen/runtime/entrypoints/openai/realtime/adapters/lingbot_world_realtime_adapter.py`（实时适配器）  
- KV‑Cache 实现：`sglang/multimodal_gen/runtime/layers/kvcache/causal_attention_cache.py`（最近窗口逻辑）  
- 管线核心：`.../pipelines_core/*`（新增请求字段、缓存策略、调度）  
- LingBot‑World 专属阶段：`.../stages/model_specific_stages/lingbot_world/*`（缓存键、交互 KV 窗口、跨注意力缓存重置）  
- 单元测试与部分测试代码更新。  

**🔍 技术洞察**  

- **架构影响**  
  - **模块化增强**：通过 `constants.py` 统一管理缓存键 & 条件键，降低硬编码风险；`__getattr__` 延迟加载阶段类，减小启动时依赖加载成本。  
  - **实时管线扩展**：在 `Req` 中新增 `realtime_causal_kv_sample_tokens`，以及对应的调度逻辑，使得实时推理的 KV 大小可动态调节。  
  - **交互 KV 窗口**：在 `LingBotWorldCausalDMDDenoisingStage` 中加入一套“交互式 KV 窗口”状态机（still‑chunk、moving‑window、still‑window），并通过 `RealtimeCausalDiTState.runtime_cache` 保存跨块状态，实现 **动态 KV 大小自适应**。  
  - **懒惰 VAE**：在 `LingBotWorldCausalDMDConfig.preprocess_vae_encode` 中，根据环境变量或配置裁剪待编码帧，避免对大量黑帧进行昂贵的 VAE 编码。  

- **性能影响**  
  - **显存**：最近窗口采样 (`recent_window_tokens`) 只保留 sink + 最近 N tokens，显著降低 KV 长度；在测试 `test_lingbot_realtime_attention_cache_samples_sink_and_recent_window` 中可见 KV 规模从 8 → 6（约 25% 减少）。  
  - **计算**：交互 KV 窗口根据相机动作自动切换 “moving” 与 “still” 模式，静止阶段仅保留少量 KV（例如 `still_window=3`），可把每帧的注意力计算量从 `O(window)` 降至 `O(sink+still)`。  
  - **前置处理**：懒惰 VAE 对黑帧的裁剪可把 VAE 前向次数减少至 `1 + lazy_black_frames`，在长序列视频中可降低 30‑50% 的前置耗时。  
  - **缓存分配**：`_initialize_kv_cache` 现在接受 `kv_cache_size` 参数，避免二次计算并能一次性分配正确大小的 KV，提升启动时的分配效率。  

- **安全考虑**  
  - **环境变量注入**：新增 `SGLANG_LINGBOT_ENABLE_INTERACTIVE_KV_WINDOW`、`SGLANG_LINGBOT_LAZY_VAE_ENCODE_BLACK_FRAMES` 均通过 `envs` 的 `_lazy_int/_lazy_bool` 读取，默认 **None**，在未显式配置时不会导致意外行为。  
  - **事件解析**：复合输入事件的解析加入严格校验（检查 `input_types`、字段存在性、类型正确性），并在验证失败时抛出 `ValueError`，防止恶意或格式错误的 payload 进入内部状态。  
  - **KV 范围控制**：`recent_window_tokens` 参数在 `_visible_attention_kv` 中强制非负、并对负值直接报错，避免出现非法切片导致内存泄漏或越界读取。  

**⚠️ 潜在风险**  

| 风险点 | 可能影响 | 缓解措施 |
|--------|----------|----------|

| **旧版代码兼容性**：为防止破坏已有非 LingBot 场景的流水线，新增字段与缓存键使用了前缀 `lingbot_`，但已有代码可能仍通过硬编码字符串访问 `extra["lingbot_cam_conditioner"]`。 | 运行时 `KeyError` 或缓存
