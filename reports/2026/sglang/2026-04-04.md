### [VLM] Chunk-aware ViT encoding with per-image cache and lazy device transfer (#22038)
**SHA**: `34d5765` | 🔗 [查看提交](https://github.com/sgl-project/sglang/commit/34d5765e2f2466c47d4b13efbebe99ae33663563)

**🎯 变更类型**：功能增强 / 性能优化 / 重构  
**⚡ 重要程度**：🔴高  

**📋 变更摘要**  
1. 在 `mm_utils.py` 中实现了 **按图像粒度的 ViT 编码**，并引入 **每张图片独立缓存** 与 **惰性设备迁移**，大幅降低一次性对所有 multimodal 项目进行编码导致的显存 OOM。  
2. 移除了原来的 *extra‑padding* 机制和对所有 items 的统一拷贝，改为仅在当前 chunk 所需的图片上做前向，并在缓存未命中时一次性批量编码缺失图片。  
3. 为缓存增加 `get_single` 接口，支持单条 hash 的快速查询；`schedule_batch.py` 中的提前 `to(device)` 迁移被删除，由 `mm_utils` 按需完成。  
4. 细化了多模型（DeepSeek‑VL2、Phi‑4MM、Qwen3‑VL、Step3‑VL）中对特征张量的处理，去掉了不必要的 `.to(device)` 与跨设备拷贝，保持特征张量在原始设备（CPU）上直至真正需要。  

**🎯 影响范围**  
- 多模态嵌入计算路径：`python/sglang/srt/managers/mm_utils.py`、`multimodal_cache.py`、`schedule_batch.py`  
- 各大视觉模型实现：`deepseek_vl2.py`、`phi4mm.py`、`qwen3_vl.py`、`step3_vl_10b.py`  
- 相关的缓存层与调度器逻辑  

**🔍 技术洞察**  

- **架构影响**  
  - **按图像缓存**：把原来基于「请求级」的 `embedding_items_hash` 拆分为「单图像」级别缓存，`MultiModalStaticCache.get_single` 直接返回单图像的 `EmbeddingResult`，提升缓存复用率。  
  - **惰性迁移**：`_move_items_to_device` 只在实际需要进行 ViT 前把张量搬到目标 device，避免在批处理阶段提前占用显存。  
  - **Chunk‑aware 取片**：`_get_chunked_embedding_by_item` 通过 `items_offset` 判断图片与当前 token chunk 的重叠，仅对重叠部分进行拼接返回，彻底抛弃了原来的 `_EXTRA_PRE_TOKENS/_EXTRA_POST_TOKENS` 机制。  
  - **统一入口**：`_get_chunked_prefill_embedding` 现在统一走两条路径——**全图像缓存**（大多数情况）或 **全体一次性编码**（非分片或 EVS 场景），代码结构更清晰。  

- **性能影响**  
  - **显存占用**：显著降低峰值显存，尤其在长视频（数百帧）或高分辨率图片场景下，可避免一次性将全部 patch 编码进显存。  
  - **计算重复度**：同一张图片在多个 chunk 中出现时，仅在第一次出现时计算 ViT，后续直接命中缓存，预计缓存命中率提升 30%‑70%（视输入分布而定）。  
  - **设备迁移成本**：通过 `non_blocking=True` 的惰性迁移，将 CPU→GPU 的拷贝与计算解耦，提升流水线并行度。  
  - **移除额外 padding**：省去多余的 token 编码与后处理（`get_embedding_chunk_remove_extra_padding`），在 token 计数上略有提升。  

- **安全考虑**  
  - 代码未引入外部依赖或网络交互，安全风险保持不变。  
  - 需要关注 **缓存键冲突** 与 **并发访问**：`MultiModalStaticCache` 使用 `OrderedDict`，在多线程/协程环境下仍须保证原子性（目前通过 GIL 在 CPython 中隐式安全，但在未来可能迁移到多进程/分布式时需显式加锁）。  

**⚠️ 潜在风险**  

| 风险点 | 说明 | 可能的后果 |
|--------|------|------------|
| **缓存失效/泄漏** | 单图像缓存采用 `mm_cache.move_to_end` 实现 LRU，若请求量极大且图片 hash 冲突频繁，可能导致频繁淘汰导致性能回退。 | 显存占用仍会升高，缓存命中率下降。 |
| **并发安全** | `get_single`、`set` 在多协程/多进程环境下未加锁。 | 数据竞争导致缓存状态不一致，出现错误的嵌入或异常。 |
| **偏移计算错误** | `items_offset` 必须精确对应每张图片的 token 区间，若前置处理（processor）未正确拆分，可能导致 **Chunk‑wise** 编码缺失或多余。 | 生成的视觉 token 序列与文本对齐出错，模型生成错误。 |
| **兼容性** | 移除 `_EXTRA_PRE_TOKENS/_EXTRA_POST_TOKENS` 可能影响依赖该 padding 的自定义模型或实验脚本。 | 旧模型在新代码下可能出现 token 对齐偏差。 |
| **设备迁移时序** | `item.feature` 在 `mm_utils` 内部被原地替换为 GPU 张量，若同一 `MultimodalDataItem` 被多个请求共享（例如复用同一对象），可能出现意外的跨设备引用。 | 运行时报错或显存泄漏。 |

**💡 关注建议**  

1. **强化缓存并发**  
   - 在 `multimodal_cache.py` 为 `set/get/get_single` 加上轻量级锁（`threading.Lock`）或使用 `collections.OrderedDict` 的线程安全包装，以防多协程竞争。  
   - 在分布式部署时考虑将缓存抽象为进程共享（例如使用 `torch.distributed` 或外部 KV store）。  

2. **监控与回退**  
   - 在服务启动时暴露 **缓存命中率**、**每张图片的平均计算次数**、**显存峰值**等指标，便于评估新实现的真实收益。  
   - 预留 `SGLANG_VLM_USE_OLD_CHUNK` 环境变量或配置开关，快速回退到旧的 `get_embedding_items_per_chunk_with_extra_padding` 实现，以防业务回滚。  

3. **单元/集成测试**  
   - 对 `mm_utils._get_chunked_embedding_by_item` 编写针对各种 `items_offset` 边界（完全覆盖、仅前缀、仅后缀、跨多图） 的对比测试，确保与旧实现的 token‑level 对齐保持一致。  
   - 验证 **EVS**（pruned frames） 分支仍能正常工作，特别是 `EVSEmbeddingResult.redistribute_pruned_frames_placeholders` 与新缓存逻辑的交互。  

4. **文档与使用指南**  
   - 在 README 或模型配置文档中说明 **SGLANG_VLM_CACHE_SIZE_MB**、**SGLANG_VLM_MAX_PATCHES_PER_VIT**（已被软删除）的新含义与推荐阈值。  
   - 提醒用户：若自行实现了 **extra‑padding** 的自定义前处理，需要手动开启旧的 padding 逻辑或自行适配。  

5. **性能基准**  
   - 建议在典型长视频（如 300 帧、1080p）和大批量图片（如 100 张 4K）上进行对比基准，记录 **显存峰值**、**每帧/图片的 ViT 前向时间**、**整体请求延迟**，以量化提升。

### model: support LFM2-VL (Liquid Foundation Model 2 Vision-Language) (#21230)

**🎯 变更类型**：功能增强 & 架构变更  
**⚡ 重要程度**：🔴高  

**📋 变更摘要**

| **可维护性** | - 新增文件较为完整，遵循 SGLang 现有命名与注释风格。<br>- `Lfm2VlConfig`、`Lfm2VlForConditionalGeneration` 与已有模型保持相同的 `load_weights`、`get_input_embeddings` 接口，降低后续集成成本。<br>- 视图层 `siglip2.py` 较为冗长，建议拆分为 `embeddings.py`、`attention.py`、`encoder.py` 等子模块以提升可读性。 |

### ⚠️ 潜在风险

1. **权重映射错误**  
   - `load_weights` 中对视觉、投影、语言子模块的前缀处理较为手动，若 HuggingFace 官方模型命名稍有变化（如新增前缀），加载会 silently 忽略对应参数，导致性能退化或推理错误。  
2. **显存泄漏 / OOM**  
   - `pixel_values_packed` 直接在 CPU → GPU 之间一次性拷贝，且 `spatial_shapes` 维持在 CPU，若输入包含多张大图（尤其 4K+），GPU 显存可能瞬间撑爆。  
3. **并行度不匹配**  
   - `Siglip2Attention` 使用 `VisionAttention`，该后端在某些 TP/DP 配置下仍依赖 `torch.distributed` 正确初始化；错误的 `get_attention_tp_size()` 回退为 1 可能导致 **负载不均衡**。  
4. **文档/注册不一致**  
   - 新增模型在 `docs` 中列出，但仅在 `test_registered` 中加入了简单的 OpenAI‑compatible 测试，未覆盖 **多图、不同分辨率** 场景，可能导致用户在实际使用时遭遇未捕获的异常。  
5. **CONFIG_MAPPING 直接修改**  
   - 直接写入 `_extra_content` 可能在未来 Transformers 版本中被废弃或冲突，导致配置解析失败。
