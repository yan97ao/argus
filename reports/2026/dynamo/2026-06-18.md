### feat(protocols): move nvext handling to llm (#10784)
**SHA**: `6fb0d6e` | 🔗 [查看提交](https://github.com/ai-dynamo/dynamo/commit/6fb0d6e0da43574e5f17bf031ec7c24165407350)

**🎯 变更类型**：功能增强 / 架构迁移 / 重构  
**⚡ 重要程度**：🔴 高  
**📋 变更摘要**  
本次提交将 **nvext**（Dynamo 扩展字段）的处理从各协议实现（OpenAI、Anthropic 等）迁移到 **LLM 通用层** `protocols::common::extensions` 中。新增统一的 `NvExt` 结构体、解析、验证、路由约束转换、Header‑Routing 覆写、以及响应字段选择逻辑。所有协议实现统一使用 `NvExtProvider`，并在前端入口、预处理器、路由、以及响应聚合处统一调用。文档、测试、以及部分内部模块也同步更新，以支持新的统一实现。

---

### 🎯 影响范围
| 受影响模块 / 组件 | 说明 |
|-------------------|------|
| `lib/llm/src/protocols/common/extensions.rs` | 新增 `NvExt`、`AgentHints`、`AgentContext`、`RoutingConstraints`、`RouterParams`、`MetadataUpload`、验证函数、Header‑Routing 逻辑、`merge_response_nvext` 等。 |
| 各协议实现 (`openai`, `anthropic`) | 改为 `use dynamo_llm::protocols::common::extensions::*`，调用统一的 `NvExt` 与 `NvExtProvider`。 |
| 预处理器 (`preprocessor.rs`) | 引入 `routing_constraints_to_kv`，将 `nvext.routing_constraints` 转换为 KV‑router 所需结构。 |
| HTTP 服务层 (`service_v2.rs`, `openai.rs`, `anthropic.rs`) | 在请求入口统一 `gate_*_nvext`、`apply_*_header_routing_overrides`、`validate_*_nvext`，并在响应聚合中使用统一的 `NvExtResponseFieldSelection::build_response_nvext`。 |
| 文档 (`docs/components/frontend/configuration.md`) | 更新 `DYN_ENABLE_FRONTEND_NVEXT` 环境变量说明，增加对 `/v1/messages` 的影响。 |
| 单元测试 | 大量新增/迁移测试，确保 nvext 解析、验证、路由覆写、响应字段选择在各协议下均保持行为一致。 |
| 其他模块 (`kv_router`, `request_trace`, `protocols/tensor.rs`, `unified.rs`) | 迁移对应的 `use` 路径，保持编译通过。 |

### 🔍 技术洞察

#### 1. 架构影响
| 维度 | 影响 |
|------|------|
| **统一扩展模型** | 将所有协议共用的扩展字段抽象到 `common::extensions`，实现 **一次定义，多处复用**。这降低了不同协议实现间的代码重复度，提升了可维护性。 |
| **职责划分** | 原先每个协议自行实现 `nvext` 的解析、校验、路由覆写逻辑，导致语义分散且易产生不一致。迁移后这些职责集中到 LLM 通用层，协议层只负责**转换**（`TryFrom`）和**调用**，更符合单一职责原则。 |
| **模块依赖** | 通过 `use dynamo_llm::protocols::common::extensions::*`，消除 `openai::nvext` 与 `anthropic::nvext` 的循环依赖，使得 **协议层不再依赖 LL**（LLM 核心）内部实现，仅通过公开的结构体交互。 |
| **环境变量镜像** | `DYN_ENABLE_FRONTEND_NVEXT` 的语义仍在 `HttpService` 中映射，但实现已统一到 `state.nvext_enabled()`，简化了配置路径。 |
| **路由约束** | `routing_constraints` 现在是 `extensions::RoutingConstraints`，并通过 `routing_constraints_to_kv` 转换为 `dynamo_kv_router::protocols::RoutingConstraints`。这把 **业务约束** 与 **底层路由实现** 明确分离。 |
| **响应字段选择** | `NvExtResponseFieldSelection::build_response_nvext` 现在位于 `extensions`，统一处理 `worker_id`、`token_ids`、`timing`、`routed_experts`、`engine_data`、`stop_reason`、`completion_token_ids`、`prompt_logprobs` 的生成与聚合。 |
| **向后兼容** | 保持 `NvExt` 的 `serde` 结构不变（仍 `deny_unknown_fields`），但新增 `#[builder]` 便于构造；旧的 `nvext` JSON 仍可被 `parse_nvext` 正确解析，确保已有客户不受影响。 |

#### 2. 性能影响
| 维度 | 影响 |
|------|------|
| **请求路径额外开销** | 统一的 `parse_nvext`、`validate_nvext_semantics`、`apply_header_routing_overrides` 在请求入口额外执行一次 JSON 解析和字段校验。对大流量服务来说，额外的 `serde_json` 解析成本在 **毫秒级**，但已在 **builder** 中使用 `Option` 惰性创建，且仅在 `nvext` 存在时触发，影响可接受。 |
| **路由阶段** | `routing_constraints_to_kv` 是一次浅拷贝（集合搬移），开销极低；对 KV‑router 的调用保持不变。 |
| **响应聚合** | `merge_response_nvext` 通过对象合并避免重复拷贝，仅在需要合并 `completion_token_ids` 与 `prompt_logprobs` 时进行少量内存拷贝。整体吞吐不受负面影响。 |
| **缓存/预处理** | `NvExt` 中的 `token_data`、`metadata_upload` 等字段仍在原位使用，未改变缓存行为。 |

#### 3. 安全考虑
| 维度 | 影响 |
|------|------|
| **输入验证** | 新增 `validate_nvext_semantics` 对 `agent_context`、`session_control` 中的必填字符串执行 **非空校验**。防止恶意提交空字符串导致后端逻辑异常或 KV 隔离失效。 |
| **未知字段** | 仍保留 `#[serde(deny_unknown_fields)]`，任何未知字段都会导致解析错误，防止潜在的 **反序列化漏洞**（例如隐蔽的语义注入）。 |
| **Header 覆写** | `apply_header_routing_overrides` 仅在 `nvext_enabled` 为 true 时采纳 HTTP Header，且对每个字段都有显式的 `parse::<u64>()`/`parse::<u32>()` 检查，防止 **HTTP 注入** 或 **类型混淆**。 |
| **环境变量关闭** | 当 `DYN_ENABLE_FRONTEND_NVEXT=0` 时，所有 `nvext` 会在入口被 **dropped** 并记录警告，避免泄露内部路由信息。 |
| **元数据上传** | `MetadataUpload.url` 通过自定义反序列化函数强制非空；仍需业务层对 URL 做安全检查（如禁止 `file://`），但所做校验已降低 **目录遍历** 风险。 |
| **同步/并发** | 所有新增函数为 **纯函数**（不持有全局可变状态），不会引入 race 条件。 |

#### 4. 可维护性 & 可测试性
- **统一 builder**：`NvExt::builder()` 与 `AgentContext::builder()` 为结构体提供强类型构造，降低手写 JSON 出错概率。  
- **测试覆盖**：大量单元测试覆盖：解析、注释检测、Header 覆写、验证、响应字段选择、协议转换、序列化错误等，提升回归安全。  
- **文档同步**：`docs/components/frontend/configuration.md` 已更新，避免用户因文档滞后导致配置错误。

### feat(kv-indexer): add access log and log reopen support (#10700)
**SHA**: `298f1e0` | 🔗 [查看提交](https://github.com/ai-dynamo/dynamo/commit/298f1e00b87e55ac4969dbf93513d1ced88a9375)

**🎯 变更类型**：功能增强  
**⚡ 重要程度**：🔴高  
**📋 变更摘要**：在 KV‑Indexer 的 standalone HTTP 模式中新增可写访问日志功能，支持 JSON‑lines 日志、可配置的 trace‑id Header、UTC/本地时间切换，以及运行时日志文件 reopen（日志轮转）接口。为实现该特性，引入 `chrono`、`tracing-appender`、`tempfile` 与 `symlink` 依赖，新增 CLI 参数并在服务端状态中持有 `AccessLogSink`，通过 Axum 中间件统一记录请求信息。

**🎯 影响范围**：  
- `lib/kv-router`（核心 KV‑Indexer HTTP 服务）  
- `lib/bindings/python`（Python 绑定的 Cargo.lock 同步更新）  
- 构建依赖：新增 `chrono`、`tracing-appender`、`tempfile`、`symlink`  
- 公开 CLI 参数：`--access-log`、`--trace-id-header`、`--access-log-local-time`  

---

### 🔍 技术洞察

| 维度 | 影响描述 |
|------|----------|
| **架构影响** | <ul><li>在 `AppState` 中加入 `access_log_sink: Option<Arc<AccessLogSink>>`，保持最小侵入性，原有业务路径不受影响。</li><li>通过 Axum 中间件 `access_log_middleware` 注入日志记录，统一在请求入口/出口完成记录，保持 **单一职责**。</li><li>新增 `/reopen_logs` 管理端点，提供运行时日志轮转能力，配合外部 logrotate 脚本使用。</li><li>CLI 参数解析后直接注入 `IndexerConfig`，保持配置统一入口。</li></ul> |
| **性能影响** | <ul><li>日志写入使用 `tracing_appender::non_blocking::NonBlocking`，写入在独立的后台线程完成，几乎不阻塞业务请求。</li><li>在 `AccessLogSink::write_line` 中若写入失败，仅记录警告且计数，避免因磁盘错误导致业务崩溃。</li><li>额外的 `Mutex`（`parking_lot::Mutex`）仅在日志写入和 reopen 时加锁，冲突概率极低。</li><li>开启日志会产生额外的 JSON 序列化成本，约 1‑2 µs/条，整体对吞吐影响可忽略。</li></ul> |
| **安全考虑** | <ul><li>日志中记录的 `trace_id` 来自可配置的 Header，默认 `x-trace-id`，若上游未清理可能泄露业务唯一标识，需在部署文档中提醒。</li><li>日志文件路径通过 CLI 直接指定，若误指向敏感目录会产生权限风险。建议在启动脚本中使用受限用户或 `chmod 0600`。</li><li>日志写入错误被捕获，不会导致 panic，提升容错。</li></ul> |
| **可维护性** | <ul><li>新增模块 `logging.rs` 完全自包含，提供 `AccessLogSink`、中间件、解析函数，易于单元测试（已包含测试）。</li><li>使用 `serde_json` 直接序列化结构体，日志格式固定为 JSON Lines，便于 downstream 分析。</li><li>对外接口（CLI 与 HTTP）保持向后兼容，未开启日志时行为不变。</li></ul> |
| **兼容性** | <ul><li>Feature `standalone-indexer` 增加 `chrono`、`tracing-appender` 可选依赖，默认不启用日志功能，已有用户无需改动。</li><li>在 `Cargo.toml` 中仅在 `standalone-indexer` 里加入新依赖，避免对其它子crate造成依赖膨胀。</li></ul> |

---

**⚠️ 潜在风险**

1. **磁盘空间耗尽**：开启访问日志后若未进行轮转，日志文件可能快速增长。  
2. **日志路径/权限错误**：启动时提供的路径不可写导致 `AccessLogSink::new` 失败，启动即报错。  
3. **Header 名称冲突**：用户自定义 `--trace-id-header` 若与业务已有 Header 重名，可能误记录不应公开的内容。  
4. **Reopen 失败**：在文件被外部删除或权限变更后调用 `/reopen_logs` 可能返回错误，需要上层监控该端点返回。  
5. **时间偏差**：使用本地时间 (`--access_log_local_time`) 时若机器时钟不准，日志时间会产生偏差，建议统一使用 UTC。  

---

**💡 关注建议**

| 对象 | 建议 |
|------|------|
| **运维 / 部署** | • 配置日志轮转（如 `logrotate`）并定期检查磁盘剩余空间。<br>• 使用受限用户运行服务，限制日志文件写权限。<br>• 如需本地时间，确保 NTP 同步。 |
| **开发者** | • 在单元测试或 CI 中加入 `access_log_sink` 为 `None` 的路径，以验证不影响业务。<br>• 若新增自定义 `trace-id-header`，在文档中说明 Header 必须符合 RFC 7230。 |
| **安全审计** | • 对写入的日志内容进行审计，确认不泄露敏感信息（如内部鉴权 token）。 |
| **用户** | • 默认不启用日志，只有在需要审计或调试时才加 `--access-log /path/to/file.log`。<br>• 通过 `/reopen_logs` 手动触发日志轮转，或配合外部脚本自动调用。 |

--- 

整体来看，此次改动为 KV‑Indexer 引入了可观测性增强，架构清晰、性能开销可接受，且通过特性门控保持向后兼容。只要在部署时做好日志轮转和权限管理，即可安全投入生产使用。

### feat(router): expose registered worker metric (#10587)
**SHA**: `b8f677e` | 🔗 [查看提交](https://github.com/ai-dynamo/dynamo/commit/b8f677e7a21e922361d1cffc1a827bf642ca42ea)

**🎯 变更类型**：功能增强  

**⚡ 重要程度**：🔴高  

**📋 变更摘要**：  
1. 在路由器（router）层新增 `router_worker_registered` 指标，用于暴露每个已注册的 worker/dp_rank 的存活状态（1=已注册）。  
2. 为 `SequencePublisher` 接口补充 `observe_worker_registered` 与 `observe_worker_removed` 回调，并在 `ActiveSequencesMultiWorker` 中于拓扑变更时触发。  
3. 引入 `RouterWorkerStatusMetrics` 结构体实现基于 `IntGaugeVec` 的组件级仪表，完成注册、注销的实际写入，并在 Prometheus 名称表、Python 绑定及单元测试中同步更新。  

**🎯 影响范围**：  
- `lib/kv-router/src/sequences/multi_worker.rs`（拓扑变更逻辑）  
- `lib/llm/src/kv_router/metrics.rs`（新增 `RouterWorkerStatusMetrics`）  
- `lib/llm/src/kv_router/sequence.rs`（RuntimeSequencePublisher 实现）  
- `lib/runtime/src/metrics/prometheus_names.rs` 与 Python 绑定 `prometheus_names.py`（新增常量）  
- 相关测试文件（验证注册/注销观察）  

**🔍 技术洞察**：  
- **架构影响**：  
  - 仅在观察层面增加了仪表收集，未改动路由核心调度算法。  
  - `RouterWorkerStatusMetrics` 采用 `Component::metrics()` 创建的 component‑scoped gauge，保持与现有 `MetricsHierarchy` 的标签体系一致（`dynamo_namespace`、`dynamo_component`），因此对整体架构影响微小。  
  - 新的回调使 `SequencePublisher` 与 worker 拓扑绑定得更紧密，提升了可观测性但略微增加了耦合度。  

- **性能影响**：  
  - 每次 worker 注册/注销会执行一次 `IntGaugeVec::with_label_values(...).set(1)` 或 `remove_label_values`，成本在微秒级，且拓扑变更本身就相对稀疏，整体性能影响可忽略。  
  - 额外的锁 (`Mutex` 用于测试状态) 仅在单元测试中出现，对生产环境无影响。  
  - 增加了仪表的标签组合数，需关注 Prometheus 的卡片基数，以防卡片爆炸。  

- **安全考虑**：  
  - 暴露的标签仅包含 `router_worker_id`（内部 worker ID）、`dp_rank`、`worker_type`，均为非敏感信息，不会导致信息泄露。  
  - 未引入额外的网络交互或跨组件调用，安全风险极低。  

**⚠️ 潜在风险**：  
- **卡片基数**：大量 worker/dp_rank 组合会导致 `router_worker_registered` 卡片数量线性增长，若集群规模极大需监控 Prometheus 的卡片上限。  
- **标签清理不完整**：若 worker 异常退出未触发 `unregister_worker`，对应 gauge 可能残留为 “1”，导致监控误报。  
- **初始化顺序**：在极端启动路径中，如果 `RouterWorkerStatusMetrics::from_component` 在 `Component` 尚未完成初始化前被调用，可能出现 `OnceLock` 未就绪的 panic。  
- **并发拓扑变更**：`apply_worker_topology_change` 在遍历 `added` 与 `removed` 时未加额外同步，若同一时间有并发变更，可能出现重复注册/删除调用，导致 gauge 状态不一致。  

**💡 关注建议**：  
1. **监控卡片基数**：在部署环境的 Prometheus 配置中设定合理的 `max_labels` 限制，并在监控仪表板上增加 `router_worker_registered` 卡片数量告警。  
2. **健全注销路径**：在 worker 失联检测（如心跳超时）中确保调用 `unregister_worker`，必要时在 `worker_monitor` 中加入强制清理逻辑。  
3. **初始化防护**：为 `RouterWorkerStatusMetrics::from_component` 添加容错日志，确认 `Component` 已完成 metric 注册后再实例化。  
4. **并发安全**：如果未来支持并发拓扑变更，考虑在 `apply_worker_topology_change` 外部加 `RwLock` 或使用事务化的拓扑更新，以避免重复的 `observe_worker_*` 调用。  
5. **文档同步**：在项目的可观测性章节补充 `router_worker_registered` 指标的含义、标签解释以及最佳实践，帮助运维团队快速定位 worker 生命周期问题。

### feat(bindings): expose SelectionService selection core as a Python class (#10766)
**SHA**: `8aa43af` | 🔗 [查看提交](https://github.com/ai-dynamo/dynamo/commit/8aa43afceefda293a51c7c25f1303f51244450c4)

**🎯 变更类型**：功能增强 / 重构  
**⚡ 重要程度**：🔴 高  

**📋 变更摘要**  
本次提交在 `dynamo` 项目的 Python 绑定层新增了 `SelectionService` 类，向 Python 使用者暴露了原本仅在 Rust 端运行的 KV‑selection 核心（`SelectionCore`）。同时对内部实现加入了基于 `tokio_util::sync::CancellationToken` 的可取消任务体系，提供统一的 `shutdown` 接口以及在服务关闭时快速失败的能力。此外，错误类型 `SelectionError` 被扩展以返回机器可读的 `kind` 与 HTTP‑style `status_code`，并在 Python 侧映射为 `SelectionServiceError`。

---

### 🎯 影响范围
- **Python 绑定层**：`dynamo._core.SelectionService`、`SelectionServiceError`、对应的 `.pyi` 类型声明以及 `dynamo.llm` 包的导入。
- **Rust 业务层**：`kv-router/src/services/selection`（核心实现、错误结构、服务生命周期管理）。
- **KV‑Indexer**：`WorkerRegistry` 与 `AppState` 现在接受根级 `CancellationToken`，允许全局统一取消。
- **测试套件**：新增 `SelectionService` 相关单元/集成测试，以及 `WorkerRegistry` 的取消 token 单元测试。

---

### 🔍 技术洞察

| 维度 | 影响分析 |
|------|----------|
| **架构影响** | - 在 **Python‑Rust 边界** 增加了新的类 `SelectionService`，实现了 “in‑process” 调用而不需要启动独立的 HTTP 服务器；<br>- 引入根级 `CancellationToken`，实现 **层级取消**（根 → 子 token），对 KV‑Indexer 线程池、ZMQ 监听器以及调度器统一控制，提升资源回收的确定性。<br>- `SelectionCore.shutdown` 与 `SelectionService.__del__` 实现 **幂等** 关闭，符合 RAII 思想。 |
| **性能影响** | - 通过 `future_into_py` 将 Rust `async` 任务直接映射为 Python `awaitable`，额外的 GIL 获得与释放开销极小；<br>- 取消 token 的加入不会对正常路径产生显著额外成本，仅在关闭时触发 `cancelled()` 检查（相当于一次原子读）。<br>- `WorkerRegistry.start_pending` 与 `resume` 现在使用子 token，避免因根 token 失效后仍产生泄漏线程。 |
| **安全考虑** | - 新增的 `SelectionServiceError` 向上层暴露 `kind` 与 `status_code`，帮助调用方基于 **结构化错误** 做安全审计和容错。<br>- 错误信息仍然通过 `to_string()` 作为人类可读描述，未泄漏内部实现细节。<br>- `SelectionError::BadRequest` 仍映射为 Python 的 `ValueError`，保持对 **输入验证** 的明确分层。 |
| **可维护性** | - 通过 `SelectionCore.ensure_running` 统一检查服务是否已关闭，降低各业务入口忘记检查的风险。<br>- `WorkerRegistry` 新增 `listener_cancelled` 供测试使用，提升可验证性。<br>- 代码中大量 `#[cfg(feature = "select-service")]` 条件编译，保持向后兼容；若未启用该特性，Python 包仍可导入（只会 `ImportError`，不影响其他功能）。 |
| **兼容性** | - 仅在编译时开启 `select-service` Cargo feature 时才会提供 Python 类；已有用户若未开启此特性，`import SelectionService` 将得到 `ImportError`，不影响原有 API。<br>- 对现有 `DynamoException` 层级未作破坏性改动，保持向下兼容。 |

### ⚠️ 潜在风险
1. **特性启用不一致**：如果库在 Rust 侧开启 `select-service` 而 Python 包未同步更新（或相反），会导致运行时 `ImportError` 或缺失的异常类。需在发布说明中强调 Feature 与 Python 包版本对应关系。  
2. **取消 Token 竞争**：在极端并发场景下，`shutdown` 可能在已提交的 `select` 任务获取 `cancel_token` 之前触发，导致任务直接返回 `SubscriberShutdown`，这在业务层需要做好重试或回退处理。  
3. **资源泄漏风险**：`SelectionService` 的 `drop` 实现仅调用 `inner.shutdown()`，若用户手动 `shutdown` 后仍保留 `SelectionService` 引用且继续使用内部 `Arc<SelectionCore>`，可能出现 **使用已取消对象** 的错误。建议在文档中指出 `shutdown` 后对象不应再被使用。  
4. **异常映射**：`SelectionError::BadRequest` 被映射为纯 `ValueError`，而不是 `SelectionServiceError`，这可能导致调用方在 `except SelectionServiceError` 时遗漏输入错误。需要在文档中清晰说明两类错误的区别。  

---

### 💡 关注建议
- **文档更新**：在 Python SDK 文档中添加 `SelectionService` 使用示例、特性开启方式 (`pip install dynamo[select-service]` 类似），并说明 `shutdown` 的幂等性与后续不可再用的约束。  
- **错误处理指南**：提供示例代码展示如何捕获 `SelectionServiceError` 并依据 `exc.kind` / `exc.status_code` 进行业务分支（如重试、降级）。  
- **监控与日志**：在 `SelectionCore.shutdown` 与 `SelectionCore.ensure_running` 添加 `tracing::debug!` 日志，帮助运维在服务关闭时快速定位因取消导致的异常。  
- **测试覆盖**：考虑加入 **跨语言** 集成测试，验证 Python `await service.select(...)` 在 `service.shutdown()` 之后会抛出期望的 `SelectionServiceError`，以及在高并发下的 **取消安全性**。  
- **兼容性验证**：在 CI 中加入对不启用 `select-service` 特性的构建与运行测试，确保 `import SelectionService` 正常抛出 `ImportError`，而其它模块不受影响。  

--- 

**结论**：此变更为 `dynamo` 引入了强大的 **内嵌式 Selection Service**，显著提升了 Python 开发者的可操作性与错误可辨识度，同时通过层级取消机制增强了资源管理和服务可控性。只要在发布说明中明确特性依赖并提供正确的使用指南，整体风险可控，业务价值与架构可维护性均得到提升。

### fix(sglang): skip model prefetch for object storage paths (#10656)
**SHA**: `e1e6faa` | 🔗 [查看提交](https://github.com/ai-dynamo/dynamo/commit/e1e6faae12daa9ca6d1c993d3a3e6c93bcc43ca3)

**🎯 变更类型**：Bug修复  
**⚡ 重要程度**：🔴高  
**📋 变更摘要**：  
在 `sglang` 参数处理模块中，引入 `is_object_storage_path` 用于检测对象存储（S3、GCS、Azure）路径，并在 `should_fetch_model` 中加入判断，使得对这些远程路径不再执行本地模型预取。相应单元测试也被补全，确保新逻辑在各种对象存储前缀下均生效。  

**🎯 影响范围**：  
- `components/src/dynamo/sglang/args.py`（模型路径预取决策）  
- `components/src/dynamo/sglang/tests/test_sglang_unit.py`（新增测试）  
- 依赖 `sglang` 注册流程的所有运行时实例（尤其是使用 `runai_streamer` 加载方式时）  

**🔍 技术洞察**：  
- **架构影响**：  
  - 仅修改了 `sglang` 子模块的路径判定逻辑，对整体 Dynamo 架构影响有限。  
  - 引入的 `is_object_storage_path` 为后续扩展（如支持更多对象存储协议）提供了统一入口，提升了代码可维护性。  
- **性能影响**：  
  - 通过跳过对对象存储路径的本地文件 existence 检查和潜在的模型下载，显著降低启动时的网络 I/O 与延迟，尤其在大模型环境下可节省数十 GB 的流量和数分钟的启动时间。  
- **安全考虑**：  
  - 防止误将对象存储路径当作本地路径去 `os.path.exists`，从而避免因路径解析错误触发不受控的文件系统访问。  
  - 仍然依赖后续的 `register_model` 流程对对象存储进行安全凭证检查（如 IAM、服务账号），本次改动未改变该安全模型。  

**⚠️ 潜在风险**：  
1. **路径检测不完全**：仅匹配 `s3://`, `gs://`, `az://` 三种前缀，若用户使用其他对象存储协议（如 `oss://`, `cos://`）或自定义前缀，将仍走旧的预取路径，可能导致不一致行为。  
2. **误删预取**：如果某些部署场景下对象存储路径实际上是挂载在本地文件系统的映射路径（如通过 fuse），此判断会错误地跳过必要的 existence 检查，进而导致后续加载失败。  
3. **兼容性**：其他模块（非 `sglang`）若也依赖 `should_fetch_model` 逻辑，需确认它们同样期望对象存储路径被直接跳过，否则可能出现未预取而导致后续错误。  

**💡 关注建议**：  
- **扩展路径检测**：将前缀列表抽离为可配置常量或环境变量，便于在未来添加新对象存储协议。  
- **文档更新**：在 Dynamo 的模型加载文档中明确说明对象存储路径将被视为“已在 runtime 管理”，并提醒用户确保相应的凭证和挂载方式已准备就绪。  
- **回归测试**：在 CI 中加入针对非标准对象存储前缀的负面用例，以捕捉误判情况。  
- **监控/日志**：在 `should_fetch_model` 中添加调试级日志，记录被跳过的对象存储路径，帮助运维排查意外行为。  

通过以上措施，可进一步降低因路径判定错误导致的启动失败风险，同时保持对新对象存储类型的可扩展性。
