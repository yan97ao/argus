### fix: Use vllm to load prompt embeds (#8228)
**SHA**: `d644d88` | 🔗 [查看提交](https://github.com/ai-dynamo/dynamo/commit/d644d88daa18c7ad9f52a7460da59bf89a9e9565)

**🎯 变更类型**：Bug修复 / 安全修复  
**⚡ 重要程度**：🔴高  
**📋 变更摘要**：  
1. 将原先手动 `base64 → torch.load` 的 Prompt Embedding 解码逻辑替换为 vllm 提供的 `safe_load_prompt_embeds`，防止恶意构造的 tensor 导致内存写越界。  
2. 为所有 VLLM Worker Handler 增加 `model_config` 参数，并在解码前进行模型配置可用性检查与维度校验。  
3. 相应地更新了工厂函数、构造函数签名以及单元测试，以覆盖新路径并移除不再使用的 `base64`、`io`、`binascii` 依赖。  

**🎯 影响范围**：  
- `components/src/dynamo/vllm/handlers.py`（DecodeWorkerHandler、PrefillWorkerHandler、其他子类）  
- `components/src/dynamo/vllm/worker_factory.py`（创建 worker 时传递 `model_config`）  
- 相关单元测试文件（prompt embeds 解码、handler 构造）  

**🔍 技术洞察**：  

- **架构影响**  
  - 引入 `ModelConfig` 作为显式依赖，使得 Prompt Embedding 解码与模型本身的配置绑定，更符合“模型驱动”设计。  
  - `BaseWorkerHandler` 及其子类的构造签名变化可能影响外部调用方（如自定义插件或旧版集成），需要确认所有实例化点已同步更新。  
  - 通过统一使用 vllm 的安全加载函数，解耦了底层 `torch.load` 的实现细节，降低了后续维护成本。  

- **性能影响**  
  - `safe_load_prompt_embeds` 在内部仍会对 `torch.load` 进行包装，额外的安全检查（如维度、dtype、写入边界）会带来极小的 CPU 开销，几乎可以忽略不计。  
  - 移除 `io.BytesIO` 的创建与手动 `base64` 解码，改为一次性 `encode()` + vllm loader，整体执行路径更简洁，潜在的微小性能提升。  

- **安全考虑**  
  - 关键安全漏洞修复：原实现直接 `torch.load`（`weights_only=False`）会执行任意 pickle 代码，且未校验 tensor 大小，攻击者可构造巨型或越界的 tensor 导致 OOM 或内存写越界。  
  - `safe_load_prompt_embeds` 强制 `weights_only=True`，并在 `ModelConfig` 中检查 `enable_prompt_embeds` 开关，确保只有明确开启的模型才接受此路径。  
  - 新增对 `prompt_embeds_base64` 类型的校验以及对解码后 tensor 维度的强制要求（必须是 2‑D），进一步降低异常输入的攻击面。  

**⚠️ 潜在风险**：  
1. **向后兼容性**：所有使用 `BaseWorkerHandler` 的代码必须传入 `model_config`（即使为 `None`），否则会在运行时抛 `ValueError`。如果有遗漏的调用点，服务启动会失败。  
2. **模型配置缺失**：在某些部署场景（如仅使用 vllm 旧版配置）可能没有 `model_config`，此时 Prompt Embeds 功能将被禁用，需提前在配置文件中显式提供 `model_config`。  
3. **测试覆盖不足**：当前单元测试只保留 2‑D tensor 场景，3‑D（batch）路径被移除。如果以后业务需要支持 batch‑wise embeddings，需额外实现兼容逻辑。  
4. **依赖版本**：`safe_load_prompt_embeds` 属于 vllm 1.x+，若用户使用较旧的 vllm 版本会出现 `ImportError`，需要在 `pyproject.toml` 中锁定最低兼容版本。  

**💡 关注建议**：  
- **审查所有实例化路径**：在 `dynamo` 代码库以及可能的外部插件中搜索 `BaseWorkerHandler(` 的调用，确保都传入 `model_config`（即使是 `None`），并在文档中注明新增参数。  
- **升级 vllm 依赖**：在 CI/生产环境统一升级到包含 `safe_load_prompt_embeds` 的 vllm 版本，并在 `requirements.txt`/`Cargo.toml` 中添加版本约束。  
- **完善测试**：补充对 `model_config=None`、`enable_prompt_embeds=False`、以及异常 base64/非 tensor 输入的覆盖，确保错误路径的日志和异常一致。  
- **监控与告警**：在日志系统中关注 `Failed to decode prompt_embeds`、`ModelConfig is unavailable` 等错误级别的日志，及时发现配置缺失或潜在恶意请求。  
- **文档更新**：在用户手册中说明：*要使用 Prompt Embeds 功能，请在模型配置中启用 `enable_prompt_embeds=True`，并确保使用 vllm ≥ X.Y*。  

---  

通过这次改动，项目在安全性上实现了关键提升，潜在的性能影响极小，但需要注意向后兼容和依赖版本的同步更新。只要按以上建议完成验证与文档同步，风险可控且能够显著提升整体系统的安全防护水平。

### fix(health): make canary sole authority on endpoint readiness when enabled (#8165)
**SHA**: `61d4674` | 🔗 [查看提交](https://github.com/ai-dynamo/dynamo/commit/61d4674c868b4fd9a25f3dba6dfac704eb9e7d83)

**🎯 变更类型**：Bug 修复  
**⚡ 重要程度**：🔴 高  

**📋 变更摘要**  
- 引入 `health_check_enabled` 配置，使得在 canary（健康检查）启用时，系统不再在注册端点后直接标记为 `Ready`，而是交由 canary 完成验证后再设为 `Ready`。  
- 新增 `SystemHealth::set_endpoint_registered` 方法，统一处理“端点已注册”事件，并在 canary 关闭时保持原有的立即 `Ready` 行为。  
- 相关入口（HTTP、TCP、Push）均改为调用 `set_endpoint_registered`，去掉直接设置 `Ready` 的逻辑。  

**🎯 影响范围**  
- `runtime` 核心模块：`distributed.rs`、`system_health.rs`、`pipeline/network/ingress/*`（HTTP、TCP、Push）  
- 配置层：`Config`（新增 `health_check_enabled` 字段）  

**🔍 技术洞察**  

- **架构影响**  
  - **新增配置维度**：`health_check_enabled` 成为系统健康检查的全局开关，影响 `SystemHealth` 的初始化与运行时行为。  
  - **职责划分更清晰**：端点注册与健康状态的转换职责从各入口散落的代码统一到 `SystemHealth::set_endpoint_registered`，提升可维护性。  
  - **Canary 主导就绪判断**：在 canary 开启时，只有 canary 成功通过后才会将端点标记为 `Ready`，避免因提前标记导致外部流量在验证完成前进入。  

- **性能影响**  
  - **微小开销**：新增布尔判断 (`if !self.health_check_enabled`) 在注册路径上执行一次，对吞吐量和延迟几乎无影响。  
  - **更少的锁竞争**：原先在每个入口直接 `set_endpoint_health_status` 的实现会导致多次写锁，改为统一的 `set_endpoint_registered`（仅在 canary 关闭时触发一次写锁），潜在降低锁争用。  

- **安全考虑**  
  - **对外健康检查暴露**：`health_check_enabled` 默认为 `false`（保持原有行为），对已开启 canary 的部署，健康检查路径仍然可访问，不引入新的泄露风险。  
  - **配置错误风险**：若用户误将 `health_check_enabled` 设置为 `true` 而未部署 canary，系统将始终保持 `NotReady`，导致服务不可用。应在文档中明确该选项的前置条件。  

**⚠️ 潜在风险**  
1. **配置不一致**：旧版本的配置文件未包含 `health_check_enabled`，升级后默认值为 `false`，但业务方可能误以为已开启 canary，导致健康状态不符合预期。  
2. **Canary 未启动却开启 `health_check_enabled`**：系统会始终保持 `NotReady`，影响业务可用性。  
3. **并发注册顺序**：如果在极端并发场景下，端点注册与 canary 验证的顺序出现竞争，可能出现端点标记 `Ready` 前后不一致的短暂状态，需要在 canary 实现中确保原子性。  

**💡 关注建议**  
- **文档与示例**：在配置文档中明确 `health_check_enabled` 的默认值、开启条件以及与 canary 的耦合关系，提供示例启动脚本。  
- **升级指南**：在发布说明中提醒用户在升级后检查 `Config`，必要时显式设置 `health_check_enabled` 为 `false`（若未使用 canary）。  
- **监控告警**：建议在运维监控中加入 `system_health` 为 `NotReady` 超时告警，快速定位因 canary 未启动导致的服务不可用。  
- **单元/集成测试**：补充针对 `health_check_enabled = true` 场景的测试，确保在 canary 完成验证前系统不误报 `Ready`。  
- **代码审计**：后续如在 `SystemHealth` 增加更多健康检查策略时，保持 `set_endpoint_registered` 为单一入口，避免再次出现散落的状态写入。

### fix(jail): preserve logprobs through tool-call jailing (#8072)
**SHA**: `59d18d8` | 🔗 [查看提交](https://github.com/ai-dynamo/dynamo/commit/59d18d8e9c02231067897035b7654feec54a17ec)

**🎯 变更类型**：Bug修复 / 功能增强  
**⚡ 重要程度**：🔴高  
**📋 变更摘要**：在 OpenAI Chat 完成流的 *tool‑call* jailing 机制中，新增对 `logprobs` 的累计与传递，确保在工具调用被“监狱”包裹的过程中，原始的 token‑level 对数概率信息不被丢失。实现方式是在 `ChoiceJailState` 中添加 `accumulated_logprobs` 字段，并在累计、结束监狱、以及最终输出阶段对其进行合并、转移。对应单元测试也同步加入，验证单块、跨块以及文本+工具调用混合场景下的 `logprobs` 保留情况。  

**🎯 影响范围**：  
- `lib/llm/src/protocols/openai/chat_completions/jail.rs`（核心流处理逻辑）  
- 相关的 `ChatChoiceLogprobs`、`ChatCompletionStreamResponseDelta` 数据结构  
- 受影响的单元测试文件（新增 `*_with_logprobs*` 测试）  

**🔍 技术洞察**：  
- **架构影响**：  
  - 在原有的 *jail* 状态机中引入了 `Option<ChatChoiceLogprobs>` 的累计层，保持了原有抽象不变，仅在 `ChoiceJailState` 内部做状态扩展，未对外部接口产生破坏性改动。  
  - 通过 `take_accumulated_logprobs` 将累计的 logprobs 与内容同步释放，确保生命周期统一，避免悬挂引用。  

- **性能影响**：  
  - 每次累计都会 `clone`（甚至深拷贝）部分 `logprobs` 向量，导致在高吞吐量、长内容的工具调用场景下会产生额外的内存分配与拷贝开销。  
  - 由于 `logprobs` 本身是可选的，且多数用户可能关闭此功能，实际性能回退在普通文本流中几乎可以忽略不计。  

- **安全考虑**：  
  - `logprobs` 包含每个 token 的概率信息，可能泄露模型推理细节或输入 token 内容。把它们在工具调用中同样返回给客户端，需确认下游消费者的隐私策略。  
  - 改动未引入新的并发或不安全代码，仍然遵循 Rust 的所有权与借用规则。  

**⚠️ 潜在风险**：  
1. **内存使用激增**：在大量工具调用且开启 `logprobs` 的情况下，累计向量可能导致显著的堆内存增长。  
2. **向后兼容性**：部分下游实现可能假设工具调用的 `logprobs` 为 `None`，现在会收到非空值，可能触发未处理的分支或导致误判。  
3. **隐私泄露**：若用户对 `logprobs` 有严格的保密要求，工具调用中返回这些信息可能违背合规要求。  
4. **错误合并**：在极端并发或异常流（如中途取消）下，`accumulated_logprobs` 的合并逻辑如果出现 `None` 与 `Some` 不匹配的情况，可能导致部分 token 的概率丢失或重复。  

**💡 关注建议**：  
- **监控与阈值**：在生产环境中监控每个请求的 `logprobs` 大小，设定合理的上限（如 1 MiB），超过阈值时可选择截断或直接丢弃。  
- **配置开关**：考虑在 `JailedStream` 构造器中添加 `preserve_logprobs: bool` 参数，默认保持当前行为（即不返回工具调用的 logprobs），让使用者自行决定是否启用。  
- **文档更新**：在公共 API 文档中明确说明工具调用现在会携带 `logprobs`，并提示可能的隐私与性能影响。  
- **回归测试**：加入对极端长工具调用、空 `logprobs`、以及流提前终止的测试，确保 `accumulated_logprobs` 在所有异常路径下都能安全 `take` 或清空。  
- **下游适配**：提醒使用该库的上层服务检查 `ChatChoiceLogprobs` 是否为 `Some`，避免因未预料的非空值导致业务逻辑错误。

### fix(kv-router): prune stale compressed-tree children (#8127)
**SHA**: `0652441` | 🔗 [查看提交](https://github.com/ai-dynamo/dynamo/commit/0652441fe2014b57946a18764576053eec6decd2)

**🎯 变更类型**：Bug修复 / 性能优化（清理过期压缩树子节点）  
**⚡ 重要程度**：🔴高  
**📋 变更摘要**  
在 KV‑Router 的并发压缩基数树实现中，新增了“清理过期子节点”任务。当子节点不再被任何 worker 持有且没有子树时，会在合适的时机遍历整棵树并将其从父节点的 edge 表中剔除。通过 `WorkerTask::CleanupStaleChildren` 在工作线程中调度此任务，并在 `ThreadPoolIndexer` 的关键路径（store / remove / worker‑remove‑dp‑rank）后尝试触发。此改动可防止因频繁的 split/merge 操作导致的内存泄漏，并提供了对应单元测试验证。

**🎯 影响范围**  
- `lib/kv-router/src/indexer/concurrent_radix_tree_compressed.rs`（核心树实现、清理逻辑、状态管理）  
- `lib/kv-router/src/indexer/concurrent_radix_tree.rs`（新增任务分支）  
- `lib/kv-router/src/indexer/positional.rs`（同样加入清理任务）  
- `lib/kv-router/src/indexer/thread_pool.rs`（调度清理任务的入口）  
- `lib/kv-router/src/indexer/traits.rs`（为所有 Indexer 增加清理相关的 trait 方法）  
- `lib/kv-router/src/indexer/types.rs`（WorkerTask 枚举扩展）  
- 相关单元测试文件 `tests.rs`

**🔍 技术洞察**  

- **架构影响**  
  - 为所有实现 `SyncIndexer` 的后端统一引入了“维护任务”概念，`try_schedule_cleanup` / `run_cleanup_task` 成为接口的一部分。  
  - `ConcurrentRadixTreeCompressed` 增加了 `CleanupState`（基于 `Instant`、`AtomicU64`、`AtomicBool`）来控制清理频率和防止重复调度。  
  - `ThreadPoolIndexer` 在每次对树进行写操作后调用 `maybe_enqueue_cleanup`，若满足间隔则向对应 worker 发送 `CleanupStaleChildren`。这保持了“无额外线程”的实现方式，仍然利用现有工作线程完成维护。  

- **性能影响**  
  - **正面**：释放不再使用的子节点，显著降低长期运行时的内存占用，防止因大量 dead edge 导致的树遍历成本上升。  
  - **负面**：清理过程需要一次全树遍历（O(N)），但受 `CLEANUP_INTERVAL_MS = 5 min` 限制，实际触发频率低。调度检查本身只做两次原子读/写，开销可忽略。  
  - 通过 `try_write` 在子节点上尝试获取写锁，若被其他线程占用则安全跳过，避免阻塞。  

- **安全考虑**  
  - 代码未引入外部 I/O 或不安全块，主要使用 `Arc`/`Weak` 与原子类型，符合 Rust 的所有权模型。  
  - 采用 `Arc::strong_count(&child) != 2` 检查子节点仅被父节点与自身的边持有，防止误删仍在使用的节点。  
  - 由于清理在持有父节点写锁的同时仅通过 `try_write` 访问子节点，避免了潜在的死锁，且在获取不到锁时直接放弃删除，保持安全性。  

**⚠️ 潜在风险**  

1. **并发竞争导致延迟删除**  
   - 在高并发写入期间，子节点可能被频繁锁定，`try_write` 失败导致该轮清理跳过，短时间内内存回收不及时。  
2. **清理任务对业务请求的影响**  
   - 清理在工作线程上运行，如果正好在处理大量 `Store`/`Remove` 事件时触发，可能略微增加单次请求的延迟。  
3. **错误的强计数判断**  
   - 强计数检查假设除父‑edge 之外仅有两条强引用（父与子本身）。若项目后续在节点上添加了额外的强引用（例如调试统计），则节点将不被删除，导致潜在泄漏。  
4. **未覆盖所有后端实现**  
   - 只在 `ConcurrentRadixTreeCompressed` 和 `PositionalIndexer` 中实现了清理逻辑，其他可能的后端若实现 `SyncIndexer` 但未覆盖 `try_schedule_cleanup`，仍会出现内存增长。  

**💡 关注建议**  

- **监控与调优**：在生产环境加入内存使用监控，观察清理间隔是否需要根据实际负载调低（如改为 1 min）或调高。  
- **基准测试**：补充基准（bench）测评清理任务在不同树规模下的耗时，确保对峰值请求延迟的影响在可接受范围。  
- **文档与迁移**：在项目文档中说明新增的 `WorkerTask::CleanupStaleChildren` 以及 `SyncIndexer` 默认实现的行为，提醒后续实现自定义后端时进行相应的实现。  
- **可配置间隔**：考虑将 `CLEANUP_INTERVAL_MS` 设为可配置参数（环境变量或 runtime 配置），以便在不同部署场景下灵活调节。  
- **更严格的测试**：加入并发场景的随机化测试，验证在高并发写入/删除时清理不会误删活跃节点，也不会导致死锁或任务丢失。  

整体来看，此次补丁通过在现有工作线程中巧妙地加入一个低频率的维护任务，解决了压缩基数树的内存泄漏问题，对系统的稳定性与资源利用率提升显著，风险可控。

### perf(discovery): replace polling with Notify in concurrent registration wait (#8291)
**SHA**: `f05caba` | 🔗 [查看提交](https://github.com/ai-dynamo/dynamo/commit/f05cabac8ed4fd99e2024be3c2e2f789cf7a5ccd)

**🎯 变更类型**：性能优化（perf）  
**⚡ 重要程度**：🟡 中  
**📋 变更摘要**：将 `ModelWatcher::recover_concurrent_registration` 中原有的轮询式等待（最多 30 s、每 100 ms 检查一次）改为基于 `tokio::sync::Notify` 的事件驱动等待。`RegistrationGuard` 在 `drop` 时会 `notify_waiters`，阻塞的任务在 `notification` 到达或超时后立即恢复，从而消除最多 100 ms 的延迟并降低 CPU 占用。  

**🎯 影响范围**：  
- `lib/llm/src/discovery/watcher.rs`（模型注册/并发检测）  
- 相关的 `ModelWatcher` 实例（所有使用 `registering_worker_sets` 的路径）  

**💡 关注建议**：

1. **并发可靠性**  
   - `Notify` 是 **广播** 而非点对点，任何注册完成都会唤醒所有在同一 `Notify` 上等待的任务。虽然代码在每轮循环前重新 `enable()` 并再次检查集合，仍可能导致 **惊群效应**（大量任务同时唤醒后竞争 `registering_worker_sets`），在高并发场景下要关注 CPU 负载。若出现显著抖动，可考虑为每个注册键单独维护 `Notify`（如 `DashMap<String, Notify>`）以降低无关唤醒的概率。  

2. **丢失唤醒的竞态**  
   - 当前实现先 `enable()` 再检查集合，避免了 “检查‑等待‑丢失” 的竞态。但在 `notify_waiters` 与 `enable()` 之间仍存在极短窗口（guard drop 与等待任务注册 `Notify` 之间）。推荐在 `RegistrationGuard::drop` 中 **先 `notify_waiters` 再 `remove`**，或使用 `Notify::notify_one`（若改为每键单独 `Notify`）以进一步降低误触风险。  

3. **超时与 spurious wake‑ups**  
   - 代码使用 `tokio::time::timeout` + 截止时间确保不会因无关唤醒一直卡住。请确认在单元/集成测试中覆盖 **超时路径**（30 s 后仍未解除），防止潜在死锁。  

4. **资源清理**  
   - `RegistrationGuard` 持有对 `Notify` 的不可变引用，生命周期由 `ModelWatcher` 管理，安全性已符合 Rust 所求。但需确保 `ModelWatcher` 在所有注册任务结束前不会被提前 drop（如在异常关闭时），否则可能出现悬空引用。  

5. **文档与日志**  
   - 为新行为添加注释或 README，说明等待改为事件驱动，并记录 `registration_notify.notify_waiters()` 的调试日志（可使用 `tracing::debug!`），便于后续排查 “注册卡住” 的根因。  

**总体评价**：改动显著削减了轮询带来的延迟与 CPU 开销，符合性能目标。只要注意上述并发细节并补充相应测试，即可安全合入。
