### fix(health): use local endpoint registry for canary health checks (#8294)
**SHA**: `c2ec335` | 🔗 [查看提交](https://github.com/ai-dynamo/dynamo/commit/c2ec33597fd40739a4da0089b6b49ef8b739aa46)

**🎯 变更类型**：Bug修复 / 性能优化  
**⚡ 重要程度**：🔴 高  

**📋 变更摘要**  
1. 在 `EndpointConfigBuilder::build` 中加入校验：当 Canary（健康检查）开启且提供 `health_check_payload` 时，必须先在本地引擎注册表中注册对应的本地引擎，否则直接返回错误。  
2. 完全重构 `HealthCheckManager`：去掉基于 `PushRouter` 的跨进程/跨节点健康检查路径，改为通过 **本地端点注册表** 与 `AsyncEngine` 直接发起健康检查请求。相应地删除了大量缓存、etcd 监听、路由创建等代码，简化了依赖并显著提升了执行效率。  

**🎯 影响范围**  
- `lib/runtime/src/component/endpoint.rs`（端点构建与健康检查注册）  
- `lib/runtime/src/health_check.rs`（健康检查管理器实现）  
- 相关的 `DistributedRuntime`、`SystemHealth`、`AsyncEngine` 接口（行为被新的本地调用方式取代）  

**🔍 技术洞察**  

| 维度 | 影响分析 |
|------|----------|
| **架构影响** | • 由分布式 **PushRouter → etcd → 实例发现** 的健康检查路径，切换为 **本地端点注册表 → AsyncEngine** 的同步路径。<br>• 通过 `register_local_engine()` 完成的本地引擎注册变成 Canary 健康检查的硬依赖，进一步明确了 “本地引擎 = 健康检查目标”。<br>• 移除 `router_cache`，降低了 `HealthCheckManager` 的状态复杂度，简化了内存模型。 |
| **性能影响** | • 去掉了对 `etcd` 监视流的等待（`router.client.wait_for_instances()`）以及 10 s 超时逻辑，健康检查立刻走本地路径，延迟从数毫秒降低到微秒级。<br>• 移除 `HashMap` 缓存和 `Arc<Mutex<>>` 的频繁锁竞争，减少了并发开销。<br>• 体积上 `health_check.rs` 删除约 100 行代码，编译时间与二进制体积都有轻微下降。 |
| **安全考虑** | • 仅在进程内部调用 `AsyncEngine.generate`，不再通过网络/etcd 进行健康检查，攻击面进一步收窄。<br>• 错误信息在 `EndpointConfigBuilder` 中使用 `anyhow::bail!` 明确提示缺失本地注册，引导开发者在部署阶段完成正确的调用顺序，避免因误配置导致健康检查失效。 |
| **可维护性** | • 删除了大量与旧路由、实例发现相关的代码，代码行数下降约 50%，逻辑更清晰。<br>• 通过显式的 `register_local_engine()` 前置检查，业务侧错误更早可见，调试成本降低。 |

**⚠️ 潜在风险**  

1. **启动顺序硬依赖**：如果调用方忘记在 `Runtime::start()` 前执行 `register_local_engine()`，`EndpointConfigBuilder` 会直接返回错误，导致整个系统启动失败。必须在文档或示例中明确此前置条件。  
2. **分布式场景兼容性**：原实现支持跨节点通过 etcd 发现实例进行健康检查；新实现仅在本地进程内工作。如果用户在多进程/多机器部署且仍期望 Canary 健康检查跨节点执行，功能将失效。  
3. **代码依赖遗漏**：`HealthCheckManager` 的结构体字段 `router_cache` 被删除，若其他模块仍持有该字段的引用（例如单元测试或旧插件），会导致编译错误。需要确保全库已经同步修改。  
4. **错误传播**：`anyhow::bail!` 产生的错误会在 `EndpointConfigBuilder::build` 中被传播到调用方。若调用方未捕获并处理，可能导致未预期的 panic。  

**💡 关注建议**  

- **文档与示例**：在 `README` 或发布说明中明确 “使用 Canary 健康检查时必须在 `Runtime.start()` 前调用 `register_local_engine()`”。提供最小可运行示例演示正确顺序。  
- **兼容性保护**：若仍需支持跨节点 Canary（例如在大规模集群），考虑保留旧实现的特性开关（`use_remote_canary = true`），或者在配置中明确声明仅支持本地模式。  
- **单元/集成测试**：新增测试验证：① 未注册本地引擎时启动会返回错误；② 正常注册后健康检查能够成功发送并收到响应。  
- **监控与告警**：在 `SystemHealth` 中加入 “Canary 配置错误” 的告警项，以便运维快速定位启动失败的根因。  
- **回滚路径**：如果后续发现跨节点健康检查仍是业务必需，保持当前提交的 `git` 分支历史，以便快速恢复 `router_cache` 与 `PushRouter` 逻辑。  

通过上述改动，Dynamo 的健康检查机制变得更轻量、快速且安全，但也引入了对本地引擎注册的强制依赖，需要在使用文档和测试上做好对应的补充。

### fix(snapshot): resolve DRA GPU UUIDs from claims (#8292)
**SHA**: `2cc6d1e` | 🔗 [查看提交](https://github.com/ai-dynamo/dynamo/commit/2cc6d1e2b7a19e5c6990b78de8f14b82f54e39d7)

**🎯 变更类型**：Bug修复 / 功能增强  
**⚡ 重要程度**：🔴高  
**📋 变更摘要**：  
1. 为 GPU UUID 发现新增统一入口 `DiscoverGPUUUIDs`，在 DRA‑backed Pod、传统 `nvidia.com/gpu` Pod 与 nvidia‑smi 三条路径之间自动切换。  
2. `GetGPUUUIDsViaDRAAPI` 重构，返回是否存在 NVIDIA DRA 分配的标识并提升错误信息。  
3. 单元测试覆盖 DRA、PodResources 与 fallback 场景，并对旧实现做了兼容性验证。  
4. 快照检查点/恢复流程改用 `DiscoverGPUUUIDs`，去除冗余手动 fallback 逻辑。

---

### 🎯 影响范围
- **核心模块**：`deploy/snapshot/internal/cuda`（`cuda.go`, `dra.go`, `dra_test.go`, `cuda_test.go`）  
- **快照执行器**：`deploy/snapshot/internal/executor/checkpoint.go`, `restore.go`  
- **依赖**：引入 `k8s.io/client-go/kubernetes` 与 `k8s.io/client-go/kubernetes/fake`（用于测试）  
- **后端**：与 Kubernetes DRA（Device Resource Allocation）API 的交互路径。

---

### 🔍 技术洞察

| 维度 | 影响描述 |
|------|----------|
| **架构影响** | • 引入 **Discovery 层**，统一了多种 GPU UUID 获取方式，降低上层业务对实现细节的耦合。<br>• `DiscoverGPUUUIDs` 需要 `clientset` 参数，意味着调用方必须拥有对 Kubernetes API Server 的访问权限，这在原有仅依赖 PodResources socket 的设计上提升了权限需求。<br>• DRA 路径的实现被拆分为 `getAllocatedNVIDIADRADevices`（解析 Pod → Claim → Allocation）与主函数，便于未来复用或单独测试。 |
| **性能影响** | • 新增一次 **DRA API** 查询（Pod、Claim、ResourceSlice）以及一次 **PodResources** 调用（若 DRA 失效），最坏情况仍会落到原有 `nvidia‑smi` 调用。<br>• 这些 API 调用在本地集群内的延迟通常在毫秒级，对快照检查点/恢复的总体耗时影响可忽略（原本已需要读取 `/proc` 与 `nvidia‑smi`）。<br>• 代码路径已在成功获取 UUID 后提前返回，避免不必要的额外查询。 |
| **安全考虑** | • 新增对 Kubernetes **client-go** 的使用，若 `clientset` 配置错误或被恶意篡改，可能导致信息泄露（如获取其它命名空间的 ResourceClaim）。但函数仅读取 **只读** 资源，且已通过 `hasNVIDIADRAAllocation` 标识区分是否为 DRA‑GPU 场景，未对对象做写操作。<br>• `log.Error` 与 `log.Info` 已记录错误来源，便于审计。 |
| **可维护性** | • 逻辑分层清晰：DRA 解析 → PodResources → nvidia‑smi 逐级回退。<br>• 单元测试涵盖 **经典 Pod、DRA Pod、错误回退、模板‑backed claim** 四类情形，提升后续改动的安全网。<br>• 旧的 `GetPodGPUUUIDs` 仍保留，供内部或外部直接调用，保持向后兼容。 |
| **兼容性** | • 旧实现的行为保持不变：若 `clientset` 为 `nil` 或 Pod 未使用 DRA，仍会走 PodResources → nvidia‑smi 途径。<br>• 调用方需要更新签名（额外的 `clientset`、`hostProcPath`、`log` 参数），已在快照执行器同步修改，外部直接使用 `GetGPUUUIDsViaDRAAPI` 的代码需适配返回值变化（额外 `bool`）。 |

---

### ⚠️ 潜在风险
1. **权限不足**：在某些受限的运行时环境（如仅挂载 `kubelet` socket的 sidecar）可能没有足够的 RBAC 权限去访问 `ResourceClaim`、`ResourceSlice`。此时 `DiscoverGPUUUIDs` 会返回错误并回退至 nvidia‑smi，若 `nvidia‑smi` 也不可用会导致快照失败。  
2. **API 版本/兼容性**：`ResourceSlice` 与 DRA 仍在 **Alpha/Beta** 阶段，API 变更可能导致字段名或筛选器失效，需要在未来的 Kubernetes 版本升级时进行回归测试。  
3. **资源泄露**：错误的 `clientset`（例如使用了高权限的 ServiceAccount）可能让此组件在不必要时访问集群资源，建议在部署时使用最小化的 RBAC（仅 `get`/`list` 所需的 CRDs）。  
4. **性能突增**：在极端大规模集群中，`List` 所有 `ResourceSlice` 可能返回大量对象。当前代码通过 `FieldSelector` 限定 `driver` 与 `nodeName`，但仍可能产生较大响应，建议监控列表耗时并在必要时加入分页或缓存。  

---

### 💡 关注建议
- **RBAC 最小化**：在部署快照插件时，加入如下 ClusterRole（或对应 Namespace‑Scoped Role）：
  ```yaml
  - apiGroups: ["resource.k8s.io"]
    resources: ["resourceclaims", "resourceslices"]
    verbs: ["get", "list"]
  - apiGroups: [""]
    resources: ["pods"]
    verbs: ["get"]
  ```
- **监控 & 报警**：记录 `DiscoverGPUUUIDs` 的每一次路径选择（DRA / PodResources / nvidia‑smi）到 Metrics（如 Prometheus Counter），监测异常回退频率。  
- **回退日志**：当前已在 `log.Info` 中输出回退原因，建议统一使用结构化日志字段（`fallback_reason`）以便日志查询。  
- **升级兼容**：在未来的 Kubernetes 1.30+（或对应 DRA 正式 GA）发布后，验证 `resourceSlice` 的 API Group/Version 是否仍为 `resource.k8s.io/v1alpha2`（或新版本），并更新代码中的 import/结构体。  
- **测试覆盖**：持续在 CI 中运行新增的 DRA 测试；若项目启用 `go test -race`，确认并发安全（`clientset` 为只读且不共享状态，已安全）。  
- **文档更新**：在项目的 README/插件指南中添加 “GPU UUID discovery flow” 与所需 RBAC 示例，帮助使用者了解新行为。  

--- 

**结论**：本次变更在保持原有功能的同时，引入了对 NVIDIA DRA GPU 分配的原生支持，统一了多路径 GPU UUID 检索逻辑。对架构的正向影响明显，性能与安全影响可控，只要在部署时确保适当的 RBAC 与监控，即可安全地在生产环境推广。

### feat(benchmarks): add sliding-window JSONL generation strategy (#8201)
**SHA**: `32ec044` | 🔗 [查看提交](https://github.com/ai-dynamo/dynamo/commit/32ec044ccd24d68981e30f1ed76ad80d680f291c)

**🎯 变更类型**：功能增强  

**⚡ 重要程度**：🔴高  

**📋 变更摘要**：在 `benchmarks/multimodal/jsonl` 中引入了 **sliding‑window** 生成策略，并对原有的 JSONL 生成脚本进行结构化重构。CLI 现在采用子命令（`single-turn`、`sliding-window`）统一管理公共参数；增加了参数校验、默认子命令自动选择、以及基于策略的入口分发。相应单元测试覆盖了两种策略的基本行为与图片重叠逻辑。

**🎯 影响范围**：  
- `benchmarks/multimodal/jsonl/args.py`（CLI 参数组织）  
- `benchmarks/multimodal/jsonl/main.py`（生成逻辑、策略分发）  
- 新增测试 `tests/benchmarks/multimodal/jsonl/test_main.py`  
- 相关文档/示例脚本（README/使用说明）  

**🔍 技术洞察**  

- **架构影响**  
  - 将原本单一脚本的参数解析拆分为 **_common_parser**、**_image_parser** 与子命令层级，实现了 **模块化、可扩展** 的 CLI 架构。后续若要再添新策略（例如 “multi‑turn”）只需在 `STRATEGIES` 表中注册即可，无需改动核心逻辑。  
  - 引入 `STRATEGIES` 映射表 +统一入口 `main()`，实现 **策略模式**，提升代码可读性与可维护性。  
  - 默认子命令逻辑（当未显式指定 subcommand 时自动使用 `single-turn`）保持向后兼容，但在直接调用 `python main.py -h` 时的帮助信息展示稍有变化，需要在文档中说明。  

- **性能影响**  
  - 生成滑动窗口时会一次性构造所有用户所需的图片池 (`total_images = num_users * images_per_user`) 并在写入时通过切片复用。内存占用随 `num_users * (window_size + turns_per_user - 1)` 线性增长，极端配置（如数千用户、上百 turns）可能出现显存/内存压力。  
  - 其它路径（单轮）保持原有实现，仅因函数拆分与子命令解析产生极微小的 CPU 开销，基本可以忽略。  
  - 随机种子统一在 `main()` 中生成，确保 **可复现**，对 benchmark 的对比实验非常有帮助。  

- **安全考虑**  
  - 新增 `--seed` 参数已保持原有行为，无额外安全风险。  
  - `--image-mode http` 仍然直接读取外部 COCO URL，若在受限网络环境或恶意 URL 列表中使用，可能导致 **外部请求** 或 **信息泄露**。建议在生产/CI 环境中默认使用 `base64` 模式或对 URL 做白名单校验。  
  - 参数校验使用了自定义 `_positive_int`，防止负数或零引发逻辑错误，提升了 **输入安全性**。  

**⚠️ 潜在风险**  

1. **向后兼容性**  
   - 原始脚本默认行为是 `single-turn`，但现在通过子命令实现，若调用者在脚本前加了未知参数或自行解析 `argv`，可能误判为子命令从而进入异常分支。  
   - 文档/CI 脚本中硬编码的 `-n`、`--images-per-request` 等参数顺序仍可使用，但若在 CI 中直接调用 `python benchmarks/.../jsonl/main.py`（不带子命令），仍会自动切换为 `single-turn`，兼容性基本保持。  

2. **依赖 Pillow**  
   - 测试中显式 `pytest.importorskip("PIL")`，但在未安装 Pillow 的环境下，`base64` 模式会报错。需要在项目的 `requirements` 中明确标记 `Pillow` 为 benchmark 运行时依赖。  

3. **大规模滑动窗口内存占用**  
   - 随着 `num_users`、`turns_per_user`、`window_size` 的组合指数增长，`total_images` 可能导致 O(N) 的列表占用。若在资源受限的 CI 中使用极端参数，会出现 OOM。建议在 CLI 中加入上限检查或分批写入。  

4. **路径与文件写入**  
   - 自动生成的默认输出路径基于 `Path(__file__).parent`，在某些 CI 运行时可能无写权限。虽然用户可以自行指定 `-o`，但最好在文档中提醒输出目录必须可写。  

**💡 关注建议**
