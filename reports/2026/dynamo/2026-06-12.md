### feat: support multimodal inputs in sglang disaggregated prefill/decode (#10643)
**SHA**: `2e50e0b` | 🔗 [查看提交](https://github.com/ai-dynamo/dynamo/commit/2e50e0b5a2af714dd4fdcbeaaa16fd844504c086)

**🎯 变更类型**：功能增强  

**⚡ 重要程度**：🔴高  

**📋 变更摘要**  
- 为 SGLang 的 **disaggregated（前置 Prefill + Decode 分离）** 工作流新增对多模态（图片/视频）输入的统一支持。  
- 抽取并复用 URL 解析逻辑至 `mm_disagg_utils.py`，在 Prefill 与 Decode 两侧均以相同的 `image_data` / `video_data` 参数调用底层 LLM 引擎，确保 KV 传递的 token 布局一致。  
- 同步更新测试、启动脚本以及多模态性能 profile，加入对模型名称的可选覆盖以及对异常/错误 payload 的显式校验。  

**🎯 影响范围**  
- `components/src/dynamo/sglang/request_handlers/llm/decode_handler.py`  
- `components/src/dynamo/sglang/request_handlers/llm/prefill_handler.py`  
- 新增 `components/src/dynamo/sglang/request_handlers/llm/mm_disagg_utils.py`（核心抽取与参数构造）  
- 相关单元测试 `components/src/dynamo/sglang/tests/…`  
- 示例启动脚本 `examples/backends/sglang/launch/disagg_same_gpu.sh`  
- 多模态 profile 配置 `tests/serve/multimodal_profiles/sglang.py`  

---

### 🔍 技术洞察  

| 维度 | 影响描述 |
|------|----------|
| **架构影响** | - 将多模态 URL 解析抽离成公共工具，消除了 Prefill 与 Decode 两端实现差异，提升代码可维护性。<br>- `async_generate` 接口在 disaggregated 路径上新增 `image_data` / `video_data` 参数，要求底层引擎保持向后兼容。<br>- launch 脚本加入 `--model` 参数，方便在同一 GPU 上切换普通 LLM 与 VLM，进一步统一了部署流程。 |
| **性能影响** | - 只在请求入口做一次列表遍历与简单类型检查，CPU 开销极低（O(N)），对整体吞吐影响可忽略。<br>- 对于 **前端解码** 场景（`_enable_frontend_decoding=True`），仍会触发图片批量下载，新增的 `extract_media_urls` 在该分支外返回 URL 列表，不产生额外 I/O。<br>- 通过统一 token 布局避免了 KV 不匹配导致的重算或错误回滚，间接提升了端到端一致性与成功率。 |
| **安全考虑** | - 新增 **严格校验**：非列表、未包含 `Url` 字段或携带 `Decoded` 键的项将抛出 `ValueError`，防止后端误将已解码的二进制数据当作 URL 处理。<br>- 仍然接受任意字符串 URL，若后端在 `image_loader` 中直接下载，可能受到 **SSRFD**、**恶意 URL** 的风险。建议在更高层加入白名单或 URL Scheme 限制（只允许 `http(s)://`、`file://` 本地路径等）。 |
| **可维护性** | - 通过 `mm_disagg_utils` 实现 **单点修改**，后续若需支持更多媒体类型（音频、3D）只需在该文件添加对应键即可。<br>- 测试覆盖从原来的私有 `_extract_media_urls` 转为公开 `extract_media_urls`，并加入异常路径的断言，提升质量门槛。 |
| **兼容性** | - 旧的调用路径仍可工作（未使用 `image_data`/`video_data` 参数时为 `None`），但在 **disaggregated** 场景下如果请求中携带错误的多模态 payload，将不再默默降级为纯文本，而是抛异常，可能导致现网请求失败。需要评估现有用户的调用容错策略。 |

---

### ⚠️ 潜在风险  

1. **异常传播**：`extract_media_urls` 抛出的 `ValueError` 若未在更高层捕获，会中断整个请求的生成流程，导致 5xx 错误。  
2. **URL 可信度**：直接把用户提供的 URL 交给图片加载器（`self._image_loader.load_image_batch`）可能引入外部资源下载风险（恶意文件、内部网络探测）。  
3. **部署脚本变更**：`disagg_same_gpu.sh` 新增 `--model` 参数，误用或忘记传递导致默认模型不符合预期，尤其在 CI/benchmark 环境中。  
4. **后端引擎兼容**：`async_generate` 必须接受 `image_data`、`video_data` 两个可选 kwarg；如果有自定义或旧版引擎未同步更新，会出现 “unexpected keyword argument” 错误。  

---

### 💡 关注建议  

| 对象 | 建议 |
|------|------|
| **后端引擎维护者** | - 确认 `async_generate` 接口已向后兼容新增 kwarg，若有语言层面（C/Go）实现，补充对应参数的默认值处理。<br>- 在加载图片前加入 URL Scheme 与域名白名单检查，防止 SSRF。 |
| **前端/调用方** | - 在使用 disaggregated Prefill+Decode 时，确保多模态 payload **仅提供 URL 列表**（不含 `Decoded`），否则会被断言为错误。<br>- 若需要前端直接解码（`_enable_frontend_decoding=True`），仍使用旧的 `_image_loader` 路径，保持兼容。 |
| **运维/测试** | - 在 CI 中加入对 `disagg_same_gpu.sh --model` 参数的验证，防止默认模型被误写。<br>- 为异常路径（malformed payload）添加健康检查，确认服务返回 400/422 而非 500。 |
| **文档/示例** | - 在 SGLang 文档中明确说明 **disaggregated 多模态请求** 的格式要求（`multi_modal_data` 必须是 `{ "image_url": [...], "video_url": [...] }`），并标注不支持 `Decoded` 键的原因。<br>- 示例脚本中加入 `--help` 输出，帮助新手快速定位可配置项。 |

**结论**：此次提交为 SGLang 在 **disaggregated** 工作流中引入多模态支持的关键补丁，提升了功能完整性并通过统一抽取函数降低了代码重复度。风险主要集中在异常处理和 URL 安全上，建议在生产环境前做好输入校验、异常捕获以及对应的监控告警。整体收益（功能扩展 + KV 一致性保证）远大于潜在风险，属于 **高优先级** 的功能增强。

### fix(renderer): render chat templates closer to vLLM/SGLang behavior (#10624)
**SHA**: `2623ed7` | 🔗 [查看提交](https://github.com/ai-dynamo/dynamo/commit/2623ed7dac2dbd8314277bc2df49d1ebec89a884)

**🎯 变更类型**：Bug修复 / 功能增强（对渲染行为做精准对齐）  
**⚡ 重要程度**：🔴高  
**📋 变更摘要**：本次提交为渲染层引入 `minijinja` 的 `preserve_order` 特性并实现自定义的 `PyJsonFormatter`，使得 JSON 对象在模板渲染后保持客户端（Python）插入顺序，并采用 Python `json.dumps` 的默认分隔符（`, ` 和 `: `）以及不进行 HTML 转义，从而与 vLLM / SGLang 的行为保持一致。相应的快照测试已更新以验证新顺序与格式。  

**🎯 影响范围**：  
- `lib/renderer`（模板渲染、`tojson` 过滤器）  
- `lib/llm`（预处理器中使用的 Minijinja）  
- 依赖 `indexmap`（所有使用 Minijinja 的 crate）  
- 单元测试快照（`lib/llm/tests/snapshots`）  

**🔍 技术洞察**  

- **架构影响**  
  - 引入 `indexmap` 并开启 `minijinja` 的 `preserve_order` 特性，使得 Minijinja 在内部使用 `IndexMap` 替代默认的 `BTreeMap` 保存对象键序。此更改在渲染层是向后兼容的（仅改变键的遍历顺序），但会影响所有通过 Minijinja 生成 JSON 的路径。  
  - 新增 `PyJsonFormatter` 实现了 `serde_json::ser::Formatter`，在 `tojson` 过滤器中使用，确保序列化时使用 Python 风格的逗号与冒号加空格分隔符，匹配 HuggingFace Transformers `tojson` 行为。  

- **性能影响**  
  - `IndexMap` 在大多数情况下比 `BTreeMap` 更快（O(1) 插入/查找），因此渲染大量工具调用参数时可能略有提升。  
  - `PyJsonFormatter` 相比默认 `CompactFormatter` 多了一次 `write_all` 调用来写空格，开销极小，可忽略不计。整体渲染性能基本保持不变。  

- **安全考虑**  
  - 之前 `tojson` 会对 `< > & '` 进行 Unicode 转义，以防止在 HTML 环境中产生 XSS。现在移除了这层转义，直接返回 *安全字符串*（`Value::from_safe_string`），因为渲染结果主要用于模型提示而非直接嵌入 HTML。若用户误将渲染结果用于 Web 页面，需自行进行相应的 HTML 转义。  
  - 依赖版本更新至 `indexmap 2.14.0`，没有已知的安全漏洞，但建议在 CI 中继续使用 `cargo audit` 检查。  

**⚠️ 潜在风险**  

1. **行为兼容性**  
   - 之前依赖键的字母序（BTreeMap）进行比较或签名的系统可能会因键顺序变化导致不一致。  
   - 已经持有的快照或缓存（如旧模型的 prompt）若未同步更新，可能产生意料之外的输出差异。  

2. **HTML 使用场景**  
   - 由于去掉了转义，如果 `tojson` 的结果被直接插入到网页模板或日志系统，可能触发 XSS。使用方需要自行确保后端/前端进行适当的转义。  

3. **依赖升级冲突**  
   - 部分库可能对 `indexmap` 有特定的 `serde` 实现或特性要求，升级到 2.14.0 可能导致编译冲突。当前仓库已通过 CI，但外部使用者在集成时需注意 Cargo 版本冲突。  

**💡 关注建议**  

- **回归测试**：在所有使用 Minijinja 渲染 JSON 的端到端场景（包括工具调用、函数调用、系统提示）跑一次完整的集成测试，确保顺序和格式符合预期。  
- **文档更新**：在 `README` 或渲染模块的使用说明中明确指出 `tojson` 不再进行 HTML 转义，并推荐在 Web 场景自行调用 `html_escape`。  
- **兼容层**：如果需要兼容旧行为，可考虑在 `tojson` 中添加可选参数（如 `escape_html=true`）来恢复原有转义逻辑，避免强制破坏旧用户代码。  
- **依赖监控**：将 `indexmap` 的安全审计纳入 CI（`cargo audit`），并在发布日志中标明此依赖的升级。  
- **性能基准**：可在大型工具调用列表上跑一次基准（`cargo bench`），确认 `IndexMap` 带来的微小性能提升或不下降。

### fix(xpu): align DYN_SYSTEM_PORT defaults and add ZE_AFFINITY validation (#10488)
**SHA**: `1f977b5` | 🔗 [查看提交](https://github.com/ai-dynamo/dynamo/commit/1f977b53f4556f4a4a13ce773c410db9e5378485)

**🎯 变更类型**：Bug修复 / 功能增强  
**⚡ 重要程度**：🔴高  
**📋 变更摘要**：  
- 将 XPU 启动脚本中 `DYN_SYSTEM_PORT` 的默认值对齐到 `VLLM_SYSTEM_PORT_BASE`（18081），并在多 Worker 场景下支持 `DYN_SYSTEM_PORT1/2/...` 的显式指定。  
- 新增 `ZE_AFFINITY_MASK` 解析与完整性校验，确保多 Worker 时每个进程都有合法的 GPU 亲和性配置。  
- 相应地更新了测试用例，加入对上述环境变量的使用与验证，避免因缺失或格式错误导致脚本异常。

**🎯 影响范围**：  
- `examples/backends/vllm/launch/xpu/agg_multimodal_router_chat_processor_xpu.sh`  
- `examples/backends/vllm/launch/xpu/agg_multimodal_router_xpu.sh`  
- `tests/serve/multimodal_profiles/vllm_xpu.py`  
- `tests/serve/test_vllm_xpu.py`  
- 受影响的 CI XPU 测试流程与用户自定义的 XPU 启动脚本。

**🔍 技术洞察**：

- **架构影响**  
  - 脚本层面实现了对多 Worker 环境的显式端口与 GPU 亲和性配置，提升了 XPU 后端在多卡、多进程部署时的可预测性。  
  - 通过 `DYN_SYSTEM_PORT{i}` 与 `ZE_AFFINITY_MASK` 的映射，解耦了硬编码端口/GPU 与运行时环境，降低了跨节点部署的配置冲突风险。  

- **性能影响**  
  - 仅在脚本启动阶段进行一次字符串分割与校验，开销可忽略不计。  
  - 正确的 `GPU_ID` 分配和端口对齐可避免因端口冲突或 GPU 亲和性错误导致的进程重启或资源争用，从而间接提升多 Worker 推理吞吐。  

- **安全考虑**  
  - 对 `ZE_AFFINITY_MASK` 进行非空、去空格校验，防止空值或仅包含空格的参数导致意外的 `GPU_ID` 解析错误，减少潜在的资源泄漏或异常退出。  
  - 采用 `set -euo pipefail` 环境，脚本在检测到不合法输入时直接 `exit 1`，防止后续命令在错误状态下继续执行，提升了运行时的鲁棒性。  

**⚠️ 潜在风险**：

1. **向后兼容性**：已有 CI 或本地脚本未显式设置 `ZE_AFFINITY_MASK` 或 `DYN_SYSTEM_PORT{i}`，在多 Worker 场景下将触发脚本退出，可能导致构建/测试失败。  
2. **环境变量拼写错误**：若用户误把 `DYN_SYSTEM_PORT` 写成单个变量（而不是 `DYN_SYSTEM_PORT1` 等），单 Worker 时仍能工作，但多 Worker 时会回退到默认端口 18081，可能产生端口冲突。  
3. **端口冲突**：如果用户自定义的 `DYN_SYSTEM_PORT{i}` 与其他服务（如 ZMQ、HTTP）冲突，脚本本身不会检测，需要在外部自行保证端口唯一。  

**💡 关注建议**：

- **CI / 文档**：在项目文档（README/部署指南）中补充 `ZE_AFFINITY_MASK` 与 `DYN_SYSTEM_PORT{i}` 的使用说明，明确多 Worker 必须提供对应的逗号分隔列表。  
- **迁移指南**：提供一个小脚本或示例 CI 配置，自动生成 `ZE_AFFINITY_MASK=0,1,...` 与 `DYN_SYSTEM_PORT{i}=18081+2*(i-1)`，帮助用户平滑迁移。  
- **回退方案**：若业务需要保留旧行为，可在脚本最前加入兼容层（例如检测旧变量 `DYN_SYSTEM_PORT` 并映射），但建议在下个大版本中正式去除。  
- **监控/调试**：在多 Worker 调试时打印 `GPU_ID` 与 `WORKER_PORT` 的最终取值，有助于快速定位环境变量配置错误。

  - **请求路径扩展**：在原有 `nvext` 与 `extra_args` 两层结构中统一读取 `metadata_upload`，保持向后兼容；`extra_args` 为后端专用的透传字段，避免对已有 OpenAI API 接口产生影响。  
  - **后端抽象**：`MetadataUploader` 通过 `fsspec` 抽象文件系统，实现对 S3、GCS、POSIX 等多种存储的统一写入，符合 Dynamo “插件化后端” 设计。  
  - **异步 I/O**：上传在 `asyncio.to_thread` 中完成，防止阻塞生成流；仅在 **final** token（或流结束）时触发，减小对生成路径的干扰。  
  - **Rust↔Python 边界**：在 Rust `NvExt` 中加入 `metadata_upload` 字段，使得前端可以直接在 JSON 请求中声明上传 URL，保持一致的 schema 校验。

- **性能影响**  
  - **CPU 开销**：`msgspec` 编码 + `zstd` 压缩在上传前会占用额外 CPU，尤其是大规模张量/数组时。实现使用 `asyncio.to_thread` 将其放到线程池，减轻事件循环压力。  
  - **网络 I/O**：上传过程是 **可选** 的，仅在 `enable_rl=true` 且 `metadata_upload` 配置存在时才触发；若后端存储带宽较低，可能延长最终响应的整体时延。  
  - **内存占用**：压缩前完整的元数据会在内存中保持一次（`bytes`），随后在上传完成后立即 `del data`，对内存泄漏风险有限。  

- **安全考虑**  
  - **URL 校验**：仅进行非空字符串检查并去除空格，未验证协议合法性或路径安全性，仍依赖 `fsspec` 后端实现的认证/签名机制。若用户自行提供恶意 URL，可能导致数据泄露或写入未经授权的存储。  
  - **异常传播**：缺少 `fsspec`、`zstandard`、`msgspec` 时会抛出 `RuntimeError`，会把整个生成请求中断；需要在生产环境对这些错误进行捕获并回滚或返回可解释的错误码。  
  - **数据隐私**：上传的元数据可能包含模型内部状态（如路由专家、张量），若未加密传输（取决于后端 URL），会有敏感信息泄露风险。建议在文档中明确要求使用 TLS/加密存储。

**⚠️ 潜在风险**  

| 风险点 | 可能后果 | 缓解建议 |
|--------|----------|----------|
| 未安装 `fsspec`/`zstandard`/`msgspec` | 运行时抛异常、整个生成任务失败 | 在部署脚本中强制安装对应 extras，或在启动时检测缺失并给出明确提示。 |
| `enable_rl` 未开启但仍提供 `metadata_upload` 配置 | 上传器不会创建，导致用户预期的元数据未上报 | 在前端返回警告或在文档中强调必须配合 `enable_rl=true`。 |

| 存储写入错误未被捕获（如网络中断、权限不足）导致请求异常终止 | 客户端收到错误响应， RL 训练数据丢失 | 在 `_process_*_stream` 中已捕获 `RuntimeError` 并让异常冒泡，可在上层捕获并记录失败，不影响已返回的 token。 |
| URL 直接使用用户提供的字符串，未做白名单或协议校验 | 可能写入不受控位置，安全隐患 | 在 `MetadataUploader.from_settings` 中加入协议白名单（s3://, gs://, file:// 等），并对路径做安全清理。 |

**💡 关注建议**  

1. **部署检查**：确保 `fsspec`（含所需后端 extras）和 `zstandard` 已经安装；在 CI 中加入对应的依赖检查。  
2. **监控与告警**：对 `MetadataUploader.upload_choice` 的异常率、上传时长、压缩比进行监控，从而快速捕获后端存储或网络异常。  
3. **文档完善**：在使用手册中明确说明 `metadata_upload` 只在 RL 场景（`enable_rl=true`）下生效，且推荐使用 TLS/加密存储。  
4. **安全加固**：如果产品对数据隐私有严格要求，考虑在上传前对压缩数据进行加密（如 `cryptography`）或仅支持 HTTPS/STS/签名的存储协议。  
5. **性能基准**：建议在实际部署前对典型元数据（张量/数组）进行基准测试，评估压缩+上传对整体响应时间的额外贡献，并根据需求调节是否开启该功能。

此次改动为 Dynamo 引入了可扩展的元数据外部化机制，为强化学习等大规模实验提供了实用的后端数据收集手段，但也伴随额外的依赖、CPU/网络开销以及安全风险。做好依赖、监控、文档及安全防护，能够将风险降至最低并充分发挥该功能的价值。

- `deploy/helm/charts/platform/components/operator/templates/prometheus.yaml`（Prometheus PodMonitor 配置）  
- 运行时的 `worker`、`decode`、`prefill` 三类 Dynamo 组件的 metrics 暴露与采集  
- 监控平台的 Grafana/Alert 规则（若依赖 `dynamo_component_type`）  

**关注建议**  
1. **功能验证**：在测试集群部署更新后，确认 `decode`、`prefill` Pod 的 `/metrics` 能被 Prometheus 成功抓取，且 `dynamo_component_type` 标签出现。  
2. **回归检查**：确保其他非上述组件（如 `parameter-server`）仍然不被此 PodMonitor 捕获，防止误报。  
3. **CI/Lint**：新增 Helm lint 检查，确保 `matchExpressions` 语法正确，且 `relabelings` 块符合 Prometheus Operator schema。  
4. **文档同步**：在 chart README 与监控手册中补充 “decode/prefill metrics now exposed” 的说明，提醒用户升级后可查看对应监控图表。  
5. **版本管理**：由于行为变化，建议在 Chart 版本号上升 `patch` 或 `minor`，并在发布日志中明确列出此 fix。  

总体来看，此次修改只涉及监控配置层面，对业务逻辑无侵入，但需确保 Helm 渲染和 Prometheus Operator 正确解析新规则，以避免监控数据缺失或误抓取。
