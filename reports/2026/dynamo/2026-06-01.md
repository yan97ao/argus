### refactor: extract chat-template rendering into standalone dynamo-renderer crate (#9946)

**⚡ 重要程度**：🔴高  

**📋 变更摘要**  
- 将原本散落在 `lib/llm/src/preprocessor/prompt.rs` 中的 **chat‑template 渲染逻辑**抽离为独立的 `dynamo‑renderer` crate（`lib/renderer`），实现 **运行时‑无依赖、可单独使用** 的渲染库。

### feat(sglang): fix mem leak with request cancellation (#10082)
**SHA**: `7c4d40b` | 🔗 [查看提交](https://github.com/ai-dynamo/dynamo/commit/7c4d40b0cc2521a23fff5827158858229c8dac08)

**🎯 变更类型**：Bug修复 / 功能增强  
**⚡ 重要程度**：🔴高  
**📋 变更摘要**：在 `handler_base.py` 中为请求取消监控引入了显式的 `cancellation_future` 与 `shutdown_task` 变量，并在 `finally` 块统一进行取消与回收，防止因异常退出导致的协程或任务残留，从而修复了内存泄漏问题。  

**🎯 影响范围**：  
- `components/src/dynamo/sglang/request_handlers/handler_base.py`（SGLang 请求处理基类）  
- 受影响的整体 SGLang 请求生命周期管理（包括取消、shutdown 监控）  

**🔍 技术洞察**：  
- **架构影响**：  
  - 引入显式的资源管理（`cancellation_future`、`shutdown_task`），遵循 “获取即释放” 的原则，使请求处理的异常路径更加明确。  
  - 通过统一的 `finally` 清理，降低了分支路径中遗漏取消任务的风险，提升了系统的健壮性。  
- **性能影响**：  
  - 额外的 `await awaitable.cancel()` 与异常捕获开销极低，只有在请求被取消或服务关闭时才会触发，对常规请求的吞吐量几乎没有影响。  
  - 通过及时回收未完成的协程，避免了长时间持有的对象导致的堆内存增长，对长期运行的服务有正面性能收益。  
- **安全考虑**：  
  - 本次改动不直接涉及安全功能，但通过消除悬挂协程可以防止潜在的资源耗尽（Denial‑of‑Service）攻击向量，提升系统的可用性安全。  

**⚠️ 潜在风险**：  
1. **协程取消时的副作用**：如果外部代码在 `cancellation_future` 或 `shutdown_task` 完成前已经对相关资源做了清理，取消后可能触发额外的异常处理，需要确认业务代码在取消后仍能安全容错。  
2. **未初始化的 `cancellation_future`**：在极端异常路径（如 `self.shutdown_event` 为 `None` 且提前抛出异常）下，`cancellation_future` 仍可能为 `None`，但 `finally` 已做空值检查，风险已基本消除。  
3. **兼容性**：若已有自定义子类覆盖了 `_handle_cancellation` 实现，需确认其仍遵循相同的清理约定，否则可能出现重复取消或遗漏。  

**💡 关注建议**：  
- **单元测试覆盖**：添加包含请求正常完成、取消、以及服务 shutdown 三种情形的异步测试，验证 `cancellation_future` 与 `shutdown_task` 均能在 `finally` 中被正确取消。  
- **监控告警**：在生产环境开启内存使用监控，观察该改动后是否出现预期的内存回收曲线，防止其他未捕获的泄漏。  
- **文档更新**：在 SGLang 请求处理模块的开发文档中标注 “请求取消需要在 finally 中显式取消关联的 Future/Task”，帮助后续维护者避免再次出现泄漏。  
- **代码审查注意点**：审查新增加的 `await awaitable` 逻辑时，确认不会引入死锁（如在同一事件循环中同步等待），并确保异常捕获范围不吞掉关键错误信息。

**📋 变更摘要**  
本次提交在 `components/src/dynamo` 下新增 `thunderagent_router` 子模块，实现了 **ThunderAgent** 论文中的程序级调度器。它在 Dynamo 的原生 KV Router 之上，按 `nvext.agent_context.trajectory_id` 将一次完整的 “LLM turn → tool call → 下一轮” 视为一个 **Program**，在内存压力大时在工具边界 **pause** 程序、在资源空闲时 **resume**，从而显著降低 KV 缓存占用与重预填成本。新增的 CLI、配置解析、容量采集、程序状态模型以及完整的异步调度循环，使其能够直接作为 Dynamo 路由服务运行（`python -m dynamo.thunderagent_router`），并可选向前端注册模型。

### 🔍 技术洞察

| 维度 | 分析 |
|------|------|
| **架构影响** | 1. **新增服务层**：`thunderagent_router` 以独立的 Dynamo worker 形式出现，内部持有 `KvRouter` 实例并向前端注册模型，形成 “frontend → thunderagent_router → KvRouter → vLLM workers” 的三层链路。<br>2. **程序状态表**：`ProgramTable` 在内存中维护所有运行中的 program，加入 `lifecycle`（ACTIVE/PAUSED/TERMINATED）与 `status`（REASONING/ACTING）两维状态。<br>3. **容量感知**：`WorkerCapacityProvider` 通过 FPM 事件的 MDC（模型卡）实时获取每个 worker 的 KV Pool 大小，提供给调度器做“工作集-容量”比较。<br>4. **异步调度循环**：后台 `asyncio` 任务每 `scheduler_interval_seconds` 触发一次，执行 **软降级 → greedy resume → pause‑until‑safe** 三步，改变了原有的“请求级”调度模型。 |
| **性能影响** | - **正向**：论文报告在 8×H100 上对 SWE‑bench‑Lite 提升 12‑16% 吞吐，主要得益于：<br>  • 程序级 KV 缓存回收，降低重复 pre‑fill；<br>  • 在 tool‑boundary 暂停，避免不必要的 decode。<br>- **负向**：<br>  • 额外的 Python asyncio 调度、锁竞争、事件等待可能在高并发（>128）时成为瓶颈；<br>  • `WorkerCapacityProvider` 依赖 FPM 事件流，若订阅延迟或丢失，调度决策会基于过期/空白快照，导致误判。<br>  • 程序状态表随 program 数增长而线性膨胀，内存占用略增（每条记录约 200 B）。  |
| **安全考虑** | - **输入验证**：`program_id` 直接取自 `nvext.agent_context.trajectory_id`，未做长度或字符集合校验，可能被恶意构造极长或含特殊字符的 ID，导致内存耗尽或日志注入。<br>- **资源抢占**：调度器通过 `pause_until_safe` 将新请求置于 PAUSED 状态，若攻击者大量提交合法 `program_id`，可触发频繁 pause/resume，形成 **DoS**。<br>- **环境变量**：所有调度参数通过 envvar 暴露，若运行在多租户容器中，未做 whitelist 可能被旁路修改。<br>- **代码路径**：新入口 `dynamo.thunderagent_router.generate` 直接调用 `KvRouter.generate_from_request`，若 downstream worker 返回异常对象，缺少统一的异常屏蔽，可能泄露内部栈信息。 |
| **可维护性** | - 代码量大（≈2 k 行），但模块划分清晰：`args`, `capacity`, `program_state`, `router`, `__main__`。<br>- 使用了大量 `asyncio`、`Event`、自定义 `dataclass`，对 Python 异步不熟悉的贡献者上手难度提升。<br>- 单元测试覆盖了核心逻辑（capacity 解析、程序状态、调度入口），但缺少 **集成测试**（真实 vLLM worker 与前端交互）。<br>- `README.md` 已说明实验性并提供完整使用指南，降低误用风险。 |

### ⚠️ 潜在风险

1. **调度死锁 / 失效**：`_lock` 在 `before_request`、`after_request`、`_scheduler_tick` 之间反复获取，若某个路径抛异常而未释放，可能导致后续请求全部阻塞。  
2. **容量快照失效**：在 worker 启动初期或 MDC 更新滞后时，`WorkerCapacityProvider.snapshot()` 可能返回空 dict，导致所有新 program 被错误地标记为 PAUSED（或直接放行导致 KV 缓存溢出）。  
3. **软降级时间戳冲突**：`soft_demoted_until` 采用全局 “当前时间 + interval*1.5”，在高并发情况下会频繁刷新，同一 program 可能被反复软降级，导致优先级跳变异常。

| **架构影响** | 1. **后端抽象统一化**：`TransferBackendKind` Enum 把原有散字符串统一为强类型枚举，所有创建点均走 `create_transfer_backend`，降低拼写错误风险。<br>2. **后台准备并行化**：`_NixlPosixStagingTransferSession` 现在在后台线程池中提前创建 NIXL 代理、分配 pinned copy slots，并在 `restore()` 前完成准备，实现 “准备 ⇢ 恢复” 重叠，整体调用链更清晰。<br>3. **分片 SSD 队列拆分**：`_group_sources_by_root` 会依据 `sharded_ssd_queues_per_root` 将同一根目录的文件分配到多个逻辑工作组（`#q0`, `#q1`…），从而让 NIXL POSIX 后端能够利用多 NIXL agents 并行读取同一 SSD。<br>4. **Manifest 简化**：去除 `version` 字段，改为纯数据结构，减少不必要的校验。 |
| **性能影响** | 1. **启动/恢复时延**：避免在每个设备线程中 `import torch`，并提前加载 NIXL API，使 `loader` 的热路径更轻。<br>2. **更高并行度**：默认 `max-workers` 从 8→16（loader）/8→16（storage‑runner），以及 `sharded_ssd_queues_per_root` 默认 2，提升磁盘 → PCIe → GPU 的吞吐。<br>3. **更高的带宽利用**：`restore_file_groups_with_nixl_staging` 现在在一次 `close` 之后只释放 pinned slots一次，且 `slot.copy_to_device_async` 与 `slot.wait` 交叉使用，降低同步阻塞。<br>4. **内存分配重映射**：在 `remap_all_vas` 中批量 `cuda_synchronize` 与 `cuda_validate_pointer`，从 N * sync → 1 sync，显著降低 O(N) 同步成本。 |
| **安全考虑** | - 新增 `device_memory_info` 通过 NVML 查询设备内存，若系统未装 `pynvml` 会抛出 `ImportError`; 目前未捕获，可能导致服务启动失败。<br>- CLI 参数校验已加强（正整数检查、必传 checkpoint‑dir），降低用户误用导致的异常。<br>- 移除 `torch` 直接依赖，减少潜在的二进制兼容风险。 |
| **可维护性** | - 使用 `Enum`、类型注解、统一的日志模板，提高代码可读性与 IDE 自动补全支持。<br>- 大量单元测试覆盖新逻辑，确保未来改动不会轻易回归。<br>- 将 NIXL 相关的硬编码工作组划分抽离为 `split_work_groups`，便于单独调优或替换实现。 |

### ⚠️ 潜在风险  

1. **向后兼容性**  
   - **CLI 参数默认值变更**：`--max-workers`、`--sharded-ssd-queues-per-root` 默认提升，旧脚本在资源受限的环境（如容器 CPU 限额）可能触发 OOM / 过度调度。  
   - **Manifest 结构**：`SaveManifest` 不再包含 `version`，依赖该字段的外部工具（第三方备份、审计系统）会在解析时失败。  
   - **后端名称**：虽然 Enum 的 `value` 与原字符串保持一致，但直接使用 `TransferBackendKind` 类型的对象在旧代码中可能出现类型不匹配（如 `isinstance(..., str)` 检查）。  

2. **NVML 依赖**  
   - `device_memory_info` 直接 `import pynvml`。在没有 NVML 库的机器上会抛 `ImportError`，导致 GMS 启动或分配重试失败。  

3. **并行度提升导致资源争用**  
   - `max_workers=16` + `sharded_ssd_queues_per_root=2` 会在高并发时产生大量线程/文件描述符。如果系统 `ulimit -n` 较低，可能触发 “Too many open files”。  

4. **工作组划分算法**  
   - `split_work_groups` 采用 **最大‑先** 贪心分配，可能在极度不均衡的文件大小分布下出现轻微负载不均，但不影响功能正确性。  

5. **错误传播**  
   - 在 `_NixlPosixStagingTransferSession.restore` 中，后台 `prepare` 步骤抛异常时会立即 `set` 取消事件，但已提交的工作线程仍可能在 `as_completed` 循环中尝试 `restore_prepared_group`，需确认异常路径已全部关闭资源（已实现但需回归验证）。
