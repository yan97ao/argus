### feat(replay): add agg/disagg offline replay optimization [DYN-2566] (#7774)
**SHA**: `4ea2107` | 🔗 [查看提交](https://github.com/ai-dynamo/dynamo/commit/4ea21079672c1840670f40560574d60cf76081ee)

**🎯 变更类型**：功能增强  

**⚡ 重要程度**：🔴高  

**📋 变更摘要**  
- 在 `components/src/dynamo/profiler/utils/replay_optimize` 新增完整的离线 Replay 优化子系统，包括模型、搜索、评估、评分、对比以及 AIC（aiconfigurator SDK）集成，支持聚合（agg）和去聚合（disagg）两种调度模式的 TP、GPU 预算、路由器配置等多维度搜索。  
- 为 Python 绑定的 Rust 实现 `KvRouterConfig` 与 `MockEngineArgs` 添加了 JSON 序列化/反序列化、拷贝、属性 getter/setter、`with_overrides` 方法以及对 `overlap_score_weight` / `worker_type` 等字段的合法性校验。  
- 新增 900 行pytest 单元测试，覆盖搜索过程、约束惩罚、路由模式、缓存复用、Trace 与 Synthetic 工作负载等场景。  
- 调整 `MockEngineArgs` JSON 结构实现，使 `max_num_seqs`、`max_num_batched_tokens` 等可为 `null`，并去除已废弃的 `has_perf_model` 字段。  

**🎯 影响范围**  
- **Profiler 核心**：`dynamo.profiler.utils.replay_optimize` 包全新加入的 7 个模块（`__init__`, `aic`, `bench`, `constants`, `engine_args`, `evaluate`, `models`, `scoring`, `search`）。  
- **Python‑Rust 绑定**：`lib/bindings/python/rust/llm/entrypoint.rs`、`lib/bindings/python/rust/llm/replay.rs`、`lib/bindings/python/src/dynamo/_core.pyi` 以及底层 `mocker` 中的 `MockEngineArgs` 实现。  
- **外部依赖**：引入对 `aiconfigurator` SDK 的可选依赖；若未安装，会在运行时抛出明确的 `RuntimeError`。  
- **测试套件**：新增 `components/src/dynamo/profiler/tests/test_replay_optimize.py`，对搜索、评估与约束逻辑进行回归验证。

### fix(kv-router): compressed tree sizes accounting (#7800)
**SHA**: `b55277c` | 🔗 [查看提交](https://github.com/ai-dynamo/dynamo/commit/b55277c9e756f9911071d90f219e7f57b2fb6fac)

**🎯 变更类型**：Bug修复  
**⚡ 重要程度**：🔴高  
**📋 变更摘要**：在 `ConcurrentRadixTreeCompressed` 中，修正了树大小（tree size）统计的逻辑。`insert_blocks_from` 现在返回实际新增的块数，并仅在真正插入新键时计数；相应地在 `apply_event` 中使用该返回值更新 `tree_sizes`。同时补充了针对所有索引器变体（single、sharded、concurrent、concurrent_compressed）的单元测试，验证重复 store/remove 不会导致树大小膨胀或结构变化。

**🎯 影响范围**：  
- `lib/kv-router/src/indexer/concurrent_radix_tree_compressed.rs`（核心索引器实现）  
- `lib/kv-router/src/indexer/tests.rs`（测试套件，覆盖所有索引器实现）  

**🔍 技术洞察**：  
- **架构影响**：  
  - 改动局限于压缩并发 radix 树实现，属于索引器层面的内部计数机制，不会影响外部 API。  
  - 引入返回值 `usize` 使 `insert_blocks_from` 更具函数式语义，提升代码可读性与可组合性。  
  - 新增的统一测试模板 `tree_size_indexer_template` 扩展了 CI 覆盖，确保跨实现的一致性。  

- **性能影响**：  
  - 额外的 `if wl.insert(...).is_none()` 检查在插入阶段进行一次哈希查找，开销极小（已在同一锁/写入上下文中）。  
  - 计数改为基于实际新增块数，可避免因重复插入导致的 `tree_sizes` 虚增，从而在长时运行中降低不必要的内存统计与后续监控开销。  
  - `AtomicUsize::fetch_add` 仍使用 `Relaxed`，保持原有低成本原子计数；若对计数的可见性有更强需求，可考虑升级为 `AcqRel`。  

- **安全考虑**：  
  - 本次改动仅涉及内部状态计数与测试，不涉及网络、文件或加密路径，无直接安全风险。  
  - 由于计数依据 `lookup` 哈希表的插入结果，若未来 `lookup` 变为跨线程共享（目前为 per‑worker），应审慎评估竞争条件和潜在的双写导致的计数错误。  

**⚠️ 潜在风险**：  
1. **并发计数不对称**：在极端并发下，不同线程可能同时判断 `wl.insert(...).is_none()` 为 `true`（因为 `lookup` 仍局部），导致同一块被计数多次。当前实现仍然基于每个 `worker` 的独立 `lookup`，风险低，但若实现细节改变需重新评估。  
2. **原子计数顺序**：使用 `Relaxed` 可能在调试工具或跨线程观察时出现计数短暂不一致的现象，虽不影响功能，但可能导致监控数据瞬时波动。  
3. **测试覆盖边界**：新增测试重点验证“重复 store/remove”场景，未覆盖“中间链段删除后懒惰清理导致的计数偏差”。如果后续对清理策略做改动，可能重新出现计数不准的问题。  

**💡 关注建议**：  
- **代码层面**：  
  - 为 `insert_blocks_from` 添加文档注释，说明返回值语义以及计数行为，帮助后续维护者快速了解。  
  - 考虑将 `AtomicUsize::fetch_add` 的内存序升级为 `Ordering::AcqRel`，在需要对计数进行跨线程可见性检查时提供更强保障。  
- **测试层面**：  
  - 增加一组针对“中间链段删除后重新插入” 的回归测试，以捕获当前已知的懒惰清理计数差距。  
  - 在 CI 中开启对 `tree_sizes` 的随机并发重复插入场景（使用 `tokio::spawn` 多任务）以验证计数在高并发下的稳健性。  
- **运维层面**：  
  - 监控 `KvIndexerMetrics.tree_sizes` 的增长趋势，确保在生产环境中计数不出现异常跳升。  
  - 若发现计数与实际键数不匹配，可快速定位是否是新的并发路径导致的重复计数。  

此修复提升了压缩并发 radix 树的计数准确性，降低了因重复事件导致的统计误差，对系统的可观测性和资源使用监控具有积极意义。

### fix: tool_choice=required bypasses format-specific parsers (#6821) (#7589)
**SHA**: `07af8dc` | 🔗 [查看提交](https://github.com/ai-dynamo/dynamo/commit/07af8dca3ef34aad1c4ef85add4af23482d8a580)

**🎯 变更类型**：Bug修复 / 功能增强  
**⚡ 重要程度**：🔴 高  

**📋 变更摘要**  
1. 修复 `tool_choice=required` 在有 `tool_call_parser` 时仍走 Immediate JSON 模式，导致格式特定解析器（如 qwen3_coder XML）失效的问题。  
2. 在 `tool_choice=named` 场景下加入 `named_tool_filter`，强制仅保留符合指定工具名称的解析结果，并在标记式 Jail 中使用。  
3. 相应更新 `JailedStream` 的实现：新增 `named_tool_name`、过滤逻辑、finish‑reason 处理以及 Builder 接口。  
4. 增加覆盖上述行为的回归测试，验证解析器、过滤及 finish‑reason 的正确性。

**🎯 影响范围**  
- `lib/llm/src/preprocessor.rs`（OpenAIPreprocessor 决策层）  
- `lib/llm/src/protocols/openai/chat_completions/jail.rs`（JailedStream 核心实现与 Builder）  
- 单元测试 `lib/llm/tests/test_jail.rs`、`lib/llm/tests/tool_choice.rs`  
- 受影响的模块：OpenAI 兼容的聊天完成协议、工具调用（tool calling）子系统、运行时解析器插件（如 qwen3_coder、hermes）。

**🔍 技术洞察**  

- **架构影响**  
  - 引入 `named_tool_name` 作为 `JailedStream` 状态，保持向后兼容（默认 `None`）。  
  - 在 `OpenAIPreprocessor` 中根据是否配置 `tool_call_parser` 选择 **MarkerBased**（标记式）或 **Immediate**（直接 JSON）模式，避免硬编码路径。  
  - 新增 Builder 方法 `named_tool_filter`，实现 “tool_choice=named + parser” 的闭环，使高层 API 能直接表达业务意图。  
  - 通过 `named_tool_active` 标记在 `fix_finish_reason` 中区分 `Stop`（named）与 `ToolCalls`（required） 的 finish‑reason，符合 OpenAI 规范。  

- **性能影响**  
  - 额外的过滤步骤（遍历 `tool_calls` 并比对名称）在最坏情况下会增加 O(N) 的开销，其中 N 为解析出的工具调用数量。通常 N 很小（1‑2），对整体流式响应延迟影响可忽略。  
  - 采用标记式 Jail 时仍需要完整的解析器运行，和之前的 Immediate 模式相比成本相当，但显式分支避免了错误路径导致的后续二次处理（如泄漏原始 XML），整体响应质量提升。  

- **安全考虑**  
  - 修复后原始 XML/自定义格式不再直接作为内容泄漏，降低了下游对未解析数据进行误处理或注入攻击的风险。  
  - `named_tool_filter` 只在 parser‑aware 路径下生效，避免因误过滤导致意外的空响应或错误的 finish‑reason。  

**⚠️ 潜在风险**  

1. **误过滤导致功能失效**  
   - 如果模型在 `tool_choice=named` 场景中误返回工具名称（大小写、前后空格不同），当前过滤会将其全部丢弃，导致用户侧看不到任何工具调用。  
2. **兼容性回退**  
   - 旧的自定义 `tool_call_parser` 实现若未完全遵守标记式 Jail 的输入/输出约定，可能在新路径下产生解析错误或空调用。  
3. **Finish‑reason 差异**  
   - `named_tool_active` 为 true 时统一使用 `Stop`；若外部系统依赖 `ToolCalls` finish‑reason 来判断工具调用结束，可能需要适配。  
4. **Builder 使用错误**  
   - 开发者若在没有配置 `tool_call_parser` 的情况下仍调用 `named_tool_filter`，会得到 Immediate JSON 模式且过滤不会生效，可能产生意料之外的输出。  

**💡 关注建议**  

- **测试升级**：在 CI 中加入对大小写/空格变化的工具名称容错测试，确保 `named_tool_filter` 不因细微差异误过滤。  
- **文档说明**：在 `OpenAIPreprocessor` 与 `JailedStreamBuilder` 文档中明确阐述 `named_tool_filter` 仅在 *parser‑aware*（标记式）路径下生效，并说明对 `finish_reason` 的行为差异。  
- **监控与告警**：在生产环境记录 `named_tool_active` 且 `tool_calls.is_empty()` 的警告日志，以便快速定位模型返回错误工具名称的情况。  
- **向后兼容**：若已有用户依赖 `tool_choice=named` 且未使用解析器，保持默认 Immediate JSON 行为不变；但建议在迁移期显式声明 `tool_call_parser=None` 以避免隐式切换。

- **性能影响**  
  - **缓存命中**：在同一批次或跨批次出现相同图片 URL 时，可显著减少 GPU 编码次数，降低显存占用与推理时延。单测展示了部分命中情况下仅一次远程编码调用。  
  - **CPU 内存占用**：嵌入向量以 `torch.Tensor`（CPU）形式存放，受配置容量限制；大规模缓存可能导致系统内存压力，需要监控。  
  - **额外开销**：哈希计算、LRU 查表、张量分割与拼接的成本相对编码本身可忽略不计；但在全部未命中且图片数量极大时，会产生一次额外的 `torch.split` 与 `torch.cat`，对 CPU 产生轻微负担。  

- **安全考虑**  
  - **哈希冲突**：使用 blake2b（256 位）几乎杜绝冲突风险，冲突导致缓存错误覆盖概率可视为可接受。

**⚠️ 潜在风险**  

1. **内存泄漏 / OOM**  
   - 若缓存容量配置过大且嵌入维度较高，累计的 CPU Tensor 可能耗尽系统内存。  
2. **并发竞争**  
   - LRU 实现若未加锁，在高并发请求下可能出现写入竞争，引发缓存状态不一致或异常。  
3. **缓存失效一致性**  
   - 代码假设同一 URL 对应的视觉内容不变；若 URL 指向动态图片（如 CDN 变更），旧缓存将被错误复用，导致生成内容偏差。  
4. **设备不匹配**

5. **错误的 token 切分**  
   - `_split_token_counts` 依赖 `total_grid_tokens % total_tokens == 0` 条件；若未来模型输出的 token 计数规则变化，切分将抛异常，导致服务不可用。

- **容量监控**：在服务启动时打印缓存容量上限，运行时监控 `process_memory_rss`，并在监控系统中添加 “embedding cache memory usage” 指标。  
- **并发防护**：为 `MultimodalEmbeddingCacheManager` 增加 `asyncio.Lock` 包装的 `get/set`，或改用 `threading.RLock`（结合 `asyncio.to_thread`）确保多协程安全。  
- **缓存失效策略**：考虑在配置中加入 TTL（时间到期）或基于 HTTP `ETag/Last-Modified` 的可选校验，以防止动态图片导致的错误复用。  
- **文档与默认值**：在 README/配置文档中明确 “默认关闭缓存”，并给出推荐的容量（如 2~4 GB）和典型的缓存命中率预估。  
- **回退机制**：在捕获 `_split_token_counts` 或缓存读取异常时，回退到直接调用 `_encode`，确保单点故障不影响整体服务。  
- **性能基准**：建议在真实工作负载下做 A/B 对比（开启 4 GB 缓存 vs 关闭），记录编码时延、GPU 利用率、CPU 内存峰值，以量化收益并指导容量调优。

  - 该改动不涉及网络、权限或加密等安全机理。唯一需关注的是 **配置一致性**：如果不同节点或进程在获取 `block_size` 前已经基于错误的默认值分配资源，可能出现资源不匹配导致的错误或崩溃。当前实现在引擎创建完毕后立即同步，降低了该风险。

**⚠️ 潜在风险**：

1. **下游代码仍假设 `block_size` 已经在 `EngineConfig` 中存在**  
   - 任何在 `setup_vllm_engine` 之前访问 `engine_config.block_size` 的逻辑可能会得到 `None`，导致异常或错误的内存计算。  
2. **测试或脚本中硬编码的 `--block-size` 参数**  
   - 老的 CI / 本地脚本仍可能显式传递 `--block-size 16`，在新环境下可能与运行时推导值不一致，产生资源冲突。  
3. **跨进程共享的缓存配置不一致**  
   - 在多进程/多节点部署时，如果某些进程在获取运行时 `block_size` 前已经启动缓存相关的子模块，可能出现不同 `block_size` 的实例，导致 KV‑Cache 不兼容。

**💡 关注建议**：

- **代码层面**  
  1. 在所有需要 `block_size` 的入口处添加容错检查（如 `engine_config.block_size or runtime_cache.block_size`），确保在配置尚未同步前不会抛异常。  
  2. 将 `block_size` 的获取封装为公共函数（如 `resolve_block_size(engine_config, runtime_info)`），统一逻辑，防止散落的硬编码。  

- **测试/CI**  
  1. 更新所有旧的 vLLM 相关 CI 脚本，去除或改为可选的 `--block-size` 参数，确保在没有显式设置时能够使用新默认路径。  
  2. 将新增的 “without block size” 测试设为 **必跑**（非 pre‑merge），以防未来回退到硬编码行为。  

- **运维部署**  
  1. 在生产部署文档中明确说明：`block_size` 如未在配置文件中显式指定，将由后端根据模型自动决定，且该值会在 engine 启动后写回 `vllm_config.cache_config.block_size`，用于后续监控和调度。  
  2. 对已有的监控/报警系统检查是否对 `block_size` 有硬编码阈值，需要相应放宽或改为动态读取。  

通过上述措施，可最大化利用此次修复带来的灵活性，同时将潜在的配置不一致风险控制在可接受范围内。

#### 架构影响
- **统一请求层**：在 HTTP 前端与后端预处理之间增设一层抽象，使不同协议的专有字段不再“消失”。整体架构从 “多协议 → OpenAI Chat” 的单向 funnel 变为 “多协议 → UnifiedRequest → (保持原有 Funnel)”。  
- **上下文保留**：`ApiContext` 通过枚举将协议来源及其特有字段绑定到请求实例，后续流转换器、响应包装器可以安全、显式地获取这些信息，避免了之前依赖全局变量或硬编码的“猜测”。  
- **最小侵入式迁移**：`UnifiedRequest` 实现了所有原有 `NvCreateChatCompletionRequest` 需要的 trait，几乎不需要修改现有预处理逻辑，仅在服务入口处包装/解包装即可，迁移成本低。

#### 性能影响
- **内存开销**：新增 `UnifiedRequest` 包含一次完整的 `NvCreateChatCompletionRequest`（已在系统中存在）以及少量额外结构体（`AnthropicContext`、`ResponsesContext`、缓存断点向量）。大多数字段为 `Option` 或 `Vec`，在不使用时几乎为零开销。整体复制/移动成本与之前直接使用 `NvCreateChatCompletionRequest` 相比提升 < 5%。

- **用户可控字段**：`AnthropicContext::metadata`、`ResponsesContext::include` 等直接来源于请求体，若后端错误地将其原样返回或写入日志，可能泄露用户敏感信息。代码已在响应层仅在需要时读取（如 `store`、`previous_response_id`），但仍建议：
