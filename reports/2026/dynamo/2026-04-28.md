### feat(snapshot): CRI-O support + OpenShift chart enablement (#8427)
**SHA**: `9c9a02e` | 🔗 [查看提交](https://github.com/ai-dynamo/dynamo/commit/9c9a02e83c3c6a10c263a4510d012b0f300fd90f)

**🎯 变更类型**：功能增强 / 重构 / 架构变更  
**⚡ 重要程度**：🔴 高  
**📋 变更摘要**：  
- 为 **Snapshot** Helm Chart 添加了对 **CRI‑O** 运行时及 **OpenShift** 环境的原生支持（runtime.type、runtime.socketPath、openshift.enabled）。  
- 抽象出容器运行时接口 `runtime.Runtime`，实现 `ContainerdRuntime` 与 `CRIORuntime` 两套后端，统一解析容器 PID 与 OCI spec。  
- 更新 Agent 启动参数、控制器、执行器以及 Helm 模板以适配新运行时；删除旧的 `oci.go`，改为通用实现并在 `runtime.go` 中提供公共工具（路径规范化、OCI 管理路径收集、CRIScheme 剥离）。  
- 新增 OpenShift‑专用 RBAC 与 SCC 注解，完善 Chart README 与文档。  

**🎯 影响范围**：  
- `deploy/helm/charts/snapshot/*`（Chart 模板、README、values）  
- Snapshot Agent 二进制（`cmd/agent/main.go`、内部控制器、执行器）  
- 新增 runtime 包 `deploy/snapshot/internal/runtime/*`（容器运行时抽象层）  
- 相关测试代码与依赖（新增 `k8s.io/cri-api`、`k8s.io/cri-client` 等）

### 架构影响  
| 维度 | 影响描述 |
|------|----------|
| **运行时抽象层** | 将原本硬编码的 `containerd` 客户端抽离为统一的 `runtime.Runtime` 接口，新增 `ContainerdRuntime` 与 `CRIORuntime` 实现，实现了 **依赖倒置**，未来可以再平滑加入其他 CRI（如 Docker、container‑engine‑shim）而无需改动业务逻辑。 |
| **配置驱动** | Helm `values.yaml` 现在暴露 `runtime.type`、`runtime.socketPath`、`openshift.enabled` 三个键，Chart 模板通过 `snapshot.validateRuntime` 进行提前校验，防止在部署阶段出现不可用配置。 |
| **OpenShift 支持** | 通过 `openshift.enabled` 自动生成对内置 **privileged SCC** 的 `Role/RoleBinding`，并在 DaemonSet 注解中加入 `openshift.io/required-scc: privileged`，保证在 OCP 环境下仍能获得必须的特权。 |
| **代码组织** | 删除 `oci.go`（仅 containerd 实现）后，新文件 `oci_containerd.go`、`oci_crio.go` 分别实现两套后端；公共工具被迁移到 `runtime.go`，避免重复实现。 |

### 性能影响  
| 维度 | 正面影响 | 负面/注意点 |
|------|----------|------------|

### 安全考虑  
- **特权需求**：Snapshot Agent 仍需 `privileged` 权限（hostPID、hostIPC、hostNetwork）。在 OpenShift 中通过 `openshift.io/required-scc: privileged` 强制使用内置 SCC，避免因自定义 SCC 优先级导致权限不足。  
- **RBAC 生成**：仅在 `openshift.enabled && .Values.rbac.create` 为 true 时生成 `Role/RoleBinding`，默认不启用，降低不必要的权限扩大。  
- **Socket 路径暴露**：`runtime.socketPath` 通过 Helm 可自定义，若误配置指向不受信任的 socket（如宿主机的其他服务），可能导致非授权进程被 Snapshot Agent 通过 CRI 接口操作。建议在生产环境通过 `PodSecurityPolicy` / `OPA` 检查此路径的文件系统权限。  
- **输入校验**：新增 `snapshot.validateRuntime` 在模板渲染阶段即失败，防止错误 runtime.type 进入集群，提升防御深度。

## ⚠️ 潜在风险  

1. **运行时选择错误**  
   - 若 `runtime.type` 与实际节点运行时不匹配（例如声明 `crio` 而节点使用 `containerd`），Agent 将在启动时因 socket 连接失败而退出，导致 Snapshot 功能不可用。  
   - **建议**：在 CI/Helm lint 阶段加入 `kubectl get node -o jsonpath='{.status.nodeInfo.containerRuntimeVersion}'` 检查，或在 Helm `post-install` Hook 中动态检测并警告。  

2. **CRIO gRPC 超时**  
   - `crioCallTimeout` 设为 10 s，若节点网络或 CRI‑O 服务卡顿，checkpoint/restore 可能因超时返回错误，导致业务回滚或挂起。  
   - **建议**：提供可覆盖的 Helm 参数（如 `runtime.callTimeout`) 让运维根据集群特性调节。  

3. **OpenShift SCC 竞争**  
   - OpenShift 中如果集群已经为项目自定义了更高优先级的 SCC，`openshift.io/required-scc: privileged` 注解仍会强制使用 `privileged`，但如果自定义 SCC 明确限制 `privileged`（通过 `allowPrivilegedContainer: false`），Agent 仍可能被拒绝。  
   - **建议**：在 `README` 中提醒用户检查项目的 SCC 限制，或在 Chart 中提供 `openshift.sccName` 参数以自定义 SCC 名称。  

4. **依赖升级冲突**  
   - 新增 `k8s.io/cri-api` 与 `k8s.io/cri-client` 可能与集群所使用的 Kubernetes 版本产生微小不兼容（如 API 字段变更）。虽然使用 `v0.34.3` 对齐当前项目的 k8s 依赖，但在后续升级到更高 K8s 版本时需同步更新。  

5. **路径规范化安全**  
   - `normalizeOCIPath` 通过 `filepath-securejoin` 防止路径穿越，但仍依赖容器根文件系统的真实挂载结构。若容器根目录被恶意挂载到宿主机的敏感目录（例如 `/proc` → `/host/proc`），仍可能导致 CRIU 访问不应暴露的文件。  
   - **建议**：在 Snapshot 前做一次 **路径白名单** 检查，仅允许已知的 OCI 管理路径（/proc, /sys, /etc/hosts 等）进入 `collectOCIManagedPaths`。

### feat: add reusable telemetry primitives (#8788)
**SHA**: `da525a7` | 🔗 [查看提交](https://github.com/ai-dynamo/dynamo/commit/da525a7609009aef5ba1a811796f7e36993af22a)

**🎯 变更类型**：功能增强 / 重构  
**⚡ 重要程度**：🔴高  
**📋 变更摘要**：本次提交为 `ai-dynamo/dynamo` 引入了一套可复用的遥测（telemetry）基础设施，包括通用的 `TelemetryBus`、JSONL 持久化工作者以及流完成通知工具。同时对现有审计总线改为使用新总线，并对 `Recorder` 增加可配置选项（文件轮转、缓冲、刷新间隔等），提升了可维护性和灵活性。  

**🎯 影响范围**：  
- `lib/llm/src/audit/bus.rs`（审计总线改造）  
- `lib/llm/src/recorder.rs`（录制器 API 扩展）  
- 新增模块 `lib/llm/src/telemetry/*`（bus、jsonl、stream）  
- `lib/llm/src/lib.rs`（公开 `telemetry` 模块）  

**🔍 技术洞察**：

- **架构影响**  
  - 引入 **统一的遥测总线** (`TelemetryBus<T>`) ，取代了之前散落的 `OnceLock<broadcast::Sender<_>>` 实现，形成了 **Typed、单例、可复用** 的内部发布/订阅层。  
  - `TelemetryBus` 通过 `OnceLock` 保证一次初始化，兼容“先 publish 后 init”场景，降低了初始化顺序的耦合。  
  - `Recorder` 通过 `RecorderOptions` 解耦了硬编码参数（如缓冲大小、文件轮转阈值），实现 **配置即行为**，对上层调用者更加友好。  
  - 新增 **JSONL Sink**（`spawn_jsonl_worker_with_shutdown`）将总线数据异步写入磁盘，配合 `CancellationToken` 实现优雅关闭，完善了遥测的持久化路径。  
  - `stream::notify_on_completion` 为异步流提供 **完成通知**（drop 或结束时）机制，可在上层实现资源回收或后置统计，提升了流式处理的可观察性。  

- **性能影响**  
  - `TelemetryBus` 复用了 `tokio::sync::broadcast`，保持 **O(1) 复制**（内部使用 `Arc`），对吞吐量影响极小。  
  - `Recorder` 现在支持 **自定义缓冲区大小** 与 **定时刷新**（`flush_interval`），可根据业务需求调优 I/O，降低磁盘写入频率，提升写入峰值吞吐。  
  - 实现了 **轮转文件**（基于行数或计数）和 **最大时间** 限制，防止单文件过大导致磁盘 I/O 性能下降。  
  - `jsonl` sink 在关闭期间会 **尝试一次性 flush 剩余记录**，确保数据完整性且不会产生长时间阻塞。  

- **安全考虑**  
  - 代码保持 **只读/写** 权限在 `OpenOptions` 中明确设置，`append` 选项可防止意外覆盖已有遥测文件。  
  - 所有跨线程共享的结构均要求 `T: Clone + Send + 'static`，避免数据竞争与未定义行为。  
  - `TelemetryBus::publish` 对未初始化的总线采用 **安全的空操作**（不会 panic），降低因初始化顺序错误导致的运行时异常。  
  - `jsonl` sink 在接收阶段对 `RecvError::Lagged` 做日志告警，防止因背压导致的记录丢失被忽视。  

**⚠️ 潜在风险**：  
1. **单例全局状态**：`TelemetryBus` 仍是 `static` 单例，若在多进程/插件环境中重复加载库可能导致 **重复初始化冲突**。  
2. **缓冲区与刷新间隔不匹配**：过小的 `buffer_bytes` 配合极短的 `flush_interval` 可能导致 **频繁磁盘写入**，削弱性能提升。需在生产环境进行基准调优。  
3. **文件轮转策略**：当前轮转仅基于行数/计数/时间，不考虑文件大小或磁盘空间，极端情况下可能仍产生大文件。  
4. **错误处理**：`Recorder` 中对 `writer.flush().await` 错误仅记录日志，未上报错误，可能在磁盘写满或权限错误时导致数据丢失而不被感知。  

**💡 关注建议**：  
- 在项目初始化阶段统一调用 `TelemetryBus::init`，并在文档中说明 **只能初始化一次**，避免隐藏的竞争。  
- 为不同业务场景提供 **推荐的 RecorderOptions**（如日志级别、监控指标）以及对应的基准测试结果，帮助用户快速选型。  
- 考虑在 `Recorder` 中加入 **错误回调或监控报警**，在写入失败时触发上层的异常处理逻辑。  
- 若计划在多实例或插件化场景使用，评估将 `TelemetryBus` 改为 **依赖注入**（传入实例）而非全局单例，以提升可测试性和隔离性。  
- 在 CI 中加入对 **TelemetryBus**、**JSONL sink**、**stream 完成通知** 的集成测试，覆盖发布后未初始化、背压、关闭流程等边界情况，确保可靠性。

**🎯 影响范围**  
- **CI/CD 流水线**：所有涉及框架镜像（vllm、sglang、trtllm、dynamo planner、frontend）的构建、测试、合规、拷贝均受影响。
