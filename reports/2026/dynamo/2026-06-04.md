### feat(tests): XPU router e2e test support with TOCTOU port fixes (#8675)
**SHA**: `983a0a8` | 🔗 [查看提交](https://github.com/ai-dynamo/dynamo/commit/983a0a8c9cbb0ff5c911bdee4392d7ce76abc205)

**🎯 变更类型**：功能增强 / 重构  
**⚡ 重要程度**：🔴高  
**📋 变更摘要**：  
- 为测试套件引入 `_create_kv_router_with_timeout`，在独立守护线程中创建 `KvRouter` 并轮询 worker 存活，以避免 Rust FFI 调用因 worker 崩溃或端口竞争而导致的永久阻塞。  
- 新增 XPU（Intel GPU）专用的端到端路由测试 `test_router_e2e_with_vllm_xpu.py`，包括设备可见性、TOCTOU‑安全端口分配、延长超时以及针对 XPU 的 block‑size 调整。  
- 在部分通用测试函数中加入 `initial_wait` 参数，以兼容 XPU 启动延迟。  

**🎯 影响范围**：  
- `tests/router/common.py`（核心测试工具类）  
- `tests/router/e2e_harness.py`（运行器包装）  
- 新增 `tests/router/test_router_e2e_with_vllm_xpu.py`（XPU e2e 测试）  
- 受影响的模块：`KvRouter`（Rust FFI 调用的 Python 包装）、测试进程管理、端口分配与健康检查逻辑。

**🔍 技术洞察**  

- **架构影响**  
  - **测试层解耦**：通过在守护线程中创建 `KvRouter`，把阻塞的 FFI 调用与 pytest 主线程解耦，提升整个 CI 流水线的鲁棒性。  
  - **TOCTOU 防护**：在 XPU 测试中显式使用 `health_check_ports=[kv_event_port]`，让 harness 在端口真正绑定后才继续，消除了 “端口预占‑使用” 的竞态。  
  - **设备可见性抽象**：引入 `ZE_AFFINITY_MASK` 环境变量的解析，使得 XPU 卡的分配逻辑与 CUDA 的 `CUDA_VISIBLE_DEVICES` 相对应，保持跨硬件平台的一致性。

- **性能影响**  
  - **额外轮询开销**：`_create_kv_router_with_timeout` 每 2 秒一次 `join` 检查，线程本身极轻，且仅在测试启动阶段运行，对整体执行时间的影响可忽略（仅在卡启动慢的 XPU 场景下略有 1‑2 s 额外延迟）。  
  - **更长的测试超时**：XPU 测试将超时提升至 600 s，以容忍模型加载的慢速路径，这会占用 CI 资源更久，但换来更可靠的验证。

- **安全考虑**  
  - 变更仅限于测试代码，未触及生产逻辑。  
  - 使用守护线程意味着即使 `KvRouter` 卡在 FFI 调用内，进程退出时线程会被强制终止，避免资源泄露。  
  - 新增的端口健康检查防止端口冲突导致的 “EADDRINUSE” 未捕获异常，提升了错误可见性。

**⚠️ 潜在风险**  

1. **残留阻塞线程**：如果 `KvRouter` 在守护线程里长时间卡住且测试进程因超时提前退出，可能留下未清理的 ZMQ / etcd 句柄，影响后续测试的端口占用。  
2. **误判 worker 死亡**：轮询逻辑仅检查 `proc.poll()`，若 worker 进入卡死但进程仍存活（未退出），测试会继续，可能掩盖潜在的稳定性问题。  
3. **环境变量依赖**：XPU 测试强制要求 `ZE_AFFINITY_MASK` 正确设置；在不具备 XPU 硬件的 CI 环境中运行会快速失败。  
4. **超时设置不当**：`timeout=120`（默认）对某些极端慢启动的模型可能仍不足，引发不必要的 `RuntimeError`。  

**💡 关注建议**  

- **资源清理**：在 `tearDown` 或 `pytest` 的 `finalizer` 中显式调用 `router_thread.join(timeout=0)`，确保即使卡住也能及时释放资源。  
- **增强死亡检测**：可在轮询时加入 ZMQ 心跳或 Etcd/Redis 键的健康检查，以捕获 “进程未退出但失去服务” 的情形。  
- **可配置超时**：将 `timeout` 参数提升为可通过环境变量或 pytest 参数配置的值，方便在不同硬件/模型下调优。  
- **文档与 CI 标记**：在项目文档中明确标记 XPU 测试为 “hardware‑specific”，并在 CI 中使用 `only-if` 条件（如 `runs-on: [self-hosted, xpu]`）避免在普通 GPU/CPU runner 上触发。  
- **回归测试**：在提交前运行完整的 GPU、CPU 与 XPU 测试矩阵，确认引入的守护线程不影响已有的 CUDA/CPU 路由测试的通过率。  

---  

此轮改动显著提升了 Dynamo 路由器测试的可靠性，尤其在 XPU 环境下避免了 TOCTOU 导致的卡死情形，同时保持了对现有 GPU/CPU 测试的兼容。通过适度的风险控制与后续的监控改进，可将这些增强平滑地推广至主分支。

### feat(xpu): enable multimodal CI tests for XPU (#9611)
**SHA**: `4827625` | 🔗 [查看提交](https://github.com/ai-dynamo/dynamo/commit/48276259440a0edaa3e851c267e5d2ee8dad9c2d)

**🎯 变更类型**：功能增强  

**⚡ 重要程度**：🔴高  

**📋 变更摘要**：  
1. 为 XPU CI 工作流打开了多模态（图片+视频）测试，调整了 workflow 标记、开启 Git‑LFS 下载并在 CI 中启动媒体服务器。  
2. 在 `tests/serve` 中加入对视频文件的加载、环境变量 `DYN_MM_ALLOW_INTERNAL` / `DYN_MM_LOCAL_PATH` 的配置，以及更宽松的超时/缓存限制，以支撑 Qwen 系列大模型的多模态推理。  

**🎯 影响范围**：  
- `.github/workflows/pr-xpu.yaml`、`.github/workflows/xpu-ci.yaml`（CI 流水线）  
- `tests/serve/conftest.py`、`tests/serve/test_vllm_xpu.py`（单元/集成测试）  
- `dynamo/common/utils/paths.rs`（引用 `WORKSPACE_DIR`）  
- 相关的 CI 依赖（Git‑LFS、Docker 镜像）  

**🔍 技术洞察**  

- **架构影响**  
  - 引入了在测试阶段通过本地 HTTP 服务器提供图片/视频资源的子系统，属于测试层面的临时服务，对生产代码无直接影响。  
  - 新增的环境变量 `DYN_MM_ALLOW_INTERNAL` 与 `DYN_MM_LOCAL_PATH` 让后端在 XPU 环境下能够直接访问工作区文件，若在非 CI 环境误用，可能导致意外的本地文件泄露。  
  - CI workflow 中 `lfs: true` 使得大文件（视频）在 checkout 时自动拉取，增加了镜像大小和网络 I/O，需确保 runner 的磁盘配额充足。  

- **性能影响**  
  - 多模态模型（尤其是视频）加载时间显著提升，测试超时从 220 s 增至 360 s，整体 CI 时长预计增加 30%‑50%。  
  - 启动本地 `HTTPServer` 并在每次测试中读取完整视频文件会占用额外的 CPU/IO，若并发跑多套 XPU 测试，可能出现端口竞争或资源争用。  
  - 为避免 “TOCTOU” 端口分配竞争，新增了仅在标记 `multimodal` 的测试中才启动 `image_server` 的 guard，降低非多模态用例的开销。  

- **安全考虑**  
  - 测试代码通过 `http://localhost:{port}` 暴露本地文件，仅在 CI 环境内部可达，风险有限。  
  - 依赖 Git‑LFS 拉取的视频文件如果出现指针文件（`version ` 前缀），会被错误地返回；新增的检查 `if not video_data.startswith(b"version ")` 防止此类泄漏。  
  - 环境变量 `DYN_MM_LOCAL_PATH` 将工作区绝对路径暴露给被测进程，若 CI 镜像被外部使用，可能导致路径遍历或未受限的文件访问，需要在镜像层面限制权限。  

**⚠️ 潜在风险**  

1. **CI 资源耗尽**：拉取大文件、延长超时、启动额外 HTTP 服务器，可能导致 GitHub Actions 的磁盘或时间配额被快速耗尽。  
2. **测试不确定性**：视频文件加载速度受网络/磁盘 I/O 影响，偶发的超时或端口冲突仍可能出现。  
3. **环境变量泄漏**：`DYN_MM_LOCAL_PATH` 在生产容器中若被误用，会把宿主机路径暴露给容器进程。  
4. **回归风险**：新增的多模态标记未在所有 XPU 相关测试中统一使用，可能出现未覆盖的老测试仍使用旧的 `pytest_markers`（如 `pre_merge`），导致 CI 行为不一致。  

**💡 关注建议**  

- **CI 配额监控**：在 XPU workflow 中加入磁盘使用和运行时长的监控告警，防止因 LFS 拉取导致的配额超限。  
- **资源隔离**：考虑在 CI 中为多模态测试单独配置更大的 runner（如 `large` 或 `xlarge`），或将视频文件抽象为 Mock 数据以降低 I/O。  
- **安全审计**：限制生产镜像中 `DYN_MM_ALLOW_INTERNAL`/`DYN_MM_LOCAL_PATH` 的默认值，确保只有 CI 环境开启。  
- **测试可靠性**：为 `image_server` 添加端口复用重试逻辑，并在测试结束后显式关闭服务器，防止端口残留。  
- **文档同步**：更新 CI 文档，说明开启 LFS、所需磁盘空间、以及如何在本地运行多模态 XPU 测试（包括 `export DYN_MM_ALLOW_INTERNAL=1`）。  
- **回归检查**：在 PR 模板中加入 “是否需要 multimodal 标记” 检查，避免遗漏导致测试漏跑或误跑。  

通过上述措施，可在保持新功能（XPU 多模态 CI）价值的同时，将资源使用、可靠性和安全风险控制在可接受范围内。

### fix(trtllm): summarize KV event debug logs (#10298)
**SHA**: `0d5699f` | 🔗 [查看提交](https://github.com/ai-dynamo/dynamo/commit/0d5699f0d2a8b04c5c5f7a1e7acd5799d937a202)

**🎯 变更类型**：功能增强 / Bug修复  
**⚡ 重要程度**：🔴高  

**📋 变更摘要**  
1. 为 TRT‑LLM KV 缓存事件引入完整的可观测性：新增 `KV_EVENT_BUFFER_CAPACITY`、`KV_EVENT_DRAIN_BATCH_SIZE`、`KV_EVENT_ID_GAP_EVENTS_TOTAL` 指标，并在 `AdditionalMetricsCollector` 中实现对应的上报逻辑。  
2. 将 KV 事件缓冲区默认上限从 1 024 提升至 **100 000**，并通过 `kv_cache_config.event_buffer_max_size` 进行显式配置与校验。  
3. 修复并强化 KV 事件处理：改为 **按 attention‑DP rank** 维护独立的 `last_event_id`，避免跨 rank 产生误报；对非全局注意力层的事件进行过滤后不再误计缺失。  
4. 大量日志改写：在多种请求处理入口（Embedding、Image Diffusion、Multimodal、TRT‑LLM）中加入结构化、尺寸感知的调试信息，提升故障排查效率。  
5. 新增/完善单元测试覆盖 KV 事件指标、异常路径以及日志配置行为，确保行为在 CI 中可验证。  

**🎯 影响范围**  
- `components/src/dynamo/trtllm/*`（engine、metrics、publisher、request_handlers、workers）  
- `components/src/dynamo/sglang/request_handlers/*`（embedding、image_diffusion）  
- `components/src/dynamo/vllm/*`（multimodal handlers）  
- `lib/bindings/python/src/dynamo/prometheus_names.py`、`runtime/src/metrics/prometheus_names.rs`  
- `lib/bindings/python/src/dynamo/runtime/logging.py`（日志桥接）  
- 测试套件（`trtllm` 相关测试、日志配置测试）  

**🔍 技术洞察**  

- **架构影响**  
  - **KV 事件缓冲层**：引入显式可配置的缓冲区大小，`Engine` 在构造时解析 `kv_cache_config` 并把 `event_buffer_max_size` 传递给 `Publisher`，实现 **配置即行为**。  
  - **分层事件 ID 追踪**：改为 `self._last_engine_event_id_by_rank`（`dict[int, int]`），解决多 attention‑DP 并行场景下的误报，提升 **分布式一致性监控**。  
  - **Metrics 集成**：在 `AdditionalMetricsCollector` 中新增 **Gauge**（容量）和 **Histogram**（批量大小）以及 **Counter**（ID 缺失），并在 `Publisher` 的轮询循环与事件分发处调用对应记录函数，形成 **完整的事件生命周期可观测链路**。

## 🔍 技术洞察

### 架构影响
| 维度 | 影响描述 |
|------|----------|
| **协议层** | 新增 **request‑stream**（单向数据流）与已有 **response‑stream**（双向）并行工作。每个 `StreamType` 对应独立 TCP 连接，保持 Dynamo “分离请求/响应流” 的设计原则。 |
| **注册/发现** | `associate_instance` 现在接受 **recv_subject**（response）和可选 **send_subject**（request），并在 `instance_subjects` 中记录 `(StreamType, subject)`，实现 **双向** 资源回收，防止因只注销 response‑stream 导致 request‑stream 的 oneshot 泄漏。 |
| **资源回收** | `cancel_instance_streams`、`cancel_send_stream`、`cancel_recv_stream` 统一在 `instance_subjects` 中删掉对应 `(StreamType, subject)`，避免残留的 pending entry 引起内存泄漏或后续误关联。 |
| **错误/取消语义** | 请求流在客户端只读（`StreamReceiver`），服务器只写（`StreamSender`），并在 `request_stream_send_handler` 中统一把 `context.kill/stop` 映射为 `ControlMessage::Kill/Stop`，保证上游对下游的取消意图能够跨 TCP 正确传递。 |
| **模块划分** | 客户端 `create_request_stream` 与 `create_response_stream` 对称，实现上层对 API 的统一调用体验；服务器端 `process_request_stream` 与 `process_response_stream` 对称，代码结构更易维护。 |
| **测试覆盖** | 新增的集成/单元测试覆盖 handshake、控制帧、异常路径、实例 tombstone、双向取消等场景，大幅提升代码可靠性。 |

### 性能影响
| 维度 | 正面 | 负面 | 风险 |
|------|------|------|------|
| **CPU** | 只在握手后转发 `DataOnly` 帧，和旧的 response‑stream 解码路径相同，基本无额外开销。 | 为 request‑stream 新增 `handle_request_reader` 循环，额外的 `select!` 用于捕获 `context.kill/stop`，在极端高并发下可能略微增加调度开销（但仍在毫秒级）。 | 低 |
| **内存** | 引入 `bytes_tx` / `bytes_rx` 64‑deep channel，和 response‑stream 相同的 buffer 大小。 | `instance_subjects` 现在持有 `(StreamType, String)`，每条 entry 额外 1 字节标记（实际占用取决于 `Enum` 大小），对大规模实例（数万）影响可忽略。 | 低 |
| **网络** | 仍使用单独的 TCP 连接进行请求流传输，保持原有的无竞争、全双工特性。 | 多建立一次 TCP 连接（请求流额外），在极端短链路/高延迟环境会导致额外的握手 RTT（≈ 1 ms），可在 `StreamOptions` 中通过 `enable_request_stream` 控制。 | 中等（仅在开启 request‑stream 时） |

### 安全考虑
- **协议完整性**：在 `handle_request_reader` 中对 `ControlMessage` 进行 `serde_json::from_slice` 解析，若解析失败会立即 `context.kill()` 并记录取消，防止恶意或损坏的控制帧导致未定义行为。  
- **资源泄露**：通过 `instance_subjects` 的双向标签以及 `cancel_send_stream`/`cancel_recv_stream`，避免了因实例 tombstone 只清除 response‑side 导致 request‑stream pending entry 残留，降低了潜在的 DoS（无限等待）。  
- **取消传播**：`ControlMessage::Kill` / `Stop` 只能从上游（前端）流向下游（Worker），保持了 Dynamo 设计的“只上游可取消”原则，防止下游恶意关闭上游流。  
- **身份验证**：仍然依赖 `ConnectionInfo.context` 与 `subject` 进行匹配，`create_request_stream` 中已检查 `info.context == context.id()`，防止跨上下文的流冒用。  
- **TLS/加密**：此 PR 未更改传输层加密策略，仍然依赖外部配置（如 `TcpStreamServer::ServerOptions`），不引入新漏洞。

## ⚠️ 潜在风险
1. **并发竞态**：`associate_instance` 在检查 tombstone 与插入 `instance_subjects` 之间存在短暂窗口，若两条并发的 `associate_instance` 同时尝试关联同一实例且其中一路已 tombstoned，可能出现 “先插入后被 tombstone” 的情况。当前通过 `prune_tombstones` 及 `contains_key` 检查减轻，但仍建议在极端并发下添加原子 CAS（如 `RwLock` + 双检查）或在 `associate_instance` 完整加锁。  
2. **遗漏 `cancel_send_stream`**：在 `process_request_stream` 中手动调用 `state.tx_subjects.remove(&subject)` 已在 `cancel_instance_streams` 里做清理，但如果调用方忘记在异常路径手动 `cancel_send_stream`，仍可能留下 pending entry。建议在 `process_request_stream` 中出现错误时确保调用 `cancel_send_stream`（已在代码中实现，但需保持同步）。  
3. **测试依赖时间**：部分测试使用 `tokio::time::timeout(1s)`，在 CI 负载高时可能出现偶发超时，导致误报。可将超时时间适当放宽或使用 `tokio::test(flavor = "multi_thread")]` 控制并发。  
4. **升级兼容性**：`StreamOptions::enable_request_stream` 以前是未实现的占位选项，现在正式实现。下游（使用该库的上层服务）若在旧版 Dynamo 仍使用 `enable_request_stream(true)` 会触发 panic（旧实现），但在升级到本版本后无需改动。需在发布说明中强调 **“从 0.12.x 升级到本版本后，请确认所有已编译二进制已重新链接”**。  
5. **错误控制帧的容错**：如果上游在请求流中发送了非法控制帧（如 `Sentinel` 在数据中途），`handle_request_reader` 会 `kill` 并计数一次，这是预期行为；但下游可能无法区分 “非法帧” 与 “正常 kill”。目前日志已记录 `invalid control message`，建议在监控端对异常比例进行报警。

  - 仅在 `encoding_messages` 循环中对 `content` 做一次列表遍历并拼接字符串，时间复杂度 O(n)（n 为文本块数），在常规对话长度（≤ 10‑20 条）下几乎可忽略。
