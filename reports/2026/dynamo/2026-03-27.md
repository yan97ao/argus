#### 🔴 高重要度变更 (1)

### feat: add Intel XPU deployment templates with Kubernetes DRA support (#7464)
**SHA**: `310f8ca` | 🔗 [查看提交](https://github.com/ai-dynamo/dynamo/commit/310f8ca91670f23ecea28243f57466b75aa28856)

**🎯 变更类型**：功能增强  

**⚡ 重要程度**：🔴高  

**📋 变更摘要**  
1. 在 `examples/backends/vllm/deploy` 中新增两套针对 Intel XPU 的部署模板（聚合式 `agg_xpu_dra.yaml`、解聚式 `disagg_xpu_dra.yaml`），并通过 Kubernetes Dynamic Resource Allocation (DRA) 动态申请 `gpu.intel.com` 设备。  
2. README 文档同步更新，说明使用前置条件、部署步骤以及如何验证资源 Claim。  
3. 模板中显式设置 `VLLM_TARGET_DEVICE=xpu`，并在解聚式模板的 Prefill Worker 中配置 XPU‑buffer 的 Nixl KV 传输。  

**🎯 影响范围**  
- `examples/backends/vllm/deploy/`（新增/修改的 README 与两份 YAML）  
- `DynamoGraphDeployment` CRD 的使用方式（加入 `resourceClaims`）  
- 依赖 Kubernetes 1.34+ 与 Intel XPU Resource Driver 的集群环境  

**🔍 技术洞察**  

- **架构影响**  
  - 引入 **可选的硬件专属路径**：在原有 GPU‑only（CUDA）部署基础上，提供 XPU‑专用的聚合与解聚模式。  
  - 通过 **ResourceClaimTemplate** + pod‑level `resourceClaims` 实现 DRA 动态设备分配，避免手动设置 `nodeSelector`/`devicePlugin`。  
  - 对核心源码无侵入，仅在示例层面扩展，保持向后兼容。  

- **性能影响**  
  - 使用 Intel XPU 可在特定工作负载上获得与 NVIDIA GPU 相近或更好的吞吐／延迟，取决于底层硬件驱动与模型规格。  
  - DRA 的动态调度可能在高并发场景下提升资源利用率，但会在调度阶段增加少量延迟（调度器需要完成 Claim → Allocation 流程）。  
  - KV 传输采用 `NixlConnector` + XPU 缓冲区，理论上降低跨设备拷贝成本，提升解码/预填阶段的带宽利用。  

- **安全考虑**  
  - 依赖 **自定义容器镜像**（需自行构建或在 NGC 之外的镜像），因此必须确保镜像来源可信、镜像内部不包含未授权的二进制。  
  - 新增的 **ResourceClaim**、**ResourceClaimTemplate** 对 RBAC 权限提出要求；若权限过宽可能被恶意 pod 申请 XPU 资源。  
  - 与原有 GPU 资源分离，防止误将 XPU 资源误用于仅支持 CUDA 的容器，降低资源泄露风险。  

**⚠️ 潜在风险**  

1. **集群版本 / 驱动不匹配**：DRA API 仅在 Kubernetes 1.34 及以上可用；未安装 Intel XPU Resource Driver 的集群会导致 `ResourceClaim` 创建失败。  
2. **镜像兼容性**：模板默认使用 `nvcr.io/nvidia/ai-dynamo/vllm-runtime:my-tag`，若未在镜像中编译 XPU 支持（`--device xpu`），容器启动会报错。  
3. **资源 Claim 命名冲突**：多个部署共用同名 `gpu-template`，在同一命名空间内可能产生冲突或残留 Claim，导致后续部署资源不足。  
4. **文档与默认路径歧义**：新增模板标记为 *Optional*，但 README 与示例文件混杂，初学者可能误以为是唯一入口。  
5. **调度延迟 / 资源碎片**：DRA 在大规模多租户环境下，若 XPU 设备稀缺，调度器可能因 Claim 未满足而导致 pod 长时间 Pending。  

**💡 关注建议**  

- **CI / 测试**：在项目 CI 中加入对 `resource.k8s.io/v1` API 的兼容性检测，以及对 `gpu.intel.com` DeviceClass 的可达性验证。  
- **镜像说明**：提供一个最小化的 Dockerfile 示例，展示如何在 `vllm-runtime` 中启用 `--device xpu` 编译选项，并在 README 明确标注。  
- **RBAC 与清理**：建议在部署文档中加入对应的 Role/ClusterRole 示例，并说明在删除 Deployment 后需手动清理残留的 `ResourceClaim`（`kubectl delete resourceclaim -n <ns> --all`）。  
- **命名约束**：为防止 Claim 冲突，可在模板中使用 `metadata.name: "{{ .Release.Name }}-gpu-claim"`（使用 Helm 或 Kustomize），或在文档中建议用户自行修改。  
- **回退路径**：保留原有 `agg.yaml` / `disagg.yaml` 作为默认入口，明确在 README 中使用 “如果集群不满足 DRA 条件，请使用传统模板”。  
- **监控与调度**：推荐在集群监控平台（Prometheus + kube‑state‑metrics）中关注 `resourceclaim`、`dynamographdeployment` 的 Pending 状态，以便快速定位调度瓶颈。  

通过上述措施，可最大化该功能对 Intel XPU 用户的价值，同时将潜在的兼容性、权限和调度风险降至可接受范围。

#### 🟡 中重要度变更 (1)

### chore(mocker): batch live output signal sends (#7647)
**SHA**: `a2154ba` | 🔗 [查看提交](https://github.com/ai-dynamo/dynamo/commit/a2154ba5606ebda6b7f6d981c365b151f7050665)

**核心变更**  
1. **启动门统一**：将原 `KvRouterConfig.min_initial_workers` 参数抽离，改为全局环境变量 `DYN_ROUTER_MIN_INITIAL_WORKERS`。新增 `min_initial_workers_from_env()`，在 frontend、standalone‑indexer、LLM entrypoint、kv‑router、offline‑agg/disagg、online‑runtime、MockEngine 等入口处读取并在启动前阻塞等待足够的 worker。文档、Python 绑定与示例均同步更新。  
2. **OutputSignal 批处理**：原先 `mpsc::UnboundedSender<OutputSignal>` 改为 `…<Vec<OutputSignal>>`，所有发送方（MockEngine、Scheduler、KvRouter、offline/online demux、tests）均改为一次性发送向量；相应的 `flush_output_signals`、`process_output_signal`、`run_demux` 等函数改为遍历批量处理。这样可显著降低跨线程消息发送次数，提升高并发 replay 与真实运行时的吞吐。  
3. **API 简化**：kv‑router 中 `ActiveSequencesMultiWorker` 的添加/释放接口由 async → sync（已去掉 `await`），并在内部通过 `spawn_publish_event` 异步发布 KV 事件；同步包装后保持向后兼容。  
4. **配置清理**：删除 `KvRouterConfig.min_initial_workers` 及相关校验，`RouterConfig` 默认不再强制等待；相关 `serde`、`validator`、Python stub、测试数据均同步移除。  
5. **文档/示例**：更新 router‑examples、standalone‑indexer、router‑common 文档，加入环境变量使用说明。  
6. **测试与 bench 调整**：所有离线/在线 replay、bench、mock‑engine、scheduler、kv‑router 的单元/集成测试改为批量信号、无等待 worker 起始门的情况，并相应更新快照结构字段名称。

**影响范围**  
- **运行时**：frontend、standalone‑indexer、LLM router、KvRouter、offline agg、offline disagg、online runtime、MockEngine、Scheduler（VLLM、Sglang）全部使用新 env 变量和批量信号。  
- **绑定层**：Python `KvRouterConfig` 构造函数去掉 `min_initial_workers` 参数；测试工具 `min_initial_workers_env` 新增。  
- **配置文件**：原 `router-min-initial-workers` CLI/Env 已失效，转为 `DYN_ROUTER_MIN_INITIAL_WORKERS`。  
- **API**：`ActiveSequencesMultiWorker.{add_request, free, mark_prefill_completed}` 从 async → sync，调用方已更新。  

**关注建议**  
- **兼容性**：对外发布前保持旧 `--router-min-initial-workers` 参数的兼容层（可在 `add_arguments` 中保留，仅做警告），防止已有部署瞬间失效。  
- **默认行为**：当前默认为 0（不阻塞），应在发布说明明确该变更可能导致 router 在 worker 未就绪时提前接受请求，业务方需自行设定 env 变量。  
- **错误处理**：`min_initial_workers_from_env()` 目前在 env 解析错误时返回 `anyhow`，确保所有调用处均捕获并在启动日志中给出明确提示。  
- **性能基准**：建议在 CI 中加入批量信号 vs 单条信号的对比基准，验证在高并发 replay 场景下的提升幅度。  
- **文档同步**：所有 `README`、`router‑examples.md`、Python SDK 文档需同步注明新 env 变量及其作用，防止用户使用旧 CLI 选项导致配置失效。  

整体来看，此次改动统一了 router 启动门逻辑、显著降低信号传递开销并简化了部分 API，属于中等重要度的功能增强。只要注意向后兼容和文档同步，即可顺利上线。
