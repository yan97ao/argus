### feat(router): propagate session identity through scheduling (#11089)
**SHA**: `b6d3d62` | 🔗 [查看提交](https://github.com/ai-dynamo/dynamo/commit/b6d3d622a546cb3b60192a11138c8cff7f8ef8e4)

**🎯 变更类型**：功能增强  
**⚡ 重要程度**：🔴高  
**📋 变更摘要**：在 KV‑Router 调度链路中引入 `session_id` 字段，实现会话标识从请求入口一路向下传递至调度、选路和离线回放等所有相关模块。所有调度请求结构体、选路参数、路由实现以及离线回放状态均新增该可选字段，并在相应路径中进行传递和测试验证。  

**🎯 影响范围**：  
- `lib/kv-router/src/scheduling/*`（本地调度、策略、队列、选择器）  
- `lib/kv-router/src/services/selection/*`（选路核心、类型）  
- `lib/llm/src/kv_router.rs` 与 `push_router` 相关实现  
- Python 绑定层 `kv.rs`（新增参数占位）  
- 离线回放模块 `lib/mocker/src/replay/offline/*`（路由、状态、聚合、分块）  
- 相关单元测试用例  

**🔍 技术洞察**  

- **架构影响**：  
  - **会话链路完整化**：`session_id` 现在在 `ScheduleRequest` → `SchedulingRequest` → `SelectionCore` → `KvPushRouter` → `OfflineReplayRouter` 等路径上统一传递，解决了之前在跨阶段调度中丢失会话信息的问题，提升了系统可观测性与审计能力。  
  - **向后兼容**：字段均定义为 `Option<String>`，默认 `None`，不影响已有调用者，只在显式提供时生效。  
  - **序列化/反序列化**：在 `selection/types.rs` 中使用 `#[serde(default)]`，确保 JSON/YAML 等外部配置在缺失该字段时仍能正确解析。  

- **性能影响**：  
  - 仅增加一次 `String` 的拷贝/所有权转移（大多数路径中使用 `Option<String>` 的 `clone()`），对热点路径（调度选择）造成的 CPU 与内存开销可忽略不计。  
  - 引入的结构体字段改动未改变任何算法复杂度或调度逻辑，整体延迟基本保持不变。  

- **安全考虑**：  
  - `session_id` 现在可能来源于外部（如 `agent_context.session_id`），若被用于鉴权或日志记录，需要确保：  
    1. 不将未清洗的会话标识直接写入日志或监控系统，防止信息泄露。  
    2. 若后续引入基于 `session_id` 的访问控制，必须在调度层前进行校验，防止伪造。  
  - 当前实现仅作为标记传递，未在安全关键路径使用，风险极低。  

**⚠️ 潜在风险**：  
1. **序列化兼容性**：外部客户端（如 Python、JavaScript）在调用 KV‑Router API 时若使用老版协议，新增字段不会被识别，虽然默认安全，但需要确保绑定层（`kv.rs`）的函数签名已同步更新，防止参数位置错位。  
2. **日志/监控膨胀**：若在大量请求中开启 `session_id` 并将其写入统一日志，可能导致日志体积激增。建议在生产环境通过采样或统一脱敏后记录。  
3. **测试覆盖不足**：虽然新增了单元测试验证 `session_id` 能到达调度请求，但没有覆盖跨进程、网络序列化的路径，建议在集成测试或 CI 中加入端到端的会话传递验证。  

**💡 关注建议**：  
- **文档同步**：更新公共 API 文档（OpenAPI、Python SDK）说明 `session_id` 为可选字段及其语义（会话追踪），并在示例中演示如何使用。  
- **日志脱敏**：在 `tracing`、`dynamo_nvtx_range!` 等日志点加入条件编译或配置开关，仅在调试模式下打印 `session_id`，生产环境可屏蔽。  
- **兼容性检查**：在 CI 中加入旧版客户端的兼容性测试，确保不提供 `session_id` 时仍能完整工作。  
- **后续扩展**：若计划基于 `session_id` 实现跨请求缓存或资源隔离，请在调度层加入验证钩子，确保会话标识的唯一性和可信度。

### fix(ci): OPS-7471 make baked /workspace dirs writable in runtime test images (#11198)
**SHA**: `aeda23b` | 🔗 [查看提交](https://github.com/ai-dynamo/dynamo/commit/aeda23b483831df2f75b697b8ccf65f4ffd9f3d6)

**🎯 变更类型**：Bug 修复  
**⚡ 重要程度**：🟡 中  

**📋 变更摘要**  
在 CI 用的测试镜像中，`/workspace` 目录原本只对 `dynamo` 用户可写，导致 ARC runner 以 uid 1001（非 `dynamo`）执行时在运行时无法创建或修改文件。此提交在 Dockerfile 中加入两段 `chmod o+w` 对所有子目录授予 “others” 写权限，并在后续的 `github_runner_tools` 阶段复制 `find`、`chmod` 等工具，以确保即使在 distroless 基础镜像缺失 `find` 时也能完成权限修正。

**🎯 影响范围**  
- `container/Dockerfile.test`（构建测试镜像）  
- CI 测试执行环境（涉及所有依赖 `/workspace` 的单元/集成测试）  

**💡 关注建议**  
1. **权限安全审查**：仅对目录加 `o+w`，若未来加入敏感文件（如凭证、密钥）需重新评估是否仍适用。  
2. **镜像层大小**：`find … -exec chmod` 会遍历整个 `/workspace`，在大体积工作区可能增加构建时间；可考虑在 Dockerfile 中使用 `--chmod=777` 直接设置目录权限，避免额外层。  
3. **兼容性验证**：在本地及其他 CI 平台（GitHub Actions、GitLab）跑一遍完整测试，确保 `find` 已经被成功复制，否则仍会出现 “skipped (distroless planner)” 的日志。  
4. **文档更新**：在 CI 文档或镜像说明中标明此权限变更，方便后续维护者了解为何 `dynamo` 用户不再是唯一可写用户。  

整体来看，此修复解决了运行时写权限导致的测试失败，影响范围局限于 CI 镜像构建，风险可控。后续可关注镜像体积与安全策略的平衡。

**📋 变更摘要**

**🎯 影响范围**  
- `deploy/helm/charts/platform/components/operator/templates/webhook-configuration.yaml`（Helm 部署）  
- `deploy/operator/internal/webhook/defaulting/dynamographdeployment_handler.go`（默认化 webhook）  
- `deploy/operator/internal/webhook/validation/dynamographdeployment_handler.go`（校验 webhook）  

**💡 关注建议**  

1. **CRD 兼容性**：确认集群已安装 `nvidia.com/v1beta1` 版本的 `DynamoGraphDeployment` CRD，否则 webhook 将因找不到对应的 API 版本而返回 404。  
2. **升级路径测试**：在实际升级（1.2 → 1.3）或回滚（1.3 → 1.2）场景下，验证 operator 能同时监听 `v1alpha1` 与 `v1beta1`，确保所有运行实例在切换 Helm 注册前已提供双端点。

4. **后续清理**：在 1.5 发行前移除 `v1alpha1` 相关代码与二进制注册，防止残留的旧入口导致安全或性能隐患。  
5. **CI 检查**：加入针对 `v1beta1` webhook 的单元/集成测试，确保 `RegisterWithManager` 正确注册新路径，并验证请求成功通过默认化与校验逻辑。  

通过上述检查可确保本次回退不会在生产环境引入不可预期的 webhook 注册错误，并为后续彻底去除 `v1alpha1` 做好准备。
