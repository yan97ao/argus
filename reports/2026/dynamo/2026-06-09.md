### feat(container): add SGLang XPU Docker support (#7990)
**SHA**: `9b978b3` | 🔗 [查看提交](https://github.com/ai-dynamo/dynamo/commit/9b978b311233a867f1309aa18a3dbed45f20c94e)

**🎯 变更类型**：功能增强、架构变更、安全修复  
**⚡ 重要程度**：🔴高  
**📋 变更摘要**：本次提交为 `ai-dynamo/dynamo` 添加 Intel XPU（GPU）对 SGLang 的 Docker 支持。通过在 `container/render.py`、`run.sh` 以及多个 Dockerfile 模板中引入 `--device=xpu` 参数，实现了基于 Intel deep‑learning‑essentials 镜像自研 SGLang、NIXL、UCX 等组件的构建与运行，并在文档中提供完整的使用示例。  

**🎯 影响范围**：  
- `container/` 目录下的所有 Dockerfile、脚本与模板（runtime、dev、local‑dev、wheel_builder 等）  
- `container/context.yaml` 中新增的 `sglang.xpu` 配置块  
- `container/render.py` 参数校验逻辑  
- `container/run.sh` 的设备选择与权限挂载逻辑  
- 主要针对 **SGLang** 框架，其他框架保持原有 CUDA 流程不变  

**🔍 技术洞察**  

- **架构影响**  
  - 引入了 *device* 维度（`cuda` / `xpu`），使 Docker 构建流程从单一 CUDA 路径分支为两条：  
    1. 维持原有 `sglang` CUDA 运行时（基于 upstream `lmsysorg/sglang` 镜像）。  
    2. 新增 XPU 路径：从 `intel/deep-learning-essentials` 基础镜像构建 SGLang 源码，集成 Conda 环境、Intel NIXL、UCX、Level‑Zero 等堆栈。  
  - 新增 `sglang_xpu_framework.Dockerfile`，作为 XPU **framework** 阶段的入口，显式分离了 **framework** 与 **runtime**，保持运行时镜像的最小化。  
  - `wheel_builder` 阶段对 XPU 也强制走自建 NIXL 流程，保证运行时拥有对应的本地库，避免依赖上游未提供的 XPU 包。  

- **性能影响**  
  - **构建时间**：XPU 路径包含源码克隆、Conda 环境创建、UCX/NIXL 编译等步骤，预计比 CUDA 直接使用预编译镜像慢 2‑3 倍。  
  - **运行时体积**：额外引入 Intel oneAPI、Conda、NIXL、UCX 等二进制，镜像大小约从 4 GB 增至 6‑7 GB。  
  - **推理性能**：在支持 Intel Xe GPU 的平台上，可利用 Level‑Zero 与 NIXL 的 P2P/GPUDirect Storage，理论上对大模型推理的内存带宽与延迟有显著提升。实际收益取决于硬件驱动和模型实现的 XPU 优化程度。  

- **安全考虑**  
  - `run.sh` 为 XPU 添加了 `--device /dev/dri` 与 `--group-add render`，将宿主机的 **render** 组加入容器，允许容器直接访问 GPU 设备。若容器被攻破，攻击者可能直接利用 GPU 进行计算或信息泄露，需要在生产环境通过 **least‑privilege**（最小化组权限）以及容器安全基线（e.g., SELinux/AppArmor）进行约束。  
  - 新增的 `intel/deep-learning-essentials` 基础镜像包含大量系统工具与驱动，扩大了供应链攻击面。建议在 CI 中加入镜像签名校验及漏洞扫描。  

**⚠️ 潜在风险**  

1. **平台限制**：XPU 仅支持 `--platform linux/amd64`，在 arm64 环境会直接报错，易导致 CI 脚本误用。  
2. **依赖同步**：SGLang、sgl‑kernel、NIXL、torch‑xpu 等仓库采用不同的 tag/commit，若 upstream 更新未同步，可能导致编译失败或运行时异常。  
3. **环境冲突**：在同一宿主机上混用 CUDA 与 XPU 容器，`run.sh` 默认仍会传递 `--gpus` 参数，若未显式清除可能导致 Docker 启动失败。  
4. **镜像体积与构建资源**：XPU 构建需要大量磁盘（>20 GB）和内存（>8 GB），在资源受限的 CI 环境可能导致 OOM 或磁盘耗尽。  
5. **权限提升**：容器内部的 `sudo chown`、`NOPASSWD` 设置在 `local‑dev` 阶段被保留，若未限制用户可访问的路径，可能被用于横向移动。  

**💡 关注建议**  

- **CI/测试**：  
  - 为 XPU 构建新增专属 CI 作业，使用 `--platform linux/amd64` 并配合硬件加速器（或 GPU 虚拟化）进行 smoke‑test。  
  - 将 `context.yaml` 中的 XPU 相关字段加入 schema 检查，防止手动编辑出错。  
- **文档与 UX**：  
  - 在 README 中明确 `--device=xpu` 需要 Intel GPU、对应驱动、以及 `docker run` 必须加 `--privileged`（或 `--device /dev/dri` + `--group-add render`）的说明。  
  - 为常用的 “local‑dev” 场景提供一键脚本，自动执行 `chown`、`uv pip install` 等步骤，降低新手使用门槛。  
- **安全加固**：  
  - 在 `run.sh` 中对 `--device` 参数进行白名单校验，禁止传入其它路径。  
  - 推荐在生产部署时使用 AppArmor/SELinux profile 限制对 `/dev/dri` 的访问范围，仅保留 `render` GID。  
  - 对 `intel/deep-learning-essentials` 镜像执行定期 CVE 扫描（如 Trivy），并在发现高危漏洞时快速发布补丁。  
- **性能监控**：  
  - 在 XPU 镜像中预装 `nvidia‑tools` 对应的 **Intel** 监控工具（e.g., `intel‑gpu‑tools`），便于用户对推理延迟、带宽进行基准评估。  
  - 在运行时入口 (`sglang_runtime.Dockerfile`) 添加 `LD_LIBRARY_PATH` 与 `NIXL_*` 环境变量的显式说明，避免因路径冲突导致的库加载失败。  

通过上述措施，可在保证新功能可用性的同时，降低因平台、依赖与安全引入的风险，帮助社区更快、稳健地在 Intel XPU 环境上使用 Dynamo 与 SGLang。

### feat(RL): add RL worker admin routes (#9680)
**SHA**: `81f7f89` | 🔗 [查看提交](https://github.com/ai-dynamo/dynamo/commit/81f7f89c9f9f7fbc2ee3ec627019ec90b04ec52e)

**🎯 变更类型**：功能增强（RL Worker Admin Routes）  
**⚡ 重要程度**：🔴高  

**📋 变更摘要**  
1. 在 `components/src/dynamo/common/rl` 中新增 `admin.py` 与 `__init__.py`，实现 RL（Reinforcement‑Learning）管理侧的通用工具：路由注册、请求验证、首个生成块返回、环境变量布尔解析等。  
2. 为 VLLM worker（`vllm/handlers.py`）注入 RL 管理路由，包括 liveness probe、pause/resume、cache flush、abort request、权重更新、LoRA load/unload 等，并完善 **deferred abort** 机制以避免在 “首 token 前” 调用 engine abort 的 unsafe 场景。  
3. `worker_factory.py` 通过 `register_rl_routes` 把上述 RL 入口统一注册到 Runtime，支持可选开启 `--enable-rl`（`DYN_ENABLE_RL`）。  
4. 完善单元测试，覆盖请求体验证、错误传播、deferred abort 行为等。

---

**🎯 影响范围**  
- `components/src/dynamo/common/rl`（全新模块）  
- `components/src/dynamo/vllm/handlers.py`（核心请求处理、生成路径）  
- `components/src/dynamo/vllm/worker_factory.py`（Worker 启动与路由注册）  
- 相关测试文件 `components/src/dynamo/vllm/tests/*`、`components/src/dynamo/common/tests/test_rl_admin.py`  
- 运行时配置：`config.enable_rl`、环境变量 `DYN_LORA_HOTSWAP_ENABLED`、`DYN_ENABLE_RL` 等  

---

**🔍 技术洞察**  

| 维度 | 影响描述 |
|------|----------|
| **架构影响** | • 引入 **RLRouteRegistry**，提供统一的请求‑平面路由描述与分发（`rl_dispatch`）。<br>• `BaseWorkerHandler` 现在持有 `rl_route_registry` 与 `_deferred_aborts`（请求ID → DeferredAbort 实例），实现 **out‑of‑band admin** 能力。<br>• `worker_factory` 为每个 worker 动态创建 `.rl` endpoint，仅在 `enable_rl` 为真时暴露。<br>• LoRA‑load/unload 通过 `first_endpoint_response` 包装为生成式 endpoint，保证资源及时释放。 |
| **性能影响** | • RL 管理接口均为 **异步、单次请求**，除 `rl_dispatch`（返回路由列表）外，其他路由仅在管理员调用时触发，业务路径的额外开销可忽略。<br>• 通过 `asyncio.shield` 与 **deferred abort guard**，防止 admin abort 在首 token 前阻塞请求线程，提升高并发下的响应及时性。<br>• 新增的 `_deferred_aborts` 字典在极端并发情况下可能稍增内存占用，但在 `close()` 时会及时清除。 |
| **安全考虑** | • 所有 RL admin 入口 **未内置鉴权**，依赖部署层（Istio/Envoy、K8s 网络策略）进行访问控制。若直接暴露，会产生未授权的 **pause/resume/weight‑update** 风险。<br>• `env_bool` 与 `RLAdminValidationError` 对请求体进行严格校验，防止 **JSON 注入** 或 **类型混淆**（只接受 `dict`，字段必须为 `str`）。<br>• 权重更新路径捕获 `EngineDeadError` 并触发进程 **强制退出** (`os._exit(1)`)，保证在底层引擎失效时不出现不一致状态。 |
| **可维护性** | • RL 相关代码抽离到独立模块，职责清晰，后续可以单独演进。<br>• `register_rl_routes` 通过 `enable_dispatch` 控制是否向 request‑plane公开路由描述，兼容老版客户端。<br>• 单元测试覆盖率提升，特别是异常路径（非 dict body、deferred abort 失败、hot‑swap 回滚）。 |

---

**⚠️ 潜在风险**  

1. **未授权访问**：如果运行时未通过外部网关限制 `.rl` endpoint，攻击者可直接调用 `pause_generation`、`update_weights_*`、`load_lora` 等导致服务不可用或模型状态不一致。  
2. **注册泄漏**：`_deferred_aborts` 仅在 `close()` 时删除；若异常退出未调用 `close()`（如进程被 OOM 杀死），字典可能暂时膨胀，需监控内存占用。  
3. **并发 LoRA hot‑swap**：在 `DYN_LORA_HOTSWAP_ENABLED` 打开时，多请求同时尝试 hot‑swap 同一个 LoRA，虽然加锁序列化，但回滚逻辑仍可能留下 **engine‑side** 的半残留适配器，需要确认 `engine_client.remove_lora` 与 `add_lora` 在错误情形下的幂等性。  
4. **EngineDeadError 处理**：`_shutdown_on_engine_dead` 直接 `os._exit(1)`，会导致 **core dump** 与容器重启。如果业务需要快速恢复，可能需要更温和的重启策略。  
5. **API 兼容性**：新增的 RL admin endpoint 采用 `dyn://<ns>.<comp>.rl`，旧版工具若误用 `rl_dispatch` 的 `method` 字段可能收到 “only supports method='routes'” 错误；需要在客户端文档中明确说明。

**💡 关注建议**  

| 推荐对象 | 建议 |
|----------|------|
| **运维 / 平台** | • 在部署时强制在 Service Mesh、Ingress 或防火墙层对 `*.rl` 端点进行 **网络隔离**（仅内部可达）。<br>• 通过 `DYN_ENABLE_RL` 环境变量在生产环境关闭 RL 管理功能，若不需要热更新或 LoRA 管理。 |
| **后端开发** | • 为 RL admin 路由实现 **审计日志**（记录调用者身份、时间、操作类型），便于事后追踪。<br>• 考虑在 `first_endpoint_response` 捕获 `async_generator` 的异常并统一返回 `status: error`，避免隐藏异常。 |
| **代码维护** | • 为 `RLRouteRegistry.routes` 添加 **弱引用**（`WeakValueDictionary`）或周期性清理，以防极端异常导致残留。<br>• 在 `load_lora`/`unload_lora` 的 hot‑swap 分支加入 **幂等检查**（如 `engine_client.has_lora(id)`），降低回滚失败概率。 |
| **测试** | • 增加 **权限/身份验证** 测试模拟未授权请求，确保不在业务环境中意外开放。<br>• 在高并发场景下（数千并发 `load_lora`）进行压力测试，验证锁竞争与 `register_rl_routes` 的性能表现。 |
| **监控** | • 暴露 `rl_route_registry` 的路由列表指标（如 `dynamo_rl_routes_total`），监控是否出现意外路由泄露。<br>• 对 `abort_request`、`pause_generation`、`resume_generation` 等关键操作增加 **Prometheus** 计数器，监测异常频率。 |

### fix(bindings): propagate Python exception HTTP status through the boundary (#10364)
**SHA**: `7be38df` | 🔗 [查看提交](https://github.com/ai-dynamo/dynamo/commit/7be38df572fb7d312ca2d68ee041d81d8ce0b315)

**🎯 变更类型**：Bug修复  
**⚡ 重要程度**：🔴高  
**📋 变更摘要**：在 Python 绑定层引入 `extract_http_like_error`，使得 Python 异常中携带的 HTTP 状态码 (如 `.code`、`.status`) 能够以 JSON 结构 `{message, code}` 透传到 Dynamo 后端错误对象，进而在 OpenAI 前端返回对应的 HTTP 响应码，而不再统一映射为 500。新增单元与端到端集成测试，验证 4xx 错误在整个请求链路（Python engine → TCP/etcd → 前端）中的正确传播。

  - 新增的 `ToolCallDispatchPayload`、`ReasoningDispatchPayload` 结构体提升了序列化代码的可维护性，避免在各处硬编码 JSON 字段。

  - **降低 CPU 开销**：一次性 `serde_json::to_string` + `Event::data` 替代 `Event::json_data`，避免 Axum 对每个字符的逐字过滤（尤其是换行、回车），在高吞吐的聊天流场景下可显著提升 QPS。

- **性能影响**  
  - 额外的字典拷贝与 `json.dumps`（对 metrics 的序列化）在每个流式块都会产生微小的 CPU 与内存开销；在高吞吐、短块（每 token）场景下可能累计到 **≈5‑10%** 的额外延迟。

  - 注解内容仅为数字计数，未引入可执行代码或外部输入的直接渲染，安全风险低。

- **正向**：显式关闭程序后，调度器不再对已完成的程序计数，能够更快降低 KV 使用率，提升 **内存回收效率** 与 **吞吐提升**（原实验报告显示 8‑16%）。在高并发 Agent 场景下可避免因历史程序残留导致的 “缓存占用 blow‑up”。

#### 安全考虑
- **输入验证**：`_is_trajectory_final` 仅检查 `agent_context` 为 dict 并读取布尔字段，未对字段类型进行严格校验。若恶意客户端发送非布尔或非 dict 结构，仍会被安全地解析为 `False`，不触发异常。

- `lib/llm/src/protocols/common.rs`、`openai.rs`（协议层结构）  
- `lib/parsers/src/tool_calling/harmony`（工具调用解析器）  
- `lib/renderer/src/template/formatters.rs`（模板渲染）  

**🔍 技术洞察**  
- **架构影响**：  
  - 在请求层统一抽象了可见/隐藏停止标记的处理，保持后端（TRT‑LLM/PyTorch）与 Dynamo 之间的职责分离。  
  - 新增的 `VisibleStopTokenDetected` 通过 `StopTrigger` 与已有停止路径兼容，最小化对其它模块的侵入。  
  - `StopConditions` 结构的扩展要求所有调用方（OpenAI 接口、内部 SDK 等）同步更新以避免字段缺失。  

- **性能影响**：  
  - 仅在检测到可见停止标记时会额外进行一次 HashSet 查找 (`visible_stop_ids.contains`)，成本极低（O(1)），对整体吞吐基本无感。  
  - 迁移停止标记到 `visible` 集合后，原先的 “隐藏”过滤步骤被省去，略微减小了 token‑level 的过滤开销。  

- **安全考虑**：  
  - 由于停止标记直接决定生成何时终止，若误将恶意 token 标记为可见停止，可能导致生成在未预期的地方截断或泄露内部状态。当前实现仅在工具调用解析器明确声明的结束 token 中加入，可控风险。  
  - 新增的 `ignore_eos` 强制为 `true`，确保后端不因 EOS 误判提前结束，避免潜在的输出截断攻击。  

**⚠️ 潜在风险**  
1. **向后兼容性**：旧版客户端/插件如果未识别 `stop_token_ids_visible`，可能仍把这些 token 当作隐藏停止，导致工具调用仍被裁剪。  
2. **字段同步错误**：在未更新的自定义 `StopConditions` 实例中 `stop_token_ids_visible` 为 `None` 时，后端仍会把对应 token 当作隐藏停止，行为回退到老版本，可能出现不一致的日志/错误。  
3. **模板渲染改动**：`normalize_dict_method_calls` 过滤器对所有 Jinja2 模板生效，若模板中有意使用 `.items()` 方法名而非字典迭代，可能被错误改写。  
4. **并发路径**：`handler_base` 中对 `visible_stop_token_ids` 的集合操作在异步上下文里仍是局部副本，风险低，但若未来改为共享可变状态需额外同步。  

**💡 关注建议**  
- **升级文档**：在 OpenAI/内部 SDK 文档中明确 `stop_token_ids_visible` 的语义及使用场景，提醒调用方在构造请求时正确填充。  
- **回归测试**：加入对 `visible` 与 `hidden` 停止标记组合的端到端生成测试，覆盖不同模型（gpt‑oss、vLLM、TRT‑LLM）以及多工具调用情形。  
- **监控/告警**：在生成服务中增添 `VisibleStopTokenDetected` 的统计指标，以快速捕获异常高频触发（可能是工具调用配置错误）。  
- **模板审计**：对已有 Jinja2 模板执行一次批量校验，确保 `normalize_dict_method_calls` 不会意外改写业务逻辑；如有冲突，可在模板头部使用 `{% raw %}` 包裹或手动调整。  
- **回滚路径**：保留旧的 `ignore_eos` 逻辑（通过环境变量或特性开关）作为紧急回滚手段，以防新字段在某些边缘案例导致生成异常。

### fix(frontend): merge leading system messages to fix Codex CLI first-turn failure (#10283)
**SHA**: `e0be711` | 🔗 [查看提交](https://github.com/ai-dynamo/dynamo/commit/e0be711f8f2f3048d79e0b171c2dd303bcbaee24)

**⚡ 重要程度**：🔴 高  
**📋 变更摘要**：在 OpenAI 协议的响应转化逻辑中，新增对开头连续多个 System 消息的合并处理。将它们合并为单一 System 消息（以两个换行分隔），解决了 Codex CLI 第一次请求产生的“双 System”导致 Qwen 等模型模板请求被拒的问题。同时补充了对应单元测试，以验证合并行为的正确性。  

**🎯 影响范围**：  
- `lib/llm/src/protocols/openai/responses/mod.rs`（核心协议转换层）  
- 依赖 OpenAI 协议的所有上层业务（Chat、Codex CLI、第三方模型适配层）  

**🔍 技术洞察**  
- **架构影响**：  
  - 只在 **响应 → 请求** 的转换阶段做了局部改动，属于 **业务规则层** 变更，不涉及底层网络、序列化或线程模型。  
  - 引入了对消息序列的前置检查与 `drain`/`insert` 操作，保持原有 `Message` 向量的所有权，符合 Rust 所有权模型，未引入额外抽象或公共 API 改动。  

- **性能影响**：  
  - 合并逻辑仅在检测到 **>1** 条前置 System 消息时执行，最坏情况下遍历一次前导消息并进行一次 `Vec::drain` + `Vec::insert`，时间复杂度 O(k)（k 为前导系统消息数量），对整体请求构造的影响可忽略。  
  - 额外的 `tracing::debug!` 仅在出现非文本系统消息时触发，对 Release 编译几乎无运行时开销。  

- **安全考虑**：  
  - 合并过程仅处理 **文本** 内容；对非文本（如未来可能出现的 `Array`）会被 **丢弃** 并记录调试日志。若后端对系统消息的非文本形式有安全校验需求，可能导致信息丢失或意外行为，需要关注后续协议扩展。  
  - 日志信息不泄露用户敏感数据，仅输出 `Debug` 表示的枚举变体，风险极低。  

**⚠️ 潜在风险**  
1. **信息丢失**：在出现非 `Text` 系统消息时，内容会被直接抛弃，仅留下空字符串，可能导致请求语义不完整。  
2. **兼容性**：某些后端可能依赖多个系统消息（例如分层指令），合并后可能改变其期望的指令顺序或分段。  
3. **边界情况**：如果前导系统消息列表为空或只有一条，当前逻辑仍会触发 `take_while`，但不会执行 `drain/insert`，行为保持不变；但若消息顺序被其他代码提前修改，可能导致 `unreachable!()` panic（在 `match m` 的 `_ => unreachable!()` 分支）。  

**💡 关注建议**  
- **测试覆盖**：在 CI 中加入对非文本系统消息的回归测试，确保日志能够捕获并不会导致功能回退。  
- **文档说明**：在库的协议转换文档或 CHANGELOG 中标注 “前导多个 System 消息会被合并”，提醒使用者在自定义模板时注意该行为。  
- **可配置开关**（可选）：若业务场景确实需要保留多个系统消息，考虑提供一个特性标记或配置参数，以禁用自动合并。  
- **监控与日志**：在生产环境开启 `DEBUG` 级别的 tracing，以便在出现非文本系统消息时快速定位并决定是否需要升级库或后端。

此变更主要解决了 **Codex CLI 首轮请求** 在特定模型（如 Qwen）下因多余系统消息导致的请求失败，风险可控且收益显著，建议尽快在下一次发布中推送。
