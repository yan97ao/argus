### fix: disable Allure reporting to stop gh-pages bloat (#7982)
**SHA**: `a1f230e` | 🔗 [查看提交](https://github.com/ai-dynamo/dynamo/commit/a1f230e996f292f12eff5d1ecdca09dbf3c3aa8a)

**🎯 变更类型**：Bug修复 / 功能调整  
**⚡ 重要程度**：🔴 高  
**📋 变更摘要**：为了解决 `gh-pages` 分支因持续生成 Allure 报告而膨胀至约 1 GB 的问题，作者将所有与 Allure 报告生成、上传以及在 PR 工作流中调用的步骤全部 **禁用**。同时对报告生成脚本做了轻量化的整理与冲突重试逻辑，以免在未来重新开启时出现推送冲突。  

**🎯 影响范围**：  
- `.github/workflows/allure-report.yml`（整个 Allure 报告流水线被强制 `if: false`）  
- `.github/workflows/generate-allure-report.yml`（报告生成、注入 CI 链接、推送至 `gh-pages` 的全部逻辑仍保留，但在调用方已被屏蔽）  
- `.github/workflows/pr.yaml`（PR 合并后统一调用 Allure 报告的步骤同样被禁用）  

---

### 🔍 技术洞察

| 维度 | 影响描述 |
|------|----------|
| **架构影响** | - CI/CD 流程中 **Allure 报告** 仅是可选的可视化层，关闭后不影响核心编译、测试、部署等阶段。<br>- 仍保留生成脚本（`generate-allure-report.yml`），为以后重新启用提供最小侵入路径。<br>- 对 `gh-pages` 分支的写入逻辑做了 **深度 fetch**、冲突重试包装，提升未来并发推送的稳健性。 |
| **性能影响** | - **CI 运行时显著下降**：原本的 Allure 报告收集、注入标签、打包、上传等步骤约耗时 2–5 min，现被直接跳过。<br>- **存储成本降低**：`gh-pages` 分支不再累计数百 MB 的报告文件，仓库体积保持在几百 MB。 |
| **安全考虑** | - 关闭 Allure 报告本身不涉及安全风险。<br>- 仍保留 `git config http.https://github.com/.extraheader "AUTHORIZATION: basic …"` 以便推送，若未来重新启用，需要确保 token 权限最小化（仅 `pages:write`）。 |

---

### ⚠️ 潜在风险

1. **失去可视化回归信息**  
   - 开发者、QA 以及外部合作伙伴将看不到测试结果的 Allure UI，可能导致定位错误或回归检测变慢。  
2. **仪表盘/报告依赖中断**  
   - 项目内部或 CI 报告聚合平台（如统一的 Allure Dashboard）若依赖 `gh-pages` 上的历史数据，可能出现空白或错误。  
3. **误删导致的回滚难度**  
   - 由于 `gh-pages` 已被清理，若需要恢复历史报告，需要从备份或手动重新运行历史工作流。  
4. **潜在的脚本错误被隐藏**  
   - 该提交仅把工作流条件改为 `false`，并未删除脚本本身，若脚本本身存在未捕获的错误，在未来重新开启时仍会触发。

### fix(operator): handle GPU resources in requests, not just limits (#8005)
**SHA**: `99199f4` | 🔗 [查看提交](https://github.com/ai-dynamo/dynamo/commit/99199f4957f6fcec5c8e18b496c43dbcf44559c8)

**🎯 变更类型**：Bug修复 / 功能增强  
**⚡ 重要程度**：🔴高  
**📋 变更摘要**：  
1. 在 `GetResourcesConfig` 中新增对 `Resources.Requests.GPU` 的解析，使得 Operator 在生成 `corev1.ResourceRequirements` 时同时填充 GPU **requests**，而不仅仅是 **limits**。  
2. 为此添加相应的单元测试，覆盖无 GPU 类型、指定自定义 GPU 类型，以及限额与请求同时存在的场景。

**🎯 影响范围**：  
- `deploy/operator/internal/controller_common/resource.go`（核心资源组装逻辑）  
- `deploy/operator/internal/controller_common/resource_test.go`（测试覆盖）  
- 受影响的模块：Operator 调度器/控制器在创建 Pod‑spec 时的资源声明；所有依赖 `GetResourcesConfig` 的自定义资源 (CR) 生成流程。

**🔍 技术洞察**  

| 维度 | 影响分析 |
|------|----------|
| **架构影响** | - 资源组装层新增对 GPU **requests** 的处理，保持 `limits` 与 `requests` 两套资源列表独立。 <br> - 通过 `getGPUResourceName(resources.Requests)` 自动选取自定义 GPU 类型，保持与已有 `limits` 逻辑一致。 <br> - 该改动在 Operator 内部是 **向后兼容** 的：未设置 `Requests` 的 CR 仍然只返回 `limits`。 |
| **性能影响** | - 仅在 CPU‑side解析字符串并构造 `resource.Quantity`，占用微秒级别的额外 CPU 与内存，几乎可以忽略。 <br> - 对 Pod 创建速率或调度延迟无实质影响。 |
| **安全考虑** | - 新增的 `resource.ParseQuantity` 仍会返回错误并被包装后返回，避免将非法 GPU 数量写入 PodSpec。 <br> - 需要注意 **空指针**：当前实现直接访问 `resources.Requests.GPU`，若 `resources.Requests` 为 `nil` 将导致 panic，属于安全/可靠性风险。 |

**⚠️ 潜在风险**  

1. **空指针风险**  
   ```go
   if resources.Requests.GPU != "" { … }
   ```  
   当 `resources.Requests` 为 `nil` 时会触发运行时 panic，影响 Operator 的可用性。  
2. **重复资源名称冲突**  
   - 若 `limits` 与 `requests` 使用相同 GPU 资源名称（默认 `nvidia.com/gpu`），两个 map 条目分别写入 `Limits` 与 `Requests`，不会冲突，但 downstream 代码若假设只有 `Limits` 存在，可能需要适配。  
3. **错误信息传播**  
   - 现有错误包装为 `parse requests gpu quantity: %w`，调用方需要正确处理错误路径，防止因解析失败导致 CR 持续处于 “Pending” 状态。  

**💡 关注建议**  

| 建议对象 | 具体措施 |
|----------|----------|
| **开发者** | - 在 `GetResourcesConfig` 前添加空指针防护：<br>`if resources.Requests != nil && resources.Requests.GPU != "" { … }`。<br>- 为 `nil` 场景补充单元测试，确保函数在 `resources.Requests` 为 `nil` 时仍成功返回仅包含 `limits`（或 `nil`）。 |
| **测试** | - 增加 **边界测试**：`Resources{Limits:…, Requests: nil}`、`Resources{Requests: &ResourceItem{}}` 等，验证函数不 panic。 |
| **文档** | - 更新 Operator CRD 文档，说明 GPU **requests** 现在被支持，并解释默认 GPU 类型与自定义类型的映射规则。 |
| **运维/用户** | - 若已有监控或审计依赖于 `GPU` 只出现在 `limits`，确认这些查询逻辑已兼容 `requests`。<br>- 在升级 Operator 前，检查已有 CR 是否已经设置了 `requests.gpu`（可能是旧的空字符串），避免因解析错误导致升级回滚。 |
| **后续优化** | - 将 GPU 资源解析抽象为通用函数（如 `parseGPU(resources *v1alpha1.ResourceItem, target corev1.ResourceList)`），避免代码重复并统一错误处理。 |

---  
**结论**：此次改动提升了 Operator 对 GPU 资源的表达能力，使得 GPU **请求** 能被显式声明，对调度器的资源预留更为精准。风险主要集中在空指针检查和向后兼容性上，建议在代码中加入防护并补全相应测试，以确保升级过程的平滑与系统的高可用。

### refactor(snapshot): tighten operator protocol boundary (#8018)
**SHA**: `debda33` | 🔗 [查看提交](https://github.com/ai-dynamo/dynamo/commit/debda33277c5a6cfb9e4e828de95273240b6e5d2)

**🎯 变更类型**：重构 / 架构变更  
**⚡ 重要程度**：🔴高  
**📋 变更摘要**：本次提交将快照（snapshot）相关的标签、注解、默认配置以及检查点作业观察逻辑从 Operator 代码库中抽离到统一的 `deploy/snapshot/protocol` 包。所有原先散落在 `internal/consts` 与 `internal/checkpointjob` 的常量与函数统一迁移，并对相应的控制器、测试以及生成的资源做了适配。删除了已废弃的 `checkpointjob` 观察实现与不再使用的测试文件，实现了“协议边界”更清晰的模块划分。

**🎯 影响范围**  
- `deploy/operator/internal/checkpoint/*`（标签/注解使用）  
- `deploy/operator/internal/controller/*`（Checkpoint 控制器、ComponentDeployment 控制器）  
- `deploy/operator/internal/checkpointjob/*`（全部删除）  
- 新增/修改 `deploy/snapshot/protocol`（常量、`GetCheckpointJobName`、`ObserveCheckpointJob`、相关单元测试）  
- 受影响的公共 API：`snapshotprotocol.CheckpointIDLabel`、`snapshotprotocol.CheckpointArtifactVersionAnnotation`、`snapshotprotocol.GetCheckpointJobName`、`snapshotprotocol.ObserveCheckpointJob` 等。

---

### 🔍 技术洞察

| 维度 | 影响分析 |
|------|----------|
| **架构影响** | - **协议统一**：所有与快照/检查点相关的 K8s 标签、注解、默认值现在集中在 `snapshot/protocol` 包，避免了 Operator 与 Snapshot 之间的隐式耦合。<br>- **职责清晰**：Operator 只负责调度与调和，快照协议本身负责定义“协议层面”的常量与业务规则，符合单一职责原则。<br>- **代码体积**：删除了 84 行的 `observation.go` 与 24 行的 `job_test.go`，代码库更简洁，维护成本降低。 |
| **性能影响** | - 迁移仅涉及常量、函数调用路径的变更，运行时没有额外计算或资源开销。<br>- `ObserveCheckpointJob` 仍然在控制器循环中执行，行为保持不变。 |
| **安全考虑** | - `DefaultSeccompLocalhostProfile` 常量从 `internal/consts` 移到协议层，语义未变，仍然使用本地 seccomp profile 来阻断 `io_uring` 系统调用，安全姿态保持。<br>- 通过统一的协议常量，减少因手动写错标签/注解导致的安全策略失效风险。 |
| **可维护性** | - **统一入口**：任何对快照标签/注解的修改只需在 `snapshot/protocol` 中完成，避免在多个目录中同步更改。<br>- **测试覆盖**：新增 `GetCheckpointJobName` 与 `ObserveCheckpointJob` 的单元测试，保障未来改动的回归安全。 |
| **兼容性** | - 删除了 `internal/consts` 中的快照相关常量（`KubeLabelCheckpointID`、`KubeAnnotationCheckpointArtifactVersion` 等）以及 `checkpointjob.DesiredCheckpointJobName`。外部项目若直接引用这些已删除的符号将编译失败。<br>- 对内部代码已全部迁移，但若有未在本次提交中捕获的引用（如其他子模块或第三方插件），会出现构建错误。建议保留 **向后兼容的别名**（如在 `consts` 中保留已废弃的常量并标记 `// Deprecated`），或在发布说明中明确破坏性变更。 |

**⚠️ 潜在风险**

1. **编译破坏** – 任何仍然使用旧常量（`consts.KubeLabelCheckpointID`、`consts.KubeAnnotationCheckpointArtifactVersion`）或旧函数（`checkpointjob.DesiredCheckpointJobName`）的代码将因未找到符号而编译失败。  
2. **运行时标签不匹配** – 若旧的 CRD 或外部控制平面仍依赖旧标签名称，K8s 对象的标签/注解将不再匹配，导致资源调和失效（例如，旧版的 DaemonSet 仍在监听 `nvidia.com/snapshot-checkpoint-id`）。  
3. **文档/示例未同步** – 项目文档、 Helm chart、示例 YAML 中仍可能出现已废弃的标签/注解字段，使用者升级后会产生不可预期的行为。  
4. **测试遗漏** – 虽然核心控制器覆盖率提升，但仍有可能遗漏对 `snapshotprotocol.RestoreTargetLabel`、`CheckpointVolumeName` 等在其他模块（如 `restore` 逻辑）中的使用检查。

**SHA**: `9b3e924` | 🔗 [查看提交](https://github.com/ai-dynamo/dynamo/commit/9b3e9249fda1825587313c1a6d9ce561121e2389)

**🎯 变更类型**：功能增强  
**⚡ 重要程度**：🔴高  
**📋 变更摘要**：在 KV‑Router 的事件处理路径中加入 `EventDedupFilter`，对 vLLM 产生的 KV‑Cache **store/remove** 事件按 DP rank 进行引用计数去重，并在 `Cleared` 事件到来时统一清理。仅在实际需要向下游发布时才递增 `publish_id`，并为该过滤器补充了完整单元测试。

**🎯 影响范围**：  
- `lib/llm/src/kv_router/publisher/event_processor.rs`（核心去重逻辑、批处理 flush）  
- `lib/llm/src/kv_router/publisher/mod.rs`（导出 `EventDedupFilter`）  
- `lib/llm/src/kv_router/publisher/tests.rs`（新增去重单元测试）  
- 整体 KV‑Router 发布子系统（所有依赖 `RouterEventSink` 的消费者间接受影响）

**🔍 技术洞察**  
- **架构影响**  
  - 引入 ** per‑rank 引用计数表**（`HashMap<u32, HashMap<ExternalSequenceBlockHash, usize>>`），将去重状态本地化到单个事件处理协程，保持原有的单生产者‑多消费者模型。  
  - `BatchingState::flush` 现在需要显式传入 `EventDedupFilter`，并在 `Removed` 事件前调用 `filter_remove`，`Stored` 事件后调用 `track_store`，保证前后顺序一致。  
  - `Cleared` 事件触发 `dedup.clear()`，保持与 indexer “全部块被清除” 的语义同步。  

- **性能影响**  
  - **时间复杂度**：对每个块的 `store` / `remove` 操作均为哈希表的 O(1)（均摊），对批量事件的额外开销与块数线性相关。  
  - **空间复杂度**：每个 DP rank 保存活跃块的引用计数，最坏情况是所有未清除的块哈希都保留在 map 中，可能导致内存占用随 KV‑Cache 大小增长。  
  - **整体收益**：过滤掉重复的 `remove` 事件显著降低下游索引器、网络、日志等组件的工作量，减少不必要的 I/O 与 GC 压力。对常见的 “同一块多次 remove” 场景尤为有效。  

- **安全考虑**  
  - 只涉及纯数据结构与同步，无 `unsafe`。  
  - 去重逻辑采用 **防御式** 处理（对未登记的 `remove` 直接放行），可避免因状态不一致导致的块泄漏或错误删除。  
  - 通过限制潜在的重复删除，间接降低因恶意或异常 vLLM 产生的大量冗余事件导致的 **DoS** 风险。  

**⚠️ 潜在风险**  
1. **内存泄漏风险**：如果 `Cleared` 事件未被触发或 `dedup.clear()` 被遗漏，`per_rank` 中的计数会一直增长，可能在长时间运行的节点上导致 OOM。  
2. **并发假设**：`EventDedupFilter` 仅在单一处理任务内使用，若未来把过滤器迁移到跨任务共享的结构，需要额外的同步保护。  
3. **计数漂移**：异常路径（如进程崩溃后恢复）可能导致计数与实际块不匹配，导致后续 `remove` 被错误过滤。  
4. **发布 ID 递增逻辑**：现在仅在实际产生事件时递增，若外部依赖连续的 `publish_id`（极少数情况），可能产生兼容性差异。  

**💡 关注建议**  
- **监控 & 报警**：在指标系统中加入 `dedup_map_size_per_rank`、`filtered_remove_count` 等监控，及时发现计数地图异常膨胀。  
- **清理策略**：考虑在极端情况下对 `per_rank` 实施上限或 LRU 清理（例如在 `max_batch_blocks` 超过阈值时强制 `clear()`），以防止长时间无 `Cleared` 事件导致的内存泄漏。  
- **文档说明**：在 KV‑Router 使用手册中明确 `Cleared` 事件的语义及去重机制，提醒运维保证 indexer 在需要全局清除时能够发送相应事件。  
- **回退路径**：提供配置开关（如 `enable_dedup_filter: bool`），以便在极端兼容性需求下可以临时关闭去重逻辑进行故障定位。  
- **持续测试**：在 CI 中加入高频 `store`/`remove` 场景的压力测试，验证去重对延迟和吞吐的正面影响，并确保在异常恢复（进程重启）后计数能够重新初始化而不影响功能。

### ⚠️ 潜在风险

1. **配置不匹配**  
   - Python 端若忘记在 `KvTensorLayout.from_vllm_config` 中设置 `use_mla=True`（或误传 `use_mla=False`），将导致 `outer_dim=None`、`inner_dim=None`，Rust 会尝试基于 stride 推断，可能产生错误的布局，进而导致 KV offload 错误或性能回退。  
   - 解决方案：在模型配置 `KvbmModelConfig.use_mla` 与实际 VLLM `attention_backend` 之间保持严格映射，建议在 `KvbmModelConfig` 中加入断言或在 CI 中显式检测。

2. **跨字段校验异常**  
   - `KvbmWorkerConfig.validate()` 会在 `outer_dim` 与 `inner_dim` 不成对出现时返回错误。若已有脚本直接调用 Rust API（不经过 Python 桥接）而未提供两者，可能导致启动失败。  
   - 建议在所有自定义 Rust 调用路径（如自研插件）统一使用 `Builder` 并显式调用 `validate()`，或在文档中明确“外/内维度必须一起提供”。

3. **页面尺寸限制**  
   - 强制 `page_size ≤ 1024` 可能与极端模型（超大块）不兼容，CI 需要提前确认模型的块大小是否在此范围内，否则会报错。  
   - 如需更大块，请在 `KvbmWorkerConfigBuilder` 中提供覆盖选项或在 `validate_page_size` 中放宽限制。

4. **测试超时**  
   - 新增的 `_SERVER_START_TIMEOUT` 默认 600 s，对大型模型友好，但在资源受限的 CI 环境（GPU Mem < 16 GiB）可能导致测试卡死。  
   - 可通过 `KVBM_SERVER_START_TIMEOUT` 环境变量降低，或在 CI 中检测可用 GPU 资源后动态设置。

5. **MLA 布局推断错误**  
   - 当 `outer_contiguous` 为 `false` 且 `shape[1]`（被误解为 outer_dim）> 2 时，当前代码会把它当作标准布局并计算 `inner_dim`，可能在极端形状下产生错误。  
   - 建议在 `outer_contiguous` 分支中加入对 `candidate <= 2` 的判定后，若不满足则直接使用 `outer_dim=1`（MLA）并重新计算 `inner_dim`，以防误判。
