### feat: wire nvext.cache_control TTL-based pinning through Dynamo router (#6213)
**SHA**: `f1d5c95` | 🔗 [查看提交](https://github.com/ai-dynamo/dynamo/commit/f1d5c95a4e646e006422e8c7fd22e22a30fa5972)

**🎯 变更类型**：功能增强 / 架构变更 / 安全修复（实验性质）  

**⚡ 重要程度**：🔴高  

**📋 变更摘要**  
本次 PR 为 Dynamo KV Router 引入 **TTL‑based KV‑cache pinning**（`nvext.cache_control`），在前端提供 `--enable-cache-control` 开关，仅在 `--router-mode=kv` 时可用。实现了请求后自动调用 SGLang worker 的 `pin_prefix` RPC，以在 HiCache 中对指定 token 前缀进行 TTL 限制的固定，防止在高负载下被驱逐。代码涉及前端参数、路由器配置、预处理器、KV Router 新增 `CacheControlClient`、`PinState`、异步 fire‑and‑forget 调用以及完整文档更新。

---

### 🎯 影响范围
- **Frontend**：`components/src/dynamo/frontend/*`（新增 CLI 标志 `--enable-cache-control`、配置校验、参数传递）。  
- **Router 核心**：`lib/llm/src/kv_router/*`（配置结构、PushRouter、PinState、lazy 初始化 `CacheControlClient`、在请求完成后触发 `spawn_pin_prefix`）。  
- **预处理器**：`lib/llm/src/preprocessor.rs`、`lib/llm/src/protocols/common/preprocessor.rs`（解析 `nvext.cache_control` 并把 TTL 写入 `RoutingHints`）。  
- **协议层**：`lib/llm/src/protocols/openai/nvext.rs`（新增 `CacheControl` 类型、TTL 解析逻辑）。  
- **Python 绑定**：`lib/bindings/python/rust/llm/entrypoint.rs`（暴露新增 `router_enable_cache_control` 参数）。  
- **SGLang 端点**：`components/src/dynamo/sglang/request_handlers/handler_base.py`（实现 `pin_prefix` RPC 与 `cache_control` service mesh）。  
- **文档**：新增/更新 SGLang Agentic 工作负载、Cache Control、nvext 字段、导航等。  

---

## 🔍 技术洞察

| 维度 | 影响分析 |
|------|----------|
| **架构影响** | - **服务网格**：在 KV Router 与 SGLang worker 之间新增 `cache_control` 服务网格通道，保持与现有 `engine`、`prefill`、`decode` 端点的一致性。<br>- **懒加载**：通过 `tokio::sync::OnceCell` 延迟创建 `CacheControlClient`，仅在第一次需要 PIN 时才建立连接，避免不必要的 RPC 初始化成本。<br>- **路由决策**：`RoutingHints` 里加入 `cache_control_ttl`，对已选 worker 记录 token 前缀，用于后置 PIN，保持原有调度路径不变。 |
| **性能影响** | - **正向路径**：新增字段解析、PinState 构造的微量 CPU/内存开销（< 1 µs），对吞吐几乎无影响。<br>- **额外 RPC**：完成一次生成后会 **fire‑and‑forget** 调用 `pin_prefix`，产生一次短小的 JSON 请求/响应（< 1 KB），对单请求延迟影响微乎其微（网络 RTT‑< 1 ms 在同节点内部）。<br>- **缓存命中率**：成功 PIN 后可显著提升后续对同前缀的 `cached_tokens` 比例，降低 KV 重复计算，尤其在长对话、多轮交互的 Agent 场景下可削减 10‑30% 的显式推理时间。<br>- **内存占用**：Pinned token 会占用 HiCache 主机端内存，受 `SGLANG_HICACHE_MAX_PINNED_RATIO` 限制，超出配额的 PIN 请求会被拒绝，避免 OOM。 |
| **安全考虑** | - **权限**：此功能仅在内部 Router‑Worker 通信（受 Service Mesh 认证/授权机制）中使用，未向外部暴露。<br>- **输入校验**：TTL 通过 `CacheControl::ttl_seconds()` 强制在 `[300, 3600]` 秒范围内，防止异常长时间占用缓存。<br>- **异常处理**：`spawn_pin_prefix` 中捕获 RPC 错误并记录 `warn`，不会导致请求失败或进程崩溃。 |
| **可维护性** | - **代码分离**：`cache_control` 模块独立于核心路由逻辑，易于单元测试和未来扩展（如 `unpin_prefix`、优先级 pin）。<br>- **文档同步**：完整的 Markdown 文档、导航、API 示例已更新，降低使用门槛。<br>- **向后兼容**：默认 `router_enable_cache_control = false`，旧版部署不受影响；新增 CLI 参数为可选，且在非 KV 模式下会提前报错。 |

## ⚠️ 潜在风险

1. **HiCache 必须开启**  
   - 若用户忘记 `--enable-hierarchical-cache`，PIN 请求会被 worker 拒绝，导致日志噪声且失去功能。  
2. **Pin 预算耗尽**  
   - `SGLANG_HICACHE_MAX_PINNED_RATIO` 默认 `0.0`，若未显式打开，所有 PIN 会被 silently拒绝；若设置过高，可能导致大量无效缓存占用，影响普通请求的 eviction。  
3. **TTL 过期时的瞬时抖动**  
   - 大量 PIN 同时到期可能触发一次集体 eviction，短时导致缓存命中率下降。  
4. **异步 fire‑and‑forget**  
   - `spawn_pin_prefix` 中的 `stream.next()` 仅读取首响应，忽略后续流，若实现方后续返回错误信息，将不被捕获。  
5. **跨语言/版本兼容**  
   - Python SDK、OpenAI 兼容层需要同步更新 `extra_body.nvext.cache_control`，否则用户可能误以为已开启。  
6. **监控缺失**  
   - 当前 Grafana 面板中移除了 “Active PIN Count” 面板，导致运维难以直观看到 PIN 使用情况；需补充新的监控指标。

### feat: allow deepseek_v3 architecture to use Kimi's bpe pattern (#6653)
**SHA**: `a9b6f95` | 🔗 [查看提交](https://github.com/ai-dynamo/dynamo/commit/a9b6f95c2dd9bdbcf1fe69a627f0ea035351d03a)

**🎯 变更类型**：功能增强  

**⚡ 重要程度**：🔴 高  

**📋 变更摘要**  
- 在 `detect_bpe_pattern()` 中加入对 `deepseek_v3` 模型类型的匹配，使其复用已有的 KIMI BPE 正则（`KIMI_PATTERN`）。  
- 解决 DeepSeek V3（实际为 Kimi‑2.5）模型在 `config.json` 中使用 `deepseek_v3` 但仍提供 KIMI‑style `tiktoken` 文件的问题。  
- 让该架构的模型可以直接使用现有的 tokenizer 而不必额外提供 `tokenizer.json`。

**🎯 影响范围**  
- `lib/llm/src/tokenizers/tiktoken.rs`（Tokenizer 识别逻辑）  
- 受影响的模型加载路径：所有基于 `tiktoken` 的模型，尤其是 DeepSeek V3（Kimi‑2.5）系列。  

**🔍 技术洞察**  
- **架构影响**：  
  - 仅在 tokenizer 子模块新增一种 model_type → pattern 的映射，属于局部扩展，不会改变整体模块接口或依赖结构。  
  - 通过注释明确了此映射的前置假设（DeepSeek V3 不使用纯 tiktoken 模型文件），保持了代码可读性。  

- **性能影响**：  
  - 代码路径仅多一次字符串匹配，CPU 开销与原实现相同，几乎不可感知。  
  - 不会影响 tokenization 速度或内存使用。  

- **安全考虑**：  
  - 变更不涉及外部输入的解析或执行，仅是内部枚举匹配，未引入新的攻击面。  
  - 唯一需要关注的是误将非 KIMI‑compatible 模型标记为 `deepseek_v3`，导致 tokenization 失效，从而可能在上层触发不可预测的异常（但不涉及安全泄露）。  

**⚠️ 潜在风险**  
1. **误判风险**：若未来出现真正使用 `deepseek_v3` 且提供 `tiktoken.model` 文件的模型，仍会走 `KIMI_PATTERN`，可能产生 tokenization 错误。  
2. **文档同步**：目前错误信息仍列出 “Currently supported: kimi, kimi_k2, kimi_k25, deepseek_v3”。若忘记更新对应文档或发行说明，使用者可能对支持范围产生误解。  
3. **回归风险**：对 `detect_bpe_pattern()` 的单元测试覆盖不足，新增分支如果出现拼写错误或忘记引入对应常量，会导致编译通过但运行时 panic。  

**💡 关注建议**  
- **单元测试**：为 `deepseek_v3` 添加专属测试用例，验证在该 `model_type` 下返回 `KIMI_PATTERN`，并确保 tokenization 结果与预期一致。  
- **注释与文档**：在项目 README 或 tokenizer 文档中补充 “DeepSeek V3 (Kimi‑2.5) 兼容 KIMI BPE 模式”，避免使用者疑惑。  
- **未来兼容性**：考虑将 `deepseek_v3` 与 “pure DeepSeek V3” 区分为两个枚举值（例如 `deepseek_v3_kimi`），以便未来真正的 DeepSeek V3 模型出现时能够轻松切换。  
- **错误提示**：在 `Unsupported tiktoken model_type` 错误信息中加入 “若模型提供 tokenizer.json，请使用 tokenizer.json 方式”。  
- **代码审查**：确保 `KIMI_PATTERN` 常量在其他地方未被误修改，保持 BPE 正则的稳定性。  

整体来看，此次改动为低风险的功能增强，提升了对 DeepSeek V3（Kimi‑2.5）模型的即插即用兼容性，建议快速合入主线并配合上述注意事项进行回归验证。

### feat: improve mooncake_bench sweep logic and throughput accounting (#6631)
**SHA**: `824b456` | 🔗 [查看提交](https://github.com/ai-dynamo/dynamo/commit/824b456ffbba3362816c97cd0621c8b426607392)

**🎯 变更类型**：功能增强 / 性能优化 / 重构  

**⚡ 重要程度**：🔴高  

**📋 变更摘要**  
1. 重构 `mooncake_bench` 基准测试的 sweep 逻辑：引入正向/逆向持续时间序列、基于索引器特性的多线程/单线程不同遍历顺序、以及提前终止策略（连续 5 步维持 95% 产出或出现 5 倍饱和）。  
2. 改进吞吐量统计：增加 `count_events` 参数，仅对支持删除（`remove`）的索引器计入事件数量，提高指标的真实性。  
3. 新增序列化结构体并将 sweep 结果导出为 JSON，同时把默认 GPU block 数从 1 048 576 降到 16 384，以更好触发驱逐和淘汰路径。  

**🎯 影响范围**  
- `lib/kv-router/benches/mooncake_bench.rs`（基准测试主入口、统计与绘图）  
- 依赖 `serde` 的序列化/写文件路径（可能影响编译特性）  
- 受 `IndexerArgs::supports_remove` 与 `IndexerArgs::is_multi_threaded` 判定的索引器实现（如 `nested-map`、`concurrent-radix-tree` 等）  

**🔍 技术洞察**  

- **架构影响**  
  - 基准测试层代码的抽象进一步分离：新增 `supports_remove` / `is_multi_threaded` 辅助函数，使得不同索引器的特性在 benchmark 中得到统一判定。  
  - sweep 生成顺序（low→high 与 high→low）基于多线程特性动态切换，避免在高并发索引器上出现“先满后慢”的误判。  
  - 增加 `SweepStepResult`、`BenchmarkResults` 的 `serde::Serialize` 派生，实现了 JSON 输出，提升了结果可消费性（用于 CI、可视化或后续分析）。  

- **性能影响**  
  - **更真实的吞吐量计数**：仅在支持 `remove` 事件的索引器上计入事件块，防止那些本身不产生删除的实现（`naive-nested-map`、`inverted-index`）误报吞吐量。  
  - **提前终止**：  
    - 对多线程索引器，连续 5 步保持 `>= 95%` 的 offered 与实际块吞吐率即停止，显著缩短长时间的 sweep（尤其在高负载时）。  
    - 对单线程索引器，出现 `offered_block_throughput > 5 × block_throughput` 即停止，防止在“供大于求”阶段耗时无意义的遍历。  
  - **默认 GPU block 数降低**：把默认值从 `1 048 576` 降至 `16 384`，在基准测试中更容易触发块迁移与驱逐路径，能够更快暴露缓存淘汰瓶颈。  

- **安全考虑**  
  - 新增文件写入（JSON）使用 `std::fs::write`，路径来源于 CLI 参数 `args.sweep_output`，默认受用户控制，未引入外部输入解析风险。  
  - 引入 `serde` 序列化，不涉及敏感数据，仅是 benchmark 结果，风险可忽略。  

**⚠️ 潜在风险**  

| 风险点 | 描述 | 可能影响 |
|--------|------|----------|
| **提前终止逻辑误判** | 对多线程索引器使用 “连续 5 步 >=95%” 作为停止条件，若负载波动导致短暂满足后仍有潜在瓶颈，可能导致报告的最佳点过早收敛。 | 产生过于乐观的性能评估，误导 CI 或发布决策。 |
| **`count_events` 判定错误** | `supports_remove` 只排除 `naive-nested-map` 与 `inverted-index`，若未来新增不产生 `remove` 的实现未在白名单中，仍会错误计入事件吞吐。 | 统计偏差，导致不同实现之间不可比。 |
| **默认 GPU block 变更** | 将默认块数调低可能导致已有基准脚本（未显式指定 `--num-gpu-blocks`）在迁移到新版本后表现异常。 | CI 基准结果突变，需要更新文档或迁移脚本。 |
| **序列化依赖** | 引入 `#[derive(Serialize)]` 需要在 `Cargo.toml` 中开启 `serde` feature；若仓库或 downstream 项目未启用该 feature，编译会失败。 | 编译阻断，CI 需要同步更新。 |
| **JSON 路径拼接** | 使用 `replace(".png", ".json")` 等方式生成文件名，若用户指定的输出路径不含这些后缀，生成的文件名可能异常。 | 结果文件可能写入错误位置或被覆盖。 |

**💡 关注建议**  

1. **验证提前终止的鲁棒性**  
   - 在 CI 中加入对 “连续 5 步 >=95%” 与 “5×供给” 两种条件的对照实验，确保早停不会隐藏后期的性能下降。
