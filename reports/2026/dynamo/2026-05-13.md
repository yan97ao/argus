### fix: benchmark docs to use DYN_TOKENIZER env var (#9477)
**SHA**: `b79080c` | 🔗 [查看提交](https://github.com/ai-dynamo/dynamo/commit/b79080cb315f4bd4df06c4a202d2433dda79280d)

**🎯 变更类型**：Bug修复 / 文档/配置同步  
**⚡ 重要程度**：🔴高  
**📋 变更摘要**：将所有基准测试相关的环境变量及模板占位符从 `DYN_TOKENIZER_BACKEND` 统一改为 `DYN_TOKENIZER`，并在 sweep 脚本中更新映射表，使其接受 `"default"` 与 `"fastokens"` 两个取值。文档同步对应说明，确保使用者在本地或 Kubernetes 部署时只需关注单一变量 `DYN_TOKENIZER`。

**🎯 影响范围**：  
- `benchmarks/frontend/README.md`  
- `benchmarks/frontend/dgd/templates/*.yaml`（mocker、vllm、vllm-gpt-oss-20b）  
- `benchmarks/frontend/scripts/scaling-test.md`  
- `benchmarks/frontend/scripts/sweep_k8s/dgd.py`、`templates/template.py`  
- `docs/features/tokenizer/README.md`  

**🔍 技术洞察**  

- **架构影响**：  
  - 仅涉及基准测试层面的部署脚本与文档，不会影响 Dynamo 核心服务或模型推理路径。  
  - 统一变量名后，Grove Operator（负责重建 Frontend Pod）仍然通过读取 `DYN_TOKENIZER` 环境变量来决定使用的分词器后端，保持原有自动重启机制。  

- **性能影响**：  
  - 变量名的更改本身不影响运行时性能。  
  - 映射表调整后，`fast` 与 `fastokens` 均映射为 `"fastokens"`，避免用户在本地使用 `--tokenizer fast` 时出现不一致的后端选项，从而确保基准测试能真实比较 HF 与 fastokens 两种实现。  

- **安全考虑**：  
  - 变更仅涉及配置层面，无新增外部依赖或安全接口。  
  - 需要注意旧的 `DYN_TOKENIZER_BACKEND` 环境变量若仍被保留在生产/CI 环境中，将被忽略，可能导致意外使用默认分词器，进而产生安全或合规审计差异（如未使用预期的 fasttokens 加速）。  

**⚠️ 潜在风险**  

1. **向后兼容性**：已有脚本、CI/CD 流水线或用户手动部署仍使用 `DYN_TOKENIZER_BACKEND`，在升级后将失效，导致前端 Pod 启动时默认回退到 HuggingFace 分词器。  
2. **文档/示例同步遗漏**：若项目其他未被扫描到的 README、Jupyter Notebook 或第三方指南仍引用旧变量，使用者可能产生困惑。  
3. **映射表错误**：`TOKENIZER_BACKEND_MAP` 中将 `"fast"` 映射为 `"fastokens"`，若后续引入新的后端名称（如 `"custom"`），需要同步更新映射，否则会出现不可预期的值。  

**💡 关注建议**  

- **迁移指南**：在项目的 `CHANGELOG` 或 `UPGRADE.md` 中添加一步迁移说明，提示用户将环境变量、模板占位符统一改为 `DYN_TOKENIZER`。  
- **兼容性保护**：在 `sweep_k8s/dgd.py` 与模板渲染逻辑中加入对 `DYN_TOKENIZER_BACKEND` 的向后兼容检查（如果检测到旧变量则打印警告并自动迁移到新变量），降低突发性部署失败的概率。  
- **文档审查**：使用仓库全局搜索确保没有遗漏的 `DYN_TOKENIZER_BACKEND` 引用，尤其是在 `examples/`、`tests/`、`ci/` 等目录。  
- **测试覆盖**：在 CI 中添加针对 `DYN_TOKENIZER` 与旧变量的回归测试，验证在两种变量存在时前端容器能够正确读取并呈现期望的 tokenizer 后端。  
- **监控/日志**：在前端启动日志中明确打印 “Using tokenizer backend: <value> (env=DYN_TOKENIZER)” 以便用户快速确认实际生效的配置。  

通过上述措施，可确保此次变量名统一的修复平滑落地，避免因老旧脚本而导致的基准测试不一致或部署错误。

### fix: support ModelExpress non-shared-storage mode (#9456)
**SHA**: `31a3639` | 🔗 [查看提交](https://github.com/ai-dynamo/dynamo/commit/31a3639f7a5e567afb21fab7db30c4956dda5606)

**🎯 变更类型**：Bug 修复 / 功能增强  
**⚡ 重要程度**：🔴 高  
**📋 变更摘要**：  
本次提交在 ModelExpress 客户端中新增对 `MODEL_EXPRESS_NO_SHARED_STORAGE` 环境变量的支持，使其在 **非共享存储**（如 RWO PVC、跨命名空间部署）场景下能够通过 gRPC 流式传输模型文件，而不再强制依赖共享文件系统。文档同步更新，提供了 Kubernetes 部署的配置示例及使用注意事项。

**🎯 影响范围**：  
- `lib/llm/src/hub.rs`（模型下载路径决策）  
- `lib/runtime/src/config/environment_names.rs`（新增环境变量常量）  
- `docs/getting-started/local-installation.md`、`docs/kubernetes/model-caching.md`（用户文档）  
- 受 ModelExpress 客户端及所有使用该客户端的运行时（vLLM、Shadow Engine 等）影响。

**🔍 技术洞察**  

- **架构影响**  
  - 在原有“共享存储模式”基础上，加入 **非共享‑存储模式** 分支：通过 `config.cache.shared_storage = false` 使 `MxClient` 在请求模型后，以 gRPC 流式方式把文件直接推送到工作节点本地缓存。  
  - 该改动不破坏现有的共享‑文件系统路径逻辑，向后兼容：只有在环境变量显式开启时才切换。  
  - 对 Operator/Helm 部署仅需在 worker pod 上添加 `MODEL_EXPRESS_NO_SHARED_STORAGE=1`，无需改动 ModelExpress server 本身。

- **性能影响**  
  - **正面**：避免因共享卷不可达导致的二次 HuggingFace 下载，提升首次模型分发成功率，尤其在跨命名空间或 RWO PVC 场景。  
  - **负面**：相比直接读取本地共享卷，gRPC 流式传输会产生额外的网络开销和 CPU 负载（文件分块、校验），在大模型（数十 GB）下首次启动延迟会略高。  
  - 生产环境可通过调节 gRPC chunk 大小（Modelexpress 文档提供）来平衡吞吐与延迟。

- **安全考虑**  
  - 文件通过 **gRPC** 传输，受当前 Modelexpress 通道的 TLS/Authentication 机制保护，未引入新安全漏洞。  
  - 环境变量误配置（未设置或设置错误）会导致回退到直接下载，可能泄漏内部模型访问凭证到外部网络；因此建议在 CI 中加入 `env` 校验。

**⚠️ 潜在风险**  

1. **环境变量误用**：`MODEL_EXPRESS_NO_SHARED_STORAGE` 未显式设置或拼写错误会导致默认共享模式失败，进而触发自动回退的 HuggingFace 下载，增加带宽成本。  
2. **缓存一致性**：在非共享模式下，每个 worker 会在本地保存一份缓存，若后续更新模型，需要在所有节点上统一清理或使用对象存储统一管理，避免出现“版本不一致”问题。  
3. **资源占用**：首次流式下载时工作节点的本地磁盘会瞬时占用完整模型大小，需确认节点磁盘余量足够。  
4. **兼容性**：旧版 Modelexpress 镜像（v0.2 及以下）不支持统一 `mx` loader，仍依赖 `mx-source`/`mx-target`，在非共享模式下可能需要额外的 `--load-format` 参数或升级镜像。

**💡 关注建议**  

- **对开发者**  
  - 在 CI/CD 流水线中添加对 `MODEL_EXPRESS_NO_SHARED_STORAGE` 的检查，确保仅在需要时开启。  
  - 为使用非共享模式的部署编写自动化脚本，统一清理 worker 本地缓存（如 `rm -rf /tmp/model_cache`）以防磁盘泄漏。  
  - 若使用旧版 Modelexpress，建议升级至 v0.3+ 以获得统一 `mx` loader 与流式传输的完整支持。  

- **对用户/运维**  
  - 在 Kubernetes 部署时，依据存储类选择：  
    - **RWX PVC** → 保持默认共享模式（更快）。  
    - **RWO PVC / 跨命名空间** → 添加 `MODEL_EXPRESS_NO_SHARED_STORAGE=1`，并监控网络流量与磁盘使用。  
  - 使用 `kubectl describe pod` 或 `env` 检查变量是否生效；在出现 “Failed to resolve local model path after server download” 警告时，确认是否已启用非共享模式。  
  - 评估 gRPC 带宽需求，尤其在多节点大模型部署场景下，适当调整 `ModelExpress` 的 `chunk_size` 参数以降低突发流量。  

---  

通过这次改动，Dynamo 在 Kubernetes 环境下对 ModelExpress 的部署灵活性大幅提升，能够在缺乏共享存储的云原生场景中依然实现高效的模型分发，风险主要集中在配置和资源管理上，建议结合上述建议进行验证与监控。

### refactor: move branch-sharded indexer to canonical module (#9334)
**SHA**: `6da48be` | 🔗 [查看提交](https://github.com/ai-dynamo/dynamo/commit/6da48be4d2a39cd87e5a0b213ea4c0f89fe7e0b9)

**🎯 变更类型**：重构/架构变更  
**⚡ 重要程度**：🔴高  

**📋 变更摘要**  
- 将原先的 `anchor_aware_branch_sharded.rs`（约 1.5 k 行） 删除，迁移并精简实现到 `branch_sharded.rs`（约 1.3 k 行）。  
- 在 `lib/kv-router/src/indexer/mod.rs` 中改为公开 `branch_sharded`，并保留 `AnchorAwareBranchShardedIndexer` 的 *deprecated* 别名以兼容旧代码。  
- 同时去除了未使用的 `ShardReadPool`、FNV‑1a 前缀散列逻辑以及一些冗余的计数/广播实现，统一使用基于 **RoutingTrie** 的分片路由。  

**🎯 影响范围**  
- **核心模块**：`kv-router` 中的索引器实现（所有使用 `BranchShardedIndexer` / `AnchorAwareBranchShardedIndexer` 的地方）。  
- **公开 API**：`kv-router::indexer` 的导出层，外部依赖若直接引用 `anchor_aware_branch_sharded` 将编译失败。  
- **测试与基准**：相关单元测试、benchmark 仍然引用 `BranchShardedIndexer`，但可能出现新的编译警告（已标记为弃用）。  

**🔍 技术洞察**  

| 维度 | 影响说明 |
|------|----------|
