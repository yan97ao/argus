### feat(kv-router): share core indexer metrics (#10768)
**SHA**: `2d5d7eb` | 🔗 [查看提交](https://github.com/ai-dynamo/dynamo/commit/2d5d7eb603af21926887580fe3a212c7926fbaad)

**🎯 变更类型**：功能增强  

**⚡ 重要程度**：🔴高  

**📋 变更摘要**  
- 为 KV‑router 的核心索引器引入统一的 Prometheus 指标（事件计数、警告计数），并在运行时通过 `AppState::new` 自动注册。  
- 添加 `create_indexer_with_metrics` 等新建函数，使每个 `Indexer` 在创建时接入共享的 `KvIndexerMetrics` 实例，实现跨模型/租户的全局聚合。  
- 更新文档、注册流程以及单元测试，以确保新指标在 `/metrics` 端点可被正确导出。  

**🎯 影响范围**  
- `lib/kv-router/src/indexer/metrics.rs`（指标定义与注册）  
- `lib/kv-router/src/services/indexer/backend.rs`（创建索引器入口）  
- `lib/kv-router/src/services/indexer/registry.rs`（WorkerRegistry 现在持有共享指标）  
- `lib/kv-router/src/services/indexer/server.rs`、`mod.rs`（AppState 初始化、Prometheus 注册）  
- 文档 `docs/components/router/standalone-indexer.md`  
- 新增/扩展的单元测试 `standalone_indexer_http.rs`  

**🔍 技术洞察**  

| 维度 | 影响分析 |
|------|----------|
| **架构影响** | - 引入 **全局共享指标对象** (`Arc<KvIndexerMetrics>`) 替代每个索引器单独创建的计数器，简化了指标注册路径，避免了同一进程内的重复注册错误。<br>- `AppState::new` 负责一次性创建并注册所有指标，随后 `WorkerRegistry::new_with_indexer_metrics` 将同一实例注入每个 `Indexer`，实现 **横跨模型/租户的统一聚合**。<br>- 通过 `#[cfg(any(feature = "metrics", feature = "runtime‑protocols"))]` 条件编译，保持在未启用 metrics 时仍能编译运行。 |
| **性能影响** | - 每次 KV 事件或警告计数时仅执行一次 `IntCounterVec::inc_by`，开销与原有单独计数器相当，且**减少了注册阶段的锁竞争**（原先多次注册同名指标会触发 `AlreadyRegistered` 错误）。<br>- 共享 `Arc` 引用带来极小的引用计数开销，整体性能影响可忽略。<br>- 由于所有线程写入同一 `IntCounterVec`，Prometheus 内部使用原子计数器，已具备良好并发性。 |
| **安全考虑** | - 新增的 `/metrics` 端点仍旧受限于已有的 HTTP 监听配置，未改变访问控制。<br>- 指标暴露的细粒度信息（如 `duplicate_store` 警告计数）可能帮助外部观察系统内部恢复行为，但这类信息在生产环境的监控系统中通常是公开的，风险极低。 |
| **可维护性** | - 将指标创建抽象为 `KvIndexerMetrics::new_prometheus` 与 `new_registered`，代码位置集中，后续修改或扩展更易定位。<br>- 新的 `create_indexer_with_metrics` 使调用方显式提供指标实例，提升可测试性（单元测试直接使用 `new_unregistered`）。 |

**⚠️ 潜在风险**  
1. **Metrics Feature 未打开** 时仍会创建 `KvIndexerMetrics::new_unregistered()`，其内部会实例化 `IntCounterVec` 并在没有注册的情况下使用，可能导致 **内存泄漏**（未被 `prometheus::Registry` 清理）或 **重复注册** 如果以后不小心在多个 `AppState` 实例中调用 `new_registered`。  
2. **并发写入同一 CounterVec** 虽然是原子的，但在极高 QPS 场景下可能产生 **热点竞争**（CPU 缓存行争用），需在高负载环境监控指标采集延迟。  
3. **初始化路径改变**：`run_server` 现在先构造 `AppState::new`，若该函数返回错误（如注册失败），服务启动会直接失败，可能影响现有部署的容错策略。  
4. **文档/监控工具兼容性**：新增指标名称和标签需要在监控仪表盘中同步更新，避免出现“未识别指标”警告。  

**💡 关注建议**  
- **上线前测试**：在 CI 中开启 `metrics` feature，运行新增单元测试，确保 `duplicate_store` 警告计数能够被正确导出。  
- **监控准备**：在 Prometheus 抓取配置和 Grafana 仪表盘中加入 `dynamo_kvrouter_kv_cache_events_applied` 与 `dynamo_kvrouter_kv_cache_event_warnings`，并定义相应的告警阈值（如 duplicate_store 警告激增）。  
- **容错处理**：如果指标注册失败（如端口冲突或权限问题），考虑在生产脚本中提供 `--no-metrics` 选项或捕获错误并回退到 `new_unregistered`，避免服务完全无法启动。  
- **性能观察**：在高并发写入场景下观察 `prometheus::process_cpu_seconds_total` 与指标采集延迟，必要时可考虑**分片**（为每个线程/租户创建独立 CounterVec）以降低热点。  
- **代码审查**：确认所有新创建的 `Indexer`（包括可能的自定义或测试实现）都使用 `create_indexer_with_metrics` 或显式传入 `KvIndexerMetrics::new_unregistered()`，防止遗漏导致 *“未注册的指标”* 警告。  

---  

通过以上改动，KV‑router 的核心索引器现在能够在单一进程内统一、可观测地记录关键事件与异常，提升运维可视化能力，同时保持对现有非 metrics 环境的兼容性。

### fix(profiler): keep decode num_request sweep within KV-cache capacity (#10125)
**SHA**: `8b19d75` | 🔗 [查看提交](https://github.com/ai-dynamo/dynamo/commit/8b19d756c7b5874e8cede2f3678060238f7b0bda)

**🎯 变更类型**：Bug修复  

**⚡ 重要程度**：🔴高  

**📋 变更摘要**：  
此提交修复了 `get_num_request_range()` 在解码阶段对 KV‑cache 并发容量的计算错误。原实现使用 `step = (conc_per_dp - 1) * attn_dp_size / (granularity - 1)`，导致上界被放大到 `max_concurrency * attn_dp_size`，在 `attn_dp_size > 1` 时会超出 KV‑cache 限制，引发临时解码工作者的 liveness probe 重启循环。修复后采用 `step = (conc_per_dp - 1) / (granularity - 1)`，在保持每个探测值为 `attn_dp_size` 的整数倍的同时，严格限制在 KV‑cache 能容纳的并发上限。  

**🎯 影响范围**：  
- `components/src/dynamo/profiler/utils/profile_decode.py`（核心业务逻辑）  
- 受影响的上层模块：`profiler`、`planner`、调度器中使用 decode 并发探测的代码路径  
- 相关测试：`components/src/dynamo/profiler/tests/unit/test_get_num_request_range.py`  

**🔍 技术洞察**：  
- **架构影响**：  
  - 仅限于 profiler 子系统的工具函数，没有改变公共 API 或调用约定，整体架构保持不变。  
  - 通过修正 sweep 计算，避免了临时 decode worker 由于 KV‑cache 溢出而被频繁重启，对调度器的稳定性提升明显。  

- **性能影响**：  
  - **正面**：阻止了 KV‑cache 被过度写入导致的内存抖动和频繁的工作线程重启，从而提升整体推理吞吐和延迟的稳定性。  
  - **负面**：在极端 `conc_per_dp` 与 `granularity` 组合下，使用整数除法后产生的 `step` 可能略小，导致最后一次探测值略低于理论上限，可能少探测一些并发点，但对整体性能影响可忽略。  

- **安全考虑**：  
  - 变更不涉及网络、权限或数据加密，仅是内部资源上限的校正，暂无安全风险。  
  - 通过避免 KV‑cache 溢出，间接降低了因内存异常触发的服务崩溃，提升了系统可用性。  

**⚠️ 潜在风险**：  
1. **整数除法舍入**：`step = (conc_per_dp - 1) / (granularity - 1)` 使用 Python 默认的浮点除法后再取 `int(i * step) * attn_dp_size`，在某些极端比例（如 `conc_per_dp` 与 `granularity` 相近）可能导致最后一个探测值比 `conc_per_dp * attn_dp_size` 低 1‑2 个单位，可能少跑一次并发评估。  
2. **向后兼容**：已有外部脚本或监控工具可能依赖旧实现产生的“超限”探测序列，升级后其统计结果会变化。  
3. **分支覆盖不足**：`conc_per_dp < granularity` 分支未受影响，若未来对该分支的实现进行修改，需确保同样不突破 KV‑cache 上限。  

**💡 关注建议**：  
- **回归测试**：在 CI 中加入更广泛的参数矩阵（包括 `attn_dp_size` 大于 8、`engine_max_concurrency` 超过 `DECODE_MAX_CONCURRENCY` 的组合），确保上限始终被正确 clamp。  
- **监控报警**：部署后在生产环境开启 KV‑cache 使用率的监控报警，验证修复是否彻底消除之前的 liveness probe 重启现象。  
- **文档更新**：在 `profile_decode` 模块的 docstring 中注明 `get_num_request_range` 的边界约束以及对 `attn_dp_size` 的整数倍要求，帮助使用者理解返回值的意义。  
- **兼容层**：若担心外部依赖旧行为，可考虑在函数中加入可选的 `legacy=True` 参数，显式开启旧实现，方便平滑迁移。
