### fix: Don't double-count duplicate store events in the tree size. (#7549)
**SHA**: `52382c9` | 🔗 [查看提交](https://github.com/ai-dynamo/dynamo/commit/52382c91ac4b56b344130c6fccc3441c9c471b01)

**🎯 变更类型**：Bug 修复  
**⚡ 重要程度**：🔴高  
**📋 变更摘要**：修正了 `ConcurrentRadixTree` 在处理重复 Store 事件时会出现的 **双重计数** 问题。核心逻辑改为仅在 `worker_lookup.insert` 实际插入新条目时才增加 `num_blocks_added`，并将对最后节点的 worker 插入移动到循环之外，保证与单线程实现的 `lookup.len()` 语义保持一致。新增单元测试验证相同事件再次写入不会导致树尺寸膨胀。  

**🎯 影响范围**：  
- `lib/kv-router/src/indexer/concurrent_radix_tree.rs`（核心索引结构）  
- `lib/kv-router/src/indexer/tests.rs`（新增并行安全性测试）  
- 受影响的业务模块：KV‑Router 的索引器、匹配调度、每个 Worker 的树大小统计。

---

### 🔍 技术洞察

| 维度 | 影响 |
|------|------|
| **架构影响** | - `ConcurrentRadixTree` 仍然保持原有的 **无锁/细粒度锁** 设计，只是把 **计数职责** 从 `op.blocks.len()`（假设全部为新块）迁移到 **实际插入成功** 的判断上。<br>- 计数现在与单线程 `RadixTree` 的 `lookup.len()` 语义统一，提升了 **实现一致性** 与 **可理解性**。 |
| **性能影响** | - 额外的 `if worker_lookup.insert(...).is_none()` 检查和一次条件分支，开销极小（HashMap 插入本身已需锁/原子操作）。<br>- 通过避免错误的 `fetch_add`，在高并发重复写入场景下 **避免了不必要的内存增长** 与 **GC/回收成本**，整体性能预计略有提升。 |
| **安全考虑** | - 本次修改不涉及网络、加密或权限控制，**安全风险几乎为零**。<br>- 通过保证计数的准确性，间接提升了 **资源配额/限流** 等安全/运维相关判断的可靠性。 |
| **可维护性** | - 代码注释清晰说明了 **“锁一次、计数一次”** 的意图，降低了未来误删计数逻辑的风险。<br>- 新增的单元测试覆盖了关键路径，提升了 **回归安全性**。 |

---

### ⚠️ 潜在风险

1. **并发竞态**：虽然 `worker_lookup.insert` 在内部使用了互斥锁（`RwLock`/`Mutex`），在极端高并发下仍可能出现两条线程几乎同时判断 `is_none()` 为 `true`，导致两次 `fetch_add`（计数仍会被双计）。需要确认 `insert` 的原子返回值在锁范围内完整决定计数增量。  
2. **`needs_worker_insert` 逻辑迁移**：将对 `current.write().workers.insert(worker)` 的调用移到循环后，若 `needs_worker_insert` 在循环内部被错误地清除，可能导致最后一个节点的 worker 未被插入。当前代码看似保持原有语义，但在未来对 `needs_worker_insert` 的修改时要注意保持此顺序。  
3. **测试覆盖不足**：新增的测试仅覆盖单线程重复写入的情形，未覆盖 **多线程并发重复写入**（例如 `tokio::join!` 多任务）以及 **跨 worker、跨块的竞争**。若出现未检测的竞态，计数仍可能不准确。

---

### 💡 关注建议

1. **加强并发测试**  
   - 增加 `#[tokio::test]` 的并行写入场景，使用 `tokio::spawn` 同时对同一块调用 `apply_event`，断言 `tree_sizes` 仍为唯一计数。  
   - 引入 `loom` 或类似的模型检查工具，验证 `worker_lookup.insert` 与 `size.fetch_add` 之间的原子性。

2. **审计 `worker_lookup` 实现**  
   - 确认 `insert` 方法在内部已经持有写锁，并在返回 `Option<V>` 前不释放锁，以防出现 “检查‑插入‑计数” 三段式的竞态。若不是，考虑使用 `RwLock::write` 包裹一次完整的 **检查‑插入‑计数** 操作。

3. **文档与注释**  
   - 在 `ConcurrentRadixTree` 的结构体文档中注明 **tree size 计数遵循单线程 `lookup.len()` 语义**，帮助新贡献者快速理解计数来源。  
   - 对 `needs_worker_insert` 变量的作用进行更细致的说明，提醒后续修改者不要在循环内部提前执行插入。

4. **监控 & 报警**  
   - 若系统在生产环境中依据 `tree_sizes` 进行配额/负载均衡，建议在关键指标（如 `tree_sizes` 突增）上加报警，以便在计数异常时快速定位问题。

5. **回滚策略**  
   - 该变更仅影响计数，不会破坏已有数据结构，回滚风险极低。但在升级期间可以保留旧的计数字段作为只读备份，以防意外回滚导致统计不一致。

---

**结论**：此次提交解决了一个 **关键的计数错误**，保证了 `ConcurrentRadixTree` 在重复 Store 场景下的 **数据准确性** 与 **资源统计一致性**，对整体系统的可靠性提升显著。只要在后续加入更严格的并发测试并确认 `insert` 的原子性，即可安全推广到生产。

### feat(gms): e2e failover support for vLLM (#6818)
**SHA**: `1c65a58` | 🔗 [查看提交](https://github.com/ai-dynamo/dynamo/commit/1c65a588d05aedce014e1da901672bfa5f40fc08)

**🎯 变更类型**：功能增强 / 架构变更 / 安全修复  
**⚡ 重要程度**：🔴 高  

**📋 变更摘要**  
本次提交为 vLLM 引入了针对 GPU Memory Service（GMS）的 **shadow（备份）模式**，实现了在同一 GPU 上多实例（主/备）失效切换时的 KV‑cache 共享与延迟分配。新增 `--gms-shadow-mode` 参数、相应配置校验、运行时环境变量 `DYN_GMS_SHADOW_MODE=1`，并在 vLLM 的 worker、模型加载、内存管理等关键路径注入了 GMS‑specific 的锁模式、CUDAGraph 限制、以及一套防止 KV‑cache 冲突的 **shadow‑mode monkey‑patch**。此外，Python 绑定层新增 `DistributedRuntime.set_health_status` 接口，以便在 shadow 引擎的睡眠/唤醒阶段向控制平面报告健康状态。

**🎯 影响范围**  
- `components/src/dynamo/vllm/*`（CLI 参数、配置结构、启动入口）  
- `lib/gpu_memory_service/integrations/vllm/*`（模型加载、worker、补丁、实用工具、shadow‑mode runner）  
- Python 绑定 `lib/bindings/python/*`（新增健康状态接口）  
- 任何使用 `--load-format=gms` 的 Dynamo‑vLLM 部署，尤其在多‑Replica / TP>1 的容错场景下。

---

### 🔍 技术洞察

| 维度 | 影响描述 |
|------|----------|
| **架构影响** | 1. **新增 Shadow 模式**：在同一 GPU 上同时运行 *主动* 与 *备份* 两套 vLLM 实例，备份实例在启动时跳过 KV‑cache 分配、进入睡眠并通过 `FlockFailoverLock` 进行互斥控制。<br>2. **GMS 锁模式**：`configure_gms_lock_mode` 根据 `ENGINE_ID` 自动将非零实例的 `gms_read_only` 设为 `True`，避免多实例 RW 锁竞争导致死锁。<br>3. **健康状态暴露**：`DistributedRuntime.set_health_status` 让 orchestrator（如 K8s）能够依据 “Ready/NotReady” 判断备份实例是否已经进入可接管状态。 |
| **性能影响** | - **启动时内存占用大幅下降**：备份实例不分配 KV‑cache，显著降低显存需求，使得在同一卡上可以容纳更多副本（尤其在 TP>1 场景）。<br>- **切换延迟**：唤醒时会重新分配 KV‑cache，可能触发 OOM 重试循环（最多 60 s），并进行显存可用性检查，导致短暂的恢复窗口。<br>- **CUDAGraph 限制**：强制 `PIECEWISE` 或 `NONE`，放弃 `FULL_*` 模式，可能降低部分图优化带来的吞吐提升。<br>- **额外的锁争用**：`FlockFailoverLock` 在获取锁期间会阻塞备份实例的注册，若锁实现不够高效，可能在高并发失效恢复时出现排队。 |
| **安全考虑** | - **环境变量驱动**：`DYN_GMS_SHADOW_MODE` 与 `FAILOVER_LOCK_PATH` 通过环境变量注入，需在部署模板中严格控制，防止未授权节点误启 shadow 模式导致资源滥用。<br>- **锁模式错误**：若 `gms_read_only` 与 `ENGINE_ID` 不匹配，当前实现会 `raise ValueError`，避免写入冲突，但也可能因配置错误导致服务不可用。<br>- **健康状态 API**：新的 `set_health_status` 直接影响调度判断，若被误调用可能导致 orchestrator 错误回收或重启实例，需要权限校验（目前在 Python 层无额外检查）。 |
| **可维护性** | - 引入大量 **monkey‑patch**（`patch_request_memory`、`patch_register_kv_caches`），代码分散在 `patches.py`，对上游 vLLM 版本升级的兼容性风险提升。<br>- Shadow‑mode 逻辑通过环境变量和配置 flag 隐式激活，代码路径分支多，调试时需要确认 `DYN_GMS_SHADOW_MODE` 状态。<br>- 新增 `GMSShadowModelRunner` 通过 `__class__` 动态替换，虽然保留对象身份，但对类型检查工具（mypy、IDE）不友好。 |

---

**⚠️ 潜在风险**

1. **显存竞争超时**  
   - 在显存紧张场景下，备份实例在 `allocate_kv_cache_on_wake` 可能在 60 s 内仍无法获取足够显存，导致 RuntimeError 并让整个 Pod 重启。  
2. **锁获取失效**

   - 仅在 `cross_validate_config` 中检查 `--gms-shadow-mode` 与 `--load-format=gms` 的对应关系，但未防止用户在运行时通过环境变量直接开启 shadow，可能导致运行时错误。  
6. **健康状态误报**  
   - `set_health_status` 在 shadow 初始化阶段被调用两次（一次在 `run_dynamo_headless` 中，一次在 `Worker.wake_up`），若调用顺序错乱，外部调度系统可能误判实例状态。

1. **标签 Cardinality 爆炸**

### 🔍 技术洞察

| 维度 | 分析 |
|------|------|
| **架构影响** | - 引入 **NVTX** 作为可选插件，使用 **两层 gating**（Cargo `nvtx` feature + 环境变量 `DYN_ENABLE_RUST_NVTX`），在默认情况下对二进制体积和运行时开销几乎为 0。<br>- 为运行时的 **request‑plane** 与 **transport** 层加入显式的 `RequestControlMessage` 字段 `frontend_send_ts_ns`；通过 `serde(default, skip_serializing_if = "Option::is_none")` 保持向后兼容。<br>- 新增 **profiling** profile，保留 `debug` 符号，可在不影响发布版优化的前提下进行高精度采样。<br>- 通过 **RAII guard**（`InflightGuard`、`InflightDecStream`）确保 `REQUEST_PLANE_INFLIGHT` gauge 正确增减，防止异常路径泄漏。 |
| **性能影响** | - **关闭** `nvtx`：编译后宏被完全剔除，理论上 **0 ns** 开销。<br>- **打开** `nvtx` 但未启用环境变量：每个标记只读取一次 `AtomicBool`（≈ 1 ns）。<br>- **打开** 并设置 `DYN_ENABLE_RUST_NVTX=1`：实际调用 `cudarc::nvtx`，约 **50 ns/annotation**，对大多数请求仍可接受，尤其在 profiling 环境下用于定位热点。<br>- 新增的 **Prometheus Counter** 与 **Histogram** 对采样频率几乎无影响，但会略增内存占用（计数器 & 桶）。<br>- `frontend_send_ts_ns` 采用 `SystemTime`，对单机部署几乎无额外开销，跨机器时依赖 NTP 同步。 |
| **安全考虑** | - 只读取本地环境变量 `DYN_ENABLE_RUST_NVTX`，无外部输入。<br>- 新增的网络字段 `frontend_send_ts_ns` 仅用于内部度量，未影响业务逻辑或授权检查。<br>- 引入 `cudarc`（可选）时，需要链接 `libnvToolsExt.so`，其本身不引入安全风险。<br>- 代码路径已在异常分支中 **记录错误计数**（如 TCP/NATS 错误计数），有助于监控异常行为。 |
| **可维护性** | - 使用 **宏 + RAII** 抽象，使得在高层业务代码中仅需一行 `let _nvtx = dynamo_nvtx_range!("name");`，保持代码可读性。<br>- 所有新指标均通过 `ensure_*_registered_prometheus` 函数一次性注册，具备 **幂等**，降低重复注册风险。<br>- `profiling` profile 明确放在根 `Cargo.toml`，便于 CI/Bench 使用。<br>- 对现有 `frontend_perf` 中的 **detokenize** 直方图改为 **累计计数器 + token计数器**，更符合业务需求且避免每 token 记录直方图导致高开销。 |

### ⚠️ 潜在风险

1. **跨机器时间漂移**  
   - `frontend_send_ts_ns` 基于 `SystemTime`，如果前后端机器时钟不同步，网络传输时延统计会出现负值或异常的大数值。  
   - **建议**：在生产环境通过 NTP 或 Chrony 保持时钟同步，或者在监控层对负值/异常值进行过滤。

2. **未开启 `nvtx` 时忘记初始化**  
   - `crate::nvtx::init()` 已在 `Runtime::new` 调用，若用户自行创建 runtime（绕过此入口）可能导致 `enabled()` 始终返回 `false`，但这只影响可视化，不影响功能。  
   - **建议**：在文档中明确必须使用 `Runtime::new` 或自行调用 `nvtx::init`。

3. **特性冲突 / 编译时间**  
   - `nvtx` 依赖 `cudarc`（CUDA 相关），在不具备 CUDA 环境的机器上如果误打开特性，会导致链接错误。  
   - **建议**：在 `Cargo.toml` 中将 `cudarc` 标记为 *optional* 并在对应 `features` 中加入 `nvtx`，已实现此点；仍需在 README 中提醒用户仅在需要时启用。

4. **Metric Cardinality 增长**  
   - 新增的 `request_plane`、`transport`、`routing_overhead` 等指标带有多个标签（如 `router_id`、`transport_kind`），在大规模集群中可能导致 **标签爆炸**。  
   - **建议**：监控 Prometheus 的 `target/metadata`，确保标签集合保持可控；如必要提供 `metric_relabel_configs` 过滤不常用标签。

5. **并发计数泄漏**  
   - `REQUEST_PLANE_INFLIGHT` 采用 `InflightGuard` + `InflightDecStream`，若后续代码在 `generate` 前返回错误而未触发 guard 的 `disarm`，可能导致计数

7. **Python 日志配置** – 为 `gpu_memory_service` 添加统一的 logger，使其在 SGLang/VLLM 中的日志与 Dynamo 其余部分保持一致。

- `components/src/dynamo/*`（snapshot、sglang、vllm）  
- `deploy/helm/charts/snapshot`（RBAC、RoleBinding）
