### fix: prevent false kube discovery removals from transient metadata CR reads (#9413)
**SHA**: `1f9d68d` | 🔗 [查看提交](https://github.com/ai-dynamo/dynamo/commit/1f9d68d7dc1fe563de590f88a5bc4ff48e101938)

**🎯 变更类型**：Bug修复  
**⚡ 重要程度**：🔴高  
**📋 变更摘要**：  
1. 在模型监听器中，避免因为 `ModelDeploymentCard` 已被提前删除而导致 panic，改为记录警告并安全返回。  
2. 为 Kube discovery daemon 引入 **CR 元数据缓存**，在读取到 `spec.data` 为 `null` 或反序列化失败的瞬时状态时，复用上一次成功读取的元数据，防止错误地将已经就绪的实例从快照中剔除。  
3. 新增日志辅助函数 `managed_fields_summary` 与单元测试，提升调试可观测性。

**🎯 影响范围**：  
- `lib/llm/src/discovery/watcher.rs`（模型卡管理）  
- `lib/runtime/src/discovery/kube/daemon.rs`（Kubernetes 发现守护进程）  
- 相关单元测试文件  

**🔍 技术洞察**：

- **架构影响**：  
  - 为 `DiscoveryDaemon` 增加了 `CachedCrMetadata` 结构体与 `valid_cr_cache` 哈希表，实现 **一次性缓存** 与 **基于 UID 的验证**。  
  - 快照聚合逻辑从仅依赖当前 CR 状态，转为 “当前有效或缓存的已确认元数据” 两条路径，提升对 **Kubernetes 控制面瞬时不一致**（如 CR 被标记删除但还在同步）的容错能力。  
  - `ModelWatcher` 的错误处理从 `anyhow::bail!`（会中止整个任务）改为 `tracing::warn!` 并返回 `Ok(None)`，使模型删除事件更为 **幂等**，防止因重复或延迟的删除事件导致进程崩溃。

- **性能影响**：  
  - 引入的缓存是 `HashMap<String, CachedCrMetadata>`，对每个 CR 只存一份 `Arc<DiscoveryMetadata>`，内存占用极低（数百 KB 级别，取决于实例数量）。  
  - 每次聚合快照多一次哈希查找和一次 `retain` 操作，时间复杂度保持在 O(N)（N 为 CR 数量），对已有的 **500ms 防抖** 窗口影响可忽略。  
  - 日志新增的 `managed_fields_summary` 只在出现错误或空数据时触发，开销同样可接受。

- **安全考虑**：  
  - 代码未引入外部输入的执行路径，仅在日志中打印 CR 的 `uid`、`resource_version` 与 `managed_fields`，这些信息本属于集群内部可见，不会泄露额外敏感数据。  
  - 通过 UID 校验确保 **不误用已被删除并重新创建的 CR**，避免潜在的“旧数据被误当成新实例”攻击向量。  

**⚠️ 潜在风险**：  
1. **缓存陈旧**：如果 CR 长时间保持无效（如 `spec.data` 持续为 `null`），守护进程会一直使用旧的元数据，可能导致实际已下线的实例仍被视为就绪。  
2. **内存增长**：在极端情况下（大量瞬时出现后立即消失的 CR），缓存会保留这些键，虽然 `retain` 已在每轮聚合中清除未观察到的条目，但仍需确保 `observed_crs` 集合覆盖所有删除路径。  
3. **错误隐藏**：将原先的 `bail!` 改为警告后，某些真正的逻辑错误（比如模型卡意外缺失）可能被忽略，需要运维监控相应的 WARN 日志。  
4. **并发一致性**：缓存的写入与读取在单线程 async 循环内完成，理论上安全，但若后续代码改为多任务并发访问，需要加入同步原语。  

**💡 关注建议**：  
- **监控**：在生产环境开启 `tracing::warn` 的日志聚合，关注 “ModelDeploymentCard already absent” 与 “DynamoWorkerMetadata CR has null spec.data” 警告频率，异常升高时排查 CR 同步或控制面故障。  
- **指标**：考虑添加 Prometheus 指标 `dynamo_cr_cache_hits` / `dynamo_cr_cache_misses`，帮助评估缓存命中率与潜在陈旧数据比例。  
- **缓存淘汰**：若业务出现大量短暂 CR（如滚动升级），可在 `retain` 逻辑中加入基于时间戳的过期策略，以防止偶发的键残留。  
- **回滚检查**：确认 `ModelWatcher` 的调用方在收到 `Ok(None)` 时能够安全处理 “无模型卡” 的情况，防止业务层因返回值忽略而产生隐蔽错误。  
- **持续测试**：在 CI 中加入包含 `null`、反序列化错误以及 UID 变化的端到端场景，确保缓存逻辑在各种 Kube 状态下均保持一致性。

### fix: enable DGDSA for planner-managed DGDR workers (#9450)
**SHA**: `7997117` | 🔗 [查看提交](https://github.com/ai-dynamo/dynamo/commit/7997117780e05eb081c0caf106c948c79289f6da)

**🎯 变更类型**：Bug修复 / 功能增强  
**⚡ 重要程度**：🔴高  

**📋 变更摘要**  
本次提交在 `dgd_generation.py` 中实现 `enable_planner_worker_scaling_adapters`，使得由 Planner（非 advisory）管理的 DGDR worker 服务默认开启 DGDSA（Scaling Adapter）。同时补充了大量单元/集成测试，验证在不同 Planner 模式（prefill、decode、agg、disagg）以及服务命名/缺失 `subComponentType` 场景下的正确行为。  

**🎯 影响范围**  
- `components/src/dynamo/profiler/utils/dgd_generation.py`（核心配置生成逻辑）  
- `components/src/dynamo/profiler/tests/unit/test_helpers_profile_sla.py`、`components/src/dynamo/profiler/tests/integration/test_profile_sla_dgdr.py`（新增测试）  
- 受 Planner 生成的 DGDR 配置影响的全部部署路径（Kubernetes‑Planner、DGDSA 控制器）  

**🔍 技术洞察**  

- **架构影响**  
  - 在 **Dynamo Graph Deployment Request (DGDR)** 配置生成阶段，新增一次对 `services[].scalingAdapter.enabled` 的写入，强化了 Planner 与 DGDSA 的耦合：Planner 负责决定 replica 数，DGDSA 负责实际伸缩。  
  - 通过 `PlannerConfig.advisory` 检查，保证在“仅建议”模式下不自动开启，保持向后兼容。  
  - 引入 **service‑name 推断** (`_infer_subcomponent_from_service_name`) 与 **未标注 worker 计数** (`untyped_worker_count`) 逻辑，可在缺少 `subComponentType` 时自动补全，提升对自定义模板的适配能力。  

- **性能影响**  
  - 代码仅在配置字典层面做一次遍历（O(N)）并添加/修改少量键值，对启动时的配置生成几乎没有可感知的性能开销。  
  - 对已开启的 `scalingAdapter` 不会产生额外计算；对未开启的服务保持原样，运行时行为保持不变。  

- **安全考虑**  
  - 该改动本质上是对内部配置的写操作，不涉及网络、权限或加密路径。  
  - 唯一潜在安全点是 **误开启 ScalingAdapter**：若误将不应受 Planner 管理的 worker 标记为可伸缩，可能导致资源超额分配或服务意外重启。代码已通过 `planner_config.advisory` 与子组件匹配进行过滤，风险有限。  

**⚠️ 潜在风险**  

1. **误判子组件**：基于服务名称的推断 (`prefill` / `decode` 关键字) 在自定义 worker 名称中可能出现误判，导致错误地开启或关闭 ScalingAdapter。  
2. **单一未标注 worker 的 agg 模式**：当 `planner_mode == "agg"` 且仅有一个未标记的 worker 时，会强制标记为 `decode`。若模板实际是 `prefill`（极少见），会产生不符合预期的伸缩行为。  
3. **配置突变副作用**：`enable_planner_worker_scaling_adapters` 直接在入参 `config_dict` 上就地修改，若调用方在后续仍需要原始配置的快照，可能产生不可预料的副作用。  
4. **兼容性**：已有外部工具可能在生成 DGDR 时手动设置 `scalingAdapter`，本次自动覆盖可能冲突（虽然实现时优先保留已有 `scalingAdapter`，但会强制 `enabled=True`）。  

**💡 关注建议**  

- **文档同步**：在 Dynamo 官方文档中明确说明 Planner（非 advisory）会自动为 worker 开启 DGDSA，并阐述 “service name 推断” 与 “agg 模式单 worker” 的实现细节。  
- **日志与审计**：在 `enable_planner_worker_scaling_adapters` 中加入调试/信息日志，记录哪些服务被自动标记为 `subComponentType` 与 `scalingAdapter.enabled`，方便用户排查意外行为。  
- **防御性检查**：若未来引入更复杂的子组件类型，建议将推断逻辑抽象为可插件的策略，避免硬编码关键字导致误判。  
- **配置快照**：在调用前可考虑 `deepcopy` 配置字典，或在函数文档中标明“该函数会原地修改输入”。  
- **回归测试**：补充针对以下情况的测试，以进一步降低风险：  
  1. 多个未标注 `subComponentType` 的 worker（agg 模式下）  
  2. 非标准命名但包含 “prefill”/“decode” 子串的 worker  
  3. 已经存在 `scalingAdapter` 但 `enabled=False` 的情况，确保仍被强制启用。  

通过上述措施，可确保本次功能提升在保持高可用伸缩的同时，最小化潜在的误配置与运维风险。

**📋 变更摘要**：  
- 将原本的全局 `g_malloc_callback` / `g_free_callback` 改为基于 C++11 magic‑static 单例的 `Callbacks` 结构，保证在 CPython 3.13+ 的 *free‑threaded* 运行时下，无需显式同步即可安全读取回调指针。  
- 在模块初始化 `PyInit__allocator_ext` 中加入对 `Py_GIL_DISABLED` 的检测，若 CPython 编译开启无 GIL 支持，则通过 `PyUnstable_Module_SetGIL` 声明模块不使用 GIL。  
- 移除旧的引用计数手动管理代码，改为在单例构造时一次性 `Py_XINCREF`，并在后续调用中直接使用已持有的引用。

**🎯 影响范围**：  
- `core` → `gpu_memory_service/client/torch/extensions/allocator.cpp`（CUDA Pluggable Allocator 与 PyTorch 交互层）  
- Python C‑extension 模块 `_allocator_ext`（被 PyTorch 的自定义 CUDA 分配器加载）  

**🔍 技术洞察**  

- **架构影响**  
  - **解耦全局状态**：通过 `Callbacks` 单例取代裸全局指针，降低模块间的隐式耦合，符合 “单例懒初始化 + 隐式只写一次” 的设计模式。  
  - **兼容 free‑threaded CPython**：在 `PyInit__allocator_ext` 中声明 `Py_MOD_GIL_NOT_USED`，使得该扩展在 CPython 3.13+ 的无 GIL 模式下能够被加载且不触发运行时断言。  
  - **单例初始化顺序**：利用 C++11 静态局部变量的线程安全懒初始化，保证 `callbacks()` 第一次调用前已经完成 `init_module`（即 Python wrapper 的 `_ensure_callbacks_initialized`）的同步调用，从而满足 “first call wins” 的契约。

- **性能影响**  
  - **无锁读取**：回调指针在单例构造后属于只读状态，后续 `my_malloc` / `my_free` 仅执行一次指针读取，无需锁或原子操作，保持原有的微秒级调用开销。  
  - **初始化开销**：单例的懒初始化在第一次 `my_malloc`/`my_free` 前完成，涉及一次 `Py_XINCREF` 与结构体拷贝，成本极低且只在首次分配时触发。  
  - **模块加载**：额外的 `PyUnstable_Module_SetGIL` 调用在模块初始化阶段执行，几乎可以忽略不计。

- **安全考虑**  
  - **引用计数安全**：在单例构造时对回调对象 `Py_XINCREF`，随后不再对其 `DECREF`，避免在模块已经卸载后仍持有悬空指针；但这意味着在解释器终止前这些回调对象始终保持引用，需确保解释器关闭前已完成全部分配/释放操作，否则可能在解释器清理阶段出现 “leaked reference” 警告。  
  - **再次初始化容错**：后续对 `callbacks(malloc_cb, free_cb)` 的调用会被静默忽略（单例已经构造），若上层误以为可以重新绑定回调，则实际不会生效，可能导致难以定位的错误。建议在文档中明确“一次性绑定”。  
  - **兼容性**：`PyUnstable_Module_SetGIL` 为实验性 API，仅在启用了 `Py_GIL_DISABLED` 的 CPython 构建中可用；在旧版 CPython（未定义该宏）下仍保持原有行为，兼容性良好。

**⚠️ 潜在风险**  

| 风险点 | 可能后果 | 缓解措施 |
|--------|----------|----------|
| 回调在解释器已销毁后仍被调用 | 崩溃或未定义行为 | 确保所有 GPU 内存已在解释器关闭前被释放；可在 `Py_AtExit` 注册清理回调（若需要） |
| 多次调用 `py_init_module`（误用）导致回调未更新 | 使用旧的回调实现，功能不符合预期 | 在 `py_init_module` 中加入检测并在调试构建中抛出警告；或在文档中强调“一次性调用”。 |
| 编译环境未开启 `Py_GIL_DISABLED`，但运行时使用了 free‑threaded CPython | 模块仍被标记为使用 GIL，可能导致运行时 GIL 检查失败 | 在构建脚本中检测 `Py_GIL_DISABLED` 宏并确保对应的 CPython 版本兼容；或提供 fallback。 |
| 单例的 `static const Callbacks instance` 持有对 Python 对象的引用，导致解释器停机时延迟释放 | 内存泄漏（可接受的进程结束前保持） | 这在当前设计下是预期行为，泄漏量极小；若要彻底释放，可在 `Py_AtExit` 手动 `Py_XDECREF`。 |

**💡 关注建议**  

1. **文档与使用约定**  
   - 明确说明 `_ensure_callbacks_initialized` 必须在任何分配路径触发前调用一次，且后续调用不会重新绑定。  
   - 在 `README` 中标注该扩展在 CPython 3.13+ 的 free‑threaded 环境下已得支持，推荐在多线程 GPU 计算场景下使用。  

2. **测试覆盖**  
   - 添加 CI 测试：在开启 `Py_GIL_DISABLED` 的 CPython 构建下，使用多线程并发调用 `my_malloc`/`my_free`，验证不会触发 GIL 检查或出现数据竞争。  
   - 增加单元测试：在解释器即将退出前再次调用 `my_malloc`/`my_free`，确认不会因已析构的对象导致段错误。  

3. **异常安全**  
   - 在 `my_malloc` / `my_free` 中对 `PyObject_CallObject` 返回值做更细致的错误检查（如 `if (!result) PyErr_Print();`），防止因 Python 回调异常导致 C++ 端忽略错误。  

4. **兼容性检查**  
   - 在 `setup.cfg` 或 `Cargo.toml` 中加入对 `Py_GIL_DISABLED` 宏的检测逻辑，确保在不支持的 Python 发行版上仍能成功编译（保持原有行为）。  

5. **性能基准**  
   - 虽然预计无锁读取开销极低，仍建议在高频分配路径上做一次微基准（如 10 M 次调用）对比旧实现，确保没有意外的回归。  

通过上述改动，`_allocator_ext` 已为即将到来的 free‑threaded CPython 生态做好准备，提升多线程 GPU 计算的可伸缩性，同时保持向后兼容和低额外开销。务必在文档、测试与 CI 中同步更新，以防潜在的单例绑定误用和解释器关闭时的资源泄露。

| **架构影响** | • `match_blocks` 从 **N 次 acquire‑for‑hash**（每次都要 `lock()`）改为 **一次锁批处理**，使活跃/非活跃前缀匹配的控制流更紧凑。<br>• `LineageBackend` 由原来的 `HashMap<…, BTreeMap>` 替换为 **单一 slab + 哈希索引 + 可插拔 LeafPolicy**，显著降低了堆分配与碎片化风险。<br>• 新增 `LeafPolicy` 抽象，使后端能够在 **O(1)** FIFO 与 **O(log n)** Tick 两种策略间自由切换，提升可配置性。 |
| **性能影响** | • **锁竞争减半**：`match_blocks` 只锁一次，频繁调用的 KV‑cache 热路径从 34% → ~20%（基准显示约 30% 左右的总体耗时下降）。<br>• **减少堆分配**：`LineageBackend` 的 slab 与 pre‑sized `SeqHashMap` 在常规链式插入情况下不产生额外分配，提升吞吐并降低 GC/分配压力。<br>• **哈希成本削减**：`IdHasher` 只做一次 128→64 位折叠，取代 SipHash，提升 `active_by_hash`、`LruBackend`、`MultiLruBackend` 的查找速度（约 2–3× 加速）。 |
| **安全考虑** | • `SequenceHash` 已经是强随机的 128 bit 内容哈希，使用 **身份哈希** 不会降低碰撞强度；相反避免了 SipHash 的 **Hash‑DoS** 攻击面，因为攻击者无法控制哈希值。<br>• 新的 `LineageBackend` 逻辑保持 `Arc<…>`、`Mutex` 保护，未引入 `unsafe`，符合 Rust 的安全模型。<br>• 所有公开 API（`with_lineage_backend_eviction`）仍保持向后兼容，默认仍是历史 `Tick` 行为。 |
| **可维护性** | • 代码分层更加清晰：`LeafPolicy` 抽象、`LineageBackend` 自包含；未来可在 `LeafPolicy` 中加入 **频率分层**（TinyLFU）等新策略而无需改动核心。<br>• 引入的基准 (`criterion`) 与 **大量回归测试**（包括前缀匹配、频率触碰、错误路径）保证后续改动的安全性。<br>• `IdHasher` 的实现简单且自解释，降低了后端 map 的认知负担。 |
| **兼容性** | • 通过 `builder.with_lineage_backend()` 默认仍使用 `Lineage` + `Tick`，对现有用户透明。<br>• 新增 `with_lineage_backend_eviction` 为可选 API，老代码不受影响。<br>• `match_blocks` 的返回值、语义保持不变（已通过全套单元测试验证），外部调用无需修改。 |

**⚠️ 潜在风险**  
1. **行为回归**：单锁实现在极端并发下可能出现 **先后顺序不同** 的活跃/非活跃竞争（原实现每次都立即触发频率计数），虽然已在测试中覆盖，但在真实多线程工作负载中仍建议进行 **压力/混沌测试**。  
2. **LeafPolicy 错误使用**：如果用户自行指定 `Fifo` 并期望与 `Tick` 完全相同的淘汰顺序，可能导致 **缓存命中率下降**。文档需明确两者差异及使用建议。  
3. **索引哈希冲突**：`IdHasher` 将 128 bit 折叠为 64 bit，理论上仍存在碰撞概率（极低）。在极大规模（>2⁶⁴）块的场景下，若出现冲突会导致 **活跃缓存误命中**。可通过在 `SeqHashMap` 创建时加上 `HashMap::with_hasher` 的 `RandomState` 作为后备或在关键路径加入完整相等检查来降低风险。  
4. **内存泄漏/碎片**：`LineageBackend` 使用手动 `free_head` 管理 slab，若在异常路径（panic）前未正确回收，可能导致 **槽位泄漏**。目前所有错误路径均 `panic!`，但在未来加入 `Result` 时需审查 `free_slot` 的调用完整性。  

**💡 关注建议**  

| 针对对象 | 建议 |
|----------|------|
| **开发者** | • 在高并发环境下运行 **stress / loom** 测试，确保 `match_prefix_locked_batch` 的一次锁实现不导致活跃/非活跃状态错乱。<br>• 为 `LeafPolicy::Fifo` 在文档中添加 “适用于对延迟不敏感、希望避免 BTree 分配的场景” 的使用说明。<br>• 将 `IdHasher` 的 `write` 明确标记为 `#[cold] unreachable!`，防止未来对 `SequenceHash` 增加自定义 `Hash` 实现时遗忘更新。 |
| **用户** | • 若对缓存淘汰顺序有严格需求（如与旧版行为完全一致），保持默认 `Tick`，否则可通过 `with_lineage_backend_eviction(LineageEviction::Fifo)` 获得 **O(1)** 的链表淘汰。<br>• 监控 `match_blocks` 的 **match_hashes_requested** 与 **match_blocks_returned** 指标，确认新实现未引入异常的前缀截断。 |

  - **数据流**：调用链从 `DynamoRouter` → `KvRouter.get_overlap_scores` → `Indexer.find_matches_by_tier` + （可选）`SharedCache.check_blocks`。不再依赖 `best_worker_id` 的路由决策，只返回原始匹配分布。

  - **计算开销**：与 `best_worker_id` 基本相同——需要计算块哈希、查询 KV 索引、可选的共享缓存检查。额外的遍历和聚合步骤（构造 `workers` 集合、统计扩展块）在数据量（worker 数量）上呈线性增长。  
  - **响应体大小**：返回的 JSON 包含每个 worker‑dp‑rank 的多字段统计，若集群规模大（上百 worker）可能导致网络传输和序列化成本显著上升。建议在高并发场景下对该端点做速率限制或分页。

  - **信息泄露**：返回的层级块匹配细节（每个 worker 的 device/host‑pinned/disk 命中数）可被外部利用推断集群的 KV 分布与缓存布局，属于潜在的内部实现泄露。若部署在多租户环境或对安全有严格要求的场景，需要在网络层或服务层加入访问控制（OAuth、mTLS）或对返回数据进行脱敏。

1. **未初始化的 KvRouter**：调用 `get_overlap_scores` 前未创建 `kv_router` 会导致运行时错误。虽然已有显式检查并抛异常，但如果上层代码未捕获，可能导致服务进程崩溃。

3. **响应体膨胀**：大规模集群（数百 worker）下返回的 `workers` 列表可能触发序列化瓶颈或网络超时。  
4. **并发竞争**：高并发调用该端点可能与生成请求竞争 CPU/IO，导致生成延迟上升。

- **回退方案**：如果出现性能回退，可在配置中关闭 `include_shared` 或直接禁用该端点，以免影响主业务路径。  

通过以上措施，可在充分利用新接口提供的细粒度 KV 重叠信息的同时，降低对系统整体稳定性与安全性的潜在冲击。

  - Planner 通过 `_frontend_metric_name` 与 `_sum_frontend_metric` 两个辅助函数统一处理前缀与大小写，提升代码可读性并避免跨语言前缀硬编码。  
  - 新增的回退逻辑保证向后兼容，使得在旧版前端未暴露 `*_started_total` 时仍能正常工作。

- **性能影响**：  
  - 增加一个 `IntCounterVec`，每个请求多一次原子递增，开销极小（与已有 `request_counter` 同阶）。  
  - Planner 在查询时先做一次 `increase(requests_started_total[interval])`，若返回空再查询 `requests_total`，最多两次 Prometheus 查询，仍在毫秒级响应范围。  
  - 对 Prometheus 存储的额外时间序列量极低（仅增加一类计数），不会对服务端吞吐产生可感知影响。

- **安全考虑**：  
  - 仅是对外暴露的监控指标，未改变业务逻辑或数据访问路径，对安全属性不产生新风险。  
  - 需要确认指标标签（`model`、`endpoint`、`request_type`）的可见性符合组织的监控合规策略。

**⚠️ 潜在风险**  

1. **兼容性**：在滚动升级期间，部分前端实例可能仍未实现 `requests_started_total`，导致 Planner 在这些实例上只能使用已完成计数，可能出现短暂的负载估计偏差。  
2. **监控查询成本**：新增的 `increase(...started_total[interval])` 查询在高并发环境下会稍增 Prometheus 查询负载，需监控查询延迟。  
3. **指标拼写错误**：新增的前缀处理函数 `_frontend_metric_name` 必须保持与其它语言（Rust、Python）实现一致，防止出现 “dynamo_frontend_requests_started_total” 与 “frontend_requests_started_total” 不匹配的情况。  
4. **标签不匹配**：如果前端在递增 `request_started_counter` 时遗漏了某些标签（例如 `endpoint`），则对应的查询可能得不到匹配，导致回退到 `requests_total`，出现误判。

**💡 关注建议**  

- **滚动升级**：在部署新前端二进制时，先确保所有实例已成功导出 `requests_started_total`，或在升级窗口内监控 Planner 的查询日志，确认回退路径的触发频率趋近于 0。  
- **监控告警**：为 `increase(dynamo_frontend_requests_started_total[interval])` 添加查询延迟和返回为空的告警，以便快速定位前端指标缺失。  
- **指标验证**：在 CI 中保留对新指标的可见性检查（已在测试中新加入），并在生产部署后通过 Grafana/Prometheus UI 验证 `*_started_total` 随请求流入而递增。  
- **文档同步**：更新内部文档和外部用户手册，说明 “started” 指标的语义以及 Planner 使用该指标进行负载规划的原理，帮助运维团队理解监控数据的变化。  
- **日志追踪**：在 Planner 中加入调试日志，记录是使用 “started” 还是 “total” 计数，以便在异常负载情况下排查根因。
