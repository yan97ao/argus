### feat: add experimental DeepSeek-V4-Flash + V4-Pro vLLM agg recipes (#8668)
**SHA**: `9c07899` | 🔗 [查看提交](https://github.com/ai-dynamo/dynamo/commit/9c07899613fdc3b990654c469a9a5a62be047551)

**🎯 变更类型**：功能增强  
**⚡ 重要程度**：🔴高  

**📋 变更摘要**  
本次提交为 Dynamo 项目新增两套实验性 **DeepSeek‑V4‑Flash** 与 **DeepSeek‑V4‑Pro** 的 vLLM 聚合（agg）部署 Recipes。包括完整的 Kubernetes 资源清单（PVC、Job、DynamoGraphDeployment）、模型下载脚本、容器镜像 Dockerfile 与构建说明，以及对应的 README 文档。目标是让用户能够在单节点 4 × B200（Flash）或 8 × B200（Pro）GPU 上运行 284B/13B‑active 与 1.6T/49B‑active 的 MoE 模型，提供 FP8 KV 缓存、FP4 Indexer、CSA+HCA 混合注意力、推理/工具调用解析等特性。

**🎯 影响范围**  
- `recipes/` 目录：新增 `deepseek-v4-flash`、`deepseek-v4-pro` 两套完整的部署示例。  
- `recipes/README.md`：表格中加入两条新条目。  
- Docker 镜像层：新增 `container/Dockerfile.dsv4`（分别对应 Flash 与 Pro）以及对应的 README。  
- Kubernetes 资源：新增 `model-cache/*.yaml`、`model-download/*.yaml`、`vllm/agg/vllm-dgd.yaml`（两个模型各一份）。  
- 依赖：引入了 `libibverbs*`, `rdma-core`, `ucx`, `nixl` 等运行时库；对上游 vLLM `sparse_attn_indexer.py` 做了临时补丁。  
- 文档：两个模型的使用手册、构建指南、验证步骤等。

---

**🔍 技术洞察**  

| 维度 | 影响分析 |
|------|----------|
| **架构影响** | 1. **容器叠加层**：通过多阶段 `Dockerfile` 将 Dynamo 运行时（wheels、nats、etcd、UCX、NIXL）叠加到官方 DeepSeek‑V4 vLLM 基础镜像上，实现 **零重装 vLLM**，保持官方功能（CSA、HCA、MTP、FP4 Indexer）不变。<br>2. **DynamoGraphDeployment**：新增两套聚合（agg）部署规格，分别采用 **DP+Expert Parallel**（Flash）和 **TP+Expert Parallel**（Pro）两种并行策略，符合模型大小与 GPU 资源匹配的最佳实践。<br>3. **模型缓存**：使用 RWX PVC（400 Gi / 1500 Gi）共享给 Frontend 与 DecodeWorker，配合 `HF_HUB_OFFLINE=1` 实现完全离线加载，降低对外网的依赖。 |
| **性能影响** | 1. **FP8 KV 缓存 + 256‑Block**：显著降低显存占用（约 1/4），使 1 M 长上下文模型得以在单节点运行。<br>2. **FP4 Indexer Cache**（`--attention-config '{"use_fp4_indexer_cache":true}'`）配合 CSA+HCA，提升稀疏注意力的吞吐并降低算力开销。<br>3. **FlashInfer + cudagraph**：首次启动会进行权重加载、FlashInfer autotune、CUDA‑graph 编译，导致 **启动延迟**（Flash≈60 min，Pro≈90 min），但后续推理延迟与吞吐将大幅提升。<br>4. **并行策略**：Flash 使用 **DP=4 + Expert Parallel**（更好地利用显存碎片），Pro 使用 **TP=8 + Expert Parallel**（模型太大无法 DP），两者在不同硬件配置下的性能表现会有显著差异。 |

**⚠️ 潜在风险**  

1. **镜像兼容性**  
   - 基础 vLLM 镜像仅支持 Python 3.12 与 CUDA 13.0（或 12.9），若集群节点 CUDA 版本不匹配，容器将启动失败。  
   - 通过 `sed` 修改上游源码的做法在基础镜像升级后可能出现路径或 API 变更，引发运行时错误。  

2. **存储要求**  
   - Flash 需要 **≥400 Gi**，Pro 需要 **≥1500 Gi** 的 RWX PVC；若未提供足够容量或适配的 StorageClass（尤其是跨节点共享），Pod 将卡在 `Pending`。  

3. **启动时长**  
   - 长达 60‑90 分钟的首轮加载会导致 **probe 超时** 或 **资源占用**（sharedMemory 200 Gi）。业务方必须提前预留时间，否则会误判为部署失败。  

4. **资源配置错误**  
   - `gpu: "4"`/`"8"` 必须匹配实际节点的 GPU 数量，若调度到 GPU 数量不足的节点，Deployment 将不可达。  

5. **安全凭证泄露**  
   - HF Token 通过 `Secret` 注入至 Job 与 Worker，若 Pod 以 `root` 权限运行且未设置 `runAsNonRoot`，攻击者可能通过容器逃逸获取凭据。  

6. **依赖的 NIXL / UCX 版本**  
   - NIXL 与 UCX 通过 `COPY --from=` 引入，版本锁定在本地 Dynamo runtime 镜像。如果 Dynamo runtime 升级但未同步更新 overlay 镜像，可能出现二进制不兼容或 ABI 错误。

### fix(sglang): canary probe for disagg prefill + consolidate HEALTH_CHECK_KEY (#8611)
**SHA**: `c692ce6` | 🔗 [查看提交](https://github.com/ai-dynamo/dynamo/commit/c692ce60936a3ab89f4547a715bc43a5ec484cc5)

**🎯 变更类型**：Bug修复 / 功能增强（新增 Canary 探活标记并统一 `HEALTH_CHECK_KEY`）  
**⚡ 重要程度**：🔴 高  
**📋 变更摘要**：本次提交在 `dynamo.health_check` 中统一了健康检查标记键 `HEALTH_CHECK_KEY`，并在 Sglang、TRT‑LLM、VLLM 三大后端的健康检查 payload 中实现 `to_dict` 覆写，将该标记层叠进去。随后在 Sglang 的 prefill handler 中加入对标记的检测与专门的 Canary 路径，以避免发送首次‑yield 的合成信息导致探活失败。对应测试也同步补全，确保标记在环境变量覆盖的情况下仍然保留。  

**🎯 影响范围**：  
- `components/src/dynamo/sglang/*`（health_check、prefill_handler、tests）  
- `components/src/dynamo/trtllm/*`（health_check、handler_base、tests）  
- `components/src/dynamo/vllm/*`（health_check、tests）  
- `lib/bindings/python/src/dynamo/health_check.py`（统一常量）  
- 相关单元测试与故障容错测试 (`tests/fault_tolerance/test_canary_rank_pause.py`)  

**🔍 技术洞察**：  
- **架构影响**：  
  - **统一标记键**：将原先散落在各后端的健康检查键抽取到公共模块 `dynamo.health_check`，实现了跨后端的协议层一致性，降低了后续新增后端时遗漏标记的风险。  
  - **Canary 探活路径**：在 Sglang 的 prefill handler 中加入 `if inner_request.get(HEALTH_CHECK_KEY)` 分支，使探活请求能够直接把生成的结果流回 Rust 调度层，而不走常规的 `bootstrap_info` 逻辑，避免了探活期间因 “synthetic first‑yield” 被遮蔽导致的误判。  
  - **payload 兼容层**：所有后端的 `HealthCheckPayload` 通过 `to_dict` 重新包装，保证在 `DYN_HEALTH_CHECK_PAYLOAD` 环境变量覆盖默认 payload 时，标记仍然被保留，实现了“环境变量覆盖 + 标记不丢失”的设计目标。  

- **性能影响**：  
  - 仅在健康检查路径（极少数探活请求）会多一次字典拷贝与键插入，成本可忽略不计。  
  - 对正常推理路径没有任何额外开销，因为标记键只在探活请求中被读取。  

- **安全考虑**：  
  - 标记键为内部使用的约定字符串 `"_HEALTH_CHECK"`，不影响外部业务数据。  
  - 该键的加入不会泄露敏感信息，也不改变现有的权限模型。唯一需要关注的是，若外部系统误将该键写入业务 payload，可能导致后端误判为探活请求；建议在业务层对输入做白名单过滤。  

**⚠️ 潜在风险**：  
1. **兼容性**：外部自定义的健康检查 payload（或直接使用 `HealthCheckPayload.from_env`）若在覆盖后未调用 `to_dict`，可能会丢失标记，导致探活路径失效。  
2. **键名硬编码**：`HEALTH_CHECK_KEY` 仍是字符串常量，历史代码若仍硬编码 `"_HEALTH_CHECK"`（如某些未同步的插件），会出现不匹配的情况。  
3. **测试覆盖不足**：目前只有单元测试验证了标记在三大后端的存在，未覆盖实际探活请求在分布式调度层的完整链路，生产环境中仍可能出现未预料的分支遗漏。  

**💡 关注建议**：  
- **代码审查**：确认所有自定义或第三方实现的 `HealthCheckPayload` 在使用前均走统一的 `to_dict` 接口，避免手动构造字典导致标记缺失。  
- **部署验证**：在上线前使用真实的 Canary 探活脚本跑一次完整的分布式推理流程，确保 Rust 调度层可以收到带有 `_HEALTH_CHECK` 标记的消息并正常返回。  
- **监控告警**：在探活失败的监控中加入对 `HEALTH_CHECK_KEY` 是否被正确转发的校验，一旦探活请求未返回预期的标记，即触发告警。  
- **文档同步**：更新开发者文档，明确 `HEALTH_CHECK_KEY` 为公共常量，所有后端健康检查 payload 必须通过 `to_dict` 方法自动层叠该标记。  

通过上述改动，系统的探活可靠性得到显著提升，且保持了良好的可维护性和扩展性。

### fix: Delay vLLM Request Cancellation until KV transfer is complete (#8624)
**SHA**: `1eb856c` | 🔗 [查看提交](https://github.com/ai-dynamo/dynamo/commit/1eb856c35f624d448b2ac148305cbdee7458a95a)

**🎯 变更类型**：Bug修复  
**⚡ 重要程度**：🔴高  
**📋 变更摘要**：在 vLLM 的 disaggregated‑decode 模式下，取消请求（`abort`）可能在 KV 转移仍在进行时被执行，导致 EngineCore 崩溃。本次提交引入 `_DeferredAbort` 与 `_deferred_abort_guard`，将实际的 `engine_client.abort` 延迟到收到首个 token 之后执行，并在生成循环结束时安全地清理可能残留的后台任务。相应的生成路径与 abort 监控逻辑完成改造，同时补充了大量单元测试以验证行为。

**🎯 影响范围**：  
- `components/src/dynamo/vllm/handlers.py`（核心请求处理、abort 监控、token 生成流程）  
- `components/src/dynamo/vllm/tests/test_vllm_worker_handler.py`（新增单元测试）  
- `tests/fault_tolerance/cancellation/test_vllm.py`（调整 skip 标记）  

**🔍 技术洞察**  
- **架构影响**：  
  - 引入了 “延迟 abort” 的防护层，仅在 **decode‑only**（disaggregated decode）模式下激活，保持了现有的同步 abort 语义。  
  - 通过 `asynccontextmanager` 把 guard 生命周期与请求绑定，保证在请求结束（无论是否生成 token）时一定会调用 `guard.close()`，避免孤儿任务。  
  - 修改了 `_monitor_abort` 与 `_abort_monitor` 的签名，传递 `abort_guard`，对现有调用链影响最小（仅在 decode‑only 场景传参）。  

- **性能影响**：  
  - 新增的 `asyncio.Event`、单次 `asyncio.create_task` 与一次额外的协程切换（等待首 token）带来的开销极低（微秒级），对吞吐基本没有可感知影响。  
  - 在**不需要延迟 abort**的普通（prefill/agg‑decode）路径中，仍然走原始 `engine_client.abort`，不受额外成本影响。  

- **安全考虑**：  
  - 变更本质是 **避免在 KV 传输期间调用 abort**，防止 EngineCore 崩溃，是一次 **可靠性/稳定性** 修复，未引入外部输入依赖或权限提升风险。  
  - 通过 `close()` 中的 `await self._abort_task` 处理，确保即使后台任务因异常结束也会记录警告，不会泄露异常信息给调用方。  

**⚠️ 潜在风险**  
1. **首 token 永不到达**：如果生成任务卡在 KV 传输或其他内部错误，`abort_guard` 会在 `close()` 时 **取消** 待执行的 abort，而不会真正取消 EngineCore，导致该请求在 Engine 内部可能仍占用资源。  
2. **并发使用 guard**：当前实现假设每个请求仅创建一个 `_DeferredAbort` 实例。若后续在同一请求上下文中误创建多个 guard，可能产生竞争（`_first_token_event` 被多次 set），虽然不会破坏功能，但会产生多余的后台任务。  
3. **异常传播**：`_wait_and_abort` 中捕获所有异常并仅记录 warning，若 `engine_client.abort` 抛出关键错误（如网络不可达），上层可能不感知，可能导致请求状态不一致。  

**💡 关注建议**  
- **监控 & 超时**：在 guard 的 `close()` 前可加入可选的 “最大等待时间” 参数，超过阈值则强制调用 `engine_client.abort`，防止资源悬挂。  
- **文档说明**：在项目文档或代码注释中明确 “仅在 disaggregated decode 模式下使用” 的前提，防止误在其他模式开启 guard。  
- **测试覆盖**：已有丰富的单元测试，建议在 CI 中加入 **压力/超时** 场景，验证在长时间无 token 输出的情况下系统能正确回收。  
- **日志细化**：目前日志使用 `debug`/`warning`，可在生产环境通过配置打开 `info` 级别，帮助快速定位因延迟 abort 引起的异常行为。  
- **复审调用方**：确认所有调用 `_monitor_abort` 的地方（如未来新增的生成路径或其他模型后端）已同步传递 `abort_guard` 参数，避免遗漏导致旧的 “立即 abort” 行为在 decode‑only 场景仍然触发。

  - 引入 **进程隔离**：原先的单进程服务器在 pytest 主进程中持有 CUDA 上下文，导致显存被“黏在” pytest PID 上；现在通过 `subprocess.Popen` 启动独立进程，PID 生命周期与测试模块绑定，进程结束即自动释放所有 CUDA 资源（上下文、fd、驱动分配）。  
  - 增加 **socket 管理**：在子进程启动前会检查并清理残留的 Unix domain socket，避免“僵尸 socket”导致端口冲突。  
  - 引入 **上下文管理器** (`GMSServer.__enter__/__exit__`) 把启动/停止逻辑封装，提升可读性并统一异常处理。  

- **性能影响**：  
  - 启动子进程以及 RPC 服务器的初始化会带来 **数十毫秒的额外开销**（进程创建、CUDA 上下文创建），在单个测试模块的启动成本上可感知，但相对于避免 2‑3 GiB 的显存泄漏造成的 OOM/重跑成本，属于正向 trade‑off。  
  - 子进程在结束时立即释放显存，整体测试套运行时间可能会略有下降，因为后续测试不再因显存不足被迫跳过或重试。  

- **安全考虑**：  
  - 仅在测试环境中使用子进程，没有向外部暴露网络端口（使用 Unix domain socket），安全风险极低。  
  - 调用了 `sys.executable -m tests.gpu_memory_service.common._gms_server_runner`，确保只执行项目内部受信代码。  
  - 依赖 `nvidia-smi` 进行显存查询，若在生产环境误打开该 fixture，可能泄露 GPU 使用情况，但该 fixture 标记为 `autouse` 且仅在测试目录下加载，不会进入生产二进制。  

**⚠️ 潜在风险**：  
1. **子进程启动失败或超时**：若 `GMSRPCServer` 初始化慢于 `_SERVER_START_TIMEOUT_SECONDS`（默认几秒），`GMSServer.__enter__` 会抛 `TimeoutError`，导致整个测试模块报错。  
2. **残留进程**：异常路径或 CI 被强制中断时，子进程可能未被 `terminate/kill`，导致显存仍被占用。  
3. **跨平台兼容性**：当前实现使用 Unix domain socket 与 `os.unlink`，在 Windows 环境下可能需要改为命名管道或 `socket` 文件路径处理。  
4. **依赖 `nvidia-smi`**：若测试机器上缺少该二进制或权限不足，fixture 会 silently 跳过检查，可能掩盖泄漏。  
5. **资源竞态**：同一 socket 路径的并发测试（比如 `pytest-xdist` 并行运行）可能出现竞争，需确保每个 worker 使用独立 `device/tag` 参数。  

**💡 关注建议**：  
- **CI 监控**：在 CI 中显式检查子进程的退出码，若返回非 0，打印 `stderr` 以帮助定位服务器启动错误。  
- **可配置超时**：将 `_SERVER_START_TIMEOUT_SECONDS`、`_SERVER_STOP_TIMEOUT_SECONDS` 与 `LEAK_THRESHOLD_MIB` 暴露为环境变量或 pytest‑ini 配置，便于在资源紧张的机器上调节。

  - 变更没有引入新的网络入口或数据处理路径，仅是调度方式的调整。  
  - `health_check_payload` 只携带简单的 `{ "text": "health" }`，不泄露敏感信息，且与 `HealthCheckRequest` 结构保持一致。  
  - 需要确保 `health_check` handler 本身不执行除状态返回外的业务逻辑，以免成为潜在的攻击面。  

**⚠️ 潜在风险**  

1. **异常传播**：`asyncio.gather` 默认在任一子任务抛异常时立即传播，可能导致两端点都被关闭。若 `scale_request` 启动失败，`health` 也会随之失效，导致 pod 仍旧不可用。需在生产代码中考虑使用 `return_exceptions=True` 或捕获并记录异常后继续运行。  
2. **Future 取消行为**：`serve_endpoint` 返回的 Future 在测试中被模拟为永不完成的对象。实际运行时如果 operator 发起 shutdown，必须确保 `Runtime` 能正确取消这些 Futures，否则可能出现资源泄漏。  
3. **依赖实现细节**：`health_check_payload` 必须严格匹配 `HealthCheckRequest` 的序列化格式；若未来 `HealthCheckRequest` 结构变化，当前硬编码的字典可能导致反序列化错误，影响系统健康切换。  

**💡 关注建议**  

- **异常容错**：在 `asyncio.gather` 周围加入显式的异常捕获日志，确保单一 endpoint 启动失败不会导致整个服务崩溃，或在必要时实现局部回退（如仅记录错误、继续运行健康检查）。  
- **优雅停机**：确认 `Runtime` 在接收到 SIGTERM 等信号时能正确取消 `serve_endpoint` 返回的 Futures，避免 pod 停止时出现 hanging 进程。  
- **健康检查 payload 统一**：抽象出 `DEFAULT_HEALTH_PAYLOAD = {"text": "health"}` 常量，供所有 planner 使用，防止未来结构不一致导致的回归。  
- **持续回归**：将新增的单元测试加入 CI 的长期回归套件，并在 `cargo test` / `pytest` 阶段确保覆盖率。若后续有其它 planner 引入类似注册逻辑，复用该测试模板。
