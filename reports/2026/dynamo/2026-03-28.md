### feat: Support a dynamic default max_tokens for TensorRT-LLM backend (#5152)
**SHA**: `98d0ce9` | 🔗 [查看提交](https://github.com/ai-dynamo/dynamo/commit/98d0ce91b8c7ff0fceefe705a0b308ab158528af)

**🎯 变更类型**：功能增强  
**⚡ 重要程度**：🔴高  

**📋 变更摘要**  
1. 在 `RequestHandlerConfig` 中新增 `max_seq_len` 参数，用于在请求未显式指定 `max_tokens` 时提供动态默认值。  
2. `HandlerBase` 将该配置保存为实例属性，并在 `generate_locally` 流程中根据输入长度计算 `sampling_params.max_tokens`（`max_seq_len - input_length`），多模态请求则保持原有行为。  
3. `llm_worker` 在初始化时把 `max_seq_len` 透传至 `HandlerBase`。  

**🎯 影响范围**  
- `components/src/dynamo/trtllm/request_handlers/handler_base.py`  
- `components/src/dynamo/trtllm/workers/llm_worker.py`  
- `components/src/dynamo/trtllm/request_handlers/config`（新增字段）  

**🔍 技术洞察**  
- **架构影响**：  
  - 引入了一个全局的 “最大序列长度” 配置，属于 **运行时配置层** 的扩展。  
  - 通过 `HandlerBase` 向下游的 `generate_locally` 逻辑注入默认 `max_tokens`，保持了原有抽象（`RequestHandlerConfig → HandlerBase → 生成流`）不变，仅在中间层加入了计算步骤。  
  - 对多模态处理保持向后兼容，避免在视觉/音频等非文本输入上误用动态默认值。  

- **性能影响**：  
  - 计算 `dynamic_default = max(1, self.max_seq_len - input_length)` 的开销极低（常数时间），对整体吞吐几乎无感。  
  - 动态限制 `max_tokens` 能防止模型在序列过长时无限生成，从而 **降低显存/算力占用**，在高并发场景可能提升整体吞吐。  
  - 若 `max_seq_len` 设置得过低，可能导致频繁截断，增大重试/补全次数，对性能产生轻微负面影响。  

- **安全考虑**：  
  - 通过默认上限防止恶意请求指定极大 `max_tokens`（潜在 DoS），提升系统的 **资源滥用防护**。  
  - 该字段本身不涉及权限或注入风险，但若配置文件被未授权修改，可能导致生成被意外截断；建议把 `max_seq_len` 纳入配置审计流程。  

**⚠️ 潜在风险**  
1. **兼容性**：  
   - 旧版配置文件若未包含 `max_seq_len`，行为保持不变；但在新版中若误将 `max_seq_len` 设为 `0` 或负数，会导致 `dynamic_default` 为 1（虽然代码已 `max(1, ...)`），但可能不符合业务预期。  
2. **多模态请求漏掉默认**：  
   - 代码在检测到 `self.multimodal_processor` 且 `processed_input` 不为 `None` 时跳过动态默认，这一判断若未来改变（如新增多模态前置处理），需要同步更新此处逻辑。  
3. **配置传播**：  
   - `max_seq_len` 必须在所有创建 `HandlerBase` 的路径（除 TRT‑LLM 之外的后端）同步添加，否则不同后端的行为会出现不一致。  
4. **测试覆盖不足**：  
   - 需要验证：  
     - `max_seq_len` 为 `None` 时保持原有行为。  
     - 输入长度接近/超过 `max_seq_len` 时的边界值（如 `input_length == max_seq_len`）。  
     - 多模态请求不受动态默认影响。  

**💡 关注建议**  
- **文档**：在配置文档中明确 `max_seq_len` 的含义、推荐取值范围以及与 `max_tokens` 的关系。  
- **验证**：在启动时对 `max_seq_len` 做基本校验（>0），并在日志中提示实际使用的动态默认值。  
- **单元/集成测试**：添加覆盖以下场景的测试用例：  
  1. `max_tokens` 显式提供 → 覆盖 `max_seq_len`。  
  2. `max_tokens` 为 `None`，`max_seq_len` 有效 → 检查 `sampling_params.max_tokens` 正确计算。  
  3. 多模态请求 → 确认仍使用 `None`（不自动截断）。  
- **监控**：在运行时监控 `max_tokens` 的实际取值分布，确保不会出现异常截断或意外的大值请求。  
- **后端一致性**：如果项目计划在未来支持其他后端（如 PyTorch、vLLM），请同步实现相同的 `max_seq_len` 逻辑，避免行为碎片化。
