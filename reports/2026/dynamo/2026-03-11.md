### fix: removing reference to async-ucx (#7213)
**SHA**: `ddf5035` | 🔗 [查看提交](https://github.com/ai-dynamo/dynamo/commit/ddf503596c1c7913c5382c6731abb93eeb42c07b)

**🎯 变更类型**：Bug修复  
**⚡ 重要程度**：🔴高  
**📋 变更摘要**：本次提交移除了对 `async-ucx` 依赖的所有引用，包括 Cargo.toml 中的可选 feature、README 中的功能说明以及测试代码中的 UCX 相关实现。相应地，`Cargo.lock` 经过大量删减，清理了已不再使用的 crate（如 `async-ucx`、`ucx1-sys`、旧版 `bindgen`、`futures-lite` 等），并对若干依赖版本进行了升级。目标是解决因 `async-ucx` 仓库不可用或构建冲突导致的编译失败。

**🎯 影响范围**：  
- `lib/velo-transports`（核心传输层）  
- `Cargo.toml`、`Cargo.lock`（整个工作区的依赖树）  
- `README.md` 中的传输特性表格  
- `tests/common/mod.rs` 中的 UCX 测试工具代码  

**🔍 技术洞察**：  
- **架构影响**：去除 UCX/RDMA 传输后，`velo-transports` 仅保留 TCP、HTTP、NATS、gRPC 四大传输实现，架构更加简洁。对外部使用者而言，原本依赖 `ucx` feature 的代码将直接编译错误，需要改用其他传输或自行实现 UCX 支持。  
- **性能影响**：删除大量可选依赖（尤其是 `async-ucx` 及其 `bindgen`、`socket2` 等旧版）显著降低了 crate 编译时间和最终二进制体积；运行时性能不受影响，因为 UCX 功能本身已不再提供。  
- **安全考虑**：去除 `async-ucx`、`bindgen` 旧版等外部代码，缩小了攻击面，降低了因未审计的 native 依赖（如 UCX 库）导致的潜在漏洞风险。  

**⚠️ 潜在风险**：  
1. **下游兼容性**：依赖 `ucx` feature 的内部或第三方项目在升级后会出现编译错误。  
2. **文档/示例同步**：如果还有未更新的示例或 CI 脚本仍启用 `ucx`，会导致构建失败。  
3. **功能缺失**：对需要高性能 RDMA 通信的用户，失去官方提供的 UCX 实现，可能需要自行维护或等待新的 `nixl` 实现。  

**💡 关注建议**：  
- 在发布日志中明确标记 `ucx` feature 已被移除，并提供迁移指引（如改用 TCP/HTTP 或自行实现 UCX）。  
- 检查 CI/CD、示例仓库以及任何依赖 `velo-transports` 的子项目，确保未再激活 `ucx`。  
- 若计划在未来以 `nixl-sys` 替代 UCX，建议提前创建 `nixl` feature 分支并在文档中注明“实验性/待实现”，避免用户误以为已可用。  
- 考虑在 `Cargo.toml` 中加入 `ucx = []` 的废弃警告（如使用 `#[deprecated]` 在代码层面提示），帮助使用者平滑过渡。

**📋 变更摘要**：  
- 解决了 TRT‑LLM 与 Dynamo Rust 传输层之间的键类型不匹配问题：`first_gen_log_probs` 中使用整数 `token_id` 作为字典键，导致 pythonize/serde_json 在反序列化时抛出 “dict_key_not_string”。  
- 新增 `DisaggregatedParamsCodec.serialize_first_gen_log_probs` 与 `deserialize_first_gen_log_probs`，在网络往返前后完成整数‑> JSON‑安全结构（list‑of‑dict）以及逆向恢复。  
- 在 `handler_base.py` 中分别调用序列化/反序列化，确保前端/后端协同工作，并补充完整单元测试验证 round‑trip 行为。

**🎯 影响范围**：  
- `components/src/dynamo/trtllm/request_handlers/handler_base.py`（prefill‑→‑decode 参数解码、prefill 参数编码）  
- `components/src/dynamo/trtllm/utils/disagg_utils.py`（新增序列化/反序列化实现）  
- 新增 `tests/serve/test_disagg_logprobs_serialization.py`（覆盖多种 edge case）  
- 依赖：`tensorrt_llm.executor.result.Logprob`、`tensorrt_llm.llmapi.DisaggregatedParams`

**🔍 技术洞察**  
- **架构影响**：  
  - 在 Python→Rust 传输链路中插入了 “codec layer”，保持原有 TRT‑LLM 内部数据结构不变，仅在网络边界做一次轻量转换。  
  - 该层解耦了业务逻辑（Logprob 计算）与传输格式，提升了跨语言兼容性，且不影响已有的 `DisaggregatedParams` 编码路径。  
- **性能影响**：  
  - 序列化/反序列化均为 O(N) 线性遍历，N 为 `first_gen_log_probs` 中 token 数量，通常在数百级别，对整体预填/解码延迟影响可忽略 (< 0.5 ms)。  
  - 只在需要携带 `first_gen_log_probs` 时触发，未改变其余字段的处理路径。  
- **安全考虑**：  
  - 只对受信任的内部结构进行转换，无外部输入直接写入；使用标准 Python 数据结构并通过 `json.dumps` 验证 JSON‑安全性，避免了潜在的注入或序列化漏洞。  
  - 未引入新的外部依赖或可执行代码，风险极低。  

**⚠️ 潜在风险**  
1. **遗漏调用**：若未来在其他入口（如自定义插件）未显式调用 `serialize_first_gen_log_probs`/`deserialize_first_gen_log_probs`，仍会触发 Rust 端的 “dict_key_not_string” 错误。  
2. **向后兼容**：老版本的 Rust 客户端若未更新对应的解码逻辑，将收到已被序列化的列表结构，导致业务逻辑误读。需要确保协议版本同步。  
3. **字段结构变化**：`Logprob` dataclass 若在未来添加字段，需要同步更新序列化/反序列化实现（目前仅处理 `logprob` 与 `rank`）。  

**💡 关注建议**  
- **文档 & 协议**：在项目的 OpenAPI/协议文档中标注 `first_gen_log_probs` 的传输格式（list‑of‑dict），并说明兼容层的存在。  
- **统一入口**：考虑将序列化/反序列化调用封装到 `DisaggregatedParamsCodec.encode` / `decode` 中，避免在业务代码里手动调用，降低遗漏风险。  
- **回归测试**：将新增的单元测试纳入 CI 全流程，并在多节点/多 GPU 环境下执行一次集成测试，确保跨进程/跨机器的实际网络序列化不出现异常。  
- **监控报警**：在 Rust 前端捕获 `serde_json::Error` 时上报告警，快速定位若有协议不匹配回退的情况。  
- **兼容性迁移**：如果需要支持旧版 Rust 客户端，可在 Python 端根据客户端版本决定是否进行序列化，以实现平滑升级。

### fix: Fix mocker+router segfaults + reliability fixes (#7162)
**SHA**: `761e848` | 🔗 [查看提交](https://github.com/ai-dynamo/dynamo/commit/761e848b5b388b8fcd75ff72ae0c17d807a20b41)

**🎯 变更类型**：Bug修复 / 可靠性提升  
**⚡ 重要程度**：🔴 高  
**📋 变更摘要**：  
1. 在 `EventPublisher` 中引入 `runtime_handle`，在 `Drop` 时使用创建该对象时的 Tokio 运行时进行异步注销，避免在没有 Tokio 上下文（如 PyO3 finalizer）中调用 `tokio::spawn` 导致段错误。  
2. 为 `ManagedProcess` 添加更细粒度的停止/强制停止逻辑及专用异常类，提升子进程清理的可靠性，防止僵尸进程和资源泄漏。  
3. 更新测试与 Python 项目配置以适配新实现并消除 Python‑3.12 的 SWIG 警告。

---

### 🎯 影响范围
- **核心运行时**：`lib/runtime/src/transports/event_plane/mod.rs`（EventPublisher 生命周期管理）  
- **测试框架**：`tests/utils/managed_process.rs`（进程管理工具）以及若干 E2E 测试文件  
- **Python 包**：`pyproject.toml`（过滤 Python 3.12 SWIG 警告）  

---

### 🔍 技术洞察

| 维度 | 影响 |
|------|------|
| **架构影响** | - 引入 `runtime_handle: tokio::runtime::Handle` 作为 `EventPublisher` 的成员，使其不再依赖全局或 “ambient” Tokio 运行时；从而在跨语言（Python → Rust）边界的 `Drop` 时能够安全地调度异步注销任务。<br>- `ManagedProcess` 的停机流程被抽象为 `_stop_started_processes`，并在异常路径中统一使用 `ManagedProcessStopError` / `ManagedProcessStopTimeoutError`，提升了资源管理子系统的模块化与可测试性。 |
| **性能影响** | - 新增的 `runtime_handle.clone()` 与一次 `catch_unwind` 的开销极低（微秒量级），对正常业务路径几乎不可感知。<br>- 强制终止子进程时多了一轮 `process.wait(timeout)` 再次确认退出，同样仅在异常/停机场景出现，属于可靠性而非吞吐量优化。 |
| **安全考虑** | - 通过 `catch_unwind` 防止因运行时已 shutdown 而触发的 panic 传播至外部（尤其 Python 解释器），避免进程崩溃，提升了安全性（防止拒绝服务）。<br>- 新增的异常类对错误信息提供了更精确的上下文，有助于审计与调试。未引入新的攻击面。 |
| **可维护性** | - 代码路径明确：`Drop` → `runtime_handle.spawn` → `unregister`，并在无法获取运行时时记录警告而非 panic。<br>- `ManagedProcess` 的停止逻辑被明确拆分为普通停止、强制停止、错误包装，后续若需修改策略（如增添日志或 metric）更为方便。 |
| **兼容性** | - 对外 API（`EventPublisher::new`）保持不变，仅内部增加字段；对使用者透明。<br>- 新增的 Python 警告过滤仅影响测试/CI 环境，不改变运行时行为。 |

---

### ⚠️ 潜在风险
1. **运行时已关闭但仍尝试 `spawn`**  
   - 虽然已捕获 `panic`，但若业务在运行时被提前关闭（如手动 `shutdown_background`），`runtime_handle.spawn` 可能返回 `Err`, 业务逻辑仍会继续执行而不知情。当前实现仅记录警告，若注销失败可能导致发现服务残留。  
2. **异常屏蔽**  
   - `catch_unwind` 会吞掉所有 panic，包括可能指示严重逻辑错误的 panic，导致后续调试难度提升。建议在 CI 中打开 `RUST_BACKTRACE=1` 并监控警告日志。  
3. **强制终止子进程的副作用**  
   - 使用 `kill`/`terminate_process_tree` 可能导致子进程未完成写入日志或持久化数据的清理工作。若业务依赖子进程的 graceful shutdown，需确认在异常情况下仍能接受强制杀死的后果。  

---

### 💡 关注建议
- **运行时监控**：在生产部署中加入监控，捕获 `EventPublisher` 注销失败的 warning，必要时触发报警或自动重注册。  
- **测试覆盖**：新增单元测试验证 `EventPublisher::drop` 在没有 Tokio 运行时的环境下不会 panic，并检查日志输出。  
- **文档更新**：在 `EventPublisher` 的公开文档中说明其在 Python/跨语言使用场景下的生命周期行为，以及在 `Drop` 时可能出现的 “runtime unavailable” 警告。  
- **异常可视化**：考虑在 `ManagedProcessStopError` 中加入错误码或业务标签，方便在 CI 中区分“超时”与“强制结束”两类故障。  
- **回滚计划**：如果在大规模部署后出现大量 “runtime unavailable” 警告，准备回滚到旧的 `tokio::spawn` 实现并调查是否有更根本的运行时管理问题（如主进程提前关闭 Tokio）。  

--- 

**结论**：本次提交通过显式保存创建时的 Tokio 运行时句柄并在 `Drop` 中安全地调度注销任务，彻底消除了在非 Tokio 环境下的段错误；同时，增强了 `ManagedProcess` 的停止流程，使子进程清理更可靠。风险主要集中在异常/关闭路径的可观测性上，建议通过日志/监控补足可见性。整体上，这是一次高价值的可靠性修复，值得在所有发布渠道尽快推广。
