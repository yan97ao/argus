### feat: expose WorkerType + extend register_model (#8700)
**SHA**: `cd3c322` | 🔗 [查看提交](https://github.com/ai-dynamo/dynamo/commit/cd3c322b4e5453940e8aaa2f88ed184da14a4495)

**🎯 变更类型**：功能增强、架构变更、兼容性改进  
**⚡ 重要程度**：🔴 高  
**📋 变更摘要**：本次提交在 Rust 核心库以及 Python 绑定中新增 `WorkerType` 枚举，并在模型注册流程中引入 `worker_type` 与 `needs` 两个字段，用于描述分布式解耦（disaggregation）拓扑的角色与依赖关系。相应地，模型卡（`ModelDeploymentCard`）的校验摘要（mdcsum）加入了这两项，以确保拓扑元数据变更能够触发滚动更新。大量调用点（`attach`、`register_model`、入口示例、单元测试）同步迁移至新接口，并提供了向后兼容的 shim。

---

### 🎯 影响范围
- **核心库**：`lib/backend-common/src/worker.rs`、`lib/llm/src/local_model.rs`、`lib/llm/src/model_card.rs`、`lib/llm/src/discovery/model.rs`、`lib/llm/src/entrypoint/input/endpoint.rs`、相关测试文件。  
- **Python 绑定**：`lib/bindings/python/rust/lib.rs`、`lib/bindings/python/src/dynamo/_core.pyi`。  
- **文档/示例**：已在 `docs/proposals/health-disagg-readiness.md` 中描述（隐含更新）。  
- **用户代码**：所有调用 `register_model` / `Model.attach` 的用户（Python、Rust）需要适配新参数，旧调用仍可工作（通过 shim），但会得到默认 `Aggregated` 行为。

---

### 🔍 技术洞察

| 维度 | 影响分析 |
|------|----------|
| **架构影响** | - 引入 **WorkerType**（Prefill / Decode / Encode / Aggregated）以及 **needs**（DNF 形式的依赖集合），为 **分布式解耦**（disaggregation）提供正式的拓扑描述。<br>- `ModelDeploymentCard` 现在记录角色与依赖，`ModelDiscovery` 通过 `ws_role_and_needs` 能够判断拓扑是否已就绪，从而在健康检查 / 动态扩缩容层面做出更精准决策。<br>- 通过在 `mdcsum` 中加入这些字段，确保任何拓扑元数据变动都会导致滚动更新（drain‑and‑redeploy），防止旧拓扑误用新模型。 |
| **性能影响** | - 仅在模型注册与校验阶段多做几次枚举匹配和向量遍历，开销极低（O(#needs)），对推理路径无影响。<br>- `mdcsum` 计算多了 ~20 字节的哈希输入，同样可忽略。 |
| **安全考虑** | - 新增字段不涉及网络 I/O、凭证或执行权限。唯一的安全关注点是 **拓扑错误**（如错误的 `needs` 配置导致错误的依赖解析），可能导致流量路由到不具备所需能力的 worker，进而产生错误的响应。建议在服务端加入 **验证**（比如检查 `needs` 中的 WorkerType 是否在当前部署中可达）。 |
| **可维护性** | - 将 `WorkerType` 通过 Python `pyclass` 暴露，保持语言间的一致性，提升可读性。<br>- 通过 `resolve_worker_type_and_needs` 在 `worker.rs` 集中逻辑，降低重复实现风险。<br>- 为后向兼容提供 shim（`worker_type = None` → `Aggregated`），避免突发破坏。 |
| **兼容性** | - 旧版调用仍能编译运行，默认行为与之前的 `Aggregated` 保持一致。<br>- `ModelDeploymentCard` 的 JSON 反序列化已加入默认值，保证历史卡片能够无缝加载。<br>- 仅在 **Phase 3**（严格后端注册）时可能出现兼容性错误，文档已提前说明。 |

---

### ⚠️ 潜在风险
1. **业务逻辑错误**：用户未显式提供 `worker_type`/`needs`，仍使用默认聚合模式，可能在实际部署的分布式场景下导致拓扑不完整却被误判为就绪。  
2. **后向兼容性失效**：在 Phase 3 之前如果后端开始强制校验 `worker_type`，未迁移的客户端会收到运行时错误。  
3. **配置错误导致循环依赖**：`needs` 采用 DNF 表达，若配置形成循环（如 Prefill 需要 Decode，Decode 需要 Prefill），当前实现不检测，会导致拓扑永远不满足就绪条件。  
4. **测试覆盖不足**：虽然添加了模型卡 checksum 测试，但对 `needs` 解析、拓扑就绪判定的路径仍缺少端到端集成测试。  
5. **文档同步**：Python 类型提示 (`_core.pyi`) 已更新，但示例代码或第三方 SDK 仍使用旧签名，可能产生运行错误。

---

### 💡 关注建议
- **立即升级**：在所有后端服务（Rust 与 Python）统一使用新 `register_model` 接口并显式声明 `worker_type` 与 `needs`，防止依赖默认 shim 带来的隐蔽错误。  
- **拓扑校验**：在模型注册前添加一次 **needs 校验**：确保所有 `needs` 中的 `WorkerType` 在同一 `namespace` 的 `WorkerSet` 中至少有一个满足的候选。可以在 `ModelDiscovery::ready` 中提前抛出清晰错误。  
- **循环检测**：实现一次简单的 **有向图环检测**（如在 `model.rs` 中）来提示用户错误的 DNF 配置。  
- **滚动更新验证**：验证 `mdcsum` 变更确实触发了预期的重新部署路径（`watcher.rs`），防止因为缓存或 `OnceLock` 导致旧哈希残留。  
- **文档与示例**：完善 `docs/proposals/health-disagg-readiness.md` 中的使用指南，提供完整的 Python 示例（包括 `WorkerType.Prefill` 等）。同时更新 SDK 的类型注解与自动生成的文档。  
- **回归测试**：在 CI 中加入 **拓扑就绪** 场景的集成测试：不同 `worker_type`/`needs` 组合下，`ModelDiscovery::is_ready` 返回预期结果。  
- **监控告警**：在运行时加入 **拓扑不完整** 警报（如 `needs` 未满足），帮助运维快速定位配置问题。  

通过上述措施，可在保证新功能可用性的同时，最小化对现有部署的冲击，并提升系统在分布式解耦场景下的可观测性与安全性。

### fix(operator/lws): preserve legacy worker pod labels (#9738)
**SHA**: `959b9ee` | 🔗 [查看提交](https://github.com/ai-dynamo/dynamo/commit/959b9eebf21d92237820eb59e7aa296df033d0ba)

**🎯 变更类型**：功能增强 / Bug修复  
**⚡ 重要程度**：🔴高  

**📋 变更摘要**  
1. 在 `DynamoComponentDeploymentReconciler` 中新增 `getDCDWorkloadPodLabels`，用于在生成 LeaderWorkerSet（LWS）时把 **Legacy Worker** 的 pod 标签与当前工作负载的组件类型保持一致。  
2. `generateLeaderWorkerSet` 现在使用该函数返回的 `podLabels` 来构建 leader 与 worker 的 pod 模板，而不再直接复用 `labels`（此前只包含组件层面的标签）。  
3. 删除了一些已失效的测试代码，新增大量升级路径单元测试（包括 Deployment、LeaderWorkerSet、Grove PodCliqueSet）的验证，确保在从 v1alpha1 到 v1beta1 的迁移过程中 **不会触发不必要的滚动升级**，且旧的 worker‑identity 标签仍被保留。  

**🎯 影响范围**  
- `deploy/operator/internal/controller/dynamocomponentdeployment_controller.go`（核心业务逻辑）  
- `deploy/operator/internal/controller/dynamographdeployment_controller_test.go`（测试）  
- 新增 `deploy/operator/internal/controller/upgrade_test.go`（升级路径测试）  
- 相关的 mock、常量以及标签处理代码均受到影响。  

**🔍 技术洞察**  

| 维度 | 影响描述 |
|------|----------|
| **架构影响** | - 引入 `getDCDWorkloadPodLabels` 统一了 **DynamoComponentDeployment**（DCD） 与 **Legacy Worker** 的标签生成逻辑，避免了在不同代码路径中重复实现。<br>- 通过维护 `ComponentType` 与 `SubComponentType` 的标签映射，确保 LWS 的 **leader/worker** pod 与旧版 Deployment/Service 的标签保持兼容，提升了operator在跨版本升级时的可预测性。<br>- 新增的升级测试覆盖了三种工作负载类型（Deployment、LWS、Grove PodCliqueSet），强化了对 **v1alpha1→v1beta1** 转换的回归保障。 |
| **性能影响** | - 额外的标签查询 `r.getDCDWorkloadComponentType`（已有的内部函数）只在 **生成 LWS** 时调用一次，计算量极小，对控制器的调和速率（reconcile）几乎没有可感知的开销。<br>- 通过 `make(map[string]string)` 创建局部副本的方式保持了 O(N) 的线性复杂度，且 N（标签数）通常只有十几条，影响可忽略。 |
| **安全考虑** | - 该改动仅涉及 **metadata.labels**，不修改任何 `spec` 或 `securityContext`，不引入新的安全风险。<br>- 保留了旧标签（如 `nvidia.com/dynamo-component-type=worker`），避免因标签不匹配导致的 **RBAC**、**NetworkPolicy** 或 **Service** 选择器错误，从而维持已有的安全策略。 |
| **可维护性** | - 将标签构造抽取为独立函数后，后续若需调整标签策略（比如新增组件类标签）只需在一处修改，降低了维护成本。<br>- 单元测试覆盖率显著提升，尤其是 **升级路径** 的行为被明确断言，未来代码改动更易定位因标签变化导致的回滚或不期望的滚动升级。 |
| **向后兼容性** | - 通过在 `getDCDWorkloadPodLabels` 中仅在需要时追加 `ComponentType` 与 `SubComponentType`，若原始 `labels` 已经包含这些键，则保持原样，确保对已使用 **Native Worker**（非 Legacy） 的集群不产生副作用。<br>- 测试验证了 **native**（非 Legacy）工作负载的标签保持不变，防止误迁移。 |

**⚠️ 潜在风险**  
1. **标签冲突**：如果用户在自定义 `metadata.labels` 中自行添加了与内部常量相同的键，`getDCDWorkloadPodLabels` 的覆盖行为可能导致意外修改。  
2. **遗漏组件类型**：`getDCDWorkloadComponentType` 若返回空字符串（例如新组件尚未在映射中），当前实现会直接返回原始 `labels`，这在未来引入新组件时可能导致 **Legacy Worker** 与新组件不对齐。  
3. **测试环境依赖**：新增的大量升级测试在 CI 中会消耗更多资源，若未及时维护（比如底层 API 变更导致 yaml 解析失效），可能导致测试失效并掩盖真实回归。  
4. **旧版 Operator 兼容**：该提交仅在当前 master 分支上生效，若用户仍在使用旧版 Operator（未包含此补丁），在升级到新 CRD 版本后仍会出现标签不匹配的情况，需要在升级指南中强调此变更。  

**💡 关注建议**  
- **文档与升级指南**：在 Operator 的升级说明中明确指出 **Legacy Worker** 标签的保持机制，建议在升级前检查已有工作负载的标签是否正确。  
- **防御性检查**：在 `getDCDWorkloadPodLabels` 中加入日志（或事件）提示当用户自定义标签覆盖内部标签时，以便运维快速定位。  
- **扩展映射**：为 `getDCDWorkloadComponentType` 提供单元测试或配置化映射，以便将来新增组件类型时不遗漏标签同步。  
- **CI 稳定性**：将升级路径测试拆分为独立的 `integration` 目录或使用 `go test -run Upgrade` 标记，以免影响常规单元测试的执行时间。  

总体而言，此次提交通过统一标签处理、补全升级路径测试，显著提升了 **Operator** 在跨版本迁移时的可靠性与可观测性，风险可控且收益明显。

### fix(routing): exclude over-threshold workers from load-balancer candidate sets (#9688)
**SHA**: `9c7a9d2` | 🔗 [查看提交](https://github.com/ai-dynamo/dynamo/commit/9c7a9d2d4a6d3de9ca9e6d2c21987dba2d8b61de)

**🔍 技术洞察**  

- **架构影响**  
  - **实例过滤层提升**：在客户端内部将 “可路由实例集合” (`routable_ids`) 与 “实际可用实例集合” (`free_ids`) 明确区分，遵循单一职责原则，避免上层路由逻辑再次进行过滤。  
  - **错误统一**：新增 `empty_free_pool_error`，统一返回 `DynamoError`（`ErrorType::ResourceExhausted`），对外提供更语义化的错误码，便于调用方做幂等或重试控制。  
  - **API 稳定性**：对外仅暴露 `RoutingInstances`（`pub(crate)`），因此对外部 crate 的破坏性最小；但内部使用方需从 `.routable_ids()` 切换到 `.free_ids()`，已在库内部完成迁移。  

- **性能影响**  
  - **计算开销**：`derive_free_ids` 在每次 `RoutingInstances` 重建时遍历一次 `routable_ids`（通常为十几到几百个实例），时间复杂度 O(N)。相较于之前的每次路由都在 `routable_ids` 上做过滤，这是一种 **一次性预计算**，整体请求路径的 CPU 开销会略有下降。  
  - **内存占用**：`free_ids` 额外持有一个 `Vec<u64>`，与已有 `routable_ids` 大小相当，整体内存占用几乎不变。

  - 通过统一错误类型，避免将内部实现细节泄露给上层，同时使监控系统能够区分 **资源耗尽** 与 **实例未发现** 两类错误。

---

**⚠️ 潜在风险**  

1. **并发更新的瞬时不一致**  
   - `free_ids` 是在 `RoutingInstances` 重建时一次性生成的。如果在极短时间内实例的 overload 状态频繁切换，可能出现 “瞬时空闲池” 与实际占用状态不一致的窗口，导致路由器选择到一个刚刚被标记为 overloaded 的实例，随后在 `route` 的后置检查中被拒绝，返回 503。该窗口已在原实现中通过后置检查存在，只是现在提前过滤，风险并未增大。  

2. **向后兼容性**  
   - 虽然 `RoutingInstances` 仍然公开 `routable_ids()`，但库内部已转向使用 `free_ids()`。若外部用户直接依赖 `routable_ids()` 并自行进行 overload 判断，可能在升级后出现行为差异（如仍会得到已 overload 的实例）。文档需明确 `free_ids` 为路由器推荐使用的集合。  

3. **错误处理路径变化**  
   - 之前路由器在实例为空时返回 `anyhow::anyhow!("no instances found…")`，现在改为 `DynamoError`（`ResourceExhausted`）。调用方如果只匹配字符串或 `anyhow::Error`，可能失去对错误类型的细粒度判断，需要相应更新。  

---

**💡 关注建议**  

1. **文档更新**  
   - 在 `RoutingInstances` API 文档中加入 `free_ids` 的说明，强调其为 “已过滤掉 overload 实例的可调度实例集合”。  
   - 在路由器使用说明中标明错误类型已统一为 `ErrorType::ResourceExhausted`，并提供示例如何基于 `DynamoError` 进行重试或降级。  

2. **监控与告警**  
   - 引入对 `ResourceExhausted` 错误的专门告警，帮助运维快速发现系统整体过载情况，而不是仅看到散落的 “no instances found” 错误。  

3. **回退策略**  
   - 若业务对瞬时空闲池不一致导致的偶发 503 敏感，可在上层实现 **重试**（指数退避）或 **熔断**，利用统一的错误类型做精准判断。  

4. **测试覆盖**  
   - 保持现有的单元测试，同时在集成测试中加入 **高并发 overload 状态切换** 场景，验证 `free_ids` 与后置 overload 检查的协同效果，防止潜在的竞争条件。  

5. **性能基准**  
   - 在生产规模（上千实例）下跑一次路由路径基准，确保一次性预计算 `free_ids` 的开销仍在可接受范围内，若出现回归再考虑增量更新策略。  

--- 

此修复通过明确分离 **可路由** 与 **可用** 实例集合，提升了负载均衡的正确性和错误语义，对系统的可靠性有显著正面影响。只要关注上述兼容性与监控要点，即可安全升级。

  2. **依赖表达（DNF）**：`needs: Vec<Vec<WorkerType>>` 采用析取范式，使得 “A 需要 (B 且 C) 或 D” 能够简洁描述，提升拓扑表达能力并为 **健康检查**、**调度** 提供机器可判定的依据。  
  3. **就绪度判定**：`Model::is_workers_ready`、`first_ready_workers`、`has_ready_workers` 通过遍历同一 namespace 的 `WorkerSet`，基于当前活跃 worker 数与 `needs` 的 DNF 计算，决定该拓扑是否能对外服务。该判定是 **无锁、只读**，但依赖 `Arc<WorkerSet>` 与 `watch` 机制的更新，保持与现有事件驱动模型兼容。  
  4. **向后兼容**：使用 `#[serde(default)]` 与临时 shim (`worker_type == None → Aggregated`) 让旧版后端在未提供 `worker_type` 时仍然可被视为聚合节点，避免首次升级即导致不可用。文档已明确此 shim 为 **临时**，后续必须移除。  

- **性能影响**  
  - 新增的 `distinct_namespaces_sorted`、`is_workers_ready` 等函数在模型查询时会 **遍历所有 `WorkerSet`**（目前是 `DashMap`），时间复杂度 O(N)（N 为 worker sets 数），对大规模集群（数千 worker sets）仍在毫秒级。  
  - `HashSet<WorkerType>` 的创建、`needs` 的 DNF 检查均是轻量级 CPU 操作，内存占用仅数十字节。  
  - 由于就绪度检查是 **惰性调用**（仅在 API / 调度逻辑需要时触发），对常规请求路径的影响极小。  
  - 若业务对就绪度查询频繁（如轮询健康探针），可考虑**缓存** `has_ready_workers` 的结果并在 `WorkerSet` watcher 触发时失效。  

- **安全考虑**  
  - 变更本身不涉及网络、权限或加解密逻辑。  
  - 主要风险是 **错误的 `needs` 表达** 或 **缺失的 `worker_type`** 可能导致模型错误地被标记为 “ready”，进而对外暴露未准备好的服务，产生 **服务不可用（Denial of Service）** 或 **错误返回**。  
  - 临时 shim 将缺失 `worker_type` 视作 `Aggregated`，可能在特定拓扑（如仅需要 Prefill）误判为可用，需在生产环境 **严格审计** 后端卡片确保字段完整。  

**⚠️ 潜在风险**  

| 风险点 | 说明 | 缓解措施 |
|--------|------|----------|
| **后端未同步 `worker_type/needs`** | 仍使用旧卡片会被 shim 当作 `Aggregated`，在复杂拓扑（如 Encode 需要 Prefill+Decode）可能误判为就绪 | 在升级计划中强制所有后端填写 `worker_type`，并在 shim 移除前进行 **回归测试**。 |

| **可维护性** | - 大量新增 `*_assigned`、`on_session_metadata` 等方法，保持 **单一职责**（Collector 负责记录，Runtime 负责调度）。<br>- `PerRequestRecord` 序列化通过 `serde_json` 完全自动，无手写拼接，易于后续字段扩展。<br>- 新增的测试覆盖了捕获开启、字段填充、排序、序列化等关键路径，提升可靠性。 |

### ⚠️ 潜在风险
1. **内存溢出**：在极大规模的离线重放（>10 M 请求）且开启 `--report-jsonl`，`per_request` 向量会一次性占用数十 GB 内存。  
2. **磁盘空间耗尽**：每条记录约 300 ~ 500 字节，10 M 请求约 3–5 GiB，若磁盘空间不足会导致写入错误，进而导致整体重放失败。  
3. **I/O 性能瓶颈**：对慢速磁盘（网络挂载、SSD 低写速）可能成为最慢环节，导致 `run_mocker_trace_replay` 的总耗时显著增加。  
4. **错误路径**：当前仅检查 `replay_mode=="offline"`，未对 `report_jsonl_path` 的可写性进行提前验证，可能在写入阶段抛出 `std::io::Error`，导致 Python 抛异常但不易定位。  
5. **数据一致性**：如果运行过程中被 **Ctrl‑C** 中断，已写入的部分文件会保留不完整的 JSONL，后续消费工具需能够容错或重新生成。

### 🔍 技术洞察

#### 1. 架构影响
- **层次化 Offload 模型**：原仅 G1↔G2（GPU ↔ Host）模型扩展为 **G1↔G2↔G3**，G3 通过 **进程共享** (`SharedG3Pool`) 实现多 worker 之间的写‑回竞争与容量共享。  
- **容量预留抽象**：`CapacityReservations` 通过原子计数与 `Arc` 保证跨线程/worker 的 **预留/释放**，防止超过物理块数导致 “假命中”。  
- **调度入口统一**：`EngineCore` 与 `OfflineWorkerState` 增加 `tick_offload_only` 与 `earliest_offload_deadline`，让离线/聚合调度在 **不前进时间** 的情况下先驱动所有纯 Offload 完成，随后再进行 admission。  
- **配置/校验统一**：`KvbmOffloadConfig::from_args` 现在返回 `Result<Option<…>>`，在缺失 `kv_bytes_per_token`、`num_g2_blocks`、`num_g3_blocks` 任一必需条件时直接返回错误，避免运行时 panic。  
- **序列化兼容**：`MockEngineArgsSerde` 结构体同步新增字段，保持向后兼容（默认 `None` 表示禁用）。  

#### 2. 性能影响
| 影响 | 说明 |
|---|---|
| **模拟开销** | 新增两条 **PS（Processor‑Sharing）模型**（本地 G1↔G2、共享 G2↔G3），每次 `tick` 必须遍历两套活动传输集合并计算最早完成时间，理论 CPU 开销约 1‑2%（取决于批次规模）。 |
| **并发争用** | 共享 G3 使用单一 `PendingTracker` 与 `CapacityReservations`，所有 worker 对 G3 的写‑回会排队（`max_concurrent_transfers = 1`），这在大规模离线 replay 中可能成为 **瓶颈**（模拟真实 NVMe 共享带宽）。 |
| **内存占用** | G3 块数默认 1 024，块大小默认 1 000 000 B（≈1 MiB），最高约 1 GiB 虚拟内存；若用户自行设定更大值可能导致 **OOM**。 |
| **报告统计** | 新增 `first_admission_prefix_cache_reused_ratio`，在 `TraceCollector` 中额外累计一次 `first_admission_reused_input_tokens`，对离线统计的时空复杂度提升几乎可以忽略。 |

#### 3. 安全/健壮性考虑
- **输入校验**：`non_negative_int/float` 现在在 Python 层提前拦截负数，防止下游出现负值导致 `usize`/`f64` 解析错误或 overflow。  
- **跨进程共享**：`SharedG3Pool` 使用 `OnceLock<Mutex<Option<Weak<…>>>` + `Arc`，确保 **一次性初始化** 与 **弱引用自动回收**，避免资源泄漏。  
- **并发原子**：`CapacityReservations` 采用 `fetch_update`/`compare_exchange_weak`，防止 **ABA** 与 **竞争**，并在 `Drop` 中安全释放。  
- **错误传播**：`KvbmOffloadConfig::from_args` 现返回 `anyhow::Result`，所有调用已更新为 `?` 处理，保证错误信息清晰、不会因 `unwrap` 导致进程崩溃。  
- **资源释放**：`MockOffloadEngine::drop` 关闭 `PendingTracker`、`EventManager`，确保线程安全退出。  

#### 4. 兼容性
- **默认行为保持不变**：若不传递任何新参数或将其设为 `0`，离线/实时路径仍
