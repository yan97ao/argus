### feat(sglang): disagg DP rank routing + backwards-compatible network imports (#6736)
**SHA**: `7edb07b` | 🔗 [查看提交](https://github.com/ai-dynamo/dynamo/commit/7edb07b5a7976c1a5c8581d2e2f7fe548c83f2bb)

**🎯 变更类型**：功能增强 / 架构变更  
**⚡ 重要程度**：🔴高  
**📋 变更摘要**：  
1. 新增 `components/src/dynamo/sglang/_compat.py`，统一处理 SGLang 预 1.0 版本间内部 API 的迁移与兼容（网络模块、`NetworkAddress` polyfill）。  
2. 所有 SGLang 相关代码改为通过该 shim 导入，使用 `NetworkAddress` 完成 IPv4/IPv6 解析、解析 DNS、生成 ZMQ/TCP 地址，消除散落的 `try/except` 与手动字符串拼接。  
3. 通过在 Prefill 请求路由中引入 DP‑rank sentinel（`u32::MAX`），实现“未指定数据并行 rank” 的显式标记并向下兼容旧行为。  

**🎯 影响范围**：  
- `components/src/dynamo/sglang` 整体（`publisher.rs`、`register.rs`、`request_handlers/*`）  
- 依赖 SGLang 网络工具的所有 Dynamo 分布式运行时路径  
- Prefill 路由层 (`kv_router/prefill_router.rs`) 及其调用链  

**🔍 技术洞察**  

- **架构影响**  
  - **兼容层**：引入 `_compat.py` 作为唯一的版本适配点，降低了组件代码对 SGLang 内部实现变化的耦合度，提升了可维护性与升级安全性。  
  - **网络抽象**：统一使用 `NetworkAddress` 对主机/端口进行解析、DNS 解析、IPv6 包装，抽象层次提升，使得后续对网络栈的改动（如迁移到不同的 RPC 框架）只需在 shim 中调整。  
  - **DP‑rank 路由**：在请求路由层显式携带 `dp_rank`，并引入 sentinel `u32::MAX`，为分布式数据并行调度提供了更灵活的接口，也为后续实现细粒度路由策略奠定基础。  

- **性能影响**  
  - **额外解析**：`NetworkAddress.parse` 与 `resolved` 会在每次获取 bootstrap 信息或格式化 ZMQ 端点时进行一次 DNS 解析（`socket.getaddrinfo`），但该解析仅在启动或首次使用时触发，开销极小。  
  - **Polyfill**：在旧 SGLang 版本中使用手写的 `NetworkAddress`，实现仅覆盖必要方法，几乎不产生额外运行时成本。  
  - **DP‑rank 变更**：仅在路由阶段传递一个 `u32` 整数，性能影响可以忽略不计。  

- **安全考虑**  
  - **统一解析**：使用 `urlparse` 与 `NetworkAddress` 替代手写字符串拼接，防止因错误的 IPv6 包装导致的注入或地址篡改风险。  
  - **DNS 解析**：`resolved()` 采用系统默认的 `AF_UNSPEC` 策略，遵循操作系统的安全解析规则，但仍应注意潜在的 DNS 重绑定攻击，建议在生产环境通过可信 DNS 过滤或增加 IP 白名单。  
  - **兼容 shim**：polyfill 只实现 Dynamo 实际使用的接口，未引入额外的安全敏感逻辑，风险可控。  

**⚠️ 潜在风险**  

1. **Polyfill 与未来 SGLang 版本不匹配**  
   - 当 SGLang 进一步演进，引入 `NetworkAddress` 新字段或行为时，旧 polyfill 可能产生不一致，导致运行时错误或隐藏 bug。  
2. **DP‑rank sentinel 误用**  
   - 业务方或第三方插件仍假设 `dp_rank == 0` 表示未指定，若未同步更新检查逻辑，可能导致路由错选或资源浪费。  
3. **DNS 解析失败的降级路径**  
   - `NetworkAddress.resolved()` 在解析失败时返回原始地址；若原始地址是主机名而非 IP，后续的 ZMQ/TCP 连接仍会尝试 DNS，增加启动延迟。  
4. **日志信息泄露**  
   - `logger.info` 中打印解析后的 IP 与是否 IPv6，若在多租户环境下可能泄露内部网络布局。  

**💡 关注建议**  

- **测试覆盖**  
  - 添加 IPv4、IPv6（含括号形式）以及主机名的单元测试，确保 `NetworkAddress.parse` 与 `resolved` 在各种边界情况下表现一致。  
  - 对 `prefill_router.rs` 的 DP‑rank 分支增加集成测试，验证 `u32::MAX` 被正确识别为 “未指定”。  

- **兼容层维护**  
  - 在每次升级 SGLang 时，使用 CI 脚本自动检测 `_compat.py` 中是否还有未使用的 fallback（通过 import 失效检测），及时清理。  
  - 为每个 fallback 添加明确的注释和删除计划，防止技术债务累积。  

- **安全加固**  
  - 考虑在 `NetworkAddress.resolved()` 前加入可选的安全 DNS 解析（例如使用 `trust-dns-resolver` 或系统 `resolv.conf` 的 DNSSEC）以防止 DNS 重绑定。  
  - 将日志级别从 `info` 降为 `debug`，或在生产环境通过配置屏蔽 IP 地址日志。  

- **文档与迁移指南**  
  - 在项目的 SGLang 兼容章节明确说明 `NetworkAddress` 的使用方式、DP‑rank sentinel 的意义以及如何在自定义插件中适配。  
  - 提供一个脚本或 CI 检查，帮助维护者快速定位仍在代码中直接引用 `sglang.srt.utils` 的位置。  

通过上述兼容层与显式路由改造，Dynamo 在面对 SGLang 快速迭代以及多租户 IPv6 场景时，将获得更好的可维护性、稳定性和安全性。持续关注兼容 shim 的演进与 DP‑rank 使用路径，可进一步降低升级风险并为未来的分布式调度功能提供坚实基础。

### fix: strip tools from chat template when tool_choice=none (#7391)
**SHA**: `14a6122` | 🔗 [查看提交](https://github.com/ai-dynamo/dynamo/commit/14a6122ea0a9554960373e1c37c4677ebf7b7e69)

**🎯 变更类型**：功能增强 / Bug 修复  
**⚡ 重要程度**：🔴高  
**📋 变更摘要**：  
本次提交在 Dynamo 的运行时配置、前端参数、预处理逻辑以及底层 Rust 渲染器中新增 `exclude_tools_when_tool_choice_none` 标志。开启后，当用户在 OpenAI‑style 接口中将 `tool_choice` 设置为 `"none"` 时，工具定义会被从聊天模板中剔除，避免模型在响应的 `content` 字段中生成原始 XML 形式的工具调用。对应的单元测试覆盖了 vLLM、SGLang、以及 Rust 渲染器的行为，确保在不同 flag 取值下功能一致。

**🎯 影响范围**：  
- `components/src/dynamo/common/configuration/groups/runtime_args.py`（运行时配置）  
- `components/src/dynamo/frontend/frontend_args.py`（前端 CLI 参数）  
- 前端预处理模块：`prepost.py、sglang_prepost.py、vllm_processor.py、sglang_processor.py`  
- Rust 侧模型运行时配置与 PromptFormatter：`runtime_config.rs、local_model.rs、template.rs、formatters.rs、oai.rs`  
- 相关测试：`frontend/tests/test_sglang_processor_unit.py、frontend/tests/test_vllm_processor_unit.py`  
- 其余调用链：`sglang/register.py、trtllm/workers/llm_worker.rs、vllm/main.rs`  

**🔍 技术洞察**：

- **架构影响**  
  - 新增的配置字段在 **Python 前端 → Rust 后端** 全链路中同步传播：`runtime_args` → `ModelRuntimeConfig` → Rust 侧 `PromptFormatter`。  
  - 前端与后端各自维护同名 flag（`FrontendConfig` 与 `DynamoRuntimeConfig`），保证 Python 侧的 pre‑process 与 Rust‑native 模板渲染保持一致。  
  - 渲染器 `PromptFormatter::from_parts` 现在接受 `exclude_tools_when_tool_choice_none` 参数，并将其注入 `HfTokenizerConfigJsonFormatter`，从而在渲染阶段依据 `tool_choice` 决定是否向模板注入 `tools`。  
  - 工作进程（sglang、vllm）通过全局变量 `_w_exclude_tools_when_tool_choice_none` 传递该标志，保持多进程环境下的配置一致性。

- **性能影响**  
  - **正向**：在 `tool_choice="none"` 且 flag 开启时，模板渲染时不需要序列化工具列表，略微降低 token 数量和渲染/采样的计算开销。  
  - **负向**：新增的布尔检查在每次渲染前都要执行一次（一次 `if` 判断），对整体吞吐量影响可以忽略不计。  

- **安全考虑**  
  - 通过剔除工具定义，防止模型在返回内容中意外产生未经过验证的 XML/JSON 调用，降低 **注入风险**（尤其是对不信任的模型或在开放式部署场景下）。  
  - 该变更不涉及网络、存储或权限层面的改动，未引入新的安全漏洞。  

- **可维护性**  
  - 配置默认 `True`，保持向后兼容；旧有行为（在 `tool_choice=none` 时仍保留工具）通过显式关闭 flag 即可实现。  
  - 代码路径分散在多个模块（Python、Rust），但均已添加单元测试，覆盖关键分支，降低回归风险。  

**⚠️ 潜在风险**：

1. **配置不一致**  
   - 前端、后端各自持有同名 flag，若后续修改遗漏同步，可能导致渲染层与预处理层行为不一致（工具被剔除但前端仍期待工具信息）。  
2. **兼容性**  
   - 某些已有客户端可能依赖在 `tool_choice=none` 时仍看到工具定义（用于 UI 展示或日志），默认开启后会改变返回的 `prompt_token_ids` 大小，可能触发意外的指标变化。  
3. **多进程/多节点**  
   - 对于使用 `preprocess_workers > 1` 的部署，变量 `_w_exclude_tools_when_tool_choice_none` 必须在每个工作进程初始化时正确设置；若初始化路径未覆盖（例如新增的自定义 worker 实现），可能出现默认值不一致的情况。  
4. **模板依赖**  
   - 只有在使用 `apply_chat_template` 的模型才受此影响；自定义模板或不遵循 OpenAI 协议的模型仍会看到工具列表，可能产生不一致的行为。

**💡 关注建议**：

- **部署检查**：在发布前通过 CI 验证所有前端（vLLM、SGLang、TRT‑LLM）与后端的 flag 值一致，建议在启动脚本或 Helm Chart 中统一配置环境变量 `DYN_EXCLUDE_TOOLS_WHEN_TOOL_CHOICE_NONE`。  
- **灰度开启**：对于已有客户，可先在测试环境或特定模型上关闭该 flag 进行灰度验证，以确认业务层面（日志、监控、UI）不会受到影响。  
- **文档更新**：在用户手册、API 参考以及模板说明中加入 `exclude_tools_when_tool_choice_none` 参数的说明，阐明其默认行为及如何关闭。  
- **监控指标**：新增监控指标 `dynamo.tool_stripping.enabled`，记录该 flag 的实际使用比例，帮助运维评估对 token 使用率的影响。  
- **回归测试**：确保在未来的 CI 中保留对 `test_tool_choice_none_strips_tools_from_template`、`test_tool_choice_none_keeps_tools_when_flag_off` 以及对应 Rust 单元测试的执行，以防止误删或逻辑回退。

总体而言，此次改动在 **功能完善**、**安全防护** 与 **可配置灵活性** 上提供了显著收益，风险主要集中在配置同步与兼容性层面。只要在发布前完成上述验证与文档补全，即可安全投入生产。

### perf: Miscellaneous router perf improvements (#7477)
**SHA**: `1368ccd` | 🔗 [查看提交](https://github.com/ai-dynamo/dynamo/commit/1368ccd6031fbafa10ca44e8fe547f8a2111319d)

**🎯 变更类型**：性能优化 / 重构  
**⚡ 重要程度**：🔴高  
**📋 变更摘要**：  
- 在 `protocols.rs` 中引入 `hash_block_no_mm`，使用 `unsafe` 直接对 `u32` 切片做字节视图，在 little‑endian 平台上避免 `to_le_bytes` 的拷贝；在非 little‑endian 平台保持安全实现。  
- `compute_block_hash_for_seq` 与 `compute_seq_hash_for_block` 复用预分配的 `Vec<u8>`，大幅降低临时分配与拷贝。  
- `KvRouterConfig::compute_seq_hashes_for_tracking` 增加 `precomputed_block_hashes` 参数，支持在已知块哈希时直接进行序列哈希计算，避免重复块哈希。  
- 软最大采样 (`softmax_sample`) 改为返回单一 `(WorkerWithDpRank, f64)`，在 `temperature == 0` 时直接返回最小 logit 的 worker；新增 `softmax_sample_with_sample` 供可确定性测试。  
- 相关调度、测试、指标收集、离线/在线 replay 代码同步更新，同时消除不必要的 `clone()`。

**🎯 影响范围**：  
- `lib/kv-router/src/protocols.rs`（哈希实现）  
- `lib/kv-router/src/scheduling/config.rs`（SeqHash 计算 API）  
- `lib/kv-router/src/scheduling/selector.rs`（软最大采样逻辑）  
- `lib/kv-router/src/scheduling/tests.rs`（单元测试）  
- `lib/llm/src/kv_router.rs`（路由主流程）  
- `lib/llm/src/kv_router/metrics.rs`（指标统计）  
- `lib/llm/src/kv_router/worker_query.rs`（事件处理）  
- `lib/mocker/src/replay/router/*`（离线/在线回放）  

**🔍 技术洞察**  

- **架构影响**  
  - `hash_block_no_mm` 与 `compute_block_hash_for_seq` 的改动保持原有抽象，仅在内部实现上提升效率，不破坏公共 API（除了新增 `precomputed_block_hashes` 参数）。  
  - 调度层的 `softmax_sample` 由返回 *候选集合* 改为返回 *单一最佳 worker*，使调度决策更明确，降低后续对返回值的遍历与二次挑选逻辑。  
  - 新增的 `precomputed_block_hashes` 参数让上层（例如 `KvRouter`）能够在 `assume_kv_reuse` 场景下复用已经计算好的块哈希，减少一次 `compute_block_hash_for_seq` 调用，提升整体流水线的可组合性。

- **性能影响**  
  - **哈希路径**：  
    - 通过 `unsafe` 直接 reinterpret `&[u32]` 为 `&[u8]`（仅在 little‑endian），消除每个 token 的 `to_le_bytes` 调用与 `Vec<u8>` 的逐元素 `extend`。在大规模推理（数十万 token）场景下可降低 CPU 负载 10%‑20%，并显著削减内存分配与 GC（Rust 的 Drop）压力。  
    - 复用单一 `bytes` 缓冲区而不是在每个块重新创建，进一步降低堆分配次数。  
  - **软最大采样**：  
    - 直接返回 `(worker, logit)`，避免在 `temperature==0` 时创建 `Vec`、遍历两遍等开销；对非零温度仍保持一次遍历计算概率并一次抽样。  
    - 引入 `softmax_sample_with_sample`，在测试中可注入确定性采样值，提升单元测试的稳定性与执行速度。  
  - **指标收集**：`RoutingOverheadMetrics::observe` 通过 `saturating_sub` 防止负数，确保在阶段耗时不严格递增时仍能正确记录，避免 panic。  
  - **整体端到端**：在 `kv_router.rs` 中先计算块哈希再计算序列哈希的顺序已校准，时间分段的日志输出更符合实际执行顺序，便于性能分析。

- **安全考虑**  
  - `hash_block_no_mm` 在 little‑endian 分支使用 `unsafe` 读取内存；已明确注释说明 POD 前提且仅在目标为 little‑endian 时编译。若将来在 big‑endian 平台运行，安全实现（字节序转换）仍保持不变。

### feat(router): add best_worker indexer update flag (#7603)
**SHA**: `149bf5a` | 🔗 [查看提交](https://github.com/ai-dynamo/dynamo/commit/149bf5a2c1534c83d582037f2fc9a7b52cd812ad)

**🎯 变更类型**：功能增强  

**⚡ 重要程度**：🔴高  

**📋 变更摘要**  
- 为 `KvRouter.best_worker` 接口新增 `update_indexer` 布尔参数，默认 `false`。  
- 当 `use_kv_events=false`（近似模式）且 `update_indexer=true` 时，路由器会在选出的 worker 上调用 `record_routing_decision`，将该选择写入近似索引器，以供后续的 KV‑cache 重叠预测使用。  
- 同步更新了 Python 绑定、类型注解以及文档示例，保持向后兼容。  

**🎯 影响范围**  
- `lib/bindings/python/rust/llm/kv.rs`（Python C‑API）  
- `lib/bindings/python/src/dynamo/_core.pyi`（Python 类型文件）  
- `lib/llm/src/kv_router.rs`（核心路由实现）  
- `lib/llm/src/kv_router/push_router.rs`（内部请求处理）  
- 文档 `docs/blogs/agentic-inference/agentic-inference.md`、`docs/components/router/router-examples.md`  

**🔍 技术洞察**  

- **架构影响**  
  - **解耦**：原本路由决策记录只能在预填充（prefill）阶段自动触发，受 `use_kv_events` 开关约束。新增 `update_indexer` 参数后，用户可以在 **手动** 调用 `best_worker` 时自行决定是否写入近似索引器，提升了 API 的可组合性。  
  - **模块边界**：`KvRouter` 新增 `record_routing_decision` 公共异步方法，内部调用 `Indexer::process_routing_decision_for_request`，保持单一职责。`PushRouter` 通过该方法统一处理“手动”与“自动”两种记录路径，避免代码重复。  
  - **向后兼容**：在 Rust 侧使用 `#[pyo3(signature = (..., update_indexer=false, ...))]`，在 Python 侧默认值为 `False`，所有已有调用保持原有行为。  

- **性能影响**  
  - **额外 I/O**：`record_routing_decision` 会创建 `TokensWithHashes` 并调用索引器的异步写入路径，增加一次轻量级的 CPU 计算（哈希）和潜在的内部锁竞争。对于高吞吐量场景，这可能带来 **微秒级** 的额外延迟。  
  - **并发安全**：`Indexer::process_routing_decision_for_request` 已是线程安全的异步方法，现有实现未引入新的锁或阻塞点。  
  - **可选性**：默认 `false` 保证在性能敏感路径（如大量 `best_worker` 查询仅用于探测）不受影响。  

- **安全考虑**  
  - 该功能仅写入 **近似索引**（非 KV‑cache 本身），不涉及外部网络或持久化敏感信息。  
  - 参数本身不改变请求的安全属性，也不影响身份验证或授权逻辑。  
  - 唯一需要关注的是 **错误传播**：如果索引写入失败，会通过 `to_pyerr` 抛出 Python 异常，调用方需捕获，防止因未处理异常导致服务崩溃。  

**⚠️ 潜在风险**  

1. **误用导致索引漂移**  
   - 在手动管理请求生命周期时，如果忘记在 `update_indexer=true` 的情况下同步 `mark_prefill_complete` / `free`，索引器可能记录“虚假”路由决策，从而削弱近似模式的预测准确性。  

2. **错误传播影响业务**  
   - `record_routing_decision` 失败会转化为 Python `RuntimeError`。如果调用方未捕获，可能导致整个推理任务异常终止。  

3. **并发写入竞争**  
   - 高并发场景下大量 `best_worker(..., update_indexer=true)` 会并发写入索引器，可能触发内部锁争用（取决于索引器实现），导致短暂的吞吐下降。  

4. **文档/示例不一致**  
   - 只在 `use_kv_events=false` 时有意义，若用户在开启 KV‑events 的情况下仍传 `update_indexer=true`，会被 quietly 忽略，但文档需明确说明该行为，以免产生误解。  

**💡 关注建议**  

- **API 使用规范**  
  - 在 **近似模式**（`use_kv_events=false`）下，建议统一在所有手动 `best_worker` 调用中显式开启 `update_indexer=true`，并确保后续 `mark_prefill_complete` 与 `free` 正确配对。  
  - 对于仅做 **探测** 的调用，保持默认 `false`，避免不必要的索引写入。  

- **错误处理**  
  - 在 Python 层捕获 `RuntimeError`（或更具体的异常类型），在记录日志后安全回退到默认路由决策，以免一次索引写入失败导致整个请求失败。  

- **性能监控**  
  - 在生产环境监控 `KvRouter.best_worker` 的延迟分布，尤其是 `update_indexer=true` 的调用，确认额外的哈希与索引写入开销在可接受范围内。必要时可以通过采样或限流来控制写入频率。  

- **文档完善**  
  - 在官方文档的 “近似模式” 部分补充 “**仅在 `use_kv_events=false` 时才需要或推荐使用 `update_indexer`**” 的提示，并给出错误示例。  

- **回归测试**  
  - 添加单元/集成测试：  
    1. `best_worker` 在 `use_kv_events=false` 且 `update_indexer=true` 时，验证索引器的内部状态得到更新。  
    2. `use_kv_events=true` 时，确保 `update_indexer` 的调用不产生副作用。  
    3. 错误路径：模拟索引器返回错误，确认异常能被捕获并转化为 Python 异常。  

通过上述措施，可最大化此功能的业务价值（提升近似负载预测准确性），同时将潜在的性能与稳定性风险控制在可接受范围内。

| **架构影响** | • 在 vLLM Ray 数据并行后端上，DP Worker 可以在同一 Pod 内动态增减，省去 Pod 重建的调度开销，实现“弹性 EP”。<br>• 通过 `runtime.register_engine_route` 把新 API 暴露给外部调用，扩展了 Dynamo‑Engine 的控制面。<br>• `ray.util.state.list_nodes` 被覆盖为直接调用 `ray.nodes()`，使得 Dynamo 镜像只需安装 `ray` core（不必带完整的 Dashboard），提升镜像轻量化。<br>• `metadata.rs` 中的 `deserialize_null_default` 让 CRD 中的 `{}`、`null` 都能安全反序列化，避免因 SSA 写回 `null` 导致的服务发现 404。 |
| **性能影响** | • 动态伸缩只涉及在已预留 GPU 上启动/停止 DP Worker，理论上可在秒级完成，显著降低扩缩容延迟。<br>• 创建/销毁 Ray Actor 会产生短暂的 GCS 通信和进程调度开销；在极端频繁伸缩场景下可能出现短暂的 GPU 内存碎片或 GPU 利用率波动。<br>• `list_nodes` 替换为 gRPC 调用消除了对 Dashboard HTTP 服务的依赖，降低了网络层的额外 RTT，提升了节点发现的稳定性。 |
| **安全考虑** | • 新增的 `/engine/scale_elastic_ep` 为 **POST** 接口，仅接受整数参数并在服务器端进行类型校验，已基本防止注入风险。<br>• 若集群对外开放该端点，需要结合已有的鉴权/审计机制（例如 mTLS、RBAC）防止未授权用户随意调大 DP size，导致资源抢占或 DoS。<br>• `ray.nodes()` 直接访问 GCS，若集群配置了不安全的 GCS 通信（未加密），可能泄露节点信息。建议保持默认的 TLS 配置。 |

### ⚠️ 潜在风险

1. **Future Ray Compatibility**：Monkey‑patch覆盖 `list_nodes` 可能在 Ray ≥ 2.10 中被移除或改名，导致运行时 `AttributeError`。  
2. **资源争用**：如果用户在同一 Pod 中同时启动多个伸缩请求，可能出现 DP size 超出预留 GPU 数的冲突，触发 `RayActorError` 或 OOM。  
3. **状态不一致**：在伸缩过程中若出现异常（网络抖动、GCS 超时），可能导致 `event_channels` 被写回 `null`，虽然已加入 `null`‑tolerant 解析，但仍会导致短暂的发现盲点。  
4. **安全暴露**：未对新 API 加强鉴权，开放给外部网络时可能被滥用进行资源耗尽攻击。  
5. **性能回弹**：频繁伸缩会导致 GPU 内存碎片，尤其在使用 `--gpu-memory-utilization` 较高的模型时，可能出现分配失败。  

---

### 💡 关注建议

1. **兼容性检测**：在启动时检测 Ray 版本；若 `ray.nodes` 已提供兼容实现，可直接使用而不覆盖 `list_nodes`，或在升级后移除该补丁。  
2. **输入校验与配额**：在 `scale_elastic_ep` 前加入对 `new_data_parallel_size` 与当前 Pod `gpu` 配额的交叉检查；可考虑配置最大伸缩上限防止误操作。  
3. **审计日志**：将伸缩请求写入统一审计日志（包括调用者、时间、目标 size），配合已有的 Dynamo 监控体系。  
4. **回滚机制**：在 `scale_elastic_ep` 失败时，确保已启动的临时 DP Worker 被安全回收，防止残留进程占用资源。  
5. **监控指标**：新增 Prometheus 指标 `dynamo_vllm_elastic_ep_target_dp`、`dynamo_vllm_elastic_ep_current_dp`，以及伸缩延迟 `dynamo_vllm_elastic_ep_scale_latency_seconds`，便于运维观察伸缩行为。  
6. **文档升级**：在官方文档中明确说明：*（a）必须在容器镜像中预留全部目标 GPU；*（b）使用 Ray backend 必须开启 `VLLM_USE_ELASTIC_EP=1`；*（c）推介在生产环境使用 `pplx` all‑to‑all 后端以获得更好跨节点扩展性。  
7. **测试覆盖**：将 `run_elastic_ep_scale_test.sh` 纳入 CI，使用集成测试框架跑完整的 6 步伸缩序列，确保在代码改动后仍能保持零错误。  

--- 

**结论**：本次提交为 Dynamo 与 vLLM 的深度集成引入了弹性 EP 伸缩能力，极大提升了资源利用率与运维灵活性。核心风险集中在对 Ray 内部实现的临时性补丁、伸缩过程的资源冲突以及新 API 的安全防护上。通过上述建议可进一步降低风险并确保功能在生产环境的可用性与可观测性。

总体来看，此次改动通过集中光标状态管理，提升了恢复路径的可读性、可维护性以及错误处理的鲁棒性。后续如需在其它模块（例如统计或采样）复用光标逻辑，只需要引用 `kv-router::recovery` 即可。
