### feat(DGH-706): Phase 3 — disaggregated topology readiness (#9815)
**SHA**: `cfc3035` | 🔗 [查看提交](https://github.com/ai-dynamo/dynamo/commit/cfc303534c0a2ce2dde0b899618581207663d7e2)

**🎯 变更类型**：功能增强 / 架构变更 / 安全修复  
**⚡ 重要程度**：🔴高  

**📋 变更摘要**  
本次 PR（Phase 3）为 Dynamo 引入 **`WorkerType`** 枚举，彻底把「前缀/Decode」的拓扑角色从 `ModelType::Prefill` 中抽离，改为显式的 `worker_type`（Prefill、Decode、Encode、Aggregated）+ `needs`（DNF 依赖）。  
- 注册层面：`register_model` 必须携带 `worker_type`，并对 `Prefill/Encode` 强制要求 `ModelInput::Tokens`，不再允许在 Prefill 上暴露 OpenAI Surface。  
- 发现层面：WorkerSet 键改为 `{namespace}:{model_type}:{worker_type}`，并在旧卡缺失 `worker_type` 时使用兼容 shim（把带 `Prefill` 标记位的旧 pre‑fill 视为 `Prefill`，其他视为 `Aggregated`）。  
- 可用性层面：新增模型级 **拓扑可用性** 检查（`Model::is_workers_ready`），在 HTTP OpenAI 接口入口统一返回 **503 Service Unavailable** 并带诊断体；`GET /v1/models` 仅列出至少存在一个 **拓扑完整**（ready）的命名空间的模型。  
- 生态层面：所有后端（vLLM、TRT‑LLM、SGLang、Omni、示例后端等）以及测试、文档同步迁移到新 API；`ModelType::Prefill` 被标记为 legacy，仅用于跨版本兼容的双发射。  

**🎯 影响范围**  
- **核心库**：`lib/llm`（模型卡、发现、调度、入口、绑定层）  
- **Python 绑定**：`lib/bindings/python`（`register_model` 签名、`EntrypointArgs`、`is_decode` 标记）  
- **后端实现**：`components/src/dynamo/*`（vLLM、TRT‑LLM、SGLang、thunderagent、global_router、sample、triton、examples 等）  
- **前端/Router**：模型列表、请求入口、KV‑router、PrefillRouter 的激活/注销逻辑  
- **文档/提案**：新增健康检查方案、README、router‑guide、示例脚本等  
- **测试套件**：单元/集成测试全部迁移至新字段，新增拓扑可用性断言  

**🔍 技术洞察**  

- **架构影响**  
  1. **WorkerSet Key 重构**：从 `{ns}` / `{ns}:prefill` 迁移到三元组 `{ns}:{model_type}:{worker_type}`，实现了 **真正的多角色分区**，解决了 encode worker 与 decode worker 在同 namespace 冲突的问题。  
  2. **拓扑可用性模型**：在 `Model` 中实现 DNF 依赖解析，能够在 **运行时** 动态判断每个 namespace 是否满足所有 `needs`，并把结果用于 503 报错、模型列表过滤以及后续的 **按命名空间调度**（Phase 4 计划）。  
  3. **兼容层**：保留旧卡 `worker_type=None` 的 shim，针对 legacy `ModelType::Prefill`（marker 位）自动映射为 `Prefill`，并在首次触发时记录一次 `warn_legacy_readiness_once`，保证平滑滚动升级。  
  4. **后端统一注册**：所有后端现在统一通过 `register_model(..., worker_type, needs)` 生成 MDC，编码、推理、LoRA 等路径均对 `worker_type` 与 `needs` 做显式声明，避免过去的暗码（`ModelType::Prefill`）产生歧义。  

- **性能影响**  
  - **请求路径**：每一次 OpenAI 接口会调用 `Model::is_workers_ready`，该函数遍历模型的 `worker_sets`（通常 1~3 条）并做集合运算，**时间复杂度 O(N)**，且 N 极小，几乎可以忽略。  
  - **注册与发现**：新增 `worker_set_key` 计算及 `effective_worker_type` 判定带来极低的 CPU/内存开销；对 etcd/etcd‑watcher 的键空间改动不会产生显著负载。  
  - **潜在回归**：在高 QPS 场景下，**每请求一次拓扑检查** 仍会产生一定的锁竞争（`ModelManager` 中的 `DashMap` 读取），建议后续（Mechanism 3）实现 **ready‑instance池** 来把判断结果缓存至 O(1) 调度。  

- **安全考虑**  
  - **输入校验加强**：`register_model` 现在在 Rust 端强制 `worker_type` 非空、Prefill/Encode 必须使用 `ModelInput::Tokens`、Prefill 不得携带任何 OpenAI surface（`ModelType` 必须为 `Empty` 或仅保留 legacy marker位）。这防止了误配置导致的 **非法暴露接口** 或 **未授权的 KV 传输**。  
  - **错误信息泄漏**：503 响应体会返回缺失的 worker 类型，信息对运维友好且不泄露内部实现细节，属于安全可接受范围。  
  - **兼容 shim**：对旧卡的 fallback 只在日志层面提示，未对外暴露内部版本信息，安全风险低。  

**⚠️ 潜在风险**  

| 风险点 | 说明 | 可能后果 |
|--------|------|----------|
| **旧二进制仍在运行** | 仍有未升级的 worker 未携带 `worker_type`，会走兼容 shim 并**隐藏模型**（因为 `list_models` 过滤）或误报可用（只要有存活 worker）。 | 业务方在滚动升级期间可能看到模型突现/突消，导致请求 503 或流量错发到不完整 namespace。 |
| **`ModelType::Prefill` 完全移除** | 任何残留代码仍使用 `ModelType::Prefill` 进行 surface 注册将编译失败或在运行时被拒绝。 | 部署脚本、实验代码或第三方插件因未同步改动

### feat: Atn-dp fpm upstream integration (#9059)
**SHA**: `f52e2b2` | 🔗 [查看提交](https://github.com/ai-dynamo/dynamo/commit/f52e2b2bf133add2c2c86dfd8336d0196bb310ed)

**🎯 变更类型**：功能增强  

**⚡ 重要程度**：🔴 高  

**📋 变更摘要**  
1. 将 ForwardPassMetrics（FPM）发布从 “仅在非‑attention‑DP（单 rank）时开启” 改为在 **所有 attention‑DP 场景下均开启**，并为每个 attention‑DP rank 创建独立的逻辑通道。  
2. 在 `Publisher.__init__` 与 `initialize` 中移除旧的 `self.fpm_enabled` 判断，改为始终实例化 `FpmDirectPublisher`（dp_size 为实际 attention‑DP 大小），并在异常时安全降级。  
3. 更新统计处理逻辑：`attentionDpRank` 直接映射为 FPM 的 `dp_rank`，并在缺失或为 `None` 时默认 `0`。  
4. 同步更新单元测试，验证多 rank 场景下的队列/调度字段不被错误复制、默认行为以及异常路径。  

**🎯 影响范围**  
- `components/src/dynamo/trtllm/publisher.py`（核心指标发布模块）  
- `components/src/dynamo/trtllm/tests/test_trtllm_fpm_publisher.py`（对应单元测试）  
- 依赖 `FpmDirectPublisher` 的所有 Dynamo 工作节点（尤其是使用 TRT‑LLM Attention‑DP 的部署）  

**🔍 技术洞察**  

| 维度 | 影响描述 |
|------|----------|
| **架构影响** | • **从单通道 → 多通道**：FPM 现在按 attention‑DP rank 进行逻辑分片，Planner 可以获取每个 rank 的真实负载，避免 “planner poison” 场景。<br>• **去除 `self.fpm_enabled` 标记**，统一初始化路径，降低代码分支复杂度。<br>• **向上兼容**：对非‑attention‑DP（dp_size=1）行为保持不变，仅在 `dp_size>1` 时多通道生效。 |
| **性能影响** | • **更精准的负载报告**：Planner 能基于每个 rank 的实际调度/排队数做调度决策，提升整体系统吞吐与延迟。<br>• **额外的 Python‑Rust 边界调用**：每个 rank 创建独立的 `FpmDirectPublisher`，在高并发 DP 环境下会产生 `dp_size` 次初始化及心跳维持，开销与原先的单通道相比线性增长，但通常 `dp_size` ≤ 8，开销可接受。<br>• **错误路径已捕获**：初始化失败只会记录警告并将 `fpm_publisher` 置 `None`，不会导致整个节点崩溃。 |
| **安全考虑** | • 仅涉及监控数据的收集与上报，无敏感业务数据泄露风险。<br>• `FpmDirectPublisher` 的实例化仍受原有的 endpoint 权限校验约束，未引入新网络入口。<br>• 对异常（`RuntimeError`）的捕获防止了因为内部 PyO3 错误导致的未捕获异常传播。 |
| **可维护性** | • 删除了旧的 `self.fpm_enabled` 变量和对应的分支，代码路径更简洁。<br>• 添加了明确的注释，解释 attention‑DP rank 如何映射到 FPM。<br>• 单元测试覆盖了缺失/为 `null` 的 `attentionDpRank`、多 rank 数据分发以及初始化异常两大情形，提升回归安全性。 |

**⚠️ 潜在风险**  

1. **初始化并发冲突**：在极端大规模 DP（例如 `dp_size > 16`）时，短时间内创建多个 `FpmDirectPublisher` 可能触发底层 Tokio runtime 初始化竞争，导致偶发的 `RuntimeError`。  
2. **心跳/空闲报告激增**：每个 rank 均会产生 1 s 心跳；若监控后端未针对多通道做限流处理，可能出现短期流量激增。  
3. **旧版本兼容性**：若用户的 Planner 仍假设单通道 FPM（仅 rank 0），收到多 rank 报文后可能误判负载，需要同步升级 Planner 或在后端做聚合。  
4. **测试依赖的 Mock 行为**：单元测试中对 `FpmDirectPublisher` 的 mock 只验证一次实例化；实际运行时仍会创建 `dp_size` 个通道，若底层实现在未来更改（如每 rank 独立进程），需要相应更新测试。  

**💡 关注建议**  

- **部署指南更新**：在使用 TRT‑LLM Attention‑DP 时，明确说明 Planner 必须支持 per‑rank FPM，以防出现负载误判。  
- **监控后端适配**：确认后端（例如 Dynamo Planner）能够接受并聚合多 rank 的 FPM 消息，或在入口层加入 `dp_rank` 聚合逻辑。  
- **异常监控**：在生产环境中打开 `FpmDirectPublisher` 的初始化日志级别（INFO），以便快速定位可能的 `RuntimeError`（如 endpoint 未就绪）。  
- **资源预估**：针对大规模 DP 部署，评估每个 rank 心跳对网络与后端处理的影响，必要时在配置中提供 `fpm_heartbeat_interval` 参数（若底层支持）。  
- **回滚路径**：保留 `self.fpm_enabled` 旧实现的注释或分支，以便在紧急回滚时可以快速恢复到单通道模式。  

---  

*此变更为 Dynamo 在多‑attention‑DP 场景下提供更细粒度的负载监控，提升调度质量，但需要确保下游 Planner 与监控后端同步升级以完整利用新特性。*

### refactor(mm-routing): unify on canonical pad_value, drop backend flags (#10328)
**SHA**: `aa0e1b9` | 🔗 [查看提交](https://github.com/ai-dynamo/dynamo/commit/aa0e1b90b6b92f8128c4edbe85151a6ca9b3dad1)

**🎯 变更类型**：重构 / 架构变更  
**⚡ 重要程度**：🔴高  

**📋 变更摘要**  
此次提交统一了多模态路由的实现方式：所有后端现在使用 **canonical `pad_value`** 方案，在路由 token 流中直接用 `pad_value(mm_hash)` 代替图像占位符，彻底废弃了原来的 `block_mm_infos` side‑channel 以及 `backend_framework` 标记。为此在 Python 与 Rust 两侧同步了常量 `MM_PAD_SHIFT_VALUE`、`MM_PAD_HASH_MASK`，并在 KV‑router、ZMQ 归一化、vLLM 适配层等关键路径中传递模型的 **image placeholder token id**，使得 vLLM 事件也可以在发布阶段被归一化为与前端完全一致的 `pad_value` 哈希。  

**🎯 影响范围**  
- `components/src/dynamo/common/multimodal/routing_utils.py`（核心路由工具）  
- 单元测试 `test_routing_utils.py`、`test_vllm_worker_handler.py`、`kv-router` 相关测试  
- `components/src/dynamo/frontend/vllm_processor.py`（前端调用）  
- `components/src/dynamo/sglang/register.py`、`components/src/dynamo/vllm/main.py`（后端注册）  
- `components/src/dynamo/vllm/handlers.py`、`vllm/main.py`（vLLM 参数处理）  
- Rust 绑定层 `lib/bindings/python/rust/*`（`resolve_routing_image_token_id`、`KvEventPublisher` 参数）  
- KV‑router 核心 `lib/kv-router/src/protocols.rs`、`zmq_wire/convert.rs`、`zmq_wire/mod.rs`（哈希、事件归一化）  
- 运行时配置结构 `ModelRuntimeConfig`（删除 `backend_framework`）  
- 预处理模块 `lib/llm/src/preprocessor.rs`（MM‑routing 逻辑统一）  

**🔍 技术洞察**  

- **架构影响**  
  1. **统一路由协议**：原先依赖 `runtime_config.backend_framework` 区分 SGLang 与 vLLM 的两套协议（pad‑value vs. 64‑char hash + `block_mm_infos`）被合并为单一的 `pad_value` 方案。后端仅需要提供 **image placeholder token id**，其余工作均由 KV‑router 在 `convert_event` 中完成。  
  2. **消除 side‑channel**：`block_mm_infos` 不再产生，从 `KvEventPublisher` → `ZmqEventNormalizer` → `convert_event` 的数据流更简洁，减少了跨语言结构的同步点。  
  3. **常量统一**：`MM_PAD_SHIFT_VALUE`、`MM_PAD_HASH_MASK` 以及 `pad_value_for_mm_hash` 在 Rust 与 Python 中保持同一实现，防止协议漂移。  
  4. **新绑定函数** `resolve_routing_image_token_id` 让 Python 端在启动时直接获取模型对应的 image placeholder token，确保前后端使用同一 token。  

- **性能影响**  
  - **降低内存/CPU 开销**：不再为每个 KV‑block 构造 `BlockExtraInfo`，`block_mm_infos` 被裁剪为 `[]`，在 `compute_block_hash_for_seq` 时可以直接跳过 `block_mm_infos` 参数，哈希过程更轻量。  
  - **额外的 token 替换成本**：在 `create_stored_block_from_parts` 中加入 `substitute_pad_values`，遍历一次 token 序列进行 run‑level替换，复杂度 O(N)，但 N 为 KV‑block 大小（数百），对整体吞吐影响可忽略。  
  - **ZMQ 事件路径**：`ZmqEventNormalizer` 需要保存 `image_token_id` 并在每帧事件归一化时调用 `substitute_pad_values`，引入一次额外函数调用，但同样是线性遍历，几乎不影响延迟。  

- **安全考虑**  
  - **攻击面缩减**：删除 `backend_framework` 配置字段后，前端不再根据外部字符串决定路由策略，避免因非法或错误的框架标识导致意外的路由路径或信息泄露。  
  - **哈希一致性**：统一 `pad_value` 公式后，前后端哈希完全相同，避免因协议不一致产生的 “hash 冲突/误匹配” 导致的潜在数据完整性问题。  
  - **输入校验**：`_pad_mm_hashes_to_64` 在 vLLM 侧对 mm_hash 长度进行补齐，确保不会因长度不足触发解析异常。  

**⚠️ 潜在风险**  

1. **兼容性回退**：旧的后端（或自定义插件）仍可能依赖 `block_mm_infos` 或 `backend_framework`，在未同步更新的情况下会收到空 `block_mm_infos`，导致路由错误或回退到文本前缀。  
2. **常量漂移**：`MM_PAD_SHIFT_VALUE`、`MM_PAD_HASH_MASK` 必须与 SGLang 实现保持同步；若上游 SGLang 修改了这些常量而本仓库未同步，路由哈希将不匹配，表现为“看似正常但检索不到图像”。  
3. **image_token_id 未解析**：`resolve_routing_image_token_id` 可能返回 `None`（模型未注册或缺少配置），此时 `substitute_pad_values` 不会执行，导致 vLLM 事件仍携带原始 placeholder token，进而在 KV‑router 侧产生不一致的 hash。  
4. **外部依赖**：`ModelRuntimeConfig` 删除了 `backend_framework`，如果外部代码（例如用户自定义的监控或日志工具）仍读取该字段，会触发 `AttributeError`。  
5. **测试覆盖不足**：虽然已更新单元测试，但对跨进程、跨节点的端到端路由一致性（尤其在分布式 DP 场景）缺少集成测试，潜在的同步错误难以及时发现。  

**💡 关注建议**  

- **迁移指南**：在发布说明中明确指出所有自定义后端需要移除对 `block_mm_infos` 与 `backend_framework` 的依赖，改为使用 `image_token_id` 与统一 `pad_value`。  
- **常量同步机制**：建议在 CI 中加入对 SGLang 仓库 `MultimodalItem._compute_pad_value` 常量的自动比对，以防未来上游改动未同步。  
- **兼容层 fallback**：在关键路径（如 `create_stored_block_from_parts`）保留对旧 `block_mm_infos`

### fix(kv-router): harden cancellation and replica sync (#10331)
**SHA**: `f68be6d` | 🔗 [查看提交](https://github.com/ai-dynamo/dynamo/commit/f68be6d409efb3ac3eb6874bebcd3077a9c131e6)

**🎯 变更类型**：功能增强 / Bug 修复 / 性能优化 / 重构 / 安全修复  
**⚡ 重要程度**：🔴 高  

**📋 变更摘要**  
本次提交对 KV‑Router 的调度与副本同步逻辑做了重大硬化：  
1. **调度取消安全**：在 `SchedulerQueue::admit_one` 中先检查响应接收方是否已关闭，若已关闭则跳过状态预订；若在预订后响应投递失败，则回滚已预订的调度状态，防止“幽灵”占用。  
2. **统一错误类型**：新增 `KvSchedulerError::BookingFailed`，统一报告预订失败。`SchedulingRequest::respond` 改为返回布尔值，区分成功/被丢弃的情况。  
3. **副本同步（replica‑sync）**：抽离为独立模块 `replica_sync.rs`，实现 **批量** 事件处理、负载发布合并、及时唤醒调度器以及 PromptTrie 清理。通过 `poll_next_event` 与 `next_event` 双通道实现无阻塞的事件拉取与取消。  
4. **请求生命周期管理**：新增 `request_guard.rs`，将调度状态释放、会话关闭、指标记录等职责封装为 **RAII** Guard，确保即使在流提前 drop、取消或错误路径也能安全回收资源。  
5. **选择逻辑抽离**：`selection.rs` 把工作者选取、固定‑worker 解析、后备‑pin、路由哈希记录等拆分，代码结构更清晰、单元可测。  
6. **其它改动**：`publish_load_batch`、`publish_load` 合并、`SequenceSubscriber::poll_next_event` 新增以支持非阻塞轮询；`PromptRegistry` 增加清理计数用于测试验证。

**🎯 影响范围**  
- `components/src/dynamo/router/CLAUDE.md`、`lib/kv-router/src/scheduling/*`、`lib/kv-router/src/sequences/*`、`lib/llm/src/kv_router/*`（尤其是 `push_router.rs`）以及运行时事件层 (`runtime/src/transports/event_plane`)。  
- 所有使用 KV‑Router 进行调度的入口（Python Stand‑alone router、embedded Rust pipeline）都将受益。  
- 依赖 `KvRouter` 的业务方（如 LLM 服务、分布式推理网关）将直接感受到行为变化。

**🔍 技术洞察**  

| 维度 | 影响说明 |
|------|----------|
| **架构** | - 将原本散落在 `push_router.rs` 与调度队列中的 “取消/回滚” 逻辑抽离为独立组件，降低模块耦合度。<br>- `ReplicaBatchEffects` 引入批处理层，显式区分 **状态预订** 与 **负载发布**，提升副本同步的可扩展性。<br>- `RequestGuard` 采用组合模式（`RequestCleanup` + `RequestObservability`），实现了 **RAII** 资源管理，确保即使在 `Drop` 期间也能安全异步清理。 |
| **性能** | - 在调度路径中提前检测 `resp_tx.is_closed()`，避免无意义的状态预订，降低调度延迟与内存占用。<br>- 副本同步现在 **合并** 同一批次的负载更新（`publish_load_batch`），相较于每条事件单独 publish，显著降低 NATS/事件总线的网络压制和系统调用开销。<br>- `MAX_REPLICA_BATCH_EVENTS = 256` 与 `MAX_REPLICA_BATCH_DURATION = 1 ms` 让同步在高频率情况下保持低延迟，同时在低负载时仍能及时唤醒调度器。 |
| **安全** | - 通过统一的 `BookingFailed` 错误返回，调度系统不再出现 “状态漂移” 导致的资源泄露，降低 DoS 风险。<br>- 取消路径使用 `cancel_on_stop` 统一检测 `AsyncEngineContext::stopped()`，防止在请求被关闭后仍继续执行底层 RPC，避免潜在的泄露或未授权的后端调用。 |
| **可维护性** | - 代码划分为 `cancellation.rs`、`request_guard.rs`、`selection.rs`、`replica_sync.rs` 四个职责明确的文件，单元测试覆盖率大幅提升（新增大量 `#[tokio::test]`）。<br>- `SequenceSubscriber::poll_next_event` 的引入，使得 event 拉取可以在 `select!` 循环中非阻塞地检测取消，整体流程更易理解。 |
| **兼容性** | - 对外 API（`KvRouter::find_best_match`、`KvRouter::record_routing_decision` 等）保持不变，唯一的破坏性改动是 `SchedulingRequest::respond` 现在返回 `bool`，但该方法仅在内部使用，外部不受影响。 |

**⚠️ 潜在风险**  
1. **异步 Drop 产生竞争**：`RequestCleanup::drop` 在没有 Tokio runtime 的上下文中会直接放弃清理，若系统在某些线程上未启动 Runtime（例如自行创建的线程池），可能导致调度状态和 Session KV 没被释放。  
2. **批处理边界丢失**：在极端高频副本事件下，如果 `MAX_REPLICA_BATCH_DURATION` 触发过早，可能出现“未发布的负载”被丢弃（但实现已在 `flush_replica_batch_effects` 中保证即使未达到事件数也会在超时后立即发布）。仍需监控实际批次大小，防止负载统计偏差。  
3. **错误路径的错误传播**：`cancel_on_stop` 会把取消错误包装为 `DynamoError::Cancelled`，如果调用方未显式处理这种错误，可能导致请求被误认为是普通失败而计入错误率统计。  
4. **测试依赖的内部计数**（如 `PromptRegistry::cleanup_attempts`）在生产环境中仍会计数，产生微小的额外原子操作开销。  

**💡 关注建议**  
- **监控批处理指标**：在生产中开启 `replica_sync` 的监控（批次数、最长批时长、发布负载条数），确保 `MAX_REPLICA_BATCH_*` 参数仍适配实际流量。  
- **确保所有工作线程都有 Tokio Runtime**：在自定义线程池或 C++ 绑定层启动时，显式创建 `tokio::runtime::Handle`，以免 `RequestCleanup::drop` 因缺少 runtime 而失效。  
- **审计取消错误的统计口径**：在监控仪表盘中区分 `Cancelled` 与 `ResourceExhausted` 等错误，防止误报。  
- **回归测试**：针对调度取消场景（响应接收方提前关闭、`select_worker` 期间取消）进行端到端压测，验证预约回滚与 `remote_state_update` 的正确性。  
- **文档更新**：在用户手册中说明 “请求被取消后仍会自动释放 scheduler slot 与 session KV”，帮助运维理解资源回收的时序。  

> 综合来看，此次改动大幅提升了 KV‑Router 在高并发、频繁取消以及多副本同步环境下的鲁棒性，架构更清晰、性能更优且安全风险得到降低，推荐尽快在所有生产集群上推广。

**🎯 变更类型**：Bug 修复（runtime）  
**⚡ 重要程度**：🟡 中  

**📋 变更摘要**  
为直接 ZMQ 事件订阅器引入有界的合并事件通道，默认容量 100 000（匹配 ZMQ_RCVHWM），通过环境变量 `DYN_ZMQ_EVENT_SUBSCRIBER_CHANNEL_CAPACITY` 可调。改为 `mpsc::channel` 并在满时 `try_send` 丢弃事件，避免之前出现的 RSS 暴涨（单节点 ~80 GiB）。

**🎯 影响范围**

- `runtime::transports::event_plane::dynamic_subscriber`（通道构造、发送逻辑）  
- 相关测试文件中加入新 env 的检查  

**💡 关注建议**  
1. **文档与部署**：在 README/部署指南中说明该 env 的意义、默认值及调优建议，防止运维误以为丢弃事件会导致错误。  
2. **监控**：利用已有的 `tracing::trace` 日志监控 “channel full” 情况，必要时在监控平台报警，以判断是否需要提升容量。  
3. **兼容性**：老版二进制仍使用默认 100 000，保持行为一致；若用户显式设为 0，仍会回退默认，避免 panic。  
4. **测试**：增加高并发发布/消费测试，验证在容量被占满时不会出现内存泄漏或死锁。  
5. **性能评估**：在 replica‑sync 场景下跑一次基准，对比内存峰值和事件延迟，确保 “丢弃” 对路由估计的影响在可接受范围内。  

总体而言，此改动显著降低了因无界通道导致的内存爆炸风险，唯一副作用是极端负载下会出现少量事件丢失，属可接受的最佳‑努力模型。请在生产部署前评估实际发布速率与默认容量的匹配度。
