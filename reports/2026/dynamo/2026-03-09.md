### feat(skill): add main branch cross-reference to pr-monitor (#6889)
**SHA**: `3e69e37` | 🔗 [查看提交](https://github.com/ai-dynamo/dynamo/commit/3e69e37f281137850614764e14bfc5a45b184047)

**🎯 变更类型**：功能增强（文档层面的技能扩展）  
**⚡ 重要程度**：🟡 中  

**📋 变更摘要**  
在 `.claude/skills/pr-monitor/SKILL.md` 中新增 **“Step 3b: Main Branch Cross‑Reference”**，为 PR‑monitor skill 添加了通过 GitHub API 对比 PR 与 `main` 分支最近 CI 结果的步骤。该步骤帮助区分 PR 引入的实际回归与已有的 flaky/已修复问题，并在报告中给出 `[PR‑CAUSED]、[PRE‑EXISTING]、[UNCLEAR]、[NEW JOB]` 四类标记及相应建议。

**🎯 影响范围**  
- **文档/技能**：仅修改了 `pr-monitor` 的 Markdown 文档，未触及代码。  
- **CI 使用者**：依赖 `pr-monitor` 的团队会在生成报告时得到更细致的回归定位信息。  

**💡 关注建议**  

1. **验证脚本兼容性**：新增的 Bash/gh‑cli 调用假设 CI 环境已预装 `gh` 并具备相应的 token 权限，使用前请在本地或 CI 中做一次 dry‑run，避免因权限或速率限制导致报告生成失败。  
2. **处理大幅落后 PR**：当 `behind_by > 20` 时报告中会提醒“可能已在 main 修复”，建议在实际使用时加入自动提示：若落后较多，可自动建议 rebase，防止误判。  
3. **结果去重与超时**：`MAIN_RUNS` 只取最近 3 次 workflow，从而降低 API 调用次数；如果项目 CI 频繁触发，可考虑将 `per_page` 或 `status=completed` 参数调高，以确保始终拿到有效的比较数据。  
4. **文档同步**：若后续对 `pr.yaml` 工作流或 job 命名进行更改，需要同步更新此文档中的示例脚本，否则可能出现 “job not found” 的误报。  

总体来说，此次改动对功能本身没有代码层面的风险，主要提升了 PR‑monitor 的可用性和定位准确度。建议在正式部署前在几个真实 PR 上跑通，确认 `gh` 命令的返回结构与文档保持一致，再向团队推广。

### chore: rename chrek to Dynamo Snapshot (#7028)
**SHA**: `6831020` | 🔗 [查看提交](https://github.com/ai-dynamo/dynamo/commit/6831020f35c4eba350b40616bf721922420bc010)

**🔧 变更类型**：功能增强（名称统一 & 代码整洁）  
**⚡ 重要程度**：🟡 中——涉及跨模块的大量符号重命名，需完整回归验证。  

**📋 变更摘要**  
本次提交将原先的 **“chrek”**（Checkpoint/Restore）全部更名为 **“snapshot”**，包括：  
- 核心代码中 `checkpoint_restore_engine` ➜ `snapshot_engine`，相关函数签名统一改为 `handle_checkpoint_mode` → `handle_snapshot_mode`（实际保持同名）。  
- SGLang 与 vLLM 入口层的参数、调用及文档同步修改。  
- Helm Chart、Operator 配置、PV‑PVC 名称、K8s 标签、Env/日志变量等全部改为 `snapshot‑*`。  
- 新增 `deploy/snapshot` Go module，迁移原 `chrek` 包路径、Dockerfile、Makefile、CI 脚本等。  
- 文档、README、API reference 统一更换为 snapshot 相关链接与示例。  

**🎯 影响范围**  
- `components/src/dynamo/sglang/*`、`components/src/dynamo/vllm/*`（运行时初始化、worker factory）  
- `deploy/operator/*`（CRD 默认、常量、注入逻辑）  
- `deploy/helm/charts/*`（Chart 名称、模板、values、标签、PVC 名称）  
- `deploy/snapshot/*`（全新 Go module）  
- 所有单元测试、CI 脚本与文档。  

**💡 关注建议**  
1. **兼容性**：如果已有用户仍在使用旧的 `chrek` Helm chart 或 PVC 名称，建议保留一个短期别名（如 `chrek-pvc → snapshot-pvc`）或在升级指南中明确迁移步骤。  
2. **CI 全链路**：确保 `go test ./...`、`cargo`（如有）以及 Helm lint、helm install/upgrade 流程全部通过；重点检查跨语言（Python、Rust）调用的参数签名是否已同步。  
3. **文档同步**：所有外部链接（GitHub、Helm repo）均已指向 `snapshot`，但仍需更新外部站点或 CI 生成的文档索引。  
4. **版本号**：因 API/CRD 变更，建议在 Operator 与 Chart 中提升 major/minor 版本，以防用户误升级导致资源不可用。  
5. **回滚准备**：保留 `chrek` 分支或在代码中加入 `// DEPRECATED` 注释，便于快速回滚或提供迁移脚本。  

整体来看，改名工作已覆盖主要实现路径，关键在于确保所有依赖（尤其是 Helm + Operator 的 PVC/Label）保持一致，并提供清晰的升级指引。
