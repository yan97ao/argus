### feat: request lifecycle logging + metrics capture [DIS-1643] (#7840)
**SHA**: `679d9f1` | 🔗 [查看提交](https://github.com/ai-dynamo/dynamo/commit/679d9f145236659e11e25286b9849d874e59a4c0)

**🎯 变更类型**：功能增强  
**⚡ 重要程度**：🔴高  
**📋 变更摘要**  
本次提交在整个请求生命周期中引入了统一的 **request‑id** 传播、详细的日志记录以及更丰富的指标采集。新增 `InflightGuard` 与 `ResponseMetricCollector` 的字段（如 `request_id、input/output token 长度、TTFT、ITL`），并在网络层的入口/出口、所有 HTTP 与 gRPC 服务实现中将 `request‑id` 通过 headers 传递至业务层，确保每一次请求的开始、错误、完成都能在 tracing 与 Prometheus 中完整体现。

**🎯 影响范围**  
- **核心层**：`lib/llm/src/http/service/*`, `lib/llm/src/grpc/service/*`  
- **指标模块**：`lib/llm/src/http/service/metrics.rs`（InflightGuard、ResponseMetricCollector）  
- **运行时网络层**：`lib/runtime/src/pipeline/network/*`（PushWorkHandler、Ingress/PushEndpoint）  
- **测试**：`lib/llm/tests/http_metrics.rs`、`lib/runtime/src/pipeline/network/*` 单元测试  

**🔍 技术洞察**  

| 维度 | 影响 |
|------|------|
| **架构影响** | - 在所有入口点统一引入 `request-id`，形成跨层（Ingress → Service → Metrics → Tracing）的 **统一请求上下文**，提升可观测性。<br>- 通过在 `PushWorkHandler`、`Ingress` 等抽象层加入 `request_id` 参数，保持向后兼容（默认 `None`），对已有实现影响最小。<br>- `InflightGuard` 现在负责 **日志注入、计数、错误捕获**，并在 `Drop` 时统一记录完成日志，简化业务代码。 |
| **性能影响** | - 每个请求额外分配并复制一次 `String`（request‑id），对毫秒级请求几乎不可感知；在高 QPS 场景下可通过使用 `Arc<str>` 或 `Uuid` 轻量化进一步优化。<br>- 额外的 tracing 记录（`info!`/`error!`）在 `INFO` 级别启用时会产生少量 CPU 与 I/O 开销，建议在生产环境使用 `WARN`/`ERROR` 或开启采样。<br>- 新增的指标（TTFT、ITL、输入/输出 token 长度）使用已注册的 Prometheus `Histogram` 与 `Counter`，对监控端的聚合开销有限。 |
| **安全考虑** | - `request-id` 属于内部追踪标识，默认不包含用户数据，泄露风险低。但若上层业务自行将业务信息拼接进 `request-id`，需做好 **脱敏**。<br>- 通过 HTTP headers 传播 `request-id` 时，需确保 **Header 注入** 不被外部篡改（已使用固定名称 `request-id` / `x-dynamo-request-id`，可在网关层做签名或校验）。 |
| **可维护性** | - 统一的 `InflightGuard` 与 `ResponseMetricCollector` 使得 **日志、错误、指标** 的统一管理更易维护。<br>- 代码中多处 `create_inflight_guard` 调用已更新为新签名，若后续继续添加额外维度，只需在该工厂函数处统一更新。<br>- 为保证向后兼容，所有测试已适配新增参数，降低回归风险。 |

**⚠️ 潜在风险**  

1. **遗漏 `request-id`**：某些自定义 `PushWorkHandler` 实现或外部代理未传递 `request-id`，导致日志与指标出现 “空” 标识，影响追踪完整性。  
2. **日志噪声**：在 `INFO` 级别下每个请求都会产生两条日志（开始/完成），在高流量场景可能导致日志存储激增。  
3. **内存占用**：`InflightGuard` 持有 `String`，在极端并发（数万并发请求）下累计占用的内存会略有提升。  
4. **误用 Header**：若外部客户端故意或误将敏感信息写入 `request-id`，会被原样记录在日志与监控系统，产生泄露风险。  
5. **向后兼容**：旧版插件或自行实现的 `PushWorkHandler` 若未更新签名，会在编译阶段报错；需要在文档中明确迁移指南。

**💡 关注建议**  

- **迁移指南**：在项目文档中标注 `create_inflight_guard` 新签名以及 `handle_payload` 的 `request_id` 参数，提供示例代码。  
- **默认值**：在 `create_inflight_guard` 中对空 `request_id` 使用 `"<none>"` 或生成 UUID，避免出现空字符串导致的日志分片。  
- **日志采样**：在生产环境将 `request` 相关日志级别调至 `WARN`，或使用 OpenTelemetry/Jaeger 的采样策略，仅保留关键错误路径。  
- **安全审计**：在网关层对 `request-id` 做白名单校验（长度、字符集），防止注入恶意内容。  
- **性能监控**：监控新增指标（`ttft_ms`, `avg_itl_ms`）的收集频率及 Prometheus 存储使用，确保不因过细的 Histogram 桶导致磁盘压力。  
- **回归测试**：增加对 `request-id` 传播链路的端到端测试，验证在异常、超时、取消等路径下日志与指标均能完整记录。  

通过上述改动，系统在 **可观测性、错误追踪和业务洞察** 上获得显著提升，同时只带来了可接受的性能与复杂度增长。后续关注日志噪声、内存占用以及 Header 安全即可确保改动的平稳运行。

### refactor(vLLM): Move video support from example to backend (#7663)
**SHA**: `4791aaa` | 🔗 [查看提交](https://github.com/ai-dynamo/dynamo/commit/4791aaaa2ca209ecaad82f872832081fa3e43470)

**🎯 变更类型**：重构 / 架构变更  
**⚡ 重要程度**：🔴高  

**📋 变更摘要**  
- 将视频多模态支持从旧的 *examples/multimodal* 示例实现中抽离，迁移至核心后端 `dynamo.vllm`。  
- 新增 `VideoLoader` 负责使用 vLLM 的 `MediaConnector` 进行远程/本地视频下载、解码以及（可选）前端 NIXL 传输的解码帧。  
- `handlers._extract_multimodal_data` 现在会统一加载 `image_url` 与 `video_url`，并在 `construct_mm_data` 中仅保留对视频模型的通用处理（删除对已废弃 LLaVA‑Next‑Video 的特化）。  
- 删除旧的 `video_encode_worker` 与相关脚本，简化部署流程；新增聚合/分布式视频服务 launch 脚本。  
- 完善文档、单元测试以及 `media_nixl` 的类型重载。  

**🎯 影响范围**  
- **后端核心**：`dynamo.common.multimodal.video_loader`, `dynamo.vllm.handlers`, `dynamo.vllm.multimodal_utils.model`, `dynamo.common.utils.media_nixl`。  
- **示例/脚本**：`examples/backends/vllm/launch/video_agg.sh`, `video_disagg.sh`；旧的 `examples/multimodal` 相关代码被删除。  
- **文档**：`docs/features/multimodal/README.md`, `multimodal-vllm.md`。  
- **测试**：新增 `test_video_loader.py`, `test_vllm_video_handler.py`；修改 `tests/serve/test_vllm.py` 中视频 CI 场景。  

**🔍 技术洞察**  

- **架构影响**  
  1. **统一入口**：视频加载逻辑从示例工程迁移到 `components/src/dynamo/common/multimodal`，实现了与图片加载同等的抽象层，后端仅需一次 `VideoLoader` 实例即可处理所有视频请求。  
  2. **解耦前端/后端**：通过 `enable_frontend_decoding` 开关，支持两条路径：  
     - **后端解码**（默认）— 依赖 vLLM `MediaConnector` 在后端直接下载并抽帧。  
     - **前端解码** — 通过 NIXL 将已解码的帧从前端传递到后端，适用于分布式部署或需要降低后端网络 IO 场景。  
  3. **去除旧 VideoEncodeWorker**：原来的多进程、PyAV、GPU‑加速的 video 编码流水线不再维护，简化了组件依赖（`av`, `cupy` 等），降低了维护成本。  
  4. **模型支持抽象**：`construct_mm_data` 不再硬编码 LLAVA‑Next‑Video，仅保留对通用视频模型的「存在 video_numpy」检查，提升后端对未来 vLLM 新增视频模型的兼容性。  

- **性能影响**  
  - **解码路径**：使用 vLLM 原生 `VideoMediaIO`，在 C++ 层完成采样与解码，通常比 Python+PyAV 更快且内存占用更可控。  
  - **默认帧数** `NUM_FRAMES_DEFAULT=32`（可通过 `DYN_MM_VIDEO_NUM_FRAMES` 调整），比旧实现的 8 帧采样更丰富，可能提升推理质量但也会增加带宽和显存占用。  
  - **异步批处理**：`load_video_batch` 采用 `asyncio.gather` 并行下载/解码，提升并发吞吐；不过在高并发场景下仍需关注网络 IO 限制与 NIXL 读写竞争。  
  - **缓存**：代码中已留 TODO 进行远程 video URL 缓存，当前未实现，短期内对重复请求的性能提升有限。  

- **安全考虑**  
  - **URL 正规化**：`_normalize_video_url` 对本地路径做文件存在检查并强制转为 `file://` URI；对已有 scheme（http/https/data）直接返回。  
  - **本地文件访问**：`MediaConnector` 被构造为 `allowed_local_media_path="/"`，允许任意本地文件读取，可能被利用进行本地文件泄露或 SSRF（如果外部用户能够控制 `video_url`).  
  - **依赖导入**：`_require_vllm_video_media` 在缺失 vLLM 多模态库时抛出明确错误，防止因隐藏 ImportError 导致运行时异常。  
  - **异常记录**：所有异常均记录日志并包装为 `ValueError`，避免泄露内部堆栈给调用方。  

**⚠️ 潜在风险**  

| 风险点 | 可能后果 | 缓解措施 |
|--------|----------|----------|
| **本地文件无限制读取** (`allowed_local_media_path="/"`) | 攻击者若能提交任意 `file://` URL，可能读取系统敏感文件（/etc/passwd 等） | 在生产环境通过环境变量或配置限制 `allowed_local_media_path`，或在 `_normalize_video_url` 中限制仅允许工作目录下的子路径。 |
| **未实现远程缓存** | 高并发时对同一远程视频的重复下载会导致网络带宽浪费、显存峰值提升 | 及时实现缓存（基于 LRU），并在 `load_video_batch` 中复用已下载的 `BytesIO`。 |
| **前端解码路径错误配置** (`enable_frontend_decoding=False` 时仍收到 `Decoded` 项) | 运行时抛 `ValueError`，导致请求失败 | 在 API 文档中明确说明开启前端解码的前置条件；在入口层加入参数校验，提前返回友好错误。 |
| **依赖 vLLM 多模态库缺失** | `ImportError` 被包装为 `RuntimeError`，启动即失败 | CI 已加入对应依赖；生产镜像需确保 `vllm[mm]`（或等价）已安装。 |
| **显存占用增长**（默认 32 帧 × 3 通道 × 336×336）| 在显存紧张的 GPU 上可能 OOM，尤其在分布式预填/解码模式下 | 提供 env `DYN_MM_VIDEO_NUM_FRAMES` 调整，或在 `VideoLoader` 中加入显存感知采样策略。 |
| **异步异常聚合** (`asyncio.gather(..., return_exceptions=True)`) | 若单个视频解析异常，整体 `load_video_batch` 会抛统一 `Exception`，可能隐藏单个错误源 | 在日志中已记录每个失败 URL，建议在上层捕获后返回局部错误信息而非一次性中止。 |

### feat: enable KVBM Disk allocation support different storage types (#7839)
**SHA**: `234a89c` | 🔗 [查看提交](https://github.com/ai-dynamo/dynamo/commit/234a89c0c22d3ce29e52426e931a9c4ac31c62b7)

**🎯 变更类型**：功能增强  
**⚡ 重要程度**：🔴 高  
**📋 变更摘要**：  
- 为 KVBM 磁盘分配器引入可插拔的打开策略（`DiskOpenStrategy`），通过环境变量 `DYN_KVBM_DISK_ALLOCATOR_TYPE` 在运行时选择 `default`（采用 `fcntl` 设置 `O_DIRECT`）或 `open-direct`（在 `mkostemp` 时直接传入 `O_DIRECT`）。  
- 为 `DiskAllocator` 增加 `from_env` 与显式策略构造函数，`DiskStorage::new` 现接受策略引用。  
- 文档同步更新，说明了新环境变量的取值、使用场景以及故障排查步骤。

**🎯 影响范围**：  
- `lib/llm/src/block_manager/storage/disk.rs`（核心磁盘分配实现）  
- `lib/llm/src/block_manager.rs`、`worker.rs`、`offload.rs`（调用方）  
- `docs/components/kvbm/kvbm-guide.md`（用户手册）  
- 受影响的单元测试文件以及相关 CI。

**🔍 技术洞察**  
- **架构影响**：  
  - 引入了 **策略模式**（`DiskOpenStrategy`）解耦了磁盘文件打开方式与上层分配逻辑，使得不同文件系统可以在不修改业务代码的前提下切换实现。  
  - `DiskAllocator` 从单例实现转为持有 `Box<dyn DiskOpenStrategy>`，符合依赖注入原则，提升可扩展性。  
  - 对外 API（`DiskStorage::new`）从无参改为接受策略引用，内部调用已同步更新，保持向后兼容（默认仍使用 `DefaultDirectIo`）。

- **性能影响**：  
  - 在支持 `fcntl` 的文件系统上行为保持不变。  
  - 对于 IBM Storage Scale 等 `fcntl(F_SETFL, O_DIRECT)` 被忽略的环境，使用 `open-direct` 直接在文件创建时开启 `O_DIRECT`，可恢复 GPU DirectStorage 的零拷贝性能，提升 I/O 带宽与延迟。  
  - 额外的环境变量读取与策略对象创建开销极小（一次性），对整体吞吐无显著影响。

- **安全考虑**：  
  - `open-direct` 直接在 `mkostemp` 时传入 `O_DIRECT`，仍然使用 `mkostemp` 的安全特性（原子创建、`O_CLOEXEC`），未引入路径遍历或特权提升风险。  
  - 若用户误设 `DYN_KVBM_DISK_ALLOCATOR_TYPE` 为未知值，`disk_open_strategy_from_env` 将返回错误并阻止分配，避免在不受支持的文件系统上产生不可预测的 I/O 行为。  
  - 环境变量控制的行为在生产环境应通过配置审计，防止因误关闭 `O_DIRECT`（`DYN_KVBM_DISK_DISABLE_O_DIRECT`）而导致性能泄漏。

**⚠️ 潜在风险**  
1. **API 兼容性**：外部程序直接调用 `DiskStorage::new`（未传递策略）将编译错误，需要使用 `DiskAllocator::default()` 或 `DiskAllocator::from_env()` 包装。  
2. **环境变量误配置**：  
   - 设置 `open-direct` 在不支持 `O_DIRECT` 的文件系统上可能导致文件创建失败（`mkostemp` 返回错误），从而导致 KVBM 启动失败。  
   - 未显式关闭 `O_DIRECT` 时，若底层文件系统不满足对齐要求，仍可能触发 `EINVAL`，需要用户自行排查。  
3. **资源泄露**：在 `open_temp_file` 异常路径中已调用 `unlink`，但若后续逻辑未能及时关闭 `RawFd`，可能产生文件句柄泄漏。  
4. **并发安全**：`DiskOpenStrategy` 实现对象在 `DiskAllocator` 中通过 `Arc`（间接）共享，当前实现是只读且线程安全，但未来若加入内部可变状态，需要额外同步。

**💡 关注建议**

**SHA**: `1f10ba5` | 🔗 [查看提交](https://github.com/ai-dynamo/dynamo/commit/1f10ba55e69798cf2f5a6598ead32e5100a8cc65)

**🎯 变更类型**：重构（测试代码整理）  
**⚡ 重要程度**：🟡 中  
**📋 变更摘要**：在 `kv-router/src/indexer/tests.rs` 中抽取并实现了一批查询/断言辅助函数（`query_scores、assert_score、assert_no_scores、assert_exact_scores`），并将原先散落在各测试用例里的 `find_matches` 调用及结果校验统一到这些函数中，从而消除了大量重复代码。  

**🎯 影响范围**：`kv-router` 模块的单元测试（所有 `interface_tests`）以及可能的 CI 测试脚本。

4. **覆盖完整性**：运行完整的 `cargo test --all-features`，确认所有原有断言仍被覆盖，尤其是涉及 `tree_sizes` 的用例。  

总体而言，此次改动提升了测试代码的可维护性和可读性，对功能本身没有影响，只需确保新 helper 的行为与旧代码保持一致即可。
