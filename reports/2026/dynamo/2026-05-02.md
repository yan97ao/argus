### feat(benchmarks): emit image_uuids in jsonl generator for vLLM cached-mm (#8854)
**SHA**: `297119e` | 🔗 [查看提交](https://github.com/ai-dynamo/dynamo/commit/297119eac6c4ff6ae98ada191605a2ec64cfc383)

**🎯 变更类型**：功能增强  
**⚡ 重要程度**：🔴高  
**📋 变更摘要**：在多模态基准 JSONL 生成器中新增 `--uuid` 参数，开启后会为每张图片生成并输出稳定的 `image_uuids` 列表。实现了 `compute_image_uuid` 辅助函数，使用 UUIDv5（OID 命名空间）对图片引用（路径或 URL）进行确定性哈希，确保跨运行的缓存键保持不变。  

**🎯 影响范围**：  
- `benchmarks/multimodal/jsonl/args.py`（命令行解析）  
- `benchmarks/multimodal/jsonl/generate_images.py`（UUID 计算实现）  
- `benchmarks/multimodal/jsonl/main.py`（JSONL 行生成逻辑）  

**🔍 技术洞察**  
- **架构影响**：仅作用于基准生成脚本，未触及核心库或运行时代码，属于工具层面的向后兼容扩展。新增的 `image_uuids` 字段是 vLLM 的自定义扩展，保持 OpenAI‑compatible schema 的兼容性。  
- **性能影响**：UUID v5 采用 SHA‑1 哈希，计算开销极低（微秒级），对大规模基准生成的整体性能影响可忽略不计。唯一的额外成本是每行 JSON 体积增加约 36 B（UUID 字符串）+ 列表分隔符。  
- **安全考虑**：使用标准库 `uuid`，不引入外部依赖。生成的 UUID 为确定性哈希，不泄露图片内容，仅基于路径/URL 字符串。无显著安全风险。  

**⚠️ 潜在风险**  
1. **向后兼容**：下游消费方（如自研解析器或第三方工具）若未预料到 `image_uuids` 字段，可能会出现未知字段错误或忽略。  
2. **路径规范**：若不同机器上相同图片使用不同的相对路径，UUID 将不一致，导致缓存失效。  
3. **异常输入**：空字符串或无效路径仍会生成 UUID，可能与业务预期不符。  
4. **数据量膨胀**：在极大规模基准（数十亿行）下，额外的 UUID 字段会略微增大磁盘与网络传输成本。  

**💡 关注建议**  
- **文档同步**：在基准使用手册中明确 `--uuid` 参数的作用、开启场景以及对 vLLM `mm-cache-mode` 的对应关系。  
- **输入规范**：在 `compute_image_uuid` 前加一次路径正规化（`os.path.abspath` / URL 标准化），确保跨机器 UUID 一致。  
- **测试覆盖**：加入单元测试，校验同一引用在多次运行中得到完全相同的 UUID；以及空/非法引用的异常行为。  
- **兼容性验证**：在基准生成后，用 `jq` 或自研解析器检查 JSONL 行是否能够被原有流水线无错误读取。  
- **可选字段策略**：若后续需要兼容更广的工具链，可考虑在生成时加入 `image_uuids` 前的 `if args.uuid and images:` 判断，避免在没有图片的请求中产生空列表字段。

**⚡ 重要程度**：🔴高  
**📋 变更摘要**：  
- 引入 `ModelFamily` 枚举并实现 `resolve_model_family`，统一模型族的判定逻辑。  
- 所有原先使用 `is_qwen_vl_model` / `is_model_supported` 的代码改为通过 `ModelFamily` 进行分支。  
- 新增对模型配置文件 (`config.json`) 的读取、缓存以及基于名称模式的双阶段解析，提升对新模型变体的兼容性。  

**🎯 影响范围**：  
- `components/src/dynamo/vllm/handlers.py`  
- `components/src/dynamo/vllm/multimodal_handlers/encode_worker_handler.py`  
- `components/src/dynamo/vllm/multimodal_utils/__init__.py`  
- `components/src/dynamo/vllm/multimodal_utils/encode_utils.py`  
- `components/src/dynamo/vllm/multimodal_utils/model.py`（核心实现）  
- 相关单元测试 `components/src/dynamo/vllm/tests/multimodal_utils/test_vllm_model.py`  

**🔍 技术洞察**：

| 维度 | 影响 |
|------|------|
| **架构影响** | - **抽象层提升**：模型族判定从散落的布尔函数集中到 `ModelFamily` + `resolve_model_family`，形成统一的入口，后续新增模型族只需在 `_FAMILY_ARCHITECTURES` / `_FAMILY_NAME_PATTERNS` 中登记即可，降低耦合。<br>- **依赖链变化**：多处 `vllm` 处理逻辑现在依赖 `ModelFamily`，减少对具体模型名称的硬编码，使代码更具可读性和可维护性。 |
| **性能影响** | - 解析函数使用 `functools.lru_cache(maxsize=8)`，在常驻进程内几乎无额外开销。<br>- 读取 `config.json` 仅在首次解析对应路径时发生，后续均走缓存。整体性能影响可以认为是 **可忽略**。 |
| **安全考虑** | - 只涉及本地文件读取（`config.json`），未引入网络请求或外部输入的直接执行。<br>- 使用 `json.loads` 并捕获异常，防止恶意或破损的配置文件导致进程崩溃。总体上 **无安全风险**。 |
| **可维护性** | - 通过枚举和集中化的解析函数，后续对新模型族的适配只需更新同一处映射表，降低遗漏风险。<br>- 单元测试覆盖了字符串解析、文件元数据解析以及回退行为，提升回归保障。 |

**⚠️ 潜在风险**：

1. **`resolve_model_family` 返回 `None`**：现有调用处（如 `if resolve_model_family(self.config.model) is ModelFamily.QWEN_VL:`）假设返回值必定是 `ModelFamily` 成员，未对 `None` 做 guard。若用户提供未登记的模型，代码会进入默认分支（可能导致非预期的行为或 `AttributeError`）。  
2. **缓存失效**：`lru_cache` 按模型标识缓存解析结果。如果同一标识在不同文件系统位置对应不同配置（极端情况），缓存可能返回错误家族。  
3. **路径解析误判**：名称模式匹配是基于子串判断，可能出现误判（比如模型名中意外出现 `qwen3` 但实际上是非视觉模型），导致错误分支。  
4. **向后兼容**：旧的 `is_qwen_vl_model`、`is_model_supported` 已被删除，若外部插件仍直接引用这些函数会导致 ImportError。  

**💡 关注建议**：

- **防御性编程**：在所有使用 `resolve_model_family` 的分支前加入 `family = resolve_model_family(...); if family is None: fallback_to_default_or_error`，避免 `None` 引发隐蔽错误。  
- **文档更新**：在项目的模型支持章节明确说明新增的 `ModelFamily` 与 `resolve_model_family`，并提供 “如何为新模型族添加支持” 的指南。  
- **API 兼容层**：如果仍有外部代码可能依赖 `is_qwen_vl_model`，可以保留一个轻量包装函数（调用 `resolve_model_family`）以保持向后兼容。  
- **扩展测试**：加入对误匹配的负面用例（例如 `model_name="my-qwen3-not-vision"`），确保误判不会进入视觉路径。  
- **监控缓存命中率**：在生产环境中可以加入日志或度量（如 Prometheus）监控 `resolve_model_family` 的缓存命中率，确保 LRU 缓存的有效性。  

总体而言，此次改动在 **架构清晰度** 与 **可扩展性** 上带来显著提升，风险主要集中在未处理的 `None` 情况和潜在的名称误匹配。通过上述防御性检查与文档同步，可将风险降至可接受水平。

### feat(snapshot): make multi-container checkpoint/restore an annotation contract (#8631)
**SHA**: `1857904` | 🔗 [查看提交](https://github.com/ai-dynamo/dynamo/commit/18579047159ddaabc91c481f08c036205b7a7ef1)

**🎯 变更类型**：功能增强 / 重构 / 架构变更  
**⚡ 重要程度**：🔴高  

**📋 变更摘要**  
本次提交将快照（snapshot）子系统从“仅针对单一 `main` 容器”升级为 **多容器目标**，实现了以下关键点：  
1. 引入 `nvidia.com/snapshot-target-containers` 注解，显式声明 **checkpoint/restore** 所作用的容器列表（支持单容器、多个容器或 failover‑engine 列表）。  
2. 运算符（operator）在生成 PodSpec、CheckpointJob、RestorePod 时统一读取、写入并校验该注解，确保 **Checkpoint** 必须只对应 **一个容器**，而 **Restore** 可以对应 **一个或多个容器**。  
3. 为 intra‑pod failover 场景（engine‑0、engine‑1）实现自动注解、子路径挂载（`subPath=<containerName>`）以及多容器的 `sleep infinity` 占位与启动探针。  
4. 重构 checkpoint 注入流程，去除对 “第 0 个容器即为 main” 的假设，改为依据注解解析容器名称。  
5. 更新 `snapshotctl` CLI，强制使用 `--container` / `--containers` 标记或预先注解，统一校验并在 Manifest 中写入注解。  
6. 清理旧的标签（`snapshot-is-restore-target`）与相关逻辑，统一使用 `snapshot-checkpoint-id` + 目标容器注解来标识 restore pod。  
7. 重新实现控制卷挂载的 **per‑container subPath**，防止不同容器共享同一目录导致状态冲突。  
8. 大量单元测试与文档同步，验证多容器、GMS、Failover、 operator‑generated podClique 等全链路行为。

**🎯 影响范围**  
- `deploy/operator/internal/checkpoint/*`（注入、GMS sidecar、PodSpec 处理）  
- `deploy/operator/internal/controller/*`（CheckpointJob、DynamoComponentDeployment、DynamoGraphDeployment 生成）  
- `deploy/operator/internal/dynamo/*`（failover 检测、engine‑container 名称）  
- `deploy/snapshot/*`（protocol、CLI、controller）  
- 相关 CRD（`DynamoCheckpoint`、`DynamoComponentDeployment` 等）及其注解/标签处理  
- 文档 `docs/kubernetes/snapshot.md` 以及 Helm chart 示例  

**🔍 技术洞察**  

| 维度 | 影响 |
|------|------|
| **架构影响** | - **注解驱动**的目标容器选择取代了硬编码的 `main` 逻辑，形成统一的契约层（annotation contract）。<br>- 通过 `TargetContainersFromAnnotations` 统一解析、校验并返回容器列表，所有上层（operator、snapshotctl、snapshot-agent）共享同一实现，降低了实现分歧。<br>- 在 **Intra‑Pod Failover** 场景，`IntraPodFailoverEngineContainerNames()` 动态生成 `engine-0,engine-1`，并在 `GenerateGrovePodCliqueSet`、`InjectCheckpointIntoPodSpec` 等关键路径自动写入注解，做到“一次声明、多处生效”。 |
| **性能影响** | - **Checkpoint/Restore 目标容器数量** 直接决定 `sleep infinity` 替换的容器数目，额外的 `EnsureControlVolume` 和 `VolumeMount` 操作仅在 PodSpec 生成阶段执行，运行时无额外开销。<br>- 对 **control volume** 使用 per‑container `subPath`，避免了旧实现中所有容器共享同一目录导致磁盘 I/O 冲突，理论上提升并发 restore 稳定性。 |
| **安全考虑** | - 引入 `TargetContainersAnnotation` 必须遵循 K8s 合法名称校验，已使用 `validation.IsQualifiedName` 防止注解键超长或非法导致 `metadata.annotations` 被拒绝。<br>- `RestoreStatusAnnotationKeysFor` 生成 per‑container状态注解键，确保容器名称不会突破 DNS‑1123 限制，防止潜在的 API Server 拒绝或注入攻击。<br>- `PrepareRestorePodSpec` 仍然强制注入本地 seccomp profile，保持对 I/O 系统调用的拦截。 |
| **可维护性** | - 所有关于容器定位的代码从 “硬编码 `main`/`worker`” 迁移为 **注解+函数**，后续若需支持更多容器或自定义名称，仅需修改注解值即可，无需遍历调用点。<br>- `checkpoint` 与 `restore` 的注解校验统一放在 `reconcileTargetContainers`，CLI 与 operator 共用，实现 DRY。<br>- 移除已废弃的 `snapshot-is-restore-target` 标签，简化标签空间，避免混淆。 |
| **兼容性** | - 旧的单容器 checkpoint 仍然兼容，只需在 PodTemplate 中保留 `snapshot-target-containers: main` 注解（operator 自动写入）。<br>- 对已有的 checkpoint Job，如果未带注解，operator 会在 `buildCheckpointJob` 中自动补全 `TargetContainersAnnotation=main`，保证平滑升级。 |
| **监控/可观测性** | - `RestoreStatusAnnotation` 仍然使用 `snapshot-restore-status.<container>`，但现在可针对多容器分别观察，每个容器的恢复进度独立记录。 |
| **错误处理** | - 强化了缺失、重复、非法容器名的报错信息，CLI 在注解冲突时会直接返回错误，避免产生不可预期的 checkpoint/restore。<br>- `NodeController` 重新按照 `CheckpointIDLabel` + `!CheckpointSourceLabel` 筛选 restore pods，减少误匹配。 |

**⚠️ 潜在风险**  

1. **注解缺失或不一致**  
   - 若用户手动编辑 Manifest，但未同步 `--container(s)` 标记，`snapshotctl` 将报错，可能导致使用流程受阻。需要在文档与 CI 中强化提示。  

2. **容器名称超长或包含非法字符**  
   - 虽已使用 `validation.IsQualifiedName` 检查，但若用户自定义极长名称（>63字符）仍会被拒绝，导致 restore 任务不启动。  

3. **多容器 Restore 时的 `sleep infinity` 替换**  
   - 某些容器在被替换后会失去原始启动命令，若未在 `PrepareRestorePodSpec` 前做好挂载/文件准备，可能出现恢复后容器启动异常（尤其是非‑Python 包装的 workload）。  

4. **跨版本兼容**  
   - 旧版快照代理（snapshot-agent）如果未升级到支持 `snapshot-target-containers` 注解的版本，将忽略多容器信息，导致恢复仅作用于第一个容器。确保集群中所有快照组件同步升级。  

5. **Failover 与 GMS 并存的复杂性**  
   - 代码已在 `graph.go` 中对 `RoleGMS` 排除注解写入，但如果未来出现 **Intra‑Pod Failover + GMS** 的混合模式，需额外判断避免把 GMS weight‑server 当作 restore 目标。  

**💡 关注建议**  

1. **升级指引**：在 operator release notes 中明确说明 “从 1.x 升级到 2.x 需要保证所有快照组件（snapshot‑agent、snapshotctl）同步到最新版本”。提供脚本检测注解是否缺失。  

2. **CI/Lint 检查**：在 CI 流程中新增校验，确保所有 `DynamoCheckpoint`、`DynamoComponentDeployment`、`DynamoGraphDeployment` 生成的 PodTemplate 均包含合法 `snapshot-target-containers` 注解。  

3. **监控告警**：为 `snapshot-target-containers` 注解缺失/解析错误的情况添加 Prometheus 报警，以便快速定位因注解错误导致的 checkpoint/restore 失败。

### fix: prevent false cache misses for short queries in BranchShardedIndexer (#8939)
**SHA**: `6c5a728` | 🔗 [查看提交](https://github.com/ai-dynamo/dynamo/commit/6c5a728b5fe460252b2a005c1a17b1570c04375d)

**⚡ 重要程度**：🔴高  
**📋 变更摘要**：  
- 修复 `BranchShardedIndexer` 在 `prefix_depth > 1` 时对长度小于 `prefix_depth` 的查询产生误报（false miss）的问题。  
- 在存储根事件时额外记录所有中间 FNV 前缀哈希，使得短查询能够路由到正确的分片并获得浅层匹配而不是直接失效。

**🎯 影响范围**：  
- `lib/kv-router/src/indexer/branch_sharded.rs`（核心索引器实现）  
- `lib/kv-router/README.md`、`lib/bench/README.md`（文档）  
- 受影响的组件：KV Router、BranchShardedIndexer、分片调度（sharding）逻辑、缓存命中统计。  

**🔍 技术洞察**：  
- **架构影响**  
  - 新增 `register_prefix_keys` 与 `prefix_keys_for_stored_blocks_from_state` 两个私有函数，用于在 `apply_event` 期间把 **每个前缀 FNV 哈希 → 分片索引** 的映射写入 `branch_to_shard`。  
  - 该映射原本只记录完整的 `prefix_depth` 哈希，现在会额外保存深度 < `prefix_depth` 的哈希，以实现 “短查询” 的正确路由。  
  - 逻辑仍然保持原有的 “branch count 不计入” 语义，仅在 **根或跨越 `prefix_depth` 的批次** 时调用 `register_prefix_keys`，不影响已有的分片计数统计。  

- **性能影响**  
  - **额外计算**：在每次 `Stored` 事件处理时，需要遍历最多 `prefix_depth` 条块，累计 FNV 哈希并保存到 `Vec<u64>`。这在极端情况下（大量小批次存储）会带来 O(`prefix_depth`) 的轻微 CPU 开销。  
  - **内存开销**：`branch_to_shard` 会记录更多键值对（每个完整根哈希对应 `prefix_depth` 条前缀），在高并发、大量根事件的工作负载下可能导致哈希表膨胀。实际影响取决于 `prefix_depth`（默认 2~4）与根事件频率。  
  - **查询收益**：短查询（长度 < `prefix_depth`）命中率显著提升，避免了因前缀不匹配而直接返回 miss，提升了整体 KV 路由的响应质量，尤其在 **“短查询占比较高”** 的场景（如微服务内部缓存、RAG 检索）中收益可观。  

- **安全考虑**  
  - 变更仅涉及哈希映射与路由逻辑，无外部输入的直接执行路径，未引入新的安全风险。  
  - 唯一需关注点是 **哈希碰撞**：FNV 哈希虽然冲突概率低，但额外存储的前缀键增加了冲突概率的理论上限，但对业务安全影响仍然可接受。  

**⚠️ 潜在风险**  
1. **内存增长**：`branch_to_shard` 记录的前缀键数量是根事件数 × `prefix_depth`，在极端的大规模根批次场景可能导致显著内存占用。  
2. **清理不完整**：`Removed` 事件目前只删除完整根键对应的 `branch_to_shard` 条目，**不会自动清理已注册的短前缀键**，可能留下无用映射，导致长期内存泄漏。  
3. **并发写入竞争**：使用 `DashMap::entry(...).or_insert` 在多线程环境下是安全的，但在高并发写入同一前缀键时仍会产生锁竞争，极端情况下可能轻微影响写入吞吐。  
4. **错误的 `prefix_depth` 配置**：如果 `prefix_depth` 设置过大，前缀键数量激增，且短查询本身并不常见，反而会带来不必要的开销。  

**💡 关注建议**  
- **监控内存使用**：在生产环境中监控 `branch_to_shard` 的大小，尤其在高根频率的工作负载下。可通过 Prometheus/Metric 导出条目数量进行预警。  
- **实现前缀键清理**：在 `Removed` 或 `Cleared` 事件处理中，考虑加入 **逆向映射**（前缀键 → 根键）或引用计数，以便在根被删除时同步清理对应的短前缀键。  
- **配置审查**：根据业务查询特征（短查询比例）合理调节 `prefix_depth`；若短查询占比低，保持默认（2）或更小，以控制内存开销。  
- **性能回归测试**：在大规模基准（`mooncake_bench`）中加入 **短查询命中率** 与 **内存占用** 两项指标，确保新实现未对吞吐或延迟产生负面回归。  
- **文档同步**：确保所有外部用户文档（README、bench guide）已指向新的 `INDEXER_BENCH.md`，避免旧链接导致的 benchmark 运行错误。

- `deploy/inference-gateway/epp/Dockerfile`

- `recipes/glm-5-nvfp4/Dockerfile`  
- `tests/fault_tolerance/deploy/container/Dockerfile.local_vllm`  

**💡 关注建议**  
1. **CI 运行时验证**：运行一次完整的 CI（包括并发的 Docker 镜像构建）确保没有因缓存锁定导致的 dead‑lock 或超时。  
2. **缓存清理**：若出现缓存损坏或不一致，使用 `docker builder prune --filter type=cache` 手动清理对应的缓存层。

总体来看，此次改动通过显式声明缓存共享策略，降低了并发构建中因缓存竞争导致的错误几率，对 CI 稳定性和构建时间都有正面影响。只要 CI 环境使用支持该语法的 Docker BuildKit（>=23.0），即可安全上线。
