### feat(frontend): standardize Dynamo routing headers (#10873)
**SHA**: `82f6f0b` | 🔗 [查看提交](https://github.com/ai-dynamo/dynamo/commit/82f6f0ba13c12832ff50c7f9c3ed4d1c577017f6)

**🎯 变更类型**：功能增强  
**⚡ 重要程度**：🔴高  
**📋 变更摘要**：将 Dynamo 路由相关的所有 HTTP Header 统一加上 `x-dynamo-` 前缀，以避免与用户自定义或第三方服务的同名冲突。代码实现中同时保留原始未加前缀的兼容别名，并在文档与测试中同步更新。目标是提升路由字段的可辨识性与未来的可演进性。  

**🎯 影响范围**：  
- `deploy/inference-gateway/epp`（Go 与 Rust 实现的 External Processor）  
- `deploy/inference-gateway/epp/cmd/epp`（入口程序）  
- `lib/llm`（Rust KV‑router、HTTP 服务层、协议扩展常量）  
- 文档层面：frontend、nvext、kubernetes 部署指南  
- 单元/集成测试及 E2E 脚本（Python、Rust）  

**🔍 技术洞察**  

- **架构影响**  
  - **路由链路统一**：所有路由决策（解码 worker、prefill worker、DP rank）均通过统一命名的 Header 进行传递，避免了在不同组件间出现 “同名但语义不同” 的隐蔽错误。  
  - **向后兼容层**：通过在 `extensions.rs` 中引入 `*_ALIAS` 常量并在 `apply_header_routing_overrides` 中做 `or_else` 搜索，现有使用旧 Header 的客户端仍可正常工作，保证平滑升级。  
  - **文档同步**：所有公开文档已更新，明确提示旧 Header 为 “compatibility alias”，并在未来计划逐步废除，帮助生态提前适配。  

- **性能影响**  
  - **微乎其微**：仅是 Header 名称的字符串常量更换与一次额外的 HashMap 查找（别名兼容），对请求路径的 CPU、内存、延迟均无可感知影响。  
  - **编译期优化**：常量折叠和内联不会引入额外运行时开销。  

- **安全考虑**  
  - **命名冲突防护**：通过前缀化降低了恶意用户利用相同 Header 名称伪造路由信息的风险。  
  - **别名接受策略**：仍然接受旧 Header，意味着在混合部署的短暂窗口期内，攻击者仍可使用未经前缀的 Header。建议在生产环境中尽快切换至新 Header，并在配置中关闭别名（若有此选项）。  
  - **日志与监控**：部分日志信息已同步更新，确保审计时能够识别新旧 Header 的来源。  

**⚠️ 潜在风险**  

1. **部署版本不对齐**：如果前端（EPP）使用新 Header，而后端（KV‑router、LLM 服务）仍运行旧二进制，路由信息会缺失，导致 fallback 或错误的 worker 选举。  
2. **别名误用**：在混合使用新旧 Header 时，优先级实现为 “新 Header 覆盖旧 Header”。若业务方误在同一请求中发送冲突值，可能导致不可预期的路由。  
3. **文档/脚本遗漏**：自动化 CI 或第三方 SDK 仍硬编码旧 Header，升级后会出现隐藏的 4xx/5xx 错误。  
4. **外部代理/网关**：某些自建或云厂商的 HTTP 代理可能对 `x-` 前缀做特殊处理，需要确认不拦截 `x-dynamo-` 系列 Header。  

**💡 关注建议**  

- **发布策略**：在正式发行前，提供明确的 “兼容期（建议 4‑6 周）” 文档，说明在此期间两套 Header 均被接受。  
- **监控与告警**：在网关或 router 端加入指标 `router.headers.deprecated_used_total`，追踪仍使用旧 Header 的请求比例，以便在兼容期结束前强制关闭 alias。  
- **CI 检查**：在仓库增加 lint 步骤，确保代码新增或修改时使用 `HEADER_*` 常量而非硬编码字符串，以防再次出现命名不统一的情况。  
- **默认安全选项**：可考虑在配置（如 `DYN_ENABLE_HEADER_ALIAS`）中提供开关，默认在新发布中关闭别名，强制使用新 Header，进一步降低冲突风险。  
- **文档推广**：在所有对外 SDK、示例代码和 API 参考中同步切换为 `x-dynamo-` 前缀，标记旧 Header 为已废弃。  

总体来看，此次改动在提升可维护性和安全性的同时，保持了向后兼容，风险主要集中在版本同步和别名使用上。只要按上述建议做好发布节奏与监控，影响可控且收益显著。

### feat(sglang): bump to 0.5.13.post1 (#10734)
**SHA**: `b331b15` | 🔗 [查看提交](https://github.com/ai-dynamo/dynamo/commit/b331b15a998a7d25f3b61e166b3b24a616b64f0c)

**⚡ 重要程度**：🔴高  
**📋 变更摘要**：将 Dynamo 对 SGLang 的依赖从 0.5.12.post1 升级到 0.5.13.post1，去除旧版兼容层 `get_scheduler_info`，改为直接访问 `engine._scheduler_init_result.scheduler_infos[0]；同步更新容器镜像标签、CI 超时时间、视频编码实现以及相关文档和测试，以利用上游已实现的 `mm_hashes` 互操作特性。  

**🎯 影响范围**：  
- `components/src/dynamo/sglang/_compat.py`（删除 `get_scheduler_info`）  
- `components/src/dynamo/sglang/llm_engine.rs`、`register.rs`（直接读取 scheduler 信息）  
- CI 工作流（GPU 测试超时从默认提升至 45 分钟）  
- Dockerfile 与容器镜像标签（指向新的 SGLang runtime）  
- 视频生成模块（新增 codec 回退逻辑）  
- 多处文档、测试与 README 的版本说明  

**🔍 技术洞察**  
- **架构影响**  
  - **耦合度提升**：原本通过 `_compat` 提供的抽象层被移除，代码直接依赖 SGLang v0.5.13 内部结构 (`_scheduler_init_result.scheduler_infos[0]`)。这简化了实现，但削弱了对未来 SGLang 结构变化的容错能力。  
  - **补丁淘汰**：`container/deps/sglang/patches/v0.5.12.post1/0001‑pr25300‑mm_hashes‑interop.patch` 被删除，因为对应功能已在上游 0.5.13 中实现，减少了自维护补丁的维护成本。  

- **性能影响**  
  - **视频编码**：在使用 `h264_nvenc` 失败时自动回退到 CPU 编码器 `libx264`，提升了在不具备 NVENC 支持的 GPU 实例上的稳健性，代价是 CPU 带宽和时延可能上升。  
  - **GPU 测试**：CI 超时上调至 45 分钟，可防止因 SGLang 启动或大模型加载慢导致误报失败，间接提升 CI 稳定性。  

- **安全考虑**  
  - 本次改动未涉及网络、权限或加密逻辑，安全风险基本为零。唯一需要关注的是 **依赖升级** 可能引入上游未审计的安全漏洞，建议在生产镜像中锁定具体 SGLang 版本，并定期监控 upstream 安全通报。  

**⚠️ 潜在风险**  
1. **运行时兼容性**  
   - 若用户在本地或 CI 环境中仍使用旧版 SGLang（< 0.5.13），直接访问 `_scheduler_init_result.scheduler_infos[0]` 会抛出 `AttributeError`，导致 Dynamo 启动失败。  
2. **上游结构变更**  
   - 未来 SGLang 可能改动 `Engine` 的内部属性名称或结构，当前硬编码访问将失效，缺少回退逻辑。  
3. **视频回退路径**  
   - 在 `h264_nvenc` 失败后转为 `libx264`，如果系统缺少 `ffmpeg` 编码库或对应的环境变量不正确，仍会报错。  
4. **容器镜像同步**  
   - `container/context.yaml`、`README`、CI 镜像标签均已更新为 0.5.13；若实际构建过程中拉到旧镜像或缓存未清理，可能出现版本不匹配。  

**💡 关注建议**  
- **兼容层增强**：在关键位置（如 `llm_engine.rs`、`register.rs`）加入显式的异常捕获，若属性不存在则回退到旧版兼容实现或抛出易读的错误提示。  
- **CI 与发布流程**：在 CI 前置步骤中添加 `sglang --version` 检查，确保使用的镜像版本正好是 0.5.13.post1，防止因缓存导致的版本回退。  
- **文档同步**：更新 `components/src/dynamo/sglang/_compat.py` 的注释，说明此文件已不再提供 `get_scheduler_info`，并在 README 中明确最低支持的 SGLang 版本。  
- **视频编码依赖**：在 Docker 镜像中确认已安装 `ffmpeg` 与 `libx264`，并在异常日志中加入完整的 `ffmpeg -codecs` 输出以便调试。  
- **监控与回滚**：上线后监控 `engine._scheduler_init_result` 访问日志，一旦出现属性缺失异常，快速回滚到使用 `_compat.get_scheduler_info` 的容错实现。

### fix(operator): fail closed conversion admission in manual cert mode (#10668)
**SHA**: `d8cdf13` | 🔗 [查看提交](https://github.com/ai-dynamo/dynamo/commit/d8cdf133ca6b0804629c9e253db66814d8ab0609)

**🎯 变更类型**：Bug修复  
**⚡ 重要程度**：🔴 高  

**📋 变更摘要**  
- 解决了在 **手动证书模式**（`certProvisionMode: Manual`）下，CRD 转换 Webhook 因缺失 CA Bundle 而导致的“fail‑open”风险，改为 **fail‑closed**。  
- 为多版本 CRD 自动注入 Webhook conversion 配置（strategy = Webhook），并在 Helm Chart 中加入可配置的 Service Name/Namespace 参数。  
- 通过 `CertManager.SetupAndRunOnce`、`CABundleInjector`、以及新增的 **GVK 校验**（`ValidateAdmissionGVK`）实现证书的同步准备、CA Bundle 注入以及 admission 请求版本安全检查。  

**🎯 影响范围**  
- `deploy/operator`（核心 operator 代码、证书管理、Webhook 注册）  
- `deploy/operator/internal/cert`（证书轮转、占位 Secret、挂载等待）  
- `deploy/operator/internal/webhook`（默认化、验证处理器、GVK 校验）  
- Helm Chart `charts/platform/components/operator`（CRD YAML 与 Deployment 模板）  
- `api/v1alpha1`、`api/v1beta1`（新增 `GroupVersionKind` 常量）  

---

### 🔍 技术洞察

| 维度 | 影响描述 |
|------|----------|
| **架构影响** | 1. 为所有 Dynamo 相关 CRD（`DynamoComponentDeployment`、`DynamoGraphDeployment`、`DynamoGraphDeploymentRequest`、`DynamoGraphDeploymentScalingAdapter`）统一添加 **conversion webhook** 配置，形成 **CRD → Webhook → Operator** 的全链路转换路径。<br>2. `cert` 包的启动顺序被重构：<br>&nbsp;&nbsp;• `SetupAndRunOnce` 在 `mgr.Start` 之前同步完成一次证书生成/刷新，保证 TLS 证书与 CA Bundle 已经在 Secret 中可用。<br>&nbsp;&nbsp;• 通过 `WaitForMountedCertificate` 等待 kubelet 将证书文件投射到容器，防止 “证书文件未就绪但已经启动 Webhook” 的 race。<br>3. 引入 **GVK 校验**（`ValidateAdmissionGVK`），在每个 admission/defaulting 处理器入口强制检查请求的 `GroupVersionKind` 与 operator 支持的版本一致，提升 **fail‑closed** 能力，防止意外在未转换的旧版本对象上执行业务逻辑。 |
| **性能影响** | - 证书轮转仍由 `certrotator` 异步完成，除首次启动的同步检查外，对运行时请求的 **延迟** 基本无影响。<br>- 新增的 `CABundleInjector` 在注入阶段使用 `client.Patch`（MergeFrom），对 API Server 的负载极低。 |
| **安全考虑** | - **手动模式**下，原实现仅在 Admission Webhook 注入 CA Bundle，**conversion** CA Bundle 却保持空，导致 API Server 在转换失败时仍接受对象（fail‑open）。现在 `InjectCRDConversionCA` 会 *阻塞*（通过 `wait.PollUntilContextCancel`）直至 Secret 中的 `ca.crt` 可用，确保 **所有** webhook（Admission + Conversion）均拥有有效 CA，实现 **fail‑closed**。<br>- 在 `CertManager.SetupAndRunOnce` 中，自动模式会先 **bootstrap** 证书后再注册 rotator，防止在 HA API Server 环境中出现 “CA 与 webhook 配置不匹配” 的安全窗口。<br>- 增加 `ValidateAdmissionGVK`，若 API Server 调度了错误版本的对象，Admission Webhook 会直接返回错误，阻止潜在的 schema 不兼容或未经过转换的对象进入业务层。 |
| **可维护性** | - 将 CRD conversion 注入抽离为独立函数 `configureConversionWebhookService`，参数化 Service Name/Namespace，便于 Helm Chart 与自定义部署的灵活配置。<br>- `groupversion_info.go` 中统一导出 `GVK` 常量，降低硬编码风险。<br>- 新增单元测试覆盖 **CA Bundle 注入**、**挂载等待**、**GVK 校验**，提升回归安全性。 |

### ⚠️ 潜在风险

1. **Helm 参数未同步**：若使用旧版 Chart 未填入 `--conversion-webhook-service-name` / `--conversion-webhook-service-namespace`，CRD 中的 Service Reference 会指向错误的 Service，导致 Conversion Webhook 调用失败。  
2. **手动模式的 Secret 不存在或缺少 `ca.crt`**：`CABundleInjector.waitForCABundle` 会在启动阶段阻塞直至 Secret 完整。如果运维忘记提前创建该 Secret，Operator 将因 `context.DeadlineExceeded` 超时而无法启动。  
3. **GVK 校验导致兼容性回退**：旧版客户端（或手动 `kubectl apply`）如果在未启用 conversion 的情况下直接提交 `v1beta1` 对象，会被 webhook 拒绝。需要确保所有入口都走 conversion 流程。  
4. **证书轮转期间的 Pod 重启**：`WaitForMountedCertificate` 依赖文件系统投射，一旦 Secret 更新但 kubelet 尚未同步，Operator 暂停启动。若 kubectl/helm 在此期间触发滚动更新，可能出现短暂不可用。

- 引入 `remove_worker_for_hashes` 与 `RemoveBatchOutcome`，一次性定位最小 edge 位置、一次性删除并返回未匹配的哈希，降低锁竞争和多次遍历的开销。  
- 更新 `remove.rs` 通过“分组‑批量‑回退”策略，在同一父节点上聚合哈希进行批删，并在必要时回退到单哈希路径。新增大量单元测试验证行为。

- `kv-router::core::indexer::concurrent_radix_tree_compressed`（核心索引模块）

- 相关测试文件 `tests.rs`（新增批量删除用例）  

---

### 🔍 技术洞察

| 维度 | 影响描述 |
|------|----------|
| **架构影响** | - **局部改动**：仅限于压缩基数树内部的删除逻辑，未修改公开 API 或外部模块，保持向后兼容。<br>- **职责划分**：抽象出 `apply_removed_group` 与 `apply_removed_hash`，使批量删除与单哈希回退路径清晰分离，提升代码可维护性。<br>- **数据结构**：新增 `RemoveBatchOutcome`，在 `Node` 层返回 `stale_hashes`（实际被删除的哈希）与 `unmatched_hashes`（在当前节点未匹配的哈希），增强信息传递能力。 |
| **性能影响** | - **锁粒度降低**：原先每个哈希都需要一次 `shape_gate.write()` 与 `state.write()`，现在一次 `remove_worker_for_hashes` 完成多哈希的定位与删除，仅一次写锁即可。<br>- **遍历次数减少**：通过 `min_match` 选取最小 edge 位置，避免对同一压缩边的重复定位；批量删除后仅在必要时回退到子树寻找。<br>- **测试基准**（未在 PR 中提供）预计在高并发、同一父节点上大量删除场景下吞吐量提升 30%‑50%，延迟下降约 20%。 |
| **安全考虑** | - **并发安全**：仍然依赖 `parking_lot::RwLock` 与 `DashMap` 的原子性，新增的批量路径未引入跨锁的顺序依赖，风险可控。<br>- **错误处理**：`unmatched_hashes` 在批量删除后会被重新走单哈希路径，确保不产生遗漏或误删。<br>- **内存安全**：仅在 `Node` 中短暂创建 `Vec`，没有循环引用或泄漏风险。 |

| **兼容性** | - 仅内部实现变化，外部调用 `remove` 仍保持相同签名与语义，**向后兼容**。 |

### ⚠️ 潜在风险

1. **分组错误导致误删**  
   - 逻辑判断 `node.contains_edge_hash(block_hash)` 决定是否继续聚合。若 `contains_edge_hash` 实现不严谨，可能将本不相同的哈希错误归入同一组，从而在批量删除时触发错误的 `min_match` 选取。  
2. **并发分裂期间的竞争**  
   - 在另一个线程对同一边进行 `split` 并移动部分哈希到子树时，`remove_worker_for_hashes` 只返回匹配的最小位置，未匹配的哈希会进入 `unmatched_hashes` 并走回退路径。若回退路径出现错误（如 `find_in_subtree` 未能正确定位），可能导致哈希永久丢失。  
3. **性能回退路径**  
   - 当大量哈希都不在最小位置（如大量分裂导致哈希分散），批量删除会退化为一次次单哈希回退，产生额外的 `find_in_subtree` 调用，可能在极端情况下比原实现更慢。  
4. **内存峰值**  
   - `remove_worker_for_hashes` 暂存 `unmatched_hashes` 与 `min_match`，在一次批量删除中若 `block_hashes` 极大（如上千），会产生一次性 `Vec` 分配，可能导致短暂的内存峰值。  
5. **测试盲区**  
   - 当前测试覆盖了常见路径，但未覆盖跨线程高并发下的“分裂‑删除‑恢复”交叉场景，仍有潜在竞态未被捕获。
