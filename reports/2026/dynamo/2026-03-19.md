### fix: increase `/dev/shm` for CI pytest containers to fix NIXL UCX segfault (#7515)
**SHA**: `4fc2b5e` | 🔗 [查看提交](https://github.com/ai-dynamo/dynamo/commit/4fc2b5e84b19c7583069dd00876bf69eb06866ec)

**🎯 变更类型**：Bug修复  
**⚡ 重要程度**：🔴高  
**📋 变更摘要**：  
- 在 GitHub Actions CI 中为 `pytest` 容器显式扩大 `/dev/shm`（共享内存）到 200 MiB，解决 NIXL‑UCX 在多 Agent 场景下因共享内存不足导致的 segfault。  
- 同时更新了相关测试配置：去除导致 segfault 的 `UCX_TLS` 环境变量、统一了测试名称、并为 VLLM 添加 `DYN_VLLM_EMBEDDING_TRANSFER_MODE=nixl-write`。  

**🎯 影响范围**：  
- `.github/actions/pytest/action.yml`（CI 执行环境）  
- `tests/serve/test_sglang.py`、`tests/serve/test_vllm.py`（单元/集成测试）  
- 受影响的模块：CI Docker 运行时、SGLang 与 VLLM 的测试脚本与配置。

**🔍 技术洞察**：

- **架构影响**  
  - 仅限 CI 环境的 Docker 启动参数变化，未修改项目核心代码或运行时架构。  
  - 去除 `--ipc=host` 并改用 `--shm-size=200m`，避免宿主机共享内存大小（K8s 默认 64 MiB）覆盖自定义大小，保持容器内部共享内存独立。  
  - 新增 VLLM 环境变量 `DYN_VLLM_EMBEDDING_TRANSFER_MODE=nixl-write`，为 NIXL 写入模式提供显式开关，提升可配置性。

- **性能影响**  
  - 增大 `/dev/shm` 会略微提升内存占用（约 +136 MiB 对比默认），对 CI 资源消耗有轻微上升，但能够避免 UCX 分配失败导致的测试崩溃，整体提升测试可靠性。  
  - 移除 `--ipc=host` 使容器不再共享宿主机的 IPC namespace，可能对跨容器通信略有影响，但在当前 CI 场景下无实际影响。

- **安全考虑**  
  - 仅调整 Docker 运行时选项和测试环境变量，无新增网络或权限类风险。  
  - 通过显式限制 `shm-size`，防止容器因意外大量共享内存占用而影响宿主机安全，属于正向安全改进。

**⚠️ 潜在风险**：

1. **CI 资源紧张**：增加 200 MiB 共享内存会提升每个测试容器的内存基准占用，若 CI 并行度很高，可能触发 OOM 或导致排队时间延长。  
2. **Docker 参数兼容性**：某些自行维护的 CI 环境或老旧 Docker 版本可能不支持 `--shm-size` 或对 `--ulimit` 选项有不同默认行为，可能导致启动失败。  
3. **环境变量变更**：`UCX_TLS` 被移除后，若外部用户在本地自行运行相同测试脚本，仍可能遇到相同的 UCX segfault，需要自行加入合适的共享内存配置。

**💡 关注建议**：

- **CI 侧**：监控 CI 运行时的内存使用峰值，确保总并行容器数与可用节点资源匹配；必要时在 CI 配置中添加资源上限（如 `resource_class`）或调低并行度。  
- **本地调试**：对本地开发者，建议在 Docker 运行时手动添加 `--shm-size=200m`（或更大）以复现 CI 环境，避免因默认 64 MiB 导致的 UCX segfault。  
- **文档更新**：在项目 README 或 CI 章节说明 `/dev/shm` 大小需求以及相关环境变量的使用场景，帮助外部贡献者快速定位类似问题。  
- **回退措施**：若后续发现共享内存需求增长，可将 `--shm-size` 调整为更大值或恢复 `--ipc=host`（并在 K8s 环境中自行增大 host `/dev/shm`），保持灵活性。

### feat: add NixlEmbeddingSender / NixlEmbeddingReceiver in sglang backend (#7153)
**SHA**: `70892fc` | 🔗 [查看提交](https://github.com/ai-dynamo/dynamo/commit/70892fc1b83be513175739915ff24f4dd960e75b)

**🎯 变更类型**：功能增强 / 架构变更  

**⚡ 重要程度**：🔴高  

**📋 变更摘要**：  
本次提交为 SGLang 后端引入了 Nixl（共享内存）方式的多模态嵌入传输，实现了 `NixlEmbeddingSender` 与 `NixlEmbeddingReceiver`。通过 `EmbeddingTransferMode` 枚举和工厂映射，用户可以在 *local*、*nixl‑write*、*nixl‑read* 三种模式间切换。相应地，协议结构从 `RdmaMetadata` 替换为统一的 `TransferRequest`，并在编码/解码 worker 中完成了初始化、发送、接收以及张量释放的全链路改造，同时去除了原有 `async_init` 中的 Nixl connector 初始化逻辑。

**🎯 影响范围**：  
- `components/src/dynamo/common/multimodal/*`（工厂、常量、协议）  
- `components/src/dynamo/sglang/backend_args.py`（新增 CLI/Env 参数）  
- `components/src/dynamo/sglang/request_handlers/multimodal/*`（Encode、Worker、Prefill）  
- `components/src/dynamo/sglang/protocol.py`（请求模型）  
- 测试 `tests/serve/test_sglang.py`（加入 UCX 环境规避）  

**🔍 技术洞察**  

- **架构影响**  
  - 引入 **EmbeddingTransferMode** 抽象，使多模态嵌入的传输方式可插件化，提升系统可扩展性。  
  - 通过 **EMBEDDING_SENDER_FACTORIES / RECEIVER_FACTORIES** 将具体实现（本地 vs Nixl）解耦，后端 handler 只依赖工厂返回的对象，降低了代码耦合度。  
  - `TransferRequest` 成为统一的传输描述，取代原有 `connect.RdmaMetadata`，为后续支持其它传输协议（如 gRPC、NVLink）预留空间。  
  - 删除了 `async_init` 中的 Nixl connector 初始化，改为 **惰性初始化**（在发送/接收时创建），简化了 worker 启动流程，也避免了启动阶段不必要的资源占用。  

- **性能影响**  
  - **Nixl‑write / Nixl‑read**：利用共享内存实现零拷贝，理论上可将嵌入张量的序列化/反序列化成本降至几乎为 0，显著提升跨进程/跨机器的吞吐。  
  - 额外 **Future**（`transfer_future`）的等待使得发送端在获取响应前仍需等待传输完成，若网络/共享内存布局不佳，可能出现微小的阻塞。  
  - 通过 `release_tensor` 在使用完毕后主动回收资源，防止显存泄漏，保持长时间服务的内存稳定。  
  - 默认模式改为 `NIXL_WRITE`，在不具备 Nixl 环境的部署上会因库缺失或初始化失败而回退到 `local`（未实现自动回退），需要运维注意。  

- **安全考虑**  
  - 共享内存的使用在同一机器上通常安全，但在多租户或容器化部署时，**内存映射权限** 需要严格控制，防止未授权进程读取嵌入张量。  
  - 环境变量 `DYN_SGL_EMBEDDING_TRANSFER_MODE` 暴露了传输方式的配置入口，若被恶意篡改为 `nixl‑write` 而节点未做好安全隔离，可能导致共享内存被滥用。建议在生产环境通过配置管理（如 ConfigMap）锁定此值。  
  - 新增的 `TransferRequest` 通过 Pydantic 验证，降低了非法 payload 注入的风险。  

**⚠️ 潜在风险**  

1. **库依赖不匹配**：Nixl 库（`dynamo.nixl_connect`）在某些平台或容器中可能不可用，导致 worker 在启动时报错。  
2. **资源泄漏**：若异常路径（如 `await transfer_future` 抛异常）未触发 `release_embeddings`，会残留未释放的共享内存句柄。  
3. **默认模式冲突**：默认 `NIXL_WRITE` 在 CI/测试环境会触发 segfault（已在测试里通过 `UCX_TLS=^mm` 暂时规避），可能影响新用户的即装即用体验。  
4. **并发释放 race**：`release_embeddings` 在 `finally` 里再次调用可能出现二次释放，如果接收端已提前回收张量会触发错误。  
5. **兼容性**：`EmbeddingTransferMode` 枚举值从字符串转为枚举在配置解析中增加了路径，旧版配置文件仍使用字符串时需确保 `validate` 步骤覆盖。  

**💡 关注建议**  

- **兼容性回退**：在 `backend_args` 中加入检测 Nixl 库可用性的兜底逻辑，若检测失败自动切换为 `LOCAL`，并记录警告日志。  
- **资源清理强化**：在 `EmbeddingsProcessor.process_embeddings` 的异常捕获块里，确保在任何异常后都调用 `release_embeddings`（使用 `try/finally` 包装）。  
- **安全加固**：在容器化部署时，显式限制共享内存的 `mmap` 权限，仅允许可信进程访问；同时把 `DYN_SGL_EMBEDDING_TRANSFER_MODE` 写入只读的配置文件或环境变量。  
- **文档与示例**：更新 README 与部署手册，明确说明三种模式的前置条件（Nixl Agent、UCX 配置等）以及在不满足条件时的推荐操作。  
- **测试覆盖**：完善单元/集成测试，分别在 `LOCAL`、`NIXL_WRITE`、`NIXL_READ` 三个模式下跑一次完整的 encode → worker → prefill 流程，确保张量释放路径始终被走通。  
- **监控埋点**：在 `embedding_sender.send_embeddings` 与 `embedding_receiver.receive_embeddings` 前后加入 NVTX/Prometheus 埋点，便于后期对共享内存传输时延进行量化评估。  

通过上述改进，SGLang 后端的多模态嵌入传输将更加灵活高效，同时也需要在部署、监控和安全层面做好相应的防护和验证。

## 🔍 技术洞察

| 维度 | 影响描述 |
|------|----------|
| **架构影响** | - **职责单一化**：`MultiModalEmbeddingLoader` 负责“请求→接收→缓存”全链路，避免在 `MultimodalPDWorkerHandler` 中散布业务逻辑，提升可读性与可维护性。<br>- **解耦前端解码**：`ImageLoader` 自己管理 NIXL 连接，调用方只需要传 `enable_frontend_decoding` 开关，削减跨模块的依赖（取消了在 `handlers.py` 中持有 `_nixl_connector`）。<br>- **统一异步桥接**：`run_async` 成为唯一的“同步调用异步”入口，避免每处手写 `asyncio.get_running_loop` + `ThreadPoolExecutor` 的重复实现。 |
| **性能影响** | - **连接创建成本**：`run_async` 在同步上下文中使用 `asyncio.run`（一次性创建并关闭事件循环），在异步上下文中使用 `ThreadPoolExecutor` 启动子线程并在子线程中再次调用 `asyncio.run`。这会比原来的 “在同一线程中直接 `await`” 多一次线程切换与循环创建，短期内对少量连接影响可忽略，但在高并发场景下可能产生 **线程池增长/竞争**。<br>- **缓存命中**：抽象层未改变缓存逻辑，本身不影响性能。<br>- **内存占用**：`ImageLoader` 现在在实例化时可能创建额外的 NIXL `Connector`（即使未使用），但仍为单例级别，影响微乎其微。 |
| **安全考虑** | - `run_async` 会在子线程中执行任意异步函数，若传入的函数内部泄露了 `await` 的上下文（例如调用 `loop.add_signal_handler`）可能导致 **信号处理冲突**。当前实现仅用于内部工具（连接初始化、读取），风险可控。<br>- 代码路径中未出现外部用户可注入的代码执行点，安全影响有限。 |
| **可维护性** | - **代码复用**：`run_async` 与 `MultiModalEmbeddingLoader` 提供统一入口，未来如果需要在其它组件使用相同模式，只需复用，不必复制逻辑。<br>- **测试简化**：测试直接对 `embedding_loader` 进行 `patch`，不再需要手动构造 NIXL 连接对象，降低测试维护成本。<br>- **类型清晰**：引入显式类后，参数签名更易阅读（`encode_worker_client`, `receiver`, `embedding_cache_manager`），有助于 IDE 自动补全和 static analysis。 |

## ⚠️ 潜在风险

1. **线程池滥用**  
   - `run_async` 每次在异步上下文中都会新建一个 `ThreadPoolExecutor`（默认 `max_workers=None` → 根据系统 CPU 数决定）并立即 `shutdown`。在高并发路径（如每个请求都要创建 NIXL 连接）会导致大量短命线程，增加调度开销，甚至触发 `OSError: [Errno 24] Too many open files`（因为每个线程会打开文件描述符）。  
   - **建议**：改为全局复用 `ThreadPoolExecutor`（单例）或直接使用 `asyncio.to_thread`（Python 3.9+），后者内部共用线程池，开销更低。

2. **异常传播与资源泄露**  
   - `run_async` 在子线程中调用 `asyncio.run`，若内部 `func` 抛出异常，异常会被 `Future.result()` 重新抛出，但线程的事件循环已被关闭，可能导致 **未关闭的 NIXL 句柄**（如果在 `func` 中创建了异步资源但未在异常路径里释放）。  
   - **建议**：在 `run_async` 中捕获 `Exception`，在必要时显式调用资源清理（如 `connector.shutdown()`），或让 `func` 自己管理 `try/finally`。

3. **并发安全**  
   - `ImageLoader` 在 `__init__` 中若 `enable_frontend_decoding=True` 会立即 `run_async(self._nixl_connector.initialize)`. 如果多个 `ImageLoader` 实例并行创建，可能对底层 NIXL 库产生竞争（如同一机器上多次创建 RDMA 通道）。目前代码未做锁保护。  
   - **建议**：将 NIXL 连接提升为进程/线程单例，或在 `ImageLoader` 中添加 `asyncio.Lock` 来序列化 `initialize` 调用。

4. **向后兼容性**  
   - 移除 `handlers.py` 中的 `self.enable_frontend_decoding`、`_nixl_connector` 成员，外部如果仍通过属性访问会触发 `AttributeError`。若有第三方插件或旧代码依赖这些属性，需要提供兼容 shim（如属性代理到内部 `ImageLoader`）。
