### fix(mocker): cut AIC replay overhead (#7692)
**SHA**: `4abafff` | 🔗 [查看提交](https://github.com/ai-dynamo/dynamo/commit/4abafffe66c960814305e90b5a48c4046da94074)

**🎯 变更类型**：性能优化 / Bug 修复  
**⚡ 重要程度**：🔴 高  

**📋 变更摘要**  
1. 在 `components/src/dynamo/mocker/aic_session.py` 中，用手动计算的方式替代对 `InferenceSession.run_static` 的完整 AIC 运行，显著削减了 mock 环境下的回放（replay）开销。  
2. 新增 `DEFAULT_STATIC_STRIDE`、模型/数据库缓存以及两套专用延迟预测函数 `_predict_context_latency` 与 `_predict_generation_latency`，实现对 context 与 generation 两阶段的逐 op 查询并累计。  
3. 统一了模型名称获取逻辑，防止部分模型缺失 `model_name` 时产生异常。

**🎯 影响范围**  
- `components/src/dynamo/mocker/aic_session.py`（核心 mocker 实现）  
- 依赖 AIC mocker 的单元测试、benchmark、CI 中的运行时成本评估模块  
- 任何直接或间接通过 `create_session` 创建 `AicSession` 的上层业务逻辑  

**🔍 技术洞察**  

| 维度 | 影响说明 |
|------|----------|
| **架构影响** | - 将 `AicSession` 与 `InferenceSession.run_static` 的强耦合拆除，改为 “查询‑累加” 的轻量实现。<br>- 引入对 `model.context_ops`、`model.generation_ops`、`model._nextn` 等内部属性的直接访问，增加了对模型实现细节的依赖。 |
| **性能影响** | - 省去了完整的 AIC 运行图（graph）构建、调度与执行，降低了 CPU/内存占用，预计在 CI/benchmark 环境中可将 mock 运行时间降低 **80%+**（具体收益视模型规模而定）。<br>- 通过固定步幅 `DEFAULT_STATIC_STRIDE = 32` 批量累计生成阶段的延迟，减少了循环次数。 |
| **安全考虑** | - 代码仅在本地 mock 环境执行，未涉及网络、文件或权限操作；安全风险极低。<br>- 唯一潜在安全点是使用 `getattr(op, "_name", "")` 访问私有属性，若底层库在异常情况下泄露或利用未公开字段，理论上不影响安全边界。 |
| **可维护性** | - 使用私有字段 (`_name`, `_nextn`) 使代码对 AIC 内部实现的变化敏感，未来库升级可能导致 `AttributeError`。<br>- 增加了两个较为复杂的手写预测函数，需配套单元测试以防止计算模型与真实运行结果不一致。 |
| **兼容性** | - 通过 `getattr(model, "model_name", None) or model_path` 兼容了缺失 `model_name` 的模型，提升了对不同模型实现的容错。<br>- 若模型不提供 `context_ops`/`generation_ops`，当前实现会抛出 `AttributeError`，需要在上层做好异常捕获或提前校验。 |

**⚠️ 潜在风险**  

1. **内部 API 变更**：依赖 `op._name`、`model.context_ops`、`model.generation_ops`、`model._nextn` 等私有字段，一旦 AIC 库改动，这部分代码会失效。  
2. **延迟预测误差**：手动累加的方式与真实 `run_static` 可能在并行调度、缓存效应等细节上产生偏差，导致基准测试结果偏离实际。  
3. **默认步幅硬编码**：`DEFAULT_STATIC_STRIDE = 32` 固定在代码里，若模型的最佳步幅不同，预测精度会下降。  
4. **异常处理不足**：`_predict_context_latency` 对 `isl <= prefix` 抛 `ValueError`，但外层 `predict_prefill` 未捕获，使用者若提供非法参数会直接中断。  

**💡 关注建议**  

- **增加防护层**：在 `AicSession.__init__` 中校验 `model` 是否具备 `context_ops`、`generation_ops`、`_nextn`，若缺失给出明确错误信息或回退到原有 `run_static` 实现。  
- **抽象私有属性访问**：封装为内部工具函数，如 `op_name(op)`、`model_name(model)`，便于后期适配 AIC API 变更。  
- **配置化步幅**：将 `DEFAULT_STATIC_STRIDE` 暴露为可配置项（环境变量 / config），并在文档中说明其调节意义。  
- **对比基准**：新增单元测试/benchmark，对比 `run_static` 与新预测函数在多种模型、不同 `isl/osl` 组合下的延迟差异，确保误差在可接受范围（如 <5%）。  
- **异常包装**：在 `predict_prefill` 与 `predict_decode` 中捕获并转化内部 `ValueError` 为业务层友好的错误码，防止 CI 因非法输入直接崩溃。  
- **文档更新**：在 README / developer guide 中说明 AIC mocker 的新实现原理、使用限制以及推荐的参数范围。  

通过上述措施，可在保持明显性能收益的同时，降低因内部实现变更带来的回归风险，提升代码的可维护性与可预测性。

### refactor: restructure planner package into subpackage hierarchy (#7689)
**SHA**: `95c4550` | 🔗 [查看提交](https://github.com/ai-dynamo/dynamo/commit/95c45509ea917c0ea5697edf259df10445eeffac)

**🎯 变更类型**：重构 / 架构变更  
**⚡ 重要程度**：🔴高  

**📋 变更摘要**  
本次提交将原本散落在 `dynamo/planner` 下的代码重新组织为 **四层子包**：`config`（配置与默认值）、`connectors`（各类连接器抽象与实现）、`core`（业务核心逻辑、预算、负载/吞吐预测等）以及 `monitoring`（Prometheus 指标、服务发现等）。相应地，所有原有的 `utils.*`、`defaults.*`、`planner_*` 等模块被迁移或删除，`__init__.py` 重新导出统一的公共 API。  
目标是提升模块边界清晰度、降低耦合度、便于后续功能扩展与单元测试。预期效果是 **代码可维护性显著提升**，而功能行为保持不变。

---

### 🎯 影响范围
- **核心 Planner 包**：`components/src/dynamo/planner/**`（全部子模块）  
- **依赖 Planner 的其它模块**：`components/src/dynamo/global_planner/*`, `components/src/dynamo/profiler/*`, `deploy/*`、`tests/planner/**`、`tests/global_planner/**` 等全部引用旧路径的地方。  
- **对外 API**：任何直接 `from dynamo.planner.utils.*`、`dynamo.planner.defaults`、`dynamo.planner.kube`、`dynamo.planner.scale_protocol` 的用户代码。

| **部署/CI** | - `deploy/operator/...` 中的 CRD 文档已同步更新，生成的 Go 代码也做了对应修改。<br>- 测试用例全量迁移，CI 通过。 |

### fix: Anthropic streaming double-parsing + reasoning_content roundtrip (#7358)
**SHA**: `f0d3ce6` | 🔗 [查看提交](https://github.com/ai-dynamo/dynamo/commit/f0d3ce639aff1585b3e7012c7e9e7aa70096327f)

**🎯 变更类型**：Bug修复  

**⚡ 重要程度**：🔴高  

**📋 变更摘要**  
- 修复 Anthropic 流式接口在使用推理 (reasoning) 模型时出现的“双重解析”问题，并确保 `reasoning_content` 能在请求‑响应往返中完整保留。  
- 在 Python 端新增 `_inject_reasoning_content`，将 `reasoning_content` 转换为 `<think>` 块并注入到 `content`，并为其编写完整单元测试。  
- 在 Rust 端对 Anthropic 处理逻辑、聊天模板元数据以及预处理器做了多处增强：  
  - 引入 `template_handles_reasoning` 标识，避免对已经在模板中处理 `reasoning_content` 的模型进行二次注入。  
  - 当模型配置了推理解析且客户端未显式禁用 thinking 时，自动向 `chat_template_args` 添加 `enable_thinking=true` 与 `truncate_history_thinking=false`。  
  - 移除原先在 Anthropic 路径中错误的 `OpenAIPreprocessor` 推理解析，防止流式输出被二次切分。  
- 更新 tokenizer 配置、模板渲染器以及相关单元测试，覆盖文本、分段、多模态以及 agentic 多轮调用场景。

- **核心库**：`components/src/dynamo/common/utils/input_params.py`（Python）  
- **LLM 预处理/模板系统**：`lib/llm/src/preprocessor/prompt/template.rs`、`formatters.rs`、`oai.rs`（Rust）  
- **Anthropic 端点实现**：`lib/llm/src/http/service/anthropic.rs`（Rust）  
- **推理解析器**：`lib/parsers/src/reasoning/mod.rs`（Rust）

### 🔍 技术洞察

| 维度 | 影响描述 |
|------|----------|
| **架构影响** | - 在 **聊天模板加载阶段** 增加 `template_handles_reasoning` 标记，形成“模板自处理 vs. 代码注入”两条分支路径。<br>- `InputParamManager` 现在负责将 `reasoning_content` 注入请求前的 **统一入口**，保证所有后端（包括 OpenAI、Anthropic）在渲染模板前都能得到统一处理。<br>- Anthropic 路径不再走 OpenAI 的 `OpenAIPreprocessor`，而是直接依据 `parsing_options` 与 `thinking` 参数决定是否向模板传递 `enable_thinking`。 |
| **性能影响** | - 注入逻辑为 **O(N)**（遍历消息列表），对单次请求的额外 CPU 开销极低。<br>- 通过 `chat_template_args` 自动开启 `enable_thinking`，避免在模板渲染后再次做字符串拼接，保持流式生成的 **低延迟**。<br>- 额外的 `<think>` 标记会略微增加 token 数量（约 2‑3 token/块），对吞吐量影响可忽略。 |
| **安全考虑** | - `reasoning_content` 源自模型输出，理论上不应直接暴露给用户，但注入后仍会进入模型的下一轮提示中，属于 **受信任的内部循环**；不引入外部注入风险。<br>- `template_handles_reasoning` 检测采用简单的字符串搜索 (`"reasoning_content"`)，若恶意模板故意隐藏该关键字仍会触发注入，导致 **重复 `<think>`**，可能被模型误解。建议在生产环境对自定义模板进行审计或使用更可靠的语法检测。 |
| **可维护性** | - 新增的 Python 注入函数与对应测试提升了 **可验证性**；Rust 端的 `template_handles_reasoning` 与 `inject_reasoning_content_into_messages` 实现保持了 **单一职责**。<br>- 代码变动集中在 **输入参数管理** 与 **模板渲染** 两个层面，改动范围明确，后续扩展新模型或新模板时只需关注这两个入口。 |
| **兼容性** | - 对已存在的 **支持 `reasoning_content` 的模型**（如 Nemotron、Qwen3）保持向后兼容：当模板内部已有对 `reasoning_content` 的引用，注入将被自动跳过。<br>- 对 **不支持推理的模型** 仍保持原有行为，因为 `template_handles_reasoning` 为 `false` 时才会进行注入。 |

### feat(runtime): scope MDC checksum validation to per-WorkerSet (#7368)
**SHA**: `d6136f4` | 🔗 [查看提交](https://github.com/ai-dynamo/dynamo/commit/d6136f4ac6f9c68b2be0b7a05642e550d898205a)

**🎯 变更类型**：功能增强 / 重构  
**⚡ 重要程度**：🔴高  
**📋 变更摘要**：  
1. 将模型的 MDC 校验范围从“全模型唯一校验和”放宽到“同一 WorkerSet（命名空间 + 模型类型）唯一”。  
2. 移除 `Model` 中的 `OnceLock`‑式全局校验和与对应错误，简化 `add_worker_set` 为不返回 `Result`。  
3. 新增 `Model::is_checksum_compatible` 与 `ModelManager::add_worker_set` 的调用方逻辑，使检查在 `ModelWatcher` 中按 WorkerSet 进行，允许不同命名空间在同一模型上并行部署不同版本（滚动更新）。  

**🎯 影响范围**：  
- `lib/llm/src/discovery/model.rs`（`Model` 结构、API）  
- `lib/llm/src/discovery/model_manager.rs`（错误枚举、`add_worker_set`、相关校验函数）  
- `lib/llm/src/discovery/watcher.rs`（WorkerSet 注册时的 checksum 验证）  
- 单元测试文件中涉及 checksum 的全部用例  

**🔍 技术洞察**：  
- **架构影响**：  
  - 由“全模型唯一校验和”→“同一 WorkerSet 唯一校验和”，实现了**按命名空间/模型类型的版本并行**，更好支持滚动升级与多租户部署。  
  - 去除 `OnceLock`，降低了跨线程一次性写入的同步成本，也消除了 `ModelManagerError::ChecksumMismatch` 错误路径，使模型管理层更简洁。  
  - 新的 `is_checksum_compatible` 直接在 `DashMap` 中查找对应 `WorkerSet`，使校验逻辑更局部化，降低了全局状态依赖。  

- **性能影响**：  
  - 删除 `OnceLock` 的原子写入与读取，稍微减少了竞争。  
  - `add_worker_set` 现在不返回 `Result`，避免了错误分支的额外分配；校验只在 `ModelWatcher` 中进行一次 `model.get` + `model.is_checksum_compatible`，整体开销微乎其微。  
  - 对常规路由/调度路径没有额外影响。  

- **安全考虑**：  
  - 该变更不涉及网络或权限安全，仅影响内部一致性。  
  - 放宽校验范围的风险在于**同一命名空间**若出现 checksum 不匹配的 WorkerSet，将不再抛出错误而是被日志记录并被拒绝加入（由 `ModelWatcher` 记录 error）。因此必须确保 `worker_set_key` 生成规则唯一且不被误用，以防出现跨命名空间误判。  

**⚠️ 潜在风险**：  
1. **向后兼容性**：外部代码若仍依赖 `ModelManager::add_worker_set` 返回 `Result`（或捕获 `ChecksumMismatch`），编译会失败。需同步更新调用方。  
2. **误判冲突**：若 `worker_set_key`（`namespace + model_type`）的生成逻辑被修改或出现冲突，可能导致本应不同的 WorkerSet 被误判为同一组，从而错误地拒绝合法的 checksum。  
3. **日志依赖**：校验失败现在只记录错误日志，未向调用方返回错误，若上层业务依赖错误返回做回滚或告警，需自行监听日志或实现额外的回调。  

**💡 关注建议**：  
- **代码迁移**：检查所有业务层或插件仍然使用 `ModelManager::add_worker_set` 的旧签名，统一改成无返回值的调用。  
- **监控**：在生产环境中为 `ModelWatcher` 的 checksum‑mismatch 日志添加监控/告警，防止因日志被忽略导致隐藏的版本不一致问题。  
- **文档**：更新项目文档，明确说明同一模型不同命名空间可以拥有不同的 MDC checksum，且同一命名空间内 checksum 必须保持一致。  
- **测试**：补充针对同一 `namespace`、不同 `model_type` 场景的兼容性测试，确保 `is_checksum_compatible` 的键生成逻辑覆盖所有组合。  
- **回滚路径**：如果需要在紧急情况下恢复全局唯一校验，可考虑通过 feature flag 暂时重新启用 `canonical_checksum` 检查，避免对已有部署产生突发影响。

### fix: race condition in multi modal PD worker (#7679)
**SHA**: `f6e5023` | 🔗 [查看提交](https://github.com/ai-dynamo/dynamo/commit/f6e5023be86385b4a32469b50aa651d182d10348)

**🎯 变更类型**：Bug 修复  
**⚡ 重要程度**：🟡 中  
**📋 变更摘要**：在 `multimodal_pd_worker_handler.py` 的异步生成器 `_generate_agg` 中，原先在 `yield` 之后才更新 `num_output_tokens_so_far`，导致 vLLM 在协程挂起期间就地修改 `response.outputs[0].token_ids`，产生竞争条件。现在先将 `chunk` 生成并在 **yield 前** 记录 token 数量，再返回 `chunk`，确保计数基于不被后续 mutation 影响的快照。  

**🎯 影响范围**：  
- `components/src/dynamo/vllm/multimodal_handlers/multimodal_pd_worker_handler.py`（唯一改动文件）

**💡 关注建议**  
1. **同步检查**：确认上层消费者在收到 `chunk` 后不再依赖 `response` 的实时 `token_ids`，否则需对其做深拷贝。  
2. **并发安全**：审视项目中其他 async generator 或回调，是否也存在 “yield 后立即使用被共享对象” 的模式，避免类似竞态。  
3. **测试覆盖**：新增或扩展多模态 PD worker 的并发单元测试，模拟高并发 token 生成情形，验证 `num_output_tokens_so_far` 与实际输出一致。  
4. **日志/监控**：由于 `logger.debug` 仍在 `yield` 前后执行，确认日志信息仍然准确，必要时可加入 `chunk` 的快照日志。  

总体来看，此次改动局部且安全，解决了已知的 race 条件，对外行为未变，只要确保上层不直接依赖 `response` 的即时 mutability，即可平滑上线。

### fix: make sleep/wake errors fatal in failover mode and correct RO memory accounting (#7681)
**SHA**: `a565f10` | 🔗 [查看提交](https://github.com/ai-dynamo/dynamo/commit/a565f105770f15edfc7fa726f2599fe681f36baa)

**🎯 变更类型**：Bug 修复 / 功能增强  
**⚡ 重要程度**：🟡 中

**💡 关注建议**

2. **并发安全**：`mark_resumed` 在恢复后立即调用，确保该方法是幂等且线程安全的，避免在高并发恢复路径出现竞争。  
3. **显存统计准确性**：`model_memory_usage` 取值来源于 `model_runner`，请确认所有模型实现都实现该属性，否则会默认 0 导致潜在低估。可以在初始化时加入断言或警告。  
4. **日志可观测性**：新增的显存日志信息对调试有帮助，建议在 `debug` 级别再输出更细粒度的 KV‑cache 与其它占用，以便运营监控。  
5. **回滚检查**：在非 Shadow 模式下，旧路径保持不变，确保单节点或普通多引擎部署不受影响。可在 CI 中加入针对 `is_shadow_mode()` 为 true/false 的两套集成测试。  

总体来看，此次改动提升了 failover 场景的鲁棒性并修正了只读权重的显存漏算问题，风险主要集中在异常处理与 `model_memory_usage` 的完整性，建议相应补全单元测试并做好监控。
