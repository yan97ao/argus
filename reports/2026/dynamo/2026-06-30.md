### fix(toolcalling): publish fragmented tool calls on finish instead of end-of-stream (#10988)
**SHA**: `370dc70` | 🔗 [查看提交](https://github.com/ai-dynamo/dynamo/commit/370dc707f057a951bfd595cf37f8de850b785fcf)

**🎯 变更类型**：功能增强 / Bug修复 / 重构  
**⚡ 重要程度**：🔴 高  

**📋 变更摘要**  
本次提交对 **Anthropic** 与 **OpenAI** 两套流式返回协议的 `stream_converter` 实现进行重大改造，解决了工具调用（tool‑call / function‑call）在 **碎片化 (fragmented) 参数** 场景下的错误行为。核心改动包括：  
1. **延迟发布** 工具块：不再在收到首个包含 `id/name` 的增量时立即产生 `content_block_start/stop`，而是等到后端明确的 *finish_reason*（`ToolCalls` / `FunctionCall`）或流结束时统一 flush。  
2. 新增 **状态结构**（`ToolCallState`、`FunctionCallState`）以及 **缓冲区**，累计参数片段直至完整后一次性生成 `content_block_start`、若干 `content_block_delta` 与 `content_block_stop`。  
3. 引入 **去重机制**，防止同一工具调用 ID 被重复发送。  
4. 为 OpenAI 协议实现了相同的“身份‑先行、参数‑后发”逻辑，并在流结束阶段统一补齐 `function_call_arguments.done` 与 `output_item.done`。  
5. 大量 **单元测试**、**端到端 HTTP 回放** 以及 **SSE 解析工具** 被加入，以验证碎片化、并行、仅标识、EOF fallback 等多种边缘情况。  

**🎯 影响范围**  
- `lib/llm/src/protocols/anthropic/stream_converter.rs`  
- `lib/llm/src/protocols/openai/responses/stream_converter.rs`  
- 相关的 `types`（`ChatCompletionMessageToolCallChunk`、`FinishReason`）  
- 业务层面：所有依赖 Dynamo **Anthropic Messages** 与 **OpenAI Responses** 接口的客户端（包括 `dynamo-cli`、SDK、外部集成）  
- 测试套件：新增 `anthropic_http_replay.rs`、`responses_http_replay.rs` 以及大量 fixture、通用 HTTP/SSE 辅助代码

### fix(py): preserve trace context in detached contexts (#10783)
**SHA**: `acc659b` | 🔗 [查看提交](https://github.com/ai-dynamo/dynamo/commit/acc659b615a240e54edb60ec7491ea87496b980e)

**🎯 变更类型**：功能增强  

**⚡ 重要程度**：🔴高  

**📋 变更摘要**：在 Python 绑定的 `Context` 类型中新增 `detached` 方法，使得在保持原有 trace context、span 与元数据快照的同时，能够创建一个拥有全新取消控制器和请求 ID 的上下文。此功能允许在 “分离” 的执行阶段（如解码、后处理）保持可观测性链路而不共享取消所有权。  

**🎯 影响范围**：  
- `lib/bindings/python/rust/context.rs`（核心 Rust 实现）  
- `lib/bindings/python/src/dynamo/_core.pyi`（Python 类型提示）  
- `lib/bindings/python/tests/test_context_metadata.py`（测试套件）  

**🔍 技术洞察**：  
- **架构影响**：  
  - 引入 `detached` 构造函数属于 **功能层面的扩展**，不改变现有的对象模型，只在 `Context` 上添加一种新的创建路径。  
  - 通过 `Arc::new(Controller::new(id))` 创建独立的取消控制器，确保 **分离上下文** 不会因父上下文的取消而提前终止，符合 “分层取消” 的设计理念。  
  - `metadata` 使用 `Arc<Mutex<_>>` 包装的快照，实现 **写时复制**（Copy‑on‑write）的轻量隔离，避免在并发环境下的竞争。  
- **性能影响**：  
  - 创建新 `Context` 时仅复制指针（`Arc`）和少量元数据（浅拷贝），开销极低，几乎与原有 `Context::new` 相同。  
  - 额外的 `Mutex` 包装在读取/写入元数据时会产生一次锁竞争，但在典型的单线程或低并发使用场景下可以忽略不计。  
- **安全考虑**：  
  - 没有引入外部输入的直接解析或系统调用，安全风险极低。  
  - 唯一需要注意的是 **元数据快照的可变性**：复制的是内部 `HashMap` 的快照，后续对原 `Context` 的元数据修改不会影响已分离的上下文，符合预期，但若业务依赖实时同步元数据，需要明确此行为。  

**⚠️ 潜在风险**：  
1. **资源泄漏**：分离的 `Context` 持有独立的 `Controller`，如果调用方忘记在使用完毕后调用 `stop_*` 系列方法，可能导致后台线程/计时器未被回收。  
2. **并发争用**：虽然元数据使用 `Mutex`，但高并发下大量 `detached` 调用仍会产生锁竞争，可能在极端负载下出现短暂的吞吐下降。  
3. **错误预期**：`first_token` 被显式置 `None`，意味着分离上下文不参与“首次 token”信号的传播。若上层逻辑依赖该信号进行资源分配或计费，需额外处理。  

**💡 关注建议**：  
- **文档**：在官方文档和 Python SDK 的 `Context` 类说明中明确 `detached` 的语义、使用场景以及 **不共享取消所有权**、**不参与 first‑token** 的细节。  
- **资源管理**：推荐在 `detached` 返回的 `Context` 使用完毕后显式调用 `stop_generating()`（或相应的 `kill`）进行清理，或在 `Context` 实现 `Drop` 时确保底层 `Controller` 能够自行超时回收。  
- **并发测试**：加入压力测试，模拟多线程创建大量 `detached` 上下文，验证 `metadata` 锁的竞争度，并评估是否需要改为 `RwLock` 或无锁数据结构（如 `dashmap`）以提升并发写入性能。  
- **监控**：在追踪系统（OpenTelemetry 等）中验证 `detached` 上下文的 trace 链路是否如预期保持父子关系，防止因实现错误导致孤立 span。  
- **回归测试**：继续扩展测试覆盖 “取消 propagation” 场景，确保父上下文停止后，分离上下文仍保持运行；而分离上下文自行停止后，不会影响父上下文。  

总体来看，此次变更为 Python SDK 引入了更灵活的上下文管理能力，对可观测性和资源隔离都有积极意义，风险可通过文档、测试与资源回收机制得到有效控制。

**🔍 技术洞察**  

- **架构影响**  
  - **解耦校验与配置**：通过 `validate_no_unsupported_fields_with_ignore` 将配置读取与业务校验分离，使得单元测试能够直接控制“忽略”行为，提升可测试性。  
  - **全局开关**：使用 `LazyLock<bool>` 将环境变量在首次使用时解析一次，避免每次请求都进行环境查询，符合“懒加载+只读全局”模式。  
  - **向后兼容**：默认行为保持不变（不忽略），已有调用 `validate_no_unsupported_fields` 的路径仍然得到相同的错误检查，保证向后兼容。  

- **性能影响**  
  - **微小提升**：在开启忽略模式后，校验路径仅会过滤 `PASSTHROUGH_EXTRA_FIELDS`，省去对所有未知字段的错误构造与 `anyhow::bail!`，对高并发请求的微量 CPU 开销有正向影响。  
  - **一次性代价**：`LazyLock` 初始化一次读取环境变量，成本极低，且后续请求不再产生额外开销。整体对吞吐量的影响可以忽略不计。

  - **潜在风险**：忽略未知字段会导致前端请求中潜在的恶意或错误参数被 silently 丢弃，可能隐藏 API 使用者的错误配置，也可能让后端失去对异常输入的可审计性。
