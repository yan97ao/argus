### feat: vLLM multinode elastic EP scaling support test (#8183)
**SHA**: `f460f4e` | 🔗 [查看提交](https://github.com/ai-dynamo/dynamo/commit/f460f4e25e30253f8094b8e467cbd369a0d09ca8)

**🎯 变更类型**：功能增强  
**⚡ 重要程度**：🔴高  
**📋 变更摘要**：新增两套 Kubernetes 部署模板和对应的 Bash 脚本，用于在裸 vLLM 环境下测试多节点 Elastic EP（专家并行）跨节点弹性扩缩容能力。通过 Ray 集群、NCCL 环境变量以及 vLLM 原生 `POST /scale_elastic_ep` 接口，验证从 2→3→4→… 的 DP（data‑parallel）规模变化是否能够成功调度到第二节点的 GPU 上，从而评估 vLLM 本身对跨节点 Elastic EP 的支持程度。  

**🎯 影响范围**：  
- `tests/fault_tolerance/deploy/templates/vllm/` 目录下的 YAML 与 Shell 脚本。  
- CI/CD（如果启用 fault‑tolerance 测试）将会执行该多节点弹性扩缩容场景。  
- 与核心业务代码无直接耦合，仅影响测试套件与相关 CI 流程。  

**🔍 技术洞察**：  

- **架构影响**  
  - 引入 **裸 vLLM + Ray** 双节点拓扑的测试框架，独立于 Dynamo Operator。  
  - 使用 **Headless Service** 实现 leader‑worker DNS 通信，确保跨 Pod 能够通过固定域名访问。  
  - 通过 **NCCL 环境变量**（`NCCL_IB_DISABLE=1`、`NCCL_SOCKET_IFNAME=eth0`）强制使用 TCP，规避集群内部缺乏可路由的 RoCE GID 的网络限制。  
  - 脚本中显式 **等待 leader 健康** 后才加入 Ray，确保 DP Workers 初始全部落在 leader 节点，为后续弹性扩容提供“闲置” GPU 资源。  

- **性能影响**  
  - 仅为测试资源，不会进入生产代码路径，对运行时性能无直接影响。  
  - 测试过程中会 **启动两个 Ray 节点、加载模型、执行多次弹性扩缩容**，对 CI 资源（GPU、网络、磁盘）有较高消耗，建议在专用的硬件环境或 nightly CI 中运行。  
  - 通过 `nvidia‑smi` 快照与 `ps` 检查，能够捕获实际 GPU 利用率与 Ray Actor 分布，为后续性能基准提供数据。  

- **安全考虑**  
  - 使用 `envFrom.secretRef` 将 HF Token 注入 leader Pod，保持密钥在 Kubernetes Secret 中，不会硬编码在源码。  
  - 脚本通过 `kubectl exec` 与 `curl` 与集群交互，需在拥有相应 RBAC 权限的 CI ServiceAccount 中运行；若在不受信任的环境下执行，可能泄露集群内部地址或模型下载凭证。  
  - 脚本内未对 `curl` 响应进行签名校验，属于普通的 HTTP 通信，风险可接受。  

**⚠️ 潜在风险**：  
1. **硬编码节点选择器**（`kubernetes.io/hostname`）使测试只能在特定节点名称的集群上通过，导致 CI 不可移植。  
2. **NCCL 配置依赖特定网络拓扑**（禁用 IB、使用 eth0），在不同 CNI 或跨云提供商环境下可能失效。  
3. **等待逻辑（轮询）** 与 **Python 字节码编译延迟** 最高可达 10 分钟，若集群出现异常会导致 CI 超时或堵塞。  
4. **资源泄露**：脚本在结束时仅 `kill $PF`，未显式 `ray stop` 或 `kubectl delete` 资源，可能残留 Ray 进程或占用端口。  
5. **模型下载** 采用 `nvcr.io/...:main` 镜像，若镜像更新频繁或失效，测试会因拉取失败而中断。  

**💡 关注建议**：  
- **参数化节点选择**：改用标签选择（`nodeSelector: {gpu-node: "true"}`）或通过环境变量/CI 参数传入，使测试在不同集群上均可运行。  
- **抽象 NCCL 配置**：提供一个可选的 `use_infiniband` 开关，依据集群检测自动决定是否禁用 IB，避免在具备可路由 GID 时失去高性能网络优势。  
- **完善清理流程**：在脚本末尾加入 `ray stop --force`、`kubectl delete pod ...` 或 `kubectl delete -f yaml`，确保测试结束后资源全部回收。  
- **超时与重试机制**：对 `wait_worker_in_ray`、`curl /health` 等关键等待添加更细粒度的 timeout 与错误日志，以免 CI 长时间卡死。  
- **CI 集成**：将此测试标记为 *slow* / *gpu‑heavy*，仅在 nightly 或专用 GPU 集群中执行；并在 CI 报告中输出关键的 `nvidia‑smi` 快照与 Ray 资源分配图，以便快速定位扩容失败的根因。  
- **安全审计**：确认 `hf-token-secret` 的 RBAC 范围仅限该命名空间，并在 CI 完成后立即撤销对该 Secret 的访问。  

---  

以上分析从架构、性能、安全及运维角度评估了本次新增多节点 Elastic EP 扩容测试的潜在影响，并提供了降低风险、提升可维护性的实操建议。祝测试顺利，期待在未来的正式发布中看到对 vLLM 跨节点 Elastic EP 的原生支持。

### fix: propagate vLLM --stream-interval to Dynamo frontend (#8101)
**SHA**: `714b423` | 🔗 [查看提交](https://github.com/ai-dynamo/dynamo/commit/714b42343917d5033ec204ec8b992d356ea3b7e5)

**🎯 变更类型**：功能增强 / Bug修复  
**⚡ 重要程度**：🔴高  
**📋 变更摘要**：  
- 在 vLLM 前端（`vllm_processor.py`）中加入对 `stream_interval` 的解析逻辑，支持环境变量 `DYN_VLLM_STREAM_INTERVAL`、后端 `runtime_config` 中的 `stream_interval`，并在缺省时回退到默认值 20。  
- 在 `args.py` 的交叉校验中将原来的警告改为信息提示，明确该参数将被前端使用并可通过环境变量覆盖。  
- 在模型注册入口 `main.py` 中将 `engine_args.stream_interval` 写入 `runtime_config` 的 `engine_specific` 字段，使后端能够将该值传递给前端。  

这些改动统一了 `--stream-interval` 参数在 Dynamo 生态链路中的传播，使前端的输出处理器能够按照用户期望的间隔进行流式输出。

---

**🎯 影响范围**：  
- `components/src/dynamo/frontend/vllm_processor.py`（前端流式输出处理）  
- `components/src/dynamo/vllm/args.py`（参数校验与日志）  
- `components/src/dynamo/vllm/main.py`（模型注册与 runtime_config 传递）  

涉及的主要模块：`frontend`、`vllm` 子系统、`runtime_config` 管理。

---

**🔍 技术洞察**  

- **架构影响**：  
  - 新增了前端对后端 `runtime_config`（`engine_specific`）的读取，形成了前后端之间的 **配置同步** 链路。  
  - 通过环境变量提供最高优先级的覆盖路径，保持了 **“外部化配置”** 的灵活性。  
  - 对 `runtime_config` 的写入仍保持原有的 JSON‑string 方式，兼容现有 Rust 端的 `serde_json` 解析。

- **性能影响**：  
  - `stream_interval` 决定了输出缓冲的批次大小，直接影响 **流式响应的延迟** 与 **吞吐量**。  
  - 正确传播后，用户可通过调大该值降低网络/IO 调用频率，提升整体吞吐；调小则提升响应实时性。  
  - 代码加入了 `max(1, int(...))` 防御，避免因 0 或负数导致无限循环或异常阻塞。

- **安全考虑**：  
  - 环境变量 `DYN_VLLM_STREAM_INTERVAL` 由运行时进程读取，属于本地可信输入，风险极低。  
  - `runtime_config.set_engine_specific` 仍采用字符串写入，未引入二进制或脚本执行路径，未产生新攻击面。  
  - 添加的异常捕获（`TypeError`, `ValueError`）防止恶意或误配置导致服务崩溃。

---

**⚠️ 潜在风险**  

1. **配置不一致**：如果后端和前端分别读取不同来源的 `stream_interval`（例如环境变量 vs. runtime_config），可能出现 **间隔不匹配**，导致输出顺序或速率异常。  
2. **默认值硬编码**：前端默认值写死为 20，若后端默认值在未来改动，需要同步更新，否则会出现隐藏的偏差。  
3. **回退逻辑**：当后端提供的 `stream_interval` 为非整数或非法字符串时，仅会记录警告并使用前端已有的 `self.stream_interval`（默认 20），可能导致 **意外的高延迟**。  
4. **旧版客户端兼容**：某些依赖默认 `stream_interval=1`（如原 vLLM 的内部实现） 的工具链可能在迁移后出现 **输出过慢** 的感知。

---

**💡 关注建议**  

- **测试覆盖**：为 `vllm_processor.py` 增加单元测试，验证三条路径（env → runtime_config → default）的优先级和异常回退。  
- **文档同步**：在 Dynamo 官方文档中明确说明 `--stream-interval` 的传播机制、默认值以及 `DYN_VLLM_STREAM_INTERVAL` 的使用方法。  
- **配置统一**：考虑在 `runtime_config` 中加入统一的 “fallback” 机制，使前端只读取一次 `runtime_config`，避免多点分支。  
- **监控报警**：在生产环境加入监控，捕获 `logger.warning` 中的 “Invalid stream_interval” 信息，以便快速定位错误配置。  
- **向后兼容**：如果已有用户依赖 `stream_interval=1` 的行为，可在发布说明中提供迁移指南或提供 `--disable-stream-interval-propagation` 开关以保持旧行为。

### fix(sglang): add enable_trace to diffusion worker ServerArgs stub (#8332)
**SHA**: `1d154dc` | 🔗 [查看提交](https://github.com/ai-dynamo/dynamo/commit/1d154dcbecc6ae933b1771ccc6eefaa5e737b91f)

**🎯 变更类型**：Bug 修复  
**⚡ 重要程度**：🟡 中  
**📋 变更摘要**：为 SGLang Diffusion Worker 增加 `enable_trace` 参数的默认填充。`args.py` 在构造 `ServerArgs` 时写入 `enable_trace`，并在 `handler_base.py` 中使用 `getattr(..., "enable_trace", False)` 防止属性缺失导致异常。  

**🎯 影响范围**：  
- `components/src/dynamo/sglang/args.py`（ServerArgs 结构体创建）  
- `components/src/dynamo/sglang/request_handlers/handler_base.py`（所有基类请求处理器）  

**💡 关注建议**：  
1. **兼容性**：新字段默认 `False`，对已有部署无行为变化；但若外部工具依赖 `ServerArgs` 的 schema，需同步更新文档/API。  
2. **日志与监控**：`enable_trace` 目前仅在处理器内部保存，若后续计划输出 tracing 信息，请确保相应的日志级别或监控管线已准备。  
3. **测试**：加入包含 `--enable-trace` 参数的单元/集成测试，验证在开启和关闭两种情况下工作流不抛异常。  
4. **代码风格**：使用 `getattr(..., "enable_trace", False)` 已保证向后兼容，建议在其他可能访问 `ServerArgs` 的位置统一该写法，防止未来属性缺失导致崩溃。  

总体而言，此次修改稳妥地引入了可选的 tracing 开关，对现有功能影响极小，但后续若真正利用 tracing，需补充实现和文档。
