### test: add reasoning parser parity matrix (part 3) (#9781)

**🎯 变更类型**：功能增强 / 重构 / 架构变更 / 测试与工具升级  

**⚡ 重要程度**：🔴高  

**📋 变更摘要**  
1. 在 `lib/bindings/python/rust/parsers.rs` 中为 Dynamo 的 **Reasoning** 解析器新增两套统一的 PyO3 接口 `parse_reasoning_batch` 与 `parse_reasoning_stream`，支持一次性解析和增量流式解析，并能够在“已在 reasoning 模式”时强制开启解析。  
2. 为 Python 侧提供对应的 `.pyi` 类型声明与单元测试，验证基本闭合标记、`in_reasoning` 启动模式以及 token‑chunk 长度校验。  
3. 在底层 Rust 实现里为所有 ReasoningParser 实现（`BasicReasoningParser`、`Gemma4ReasoningParser`、`GraniteReasoningParser`）实现 `finish_reasoning_stream`，保证流式结束时缓冲区能够被正确冲刷，避免遗漏未闭合的 delimiter。  
4. 对 `OpenAIPreprocessor` 注释进行细化，明确不同模型的 reasoning 标记。  
5. 扩展 parity‑testing 基础设施：  
   - 新增 `reasoning` 子表生成脚本、数据结构与 HTML 渲染逻辑。  
   - 新增大量 UI/样式（tooltip、颜色、信息面板）以及工具函数 `colorize_markup`、`colorize_stream_deltas` 等，以在 parity 表中更好地展示 markup、漏泄与研究标记。  
6. 增加了对 DeepSeek‑V3 系列、Gemma‑4、MiniMax‑Append‑Think 等模型的合理映射，使 Reasoning 表能够显示这些模型共享同一底层 parser 的情况。  
7. 更新了 test 围绕 `parse_reasoning_*` 的 Python 单元测试，并为 parity‑reasoning、tool‑calling 表提供了完整的生成、渲染与校验脚本。  

**🎯 影响范围**

### feat(kv-router): refresh overlap scores at dequeue time (#9663)
**SHA**: `ed2f3e1` | 🔗 [查看提交](https://github.com/ai-dynamo/dynamo/commit/ed2f3e1eb653738443246738b6d39c3e1de04ea2)

**🎯 变更类型**：功能增强 / 架构变更  
**⚡ 重要程度**：🔴高  

**📋 变更摘要**  
1. 为 KV‑Router 引入 **dequeue‑time overlap‑score 刷新**（`OverlapScoresRefresh`）。  
2. 在调度队列中记录请求的块哈希、入队时间，并在满足等待阈值时调用刷新器重新查询 KV‑Indexer，得到最新的 `tier_overlap_blocks`、`effective_overlap_blocks` 与 `effective_cached_tokens`。  
3. 为 `LocalScheduler` 与 `SchedulerQueue` 增加泛型 `RF`（刷新器）以及对应的构造函数族，默认使用 `NoopOverlapScoresRefresh` 保持向后兼容。  
4. 对外暴露 `schedule_with_block_hashes` 接口，使上层 `KvRouter` 能在调度时把块哈希传递下去。  
5. 新增环境变量 `DYN_ROUTER_OVERLAP_REFRESH_AFTER_SECS`（默认 10 s）控制刷新阈值，文档同步更新。  
6. 完整单元测试覆盖刷新逻辑、阻塞刷新以及刷新后重新排队的行为。

## 🔍 技术洞察

| 维度 | 影响 |
|------|------|
| **架构影响** | - 在调度层引入 **可选的刷新模块**（plug‑in），通过泛型 `RF` 将刷新器与调度器解耦。<br>- `SchedulerQueue` 现在持有 `Option<Arc<RF>>` 与阈值 `Duration`，并在 `run → handle_update` 中执行刷新。<br>- `KvRouter` 在启动时根据 `Indexer` 类型（`KvIndexer` / `Concurrent`）创建 `KvRouterOverlapRefresher`，并把 `Arc<_>` 注入调度器。<br>- 通过 `supports_overlap_refresh()` 与 `prepare_block_hashes_for_refresh` 保证在不支持刷新时不额外分配或传递块哈希。 |
| **性能影响** | - **正向**：对长时间等待的请求可在出队前使用最新的缓存状态，降低因缓存失效导致的错误 worker 选择，从而提升整体吞吐与延迟。<br>- **负向**：每次刷新会触发一次 KV‑Indexer 查询（`find_matches_by_tier`），可能导致额外的 I/O/CPU 开销。刷新只在 **阈值**（默认 10 s）后触发，且在 **无块哈希** 或 **刷新器未配置** 时直接跳过，控制了成本。<br>- 通过 `await` 并行执行刷新（仅在 `handle_update` 中），不会阻塞调度器对其他请求的处理。<br>- 新增 `enqueue_with_block_hashes` 复制块哈希（`Vec<LocalBlockHash>`），在未开启刷新时会被 `None` 丢弃，几乎不产生额外开销。 |
| **安全考虑** | - `OverlapScoresRefresh::refresh` 被要求 **不 panic**、**可安全克隆**，实现（`KvRouterOverlapRefresher`）捕获 Indexer 错误并返回 `None`，保证刷新失败不会导致调度器崩溃或丢弃请求。<br>- 环境变量解析严格，非数值或负数都会回退到默认或禁用刷新，防止配置注入异常。<br>- 所有跨线程共享的对象使用 `Arc` + `Send + Sync` 约束，保持 Rust 的所有权安全。 |
| **可维护性** | - 通过 **trait** 和 **默认 no‑op 实现**，原有代码路径保持不变，兼容性高。<br>- 代码分层明确：`overlap_refresh`（刷新抽象） → `queue`（调度时点调用） → `scheduler`（接口暴露） → `kv_router`（业务层使用）。<br>- 单元测试覆盖关键路径，包括刷新触发、刷新后重新排队、并发刷新阻塞等，降低未来回归风险。 |
| **兼容性** | - 默认构造函数 `LocalScheduler::new`、`SchedulerQueue::new` 仍使用 `NoopOverlapScoresRefresh`，对现有部署无任何行为变更。<br>- 新增的 `new_without_overlap_refresh`、`new_with_overlap_refresh` 仅是显式命名的包装，保持 API 稳定。<br>- `KvRouter` 在启动时自动决定是否使用刷新器，用户无需更改代码，只需在需要时通过环境变量调节阈值。 |

## ⚠️ 潜在风险

| 风险点 | 描述 | 缓解措施 |
|--------|------|-----------|
| **刷新器查询成本** | 对高并发、块哈希数量大的请求，频繁刷新可能导致 KV‑Indexer 负载激增。 | - 阈值默认 10 s，且仅在 **请求排在队首** 时刷新，降低并发刷新次数。<br>- 可通过 `DYN_ROUTER_OVERLAP_REFRESH_AFTER_SECS=0` 完全关闭。<br>- 监控 `overlap_refresh_after` 与实际刷新调用次数，必要时调高阈值或实现更高效的索引查询。 |
| **块哈希泄露** | `LocalBlockHash` 通过 `Arc` 传递给刷新器，若实现不当可能泄露内部 KV‑cache 结构信息。 | - `LocalBlockHash` 为轻量 opaque 类型，仅是 `u64`（或类似），不携带敏感数据。<br>- 刷新器实现已限制在 `KvRouterOverlapRefresher`（内部），外部实现仍受 `Send + Sync` 约束。 |
| **竞争导致请求错失调度** | 刷新期间工作者状态可能再次变化，导致刷新后仍无法调度，从而再次进入队列，形成 “刷新—再入队” 循环。 | - 刷新后若 **所有工作者仍忙**，代码会把请求 **重新压回队列**（保持原 key），并在下次 `handle_update` 再次检查。<br>- 该循环只会在极端高负载、刷新阈值过低的情况下出现，可通过增大阈值或提升工作者容量解决。 |
| **兼容性回退** | 旧的 `SchedulerQueue::new` 已被标记为 deprecated，外部库如果直接调用该函数可能失去刷新能力。 | - 保留 `new` 与 `new_with_overload_provider` 仍指向 **no‑op** 实现，完全兼容旧代码。<br>- 文档已说明新函数 `new_with_overlap_refresh` 为推荐入口。 |
| **测试环境时间控制** | 单元测试依赖 `tokio::time::advance`，在实际运行时若未开启 `start_paused` 可能导致不确定行为。 | - 生产代码不受此机制影响，只在测试中使用。<br>- 代码路径已在生产环境下使用真实 `Instant::now()`，不受 `paused` 状态影响。 |

### 🔍 技术洞察

| 维度 | 影响描述 |
|------|----------|
| **架构影响** | - 将 DP 相关的计算抽离为 `capacity` 公共模块，提升代码可维护性，降低不同子系统之间的实现差异。<br>- `runtime_capacity()` 统一返回 `RuntimeCapacity`，供 Engine、Register、Publisher 统一消费，削减了以前在每个模块里复制的 `dp_size / nnodes / node_rank` 逻辑。 |

| **可维护性** | - 统一入口后，后续对 DP 计算的修改只需在 `capacity.py` 中完成，降低了重复代码和潜在的疏漏。<br>- 增加了明确的错误信息（如 `RuntimeError("Engine not initialized")`），提升调试效率。 |
| **兼容性** | - 对外 API（EngineConfig、Publisher）保持不变；仅内部数值来源更换为统一实现，理论上不会破坏已有用户的行为。<br>- 对未开启 DP 的单节点部署仍返回 `(0,1)`，保持向后兼容。 |
| **监控/可观测性** | - `kv_metrics_block_values` 统一在所有后端将 KV block 数量转化为 “块” 单位，避免了 SGLang 与 TRT‑LLM 之间因 `page_size` 不一致导致的监控偏差。<br>- `per_rank_kv_blocks` 为 vLLM 增加了边界检查与警告日志，方便运维快速定位 DP 与 KV block 不匹配的异常。 |

---

### ⚠️ 潜在风险

1. **属性缺失或类型不匹配**  
   - 代码直接通过 `getattr(server_args, "...", default)` 读取参数，若外部传入的 `server_args` 对象缺少这些属性或属性类型非整数（如字符串），可能导致异常或错误的 DP 范围。  
2. **`per_rank_kv_blocks` 返回 `None`**  
   - 在 vLLM 启动阶段，如果 `num_gpu_blocks` 为 `None`，会触发 `RuntimeError("per-rank KV block count is not set")`，导致服务启动失败。需要确保 vLLM 始终返回有效的 block 数量。  
3. **DP size 为 0 或负数**  
   - 虽然 `dp_size` 默认 `1`，但若用户手动设置为 `0`/负数，`local_dp_rank_bounds` 仍会返回 `(0,1)`，这可能与用户期望不符。最好在参数校验层提前捕获。  
4. **日志噪声**  
   - `per_rank_kv_blocks` 在不整除情况下会产生警告日志，若部署大规模 DP（如 128‑rank）且总块数不恰好可整除，日志量可能增大。可考虑通过环境变量调节日志级别。
