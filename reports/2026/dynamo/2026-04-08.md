### fix: dp_rank always 0 in non-KV router mode (#7984)
**SHA**: `a210efa` | 🔗 [查看提交](https://github.com/ai-dynamo/dynamo/commit/a210efa6dc58490fc58942ec7b7ce6997932568b)

**🎯 变更类型**：Bug修复  
**⚡ 重要程度**：🔴高  

**📋 变更摘要**  
- 修复在非 KV‑router 模式下 `dp_rank` 始终为 0 的错误。  
- 将与 `dp_rank` 相关的返回值改为 `Option<u32>`，在 C 绑定层统一转为 `u32::MAX`（表示“未定义”），并在 Rust 路由实现中传播 `None`。  
- 更新路由器内部逻辑以避免对未设置的 `dp_rank` 进行错误的默认使用，提升路由决策的准确性。

**🎯 影响范围**  
- `core` 模块的路由层：`RouterHandles::query_prefill_worker`、`PrefillRouter`（`execution.rs`、`mod.rs`、`types.rs`）。  
- C 绑定层：`lib/bindings/c/src/lib.rs`。  
- 任何直接使用 `dp_rank`（如监控、日志或自定义调度策略）的调用方。

**🔍 技术洞察**  

- **架构影响**：  
  - `dp_rank` 从必定存在的 `u32` 变为可选 `Option<u32>`，导致 API 向下兼容性破坏。  
  - 通过在 C 绑定层使用 `u32::MAX` 作为哨兵值，将可选语义平滑映射到 FFI 接口，保持外部 C 调用的二进制兼容。  
  - 路由器内部在选取预填充 worker 时不再强制使用 `0`，从而在 KV‑router 与非 KV‑router 两种路径之间保持行为一致。

- **性能影响**：  
  - 改动仅涉及少量 `Option` 包装/解包，额外的分支判断对吞吐量和延迟影响可以忽略不计。  
  - 在 C 端使用 `unwrap_or(u32::MAX)` 也几乎没有运行时开销。

- **安全考虑**：  
  - 之前的 `dp_rank = 0` 可能导致错误的调度路径，间接影响资源隔离和服务质量（QoS），属于业务层安全风险。  
  - 采用 `Option` 可显式区分 “未设置” 与 合法的 `0`（如果业务上 `0` 有意义），降低误用风险。  
  - 使用 `u32::MAX` 作为哨兵值时需确保后续逻辑在 C 端不会把它当作合法 rank 进行比较或哈希，否则可能引入整数溢出或错误路由。

**⚠️ 潜在风险**  
1. **API 破坏**：其他 Rust 代码或外部语言绑定仍然期待 `Result<(u64, u32)>`，会因类型不匹配编译失败。  
2. **哨兵误用**：C 端若未对 `u32::MAX` 做特殊处理，可能把它误认为真实的 `dp_rank`，导致调度异常。  
3. **日志/监控**：现有监控/日志系统可能假设 `dp_rank` always present; 需要更新解析逻辑以适配 `Option`。  
4. **错误传播**：如果后续业务层仍然使用 `unwrap()` 直接解包，仍会在 `None` 场景下 panic。

**💡 关注建议**  
- **代码层面**：  
  - 完全审查所有调用 `query_prefill_worker`、`prefill_router` 接口的地方，确保已经改为接受 `Option<u32>`。  
  - 在 C 绑定的文档中明确说明 `dp_rank == UINT32_MAX` 表示 “未定义”。  
  - 为防止误用，建议在 Rust 侧提供 `fn dp_rank_or_default(&self) -> u32` 之类的辅助方法。

- **测试层面**：  
  - 增加针对非 KV‑router 模式的单元/集成测试，验证 `dp_rank` 为 `None` 时路由行为与预期一致。  
  - 添加 FFI 测试，确保在 C 调用返回 `UINT32_MAX` 时上层逻辑不会崩溃。

- **运维层面**：  
  - 更新监控告警规则，排除 `dp_rank == UINT32_MAX` 的异常报告。  
  - 在发布说明中突出该兼容性变化，提醒使用者迁移代码。

- **长期建议**：  
  - 考虑在公共 API 中统一使用 `Option<u32>`，并逐步淘汰哨兵值，以提升类型安全性。  
  - 若业务上需要区分 “未设置” 与 “显式 0”，应在 protobuf/JSON schema 中明确 optional 标记。

### feat(kv-router): split Dynamo-native remote indexer [DYN-2593] (#7973)
**SHA**: `49eb397` | 🔗 [查看提交](https://github.com/ai-dynamo/dynamo/commit/49eb397a5e6908e4eb58c0f111f8f2ac7357d4ce)

**🎯 变更类型**：功能增强 / 架构改动  
**⚡ 重要程度**：🔴 高  

**📋 变更摘要**  
本次提交在 **kv‑router** 中引入了 *Dynamo‑native Remote Indexer* 的概念，并实现了两套新功能：  
1. `--serve-indexer`：在 **frontend / router** 进程上暴露本地 KV 索引器的查询（`kv_indexer_query`）以及（在近似模式）写入（`kv_indexer_record_routing_decision`）接口，使其能够被同一 namespace 内的其他 router/frontend 远程查询。  
2. `--use-remote-indexer`：在消费侧 router/frontend 中关闭本地 radix‑tree，改为通过请求平面（request plane）调用前述服务获取 overlap 分数。  
为此，代码库在多个层面做了拆分与重构：  
- 配置结构从 `remote_indexer_component`（字符串）迁移到布尔开关 `use_remote_indexer` / `serve_indexer`。  
- `kv-router` 的内部索引器实现统一为 `Indexer::KvIndexer`（本地）与 `Indexer::Remote`（远程）两种；对应的 RPC 逻辑、度量、异常处理均已更新。  
- 移除原有的 **runtime‑indexer**（基于 Dynamo 运行时的发现/事件平面）实现，改为 **pure request‑plane**（亦支持近似模式）实现；相应的 Cargo feature `kv-indexer-runtime` 被删除。  
- Dockerfile、Python 绑定、文档、测试均同步更新，以覆盖新拓扑（事件驱动、近似、远程）三种模式的使用场景。  

**🎯 影响范围**  
- `components/src/dynamo/common/configuration/groups/kv_router_args.py`、`frontend_args.py`、`router/args.py`（配置层）  
- `lib/bindings/python`（Python 绑定、入口、Prometheus 指标）  
- `lib/kv-router`（核心 Rust 实现：`indexer/kv.rs`、`indexer/sharded.rs`、调度配置）  
- `lib/kv-router/src/standalone_indexer/*`（原 runtime‑indexer 代码被删除）  
- `container/templates/wheel_builder.Dockerfile`（特性开关）  
- 文档 `router-guide.md`、`standalone-indexer.md`（说明更新）  
- 测试套件 `tests/router/*`（新增远程索引器验证）

## 🔍 技术洞察

### 架构影响
| 维度 | 变化 | 结果 |
|------|------|------|
| **部署拓扑** | 引入 *served indexer*（在 router/frontend 中提供查询/写入服务）与 *remote indexer client*（消费端调用） | 1）消除了对独立 `dynamo.indexer` 微服务的硬性依赖；2）可以在同一进程内实现 KV‑indexer 与 router 的“一体化”，降低网络跳数；3）支持 **近似模式**（无 KV 事件）下的单实例查询/写入路径。 |
| **发现机制** | 完全去除基于 `dynamo-runtime`（MDC/Discovery）的 worker 自动注册与事件平面订阅；只保留原有的 **event‑driven**（KV 事件）模式下的 ZMQ/NATS 事件平面。 | 简化了运行时依赖，降低了运行时启动顺序的耦合；但也失去了一键通过 `--dynamo-runtime` 启动的“全自动”模式，用户必须手动使用 `--serve-indexer` / `--use-remote-indexer`。 |
| **组件解耦** | `Indexer` 统一抽象为本地、远程、并发三种实现，`RemoteIndexer` 抽出为独立模块，内部通过 **PushRouter**（请求平面）实现 RPC。 | 代码结构更清晰，Indexer 与 KV‑router 的耦合度下降，后续可以单独演进远程索引服务（如语言/协议迁移）。 |
| **配置模型** | `KvRouterConfig` 新增 `use_remote_indexer`、`serve_indexer` 两个布尔字段，原 `remote_indexer_component` 被废弃。 | 配置更直观（布尔开关），避免了手动填写组件名导致的拼写错误；同时为 **互斥校验**（两者不能同时开启）提供了明确报错。 |
| **服务注册** | `ServedIndexerService` 为一个 **进程内单例**（基于 `DashMap` + `RwLock`），在同一 namespace/组件下多实例 router 能共享同一个服务（query + record 端点），并通过 discovery 检查拓扑一致性。 | 防止出现多个写入端点导致状态不一致的情况；但需要保证在同一组件内部只会出现 *单实例*（或多个实例但共享同一实例 ID）时才能启动。 |

### 性能影响
- **查询路径**：原本在本地 radix tree 中 O(log N)（N 为缓存块数）完成 overlap 计算；使用 `--use-remote-indexer` 后，会多一次请求平面往返（序列化/反序列化 + 网络 RTT）。在同机房、NATS/GRPC 低延迟情况下，影响可接受，且**不再占用本地内存**（在资源受限的 pod 中有利）。
- **写入路径**（近似模式）：`--serve-indexer` + `--use-remote-indexer` 时，消费端会在每次 routing decision 完成后调用 `kv_indexer_record_routing_decision` 将本地计算的哈希写回远程索引器。写入成本同样是一次网络调用，且仅在近似模式下使用，写入频率受 TTL/Prune 策略控制，不会显著放大。
- **CPU/内存**：`serve_indexer` 会在 router 进程里多维护一个 `KvIndexer` 实例（或 `KvIndexerSharded`），其 CPU/内存占用与原本启动 `dynamo.indexer` 服务相近，但省去了额外进程的调度开销。整体资源使用略有上升（约 10‑20%），但削减了容器数量与网络转发层数。
- **吞吐量**：实验性指标 `router_remote_indexer_query_failures_total` 与 `router_remote_indexer_write_failures_total` 已加入监控，便于在高负载下观察因网络抖动导致的回退。若远程索引服务节点故障，router 会降级为 `KvRouterError::IndexerOffline`（旧行为）并继续调度，保持系统可用性。

### 安全考虑
- **网络安全**：远程索引器通过 **request plane**（默认使用 Dynamo 组件的安全通道）进行 RPC，继承了 `dynamo-runtime` 对身份验证、TLS、ACL 的校验。新增的 `kv_indexer_record_routing_decision` 端点同样受到同一套安全策略约束，未出现额外暴露。  
- **权限校验**：`ServedIndexerService.verify_service_topology` 在启动时检查 **query 与 record** 端点是否来自同一实例，防止恶意注册多个写入口进行状态篡改。  
- **错误传播**：`RemoteIndexer` 将查询/写入错误转化为 Prometheus 计数器，且在查询失败时返回空分数（`OverlapScores::new()`），避免异常抛出导致路由器崩溃。  
- **资源隔离**：`serve_indexer` 与 `use_remote_indexer` 互斥，防止同一进程既提供又消费本地索引导致循环调用或双写。配置校验在 `KvRouterConfig.validate` 中已经实现。

### 可维护性 & 可测试性
- **代码抽象**：`Indexer::Remote` 与 `Indexer::KvIndexer` 实现统一接口，调用方无需感知本地/远程差异，后续增加其他远程实现（比如 gRPC）时只需实现 `record_hashed_routing_decision` 与 `find_matches`。  
- **特性开关**：以前的 `kv-indexer-runtime` Feature 已被移除，简化了 Cargo.toml（仅保留 `kv-indexer`、`kv-indexer-metrics`），降低

### feat(sglang): add logprob passthrough in decode handler (#6837)
**SHA**: `5802db8` | 🔗 [查看提交](https://github.com/ai-dynamo/dynamo/commit/5802db87c9bf7a226ec132405b0de4c8b20ccbfc)

**🎯 变更类型**：功能增强  

**⚡ 重要程度**：🔴高  

**📋 变更摘要**  
1. 在 SGLang 的 `DecodeWorkerHandler` 中实现了 logprob（对数概率）透传功能，前端通过 Rust `OutputOptions` 提供 `logprobs` 与 `prompt_logprobs` 参数。  
2. 新增 `_build_logprob_kwargs` 将这些参数映射为 SGLang `async_generate` 所需的 `return_logprob`、`top_logprobs_num`、`logprob_start_len`。  
3. 新增 `_extract_logprobs` 在流式输出时从 SGLang 累积的 `meta_info` 中切分出本次块对应的 logprob 与 top‑logprob 数据，并组装到 Dynamo 返回结构中。  

**🎯 影响范围**  
- `components/src/dynamo/sglang/request_handlers/llm/decode_handler.py`（核心解码处理器）  
- `components/src/dynamo/sglang/CLAUDE.md`（文档）  
- Rust 前端 `lib/llm/src/protocols/common.rs` 中的 `OutputOptions`（间接影响）  
- 依赖 SGLang 解码的所有 Dynamo LLM 服务实例（包括 vLLM、TRT‑LLM 等兼容后端）  

**🔍 技术洞察**  

- **架构影响**  
  - 在已有的 “preprocess → Dynamo RPC → Python handler → SGLang engine → postprocess” 流程中，新增了 **logprob 参数的跨语言传递** 与 **流式日志概率切分** 两个关键节点。  
  - 通过 `_build_logprob_kwargs` 把 Rust 前端的 `output_options` 转化为 SGLang 的内部 kwarg，保持了 **前后端解耦**：前端只需关注 `OutputOptions`，后端无需感知 SGLang 细节。  
  - `_extract_logprobs` 解决了 **SGLang 累积 logprob 与 Dynamo 分块 token_ids 不一致** 的匹配问题，保证了流式输出的正确性。  

- **性能影响**  
  - **正向影响**：logprob 计算已在 SGLang 引擎内部完成，Dynamo 仅做切片与结构转换，额外开销极小（O(新增 token 数) 的列表切片与少量字典构造）。  
  - **潜在负担**：当 `top_logprobs_num` 被设为较大值（如 100+）时，SGLang 需要额外的 softmax 与排序工作，可能导致显存与计算时间上升，尤其在高并发流式场景下。  
  - 代码使用 `logging.warning` 进行输入校验，对正常路径几乎没有运行时成本。  

- **安全考虑**  
  - 新增的输入字段仍由前端受信任的 `OutputOptions` 产生，未引入外部文件或网络请求。  
  - 对 `logprobs` 与 `prompt_logprobs` 进行 **整数解析与非负校验**，防止恶意构造的负数或非整数导致异常行为。  
  - 仍需关注 **资源消耗攻击**：如果用户能够控制 `top_logprobs_num`，可能通过极大数值逼迫后端进行大规模排序，引发 OOM 或 CPU 飙升。建议在后端配置上设上限。  

**⚠️ 潜在风险**  

1. **累计切片错误**

2. **性能回退**

**SHA**: `c53ef73` | 🔗 [查看提交](https://github.com/ai-dynamo/dynamo/commit/c53ef736ccac4a2bdc2744d4487a64b393f701ef)

**🎯 变更类型**：Bug修复  
**⚡ 重要程度**：🔴高  
**📋 变更摘要**：  
在请求迁移（migration）机制中，针对使用 **guided‑decoding（结构化输出/JSON schema）** 的请求禁用了迁移。实现上通过在 `RetryManager::build` 中检测 `guided_decoding` 配置并将 `retries_left` 强制设为 0，从而让迁移直接失效，错误直接回 propagates 给客户端。同时在文档中新增了该限制的说明，并补充了单元测试验证行为。

**🎯 影响范围**：  
- `lib/llm/src/migration.rs`（请求迁移核心实现）  
- `docs/fault-tolerance/request-migration.md`（迁移特性文档）  
- 迁移相关的单元测试 `lib/llm/src/migration.rs::tests`

**🔍 技术洞察**  

- **架构影响**：  
  - **请求迁移路径**：在构造 `RetryManager` 时加入了对 `guided_decoding` 的判定，强制关闭迁移分支。该改动局部化在 `RetryManager::build`，不影响整体请求流水线的其它路径。  
  - **统一行为**：明确了所有后端（vLLM、SGLang、TRT‑LLM）在结构化输出场景下的行为一致性，避免了之前因 FSM 状态不同步导致的输出损坏。  
  - **可观测性**：通过 `tracing::warn!` 记录禁用迁移的警告，便于运维快速定位此类错误。

- **性能影响**：  
  - 仅在每次构建 `RetryManager` 时额外进行一次 `guided_decoding` 检查，成本可忽略不计。  
  - 迁移被禁用后，原本可能的二次请求（重新调度）不再执行，极端情况下对受影响请求的延迟会略有提升（因为直接返回错误），但整体系统吞吐量不会受负面影响。

- **安全考虑**：  
  - 防止了因 FSM 状态不匹配而产生的 **重复/嵌套 JSON**，降低了下游服务解析错误或潜在的注入风险。  
  - 迁移关闭后，错误直接向客户端暴露，仍然遵循 Dynamo 统一的错误类型（`Disconnected`），不会泄漏内部实现细节。

**⚠️ 潜在风险**  

1. **可用性下降**：结构化输出请求在 worker 失效时不再利用迁移进行容错，服务可用性会回退到 “fail‑fast”。在高可用场景下，这可能导致用户感受到更频繁的错误。  
2. **误判禁用**：如果 `guided_decoding` 标记在请求中被误设（例如普通文本请求误带 `guided_decoding`），同样会失去迁移能力。需要确保上层 API 对该字段的校验足够严格。  
3. **Metrics 偏差**：原有的迁移计数（`migration_new_request_count`、`migration_ongoing_request_count`）在此类请求上不再递增，仪表盘可能出现异常下降，需要在监控层面注明该限制。  
4. **后续实现耦合**：若后续实现 FSM 状态序列化，当前的 “zero‑retries” 逻辑必须同步更新，否则可能出现 “禁用迁移 + 仍然尝试迁移” 的不一致。

**💡 关注建议**  

- **监控与告警**：在监控仪表盘中新增 “guided‑decoding migration disabled” 的计数指标，以便评估该限制对业务的实际影响。  
- **文档同步**：确保所有 SDK、API 文档都明确说明结构化输出请求不支持迁移，避免用户在使用时产生误解。  
- **回退策略**：如果业务对结构化输出的可用性要求极高，可考虑在业务层实现 **重试**（而非迁移）或 **请求分片**，以降低单节点故障影响。  
- **未来实现路线**：对于计划支持迁移的 roadmap，提前在代码中预留 **FSM 序列化/恢复** 的抽象接口（如 `GuidedDecodingState::serialize/deserialize`），以降低后期改动成本。  
- **单元/集成测试**：当前已加入针对 guided‑decoding 的单元测试，建议在 CI 中加入 **跨后端**（vLLM、SGLang、TRT‑LLM）组合的集成测试，确保所有实现保持一致的禁用行为。  

---  

此改动通过在迁移入口处直接阻断不兼容的 guided‑decoding 场景，修复了已知的 JSON 输出损坏 bug，提升了系统的正确性与安全性，但也在特定请求类型上牺牲了迁移带来的容错能力。后续如果实现 FSM 状态迁移，可在此基础上平滑恢复迁移特性。

| **性能影响** | - 仅在对象转换为 `Unstructured` 后遍历一次顶层 map，额外的 `map` 拷贝开销极小（O(N) 与对象字段数成正比），对整体控制器吞吐量几乎无感。<br>- 测试覆盖率提升，有助于在 CI 中捕获潜在回归，间接提升系统可靠性。 |
| **安全考虑** | - **Secret 复制**：`CopySpec` 现在会把 **所有**非 envelope 字段复制到目标，包括 `data`（即 Secret 的明文值）。在同一控制平面内部这仍是合法操作，但需确认 RBAC 允许 operator 读取/写入对应 Secret，否则可能触发权限错误。<br>- **注解泄露**：`GetSpecHash` 仍使用哈希后写入 `nvidia.com/hash` 注解，未对数据本身做额外加密，保持原有行为；无新安全风险。 |

1. **覆盖不可变字段**：某些资源（如 Service 的 `spec.clusterIP`）在创建后不可修改。虽然 `CopySpec` 只在 `SyncResource` 中用于“desired 与现状对齐”，若目标对象已经存在且包含不可变字段，直接写入可能导致 `api` 返回错误。

4. **意外覆盖用户自定义字段**：`nonEnvelopeFields` 直接复制所有除 envelope 之外的字段。如果未来出现资源在 envelope 之外加入与 operator 无关的自定义字段（如第三方注解），也会被同步进去，可能导致冲突。

3. **升级迁移策略**：对已有集群，在升级前可通过一次 “dry‑run” 或 `--validate` 模式检查 hash 变化，必要时在升级脚本中先为所有 spec‑less 资源补上 `nvidia.com/hash` 注解，以避免因 hash 差异导致的大规模滚动更新。  
4. **扩展测试**：加入 Secret、Role、ClusterRole 等资源的 `GetSpecChangeResult` 测试，确保相同的行为在所有常见 envelope‑less 类型上得到验证。  
5. **文档更新**：在 operator 文档中说明 **Spec‑less 资源的同步行为**，尤其是 “所有非 envelope 字段均视为 spec”，帮助使用者理解为何修改 ConfigMap 的 `metadata` 不会触发滚动，而修改 `data` 则会。
