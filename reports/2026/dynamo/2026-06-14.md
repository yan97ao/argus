### feat(routing): unified structured Selected-worker INFO log across all router modes (#10554)
**SHA**: `7897369` | 🔗 [查看提交](https://github.com/ai-dynamo/dynamo/commit/789736997696e0a874e07a70b507ffaffd1069b3)

**🎯 变更类型**：功能增强（统一、结构化的 Selected‑worker INFO 级日志）  
**⚡ 重要程度**：🔴高  

**📋 变更摘要**  
- 将 KV‑router、Push‑router 等多种路由模式在选取 worker 时的日志统一为结构化的 `INFO` 级别，并使用 `tracing::info!` 通过键值对记录 `router_mode、worker_id、worker_type、dp_rank、logit、host_pinned_blocks、disk_blocks、candidate_count、load 等字段。  
- 删除或降级原先的 `trace`/`debug` 日志，改为统一的 `"Selected worker"` 信息。  
- 相应测试用例更新，以匹配新的结构化日志格式，并禁用 ANSI 彩色输出以保证正则匹配。

**🎯 影响范围**  
- `lib/kv-router/src/scheduling/selector.rs`（KV 选路）  
- `lib/runtime/src/pipeline/network/egress/push_router.rs`（Power‑of‑Two‑Choices、Round‑Robin、Random、Direct、DeviceAwareWeighted、LeastLoaded 路由）  
- 单元/集成测试 `tests/serve/test_trtllm.py`（日志正则匹配）  

**🔍 技术洞察**  

- **架构影响**  
  - 仅影响日志层面，业务逻辑保持不变；对外观测系统（Prometheus、Grafana、ELK 等）提供统一的结构化字段，便于聚合与告警。  
  - 通过 `tracing` 的结构化字段，后端可直接把 `router_mode`、`worker_id` 等作为标签，提升可观测性与排查效率。  

- **性能影响**  
  - 将原先的 `trace`（默认在高负载下关闭）提升至 `info`，在生产环境开启 INFO 级别时会产生额外的日志写入与序列化开销，尤其在高 QPS 场景下可能导致 I/O 增压。  
  - 结构化字段的序列化成本微乎其微，主要瓶颈在日志输出目的地（文件、STDOUT、云日志服务）。  
  - 代码路径内仅增加 `tracing::info!` 调用，几乎不影响路由选择的计算路径。  

- **安全考虑**  
  - 日志中暴露 `worker_id、dp_rank、logit、block counts` 等内部调度信息。虽然不包含用户数据或密钥，但在多租户或受监管环境中可能被视为敏感的拓扑信息。  
  - 建议通过环境变量或 `DYN_LOG` 控制日志级别，防止在生产环境误开启 INFO 级别导致信息泄露。  

**⚠️ 潜在风险**  

1. **日志噪声 & 存储压力**  
   - 在高并发推理服务下，选路日志频率与请求数成正比，开启 INFO 可能导致日志量激增，进而耗尽磁盘空间或使日志聚合系统负载飙升。  

2. **回归兼容性**  
   - 依赖旧 `trace`/`debug` 日志进行监控或内部工具的用户需要更新正则或筛选规则。  

3. **ANSI 颜色导致正则失效**  
   - 如未禁用 `DYN_SDK_DISABLE_ANSI_LOGGING`，ANSI 颜色码会插入字段之间，破坏现有日志解析。  

4. **误配置导致泄露**  
   - 若在包含敏感租户信息的环境中误将日志级别设为 INFO，可能把内部拓扑信息暴露给不该看到的用户。  

**💡 关注建议**  

- **日志级别治理**  
  - 在生产环境默认将 `router_mode` 相关日志保留在 `debug`/`trace`，仅在调试或故障排查时通过 `DYN_LOG=...=info` 临时提升。  
  - 为避免意外泄漏，可在 CI/CD 中加入检查，确保 `INFO` 级别日志不被默认打开。  

- **日志采集策略**  
  - 在日志收集层（Fluent Bit、Filebeat 等）对 `router_mode=kv|power-of-two-choices|...` 打标签，便于按路由模式统计选路延迟与负载分布。  
  - 对 `candidate_count`、`load` 字段建立监控阈值，及时发现异常负载倾斜。  

- **测试与 CI**  
  - 保持 `DYN_SDK_DISABLE_ANSI_LOGGING=1` 在所有日志匹配测试中生效，防止因终端彩色输出导致正则失效。  
  - 为防止日志量过大，可在 CI 中加入 “日志大小阈值” 检查，确保单次测试产生的日志不超过合理上限。  

- **安全审计**  
  - 审视是否有业务需求将这些选路字段写入外部审计日志；如有，需在审计策略中标记为非敏感或进行脱敏。  

- **回滚路径**  
  - 如出现异常日志爆炸，可快速回退到原 `trace`/`debug` 调用（提交前的代码已保留），只需改回 `tracing::trace!` 或降低 `DYN_LOG` 级别。  

通过上述统一的结构化日志，团队将获得更一致、更易查询的选路可观测性，但务必在生产环境做好日志级别管控，以免产生性能与安全副作用。

### fix(kv-router): unbreak main — pass strict_priority in selection service call sites (#10689)
**SHA**: `8ddba58` | 🔗 [查看提交](https://github.com/ai-dynamo/dynamo/commit/8ddba58342b7e1245fa1c548396e8740f8e4c70e)

**🎯 变更类型**：Bug修复 / 功能增强  
**⚡ 重要程度**：🔴高  
**📋 变更摘要**：  
- 向 `SelectRequest` 与 `SelectAndReserveRequest` 中新增 `strict_priority` 参数，并在 `SelectionCore` 调用路径上向下传递。  
- 在 VLLM 单元测试中为 `model_config` 补充 `get_diff_sampling_param` 回调，以适配新的路由选择逻辑。  
- 目标是恢复主分支的编译通过（“unbreak main”），并让路由器能够依据严格优先级进行调度。

**🎯 影响范围**：  
- `kv-router` 服务层（`lib/kv-router/src/services/selection/*`）  
- 所有调用 KV‑Router 选路的上层组件（如 VLLM、光纤模型加载器等）  
- 序列化/反序列化层（`serde`），因为结构体字段新增  

**🔍 技术洞察**  

- **架构影响**：  
  - `strict_priority` 作为新的调度维度加入，扩展了请求路由的决策树。  
  - 只在 `SelectionCore` 的内部 `select` 与 `select_and_reserve` 两条分支中使用，保持了向后兼容（默认 `0`），不破坏现有架构。  
  - 增加了结构体字段后，所有通过 JSON/MsgPack 等方式跨进程传递的 `Select*Request` 必须兼容新字段，序列化层已通过 `#[serde(default)]` 做了向后兼容处理。

- **性能影响**：  
  - 仅在函数签名中多传递一个 `u32` 整型参数，内存/CPU 开销可忽略不计。  
  - 若后续在调度算法中加入基于 `strict_priority` 的排序或过滤，可能会额外的比较开销，但仍在 O(1)‑O(log N) 范围，当前提交未引入实际业务逻辑，只是参数转发。

- **安全考虑**：  
  - 新增的字段本身不涉及敏感信息或权限检查。  
  - 需要确保外部客户端不能通过构造异常的 `strict_priority`（如极大数值）导致资源分配失衡，建议在业务层对该字段设定上限（如 `<= MAX_PRIORITY`)。

**⚠️ 潜在风险**  

1. **向后兼容性**：旧版客户端在序列化请求时若使用严格模式（`deny_unknown_fields`）可能因缺少 `strict_priority` 报错。  
2. **调度行为变更**：虽然默认值为 `0`，但若有代码在创建 `SelectRequest` 时显式设置 `strict_priority`，可能导致现有流量调度策略被意外覆盖。  
3. **测试覆盖不足**：本次仅在 VLLM 单元测试中加入 `get_diff_sampling_param`，其他模块（如 `text-generation`, `embedding`）未验证新字段的路径。  
4. **序列化/反序列化冲突**：如果之前有自定义 `Deserialize` 实现，需确认已同步处理新字段的默认值。

**💡 关注建议**  

- **兼容性验证**：在 CI 中加入跨语言（Python、C++) 客户端的序列化兼容性测试，确保在 `strict_priority` 缺失时仍能成功解析。  
- **字段校验**：在入口层（如 `SelectionService::handle_request`）加入对 `strict_priority` 合法性的检查（范围、与 `priority_jump` 的冲突检测），避免异常高优先级占用所有资源。  
- **文档更新**：完善 API 文档，说明 `strict_priority` 的语义、默认值以及使用场景，防止误用。  
- **回归测试**：对原有调度路径（不使用 `strict_priority`）执行完整回归测试，确认业务行为保持不变。  
- **监控告警**：在生产环境添加监控指标 `kv_router.strict_priority_used_total`，并在异常激增时触发告警，以快速定位潜在的优先级滥用。

### fix(routing): log request rejection at warn, not error (#10561)
**SHA**: `dd667fb` | 🔗 [查看提交](https://github.com/ai-dynamo/dynamo/commit/dd667fbbbb22f22255fe9a25d97a80dafc91deca)

**🎯 变更类型**：Bug 修复  
**⚡ 重要程度**：🟡 中  
**📋 变更摘要**：将预填充路由因容量导致的请求被拒的日志级别从 `error` 降为 `warn`，防止误报错误率；并在远程预填充失败的错误分支中，通过 `match_error_chain` 区分 `ResourceExhausted`（容量）与其他错误，分别使用 `warn` 与 `error` 记录。  

**🎯 影响范围**：  
- `lib/llm/src/kv_router/prefill_router/mod.rs`（预填充路由）  
- 依赖 `dynamo_runtime::error` 的监控/告警系统  

**💡 关注建议**：  
1. **监控阈值**：确认告警规则已把 `warn` 级别的 “request rejected: prefill router backpressure” 纳入容量监控，避免遗漏关键容量告警。  
2. **日志一致性**：检查其他模块（如 KVRouter、decoder）是否仍使用 `error` 记录相同的容量拒绝，保持日志级别统一。  
3. **回退兼容**：`match_error_chain` 引入的错误链匹配逻辑在老版本运行时是否可用；若存在旧二进制，确保依赖库版本兼容。  
4. **测试覆盖**：新增单元/集成测试，验证在 `ResourceExhausted` 场景下返回 `warn`，而其他错误仍为 `error`，防止回归。  
5. **文档更新**：在运维手册中注明该日志已改为 `warn`，并说明如何在仪表盘中筛选对应指标。  

总体而言，此改动对业务逻辑无影响，仅优化了日志噪声和监控可读性。建议在下一个发布周期验证监控阈值后推广。
