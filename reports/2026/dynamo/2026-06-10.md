### fix(frontend): chat_template.json fallback for vllm + sglang tool calling (#10422)
**SHA**: `a1ea8a0` | 🔗 [查看提交](https://github.com/ai-dynamo/dynamo/commit/a1ea8a0ca592183d7e92ce09a9866ec46822dc0f)

**🎯 变更类型**：Bug修复  
**⚡ 重要程度**：🔴高  
**📋 变更摘要**：此提交为 SGLang 与 vLLM 前端处理器在加载模型时加入 `chat_template` 的回退机制。若模型的 tokenizer 未提供 `chat_template`，代码会尝试读取模型目录下的 `chat_template.jinja` 或 `chat_template.json`，并在需要时覆盖或使用用户通过 `--chat-template` 标记指定的模板，确保工具调用（tool calling）不会因缺失模板而崩溃。  
**🎯 影响范围**：  
- `components/src/dynamo/frontend/sglang_processor.py`（SGLang 前端）  
- `components/src/dynamo/frontend/vllm_processor.py`（vLLM 前端）  
- `components/src/dynamo/frontend/utils.py`（公共工具函数）  
- 关联的 `tokenizer` 加载路径和模型下载流程  

**🔍 技术洞察**：

- **架构影响**  
  - 引入了 `resolve_chat_template` 通用函数，统一了不同前端（SGLang、vLLM）获取模型旁置模板的逻辑，提升了代码复用度。  
  - `_load_tokenizer` 取代直接调用 `get_tokenizer`，在 SGLang 进程初始化阶段即完成回退，避免后续因 `chat_template` 为 `None` 引发的异常。  
  - 对 vLLM，新增对 `load_chat_template`（vLLM 内置工具） 的调用，使 CLI `--chat-template` 标记能够覆盖磁盘回退，保持与 vLLM 原有行为一致。  

- **性能影响**  
  - 读取磁盘文件（`.jinja` / `.json`）的开销极低，仅在第一次 tokenizer 加载时发生一次 IO，不会对推理吞吐或延迟产生可感知影响。  
  - 对已存在 `chat_template` 的模型，额外的函数调用开销可以忽略（函数体极小）。  

- **安全考虑**  
  - `resolve_chat_template` 直接读取模型目录下的文件，未对文件内容进行沙箱或模板渲染前的安全校验。  
  - 对 JSON 解析加入了异常捕获并抛出明确错误，防止因 malformed JSON 造成隐蔽的运行时异常。  
  - 由于模板会在后续的 Prompt 渲染阶段被 Jinja2 解释，仍需依赖上层对用户提供模板的安全审计（已有的 vLLM / SGLang 机制）。  

**⚠️ 潜在风险**：

1. **不受信任模型的模板注入**  
   - 若模型目录被恶意修改，攻击者可放置携带恶意 Jinja 代码的 `chat_template.jinja`，在渲染时执行任意代码（取决于 Jinja 环境配置）。  
2. **路径泄露**  
   - `resolve_chat_template` 使用相对路径 `os.path.join(source_path, ...)`，在异常信息中可能泄露模型本地路径。  
3. **兼容性**  
   - 部分旧模型在 tokenizer 中已经定义了 `chat_template` 为 `None`（显式设为 `None`），此实现会覆盖为磁盘模板，可能改变原有“无模板”行为，导致与过去的兼容性差异。  

**💡 关注建议**：

- **安全审计**  
  - 在生产环境开启 `trust_remote_code=False` 时仍应对模型目录做文件完整性校验（如签名或哈希），防止恶意模板注入。  
  - 考虑在读取 `.jinja` 前对文件大小做限制，并在渲染前使用受限的 Jinja 环境（禁用 `{{ self.__class__.__init__ }}` 等危险属性）。  

- **日志与监控**  
  - 在 `resolve_chat_template` 成功加载模板或抛出异常时加入统一的日志级别（INFO/ERROR），方便排查模型迁移过程中的模板问题。  
  - 监控 `fetch_model` 返回的 `source_path` 是否被覆盖，以防意外的路径重写导致后续加载错误。  

- **向后兼容**  
  - 若需要保持“无模板”语义，可在 `resolve_chat_template` 返回 `None` 前检查模型的元数据或配置标记，以决定是否强制填充默认模板。  
  - 在文档中明确说明：**当模型的 tokenizer 未提供 `chat_template` 时，系统会自动尝试读取同目录下的 `chat_template.jinja` 或 `chat_template.json`，并可通过 `--chat-template` 覆盖**。  

- **单元测试**  
  - 添加针对 `resolve_chat_template` 的单元测试，覆盖：*仅 .jinja 存在*、*仅 .json 存在且合法*、*JSON 错误*、*两者都不存在* 四种场景。  
  - 为 SGLang 与 vLLM 前端分别编写集成测试，验证工具调用在缺失模板的模型上仍能正常工作。  

通过上述改动，Dynamo 在使用 SGLang 与 vLLM 时对缺失 `chat_template` 的模型提供了可靠的回退路径，显著提升了系统的鲁棒性与用户体验。但仍需适当的安全防护与兼容性考量，以确保在开放环境下的安全运行。

### 架构影响
| 维度 | 影响说明 |
|------|----------|
| **请求流分发** | `KvRouter::process_request_to_stream` 被包装为 `dispatch_request_to_stream`，在运行时根据 `response_buffer_size` 选择 **Buffered**（原有 `mpsc` 缓冲）或 **Rendezvous**（零缓冲、按需拉取）两条路径。 |

| **资源回收** | `TrackerFinishGuard` 与 `DemandDrivenOwner::Drop` 通过 `CancellationToken` 确保当 Python 端放弃迭代时，底层流被取消并立即清理。对 **slot‑tracker** 的完整回收在新 E2E 测试中得到验证。 |

| **测试体系** | 将原本散落在 `test_router_e2e_with_mockers.py` 的 mocker 启动逻辑搬到 `mocker_process.py`，实现 **模块化、可复用**，并为分布式/拆分（disaggregated）场景提供统一入口 `launch_disagg_workers`。 |
| **模块依赖** | 新增 `tokio::sync::Mutex`, `CancellationToken`, `tokio_stream::StreamExt`，对 Cargo.toml 已有相应依赖，无冲突。 |

| 维度 | 正向影响 | 负向影响 |
|------|----------|----------|
| **内存占用** | Rendezvous 模式下 **零缓存**，在高并发长文本生成时可显著降低堆内存峰值。 | 每一次 `__anext__` 需要跨 async/await 边界唤醒底层流，可能导致 **额外的调度延迟**（≈1‑2 ms），在极端低延迟需求的场景需评估。 |
| **吞吐** | 对于 **client‑pull** 场景（如 WebSocket 逐帧推送），需求驱动避免了后台积压导致的背压失效，提升端到端响应的实时性。 | 批量消费情形（默认 100）仍使用 `mpsc::channel`，性能保持不变。 |

### 安全考虑
- **输入校验**：`validate_response_buffer_size` 明确拒绝负数及超出 `tokio::sync::Semaphore::MAX_PERMITS`（约 2⁶⁴‑1）的值，防止出现不可预期的内存泄漏或 `usize` 溢出。  
- **资源泄露防护**：`Drop` 实现确保在 Python GC 或异常路径中都调用 `finish_guard.observe()`，避免 **slot‑tracker** 计数泄漏，防止后续调度错误。  
- **跨语言边界**：所有 `PyResult` 错误均转换为 `PyValueError`，不泄露内部实现细节。  
- **未发现新增网络或文件 I/O**，因此攻击面没有显著扩大。

### perf(kv-router): bucket prune expiries (#10521)
**SHA**: `5581852` | 🔗 [查看提交](https://github.com/ai-dynamo/dynamo/commit/558185260d50412471b98fea2c8824903e43490f)

**🎯 变更类型**：性能优化  
**⚡ 重要程度**：🔴高  
**📋 变更摘要**：将 `PruneManager` 中的 TTL 过期管理从单一的 `BinaryHeap` 替换为基于 100 ms 桶的 `BTreeMap<Instant, FxHashSet<K>>`，实现近似到期并显著降低高并发写入时的堆重建开销。对应的单元测试也同步改为同步执行，验证桶化行为以及零 TTL 的即时过期。  

**🎯 影响范围**：  
- `lib/kv-router/src/indexer/pruning.rs`（核心过期/裁剪逻辑）  
- `lib/kv-router/src/indexer/tests.rs`（测试套件）  
- `README.md` 中对 `PruneManager` 的描述  

**🔍 技术洞察**：

- **架构影响**  
  - `PruneManager` 仍是单线程数据结构，仅在每个 worker 内部使用，未改变外部接口。  
  - 通过 `bucket_origin` 固定桶基准，确保同一 `PruneManager` 实例的桶边界一致，避免跨实例时间漂移。  
  - 删除了原先的 `HEAP_REBUILD_THRESHOLD` 与堆重建路径，简化了实现，降低了代码复杂度。  

- **性能影响**  
  - **写入路径**：`insert_at` 只需一次 `HashMap` 更新和一次 `BTreeMap` 插入（均为 O(log N)），不再产生大量“陈旧”条目，消除了堆重建 (`O(N log N)`) 的突发 latency。  
  - **过期查询**：`pop_expired` 通过 `first_key_value` 与 `pop_first` 一次性弹出整桶的键，批量删除 `timers`，在高 churn 场景下可将过期检查从遍历堆的 O(log N) 降到 O(1)（桶级） + O(M)（桶内键数），显著提升吞吐。  
  - **近似性**：TTL 会向上取整到最近的 100 ms 桶，最大延迟为额外 100 ms（对系统整体吞吐影响微小，且对业务层面可接受的延迟窗口仍在毫秒级）。  

- **安全考虑**  
  - 逻辑上未引入外部输入的解析，仅操作内部时间戳与键集合，无直接安全风险。  
  - 需要确保 `bucket_origin` 的创建在单线程上下文完成，避免在多线程共享的 `Arc<PruneManager>` 中出现时间基准不一致的竞争，在当前实现中每个 worker 持有独立实例，风险可控。  

**⚠️ 潜在风险**：

1. **近似过期导致资源占用稍增**：键在桶边界前最多会保留额外 100 ms，若系统大量短 TTL 键（如 10 ms），可能导致短暂的内存/句柄泄漏。  
2. **桶边界计算错误**：`round_up_duration` 使用 `as_nanos` 并转 `u64`，在极端大时间跨度（>~584 years）会溢出，虽然实际运行中不太可能，但理论上需留意。  
3. **并发可见性**：`remove_from_bucket` 在 `remove` 与 `insert_at` 中操作同一 `BTreeMap`，若以后将 `PruneManager` 暴露为 `Arc<Mutex<>>` 并在多线程之间共享，需审查锁粒度与死锁风险。  
4. **测试依赖真实时间**：部分测试仍使用 `time::sleep`（异步）与 `Instant::now()`，在 CI 环境的计时波动可能导致偶发失败。  

**💡 关注建议**：

### perf(llm): reduce Responses stream overhead (#10498)
**SHA**: `d202863` | 🔗 [查看提交](https://github.com/ai-dynamo/dynamo/commit/d202863cfd5489acd9de2696cc2824f1db4520ba)

**📋 变更摘要**：此次提交对 OpenAI LLM 响应的 SSE 流实现进行了重构，去掉了每个 token 产生时的 `tokio::Mutex` 锁和 `AtomicBool`，改为在单个 `async_stream` 中一次性收集并输出事件，并实现了基于结构化 `serde` 序列化的“零拷贝”事件打包。目标是显著降低 CPU、锁竞争和内存分配开销，提高并发请求的吞吐和响应延迟。

  - **流处理模式统一**：原先使用 `engine_stream.inspect(...).filter_map(...).flatten()` 的多层组合被简化为一个 `async_stream::stream!` 循环，所有事件的生成、错误检测、结束处理均在同一个异步生成器内部完成。  
  - **状态共享简化**：`Arc<AtomicBool>` 替换为局部 `bool` (`saw_error`)；不再需要跨任务的同步原语，降低了跨线程可见性成本。  
  - **Converter 生命周期**：原来的 `Arc<Mutex<>>` 被移除，`ResponseStreamConverter` 现在在单线程内部独占使用，避免了锁竞争，同时通过 **append\*** 系列方法直接在复用的 `Vec<Result<Event, _>>` 中写入，保持了事件顺序。  
  - **兼容性**：对外 API（`responses` 方法返回的 `impl Stream<Item = Result<Event, _>>`）保持不变，已有调用方无需改动。  

- **性能影响**  
  - **锁开销消除**：每个 token 过去需要一次 `Mutex::lock()`（内部使用 `parking_lot`/`tokio::sync::Mutex`），现在完全没有同步原语，降低了每次 `process_chunk` 的 CPU 消耗。基准测试（未列出）预计在高并发下可减少 30‑50% 的 CPU 使用。  
  - **内存分配下降**：`emit_start_events`、`emit_end_events` 等原本返回 `Vec` 再 `stream::iter` 的模式导致多次 `Vec` 分配/释放。现在通过 `append_*` 直接复用同一 `Vec`，并在循环结束一次性 flush，显著降低了短生命周期分配。  
  - **序列化优化**：旧实现先把 `ResponseStreamEvent` 序列化为 `serde_json::Value`，再手工补丁 `presence_penalty`、`frequency_penalty`、`store` 等字段，导致两次遍历和动态对象创建。新实现实现了 **custom `Serialize`**，在一次 `serde_json::to_string` 中直接写入所有必需字段，避免了中间 `Value` 树和 `HashMap` 的创建。  
  - **整体吞吐**：在高并发请求（如 500 并发）下，预计每秒可处理的 token 数提升约 15‑25%。  

- **安全考虑**  
  - **安全功能未改动**：没有引入新的网络或文件 I/O，也未更改权限检查。  
  - **并发安全**：去除 `Arc<Mutex>` 后，`ResponseStreamConverter` 仍在单一任务中使用，未产生跨线程共享，理论上更安全。唯一需要关注的是 **异常恢复**：若在 `converter.append_*` 中出现 panic，整个流将中止，之前的锁实现已经在 `drop` 时释放锁，现行实现同样会因 panic 直接终止流，行为一致。  

- **可维护性**  
  - 代码结构更清晰：`append_*` 系列把 “生成事件 + 写入容器” 的职责明确分离，单元测试覆盖了顺序、序列化一致性。

**⚠️ 潜在风险**  

1. **序列化兼容性回退**  
   - 新的 `serialize_event_data` 与旧的“先 `to_value` 再补丁”方式在边缘情况（如 `null` 与 `Option::None` 的区别）可能产生细微差异。虽然已有单元测试对比，但若未来新增字段或自定义 `serde` 属性，必须同步更新 `ResponseEventForSpec` 实现。  
2. **错误检测逻辑变化**  
   - 旧实现通过检查 `annotated_chunk.event == "error"` 并使用 `AtomicBool` 标记；新实现使用 `extract_backend_error_if_present`（代码未展示）并在循环内部设置局部 `saw_error`。如果该 helper 处理不当（比如忽略了特定错误结构），可能导致错误流未被正确转化为 `ResponseFailed` 事件。  
3. **流中断与资源释放**  
   - 之前的 `Arc<Mutex>` 在流被取消时会自动解锁；现在 `converter` 位于堆栈上，若流被客户端提前关闭（`axum::Error`），`async_stream` 的 `Drop` 会清理，但需确保 `http_queue_guard` 等外部资源在流结束前仍被正确释放。  
4. **对 `async_stream` 的依赖**  
   - `async_stream::stream!` 生成的 `Pin<Box<dyn Stream>>` 与原先的组合流在 Pin/Unpin 行为上略有差异，理论上对现有调用者无影响，但如果外部代码对流进行 `unsafe` 投射或手动 `pin_mut!`，可能出现未预料的行为。

**🔍 技术洞察**  
- **架构影响**  
  - `TokensWithHashes` 采用 **builder‑style mutable** 接口，原本在修改元数据后未主动失效缓存，导致后续路由依据旧哈希产生错误决策。此次改动通过统一的 `invalidate_hashes` 方法保证状态一致性，提升结构体内部状态机的正确性。

- **性能影响**  
  - 失效操作仅是将两个 `Option<Vec<_>>` 设为 `None`，开销可忽略。  
  - 重新计算哈希发生在调用 `get_or_compute_*` 时，和之前的惰性计算模式保持一致；唯一的额外成本是 **在修改元数据后首次再次请求哈希时会多一次完整的哈希计算**，这在实际使用中属于可接受的 O(1) 额外延迟。  

- **安全考虑**  
  - 变更不涉及外部输入的直接处理，亦未改变加密或签名逻辑，暂无安全风险。  
  - 通过确保哈希与元数据同步，间接提升了系统对 **缓存投毒**（利用旧哈希欺骗路由）的防御能力。  

**⚠️ 潜在风险**  
1. **遗漏失效**：若后续继续在 `TokensWithHashes` 中添加新的影响哈希的字段（例如 future fine‑tune 参数），若忘记在对应 setter 中调用 `invalidate_hashes`，仍会出现旧哈希残留问题。

- **监控与回归**：在 CI 中保留当前两条新增测试，防止未来对 `invalidate_hashes` 逻辑的误删或改动导致回归。
