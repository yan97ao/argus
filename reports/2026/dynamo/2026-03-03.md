### feat: FlexKV integration in Dynamo (#5858)
**SHA**: `4d7c984` | 🔗 [查看提交](https://github.com/ai-dynamo/dynamo/commit/4d7c98458ba578e88b481a827fefef75d4a86565)

**🎯 变更类型**：功能增强  
**⚡ 重要程度**：🔴高  

**📋 变更摘要**  
本次提交为 Dynamo 项目新增 FlexKV（版本 1.2.1）后端的集成支持。通过在 `vllm` 参数解析层加入 `"flexkv"` 关键字映射到 `FlexKVConnectorV1`，并补充对应的单元测试、文档以及示例脚本，使用户能够通过环境变量 `DYNAMO_USE_FLEXKV=1` 启用 FlexKV 的 KV 缓存离线与分布式服务能力。  

**🎯 影响范围**  
- `components/src/dynamo/vllm/args.rs`（Python 对应实现）  
- `components/src/dynamo/vllm/tests/test_vllm_unit.rs`  
- 文档目录 `docs/backends/vllm/`、`docs/integrations/`、`docs/design-docs/`  
- 示例脚本 `examples/backends/vllm/launch/*.sh`  
- 项目许可证归属文件 `ATTRIBUTIONS-Python.md`  

**🔍 技术洞察**  
- **架构影响**：  
  - 在 KV‑BM（Key‑Value Buffer Manager）层新增 `FlexKVConnectorV1`，实现了 “KV cache offloading” 的统一接口，保持了原有 connector 抽象不变。  
  - 通过环境变量 `DYNAMO_USE_FLEXKV` 与 `kv-transfer-config` 参数实现运行时可插拔，使 Dynamo 能够在同一集群中混用 FlexKV、Nixl、LMCache 等多种后端，提升了系统的可组合性和部署灵活性。  
- **性能影响**：  
  - FlexKV 主打跨机器 KV 缓存共享与磁盘/NVMe 离线，理论上可显著降低 GPU 显存压力并缩短 KV 重算的延迟，特别适用于大模型长会话场景。  
  - 但引入网络通信（ZMQ）和进程间 IPC（共享内存）会带来额外的序列化与传输开销，实际收益取决于业务的 KV 重用率、网络带宽与 FlexKV 服务端配置。  
- **安全考虑**：  
  - FlexKV 使用 Apache‑2.0 许可证，已在 `ATTRIBUTIONS-Python.md` 中明确归属，符合项目的许可证合规性。  
  - 对外提供的 FlexKV 进程可能暴露 IPC 或 ZMQ 端口，需确保在生产环境中对这些端口进行访问控制，以防止未授权的 KV 数据读取或篡改。  

**⚠️ 潜在风险**  
1. **依赖兼容性**：FlexKV 目前锁定在 1.2.1 版本，若上游升级可能导致 API 不匹配，需在发布说明中明确版本要求。  
2. **运行时缺失**：未安装 FlexKV 二进制或库时，Dynamo 启动仍会尝试加载 `FlexKVConnectorV1`，可能导致启动失败或误报。  
3. **测试覆盖不足**：仅在单元测试中验证 `FlexKVConnectorV1` 不会被误判为 Nixl，未覆盖实际 KV 迁移、故障恢复等关键路径。  
4. **安全面**：FlexKV 进程默认不做身份验证，若在开放网络环境下使用，可能成为攻击面。  

**💡 关注建议**  
- **发布前**：在 CI 中加入可选的 FlexKV 集成测试（使用 mock 或本地轻量版 FlexKV），确保在缺失依赖时系统能够优雅降级。  
- **文档**：明确 FlexKV 的版本要求、安装步骤以及安全最佳实践（如 IPC 路径权限、ZMQ 认证方式）。  
- **监控**：建议在 Dynamo 的监控面板中加入 FlexKV 相关指标（缓存命中率、网络延迟、后端错误率），帮助运维评估实际收益。  
- **回退机制**：在 `args` 解析层加入容错逻辑：当 `DYNAMO_USE_FLEXKV=1` 且 FlexKV 启动失败时，自动回退到本地 Nixl 或 LMCache，避免服务不可用。  
- **许可证合规**：继续在 `ATTRIBUTIONS-Python.md` 中维护第三方许可证信息，防止未来的合规审计风险。

### ⚠️ 潜在风险
1. **特性泄漏**  
   - `pub use` 导致的公开 API 可能在未开启 `testing` 特性时仍被误引用，引发编译错误。需确保 `Cargo.toml` 中的 `[features]` 正确标记 `testing` 为 *非默认*（已是）并在 `lib.rs` 使用对应 `cfg`。  
2. **依赖升级副作用**  
   - `aws-lc-sys` 从 0.37.x 升级至 0.38.0，引入了新平台支持及可能的 ABI 变化（虽然属于 `patch`），仍建议在 CI 中跑一次全平台交叉编译验证。  
3. **名称冲突**  
   - 新增的 `create_token_block`、`create_sequential_block` 等函数与 downstream crate 中已有同名函数（若有）可能产生冲突，建议在文档中明确使用路径（`kvbm_logical::testing::create_token_block`）。

### fix: auto-scale request count in benchmarks (#6777)
**SHA**: `8bd5966` | 🔗 [查看提交](https://github.com/ai-dynamo/dynamo/commit/8bd5966b82b360d38f6fde4ac03bc02082c0cbf3)

**🎯 变更类型**：Bug修复 / 性能优化  
**⚡ 重要程度**：🔴高  
**📋 变更摘要**：在 `benchmarks/utils/aiperf.py` 中引入 `REQUEST_COUNT` 环境变量及自动计算逻辑，默认根据并发度自动生成请求数 `max(concurrency * 3, 10)`，并将该参数传递给 `aiperf`。文档同步更新，说明配置方式及注意事项。此改动提升基准测试的稳定性，避免因请求数过少导致测量噪声。

**🎯 影响范围**：  
- `benchmarks/utils/aiperf.py`（基准测试执行脚本）  
- `benchmarks/utils/benchmark.py`（调用入口）  
- `docs/benchmarks/benchmarking.md`（使用文档）  
- 依赖 `aiperf` 的 CI / 性能回归测试流水线

**🔍 技术洞察**  
- **架构影响**：仅限 Benchmarks 子系统，与核心库（`dynamo`）解耦；新增的环境变量解析和默认策略不会影响运行时服务架构或 API。  
- **性能影响**：  
  - 正面：确保每个并发槽至少执行 `REQUEST_COUNT_SCALE_FACTOR` 次请求，降低统计方差，提高测量可靠性。  
  - 负面：在高并发场景下自动放大请求数会显著延长基准耗时，可能占用更多计算/网络资源。  
- **安全考虑**：对 `REQUEST_COUNT` 进行非负整数校验，避免注入异常值。未涉及网络、文件系统或权限的安全风险。  

**⚠️ 潜在风险**  
1. **兼容性风险**：若部署环境的 `aiperf` 版本不支持 `--request-count` 参数，脚本将报错并中止基准。  
2. **资源耗尽**：`max(concurrency * 3, 10)` 在极高并发（如 10k）时会生成上万请求，可能导致机器 OOM、CPU/网络饱和或超时。  
3. **误用风险**：用户自行设置 `REQUEST_COUNT` 小于并发度，只给出警告而继续运行，导致实际并发被限制，基准结果失真。  
4. **CI 失稳**：CI 中未显式设定 `REQUEST_COUNT`，自动放大的请求数可能导致 CI 超时或资源配额超标。  

**💡 关注建议**  
- **兼容性检查**：在脚本启动前检测 `aiperf --help` 是否包含 `--request-count`，若缺失则回退至原有调用方式或提示升级。  
- **可配置放大因子**：将 `REQUEST_COUNT_SCALE_FACTOR` 也暴露为环境变量，以便在资源受限的环境中调低放大比例。  
- **资源上限保护**：在计算 `request_count` 之前加入上限（例如 `MAX_REQUEST_COUNT=5000`），防止极端并发导致资源耗尽。  
- **CI 推荐做法**：在 CI 配置文件中显式设置 `REQUEST_COUNT=100`（或适当值），并在 `benchmarks` 目录添加超时/资源限制的 wrapper。  
- **测试覆盖**：新增单元测试验证 `get_request_count` 与 `run_aiperf` 在不同 env 配置下的行为，包括负数、非整数、空值等异常情况。

| **架构影响** | - 纯外部工具链，**不改动 Dynamo 主代码**，对现有服务二进制完全透明。<br>- 通过 **PID‑attach** 或 **系统级** tracepoints，提供横向对比的观测层，可在不重启或重新编译的情况下快速加入到现有 benchmark pipeline。<br>- 脚本的 batch 模式在 `run.sh` 中被设计成后台并行执行，输出统一到 `OUTPUT_DIR`，便于后续 CI 报表统一收集。 |
| **性能影响** | - BPF 本身是 **零拷贝、在内核执行** 的低开销采集方式；相较于 `perf`/`nsys`，对前端业务的干扰更小（尤其在 batch‑mode 下仅记录聚合统计）。<br>- 仍然会占用一定的内核资源（map、tracepoint 累计），在 **高并发 (64 并发, 4k 请求)** 场景下可能导致 minor CPU/内存开销。<br>- 脚本默认过滤 `tokio-runtime-w` 进程，可通过 `--pid` 限定，进一步降低噪声。 |
| **安全考虑** | - **需要特权**：`perf_event_paranoid=-1`、`kptr_restrict=0`，并授予 `CAP_BPF、CAP_PERFMON、CAP_NET_ADMIN、CAP_SYS_PTRACE` 给 `bpftrace` 可实现 **非 sudo** 运行。若误授予给不受信二进制，攻击者可利用 BPF 实现内核信息泄露或 DoS。<br>- 脚本通过 `setup.sh` 自动修改 `/proc/sys/...`，这些改动是 **临时的，重启即恢复**，但在生产机器上仍需运维审批。<br>- 所有脚本均以 **只读** 方式读取内核数据，未执行写入或修改操作，风险主要在权限提升。 |
| **可维护性** | - 使用 Bash + bpftrace，代码量大但结构清晰；每个探针文件保持单一职责，易于增删。<br>- `run.sh` 采用 `set -euo pipefail`、参数校验、SIGINT 转发等最佳实践，降低脚本运行时的意外中断风险。<br>- 文档 (`README.md`) 完整，覆盖 **install → check → run** 三个阶段，可降低新手使用门槛。 |

**⚠️ 潜在风险**  
1. **权限泄露**：若 `grant_capabilities` 步骤在生产机器上执行，`bpftrace` 将拥有 `CAP_BPF` 等强大能力，任何拥有该二进制的普通用户都能执行任意 BPF 程序，潜在攻击面扩大。  
2. **系统稳定性**：不当的 `perf_event_paranoid`、`kptr_restrict` 调整会暴露内核符号，增加信息泄露风险；大规模并行运行多个 BPF 脚本（尤其系统级 `tcplife`、`tcpretrans`）会导致内核 BPF map 资源耗尽，引发 `bpftrace: failed to create map` 错误。  
3. **兼容性**：仅在 **Linux ≥ 4.18**、且内核开启 `CONFIG_BPF`、`CONFIG_DEBUG_INFO_BTF` 等选项时才能正常运行；老旧发行版或定制内核可能缺失必要的 tracepoint/kprobe。  
4. **CI 环境干扰**：若 CI 集群共用节点，BPF 脚本的内核调参会影响同节点的其他任务，需在隔离的容器/节点上执行。  

**💡 关注建议**  
- **权限管理**：生产环境**强烈建议**仅在专用调试机器上执行 `setup.sh --caps`，或在完成采集后手动撤销 `setcap`（`sudo setcap -r $(which bpftrace)`）。  
- **资源限制**：在 `run.sh` 的 batch 模式加入 `ulimit -n 65536` 或 `bpftrace -m <max_map_size>`，防止 map 爆炸。  
- **监控脚本本身**：使用 `timeout --signal=INT` 已在实现中加入，建议在 CI 中显式设定超时（如 `--duration 60`），防止长时间卡住。

- **后续集成**：考虑在 `benchmarks/frontend` 的 `Cargo.toml` 中新增一个 feature `bpf-tracing`，让用户通过 `cargo bench --features bpf-tracing` 自动拉取并执行这些脚本，保持一致的构建/测试流程。
