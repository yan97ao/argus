### feat: add Planner schema to DGDR and Profiler input (#6463)
**SHA**: `659122e` | 🔗 [查看提交](https://github.com/ai-dynamo/dynamo/commit/659122ed070786d5bdf39d5434669526a5bf2b12)

**🎯 变更类型**：功能增强 / 架构变更  
**⚡ 重要程度**：🔴 高  

**📋 变更摘要**  
1. 将原先在 Go 代码中重复实现的 `PlannerSpec`/`PlannerPreDeploymentSweepMode` 删除，改为在 Python 侧统一使用 `dynamo.planner.utils.planner_config.PlannerConfig` 作为唯一真相。  
2. 在 DGDR（Dynamo Graph Deployment Request）CRD 中把 `planner` 字段改为 `runtime.RawExtension`（opaque JSON），并在转换逻辑里实现 **透明的 JSON‑blob 读取/写入**。  
3. 更新了 Helm/CRD、文档、pydantic 代码生成与验证脚本，使其能够 **直接引用** `PlannerConfig`，避免代码生成重复定义。  
4. CI 中的 operator 工作流额外安装 `pyyaml`，Makefile 与测试脚本相应改名并加入 “operator‑only‑build‑context” 的容错逻辑。  

**🎯 影响范围**  
- `components/src/dynamo/planner/utils/planner_config.py`（单一配置定义）  
- `deploy/operator/api/v1beta1`、`v1alpha1` 相关的 Go 类型、转换函数、deep‑copy、CRD yaml、Helm chart  
- `deploy/operator/api/scripts/generate_pydantic_from_go.py`、`validate_pydantic_models.py`（代码生成 & 测试）  
- 文档 `docs/pages/kubernetes/api-reference.md`  
- GitHub Actions 工作流（operator CI）  

---  

### 🔍 技术洞察  

| 维度 | 影响说明 |
|------|----------|
| **架构影响** | • 将 Planner 的配置 **抽离为统一的 Pydantic 模型**，消除了 Go 与 Python 两套重复定义的风险。<br>• 在 CRD 中使用 `runtime.RawExtension` 让 Go 侧只负责 **透传**，真正的 Schema 校验交给 Planner 服务启动时完成，提升了跨语言一致性。<br>• `FeaturesSpec.Planner` 从结构化对象转为 **opaque JSON**，需要在 Operator 转换代码中正确实现 `applyPlannerFromBlob` 与 `mergePlannerIntoBlob`，否则可能丢失/破坏配置。 |
| **性能影响** | • 读取/写入 Planner 配置的额外 JSON 序列化成本极低（微秒级），对整体调度路径影响可以忽略不计。<br>• 代码生成脚本在 operator 镜像构建时会 **跳过** 生成步骤（因为 `components/src` 未复制），避免不必要的 IO。 |
| **安全考虑** | • `runtime.RawExtension` 直接携带用户提供的 JSON，理论上可能成为 **注入攻击** 的入口（例如非法字段、巨型 payload）。目前依赖 Planner 服务在启动时进行 schema 验证，建议在 Operator 中加入 **大小限制** 与 **基本的 JSON 结构检查**，防止 DoS。 |
| **可维护性** | • 单一来源的 PlannerConfig 大幅降低了文档/代码同步工作量，新增/修改字段只需在 `planner_config.py` 中改动一次。<br>• DeepCopy 代码自动从 `PlannerSpec` 迁移到 `runtime.RawExtension`，简化了生成器的维护。 |
| **兼容性** | • 旧版 `PlannerSpec` 已在 CRD 中删除，对 **已有集群** 进行升级时会出现 **字段缺失**（v1beta1→v1beta1 升级）或 **字段被忽略**（v1alpha1→v1beta1 转换）。需要提供升级指南或迁移脚本，将旧的 `planner.enabled`、`plannerPreDeploymentSweeping`、`plannerArgsList` 映射至新的 JSON 结构。 |
| **CI/测试** | • CI 增加 `pyyaml>=6.0` 依赖，防止在 operator 工作流中缺少 yaml 解析。<br>• `validate_pydantic_models.py` 现在在 “operator‑only‑build‑context” 环境下会 **提前退出**，避免因缺少 `components/src` 而报错。 |

---  

### ⚠️ 潜在风险  

1. **升级路径不完整**：若用户直接升级到此版本而仍保留旧 `PlannerSpec`，转换函数会把旧字段丢失，导致 Planner 服务启动失败（找不到必需字段）。  
2. **RawExtension 失去结构化校验**：Operator 不再检查 Planner 配置内部合法性，全部依赖 Planner 服务。若 Planner 服务升级后 schema 变化，旧版本 Operator 仍会把旧 JSON 原封不动转发，可能导致运行时错误。  
3. **JSON 大小/结构攻击**：未对 `blob["planner"]` 的大小或深度做限制，恶意用户可提交极大或循环引用的 JSON，导致 Operator 在 `json.Marshal`/`json.Unmarshal` 时耗尽资源。  
4. **代码生成脚本入口变化**：`generate_pydantic_from_go.py` 现在会在 operator‑only 构建中直接返回 0，若后续有人在该镜像里手动运行生成脚本，可能误以为已成功而忘记更新模型。  
5. **文档/CRD 不一致**：Docs 已改为展示 RawExtension，但仍保留旧的 `PlannerPreDeploymentSweepMode` 章节被删除，可能导致搜索引擎或旧文档的误导。

### feat: embedding cache in agg vLLM node (#6153)
**SHA**: `33b3cb8` | 🔗 [查看提交](https://github.com/ai-dynamo/dynamo/commit/33b3cb8aba3b66f4be89f094b5461d8fa8343655)

**🎯 变更类型**：功能增强  
**⚡ 重要程度**：🔴高  
**📋 变更摘要**：在 vLLM 节点中引入了 DynamoMultimodalEmbeddingCacheConnector，实现了基于 CPU 的多模态嵌入缓存（LRU 逻辑），通过 ECTransferConfig 在调度器与工作节点之间下发 load/save/evict 指令。相应地在 `setup_vllm_engine` 中注入 `ec_both` 模式配置，并为 `WorkerFactory.create` 接口新增 `shutdown_endpoints` 参数以支撑多模态 worker 的关闭流程。新增单元测试覆盖缓存逻辑与工厂调用。

**🎯 影响范围**：  
- `components/src/dynamo/vllm/main.py`（引擎初始化）  
- `components/src/dynamo/vllm/multimodal_utils/multimodal_embedding_cache_connector.py`（核心缓存实现）  
- `components/src/dynamo/vllm/tests/*`（单元测试）  
- `components/src/dynamo/vllm/worker_factory.py`（工厂接口签名变更）  

**🔍 技术洞察**  

| 维度 | 影响分析 |
|------|----------|
| **架构影响** | • 新增 **EC（Embedding Cache）Transfer** 配置，实现调度器对 CPU 缓存的统一管理，遵循 vLLM “scheduler‑authoritative” 模式。<br>• 引入 `DynamoMultimodalEmbeddingCacheConnector` 作为 vLLM 的 `ECConnectorBase` 子类，负责在调度器端生成元数据（loads / saves / evicts）并在 worker 端执行对应的张量搬迁。<br>• `setup_vllm_engine` 现在在 Engine 实例化前提前设置 `ec_transfer_config`，确保 vLLM 能识别并使用该 connector。<br>• `WorkerFactory.create` 参数加入 `shutdown_endpoints`，为后续多模态 worker 的优雅关闭预留扩展点。 |
| **性能影响** | • **正向**：CPU 缓存降低了对同一多模态输入的重复编码开销，尤其在批次中出现相同图片/音频时可显著提升吞吐。<br>• **负向**：缓存命中需要在调度器端维护 LRU 数据结构，额外 O(1) *amortized* 操作；在极端高并发场景下可能产生轻微的调度器 CPU 竞争。<br>• 通过 **容量 (GB)** 限制，避免 OOM；使用 `bytes_per_embed` 计算精确大小，提升内存利用率。 |
| **安全考虑** | • 仅在本进程内的 CPU ↔ GPU 共享张量，不涉及网络传输，安全风险极低。<br>• 需要确保 **多模态数据（图片、音频等）** 不被意外持久化到磁盘；当前实现存于内存 `dict[str, Tensor]`，符合现有安全模型。<br>• 版本检查 (`MINIMUM_VLLM_VERSION`) 防止在不支持 ECTransfer 的 vLLM 版本上出现未定义行为。 |
| **可维护性** | • 代码遵循 vLLM 的 ECConnector 接口，后续升级 vLLM 时只需保持接口兼容。<br>• 单元测试覆盖 LRU、容量限制、保存/加载/驱逐路径，降低回归风险。<br>• 通过 `logger.info` 与 `logger.warning` 提供可观测性，便于运维排查。 |

**⚠️ 潜在风险**  
1. **版本兼容性**：在 vLLM 低于 0.17.0 时仅给出 warning，仍会尝试实例化 connector，可能导致运行时属性缺失或行为异常。  
2. **签名变更破坏**：`WorkerFactory.create` 新增 `shutdown_endpoints` 参数，所有内部或外部调用点必须同步更新，否则会出现 `TypeError`。  
3. **内存泄漏**：CPU 端 `self._cpu_store` 持有 `torch.Tensor`，如果缓存未被正确驱逐或在异常路径中未清理，可能导致长期占用内存。  
4. **并发安全**：当前实现的 LRU 是单线程 (`OrderedDict`)；在多线程调度器（如使用 `torch.distributed`）环境下需要确认所有调用在同一线程执行，否则会出现竞态。  
5. **容量计算误差**：`dtype_bytes` 通过创建空张量推算，若模型在运行时切换 dtype（例如通过 `torch_dtype` 参数），缓存大小的估算可能不匹配实际占用。  

**💡 关注建议**  

| 建议 | 说明 |
|------|------|
| **版本保护** | 在 `__init__` 中若检测到 vLLM 版本低于最低要求，直接 `raise RuntimeError`，避免后续隐式错误。 |
| **向后兼容** | 为 `WorkerFactory.create` 添加 `*args, **kwargs` 包装层或提供默认空列表的 `shutdown_endpoints` 参数，以免旧调用崩溃。 |
| **显式清理** | 在 connector 关闭（如进程退出）时提供 `def shutdown(self): self._cpu_store.clear()`，确保内存及时释放。 |
| **并发安全** | 若调度器可能在多线程/多进程环境运行，考虑使用 `threading.Lock` 包裹 `_cache_order`、`_loads_this_step` 等集合的修改。 |
| **容量校准** | 在每次 `update_state_after_alloc` 前重新计算 `self._bytes_per_embed`（基于 `vllm_config.model_config.dtype`），防止 dtype 动态变化导致容量溢出。 |
| **监控指标** | 引入 Prometheus/StatsD 指标：缓存命中率、当前占用字节、驱逐次数等，帮助运维评估实际收益。 |
| **文档与示例** | 在项目 README 或 Dynamo 文档中补充 `multimodal_embedding_cache_capacity_gb` 配置说明、典型容量选取（如 2‑4 GB 对 7B 模型的推荐），以及 `ec_both` 模式启用方式。 |
| **回归测试** | 增加跨进程（多 GPU）场景的集成测试，验证在真实 vLLM 集群下缓存同步与驱逐行为。 |

> **结论**：此次提交为 **功能增强**，为多模态推理提供了显著的 CPU 端嵌入缓存机制，理论上可降低重复计算、提升吞吐。风险主要集中在版本兼容、接口变更以及并发安全上。通过上述风险缓解措施即可在生产环境安全、平滑地 rollout。

### feat: Standardized Dynamo Error Type (#6303)
**SHA**: `c8276cd` | 🔗 [查看提交](https://github.com/ai-dynamo/dynamo/commit/c8276cd28b3b12f4a7779784f57cd952d4d74da3)

**🎯 变更类型**：功能增强 / 架构变更 / 安全修复  
**⚡ 重要程度**：🔴高  

**📋 变更摘要**  
本次 PR 引入 **统一的 DynamoError** 错误体系（`lib/runtime/src/error.rs`），并在整个代码库中替换原有的字符串/`anyhow` 错误表示方式。核心改动包括：

1. 新增 `ErrorType`、`BackendError` 枚举以及 `DynamoErrorBuilder`，实现可链式构造、序列化与 `std::error::Error` 的互转。

- **架构影响**  
  - **职责划分更清晰**：调度与容量检查的完整流程现在由 `SchedulerQueue` 单独管理，`KvScheduler` 只负责接收请求与周期性触发 `update`。

  - **正向**：  
    - 减少了两次锁（`ready` + `pending`）的竞争，改为仅操作 `pending`（单一 `Mutex<BinaryHeap>`）并在调度成功后直接调用 `schedule`，降低了锁持有时间。

    - `schedule` 中会在每次调度前再次调用 `slots.potential_blocks_and_tokens`，这一步仍是异步 I/O/计算，若 `pending` 中积压大量请求，单次 `update` 可能导致一次性大量并发的 `potential_*` 调用，需关注 CPU/内存占用峰值。

  - 代码路径未引入新的 unsafe，安全属性未受影响。  
  - `SchedulingRequest::respond` 现在接受 `Result<SchedulingResponse, KvSchedulerError>`，避免了在调度失败时“失去错误信息”而导致调用方假设成功的风险，提升了错误可见性。  

- **可维护性**  
  - 将调度核心抽象为 `SchedulerQueue::schedule`，后续若要更换调度算法（如加入更复杂的负载预测）只需改动此函数，降低耦合。  
  - 代码量压缩：原来在 `KvScheduler` 中的约 150 行调度逻辑被削减至约 30 行，仅保留请求收发与周期性更新。  

**⚠️ 潜在风险**  

1. **错误传播不匹配**  
   - `KvScheduler::schedule` 调用 `request.respond(Ok(...))` 已改为 `Result`，若其他模块仍以 `SchedulingResponse` 为返回类型使用 `await`，会出现编译错误或运行时 panic。需确认所有调用方已更新为 `Result<..., KvSchedulerError>`。  

2. **背压恢复的时延**  
   - `update` 仅在 `enqueue` 后尝试一次调度（若仍忙）以及每 60 s 的定时器触发。极端情况下，worker 可能在短暂瞬间释放容量但 `update` 未即时运行，导致请求在 `pending` 中多等一次 `recheck_interval`。可以考虑把 `recheck_interval` 调小或在 `prefill_complete/free` 之后显式调用 `queue.update()`（已经在 `update` 中实现，但需确保调用路径仍可触发）。  

3. **并发争用**  
   - `pending` 使用单一 `Mutex<BinaryHeap>`，在极高请求速率下可能出现锁竞争。若出现性能瓶颈，可考虑使用 `tokio::sync::Mutex` 的公平性调优或改用分段堆（sharded priority queue）。  

4. **资源泄漏**  
   - `SchedulingRequest` 中的 `resp_tx` 若未及时 `take()`，会在 `drop` 时自动关闭通道，但若调用 `respond` 前已经出现错误并未发送，调用方的 `await` 将得到 `RecvError`，需要在上层做好超时或错误处理。  

**💡 关注建议**  

- **测试覆盖**：  
  - 添加/强化高并发压力测试，验证在 `threshold_frac` 较低（强背压）时，`pending` 能够稳定增长且不会导致请求永久卡死。  
  - 检查所有 `await scheduler.schedule(...).await` 的调用点，确保错误 `Result` 被正确解包。  

- **调度频率**：  
  - 评估是否需要将 `recheck_interval` 从 60 s 调整为更小的值（如 1 s），或在 `prefill_complete`/`free` 事件后主动触发 `queue.update()`（已在 `update` 中实现，但应确认这些事件真正调用到了 `update`).  

- **监控指标**：  
  - 暴露 `pending` 大小、`ready`（已删除）以及 `all_workers_busy` 检查次数等关键指标，以便运营时快速定位背压异常。  

- **文档更新**：  
  - 在项目的调度器使用说明中标明 `SchedulerQueue` 现已负责完整调度，外部调用者只需要通过 `KvScheduler::schedule` 发送 `SchedulingRequest` 并等待 `Result<SchedulingResponse, KvSchedulerError>`。  

- **依赖升级审计**：  
  - 伴随 `Cargo.lock` 的大量版本升级，运行 `cargo audit` 检查是否引入新的安全漏洞或不兼容的 API。  

通过上述改动，dynamo 的 KV 路由层在高负载场景下能够正确执行背压，调度路径更简

**⚡ 重要程度**：🔴高  

**📋 变更摘要**  
本次提交彻底去除 `Component` 在公开 API 中的使用，改为直接通过 `DistributedRuntime.endpoint(..)` 获取和操作 `Endpoint`。代码层面删除了 `Component` 的 Python、Rust 绑定以及相应的内部缓存，实现了 “endpoint‑only” 的设计，以避免因复用同一 Component 导致的重复指标注册和资源冲突。  

**🎯 影响范围**  
- `components/src/dynamo/*`（所有子项目：global_planner、global_router、router、sglang、trtllm、vllm、router）

- 运行时 `runtime/src/component.rs`（新增组件缓存）  

**🔍 技术洞察**  

- **架构影响**  
  - 1️⃣ 将 **Component** 从公共入口剥离，仅保留内部缓存，所有外部调用必须使用 `Runtime.endpoint(...)`。  
  - 2️⃣ 在 Rust 端 `Namespace::component` 增加了基于 `DashMap` 的 **component_cache**，保证同名 Component 复用，依旧避免重复指标注册。  
  - 3️⃣ `WorkerMetricsPublisher.create_endpoint`、`KvEventPublisher` 等 API 由接收 `Component` 改为接收 `Endpoint`，内部通过 `endpoint.inner.component()` 取得组件信息，实现向后兼容的最小侵入。

- **性能影响**  
  - **正面**：去除跨多个 endpoint 重复创建 `Component` 的开销，避免多次注册同一 metrics registry，显著降低启动时的初始化成本和内存占用。  
  - **负面**：首次调用 `runtime.endpoint` 时仍需在 Rust 端创建（或缓存） `Component`，但引入 `DashMap` 实现 lock‑free 读取，整体延迟对比旧实现可忽略不计。

  - 组件的 NATS/Etcd 注册逻辑保持不变，未引入新的网络交互。  
  - 因去除 `Component` 对外暴露，外部代码无法直接获取底层 `Component` 对象，降低了误用或恶意篡改 metrics registry 的风险。  

**⚠️ 潜在风险**  

1. **向后兼容**：老代码仍可能在运行时尝试调用 `endpoint.component()`，若未更新将抛出 `AttributeError`。  
2. **缓存一致性**：`DashMap` 缓存的组件在高并发创建时可能出现短暂的双实例（虽最终会统一），若用户在组件创建后立即进行副作用操作（如手动注册自定义 metric），可能产生重复。

4. **文档/示例**：示例和 README 中仍可能出现 “Component” 相关说明，用户在阅读时会产生困惑。  

**💡 关注建议**  

- **代码迁移**：在所有业务代码中搜索 `component` 参数或 `endpoint.component()` 调用，统一改为直接使用 `Endpoint`。

- **监控验证**：部署后对比启动时的 Prometheus metrics 注册数量，确认重复注册已消除，且新注册的指标仍完整。  
- **并发创建测试**：加入高并发获取同名 endpoint 的压力测试，确认 `component_cache` 的锁自由特性在实际生产负载下不会产生竞争错误。  
- **文档清理**：更新 `dynamo/_core.pyi`、官方文档、示例代码和 README 中所有关于 `Component` 的描述，避免误导新用户。  
- **回滚准备**：保留 `Component` 的内部实现（未在 `__all__` 中导出），以便在出现不可预见的兼容性问题时能够快速恢复旧接口。

### ⚠️ 潜在风险

1. **持久化不兼容**  
   - `start_kv_block_indexer` 明确禁止 `durable_kv_events`。若用户在配置文件中误开启此选项，将导致运行时 panic。需要在文档或 CLI 参数层提前校验，防止误操作。  

2. **跨进程网络错误**  
   - 查询端点通过 TCP（或 RPC）与独立索引器通信，网络抖动或防火墙策略变化可能导致查询超时。当前实现依赖 `tokio::spawn`，未对重试或超时做细粒度控制。  

3. **资源泄漏**  
   - `start_kv_block_indexer` 返回 `Indexer` 实例并在内部 spawn 了查询服务任务。若调用方忘记保留返回值或在进程退出前未显式关闭，可能出现后台任务残留。  

4. **版本兼容**

5. **全局 TCP 取消 Token**

**🔍 技术洞察**  

- **架构影响**  
  - 引入 **系统管理 API（DYN_SYSTEM_PORT）** 让 LoRA 适配器可以在运行时通过 HTTP 动态加载/卸载，保持与已有的模型管理框架一致。

- **性能影响**  
  - **加载时**：首次加载 LoRA 会触发网络 I/O（S3 下载）并在 GPU 内存中分配额外显存，`DYN_LORA_PATH` 的本地缓存可以避免重复下载。  
  - **推理时**：LoRA 参数会与基础模型参数融合，增加的显存占用约为 `rank * hidden_dim * 4`（取决于 LoRA rank）。示例中默认 `--max-lora-rank 64`，对 1‑GPU 环境影响可控。  
  - **吞吐量**：在聚合单 worker 场景，LoRA 的额外计算开销通常在 5%–15% 范围，受模型大小与 LoRA rank 的影响。  
  - **网络**：系统 API 暴露在 `9090` 端口，仅用于内部集群间通信；若对外开放需注意额外的带宽与 latency。

- **安全考虑**  
  - **凭证管理**：使用 `minio-secret.yaml` 通过 Kubernetes Secret 注入 MinIO 访问密钥，符合最小权限原则。  
  - **系统 API**：`DYN_SYSTEM_ENABLED=true` 与 `DYN_SYSTEM_PORT=9090` 使 LoRA 管理接口可被任意 HTTP 客户端调用，若未做网络隔离（NetworkPolicy）或身份校验，可能导致未授权用户加载任意 LoRA（潜在模型后门）。  
  - **对象存储**：S3 URI 直接指向 MinIO bucket，需确保 bucket 权限严格控制，防止恶意覆盖或读取其他租户的 LoRA。  
  - **容器镜像**：示例使用官方 `vllm-runtime` 镜像，已签名；若使用自定义镜像，请保持同等安全审计。  

**⚠️ 潜在风险**  

1. **未授权 LoRA 加载**：管理 API 若在公开网络或未受限的 Namespace 中暴露，攻击者可注入恶意 LoRA，导致模型行为被篡改。  
2. **显存超限**：在单 GPU 部署时，多个 LoRA 同时加载可能导致显存溢出，触发 OOM，尤其在高 rank / 多 LoRA 场景。  
3. **S3 连接失效**：MinIO/Network 问题会导致 LoRA 下载失败，导致工作者启动错误或运行时请求报错。

5. **拆分部署不一致**：在 prefill/decode 拆分模式下，仅在 prefill 加载 LoRA 会导致 decode 使用基模型，产生输出不一致的疑惑。

- **LoRA 生命周期管理**：使用 `DynamoModel` CRD 作声明式管理，配合 GitOps 流程；在 CI 中加入 LoRA 与基模型版本匹配检查。  
- **异常处理**：在 worker 启动脚本或容器入口中加入对 `aws s3`、`curl` 等网络操作的重试与超时控制，提升弹性。  
- **文档同步**：保持 `README.md` 与实际 `agg_qwen_lora.yaml` 参数一致；建议在 CI 中加入文档 lint 检查，防止示例代码与文档漂移。  
- **安全审计**：定期审计 MinIO bucket 权限以及 `hf-token-secret`、`minio-secret` 的访问范围，避免凭证泄露。  

总体而言，此次提交为多模态 LoRA 的 **运行时动态化** 与 **K8s 可复制部署** 提供了完整的用户指引和示例实现，提升了 Dynamo 在视觉‑语言大模型细调场景的可用性。但需要在实际集群中通过网络策略、资源配额以及版本兼容性检查来降低潜在风险。
