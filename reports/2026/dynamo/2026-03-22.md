## 🔍 技术洞察

### 架构影响
1. **新增依赖链**  
   - **Grove**：Dynamo 现在显式依赖 Grove 提供的 `ClusterTopology` CR 与 `PodCliqueSet`/`PodCliqueScalingGroup` API。Operator 必须在运行时确认 `runtimeConfig.GroveEnabled` 为 true，才能使用拓扑相关字段。  
   - **ClusterTopology**：作为用户配置的全局资源，Operator 通过只读 RBAC 读取其定义，用于 webhook 验证和层级检查。  

2. **CRD 扩展**  
   - 新增 `spec.topologyConstraint`（部署级）与 `spec.services[].topologyConstraint`（服务级），保持 **向后兼容**（字段为 optional），但在 **创建** 时若任意层出现约束则必须提供 `topologyProfile`。  

3. **状态层**  
   - 新增 `ConditionTypeTopologyLevelsAvailable`，通过 `propagateTopologyCondition` 将 Grove PCS 的 `TopologyLevelsUnavailable` 条件映射到 DGD。该条件在 UI / `kubectl describe` 中可直接看到，以便运维监控。

4. **不可变性**  
   - 拓扑约束被标记为 **immutable**（创建后不可更改），与 Grove 框架的不可变约束保持一致，简化调度实现但需要在升级/变更时提醒用户 **删除‑重建**。

### 性能影响
| 维度 | 正面影响 | 潜在负面 |
|------|----------|----------|
| 调度延迟 | 拓扑约束帮助调度器在同一拓扑域内打包 pod，减少跨节点通信延迟，提升推理吞吐。 | 约束会缩小调度搜索空间，若集群中满足域的资源不足，调度器可能需要更长时间回滚或进入 `Pending` 状态。 |
| 运行时开销 | 只在生成 PCS 及 ScalingGroup 时写入一次字段，运行时几乎无额外成本。 | Operator 在每次 **Reconcile** 时会额外读取 `PodCliqueSet` 条件并更新 DGD 状态，产生一次 `GET`+`UPDATE`，对高频 Reconcile 场景影响可忽略。 |
| API Server 负载 | 新增 `clustertopologies` 只读权限，查询频率仅在 webhook 创建时（单次）及偶尔的状态同步，影响极小。 |

### 安全考虑
- **最小权限**：仅授予 `get/list/watch` `clustertopologies` 的 ClusterRole，且绑定至 Operator ServiceAccount，未授予写权限，风险可控。  
- **输入校验**：Webhook 对 `packDomain` 格式、必填、层级关系、ClusterTopology 存在性做了强校验，防止恶意或错误的 CR 触发调度错误或资源泄漏。  
- **不可变性**：防止运行时对拓扑约束的修改导致已有 Pods 与新约束不一致，引发调度不确定性。  
- **潜在信息泄露**：Operator 通过 RBAC 能读取整个 `ClusterTopology`（包括节点标签），但该资源本身是集群级公开的，风险与现有 Grove 权限等同。

### 兼容性与迁移
- **向后兼容**：未填写 `topologyConstraint` 的旧版 DGD 仍可正常调度，所有新增字段均为 `omitempty`。  
- **升级路径**：在升级至支持 TAS 的 Operator 版本时，已存在的 DGD **不会** 自动获得拓扑约束；若用户想启用，需要 **删除‑重建**（因为约束不可变）。  
- **Helm Chart**：默认不启用 TAS，只有在 `spec.topologyConstraint` 被用户显式填写时才生效，保持现有部署的原样运行。

### fix: fix MyPy issue introduced in #7517 (#7562)
**SHA**: `725a845` | 🔗 [查看提交](https://github.com/ai-dynamo/dynamo/commit/725a845f2e7e9be9f29f2527279b986171f4a66f)

**🎯 变更类型**：Bug修复  
**⚡ 重要程度**：🟡 中  

**📋 变更摘要**  
本次提交修复了在 PR #7517 中引入的 MyPy 类型检查错误。具体做法是：在 `encode_worker_handler.py` 中将 `raw_request`（原本是 `dict`）包装为 `PreprocessedRequest` 实例后再传递给 `SglangMultimodalRequest`，从而满足 `SglangMultimodalRequest.request` 字段的类型要求。

**🎯 影响范围**  
- `components/src/dynamo/sglang/request_handlers/multimodal/encode_worker_handler.py`（核心业务逻辑）  
- 相关的类型定义：`dynamo.sglang.protocol.PreprocessedRequest`、`SglangMultimodalRequest`  
- 可能波及到调用该处理器的上层工作流（如多模态编码管线），但行为保持不变，仅类型层面更严谨。

**💡 关注建议**  
1. **代码可读性**：建议在注释中说明 `PreprocessedRequest` 的作用，避免后续维护者误以为是功能性改动。  
2. **测试覆盖**：确认单元测试已覆盖 `encode_worker_handler` 的路径，尤其是涉及多模态请求的序列化/反序列化，防止因包装对象导致的运行时错误。  
3. **MyPy 配置**：检查项目的 `mypy.ini` 是否已包含 `PreprocessedRequest` 的路径，确保 CI 中的类型检查不再报错。  
4. **向后兼容**：因为只是类型包装，运行时行为未变，但若外部代码直接访问 `SglangMultimodalRequest.request` 的字典属性，需确认仍能通过属性访问得到相同结构。  

总体而言，此次修改对功能无影响，仅提升了类型安全性，建议尽快合入主线并在 CI 中验证 MyPy 通过。
