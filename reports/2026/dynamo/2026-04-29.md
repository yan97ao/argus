### fix: add optimizationType enum validation to DGDR (#8796)
**SHA**: `2e31245` | 🔗 [查看提交](https://github.com/ai-dynamo/dynamo/commit/2e3124546c6d6c6ec1765c7ce18958bfdb6237ff)

**🎯 变更类型**：Bug修复 / 功能增强（新增 `OptimizationType` 枚举并加入校验）  
**⚡ 重要程度**：🔴高  
**📋 变更摘要**：在 DGDR（DynamoGraphDeploymentRequest）SLA 规格中引入 `OptimizationType`（`latency`、`throughput`）枚举，完善 Python、CRD、Go 结构体、转换逻辑、Webhook 校验以及文档；同时补充单元测试确保新字段的合法性与向后兼容性。  

**🎯 影响范围**：  
- `components/src/dynamo/profiler/utils/dgdr_v1beta1_types.py`（Python SDK）  
- Helm/CRD 相关 YAML（`nvidia.com_dynamographdeploymentrequests.yaml`）  
- Operator Go 代码：`api/v1beta1/dynamographdeploymentrequest_types.go`、`conversion.go`、`validation`、`deepcopy`、对应单元测试  
- 文档 `docs/kubernetes/api-reference.md`  

---

**🔍 技术洞察**  

- **架构影响**  
  - **新增可选字段**：`OptimizationType` 作为 SLA 的可选属性，对现有业务流程无侵入式改动，保持向后兼容。  
  - **CRD 与 Operator 同步**：通过 `kubebuilder:validation:Enum` 在 CRD 中声明枚举，Operator 在转换、Webhook、DeepCopy 中同步处理，保证集群层面的数据一致性。  
  - **跨语言一致性**：Python Enum 与 Go `type OptimizationType string` 对齐，避免跨语言序列化差异。  

- **性能影响**  
  - **几乎无运行时开销**：仅在 Admission Webhook 中进行一次字符串比较，CPU 与内存消耗可忽略。  
  - **转换路径**：新增字段的序列化/反序列化在 `mergeSLAWorkloadIntoBlob` 与 `applySLAAndWorkloadFromBlob` 中完成，仍然是浅拷贝，且已有路径已经过性能验证。  

- **安全考虑**  
  - **输入校验**：Webhook 现在显式检查 `OptimizationType` 是否为合法枚举，防止恶意提交导致 Operator 进入未知分支或触发 panic。  
  - **异常处理**：在 `applySLAAndWorkloadFromBlob` 中仅在值匹配既有枚举时写入，防止非法字符串写入内部结构体，降低潜在的注入风险。  

---

**⚠️ 潜在风险**  

1. **向后兼容性**  
   - 老版本的 Operator（未包含此字段的代码）在读取带有 `optimizationType` 的 CR 时会忽略该字段，可能导致用户期望的优化目标未被使用。  
   - 老版本的客户端（如旧的 Python SDK）如果未同步 Enum 定义，可能在生成 YAML 时出现未知字段错误。  

2. **字段默认值**  
   - 目前未设定默认值，若用户未显式填写，内部逻辑需确保 `nil`（Go）或 `None`（Python）被安全处理，防止在后续基于该字段做决策时出现 `nil` 引用或业务分支错误。  

3. **测试覆盖不足**  
   - 新增的 Validation 只覆盖了枚举合法性，未覆盖与其他 SLA 字段（如 `ttft`、`itl`）组合的业务逻辑，后续可能出现冲突或不一致的配置。  

---

**💡 关注建议**  

- **版本升级指引**：在发布新 Operator 版本时，提供升级说明，提醒用户同步 Operator、Helm chart 与 SDK 至相同版本，以免出现字段不识别的情况。  
- **默认行为文档**：在文档中明确若 `optimizationType` 为空或未设置时的默认优化策略（如保持原有行为），帮助用户理解系统行为。  
- **持续集成**：在 CI 中加入 CRD 与 SDK 生成的对比测试，防止手动忘记同步枚举定义导致不一致。  
- **监控告警**：在 Admission Webhook 记录 `optimizationType` 校验失败的事件，便于快速定位用户配置错误。  
- **业务层适配**：检查调度器、Profiler 等后端组件是否已经支持基于 `OptimizationType` 的不同调优路径，若尚未实现，需规划后续功能实现路线。

### fix: surface DGD name-length violation as terminal failure (#8807)
**SHA**: `4270c93` | 🔗 [查看提交](https://github.com/ai-dynamo/dynamo/commit/4270c93a36d99f68eded2625b2a96cf1952c4fb4)

**🎯 变更类型**：Bug修复 / 安全修复 / 性能优化  
**⚡ 重要程度**：🔴高  
**📋 变更摘要**：  
1. 在 Python Profiler 中加入 `_validate_dgd_service_name_lengths`，在生成 DGD 配置后检测「DGD 名称 + Service 名称」的组合长度是否超过 Kubernetes Pod‑name（45字符）限制，超限直接抛 `ValueError` 并终止 profiling。  
2. 为相关测试补充 `DGDR_NAME` 环境变量，确保验证逻辑在 CI 中被触发。  
3. 所有 `config_modifiers`（protocol、sglang、trtllm、vllm）改为使用 `uuid4().hex[:8]` 为 `metadata.name` 添加随机后缀，防止名称冲突及长度意外。  
4. 在 Operator 控制器的 `createDGD` 过程中，将 400/403/422 等永久性 API 拒绝视为 **终止失败**（PhaseFailed），避免无限重试。

---

### 🎯 影响范围
- **Python 侧**：`components/src/dynamo/profiler/profile_sla.py`、各单元/集成测试文件。  
- **配置生成**：`components/src/dynamo/profiler/utils/config_modifiers/*`（protocol、sglang、trtllm、vllm）。  
- **Kubernetes Operator**：`deploy/operator/internal/controller/dynamographdeploymentrequest_controller.go`。  
- **CI/UT**：所有新增/修改的测试用例。

---

### 🔍 技术洞察

| 维度 | 影响 |
|------|------|
| **架构影响** | • 在 **profiling** 流程中加入前置校验，提升了错误定位的前移，防止后端创建 Pod 失败后才暴露问题。<br>• Operator 现在能在 **Reconcilation** 阶段快速判定“永久性”错误，直接转为 `Failed`，避免了无意义的Retry循环，提升控制平面的可靠性。 |
| **性能影响** | • 额外的长度检查和 `uuid4()` 生成开销极低（微秒级），对整体 profiling 运行时间几乎没有可感知影响。<br>• 通过提前失败可省去后续的资源渲染与 API 调用，整体算是 **性能提升**。 |
| **安全考虑** | • 新增对 **403 Forbidden** 的专门处理，防止因权限错误导致的无限重试，降低对 API Server 的压力。<br>• 抛出的 `ValueError` 只在本地 profiling 进程中出现，不会泄露敏感信息。 |
| **可维护性** | • 验证逻辑被抽成独立函数，单元测试覆盖完整，后续若需调整长度阈值或命名规则，只需修改该函数。<br>• `metadata.name` 改为 `uuid` 后缀，使得不同 DAG/运行实例的 DGD 名称必然唯一，降低因手动重复而产生的错误。 |
| **兼容性** | • 旧版 `metadata.name`（固定字符串）仍然可用，只是现在会被随机后缀覆盖，可能对 **外部监控/日志** 中依赖固定名称的脚本产生影响，需要相应更新。<br>• 对非 Operator（Standalone CLI）路径仍保持原有行为，仅在 `DGDR_NAME` 未设置且配置中无 `metadata.name` 时 **跳过** 校验，保持向后兼容。 |

---

### ⚠️ 潜在风险
1. **用户体验**：当名称超长时 `ValueError` 会直接终止 profiling，若调用方未捕获该异常，会导致 CLI 退出且返回码可能为 1，需在文档或用户手册中说明。  
2. **监控/脚本依赖**：`metadata.name` 加上随机 UUID 可能导致已有依赖「固定名称」的监控或告警规则失效。  
3. **环境变量依赖**：在 Operator 环境中 `DGDR_NAME` 必须始终被设置，否则会走「fallback」路径并仅打印 DEBUG，可能错过真正的长度违规。若环境变量意外缺失，可能导致不符合预期的名称通过验证。  
4. **错误信息泄露**：错误信息直接拼接了原始 DGDR 名称，虽然不涉及敏感信息，但在共享日志时可能暴露内部命名约定，建议在生产环境对日志级别进行审查。  

---

### 💡 关注建议
- **文档更新**：在用户手册和 Operator README 中注明 **DGDR 名称 + Service 名称 ≤ 45** 的限制，以及如何通过 `DGDR_NAME` 或 `overrides.dgd.metadata.name` 自行调整。  
- **监控适配**：如果有基于 `metadata.name` 进行的外部查询或告警，改为匹配 `*-{8位uuid}` 的正则或使用标签而非名称。  
- **CI/回归**：保持当前单元/集成测试的覆盖率；建议在 CI 中加入 **长名称自动生成** 的随机测试，确保阈值改动不破坏现有逻辑。  
- **可配置阈值**：将 `45` 设为常量或配置项，便于未来若 Kubernetes 对名称长度上限调整时，只需改动一处。  
- **错误处理统一**：在调用 `run_profile` 的上层（CLI/SDK）捕获 `ValueError`，并转化为用户友好的错误码/提示，避免栈追踪直接输出。  
- **日志级别**：将 `logger.debug("skipping DGD service name length validation.")` 的信息提升至 `info` 或提供 `--verbose` 开关，以利于排查实际运行中的名称来源。  

---  

**结论**：本次提交通过前置校验、唯一化命名及对永久性 API 错误的快速终止，显著提升了系统的可靠性和可维护性，属于 **高优先级** 的安全/可靠性修复。只需在文档、监控和异常包装上做好配套工作，即可平滑落地。

### feat(container): add planner test stage and unify test image builds via inline _test stages (#8739)
**SHA**: `fa99196` | 🔗 [查看提交](https://github.com/ai-dynamo/dynamo/commit/fa99196cb0976252e8e4492e3c5f3472cf4dab19)

**🎯 变更类型**：功能增强 / 架构变更  

**⚡ 重要程度**：🔴高  

**📋 变更摘要**  
- 在容器构建流程中引入统一的 “_test” 阶段，废除单独的 `docker-test-image-build` Action，改由 `docker-remote-build` 支持 `build_target` 与 `tag_suffix` 参数，实现测试镜像与运行时镜像共用同一渲染的 `Dockerfile`。  
- 新增 `planner_test` 阶段并在 `planner.Dockerfile` 中提供完整的测试环境（shell、tar、grep 等），同时在模板化的 `Dockerfile.template` 中自动插入 `test_stage.Dockerfile`。  
- 改进了 `container/render.py` 的内部结构，抽取 Jinja2 环境创建与渲染上下文，提升可维护性。  
- 更新 CI 配置，使测试镜像构建不再冲突，并同步调整过滤规则、构建缓存逻辑与构件命名。

**🎯 影响范围**  
- CI/CD 工作流：`.github/workflows/shared-build-image.yml`、`.github/actions/docker-remote-build`、`.github/actions/docker-test-image-build`（已删除）  
- Dockerfile 生成与渲染：`container/render.py`、`container/Dockerfile.template`、`container/templates/test_stage.Dockerfile`、`container/templates/planner.Dockerfile`  
- 单元测试代码：`components/src/dynamo/planner/tests/unit/test_kube.py`（添加默认命名空间 fixture）  
- 构建缓存与制品命名：所有涉及 `tag_suffix` 的步骤  

**🔍 技术洞察**  

- **架构影响**  
  - **统一构建入口**：通过 `docker-remote-build` 完成 runtime、runtime_test、planner_test 等所有阶段的构建，消除了专用的 test‑image Action，简化了工作流依赖图。  
  - **阶段化镜像**：在 `Dockerfile.template` 中根据 `target` 动态插入 `test_stage.Dockerfile`，实现“同一文件多阶段”模式，降低 Dockerfile 维护成本，并保证 runtime 与 test 镜像使用相同的基础层，从而提升镜像一致性。  
  - **缓存策略升级**：引入 `tag_suffix` 参数后，测试构建会使用独立的缓存命名空间（如 `…-test-cache`），但仍回溯到基础缓存，兼顾层复用与并行构建。  

- **性能影响**  
  - **构建时间**：统一渲染后，测试镜像构建无需额外 `FROM` 步骤或再次拉取 runtime 镜像，层复用率提升，预计可将 CI 中的测试镜像构建时间降低 20%–30%。  
  - **缓存命中率**：通过 “先读 suffixed cache → 再读 base cache” 的双层缓存策略，能够在同一 CI 运行中共享 runtime 与 test 阶段的中间层，进一步缩短后续构建。  

- **安全考虑**  
  - **权限与用户**：`test_stage.Dockerfile` 仍然在 `runtime` 阶段后切换到非 root `dynamo` 用户，保持运行时最小权限原则。  
  - **构建参数泄露**：新增 `build_target`、`tag_suffix` 通过 GitHub Actions 输入传递，默认均为空，未引入额外敏感信息。  
  - **废弃的 Action**：删除的 `docker-test-image-build` 不再在工作流中使用，避免了潜在的未维护代码路径。  

**⚠️ 潜在风险**  

1. **模板渲染兼容性**  
   - 新增 `test_stage.Dockerfile` 使用 Jinja2 变量 `{{ target }}`。若旧的渲染脚本或手动 `docker build -f` 仍引用 `container/Dockerfile.test`（已从过滤列表移除），可能导致渲染失败。  
2. **缓存命名冲突**  
   - `tag_suffix` 为 `test` 时，主缓存槽改为 `"{framework}-cuda{N}-{arch}-test-cache"`，若已有同名缓存（例如手动标记为 `test` 的正式镜像），会发生意外覆盖。建议在正式发布分支仍保持默认空 suffix。  
3. **CI 并发构建**  
   - 同一 job 中同时运行 runtime 与 runtime_test，虽然共享缓存，但若 `CACHE_ARGS` 中顺序错误或网络抖动，可能导致某一次缓存拉取失败导致整体构建回退到全新构建，增加时延。  
4. **单元测试改动**  
   - 新增 `mock_namespace` fixture 依赖 `patch` 路径 `dynamo.planner.connectors.kubernetes_api.get_current_k8s_namespace`，如果未来该函数搬迁或重命名，测试会失效。  

**💡 关注建议**  

- **CI 稳定性**：在 `shared-build-image.yml` 中为 `tag_suffix` 加入显式校验（如只能是 `test` 或空），防止误传导致缓存命名冲突。  
- **文档同步**：更新 README 与内部文档，明确 “runtime_test” 与 “planner_test” 阶段的构建方式以及 `docker-remote-build` 新增参数的使用示例。

### feat(gms): initialize with scrach-aliased kv cache (#8686)
**SHA**: `e7dc0fd` | 🔗 [查看提交](https://github.com/ai-dynamo/dynamo/commit/e7dc0fd5faa014c18ff7b8b5c6c9a7e4db38ddf6)

**🎯 变更类型**：功能增强  

**⚡ 重要程度**：🔴 高  

**📋 变更摘要**  
本次提交为 GPU Memory Service (GMS) 引入 “scratch‑aliased KV 缓存” 的两阶段初始化机制。通过在启动阶段使用单个物理页面映射到多个虚拟地址（scratch KV），实现 KV 缓存占用的内存仅在 **shadow‑init → wake‑up** 之间保留 VA 而不消耗真实显存，随后在 wake‑up 时迁移为真实的 per‑tensor backing。新增环境变量 `DYN_GMS_SCRATCH_KV_ENABLED`、相应的 allocator、manager、patch 与 worker 逻辑，并删除旧的 shadow‑mode 实现。

**🎯 影响范围**  
- `components/src/dynamo/vllm`（启动脚本、engine 参数）  
- `lib/gpu_memory_service/client`（`memory_manager.py`、`torch/allocator.py`）  
- `lib/gpu_memory_service/common`（`utils.py`）  
- `lib/gpu_memory_service/integrations/vllm`（`patches.py`、`utils.py`、`worker.py`）  
- 示例部署文件 `examples/backends/sglang/deploy/agg_gms.yaml`  

**🔍 技术洞察**

| 维度 | 影响 |
|------|------|
| **架构影响** | 1. **双路径分配**：在同一 tag 下，`_TagState.is_scratch` 控制走 **scratch‑mapping**（本地仅 VA）还是 **server‑backed mapping**（通过 GMS RPC）。<br>2. **两阶段 KV 生命周期**：<br>   - **init**：`create_scratch_mapping` 分配 1 块物理页面并在 VA 上多次映射。<br>   - **sleep**：`unmap_all_vas` 同时释放 real 和 scratch 物理，保留 VA 预留。<br>   - **wake**：`prepare_scratch_for_reallocation` 将 scratch 记录迁入 `_mappings`，`ensure_scratch_disabled` 切换路由，随后走标准的 `reallocate_all_handles` + `remap_all_vas`。<br>3. **统一环境变量**：`ENV_SCRATCH_KV_ENABLED` 与 operator/launcher 中的常量保持同步，降低配置漂移风险。 |
| **性能影响** | - **启动时显存占用几乎为 0**（仅 1 MiB scratch 页面），显著降低多实例 failover 场景的显存竞争及 dead‑lock 概率。<br>- **wake‑up 期间额外一次 reallocation**（遍历所有 VA），但这只在发动机切换时发生，成本可接受。<br>- **patches** 中对 `MemorySnapshot.measure`、`request_memory` 的轻量改写对运行时性能基本无影响。 |
| **安全考虑** | - 新增的环境变量解析使用白名单的 “truthy” 检查 (`is_truthy_env`)；不存在注入风险。<br>- 所有 CUDA 调用仍在同步 (`cuda_synchronize`) 后执行，避免并发释放导致的 UB。<br>- 通过 `ensure_scratch_disabled` 强制在 RW 连接后才切换为 server‑backed，防止在只读模式下误写。 |
| **可维护性** | - 引入 `_ScratchMapping` 数据结构以及 `_scratch_mappings` 字典，使 scratch 与常规映射在同一 `GMSClientMemoryManager` 中清晰分离。<br>- allocator 中的分支逻辑通过 `state.is_scratch` 标记，避免在外部代码中传播判断。<br>- 旧的 `shadow` 代码被删除，减少冗余路径。<br>- 代码注释与日志信息充分，便于后续调试。 |

**⚠️ 潜在风险**  
1. **迁移错误**：若 `prepare_scratch_for_reallocation` 没有在 `wake_up` 前被调用，scratch VA 会保持 `scratch_handle == 0`，导致后续 `remap_all_vas` 失败或出现未映射访问。  
2. **多进程竞争**：在同一设备上启动多个进程，只有一个进程会在 `wake_up` 时执行 `ensure_scratch_disabled`。若两个进程几乎同时完成 sleep→wake，可能出现 **双重迁移**（重复创建 `LocalMapping`）。  
3. **兼容性**：依赖的 vLLM 接口（如 `NixlConnector.register_kv_caches`）在不同 vLLM 版本中可能更名或行为变化，导致补丁失效或未能正确恢复 KV 注册。  
4. **资源泄漏**：`destroy_scratch_mapping` 只在 `close` 或显式 `_gms_free` 路径调用；如果异常路径跳过 `close`，残留的 scratch_handle 可能未释放。  
5. **错误配置**：环境变量拼写错误或未设置导致功能意外关闭，用户可能仍面临原始 KV 分配压力而不易察觉。

**💡 关注建议**  
- **测试覆盖**：在 CI 中加入 **多实例、TP>1、不同 GMS lock 模式** 的组合测试，确保 `sleep → wake` 的完整迁移流程以及 Nixl 注册恢复能够在并发场景下稳健运行。  
- **监控与日志**：建议在生产中开启 `DYN_GMS_SCRATCH_KV_ENABLED=1` 时，监控日志 `"[GMS] Reserved … scratch"`、`"[GMS] Promoted … scratch VAs"` 与 `"[GMS] Registered … kv_cache tensors"` 的出现次数，以验证每次启动/重启都完成了完整的两阶段流程。  
- **安全审计**：虽然当前没有输入直接影响 CUDA 调用，但仍应确认 `cumem_create_tolerate_oom` 对 OOM 处理符合预期，防止因单页分配失败导致启动异常。  
- **回滚路径**：保留 `DYN_GMS_SCRATCH_KV_ENABLED` 的默认关闭（false），并在文档中明确说明在 **failover / 多副本** 场景才开启此特性。  
- **兼容性检查**：在每次升级 vLLM（或 GMS）时，运行 `apply_scratch_kv_patches()` 的检测路径，确认 `patch_register_kv_caches`、`patch_request_memory` 等仍能成功导入并作用。可考虑在 `apply_scratch_kv_patches` 中加入 **版本检测**，在不兼容时给出清晰错误信息。

**⚡ 重要程度**：🔴高  

**📋 变更摘要**  
在 `profiling` Job 的创建流程中加入 `ensureOutputCopierKubeAPIAccess`，当用户在 pod 级别将 `automountServiceAccountToken=false` 时，自动为 operator 控制的 *output‑copier* sidecar 注入一个 **Projected Volume**，内含 ServiceAccount Token、Kubernetes root CA ConfigMap 与 Namespace Downward API。该 volume 只会挂载到 `output‑copier` 容器，避免影响用户业务容器，同时在用户自行覆盖同名 Volume 时进行保护恢复。  

**🎯 影响范围**  
- `deploy/operator/internal/controller/dynamographdeploymentrequest_controller.go`（Job 创建入口）  
- `deploy/operator/internal/controller/profiling_job_overrides.go`（新增函数、常量）  
- 相关单元测试 `profiling_job_overrides_test.go`  
- 受影响的运行时组件：**Operator**（尤其是 profiling job）  

**🔍 技术洞察**  

- **架构影响**  
  - 在原有的 profiling job 结构上添加了一个 **专用的 Projected Volume**（`output-copier-kube-api-access`），仅供 operator 拥有的 `output‑copier` sidecar 使用。  
  - 通过 `mergeNamedSlice` 与 `removeVolumeMountByName` 实现 **幂等且防冲突** 的 volume 合并与 mount 整理，保证用户自定义的同名 volume 不会导致 sidecar 失去访问能力。  
  - 引入常量 `ServiceAccountTokenExpirationSeconds = 3600`，使 token 生命周期受控（默认 1 h），提升安全性。  

- **性能影响**  
  - 投入的资源非常有限：创建一个 projected volume、读取 token、CA、namespace。对 Job 启动时间的影响在毫秒级，可忽略。  
  - 由于 token 会在 1 h 后自动刷新，sidecar 不需要频繁重新挂载或重新创建 Pod。  

- **安全考虑**  
  - **最小权限**：仅在 `output‑copier` 容器中挂载 token，其他业务容器不暴露。  
  - **只读挂载** (`ReadOnly: true`) 防止容器写入或篡改 token。  
  - **短期 token**（3600 s）相较于默认 ServiceAccount token（默认 1 y）大幅降低泄漏风险。  
  - 采用 **ConfigMap 投射** 方式获取 `kube-root-ca.crt`，确保 TLS 验证链完整。  
  - 需要确认目标集群已提供 `kube-root-ca.crt` ConfigMap（K8s ≥ 1.20 自动创建），否则 Pod 将因找不到 ConfigMap 而挂起。  

**⚠️ 潜在风险**  

1. **用户自定义同名 Volume**  
   - 如果用户显式声明了名为 `output-copier-kube-api-access` 的 Volume（例如 EmptyDir），`ensureOutputCopierKubeAPIAccess` 会覆盖为 projected volume。虽然已在测试中验证“保护”行为，但用户若期望自己的 Volume 生效会被意外改写。  

2. **Cluster 兼容性**  
   - `ServiceAccountTokenProjection` 与 `ProjectedVolumeSource` 在较老的 K8s 版本（< 1.12）并不受支持。若在这些版本上运行，Job 会因不识别字段而失败。  

3. **ConfigMap `kube-root-ca.crt` 不存在**  
   - 某些自建集群或极旧的发行版可能未自动创建该 ConfigMap，导致投射失败，进而阻塞 profiling job。  

4. **Sidecar 权限泄漏**  
   - 虽然 token 只读挂载，但 sidecar 仍拥有对 `output` ConfigMap 的写权限。若 sidecar 被利用（例如容器逃逸），攻击者可借此写入任意 ConfigMap 内容。  

**💡 关注建议**  

- **文档更新**：在 Operator 使用手册中说明 “当 `automountServiceAccountToken=false` 时，Operator 会自动为 output‑copier 注入专用 token”，并提醒用户不要自行创建同名 Volume。  

- **版本检查**：在 Operator 启动时加入对集群版本的检测，若低于 1.12 或缺少 `kube-root-ca.crt` ConfigMap，给出明确的错误提示或回退方案（如使用默认 ServiceAccount token）。  

- **可配置化**：考虑将 `ServiceAccountTokenExpirationSeconds` 与 `VolumeNameOutputCopierKubeAPIAccess` 通过 CRD/CLI 参数暴露，方便在安全合规要求更严格的环境下调整。  

- **安全审计**：定期审计 `output‑copier` 容器的 RBAC 权限，仅授予更新 DGDR 输出 ConfigMap 所需的最小 `patch` 权限。  

- **测试覆盖**：保持现有单元测试，并在 CI 环境加入 **集群兼容性**（不同 K8s 版本）以及 **ConfigMap 缺失** 场景的集成测试。  

- **监控告警**：若 profiling job 因 volume 挂载错误或 token 投射失败而进入 `Failed` 状态，触发告警，帮助运维快速定位问题。
