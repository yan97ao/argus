### fix(container): use pip_target in vllm runtime Dockerfile (#10647)
**SHA**: `5a2eb79` | 🔗 [查看提交](https://github.com/ai-dynamo/dynamo/commit/5a2eb798d0162fb7452dc7d736059c038cd4018c)

**🎯 变更类型**：Bug 修复  
**⚡ 重要程度**：🟡 中  

**📋 变更摘要**  
在 `container/templates/vllm_runtime.Dockerfile` 中，将原硬编码的 `uv pip install --system --no-deps` 替换为 `uv pip install {{ pip_target }} --no-deps`，使 Dockerfile 能够通过模板变量 `pip_target` 动态指定 pip 安装源或额外参数，从而修正之前构建 vLLM Runtime 镜像时缺失自定义 pip 目标的问题。  

**🎯 影响范围**  
- `container/templates/vllm_runtime.Dockerfile`（Docker 镜像构建）  
- 依赖该模板的 CI/CD 流程与本地构建脚本  
- 可能波及使用 `Modelexpress` 包的下游镜像  

**💡 关注建议**  
1. **变量定义**：确保在渲染模板前始终传入合法的 `pip_target`（如 `-i https://pypi.org/simple` 或本地 wheel 路径），否则会导致构建失败。  
2. **安全审计**：`pip_target` 直接拼入命令，需防止注入风险；若接受外部输入，建议在上层进行白名单校验。  
3. **兼容性验证**：在本地及 CI 中重新跑一遍 vLLM Runtime 镜像的构建测试，确认老版本镜像仍能正常生成。  
4. **文档更新**：在容器构建说明中加入 `pip_target` 参数的使用指南与示例。  
5. **回滚路径**：如出现不可预期的安装错误，可临时恢复原 `--system` 参数或在 `pip_target` 中显式写入 `--system` 以保持兼容。  

总体来说，此改动提升了镜像构建的灵活性，但需要确保模板渲染阶段对 `pip_target` 的正确传递与安全性检验。
