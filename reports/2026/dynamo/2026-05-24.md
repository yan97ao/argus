### fix(snapshot): fail checkpoint jobs on helper container errors (#9850)
**SHA**: `2aaade3` | 🔗 [查看提交](https://github.com/ai-dynamo/dynamo/commit/2aaade3b54b417570533160991e12d4b41bf982d)

**🎯 变更类型**：Bug修复  
**⚡ 重要程度**：🔴高  

**📋 变更摘要**  
1. 修正 checkpoint‑job 在 *helper* 容器异常退出时未标记失败、未停止正在运行的 checkpoint 容器的问题。  
2. 引入对所有非‑helper 容器的终止检测、事件上报、K8s Job 注解更新以及对仍在运行的 checkpoint 容器发送 `SIGKILL`。  
3. 对 `setCheckpointStatus` 做幂等化与冲突重试：只在完成状态时使用 `RetryOnConflict`，并返回是否真正写入，以防已标记为失败后仍误报成功。  
4. 常量命名整理、引入 `k8s.io/client-go/util/retry`、改进 `restoreContainerResolveAttemptContext` 超时计算。  
5. 单元测试覆盖：当任意普通容器（包括 helper）失败时，checkpoint job 必须被标记为 Failed，且不会启动快照工作流。  

**🎯 影响范围**  
- `deploy/snapshot/internal/controller/controller.go`（NodeController 核心逻辑）  
- `deploy/snapshot/internal/controller/controller_test.go`（新增/修改测试）  
- 依赖的 K8s 客户端交互：Job 注解更新、Event 创建、Pod 状态查询  
- 运行时调用 `snapshotruntime.SendSignalToPID` 与 `ResolveContainer`  

**🔍 技术洞察**  

- **架构影响**  
  - **可靠性提升**：原先仅在 checkpoint 容器本身失败时才标记 job 为 Failed，导致 helper 容器异常会留下 “Completed” 或 “Running” 状态的僵尸 job。现在统一由 NodeController 负责检测所有容器的终止状态，确保 checkpoint 过程在任何辅助容器异常时都能快速回滚。  
  - **职责划分更清晰**：`setCheckpointStatus` 只负责持久化终态，完成状态走冲突重试路径，避免与已标记为 Failed 的 Job 产生写冲突。  

- **性能影响**  
  - 额外的容器状态遍历与一次 `ResolveContainer` 调用（仅在检测到错误时），对整体 checkpoint 流程的 CPU/IO 开销几乎可以忽略。  
  - 使用 `retry.RetryOnConflict` 在极端高并发场景下可能产生数次 API 重试，但这种情况仅在 checkpoint 已经成功并尝试写完成状态时出现，属于可接受的突发成本。  

- **安全考虑**  
  - 通过 `snapshotruntime.SendSignalToPID` 向仍在运行的 checkpoint 进程发送 `SIGKILL`，需要确保信号只针对当前 checkpoint 容器的 PID，防止误杀其它容器进程。代码已在失败路径中遍历 **仅** `Running` 状态且 `ContainerID` 不为空的容器，风险已被最小化。  
  - 对 Job 注解的写入使用了 RBAC 授权的 `batch/v1` 接口，不引入新的权限需求。  

**⚠️ 潜在风险**  
1. **并发写冲突**：在极端情况下，多个控制器实例可能同时尝试将同一 Job 标记为 Completed/Failed，虽然已加入冲突重试，但仍可能导致短暂的写入失败，需要监控 `RetryOnConflict` 错误率。  
2. **误杀风险**：如果 pod 中出现多个运行中的容器且其中一个是 checkpoint 容器，循环中会对所有 `Running` 容器发送 `SIGKILL`，若误将非 checkpoint 容器标记为 “running checkpoint” 可能导致业务容器被意外终止。确保容器命名约定（checkpoint 主容器唯一）或在信号发送前增加容器名称校验可进一步降低风险。  
3. **事件风暴**：对于批量启动的 checkpoint job，若大量 helper 容器快速失败，可能在短时间内产生大量 `CheckpointFailed` 事件，需评估监控系统的事件吞吐能力。  

**💡 关注建议**  
- **监控**：新增监控指标 `snapshot_checkpoint_job_failure_total{reason="helper_error"}`，及时发现异常增长。  
- **RBAC审计**：确认运行 snapshot‑controller 的 ServiceAccount 拥有 `batch/jobs` 的 `update` 权限以及 `events` 的 `create` 权限。  
- **回归测试**：在 CI 中加入高并发模拟（多 NodeController）以验证 `RetryOnConflict` 的稳定性。  
- **文档**：更新 checkpoint‑job 状态说明，明确 “Failed” 可能由任意容器异常触发，并说明 `SIGKILL` 的作用范围。  
- **防误杀**：如果业务场景中 pod 包含多个长期运行的容器，考虑在发送信号前加入容器标签或名称前缀过滤，确保只对 checkpoint 容器执行杀死操作。

### feat(operator): support user-declared GMS checkpoint clients (#9641)
**SHA**: `da85b01` | 🔗 [查看提交](https://github.com/ai-dynamo/dynamo/commit/da85b01152edd1e6b6aee7a2d355c2530f7060f4)

**🎯 变更类型**：功能增强 / 架构变更  
**⚡ 重要程度**：🔴高  

**📋 变更摘要**  
- 在 **GPU Memory Service (GMS)** 配置中新增 `extraClientContainers`、`extraClientPods`（预留）字段，允许用户自行声明需要接入 GMS 的容器。  
- 为 **Checkpoint** 增加 `targetContainerName` 与 `job` 子结构（可自定义 GMS client容器与 PodTemplate），并在 Operator 中相应地处理这些新属性。  
- 完善 CRD 深拷贝、转换、默认值、验证（名称合法性、模式约束）以及 webhook 逻辑，确保向后兼容。  
- 将 GMS 客户端注入抽象为 **`gms.EnsureClient`**，取代旧的 side‑car‑only 实现；在组件生成 PodSpec 时自动为声明的容器添加共享 socket、DRA Claim 与环境变量。  
- 引入 **GMS checkpoint 迁移路径**：在恢复阶段不再硬编码 loader/saver 容器，而是依据 `targetContainerName` 与 `extraClientContainers` 动态决定。  
- 为 **Snapshot Restore** 添加三个可选的传输后端：  
  - `nixl`（POSIX‑direct‑IO + pinned‑host staging）  
  - `nixl-gds`（直接 FILE → VRAM 的 GDS‑MT）  
  - `sharded-ssd`（在本地 SSD 根目录上并行读取）  
- 新增大量内部实现（NIXL API 包装、资源管理、分片调度、错误处理），并废除已不再使用的 `gms.loader`/`gms.saver` side‑car 实现。

### 🔍 技术洞察

| 维度 | 影响说明 |
|------|----------|
| **架构** | 1. **CRD 扩展**：`GPUMemoryServiceSpec` 与 `ServiceCheckpointConfig` 变得可组合（extra client containers/pods、target container）。<br>2. **Operator 代码分层**：<br>   - 新增 `internal/gms` 包负责统一的 client 注入逻辑，提升复用度。<br>   - `internal/dra.ApplyClaim` 仍然负责 DRA Claim，但现在会在每个 GMS 客户端容器上同步创建 claim。<br>3. **Snapshot Restore 流程**：从 “固定 loader/saver side‑car” 迁移到 “按需求动态注入”，并通过统一的 **TransferBackend** 接口抽象磁盘 → GPU 的搬迁路径。<br>4. **后端插件化**：通过 `TransferBackend` 接口实现多种高性能搬迁方式，可在运行时通过 `--transfer-backend` 切换。 |
| **性能** | - **GMS 客户端注入**：额外容器共享同一 `SharedVolume` 与 DRA claim，几乎零额外网络/PCIe 开销，保持原有 intra‑pod 性能。<br>- **Snapshot Restore**：<br>  • `nixl` 使用 **POSIX O_DIRECT** + **pinned host staging**，在 NFS / NAS 等远程文件系统上实现 I/O 深度（`_IO_DEPTH=16`）和 CPU‑GPU 并行拷贝，理论吞吐提升 1.5‑3×。<br>  • `nixl-gds` 直接使用 **GDS_MT**（GPUDirect‑Storage）省去主机 staging，适用于支持 GDS 的平台，可接近磁盘带宽上限。<br>  • `sharded-ssd` 通过本地 SSD 多根并行读取，显著降低单根磁盘瓶颈，适用于大模型快照（>10 GiB/设备）。<br>- **资源占用**：每个恢复工作线程持有 **pinned host buffers**（默认 64 MiB）x2；在高并发（`max_workers`≈16）下约占用 2 GiB 主机内存，需监控节点可用内存。 |
| **安全** | 1. **Webhook 验证**：新增 `validateCheckpointConfig`、`validateGMSClientContainerNames`，对容器名、`targetContainerName`、`checkpoint.job.gmsClientContainers` 进行 DNS‑1123 检查，防止非法字符导致注入异常。<br>2. **GMS socket 共享**：`gms.EnsureClient` 会覆盖已存在的 `SharedVolume`、`Env`、`ResourceClaims`，但仅在同一 Pod 内部共享，外部容器仍受 Pod‑Security‑Policy / PSP 限制。<br>3. **额外容器的权限**：若用户自行声明容器为 GMS 客户端，必须自行确保该容器拥有适当的安全上下文（例如 `runAsUser`、`allowPrivilegeEscalation`）——Operator 不再自动注入安全上下文，防止潜在的 **privilege escalation**。<br>4. **文件 I/O**：`nixl`、`nixl-gds`、`sharded-ssd` 均使用 **O_DIRECT**（或回退警告），避免缓冲区污染；但若底层文件系统不支持 O_DIRECT，恢复会回退到普通读并记录警告。 |
| **可维护性** | - 代码组织更清晰：GMS 客户端注入抽象、深拷贝、转换、验证分离。<br>- 新增 `TransferBackend` 接口，后续可无侵入式加入其他后端（例如 RDMA‑direct）。<br>- 旧的 `gms.loader`/`gms.saver` 侧车已被废弃，降低死代码维护成本。<br>- 增加的 `extraClientPods` 仍标记为 “reserved”，保持向后兼容但不影响当前运行。 |
| **兼容性** | - 现有 CRDs 在不填充新字段时仍保持行为不变（默认 `enabled=false`，`extraClientContainers` 空）。<br>- 由于 `extraClientContainers` 只能在 `mode=intraPod` 使用，旧的 `interPod` 部署不会受到影响。<br>- 通过 `env` 注入和 `volumeMount` 覆盖，已存在的 `main` 容器行为保持一致。<br>- `checkpoint.job` 结构的新增字段在 v1alpha1 中是可选的，现有用户的 Checkpoint CR 不受破坏。 |
| **风险点** | - **资源泄漏**：`DeviceToFileWriter` 与 `PinnedCopySlot` 在异常路径下的释放依赖 `finally`，若新代码路径未覆盖可能导致文件句柄或 CUDA 句柄泄漏。<br>- **并发错误**：恢复时多线程并发访问同一 `GMS` 句柄（如 `nixl-gds`）需要正确的同步；现实现使用单线程 per‑root 并发，仍有潜在的 **RaceCondition**。<br>- **平台依赖**：`nixl-gds` 依赖 NVIDIA **GDS**，非 GDS 机器会在运行时报错；需要在文档中明确要求。<br>- **O_DIRECT 不可用**：在某些文件系统（e.g., ext4 on older kernels）上 `O_DIRECT` 会返回 `EINVAL`，代码会回退并记录警告，但恢复性能会大幅下降。<br>- **升级路径**：CRD 版本
