### fix: respect per-request chat_template_kwargs for DeepSeek V3.2 thinking mode (#7286)
**SHA**: `745d7e4` | 🔗 [查看提交](https://github.com/ai-dynamo/dynamo/commit/745d7e4ca240c54f30c067248c0c5f2fc5a38760)

**🎯 变更类型**：Bug修复  
**⚡ 重要程度**：🔴高  
**📋 变更摘要**：  
- 为 DeepSeek V3.2（deepseek_r1）模型在 **OpenAIPreprocessor** 中加入对 `chat_template_args` 的解析，支持通过 `"thinking"` 布尔或 `"thinking_mode"` 字符串来关闭/切换推理模式。  
- 在 DeepSeek V3.2 的 **formatter** 中实现 `resolve_thinking_mode`，使其能够在每一次请求层面按上述参数覆盖默认的 `ThinkingMode`（Chat / Thinking）。  
- 完善单元测试，验证不同参数组合（`thinking`, `thinking_mode`, 空参数）对模型推理模式的影响。

**🎯 影响范围**：  
- `lib/llm/src/preprocessor.rs`（`is_reasoning_disabled_by_request` 扩展）  
- `lib/llm/src/preprocessor/prompt/deepseek_v32.rs`（思考模式解析与渲染）  
- 相关单元测试文件（`mod tests`）  
- 任何使用 `OpenAIPreprocessor` 与 DeepSeek V3.2 formatter 的调用方（如上层 API、推理服务）。

**🔍 技术洞察**  

- **架构影响**  
  - 引入 **per‑request** 参数解析，提升了输入层的灵活性，且实现仅在两处（预处理器与 formatter）进行，保持了现有模块边界。  
  - 通过 `chat_template_args` 向下传递，仅在 DeepSeek V3.2 特化实现中使用，不会影响其他模型的预处理逻辑。  
  - 新增的 `resolve_thinking_mode` 方法遵循“先布尔后字符串”的优先级，保持与 Kimi 等模型的行为一致，降低了跨模型配置差异带来的认知成本。

- **性能影响**  
  - 每次渲染会额外做一次 `HashMap` 查找（最多两次），对整体吞吐量的影响在毫秒级以下，可忽略。  
  - 编译期未引入额外分配，运行时仅读取已有的 `chat_template_args`，不产生显著内存负担。

- **安全考虑**  
  - 解析仅依赖 `serde_json::Value::as_bool` 与 `as_str`，对不符合预期的类型返回 `None`，不会触发 panic。  
  - 参数来源仍是用户请求的 JSON，未进行额外的验证或执行代码，属于安全中性改动。  
  - 建议在上层 API 对 `chat_template_args` 做 schema 校验，以免误传导致思考模式异常（虽不直接导致安全风险，但影响业务正确性）。

**⚠️ 潜在风险**  
1. **兼容性**：旧版 DeepSeek V3.2 客户端若仍使用 `"thinking_mode"` 为 `null` 或非 `"chat"/"thinking"`，会回退到默认 `thinking_mode`，可能与已有预期不符。  
2. **误配置**：如果用户误把 `"thinking"` 设为字符串或其他非布尔值，当前实现会忽略该键，导致默认模式生效，潜在的“配置失效”风险。  
3. **测试覆盖不足**：虽然已加入丰富的单元测试，但未覆盖 `thinking` 与 `thinking_mode` 同时出现且值冲突的更多边界（如 `"thinking": true` 与 `"thinking_mode": "chat"`），需确保业务文档明确优先级。  
4. **代码重复**：相同的参数解析逻辑在 `preprocessor.rs` 与 `deepseek_v32.rs` 中出现，若后续新增模型可能出现维护分歧。

**💡 关注建议**  
- **文档更新**：在模型说明或 API 文档中明确 `chat_template_args` 支持的键、取值以及优先级规则。  
- **统一实现**：考虑抽象出通用的 “思考模式解析器” 工具函数，供不同模型共享，降低重复代码风险。  
- **输入校验**：在请求入口层（如 OpenAI 兼容层）添加 schema 校验，确保 `thinking` 为布尔、`thinking_mode` 为枚举，提前捕获错误配置。  
- **回归测试**：在 CI 中加入对混合键冲突的组合测试，验证“布尔优先于字符串”逻辑始终如预期。  
- **监控指标**：如果业务对思考/聊天模式敏感，可在日志或监控中记录每次渲染使用的 `ThinkingMode`，帮助快速定位因参数错误导致的异常行为。

### feat: replaces PersistentConnector monkey-patch with proper nixl_conn… (#6913)
**SHA**: `387100c` | 🔗 [查看提交](https://github.com/ai-dynamo/dynamo/commit/387100c8d5126c9a744fe33023a3c126eba71614)

**🎯 变更类型**：功能增强  
**⚡ 重要程度**：🔴高  
**📋 变更摘要**：  
- 删除了原有 `PersistentConnector` 的 monkey‑patch 实现，改为在 `nixl_connect` 包中直接提供 “共享单例 Connection” 与 “Remote‑agent 引用计数” 机制。  
- `EmbeddingTransfer` 的发送/接收类改用新的 `nixl_connect.Connector`，并在 `Connection`、`Remote` 中加入线程/异步锁及引用计数逻辑，以保证跨多次 send/receive 时保持同一 NIXL agent 并安全地释放资源。  

**🎯 影响范围**：  
- `components/src/dynamo/common/multimodal/embedding_transfer.py`（所有使用 NIXL READ 传输的发送/接收路径）  
- `lib/bindings/python/src/dynamo/nixl_connect/__init__.py`（Connector、Connection、Remote 实现）  
- 可能的其他使用 `nixl_connect.Connector` 的模块（如未来的 KV‑cache、tensor 传输等）  

**🔍 技术洞察**  

- **架构影响**  
  - 由“每次操作新建/销毁 Connection + monkey‑patch Remote._release”迁移为 **真正的共享单例 Connection**，统一由 `Connector._create_connection` 管理。  
  - 引入 **引用计数 (ref‑count) 机制** (`Connection.acquire_remote_ref / release_remote_ref`) 来决定何时真正调用 NIXL 的 `remove_remote_agent`，消除之前因每次 `release` 都 deregister 导致的 inflight 操作被强制终止的问题。  
  - 新增 **线程锁 (`threading.Lock`)** 与 **异步锁 (`asyncio.Lock`)**，确保在多线程/协程环境下对 `_remote_refs` 与 `_shared_connection` 的并发安全。  

- **性能影响**  
  - **显著降低连接建立开销**：原实现每次 send 前都会创建 `Connection` 并进行 agent 注册，成本在数百毫秒；改为首次创建后后续复用，预计整体延迟下降 30%‑50%。  
  - **资源使用更平稳**：避免频繁的 `remove_remote_agent`/`register_remote_agent`，降低 NIXL 内部缓存碎片与系统调用次数。  
  - **潜在微小锁竞争**：在高并发场景下，`_remote_refs_lock`、`_shared_connection_lock` 可能产生短暂的排队，但锁持有时间极短（仅计数或检查单例），对整体吞吐影响可忽略。  

- **安全考虑**  
  - 引入显式的 **引用计数** 防止提前 deregister，提升远程 agent 生命周期的可预测性，降低因资源提前释放导致的异常或崩溃。  
  - 仍然依赖底层 NIXL API 的安全性，未暴露新的外部接口；但需要确保 **正确调用 `Remote.release()`**（即 `descriptor._release()`），否则可能出现 “引用泄漏” 导致 agent 永久残留。  

**⚠️ 潜在风险**  

1. **引用计数不匹配**  
   - 如果某些代码路径因异常或提前退出未调用 `_release`，会导致 `_remote_refs` 中残留计数，进而 **永不 deregister**，可能造成资源泄漏。  
2. **并发死锁 / 锁竞争**  
   - 虽然锁粒度小，但在极端高并发（数千并发协程）下，`_shared_connection_lock` 可能成为热点，需通过压测验证。  
3. **向后兼容性**  
   - 任何仍旧直接使用旧的 `PersistentConnector`（或自行 monkey‑patch `Remote._release`）的外部插件将失效，需要更新文档或提供兼容层。  
4. **初始化顺序**  
   - `Connector._create_connection` 现在是 **惰性单例**，如果在某些情形下需要提前初始化（例如在子进程 fork 前），需要额外注意。  

**💡 关注建议**  

- **测试**  
  - 编写并运行 **并发压力测试**：在同一进程/多进程下并发多次 `send_embeddings` / `receive_embeddings`，验证连接仅创建一次、Remote.agent 正确 deregister。  
  - 添加 **异常路径测试**：异常抛出后确保 `Remote._release` 仍被调用，避免引用泄漏。  

- **监控**  
  - 在运行时通过日志或监控指标（如 `connection_created_count`、`remote_refs_total`）观察单例创建次数与引用计数变化，确保预期行为。  

- **文档与迁移**  
  - 更新开发者文档，说明 **不再需要** `PersistentConnector` 与手动 monkey‑patch，统一使用 `nixl_connect.Connector()`。  
  - 对外提供 **兼容 shim**（可选），在 `PersistentConnector` 中直接继承新 `Connector`，防止第三方插件瞬间失效。  

- **代码审查点**  
  - 确认所有 `Descriptor`/`Remote` 对象的 `__del__` 或上下文管理器路径最终会调用 `_release`。  
  - 检查是否有仍然保留的 `nixl_connect.Remote._release = ...` 赋值，防止旧的 monkey‑patch 残留。  

通过以上改动，系统在 **保持功能完整** 的同时，显著提升 **连接复用效率** 与 **资源管理安全性**，但仍需关注并发和异常路径下的引用计数一致性。

### feat: follow up of optinal, batching-of-client-side-events (#6741)
**SHA**: `fca0a80` | 🔗 [查看提交](https://github.com/ai-dynamo/dynamo/commit/fca0a801dc8f0b91343dd702125de31290643f4f)

**🎯 变更类型**：功能增强 / 架构变更 / 性能优化  
**⚡ 重要程度**：🔴高  

**📋 变更摘要**  
1. 为 KV 事件发布器引入 **可选的批处理窗口（毫秒）**，默认关闭批处理，支持 `None/0` 立即发送，`50` 表示 50 ms 批处理窗口，最大限制为 15 s 并在超出时自动截断。  
2. 新增 **KV Publisher 监控指标**，统计引擎在进入 Publisher 前因事件 ID 间隙而丢失的原始事件数量，并通过 `MetricsHierarchy` 自动挂载层级标签。  
3. 重构批处理状态 `BatchingState`，加入 **last_flush_time**、**pending_block_count**、**record_flush_time** 等辅助方法，实现 **空闲期间的立即刷新**（stale‑timer）以及 **批大小上限**（默认 128 块）。  
4. 同步更新 Python 绑定签名、文档注释以及大量单元测试，以覆盖新行为。  

**🎯 影响范围**  
- `lib/llm/src/kv_router/publisher.rs`（核心 KV 事件发布逻辑）  
- `lib/bindings/python/rust/llm/kv.rs`（Python 接口签名）  
- `lib/runtime/src/metrics/prometheus_names.rs`（Prometheus 指标名称）  
- 相关单元测试模块（`event_processor_tests`、`batching_state_tests`）  

**🔍 技术洞察**  

- **架构影响**  
  - 新增 **`batching_timeout_ms: Option<u64>`** 参数，使 KV Publisher 的批处理可在运行时由用户显式开启/关闭，兼容 `None`（立即发送）与 `Some(ms)`（启用批处理）。  
  - 通过 `KvPublisherMetrics::from_component` 将指标注册到 **`MetricsHierarchy`**，实现统一的命名空间与层级标签（`dynamo_namespace`、`dynamo_component`），提升可观测性。  
  - 引入 `MAX_BATCHING_TIMEOUT_MS` 与 `DEFAULT_MAX_BATCH_BLOCKS`，在 `run_event_processor_loop` 中统一控制窗口大小与批块上限，防止误配置导致内存激增或延迟失控。  
  - `BatchingState` 由原来的 **`batch_start_time`** 改为 **`last_flush_time`**，并在每次 flush 后记录，以实现 **“空闲后立即 flush”** 的低延迟特性。  

- **性能影响**  
  - **正向**：启用批处理后，多个 `Removed/Stored` 事件会合并为一次网络发送，显著降低 ZMQ / Jetstream 的带宽占用与系统调用次数，提升吞吐。  
  - **负向**：在默认 **关闭批处理**（`None`）的情况下行为保持不变；当用户打开大窗口（接近 15 s）且事件产生速率低时，会增加单批次的内存占用（最多 128 blocks），但已有上限防止 OOM。  
  - 新的 **stale‑timer**（空闲检测）在长时间无事件后立即 flush，避免因超时等待导致的延迟累计，对低频关键事件的响应时间有明显改进。  

- **安全考虑**  
  - 变更未引入外部输入的直接执行路径，安全风险极低。唯一需要关注的是 **Prometheus 指标的暴露**：如果指标注册失败，系统会回退为未注册的本地计数器，仍然安全，只是监控缺失。  
  - 参数上限（15 s）防止恶意或误配置导致的资源耗尽（长时间缓存大量事件）。  

**⚠️ 潜在风险**

2. **队列溢出导致度量丢失**  
   - 当发布速度跟不上生成速度（极端高 QPS）时，`queue.Full` 会 silently drop指标，监控端可能观察到不完整的数据。  
3. **后台线程异常**  
   - ZMQ 发送异常（如网络中断）只记录 warning，未上报给调度层，可能在长时间故障后导致度量全失。  
4. **依赖升级冲突**  
   - `msgspec >=0.19.0` 与已有的 `msgpack==1.1.2` 没有直接冲突，但如果项目中其他子模块锁定了更旧的 `msgspec`，会出现 pip 解析错误。  
5. **安全面**  
   - 未限制订阅者的来源，任何能访问该端口的机器都能获取调度内部细节（如请求数量、KV‑cache 大小），在多租户环境下可能泄露隐私信息。  

---

**💡 关注建议**  

1. **部署安全**  
   - 在生产环境强制将 ZMQ 绑定到 `127.0.0.1` 或使用 Unix domain socket（`ipc://`），或在防火墙层面仅允许可信 IP。  
2. **监控与告警**  
   - 为 `InstrumentedScheduler` 添加内部计数（如 `publish_failed`, `queue_dropped`）并通过现有 `StatLogger` 上报，以便在度量发送异常时及时告警。  
3. **可配置化**  
   - 暴露 `FPM_PUBLISHER_QUEUE_SIZE`、`FPM_HEARTBEAT_INTERVAL` 等参数，方便在高负载下调优。  
4. **兼容性验证**  
   - 在 CI 中加入跨版本测试：当 `DYN_VLLM_FORWARDPASS_METRIC_PORT` 未设置或已被用户自定义 `scheduler_cls` 时，确保原始调度行为不受影响。  
5. **文档与示例**  
   - 在项目 README 增加 “Forward Pass Metrics” 使用指南，示例代码演示如何使用 ZMQ SUB 订阅并解码 `ForwardPassMetrics`。  
6. **资源清理**  
   - 确认在 `Engine` 关闭流程中调用 `InstrumentedScheduler.shutdown()`，避免 ZMQ socket 泄漏。  

通过上述建议，可在保持低开销可观测性的同时，降低部署风险并提升系统的可维护性与安全性。

| 关键点 | 风险/注意事项 | 建议 |
|--------|--------------|------|
| **TARGETARCH 与旧 ARG 的兼容** | 部分旧的外部构建脚本或 CI 仍会显式传 `--build-arg ARCH=…`，会被忽略或导致冲突。 | 在文档和 CI 模板中删除 `ARCH`/`ARCH_ALT`，统一使用 `--platform`。如必须保留向后兼容，考虑在 Dockerfile 开头增加 `ARG ARCH=; ARG ARCH_ALT=` 但不使用。 |

| **LD_LIBRARY_PATH 与 ld.so.conf** | 多架构时在 `trtllm_runtime.Dockerfile` 同时写入两套 TensorRT 路径，可能在只构建单架构时产生不存在的路径。 | 通过 `TARGETARCH` 条件化写入，仅保留实际存在的路径，保持镜像体积和 ldconfig 警告最小化。 |

**总体评估**  
此次改动成功将 Docker 构建从手动 `ARCH/ARCH_ALT` 参数迁移到 BuildKit 原生的 `TARGETARCH`，实现了官方推荐的多架构支持。核心逻辑已在 `render.py` 中统一，后端模板也全部适配。若 CI、文档和外部脚本同步更新，兼容性风险可控。建议在 CI 中加入一次 “multi‑arch build (amd64+arm64) → `docker run --platform …` 测试”，确保所有路径、库文件和 sccache 缓存均按预期工作。
