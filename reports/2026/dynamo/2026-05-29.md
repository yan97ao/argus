### feat: add combined parity table UI (#10113)
**SHA**: `cfdb49f` | 🔗 [查看提交](https://github.com/ai-dynamo/dynamo/commit/cfdb49f2555d692b81753f9bda19162cc16c0a3b)

**🎯 变更类型**：功能增强  
**⚡ 重要程度**：🔴高  

**📋 变更摘要**  
本次提交在 `tests/parity` 目录下新增了「Combined Parity Table」生成能力，能够一次性渲染 Tool‑Calling 与 Reasoning 两大解析器的对齐（Parity）表格，并通过前端标签页（tab）在同一 HTML 页面中切换展示。为实现该功能，新增了 CLI 参数 `all --html`、一套 HTML 组装逻辑以及相应的 Jinja2 模板改造，同时对测试用例、文档注释及正则匹配做了统一命名更新。

**🎯 影响范围**  
- `tests/parity/generate_parity_table.py`（核心生成脚本）  
- `tests/parity/parity_table.html.j2`（模板）  
- `tests/parity/toolcalling/table.py`、`tests/parity/reasoning/table.py`（内部加载逻辑）  
- `lib/parsers/src/tool_calling/parsers.rs` & `lib/llm/tests/test_streaming_tool_parsers.rs`（测试文档命名统一）  
- `tests/parity/README.md`（示例流程）  
- 新增/扩展的单元测试 (`tests/parity/toolcalling/test_generate_parity_table.py`)  

**🔍 技术洞察**  

- **架构影响**  
  - 通过在生成脚本中统一调度两套 Parity 表格的加载、路径重写与 Jinja2 渲染，实现了 **“组合页面”** 的新抽象层，保持原有单独页面生成逻辑不受破坏。  
  - 引入了 `render_combined_html()` 与 `_combined_*_panels()` 辅助函数，把原本分散的渲染步骤聚合，提升了代码复用度。  
  - 前端使用 **tab‑button / tab‑panel** 机制，并在 JavaScript 中加入 URL 查询参数同步（`?tab=...`）的逻辑，改善了使用者在浏览器刷新后仍能保持选中标签页的体验。  

- **性能影响**  
  - 生成过程仅在 CI / 本地开发阶段运行，额外读取两套 fixture、两套 Markdown 与两套 Jinja 模板，预计 **CPU、内存开销 < 200 ms**（基于现有的单表生成时间 ≈ 80 ms）。  
  - 对运行时二进制（库）没有影响；仅在测试/工具链中增加一次完整渲染的 IO，属于可接受的开发成本。  

- **安全考虑**  
  - HTML 输出使用 `html.escape` 对动态属性（panel id、label、title）进行转义，避免 XSS。  
  - 读取的唯一外部输入为 **本仓库内部文件**，不存在未受信任数据注入风险。  
  - 引入 `zoneinfo` 读取特定时区时间戳；在不支持 `zoneinfo`（如旧版 Windows）时会抛异常，建议在 CI 环境明确 Python≥3.9 并安装 `tzdata`。  

**⚠️ 潜在风险**  
1. **兼容性**：`zoneinfo` 在某些平台（Windows 7/8、Python 3.8 及以下）不可用，可能导致脚本在旧环境崩溃。  
2. **CLI 破坏**：新增 `all` 选项后，若未来继续扩展 `stage` 参数，需保持向后兼容；当前实现将 `all` 与 `--html` 绑定，误用 `all` 而未提供 `--html` 会触发错误信息。  
3. **路径重写逻辑**：`_rewrite_panel_paths` 通过硬编码的字符串替换实现链接修正，若未来目录结构变动（如 `lib/` 移动），可能产生失效链接。  
4. **测试依赖**：新测试对 HTML 结构（按钮 `data-tab-target`、ID 命名）做了大量硬编码断言，模板细微改动（如 class 名称）会导致测试失效，需要同步维护。  

**💡 关注建议**  
- **平台依赖**：在 CI 与文档中明确声明 Python≥3.9 并确保 `tzdata` 包已装，以防 `zoneinfo` 不可用。  
- **模块化**：考虑将路径重写抽象为统一的 **URL Builder**，以便在目录迁移时仅修改一次实现。  
- **测试维护**：将 UI 断言抽象为“关键属性”集合（如 `data-tab-target`、`aria-label`），避免因微调样式导致频繁失效。  
- **文档同步**：在 `README.md` 与项目官网（若有）更新示例，说明如何使用 `python3 tests/parity/generate_parity_table.py all --html` 生成统一表格。  
- **未来扩展**：如果后续再加入更多解析器（如 “Embedding”），可直接在 `render_combined_html` 中追加对应 `_combined_*_panels()`，保持当前插件化结构不变。

### fix(omni): handle StageRuntimeInfo object in get_stage_metadata (#9875)
**SHA**: `d8bc6c6` | 🔗 [查看提交](https://github.com/ai-dynamo/dynamo/commit/d8bc6c63434bef2eddc23d0930cc7112f356f4ab)

**🎯 变更类型**：Bug修复  
**⚡ 重要程度**：🔴高  
**📋 变更摘要**：  
- 解决了 `engine_client.engine.get_stage_metadata` 返回值类型由字典改为 `StageRuntimeInfo` 对象后导致的属性读取异常。  
- 将原先的 `dict.get` 调用替换为 `getattr`，并在测试中使用 `SimpleNamespace` 模拟新对象。  
- 为 Omni 运行时在多硬件（GPU/XPU）环境下的元数据获取提供兼容性，确保后续功能不受返回类型更改影响。

**🎯 影响范围**：  
- `components/src/dynamo/vllm/omni/omni_handler.py`（采样参数构建逻辑）  
- 单元测试 `components/src/dynamo/vllm/tests/omni/test_omni_handler.py`  
- 集成测试 `tests/serve/test_vllm_omni.py`（新增 XPU 标记）  

**🔍 技术洞察**  
- **架构影响**：  
  - 该修改位于 **Omni Handler** 与底层 **Engine** 之间的适配层，提升了代码对 Engine 元数据返回类型的容错性。  
  - 通过使用 `getattr`，降低了对具体返回结构的耦合，使得后续若引入更丰富的 `StageRuntimeInfo`（如新增字段）时，已有逻辑仍能平滑工作。  
- **性能影响**：  
  - `getattr` 与 `dict.get` 的成本相当，且仅在构建采样参数的循环中执行，整体性能影响可以忽略不计。  
  - 测试标记加入 XPU 运行路径，可能在 CI 中触发更多硬件配置的测试，但对运行时性能无直接影响。  
- **安全考虑**：  
  - 读取属性改为 `getattr(metadata, "stage_type", "llm")`，避免因属性不存在抛异常导致服务崩溃，提高了容错性。  
  - 未引入外部输入的直接处理，安全风险极低。  

**⚠️ 潜在风险**  
1. **类型假设破坏**：如果未来 `get_stage_metadata` 返回 `None` 或非对象（例如自定义实现的映射），`getattr` 将返回默认值，可能隐藏真正的配置错误。  
2. **测试覆盖不足**：目前仅用 `SimpleNamespace` 模拟单一属性，未覆盖 `StageRuntimeInfo` 可能包含的方法或属性访问失败的情形。  
3. **兼容旧实现**：旧版 Engine 仍返回字典，`getattr` 对字典仍返回默认值（因为字典没有属性），会导致隐式把 `stage_type` 视为 `"llm"`，若旧版系统依赖 `"diffusion"` 类型会出现功能退化。  

**💡 关注建议**  
- **兼容性验证**：在 CI 中加入对老版本 Engine（返回 dict）和新版本（返回对象）的双向兼容性测试，确保两种返回结构均能正确解析。  
- **防御性编程**：可在获取 metadata 前加入类型检查或日志告警，例如：

  ```python
  metadata = self.engine_client.engine.get_stage_metadata(i)
  if isinstance(metadata, dict):
      stage_type = metadata.get("stage_type", "llm")
  else:
      stage_type = getattr(metadata, "stage_type", "llm")
  ```

  这样既保留向后兼容，又能在日志中捕获异常返回类型。  
- **文档同步**：更新 `Engine` 接口文档，明确 `get_stage_metadata` 的返回类型已从 `Dict` 更改为 `StageRuntimeInfo`，并说明推荐的获取方式。  
- **监控与告警**：在生产环境中加入对 `stage_type` 为默认值 `"llm"` 的异常比例监控，及时发现因元数据获取异常导致的行为偏差。  

通过上述措施，可确保此次修复在提升稳定性的同时，保持向后兼容并降低潜在风险。

### ⚠️ 潜在风险

1. **配置错误导致服务不可达**  
   - `tlsMode=MUTUAL` 必须配对 `clientCertificate` 与 `privateKey`，若路径错误或文件不可读，Istio 会拒绝 DestinationRule，导致 EPP 相关服务的流量全部中断。  
2. **隐式默认行为变化**  
   - 之前 `insecureSkipVerify` 默认 `true`，现在模板会显式渲染用户提供的值，若用户未显式设置而期望保留旧的默认 `true`，仍会得到 `true`（因为模板保留默认），但在 Helm `--set` 覆盖时可能出现意外的 `false`。需要确认 CI/CD 流水线是否有相应的值覆盖。  
3. **NATS 注解默认关闭 Sidecar**  
   - 在启用了 Istio 且希望对 NATS 进行流量监控或安全拦截的场景，默认注解会导致 sidecar 不注入，可能破坏该场景的安全策略。用户需手动覆盖，若未意识到会产生安全盲点。  
4. **向后兼容性**  
   - 旧版 CR 没有 `istio` 块但 `provider=istio` 时，新增的校验会报错（`istio configuration is required`），这在升级路径上可能导致集群启动失败。需要在升级指南中说明此变化并提供迁移示例。
