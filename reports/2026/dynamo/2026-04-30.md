### feat(kv-router): carry tiered match data over the remote indexer wire (#8632)
**SHA**: `883a07e` | 🔗 [查看提交](https://github.com/ai-dynamo/dynamo/commit/883a07e83210c633889f11936531f90586723a9e)

**🎯 变更类型**：功能增强 / 架构变更  
**⚡ 重要程度**：🔴高  

**📋 变更摘要**  
1. 为 KV‑Router 的索引查询请求 `IndexerQueryRequest` 新增 `device_only` 布尔字段，支持仅返回设备层（Device tier）的匹配分数，省去低层遍历与传输开销。  
2. 引入一套 **WireTieredMatchDetails**、**WireLowerTierMatchDetails** 与 **WireOverlapScores** 的序列化结构，统一在 JSON RPC 中传递 **分层匹配结果**（device + 各 lower tier 命中）。  
3. 调整 `DedupCacheStatusTracker` 在去重模式下强制使用 `StorageTier::Device` 打标签，保证 STORE 与对应的 REMOVE 在相同 tier 的 indexer 里配对，防止 orphaning。  
4. RemoteIndexer 与服务端查询引擎改为通过 `find_matches_by_tier`（返回 `TieredMatchDetails`）统一处理，且在 `device_only=true` 时可直接走 `find_match_details` 跳过 lower‑tier 计算。  
5. 完善大量单元/集成测试，覆盖 dedup‑mode tier 正规化、共享 hash‑mapping 的多源删除、以及远程查询的 tiered 与 device‑only 两种路径。

---

**🎯 影响范围**  
- `lib/kv-router/src/indexer/types.rs`（请求/响应序列化结构）  
- `lib/llm/src/block_manager/kv_consolidator/tracker.rs`（去重缓存追踪逻辑）  
- `lib/llm/src/kv_router/indexer/mod.rs`（内部 `TieredMatchDetails` ↔ `WireTieredMatchDetails` 转换、Remote 查询入口）  
- `lib/llm/src/kv_router/indexer/remote.rs`（远程查询 RPC）  
- 相关测试模块（`tracker.rs`、`mod.rs`、`remote.rs`）  

---

**🔍 技术洞察**

| 维度 | 影响 | 说明 |
|------|------|------|
| **架构影响** | ★☆☆☆☆ | - 新增的 wire 层结构把 **分层匹配结果** 纳入统一 RPC，保持后端 `TieredMatchDetails` 与前端 JSON 之间的显式映射。<br/>- `device_only` 为向后兼容的可选字段，使用 `#[serde(default)]`，老客户端仍会得到完整 tiered 结果。<br/>- `DedupCacheStatusTracker` 现在统一使用 `Device` tier，强化了去重模式下的**索引一致性**，防止因不同 tier 路由导致的 orphan block。 |
| **性能影响** | ★★☆☆☆ | - **正向收益**：`device_only=true` 可显著削减服务器端 lower‑tier 遍历 CPU 与网络带宽（尤其在大模型、长序列时）。<br/>- **负向成本**：引入 `WireTieredMatchDetails` 与 `WireLowerTierMatchDetails` 增加了序列化/反序列化的字段数目和大小（每个 tier 多一次结构包装），但相较于原始 `OverlapScores` 的 **复制** 开销仍在可接受范围。<br/>- 去重路径强制使用 `Device` tier，避免了在 `lower_tier` 产生冗余 STORE/REMOVE，间接提升整体写入路径的吞吐。 |

| **可维护性** | ★★☆☆☆ | - 新增的 `From` 实现将内部结构映射到 wire 结构，使转换集中管理，易于后续扩展（如新增 tier）。<br/>- `DedupCacheStatusTracker` 中的注释明确说明了为何统一为 `Device`，降低后续误改风险。<br/>- 测试覆盖率大幅提升，帮助阻止回归。 |
| **兼容性** | ★☆☆☆☆ | - 增加的 `device_only` 带默认值 `false`，保持向后兼容。<br/>- 旧服务端仍会返回 `TieredScores`（完整），客户端可自行忽略 lower‑tier 部分。 |

---

**⚠️ 潜在风险**

1. **序列化体积增大**：在极端低带宽环境下，返回完整 tiered 数据仍可能占用显著流量。需要监控实际 payload 大小，确保 `device_only` 能被正确使用。  
2. **重复 StorageTier 处理**：`WireTieredMatchDetails` 转换时仅保留最后出现的同 tier 条目，若业务逻辑误生成重复条目，可能导致信息丢失。应在生成端避免重复或在转换层加入更严格校验。  
3. **去重路径强制 Device**：若后续出现需要保留 lower‑tier 信息的业务（如调度策略依赖 tier），当前实现会屏蔽该信息。文档需明确去重模式的约束。  
4. **并发删除竞争**：`hash_mapping` 的 retain 逻辑已经改为在最后一个 source 删除时才清理，但仍需确保在高并发下不会出现短暂的“missing mapping” 警告导致误删。  
5. **远程查询分支**：`RemoteIndexer::find_matches_by_tier` 新增 `device_only` 参数，若在其他未更新的调用方仍使用旧签名，将编译错误或运行时 panic。需检查全仓库调用路径。  

---

**💡 关注建议**

- **性能监控**：在生产环境开启指标（如 `remote_indexer_query_latency`, `wire_payload_bytes`）对比 `device_only=false/true` 的响应时间与网络使用，验证预期的 CPU/带宽节省。  
- **兼容性验证**：对未升级的旧客户端进行回归测试，确保在遇到新字段时能正常解析（默认 `false`) 并继续工作。  
- **去重模式文档**：在用户手册或配置说明中明确指出去重模式下所有块都会以 `Device` tier 上报，避免用户误以为 lower‑tier 仍可查询。  
- **重复 tier 检查**：在 `From<WireTieredMatchDetails> for TieredMatchDetails` 中加入 debug/警告级别的冗余检测，或在生产环境抛出错误以防数据不一致。  
- **错误路径审计**：当前在 `hash_mapping` 清理时仅在 `metadata.exists_in_any_source()` 为 false 时执行，建议在 `handle_remove` 结束后再次确认 `hash_mapping` 中不存在孤立条目，以免长期累计的 “stray entries”。  
- **回归测试**：确保 CI 包括 `dedup_normalizes_tier_to_device`、`test_shared_block_hash_across_sources_two_removes_succeed`、以及 remote 查询的 tiered 与 device‑only 两条路径，防止未来改动破坏该行为。  

---  

此提交在 **功能** 与 **架构** 上提供了更细粒度的查询能力，并通过去重路径的统一 tier 处理提升了系统一致性。只要注意上述风险点并做好监控/文档工作，整体收益大于成本，值得在生产环境快速推广。

### 架构影响  
| 维度 | 影响描述 |

| **模块划分** | 新增 `agents::trace` 独立子模块，封装 **配置 → 总线 → Sink** 三层结构。`TelemetryBus` 为 **一次性广播**，实现无锁 `tokio::sync::broadcast`，在热点请求路径仅调用 `BUS.publish`（O(1)）**。 |

| **后端异步** | `sink::spawn_workers_from_env` 在 `init_from_env` 时创建并持有 `CancellationToken`，每个 sink 通过 `mpsc`/`broadcast` 异步消费，避免阻塞主请求流。|

### 性能影响  
| 场景 | 影响点 | 预估成本 |
|------|--------|----------|
| **trace 关闭** | 仅一次 `is_enabled()` 检查 & `X_REQUEST_ID_CONTEXT_KEY` 插入（常数时间） | **可忽略** |
| **trace 开启 – 仅 stderr** | 序列化 `AgentTraceRecord` → `serde_json` → `tracing::info!`（异步日志后端） | 约 **0.5‑1 µs**/record（取决于日志后端） |
| **trace 开启 – jsonl** | `mpsc::Sender::send`（非阻塞）+ 单独 writer 线程压缩/写入 | **写入延迟 ≤ flush_interval**（默认 1 s），CPU 开销约 **0.2 %**（单核）在 10 k record/s 场景下 |
| **trace 开启 – jsonl_gz** | 与 jsonl 相同 + gzip 压缩（阻塞线程）+ segment roll 检查 | 约 **1‑2 µs**/record（压缩）+ **I/O 延迟**（磁盘）; 推荐在 SSD / NVMe 环境 |
| **请求内部计量** | `record_llm_metric_tokens` 在每个 token 生成阶段调用一次 `if trace_enabled` 检查并可能更新 `RequestTracker` | 额外分支开销 **< 100 ns**，对推理吞吐几乎无感知 |
| **内存** | 每条 trace 记录复制一次（`AgentTraceRecord`），默认 `capacity=1024`，最高占用约 **< 2 MiB**（每条 ~2 KiB） | 可通过 `DYN_AGENT_TRACE_CAPACITY` 调整 |

### 安全考虑  
- **数据脱敏**：trace 仅记录 **元数据**（模型、token 计数、时延、worker ID），**不包含 Prompt/Response 内容、采样参数、FinishReason**，符合隐私/合规要求。  
- **环境变量注入**：`DYN_AGENT_TRACE_SINKS`、`OUTPUT_PATH`、`CAPACITY` 等均为 **字符串/整数**，未做特殊权限校验；不应在不受信任的环境中暴露强制写入路径，否则可能导致 **路径遍历** 或 **磁盘消耗**（但 serde 直接使用提供的路径，不进行清理）。建议在生产环境限制可写目录。  
- **错误处理**：所有 I/O/序列化错误均记录 `tracing::warn!`，不会导致请求失败；但在极端错误（磁盘满、权限）下可能 **丢失 trace**，这属于可接受的“调试/分析”级别。  
- **并发安全**：`TelemetryBus` 使用 `tokio::sync::broadcast`，在高并发下会出现 **lagged** 丢弃记录的情况（已记录警告）。若业务对完整性有强需求，可提升 `capacity` 或使用持久化 sink（外部审计）。  
- **资源泄漏**：Sink 实现了 `Drop`，在 `CancellationToken` 触发时优雅关闭；但若进程异常退出（SIGKILL）仍可能留下未刷新缓冲。

## ⚠️ 潜在风险  

| 风险 | 说明 | 可能影响 |
|------|------|----------|
| **磁盘/IO 过载** | 开启 `jsonl_gz` 并在高吞吐（>10k records/s）情况下，磁盘写入或 gzip 压缩可能成为瓶颈。 | 可能导致 `lagged` 报警，trace 记录被丢弃。 |
| **记录丢失** | `broadcast` 的缓冲大小受 `capacity` 控制；若消费端处理不及时（例如 sink 坏掉）会出现 “bus lagged”。 | 调试信息缺失，尤其在短时高并发突发。 |

**SHA**: `ab18655` | 🔗 [查看提交](https://github.com/ai-dynamo/dynamo/commit/ab1865542a29d0a0da3f2fabb0c910f5f772916d)

**变更概览**  
本次提交把 ZMQ KV 事件的解析、过滤与规范化抽离到 `lib/kv-router/src/zmq_wire` 模块，实现 **ZmqEventNormalizer**。原本散落在 `listener.rs`、`zmq_wire.rs`（1132 行）中的反序列化、额外键处理、缓存规格过滤、警告计数等逻辑被拆分为四个子文件：`types.rs`、`deserialize.rs`、`extra_keys.rs`、`filter.rs`，并在 `convert.rs` 中保留原有的 `convert_event` 实现。`listener.rs` 只负责读取 ZMQ 帧、调用 `decode_event_batch` → `normalizer.preprocess` → `normalizer.normalize_preprocessed`，从而统一事件过滤与警告计数。  
同时，LLM 层的 KV‑router 发布链路被拆分为 **batch‑dedup‑sink** 三层，原来的 `EventDedupFilter`、`BatchingState`、`emit` 迁移到独立文件， `event_processor.rs` 只保留调度逻辑。

**影响范围**

   - 之前在 `listener` 中直接持有 `Arc<AtomicU32>`；现在改为 `ZmqEventNormalizer` 内部的 `warning_count`。如果外部监控仍然通过旧路径读取该计数，会失效。建议在 `ZmqEventNormalizer` 上暴露 `pub fn warning_count(&self) -> Arc<AtomicU32>` 或通过指标上报（已在 `kv-router` 的 metrics 中实现）。

6. **测试覆盖**
