### fix(sglang): advertise global DP ranks for routing (#10196)
**SHA**: `ad2e264` | 🔗 [查看提交](https://github.com/ai-dynamo/dynamo/commit/ad2e2642d818f3d633a58b46d061053b08b2fd78)

**🎯 变更类型**：Bug修复  
**⚡ 重要程度**：🔴高  
**📋 变更摘要**：  
1. 修正了 SGLang 多节点场景下模型注册时的 DP（Data‑Parallel）rank 可见性，确保模型卡在全局 DP 范围 `[0, dp_size)` 上注册，而不是仅使用本地节点的子区间。  
2. 新增 `model_card_dp_rank_bounds` 用于获取全局 DP 范围，更新注册流程及文档说明，并补充对应单元测试，防止未来回归。

**🎯 影响范围**：  
- `components/src/dynamo/sglang` 目录下的 **模型注册、路由调度、容量计算** 相关代码。  
- 受影响的子模块包括 `capacity.rs`, `register.rs`, `CLAUDE.md` 文档以及对应的单元测试。  

**🔍 技术洞察**：  
- **架构影响**：  
  - 将模型注册视角从“本地 DP 子区间”提升为“全局 DP 范围”，使得路由器能够跨节点感知并调度所有 DP rank。  
  - 保持原有 KV 事件、FPM、指标发布仍使用本地 DP slice，确保数据流向不变，仅在注册层面统一全局视图，符合现有多进程/多节点解耦设计。  
- **性能影响**：  
  - 注册阶段多返回 `dp_size`（而非本地子区间），增加的计算量极小，几乎不影响启动时延。  
  - 正确的全局 rank 可见性可以避免因路由找不到目标 DP 而产生的重试或任务迁移，间接提升整体吞吐和响应 latency。  
- **安全考虑**：  
  - 未直接涉及安全漏洞，但错误的 DP 可见性可能导致请求被错误路由或被丢弃，形成服务不可用（DoS）表现。修复后提升了系统的可靠性和可用性。  

**⚠️ 潜在风险**：  
1. **配置不一致**：如果外部部署脚本仍自行截取本地 DP slice 用于 `ModelRuntimeConfig`，可能导致注册与调度不匹配。  
2. **遗留代码**：其他未被检测到的路径仍可能使用 `local_dp_rank_bounds` 进行注册，需在代码审查中确认已全部替换。  
3. **文档同步**：文档已更新，但如果有外部依赖（如 Helm chart、Terraform 模块）仍依据旧说明，部署时可能误配置 `dp_size`。  

**💡 关注建议**：  
- **开发者**：在新增的或已有的模型注册逻辑中，务必使用 `model_card_dp_rank_bounds` 而非 `local_dp_rank_bounds`，并在代码审查 checklist 中加入 “全局 DP 注册路径校验”。  
- **测试**：持续运行 `test_model_card_registration_keeps_global_dp_range`，并在 CI 中加入对 `dp_size` 与 `nnodes*local_slice` 一致性的校验，以防回归。  
- **运维**：检查部署脚本或配置管理工具中关于 DP 范围的注释/说明，确保与文档统一；监控路由器日志中的 “routable data_parallel rank range” 信息，确认启动时已正确广播全局 DP 范围。  
- **文档**：在 `README` 或部署手册中突出 “仅 leader 进程负责模型注册，需使用全局 DP 范围” 的注意事项，避免用户误用本地 DP 范围导致路由失效。

### fix(tool-calling): recover bare DeepSeek tool calls without outer wrapper (#10133)
**SHA**: `6109eb7` | 🔗 [查看提交](https://github.com/ai-dynamo/dynamo/commit/6109eb7b558ec1452d3e7748e1407cb4a5cd1c55)

**🎯 变更类型**：Bug修复 / 功能增强（恢复 DeepSeek V3/V3.1 **bare** 工具调用，防止标记泄漏）  
**⚡ 重要程度**：🔴高  
**📋 变更摘要**  
- 在 `deepseek_v3` 与 `deepseek_v3_1` 的 JSON 解析器中，加入对 **缺少外层 `<｜tool▁calls▁begin｜>` 包裹** 的裸内部工具调用的检测与恢复逻辑。  
- 当检测到 `<｜tool▁call▁begin｜>` 时，解析器会提取所有裸调用块，生成 `ToolCallResponse`，并返回 **正常文本**（剔除工具调用标记）。  
- `detect_tool_call_start_*` 现在只要出现内部起始标记即返回 `true`，避免因为外层包装缺失而漏检。  
- `find_tool_call_end_position_json` 对 DeepSeek V3 系列补充了多裸调用的结束位置计算，确保一次性消费完整的连续裸调用块。  
- 同时补充了大量单元测试与 parity fixtures，验证恢复行为、标记不泄漏以及流式多调用场景。  

**🎯 影响范围**  
- `lib/parsers/src/tool_calling/json/deepseek_v3_parser.rs`  
- `lib/parsers/src/tool_calling/json/deepseek_v3_1_parser.rs`  
- `lib/parsers/src/tool_calling/json/mod.rs`（结束位置计算）  
- 相关检测函数 `detect_tool_call_start_*`  
- 单元测试与 parity 测试文件（DeepSeek V3 / V3.1）  

**🔍 技术洞察**  

- **架构影响**  
  - 变更局限在 **解析层**，不涉及上层业务逻辑或模型推理路径。  
  - 通过在 `parse_tool_calls_*` 中提前返回 `Ok((tool_calls, Some(normal_text)))`，保持原有 `Parser` 接口不变，兼容所有调用方。  
  - 新增的 `extract_tool_call_blocks_v3(_1)` 与 `parse_single_tool_call_v3(_1)` 已在同文件中实现，未改动外部模块依赖。  

- **性能影响**  
  - 仅在缺失外层包装的情况下多一次 `contains` 与 `find` 操作，时间复杂度仍为 **O(N)**（N 为输入长度），对正常流量几乎无感。  
  - 生成 `Uuid` 为每个恢复的调用分配唯一 ID，成本极低。  
  - 新增的 `tracing::warn!` 在生产环境默认关闭，仅在调试/日志级别开启时产生轻微开销。  

- **安全考虑**  
  - 仍然使用 `serde_json::to_string` 序列化解析得到的 `arguments`，与现有实现保持一致，无新增安全风险。  
  - 解析器现在会在找不到可恢复块时 **返回空 normal_text**，防止未过滤的 `<｜tool▁call▁begin｜>` 标记泄漏到消息内容，降低潜在的注入或混淆风险。  

**⚠️ 潜在风险**  

| 风险点 | 描述 | 缓解措施 |
|--------|------|----------|
| **误判普通文本包含 `<｜tool▁call▁begin｜>`** | 如果真实对话中出现该字符串（如代码示例），解析器会把后面的内容当作工具调用，导致误解析。 | - 新增 `detect_tool_call_start_*` 仍会先检查外层包装；若仅出现内部标记且前缀不符合工具调用结构，则 `parse_single_tool_call_*` 可能返回 `None`，最终返回空调用。<br>- 建议在生产配置中加入白名单或更严格的正则，或提供可选的 “strict mode”。 |
| **大量连续裸调用导致一次性内存占用** | `find_tool_call_end_position_json` 会一次性消费所有连续 `<｜tool▁call▁end｜>`，在极端情况下（大量重复标记）可能产生较大临时字符串。 | - 该情况极少见，且已限制为 `Vec<ToolCallResponse>`，每个调用仅存放函数名与 JSON 参数。<br>- 如有担忧，可在 `extract_tool_call_blocks_*` 加入长度上限检查。 |
| **日志泄漏** | `tracing::warn!` 中记录了 `stripped_bytes`、`recovered_calls` 等信息，若开启高日志等级，可能暴露内部模型输出。 | - 日志层级默认为 `WARN`，生产环境可调为更高等级（`ERROR`）或通过环境变量关闭；代码已使用结构化日志，易于过滤。 |

### feat(kv-router): optional binary-search mode for positional indexer find_matches (#10181)
**SHA**: `35ec918` | 🔗 [查看提交](https://github.com/ai-dynamo/dynamo/commit/35ec918733a327a379a9ba4f8b1e595870345e2c)

**🎯 变更类型**：功能增强 / 性能优化  
**⚡ 重要程度**：🔴高  
**📋 变更摘要**：在 `PositionalIndexer` 中加入可选的二分搜索模式 (`SearchMode::Binary`) 作为 `find_matches` 的实现替代原始的跨距跳跃搜索 (`SearchMode::Strided`)。搜索模式可通过环境变量 `DYN_ROUTER_POSITIONAL_SEARCH_MODE` 或显式构造函数进行控制，并在核心实现中新增 `binary_search_matches`。对应单元测试、基准和辅助断言也同步更新，确保两种模式在业务约束下产生一致的 `OverlapScores`。  

**🎯 影响范围**：  
- `lib/kv-router/src/indexer/positional.rs`（核心索引逻辑）  
- `lib/kv-router/src/indexer/tests.rs`（新增对二分模式的差分测试）  
- `lib/kv-router/src/test_utils.rs`（新增 `assert_overlap_scores_eq` 辅助函数）  
- 所有使用 `PositionalIndexer` 的调用方，尤其是 `ThreadPoolIndexer` 的构造路径（`flat` 与 `flat_binary` 变体）  

---

### 🔍 技术洞察

| 维度 | 影响分析 |
|------|----------|
| **架构影响** | • 通过 `SearchMode` 枚举把搜索策略抽象为可配置的运行时选项，保持原有 API（`PositionalIndexer::new`）不变，仅在内部引入环境变量读取，遵循 **“默认向后兼容 + 显式覆盖”** 的设计原则。<br>• 业务层 `KvIndexerInterface` 仍通过统一的 `find_matches` 接口调用，代码路径分支在 `PositionalIndexer` 内部完成，外部模块感知不到实现差异。<br>• 新增 `PositionalIndexer::new_with_mode` 与 `with_search_mode` 构造器，提供明确的模式绑定，支持测试/基准驱动的可控实验。 |
| **性能影响** | • **二分搜索** 在工作集 **单调递减**（即“活跃 workers 集合随位置只能减少”）的常见场景下，理论查询次数从 `O(len / jump_size)` 降至 `O(W·log L)`，其中 `W` 为起始活跃 worker 数，`L` 为序列长度，显著降低大序列、跳距较大时的探测次数。<br>• 对 **紧窗口**（`len - lo <= jump_size`）仍回退线性扫描，避免二分带来的额外分支开销，兼顾最坏情况。<br>• 通过 `ensure_seq_hash_computed` 复用已有的序列哈希向量，保持 **缓存友好**，避免重复哈希计算。<br>• 代码在 `binary_search_matches` 中直接在原 `active` 集合上原地 `reconcile_active_workers`，避免一次性克隆导致的堆分配，提升并发路径的内存占用和 GC 压力。 |
| **安全考虑** | • 新增的环境变量读取仅影响 **行为选择**，不会泄露或改变数据本身。错误值会记录 `tracing::warn!`，保持安全边界。<br>• `binary_search_matches` 的实现引用了相同的内部检查 (`count_workers_at == active.len()`) 来保证 **集合相等** 判定的正确性；若 `workers(p) ⊄ active`（稀疏绝对位置存储）导致误判，当前实现已在文档中明确说明并限定使用场景。<br>• 代码未引入 `unsafe`，且所有并发访问继续使用 `DashMap`，保持线程安全。 |
| **可维护性** | • `SearchMode` 采用 `#[derive(Default)]`，默认行为保持不变，降低回归风险。<br>• 新增的 `binary_search_matches` 与原实现结构相似，分离为私有方法，便于后续单独调优或替换。<br>• 测试覆盖显著提升：<br> - 单元测试 `binary_matches_strided_both_early_exit_settings` 验证两种实现在所有边界下等价。<br> - 大规模随机差分测试 `positional_binary_matches_strided_differential` 在真实工作负载下进行端到端对比。<br> - 新增断言 `assert_overlap_scores_eq` 把相等检查显式化，提升可读性。 |
| **兼容性** | • 只在 `PositionalIndexer::new` 中读取环境变量，已有部署若未设置该变量仍使用默认 `Strided`，不会产生行为变化。<br>• 新增的 `flat_binary` 变体在测试矩阵中使用显式构造函数，生产代码若仍使用 `flat`，行为保持不变。 |

### ⚠️ 潜在风险

| 风险点 | 说明 | 缓解措施 |
|--------|------|-----------|
