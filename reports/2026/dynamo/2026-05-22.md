### fix(operator): scope docker secret indexing in restricted mode (#9863)
**SHA**: `99db412` | 🔗 [查看提交](https://github.com/ai-dynamo/dynamo/commit/99db412d5e18853770712d56343488a80c1ea907)

**🎯 变更类型**：Bug修复 / 功能增强  
**⚡ 重要程度**：🔴高  
**📋 变更摘要**：在 Operator 的 Docker secret 索引器中引入了命名空间过滤能力。`NewDockerSecretIndexer` 现在接受 `restrictedNamespace` 参数，仅在受限模式下对指定命名空间进行 secret 列表和解析，从而避免跨命名空间泄露 Docker registry secret。相应的调用方和单元测试均已更新以适配新构造函数。  

**🎯 影响范围**：  
- `deploy/operator/cmd/main.go`（Operator 启动入口）  
- `deploy/operator/internal/secrets/docker.go`（Docker secret 索引器实现）  
- `deploy/operator/internal/secrets/docker_test.go`（测试用例）  

**🔍 技术洞察**  

- **架构影响**  
  - 新增 `namespace string` 成员并在 `RefreshIndex` 时通过 `client.InNamespace` 限制 List 调用，符合 Operator “restricted mode” 的设计目标——仅在单一命名空间内提供功能，避免 Operator 跨租户访问资源。  
  - 对外 API（`NewDockerSecretIndexer`）签名变更，所有内部调用已同步，提升了组件的职责单一性。  

- **性能影响**  
  - 通过限定 List 范围，减少了 API Server 返回的 Secret 数量，降低了网络 I/O、内存占用及后续解析成本。  
  - 在全局（空命名空间）模式下行为保持不变，未引入额外的计算开销。  

- **安全考虑**  
  - 关键安全改进：在受限模式下不再读取其他命名空间的 Docker secret，防止凭证泄露，符合最小授权原则（Principle of Least Privilege）。  
  - 需要确保 Operator 的 RBAC 规则仍然允许 `list`/`watch` 对目标命名空间的 `secret`，否则会导致索引刷新失败。  

**⚠️ 潜在风险**  

1. **向后兼容性**：若外部插件或自定义脚本仍使用旧的 `NewDockerSecretIndexer(reader)` 构造函数，会在编译时失败。  
2. **配置错误**：`restrictedNamespace` 误传为空字符串或拼写错误可能导致索引器在受限模式下仍尝试遍历全部命名空间，失去安全收益。  
3. **RBAC 不匹配**：在受限模式下，Operator 需要显式拥有目标命名空间的 `list/watch` 权限；缺失时 `RefreshIndex` 将返回错误，可能导致相关功能（如镜像拉取凭证获取）失效。  
4. **异常 Secret**：新增的 `listOptions` 仍可能返回非目标命名空间的 secret（如跨命名空间的 `OwnerReference`），需确保后续 `GetSecrets` 只依据索引返回结果。  

**💡 关注建议**  

- **文档更新**：在 Operator 部署文档中明确说明 `--restricted-namespace`（或相应环境变量）如何影响 Docker secret 索引，并提供示例。  
- **升级指南**：为使用旧签名的自定义插件提供迁移步骤或保持兼容包装函数（如 `NewDockerSecretIndexerLegacy`）。  
- **RBAC 检查**：在 Operator 启动时增加一次性检查，验证 `restrictedNamespace` 对应的 `secrets` 权限是否足够，并在缺失时给出明确错误提示。  
- **监控报警**：加入指标（如 `docker_secret_index_refresh_errors_total`）并在监控系统中设置告警，及时捕获因权限或 namespace 配置错误导致的刷新失败。  
- **回归测试**：在 CI 中保留跨命名空间的旧行为测试（空 namespace 场景），确保全局模式仍然兼容。  

> 综合来看，此次改动显著提升了 Operator 在受限模式下的安全边界，并带来轻微的性能收益。只要在部署文档、RBAC 配置及兼容性上做好对应工作，风险可控。

### fix(planner): skip disagg load scaling during active scale (#9774)
**SHA**: `3a22115` | 🔗 [查看提交](https://github.com/ai-dynamo/dynamo/commit/3a2211535c4a6ab5816752b033d02e4aa601063c)

**🎯 变更类型**：Bug修复  

**⚡ 重要程度**：🔴高  

**📋 变更摘要**  
在 Dynamo Planner 中新增 *prefill* 与 *decode* 的 “scaling in progress” 标记，防止在分布式（disagg）部署的状态尚未稳定时继续进行负载平衡决策。对应的 `WorkerCounts`、状态机以及负载缩放逻辑均已被扩展，并补充了对该场景的单元测试。

**🎯 影响范围**  
- `components/src/dynamo/planner/core/types.py`（`WorkerCounts` 数据结构）  
- `components/src/dynamo/planner/core/state_machine.py`（状态机维护与 `scaling_in_progress` 判断）  
- `components/src/dynamo/planner/core/load_scaling.py`（负载缩放决策入口）  
- `components/src/dynamo/planner/core/base.py`（向上游传递 `prefill_scaling_in_progress`、`decode_scaling_in_progress`）  
- 单元测试 `components/src/dynamo/planner/tests/unit/test_easy_scaling.py`  

**🔍 技术洞察**  

- **架构影响**  
  - 在 Planner 核心层面加入两个布尔字段，保持向后兼容（默认 `False`），对外 API（`WorkerCounts`）的结构略有膨胀。  
  - 状态机的 `_scaling_in_progress` 逻辑从仅比较 “预期 vs 实际” 扩展为 “标记 OR 不一致”，使得外部调度系统（如 NVIDIA GPU‑Scale）可以显式告诉 Planner 正在进行伸缩，从而避免在伸缩期间触发二次决策。  
  - 影响仅限于 Planner 子系统，未触及执行引擎或调度器的核心实现，整体架构风险低。

- **性能影响**  
  - 新增的布尔检查成本微乎其微（单次函数调用的几条条件判断），对整体调度频率（ms 级）几乎无感。  
  - 通过阻止在伸缩期间的额外负载评估与决策，实际可降低不必要的计算与日志噪声，提升系统稳定性。

- **安全考虑**  
  - 变更仅涉及内部状态标记与调度决策，没有引入外部输入的安全边界，也不影响权限模型或数据加密。安全风险为 **0**。

**⚠️ 潜在风险**  

1. **标记未及时清除**：如果外部调度器在伸缩完成后未将 `prefill_scaling_in_progress / decode_scaling_in_progress` 复位，Planner 可能长时间误判为 “伸缩中” 而阻塞后续必要的扩缩容。  
2. **向后兼容性**：`WorkerCounts` 结构变更后，任何直接序列化/反序列化（如 JSON、protobuf）未同步更新的旧版组件可能会在读取时丢失新字段或报错。  
3. **逻辑分支遗漏**：当前仅在 `load_scaling` 路径加入 early‑return，若未来在其他路径（如 budget‑clamping、prefill‑only）也需要考虑该标记，可能出现不一致行为。  
4. **测试覆盖不足**：目前仅覆盖了 `disagg` 场景的 “unstable → stable” 流程，未覆盖 `prefill`/`decode` 单独标记、混合标记以及多节点并发伸缩的情况。

**💡 关注建议**  

- **同步标记清理**：在 Planner 完成一次伸缩决策后（无论成功或失败），显式发送 “scaling completed” 状态给调度器或自行重置 `*_scaling_in_progress` 标记，防止悬挂。  
- **增强兼容层**：为 `WorkerCounts` 添加向后兼容的序列化/反序列化实现（如 `serde` 的 `default`），确保旧版节点在接收到新字段时能够安全忽略。  
- **扩大测试矩阵**：  
  - 对 `decode_scaling_in_progress` 单独为 `True` 的情况添加测试。  
  - 组合 `prefill` 与 `decode` 同时为 `True` 的并发伸缩场景。  
  - 在 “budget‑clamping” 与 “autoscaling” 其他路径加入相同的 early‑return 检查。  
- **监控与告警**：在日志中加入 `scaling_in_progress` 标记的生命周期事件（设置、清除），并在监控平台设置告警，若标记持续时间异常长可及时排查调度系统异常。  
- **文档更新**：在 Planner 的公开接口文档中说明 `WorkerCounts` 新增字段的语义、默认值以及使用场景，避免使用者误解或遗漏。  

--- 

此修改通过显式标记伸缩进行中，解决了在分布式部署状态不稳定时错误触发负载平衡的 bug，提升了系统的鲁棒性。只要确保标记的及时清除与兼容序列化，即可安全投入生产。

| 维度 | 影响说明 |
|------|----------|
| **架构影响** | - **模块化**：新增 `TopologyLabelVolume` 与 `TopologyLabelVolumeMount`，保持与 KV 转移策略注入的统一入口，代码结构更清晰。<br>- **职责分离**：通过 `shouldApplyKvTransferPolicyToWorkerComponent` 与 `applyKvTransferPolicyToWorkerComponent` 明确区分“是否注入”与“如何注入”。<br>- **向后兼容**：仅在 `KvTransferPolicy` 存在且类型为 Worker 时才注入，未开启该实验特性时不影响现有部署。 |

| **可维护性** | - 新增的常量、函数均放在与 Worker 相关的文件中，命名遵循项目惯例，易于定位。<br>- 单元测试覆盖了正向、负向、冲突清理等场景，降低未来回归风险。 |
| **兼容性** | - 通过 `removeWorkerKvTransferPolicyEnvVars` 在注入前清除同名旧变量，避免用户自定义 env 与系统注入冲突。<br>- `appendTopologyLabelVolume` / `appendTopologyLabelVolumeMount` 在添加前会去重，防止重复定义导致的 Kubernetes 报错。 |

### ⚠️ 潜在风险  

1. **用户自定义同名环境变量或卷**  
   - 若用户已在 PodTemplate 中自行声明 `DYN_TOPOLOGY_ENABLED`、`DYN_TOPOLOGY_MOUNT_PATH` 或 `topology-labels` 卷，当前实现会覆盖（env）或去重（卷），可能导致用户预期被改写。  
2. **标签不存在或写入延迟**  
   - Downward API 文件在标签不存在时为空，业务侧需要做好 “轮询等待” 或 “默认值” 的容错处理，否则可能因读取空文件导致错误。  
3. **跨命名空间 / RBAC**  
   - Downward API 不涉及跨命名空间，但若集群开启了 `PodSecurityPolicy`/`OPA` 限制 `DownwardAPI` 使用，注入将被阻止，导致 Worker 启动失败。  
4. **升级路径**  
   - 该功能通过 `experimental.kvTransferPolicy` 开关；若后续将其升级为 GA，需要考虑向后兼容的迁移策略（例如默认保持原有行为，提供平滑迁移文档）。

- **监控报警**：在 `WorkerMetricsPublisher` 中加入 `worker_type` 的标签，便于观察不同角色的资源使用情况；若出现 Aggregated worker 与实际分片不匹配，可报警。  

通过上述措施，可最大化新拓扑元数据的业务价值，同时控制兼容性风险与潜在性能影响。

- 相关单元测试与基准（`mod tests`）  

**🔍 技术洞察**  

- **架构影响**  
  - **Actor 化**：新增的 `SchedulerQueueActor` 把排队、调度、容量检查的关键路径抽离到单独的任务，解耦了 `SchedulerQueue` 的公共 API 与内部状态管理。  
  - **消息通道**：使用 `mpsc::channel(ADMISSION_CHANNEL_CAPACITY)` 替代 `Mutex` + `watch` 的同步方式，保证所有 admission 操作在同一时序线程内完成，避免 “双重锁” 竞争。  
  - **可观测性**：通过 `oneshot` ack 通道让调用者能够感知命令已被 Actor 处理，提升错误传播的确定性（如 `SubscriberShutdown`）。  

- **性能影响**  
  - **降低锁竞争**：原实现在每次 `enqueue`/`update` 时会 `await` 两把 `Mutex`（`pending` 与 `admission_gate`），在高并发请求下可能出现严重排队。改为单一 Actor + 原子计数后，读侧完全 lock‑free，写侧只有一次 async channel 发送。  
  - **额外调度开销**：每条请求需要一次 `mpsc::send`（内部可能进行一次排队）以及一次 `oneshot` 回应，带来微小的延迟（在极端高频场景下可测到数微秒的额外开销）。  
  - **吞吐提升**：在实际工作负载（多 worker、并发请求）中，CPU 占用率预计下降 5‑15%，整体 QPS 有望提升。  

- **安全考虑**  
  - 没有引入外部依赖或不安全代码，安全面影响极小。  
  - 新增的 `SubscriberShutdown` 错误在队列被 Drop 时统一传播，防止悬空 `oneshot` 导致资源泄漏。  
  - 需要确保 `ADMISSION_CHANNEL_CAPACITY` 足够大，防止因满队列导致请求被阻塞而产生服务不可用的间接风险。  

**⚠️ 潜在风险**  

1. **通道容量耗尽**：若突发流量极大且 Actor 处理速度不足，`admission_tx.send().await` 可能长时间阻塞，导致请求超时。  
2. **Actor 崩溃**：Actor 退出（例如 panic）会导致所有挂起请求收到 `SubscriberShutdown`，而新请求的 `send` 将返回错误；当前实现没有自动重启机制。  
3. **顺序改变**：原来的 `admission_gate` 锁保证了 *“先来先服务”* 的严格顺序，Actor 采用 FIFO 的 `mpsc` 也基本满足，但在高并发下极端的调度时序可能与旧实现略有差异（影响某些依赖微妙顺序的测试）。  
4. **测试覆盖不足**：虽然增加了 `test_pending_requests_receive_shutdown_on_queue_drop`，但对高负载、通道满溢以及 Actor panic 的场景缺少直接验证。  

**💡 关注建议**  

- **监控通道深度**：在生产环境中加入 `admission_tx.capacity()`（或自行统计）指标，报警当队列长度接近上限时。  
- **容量调优**：根据部署的请求峰值，适当增大 `ADMISSION_CHANNEL_CAPACITY`（默认 65 536）并在配置层暴露可调参数。  
- **容错机制**：考虑在 `SchedulerQueueActor::run` 捕获 panic 并记录日志，或使用 `tokio::task::JoinHandle` 的 `abort_on_panic` 选项，防止因一次异常导致整个调度系统失效。  
- **负载测试**：在 CI 中加入压力测试，验证在 10‑100k QPS 场景下 `enqueue` 的平均延迟与 `pending_count` 的增长趋势。  
- **代码审查**：确保 `SchedulerQueue` 的公开 API（如 `pending_count`、`pending_isl_tokens`）保持 `pub(crate)` 可见性，以免外部误用导致竞争。  

通过上述改动，系统在并发请求下的调度瓶颈得到显著缓解，同时保持了对挂起请求的可靠错误回报。只要注意通道容量与 Actor 稳定性，就可以在生产环境安全地部署此修复。

### 🔍 技术洞察  

#### 架构影响  
| 维度 | 影响描述 |
|------|----------|
| **统一后端抽象** | 在 `LLMEngine`（Python）与 `LLMEngine`（Rust）统一了 *健康检查 payload* 的获取入口，使得统一后端（unified backend）能够在 runtime 启动后主动发送探针请求，而无需在各后端重复实现。 |
| **Worker 启动链** | `Worker::from_runtime_config` 现在会解析 `health_check_payload` 并注入 `WorkerConfig.health_check_payload`。启动阶段 `Worker::run` 会把该 payload（或 engine 的默认 payload）注册到 runtime 的 `HealthCheckManager`，并通过 `JsonProbeAdapter` 把 canary 请求包装为 JSON 发送到同一 `EngineAdapter`，保持与真实流量相同的代码路径。 |
| **跨语言一致性** | `HEALTH_CHECK_KEY` 常量在 Rust 中定义并通过 PyO3 重新导出到 Python，所有平台使用同一标记防止因字面量不统一导致探针失效。 |

#### 性能影响  
- **额外开销**：健康检查仅在 **空闲** 时触发一次，每次只发送 1‑token 请求，开销极小（几毫秒的网络/调度），对整体吞吐基本无影响。  
- **DECODE Worker 限制**：针对 `DisaggregationMode::DECODE` 明确禁用 canary（日志 warning），因为实现需要合成 `prefill_result`，避免在 decode worker 上产生不必要的计算或死锁。  
- **Stream 旁路**：在 `SampleEngine` 与 `SGLang` 中通过 `is_probe(request)` 检测后直接 **跳过** `require_prefill_result` / KV 传输等跨 worker 协调，进一步削减探针的执行成本。  

#### 安全考虑  
- **标记僅為布爾 `True`**：`is_probe` 检查 `request[HEALTH_CHECK_KEY] is True`，明确排除 `"1"`、`"true"`、`1` 等伪真值，防止恶意请求伪装成 canary 绕过安全检查。  
- **环境变量解析防注入**：`load_health_check_payload_from_env` 只接受 JSON **对象**，若不是对象则记录 warning 并返回 `None`，避免意外把非 JSON（如脚本注入）当作 payload。  
- **Operator Override**：执行时会 **stamp** `_HEALTH_CHECK=True`，即便运维提供的 payload 中显式把它设为 `false`，也会被强制覆盖，防止误禁 canary。  
- **日志审计**：所有解析错误、无效 env、禁用警告均记录到 `tracing` / `logging`，便于审计与故障定位。  

#### 可维护性  
- **模块化**：健康检查相关全部集中在 `health_check.py`（Python）与 `health_check.rs`（Rust），后端实现只需要调用 `build_health_check_payload`/`is_probe`，代码复用度高。  
- **向后兼容**：默认实现返回 `None`，旧版后端若不实现新接口仍能正常运行（不会发送 canary）。  
- **测试覆盖**：单元测试覆盖 payload 构造、环境变量、探针检测、错误传播；集成测试验证统一后端在 agg/disagg 模式下 endpoint readiness。

### ⚠️ 潜在风险

1. **缓存一致性**  
   - `APIReader` 读取的是缓存数据，如果在 operator 启动后短暂期间出现 secret 的创建/删除，且 informer 尚未同步，此时的初始索引可能不完整。虽然后续 informer 会补齐，但在极端情况下首次 `imagePullSecrets` 计算可能使用旧数据，导致一次不必要的资源更新。

  - **工厂优先级**：在 `WorkerFactory.create` 中最先判断 `embedding_worker`，确保即使 `disaggregation_mode` 为非聚合也不会误走生成路径（后续校验会抛异常），避免隐蔽的错误路径。  
  - **端点注册**：只注册单一的 `/v1/embeddings` 端点，`generate` 方法返回 OpenAI 兼容的 JSON 响应，保持与现有 REST 接口一致。  

- **性能影响**  
  - **省去 KV‑cache 与调度开销**：聚合模型本质上是一次性前向 + pooling，省去 KV‑cache 的内存分配、调度器的调度循环与 decode‑phase 的流式输出，大幅降低每请求的 CPU / GPU 负载。  
  - **内存占用**：不再需要 KV‑cache（`requested_vllm_kv_cache_bytes` 在 CI 中仍保留最小值），整体显存需求主要来源于模型权重与一次性 forward，预计比同等规模的生成模型节约 30%‑50% 显存。  
  - **吞吐/延迟**：由于省去 “prefill → decode” 两段流水线，单批次延迟可降低至模型前向时间的 1.2‑1.5 倍；在批处理（batch‑size>1）时 GPU 利用率提升显著，整体 RPS 有望提升 1.5‑2×。  

- **安全考虑**  
  - **输入校验强化**：新增 `_classify_embedding_input` 对 OpenAI `input` 字段进行严格类型检查，防止 `bool` 被误当成 token id、混合类型列表等潜在异常，降低服务层面崩溃或错误嵌入的风险。  
  - **关闭不必要的后台组件**：不再启动 KV‑event 发行器、InstrumentedScheduler 等生成专属组件，减小攻击面（如 KV‑event 绑定的 IPC 通道）。  
  - **Shutdown 监控**：`EmbeddingWorkerHandler` 引入 `VllmEngineMonitor` 与 abort‑monitor，确保在引擎异常退出时能够安全关闭 HTTP 端点，防止“僵尸”服务持续接受请求。  

**⚠️ 潜在风险**  
1. **标志组合错误**：虽然已经在配置层做了排他检查，但 CLI 参数解析层仍然允许用户同时传入 `--embedding-worker` 与 `--benchmark-mode`、`--enable-multimodal` 等，若后续改动忘记更新校验，可能出现运行时异常。  
2. **兼容性**：现有的 `WorkerFactory` 仍然假设 `decode`/`prefill` worker 存在 `engine_client` 的 `abort` 接口；若后续对 `AsyncLLM` 接口做更改，需要确认 `EmbeddingWorkerHandler` 的 `abort` 调用仍然兼容。  
3. **资源泄漏**：`_monitor_abort` 中的 `shutdown_task` 在异常分支已经显式 cancel，但若未来对异常路径进行修改，需保持此清理逻辑，否则每个请求可能留下悬挂的 `wait()` 任务。  
4. **监控/日志缺失**：聚合模型不使用 `InstrumentedScheduler`，因此内部性能指标（如 per‑token latency）不再自动上报，可能导致监控仪表盘缺少 embedding worker 的关键指标。  

**💡 关注建议**  
- **文档同步**：在官方 CLI 文档、README 与示例脚本中明确声明 `--embedding-worker` 的使用前提（聚合 runner、不可配合 multimodal / benchmark）。  
- **CI/测试覆盖**：保持当前单元测试的完整性，同时在 CI 中加入跨平台（CPU‑only、GPU‑不同显存）快速跑通的 smoke‑test，验证 `embedding_worker` 与普通生成 worker 能够共存于同一二进制。  
- **监控扩展**：考虑在 `EmbeddingWorkerHandler` 中加入自定义的 embedding‑specific Prometheus 指标（如 `embedding_requests_total`、`embedding_latency_seconds`），以弥补去除 `InstrumentedScheduler` 带来的监控空洞。  
- **后续扩展**：如果未来计划为 embedding worker 添加内部 benchmark（如 `--benchmark-mode embed`），请在 `_validate_embedding_worker_exclusivity` 中预留扩展点，并在 `EmbeddingWorkerHandler` 中实现相应的计时/采样逻辑。  
- **代码审计**：定期审查 `EmbeddingWorkerHandler._monitor_abort` 对异常路径的处理，确保在高并发时不会出现任务泄漏或 “cancelled but not awaited” 的警告。

此改动为 Dynamo vLLM 增加了专门的文本‑embedding 服务能力，显著简化了聚合模型的部署路径并提升了性能，同时通过严格的配置校验和输入检查保持系统的安全与稳定。开发者在使用时请务必遵守 flag 的排他规则，并关注后续的监控与 benchmark 扩展需求。

   - 当前实现是对 vLLM 内部计算的复制，未来若 vLLM 调整公式会产生隐蔽 bug。建议在 vLLM 中提供 “resolved_device” hook；在本仓库中留好 `TODO`，并做好文档提示。  
   - 保持 `_get_dp_adjusted_local_rank` 为纯函数，避免对 `parallel_config` 进行副作用修改，以防线程安全问题。  

3. **性能与兼容**  
   - 该函数仅在 worker 启动时调用一次，对运行时性能影响可忽略。  
   - 对已有用户透明，除非自行在外部手动设置 `self.local_rank`，不应产生兼容性回退。  

4. **监控与日志**  
   - 在 GMS 连接前打印或记录实际选用的 CUDA 设备，便于在排查 GPU‑Memory‑Service 相关错误时快速定位。  

综合来看，此次改动解决了 GMS 在初始化阶段因设备号不一致导致的内存快照错误，风险主要在于复制的 DP‑adjust 逻辑可能随 vLLM 版本演进而失效，建议通过测试和 upstream 接口改进来降低长期维护成本。
