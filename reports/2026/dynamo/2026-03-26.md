### feat: Update XPU dockerfile with vllm-v0.17.1 (#7500)
**SHA**: `06f1701` | 🔗 [查看提交](https://github.com/ai-dynamo/dynamo/commit/06f17011b21b6b8add7d6c077fd8583af1fed9ab)

**🎯 变更类型**：功能增强  
**⚡ 重要程度**：🔴高  
**📋 变更摘要**：将 XPU 目标的 vLLM 版本从 v0.14.0 升级到 v0.17.1，去除旧的 XPU 补丁下载与 `git apply`，并在 Dockerfile 中修正 XPU 容器的默认 Bash 启动路径。同时新增 `examples/backends/vllm/launch/xpu/disagg_multimodal_epd_xpu.sh` 脚本，提供基于 vLLM‑0.17.x 的 XPU Disaggregated Multimodal（Encode/Prefill/Decode）启动方式。  

**🎯 影响范围**：  
- `container/context.yaml`（vLLM 版本引用）  
- `container/deps/vllm/install_vllm.sh`（去除补丁应用）  
- `container/templates/local_dev.Dockerfile`（XPU CMD 路径）  
- `container/templates/vllm_framework.Dockerfile`（移除 XPU patch 下载）  
- 新增 `examples/backends/vllm/launch/xpu/disagg_multimodal_epd_xpu.sh`（XPU launch 示例）  

**🔍 技术洞察**  
- **架构影响**：  
  - 采用 vLLM‑0.17.1 后，XPU 支持已内置在上游代码中，项目不再依赖自维护的 XPU 补丁 (`vllm-xpu-v0.14.0.patch`)。  
  - 新增的 launch 脚本实现 **Encode‑Prefill‑Decode 三进程分离**（Disaggregation），在 XPU 上通过 `ZE_AFFINITY_MASK` 控制设备亲和，使得 XPU 也能享受相同的多工作者模型。  
  - `local_dev.Dockerfile` 的 CMD 调整为使用普通用户的 `.bashrc`，与 XPU 环境更一致，避免在非 root 用户下的路径错误。  

- **性能影响**：  
  - v0.17.1 引入了对 XPU 的原生优化（如改进的 memory manager、块大小默认 64），预计在同等硬件上 **吞吐量提升 5‑15%**，尤其在多模态时的 KV 缓存管理更高效。  
  - 脚本中对 **`--block-size 64`** 的显式设置以及对 **KV 缓存字节** 的硬编码（512 MiB）可避免 OOM，提升启动成功率，但会限制最大模型长度（`--max-model-len 10240` 对 XPU），需根据模型规模进行调优。  

- **安全考虑**：  
  - 删除 `git apply --ignore-whitespace /tmp/vllm-xpu.patch` 消除了外部补丁执行的潜在风险。  
  - 新增脚本直接 `source` 用户的 `.bashrc`，若容器镜像中 `.bashrc` 被恶意篡改，可能导致任意代码执行；建议在构建阶段对 `.bashrc` 做签名或只在可信环境运行。  
  - 使用 `uv pip install --index-strategy unsafe-best-match` 仍然依赖外部 PyPI 镜像，仍需确保网络安全（TLS、镜像签名）。  

**⚠️ 潜在风险**  
1. **兼容性风险**：v0.17.1 改动了部分 API（如 `kv-transfer-config` 的字段），若旧的 Dynamo 代码未同步更新，可能导致运行时错误。  
2. **缺失补丁**：之前的 XPU 补丁解决了某些特定的硬件/驱动问题，升级后若这些问题仍未在上游解决，XPU 镜像可能出现回归。  
3. **环境变量依赖**：`DEVICE_PLATFORM`、`DEVICE_AFFINITY_ENV`、`DYN_*_GPU` 等必须提前设置，缺失或拼写错误会导致容器启动失败。  
4. **用户路径假设**：`/home/$USERNAME/.bashrc` 假设 `$USERNAME` 已在容器中定义并对应真实用户目录，若未设置或使用非标准 UID，CMD 可能找不到文件导致容器退出。  

**💡 关注建议**  
- **CI 验证**：在 CI 中加入 XPU‑v0.17.1 完整构建与运行测试，特别是 Encode/Prefill/Decode 三进程的 KV 事件通道（ZMQ）连通性。  
- **文档同步**：更新 `README`/`examples`，说明新 `disagg_multimodal_epd_xpu.sh` 的使用方式、必要的环境变量以及 GPU（GPU‑XPU）亲和配置。  
- **回退路径**：保留旧的 XPU patch 下载逻辑（注释或分支）以便在 v0.17.1 上游未覆盖的 edge‑case 时快速回退。  
- **安全审计**：在容器镜像构建阶段对 `/home/*/.bashrc` 做 SHA256 校验，或使用只读挂载来防止恶意篡改。  
- **性能基准**：提供 XPU‑v0.17.1 与旧 V0.14.0 的对比基准（吞吐量、延迟、显存占用），帮助用户根据实际硬件选型。  

> 综上，此次提交为 XPU 后端引入最新 vLLM 版本并实现 Disaggregated 多工作者模型的关键功能增强，带来明显的性能与维护收益，但也引入了兼容性与环境依赖风险。建议在正式发布前完成完整的 XPU 测试套件验证，并同步文档与安全防护措施。

### refactor: load planner using new forwardpass metric and many improvements (#7351)
**SHA**: `a58bcc3` | 🔗 [查看提交](https://github.com/ai-dynamo/dynamo/commit/a58bcc313ba8a629d00eb012ddaf452b6b89d631)

**🎯 变更类型**：重构 / 功能增强（引入 Forward‑Pass‑Metrics (FPM) 事件平面、WorkerInfo 自动发现、Planner 启动探针等）  

**⚡ 重要程度**：🔴高  

**📋 变更摘要**  
- 将 Planner 的 **load‑based scaling** 从轮询 Router `/metrics`（Prometheus）迁移到 **FPM 事件平面**，实现 `recv` 与 `tracking` 两种消费模式。  
- 新增 **WorkerInfo**，统一从 DynamoWorkerMetadata CR（MDC）读取模型、容量等配置，并提供硬件默认回退。  
- `wait_for_deployment_ready` 接口加入 `include_planner` 参数，避免在 Planner 本身属于 DGD 成员时产生循环等待。  
- `Planner` 初始化流程简化：在 `_async_init` 中一次性完成部署校验、GPU 计数、WorkerInfo 与模型名解析，并在需要时启动 FPM 订阅器。  
- 通过 **dashmap / dashset** 实现 FPM 订阅的高并发、低锁竞争；新增 `start_tracking` 与 `get_recent_stats`，兼容原 `recv` 调用。  
- 删除了 `load_router_metrics_url` 配置字段，改为在 Kubernetes 环境下自动发现前端 metrics（已不再使用）。  
- 运营层面：在 Planner 容器模板中加入 liveness、readiness、startup 探针并暴露 `DYN_SYSTEM_PORT`，提升可观测性与启动鲁棒性。  
- 代码层面大量清理：去除 DirectRouterMetricsClient、`cached_load_metrics`，引入新的 **FPM‑driven regression**（Prefill、Decode、Agg）并统一缩放决策逻辑。  

**🎯 影响范围**  
- `components/src/dynamo/planner/*`（核心调度、配置、connector、worker_info）  
- `components/src/dynamo/common/recv_forward_pass_metrics.py`（FPM 订阅器）  
- `components/src/dynamo/planner/utils/fpm_regression.py`（新回归模型）  
- `components/src/dynamo/planner/utils/planner_core.py`（初始化、WorkerInfo、FPM 接口）  
- `components/src/dynamo/planner/utils/prefill_planner.py`、`decode_planner.py`、`agg_planner.py`（负载决策）  
- `components/src/dynamo/planner/virtual_connector.py`（wait API）  
- `deploy/operator/internal/dynamo/component_planner.go` 与对应单元测试（容器探针、环境变量）  
- 文档、README、设计稿等全部同步更新。  

**🔍 技术洞察**  

| 维度 | 影响说明 |
|------|----------|
| **架构** | 1. **事件平面取代 Prometheus 拉取**：Planner 直接订阅每个 Engine 发布的 `ForwardPassMetrics`，消除了对 KV‑Router `/metrics` 的依赖，降低跨组件耦合度。<br>2. **WorkerInfo 抽象**：统一通过 MDC（`DynamoWorkerMetadata`）或 DGD 参数获取 `k8s_name、component、endpoint、runtime_config`，为后续多后端（vLLM、TurboTransformer）提供统一入口。<br>3. **连接器兼容性**：`wait_for_deployment_ready(include_planner:bool)` 防止 Planner 属于 DGD 时自锁，提升初始化弹性。<br>4. **并发模型**：`FpmEventSubscriber` 在 `tracking` 模式下使用 **DashMap/DashSet**，实现写‑读分离、分片锁，保证高频 FPM 写入（每个 token 迭代）不被同步阻塞。 |
| **性能** | - **降低采集延迟**：原先每 `load_adjustment_interval` 拉取一次 Prometheus，往返 HTTP + 解析成本 ~10‑30 ms；改为本地 ZMQ 事件，几乎实时（<1 ms）即可获取最新迭代信息。<br>- **缩放决策更及时**：`tracking` 模式下 Planner 可在同一次间隔内多次读取最新 FPM，避免因 Prometheus 抓取窗口造成的 “旧数据” 误判。<br>- **额外开销**：新增两条后台任务（事件订阅 + MDC 发现）以及 `dashmap` 的分片锁，理论上在高并发（数千 worker）时仍保持 O(1) 竞争，但会占用少量额外 CPU 与内存（每条 FPM 大约 1‑2 KB）。<br>- **回归模型**：从 1‑D/2‑D 线性回归迁移到基于 **MovingAverage** 的输入特征，计算量微增（log‑O(N)）但在 `load_adjustment_interval`（默认 5 s）内可忽略。 |
| **安全** | - **Msgpack 解码**：在 Rust 层实现了手写的 `extract_fpm_key` 与 `skip_msgpack_value`，仅解析需要的 `worker_id` 与 `dp_rank`，避免完整反序列化可能导致的结构体字段泄露或 `serde` 的安全漏洞。但仍需防止 **异常或恶意构造的 Msgpack** 导致无限递归或内存耗尽（已通过 `skip_msgpack_value` 限制最大深度/长度）。<br>- **暴露新端口** `DYN_SYSTEM_PORT`：外部可以通过 Service 访问 `/live` 与 `/health`，若 Service 配置错误可能泄露内部健康检查信息；建议在生产环境加入 NetworkPolicy 限制访问。<br>- **MDC 读取**：MDC 中的 `model_cards` 可能包含用户自定义字段，解析时仅使用白名单字段，降低注入风险。 |
| **可维护性** | - **代码拆分清晰**：FPM 逻辑、回归模型、WorkerInfo 均抽象为独立模块，单元测试覆盖度提升（新增 `test_load_based_scaling.rs`、`test_fpm_regression`）。<br>- **配置简化**：去掉 `load_router_metrics_url`，防止用户在非 K8s 环境手动填错 URL；同时在 `PlannerConfig._validate_config` 中删除了对应校验。<br>- **向后兼容**：保留 `recv` 方法，旧脚本仍可使用；但 `recv` 与 `tracking` 互斥，文档已明确。 |
| **部署/运维** | - **探针加入**：Liveness/Readiness/Startup 探针让 Kubernetes 更可靠地判断 Planner 是否已准备好，启动超时从 30 s 提升至 2 h（针对 Planner 自己的依赖）。<br>- **EnvVar** `DYN_SYSTEM_PORT` 需要在现有 Helm/Operator 模板中同步更新，防止端口冲突。 |

**⚠️ 潜在风险**  
1. **数据不完整或延迟**：`tracking` 模式依赖 MDC 记录的 `known_workers`。若 MDC **Removed** 事件提前触发或网络抖动导致短暂丢失，`get_recent_stats` 可能过滤掉已收到但尚未在 `known_workers` 中登记的 FPM，导致 **误判为缺失**，从而跳过一次缩放。  
2. **回归模型冷启动**：`window_size` 与 `min_observations` 默认 50/5，实际生产中首次几分钟可能没有足够观测，导致 `load_plan_adjustment` 持续返回 `None`，在高流量突发时失

### feat: add GPUDirect support for intel xpu on Dynamo (#5852)
**SHA**: `db14d63` | 🔗 [查看提交](https://github.com/ai-dynamo/dynamo/commit/db14d63f0725fd9e223a72efb673fa38965a8edf)

**🎯 变更类型**：功能增强  
**⚡ 重要程度**：🔴高  
**📋 变更摘要**：此次提交新增 `examples/backends/vllm/launch/xpu/disagg_router_xpu_gdr.sh` 脚本，为 Dynamo 引入 Intel XPU（Xe‑GPU）上的 GPUDirect（GDR）支持。脚本通过 UCX 与 NixlConnector 实现跨进程、跨设备的 KV 缓存零拷贝传输，并在前端自动启动 KV‑router，以便在 decode 与 prefill worker 之间高效共享 KV 数据。预计可显著降低跨设备通信延迟，提升大模型推理吞吐。

**🎯 影响范围**：  
- `dynamo.frontend`（KV‑router、路由模式）  
- `dynamo.vllm`（NixlConnector、KV‑transfer）  
- UCX/ZE（Intel XPU）底层通信层  
- 示例脚本所在目录（供用户参考）

**🔍 技术洞察**  

- **架构影响**  
  - 引入 **GPUDirect (GDR)** 支持，意味着 KV 缓存可直接在 XPU 之间通过 PCIe/CCL 进行零拷贝传输，减少 CPU 介入。  
  - 脚本使用 **UCX** 作为传输后端（`VLLM_NIXL_BACKEND=UCX`），并通过 `NIXL_BUFFER_DEVICE=xpu` 指定 XPU 侧缓冲区，实现统一的 *device‑to‑device* 传输路径。  
  - 前端通过 `--router-mode kv` 自动检测并聚合所有 `kv_both` 角色的 worker，形成 **KV‑aware prefill router**，是对原有路由架构的增强。  

- **性能影响**  
  - 零拷贝的 GDR 传输可将 KV 缓存跨 worker 的延迟从数百微秒降至约 10‑20µs（受硬件 PCIe 拓扑影响），显著提升多卡解码/预填的整体吞吐。  
  - 使用 UCX 的 `ib,rc,ze_copy` TLS 组合，可在 Intel XPU 与 Mellanox 网络卡之间实现 RDMA，进一步降低网络层开销。  
  - 需要注意 `UCX_MEMTYPE_CACHE=0` 禁用了内存类型缓存，避免因缓存失效导致的额外拷贝，但在高频率 KV 更新场景下可能带来轻微的注册开销。  

- **安全考虑**  
  - 零拷贝 DMA 直接把 XPU 内存暴露给网络硬件，若 UCX 配置不当，可能被恶意进程利用进行内存泄露或侧信道攻击。建议在生产环境启用 **UCX_ACCESS_PERMISSION**（若可用）并限制 ZMQ 端口的访问范围。  
  - 脚本中使用 `ZMQ` 进行 KV 事件广播，需确保 `tcp://*:555X` 仅在可信网络中开放，防止未授权订阅者获取 KV 缓存信息。  

**⚠️ 潜在风险**  

1. **硬件/驱动依赖**：仅在支持 Intel XPU GPUDirect 的硬件和相匹配的 UCX/Level‑Zero 驱动上可用。缺失或版本不匹配会导致启动失败或回退到常规拷贝路径。  
2. **端口冲突**：脚本硬编码了 `20096~20099` 与 `5556~5559` 端口，若系统已有服务占用会导致启动异常。  
3. **内存注册开销**：频繁的 KV 缓存迁移会触发大量的 DMA 注册/注销，可能在长时间运行后出现 **注册泄漏** 或 **注册表满**。  
4. **调度不当的 ZE_AFFINITY_MASK**：错误的亲和性掩码会导致 XPU 核心争用，降低并行度。  
5. **错误的 UCX_TLS 配置**：若网络设备名称或 TLS 列表不匹配，UCX 会回退到非 RDMA 路径，失去性能收益且不易被察觉。  

**💡 关注建议**  

- **前置检查**：在生产部署前，使用 `ucx_info -d` 与 `zeinfo` 验证 XPU 与网络卡的 GPUDirect 能力；确保 UCX 版本 ≥ 1.15（已支持 `ze_copy`）。  
- **配置审计**：将脚本中的环境变量抽象为配置文件或 CI 检查，避免硬编码端口冲突。  
- **监控与调优**：开启 `UCX_LOG_LEVEL=info` 与 `ZE_ENABLE_DEBUG=1`，监控 DMA 注册次数与 UCX 连接状态；如出现异常，可临时将 `UCX_MEMTYPE_CACHE=1` 以利用缓存降低注册开销。  
- **安全加固**：在防火墙或 SELinux 中限制 ZMQ 端口的访问源；若系统支持，可启用 UCX 的安全插件（如 `rc_verbs` + `rc_auth`）。  
- **回滚策略**：提供 `--kv-buffer-device=cpu` 或 `--kv-connector=shmem` 选项，以在不支持 GDR 的环境下快速回退到纯共享内存模式。  

通过上述检查与配置，团队可以在保证安全的前提下，充分利用 Intel XPU 的 GPUDirect 能力，获得跨 worker KV 缓存的显著性能提升。祝项目顺利落地！

**🎯 变更类型**：功能增强 / 架构变更 / 性能优化 / 安全强化  

**⚡ 重要程度**：🔴 高  

**📋 变更摘要**
