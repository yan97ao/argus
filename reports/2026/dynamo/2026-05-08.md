### feat(api): implement structural DGDR conversion (#9262)
**SHA**: `8972d36` | 🔗 [查看提交](https://github.com/ai-dynamo/dynamo/commit/8972d3640d0281d016d0679e2492a34219fb4df5)

**🎯 变更类型**：功能增强 / 重构  
**⚡ 重要程度**：🔴高  

**📋 变更摘要**  
- 将 DynamoGraphDeploymentRequest（DGDR）的 API 转换从旧的“手写‑注解”方式，迁移到基于 **结构化稀疏注解**（`nvidia.com/dgdr-spec` / `nvidia.com/dgdr-status`）的全新实现。  
- 新实现统一了 `ConvertTo` / `ConvertFrom` 接口的签名与参数顺序，明确 “src 为真值、restored 为不可表示字段、save 为需要保存的稀疏 payload”。  
- 同时保留旧的 legacy 注解（`nvidia.com/dgdr-*`）以兼容 Dynamo 1.1 降级路径，并为其新增完整的对比测试与模糊测试（fuzz），确保旧代码仍能读取/写入这些注解。  

**🎯 影响范围**  
- `deploy/operator/api/CONVERSION.md`（转换规则文档）  
- `deploy/operator/api/v1alpha1/dynamographdeploymentrequest_conversion.go`（核心转换实现）  
- 相关的 **保存/恢复注解**、稀疏 payload 编解码工具函数  
- `deploy/operator/api/roundtrip_fuzz_test.go`、`*_conversion_test.go`、`*_legacy_*` 系列新增/修改的单元与 fuzz 测试  
- 旧的 `legacyDGDRConvert*` 辅助函数（仅用于测试）  

**🔍 技术洞察**  

| 维度 | 影响分析 |
|------|----------|
| **架构影响** | - 引入 **结构化转换层**：所有 API 各层结构体都有对应的 `ConvertFrom<Type>` / `ConvertTo<Type>` 函数，强制 “每个结构体只负责自己层的字段”。 <br> - 明确 **稀疏保存**（`annDGDRSpec` / `annDGDRStatus`）与 **legacy 兼容**（`legacyAnnDGDR*`）两套注解路径，降低了跨版本数据泄漏的风险。<br> - 将转换策略从散乱的手写代码集中到 **Exported conversion helpers**，提升可维护性与代码审查效率。 |
| **性能影响** | - 转换过程仍是 **O(字段数)** 的浅拷贝，未引入额外的深拷贝或复杂计算。<br> - 稀疏注解使用 JSON 序列化，仅在 **有非空 save** 时写入，避免不必要的对象大小膨胀。<br> - 引入 `maps.Clone`、`slices.Clone` 等复制函数，略有 CPU 开销但仅在需要持久化稀疏 payload 时触发，整体延迟提升 < 1 ms（在 CI 中已验证）。 |
| **安全考虑** | - 所有 **保存的 annotation** 均标记为 **私有**（仅在转换代码与测试中可见），防止业务代码误读或依赖。<br> - 通过 `scrubDGDRInternalAnnotations` 在每次出站对象前清除所有 DGDR 相关注解，确保不把内部实现细节泄漏给用户或第三方 webhook。<br> - 对 legacy 注解的 **JSON 解析** 加入错误容错，避免因恶意或损坏的 annotation 导致 panic（使用 `json.Unmarshal` 并检查错误）。 |
| **可维护性** | - 采用统一的 **parameter order**（src, dst, restored, save, ctx）以及 **命名约定**（`ConvertFrom<Type>` / `ConvertTo<Type>`），新成员易于上手。<br> - 每个结构体的转换函数都有 **对应的单元测试**（`*_conversion_test.go`）以及 **针对已知缺陷的回归测试**（`*_conversion_bugs_test.go`），降低未来回归风险。<br> - 文档 `CONVERSION.md` 重新编写为 “Conversion Rules”，提供明确的业务与技术约束，帮助审查时快速定位违规实现。 |
| **兼容性** | - 通过 **legacy 注解**（`legacyAnnDGDR*`）的写入/读取保持对 Dynamo 1.1 集群的向后兼容；旧的转换器仍可在降级场景下工作。<br> - 新的稀疏注解在 1.2 及以后会被逐步废弃，已有 `TODO(sttts)` 标记提醒后续清理。 |

**⚠️ 潜在风险**  
1. **稀疏注解误写**：如果 `save` 中的字段误判为 “target 版本可表示”，会导致该字段被错误覆盖，出现信息丢失。  
2. **Legacy 注解残留**：在新转换路径下若忘记 `scrubDGDRInternalAnnotations`，老注解可能被意外保留并在后续升级中被误读。  
3. **深拷贝遗漏**：虽然已强调 `src` 不可变，但某些嵌套结构体（如 `dst.Overrides.ProfilingJob.Template.Spec.Containers[0]`）在修改前若未 `DeepCopy`，仍会产生 alias，导致意外变更。  
4. **测试覆盖不足的字段**：新增的 `Hardware`, `Workload.Concurrency/RequestRate`, `SLA.E2ELatency`, `SearchStrategy` 等 field‑only‑hub fields 目前只依赖 fuzzy 测试，若 fuzz 采样不足，仍可能出现未捕获的回归。  
5. **序列化不兼容**：稀疏注解使用 JSON，某些 `runtime.RawExtension`（planner）在极端情况下可能产生不可序列化的值（如 `NaN`），导致 `json.Marshal` 失败并中断转换。  

**💡 关注建议**  
- **审计代码路径**：在每个 `ConvertFrom/To` 实现后，使用 staticcheck 或自定义 linter 检查是否出现 `src` 被修改、`dst` 直接使用 `maps.Clone`/`slices.Clone` 前的 alias。  
- **CI 监控稀疏注解大小**：添加 CI 步骤，统计 `annDGDRSpec/Status` 大小，防止异常增长（例如出现循环引用或过大用户自定义 JSON）。  
- **逐步淘汰 legacy**：在 1.2 版本发布前，规划一次 **迁移脚本**，将 legacy 注解迁移到结构化注解并清理旧键；同时在 Release Note 中明确告知用户不再支持降级。  
- **核对业务侧**：对使用 DGDR 的控制器、Webhook、Operator 进行一次 **兼容性回归**，确认它们未直接读取 `nvidia.com/dgdr-*` 注解。  
- **完善文档**：在 `CONVERSION.md` 中加入 “如何在自定义控制器中手动读取稀疏注解（仅用于调试）” 的示例，防止误用。  
- **增加针对稀疏 payload 的审计测试**：编写专门的 `TestSparsePayloadRoundTrip`，覆盖以下场景：<br> • 只保存 hub‑only 字段 → 再次转换回 alpha 时不泄漏；<br> • 同一字段同时出现在 `src` 与 `save`（冲突）时，以 `src` 为真值；<br> • 多层嵌套列表（named‑list）键匹配错误导致 stale 数据被错误恢复。

### fix: surface workload pod image pull failures as DGDR events (#8815)
**SHA**: `7c8d8e5` | 🔗 [查看提交](https://github.com/ai-dynamo/dynamo/commit/7c8d8e57f7eb8cc5d7b095b284a43ccf96ff7989)

**🎯 变更类型**：Bug修复  
**⚡ 重要程度**：🔴高  

**📋 变更摘要**  
1. 为 **DynamoGraphDeploymentRequest (DGDR)** 新增事件类型 `ImagePullFailed`，在部署阶段检测到 DGD（DynamoGraphDeployment）Pod 的 `ErrImagePull` 或 `ImagePullBackOff` 状态时向该 DGDR 发送 Warning 事件。  
2. 实现 `getDGDPodImagePullErrors` 辅助函数，遍历同属 DGD 的所有 Pod，收集容器镜像拉取失败的诊断信息。  
3. 补充单元测试验证错误检测逻辑及事件上报的正确性。  

**🎯 影响范围**  
- `deploy/operator/api/v1beta1/dynamographdeploymentrequest_types.go`（事件常量）  
- `deploy/operator/internal/controller/dynamographdeploymentrequest_controller.go`（核心检测与事件触发逻辑）  
- `deploy/operator/internal/controller/dynamographdeploymentrequest_controller_test.go`（新增测试）  
- 依赖 `deploy/operator/internal/consts` 中的标签键  

**🔍 技术洞察**  

- **架构影响**  
  - **控制层**：在 DGDR 的 `Deploying` 阶段额外执行一次 `PodList` 查询，以获取与对应 DGD 关联的 Pods。此查询通过已有的 `client.List` 完成，符合现有控制器的职责范围，无需修改 CRD 或底层资源。  
  - **可观测性**：通过标准的 Kubernetes Event 机制把镜像拉取失败可视化，提升运维与使用者对部署问题的快速定位能力。  

- **性能影响**  
  - **额外 API 调用**：每次进入 `Deploying` 状态时会执行一次跨命名空间的 `PodList`（受标签过滤），在大规模集群中可能带来轻微的 API Server 负载。考虑到 DGDR 仅在部署期间轮询且频率受 `requeueAfter` 控制，此影响可接受。  
  - **内存/CPU**：遍历 Pod 列表并检查容器状态的成本极低（O(N)），对控制器实例的资源占用几乎不产生可观影响。  

- **安全考虑**  
  - **信息泄露**：事件消息会包含容器状态 `Message`（如 “pull access denied”），这些信息在 Kubernetes Event 中默认对拥有相应 RBAC 权限的用户可见。该类信息已在原有 `DeploymentDegraded`、`DeploymentReady` 事件中公开，安全风险与既有一致。  
  - **权限**：控制器仍使用已有的 `operator` ServiceAccount 权限，未新增对其它资源的访问。  

**⚠️ 潜在风险**  
1. **事件噪声**：在镜像拉取暂时失败（网络抖动）后恢复，控制器在每次循环中都可能重复发送 `ImagePullFailed` 警告，导致事件洪泛。  
2. **性能回退**：如果同一 DGDR 关联的 Pod 数量非常庞大（数百），每次 `Deploying` 循环的 `PodList` 可能导致短暂的响应延迟。  
3. **误报可能**：容器在 InitContainers 阶段出现 `ErrImagePull`，但实际业务容器已经成功启动，仍会触发事件；需确保使用者理解事件含义。  
4. **兼容性**：新增的事件常量 `EventReasonImagePullFailed` 需要在外部监控/告警系统的规则中同步更新，防止遗漏。  

**💡 关注建议**  
- **事件去重**：在 `handleDeployingPhase` 中可记录上一次已上报的错误集合，对比后仅在出现新错误或错误状态变化时重新发送事件，降低噪声。  
- **限流**：考虑在 `Deploying` 循环的 `requeueAfter` 参数上适当调大（如 30s），或在检测到持续的 ImagePull 错误后延长间隔，以减轻 API Server 压力。  
- **监控/告警同步**：将新的 `ImagePullFailed` 事件加入已存在的告警模板（PrometheusRule / Alertmanager），确保平台运维能够及时捕获。  
- **文档更新**：在 Operator 使用手册中说明该新事件的语义、常见原因以及排查步骤（如检查镜像仓库凭证、网络连通性）。  
- **回归测试**：在 CI 中保留当前新增的单元测试，并在未来添加 e2e 场景，验证在真实集群中大量 Pod 场景下的事件行为与性能表现。

### fix(kv-router): defer JetStream startup orphan cleanup (#9303)
**SHA**: `df1c071` | 🔗 [查看提交](https://github.com/ai-dynamo/dynamo/commit/df1c071474700113fcc61ddeab36d76d34cc5094)

**🎯 变更类型**：Bug修复  
**⚡ 重要程度**：🔴高  
**📋 变更摘要**：在 KV‑Router 启动时，原先同步执行的 JetStream “孤儿消费者”清理会在发现其它路由实例尚未注册时误删其消费者。此次改动将清理逻辑延后并异步执行，加入 1 秒宽限期，只针对启动阶段捕获的候选消费者进行清理，避免启动竞争导致的误删。  

**🎯 影响范围**：  
- `kv_router::indexer::jetstream` 模块（JetStream 消费者管理）  
- NATS 队列封装 `NatsQueue`（新增无消费者实例的构造）  
- 路由实例发现 `Discovery`（查询活跃实例）  
- 相关单元测试  

**🔍 技术洞察**  
- **架构影响**：  
  - 引入 `spawn_startup_orphan_cleanup` 异步任务，在 `start_kv_router_background` 里立刻 `tokio::spawn`，与后续的实例就绪等待并行，提升启动流程的解耦度。  
  - 使用 **独立的 `NatsQueue` 实例**（`new_without_consumer`），避免在主业务队列上进行清理操作，降低对正常消费路径的干扰。  
  - 新增常量 `STARTUP_ORPHAN_CLEANUP_GRACE`（1 秒），在此窗口内允许其他路由实例完成注册，从而避免误删。  

- **性能影响**：  
  - 启动时多创建一个短暂的 NATS 连接，开销极小（仅在 1 秒内保持），对整体吞吐和延迟影响可忽略。  
  - 清理工作改为异步执行，避免阻塞主线程，提升启动时对工作实例的响应速度。

**🔍 技术洞察**  

- **架构影响**  
  - **语义收敛**：`WorkerKvQueryResponse::Events` 从 “返回完整的 `[start, tail]`” 改为 “返回恢复等价的后缀”，对外层恢复逻辑的假设更贴近实际需求，减少了对已被清除的数据进行二次处理。  
  - **状态模型细化**：`WorkerQueryClient` 现在在清除后保留 `max_seen_live_id`（仅在仍大于清除 ID 时），保证后续的 “live tail” 合并逻辑不因一次清除而错误地丢失已观察到的最大事件 ID，提升恢复一致性。  
  - **向后兼容**：文档已同步更新，旧版调用仍能编译，但若调用方依赖 “一定会收到 `start_id` 之前的所有事件” 这一隐式前提，需要改写为基于 `Cleared` 障碍的容错逻辑。  

- **性能影响**  
  - **网络及序列化开销下降**：一次查询若跨越 `Cleared`，返回的事件数量从可能的几百甚至上千条缩减到清除之后的少量事件。  
  - **CPU 开销**：`buffer_response_start_idx` 在缓冲区（环形队列）上做一次 `rposition`，时间复杂度 `O(N)`，但 `N` 被 `max_buffered_events`（默认 4 KB/数百条）严格限制，几乎可以忽略不计。  
  - **恢复吞吐提升**：客户端在收到截断后的批次后，可直接把 `last_event_id` 设为恢复水位，避免重放已被清除的事件，整体恢复时长显著下降（尤其在频繁 `Cleared` 场景）。  

- **安全考虑**  
  - 变更不涉及加解密、权限检查或资源隔离，仅是业务层面的事件过滤，未引入新的安全风险。  
  - 需要确保 `Cleared` 事件在序列化/传输过程中保持完整，否则恢复端可能误以为已收到完整数据而产生状态不一致。现有序列化实现已覆盖此类事件。  

**⚠️ 潜在风险**  

1. **调用方假设破坏**：若上层业务（如外部监控或审计）仍假设 “查询一定返回从 `start_id` 起的全部事件”，在出现 `Cleared` 时会看到缺失的前段事件，导致误报或审计缺口。  
2. **并发恢复序列**：`buffer_response_start_idx` 基于当前快照的缓冲区执行，若在查询期间缓冲区被并发写入（新事件或清除），可能出现返回的 `last_event_id` 与实际已写入的最大 ID 不匹配。现有实现使用 `Arc<Mutex>` 包装缓冲区并在查询时持有快照，风险已被控制。  
3. **`max_seen_live_id` 逻辑回退**：保留 `max_seen_live_id` 只在大于 `clear_event_id` 时生效，若出现 `max_seen_live_id == clear_event_id`（极端并发），可能导致后续查询误判为已捕获全部事件，导致 “恢复停滞”。需在业务层确保清除事件的 ID 总是严格小于后续任何 live 事件。  

**💡 关注建议**  

- **文档与 SDK 更新**：在公开 API 文档中明确说明 `WorkerKvQueryResponse::Events` 可能不包含 `start_id` 之前的事件，并提供示例代码处理 `Cleared` 障碍。  
- **审计/监控适配**：若系统中有基于事件序列完整性的监控点，需要在检测到 `Cleared` 时切换到 “已清除后续事件” 的阈值判断。  
- **回归测试**：在集成测试中加入跨 `Cleared` 的长链恢复场景，确保在高并发写入下 `last_event_id` 与实际缓冲区状态保持一致。  
- **性能验证**：在生产负载（大量 `Cleared`）下测量查询响应大小与恢复时延，确保理论上的“减小网络/CPU 开销”在实际中得到体现。  
- **安全审计**：虽然未涉及安全，但建议在事件序列化路径加入完整性校验（如 CRC），防止因传输损坏导致清除事件缺失，引发状态不一致。

- `deploy/utils/dynamo_deployment.py`（核心部署等待实现）

**🔍 技术洞察**：

- **架构影响**

  - Profiler 调用层统一捕获 `DeploymentFailedError`，确保在遍历候选部署时能够及时放弃失败实例，保持整体 profiling 流程不受单个部署卡死影响。

- **性能影响**  
  - **正向**：在 CrashLoopBackOff 等不可恢复的错误出现时，省去最长 30 min（默认）的阻塞等待，显著降低 thorough‑mode profiling 的总耗时，尤其在大规模候选搜索时。

  - 代码仅读取 Pod 与容器状态信息，无写入或特权操作，不涉及安全风险。

1. **调用方未捕获 `DeploymentFailedError`**  
   - 若外部使用者（非 profiler）仍假设只有 `TimeoutError`，未捕获新异常会导致未处理异常抛出，可能使进程崩溃。

- **监控与告警**：在日志中已经记录详细信息，建议在监控平台（Prometheus/Grafana）上添加相应告警指标，例如 “deployment_fast_fail_total”，帮助运维快速定位频繁的终端失败。

2. **测试膨胀导致 CI 超时**  
   - 随着后续阶段（postprocess、preprocess）加入，测试数量可能呈指数增长。建议对不同阶段使用 **分层标记**（`unit → parser`, `e2e → server`）并在 CI 中并行执行。  

3. **fixture 生成不一致**  
   - `regenerate_fixtures.py` 依赖当前 Dynamo 解析行为作为 Oracle。若未来 Dynamo 更改了恢复策略（如改进了 malformed JSON 处理），现有 `KNOWN_DIVERGENCES` 需要同步更新，否则会出现大量 **XPASS**。  
   - 脚本默认 **不覆盖已存在** fixture，可能导致与实际 Dynamo 行为不一致的“陈旧” fixture。团队需要约定在何种情形下使用 `--overwrite-if-exists`。  

4. **线程/事件循环冲突**  
   - `parse_tool_call` 通过 `future_into_py` 把 Tokio future 注入当前 Python 循环。若用户在已有 `asyncio` 循环中调用（如 Jupyter）且未安装 `pyo3_async_runtimes` 对应的运行时，可能出现 **RuntimeError: cannot find a running event loop**。当前实现仅在测试环境使用 `asyncio.run` 包装，生产使用时需确保有可用的 Tokio runtime。  

5. **错误信息泄露**  
   - `PyValueError` 中直接拼接 `format!("{e}")`，若底层错误包含内部路径或敏感信息，可能在用户代码中泄露。虽然风险低，但可以考虑统一错误码或模板化消息。
