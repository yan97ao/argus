### fix: predownload LoRA adapter so offline-mode tests find it in the HF cache (#9207)
**SHA**: `d61b728` | 🔗 [查看提交](https://github.com/ai-dynamo/dynamo/commit/d61b72815f65615a7a31aba929123af383a7dad0)

**🎯 变更类型**：Bug修复  
**⚡ 重要程度**：🔴高  

**📋 变更摘要**  
- 通过在 pytest 标记 `@pytest.mark.model` 中遍历所有模型标记并收集模型 repo‑id，修复了离线模式下 LoRA 适配器无法被预下载的问题。  
- 将 LoRA 的下载实现从临时目录拷贝改为直接从本地 HuggingFace 缓存 (`snapshot_download(..., local_files_only=True)`) 解析快照路径，消除了对 `HF_HUB_OFFLINE` 环境变量的临时切换。  
- 为相关测试用例新增了对额外 LoRA 仓库的标记，确保离线测试能够正确定位该适配器。  

**🎯 影响范围**  
- `tests/conftest.py`（模型标记收集逻辑）  
- `tests/serve/lora_utils.py`（LoRA 下载/解析实现）  
- `tests/serve/test_vllm.py`（新增 LoRA 适配器的测试标记）  
- 仅限测试套件和 CI 环境，运行时库功能不受影响。  

**🔍 技术洞察**  
- **架构影响**：  
  - 仅涉及测试层的 fixture 与工具函数，未改动核心业务代码，保持了库的模块化边界。  
  - 通过 `item.iter_markers("model")` 统一收集所有模型标记，提升了标记解析的通用性，未来可 seamlessly 支持多个模型标记。  

- **性能影响**：  
  - 直接返回缓存快照路径，避免了创建临时目录、复制文件及后续清理的 I/O 开销，离线模式下测试启动更快。  
  - 减少了对环境变量 `HF_HUB_OFFLINE` 的切换，降低了潜在的并发冲突风险。  

- **安全考虑**：  
  - 只读取本地 HF 缓存 (`local_files_only=True`)，不触发任何网络请求，符合离线安全需求。  
  - 未引入新的外部依赖或权限提升，安全风险基本为零。  

**⚠️ 潜在风险**  
- **缓存依赖**：若 CI 或本地机器的 HF 缓存中未预先存在对应 LoRA 仓库，`snapshot_download` 将抛出 `FileNotFoundError`，导致相关测试失败。  
- **路径使用假设**：`download_lora` 现在返回的是快照目录而非临时复制目录，若其他测试代码（或未来的工具）仍假设返回的是可写的临时目录，可能产生写入权限错误。  
- **标记兼容性**：改用 `iter_markers` 收集模型标记后，若旧的自定义标记实现依赖单一 `model` 标记的 `args` 结构，可能出现兼容性问题。  

**💡 关注建议**  
1. **确保预下载**：在 CI 前置步骤或本地开发环境中，使用 `predownload_models` fixture 明确列出所有 LoRA repo‑id，保证缓存完整。  
2. **容错处理**：在 `download_lora` 内部加入缓存缺失的友好提示或回退逻辑（如 `pytest.skip`），避免因单个缺失模型导致整套测试中断。  
3. **文档更新**：在项目的测试贡献指南中说明新增的 `@pytest.mark.model` 支持多值传递，以及离线模式下对 HF 缓存的依赖。  
4. **回归验证**：运行完整的离线测试矩阵，确保未因路径返回变化而影响到其他测试用例（尤其是涉及文件写入的 fixture）。  
5. **监控变更**：若后续在业务代码中复用 `download_lora`，需评估是否仍需要返回可写的临时目录，必要时提供参数化选项以兼容两种使用场景。

### fix(operator): add v1beta1 conversion fuzz coverage (#9164)
**SHA**: `b3afdf6` | 🔗 [查看提交](https://github.com/ai-dynamo/dynamo/commit/b3afdf6a8b4ecdb7a37ee6dc7d196aa56a9a465a)

**🎯 变更类型**：Bug修复 / 性能优化 / 重构  
**⚡ 重要程度**：🔴高  
**📋 变更摘要**：  
- 为 `DynamoGraphDeploymentRequest`（DGDR）新增完整的 round‑trip fuzz 测试，覆盖 hub↔spoke 两向转换并加入对内部注解的过滤。  
- 在转换实现中对 `ObjectMeta` 做深拷贝、引入统一的 `dgdrDeploymentStatusAnnotation` 结构以在注解中保存部署状态及请求状态，并实现安全的反序列化与校验。  
- 更新保留注解前缀列表、提供 `oneOf` 辅助函数、优化 diff 选项及清理旧代码，配套单元测试验证新实现的正确性和向后兼容性。

**🎯 影响范围**：  
- `deploy/operator/api`（核心 CRD 转换层）  
- 相关 fuzz 测试文件 `roundtrip_fuzz_mutation_test.go`、`roundtrip_fuzz_test.go`  
- `v1alpha1/dynamographdeploymentrequest_conversion.go` 与对应的单元测试  
- 任何使用 DGDR CR 的 Operator 部署与升级路径

**🔍 技术洞察**：

| 维度 | 影响 |
|------|------|
| **架构影响** | - 引入 `dgdrDeploymentStatusAnnotation` 结构，将 **alpha → beta → alpha** 的状态信息通过注解安全持久化，提升了转换层的**可追溯性**。<br>- 深拷贝 `ObjectMeta` 防止在多版本对象间意外共享指针，改善了对象生命周期管理。<br>- 扩展 `reservedAnnotationPrefixes`，显式声明 `dgdr` 前缀，保持注解命名空间的一致性。 |
| **性能影响** | - 运行时影响极小：仅在 CR 转换时进行一次 JSON 编解码和一次浅拷贝。<br>- 测试层面新增约 350+ 行 fuzz 测试，将 CI 运行时间略有提升（约 10–20%），但对生产代码无直接性能劣化。 |
| **安全考虑** | - `restoreDGDRDeploymentStatus` 对注解内容进行 **结构化校验**（合法的 `DGDRState`、DGD 名称匹配、相位一致性），防止恶意或损坏的注解导致状态错乱或信息泄露。<br>- 通过 `cmpopts.AcyclicTransformer` 忽略内部 `dgdr-` 注解的比较，避免因未预期字段导致误判。 |
| **可维护性** | - 增加统一的 `oneOf` 辅助函数，简化随机化 fuzz 数据生成。<br>- 测试代码对不同转换路径使用统一的 `fuzzHubSpokeHub` / `fuzzSpokeHubSpoke`，提升后续扩展的可复用性。<br>- 对旧的注解格式进行回退检查，保持向后兼容，降低升级风险。 |

**⚠️ 潜在风险**：

1. **旧集群兼容性**  
   - 已存在的 DGDR 对象若仅携带 **legacy** 注解（仅 `DeploymentStatus`，无 `requestState`），在新代码路径下会被 **忽略**（返回 `ok == false`），可能导致状态恢复失败。  
   - 迁移期间需要确保已有对象在升级后仍能正常恢复，或提供迁移脚本/兼容层。

2. **注解体积增长**  
   - 将 `requestState` 与完整 `DeploymentStatus` 统一序列化后写入注解，注解大小上限（256KB）基本不受影响，但在极端大量字段的 `DeploymentStatus` 场景下需监控。

3. **测试依赖随机性**  
   - Fuzz 测试使用随机填充，若出现不可重复的 flaky 失败，可能导致 CI 不稳定。建议锁定随机种子或在 CI 中保存失败的输入用于回归。

4. **误用 `cmpopts.AcyclicTransformer`**  
   - 该 transformer 只在测试中使用，若误将其泄露至生产代码（例如通过公共 API）可能导致注解过滤被错误使用。

**💡 关注建议**：

- **升级指引**：在 Operator 版本升级文档中注明对 DGDR 注解格式的变更，建议在升级前执行一次 `kubectl get dgdr -o json | jq` 检查是否存在仅旧格式的注解，必要时手动补齐 `requestState`（可使用 `kubectl annotate`）。
- **回退兼容**：考虑在转换逻辑中对 legacy 注解做一次 **自动迁移**（在检测到旧格式且符合当前对象状态时，重新写入新的结构），以减轻用户手动修复负担。
- **监控告警**：在 Operator 中增加日志/指标，记录每次 `restoreDGDRDeploymentStatus` 的 `ok == false` 场景，帮助快速定位仍残留的不兼容对象。
- **测试治理**：将 fuzz 测试的随机种子固定在 CI 中的构建变量，以便 reproducing 失败；同时保持 `go test -run TestFuzzRoundTrip* -count=1 -timeout=20m` 的超时阈值，防止 CI 被卡死。
- **文档更新**：在 CRD 文档中明确 `nvidia.com/dgdr-*` 注解的语义与存储方式，帮助开发者理解注解背后的状态同步机制。

---  
**结论**：本次提交显著提升了 DGDR 在不同 API 版本之间的转换可靠性，修复了注解泄漏及状态不一致的历史缺陷，同时通过严格的校验和回退机制降低了安全风险。只要在升级时注意旧注解的兼容处理，即可安全推广到生产环境。

### feat(operator): Migrate DGSA controller to v1beta1 (#9194)
**SHA**: `737d41e` | 🔗 [查看提交](https://github.com/ai-dynamo/dynamo/commit/737d41e32c317d181ec70381c420523166849112)

**🎯 变更类型**：功能增强  

**⚡ 重要程度**：🔴高  

**📋 变更摘要**  
- 将 Operator 中与 DGSA（DynamoGraphDeploymentScalingAdapter）相关的控制器全部从 `v1alpha1` API 迁移到 `v1beta1`，并把原来的 `services` map 改为 `components` 列表结构。  
- 相应地重构了通用工具函数、状态选择器构造、Watcher 过滤逻辑以及单元测试，使其兼容新的 CRD 定义并提升了代码可读性和查询效率。  

**🎯 影响范围**  
- `deploy/operator/internal/controller/common.go`（公共工具函数）  
- `deploy/operator/internal/controller/dynamographdeploymentscalingadapter_controller.go`（核心控制器）  
- `deploy/operator/internal/controller/dynamographdeploymentscalingadapter_controller_test.go`（单元测试）  
- 依赖 `v1beta1` API 的 CRD 包 `deploy/operator/api/v1beta1`（包括 `DynamoGraphDeployment`, `DynamoGraphDeploymentScalingAdapter`）  

**🔍 技术洞察**  

| 维度 | 影响描述 |
|------|----------|
| **架构影响** | - 统一使用 `v1beta1` 版本，简化了 API 维护，避免了在同一代码库中同时维护 `alpha` 与 `beta` 两套结构。<br>- 将 `spec.services`（`map[string]*...`）转为 `spec.components`（`[]...`）后，**组件定位从键查找改为线性遍历 + 通过 `ComponentName` 索引**，配合 `GetComponentByName` 方法保持对外行为不变。<br>- `buildPodSelector` 由手写字符串改为 `labels.SelectorFromSet`，提升了标签组合的可靠性并消除了硬编码错误风险。 |
| **性能影响** | - `componentsEqual` 在比较新旧列表时先构建 `map[string]*Component`，实现 **O(N)** 的对比而非原来的 **O(N²)**（原 map 直接 O(1) 但在 `v1beta1` 中仍需一次遍历），对大规模组件列表仍保持线性性能。<br>- 在 `findAdaptersForDGD` 中去掉了 `MatchingLabels` 参数，改为在列表遍历时自行检查 `Spec.DGDRef.Name`，避免因标签不一致导致的遗漏，同时仍保持 **API‑层过滤**（只列出同命名空间的对象）。<br>- 使用 `labels.SelectorFromSet` 生成选择器字符串，降低手动拼接产生的错误并可能在后续的 List/Watch 中直接使用标签选择器，提高查询效率。 |
| **安全考虑** | - 变更仅涉及内部控制器逻辑和 CRD 结构，没有新增网络、权限或加密相关代码。<br>- 迁移到 `v1beta1` 需要确保 **CRD 升级路径** 已在集群中完成，避免因旧版对象缺失字段导致控制器 panic（已通过 `nil` 检查和默认值保护）。 |
| **可维护性** | - 代码统一使用 `v1beta1` 类型，降低理解成本。<br>- 通过抽取 `getComponentNames`、`componentsEqual` 等通用函数，提升了可读性和单元测试覆盖度。<br>- 单元测试已同步更新，覆盖了默认副本、缺失组件、标签不一致等边界情况，增强了回归防护。 |

**⚠️ 潜在风险**  
1. **CRD 版本不匹配**：如果集群仍保留 `v1alpha1` CRD，控制器将无法读取旧对象，导致资源“未找到”或字段缺失错误。  
2. **兼容性破坏**：外部用户仍在使用 `v1alpha1` YAML 部署时，会在升级后遇到字段名变化（`services` → `components`、`serviceName` → `componentName`），可能导致部署失败。  
3. **列表过滤逻辑**：`findAdaptersForDGD` 现在依赖 `Spec.DGDRef.Name` 检查而非标签过滤，若用户手动修改 `spec.dgdRef.name` 与标签不一致，仍会被正确关联，但潜在的 “stale label” 场景需要在文档中说明。  
4. **误删或误更新**：`componentsEqual` 在比较时先把新列表转为 map，如果同名组件出现多次（非法但可能的输入），后面的会覆盖前面的，导致判断错误。  

**💡 关注建议**  
- **升级指南**：在发布说明中提供 **CRD 迁移手册**，包括 `kubectl apply -f <v1beta1-crd>.yaml --server-side --force-conflicts` 以及示例 YAML 中 `services` → `components` 的转换示例。  
- **兼容层**：如果必须兼容老版本，考虑在启动时检测 `v1alpha1` CRD 是否存在，给出明确错误提示或提供临时的转换 webhook。  
- **监控与日志**：在控制器中加入 **事件/指标**，记录 “component not found” 场景，以便运维快速定位因配置错误导致的无效 scaling。  
- **单元/集成测试**：在 CI 中加入 **跨版本回滚测试**（先部署 `v1alpha1` 再升级到 `v1beta1`），确保已有集群可以平滑迁移。  
- **文档同步**：更新 `README`、API 文档以及 Helm chart 中的默认值，使其全部指向 `v1beta1` 定义。

### 性能影响  
| 维度 | 正向/负向影响 |
|------|----------------|
| **查询路径** | 多层查询 **额外遍历 0‑3 个下层索引**，每层在 `ThreadPoolIndexer` 中并行搜索。对大多数查询（仅 Device‑Tier 匹配）影响极小；对需要全链路匹配的查询会产生 **O(层数)** 的额外 CPU/内存开销。 |
| **写入路径** | 写入时只进入 **单一目标层级**（除 `Cleared` 需要广播），因此写放大成本仍在 **每层单独的后台线程池**。懒创建避免了空层级的资源浪费。 |

### feat(mocker): publish tiered KV events for G2 offload (#8961)
**SHA**: `cc141cc` | 🔗 [查看提交](https://github.com/ai-dynamo/dynamo/commit/cc141cc853682b17e2240651686dbd53b1be9e89)

**🎯 变更类型**：功能增强（在 KV 路由器中加入分层（tiered）KV 事件，以支持 G2 off‑load）  

**⚡ 重要程度**：🔴 高  

**📋 变更摘要**  
本次提交为 **mocker** 子系统引入了 “tiered KV events” 支持，使得在 G2（Host‑Pinned）层面的 KV block off‑load 能够向路由器发布 **StorageTier** 信息（Device / HostPinned / Disk / External）。为此完成了：

1. 在 `StorageTier` 中加入 `to_kv_medium`、序列化字段 `medium`（ZMQ）。  
2. 新增 `publish_with_storage_tier` 接口，所有 KV 事件发布路径统一走该接口。  
3. **kvbm‑offload** 引擎在 G1→G2 迁移完成后产生 **G2RouterEvent**（Stored / Removed），并通过 `KvManager` 再发布为 `HostPinned` KV 事件。  
4. 增加 `SwapInRegistrationBlock`、`G2BlockEventMetadata`、`G2RouterEvent` 等数据结构，记录 G2 事件所需的本地 hash 与 token‑ids。  
5. 在 replay 与 benchmark 流程中加入对 `StorageTier` 的过滤，确保离线/在线重放仅对 GPU（Device）层事件进行索引。  
6. CI、benchmark、测试代码全面改动，加入针对 G2‑offload 场景的单元与集成测试，并在 `pre‑merge` 工作流中添加对应的 cargo test / clippy 检查。  

**🎯 影响范围**  
- `lib/bench/*`（benchmark 入口、common、tests）  
- `lib/kv-router/src/protocols.rs`（StorageTier 扩展）

- `lib/mocker/src/scheduler/*`（事件捕获、延迟发布、G1 前缀 pin）  
- CI workflow (`.github/workflows/pre-merge.yml`)  

**🔍 技术洞察**  

| 维度 | 影响说明 |
|------|----------|
| **架构影响** | - 引入 **StorageTier** 概念贯通整个路由层，从 `KvEventPublisher` → `KvCacheEventSink` → `MockOffloadEngine` → `KvManager`。<br>- `MockOffloadEngine` 现在负责维护 **G2 生命周期事件流**（EventsManager + metadata map），并在事件完成时回推至路由器，形成双向路径（G1→G2 与 G2→G1）。<br>- 为防止 G1 前缀在 G2 迁移期间被淘汰，`KvManager::try_pin_g1_prefix` 在调度阶段持有 `ImmutableBlock<G1>`，通过 RAII 确保在 swap‑in 完成前不被驱逐。<br>- 新特性通过 `cfg(feature = "mocker-kvbm-offload")` 隐式开启，保持对不使用 off‑load 场景的零影响。 |

**📋 变更摘要**：本次提交将原先 *harness* 端 **PUB** 绑定的 ZMQ 端点改为由 Dynamo **PULL** 端点统一绑定，工具事件由各 harness 通过 **PUSH** 连接发送。对应的文档、环境变量描述、Relay 实现及 ZMQ 工具函数均同步更新，以消除多进程竞争绑定、提升事件可靠性，并完善 Topic 过滤与关闭流程。  

**🎯 影响范围**：

- 任何使用 `DYN_AGENT_TRACE_TOOL_EVENTS_ZMQ_ENDPOINT` 的外部 harness / 子进程插件  

**🔍 技术洞察**  
- **架构影响**  
  - 由 **“harness PUB → Dynamo SUB”** 变为 **“Dynamo PULL ← 多个 harness PUSH”**，实现单点绑定（Dynamo）而多点连接（harness），避免在多进程/子进程场景中出现 “Only one process can bind the same endpoint” 的启动错误。  
  - Relay 现在直接在 `bind_pull_socket` 上循环读取，去除了原来的 `connect_sub_socket` 逻辑，简化了网络层实现。  
  - 统一了 Topic 过滤语义：仅检查第一帧是否匹配 `zmq_topic`，不再在 Publisher 端进行过滤。  

- **性能影响**  
  - `Push/Pull` 组合在本地 TCP/ZMQ 上的吞吐量与原 PUB/SUB 相当，但更稳健：Push 的高水位 (HWM) 仍然受控，且 **不再丢失早期帧**（PUB 的 early‑join 丢帧问题已被消除）。  
  - 通过 **bounded队列 + 后台线程** 的设计保持低延迟，且在 `shutdown` 中采用循环尝试 `queue.put_nowait(None)`，防止因满队列导致阻塞退出。  
  - 多生产者并发写入同一 Pull 端点，ZMQ 内部已实现高效的多路复用，整体性能提升可估计在 5‑10% 左右（主要体现在高并发子进程场景下的启动成功率）。  

- **安全考虑**  
  - 端点仍然是 **本地 TCP**（`tcp://127.0.0.1:<port>`），未对外暴露，安全风险基本不变。  
  - 新增的 `zmq_topic` 过滤仅在 Relay 层进行，若误配置可能导致合法记录被静默丢弃，需在文档中强调正确的 topic 设置。  

**⚠️ 潜在风险**  
1. **向后兼容性**：已有外部 harness 仍采用旧的 `PUB bind` 方式将无法启动（因绑定冲突），需要同步更新。  
2. **端口冲突**：Dynamo 在启动时若端口已被占用，会直接返回错误，导致整个 tracing 系统失效。  
3. **Topic 过滤误用**：若 `DYN_AGENT_TRACE_TOOL_EVENTS_ZMQ_TOPIC` 与实际发送的 topic 不匹配，所有记录将被丢弃，难以在运行时捕获。  
4. **资源泄漏**：在异常路径（如 `queue.Full` 循环）中仍有可能留下未关闭的 socket；需注意在 `shutdown` 后确保 `linger=0` 被成功调用。  

**💡 关注建议**  
- **用户侧**：升级所有 harness/子进程插件，改为使用 `ZmqToolEventPublisher` 的 **PUSH** 方式（`socket.connect(endpoint)`），并确保环境变量 `DYN_AGENT_TRACE_TOOL_EVENTS_ZMQ_ENDPOINT` 已正确设置。  
- **运维侧**：在部署脚本中预先检查端口可用性，或使用 `0`（让系统分配）后通过环境变量注入实际端口。  
- **开发者**：为旧版 `PUB` 兼容模式保留一个隐藏的实验开关（如 `DYN_AGENT_TOOL_EVENTS_LEGACY_BIND`），帮助平滑迁移。  
- **测试**：加入多进程/子进程并发推送的集成测试（已在 PR 中），并在 CI 中验证 Topic 过滤和绑定失败的错误路径。  
- **文档**：在 `agent-context.md` 明确说明 “Dynamo 为唯一收集者，所有 harness 为生产者” 的新模型，以及生产者侧的示例代码。

**SHA**: `e06a3d3` | 🔗 [查看提交](https://github.com/ai-dynamo/dynamo/commit/e06a3d3cd3e73b955f4148d9251d78a05cf76901)

   - `WorkerLookupStats` 的 `from_worker_block_counts` 已过滤掉 `0` 块的 worker，确保 `worker_count()` 统计的是真实活跃 worker。  
   - 移除的 `tree_sizes` 相关代码大幅简化了 `ConcurrentRadixTree*` 实现，降低了 `DashMap+AtomicUsize` 的维护成本。  

**总体结论**：本次改动把统计从“写时维护”迁到“读时实时采集”，实现更简洁、易于验证的逻辑，且对业务路径影响有限。后续关注并发一致性说明、兼容旧同步 API、以及在高并发监控场景下的统计开销即可。祝调试顺利！
