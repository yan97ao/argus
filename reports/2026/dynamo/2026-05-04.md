### fix(vllm): fix prefill cancellation log race in _monitor_abort [DYN-2880] (#8768)
**SHA**: `0db9885` | 🔗 [查看提交](https://github.com/ai-dynamo/dynamo/commit/0db98859f469b6514d530b88b9f3a4372f31d831)

**🎯 变更类型**：Bug修复/性能优化/重构  
**⚡ 重要程度**：🔴高  
**📋 变更摘要**：本次提交修复了 VLLM 前缀（prefill）阶段取消请求时日志与实际 abort 的竞争问题。通过为 `DeferredAbort` 引入持久任务引用、拆分 `abort` 为 `_run_abort` 与 `_wait_and_abort` 两个协程，并在 `_monitor_abort` 中对 `engine_client.abort` 使用 `asyncio.shield`，确保即使外部任务被 `CancelledError` 中断，abort 仍能完整执行并记录日志。同步测试用例也同步更新以验证新行为并恢复了原本被 skip 的 prefill 取消测试。

**🎯 影响范围**：  
- `components/src/dynamo/vllm/handlers.py`（核心请求处理与取消逻辑）  
- `components/src/dynamo/vllm/tests/test_vllm_worker_handler.py`（单元测试覆盖）  
- `tests/fault_tolerance/cancellation/test_vllm.py`（集成/端到端取消测试）  

**🔍 技术洞察**：  
- **架构影响**：  
  - 引入了 `_run_abort` 统一的 abort 执行路径，使得 `DeferredAbort` 在 `abort`、`close` 与 `_monitor_abort` 中复用相同日志与异常处理逻辑，降低代码重复率。  
  - 通过在 `abort` 中创建 `asyncio.create_task` 并保存到 `_abort_task`，为异步取消提供了“强引用”，防止协程在被外部 `CancelledError` 同时抛出时被意外丢弃，提升了系统的鲁棒性。  
- **性能影响**：  
  - 额外创建的后台任务带来极低的调度开销（单个 `Task`），对整体吞吐无显著影响。  
  - 使用 `asyncio.shield` 包装 `engine_client.abort` 可避免因外部取消导致的中途中断，从而防止因半执行的 abort 导致资源泄漏或状态不一致，间接提升系统稳定性。  
- **安全考虑**：  
  - 该改动主要涉及控制流与日志一致性，对安全面无直接影响。  
  - 通过确保 abort 必定执行，避免了因取消竞争导致的潜在“挂起请求”泄露，间接提升了资源使用的安全性（防止 DoS 场景下的资源堆积）。  

**⚠️ 潜在风险**：  
1. **长时间阻塞风险**：`engine_client.abort` 若出现网络卡顿或内部阻塞，`asyncio.shield` 会阻止外层任务被取消，可能导致系统在高并发下出现“不可中断的” abort 任务，进而占用工作线程。  
2. **任务泄漏**：若 `close()` 未被调用（异常路径或进程异常退出），已创建的 `_abort_task` 仍会在后台运行，需确认进程结束时有统一的任务回收机制。  
3. **测试时序依赖**：新的单元测试大量依赖 `await asyncio.sleep(0)` 与任务调度顺序，可能在不同 Python/asyncio 版本或硬件负载下出现间歇性失效。  

**💡 关注建议**：  
- 为 `engine_client.abort` 增加合理的超时包装（例如 `asyncio.wait_for(..., timeout=5)`），在超时后记录警告并安全退出，防止阻塞。  
- 在 `DeferredAbort.close()` 中加入对 `_abort_task` 的 `cancel()` 并 `await` 超时处理，确保即使外部未显式调用 `close`，也能在对象析构或上下文退出时安全终止任务。  
- 在 CI 中加入跨平台（Linux/macOS/Windows）以及不同 Python 版本的并发压力测试，验证 `await asyncio.sleep(0)` 这种调度依赖不会导致假阴性。  
- 更新文档，明确 `abort()` 需要在 `close()` 前或同一事件循环中使用，以避免因忘记调用 `close` 导致后台任务残留。  

整体来看，此次改动显著提升了 VLLM 前缀阶段取消的可靠性，避免了日志遗漏与请求状态不一致的罕见 race 条件，属于高价值的 bug 修复。只要在生产环境关注 abort 超时与任务清理，即可安全部署。

### feat(kv-router): add anchor-aware branch sharding (#9007)
**SHA**: `ae6f181` | 🔗 [查看提交](https://github.com/ai-dynamo/dynamo/commit/ae6f18105525fe92bfdb99f366d06da608f52705)

**🎯 变更类型**：功能增强 / 架构变更  

**⚡ 重要程度**：🔴高  

**📋 变更摘要**  
- 为 KV‑router 引入 **Anchor‑aware Branch‑sharded Indexer**（`AnchorAwareBranchShardedIndexer`），在跨分片的 KV‑cache 中通过 **结构锚点**（anchor）实现前缀‑后缀路由，避免跨 shard 的前缀重复存储。  
- 完全重写 `ConcurrentRadixTreeCompressed`：采用 **分离式锁**（shape‑gate + shape‑version）、**压缩边缘**（edge compression）以及 **每工作者覆盖截点**（cutoff）实现高并发写入、惰性查找修复和高效删除。  
- 新增大量测试：并发写入、并发删除、清理、读‑写竞争、锚点安装/复用等场景的细粒度验证。  
- 细化 metric 与日志：  
  - `KvIndexerMetrics` 迁移至 `ShardedIndexerMetrics`（计数/计时分离），仅在 `bench` feature 开启时启用。  
  - 将 `apply_event` 过程的 tracing 限制在 `TRACE` 级别才产生，降低生产环境日志噪声。  
- 代码层面清理：移除 `RadixTree`、`Positional` 中冗余的 `tree_sizes` 填充逻辑，统一交给后端实现；删除 `lower_tier` 中已废弃的 `remove_worker_from_edge` 分支等。  
- `ThreadPoolIndexer` 新增 `enqueue_anchor` 用于在同一工作线程上保序地写入 **anchor** 任务。  
- 配置层面公开 **anchor‑aware‑branch‑sharded‑crtc** 选项，兼容已有 CLI/文档。

#### 架构影响
| 维度 | 说明 |
|------|------|
| **模块化** | 将压缩 Trie 的实现拆分为 `node`, `node_state`, `remove`, `store`, `repair`, `dump`, `matches` 等子模块，职责更清晰，易于独立演进和基准。 |
| **锚点机制** | `AnchorRef`/`AnchorTask` 让分片之间共享 **“边界锚点”**（父块哈希），实现 **“前缀‑后缀跨 shard”** 的准确路由而不产生重复存储。锚点通过 `ThreadPoolIndexer::enqueue_anchor` 保证在同一 worker 队列中 **先锚点后存储** 的顺序。 |
| **锁粒度** | 每个 `Node` 维护 **两把锁**：`shape_gate`（读/写短暂互斥） + `state`（RwLock）以及 `children` 使用 `DashMap`。这种 **读‑写分离 + 版本校验** 能在高并发读期间避开全局锁，提升 `find_matches` 的吞吐。 |
| **惰性查找修复** | 当写入线程分裂 edge 时，其他工作线程的逆向查找（`worker_lookup`）可能指向旧节点。`resolve_lookup`+`repair_lookup_for_resolved_node` 在 **需要时** 扫描子树并更新所有受影响的 worker lookup 表，避免一次性全局更新的写放大。 |
| **后端统一** | `SyncIndexer` 接口新增 `apply_anchor` / `find_matches_from_anchor`，默认返回 `Unsupported`，只有新后端实现覆盖。旧实现保留兼容性，调用路径无需改动。 |
| **清理机制** | `CleanupState` 与 `CleanupGuard` 仍保留，但 `ConcurrentRadixTreeCompressed` 在 `run_cleanup_for_test` 中实现 **自底向上的子树收割**，并在 `ThreadPoolIndexer::maybe_enqueue_cleanup` 中自动调度。 |

#### 性能影响
| 场景 | 预期变化 | 关键实现 |
|------|----------|----------|
| **高并发写入** (多 worker 同时向同一前缀存储不同后缀) | **提升**：写入冲突大幅减少，仅在 **split** 或 **anchor 安装** 时获取 `shape_gate` 写锁；普通子节点插入使用 `DashMap` 并发安全。 | `Node::with_shape_plan` + 多次 `shape_version` 重试 (`SHAPE_PLAN_RETRIES=4`)。 |
| **跨 shard 解码延伸** (decode 继续写入已有前缀后缀) | **提升**：通过锚点复用 **prefix‑suffix**，避免在每个 shard 重新存储前缀块，显著降低内存占用和网络 I/O。 | `apply_anchor` + `find_matches_from_anchor`。 |
| **查询 (`find_matches`)** | **轻微回退**：在 split 过程中 `find_matches` 可能看到 **相邻两棵树的状态**，导致 **最坏情况下少量 over‑count**（已被容忍且不会 panic）。 | `find_match_step` 不持 `shape_gate`，仅读 `NodeState`，在版本不匹配时容忍。 |
| **删除 (`apply_removed`)** | **提升**：删除只调整 `cutoff`，无需拆分 edge，且**惰性清理**子树，保持 O(1) 删除 + O(#workers) 更新。 | `Node::remove_worker_for_hash` → `Node::remove_child_if_stale_leaf`。 |
| **基准计数** | 新增 `ShardedIndexerMetrics`：`find_match_dispatches`、`find_match_early_returns`、`anchor_installs`、`anchor_reuses`、`remove_broadcasts`、`node_splits`、`lookup_repair_scans/entries`，帮助定位瓶颈。 | `cfg(feature="bench")` 条件编译。 |
| **日志** | 生产环境日志负

### fix: deepseek v4 inter-leaved tool and reasoning parser (#8901)
**SHA**: `cddc3c8` | 🔗 [查看提交](https://github.com/ai-dynamo/dynamo/commit/cddc3c8ffcc4f5eec1299362534058189b36ed84)

**🎯 变更类型**：Bug修复 / 功能增强（针对 DeepSeek V4 与 Kimi K2.5 工具后续回合的推理解析）  

**⚡ 重要程度**：🔴高  

**📋 变更摘要**  
1. 在 `OpenAIPreprocessor::postprocessor_parsing_stream` 中，引入 `suppress_reasoning_after_tool` 标志，使得：  
   - 当请求的最后一条消息是 `tool` 且使用 Kimi K2.5 解析器时，仍然关闭强制推理（保持原有行为）。  
   - 对 DeepSeek V4 则不再自动关闭，保留“prompt‑injected reasoning” 信号，以正确处理模型在工具后直接进入推理块的情况。  
2. 新增 3 组针对不同模型（DeepSeek V4、Kimi K2.5）和不同切分情况的回归测试，验证：  
   - 推理内容被正确抽取为 `reasoning_content`。  
   - `</think>` 结束标签不泄漏到普通 `content`。  
   - 对 Kimi K2.5 的旧行为保持不变。  

**🎯 影响范围**  
- `lib/llm/src/preprocessor.rs`（核心推理预处理逻辑）  
- `lib/llm/tests/postprocessor_parsing_stream.rs`（流式后处理的单元测试）  
- `lib/llm/tests/test_reasoning_parser.rs`（推理解析器层面的测试）  

**🔍 技术洞察**  

| 维度 | 影响描述 |
|------|----------|
| **架构影响** | - 仅在 **OpenAIPreprocessor** 的流式后处理路径加入了模型特定的分支 (`suppress_reasoning_after_tool`) ，保持了现有插件化/配置驱动的设计。<br>- 通过 `runtime_config.reasoning_parser` 决定行为，使得新模型（DeepSeek V4）无需修改上层调用者，兼容性良好。 |
| **性能影响** | - 新增的布尔判断和 `matches!` 检查成本极低（常数时间），对整体流式处理中吞吐量几乎没有可观测影响。<br>- 额外的测试用例增加 CI 运行时间约 0.2 s，属于可接受范围。 |
| **安全考虑** | - 代码仅在解析阶段做字符串标签的过滤与路由，不涉及外部 I/O、内存分配的改变，未引入新的攻击面。<br>- 仍需要确认 `runtime_config.reasoning_parser` 的来源受信任，防止恶意配置导致错误的强制推理开启，导致信息泄露或模型输出被误分类。 |
| **可维护性** | - 通过显式变量 `suppress_reasoning_after_tool` 将模型特定的条件抽离，代码可读性提升。<br>- 新增的测试覆盖了关键分支及边界（闭合标签跨块） ，为未来添加其他模型的特殊规则提供模板。 |
| **兼容性** | - 对已有模型（如 Kimi K2.5）保持原有 “在 tool‑turn 后关闭强制推理” 行为，防止回归。<br>- 对 DeepSeek V4 现有用户则无需额外配置即可获得正确的推理抽取。 |

**⚠️ 潜在风险**  

1. **模型识别误判**  
   - 依赖 `runtime_config.reasoning_parser` 字符串匹配（`"kimi_k25"`），若用户自行设置错误的标识或拼写错误，可能导致推理被错误地强制或被错误地抑制。  
2. **未来模型的特殊需求**  
   - 目前仅针对 DeepSeek V4 和 Kimi K2.5 增加了判断，若后续出现类似“工具后仍需保留推理信号”的模型，需要再次修改此处逻辑，可能导致分支增长。  
3. **测试覆盖不足的极端流**  
   - 虽然已加入 `</think>` 跨块的测试，但仍未覆盖极端的多块、嵌套或缺失闭合标签的情况，可能在生产环境出现未捕获的 tag 泄漏。  

**💡 关注建议**  

1. **配置校验**  
   - 在启动时对 `runtime_config.reasoning_parser` 做白名单校验，确保仅接受已知的解析器标识，避免拼写错误导致行为异常。  

2. **抽象模型特性**  
   - 将 “是否在 tool‑turn 后保持 injected reasoning” 抽象为模型元数据（例如 `preserve_reasoning_after_tool: bool`），放入配置结构体中，减少硬编码的字符串匹配。  

3. **增强流式解析容错**  
   - 在 `parse_reasoning_content_from_stream` 中加入对缺失 `</think>` 的容错（如在流结束时自动关闭推理块），防止意外泄漏。  

4. **持续回归测试**  
   - 将新增的三组测试纳入每日 CI，确保在引入其他模型或改动预处理器时不产生回归。  
   - 考虑加入 fuzz 测试，随机生成多块流，验证 `reasoning_content` 与 `content` 的分离始终正确。  

5. **文档更新**  
   - 在项目的模型兼容性文档或 README 中注明 DeepSeek V4 与 Kimi K2.5 在工具后续回合的特殊行为及对应的配置要求，帮助使用者避免误用。  

---  

**结论**  
此提交通过细粒度的条件判断纠正了 DeepSeek V4 在工具‑后回合中推理标记被错误清除的缺陷，同时保留了对 Kimi K2.5 的原有抑制策略。改动对系统架构影响极小，性能开销可忽略，安全风险有限。只要在配置管理和未来模型扩展上保持审慎，即可安全合并到主线。

**🔍 技术洞察**  

- **架构影响**  
  - **配置向下传播**：`OperatorConfiguration.Checkpoint` 现在携带 `Seccomp` 子结构，所有生成 PodSpec 的入口（Job、ComponentDeployment、Clique）统一通过 `EffectiveSeccompProfile()` 取得最终注入路径，保持单点配置源，降低了散落硬编码的风险。  
  - **兼容 OpenShift**：通过 `Seccomp.Disabled` 可在 OpenShift（SCC 限制）环境中关闭注入，避免因自定义 localhost profile 需要 privileged SCC 导致的调度失败。  
  - **模块解耦**：`EnsureLocalhostSeccompProfile` 只负责在 `PodSecurityContext` 中写入 profile，使用空字符串实现“无注入”，调用侧不需再判断平台差异，提升代码可维护性。  

- **性能影响**  
  - 仅在 PodSpec 初始化阶段写入一个字符串字段，CPU 与内存开销可忽略不计。  
  - 对已有工作负载的启动时间基本不受影响，唯一潜在的微小延迟来自于额外的 API 读取（`EffectiveSeccompProfile`）以及 Helm 渲染的几行额外 YAML。  

- **安全考虑**  
  - 默认注入的 `block-iouring.json` profile **主动限制**了 CRIU 不能安全快照的 `io_uring` 系统调用，提升了 checkpoint/restore 的可靠性，防止因未知系统调用导致的恢复失败或潜在的系统不一致。  
  - `Seccomp.Disabled=true` 让用户在已信任的环境（如已使用支持 `io_uring` 的 CRIU）或在 OpenShift 必须使用 SCC 特权时，**关闭**此限制，避免因不兼容而导致 Pod 被拒绝。  
  - 需要确保自定义 `profile` 路径的文件在容器镜像或 Volume 中可访问，否则会导致 Pod 启动失败。  

**⚠️ 潜在风险**  

1. **默认 profile 不可用**  
   - 如果部署的 Operator 镜像未随附 `profiles/block-iouring.json`，或用户在自定义镜像中删除了该文件，将导致 Pod 创建失败（seccomp profile not found）。  
2. **OpenShift SCC 冲突**  
   - 在 OpenShift 上忘记将 `Seccomp.Disabled` 设为 `true`，仍尝试注入 localhost profile，会因 SCC 不允许自定义 `seccompProfile` 而被拒绝。  
3. **自定义 profile 错误**  
   - 用户自行提供的 profile 若格式不符合 seccomp 规范，容器运行时会直接报错，影响业务可用性。  
4. **深拷贝遗漏**  
   - `DeepCopyInto` 已补全，但若后续在代码中新增字段而忘记更新 deepcopy，可能导致配置在控制循环中丢失。  

**💡 关注建议**  

- **发布检查**：在 Operator 打包镜像阶段校验 `profiles/block-iouring.json` 是否已包含；若计划移除默认 profile，请同步更新文档和 `DefaultSeccompProfile` 常量。  
- **部署指南**：在 Helm `README`、OpenShift 部署手册中明确说明：  
  - 对 OpenShift 必须将 `checkpoint.seccomp.disabled=true`，或自行绑定 permissive SCC。  
  - 如使用自定义 CRIU（已支持 `io_uring`），同样建议关闭注入。  
- **监控/告警**：可以在 Operator 的状态报告中新增 `seccompProfileApplied` 字段，帮助运维快速定位因 seccomp 引起的 Pod 启动异常。  
- **单元测试覆盖**：已加入 `EffectiveSeccompProfile` 的单元测试，建议继续在更多调用路径（如 `InjectCheckpointIntoPodSpec`）添加对 `seccompProfile` 为 `""` 的场景测试，确保在 OpenShift 环境下不触发注入。  
- **文档同步**：确保 API 文档、Helm values 示例、以及 `docs/kubernetes/api-reference.md` 中的默认值描述保持一致，防止用户误解默认行为。
