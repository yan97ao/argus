**⚡ 重要程度**：🔴 高  

**📋 变更摘要**  
- 将散落在 `tests/gms/*` 的大量 GPU Memory Service（GMS）集成/端到端测试全部删除，移除与外部进程、端口管理、托管进程等复杂测试基础设施。

**🎯 影响范围**  
- **核心代码**：`gpu_memory_service` 运行时代码未改动，仅影响 **测试套件**。

- **CI / CI‑pipeline**：依赖原有 `tests/gms/*` 的 CI 作业将不再运行，改为更轻量的测试集合。

### feat: expose multimodal embedding cache metrics via Prometheus (#8031)
**SHA**: `96476be` | 🔗 [查看提交](https://github.com/ai-dynamo/dynamo/commit/96476be5e153843739b0f4d17fcafeb2b4eea428)

**🎯 变更类型**：功能增强  
**⚡ 重要程度**：🔴高  
**📋 变更摘要**：在 `MultimodalEmbeddingCacheManager` 中加入驱逐计数，并通过 `register_embedding_cache_metrics` 将命中、未命中、驱逐以及缓存容量等信息以 Prometheus 指标形式暴露。新增统一的 `EmbeddingCacheMetrics` 枚举、指标注册回调以及在多模态、LLM、VLLM 工作进程的初始化路径中完成自动注册。配套单元测试验证计数累加、Gauge 快照以及标签完整性。  

**🎯 影响范围**：  
- `components/src/dynamo/common/memory/multimodal_embedding_cache_manager.py`（缓存统计）  
- `components/src/dynamo/common/utils/prometheus.py`（指标枚举、注册函数）  
- 多模态编码、LLM、VLLM 工作进程的初始化入口（`init_multimodal.py`、`llm_worker.py`、`worker_factory.py`）  
- 新增单元测试 `components/src/dynamo/common/tests/memory/test_embedding_cache_metrics.py`  

**🔍 技术洞察**  

- **架构影响**  
  - 引入 **EmbeddingCacheMetrics** 统一定义，避免硬编码指标名称，提升可维护性。  
  - 每个 **Endpoint** 通过 `register_prometheus_expfmt_callback` 持有独立的 **CollectorRegistry**，实现指标隔离，兼容 SGLang 的多进程模式。  
  - 通过在组件初始化阶段自动调用 `register_embedding_cache_metrics`，将监控层与业务层解耦，符合观察性（Observability）最佳实践。  

- **性能影响**  
  - **运行时开销**：在每次 `/metrics` 抓取时会读取缓存的 `stats` 并执行一次锁保护的增量计数及 Gauge 设值。读取的都是整数/浮点属性，受 GIL 保护，开销极低。  
  - **锁竞争**：使用 `threading.Lock` 仅在并发抓取时序列化回调，常规单线程抓取（Prometheus 默认单实例）不会产生竞争。  
  - **内存占用**：每个组件额外创建一个 `CollectorRegistry`（约数 KB）以及若干 Counter/Gauge 对象，整体可接受。  

- **安全考虑**  
  - 仅读取内部状态，无写入操作，未引入外部输入路径，安全风险极低。  
  - 依赖 `prometheus_client` 动态导入，避免在多进程启动前加载产生的 Fork‑Safety 问题。  

**⚠️ 潜在风险**  

1. **重复注册**：若同一 `Endpoint` 多次调用 `register_embedding_cache_metrics`（例如在重启/重新加载时），可能导致同名指标在同一 Registry 中冲突。当前实现未检测重复，需要调用方确保一次性注册。  
2. **Metric 拉取耗时**：在高并发服务中若 `/metrics` 被频繁抓取，回调内部的 `cache.stats` 读取虽然轻量，但仍会阻塞其他抓取线程。若使用多进程模式，每个进程都有独立回调，总耗时会累积。  
3. **多进程兼容性**：`prometheus_client` 在多进程模式下要求在子进程启动前设置 `multiprocess_dir`。虽然代码已延迟导入，但在极端部署（如 fork 后立即创建 Endpoint）仍可能触发导入顺序错误。  

**💡 关注建议**  

- **防止重复注册**：在 `register_embedding_cache_metrics` 中加入 idempotent 检查（如在 `endpoint` 上保存已注册标记），或在调用方使用 guard 防止多次注册。  
- **监控回调耗时**：在生产环境通过 `prometheus_client` 的 `process_cpu_seconds_total` 或自定义计时器监控 `/metrics` 响应时间，必要时对回调做异步缓存（只在上一次抓取后 1s 内复用结果）。  
- **多进程部署验证**：在使用 SGLang 多进程模式时，执行一次完整的启动‑抓取‑关闭循环，确保没有 “CollectorRegistry already exists” 或 “Metric name already registered” 错误。  
- **文档与标签统一**：更新项目文档，说明新增的 `model` 与 `dynamo_component` 标签的意义，以便在 Prometheus 查询中正确聚合。  
- **回归测试**：已加入单元测试覆盖计数累加、Gauge 快照、标签完整性，建议在 CI 中加入性能基准（如每秒抓取 100 次的 CPU/latency）以防止未来改动引入显著开销。

### feat(gms): exported GMS handles on allocation (#8108)
**SHA**: `49feb28` | 🔗 [查看提交](https://github.com/ai-dynamo/dynamo/commit/49feb28404094d986738d41c63d6f6b64908a858)

**🎯 变更类型**：功能增强

**📋 变更摘要**：  
- 在 GMS（GPU Memory Service）的分配流程中，引入 `export_fd` 缓存：每次分配时仅调用一次 CUDA 的 `cuMemExportToShareableHandle`，随后通过 `os.dup` 复制已缓存的文件描述符供客户端使用。

**🔍 技术洞察**：

- **架构影响**：  
  - 将 “导出句柄” 的生命周期从 **每次导出 RPC** 转为 **分配时一次**，缓存于 `AllocationInfo.export_fd`。  
  - 通过 `os.dup` 实现句柄复用，减少对 CUDA VMM 接口的调用次数，简化后端与客户端的交互模型。  
  - 改动对外部协议无影响，仍然通过 Unix socket 传递 `SCM_RIGHTS`，但后端服务的内部状态更为丰富（多一个 FD 字段）。  

- **性能影响**：  
  - **正面**：避免了在高并发导出场景下重复调用 `cuMemExportToShareableHandle`（一次系统调用 + CUDA 调用），降低了 CPU 与 CUDA 驱动的开销，降低延迟。  
  - **负面**：每个分配会多保留一个打开的文件描述符，若大量小块分配可能逼近进程的 FD 上限，需要关注 FD 资源使用情况。  

- **安全考虑**：  
  - 句柄复用通过 `dup` 实现，保持原始 FD 的权限与属性，无新增的安全漏洞。  
  - 代码在 `free_allocation` 与 `clear_all` 中显式 `os.close(export_fd)`，防止泄漏导致的“资源占用”攻击。  

**⚠️ 潜在风险**：  
1. **FD 泄漏**：若未来在分配路径加入异常分支或提前返回，未触发 `free_allocation`，可能导致缓存的 FD 未关闭。  
2. **并发导出**：在高并发 `export_allocation` 调用时，虽然 `dup` 是原子的，但如果同一 allocation 被多线程同时 `free`，可能出现 “已关闭的 FD 再 dup” 场景，需要保证 `export_allocation` 与 `free_allocation` 的调用顺序安全（目前在单进程 async 环境下已足够）。  
3. **FD 上限**：极端情况下，大量分配且未及时释放会占满进程的文件描述符表，触发 `EMFILE` 错误。  

**💡 关注建议**：  
- **资源监控**：在部署环境中监控进程的打开文件描述符数量，必要时调高 `ulimit -n`。  
- **异常保护**：在分配函数的异常路径（如 `cumem_create_tolerate_oom` 失败后）提前释放可能已创建的 `export_fd`，避免泄漏。  
- **文档同步**：确保 SDK 使用者了解在 `RW_OR_RO` 模式下，首次 writer 会持有写锁并在 abort 时可以立即释放，防止误以为需要显式清理导出句柄。  
- **代码审查**：后续如引入多进程或多线程的分配管理，需要在 `export_allocation` 与 `free_allocation` 之间加入锁或原子标记，以防 “使用已关闭的 FD”。

  - **输入验证**：`DYN_ENCODER_CUDA_TO_CPU_RATIO` 被解析为 `usize` 并强制 `>=1`，防止异常负数或零导致除零错误。  

- **可维护性**  
  - **代码分离度良好**：核心预算算法被抽离为 `device_aware_candidate_group`，单元测试覆盖充分，后续可独立演进。  
  - **文档同步**：README、router‑guide 已同步更新，降低学习成本。  
  - **向后兼容**：默认 `device_type=None` 时仍可使用旧路由模式，不会破坏现有部署。  

**⚠️ 潜在风险**  

1. **错误的设备检测**  
   - 当容器内部未正确设置 `CUDA_VISIBLE_DEVICES` / `NVIDIA_VISIBLE_DEVICES`，但实际无 GPU 可用，仍可能被误判为 `Cuda`，导致路由器把请求分配给不可用 GPU 实例，触发运行时错误。  
2. **比例配置不当**  
   - `DYN_ENCODER_CUDA_TO_CPU_RATIO` 若设置过高或过低，会导致 CPU 或 GPU 过载/空闲，削弱负载均衡效果。缺省值 8 基于典型模型，特定工作负载需调优。  
3. **极大规模实例列表的调度延迟**  
   - O(N) 的遍历在上千实例环境下可能增加路由路径的延迟，尤其在高 QPS 场景。  
4. **兼容性回退**  
   - 当所有实例的 `device_type` 为 `None`（旧版本未上报），`device_aware_weighted` 将退化为 `least_loaded`，但在混合集群中若仅部分实例升级，可能出现“半可见”行为，导致调度策略不一致。  

**💡 关注建议**  

- **部署检查**：在使用 `device-aware-weighted` 前，验证容器/节点的 `CUDA_VISIBLE_DEVICES`、`NVIDIA_VISIBLE_DEVICES` 是否符合预期，确保 GPU 实例真实可用。  
- **比例调优**：建议在首次上线后监控每类设备的 `inflight` 与响应时延，通过指标（如 `frontend_perf`）调节 `DYN_ENCODER_CUDA_TO_CPU_RATIO`，找到最适合业务的值。  
- **监控告警**：在运行时加入对 `device_type` 为 `None` 的实例比例的监控，若出现异常高占比，提醒运维检查版本升级或环境变量配置。  
- **性能基准**：对大规模（>500）实例的集群进行基准测试，比对 `least_loaded` 与 `device-aware-weighted` 的调度延迟，评估是否需要对算法做分片或缓存化优化。  
- **回滚方案**：如出现异常行为，可通过 CLI `--router-mode least-loaded` 快速回退，保持服务可用。  

总体而言，此次功能增强为异构硬件环境提供了更合理的调度策略，提升资源利用率与吞吐，风险可通过配置验证、监控和渐进式 rollout 有效管控。

总体来说，此改动提升了跨容器、跨 NCCL 发行版的鲁棒性，但需关注最低 NCCL 版本要求及符号链接的兼容性。
