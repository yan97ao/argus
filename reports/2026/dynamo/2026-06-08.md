### fix(sglang): forward repetition sampling controls (#10310)
**SHA**: `c3f80f1` | 🔗 [查看提交](https://github.com/ai-dynamo/dynamo/commit/c3f80f172e49d88a3832412f973d568804f56864)

**🎯 变更类型**：Bug修复 / 功能增强  
**⚡ 重要程度**：🔴 高  
**📋 变更摘要**：  
- 为 SGLang 的 `decode_handler` 与 `prefill_handler` 新增统一的采样参数抽取逻辑，完整转发 `presence_penalty`、`frequency_penalty`、`repetition_penalty`、`min_p` 与 `seed`（映射为 `sampling_seed`）等控制字段。  
- 通过 `_sampling_option_params` 函数统一处理，避免在不同代码路径中手动拼接导致的遗漏。  
- 新增单元测试验证在 token‑request 与 OpenAI‑style request 两种入口下，这些采样控制能够正确传递。  

**🎯 影响范围**：  
- `components/src/dynamo/sglang/request_handlers/llm/decode_handler.py`  
- `components/src/dynamo/sglang/request_handlers/llm/prefill_handler.py`  
- 相关单元测试 `components/src/dynamo/sglang/tests/test_sglang_decode_handler.py`  

---

### 🔍 技术洞察  

| 维度 | 影响描述 |
|------|----------|
| **架构影响** | - 引入 `_SAMPLING_OPTION_FIELDS` 常量与 `_sampling_option_params` 辅助函数，实现采样参数抽取的 **单点统一**，降低了代码重复度。<br>- `prefill_handler` 通过 `from …decode_handler import _sampling_option_params` 复用该逻辑，形成 **跨模块共享**，提升了内部 API 的一致性。|
| **性能影响** | - 参数抽取使用一次 `dict` 推导，运行时开销几乎可以忽略（O(N) 的 N 为字段数，固定且极小）。<br>- 额外的函数调用层级对整体请求吞吐量影响微乎其微。|
| **安全考虑** | - 仅把前端传来的数值字段原样转发，无额外字符串拼接或代码执行，未引入新的攻击面。<br>- `seed` 直接映射为 `sampling_seed`，仍然是数值型，满足现有 SGLang 对种子的安全校验。<br>- 代码未对字段类型做显式校验，若上游传入非数值可能在 SGLang 端触发异常，但不会导致安全泄露。|

### ⚠️ 潜在风险  

1. **向后兼容性**：旧版 SGLang 可能不认识 `presence_penalty`、`frequency_penalty`、`repetition_penalty`、`min_p`、`sampling_seed` 等字段，导致请求被拒绝或返回错误。  
2. **类型假设**：`seed` 等字段被直接传递，若前端误传字符串或 `null`，SGLang 端可能抛异常，未在本层捕获。  
3. **循环依赖风险**：`prefill_handler` 依赖 `decode_handler` 的私有函数 `_sampling_option_params`。虽然目前不形成循环导入，但后续若对模块结构进行大幅调整，可能引入不可预期的导入错误。  
4. **测试覆盖不足**：仅覆盖了核心采样参数的转发，未覆盖异常路径（如缺失字段、字段类型错误、SGLang 不支持的字段）。

- **文档同步**：更新 SGLang 接口文档与 Dynamo 的 API 说明，明确哪些采样控制字段会被转发以及对应的字段名映射（`seed → sampling_seed`）。  
- **模块封装**：考虑将 `_sampling_option_params` 迁移到 `dynamo.sglang.utils` 等公共模块，并使用显式的 `public` 命名（去掉前导下划线），避免因私有函数跨模块使用导致的维护负担。  
- **回归测试**：在 CI 中加入老版本 SGLang 环境的兼容性测试，确保在不支持新字段的环境中仍能正常降级运行。  

总体来看，此次改动提升了采样控制参数的完整性与代码可维护性，风险可控，建议尽快合并并在生产环境监测异常率。

### feat(kv-router): add standalone slot-tracker replica sync (#10394)
**SHA**: `ca9d4ae` | 🔗 [查看提交](https://github.com/ai-dynamo/dynamo/commit/ca9d4aed32613119209c2d1374684b2f78508971)

**🎯 变更类型**：功能增强 / 架构变更 / 安全修复  

**⚡ 重要程度**：🔴高  

**📋 变更摘要**  
- 为 **standalone slot‑tracker** 增加可选的 **replica‑sync** 功能，基于 ZeroMQ PUB/SUB 实现跨实例的生命周期事件广播。  
- 引入全新的 **WorkerDpRange** 抽象以及 **ReplicaWorkerPolicy**，实现对 worker 拓扑的严格校验、增删改统一管理，并将原有的 `register_external_workers` 迁移为 **upsert** / **reconcile** / **unregister** 接口。  
- 扩展命令行、文档、HTTP API（`/register_peer`、`/deregister_peer`、`/peers`）以及内部服务模块，以支持动态 peer 管理、过程 ID 生成以及事件过滤。  
- 对 **kv‑router** 的调度、拓扑以及序列管理层做了大量重构，提升可维护性并为复制提供统一的发布/订阅路径。  

**🎯 影响范围**  
- `kv-router` 核心库（`lib/kv-router/src/sequences/*`、`scheduling/*`）  
- Standalone 服务：`slot_tracker`（服务入口、registry、replica_sync）  
- 相关 CLI 参数、文档、测试用例以及 Cargo 特性（`standalone-slot-tracker`）  
- ZeroMQ 通讯层（`services/zmq.rs`）以及公用的 ZMQ endpoint 校验函数  

**🔍 技术洞察**  

| 维度 | 影响描述 |
|------|----------|
| **架构影响** | 1. **复制层加入**：在单机 slot‑tracker 基础上通过 ZMQ PUB（推送）+ SUB（订阅）实现 *best‑effort* 的跨实例状态同步。<br>2. **拓扑抽象统一**：`WorkerDpRange` 代替原来的 `(worker_id, dp_start, dp_size)` HashMap，所有拓扑增删改通过 `WorkerTable` 完成，保证 **唯一性、合法性**（dp_range 校验）并提供查询接口。<br>3. **ReplicaWorkerPolicy**：可在 `LazyRegister`（保持旧行为）与 `RequireRegistered`（仅处理已登记的 worker）之间切换，防止复制产生的 “幽灵 worker”。<br>4. **服务解耦**：`SlotTrackerRegistry` 现在持有可选的 `ReplicaConfig`，在未启用复制时不创建任何 ZMQ socket，保持原有纯 HTTP 运行模式。 |
| **性能影响** | - **额外网络开销**：复制事件通过 ZMQ PUB 发送，串行化为 MessagePack（≈ 200 B/event），对 CPU 与网络带宽产生轻微负担。<br>- **异步通道**：内部 `mpsc::channel` (capacity 100 k) 为复制流提供背压，若消费速度不足会 **丢弃** 新事件（记录 trace），避免阻塞 HTTP 线程。<br>- **锁竞争**：`TrackerEntry` 现在使用 `parking_lot::Mutex` 包装 `lifecycle_lock` 与 `worker_ranges`（已重构为 `WorkerTable`），锁颗粒度更细，理论上不应出现显著性能回退。<br>- **调度路径**：`ActiveSequencesMultiWorker::reconcile_workers` 通过一次性批量更新，避免多轮 `register_external` 产生的 O(N²) 重建。 |
| **安全考虑** | - ZMQ **未加密、无认证**，文档已提醒只能在受信内部网络使用。若外部网络误连，可能导致 **恶意伪造** replica 事件（如 `AddRequest`、`Free`），进而引起资源耗尽或状态错乱。<br>- `--replica-sync-advertise` 与 `--replica-sync-peers` 参数未做白名单校验，攻击者可通过配置恶意 **peer endpoint** 实现 **DoS**（发送大量无效事件导致通道溢出）。<br>- 复制事件未携带签名或完整性校验，依赖 ZMQ 的可靠性保证，仍存在 **中间人注入** 风险。 |
| **可维护性** | - 引入的 `WorkerDpRange` 与 `WorkerTopologyError` 将拓扑错误集中统一，提升错误定位能力。<br>- `replica_sync` 模块代码组织良好，使用 `Result` + `anyhow::Context`，但对 **异步任务的生命周期**（`CancellationToken`）的管理稍显复杂，需要保证在服务关闭时 **正确关闭** ZMQ socket 与所有 `mpsc` 接收端。<br>- 新增的 HTTP 接口（peer 管理）与对应错误映射 (`PeerError`)，保持了 axum 路由的统一风格。 |

**⚠️ 潜在风险**  

1. **事件丢失 / 不一致**  
   - 复制采用 **best‑effort、无确认**，在高负载或网络抖动时可能出现 **消息丢失**，导致各 replica 的 slot‑tracker 状态不一致（尤其在 `RequireRegistered` 策略下，丢失的 `Add` 事件会导致请求永远不被计数）。  
2. **通道阻塞与背压**  
   - `REPLICA_EVENT_CHANNEL_CAPACITY = 100_000` 在极端流量（如 10k requests/秒）下可能填满，随后新事件被直接丢弃，仅记录 trace，可能隐藏业务异常。  
3. **错误的 peer 配置**  
   - `SelfEndpoint` 检查仅在 `register_peer` 时执行，若业务手动在 `--replica-sync-peers` 中写入自己的地址，仍会尝试 `connect`，导致 ZMQ 报错并可能导致服务启动失败。  
4. **资源泄露**  
   - 每次 `TrackerEntry::new`（包括在 `register`/`unregister` 场景下重新创建 tracker）都会创建新的 **replica_tx** 与 **replica_rx**，若旧 `tracker` 没有及时退出，其内部的 `run_replica_sync` 仍在后台运行，可能导致 **泄漏**。  
5. **锁竞争 / 死锁**  
   - `TrackerEntry` 现在持有 `lifecycle_lock`（普通 `Mutex`），在 `register`/`unregister` 与 `reconcile_workers` 中交叉使用时，若外部代码在同一线程持有锁并尝试调用注册函数，可能产生 **死锁**（虽然目前调用路径单线程，但未来扩展需注意）。  
6. **安全漏洞**  
   - 未限制 `--replica-sync-bind` 的绑定地址，若在公网机器上误开启，攻击者可直接向该 PUB 端点推送伪造事件，导致资源占用或错误计数。  
   - 未对复制事件的 `model_name`、`tenant_id` 进行白名单校验，恶意 peer 可制造 **模型/租户漂移**（跨租户注入请求）。

### fix(gms): configure scratch alias size (#10329)
**SHA**: `b9892e6` | 🔗 [查看提交](https://github.com/ai-dynamo/dynamo/commit/b9892e61eb34605f42f625a8b208802c73df5df0)

**🎯 变更类型**：功能增强 / Bug修复  
**⚡ 重要程度**：🔴高  

**📋 变更摘要**

- 重构 `_ScratchMapping` 与 `LocalMapping`，加入 `va_reserved_size` 与 `size` 字段，使得 VA reservation 与实际请求大小解耦。  
- 移除原有 `ensure_scratch_disabled` 辅助函数，改为在 `prepare_scratch_for_reallocation` 中完成「搬迁+路由翻转」的完整职责。  
- 更新 Torch allocator、VLLM worker 以及单元测试，以适配新接口并验证 Scratch 重分配在 CUDA 颗粒粒度下仍保持已提交分配。  

**🎯 影响范围**  
- `gpu_memory_service/client/memory_manager.py`（核心内存管理逻辑）  
- `gpu_memory_service/client/torch/allocator.py`（Torch 分配器注册/获取）  
- `integrations/vllm/worker.py`（VLLM 对 GMS 的使用）  
- 单元测试 `tests/test_runtime_flows.py`（新增 Scratch 重分配验证）  

**🔍 技术洞察**  

- **架构影响**  
  1. **职责划分更清晰**：`scratch_size` 成为 `GMSClientMemoryManager` 的显式配置，避免以前硬编码的 `granularity` 被误用为 Scratch 大小。  
  2. **数据结构重构**：`_ScratchMapping` 由「n_chunks」改为「size」+「va_reserved_size」，并在 `LocalMapping` 中加入同等字段，使 VA 预留尺寸与实际请求尺寸分离，提升了后续迁移及统计的准确性。  
  3. **迁移流程合并**：`prepare_scratch_for_reallocation` 现在一次性完成「搬迁 + 路由翻转」；原先的 `ensure_scratch_disabled` 被废除，降低了 API 表面层的冗余调用风险。  
  4. **跨组件兼容**：`get_or_create_scratch_manager` 统一传入 `scratch_size`，确保所有使用者（包括 VLLM）在创建 manager 时获得统一配置。

- **性能影响**  
  1. **映射次数下降**：Scratch 现在以 `scratch_size`（默认 512 MiB）为单位一次性映射多个 VA 段，原来的每颗粒 (`granularity`) 映射被合并，大幅减少 `cuMemMap`/`cuMemUnmap` 调用次数，降低驱动开销。  
  2. **VA 预留空间更紧凑**：`va_reserved_size` 按 `scratch_size` 对齐，比原先按照 `granularity` 对齐往往能减少预留的碎片（尤其在 KV 缓存尺寸非 4 MiB 整数倍时）。  
  3. **内存占用**：如果 `scratch_size` 设得过大，可能导致 VA 地址空间预留远大于实际需求（但不会导致物理显存浪费，因为仍只分配单个 Scratch 物理块）。在显存紧张的高并发场景下，建议保持默认或根据实际 KV 大小调节。  
  4. **额外校验开销**：构造函数与 `create_scratch_mapping` 添加的 4 条 `ValueError` 检查在 Python 层几乎可以忽略不计。

- **安全考虑**  
  - **输入验证**：对 `scratch_size` 强制要求 ≥ CUDA 颗粒、是颗粒的整数倍且为 2 的幂，防止非法大小导致驱动报错或地址冲突。  
  - **未见权限或特权提升风险**：所有变更仅在用户空间的 Python 包中，未涉及底层驱动安全路径。

**变更类型**：Bug 修复  
**重要程度**：🟡 中  

**变更摘要**：在 `kv_router::publisher::worker_metrics` 中，改进了空闲时指标发布器的唤醒逻辑。原实现使用一个始终重置为 1 h 的 `publish_timer`，导致即使指标未变化也会频繁触发 `select!`，浪费 CPU 并可能产生不必要的网络流量。现在改为：① 使用 `tokio::time::sleep(Duration::ZERO)` 并 `tokio::pin!` 创建一次性计时器；② 当新指标与上一次相同直接 `continue`，仅在实际变化时才设置 1 ms 的延迟并保存待发布数据；③ 只在 `pending_publish` 为 `Some` 时才等待计时器，避免空闲轮询。

**影响范围**：`lib/llm/src/kv_router/publisher/worker_metrics.rs`（指标发布子系统），间接影响所有使用 KV‑Router 采集工作节点负载的组件。

**关注建议**  
- **开发者**：为“指标未变化保持沉默”的分支新增单元测试，确保 `publish_timer` 不会在空闲时触发。验证在高并发更新下 `pending_publish` 的覆盖行为是否符合预期，避免因连续相同值导致丢失一次发布。  
- **用户**：升级后监控后台 CPU 利用率应下降，若出现 “metrics publish delayed” 警告请检查网络或 NATS 客户端是否因 1 ms 防抖导致累计延迟。  
- **代码维护**：保留对 `tokio` 版本的兼容性说明，若未来 `tokio` 改进 `sleep` 的 `reset` 行为，及时评估是否仍需手动 `pin!`。  

总体而言，此改动消除了无意义的唤醒，提升了运行时效率，风险主要在于计时器重置的细粒度可能在极端负载下产生轻微的恼人抖动，建议在发布前进行压测确认。
