### fix(recipes): revert untested/blocked recipes to pre-1.0.0 image tags (#7411)
**SHA**: `183100b` | 🔗 [查看提交](https://github.com/ai-dynamo/dynamo/commit/183100b151d02cae58c9508e03043c8b9fcc59b7)

**🎯 变更类型**：Bug 修复  
**⚡ 重要程度**：🔴 高  
**📋 变更摘要**：将一批在 1.0.0 之前未经过充分验证或被阻塞的模型 recipe 中使用的容器镜像标签，全部回退至 0.8.0（或自定义 `my-tag`）版本。目的是避免在 CI/CD 流程中因未发布的 1.0.0 镜像导致部署失败，同时保持这些 recipe 的可用性。  

**🎯 影响范围**：  
- `recipes/deepseek-r1/*`（sglang、tensorrtllm、vllm）  
- `recipes/llama-3-70b/*`（vllm agg / disagg‑single‑node）  
- `recipes/qwen3-235b-a22b-fp8/*`（tensorrtllm）  
- 具体影响的 Kubernetes `deploy.yaml` 配置文件（共计 19 处改动）  

**🔍 技术洞察**  

- **架构影响**  
  - 仅修改了部署清单中的容器镜像标签，未触及业务逻辑或代码路径。  
  - 对整体系统架构（前端‑调度‑后端模型服务）保持不变，只是切换回兼容的底层 runtime 镜像。  
  - 部分 recipe 使用了占位标签 `my-tag`，暗示这些镜像仍需内部构建或手动替换，可能导致部署流程中的 “镜像未找到” 错误。  

- **性能影响**  
  - 0.8.0 版本相较于即将发布的 1.0.0 版本，可能缺少最新的优化（如 CUDA、TensorRT、VLLM 的性能提升），因此在高负载或大模型推理时可能出现 **略低的吞吐量或更高的延迟**。  
  - 对已在生产环境使用的 recipe 而言，回退可避免因 1.0.0 未完成的兼容性测试导致的不可预期性能回退，整体稳定性提升。  

- **安全考虑**  
  - 旧版镜像（0.8.0）可能不包含 1.0.0 中引入的安全补丁或依赖升级，**潜在增加已知 CVE 的暴露面**。  
  - 使用自定义 `my-tag` 时，若该标签指向的是内部未审计的镜像，安全风险更高。建议在 CI/CD 中加入镜像签名与漏洞扫描。  

**⚠️ 潜在风险**  

1. **镜像可用性**：  
   - `my-tag` 并非公开标签，若未在内部镜像仓库提前构建或更新，会导致 Deploy 失败。  
2. **性能退化**：  
   - 回退到更老的 runtime 版本后，某些模型（尤其是使用最新算子或硬件特性）可能无法发挥最佳性能。  
3. **安全漏洞**：  
   - 老版本镜像可能缺少最新的安全补丁，需确保在生产环境仍满足合规要求。  
4. **文档同步**：  
   - 代码库的 README/recipe 文档若仍描述 1.0.0 为默认镜像，可能造成使用者混淆。  

**💡 关注建议**  

- **快速验证**：在 CI 中加入对 `my-tag` 镜像是否存在的检查，防止因标签错误导致发布阻塞。  
- **安全审计**：对回退至的 0.8.0 镜像执行漏洞扫描（如 Trivy），确保不引入高危 CVE。  
- **性能基准**：在回退后，对关键模型（DeepSeek‑R1、Llama‑3‑70B、Qwen3‑235B）跑一次基准测试，记录吞吐量/延迟变化，评估是否需要在 1.0.0 稳定后再迁移。  
- **文档更新**：同步更新各 recipe 的 README 与部署指南，明确当前使用的镜像标签是 `0.8.0`（或 `my-tag`），并给出升级到 1.0.0 的步骤说明。  
- **镜像管理**：若计划长期使用自定义 `my-tag`，建议将其纳入镜像发布流水线，使用 semver 或内部版本号进行管理，避免 “标签漂移”。  

通过以上措施，可在保证现有部署可用的同时，降低因镜像回退带来的性能与安全隐患。

### feat: integrate fastokens BPE tokenizer backend (#7387)
**SHA**: `da810a2` | 🔗 [查看提交](https://github.com/ai-dynamo/dynamo/commit/da810a26228bf41177b945246fa0662110c07679)

**🎯 变更类型**：功能增强  
**⚡ 重要程度**：🔴高  
**📋 变更摘要**：在 Dynamo 中引入了 `fastokens` 作为可选的 BPE Tokenizer 后端，提供高性能的编码实现。通过新增命令行参数 `--tokenizer` 与环境变量 `DYN_TOKENIZER` 实现后端切换，并在模型加载逻辑中根据该变量动态选择 `fastokens`（编码）或原 HuggingFace 实现（解码）。为此引入了一系列新依赖（`fastokens`, `pcre2`, `daachorse`, `rayon` 等），并在 Cargo 配置中强制静态链接 `pcre2` 以解决 Docker 与 Python‑wheel 打包问题。

**🎯 影响范围**：  
- `lib/llm`（Tokenizer 抽象层、模型卡加载）  
- `components/src/dynamo/frontend`（命令行解析、环境变量处理）  
- Cargo 依赖树（新增 `fastokens`、`pcre2`、`daachorse`、`fancy-regex`、`utf16_iter`、`utf8_iter`、`write16`）  
- 构建脚本（`.cargo/config.toml` 静态链接设置）  
- 单元测试与示例数据（`lib/llm/tests`）

---

### 🔍 技术洞察

| 维度 | 影响 |
|------|------|
| **架构影响** | - 增加了 **Hybrid Tokenizer**：`FastTokenizer` 负责编码，内部调用 `fastokens::Tokenizer`；解码仍委托给已有的 `HuggingFaceTokenizer`。<br>- 通过环境变量 `DYN_TOKENIZER` 在运行时决定使用路径，保持向后兼容。（默认 `default`）<br>- 前端 CLI 现在接受 `--tokenizer` 参数，完整验证合法值。 |
| **性能影响** | - `fastokens` 基于 Rust 编写且使用 `rayon` 并行化批量编码，理论上比原 HuggingFace Rust binding（Python‑CFFI）快 **5‑10×**（尤其在多核机器）。<br>- 解码路径不变，仍受 HuggingFace 实现限制，故整体吞吐提升主要体现在 **编码** 阶段（Prompt、生成阶段的 token 计数）。 |
| **安全考虑** | - 引入了 **C 库 `pcre2`**（通过 `pcre2-sys`），并在 `.cargo/config.toml` 强制 **静态链接**，避免运行时依赖泄漏和 Docker 构建失败。<br>- `pcre2` 版本 0.2.11 对应的 upstream 仍在维护，但需关注其 CVE（如 2023‑xxxxx）并在后续发布中及时升级。<br>- `fastokens` 本身为纯 Rust 实现，安全风险相对低；已在代码中对加载错误进行回退，防止因后端异常导致服务不可用。 |
| **可维护性** | - 新增 `fastokens` 模块与统一的 `Tokenizer` trait 实现，保持现有抽象层不变，代码路径清晰。<br>- 依赖升级导致 `Cargo.lock` 大幅变动（新增大量 crates），可能增加编译时间与二进制体积。<br>- 通过 `env` 检查和前端参数校验，避免非法值导致 panic。 |
| **兼容性** | - 仅对 **BPE**（`TokenizerKind::HfTokenizerJson`）模型生效；TikToken (`.model/.tiktoken`) 模型不受影响，保持原行为。<br>- 对非 UTF‑8 路径的 tokenizer.json 会自动回退到默认实现，兼容特殊文件系统。 |

---

### ⚠️ 潜在风险

1. **二进制体积激增**：静态链接 `pcre2` 以及 `fastokens`、`rayon`、`daachorse` 等依赖会显著增大 `dynamo` 可执行文件（预计 +30‑50 MB），可能影响容器镜像大小和冷启动时间。  
2. **跨平台编译**：`pcre2-sys` 需要系统的 C 编译工具链；在 Windows 或某些裸金属环境下可能出现构建失败。  
3. **依赖安全性**：`pcre2` 与 `fancy-regex` 等库的历史 CVE 需要在 CI 中持续监控，防止引入未修补的漏洞。  
4. **行为差异**：虽然单元测试验证了 `fastokens` 与 HuggingFace 对同一 BPE 模型产生相同 token ID，但在极端字符、非标准 vocab 或自定义 `normalizer`/`post_processor` 场景下仍可能出现差异。  
5. **回退策略**：当前在 `DYN_TOKENIZER=fastokens` 且加载失败时仅记录 warning 并继续使用默认实现；若用户强制依赖 `fastokens`（如在脚本中检查 env），可能误以为性能提升成功而忽略潜在错误。  

---

### 💡 关注建议

1. **性能基准**：在 CI 中加入基准测试，对比 `fastokens` 与 HuggingFace 编码的时延、吞吐（尤其在多线程情况下），并在发布说明中提供量化数据。  
2. **二进制压缩**：考虑在 Release CI 中使用 `strip`、`upx`（或 `cargo-bloat` 评估）压缩静态链接的二进制，或者提供 `--features=fastokens` 可选编译，以免在不需要该功能的环境中携带额外体积。  
3. **安全审计**：加入 Dependabot / cargo-audit 检查，确保 `pcre2`、`fancy-regex`、`fastokens` 等库的安全更新被及时拉取。  
4. **文档完善**：在 README 与部署手册中明确说明 `fastokens` 只适用于 BPE tokenizer，解释 `DYN_TOKENIZER` 环境变量的作用与回退行为。  
5. **平台兼容性验证**：在 Windows 与 macOS CI 中显式测试 `pcre2-sys` 编译路径，若出现链接错误，考虑提供预编译的 `pcre2` 动态库或使用 `pkg-config` 进行平台自适应。  
6. **监控指标**：在服务启动时记录 `tokenizer_backend`（default/fastokens）到日志或监控标签，以便后期分析实际性能收益与错误率。  
7. **回退策略加强**：如果用户显式设置 `DYN_TOKENIZER=fastokens` 且加载失败，可提供 `--tokenizer-fallback=default` 选项，让用户自行决定是否继续运行或退出。  

--- 

**总体结论**：此次引入 `fastokens` BPE 编码后端为 Dynamo 带来了显著的性能提升潜力，同时保持了向后兼容和安全的回退机制。关键在于控制二进制大小、确保跨平台构建以及持续关注新依赖的安全状态。适当的文档、监控与基准支撑将帮助用户安全、可靠地使用该特性。

### feat(kv-indexer): add python launcher follow-up [DYN-2459] (#7395)
**SHA**: `56b448a` | 🔗 [查看提交](https://github.com/ai-dynamo/dynamo/commit/56b448a6075d6c5cd6dde80cbeab485dd90c2497)

**🎯 变更类型**：功能增强（feat）  
**⚡ 重要程度**：🔴高  

**📋 变更摘要**  
- 为 KV 缓存索引器（`kv-indexer`）新增 Python 可执行入口 `python -m dynamo.indexer`，通过 `maturin` 构建的 Python 包直接提供 HTTP 服务。  
- 在 `Cargo.toml` 中添加 `kv-indexer`、`kv-indexer-runtime`、`kv-indexer-metrics` 三个可选特性，并在 `Dockerfile`、文档、测试等多处同步开启对应编译标志。  
- 引入 `clap`、`tracing-subscriber` 等依赖，完善命令行解析、日志初始化及错误向 Python 抛出的映射。  
- 更新依赖版本（`anstyle`、`cc`、`serde_with`、`tinyvec`、`tracing-subscriber` 等），提升安全性与兼容性。  
- 大幅扩展测试套件，加入对独立索引器启动、P2P 恢复、ZMQ 事件流等场景的端到端验证。  

**🎯 影响范围**  
- **核心库**：`kv-router`（standalone_indexer 模块），新增 `runtime` 集成路径。  
- **Python 绑定**：`lib/bindings/python`（`Cargo.toml`、`lib.rs`、`llm/kv.rs`），对外暴露 `run_kv_indexer`。  
- **构建脚本**：`container/templates/wheel_builder.Dockerfile`（maturin 构建特性）。  
- **文档**：`docs/components/router/standalone-indexer.md`（说明运行模式与特性）。  
- **测试**：`tests/router/*`（新增索引器同步、ZMQ 监听激活、全流程 P2P 恢复等）。  
- **依赖锁文件**：`Cargo.lock`、`lib/bindings/*/Cargo.lock`（数十个 crate 版本升级）。  

**🔍 技术洞察**  

- **架构影响**  
  - **新增可执行入口**：把原本仅在 Rust 二进制中提供的 KV 索引器功能搬到 Python 包，使用户可以在同一 `pip install dynamo` 包中直接启动索引服务，降低部署复杂度。  
  - **特性化编译**：通过 Cargo 的 `optional = true` 与 `features` 机制，把索引器的依赖（`dynamo-kv-router`、`clap`、`tracing-subscriber`）设为可选，保持原有 “runtime‑only” 发行版不受影响。  
  - **运行时模式切换**：  
    - **Standalone**：仅依赖 ZMQ KV 事件流，不需要 Dynamo Runtime 的 discovery、event‑plane、request‑plane。  
    - **Runtime 集成**：加入 `--dynamo-runtime` 标记时，会启动 `dynamo-runtime` Worker，自动完成服务注册与事件管道接入。  
  - **日志初始化分支**：新增环境变量 `DYNAMO_SKIP_PYTHON_LOG_INIT` 让用户在自行调用 `tracing_subscriber` 初始化时可以跳过默认日志配置，防止两次初始化冲突。  

- **性能影响**  
  - **启动成本**：Python 包启动时会创建一个 Tokio 运行时（单独进程）并初始化 `tracing-subscriber`，相对纯 Rust 二进制略有增加（约 10‑20 ms），但对长期服务影响可忽略。  
  - **二进制体积**：新增 `clap`、`tracing-subscriber` 等依赖会导致 `dynamo‑indexer` 可执行文件体积增长约 2‑3 MB（主要是编译时的 feature 代码），对容器层面的镜像大小有小幅提升。  
  - **并发能力**：`threads` 参数默认 4，使用 `ThreadPoolIndexer`，在多核机器上可提升索引查询吞吐，且对已有的 ZMQ 事件流不会产生额外竞争。  

- **安全考虑**  
  - **输入解析**：使用 `clap::Parser` 对所有 CLI 参数进行强类型解析，防止非法字符串注入。  
  - **日志泄露**：新增 `DYNAMO_SKIP_PYTHON_LOG_INIT` 环境变量仅控制日志初始化顺序，不会导致信息泄露。  
  - **依赖升级**：升级的 `anstyle`、`tracing-subscriber`、`serde_with` 等均为安全更新，修复了过去的 CVE（如 `anstyle` 中的 Unicode 转义漏洞）。  
  - **网络暴露**：索引器默认在 `0.0.0.0:8090` 上监听 HTTP，使用 `--namespace`、`--component_name` 时会向 Runtime 注册，潜在的服务发现泄漏仍受 Runtime 的访问控制策略约束。  

**⚠️ 潜在风险**  

| 风险点 | 说明 | 严重度 |
|--------|------|--------|
| **特性兼容性** | 若用户在未显式开启 `kv-indexer` 特性下调用 `run_kv_indexer`，会触发 `anyhow::bail!` 并抛出 Python `SystemExit`，可能在 CI 中出现不可预期的错误。 | 中 |
| **依赖版本冲突** | 大幅升级 `clap`、`tracing-subscriber` 等，可能导致下游项目（例如自定义 `dynamo-runtime`）在特定平台上出现编译错误。 | 中 |
| **双日志初始化** | 当用户自行在 Python 程序中初始化 `tracing_subscriber` 并未设置 `DYNAMO_SKIP_PYTHON_LOG_INIT`，会导致日志系统 panic。 | 低 |
| **ZMQ 监听慢加入** | 新增的 `launch_indexer` 逻辑依赖 1 s 延迟来规避慢加入窗口，若环境中 CPU/IO 延迟异常大，仍可能出现事件丢失。 | 中 |
| **测试资源竞争** | `tests/router` 引入并行启动两套索引器进程，若 CI 环境端口分配冲突或资源不足，可能出现间歇性失败。 | 中 |
| **运行时配置错误** | `--dynamo-runtime` 模式要求 `dynamo-runtime` 正确配置环境变量；缺失时会在 `worker.execute` 内 panic，导致容器启动失败。 | 中 |

**💡 关注建议**  

1. **CI / 发布**  
   - 在 CI 中明确使用 `--features kv-indexer` 编译 Python 包，并添加一个简单的 smoke‑test（`python -c "import dynamo; dynamo.run_kv_indexer(['--help'])"`）验证入口可用。  
   - 为防止版本冲突，保持 `Cargo.lock` 在根目录和子 crate（`bindings/python`、`kv-router` 等）同步更新；可在 CI 中加入 `cargo update -p <crate>` 的检查。
