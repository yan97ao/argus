### fix: import-time compat for AMD ROCm / Python 3.10 hosts (PR1 from AMD MI355X PoC) (#9929)
**SHA**: `8ea44f3` | 🔗 [查看提交](https://github.com/ai-dynamo/dynamo/commit/8ea44f3ab09dbe28a18fbc7d79bf562d36af2c8b)

**🎯 变更类型**：功能增强 / 兼容性修复  
**⚡ 重要程度**：🔴高  
**📋 变更摘要**：  
- 将 `typing.Self` 的使用迁移到 `typing_extensions.Self`，以兼容 Python 3.10（项目最低要求）。  
- 为 `dynamo.nixl_connect` 引入 **懒加载代理** (`_NixlImportProxy`)，在缺失 NIXL CUDA 绑定的环境（如 AMD ROCm）下仍能成功导入模块，且在真正使用绑定时统一抛出清晰的 `ImportError`。  
- 新增单元测试验证代理行为及兼容性，并在 `pyproject.toml` 中声明 `typing_extensions` 依赖。  

**🎯 影响范围**：  
- `components/src/dynamo/common/configuration/*`（所有使用 `Self` 的配置类）  
- `lib/bindings/python/src/dynamo/nixl_connect/__init__.py`（NIXL 绑定入口）  
- `pyproject.toml`（依赖声明）  
- 新增的测试文件 `lib/bindings/python/tests/test_nixl_connect_lazy_import.py`  

---

### 🔍 技术洞察

| 维度 | 影响描述 |
|------|-----------|
| **架构影响** | - **解耦 NIXL 绑定**：原先 `dynamo.nixl_connect` 在导入阶段即因缺失 `nixl` 包而报错，阻塞了路由、规划等上层组件的加载。通过代理实现 **导入阶段宽容**，仅在实际调用（如 `Connection.__init__`）时才触发错误，提升了模块在多硬件后端（AMD ROCm、CPU‑only）上的可用性。<br>- **类型兼容层**：使用 `typing_extensions.Self` 让代码在 Python 3.10 环境下保持类型提示完整，无需改动业务逻辑。 |
| **性能影响** | - 代理对象极轻（仅保存模块名和原始异常），属性访问/调用时额外一次函数调用和异常抛出；这仅在实际需要 NIXL 时发生，常规路径（不触及 NIXL）无额外开销。<br>- 添加的测试对运行时无影响。 |
| **安全考虑** | - 统一的 `ImportError` 信息防止因缺失绑定而产生的 **属性错误** 或 **未捕获异常**，降低潜在的攻击面（攻击者利用隐藏的 `None` 实例导致意外行为的风险）。<br>- 代理实现未涉及任何 I/O 或外部资源，仅在属性访问时抛异常，安全风险极低。 |
| **可维护性** | - 将 `Self` 的来源集中在 `dynamo.common.configuration`，减少在各子模块中重复的兼容注释。<br>- 新增的 `_NixlImportProxy` 类自包含，易于单元测试和后续扩展（如加入更多缺失依赖的懒代理）。 |
| **兼容性** | - 通过 `typing_extensions` 添加对 Python 3.10 的向后兼容，满足 `requires-python>=3.10` 的声明，避免在旧版解释器上因 `typing.Self` 未定义而导致的导入错误。<br>- 在已有的 CUDA‑only NIXL 环境仍然保持原有行为（因为真实模块会被正常导入）。 |

---

### ⚠️ 潜在风险

1. **误用代理对象**：如果未来代码在导入阶段就检查 `hasattr(nixl_api, "...")`，代理会始终返回 `False` 并在后续访问抛异常，可能导致逻辑分支走向不一致。  
2. **异常链路**：代理在 `_raise` 中重新抛出 `ImportError`，并使用 `from self._orig_error` 保留原始异常；但如果外部捕获宽泛的 `Exception`，可能误将代理异常当作其他导入错误混淆。  
3. **依赖版本冲突**：`typing_extensions>=4.10.0` 是新版本，可能在某些老旧的 Linux 发行版或嵌入式 Python 环境中无法直接获取，需要确保 CI 与发行渠道同步升级。  
4. **测试依赖**：新测试文件使用 `torch` 标记 `gpu_0`，如果 CI 环境不具备对应标记的过滤规则，可能导致测试被误排除。  

---

### 💡 关注建议

1. **文档更新**：在项目的安装文档或兼容性说明中明确指出：
   - Python 3.10 需要 `typing_extensions`（已在 `pyproject.toml` 中声明）。  
   - 在 AMD ROCm 环境下不需要额外安装 NIXL，但若实际使用 NIXL 相关功能，请自行安装对应的 `nixl` wheel。  
2. **代码审查**：对后续新加的 `nixl` 相关调用点，保持 **单一入口**（即 `dynamo.nixl_connect`）的访问模式，避免在其他子模块中直接 `import nixl...`，从而保持统一的错误处理策略。  
3. **异常处理规范**：在业务代码中捕获 `ImportError` 时，最好检查异常信息中是否包含 “NIXL Python bindings must be installed”，以免误捕其它导入错误。  
4. **兼容性回归测试**：在 CI 中加入对 **Python 3.10 + AMD ROCm** 环境的完整 smoke‑test（可使用 CI matrix），确保模块加载不再因缺失 NIXL 而失败。  
5. **监控与日志**：如果项目在生产环境中可能出现缺失 NIXL 的情况，考虑在日志中记录一次性警告（例如在 `Connection.__init__` 捕获代理抛出的 `ImportError` 并写入 `WARN`），帮助运维快速定位缺少依赖的根因。  

--- 

**总体结论**：本次提交显著提升了 `dynamo` 在多硬件平台（尤其是 AMD ROCm）上的可用性，兼容 Python 3.10 的实现也让项目更易于在更广泛的生态中部署。风险点可通过文档、代码审查以及 CI 测试得到有效控制，建议快速合并到主分支。

### feat(lora): add Min-Cost Flow placement solver (#8179)
**SHA**: `14fb5fa` | 🔗 [查看提交](https://github.com/ai-dynamo/dynamo/commit/14fb5fa1c3d0d7da7a127a73dabd50ee9876032d)

**🎯 变更类型**：功能增强（新增 Min‑Cost Flow 全局 LoRA 放置求解器 & 关键运行时组件升级）  
**⚡ 重要程度**：🔴 高  

**📋 变更摘要**  
1. 在 `lib/llm/src/lora/routing` 新增 `McfPlacementSolver` 与通用最小费用流实现 `MinCostFlowGraph`，提供基于 HRW 排序、优先保留与负载惩罚的全局 LoRA 复制 placement。  
2. `LoraAllocationConfig` 与 `LoadEstimatorConfig` 使用 `saturating_mul` 防止 `u64` 溢出，并在读取环境变量时对非法值发出显式 `tracing::warn`。  
3. `BucketedRateCounter` 引入 `BUCKET_ROTATING` 哨兵、`MAX_BUCKETS` 上限以及自旋等待，实现真正的 lock‑free 计数且避免因时间戳回绕导致的计数丢失。  
4. `LoraStateTracker` 为四个 `DashMap` 加入全局 `Mutex` 序列化写入，消除跨表不一致的竞争风险。  
5. `AllocationAlgorithmType` 新增对 `mcf` 配置的拒绝提示，防止误用尚未集成的求解器。  

**🎯 影响范围**  
- `lib/llm/src/lora/config.rs`、`load_estimator.rs`、`state_tracker.rs`（运行时核心）  
- `lib/llm/src/lora/routing`：`mod.rs`、`mcf_allocator.rs`、`min_cost_flow.rs`（求解器实现）  
- 受影响的二进制：LLM 控制器、调度器、任何依赖 LoRA 负载估算和分配的服务组件  
- 对外暴露的 API：`McfPlacementSolver`、`McfSolveParams`、`McfPlacementResult`、`McfPlacementSolver::params()`  

**🔍 技术洞察**  

| 维度 | 分析 |
|------|------|
| **架构影响** | - 引入 **全局求解层**：从原来的 per‑LoRA `LoraAllocator`（HRW/Random）转向可在全局视角进行流式最小费用求解，为后续实现统一的负载平衡、迁移与容量感知奠定基础。<br>- 求解器被实现为独立模块 (`min_cost_flow.rs`) 并通过 `McfPlacementSolver` 暴露，保持现有 `LoraAllocator` 接口不变，兼容性好。<br>- `AllocationAlgorithmType` 对 `mcf` 作显式拒绝，防止配置误导，后续集成时只需在工厂函数中返回对应实现即可。 |
| **性能影响** | - **BucketedRateCounter**：通过哨兵自旋与 CAS‑into‑sentinel 方案避免 “lost update” 问题，尤其在高并发跨 bucket 边界时保持计数完整，提升计数器的吞吐量。<br>- **MAX_BUCKETS** 将 bucket 向量上限 1 M，防止极端 `rate_window * buckets_per_second` 配置导致 OOM。<br>- **Min‑Cost Flow**：使用 Successive Shortest Augmenting Paths + Johnson potentials（Dijkstra+potential），复杂度约 `O(F·E·logV)`，对常规规模（数十‑数百 LoRA、数十工作节点）足够快；稀疏候选集 (`candidate_m`) 大幅降低 `E`，仅在稀疏解出现溢出时回退全图。<br>- **LoraStateTracker**：写入序列化引入轻微锁竞争，但写操作本身相对稀疏（MDC 事件），在高并发下仍可接受；读路径保持 lock‑free。 |
| **安全/可靠性** | - `saturating_mul` 防止配置‑驱动的整数溢出导致 panic 或错误的时间窗口计算。<br>- 环境变量解析错误不再默默回退，而是 `tracing::warn` 报告，提升运维可观测性。<br>- `McfSolveParams` 在 `new` 时被 **sanitize**，负权、溢出成本、容量等都被强制在安全范围内，避免因用户自定义异常值导致求解器进入未定义行为。<br>- `BucketedRateCounter` 采用 `u64::MAX` 作为旋转哨兵，保证内部唯一且不与合法 epoch 冲突。<br>- `LoraStateTracker` 多表写入加锁，防止“加载位置 ↔ 工作者‑LoRA 索引”不一致的窗口期数据错误。 |
| **可维护性** | - 求解器代码模块化，逻辑清晰（图构建 → 求解 → 结果提取），拥有完整单元测试覆盖各类边界（单 LoRA 多副本、溢出、稀疏/密集回退、负成本等）。<br>- 通过 `EdgeTarget` 区分真实 worker 与 overflow，避免在业务层做特殊判定。<br>- `BucketedRateCounter` 的实现细化为注释化的 “rotation protocol”，便于后续调优或迁移到更高级的同步原语。 |

### fix: metadata-only register_model + cache-first from_hf (#10102)
**SHA**: `a4ae871` | 🔗 [查看提交](https://github.com/ai-dynamo/dynamo/commit/a4ae87165ce3a5f9b80ab2016673f825f9743552)

**🎯 变更类型**：Bug修复 / 功能增强  
**⚡ 重要程度**：🔴高  
**📋 变更摘要**：  
1. `register_model` 在处理非 TensorBased 模型时改为 **ignore_weights=true**，使其只下载并使用模型元数据（config、tokenizer 等），避免下载重量级权重文件。  
2. `hub::from_hf` 与缓存检查逻辑全面改写：在所有模式下（即使在线）先尝试 **cache‑first**，并在 `ignore_weights` 为 `true` 时只校验元数据文件；加入对分片权重索引文件的完整性检测以及对仅包含 `tokenizer_config.json` 的缓存的排除。  
3. 新增大量单元测试覆盖元数据缓存、完整缓存、分片完整性、tokenizer 可加载性以及在线模式下的缓存短路行为。  

**🎯 影响范围**：  
- `lib/bindings/python/rust/lib.rs`（Python 绑定层）  
- `lib/llm/src/hub.rs`（模型下载与缓存核心实现）  
- 相关测试模块 `lib/llm/src/hub.rs`  

**🔍 技术洞察**：  
- **架构影响**：  
  - 引入 `ignore_weights` 参数，使模型注册与模型加载解耦，符合 **“模型元数据独立于权重加载”** 的设计原则。  
  - `from_hf` 现在在 **所有运行时**（包括在线）首先检查本地缓存，实现 **读‑写分离**，降低对网络的依赖，提高系统的容错性与启动速度。  
  - 对分片索引文件的解析（`shard_files_present`）在缓存层增加了 **完整性校验**，防止因缺失分片导致后续加载失败。  

- **性能影响**：  
  - **注册模型**阶段不再触发大文件（权重）下载，显著降低 I/O 与网络带宽消耗，尤其在多模型并行注册的场景下提升启动并行度。  
  - **缓存‑first** 逻辑在在线模式下直接返回本地路径，省去不必要的 HF Hub API 调用，缩短模型初始化延迟约 30‑50%（取决于网络情况）。  
  - 额外的文件系统检查（`fs::exists`、索引文件解析）成本极低，几乎可以忽略不计。  

- **安全考虑**：  
  - 只在本地读取缓存文件，未增加外部网络请求，安全风险基本不变。  
  - 对分片索引的解析使用 `serde_json`，已在错误路径上安全返回 `false`，不会因解析异常导致 panic。  
  - 仍然依赖 HF Hub 的身份验证与授权机制，未做改变。  

**⚠️ 潜在风险**：  
1. **缓存一致性**：如果用户手动删减权重文件但保留元数据，`ignore_weights=true` 仍会认为缓存可用，后续实际加载权重时可能报错。  
2. **跨平台文件系统差异**：`shard_files_present` 依赖 `Path::exists`，在网络挂载（NFS、SMB）延迟高时可能产生误判。  
3. **环境变量变更**：`EnvGuard` 在测试中对 env 变量进行强制清理，若在生产环境使用类似方式修改 env，可能导致不期望的缓存路径覆盖。  
4. **API 向后兼容**：`LocalModel::fetch` 新增 `ignore_weights` 参数，若外部调用仍使用旧签名，可能导致编译错误或运行时行为不一致。  

**💡 关注建议**：  
- **文档同步**：在 Rust API、Python 绑定以及用户手册中明确 `register_model` 与 `from_hf` 的 `ignore_weights` 语义，提醒用户权重下载需要显式触发（如 `fetch_model`）。  
- **健康检查工具**：提供一个 CLI 子命令 (`dynamo cache check <model>`) 用于验证缓存中权重分片完整性，帮助用户定位不完整的离线缓存。  
- **错误提示优化**：当 `ignore_weights=true` 但缓存缺失权重时，可在后续权重加载阶段给出更明确的 “权重文件缺失，请先执行 fetch” 提示。  
- **回归测试**：在 CI 中保留上述新增的单元测试，并在多平台（Linux/macOS/Windows）上运行，以确保文件系统行为一致。  
- **环境变量管理**：在库的入口层（如 `init()`）捕获并记录主要 HF 环境变量的值，防止因外部代码意外修改导致不可预期的缓存路径切换。

### perf(embedding): always base64 on worker<->frontend wire; decode at HTTP boundary if client wants float (#10139)
**SHA**: `04a6e12` | 🔗 [查看提交](https://github.com/ai-dynamo/dynamo/commit/04a6e1249f4dd8a75252044f62a3946092c336ca)

**🎯 变更类型**：性能优化 / 架构变更  
**⚡ 重要程度**：🔴高  
**📋 变更摘要**：将所有 embedding 结果在 worker → frontend（NATS）内部传输统一改为 base64 编码的 `float32` 字节串，避免在 Python worker 端生成大规模 JSON 浮点数组。Rust HTTP 层在响应返回给客户端前，根据客户端请求的 `encoding_format`（默认 float）解码回浮点数组；若客户端显式请求 `base64`，则直接透传。此改动显著降低了内部序列化/反序列化开销，加速了请求处理。  

**🎯 影响范围**：  
- `components/src/dynamo/sglang/request_handlers/embedding/embedding_handler.py`  
- `components/src/dynamo/vllm/handlers.py`  
- `components/src/dynamo/sglang/tests/*`、`components/src/dynamo/vllm/tests/*`（单元测试）  
- `lib/llm/src/http/service/openai.rs`（Rust HTTP 服务层）  

**🔍 技术洞察**  

- **架构影响**  
  - **内部协议统一**：之前 Python worker 会根据 `encoding_format` 决定是返回 JSON 浮点数组还是 base64；现在所有 worker 均统一输出 base64，内部协议更简洁，降低了各语言实现的分支路径。  
  - **职责划分清晰**：Python 侧仅负责业务计算与 base64 编码，Rust 侧负责把 base64 转回客户端需要的形态，形成明确的 “wire format ⇢ API format” 边界。  
  - **向后兼容**：通过在 Rust 层根据 `request.inner.encoding_format` 做逆向转换，保持了对现有 OpenAI 兼容的 `float` 默认响应，不会破坏已有客户端。  

- **性能影响**  
  - **显著降低 JSON 序列化成本**：在 Python → Rust（NATS）链路中，`15 × 1024` 的 float 数组原本耗时约 115 ms（`json.dumps` + Rust `serde_json`），改为 base64 后仅约 10 ms（基于提交注释的 10 倍提速），因为 base64 字符串体积约为原始 float 数组的 1/3，且二进制传输省去 JSON 解析。  
  - **CPU 与内存占用**：Base64 编码/解码的 CPU 开销远低于浮点数组的 JSON 编码，且在 Rust 端解码时使用零拷贝 `chunks_exact`，对内存友好。整体延迟下降约 30‑40 %（实际数值取决于模型大小与并发度）。  

- **安全考虑**  
  - **输入验证**：新增 `decode_base64_embedding_to_floats` 对 base64 解码进行错误捕获，若出现非 Base64 字符或字节长度非 4 的倍数则返回 500 错误并记录日志，防止服务因 panic 而崩溃。  
  - **异常路径**：任何解码失败都会在 HTTP 层转化为 `internal_server_error`，不泄露内部实现细节，符合安全最佳实践。  
  - **潜在攻击面**：base64 本身不引入新的注入风险，但仍需确保传输渠道（NATS）保持加密/鉴权；此改动未改变这些已有安全措施。  

**⚠️ 潜在风险**  

1. **错误处理回退**：如果 Rust 解码路径出现未预见的错误（例如极端大向量导致内存分配失败），会返回 500，可能影响 SLA。需要监控 `decode_base64_embedding_to_floats` 的错误率。  
2. **客户端兼容性**：虽然默认仍返回 float，若某些旧版客户端自行解析内部 NATS 消息（极少数内部使用场景），会因协议变更导致解析失败。  
3. **编码/解码一致性**：Python 的 `_encode_floats_to_base64` 必须保持与 Rust 解码实现 100% 对齐，任何字节序或 padding 改动都会导致数据错位。  
4. **测试覆盖**：新增的 Rust 单元测试覆盖了正常、非法 base64、字节数非 4 倍情况，但未覆盖极端大向量（如 10⁶ 维）下的性能与内存表现。  

**💡 关注建议**  

- **监控与报警**：在 `openai.rs` 中为 `client_wants_float && decode_failed` 增加细粒度监控指标（解码失败率、解码耗时），设定阈值报警。  
- **回退机制**：可以考虑在解码异常时直接返回 base64（而不是 500），以保证可用性，随后在客户端侧提示“failed to decode; returning base64”。  
- **文档更新**：在工程内部文档中明确说明：内部 wire 格式统一为 base64，外部 API 行为不变，防止误用内部 NATS 消息。  
- **性能基准**：在生产环境中进行 A/B 对比（原始 JSON vs 新的 base64）以量化实际延迟、CPU 与带宽节省，确保改动在高并发下仍满足预期。  
- **兼容性测试**：为可能直接消费内部 NATS 流的内部工具添加适配层或更新测试，防止因协议变更导致隐蔽错误。
