### feat: autodetect agent behavior in agent traces (#9817)
**SHA**: `39251bc` | 🔗 [查看提交](https://github.com/ai-dynamo/dynamo/commit/39251bcf13f854b305331101014d2a3980a2aab6)

**🎯 变更类型**：功能增强、架构变更、可观测性增强  

**⚡ 重要程度**：🔴高  

**📋 变更摘要**  
本次提交为 Dynamo **agent trace** 添加了对请求结束时“完成原因”(finish reason) 的自动检测与记录。核心改动包括：  
1. 在 `AgentTraceRecord` 中新增 `finish_reason_metadata` 结构体，用以保存后端原始 finish reason、stop reason、工具调用信息以及每个 choice 的独立 finish metadata。  
2. 引入 `SharedFinishReasonMetadata` 线程安全的共享状态，负责在后端流、Chat/Completion 解析阶段实时收集并合并上述信息。  
3. 在 `OpenAIPreprocessor`、`wrap_agent_trace_*_request_end_stream` 等关键路径中接入该状态，确保所有 LLM 流程（包括非流式）都能记录。  
4. 更新 Benchmarks、文档、Perfetto 转换脚本，以展示和度量新的元数据开销。  

**🎯 影响范围**  
- `lib/llm/src/agents/trace/*`（结构体、集成层、记录、Sink）  
- `lib/llm/src/preprocessor.rs`（流处理、后端 finish reason 注入）  
- `benchmarks/agent_trace/*`（Perfetto 转换、验证、文档）  
- `lib/llm/benches/agent_trace_finish_metadata.rs`（新基准）  
- Cargo features: 新增 `agent-trace-bench` 用于基准；影响 `Cargo.toml` 中的 bench 列表。

### 🔍 技术洞察  

| 维度 | 影响描述 |
|------|----------|
| **架构影响** | - **新增共享状态** `SharedFinishReasonMetadata`（Arc\<Mutex\>)，在多线程/异步环境中安全收集 finish metadata。 <br>- **追踪链路扩展**：`wrap_agent_trace_chat_request_end_stream` 与 `wrap_agent_trace_completion_request_end_stream` 分别对 Chat 与 Completion 流进行拦截，记录 tool‑call 与 finish 信息，然后统一走 `wrap_agent_trace_request_end_stream` 生成统一的 `RequestEnd` 记录。 <br>- **API 向后兼容**：`AgentTraceRequestEndState` 新增字段 `finish_reason_metadata`，默认 `None`，不影响已有代码。 <br>- **文档/转换层**：Perfetto 转换脚本现在会把这些平铺的字段写入请求 slice，Mooncake replay 则仍旧忽略（保持既有行为）。 |
| **性能影响** | - **CPU**：在每个后端 token输出 (`BackendOutput`) 上会执行一次 `record_backend_finish_reason_metadata`，以及在 Chat/Completion chunk 解析时额外遍历 tool‑call 列表，预计 **< 0.5 µs/ token** 的额外开销（基准 `agent_trace_finish_metadata` 已测量）。 <br>- **内存**：`FinishReasonMetadata` 在最坏情况下（大量 tool‑calls、choices）会存储若干 `String`（id、name），每条约 24 B + 内容大小，整体量级仍然是 **O(#tool_calls + #choices)**，对常规请求几乎可以忽略。 <br>- **锁竞争**：使用 `parking_lot::Mutex`，在高并发的长时间请求（上千 token + 多 tool‑calls）可能出现轻微争用；但因为大多数记录在单个请求的上下文中完成，争用窗口极短。 |
| **安全考虑** | - **信息泄露**：`ToolCallMetadata` 仅保存 `id` 与 `name`（无参数），已在注释中说明“arguments are intentionally omitted”。因此风险有限。 <br>- **可觀測性滥用**：如果外部系统读取 Perfetto 导出的 trace，能够看到模型完成的具体 tool‑call 名称，可能被用于逆向模型行为；建议在生产环境中通过配置关闭 `agent-trace` 或在导出前过滤 `finish_reason_metadata`。 <br>- **特权访问**：`SharedFinishReasonMetadata` 通过 `Arc` 在整个请求生命周期共享，未暴露给未授权代码，安全边界与现有 trace 体系相同。 |
| **可维护性** | - **代码分离**：大部分新逻辑被封装在 `integration.rs` 与 `types.rs`，对核心业务路径改动最小。 <br>- **单元测试**：新增 `#[tokio::test]` 覆盖 Chat 与 Completion 两类结束记录的路径，确保元数据正确写入。 <br>- **基准**：提供 `agent_trace_finish_metadata` 基准，可在未来检测性能回归。 |
| **兼容性** | - **向后兼容**：旧的 trace 消费者（如 Mooncake replay）依旧忽略新字段。 <br>- **序列化**：`serde(skip_serializing_if = "Option::is_none")` 保证在不记录时不产生额外字段，避免破坏已有 schema。 |

---

**⚠️ 潜在风险**  

1. **锁竞争导致延迟**  
   - 在极端高并发、单请求携带大量 tool‑calls（如 10k）时，`Mutex` 可能成为瓶颈。  
2. **元数据体积膨胀**  
   - 大量 tool‑call（尤其是每次都有 `id`/`name`）会让 trace 体积显著上升，影响存储与传输成本。  
3. **隐私/合规**  
   - 虽然不记录工具参数，但工具 *名称* 本身可能泄露业务逻辑，需在合规环境中审查。  
4. **错误的后端 finish_reason 映射**  
   - `record_backend_finish_reason_metadata` 直接把 `BackendFinishReason::Stop` 转为字符串 `"stop"`；若后端新增其他枚举值，未同步更新可能导致信息缺失。

**💡 关注建议**  

| 给开发者的建议 | 给使用者/部署者的建议 |
|----------------|----------------------|
| **性能监控**：在生产环境开启 `agent-trace` 时，使用新基准 `agent_trace_finish_metadata` 对比开启/关闭的延迟，关注 `Mutex` 持有时间（可在 `record_backend_finish_reason_metadata` 前后加 `tracing::debug!`）。 | **控制采样**：如果 trace 数据量成为瓶颈，使用 Dynamo 配置关闭 `finish_reason_metadata`（仍保留基本 token‑level trace）。 |
| **审计**：在审计/合规需求下，确保导出的 Perfetto trace 在交付前使用脚本剔除 `finish.*` 字段。 | **文档宣传**：更新监控仪表盘说明，新字段 `finish.*` 只在 **Perfetto** 视图中可见，Mooncake replay 不受影响。 |
| **代码演进**：如果后端引入新 finish reason 类型，扩展 `record_backend_finish_reason_metadata` 的映射逻辑并添加对应单元测试。 | **Feature Flags**：在 CI 中开启 `agent-trace-bench` 只在 benchmark 场景使用，防止误将其推到生产 crate。 |
| **异常处理**：在 `wrap_agent_trace_*_request_end_stream` 中加入 `debug_assert!` 检查 `metadata.snapshot()` 不为 `None` 时字段完整性，防止遗漏。 | **存储规划**：对长期保存的 trace，考虑对 `finish_reason_metadata` 进行压缩（如使用 protobuf/flatbuffer）以降低磁盘占用。 |

**总体结论**  
此次改动为 Dynamo 的 **agent trace** 引入了可观测的完成原因与工具调用元数据，提升了调试、性能分析与错误定位能力。架构上保持了向后兼容，新增的锁与字段带来的性能与存储开销在常规使用场景下可接受，并已配套基准与测试验证。唯一需要重点关注的是在极端高并发/高 tool‑call 场景下的锁竞争以及在合规环境中对工具名称的泄露风险，建议通过采样、字段过滤或配置关闭来治理。

## ⚠️ 潜在风险  

1. **CI / 环境依赖**  
   - 多 GPU 测试 (`gpu_2`) 需要两块可用的 GPU；在不具备此硬件的 CI 环境会导致测试卡死或被跳过，可能隐藏回归。  
   - `agg_embed_multiworker.sh` 假设系统端口 `8081`、`8082` 可用，若已有服务占用这些端口会导致启动失败。  

2. **健康检查超时**  
   - 健康检查现在执行一次真实的 forward 计算，若模型加载慢或 GPU 资源争抢，探针可能超过默认超时导致容器被判为不健康。  

3. **后向兼容**  
   - 现有用户如果在自定义 `DYN_HEALTH_CHECK_PAYLOAD` 中已经提供了 chat‑style payload，且未设置 `DYN_HEALTH_CHECK_ENABLED`，embedding worker 仍会使用默认 payload（`input: probe`），但在极端情况下可能出现 **“missing required 'input'”** 报错。  

4. **日志噪声**  
   - `model` 名称会随每次健康检查写入 worker 日志，若在大规模部署中开启大量 embedding worker，可能产生不必要的日志量。  

5. **命名空间冲突**  
   - 脚本强制为每个 worker 指定唯一 `endpoint`（`embed-worker-1.vllm.generate`、`embed-worker-2.vllm.generate`），但如果用户在同一进程中自行创建相同 endpoint，仍会出现覆盖。

### perf(embedding): submit prompts concurrently via asyncio.gather (#10117)
**SHA**: `d27f61b` | 🔗 [查看提交](https://github.com/ai-dynamo/dynamo/commit/d27f61b1c53e873607fe9c393f6550a38167572e)

**🎯 变更类型**：性能优化 / 重构  
**⚡ 重要程度**：🔴高  
**📋 变更摘要**：  
- 将 `EmbeddingWorkerHandler.generate` 中对每个 prompt 的 `engine.encode` 调用改为在同一个事件循环 tick 中并发提交，通过 `asyncio.gather` 实现批量调度，利用 vLLM 的 continuous‑batching 能力一次性前向计算。  
- 为避免 `gather` 抛异常后遗留的未取消任务，新增了显式的 `finally` 取消逻辑，并在完成后统一 await 这些任务的异常。  
- 补充了两套单元测试，验证 **部分失败时的任务取消** 与 **全部成功时的正常返回**，确保行为可靠。

**🎯 影响范围**：  
- `components/src/dynamo/vllm/handlers.py`（主要业务逻辑）  
- `components/src/dynamo/vllm/tests/test_vllm_worker_handler.py`（新增测试）  
- 依赖 `vLLM` 引擎的所有嵌入（embedding）调用路径

### 🔍 技术洞察

| 维度 | 影响说明 |
|------|----------|
| **架构影响** | - 代码从 **顺序循环** 变为 **并行任务集合**，但仍保持原有的 API 与返回结构，外部不可感知。<br>- 引入了 `asyncio.create_task` + `gather` + `finally‑cancel` 的模式，提升了与 vLLM **continuous‑batching scheduler** 的协同效率。<br>- 通过显式取消未完成任务，防止后台仍占用 vLLM 引擎资源，避免 “资源泄漏” 影响后续请求。 |
| **性能影响** | - **吞吐量提升**：多个 prompt 在一次调度中合并，单次 forward pass 完成所有嵌入计算，相比原来的 N 次顺序调用可降低调度开销和 GPU 轮转时间。<br>- **延迟下降**：在批量足够大时，单个请求的响应时间会明显缩短；在单条请求（batch=1）时几乎无负面影响，仅多出 `gather` 的微小调度开销。<br>- **资源利用率提升**：并发提交让 vLLM 能更好地填满 GPU，提升算力使用率。 |
| **安全考虑** | - 新增的任务取消逻辑在异常路径显式调用 `task.cancel()` 并 `await`，避免了“僵尸协程”继续持有内部句柄，降低了潜在的 **Denial‑of‑Service**（持续占用引擎）风险。<br>- 仍然依赖 `engine_client.abort`（在其它路径会被调用），未改变权限模型，未引入新攻击面。 |

### ⚠️ 潜在风险

1. **并发规模过大**  
   - 当一次请求中 `prompts` 数目非常大（几千以上），一次性创建大量 `Task` 可能导致事件循环调度开销或内存占用激增。建议在上层（API 网关）对 batch 大小做上限控制（例如 ≤ 128）。  

2. **vLLM 引擎内部异常**  
   - 若 vLLM 在批处理阶段抛出异常，`asyncio.gather` 会把异常传播到外层，随后 `finally` 会尝试取消仍在执行的任务。若某些任务已经进入底层 C++ 调用且不响应 Cancel，可能导致资源迟迟释放。可在未来考虑使用 `asyncio.wait_for` 设置超时或在 vLLM 侧实现更快的取消响应。  

3. **测试依赖的 Mock 行为**  
   - 新增的测试使用 `MagicMock`、`AsyncMock`，如果底层实现改变（如 `encode` 返回非生成器），测试可能失效。保持测试与真实 `engine_client.encode` 接口的一致性很重要。  

4. **异常传播顺序**  
   - `asyncio.gather` 默认在第一个异常出现时立即抛出，后续任务仍会继续运行直至被取消。若业务上需要收集所有错误信息，当前实现只能看到第一个错误。若这种需求出现，需要改用 `return_exceptions=True` 并自行聚合。

### 💡 关注建议

1. **批量大小限制**  
   - 在请求校验层加入 `max_embedding_batch_size`（例如 64~128），防止单次请求创建过多协程。  

2. **监控与报警**  
   - 新增指标：`embedding_concurrent_tasks`, `embedding_cancelled_tasks`, `embedding_gather_exceptions`。一旦取消任务异常率升高，可快速定位 vLLM 或网络异常。  

3. **文档更新**  
   - 在项目的 API 文档中注明 “Embedding 请求会在同一 tick 中并发提交”，并说明在异常情况下会自动取消其它子请求。  

4. **回归测试**  
   - 将现有的两套测试加入 CI 的并发/性能基准（如 `pytest-benchmark`），监控平均 latency 与吞吐提升是否符合预期。  

5. **未来扩展**  
   - 若 vLLM 进一步支持 “batch‐wide” 参数（如统一的 pooling 参数），可以将 `_encode_one` 的 `request_id` 生成逻辑抽象为可配置，以便在更高层做 batch 合并（跨请求的全局调度）。

### fix: surface NATS max payload errors (#9769)
**SHA**: `a372599` | 🔗 [查看提交](https://github.com/ai-dynamo/dynamo/commit/a37259950b939f4440087487236e3f413aff59ff)

**🎯 变更类型**：Bug修复 / 功能增强  
**⚡ 重要程度**：🔴高  

**📋 变更摘要**  
- 在 OpenAI HTTP 服务层通过错误链遍历，能够正确识别并从任何层级的 `DynamoError::InvalidArgument`（尤其是 NATS 超大负载错误）转化为 HTTP 400 返回，而不是隐藏为 500。  
- NATS 客户端在发送请求前加入对服务器 `max_payload` 限制的本地提前检测，并返回统一的、对外安全的 `InvalidArgument` 错误信息。  
- 相关单元测试覆盖错误链检测、用户安全信息以及 NetworkManager 对不同请求平面模式的行为。

**🎯 影响范围**  
- `lib/llm/src/http/service/openai.rs`（错误处理与响应生成）  
- `lib/runtime/src/pipeline/network/egress/nats_client.rs`（NATS 请求发送实现）  
- `lib/runtime/src/pipeline/network/manager.rs`（网络管理器创建客户端逻辑）  
- 全局错误定义 `dynamo_runtime::error`（间接使用）  

**🔍 技术洞察**  

- **架构影响**  
  - 引入 `find_dynamo_error_in_chain` 公用函数，将错误链遍历的逻辑抽取，提升了错误处理的可复用性与可维护性。  
  - NATS 客户端现在依赖 `client.server_info().max_payload` 并在构造时快照，避免在运行时频繁查询，提高了层与层之间的职责分离：网络层负责前置校验，业务层只处理统一的 `DynamoError`。  
  - `NetworkManager` 的单元测试保障了不同 `RequestPlaneMode`（TCP vs NATS）下的客户端创建路径，防止在缺失 NATS 客户端时产生不明确的运行时错误。  

- **性能影响**  
  - **正面**：提前在客户端本地检查 payload 大小，避免了向 NATS 服务器发送必然被拒的超大请求，减少了网络 I/O 与服务器端的负载。  
  - **负面**：在每次请求前要读取 `payload.len()` 与已缓存的 `max_payload`，这是一条 O(1) 的轻量比较，对整体吞吐几乎没有可感知影响。  

- **安全考虑**  
  - 对外返回的错误信息已去除敏感实现细节（如 NATS subject、字节计数、服务器配置指示），仅提供“payload 过大，请缩减” 的通用提示，符合最小信息泄露原则。  
  - 通过 `NATS_ERRORS_TOTAL` 计数器记录内部错误，以便运营监控而不暴露给用户。  
  - 仍然使用 `anyhow::anyhow!` 包装 `DynamoError`，保持错误链的可追溯性，便于内部排障。  

**⚠️ 潜在风险**  

1. **快照失效**  
   - `max_payload` 在客户端构造时快照，如果在运行期间 NATS 服务器重新配置了更小的 `max_payload`，客户端可能不会立即感知，导致超大请求仍被发送至服务器并被拒。虽然服务器会再一次返回错误并被转换为 400，但会产生额外的网络往返。  
2. **误判边界**  
   - 检查仅针对 payload 本体大小，不包括 NATS header/subject 的额外字节。极端情况下，payload 接近 `max_payload` 而加上大 Header 仍会被服务器拒绝，导致错误在服务器侧再次产生。  
3. **错误链穿透**  
   - `find_dynamo_error_in_chain` 会在任意层级捕获 `InvalidArgument`，这可能把原本应该映射为其它 HTTP 状态码的内部错误误报为 400。需确保业务上所有 `InvalidArgument` 均确实代表请求错误。  

**💡 关注建议**  

- **监控 & 报警**：在 `NATS_ERRORS_TOTAL` 计数器上添加阈值报警，关注 `max_payload_exceeded` 的增长趋势，以便发现潜在的业务层面输入异常。  
- **动态刷新**：考虑在 `NatsRequestClient` 中实现可选的 `refresh_max_payload()` 接口，在连接重连或定时任务时重新获取服务器的 `max_payload`，降低快照失效风险。  
- **文档更新**：在 SDK/API 文档明确说明 “请求负载大小受部署限制”，并提供查询 `max_payload` 的方式（如通过管理接口），帮助用户自行规避。  
- **回归测试**：加入针对 “header 导致超限” 场景的集成测试，确保服务器端错误仍能被正确映射为 400，且不会泄露内部细节。  
- **代码审查**：审视项目中其他可能出现的 `anyhow::Error` 链，确保所有 `InvalidArgument` 均已在业务层面得到适当的 HTTP 映射，防止意外的错误分类。
