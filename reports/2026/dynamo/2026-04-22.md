### fix(planner): match MDC component field against backend default, not DGD key (#8489)
**SHA**: `2cd4288` | 🔗 [查看提交](https://github.com/ai-dynamo/dynamo/commit/2cd4288a59054ace109b16b2c072ba4af4cdb7da)

**🎯 变更类型**：Bug修复  
**⚡ 重要程度**：🔴高  
**📋 变更摘要**：  
修复了 Dynamo Planner 在 Kubernetes 环境下通过 MDC（Metadata Context）过滤 Worker 时，错误地使用 DGD（Dynamic Graph Definition）服务键作为组件名称进行匹配的问题。现在会优先使用后端默认的 component 名称或用户通过 `--endpoint` 参数自定义的名称，从而确保 MDC 过滤能够正确命中真实的 Worker 实例。新增了丰富的单元测试覆盖各种后端、默认值、用户覆盖以及异常情况。

**🎯 影响范围**：  
- `components/src/dynamo/planner/connectors/kubernetes.py`（Planner 中的 DGD 服务解析逻辑）  
- `components/src/dynamo/planner/monitoring/dgd_services.py`（`Service` 类新增解析 `--endpoint` 的方法）  
- `components/src/dynamo/planner/tests/unit/test_kubernetes_connector.py`（新增 250 行单元测试）  
- 受影响的子系统：Planner → Kubernetes Connector → MDC 过滤、DGD 解析、Worker 启动流程  

**🔍 技术洞察**：

| 维度 | 影响描述 |
|------|----------|
| **架构影响** | 1. **职责划分更清晰**：将 “从容器 args 中解析用户自定义 component 名称” 抽离为 `Service.get_component_name_from_endpoint_arg`，保持了 `KubernetesConnector._resolve_dgd_service` 的单一职责。<br>2. **正确的优先级**：实现了 “用户显式 `--endpoint` > 后端默认 > DGD key” 的匹配顺序，符合运行时（Rust）向 MDC 写入的真实来源，避免了之前跨层不匹配的隐蔽错误。<br>3. **兼容多后端**：通过 `build_worker_info_from_defaults` 统一获取后端默认 component，保持对现有 vLLM、SGLang、TRT‑LLM 的兼容。 |
| **性能影响** | 仅在 Planner 启动或重新调度时解析一次服务定义，新增的 `parse_endpoint` 与 `break_arguments` 调用开销极低（O(N) 参数遍历），对整体调度延迟可忽略不计。 |
| **安全考虑** | - **输入验证**：`Service.get_component_name_from_endpoint_arg` 对 `--endpoint` 参数进行结构化解析，捕获 `ValueError` 并安全回退，不会因恶意或异常的容器 args 导致异常抛出或信息泄露。<br>- **潜在风险**：若后端容器自行在 args 中拼接不受信任的字符串，解析后得到的 component 可能被用于 MDC 过滤，理论上可能导致误过滤或信息泄漏（但过滤本身是内部匹配，风险极低）。 |
| **可维护性** | - 新增的 Helper 方法和单元测试显著提升了代码可读性和防回归能力。<br>- `expected_component` 的命名与注释清晰表达了业务含义，易于后续维护者理解。 |

**⚠️ 潜在风险**：

1. **向后兼容性**：旧版 DGD（仅依赖服务键）在极端情况下若未提供 `extraPodSpec` / `args`，仍会回退到后端默认，这与之前的行为一致，风险低。但如果有用户在容器 args 中使用了非标准的 `--endpoint` 位置（比如在脚本内部自行拼接），解析可能失效，导致过滤不匹配。  
2. **解析函数 `parse_endpoint`** 假设字符串形如 `<ns>.<comp>.<ep>` 或以 `dyn://` 前缀；若未来后端改变协议或新增前缀，需同步更新此函数，否则会误回退到默认。  
3. **测试覆盖** 已经非常全面，但仍可能缺少对 **多 `--endpoint`** 出现或 **其他参数同名**（如 `--endpoint` 在 `args` 中出现多次）的场景。  

**💡 关注建议**：

- **文档同步**：在 Dynamo 官方文档或 `README` 中明确说明 “Planner 过滤使用的 component 名称来源”，包括用户自定义 `--endpoint` 的优先级，以免使用者产生误解。  
- **监控报警**：可在 Planner 中加入一条 DEBUG/INFO 日志，记录实际使用的 `expected_component` 与 DGD key 的对应关系，帮助运维快速定位过滤失效的根因。  
- **防御性编程**：在 `Service.get_component_name_from_endpoint_arg` 中加入对重复 `--endpoint` 参数的检测（取第一次或报错），防止潜在的配置错误。  
- **未来扩展**：若计划支持更多后端或自定义调度器，建议将 “component name 解析” 抽象为插件接口，统一由后端实现提供，Planner 只调用统一的 `get_component_name()` 方法。  

---  

*整体来看，此次修复消除了因 DGD 键大小写不匹配导致的 MDC 过滤失效问题，提升了 Planner 对不同后端以及用户自定义 endpoint 的适配能力，风险可控，建议尽快在主分支合并并发布对应的 Patch 版本。*

### feat: address MR comments from TRTLLM/GMS integration (#8255)
**SHA**: `dfd3aa9` | 🔗 [查看提交](https://github.com/ai-dynamo/dynamo/commit/dfd3aa99e4a5851ccd4b1abd94d2d7a76225a7eb)

**🎯 变更类型**：功能增强 / 重构  
**⚡ 重要程度**：🔴 高  
**📋 变更摘要**：  
- 移除了 `_collective_rpc` 中对 `wakeup` 方法的异常捕获与回退到 `wake_up` 的逻辑，改为直接调用 RPC，假设 TRT‑LLM 接口统一使用 `wakeup`。  
- 测试文件 `test_trtllm_sleep_wake_handlers.py` 中的 `quiesce` Mock 由单纯返回 `True` 改为在调用时同步更新 `is_quiesced` 状态，使其更贴近真实实现，并去掉了手动设置 `is_quiesced` 的步骤。

**🎯 影响范围**：  
- `components/src/dynamo/trtllm/request_handlers/handler_base.py`（所有 TRT‑LLM 相关的请求处理器）  
- `components/src/dynamo/trtllm/tests/test_trtllm_sleep_wake_handlers.py`（单元测试套件）

---

**🔍 技术洞察**  

- **架构影响**  
  - `handler_base._collective_rpc` 现在不再承担兼容旧版 TRT‑LLM（使用 `wake_up`）的职责，简化了代码路径。  
  - 这使得请求处理层对外部依赖的耦合度提升，若未来出现 API 变更，需要在更高层（如适配层）统一处理，而不是在每个 handler 内散布兼容逻辑。  

- **性能影响**  
  - 移除 `try/except` 块以及额外的回退 RPC 调用，微幅降低了函数调用开销与异常捕获成本（对吞吐量影响极小）。  
  - 测试中对 `quiesce` 的 side‑effect 实现，使得 `AsyncMock` 的行为更接近真实异步调用，提升了测试的真实性，间接帮助发现潜在性能瓶颈。  

- **安全考虑**  
  - 代码未引入新的 I/O、文件或网络操作，安全风险基本保持不变。  
  - 唯一需要关注的是 **兼容性回退**：若在生产环境中误调用了不存在的 `wakeup` 方法，将抛出未捕获异常，可能导致服务进程崩溃，需通过监控或版本检查防止此类情况。  

---

**⚠️ 潜在风险**  

1. **向后兼容性断裂**  
   - 旧版 TRT‑LLM（仍使用 `wake_up`）在部署时会因 `rpc("wakeup", …)` 抛出异常而导致请求处理器失效。  
2. **异常未捕获导致服务不稳定**  
   - 现在任何 `rpc` 调用的异常都会直接向上抛出，若上层未做好异常隔离，可能导致整个 Dynamo 服务进程重启。  
3. **测试行为与真实实现不完全一致**  
   - 虽然 `quiesce` Mock 现在会同步更新 `is_quiesced`，但仍缺少对 `resume` 之后状态恢复的完整验证，可能隐藏状态管理的细微 bug。  

---

**💡 关注建议**  

1. **增加版本检测或后备路径**  
   - 在调用 `_collective_rpc` 前读取 TRT‑LLM 版本信息（或通过一次探测调用），在检测到不支持 `wakeup` 时自动回退到 `wake_up`，或在配置文件中显式声明兼容模式。  

2. **在上层加入异常容错**  
   - 为 `handler_base._collective_rpc` 的调用方加上 `try/except`，记录异常并保持服务可用（例如进入安全降级模式），避免单点崩溃。  

3. **扩充集成测试**  
   - 引入针对不同 TRT‑LLM 版本的集成测试，用真实的 `wakeup`/`wake_up` 实现验证兼容性。  
   - 在单元测试中加入对 `resume` 后 `is_quiesced` 状态回退的断言，确保状态机完整性。  

4. **监控与报警**  
   - 在生产环境中监控 `rpc` 调用的错误率，一旦出现 “method not found” 类错误立即触发警报，以便快速定位因版本不匹配导致的故障。

### fix(operator): run GPU discovery when any hardware field is missing (#8267)
**SHA**: `d07288c` | 🔗 [查看提交](https://github.com/ai-dynamo/dynamo/commit/d07288c54251c9437e3ac493fc072c5dc46dbf07)

**🎯 变更类型**：功能增强 / Bug 修复 / 架构变更  
**⚡ 重要程度**：🔴 高  

**📋 变更摘要**  
- 将 `HardwareSpec` 的四个字段（`gpuSku`、`vramMb`、`numGpusPerNode`、`totalGpus`）全部标记为 **可自动发现**，仅当全部四项都显式提供时才跳过发现。  
- 引入 **GPU discovery 的 SKU 过滤**（`DiscoverGPUsFromDCGMFiltered`）以及 **按 SKU 缓存**，解决混合 GPU 集群中只发现单一 SKU 的问题。  
- `totalGpus` 现在默认由发现计算（受 32 的临时上限），并在 CRD 文档、代码注释和 Helm Chart 中同步更新。  
- 校验逻辑从 “只要任意硬件字段缺失就报错” 改为 “只要并非全部字段都有值且发现不可用则报错”。  
- 大量文档、CRD、测试同步更新，确保用户在未开启 GPU discovery 时能够得到明确的错误提示。  

**🎯 影响范围**  
- `deploy/operator/api/v1beta1/dynamographdeploymentrequest_types.go`（CRD Go 类型）  
- `deploy/operator/internal/controller/*`（DGDR 控制器）  
- `deploy/operator/internal/gpu/discovery.go`（GPU 发现实现）  
- Helm Chart CRDs 与 `docs/kubernetes/api-reference.md`（OpenAPI 文档）  
- 测试套件（`*_test.go`）以及 Python SDK 端的 `dgdr_v1beta1_types.py`  

**🔍 技术洞察**  

| 维度 | 影响描述 |
|------|----------|
| **架构影响** | 1. **发现层解耦**：原来 `DiscoverGPUsFromDCGM` 只能返回单一全局结果；现在引入 `DiscoverGPUsFromDCGMFiltered(sku)`，通过 SKU 过滤实现 **按目标 GPU 类型聚合**，更好地支持 **混合 GPU 集群**。<br>2. **缓存粒度提升**：缓存键由 `GPUSKUType` 组成，能够同时缓存不同 SKU 的发现结果，避免跨 SKU 的错误复用。<br>3. **控制器校验路径简化**：只在 **全部四字段均已提供** 时完全跳过发现，其他任意缺失均会触发 discovery，从而确保 `totalGpus` 始终基于真实集群状态。 |
| **性能影响** | - **发现开销**：在未提供完整硬件信息的 DGDR 创建场景下，仍会进行一次 DCGM 抓取（最多 60 s 缓存），但这本就是必要的操作。<br>- **缓存命中率提升**：同一 SKU 的多次 DGDR 创建可复用缓存，降低对 DCGM Exporter 的并发抓取。<br>- **额外过滤成本**：在混合集群中会遍历所有抓取到的节点并过滤 SKU，成本为 O(N)（N 为节点数），在常见的 10–100 节点规模下可忽略。 |
| **安全考虑** | - **错误信息泄露**：`GetGPUDiscoveryFailureReason` 会在日志中输出底层错误（如网络不可达、权限不足），但仍在 `Info` 级别，符合现有日志策略；未对外暴露。<br>- **权限检查**：发现路径仍依赖 `APIReader` 对节点/Pod 的读取权限，若 Operator 运行在命名空间受限模式且缺少权限，仍会返回明确错误，不会提升权限。<br>- **输入校验**：新增字段 `totalGpus` 仍受 `+optional`，未做数值上限检查（仅在文档里说明 32 上限），业务层仍需在后续版本强化。 |

  - **共享常量**：`DEFAULT_TCP_MAX_MESSAGE_SIZE` 与 `TCP_MAX_MESSAGE_SIZE` 的 `OnceLock` 实例化，使得启动时即确定大小，后续无锁读取，符合“只读全局配置”模式。  
  - **零拷贝解码器增强**：引入 `shrink_threshold` 与 `SHRINK_MESSAGE_SIZE`，实现了在空闲时自动回收过大的 `BytesMut`，降低长期连接的内存占用峰值。  

- **性能影响**  
  - **正向**：长时间运行的 TCP 服务在处理一次性大消息后能够将缓冲区收缩，显著减少常驻内存（尤其在高并发、短消息场景下），有助于整体系统的 GC 与内存碎片控制。  
  - **潜在代价**：收缩后再次遇到大消息会触发重新分配，导致一次额外的内存分配与复制（`BytesMut::with_capacity`），在消息大小频繁波动的工作负载下可能产生轻微的延迟抖动。整体影响预计在 **≤ 5 %** 的极端情况下。  
  - **同步成本**：`OnceLock` 在首次读取后变为只读，几乎零成本；环境变量解析仅在首次访问时执行，后续调用为普通读取。  

- **安全考虑**  
  - **输入校验**：`resolve_shrink_message_size` 对阈值进行上下界限制（`[INITIAL_BUFFER_SIZE, max_message_size]`），并在越界时记录 `tracing::warn!`，防止恶意或错误配置导致缓冲区过小/过大，引发 DoS 或内存泄漏。  
  - **日志可审计**：当阈值被裁剪时会产生可审计的警告日志，有助于运维快速定位配置错误。  
  - **无新增攻击面**：仅涉及只读配置与内部缓冲区管理，不改变网络协议或权限模型。  

**⚠️ 潜在风险**  

| 风险点 | 描述 | 严重度 |
|--------|------|--------|
| **缓冲区频繁收缩/扩张** | 在消息大小剧烈波动的场景下，`ZeroCopyTcpDecoder` 可能频繁触发 `BytesMut::with_capacity`，导致额外的内存分配和短暂的 GC 压力。 | 中 |
| **环境变量解析失效** | 如果 `DYN_TCP_SHRINK_MESSAGE_SIZE` 设置了非法非数字字符串，会回退到默认值并记录警告，可能导致预期的内存回收行为未生效。 | 低 |
| **文档同步** | 新增的 env 变量若未在运维手册或 CI 环境变量白名单中声明，可能导致部署时漏配。 | 低 |
| **兼容性** | 旧的 `DYN_TCP_MAX_MESSAGE_SIZE` 仍然有效，行为未变；但已有代码直接使用硬编码常量（如测试）需确保已改为 `with_capacity` 或 `get_tcp_max_message_size`。 | 低 |

**💡 关注建议**  

1. **监控与告警**  
   - 在生产环境开启 `DYN_TCP_SHRINK_MESSAGE_SIZE` 相关的 **Prometheus 警告**（如 `tcp_decoder_buffer_shrink_total`）并监控 `process_resident_memory_bytes`，确认收缩效果。  
   - 关注 `tcp_decoder_reallocations_total`（可自增）指标，若异常升高，说明业务出现频繁大小波动，需要调优阈值或业务侧分批发送。  

2. **容量基准**

3. **配置审计**  
   - 在 CI/CD 流程中加入对 `DYN_TCP_SHRINK_MESSAGE_SIZE` 的 **合法性检查**（数值范围），防止错误配置进入正式环境。  

4. **代码维护**  
   - 将所有仍旧出现的硬编码 `32 * 1024 * 1024` 替换为 `get_tcp_max_message_size()`，避免未来出现配置不一致。

通过以上措施，可充分利用本次功能增强带来的内存回收优势，同时将潜在的性能抖动和配置错误风险降到最低。祝开发顺利 🚀

**⚠️ 潜在风险**  

1. **误报/漏报**  
   - 当前的 “duplicate” 判定仅基于 **同一 worker** 在同一次 `store` 事件内部或跨事件的 **块哈希冲突**。如果不同 worker 正常地存储相同块（在多租户场景中可能出现），会误报为 duplicate。

2. **并发竞争**  
   - `ConcurrentRadixTree`/`ConcurrentRadixTreeCompressed` 在多线程并发插入时，`duplicate_store` 标记在多次竞争回滚后可能被错误地保留（例如一次成功插入后另一次竞争导致 `insert` 返回 `Some`，但未清除 `duplicate_store`）。虽然代码已在重要分支（worker 插入、lookup 插入、edge 匹配）中手动 `duplicate_store = false`，但仍需在高并发压力下进行压力测试验证。

4. **API 漏传递**

### feat(kv-router): add start positions to flat KV events (#8426)
**SHA**: `4b3cd45` | 🔗 [查看提交](https://github.com/ai-dynamo/dynamo/commit/4b3cd459dffb08fd462cab9e4df34ca39f4aa7f9)

**🎯 变更类型**：功能增强  

**⚡ 重要程度**：🔴高  

**📋 变更摘要**  
为 KV 路由器的 *flat* KV 事件引入了 `start_position` 字段，用以显式指示批次中第一个块的绝对位置。代码在 `KvCacheStoreData` 中新增该可选字段，并在 `PositionalIndexer`、事件转储、各语言绑定及相关测试中加入对应的处理逻辑，使得在没有 `parent_hash` 时仍能通过 `start_position` 正确恢复序列位置。

**🎯 影响范围**  
- `kv-router` 核心模块：`positional.rs、radix_tree.rs、radix_tree_compressed.rs、concurrent_radix_tree.rs、indexer/README.md、tests.rs`  
- 事件协议定义：`protocols.rs`（序列化/反序列化）  
- 语言绑定：C (`bindings/c/src/lib.rs`)、Python (`bindings/python/rust/llm/kv.rs`)  
- 基准/示例代码：`llm/benches/kv_router_bench.rs`、`llm/src/kv_router/indexer/worker_query.rs`、`llm/src/kv_router/publisher/tests.rs`  
- 测试工具与实用函数：`test_utils.rs`、大量单元/集成测试的更新  
- 文档：`indexer/README.md`  
- 模拟后端：`mocker/src/kv_manager/*.rs`  

**🔍 技术洞察**  

- **架构影响**  
  - 在事件模型 (`KvCacheStoreData`) 中加入 `start_position: Option<u32>`，为位置驱动的回放提供直接锚点。  
  - `PositionalIndexer::apply_event` 先检查 `start_position`，若存在则直接使用，否则回退到原有的 `parent_hash → position+1` 逻辑，保持向后兼容。  
  - `dump_events` 在生成 flat `RouterEvent::Stored` 时不再合成 `parent_hash`，而是写入 `start_position`，从而实现“位置唯一、父哈希可选”的事件流。  
  - 绑定层、序列化层均通过 `#[serde(default)]` 自动填充默认 `None`，保证旧版二进制仍能解码。  
  - 文档明确了优先级顺序（`start_position` > `parent_hash` > 0），提升了系统对不完整/非树形序列的容错能力。

- **性能影响**  
  - **CPU**：判断 `start_position` 的分支极轻，整体路径长度与原实现相同。  
  - **内存**：每条 `Stored` 事件增加最多 4 byte（`Option<u32>` 编码），在大规模 KV 场景下影响微乎其微。  
  - **I/O**：事件序列化时多出一个可选字段，序列化字节数轻微增长，且在多数情况下仍为 `None`，对网络/磁盘吞吐基本透明。  
  - **回放速度**：当提供 `start_position` 时，可直接定位起始位置，省去一次 `parent_hash` → `position` 查询，极端情况下略有提升。

- **安全考虑**  
  - 仅为元数据的扩展，不涉及权限、加密或执行路径，未引入新攻击面。  
  - 通过 `serde(default)` 防止因缺失字段导致解析错误，避免因恶意构造的缺失字段触发 panic。  

**⚠️ 潜在风险**  
1. **向后兼容性**：外部系统（如自定义 C/Python 客户端）若对 `KvCacheStoreData` 进行手动结构体匹配，未更新对应定义可能导致编译或运行时错误。  
2. **序列化一致性**：不同语言的绑定若未同步 `start_position` 的默认序列化行为，可能出现跨语言事件不匹配（如 C 端默认不写入字段导致 `None` 与 `Some(0)` 的歧义）。  
3. **逻辑冲突**：在同一事件中同时提供 `parent_hash` 与 `start_position`，当前实现优先使用 `start_position`，但未做显式校验，若两者不一致可能导致数据不一致的隐患。  
4. **测试覆盖**：虽然已添加大量单元测试，但未覆盖 “同时提供错误的 `parent_hash` 与 `start_position`” 场景，潜在的验证缺失。  

**💡 关注建议**  
- **文档与发布说明**：在项目的 Release Notes 中注明 `KvCacheStoreData` 新增 `start_position`，并提供跨语言绑定的更新指南。  
- **迁移指引**：若用户自行构造 `Stored` 事件，请确保在新版本中显式设置 `start_position`（若已知），否则保持 `None` 以兼容旧行为。  
- **输入校验**：在 `PositionalIndexer::apply_event` 中加入断言或日志，检测 `parent_hash` 与 `start_position` 同时出现且不匹配的情况，以防误用。  
- **性能监控**：在高吞吐量的 KV 路由部署中监控事件序列化大小和内存占用，确认新增字段未导致意外的资源膨胀。  
- **持续测试**：补充针对混合使用 `parent_hash` 与 `start_position` 的边界测试，以及跨语言序列化/反序列化的集成测试，确保所有绑定保持行为一致。  

总体而言，此次改动为 KV 路由器的“位置驱动回放”提供了更明确的锚点，提升了系统的健壮性和可预测性，风险可通过文档、兼容层及少量额外校验得到有效控制。

### fix(mocker): cancel replay router scheduler tasks (#8429)
**SHA**: `6f87f93` | 🔗 [查看提交](https://github.com/ai-dynamo/dynamo/commit/6f87f9322786d0ab300f3ebc8aa9bf052cef8766)

**🎯 变更类型**：Bug 修复  
**⚡ 重要程度**：🟡 中  

**📋 变更摘要**  
在 `KvReplayRouter` 中新增 `scheduler_cancel: CancellationToken`，在构造时创建并传递给底层 `LocalScheduler`，随后在 `shutdown` 时调用 `cancel()`，确保路由器关闭时能够正确终止调度器的异步任务，防止因旧的 `CancellationToken` 未被持有而导致任务泄漏或卡死。

**🎯 影响范围**  
- `core` 模块的 `lib/mocker/src/replay/online/router.rs`（路由器实现）  
- 迁移到 `dynamo_kv_router::LocalScheduler` 的调度逻辑  
- 可能波及使用 `KvReplayRouter` 的单元/集成测试及上层服务的生命周期管理  

**💡 关注建议**  

1. **代码完整性**：确认已在文件顶部 `use tokio_util::sync::CancellationToken;`（或相应路径）导入，避免编译错误。  
2. **并发安全**：`scheduler_cancel.clone()` 传入 `LocalScheduler` 后，`shutdown` 里仅调用一次 `cancel()`，但若多次调用 `shutdown`，`CancellationToken` 的多次 `cancel()` 是幂等的，仍安全。可在 `shutdown` 前加 `if self.scheduler_cancel.is_cancelled()` 之类的 guard，以提升可读性。  
3. **资源泄漏检测**：建议在 CI 中加入针对 `KvReplayRouter::shutdown` 的行为测试，验证调度器内部 `JoinHandle` 被正确结束（如 `await` 或 `abort`），防止残留任务占用线程池。  
4. **文档更新**：在结构体及 `shutdown` 注释中说明此字段的职责，让后续维护者了解 “路由器关闭 → 调度器任务统一取消”。  
5. **向后兼容**：其它调用方仍使用原构造函数，不会感知此字段；若有自定义 `KvReplayRouter` 的 `new` 实现，需同步更新以传递 token。  

总体来看，此改动解决了调度器任务在路由器关闭后未被终止的 bug，提升了资源回收的可靠性。只要注意上述细节并补充相应测试，即可安全合并。
