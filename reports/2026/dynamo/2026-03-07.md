### fix: profiling data format in disagg_planner.yaml (#7019)
**SHA**: `fb60cdc` | 🔗 [查看提交](https://github.com/ai-dynamo/dynamo/commit/fb60cdc56fffdb1bf43b5bb221e5afcd8030b053)

**🎯 变更类型**：Bug修复  
**⚡ 重要程度**：🔴 高  
**📋 变更摘要**：  
- 修正了 `disagg_planner.yaml` 中 `decode_raw_data.json` 的数据格式，将原本的二维矩阵（`z_itl`、`z_thpt_per_gpu`）改为统一的平铺一维数组，并在注释中明确说明四个关键字段必须保持相同长度。  
- 该修改同步到了 `sglang`、`trtllm`、`vllm` 三个后端示例，确保示例配置与解析逻辑保持一致。  

**🎯 影响范围**：  
- `examples/backends/sglang/deploy/disagg_planner.yaml`  
- `examples/backends/trtllm/deploy/disagg_planner.yaml`  
- `examples/backends/vllm/deploy/disagg_planner.yaml`  

**🔍 技术洞察**：  
- **架构影响**：此改动仅限于示例配置文件，属于 **配置层** 的兼容性修正。代码层面的解析逻辑已假设所有测量点以平铺 1‑D 数组提供，新的格式消除了原有的二维矩阵假设，从而避免在 `DisaggPlanner` 读取配置时出现维度不匹配的错误。对整体系统架构（调度、模型执行等）无直接影响。  
- **性能影响**：配置文件体积略有增加（展开后的数组长度为 20 条记录），但这只在启动时一次性读取，几乎不产生运行时性能开销。相反，统一的 1‑D 结构可以简化解析路径，略微提升加载速度。  
- **安全考虑**：该变更不涉及代码执行或外部输入的安全检查，仅是数据格式的统一。不存在新的攻击面。唯一需要注意的是，若用户在自定义配置时仍使用旧的二维格式，解析器可能因结构不匹配抛出错误，导致服务启动失败。  

**⚠️ 潜在风险**：  
1. **向后兼容性**：已有的用户若基于旧的二维矩阵格式自行编写或复制 `disagg_planner.yaml`，升级后将触发配置解析错误。  
2. **文档/示例同步**：如果项目文档未同步更新为“所有数组必须为等长的 1‑D”，用户可能仍误用旧格式。  
3. **CI/测试覆盖**：目前仅修改了示例文件，若项目内部有针对配置解析的单元测试，需要确认已覆盖平铺数组的路径，防止回归。  

**💡 关注建议**：  
- **向后兼容**：在解析模块（读取 `decode_raw_data.json`）加入友好的错误提示，明确告知用户需使用平铺 1‑D 数组，并在检测到二维结构时给出迁移建议。  
- **文档更新**：同步更新项目 README、配置文档及示例说明，突出 “四个字段必须为同长度的 1‑D 数组”。可以提供一个自动转换脚本，将旧的二维矩阵快速展平。  
- **测试补全**：新增或扩展配置解析的集成测试，用旧格式和新格式分别验证失败与成功路径，确保未来改动不会破坏兼容性检测。  
- **用户通知**：在发布日志（CHANGELOG）中标注此格式变更，并在关键用户渠道（GitHub Discussions、邮件列表）提示已有部署的用户检查并更新其自定义 `disagg_planner.yaml`。  

通过上述措施，可将此次修复的价值最大化，同时将潜在的兼容性风险降至最低。

### feat(trtllm): add additional metrics for dynamo-trtllm (#6668)
**SHA**: `1008192` | 🔗 [查看提交](https://github.com/ai-dynamo/dynamo/commit/100819299f6b7fa55b8b56e368ddc9f863a9fb48)

**🎯 变更类型**：功能增强  
**⚡ 重要程度**：🔴高  
**📋 变更摘要**：  
- 新增 `components/src/dynamo/trtllm/metrics.py`，实现 **AdditionalMetricsCollector**，在 Prometheus 中暴露 5 类额外的 TRT‑LLM 运行时指标（请求类型、KV‑Cache 迁移、请求中止等）。  
- 在 `handler_base.py` 中对请求进行类型检测、请求中止以及 KV‑Cache 迁移性能记录，调用上述 collector。  
- `llm_worker.py` 在工作进程启动时实例化 `AdditionalMetricsCollector` 并将其注入 `RequestHandlerConfig`，统一使用 `trtllm_` 前缀的指标过滤。  
- 为新的指标在 Rust 端定义常量 (`prometheus_names.rs`) 并在 Python 端对应绑定 (`prometheus_names.py`)。  
- 补充文档、单元测试确保指标注册、采集逻辑的正确性。  

**🎯 影响范围**：  
- **核心模块**：`components/src/dynamo/trtllm/*`（metrics、request_handlers、workers）  
- **运行时**：TensorRT‑LLM 工作进程的 Prometheus 端点  
- **监控层**：运营平台将看到新增的 `trtllm_*` 指标。  
- **文档与 CI**：`docs/backends/trtllm/trtllm‑prometheus.md`、测试套件、Rust 常量定义。  

---

### 🔍 技术洞察

| 维度 | 影响概述 |
|------|----------|
| **架构影响** | - 通过 **AdditionalMetricsCollector** 把 Python 层的业务感知（请求类型、KV‑Cache 迁移）与现有的 C++/Rust 引擎指标统一到同一个 Prometheus 前缀，实现 **跨语言统一监控**。<br>- 新增 `additional_metrics` 字段到 `RequestHandlerConfig`，保持向后兼容（默认 `None`），不影响已有部署路径。<br>- `handler_base.generate_locally` 中的检测逻辑使用原始 `request` 字段，避免对生成路径的深层改动。 |
| **性能影响** | - **CPU 开销**：每个请求会调用几次 `Counter.inc()` 和 `Histogram.observe()`，这些操作在 `prometheus_client` 中是轻量锁‑free（使用 `_lock`），对高 QPS 场景的额外开销可忽略（≈ µs 级）。<br>- **内存占用**：每个 Histogram 按 bucket 维持计数器，已选用合理的 bucket 范围（最大到 5 GB），在极端大 KV‑Cache 迁移情况下仍在数十个 bucket 内，内存增长可控。<br>- **延迟**：仅在 **请求完成** 时记录 KV‑Cache 迁移指标，额外的同步调用对整体生成延迟影响极小。 |
| **安全考虑** | - 仅暴露 **计数/时间/大小** 类型的数值，不涉及用户数据或敏感内容。<br>- 使用 **默认的 Prometheus 注册表**，依赖现有的网络访问控制（e.g., 防火墙、Auth Proxy）进行保护，无新安全面。 |
| **可维护性** | - 代码分离在 `metrics.py`，便于以后扩展或删减单独指标。<br>- 使用 `TYPE_CHECKING` guard 防止在不同 TensorRT‑LLM 版本缺失 `MetricsCollector` 时导致导入错误。<br>- 单元测试覆盖指标创建、计数、Histogram 记录以及 `handler_base` 中的检测逻辑，降低回归风险。 |
| **兼容性** | - 通过 `metric_prefix_filters=["trtllm_"]` 过滤，确保 **仅** 需要的指标被暴露，老版本的监控系统若只关注前缀仍可兼容。<br>- 当 `--publish-events-and-metrics` 未开启，`additional_metrics` 为 `None`，保持完全向后兼容。 |

### ⚠️ 潜在风险

| 风险点 | 说明 | 缓解措施 |
|--------|------|----------|
| **指标注册冲突** | 若用户自行在其他代码里使用相同名称的 Counter/Histogram，可能导致 **DuplicateMetric** 异常。 | - 所有名称均通过 `prometheus_names.trtllm_additional` 统一定义。<br>- 在 CI 中已加入 **test_no_duplicate_metrics**，确保不出现已被前端/运行时覆盖的重复指标。 |
| **高 QPS 场景下锁竞争** | `prometheus_client` 的内部锁在极端并发下可能稍微争用。 | - 采用 Counter/Histogram 的 **no‑label** 高并发路径已做优化；如果出现瓶颈，可考虑 **multiprocess** 模式或将采集转移到 C++/Rust 端。 |
| **KV‑Cache Transfer 统计误差** | 仅在 **prefill** 阶段记录成功计数，若用户自行在 decode 端触发迁移，计数可能不完整。 | - 文档已明确计数仅在 **PREFILL** 模式下记录；如有需求可在未来统一计数点。 |
| **依赖 Prometheus 客户端版本** | 使用 `Histogram` 的自定义 bucket，需要保证 `prometheus_client` 版本兼容。 | - 项目已在 `pyproject.toml` 中锁定适配的版本，CI 会在升级时捕获不兼容错误。 |
| **缺失 `tensorrt_llm.metrics.MetricsCollector`** | 某些发行版或自定义构建可能不提供该类，导致 `ImportError`。 | - 已使用 `TYPE_CHECKING` 包装导入，仅在类型检查阶段出现；运行时不强制导入，避免启动失败。 |

### 💡 关注建议

1. **监控平台**  
   - 将新增的 `trtllm_request_type_*`、`trtllm_num_aborted_requests_total`、`trtllm_kv_transfer_*` 融入现有仪表盘（如 Grafana），并设置 **告警阈值**（如异常的 Abort 率、KV‑Transfer 延迟突增）。  

2. **容量规划**

3. **性能回归检查**  
   - 在高负载 benchmark（≥ 10k RPS）中加入 **Prometheus 客户端耗时** 监控，确保额外指标采集不超过 1 ms/req。  

4. **文档同步**  
   - 确认 `docs/backends/trtllm/trtllm-prometheus.md` 中的指标列表与代码常量保持一致；若后续添加或删减指标，务必同步更新。

   - 保持 `test_trtllm_additional_metrics.py` 的覆盖率在 100%，并在 CI 中开启 **race detection**（如 `pytest -n auto`）以捕获并发下的潜在锁争用。  

6. **未来演进**

   - 若计划在 **多实例**（分布式）部署中聚合这些指标，建议在 `AdditionalMetricsCollector` 中加入 **worker_id** 标签，以便后端进行去重或聚合。

### fix: ETCD cluster unhealthy when adding new member during E2E test & fix vLLM disagg launch config changed (#6976)
**SHA**: `0f01e72` | 🔗 [查看提交](https://github.com/ai-dynamo/dynamo/commit/0f01e724065db5fe6067da98d9791b8791521692)

**🎯 变更类型**：功能增强 / Bug修复  
**⚡ 重要程度**：🔴高  
**📋 变更摘要**：  
1. 为 `DynamoWorkerProcess` 引入 `WorkerMode` 枚举，统一管理 *aggregated*、*prefill*、*decode* 三种 worker 模式，取代原先仅靠 `is_prefill` 布尔的实现。  
2. 对 ETCD HA 测试中的成员替换逻辑加入重试机制，解决删除成员后集群短暂不健康导致新成员添加失败的问题。  

**🎯 影响范围**：  
- `tests/fault_tolerance/etcd_ha/test_vllm.py`（新增枚举、改造 `DynamoWorkerProcess` 调用）  
- `tests/fault_tolerance/etcd_ha/utils.py`（ETCD 成员添加重试）  
- 可能波及项目中其它直接实例化 `DynamoWorkerProcess` 的代码（若有）  

**🔍 技术洞察**  

- **架构影响**  
  - **模块化**：通过 `WorkerMode` 将工作模式抽象为枚举，消除了布尔标识的二义性，使得后续可以轻松扩展更多模式（如 *embedding*、*rerank* 等），提升代码可读性与可维护性。  
  - **配置统一**：不同模式对应的命令行参数、健康检查 URL、KV 传输配置在同一位置集中处理，降低了散落在多个分支的硬编码风险。  
  - **向后兼容**：`__init__` 参数签名从 `is_prefill: bool` 改为 `mode: WorkerMode = WorkerMode.AGGREGATED`，若项目其余位置仍使用旧签名会导致运行时错误，需要同步更新。  

- **性能影响**  
  - **运行时开销**：枚举检查与少量列表拼接的性能成本几乎可忽略不计。  
  - **ETCD 重试**：在 `utils.py` 中新增的最多 20 次、每次 0.5 s 的重试，最坏情况会增加约 10 s 的启动时延，但换来更高的集群稳定性，整体吞吐与延迟影响极小。  

- **安全考虑**  
  - 未引入新的外部依赖或权限提升，仅在重试期间重复执行 `etcdctl member add`，其安全模型与原实现相同。  
  - 通过 `json.dumps` 生成 KV 配置，确保不出现注入风险。  

**⚠️ 潜在风险**  

1. **签名破坏**：非测试代码如果仍以 `DynamoWorkerProcess(request, etcd_endpoints, is_prefill=True)` 方式调用，将触发 `TypeError`。  
2. **默认模式误用**：未显式指定 `mode` 时默认为 `AGGREGATED`，若原有调用默认使用 *prefill*，行为会悄然改变。  
3. **重试阈值硬编码**：`max_attempts = 20`、`sleep 0.5s` 固定在代码中，极端网络/机器慢速环境可能仍超时，需要可配置化。  
4. **日志噪声**：重试期间会产生多条 `warning` 日志，可能掩盖其他重要信息。  

**💡 关注建议**  

- **代码迁移**：在项目中搜索所有 `DynamoWorkerProcess(..., is_prefill=…)` 的调用，统一改为 `mode=WorkerMode.PREFILL` 或 `mode=WorkerMode.DECODE`。若有需要保留向后兼容，可在 `DynamoWorkerProcess` 中加入临时兼容层（接受 `is_prefill` 并映射）。

| **可维护性** | - 大量 `#[serde(default)]`、`Option` 的使用提升了向前兼容性，未来若继续添加字段，只需加 `Option` 即可。<br>- 代码分离明确：映射函数、响应构造函数、流式转换器各自职责单一。<br>- 测试覆盖率显著提升（包括缺失字段、完整字段、`include` 过滤、`truncation` 回显等），降低回归风险。 |
| **兼容性** | - **向后兼容**：已保留旧版 `id: String`、`status: OutputStatus` 的序列化形式（因为 `Option` 在有值时会序列化为同样结构），旧客户端仍能解析。<br>- **向前兼容**：新增可选字段在旧版客户端中被忽略，不会导致反序列化错误。 |

1. **后端功能未实现导致错误**  
   - 现在前端可以把 `reasoning`, `text.format`, `service_tier`, `include`, `truncation` 直接传给后端。如果后端仍返回 “不支持” 错误，前端会把错误直接暴露给用户，可能导致意外的 4xx/5xx。  
2. **错误的 `include` 过滤**

### fix: use checked arithmetic in TwoPartCodec to prevent integer overflow (#6959)
**SHA**: `64ae47f` | 🔗 [查看提交](https://github.com/ai-dynamo/dynamo/commit/64ae47fdc91b4e283cf1fd6a053f40d85942794e)

**🎯 变更类型**：Bug 修复  
**⚡ 重要程度**：🟡 中  
**📋 变更摘要**：在 `TwoPartCodec` 的 `Encoder` 与 `Decoder` 中，原先直接使用 `24 + header_len + body_len` 计算总长度，可能在 `header_len`、`body_len` 极大时产生 usize 溢出。现在改为使用 `checked_add` 链式检查，若溢出则返回 `TwoPartCodecError::MessageTooLarge`，并在后续继续对 `max_message_size` 进行限制。  

**🎯 影响范围**：

**💡 关注建议**  
1. **单元/集成测试**：新增覆盖极端大报文（如 header/body 接近 `usize::MAX`）的测试，确保能够捕获 `MessageTooLarge` 错误且不会触发 panic。  
2. **错误传播**：调用方在 `encode`/`decode` 时需要处理 `TwoPartCodecError::MessageTooLarge`，检查是否已有相应的错误映射或日志。  
3. **兼容性**：若有旧版节点仍使用溢出路径，可能会在收到异常报文时直接 panic。建议在发布说明中说明此变更提升了安全性，且行为从 “进程崩溃” 变为 “返回错误”。  
4. **性能影响**：`checked_add` 的开销极小，几乎可以忽略；但在高吞吐量场景下可通过基准测试确认无显著回退。  

总体来说，此次改动通过安全算术防止了潜在的整数溢出，提升了网络层的鲁棒性，只需确认错误处理路径已覆盖即可。
