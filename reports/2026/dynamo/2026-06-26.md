### fix(operator): default DGD admission on updates (#10987)
**SHA**: `91b6259` | 🔗 [查看提交](https://github.com/ai-dynamo/dynamo/commit/91b6259ba1e3abf99f7983a4c729ed414d0c3e4a)

**🎯 变更类型**：Bug修复 / 功能增强  
**⚡ 重要程度**：🔴高  
**📋 变更摘要**：  
- 将 Operator 的 Admission Webhook 配置从仅处理 `CREATE` 扩展为同时接受 `UPDATE`，并在代码层面明确在 `UPDATE` 场景下不进行默认填充或变异，保持资源原样。  
- 补充对应的单元测试，验证 `UPDATE` 请求被允许且不产生 Patch。  
- 该改动修复了在资源更新时被错误拦截的行为，提升了运维可用性。

**🎯 影响范围**：  
- `deploy/helm/charts/platform/components/operator/templates/webhook-configuration.yaml`（Webhook 注册）  
- `deploy/operator/internal/webhook/defaulting/*`（DCD、DGDR 默认化处理）  
- `deploy/operator/internal/webhook/mutation/pod_checkpoint_restore_handler.go`（Pod 检查点恢复注入）  
- 相关单元测试文件  

**🔍 技术洞察**  
- **架构影响**：  
  - Admission Webhook 仍然在 Operator 中注册，新增 `UPDATE` 支持后，整体调用链保持不变，仅在 `UPDATE` 时直接 `Allowed` 并返回空 Patch。  
  - 通过 Helm Chart 同步更新 `operations` 列表，确保 K8s API Server 正确路由 `UPDATE` 请求到对应的 Webhook。  
- **性能影响**：  
  - 处理 `UPDATE` 时比 `CREATE` 少一次默认填充/变异步骤，CPU/内存开销微乎其微。  
  - 额外的 `UPDATE` 请求路径只做一次 “allow” 判断，整体延迟基本不增。  
- **安全考虑**：  
  - 只在 `UPDATE` 时 **不** 执行任何默认化或变异，避免了因自动写入默认值而导致的意外覆盖，安全风险降低。  
  - 仍然依赖其他 Validation Webhook（若存在）对字段合法性进行检查，未削弱安全防线。  

**⚠️ 潜在风险**  
1. **业务约束缺失**：若某些资源在更新时本应触发额外业务校验（如名称不可变），此改动可能让非法更新直接通过，需要确认已有的 Validation Webhook 能覆盖这些规则。  
2. **默认值丢失**：在 `UPDATE` 时不再补全缺失的字段，可能导致用户在更新对象时意外遗漏必需字段而产生不完整的 CR；但这正是设计意图（避免覆盖用户意图），需在文档中说明。  
3. **兼容性**：旧版客户端若仍假设 `UPDATE` 会被拒绝，可能出现行为差异；但影响范围极小，主要在 Operator 本身。  

**💡 关注建议**  
- **文档更新**：在 Operator 使用手册中明确说明：`UPDATE` 请求将被直接允许且不进行默认化/变异，用户自行负责字段完整性。  
- **验证检查**：确保已有的 Validation Webhook 对 `UPDATE` 场景执行必要的业务校验（如名称不可变、字段约束等），防止误更新。  
- **监控告警**：在 Admission Webhook 旁边添加 metric（如 `admission_update_requests_total`），观察是否有异常增多的 UPDATE 请求，及时发现潜在误操作。  
- **回滚预案**：如果后续发现业务逻辑需要在 UPDATE 时仍进行特定默认化，可通过重新打开相应的逻辑或增加细粒度的开关（configmap）进行控制。  

---  
此改动提升了 Operator 对资源生命周期的兼容性，避免了因误拦截 `UPDATE` 而导致的部署卡死或运维困扰，同时保持了安全边界。只要配套的 Validation Webhook 仍然做必要校验，即可安全上线。

### fix: Bug fix DGH-1013 (#10843)
**SHA**: `c0bc4a0` | 🔗 [查看提交](https://github.com/ai-dynamo/dynamo/commit/c0bc4a0f1fd2f915e9c5a7370a8127c65789c707)

**🎯 变更类型**：Bug 修复  
**⚡ 重要程度**：🔴高  
**📋 变更摘要**：修复了 `dynamo.profiler.__main__` 中 `--config` 参数解析的边界错误。新增对 **inline JSON**（尤其是超长 JSON）直接解析的路径，避免因路径长度超过系统限制而抛出 `OSError(ENAMETOOLONG)`；并在解析前先尝试 JSON 解析，以免误把 JSON 当成文件路径检查。同步补充了覆盖该逻辑的单元测试。  

**🎯 影响范围**：  
- `components/src/dynamo/profiler/__main__.py`（配置解析入口）  
- 相关单元测试 `components/src/dynamo/profiler/tests/unit/test_main_parse_config.py`  

**🔍 技术洞察**  
- **架构影响**：  
  - 仅作用于 **profiler CLI** 子系统，未触及核心 Rust 运行时或其他服务结构。  
  - 引入 `errno` 模块与异常捕获逻辑，保持与现有 `Path.is_file()` 调用兼容，属于局部功能增强，不改变整体模块依赖图。  
- **性能影响**：  
  - 对 **inline JSON** 的先行解析可绕过一次文件系统 `stat` 调用，略微降低 I/O 开销，尤其在大量或超长 JSON 场景下更显优势。  
  - 额外的 `json.loads` 尝试在成功解析时即返回，失败时再进行文件路径检查，整体耗时对常规（短）JSON 影响可忽略。  
- **安全考虑**：  
  - 新增的异常过滤仅针对 `errno.ENAMETOOLONG`，防止因异常泄漏导致程序崩溃，提升了 **容错性**。  
  - 未引入外部输入的代码执行路径，仍然依赖 `json.loads` 的安全解析，未出现新的注入风险。  

**⚠️ 潜在风险**  
1. **异常传播**：除了 `ENAMETOOLONG` 之外，若 `Path.is_file()` 抛出其他未预料的 `OSError`（如权限错误），会被重新抛出，导致调用方得到未捕获异常。  
2. **行为一致性**：对已经存在的同名目录或特殊文件（如符号链接）仍然依赖原有 `is_file()` 判定，若此前有特定容错逻辑，需确认不存在回归。  
3. **单元测试依赖**：测试中使用 `monkeypatch` 替换 `Path.is_file`，如果未来对 `Path` 进行更复杂的包装，可能需要同步更新测试。  

**💡 关注建议**  
- **对外文档**：在 profiler CLI 使用手册中注明 `--config` 现在会优先解析 inline JSON，特别是“大型 DGDR 规范”场景，可直接使用 `--config '{"model":...}'` 而无需临时文件。  
- **异常信息统一**：考虑在捕获 `OSError` 后统一包装为 `ValueError`（如当前测试所期望），以提供更友好的错误提示。  
- **回归测试**：在 CI 中保留当前新增的单元测试，同时加入对 **权限不足**、**路径是目录** 等边界情况的测试，以防未来改动破坏异常处理。  
- **监控与日志**：在捕获 `ENAMETOOLONG` 以后，可在日志中记录该异常类型（DEBUG 级别），帮助运维快速定位因输入过长导致的路径探测失效。  

---  
此修复提升了 profiler 工具在大规模部署场景下的可靠性和用户体验，风险可控且建议的进一步完善措施能够进一步巩固稳健性。

### feat: route Qwen3-Coder + DeepSeek-V4 tool calls through dynamo-parsers-v2 (DYN_ENABLE_EXPERIMENTAL_PARSERS_V2) (#10853)
**SHA**: `1b37e82` | 🔗 [查看提交](https://github.com/ai-dynamo/dynamo/commit/1b37e82ea778a1982eee5318fa78d70d21027370)

**🎯 变更类型**：功能增强（实验性工具调用解析器 v2）  
**⚡ 重要程度**：🔴高  
**📋 变更摘要**：  
- 新增 `dynamo-parsers-v2` 依赖并通过环境变量 `DYN_ENABLE_EXPERIMENTAL_PARSERS_V2` 开启实验性路由。  
- 对 Qwen3‑Coder、DeepSeek‑V4 两个模型族的 tool‑call **流式** 与 **batch** 解析路径全部改为使用 `dynamo‑parsers‑v2`，绕过原有的 jail + aggregate‑finalize 机制。  
- 新增 `ParsingOptions::experimental_v2_batch_eligible`、相应的 gate 判断以及 `tool_parser_v2` 模块实现完整的流式、批量解析与回填逻辑。  

**🎯 影响范围**：  
- `lib/llm`（核心 LLM 处理、preprocessor、protocols/openai、aggregator）  
- `lib/runtime`（新增环境变量常量）  
- `Cargo.toml` / `Cargo.lock`（全仓库新增 `dynamo-parsers-v2` crate）  
- 绑定层 `bindings/kvbm`、`bindings/python`（同步更新依赖）  
- 测试套件（新增 `tool_parser_v2` 单元测试、端到端实验性解析测试）

### 🔍 技术洞察  

| 维度 | 影响分析 |
|------|----------|
| **架构影响** | - **可选插件化**：通过环境变量在运行时决定是否使用 `dynamo-parsers-v2`，实现了 **可插拔** 的解析层。<br>- **模块拆分**：新增 `tool_parser_v2` 子模块，职责单一（流式解析、批处理），代码结构更清晰。<br>- **依赖层次**：`dynamo-parsers-v2` 依赖 `dynamo-parsers` 与 `openai-harmony`，形成了 **双层解析链**，但仅在实验性开启时激活，避免对默认路径的侵入。 |

| **安全考虑** | - **新增第三方 crate**：`dynamo-parsers-v2` 代码体量约 5k LOC，需审计其对外部输入的处理（尤其 `serde_json`/`regex`），防止 **JSON 注入**、**正则拒绝服务** 等风险。<br>- **错误容错**：在流式 `push`/`finish` 失败时会 **回退** 为原始 chunk（`tracing::warn!`），防止 panic 传播到前端，提升容错性。<br>- **环境变量**：默认关闭，降低误用风险；但运营方如果误开启，可能导致 **截断参数值被直接丢弃**，业务层需要接受此行为。 |
| **可维护性** | - 代码路径分叉（v1 vs v2）导致 **分支复杂度** 上升，未来若再加入新模型族需同步修改两个分支的 gate 条件。<br>- `ParsingOptions` 结构体新增字段，涉及序列化/反序列化，需要保持向后兼容（已提供默认 `false`）。<br>- 单元测试覆盖 `apply_stream`、`parse_complete`、以及全链路 `postprocessor_parsing_stream`，但仍缺少 **错误路径**（如 `create_tool_parser_for_family` 报错） 的显式测试。 |

### ⚠️ 潜在风险  

1. **误开启导致功能回退**  
   - 若 `DYN_ENABLE_EXPERIMENTAL_PARSERS_V2=1` 被意外设置，所有 Qwen3‑Coder / DeepSeek‑V4 的 tool‑call 将走 v2 路径。v2 解析在参数 **截断** 时会 **直接丢弃** 调用，可能导致业务侧收到空的 tool‑call，进而产生业务错误。  

2. **兼容性冲突**  
   - `dynamo-parsers-v2` 依赖的 `openai-harmony` 与项目已有版本保持一致（`Cargo.lock` 已锁定），但若未来主仓库升级 `openai-harmony`，可能出现 **SemVer 不兼容** 导致编译或运行时错误。  

3. **性能回退**  
   - 对不支持的模型族仍会尝试 `create_tool_parser_for_family`（在 `apply_stream` 开头）并在失败后回退为原始流。虽然已记录 `warn!`，但每次创建失败都会进行一次 **额外 hash‑lookup**，在高并发环境下可能产生微小的背压。  

4. **错误处理不充分**  
   - `parse_complete` 中对 `parser.parse_complete` 的错误直接返回 `anyhow::Result`，调用方在 aggregator 中 `match` 时只打印 debug 并 **continue**。若错误频繁（例如模型输出不符合 v2 语法），会导致大量 **工具调用丢失**，而用户难以定位根因。  

5. **测试覆盖不足**  
   - 目前测试只覆盖 **正常完整调用** 与 **截断丢弃** 两种场景，未覆盖：<br>   • `create_tool_parser_for_family` 失败路径<br>   • 多 tool‑call、跨 choice 场景<br>   • 并发流的状态共享冲突（HashMap/HashSet 的并发安全未显式声明，但在单线程 async 环境中安全）。

1. **监控与告警**

   - 当错误率超过阈值（例如 0.5%）触发告警，防止意外开启后导致生产环境大量工具调用失效。

### perf(kv-router): reduce active sequence load tracking locks (#10967)
**SHA**: `73f036c` | 🔗 [查看提交](https://github.com/ai-dynamo/dynamo/commit/73f036c3a998be276ef51da8bba68083aa2f2ff6)

**🎯 变更类型**：性能优化  
**⚡ 重要程度**：🔴高  
**📋 变更摘要**：通过引入 `seqlock` 与 `indexmap`，把原来对每个 worker 的 `RwLock<WorkerLoadSnapshot>` 替换为无锁读的 `SeqLock<WorkerLoadSnapshot>`，并用 `IndexMap` 取代手写的 `Vec+FxHashMap` 结构。与此同时，优化了 worker 注册的路径，避免在读锁持有期间进行写入，进一步降低竞争。基准测试代码使用 `black_box` 防止编译器优化，使测量更可信。

**🎯 影响范围**：
- `lib/kv-router`（核心 KV‑router 负载调度逻辑）  
- 依赖此模块的所有语言绑定（Python、Rust、C++ 等）  
- 相关基准测试 `lib/bench/kv_router`  
- Cargo 依赖：新增 `seqlock`、`indexmap`

**🔍 技术洞察**：

- **架构影响**  
  - **加载表结构**：原来的 `WorkerLoadTable` 采用 `Vec<WorkerLoadSlot>` + `FxHashMap<WorkerWithDpRank, usize>` 手动维护稠密向量和索引映射；改为 `IndexMap<WorkerWithDpRank, WorkerLoadSlot>`，保留插入顺序且提供 O(1) 的键查找与稠密遍历，代码更简洁，错误面更小。  
  - **并发同步**：从每个 worker 用 `RwLock`（读锁需要获取共享锁）改为 `SeqLock`（读操作无锁、写操作仍序列化），大幅降低了热点路径（读取所有 worker 负载进行调度投影）的锁竞争。  
  - **注册流程**：`ensure_worker_registered` 先在只读锁下检查，若未命中才释放读锁并进入写路径 `ensure_worker_registered_after_miss`，避免了“读‑写升级”导致的死锁/饥饿。  

- **性能影响**  
  - **读取路径**：每次 `project_worker_loads` 需要遍历所有 worker 并读取快照。`SeqLock` 让读取仅是一次原子的 `load.read()`（内部使用 volatile + sequence 检查），比 `RwLock::read` 的锁获取和解锁快约 2–5 倍（取决于并发度）。  
  - **写入路径**：写入仍需获取 `SeqLock` 的内部互斥锁，性能与 `RwLock::write` 类似，但写入频率远低于读取（调度投影是热点），整体提升显著。  
  - **数据结构**：`IndexMap` 的紧凑布局提升了缓存局部性，遍历时的预取效果好于分离的 `Vec` + `HashMap`。  
  - **基准测试**：使用 `black_box` 防止编译器把 `project_worker_loads` 整体消除，确保测量真实。  

- **安全考虑**  
  - `seqlock` 在内部使用 `read_volatile` 读取可能在写并发时产生“数据竞争”。虽然 crate 官方声称“逻辑上无竞争”，但在 LLVM/Rust 中这仍属于 **未定义行为（UB）**，因为 volatile 并不保证原子性。此风险在 **单线程写、 多线程读** 场景下基本可接受，但若未来出现写‑写并行（如多线程并发更新同一 worker），则可能触发 UB。  
  - 代码已显式限制写入只有单一写者（`ActiveSequencesMultiWorker` 的调度路径），但仍请在文档中注明此前提，防止误用。  

**⚠️ 潜在风险**：
1. **SeqLock UB**：若出现写‑写并发或读‑写之间的极端竞争，可能导致读取到不完整的快照或触发 UB，导致进程崩溃或错误结果。  
2. **IndexMap 迭代顺序改变**：`IndexMap` 保持插入顺序，原来的 `Vec` 也是顺序的，但在某些极端情况下（如删除后重新插入）顺序可能不同，若外部代码依赖遍历顺序进行统计，可能产生细微差异。  
3. **Cargo 依赖更新**：新增 `seqlock` 与 `indexmap` 可能引入额外的 transitive 依赖冲突或安全审计遗漏，需要在 CI 中保持 `cargo audit`。  
4. **回退兼容性**：旧的 `RwLock` 读取路径在异常（panic）后仍能安全释放锁；`SeqLock` 在同一线程 panic 时仍会正确解锁，但务必确保没有跨线程 panic 导致锁状态不一致。  

**💡 关注建议**：
- **并发测试**：在 CI 中添加压力测试（比如 `loom`）模拟高并发的读‑写场景，验证 `SeqLock` 在极端竞争下不会触发 UB。  
- **安全审计**：对 `seqlock` crate 进行审计，确认其 `read_volatile` 用法在当前平台（Linux x86_64、macOS ARM）不会被优化成真正的竞争。若有顾虑，可考虑使用 `atomic::AtomicU64` 替代自定义序列号实现。  
- **文档说明**：在 `WorkerLoadSlot` 的注释里明确写入只能单线程执行，说明 `SeqLock` 的使用前提以及不推荐在多写者环境使用。  
- **回退路径**：保留一个 feature flag（例如 `use_rwlock`），让用户在遇到平台兼容性问题时可以切回 `RwLock` 实现，避免强依赖 `seqlock`。  
- **基准回归**：在发布前保留原始 `RwLock` 基准对比，确保在不同硬件（低核数、NUMA 环境）下仍有显著收益。

**🎯 影响范围**  
- `kv_router.rs`（核心路由模块）  
- 依赖 `KvScheduler` 调度的所有 LLM 推理入口（包括 `RouterRequest::New`）  
- 错误传播层面：新增 `Cancelled` 错误类型的处理路径  

**🔍 技术洞察**  

- **架构影响**  
  - 引入 **异步取消** 机制，使用 `tokio::select!` 让调度与上下文生命周期解耦。  
  - 通过 `Box::pin` 将调度 Future 包装后可在 `select!` 中引用，保持原有调度逻辑不变，仅在外层添加取消分支，侵入度低，兼容性好。

  - **正面**：对已被取消的请求不再继续耗费 CPU / I/O，降低系统在高并发、频繁取消场景下的资源占用。

  - `biased;` 关键字确保取消分支先于调度完成分支竞争，降低取消延迟。

  - 该改动不涉及网络协议或权限检查，仅是错误类型的细化。  
  - 新增 `ErrorType::Cancelled`，如果上层对错误进行分类显示，需要确保不泄露内部 `context_id` 等实现细节；当前实现仅在日志中记录，向客户端返回的错误信息（`Request {id} was cancelled`）属于可接受的业务层信息。  

**⚠️ 潜在风险**  
1. **状态清理失败**：如果 `self.free(&context_id).await` 返回错误，当前实现只记录 `warn!`，但仍返回 `Cancelled`。在极端情况下，调度器内部可能残留未释放的令牌或占用，导致后续调度不均衡。  
2. **Future 迁移错误**：`Box::pin` 包装后若后续代码有 `.await` 再次移动该 Future，可能触发编译器 “future moved after being polled” 错误；需要确保所有后续使用都保持同一 `Pin<&mut>`。  
3. **误判取消**：如果 `request_context.stopped()` 触发的时机不够精准（比如在网络层面短暂中断后恢复），可能导致本应继续的请求被错误取消。

- **性能评估**：在高并发基准下测量 `Box::pin` 带来的 GC/分配开销，必要时考虑使用 `pin_utils::pin_mut!` 或自定义池化方案来降低堆分配频率。  
- **错误处理**：如果业务方需要对取消错误做特殊重试或上报，可在 `DynamoError::Cancelled` 上层统一映射为幂等的状态码（如 HTTP 409/499），避免因频繁取消导致的流量异常。  
- **回滚检查**：确保 `self.free` 在所有路径（包括调度成功、错误、取消）都是幂等的，并在单元测试中覆盖取消后再次调用 `free` 的情形。  
- **文档更新**：在项目的错误码说明或 API 文档中加入 `Cancelled` 错误的语义解释，提醒调用方在收到此错误时可以安全地放弃或重新发起请求。

- 运行时：Pod spec、Toleration、ResourceClaim、Volume、Portworx、UserNamespaces 等字段的描述、gate 要求与可用性。  

**💡 关注建议**  

1. **兼容性检查**  
   - v0.36 库要求的 API 版本在 Kubernetes 1.30+ 才完全受支持。若仍在 1.28/1.29 集群上运行，未知字段会被忽略，但可能出现 webhook 校验错误。建议先在测试集群验证升级。  
   - `procMount`、`UserNamespacesSupport`、`RecoverVolumeExpansionFailure` 等已不再强制 feature‑gate，字段描述被删减，可直接使用，避免因 “requires feature flag” 警告导致 webhook 拒绝。  

2. **新特性启用**  
   - `DynamicResourceAllocation` 已从 alpha 改为 stable，默认可用但仍需要在 apiserver 打开相应 gate 才能生效。  
   - `DRAWorkloadResourceClaims`、`TaintTolerationComparisonOperators`、`HPAConfigurableTolerance`、`PodGroupResourceClaims` 均为可选特性，若业务依赖请在集群启动参数中开启对应 gate。  
   - 新增 `schedulingGroup`（pod‑group 名称）会被 scheduler 用作分组调度，若不想使用，请保持空；否则需配合对应的 `PodGroup` 资源（本仓库已提供）并确保调度器支持。  

3. **CRD 与 Helm 维护**  
   - 所有 CRD 的 `description` 均已改为多行块 (`|-`) 并补充了 “cannot be set on ephemeral containers” 等限制，升级后旧的 JSON/YAML 客户端仍能解析。  
   - `userAnnotations` 现在作为 `PodCertificateRequest` 的 `unverifiedUserAnnotations` 直接复制，使用前请确认业务不依赖签名方对该字段的过滤逻辑。  

4. **代码迁移**  
   - `informers.NewFilteredSharedInformerFactory` 被废弃，已改为 `informers.NewSharedInformerFactoryWithOptions(..., informers.WithNamespace(...))`；自定义过滤逻辑如果仍需保留，请在外层自行实现。  
   - `ctrl.NewWebhookManagedBy(mgr, &Obj{})` 接口签名更新，所有自定义 webhook 代码需同步修改，否则编译错误。  

5. **CI / Lint**  
   - `golangci.yml` 新增对 controller‑runtime v0.24 已废弃的 `scheme.Builder`、`CustomValidator`、`WithCustomValidator`、`GetEventRecorderFor` 的 suppress，后续可计划迁移至新版 admission/event API（#10795）。  

**📌 小结**：此次升级把 operator 对 Kubernetes 的依赖对齐到最新的 0.36 系列，兼容性提升、废弃字段清理、若干新特性默认升级为 stable。升级前请在非生产环境完整跑一遍 CRD‑apply、Webhook‑conversion、以及需要开启的 feature‑gate，确认业务没有依赖已移除的 alpha 行为。生产集群升级时建议采用滚动重启并保留旧版 operator 作为回滚路径。
