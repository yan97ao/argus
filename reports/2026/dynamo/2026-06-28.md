### feat(bench): add disaggregated KVBM offline replay (#11010)
**SHA**: `7d84764` | 🔗 [查看提交](https://github.com/ai-dynamo/dynamo/commit/7d84764b180c426437bed801a03f2d5c2ccaef95)

**🎯 变更类型**：功能增强、架构变更、性能优化（离线回放基准新增“分离式 KVBM”模拟）  
**⚡ 重要程度**：🔴高  
**📋 变更摘要**：  
- 在离线回放基准工具 `offline_replay_bench` 中加入了 **disaggregated（分离式）服务拓扑** 支持，能够分别模拟 prefill 与 decode 工作池，并引入 KV‑Cache 带宽、容量、离线转移时延等参数。  
- 新增 `--serving-mode disagg`、`--num-prefill-workers`、`--num-decode-workers`、`--kv‑transfer‑bandwidth`、`--kv‑bytes‑per‑token` 等 CLI 选项；在代码层面加入 `ServingModeArg`、`KvTransferTimingModeArg`、`WorkerType`、`OfflineDisaggReplayConfig` 等新枚举与结构体，并在 `MockEngineArgs` 中暴露对应字段。  
- 通过 `#[cfg(feature = "mocker-kvbm-offload")]` 将 KVBM（KV‑Buffer‑Manager）离线 offload 相关逻辑做特性门控，保证不影响默认构建。  

**🎯 影响范围**：  
- `lib/bench/offline_replay_bench.rs`（CLI 参数解析、调度入口）  
- `lib/mocker/src/replay/offline/entrypoints.rs`（离线回放调度实现、单元测试）  
- `lib/mocker/src/common/protocols.rs`（`MockEngineArgs` 扩展）  
- `README.md`（文档示例）  

**🔍 技术洞察**：

| 维度 | 影响描述 |
|------|----------|
| **架构影响** | - 引入 **分离式工作池**（prefill / decode）以及 **WorkerType**，使离线回放能够更贴近实际的推理服务拓扑。<br>- 新增 `OfflineDisaggReplayConfig` 将两套 `MockEngineArgs`（prefill 与 decode）组合在一起，解耦了原本的单一 `MockEngineArgs`，对内部调度层（`simulate_trace_workload_disagg_with_router_mode`）产生了结构性改动。<br>- KVBM offload 通过特性 `mocker-kvbm-offload` 集成，涉及 GPU KV‑Cache、G2/G3/G4 多层存储模型，整体架构更复杂但保持向后兼容。 |
| **性能影响** | - 新增 **KV‑transfer‑bandwidth**、**kv‑bytes‑per‑token**、**kv‑transfer‑timing‑mode** 参数，能够在基准中显式模拟 KV‑Cache 带宽瓶颈与转移时延，提升 benchmark 的真实性。<br>- 分离 prefill / decode 工作池后，可分别配置 worker 数量与资源（如 `num_gpu_blocks`），对吞吐量与 latency 的耦合关系更加细粒度。<br>- 需要注意：在 **offload** 模式下，额外的内存复制与网络带宽模拟会显著拖慢基准运行时间，可能导致误判性能瓶颈。 |
| **安全考虑** | - 代码改动主要是 **仅在测试/benchmark 环境** 使用的 mock 层，未触及生产服务路径，安全风险极低。<br>- 通过 `ensure!` 对 offload 参数进行简单校验（engine 必须是 `vllm`，且必须提供 `kv_bytes_per_token`），避免错误配置导致 panic，但仍建议在特性开启时加入更严格的参数完整性检查。 |
| **可维护性** | - 新增大量 CLI 参数和对应结构体字段，代码路径增多，后续演进时需保持文档同步。<br>- 对 `MockEngineArgs::builder()` 的链式调用进行了扩展，若后续再加入新特性，需要注意 **builder** 的兼容性与默认值。 |
| **兼容性** | - 通过 `#[cfg(feature = "mocker-kvbm-offload")]` 隔离 offload 相关字段，默认编译（不启用该特性）仍保持原有行为，兼容性良好。<br>- 老用户如果仍使用 `--serving-mode aggregated`（默认），不会感知任何变化。 |

**⚠️ 潜在风险**：

1. **参数组合不完整导致 panic**  
   - 现有校验仅检查 `engine_type == vllm` 与 `kv_bytes_per_token` 是否提供，未校验 `num_g2_blocks`、`bandwidth_*` 等参数的相互依赖关系。错误或缺失的组合可能在运行时触发 `unwrap`/`panic!`。  
2. **特性门控遗漏**  
   - 某些代码路径（如 `simulate_trace_workload_disagg_with_router_mode`）在未启用 `mocker-kvbm-offload` 时仍会被编译进去，可能导致未定义的字段访问或编译错误。  
3. **基准结果误导**  
   - 用户如果忘记提供真实的带宽/容量参数，benchmark 会使用默认 `None`（即无带宽限制），从而产生过于乐观的 latency/throughput 报告。  
4. **文档与实际实现不一致**  
   - README 中示例使用 `--kv-transfer-bandwidth 64`（GB/s），但代码中 `kv_transfer_bandwidth` 的单位是 **GB/s**，若用户误以为是 **GB/s** 的整数（如 64）而实际需要 `64.0`，可能导致解析误差。  
5. **测试覆盖不足**  
   - 当前仅新增了一个 `disagg_replay_initializes_kvbm_workers` 单元测试，未覆盖错误参数、不同 `WorkerType` 组合、带宽模型切换等情形。  

**💡 关注建议**：

- **完善参数校验**  
  - 在 `build_engine_args` 中加入 `ensure!` 检查所有带 `#[cfg(feature = "mocker-kvbm-offload")]` 的字段与前置条件的匹配（例如 `num_g2_blocks` 必须非空且大于 0，带宽参数必须提供且为正数）。  
- **强化文档**  
  - 在 README 明确每个新增 flag 的单位、必选性以及与 `mocker-kvbm-offload` 特性的关系。提供一个“最小可运行配置”的示例帮助新手快速上手。  
- **默认值与安全阈值**  
  - 为 `kv_transfer_bandwidth` / `kv_bytes_per_token` 设置保守的默认值（如 10 GB/s、128 KiB），防止用户忘记显式设置时产生不合理的基准结果。  
- **增强测试**  
  - 增加参数错误路径的负面测试（缺少必填 flag、非法数值等），以及不同 `KvTransferTimingMode`（`FullPrompt` 与 `DestinationMissing`）对延迟模型的影响对比。  
- **CI/benchmark CI 更新**  
  - 在 CI 中加入 `--features mocker-kvbm-offload` 的基准跑分，确保新特性在 Linux 环境下能够成功编译并产生合理的报告。  
- **后续演进计划**  
  - 若计划将该离线 offload 模型迁移到真实服务层，建议抽象出 `KvTransferModel` 接口，统一 CPU‑GPU、GPU‑Host、Host‑ObjectStorage 的带宽与延迟模型，减少特性门控的散布。

### fix(mocker): enforce bandwidth and capacity invariants (#11012)
**SHA**: `ec8ffdd` | 🔗 [查看提交](https://github.com/ai-dynamo/dynamo/commit/ec8ffdd0bd5bf8a04203497bb919a44a4d936fca)

**🎯 变更类型**：Bug修复  
**⚡ 重要程度**：🔴高  
**📋 变更摘要**：在 `BandwidthSharingModel` 与 `CapacityReservations` 中加入严格的数值合法性断言，防止带宽换算溢出、时间回滚以及容量释放下溢。新增 `assert_valid_time` 辅助函数并对 transfer‑id 计数器使用 `fetch_update` 检查上限，确保模型在异常输入下直接 panic 而不是产生错误的内部状态。  

**🎯 影响范围**：  
- `lib/mocker/src/kvbm_offload/bandwidth_sharing_model.rs`（带宽共享模型）  
- `lib/mocker/src/kvbm_offload/capacity_reservation.rs`（容量预留）  
- 依赖该模型的模拟/回放组件、以及任何使用 `CapacityReservations` 的业务逻辑  

**🔍 技术洞察**  

- **架构影响**  
  - 仅在模型层面加入运行时不变量检查，未改变公共 API。  
  - 通过 `assert!` 强化了模型的自洽性，使得错误的调用路径能够在早期暴露，提升了整体系统的鲁棒性。  
  - `fetch_update` 取代了单纯的 `fetch_add`，在计数器即将溢出时会触发 panic，防止出现重复 `TransferId`，对后续依赖唯一 ID 的调度器更安全。  

- **性能影响**  
  - 断言在 **debug** 与 **release** 编译模式下均会执行（`assert!` 在 release 中仍会保留），但仅涉及几次浮点/整数比较与一次原子操作，开销极低（< 0.1%），对大多数模拟场景无感知。  
  - 额外的检查（如 `is_finite`、非负校验）避免了潜在的异常路径导致的 CPU 重新计算或错误的内存写入，间接提升了在异常输入下的稳定性。  

- **安全考虑**  
  - 防止 **数值溢出**（带宽换算、TransferId）和 **下溢**（容量释放），消除潜在的整数漏洞，降低因未定义行为导致的 DoS 或越界访问风险。  
  - 断言导致的 panic 会终止当前模拟进程，属于 **fail‑fast** 策略，确保错误不会悄悄传播到持久层或网络边界。  

**⚠️ 潜在风险**  

1. **业务级 panic**：如果上层代码在异常时间顺序或容量释放顺序上有 bug，会因为新增的断言而直接 panic，可能导致整个模拟服务崩溃。需要确认调用方已在适当层次捕获或避免这些异常情形。  
2. **TransferId 上限**：`AtomicU64::fetch_update` 在达到 `u64::MAX` 时会 panic。虽然实际使用中几乎不可能达到，但极端负载（如长时间不重启的服务）可能触发，需要评估是否需要回滚或循环策略。  
3. **断言在 release 中仍会触发**：如果项目原本依赖“宽容”行为（例如让溢出回环），此更改会改变行为并可能破坏兼容性。  

**💡 关注建议**  

- **文档升级**：在 `BandwidthSharingModel::new`、`start_transfer`、`advance_to` 与 `CapacityReservations::release` 的公开文档中明确说明这些函数在非法输入时会 panic。  
- **上层容错**：在使用模型的代码（如 trace replay、模拟调度器）加入前置检查或捕获 `panic`（例如 `std::panic::catch_unwind`）以提供更友好的错误报告。  
- **监控/告警**：在生产环境中加入 panic 统计或监控，以快速定位因时间倒退或容量误用导致的崩溃。  
- **极端负载评估**：如果业务可能长期运行且累计产生大量 `TransferId`，考虑在 `BandwidthSharingModel` 实现上提供 “wrap‑around” 或 “reset” 机制，而不是直接 panic。  
- **回归测试**：确保 CI 包含本次新增的单元测试（已在代码中），并在已有的负载测试场景下验证不会出现额外的性能回退。

*总体而言，此次提交通过明确的运行时不变量提升了核心模拟模型的可靠性与安全性，属于高价值的 bug 修复。只需注意对上层调用者的兼容性和异常处理策略即可平稳过渡。*

### fix(sglang): precompile bytecode in runtime image to cut CI test time (#10978)
**SHA**: `f8a8591` | 🔗 [查看提交](https://github.com/ai-dynamo/dynamo/commit/f8a8591e7a4f7b505866a39487059b6bed6b6828)

**🎯 变更类型**：Bug修复 / 性能优化  
**⚡ 重要程度**：🟡 中  

**📋 变更摘要**  
1. 在 CI 工作流 `pr.yaml` 中删除了 `gpu_test_timeout_minutes: 45`，不影响功能，仅是清理。  
2. 在 `sglang_runtime.Dockerfile` 中新增步骤（除 XPU 设备外），在镜像构建阶段对 site‑packages 以及 `/sgl-workspace/sglang/python` 进行 `python -m compileall`，预编译 `.pyc`。这消除了 CI 测试中每个进程第一次导入 `torch/transformers/sglang` 时的重新编译开销，将原本 8‑10 分钟的慢速显著缩短。  

**🎯 影响范围**  
- Docker 镜像构建过程（`container/templates/sglang_runtime.Dockerfile`）。  
- CI‑GPU 测试执行时间。  
- 仅在非 XPU 设备的 runtime 镜像中生效。  

**💡 关注建议**  
- 确认编译后镜像体积仍在可接受范围，若显著增大可考虑在 CI 中使用分层缓存。  
- 注意 `compileall` 的并行度 `-j0`，在资源受限的构建机器上可能导致 OOM，必要时添加 `--max-workers` 限制。  
- 由于预编译在 root 权限下完成，切换到 `dynamo` 用户后不要再尝试写入 site‑packages，防止权限错误。  
- 对于 XPU 设备，保持原有逻辑；若以后需要同样的优化，请在对应的 Dockerfile 分支添加对应条件。  
- CI 触发后可观察 `sglang` 相关测试的耗时是否如预期下降，若仍有异常编译耗时，检查是否有路径未被包含在 `compileall` 范围内。
