### fix(tcp): replace panic with warn+break on stream read errors (#8254)
**SHA**: `eec0d54` | 🔗 [查看提交](https://github.com/ai-dynamo/dynamo/commit/eec0d54322132770ff974dbffd460abd8b20a967)

**🎯 变更类型**：Bug修复 / 安全修复  
**⚡ 重要程度**：🔴高  
**📋 变更摘要**：  
本次提交通过将 TCP 客户端/服务端在异常情况下的 `panic!` 替换为日志记录（`warn`/`debug`）并妥善结束任务，避免因单个流错误导致整个进程崩溃。新增 `wait_for_connection_tasks` 与 `wait_for_server_shutdown` 辅助函数，使连接监控的错误处理更明确；在读取、协议违例、序列化等场景统一使用 `tracing::warn` 并对上下文进行 `kill/stop`，同时补充大量单元测试验证新行为。  

**🎯 影响范围**：  
- `lib/runtime/src/pipeline/network/tcp/client.rs`（TCP 客户端实现）  
- `lib/runtime/src/pipeline/network/tcp/server.rs`（TCP 服务端实现）  
- 相关测试模块（`tests/fault_tolerance/cancellation/*`、`lib/runtime/src/pipeline/network/tcp/tests`）  

**🔍 技术洞察**  

- **架构影响**  
  - **错误边界收敛**：原先在多处使用 `panic!`，导致单个流异常会直接导致整个 `dynamo` 进程退出，影响整个服务可用性。现在改为在流级别捕获错误、记录并返回 `Result`，仅关闭对应连接，提升系统容错能力。  
  - **职责划分更清晰**：`wait_for_connection_tasks` 把 “等待 reader/writer 任务 + 处理服务器 FIN” 抽离为独立函数，降低 `TcpClient::new` 中的嵌套层级，增强可读性并为后续复用提供入口。  
  - **上下文统一**：对 `AsyncEngineContext`（`kill` / `stop`）的调用统一放在异常路径，确保异常传播到引擎层，避免“死锁”或“资源泄漏”。  

- **性能影响**  
  - **微小开销**：新增的 `tokio::join!`、日志记录以及 `Result` 包装带来的运行时开销极低，且在正常路径（无错误）下基本保持原有逻辑。  
  - **更快的错误回收**：以前 `panic!` 触发堆栈展开会耗时较长，改为 `warn` + 直接 `break`，在异常情况下可以更快释放资源，提升整体吞吐量的稳定性。  

- **安全考虑**  
  - **避免进程级崩溃**：`panic!` 可能导致未捕获的内存泄露或未完成的清理工作（如文件句柄、网络端口），改为受控错误处理降低了潜在的 DoS 风险。  
  - **日志泄露控制**：使用结构化日志（`tracing::warn!(err = ?e, ...)`）而非直接 `panic!`，更易通过日志审计发现异常来源，提升审计可追踪性。  
  - **序列化错误安全**：在服务端发送控制消息时捕获 `serde_json::to_vec` 可能的错误，防止因内部序列化异常直接 panic。  

**⚠️ 潜在风险**  

1. **错误沉默**：过度捕获错误后如果上层未及时检查返回的 `Result`，可能导致错误被忽视而进入错误的业务状态。需要确保调用方（如 `TcpClient::new`）在 `wait_for_connection_tasks` 结果上进行适当的错误上报。  
2. **日志噪声**：将多处 `panic!` 改为 `warn`，在高负载场景下如果异常频繁，日志量会激增，可能影响日志存储或监控系统。建议对频繁错误进行聚合或限速。  
3. **兼容性**：部分外部依赖（如自定义 `Controller` 实现）可能仍期待 `panic!` 抛出异常来终止流程。需要确认所有调用方已适配返回 `Result` 的新语义。  

**💡 关注建议**  

- **监控与告警**：在生产环境为 `warn` 日志添加对应告警阈值，防止异常频繁埋藏在大量日志中。  
- **回归测试**：运行完整的集成测试套件（包括 fault‑tolerance 场景），确保在异常路径下上下文状态（`killed`/`stopped`）与计数器 `cancellation_counter` 的增减行为符合预期。  
- **文档更新**：在项目的错误处理章节补充说明：流级错误不再导致进程 panic，错误将通过 `AsyncEngineContext` 传播并在 `TcpClient`/`TcpServer` 中记录。  
- **代码审查重点**：后续对 `TcpClient`、`TcpServer` 的功能扩展（如 TLS、QUIC）时，保持同样的“错误 → warn → kill/stop → graceful shutdown”模式，避免再次出现 `panic!`。

### feat: Support a single snapshot agent across all workload namespaces (#8740)
**SHA**: `38782e0` | 🔗 [查看提交](https://github.com/ai-dynamo/dynamo/commit/38782e0a052944a64633359b66fa85b68f112aa0)

**🎯 变更类型**：功能增强  

**⚡ 重要程度**：🔴 高  

**📋 变更摘要**  
本次提交实现了 **单实例 Snapshot Agent**，即可以在集群级别只部署一个 privileged DaemonSet 并为所有工作命名空间提供 checkpoint 存储访问。为此：

1. 在 Helm charts 中新增 `storage.accessMode`（`agentMount` / `podMount`），并把原来 Operator 里的 `checkpoint.storage` 移至 **snapshot chart**。  
2. 在 Operator 中加入 PVC 自动创建/校验逻辑（`EnsureStoragePVC`），并将 storage 配置透传到 checkpoint 注入路径。  
3. 扩展 checkpoint 协议，支持在注解中携带 storage type 与 basePath；提供 `ApplyCheckpointStorageMetadata`、`ResolveStorage` 等工具函数。  
4. Snapshot‑agent 控制器改造：在 `podMount` 模式下通过 `/host/proc/<pid>/root` 访问工作负载挂载的 PVC，加入 PID‑一致性校验与路径合法性检查。  
5. 大量代码路径（PodSpec 注入、Controller、测试、文档）同步更新，移除旧的 `checkpoint.storage` 验证并加入兼容性错误提示。  

**🎯 影响范围**  
- `deploy/operator`：API (`types.go`)、内部 checkpoint 包、所有控制器 (`dynamocomponentdeployment`, `dynamographdeployment`, `clique`)、Helm values 与文档。  
- `deploy/snapshot`：Helm chart（README、values、templates）、Agent Controller、Executor（checkpoint / restore）以及协议层 (`protocol/*`)。  
- 依赖 `snapshotprotocol` 的其他内部库（如 `dra`、`runtime`）。  
- 单元测试与 CI。  

**🔍 技术洞察**  

| 维度 | 影响分析 |
|------|----------|
| **架构影响** | 1. **从 per‑namespace DaemonSet → 单实例 DaemonSet**，显著简化资源拓扑，降低 DaemonSet 数量。<br>2. **存储发现职责转移**：原先 Operator 从所在命名空间的 snapshot‑agent 直接读取 PVC；现在在 `podMount` 场景下 Operator 负责在每个工作命名空间创建/校验 PVC，存储路径通过注解在 workload pod 与 agent 之间共享。<br>3. 新增 `storage.accessMode` 配置，使 chart 与 Operator 配置解耦，提升可插拔性。 |

### feat(mm-routing): lightseek-mm Rust-frontend MM-aware KV routing (#9272)
**SHA**: `c2fdd14` | 🔗 [查看提交](https://github.com/ai-dynamo/dynamo/commit/c2fdd14a99b071a290bee13c98e5b47abf3b830a)

**🎯 变更类型**：Feature (新增 MM‑aware KV 路由 + 相关实现、脚本、测试)  
**⚡ 重要程度**：🔴 高  

**📋 变更摘要**  
1. 在 Rust 前端引入 **lightseek‑mm** 特性，使用 `llm‑multimodal` 计算每张图片的 token 数、展开占位符并生成 **精确的 MM‑aware KV 路由信息**。  
2. 增加 `async-lock / moka / tagptr` 依赖，实现 **并发安全的 LRU 维度缓存**，以及 **单飞锁** 用于跨请求的单飞（single‑flight）获取。  
3. 在 `OpenAIPreprocessor` 中完成 **图片哈希 (xxh3‑64)**、维度获取（Header‑only HTTP / data‑URI 解析）以及 **路由侧 `mm_hashes` 组装**，并在 `gather_multi_modal_data` 与 `gather_mm_exact_routing_info` 中把这些信息注入 `PreprocessedRequestBuilder`。  
4. 为不同聊天模板（Phi‑3‑vision、LLaVA‑1.5 等）实现 **占位符模板**（`<|image_{n}|>` / `<image>`），并在需要时把混合 `text+image` 内容平铺为单字符串。  
5. 拓展启动脚本：`agg_multimodal_router.sh`（Rust 前端 + lightseek‑mm） 与 `agg_multimodal_router_chat_processor.sh`（Python‑processor 方案），并加入 **前端解码** (`--frontend-decoding`) 变体。  
6. 大量单元/端到端测试覆盖 **content‑addressed 哈希、data URI、Range 请求、日志输出**，确保路由与 vLLM KV 缓存命中行为一致。
