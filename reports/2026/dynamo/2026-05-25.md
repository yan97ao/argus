#### 🔴 高重要度变更 (2)

### refactor(parsers): decouple dynamo-parsers from dynamo-protocols (#9922)
**SHA**: `8a102e5` | 🔗 [查看提交](https://github.com/ai-dynamo/dynamo/commit/8a102e5a0ce098006764c6e4747c05c4efb0ca07)

**🎯 变更类型**：重构（decouple dynamo‑parsers from dynamo‑protocols）  
**⚡ 重要程度**：🔴高  
**📋 变更摘要**：本次提交将 `dynamo-parsers` 与 `dynamo-protocols` 的直接依赖拆除，改为在使用方完成协议层映射。为此新增了 parser‑native 的工具调用结构体（`ToolCallResponse`、`ToolCallResponseChunk` 等），并在 `openai::chat_completions` 模块实现了映射函数及对应单元测试；同时删除 `Cargo.toml` 中的跨 crate 依赖并调整聚合/流式解析 API 返回值。  

**🎯 影响范围**：  
- `lib/parsers`（所有 tool‑call 相关解析入口）  
- `lib/llm/src/protocols/openai/chat_completions`（映射函数、聚合器）  
- `lib/llm` 及其 Rust 绑定（Python、KVBM）等消费 `dynamo-parsers` 的 crates  
- CI/Test 用例（新增大量映射验证）  

**🔍 技术洞察**  

- **架构影响**  
  - **解耦层级**：`dynamo-parsers` 只输出与协议无关的原生结构，`dynamo-protocols` 仅在上层或消费方进行映射，形成了**清晰的职责分离**（解析 vs. 传输协议）。  
  - **向上兼容**：通过在 `openai::chat_completions` 中提供 `tool_call_response_to_protocol` / `..._chunk_to_protocol`，保持了旧有协议输出不变，避免了突发性破坏。  
  - **API 演进**：公开的 `try_tool_call_parse_*` 系列函数签名从返回协议类型改为返回原生结构，属于**破坏性 API 变更**，需要下游 crate 更新。  

- **性能影响**  
  - **额外映射成本**：在聚合器 (`DeltaAggregator`) 中多了 `map(super::tool_call_response_to_protocol)`，以及用户侧可能的映射调用，带来极小的 O(N) 结构复制开销（N 为 tool‑call 数量），对整体请求响应时间影响可忽略。  
  - **编译时间**：移除 `dynamo-parsers` 对 `dynamo-protocols` 的依赖，降低了 crate 间的循环依赖，理论上可缩短增量编译时间。  

- **安全考虑**  
  - **攻击面未扩大**：仅是结构体搬迁与映射，无新增 I/O、网络或 unsafe 代码。  
  - **数据完整性**：映射函数使用字段逐一复制，保持原始解析结果不变，降低因序列化/反序列化错误导致的数据泄露风险。  
  - **测试覆盖**：新增的测试对比了结构体相等性及 JSON 序列化，确保在解耦后**输出完全等价**，提升了安全可靠性。  

**⚠️ 潜在风险**  

1. **向下兼容破坏**  
   - 外部项目直接依赖 `dynamo-parsers` 并期待返回 `dynamo-protocols` 类型将编译失败。  
2. **映射实现分散**  
   - 新增的映射函数位于 `openai` 协议层，若有其他协议（如 Azure、Anthropic）需要同样映射，可能出现代码重复。  
3. **错误映射导致细微差异**  
   - 只要映射实现不完整（如忘记复制某字段），会导致协议层 JSON 与历史版本不一致，潜在兼容性问题。  
4. **测试依赖的隐藏假设**  
   - 测试使用 `serde_json::to_string` 进行比较，若协议结构加入新可选字段且未在映射中设置，序列化差异可能在未来漏报。  

**💡 关注建议**  

- **文档与迁移指南**：在 `dynamo-parsers` README 中明确说明返回类型已改为 parser‑native，并提供示例映射代码，帮助使用者快速迁移。  
- **版本号策略**：由于是破坏性更改，建议将主版本号提升（例如 `2.0.0`），并在 `Cargo.toml` 中加入 `#[deprecated]` 警告（如果保留旧函数）。  
- **统一映射库**：考虑在 `dynamo-protocols` 或单独的 `dynamo-mappers` crate 中抽象通用映射实现，避免在每个协议实现里重复相同代码。  
- **CI 防护**：在 CI 中加入对比旧二进制输出（如使用 `cargo test --release` 的基准）与新输出的差异检查，确保未来改动不意外更改协议层 JSON。  
- **兼容层 shim**：如果有大量下游仍依赖旧签名，可在 `dynamo-parsers` 中提供一个兼容层（如 `pub fn try_tool_call_parse_aggregate_legacy(...) -> Result<Vec<ChatCompletionMessageToolCall>, _>`），内部调用新实现并映射，以平滑过渡。  

---  

此次解耦提升了代码库的模块化和可维护性，短期需要注意 API 迁移和映射实现的完整性，长期将为多协议支持与独立演进奠定更稳固的基础。

### refactor: Use delta output kind for vLLM token streaming (#9915)
**SHA**: `2b980ba` | 🔗 [查看提交](https://github.com/ai-dynamo/dynamo/commit/2b980ba10c9e0e54ab4b0fab4f09bdec31208e46)

**🎯 变更类型**：功能增强 / 重构  
**⚡ 重要程度**：🔴高  

**📋 变更摘要**  
1. 将 Dynamo 对 vLLM 的 token 流式请求统一为 `RequestOutputKind.DELTA`，并强制 `detokenize=False`，以实现“增量”(delta) 输出模式。  
2. 重构 `handlers.py` 与 `llm_engine.py` 的流式处理逻辑：改为直接使用 vLLM 返回的增量 token / log‑probs 切片，去除累积截取，并在统计 `completion_usage` 时通过累计计数器得到正确的 token 数。  
3. 新增完整的单元测试覆盖 delta‑mode 的 token、logprob、finish_reason、空 delta 以及多索引交叉输出等场景，确保行为一致性。

**🎯 影响范围**  
- `components/src/dynamo/vllm/handlers.py`（核心流式处理）  
- `components/src/dynamo/vllm/llm_engine.py`（统一 LLM 引擎入口）  
- 相关测试文件 `test_vllm_delta_streaming.py`、`test_vllm_engine.py`  
- 任何使用 Dynamo‑vLLM 集成进行流式生成的上层服务（Chat、API、SGLang 等）

**🔍 技术洞察**  

- **架构影响**  
  - 引入 **Delta 输出模式** 作为内部统一约定，降低了 Dynamo 与 vLLM 之间的“桥接”压力：后端仅在 `stream_interval` 触发时发送 token 增量，而不是每次都发送完整累计列表。  
  - `RequestOutputKind` 的强制覆盖使 Dynamo 内部不再依赖调用方的 `output_kind` 配置，提升了内部一致性。  
  - 通过 `total_output_tokens_by_index` 计数器实现 **跨索引并行流** 的正确 token 计数，支撑多流（如并行批次）场景的使用。

- **性能影响**  
  - **带宽与延迟**：增量传输显著减少每次 RPC 的 payload，大幅降低网络/IPC 带宽占用，尤其在高 `stream_interval`（如 4）时可减少 70%+ 的数据量。  
  - **CPU/内存**：省去对累计 token 列表的切片操作，减少 Python 列表拷贝与 GC 负担；计数器是 O(1) 维护。  
  - **日志/监控**：保留 `completion_usage` 的累计统计，兼容现有监控仪表盘，无额外计算开销。

- **安全考虑**  
  - 变更仅涉及内部数据流向与统计逻辑，不涉及凭证、访问控制或外部输入的安全校验，暂无直接安全风险。  
  - 需确认 **输出模式强制** 不会突破用户对 `output_kind` 的安全/合规期待（例如禁止完整累计输出以防泄露上下文），但该风险已在上层通过 `DisaggregationMode` 与 `stream_interval` 控制。

**⚠️ 潜在风险**  

1. **向后兼容性**  
   - 直接强制 `output_kind = DELTA` 可能导致使用方（自定义插件或第三方集成）仍期待 CUMULATIVE 输出而未做适配，出现 token 列表为空或不完整的错误。  
2. **空 delta 块的处理**  
   - 虽已在代码中保留空块以生成 `completion_usage`，但若下游强制忽略空块（如过滤逻辑），统计可能不完整。  
3. **日志概率对齐**  
   - `logprobs` 现在基于 **零偏移** 读取；若 vLLM 将来改变内部 `logprobs` 排列（例如加入 padding），现有切片逻辑可能产生错位。  
4. **并发索引计数**  
   - `total_output_tokens_by_index` 通过字典累计，若出现极端高并发（数千索引）可能导致字典膨胀，虽概率低但需监控。  
5. **测试覆盖不足的边界**  
   - 当前单元测试未覆盖 `stop_reason` 与 `finish_reason` 同时出现的情况，以及 `output_kind` 既非 DELTA 也非 CUMULATIVE 的异常路径。

**💡 关注建议**  

- **文档与使用说明**：在 `dynamo` 文档中明确指出内部已强制使用 DELTA 模式，建议用户在自定义 `SamplingParams` 时不再设置 `output_kind` 或保持默认。  
- **兼容层保护**：在 `handlers.py` 前置一个兼容检查，如检测请求显式请求 CUMULATIVE 且业务确实需要，可抛出友好错误或提供后备路径。  
- **监控计数**：在 `generate` 与 `generate_tokens` 中加入 DEBUG 级别日志，记录 `total_output_tokens_by_index` 的大小变化，以便捕获异常增长。  
- **回归测试**：补充以下情景的测试：  
  1. 同时出现 `finish_reason` 与 `stop_reason`；  
  2. `logprobs` 为 `None` 或仅部分 token 提供；  
  3. 大批量并行索引（>1000）下的计数准确性。  
- **性能基准**：在真实 GPU/CPU 部署上对比 **stream_interval=1**（旧累计模式） 与 **stream_interval=4**（新增 delta 模式）的吞吐量和延迟，确保改动实现预期的 30%+ 性能提升。  
- **安全审计**：虽然当前无直接安全影响，但建议在 CI 中加入 **敏感信息泄露检测**，防止因增量输出现有日志或监控系统意外输出完整上下文。  

---  
通过上述改动，Dynamo 在与 vLLM 的流式交互上实现了更轻量、低延迟的增量传输，同时保持了完整的使用统计。不过在正式发布前，请重点验证向后兼容性和极端并发场景。祝发布顺利！
