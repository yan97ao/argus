### fix: restore --enforce-disagg to reject requests before prefill router activates (#6957)
**SHA**: `f934744` | 🔗 [查看提交](https://github.com/ai-dynamo/dynamo/commit/f934744fb9a9f38f22584cdf8f8c39d6f1a0c710)

**🎯 变更类型**：Bug修复 / 功能增强  
**⚡ 重要程度**：🔴高  
**📋 变更摘要**：将原先的 `--decode-fallback` 标记重命名为 `--enforce-disagg`，并恢复其“在 prefill 路由器未激活时直接拒绝请求”的行为。默认值改为 `false`（即不强制），保持原有的 fallback 机制；开启 `--enforce-disagg` 时若仍未发现 prefill worker，则立即返回错误。相关代码在前端参数、C/Go 绑定、Python/Rust 绑定、路由实现以及测试用例中统一更新。

**🎯 影响范围**：  
- `components/src/dynamo/frontend`（参数解析、入口调用）  
- `deploy/inference-gateway/epp`（插件 Go 实现）  
- `lib/bindings/*`（C、Rust、Python 绑定）  
- `lib/llm/src/*`（路由器配置、PrefillRouter 逻辑）  
- 文档 `docs/kubernetes/inference-gateway.md`、示例 YAML、相关单元/集成测试

**🔍 技术洞察**  
- **架构影响**：  
  - 引入显式的 “强制分离” 开关，使前置路由层能够在 prefill worker 尚未注册时立即失败，防止请求误入聚合解码路径导致不一致的服务行为。  
  - 通过统一的配置字段 `enforce_disagg`，在多语言绑定（C、Go、Python、Rust）和内部 Rust 结构体之间保持一致，降低参数错位风险。  
- **性能影响**：  
  - 对正常流量影响极小，仅在无 prefill worker 场景下增加一次快速错误返回，省去后续的 fallback 处理路径。  
  - 对异常流量（启动阶段或 worker 暂不可用）可避免不必要的 decode‑only 处理，提高整体算力利用率。  
- **安全考虑**：  
  - 新增的错误路径不会泄露内部实现细节，仅返回统一的 `PrefillError::NotActivated`，安全性保持。  
  - 由于默认改为非强制，部署时若误开启 `--enforce-disagg` 可能导致服务在 worker 启动慢时不可用，需在运维文档中强调使用场景。  

**⚠️ 潜在风险**  
1. **默认行为变化**：原项目默认在没有 prefill worker 时直接失败（`decode_fallback=false`），现在默认会 fallback（`enforce_disagg=false`），可能导致已有部署在启动阶段出现意外的 decode‑only 路由，进而影响性能基准。  
2. **兼容性**：旧的环境变量 `DYN_DECODE_FALLBACK` 已被淘汰，仍在使用者脚本中留存会被忽略，可能产生配置失效的困惑。  
3. **测试覆盖**：部分测试仍使用旧参数名或默认值，需要确保 CI 中所有相关测试已更新，否则可能出现讹误的通过率。  
4. **错误提示统一**：在 `PrefillRouter::disabled` 分支仍会返回 `PrefillError::NotActivated`，但错误信息未区分 “fallback 被禁用” 与 “强制分离开启”，调试时可能稍显模糊。  

**💡 关注建议**  
- **运维**：在升级指南中明确标出 `DYN_DECODE_FALLBACK` 已废除，推荐使用 `DYN_ENFORCE_DISAGG`，并提示默认行为已改为 “允许 fallback”。对于需要严格的分离模式，务必在启动脚本中显式添加 `--enforce-disagg`。  
- **监控**：在监控面板添加 `prefill_router_not_activated` 计数，区分是因为 fallback 还是强制拒绝，以便快速定位启动阶段的 worker 注册问题。  
- **回滚方案**：若业务需要恢复旧的默认“直接失败”行为，可在部署时显式传入 `--enforce-disagg=true`，无需回滚代码。  
- **文档与示例**：同步更新所有示例 YAML、README 以及 CI 测试脚本，去除已废弃的 `DYN_DECODE_FALLBACK` 环境变量示例。  
- **代码审查**：后续若再次调整默认行为，请确保在 `RouterConfig`、`FrontendConfig`、以及语言绑定层的默认值保持一致，避免潜在的配置漂移。

### fix: TRT-LLM multimodal preprocessor - remove default_multimodal_input_loader from the embedding paths (#6924)
**SHA**: `e6ddf0e` | 🔗 [查看提交](https://github.com/ai-dynamo/dynamo/commit/e6ddf0eaf9ae7a662ec2c655541eca1d7e09df45)

**🎯 变更类型**：Bug修复 / 功能增强  
**⚡ 重要程度**：🔴 高  

**📋 变更摘要**  
1. 删除了 `default_multimodal_input_loader` 的调用，改为在 Python 端自行组织 multimodal 输入（包括直接传递 NIXL / embedding 数据和可选的文本 prompt）。  
2. Rust 侧的 `OpenAIPreprocessor` 现在会在 `extra_args` 中保留 `formatted_prompt`，并在多模态数据收集函数中接受该参数，以配合新版 Python 预处理器。  
3. 新增对 `.pt/.pth/.bin` 文件的直接加载路径支持，并在相应分支返回结构化的 `multi_modal_embeddings`，实现对 TRT‑LLM 的嵌入路径/ NIXL 流的兼容。  

**🎯 影响范围**  
- `components/src/dynamo/trtllm/multimodal_processor.py`（Python multimodal processor）  
- `lib/llm/src/preprocessor.rs`（Rust OpenAIPreprocessor）  
- 触及的核心模块：`dynamo::trtllm`、`llm::preprocessor`、`common::multimodal`、前端 token‑generation 接口  

**🔍 技术洞察**  

- **架构影响**  
  - 将原本由 `tensorrt_llm.inputs.default_multimodal_input_loader` 负责的图像下载、预处理逻辑下放至项目内部，实现 **“前端 → 后端 → TRT‑LLM”** 的更清晰数据流。  
  - 新的 `processed_inputs` 结构在 Python 端统一组织：`prompt_token_ids`、`prompt`、`mm_processor_kwargs`、`multi_modal_embeddings`，使 TRT‑LLM 仅关心最终的张量，而不再依赖内部加载器的副作用。  
  - Rust 端通过 `extra_args.formatted_prompt` 把模板渲染后的 Prompt 传递到 Python，保持了原有的 **“模板 → tokenization → multimodal”** 路径，只是把 responsibility 移植到了 Python。  

- **性能影响**  
  - 移除 `asyncio.to_thread(default_multimodal_input_loader…)`，消除了对同步下载/预处理的线程切换开销，在高并发场景下 **提升吞吐**。  
  - 对于仅使用预先生成的 embeddings（NIXL / 路径），直接在内存中转发，避免了不必要的磁盘/网络 I/O。  
  - 需要注意：对 `image_url` 的普通图片仍然通过 `ImageLoader.load_image_batch` 同步加载，若大量图片上传仍可能成为瓶颈；未来可考虑再次引入异步/并行加载。  

- **安全考虑**  
  - 去掉了 `default_multimodal_input_loader` 的网络下载路径，降低了 **外部资源注入**（如恶意图片 URL）攻击面。  
  - 新增的 `load_tensor_from_path_or_url` 仍支持 URL，仍需对 URL 的来源进行审计或在生产环境使用白名单/签名校验。  

**⚠️ 潜在风险**  

| 风险点 | 说明 | 可能影响 |
|-------|------|----------|
| **回退路径移除** | 依赖 `default_multimodal_input_loader` 处理普通 image URL 的老业务会直接返回 `None` 或报错。 | 造成请求失败、服务不可用。 |
| **`formatted_prompt` 缺失** | Python 端在未获取 `formatted_prompt` 时直接 `return None`，可能导致 token‑ids 正常但 Prompt 丢失的情况。 | 数据不完整、生成结果异常。 |
| **Embedding 加载错误** | `load_tensor_from_path_or_url` 在网络/磁盘异常时返回 `None`，当前代码会 `return None` 中止整个请求。 | 影响 NIXL 流的可靠性。 |
| **结构兼容性** | `processed_inputs` 结构变化（不再一定包含 `prompt_token_ids`）需要调用方（TRT‑LLM）同步更新。 | 运行时类型错误或字段缺失。 |
| **并发文件 I/O** | 仍采用同步方式读取 `.pt/.pth/.bin`，在高并发下可能出现磁盘争用。 | 增加响应延迟。 |

**💡 关注建议**  

1. **回退兼容**：在文档或代码注释中明确指出，普通图片 URL 必须通过项目自带的 `ImageLoader` 或外部前置服务处理；若仍需旧行为，考虑保留可选的 `enable_default_loader` 开关。  
2. **完善异常处理**：对 `load_tensor_from_path_or_url` 加入重试/超时机制，或在加载失败时返回空 embedding 并记录监控指标，避免一次异常导致整批请求被截断。  
3. **增加单元/集成测试**：  
   - NIXL / embedding‑path + formatted_prompt 完整路径。  
   - 仅图像 URL（无 embedding）场景，确认返回 `None` 并记录合理错误。  
   - token_ids 正常、prompt 缺失等边界情况。  
4. **监控与报警**：在 `processed_inputs` 为 `None` 或 `multi_modal_embeddings` 加载异常时上报告警，以快速捕获不兼容的调用方。  
5. **性能评估**：在生产流量下对比去除 `default_multimodal_input_loader` 前后的 QPS、CPU/IO 使用率，评估是否需要对同步文件读取做异步化或批量缓存。  
6. **安全审计**：对 `load_tensor_from_path_or_url` 进行 URL 白名单或签名校验，防止恶意文件被加载进模型。  

通过以上措施，可最大化本次修复的收益（消除 TRT‑LLM 的已知 bug、提升吞吐），并把潜在的回退兼容和安全隐患控制在可接受范围。

### feat: enable KVBM metrics on k8s for kimi k2.5 recipe (#6963)
**SHA**: `a620a9c` | 🔗 [查看提交](https://github.com/ai-dynamo/dynamo/commit/a620a9cf767ebcb55918978703f7b55f2b44d18d)

**🎯 变更类型**：功能增强  

**⚡ 重要程度**：🔴高  

**📋 变更摘要**  
- 为 Kimi‑K2.5 `agg` 方案在 Kubernetes 上启用 KVBM（Key‑Value‑Block‑Manager）指标，新增 `DYN_KVBM_METRICS` 环境变量和对应端口 (`6880`)。  
- 新增 `README.md`、`podmonitor‑kvbm.yaml`，以及在 `deploy‑kvbm.yaml` 中暴露 `system`（9090）、`nixl`（19090）和 `kvbm`（6880）端口，便于 Prometheus Operator 抓取 `/metrics`。  
- 通过环境变量 `DYN_KVBM_CPU_CACHE_GB=10` 默认开启小容量 CPU 缓存，以加快容器启动。  

**🎯 影响范围**  
- **trtllm/agg/nvidia** 目录下的 Kimi‑K2.5 配方（包括 `deploy‑kvbm.yaml`、`podmonitor‑kvbm.yaml`）。  
- **Dynamo Operator** 在 Kubernetes 上管理的 `DynamoGraphDeployment` 实例。  
- 依赖 **Prometheus Operator** 的监控链路（PodMonitor、ServiceMonitor）。  

**🔍 技术洞察**  

- **架构影响**  
  - **新增 Metrics 接入层**：通过在 worker 容器中打开 `6880` 端口并暴露 `DYN_KVBM_METRICS`，把 KVBM 的内部状态（KV‑cache 使用率、命中率、回收率等）通过 Prometheus 指标公开。  
  - **组件耦合度下降**：指标采集是旁路的（side‑car 方式），不影响业务路径，仅在 `worker` Pod 上增加一个 HTTP 端点。  
  - **部署流程更新**：需要用户在 `deploy‑kvbm.yaml` 中填入实际镜像 tag，并确保 `hf-token-secret`、`model-cache` PVC 已就绪；同时提供 `podmonitor‑kvbm.yaml` 供监控系统使用。  

- **性能影响**  
  - **CPU/内存开销**：开启 KVBM metrics 会在每个 worker 上启动 `prometheus` 客户端库的采集线程，带来极低的 CPU (≈0.5‑1% 单核) 与内存占用（数 MB）。  
  - **网络开销**：`kvbm` 端口仅在本地 Pod 与 Prometheus 之间通信，流量极小（指标大小 < 10 KB/抓取），对节点网络影响可以忽略。  
  - **启动时间**：将默认 CPU cache 从 100 GB 降至 10 GB，可显著缩短容器启动时间，提升调度灵活性。  

- **安全考虑**  
  - **暴露端口**：新增 6880 端口在 Pod 中对外可达，仅限于拥有 `metrics-enabled=true` 标签的 worker，且默认仅在内部集群网络中可访问。  
  - **指标泄露**：Prometheus 指标内可能包含模型缓存大小、KV‑block 使用率等内部信息，若监控系统对外开放，需要通过网络策略或 RBAC 限制访问。  
  - **配置安全**：`DYN_KVBM_METRICS=true` 为显式开关，默认开启可避免因忘记设置导致监控缺失，但也必须在安全审计中明确此入口。  

**⚠️ 潜在风险**  

1. **端口冲突**：如果集群节点上已有服务占用 `6880`（或 `9090/19090`），Pod 启动会失败。  
2. **监控误报**：Prometheus 抓取频率为 5 s，若集群规模很大（数百个 worker），可能导致 Prometheus 服务器负载升高，需要审慎规划抓取间隔。  
3. **资源泄漏**：`DYN_KVBM_CPU_CACHE_GB` 默认 10 GB，如果实际业务需要更大缓存而未调优，可能导致频繁缓存回收，性能下降。  
4. **安全误配置**：若 `nvidia.com/metrics-enabled: "true"` 标签被误加到非 KVBM worker，Prometheus 仍会尝试抓取，导致 404/500 错误日志。  

**💡 关注建议**  

- **部署前检查**：  
  - 使用 `kubectl get svc -A | grep 6880` 确认端口未被占用。  
  - 确认 `hf-token-secret`、`model-cache` PVC 已创建。  
- **监控侧配置**：  
  - 在 Prometheus Operator 中为 `PodMonitor` 设置合理的 `scrape_interval`（如 30 s）并开启 `relabel_configs` 过滤掉不需要的标签。  
  - 若需要跨命名空间抓取，确保 `PodMonitor` 的 `namespaceSelector` 与 RBAC 配置匹配。  
- **性能调优**：  
  - 依据实际 KV‑cache 命中率监控数据，适时增大 `DYN_KVBM_CPU_CACHE_GB`（如 20‑40 GB）以提升吞吐。

  - **内存占用**大幅下降：原实现在大文件（数百 MB 甚至 GB）时会一次性加载到内存，导致 OOM；改为流式写入后内存保持在单块 Chunk 大小（由 `object_store` 决定，通常几 KB~几 MB）。  
  - **磁盘 I/O** 增加但可接受：写入本地磁盘是必然开销，且在大多数部署场景下磁盘写入速度远高于网络下载速度，整体下载时间基本不变。  
  - **重试与退避**：在网络不稳定时可自动恢复，避免因瞬时错误导致整体任务失败。

- **安全考虑**：  
  - 默认仍使用 HTTPS，只有在环境变量 `AWS_ALLOW_HTTP=true` 时才降级为 HTTP，保持原有安全策略。  
  - 新增的 `LORA_DOWNLOAD_TIMEOUT_SECS` 环境变量仅控制客户端超时，不会泄露凭证或改变访问权限。  
  - 采用 `tokio::fs::File` 异步写入，避免阻塞主线程，降低 DoS 风险。

- **可维护性**：  
  - 代码分离为 `stream_to_file`、`download_file_with_retry` 两个职责清晰的函数，易于单元测试和后续扩展（例如支持进度回调）。  
  - 错误处理统一使用 `anyhow::Context`，提供丰富的错误链，便于调试。  
  - 临时目录清理逻辑封装在 `cleanup_on_error` 闭包，确保异常情况下不会残留垃圾文件。

**⚠️ 潜在风险**：

1. **磁盘空间不足**：大 LoRA 包下载到本地临时目录后会占用完整空间，若磁盘配额不足可能导致下载失败。  
2. **并发下载冲突**：同一目标路径被多个任务并行下载时，临时目录命名基于时间纳秒，冲突概率低但仍需上层保证调用序列。  
3. **环境变量失效**：`LORA_DOWNLOAD_TIMEOUT_SECS` 若设置不合理（如过低）会导致正常下载被提前中止。  
4. **对象存储兼容性**：`ClientOptions::with_timeout` 依赖底层 `reqwest` 实现，极端情况下某些自定义 S3 兼容实现可能不完全遵循超时语义。

**💡 关注建议**：

- **部署前检测**：在生产环境中监控磁盘余量，尤其是首次下载大模型时；可在启动脚本中预留足够的 `/tmp` 或指定目录。  
- **超时配置**：根据网络带宽和模型大小，合理设置 `LORA_DOWNLOAD_TIMEOUT_SECS`（默认 3600 s），避免误判为网络故障。  
- **并发控制**：若业务会并发拉取同一 LoRA，建议在上层加锁或使用下载缓存，以防重复下载和临时目录冲突。  
- **监控与告警**：在日志中关注 `S3 download failed` 警告，设置告警阈值，以便及时发现网络或存储异常。  
- **回归测试**：加入对极大文件（>1 GB）下载的集成测试，验证在低内存节点（如 2 GB）仍能成功。

**🔍 技术洞察**  

- **架构影响**  
  - **定位层**：`get_device_numa_node` 由进程层调用，向系统层（`nvidia-smi`）查询 GPU‑CPU 拓扑。原实现直接使用进程本地的 `device_id`，在 `CUDA_VISIBLE_DEVICES` 重映射后会得到错误的物理 GPU，导致 NUMA 绑定错位。  
  - **新增映射层**：`cuda_device_id_to_nvidia_smi_id` 成为桥梁，使得上层逻辑无需感知环境变量，仅通过统一的 `device_id` 接口获得正确的物理标识，保持了模块边界的清晰。  
  - **向后兼容**：若未设置 `CUDA_VISIBLE_DEVICES`，函数直接返回原索引，保持原有行为。  

- **性能影响**  
  - **额外开销**：在每次查询前解析一次 `CUDA_VISIBLE_DEVICES`（字符串分割、遍历），该操作的时间复杂度为 O(N)（N 为可见 GPU 数），在常规机器（≤8 张卡）中可忽略。  
  - **命令行调用保持不变**：仍通过 `nvidia-smi` 执行一次外部进程，成本主导仍为外部工具而非 Rust 代码。  

- **安全考虑**  
  - **环境变量可信度**：`CUDA_VISIBLE_DEVICES` 属于用户可控的环境变量。函数仅解析逗号分隔的标识并直接作为 `nvidia-smi -i` 参数，未执行任何 shell 解析或拼接，避免了命令注入风险。  
  - **日志信息**：新增的 `tracing::warn!` 与 `tracing::trace!` 不泄露敏感信息，仅包含设备索引。

1. **异常格式**：如果 `CUDA_VISIBLE_DEVICES` 包含非法字符、空项或 UUID 与整数混杂，当前实现会把原始子串直接传给 `nvidia‑smi`，可能导致命令执行失败并回退到 heuristic，进而产生错误的 NUMA 绑定。  
2. **运行时环境变化**：`CUDA_VISIBLE_DEVICES` 在进程启动后若被修改，已有的缓存（若后续加入）可能不一致；当前实现每次查询都重新读取，虽然安全但会产生稍微的重复解析开销。  
3. **兼容性**：在旧版 `nvidia-smi`（不支持 UUID）或异常的 `-i` 参数行为上，仍可能出现错误提示。  

**💡 关注建议**  

- **加强输入校验**：在 `cuda_device_id_to_nvidia_smi_id` 中加入对每个 token 的基本校验（是否为纯数字或符合 `GPU-` 前缀的 UUID），并在异常时记录更明确的错误信息。  
- **单元/集成测试**：添加覆盖以下场景的测试：  
  - 未设置 `CUDA_VISIBLE_DEVICES`（直接映射）。  
  - 只保留单个 GPU（如 `CUDA_VISIBLE_DEVICES=2`）。  
  - 多 GPU 重映射（如 `0,2,4`）。  
  - 包含 UUID（如 `GPU-1234abcd,GPU-5678efgh`）。  
  - 非法格式（空项、非法字符）。  
- **缓存优化（可选）**：如果后续发现 `get_device_numa_node` 被频繁调用，可考虑在首次解析后缓存映射表，避免重复字符串分割。缓存需在进程生命周期内保持不变，以防环境变量后期被修改导致不一致。  
- **文档更新**：在项目的 NUMA 绑定或 KVBM 使用说明中，标注该函数已经对 `CUDA_VISIBLE_DEVICES` 做了兼容，建议用户在多租户或容器环境下使用该变量时，确认 NUMA 亲和性仍然符合预期。
