### fix(kv-router): expose kv_cache_events_applied in multithreaded indexer (#8088)
**SHA**: `d0d9c03` | 🔗 [查看提交](https://github.com/ai-dynamo/dynamo/commit/d0d9c03013e5d389cf60e88ce355717c04be7c88)

**🎯 变更类型**：功能增强 / 重构  
**⚡ 重要程度**：🔴高  
**📋 变更摘要**：  
本次提交在 KV‑Router 的多线程索引器路径中，新增对 `kv_cache_events_applied` 计数器的暴露与高效记录。通过引入 `EventKind`、`PreBoundEventCounters`，在每个工作线程启动时预先绑定对应的 Prometheus `IntCounter`，并在事件处理循环中直接递增，避免了原先在 `apply_event` 里每次通过 `with_label_values` 进行哈希查找的开销。同时扩展了 `SyncIndexer` 接口以接受可选的 `KvIndexerMetrics`，并在 `ThreadPoolIndexer`、各具体索引实现以及 `KvIndexer` 主体中传递该指标对象。

**🎯 影响范围**：  
- `lib/kv-router/src/indexer/*`：所有实现 `SyncIndexer`（`ConcurrentRadixTree`、`ConcurrentRadixTreeCompressed`、`PositionalIndexer`）的工作线程入口。  
- `lib/kv-router/src/indexer/thread_pool.rs`：线程池创建逻辑。  
- `lib/kv-router/src/indexer/metrics.rs`：计数器结构与预绑定实现。  
- `lib/kv-router/src/indexer/kv_indexer.rs`：全局索引路径的事件应用计数。  
- `lib/llm/src/kv_router/indexer/mod.rs`：入口根据线程数选择并注入 `KvIndexerMetrics`。

**🔍 技术洞察**  

- **架构影响**  
  - **接口演进**：`SyncIndexer::worker` 从单参数扩展到接受 `Option<Arc<KvIndexerMetrics>>`，保持向后兼容（原调用仍可传 `None`），但所有内部实现必须适配新签名。  
  - **职责分离**：计数逻辑从 `KvIndexerMetrics::increment_event_applied` 移至 `PreBoundEventCounters::inc`，实现“预绑定”以把标签解析搬到线程启动阶段，降低热路径复杂度。  
  - **可观测性统一**：单线程 `KvIndexer` 与多线程 `ThreadPoolIndexer` 现在在同一指标 `kv_cache_events_applied` 上保持一致，便于统一监控与告警。  

- **性能影响**  
  - **计数开销显著降低**：原实现在每次事件处理时调用 `IntCounterVec::with_label_values`，涉及哈希查找与锁竞争。新实现在工作线程启动时一次性完成所有标签的 `IntCounter` 解析，后续仅执行 `counter.inc()`（几乎是原子增量），预计可减小热点路径的 CPU 使用 30%‑50%，尤其在高并发（数千 QPS）场景下效果更加明显。  
  - **内存占用**：`PreBoundEventCounters` 持有固定数量的 `IntCounter`（12 个），对内存影响可忽略。  
  - **线程启动成本**：首次绑定时会额外进行 12 次 `with_label_values` 调用，成本极低且仅在工作线程创建时触发。  

- **安全考虑**  
  - 代码未涉及外部输入的直接安全风险。  
  - 新增的 `Arc<KvIndexerMetrics>` 在多线程间共享，只读取内部 `IntCounterVec`，符合 Prometheus 线程安全设计。  
  - `EventKind::of` 对 `KvCacheEventData` 进行模式匹配，未出现未覆盖分支，编译期已保证完整性。  

- **可维护性**  
  - 引入 `EventKind` 统一了事件类型标签的来源，减少了散落的字符串常量使用。  
  - `PreBoundEventCounters` 的实现通过 `cfg(feature = "metrics")` 隐蔽了无指标构建时的空结构，保持编译兼容。  
  - 通过 `SyncIndexer` 签名的变更，所有实现必须同步更新，防止遗漏导致运行时 panic。  

**⚠️ 潜在风险**  

| 风险点 | 说明 | 严重度 |
|--------|------|--------|
| **实现遗漏** | 若未来新增的 `SyncIndexer` 实现未按照新签名实现 `worker`，编译会失败但可能在分支代码中遗漏 `metrics` 传递，导致监控缺失。 | 中 |
| **指标标签不匹配** | `PreBoundEventCounters::new` 假设 `KvIndexerMetrics::kv_cache_events_applied` 已包含全部标签组合；若指标初始化变更（新增状态标签），未同步更新 `PreBoundEventCounters` 将导致运行时 panic（`with_label_values` 返回错误）或计数缺失。 | 中 |
| **特性切换** | 在 `metrics` feature 关闭时，`PreBoundEventCounters` 为空结构，`inc` 成为空操作。若代码误用了返回的计数器（如尝试读取值），将得到默认或零值，可能掩盖业务问题。 | 低 |
| **线程安全** | `Arc<KvIndexerMetrics>` 在多线程共享，若后续在 `KvIndexerMetrics` 中加入非线程安全字段，可能产生竞争。 | 低 |
| **回滚兼容** | 老版本的 `ThreadPoolIndexer::new` 仍被保留，未传递 metrics，可能导致监控数据不完整。需要在部署文档或调用方明确使用 `new_with_metrics`。 | 低 |

**💡 关注建议**  

1. **测试覆盖**  
   - 为 `PreBoundEventCounters::inc` 添加单元测试，覆盖每种 `EventKind` + 每种 `KvCacheEventError` 的组合，确保计数路径完整。  
   - 在高并发负载（≥ 10k QPS）下跑基准，对比旧实现的 CPU 使用与新实现的差异，验证性能提升。  

2. **文档与迁移**  
   - 在项目的迁移指南中说明 `ThreadPoolIndexer::new` 已被废弃，推荐使用 `new_with_metrics` 并传入 `KvIndexerMetrics::from_component`。  
   - 更新 `SyncIndexer` trait 文档，明确 `metrics` 参数为可选，且若提供，计数器将在热路径中使用。  

3. **监控验证**  
   - 部署后检查 Prometheus 中 `kv_cache_events_applied_total`（假设为总计）是否出现预期的 12 组时间序列；若缺失，确认 `PreBoundEventCounters::new` 与指标初始化一致。  

4. **未来扩展**  
   - 如需新增事件类型或错误码，只需在 `EventKind`、`KvCacheEventError` 与 `PreBoundEventCounters::new`/`inc` 中对应添加分支，编译器会强制完整性。  
   - 考虑把 `PreBoundEventCounters` 生成逻辑抽象为宏，以免手动维护大量 `with_label_values` 调用。  

通过上述改动，项目在多线程路径上实现了更高效的可观测性记录，兼顾了性能与代码可维护性。只要在后续实现新索引器或扩展事件种类时同步更新相关映射，即可保持系统的可靠与可监控。

### feat(async-openai): accept dict format for tool-call arguments (#7772)
**SHA**: `59f474a` | 🔗 [查看提交](https://github.com/ai-dynamo/dynamo/commit/59f474a2d971399ae769dec7eb9fe3e570e36d66)

**🎯 变更类型**：功能增强  

**⚡ 重要程度**：🔴高  

**📋 变更摘要**  
在 `async-openai` 的 `FunctionCall`、`FunctionCallStream` 与 `ChatCompletionMessageToolCallChunk` 类型上，新增了自定义反序列化器，使 `arguments` 字段既可以接受 JSON 字符串，也可以直接接受已经解析为对象（即字典）形式。实现了统一的 `String` 语义并在单元测试中覆盖了多种合法/非法输入，提升了对 LangChain 等上层框架的兼容性。

**🎯 影响范围**  
- `lib/protocols/src/types/chat.rs`（核心协议层）  
- 依赖 `async-openai::types::chat` 的所有上层模块（如对话管理、工具调用、流式响应处理）  
- 与工具调用（Tool Call）相关的序列化/反序列化逻辑（包括 `ChatCompletionMessageToolCall`、`ChatCompletionStreamResponseDeltaFunctionCall`）  

**🔍 技术洞察**  

- **架构影响**  
  - **局部类型复写**：原本直接 `pub use async_openai::types::chat::{FunctionCall, FunctionCallStream, ChatCompletionMessageToolCallChunk}` 的重导出被改为本地结构体定义，以挂载自定义 `#[serde(deserialize_with)]`。这在模块内部形成了 **“包装层”**，对外仍保持相同命名，兼容性良好。  
  - **向后兼容**：由于仍实现了 `Serialize`，序列化结果保持与上游相同（`arguments` 始终输出为字符串），对已有服务端或客户端不产生破坏。  
  - **依赖隔离**：仅在本 crate 内部使用本地定义，避免对 upstream crate 产生修改需求，降低维护成本。  

- **性能影响**  
  - **反序列化开销**：在 `deserialize_arguments`/`deserialize_arguments_opt` 中，对对象 `Value` 进行一次 `serde_json::to_string`（内存分配 + 字符串生成）。在大批量工具调用或高频流式输出时会产生 **额外的 O(N)** 序列化成本。  
  - **正常路径影响可接受**：工具调用的 `arguments` 通常体积不大（几百字节），额外开销在微秒级别，整体吞吐影响有限。  
  - **编译产出**：新增约 220 行代码和若干测试，二进制体积微增（主要是 `serde_json` 的 `to_string` 实现已在依赖中），对运行时内存占用几乎无影响。  

- **安全考虑**  
  - **输入验证**：自定义解码器仅接受 **字符串或对象**，对其他 JSON 类型（数字、布尔、数组、null）直接报错，防止意外的非预期结构进入业务层。  
  - **JSON 注入**：`serde_json::to_string` 对任意 `Value::Object` 完全转义，生成合法 JSON 字符串，不会导致注入风险。  
  - **错误信息**：返回的错误信息会暴露原始 `serde_json::Value` 的 `Debug` 表现，略显技术细节，但不泄露敏感数据。  

**⚠️ 潜在风险**  

1. **类型冲突**：本地重新定义的 `FunctionCall` 等与 upstream 同名，但在同一个作用域中使用 `pub use` 进行混合可能导致 **名称冲突** 或 **意外的类型不匹配**（尤其在其它 crate 对 `async_openai::types::chat::FunctionCall` 直接使用时）。  
2. **序列化不一致**：如果上游在未来更改了这些结构体的序列化细节（例如添加新字段），本地复制的实现可能不再同步，导致 **隐藏的兼容性裂痕**。  
3. **性能回退**：在极端高并发的工具调用场景（如每秒数万次调用）额外的 `to_string` 可能累计为 **显著的 CPU 与内存压力**。  
4. **错误传播**：`deserialize_arguments` 中使用 `unwrap()` 将 `serde_json::to_string` 的错误直接 panic（虽然该函数理论上永不错误），但若未来 `serde_json` 实现改变，可能导致不可预期的 panic。  

**💡 关注建议**  

- **文档与迁移指南**：在项目的 CHANGELOG 与开发者文档中明确说明 `arguments` 现在支持字典格式，并指出仍会被序列化为字符串，避免使用方对字段类型产生误解。  
- **统一类型别名**：考虑在 `pub use` 前加入 `pub(crate) use` 并在公共 API 中只暴露本地实现，防止外部 crate 误用 upstream 原始类型。  
- **性能压测**：在高并发场景下（如 10k+ RPS 的工具调用），加入基准测试，评估 `to_string` 开销，必要时提供可选的 “原始对象” 模式或缓存路径。  
- **错误处理改进**：将 `serde_json::to_string(&v).unwrap()` 替换为 `serde_json::to_string(&v).map_err(D::Error::custom)?`，消除潜在的 panic 源。  
- **同步上游变更**：设立 CI 检查，比较本地重新定义的结构体字段集合与 upstream 的对应版本，若出现差异自动提示更新。  
- **安全审计**：虽然当前实现安全，但建议在接受外部 `arguments` 前，引入 **schema 校验**（如使用 `jsonschema`）或白名单检查，以防止业务层对恶意 JSON（超大嵌套、深度）产生解析耗时或 DoS。

通过上述改动，项目对多种上层框架的兼容性大幅提升，同时保持了对原有协议的向后兼容。只要在文档、测试和 CI 中持续跟踪上游类型的演进，就能将潜在风险控制在可接受范围内。

### fix(llm): close push-router cleanup gap (#8181)
**SHA**: `3efc733` | 🔗 [查看提交](https://github.com/ai-dynamo/dynamo/commit/3efc733f8a7d7990e9080d4239ede97cc66c91c1)

**🎯 变更类型**：Bug修复  
**⚡ 重要程度**：🔴高  
**📋 变更摘要**  
- 引入 `PendingDispatchGuard` 与 `spawn_cleanup_task`，在请求调度阶段（`direct` 调用返回之前）就准备好释放调度器槽位与执行延迟关闭的逻辑。  
- 修复原有实现的“drop‑guard 漏洞”：当请求在取得流之前就被 Drop，原来的 `RequestGuard` 无法触发 `free`/`deferred_close`，导致调度器资源泄漏。  
- 通过在 `direct` 前构造 `PendingDispatchGuard`、在成功取得流后 `disarm`，确保无论是正常流式消费还是异常提前 Drop，都能正确清理。

- `core/llm` 子模块：`push_router.rs`（关键的 KV‑Router 调度路径）  
- 依赖该路由的所有 LLM 引擎实例（单节点、分布式等），尤其是使用 `AsyncEngine` 的调用方。

**🔍 技术洞察**  

| 维度 | 影响描述 |
|------|----------|

2. **双重释放风险**

3. **异常抛出影响**  
   - `spawn_cleanup_task` 本身不捕获 `chooser.free` 的错误，除日志外不做回退；如果 `free` 失败且对业务有强依赖（如配额计数），仍会留下错误状态。

| **调度效率** | DRA 的 **DeviceClaim** 仍然走 scheduler，但因为不再使用 GPU‑device‑plugin 的 `ExtendedResource`，调度器需查询 `DeviceClass` 与 `ResourceClaim`，这在大规模集群中略微增加调度计算成本。 |

### 安全考虑  
| 维度 | 说明 |
|------|------|
| **RBAC 扩展** | 新增对 `resource.k8s.io`（DeviceClass、ResourceClaimTemplate）的 `create/list/watch` 权限。若集群中存在未受控的 ServiceAccount，可能被滥用创建任意 GPU 资源声明，建议仅为 Operator 的 ServiceAccount 授权。 |
| **Sidecar 代码执行** | GMS 侧车运行 `python3 -m gpu_memory_service`（用户可自定义镜像），如果镜像来源不受信任，可能成为攻击面。应在 CI/CD 流程中对 GMS 镜像进行签名、扫描。 |
| **共享 Volume & UDS** | 主容器与 GMS 通过共享 `emptyDir` 与 Unix Domain Socket 交互，容器之间的文件系统访问权限默认受同一 Pod 的安全上下文控制，风险较低。但若用户自行修改 `extraPodSpec` 绕过默认安全上下文，可能导致跨容器的文件访问。建议在 Operator 中强制设置 `volumeMounts` 为只读（仅主容器写入 TMPDIR，GMS 只读）或使用 `ReadOnlyMany`（不适用于 emptyDir）。 |
| **DRA 资源泄漏** | 若 GMS 相关的 `ResourceClaimTemplate` 没有被及时删除（例如组件切换为不启用），可能导致 orphaned Claim 对象占用 GPU 资源。实现中通过 `toDelete` 标记在 `SyncResource` 中自动清理，仍需在大规模滚动升级时验证。 |

---

## ⚠️ 潜在风险  

1. **集群不满足 DRA 条件**  
   - Kubernetes 版本低于 1.32，或缺失 NVIDIA DRA driver。Operator 会报错，但用户可能忽视日志，导致部署卡在 `Reconciling`。  
2. **DeviceClass 不存在**  
   - `GenerateGMSResourceClaimTemplate` 在检查 `DeviceClass` 时返回错误。如果用户自定义 `deviceClassName` 而未提前创建对应 `DeviceClass`，部署会失败。  
3. **资源配额冲突**  
   - `gpuMemoryService.enabled=true` 同时仍在 `resources.limits.gpu` 中声明 GPU 数量。虽然代码会删除这些字段，但若其他控制器（如 HPA、KEDA）基于这些字段做 scaling，可能产生不一致。  
4. **Sidecar 失败导致 Pod 无法启动**  
   - `startupProbe` 超时或 GMS 初始化进程崩溃会阻塞整个 Pod 启动，影响业务可用性。  
5. **RBAC 过宽**  
   - 为所有 `DeviceClass` 与 `ResourceClaimTemplate` 开放 `create` 权限
