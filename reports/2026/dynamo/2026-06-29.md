### fix(operator): skip initial dgd restart request (#10955)
**SHA**: `82ce03d` | 🔗 [查看提交](https://github.com/ai-dynamo/dynamo/commit/82ce03df22b6e92089702e0e848e06c55e53e20d)

**🎯 变更类型**：Bug修复 / 功能增强（新增 CRD 校验，防止创建时误发 Restart 请求）  
**⚡ 重要程度**：🔴 高  

**📋 变更摘要**  
- 在 `DynamoGraphDeployment`（v1alpha1 与 v1beta1）CRD 中加入 **X‑validation**，强制 `spec.restart` 在对象创建时必须为空，仅在创建后才能设置 `restart.id` 以触发重启。  
- 同步更新 Helm Chart、Operator 配置中的 CRD 文件与 `controller-gen` 版本至 `v0.17.3`，并把 `envtest`、`controller-tools` 版本提升至对应的 Kubernetes 1.30 兼容套件。  
- 新增单元测试 `dynamographdeployment_validation_test.go` 验证上述校验行为。  

**🎯 影响范围**  
- `deploy/operator/api/v1alpha1`、`v1beta1` 的 Go 类型定义  
- 所有 `deploy/operator/config/crd/bases/*.yaml` 与 Helm Chart 中对应的 CRD YAML  
- `Makefile` 与 `suite_test.go` 中的 `ENVTEST_K8S_VERSION`、`CONTROLLER_TOOLS_VERSION`  
- CI / 本地测试环境（需要 1.30 版的 k8s test 二进制）  

**🔍 技术洞察**  

- **架构影响**  
  - 将业务约束从控制器代码迁移到 **CRD 校验层**，提升了数据一致性，避免了控制器在 `Reconcile` 前即收到非法对象。  
  - 通过 `x-kubernetes-validations`（OpenAPI v3）实现声明式校验，使得所有使用标准 Kubernetes API 的客户端（kubectl、helm、client‑go）都能统一得到错误反馈。  
  - 版本升级至 `controller-tools v0.17.3` 与 `envtest 1.30`，确保生成的 OpenAPI 与校验规则在目标集群上可用，保持向后兼容性。  

- **性能影响**  
  - 校验在 API Server **Admission** 阶段执行，逻辑极其轻量（一次旧对象属性检查），对吞吐量影响可忽略。  
  - 由于新增了自动化测试，CI 运行时间略增（envtest 启动 1.30 版镜像），对生产运行无直接影响。  

- **安全考虑**  
  - 防止用户在创建时即写入 `spec.restart`，避免 **恶意或误操作** 触发无限重启循环，降低潜在的服务拒绝（DoS）风险。  
  - 规则使用 `optionalOldSelf: true`，确保在 **更新** 场景仍能正常接受 `restart.id`，不影响授权模型。

1. **集群版本兼容性**

2. **工具链升级副作用**  
   - `controller-tools v0.17.3` 生成的代码可能与旧版 `kubebuilder` 项目产生不兼容警告，需确保所有本地开发者和 CI 环境同步升级。  

3. **测试环境依赖**  
   - `suite_test.go` 已硬编码 `1.30.0` 的二进制路径，若 CI 未预装对应二进制或网络访问受限，测试会失败。  

4. **回滚策略**  
   - 若已有 `DynamoGraphDeployment` 对象在创建时意外包含 `spec.restart`，升级后这些对象在 `kubectl apply` 时会被拒绝，需要手动清理或在升级脚本中先删除该字段。  

**💡 关注建议**

- **回滚演练**：提供迁移脚本，自动遍历现有 `DynamoGraphDeployment`，若检测到 `restart` 字段则清除后再重新应用，以免升级阻塞。

### fix(mocker): make KVBM allocation and offload lifecycle race-free [DYN-3299]
**SHA**: `65bdf67` | 🔗 [查看提交](https://github.com/ai-dynamo/dynamo/commit/65bdf67a0fb7a0bb43c0df309f8dd4f01fbe19fc)

**🎯 变更类型**：Bug修复 / 代码改进（使 KVBM 的 `Use`、`Decode` 与离线 offload 的生命周期 **彻底无竞态**）  
**⚡ 重要程度**：🔴 高  
**📋 变更摘要**  
- 为 **KvManager** 与 **MockWorker** 引入 **`G1Acquire`** 枚举，统一表示 G1 资源分配结果：`Ready(..)`、`CapacityExhausted`、`BlockedOnOffload{offload_id,deadline}` 与 `RetryNow{generation,released}`。  
- 将 **`Use`**、**`Decode`**、**`reserve_decode_blocks`** 等路径改为 **原子（一站式）** 的分配‑提交流程，防止 “先分配后失败” 的部分成功情形。  
- 新增 **离线 offload 结算机制**：`SettlementToken`、`PipelineCheckpoint`、`OffloadCoordinator` 与一整套 **准备 → 预提交 → 确认** 的状态机，确保 **G1→G2、G2→G3、G2→G4** 三条流水线在同一 offload 事务中 **因果一致**。  
- `MockOffloadEngine` 由仅返回 `SwapInHandle`（`AtomicBool`）改为 **`SwapInHandle { id, status_rx }`**，通过 `watch` 通道发布 **Swap‑in 完成、取消、失败** 状态。  
- `MockWorker` 使用 `watch` 通道报告 **reservation count** 与 **swap‑in状态**，并暴露 `subcribe_reservation_count()` 与 `reservation_notifier()`（后者已改为 `watch`）。  
- 调度器 (VLLM) 现在在 **请求入口、调度、预取、解码** 时统一检查 `G1Acquire`，并在 **离线 offload 受阻** 时记录 `OffloadDependency`，在后续 tick 中自动刷新。  
- 大量 **单元测试** 与 **稳定性回归**（包括离线 replay、worker 硬件故障注入、settlement 错误路径等）随之加入。

## 🔍 技术洞察

### 1. 架构层面
| 维度 | 影响 |
|---|---|
| **状态一致性** | 原先 `KvManager::process` 直接返回 **已分配块数**，在 `Use` 路径上若后续 `reserve_decode_blocks` 失败会产生 **部分已分配‑未提交** 的状态，导致 G1 资源泄漏或错误的 `KVCacheEvent`。新的 `G1Acquire` 将分配、检查、提交合并为 **一次性原子操作**，保证 **要么全成功，要么全不分配**。 |
| **离线 offload 事务化** | 通过 `SettlementToken` 捕获每条 pipeline（G1→G2、G2→G3、G2→G4）在 **准备阶段** 的检查点，`OffloadCoordinator` 负责 **准备‑确认‑完成** 的三阶段闭环。所有下游链路（G2→G3、G2→G4）只在 **父 lease 完成** 后才注册，防止 **“先提交子链，后失效父链”** 的竞态。|
| **调度器与 offload 的耦合** | `VllmCore` 现在在 **请求入队、swap‑in、预取** 时都能捕获 `OffloadDependency`，并在每轮 `tick_offload_only` 中自动刷新。这样 **调度器不再隐式假设 offload 必然成功**，而是显式等待或取消。 |
| **错误传播** | `SettlementError`、`PipelineFailure`、`AdvanceAcknowledgementError` 为 **结构化错误**，可在 `MockOffloadEngine::settle_after` 中直接 panic（在测试/模拟环境），或在生产代码中转换为可观测的错误码。 |
| **资源所有权模型** | `SwapInHandle` 由 `OffloadCoordinator` 统一管理，使用 `watch` 频道而不再依赖 `AtomicBool`，**避免**因为多线程 `Relaxed` 读取导致的 “错过完成信号” 或 “提前完成” 的竞态。 |

### perf(mocker): reduce scheduler and FPM publish overhead (#11025)
**SHA**: `81cf355` | 🔗 [查看提交](https://github.com/ai-dynamo/dynamo/commit/81cf35564312d64039509494cd3410a319674de5)

**🎯 变更类型**：性能优化  

**⚡ 重要程度**：🔴高  

**📋 变更摘要**  
1. 重构 Forward Pass Metrics (FPM) 序列化流程，改为在单一可复用的 `Vec<u8>` 缓冲区上就地序列化，避免为每次快照分配新内存。  
2. 引入 `PendingFpm` 与 `serialize_fpm_into`，将快照的发送与序列化解耦，使调度器只负责把快照放入通道，真正的序列化与发布在统一的 publisher 任务中完成。  
3. 为事件平面编码加入 `encode_envelope_parts`，在发布原始字节时直接借用 payload，进一步消除一次拷贝。  
4. `sleep_until_precise` 在已超时的情况下不再创建 `timerfd`，直接 `yield_now`，降低调度器在高加速比模拟中的开销。  
5. NATS 传输层新增 `publish_subject` 参数，Publisher 端可预先确定主题，省去每条消息的字符串拼接。  

**🎯 影响范围**  
- `lib/llm/src/fpm_publisher.rs`（FPM 快照序列化与发布）  
- `lib/mocker/src/common/utils.rs`（精确睡眠实现）  
- `lib/runtime/src/distributed.rs`（KV‑router NATS 发送接口）  
- `lib/runtime/src/transports/event_plane/codec.rs`（Msgpack 编码）  
- `lib/runtime/src/transports/event_plane/mod.rs`（EventPublisher）  
- `lib/runtime/src/transports/event_plane/nats_transport.rs`（NATS 传输实现）  

**🔍 技术洞察**  

| 维度 | 影响 |
|------|------|
| **架构影响** | - 引入 **生产者‑消费者** 双层通道：调度器仅产出 `PendingFpm`，统一的 publish 任务负责序列化并写入事件平面。<br>- `EventPublisher` 现在持有 **完整主题字符串** (`subject`)而非在每次发布时拼接，降低了字符串分配与处理路径。<br>- `Codec::encode_envelope_parts` 为 **零拷贝** 接口，保持了现有 `EventEnvelope` 抽象不变，兼容旧的 `encode_envelope`。 |
| **性能影响** | - **内存分配**：原先每条 FPM 快照会 `rmp_serde::to_vec_named` 产生新 `Vec<u8>`，现在复用同一缓冲区，仅在第一次使用时分配，后续 `clear` 重用，显著降低 GC/allocator 压力。<br>- **CPU**：序列化函数改为 `Serialize::serialize`（使用 `Serializer::with_struct_map`）比 `to_vec_named` 略快；`publish_bytes_ref` 省去一次 `Bytes::from(Vec<u8>)` 的拷贝。<br>- **调度器**：`sleep_until_precise` 在已超时的情况下直接 `yield_now`，避免创建 `timerfd`，在高加速比模拟（如 1000×）下可削减调度循环的系统调用开销。<br>- **吞吐**：统一的 publisher 任务保持 **事件顺序**，且序列化/发布在单线程内进行，避免多线程争抢同一 NATS 连接导致的流控抖动。 |
| **安全考虑** | - 新增的 `encode_envelope_parts` 直接借用 `&[u8]`，若调用方在 `publish_bytes_ref` 返回前修改该 slice，可能导致发布的 payload 受污染。当前实现立即把数据序列化成独立的 `Bytes`，因此安全性保持不变。<br>- `serialize_fpm_into` 仍然返回 `Result<()>`，错误路径会记录警告，不会泄露内部状态。<br>- 对外暴露的 `kv_router_nats_publish_subject` 与 `publish_subject` 并未引入额外的权限检查，保持与原有 `kv_router_nats_publish` 相同的安全模型。 |
| **可维护性** | - 新增的 `PendingFpm`、`serialize_fpm_into`、`encode_envelope_parts` 均被放在私有模块并配有单元测试，代码可读性提升。<br>- 将 `serialize_fpm` 重命名为内部测试函数，防止外部误用旧接口。<br>- `EventPublisher` 的字段改动（去除 `scope`）需要更新文档和可能的外部构造器，已在代码中统一使用 `subject`。 |

**⚠️ 潜在风险**  
1. **共享缓冲区竞争**：`payload` 缓冲区在 publisher 任务内部复用，若未来将 `publish_bytes_ref` 暴露给多任务并行调用，需确保每次调用在前一次序列化完成后才复用，否者会出现数据竞争。当前实现是单一 async 任务，风险低。  
2. **错误处理**：`serialize_fpm_into` 错误仅记录警告并继续循环，若序列化持续失败（例如结构体不兼容），可能导致大量空消息被丢弃。建议在监控中追踪此类警告频率。  
3. **主题拼接改变**：`EventPublisher` 现在把 `subject` 预先计算一次，若业务方在运行时动态改变 `scope`（极少见），可能出现不一致。需在变更说明中明确 “topic 与 scope 在创建时固定”。  
4. **测试覆盖**：新增的 `test_serialize_fpm_into_reuses_buffer` 与 `test_msgpack_codec_borrowed_envelope_roundtrip` 只在单线程环境下执行，建议在多线程/生产环境做压力测试，确保不会出现内部缓冲被提前释放的情况。  

**💡 关注建议**  
- **监控**：在生产环境打开 `FPM direct publisher` 的 **序列化错误计数** 与 **发布失败计数**，结合现有 Metrics 看是否还有遗漏的热点。  
- **基准**：与之前的版本在相同负载 (如 10k FPM 快照/秒) 对比 **CPU 使用率、GC/allocator 频率、内存峰值**，确保优化带来的提升符合预期。  
- **文档**：更新 `EventPublisher` 与 `FpmDirectPublisher` 的公开 API 文档，说明 `publish_bytes_ref` 与 `publish_bytes` 的区别以及何时使用 `PendingFpm`。  
- **代码审查**：确认所有新加入的 `pub async fn kv_router_nats_publish_subject` 只在内部使用，防止外部误传错误的 `Subject`（如未加前缀）。

总体来看，此次提交通过 **一次性复用缓冲区 + 零拷贝发布** 有效降低了 FPM 采集路径的内存分配与系统调用开销，对高吞吐的模拟场景（尤其是 `mocker`）带来显著的性能提升，风险可控。建议在正式发布前进行 **高并发、长时运行** 的集成压力测试，以验证缓冲区复用及 NATS 主题预设在极端负载下的稳健性。
