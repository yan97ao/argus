### fix(vllm): support unified multimodal routing (#10375)
**SHA**: `37ae347` | 🔗 [查看提交](https://github.com/ai-dynamo/dynamo/commit/37ae347e6afaccd7a84ff1a9dbc90cc70a1d9618)

**🎯 变更类型**：功能增强 / 重构 / 安全修复  
**⚡ 重要程度**：🔴高  

**📋 变更摘要**  
本次提交在 Dynamo 的 multimodal 路由层引入了统一的多模态（image、audio、video 等）路由与元数据分组机制。核心改动包括：  
1. 新增对 `mm_position.is_embed` 掩码的解析与在路由填充时的细粒度控制。  
2. 在前端 `vllm_processor` 中实现对多模态特征的 **模态分组**、**占位符序列化** 以及 **单模态传输判断**，并在向后端传递时保留分组信息。  
3. 后端 handler 通过统一接口读取分组后的 hash 与 placeholder（包括 `is_embed`），并在 KV‑cache 构造时保持模态对应。  
4. 大量新增/调整单元测试，覆盖 embed‑mask、分组元数据、统一 vision‑chunk 映射以及 e2e KV‑cache 重用场景。  

**🎯 影响范围**  
- `components/src/dynamo/common/multimodal/routing_utils.py` → 路由填充逻辑  
- `components/src/dynamo/frontend/vllm_processor.py` → 前端特征分组、模态判定、发送器初始化逻辑  
- `components/src/dynamo/vllm/handlers.py` → 后端接收与 KV‑cache 构造路径  
- 相关测试文件（unit、e2e）以及辅助函数 `__init__` 中的导入方式  
- 任何使用 Dynamo‑vLLM 多模态功能的模型（尤其是开启 `use_unified_vision_chunk` 的模型）  

---

### 🔍 技术洞察  

| 维度 | 影响描述 |
|------|----------|
| **架构影响** | - 引入了 **模态分组元数据**（`mm_hashes_by_modality`、`mm_placeholders_by_modality`），使前后端对多模态特征的约定由单一列表升级为 **键值对结构**，兼容 legacy image‑only 场景。<br>- 添加 `_single_transfer_modality` 判断，避免在混合模态下错误使用单模态共享内存传输，从而提升系统鲁棒性。<br>- `is_embed` 掩码的序列化/反序列化使路由层能够在 **Embedding‑only** token（如 CLIP‑vision encoder 输出）上保持原始 token 不被 padding，提升模型兼容性。 |
| **性能影响** | - **正向**：在仅 image 场景仍保持原有 “flat” 结构，避免额外的 dict 查找开销；在混合模态时通过 **分组传输** 可以绕开不必要的共享内存拷贝，仅在模态统一时启用高效运输。<br>- **潜在**：对每个 placeholder 进行 `is_embed` 列表转换（`torch.as_tensor`）会在路由阶段产生轻微 CPU 开销，影响极小 (< 0.5 ms)。<br>- E2E 测试加入了 **total blocks 稳定性** 与 **重用率** 断言，确认 KV‑cache 重用未因分组而退化。 |
| **安全考虑** | - 新的 `is_embed` 处理链路通过 `detach().cpu().tolist()` 规避了潜在的 **CUDA 同步** 与 **GPU 直接泄露**，保持了原有的安全边界。<br>- 对 `extra_args` 中的模态字典进行 **类型检查**（`isinstance(..., dict)`），防止恶意构造的非‑dict 参数导致后端误解析。整体未引入新外部依赖，安全面影响可视为 **低**。 |
| **可维护性** | - 代码采用 **函数抽象**（`_mm_feature_modality`、`_serialize_mm_placeholder` 等）提升可读性。<br>- 通过统一的分组返回值，后续若新增模态（如 `depth`、`pointcloud`）只需在字典中添加键值，无需改动路由核心。<br>- 测试覆盖率显著提升，尤其是对 `is_embed` 边缘情况的验证，有助于防止回归。 |

---

### ⚠️ 潜在风险  

1. **兼容性回退**  
   - 老版前端（未使用 `mm_hashes_by_modality`）仍能工作，但如果后端在某些实现中硬编码 `image` 键，可能导致 **模态映射失效**。已在代码中保留 legacy flat 兼容，但第三方插件需注意。  

2. **混合模态错误的发送器选择**  
   - `_single_transfer_modality` 只在全部特征模态相同且非空时返回模态；若特征列表中出现 `modality=None`（默认 `image`），但实际包含 audio，则会错误判定为单模态。建议在特征构造阶段显式填写 `modality`。  

3. **`is_embed` 长度不匹配**  
   - 当 `is_embed` 长度与 `length` 不一致时会抛 `ValueError`，这在某些自定义前端实现中可能触发。可考虑在前端提前对齐长度并给出更友好的错误信息。  

4. **性能回退**  
   - 在极端大批量 multimodal 请求（数千个占位符）时，遍历 `is_embed` 掩码并进行列表转化的 CPU 开销可能累计。若出现性能瓶颈，可考虑 **批量 numpy** 或 **torch** 向量化实现。  

---

### 💡 关注建议  

| 对象 | 建议 |
|------|------|
| **前端开发者** | - 对所有 multimodal 特征显式设置 `modality`，确保分组正确。<br>- 若使用 `is_embed`，务必保证长度等于 `length`，并在必要时提前 `torch.tensor` 转换以降低路由阶段的 Python 循环。 |
| **后端/运维** | - 监控 `mm_hashes_by_modality` 与 `mm_placeholders_by_modality` 的实际大小，防止异常大 dict 导致内存抖动。<br>- 在使用 `use_unified_vision_chunk` 的模型上，验证 `vision_chunk` 键是否在 downstream 组件（如 vLLM cache）被正确识别。 |
| **测试/CI** | - 保持新增的单元测试与 e2e 场景在 CI 中运行，特别是 `test_build_prompt_preserves_grouped_forwarded_hashes_for_fallback` 与 `test_http_vs_data_uri_same_image`，它们能够捕获模态键映射错误。 |
| **未来扩展** | - 设计时已采用字典式分组，后续如果加入 **视频**、**深度图** 等，只需在 `mm_feature.modality` 中声明新标识即可，无需改动路由核心。<br>- 若计划支持 **多模态跨请求共享**（如跨请求 cache），可以直接复用已有的 `mm_hashes_by_modality` 结构。 |

---

**总体结论**  
此提交为 Dynamo‑vLLM 多模态路由提供了更通用、细粒度且向前兼容的实现，对 **模型缓存复用** 与 **跨模态特征传输** 产生了积极的正向效应。风险主要集中在兼容性与 `is_embed` 长度校验上，建议在上线前在多模态混合集群中做一次完整的 smoke‑test，以确认所有模态键均被后端正确解析。

### feat(examples): add XPU DRA deployment examples (aggregation, disaggregation, tracing) (#9253)
**SHA**: `d17cdf7` | 🔗 [查看提交](https://github.com/ai-dynamo/dynamo/commit/d17cdf71781e65b4a00750b08845e984e4a887ae)

**🎯 变更类型**：功能增强  
**⚡ 重要程度**：🔴高  

**📋 变更摘要**  
本次 PR 为 `examples/backends/vllm/deploy` 添加了一套针对 Intel XPU GPU 的部署示例（aggregation、disaggregation、tracing、KV Router、Planner 等），完整迁移到 Kubernetes Dynamic Resource Allocation（DRA）模式。包括新增 README、若干 `*.yaml` 模板、镜像切换至 `vllm-runtime‑xpu`、以及 KV‑Transfer 与 OpenTelemetry 的默认配置。

**🎯 影响范围**  
- `examples/backends/vllm/deploy/README.md`（文档路径）  
- 新增 `examples/backends/vllm/deploy/xpu/` 目录下 10+ 份部署模板  
- `examples/global_planner/README.md` 与 `global-planner-vllm-test-xpu-dra.yaml`（全局调度示例）  
- 部分原有 `v1beta1/xpu/` 文件的重命名与镜像/环境变量更新  

**🔍 技术洞察**  

- **架构影响**  
  1. **硬件抽象层切换**：原先基于 NVIDIA GPU（`resources.limits.gpu`）的部署改为使用 Intel XPU 通过 DRA `ResourceClaimTemplate` 与 `ResourceClaim`、`ResourceSlice` 进行资源分配。  
  2. **组件镜像分离**：新增 `vllm-runtime-xpu` 镜像，所有涉及 XPU 的 Worker（Prefill/Decode）统一使用该镜像，并通过 `VLLM_TARGET_DEVICE=xpu` 环境变量告知 vLLM 使用 XPU 后端。  
  3. **KV 传输适配**：在 Disaggregated 模式下显式加入 `--kv-transfer-config`，使用 `NixlConnector` 将 KV 缓冲区放在 XPU 上，以避免跨设备拷贝。  
  4. **路由器增强**：提供两种 KV Router 实现：  
     - **完整 KV 事件模式**（依赖 NATS/JetStream）  
     - **本地预测模式**（`--no-kv-events`，通过 TTL/缓存预测）  
  5. **全局 Planner 扩展**：新增 XPU‑DRA 版 `global-planner-vllm-test-xpu-dra.yaml`，在 GlobalPlanner 中同样使用 DRA 资源声明，保持与原 GPU 示例一致的调度策略。  

- **性能影响**  
  1. **资源调度效率**：DRA 通过 `ResourceClaimTemplate` 在调度阶段即完成 GPU （XPU）绑定，避免 Pod 启动后再进行 `DevicePlugin` 分配，理论上可降低调度延迟。  
  2. **KV 共享成本**：使用 `NixlConnector` 将 KV 缓冲区置于 XPU 设备内存，可显著降低跨设备（CPU ↔ XPU）拷贝开销，提升分布式 Prefill/Decode 场景的吞吐与时延。  
  3. **Tracing 开销**：OpenTelemetry 采集会带来额外的 CPU 与网络负载，推荐在生产环境通过 `OTEL_EXPORT_ENABLED=false` 关闭，或对采样率进行调优。  
  4. **Router 计算**：本地预测模式省去 KV 事件的序列化/网络传输，但会引入 TTL 失效与缓存误判的潜在性能波动，需要通过实验调优 `kv_router` 参数。  

- **安全考虑**  
  1. **Secret 暴露**：所有模板仍通过 `hf-token-secret` 将 HuggingFace Token 注入容器，需确保命名空间的 RBAC 限制只能被授权的 ServiceAccount 读取。  
  2. **RBAC 对 DRA**：使用 `ResourceClaimTemplate`、`ResourceSlice` 需要对应的 ClusterRole（如 `dynamo-platform-dynamo-operator-planner`）具备 `resourceclaims`、`resourceslices` 的 `get/list/watch` 权限，若缺失将导致部署失败。  
  3. **镜像来源**：新镜像 `nvcr.io/nvidia/ai-dynamo/vllm-runtime-xpu` 为内部构建，发布前应完成镜像签名与 CVE 扫描，防止供应链风险。  
  4. **驱动/插件依赖**：集群必须安装 Intel XPU 设备驱动和 `intel-resource-drivers-for-kubernetes`，否则 DRA 资源声明会被调度器拒绝，可能导致资源泄露或挂起。

**⚠️ 潜在风险**  
| 风险点 | 可能后果 | 缓解措施 |
|--------|----------|----------|
| 集群未升级至 K8s v1.34 或未启用 DRA API | Deployments 无法创建 `ResourceClaim`，导致启动失败 | 在 CI/文档中明确检查 `kubectl api-versions | grep resource.k8s.io/v1`，并提供升级指南 |
| Intel 资源驱动版本不匹配（DeviceClass 名称、API） | GPU 资源无法分配，Pod 卡在 `Pending` | 在 README 中列出兼容的驱动版本，使用 `kubectl describe resourceclaim` 排查 |
| 镜像未包含 XPU 运行时或缺少 `VLLM_TARGET_DEVICE` 环境变量 | Worker 启动后报 “Unsupported device” | CI 增加镜像构建验证步骤，确保镜像标签 `-xpu` 包含对应二进制 |
| KV Transfer 配置错误（`kv_buffer_device` 仍为 `gpu`） | KV 数据跨 PCIe 迁移导致高时延或 OOM | 提供模板校验脚本或 lint 检查 `kv_transfer_config` 中的 `kv_buffer_device` 必须为 `xpu` |
| OpenTelemetry 未正确配置（OTLP endpoint 不可达） | Worker 启动卡住或日志大量重试 | 在模板中加入 `OTEL_EXPORT_ENABLED` 默认 `false`，仅在需要时手动开启 |
| RBAC 权限不足导致 `resourceclaims`、`resourceslices` 读取失败 | Operator 报错 “forbidden” | 在 `global-planner-vllm-test-xpu-dra.yaml` 中已经加入 `ClusterRoleBinding`，确保用户自行创建相同绑定或在 CI 中检测 |  

**💡 关注建议**  
1. **验证环境准备**：在使用前执行 `kubectl apply -f examples/backends/vllm/deploy/xpu/README.md` 中的前置检查脚本，确认 K8s 版本、DRA 启用、Intel 驱动、Secret 权限。  
2. **镜像构建流程**：建议在项目 `container/` 目录添加 `render_xpu.sh`，统一生成 `vllm-runtime-xpu` 镜像并推送至内部 registry，避免手动复制命令出错。  
3. **CI 自动化**：加入对 `*.yaml` 中 `resourceClaimTemplateName`、`deviceClassName`、`VLLM_TARGET_DEVICE` 的 lint 检查，确保所有 XPU 示例保持一致。  
4. **监控与调优**：部署后通过 `kubectl get resourceclaim,resourceslices -n <ns>` 确认 XPU 绑定成功；开启 `OTEL` 时，监控 Tempo/Prometheus 的采样率，防止对高并发推理产生额外负载。  
5. **文档同步**：README 中的示例路径已经从 `agg_xpu_dra.yaml` → `xpu/agg_xpu_dra.yaml`，请在所有引用位置（包括 CI、发布脚本）同步更新，防止用户因路径错误导致部署失败。  
6. **安全审计**：对 `nvcr.io/nvidia/ai-dynamo/vllm-runtime-xpu` 镜像

### feat(vllm): elastic EP scaling and headless on unified backend (#10834)
**SHA**: `b6edcd4` | 🔗 [查看提交](https://github.com/ai-dynamo/dynamo/commit/b6edcd4230a18a45f03f66094d1ab7b372cb90d0)

**🎯 变更类型**：功能增强 / 架构变更 / 性能优化  
**⚡ 重要程度**：🔴高  

**📋 变更摘要**  
本次提交为统一后端（unified backend）引入了两大新特性：  
1. **Elastic EP（Expert Parallelism）弹性扩缩容**：在 Ray‑DP 后端上实现 `/engine/control/scale_elastic_ep` 控制接口，支持在单节点内动态增减数据并行（DP）规模（2→4→2），并对 `ray.util.state.list_nodes` 进行一次性 shim 替换以兼容 `--minimal` 启动模式。  
2. **Headless 多节点模式**：通过 `--headless` 让除主节点外的副节点只启动 vLLM worker（无 Engine、Scheduler、Dynamo 端点），实现 TP/PP 多节点部署的轻量化路径。  
为支撑以上特性，统一后端启动入口 `run` 增加 `engine_factory` 参数，避免二次解析 CLI；相应单元/集成测试、CI 标记与脚本也同步更新。

### 🎯 影响范围

## 🔍 技术洞察

### 架构影响
- **审计路径独立**：新增 `OtelSink` 作为 `AuditSink` 的实现，使用 OpenTelemetry SDK 的日志导出器（batch processor），与原有的 `stderr`、`nats`、`jsonl`、`jsonl_gz` 完全并行。审计记录不再走 `tracing` 桥接的日志路径，避免 **双重 OTLP 导出**。
- **状态机**：在 `audit::config` 中加入 `CAPTURE_STATE`（UNINITIALIZED / ACTIVE / INACTIVE）` 与 `capture_enabled()`，确保只有在 `init_from_env_with_shutdown` 完成并标记 `ACTIVE` 之后才会创建审计句柄，防止“早期”创建的句柄在 shutdown 后仍尝试发布。

### 性能影响
| 影响点 | 说明 |
|--------|------|
| **请求路径** | `AuditHandle::emit` 已从同步记录改为 `async` 的 sink 调用，**不再阻塞**请求线程。唯一的同步成本是 `Arc::new(request.clone())` 与 `SystemTime::now()`，极小。 |

- **敏感数据泄露**：审计记录包含完整的请求体（包括系统提示、用户消息、工具调用等），如果 OTLP endpoint 对外暴露或未使用 TLS，可能导致 PII 泄漏。建议在部署时：
