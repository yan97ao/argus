### feat(planner): support a static bearer token for Prometheus (#9512)
**SHA**: `d6240e6` | 🔗 [查看提交](https://github.com/ai-dynamo/dynamo/commit/d6240e619a2f26daa789f87a4b8c299cf6b26a15)

**🎯 变更类型**：功能增强  
**⚡ 重要程度**：🔴高  

**📋 变更摘要**  
在 Planner 模块中新增对 Prometheus 拉取接口的静态 Bearer Token 支持，允许通过 `PROMETHEUS_TOKEN` 环境变量在每次 PromQL 请求中添加 `Authorization: Bearer <token>`。该 Token 在服务启动时读取一次，配置字段被标记为 `exclude=True` 以防止序列化泄漏。  

**🎯 影响范围**  
- `components/src/dynamo/planner/config/defaults.py`（默认配置）  
- `components/src/dynamo/planner/config/planner_config.py`（Pydantic 配置模型）  
- `components/src/dynamo/planner/core/base.py`（Planner 初始化）  
- `components/src/dynamo/planner/monitoring/traffic_metrics.py`（Prometheus API 客户端）  

**🔍 技术洞察**  

- **架构影响**  
  - 在 Planner 与 Prometheus 之间的 HTTP 调用链中加入了可选的 `Authorization` 头部，实现方式保持向后兼容：若环境变量未设置，行为与原先完全相同。  
  - 通过 `exclude=True` 防止 token 被写入日志或配置快照，符合最小权限原则。  
  - 只在启动时读取一次，避免在运行时频繁读取文件或环境变量，简化实现但限制了动态轮转能力。  

- **性能影响**  
  - 仅在 `PrometheusConnect` 会话对象的 Header 中追加一个字符串，几乎没有额外的 CPU 或网络开销。  
  - 读取环境变量的成本极低，且在启动阶段完成，对运行时性能基本无影响。  

- **安全考虑**  
  - **机密泄露风险**：Token 通过环境变量注入，若容器/进程的环境被意外导出或日志记录，可能导致凭证泄露。已通过 `exclude=True` 避免序列化泄露，但仍需运维层面确保环境变量的保密。  
  - **一次性读取**：Token 在启动时固定，若使用的是短期 ServiceAccount token（会轮转），将失效导致监控采集中断。计划中已提出 `token_file` 方案，以支持文件监控轮转。  
  - **授权范围**：使用 Bearer Token 说明 Prometheus 前端已被 OAuth 或类似代理强化，确保仅对可信服务开放此功能，防止误用导致权限提升。  

**⚠️ 潜在风险**  

1. **凭证泄露**：不恰当的部署脚本或 CI/CD 环境可能将 `PROMETHEUS_TOKEN` 写入日志或配置快照。  
2. **Token 失效**：对使用 kube‑service‑account token 的集群，静态读取会在 token 轮转后失效，导致监控数据采集中断。  
3. **错误的 Header**：若 `bearer_token` 包含空格或换行，可能导致 Prometheus 返回 401/403，进而影响 Planner 的调度决策。  
4. **兼容性**：部分老版本的 `prometheus-api-client` 可能不支持直接修改 `_session.headers`，在极端情况下会抛出属性错误。  

**💡 关注建议**  

- **部署安全**：务必使用容器运行时的 secret 管理（如 Kubernetes `Secret` + `envFrom`）注入 `PROMETHEUS_TOKEN`，并在 CI/CD 中屏蔽该变量的打印。  
- **监控 Token 轮转**：在需要自动轮转的环境中，尽快实现后续的 `token_file` 配置，以让 Planner 在运行时重新加载 token。  
- **错误监控**：在 Planner 启动后检测 Prometheus 客户端是否成功添加 Header，若出现 401/403，记录明确的告警信息。  
- **文档与示例**：在项目文档中补充示例（如 OpenShift Thanos‑Querier）以及最佳实践，提醒用户不要在本地开发环境中硬编码 token。  
- **回退方案**：提供环境变量 `PROMETHEUS_TOKEN` 的默认空值行为，使旧部署在不提供 token 时仍能继续工作，无需代码改动。  

通过上述措施，可在保证监控安全性的同时，最小化对现有功能的冲击，并为后续动态凭证管理奠定基础。

### test(parity): resolve Harmony divergences and remove tool-call leaks (#9593)
**SHA**: `1bd96f9` | 🔗 [查看提交](https://github.com/ai-dynamo/dynamo/commit/1bd96f97de4c2c803cc618f9f2656ef20c9fbcbb)

**🎯 变更类型**：Bug修复 / 功能增强  
**⚡ 重要程度**：🔴高  

**📋 变更摘要**  
1. 修复 `JailedStream` 在生成 Tool Call 时遗漏 `finish_reason`，确保调用结束标记被正确传播。  
2. 大幅重构 Harmony 协议解析：  
   - 新增 token‑ID 检测、调用标记检测、自动恢复、以及对非工具文本的安全剥离。  
   - 对解析失败、JSON 错误、缺失 `<|call|>` 等异常情况实现更严谨的回退逻辑，并防止协议内容泄露到 `normal_text`。  
3. 更新测试、Parity 夹具以及 HTML 报表生成，使其能够验证新的解析行为并在报告中高亮 Harmony 标记。  
4. 为 vLLM 集成添加 Harmony 编码不可用时的降级路径，保持 parity 与 Dynamo 行为一致。  

**🎯 影响范围**  
- `lib/llm/src/protocols/openai/chat_completions/jail.rs`（核心流控制）  
- `lib/parsers/src/reasoning/gpt_oss_parser.rs`（GPT‑OSS 推理解析）  
- `lib/parsers/src/tool_calling/harmony/harmony_parser.rs`（Harmony 协议主解析）  
- `lib/parsers/src/tool_calling/parsers.rs`（工具调用统一入口）  
- `lib/llm/tests/test_jail.rs`、所有 Harmony 相关 parity fixture YAML（测试层）  
- `tests/parity/parser/generate_parity_chart.py`（HTML 报表生成）  
- `tests/parity/parser/vllm.py`（vLLM 适配层）  

**🔍 技术洞察**  

- **架构影响**  
  - 引入 `OnceLock` 单例 `HARMONY_CALL_TOKEN_IDS`、`GLOBAL_HARMONY_GPTOSS_ENCODING`，在全局层面缓存 tokenizer 编码，避免重复加载。  
  - 通过 `input_contains_call_marker`、`raw_input_text_for_tool_parser` 等函数，将 **文本‑→‑token** 的判定抽象到解析器前置层，保持 `ReasoningParser` 与 `ToolCallParser` 的职责单一。  
  - Harmony 解析器从单一正则匹配扩展为 **多阶段清理+日志**：先正则抽取，再逐步剔除 `commentary`、`analysis`、`message`、特殊 token，确保 `normal_text` 只保留用户可见内容。  
  - 增加 `parse_tool_calls_harmony_complete` 的 **恢复分支**：在 tokenizer/编码失效时回退到正则提取，且在回退后仍会调用 `strip_harmony_protocol_from_normal_text` 防止协议泄漏。  

- **性能影响**  
  - 新的正则与字符串清理在异常路径（解析失败）才会触发，对常规成功路径的 CPU/内存开销几乎不变。  
  - `OnceLock` 缓存使得首次加载后后续调用成本极低。  
  - vLLM 的 fallback ` _harmony_token_ids` 仅在编码可用时调用 `encode`，对大多数模型仍保持原有 token‑ID 流程。  

- **安全考虑**  
  - 通过 **严格剥离** Harmony 标记和原始 JSON（包括不合法的 JSON），防止协议元信息（如 `<|channel|>`、函数名称）泄漏至用户可见的 `normal_text`，降低信息泄露风险。  
  - 解析错误时不再“fabricate”工具调用，仅返回原始文本或空调用，避免因错误的 JSON 合成导致 downstream 业务执行错误。  
  - 所有新日志均使用 `tracing::warn!`/`debug!`，不记录用户实际内容，仅记录结构化的剥离信息，符合最小化敏感信息日志的原则。  

- **可维护性**  
  - 代码拆分为 **检测‑清理‑序列化** 三级函数，职责清晰，后续若 Harmony 协议迭代，只需调整对应正则或清理函数。  
  - 测试覆盖面大幅提升：新增针对隐藏 analysis、异常 JSON、缺失 `<|call|>`、多调用等场景的单元测试，确保未来改动不易回归。  
  - HTML 报表增强了 Harmony token 彩色显示与族过滤功能，提升了内部调试与规范对齐的可视化效率。  

**⚠️ 潜在风险**  

| 风险点 | 说明 | 严重度 |
|---|---|---|
| 正则/清理逻辑不兼容未来 Harmony 小幅变更 | 当前正则硬编码了 `<|call|>` 结束标记，若 OpenAI 增加新结束 token（例如 `<|finish|>`），解析会失效并把内容错误归入 `normal_text`。 | 中 |
| `OnceLock` 初始化失败导致全局降级 | 如果 `get_harmony_encoding()` 抛异常，所有后续调用走回退路径，可能导致大量解析转为正则，性能下降。 | 低 |
| 与旧版 SDK 或自定义 `tool_call_parser` 的兼容性 | 旧版依赖 “envelope 泄漏到 normal_text” 的行为可能被破坏，导致业务侧需要适配。 | 中 |
| vLLM fallback 逻辑的双重 `normal_text` 合并 | `residual_normal_text` 与 `info.content` 合并时的空白处理若不一致，可能出现意外空格或丢失。 | 低 |

**💡 关注建议**  

1. **文档同步**：在项目的 parser 文档（`PARSER_CASES.md`）中明确说明 Harmony 必须以 `<|call|>` 结束，且 `analysis` 部分为隐藏内容。  
2. **回退开关**：考虑加入环境变量或配置项 `harmony.strict_mode`，在需要兼容旧行为时可以关闭 `strip_harmony_protocol_from_normal_text`。  
3. **监控与告警**：在生产环境中，对 `tracing::warn!` 日志设定告警阈值，及时捕获意外的 “protocol content leaked” 警告。  
4. **兼容性测试**：在 CI 中加入对外部 SDK（如 OpenAI 官方客户端）与本库的兼容性验证，防止因协议细节变化导致集成方破坏。  
5. **性能回归**：虽然常规路径无显著变化，仍建议在性能基准测试中加入 “大批量 Harmony 调用” 场景，确保正则回退不在高 QPS 场景中成为瓶颈。

### fix(sglang): align multinode KV event worker ids (#9505)
**SHA**: `bf67d04` | 🔗 [查看提交](https://github.com/ai-dynamo/dynamo/commit/bf67d04c9d8fcd588e79c4542dda58a0b3f03deb)

**🎯 变更类型**：功能增强 / Bug 修复 / 架构变更  
**⚡ 重要程度**：🔴 高  

**📋 变更摘要**  
- 为 `EngineConfig` 增加 `runtime_data` 字段，并在 Python/Rust 绑定、后端注册流程以及测试中全链路传递。  
- 在 SGLang 的多节点部署场景下，引入 *worker group id*（`sglang_worker_group_id`）概念，用 `dist_init_addr` 统一生成并写入 `runtime_data`。  
- KV 事件发布路径改造：非 leader 节点在启动时通过 `runtime_data` 解析 leader worker id，随后在 `KvEventPublisher` 中以该 id 进行 KV 事件归属，从而实现跨节点的 KV 事件统一归档。  
- 新增 Discovery 查询端点命名规则 `worker_kv_indexer_query_dp{N}_worker{W}`，并在 `WorkerQueryClient` 中维护 “查询端点目录” 以支持 worker‑scoped 查询、实例切换、流取消/清理等操作。  
- 暴露 Python 客户端 `wait_for_instance_by_runtime_data` 方法，允许用户基于 `runtime_data` 键值定位唯一实例。  
- 依赖层面加入 `serde_json`，并对若干内部 API（如 `KvEventPublisher`）的签名做了向后兼容的可选化处理。  

**🎯 影响范围**  
- `components/src/dynamo/common/backend/engine.py` 与所有对应的 Python 绑定。  
- SGLang 相关模块：`_disagg.py`、`llm_engine.py`、`publisher.py`、`register.py`、`tests/*`。  
- 后端公共库（Rust）：`backend-common`（EngineConfig、ModelRuntimeConfig）、`kv-router`（查询端点、Transport、Indexer）。  
- Runtime / Pipeline 层：`lib/runtime/src/component.rs`、`pipeline/network/egress/addressed_router.rs`、`push_router.rs`。  

**🔍 技术洞察**  

- **架构影响**  
  - **统一的 runtime_data 机制**：`EngineConfig.runtime_data` 通过注册元数据把 SGLang 的 worker‑group 信息写入 MDC，形成从 **启动 → 注册 → 发现 → KV 事件归属** 的闭环。  
  - **查询端点分层**：新增 `worker_kv_indexer_query_endpoint_for_worker` 使查询流可以固定指向特定 logical worker，即使实际路由实例（instance_id）在不同物理节点上。`WorkerQueryEndpointDirectory` 实现了发现→解析→缓存的全局映射，保证 **每个 `(worker_id, dp_rank)` 只对应唯一的实例**。  
  - **实例生命周期管理**：在端点被删除或替换时，`WorkerQueryClient` 会主动 `cancel_instance_streams` 并 `clear_instance_tombstone`，避免残留流导致资源泄漏或错误的 KV 事件恢复。  
  - **API 向后兼容**：`KvEventPublisher` 的 `worker_id` 参数改为 `Option<WorkerId>`，默认采用 endpoint 本地 connection_id，兼容已有调用。  

- **性能影响**  
  - **额外的 discovery 查找**：启动阶段会对 `runtime_data` 进行一次解析，并在非 leader 节点执行一次 `wait_for_instance_by_runtime_data`（含一次网络轮询），对冷启动有轻微的延迟（毫秒级），对持续吞吐基本无影响。  
  - **查询路由成本**：在 `WorkerQueryTransport::query_worker` 中加入了对特定实例的直连（`AddressedRequest::for_instance`），避免了原来的 round‑robin 负载均衡，提升了恢复查询的稳定性与时延。  
  - **内存占用**：`WorkerQueryEndpointDirectory` 为每个 worker‑dp 维度维护 `DashMap< (WorkerId, DpRank), Instance>`，在大规模多节点（上千 worker）情况下占用可忽略的 O(N) 条目。  
  - **新增的流取消/清理**：在端点切换时触发一次 `cancel_instance_streams`，涉及一次内部 RPC，成本极低。  

- **安全考虑**  
  - **runtime_data 直接来源于 `dist_init_addr`**：仅在 SGLang 多节点内部使用，未对外暴露。`get_sglang_worker_group_id` 对解析异常进行容错并记录 warning，避免因恶意/错误地址导致 panic。

  - **超时与异常**：`wait_for_instance_by_runtime_data` 对超时返回 `PyTimeoutError`，防止无限阻塞。异常信息不泄露内部实现细节。  

**⚠️ 潜在风险**  

1. **发现延迟导致 KV 事件丢失**  
   - 非 leader 节点在等待 leader worker_id 超时（默认 5 s）后会直接跳过 KV 事件发布，可能导致该节点的 KV 事件不被记录。  
2. **注册元数据不一致**  
   - 若 `runtime_data` 在某些路径（如旧版 `EngineConfig` 实例）未正确填充，`get_sglang_worker_group_id` 返回 `None`，导致多节点之间缺失统一的 worker group id。  
3. **实例切换竞争**  
   - 多个 discovery 事件并发到达时，`WorkerQueryEndpointDirectory.insert` 可能出现“先前实例被覆盖”而未及时取消旧流，虽然已加入 `cancel_instance_streams`，但仍有概率出现短暂的双发请求。  
4. **向后兼容性**  
   - Python 端 `KvEventPublisher` 构造函数签名变更（`worker_id` 可选），若用户自定义实例化而未显式使用关键字参数，可能触发参数顺序错误。  
5. **依赖升级冲突**  
   - `serde_json` 被移动到核心依赖，可能影响自定义 Cargo workspace 中的版本锁定，导致编译冲突。

- `deploy/snapshot/protocol/restore_test.go`（测试套件）  
- 整个快照恢复子系统，间接影响 kubelet 对 Pod Startup / Readiness 的判断。

**💡 关注建议**  

1. **兼容性检查**：当前实现会强制把 `InitialDelaySeconds` 设为 0、`PeriodSeconds` 设为 1、`TimeoutSeconds` 设为 1，可能覆盖用户在 Liveness/Readiness 探测里自行配置的超时值。建议在注释或文档中明确说明此行为，或在实现里仅在“没有用户探测”时使用 sentinel‑probe，保留用户自定义的 `TimeoutSeconds`。  

2. **恢复窗口**：`FailureThreshold = math.MaxInt32` 确保恢复期间无限重试，但也会让 kubelet 长时间保持 “Starting” 状态。确认这不会导致资源调度或 QoS 评估异常。  

3. **测试覆盖**：新测试已去除对周期/超时的断言，确保未来对 probe 参数的微调不易导致回归。但仍应补充边界测试，例如：用户已有 `StartupProbe` 并明确设置了 `PeriodSeconds>1`，确保该值在深拷贝后被合理覆盖（或保持）。  

4. **文档同步**：更新 README 或内部设计文档中关于快照恢复时探测行为的描述，帮助运维在调试恢复慢启动问题时快速定位。  

5. **监控报警**：若恢复时间异常增长，kubelet 仍会以 “Starting” 状态报告，建议在监控面板加入 `container_startup_probe_failure_total` 的阈值告警，以便及时发现因 probe 失效导致的恢复阻塞。  

总体而言，此次修改提升了恢复阶段的快速感知能力，代码路径简洁且测试覆盖良好。只需注意上述兼容性与可观测性细节，即可安全合并。
