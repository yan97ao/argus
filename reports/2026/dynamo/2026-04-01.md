### feat: add least-loaded router (#6314)
**SHA**: `cc22114` | 🔗 [查看提交](https://github.com/ai-dynamo/dynamo/commit/cc22114d7825b2be78b2790373bd47582dbae46f)

**🎯 变更类型**：功能增强  
**⚡ 重要程度**：🔴高  

**📋 变更摘要**  
本次提交为 Dynamo 前端路由器新增 *least‑loaded*（最少负载）调度策略，并在运行时实现统一的请求占用计数器（`RoutingOccupancyState`）。相关改动覆盖 Python 绑定、CLI 参数、文档、路由枚举、运行时组件以及 PushRouter 实现，同时补充了大量单元测试以验证计数、并发分配与资源清理行为。

**🎯 影响范围**  
- `components/src/dynamo/frontend/*`（CLI 参数、文档）  
- `lib/bindings/python/*`（RouterMode 枚举、类型提示）  
- `lib/runtime/src/component/*`（共享占用状态实现）  
- `lib/runtime/src/distributed.rs`（运行时结构扩展）  
- `lib/runtime/src/pipeline/network/egress/push_router.rs`（路由逻辑、占用计数）  
- 相关单元测试文件  

**🔍 技术洞察**  

- **架构影响**  
  - **统一占用状态**：新增 `RoutingOccupancyState`（基于 `dashmap` + `AtomicU64`），由 `DistributedRuntime` 持有 `routing_occupancy_states` 映射，实现跨 `Endpoint` 的共享计数。  
  - **路由枚举扩展**：`RouterMode` 增加 `LeastLoaded`，并在 Python 绑定、C-FFI、`.pyi` 中同步。  
  - **PushRouter 重构**：新增 `OccupancyPermit` 与 `OccupancyTrackedStream`，在请求生命周期结束时自动 decrement。`select_next_worker` 与 `peek_next_worker` 在 `LeastLoaded` 与 `PowerOfTwoChoices` 场景返回 `None`，强制走带生命周期的 `generate`。  
  - **实例清理**：`Client` 在实例列表更新时自动调用 `state.retain`，清除已下线 worker 的计数，防止泄漏。  
  - **兼容性**：原有路由模式（round‑robin、random、kv、direct、power‑of‑two）保持不变，仅在配置中加入新选项，默认仍为 `round-robin`。  

- **性能影响**  
  - **计数成本**：每次请求在入口处 `increment`（一次 `fetch_add`）以及在流结束时 `decrement`（一次 `fetch_update`），均为无锁原子操作，性能开销极低。  
  - **选择成本**：`least-loaded` 通过 `select_exact_min_and_increment` 在一次 `Mutex`（`exact_selection_lock`）保护下扫描当前可用实例列表，时间复杂度 `O(N)`（N 为实例数），在实例数 ≤ dozens 的常规部署中可忽略。  
  - **潜在瓶颈**：在极大规模（上千实例）场景，`exact_selection_lock` 可能成为争用点；可考虑改为分段锁或基于 `AtomicU64` 的 `compare_and_swap` 实现。  
  - **内存占用**：`DashMap<u64, AtomicU64>` 为稀疏计数结构，仅在出现实例时才分配，整体内存消耗与实例数线性相关。  

- **安全考虑**  
  - 未引入外部 I/O、文件或网络交互，仅在内存中维护计数，未增加攻击面。  
  - `Arc<RoutingOccupancyState>` 与 `Weak` 的使用确保生命周期受控，不会出现悬空指针。  
  - 代码中已加入单元测试覆盖计数的原子性、并发分配以及下线清理，可降低因竞态导致的计数错误风险。  

**⚠️ 潜在风险**  

| 风险点 | 可能后果 | 缓解措施 |
|--------|----------|----------|
| `exact_selection_lock` 在极高并发下出现排队 | 请求延迟升高，路由响应时间波动 | 在后续版本优化为无锁最小值搜索或分段锁；监控 `occupancy_state.lock` 的等待时间。 |
| `RoutingOccupancyState.retain` 未及时触发导致旧实例计数残留 | 负载误判、某实例被错误地视为“最少负载” | 确保 `Client` 实例列表更新后立即调用 `retain`（已实现），并在关键路径加上日志/度量。 |
| `Arc` 循环引用导致内存泄漏 | 长期运行后内存逐渐增长 | 采用 `Weak` 存储在 `routing_occupancy_states` 中，已避免循环引用。 |
| 兼容性遗漏：旧版插件/脚本仍使用 `router_mode` 参数时未包含 `least-loaded` | 启动时报错或默认回退到 `round-robin` | 文档已更新；建议在发布说明中明确向后兼容处理。 |
| 测试覆盖不足的异常路径（例如 `generate_with_fault_detection` 失败后未释放计数） | 计数泄漏 | `OccupancyPermit` 通过 `armed` 标识，确保在异常返回前仍持有计数；需要在异常分支确认 `drop` 被触发（已通过测试验证）。 |

**💡 关注建议**  

1. **监控与度量**  
   - 在 `runtime` 中暴露 `routing_occupancy_state` 的计数指标（如 `dynamo_router_occupancy{instance}`），便于观察负载分布与锁争用。  
   - 对 `exact_selection_lock` 的等待时间添加 `Histogram`，提前发现瓶颈。  

2. **性能基准**  
   - 在典型集群规模（10、100、500 实例）下跑基准，比较 `least-loaded` 与 `power-of-two` 的延迟分布，确保新策略在大规模场景仍具竞争力。  

3. **代码审查点**  
   - 确认所有新增 `Arc<RoutingOccupancyState>` 的 `clone` 只在必要时进行，避免不必要的引用计数增长。  
   - 检查 `PushRouter::select_next_worker` 与 `peek_next_worker` 的 `panic!` 信息在未来可能被 `LeastLoaded` 调用时保持一致。  

4. **文档与用户指引**  
   - 在 Frontend Guide 中补充 “least‑loaded 与 power‑of‑two 在分散式 prefill 模式下的行为相同，都会回退至同步 prefill”。  
   - 更新 Python SDK 示例，展示如何通过 `RouterMode.LeastLoaded` 使用新路由。  

5. **后续扩展**  
   - 如果业务需要更细粒度的权重（如 CPU、GPU 利用率），可以在 `RoutingOccupancyState` 中加入多维度计数或自定义 `load_metric` 回调。  
   - 考虑在 `RoutingOccupancyState` 加入 TTL 清理机制，自动剔除长时间不活跃的实例计数，以进一步降低 stale data 的风险。  

---  
此变更为 Dynamo 提供了更直观的负载感知路由选项，架构上实现了统一的占用计数设施，整体风险可控，只需留意锁竞争与实例清理的细节即可安全上线。

### fix(backend): Reject conflicting model registrations on the same endpoint (#7686)
**SHA**: `21ba0c4` | 🔗 [查看提交](https://github.com/ai-dynamo/dynamo/commit/21ba0c4be37bd1fe41f93aeb360872fb965477d3)

**🎯 变更类型**：Bug修复  
**⚡ 重要程度**：🔴高  
**📋 变更摘要**：  
1. 在 `Discovery` trait 中加入默认实现的冲突检测逻辑，禁止在同一 `namespace/component/endpoint` 上注册不同的模型（除非是同一基模型的 LoRA 适配器）。  
2. 将各后端实现的原始 `register` 方法重命名为 `register_internal`，统一由 trait 的默认 `register` 包装调用。  
3. 新增模型标识抽取、兼容性判定以及冲突查找的辅助函数，并在注册失败时提供明确错误信息。  
4. 增加大量单元测试，验证同名模型、不同模型、不同 endpoint、LoRA 适配器等场景的行为。

**🎯 影响范围**：  
- `lib/runtime/src/discovery/mod.rs`（核心 Discovery trait）  
- `lib/runtime/src/discovery/kube.rs`、`kv_store.rs`、`mock.rs`（三种后端实现）  
- 依赖 `Discovery` trait 的所有业务模块（模型拉取、服务路由等）  
- 单元测试套件  

**🔍 技术洞察**：

- **架构影响**  
  - **统一注册流程**：通过 trait 的默认实现一次性完成冲突检查与实际注册，后端只需实现 `register_internal`，降低各实现的重复代码。  
  - **模型身份抽象**：引入 `ModelRegistrationIdentity`，把 “display_name / source_path / 是否 LoRA” 作为冲突判定的核心，提升可维护性与可扩展性（未来可以在此加入更多字段）。  
  - **向后兼容**：外部调用仍使用原来的 `register` 接口，编译层面兼容性保持，但自定义后端若未实现新的 `register_internal` 会编译失败，需要同步更新。

- **性能影响**  
  - **额外查询**：在注册前后各执行一次 `list`（`DiscoveryQuery::EndpointModels`）来扫描同一 endpoint 上的已有模型，时间复杂度 O(N)（N 为该 endpoint 已注册模型数）。在模型数量极大或注册频繁的场景下可能产生可感知的延迟。  
  - **一次性回滚**：若冲突在第二次检查时才被发现，会执行一次 `unregister`，增加一次写操作。整体开销仍在可接受范围（注册本身并非高频操作），但建议在高并发环境评估实际影响。

- **安全考虑**  
  - 本次改动未引入网络或文件系统的安全风险。相反，通过防止不同模型意外覆盖同一 endpoint，间接提升系统的可用性与一致性，降低因错误配置导致的服务不可用（DoS）风险。  
  - 错误信息中直接泄露了已有模型的 `display_name`，在多租户环境下需确认该信息不属于敏感数据。

**⚠️ 潜在风险**：

1. **并发竞态**  
   - 两个实例几乎同时调用 `register`，可能都在 `list` 检查阶段看不到对方的注册，导致冲突模型同时被写入。后续再次注册或查询会出现不一致状态。  
   - 解决方案：在底层存储（如 KVStore、Kubernetes CRD）实现原子“检查‑并‑写”事务，或在 `register` 中加入分布式锁。

2. **自定义后端兼容性**  
   - 外部实现如果仍实现旧的 `register` 而未迁移到 `register_internal`，编译会失败。需要在文档中明确迁移步骤。

3. **性能退化**  
   - 大规模模型库（上千个模型）在同一 endpoint 上注册时，列表遍历成本显著。若出现频繁的注册/注销循环，可能导致控制面（Discovery）负载升高。

4. **错误回滚路径**  
   - 当第二次冲突检测失败时，尝试 `unregister` 可能本身也出错，此时报错信息会被包装成新的 `anyhow`，原始错误可能被隐藏。应确保日志记录足够详细，或使用更细粒度的错误链。

**💡 关注建议**：

- **并发安全**：在生产环境启用后，监控 `register` 的错误率，特别是 “Cannot register model … a different model … is already registered” 的出现频次，评估是否需要在底层实现事务或乐观锁。  
- **性能基准**：在模型数目大于 500 时进行基准测试，确认列表查询的时延对整体注册延迟的影响，必要时可以在 `Discovery` 中缓存 endpoint‑>model 列表并使用版本校验来降低查询次数。  
- **文档与迁移**：更新 `README` 与 SDK 文档，说明新接口 `register_internal` 必须实现，并提供示例实现模板。对现有插件提供迁移补丁。  
- **错误可观测性**：在 `register` 的冲突检测分支加入结构化日志（例如 `model_name`, `endpoint`, `conflicting_name`），便于运维快速定位错误根因。  
- **扩展性**：如果以后需要支持多模型共存（如 A/B 测试），可以在 `ModelRegistrationIdentity::is_compatible_with` 中加入策略标记，而不是直接返回 `false`。当前实现已为此留有清晰的抽象点。

### feat: propagate OTEL trace context across SGLang multimodal E/P/D workers (#7580)
**SHA**: `8886d18` | 🔗 [查看提交](https://github.com/ai-dynamo/dynamo/commit/8886d18479124424d9fa601e899de9d51ccff3c4)

**🎯 变更类型**：功能增强  

**⚡ 重要程度**：🔴高  

**📋 变更摘要**  
在 SGLang 多模态工作流（Embedding、LLM Prefill/Decode/Diffusion、Image Diffusion、Video Generation）中引入 OpenTelemetry（OTEL）追踪头的统一生成与传播。新增 `build_trace_headers` 工具函数，统一在所有处理器里根据 `Context` 生成 W3C `traceparent`，并将对应的 `trace_id`（`rid`）随 SGLang 引擎调用一起传递，实现跨进程、跨 worker 的端到端可观测性。  

**🎯 影响范围**  
- `components/src/dynamo/sglang/request_handlers/*`（Embedding、LLM Prefill/Decode/Diffusion、Image Diffusion、Video Generation、Multimodal Worker）  
- `components/src/dynamo/sglang/request_handlers/handler_base.py`（添加 `enable_trace` 标记）  
- `dynamo/common/utils/otel_tracing.py`（提供 `build_trace_headers`）  
- 单元测试及文档同步  

**🔍 技术洞察**  

- **架构影响**  
  - **可观测性层面提升**：通过在 `Context` 中携带全链路 trace 信息，并在每一次对下游 SGLang 引擎的调用（`async_generate`、`async_encode` 等）注入 `external_trace_header` 与 `rid`，实现从入口 API 到各子 worker（Prefill、Decode、Embedding、Multimodal）的一致追踪。  
  - **配置统一**：在基类 `BaseWorkerHandler` 中读取 `config.server_args.enable_trace`，统一开启/关闭追踪，避免各子类自行实现。  
  - **最小侵入**：仅在调用 SGLang Engine 时额外传递两个可选参数，保持原有业务逻辑不变，兼容未开启 tracing 的场景。  

- **性能影响**  
  - **微小开销**：生成 `traceparent` 只涉及字符串拼接与一次字典创建（O(1)），对高并发请求的 CPU 消耗可忽略。  
  - **网络负荷**：额外的 HTTP/GRPC（或 NATS）头部会随每次 RPC 传输，典型大小 < 100 B，对带宽影响极小。  
  - **异步路径未受阻塞**：所有 trace 生成在同步代码路径完成，不影响底层 `async_generate` 的执行流水线。  

- **安全考虑**  
  - **信息泄露风险**：`trace_id` 与 `span_id` 在公开 API 之间传递，属于观测数据，一般不视作敏感信息，但在多租户环境下可能被用于追踪其他租户的请求路径。建议：  
    - 默认在生产环境开启 tracing 时对外暴露的 trace 信息做脱敏或限制访问（例如仅内部监控系统可查询）。  
    - 确认 `trace_id` 不被写入用户可见的返回体或日志（除调试级别外）。  
  - **授权传播**：当前实现仅依赖 `Context`，若未来引入跨集群传播，需要在传输层（如 NATS）确保加密或认证防止伪造 trace。  

**⚠️ 潜在风险**  

1. **遗漏 `enable_trace` 检查**：部分新调用添加了 `external_trace_header` 但未确保 `self.enable_trace` 为真，可能在未开启 tracing 时仍产生无意义头部。虽然影响不大，但会导致日志噪声。

## ⚠️ 潜在风险  

1. **依赖缺失**：`soundfile`、`librosa` 是新增的 runtime 依赖，若未在容器/环境中正确安装，音频编码/变速会在运行时抛异常并导致请求 500。

- Rust Python 绑定层 (`lib/bindings/python/rust/*`)  
- 错误定义 (`lib/runtime/src/error.rs`)  
- 文档与测试 (`docs/development/backend-guide.md`、pytest suite）  

**🔍 技术洞察**  

- **架构影响**：  
  - 将错误抽象上移到统一的 `BackendError` → `DynamoError` 体系，Python 层直接抛出对应异常，Rust 层通过 `py_exception_to_backend_error` 统一映射，形成“一站式”错误桥梁。  
  - 替换 `GeneratorExit` 为业务语义更清晰的 `EngineShutdown`，避免与 Python 生成器协议混用，引入的异常类型更贴合 Dynamo 的业务模型。  
  - 新增的 `Cancelled`、`StreamIncomplete` 等错误类型，使迁移（request‑migration）判定逻辑更细粒度，`is_migratable` 通过错误链判断可迁移性。  

- **性能影响**：  
  - 额外的异常类型注册与映射在错误路径才会触发，对正常请求路径几乎无影响。  
  - 在 `engine.rs` 中将原先的 `ResponseProcessingError::PythonException`、`PyGeneratorExit` 合并为单一 `Dynamo` 分支，简化匹配路径，略微提升错误分支的匹配效率。  
  - 通过 `py_exception_to_backend_error` 直接在 Python GIL 中完成映射，避免了多层次的错误包装，降低了内存分配次数。  

- **安全考虑**：  
  - 新异常映射层不泄露内部实现细节，仅将异常类别与简要信息返回给上层，符合最小信息泄露原则。  
  - 对所有未知 Python 异常统一映射为 `BackendError::Unknown`，防止异常种类膨胀导致的异常枚举不匹配风险。  

- **可维护性**：  
  - 通过宏 `define_dynamo_exceptions!` 自动生成异常类与注册代码，降低手工维护成本；新增错误只需在宏调用中添加一行即可。  
  - Python 端的异常导入统一在 `dynamo.llm.exceptions` 中完成，使用者可以一次性获取全套异常。  
  - 文档已同步更新，避免因旧文档误导导致的代码破坏。  

**⚠️ 潜在风险**  

1. **向后兼容性**：  
   - 现有用户仍可能捕获 `GeneratorExit`，而新实现不再抛出该异常，可能导致业务逻辑失效。需要在迁移指南中明确说明升级步骤。  
2. **异常捕获顺序**：  
   - Python 代码若捕获基类 `DynamoException` 而非具体子类，可能误将 `EngineShutdown` 当作普通错误处理，导致迁移逻辑失效。  
3. **错误链检测**：  
   - `is_migratable` 现在把 `Cancelled` 视为非迁移错误，若业务场景错误地把可迁移的取消视为不可迁，可能导致请求被错误丢弃。  
4. **宏生成的异常类名称冲突**：  
   - 若未来在 Rust 新增 `BackendError` 变体但忘记在宏中同步，可能导致 Python 侧缺失相应异常类，引发运行时 `AttributeError`。  
5. **测试覆盖不足**：  
   - 仅在少数路径添加了 `EngineShutdown` 捕获测试，其他异常（如 `InvalidArgument`、`StreamIncomplete`）未加入对应单元测试，可能隐藏映射错误。  

**💡 关注建议**  

- **升级指引**：在发布说明中强调 `GeneratorExit` 已被废弃，建议用户改为捕获 `EngineShutdown` 或更通用的 `DynamoException`。可提供兼容 shim（例如在 `exceptions.py` 中保留 `GeneratorExit = EngineShutdown`）供旧代码平滑迁移。  
- **异常捕获最佳实践**：文档中加入建议：*“在需要区分不同错误时请捕获具体子类；若只关心是否为 Dynamo 错误，捕获 `DynamoException` 即可。”*  
- **宏维护流程**：在 CI 中加入检测脚本，确保 `BackendError` 枚举所有变体均出现在 `define_dynamo_exceptions!` 调用中。  
- **扩大测试**：为每个新异常添加对应的 Python 抛出‑Rust 捕获回环测试，确保映射准确；在 `is_migratable` 的单元测试中分别验证 `Cancelled` 与 `EngineShutdown` 的迁移判定。  
- **性能监控**：在生产环境开启错误统计（如 `runtime::error_counter`），观察 `EngineShutdown`、`Cancelled` 的出现频率，以评估迁移机制的实际效果。  

通过上述改动，Dynamo 错误处理从“临时 hack”升级为结构化、跨语言统一的错误体系，提升了可维护性和可靠性，但升级过程需特别关注向后兼容与测试覆盖。
