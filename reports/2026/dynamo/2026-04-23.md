### fix(planner): restore documented Prometheus metric behaviour [DYN-2855] (#8575)
**SHA**: `dfde02c` | 🔗 [查看提交](https://github.com/ai-dynamo/dynamo/commit/dfde02c5b9f82427d16cb9ea8fd374f5615547de)

**🎯 变更类型**：Bug修复 / 性能优化 / 可观测性增强  
**⚡ 重要程度**：🔴高  
**📋 变更摘要**：  
1. 修复 Prometheus 指标行为，使 replica count 与累计 GPU 使用时长（gpu_hours）在每个 tick 中均得到正确上报，且即使 throughput‑scaling 被关闭或 Prometheus 端口为 0 也会继续累积。  
2. 通过 `_publish_inventory_and_gpu_hours` 将累计 GPU 时长的计算与时间差解耦，避免过去实现中只在 `throughput` 路径上统计导致数据缺失。  
3. 改进 `_report_diagnostics`，仅在对应的 scaling 路径（load / throughput）实际执行时写入枚举 gauge，防止相互覆盖。  
4. 细化 load‑scaling 逻辑，保留子决策的 “insufficient_data” 等真实原因，防止被 “no_change” 覆盖。  
5. 为诊断页面新增 “Cumulative GPU Hours” 行，并相应扩展测试覆盖。

---

**🎯 影响范围**  
- `components/src/dynamo/planner/core/base.py` – 核心调度循环、Prometheus 指标发布。  
- `components/src/dynamo/planner/core/load_scaling.py` – Load‑based scaling 决策诊断信息。  
- `components/src/dynamo/planner/monitoring/diagnostics_recorder.py` – HTML 报表的行数/高度调整。  
- 单元测试目录：新增 `test_metric_publication.py`、对 `test_easy_scaling.py` 的扩展。  

---

**🔍 技术洞察**  

| 维度 | 影响 |
|------|------|
| **架构影响** | - 将指标发布从 “throughput‑only” 移至 “每 tick” 的通用路径，解耦了 **调度业务逻辑** 与 **可观测性**。<br>- 引入 `_last_gpu_hours_update_ts` 维护累积时长的状态，属于 **Planner** 实例的生命周期状态；重启后会重新计数，这在业务上是可接受的。 |
| **性能影响** | - 额外的时间戳差值计算和 gauge `set` 调用，开销极低（O(1)）并在原有 `run` 循环中完成。<br>- 只在 Prometheus 启用时才触发网络写出，关闭时仍保留累计计算，基本不影响主流程。 |
| **安全考虑** | - 仅涉及内部监控数据，无外部输入参与，未引入安全风险。<br>- `prometheus_port` 为 0 时会跳过网络导出，避免在未授权环境暴露指标。 |
| **可维护性** | - 代码路径更清晰：指标发布与调度决策分离，后续若要替换监控后端（如 OpenTelemetry）只需修改 `_publish_inventory_and_gpu_hours` / `_report_diagnostics`。<br>- 新增单元测试覆盖关键分支，提升回归安全性。 |
| **业务价值** | - 累计 GPU 小时是计费与容量规划的重要指标，准确上报可直接提升客户的成本可视化与 SLA 报告。<br>- 保留子决策原因后，运维人员可快速定位 “insufficient_data” 等根本问题，降低故障排查时间。 |

---

**⚠️ 潜在风险**  

1. **时间戳回拨**：如果 `tick_input.now_s` 产生非递增值（如系统时间调整），`dt_s` 会变为负数，累计 GPU 时长可能出现倒退。当前实现使用 `max(0.0, now - last_ts)` 抑制负值，但仍会导致“停滞”期间的计时缺失。  
2. **状态不持久化**：`_cumulative_gpu_hours` 与 `_last_gpu_hours_update_ts` 在进程重启后会归零，若运维依赖跨重启的累计值，需要外部持久化或在 Prometheus 中使用 `increase()`。  
3. **并发安全**：Planner 本身在单线程 `asyncio` 循环中运行，当前实现未加锁；若未来引入多线程或并发调用 `_publish_inventory_and_gpu_hours`，需保证原子更新。  
4. **指标导出频率**：每 tick 均会 `set` gauge，若 tick 频率极高（例如 <100 ms）可能导致 Prometheus 抓取压力增加。建议在生产环境通过 `prometheus_port` 控制或在 `set` 前做限流。  

---

**💡 关注建议**  

1. **防止时间回拨影响**：  
   - 在 `_publish_inventory_and_gpu_hours` 中记录日志当 `dt_s == 0`（可能是回拨）以便监控。  
   - 如有需求，可使用系统单调时钟（`time.monotonic()`) 替代 `now_s`。  

2. **持久化累计 GPU 时长**：  
   - 若业务需要跨进程累计，可在 `Planner` 启动时从持久化存储（如 ConfigMap、etcd）读取上一次的 `gpu_hours` 与时间戳。  

3. **监控导出频率控制**：  
   - 添加可配置的 **metric_publish_interval**，在低频模式下只每 N 个 tick写一次 gauge，以降低 Prometheus 拉取负载。  

4. **单元测试与回归**：  
   - 当前已覆盖核心路径，建议在 CI 中加入 **时间回拨** 场景的测试，确保累计不出现负增长。  
   - 对 `prometheus_port` 为 `0` 时的日志输出做一次快照，防止未来误删 “if self.prometheus_port == 0: return”。  

5. **文档更新**：  
   - 在项目的监控章节补充 “Cumulative GPU Hours” 的解释、计量单位（GPU‑hours）以及累计方式（基于 wall‑clock delta）。  
   - 将 “load_scaling 诊断原因保留” 的行为说明写入 Planner 调度的设计文档，以免后续误删。  

--- 

**总结**：此次提交恢复并增强了 Prometheus 相关指标的完整性，尤其是累计 GPU 使用时长的持续追踪与 replica count 的每 tick 上报，显著提升了系统可观测性和运营成本透明度。风险点主要集中在时间戳管理和状态持久化上，建议通过上述措施进一步稳固实现。

### fix: add hardware discovery fallbacks and infer attempt (#8507)
**SHA**: `71fafe3` | 🔗 [查看提交](https://github.com/ai-dynamo/dynamo/commit/71fafe3fa69fb6974d2420322eae6c732341aa49)

**🎯 变更类型**：Bug修复 / 功能增强  
**⚡ 重要程度**：🔴高  
**📋 变更摘要**：  
- 在 `enrichHardwareFromDiscovery` 中加入对 GPU 型号的推断逻辑，若 DCGM 返回的 `system` 字段为空，先尝试通过 `gpu.InferHardwareSystem` 根据模型名称自动补全 SKU；否则回退到原始模型字符串。  
- 为此新增 `InferHardwareSystem` 的“裸模型” 推断分支（如 `NVIDIA H200` → `h200_sxm`），并补齐相应单元测试，验证正常化与回退行为。  

**🎯 影响范围**：  
- `deploy/operator/internal/controller/dynamographdeploymentrequest_controller.go`（硬件信息填充）  
- `deploy/operator/internal/gpu/discovery.go`（SKU 推断实现）  
- `deploy/operator/internal/gpu/discovery_test.go`、`controller/enrich_hardware_test.go`（单元测试）  
- 相关的 CI/CD 流水线（测试用例数量增加）  

**🔍 技术洞察**：  
- **架构影响**：  
  - 仅在 Operator 的控制器层面引入了业务逻辑补全，不改变 CRD、API 版本或调度框架。  
  - `GPUDiscovery` 仍保持原有接口，推断逻辑被封装在 `InferHardwareSystem`，对外部调用透明。  
- **性能影响**：  
  - 推断仅基于字符串匹配（正则/简单规则），耗时微乎其微，对控制器的 reconcile 循环无可感知的性能回退。  
  - 新增的单元测试会稍微延长 CI 运行时间，但对生产运行无影响。  
- **安全考虑**：  
  - 变更不涉及网络、权限或加密，仅处理元数据。  
  - 通过回退到原始模型名称避免因错误推断导致的配置非法错误，从安全角度看是更保守的做法。  

**⚠️ 潜在风险**：  
1. **误推断**：如果 `InferHardwareSystem` 的规则表未覆盖新发布的 GPU，或规则误判（例如误将 PCIe‑only GPU 当作 SXM），可能导致生成的 `GPUSKU` 与实际硬件不匹配，进而触发 profiling 作业失败。  
2. **向后兼容**：旧版 Operator 仍会使用原始模型字符串；升级后若外部系统（如自研调度器）依赖于原始字符串进行自定义处理，可能出现行为差异。  
3. **测试依赖**：单元测试使用了 `fake.NewClientBuilder`，若后续代码对 client 接口做了改动，测试可能失效，导致回归覆盖缺失。  

**💡 关注建议**：  
- **监控 & 日志**：在 `enrichHardwareFromDiscovery` 中加入 DEBUG/INFO 级别的日志，记录 “推断得到的 GPUSKU / 回退使用原始模型”，便于在生产环境快速定位异常。  
- **规则维护**：将 `InferHardwareSystem` 的匹配规则抽离为配置文件或 ConfigMap，方便在新 GPU 发布时无需重新编译即可热更新。  
- **回归测试**：在 CI 中保留现有的两类测试（裸模型正常化、未知 GPU 回退），并每次发布前对官方 NVIDIA GPU 支持矩阵进行一次同步校验。  
- **文档更新**：在 Operator 的 README/CHANGELOG 中标注 “DCGM 可能返回无 form‑factor 的模型名，Operator 会自动推断为 SXM SKU”，提醒使用者不必手动填补 `GPUSKU`。  

---  
此次提交主要通过 **推断 + 回退** 两步确保 `GPUSKU` 字段始终为 Dynamo Profiling 所能接受的枚举值，提升了系统的稳健性，风险主要集中在匹配规则的完整性和向后兼容性上。适当的监控、规则可配置化以及文档说明即可将风险降到可接受水平。

### fix(omni): fix disaggregated Qwen2.5-Omni serving pipeline (#8301)

- 相关单元测试文件（`test_omni_*`）以及 mock 实现。  

**🔍 技术洞察**  

| 维度 | 影响描述 |
|------|----------|
| **架构影响** | - 引入了 **`_build_engine_core_request_from_upstream`**，让 stage Worker 能在没有自定义 processor 的情况下复用原生 orchestrator 的 “tokens → EngineCoreRequest” 逻辑。<br>- 通过 `StageEngine.get_tokenizer` 为 `parse_omni_request` 提供可选的 tokenizer，保持与 `vllm‑omni serve --omni` 的统一行为。<br>- connector 写入顺序的调整，使得 **pipeline topology**（由 YAML 定义）成为决定是否走 connector 的唯一依据，提升 disaggregated 部署的可预测性。 |
| **性能影响** | - 新增的 async tokenizer 调用在大多数场景（聊天模板可用）只会执行一次，开销极小，且已使用 `await`，不会阻塞事件循环。<br>- `build_engine_core_request_from_tokens` 与手动注册 output processor 的额外函数调用成本可忽略。<br>- 通过提前检查 `connector` 存在并直接返回，可略微减少不必要的 SHM 写入路径。 |
| **安全考虑** | - 变更仅涉及内部数据结构的转换和异步 IO，没有引入外部资源访问或网络调用。<br>- 对上游输出结构的假设（`outputs[0].token_ids`）如果被恶意构造的对象破坏，可能触发 `RuntimeError` 并被捕获返回错误信息，不会导致服务崩溃。 |
