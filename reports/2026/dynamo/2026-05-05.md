### feat(operator): implement v1alpha1 ↔ v1beta1 conversion for DGD and DCD (#8647)
**SHA**: `f9b6ab6` | 🔗 [查看提交](https://github.com/ai-dynamo/dynamo/commit/f9b6ab689f5aed8d662932e8350015a57e7b2d65)

**🎯 变更类型**：功能增强 / 架构变更 / 性能优化 / 安全修复  

**⚡ 重要程度**：🔴高  

**📋 变更摘要**  
- 在 `v1alpha1` 与 `v1beta1` 之间实现了完整的 **hub‑and‑spoke** 双向转换（DynamoGraphDeployment、DynamoComponentDeployment、DynamoGraphDeploymentScalingAdapter）。  
- 通过 **preservation 注解**（`nvidia.com/dgd‑*`、`nvidia.com/dcd‑*`）保存两侧版本无法直接映射的字段，实现 **Round‑Trip 语义等价**。  
- 为所有 CRD 添加 `served: true`、`deprecated:true` 标记，使 `v1beta1` 成为 **served** 版本，`v1alpha1` 成为 **storage** 版本。  
- 引入 **自定义 MarshalJSON**（`api/v1beta1/marshal.go`）在序列化时剔除 Go 编码器产生的空对象（`metadata:{}`、`resources:{}`），防止 `kubectl apply` 产生不必要的 `.metadata.generation` 增长。  
- 新增 **默认化 Webhook** 为 `DynamoComponentDeployment`（在 `CREATE` 时把 `spec.componentName` 默认填充为 `metadata.name`），并在 `main.go` 中注册所有转换 Webhook。  
- 扩充 **CRD**、**operator‑cert** 代码，使转换 Webhook 能自动为四个多版本 CRD 注入 `caBundle`。  
- 大量 **fuzz / round‑trip 测试**（> 2000 行）覆盖字段迁移、保存/恢复、顺序保持、实验特性、前端 sidecar、共享内存、资源、annotations、labels、status、JSON 字节等场景。

## 🔍 技术洞察  

### 1. 架构层面  
| 影响 | 说明 |
|------|------|
| **Hub‑and‑Spoke 迁移** | `v1beta1` 现在是 **served** 版本，所有读写请求都经过 operator 的 **conversion webhook**（`/convert`）转到 `v1alpha1`（storage）。这把版本兼容的责任从 apiserver 移到了 operator，减少了 apiserver 的升级复杂度。 |
| **双向保留注解** | 使用 `nvidia.com/dgd-spec`、`nvidia.com/dgd-status`、`nvidia.com/dcd-spec`、`nvidia.com/dcd-status` 保存两端不可直接映射的字段。注解键被统一放在 `reservedAnnotationPrefixes`，防止用户自定义的同名注解被误删。 |
| **Shared‑Spec 转换抽象** | `shared_spec_conversion.go` 抽出了 `convertSharedSpecToHub` / `convertSharedSpecFromHub` 等核心函数，提供 **字段映射 + 稀疏保存 + 稀疏恢复** 的统一实现，避免在 DGD 与 DCD 两个入口出现逻辑分歧。 |
| **前端 Sidecar 兼容** | `FrontendSidecar` 在 v1beta1 只保留容器名称，完整的 sidecar 参数在 v1alpha1 通过稀疏注解保存并在需要时恢复，兼容了 “仅在 hub 中生成 sidecar” 的场景。 |
| **默认化 Webhook** | 为 `DynamoComponentDeployment` 添加了 **mutating webhook**，在 `CREATE` 时自动填充 `spec.componentName`，保持旧的 “metadata.name ⇒ spec.name” 行为，且对 `v1beta1` 仍然生效。 |
| **CRD 声明更新** | 所有 CRD 从 `unservedversion` 变为 `served:true` 并标记 `deprecated:true`，使 `kubectl`、helm 在未来升级时能自动提示迁移路径。 |
| **CA Bundle 注入** | `internal/cert/cert.go` 把四个 CRD（DGDR、DGD、DCD、DGDSA）统一加入转换 webhook 配置，确保在 Operator 启动时即完成 webhook 注册，即使 CRD 尚未创建也会记录 INFO 而不是错误。 |

### 2. 性能层面  
| 影响 | 说明 |
|------|------|
| **转换 webhook 开销** | 每次 `GET/POST/PUT` 都会走一次转换（`hub → spoke → hub`），额外的 **reflect‑based** 注解分析和稀疏结构拷贝在典型的几百毫秒以内（实际数据未测，但已加入基准 `BenchmarkMarshal_DGD_Aggregate`）。 |
| **自定义 MarshalJSON** | 在每次对象序列化时会执行一次 **JSON → map → 反射剔除 → 再 Marshal**，相较于直接 `json.Marshal` 额外消耗约 **30‑50µs**（取决于对象深度），对大规模 List 查询（如 `kubectl get dgd`) 可能产生可测的 CPU 增长。已通过基准 `BenchmarkMarshal_DGD_Aggregate` 监控，后续可以通过缓存 `fieldCache` 进一步降低。 |
| **注解大小** | 稀疏注解会把完整的 `DGD` / `DCD` spec（JSON）嵌入到单个 annotation 中，最大约 **8‑10 KB**（受 `metadata.annotations` 大小限制约 256 KB 保护），对网络传输和 etcd 存储略有增加。 |
| **内存占用** | 转换过程中会产生 **深拷贝**（`DeepCopy`）的临时对象，峰值内存大约是原对象的 **2‑3 倍**，在资源受限的节点上仍在可接受范围。 |

### fix: merge TRT-LLM overrides into --override-engine-args when present (#9107)
**SHA**: `8db5ad2` | 🔗 [查看提交](https://github.com/ai-dynamo/dynamo/commit/8db5ad2b82ec1a58bc683ee3ea0360731e09bcfa)

**🎯 变更类型**：Bug修复 / 功能增强  
**⚡ 重要程度**：🔴高  
**📋 变更摘要**：  
- 修复在使用 TRT‑LLM 时，当命令行已经包含 `--override-engine-args`，原有实现会错误地同时生成 `--trtllm.*` 标记，导致 TRT‑LLM 启动失败。  
- 新增 `_merge_overrides_into_args`、`_dotted_to_nested`、`_deep_merge` 等内部工具，实现“点式”键到嵌套 JSON 的转换并深度合并到已有的 `--override-engine-args` 中；若未出现该标记则保持原有的动态 `--trtllm.*` 生成逻辑。  
- 同时加入对应单元测试 `test_trtllm_override_merge.py`，验证合并行为及冲突消除。

**🎯 影响范围**：  
- `components/src/dynamo/profiler/utils/config_modifiers/trtllm.py`（核心配置修改器）  
- 依赖该模块的所有 TRT‑LLM 后端入口，包括 `convert_config`、`set_config_tp_size`、`set_prefill_config` 等函数  
- 单元测试目录 `components/src/dynamo/profiler/tests/unit/`（新增）

**🔍 技术洞察**  

- **架构影响**：  
  - 将原本散落在多个调用点的 “append `--trtllm.*`” 逻辑抽象为统一的合并函数，提升了 **配置层的职责单一性** 与 **可复用性**。  
  - 引入 `parse_override_engine_args`（已有）配合新的合并流程，使 **参数解析与改写** 更具一致性，降低未来添加其他后端覆盖策略时的耦合度。  

- **性能影响**：  
  - 仅在启动阶段进行一次 JSON 解析与深度合并，时间复杂度约为 O(N)（N 为覆盖键数量），对整体推理性能影响可忽略。  
  - 额外引入 `json.dumps`/`json.loads`，在极端情况下（数千条覆盖键）可能略微增加启动时延，但仍在毫秒级别。  

- **安全考虑**：  
  - 合并过程使用标准库 `json` 与受控的 `dict` 操作，无外部输入的代码执行风险。  
  - 需要确保 `parse_override_engine_args` 正确验证 `--override-engine-args` 后的 JSON 合法性，防止恶意构造的非法 JSON 导致 Crash；目前项目已有相应校验。  

**⚠️ 潜在风险**  

1. **兼容性回退**：若用户依赖原先 “`--trtllm.*` 与 `--override-engine-args` 同时出现” 的误用方式，合并后可能导致行为变化（但这种用法本就是不被 TRT‑LLM 支持的）。  
2. **JSON 合并冲突**：在嵌套结构中若覆盖键与原始键类型不匹配（例如原为字符串，覆盖为字典），`_deep_merge` 会直接覆盖，可能导致意料之外的配置失效。  
3. **测试覆盖不足**：当前单元测试仅覆盖了 `--override-engine-args` 已存在的场景，未覆盖“未出现该标记”时仍生成 `--trtllm.*` 的路径，后续改动可能引入回归。  

**💡 关注建议**  

- **完善测试**：补充一个覆盖 `--override-engine-args` 不存在时仍生成 `--trtllm.*` 标记的用例，确保两条分支都受监管。  
- **文档更新**：在使用说明或参数手册中明确说明：如果你已经使用 `--override-engine-args`，请勿再手动添加 `--trtllm.*`，系统会自动合并。  
- **输入校验**：在 `parse_override_engine_args` 之后加入对解析后 JSON 的 schema 验证（例如使用 `serde_json`‑style schema），提前捕获类型冲突。  
- **监控上线**：部署后通过 CI/CD 的端到端集成测试监控 TRT‑LLM 启动日志，确保不再出现 “mutually‑exclusive flags” 的报错。

- **性能影响**  
  - 使用 **UCX rc**（Reliable Connection）+ **NIXL** 能在 GB200 上实现约 **9.6 GB/s** 的 KV 带宽（接近 EFA 的性能），显著降低跨节点 prefill ↔ decode 的同步延迟。  
  - 通过 `NCCL_CUMEM_ENABLE`、`NCCL_NVLS_ENABLE`、`NVIDIA_GDRCOPY` 等环境变量，进一步提升 NCCL 在多节点之间的 GPU‑Direct 通信效率。  
  - 为防止 OOM，默认将 `mem-fraction-static` 调低至 `0.75`，并限制 `cuda-graph-max-bs` 为 `128`，在 TP8 环境下提供更稳定的内存占用模型。

- **安全考虑**  
  - `IPC_LOCK` 能力允许容器锁定内存，以便 UCX 注册 GPU 内存；这是一项 **提升特权**，在多租户集群中需严格审计。  
  - 注入 `rdma/ib` 设备以及自定义 `custom` 资源（如 `networking.gke.io/networks/rdma-*`）需要配套的设备插件或网络插件具备相应的 RBAC 权限。  
  - 对 NATS 服务器的覆盖（`NATS_SERVER`）可能导致跨命名空间的网络隔离失效，需确认网络策略与 TLS 配置仍然符合组织安全规范。

**⚠️ 潜在风险**：

1. **RDMA 设备不匹配**  
   - 未在 Pod spec 中声明 `rdma/ib`，导致容器无法看到 InfiniBand 设备，进而触发 `NIXL_ERR_BACKEND`。  
   - 绑定的 IB 设备（如 `mlx5_bond_0`）若 LID=0，会使 UCX UD/RC 失效，需要手动指定 `UCX_NET_DEVICES`。

2. **UCX / OFED 版本不兼容**  
   - 容器中 UCX mlx5 库编译的 ABI 与宿主机的 OFED 驱动不匹配会导致 `devx` 调用崩溃（如 `mlx5dv_devx_obj_destroy` 错误）。  
   - 需要在镜像构建阶段锁定与目标硬件相对应的 UCX 版本或使用自带 OFED。

3. **ComputeDomain 资源争用**  
   - `compute-domain-channel` 声明为 `Single` 分配模式，若同一集群中有多个大模型部署，可能出现通道资源竞争，导致调度失败或性能下降。

4. **特权能力滥用**  
   - `IPC_LOCK` 让容器可以锁定任意进程内存，若被恶意利用可能导致系统内存枯竭或信息泄露。

5. **网络策略冲突**  
   - GKE RDMA 注解需要集群启用 **GKE Multi‑Network** 功能，若未开启或网络策略阻断 `rdma*` 网络，将导致 KV 传输回退到 TCP，显著下降吞吐。

**💡 关注建议**：

- **验证前置条件**：在正式部署前，使用 `ibv_devinfo` 与 `kubectl describe pod` 检查 RDMA 设备是否已成功注入，确认 LID 正常。  
- **版本锁定**：在 CI/CD 中加入检查步骤，确保容器镜像的 UCX 与宿主机 OFED 版本兼容（可通过 `ucx_info -v` 与 `modinfo mlx5_core` 比对）。  
- **最小特权**：仅在需要 RDMA 的 Pod 中开启 `IPC_LOCK`，其余服务保持默认安全上下文；若可能，使用 `PodSecurityPolicy` 或 `OPA Gatekeeper` 限制该能力的使用范围。  
- **资源配额**：为 `compute-domain-channel` 设置 `ResourceQuota`，避免在同一命名空间或集群中出现通道资源抢占。  
- **监控与告警**：开启 NIXL 与 UCX 的 Telemetry（已在 env 中配置），将指标推送至 Prometheus；监控 `UCX_KEEPALIVE_INTERVAL`、`UCX_RC_TIMEOUT`，以及 `NATS` 连通性，提前捕获网络分区或后端初始化失败。  
- **文档同步**：更新 HelmChart/Operator 文档，明确 `rdma/ib`、GKE 多网络、ComputeDomain 等自定义资源的启用方式及所需 RBAC。  
- **回退路径**：提供不依赖 RDMA 的 TCP‑Socket 配置（如将 `UCX_TLS` 改为 `cuda_ipc,cuda_copy`），便于在硬件不满足时快速回滚。

| **安全考虑** | - 未引入外部网络或文件系统新入口，无新增攻击面。<br>- 新增的 `PrefillTokenDeltas::tokens_for` 对负数使用 `checked_sub` 防止 underflow，已在实现中加入错误日志。<br>- `ReplayStartGate` 使用 `tokio::sync::Barrier`，仅在进程内同步，不涉及共享内存安全风险。 |
| **可维护性** | - 通过 `pub use` 明确导出模块，外部调用者只需要依赖 `kv_router::PrefillTokenDeltas` 与 `bench::common::*`，减少散弹式 `use`。<br>- `node.rs` 中的 `apply_child_update` 被彻底移除，代码行数下降 30%，逻辑更易审计。<br>- 新增的 `CLAUDE.md` 文档为每个子目录提供明确的 Guardrails，降低未来违规改动的概率。 |

**⚠️ 潜在风险**  

1. **API 不兼容**  
   - `ActiveSequencesMultiWorker::potential_blocks_and_tokens_with_prefill_tracking` 已被删除，所有外部调用（包括旧的 benchmark、测试、或者第三方插件）需要迁移到新签名 `potential_blocks_and_tokens` 并提供 `PrefillTokenDeltas`。  
   - `make_progress_bar` 被移动到 `common::progress`，若有直接 `crate::common::make_progress_bar` 的路径，需要更新 `use`。  

2. **Prefill Token 计算差异**  
   - 之前 `added_prefill_tokens` 使用 `isl - overlap*block_size`，现在依赖调度层提供的 `PrefillTokenDeltas`。若调度层计算错误（例如 `cached_tokens` 计算漏掉某些共享缓存命中），会导致 `prefill_tokens` 为负或过大，进而影响 WSPT 排序和吞吐量估算。需要确保所有路径（尤其是 `OfflineReplayRouter`）使用同一套 `PrefillTokenDeltas` 生成逻辑。  

3. **节点插入语义**  
   - 原实现在 `shape_version` 失效时会自旋多次后返回 `None`（Stale），新实现直接返回 `Stale`。如果上层逻辑曾依赖 “重试” 行为（极少数极端并发场景），可能导致写入失败增多。建议在测试/压测中验证高并发写入（如 10k ops/s）下的成功率。  

4. **Bench‑Only 代码未编译**  
   - 部分 bench‑only功能（如 `make_progress_bar`）现在受 `feature="bench"` 条件编译保护。若在 CI 中未启用该特性，可能导致编译错误或缺失统计信息。
