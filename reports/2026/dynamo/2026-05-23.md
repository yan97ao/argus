### feat(tokenizers): add opt-in L1 prefix cache for tokenization (#9742)
**SHA**: `cba4560` | 🔗 [查看提交](https://github.com/ai-dynamo/dynamo/commit/cba456042a3c1cec12a1d779d333577b7130ef91)

**🎯 变更类型**：功能增强 / 架构变更 / 性能优化  

**⚡ 重要程度**：🔴高  

**📋 变更摘要**  
- 为 `tokenizers` 模块新增 **可选 L1 前缀缓存**，仅在特殊 token（`special: true`）边界上记录完整的前缀 tokenization，显著加速多轮对话中共享系统前缀的场景。  
- 通过环境变量 `DYN_TOKENIZER_CACHE=1` 和 `DYN_TOKENIZER_CACHE_BYTES` 控制缓存开启与内存上限，默认关闭，开启时几乎零额外开销。  
- 引入 `blake3`（高速哈希）与 `dashmap`（分片并发哈希表）实现线程安全、分片 LRU 缓存。  
- 新增 Prometheus 监控指标 `tokenizer_cache_hits_total`、`tokenizer_cache_misses_total`，并在 `ModelDeploymentCard::tokenizer` 中对后端（fastokens / HuggingFace / tiktoken）做统一包装。  
- 代码库 Cargo.toml 与 Cargo.lock 均加入新依赖，新增 `cache` 子模块及大量单元/集成测试，确保跨模型族（TinyLlama、Mock‑Llama‑3.1）的一致性与正确性。  

**🎯 影响范围**  
- `lib/tokenizers`（新增 `cache`、`l1.rs`、公开 `CachedTokenizer`、`L1CacheStats`）  
- `lib/llm/src/model_card.rs`（Tokenizer 加载路径改为可选缓存包装）  
- `lib/runtime/src/metrics`（前端性能指标扩充）  
- 依赖链：`blake3`, `dashmap` 被全局加入，影响所有编译目标（core、bindings）。  
- 受影响的二进制：`dynamo` 主程序、Python/KVBM 绑定、任何直接使用 `dynamo::tokenizers` 的第三方。  

**🔍 技术洞察**  

| 维度 | 影响 |
|------|------|
| **架构影响** | - 在 `Tokenizer` 接口之上增加 **装饰器层** `CachedTokenizer`，实现了 “包装者模式”。<br>- 通过 `Arc<dyn Tokenizer>` 维持对底层实现的抽象，使得 `fastokens`、`HuggingFace`、`tiktoken` 任意组合均可被统一缓存，无需修改各自实现。<br>- L1 缓存采用 **分片 DashMap**（16 shards）+ **Blake3 哈希**，实现 lock‑free 读/写路径，保持原有并发模型（Rayon）不受干扰。 |
| **性能影响** | - **热点场景**（多轮对话，共享系统提示）可在一次完整 tokenization 后，对后续请求仅重新 token 化 **后缀**，理论上将 **O(N)** 的 tokenization 降到 **O(suffix_len)**，在大多数聊天工作负载下提升 5‑10×+（取决于前缀长度与特殊 token 分布）。<br>- 哈希成本极低（Blake3 ~ 1‑2 ns/KB），并且缓存命中率通过 **特殊 token 边界** 保障不产生错误合并。<br>- 内存占用受 `DYN_TOKENIZER_CACHE_BYTES` 限制，默认 50 MiB，LRU 近似驱逐策略避免 OOM。<br>- 在缓存关闭时（默认）几乎 **零运行时开销**：`special_tokens.is_empty()` 直接短路，避免无意义的边界扫描与计数。 |
| **安全考虑** | - 代码未引入外部 I/O 或不可信数据的直接执行，仅在 **哈希** 与 **内存管理** 之间进行，仅使用 Rust 安全抽象。<br>- `blake3` 与 `dashmap` 均为成熟、审计通过的外部 crate，未发现已知安全漏洞（截至 2026‑05）。<br>- 通过 `Arc<AtomicU64>` 记录最近访问时间，避免出现 **数据竞争**。<br>- 对 `tiktoken` 分支未提取特殊 token，缓存仍会创建但永远 miss → 不会产生错误的合并。 |
| **可维护性** | - 新增的 `cache` 模块代码量约 800 行，单元/集成测试覆盖率 > 90%，提供了 **强保证**。<br>- 通过 `CacheEventFn` 把命中/未命中回调外置，指标埋点与业务代码解耦。<br>- `ModelDeploymentCard::tokenizer` 中的分支逻辑稍显臃肿，未来可考虑抽象为工厂函数，以降低维护成本。 |
| **兼容性** | - 默认关闭，所有现有用户行为保持不变。<br>- 对已有 env‑var `DYN_TOKENIZER` 的兼容未变。<br>- 对 `tiktoken` 的包装仍保留原有行为，只是多了一层透明的 `CachedTokenizer`，不会影响已使用的 `encode_batch` 并发实现。 |

**⚠️ 潜在风险**  

1. **内存泄漏 / 超限**  
   - 虽然有整体上限 (`max_memory`)，但每条 `insert_at_boundaries` 调用在 **批量** 插入时会先检查 `total_size_needed > max_memory` 并直接 **放弃**，这可能导致 **缓存永远不被填充**（当单条输入本身就超出上限）。如果用户误设极小 `CACHE_BYTES`，将导致全局 miss，仍然保持正确性，只是失去性能收益。  
2. **LRU 近似实现**  
   - 采用随机抽样的方式驱逐，极端情况下（大量小条目频繁命中同 shard）可能产生**热点失衡**，导致某些 shard 失去可用空间而整体命中率下降。  
3. **特殊 token 提取失误**  
   - 仅在 `HuggingFace` 路径提取 `special` 标记，其余后端（如 `fastokens`、`tiktoken`）若内部也拥有原子特殊 token 但未暴露，则缓存边界会少于理论上可能的边界，导致 **潜在的性能下降**（非功能错误）。  
4. **并发插入竞争**  
   - `insert_at_boundaries` 在大量线程并发写入同一 `L1Cache` 时，可能出现 **热点键冲突**（同一前缀被多线程同时插入），导致旧条目被频繁替换，计数上会出现轻微偏差，但不影响正确性。  
5. **Metric 注册遗漏**  
   - 新增的 Prometheus 指标在 `ensure_frontend_perf_metrics_registered` 中已注册，但若用户自行创建第二个 `MetricsRegistry` 而未调用该函数，可能导致监控缺失。  

**💡 关注建议**  

1. **文档 & 示例**  
   - 在 README / 环境变量说明中明确 `DYN_TOKENIZER_CACHE=1` 的开启方式、默认上限（50 MiB）以及调参建议（如对长对话 100 MiB 可提升命中率）。  
2. **监控告警**  
   - 基于新指标，建议在运营平台添加 **缓存命中率**（hits/(hits+misses)）告警，若长期低于 20% 则可能是特殊 token 列表错误或内存上限设置过低。  
3. **内存上限自适应**  
   - 考虑在未来版本引入 **动态伸缩**：当缓存命中率持续提升，可自动提升 `CACHE_BYTES`（最多受进程可用内存限制），反之缩小。  
4. **LRU 改进**  
   - 若实测出现热点 shard 失衡，可改为 **全局 LRU** 或 **分段计数 + 迁移**，或增大 `NUM_SHARDS`（通过 env‑var）以降低冲突概率。  
5. **特殊 token 抽取统一化**  
   - 为 `fastokens` 与 `tiktoken` 实现一个统一的 `Tokenizer::special_tokens()` 方法（或在 `CachedTokenizer::new` 中接受一个回调）以避免因后端缺失导致边界稀疏。

**⚡ 重要程度**：🔴高  

**📋 变更摘要**：  
1. 在 OpenAI HTTP 服务实现中，完善了对流式聊天响应的空块检测逻辑，新增对 `ChatCompletionMessageContent::Text("")` 与 `ChatCompletionMessageContent::Parts(Vec::new())` 两类“结构上为空”的内容的过滤。  
2. 引入 `ChatCompletionMessageContent` 类型，扩展 `is_empty_stream_response` 判空条件，并确保在出现 `logprobs`、`finish_reason` 等语义字段时不错误过滤。  
3. 添加覆盖上述场景的单元测试，验证空块过滤行为以及非空块的保留。  

**🎯 影响范围**：  
- `lib/llm/src/http/service/openai.rs`（OpenAI 兼容 HTTP 接口的流式返回处理）

### 🔍 技术洞察  

| 维度 | 影响描述 |
|------|----------|
| **架构影响** | 此次修改仅限于 **LLM HTTP 层** 的流式响应过滤逻辑，属于 **业务逻辑** 的细化，不会牵连到核心推理引擎或协议层。新增的 `ChatCompletionMessageContent` 引入，使得代码对多模态（文本 + 其它）内容的抽象更统一，提高了后续扩展（如图像、音频）时的兼容性。 |
| **性能影响** | 判空逻辑从单一 `content.is_none()` 扩展为匹配 `Text("")` 与 `Parts([])`，以及额外检查 `logprobs` 是否为 `None`。这些检查都是 **O(1)** 的模式匹配，运行时开销可以忽略不计；相反，过滤掉无意义的空块可减少网络 I/O 与下游消费者的处理量，整体上对吞吐量有微小正向提升。 |
| **安全考虑** | 变更仅涉及数据过滤，没有引入外部输入的解析或权限校验，安全风险极低。唯一需要关注的是 **误过滤** 可能导致合法的 “空消息” 被意外丢弃（例如用户明确发送空字符串以触发某些业务），但此类场景在 OpenAI 协议中几乎不存在，风险可接受。 |
| **可维护性** | 新增的 `ChatCompletionMessageContent` 导入以及更细致的判空分支，提高了代码可读性与自解释性。单元测试覆盖了四种关键路径（纯空文本、空多模态、含语义字段、含 logprobs），为后续改动提供了安全网。 |

### ⚠️ 潜在风险  

1. **误过滤合法空内容**：如果下游业务在未来需要接受显式的空文本（例如占位符或触发特殊逻辑），当前判空会直接抛弃，可能导致行为不一致。  
2. **多模态扩展兼容性**：当前只对 `Parts(Vec::new())` 进行过滤，若未来新增的多模态子类型（如 `Image`, `Audio`）在空状态下使用不同的表示方式，需要同步更新判空逻辑。  
3. **错误传播**：`is_empty_stream_response` 被用于过滤流块后直接 `continue`，若后续修改忘记更新此函数的调用点，可能导致空块仍被发送，破坏修复效果。

- **兼容性检查**：在发布新版本前，使用真实的 OpenAI 兼容后端（如 `gpt-4o-mini`）进行端到端流式对话测试，确保不会因过滤导致对话截断或意外结束。  
- **扩展判空**：为防止未来多模态类型遗漏，建议将判空实现抽象为 `impl IsEmpty for ChatCompletionMessageContent`，统一管理所有空态判定。  
- **监控与日志**：在 `is_empty_stream_response` 的分支中加入调试日志（可通过环境变量控制），便于在生产环境快速定位因空块过滤导致的异常情形。

### feat(llm): add nvext Tokens-in-Tokens-Out protocol (#9649)

**⚡ 重要程度**：🔴高  

**📋 变更摘要**  
- 在 OpenAI 接口路径上实现 “Tokens‑in‑Tokens‑Out”（nvext）协议，允许 **后端回传 token_id 列表** 与 **每个提示 token 的 logprob**。  
- 新增 **PRESERVE_OMITTED_MAX_TOKENS_CONTEXT_KEY**，用于在请求中显式保留 `max_output_tokens` 被省略的情况，以便后端依据实际 Prompt 长度动态计算生成上限。  
- 扩展 **Preprocessor**，引入 `PreprocessRequestOptions`、后台额外参数 (`backend_extra_args`) 以及对 `max_output_tokens` 省略的默认计算逻辑。  
- 增加 **NvExt** 与 **NvExtProvider** 的实现，支持 `cache_salt`、`detokenize`、`allowed_token_ids`、`bad_words_token_ids` 等 **PASSTHROUGH_EXTRA_FIELDS**，并在响应构造时可选地返回 `completion_token_ids` 与 `prompt_logprobs`。  
- 完善请求校验、序列化、聚合逻辑以及对应单元测试，确保新字段的安全校验与向后兼容。

**🎯 影响范围**  
- `lib/llm/src/http/service/openai.rs`（请求入口与上下文注入）  
- `lib/llm/src/preprocessor.rs`（预处理流水线、默认 max_token 计算、额外参数生成）  
- `lib/llm/src/protocols/common/llm_backend.rs`（新增 PromptLogprob 结构体）  
- `lib/llm/src/protocols/openai/*`（chat、completion 请求/响应、nvext 处理、验证器）  
- `lib/llm/src/protocols/unified.rs`（统一请求的 `prompt_logprobs` 透传）  
- 相关单元测试文件，覆盖新的字段、错误路径以及聚合行为。

**🔍 技术洞察**  

| 维度 | 影响描述 |
|------|----------|

| **可维护性** | - **代码结构**：控制器实现与其他 reconciler 保持相同的文件组织（`internal/controller`），使用公共 `predicate` 与 `runtimeConfig`，便于后续统一管理。<br>- **注解/常量**：统一放在 `consts`，并在图生成、Clique 生成、DCD 注解获取路径中统一处理，避免碎片化。<br>- **测试覆盖**：单元测试覆盖了正常复制、已经存在标签、未调度、节点缺失、无注解、非 Dynamo pod 等路径，提升回归安全性。 |
| **兼容性** | - 对已有部署 **向后兼容**：如果未在 DGD 中配置 `kvTransferPolicy.labelKey`，控制器不会执行任何操作，原有工作负载不受影响。<br>- **Alpha→Beta 转换**：在 `graph.go` 中明确删除旧的拓扑注解，只保留在 `DCD`/`Clique` 中的注解，防止重复或冲突。 |
| **监控/可观测性** | - 通过 `EventRecorder` 记录 `TopologyLabelMissing` 警告，运维可通过 `kubectl get events` 监控。<br>- `logger.Info` 打印复制成功日志，便于调试。 |

---

## ⚠️ 潜在风险
1. **RBAC 误授**：ClusterRole 赋予所有 Operator ServiceAccount `get nodes` 权限，若集群中存在多个 Operator 实例或其他组件复用同一 ServiceAccount，需确认该权限的最小化范围。  
2. **标签冲突**：如果 pod 已经拥有同名标签但值不同，控制器会 **不覆盖**（因为 `needsTopologyLabelCopy` 会返回 false），可能导致实际标签与节点不一致，需在业务层做好一致性检查。  
3. **延迟复制**：在节点调度后第一次 reconcile 完成之前，pod 可能已经启动并读取 Downward API，导致首次启动时缺失拓扑信息。虽然大多数容器会在第一次读取时即捕获更新，但极端场景下可能出现短暂不一致。可通过在 Pod 初始化容器中等待标签存在来缓解。  
4. **错误的 `labelKey` 配置**：如果用户误填不存在的节点标签键，所有相应 pod 将持续产生 Warning 事件，可能掩盖其它真警告。建议在 CR 验证 webhook 中加入 `labelKey` 是否在节点上存在的检查（可选），或在 UI/CLI 中提供预检查。  
5. **跨命名空间部署**：ClusterRoleBinding 绑定到 Operator 的 ServiceAccount（命名空间固定），但节点是集群级资源，若未来出现多命名空间的 Operator 实例，需要确保每个实例都拥有相同的 ClusterRole，否则部分实例会失去复制能力。  

---

## 💡 关注建议
1. **审计 RBAC**：确认只在需要的 Operator ServiceAccount 上绑定 `topology-label` ClusterRole，避免不必要的 cluster‑wide 权限泄露。  
2. **配置校验**：在 `DynamoGraphDeployment` 的 webhook 中加入对 `spec.experimental.kvTransferPolicy.labelKey` 的合法性检查（如正则匹配 `^[a-z0-9\.\-_/]+$`）以及非空检查。  
3. **启动顺序**：若业务对拓扑标签高度依赖，考虑在 worker pod 的 `initContainer` 中使用 `kubectl wait` 或 `sleep` 等方式等待标签出现，以确保首次读取到正确值。  
4. **监控告警**：将 `TopologyLabelMissing` 事件聚合到监控系统（Prometheus Alertmanager）中，及时发现节点标签缺失或配置错误。  
5. **文档同步**：在官方文档中说明：
   - 必须在 `DynamoGraphDeployment.experimental.kvTransferPolicy.labelKey` 中指定节点标签键。  
   - 此标签会自动复制到 pod，且可通过 Downward API 读取。  
   - 如不需要该功能，保持 `kvTransferPolicy` 为 `nil` 或 `labelKey` 为空即可。  
6. **回滚方案**：若出现异常行为，只需删除 `topology-label` ClusterRole/Binding 并把 `DynamoGraphDeployment` 中的 `kvTransferPolicy` 移除或 `labelKey` 设为空，即可让控制器不再进行任何复制。  

---  

**总体评估**：本次改动在 **功能、可观测性、可维护性** 方面都有明显提升，且对已有运行环境影响极小（仅在显式开启 `kvTransferPolicy` 时生效），属于 **高价值、低风险** 的功能增强。建议在下一个发布候选版本中开启，并配合相应的 RBAC 审计与文档更新。
