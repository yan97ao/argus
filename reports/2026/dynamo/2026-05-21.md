### feat: add AsyncShardHandle trait and generalize BranchShardedIndexer (#9763)
**SHA**: `c8f29d2` | 🔗 [查看提交](https://github.com/ai-dynamo/dynamo/commit/c8f29d2725cd3aa1f56c62d9ed0dbe5ce3d47b36)

**🎯 变更类型**：功能增强 / 架构变更 / 重构  
**⚡ 重要程度**：🔴高  

**📋 变更摘要**  
1. 新增 `AsyncShardHandle` trait，将分片后端抽象为统一的异步接口，使 `BranchShardedIndexer` 能够同时支持本进程的 `ThreadPoolIndexer<T>` 与跨进程的 Velo‑backed 客户端（`VeloShardClient`）。  
2. `BranchShardedIndexer` 改为泛型 `BranchShardedIndexer<S: AsyncShardHandle>`，并提供向后兼容的构造函数与类型别名。  
3. 迁移若干同步调用到异步实现（`spawn_blocking`、`JoinSet` 并发收集），并在单元测试、模块导入及序列化实现上做了相应调整。  

**🎯 影响范围**  
- `lib/kv-router/src/indexer/branch_sharded.rs`（核心路由分片实现）  
- 新增 `lib/kv-router/src/indexer/shard_handle.rs`（Trait 与 in‑process实现）  
- `mod.rs` 中引入 `shard_handle` 模块  
- `traits.rs`、`types.rs` 中结构体序列化派生（用于跨进程/持久化）  
- 相关单元测试文件的适配改动  

**🔍 技术洞察**  

- **架构影响**  
  - **抽象层提升**：`AsyncShardHandle` 将后端分片统一为异步接口，解耦了路由层与实际存储/计算实现，打开了以下两条重要路径：  
    1. **本进程多线程**（`ThreadPoolIndexer<T>`）保持不变，只是通过 trait 包装。  
    2. **跨进程/远程**（`VeloShardClient`），在 `features = ["velo-runtime"]` 下可直接替换，帮助 Dynamo 在多机器、多卡部署场景下使用统一代码路径。  
  - **向后兼容**：通过 `AnchorAwareBranchShardedIndexer<T>` 别名以及 `new_from_backends` / `new_with_options_from_backends` 构造函数，旧版基于 `Vec<T>` 的 API 仍可编译，降低迁移成本。  
  - **并发模型改进**：`flush`、`shard_sizes` 改为使用 `tokio::task::JoinSet` 并发执行，提升多分片情况下的总体吞吐与响应时间。  

- **性能影响**  
  - **读取路径**：`find_matches_from_anchor` 仍在本进程实现中通过 `spawn_blocking` 包装同步 trie 查询，避免阻塞异步运行时；在远程实现中可直接使用网络 RPC，无额外 `spawn_blocking` 开销。  
  - **写入路径**：写操作保持 fire‑and‑forget 语义，远程实现会在底层使用 Velo 的 active‑messaging，理论上不会比原来的同步通道慢太多，且可以通过网络并行化。  
  - **并行聚合**：`shard_sizes` 与 `flush` 使用 `JoinSet` 并行收集结果，放大了 CPU 利用率，尤其在 shard 数量（≥8）时可显著降低 latency。  
  - **潜在开销**：Trait 的动态分发已在编译时单态化（没有 `dyn`），但每个调用会产生一个返回 `impl Future` 的匿名类型，编译器会内联；因此运行时开销基本可忽略。  

- **安全考虑**  
  - 新增的远程分片路径会引入网络交互，需要确保 Velo 通道的鉴权、加密以及错误恢复机制已在 `VeloShardClient` 实现中完善。当前变更本身未加入安全检查，安全责任转移到 `VeloShardClient`。  
  - 所有公共结构体（`ShardSizeSnapshot`、`AnchorRef`、`AnchorTask`）加入了 `serde::{Serialize, Deserialize}` 派生，便利了持久化与跨进程序列化，但也要防止不受信任数据的反序列化漏洞（如使用 `bincode` 时需开启安全限制）。  

**⚠️ 潜在风险**  
1. **语义一致性**：`remove_worker_dp_rank` 必须只删除指定的 dp_rank，之前的实现误用了 `remove_worker`，新测试覆盖了该情形；仍需确保所有远程实现遵守相同语义。  
2. **写入顺序保证**：`enqueue_anchor` 文档要求“anchor 必须在后续 suffix 之前可见”。若远程实现未实现严格的顺序（例如异步发送后不等待 ack），可能导致路由错误。  
3. **并发错误传播**：`JoinSet` 中的任务若 panic，会被记录为警告而不导致整体错误返回，可能掩盖关键分片故障。生产环境建议监控并在必要时强制错误传播。  
4. **向后兼容性**：虽然提供了过时的构造函数，但已标记为 `#[deprecated]`，长久使用会产生编译警告，团队需要在后续迭代中迁移到 `BranchShardedIndexer::new`。  
5. **序列化兼容**：为 `ShardSizeSnapshot`、`AnchorRef`、`AnchorTask` 添加 `serde` 派生后，原有二进制持久化文件的格式会变化，需要版本迁移或保持旧版兼容层。  

**💡 关注建议**  
- **测试覆盖**：在开启 `velo-runtime` feature 时，加入针对 `VeloShardClient` 的集成测试，特别是 **anchor 顺序** 与 **remove_worker_dp_rank** 场景。  
- **监控 & 报警**：对 `ShardHandle::flush`、`shard_sizes`、`find_matches_from_anchor` 的错误计数与延迟做细粒度监控；若 `JoinSet` 中出现 panic，及时上报。  
- **安全审计**：审查 `VeloShardClient` 的网络实现，确保传输加密、身份校验以及防重放攻击；对所有通过 `serde` 反序列化的入口点使用白名单或版本校验。  
- **迁移计划**：在下一个主要版本中移除已废弃的构造函数与别名；提供迁移指南，示例代码展示如何显式创建 `ThreadPoolIndexer` 并调用 `BranchShardedIndexer::new`.  
- **性能基准**：在多节点实验环境下对比单机 `ThreadPoolIndexer` 与远程 `VeloShardClient` 的写入/读取延迟，评估 `spawn_blocking` 与网络 RPC 的占比，决定是否在未来把 trie 查找也迁移为完全异步实现。

### fix: filter empty stream chunks caused by multi-byte token assembly (#8036)
**SHA**: `5bff353` | 🔗 [查看提交](https://github.com/ai-dynamo/dynamo/commit/5bff35311517b0863549de00e1969810f853c6f5)

**🎯 变更类型**：Bug修复 / 性能优化  

**⚡ 重要程度**：🔴高  

**📋 变更摘要**  
1. 在 OpenAI HTTP 服务的流式响应路径中新增 `filter`，剔除因多字节 Token（如 Emoji）拼接产生的空 stream chunk。  
2. 实现 `is_empty_stream_response` 与 `is_empty_completion_stream_response` 两套检测逻辑，用于辨别“空块”。  
3. 补充单元测试，验证空块的判定规则在 ChatCompletion 与 Completion 两种返回结构下均符合预期。  

**🎯 影响范围**  
- `lib/llm/src/http/service/openai.rs`（流式 ChatCompletion 与 Completion 的实现）  
- 相关的事件转化、监控、SSE 推送链路  
- 测试模块 `lib/llm/src/http/service/openai.rs::tests`  

**🔍 技术洞察**  

- **架构影响**  
  - 该改动位于 **LLM → OpenAI 接口 → Streaming Pipeline** 的入口层，属于 **数据清洗** 步骤。  
  - 通过 `filter` 直接在 `Stream` 上拦截，避免空块继续进入 `EventConverter`、`MetricObserver`、SSE 输出等后续组件，降低了不必要的处理路径。  
  - 代码保持了现有的 `flat_map`、`map` 链式结构，未引入新的抽象层或全局状态，对整体架构兼容性良好。  

- **性能影响**  
  - 额外的过滤判断在每个流块上执行一次，主要是一次 `Option::as_ref` 与 `is_empty_*` 的结构体遍历，计算开销极低（O(1)）。  
  - 通过提前剔除空块，可显著减少后续 `process_response_using_event_converter_and_observe_metrics`、SSE 编码、日志记录等 **I/O 与序列化** 工作量，尤其在高吞吐、频繁出现多字节 Token（emoji、中文等）时可提升整体吞吐与延迟。  
  - 对单次请求的峰值延迟影响可忽略不计。  

- **安全考虑**  
  - 仅在业务层对响应进行过滤，不涉及身份验证、加密或资源访问控制，未引入新的安全威胁。  
  - 过滤逻辑基于响应内容本身，未改变协议字段或序列化方式，亦不影响已有的安全审计或追踪。  

**⚠️ 潜在风险**  

| 风险点 | 说明 | 可能的后果 |
|--------|------|------------|
| 判定规则不完整 | `is_empty_*` 只检查了当前已知字段（content、reasoning、tool_calls、usage、finish_reason、refusal 等），若未来后端引入新的可空字段或业务约定，空块可能被误判为有效块或相反。 | 产生额外的空 SSE 事件或误删合法的空内容（如某些工具调用需要显式发送空 Payload）。 |
| 与监控/指标的兼容性 | 过滤空块会导致 `EventConverter`、`MetricObserver` 收不到这些块，若监控原本统计“接收的块数量”，统计值会下降。 | 监控报警阈值需相应调整，避免误报。 |
| 客户端兼容性 | 部分自定义客户端可能依赖收到的每个块（即使为空）来维持流状态或做心跳检测。 | 客户端在收到间断的块时可能误判流已结束或出现超时。 |
| 未来协议变更 | 当 OpenAI 或兼容后端在空块中携带 meta 信息（如 token‑usage 增量），过滤会导致信息丢失。 | 用户侧统计不完整。 |

**💡 关注建议**  

1. **回归测试与监控**  
   - 将空块过滤计数加入监控（例如 `stream.empty_chunks_filtered`），确保在异常增多时可以快速定位。  
   - 在 CI 中加入对新增字段的判空回归测试，防止未来协议演进造成误删。  

2. **可配置开关**  
   - 考虑通过环境变量或运行时配置打开/关闭空块过滤（如 `DYN_FILTER_EMPTY_STREAM_CHUNKS`），为特殊场景（自研客户端、调试）提供回退路径。  

3. **文档与客户端说明**  
   - 在 SDK 与文档中明确说明服务器端已自动剔除空块，提醒使用者不要依赖空块进行流状态管理。  

4. **性能基准**  
   - 在高并发、emoji‑密集的对话场景下进行基准（benchmark），量化过滤前后的延迟与带宽节省，以便在 RELEASE NOTES 中给用户提供实测数据。  

5. **代码维护**  
   - 将 `is_empty_*` 的实现抽离为公共库（如 `dynamo_protocols::utils::empty_chunk`)），以便在其他服务（如 Anthropic、Mistral）复用，保持判定逻辑统一。  

---  

总体而言，此次提交通过轻量级过滤解决了多字节 Token 拼装导致的空 SSE chunk 泄漏问题，提升了流式响应的可靠性与资源利用率，风险可控且易于通过配置和监控进行管理。建议尽快合并并在下一个发布周期中正式提供。

### feat(sglang): OpenAI embeddings dimensions (DIS-2093) (#9722)
**SHA**: `92f5b3b` | 🔗 [查看提交](https://github.com/ai-dynamo/dynamo/commit/92f5b3b8d7dd5ab9179d4b1034bd2c1c0803693e)

**🎯 变更类型**：功能增强  
**⚡ 重要程度**：🔴高  
**📋 变更摘要**：在 SGLang 嵌入（embedding）请求处理器中新增 `dimensions` 参数，允许调用方对返回的向量进行“Matryoshka‑style”截断。实现了前置下界校验、在响应转换阶段的上界校验与截断，并在测试套件中加入对应用例。同时扩展了 payload 构造函数以支持传入额外字段。  

**🎯 影响范围**：  
- `components/src/dynamo/sglang/request_handlers/embedding/embedding_handler.py`（核心业务逻辑）  
- `tests/serve/test_sglang.py`（新增单元测试）  
- `tests/utils/payload_builder.py`（payload 构造函数）  
- 受 OpenAI 兼容层调用 `embeddings` 接口的所有服务实例  

**🔍 技术洞察**  

- **架构影响**  
  - 仅在 Python 层的 SGLang 嵌入处理器中引入了 `dimensions` 参数，未触及底层模型推理或 Rust 前端协议栈，保持了现有微服务边界不变。  
  - 通过在 `generate` 前置校验（下界）以及在 `_transform_response` 中进行上界校验，保证了业务规则在最早可能的阶段被拦截，符合“防御深度”设计。  
  - 代码注释已明确指出 `encoding_format=base64` 仍需后端 Rust 协议层支持，避免误导。  

- **性能影响**  
  - 前置的 `dimensions < 1` 检查与后置的切片 `embedding[:dimensions]` 均为 O(1)‑O(k)（k 为截断长度），对推理计算本身没有额外开销。  
  - 只在返回结果的 Python 端执行截断，避免了在模型推理阶段额外的张量切割，保持了原有的高吞吐量特性。  

- **安全考虑**  
  - 参数校验阻止了明显的非法请求（如 `dimensions=0`），降低了异常路径下潜在的资源浪费或服务崩溃风险。  
  - 由于 `dimensions` 只控制返回数据的长度，对内部模型权重或推理过程没有直接影响，未引入新的安全威胁。  

**⚠️ 潜在风险**  

1. **异常映射不一致**  
   - 当前在 `generate` 与 `_transform_response` 中直接抛出 `ValueError`，若上层 HTTP 框架未统一捕获并映射为 400 Bad Request，可能导致 500 Internal Server Error。  
2. **上界校验位置**  
   - 上界检查放在响应转换阶段，意味着在 `async_encode` 完成整个前向传播后才发现维度超出，仍会消耗一次完整的推理计算。对极端情况（如意外提交极大 `dimensions`）可能导致不必要的算力浪费。  
3. **兼容性**  
   - 新增 `dimensions` 参数对现有客户端是向后兼容的，但如果后端协议（Rust 前端）未来实现了 `encoding_format=base64`，需要同步更新此处的注释与实现，避免出现格式不匹配的错误。  

**💡 关注建议**  

- **异常统一**：在统一的请求入口层（如 `dynamo/sglang/api.py` 或对应的 FastAPI/Starlette 中间件）捕获 `ValueError` 并返回 400，确保错误信息对外可见且不泄漏内部实现细节。  
- **上界预检查**：如果模型的最大向量维度可以在运行时获取（例如通过模型元信息），建议在 `generate` 前加入上界预检查，以避免不必要的完整推理。  
- **文档同步**：更新 OpenAI 兼容协议文档，说明 `dimensions` 为可选字段，当前仅支持截断返回 `float[]`，不支持 `base64` 编码。  
- **性能基准**：添加微基准测试，验证在大批量嵌入请求并使用 `dimensions` 截断时的吞吐量与延迟，确保新逻辑不会引入不可感知的回归。  
- **未来扩展**：如果计划在 Rust 端直接支持 `encoding_format=base64`，应提前规划统一的协议结构（在 `lib/protocols` 中新增对应字段），以便一次性迁移而不是分散在多层实现中。

- **Grove 资源交互层**：`PodCliqueSet`（水平副本、缩放组副本）  
- **相关工具函数**：`orderLikeExisting`（未改变，仅配合新函数使用）

**🔍 技术洞察**  

- **架构影响**  
  - **保持状态一致性**：Grove `PodCliqueSet` 的副本数是通过 K8s `scale` 子资源动态调节的。此前在每次 DGD 重新渲染并同步时，会把 `PodCliqueSetSpec` 中的 `replicas` 直接覆盖，导致水平伸缩失效。新函数在 **渲染阶段** 只复制 *已有* 副本值到目标对象，从而实现 **“声明式渲染 + 状态保留”** 的混合模型，兼容了 Grove 自身的水平伸缩机制。  
  - **解耦 DGD 与 Grove 的副本控制**：原有实现把 DGD 的 `ComponentSpec.Replicas` 直接写入 PCS，违反了 **单一职责**（DGD 负责业务层面的副本意图，Grove 负责实际伸缩）。修复后，两者职责更清晰，提升整体可维护性。  

- **性能影响**  
  - **额外遍历成本**：函数对 `desired` 与 `existing` 中的 `Cliques` 与 `ScalingGroupConfigs` 各做一次 O(N) 的 map 构建和遍历。由于这些列表通常只有几到十几个条目（典型 DGD 部署），对控制平面的吞吐量影响可忽略不计。  
  - **避免不必要的 Update**：保留副本后，`SyncResource` 检测到 spec 没有变化的情况会跳过 `Patch`，反而能减少 API Server 的写入负载。  

- **安全考虑**  
  - **无直接安全风险**：修改仅涉及内部资源字段的复制，不涉及 RBAC、网络策略或数据泄露。  
  - **防御性检查**：函数对 `nil` 参数、空名称、缺失 `Replicas` 等情况做了容错判断，降低因异常对象导致 panic 的风险。  

**⚠️ 潜在风险**  
1. **副本不一致的隐式状态**——如果管理员手动修改 `PodCliqueSet` 副本数，而 DGD 中的副本数保持旧值，后续 DGD 重新渲染时仍会保留手动修改的值，可能导致 **意图漂移**（DGD 声称的副本数与实际运行数不匹配）。  
2. **缩放组配置名称冲突**——函数依赖 `ScalingGroupConfig.Name` 唯一且不变；若未来引入自动生成的名字或对名称做修改，可能导致复制失效。  
3. **测试覆盖不足**——当前单元测试覆盖了普通 Clique 与 ScalingGroup 两种路径，但未覆盖 **全部删除** 场景（如删除已有 Clique 后再次创建），可能出现残留副本值导致不期望的副本数。  

**💡 关注建议**  
- **文档与使用约定**：在运营手册中明确说明：**Grove 的水平伸缩应通过 `kubectl scale` 或 HorizontalPodAutoscaler 完成，DynamoGraphDeployment 中的 `replicas` 仅用于首次创建或“非自动调节”场景**。  
- **监控与审计**：在 operator 中加入对 `PodCliqueSet.Spec.Template.Cliques[*].Spec.Replicas` 与对应 DGD `ComponentSpec.Replicas` 的差异检测（可通过 Prometheus 指标），帮助运维快速发现意图漂移。  
- **增加回滚路径**：如果出现副本不一致导致业务异常，提供一个快速 “强制同步” 参数或注解，使 operator 能够覆盖现有副本数（如 `operator.ai-dynamo.io/force-replicas-sync: "true"`），以便手动纠正。  
- **扩展测试**：补充以下场景的测试：  
  1. **删除已有 Clique** → 确认不存在的 Clique 不会把旧副本数写回。  
  2. **ScalingGroup 名称变更** → 验证函数在名称不匹配时不会误复制。  
  3. **并发更新** → 使用 `fake.Client` 并发调用 `reconcileGrovePodCliqueSet`，确保不会产生竞争导致副本被错误覆盖。  

通过上述措施，可进一步提升此修复的可靠性，确保在大规模模型部署场景下，operator 与 Grove 的伸缩协同能够保持一致且可预测。
