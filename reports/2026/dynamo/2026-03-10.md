### feat: add an auto GPU VRAM estimator for disagg-same-GPU (#6868)
**SHA**: `423e4b4` | 🔗 [查看提交](https://github.com/ai-dynamo/dynamo/commit/423e4b45aaf1e72c2c85654c64036bf8e3671429)

**🎯 变更类型**：功能增强 / 性能优化 / 重构  
**⚡ 重要程度**：🔴 高  

**📋 变更摘要**  
- 在 `examples/backends/*/launch/disagg_same_gpu.sh` 中引入自动 GPU VRAM 估算逻辑，依据模型参数、上下文长度与并发请求数动态计算每个 worker 的显存占比。  
- 新增 `examples/common/gpu_utils.sh` 与 `examples/common/gpu_utils.md`，统一提供模型参数查询、显存估算、不同后端（vLLM、sglang、TensorRT‑LLM）对应的显存‑fraction 计算函数。  
- 支持通过 `DYN_GPU_MEMORY_FRACTION_OVERRIDE` 环境变量全局覆盖，且在单 GPU 场景自动将覆盖值均分给两个 worker。  

**🎯 影响范围**  
- `examples/backends/sglang/launch/disagg_same_gpu.sh`  
- `examples/backends/trtllm/launch/disagg_same_gpu.sh`  
- `examples/backends/vllm/launch/disagg_same_gpu.sh`  
- 新增 `examples/common/gpu_utils.sh`、`examples/common/gpu_utils.md`（所有后端的启动脚本均可 `source`）  
- 相关 CI / 文档（README 级别的示例）  

**🔍 技术洞察**  

| 维度 | 分析 |
|------|------|
| **架构影响** | - 将显存估算抽取为公共库 `gpu_utils.sh`，降低了各后端脚本的重复实现，提升可维护性。<br>- 通过统一的 `estimate_worker_vram` → `_EW_*` 变量 → `gpu_worker_fraction` 流程，使得不同后端的显存 fraction 计算能够在同一套模型参数下保持一致，避免了之前手动硬编码的 0.45/0.12 等经验值。<br>- 新的 `gpu_utils.sh` 依赖 `nvidia‑smi` 与 `python3`，在容器/裸机环境中仍保持轻量。 |
| **性能影响** | - **正向**：显存占用更精准（基于模型权重、KV cache 以及经验 overhead），可在相同 GPU 上跑更多并发请求或更长上下文，提升吞吐。<br>- **负向**：启动时额外执行 `estimate_worker_vram` (纯算数) 与一次 `nvidia‑smi` 查询，开销 < 10 ms，几乎可以忽略。<br>- 对 GPU 直接的运行时性能无影响，仅影响启动阶段的资源配置。 |
| **安全考虑** | - 脚本通过 `source` 方式执行，任何 **环境变量**（尤其 `DYN_GPU_MEMORY_FRACTION_OVERRIDE`、`OVERRIDE_ENGINE_ARGS`）均会直接写入进程命令行，若在不受信任的 CI 环境中使用，可能被注入恶意参数。<br>- 使用 `python3 -c` 读取显存信息，若系统中 `python` 被篡改或执行了恶意代码，可能导致信息泄漏或任意代码执行。<br>- 脚本对 `nvidia‑smi` 输出未做完整校验，理论上若 `nvidia‑smi` 被恶意包装，可能导致错误的 fraction 计算，进而触发 OOM。总体风险仍属低。 |
| **可维护性** | - 将模型参数硬编码集中到 `get_model_params`，新增模型只需在一个函数中补全，降低未来维护成本。<br>- 文档 `gpu_utils.md` 对每个后端的显存语义作了系统化说明，降低新手使用误解的概率。 |
| **兼容性** | - 仍保留对原有 `GPU_MEM_FRACTION` 参数的兼容（通过 env var 覆盖），旧有脚本在不 `source` 新工具的情况下仍能运行。<br>- 对于使用 `--kv-cache-memory-bytes`（vLLM）或手动指定 `free_gpu_memory_fraction` 的用户，新增的覆盖逻辑会被警告忽略，保持行为一致。 |

**⚠️ 潜在风险**  
1. **显存估算误差**：`estimate_worker_vram` 中的 overhead 采用经验公式，极端模型（如超大 MoE）可能出现显著偏差，导致 OOM。  
2. **环境变量误用**：`DYN_GPU_MEMORY_FRACTION_OVERRIDE` 在单 GPU 场景会被自动除以 2，若用户误以为是整体 fraction，可能导致显存配置不足。  
3. **外部依赖失效**：`nvidia‑smi`、`python3` 若不可用（如某些容器未装 NVIDIA 驱动或 Python），脚本会提前退出，影响 CI/自动化。  
4. **子脚本参数冲突**：`OVERRIDE_ARGS` 在 trtllm 脚本中使用 JSON 拼接，若用户自行追加 `--override-engine-args`，可能出现 JSON 合并错误。  

**💡 关注建议**  
- **测试覆盖**：在 CI 中加入多模型（0.6B、8B、30B）以及不同 GPU（24 GiB、48 GiB、80 GiB）的显存估算验证，确保误差在可接受范围（< 5 %）。  
- **警告机制**：当 `estimate_worker_vram` 计算出的总显存需求接近 GPU 上限（> 90 %），脚本应打印显式警告并建议用户调低 `MAX_…` 参数或使用 `DYN_GPU_MEMORY_FRACTION_OVERRIDE`。  
- **安全加固**：在入口脚本对 `python3`、`nvidia‑smi` 的路径做白名单校验（`command -v`），并在读取显存前验证返回值的整数范围。  
- **文档同步**：在项目根目录的 README/文档中注明 “显存估算仅适用于非量化模型，量化后请自行调整 `weight_bytes`”。  
- **参数显式化**：为避免除以 2 的隐式行为，可在帮助信息中专门说明 `DYN_GPU_MEMORY_FRACTION_OVERRIDE` 在单 GPU 模式下会被平分，并提供 `DYN_GPU_MEMORY_FRACTION_OVERRIDE_SINGLE=0.24` 之类的直通变量。  

---  

**总结**：本次提交通过统一的 GPU 显存估算库显著提升了 Dynamo 示例脚本在单 GPU 上的可配置性和可靠性，兼顾了不同后端的显存语义差异。风险点主要集中在参数估算的经验误差和外部依赖可用性，建议通过 CI 覆盖、警告提示及输入校验进一步降低风险。

### fix: handle early arrivals (#7126)
**SHA**: `b685291` | 🔗 [查看提交](https://github.com/ai-dynamo/dynamo/commit/b685291960550b4a7d6f1d21e471226f09f8b771)

**🎯 变更类型**：Bug修复 / 重构 / 性能优化  
**⚡ 重要程度**：🔴高  
**📋 变更摘要**：  
- 为 NIXL 通知的转移完成处理引入抽象层（`NotificationSource`、`Registration`），使核心循环可被单元测试并支持替换实现。  
- 增加对“提前到达”（notification 在转移注册前到来的）情形的检测与处理，防止因 race condition 导致的转移卡死。  
- 引入 `early_arrivals` 缓冲并在需要时触发即时完成，同时加入警告与可选的 debug panic。  

**🎯 影响范围**：  
- `kvbm-physical` crate（核心 NIXL 通知处理模块）  
- 依赖 `kvbm-physical` 的上层 Transfer、EventManager 代码路径  
- 测试套件（新增大量 `#[tokio::test]` 单元测试）  

**🔍 技术洞察**  

- **架构影响**  
  1. **解耦**：原本 `process_nixl_notification_events` 直接耦合 `NixlAgent` 与 `NotificationMap`，现在通过 `NotificationSource` 抽象，将外部 I/O（NIXL agent）与业务逻辑分离，极大提升可组合性与可测试性。  
  2. **泛型注册**：`Registration` trait 把 `RegisterNixlNotification` 与可能的未来注册消息统一抽象，允许在测试中使用轻量结构而不需要真实的 `XferRequest`。  
  3. **早期到达缓冲**：`early_arrivals: HashSet<Uuid>` 记录尚未出现对应注册的通知，配合 `process_events_core` 中的即时完成逻辑，消除了原有的 race 条件。  
  4. **模块化**：核心循环被拆分为 `collect_completed`、`complete_transfers`、`process_events_core` 三个纯函数，代码可读性、复用性提升。  

- **性能影响**  
  - **额外内存**：为每个未注册的提前通知维护一个 `HashSet<Uuid>`，在极端情况下（大量提前到达且未及时注册）会占用额外内存。  
  - **CPU 开销**：每次轮询后都会遍历一次 `outstanding`（HashMap）与 `early_arrivals`（HashSet），但这些操作的时间复杂度均为 O(1)（哈希查找），对整体吞吐影响可以忽略。  
  - **延迟改进**：提前到达现在可以立即完成，降低了最坏情况的转移完成延迟（原本需要等待下一轮轮询）。  

- **安全考虑**  
  - 没有引入 `unsafe`，所有新代码均保持 `Send + Sync` 约束，符合 async 环境的线程安全要求。  
  - `#[cfg(all(not(test), debug_assertions))] panic!` 在非测试的 debug 编译下会因提前到达而 panic，这是一种 **显式的开发阶段安全检查**，不会在 release 构建中触发。  
  - 新增的 `tokio = { workspace = true, features = ["test-util"] }` 仅用于测试，对生产运行时无影响。  

**⚠️ 潜在风险**  

1. **early_arrivals 缓冲无界增长**  
   - 如果系统出现异常大量提前通知（比如外部 NIXL 产生噪声），`early_arrivals` 可能持续增长，导致内存泄漏。  
2. **注册丢失导致警告泛滥**  
   - `warn!` 每 30 s 打印一次 `early_arrivals` 长度，如果该集合始终不被清除，日志可能被刷屏。  
3. **泛型实现的误用**  
   - `Registration::Payload` 必须在转移完成前保持有效性；如果自定义实现忘记保留必要资源（如 `XferRequest`），可能导致资源提前释放，产生隐藏的竞争。  
4. **调试 panic**  
   - 在非测试的 debug 环境中，提前到达会导致 panic。若生产环境误用了 debug 编译，可能在稀有 race 场景下导致服务崩溃。  

**💡 关注建议**  

- **资源清理**：在 `process_events_core` 中周期性清理 `early_arrivals`（例如超过 5 min 未匹配的 UUID 自动移除），防止无界增长。  
- **日志阈值**：可以将 `early_arrivals` 警告阈值配置化（如 > 100 条），或在生产中仅记录计数而不列出 UUID。  
- **文档与实现约束**：为 `Registration` trait 增加文档说明，明确 `Payload` 必须保持 “在通知到达前有效”。必要时提供 `#[must_use]` 或 `PhantomData` 编译时检查。  
- **CI 检测**：在 CI 中保留当前的单元测试，同时加入 **压力测试**，模拟大量提前到达的场景，验证 `early_arrivals` 的内存行为。  
- **调试模式控制**：如果团队希望在调试构建中避免 panic，可将该 `panic!` 包裹在可配置的 feature（如 `strict-early-arrival`），默认关闭。  

---  

总体而言，此次改动显著提升了 NIXL 通知路径的可靠性与可测试性，解决了原有的 race 条件 bug，并以最小的性能代价实现了更清晰的抽象。只要对 `early_arrivals` 的潜在增长进行适当的监控与限制，风险可控。

  - **ZMQ Listener 变更**：通过 `tokio::sync::watch::Receiver<bool>` 等待 `ready` 信号后才进入 `recv` 循环，确保恢复期间的事件不会被误删或重复。  

- **性能影响**  
  - **启动时额外 I/O**：单次 HTTP GET（默认 10 s 超时）和 1 秒的强制延迟，影响启动时延但仅在首次恢复时发生。  
  - **运行时开销**：`watch` 通过原子标志实现，几乎为零；`Peer` 列表仅在恢复阶段遍历，一般规模为几台，成本可忽略。  
  - **内存占用**：`dump` 包含完整事件序列，可能在大模型/高并发场景中占用数百 MB；建议在生产部署中使用 `--block-size` 限制树深度或压缩传输（未来可考虑）。  

- **安全考虑**  
  - **未加密的 HTTP**：恢复通过明文 HTTP 请求，同伴 URL 需要在可信内部网络中使用；若在不受信网络使用，应配合 TLS（`reqwest` 已支持）或启用 mTLS。  
  - **入站 peer 管理**：`/register_peer`、`/deregister_peer` 端点未做身份验证，潜在被恶意脚本添加恶意 URL，导致指数级的恢复尝试或信息泄露。建议加入基于 token/role 的访问控制或 IP 白名单。  
  - **Dump 内容可信度**：恢复时对 `block_size` 不做校验，只在日志中警告不匹配；如果攻击者伪造 dump，可能导致错误的 block 大小并影响后续查询。可考虑在 dump 中签名或加入一致性校验。  

**⚠️ 潜在风险**  

1. **恢复竞态**：若 `ready` 信号未及时发送（如恢复线程 panic），所有 ZMQ 监听器会永久卡在 `wait_for`，导致业务不可用。  
2. **不一致的块大小**：已有 indexer 已使用某 `block_size`，而从 peer 恢复时 `block_size` 不同，仅记录警告不阻止创建，可能导致后续 `apply_event` 失败或查询错误。  
3. **大规模 dump 导致 OOM**：在极端稠密缓存场景，完整 dump 可能超过可接受内存，导致恢复进程 OOM。  
4. **向后兼容性**：旧版 CLI 不支持 `--peers`，但 `WorkerRegistry::new` 仍创建 `ready` 通道，导致旧二进制启动后所有 ZMQ 监听器因未收到 `ready` 而卡死。  
5. **依赖升级风险**：新增 `reqwest 0.12` 可能带来 TLS/代理兼容性问题，需要在 CI 中验证各种平台。  

**💡 关注建议**  

- **验证与防护**  
  - 在 `/register_peer`、`/deregister_peer` 加入身份验证（如 JWT、API Key）或仅对本地回环地址开放。  
  - 为 `dump` 添加可选签名字段（例如使用 `ring`/`ed25519`），在恢复前校验完整性。  
  - 对 `block_size` 进行严格校验：若不匹配直接终止恢复并回退到空状态或提示用户手动干预。  

- **资源防护**  
  - 为 `recovery::recover_from_peers` 设置最大事件数阈值或流式解码，以防一次性加载大 dump 导致 OOM。  
  - 可在 CLI 提供 `--max-dump-size`、`--dump-compress`（gzip）等开关。  

- **可观测性**  
  - 在 `ready` 信号发送前后分别记录 `metrics::counter!("kv_indexer_ready", ...)`，帮助监控启动阶段的卡点。  
  - 暴露恢复时长和成功率的 Prometheus 指标（`kv_indexer_recovery_seconds`、`kv_indexer_recovery_success_total`）。  

- **兼容性与回滚**  
  - 为旧二进制保留默认 `ready` 为 `true`（即立即进入循环），或在 `main.rs` 检测 `--peers` 是否为空后自动 `signal_ready()`，防止老版本升级后卡死。  
  - 在发布说明中明确说明 “若使用 `--workers` 而不指定 `--peers`，请在启动后手动调用 `POST /signal_ready`（未来可实现）”。  

- **测试与持续集成**  
  - 保持现有的端到端测试（已覆盖两副本恢复），并新增单元测试：  
    - `registry::signal_ready` 在未发送前阻塞；  
    - `recovery::try_recover_from_peer` 对异常 HTTP 状态码、超时、无效 JSON 的鲁棒性。  
  - 在 CI 中加入安全审计（`cargo audit`）以及对 `reqwest` TLS 配置的 lint。  

通过上述措施，P2P 恢复功能将在提升故障容忍度的同时，保持启动可用性、资源安全和运维可控性。
