### feat(runtime): worker-side request admission (--engine-request-limit / --dynamo-request-queue-limit) (#10509)
**SHA**: `ff60ed8` | 🔗 [查看提交](https://github.com/ai-dynamo/dynamo/commit/ff60ed85ca1099acb793592fc0070b7583a2996b)

**🎯 变更类型**：功能增强 / 可靠性提升 / 架构变更  
**⚡ 重要程度**：🔴高  
**📋 变更摘要**：  
1. 在运行时配置中新增 `--engine-request-limit`（对应环境变量 `DYN_ENGINE_REQUEST_LIMIT`），实现 **Worker 端请求 admission**，在并发请求达到上限时直接返回 `503 ResourceExhausted`。  
2. 引入 overflow 队列大小 `DYN_DYNAMO_REQUEST_QUEUE_LIMIT`（默认 16），形成 **硬容量 N+Q** 的请求上限。  
3. 在网络入口/出口、调度与路由层面加入对超载的检测、标记与 Metrics（`dynamo_rejection_request_total`、`dynamo_engine_request`、`dynamo_request_queue`），并在前端将超载映射为 HTTP 503。  

**🎯 影响范围**：  
- `components/src/dynamo/common/configuration/groups/runtime_args.py`（配置解析）  
- `lib/runtime/src/pipeline/network/ingress/shared_tcp_endpoint.rs`（TCP 入口主逻辑）  
- `lib/runtime/src/pipeline/network/egress/addressed_router.rs`、`push_router.rs`（请求发送与错误处理）  
- `lib/llm/src/kv_router/prefill_router/execution.rs`（Prefill 错误链）  
- `lib/runtime/src/component/client.rs`（实例状态管理）  
- `lib/runtime/src/metrics/*`（新指标注册）  
- 文档 `docs/fault-tolerance/request-rejection.md`（使用说明）  

**🔍 技术洞察**  

- **架构影响**  
  - **新增并发调度层**：引入 `engine_sem`（`tokio::sync::Semaphore`）作为 **engine‑side 并发限制**，并在 `read_loop` 中实现 **直接调度**（无排队）与 **溢出队列调度** 两条路径。  
  - **容量分层**：`resolve_sizing()` 将 `DYN_ENGINE_REQUEST_LIMIT` 与 `DYN_DYNAMO_REQUEST_QUEUE_LIMIT` 解耦，保持对已有大容量默认（未设置时）兼容。  
  - **后端状态标记**：`RoutingInstances` 与 `Client` 增加 `mark_overloaded_immediate`，实现 **即时背压**（不走故障检测路径），前端路由会在下一个 metric 事件中重新评估。  
  - **错误链统一**：在 Prefill 与网络层统一将 “worker overload” 转为 `ErrorType::ResourceExhausted`，确保 HTTP 503 判定路径唯一。

- **性能影响**  
  - **正向**：在请求能够直接获得 `engine_sem` 时 **绕过** 工作队列 `mpsc`，降低调度延迟（尤其在低并发、队列为空的场景）。  
  - **负向**：引入 **额外的 semaphore acquire** 操作以及 `queue_capacity` 判断，极端高并发下可能出现轻微的竞争开销；但整体开销远低于一次 `send().await` 的排队路径。  
  - **监控**：三个新指标提供 **实时容量利用率**（`engine_request`、`request_queue`、`rejection_request_total`），帮助定位背压瓶颈。

- **安全考虑**  
  - 未引入外部可执行代码或网络协议改动，安全面风险极低。  
  - 唯一注意点：**错误返回的负载**（如 `"Server overloaded: …"`）在日志中直接打印，需确保不泄露内部实现细节或敏感配置信息。建议在生产环境日志中对该类消息做适度脱敏。

**⚠️ 潜在风险**  

| 风险 | 描述 | 可能导致的后果 |
|------|------|----------------|
| **配置不当** | `--engine-request-limit` 设为过小或 `DYN_DYNAMO_REQUEST_QUEUE_LIMIT` < 2 | 触发大量 503，服务可用性严重下降。 |
| **资源泄露** | `ActiveTaskGuard` 与 `ENGINE_REQUEST_GAUGE` 必须成对 inc/dec；若 future 在异常路径未 Drop，可能导致 gauge 偏移。 | 监控失真，误判系统是否已饱和。 |
| **竞态标记** | `mark_overloaded_immediate` 与 `set_overloaded_instances`（基于 metric）并发更新 `overloaded_ids`。若实现不具备原子性，可能出现 **短暂的错误路由**（请求仍被选中）。 | 产生间歇性 503 或请求错误转发。 |
| **兼容性** | 老版本的 worker 不读取 `DYN_ENGINE_REQUEST_LIMIT`，在混合部署时可能出现 **容量不一致**（部分 worker 仍使用大容量）。 | 部署滚动升级期间出现不均匀负载、异常拒绝。 |
| **测试覆盖不足** | 新的直调路径和 overflow 路径在单元/集成测试中覆盖有限。 | 隐蔽的死锁或 panic 在生产中被触发。 |

1. **上线前验收**  
   - 在预发布或金丝雀环境下，打开 `--engine-request-limit` 并配合 `--dynamo-request-queue-limit` 进行 **容量压测**，确保 `dynamo_engine_request`、`dynamo_request_queue` 与业务 SLA 匹配。  
   - 通过 `prometheus` 报表监控 `dynamo_rejection_request_total`，确认峰值在预期阈值内。

2. **配置治理**

   - 使用 `dynamoctl validate-config`（若已有）对运行时配置进行提前校验。

3. **日志与监控**  
   - 将 `Server overloaded:` 前缀的日志级别调至 **WARN**，并在日志聚合平台中设置告警，防止因误删导致的 “隐藏的 503”。  
   - 为新指标添加 **Dashboard**（例如 “Worker 并发使用率 & Overflow 队列深度”），帮助运维快速定位背压根因。

4. **回滚路径**  
   - 若在生产中观察到异常的 503 激增，可 **快速禁用** 环境变量 `DYN_ENGINE_REQUEST_LIMIT`（置空），系统会自动回退到原有大容量模式。确保 CI/CD 可在同一版本中切换。

### fix(llm): resolve eos/bos token ids from tokenizer_config and special_tokens_map (#10672)
**SHA**: `3a5a20d` | 🔗 [查看提交](https://github.com/ai-dynamo/dynamo/commit/3a5a20d50978dbc12c2758622a2ef6083b4fd822)

**🎯 变更类型**：Bug修复 / 兼容性提升  
**⚡ 重要程度**：🔴高  
**📋 变更摘要**：  
- 修正了 HuggingFace 模型在加载时对 BOS/EOS token id 的解析逻辑，加入对 `tokenizer_config.json` 与 `special_tokens_map.json` 的支持，形成四层回退链。  
- 新增一系列内部帮助函数（读取 JSON、提取 token 字符串、在多种文件中查找 token id）以及更详尽的错误信息。  
- 引入 3 条单元测试，覆盖仅靠 `tokenizer_config.json`、仅靠 `special_tokens_map.json` 两种极端模型布局以及全链路缺失的错误提示。

---

### 🎯 影响范围
- **`lib/llm/src/model_card.rs`**：核心模型配置解析实现。  
- **`lib/llm/tests`**：新增测试数据与测试用例。  
- **模型加载路径**：所有使用 `HFConfig::from_json_file` 的模型（包括自定义、社区、HF 官方模型）都会受此解析链影响。

---

### 🔍 技术洞察

| 维度 | 影响分析 |
|------|----------|
| **架构影响** | - 将 token id 解析抽象为“rungs”链式查找，提升了 **配置加载层的容错能力**，无需每个模型都复制 `bos/eos` 到 `generation_config.json`。<br>- 新增的 `resolve_token_id_from_tokenizer_files`、`resolve_token_id_from_special_tokens_map` 等函数，使代码职责更清晰，便于后续在不同 tokenizer 方案（如 per‑architecture tokenization）中继续扩展。 |
| **性能影响** | - 额外读取 `tokenizer_config.json`、`special_tokens_map.json`、`tokenizer.json`（最多 3 次 I/O），但仅在模型首次装载时执行，整体开销在毫秒级，可忽略。<br>- 使用 `serde_json::from_str` 解析，小文件体积（几 KB）对内存与 CPU 影响微乎其微。 |
| **安全考虑** | - 仍然是 **受信任文件**（模型权威发布），解析过程未引入外部网络请求。<br>- 通过 `anyhow::Context` 提供错误上下文，防止因缺失或格式错误导致 panic，提升 **稳健性**。<br>- 没有新增 `unsafe` 或外部依赖，安全风险保持低。 |
| **可维护性** | - 将复杂的“优先级链”抽离为独立函数，代码可读性大幅提升。<br>- 测试覆盖新增的四层解析路径，降低回归风险。<br>- 错误信息现在列出所有查找来源，运维人员可快速定位缺失的文件。 |

---

### ⚠️ 潜在风险
1. **文件路径假设**：代码使用 `file_path.parent().unwrap_or_else(|| Path::new(""))`，在极端情况下（如根目录文件）可能返回空路径，导致后续 `join` 产生相对路径，读取错误文件。  
2. **JSON 结构兼容**：`extract_token_string` 仅处理 `String` 或 `{"content": "..."} `两种形式；若未来 HF 改为其他结构（如带 `id` 字段的对象），解析会失败。  
3. **重复 ID 合并**：在 `final_eos_token_ids` 中手动 `push` 可能导致重复（虽然代码检查 `contains`），但若后续改为 `Vec` 之外的集合，需要确保去重逻辑同步。  
4. **错误信息泄露**：错误中会输出完整的文件路径，对内部使用者无碍，但对公开发行的二进制可能泄露本地文件系统结构，风险极低。

---

### 💡 关注建议
- **审计路径处理**：在极端路径（根目录或相对路径）下加一层防御，确保 `model_dir` 永远是合法目录。  
- **未来兼容**：在 `extract_token_string` 中加入对可能出现的其他字段（如 `token`、`id`) 的容错，或在注释中注明仅支持当前 HF 版本。  
- **性能基准**：可在 CI 中加入基准测试，确保在大模型目录（包含多余文件）下读取开销仍在可接受范围。  
- **日志级别**：`tracing::debug!` 已足够，生产环境默认关闭；若需要排查模型加载问题，可通过环境变量提升日志等级。  
- **文档更新**：在项目的模型加载说明中加入“BOS/EOS token 解析顺序”章节，帮助用户了解哪些文件是必须的，以及缺失时的错误提示。  

总体而言，此次修复显著提升了 **模型兼容性** 与 **错误可诊断性**，对整体架构的正向影响大于潜在风险，建议尽快合并到主分支并在下一个发布版本中同步。

**🔍 技术洞察**：  
- **架构影响**：  
  - 引入 `_served_model_name`（已在 `VllmLLMEngine.__init__` 中计算）作为统一的模型标识，避免在后续流程中出现列表‑字符串不一致。  
  - `EngineConfig` 的 `served_model_name` 字段从“可接受列表”转为“必为字符串”，对上层统一模型注册、路由与监控组件提供更明确的契约。

  - 额外的单元测试执行时间微增（≈ 0.1 s），不影响生产性能。  

- **安全考虑**：  
  - 变更不涉及外部输入的直接使用或网络交互，安全风险极低。

3. **文档同步**：`EngineConfig` 的类型声明或 API 文档需同步更新，否则使用者会产生误解。  

**💡 关注建议**：  
- **兼容层**：在 `EngineConfig` 读取端加入类型容错（如 `if isinstance(name, list): name = name[0]`），或在 `VllmLLMEngine` 中加入显式校验并抛出友好错误提示。  
- **版本发布说明**：在 changelog 中注明 `served_model_name` 归一化的行为变更，提醒使用者检查自定义代码。  
- **文档更新**：更新 Dynamo 开发者手册、代码注释以及 `EngineConfig` 的结构说明，明确字段类型为 `str`。  
- **回归测试**：在完整的集成测试套件中加入对旧版配置的兼容性检查，确保升级不会导致服务不可用。  
- **监控告警**：若模型注册依赖 `served_model_name` 进行统计或配额管理，确保监控规则已适配新的单字符串字段。

**🔍 技术洞察**  

| 维度 | 影响分析 |
|------|----------|
| **架构影响** | - 仅在 **监控线程**（单线程 async loop）内部使用 `OverloadedWorkerTracker`，不牵涉跨线程共享，保持原有架构不变。<br>- 通过 `collect_overloaded_workers` 将 overload 判定抽象为函数，提升代码可读性，便于后续统一阈值策略。 |
| **性能影响** | - **大幅降低** 每次 `ActiveLoad` 更新时的遍历成本：从 `O(N)`（遍历全部 worker）降至 **O(1)**（仅对本次更新的 worker 进行状态切换），在 worker 数量成百上千时尤为明显。<br>- 仅在阈值配置更改时才会重新遍历，全局锁 (`RwLock`) 的持有时间仍保持极低。<br>- 引入 `HashSet<u64>` 记录 overload ID，查询 / 插入 / 删除均为常数时间。 |

- **扩展性**：如果后续需要在多个监控线程间共享 overload 信息，可考虑将 `OverloadedWorkerTracker` 抽象为线程安全的共享结构（如 `Arc<RwLock<HashSet>>`），但目前保持单线程足够。  
- **文档更新**：在项目的监控章节说明 “incremental overload tracking” 的原理与使用场景，帮助新维护者快速理解为何不再每轮遍历全部 worker。

此改动在 **性能层面** 贡献显著，架构保持稳定，风险可控。建议在正式发布前进行 **负载压测**（数千 worker 场景）以及 **阈值热更新** 的端到端验证，确保全量 recompute 与增量更新在所有路径下都能保持正确的 overload 状态同步。

**⚠️ 潜在风险**  

1. **兼容性回滚风险**  
   - 老版 worker 在升级后若仍未提供 `worker_type`，仍会走 legacy shim，导致 **模型在列表中出现但实际不可用**（因为后端仍是 decode‑only）。运维需要确保所有 worker 全部迁移到带 `worker_type` 的新版。  
2. **请求路径性能**  
   - 在极端部署（数千个 `WorkerSet` / 模型）下，请求的就绪遍历仍是线性成本，可能在 10k QPS 场景出现微小 latency 峰值。需在后续机制 3 中实现“就绪实例池”。  
3. **错误体误用**  
   - 503 错误体现在客户端可能被误认为是网络故障，导致重试风暴。需要在 SDK/客户端层面做好 **幂等 / 重试策略**（只对 5xx 且非 503 重试）。  
4. **路由优先级冲突**  
   - `GET /v1/models/{model}/ready` 与模型名本身包含 `/ready` 的情况已通过“先 exact‑match 再子资源”处理，但若未来引入通配符路由或自定义前缀，仍需保持此判定顺序不被意外修改。  

**💡 关注建议**  

1. **发布前验证**  
   - 在预发环境使用 **滚动升级**，确保所有 worker 在升级后都能正确上报 `worker_type`，并检查 `GET /v1/models` 是否不再出现 decode‑only 只注册的模型。  
2. **监控告警**  
   - 基于新端点创建监控：`model_readiness.ready == false` 持续出现 → 触发告警，帮助运维快速定位缺失的 prefill/encode/decoder。  
   - 监控 503 响应率，区分是由于**服务未就绪**（process ready）还是**拓扑不完整**（业务可观测），避免重试风暴。  
3. **客户端适配**  
   - SDK/CLI 在收到 503+结构化错误时建议直接抛出 “模型未就绪”异常而非进行指数重试。  
4. **逐步移除 legacy shim**  
   - 在下一个主要版本（或通过内部里程碑）强制所有 worker 必须声明 `worker_type`，并在 CI 中加入校验，使 legacy 路径最终被删除，防止隐式兼容导致的误判。  
5. **后续机制 3 实施计划**  
   - 为避免高 QPS 场景的遍历成本，尽快实现 “就绪实例池” 并在路由层使用该池子进行 O(1) 调度，以保持系统在数千 QPS 规模下的低 latency。
