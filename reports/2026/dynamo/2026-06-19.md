### fix(trace): infer tool slices from request finish metadata (#10802)
**SHA**: `3985168` | 🔗 [查看提交](https://github.com/ai-dynamo/dynamo/commit/398516884fa3ca1547e20a6066c3da011a927e1c)

**🎯 变更类型**：Bug修复 / 功能增强  
**⚡ 重要程度**：🔴高  
**📋 变更摘要**：  
- 在 `benchmarks/request_trace/convert_to_perfetto.py` 中实现了基于 `request.finish_reason_metadata.tool_calls` 的 **推断工具切片**（inferred tool slices），当实际的工具事件缺失时仍能在轨迹图中展示工具调用的时间范围。  
- 将原来的 **session‑>pid** 映射简化为单一进程（pid=1），统一所有会话在同一进程下，降低元数据开销。  
- 新增大量单元测试以验证推断切片、跨会话进程统一以及原有工具切片的正确性。  

**🎯 影响范围**：  
- `benchmarks/request_trace/`（Python 转换脚本、README、验证脚本）  
- 依赖该转换脚本的 CI/性能基准流程、外部可视化工具（Perfetto、Chrome Trace Viewer）  

---

### 🔍 技术洞察

| 维度 | 影响描述 |
|------|----------|
| **架构影响** | 1. **进程抽象统一**：原先为每个 `session_id` 分配独立的 PID，现改为统一的 PID=1，所有会话共享同一进程名称 `Dynamo request trace`。这简化了 `TrackTable` 的内部结构，去除了 `_session_pids`。<br>2. **推断工具切片层**：新增 `_inferred_tool_calls` 与 `_prepare_inferred_tool_items` 两个函数，在 `convert_records` 中通过 `request_items` 与已有 `tool_records` 交叉匹配，生成 “inferred_tool” 记录并在后续渲染阶段统一处理。 |
| **性能影响** | - **内存/CPU**：去除每会话的 PID 映射表，`TrackTable` 只维护 `track_tid`，显著降低内部字典大小（尤其在高并发、长时间追踪的大量会话场景）。<br>- **转换速度**：新增一次遍历 `request_items` + `tool_records`（O(N)），但已在同一次 `convert_records` 调用中完成，整体耗时提升可忽略不计。 |
| **安全考虑** | - 仅涉及离线 trace 数据的后处理，未修改生产代码路径，未引入外部输入的执行或资源泄露风险。<br>- 推断逻辑默认对 `finish_reason_metadata.tool_calls` 进行严格类型检查（`isinstance(dict)` / `list`），避免恶意或异常结构导致异常抛出。 |
| **可维护性** | - 代码模块化：推断功能封装在独立函数，易于单独测试和后续演进。<br>- 通过新增测试用例覆盖多种边界（单请求、跨请求、重叠请求、无后续请求），提升回归安全性。<br>- 移除 `_session_pids` 相关代码，降低维护负担。 |
| **兼容性** | - **向后兼容**：将所有会话统一到单一 PID，可能影响使用 **pid** 区分会话的外部脚本或可视化过滤。<br>- **功能兼容**：已保留原有真实工具事件的渲染路径，仅在缺失时补充推断切片。 |

---

### ⚠️ 潜在风险

1. **外部依赖破坏**  
   - 任何基于 `pid` 区分会话的分析脚本（例如在 Perfetto 中按进程过滤）将不再能区分不同会话，可能需要适配新的 `thread_name`（轨迹）或自定义 `session_id` 字段。  
2. **推断误差**  
   - 依赖 `finish_reason_metadata.tool_calls` 的准确性；如果后端在异常情况下误报或漏报工具调用，会产生 **错误的时间跨度**（过长或过短），误导性能分析。  
3. **合并进程导致线程ID冲突**  
   - 虽然 `track_for` 仍返回唯一 `tid`，但在极端并发场景下（大量轨迹合并到同一 PID）可能导致 **tid** 分配顺序变化，进而影响可视化中轨迹的排列顺序。  
4. **测试覆盖不足的边缘情况**  
   - 目前仅在 benchmark 目录下验证，未在生产环境的 trace 导出流程中跑完整回归，可能遗漏极端 schema 变更（如 `tool_calls` 包含非标准字段）。  

---

### 💡 关注建议

1. **文档与迁移**  
   - 在 README 或官方文档明确说明 **“所有会话现在统一为 PID=1”**，并提供 **过滤会话的替代方案**（如通过 `session_id`、`trajectory_id` 或自定义 `args` 中的 `session_id` 字段）。  
2. **可选开关**  
   - 考虑在 `convert_to_perfetto.py` 增加 CLI 参数（如 `--separate-pids`），让用户在需要保留会话‑PID 映射时可以恢复旧行为，降低迁移风险。  
3. **推断逻辑容错**  
   - 为 `finish_reason_metadata.tool_calls` 添加 **日志/警告**，在结构异常或缺失关键字段时记录，以便后续排查。  
4. **回归测试**  
   - 在 CI 中加入真实生产 trace（含多会话、多工具调用）跑一次全链路验证，确保 **pid 合并** 与 **推断切片** 在真实数据下不产生意外的时间偏移或缺失。  
5. **监控性能**  
   - 在大规模基准运行时（如成千上万的请求）监控 `convert_records` 的执行时间和内存占用，确保 O(N) 的遍历不会因列表扩容导致峰值抖动。  

> **总体结论**：此次提交通过 **简化进程抽象** 与 **补全缺失工具切片** 两个维度提升了 trace 可视化的完整性与转换脚本的执行效率。风险主要集中在外部依赖的兼容性上，建议提供迁移指引或可选开关以平滑过渡。只要在文档中明确说明并在 CI 中覆盖上述场景，变更的价值大于潜在风险。

### fix(llm): accept generic trajectory identity (#10800)
**SHA**: `2c1bb16` | 🔗 [查看提交](https://github.com/ai-dynamo/dynamo/commit/2c1bb1696b6bdd98b22991b1b0c16520ae645eca)

**🎯 变更类型**：功能增强 / Bug 修复  
**⚡ 重要程度**：🔴 高  

**📋 变更摘要**  
本次提交将 `AgentContext` 中的 `session_type_id`、`session_id` 改为可选，并新增 `x-dynamo-trajectory-id` 头部支持，默认 `session_type_id = "dynamo"`。文档同步更新，所有相关序列化/反序列化、路由与追踪逻辑均已改写，以便在仅携带轨迹 ID（如普通 HTTP 客户端）时仍能产生完整的 `agent_context`。  

**🎯 影响范围**  
- `lib/llm/src/protocols/agents.rs`（头部解析与判定）  
- `lib/llm/src/protocols/common/extensions.rs`（`AgentContext` 结构体与 `From` 实现）  
- `lib/llm/src/request_trace/*`（全链路追踪）  
- `lib/llm/src/http/service/anthropic.rs`（请求构造）  
- 文档 `docs/agents/agent-tracing.md`、`docs/components/frontend/nvext.md`  
- 相关单元测试与示例  

**🔍 技术洞察**  

- **架构影响**  
  - **更松耦合的身份模型**：`AgentContext` 从 “强制两层（session+trajectory）” 变为 “trajectory‑only 也可接受”，使 Dynamo 能直接服务于不具备完整 agent 框架的通用 HTTP 客户端。  
  - **路由/调度层兼容性**：`has_agent_headers` 与 `apply_header_routing_overrides` 增加对 `x-dynamo-trajectory-id` 的检测，保持旧有专有头部优先级不变。  
  - **向后兼容**：所有原有必填字段仍保持 `Option` 包装，代码在使用前需显式 `as_ref()` / `as_deref()`，因此已有业务若未适配可能出现编译错误，提醒使用者升级。  

- **性能影响**  
  - 解析路径仅增加一次对 `HEADER_DYNAMO_TRAJECTORY_ID` 的查找，时间复杂度仍为 O(1)。  
  - 序列化时 `skip_serializing_if = "Option::is_none"` 让空字段不写入 JSON，略微降低网络负载。整体性能基本不变。  

- **安全考虑**  
  - 新增的通用头部 `x-dynamo-trajectory-id` 只负责标识轨迹，不包含用户可控的敏感信息，风险极低。  
  - 仍保留对 `session_type_id`、`session_id` 的非空校验（仅在提供时），防止恶意空字符串导致后端状态冲突。  
  - 需要确保在边缘代理或网关层对该头部进行适当的长度/字符校验，防止 Header Injection。  

**⚠️ 潜在风险**  

1. **编译/运行时破坏**：大量内部代码仍假设 `session_type_id`、`session_id` 为 `String`，若未迁移为 `Option`，会导致编译错误或 `unwrap()` panic。  
2. **业务逻辑偏差**：一些业务（如统计、计费）可能依赖 `session_id` 分组，使用仅轨迹 ID 时会导致报表碎片化。  
3. **兼容性隐患**：外部系统若仍发送旧的必填字段但值为空字符串，新的 `validate_non_empty` 仍会报错，需检查互操作方。  
4. **日志/追踪查询**：查询工具若基于 `session_id` 过滤，需要在 UI/SQL 中加入对 `NULL` 或缺失的情况的兼容。  

**💡 关注建议**  

- **代码迁移**：尽快在项目内部将所有对 `AgentContext.session_type_id`、`session_id` 的直接使用改为 `Option`‑aware（如 `if let Some(id) = &ctx.session_id { … }`），并补充相应单元测试。  
- **文档与 SDK 更新**：确保公开的 SDK（Python、JS 等）也同步把这两个字段设为可选，并在 README 中明确 “仅提供 `x-dynamo-trajectory-id` 时 Dynamo 将自动补全 `session_type_id = dynamo`”。  
- **监控与报警**：在追踪后端加入对缺失 `session_id` 的告警，帮助运营判断是否来自通用 HTTP 客户端。  
- **安全审计**：在网关层限制 `x-dynamo-trajectory-id` 的长度（推荐 ≤ 128）和字符集，防止异常请求导致后端资源耗尽。  
- **回滚策略**：若出现大规模查询错误，可通过环境变量禁用 `HEADER_DYNAMO_TRAJECTORY_ID` 的解析，回退到原有严格身份模型。  

---  

*总体而言，此次改动显著提升了 Dynamo 在跨语言、跨平台场景下的可接入性，风险主要集中在向后兼容的代码迁移和业务统计的适配上，建议在下一个版本发布前完成相应的适配和监控检查。*

### fix(llm): normalize coding agent headers (#10782)
**SHA**: `0b6cc91` | 🔗 [查看提交](https://github.com/ai-dynamo/dynamo/commit/0b6cc91c5fadf509af7a69cdd761ec0137d29f19)

**🎯 变更类型**：Bug修复（对编码‑agent Header 的统一与解析）  
**⚡ 重要程度**：🔴高  
**📋 变更摘要**：  
1. 新增 `agents` 协议模块，统一解析 Claude Code、Codex、OpenCode 等编码‑agent 在 HTTP 边界上传递的会话/轨迹信息。  
2. `apply_header_routing_overrides` 现在会在缺省时自动依据这些 Header 注入 `NvExt.agent_context`，并在已有显式 `agent_context` 时保持优先。  
3. `warn_nvext_disabled` 通过 `has_agent_headers` 检测到任意 agent Header 时触发一次警告，避免遗漏。  
4. 为上述改动补充单元测试及前端 CI‑smoke 测例，提升整体覆盖率。

**🎯 影响范围**：  
- `lib/llm/src/http/service/openai.rs`（警告函数）  
- `lib/llm/src/protocols.rs`（公开 `agents` 子模块）  
- `lib/llm/src/protocols/agents.rs`（新实现）  
- `lib/llm/src/protocols/common/extensions.rs`（Header 路由与 `NvExt` 合并逻辑）  
- 前端测试套件 `tests/frontend/*`（新增 agent smoke 与请求追踪）  

**🔍 技术洞察**  

| 维度 | 影响说明 |
|------|----------|
| **架构影响** | - 引入 `agents` 模块形成 **统一的 Header‑→ AgentContext** 映射层，解耦了业务代码（OpenAI Service）与具体 agent 实现。<br>- `NvExt` 结构体的 `agent_context` 字段现在可以由 HTTP Header 自动填充，降低了上层业务手动组装的负担。<br>- 警告路径 `warn_nvext_disabled` 增加 `has_agent_headers` 检测，使得在 **NVExt 失效** 时仍能捕获 agent Header 警告，提升可观察性。 |
| **性能影响** | - 解析过程只遍历常量数组 `AGENT_HEADER_MAPPINGS`（长度 3）并检查 `HeaderMap`，时间复杂度 O(1)（常量因子极小）。<br>- 额外的 `HeaderMap::contains_key` 调用对请求路径的整体延迟基本可忽略（微秒级）。 |
| **安全考虑** | - 新增的 Header 仅用于内部追踪 `session_id / trajectory_id`，不在响应体中回显，未引入信息泄露风险。<br>- `warn_nvext_disabled` 的日志级别为 `warn!`，仍需在生产环境审查日志脱敏策略，防止意外记录敏感会话 ID。 |
| **可维护性** | - 采用 **数据驱动的映射结构** (`AgentHeaderMapping`)，便于后续新增或修改 agent 类型。<br>- 通过 `From<AgentContextHeaderValues> for AgentContext` 实现转换，保持类型安全。<br>- 单元测试覆盖多种 Header 组合，降低回归风险。 |

**⚠️ 潜在风险**  

1. **兼容性风险** – 旧版客户端若发送自定义 Header（恰好与新定义的 Header 同名）可能触发 `warn_nvext_disabled` 警告，导致日志噪声或误判。  
2. **显式 `agent_context` 与 Header 冲突** – 虽已实现 “显式优先”，但若业务逻辑误以为 Header 能覆盖已填充的 `agent_context`，可能导致隐藏错误。  
3. **日志泄漏** – 警告日志中未过滤 Header 值，若生产环境开启 `WARN` 级别，可能记录会话 ID。  
4. **CI 运行时长** – 新增的 OpenCode、Claude‑subagent smoke 测试以及请求追踪会显著延长 CI（约 +250 s），需确保 CI 资源充足。  

**💡 关注建议**  

- **文档同步**：在项目的 HTTP 接口文档中加入 **coding‑agent Header 规范**（包括 `x-claude-code-session-id`、`x-claude-code-agent-id`、`session-id`、`x-session-id`、`x-parent-session-id`），并指明仅在 `DYN_ENABLE_FRONTEND_NVEXT` 开启时生效。  
- **日志脱敏**：在生产环境对 `warn_nvext_disabled` 进行统一过滤，确保 `session_id`、`trajectory_id` 不会写入对外可见的日志。  
- **回退开关**：提供环境变量（如 `DYN_DISABLE_AGENT_HEADER_PARSING`）让运维在极端兼容性需求时暂时关闭新解析逻辑。  
- **监控告警**：在监控系统中增加 “agent_header_detected” 计数器，帮助运维快速定位异常或意外的 Header 注入。  
- **CI 优化**：考虑将 OpenCode 与 Claude‑subagent smoke 测试设为 **可选**（`--run-agent-smoke`），在常规 PR 中默认跳过，仅在 nightly 或特定标签下执行，以控制总体 CI 时长。  
- **后续扩展**：若未来引入更多 coding‑agent（如 Gemini‑Code），只需在 `AGENT_HEADER_MAPPINGS` 中添加对应条目，无需改动核心路由逻辑。  

通过上述改动，Dynamo 在对编码‑agent 场景的 **Header 统一化、上下文追踪** 上实现了关键的可靠性提升，但仍需在日志管理、兼容性与 CI 资源方面做好配套措施。

### fix(kv-router): cancel in-flight recovery when worker is removed (#10616)
**SHA**: `8d94bbb` | 🔗 [查看提交](https://github.com/ai-dynamo/dynamo/commit/8d94bbb127eece199767730fd8153a42417bfb86)

**🎯 变更类型**：Bug修复  
**⚡ 重要程度**：🔴 高  
**📋 变更摘要**：在 `kv_router` 的恢复逻辑中引入了基于 `CancellationToken` 的 per‑rank 取消机制。现在当工作节点（worker）或其 DP‑rank 被移除时，所有正在进行的恢复任务会被立刻取消，避免在已下线的 rank 上继续重试轮询。此修改同时添加了两项单元测试，验证“移除后停止重试”以及“竞态产生的任务不会误查询”。  

**🎯 影响范围**：  
- `lib/llm/src/kv_router/indexer/recovery/worker_query.rs`（核心恢复模块）  
- 依赖此模块的 `kv_router`、`kv_indexer` 以及整体 Dynamo 服务的查询/恢复路径  
- 相关的测试套件 (`tests` 中新增两例)  

---

### 🔍 技术洞察

| 维度 | 影响分析 |
|------|----------|
| **架构影响** | - 引入 `DashMap<RecoveryKey, CancellationToken>` 作为全局的取消登记表，保持与已有的 `worker_states`、`recovery_semaphore` 同等的并发安全模型。<br>- 取消操作在 `remove_rank` 流程中完成，保证 **先撤销再销毁**，消除了之前的 “在 rank 已下线后仍会进行恢复重试” 的架构缺陷。<br>- 对外行为保持不变：对上层调用者仍然是 **异步恢复任务**，只是内部在 rank 被移除时会更早退出。 |
| **性能影响** | - **正向**：避免了对已下线 worker 的不必要网络请求和指数回退重试，降低了 CPU、网络以及目标服务的负载，尤其在集群规模大、频繁缩容的场景下收益明显。<br>- **负向**：每次 `spawn_recovery_task` 需要一次 `DashMap` 的读写以及一次 `CancellationToken` 的创建/克隆，开销极低（微秒级），对整体吞吐几乎无影响。 |
| **安全考虑** | - 该变更仅涉及内部任务取消逻辑，未暴露新的外部接口或修改协议，不会产生安全风险。<br>- 使用 `tokio_util::sync::CancellationToken`，该实现已在 Tokio 生态中得到广泛检验，可靠性高。 |
| **可维护性** | - 代码结构更清晰：取消逻辑与恢复任务的生命周期绑定，避免了隐藏的资源泄漏。<br>- 新增的单元测试覆盖了关键 race‑condition 场景，提升了后续改动的回归防护。 |
| **可靠性** | - 解决了之前 “rank 删除后仍会继续重试” 导致的 **无限重试** 与 **资源泄漏**（`recovery_cancels` 条目残留）问题，提升系统在缩容或失效恢复过程中的稳健性。 |

---

### ⚠️ 潜在风险

1. **CancellationToken 泄漏**  
   - 若在 `remove_rank` 前已经成功获取了 `cancel` 并随后因其它路径（例如异常提前退出）未调用 `cancel.cancel()`，对应的条目仍会在 `recovery_cancels` 中残留。当前实现仅在 `remove_rank` 时删除，理论上不会出现未被清理的情况，但若将来出现额外的 **手动撤销** 场景，需要确保每个 token 都能被移除。  
2. **竞争窗口细微**  
   - `spawn_recovery_task` 在注册 token 前先检查 `worker_states`，但在极端高并发下仍可能出现 “先检查后删除” 的极短窗口。即便如此，若窗口内生成了 token，随后 `remove_rank` 会立即取消并移除，任务在 `select!` 中会收到取消信号并安全退出，影响已被控制。  
3. **依赖版本兼容**  
   - 引入 `tokio_util::sync::CancellationToken` 需要确保项目的 `tokio-util` 版本兼容当前的 Tokio 运行时。若上游升级导致 API 变化，可能需要同步更新。  

---

### 💡 关注建议

1. **监控 & 告警**  
   - 在生产环境中添加 **“recovery_cancels 大小”** 的指标或日志，监控是否出现异常增长，及时捕获潜在泄漏。  
2. **代码审查要点**  
   - 新增的 `DashMap` 条目必须在 `remove_rank` 中配对 `cancel.cancel()` 并 `remove`，避免忘记清理。  
3. **文档更新**  
   - 在模块注释或 `README` 中补充“恢复任务取消机制”说明，帮助后续维护者了解 token 生命周期。  
4. **回归测试**  
   - 将新增的两个单元测试加入 CI，并在大规模集群缩容/扩容的集成测试中验证 “不会出现长时间的恢复请求”。  
5. **依赖审计**  
   - 定期检查 `tokio-util` 的安全报告，确保使用的版本没有已知漏洞。  

--- 

**总体结论**：此提交通过合理的并发控制与取消机制修复了关键的恢复任务泄漏与无效重试问题，提升了 Dynamo 在动态拓扑变化下的可靠性与资源利用率。风险可控，建议尽快合并到主分支并在 CI 中加入对应的监控点。

### fix(runtime): bound graceful shutdown drain (#10705)
**SHA**: `65d5e76` | 🔗 [查看提交](https://github.com/ai-dynamo/dynamo/commit/65d5e76275ad511f2dd5839f6a9ea70b60855514)

**🎯 变更类型**：Bug修复 / 功能增强（为运行时优雅关闭添加超时限制）  
**⚡ 重要程度**：🔴 高  
**📋 变更摘要**：  
- 为 `runtime.shutdown()` 引入 `DYN_RUNTIME_GRACEFUL_SHUTDOWN_TIMEOUT_SECS` 环境变量，默认 15 分钟，限制在等待所有标记为 `graceful_shutdown` 的端点完成时的最大阻塞时间。  
- 超时后记录剩余未完成的端点数量并强制继续运行时拆除，防止 Shutdown 卡死。  
- 同步更新文档、环境变量常量、单元测试，确保新行为受控且可验证。

**🎯 影响范围**：  
- `lib/runtime/src/runtime.rs`（核心关闭逻辑）  
- `lib/runtime/src/config/environment_names.rs`（新增环境变量常量）  
- 文档 `docs/fault-tolerance/graceful-shutdown.md`  
- 相关单元测试 `runtime.rs`  

**🔍 技术洞察**  

- **架构影响**  
  - 在原有的三阶段优雅关闭流程中，**第 2 阶段**（等待端点完成）由“无限阻塞”改为**受限等待**。  
  - 新增 `graceful_shutdown_timeout()` 辅助函数，将环境变量解析为 `Duration`，并在 `tokio::time::timeout` 包装 `tracker.wait_for_completion()`。  
  - 通过 `tracing::error!` 记录超时信息，保持对外行为不变（仍然会在第 3 阶段取消主 token），只是不再因单个端点卡死导致进程永久挂起。  
  - 对外暴露的环境变量遵循已有 `DYN_` 前缀约定，保持命名一致性。

- **性能影响**  
  - 引入 `tokio::time::timeout` 带来极小的调度开销（一次额外的定时器），对整体性能可忽略不计。  
  - 防止因端点长时间未结束导致的 **Shutdown 阶段阻塞**，提升系统在异常情况下的恢复速度和可观测性。  
  - 默认 15 分钟仍然相对宽松，避免对正常长请求（如大文件上传）产生过早中断；若业务需要更短/更长可通过 env 调整。

- **安全考虑**  
  - 通过强制超时避免 **Denial‑of‑Service** 场景：恶意或 buggy 端点若不响应关闭请求，会导致整个服务进程无法退出，进而阻塞部署/滚动更新，引发资源占用。  
  - 只在日志层面记录剩余端点数量，不泄露敏感业务数据。  
  - 环境变量解析采用 `parse::<u64>()`，若值非法则回退到安全的默认值，避免因配置错误导致 panic。

**⚠️ 潜在风险**  

1. **误配置导致请求被截断**  
   - 若运维将超时时间设得过短（如几秒），仍在处理中的请求将被强制中断，可能导致业务错误或数据不一致。  
2. **日志噪声**  
   - 在高并发或频繁部署的环境中，超时可能频繁出现，导致大量 `error!` 日志，埋没真实故障信息。  
3. **测试覆盖局限**  
   - 当前仅验证了 “等待超时后主 token 被取消” 场景，未覆盖 **超时前正常完成**、**空 tracker**、以及 **环境变量解析异常**（如负数、非数字） 的行为。  
4. **跨平台时间精度**  
   - `tokio::time::timeout` 依赖系统计时器，极端情况下（系统时间跳变）可能提前触发，虽然概率极低，但在高度同步要求的环境中值得关注。
