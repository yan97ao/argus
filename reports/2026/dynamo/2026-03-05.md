### fix: revert accidental change of trtllm/multimodal_processor (#6921)
**SHA**: `9c6f36e` | 🔗 [查看提交](https://github.com/ai-dynamo/dynamo/commit/9c6f36e9a9a18976ef18f8b9e7e1ead3bdb4f12e)

**🎯 变更类型**：Bug修复 / 功能增强  
**⚡ 重要程度**：🔴高  
**📋 变更摘要**：  
1. 纠正 `trtllm/multimodal_processor.py` 中因误提交导致的多模态输入处理逻辑回退，恢复对 NVIDIA‑TRT‑LLM 多模态嵌入的正确传递。  
2. 引入 `asyncio.to_thread` 将 `default_multimodal_input_loader` 的同步图片下载与预处理搬到线程池，提升高并发请求下的吞吐。  
3. 调整 E2E 测试超时时间，从 150 s 改为 300 s，防止因新异步路径导致的偶发性超时失败。  

**🎯 影响范围**：  
- `components/src/dynamo/trtllm/multimodal_processor.py`（核心多模态请求处理）  
- 依赖此处理器的 TRT‑LLM 生成路径（包括 EPD 与 PD 两种流）  
- `tests/router/test_router_e2e_with_trtllm.py`（TRT‑LLM KV 路由端到端测试）  

**🔍 技术洞察**：  
- **架构影响**：  
  - 将原本完全同步的 `default_multimodal_input_loader` 包装为异步调用，保持 `process_openai_request` 的整体 `async` 接口不变，只是内部通过线程池并行化 I/O 与图像预处理。  
  - 重新组织了对嵌入（embeddings）与嵌入文件路径的处理逻辑，使两种情况（NIXL 直接传递 & PD 流加载本地/远端文件）统一走同一分支，降低代码分支复杂度。  
- **性能影响**：  
  - 在高并发场景下，图片下载与预处理不再阻塞事件循环，可实现 *请求级* 并行，预期在 GPU‑bound 推理阶段前的 CPU‑bound 开销下降 20%~40%（视图像大小与网络条件而定）。  
  - 引入 `asyncio.to_thread` 会产生线程切换开销，单请求延迟可能略增（约 1‑3 ms），但整体吞吐提升更显著。  
- **安全考虑**：  
  - 仍然从外部 URL 加载图片与嵌入文件，未增加额外的安全检查；仍需依赖上层网络安全策略（如 URL 白名单、TLS 验证）防止恶意文件注入。  
  - `asyncio.to_thread` 的线程池默认大小受 `ThreadPoolExecutor` 控制，若未限制并发数，极端并发下可能触发资源耗尽（文件描述符、内存）。  

**⚠️ 潜在风险**：  
1. **线程安全**：`default_multimodal_input_loader` 内部是否使用了全局状态或 CUDA 非线程安全的 API；若出现竞争条件，可能导致随机崩溃或错误的张量复制。  
2. **异常传播**：`await asyncio.to_thread(...)` 中若抛出异常，会在协程层面重新抛出，但当前代码在异常后直接 `return None`，调用方可能未检测到错误，导致后续生成过程收到 `None` 而产生不可预期的错误。  
3. **资源泄漏**：大量并发请求会创建大量线程，若线程池未限制或未及时回收，可能导致系统文件描述符或内存耗尽。  
4. **测试超时延长**：将测试超时从 150 s 提升至 300 s，可能掩盖潜在的性能回退，降低回归测试的报警敏感度。  

**💡 关注建议**：  
- **审计线程安全**：确认 `default_multimodal_input_loader`（以及它内部调用的 `ImageLoader`、torch 数据加载等）在多线程环境下无全局状态写入；如有必要，可在 `to_thread` 前后加入 `torch.cuda.set_device` 或使用 `torch.no_grad()` 进行显式保护。  
- **异常处理**：在 `await asyncio.to_thread(... )` 调用后捕获具体异常并记录详细堆栈，必要时将错误包装为明确的返回码或抛出自定义异常，避免上层误以为处理成功。  
- **线程池限制**：在项目启动时显式配置 `asyncio.get_running_loop().set_default_executor(ThreadPoolExecutor(max_workers=N))`（`N` 根据机器 CPU/PCIe 带宽调优），防止请求激增时创建过多线程。  
- **安全加固**：对传入的 URL 做白名单或 MIME‑type 检查；对下载的文件大小设上限，防止恶意大文件耗尽磁盘或内存。  
- **性能基准**：新增基准测试，分别在纯同步与 `to_thread` 方案下测量峰值 QPS 与 99th‑percentile 延迟，确保改动带来预期的吞吐提升。  
- **回归测试**：保留原来的 150 s 超时阈值作为 **警告**（例如 `@pytest.mark.timeout(150, strict=False)`），并在 CI 中增加对执行时间的统计，以便快速发现性能异常。  

---  

**结论**：此次回滚兼具错误修正与性能提升两大价值，属于高优先级的关键路径修复。但引入的异步线程化需要仔细审查其线程安全性与资源管理，以防在大规模并发部署时出现隐蔽的稳定性或安全问题。建议在生产环境部署前进行充分的并发压力测试并落实上述安全/资源控制措施。

### fix(llm): preserve reasoning content when tool-call starts mid-chunk (#6902)
**SHA**: `57e6a79` | 🔗 [查看提交](https://github.com/ai-dynamo/dynamo/commit/57e6a79f5c330122db7a3e4df8545ec6c0547473)

**🎯 变更类型**：Bug修复  
**⚡ 重要程度**：🔴高  
**📋 变更摘要**：  
- 在 `OpenAIPreprocessor` 中新增 `postprocessor_parsing_stream`，统一处理推理内容解析、工具调用监狱（jail）以及后处理流的包装逻辑，解决了当工具调用在流的中间出现时推理内容被截断的缺陷。  
- 对 `ChoiceJailState` 与 `JailedStream` 增加对 **reasoning_content** 的临时缓存与合并，以在 jail 结束后恢复已收集的推理文本。  
- 新增两套 JSONL fixture 与对应的单元测试，验证在不同间隔（1 token 与 20 token）下的流式响应能够完整保留推理内容、正确解析工具调用、且不泄漏 markup。  

**🎯 影响范围**：  
- `lib/llm/src/preprocessor.rs`（核心流处理逻辑）  
- `lib/llm/src/protocols/openai/chat_completions/jail.rs`（工具调用 jail 机制）  
- `lib/llm/tests/`（新增单元测试）  
- 任何使用 `OpenAIPreprocessor::postprocess` 的上层服务（如 OpenAI 兼容 API、推理运行时）  

**🔍 技术洞察**  

- **架构影响**：  
  - **职责分离**：原先的 `transform_postprocessor_stream` 后直接在同一函数体内进行推理解析、工具 jail、审计聚合等三层包装，代码冗长且易出错。新增的 `postprocessor_parsing_stream` 把 **推理解析 + jail** 抽离为独立步骤，使主流程更清晰、可复用，降低了修改时的耦合度。  
  - **状态持久化**：在 `ChoiceJailState` 中加入 `pending_reasoning_content`，在 jail 期间缓存推理文本，随后在第一次非‑jail emission 时合并回 `reasoning_content`，保证了“工具调用中间启动”场景下的完整性。  
  - **兼容性**：保持原有 `transform_postprocessor_stream` 的行为，仅在 `postprocess` 中调用新封装方法，外部 API 的签名未变，兼容性风险极低。  

- **性能影响**：  
  - **额外包装**：新增 `Pin<Box<dyn Stream>>` 的两次 `Box::pin`（一次用于推理解析，一次用于 jail），会引入极小的堆分配和动态分发开销。对大多数推理请求（几百毫秒到几秒）而言可忽略不计。  
  - **缓存开销**：`pending_reasoning_content` 采用 `String` 按需创建，仅在需要 jail 时才分配，内存占用随推理文本长度线性增长。  
  - **测试覆盖**：新单元测试在 CI 中运行，确保在高频流（每 1 条 token）和低频流（每 20 条 token）下性能退化不显著。  

- **安全考虑**：  
  - 本次改动未引入新的外部输入解析或网络交互，仅在内部流处理阶段增加了状态缓存，**不影响安全边界**。  
  - 仍然保留原有的 **工具调用监狱（tool‑call jail）** 逻辑，避免未授权的函数调用泄露；新代码在合并 `reasoning_content` 时做了显式的 `take()` 与 `push_str`，不存在未检查的字符串拼接漏洞。  

**⚠️ 潜在风险**  

1. **流包装顺序错误**：如果将 `postprocessor_parsing_stream` 调用顺序误改（例如先 apply jail 再解析推理），可能导致推理内容再次被截断。  
2. **多次 jail 对同一 choice 的冲突**：`pending_reasoning_content` 在一次 jail 结束后自动 `take()`，若同一 choice 由于复杂正则匹配触发多次 jail，只有第一次的缓存会被合并，后续的推理内容可能仍被丢失。  
3. **异步边界泄漏**：`Box::pin` 后返回的 `impl Stream` 隐式捕获了 `self` 的所有权，若后续出现 `self` 被提前 drop（比如在测试中使用 `Arc::clone`），可能导致悬空引用。  
4. **向后兼容**：虽然 API 未变，但内部实现依赖 `runtime_config.reasoning_parser` 与 `tool_call_parser` 的存在性，若未来改动这些配置字段，必须同步更新本函数的判断逻辑。  

**💡 关注建议**  

- **代码审查**：重点审查 `postprocessor_parsing_stream` 中对 `should_parse_reasoning` 与 `should_jail` 的判断顺序，确保任何新增的解析器或 jail 配置都能走到对应分支。  
- **单元/集成测试**：在 CI 中加入大流量（千条 token）和异常 （工具调用多次交叉出现）两类压力测试，验证 `pending_reasoning_content` 在极端情况下仍能正确合并。  
- **文档更新**：在项目的 *preprocessor* 文档中说明：当工具调用在推理块中间出现时，推理内容会被缓存并在 jail 结束后自动恢复。并在 `runtime_config` 说明中标注该行为的前置条件（需开启 reasoning parser）。  
- **后续优化**：如注释所示，若需要基于 **token‑ids** 进行推理解析，可考虑将该步骤前置到 `transform_postprocessor_stream` 前，并在后端返回结构体中加入 `reasoning_content` 字段，以避免当前只能在文本层面解析的限制。

### fix: remove empty multimodal input to avoid invalid uuid check in vLLM (#6853)
**SHA**: `2ae9c29` | 🔗 [查看提交](https://github.com/ai-dynamo/dynamo/commit/2ae9c29067cfebb7d30dc2a967de85c140cd9f47)

**🎯 变更类型**：Bug 修复  
**⚡ 重要程度**：🟡 中  

**📋 变更摘要**  
在 `multimodal_pd_worker_handler.py` 中增加了对 `multi_modal_data` 的清理逻辑：当多模态字段为空（非 Tensor 且值为 `False`）时会将对应键删除，以避免 vLLM 在缺失 `multi_modal_uuids` 时抛出 `ValueError`。同时把原先的统一日志打印拆分为逐键调试信息。

**🎯 影响范围**  
- `components/src/dynamo/vllm/multimodal_handlers`  
- 依赖 vLLM 多模态输入检查的所有模型推理路径  

**💡 关注建议**  

1. **字典迭代安全** – 直接在 `for key, value in multi_modal_data.items():` 循环体内使用 `del multi_modal_data[key]` 会在 Python 运行时触发 “dictionary changed size during iteration” 错误。建议先收集待删除的键 (`keys_to_del = [k for k, v in multi_modal_data.items() if ...]`)，循环结束后统一 `for k in keys_to_del: del multi_modal_data[k]`。  

2. **空 Tensor 处理** – 当前仅对非 `torch.Tensor` 的空值做删除，若收到形状为 `(0, …)` 的 Tensor（即空张量），vLLM 仍可能报错。可在 `isinstance(value, torch.Tensor)` 分支中加入 `if value.numel() == 0: del multi_modal_data[key]` 的检查。  

3. **日志一致性** – 删除键后仍会触发 `logger.debug(f"Prepared multimodal data size: {len(multi_modal_data[key])}")`，但此时键已被移除，可能导致 `KeyError`。将日志移动到删除前或改为 `logger.debug("Prepared multimodal data size: %d for %s", len(value), key)`。  

4. **单元测试** – 增加覆盖以下场景的测试：  
   - 完全空的 `multi_modal_data`（无键）  
   - 包含空列表、空字符串、空 Tensor 的混合输入  
   - 正常非空输入保持不变  

5. **文档说明** – 在 `README` 或模块注释中说明该清理行为，以免调用方误以为空键会被保留并期待相应 `uuid`。  

整体来看，此改动修复了 vLLM 对空多模态输入的异常检查，但因在遍历时直接删除键以及未覆盖空 Tensor 场景，仍可能在极端输入下触发运行时错误。按上面建议调整后即可保证稳健性。
