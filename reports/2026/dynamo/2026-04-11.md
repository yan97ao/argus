### feat: Add glm-5 nvfp4 recipe (#7780)
**SHA**: `59df10d` | 🔗 [查看提交](https://github.com/ai-dynamo/dynamo/commit/59df10d1c73423d8d1c301dcfb917c466a893552)

**🎯 变更类型**：功能增强  
**⚡ 重要程度**：🔴 高  
**📋 变更摘要**：新增 `glm-5-nvfp4` 参考实现，提供面向 NVIDIA GB200（SM100）节点的 GLM‑5 NVFP4 模型的 **SGLang + Dynamo** 分布式预填 + 解码部署方案。包括完整的 Docker 镜像构建脚本、Kubernetes 资源清单（PVC、下载 Job、ComputeDomain、ResourceClaim、DynamoGraphDeployment）以及性能基准 Job，旨在实现约 19k token/s 的吞吐与低时延推理。  

**🎯 影响范围**：  
- `recipes/` 目录下全新 **glm-5-nvfp4** 子目录（Dockerfile、README、K8s YAML）  
- Docker 构建过程依赖 `lmsysorg/sglang` 基础镜像、`flashinfer` 0.6.7 轮子、`dynamo` 源码（`maturin` 编译）  
- Kubernetes **Dynamo Operator**（`DynamoGraphDeployment`、`ComputeDomain`、`ResourceClaimTemplate`）  
- 运行时环境变量及安全凭证（HF Token Secret）

### 🔍 技术洞察  

| 维度 | 关键影响 |
|------|----------|
| **架构影响** | - 引入 **Disaggregated Prefill/Decode** 架构：prefill 与 decode 分别在 1 台节点（4 GPU）和 4 台节点（共 16 GPU）上运行，通过 **NixL** 进行跨节点缓存传输。<br>- 通过 **ComputeDomain** 与 **ResourceClaim** 实现跨节点 GPU 通信通道（MNNVL 调度），对已有 Dynamo Operator 资源模型做了扩展。<br>- Docker 镜像在 Builder 阶段从源码编译 `dynamo` Python 绑定并安装 `flashinfer`，确保运行时库版本一致。 |
| **性能影响** | - 使用 **FP8‑E4M3 KV Cache** 与 **modelopt_fp4** 量化，显著降低显存占用（约 50%），提升显存利用率。<br>- 开启 **EAGLE speculative decoding (Spec‑V2)**，实验报告显示接受率 85‑95%，吞吐约 19k token/s，TTFT≈850 ms。<br>- 大幅并行化：`tensor‑parallel=16`、`data‑parallel=16`、`expert‑parallel=16`，以及 **prefill‑round‑robin‑balance**，在 5 台 GB200 节点上实现高并发 (512 请求)。<br>- 通过 `--kv-cache-dtype fp8_e4m3`、`--mem-fraction-static 0.72`、`--chunked-prefill-size 32768` 等调优参数，进一步压缩显存占用并改善 CUDA Graph 启动时延。 |
| **安全考虑** | - 镜像基于公开 `lmsysorg/sglang:v0.5.10.post1-cu130`，但在 Builder 阶段执行 `apt-get install`、`curl`、`git clone`，需确保构建环境可信。<br>- 使用 **HF Token Secret** 通过 `envFrom` 注入容器，已限制为只读且在 Job/Deployment 中未显式挂载，符合最小权限原则。<br>- `pip install` 直接从 PyPI 与 FlashInfer 私有索引 (`https://flashinfer.ai/whl/cu130`) 拉取二进制文件，若索引被篡改可能导致供应链攻击，建议在 CI 中做镜像签名或使用 cosign/Notary。<br>- 容器安全上下文已 **drop all capabilities**、启用 **RuntimeDefault seccomp**，但仍保留 `IPC_LOCK` 能力用于锁页，需确认该能力不会被滥用。 |
| **可维护性** | - 所有 YAML 文件采用 **Kubernetes 原生 CRD** (`DynamoGraphDeployment`、`ComputeDomain`），对已有平台友好；但 CRD 版本 (`v1alpha1`) 仍在迭代，升级时需检查兼容性。<br>- 环境变量与启动命令硬编码在 `deploy.yaml`，如果模型路径或量化方式变更，需要手动同步多处。建议抽离为 ConfigMap 或 Helm chart 参数。<br>- Dockerfile 中使用 `ARG DYNAMO_COMMIT="main"`，默认编译最新主分支，可能导致不可预期的破坏性改动，生产环境应锁定具体 commit SHA。 |

### ⚠️ 潜在风险  

1. **依赖未发布的 SGLang / Dynamo 版本**  
   - 构建时使用 `BASE_IMAGE="lmsysorg/sglang:v0.5.10.post1-cu130"`（预发布）以及 `git checkout ${DYNAMO_COMMIT}`（默认 `main`），若 upstream 出现不兼容改动，镜像构建或运行时可能失败。  

2. **跨节点 ComputeDomain 配置错误**  
   - `ResourceClaimTemplate` 与 `ComputeDomainChannelConfig` 必须与集群的 NVIDIA ComputeDomain 控制器配合。若控制器未启用或频道分配不匹配，会导致 Pod 启动卡在 `Pending`。  

3. **显存/资源超预期**  
   - 虽然 KV Cache 使用 FP8，但 `tensor‑parallel=16`、`data‑parallel=16`、`expert‑parallel=16` 仍会在每个 GPU 上创建大量中间张量，尤其在预填阶段（`prefill‑round‑robin‑balance`）可能出现 OOM。  

4. **安全凭证泄露**  
   - `hf-token-secret` 在 Job、Deployment 中通过 `envFrom` 注入，若 Pod 被攻击者获取（如通过不安全的 `kubectl exec`），令牌可被窃取并下载模型。建议开启 `Kubernetes secret` 的 `immutable`、`type: Opaque` 并限制 `serviceAccount` 权限。  

5. **镜像体积 & 启动时间**  
   - 完整的 Rust 编译 + FlashInfer + Dynamo Python wheel 会导致镜像体积 ≥ 6 GB，导致节点拉取时间长，首次启动需数分钟的 CUDA Graph 捕获，影响快速弹性伸缩。

### fix: LoRA unregister drops all the LoRA variants that share the same ID (#8053)
**SHA**: `9efa460` | 🔗 [查看提交](https://github.com/ai-dynamo/dynamo/commit/9efa460cd5a435566cbe613f3e7e59e4766ec9ec)

**🎯 变更类型**：Bug修复  
**⚡ 重要程度**：🔴高  

**📋 变更摘要**  
1. 修正 `MockDiscovery::unregister` 在注销实例时使用错误的标识（`instance_id`）导致同一 `instance_id` 下的多个 LoRA 变体全部被删除。改为使用完整的 `DiscoveryInstanceId`（`id()`）进行匹配。  
2. 为 `watch_and_extract_field` 引入基于完整 `DiscoveryInstanceId` 的内部状态，并在向外暴露时 **collapse** 为 `HashMap<u64, V>`，在同一 `instance_id` 存在多个条目时优先保留「基础模型」(suffix=None)。  
3. 新增大量单元测试，验证 LoRA 注销仅移除对应变体且不会影响 worker 的运行时配置，以及跨 endpoint 的实例不会相互 alias。  

**🎯 影响范围**  
- `lib/runtime/src/discovery/mock.rs`（MockDiscovery 实现）  
- `lib/runtime/src/discovery/utils.rs`（`watch_and_extract_field` 及内部 collapse 逻辑）  
- 依赖 `Discovery`、`DiscoveryInstanceId`、`DiscoveryEvent` 的上层模块（如模型调度、worker 状态监控）  

**🔍 技术洞察**  

- **架构影响**  
  - 引入 **`DiscoveryInstanceId`** 作为内部键，显式区分同一 `instance_id` 下的不同「命名空间/组件/端点/模型后缀」组合，防止不同实体误相互覆盖。  
  - `collapse_by_instance_id` 将完整键压平为 `u64`（原 `instance_id`），实现 **“base model 覆盖 LoRA”** 的业务规则，保持对外 API 不变，同时内部保持正确的唯一性。  
  - `watch_and_extract_field` 只在 **collapsed 视图** 发生变化时发送更新，降低了对下游 watcher's 不必要的唤醒，提升系统整体响应的确定性。

- **性能影响**  
  - 额外维护 `HashMap<DiscoveryInstanceId, V>`（完整键）以及一次遍历进行 collapse，带来 **O(N)** 的轻微额外成本。  
  - 通过比较 `tx.borrow()` 与 `collapsed` 是否相等，避免了 *no‑op* 的 `watch` 通知，整体 **IO 与调度开销下降**，对大规模实例（数千）仍在可接受范围。  
  - 测试覆盖率提升，可在 CI 中捕获潜在的性能回归。

- **安全考虑**  
  - 变更仅涉及内部数据结构和注销语义，不涉及网络、权限或加密等安全面向。  
  - 通过严格使用 `DiscoveryInstanceId`，避免了因错误注销导致的 **资源泄露**（worker 状态被误删）或 **错误的负载均衡决策**，从间接层面提升了系统的鲁棒性。

**⚠️ 潜在风险**  
1. **向后兼容性**：如果外部代码直接依赖原来的 `unregister` 行为（误把 LoRA 视为独立 worker），现在会保留原 worker，可能导致业务逻辑需要适配。  
2. **Collapse 策略误判**：`collapse_by_instance_id` 采用「第一次出现的非 LoRA」或「任意 LoRA」作为回退。若未来出现更复杂的模型层级（例如多层 LoRA），此策略可能需要重新审视。  
3. **并发竞争**：内部 `state` 使用 `HashMap` 并在 `tokio::spawn` 任务中被多线程访问。虽然只在单任务内部修改，但如果未来改为多任务共享，需考虑锁或 `DashMap` 等并发容器。  
4. **测试依赖**：新加入的单元测试基于 `MockDiscovery`，真实后端实现（如 etcd、k8s）若未同步 `DiscoveryInstanceId` 的使用，可能出现行为不一致。

**💡 关注建议**  
- **审查并更新所有 `unregister` 的调用方**，确保它们传入的 `DiscoveryInstance` 与新语义匹配（使用 `id()` 而非仅 `instance_id()`）。  
- **在集成测试/CI 中加入跨端点、LoRA 多变体的场景**，覆盖 `watch_and_extract_field` 的 collapse 路径，防止回退策略因业务扩展而失效。  
- **性能基准**：在生产规模（数千 worker）下跑一次基准，确认额外的遍历与 clone 不会导致显著延迟。若出现瓶颈，可考虑在 `collapse_by_instance_id` 中使用 `Arc<V>` 或 `Rc<V>` 降低克隆成本。  
- **文档补全**：在 `DiscoveryInstanceId`、`DiscoveryInstance::id()` 与 `DiscoveryInstance::instance_id()` 的区别上做好注释，避免后续开发者混淆。  
- **关注未来扩展**：若计划支持「多层 LoRA」或「模型分片」等新特性，评估是否仍能使用当前的「首个非 LoRA 优先」规则，必要时引入可配置的优先级策略。

### fix(llm): support reading eos_token_ids from tokenizer_config.json for models like Qwen3.5 with <|im_end|> token (#8091)
**SHA**: `b425b65` | 🔗 [查看提交](https://github.com/ai-dynamo/dynamo/commit/b425b65c25d7a901c2991fd71d2482d30ce260b0)

**🎯 变更类型**：Bug修复  
**⚡ 重要程度**：🔴高  
**📋 变更摘要**：在 `HFConfig` 的加载流程中加入对 `tokenizer_config.json` 的解析，读取其中的 `eos_token` 并从 `added_tokens_decoder` 中映射到对应的 token ID，进而将其合并到最终的 `eos_token_ids` 集合。此修复解决了 Qwen3.5 等模型在 `config.json` 中声明的 EOS 与实际 tokenizer 输出的 EOS 不一致的问题，保证生成过程能够正确检测结束标记。  

**🎯 影响范围**：  
- `lib/llm/src/model_card.rs`（HF 配置读取与 EOS 处理）  
- `lib/llm/src/model_card.rs` 的单元测试（新增针对 Qwen3.5 的验证）  
- 运行时的模型加载与文本生成模块（依赖 `eos_token_ids` 的停止逻辑）  

**🔍 技术洞察**：  
- **架构影响**：  
  - 新增 `resolve_eos_token_id_from_tokenizer_config` 辅助函数，解耦了 “从 generation_config / config.json” 到 “从 tokenizer_config.json” 两套 EOS 获取路径，保持了原有抽象层次。  
  - `final_eos_token_ids` 由不可变改为可变 (`let mut`)，并在读取完 tokenizer 配置后做一次合并，未对外部 API（`HFConfig::from_json_file`）产生兼容性破坏。  
- **性能影响**：  
  - 读取 `tokenizer_config.json` 只在模型首次加载时执行一次，文件大小通常在几 KB，IO 开销极低，对整体启动时延的影响可以忽略不计。  
  - 合并操作为一次线性查找 (`contains`) 与 `push`，时间复杂度 O(N)，N 为 EOS token 数量（通常 1‑2），对运行时性能无实质影响。  
- **安全考虑**：  
  - 新增对外部 JSON 文件的解析路径，使用 `anyhow::Context` 包装错误，避免 panic。  
  - 仍然依赖 `serde_json::Value` 直接访问字段，若文件被恶意构造（如注入巨大的数组或深层嵌套对象）可能导致解析时的内存占用激增。当前实现未显式限制深度/大小，建议在未来加入文件大小检查或 `serde_json::Deserializer::from_reader` 的流式限制。  

**⚠️ 潜在风险**：  
1. **文件缺失或路径错误**：若模型目录下不存在 `tokenizer_config.json`，`resolve_eos_token_id_from_tokenizer_config` 会返回错误，导致模型加载失败。现在通过 `if let Ok(...)` 静默忽略错误，但如果未来改为 `?`，需确保兼容性。  
2. **字段格式变更**：`eos_token` 可能在新版本的 HuggingFace 中采用不同字段名称或结构（如嵌套在 `special_tokens` 中），当前实现只兼容两种已知格式，可能在后续模型上失效。  
3. **重复 EOS ID**：虽然使用 `contains` 检查避免重复，但若 `tokenizer_config.json` 中声明的 EOS 与 `config.json` 中的 ID 不同且都有效，生成过程会收到两个停止标记，理论上不会引起错误，但需确认下游停止逻辑能够处理多值集合。  
4. **错误的 ID 映射**：`added_tokens_decoder` 中若出现同一 `content` 对应多个 ID（不常见），函数会返回第一个匹配的 ID，可能导致误识别。  

**💡 关注建议**：  
- **对外容错**：在加载模型时，若 `tokenizer_config.json` 读取失败，建议记录 WARN 日志而非直接返回错误，确保已有模型仍可使用。  
- **兼容性测试**：在 CI 中加入更多不同 tokenizer 配置的模型（如 GPT-NeoX、Llama 等）进行回归，以防新解析逻辑对其它模型产生意外影响。  
- **安全加固**：对读取的 `tokenizer_config.json` 增加文件大小上限（如 1 MiB）或使用 `serde_json::Deserializer::from_reader` 进行流式解析，防止恶意大文件导致 OOM。  
- **文档更新**：在项目文档中说明 “模型加载时会自动合并 tokenizer_config 中的 EOS”，并列出已知支持的 EOS 字段格式，以便用户了解行为。  
- **性能监控**：虽然影响微乎其微，但可在 `HFConfig::from_json_file` 添加一次性计时日志，帮助定位未来因文件读取异常导致的启动慢点。

### fix(operator): prevent orphaned old worker DCDs after rolling update (#7939)

**📋 变更摘要**：

- **operator 核心控制器**：`deploy/operator/internal/controller/dynamographdeployment_rollingupdate.go`（核心滚动更新实现）。  
- **测试套件**：`*_rollingupdate_test.go`（单元测试覆盖）  
- **相关 CRD 状态**：`DynamoGraphDeployment.Status.RollingUpdate` 与 `AnnotationCurrentWorkerHash`。

**🔍 技术洞察**  

| 维度 | 影响描述 |
|------|----------|
| **架构影响** | - 通过把 **状态写入** 从各子函数中抽离，仅在 `Reconcile()` 的 `defer` 中统一调用 `r.Status().Update`（或 `r.Update`）完成持久化，降低了状态竞争的概率。<br>- `completeRollingUpdate` 现在一次性删除 **所有** 旧 Worker DCD（而不是仅一次性删除单一代数），实现 **原子清理**，防止在多代更新后出现残留资源。<br>- 引入 `r.getOrCreateRollingUpdateStatus` 统一获取或创建状态对象，避免外部手动传递指针导致的空指针或状态不一致。 |
| **性能影响** | - 删除旧 DCD 时一次性遍历所有非当前代的 DCD，理论上在大规模集群（数百 worker）时会产生一次性 API 调用的增长，但相比之前多次分散调用的总次数并无显著差异；实际影响微乎其微。<br>- 移除多余的 `Status().Update` 调用，减少 API Server 写入压力，提升 reconciliation 循环的吞吐。 |
| **安全考虑** | - 本次改动不涉及权限提升、网络攻防或数据加密，仅是内部资源清理和状态管理。<br>- 通过将清理错误向上传递（`return fmt.Errorf("failed to delete old worker DCDs: %w", err)`），可以让 controller‑runtime 的错误处理机制触发 **requeue**，避免因 “忽略错误” 导致资源泄漏，间接提升系统的可用性和安全性（防止因残留 DCD 引发意外调度或资源争用）。 |
| **可维护性** | - 代码路径更简洁，删除了大量重复的 `Status().Update` 调用，降低了以后维护时忘记持久化状态的风险。<br>- 单元测试同步更新，覆盖了新加入的“卡死检测”路径，提升了回归安全性。 |

**⚠️ 潜在风险**  

1. **错误传播导致持续重试**  
   - `completeRollingUpdate` 现在在清理旧 DCD 失败时返回错误。若底层 API Server 在短时间内不可用或 DCD 被外部手动锁定，operator 将持续重试，可能导致 **滚动更新卡住** 并产生大量日志。  
   - 建议在异常频繁时设定 **Backoff** 或 **Rate‑limit**，防止对 API Server 造成冲击。

2. **状态延迟持久化**  
   - 状态的写入被统一放在 `Reconcile()` 的延迟函数中。如果在 `completeRollingUpdate` 前出现 panic 或异常导致 `Reconcile()` 退出，状态可能未被写入，导致 **短暂不一致**。  
   - 需要确保 `Reconcile()` 的 `defer` 块始终能安全执行（已在代码中通过 `defer r.persistStatus(ctx, dgd)` 等实现），但仍要在集群升级或 operator 重启时注意日志监控。

3. **一次性删除大量 DCD**  
   - 在极端情况下（如误操作导致数千代遗留），一次性遍历和删除会产生 **大量 DELETE** 请求，可能触发 API Server 的速率限制。  
   - 可考虑在 `deleteOldWorkerDCDs` 内部加入分批（batch）删除或限流。

4. **兼容性**  
   - 该改动改变了部分内部函数签名（去除 `*RollingUpdateStatus` 参数），如果外部自定义的插件或测试框架仍基于旧签名编译，会出现编译错误。当前仓库内部已同步更新，但在 downstream fork 中需注意。
