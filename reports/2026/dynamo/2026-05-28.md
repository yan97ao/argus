### feat: context propagation metadata http side (#9726)
**SHA**: `42bef4b` | 🔗 [查看提交](https://github.com/ai-dynamo/dynamo/commit/42bef4b1932ea63e9372dccc0c79b9ff804eb747)

**🎯 变更类型**：功能增强  
**⚡ 重要程度**：🔴高  

**📋 变更摘要**  
- 为 HTTP 与 gRPC 服务引入统一的 **上下文元数据传播**（metadata），通过 `Context::with_id_and_metadata` 将请求头/metadata 注入 `dynamo_runtime::pipeline::Context`。  
- 新增 `metadata.rs` 实现 HTTP、gRPC 元数据抽取、过滤（敏感信息）与配额（条目数、总字节）限制。  
- 迁移大多数代码路径从 `Context::with_id` 到 `with_id_and_metadata`，并在健康检查、TCP 网络、单元测试等处同步更新。  

**🎯 影响范围**  
- `lib/llm/src/grpc/service/*`（KServe、OpenAI、Tensor）  
- `lib/llm/src/http/service/*`（OpenAI、Anthropic、`service.rs`）  
- `lib/runtime/src/pipeline/context.rs`（Context 构造函数）  
- `lib/runtime/src/pipeline/network/tcp.rs`、`health_check.rs`、单元测试 `mock.rs`  
- 新增模块 `lib/llm/src/http/service/metadata.rs`  

**🔍 技术洞察**  

- **架构影响**  
  - **统一化上下文**：所有入口（HTTP、gRPC、内部 TCP）现在通过同一构造函数创建带 **metadata** 的 `Context`，实现跨协议的统一元数据传递，可在后续的 pipeline、拦截器或插件中直接使用。  
  - **显式声明**：`Context::with_id` 已被标记为 **deprecated**，强制迁移至 `with_id_and_metadata`，提升 API 可维护性并避免未来的二义性。  
  - **可配置前缀**：通过环境变量 `DYN_METADATA_HEADER` 自定义元数据前缀，提升部署灵活性。  

- **性能影响**  
  - **额外解析开销**：在每次请求入口会遍历所有请求头或 gRPC metadata，做前缀匹配、大小统计及敏感过滤。该开销主要是 O(N) 的字符串操作，影响微秒级别，几乎可以忽略在整体推理延迟（毫秒‑秒级）中。  
  - **内存占用**：元数据被复制到 `BTreeMap<String,String>`，对单次请求的额外内存约为 `#entries * (key_len+value_len)`；在限制（64 条、64 KiB）下可控。  
  - **编译期优化**：使用 `OnceLock` 缓存前缀字符串，避免每次请求重复读取环境变量。  

- **安全考虑**  
  - **敏感信息过滤**：显式跳过 `Authorization` 头及其 `Bearer` 前缀的值，防止凭证泄露到下游业务。  
  - **大小/条目配额**：防止恶意请求通过海量元数据耗尽内存或 CPU，超限会返回 `REQUEST_HEADER_FIELDS_TOO_LARGE (431)`（HTTP）或 `Status::invalid_argument`（gRPC）。  
  - **非 UTF‑8 丢弃**：对不满足 UTF‑8 的 header/metadata 安全地忽略，避免潜在的解析错误或注入攻击。  

**⚠️ 潜在风险**  

1. **向后兼容性**：  
   - 所有内部调用已更新，但外部库如果仍使用 `Context::with_id`（已 deprecated）会在未来版本失效。  
   - 迁移期间若遗漏调用点，可能导致 **metadata 丢失**，影响依赖该信息的插件或审计功能。  

2. **错误路径**：  
   - 当元数据超出配额，服务会直接返回错误码 431/invalid_argument。若调用方未捕获该错误，可能导致整体请求失败而不是回退到默认行为。  

3. **配置误用**：  
   - 环境变量 `DYN_METADATA_HEADER` 若设置为空或非法前缀，可能导致所有自定义元数据被错误过滤。  
   - 前缀匹配采用 `strip_prefix`，对大小写不敏感；但若用户使用类似 `X-Dynamo-Meta-` 但期望区分大小写，可能产生冲突。  

4. **测试覆盖不足**：  
   - 新增的抽取逻辑已有单元测试，但尚未覆盖 **跨协议**（HTTP ↔ gRPC ↔ TCP）完整链路，潜在的序列化/传递错误仍可能出现。  

**💡 关注建议**  

- **迁移计划**：在下一个次要版本中将 `Context::with_id` 完全删除，提前在内部库、示例以及文档中标记 `with_id_and_metadata` 为唯一入口。  
- **配置验证**：在服务启动阶段打印实际使用的 `metadata_header_prefix`，并对 `DYN_METADATA_HEADER` 做合法性检查（非空、合法 ASCII）。  
- **监控与报警**：  
  - 新增 `metadata_extraction_errors_total` 计数器，监控因大小/条目限制导致的 431 错误率。  
  - 记录被过滤的敏感字段（不记录值，只记录键）以帮助审计。  
- **性能基准**：在高并发（≥10k RPS）场景下跑基准，确认元数据抽取对整体延迟的贡献仍在可接受范围（<1 ms）。  
- **安全审计**：确认所有下游插件在使用 `Context.metadata()` 前进行 **只读** 访问，防止意外写入导致跨请求泄漏。  
- **文档更新**：在 `README`、API 文档以及 OpenAPI/Proto 注释中说明：  
  - 哪些 header 会被自动转为元数据。  
  - 默认配额与自定义方式。  
  - 何时收到 431 响应及其意义。  

通过上述改动，Dynamo 项目实现了 **跨协议的上下文元数据统一传播**，提升了可观察性与可扩展性，同时通过配额与敏感信息过滤保持了安全性。后续重点在于妥善完成迁移、监控新引入的错误路径以及验证在高负载下的性能影响。

### fix: add catch and unwind stuff for indexer so we have a log msg (#9974)
**SHA**: `faf0f51` | 🔗 [查看提交](https://github.com/ai-dynamo/dynamo/commit/faf0f51f0a3dbb9a3d6aa7614f5e56617c8de11b)

**🎯 变更类型**：Bug修复  
**⚡ 重要程度**：🔴高  
**📋 变更摘要**：在 KV‑Indexer 的内部线程以及线程池工作线程中加入 `std::panic::catch_unwind` 包装，捕获意外 panic 并通过 `tracing` 记录清晰的 panic 信息后重新抛出。新增 `panic_payload_message` 辅助函数用于将任意 panic payload 转为可读字符串。  

**🎯 影响范围**：  
- `lib/kv-router/src/indexer/kv_indexer.rs`（Indexer 主循环）  
- `lib/kv-router/src/indexer/mod.rs`（新增 `panic_payload_message`）  
- `lib/kv-router/src/indexer/thread_pool.rs`（线程池工作线程）  

**🔍 技术洞察**  

- **架构影响**  
  - 把原本直接崩溃的线程包装为可捕获的 panic，提升整体系统的可观测性和故障定位能力。  
  - 仍然通过 `std::panic::resume_unwind` 将 panic 重新抛出，保持原有 “一旦关键线程 panic，整个进程终止” 的语义，不会导致半残的索引器继续运行产生不一致状态。  
  - 新增的 `panic_payload_message` 只在日志层面使用，不影响业务逻辑或模块间接口。  

- **性能影响**  
  - `catch_unwind` 本身在正常路径（无 panic）几乎没有运行时开销，仅在异常路径会产生一次额外的异常捕获/展开成本。  
  - 额外的字符串转换（`downcast_ref`）仅在 panic 发生时执行，对整体吞吐量影响可以忽略。  
  - 线程池 worker 现在会在 `backend.worker` 返回 `Err` 时记录错误，这属于原有错误路径的可观测性提升，同样不会显著增加延迟。  

- **安全考虑**  
  - 通过显式记录 panic payload，避免因未记录的崩溃导致隐蔽的安全风险（如数据不一致、资源泄漏未被察觉）。  
  - 记录的内容仅是 `&str` 或 `String`，没有额外的内部结构泄露，符合日志安全原则。  
  - 仍然遵循 Rust 的 “panic = unwind” 语义，不会将异常转为错误返回，从而防止异常被误判为业务错误。  

**⚠️ 潜在风险**  

1. **双重 panic**：在 `catch_unwind` 块内部再次触发 panic（例如 `tracing::error!` 本身因内部错误 panic），会导致 `resume_unwind` 再次抛出，可能产生未捕获的二次 panic。  
2. **日志噪音**：大量短暂的 panic（如测试或异常输入）会在生产环境产生大量错误日志，需要配合日志采集/过滤策略。  
3. **兼容性**：添加 `panic_payload_message` 依赖 `std::any::Any`，若未来对 panic payload 类型做自定义包装，可能需要扩展函数以正确展示信息。  

**💡 关注建议**  

- **监控告警**：在监控平台为 `dynamo_kv_router::indexer_panic`、`dynamo_kv_router::thread_pool_worker_panic` 设置告警阈值，及时捕获异常突增。  
- **测试覆盖**：编写专门的集成测试，强制让 `KvIndexer` 或线程池 worker panic（例如 `panic!("test")`），验证日志内容与进程退出行为符合预期。  
- **日志采样**：考虑对 panic 日志做采样或限流，防止异常输入导致日志洪水影响后端日志存储。  
- **文档更新**：在项目的故障排查手册中加入 “Indexer panic 捕获与日志” 章节，说明如何通过 `tracing` 输出定位根因。  
- **审计 payload 转换**：如果未来引入自定义 panic payload（例如结构体），记得在 `panic_payload_message` 中增加相应的 `downcast_ref` 处理，确保不会返回 “Unknown panic payload”。  

---  

此改动显著提升了 KV‑Indexer 与线程池的可观测性与故障定位效率，风险可控且对性能影响微乎其微。建议尽快合并并在生产环境开启相应监控。

### fix: use shared conftest for selective module skip in GMS tests (#9818)
**SHA**: `8294ddd` | 🔗 [查看提交](https://github.com/ai-dynamo/dynamo/commit/8294dddfc5828b597f8e3161d252e33d2a18c150)

**🎯 变更类型**：Bug修复  

**⚡ 重要程度**：🔴高  

**📋 变更摘要**：  
在 `gpu_memory_service` 测试目录新增 `_deps.py`，统一检测 `pynvml`、`torch`、`gpu_memory_service` 本身以及 CUDA 可用性。随后所有受影响的测试文件改为基于该共享依赖模块进行条件跳过，避免在缺少相应依赖的 CI 环境中因 `ImportError` 抛异常而导致整个测试套件失败。  

**🎯 影响范围**：  
- `lib/gpu_memory_service/tests/` 下的所有单元/集成测试：`test_failover_lock.py、test_mx_integration.py、test_runtime_flows.py、test_torch_integration.py`  
- CI/CD 流水线（尤其是不提供 GPU、torch、pynvml 的镜像）  
- 依赖检测逻辑（`_deps.py`）  

**🔍 技术洞察**：  
- **架构影响**：  
  - 引入了测试专用的共享依赖声明层 `_deps.py`，将原本分散在各测试文件中的 `try/except import` 与 `pytest.importorskip` 统一抽象，降低了代码重复度，提高了测试代码的可维护性。  
  - 不影响生产代码路径，仅对测试层面做了改动，保持了现有模块化设计不变。  

- **性能影响**：  
  - 依赖检测在模块导入时执行一次，开销极小（仅 `importlib.util.find_spec`），相较于之前在每个测试文件中重复尝试导入的方式略有提升。  
  - 对 CI 流程的整体执行时间可能略有下降，因为在缺少依赖的环境下可以更早地 `skip` 整个测试文件，省去后续的准备工作。  

- **安全考虑**：  
  - 仅涉及测试环境的条件跳过，不涉及运行时业务逻辑或外部输入处理，安全风险基本为 **零**。  

**⚠️ 潜在风险**：  
1. **依赖检测误判**：`_check_gms_usable` 只检查子模块是否可被 `importlib` 发现，若子模块在运行时因缺失其他运行时依赖（如 C 动态库）而失败，仍会进入测试并可能产生未捕获的异常。  
2. **隐藏真实回归**：在某些 CI 环境中因为依赖缺失而直接跳过测试，可能导致功能回归未被检测到。  
3. **跨平台差异**：不同平台（如 Windows vs Linux）对 `torch.cuda.is_available()` 的实现可能略有差异，导致 `HAS_CUDA` 与实际可用性不一致。  

**💡 关注建议**：  
- 为 `_deps.py` 添加简单的单元测试，确保在各种模拟缺失依赖的环境下返回值均符合预期。  
- 在 CI 配置中明确标记哪些测试需要完整 GPU 环境，避免关键功能在非 GPU runner 上被误跳过。  
- 若项目计划在非 GPU 环境运行部分 GMS 逻辑，考虑在代码层面也提供相应的 “mock / stub” 实现，以防测试全被 skip。  
- 定期审视 `HAS_*` 标志的检测方式，确保与上游库（torch、pynvml）版本升级后仍保持兼容。

### 架构影响  
| 维度 | 说明 |
|------|------|

### 性能影响  
| 维度 | 正向影响 | 潜在负面 |
|------|----------|----------|
| **指标采集开销** | - 仅在 **pass 结束** 以及 **每个 token** 时执行轻量的原子/锁操作（`IntGauge.set`、`Histogram.observe` 等）。<br>- 使用 **Arc\<NativeMockerMetrics\>**，避免重复创建 collector。 | - 每个请求会在 `request_timing` 中获取一次锁（`state.lock()`）用于 **handle 绑定**，在高 QPS（>10k rps）场景可能成为争用热点。<br>- `Histogram` 的 bucket 较多（VLLM TTFT 22 桶、ITL 18 桶等），在极高并发下会导致内部向量扩容/锁竞争。 |

### chore(kv-router): clean up worker selection routing (#10026)
**SHA**: `9ecf2d3` | 🔗 [查看提交](https://github.com/ai-dynamo/dynamo/commit/9ecf2d3350141f96f765840aa72b1c953ec60489)

**变更概览**  
本次提交对 KV‑router 的“粘性会话”实现进行了一次大幅重构：

1. **模块拆分 & 命名统一**  
   - 删除 `agent_controller`，新增 `sticky::lifecycle::SessionLifecycleController` 负责会话 open/close RPC。  
   - 新增 `sticky::coordinator::StickySessionCoordinator`，统一管理会话亲和性（绑定、解绑、刷新）以及生命周期 RPC。  
   - 原 `sticky_sessions` 改名为 `sticky::router`，对外仅保留 `StickySessionRouter`，去掉了直接接受 `PreprocessedRequest` 的 `resolve/peek` 接口，改为基于会话 ID 的操作。

2. **路由逻辑精简**  
   - 把原来在 `KvPushRouter::select_worker` 中散落的粘性查询、Pinned‑Worker 处理抽象为 `RoutingRequestParts`、`BestMatchArgs`，统一走 `KvRouter::find_best_match_details`。  
   - 引入 `resolve_pinned_worker_rank`，在未能确定唯一 DP‑rank 时直接报错（原来默认为 0 的兜底行为已去除），防止错误路由。  
   - `select_best_match` 成为独立函数，返回统一的 `WorkerSelection`，并在 `select_worker` 中只负责粘性/错误路径的前置检查。

3. **相关数据结构调整**  
   - `WorkerCacheHitEstimate` 去掉 `cached_tokens` 字段，相关统计改在 `WorkerSelection` 中返回。  
   - `KvPushRouter` 结构体由 `sticky_sessions`、`agent_controller` 改为 `sticky: Arc<StickySessionCoordinator>`，并在构造时一次性创建 `StickySessionCoordinator`（内部仍持有 `StickySessionRouter` 与 `SessionLifecycleController`）。

4. **单元测试迁移**  
   - 对 `StickySessionRouter` 的测试改为直接使用会话 ID 接口；新增 `resolve_pinned_worker_rank`、`sticky_allowed_for_phase` 的单元测试。

**影响范围**  
- **路由层**：`KvPushRouter`、`PrefillRouter`、`push_router` 的所有调用点均已更新；任何仍使用 `sticky_sessions`（或 `agent_controller`）的外部代码会编译失败。  
- **会话生命周期**：原来的 `AgentController::on_routed` 被 `StickySessionCoordinator::on_routed` 替代，返回的 `SessionRoutingResult` 只包含 `deferred_close`。  
- **缓存命中统计**：移除 `cached_tokens` 可能影响监控/日志代码，需要确认没有其他模块仍引用该字段。  

**建议**  
1. **兼容性检查**：在项目其他子模块（如 `llm_service`, `engine`）中搜索 `agent_controller`、`sticky_sessions`、`cached_tokens` 的使用，确保已全部迁移到新的 API。  
2. **错误路径测试**：当前对未能解析 DP‑rank 的情况会直接 `anyhow!` 报错，建议在单元测试中加入对应的异常路径，防止运行时 panic。  
3. **文档同步**：更新 `docs/kv_router.md`（或相应协议说明）中关于粘性会话、Pinned‑Worker 的行为说明，尤其是“不再默认使用 rank 0”。  
4. **性能基准**：重构后路由路径多了一层 `RoutingRequestParts` 的构造，建议跑一次完整的 KV‑router 吞吐量基准，确认没有显著回退。  
5. **审计日志**：`select_worker` 中仍有 `tracing::debug!` 打印旧字段（`best_worker` 等），可统一改为使用 `selection`，保持日志结构一致。  

整体来看，重构提升了职责分离和代码可维护性，但涉及的 API 改动较大，需要在全仓库范围内完成迁移并验证异常路径的健壮性。
