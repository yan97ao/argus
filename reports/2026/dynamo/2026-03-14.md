### feat(perf): add request-plane, transport, and work-handler metric definitions (#6735)
**SHA**: `cec19d4` | 🔗 [查看提交](https://github.com/ai-dynamo/dynamo/commit/cec19d4db601eb91aa228bb22e6daef340f169e4)

**🎯 变更类型**：功能增强 / 性能优化（可观测性）  
**⚡ 重要程度**：🔴 高  
**📋 变更摘要**  
本次提交为 **dynamo** 项目新增一套细粒度的 Prometheus 监控指标，覆盖请求路径（request‑plane）、传输层（transport）以及后端工作处理器（work‑handler）的关键时延与资源使用，并在 Python 代码生成器中导出对应的常量名称。同时提供了安全的注册助手 `add_metric_or_warn` 以及幂等的注册函数，提升了系统可观测性与运维调试能力。

**🎯 影响范围**  
- `lib/runtime/src/metrics/`：新增 `request_plane.rs、transport_metrics.rs、work_handler_perf.rs` 三个模块以及对 `MetricsRegistry` 的扩展。  
- `lib/runtime/src/metrics/prometheus_names.rs`：统一的常量前缀与指标名称。  
- `lib/bindings/python/...`：Python 端的指标名称导出（`prometheus_names.py`）。  
- 代码生成器 `prometheus_parser.rs`：添加了对子模块的 TODO 说明。  
- 任何使用 `MetricsRegistry` 注册/采集指标的运行时路径（如 LLM HTTP 服务、Dynamo 核心服务）均受到影响。

**🔍 技术洞察**  

- **架构影响**  
  - **模块化**：新增 `request_plane`, `transport_metrics`, `work_handler_perf` 三个独立子模块，使指标组织更贴合业务层次，符合“层‑级职责分离”。  
  - **注册路径**：引入 `add_metric_or_warn`，在注册失败时记录警告而不是返回错误，防止因单个指标冲突导致整个服务启动失败，提升容错性。  
  - **幂等注册**：使用 `OnceCell` 实现注册过程的幂等（分别针对 `MetricsRegistry` 与原生 `prometheus::Registry`），避免重复注册导致的 panic。  
  - **代码生成缺口**：当前 `prometheus_parser` 仍未递归处理 Rust 子模块，导致 Python 端手动写入的嵌套类在重新生成时会被覆盖，产生维护风险。  

- **性能影响**  
  - **内存占用**：每个 `Histogram`/`Counter`/`Gauge` 在首次创建时会分配桶结构，新增约 10+ 个指标，内存提升在毫级别，基本可以接受。  
  - **运行时开销**：在每一次 `generate → send_request`、`send_request → response`、TCP/NATS 计数等关键路径上记录 Histograms/Counter，会产生原子操作（`inc`, `observe`). 在高 QPS 场景下，累计的原子 CAS 开销可能占总 CPU 的 0.1%~0.5%。如果业务对毫秒级延迟极端敏感，建议通过 Feature Flag 控制是否启用这些指标。  
  - **注册成本**：注册在启动阶段完成，使用 `OnceCell` 确保只执行一次，几乎没有运行时影响。  

- **安全考虑**  
  - **信息泄露**：新增的指标均不携带业务敏感数据，仅统计计数与时延，风险低。  
  - **标签治理**：`NATS_ERRORS_TOTAL` 使用标签 `error_type`，如果未来加入更多 label，需防止高基数造成 Prometheus 爆炸。当前只保留固定枚举值，安全性可控。  
  - **异常处理**：`add_metric_or_warn` 将注册错误降级为日志，避免因恶意或错误的指标名称导致服务不可用，提升系统韧性。  

**⚠️ 潜在风险**  

1. **指标重复注册**  
   - 若第三方库或未来的其它模块手动创建同名指标，`add_metric_or_warn` 只记录警告，可能导致监控数据缺失而不易被发现。  
2. **代码生成覆盖**  
   - 手动在 `prometheus_names.py` 添加的 `transport.tcp`、`transport.nats` 类在下一次运行 `gen-python-prometheus-names` 时会被覆盖，导致 Python SDK 与实际指标不匹配。  
3. **高基数风险**  
   - 虽然当前标签固定，但若后续新增动态标签（如错误码、客户端 IP），可能导致 Prometheus 存储膨胀。  
4. **性能回退**  
   - 在极端高并发（>100k QPS）场景，频繁的 histogram `observe` 可能成为热点；未提供采样或开关，可能造成 CPU 争用。  
5. **兼容性**  
   - 新增的模块需在所有 `MetricsRegistry` 初始化路径显式调用对应的 `ensure_*_registered`，否则对应指标永远不会被收集，可能导致监控缺口。

**💡 关注建议**  

- **完善代码生成**：尽快实现 `prometheus_parser.rs` 对嵌套 `pub mod` 的递归解析，自动生成完整的 Python 类，避免手动维护导致的同步错误。  
- **指标开关**：在 `MetricsRegistry` 或配置层加入 Feature Flag（如 `enable_request_plane_metrics`），允许在性能敏感的部署中关闭这些细粒度指标。  
- **监控监测**：在 CI/CD 中加入检测指标注册冲突的测试，用 `prometheus::Registry::register` 的错误信息做断言，确保新指标不被意外覆盖。  
- **标签基数管理**：为 `NATS_ERRORS_TOTAL` 明确定义枚举型标签值，防止日后误用。若需要更多维度，考虑使用 `HistogramVec` 或 `CounterVec` 并设立基数上限。  
- **文档与示例**：在项目文档中补充这些新指标的含义、推荐的查询语句以及典型的阈值报警示例，帮助运维快速上手。  
- **回滚路径**：保持 `MetricsRegistry::add_metric_or_warn` 的实现不变，以便在发现指标冲突或性能异常时，仅通过日志定位问题而不影响服务可用性。  

通过上述措施，可在提升系统可观测性的同时，保持架构的整洁性、性能的可控性以及安全的稳健性。

### feat: propagate OTEL trace context across E/P/D multimodal workers (#7239)
**SHA**: `c263a99` | 🔗 [查看提交](https://github.com/ai-dynamo/dynamo/commit/c263a99ed268424692bcd168765f08525bd425a7)

**🎯 变更类型**：功能增强  

**⚡ 重要程度**：🔴高  

**📋 变更摘要**  
本次提交在多模态工作流（PD、Encode、Decode）之间新增了 OpenTelemetry（OTEL）追踪上下文的传播机制。通过在关键函数签名中加入可选的 `context` 参数，并在 RPC 调用（`round_robin`、`generate`）以及内部的 `engine_client.generate` 中注入由 `build_trace_headers` 生成的追踪头部，实现了跨进程、跨节点的链路追踪。同步更新了 Python 绑定的类型声明以支持新参数。

**🎯 影响范围**  
- `components/src/dynamo/vllm/multimodal_handlers/multimodal_pd_worker_handler.py`  
- `components/src/dynamo/vllm/multimodal_handlers/worker_handler.py`  
- `components/src/dynamo/vllm/multimodal_utils/prefill_worker_utils.py`  
- 单元测试 `test_vllm_multimodal_pd_worker_handler.py`（mock 调用签名更新）  
- Python FFI 类型文件 `lib/bindings/python/src/dynamo/_core.pyi`（新增 `context` 参数）  

**🔍 技术洞察**  

- **架构影响**  
  - **可观测性提升**：通过在所有跨 worker 的 RPC 路径中注入 OTEL trace headers，实现了从前端请求到各层 worker（Encode/Decode/PD）的统一链路追踪，便于故障定位和性能分析。  
  - **向后兼容**：`context` 参数默认 `None`，原有调用保持兼容；但必须保证在分布式运行时（`DistributedRuntime`）能够把 `Context` 对象正确序列化/反序列化。  
  - **接口统一**：在 Python 绑定层统一加入 `context`（以及 `annotated` 参数），为以后在其他语言绑定中推广 OTEL 支持奠定基础。  

- **性能影响**  
  - **微小开销**：`build_trace_headers` 只做一次字典构造，网络层额外传输的 Header 字段极小（几百字节），对吞吐量影响可忽略。  
  - **潜在阻塞**：若 `context` 对象在序列化过程中出现异常（如包含不可序列化的对象），会导致 RPC 调用失败，进而产生额外的重试或错误处理开销。  
  - **CPU/内存**：新增的参数声明和少量对象创建对 CPU/内存占用几乎没有可感知的增长。  

- **安全考虑**  
  - **信息泄露**：OTEL trace headers 会随着 RPC 传播，若集群内部未加密（如使用明文 HTTP），可能泄露调用链信息。建议在传输层使用 TLS 或内部 VPC。  
  - **上下文污染**：在多租户环境下，确保 `context` 只来源于对应请求的根 Span，避免跨租户的 trace 关联。  

**⚠️ 潜在风险**  

1. **序列化兼容性**：`Context` 对象需实现 `jsonable`（或通过 `json.dumps`）才能随 RPC 传输；若未来 OTEL 版本更改内部结构，可能导致反序列化失败。  
2. **默认行为变更**：某些内部调用可能未显式传递 `context`，导致生成的 Span 不连续，观察到的链路会出现断点。  
3. **错误传播**：`build_trace_headers` 在异常情况下可能抛出，未被捕获会导致整个生成过程中断。  
4. **测试覆盖不足**：仅在 `test_vllm_multimodal_pd_worker_handler.py` 做了 mock 参数调整，缺少对 `context` 实际传递的端到端验证。  

**💡 关注建议**  

- **完善单元/集成测试**：加入带真实 `Context`（或 Mock）对象的完整工作流测试，验证 `trace_headers` 在每一层 RPC 中都能正确到达并被下游使用。  
- **文档与示例**：在项目文档中说明如何在用户代码里创建并传递 `Context`，以及在部署时需要的网络安全配置（TLS、内部防火墙）。  
- **异常容错**：在 `build_trace_headers` 调用前后加入 `try/except`，在生成失败时回退为 `None`，防止 OTEL 问题导致业务中断。  
- **监控新增延时**：在生产环境开启对 `mm:pd:generate_*`、`mm:decode_worker_generate` 等 NVTX 区间的延时监控，确认追踪头部传递不产生可感知的额外延迟。  
- **统一 Context 处理**：考虑在 `DistributedRuntime` 层实现统一的 `inject_context` / `extract_context` 辅助函数，减少在每个业务函数中手动传递的样板代码。  

通过上述措施，可在保证可观测性提升的同时，最大限度降低对系统稳定性与安全性的冲击。祝编码顺利！

### fix: populate logprobs bytes and token fields in OpenAI-compatible responses (#6953)
**SHA**: `947939c` | 🔗 [查看提交](https://github.com/ai-dynamo/dynamo/commit/947939c727b24d9e634e87dbeac87273d1d2a173)

**🎯 变更类型**：Bug修复  

**⚡ 重要程度**：🔴高  

**📋 变更摘要**：  
- 在 OpenAI‑兼容的返回结构中为 `logprobs` 补全 `bytes` 与 `token` 字段，确保每个 token 及其 top‑logprobs 能同时提供字符串、token_id 与 UTF‑8 字节序列。  
- 为后端 `TopLogprob` 结构体新增 `bytes`，并实现统一的 `convert_backend_top_logprobs` 与 `token_to_utf8_bytes` 转换工具，统一在所有 delta 生成器中使用。  
- 更新 VLLM 处理层在缺失 `decoded_token` 时使用 tokenizer 解码，并在测试中加入相应校验。  

**🎯 影响范围**：  
- `components/src/dynamo/vllm/handlers.py`（Python）  
- Rust 侧 `lib/llm/src/protocols/common/llm_backend.rs`、`openai.rs`、`chat_completions/aggregator.rs`、`chat_completions/delta.rs`、`completions/delta.rs`  
- 测试套件 `tests/utils/payloads.py`  

---

### 🔍 技术洞察  

| 维度 | 影响分析 |
|------|----------|
| **架构影响** | - 在统一的 OpenAI 响应模型中加入 `bytes: Option<Vec<u8>>`，保持向后兼容（`skip_serializing_if`），不影响已有客户端。<br>- Python VLLM handler 通过可选 `tokenizer` 解码缺失的 `decoded_token`，实现了 “后置填充” 的桥接层，保持 core 与 VLLM 之间的解耦。<br>- 新增的 `convert_backend_top_logprobs` 在 Rust 侧统一处理 token、logprob、bytes 的映射，降低了各模块重复逻辑。 |
| **性能影响** | - 仅在出现 `decoded_token` 为 `None` 时额外调用 tokenizer 的 `decode`，以及对每个 token 执行 `as_bytes()`。这在普通请求中占比极小，预计 < 1 ms 额外开销。<br>- `bytes` 字段为 `Option<Vec<u8>>`，序列化时会被跳过（`skip_serializing_if`），对网络传输负荷几乎没有影响。 |
| **安全考虑** | - 只进行 UTF‑8 编码，无跨站脚本或注入风险。<br>- 在 Python 中捕获 `tokenizer.decode` 异常，防止异常泄露导致服务崩溃。<br>- `bytes` 字段通过 `Option` 防止意外暴露内部实现细节。 |
| **可维护性** | - 通过集中式函数 (`token_to_utf8_bytes`, `convert_backend_top_logprobs`) 降低代码重复，后续若需更改字节序列化方式，只需编辑这两处。<br>- 测试用例已同步校验 `bytes` 字段，提升回归安全性。 |
| **兼容性** | - 旧版客户端若不识别 `bytes` 字段仍能正常解析（字段被忽略），且 `serde` 的 `default`/`skip_serializing_if` 防止因缺失字段导致反序列化错误。 |

---

### ⚠️ 潜在风险  

1. **Tokenizer 不可用**：如果运行时 `self.engine_client` 缺少 `tokenizer`，`bytes` 将为 `None`，而测试期望非空 token 时会失败；下游依赖 `bytes` 的业务可能得到不完整数据。  
2. **Tokenizer 与后端不一致**：VLLM 使用的分词器与 `engine_client.tokenizer` 不同会导致 `bytes` 与实际 token 丢失匹配，产生错误的字节表示。  
3. **异常隐藏**：Python 中捕获所有异常后返回 `None`，可能掩盖分词器实现错误，建议在日志中记录调试信息。  
4. **序列化体积微增**：在高频率、极大 token 数的流式返回场景下，额外的 `bytes` 列表会稍微增加响应体积（约 1‑2 byte / token），对带宽极端受限的环境可能有影响。

- **安全考虑**：新增的 apt 安装步骤引入了额外的系统库（例如 `libtcmalloc-minimal4`、`gcc-12`），若未锁定版本号可能导致供应链风险；`LD_PRELOAD` 加载了 `libtcmalloc_minimal.so.4` 与 `libiomp5.so`，需确保这些库的来源可信且与运行时兼容。  

**⚠️ 潜在风险**：  
1. **构建兼容性**：CPU 只在 `amd64` 上默认开启 AVX512，若在不支持 AVX512 的机器上强制开启，可能导致二进制运行错误。  
2. **CI 测试缺失**：当前 PR 未显式加入 CPU 平台的 CI 验证，可能出现未捕获的构建或运行时错误。  
3. **环境变量冲突**：`LD_LIBRARY_PATH` 与 `LD_PRELOAD` 在不同设备分支间的拼接方式使用了 `${LD_LIBRARY_PATH:-}`，若上游已有同名变量，可能产生覆盖或重复路径。  
4. **镜像体积增大**：CPU 镜像额外包含 `gcc-12`、`uv`、`libtcmalloc` 等，镜像大小会显著上升，影响拉取与分发成本。  

**💡 关注建议**：  
- 在 CI 中加入 `cpu` 设备的全链路测试（包括跨编译选项的矩阵），确保 `docker build`、`vllm` 编译与运行均通过。  
- 为 AVX512/AMX 可选标志提供明确的默认值并在文档中标注不支持的 CPU 必须手动关闭（`VLLM_CPU_DISABLE_AVX512=true`）。  
- 将关键系统库（`gcc-12`、`libtcmalloc`）的版本锁定在 `apt-get install` 命令中，以降低供应链风险。  
- 考虑将 CPU 镜像拆分为 “base” 与 “runtime” 两层，以减少不必要的编译工具在最终运行镜像中的存在，控制镜像体积。  
- 更新 `README` 与 Docker 使用示例，明确 `--device cpu` 参数的含义、硬件要求以及推荐的构建选项。
