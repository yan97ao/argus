### feat(RL): add vLLM Tokens-in-Tokens-Out and RL related response support (#9651)
**SHA**: `df42116` | 🔗 [查看提交](https://github.com/ai-dynamo/dynamo/commit/df4211638cf8041e2e32700e676402387a80843a)

**🎯 变更类型**：功能增强、架构变更、性能优化、测试补全、轻量安全修复  

**⚡ 重要程度**：🔴 高  

**📋 变更摘要**  
1. 在 vLLM 后端引入 **RL（Reinforcement Learning）‑Token‑In‑Token‑Out（TITO）** 支持，新增 `--enable-rl` 标志并自动切换日志概率 (`logprobs_mode`) 为 `processed_logprobs`。  
2. 新增一套 **nvext.extra_fields** 机制，能够在响应中携带 `engine_data`（包括累积的 token_id、per‑token logprobs、prompt_logprobs、routed_experts 等），并实现相应的序列化、去噪与安全包装。  
3. 为 **非有限 logprob**（`-inf/inf/nan`）提供哨兵值 `_MIN_FINITE_LOGPROB`，防止 JSON/serde 失效。  
4. 在 Rust 运行时 (`rust/lib.rs`) 暴露 `system_status_server_url` 接口，以便 Worker 在 RL 请求路由中使用。  
5. 大量单元/集成测试覆盖新的序列化、缓存盐、采样默认、engine_data 累积以及 routed_experts 的 Base85 编码。  

**🎯 影响范围**  
- `components/src/dynamo/vllm/args.py`, `backend_args.py`, `handlers.py`, `llm_engine.py`, `main.py`（Python 层）。  
- `lib/bindings/python/rust/lib.rs`（Rust 层）。  
- 相关测试文件（`test_vllm_*`）以及 CI/pytest 依赖。  
- 所有使用 vLLM 作为后端的 Dynamo 组件（集群调度、LLM Engine、前端路由）。  

**🔍 技术洞察**  

- **架构影响**  
  - **RL‑TITO 新模式**：通过 `enable_rl` 标志在运行时切换两套行为路径：① 在普通请求中保持模型 `generation_config.json` 的采样默认；② 在 RL‑token‑in 请求中强制使用 vLLM 的原始 `SamplingParams`（不受模型默认覆盖），保证策略梯度等训练代码能够直接使用模型的原始分布。  
  - **engine_data 统一链路**：在 `generate_tokens` 流程中累积 `token_ids`、`log_probs`，在最终 chunk（含 `finish_reason`）一次性注入 `engine_data`，并通过 `nvext.extra_fields` 让前端按需消费，提高了 **后端‑前端解耦**。  
  - **nvext 统一抽象**：`_iter_nvext_sources`、`_nvext_extra_field_requested` 等函数把不同层级（`request["nvext"]`、`request["extra_args"]["nvext"]`）的 extra_fields 合并，避免因数据来源分散导致的逻辑缺失。  
  - **Rust‑Python 交互**：新增 `system_status_server_url` 方法让 Python 侧能够获取运行时自动启动的系统状态服务地址，进一步统一了 **Worker‑Frontend** 的路由描述。  

- **性能影响**  
  - **采样参数构造**：在 RL‑TITO 场景下跳过模型配置的覆盖，略微提升 `build_sampling_params` 的执行速度。  
  - **日志概率安全包装**：`_finite_logprob` 与 `math.isfinite` 检查成本极低（O(1)），对整体吞吐几乎没有可感知影响。  
  - **engine_data 累积**：对每个 token 按 `output_index` 使用 `list.append`、`deque` 展平 logprobs，内存占用随生成长度线性增长。对超长序列（>10k token）需关注 **内存峰值**，但在 RL 训练场景中往往使用较短片段，风险可接受。  
  - **routed_experts 序列化**：仅在 **最终 chunk** 执行一次 Base85 编码与 NumPy → bytes 转换，开销与 `routed_experts` 张量大小成正比。对常规模型（few‑MB）影响微乎其微；对大规模 MoE模型需评估一次性编码是否会导致 GC 瞬时抖动。  

- **安全考虑**  
  - **logprobs_mode 强制**：`configure_rl_logprobs_mode` 在 `--enable-rl` 开启时强制使用 `processed_logprobs`，防止 RL 场景意外返回原始 logits（可能泄露模型内部概率分布）。若用户误传 `raw_logits`，代码会抛 `ValueError`，避免潜在的 **信息泄露**。  
  - **非有限数值哨兵**：`_MIN_FINITE_LOGPROB = -1e30` 是有限数值，可安全序列化为 JSON，防止 `null` 失去类型信息导致前端解析错误。  
  - **系统状态 URL 暴露**：`system_status_server_url` 返回本机 IP 与端口，若未做好网络隔离或防火墙策略，可能被外部访问到 **运行时监控接口**。建议在生产环境通过安全组或容器网络限制。  
  - **Typed Deserialization**：对 `prompt_logprobs` 采用字符串化 token_id 键，符合 Rust `HashMap<u32, _>` 的字符串解析，实现 **数据完整性**，避免因键类型不匹配被 silently丢弃。  

**⚠️ 潜在风险**

2. **vLLM 私有属性依赖**：对 `SamplingParams._bad_words_token_ids` 的直接写入在 vLLM 未来版本可能改名或封装，导致运行时 `AttributeError` 并中断 RL 训练。

4. **routed_experts 大小**：MoE 模型的 `routed_experts` 可能是稀疏矩阵，Base85 编码后体积大增，若网络带宽受限，最终 chunk 发送会出现 **延迟峰值**。

### feat: Add DCO check in pre-commit (#10314)
**SHA**: `f9beb9c` | 🔗 [查看提交](https://github.com/ai-dynamo/dynamo/commit/f9beb9ce2f327c88c1f27cb7065be7a5a9d7fe5e)

**🎯 变更类型**：功能增强  

**⚡ 重要程度**：🔴高  

**📋 变更摘要**  
本次提交在项目的 pre‑commit 配置中新增了“Developer Certificate of Origin (DCO) 检查”。通过在 `scripts/dco_check.py` 脚本实现，对每一次提交的 commit‑msg 强制要求包含符合正则 `Signed‑off‑by: Name <email>` 的行，并在 `commit-msg` 阶段阻止不符合规范的提交。此举旨在统一代码贡献的签名规范，提升贡献者可追溯性和法律合规性。

**🎯 影响范围**  
- `.pre-commit-config.yaml`（新增 `commit-msg` 钩子）  
- `.github/filters.yaml`（CI 过滤规则中排除新脚本）  
- 新增脚本 `scripts/dco_check.py`（用于校验 commit‑msg）  
- 开发者本地提交流程与 CI 自动化检查

**🔍 技术洞察**  

- **架构影响**：  
  - 仅在仓库的 Git 工作流层面增加了 **commit‑msg** 钩子，对业务代码、编译、运行时架构无任何修改。  
  - CI 过滤文件加入对 `scripts/dco_check.py` 的忽略，防止该脚本本身触发不必要的 CI 检查。  
  - 与现有的 pre‑commit 钩子（isort、black、flake8 等）共存，使用 `default_install_hook_types` 明确钩子类型，保持配置可读性。

- **性能影响**：  
  - 脚本仅在提交时读取一次 commit‑msg，正则匹配极轻量，几乎对本地提交的响应时间没有感知。  
  - 对 CI 而言，额外的 `commit-msg` 检查不进入 CI 运行阶段，性能影响可忽略。

- **安全考虑**：  
  - DCO 并非安全漏洞修复，但通过强制签名提升了 **代码来源可追溯性** 与 **版权合规性**，有助于防止未经授权的代码引入。  
  - 脚本本身使用标准库，无外部依赖，攻击面极小；唯一需要注意的是确保脚本文件的执行权限在多平台（Linux/macOS/Windows）上一致。

**⚠️ 潜在风险**  

1. **误报/漏报**：当前正则仅接受 `Signed-off-by: Name <email>` 单行形式，若开发者使用其他合法格式（如多行、不同大小写、额外空格）会被误判为缺失。  
2. **阻塞贡献**：新钩子默认 **必选**，可能导致新手或外部贡献者因未阅读 CONTRIBUTING.md 而提交被拒，增加沟通成本。  
3. **跨平台兼容**：脚本使用 `python3` 入口，在 Windows 环境中可能需要额外的 shebang 适配或在 `pre-commit` 中指定 `language: python` 的解释器路径。  
4. **CI 过滤误配置**：`filters.yaml` 中的排除规则若写错，可能导致脚本本身触发 CI，浪费资源。

**💡 关注建议**  

- **完善正则**：考虑容忍大小写、前后空格等，或使用 `re.IGNORECASE`，并支持多行 `Signed-off-by:`（如在合并提交中出现多次）。  
- **文档同步**：在 `CONTRIBUTING.md`、`README.md` 明确说明 DCO 要求，并在 PR 模板中加入提示，降低阻塞率。  
- **可选开关**：提供环境变量或配置开关，让项目维护者可在需要时临时关闭 DCO 检查（例如在内部 CI 机器上）。  
- **跨平台测试**：在 Windows CI 环境下跑一遍 `pre-commit run --all-files`，确保脚本可执行且返回正确的退出码。  
- **回退方案**：若后期决定移除该检查，确保 `filters.yaml` 与 `pre-commit-config.yaml` 中对应条目能干净撤除，不留下残余路径。  

通过上述措施，可在提升合规性的同时，将对开发者体验的负面影响降至最低。

### feat(llm): add per-request Mooncake trace capture [DYN-3181] (#10381)
**SHA**: `fcc2e80` | 🔗 [查看提交](https://github.com/ai-dynamo/dynamo/commit/fcc2e80a79c67809da9c71922694857e96ac18af)

**🎯 变更类型**：功能增强 / 架构变更 / 安全修复  

**⚡ 重要程度**：🔴高  

**📋 变更摘要**  
- 引入 **Per‑request Mooncake Replay Trace**（`DYN_REQUEST_TRACE`）功能，能够为每个 OpenAI‑style chat/completion 请求捕获一条紧凑的 `request_end` 记录，供离线 Mooncake 重放使用。  
- 新增一套完整的 **配置 → 采集 → 发行 → sink** 流水线，包括 `request_trace` 模块、环境变量、文档、CLI 参数、JSONL/JSONL‑GZ 输出、广播总线 (TelemetryBus)。  
- 在 `llm::preprocessor` 中统一构建 **RequestEndTraceState**，支持 **agent trace** 与 **request trace** 并行工作，同时在 `--agentic` 模式下自动拒绝 `dynamo.request.trace.v1`。  
- 新增 `request_trace` 相关的 **replay hash** 复用 `agents::trace::replay_metrics`，并在 `load.rs` 中解析 `dynamo.request.trace.v1`，实现去重、 schema 校验、输入长度‑hashes 一致性检查。  
- 扩展 `telemetry::stream` 实现，确保在流被提前 drop 时先 **drop_inner** 再发送完成通知，防止潜在的资源泄漏。

## 🔍 技术洞察

### 架构影响
1. **新增独立的 Trace 总线**  
   - `static BUS: TelemetryBus<RequestTraceRecord>` 与现有 `agents::trace` 总线并行。  
   - 总线容量通过 `DYN_REQUEST_TRACE_CAPACITY` 参数可调，默认 1024，满足大多数工作负载。  
   - 采用 **广播** 方式，可同时向 `stderr`、`jsonl`、`jsonl_gz` 多 sink 发送，和 agent trace 的设计保持一致，保持整体架构的统一性与可扩展性。

2. **统一 Replay Hash 计算**  
   - `request_trace::replay_metrics` 直接复用 `agents::trace::replay_metrics`，保证 **agent trace** 与 **request trace** 产生的 `input_sequence_hashes` 完全一致，避免因实现细微差异导致的 Mooncake 重放不匹配。  
   - 通过 `Arc<AgentReplayMetrics>` 实现 **共享所有权**，在两条路径（agent + request）同时需要时避免二次计算。

3. **请求预处理层的协同**  
   - `preprocessor::build_request_end_trace_state` 同时决定是否创建 **agent trace**（依赖 `agent_context`）和 **request trace**（基于请求特性）。  
   - 对不支持的请求（多模态、`best_of>1`、`n>1`、嵌入、缺少 tracker）会在 **WARN** 级别打印并 **不生成** 记录，保持业务语义的 “一行 Mooncake” 要求。

4. **兼容性检查**  
   - `LoadedAgentTrace::ensure_agentic_compatible` 在转换 `agentic` 时阻止 `request_trace` 数据进入，防止用户误用 `--agentic` 参数导致不可预期的 schema 错误。  
   - 通过 `contains_request_trace` 标记和 `duplicate request_id` 检查，提高数据完整性。

5. **文档 & CLI**  
   - 新增 `request-tracing.md`，完整列出 env‑vars、默认值、滚动策略、示例命令；README 与 observability 文档同步更新。  
   - `bench/agent_trace_to_mooncake` CLI 明确提示 `--agentic` 不适用于 `request_trace`，降低误操作风险。

### 性能影响

|------|----------|----------|----------|
| **CPU** | `input_sequence_hashes` 与已有的 block‑hash计算相同，只在 `request_trace` 开启且满足条件时执行，额外的哈希一次（`replay_metrics`）在大多数请求上占用极低的 CPU（几十 µs）。 | 对超大输入（如 128k token）会进行一次完整的 block‑hash，耗时可达数百毫秒（测试已记录）。 | - 仅在 `trace_block_size > 0` 且 **replay enabled** 时计算；<br>- 通过 `DYN_REQUEST_TRACE` 开关可以随时关闭；<br>- `jsonl_buffer_bytes` 与 `flush_interval` 控制写入频率，减少 I/O 抖动。 |
| **内存** | 采用 **Arc** 共享 `AgentReplayMetrics`，避免复制；总线采用固定容量环形缓冲区 (capacity configurable)。 | 大量并发请求可能导致总线满溢，引发记录丢失与 `Lagged` 警告。 | - 环路容量可通过 `DYN_REQUEST_TRACE_CAPACITY` 调整；<br>- Sinks 报告 `lagged` 并在 shutdown 时尝试 flush，避免长期积压。 |
| **I/O** | 支持 `jsonl`（纯文本）与 `jsonl_gz`（压缩并滚动），对磁盘写入采用批量缓冲，显著降低 syscall 开销；默认滚动阈值 256 MiB，合理控制单文件大小。 | 若使用 `stderr` 或极小 `jsonl_buffer_bytes`，可能产生大量小写入，影响磁盘/日志系统。 | - 默认采用 `jsonl_gz`，在高吞吐环境下压缩并批写；<br>- 用户可自行调优 `buffer_bytes` 与 `flush_interval`。 |

- **环境变量泄露**：所有 `DYN_REQUEST_TRACE_*` 变量均为 **非敏感**（仅控制开启/路径），不涉及凭证。
