### feat: improve BSI by removing scheduler hop (#10200)
**SHA**: `5fb4bb2` | 🔗 [查看提交](https://github.com/ai-dynamo/dynamo/commit/5fb4bb29a0ca8809997b8f72718dbcb78c7c97e9)

**🎯 变更类型**：功能增强 / 性能优化  

**⚡ 重要程度**：🔴高  

**📋 变更摘要**：  
- 在 `ThreadPoolIndexer` 的 `AsyncShardHandle` 实现中，原本对 `find_matches_from_anchor` 的调用使用 `tokio::task::spawn_blocking` 在阻塞线程池中执行，现改为在调用者的 async 任务中直接同步执行。  
- 同时更新了文档注释，说明该路径已与 `ThreadPoolIndexer::find_matches` 的行为保持一致，省去调度开销。  
- 预计在同进程分片路由的热点路径上显著降低延迟和资源占用。

**🎯 影响范围**：  
- `lib/kv-router/src/indexer/shard_handle.rs`（`AsyncShardHandle` 实现）  
- 受影响的运行时组件：KV Router 的分片索引器（in‑process branch‑sharded）  
- 可能间接受影响的上层业务：依赖 `find_matches_from_anchor` 的查询/检索 API  

**🔍 技术洞察**：

- **架构影响**  
  - 该变更仅局限于 **ThreadPoolIndexer** 的同步实现，不涉及跨进程或网络层面的架构调整。  
  - 删除了 `spawn_blocking` 的包装，使得该方法不再依赖 Tokio 的阻塞线程池，简化了执行路径。  
  - 对外部接口（`AsyncShardHandle` trait）保持不变，保持向后兼容。

- **性能影响**  
  - **正向**：消除一次任务调度（`spawn_blocking` → 线程池）以及 Future 的 `await`/`poll` 开销，特别是在同进程、高并发的分片查询热点中，可降低数十微秒的延迟。  
  - **负向**：如果 `backend.find_matches_from_anchor` 的实现在极端情况下耗时较长（例如大规模前缀搜索或极端数据倾斜），直接在 async 任务中阻塞可能导致 Tokio 运行时的核心线程被占用，进而影响其它异步任务的调度。代码注释暗示该路径已与 `ThreadPoolIndexer::find_matches` 匹配，暗示实际工作负载相对轻量，风险有限。

- **安全考虑**  
  - 代码改动未引入新的外部依赖或 unsafe 块，安全属性保持不变。  
  - 仍然通过 `KvRouterError::IndexerOffline` 处理后端不可用的错误路径，错误传播逻辑未变。

**⚠️ 潜在风险**：

1. **阻塞风险**  
   - 若未来 `find_matches_from_anchor` 的实现被替换为更耗时的同步算法，可能导致 async 运行时的阻塞，进而出现请求延迟或线程饥饿。  
2. **错误传播**  
   - 删除 `spawn_blocking` 后，原本的 `JoinError` 被去除，错误仅来自底层 `backend.find_matches_from_anchor` 本身。若底层返回 `None` 并产生 panic，当前代码将直接传播 panic，可能导致整个任务崩溃。  
3. **回归测试覆盖**  
   - 需要确认已有的性能与负载测试覆盖了此热点路径，避免在生产环境出现意外的吞吐下降。

**💡 关注建议**：

- **监控 & 基准**：在部署后，开启针对 `find_matches_from_anchor` 的延迟监控（比如 95th/99th percentile），对比改动前后的指标，确保实际收益符合预期。  
- **防御性编码**：考虑在调用前加入 `tokio::task::yield_now().await` 或轻量级的超时包装，以在极端阻塞时防止任务长时间占用运行时线程。  
- **回滚机制**：如果后续发现阻塞导致系统级别的调度瓶颈，可通过配置开关（feature flag）恢复 `spawn_blocking` 包装，实现快速回滚。  
- **文档更新**：在项目的性能指南或贡献文档中标注此实现已是 **inline** 版本，提醒后续维护者在修改 `find_matches_from_anchor` 时评估其同步耗时。

### fix(operator): preserve available old rollout generations (#10184)
**SHA**: `ad2b8ce` | 🔗 [查看提交](https://github.com/ai-dynamo/dynamo/commit/ad2b8ce40a6627621fd45753d481a5f3b8eef770)

**🎯 变更类型**：Bug修复 / 功能增强  
**⚡ 重要程度**：🔴 高  
**📋 变更摘要**  
本次提交重构了 Rolling Update 期间旧 worker 代数的副本分配逻辑。新增 `OldWorkerReplicaTargetsByComponent`、`OldWorkerReplicaTargetsByDCD`、`NewWorkerReplicaTargetsByComponent` 三个字段，替代原来的 `OldWorkerReplicas`/`NewWorkerReplicas`。通过 `allocateOldWorkerDCDReplicas` 按 **可用副本** 与 **创建时间** 对多代旧 DCD 进行精准的 replica 目标分配，确保在滚动更新中 **保留已就绪的旧代副本**，避免因旧代被一次性裁剪至 0 而导致可用容量瞬间下降。

**🎯 影响范围**  
- `deploy/operator/internal/controller/dynamographdeployment_controller.go`  
- `deploy/operator/internal/controller/dynamographdeployment_rollingupdate.go`  
- `deploy/operator/internal/controller/dynamographdeployment_rollingupdate_test.go`  
- `deploy/operator/internal/dynamo/graph.go`、`graph_test.go`  

涉及的核心模块：  
- **Operator 控制器**（DynamoGraphDeploymentReconciler）  
- **RollingUpdateContext** 结构体  
- **DynamoComponentDeployment（DCD）** 副本调度与状态聚合  

**🔍 技术洞察**  

| 维度 | 影响描述 |
|------|----------|
| **架构影响** | - 引入更细粒度的副本目标（按组件 + 按 DCD）。<br>- 旧的 `OldWorkerReplicas` 映射被拆分，保持向后兼容只在本次提交内部使用。<br>- `allocateOldWorkerDCDReplicas` 成为副本分配的核心算法，解耦了 “旧代总副本目标” 与 “单个 DCD 副本目标” 的映射。|
| **性能影响** | - 新增一次对旧代 DCD 列表的遍历与排序（按 `CreationTimestamp`），复杂度 O(N log N)（N 为同组件旧代 DCD 数），在典型场景（每组件 1~3 代）影响可忽略。<br>- 额外的内存分配（maps、slice）同样微小。整体对控制器吞吐量影响不大。 |
| **安全考虑** | - 仅涉及调度副本数的内部计算与 Patch 操作，无外部输入直接写入。<br>- 新增字段在 `RollingUpdateContext` 中未持久化到 CR 状态，仅在内存中使用，防止误暴露。<br>- 代码在 `scaleOldWorkerDCDs` 中对缺失目标做了显式错误返回，避免误 patch 不确定副本数导致服务不可用。 |
| **可维护性** | - 通过分层 API（`getOldWorkerDCDsByComponent` → `allocateOldWorkerDCDReplicas` → `scaleOldWorkerDCDs`）实现职责单一，后续若要更改分配策略只需修改 `allocateOldWorkerDCDReplicas`。<br>- 单元测试大幅扩展，覆盖了多代旧代保留、副本分配、上下文构建等路径。<br>- 变量命名更具语义（`OldWorkerReplicaTargetsByComponent` 等），有助于阅读。 |
| **兼容性** | - 结构体字段变更会影响所有使用 `RollingUpdateContext` 的代码路径，已同步更新所有调用方及测试。<br>- 仍保持 `InProgress()` 判定逻辑，只是检查新字段的长度，兼容旧的空结构体。 |

**⚠️ 潜在风险**  
1. **排序依赖时间精度**：如果两旧代 DCD 创建时间相同（极端情况），代码会使用名称做次级排序，仍能保证确定性，但若集群快速生成同名时间戳的对象，可能导致副本分配顺序不符合预期。  
2. **旧版 Operator 升级过程**：如果集群中混入未升级的 Operator（仍使用旧字段），可能出现 `OldWorkerReplicas` 为 `nil` 而 `OldWorkerReplicaTargetsByComponent` 为空的情况，导致滚动更新被误判为不在进行中。需要确保升级过程为滚动升级或使用 `kubectl rollout restart` 触发新版 Operator 同步。  
3. **异常错误处理**：`scaleOldWorkerDCDs` 在缺失 `OldWorkerReplicaTargetsByDCD` 时会直接返回错误并中断后续 DCD 的缩放，可能导致部分旧代 DCD 未被正确裁剪，产生短期副本超额。建议在高可用场景加入重试或宽容策略。  

**💡 关注建议**  

| 受众 | 建议 |
|------|------|
| **开发者** | - 在后续特性（如跨区域滚动）中复用 `allocateOldWorkerDCDReplicas`，并保持其纯函数特性。<br>- 考虑将 `RollingUpdateContext` 中的三个 maps 合并为一个结构体 `ComponentReplicaPlan`，进一步提升可读性。 |
| **运维/使用者** | - 升级 Operator 前确认所有节点已同步新镜像，避免混用旧/新版导致滚动更新卡死。<br>- 在监控面板中关注 `rollingUpdateCtx.OldWorkerReplicaTargetsByComponent` 的变化，确保旧代副本数符合预期。 |
| **测试/QA** | - 增加边界测试：同一组件多于 3 代旧 DCD、创建时间相同、`spec` 与 `available` 差距极大等情形。<br>- 验证在高并发部署（大量 DGD）下 `allocateOldWorkerDCDReplicas` 的性能是否仍在可接受范围（建议 < 10ms）。 |
| **安全审计** | - 确认新字段未通过 CRD 公开给用户，防止外部写入导致副本数被恶意操控。<br>- 若未来计划将滚动更新策略配置化（如自定义 “保留最旧 N 副本”），务必在 API 层做输入校验。 |

**总体结论**  
此提交通过细化旧代副本目标并引入保守的分配算法，解决了滚动更新期间 **可用副本突降** 的 bug，提升了系统的可靠性和可观测性。对性能和安全的影响极低，主要风险在升级期间的兼容性和极端时间戳冲突，均可通过运维流程和进一步的单元/压测覆盖来降低。建议尽快在生产环境推进该版本的 Operator，配合相应的升级指引即可。

### fix(parser): DeepSeek V3 incomplete tool call markup for batch.5.b and stream.4.c (#10192)
**SHA**: `637dfd8` | 🔗 [查看提交](https://github.com/ai-dynamo/dynamo/commit/637dfd814b2aec99f06bfabed6b59c33687ba442)

**🎯 变更类型**：Bug修复  
**⚡ 重要程度**：🔴高  
**📋 变更摘要**：修正了 DeepSeek V3/V3.1 在流式或 batch 场景下出现的 “裸内部”工具调用标记（缺少外层包装）导致的解析错误。新增两组针对该边缘情况的测试，并在 `detect_tool_call_start` 中把这类裸调用视为“被监禁（jailed）”，从而在后续恢复阶段能够正确抽取工具调用信息且不泄漏到普通文本。  

**🎯 影响范围**：  
- `lib/parsers/src/tool_calling/parsers.rs`（工具调用检测与恢复逻辑）  
- `lib/llm/tests/test_streaming_tool_parsers.rs`（新增流式工具调用恢复单元测试）  
- 整个 **tool‑calling** 子系统以及依赖该子系统的 LLM 接口层（如 OpenAI、DeepSeek 等提供者的适配代码）  

**🔍 技术洞察**  
- **架构影响**：  
  - 该更改位于解析层，属于 **底层输入处理**，不涉及上层业务逻辑或公共 API。  
  - 引入 “jailed” 概念后，解析器在检测到裸 `tool▁call▁begin` 标记时会把它们放入临时容器，在后续 `parse_response_stream` 中统一恢复，这保持了对外统一的 `ToolCall` 结构输出，不破坏现有抽象。  
  - 对外接口（`parse_response_stream`、`detect_tool_call_start`）的返回签名未变，保持兼容性。  

- **性能影响**：  
  - 仅在检测阶段增加一次额外的判断（是否存在外层包装），时间复杂度仍为 **O(N)**（N 为输入字符数）。  
  - 对流式解析的额外状态维护极其轻量（布尔标记 + 缓存少量字符串），对整体吞吐量影响可忽略不计。  

- **安全考虑**：  
  - 正确分离工具调用与普通文本能防止 **工具调用注入**（Tool‑Injection）误判为普通聊天内容，降低潜在的代码执行或信息泄露风险。  
  - 由于解析器现在会在检测到裸调用时“监禁”它们，只有在恢复阶段确认完整后才会生成 `ToolCall` 对象，避免了因不完整标记导致的误触发。  

**⚠️ 潜在风险**  
1. **误判风险**：若其他模型（非 DeepSeek）意外生成类似的裸标记，当前逻辑可能把正常文本误当作工具调用，从而导致工具被错误执行。  
2. **回退兼容**：某些老版本 DeepSeek 客户端可能仍发送不完整的外层包装，若回退到旧的解析实现可能出现不一致行为。
