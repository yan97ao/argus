### fix(runtime): let runtime pick explicit event plane (#10021)
**SHA**: `a28c600` | 🔗 [查看提交](https://github.com/ai-dynamo/dynamo/commit/a28c60020af7741966909c9246113560ccf93695)

**🎯 变更类型**：功能增强 / Bug修复  

**⚡ 重要程度**：🔴高  

**📋 变更摘要**  
本次提交让 Dynamo Runtime 能够通过显式参数 `event_plane` 直接选择事件传输层（`nats` / `zmq`），不再依赖全局环境变量 `DYN_EVENT_PLANE`。Python 与 Rust 绑定均新增 `event_plane` 参数、相应的验证逻辑，并在文档、测试以及内部实现中统一使用该参数。  

**🎯 影响范围**  
- `components/src/dynamo/common/utils/runtime.py`（Python 运行时创建）  
- `lib/bindings/python/*`（Rust‑Python 绑定层）  
- `docs/components/frontend/configuration.md`（文档）  
- `lib/llm/src/kv_router/indexer/recovery/subscriber.rs`（KV 事件订阅器）  
- 测试套件 (`tests/router/*`, `lib/bindings/python/tests/*`)  

**🔍 技术洞察**  

- **架构影响**  
  - 将事件平面的选择权下放到 `DistributedRuntime` 实例化阶段，消除了对进程级别环境变量的写入/恢复操作，提升了运行时的可组合性和可预测性。  
  - `resolve_event_transport_kind` 在 Rust 端统一解析 `event_plane` 与 discovery backend 的默认行为，保持跨语言一致性。  
  - 通过 `explicit_event_plane` 与 `event_plane` 参数，使得 `enable_nats` 的隐式推断更明确，避免了不同组件之间因环境变量共享而产生的耦合。  

- **性能影响**  
  - 删除了在 `create_runtime` 中频繁的 `os.environ` 读取/写入，尤其在多线程或高并发创建多个 Runtime 实例的场景下，可略微降低系统调用开销。  
  - 运行时创建流程基本保持原有路径，新增的参数解析和一次性函数调用对整体启动性能影响可忽略不计。  

- **安全考虑**  
  - 通过显式参数而非全局环境变量，降低了意外泄漏或跨进程污染的风险，提升了安全性。  
  - 参数校验在 `resolve_event_transport_kind` 中完成，对非法值抛出 `ValueError`，防止注入非法 transport 错误配置。  

**⚠️ 潜在风险**  

1. **向后兼容**  
   - 旧代码如果仍依赖 `DYN_EVENT_PLANE` 环境变量进行手动覆盖，可能在新版中失效（除非通过 `event_plane` 参数显式指定）。  
   - `DistributedRuntime.__init__` 新增了关键字参数 `event_plane`，如果调用方使用位置参数且未更新签名，可能导致参数错位（但已有默认 `None`，风险有限）。  

2. **`enable_nats` 与 `event_plane` 交互**  
   - 当 `event_plane` 为 `"zmq"` 时，代码会忽略环境变量 `NATS_SERVER`，但仍会在 `nats_enabled` 判定中检查 `explicit_event_plane.is_none()`，可能在极端配置下产生误判，需要在后续版本持续监测。  

3. **测试/部署环境差异**  
   - CI 环境中仍可能残留 `DYN_EVENT_PLANE`，虽然运行时已不再使用，但文档和示例仍需同步更新，以免产生误导。  

**💡 关注建议**  

- **对接方升级**：在所有使用 `DistributedRuntime` 的地方，审查是否有基于 `DYN_EVENT_PLANE` 环境变量的自定义逻辑，改为显式传入 `event_plane` 参数。  
- **文档同步**：确保所有官方文档、示例脚本、Helm Chart 等均使用新的默认说明（`auto`），并标注不再推荐手动设置环境变量。  
- **兼容层实现**：如果仍需要支持老式环境变量，可在最外层包装函数中做一次兼容转译（例如读取 `DYN_EVENT_PLANE` 并转为 `event_plane` 参数），避免突发性破坏。  
- **监控与回归测试**：在生产部署后监控 NATS 与 ZMQ 连接情况，特别是混合使用 `file`/`mem` discovery 与 `tcp` 请求平面的场景，以验证 `event_plane` 参数的实际启用行为。  
- **代码审查**：后续对 `DistributedRuntime` 的其他构造参数（如 `enable_nats`）进行统一省略或废弃，以进一步简化 API。

### fix(snapshot): poll restore containers before pod running (#9984)
**SHA**: `738300c` | 🔗 [查看提交](https://github.com/ai-dynamo/dynamo/commit/738300cb23a6e929b1cb9e51c4446a1be742e360)

**🎯 变更类型**：Bug修复 / 功能增强（确保在 Pod 运行前即可恢复，并防止对已删除或已终止的 Pod 进行恢复）  

**⚡ 重要程度**：🔴 高  

**📋 变更摘要**  
- 在快照恢复控制器中加入对 Pod 状态的二次检查：在通过节点运行时解析容器 ID 之后，重新从 API Server 拉取最新的 Pod 对象并确认其仍处于 Pending 或 Running 且未被删除。  
- 将原有的 `if pod.Status.Phase != corev1.PodRunning` 判定放宽为 “Pending or Running”，并同步排除已删除或已进入终态的 Pod。  
- 新增单元测试覆盖 Pending、Succeeded、Failed、Unknown 等多种 Pod 相位以及在容器 ID 解析前后对 Pod 状态的轮询行为。  

**🎯 影响范围**  
- `deploy/snapshot/internal/controller/controller.go`（快照恢复核心业务逻辑）  
- `deploy/snapshot/internal/controller/controller_test.go`（对应单元测试）  
- 相关的依赖库：`k8s.io/apimachinery/pkg/api/errors`、`client-go` 对 API Server 的 GET 调用  

**🔍 技术洞察**  
- **架构影响**：  
  - 新增 `refreshRestorePodForStart`，在恢复流程中多了一次 `GET /pods` 调用，保持了 **控制层** 与 **数据层**（K8s API Server）之间的强一致性检查。  
  - 改动对整体架构影响有限，仍然在同一控制器内部完成，不会引入跨组件的耦合。  

- **性能影响**：  
  - 每次运行时解析容器 ID（即 `pollForContainerID`）后会发起一次 API 请求，极少数情况下会导致额外的网络往返。对大规模节点的累计开销仍在可接受范围（单次请求 RTT 通常 < 10 ms）。  
  - 通过提前过滤已删除/终态 Pod，减少了不必要的恢复尝试和后续的事件/卷操作，从而在极端场景下提升整体系统吞吐。  

- **安全考虑**：  
  - 防止在 Pod 已被删除或已进入终态时执行恢复，避免对已经失效的资源进行写入，降低了潜在的 **数据破坏** 与 **资源泄漏** 风险。  
  - 代码路径未引入新的权限需求，仍然使用已有的 `list/watch pods` 权限。  

**⚠️ 潜在风险**  
1. **API Server 瞬时不可用**：若在 `refreshRestorePodForStart` 中的 `GET` 调用短暂失败（网络抖动、API Server 超时），恢复流程会直接返回并放弃本次恢复，可能导致恢复被错失。  
2. **竞争窗口**：虽然在获取最新 Pod 后再次检查状态，但仍可能出现 **Pod 在检查后立即被标记为删除** 的微小竞争窗口，导致恢复任务在已删除的实例上启动。  
3. **额外的请求负载**：在高并发恢复场景下，额外的 `GET` 请求会放大对 API Server 的压力，需要监控 `apiserver` 的 QPS。  

**💡 关注建议**  
- **监控与告警**：在 `refreshRestorePodForStart` 的错误路径（非 NotFound）加入计数器或日志级别提升，监控因 API 调用失败而跳过的恢复次数。  
- **重试机制**：考虑在短暂的网络错误（如 `IsTimeout`、`IsServerTimeout`）时进行一次指数回退重试，以降低因瞬时不可用导致的恢复丢失。  
- **限流**：如果集群中恢复任务极多，可在控制器内部对 `GET /pods` 请求做本地缓存或批量查询，以减轻对 API Server 的冲击。  
- **文档更新**：在快照恢复的使用指南中明确说明：即使 Pod 处于 Pending 状态，只要容器 ID 能被解析，恢复即可触发；并提示用户在高并发恢复场景下关注 API Server 的负载。  

通过上述改动，系统在面对 Pod 删除或状态快速切换的边缘情况时能够更安全、可靠地执行快照恢复，整体可靠性提升。

### feat(kv-router): DP-rank-aware sticky session affinity (POC) (#9920)
**SHA**: `d97c889` | 🔗 [查看提交](https://github.com/ai-dynamo/dynamo/commit/d97c889baf78eb423db452d5640a500f566f09d0)

**🎯 变更类型**：功能增强 / 架构变更  
**⚡ 重要程度**：🔴高  

**📋 变更摘要**  
- 在 KV‑router 中实现 **DP‑rank‑aware sticky session affinity**（概念验证），即会话在同一个 `worker_id` 与 **同一 data‑parallel rank (`dp_rank`)** 上保持粘性。  
- 为 `session_control` 新增 `action:"bind"`，仅在路由层建立 **router‑only** 亲和，无需向后端发起 `open_session` RPC。  
- 引入 `AffinityKind`（`EngineBacked`、`RouterOnly`），并在 `StickySessionRouter` 中保存 `(worker_id, dp_rank, kind)`。  
- 新增 **WorkerEligibilityError** 与 `validate_worker_eligibility`，在选路时检查 DP‑rank 范围、允许 worker 集合、路由约束（taint）等。  
- `KvPushRouter::select_worker` 现在接受可选的 `sticky_worker` 参数，以便在粘性路径上直接使用已绑定的 `(worker_id, dp_rank)`。  
- `PrefillRouter` 在生成路径中尝试使用粘性预填 worker；若失效会回退到普通 KV 选路。  
- `AgentController` 现在返回 `AgentRouteOutcome`，其中包含 **SessionLifecycleOutcome** 与可选的 `deferred_close`，并在 `BindRequested` 时仅在路由层完成亲和绑定。  
- 文档、示例、测试、以及 mocker 辅助服务（ZMQ KV 事件发布）同步更新；`mocker` 代码被拆分到 `services` 模块，去除了旧的 ZMQ 路由 socket helpers。  
- 依赖新增：`bytes`, `rmp-serde`, `tmq`（用于 ZMQ KV 事件），以及若干 `Cargo.lock` 更新。

### feat(self-host): worker advertises non-typed metadata siblings (gh-8749) (#9707)
**SHA**: `53aa9b7` | 🔗 [查看提交](https://github.com/ai-dynamo/dynamo/commit/53aa9b777f2cf7dddf8200806c713c9248ac2fb4)

**🎯 变更类型**：功能增强、架构变更、性能优化  
**⚡ 重要程度**：🔴高  

**📋 变更摘要**  
- 为 `ModelDeploymentCard` 增加 `extra_files` 字段，用于存放模型目录中除已知 typed metadata（如 `config.json`、`tokenizer.json` 等）之外的兄弟文件（如 `preprocessor_config.json`、`vocab.json` 等）。

- 为 `CheckedFile` 添加 `basename()` 方法，以统一获取本地路径或 URL 的文件名。

- `lib/llm/src/local_model.rs`（模型加载、metadata 广播）

**🔍 技术洞察**  

- **架构影响**  
  - **元数据广播层扩展**：新增 `extra_files` 让 worker 能在自托管模式下完整呈现模型目录结构，兼容外部预处理器（vLLM、sglang 等）对非 typed sibling 的加载需求。  
  - **MDC 计算一致性**：通过对 extra files 按 `(basename, checksum)` 排序后再哈希，确保不同 worker（读取顺序不同）产生相同 mdcsum，防止前端缓存错误复用。  
  - **模型卡结构膨胀**：`ModelDeploymentCard` 现在持有可变长的 `extra_files` 向量，增加了序列化/反序列化的复杂度，但影响微乎其微（仅在自托管路径存在时才会有内容）。  

- **性能影响**  
  - **目录遍历成本**：`harvest_extra_files` 在模型加载阶段做一次非递归 `read_dir`，对每个文件执行 `is_file`、权重检测和 `CheckedFile::from_disk`（读取文件大小、计算 checksum）。这在模型目录包含大量大文件时会产生 I/O 开销，但仅在首次加载/缓存生成时执行，后续运行时使用已缓存的 MDC。  
  - **哈希计算开销**：在 `mdcsum` 中对 extra files 进行排序并追加到哈希流，额外的 O(N log N) 排序成本（N 为 extra 文件数），通常 N 很小（几个位于模型根目录的 JSON/YAML），对整体启动延迟影响可忽略。  
  - **内存占用**：`CheckedFile` 保存路径、大小、checksum 等信息，额外几百字节的结构体乘以少量文件，对内存影响微不足道。  

- **安全考虑**  
  - **路径泄露**：`CheckedFile::basename` 仅返回文件名，不暴露完整路径，且 `extra_files` 在序列化时仅携带 checksum 与 basename，避免泄露磁盘结构。  
  - **Symlink 处理**：`harvest_extra_files` 使用 `path.is_file()`（跟随 symlink）并显式保留 symlink 指向的实际文件，以兼容 HF 快照中的符号链接。未对 symlink 循环进行检测，但在 HF 快照中不存在此类循环，风险低。  
  - **Checksum 可信度**：仍依赖 `b3sum` 生成的校验和，未引入新的哈希算法，安全属性保持不变。  

**⚠️ 潜在风险**  

1. **I/O 高峰**：首次加载拥有大量非 typed 文件的模型目录时，`harvest_extra_files` 可能产生一次性磁盘读取和 checksum 计算，导致短暂的启动延迟。  
2. **Symlink 安全**：若模型目录被恶意构造出现指向外部敏感文件的 symlink，`harvest_extra_files` 会跟随并将其加入 `extra_files`，可能导致意外的文件暴露。  
3. **哈希冲突误判**：虽然已按照 `(basename, checksum)` 排序，仍有极小概率不同内容但同名文件产生相同 checksum（理论上不可能，但依赖 b3sum 的碰撞抗性）。  
4. **向后兼容**：旧版 worker 未实现 `extra_files`，前端仍会请求这些文件时返回 404。需要确保新版前端在缺失时能容错。  

**💡 关注建议**  

- **监控启动时长**：在生产环境中监控模型加载的时延，若出现异常增长，可考虑在 `harvest_extra_files` 中加入并行 checksum（如 rayon）或缓存已算过的 checksum。  
- **Symlink 安全强化**：在 `harvest_extra_files` 增加对 symlink 目标路径的根目录校验（仅接受位于模型根目录或其子目录内的链接），防止路径穿透。  
- **文档与兼容性**：更新自托管文档，说明新增的 `extra_files` 端点以及在旧 worker 上可能出现的 404 情况；在前端实现中对 404 做容错处理。  
- **Cache 键扩展**：确认前端缓存键已包含 `extra_files` 的 hash（已通过 mdcsum），避免因忽略导致缓存误用。  
- **测试覆盖**：保持现有单元/集成测试，并在 CI 中加入包含大量 symlink、深层目录、异常文件名的压力测试，确保 `harvest_extra_files` 的健壮性。
