### fix: Improve active sequences request expiration (#7340)
**SHA**: `1e261db` | 🔗 [查看提交](https://github.com/ai-dynamo/dynamo/commit/1e261dbd3b3686e176619f6620200055a303317a)

**🎯 变更类型**：Bug修复 / 性能优化  
**⚡ 重要程度**：🔴高  
**📋 变更摘要**：  
1. 通过为每个请求记录时间戳并引入检查频率，修正了单个工作者在长时间未收到新请求时无法触发过期清理的缺陷。  
2. 在多工作者层面新增周期性强制过期任务（默认 60 s），确保所有工作者的陈旧请求都会被及时回收。  
3. 相应地更新了单元测试，以验证新的过期逻辑在模拟时间流动下的行为。

**🎯 影响范围**：  
- `lib/kv-router/src/sequences/single.rs`（ActiveSequences 单工作者实现）  
- `lib/kv-router/src/sequences/multi_worker.rs`（ActiveSequencesMultiWorker 多工作者实现）  
- `lib/llm/src/kv_router/sequence.rs`（创建多工作者实例的入口）  
- 相关单元测试

**🔍 技术洞察**：

- **架构影响**  
  - 采用 **内部时间戳 + 定期全局检查** 的方式，摆脱了原先依赖 `expiry_timer` 与 `expiry_requests` 集合的单工作者驱动模型。  
  - 引入 `start_periodic_force_expiry_across_all_workers`，在 `Arc<ActiveSequencesMultiWorker>` 上启动后台任务，利用 `CancellationToken` 实现优雅关闭，保持原有 `Arc` + `RwLock` / `DashMap` 的并发模型不变。  
  - 新增 `FORCE_EXPIRE_REQUESTS_ACROSS_ALL_WORKERS_INTERVAL`（60 s） 常量，统一调度周期。

- **性能影响**  
  - **正向**：过期检查不再依赖 `add_request` 的调用路径，避免在高负载、请求稀疏的场景下堆积大量已失效的请求占用内存。  
  - **负向**：每分钟遍历所有工作者并遍历其 `request_timestamps`，在工作者数量极大且活跃请求多时会产生一定的 CPU 与锁竞争开销。总体预计在常规集群规模（几十至几百工作者）下影响可忽略。  
  - 新增的 `request_timestamps` HashMap 替代了原来的 `expiry_requests`，在每次 `add_request` 与 `free` 时进行一次插入/删除，开销与原实现相当。

- **安全考虑**  
  - 代码均使用 `tokio::time::Instant`（基于单调时钟），不受系统时间回调影响，避免因时间跳变导致误删或永久保留请求。  
  - 任务通过 `CancellationToken` 控制生命周期，防止在服务关闭后仍有后台任务持有 `Arc` 导致资源泄漏。  
  - 无新增外部输入或权限检查，安全风险极低。

**⚠️ 潜在风险**：

1. **并发争用**：`force_expire_requests_across_all_workers` 在遍历所有工作者时持有每个工作者的写锁，若后台任务与业务线程频繁竞争锁，可能导致短暂的请求添加或调度延迟。  
2. **资源泄漏**：如果调用方未在服务 shutdown 时取消 `expiry_cancel`（子 `CancellationToken`），后台任务会持续运行，导致 `Arc<ActiveSequencesMultiWorker>` 不能被回收。  
3. **误配置**：`FORCE_EXPIRE_REQUESTS_ACROSS_ALL_WORKERS_INTERVAL` 与 `CHECK_EXPIRY_FREQUENCY` 若被改动为非常短的间隔，可能导致频繁遍历，增加 CPU 使用。  
4. **测试依赖单调时钟**：生产环境若混用 `std::time::Instant` 与 `tokio::time::Instant`（在非 `tokio` runtime 中），可能出现行为差异。

**💡 关注建议**：

- **监控与报警**：在运行时记录 `force_expire_requests_across_all_workers` 的执行时长及 `removed_request_count`。若出现异常增长（例如每次都清除大量请求），应检查业务侧是否出现异常长时间未使用的请求。  
- **Graceful Shutdown**：确保在服务退出前调用 `expiry_cancel.cancel()`（或等价的 `child_token` 取消），以避免后台任务持有 Arc 导致进程卡死。  
- **配置审查**：默认的 60 s 周期和 30 s 检查频率已针对大多数工作负载调优，除非有明确的业务需求，否则不建议随意修改。  
- **性能评估**：在极大规模部署（> 1k 工作者）时，可在压测环境模拟 `force_expire_requests_across_all_workers` 的 CPU 使用，必要时考虑分片或分批过期而不是一次性遍历全部工作者。  
- **代码审计**：确认所有对 `ActiveSequences` 的调用路径仍然在恰当时机更新 `request_timestamps`（已在 `add_request` 与 `free` 中实现），防止出现“漂移”导致遗漏清理。  

总体而言，此次改动显著提升了请求过期的可靠性，避免了因工作者长时间空闲导致的内存泄漏或资源占用，对系统的可用性与可维护性都有积极意义。只要在部署时留意上述并发与关闭细节，即可安全上线。

### feat: Enable EPP worker discovery mode  (#6592)
**SHA**: `a441aaf` | 🔗 [查看提交](https://github.com/ai-dynamo/dynamo/commit/a441aaf8ea3ff1b9b2928df91fc42e7a239a00fb)

**🎯 变更类型**：功能增强  

**⚡ 重要程度**：🔴高  

**📋 变更摘要**  
1. 为 EPP（Inference‑Gateway）加入 **worker discovery mode**，允许路由器在运行时通过请求头部获得可用的 pre‑fill / decode worker ID，而不再依赖启动时的 discovery 同步。  
2. 新增 `PrefillWorkerIDState`、`PrefillWorkerIDStateKey`，在 `DynPrefillScorer` 中直接写入 `x-prefill-instance-id` 请求头，并同步到 `CycleState`；`DynDecodeScorer` 读取该头部并记录日志。  
3. 引入 `skip_initial_worker_wait` 配置、外部 worker 注册 API（`register_workers`、`register_external_workers`），并在 KV‑router 初始化、调度器、队列、slot‑tracker 中相应改造，实现 **外部提供 worker 列表** 的懒注册。  
4. 统一使用 `DYN_NAMESPACE_PREFIX` 环境变量、默认关闭 `DYN_ENFORCE_DISAGG`，并在 docs、helm、例子中同步更新使用说明。  

**🎯 影响范围**  
- **EPP 插件**：`disagg/prefill_scorer.go`、`disagg/decode_scorer.go`、`disagg/shared.go`。  
- **Operator**：`component_epp.go`（默认 env 注入）。  
- **KV‑router 核心库**：`lib/kv-router/src/*`（scheduler、queue、multi_worker、config、routing）。  
- **Rust 绑定**：`lib/bindings/c/src/lib.rs`、`lib/bindings/python/...`。  
- **文档 & 示例**：`docs/kubernetes/inference-gateway.md`、`examples/.../agg.yaml`、`recipes/.../deploy.yaml`。  
- **测试**：`tests/router/test_router_e2e_with_mockers.py`。  

**🔍 技术洞察**  

- **架构影响**  
  - **外部发现模式**：路由器不再强依赖 K8s discovery 在进程启动时完成 worker 列表的收集，而是接受来自 EPP 的 `allowed_worker_ids`（prefill/解码 workers）。这把 worker 生命周期的所有权从 router 移交给 EPP，适配 GAIE/standalone 部署的 “外部调度” 场景。  
  - **状态交互**：通过 `PrefillWorkerIDState` 将 prefill worker ID 从 `DynPrefillScorer` 传递给 `DynDecodeScorer`，避免依赖 `CycleState` 跨 profile 的传播限制。  
  - **配置扩展**：`skip_initial_worker_wait` 使 KV‑router 可以在无 workers 的情况下直接启动；在 `create_routers` 中加入 `init_preprocessor` 步骤，统一处理 namespace‑rolling‑hash、模型卡获取等。  
  - **插件链路**：`DynPrefillScorer` 现在负责写 Header 与 CycleState，`DynDecodeScorer` 只读取 Header，解耦了两者的依赖。  

- **性能影响**  
  - **启动时延**：`skip_initial_worker_wait=true` 移除启动阻塞，EPP 启动更快；但首次请求若未及时注册 workers，可能出现 “NoEndpoints” 错误。  
  - **运行时开销**：每次请求在 `queue.enqueue` 前若携带 `allowed_worker_ids`，会直接调度而跳过 `all_workers_busy` 检查，降低排队延迟。  
  - **锁竞争**：`register_external_workers` 与 `register_workers` 在 `slots` 与 `queue` 上使用 `RwLock`/`RefCell`，在高并发场景下可能产生轻微锁竞争，但整体影响有限（原有 `update_workers` 已经持有写锁）。  
  - **日志与 tracing**：新增大量 `tracing::info!`/`debug!`，对生产环境可通过 `RUST_LOG` 控制，不会显著影响吞吐。  

- **安全考虑**  
  - **Header 可篡改**：`x-prefill-instance-id` 由内部 scorer 设置，外部客户端仍可自行构造同名 header；当前实现只在 **prefill** 轮次后才检查 header 是否存在，若缺失会记录错误并继续（仍会走 decode 但可能走错误 worker）。建议在前置过滤（如 webhook）或在 `DynDecodeScorer` 添加校验：仅接受已在 `PrefillWorkerIDState` 中登记的 ID。  
  - **DISAGG 强制**：默认将 `DYN_ENFORCE_DISAGG=false`，在没有 prefill worker 时请求会回退到 decode‑only。若业务要求严格的 prefill/ decode 分离，需显式打开该开关，否则可能导致请求在未准备好 prefill worker 时被误路由，产生潜在的服务降级或 DoS。  
  - **Namespace 泄露**：新增 `DYN_NAMESPACE_PREFIX` 直接暴露 Dynamo 使用的命名空间前缀，若在多租户环境中未做好 RBAC，可能被用于枚举或推断内部资源。  

**⚠️ 潜在风险**  

| 风险点 | 可能影响 | 触发条件 |
|---|---|---|
| **首次请求无可用 workers** | `NoEndpoints` 错误、请求 5xx | `skip_initial_worker_wait=true` 且 worker 注册延迟 > 第一次请求到达 |
| **Header 欺骗** | 误将请求路由到错误的 prefill worker，导致缓存失效或负载不均 | 外部客户端自行注入 `x-prefill-instance-id` |
| **锁竞争** | 高并发下调度延迟略增 | 大规模并发请求同时调用 `register_workers` |
| **滚动更新的 namespace 后缀** | 若代码未正确匹配前缀，可能找不到模型卡导致初始化失败 | 使用 rolling‑update 时 namespace 变为 `my-ns-2c5f` 等

### feat: vllm omni cli (#6788)

**🎯 变更类型**：功能增强 / 重构 / 架构变更  
**⚡ 重要程度**：🔴 高  

**📋 变更摘要**  
1. 将 vLLM‑Omni 的功能从原有的 `dynamo.vllm` 主入口中抽离，创建独立的子模块 `dynamo.vllm.omni` 并提供全新入口 `python -m dynamo.vllm.omni`。

3. 为 Omni 引入专属的参数组（`OmniArgGroup`）、配置结构 (`OmniConfig`) 与启动流程 (`omni/main.py`），并实现对应的 `OmniHandler`、`BaseHandler` 调整以及单元测试。  

**🎯 影响范围**  
- **核心模块**：`components/src/dynamo/vllm`（`args.py`, `backend_args.py`, `main.py`, `health_check.py`）  
- **新增子模块**：`components/src/dynamo/vllm/omni`（`__main__.py`, `args.py`, `base_handler.py`, `main.py`）  
- **文档**：`docs/backends/vllm/vllm-omni.md`、示例启动脚本（`examples/backends/vllm/launch/*.sh`）  
- **测试**：`components/src/dynamo/vllm/tests/omni/test_omni_args.py`（新增）

## 🔍 技术洞察

| 维度 | 影响描述 |

| **性能影响** | - **正向**：Omni 专用参数（层级卸载、GPU 并行度、缓存加速等）在 `AsyncOmni` 构造时直接传递，降低了在通用代码路径中无用参数的解析开销。<br>- **潜在负面**：新增的 `OmniHandler` 引入对 `DiffusionParallelConfig` 的可选导入，若未装载对应依赖会触发 `logger.warning`，但不影响主流程。<br>- **启动时**：分离后启动脚本需要额外的 `python -m dynamo.vllm.omni` 进程，略增系统资源占用（一个进程对应一个 GPU）。 |

| **可维护性** | - **代码清晰度**：Omni 专属的 `args.py` 与 `main.py` 将业务逻辑与原 vLLM 代码解耦，后续迭代（如新 diffusion 参数）只需在单一子模块中修改。<br>- **向后兼容**：通过删除 `--omni` 标志，避免用户在不需要 Omni 时误开启，降低误配置风险。<br>- **文档同步**：CLI 文档已改为列出独立的 Omni 参数表，示例脚本也同步使用新入口。 |

## ⚠️ 潜在风险

1. **兼容性破坏**  
   - 过去使用 `python -m dynamo.vllm --omni` 的部署将失效，需手动迁移到 `python -m dynamo.vllm.omni`。未更新的脚本或 CI 可能导致服务不可启动。  
2. **依赖缺失**  
   - `omni` 子模块依赖外部的 `vllm_omni` 包（`AsyncOmniEngineArgs`、`DiffusionParallelConfig`），若用户环境未安装对应 wheel，启动会在导入时抛出 `ImportError`。  
3. **配置同步风险**  
   - `DynamoVllmConfig` 中已移除 `omni`、`stage_configs_path` 等字段；但如果用户仍在自定义 `yaml` 或环境变量中保留这些键，`Config.from_cli_args` 可能忽略导致暗藏配置不生效。  
4. **健康检查路径**  
   - `VllmHealthCheckPayload` 与 `VllmPrefillHealthCheckPayload` 仍被导入，但 `VllmOmniHealthCheckPayload` 已从主入口中剔除。若外部监控仍尝试访问 `/health/omni` 将得到 404。  
5. **日志与监控**  
   - `setup_metrics_collection` 在两个入口中重复调用，可能导致同名指标在 Prometheus 中冲突（尤其在同一进程中启动多个组件时）。

**SHA**: `9df692c` | 🔗 [查看提交](https://github.com/ai-dynamo/dynamo/commit/9df692c1046b0b20634452bed57ebaed32e98d2e)

**🎯 变更类型**：功能增强（新增基于 flock 的失效转移锁，实现引擎 Leader Election）

2. 实现 `FlockFailoverLock`，基于 POSIX `flock(LOCK_EX)` 对共享文件进行互斥，配合 `asyncio` 的非阻塞轮询，实现跨容器/进程的 leader 选举与自动失效恢复。  
3. 更新 `setup.py` 将新模块纳入打包，添入对应的 source‑tree 映射。

- **核心模块**：`gpu_memory_service`（引擎启动、故障转移逻辑）  
- **新子包**：`gpu_memory_service.failover_lock`、`gpu_memory_service.failover_lock.flock`  
- **构建/发布**：`setup.py` 中的扩展模块列表  
- **测试套件**：`tests/fault_tolerance/gpu_memory_service/test_failover_lock.py`  

**🔍 技术洞察**  

| 维度 | 影响分析 |
|------|----------|
| **架构影响** | • 通过文件系统实现分布式锁，去除了原有可能依赖的外部协调服务（如 etcd、Redis）或 side‑car 进程，减小了部署复杂度。<br>• 锁的拥有者信息写入同一文件，提供可观测性，`owner()` 可被任意进程读取，实现“无状态” leader 选举。<br>• 仍然依赖同一 *emptyDir*（或其他共享卷）在容器间可见，若部署在多节点（跨节点）环境，需要额外的共享存储或改用网络锁。 |
| **性能影响** | • `flock` 是内核级原语，获取/释放几乎是 O(1) 系统调用，延迟极低。<br>• 采用 `LOCK_NB` + `asyncio.sleep(poll_interval)` 的轮询方式，默认 100 ms，CPU 开销可忽略；在高争用情况下可调 `poll_interval` 以降低循环频率。<br>• 与基于网络协议的锁相比，省去网络 RTT 与序列化开销，提升故障转移响应速度（毫秒级）。 |
| **安全考虑** | • 只要文件路径受限且仅容器内部可写，普通用户无法劫持锁；但若共享卷被恶意容器挂载，可能出现 **锁劫持**（占用锁后不释放）。<br>• `flock` 在 NFS 等网络文件系统上行为不一致，可能导致 **锁漂移** 或 **死锁**，需在文档中明确限制仅在本地文件系统（ext4、xfs、tmpfs）使用。<br>• 锁文件内容未经验证，`owner()` 仅作观测，不影响锁的实际所有权，安全风险低。 |
| **可维护性** | • 清晰的抽象层 (`FailoverLock`) 便于未来实现其他后端（Redis、etcd）。<br>• 代码量适中，遵循 PEP‑8 与 asyncio 风格，易于审查。<br>• 测试覆盖率高，包含跨进程竞争与进程异常恢复，降低回归风险。 |
| **兼容性** | • 仅在 POSIX 平台可用（Linux、macOS），Windows 环境下 `fcntl.flock` 不可用，需要在 CI / 文档中标记。<br>• Python 3.8+（使用 `int | None` 注解），符合项目当前的最低版本要求。 |

**⚠️ 潜在风险**  
1. **文件系统依赖**：若部署在 NFS、GlusterFS、s3‑fuse 等不保证本地锁语义的卷上，`flock` 可能失效或产生不可预期的竞争行为。  
2. **误用 `acquire`**：如果同一进程并行调用 `acquire`（例如在不同协程中），会出现 *双重获取*——虽然 `flock` 本身是可重入的，但业务层可能误以为已获得独占角色，导致状态不一致。  
3. **轮询频率**：极低的 `poll_interval`（如 1 ms）在高并发争用时会导致大量系统调用，提升 CPU 使用。需要做好默认值与可调参数的文档说明。  
4. **资源泄漏**：异常路径中若 `os.close(fd)` 未执行（如在 `owner()` 中抛出未捕获异常），可能导致文件描述符泄漏，进而阻塞后续 lock。当前实现已在 `acquire` 的异常分支关闭 fd，但 `release` 前的异常未完全覆盖。  
5. **跨语言/跨进程可见性**：其他语言（如 C++）进程若不使用同一 `flock` 语义，可能出现 “锁已释放但文件仍残留 owner 信息”的误导。  

**💡 关注建议**  
1. **文档明确限制**：在 README/部署指南中声明仅在本地 POSIX 文件系统（如 `emptyDir`、`tmpfs`）上使用；若需要跨节点，请考虑实现基于 etcd/Redis 的后端。  
2. **参数化 `poll_interval`**：提供环境变量或配置项，让运维可根据争用程度调节轮询间隔，避免极端 CPU 消耗。  
3. **防御性编程**：在 `release` 中加入 `try/except` 捕获 `OSError`，确保即便 `owner()` 读取文件时出错也能安全关闭 fd。  
4. **监控 & 警报**：利用日志（已有 `logger.info`）配合集群监控，监测 lock file 的频繁切换或异常长时间未释放的情况，以提前发现锁劫持或进程僵死。  
5. **兼容性测试**：在 CI 添加 NFS 挂载的可选测试（预期失败），确保未来的代码改动不会意外在不支持的文件系统上通过。  
6. **后备实现**：保留 `FailoverLock` 抽象的接口，实现一个基于 `redis` 或 `etcd` 的后端，以便在需要跨机器选举时平滑迁移。
