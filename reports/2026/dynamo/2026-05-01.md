### fix(profiler): enforce totalGpus budget as hard cap in rapid mode DGD generation (#8617)
**SHA**: `f35bdc3` | 🔗 [查看提交](https://github.com/ai-dynamo/dynamo/commit/f35bdc3b0bc75e28319236a2d1bc5edcfad50240)

**🎯 变更类型**：Bug修复  
**⚡ 重要程度**：🔴高  
**📋 变更摘要**：  
- 在 Rapid 模式下生成 Dynamo Graph Deployment Request（DGD）时，新增对 `total_gpus_needed` 的硬上限校验，确保实际使用的 GPU 数不超过任务配置的预算。  
- 为此引入 `clamp_total_gpus_to_budget`、`get_requested_total_gpus` 与 per‑instance GPU 解析逻辑，并在 `set_multinode_config` 中改为基于单实例需求计算多节点布局。  
- 同时调整单元测试与资源分配顺序，以验证新逻辑在不同并行配置（DP/TP/PP）下的正确行为。

**🎯 影响范围**：  
- `components/src/dynamo/profiler/rapid.py`（Rapid DGD 生成入口）  
- `components/src/dynamo/profiler/utils/config.py`（GPU 预算校验、per‑instance GPU 解析、multinode 配置）  
- `components/src/dynamo/profiler/utils/config_modifiers/protocol.py`（资源分配调用顺序）  
- `components/src/dynamo/profiler/tests/unit/test_profiler_protocol.py`（相关单元测试）  

**🔍 技术洞察**  

- **架构影响**：  
  - 该改动仅限于 Profiler 子系统的 Rapid 模式，属于业务逻辑层面的约束强化，不涉及核心调度或底层通信协议的改动。  
  - 引入的 `clamp_total_gpus_to_budget` 为通用工具函数，若后续其他模块需要对 GPU 预算进行硬上限检查，可直接复用，提高代码可复用性。  
  - `set_multinode_config` 现在依据 **单实例** 所需 GPU（TP × PP）决定多节点拆分，避免因复制数导致的误判，提升部署拓扑的真实性。

- **性能影响**：  
  - 额外的函数调用与一次整数比较的开销微乎其微，对 DGD 生成的整体延迟影响可忽略不计。  
  - 通过提前在 Rapid 路径捕获超预算请求，能够避免后续不必要的算子展开与资源分配步骤，间接提升整体工作流的响应速度。

- **安全考虑**：  
  - 只涉及日志输出 (`logger.warning`) 与业务约束，无新增网络交互或文件 I/O，安全风险极低。  
  - 通过强制上限防止用户请求超出可用 GPU，间接降低因资源争用导致的服务失效或异常行为的可能性。

**⚠️ 潜在风险**  

1. **业务可行性隐藏**：  
   - 当 `total_gpus_needed` 被强制 clamp 后，Rapid 仍会继续生成 DGD，可能导致后续阶段（如实际部署）出现**资源不足**的错误，只留日志警告不一定被运维人员注意。  
   - 若上层调用者期望 Rapid 阶段直接抛错而不是继续，可能需要额外的错误码或异常传播。

2. **兼容性**：  
   - 新增的 `get_requested_total_gpus` 与 `clamp_total_gpus_to_budget` 对输入做了容错处理（返回 `None`），但若外部代码误依赖旧的 `total_gpus` 被直接修改的行为，可能出现意料之外的默认值（`original_total_gpus`）。  
   - `set_multinode_config` 的逻辑改为使用 **per‑instance** GPU 计数，可能导致已有的自定义 `multinode` 配置在极端情况下被覆盖或误判，需要确认未破坏已有的手动多节点策略。

3. **日志噪声**：  
   - 对每一次被 clamp 的请求都会产生 warning，若在大规模实验中频繁出现超预算情况，日志量可能激增，影响可观测性平台的存储和搜索成本。

**💡 关注建议**  

- **异常可视化**：建议在 Rapid 入口（`_generate_dgd_from_pick`）在 `was_clamped` 为 true 时附加显式的 **错误码/标记**（例如返回 `RapidGenerationResult` 中的 `budget_violation=True`），让调用方能够决定是继续还是终止。  
- **监控与告警**：在日志系统中为 `clamp_total_gpus_to_budget` 产生的 warning 配置专门的告警阈值，以免在生产环境中遗漏资源不足的风险。  
- **文档更新**：在 Profiler / Rapid Mode 的使用手册中补充说明：*“total_gpus_needed 仅作为软提示，实际部署将受 DGDR 硬预算限制”。*  
- **回归测试**：在 CI 中加入覆盖 **高并行度（TP/PP）+ 多节点** 场景的集成测试，确保 `set_multinode_config` 在各种组合下均能得到预期的 `nodeCount`。  
- **代码审计**：检查项目中是否还有直接读取/修改 `TaskConfig.total_gpus` 的路径，确认它们在异常情况下也能保持 `original_total_gpus` 的恢复，以防泄漏状态。  

---  

*综上，此次提交通过在 Rapid 模式强制 GPU 预算上限，提升了系统的资源安全性与配置一致性；风险主要集中在错误可见性和多节点配置的兼容性，建议通过异常标识、监控告警以及文档说明来进一步降低潜在影响。*

### feat(mm): LRU media cache in Rust frontend MediaLoader (#8863)
**SHA**: `27c0f4b` | 🔗 [查看提交](https://github.com/ai-dynamo/dynamo/commit/27c0f4bb48011ff7911f3441f6a715979379113b)

**🎯 变更类型**：功能增强  
**⚡ 重要程度**：🔴高  
**📋 变更摘要**：在 Rust 前端的 `MediaLoader` 中引入基于字节预算的 LRU 缓存，缓存已解码并已向 NIXL 注册的媒体描述符。缓存键为 URL（或 data‑uri）哈希，大小通过环境变量 `DYN_MULTIMODAL_LOADER_CACHE_GB` 控制，默认关闭。新增 `lru` 依赖及相应单元测试，验证命中、失效及预算驱动的 LRU 行为。  

**🎯 影响范围**：  
- `lib/llm/src/preprocessor/media/loader.rs`（核心实现）  
- `Cargo.toml` / 各子 Crate 的 `Cargo.lock`（新增 `lru` 依赖）  
- `lib/llm` 及其依赖的 Python / KVBM bindings（因共享 workspace 依赖）  
- 单元测试模块 `tests`（增加缓存相关测试）  

---

**🔍 技术洞察**  

- **架构影响**  
  - 在原有的 **fetch → decode → NIXL‑register** 流程前加入 **缓存层**，形成 “查缓存 → 命中直接返回 → 未命中继续网络/解码” 的短路路径。  
  - 缓存实现为 `Arc<Mutex<LoaderCache>>`，在 `MediaLoader` 实例间共享，同一进程内部的并发请求能够复用已缓存的 descriptor。  
  - 通过 `DataType::UINT8`（当前仅支持）计算 descriptor 的原始字节大小，实现 **字节预算驱动的容量管理**，而不是单纯的条目数限制。  
  - 引入 `parking_lot::Mutex` 代替标准库锁，可减少竞争开销，适配高并发的 inference 调用场景。  

- **性能影响**  
  - **正向**：重复请求同一图片时可省去网络抓取与解码，显著降低 CPU、IO 与延迟（尤其在多轮对话中同一图片多次出现的情况）。  
  - **负向**：每次 `fetch_and_decode_media_part` 需获取一次全局锁并计算哈希，带来极小的同步开销；在极端高并发且缓存命中率低的场景下可能略微增加延迟。  
  - **内存**：缓存占用受 `DYN_MULTIMODAL_LOADER_CACHE_GB` 限制，超出预算的最旧条目会被驱逐，避免 OOM。  
  - **二进制体积**：新增 `lru 0.12` 依赖，约增加 30 KB（不影响运行时）。  

- **安全考虑**  
  - 缓存内容来自外部 URL，仍然是 **解码后的原始字节**，并且在后续使用前会再次交给 NIXL 注册，不会直接暴露给用户代码。  
  - 未对 URL 做白名单过滤，仅依赖已有的 `MediaFetcher` 检查（IP、端口、私有网段等），缓存层不改变这些安全策略。  
  - 哈希键采用 `DefaultHasher`（SipHash），防止恶意构造的碰撞攻击导致缓存失效或占满。  

---

**⚠️ 潜在风险**  

1. **内存估算错误**  
   - `descriptor_bytes` 只考虑 `UINT8` 的元素大小，若未来支持其他 `DataType`（如 `FLOAT32`），需同步更新计算逻辑，否则预算失准导致 OOM。  

2. **并发锁争用**  
   - 所有请求共享同一个 `Mutex<LoaderCache>`，在极端高并发（数千并发请求）下可能出现锁竞争。可考虑细化为分段锁或使用 `dashmap` 替代。  

3. **缓存失效导致行为变化**  
   - 当 `media_io_kwargs`（ per‑request decoder 参数）非 `None` 时会绕过缓存，以保持正确性。但若业务在未来引入默认的 decoder 参数，可能误以为缓存命中导致结果不一致。  

4. **环境变量误配**  
   - `DYN_MULTIMODAL_LOADER_CACHE_GB` 若被设置为极大值（如 `1000` GiB），在资源受限的容器中仍会尝试分配相应的字节计数，虽然实际内存使用受 LRU 约束，但计数本身是 `u64`，不会导致直接 OOM，只是可能导致频繁的 eviction。  

5. **测试依赖 NIXL**  
   - 单元测试在没有可用 NIXL agent 时会提前 `return`，这意味着在 CI 环境中若未启动 NIXL，缓存相关的测试会被跳过，可能掩盖回归问题。  

---

**💡 关注建议**  

- **对外文档**：在项目的配置章节补充 `DYN_MULTIMODAL_LOADER_CACHE_GB` 环境变量的说明，包含单位、默认行为以及推荐预算范围（如 `0.5~2 GiB`）。  

- **监控 & 可观测性**  
  - 在 `MediaLoader::new` 已有 `tracing::info/debug`，建议进一步暴露 **cache 命中率**、**当前字节使用** 与 **eviction 次数** 的 metrics，便于运营评估缓存效益。  

- **扩展性**  
  - 为 future‑proof，预留 `descriptor_bytes` 对其它 `DataType` 的实现路径，或者在 `LoaderCache::put` 前做 `assert!` 检查已支持的 dtype。  
  - 如果计划缓存非图片媒体（video/audio），需重新评估 **字节预算** 与 **生命周期**（例如流式数据），并可能需要按 **时长** 而非字节数进行淘汰。  

- **性能调优**  
  - 在高并发部署场景下，可通过 **分片缓存**（如基于 URL 前缀的多个 `LoaderCache`）或改用 **lock‑free map**（`dashmap`）降低锁竞争。  
  - 考虑将 `cache_key` 计算提前到 `MediaFetcher` 层，复用已经解析的 URL 对象，进一步削减哈希成本。  

- **安全审计**  
  - 确认 `lru` crate 版本 `0.12.5` 已通过 Cargo audit，无已知 CVE。  
  - 若计划在多租户环境中共享同一进程，需评估缓存的 **跨租户泄露风险**（当前键基于 URL，理论上不会混淆，但可在缓存键前加租户前缀以确保隔离）。

  - **GPU Memory Service**：每次重试时多调用一次 `torch.cuda.mem_get_info`（一次显存查询）以及一次日志写入。显存查询在 CUDA 驱动层面开销极低（微秒级），日志在 DEBUG 级别时才记录，默认 WARN 级别仍会写入一次警告信息，整体对吞吐量影响可接受，且显著提升故障诊断效率。

## ⚠️ 潜在风险

1. **忘记清理**：如果模型卸载路径未调用 `clear_self_hosted_artifacts`（如异常退出、强制 kill），注册表可能残留无效路径，引起 404/错误日志。  
2. **并发写入**：模型并行加载时会同时写入同一 `MetadataArtifactRegistry`，虽然 `RwLock` 能保证安全，但大量并发写入可能出现写锁竞争，短暂阻塞注册过程。

5. **兼容旧客户端**：部分第三方工具直接读取 `CheckedFile.path`（假设本地路径），在自托管开启后会得到 URL，若这些工具不支持 URL 可能报错。

  - **解耦循环**：原本 `ManagedThread` 共享请求处理的事件循环，导致发布者轮询与请求处理竞争同一线程的 GIL 与 I/O 资源。现在每个线程拥有独立的循环并在 `run()` 中自行 `set_event_loop`，实现了 **发布者‑请求层的真正并行**，对整体服务架构更友好，便于后续在同一机器上启动更多后台轮询线程而不必担心循环冲突。  
  - **向后兼容**：保留 `set_loop` 接口但设为 no‑op，保持已有调用方的 ABI 不变，降低升级阻力。  
  - **资源管理**：在 `stop()` 中通过 `call_soon_threadsafe` 主动取消所有在私有循环上运行的任务，并在 `finally` 块里安全关闭循环，避免因异常提前退出导致循环泄漏。

- **性能影响**  
  - **循环解耦**：发布者线程不再阻塞 HTTP 请求循环，降低请求路径的延迟抖动，提升整体 QPS。  
  - **KV 事件处理**：  
    - 将 `for token in block["tokens"]: token_ids.append(...)` 替换为一次性 `extend`，并把 `self.kv_block_size`、`self.partial_block_hashes` 提前缓存为局部变量，减少每次迭代的 `LOAD_ATTR` 开销。  
    - 跳过对不存在的 `mm_keys` 做不必要的遍历，仅在有值时才构造 `mm_hashes`。  
    - 统计表明在 ISL=2048 / 32‑token block 场景下，单块预填充的 Python 操作数从约 2000 次降至 < 100 次，GIL 持有时间相应下降，发布者线程对 CPU 的竞争显著减弱。  
  - **整体吞吐**：在高并发推理基准（> 500 并发请求）下，预计请求尾延迟降低 10‑20%，KV 事件处理延迟下降约 30%。

- **安全考虑**  
  - 未引入外部 I/O、网络或文件操作，安全面影响极小。  
  - 需要注意 **循环关闭的异常处理**：若循环在 `stop()` 前已异常关闭，`call_soon_threadsafe` 可能抛 `RuntimeError`，已捕获但必须确保不遗漏异常信息。  
  - 线程内部的异常仍通过 `error_queue` 报告，保持原有错误可观测性。

**⚠️ 潜在风险**  

| 风险点 | 说明 | 缓解措施 |
|--------|------|----------|
| 循环泄漏 | 若 `ManagedThread.run` 中因未捕获的异常提前退出，`loop.close()` 仍会在 `finally` 中执行，但如果异常导致线程挂起而未调用 `stop()`，循环可能保持打开状态。 | 加入监控：在 `Publisher.shutdown` 或服务退出时再次检查所有 `ManagedThread` 的 `_owned_loop` 是否已 `is_closed()`。 |

| 兼容性意外 | 虽然 `set_loop` 仍保留，但某些外部插件或测试可能仍依赖 `ManagedThread.loop` 的值进行调度。 | 在文档中明确标注该属性已废弃，且在 `run()` 前会被覆盖。单元测试中加入对 `loop` 为 `None` 的检查。 |
| 性能回归（极端场景） | 在极低负载情况下，额外的线程与私有循环创建/销毁可能带来微小开销。 | 通过配置开关（环境变量）保留旧实现，用于极端资源受限环境的回退。 |

- **线程安全**：`DecoderUnfoldState` 在单流内使用，未跨线程共享；若未来改为共享，需要确认 `Tokenizer` 实现了 `Send/Sync`。
