### fix: CEL validation rule (#7550)
**SHA**: `28cfdcc` | 🔗 [查看提交](https://github.com/ai-dynamo/dynamo/commit/28cfdcc7c7ad51c867f7d1a44a331f92df04fe2d)

**🎯 变更类型**：Bug修复  
**⚡ 重要程度**：🔴 高  
**📋 变更摘要**：此次提交修正了 `sharedMemory` 相关的 CEL 验证规则。原规则 `!(self.disabled && has(self.size))` 在 `disabled` 字段缺失（即未显式设置）时会产生误判，导致在某些情况下即使 `disabled` 为 `false`（默认），仍会触发 “size 必须为空” 的校验。新规则 `!(has(self.disabled) && self.disabled && has(self.size))` 通过先检查字段是否存在，避免了该误判，提升了 CRD 校验的准确性。  

**🎯 影响范围**：  
- `deploy/operator/api/v1alpha1/common.go`（`SharedMemorySpec` 的 kubebuilder 注解）  
- Helm Chart CRDs：`nvidia.com_dynamocheckpoints.yaml`、`nvidia.com_dynamocomponentdeployments.yaml`、`nvidia.com_dynamographdeployments.yaml`  
- Operator 本地生成的 CRD 清单（`deploy/operator/config/crd/bases/...` 同上三文件）  

**🔍 技术洞察**：  
- **架构影响**：仅涉及声明式 CRD 校验规则的修改，不会影响运行时业务逻辑或组件间交互。属于 API 层面的兼容性调整，对整体系统架构影响极小。  
- **性能影响**：CEL 校验在 API Server 的 Admission 控制阶段执行，新增的 `has(self.disabled)` 判断计算量几乎可以忽略不计，不会对请求延迟产生可感知影响。  
- **安全考虑**：更严格且准确的校验防止用户在 `sharedMemory.disabled=true` 的情况下错误地同时提供 `size`，从而避免因配置错误导致的资源分配异常或潜在的内存泄漏。属于提升配置安全性的改进。  

**⚠️ 潜在风险**：  
1. **向后兼容性** – 若已有集群中某些 CR 实例在 `disabled` 字段未显式声明且 `size` 已设置，旧规则会错误地接受这些对象。新规则会拒绝它们，导致 `kubectl apply` 或 Helm 升级失败。  
2. **自动生成 CRD 不一致** – 代码中的注解和实际 YAML 文件必须保持同步，否则可能出现 “CRD 与代码不匹配” 的警告。需要在提交后重新运行 `make manifests`（或相应生成脚本）确保两者一致。  
3. **Helm Chart 用户** – 使用 Helm 部署时若在 `values.yaml` 中仅设置 `sharedMemory.size` 而未显式设置 `disabled: false`，升级到新规则后可能出现校验错误。  

**💡 关注建议**：  
- 在升级前，使用 `kubectl apply --dry-run=client -f <crd>` 检查现有资源是否符合新规则，必要时在 YAML 中显式加入 `disabled: false`。  
- 完成代码修改后，重新生成并提交最新的 CRD 文件，确保 `api/v1alpha1` 注解与 `config/crd/bases` 中的 YAML 完全同步。  
- 在 Helm Chart 的 `values.yaml` 示例中添加 `sharedMemory.disabled: false` 的默认值或注释，防止用户因字段缺失导致升级失败。  
- 对已有集群进行一次批量验证（`kubectl get <cr> -o yaml | kubeval`）或使用 CI 自动校验，以捕获可能的违规对象。

**🎯 变更类型**：重构 / 架构变更  
**⚡ 重要程度**：🔴 高  
**📋 变更摘要**  
- 将原有的 **multimodal‑processor** 组件合并并废除，改为 **multimodal‑encode‑worker** 直接作为前端入口，接收 Rust 前端预‑token 化的请求（`ModelInput.Tokens`），负责图像下载、Vision 编码、占位 token 扩展以及 NIXL 嵌入传输。  
- 相应地删除了 `processor_handler`、相关工具函数以及 `--multimodal-processor` CLI 标志，更新了初始化逻辑、参数结构、文档和示例脚本。

- **文档**：`docs/backends/sglang/sglang-reference-guide.md`, `docs/features/multimodal/multimodal-sglang.md`。  
- **示例脚本**：`examples/backends/sglang/launch/*.sh`。

### 🔍 技术洞察  

| 维度 | 影响说明 |
|------|----------|
| **架构影响** | • 以前的三层结构 **Processor → EncodeWorker → PDWorker** 简化为 **EncodeWorker (frontend‑facing) → PDWorker**。<br>• Processor 负责的 OpenAI → SGLang 转换、Tokenizer 调用、从 JSON → `PreprocessedRequest` 的工作，现全部迁移到前端（Rust）或 EncodeWorker 中。<br>• `engine=None` 的约定仍然有效，但所有共享基类代码需再次检查对 `engine` 的 guard（已在文档中提醒）。 |

| **兼容性** | • 旧的 `--multimodal-processor` 标志被移除，直接使用 `--multimodal-encode-worker` 替代。<br>• 任何仍在启动 Processor 的脚本将因缺少对应标志而报错，需要迁移。<br>• 现有的 Python 客户端或 HTTP 入口仍通过 Dynamo Frontend 调用 `encode_worker.generate`，兼容性取决于前端是否已改为发送预-token 化请求（`ModelInput.Tokens`）。 |

### ⚠️ 潜在风险  

1. **兼容性破坏**  
   - 生产环境中仍在使用 `--multimodal-processor` 或依赖 `MultimodalProcessorHandler` 的部署脚本会启动失败。  
   - 现有的第三方工具（例如自定义 OpenAI‑compatible 客户端）如果仍发送原始 `MultiModalRequest`（未经前端 token 化）会在 EncodeWorker 中触发 `ValueError: multi_modal_data is required…`。  

2. **前端 Tokenizer 负载**  
   - 迁移后前端需要完成文本 token 化并填充 `token_ids`，这在高并发或大模型（长上下文）情况下可能成为瓶颈。若前端未开启合适的并行或缓存，整体吞吐可能下降。  

3. **错误传播路径**  
   - `_extract_image_urls` 抛出的异常现在直接在 EncodeWorker 中被捕获并记录。若异常没有被前端捕获，可能导致请求 500 返回，需在前端做好错误包装。  

4. **视频/其他多模态支持缺失**  
   - 代码仍显式 `NotImplementedError` 报错 video_url。文档未提醒此限制，可能导致用户误以为视频已被支持。  

5. **NIXL 描述符同步**  
   - `transfer_future` 与 `response_generator` 的并行执行逻辑保持不变，但因 `ready_event` 初始化位置变化，仍需确认 PDWorker 在接收嵌入前已经准备好（现有测试通过，但上线前需做一次集成压测）。

### feat(mocker): integrate AIConfigurator for latency prediction (#7505)
**SHA**: `6b67c1c` | 🔗 [查看提交](https://github.com/ai-dynamo/dynamo/commit/6b67c1c4bfe31ec831718b79fc3cb0c2ee96481d)

**🎯 变更类型**：功能增强 / 架构变更 / 安全修复  
**⚡ 重要程度**：🔴高  

**📋 变更摘要**  
本次 PR 为 Dynamo mocker 引入了 **AI Configurator (AIC) SDK** 作为可选的性能预测模型。新增 `AicSession` Python 包装层并通过 PyO3 暴露 `predict_prefill` / `predict_decode` 给 Rust；在 `PerfModel` 中加入 `Aiconfigurator` 变体以及对应的回调接口 `AicCallback`。CLI 参数、JSON 引擎配置、文档和 Python‑Rust 绑定同步更新，使得用户可以通过 `--aic-perf-model` 使用经过校准的 GPU/模型/后端预测，默认仍保持多项式/插值模型。

---

**🎯 影响范围**  
- **组件**：`components/src/dynamo/mocker/*`（Python session 与参数解析）  
- **Rust 绑定**：`lib/bindings/python/rust/llm/aic_callback.rs`、`entrypoint.rs`  
- **核心模型**：`lib/mocker/src/common/perf_model.rs`（新增 `AicCallback`、`Aiconfigurator`）

### feat(mocker): add offline trace replay mode [DYN-2502] (#7543)
**SHA**: `c3908a3` | 🔗 [查看提交](https://github.com/ai-dynamo/dynamo/commit/c3908a36ca6ca0d5984f97ae7fe666230a5a696b)

**🎯 影响范围**  
- 开发人员使用的 Dev Container 镜像（`container/templates/dev.Dockerfile`、`.devcontainer/post-create.sh`）。  
- 相关 CI/CD 步骤如果直接复用该 Dev Container（例如 GitHub Actions 中的 `container` 选项）。  

**🔍 技术洞察**  

- **架构影响**  
  - 仅限于开发环境，不会影响生产二进制或库的运行时架构。  
  - 引入了额外的软件源（NVIDIA devtools、GitHub CLI），增加了镜像构建时的外部依赖，但仍保持在单层 `RUN --mount=type=cache` 中，避免层次膨胀。  

- **性能影响**  
  - 镜像体积会因为 `nsight-systems`（≈500 MB–1.5 GB）和 `gh`（≈50 MB）而显著增大，拉取与构建时间相应上升。  
  - 运行时性能基本不受影响，因为这些工具仅在容器内部使用，且未在生产路径中引入额外运行时依赖。  

- **安全考虑**  
  - 新增两条 GPG **keyring**（NVIDIA 与 GitHub CLI），若对应的上游签名被泄露或仓库被篡改，可能引入供应链风险。  
  - `apt-get install -y --no-install-recommends` 已限制额外依赖，降低不必要的攻击面。  
  - `gh` 未固定版本，未来镜像构建时可能因上游仓库的破坏性更新导致不可预期的二进制变更，建议使用特定版本或 SHA‑pinning。  

**⚠️ 潜在风险**  

1. **镜像体积膨胀**  
   - 对 CI/CD 环境的网络带宽与存储成本产生影响，尤其在频繁拉取镜像的 CI 场景。  

2. **外部仓库可用性**  
   - 若 NVIDIA 或 GitHub CLI 的 Apt 仓库临时不可达，构建将失败。  

3. **依赖冲突**  
   - `gh` 依赖的底层库可能与容器内已有的工具产生版本冲突（例如 `libcurl`），导致运行时错误。  

4. **启动脚本副作用**  
   - 警告信息在每次交互式 Bash 启动时执行，可能影响自动化脚本或非交互式 `docker exec` 场景的输出解析。  

5. **`post-create.sh` 中的 `uv pip install --no-deps`**  
   - 跳过依赖解析可能导致缺失运行时必需的 Python 包，潜在的 “ImportError”。  

**💡 关注建议**  

- **镜像体积管理**  
  - 将 `nsight-systems` 设为可选组件（如通过构建 ARG 控制），避免在不需要性能分析的开发者机器上强制安装。  

- **版本锁定**  
  - 为 `gh` 添加版本锁定或使用 `apt-get install gh=<ver>`，并在 Dockerfile 注释中记录对应的 SHA‑256 校验。  

- **构建健壮性**  
  - 为外部 Apt 源加入 `apt-get update || (sleep 5 && apt-get update)` 的重试逻辑，提升在网络抖动时的成功率。  

- **启动脚本兼容性**  
  - 将警告信息封装为函数并仅在交互式终端 (`[[ $- == *i* ]]`) 中执行，避免影响非交互式命令。  

- **Python 包依赖**  
  - 在 `post-create.sh` 添加对 `uv pip install` 前的 `uv pip check` 或显式列出 `uv pip install -e $WORKSPACE_DIR[extras]`，确保所需依赖被安装。  

- **CI/CD 适配**  
  - 在 CI 作业中使用 `--mount=type=cache` 共享 Apt 缓存，降低重复下载造成的构建时间。  
  - 如需在 CI 中跳过 `nsight-systems`，提供构建变量 `BUILD_NSIGHT=false` 并在 Dockerfile 中通过 `ARG` 条件化安装。  

> 总体来看，此次提交提升了开发者的使用体验（防止重复 banner、明确挂载提示），但也带来了镜像体积增长与外部仓库依赖的潜在风险。建议在正式推广前对镜像体积、构建成功率以及非交互式入口的行为进行回归验证。

### fix: nightly single gpu tests timeout 180 min (#7547)
**SHA**: `7fb597a` | 🔗 [查看提交](https://github.com/ai-dynamo/dynamo/commit/7fb597a2ed993b8f3a6b9ae1dcaed6e434c19784)

**🎯 变更类型**：Bug 修复  
**⚡ 重要程度**：🟡 中  

**📋 变更摘要**  
本次提交将 `.github/workflows/nightly-ci.yml` 中针对 **单 GPU** 测例的超时时间从 **35 min** 调整为 **180 min**。原有的 35 min 超时在 nightly CI 环境下经常触发，导致部分需要更长执行时间的模型（如 vllm、sglang、trtllm）被提前终止，影响 nightly 测试的完整性。  

**🎯 影响范围**  
- GitHub Actions `nightly-ci` 工作流的三个子作业：`vllm`, `sglang`, `trtllm`  
- 主要影响单 GPU（`gpu_1`）标记的测试套件，间接影响 CI 报告、代码覆盖率统计以及回归检测的准确性。  

**💡 关注建议**  
1. **监控 CI 时长**：部署后观察单 GPU 作业的实际运行时长，确保 180 min 足够且不会导致资源滥用。  
2. **资源预算**：若多数测试仍在较短时间内完成，可考虑将超时设置为更精细的值（如 90 min），以节约 GitHub Action 的运行配额。  
3. **文档同步**：更新项目 CI 文档或 CONTRIBUTING 指南，说明单 GPU 测试的超时已调高，帮助贡献者在本地调试时设定合理的 `RUST_TEST_TIMEOUT`。  
4. **回归测试**：在下一个发布周期前，运行全量 nightly CI，确认不再出现因超时导致的错误报告。  

此改动仅涉及 CI 配置，对源码功能无直接影响，但提升了 nightly 测试的可靠性和可观测性。
