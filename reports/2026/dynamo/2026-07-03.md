### fix(operator): restore v1alpha1 DGD admission compatibility (#11126)
**SHA**: `c914348` | 🔗 [查看提交](https://github.com/ai-dynamo/dynamo/commit/c914348c76400a04b4ef9e8a1273f15dfb1833cd)

**🎯 变更类型**：Bug修复 / 功能增强（恢复 v1alpha1 DGD/DCD Admission 与 v1beta1 的兼容性，并把部分校验迁移到 CRD CEL 规则）  

**⚡ 重要程度**：🔴高  

**📋 变更摘要**  
1. 为 `DynamoComponentDeployment` 与 `DynamoGraphDeployment` 的 CRD 添加 **CEL 校验**：  
   - 限制 `metadata.annotations` 大小、校验 `nvidia.com/vllm-distributed-executor-backend` 注解只能是 `mp` 或 `ray`（不区分大小写）。  
   - 强制 **sidecar** 与 **init container** 必须指定非空 `image`。  
2. 在 `Makefile` 中自动对 v1beta1 CRD 生成相同的 CEL 规则（通过 `yq` 注入 `x‑kubernetes‑validations`，并设置 `additionalProperties.maxLength = 262144`）。  
3. 移除 Operator **Webhook** 中对 PodTemplate、CompilationCache、PodTemplate 注解等的重复校验，改由 CRD schema+CEL 完成。  
4. 扩展并改写单元测试：  
   - 使用真实的 CRD schema 与 CEL 验证器对 **Create/Update** 请求进行全链路校验。  
   - 对 v1alpha1 与 v1beta1 两个 source version 的兼容性、大小写冲突、annotation 规则等进行配对测试。  
5. 其它代码清理：删除冗余错误路径、简化 `betaComponentsByName` 实现、更新请求转换辅助函数等。

---

### 🎯 影响范围
| 受影响模块/组件 | 说明 |
|------------------|------|
| **deploy/operator/crds**（`nvidia.com_dynamocomponentdeployments.yaml`、`nvidia.com_dynamographdeployments.yaml`） | 新增 `x‑kubernetes‑validations`、`maxLength`，影响所有使用这些 CRD 的集群。 |

### 🔍 技术洞察

| 维度 | 影响分析 |
|------|----------|
| **架构影响** | - 将 **业务校验** 从 Operator Webhook 向 **Kubernetes API Server**（CRD schema + CEL）迁移，提升了职责划分的清晰度。<br>- 通过统一的 CEL 规则，v1alpha1 与 v1beta1 的 **转换路径** 保持了一致的校验语义，降低了后向兼容的复杂度。 |
| **性能影响** | - **Webhook 调用次数不变**（仍在 Create/Update 时触发），但 **处理成本显著下降**：API Server 对 JSON Schema+CEL 的校验在 C++ 层实现，通常比 Go‑实现的 webhook 更快。<br>- 生成的 CRD 体积略增（约 +10 KB），对 API Server 启动无实质影响。 |
| **安全考虑** | - 强制 sidecar/init container 必须提供镜像，防止出现 **空镜像导致的容器运行失败** 或潜在的 **镜像拉取攻击面**。<br>- 注解大小限制（`maxLength: 262144`）防止 **DoS**（过大 annotation 导致 API Server 资源耗尽）。<br>- CEL 校验在 API Server 端统一执行，避免了在 webhook 中因代码错误导致的 **校验绕过**。 |
| **可维护性** | - 业务规则集中在 CRD 中，**单点修改**即可同步至所有客户端（kubectl、client‑go、helm）。<br>- 删除了冗余的 Go‑side 校验代码，代码基线更简洁，减少未来的维护负担。<br>- 新增的 `shared_test.go` 封装了 **CRD‑schema + CEL** 的通用测试工具，后续新增字段只需在同一套框架下编写测试。 |
| **兼容性** | - 对已有资源的影响主要在 **升级** 时：如果用户的旧 Manifest 中 sidecar/init‑container 未声明 `image`（过去被 webhook 忽略），升级后 **会被 API Server 拒绝**。<br>- `metadata.annotations.maxLength` 与现有 K8s 注解总大小上限（256 KB）保持一致，兼容性良好。<br>- 仍保留对 v1alpha1 **Immutable** 字段（如 `backendFramework`）的 webhook 检查，保证老版本行为不变。 |

### ⚠️ 潜在风险
1. **升级阻塞**：已有 `DynamoComponentDeployment`/`DynamoGraphDeployment` 在升级到新 CRD 版本时，如果 sidecar/init container 没有 `image`，API Server 将返回 **Invalid**，导致升级失败。  
2. **大小写冲突**：CRD 仍允许 **服务名/组件名大小写不同**（v1alpha1 转换后可能出现冲突），但 CEL 只在 v1beta1 上校验唯一性。若用户在 v1alpha1 中使用 `worker` 与 `WORKER`，升级后会产生 **冲突错误**。  
3. **Makefile 依赖**：CRD 生成依赖 `yq` 与 `yq` 的特定版本，CI 环境或本地开发若缺失可能导致 **CRD 未正确注入 CEL**。  
4. **测试环境依赖**：`shared_test.go` 通过加载本地 `config/crd/bases/*.yaml`，在自定义构建镜像或外部 CI 中若路径不一致会导致测试失败。  
5. **聚合错误差异**：部分错误从 webhook（字符串）迁移到 CEL（字段路径），旧的监控告警或日志分析可能需要 **匹配规则更新**。

### refactor(llm): remove tool-call jail, depend on dynamo-parsers (#11112)
**SHA**: `867d9ab` | 🔗 [查看提交](https://github.com/ai-dynamo/dynamo/commit/867d9ab02c8f7e413374db5d12ef5e9d79b0e840)

**🎯 变更类型**：重构 / 架构变更 / 功能迁移  
**⚡ 重要程度**：🔴高  
**📋 变更摘要**  
- 将原本在 `dynamo-llm` 中实现的 **tool‑call jail** 完全迁移到 `dynamo-parsers`，并把对应的依赖（`aho-corasick`, `async-stream`, `futures`, `dynamo-protocols`）添加到 `dynamo-parsers`。  
- 删除 `llm` crate 中的 `jail.rs` 与旧的 **前缀匹配器**（`prefix_matcher.rs`），并相应地在 `Cargo.toml`/`Cargo.lock` 中移除不再使用的依赖。  
- 为 `OpenAIPreprocessor::apply_tool_calling_jail` 实现 **适配层**：在进入 jail 前把 `Annotated<Nv<_>>` 转为共享的 `Annotated<CreateChatCompletionStreamResponse>`，离开 jail 后再把 `llm_metrics` 重新挂回 `Nv` 结构。  
- 新增测试用例，验证 **llm_metrics** 在跨 crate 边界的缓冲与恢复过程中的正确性。  

---

### 🎯 影响范围
- **核心模块**：`lib/llm`（预处理器、相关测试）、`lib/llm/src/utils`（前缀匹配器删除）  
- **依赖链**：`dynamo-parsers`（版本升至 3.1.0，新增 `aho-corasick`, `async-stream`, `futures`, `dynamo-protocols`）  
- **绑定**：`bindings/kvbm`、`bindings/python` 的 Cargo.lock 也同步了依赖升级（`windows-sys`、`itertools`、`socket2` 等）。  
- **用户代码**：所有直接 `use dynamo_llm::protocols::openai::chat_completions::jail::*` 的调用必须迁移到 `dynamo_parsers::tool_calling::jail`.  

---

### 🔍 技术洞察

| 维度 | 说明 |
|------|------|
| **架构影响** | 1️⃣ **职责分离**：tool‑call jail 只涉及对 **共享 OpenAI `CreateChatCompletionStreamResponse`** 的解析，已不再需要 `dynamo-runtime`（`Annotated<Nv<>>`）的上下文，迁入 `dynamo-parsers` 更合乎单一职责原则。<br>2️⃣ **依赖倒置**：`dynamo-llm` 现在仅依赖 `dynamo-parsers`（而非相反），形成自下而上的依赖流，降低耦合。<br>3️⃣ **API 边界**：`OpenAIPreprocessor::apply_tool_calling_jail` 成为 **适配器**，在两套结构之间完成转换与度量缓冲，保持对外 API 不变。 |
| **性能影响** | - **二进制体积**：删除 `prefix_matcher`（约 5 KB）以及 `aho-corasick` 在 `llm` 中的直接使用，整体体积略减。<br>- **运行时开销**：新增 `Arc<Mutex<PendingMetrics>>` 用于在流式处理时缓存 `llm_metrics`，在高并发流（数千条并行）下可能引入轻微锁竞争；但实际成本仅在每个 **工具调用** 的几次 chunk 中出现，影响可忽略。<br>- **异步流转**：利用 `async-stream`/`futures` 在 `dynamo-parsers` 中实现的 `apply_tool_calling_jail` 与原实现保持同等的零拷贝流式特性，性能基本持平。 |
| **安全考虑** | - 没有引入新的 *unsafe* 代码或外部网络调用，安全属性保持不变。<br>- `Arc<Mutex>` 的使用在 Rust 中天然安全，只要不出现死锁（本代码仅在单线程 `Stream` 中使用），风险极低。<br>- 迁移后 `dynamo-llm` 不再直接依赖 `aho-corasick`，减少了潜在的库漏洞曝光面。 |
| **可维护性** | - **代码集中**：工具调用解析逻辑统一在 `dynamo-parsers`，后续改进（如新增工具调用格式、提升解析算法）只需在一个 crate 中修改。<br>- **测试迁移**：原有的大量单元测试（约 4 k 行）全部转为调用新适配层，保证行为回归。<br>- **依赖管理**：同步了 `Cargo.lock`，提升了 reproducible builds 的可靠性。 |

---

### ⚠️ 潜在风险
1. **向后兼容**：外部项目仍在 `dynamo-llm` 中 `use ...::jail::*` 的代码会编译失败，需要更新 import 路径。  
2. **度量缓冲的并发安全**：虽然目前只在单线程流中使用 `Mutex`，若后续将 `apply_tool_calling_jail` 拆解为多线程并行处理，可能出现锁竞争或死锁，需要评估并可能改为 `parking_lot::Mutex` 或 `RwLock`。  
3. **版本联动**：`dynamo-parsers` 升级到 3.1.0，若项目仍锁定在 3.0.x，Cargo 解析会冲突；发布后应在 `CHANGELOG` 明确破坏性升级信息。  
4. **Windows 平台变更**：`windows-sys` 升级到 0.61.2，可能触发编译器或链接层面的细微差异，需要在 CI 中确保 Windows CI 通过。  
5. **测试覆盖**：新加入的度量缓冲测试仅覆盖单一工具调用场景，复杂多工具调用、错误路径（如 `error` 字段）未显式验证，建议补充对应测试。

- **性能基准**：在 CI 添加 **流式工具调用的基准测试**（`criterion`），对比迁移前后每 1k token 的延迟，确保锁开销未出现回归。

### fix(vllm): propagate cache salt in text mode (#11082)
**SHA**: `b9c5bd7` | 🔗 [查看提交](https://github.com/ai-dynamo/dynamo/commit/b9c5bd7a7a931986173c93d90c2e8a74481f2006)

**🎯 变更类型**：Bug修复  
**⚡ 重要程度**：🔴高  
**📋 变更摘要**：  
在 VLLM 文本生成路径中，新增对 `nvext.cache_salt` 字段的传播。通过调用内部函数 `_apply_nvext_cache_salt` 将租户/缓存标识写入 `TextPrompt`，并补充对应单元测试确保该字段被正确注入到生成请求的 Prompt 对象中。  

**🎯 影响范围**：  
- `components/src/dynamo/vllm/handlers.py`（文本模式生成入口）  
- `components/src/dynamo/vllm/tests/test_vllm_unit.py`（新增单元测试）  
- 依赖 `TextPrompt` 结构的所有下游调度/缓存层（如 KV Cache、分片调度器）  

**🔍 技术洞察**  

- **架构影响**：  
  - 该改动在 **VLLM 处理层** 引入了租户感知的缓存盐（`cache_salt`），使得同模型的不同租户在 KV Cache 命中时能够区分，提升多租户隔离能力。  
  - 仅限文本模式（`_generate_text_mode`），不影响流式或函数调用等其它模式；如果后续需要统一策略，仍需在对应路径补全。  

- **性能影响**：  
  - 只在生成前的 Prompt 对象构造阶段添加一次属性写入，时间复杂度 O(1)。对整体吞吐、延迟基本没有可感知影响。  
  - 由于缓存盐会导致同一 token 序列在不同租户下产生不同的缓存键，可能降低 KV Cache 命中率，进而略微增加显存和计算开销。此影响取决于租户数与请求相似度。  

- **安全考虑**：

### fix(kv-router): preserve cold restore ordering (#11190)
**SHA**: `db0feb0` | 🔗 [查看提交](https://github.com/ai-dynamo/dynamo/commit/db0feb03cab917209368e8b7cdb931cf963c9dca)

**🎯 变更类型**：Bug修复  
**⚡ 重要程度**：🔴 高  
**📋 变更摘要**：修复了在 *kv-router* 的 `ThreadPoolIndexer` 中，冷移除（cold removal）后恢复（restore）任务的顺序不一致问题。通过将 **WorkerId → 线程索引** 的映射扩展为 “索引器生命周期内保持不变”，并在移除/恢复路径统一使用 `enqueue_worker_removal`，确保同一 `WorkerId` 的所有任务（包括冷移除后重新加入的任务）走同一 FIFO 队列，从而维持原有的顺序保证。  

**🎯 影响范围**：  
- `core/kv-router/src/indexer/thread_pool.rs`（核心索引器实现）  
- 受影响的模块：`ThreadPoolIndexer<T>`, `KvIndexerInterface` 实现，`prune_manager` 交互，`worker_event_channels` 调度逻辑。  

**🔍 技术洞察**  

- **架构影响**  
  - 将 `worker_assignments` 的语义从 “活跃工作线程映射” 改为 “索引器生命周期的粘性分配”。这意味着即使 Worker 被 **cold‑removed**（即从活跃集合中剔除），对应的映射仍保留，后续的 **restore** 会使用同一线程的队列，从而避免 “后续任务抢先” 的情形。  
  - 引入 `enqueue_worker_removal` 公共入口，统一处理 `RemoveWorker` 与 `RemoveWorkerDpRank` 两类任务的排队、错误日志以及清理触发，使代码更易维护且行为一致。  
  - 对 `worker_assignment_count` 的注释进行了澄清：它不再表示当前活跃线程数，而是 **单调递增的预留序列**，包括冷移除时的保留位，保证后续恢复仍占用同一 FIFO。  

- **性能影响**  
  - 代码改动主要是结构化抽象和少量额外函数调用，没有增加同步原语或显式锁，预计对吞吐量和延迟几乎没有负面影响。  
  - 通过保持任务顺序，可避免因错误的并发调度导致的重复索引或不一致数据，从而在极端负载下提升整体系统正确性与潜在的性能稳定性。  

- **安全考虑**  
  - 变更不涉及网络、加密或权限检查，安全风险极低。唯一需要关注的是 **并发安全**：`worker_assignments` 仍使用 `DashMap` 并通过 `Arc` 共享，保持原有线程安全模型。  

**⚠️ 潜在风险**  

1. **映射泄漏**：因为映射项在冷移除后仍保留，若系统长期创建大量一次性 Worker，可能导致 `DashMap` 中残留大量无用键，占用内存。  
2. **死锁/饱和**：`enqueue_worker_removal` 仍依赖 `worker_event_channels[thread_idx].send`，如果某线程的通道被阻塞（如未及时消费），后续所有冷移除/恢复任务会被集中在同一通道，可能引发背压或任务堆积。  
3. **测试覆盖不足**：当前仅添加了两例单元测试，未覆盖高并发下多个 Worker 同时冷移除并恢复的交叉情况。  

**💡 关注建议**  

- **监控 & 报警**：在生产环境监控 `worker_assignments.len()` 与实际活跃 Worker 数的比例，若出现异常增长，触发内存使用警报。  
- **压力测试**：开展并发移除/恢复的压力测试，验证在 10⁴+ Worker 同时冷热切换时 FIFO 顺序仍然保持且通道不出现阻塞。  
- **代码审阅**：确认 `maybe_enqueue_cleanup` 在新路径下仍能正确触发资源回收，防止旧的清理逻辑被遗漏。  
- **文档更新**：在项目的设计文档或注释中明确 “indexer‑lifetime sticky routing” 的概念，帮助后续维护者理解为何映射不会被即时删除。  

整体来看，此次改动通过对 Worker 任务分配机制的细化，解决了关键的顺序一致性缺陷，对系统的正确性提升明显，风险可通过上述监控与测试措施得到有效管控。
