### feat(kv-router): add ActiveSequences benchmark and extract common bench utils (#6633)
**SHA**: `e5850e2` | 🔗 [查看提交](https://github.com/ai-dynamo/dynamo/commit/e5850e2385542ff5076da8b0de8e8420b63801dc)

**🎯 变更类型**：功能增强 / 重构 / 架构变更  
**⚡ 重要程度**：🔴高  

**📋 变更摘要**  
- 为 `kv-router` 引入全新的 **调度层**（`scheduling`）和 **序列管理层**（`sequences`），对原有的 `ActiveSequencesMultiWorker` 进行重构：改用 `parking_lot::RwLock` + 按工作者顺序化的 `Vec` 替代 `dashmap`，提升锁粒度与缓存局部性。  
- 新增 `WorkerConfigLike` 抽象，使调度层只依赖统一的配置接口，`ModelRuntimeConfig` 直接实现该 trait。  
- 将原本散落在多个 benchmark 文件中的公共工具抽离到 `benches/common/mod.rs`，并新增 **ActiveSequences** 基准 `active_sequences_bench.rs` 用以测量 `potential_blocks_and_tokens / add_request / mark_prefill_completed / free` 的吞吐与延迟。  
- 引入 `RouterConfigOverride`、`KvRouterConfig` 的 **验证**（`validator`）并在 `config.rs` 中统一实现，去除冗余复制，保持与核心 `dynamo_kv_router` 包同步。  
- 将原有的 `bench_utils.rs` 移除，所有 benchmark 统一使用新的 `common` 模块；相应的 `kv_indexer_bench.rs`、`mooncake_bench.rs`、`radix_tree_microbench.rs` 迁移至新实现。  
- 在 `llm` 包中重新导出调度与配置实现，剥离对旧实现的依赖，更新 `queue.rs`、`scheduler.rs` 直接复用 `dynamo_kv_router` 提供的实现。  
- 通过 `CancelGuard` 机制防止 `mocker` 中的 `Scheduler` 提前取消，保证后台任务的生命周期。  
- 更新测试、CI 环境变量指向新的日志路径（`dynamo_kv_router::scheduling::selector`），确保日志统一。  

**🎯 影响范围**  
- `lib/kv-router`：核心调度、序列追踪、配置验证、基准套件。  
- `lib/llm`：KV‑router 组件的导入路径、调度队列、Scheduler 接口、模型运行时配置。  
- `lib/mocker`：Scheduler 生命周期管理。  
- `tests/serve`、CI 环境：日志路径更改。  
- 依赖方（任何直接使用 `dynamo_kv_router::sequence`、`dynamo_kv_router::multi_worker_sequence`、`dynamo_kv_router::protocols::WorkerConfigLike`）需要注意重新导入路径或使用提供的兼容性 re‑export。  

**🔍 技术洞察**

| 维度 | 影响 |
|------|------|
| **架构影响** | • **调度层抽象**：`WorkerSelector`、`SchedulerQueue`、`KvScheduler` 被拆分至独立模块，职责单一，便于插件化（自定义选择器）。<br>• **配置抽象**：`WorkerConfigLike` 取代之前硬编码的 `ModelRuntimeConfig`，调度层不再依赖具体运行时实现，增强可组合性。<br>• **序列管理重构**：从 `DashMap<Worker, ActiveSequences>` 迁移到 `WorkerTable { slots: Vec<(Worker, RwLock<ActiveSequences>)> }`，提升遍历效率与缓存友好度，且在更新 worker 集合时保留已有 `ActiveSequences`。<br>• **公共 Benchmark 基础**：`benches/common/mod.rs` 为所有基准提供统一的 trace 处理、事件生成、图表绘制等工具，降低重复代码。|
| **性能影响** | • **锁粒度优化**：原先的 `DashMap` 为每个 worker 进行哈希查找并伴随内部锁，遍历全部 workers 时仍需多次哈希。新实现使用 `Vec` + per‑worker `RwLock`，在遍历（如 `potential_blocks_and_tokens`）时仅做一次顺序遍历，局部性更好，理论上 **降低 O(N)** 的查找与遍历开销。<br>• **调度队列**：使用 `BinaryHeap` 与 `priority_jump` 支持延迟调度，避免因所有 worker 满载导致的请求阻塞。阈值检查采用 `DEFAULT_MAX_BATCHED_TOKENS` 防止意外无限排队。<br>• **软最大化采样**：`softmax_sample` 改写为在 `selector` 中复用，避免浮点溢出并在温度=0 时返回所有最小 logits，提升概率采样的数值稳定性。<br>• **基准覆盖**：新增 `active_sequences_bench` 能直接衡量路由层关键路径（块/令牌潜在计数 + 请求加入/完成/释放）在不同负载下的吞吐与 99% 延迟，为性能回归提供细粒度指标。|

### fix: propagate vllm-distributed-executor-backend annotation from DGD metadata to backends (#6692)
**SHA**: `dba51ee` | 🔗 [查看提交](https://github.com/ai-dynamo/dynamo/commit/dba51eed823af5a0e864d0b4fb2c42ea9ea52dc2)

**🎯 变更类型**：Bug修复 / 功能增强  
**⚡ 重要程度**：🔴高  

**📋 变更摘要**  
1. 新增 `propagateDGDAnnotations` 辅助函数，统一把 DGD（DynamoComponentDeployment）级别的关键注解（包括新加的 `vllm-distributed-executor-backend`）传播到组件级别的 `Annotations`，并确保服务级别注解拥有最高优先级。  
2. 在图生成路径（单节点/多节点）以及 Grove 路径中使用该函数，去除冗余的手工复制逻辑。  
3. 为 `vllm-distributed-executor-backend` 注解加入验证规则，只接受 `"mp"` 或 `"ray"` 两个合法值，并补充相应单元测试。  

**🎯 影响范围**  
- `deploy/operator/internal/dynamo/graph.go`（DCD、PodSpec 生成）  
- `deploy/operator/internal/dynamo/graph_test.go`（新增注解传播单元测试）  
- `deploy/operator/internal/webhook/validation/shared.go`（服务级别注解校验）  
- `deploy/operator/internal/webhook/validation/shared_test.go`（验证新注解的正负案例）  

**🔍 技术洞察**  

- **架构影响**  
  - 引入统一的 `propagateDGDAnnotations`，消除了在 DCD 与 Grove 两条路径中对同一批注解的重复实现，提升代码可维护性与一致性。  
  - 将注解传播的职责集中在运营层（Operator）而非各下游模块，降低后续新增注解时的耦合度。  
  - 增加了对 `vllm-distributed-executor-backend` 注解的支持，使得后端执行器选择能够通过 DGD 元数据统一配置，优化了组件间的配置传递链路。  

- **性能影响**  
  - 传播逻辑仅在创建 Deployment/PodSpec 时执行一次，遍历的键集合固定（4 条），对整体调度或运行时性能影响可忽略不计。  
  - 通过提前在 Operator 端完成注解合并，避免了后端容器在启动后再次解析或重写注解，间接减少了启动时的配置处理开销。  

- **安全考虑**  
  - 新增的注解值校验限制只能为 `"mp"` 或 `"ray"`，防止用户误写导致后端执行器选择异常，降低因配置错误引发的服务不可用风险。  
  - 传播过程不对注解内容做任何转义或执行，仅做键值复制，未引入新的注入面向。  
  - 由于注解本身不涉及敏感信息，此变更不影响已有的安全模型。  

**⚠️ 潜在风险**  

1. **行为变更风险**：原先未传播 `vllm-distributed-executor-backend` 时，后端可能使用默认执行器；现在会强制使用 DGD 中指定的值，可能导致已在生产环境中依赖默认行为的集群出现意外切换。  
2. **向后兼容性**：旧版 DGD 对象若未携带该注解，新版本 Operator 会默认不传播；但如果用户在服务级别已设置非法值，校验会直接报错，导致升级时部署失败。  
3. **测试覆盖盲区**：仅在 Operator 单元测试中验证了传播与校验逻辑，未在集成环境（实际 K8s）跑通完整的部署流程，可能遗漏因注解冲突导致的调度或副本创建错误。  

**💡 关注建议**  

- **升级指引**：在 Operator 版本升级文档中说明新增的 `vllm-distributed-executor-backend` 注解传播规则，建议运维在升级前检查现有 DGD 是否已经显式设置该注解，或确认默认后端行为是否符合预期。  
- **回滚方案**：如果因后端执行器切换导致异常，可通过在服务级别显式覆盖注解（`mp` 或 `ray`）来临时恢复原有行为。  
- **监控与告警**：在 Operator 的日志或事件中加入 “propagated annotation X=Y” 级别 INFO，便于排查实际生效的注解来源。  
- **进一步完善**：将注解键列表 (`dgdPropagatedAnnotationKeys`) 放到公共常量文件或配置中，以便后续新增注解时只修改单处；同时考虑在 CI 中加入端到端的 K8s 集成测试，确保注解传播在真实调度链路中保持一致。

### fix: Phase out llava and make EPD single GPU (#6674)
**SHA**: `4ffa108` | 🔗 [查看提交](https://github.com/ai-dynamo/dynamo/commit/4ffa1082cd0d47fd5c46d84333693f89da621d72)

**🎯 变更类型**：Bug修复 / 架构调整  
**⚡ 重要程度**：🔴高  
**📋 变更摘要**：本次提交将原先基于 *llava* 的 EPD（Encode‑Prefill‑Decode）多 GPU 示例改为使用 *Qwen3‑VL‑2B‑Instruct* 并强制所有工作进程在同一块 GPU 上运行。为新模型补齐了 TensorRT‑LLM 的 engine 配置文件，更新了启动脚本和 CI 测试用例，以匹配单 GPU 运行模式。  

**🎯 影响范围**：  
- `examples/backends/trtllm/engine_configs/qwen3-vl-2b-instruct/*`（新增 3 份 YAML）  
- `examples/backends/trtllm/launch/epd_multimodal_image_and_embeddings.sh`（启动脚本）  
- `tests/serve/test_trtllm.py`（CI 测试配置）  
- 相关文档/模板（已删除对 `llava` 的引用）  

**🔍 技术洞察**  

- **架构影响**  
  - **单 GPU 化**：原先的 EPD 方案采用 2‑GPU（encode 与 prefill 共 GPU，decode 单独 GPU）或 3‑GPU（每阶段独占），现在全部阶段共用同一块 GPU。整体系统拓扑从 “多节点‑多 GPU” 简化为 “单节点‑单 GPU”。  
  - **模型切换**：从 `llava‑v1.6‑mistral‑7b` 替换为 `Qwen3‑VL‑2B‑Instruct`，涉及 tokenizer、vision encoder、投影层等实现细节的差异，代码层面已统一使用 `MultimodalEncoder`（仅在 encode 阶段加载），不再依赖原 llava 的特定实现。  
  - **调度/重叠**：encode 与 prefill 共享 GPU 时，`disable_overlap_scheduler` 被统一设为 `true`，避免调度器在同 GPU 上产生竞争。  

- **性能影响**  
  - **吞吐与延迟**：单 GPU 必然导致整体吞吐下降（尤其在 decode 阶段需要大量 KV 缓存），但通过 `enable_chunked_prefill` 与 `kv_cache_config.free_gpu_memory_fraction=0.10` 的调优，仍能在 4‑batch、4096‑seq 长度下保持基本可用。  
  - **显存占用**：所有工作进程共享同一块显存，需要注意 `max_batch_size`、`max_seq_len` 与 `free_gpu_memory_fraction` 设置，防止 OOM。  
  - **启动成本**：省去跨 GPU 的进程间通信与 NCCL 初始化，启动更快，适合 CI 环境或资源受限的开发机器。  

- **安全考虑**  
  - 本次改动仅涉及配置与脚本层面，无代码执行路径的安全敏感改动。  
  - 环境变量 `CUSTOM_TEMPLATE` 被移除，降低了潜在的模板注入风险。  

**⚠️ 潜在风险**  

1. **显存不足**  
   - 单 GPU 上三个工作进程（encode、prefill、decode）共用显存，若 `max_batch_size`、`max_seq_len` 或 `tensor_parallel_size` 调高，极易触发 OOM。  
2. **模型兼容性**  
   - Qwen3‑VL‑2B‑Instruct 与原 llava 在 Vision‑Encoder 接口上可能存在细微差异，若上层代码仍硬编码 llava‑特有的 tokenizer 或 projector，运行时会报错。  
3. **CI 依赖变更**  
   - CI 标记从 `nightly` 改为 `pre_merge`，如果其他工作流仍以 `nightly` 为条件筛选，将失去此测试覆盖。  
4. **脚本参数冲突**  
   - `DECODE_CUDA_VISIBLE_DEVICES` 与 `ENCODE_CUDA_VISIBLE_DEVICES` 均设为 `"0"`，若用户在同一机器上并行启动多个 EPD 实例，可能出现设备竞争。  

**💡 关注建议**  

- **显存监控**：在 CI 与本地调试时加入 `nvidia-smi` 监控，确保 `free_gpu_memory_fraction` 能覆盖实际占用；必要时调低 `max_batch_size` 或 `max_seq_len`。  
- **文档同步**：更新 README / 示例文档，明确 EPD 已从多 GPU 迁移到单 GPU，并提供显存配置建议。  
- **模板检查**：如果后续仍需自定义 Jinja 模板，请在脚本中重新加入 `CUSTOM_TEMPLATE` 参数，并做好安全审计。  
- **兼容性回退**：保留旧的 llava 配置（或提供切换开关），以防用户仍需在多 GPU 环境下运行旧模型。  
- **CI 维护**：确认 CI 流水线对 `pre_merge` 标记的处理逻辑，防止该测试被遗漏；同时在 CI 中加入显存/时间阈值的断言，提前捕获 OOM 或超时。  

---  

通过以上改动，项目在资源受限的环境（单 GPU）下也能运行 EPD 多模态推理，提升了可用性与 CI 稳定性，但需要谨慎管理显存与模型兼容性。

### fix: store nodesWithGPUs (#6690)
**SHA**: `98a6d3b` | 🔗 [查看提交](https://github.com/ai-dynamo/dynamo/commit/98a6d3b91076f8239b9db0e131175de2de990b4a)

**🎯 变更类型**：Bug修复  
**⚡ 重要程度**：🔴高  
**📋 变更摘要**：  
1. 在 GPU 发现流程中新增 `NodesWithGPUs` 字段，并在 `GPUInfo` 中记录拥有 GPU 的节点数。  
2. 基于发现结果计算 `TotalGPUs`（`GPUsPerNode * NodesWithGPUs`），并在 `DynamoGraphDeploymentRequest` 的硬件规格中自动填充该字段。  
3. 为此新增相应的日志、单元测试以及 CRD 验证用例，确保在用户未显式指定 `TotalGPUs` 时能够补全。

**🎯 影响范围**：  
- `deploy/operator/internal/gpu/discovery.go`（GPU 发现实现）  
- `deploy/operator/internal/controller/dynamographdeploymentrequest_controller.go`（硬件规格填充）  
- `deploy/operator/internal/controller/dynamographdeploymentrequest_controller_test.go`（大量测试用例）  
- 可能影响的 CRD `DynamoGraphDeploymentRequest`（新增 `TotalGPUs` 字段的兼容性）

**🔍 技术洞察**  

- **架构影响**  
  - **数据流补全**：将底层节点级 GPU 信息（节点数、每节点 GPU 数）提升到上层业务对象（DGDR）中，提升了调度器对集群整体 GPU 能力的感知。  
  - **向后兼容**：仅在 `hw.TotalGPUs` 为 `nil` 时写入，保持对已有清单的兼容性；但需要在 CRD schema 中将 `TotalGPUs` 标记为可选，否则老版集群在 `kubectl apply` 时可能因未知字段而报错。  

- **性能影响**  
  - **发现阶段**：`nodesWithGPUs` 计数在已遍历节点的循环中完成，额外的整数累加开销几乎可以忽略。
