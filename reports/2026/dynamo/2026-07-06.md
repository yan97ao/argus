### fix(recipes): bump Hugging Face Hub helper-job pins to 1.16.4 (#10986)
**SHA**: `f8ddb76` | 🔗 [查看提交](https://github.com/ai-dynamo/dynamo/commit/f8ddb7627488b598e1e1565cc98a07678b77148d)

**🎯 变更类型**：Bug修复 / 依赖升级  
**⚡ 重要程度**：🔴 高  
**📋 变更摘要**：本次提交将项目中多个 Kubernetes Job（recipes）以及示例部署脚本里对 `huggingface_hub`（或 `huggingface-hub`）的版本固定从 **1.11.0** 升级到 **1.16.4**。目标是获取最新的 bug‑fix 与安全补丁，防止因旧版库已知漏洞或不兼容导致模型下载失败。  

**🎯 影响范围**：  
- `examples/backends/vllm/deploy/lora/*` 两个示例 Job  
- 所有 `recipes/*/model-cache/model-download*.yaml` 中的模型下载任务（覆盖 DeepSeek、GLM、GPT‑OSS、Kimi、Llama‑3、Nemotron‑3、Qwen3 系列等 20+ 配置）  

**🔍 技术洞察**  

- **架构影响**：  
  - 仅在 **容器启动阶段** 执行 `pip install`，不涉及服务代码或运行时架构。  
  - 对整体系统拓扑（调度、存储、网络）无直接影响，属于底层依赖的横向升级。  

- **性能影响**：  
  - `huggingface_hub` 1.16.x 引入了对 **多线程下载**、**更高效的缓存管理** 以及 **请求复用** 的改进，理论上可缩短模型拉取时间，尤其在并发下载大模型时收益明显。  
  - 额外的依赖升级可能带来略微的镜像体积增加（约 10 MB），对启动延迟影响可忽略不计。  

- **安全考虑**：  
  - 1.12‑1.16 版本期间修复了多个 CVE（如 **CVE‑2023‑46286**、**CVE‑2024‑XXXX**），涉及 **URL 过滤、文件写入安全** 等。升级可消除已知的路径遍历与不安全请求风险。  
  - 仍然使用 `pip install --no-cache-dir`，可防止缓存中潜在的恶意包残留，保持安全实践。  

**⚠️ 潜在风险**  

1. **兼容性 Breakage**：  
   - `huggingface_hub` 1.12+ 对 **API 参数命名** 与 **返回结构** 做了细微调整，若某些自定义脚本（如 `hf download` 包装）依赖旧版返回字段，可能出现运行时错误。  
2. **传递依赖冲突**：  
   - 1.16.4 依赖的 `requests`、`tqdm` 等库版本提高，若容器镜像中已有其他版本锁定（例如在基础镜像中），可能导致 **依赖冲突** 或 **二进制不兼容**。  
3. **镜像构建缓存失效**：  
   - 版本升级会导致 Docker 镜像层重新构建，若 CI/CD 未清理缓存，可能产生 **旧层残留**，导致镜像体积膨胀或出现不可预期的行为。  

**💡 关注建议**  

- **测试验证**：在升级后立即运行一次完整的模型下载 Job（可选取前置模型如 `deepseek-ai/DeepSeek-R1`）并检查日志，确保 `hf download` 正常完成且返回码为 0。  
- **回滚机制**：保留原始 1.11.0 版本的 yaml（通过 Git tag）以便在发现兼容性问题时快速回滚。  
- **依赖审计**：在 CI 中加入 `pip check` 步骤，捕获潜在的依赖冲突；如有冲突，考虑在镜像 Dockerfile 中显式锁定必要的子依赖版本。  
- **镜像层清理**：在 CI/CD 中使用 `--no-cache` 或手动删除旧层，确保新镜像体积合理。  
- **安全监控**：启用 Dependabot（或 similar）对 `huggingface_hub` 继续追踪最新安全补丁，定期评估是否需要进一步升级。  

通过上述措施，可确保此次依赖升级带来的安全与性能收益顺利落地，同时将潜在的兼容性风险降到最低。

### fix(operator): remove lws worker gpu pre-validation (#11115)
**SHA**: `3faef7a` | 🔗 [查看提交](https://github.com/ai-dynamo/dynamo/commit/3faef7abac245a8da75381565a4f956c77baa62c)

**🎯 变更类型**：Bug修复  

**⚡ 重要程度**：🔴高  

**📋 变更摘要**  
1. 删除了 `generateWorkerPodTemplateSpec` 中对 LWS（Local Worker Service）Worker Pod 必须设置 `nvidia.com/gpu` 限制的前置校验。  
2. 新增单元测试验证在不声明 GPU 限制的情况下仍能成功生成 Worker Pod 模板。  
3. 使得使用 CPU-only 资源的 LWS Worker 部署不再因缺失 GPU 配额而被错误拦截，提升了 Operator 的兼容性和易用性。  

**🎯 影响范围**  
- `deploy/operator/internal/controller/dynamocomponentdeployment_controller.go`（核心调度逻辑）  
- `deploy/operator/internal/controller/dynamocomponentdeployment_controller_test.go`（单元测试）  
- 受影响的模块：Operator 控制器层、DynamoComponentDeployment 资源的副本生成流程。  

**🔍 技术洞察**  

- **架构影响**  
  - 解除对 GPU 限制的强制检查后，Operator 允许 LWS Worker 在 **CPU‑only** 环境下运行，降低了对底层硬件的耦合度。  
  - 对整体系统架构影响有限，仍然保留了原有的 `DynamoComponentDeployment` → `PodTemplateSpec` → `Pod` 的生成链路，只是简化了资源校验阶段。  

- **性能影响**  
  - 省去一次资源读取与条件判断，微乎其微的 CPU 开销下降（约 <1%），对调谐周期整体性能基本不产生可感知影响。  
  - 由于可以在无 GPU 节点上调度 Worker，潜在的调度成功率提升，间接提升了集群资源利用率。  

- **安全考虑**  
  - 该改动不涉及代码执行路径的安全逻辑，也不改变 RBAC、网络或密钥管理。  
  - 唯一需要注意的是 **资源配额**：若用户未显式声明 GPU 限制，Kubernetes 将视为 **0**，调度器可能会将 Pod 分配到不具备 GPU 的节点，导致运行时库（如 `torch.cuda`）抛出错误，可能被误认为安全异常。  

**⚠️ 潜在风险**  

1. **运行时错误**  
   - 若 LWS Worker 实际依赖 GPU（例如使用 `vllm` 的 GPU 加速路径），缺少 GPU 限制会导致容器启动后在代码层面报错，表现为“CUDA not available”。  

2. **资源调度误导**  
   - 失去 GPU 限制的显式约束，调度器可能将 GPU‑dependent 工作负载误放在纯 CPU 节点，影响服务可用性。  

3. **监控与计费不一致**  
   - 运营侧监控可能仍假设 Worker 使用 GPU，导致资源使用报告与实际不符，进而影响成本核算。  

**💡 关注建议**  

- **文档更新**：在 Operator 文档中明确说明 LWS Worker **不再强制要求 GPU 限制**，并提示用户在需要 GPU 的场景自行添加 `nvidia.com/gpu` 限制。  
- **可选校验**：考虑引入配置开关（如 `OperatorConfiguration.EnableLWSWorkerGPUCheck`），让集群管理员根据实际需求决定是否进行 GPU 预校验。  
- **运行时检测**：在容器入口脚本或应用层加入 **GPU 可用性检测**，若检测不到 GPU 则优雅退出或回退到 CPU 模式，避免产生不可预期的异常。  
- **监控报警**：为 LWS Worker 添加 “GPU 未分配但代码请求 GPU” 的告警规则，帮助运维快速定位因此改动导致的启动失败。  
- **回归测试**：在 CI 中保留当前新增的单元测试，同时补充针对 **GPU 依赖** 场景的集成测试，确保两类部署（CPU‑only 与 GPU‑enabled）均能正确工作。

### refactor(validation): make DGD validation structural (#11090)
**SHA**: `4d3f3a1` | 🔗 [查看提交](https://github.com/ai-dynamo/dynamo/commit/4d3f3a1d1c5afbe6c97cc150f5b1458fa3b2445a)

**🎯 变更类型**：重构 / 架构变更  
**⚡ 重要程度**：🔴高  

**📋 变更摘要**  
1. 将 DynamoGraphDeployment（DGD） 的校验从过去的“错误 + warnings + `errors.Join`”混合实现，迁移为 **Kubernetes 官方的结构化校验风格**（`field.ErrorList` + `field.Path`）。  
2. 引入统一的 **`sharedValidation`** 接收器，用于跨资源、跨 API 版本的共享校验，去除了在对象上保存遍历状态的做法。  
3. 完整拆解、抽象出大量细粒度校验函数（对象元数据、Spec、Topology、Restart、Checkpoint、KV‑Transfer、GMS、Failover、Component‑level 检查等），并在文档 `AGENTS.md` 中统一说明校验约束。  
4. 迁移旧的 v1alpha1 兼容校验为结构化实现，保持向后兼容的同时不再在业务代码里手动拼装错误字符串。  
5. 细化了 **Warning** 收集路径，所有 warning 统一通过 `sharedValidation.warn*` 方法在同一次递归期间累积。  

---

## 🔍 技术洞察

| 维度 | 影响描述 |
|------|----------|
| **架构** | • 采用 **Kubernetes API Validation** 的推荐模式：每个 API 类型对应一个 `validate<Type>` **pure function**，返回 `field.ErrorList`。 <br>• `sharedValidation` 充当 **request‑scoped context**（`ctx, mgr, warnings`），不再把遍历状态（field path、累计错误等）挂在资源对象上，提升 **可重入性** 与 **单元测试友好度**。<br>• 通过 `field.NewPath` 生成的路径保证 **错误路径的确定性**，对比旧实现的手写 `fmt.Sprintf` 更加可靠。<br>• 将所有 **跨资源/跨版本共享的校验**（如对象元数据、Topology、KV‑Transfer）抽离到 `shared_*` 包，避免代码重复并形成 **单一事实来源**（single source of truth）。 |
| **性能** | • 结构化校验本身不改变算法复杂度，但 **去掉了大量 `errors.Join` 与 `fmt` 拼接**，减少了运行时的字符串分配与 GC 负担。<br>• 警告与错误在同一次递归中完成收集，避免二次遍历对象。<br>• 受益于 **提前返回**（如 `if len(errs)==0 && …`），在大多数合法请求下仅执行一次路径检查。 |
| **安全** | • 所有返回的错误均使用 **`field.Invalid` / `field.Forbidden`**，错误对象只携带 **最小敏感信息**（标量值或 `field.NotSupported`），避免像旧代码中可能泄露完整资源子树（`field.Invalid` 中使用完整对象）。<br>• 统一了对 **注解、标签、用户输入**（e.g. `vllmDistributedExecutorBackend`、`KubeAnnotationDynamoKubeDiscoveryMode`）的校验，使用安全的 `k8s.io/apimachinery/pkg/util/validation/field` 进行 DNS‑1123、标签合法性校验，防止注入恶意字符。<br>• 在 **Grove‑pathway** 必要性检查中改为 **返回 `field.Forbidden`**，而不是抛出普通 `error`，从而在 Admission 中统一映射为 `Invalid`，不泄露内部实现细节。 |
| **可维护性** | • 所有 **校验函数签名统一**：`func (v *sharedValidation) validateXxx(..., fldPath *field.Path, …) field.ErrorList`，便于后续 **自动生成** 或 **IDE 跳转**。<br>• 通过 `v.warn*` 将 **warning** 聚合在 `sharedValidation.warnings`，调用方只需要返回该 slice，即可保持向后兼容的 **AdmissionWarning** 行为。<br>• 多处旧代码（正则、`errors` 包、手写 nil 检查）被删除，降低维护成本并减少潜在的 **nil‑panic**。 |
| **兼容性** | • 为 **v1alpha1** 保留完整的结构化校验路径（`validateDynamoGraphDeploymentV1alpha1`、`validateDynamoComponentDeploymentSharedSpecV1alpha1`），在 `handler.ValidateCreate` 中仍会 **转换旧对象** 并走同一套校验，保证 **向后兼容**。<br>• 采用 `sharedValidation` 而非原来的 `DynamoGraphDeploymentValidator` 成员字段，所有外部调用（如 `ValidateUpdate`）必须显式传 `ctx`、`old`、`new`，若忘记会在编译期报错。|
| **测试** | • 大量 **表驱动** 测试被保留并迁移到结构化校验路径（`shared_test.go`、`dynamographdeployment_test.go`），新增了 **字段路径校验**（`assertFieldPaths`），确保错误输出的 **路径一致性**。<br>• `handler` 的 `ValidateCreate/Update/Delete` 现在会在 **缺失 AdmissionRequest** 时返回明确错误 (`admission requires …`)。 |

## ⚠️ 潜在风险

| 风险点 | 说明 | 可能影响 |
|--------|------|-----------|
| **逻辑回归** | 校验规则在迁移过程中被重新实现，特别是 **拓扑约束 / KV‑Transfer** 的跨 ClusterTopology 检查。若细节（如 `domainIndex` 排序、错误消息文字）与旧实现不完全一致，可能导致 **旧的合法对象被误拒** 或 **错误的错误信息**。 |
| **Grove‑pathway 检查** | `grovePathwayForDynamoGraphDeployment` 返回 **布尔 + 错误信息**，但部分调用仍通过 `v.grovePathwayRequiredError`（已保留兼容实现）。若后续删除该旧方法，未同步调用方会出现 **未实现的函数** 编译错误。 |
| **旧依赖删除** | 移除了 `regexp`, `errors`, `strings` 等 import，部分旧的校验函数（如 `betaGPUMemoryService`、`betaFailover`）被弃用。若还有其他包仍引用这些私有 helper，编译会失败。 |
| **警告泄漏** | 警告通过 `sharedValidation.warn*` 聚合，若调用方忘记返回 `validation.warnings`，警告会被 ** silently 丢失**，可能影响用户对 **兼容性/废弃字段** 的感知。 |
| **资源泄露** | `sharedValidation` 持有 `ctrl.Manager`，在单元测试中需要提供 **fake manager**（已有 `newGroveTopologyTestManager`），但在生产路径若误将 `nil` 传入，将在 `readGroveClusterTopology` 中触发 **nil‑panic**。 |
| **API 版本同步** | 新增的结构化校验函数在 `shared_v1beta1.go` 与 `shared_v1alpha1.go` 中保持一致性较难，若以后添加新字段，必须在两个文件同步更新，否则 **版本不一致**。 |
| **错误信息兼容** | 原有错误文字被解析用于 **CI / 自动化**（例如 GitHub Actions 检查），迁移后错误文字变化可能导致 **CI 失效**。 |
