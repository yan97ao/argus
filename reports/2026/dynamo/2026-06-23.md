### feat(sglang): derive radix sessions from trajectory ids (#10214)
**SHA**: `5407083` | 🔗 [查看提交](https://github.com/ai-dynamo/dynamo/commit/5407083c160f7818b672546ab58ed63db1fa4fe6)

**🎯 变更类型**：功能增强 / 重构  
**⚡ 重要程度**：🔴高  
**📋 变更摘要**：本次提交将原先的 *session control*（流式会话生命周期）机制改为基于 *trajectory ID* 的 *radix KV 缓存标签*。在 SGLang 后端通过 `--enable-session-radix-cache` 参数，将 `agent_context.trajectory_id` 作为 `session_params.id` 传递，实现 KV 缓存的轨迹感知归属，而不再创建 router‑sticky 亲和或后端 streaming‑session。为此删除了所有 session_control 相关路由、生命周期、sticky‑router 实现以及对应的测试/文档。

---

### 🎯 影响范围
- **核心代码**：`components/src/dynamo/sglang/*`（请求处理、初始化、文档）  
- **路由层**：`lib/llm/src/kv_router/*`（`push_router`、`request_guard`）  
- **Sticky Session 组件**：`kv_router/sticky/*` 完全移除  
- **OpenAI / Anthropic 接口实现**：去除 `validate_nvext_semantics` 中对 `session_control` 的检查  
- **测试与示例脚本**：删除 `test_streaming_session_smoke.py` 与相关测试用例  
- **文档**：更新 SGLang Agentic 工作流、NVEXT 扩展说明，去掉 session control 章节，新增对 “Trajectory Radix Cache” 的描述  

---

### 🔍 技术洞察

| 维度            | 影响描述 |
|-----------------|----------|
| **架构影响**    | 1. **简化路由层**：去除了 `StickySessionRouter`、`SessionLifecycleController` 及其协同逻辑，降低了 router 与 backend 之间的耦合。<br>2. **新的 KV 标记路径**：在 `HandlerBase::_session_kwargs` 中仅在 `enable_session_radix_cache` 为 true 且 `trajectory_id` 有效时注入 `session_params.id`，让 SGLang 能在 radix cache 中关联该 ID（仍然走普通 radix eviction 流程）。<br>3. **配置开关**：新增 `--enable-session-radix-cache` 与 `--radix-eviction-policy priority`，通过 Dynamo 前端的 `nvext.session_radix_cache` 触发。 |
| **性能影响**    | - **KV 缓存利用率提升**：子代理的 KV 能以轨迹 ID 为标签，避免与主代理长前缀竞争，同一轨迹的多轮对话共享缓存，提升前缀复用率。<br>- **降低 RPC 开销**：不再在每个 turn 前后执行 `open_session` / `close_session` RPC，削减网络往返和后端锁竞争。<br>- **内存占用**：仍使用普通可驱逐的 radix KV（不再独占 session KV 槽），因此内存峰值与原有实现相当，且在高并发下更易被统一 eviction 策略回收。 |
| **安全考虑**    | - **攻击面缩减**：删除了 `session_control` RPC 接口，攻击者无法通过构造 `session_control` 请求进行资源占用或持久化 KV 攻击。<br>- **新风险**：`trajectory_id` 直接来自请求头 `X‑Dynamo‑Trajectory‑ID`，若未做有效性校验可能被滥用（如大量唯一 ID 导致 KV 表膨胀）。当前实现仅在 `enable_session_radix_cache` 时使用，无额外验证，建议在生产环境对 ID 长度、字符集做白名单或速率限制。 |
| **可维护性**    | - **代码量大幅削减**（约 3 k 行删除，200 行新增），移除冗余 sticky 组件，降低技术债务。<br>- **文档同步**：大量文档已更新，需确保外部 SDK、示例脚本同步改动。<br>- **向后兼容**：旧的 `session_control` 字段仍在 `NvExt` 中保留，但已不参与路由，保持向后兼容，但未来可能在后端完全移除。 |

---

### ⚠️ 潜在风险
1. **兼容性回退**：已有部署通过 `session_control` 实现子代理 KV 隔离，升级后将失效，可能导致 KV 溢出或错误的路由行为。  
2. **SGLang 依赖**：`--enable-session-radix-cache` 仅在 SGLang 源码编译版中提供，官方发行版尚未支持；若用户使用已发布的二进制，功能会默认关闭，误以为已生效。  
3. **Trajectory ID 滥用**：未限制 `trajectory_id` 的生成规则，恶意请求可制造大量独立 ID，导致 KV 表快速膨胀，触发 OOM 或 eviction 失效。  
4. **文档/SDK 不匹配**：OpenAI/Anthropic 客户端仍会在请求体中携带 `nvext.session_control`，但后端忽略，可能导致用户误解接口仍有效。  
5. **测试覆盖缺失**：原有 `test_streaming_session_smoke` 被删除，缺少针对新 radix 缓存标签的 end‑to‑end 验证，建议补充专门的 KV 轨迹标签测试。  

---

### 💡 关注建议
- **升级指南**：在发布说明中明确告知用户：若仍依赖 `session_control`，需改为使用 `agent_context.trajectory_id` 并开启 `--enable-session-radix-cache`，或保持旧版后端。  
- **参数验证**：在 `HandlerBase::_trajectory_id` 中加入长度、字符集检查（如仅允许字母数字、长度 ≤ 64），并对异常情况记录警告。  
- **监控与告警**：在 Dynamo 前端新增 `sglang:trajectory_kv_entries`、`sglang:trajectory_kv_bytes` 指标，帮助运维观察轨迹标签 KV 的增长趋势。  
- **回归测试**：补充针对 `trajectory_radix_cache` 的单元/集成测试，覆盖：① 正常传递 ID、② ID 为空或非法、③ 多轨迹并发、④ 与普通 KV 竞争的 eviction 行为。  
- **文档同步**：确保所有 SDK（Python、Node、Rust）以及示例脚本（`agg_agent.sh`、`_test_agg.sh`）均展示新 `--enable-session-radix-cache` 参数及其使用方式，删除所有 `session_control` 示例。  
- **废弃计划**：在后续 1‑2 个发行版中逐步移除 `session_control` 结构体与相关代码，提供正式的废弃公告和迁移路径。

### fix(frontend): propagate runtime context_length to vLLM frontend ModelConfig (#10827)
**SHA**: `7fa89b9` | 🔗 [查看提交](https://github.com/ai-dynamo/dynamo/commit/7fa89b9cfe96b5e36212ee4fb7bd5de01ad2e68b)

**🎯 变更类型**：Bug修复  

**⚡ 重要程度**：🔴高  

**📋 变更摘要**：  
1. 新增 `_runtime_config_context_length` 辅助函数，以安全地从 `ModelDeploymentCard.runtime_config` 中提取 `context_length` 并进行类型与范围校验。  
2. 在 `chat_engine_factory` 中，根据提取到的 `context_length` 动态为 vLLM 前端的 `ModelConfig` 注入 `max_model_len`，并通过环境变量 `VLLM_ALLOW_LONG_MAX_MODEL_LEN=1` 允许 vLLM 接受超出默认上限的模型长度。  
3. 增加单元测试覆盖各种合法与非法 `runtime_config` 场景，确保函数行为符合预期。  

**🎯 影响范围**：  
- `components/src/dynamo/frontend/vllm_processor.py`（核心模型配置构建逻辑）  
- `components/src/dynamo/frontend/tests/test_vllm_processor_unit.py`（测试）  
- 依赖 vLLM 的前端模型加载路径及其环境变量配置  

**🔍 技术洞察**：  
- **架构影响**：  
  - 将部署时的 `runtime_config.context_length` 直接映射到 vLLM 前端的 `ModelConfig.max_model_len`，实现了配置在运行时的透明传递，提升了部署灵活性。  
  - 引入全局环境变量 `VLLM_ALLOW_LONG_MAX_MODEL_LEN`，该变量在首次设置后会影响同一进程中所有后续的 vLLM 实例，需注意其全局性对其他模块的潜在副作用。  
- **性能影响**：  
  - `max_model_len` 越大，vLLM 在初始化时会分配更大的 KV 缓存与内部张量，显著增加显存/内存占用，可能导致在资源受限的节点上出现 OOM。  
  - 正确传递 `context_length` 能避免因默认长度截断导致的生成错误，提升模型推理的正确性和用户体验。  
- **安全考虑**：  
  - 仅接受正整数且大于 0 的 `context_length`，在类型或范围不符时安全地返回 `None`，防止恶意或错误配置直接破坏系统。  
  - 由于环境变量全局生效，若攻击者能够控制 `runtime_config`（如通过多租户的模型部署接口），可能诱导服务分配异常巨大的内存，形成 **拒绝服务（DoS）** 风险。  

**⚠️ 潜在风险**：  
1. **资源耗尽**：极大 `context_length`（如数十万令牌）会导致显存/内存超额分配，进而触发 OOM 或系统崩溃。  
2. **全局副作用**：`VLLM_ALLOW_LONG_MAX_MODEL_LEN` 在同一进程内对所有 vLLM 实例生效，可能影响到未显式设置 `max_model_len` 的其他模型。  
3. **错误掩盖**：当 `runtime_config.context_length` 为非整数（如字符串）时直接返回 `None`，会 silently 忽略配置错误，运维难以定位失效原因。  

**💡 关注建议**：  
- **上限校验**：在 `_runtime_config_context_length` 中加入合理的上限（如 8 MiB / 4 MiB 令牌），超出时记录警告并强制截断，以防止极端值导致 OOM。  
- **环境变量作用域**：考虑在创建每个 `ModelConfig` 前以局部方式设置 `os.environ["VLLM_ALLOW_LONG_MAX_MODEL_LEN"]`，或在完成后恢复原值，避免对进程其他部分产生副作用。  
- **监控与告警**：在模型加载阶段记录实际 `max_model_len`，并配合显存/内存使用监控，出现异常增长时触发告警。  
- **错误可观测性**：当 `runtime_config` 不符合预期时，输出明确的日志（当前已通过 `logger.info`），可进一步提升为 `logger.warning`，帮助运维快速定位配置问题。  
- **文档更新**：在部署文档中明确说明 `runtime_config.context_length` 的作用、取值范围以及对资源的影响，提示用户在多租户环境下做好配额控制。

### fix(vllm): cap omitted max_tokens to configured generation default (#10506)
**SHA**: `f3b1167` | 🔗 [查看提交](https://github.com/ai-dynamo/dynamo/commit/f3b1167e7908a07d760c1c2615287e973d01bc01)

**🎯 变更类型**：Bug修复  
**⚡ 重要程度**：🔴高  

**📋 变更摘要**  
- 修复了在 vLLM 采样参数构造时，当请求未显式提供 `max_tokens` 时，默认会采用模型可用上下文长度导致生成的 token 数可能远超用户配置的默认上限。  
- 新增逻辑：若 `max_tokens` 被省略，则先取配置的默认值（`default_sampling_params["max_tokens"]`），再与模型剩余可生成长度取最小值进行上限限制。  
- 同时在 OpenAI HTTP 接口层面加入 `PRESERVE_OMITTED_MAX_TOKENS_CONTEXT_KEY` 标记，以让后端适配器能够识别并使用其自身的生成默认值。

**🎯 影响范围**  
- `components/src/dynamo/vllm/handlers.py` – 采样参数构造核心逻辑。  
- `components/src/dynamo/vllm/tests/test_vllm_unit.py` – 新增单元测试覆盖该行为。  
- `lib/llm/src/http/service/openai.rs` – 对外 HTTP 接口层面加入上下文标记。  

**🔍 技术洞察**  
- **架构影响**：  
  - 该改动集中在 **vLLM 适配层** 与 **OpenAI HTTP 接口层**，属于业务逻辑层面的细化，未触及底层调度或模型加载。  
  - 引入 `PRESERVE_OMITTED_MAX_TOKENS_CONTEXT_KEY` 为后端适配器提供统一的“是否保留省略 `max_tokens` 的上下文”信号，提升了跨适配器行为的一致性。  

- **性能影响**：  
  - 代码仅在参数构造阶段多一次 `min` 运算和一次字典查询（获取默认 `max_tokens`），对整体吞吐量和延迟的影响可以忽略不计。

| **架构影响** | - **模块化**：快照相关实现抽离为 `dynamo.common.snapshot`，职责更单一，后续可单独演进（如支持其他容器运行时）。<br>- **恢复链路**：从“在容器启动后直接执行业务代码” → “先跑 standby 进程捕获真实 env → 写文件 → `sleep` → snapshot‑agent 恢复 → 再由业务进程重新解析 env”。 <br>- **Operator‑Runtime 解耦**：Operator 只负责注入 env 与控制卷，不再硬编码 `command`，降低对后端实现的侵入。 |

| **风险点** | 1. **环境变量遗漏**：若运维在自定义镜像中未把 `DYN_SNAPSHOT_RESTORE_STANDBY=1` 注入，恢复进程会直接启动业务代码，仍可能出现旧 env 失效的情况。<br>2. **JSON 读取错误**：文件写入失败（磁盘满、权限错误）会导致 `RuntimeError`，导致容器启动失败。<br>3. **异步解析兼容**：`refresh_snapshot_restore_config` 支持同步/异步 `parse_config`，但若业务侧自行实现异步 `parse_config`（返回协程）而未 await，可能出现未完成的配置复制。<br>4. **旧插件/镜像**：依赖原来 `sleep infinity` 占位命令的镜像会失去该行为，需要自行实现 standby 或自行注入 `DYN_SNAPSHOT_RESTORE_STANDBY=1`。 |
| **监控/可观测性** | - 新增日志 `Captured snapshot restore context at …`、`Applied snapshot restore context runtime env`、`Refreshed snapshot restore runtime config`，便于排查恢复阶段的 env 差异。<br>- 通过 `snapshot‑agent` 的日志仍可观察 `ready‑for‑snapshot` 文件的创建时机。 |

  - 通过 `Sampler::ParentBased(TraceIdRatioBased)` 注入可选采样率，实现了全局 trace 采样的轻量化控制，兼容 OpenTelemetry 采样链路。  
  - 新增的 `resolve_otlp_endpoint` 与 `resolve_signal_otlp_protocol` 函数对默认值的处理更符合 OTEL 规范，提升了配置的可预测性。  

- **性能影响**：  
  - **正面**：HTTP/Protobuf 相比 gRPC 在资源受限或不支持 HTTP/2 的环境下可避免额外的 Tonic 启动开销，尤其在容器化部署中可能降低 CPU 使用。  
  - **负面**：使用 `reqwest`（基于 hyper）会引入额外的连接池和 TLS 握手成本；如果用户频繁切换协议或端点，可能导致重复创建 exporter 的开销。采样比例的计算本身开销极低。建议在生产环境固定协议/端点，避免在高并发路径上多次读取环境变量。  

- **安全考虑**：  
  - 新增 `reqwest-rustls`，默认走 TLS（Rustls），符合现代安全最佳实践。  
  - 环境变量 `OTEL_EXPORTER_OTLP_ENDPOINT`、`*_TRACES_ENDPOINT`、`*_LOGS_ENDPOINT` 直接用于构造 URL，未对 scheme 做强制校验，若误配置为 `http://` 可能导致明文传输。建议在文档中强调使用 `https://`。  
  - 采样比例解析对非法输入仅打印警告，不会导致 panic，保持了容错性。  

**⚠️ 潜在风险**  
1. **向后兼容性**：旧版仅支持 gRPC；如果用户仍依赖默认 `OTEL_EXPORTER_OTLP_ENDPOINT` 而未显式指定协议，默认仍为 gRPC，行为保持不变。但新增的 `http-protobuf` 选项可能导致未测试的代理或防火墙阻断。  
2. **依赖膨胀**：引入 `reqwest-client`、`reqwest-rustls` 以及 `rustls-native-certs`，会显著增加二进制体积和编译时间，对嵌入式或极限资源场景可能不友好。  
3. **端点拼接逻辑**：`resolve_otlp_endpoint` 在 HTTP 协议下会盲目拼接 `path`，若用户已在通用端点中包含路径（如 `http://host/v1/traces`），会产生双路径（示例测试已暴露），可能导致导出失败。  
4. **日志导出分离**：日志与 trace 端点可以不一致，但在 HTTP 模式下仍使用相同的 `path` 参数。（当前实现为 `"/v1/traces"` 与 `"/v1/logs"`），若后端实现不支持自定义路径，可能出现兼容性问题。  
5. **采样器默认行为**：当未设置 `OTEL_TRACES_SAMPLE_RATIO` 时使用 `ParentBased` 默认采样器，这在根 span（无父）时会回退到 `AlwaysOn`，可能导致意外的全量采样。
