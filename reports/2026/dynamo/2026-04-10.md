### fix: prevent KV block leak from cancel during disagg KV transfer (#7489)
**SHA**: `9055b2d` | 🔗 [查看提交](https://github.com/ai-dynamo/dynamo/commit/9055b2d362d1085159ba08bf53a034bbb6ad47b5)

**🎯 变更类型**：Bug修复 / 架构变更  
**⚡ 重要程度**：🔴高  
**📋 变更摘要**：在分布式（disaggregated）KV 缓存传输期间，若请求被取消会导致 KV 块永久泄漏。此 PR 通过两项关键改动解决该问题：  
1. 在 Python `handler_base.py` 中实现 `_DeferredAbort`，在首次 token（即 KV 传输完成）之前延迟执行 `abort()`，并在 `generate_locally` 中根据解码模式自动包装。  
2. 在 Rust KV Router 中删除 `link_child_context`，不再把 Prefill 任务链接为 EngineContext 的子上下文，防止 Kill/Stop 信号在 RPC 传输层被提前切断导致块泄漏。  

**🎯 影响范围**：  
- `components/src/dynamo/trtllm/request_handlers/handler_base.py`（核心请求处理逻辑）  
- 相关的单元测试 `test_trtllm_handler_base.py`  
- Rust 侧 `lib/llm/src/kv_router/prefill_router/*`（Prefill 路由执行）  
- 任何使用 DisaggregationMode.DECODE 的部署环境  

**🔍 技术洞察**  

- **架构影响**：  
  - 引入了 `_Abortable` 协议，使得 `GenerationResult` 与 `_DeferredAbort` 能够统一被传递，解耦了取消路径与 KV 传输完成的时序依赖。  
  - 通过移除 Prefill 的子上下文关联，Kill/Stop 事件不再传播到 RPC 层，确保 KV 传输线程在请求被取消后仍能完整执行并回收块。  
  - 这两者共同形成了“取消 → 延迟 abort → KV 传输完成 → 真正释放资源”的新闭环。

- **性能影响**：  
  - `_DeferredAbort.abort()` 会在后台创建一个轻量的 `asyncio.Task`，仅在首个 token 到达前阻塞；任务本身对主流程的 CPU / 内存开销极低。  
  - 删除子上下文链接消除了不必要的跨线程同步，理论上可略微提升 Prefill 启动速度。  
  - 唯一潜在的性能下降是取消后仍会继续执行 Prefill（计算资源被浪费），但相较于永久块泄漏的代价，这是一种可接受的权衡。

- **安全考虑**：  
  - 代码路径未涉及外部输入的安全检查，也未引入新依赖或权限提升。  
  - 唯一需要关注的是后台任务在异常情况下的沉默捕获（`except Exception: pass`），确保不会隐藏潜在的安全或稳定性问题。

**⚠️ 潜在风险**  

1. **首 token 永不到达**：如果 KV 传输因底层错误卡死，`_DeferredAbort` 的后台任务将永远等待，导致 `generation_result.abort()` 永不被调用，仍可能残留资源。  
2. **异常吞噬**：`_wait_and_abort` 中的通用 `except Exception: pass` 可能掩盖调试信息，增加排查难度。  
3. **计算浪费**：取消请求后 Prefill 仍会执行完毕，可能在高并发场景下导致显著的算力浪费。  
4. **兼容性**：老版本依赖于 `GenerationResult.abort()` 直接触发的行为，若外部代码在取消前就假设资源已释放，可能需要适配新的延迟机制。

**💡 关注建议**  

- **监控与告警**：在生产环境中监控 `Deferred abort` 任务的完成率与耗时，若出现异常堆积，及时触发告警。  
- **超时保护**：考虑在 `_DeferredAbort._wait_and_abort` 上加入一个合理的超时阈值（如 5 s），超时后强制调用 `abort()` 并记录日志。  
- **资源回收审计**：在 KV Router 侧加入日志打印（或度量）确认每个 Prefill 完成后对应的块已被释放，防止旧逻辑残留。  
- **文档更新**：在项目的 “取消行为” 与 “Disaggregation 模式” 文档中加入 “取消后仍会完成 Prefill，且 abort 会在首 token 后触发” 的说明，帮助使用者预估算力消耗。  
- **后续优化方向**：实现真正的 NIXL‑level 取消（在 Prefill 仍在执行时即能安全回收块），以兼顾资源回收与计算浪费的平衡。

### fix: add worker dump throtteling (#8013)
**SHA**: `859bd6e` | 🔗 [查看提交](https://github.com/ai-dynamo/dynamo/commit/859bd6eadcd480365bc2a9cb0c925b8b37a5fc3e)

**🎯 变更类型**：性能优化 / 重构  
**⚡ 重要程度**：🔴高  
**📋 变更摘要**：引入 `likely_served_from_buffer` 启发式判断，提前识别可以直接从缓存读取的 KV 查询；为 `WorkerKvQueryEngine` 添加 `processing_semaphore`，将可能导致大量树转储的慢查询并发度限制为 1，同时配合 `SlowQueryGuard` 实现慢查询日志监控，防止多个路由器对同一工作节点发起并发重压。  

**🎯 影响范围**：  
- `lib/kv-router/src/indexer/local.rs`（本地 KV 索引器）  
- `lib/llm/src/kv_router/indexer/worker_query.rs`（工作节点 KV 查询引擎）  
- 相关单元测试文件  

**🔍 技术洞察**：  
- **架构影响**  
  - 在每个工作节点实例化一个 `Semaphore(1)`，形成对 “树转储” 这类高开销操作的全局排他控制，保持原有的 async 引擎接口不变。  
  - 新增 `likely_served_from_buffer` 方法，使 `WorkerKvQueryEngine` 能在进入业务路径前进行轻量判断，属于业务层的缓存命中预判，未改变底层存储结构。  
  - `SlowQueryGuard` 通过 RAII 在请求完成或被取消时自动停止慢查询日志任务，避免后台任务泄漏。  

- **性能影响**  
  - **正向**：防止并发大量树转储导致 CPU、内存、磁盘 IO 飙升，提升系统在高并发查询场景下的稳定性和响应时间。  
  - **负向**：对每次查询（即便是快路径）仍需一次 `event_buffer` 的 `Mutex` 锁检查，可能在极端高并发写入情况下产生短暂争用；但锁持有时间极短（仅读取 front 元素），对整体吞吐影响可忽略。  
  - `Semaphore` 的 acquire/release 对单次慢查询的额外开销几乎可以忽略。  

- **安全考虑**  
  - 新增同步原语不暴露新的网络入口或可执行代码，未引入明显安全风险。  
  - 通过限制并发重压，间接提升对 DoS 攻击的韧性（恶意大量慢查询请求会被排队）。  
  - 需要确保 `processing_semaphore` 不被误用导致永久占用（例如在异常路径中忘记释放），但使用 RAII _maybe_permit 变量可自动在函数返回时 Drop，风险较低。  

**⚠️ 潜在风险**：  
1. **误判导致阻塞**：`likely_served_from_buffer` 仅是启发式，若缓冲区状态在判断后立刻被清空，后续仍会触发树转储，导致请求在获取信号量后才发现仍需完整转储，增加排队等待时间。  
2. **信号量泄漏**：若后续代码在 `await` 期间出现 `panic!` 且未在 `Drop` 中释放 `permit`（Rust 的 Panic 会 unwind 并自动 Drop），理论上仍会释放，但在 `abort` 场景下需确认 `tokio` runtime 正常运行。  
3. **慢查询日志任务未及时终止**：若 `SlowQueryGuard` 的 `JoinHandle` 在 `abort` 后仍保留资源（如未立即回收），可能导致微量内存泄漏。  
4. **测试覆盖不足**：当前仅在测试初始化时添加信号量，未对并发慢查询、信号量排队、日志守护等场景进行完整验证，可能在实际生产中出现未捕获的竞争问题。  

**💡 关注建议**：  
- **完善测试**：加入并发压力测试，验证在高并发慢查询情况下仅有一个请求进入树转储路径，其他请求能快速返回或正确排队。  
- **监控指标**：在生产环境中添加 `semaphore_acquire_duration`、`buffer_fast_path_ratio`、`slow_query_guard_active_seconds` 等指标，帮助评估启发式的命中率和排队时间。  
- **可配置阈值**：考虑将 `Semaphore::new(1)` 替换为可配置的并发上限（基于机器资源），并在配置文件中暴露，以便不同部署规模调优。  
- **文档说明**：在项目的 KV 查询模块文档中注明 “worker dump throttling” 的工作原理、启发式判断的限制以及如何调试慢查询日志。  
- **审计异常路径**：确认在 `engine_ctx.stopped()`、`engine_ctx.killed()` 或其他错误返回分支中，已正确 Drop `_maybe_permit`，防止信号量永久占用。  

通过上述措施，可进一步降低新增同步机制的风险，并最大化其对系统稳定性和性能的收益。

### fix(snapshot): use watch-based waits in snapshotctl (#8024)
**SHA**: `a0ecd4d` | 🔗 [查看提交](https://github.com/ai-dynamo/dynamo/commit/a0ecd4dd7fc0e8e9cfee088bb6e36371861b5e93)

**🎯 变更类型**：Bug修复 / 性能优化 / 架构变更  
**⚡ 重要程度**：🔴 高  
**📋 变更摘要**：  
1. 将快照控制器（`snapshotctl`）以及内部恢复/检查点等待逻辑从基于 `wait.Poll` 的轮询改为基于 Kubernetes **watch** 的事件驱动等待，显著降低 API 轮询频率并提升超时诊断信息。  
2. 引入 *Kubernetes discovery* 相关的 ServiceAccount、Role、RoleBinding 自动创建，确保检查点作业在非默认命名空间中拥有正确的权限。  
3. 将 Helm 配置中 `restoreReadyTimeoutSeconds` 重命名为 `restoreTimeoutSeconds`，并在代码中统一使用新的超时概念（恢复整个尝试的最大时长）。  
4. 为容器恢复时的 **startupProbe** 添加合成逻辑，以最大 `FailureThreshold` 防止因探针超时导致恢复被误判为失败。  
5. 多处调用签名改动，使恢复/检查点功能显式接受 `namespace` 与 `discovery_backend` 参数，提升可测试性与可维护性。

### 🔍 技术洞察

| 维度 | 影响分析 |
|---|---|
| **架构影响** | - 引入 `watchNamedObject` 作为统一的 Watch‑wrapper，减少了对 `k8s.io/apimachinery/pkg/util/wait` 的依赖，使代码层级更清晰。<br>- 新的 Discovery 资源（SA、Role、RoleBinding）由控制器自动同步，避免了手动配置错误，提高了 **operator** 对检查点作业的自愈能力。 |
| **性能影响** | - **正向**：原先每 2 秒一次的轮询（`wait.PollUntilContextCancel`）会对 API Server 产生持续负载，改为 **一次性 Watch**，只在对象状态变化时触发，大幅降低网络·CPU 消耗。<br>- **潜在**：Watch 需要保持长连接，若网络不稳定或 Watch 长时间无事件，可能导致连接重建开销；但整体仍优于频繁轮询。 |
| **安全考虑** | - 新增的 ServiceAccount/Role/RoleBinding 采用最小权限（仅 `get/watch/list`），未引入额外的权限风险。<br>- `watchNamedObject` 在接收到 `watch.Error` 时直接返回对应 `apierrors`，避免因错误事件被误当作正常状态。<br>- `RestoreTimeout` 现在强制大于 0，防止无限期挂起导致资源泄漏。 |
| **可维护性** | - 参数化 `namespace` 与 `discovery_backend` 的传递，使函数更易单元测试，减少全局变量依赖。<br>- 将探针合成抽离为 `ensureRestoreStartupProbe`，逻辑单一，测试覆盖完整。<br>- 配置字段统一使用 `RestoreTimeoutSeconds`，所有调用统一走 `RestoreSpec.RestoreTimeout()`，降低配置分散的风险。 |
| **兼容性** | - **Breaking Change**：`values.yaml` 中的字段名称变更，旧版 Helm chart 会因找不到 `restoreReadyTimeoutSeconds` 而回退到默认值（0），导致恢复默认无限等待。升级指南需明确迁移。<br>- 调用 `reload_snapshot_restore_identity` 的代码必须传入 `namespace` 与 `discovery_backend`，已在项目内部同步，但外部插件若自行调用需更新。 |
| **错误诊断** | - 对超时场景增加 `checkpointTimeoutSummary` / `restoreTimeoutSummary`，在日志中提供 Pod/容器状态、等待原因，帮助运维快速定位问题。 |

### ⚠️ 潜在风险
1. **Watch 超时行为不一致**：若 Controller‑Manager 的 `--watch-cache` 被禁用，Watch 可能变慢，导致 `context.DeadlineExceeded` 提前触发。  
2. **RBAC 自动创建失败**：如果集群的 `ClusterRole`/`ClusterRoleBinding` 限制导致 ServiceAccount、Role 创建受阻，检查点作业将缺失必要权限，恢复/检查点会始终失败。  
3. **配置兼容性**：旧版 Helm values 中仍保留 `restoreReadyTimeoutSeconds`，在升级后会被忽略，可能导致用户误以为仍生效，从而出现意外的无限等待。  
4. **探针合成错误**：在极端情况下（容器没有任何探针），`ensureRestoreStartupProbe` 会返回 `nil`，导致恢复后缺少 `startupProbe`，但代码已在 `PrepareRestorePodSpec` 中对 `nil` 做了容错处理。  
5. **双写状态**：`runRestore` 中在失败路径上多次调用 `setRestoreStatus(RestoreStatusFailed)`，若第一次写入成功后第二次因网络错误返回错误，状态仍保持 `Failed`，但函数会返回错误，可能导致上层重复重试。该行为虽保持幂等，但需注意日志噪声。  

---

### 💡 关注建议
1. **升级指南**  
   - 在 Helm 升级时显式提醒用户将 `restoreReadyTimeoutSeconds` 替换为 `restoreTimeoutSeconds`，并根据业务需求设置合理的秒数（默认 7200 s）。  
   - 检查自定义 `Snapshot` CR 的 `spec.restore` 中是否已经使用新字段；若仍使用旧字段，自动迁移脚本或 `kubectl patch` 可能需要提供。  

2. **RBAC 验证**  
   - 在 CI 中加入对 `CheckpointJob` 所在命名空间的 ServiceAccount、Role、RoleBinding 自动同步的集成测试，确保 Operator 在受限集群（如 `OPA` 限制）下仍能成功创建。  
   - 记录并监控 `reconcileK8sDiscoveryResources` 返回的错误，必要时通过 Operator 的 `Event` 暴露给用户。
