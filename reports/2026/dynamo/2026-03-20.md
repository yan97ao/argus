### feat: surface status and errors for profiler in DGDR (#6855)
**SHA**: `740130e` | 🔗 [查看提交](https://github.com/ai-dynamo/dynamo/commit/740130eb65b947ca09fc5ba5d2336d77eceb3e55)

**🎯 变更类型**：功能增强  
**⚡ 重要程度**：🔴高  
**📋 变更摘要**：  
1. 为 DGDR（DynamoGraphDeploymentRequest）引入 *ProfilingPhase* 子状态，扩展 `ProfilerOperationalConfig.current_phase`，在 profiler Python 代码中实时写入 `profiler_status.yaml`（包括 phase、message、error）。  
2. Sidecar 脚本轮询该文件并将 phase/message 写入输出 ConfigMap；Operator Go 控制器读取该 ConfigMap，更新 DGDR 状态、Condition Reason，并提供细粒度的失败原因。  
3. CRD 与 Helm Chart 相应调整（去除 `priority`、添加 `profilingPhase` 列），以及对日志收集脚本标签查询的兼容性改进。  

**🎯 影响范围**：  
- `components/src/dynamo/profiler/*`（Python profiler 逻辑、状态写入）  
- `deploy/operator/internal/controller/dynamographdeploymentrequest_controller.go`（Go 控制器）  
- `deploy/operator/api/v1beta1/dynamographdeploymentrequest_types.go`、CRD yaml（API）  
- `deploy/operator/internal/controller/dynamographdeploymentrequest_controller_test.go`（单元测试）  
- `deploy/utils/dynamo_deployment.py`（日志收集脚本）  
- Helm Chart 与 `.gitignore`（运维文件）

### 🔍 技术洞察  

| 维度 | 影响评估 |
|------|----------|
| **架构影响** | - **新增子阶段**：在 DGDR 状态机中加入 `ProfilingPhase`，让用户和监控系统能够看到 profiler 的细化进度（初始化、并行度扫描、配置选择、曲线构建、生成 DGD、完成）。<br>- **Sidecar‑Controller 协同**：Sidecar 通过 `kubectl apply` 将 phase 写入 ConfigMap，Controller 通过 watch 读取并同步到 CR 状态，实现 UI/Kubectl 可视化。<br>- **向后兼容**：仅在新版本的 profiler 中写入 `phase`，旧版仍可工作（phase 为空时控制器不更新）。<br>- **CRD 扩展**：`status.profilingPhase` 与 `spec` 无关，仅是状态字段，兼容现有资源。 |
| **性能影响** | - **Profiler 端**：写入 `profiler_status.yaml` 的频率仅在阶段切换时（几次），几乎不影响 profiling 计算。<br>- **Sidecar**：轮询间隔从 5 s 改为 10 s，CPU/IO 开销极低；`kubectl apply` 只在 phase 变更时触发，避免频繁写入。<br>- **Operator**：额外的 ConfigMap watch 与状态更新，资源占用可忽略（每个 DGDR 最多几次状态变更）。 |
| **安全考虑** | - Sidecar 通过 `kubectl apply` 写入 ConfigMap，需要 **`configmaps` 读写** 权限；已在 Operator RBAC 中声明（`MockRBACManager` 及实际 `ClusterRole` 需同步更新）。<br>- 仅写入 **phase**、**message**、**error**，不包含敏感模型数据，风险低。<br>- `profiler_status.yaml` 位于 Profiler 容器的工作目录，仍受容器文件系统隔离。 |
| **可维护性** | - **代码结构**：Profiler 端统一使用 `write_profiler_status`，新增 `phase` 参数，保持单点维护。<br>- **Controller**：抽取 `updateProfilingSubPhase`、`profilingPhaseReason`、`profilingPhaseFailureReason`，逻辑清晰，易于单元测试（已在 `*_test.go` 中覆盖）。<br>- **CRD**：打印列已去掉 `priority`，简化 `kubectl get` 输出；新增 `profilingPhase` 打印列可直接观察。 |
| **可靠性** | - 在 profiler 异常退出时仍会写入 `FAILED` 状态并携带 `phase`（如果已经设定），Controller 能够根据 `phase` 生成更精准的失败 Reason。<br>- Sidecar 采用 **去抖**（只在 phase 变化时写入）防止 ConfigMap 更新风暴。<br>- `isValidProfilingPhase` 对未知值抛错，防止误写入非法状态。 |

### ⚠️ 潜在风险  

1. **升级兼容**  
   - 旧版 Profiler（未写 `phase`）升级后仍能与新 Operator 交互，但 Operator 可能在读取 ConfigMap 时找不到 `phase`，导致 **无进度显示**。需在文档中说明兼容策略或在 `updateProfilingSubPhase` 中容忍缺失。  

2. **RBAC 漏洞**  
   - Sidecar 使用 `kubectl apply`，若 ServiceAccount 权限不足，会导致 **phase 更新失败**，用户只能看到 “Running”。需要确保 `ClusterRole` 包含 `configmaps get,create,update,patch`。  

3. **ConfigMap 大小 & 写入冲突**  
   - 虽然仅写入少量字段，但若同一 DGDR 多个 sidecar（误配置）并发 `apply`，可能产生 **冲突**。建议在 sidecar 脚本中加 `--server-side-apply`（已默认）或使用 `kubectl patch`。  

4. **错误信息泄露**  
   - `error` 字段会原样写入 ConfigMap，若错误信息包含内部实现细节，可能被外部用户看到（通过 `kubectl get cm`). 建议在生产环境对敏感错误做脱敏。  

5. **测试环境依赖**  
   - 新增的 `profilingPhase` 列表、ConfigMap watch 以及 sidecar脚本在 CI 环境可能需要额外的 mock（已在 `*_test.go` 中加入），但如果 CI 未启用 `kubectl`，侧车脚本可能失败。确保 CI 仅运行 Go 单元测试（不调用 sidecar）。

### feat(llm): unify user-facing priority hints (#7492)
**SHA**: `ebd06a1` | 🔗 [查看提交](https://github.com/ai-dynamo/dynamo/commit/ebd06a1f320ac5ac23d979dc8ee53d419be2719b)

**🎯 变更类型**：功能增强 / 架构变更  
**⚡ 重要程度**：🔴高  
**📋 变更摘要**：  
- 将原先用于路由层的 `latency_sensitivity` 与后端引擎层的 `priority` 统一为单一的用户可见字段 `priority`，并在文档、示例、CLI 参数、Python 绑定以及内部协议中全部迁移。  
- 为向后兼容保留了 `latency_sensitivity` 的内部解析（在 `AgentHints` 中做别名），但在外部已不再推荐使用。  
- 新增对 `priority` 的整数化、取负（对应 vLLM 低值高优先级）以及对 `--schedule-low-priority-values-first` 标志的显式禁止的检查。

**🎯 影响范围**：  
- **CLI/配置**：`components/src/dynamo/sglang/args.rs`、`router` 参数文档、`--schedule-low-priority-values-first` 校验。  
- **请求处理**：`handler_base.rs`（统一优先级归一化）、`vllm/handlers.rs`（优先级取负）、`sglang/request_handlers`（同上）。  
- **预处理 & 协议**：`lib/llm/src/preprocessor.rs`、`lib/llm/src/protocols/openai/nvext.rs`（结构体字段改名）。  
- **Python 绑定 & 示例**：`benchmarks/router/*`、`python` 示例脚本、`lib/bindings/python` 类型定义。  
- **文档**：所有 `nvext`、`router`、`SGLang`、`agentic_workloads`、`features` 等章节的字段说明与示例均已同步更新。  
- **测试/示例**：基准测试脚本 `real_data_priority_benchmark.py`、`README` 中的使用示例均改为 `priority`。

**🔍 技术洞察**：

| 维度 | 影响分析 |
|------|----------|
| **架构影响** | - 统一的 `priority` 成为 Dynamo API 的唯一调度入口，简化了前端（router）与后端（SGLang/vLLM）之间的约定。<br>- 通过 `handler_base._priority_kwargs` 对 `schedule_low_priority_values_first` 进行显式阻断，防止后端实现与 Dynamo 约定不一致导致的调度错误。<br>- `priority` 在路由层转为 “effective arrival time” 进行排序，在后端层再做后端‐特定的极性归一（vLLM 取负、SGLang 保持正）。 |
| **性能影响** | - **正向**：去除两个独立字段的解析与转换，降低请求预处理的分支路径，微幅提升吞吐。<br>- **负向**：在 vLLM 路径中需要一次 `-int(priority)` 的取负，开销可忽略。<br>- 统一后，调度策略更一致，避免因 `latency_sensitivity` 与 `priority` 不匹配导致的误排队，潜在提升高优先级请求的 TTFT（尤其在 `--router-queue-threshold` 开启时）。 |
| **安全考虑** | - 新增对 `priority` 的整数解析 (`parse_int_list`) 并在 `handler_base` 中强制 `int()`，可以捕获恶意的非数值输入，提升输入校验安全性。<br>- 仍需确保 `priority` 范围合理（建议在业务层做上下限检查），防止极端负数导致资源抢占异常。 |
| **可维护性** | - 文档、示例、代码层统一使用同一字段，大幅降低概念散落的认知成本。<br>- 旧字段 `latency_sensitivity` 仍保留在内部结构体（作为别名），便于渐进式迁移，不会立即导致破坏性回归。 |

**⚠️ 潜在风险**：

1. **向后兼容**：外部客户端仍然发送 `latency_sensitivity`，若未更新到新版 SDK，路由层将不再识别该字段，从而失去调度优先级。  
2. **`--schedule-low-priority-values-first`**：该 flag 在 SGLang 集成被硬性禁用，若用户在自建 SGLang 环境中仍使用，启动时会抛 `ValueError`，可能导致服务不可用。  
3. **负优先级**：在 vLLM 中取负后，原本的负数会变成正数，可能意外提升低优先级请求的调度权重。需要业务层对 `priority` 做非负约束。  
4. **文档同步风险**：大量文档改动，若仍有遗漏（如旧 API 章节），用户搜索时会产生混淆。  
5. **测试覆盖不足**：所有涉及 `priority` 的路径（router、sglang、vllm、trt-llm）需重新跑完整回归，确保取负、归一化逻辑在极端值下行为一致。

**💡 关注建议**：

- **迁移指南**：在发行说明中提供明确的 `latency_sensitivity → priority` 迁移表，并在 SDK 中加入一次性警告日志，提醒开发者更新字段。  
- **输入校验**：在 `parse_int_list` 与 `AgentHints` 结构体加入范围检查（例如 `priority >= 0 && priority <= 1000`），防止异常值导致调度失控。  
- **回退兼容**：考虑在 `handler_base._priority_kwargs` 中检测 `latency_sensitivity` 并自动映射到 `priority`（记录一次性 deprecation 警告），以免突发的客户端升级导致业务中断。  
- **文档审计**：组织一次全链路文档审查，确保所有 `latency_sensitivity` 关键字已被替换或标注为已废弃。  
- **CI/测试**：新增单元测试覆盖：<br>  - `parse_int_list` 正负数解析。<br>  - `handler_base` 对 `schedule_low_priority_values_first` 的异常抛出。<br>  - vLLM、SGLang 两种后端下的优先级取负/保持逻辑一致性。  
- **监控/指标**：在路由层将 `priority` 值写入日志/监控标签，便于后期观察优先级分布与实际调度效果的对应关系。  

通过上述措施，可将此次 **统一调度提示** 的业务价值（简化 API、提升调度一致性）落地，同时将潜在的兼容性与安全风险降到最低。

### feat: [vLLM] add remote embedding loader to prefill/decode worker. Port E/P/D usage (#7507)
**SHA**: `4d302ab` | 🔗 [查看提交](https://github.com/ai-dynamo/dynamo/commit/4d302ab6c969871e0b7f7b83d3777c5946f1cb9f)

**🎯 变更类型**：功能增强 / 架构变更 / 重构  
**⚡ 重要程度**：🔴 高  
**📋 变更摘要**  
本次提交为 vLLM 多模态工作流引入 **远程嵌入加载器**，实现了 **Encode‑Worker → Prefill/Decode‑Worker** 的分布式编码路径。  
- 新增 `MultiModalEmbeddingLoader`、`MultimodalEmbeddingCacheManager` 与多种 **EmbeddingTransferMode**（LOCAL、NIXL_WRITE、NIXL_READ）。  
- `BaseWorkerHandler` 及其子类（Prefill、Decode、Multimodal PD）改为接收 `Config`，在构造时自动创建 **embedding_loader** 并在 `_extract_mult multimodal_data` 中优先走远程加载。  
- `WorkerFactory` 验证逻辑放宽，统一使用 `handles` 判断，移除对部分组合的 `NotImplementedError`，脚本和测试同步更新。  

**🎯 影响范围**  
- `components/src/dynamo/vllm/handlers.py`（核心工作流入口）  
- `components/src/dynamo/vllm/multimodal_handlers/*`（Prefill、Decode、PD 工作器）  
- `components/src/dynamo/vllm/multimodal_utils/prefill_worker_utils.py`（嵌入加载实现）  
- `components/src/dynamo/vllm/worker_factory.py`（工厂创建）  
- 示例启动脚本与相关单元测试  

**🔍 技术洞察**  

- **架构影响**  
  1. **解耦编码阶段**：原本在 vLLM 引擎内部完成图像/视频嵌入，现在可以由独立的 Encode‑Worker 提前计算，减轻主工作节点 GPU 负担。  
  2. **新增组件**：`MultiModalEmbeddingLoader` 负责远程请求、接收、缓存；`MultimodalEmbeddingCacheManager` 为本地缓存提供容量管理。  
  3. **配置统一**：所有工作器统一接受 `Config`，通过 `EmbeddingTransferMode` 控制传输方式，扩展性更好。  
  4. **Factory 逻辑简化**：`WorkerFactory._validate_config` 仍保留基本校验，但不再硬性阻断 “route‑to‑encoder + prefill” 场景，为后续完整的分布式多模态协议铺路。  

- **性能影响**  
  - **正向**：远程编码可在专用 GPU（或 CPU）上并行完成，对 **Prefill/Decode** 工作节点的显存占用显著下降，尤其在单卡高并发场景。  
  - **负向**：引入跨进程/网络的 **embedding fetch**，增加 **额外的网络 RTT 与序列化开销**，对端到端延迟有潜在影响。  
  - **缓存机制**：在 `multimodal_embedding_cache_capacity_gb > 0` 时会在两侧（vLLM 引擎内部 & loader）各持有一份缓存，可能导致 **显存/系统内存重复**，需监控实际使用。  

- **安全考虑**  
  - 远程 Encode‑Worker 通过 **NIXL** 或自定义本地文件进行嵌入传输，涉及 **跨机器通信**。若使用 NIXL（写/读）需确保 **共享内存/Side‑channel** 受可信环境保护，防止数据泄露或中间人篡改。  
  - `EmbeddingTransferMode` 参数来源于用户配置，建议在生产环境对其进行 **白名单校验**，防止不受信的模式被开启。  
  - 目前代码没有显式的 **身份验证** 与 **加密**，如在不可信网络中使用，需要在外层（如 TLS 隧道）自行防护。  

**⚠️ 潜在风险**

- 对 PVC、S3、OCI 三种存储实现做了路径拼接改造，加入 `versions/<version>` 目录层级，并在 `InjectCheckpointIntoPodSpec` 中保留已有的 storage 信息。

**🎯 影响范围**：  
- `deploy/operator/internal/checkpoint/*`（路径解析、PodSpec 注入、资源创建）  
- `deploy/operator/internal/controller/dynamocheckpoint_controller.go`（调度逻辑、状态更新）  
- `deploy/operator/internal/consts/consts.go`（常量新增）  
- 相关单元测试文件及 CI 测试用例。

**🔍 技术洞察**  

- **架构影响**：  
  - 引入 *artifact version* 成为 Checkpoint 路径与 Job 名称的必选组成部分，使得同一 hash 可以拥有多个不可变快照，提升了快照管理的可读性和回滚粒度。  
  - 在 Controller 中加入 `desiredArtifactVersion` 与 `desiredCheckpointJobName` 两个辅助函数，所有后续处理统一走版本化路径，增强了一致性。  
  - 新的 duplicate‑hash 检测通过 `FindCheckpointByIdentityHash` 在创建前做一次全局查找，防止资源竞争导致的不可预期覆盖。

- **性能影响**：  
  - 版本化路径的拼接仅在控制面（Controller）完成，对运行时（Pod）几乎没有额外开销。  
  - 增加一次 `FindCheckpointByIdentityHash` 查询，可能略微提升 API 调用频率，但受限于既有的缓存/Informer 机制，影响可忽略。

- **安全考虑**：  
  - 注解 `snapshot-artifact-version` 由用户可写，若恶意用户自行指定非常规 version（如空字符串、路径穿越）可能导致路径构造异常。代码已通过 `strings.TrimSpace` 并 fallback 到默认值，基本防护到位。  
  - 仍需确保底层存储（PVC、S3、OCI）的访问控制不因版本化路径改变而泄露或覆盖已有数据。

**⚠️ 潜在风险**  

1. **兼容性**  
   - 旧版 Checkpoint（无 `artifact-version` 注解）在升级后仍会使用默认 version `1`，但存储路径从 `.../<hash>` 变为 `.../<hash>/versions/1`，若已有持久化数据未迁移，可能导致“找不到快照”错误。  
   - 需要在升级文档或提供迁移脚本，将旧路径软链接或复制到新路径。

2. **重复哈希检测**  
   - 当已有的 `DynamoCheckpoint` 被标记为 `Ready` 并拥有相同 hash，新的 Checkpoint 即使指定不同 version 也会被标记为 `Failed`（因为 hash 冲突）。这可能限制用户在同一 namespace 中对同一模型的多版本快照需求。  
   - 若业务场景需要在同一 hash 上保留多个版本，当前实现会阻断，需要额外的 “allow‑multiple‑versions” 开关或采用不同 hash 生成策略。

3. **Job 名称长度**  
   - 新的 Job 名称加入 `-<version>`，在极端情况下（长 namespace+hash+annotation）可能接近 Kubernetes 名称长度上限（63 字符），导致 Job 创建失败。

4. **测试覆盖不足**  
   - 虽然大幅度补充了单元测试，但对异常路径（如 storage 配置缺失、annotation 为空、Trim 结果异常）仍缺少负面路径验证。

**💡 关注建议**  

- **升级指南**：在发布说明中明确提示用户在升级到 vX.Y 之前，需要迁移已有快照目录或在 `CheckpointConfiguration` 中保留向后兼容的旧路径。  
- **监控告警**：添加对 `DynamoCheckpointPhaseFailed` 且错误信息包含 “identity hash … already owned” 的告警，帮助快速定位重复创建冲突。  
- **容错策略**：考虑在 `desiredCheckpointJobName` 中对超长名称做哈希压缩或截断，以避免 K8s 名称上限导致的创建失败。  
- **安全审计**：对 `snapshot-artifact-version` 注解的值做更严格的校验（例如正则 `^[0-9]+$`），防止非法字符进入文件系统路径。  
- **文档与示例**：在 operator README 中补充使用案例，展示如何通过注解升级快照版本以及如何手动触发新版本快照。  

通过上述改动，Dynamo 的 checkpoint 机制在可读性与可管理性上得到显著提升，但在升级迁移和多版本并存场景仍需业务侧配合相应的流程与监控。
