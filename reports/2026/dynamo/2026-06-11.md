### feat(planner): expose SLA target metrics to Prometheus (#10032)
**SHA**: `08f1ed3` | 🔗 [查看提交](https://github.com/ai-dynamo/dynamo/commit/08f1ed3a80c3193fb0591e5978947cf5cbac1425)

**🎯 变更类型**：功能增强  

**⚡ 重要程度**：🔴高  

**📋 变更摘要**  
1. 为 Dynamo Planner 引入两项 SLA 目标指标（`sla_target_ttft_ms`、`sla_target_itl_ms`）并在 Prometheus 中以 Gauge 形式暴露。  
2. 在 `PlannerBase.__init__` 启动 Prometheus HTTP 服务后，将配置中 `ttft_ms` 与 `itl_ms` 的静态值写入对应 gauges，实现“一次写入”。  
3. 完备单元测试，验证指标在不同配置、Prometheus 开关状态下的写入行为以及指标对象的创建。

**🎯 影响范围**  
- `components/src/dynamo/planner/core/base.py`（Planner 初始化路径）  
- `components/src/dynamo/planner/monitoring/planner_metrics.py`（Prometheus metrics 注册）  
- `components/src/dynamo/planner/tests/unit/test_metric_publication.py`（单元测试）  
- 任何使用 Prometheus 抓取 Dynamo Planner 指标的监控/仪表盘（Grafana 等）  

**🔍 技术洞察**  

| 维度 | 影响描述 |
|------|----------|
| **架构影响** | - **监控子系统扩展**：在现有 `PlannerPrometheusMetrics` 中新增两个 Gauge，保持统一的 metric 前缀 (`{PREFIX}_`)。<br>- **初始化时写入**：在 `BasePlanner.__init__` 完成 Prometheus 服务器启动后立即写入 SLA 目标值，属于一次性静态指标，避免在后续 tick 中重复写入，符合“一次写入、长期展示”的监控模型。<br>- **配置依赖**：指标值直接来源于 `PlannerConfig.ttft_ms` 与 `PlannerConfig.itl_ms`，因此配置层的变更会自动体现在 Prometheus。 |
| **性能影响** | - **极小开销**：Gauge 的 `set` 调用是 O(1) 的内存写入，且仅在 Planner 启动时执行一次，对运行时的 CPU、内存和网络带宽几乎无感。<br>- **启动时路径**：若 `metric_reporting_prometheus_port` 为非零，`start_http_server` 已在原有代码中调用，新增的两次 `set` 不会显著延长启动时间。 |
| **安全考虑** | - **信息泄露风险**：SLA 目标值是运营层面的业务参数，一般不属于敏感信息，公开到 Prometheus（默认不做身份鉴权）是可接受的。<br>- **配置误用**：若运维误将 `metric_reporting_prometheus_port` 设为 0（关闭）但仍期待仪表盘展示 SLA 目标，可能导致误判。建议在文档中明确该指标仅在 Prometheus 启用时可见。 |
| **可维护性** | - **代码可读性**：新增的注释与测试文档化了“一次写入”行为，降低后续维护者误删或误改的概率。<br>- **测试覆盖**：单元测试覆盖了默认、定制以及禁用 Prometheus 三种场景，确保未来改动不易破坏该特性。 |

**⚠️ 潜在风险**  

1. **初始化顺序不一致**  
   - 若 `PlannerPrometheusMetrics` 的实例化顺序在未来代码中被调整，写入 SLA gauges 的调用可能在 Prometheus HTTP 服务器真正启动之前执行，导致指标写入失败或被丢弃。  
2. **配置缺失或类型错误**  
   - `PlannerConfig` 中的 `ttft_ms` / `itl_ms` 若未提供（`None`）或非数值类型，当前代码会直接调用 `set`，可能抛出异常或写入错误值。  
3. **重复注册**  
   - 如果在同一进程中多次实例化 `PlannerPrometheusMetrics`（例如在测试或多实例场景），Prometheus client 会因重复注册同名 Gauge 而报错。  
4. **监控数据误解**  
   - SLA 目标是静态配置值，若运维在运行时修改 `PlannerConfig` 而未重启 Planner，Prometheus 中的目标值不会更新，导致仪表盘误报。  

**💡 关注建议**  

| 建议 | 说明 |
|------|------|
| **初始化顺序防护** | 在 `PlannerBase.__init__` 中加入容错：`if hasattr(self.prometheus_metrics, "sla_target_ttft_ms"): ...` 并在 `start_http_server` 成功后再写入，或捕获异常记录日志。 |
| **配置校验** | 在 `PlannerConfig` 校验阶段明确 `ttft_ms` 与 `itl_ms` 为正数（非 `None`），并在 `PlannerBase.__init__` 做一次安全断言，防止异常写入。 |
| **重复实例防护** | 对 `PlannerPrometheusMetrics` 实例化做单例或进程级缓存，或在指标创建时使用 `Prometheus.registry.get_metric` 防止重复注册导致的 `ValueError: duplicated timeseries`。 |
| **动态 SLA 更新** | 若业务需要在运行期间调节 SLA，考虑提供一个 `update_sla_targets(ttft_ms, itl_ms)` 方法，既能更新配置，又能重新 `set` 对应 Gauge。 |
| **文档与监控** | 在运维文档中标注：*SLA 目标指标仅在 Prometheus 导出开启且 Planner 重启后更新*。仪表盘应使用 `*_target_*` 作为基准线，而非动态阈值。 |
| **测试持续性** | 将新增加的测试文件加入 CI 的覆盖率检查，确保未来对 `PlannerPrometheusMetrics` 的改动不会意外移除或改名这些 Gauge。 |

---  

**结论**：本次改动在功能层面为 Dynamo Planner 添加了对 SLA 目标的可观测性，价值明显；对系统架构、运行性能几乎没有负面影响，安全风险低。但为了防止初始化顺序、配置异常和重复注册导致的潜在故障，建议在代码层面加入防护逻辑并在文档上做好说明。整体风险属于 **低到中**，可在当前提交的基础上快速上线。

### feat: borrow BSI shard-read suffixes (#10495)
**SHA**: `3c2f491` | 🔗 [查看提交](https://github.com/ai-dynamo/dynamo/commit/3c2f49121ea8c69ec7dceba2327fb7b1497ae751)

**🎯 变更类型**：功能增强 / 性能优化  
**⚡ 重要程度**：🔴高  
**📋 变更摘要**：  
- 将 `find_matches_from_anchor` 的签名从接受 `Vec<LocalBlockHash>` 改为借用切片 `&[LocalBlockHash]`，消除不必要的内存分配与拷贝。  
- 在 `BranchShardedIndexer` 中对 `FxHashSet` 与 `scores.scores` 进行容量预留，提升集合操作的性能。  
- 相应测试及实现代码同步更新，以使用借用的 suffix。  

**🎯 影响范围**：  
- `lib/kv-router/src/indexer/branch_sharded.rs`（核心路由与调度逻辑）  
- `lib/kv-router/src/indexer/shard_handle.rs`（抽象 `AsyncShardHandle` 接口）  
- 所有实现 `AsyncShardHandle` 的具体后端（当前仅 `ThreadPoolIndexer`，但其他实现若存在也需修改）  
- 单元测试 `branch_sharded`  

**🔍 技术洞察**：

- **架构影响**  
  - 接口层从拥有所有权 (`Vec`) 转为借用 (`&[LocalBlockHash]`) ，保持业务逻辑不变的前提下，降低了跨层数据传递的所有权成本。  
  - 需要确保所有 `AsyncShardHandle` 实现均遵循新的生命周期约束 `'a`，否则会出现编译错误。  

- **性能影响**  
  - **内存**：避免了在每次读取路径上创建新 `Vec`，显著降低短生命周期分配和 GC（内存回收）压力。  
  - **CPU**：`reserve` 与 `with_capacity_and_hasher` 预分配集合容量，减少了哈希表扩容次数及散列冲突，提升 `collect_live_workers` 与 `reconcile_active_workers` 的遍历/过滤效率。  
  - **延迟**：在高并发读取场景下，削减了分配/拷贝导致的微秒级抖动，预计整体吞吐提升 3%~10%（需实际基准验证）。  

- **安全考虑**  
  - 引入了显式生命周期 `'a`，要求调用者在 `await` 之前保持 `suffix` 的有效性。由于所有调用点在同一 async 块内部 `await`，不存在跨线程持有悬空引用的风险。  
  - 未改变任何加密、授权或网络交互逻辑，安全风险基本为零。  

**⚠️ 潜在风险**：  
1. **生命周期泄漏**：若未来出现将返回的 `Future` 保存或跨线程传递（例如 `Box::pin` 后存入结构体），编译器会阻止但可能导致开发者绕过限制（使用 `unsafe`），需保持警惕。  
2. **实现遗漏**：项目中若还有未被更新的 `AsyncShardHandle` 实现（如自定义后端或测试 mock），会因签名不匹配导致编译错误或运行时未覆盖的路径。  
3. **API 兼容性**：外部使用者已经依赖 `Vec` 参数的旧接口，升级后需要同步迁移调用方代码。  

**💡 关注建议**：  
- **回归测试**：运行全套集成与压力测试，确保在高并发情况下没有因借用导致的意外 panic。  
- **基准对比**：使用 `criterion` 或内部 benchmark 开关，对比 `Vec` 版与 `&[ ]` 版在典型读取路径（例如 10k 并发）下的延迟与内存占用。  
- **文档更新**：在 `AsyncShardHandle` trait 文档中标明 `suffix` 为借用切片并解释生命周期要求，防止外部使用误解。  
- **代码审查**：检查项目中所有 `find_matches_from_anchor` 的实现与调用，确保无遗漏并且 `suffix` 的来源在 `await` 前保持有效。  
- **警惕跨线程持有**：若未来计划将 `AsyncShardHandle` 实例迁移到 `tokio::spawn` 等跨线程任务，确认 `suffix` 生命周期仍满足 `Send + 'static` 要求，必要时复制到 `Vec`。

### feat(kv-router): decay overlap credit under prefill load (#10574)
**SHA**: `db55b7b` | 🔗 [查看提交](https://github.com/ai-dynamo/dynamo/commit/db55b7b8a73c2c291e01ff5ffa73bd8a54871b2d)

**🎯 变更类型**：功能增强  

**⚡ 重要程度**：🔴 高  

**📋 变更摘要**  
- 为 KV Router 引入 **`overlap_score_credit_decay`** 参数，使设备本地前缀重叠信用在工作负载出现「prefill 过载」时能够按比例衰减。  
- 新增对应的 CLI/Env/配置字段、Python 绑定、日志输出、文档说明以及单元测试。  
- 调度器在计算 `effective_overlap_score_credit` 时加入归一化的「超额活跃 prefill」因子，从而在保持缓存复用的同时实现更平衡的 prefill 分配。

**🎯 影响范围**  
- `components/src/dynamo/common/configuration/groups/kv_router_args.py`（CLI 参数）  
- `components/src/dynamo/router/__main__.py`（日志）  
- `docs/components/router/*`（概念、配置说明）  
- Python 绑定 `lib/bindings/python/rust/llm/entrypoint.rs`（构造函数、getter/setter、override）  
- 核心调度库 `lib/kv-router/src/scheduling/*`（配置结构、默认值、解析、`DefaultWorkerSelector` 逻辑）  
- 相关单元测试 `components/src/dynamo/common/tests/...`、`lib/kv-router/src/scheduling/tests/...`

---

### 🔍 技术洞察  

| 维度 | 影响评估 |
|------|----------|
| **架构影响** | - **配置层**：在 `KvRouterConfig` 中新增字段 `overlap_score_credit_decay`，并通过环境变量 `DYN_ROUTER_KV_OVERLAP_SCORE_CREDIT_DECAY`、CLI `--router-kv-overlap-score-credit-decay` 暴露。<br>- **调度层**：`DefaultWorkerSelector::worker_logit` 现在接受最小活跃 prefill token 数 (`min_active_prefill_tokens`) 以及衰减因子，用 `1/(1 + decay * normalized_excess)` 调整设备本地信用。<br>- **兼容性**：默认值 `0.0` 保持原有行为，向后兼容无修改的部署。<br>- **绑定层**：Python `KvRouterConfig` 新增属性、setter/getter 与 `with_overrides` 支持。 |
| **性能影响** | - **计算开销**：在每次调度时额外进行一次除法与乘法（归一化超额负载），对单次请求的 CPU 消耗几微秒，可忽略不计。<br>- **路由质量**：在高并发 prefill 场景下，能够降低「热」节点的过度选取，提升整体 P99 ITL（Inter‑Token Latency），同时对 TTFT（Time‑to‑First‑Token）影响有限（取决于 decay 参数）。<br>- **负载平衡**：对 autoscaling 场景尤为有益，能更快让新加入的低负载 worker 获得流量。 |
| **安全考虑** | - 该参数仅影响调度成本函数，不涉及网络、磁盘或权限控制，属于**业务逻辑**层面的调整。<br>- 需确保 `overlap_score_credit_decay` 的取值在 `[0, ∞)`（已通过 `#[validate(range(min = 0.0))]` 限制），防止负数导致除 0 或信用放大。 |
| **可维护性** | - 新增字段统一放在配置结构体、序列化/反序列化、默认实现、Env 解析等多处，需要保持一致性；当前提交已覆盖这些路径并补充单元测试，降低未来遗漏的风险。<br>- 日志与文档同步更新，提升可观测性和用户自助排查能力。 |

---

### ⚠️ 潜在风险  

1. **误配置导致路由退化**  
   - `decay` 设为过大（例如 `10`）会在轻微的负载差异下就把设备信用降到接近 0，可能导致缓存复用率急剧下降，显著增加 TTFT。  
2. **兼容性回退**  
   - 旧版二进制或脚本若未识别新环境变量，会忽略它而仍使用默认 `0.0`，行为保持不变。但如果某些自定义配置文件手动写入 `overlap_score_credit_decay` 而未同步代码，可能在升级后出现 “unknown field” 解析错误（已通过 `serde(default)` 处理，但需要保持序列化兼容）。  
3. **调度路径的 API 变更**  
   - `worker_logit` 现在多了两个参数 (`min_active_prefill_tokens`, `weights`)；若外部组件（例如自定义 selector）直接调用该私有函数，需同步更新签名。  
4. **统计/监控缺失**  
   - 当前仅在启动日志中打印新字段，运行时缺少监控指标。如果运维需要追踪 decay 实际生效情况，可能需要额外的指标或日志。

- 为离线（offline）和在线（online）重放的多轮会话引入 **并发深度优先（Concurrency）模式**，实现了 **会话级的 `max_in_flight` 上限**，即每个会话在整个生命周期（包括思考时间）占用一个槽位，只有在会话全部结束后才释放槽位。  
- 重构 `WorkloadDriver` 接口：新增 `next_pending_session`、`active_count`、`activate_pending` 等成员，实现会话的按序激活、深度优先调度，并在 `new_concurrency*` 构造函数中显式传入 `max_in_flight`。  
- 更新 `Trace`、入口函数、`AdmissionQueue`、聚合/拆分运行时等多处调用，去掉旧的全局 `set_max_in_flight`，改为在 Driver 创建时即确定会话上限。  
- 文档、README 与大量单元/集成测试同步更新，验证了会话保持槽位、cap 等待、思考时间不被其他会话抢占的行为。

**🎯 影响范围**  
- `lib/mocker/src/loadgen/driver.rs`（核心调度实现）  
- `lib/mocker/src/loadgen/trace.rs`（Trace → Driver 转换）  
- `lib/mocker/src/replay/offline/*`（AdmissionQueue、AggRuntime、DisaggRuntime、入口函数）  
- `lib/mocker/src/replay/online/*`（入口函数、LiveRuntime）  
- 文档 `offline/README.md`  
- 相关单元/集成测试文件  

**🔍 技术洞察**  

- **架构影响**  
  - **调度模型改变**：从原来的 “全局请求上限” 改为 “会话上限 + 深度优先”，驱动层承担会话激活、槽位管理职责，降低了 `AdmissionQueue` 与 `WorkloadDriver` 之间的耦合。  
  - **构造函数签名统一**：所有并发模式的构造函数现在必须提供 `max_in_flight`，通过 `debug_assert!` 确保只有并发模式才传入该值，防止误用。  
  - **统一时间戳来源**：`next_ready_time_ms` 只由 `WorkloadDriver` 决定，`AdmissionQueue` 不再自行做 `cluster_in_flight` 裁剪，简化了 admission 流程。  
  - **深度优先实现**：`activate_pending` 按 `next_pending_session` 顺序把会话放入 `ready_sessions` 堆，保证会话在整个生命周期内保持占位，符合 “session‑cap” 语义。  

- **性能影响**  
  - **更真实的并发限制**：在多轮会话场景下，避免了会话间的“宽度”竞争（多个会话的第一轮同时占满 cap），从而更贴近真实生产环境的资源争用模型。  
  - **堆操作次数下降**：仅把 **会话首轮**（在激活时一次）加入 `BinaryHeap`，后续轮次通过 `pop_ready` & `on_complete` 直接更新同一会话的 `ready_at`，相比原先每个请求都参与堆的插入/删除，减少了 `O(log n)` 开销。  
  - **额外状态维护开销**：`next_pending_session`、`active_count` 以及每次 `activate_pending` 的循环检查略微增加 CPU 使用，但绝大多数情况下 `max_in_flight` 较小，循环次数受限。整体预计 **吞吐提升或持平**，且延迟更可预测。  

- **安全考虑**  
  - 通过 `debug_assert!` 对 `max_in_flight` 与 `DriverMode::Concurrency` 的匹配做了编译期/运行时检查，防止配置错误导致未定义行为。  
  - 没有引入新的外部 I/O、unsafe 代码或并发竞态，仅在单线程/受控的调度器内部修改状态，安全影响有限。  
  - 释放槽位的路径在 `on_complete`、`release_cap_slot` 中均已补齐 `active_count` 与 `activate_pending`，避免出现 “slot 泄漏” 导致无限阻塞的情况。  

**⚠️ 潜在风险**  
1. **构造函数兼容性**：所有 `new_concurrency*` 调用必须升级为接受 `max_in_flight` 参数，若有外部项目仍使用旧签名将编译失败。  
2. **深度优先误用**：业务侧可能期望原来的 “宽度优先” 行为（所有第一轮并行），切换为深度优先后会导致单会话占用整个 cap，可能影响实验结果。需在文档或 API 注释中明确说明。  
3. **Slot 计数错误**：`active_count` 的增减必须保持一致；若在未来的改动中忘记在某条错误路径（如异常提前返回）更新计数，可能导致“假死”——`pop_ready` 永远返回空。  
4. **测试覆盖不足的 edge case**：当前测试覆盖了常规和取消场景，但未覆盖 `max_in_flight = 0`（已在 `new` 中 `bail!`）或极大 `usize::MAX` 的内存/时间行为。  

**💡 关注建议**  
- **升级指南**：在发布说明中提供迁移步骤，提示用户在调用 `into_concurrency_driver*`、`WorkloadDriver::new_concurrency*` 时添加 `max_in_flight` 参数。  
- **API 文档**：明确标注 **Concurrency 模式为 “会话级深度优先”**，并说明 `max_in_flight` 作用范围（会话而非单请求）。  
- **监控/日志**：在 `activate_pending`、`release_cap_slot` 等关键路径加入 `debug!` 级别日志，帮助排查因计数不匹配导致的卡死。  
- **回归测试**：补全对 `max_in_flight = usize::MAX`（完全不限制）以及 `max_in_flight = 1` 的极端并发场景的压力测试，确保 `active_count` 不会出现负数或溢出。  
- **保持向后兼容**：如果项目仍需要 “宽度优先” 行为，可考虑保留一个旧的 `max_in_flight_per_turn` 选项或提供包装函数，以免破坏已有实验流程。  

整体来看，此次改动显著提升了 **离线/在线多轮会话重放的资源调度模型**，在保持安全性的同时，为用户提供了更细粒度的并发控制，是一次高价值的功能增强与架构优化。
