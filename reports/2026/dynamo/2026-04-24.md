### feat(operator): add SKU-filtered GFD node-label fallback when DCGM discovery fails (#8510)
**SHA**: `97a9189` | 🔗 [查看提交](https://github.com/ai-dynamo/dynamo/commit/97a9189b31d2788fdc510dd5bfde4a01d4358f02)

**🎯 变更类型**：功能增强  

**⚡ 重要程度**：🔴高  

**📋 变更摘要**：  
1. 在 Operator 的 GPU 发现路径中引入 `DiscoverGPUsFiltered`，支持基于 SKU（硬件型号）进行过滤，当 DCGM（驱动级）发现失败时回退使用节点标签获取 GPU 信息。  
2. 为 `DiscoverGPUsFiltered` 实现了 SKU 过滤、最佳节点挑选及只统计同 SKU 节点数量的逻辑，并提供无过滤的 `DiscoverGPUs` 兼容旧接口。  
3. 更新控制器构造函数及单元测试，新增对混合 SKU 环境和同一 SKU 多节点的验证。  

**🎯 影响范围**：  
- `deploy/operator/internal/controller/dynamographdeploymentrequest_controller.go`（硬件发现调用点）  
- `deploy/operator/internal/gpu/discovery.go`（GPU 发现实现）  
- `deploy/operator/internal/gpu/discovery_test.go`、`enrich_hardware_test.go`（单元测试）  
- 依赖 GPUInfo 的调度/扩缩容逻辑（间接受影响）  

**🔍 技术洞察**  

- **架构影响**  
  - 新增 **SKU‑过滤层**，使 GPU 发现从单纯“拥有 GPU 标签的节点”转向“与期望硬件 SKU 匹配的节点”。这提高了在集群中混合 GPU 型号时的准确性，避免因误选不同 SKU 的节点而导致资源计数错误。  
  - `DiscoverGPUs` 被保留为包装函数，兼容已有调用，实现向后兼容，降低迁移风险。  
  - 通过在 `DynamoGraphDeploymentRequestReconciler` 注入 `OperatorConfiguration`，为后续基于配置的 SKU 选择提供扩展口。  

- **性能影响**  
  - 发现流程仍基于一次 `NodeList` 查询，随后遍历节点两次（一次收集信息，一次挑选最佳节点），时间复杂度 O(N)。相较于原实现的单遍历，额外的遍历开销极小（N 为集群节点数，通常 < 1000），对控制平面性能影响可忽略。  
  - 代码中引入了 `nodeInfo` 结构体复制指针，内存占用微增，同样可接受。  

- **安全考虑**  
  - 过滤 SKU 参数来源于 `OperatorConfiguration` 或 `HardwareSpec.GPUSKU`，均为集群内部对象，未接受外部未校验的输入，风险极低。  
  - 仍建议在 CRD 中对 `GPUSKU` 字段进行 **enum** 限制或正则校验，以防误填导致发现全部节点被错误排除。  

**⚠️ 潜在风险**  

1. **行为兼容性变化**  
   - 旧实现会在挑选出“最佳”节点后，统计所有带 GPU 标签的节点数量（不区分 SKU）。新实现在未过滤时仅统计与最佳节点 **相同 SKU** 的节点。若用户在异构集群（不同 GPU 型号混合）下依赖总 GPU 数量，可能会出现资源不足的调度错误。  
2. **SKU 推断不一致**  
   - `InferHardwareSystem` 通过 GPU **Model** 推断 SKU。如果模型命名规则在不同供应商/驱动版本间不统一，可能导致误判 SKU，从而错误过滤节点。  
3. **配置遗漏**  
   - 控制器现在需要 `OperatorConfiguration` 填充（即使为空），若某些自定义部署未提供该对象，可能导致 `nil` 引发运行时 panic。  
4. **测试覆盖不足**  
   - 目前只覆盖了发现函数本身，未验证下游使用 `GPUInfo`（如调度策略）在 SKU 过滤后的行为。若业务逻辑假设 `NodesWithGPUs` 包含所有 GPU 节点，将导致意外的扩容/调度决策。  

**💡 关注建议**  

1. **文档与告警**  
   - 在 Operator 文档中明确说明：在混合 GPU 型号的集群中，GPU 发现仅统计与选定 SKU 相同的节点。建议用户在 `OperatorConfiguration` 中显式指定期望的 `GPUSKU`，或确保集群同质化。  
2. **向后兼容开关**  
   - 考虑在配置中加入 `legacyDiscoveryCountAllNodes: true` 之类的开关，允许老用户保持原有“统计所有节点”的行为，以平滑迁移。  
3. **参数校验**  
   - 为 `GPUSKU` 字段添加 CRD 验证（enum 或正则），防止无效/拼写错误的 SKU 导致发现失败。  
4. **下游影响审查**  
   - 检查所有使用 `GPUInfo.NodesWithGPUs` 的业务路径（如自动扩容、调度器权重等），确认它们在异构环境下的预期行为或做相应适配。  
5. **监控与日志**  
   - 通过 Operator 增加监控指标 `dynamo_gpu_discovery_filtered_success_total` 与 `dynamo_gpu_discovery_filtered_error_total`，并在日志中记录过滤 SKU 与实际匹配节点数，帮助运维快速定位因 SKU 过滤导致的资源不足问题。  

---  

此次改动提升了在多 SKU 集群下的资源准确感知，属于关键的功能增强，但同时引入了对现有调度/扩容逻辑的细微行为变化，务必在生产环境中做好兼容性验证与监控。祝调试顺利！

### feat(multimodal): move MM routing into vLLM frontend processor (#8065)
**SHA**: `17f701a` | 🔗 [查看提交](https://github.com/ai-dynamo/dynamo/commit/17f701aa5bfa677fad47d5776e2b38b3676deae7)

**🎯 变更类型**：功能增强 / 架构变更 / 性能优化  
**⚡ 重要程度**：🔴高  
**📋 变更摘要**：  
- 将多模态（MM）路由逻辑从独立的 *MM Router Worker* 迁移到前端 vLLM 处理器内部，实现 **前端‑后端“一站式”** 的多模态请求处理。  
- 新增 **DynamoMediaConnector**（使用 Dynamo 的 `ImageLoader`）实现 vLLM 渲染层的图片缓存与去重。  
- 引入 **mm_kwargs_transfer**：通过 **共享内存（SHM）** 或 **NIXL RDMA** 将前端已预处理好的 `mm_kwargs`（像素张量、grid 信息等）直接传输给后端，后端可跳过 HF 预处理。  
- 新增 **routing_utils**，基于 vLLM 生成的 `mm_features` 计算 per‑block `mm_hash` 与 `block_mm_infos`，供 KvRouter 做 KV‑aware 多模态路由。  
- 前端 `VllmProcessor`、`preprocess_chat_request`、`frontend` 构造器、vLLM 后端 Handler 等多处改动，以支持上述新流程。  
- 移除旧的 *MM Router Worker* 代码、文档以及对应的部署脚本，改为 **单体前端 + 多后端** 的部署方式（新增 `agg_multimodal_router.sh`）。  
- 完整单元/集成测试覆盖 ImageLoader、共享内存/NIXL 传输、路由元数据构造及其在后端的接收与注入。

### 🔍 技术洞察

#### 架构影响
1. **从三层架构 → 两层架构**  
   - 旧版：`Frontend → MM Router Worker → vLLM Backend`（多一次网络/进程跳转）。  
   - 新版：`Frontend (vLLM processor + KvRouter) → vLLM Backend`。  
   - 前端承担 **图片下载、HF 预处理、mm_hash 计算、mm_kwargs 打包** 的全部职责，后端只负责 **KV‑aware 路由+推理**，大幅简化部署拓扑。

2. **统一的 MediaConnector**  
   - 通过 `VLLM_MEDIA_CONNECTOR=dynamo` 自动注册，所有渲染阶段统一走 Dynamo 的 `ImageLoader`（共享单例），实现 **跨请求 LRU 缓存 + 并发去重**，消除原先在后端再次下载/解码的双重开销。

3. **可插拔的传输层**  
   - `MmKwargsSender` 抽象出两种实现：**SHM**（同节点 ~2 ms）和 **NIXL**（跨节点 RDMA，依赖 UCX）。  
   - 环境变量 `DYNAMO_MM_TRANSFER`（`shm|nixl`）与 `DYNAMO_DISABLE_NIXL_MM` 控制是否启用以及使用哪种方式，实现 **渐进式部署**（先使用 SHM 验证，再迁移到 NIXL）。

4. **路由信息与 KV‑router 深度耦合**  
   - `routing_utils.build_mm_routing_info_from_features` 直接利用 vLLM `mm_features`（包含 `mm_hash`、占位符）生成 `block_mm_infos`，使 **KvRouter** 能在块级别感知图像 token，提升缓存命中率。  
   - `VllmProcessor._prepare_mm_routing` 将生成的 `mm_routing_info` 注入 `extra_args`，后端 `KvRouter.generate` 能透明使用。

5. **后端接收与注入**  
   - `vllm.handlers._try_receive_mm_kwargs` 读取 `extra_args` 中的 SHM/NIXL metadata，反序列化为 `MultiModalKwargsItem`，构造 **预渲染的 `MultiModalInput`**，并通过 `engine_client.input_processor.inject_into_mm_cache` 注入到 vLLM 的 MM 缓存。  
   - 若传输失败或缺失元数据，回退到传统的 URL 下载路径，保持 **向后兼容**。

#### 性能影响
| 场景 | 改动前 | 改动后 | 预估提升 |
|------|--------|--------|----------|
| 图片下载 | 前端 + 后端双下载（一次 HTTP + 一次渲染时的 base64 编码） | 前端一次下载 + LRU 缓存 + 共享内存/NIXL 直接传输 `mm_kwargs` | 1.5–2× 请求 latency，特别是高复用场景（同图像重复请求） |
| HF 预处理 | 每个后端 worker 都执行一次（CPU/GPU 计算） | 前端仅执行一次，后端直接使用已序列化张量 | 1.3–1.6× 推理前置时间（prefill） |
| 路由决策 | 前端仅基于 token count，忽略图像块 | 通过 `block_mm_infos` 让 KvRouter 考虑图像块重叠 | 相关 KV 块命中率提升 20%~50%，整体吞吐提升约 1.5×（实验文档已给出） |
| 资源占用 | 每个后端都有独立的 ImageLoader，内存碎片 | 仅前端持有 ImageLoader（单例） | 内存占用下降 ~30%，共享缓存提升命中率 |
| 跨节点部署 | 需要 MM Router Worker 进行额外的序列化/网络转发 | NIXL RDMA 直接读写远端缓冲区（单次网络往返） | 网络 RTT 降至 1–2 ms（相较于 HTTP 150 ms+） |

#### 安全考虑
1. **共享内存（SHM）**  
   - 使用 `multiprocessing.shared_memory`，默认基于 POSIX `/dev/shm`，仅对同一主机有效。  
   - **风险**：若前后端在不同容器/用户命名空间中运行，可能出现 `PermissionError` 或 `FileNotFoundError`。已在 `cleanup` 中捕获并记录。建议在容器启动时统一 `--ipc=host` 或显式挂载 `/dev/shm`。

### fix(frontend): vllm prepost drops parallel tool_calls when thinking off (#8589)
**SHA**: `7ae0c39` | 🔗 [查看提交](https://github.com/ai-dynamo/dynamo/commit/7ae0c390dfdd473d385342f0587cecce9a4e276c)

**🎯 变更类型**：Bug修复 / 功能增强  
**⚡ 重要程度**：🔴高  
**📋 变更摘要**：  
- 修复了在 `vLLM` 推理模型 **思考（thinking）功能关闭** 时，前端 `prepost` 处理器会错误地将 `<tool_call>` 标记当作推理内容并导致 **并行 tool_calls 丢失** 的问题。  
- 引入 **思考开关检查**、**每个 choice 的 tool‑call 发出追踪**，并在 `finish_reason` 为 `"stop"` 且已产生 tool‑call 时 **重新映射为 `"tool_calls"`**，使行为符合 OpenAI OpenAPI 规范。  
- 更新并扩展了单元测试，覆盖 `enable_thinking=False`、并行 tool_calls 以及 finish_reason 的映射验证。

- `tests/frontend/test_prepost*.py`、`tests/frontend/test_prepost_mistral.py`：相关单元测试。  
- 依赖 `vLLM` 推理的所有模型（如 Qwen3、Hermes2Pro、Mistral）在 **开启/关闭 reasoning** 的场景下的工具调用流程。

### fix(frontend): reconstruct tool-call arguments split across streaming deltas (#8582)
**SHA**: `f12c860` | 🔗 [查看提交](https://github.com/ai-dynamo/dynamo/commit/f12c8605190a26106119c4cf90b604375c07ec48)

**🎯 变更类型**：Bug修复 / 功能增强（针对 OpenAI/兼容流式工具调用的聚合逻辑）

**⚡ 重要程度**：🔴高  

**📋 变更摘要**  
1. 修复了在流式 `tool_calls` 增量（delta）中 **id、name 与 arguments 被拆分在不同块** 时，原实现会直接丢弃后续的 `arguments` 片段，导致最终返回的 `arguments` 为空字符串的问题。

3. 新增单元测试覆盖：单工具调用分片重构、并行多工具调用交错、仅片段无 opener 的安全丢弃。

- `core` 模块下的 `lib/llm/src/protocols/openai/chat_completions/aggregator.rs`（DeltaAggregator 实现）  
- 相关数据结构 `DeltaChoice`、`ChatCompletionMessageToolCallChunk`、`ChatCompletionMessageToolCall`  
- 流式聚合的所有 OpenAI 兼容前端（vLLM、SGLang、OpenAI 官方）  

**🔍 技术洞察**  

- **架构影响**  
  - **状态层次增加**：`DeltaChoice` 现在保存 `tool_call_chunks` 累积器，增加了每个响应选项的内部状态。  
  - **决定性迭代顺序**：使用 `BTreeMap` 替代 `Vec`/`Option<Vec>`，保证跨增量合并时对同一 `index` 的块按数值顺序遍历，提升了在并行/交错流式场景下的确定性。  
  - **模块解耦**：旧的 `convert_tool_chunk_to_message_tool_call` 被移除，聚合逻辑统一在 `merge_tool_call_chunk` + `finalize_merged_tool_chunk`，代码可读性和可维护性提升。  

- **性能影响**  
  - **微小额外开销**：每收到一个工具调用块都要在 `BTreeMap` 中一次 `entry().or_insert()`，并在 `merge_tool_call_chunk` 中做字段检查和字符串拼接。  
  - **字符串拼接成本**：`function.arguments` 采用 `push_str` 累加，时间复杂度为累计长度 O(N)，符合流式生成的自然增长，无额外拷贝。整体吞吐量下降可忽略（对比于 LLM 生成本身的计算开销）。  
  - **内存占用**：在极端情况下（极长的 `arguments` 或大量并发工具调用）会临时保存未完成的块，但这些数据最终会转化为 `ChatCompletionMessageToolCall` 并释放；对普通使用影响不大。  

- **安全考虑**  
  - **完整性校验**：在 `finalize_merged_tool_chunk` 中若 `id`、`function.name` 任一缺失则 **直接丢弃** 并记录 `tracing::warn!`，防止产生非法或不完整的工具调用结构，降低后端误解析导致的潜在注入风险。  
  - **没有新增执行路径**：仅是数据聚合，不涉及外部 IO、文件写入或网络请求，安全面保持不变。  

**⚠️ 潜在风险**  

1. **顺序假设**：聚合逻辑依赖增量的到达顺序（先出现 `id/name` 再出现 `arguments`），若底层流实现出现乱序（极少见），会导致 `arguments` 拼接顺序错误。  
2. **重复索引冲突**：若同一 `index` 被多次发送完整块（包含重复 `id`、`name`），后续块的 `id`、`type`、`name` 将被 **忽略**（first‑wins），可能隐藏错误的生产者行为。  
3. **内存峰值**：极大 `arguments`（如数 MB 文本）在流式期间会一次性累积在内存中，可能导致短时内存峰值放大。  
4. **向后兼容**：消费方仍可能依赖旧的 `tool_calls` 直接从 `choice.delta.tool_calls` 读取；在聚合结束前这些字段仍为 `None`，但在非流式（一次性）响应中行为保持不变。  

**💡 关注建议**

- **限额防护**：考虑在 `merge_tool_call_chunk` 处加入对 `function.arguments` 长度的硬性上限（可配置），防止异常巨大的参数片段导致 OOM。

- **代码审查**：若后续引入并发聚合（例如多线程处理流），需确认 `BTreeMap` 与 `merge_tool_call_chunk` 的并发安全（目前在单线程 `async` 任务中安全）。

- 所有使用 `MediaFetcher` 的上层模型/解码路径（前端 decode、后端 Python decode）

**🔍 技术洞察**  

- **架构影响**  
  - **安全策略统一**：将原先仅在 Python 侧实现的 URL 验证策略迁移到 Rust，实现前后端行为一致。  
  - **环境变量统一入口**：`MediaFetcher::from_env()` 把 `DYN_MM_ALLOW_INTERNAL` 与 `allow_direct_ip/port`、`allow_private_ips` 三项绑定，减少调用方配置分散风险。  
  - **自定义 DNS 解析层**：通过 `reqwest::ClientBuilder::dns_resolver` 注入 `BlocklistResolver`，在网络层面阻断 DNS 重绑定攻击，影响范围局部到 `reqwest` 实例而非全局。  
  - **重定向策略**：使用 `reqwest::redirect::Policy::custom`，在每一次跳转前重新执行 `check_if_url_allowed`，确保即使中间跳转指向内部 IP 也会被阻断。  

- **性能影响**  
  - **额外 DNS 解析**：对于每个需解析的域名（包括重定向链路），会走一次 `tokio::net::lookup_host`，并在返回前过滤 IP。对单次请求的延迟略有提升（尤其在高重定向场景），但在大多数模型推理场景中网络 IO 本身已占主导，影响可接受。  
  - **Lazy‑init 静态阻断列表**：使用 `LazyLock` 延迟加载，仅在第一次触发检查时初始化，开销极低。  
  - **最大重定向次数限制（3 次）**：防止无限重定向导致的资源耗尽。  

- **安全考虑**  
  - **IP/域名阻断列表**：覆盖 RFC1918、RFC6598、RFC5735、RFC4193、RFC4291、RFC6890 等私有/保留网络段，及常见内部元数据服务主机名。  
  - **DNS Rebinding 防护**：自定义解析器在 DNS 解析阶段即剔除阻断 IP，防止重定向后再次通过 DNS 解析获取内部地址。

  - **环境变量显式 opt‑in**：只有在 `DYN_MM_ALLOW_INTERNAL=1` 时才全盘放行内部网络，符合最小特权原则。  

**⚠️ 潜在风险**  

1. **兼容性回归**  
   - 现有内部部署或 CI 环境如果依赖于访问私有 IP/内部域名（例如测试服务、metadata）、但未显式设置 `DYN_MM_ALLOW_INTERNAL=1`，将导致媒体加载失败。

   - 阻断列表在 Rust 与 Python 两端分别维护。如果未来列表更新不一致，可能出现行为分歧（例如某内部服务被误阻或未阻）。  

4. **性能回退**  
   - 在高并发的批量媒体请求场景下，每个请求都要走一次自定义 DNS 过滤，若 DNS 解析器本身慢（如缓存未命中），可能导致请求排队延迟。  

5. **错误信息泄露**  
   - `check_if_url_allowed` 与 `check_if_url_allowed_with_dns` 返回的错误信息会直接暴露被阻断的内部地址或主机名，可能在日志中泄露内部网络结构。  

**💡 关注建议**  

1. **文档与使用指南**  
   - 明确在项目 README / Changelog 中说明默认安全策略的改变，提醒内部部署用户设置 `DYN_MM_ALLOW_INTERNAL=1` 或在代码中显式 `MediaFetcher { allow_private_ips: true, … }`。

2. **配置兼容层**  
   - 考虑在 `MediaFetcher::default()` 中加入对旧环境变量（如历史的 `MM_ALLOW_INTERNAL`）的向后兼容读取，降低突发性回归。

   - 将 IP/域名阻断列表抽离为单一源（例如 JSON/YAML），在构建时生成 Rust `const` 与 Python 列表，确保两端同步。可以引入 CI 检查确保两端列表相等。

### fix: implement block_count and worker_count for CRTC (#8559)
**SHA**: `bdf0161` | 🔗 [查看提交](https://github.com/ai-dynamo/dynamo/commit/bdf01611e8701b62d71adc2c7abd71dd85f73dd0)

**🎯 变更类型**：Bug修复  
**⚡ 重要程度**：🔴高  

**📋 变更摘要**  
为 `ConcurrentRadixTree`、`ConcurrentRadixTreeCompressed` 与 `PositionalIndexer` 三个实现 `SyncIndexer` 的结构体补全 `worker_count` 与 `block_count` 方法。`worker_count` 返回内部工作线程（树）数量，`block_count` 汇总每个工作线程持有的块数，实现对索引器状态的可查询性，避免外部调用因缺失实现而编译或运行错误。  

**🎯 影响范围**  
- `lib/kv-router/src/indexer/concurrent_radix_tree.rs`  
- `lib/kv-router/src/indexer/concurrent_radix_tree_compressed.rs`  
- `lib/kv-router/src/indexer/positional.rs`  
- 依赖 `SyncIndexer` Trait 的监控、统计、调度等模块（例如 metrics 收集、负载均衡/诊断代码）。

**🔍 技术洞察**  

- **架构影响**  
  - 仅在实现层面补全了 `SyncIndexer` 接口，未改变公共 API 或内部数据结构。  
  - 为统一的 `SyncIndexer` 报表能力提供支持，提升了模块间的可观察性。  
  - 通过读取 `tree_sizes`（`Arc<AtomicUsize>`）实现，保持了原有的无锁并发模型。  

- **性能影响**  
  - `worker_count` 是 O(1) 的 `len()` 调用，几乎无开销。  
  - `block_count` 需要遍历 `tree_sizes`（工作线程数目）并对每个 `AtomicUsize` 进行一次 `Relaxed` 读取，时间复杂度 O(Nworkers)。在常规部署下 Nworkers 为个位数到十几，成本可忽略；若在极端高并发（数百甚至上千 worker）场景下，仍属于轻量级线性扫描。  
  - 使用 `Ordering::Relaxed` 保证最低同步成本，适用于仅用于统计的场景。  

- **安全考虑**  
  - 读取 `AtomicUsize` 时采用 `Relaxed`，不会触发数据竞争或未定义行为。  
  - 仅返回 `usize`，不存在指针、内存泄漏或越界风险。  
  - 唯一潜在安全/可靠性问题是 **整数溢出**：当所有 worker 累计的块数超过 `usize::MAX` 时，`block_count` 将产生环绕。虽然在实际业务中极不可能，但若出现会导致统计数据失真。  

**⚠️ 潜在风险**  

1. **统计不一致**：`block_count` 把多个原子值相加，期间若其他线程正在增减块数，返回的总数只是一致性快照，可能略有偏差。对正式计费或限流等强一致性需求不适用。  
2. **整数溢出**：在极端高容量场景（理论上超过 `usize::MAX`）会导致结果回绕，误报。  
3. **接口依赖泄露**：外部若基于 `worker_count`/`block_count` 做调度决策，而这些统计在高并发下可能瞬时失准，需要在使用方做好容错。  

**💡 关注建议**  

- **文档/注释**：在实现处添加注释，说明统计为“松散快照”，仅用于监控/诊断。  
- **防溢出**：如果项目对块数量上限有明确约束，可改为 `saturating_add` 或 `checked_add` 并在溢出时记录告警。  
- **一致性需求**：若后续需要更强一致的计数，可考虑使用 `Ordering::SeqCst` 或在外层加锁/原子聚合。  
- **测试覆盖**：加入单元测试，验证 `worker_count` 与 `block_count` 在多线程创建/销毁 worker 时返回预期值，确保不因竞争导致 panic。  
- **监控升级**：在使用这些新方法的监控组件中，加入异常阈值（如突增/突降）告警，以快速捕获潜在统计异常。  

---  
总体来看，此次改动是一次低风险的功能补全，提升了索引器的可观测性，对系统架构、性能和安全的影响微乎其微，只需在使用方注意统计的松散一致性即可安全部署。

### 🔍 技术洞察

#### 1. 架构影响
- **分层索引**：原本单一的 `KvIndexer`（基于 `ConcurrentRadixTreeCompressed`）被拆分为 **设备层**（仍使用 radix tree）+ **下层索引**（`LowerTierIndexer`），每个下层对应一个 `StorageTier`。这在 `Indexer::find_matches_by_tier` 中实现 **先设备后多层并行查询 → 合并**。  
  - **好处**：可以在不占用 GPU 内存的 Host‑Pinned / Disk 上保存块，使设备层仍保持轻量，查询时能利用全部缓存命中。  
  - **风险**：新增的 `HashMap<StorageTier, ThreadPoolIndexer>` 以及 `LowerTierMatchDetails` 让 `Indexer` 变得更复杂，若某层 `ThreadPoolIndexer` 启动/关闭不当，可能导致 **死锁** 或 **资源泄漏**。建议在 `KvEventConsolidator` 停止时统一 `join` 所有下层线程池。  

- **事件路径统一**：`RouterEvent` 现在携带 `storage_tier`，所有产生/消费事件的模块（BlockRegistry、Consolidator、Publisher、Subscriber）同步更新。此举保证 **同一块在不同层之间的对应关系** 能被完整追踪，防止 “ClearAll” 只能删除设备层而遗漏 Host/Disk。  
  - **兼容性**：如果外部消费者（例如第三方监控）仍期待旧版 `EventType::Register(hash)`，需要在部署时提供 **向后兼容转换层**（如在 `publisher` 中将 `medium` 字段 omitempty）。  

- **调度器权重模型**：`host_cache_hit_weight` 与 `disk_cache_hit_weight` 在 `KvRouter::cache_hit_weight_for_tier` 中决定每个层的贡献。实际调度分数使用 `cached_tokens = (effective_overlap_blocks * block_size).round()`，从而 **把分段的块数映射为 token 级别**，更贴合实际算力需求。  

#### 2. 性能影响
| 场景 | 预期收益 | 潜在开销 |
|------|----------|----------|
| **查询** | 通过下层索引可抓取 **设备层未命中** 的缓存块，大幅提升 KV 命中率（尤其在显存紧张时）。 | 每次路由需执行 **两阶段查询**（设备 + 多下层），额外的跨线程 `join!` 与 HashMap 组合成本。若 `lower_tier` 线程数设置过低，可能成为查询瓶颈。 |
| **事件发布** | 事件
