### fix(vllm): include image geometry in multimodal hash preimage (#8341)
**SHA**: `5103efd` | 🔗 [查看提交](https://github.com/ai-dynamo/dynamo/commit/5103efdb64c778feee206defcf9930235c157dd6)

**🎯 变更类型**：Bug修复 / 功能增强  
**⚡ 重要程度**：🔴 高  
**📋 变更摘要**：  
- 将多模态图像哈希的前置数据从单纯的像素字节改为*固定长度头部 + 像素字节*，并由 **blake3** 生成 UUID，确保图像的宽高信息被纳入哈希，避免不同尺寸但像素总数相同的图像产生冲突。  
- 在 `vLLM` 处理路径中，新增对 EPD（Embedding‑Pre‑computed‑Data）模式的分支：当使用预计算嵌入时不再生成图像 UUID，保持与 Router/Encoder 缓存的一致性。  

**🎯 影响范围**：  
- `components/src/dynamo/vllm/handlers.py`（请求构建流程）  
- `components/src/dynamo/vllm/multimodal_utils/hash_utils.py`（哈希实现）  
- `components/src/dynamo/vllm/tests/multimodal_utils/test_vllm_hash_utils.py`（单元测试）  
- 关联的 Rust KV Publisher（使用相同哈希协议）  

**🔍 技术洞察**：

| 维度 | 影响 |
|------|------|
| **架构影响** | - 引入了统一的 *preimage* 协议（12 字节固定头 + 像素），为前后端（Python vLLM ↔ Rust KV）提供了严格的兼容层。<br>- 使用 `blake3` 替代 `sha256`，统一了跨语言的哈希实现，必须保证 Rust 端同样采用相同的预像素布局与版本号。 |
| **性能影响** | - `blake3` 在 CPU 上比 `sha256` 更快且支持增量更新，额外的 12 字节头部对整体吞吐几乎没有影响。<br>- 通过 `np.ascontiguousarray` 确保 NumPy 数据连续，避免额外复制。 |
| **安全考虑** | - `blake3` 为现代密码学哈希函数，抗碰撞性优于 `sha256`，提升了缓存键的唯一性安全性。<br>- 仍属非加密用途，未引入新的安全风险。 |
| **可维护性** | - 新增严格的输入校验（模式、dtype、shape），在错误时直接抛异常，提升了异常可观测性，防止“静默碰撞”。<br>- 通过 `_HEADER_STRUCT` 与常量抽象，后续若需扩展（比如增加 Alpha 通道）只需修改常量并 bump 版本字节。 |
| **兼容性** | - 现有缓存键（基于旧 SHA256 + 像素裸流）将失效，导致缓存 miss。需要在发布说明中提醒用户清理或迁移旧缓存。<br>- Rust KV 端若未同步到相同预像素格式，会产生解码错误。 |

**⚠️ 潜在风险**：

1. **缓存失效**：哈希算法和前置数据变更后，历史 KV 条目不再匹配，可能导致瞬时性能下降（大量冷启动）。  
2. **跨语言不一致**：如果 Rust 端仍使用旧 preimage（仅像素）或仍使用 SHA256，会出现键不匹配或运行时异常。  
3. **Embedding‑Loader 分支**：在 EPD 模式下 `mm_uuids` 被设为 `None`，若下游假设始终有 UUID（比如日志、审计）可能触发空指针错误。  
4. **输入验证**：严格的 `RGB` + `uint8` 限制可能导致部分用户上传的 `RGBA` 或 `L` 模式图片直接报错，需在上游（Router）做好转换或给出友好错误信息。  

**💡 关注建议**：

- **同步协议**：确保 Rust KV Publisher 代码同步更新，使用相同的 `_HEADER_STRUCT` 与版本字节；在 CI 中加入跨语言哈希对齐的集成测试。  
- **迁移指南**：在发布说明中提供“清理旧缓存”或“使用新版本 preimage” 的步骤；若业务对缓存命中率高度敏感，可考虑在升级窗口内预热。  
- **监控与报警**：新增监控指标 `vllm.hash_collisions` 与 `vllm.hash_errors`，在升级后观察是否出现异常冲突或输入校验错误。  
- **文档更新**：在 `vLLM` 文档中明确说明仅支持 `RGB` + `uint8` 输入，并解释为何不接受 `RGBA`／`float` 等。提供示例转换代码（PIL `convert("RGB")`、`astype(np.uint8)`）。  
- **测试覆盖**：保持当前单元测试，并在 CI 中加入针对 Rust 端的对应哈希基准测试，确保“已知 digest” 随版本号变化而被强制更新。  

总体来看，此次修改显著提升了多模态缓存键的唯一性和安全性，架构上实现了更严谨的跨语言哈希协定，风险主要集中在兼容性和缓存失效的恢复上。只要配合相应的迁移与监控措施，整体风险可控且收益明显。

### fix(snapshot): replace PID-1 SIGUSR1/SIGCONT contract with file sentinels (#8403)
**SHA**: `ca15a79` | 🔗 [查看提交](https://github.com/ai-dynamo/dynamo/commit/ca15a79da7a055668c91a347f511635d13923cb1)

**🎯 变更类型**：功能增强 / 重构 / 安全修复  

**⚡ 重要程度**：🔴 高  

**📋 变更摘要**  
本次提交彻底改写了 Dynamo 检查点/恢复的控制协议：原本依赖容器 PID 1 收到 `SIGUSR1`（检查点完成）和 `SIGCONT`（恢复完成）的信号，现在改为通过挂载在工作负载容器内部的 **snapshot‑control** emptyDir 卷写入特定的 sentinel 文件（`snapshot-complete`、`restore-complete`、`ready-for-checkpoint`）进行同步。为此新增了控制卷、环境变量 `DYN_SNAPSHOT_CONTROL_DIR`、轮询实现以及原有的信号处理、环境变量配置均被移除。相应地，Operator、Helm Chart、Snapshot 协议、运行时实现以及大量单元测试均同步更新。

**🎯 影响范围**  
- **核心 Python 工具**：`components/src/dynamo/common/utils/snapshot.py`（信号改为文件轮询）  
- **Operator API 与 CRD**：`CheckpointConfiguration` 中 `ReadyForCheckpointFilePath` 被删除，相关默认值、验证逻辑全部移除。  
- **Operator 控制器**：`checkpoint_job.go`、`dynamocheckpoint_controller.go`、`podspec.go` 等实现改为使用 sentinel 文件。  
- **Helm Chart & 部署配置**：删除 `readyForCheckpointFilePath` 参数，新增 `DYN_SNAPSHOT_CONTROL_DIR` 环境变量注入。  
- **Snapshot 协议层**：新增 `control_volume.go`、`runtime/control.go`（写 sentinel）以及对应单元测试。  
- **运行时（runtime）**：移除 `SendSignalViaPIDNamespace`，改为 `WriteControlSentinel`。  
- **GMS 组件**：更新卷名与挂载路径以避免与 snapshot‑control 冲突。  
- **大量测试代码**：覆盖新逻辑的正确性与幂等性。

### ⚠️ 潜在风险  

1. **卷挂载失效**  
   - 若某些自定义 PodSpec 手动覆盖 `snapshot-control` 卷或忘记调用 `EnsureControlVolume`，工作负载将永远卡在 “ready-for-checkpoint” 状态，导致 checkpoint job 超时。  
2. **文件系统延迟或 NFS 失效**  
   - 虽然大多数情况下使用 `emptyDir`（本地节点），但在极端节点磁盘 IO 高峰时可能出现写入延迟，导致工作负载的轮询超时。  
3. **误删或覆盖 sentinel**  
   - 业务容器如果自行在 `/snapshot-control` 目录下创建同名文件，可能误触完成状态。需在文档中明确禁止业务进程写入该目录。  
4. **迁移期间的双写冲突**  
   - 若旧版系统仍保留信号处理（未升级），同时新文件 sentinel 被写入，可能导致两套状态并存，产生不一致的 checkpoint/restore 判定。  
5. **权限升级风险**  
   - `WriteControlSentinel` 通过宿主机的 `/proc/<pid>/root` 直接写入，需要容器运行时拥有足够的权限；在受限的 `runAsUser`/`readOnlyRootFilesystem` 环境下可能失败。

### fix(parsers): recover Kimi K2 tool calls when section_end is missing (#8208)
**SHA**: `47cfbd4` | 🔗 [查看提交](https://github.com/ai-dynamo/dynamo/commit/47cfbd46c91f5363a3dec69163601bc0bf44a382)

**🎯 变更类型**：Bug修复  
**⚡ 重要程度**：🔴高  
**📋 变更摘要**：

1. `find_tool_call_end_position` 由返回固定长度改为返回 `Option<usize>`，当检测到 `section_end` 缺失时返回 `None`，让上层流式监狱（jail）决定是否提前退出。  
2. `JailedStream` 在早期退出路径上加入对 `None` 的处理：若未找到完整结束位置，则不提前退出，而是延迟到流结束时统一 `finalize`，从而恢复完整的工具调用。  
3. Kimi K2 解析器 `find_tool_call_end_position_kimi_k2` 同样改为返回 `Option<usize>` 并在缺失 `section_end` 时返回 `None`，并对 `extract_tool_calls` 进行相应适配，使其在没有显式结束标记的情况下仍能提取已完整的单个工具调用。  
4. 新增一系列针对缺失 `section_end`、多工具调用以及截断场景的单元测试，验证修复效果。

**🎯 影响范围**：  
- `lib/llm/src/protocols/openai/chat_completions/jail.rs`（流式监狱层）  
- `lib/parsers/src/tool_calling/parsers.rs`（统一工具调用结束位置函数）  
- `lib/parsers/src/tool_calling/xml/kimi_k2_parser.rs`（Kimi K2 专属解析实现）  
- 对应的单元测试文件 `lib/llm/tests/test_streaming_tool_parsers.rs`  
- 其他依赖 `find_tool_call_end_position` 的工具调用解析路径（如 `nemotron_deci`、`harmony`、`pythonic` 等）受返回类型变化影响，需要重新编译通过。

**🔍 技术洞察**：

- **架构影响**  
  - **流式监狱（JailedStream）** 现在由“能否提前退出”与“是否已有完整结束位置”两层判定组合，提升了对不完整流的容错能力。  
  - 解析层的统一函数 `find_tool_call_end_position` 改为返回 `Option<usize>`，对所有解析器统一了“未闭合”语义，降低了各解析器自行判断的复杂度。  
  - 该改动保持了现有的抽象边界，没有对外 API（如 `parse_response_stream`）的签名产生变更，兼容性良好。

- **性能影响**  
  - 额外的 `Option` 包装和一次 `if let Some` 判定在流式路径上开销极小（几乎可以忽略），对高并发或大模型的吞吐量基本不产生负面影响。  
  - 通过延迟 `finalize` 而非提前切分，能够在缺失 `section_end` 的情况下避免误提前结束导致的重复解析，从整体上可能提升流式响应的准确率，间接改善用户感知的响应速度。

- **安全考虑**  
  - 变更仅涉及字符串解析与控制流，无新增外部输入的安全风险。  
  - 通过严格的正则/模式匹配仍然保持对工具调用结构的校验，未放宽对恶意构造的容忍度。

**⚠️ 潜在风险**：

1. **返回类型变化导致编译错误**：所有调用 `find_tool_call_end_position` 的地方必须适配 `Option<usize>`，如果遗漏会导致编译失败或运行时 panic。当前 PR 已覆盖主要路径，但未来新增解析器或自行实现的插件可能遗漏适配。  
2. **早期退出逻辑回退**：在极端情况下（例如真的没有后续内容），`finalize` 会在流结束时再次尝试解析，若仍未检测到 `section_end`（且仅有不完整的调用），会返回 0 工具调用并将原始标记作为普通文本。若此类情况在业务上需要特殊处理（比如错误提示），可能需要额外的业务层判断。  
3. **内存占用略增**：因为在缺失 `section_end` 时不再提前切分，累计的 buffer 会保持更久，极端情况下（大量未闭合的标记）可能导致短期内内存使用上升。  
4. **对其他模型的兼容性**：虽然大多数模型通过 `section_end` 正常结束，改动可能在极少数自定义模型中产生意外行为（如误判为“未闭合”而延迟退出），需要在模型配置中确保 `section_end` 的可选性被合理设定。

### 🔍 技术洞察  

| 维度 | 影响说明 |
|------|----------|
| **架构影响** | - 通过 **RAII Guard (StageGuard)** 将分阶段计数的生命周期绑定到代码块，保持业务逻辑与监控代码的低耦合。<br>- 新增 `ACTIVE_REQUESTS` gauge 替代原 `INFLIGHT_REQUESTS` 的语义，使监控更加直观。<br>- `STAGE_REQUESTS` 多维度（`stage`、`phase`）计数为 **前端流水线可观测性** 打下基础，可在 Grafana 中绘制细粒度流量图。 |
| **性能影响** | - 每进入/离开一个 stage 会多一次 `IntGauge.inc/dec`，在高 QPS 场景下会产生极低的原子整数操作开销（约 **数十纳秒**），对整体吞吐几乎没有可感知影响。<br>- 新增 `active_requests_gauge` 与 `stage_requests` 两套 gauge，大约每个 gauge 1–2 KiB 内存（标签卡槽），整体内存占用 < 1 MiB，属于可接受范围。 |
| **安全考虑** | - 纯监控数据，未改变业务路径或数据流向，无新增安全风险。<br>- 兼容老版本的 `inflight_requests`，避免因 metric 名称变化导致外部报警误触。 |
| **可维护性** | - `StageGuard` 把 “计数 + 解除” 合并在同一对象，降低忘记 `dec` 的概率，提升代码可读性。<br>- 通过 `ensure_frontend_perf_metrics_registered` 统一注册，防止遗漏。<br>- 新增的单元测试覆盖了 guard 增减、不同标签组合的行为，提升可靠性。 |

---

### ⚠️ 潜在风险  

1. **重复计数**  
   - `inc_inflight_gauge` 同时对 `inflight_gauge` 与 `active_requests_gauge` 做 `inc/dec`，如果后续有人在业务层自行操作 `active_requests_gauge`，会导致计数不一致。需在文档或代码注释中明确 **只能通过 `inc_inflight_gauge` / `dec_inflight_gauge`** 调整。  

2. **Metric 注册遗漏**  
   - 若在某些自定义 `MetricsRegistry` 实例中忘记调用 `ensure_frontend_perf_metrics_registered`，`STAGE_REQUESTS` 可能不会被注册，导致 Prometheus 报错。建议在项目启动的统一入口强制调用一次。  

3. **标签卡槽耗尽**  
   - `STAGE_REQUESTS` 使用 `["stage","phase"]` 两个标签。若业务中出现异常或未预期的 `phase`（比如空字符串或拼写错误），会导致卡槽泄漏，长期运行后出现 **“Too many label values”** 错误。需确保 `phase` 只能取固定枚举值（`"", "prefill", "decode", "aggregated"`）。  

4. **Guard 失效**  
   - `StageGuard` 依赖 RAII 正常析构；在 `panic!` 或 `std::process::abort` 中可能导致漏减。虽然这种情况在生产环境极少见，但可以在 `Metrics` 中加入 **fallback 清理**（如在 `Metrics::flush` 时强制重置 gauge），进一步降低风险。  

5. **向后兼容**  
   - `INFLIGHT_REQUESTS` 仍保留，但已标记为 **Deprecated**。若外部监控系统仍基于旧 metric 做告警，需要同步迁移。否则会出现 **双告警**（一个是旧的，一个是新 `active_requests`）。

|------|------|
