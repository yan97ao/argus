### fix(observability): preserve traceparent sampling flags (#10980)
**SHA**: `4350517` | 🔗 [查看提交](https://github.com/ai-dynamo/dynamo/commit/4350517f03d76057b9eb146263e86279de1c25a0)

**🎯 变更类型**：Bug修复  
**⚡ 重要程度**：🔴高  
**📋 变更摘要**：  
- 修复了分布式追踪在跨进程传递时默认强制将 `traceparent` 的采样标记写为 `01`（已采样），导致原始请求的 `unsampled` 标记被丢失的问题。  
- 在核心运行时（`runtime/src/logging.rs`）中为 `DistributedTraceContext` 新增 `trace_flags` 字段，完整解析、校验、传播以及在生成 `traceparent` 时保留原始标记。  
- Python 绑定层同步更新文档与实现，使返回的 `traceparent` 包含真实的 `flags`。  
- 扩充大量单元测试，验证版本、字段合法性、采样决定的正确传播以及在未采样情况下不导出 OTLP span。

**🎯 影响范围**：  
- `lib/runtime/src/logging.rs`（核心追踪上下文与日志层）  
- `lib/bindings/python/rust/context.rs`、`lib/bindings/python/src/dynamo/_core.pyi`（Python SDK）  
- 相关的测试文件 `tests/frontend/test_unified_worker_otlp_export.py` 以及 `runtime/src/tests` 中的追踪单元测试  
- 使用 W3C Trace Context 进行分布式追踪的所有 Dynamo 组件（frontend、worker、system）以及依赖 OpenTelemetry 的外部监控系统。

**🔍 技术洞察**  

- **架构影响**  
  - **追踪上下文模型扩展**：`DistributedTraceContext` 现在携带 `trace_flags`，并在 `TraceParent`、`PendingDistributedTraceContext`、`TraceParent::from_headers` 等路径中完整传递。  
  - **兼容性**：对外保持 `traceparent` 字符串格式不变，只是第 4 段可能为 `00`（未采样）或 `01`（已采样），符合 W3C Trace Context 规范。旧版服务仍能解析，只是会把未知 `flags` 当作采样位。  
  - **采样决策传播**：在 `get_distributed_tracing_context` 中重新写入当前 OTel span的采样标记，确保 **本节点的采样决定** 能覆盖上游的标记，实现 “parent‑based” 行为。

- **性能影响**  
  - **微量开销**：新增字段的序列化/反序列化及几次字符串切分、十六进制合法性检查，对单次请求的 CPU 负载提升可忽略（< 0.1 µs），对高并发场景没有可观测的影响。  
  - **内存占用**：`trace_flags` 只占 2 字节（在 `String` 中略有额外分配），对整体内存使用影响极小。  

- **安全考虑**  
  - **输入验证**：对 `traceparent` 的版本、trace_id、span_id、trace_flags 均加入严格的十六进制长度校验，防止异常或恶意构造的字符串导致解析错误或潜在的内存泄漏。  
  - **不泄露额外信息**：仅传播采样位（两位十六进制），符合 W3C 规范，不会暴露额外业务数据。  

**⚠️ 潜在风险**  

1. **向后兼容**：某些老版本的下游服务（或自研的追踪解析器）可能假设 `traceparent` 第四段只能是 `01`，在收到 `00` 时可能错误地丢弃或误判。  
2. **采样决策差异**：在启用了 `parentbased_traceidratio` 或自定义 sampler 时，若未正确同步 OTel 采样状态，可能导致 **本节点采样** 与 **上游采样** 不一致，进而出现预期外的 OTLP 导出行为。  
3. **测试依赖环境**：单元测试使用 `tracing_subscriber::registry().with(DistributedTraceIdLayer)`，若生产环境使用了其他层组合，可能遗漏对 `trace_flags` 的传播。  
4. **日志/监控指标变化**：原本基于 “始终采样” 的监控预期会改变，尤其是在 `unsampled` 调用不再产生任何 OTLP span，导致监控仪表板出现 “缺失” 现象。

**💡 关注建议**  

- **兼容性验证**：在部署前对关键 downstream（如日志、监控、链路追踪 UI）进行一次兼容性回归，确认它们能够正确处理 `traceparent` 中的 `00` 标记。  
- **配置显式控制**：若业务需要强制对所有请求采样，可在环境变量或 OpenTelemetry 配置中使用 `OTEL_TRACES_SAMPLER=always_on`，避免因 `unsampled` 导致监控缺失。  
- **监控采样率**：建议在 Grafana / Prometheus 中新增 `otel_trace_sampling_ratio` 之类的指标，实时监控实际导出的 span 采样率，以便快速发现采样策略异常。  
- **文档同步**：更新项目的追踪文档（README、API 手册），明确 `Context.build_trace_headers()` 现在会返回包含真实 `flags` 的 `traceparent`，并标注 “未采样时不产生 OTLP span”。  
- **回滚方案**：如果在生产环境出现不可预料的链路缺失，可临时通过环境变量 `DYN_TRACING_FORCE_SAMPLE=1`（或类似开关）强制把 `trace_flags` 固定为 `01`，快速恢复旧行为。  

---  

此变更在 **保持 W3C Trace Context 完整性**、**修正采样信息丢失** 方面具备显著业务价值，同时对系统整体架构、性能与安全的影响可控。请在上线前完成上述兼容性与监控检查，以确保平滑迁移。

### fix(mocker): per-rank batch for AIC perf under attention-DP in offline replay (#11002)
**SHA**: `3089d1e` | 🔗 [查看提交](https://github.com/ai-dynamo/dynamo/commit/3089d1e8bb4568ad1d4c41e19bd5b672104b234c)

**🎯 变更类型**：Bug修复 / 性能优化  
**⚡ 重要程度**：🔴高  

**📋 变更摘要**  
1. 在离线回放（offline replay）路径中，为 AIC PerfModel 引入 `attention_dp_size` 参数，使其能够在全局批次（global batch）与每个注意力并行（attention‑DP） rank 所需的批次之间进行正确的划分。  
2. 新增构造函数 `from_aic_callback_with_attention_dp` 并在 `materialize_replay_mocker_args` 中使用，以确保离线回放聚合引擎向 AIC SDK 发送的批次是 **每 rank** 的真实值。  
3. 为 `PerfModel` 的 `prefill` 与 `decode` 预测实现加入批次除法（向上取整），并补充单元测试验证除法逻辑在不同 `attention_dp_size` 下的行为。

---

**🎯 影响范围**  
- `lib/bindings/python/rust/llm/replay.rs`（离线回放的参数构造）  
- `lib/mocker/src/common/perf_model.rs`（PerfModel 枚举、实现及测试）  
- 相关单元测试文件（覆盖新逻辑）  

---

**🔍 技术洞察**  

| 维度 | 影响说明 |
|------|----------|
| **架构影响** | - `PerfModel::Aiconfigurator` 现在携带 `attention_dp_size`，相当于在性能模型层面加入了 DP 维度的元信息。<br>- 新增的 `from_aic_callback_with_attention_dp` 仅在离线回放聚合路径被调用，**不影响** 现有在线路径（仍使用默认 `attention_dp_size = 1`）。<br>- 由于 `PerfModel` 实现了 `Clone`，克隆时会复制该字段，保持语义一致性。 |
| **性能影响** | - 之前离线回放使用全局批次直接调用 AIC SDK，导致 **批次被放大**（实际应除以 DP 数），从而产生过于保守的预估、调度延迟、GPU 资源浪费。<br>- 现在通过 `aic_per_rank_batch` 进行 `ceil(batch/attention_dp_size)` 计算，提供 **更精确的预估**，可提升调度器对并行度的利用率，尤其在大规模 DP 场景下收益显著。 |
| **安全考虑** | - 代码仅涉及性能模型计算和回放路径的内部逻辑，没有暴露外部网络或文件 I/O，无直接安全风险。<br>- `attention_dp_size.max(1)` 防止除以 0，确保算术安全。 |
| **可维护性** | - 增加的字段和构造函数清晰文档化，注释说明了两条路径（live vs offline‑replay）的区别，降低误用风险。<br>- 单元测试覆盖了 `attention_dp_size = 1` (no‑op) 与 `>1` 的除法行为，提升后续改动的回归安全性。 |

---

**⚠️ 潜在风险**  

1. **模式匹配破坏**：如果项目外部或内部其他模块直接对 `PerfModel::Aiconfigurator { callback }` 进行模式匹配，新增的 `attention_dp_size` 字段会导致编译错误。需要检查并更新所有匹配分支为 `{ callback, attention_dp_size }`（或使用 `..` 忽略该字段）。  
2. **构造函数遗漏**：已有代码若自行创建 `PerfModel::Aiconfigurator`（而非通过提供的构造函数），在未显式提供 `attention_dp_size` 时会默认使用旧的结构，可能导致离线回放仍使用错误的全局批次。建议统一使用 `from_aic_callback` / `from_aic_callback_with_attention_dp`。  
3. **数值溢出/错误除法**：虽然使用 `usize::div_ceil` 并强制 `max(1)` 防止除 0，但若 `attention_dp_size` 传入极大值（> batch_size），`div_ceil` 会返回 1，逻辑上仍合理，但可能与业务期望不符，需要在调用方保证 `attention_dp_size` 与实际 DP 规模一致。  
4. **行为回归**：在线路径仍使用 `attention_dp_size = 1`，若意外在离线回放之外使用了 `from_aic_callback_with_attention_dp`，则会产生不必要的批次划分，导致性能下降。须在文档或代码注释中再次强调适用范围。  

---

**💡 关注建议**  

1. **代码审查**：搜索全库中 `PerfModel::Aiconfigurator` 的模式匹配或直接构造，统一改为带 `attention_dp_size`（或使用 `..` 忽略）。  
2. **文档更新**：在 `PerfModel` 的 API 文档中明确说明两种构造函数的适用场景（online vs offline‑replay），并标注 `attention_dp_size` 的语义。  
3. **回归测试**：在 CI 中加入对离线回放路径（使用 `from_aic_callback_with_attention_dp`）的集成测试，验证调度器在不同 DP 规模下的吞吐量和延迟符合预期。  
4. **监控指标**：在实际部署时，监控 `predict_prefill_time` 与 `predict_decode_time` 输出的 batch 参数，确保除法后数值符合 “每 rank” 预期，以快速捕获潜在的配置错误。  
5. **升级指南**：如果项目对外提供 `PerfModel` 的手动构造接口（例如在插件或自定义调度器中），发布版本说明中应提醒用户迁移到新构造函数或使用默认 `attention_dp_size = 1`。  

--- 

总体而言，此次修改消除了离线回放场景下因全局批次误传导致的性能模型偏差，提升了多 GPU DP 环境下的调度精准度。只要注意兼容性（模式匹配、手动构造）和配置一致性，风险可控，收益显著。 🚀

### feat(self-host): auto-clean metadata-registry on detach (gh-8749) (#10351)
**SHA**: `e6980d6` | 🔗 [查看提交](https://github.com/ai-dynamo/dynamo/commit/e6980d61be57659d93d610b3feec94b1c9ecbb96)

**🎯 变更类型**：功能增强  

**⚡ 重要程度**：🔴高  

**📋 变更摘要**：  
- 在模型分离（detach）时自动清理 metadata‑registry 条目，避免残留的注册导致后续模型冲突或资源泄漏。  
- 引入 **Owner** 概念（`instance_id + lora_slug`），并把注册键从 `(slug, suffix, filename)` 扩展为 `(namespace, component, endpoint, slug, suffix, filename)`，使得同一 DRT 上多个 endpoint、组件或命名空间的模型可以安全共存。  
- 移除 `LocalModel` 中的 `attached_self_host_suffix`，改为在 `move_to_self_host` 时直接使用 `Endpoint` 信息，并在 detach 时通过 `unregister_for_owner` 清除对应条目。  

**🎯 影响范围**：  
- `lib/llm/src/local_model.rs`（模型构建、self‑host 迁移、detach 逻辑）  
- `lib/runtime/src/metadata_registry.rs`（注册表数据结构、API、冲突检测）  
- `lib/runtime/src/system_status_server.rs`（HTTP 路由与文件查询）  
- `tests/frontend/test_self_host_metadata.py`（测试 URL 与默认组件/endpoint 配置）  

**🔍 技术洞察**  

- **架构影响**  
  - **Owner 机制**：为每个注册条目绑定唯一的 owner（`instance_id` + optional LoRA slug），实现 **auto‑cleanup**，消除了原来通过 `LocalModel.attached_self_host_suffix` 手动追踪的脆弱方案。  
  - **键值扩展**：加入 `namespace/component/endpoint` 三元组后，metadata 注册表不再受限于单一模型 slug，支持同一 DRT 上多模型、多角色（如 `generate`、`embed`）并存，提升系统可组合性。  
  - **冲突检测**：`register` 现在返回 `CollisionError`，在不同 owner 试图注册同一键时显式报错，防止隐藏覆盖导致的不可预期行为。  

- **性能影响**  
  - **内存/CPU**：`MetadataArtifactRegistry` 仍使用 `RwLock<HashMap<...>>`，键值长度略增但整体开销微乎其微；冲突检测只在写入时进行一次比较。  
  - **路径拼接**：`move_to_self_host` 组装 URL 时多了几段字符串，CPU 成本极低。  
  - **自动清理**：`unregister_for_owner` 在 detach 时一次性遍历并删除对应 owner 的所有条目，代价与原来 `unregister(slug, suffix)` 相当，且避免了残留条目导致的后续检索失效。  

- **安全考虑**  
  - **泄漏防护**：旧实现在异常路径或进程异常退出时可能留下 stale 条目，进而让其他模型误读旧文件。新实现在模型 detach 时必然清理，降低了潜在信息泄漏风险。  
  - **冲突错误暴露**：若不同 worker（不同 `instance_id`）意外尝试注册相同键，系统会返回 500（`CollisionError`），不会 silently覆盖，避免了恶意或误操作导致的文件覆盖。  
  - **路径泄露**：路由中加入 `namespace/component/endpoint`，但这些信息本身是公开的配置，不构成安全泄露。  

**⚠️ 潜在风险**  

1. **向后兼容性**：  
   - 旧版本的 worker 会仍然使用旧路由 (`/v1/metadata/{slug}/{suffix}/{*filename}`)。若集群中混合运行新旧版本，客户端需要兼容两套 URL。  
   - 任何未更新的自定义 `metadata_registry` 调用（外部插件）仍可能使用旧 `register` 签名，导致编译错误。  

2. **冲突错误导致启动失败**：  
   - 注册表冲突现在会返回 `CollisionError` 并终止 worker 启动。如果多个 endpoint（或同一 endpoint 的不同实例）意外使用相同 `(namespace, component, endpoint, slug, suffix, filename)`，会导致服务不可用。需要确保 `model_suffix`（LoRA slug）在同一 endpoint 内唯一。  

3. **锁竞争**：  
   - `RwLock` 在高并发自主持久化大量文件时仍可能出现写锁争用，虽然影响极小，但在极端场景（大量 LoRA 动态挂载/卸载）需关注。  

4. **测试完整性**：  
   - 只更新了单 worker 的集成测试，未覆盖多 endpoint、跨 namespace 并存的场景。若未来使用多模型部署，可能出现未捕获的路径错误。  

**💡 关注建议**  

1. **文档与迁移**  
   - 在发布说明中明确 **新旧路由**，建议在所有前端/SDK 中统一使用带 `namespace/component/endpoint` 的路径。  
   - 为已有部署提供迁移脚本或配置选项，使旧 worker 能自动升级（例如在 `worker_startup` 时检测并清除可能的旧键）。  

2. **冲突防护**

### feat(runtime): default event plane to ZMQ for all backends (#10902)
**SHA**: `ce82eb5` | 🔗 [查看提交](https://github.com/ai-dynamo/dynamo/commit/ce82eb56784f6818784614f511f6884b26b3b405)

**🎯 变更类型**：功能增强  
**⚡ 重要程度**：🔴高  
**📋 变更摘要**：将 Dynamo 运行时的事件平面（event plane）默认实现从 “根据发现后端自动选择” 改为统一使用 ZMQ，NATS 仅作为显式可选的传输层。相应地更新了 CLI 参数帮助、文档、内部默认解析逻辑以及测试夹具，以匹配新的默认行为。  

**🎯 影响范围**：  
- `components/src/dynamo/common/configuration/groups/runtime_args.py`、`frontend_args.py`（CLI 参数默认说明）  
- `lib/runtime/src/discovery/mod.rs`、`distributed.rs`（事件传输默认枚举与解析）  
- `docs/design-docs/event-plane.md`、`docs/components/router/router-guide.md`（文档）  
- `tests/*`（测试夹具与 E2E mocker）  
- 路由器与索引器运行时的初始化路径  

**🔍 技术洞察**  

- **架构影响**  
  - **统一默认**：所有发现后端（`file`、`mem`、`etcd`、`kubernetes`）现在默认走 ZMQ，无需外部 NATS 服务即可完成 KV 事件的发布/订阅。  
  - **可选 NATS**：仍通过 `DYN_EVENT_PLANE=nats` 显式开启，保持对已有 NATS‑依赖部署的兼容。  
  - **Discovery 与 Runtime 分离**：`DiscoveryBackend::resolve_event_transport_kind` 逻辑从“后端决定默认”简化为单一 “ZMQ 为默认”，降低了代码的上下文耦合。  
  - **路由器行为**：文档明确将 ZMQ 设为 “router 默认 (local indexer)”，对 router‑kv‑events 的实现没有功能性修改，但对测试中 ZMQ 连接延迟的处理做了补丁（`_pin_nats_event_plane_for_mocker` 夹具）。  

- **性能影响**  
  - **延迟降低**：ZMQ 基于本地套接字的 PUB/SUB，省去网络往返和 NATS 服务器的排队，理论上可提升 KV 事件的内部传播延迟，尤其在单机或小规模集群中。  
  - **资源占用**：不再强制启动 NATS 进程，减少 CPU、内存和端口占用，对 CI 与本地调试有明显加速。  
  - **可扩展性限制**：ZMQ 为点对点模型，跨机器的规模化部署仍需要 NATS；对大规模分布式场景并未提升，只是默认走本地最简路径。  

- **安全考虑**  
  - **ZMQ 本身不提供认证/加密**，在默认情况下仅用于同机或受信网络，风险相对可控。  
  - **NATS** 可配置 TLS、用户/密码等安全特性，若用户显式开启 NATS，则仍保有这些安全机制。  
  - 新的默认行为可能导致在未加防火墙的环境中意外暴露 ZMQ 端口，需要在部署手册中提醒关闭或限制本地网络访问。  

- **可维护性**  
  - 代码路径统一后，`EventTransportKind::from_env` 与 `DiscoveryBackend::resolve_event_transport_kind` 的分支逻辑大幅简化，降低了未来出错概率。  
  - 文档同步完成，减少用户对 “默认取决于后端” 的误解。  

**⚠️ 潜在风险**  

1. **生产回滚风险**：已有部署在未显式指定 `DYN_EVENT_PLANE` 时依赖 NATS（尤其是 etcd/k8s 场景）。默认切换后，若未更新配置或环境变量，将自动走 ZMQ，导致路由器/索引器失去与 NATS 交互的能力，可能出现 “KV 事件未收到” 的错误。  
2. **测试不兼容**：部分测试（如使用 durable KV events、JetStream）仍依赖 NATS。提交已经在测试夹具中加入 `monkeypatch.setenv("DYN_EVENT_PLANE","nats")`，但若未来新增测试忘记设置，可能出现隐藏的失败。  
3. **ZMQ 连接时序**：ZMQ 采用 “slow‑joiner” 模型，发布者在订阅者完全加入前的消息会丢失。当前已为 mocker 场景添加强制 NATS，但在真实用户自行搭建的 `file`/`mem` 环境中仍可能出现因启动顺序导致的事件漏失。  
4. **安全隐患**：在多租户或混合网络环境下，默认开启 ZMQ 可能无意中暴露内部事件总线，需要运维额外配置防火墙或使用 `--event-plane=nats`。  

**💡 关注建议**  

- **部署指南**：在 README 与部署手册中明确说明：生产级分布式部署（etcd/k8s）**仍建议显式设置 `DYN_EVENT_PLANE=nats`**，否则将使用 ZMQ 并可能失去跨节点事件同步。  
- **迁移脚本**：考虑在启动脚本里检测 `--discovery-backend` 为非本地后端且 `DYN_EVENT_PLANE` 未设置时，自动打印提醒或设置 `DYN_EVENT_PLANE=nats`（可通过 `--warn-on-default-event-plane` 之类的 flag 实现）。  
- **CI 增强**：新增纯 ZMQ 环境的 E2E 测试，验证在不启动 NATS 时 `runtime`、`router`、`indexer` 能正常完成 KV 事件流。保持现有 NATS‑dependent 测试的显式 env 设置不变。  
- **连接稳健性**：在 ZMQ 事件发布端实现 “首次发布前等待短暂的 subscriber 握手”，或在 router 启动时使用 `ZMQ_LINGER`、`ZMQ_SNDHWM` 等参数防止丢失首批事件。  
- **安全加固**：提供可选的 ZMQ 安全层（CurveZMQ）或在文档中提醒用户在跨机部署时必须使用 NATS 而非 ZMQ。  

通过上述措施，可最大化利用默认 ZMQ 带来的简化部署优势，同时避免在已有分布式环境或安全敏感场景中产生回归。

- **快照前置检查**：`_validate_supported_snapshot_config` 强制只能使用单 GPU、聚合（Aggregated）文本工作流，防止多 GPU、pipeline、kvbm connector 等会在恢复前产生不可恢复的外部状态。  
- **模型预取**：在捕获前若模型为远程 HF ID（非本地路径且非 GMS 格式），会触发一次 `fetch_model`，确保模型文件在 `HF_HUB_OFFLINE=1` 前已经落盘，防止恢复后因网络不可达而卡死。  
- **CUDA 设备映射**：`BuildDeviceMap` 新增全同映射检测，若所有 GPU 映射为身份映射直接返回空字符串，使用 CUDA 默认恢复路径，避免不必要的跨 GPU 迁移逻辑。  

#### 性能影响
- **捕获路径**：进程在首次捕获后立即 `os._exit(0)`，不进入 Runtime，显著降低捕获期间的资源占用（不启动 NATS、健康检查、监听端口）。  
- **恢复路径**：恢复后首次第一次访问 Runtime 时才创建实际 Runtime，导致 **一次额外的 `await` 延迟**（约几百毫秒，取决于 `create_runtime` 初始化时间）。后续请求性能与原实现相同。  
- **模型预取**：对于远程模型，额外一次 `fetch_model`（网络 I/O）在捕获前完成，捕获时间会略有提升，但可以在快照期间完成，不影响业务服务窗口。  
- **GPU 迁移**：同 GPU 恢复时 `BuildDeviceMap` 返回空，省去映射字符串生成和 CUDA 迁移路径的额外检查，略微提升恢复速度。  

#### 安全考虑
- **环境变量泄漏**：快照前会把 *restore‑time* 环境变量写入 `snapshot-control/restore-context.json`，若容器镜像中存在敏感环境变量，需要确保不被写入或被掩码。  
- **离线模式**：在快照模式下强制 `HF_HUB_OFFLINE=1`，防止恢复后自动发起外部网络请求，降低攻击面。  
- **进程强制退出**：捕获阶段 `os._exit(0)` 会直接终止进程，若外部监控/容器 orchestration 未配置正确的重启策略，可能导致 “进程永久退出” 的误判。  
- **CUDA 设备映射**：同 GPU 的身份映射返回空字符串，避免误将非身份映射当作迁移路径，降低因错误映射导致的非法 GPU 访问风险。  

---

### ⚠️ 潜在风险
1. **配置限制**：仅支持单 GPU、聚合文本路径。用户误把多 GPU、pipeline、multimodal、KVBM connector 等配置用于快照会触发 `ValueError`，导致启动失败。  
2. **恢复流程异常**：若外部 snapshot agent 未正确触发恢复，`snapshot_before_endpoint` 将返回 `False` 并调用 `os._exit(0)`，容器会直接退出，导致业务不可用。  
3. **代理对象误用**：在恢复前访问 `runtime`（如直接调用 `runtime.endpoint`）会抛 `RuntimeError`。如果第三方插件在 `init_worker` 前就尝试使用 Runtime，可能导致意外异常。  
4. **测试覆盖不足**：目前单元测试覆盖核心逻辑，但缺少集成测试（端到端快照+恢复、跨节点恢复、Kubernetes 控制器交互），在真实集群中可能暴露细节不一致。  
5. **CUDA 设备映射回退**：`BuildDeviceMap` 在同 GPU 场景返回空字符串，若后端依赖非空映射（旧版脚本硬编码），可能导致恢复失败。  
6. **资源泄漏**：`_NoOpSnapshotPauseController` 只执行 `gc.collect()`，不对 CUDA 上下文做显式同步/释放，极端情况下可能在恢复后出现残留显存。  

---

### 💡 关注建议
- **文档与告警**：在官方文档和 CRD 注释中明确标注 “仅支持单 GPU 聚合文本路径”，并在 `trtllm` 参数验证层面给出更友好的错误提示。  
- **监控/重启策略**：在 Kubernetes `DynamoGraphDeployment` 中为快照捕获容器添加 `restartPolicy: OnFailure` 或专用 sidecar 检测 `os._exit`，确保容器在捕获后能够被调度器重新启动。  
- **集成测试**：新增 e2e 测试脚本，覆盖从 `snapshot`→`capture`→`restore`→`service ready` 的完整流程，尤其验证网络 sockets、NATS 客户端是否在恢复后重新建立。  
- **安全审计**：审查 `snapshot-control/restore-context.json` 写入的变量列表，确保不包含凭证或敏感信息；在生产环境启用 `snapshot` 前最好使用只读的挂载点。  
- **可扩展性规划**：预留接口在 `snapshot.py` 中为多 GPU、pipeline、kvbm 等场景提供 `EngineSnapshotController` 的实现，便于后续迭代。  
- **异常处理**：在 `snapshot_before_endpoint` 捕获 `Exception` 时记录错误日志并返回明确的退出码，避免容器因未捕获异常而进入未知状态。
