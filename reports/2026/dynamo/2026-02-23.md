### fix(chrek): fix CI errors and replace SIGUSR2 with SIGKILL on checkpoint failure (#6478)
**SHA**: `673822e` | 🔗 [查看提交](https://github.com/ai-dynamo/dynamo/commit/673822ea931352cebe5d6e49a9b66f66a8f1d272)

**🎯 变更类型**：Bug修复 / 安全修复  
**⚡ 重要程度**：🔴高  

**📋 变更摘要**  
1. 将 **checkpoint 失败** 时的信号由 `SIGUSR2` 改为 `SIGKILL`，实现“一键不可捕获的强制终止”。  
2. 相应地删除了 Python 端对 `SIGUSR2` 的事件监听、相关 `asyncio.Event` 状态以及文档中的错误描述。  
3. 同步更新 Go 端 watcher、Helm 文档和用户指南，确保信号约定在代码、部署脚本和官方文档中保持一致。  

**🎯 影响范围**  
- `components/src/dynamo/vllm/checkpoint_restore.py`（模型进程信号处理）  
- `deploy/chrek/pkg/watcher/watcher.go`（Kubernetes DaemonSet checkpoint 监控）  
- `deploy/helm/charts/platform/README.md`、`docs/pages/kubernetes/chrek/standalone.md`（部署文档）  
- CI 配置（已修复导致构建失败的语法/格式问题）  

**🔍 技术洞察**  

- **架构影响**  
  - **信号语义统一**：原先使用 `SIGUSR2` 作为“checkpoint 失败后让进程自行恢复” 的可捕获信号，存在 race window（进程可能在信号处理器尚未安装前被触发）并导致资源（CUDA lock、CRIU 临时文件）无法及时释放。改为 `SIGKILL` 后，由 *watcher* 直接“杀死”进程，省去了在进程内部捕获、判断并自行恢复的逻辑，使 checkpoint 失败路径在架构层面更加 **明确且不可逆**。  
  - **Cleaner 失效处理**：`SIGKILL` 不可被捕获/阻塞，保证出现不可恢复的 CUDA 锁定或 CRIU 错误时，进程必定被清理，避免残留占用的 GPU、文件系统或内存。  
  - **移除 `_checkpoint_failed` 事件**：简化了 `CheckpointRestore` 状态机，降低了事件竞争的可能性，提升代码可维护性。  

- **性能影响**  
  - **正向路径**：删除对 `SIGUSR2` 的等待逻辑，`asyncio.wait` 中的 `failed` 任务不再创建，微量减小事件调度开销（几乎可以忽略）。  
  - **失败路径**：`SIGKILL` 的即时终止避免了因进程卡死而导致的调度器（Kubernetes）超时重启循环，整体集群资源回收更快。  
  - **CI 稳定性**：修复 lint/format 错误后，CI 运行时间略有下降，间接提升开发者反馈速度。  

- **安全考虑**  
  - **强制终止**：`SIGKILL` 直接结束进程，确保在出现严重错误（例如 CRIU dump 失败、GPU 驱动异常）时，不会留下具有潜在安全风险的“僵尸进程”。  
  - **不可捕获**：虽然提升了可靠性，但也意味着业务方失去了在进程内部进行自定义错误上报或资源清理的机会，需要在 **checkpoint job**（Kubernetes Job）层面补充 **后置钩子**（如 `preStop`、`postStop`）或 **Watcher** 的错误记录，以防泄露敏感信息。  
  - **信号冲突**：`SIGKILL` 是保留信号，避免与 POSIX job‑control 或容器内部已有的信号处理冲突，降低误触发概率。  

**⚠️ 潜在风险**  

1. **资源泄露**：因为 `SIGKILL` 无法在进程内部执行清理代码，如果 checkpoint 失败前已经分配了 GPU 锁、临时文件或网络资源，可能需要依赖容器运行时或节点的 **cgroup / kubelet** 来回收。若回收不及时，可能导致节点资源被占用。  
2. **监控盲区**：原来的 `SIGUSR2` 事件可以被捕获并上报至业务监控系统，改为 `SIGKILL` 后，监控必须依赖 **Watcher** 的日志/事件（如 `CheckpointFailed`）或 Pod 的 **status.phase=Failed**。如果监控体系未同步更新，可能出现“失败未被感知”的情况。  
3. **兼容性**：已有用户在自定义代码里仍然监听 `SIGUSR2` 进行业务恢复，升级后这些代码将不再触发，可能导致意外的 **silent failure**。  
4. **CI/测试用例**：部分单元/集成测试可能仍假设 `SIGUSR2` 存在，需要相应更新测试套件，否则 CI 通过率回落。  

**💡 关注建议**  

- **在 checkpoint Job 中添加 `preStop` / `postStop` Hook**，确保即使被 `SIGKILL` 终止，也能通过容器生命周期钩子执行必要的资源回收（例如 `nvidia-smi --gpu-reset`、删除 `/tmp/ready-for-checkpoint`）。  
- **完善监控**：在 Watcher 的错误分支中增加 **Prometheus metric**（如 `chrek_checkpoint_failure_total`）以及 **事件注解**，并在平台监控仪表板上显示，以弥补进程内部无法上报的缺口。  
- **文档与 SDK 同步**：确保所有公开的 SDK 示例（Python、Go、Rust）均使用最新的 `checkpoint_restore.py` 接口；若项目提供了 `chrek-client` 库，需要在下一个版本中移除对 `SIGUSR2` 的依赖。  
- **回滚策略**：如果业务方对强制 `SIGKILL` 有特殊要求（需要自定义恢复），可提供 *可选* 配置开关（如 Helm `checkpoint.failureSignal=SIGUSR2|SIGKILL`），但默认保持 `SIGKILL` 以保证一致性。  
- **CI/CD 测试覆盖**：加入针对 `SIGKILL` 场景的端到端测试，验证 Pod 在 checkpoint 失败时能够快速进入 `Failed` 状态并触发相应的 **PodDisruptionBudget** / **Job retry** 机制。  

通过上述改动，系统在 **checkpoint 失败** 情况下的可靠性与安全性得到显著提升，同时也需要在运维、监控与文档层面做好配套工作，以防止因信号语义变化而导致的业务中断。

### feat: Backend accept new requests during shutdown grace period (#6093)
**SHA**: `ea86df2` | 🔗 [查看提交](https://github.com/ai-dynamo/dynamo/commit/ea86df2985bf7a648eac28f455e238d14bad6fd2)

**🎯 变更类型**：功能增强 / 架构变更 / 安全修复  
**⚡ 重要程度**：🔴高  

**📋 变更摘要**  
本次 PR 为 Dynamo 后端引入了“优雅关闭”机制：在收到 SIGTERM/SIGINT 时，先从服务发现中注销所有 endpoint，随后进入可配置的 **grace period（默认 5 s）**，在此期间运行时仍保持活跃并可接受请求，随后再执行 `runtime.shutdown()` 完成真正的停机。为实现该行为，新增 `graceful_shutdown.py`，并在各子项目（sglang、trtllm、vLLM）中移除原有的信号处理逻辑，统一使用 `install_signal_handlers`。同时提供环境变量 `DYN_GRACEFUL_SHUTDOWN_GRACE_PERIOD_SECS` 供用户调节。

---

**🎯 影响范围**  
- **核心运行时** (`components/src/dynamo/common/utils/runtime.py`) – 删除原有信号注册， `create_runtime` 只负责实例化 `DistributedRuntime`。  
- **信号/关闭工具** (`graceful_shutdown.py`) – 新增统一的 signal‑handler、注销 discovery、等待 grace‑period、触发 runtime shutdown。  
- **所有后端入口**：`sglang/main.py`、`trtllm/main.py`、`vllm/main.py`（以及对应的 worker 初始化）均改为显式调用 `install_signal_handlers`，并在每个 worker 初始化时将自己创建的 endpoint 写入共享的 `shutdown_endpoints` 列表。  
- **文档**：`docs/pages/fault-tolerance/graceful-shutdown.md` 更新为新的关闭流程。  
- **测试**：迁移测试强制 `DYN_GRACEFUL_SHUTDOWN_GRACE_PERIOD_SECS=0`，确保旧的快速关闭行为在 CI 中仍然通过。

---

**🔍 技术洞察**  

| 维度 | 影响 |
|------|------|
| **架构影响** | - **信号处理解耦**：所有组件不再在 `create_runtime` 中自行注册 signal handler，统一由 `install_signal_handlers` 完成，提升代码可维护性。<br>- **发现注销**：在真正关闭之前主动从服务发现系统（Kubernetes/Etcd/文件等）注销 endpoint，防止调度层仍将流量转发到即将退出的实例。<br>- **共享 shutdown_endpoints**：使用可变的列表在各子模块之间传递需要注销的 endpoint，降低跨模块耦合，但也引入了“忘记填充列表” 的隐患。 |
| **性能影响** | - 额外的 **Grace Period**（默认 5 s）会延迟容器退出，影响水平伸缩/滚动升级的总体时间。<br>- 注册/注销 discovery 的异步调用成本极低（相当于几次 RPC），对运行时吞吐基本无影响。 |
| **安全考虑** | - **更安全的停机**：在关闭信号触发后立即让调度层不可见（unregister），即使在 grace period 内仍接受内部请求，也不会再被外部流量访问，降低了关闭窗口期间的潜在攻击面。<br>- 环境变量解析加入了容错（非数值、负数 → 使用默认或 0），避免因配置错误导致异常退出。 |

| **兼容性** | - 移除了 `create_runtime(..., shutdown_event)` 参数，现有调用若仍传参会编译错误。<br>- 需要在每个 worker 初始化时手动填充 `shutdown_endpoints`，否则在关闭阶段未能注销对应 endpoint，可能导致 “zombie endpoint” 留在 discovery 中。 |

**⚠️ 潜在风险**  

1. **忘记填充 `shutdown_endpoints`**  
   - 若某新组件未将自己的 endpoint 加入列表，关闭时将不会注销该 endpoint，导致外部流量仍尝试路由到已下线实例，产生 5xx 错误或请求卡死。  
2. **信号竞争**  
   - `install_signal_handlers` 通过 `loop.add_signal_handler` 捕获内部注册的 handler 并延迟执行，若某子模块在关闭前自行调用 `loop.remove_signal_handler` 或重新设置同一信号，可能导致原有 handler 失效。  
3. **Grace Period 过长**  
   - 在高频滚动升级场景下，默认 5 s 的等待会累积导致整体升级窗口显著延长。用户若不显式调小或关闭，可能触发调度平台的 “pod not ready” 超时。  
4. **异常退出路径**  
   - `graceful_shutdown_with_discovery` 在注销 endpoint 时使用 `asyncio.gather(..., return_exceptions=True)`；如果底层 discovery 客户端出现不可恢复的网络错误，仍会继续执行后续 shutdown，可能导致运行时在未完全清理的状态下退出。  
5. **测试依赖环境变量**  
   - 迁移测试强制 `GRACE_PERIOD=0`，但如果 CI 环境意外未设置该变量，实际运行时会出现非零等待，导致测试超时。

### feat: add --headless mode for multi-node TP/PP in dynamo.vllm (#6204)
**SHA**: `4ebb244` | 🔗 [查看提交](https://github.com/ai-dynamo/dynamo/commit/4ebb244b2af77af81a797569080c66c6015afb70)

**🎯 变更类型**：功能增强  

**⚡ 重要程度**：🔴高  

**📋 变更摘要**  
- 为 `dynamo.vllm` 引入 `--headless` 标志，支持在多节点 Tensor/Pipeline 并行（TP/PP）场景下仅启动 vLLM worker 而不启动 Dynamo 核心、调度器及服务端点。  
- 在 `backend_args.py` 中新增 CLI 参数并映射到 `DynamoVllmConfig.headless`；在 `main.py` 中实现构造 `argparse.Namespace`、调用 vLLM 原生 `run_headless` 并在 `worker()` 流程中根据 `config.headless` 分支走专门路径；同时补充单元测试、文档、示例脚本以及 CI 测试案例。  

**🎯 影响范围**  
- `components/src/dynamo/vllm/backend_args.py`（CLI 参数）  
- `components/src/dynamo/vllm/main.py`（运行时入口）  
- `components/src/dynamo/vllm/tests/test_vllm_unit.py`（单元测试）  
- `docs/pages/backends/vllm/multi-node.md`（使用文档）  
- `examples/backends/vllm/launch/multi_node_tp.sh`（多节点启动脚本）  
- `tests/serve/launch/multi_node_tp_headless.sh`（CI 本地多节点验证脚本）  
- `tests/serve/test_vllm.py`（端到端测试配置）  

---

### 🔍 技术洞察  

| 维度 | 影响描述 |
|------|----------|
| **架构影响** | - 在多节点部署时，**第二节点（及后续）**仅运行 vLLM worker 进程，**跳过** Dynamo 的 `DistributedRuntime`、调度器、NATS/etcd 等控制平面。<br>- 通过 `run_headless` 直接利用 vLLM 自身的多进程/分布式启动逻辑，降低了 Dynamo 对节点角色的耦合度。<br>- 需要在配置层面区分 **head** 与 **worker**，但仍保持统一的 `dynamo.vllm` 可执行文件，简化部署脚本。 |
| **性能影响** | - **Worker 节点**不再加载 Dynamo 核心组件（如模型调度器、端点服务），显著降低 CPU、内存占用，提升 GPU 利用率。<br>- 网络交互仅限于 vLLM 内部的 NCCL/torch.distributed，而不经过额外的 NATS/etcd 通道，理论上可以略微提升跨节点的推理吞吐。<br>- 对 **head 节点**的性能影响有限，仅额外引入一次 `run_headless` 的检测路径，开销可忽略不计。 |
| **安全考虑** | - **攻击面缩小**：worker 进程不对外暴露 Dynamo HTTP/gRPC 接口，也不启动 NATS/etcd 客户端，减少潜在的远程攻击入口。<br>- 需要确保 `--headless` 只能在期望的“worker”节点使用，误用在唯一节点上会导致缺失调度服务，建议在部署脚本或 CI 中加入角色校验。<br>- 环境变量 `DYN_VLLM_HEADLESS` 的默认值为 `false`，未显式开启时行为不变，安全风险极低。 |
| **可维护性** | - 新增的代码主要集中在 `main.py` 的辅助函数 `build_headless_namespace` 与 `run_dynamo_headless`，逻辑清晰且已配单元测试。<br>- 与 vLLM 的原生 `run_headless` 交互，后者若未来 API 改动，只需要更新本仓库的适配层即可。<br>- 文档和示例脚本同步更新，降低了使用者的学习成本。 |

---

### ⚠️ 潜在风险  

1. **与 Checkpoint 模式冲突**  
   - 代码已经在 `worker()` 中检测 `config.headless` 与 `DYN_CHECKPOINT_SIGNAL_FILE`（`checkpoint_cfg`）的并存并抛异常。若未来新增的 checkpoint 相关标志不走同一路径，可能出现未捕获的冲突。  

2. **误用 `--headless`**  
   - 若在仅有单节点的部署中误加 `--headless`，Dynamo 的调度、服务端点会缺失，导致请求 404/连接错误。  

3. **缺失必需的 CLI 参数**  
   - `run_headless` 期望 `api_server_count` 等字段，当前通过 `build_headless_namespace` 强行补默认 `0`。若 vLLM 在未来引入新必需字段，当前适配层可能未同步，导致运行时 `AttributeError`。  

4. **测试依赖环境**  
   - `tests/serve/launch/multi_node_tp_headless.sh` 依赖本机两块 GPU 与 `torch.distributed` 正常初始化，CI 环境若缺少对应硬件或驱动，可能导致测试失败。  

5. **日志与监控**  
   - 由于绕过 `DistributedRuntime`，worker 节点不再上报 Dynamo 侧的 Prometheus 指标，可能导致监控盲点。若监控体系依赖统一的指标收集，需要在文档说明此差异。

### 💡 关注建议  

1. **CI/部署层校验**  
   - 在 CI 或部署脚本中加入检查：`if ROLE == "worker" && !args.headless` → 警告或强制使用 `--headless`。  
   - 同理，`if ROLE == "head" && args.headless` → 直接报错，防止误用。  

2. **向前兼容适配**  
   - 将 `build_headless_namespace` 实现为逐字段复制，同时保留一个 “fallback” 机制：`for field in REQUIRED_FIELDS: setattr(ns, field, getattr(config.engine_args, field, DEFAULTS[field]))`，便于未来 vLLM 增添必需字段。  

3. **监控与日志**
