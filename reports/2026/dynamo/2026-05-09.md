### fix(planner): make SLA-mode scale-down consolidation-aware again (#9294)
**SHA**: `90b4fbf` | 🔗 [查看提交](https://github.com/ai-dynamo/dynamo/commit/90b4fbf3b56607027e386aeffc6b13491ecfa309)

**🎯 变更类型**：功能增强 / Bug修复  
**⚡ 重要程度**：🔴 高  
**📋 变更摘要**：在 SLA‑mode 的动态扩缩容调度器中引入了 **consolidation‑aware scale‑down** 逻辑，确保在合并（N→N‑1）节点时能够评估合并后对 TTFT/ITL 的实际影响并在必要时拒绝缩容。新增诊断状态 `scale_down_refused_consolidation`，并在 Prometheus 指标、单元测试以及调度器内部实现中完整支持。  

**🎯 影响范围**  
- `components/src/dynamo/planner/core/load_scaling.py`（核心调度决策）  
- `components/src/dynamo/planner/monitoring/planner_metrics.py`（新增枚举值）  
- 单元测试 `planner/tests/unit/*`（覆盖新路径）  

**🔍 技术洞察**  

| 维度 | 影响描述 |
|------|----------|
| **架构影响** | - 调度器从单纯的 “是否所有 worker 超过阈值” 判断，演进为 **两阶段**：① 先通过回归模型预测合并后（`N/(N‑1)`）的排队/KV 负载，② 依据 `load_scaling_down_sensitivity` 决定是否安全缩容。<br>- 通过在 `_agg_prefill_scaling`、`_prefill_load_decision`、`_decode_load_decision`、`_agg_decode_scaling` 中返回 **`num_workers`**（而非 `None`）来让上层调度器区分 “安全拒绝” 与 “未产生信号”。<br>- 新增诊断状态在 `PlannerMetrics` 中注册，保持对外监控的向后兼容。 |
| **性能影响** | - 每次 tick 额外计算一次 **后合并** 的 TTFT/ITL（队列/ KV 乘以 `N/(N‑1)`），以及一次 **预算检查**（`(SLA‑T_own) * sensitivity`）。<br>- 这些计算仅涉及已有回归模型的再次调用，开销在微秒级，对整体调度频率（秒级）影响可忽略。<br>- 通过提前拒绝不安全的缩容，避免了后续因为 SLA 违约导致的 **频繁放大‑缩小** 循环，整体系统吞吐与延迟有望提升。 |
| **安全考虑** | - 变更不涉及网络、权限或代码注入，仅是调度决策的业务逻辑。<br>- 新增的 `scale_down_refused_consolidation` 通过枚举显式声明，避免了在 Prometheus 导出阶段出现 `ValueError`，提升了监控系统的健壮性。 |
| **可维护性** | - 代码结构更清晰：缩容安全检查集中在各子决策函数内部，调度器核心只负责聚合结果。<br>- 引入了 `can_scale_down` 参数并在 `_scale_decision` 中统一处理，便于后续扩展（如加入其他安全规则）。<br>- 单元测试覆盖了 **decode、prefill、agg** 三种模式的拒绝路径，提升了回归可靠性。 |
| **业务价值** | - 解决了 *“SLA‑mode 缩容在 2→1 场景下出现振荡”* 的历史缺陷，提升了生产环境的资源利用率与稳定性。<br>- 为用户提供了更细粒度的诊断信息，可在运营仪表盘快速定位 “因合并安全” 拒绝的缩容。 |

**⚠️ 潜在风险**  
1. **回归模型不收敛**：在极端负载（KV / prefill token 波动剧烈）时，模型可能返回 `None`，导致 `can_scale_down` 被误判为 `True`。  
2. **配置敏感**：`load_scaling_down_sensitivity` 设得过低会导致过度保守，资源闲置；过高则可能仍出现 SLA 违规。  
3. **兼容性**：若外部监控系统仍使用旧版 `LOAD_DECISION_STATES` 常量，可能在升级后出现 `KeyError`（已通过测试捕获，但生产环境的自定义代码需同步更新）。  
4. **心跳/调度频率**：在极高频率的 `tick` 场景下，额外的回归调用略微增加 CPU 使用，需确认不会触发调度器超时。  

**💡 关注建议**  
- **监控 & 报警**：在 Prometheus 面板中加入 `scale_down_refused_consolidation` 的趋势图，观察新状态的触发频率，确保阈值设置合理。  
- **回归模型健康**：定期检查 `_prefill_regression`、`_decode_regression`、`_agg_regression` 的拟合 R²，防止因数据异常导致 `None`。  
- **配置调优**：建议在生产环境先使用 `load_scaling_down_sensitivity = 80`（默认），随后通过 A/B 实验微调至 70‑90 区间，以平衡资源利用与 SLA 稳定。  
- **回归测试**：在 CI 中加入 **负载突增后** 的全链路测试，验证缩容被正确拒绝且不出现 “scale_up‑scale_down 振荡”。  
- **文档与升级指南**：在发布说明中标明新增枚举值及其意义，提醒用户更新自定义监控/报警规则。  

---  
*总体评估：本次改动通过引入合并感知的安全检查，显著提升了 SLA‑mode 的调度可靠性，风险可控，建议尽快在全链路 CI 与预生产环境验证后推广。*

### fix: align XML parser parity whitespace (#9350)
**SHA**: `3a09b08` | 🔗 [查看提交](https://github.com/ai-dynamo/dynamo/commit/3a09b0875d1b9a0494929f25841cd641e524df99)

**🎯 变更类型**：Bug修复 / 兼容性调整  
**⚡ 重要程度**：🔴高  
**📋 变更摘要**：此次提交对 XML 风格的工具调用解析器做了细粒度的空格与后缀处理修正，使 Dynamo 在 Qwen‑3‑Coder、Minimax、Nemotron 等模型的 XML 模板下，能够 **保持与其他实现（vLLM、SGLang）在空格截断上的一致性**，并且在出现未闭合的 `<tool_call>` 块时 **不再错误地保留后缀文本**。相应的单元测试、Parity 测试用例以及文档表格也同步更新。  

**🎯 影响范围**：  
- `lib/parsers/src/tool_calling/parsers.rs`（工具调用统一抽象层）  
- `lib/parsers/src/tool_calling/xml/parser.rs`（XML 解析核心）  
- `tests/parity/*`（多模型兼容性基准测试）  
- `tests/parity/README.md`（兼容性矩阵文档）  

**🔍 技术洞察**  

- **架构影响**  
  - 解析流程中加入了 **“只在首次解析到工具调用前收集普通文本”** 的判断 (`if calls.is_empty()`) ，防止在多次工具调用出现时错误拼接后续普通文本。  
  - 对 `normal_text` 的 `trim` 处理被迁移到 **仅当未解析到任何工具调用时** 执行，保持原始空格在有工具调用的情形下不被裁剪。  
  - 移除了原先对 `<function>` 结束标记后缀的保留逻辑，统一为 **“工具调用块后视为非内容”**，符合 Qwen3‑Coder 风格的约定。  
  - 这些改动保持了现有模块接口不变，仅在内部实现上调整了文本分段与合并策略，对系统整体架构无侵入性影响。

- **性能影响**  
  - 代码改动仅涉及少量条件判断与 `Vec::push` 的额外分支，对解析路径的时间复杂度保持 **O(n)**，且没有引入额外的内存分配。  
  - 由于 `normal_parts.join("")` 仍在末尾统一执行，整体性能基本保持不变，实际运行时的微小差异可忽略。

- **安全考虑**  
  - 此次修改纯粹是文本处理层面的行为调整，不涉及外部输入的解析深度或安全敏感的字符串转义，未引入新的安全风险。  
  - 仍然保留对不完整 `<tool_call>` 结构的容错（EOF recovery），但在恢复后不再错误传播后缀文本，降低潜在的 **信息泄露**（误将非意图内容当作 normal_text 输出）的风险。

**⚠️ 潜在风险**  

1. **兼容性回归**：对已有使用 Dynamo 解析器的业务，若依赖原先在工具调用后保留的后缀文本（例如日志拼接或 UI 展示的细节），将会发现这些文本被截断或丢失。  
2. **空格敏感场景**：某些 Prompt 可能对空格保留有语义要求（如代码生成、SQL 语句），现在在有工具调用的情况下空格不再被 `trim`，可能导致 **生成文本与预期略有差异**。  
3. **测试覆盖不足**：虽然已更新核心测试和 parity 用例，但仍有可能存在未被覆盖的模型/模板组合（如自定义 `<tool_call>` 包装），需在 CI 中保持对新老行为的对比验证。

**💡 关注建议**  

- **回归验证**：在升级前，请在关键业务的 Prompt 示例上跑一次 **全链路回归**，尤其是涉及工具调用后仍有自然语言输出的场景。  
- **文档与迁移指南**：在 `README` 或 Release Notes 中明确说明 **“工具调用后不再保留 trailing normal_text”**，并提供 **如何手动拼接后缀** 的示例（如在模型侧自行追加）。  
- **监控空格差异**：如果业务对空格极其敏感，建议在生成后加入 **统一的空格标准化步骤**（`trim_end` / `trim_start`）以防止跨版本差异。  
- **扩展测试**：考虑添加针对 **多工具调用并混杂普通文本** 的案例，确保 `normal_parts` 只在首个调用之前收集文本的逻辑在所有组合下均表现一致。  

---  

此提交在提升 Dynamo 与业界其他解析实现空格处理一致性的同时，也简化了 XML 解析器的实现逻辑。只要注意上述兼容性点，升级收益大于风险。祝开发顺利 🚀

### refactor(llm): Deduplicate DeepSeek prompt encoders v3.2 and v4 (#9322)
**SHA**: `34d55a5` | 🔗 [查看提交](https://github.com/ai-dynamo/dynamo/commit/34d55a596fb8d3d44daefe425ec1e303131f4d2c)

**🎯 变更类型**：重构 / 架构变更  
**⚡ 重要程度**：🔴 高  

**📋 变更摘要**  
- 新增 `deepseek_common.rs`，抽取 DeepSeek（v3.2、v4）提示词格式化器之间的 600 多行重复实现，包括 token 常量、JSON 序列化工具、渲染模板、消息合并、工具调用参数编码、思考模式解析等公共逻辑。  
- `prompt.rs` 公开 `deepseek_common` 模块；`deepseek_v32.rs` 与 `deepseek_v4.rs` 现在只保留模型特有的少量代码，改为直接复用 `deepseek_common` 中的函数和结构。  
- 同时对思考模式解析、工具与响应格式注入的实现统一到 `deepseek_common::inject_tools_and_response_format`，并统一使用 `normalize_message_contents`（可配置非文本处理方式）。  

**🎯 影响范围**  
- `lib/llm/src/preprocessor/prompt/*.rs`（所有 DeepSeek 相关 formatter）  
- 依赖 DeepSeek formatter 的任何上层调用（如 OpenAI 兼容入口、模型推理路由）  
- 测试套件中涉及 DeepSeek v3.2/v4 的全部单元/集成测试  

**🔍 技术洞察**  

- **架构影响**  
  - **模块化提升**：公共实现集中在 `deepseek_common`，降低了代码重复率（约 30% 行数削减），便于后续维护、特性统一和 bug 修复。  
  - **清晰的职责划分**：`deepseek_common` 负责 *pure* 业务逻辑（token、JSON 格式化、消息合并等），各版本 formatter 只保留版本特有的模板常量与少量包装代码。  
  - **可扩展性**：若后续出现 DeepSeek v5，只需在对应文件中 `use deepseek_common::*` 并提供版本特有的模板即可，避免再次复制大量实现。  

- **性能影响**  
  - **运行时开销**：新增的模块层级仅是编译期的名字空间组织，未引入额外动态分配或循环结构，实际执行路径与原实现几乎相同。  
  - **序列化优化**：`to_json` 采用自定义 `PythonFormatter`，省去两次 `serde_json::to_string`（一次生成紧凑 JSON、一次手动插入空格），在大量工具参数序列化时略有提升。  
  - **内存使用**：`merge_tool_messages`、`sort_tool_results_by_call_order` 等函数在原实现中已使用 `Vec::with_capacity`，保持同等内存预分配策略，无明显回退。  

- **安全考虑**  
  - **输入校验集中**：所有对外部 JSON（如 `tool_call["arguments"]`）的解析均通过 `anyhow::Context` 报错，并在解析失败时回退到 `{ "arguments": raw_string }`，行为保持与 Python 参考实现一致，避免因异常导致 panic。  
  - **错误路径显式**：`to_json` 中对 `serde::Serialize` 的 `expect` 仍假设序列化必然成功（因为 `JsonValue` 本身可序列化），相对安全；在 `deepseek_v4` 旧实现中有额外 `warn!` 回退，已被统一为 `expect`，风险相同。  
  - **日志信息**：对非文本块的丢弃统一使用 `tracing::warn!`，可在生产环境通过日志审计检测异常输入。  

- **可维护性**  
  - **单元测试迁移**：原来分散在 `deepseek_v32.rs` 与 `deepseek_v4.rs` 的测试逻辑现在统一到 `deepseek_common`，并在各版本文件中保留少量行为验证，降低维护成本。  
  - **代码可读性**：通过 `pub use` 将 `ThinkingMode`、`tokens` 等关键类型重新导出，保持向后兼容的同时让上层代码依然使用原路径。  

**⚠️ 潜在风险**  

| 风险点 | 可能后果 | 缓解措施 |
|--------|----------|----------|
| **行为回归**：公共实现细微差异（如 `to_json` 空格处理、`normalize_message_contents` 对非文本的处理方式）可能导致与历史模型输出不完全一致，影响已有用户的提示模板匹配。 | 与期望的 Python 参考实现不符，导致模型推理结果偏差或测试失败。 | 在 CI 中加入对比旧版与新版在真实请求下的 Prompt 输出；对关键模板（工具渲染、思考块）进行回归测试。 |
