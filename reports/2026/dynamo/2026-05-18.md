### fix: kvbm breaking change - Updated kv_cache_connector lib import to reflect the updates made to trtllm 1.3.0rc14 (#9622)
**SHA**: `8fbd862` | 🔗 [查看提交](https://github.com/ai-dynamo/dynamo/commit/8fbd8623a46f5ab779c9a1789b1773dbcae922ed)

**🎯 变更类型**：Bug修复  
**⚡ 重要程度**：🔴高  
**📋 变更摘要**：此次提交更新了 `tensorrt-llm` 依赖至 `1.3.0rc14`，并相应修正了 `kvbm` 绑定层的导入路径，将原来的 `tensorrt_llm._torch.pyexecutor.kv_cache_connector` 更改为新的 `tensorrt_llm._torch.pyexecutor.connectors.kv_cache_connector`。此改动消除了因上游库模块迁移导致的 `ImportError`，确保在新的 TRT‑LLM 版本下编译和运行不产生失败。  

**🎯 影响范围**：  
- `lib/bindings/kvbm/python/kvbm/trtllm_integration/connector/kvbm_connector_leader.py`  
- `lib/bindings/kvbm/python/kvbm/trtllm_integration/connector/kvbm_connector_worker.py`  
- `pyproject.toml`（`trtllm` 可选依赖版本）  
- `benchmarks/pyproject.toml`（文档注释更新）  

**🔍 技术洞察**：  
- **架构影响**：  
  - 只涉及 `kvbm` 与 TRT‑LLM 之间的适配层，未触及核心推理或调度逻辑。  
  - 通过更新导入路径，保持了 `kvbm` 作为插件的统一入口，避免因上游模块路径变动导致的模块加载失败。  

- **性能影响**：  
  - 该改动本质上是 **代码层面的重定位**，对运行时性能影响微乎其微。  
  - 若 TRT‑LLM 在 `rc14` 中对 `KvCacheConnector*` 实现做了内部优化，`kvbm` 将自动受益，无需额外改动。  

- **安全考虑**：  
  - 未引入新的依赖或改变安全关键路径，风险极低。  
  - 需要确保新版本的 `tensorrt-llm` 本身已通过官方安全审计（一般情况下官方发布的 RC 版本已完成基本安全检查）。  

**⚠️ 潜在风险**：  
1. **API 兼容性风险**：虽然导入路径已更新，但如果 `rc14` 对 `KvCacheConnectorScheduler`/`KvCacheConnectorWorker` 的签名或行为做了细微改动（如返回类型、异常类型），可能在运行时触发 `TypeError` 或逻辑错误。  
2. **依赖冲突**：项目同时支持 `vLLM`、`SGLang` 等其他后端，升级到 `tensorrt-llm==1.3.0rc14` 可能导致这些后端的版本约束产生冲突（例如 `vLLM 0.11.2` 对 TRT‑LLM 版本的需求）。  
3. **CI/构建环境**：CI 脚本或镜像中仍可能缓存旧的 `tensorrt-llm` 包，导致本地与 CI 环境行为不一致。  

**💡 关注建议**：  
- **回归测试**：在升级后跑一遍完整的 `kvbm` 单元测试、集成测试以及 benchmark，以捕获可能的 API 兼容性问题。  
- **兼容层检查**：对比 `tensorrt-llm` 1.3.0rc11 与 rc14 的 release notes，确认 `KvCacheConnector*` 接口是否保持向后兼容。若有变动，适配相应的调用方式。  
- **依赖管理**：在 `pyproject.toml` 中使用精确版本锁定（如 `==1.3.0rc14`）的同时，更新 CI 镜像的 `pip` 缓存，防止旧版本残留。  
- **文档同步**：在项目 README / CONTRIBUTING 中注明支持的 TRT‑LLM 版本范围（最低 rc14），帮助下游使用者避免版本不匹配。  
- **监控 runtime**：部署后监控日志中的 `ImportError`、`AttributeError`，以及可能的性能回退，以便快速定位未捕获的兼容性问题。

### fix(operator): surface specific failure reason in DGDR Succeeded condition (#8227)
**SHA**: `f20ff4e` | 🔗 [查看提交](https://github.com/ai-dynamo/dynamo/commit/f20ff4e28acc37c680a4cb1993fdb68a12f2fbd7)

**🎯 变更类型**：Bug修复  
**⚡ 重要程度**：🔴 高  
**📋 变更摘要**：本次提交在 Operator 代码中统一并改进了 `Condition` 的设置方式，去除手动写入 `LastTransitionTime`，改为让 `meta.SetStatusCondition` 自动填充；在 DGDR（DynamoGraphDeploymentRequest）流程中引入 `setSucceededCondition` 返回值以便检测是否真的产生了变更，并在失败阶段聚合子条件的具体 `Reason/Message`，从而在 `Succeeded` 条件中直接展示根因。  

**🎯 影响范围**：  
- `deploy/operator/internal/controller/dynamo_model_controller.go`（DynamoModel）  
- `deploy/operator/internal/controller/dynamocheckpoint_controller.go`（DynamoCheckpoint）  
- `deploy/operator/internal/controller/dynamographdeployment_controller.go`（DynamoGraphDeployment）  
- `deploy/operator/internal/controller/dynamographdeploymentrequest_controller.go`（DynamoGraphDeploymentRequest）  

**🔍 技术洞察**  

- **架构影响**  
  - 仅限于 Operator 的状态更新逻辑，未改动核心业务流程或 CRD 结构。  
  - 通过统一 `meta.SetStatusCondition` 的调用方式，提高了条件管理的一致性，降低了手动维护 `LastTransitionTime` 与 `ObservedGeneration` 的出错概率。  
  - 在 DGDR 中新增对 `ConditionTypeValidation` 的显式 `True` 设置，使 “验证通过” 成为可观察的状态，提升了状态机器的可追溯性。  

- **性能影响**  
  - 代码改动主要是字段赋值与条件聚合，执行路径几乎不变，CPU/内存开销可忽略不计。  
  - 由于去掉了 `metav1.Now()` 的显式调用，略微减少了时间获取的系统调用，整体性能更好（但影响极小）。  

- **安全考虑**  
  - 公开了更细粒度的失败原因（如 `Validation failed`, `Profiling failed`），有助于运维定位问题。  
  - 这些信息均为内部调度/硬件错误，不涉及敏感数据泄露，安全风险极低。  

**⚠️ 潜在风险**  

1. **兼容性**  
   - 某些外部工具或脚本可能依赖 `Condition.LastTransitionTime` 字段的显式写入。如果 `meta.SetStatusCondition` 在未来的 `controller-runtime` 版本中行为改变，可能导致时间字段缺失。建议在升级前确认该函数仍会自动填充时间。  

2. **条件聚合逻辑**  
   - `setSucceededCondition` 现在会遍历子条件并选取首个 `False` 且带有 `Message` 的条件作为最终 `Reason/Message`。若后续新增子条件但忘记在此列表中登记，可能导致聚合信息不完整。  

3. **更新冲突**  
   - 在 `handleFailedPhase` 中加入对 `Succeeded` 条件的强制更新（若有变化）并立即调用 `Status().Update`，增加了一次写入操作。若该对象正被其他控制器并发更新，可能触发冲突，需要确保 `reconcile` 能正确处理 `Conflict`（已有的重试逻辑应足够）。  

**💡 关注建议**  

- **回归测试**：对所有涉及条件更新的 CR（DynamoModel、DynamoCheckpoint、DynamoGraphDeployment、DynamoGraphDeploymentRequest）执行一次完整的 E2E 流程，验证 `kubectl get <cr> -o yaml` 中的 `conditions` 字段包含正确的 `type/status/reason/message/observedGeneration/lastTransitionTime`。  

- **监控/告警**：更新监控规则，确保监控平台读取 `condition.reason` 而不是仅依赖 `condition.message`，以利用新的更精确的错误信息。  

- **文档同步**：在 Operator 的使用手册中补充说明：`Succeeded` 条件现在会提供子阶段的具体错误原因，用户可直接通过此字段定位问题。  

- **升级指南**：若用户已经在使用自行实现的 Condition 解析代码（例如基于 `LastTransitionTime` 判断是否是最新状态），建议改为使用 `ObservedGeneration` 来做一致性校验，以兼容本次改动。  

总体来看，此次改动提升了 Operator 对错误根因的可观测性，降低了条件维护的错误概率，对系统整体稳定性与可运维性有正面作用，风险可通过上述建议的测试与文档更新有效管控。

### feat(mocker): add Mooncake delta replay (#9653)
**SHA**: `fa4916d` | 🔗 [查看提交](https://github.com/ai-dynamo/dynamo/commit/fa4916d7e8a5b192f7d6db71b934bec0330812be)

**🎯 变更类型**：功能增强  

**⚡ 重要程度**：🔴高  

**📋 变更摘要**  
本次提交为 Dynamo 的 **mocker** 子模块引入了全新的 *Mooncake‑Delta* Trace 格式，支持在离线重放时将会话内的增量输入（delta）累计为完整 Prompt。为此在 Python 绑定、Trace 解析、WorkloadDriver 以及离线/并发 replay 入口等多处实现了对应的解析、驱动模式切换以及哈希计算逻辑，并补充了单元测试验证累计行为。

**🎯 影响范围**  
- `lib/bindings/python`（CLI 参数、TraceFormat 解析）  
- `lib/mocker/src/loadgen`（`TraceFileFormat`、`Trace`、`WorkloadDriver`）  
- `lib/mocker/src/replay`（离线/并发入口、模拟实现）  
- 相关公共模块：`dynamo_kv_router::protocols`（块哈希计算）  

**🔍 技术洞察**  

- **架构影响**  
  - 新增 `PromptMode`（`Full` / `DeltaCumulative`）与 `TraceFileFormat::MooncakeDelta`，在 `WorkloadDriver` 中根据模式决定是否在每个 turn 结束后把本轮增量累加到会话级 `cumulative_tokens`。  
  - `WorkloadDriver::new_*` 系列函数扩展为 “accumulating deltas” 变体，保持原有 `Full` 路径不变，确保向后兼容。  
  - 入口层（`replay/entrypoints.rs`）通过 `trace_accumulates_session_deltas` 辅助函数分流，确保只有离线（trace）模式下使用新路径；已显式 **禁止** 在分布式 (disagg) 与在线 (live) replay 中使用，以免出现不一致或 OOM。  
  - 重新计算哈希的路径抽象为 `replay_hashes_from_tokens`，依赖 `compute_block_hash_for_seq` 与 `compute_seq_hash_for_block`，保持与原始 Mooncake 计算方式一致。  

- **性能影响**  
  - **内存**：累计增量后每个会话的 Prompt 可能会快速膨胀（尤其在长对话或大块大小下），导致显著的内存占用增长；在 `parser.add_argument` 中已提示 “can use substantially more memory”。  
  - **CPU**：每次 `pop_ready` 需要重新哈希整个累计 Prompt（`replay_hashes_from_tokens`），相较于直接使用预计算的 `turn.replay_hashes` 多了一次块哈希与序列哈希的计算，单轮开销 O(N)（N 为累计 token 数），对多轮长会话会有放大效应。  
  - **吞吐**：在单线程 VLLM 场景仍保持原有实现路径，只是多了一层累计与哈希；在多线程/多 worker 场景，由于每个 worker 仍然基于同一个 `WorkloadDriver`，累计逻辑不引入额外同步开销。  

- **安全考虑**  
  - 变更未涉及网络、权限或外部输入的安全检查，属于纯内部数据处理。  
  - 唯一需要关注的是 **资源耗尽（DoS）**：如果用户误用 `mooncake-delta` 重放大规模、极深的对话，可能导致进程 OOM 或 CPU 飙升，进而影响集群稳定性。  

**⚠️ 潜在风险**  

| 风险点 | 可能后果 | 触发条件 |
|--------|----------|----------|
| 内存激增 | 进程 OOM、节点异常退出 | 长会话累计大量增量、块大小较大、并发 Workers 多 |
| 哈希计算错误 | 重放结果与实际模型不匹配（错误的 block/sequence hash） | `engine_block_size` 与 `trace.block_size` 不匹配、`compute_block_hash_for_seq` 参数变更 |
| API 不一致 | 调用方使用旧 `TraceFileFormat::Mooncake` 仍得到 Full‑Prompt 行为，误以为已开启累计 | 文档或示例未明确说明两种模式的差异 |
| 代码路径分支遗漏 | 在未来新加入的 `simulate_*_disagg` / `simulate_*_live` 中忘记添加 `trace_accumulates_session_deltas` 检查，导致未受限的累计重放 | 代码维护者新增函数未同步更新分支判断 |
| 单元测试覆盖不足 | 复杂的累计逻辑在极端边界（如空增量、跨会话切换）未被捕获 → 隐蔽 bug | 当前测试仅覆盖两轮累计，缺少异常/极端情形 |

**💡 关注建议**  

1. **文档与使用提醒**  
   - 在官方 README / CLI `--help` 中加入明确的 *memory warning* 与 *use‑case*（仅针对 delta‑compressed trace），帮助用户避免误用。  
   - 在 Python 包的 `dynamo.replay` 示例中演示 `trace_format="mooncake-delta"` 与常规 `mooncake` 的区别。

2. **资源防护**  
   - 在 `WorkloadDriver::new` 中加入可选的 **max_cumulative_tokens** 配置，超过阈值时直接报错，防止单会话占用超大内存。  
   - 或者在 `pop_ready` 前检测 `session.cumulative_tokens.len()`，若超过安全上限，输出警告并回退到 Full‑Prompt 方式（可配置）。

3. **性能监控**  
   - 为累计模式新增计时/内存统计（如 `cumulative_hash_compute_time_us`），在报告中展示，以便用户评估新增开销。  
   - 将 `replay_hashes_from_tokens` 的计算移动到一次性预计算阶段（如在 `new` 时为所有会话预先缓存），如果累计 Prompt 不会再变动（仅在 Full‑Prompt 场景），可以进一步削减 per‑turn 开销。

4. **代码维护**  
   - 将 `trace_accumulates_session_deltas` 与 `PromptMode` 的关联抽象为统一的 `TraceReplayConfig`，在后续添加新格式时只需在该结构体里声明。  
   - 为 `WorkloadDriver::new_*_accumulating_deltas` 添加 `#[cfg(test)]` 单元测试覆盖极端情况（空增量、跨会话切换、异常 block size），提升回归安全性。

5. **安全/健壮性**  
   - 在入口层（如 `simulate_trace_file_disagg_with_router_mode_and_format`）对 `engine_block_size` 与 `trace.block_size` 做一次兼容性校验，避免因不匹配导致错误的 hash 计算。  
   - 考虑在 CLI 参数中加入 `--max-cumulative-tokens` 或 `--memory-limit`，让用户显式限制此模式的资源占用。

> 总体来看，本次功能增强为 Dynamo 的离线 replay 引入了对增量压缩日志的完整恢复能力，扩展了实验与基准的适用场景。但因为累计 Prompt 本质上会把原本压缩的 delta 重新展开，导致 **显著的内存和 CPU 开销**，建议在生产或大规模基准测试中慎用，并结合上述防护措施进行资源管理。
