### feat(mocker): add SGLang engine simulation (#6977)
**SHA**: `930721c` | 🔗 [查看提交](https://github.com/ai-dynamo/dynamo/commit/930721c8c93a6473447b846d6cc5833caf4301f6)

**🎯 变更类型**：功能增强 / 架构变更 / 重构  
**⚡ 重要程度**：🔴 高  

**📋 变更摘要**  
本次提交在 *mocker* 子库中加入对 **SGLang** 引擎的完整模拟。核心改动包括：  
1. 新增 `EngineType`（`Vllm` / `Sglang`）以及对应的 CLI 参数与 JSON 配置解析。  
2. 将原来的 `MockVllmEngine` 重新命名为通用的 `MockEngine`，并通过 **factory** `create_engine` 根据 `engine_type` 动态实例化调度器。  
3. 引入 **SchedulerHandle** 抽象，使调度器实现（vLLM、SGLang）统一接口，去除 `receive` 的 `async` 依赖。  
4. 实现 **RadixCache**（基于 `slotmap`）以及 **SglangKvManager**，提供页面化 KV‑cache、前缀匹配、LRU‑eviction、块存储/移除事件。  
5. 添加 SGLang 调度器 `SglangScheduler`，实现自适应 admission‑control、chunked‑prefill、LPM 排序等 SGLang‑style 行为。  
6. 扩展 `MockEngineArgs` 以承载 SGLang 专用配置，并在 Python CLI (`components/src/dynamo/mocker/args.py`) 中暴露相应参数。  
7. 其它配套更改：Cargo 依赖加入 `slotmap`、日志追踪开关、公共测试工具、bench 代码适配等。  

**🎯 影响范围**  
- `lib/mocker`：所有调度器实现、KV‑cache 管理、engine factory、公共协议、bench、bindings。  
- `components/src/dynamo/mocker/args.py`：CLI 参数解析。  
- `lib/bench/kv_router/*`、`lib/llm/src/mocker.rs`、`lib/llm/src/mock_engine`：使用 `SchedulerHandle` 接口的调用点。  
- `Cargo.lock`、`Cargo.toml`：新增 `slotmap` 依赖。  
- Python / Rust 绑定：因为 `Cargo.lock` 变化，重新锁定构建。

### feat: enable kvbm v1 to use the v2 ::new_with_device (#7151)
**SHA**: `5b6624b` | 🔗 [查看提交](https://github.com/ai-dynamo/dynamo/commit/5b6624bda69a9dc5224ee9c10f540d00892b33aa)

**🎯 变更类型**：功能增强 / 重构 / 架构变更  
**⚡ 重要程度**：🔴高  
**📋 变更摘要**：  
- 将 KV‑Cache Block Manager (KVBM) v1 的内存分配路径迁移到 KVBM v2 的统一实现 `dynamo_memory::PinnedStorage::new_for_device`，并加入对 NUMA‑aware 分配的完整支持。  
- 引入写合并（Write‑Combined）Pinned 内存的自动探测与回退逻辑，统一通过 `malloc_host_prefer_writecombined` 实现。  
- 为 `dynamo_memory` 添加 NUMA 节点缓存、命令行校验工具 `validate_numa_placement`，并在 `CODEOWNERS`、Cargo 依赖顺序、测试代码等处同步更新。  

**🎯 影响范围**：  
- `lib/llm`（块管理器、存储实现、Allocator）  
- `lib/memory`（NUMA 模块、PinnedStorage 实现、二进制工具）  
- 代码所有依赖 `dynamo_memory`、`dynamo_runtime`、`dynamo_tokens` 的子模块  
- CI / 文档（CODEOWNERS、示例测试）  

**🔍 技术洞察**  

- **架构影响**  
  - **统一内存抽象**：原先 KVBM v1 通过自实现的 `PinnedStorage`（含手写 NUMA/写合并路径）分配内存；现在统一使用 `dynamo_memory::PinnedStorage`，实现层级从 block_manager → memory → cuda 抽离，降低耦合度。  
  - **可选 NUMA 开关**：`DYN_MEMORY_DISABLE_NUMA` 成为全局“kill switch”，`DYN_KVBM_ENABLE_NUMA` 被标记为已废弃并转为日志提醒，提升配置一致性。  
  - **缓存策略**：`NUMA_NODE_CACHE` 将 GPU PCI → NUMA 节点的映射结果缓存至进程生命周期，避免在高频分配路径中频繁读取 sysfs 或调用 `nvidia-smi`，对整体架构的查询层面产生正向影响。  
  - **API 演进**：新增 `PinnedStorage::new_for_device`（推荐）以及对应的 `From<dynamo_memory::StorageError>` 实现，旧 `PinnedStorage::new` 已标记 **deprecated**，为后续完全移除埋下伏笔。  

- **性能影响**  
  - **写合并内存**：首次分配时会探测硬件是否支持 Write‑Combined，支持时可显著提升 PCIe DMA 带宽（约 10‑30% 提升），不支持时自动回退至 `CU_MEMHOSTALLOC_DEVICEMAP`，对性能不产生倒退。  
  - **NUMA‑aware 分配**：在多‑socket GPU 系统上，分配会尝试在对应 GPU 所在的 NUMA 节点上分配页面，减少跨‑NUMA 拷贝延迟，特别是大块 KV‑Cache（数十至上百 MB）时收益明显。  
  - **缓存查询**：`NUMA_NODE_CACHE` 将一次性读取成本从 O(N)（每次 sysfs）降至 O(1)，在高并发推理场景中可降低 CPU 侧开销，间接提升吞吐。  

- **安全考虑**  
  - 变更未引入新的外部网络或文件写入，仅在本地进程内查询 `/sys` 与 `nvidia-smi`。  
  - 环境变量解析均使用 `dynamo_config::env_is_truthy`，避免因空字符串或非布尔值导致意外开启/关闭。  
  - `malloc_host_prefer_writecombined` 仍保持 `unsafe` 边界，仅在已绑定 CUDA context 的线程中调用，未改变原有安全模型。  

**⚠️ 潜在风险**  

1. **向后兼容**  
   - 老代码仍可调用被标记 **deprecated** 的 `PinnedStorage::new`，但若在不绑定 CUDA context 的线程上使用，仍会触发旧的 `malloc_host_prefer_writecombined`（已删除）导致编译错误。  
   - `DYN_KVBM_ENABLE_NUMA` 环境变量仍被读取并产生一次性警告，若用户仍依赖该变量进行精细控制，可能出现误解。  

2. **硬件探测失败**  
   - 对于旧版 CUDA 或不完整的驱动，`malloc_host` 的写合并探测可能返回错误，导致回退至 `DEVICEMAP`。虽然功能不受影响，但日志会频繁出现 “write‑combined not supported”，可能干扰监控。  

3. **线程绑定要求**  
   - 新的 `PinnedStorage::new_for_device` 仍要求分配前 *绑定* 正确的 CUDA context（`ctx.bind_to_thread()` 在内部完成），但若在多线程环境下误用同一 `CudaContext` 对象而未显式绑定，可能出现 “CUDA context not bound” 错误。  

4. **NUMA 缓存一致性**  
   - `NUMA_NODE_CACHE` 在进程生命周期内保持不变；在容器/虚拟化环境中若 GPU NUMA 拓扑在运行时被重配（极少见），缓存可能变得不准确。  

5. **二进制工具依赖**  
   - `validate_numa_placement` 通过 `move_pages(2)` 需要 `CAP_SYS_NICE`（或 root）权限，在普通用户环境下运行会报错，需要在文档中明确说明。  

**💡 关注建议**  

- **迁移计划**  
  - 在项目的下一个次要版本中，彻底移除 `PinnedStorage::new` 与 `is_numa_enabled`，并更新所有内部调用为 `new_for_device` + `dynamo_memory::numa::is_numa_enabled`。  
  - 在 CI 中加入针对 **旧 API 已废弃** 的编译警告检测，防止新代码误用。  

- **测试覆盖**  
  - 在 CI/CD 中加入 `validate_numa_placement` 的自动化跑分（仅在拥有 NUMA 环境的机器上），确保 NUMA‑aware 分配在主流平台（x86‑64, ARM, 多‑GPU）表现一致。  
  - 加入针对 “写合并不支持” 场景的单元测试，模拟 `malloc_host` 错误返回，验证回退路径的正确性。  

- **文档与运维**  
  - 更新 README / README_CN，明确说明：  
    1. 默认开启 NUMA，若需关闭请设 `DYN_MEMORY_DISABLE_NUMA=1`。  
    2. 写合并可通过 `DYN_KVBM_DISABLE_WRITE_COMBINED=1` 强制关闭。  
    3. `validate_numa_placement` 需要 root 权限或 `CAP_SYS_NICE`。  
  - 在 `CODEOWNERS` 中加入 `kvbm-v2` 团队的所有者，以便后续维护。  

- **监控 & 诊断**  
  - 在运行时加入 metric（如 `dynamo_memory_numa_allocation_success_total`、`write_combined_fallback_total`）供 Prometheus 监控，快速定位在特定机器上是否未能使用预期的优化路径。  

- **安全审计**  
  - 确认所有 `unsafe` 区块仅围绕 CUDA 分配/释放，且在分配失败时会立即 `tracing::error!`，不泄露指针。  
  - 对 `move_pages` 调用的错误处理保持警告级别，防止因权限不足导致进程 panic。  

**结论**：  
此轮改动实现了 **内存分配层的统一、可配置化与性能提升**，对 KV‑Cache 大幅度提升了跨‑NUMA 与 PCIe 传

### 🔍 技术洞察

#### 架构影响
- **事件流并行化**：将 `event_rx` 的容量提升 8 倍，配合 `apply_event_with_prune_tracking` 的单独函数抽象，使得 **KV Router** 能在高吞吐场景下保持生产者不被阻塞，同时在 `dump` 请求时强制冲刷 pending 事件，保证一致性。  
- **恢复调度中心化**：原来的 gap‑detect 与恢复逻辑散布在 `subscriber.rs` 与 `worker_query.rs`，现在统一本地化到 `WorkerQueryClient`，所有恢复（首次发现、实时 gap、清除屏障）走同一路径。  
- **状态模型**：`WorkerState`（全局 epoch） + `RankState`（cursor、high‑water、in‑flight 标记）提供 **一致性保证**：  
  - `epoch` 用于区分同一 worker 在 **clear** 之后的所有后续恢复任务，防止旧任务写入已失效的状态。  
  - `RankCursor::InvalidatedByBarrier` 捕获 `Cleared` 事件产生的全局失效，确保后续恢复仅从 barrier 之后的事件继续。

#### 性能影响
- **正向吞吐**：更大的通道 + 批量冲刷（在 `dump` 前 `while let Ok(event) = event_rx.try_recv()`) 减少了生产者因 `apply_event` 迟缓导致的 back‑pressure，尤其在 Prune 触发时仍可继续接收事件。  
- **恢复并发**：`RECOVERY_CONCURRENCY_LIMIT = 16` + `Semaphore` 控制恢复任务数量，避免在大规模 worker/DP 环境中出现 **膨胀的并发恢复请求**（网络、CPU、内存压力）。  
- **锁粒度**：`WorkerState` 采用 `Arc<Mutex<>>`，只在调度/完成恢复时短暂持锁；`RankState` 内部字段为普通值，争用预计极低。  
- **额外开销**：  
  - 每个 worker‑dp_rank 维护 `RankState`（约 32‑64 B），在数千 workers、每个 8 dp_ranks 的规模下仍在 MB 级别。

- **回收触发**：原来的剪枝触发逻辑仍保留，仅在 `apply_event_with_prune_tracking` 中检查 `current_size > max_tree_size`，未因并发改动产生额外 CPU 负担。  

#### 安全考虑
- **无新增外部攻击面**：所有新增代码均在内部进程间通过 `dynamo_runtime` 的安全通道实现，没有暴露新的网络入口。  
- **竞争风险**：引入更多并发状态（`Mutex`、`Arc`、`Semaphore`）可能产生 **死锁/竞争**，尤其在 `shutdown` 与 `epoch` 增长的交叉路径上。已通过单元测试验证，但在生产环境仍建议加入 **运行时监控（metrics）** 捕捉异常恢复延迟或锁等待时间。  
- **错误传播**：恢复错误被捕获并记录为 `warn`，不会导致索引器崩溃；但对业务层来说，长时间 “恢复失败 → 事件缺失” 可能导致 KV 查询不一致，需监控 `RECOVERY_MAX_RETRIES` 超限次数。
