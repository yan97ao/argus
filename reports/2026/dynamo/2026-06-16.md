### feat(kv-router): add standalone selector peer sync (#10745)
**SHA**: `d1a3029` | 🔗 [查看提交](https://github.com/ai-dynamo/dynamo/commit/d1a30293648dbecca7abb296346882573adbfce3)

**🎯 变更类型**：功能增强 / 架构变更 / 性能优化  
**⚡ 重要程度**：🔴高  

**📋 变更摘要**  
1. 为 Dynamo KV‑Router 引入 **Standalone Selection Service**（独立的 worker 选路/预留服务），实现 “路由‑选路‑预留” 完全脱离模型推理进程。  
2. 为选路服务、Slot‑Tracker 与 Indexer 增加 **副本同步（Replica‑Sync）** 能力：通过 ZMQ PUB/SUB 传播 admission、prefill‑complete、free 事件，实现多实例选路服务的状态一致。  
3. 重构 kv‑router 调度层：把 cache‑hit 估算、tier‑overlap 计算抽离到独立模块 `scheduling::overlap`，并在调度时使用 **effective_prefill_tokens** 统一的预填充负载计算。  
4. 对外 API 增加 `/select_and_reserve`、`/reservations`、`/replica_sync/*` 等端点；文档、CLI 参数（`--replica-sync-port`、`--replica-sync-peers`、`--indexer-peers`）同步更新。  
5. 移除 `cached_tokens` 与 `effective_overlap_blocks` 的公开字段，改为内部使用并在响应中仅保留 Mooncake‑style overlap 汇总。

## 🔍 技术洞察

### 1. 架构影响
| 维度 | 影响描述 |
|------|----------|
| **服务拆分** | 选路、索引、Slot‑Tracker 现在可以独立部署，形成 “Router‑Selector‑Indexer‑Slot‑Tracker” 四层可水平扩展的微服务结构。 |
| **副本同步** | 新的 `ReplicaSyncConfig`、`ScopedReplicaSync`、`PeerManager` 通过本地 ZMQ PUB（`tcp://*:port`）+ SUB 以及 HTTP 管理接口实现跨实例事件转发。同步仅包括 **Admission / Prefill‑Complete / Free**，**Output‑Block** 被刻意排除以避免带宽爆炸。 |
| **启动恢复** | 选路实例在启动时可从 **另一个 selector** 或 **standalone‑indexer** 通过 `/dump` HTTP 拉取 KV‑index 快照，实现“热启动+状态恢复”。 |
| **调度层抽象** | `scheduling::overlap` 将 cache‑hit 估算、tier‑overlap、Mooncake 汇总统一为 `CacheHitEstimates` 与 `MooncakeOverlapSummary`，并在 `SelectionCore::schedule_selection` 中直接构造返回体。 |
| **统一入口** | `SelectionServiceConfig` 增加 `indexer_peers`、`replica_sync_port`、`replica_sync_peers`，`run_server` 自动创建 `ReplicaSyncConfig`、PeerManager 并注入 `SelectionCore`。 |
| **兼容性** | 选路 API 保持向后兼容：`/select` 仍然是查询，仅返回 overlap 信息；`/select_and_reserve` 继续完成预留。原有 `SlotTracker` 与 `Indexer` 代码路径仍可编译（特性门控）。 |

### 2. 性能影响
| 维度 | 正面效应 | 潜在负面 |
|------|----------|----------|
| **副本同步** | 采用 **ZMQ PUB/SUB**（单向广播）和 **无阻塞 channel**（`REPLICA_EVENT_CHANNEL_CAPACITY=100 000`），在大多数场景下延迟在毫秒级，且不影响主请求路径的 I/O。 | 在极端高频的 Admission/Free 事件（如 10k req/s）仍有可能触发 channel 满溢，导致事件被丢弃；同步是 *best‑effort*（不保证排序、重放），短暂的负载不一致会导致选路略有偏差。 |
| **输出块不同步** | 移除输出块的跨副本广播，显著降低网络流量（原实现在每个 decode block 都广播），提升整体吞吐。 | 对需要跨副本精确 decode‑block 计数的外部监控/调度器失去实时视图。 |
| **effective_prefill_tokens** | 统一了 “已缓存 token” 与 “ISL token” 的计算，避免了冗余 `cached_tokens` 字段的序列化开销；调度时只进行一次 `effective_prefill_tokens` 计算。 | 若业务侧依赖旧版 `cached_tokens`/`effective_overlap_blocks` 字段（文档已删），会导致兼容性错误。 |
| **启动恢复** | 通过 HTTP `/dump` 拉取快照，避免了在 ZMQ 上做大批量数据重放；对启动时间有正向帮助。 | 恢复过程只等待 **提交** 事件，不等待 **完整处理**，因此在恢复期间仍可能出现短暂的 miss‑KV‑state（选路误判）。 |
| **CLI 参数验证** | 新增 `parse_nonzero_port`、`requires = "replica_sync_port"` 等 clap 验证，防止配置错误导致运行时 panic。 | CLI 参数增多，部署脚本和 CI 必须同步更新，否则可能出现 “missing required argument” 错误。 |

### refactor(routing): unify prefill admission lifecycle (#10740)
**SHA**: `7e28276` | 🔗 [查看提交](https://github.com/ai-dynamo/dynamo/commit/7e28276cb40fa59e40b1803bc8a3edaa52fe18b7)

**🎯 变更类型**：重构 / 架构变更  
**⚡ 重要程度**：🔴高  
**📋 变更摘要**：本次提交统一了 *prefill*（前置填充）路由的生命周期管理，移除原来的 `activated / deactivated` 两个原子布尔位，改为单一的 `PrefillLifecycleState`（Pending‑>Active‑>Unavailable）状态机；并把原先分散在 `execution.rs`、`inner.rs`、`types.rs` 的前置填充调度、查询、错误处理等逻辑搬到 `prefill_router/admission.rs`、`prefill_router/query.rs`。  
同时对 `KvPushRouter`、`PushRouter`、Sticky‑Session 以及 C‑FFI 接口做了大量拆分和简化，新增 **选路‑预占**、**可回滚的 session 绑定**、**TransportFallback** 控制等特性。

## 🔍 技术洞察

### 1. 架构影响
| 影响点 | 原实现 | 新实现 | 价值 |
|--------|--------|--------|------|
| **Prefill 生命周期** | 两个独立的 `AtomicBool`（`activated`、`deactivated`），状态判断散落在 `can_serve_requests`、`reactivate`、`deactivate` 中。 | 单一 `AtomicU8` 保存 `PrefillLifecycleState`（`Pending`、`Active`、`Unavailable`），所有状态迁移统一在 `complete_activation`、`deactivate`、`reactivate`。 | ① 消除布尔组合导致的模糊状态（如 `activated && deactivated`）；② 扩展性更好，后续可直接加入新状态；③ 代码可读性提升，所有路径均走 **状态机**。 |

| **可维护性** | - 新增的公共结构和函数有明确文档注释，命名自解释，符合现有代码风格。<br>- `PrefillQueryOutcome` 与 `PrefillResolveDecision` 增加 `permit` 字段，保持向后兼容（旧调用仍会得到 `None`）。<br>- 单元测试覆盖了 **peek** 与 **select_and_reserve** 两个关键路径，提升回归安全性。 |

---

### ⚠️ 潜在风险  

1. **Permit 泄漏**：如果在某些异常路径（如 `await` 前的 `panic!`）未显式或隐式释放 `OccupancyPermit`，计数会保持增高，导致 **假性负载**、调度倾斜甚至阻塞。  
2. **误用 `direct()`**：用户若在不需要预占的场景（如旧 `RoundRobin`）自行调用 `direct()` 而不通过 `track_dispatch`，仍会出现 **计数不匹配**。  
3. **并发竞争**：`select_and_reserve` 对 `LeastLoaded` 使用 `select_exact_min_and_increment`（内部加锁），在极高并发下可能出现锁竞争，虽不会导致错误但可能略微增加延迟。  
4. **向后兼容**：C 绑定和外部 gateway 现在必须忽略 `permit`（已改为 `_`），若有人在自定义实现中误用此字段，可能导致编译错误或运行时异常。

### 💡 关注建议  

1. **确保 Permit 正常释放**  
   - 在所有分支（成功、错误、超时）使用 `drop(permit)` 或让其作用域自然结束，防止计数残留。  
   - 可在 `spawn_prefill_task` 入口处加上 `defer!` 类似的宏或 `let _ = permit;` 明确说明生命周期。  

2. **监控负载计数**  
   - 在生产环境开启 `RoutingOccupancyState` 的监控指标（如 `load_per_instance`），观察是否出现异常上升，快速定位潜在泄漏。  

3. **审计直接 `direct()` 调用**  
   - 对外提供的 SDK / 文档明确说明：**在 LeastLoaded / P2C / DeviceAwareWeighted 模式下，请使用 `track_dispatch` 或 `select_and_reserve`**，避免直接 `direct()` 导致计数缺失。  

4. **回归测试**  
   - 将新增的单元测试（peek、select_and_reserve、permit 生命周期）加入 CI，保证未来改动不破坏原子预占语义。  
   - 对 **KV**、**RoundRobin**、**Random** 模式保持原有行为不变，添加对应的 `peek_next_worker == None` 断言，以防误改。  

5. **文档同步**  
   - 更新 `README`、API 文档以及语言绑定（C、Python 等）说明新字段 `permit` 的含义与使用场景。

| 维度 | 影响 | 说明 |
|------|------|------|
| **架构** | ★★★★★ | - 引入了 **engine‑update** 表面，和原有 **engine‑control** 分离，避免控制路由被 LoRA 生命周期膨胀。<br>- `on_endpoint_ready` handoff 让引擎在启动阶段即可获取 `Endpoint`，为动态发现（ModelDeploymentCard）提供统一入口。<br>- LoRA 采用 **stripe‑lock**（固定 32）实现名称级别序列化，避免因 per‑adapter 动态锁导致内存泄漏。<br>- 统一后端的路由命名从 `/engine/<name>` 改为 `/engine/control/<name>`，并在 `system_status_server` 中统一注册，保持向后兼容的 `/v1/loras` 别名。 |
