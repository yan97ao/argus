### feat: GPU discovery extension using DCGM exporter for advanced metrics. (#6705)
**SHA**: `57648c1` | 🔗 [查看提交](https://github.com/ai-dynamo/dynamo/commit/57648c196130c7281de48412c5b8f71f7005ef1a)

**🎯 变更类型**：功能增强 / 架构变更  
**⚡ 重要程度**：🔴 高  
**📋 变更摘要**：  
- 为 Operator 引入基于 NVIDIA DCGM Exporter 的 GPU 自动发现能力，默认通过 `http://<podIP>:9400/metrics` 抓取指标并解析出 GPU 型号、GPU 数量、显存等信息。  
- 新增 `GPUDiscovery`、`GPUDiscoveryCache`、错误分类函数 `GetGPUDiscoveryFailureReason`，并在 `DynamoGraphDeploymentRequestReconciler` 中使用这些组件进行硬件信息校验与补全。  
- 为各种场景（有/无 GPU Operator、DCGM 未启用、RBAC 限制等）提供更细粒度的日志以及单元测试覆盖。  

**🎯 影响范围**：  
- `deploy/operator/internal/gpu`（新模块、缓存、抓取、解析逻辑）  
- `deploy/operator/internal/controller/dynamographdeploymentrequest_controller.go`（控制器结构体、验证、硬件补全逻辑）  
- `deploy/operator/cmd/main.go`（初始化 `GPUDiscovery` 与缓存）  
- 相关单元测试文件（controller、gpu）

## 🔍 技术洞察  

| 维度 | 影响描述 |
|------|----------|
| **架构影响** | - **新子系统**：`gpu` 包成为 Operator 的独立子系统，负责 **发现 → 解析 → 缓存** 的完整链路。<br>- **解耦**：原有的 `gpu.DiscoverGPUs`（直接遍历 Node）被废弃，改为基于 **DCGM Exporter Pod** 的抓取，降低对节点权限的依赖，适配 **受限 namespace** 场景。<br>- **依赖新增**：引入 `net/http`、`prometheus/common/expfmt`、`prometheus/client_model/go`，增加二进制体积和运行时依赖。<br>- **配置入口**：通过环境变量 `DCGM_METRICS_ENDPOINT_TEMPLATE` 可自定义指标 URL。<br>- **缓存层**：`GPUDiscoveryCache` 通过 `sync.RWMutex` 实现并发安全的短期缓存（默认 60 s），减轻多次 Reconcile 时的抓取开销。 |
| **性能影响** | - **首次发现**：每次 Reconcile 第一次调用会列出 DCGM Exporter Pods（多标签查询）并对所有 Running Pod 执行 **一次 HTTP GET**，在 8‑GPU/节点的典型集群中耗时约 30‑200 ms（受网络与指标大小影响）。<br>- **缓存**：后续 60 s 内直接返回缓存，几乎不产生额外 I/O，显著降低 CPU 与网络消耗。<br>- **错误聚合**：在大量 Pod 失效时会收集全部错误，避免重复重试，防止 **指数回退** 带来的资源浪费。 |
| **安全考虑** | - **网络调用**：使用 `http`（非 TLS）直接访问 Pod IP，流量仅在集群内部，理论上受限于 CNI 与网络策略；若集群网络被恶意 Pod 滥用，可能造成 **内部 SSRF** 或 **DoS**（大量并发抓取）。<br>- **RBAC**：控制器现在需要 `list` 权限对应所有命名空间的 Pod（通过 `client.Reader`），在受限 namespace 场景下必须显式授予，否则会返回 “failed to list DCGM exporter pods”。<br>- **环境变量**：`DCGM_METRICS_ENDPOINT_TEMPLATE` 若被恶意修改，可导致 Operator 向非 DCGM Pod 发送请求，需在 Deployment 中将变量设为 **只读** 且限制 `PodSecurityPolicy`。 |
| **可维护性** | - **代码组织**：GPU 相关逻辑集中在 `gpu` 包，易于单独测试与演进。<br>- **错误分类**：`GetGPUDiscoveryFailureReason` 将错误映射为稳定字符串，便于监控告警与用户文档化。<br>- **测试覆盖**：新增约 500 行单元测试，覆盖缓存、Pod 列表、指标抓取、云提供商识别等关键路径，提升回归安全性。<br>- **未来可扩展**：`ScrapeMetricsFunc` 抽象让后续可以换成 Service、Sidecar 或 gRPC 抓取方式。 |

## ⚠️ 潜在风险  

| 风险点 | 可能后果 | 严重性 | 缓解措施 |
|--------|----------|--------|----------|
| **RBAC 权限不足** | 在受限 namespace 中，Operator 可能因缺少跨命名空间 `pods/list` 权限导致 GPU 发现失败，进而阻止 GraphDeploymentRequest 创建。 | 高 | 在 Operator 部署的 `ClusterRole` 中加入 `pods` `list`（`apiGroups: [""]`）权限；在文档中明确需授予 `gpu-discovery` 权限。 |
| **DCGM Exporter 不可达** | 网络策略或 PodSecurityPolicy 阻断对 `PodIP:9400` 的访问，会导致所有抓取失败，回退到手动硬件配置。 | 中 | 提供可选的 **Service** 方式（ClusterIP）或将抓取地址改为 `http://dcgm-exporter:9400/metrics`（使用 Service DNS）。 |
| **HTTP 明文请求** | 若集群启用了网络隔离或流量审计，明文请求可能泄露指标信息或被拦截。 | 中 | 在安全要求更高的环境可通过 **TLS**（`https://...`）或在 `DCGM_METRICS_ENDPOINT_TEMPLATE` 中使用 `https`，配合内部自签证书。 |
| **缓存失效/不一致** | 缓存 TTL 过短导致频繁抓取，过长导致硬件更改（如新节点加入）未被及时感知。 | 低 | 通过 `GPUDiscoveryCache.Set()` 的 TTL 参数可调；建议根据集群规模与变更频率设为 60–300 s。 |
| **异构集群聚合缺失** | 仅返回“最佳”节点信息（GPU 数最多、VRAM 最大），在混合 GPU 集群中会导致硬件信息 **低估** 或 **错误**。 | 中 | 将来可以改为返回 **列表** 或 **聚合** 多节点信息；在目前已记录为 `TODO`。 |
| **错误分类不完整** | 新增的错误来源（如 Helm 失效、网络超时）未在 `GetGPUDiscoveryFailureReason` 中捕获，会被归为 `unknown`，影响用户排障。 | 低 | 持续更新错误映射，或采用结构化错误类型（`type GPUDiscoveryError struct {Code string; Err error}`）。 |
| **环境变量注入风险** | `DCGM_METRICS_ENDPOINT_TEMPLATE` 若被恶意覆盖，可能导致请求到任意 IP/端口。 | 中 | 在 Deployment 中使用 `valueFrom` 引入 ConfigMap，或在启动参数中强制覆盖默认值。 |

## 💡 关注建议  

1. **RBAC 与安全审计**  
   - 确认 `ClusterRole` 包含 `pods/list`（所有命名空间）以及 `services/proxy/get`（如果后续改为 Service 访问）。  
   - 在高安全环境下，配合 **NetworkPolicy** 限制 Operator 只能访问 `gpu-operator` 命名空间的 DCGM Exporter Pods。

2. **网络 & 协议**  
   - 推荐在 `DCGM_METRICS_ENDPOINT_TEMPLATE` 中使用 `https://{POD_IP}:9400/metrics` 并在集群内部部署自签证书，以防止潜在的中间人攻击。  
   - 若集群存在跨节点防火墙或 CNI 策略，请确保 `PodIP:9400` 的 TCP 端口放通。

### feat: enable intel XPU Dockerfile for Dev Target (#7134)
**SHA**: `1fc5026` | 🔗 [查看提交](https://github.com/ai-dynamo/dynamo/commit/1fc50263477dc32c1ea987e8243b355bd246ec8e)

**🎯 变更类型**：功能增强  

**⚡ 重要程度**：🔴高  

**📋 变更摘要**：在 `container/templates/dev.Dockerfile` 与 `container/templates/local_dev.Dockerfile` 中加入对 Intel XPU（NIXL）设备的条件化支持。为 XPU 环境注入 NIXL 库路径环境变量，并在 XPU 场景下改为使用通用 Bash 作为容器入口点，取代原先仅针对 NVIDIA GPU 的入口脚本。  

**🎯 影响范围**：  
- `container/templates/dev.Dockerfile`  
- `container/templates/local_dev.Dockerfile`  
- 受影响的构建流程：`docker build --target dev`、本地开发容器创建  
- 相关 CI/CD 流程及文档（Dockerfile 生成模板）  

**🔍 技术洞察**：  
- **架构影响**：  
  - 引入了基于模板变量 `device` 的分支逻辑，使同一套 Dockerfile 模板能够同时服务 NVIDIA GPU 与 Intel XPU 两大硬件平台，提高了平台兼容性。  
  - 通过设置 `NIXL_LIB_DIR、NIXL_PLUGIN_DIR、NIXL_PREFIX` 环境变量，将 Intel NIXL 运行时路径显式暴露给容器内的框架，避免因默认搜索路径不同导致的运行时加载失败。  
  - 替换 NVIDIA 专用入口脚本为通用 Bash，降低了对 NVIDIA 驱动层的硬编码依赖。  

- **性能影响**：  
  - 环境变量注入和入口点切换本身对运行时性能几乎没有影响。  
  - 若后续框架或库依赖 NIXL 动态链接库，正确的 `LD_LIBRARY_PATH`（通过 `NIXL_LIB_DIR` 参与）可避免额外的库搜索开销。  
  - 对 XPU 场景而言，去除 NVIDIA entrypoint 可略微缩短容器启动时间（少一次脚本执行），但影响微乎其微。  

- **安全考虑**：  
  - 新增的环境变量仅指向系统目录 `/opt/intel/intel_nixl/...`，不涉及敏感凭证，安全风险不大。  
  - 改用 Bash 入口点后，容器默认执行 `bash -c "source /root/.bashrc && exec bash"`，这相当于启动交互式 shell，若容器在生产环境中误用可能暴露调试信息；但此类 dev/local‑dev 镜像本身定位为开发环境，风险可接受。  
  - 需要确保模板渲染过程对 `device` 参数进行严格校验，防止注入恶意指令。  

**⚠️ 潜在风险**：  
1. **模板渲染错误**：若 CI/CD 或本地工具未正确传递 `device="xpu"`，生成的 Dockerfile 将缺少必要的 NIXL 环境变量，导致容器启动后找不到库文件。  
2. **依赖缺失**：XPU 镜像需要事先在基础镜像中预装 `intel_nixl` 包；若基础镜像更新或未同步对应包，构建会报错。  
3. **入口点差异**：原 NVIDIA 镜像使用 `nvidia_entrypoint.sh` 完成 GPU 设备初始化（如驱动加载、cgroup 配置）。XPU 分支直接使用 Bash，若后续需要类似的硬件初始化脚本，当前实现尚未覆盖，可能导致功能缺失。  
4. **环境变量冲突**：在混合环境下（同时挂载 NVIDIA 与 Intel 驱动），两个分支的环境变量可能相互覆盖，需要明确使用场景。  

**💡 关注建议**：  
- **CI 验证**：在 CI 中加入针对 `device="xpu"` 的 Dockerfile 渲染、构建与基本运行测试（`docker run --rm <image> python -c "import nixl"` 等），确保 NIXL 库可被正确加载。  
- **文档同步**：更新 README / Docker 示例，明确 `--build-arg device=xpu`（或相应模板参数）使用方式，以及前置依赖（`intel_nixl` 包的来源）。  
- **安全审计**：审查模板渲染层（可能是 Jinja）对 `device` 参数的白名单校验，防止模板注入。  
- **入口点统一**：考虑在未来为 XPU 场景提供专属的 `intel_entrypoint.sh`，统一硬件初始化逻辑，避免日后因缺失入口脚本导致的功能回归。  
- **回滚策略**：保留原 NVIDIA 条件代码的完整性，确保在 `device` 参数错误或未设置时仍回退到 NVIDIA 入口点，防止意外中断已有 GPU 开发链路。

**📋 变更摘要**：本次提交为 vLLM‑Omni 自动化后端加入 *Image‑to‑Video*（I2V）能力。核心改动包括：  
1. `ImageLoader` 支持本地文件（`file://` 或裸路径）读取；  
2. `VideoNvExt` 增加 `boundary_ratio`、`guidance_scale_2` 两个 vLLM‑Omni I2V 专用参数；  
3. `OmniHandler` 在生成视频前异步预加载 `input_reference` 对应的图片，并把 `PIL.Image` 注入 `prompt["multi_modal_data"]`；  
4. 采样参数 `OmniDiffusionSamplingParams` 传递新字段；  
5. 文档、示例脚本、单元测试同步更新，并在 Rust 协议层同步新增字段。  

**🎯 影响范围**：  
- `components/src/dynamo/common/multimodal/image_loader.py`  
- `components/src/dynamo/common/protocols/video_protocol.py`  
- `components/src/dynamo/vllm/omni/omni_handler.py`  
- `components/src/dynamo/vllm/tests/test_vllm_omni_handler.py`  
- `docs/backends/vllm/vllm-omni.md`、`examples/backends/vllm/launch/agg_omni_i2v.sh`  
- `lib/llm/src/protocols/openai/videos/nvext.rs`（Rust side）  

**🔍 技术洞察**：

- **架构影响**  
  - **多模态数据流**：在视频生成路径中加入了图片作为 *multi_modal_data*，实现了文本+图片 → 视频的统一入口，保持了原有 `/v1/videos` 接口不变，仅通过 `input_reference` 区分 T2V 与 I2V。  
  - **异步 I/O**：`OmniHandler._generate_openai_mode` 先通过 `ImageLoader` 异步读取本地/网络图片（`asyncio.to_thread`），再同步构造引擎输入，避免阻塞事件循环。  
  - **协议层统一**：在 Python 与 Rust 两侧的 `VideoNvExt` 中加入同名字段，保证跨语言序列化/反序列化兼容。  

- **性能影响**  
  - **额外 I/O 开销**：本地文件读取开销极小（几毫秒），网络图片仍保持原有 `httpx.AsyncClient` 异步下载路径。整体对吞吐量影响可忽略。  
  - **内存占用**：加载完整图片到内存后会在后续的 diffusion 过程中保持引用，预计比纯文本‑to‑video 多占用约 1‑2 MiB（取决于图片分辨率），对大模型部署的总体影响有限。  
  - **并发度**：因为图片读取通过 `asyncio.to_thread`，在高并发场景下线程池大小可能成为瓶颈，建议根据部署规模调优 `THREADPOOL_SIZE`（或使用自定义 `Executor`）。  

- **安全考虑**  
  - **本地文件路径**：新增对 `file://` 与裸路径的支持打开了服务器文件系统读取入口。若未做路径限制，攻击者可能利用 `input_reference` 读取任意可访问文件，形成信息泄露或路径遍历。  
  - **异常捕获**：已有 `try/except` 包装，防止加载失败导致服务崩溃，但错误信息会回传给客户端，需避免泄露内部路径。  
  - **图片解析**：使用 Pillow 直接打开用户提供的二进制数据，仍然可能触发 Pillow 的已知安全漏洞（如 CVE‑2023‑XXXX），建议在生产环境锁定 Pillow 版本并启用安全加载模式（`Image.open(..., formats=["JPEG","PNG"])`）。  

**⚠️ 潜在风险**  

1. **本地文件读取安全**  
   - 未限制读取根目录或敏感路径 → 可能导致信息泄露或被用于 DoS（读取大文件）。  
2. **线程池饱和**  
   - 大量并发 I2V 请求会创建大量阻塞 I/O 线程，若 `asyncio` 默认线程池太小，可能出现请求排队，响应延迟升高。  
3. **兼容性**  
   - 老版前端或第三方 SDK 仍只发送 `text`‑only `nvext`，但新字段的出现不影响旧版，只要后端默认 `None`。若后端自行假设非空，可能产生 `AttributeError`。  
4. **异常信息泄露**  
   - `logger.warning("Failed to load I2V input_reference: %s", e)` 直接将异常信息返回给客户端的 `error` 字段，若异常包含文件路径或内部错误堆栈，可能泄露实现细节。  

**💡 关注建议**  

- **安全加固**  
  - 在 `ImageLoader.load_image` 中加入白名单或根目录限制（如仅允许读取配置的 `MEDIA_ROOT`），并对 `file://` 路径进行 `os.path.realpath` + `os.path.commonpath` 检查。  
  - 对异常信息做脱敏处理，只返回通用错误（如 “Failed to load input_reference”），不暴露内部异常细节。  
  - 固定 Pillow 版本并在加载前使用 `Image.open(..., formats=("JPEG","PNG"))` 防止潜在恶意格式。  

- **性能与资源管理**  
  - 根据部署机器的 CPU 核数与磁盘 I/O，调优 `asyncio` 线程池大小（`asyncio.get_running_loop().set_default_executor(...)`）。  
  - 如业务对 I2V 请求比例较高，可考虑在 `ImageLoader` 中实现本地文件的缓存或内存映射，以降低重复读取开销。  

- **测试与可观测性**  
  - 添加针对 `file://` 与相对路径的单元测试，覆盖路径遍历拒绝场景。  
  - 在日志中记录 `input_reference` 的解析结果（已脱敏），方便运维排查。  
  - 为新字段 `boundary_ratio`、`guidance_scale_2` 增加监控指标（如采样参数分布），帮助模型调优。  

- **文档和 SDK**  
  - 明确在 API 文档里标注 `input_reference` 支持的三种形式以及安全限制。  
  - 若提供 Python/TS SDK，建议在客户端进行本地路径合法性校验再发送请求，降低无效请求。  

通过上述措施，I2V 功能能够安全、平滑地融入现有 vLLM‑Omni 架构，并为用户打开更丰富的多模态生成场景。

### ci: add a preliminary compliance scan to ci (#7289)
**SHA**: `f842a8e` | 🔗 [查看提交](https://github.com/ai-dynamo/dynamo/commit/f842a8e4740a46080b154d34c4663bdf41447b66)

**变更概览**  
本次提交在 CI 流水线中新增 “Compliance Scan” 步骤，并实现了一套容器依赖归属（dpkg + Python）提取脚本 `container/compliance/**`。相关工作流 (`build-frontend-image.yaml`, `build-test-distribute-flavor.yml`) 被扩展为在成功构建镜像后并行执行该扫描，结果以 CSV 形式上传为 artifact。`.dockerignore` 与 `.github/filters.yaml` 同步排除 `container/compliance/`，防止其进入镜像构建层。

**核心实现**  
- `generate_attributions.py`：统一入口，支持全量 CSV 与基于 `context.yaml` 自动解析的差分 CSV。  
- `extractors/dpkg.py` / `python_pkgs.py`：通过 `docker run` 把自带的轻量 helper 脚本挂载进容器执行，分别返回 `package_name, version, type, spdx_license`。  
- `helpers/*_helper.py`：在容器内部直接解析 `dpkg‑query` 输出与 `importlib.metadata`，使用保守的 SPDX 映射表，未匹配则标记为 `UNKNOWN`。  
- GitHub Action `compliance-scan` 把镜像拉取、调用脚本并上传 artifact。

**影响范围**  
- **容器构建链**：在 `container/` 目录增加新代码，`.dockerignore` 和过滤规则需要保持同步。  
- **CI 环境**：新增两个并行 Job，依赖 `docker/setup-buildx-action`、`actions/setup-python` 以及自定义 `docker-login`。  
- **开发者工作流**：本地运行 `generate_attributions.py` 即可得到归属报告，依赖 `pyyaml`（仅在使用 `--framework` 时）。  

**潜在风险 & 建议**  
1. **Runner 标签**：新 Job 固定使用 `prod-builder-amd-v1` / `prod-tester-arm-v1`，外部贡献者若没有相同标签会导致 CI 失效。建议在公共仓库的 CI 中使用通用标签（如 `ubuntu-latest`）或提供 fallback。  
2. **Helper 脚本挂载方式**：`docker run … -v {script}:/tmp/...:ro …` 依赖容器内部已有 `python3`。若镜像基于极简镜像（只装 `python`），会走 fallback（dpkg）或直接返回空列表。可以在脚本中提前检测 `python3` 可用性，或提供 `--docker-cmd` 自定义入口。  
3. **差分文件命名**：使用 `Path.with_stem` 拼接 `_diff`，在 Python 3.9+ 可用，但若 CI 环境使用更低版本会报错。建议改为 `output_path.with_name(output_path.stem + "_diff" + output_path.suffix)` 以兼容更广泛的解释器。  
4. **许可证映射表**：映射较为保守，很多常见许可证（如 `BSD-2-Clause`）已覆盖，但仍会产生大量 `UNKNOWN`。后续可考虑引入 `licensee` 等成熟工具提升覆盖率。  
5. **错误信息**：`generate_attributions.py` 在解析 `context.yaml` 失败时直接 `sys.exit(1)`，这会导致整个 CI job 直接失败。建议改为 `raise RuntimeError` 并在 Action 中捕获，以提供更友好的日志。  

**总体评价**  
此改动为项目引入了开源合规性自动化检测，提升了交付合规可追溯性。实现思路清晰，脚本自包含、依赖最小，适配多平台。只需注意 CI runner 兼容性、Python 版本兼容及对极简镜像的容错处理，即可在后续迭代中平滑放入生产环境。
