### fix: broken vLLM nightly gpu_2 test and vLLM API compatibility errors (#7865)
**SHA**: `3ad7b7c` | 🔗 [查看提交](https://github.com/ai-dynamo/dynamo/commit/3ad7b7c88ac91f18225824fdc8394f066c9607b6)

**🎯 变更类型**：Bug修复  
**⚡ 重要程度**：🔴高  
**📋 变更摘要**：本次提交修复了 vLLM nightly `gpu_2` 测试失效以及 vLLM API 不兼容导致的运行时错误。主要通过升级示例脚本的 NCCL 端口管理、完善多模态模型数据构造、以及对 `chat_processor` 中的渲染与配置路径进行适配，使测试能够顺利通过并保持向后兼容。  

**🎯 影响范围**：  
- `examples/backends/sglang/launch/*`（所有 disaggregated 启动脚本）  
- `examples/multimodal/launch/*`（音频聚合/拆分脚本）  
- `examples/multimodal/utils/chat_processor.py`、`utils/model.py`（核心多模态请求处理）  
- `tests/serve/*`、`tests/utils/payloads.py`（自动化测试套件）  

**🔍 技术洞察**：  
- **架构影响**  
  - 通过在启动脚本中显式指定 `--nccl-port`（或相应环境变量 `DYN_SYSTEM_PORT`）避免了 NCCL 随机端口的 TOCTOU 竞争，提升了分布式训练/推理的部署可靠性。  
  - `chat_processor` 现在使用 `VllmConfig` 包装 `ModelConfig`，并显式注入 `OpenAIServingRender`，使渲染层与 vLLM 1.6+ 兼容，统一了 OpenAI 接口的渲染路径。  
  - `model.py` 的模型识别改为大小写不敏感并通过关键字匹配（audio、video、vl），让新增 multimodal 模型（如 LLaVA‑NeXT‑Video）可以即插即用。  

- **性能影响**  
  - 脚本层面的改动仅涉及端口分配与进程监控 (`wait_any_exit` 取代 `wait`)，不会对模型推理吞吐产生直接影响，且因减少了端口冲突导致的进程重启，间接提升了启动速度和可用性。  
  - 在 `chat_processor` 中引入 `OpenAIServingRender` 可能带来轻微的渲染开销（主要是对象创建），但该开销在单次请求层面可忽略不计。  

- **安全考虑**  
  - 明确端口分配降低了意外暴露随机端口的风险，提升了网络安全性。  
  - 代码改动未引入新的外部依赖或文件写入路径，未产生明显的安全面暴露。  

**⚠️ 潜在风险**：  
1. **脚本兼容性**：原有用户可能仍依赖旧的随机 NCCL 端口行为，若未在启动命令中提供 `--nccl-port` 或相应 `DYN_SYSTEM_PORT*` 环境变量，仍可能遭遇端口冲突。  
2. **vLLM 版本耦合**：`OpenAIServingRender` 的签名在未来 vLLM 版本可能变更，若用户自行升级 vLLM 而未同步本仓库代码，可能再次出现兼容性错误。  
3. **模型关键字匹配**：`model.py` 通过字符串关键字判断模型类型，若出现模型名称中包含误导性子串（如 “audio” 出现在非音频模型的名字），可能导致错误的数据构造路径。  

**💡 关注建议**：  
- **部署层面**：在所有使用 SGLang/torch.distributed 的脚本中显式传递 `--nccl-port <unique_port>`（或通过 `DYN_SYSTEM_PORT{1,2,…}` 环境变量），并在 CI/CD 中加入端口占用检查。  
- **CI 测试**：确保 `gpu_*` 标记对应的机器资源符合新脚本中对端口/GPU 的显式要求，必要时在 CI 设置 `timeout` 更长的阈值以避免误报。  
- **代码维护**：将 `model` 判断逻辑抽象为统一的 “model type registry”，避免靠字符串匹配判断，以提升可扩展性和防止误判。  
- **版本锁定**：在 `Cargo.toml` 中对 vLLM 的版本范围进行适当约束（例如 `>=1.6,<1.8`），并在发行说明中注明兼容的最低 vLLM 版本。

### fix: remove sanity check from CI; it is only meant for developers to quickly check for correct compilation/inference dependencies (#7892)
**SHA**: `0a4b5d4` | 🔗 [查看提交](https://github.com/ai-dynamo/dynamo/commit/0a4b5d42c60e48712363fef240abec7effcd90f2)

**🎯 变更类型**：Bug修复 / CI流程优化  
**⚡ 重要程度**：🔴高  
**📋 变更摘要**：  
1. 移除了在 CI 中针对 Runtime 镜像的 “sanity check” 步骤，避免在 CI 环境中执行仅面向开发者的快速编译/依赖检查脚本。  
2. 对 `deploy/sanity_check.py` 进行增强：加入对可编辑（editable）安装的检测、兼容 Python 3.11+ 的 `tomllib`、以及对可选依赖缺失的更友好提示，整体容错性提升。  

**🎯 影响范围**：  
- `.github/actions/docker-build/action.yml` 与 `.github/workflows/build-test-distribute-flavor.yml`（CI 工作流）  
- `deploy/sanity_check.py`（本地/运行时镜像的完整性检查工具）  
- 受影响的模块：Docker 镜像构建、CI 自动化、Python 包安装/验证逻辑  

**🔍 技术洞察**：

- **架构影响**  
  - CI 流程中剔除对 Runtime 镜像的额外容器运行检查，简化了流水线依赖链，降低了 CI 中对 Docker 运行时的耦合（不再需要在 CI 环境中启动完整的 runtime 镜像）。  
  - `sanity_check.py` 现在能够识别并报告通过 `pip install -e .`（editable）方式安装的本地源码路径，避免误报缺失的 `ai_dynamo` 包。  

- **性能影响**  
  - CI 运行时间显著下降：原来的 sanity‑check 步骤会拉起容器并执行 Python 脚本，约消耗 30‑60 s（视镜像大小而定），删除后可直接进入单元/集成测试阶段。  
  - 脚本本身的改动增加了少量 IO（读取 `direct_url.json` 与 `pyproject.toml`），对本地运行时几乎没有性能影响。  

- **安全考虑**  
  - **正面**：脚本在检测可编辑路径时加入了对 `pyproject.toml` 的解析，能够在开发者误将本地源码打包进镜像时提前提示，防止潜在的源码泄露。  
  - **负面**：去掉 CI 中的 sanity‑check，意味着如果镜像构建过程出现依赖未正确安装的情况，唯一的防线只剩下后续的测试阶段，若测试覆盖不足，可能导致缺失依赖的镜像被发布。  

**⚠️ 潜在风险**：

1. **发布缺陷风险**：失去 CI 早期的“快速失败”机制，若后续测试未覆盖到 runtime 镜像的完整启动路径，可能出现依赖缺失但仍通过 CI 的情况。  
2. **兼容性回退**：对 `tomllib` 的使用在 Python < 3.11 时设为 `None`，但后续代码仍假设其存在（如 `tomllib.TOMLDecodeError`），若在旧版 Python 环境运行仍可能抛异常。  
3. **脚本行为变更**：之前的 sanity‑check 在失败时直接退出并使 CI 失败，现在该步骤被完全删除，使用者若仍在本地手动执行可能得到不同的错误信息，需要更新文档。  

**💡 关注建议**：

- **CI 侧**：在关键的发布分支（如 `main`、`release/*`）保持一套可选的 runtime sanity‑check（可通过输入变量打开），以防止因依赖缺失导致的镜像失效。  
- **测试覆盖**：确保集成测试中包含对 runtime 镜像的完整启动和导入路径检查，尤其是对 `ai_dynamo`、`ai_dynamo_runtime` 的 import 验证。  
- **文档更新**：在 `README` 或 CI 说明中注明 sanity‑check 已被移除，推荐开发者在本地使用 `python deploy/sanity_check.py` 手动验证。  
- **兼容性**：如果项目仍需支持 Python 3.10 及以下，建议在脚本中对 `tomllib` 为 `None` 的情况提供回退（例如使用 `tomli`）或在 CI 中固定 Python 3.11+。  
- **监控**：在发布流水线后添加一个轻量的镜像启动健康检查（例如 `docker run --rm <image> python -c "import ai_dynamo"`），以弥补移除的 sanity‑check。

### fix(sglang): make prefill warmup failure fatal (#7944)
**SHA**: `eb675ae` | 🔗 [查看提交](https://github.com/ai-dynamo/dynamo/commit/eb675ae38945b9b663b61777cc82460e0f9cc6d0)

**🎯 变更类型**：Bug修复  

**⚡ 重要程度**：🔴高  

**📋 变更摘要**：  
1. 将 prefill 引擎的 warm‑up 过程从“记录警告并继续”改为“失败即抛异常”，确保在 warm‑up 超时或出现异常时 worker 启动被中止。  
2. 在 `init_prefill` 中捕获这些异常并转化为 `RuntimeError`，防止带有损坏引擎的 worker 注册到 Dynamo 集群。  

**🎯 影响范围**：  
- `components/src/dynamo/sglang/init_llm.py`（warm‑up 逻辑及 `init_prefill` 启动流程）  
- 依赖 `init_prefill` 的 Dynamo worker 启动路径  
- 可能影响使用 sglang‑disaggregation 的部署脚本与监控系统  

**🔍 技术洞察**：  
- **架构影响**：  
  - 引入了显式的错误传播机制，使得 worker 启动阶段能够感知底层引擎的不可用状态，避免了“沉默失败”导致的错误节点注册。  
  - 对上层调用者（如集群管理或监控）增加了异常路径，需要确保这些组件能够捕获 `RuntimeError` 并作相应处理（如回滚、报警或自动重试）。  

- **性能影响**：  
  - Warm‑up 本身仍保持原有的 1800 秒超时阈值，未对运行时性能产生直接变化。  
  - 通过提前 abort 启动，可避免后续请求因预填充引擎错误而产生的高延迟或重复重试，从而间接提升整体服务可用性。  

- **安全考虑**：  
  - 此次改动不涉及数据泄露、权限提升或外部攻击面。  
  - 通过阻止未准备好的引擎对外提供服务，间接降低了因异常状态触发的潜在安全风险（如异常日志泄露、异常输入导致的崩溃等）。  

**⚠️ 潜在风险**：  
1. **启动可用性降低**：在网络抖动或临时资源不足导致 warm‑up 超时的情况下，整个 worker 将无法启动，可能导致集群短暂缺失服务实例。  
2. **向后兼容性**：已有部署脚本或监控系统可能未预期 `init_prefill` 抛出 `RuntimeError`，导致未捕获异常而导致进程崩溃。  
3. **错误信息暴露**：`RuntimeError` 中直接拼接异常信息，若异常包含敏感内部细节，日志可能泄露实现细节。  

**💡 关注建议**：  
- **监控与告警**：在集群监控中加入对 `Prefill warmup timed out` 与 `Prefill warmup failed` 的特定告警，便于快速定位启动失败根因。  
- **容错与重试策略**：考虑在外层调度器（如 Dynamo 的 worker manager）实现自动重试或回退机制，在 warm‑up 失败时重新拉起实例，减少单点启动失败对可用性的影响。  
- **配置开关**：为高级用户提供可选的 `prefill_warmup_fatal` 配置项，以在测试环境或特殊场景下保持宽容的启动行为。  
- **异常信息脱敏**：审视捕获的异常 `e`，如有必要对日志进行脱敏处理，避免暴露内部实现或路径信息。  
- **回归测试**：新增单元/集成测试，验证 warm‑up 超时、异常抛出以及异常被正确转化为 `RuntimeError`，确保异常不被意外吞掉。  

通过上述改动，系统在面对底层引擎不可用时能够更快速、明确地反馈错误，提升整体可靠性；同时需要在运维层面做好启动失败的监控与容错准备，以平衡可用性与错误可见性。

  - 该提交仅涉及测试元数据，无业务逻辑改动，不引入新的安全风险。唯一需要关注的是 **GPU 资源标记**（如 `gpu_1`）如果在 CI 环境中未提供相应的硬件，测试将因资源不可用而失败，从而间接暴露 CI 环境配置问题。

### fix: latch busy load until both cool off (#7912)
**SHA**: `ffe2062` | 🔗 [查看提交](https://github.com/ai-dynamo/dynamo/commit/ffe206297d73d1ae7eba5b524b1d6b95f17fa633)

**📋 变更摘要**  
本次提交在 KV 负载监控体系中引入了 **`kv_used_blocks`** 作为工作节点（worker）真实使用的 KV 块数量，并在 `WorkerLoadState` 中加入 **解码忙碌锁存（DecodeBusyLatch）** 机制。通过对 `active_decode_blocks` 与 `kv_used_blocks` 两条信号进行 “都降至安全阈值后才解除锁存” 的策略，解决了原先仅凭 `active_decode_blocks` 可能出现的误判（busy‑latch 过早释放）问题。相应地，所有 Python/Rust 对外发布接口、协议结构体、监控逻辑以及单元测试均同步更新。

- `components/src/dynamo/*/publisher.py`（Python 侧发布器）

- KV 路由协议 `lib/kv-router/src/protocols.rs`

**🔍 技术洞察**  

- **架构影响**  
  - **双信号模型**：原有的 `active_decode_blocks`（调度侧信号）和新增的 `kv_used_blocks`（工作节点真实占用）形成了互补的监控视图。工作节点发布 `kv_used_blocks`，调度器仍可继续发布 `active_decode_blocks`，兼容现有调度逻辑。  
  - **锁存机制**：在 `WorkerLoadState` 中增加 `decode_busy_latches`，通过 `latched_busy`、`kv_used_blocks_cleared`、`active_decode_blocks_cleared` 三个布尔位实现 “只要任意一次超阈值就锁定，只有两条信号均恢复后才清除”。这提升了 busy‑detection 的稳健性，避免因某一信号短暂恢复导致调度器错误地认为节点空闲。  
  - **向后兼容**：新增的 `kv_used_blocks` 为 `Option<u64>`，保持旧有仅有 `active_decode_blocks` 的调用路径仍可工作（已在代码里默认 `None`），但强制 **至少提供一种负载指标**（`publish` 抛出错误），阻止完全空白的发布导致监控盲区。  

- **性能影响**  
  - **额外哈希表**：`kv_used_blocks` 与 `decode_busy_latches` 各自维护 `HashMap<u32, _>`，在高 dp_rank（数十甚至上百）情况下带来 O(1) 的插入/查询开销，整体内存占用在几 KB 量级，影响可忽略。  
  - **锁存判断**：在 `is_busy` 与 `update_decode_busy_latch` 中进行数次 `Option::is_some_and` 与阈值比较，计算量极低，对请求路径的延迟几乎没有可感知影响。  
  - **发布路径**：Python/Rust 发布接口加入了可选参数包装，调用开销微升（一次 `Option` 检查），对高频率的指标发送（如每 10ms）仍在毫秒级以内。  

- **安全考虑**  
  - **数据泄露**：`kv_used_blocks` 仅是内部资源占用计数，除非外部监控系统故意暴露，否则不构成安全风险。  
  - **错误路径**：`publish` 现在会在两个指标均为 `None` 时返回错误，防止业务层因忘记填充指标而导致监控失效，此类错误会在日志中显式报出，提升可观测性。  
  - **兼容性风险**：若外部系统仍使用旧版协议（仅 `active_decode_blocks`），在解析新结构时可能因新增字段导致反序列化错误。需要确保所有消费者同步升级或保持宽松的 serde 配置（如 `#[serde(default)]` 已添加）。

2. **锁存状态泄漏**：`DecodeBusyLatchState` 中的 `latched_busy` 可能在极端情况下（如 dp_rank 达到阈值后长期未收到任何新上报）永远保持 `true`，导致该节点被永久判为 busy。  
3. **向后兼容性**：外部监控或自定义插件若未适配新字段 `kv_used_blocks`，在使用 `serde_json` 等严格解析时可能报错。  
4. **测试覆盖不足**：虽然新增了大量单元测试，但真实生产环境中多进程/多机器的同步时序仍可能触发未捕获的竞态。  

**💡 关注建议**  
- **发布端强制上报**：在所有 worker 启动路径（包括 `MockEngine`、实际部署的 inference server）确认 `kv_used_blocks` 被填充，避免出现 “仅 active_decode_blocks” 导致锁存不一致的情况。

### fix: Fix multinode gang scheduling by setting MinAvailable to replica count on cliques (#7905)
**SHA**: `6e0d62d` | 🔗 [查看提交](https://github.com/ai-dynamo/dynamo/commit/6e0d62d74f5b58f1bd0bbb078134f79ad9350366)

**🎯 变更类型**：Bug 修复  
**⚡ 重要程度**：🟡 中  
**📋 变更摘要**：在 `GenerateGrovePodCliqueSet` 中，针对多节点（multinode）部署的角色，将 `PodCliqueSpec.MinAvailable` 从固定的 `1` 改为与副本数 (`r.Replicas`) 相同，以保证 gang‑scheduling 在跨节点场景下不因可用副本不足而提前启动。相应测试也同步更新期望值。  

**🎯 影响范围**  
- `deploy/operator/internal/dynamo/graph.go`：核心 DAG 生成逻辑。  
- `deploy/operator/internal/dynamo/graph_test.go`：单元测试验证。  
- 任何使用 `PodCliqueTemplateSpec` 的上层调度/运维流程（如 Dynamo Operator、Grove 调度器）。  

**💡 关注建议**  
1. **兼容性检查**：对非 multinode 场景仍保持 `MinAvailable = 1`，避免对单节点作业的调度产生不必要的限制。  
2. **性能与可靠性**：确保 `r.Replicas` 已经经过合法性校验（>0），否则可能出现 `MinAvailable` 为 0 导致调度器误判。  
3. **回滚路径**：若出现跨节点作业启动延迟，可临时通过 `isMultinode` 标识或直接修改 `MinAvailable` 覆盖此行为。  
4. **文档同步**：更新 Operator 部署手册中关于 `minAvailable` 的说明，明确多节点情况下该字段自动等于副本数。  
5. **监控验证**：部署后监控 `PodClique` 的 `availableReplicas` 与 `minAvailable` 对比，确认 gang‑scheduling 按预期阻塞/释放。  

整体来看，此修复消除了多节点作业因 `MinAvailable` 设为 1 而提前启动的调度错误，提升了跨节点 Gang 调度的正确性和可预测性。
