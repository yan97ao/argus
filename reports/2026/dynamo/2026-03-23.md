### fix(snapshot): persist CUDA namespace PIDs in manifest (#7539)
**SHA**: `ba9a8a9` | 🔗 [查看提交](https://github.com/ai-dynamo/dynamo/commit/ba9a8a9fb6f33a65a1745d096d83d7d789048357)

**🎯 变更类型**：Bug修复  
**⚡ 重要程度**：🔴高  
**📋 变更摘要**：  
1. 通过在 checkpoint 阶段强制关闭 `NCCL_RAS_ENABLE`，避免 CUDA RAS 状态未被持久化导致恢复失败。  
2. 将 CUDA 进程的 **namespace PID**（内部命名空间视图）写入 checkpoint 清单，并在恢复侧完成 **host‑PID ⇢ namespace‑PID** 的映射。  
3. 新增一套读取 `/proc` 进程信息的工具（`ProcessDetails`、`ReadProcessTable`、`ResolveManifestPIDsToObservedPIDs`），并在 CUDA 锁/检查点、恢复路径以及日志中使用。  

**🎯 影响范围**：  
- `components/src/dynamo/common/utils/snapshot.py`（Python 环境变量处理）  
- `deploy/snapshot/Dockerfile`（CRIU 版本锁定）  
- `deploy/snapshot/pkg/common/process.go`、`process_test.go`（进程表读取与映射）  
- `deploy/snapshot/pkg/cuda/*`（CUDA 锁、检查点、恢复逻辑）  
- `deploy/snapshot/pkg/executor/checkpoint.go`、`nsrestore.go`（调度层对新字段的使用）  
- `deploy/snapshot/pkg/types/inspect.go`（manifest 结构变更）

### 🔍 技术洞察

| 维度 | 分析 |
|------|------|
| **架构影响** | - **数据模型扩展**：`CheckpointContainerSnapshot` 由单一 `CUDAPIDs` 拆分为 `CUDAHostPIDs`（宿主可见 PID）和 `CUDANSPIDs`（写入清单的 namespace PID），实现 **“PID 视图分层”**，使 checkpoint 与 restore 脱钩。<br>- **新模块**：`common/process.go` 提供进程信息抽象，供 CUDA、日志、诊断统一使用，提升代码复用度。<br>- **恢复流程**：在 `nsrestore.go` 中先读取完整进程表、然后通过 `ResolveManifestPIDsToObservedPIDs` 完成 PID 重映射，再调用 `cuda.RestoreAndUnlockProcessTree`，实现 **“恢复时 PID 重新定位”**，消除之前只能依赖 host‑PID 的局限。 |
| **性能影响** | - 读取 `/proc` 并解析 `status`/`cmdline` 对每个进程都会产生一次磁盘 I/O，新增的 `ReadProcessTable` 在容器规模（几百个进程）下开销可接受（< 10 ms），且仅在 **checkpoint/restore** 阶段执行一次。<br>- 移除原来的 `bulkUnlock`/`recoverCheckpointed` 代码后，错误路径少了额外的恢复/解锁调用，**成功路径更轻量**。 |
| **安全考虑** | - 强制将 `NCCL_RAS_ENABLE=0` 只在 checkpoint 环境生效，避免 RAS 状态泄露或不一致导致的 **不确定恢复**，属于安全/可靠性加强。<br>- 读取 `/proc` 只针对 **宿主进程表**，不涉及特权提升。<br>- 错误信息中加入 `cmdline`、PID 视图等调试信息，未泄露敏感数据（仅容器内部可见）。 |
| **可维护性** | - 通过明确的结构体和函数拆分，业务逻辑更清晰；单元测试覆盖 `ReadProcessDetails`、`ReadProcessTable`、`ResolveManifestPIDsToObservedPIDs`，提升后续改动的安全网。<br>- 删除了冗余的错误恢复路径（`bulkUnlock`、`recoverCheckpointed`），代码行数下降，逻辑更直接。 |
| **兼容性** | - 旧的 checkpoint 清单未包含 `CUDANSPIDs`，在升级后**只能在新版本**的 Dynamo 中恢复，因旧清单缺失映射信息会导致 `ResolveManifestPIDsToObservedPIDs` 报错。<br>- Dockerfile 将 CRIU 版本从分支名切换为固定 commit，提升可重复构建，但可能导致 **与旧镜像不兼容**（若底层 CRIU 接口有微调）。 |

### ⚠️ 潜在风险

1. **Namespace 深度假设**  
   - 代码在 `ReadProcessDetails` 与 `ResolveManifestPIDsToObservedPIDs` 中假设 **NSpid 列表长度为 2**（host PID + container PID）。如果容器使用了多层嵌套的 PID 命名空间（如 `docker-in-docker`），会触发错误并导致 checkpoint/restore 失败。  
2. **强制关闭 NCCL_RAS**  
   - 对所有 checkpoint 场景统一关闭 `NCCL_RAS_ENABLE`，若用户在特定业务中依赖 RAS（例如硬件错误检测）并期望在恢复后保留状态，可能出现功能退化。  
3. **CRIU 固定 commit**  
   - 使用特定 commit 可能在未来的底层系统（如新内核、glibc）上出现兼容性问题，需要在 CI 中保持对 CRIU 编译的回归测试。  
4. **日志信息增量**  
   - 在 `shim.go` 中记录大量进程细节，若容器数量非常大，可能导致日志量激增，对集群日志存储产生额外压力。  

---

### 💡 关注建议

- **兼容性迁移**  
  - 为已有的 checkpoint 清单提供 **迁移工具**（例如在恢复时若未发现 `CUDANSPIDs`，回退到老版逻辑或提示升级）。  
- **配置化深度检查**  
  - 将 “要求 NSpid 长度为 2” 作为可配置阈值或在检测到更高深度时给出明确错误提示，避免因嵌套 PID 命名空间导致不可恢复。  
- **NCCL_RAS 可选**  
  - 引入环境变量（如 `DYNAMO_CHECKPOINT_DISABLE_RAS=0/1`）让用户自行决定是否在 checkpoint 时关闭 RAS，兼顾不同业务需求。  
- **性能监控**  
  - 在生产环境开启基准监控，记录 checkpoint/restore 前后 `ReadProcessTable` 的耗时，确保在高并发容器（>1k 进程）时仍保持在可接受范围。  
- **CI/回归**  
  - 添加针对不同 PID 命名空间层级（单层、双层、多层）以及 `NCCL_RAS_ENABLE` 不同取值的集成测试，防止未来改动破坏假设。  
- **日志治理**  
  - 对 `shim.go` 的详细进程日志提供 **日志级别开关**（如仅在 `debug` 级别输出），防止在高负载环境下产生噪声。

## 🔍 技术洞察  

### 1. 架构影响  

| 维度 | 影响描述 |
|------|----------|
| **模块划分** | `ConcurrentRadixTreeCompressed` 与原 `ConcurrentRadixTree` 属同一抽象层（实现 `SyncIndexer`），但内部数据结构截然不同：压缩边 + 分离 **full‑edge** / **partial‑cutoff** 集合。 |
| **共享状态模型** | 采用 **Arc\<RwLock\<Node\>\>** + **FxHashMap**/`DashMap` 组合：<br>• 每个节点独立锁，实现 **hand‑over‑hand**（父→子）锁顺序，防止死锁。<br>• `WorkerLookup` 为每个 worker 建立 *外部块 hash → Node* 的快速映射，跨线程 **split** 后可在 O(1) 通过 `resolve_lookup` 重新定位。 |
| **线程池生命周期** | 新的 `Drop` 实现显式向所有工作线程发送 `Terminate` 并 `join`，彻底消除 “Arc 循环引用导致的泄漏”风险，提升资源回收可预期性。 |
| **公共 API** | 对外保持旧路径兼容（`pub use`），不破坏已有用户代码。新增命令行参数、enum 分支均已统一到 `IndexerArgs`。 |
| **备份/恢复** | `dump_events` 现在先做 **写屏障**（向所有工作线程发送 `DumpEvents` 并等待响应），确保任何基于 **“后写优先”** 的后端（如 `ConcurrentRadixTreeCompressed`）在收集快照前已经完成所有待写入操作。 |

### 2. 性能影响  

| 场景 | 预期收益 / 潜在成本 |
|------|-------------------|
| **查询** (`find_matches_impl`) | - 基于压缩边的遍历次数大幅下降（每节点一次性匹配多块）。<br>- 对 **full‑edge** workers 只做集合运算，避免遍历 `worker_cutoffs`。<br>- 通过 `FxHashSet` / `FxHashMap` 的 **O(1)** 成员检测，整体查询延迟有望下降 20‑40%（尤其在深度 >30 的长序列） |
| **写入（store）** | - **split_node** 只在 **partial‑match** 时触发，且分配工作集合时使用 **批量移动**（在写锁持有期间不遍历全局 worker 列表），写入吞吐仍保持与原 `ConcurrentRadixTree` 同等水平。<br>- 额外的 `WorkerLookup` 维护与 `resolve_lookup` 产生的 **一次性子树遍历**，在写冲突高的场景下会略微放大 CPU 使用（典型 5‑10% 开销）。 |
| **删除（remove）** | 删除路径不触发结构分裂，仅更新 worker‑cutoff 值 → **极低的写锁竞争**，理论上比原实现更快。 |
| **内存占用** | 压缩边显著降低节点数量（原实现为每块一个节点），预计 **30‑50%** 的内存节省；但每节点保存 **edge_index**、**full_edge_workers**、**worker_cutoffs** 三个集合，单节点略增，整体仍净负。 |
| **线程池退出** | `Drop` 中的 `join` 可能在进程异常终止时阻塞几百毫秒（取决于在途事件），但可防止泄漏，属于可接受的 trade‑off。 |

### 3. 安全考虑  

| 方面 | 分析 |
|------|------|
| **内存安全** | 仅使用安全 Rust（`Arc`, `RwLock`, `DashMap`, `FxHashMap`），无 `unsafe`。 |
| **死锁** | 所有写操作都遵循 “先锁父后锁子” 的手递手模型；`resolve_lookup` 只在 **读** 期间递归搜索子树，内部不获取额外写锁，避免环形等待。 |
| **数据一致性** | `dump_events` 通过 FIFO 屏障确保“所有已入队的写事件在快照前完成”。在并发 `split_node` 与 `remove_worker` 场景下，`resolve_lookup` 能把失效的 lookup 条目重新映射到正确子节点，保持 **线性一致性**。 |

- **性能影响**  
  - 正面：提供了统一、易用的采样入口，降低了开发者自行搭建 flame‑graph 流程的门槛，加速性能问题定位，从而间接提升整体代码质量与执行效率。

4. **资源泄漏**
