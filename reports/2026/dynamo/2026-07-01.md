### fix(llm): tolerate control chars in OpenAI JSON bodies (#11114)
**SHA**: `37dd9b8` | 🔗 [查看提交](https://github.com/ai-dynamo/dynamo/commit/37dd9b8309e6152681032ecd84926f3421a1822b)

**🎯 变更类型**：Bug修复 / 安全加固  
**⚡ 重要程度**：🔴高  
**📋 变更摘要**：  
- 在 OpenAI 兼容 HTTP 接口中，新增对请求体中未转义的控制字符（≤ 0x1F）以及非法 UTF‑8 的容错处理。  
- 通过手动读取 `Bytes`、验证 `Content‑Type: application/json`，并在反序列化失败时尝试自动转义或使用 lossy‑UTF‑8，确保即使客户端发送的 JSON 含有非法字符也能得到有效的业务请求对象。  
- 伴随大量单元测试，覆盖 `is_json_content_type`、`ensure_json_content_type` 与各种异常 JSON 场景。

---

### 🎯 影响范围
- **模块/组件**：`core/llm/src/http/service/openai.rs`（OpenAI Completion & Chat Completion 入口）  
- **调用链**：`handler_completions`、`handler_chat_completions` → 新增的 `parse_json_request*`、`ensure_json_content_type`、`escape_json_string_control_chars`  
- **对外接口**：HTTP API（`/v1/completions`、`/v1/chat/completions`）的请求体解析行为。  

---

### 🔍 技术洞察  

| 维度 | 影响分析 |
|------|----------|
| **架构影响** | - 将原先的 `axum::extract::Json<T>` 自动解析改为显式的 **Bytes → 手动 JSON 解析**，在业务层加入了 *前置校验* 与 *容错*。<br>- 新的 `parse_json_request` 采用 **策略链**：① 正常反序列化 → ② 若失败尝试 **控制字符转义** → ③ 若仍失败走 **lossy UTF‑8**。<br>- 该层逻辑位于 **服务入口**，对后端业务（模型推理）无侵入，只是提升了对异常请求的容忍度。 |
| **性能影响** | - 额外的 **一次遍历**（`escape_json_string_control_chars`）和 **一次 `String::from_utf8_lossy`**，最坏 O(N) 且只在解析失败时才触发。<br>- 正常请求（合法 JSON）仍走 `serde_json::from_slice`，几乎无额外开销。<br>- 预期对吞吐量影响在 **千分之一** 级别，实际可通过压测确认。 |
| **安全考虑** | - 防止 **控制字符导致的 JSON 解析错误** 进而返回 500、泄露内部堆栈或触发服务崩溃，提升**健壮性**。<br>- 对 **Content‑Type** 做强制校验，阻止非 JSON 体被错误解析，符合 *OWASP API Security* 推荐。<br>- 转义实现只在字符串内部进行，不会改变 JSON 结构，避免注入风险；但需注意 **双反斜杠** 的处理（已在实现中保留 `escaped` 标记）。 |
| **可维护性** | - 引入 `DeserializeOwned` 泛型，使函数可复用到其他 OpenAI‑style 端点。<br>- 单元测试覆盖率大幅提升，新增 10+ 细粒度测试，降低未来回归风险。 |
| **兼容性** | - 仅更改 **handler** 参数签名（从 `Json<T>` → `Bytes`），内部仍返回相同业务模型；对外 API 行为保持**兼容**（错误码、返回结构不变）。<br>- 需要确保所有使用该 handler 的路由注册仍使用 `axum::extract::Body`（已通过 `Bytes` 自动提取），否则可能出现编译错误。 |

---

### ⚠️ 潜在风险  
1. **性能回退**：在异常请求占比极高的场景（如恶意攻击）会触发 **全体** 转义逻辑，导致 CPU 负载激增。  
2. **错误路径**：`escape_json_string_control_chars` 对 **已转义字符**（如 `\\n`）的处理依赖 `escaped` 标记，若出现未覆盖的转义序列，可能产生误转义或 JSON 结构破坏。  
3. **错误消息泄露**：`json_deserialize_error` 将原始 `serde_json::Error` 信息直接返回给客户端，可能泄露内部字段命名。  
4. **依赖版本**：使用 `serde::de::DeserializeOwned` 需确保所有编译目标的 `serde` 版本兼容，避免因特性不一致导致编译错误。  

---

### 💡 关注建议  

1. **监控与限流**  
   - 在生产环境对 `parse_json_request` 的警告日志进行采样（比如每分钟最多 100 条），防止日志洪水。  
   - 对 **异常 JSON 请求**（触发转义或 lossy 解析）单独计数，若短时间内激增可触发 **速率限制** 或 **IP 封禁**。  

2. **性能基准**  
   - 在高并发压测中模拟大量含控制字符的请求，评估 **CPU/内存** 使用峰值，必要时对转义逻辑做 **分片并行**（如使用 Rayon）或 **提前过滤**（仅在错误路径触发）。  

3. **安全审计**  
   - 检查 `json_deserialize_error` 是否泄露过多内部实现细节，考虑使用更通用的错误信息（如 `"Invalid JSON payload"`）以降低信息泄露风险。  
   - 确认 `escape_json_string_control_chars` 对所有合法转义序列均保持原样，建议加入 **模糊测试**（fuzzing）覆盖更多 Unicode 与转义组合。  

4. **回归测试**  
   - 为 **其它 OpenAI 端点**（如 embeddings、files、fine‑tunes）添加相同的容错测试，保持行为一致。  
   - 将 `ensure_json_content_type` 的行为抽象为通用中间件，以便在未来新增的 API 中复用。  

5. **文档更新**  
   - 在 API 文档中注明服务器会 **自动转义** 控制字符并返回 **warning** 日志，帮助用户排查客户端生成的非法 JSON。

### ⚠️ 潜在风险

1. **凭证误用**  
   - 在未装有 Dynamo Operator 的集群或 CI 环境运行时，`kubectl` 仍会尝试调用 webhook，可能因权限不足产生错误，甚至泄露集群凭证（若日志未过滤）。  
2. **错误的 `apiVersion` 参数**  
   - `--to` 参数若写错（如 `nvidia.com/v2alpha1`）会导致 webhook 返回 “no conversion webhook” 错误，脚本会抛异常，但错误信息可能不够直观。  
3. **临时文件泄漏**  
   - 使用 `tempfile.NamedTemporaryFile(delete=False)` 创建 JSON 请求体后手动 `os.unlink`，若在异常路径中未能执行 `finally`，可能残留敏感文件。当前实现已经在 `finally` 中删除，但仍需关注异常路径的完整性。  
4. **CI 覆盖盲点**  
   - 将该文件从 `deploy` 测试过滤后，若后续对其进行修改而未同步更新对应的独立 CI（例如专门的 lint/coverage），可能导致代码质量下降未被检测。  
5. **YAML 1.1 兼容性实现的局限**  
   - 通过强制单引号处理 `_YAML11_AMBIGUOUS`，虽覆盖常见布尔/空值关键字，但新出现的 YAML 1.1 隐式转换规则若未列入集合，仍可能出现未转义的情况。

### refactor(frontend): rename DYN_ENABLE_FRONTEND_{NVEXT,ADMIN_API} to DYN_DISABLE_* (#11101)

**🎯 变更类型**：重构 / 架构变更  
**⚡ 重要程度**：🔴高  
**📋 变更摘要**：将前端功能开关的环境变量从 `DYN_ENABLE_FRONTEND_{NVEXT,ADMIN_API}`（默认开启，需通过 **false** 值关闭）改为 `DYN_DISABLE_FRONTEND_{NVEXT,ADMIN_API}`（默认开启，需通过 **true** 值关闭），并同步更新文档、代码注释、构建器逻辑以及对应的测试。目标是统一“禁用”语义，使配置更直观、降低误配置风险。

**🎯 影响范围**：  
- `docs/components/frontend/configuration.md`（文档）  
- `lib/llm/src/http/service/*.rs`（Anthropic、OpenAI、通用服务）  
- `lib/llm/src/http/service/service_v2.rs`（核心构建器 & `State`）  
- `lib/runtime/src/config/environment_names.rs`（环境变量常量）  
- `tests/frontend/test_frontend_api_surface_compliance.py`（集成测试）  

**🔍 技术洞察**  
- **架构影响**：  
  - 环境变量解析从 `env_is_falsey` → `env_is_truthy` 迁移，所有 “是否开启” 的判定由 “默认开启 + 禁用变量为真” 改为 “默认开启 + 禁用变量为真”。  
  - `HttpServiceConfigBuilder` 中的 `enable_nvext` / `enable_admin_api` 仍保留 builder‑level 开关，现通过 `env_is_truthy(DYN_DISABLE_…)` 与之 **AND**，实现 “代码默认 ∧ 环境变量禁用” 的组合逻辑。  
  - 常量名称在 `environment_names.rs` 统一更改，避免在其他模块中出现残留的 `ENABLE_` 常量，提升可维护性。

- **代码审计**：检查是否还有其他 `#[cfg(target_os = "linux")]` 的硬编码路径（如 `O_DIRECT` 常量），防止未来出现类似编译问题。

### ⚠️ 潜在风险  

1. **CLI 兼容性**：旧版脚本或外部工具若在 `--disaggregation-mode` 参数值列表中硬编码 `agg|prefill|decode`，在新版中会接受 `encode`，但如果未在代码中处理 `encode`（如自定义子类），可能导致 `panic`。

6. **异常路径返回不完整**：在 `generate` 的 Encode 分支，若 `context.is_stopped()` 在生成 `encoder_result` 之前被触发，只返回 `cancelled`，但未返回 `encoder_result`（符合设计），但调度层如果仍然期待 `encoder_result` 可能出现空指针错误。
