### fix(operator): Preserve legacy DGD worker hash across v1beta1 conversion (#9210)
**SHA**: `cd27229` | 🔗 [查看提交](https://github.com/ai-dynamo/dynamo/commit/cd272290b513708e15acbe3ba3c7152ccb8d5f8a)

**🎯 变更类型**：功能增强 / Bug修复 / 重构 / 架构变更  

**⚡ 重要程度**：🔴 高  

**📋 变更摘要**  
1. 在 `v1alpha1` → `v1beta1` 转换过程中，保留并迁移了旧版 **DGD worker hash**（`nvidia.com/current-worker-hash`）到新的注解 `nvidia.com/dgd-legacy-worker-hash`，确保升级后仍可使用已有 worker hash 防止不必要的滚动更新。  
2. 为此新增 `legacy_worker_hash.go`（计算 hash）以及对应的单元测试，扩展了转换实现 `dynamographdeployment_conversion.go`，并在清理内部注解时同步删除旧注解。  
3. 完善了 API 文档 `CONVERSION.md`，明确 **conversion helpers** 的命名、可见性、签名及使用原则，防止以后在控制器或内部包中误用转换注解。  
4. 重构了 `shared_spec_conversion.go`：  
   - 抽离共享内存、GPU‑Memory‑Service、Failover、Checkpoint 等实验特性的 **ConvertTo / ConvertFrom** 辅助函数，提升可读性并统一错误处理。  
   - 修正 `convertSharedMemoryTo/From` 的实现细节，以避免不必要的指针创建并保持 DeepCopy 语义。  
5. 在 `v1beta1` 新增部署条件常量、拓展 `DynamoComponentDeploymentSharedSpec` 的查询方法（Inter‑Pod GMS / Failover、影子 pod 计数、总引擎 pod 数）。  
6. 新增内部工具函数 `v1beta1_helpers.go`（组件映射、PodTemplate 访问、Ingress 结构体、元数据合并等）以及相应的单元测试。  
7. 引入内部 `ingress_types.go`，为保留 v1alpha1 Ingress 行为提供结构体，且不泄露到公共 API。  
8. 将原先在 `internal/dynamo/hash.go` 中的老实现简化为对 `v1alpha1.ComputeDGDWorkersSpecHash` 的直接调用，统一 hash 计算入口。

### refactor(llm): consolidate simple OpenAI DeltaAggregators (#9223)
**SHA**: `8889910` | 🔗 [查看提交](https://github.com/ai-dynamo/dynamo/commit/88899104696d54e43afb2889f178bd5084b59970)

**🎯 变更类型**：重构  

**⚡ 重要程度**：🔴高  

**📋 变更摘要**：  
本次提交在 `openai` 协议层统一了各子模块（audio、embeddings、images、videos）原先的 `DeltaAggregator` 实现，抽象为 `stream_aggregator` 通用模块。通过 `StreamAggregable` trait 与 `aggregate_stream` 函数，实现对 `Annotated<T>` 流的通用聚合，并在每个响应类型上提供对应的 `empty` 与 `merge` 实现，移除大量重复代码并更新所有调用方。  

**🎯 影响范围**：  
- `lib/llm/src/protocols/openai/*`（audio、embeddings、images、videos）  
- 新增模块 `stream_aggregator.rs`  
- 公开的 `DeltaAggregator` 类型已被移除（可能影响直接依赖此类型的外部代码）  

**🔍 技术洞察**：  
- **架构影响**  
  - 引入 `StreamAggregable` trait，形成“一套聚合接口 + 通用实现”的微型抽象层，降低了协议子模块之间的耦合度。  
  - 每个响应结构自行实现 `empty` 与 `merge`，使聚合逻辑在业务层保持可控，同时统一错误处理路径。  
  - 移除对 `futures::StreamExt::fold` 的重复使用，改为 `while let Some(delta) = stream.next().await` 的显式循环，提升可读性并便于后续扩展（如加入日志、统计）。  

- **性能影响**  
  - `fold` 与手写 `while` 循环在底层实现上差别不大，实际运行时的 CPU/内存开销基本持平。  
  - 通过消除多余的 `DeltaAggregator` 结构体实例化，略微降低了堆分配次数，整体性能略有提升（在大规模并发流聚合场景下更为显著）。  
  - 错误短路仍通过 `?` 运算符实现，保持了原有的早停特性。  

- **安全考虑**  
  - 变更未引入新的外部依赖或不安全代码，安全风险基本为 **无**。  
  - 错误信息仍直接向上层返回 (`String`)，未改变泄露路径。  

**⚠️ 潜在风险**：  
1. **向后兼容性**：`DeltaAggregator` 已从各模块的公开 API 中移除，若有外部项目直接 `use` 该类型或调用其 `apply` 方法，将出现编译错误。  
2. **实现遗漏**：`StreamAggregable::merge` 必须在每个响应实现中完整覆盖所有业务字段（如 usage、metadata 等），若后续新增字段忘记在 `merge` 中处理，可能导致聚合结果不完整。  
3. **错误传播细节**：`aggregate_stream` 直接使用 `delta.ok()?`，若 `Annotated::ok()` 的错误信息结构变化，需同步更新此处的错误处理逻辑。  

**💡 关注建议**：  
- **升级指南**：在迁移到新版本时，检查代码中是否仍引用 `DeltaAggregator`，改为调用对应响应的 `from_annotated_stream`（已内部使用统一聚合函数）。  
- **单元测试**：为每个实现了 `StreamAggregable` 的响应类型增加 **merge‑edge‑case** 测试，确保新增字段在聚合时被正确累加或合并。  
- **文档更新**：在公开文档或 crate 的 README 中说明 `DeltaAggregator` 已废弃，推荐使用统一的流聚合方式。  
- **监控/日志**：如果未来需要对聚合过程进行监控，可在 `aggregate_stream` 中加入统一的计时或日志埋点，所有协议子模块将自动受益。  

---  

整体来看，此次重构显著提升了代码可维护性与一致性，风险可通过升级指南和完善测试得到有效管控。推荐尽快在主分支合并并同步发布迁移文档。

### fix(profiler): use client.components for decode log path (#8733)
**SHA**: `969ad3d` | 🔗 [查看提交](https://github.com/ai-dynamo/dynamo/commit/969ad3d6ac8cc6a87fe59ab7337997e5dd957630)

**🎯 变更类型**：Bug修复 / 功能增强  
**⚡ 重要程度**：🔴 高  
**📋 变更摘要**：  
修复了 Dynamo Profiler 在解析日志路径时错误地使用 `SubComponentType.DECODE`（基于配置）来推断解码服务名称的问题。改为通过 `client.components` 动态获取实际部署中的解码组件名称，并在未能识别时安全回退 `"decode"`。相应单元测试也同步更新以使用新的 `pick_decode_component` 辅助函数。

**🎯 影响范围**：  
- `components/src/dynamo/profiler/interpolation.py`  
- `components/src/dynamo/profiler/thorough.py`  
- `components/src/dynamo/profiler/utils/profile_common.py`（新增 `pick_decode_component`）  
- Profiler 相关的集成测试文件  

**🔍 技术洞察**：

- **架构影响**  
  - **解耦**：原先 Profiler 通过 `Config` 与 `SubComponentType` 直接关联解码服务，隐式假设服务名一定匹配配置。但实际部署的组件名称来源于 `client.components`（由部署规范生成），两者可能不一致。改动将路径解析逻辑从配置层迁移到运行时部署信息层，使 Profiler 与调度/部署层的耦合度降低，提升了系统的可组合性。  
  - **容错性**：新增 `pick_decode_component` 在组件列表为空或仅包含 frontend 时返回固定 `"decode"`，防止因误选 frontend 导致日志读取失败的运行时异常。  

- **性能影响**  
  - 仅在 Profiler 运行阶段（即 profiling job）执行一次简单的列表遍历查找，时间复杂度 O(n)（n 为组件数，通常 ≤ 3）。对整体 profiling 执行时间影响可以忽略不计。  
  - 通过避免错误的日志路径查找，减少了因文件不存在导致的重试或异常捕获开销，间接提升 profiling 流程的成功率与速度。  

- **安全考虑**  
  - 变更不涉及网络、权限或数据加密，仅影响内部路径拼接。`pick_decode_component` 使用 `client.components`（由可信的部署系统填充），不存在外部输入注入风险。  
  - 回退返回硬编码 `"decode"`，同样是受控字符串，不会产生路径遍历等安全漏洞。  

**⚠️ 潜在风险**：

1. **组件列表顺序变更**：如果未来部署规范把解码组件放在列表的最后（或出现多个非‑frontend 组件），`pick_decode_component` 仍会返回第一个非‑frontend 项目，可能不是实际负责解码的 worker，从而导致日志路径仍不匹配。  
2. **多解码组件场景**：当前实现仅返回单一名称，若出现多实例（例如 `decode0`, `decode1`）的情况，可能需要进一步扩展以支持完整的日志检索策略。  
3. **回退硬编码**：在极端情况下（组件列表为空且实际使用的服务名不是 "decode"），回退会导致仍然找不到日志文件。虽然此类部署极不可能，但如果出现会产生误报。  

**💡 关注建议**：

- **代码层面**  
  - 将 `pick_decode_component` 的实现文档化，注明其假设（“第一个非 frontend 即为 decode”），并在关键调用处添加注释，以防后续维护者误解。  
  - 若计划支持多 decode 实例，考虑返回列表或提供可配置的过滤规则（例如基于 `service_type == "decode"` 的标签）。  

- **测试层面**  
  - 增加针对 `client.components` 包含多种顺序或仅有 frontend 的单元测试，验证回退路径的正确性。  
  - 在 CI 中加入对 `pick_decode_component` 的覆盖率阈值，防止未来修改时遗漏。  

- **运维层面**  
  - 确认部署流程在 `create_deployment` 中始终填充 `client.components`，尤其在使用自定义插件或旧版 Planner 时。  
  - 若出现 “frontend” 仍被误选的情况，检查部署规范是否误将 decode 服务标记为 frontend。  

通过上述改动，Profiler 对日志路径的解析更加可靠，降低了因配置不一致导致的 profiling 失效风险，对整体系统的可用性与可维护性均有正面提升。

**📋 变更摘要**：

3. 新增 `compute_bootstrap_room` 辅助函数并在 PrefillRouter 中使用，实现 “room % dp_size == dp_rank” 的模运算约束，同时对 63 位上限进行安全检查。

- `lib/llm/src/discovery/model_manager.rs`（模型管理器 API）  
- `lib/llm/src/kv_router/prefill_router/execution.rs`（Prefill 路由的 Bootstrap 房间生成）

**🔍 技术洞察**：

| 维度 | 影响分析 |
|------|-----------|
| **架构影响** | - 引入 `load_block_size_override` 通过 `OnceLock` 在进程启动时读取一次环境变量，保持 **只读、全局** 的实现，不会破坏现有的 `block_size` 配置层级。<br>- `ModelManager::get_data_parallel_size` 让调度层能够获取每个 worker 的 DP 大小，这是 **跨模块** 的新依赖（PrefillRouter → ModelManager），但仅为只读查询，兼容性好。<br>- `compute_bootstrap_room` 将 DP rank 编码进 `bootstrap_room`，实现 **端到端的 DP‑aware routing**，并保持原有的 “room ≤ i64::MAX” 合约。 |
| **性能影响** | - 环境变量读取使用 `OnceLock`，仅一次初始化，运行时开销可忽略。<br>- 负载分母改为 `load_block_size_override` 后可能产生更细粒度的 `potential_prefill_block` 计算，理论上提升调度精度，极端情况下（`load_block_size` 设得非常小）会导致 **更频繁的浮点除法**，但在 10‑100 微秒级别的调度路径中仍可接受。<br>- `compute_bootstrap_room` 为 O(1) 整数运算，额外的模运算与乘除对总体吞吐量影响极小。 |
| **安全考虑** | - 读取环境变量本身无网络或文件 I/O，风险极低。<br>- `load_block_size_override` 只接受 `>0` 的正整数，防止除 0 或负数导致 panic。<br>- `compute_bootstrap_room` 对 `dp_size == 0` 做了安全过滤，避免除 0。<br>- `bootstrap_room` 仍受 63 位上限限制，防止整数溢出向下游（SGLang）泄露异常值。整体未引入新攻击面。 |
| **可维护性** | - 新增的函数均使用 `pub`/`private` 明确分层，代码注释完整。<br>- 单元测试覆盖了边界、异常和随机分布，提升了未来回归检测的可靠性。<br>- 环境变量的硬编码在文档中需要补充，以免用户误设导致调度失衡。 |

**⚠️ 潜在风险**：

1. **误配置环境变量**：若运营人员将 `DYN_ROUTER_LOAD_BLOCK_SIZE` 设为异常大或非常小，可能导致调度器对 `potential_prefill_block` 计算失真，进而出现 **负载倾斜**（某些 worker 过载或空闲）。  
2. **DP size 与实际运行不一致**：`ModelManager::get_data_parallel_size` 依赖 `runtime_configs` 中的 `data_parallel_size` 字段，若该字段在某些 worker 启动时未正确填充，`compute_bootstrap_room` 会退化为普通随机 `r`，可能破坏 **DP‑aware** 的负载分布预期。  
3. **兼容性回退**：旧版本的路由逻辑仍假设 `bootstrap_room` 与 DP 无关，升级后如果下游服务（如 SGLang）未同步更新解析逻辑，可能出现 **room 与 rank 不匹配** 的错误。  
4. **测试随机性**：虽然单元测试使用 `rand::random_range`，但在 CI 环境中若 RNG 受限或种子固定，某些极端分布可能未被检测到。

- **配置验证**：在工作节点启动阶段加入校验，若环境变量设置超出合理区间（如 > 10 000 或 < 1) 发出警告并回退到默认 `block_size`。  
- **兼容层**：在 PrefillRouter 生成 `bootstrap_room` 前，检测目标下游服务的版本或特性标记，若不支持 DP‑aware 房间，可直接使用旧实现，保持向后兼容。  
- **监控 & 报警**：在调度器中增加 `load_block_size_override` 的运行时日志（debug/info），以及 `bootstrap_room` 生成的统计（% 匹配成功率），便于快速定位异常。  
- **性能回归验证**：在 CI 中加入基准测试（benchmark）对比开启/关闭 `DYN_ROUTER_LOAD_BLOCK_SIZE` 时的调度延迟与吞吐量，确保不出现不可接受的性能回退。  
- **代码审计**：建议在未来的 PR 中将 `load_block_size_override` 的实现迁移到统一的配置中心（如 `dyn-config`），以便统一管理、热更新，并避免依赖环境变量的硬编码方式。

### fix(prefill-router): persist endpoint after handshake so decode rebuild reactivates (#8965)
**SHA**: `7ae69a3` | 🔗 [查看提交](https://github.com/ai-dynamo/dynamo/commit/7ae69a3127dc69d66dff76df779b1734f1b148d3)

**🎯 变更类型**：Bug修复 / 稳定性提升  
**⚡ 重要程度**：🔴高  
**📋 变更摘要**：  
1. 为 `PrefillActivationState` 引入 `PrefillReady(Box<Endpoint>)`，在 prefill 端点首次出现后永久缓存，以便后续 decode WorkerSet 重建时能够立刻激活而不再等待。  
2. 使用 `DashMap::entry` 实现原子化的键级状态迁移，消除 `remove → insert` 期间的竞争窗口。  
3. 新增 `remove_decode_prefill_waiter`，在 decode WorkerSet 销毁时安全清除残留的 `DecodeWaiting(sender)`，防止重建时因旧的 `oneshot` 接收器导致路由缺失。  
4. `ModelWatcher` 调整 teardown 逻辑：prefill 组件离线时彻底清除激活器并停用 decode 侧路由；decode 组件离线时仅清理 `DecodeWaiting`，保留缓存的端点。  

---

**🎯 影响范围**：  
- `lib/llm/src/discovery/model_manager.rs`（核心路由激活状态管理）  
- `lib/llm/src/discovery/watcher.rs`（WorkerSet 生命周期监控）  
- 依赖 `PrefillRouter`、`Decode WorkerSet` 的所有业务模块（模型调度、推理请求路由）  

---

**🔍 技术洞察**  

- **架构影响**  
  - 将 prefill 端点的生命周期从 “一次性 handshake” 改为 “持久缓存”，使 decode 侧在任何时间点（包括 pod 重启、滚动升级）都能快速恢复 prefill 路由，提升系统的自恢复能力。  
  - 通过 `DashMap::entry` 实现的原子化状态切换，消除了 `remove_prefill_activator` 与 `activate_prefill_router` 并发时可能出现的“脏状态”窗口，增强了分布式环境下的并发安全。  
  - 新增的 `remove_decode_prefill_waiter` 为 decode 侧的 **“先注册后等待”** 场景提供了专属清理路径，避免在 decode 重建后仍残留 `DecodeWaiting` 导致的路由失效。  

- **性能影响**  
  - **正向**：Cache 端点后，decode 重建时无需再次等待 prefetch 注册，平均降低路由激活延迟数百毫秒到秒级（取决于 prefill 启动时间）。  
  - **负向**：每次 `activate_prefill_router` 必须克隆 `Endpoint`（`Endpoint` 可能包含大型对象），但该操作仅在 prefill 注册时发生，频率低且可接受。  
  - `DashMap::entry` 相比原先的 `remove + insert` 多一次锁获取，锁粒度仍然是键级别，整体性能影响可以忽略。  

- **安全考虑**  
  - 变动仅涉及内部状态管理和并发控制，不暴露新的网络接口或修改加密/鉴权路径。  
  - 端点对象的克隆在内存中进行，无网络交互，未引入安全风险。  
  - 需要注意 `Endpoint` 中可能包含 credentials（如 Token），持久化在 `PrefillReady` 中会在内存中保留更久，建议审查 `Endpoint` 是否已对敏感信息做脱敏或加密处理。  

---

**⚠️ 潜在风险**  

1. **内存泄漏**：`PrefillReady(Box<Endpoint>)` 长期保留端点，若模型频繁创建/删除且未触发 `remove_prefill_activator`（如异常退出），可能导致旧端点残留。  
2. **竞态边界**：虽然使用了 `entry` 抽象，但仍需确保所有对同一 `(model, namespace)` 键的修改（包括 watcher 的 `remove_worker_set`、`deactivate_prefill_router_for_decode`）均走同一代码路径，防止出现未被清理的 `PrefillReady`。  
3. **错误的 `remove_decode_prefill_waiter` 调用**：若误在 prefill 组件 teardown 时调用，可能导致缓存端点被误删，回退到原来的 “等待” 行为。  
4. **日志噪声**：新增的 `debug/info` 日志在高并发环境下可能产生大量输出，需要在生产环境中适当调节 log level。  

---

**💡 关注建议**  

- **回归测试**：在 CI 中加入包含 prefill ⇄ decode 重建、滚动升级以及异常退出的端到端场景，验证路由始终可用且 latency 下降。  
- **监控/告警**：新增指标 `prefill_endpoint_cached_total` 与 `prefill_activation_latency_ms`，监控缓存命中率和激活时延。  
- **资源审计**：定期检查 `prefill_router_activators` 的大小，确保在模型卸载后对应条目被彻底删除（`remove_prefill_activator` 必须被调用）。  
- **安全审计**：确认 `Endpoint` 中不包含持久化的凭证，或在缓存前进行脱敏；若有，则在 `PrefillReady` 中存储的仅是必要的网络地址信息。  
- **代码规范**：保持 `PrefillActivationState` 枚举的大小平衡（已使用 `Box`），并在未来若加入更多字段时继续使用 `clippy::large_enum_variant` 进行检查。  

通过上述改动，系统在面对 prefill / decode WorkerSet 的生命周期波动时能保持更高的可用性与一致性，同时保持了良好的并发安全和可维护性。

#### 性能影响
- **流式 token delta 处理**：`_completion_delta_output` 依据 `meta_info.completion_tokens` 只返回新增 token，避免 Dynamo 端累计已输出的 prompt token，确保后端流式性能不受额外复制开销。

#### 安全考虑
- **自定义 Jinja 模板路径校验**：在 `args.parse_args` 中对 `custom_jinja_template` 执行文件存在性检查，防止路径遍历或读取不存在文件导致运行时异常。

### ci: nightly release automate (#9241)
**SHA**: `9606b3b` | 🔗 [查看提交](https://github.com/ai-dynamo/dynamo/commit/9606b3b807b280408f1a3040d8398307c050807b)

**🎯 变更类型**：功能增强 / CI/CD 自动化  
**⚡ 重要程度**：🟡 中  
**📋 变更摘要**：为所有核心 GitHub Actions 工作流新增 `source_ref` 参数，实现统一的 SHA 解析与镜像标签；在 `nightly-ci.yml` 中引入“解析源码 SHA → 计算 nightly 版本 → 计算发布模式 → 条件化测试 → 自动化发布”完整链路；`release.yml` 进一步支持 nightly 版本号、轮子版本生成以及可选的 GitLab release‑automation 触发。

**🎯 影响范围**  
- `.github/workflows/dynamo-pipeline.yml`、`nightly-ci.yml`、`release.yml`、`shared-build-image.yml`、`shared-test.yml`  
- 依赖这些工作流的 CI（vllm/sglang/trtllm 构建、测试、dynamo‑pipeline）  
- CI 触发的 ECR 镜像标签、NGC 镜像发布及 Artifactory wheel 版本  

**💡 关注建议**  

1. **SHA 一致性**  
   - `resolve-source-sha` 直接输出 `github.sha`，但在手动 `workflow_dispatch` 时若使用 `source_ref`，需要确保后续所有作业仍使用该 resolved SHA（已通过 `${{ needs.resolve-source-sha.outputs.source_sha }}` 传递）。建议在关键步骤（镜像 tag、wheel 构建）打印 `SOURCE_SHA` 进行 debug。  

2. **权限设置**  
   - `release.yml` 将 `contents` 权限从 `write` 降为 `read`，但后续仍有 `git push`、`tag` 等操作？确认这些操作改为在子 workflow `release.yml` 里通过 `workflow_call` 调用的上层工作流拥有足够权限。  

3. **条件化测试**  
   - `compute-release-mode` 通过 `run_tests` 控制所有 test jobs 的 `if` 条件。请验证在 schedule 触发时 `run_tests` 为 true，且在手动关闭时所有测试均真正跳过（状态为 `skipped`），避免误阻止 `release` job。  

4. **Nightly 版本生成**  
   - Nightly wheel 采用 `${BASE_VERSION}.dev${DATE}`，而 NGC 镜像为 `nightly-YYYYMMDD-<shortsha>`。确保 `pyproject.toml` 在对应 SHA 上的 `version` 与代码实际发布的版本匹配，防止 wheel 与镜像版本不一致导致 downstream CI 失效。  

5. **GitLab Pipeline 触发**  
   - 新增 `skip_gitlab_pipeline` 参数及 `trigger-gitlab-release-pipeline` job。请检查在 CI 环境变量 `GITLAB_PIPELINE_URL`、`GITLAB_TRIGGER_TOKEN` 已经在 `automated-release` 环境中安全配置；同时验证 `curl` 返回的 JSON 结构（`id`、`web_url`）在错误时不抛出未捕获异常。  

6. **文档与兼容性**  
   - `source_ref` 参数默认空，保持向后兼容；但若调用方显式传入空字符串，也会回退到 `github.sha`。建议在 README/Workflow 文档中明确说明该字段的用途与 fallback 行为。  

7. **测试覆盖**  
   - 在 PR 合并前加入一次 `workflow_dispatch` 并指定 `source_ref` 为旧的提交，验证镜像/tag、wheel 以及 GitLab trigger 是否正确使用该 SHA。  

总体来看，此次改动为 Nightly 发布建立了完整、可配置的流水线，并提供了灵活的手动控制与紧急跳过机制。只要上述细节（权限、SHA 传递、错误处理）得到验证，改动对现有 CI 的影响应可控。
