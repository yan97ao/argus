### fix(mocker): size KV cache blocks with AIC (#9598)
**SHA**: `60e3685` | 🔗 [查看提交](https://github.com/ai-dynamo/dynamo/commit/60e3685e733086ed44a33b4e665e7a9152d1103d)

**🎯 变更类型**：功能增强 / 重构 / 架构变更  
**⚡ 重要程度**：🔴高  

**📋 变更摘要**  
本次提交为 AIC（AI‑Accelerator SDK）后端的 KV‑cache 容量估算添加了端到端支持。  
- 通过 `--num-gpu-blocks-override`、`--gpu-memory-utilization`、`--mem-fraction-static` 等 CLI 参数，允许在未显式设置 `num_gpu_blocks` 时自动向 AIC SDK 询问可用 KV‑cache 块数。  
- `MockEngineArgs` 迁移至 **JSON‑like 输入**（`dict`）而非直接使用 Rust‑绑定对象；新增 `gpu_memory_utilization`、`mem_fraction_static` 字段并实现范围校验。  
- 在 Python 侧 (`dynamo._internal.aic`) 与 Rust 侧 (`estimate_aic_num_gpu_blocks`) 双方实现统一的估算逻辑，并在 `replay` 与 `replay_optimize` 流程中自动填充 `num_gpu_blocks`。  
- 移除 `MockEngineArgs.dump_json()`，改用 `from_json()` 进行序列化/反序列化，配合 `serde_path_to_error` 提供更友好的错误定位。  
- 大量单元/端到端测试更新，以验证路由 KV‑overlap 记录、AIC 估算、以及新 `nvext` 字段的正确性。  

**🎯 影响范围**  
- **核心组件**：`components/src/dynamo/mocker/*`（CLI 参数、配置解析、`MockEngineArgs` 结构）  
- **Profiler / Replay**：`components/src/dynamo/profiler/utils/replay_optimize/*`（engine‑arg 传递、跨进程桥接、Spec 序列化）  
- **Python 绑定**：`lib/bindings/python/rust/*`（Rust ⇔ Python 桥、AIC 调用、`MockEngineArgs` 新方法）  
- **内部 AIC 模块**：`lib/bindings/python/src/dynamo/_internal/aic.py`（容量估算公开函数）  
- **测试套件**：所有 MM‑router、serve、replay 相关测试，新增 `router_logs`、`router_nvext` 辅助库。  
- **依赖**：新增 `serde_path_to_error`（JSON 解析错误定位）以及若干 `pydantic` 校验规则。  

**🔍 技术洞察**  

| 维度 | 影响 |
|------|------|
| **架构影响** | 1. `MockEngineArgs` 由 Rust → Python 的 “对象 + dump_json” 交互，改为 **纯 JSON 字典** 形式，彻底解耦了 Python 与 Rust 的序列化实现。<br>2. `EngineSpec`/`ReplayOptimizeSpec` 现在直接持有 **engine‑arg dict**，不再在跨进程时手动 `dump_json()/from_json()`，降低了跨语言/跨进程的序列化错误风险。<br>3. AIC 估算入口统一到 `dynamo._internal.aic.estimate_num_gpu_blocks`，Rust 侧仅提供低层调用 `estimate_aic_num_gpu_blocks`，实现了 **单一来源的容量模型**（后端‑独立）。 |
| **性能影响** | - 动态估算仅在 **构建 engine args** 时调用一次（本地函数或轻量的 Python‑Rust FFI），对整体 replay / mocker 吞吐基本无影响。<br>- 通过默认 `gpu_memory_utilization=0.9`、`mem_fraction_static=0.88`，可在 **不显式配置** 时得到更贴近实际的 KV‑cache 大小，提升模拟器的调度准确性，间接提升吞吐/Latency 基准的可信度。 |
| **安全考虑** | - 新增字段加入 **pydantic 范围校验**（`0.0 ≤ value ≤ 1.0`），防止非法负值或 >1 的配置导致除 0 错误或内存估算失真。<br>- `serde_path_to_error` 在 JSON 解析错误时抛出详细路径信息，降低因错误输入导致的潜在服务崩溃风险。<br>- `estimate_aic_num_gpu_blocks` 对后端名称做白名单检查，只允许已实现 KV‑capacity 的后端，否则抛 `ValueError`，防止不受支持的后端被误用。 |
| **可维护性** | - 将 **字段校验、默认填充** 完全搬入 `MockEngineArgs` 的 `TryFrom` 实现，代码路径更集中，旧的手工 `from_json_str` 解析逻辑被删除，减少维护负担。<br>- 通过 `OptionalConfigValue<T>` 区分 “缺省”“显式 null”“显式非 null”，在未来新增字段时能够保持向后兼容而不破坏已有 JSON 配置。<br>- 所有路由 KV‑overlap 解析统一到 `tests/utils/router_logs.py`，测试代码不再依赖硬编码正则，提升可读性与复用度。 |
| **兼容性** | - **默认 `num_gpu_blocks` 改为 `None`**（除非显式给出或 AIC 可估算）。非 AIC 运行时仍回退到内部默认 `16384`，保持旧行为。<br>- 移除 `MockEngineArgs.dump_json()`，外部若仍调用将失效；但项目内部已改为使用 `from_json()`/`to_json()`（Pydantic）或直接字典形式。<br>- 新增的 `--gpu-memory-utilization`、`--mem-fraction-static` 为 **可选** 参数，不会影响现有 CLI 调用。 |
| **测试覆盖** | - 新增 **AIC 容量单元测试**、**KV‑event Prometheus 指标校验**、**router nvext** 验证，覆盖了正向估算、零值/边界值、错误路径以及兼容性路径。<br>- 端到端路由测试的日志解析从正则改为结构化解析，提高了对日志格式微调的鲁棒性。 |

1. **后向兼容性**  
   - 某些外部工具可能仍依赖 `MockEngineArgs.dump_json()` 或手动构造 JSON 并期望 `num_gpu_blocks` 默认存在。若这些工具在未提供 `num_gpu_blocks` 时直接使用 JSON，可能得到 `null` 并触发后端断言。

### feat(planner): make PrometheusConnect SSL verification configurable (#9510)
**SHA**: `bb65fc1` | 🔗 [查看提交](https://github.com/ai-dynamo/dynamo/commit/bb65fc1e7d0ff9be5ce16a124acda0378dddc086)

**🎯 变更类型**：功能增强（feat）  
**⚡ 重要程度**：🟡中  
**📋 变更摘要**：在 Planner 中新增对 Prometheus TLS 证书校验的可配置开关。通过 `PROMETHEUS_SSL_VERIFY` 环境变量或 `PlannerConfig.metric_pulling_prometheus_ssl_verify` 配置项，默认保持原有 `disable_ssl=True` 行为，开启后会在 `PrometheusConnect` 初始化时进行证书验证，并可配合自定义 CA bundle 使用。  

**🎯 影响范围**：  
- `components/src/dynamo/planner/config`（默认值、模型）  
- `components/src/dynamo/planner/core/base.py`（参数传递）  
- `components/src/dynamo/planner/monitoring/traffic_metrics.py`（PrometheusConnect 实例化）  

**🔍 技术洞察**：  
- **架构影响**：仅在 Planner 子系统内部引入了一个布尔配置项，传播路径从默认配置 → `PlannerConfig` → `BasePlanner` → `TrafficMetrics`。没有跨模块或跨服务的依赖改动，架构冲击极低。  
- **性能影响**：开启证书校验会在每次 TLS 握手时增加一次 CA 验证开销，延迟提升通常在毫秒级，整体吞吐量影响可忽略不计。关闭时行为与旧实现完全相同。  
- **安全考虑**：提供了可选的 TLS 证书校验，解决了之前默认跳过验证的潜在中间人攻击风险。若使用私有 CA，需要配合 `PROMETHEUS_CA_BUNDLE`（或等价机制）注入根证书，否则会导致连接失败。  

**⚠️ 潜在风险**：  
1. **配置误用**：运维人员可能误以为默认已开启校验，导致在使用私有 CA 时出现连接错误。  
2. **兼容性**：某些旧版 Prometheus 实例使用自签证书，开启校验后会导致 Planner 启动失败，影响业务可用性。  
3. **环境变量冲突**：`PROMETHEUS_SSL_VERIFY` 与已有 `PROMETHEUS_TOKEN`、`PLANNER_PROMETHEUS_PORT` 等变量共存，若拼写错误或大小写不一致会默认为关闭验证，形成安全盲点。  

**💡 关注建议**：  
- 在部署文档中明确 `PROMETHEUS_SSL_VERIFY` 的默认值、可选取值以及与自定义 CA bundle 的配合方式。  
- 为私有 CA 场景提供示例 Helm/ConfigMap 注入方法，或在代码层面支持 `PROMETHEUS_CA_BUNDLE` 环境变量读取。  
- 增加单元/集成测试覆盖：① 验证 `ssl_verify=False` 与旧行为一致；② `ssl_verify=True` 且证书有效时能够成功连接；③ 证书无效时抛出明确异常。  
- 考虑在后续大版本中将默认值改为 `True`，并在发布说明中给出迁移指引，以逐步提升整体安全姿态。

**📋 变更摘要**  
本次提交在 *tool‑calling* 章节新增《Troubleshooting Tool Calls》文档，详细说明在工具调用出现异常时通过 `logprobs: true` 捕获原始 token 流的做法。对应的导航、README 与示例页面均已加入指向该指南的提示，并在 parity 测试文档中加入了使用该文档进行用户问题定位的说明。

**💡 关注建议**  

1. **链接完整性**：确认 CI 中的文档检查（如 markdown 链接校验）已覆盖新页面，防止出现 404。
