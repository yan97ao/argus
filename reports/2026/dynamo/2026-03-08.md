#### 🔴 高重要度变更 (1)

### refactor: replace kube-rbac-proxy sidecar with controller-runtime WithAuthenticationAndAuthorization (#7045)
**SHA**: `7dbebf3` | 🔗 [查看提交](https://github.com/ai-dynamo/dynamo/commit/7dbebf3c61bd6adeb62d285f56f4d9434afb1f30)

**🎯 变更类型**：重构 / 安全修复  
**⚡ 重要程度**：🔴高  
**📋 变更摘要**：此次提交将原先通过 `kube‑rbac‑proxy` sidecar 保护的 `/metrics` 接口，改为使用 controller‑runtime 内置的 `WithAuthenticationAndAuthorization` 过滤器直接在 operator 主容器中完成身份认证与授权。为此引入了新的 RBAC（TokenReview、SubjectAccessReview 及 `/metrics` GET 权限），默认开启安全 TLS（`secure: true`），并相应更新 Helm chart、Kustomize、CRD 默认值以及文档。  

**🎯 影响范围**：  
- `deploy/helm/charts/platform/components/operator`（deployment、service、service‑monitor、RBAC）  
- `deploy/operator/config`（kustomize、rbac）  
- Operator 配置 API（`OperatorConfiguration`、defaults、types）  
- 主进程代码（`main.go`）以及依赖库（controller‑runtime metrics filter）  
- 文档（API reference、observability）  

**🔍 技术洞察**  

- **架构影响**  
  - 移除了 `kube-rbac-proxy` sidecar，降低了 Pod 资源占用与启动时间。  
  - 通过 `controller-runtime` 的过滤器在同一进程内完成 authn/authz，统一了监控链路的实现方式。  
  - 引入了 **ClusterRole** + **ClusterRoleBinding**，在 namespace‑restricted 与 cluster‑wide 两种模式下均能满足授权需求。  
  - 新增 `metrics-reader` ClusterRole，确保 Prometheus ServiceAccount 能执行 `SubjectAccessReview` 并访问 `/metrics`。  

- **性能影响**  
  - **正面**：去掉 sidecar 后，容器数从 2→1，CPU/Memory 开销约减少 5‑10 %（原 sidecar 最高 500 m CPU、128 MiB）。  
  - **负面**：每次 metrics 抓取都会触发一次 `SubjectAccessReview`（以及 `TokenReview`），在极高抓取频率下会产生额外的 API Server 调用，可能带来几毫秒的额外延迟，但对大多数集群影响可忽略。  

- **安全考虑**  
  - 默认开启 `secure: true`，metrics 通过 HTTPS（自签 TLS）对外提供，绑定地址从 `127.0.0.1` 改为 `0.0.0.0`，但只能在通过 authz 的请求后才能访问。  
  - 引入的 `TokenReview` 与 `SubjectAccessReview` 权限是 **必需** 的，若未正确绑定会导致 Prometheus 抓取失败。  
  - ServiceMonitor 中 `insecureSkipVerify: true` 仍保留，以兼容自签证书；若外部访问需自行改为校验。  
  - 删除 `kube-rbac-proxy` 相关的 `ignore-paths=/metrics` 配置，避免误认为 metrics 已被放行。  

**⚠️ 潜在风险**  

1. **兼容性**：老版本 Helm chart（或手动部署）仍可能期待 `kube-rbac-proxy` sidecar；升级后若未同步修改 `values.yaml`，Pod 将缺少 metrics 端口或监控报错。  
2. **RBAC 漏洞**：若 `metrics-reader` ClusterRole 未绑定到 Prometheus 的 ServiceAccount（或用户自定义的 SA），将导致监控系统无法读取指标，业务可观测性受损。  
3. **TLS 失效**：`insecureSkipVerify: true` 只在内部使用安全；若用户将 ServiceMonitor 暴露给外部网络，可能产生 MITM 风险。  
4. **Metrics 端口冲突**：端口从 `8443` 迁移到 `8080`，任何自定义 Service/Ingress 仍指向旧端口的场景会失效。  
5. **依赖库升级**：`controller-runtime` 新增 `metrics/filters`，会引入额外的间接依赖（如 `cel-go`、`grpc-gateway` 等），可能导致编译或二进制体积增加。  

**💡 关注建议**  

- **升级流程**：在 Helm 升级前务必检查 `values.yaml` 中 `metricsService.enabled` 为 `true`，并确认 `metricsService.ports.name` 与 `metricsService.ports.port` 已同步至新值（`metrics:8080`）。  
- **RBAC 校验**：部署后执行以下命令确认 Prometheus ServiceAccount 已拥有 `metrics-reader` 权限：  
  ```bash
  kubectl auth can-i get --non-resource-url=/metrics --as=system:serviceaccount:<ns>:prometheus
  ```  
- **监控验证**：查看 `ServiceMonitor` 中的 `bearerTokenFile` 是否指向正确的 ServiceAccount token；若使用自签证书，可在 `ServiceMonitor` 中自行开启 `tlsConfig.insecureSkipVerify: false` 并提供 CA。  
- **回滚方案**：若业务对 metrics 抓取有强依赖且出现异常，可临时恢复 `kube-rbac-proxy` sidecar（在 Helm `values.yaml` 中重新启用 `controllerManager.kubeRbacProxy`），或在 `OperatorConfiguration` 中将 `metrics.secure` 设置为 `false` 并手动暴露 `127.0.0.1:8080`。  
- **文档更新**：确保团队成员了解新 RBAC 需求，尤其在自建 Prometheus 堆栈时需要额外绑定 `metrics-reader`。  
- **性能监控**：在高抓取频率（>30 s）环境下关注 API Server `subjectaccessreviews` 请求计数，必要时通过缓存或降低抓取频率来缓解压力。
