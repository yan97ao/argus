### feat(velo): add unix domain socket (#7197)
**SHA**: `9e9ca3e` | 🔗 [查看提交](https://github.com/ai-dynamo/dynamo/commit/9e9ca3e2ddeadfa6fc96ea69249c6765e099f38b)

**🎯 变更类型**：功能增强 / 重构 / 架构变更  
**⚡ 重要程度**：🔴高  

**📋 变更摘要**  
1. 在 `velo-transports` 中新增 **Unix Domain Socket (UDS) 传输层**，实现了 `uds` 模块（listener、transport、builder），并通过 `#[cfg(unix)]` 进行条件编译。  
2. 对原 TCP writer 任务引入 `CancellationToken` 参数，实现 **写任务可在 teardown 阶段被主动取消**，避免阻塞 shutdown。  
3. 为 UDS 传输补齐了 **注册、发送、连接管理、健康检查、优雅关闭（三阶段）**的完整实现，并配套大量单元/集成测试。  

**🎯 影响范围**  
- `lib/velo-transports/src/lib.rs` – 导出 `uds` 模块  
- `lib/velo-transports/src/tcp/transport.rs` –  writer 任务签名改动 & 取消逻辑加入  
- 新增 `uds/listener.rs、uds/transport.rs、uds/mod.rs` – 完整的 UDS 实现  
- 测试层 `lib/velo-transports/tests/*` – 统一的 `TestTransportHandle` 与 `TestCluster` 扩展支持 UDS，新增 UDS 集成与 shutdown 场景  
- 依赖 `Cargo.toml`（未展示）可能已添加 `tokio` 的 `unix` 功能

## 🔍 技术洞察

### 1️⃣ 架构影响
| 维度 | 影响 |
|------|------|
| **模块化** | 引入 `uds` 子模块，实现了 **Transport trait** 的完整替代实现，与现有 TCP、（预留的 UCX）保持同一抽象层。通过 `#[cfg(unix)]` 确保仅在 Unix 系统编译，未破坏跨平台构建。 |
| **依赖复用** | 复用 `TcpFrameCodec`（基于 `AsyncRead + AsyncWrite`）进行帧编解码，避免在 UDS 中重新实现协议；代码复用率>90%，降低维护成本。 |
| **启动流程** | `UdsTransport::start` 先 **同步 bind**，返回 `Result`，确保在任务启动前就捕获端口冲突或已有监听器的错误；这与 TCP 的 `bind`/`listen` 逻辑保持一致。 |
| **连接管理** | 与 TCP 相同的 **DashMap + ConnectionHandle** 结构，实现 lock‑free 并发写入，但改为存储 `PathBuf` 而非 `SocketAddr`。通过 `get_or_create_connection` 的 **entry API** 实现原子检查‑插入，防止竞争条件。 |
| **三阶段优雅关闭** | `ShutdownState` 已在 UDS 中完全接入：<br>• **Gate**：在 `UdsListener::handle_connection` 中检测 `is_draining()` 并返回 `ShuttingDown` 帧；<br>• **Drain**：写任务通过 `CancellationToken` 观察 `teardown_token`，在 Phase‑2 中仍然可以发送 `Response/Ack/Event`；<br>• **Teardown**：监听器与 writer 任务在 token 触发后立即退出，并清理 socket 文件。 |
| **测试覆盖** | 统一 `TestTransportHandle`、`TestCluster` 通过 **工厂模式**（`UdsFactory`）生成 UDS 实例，实现 **端到端、健康检查、并发、流控、drain** 等场景的自动化验证，提升回归安全性。 |

### 2️⃣ 性能影响
| 维度 | 正面 | 负面 |
|------|------|------|
| **CPU/系统调用** | UDS 完全绕过 TCP/IP 栈，减少内核路径、降低上下文切换，尤其在同主机进程间通信时可显著提升 **吞吐** 与 **时延**（典型可达 10‑20µs vs 100µs）。 | 额外的 **socket 文件清理**、`metadata` 检查与 `connect` 探测会在 `start` 时产生一次 I/O，成本极低。 |
| **零拷贝** | 重用 `TcpFrameCodec` 的 **bytes::Bytes** 零拷贝路径，保持与 TCP 相同的 **零拷贝发送** 效果。 | `PathBuf` 的复制在 `get_or_create_connection` 中仍会产生一次小额 heap 分配（路径字符数），影响甚微。 |
| **并发度** | `DashMap` + `flume` 的有界通道仍提供 **背压**，防止写入无限积压；`CancellationToken` 的 `select!` 只在等待 `cancelled` 时额外占用一个 future，几乎无 CPU 开销。 | `select!` 中对 `cancelled()` 的轮询会略微增加任务调度次数，但在实际负载（数百并发）下可忽略不计。 |
| **资源占用** | 与 TCP 基本等价（每个连接一个 writer 任务 + channel），不再需要 **TCP 选项**（如 `nodelay`），略微降低 per‑socket 结构体大小。 | 每个 UDS 监听器会在文件系统上创建一个 **socket 文件**，需注意因 `tmpfs` 限制导致的节点数上限（一般不构成瓶颈）。 |

### 3️⃣ 安全考虑
| 关注点 | 说明 |
|--------|------|
| **文件系统权限** | UDS 文件路径的 **UNIX 权限** 决定谁能连接。当前实现未显式设置 `chmod`，默认由创建进程的 umask 决定，可能导致 **意外暴露**（例如 777）。建议在 `bind()` 后调用 `std::fs::set_permissions`，或在 `UdsListenerBuilder` 中提供 `set_mode(u32)` 参数。 |
| **路径注入** | `parse_uds_endpoint` 直接接受任意 UTF‑8 字符串并转为 `PathBuf`，如果上层调用构造了 `uds://../somefile` 并且父目录可写，可能导致 **目录遍历** 或 **socket 文件覆盖**。建议对路径做 **绝对化**(`canonicalize`) 并检查是否位于预期根目录（如 `/tmp` 或容器绑定卷）。 |
| **双绑定竞争** | `start()` 在绑定前先探测是否已有 **活跃监听器**，但仍有极端竞争窗口（两进程几乎同时间 bind），会导致 `EADDRINUSE` 错误。现有实现已返回 `Err`，可接受。 |
| **资源泄漏** | `UdsListener::serve` 在退出时 `remove_file`，但如果进程异常退出（panic）可能留下 stale socket 文件。可在 `Drop` 实现中再做一次清理，或在 `shutdown` 前确保 `teardown_token` 已触发。 |
| **DoS 防护** | 限制 `channel_capacity`（默认 256）防止单个对端发送大量消息导致内部 channel 爆满；但仍需上层使用 `TransportAdapter` 的 **流控** 机制防止滥用。 |

### 4️⃣ 潜在风险
1. **Stale 句柄竞态**：`get_or_create_connection` 使用 `DashMap::entry` 进行原子检查‑插入，但仍在 `remove_if` 前持有旧句柄的 guard，可能导致瞬时 **双写**（旧 writer 仍在执行）。概率极低且不会破坏数据完整性。  
2. **异常路径下的清理不完整**：`connection_writer_task` 在 `connect` 失败后仍会执行 `while let Ok(msg) = rx.try_recv()` 清理队列，但若 `rx` 已被关闭且还有未消费的消息，`try_recv` 仍能获得，这可能导致 **错误回调多次**。测试已覆盖，但生产环境建议在 `connect` 失败后直接 `rx.drain()`。

### fix: patch vLLM 0.17.1 multi-node TP init ordering, enable gpu_2 pre-merge CI (#7357)
**SHA**: `c7bac97` | 🔗 [查看提交](https://github.com/ai-dynamo/dynamo/commit/c7bac97902c6893df6e016adaca66ee78d8f5b39)

**🎯 变更类型**：Bug修复  
**⚡ 重要程度**：🔴高  
**📋 变更摘要**：为 vLLM 0.17.1 添加了多节点 Tensor‑Parallel（TP）初始化顺序的热修复 patch，避免 `WorkerProc.__init__` 中提前初始化消息队列导致的分布式组未创建错误；同步在 CI 工作流中标记了相关的多 GPU 测试占位，以便后续开启。  

**🎯 影响范围**：  
- `container/deps/vllm`（vLLM 依赖的构建与安装脚本）  
- `.github/workflows/pr.yaml`（CI pre‑merge 工作流的多 GPU 测试开关）  
- 使用 vLLM 0.17.1 的所有 Dynamo 运行时实例（尤其是多节点 TP 场景）  

**🔍 技术洞察**  

- **架构影响**：  
  - 修改了 `vllm/v1/executor/multiproc_executor.py` 中 `WorkerProc.__init__` 的调用顺序：先执行 `init_device()` → `load_model()`，随后再调用 `_init_message_queues()`。这使得在 **nnodes‑within‑dp > 1** 的多节点 TP 部署中，分布式进程组（`_INNER_DP_WORLD`）能够在消息队列创建前完成初始化，消除了原先的 “group not found” 错误。  
  - 通过 `install_vllm.sh` 在容器构建阶段自动对源码或已安装的 wheel 进行 patch，保持了 **源码构建** 与 **CUDA wheel** 两条路径的一致性。  

- **性能影响**：  
  - 修复的是初始化顺序错误导致的死锁/启动失败，理论上对推理吞吐量没有直接影响。  
  - 额外的 `git apply` / `patch` 步骤只在镜像构建时执行一次，耗时极小（< 1 s），对运行时性能基本为零影响。  

- **安全考虑**：  
  - 只在检测到 `VLLM_VER=0.17.1` 时才执行 patch，且在版本不匹配时直接 **fail fast**（退出并提示删除补丁），避免在未知版本上盲目修改代码。  
  - Patch 文件随代码仓库一起发布，来源可审计；使用 `git apply --ignore-whitespace` 与 `patch -p1`，不涉及网络下载，风险可控。  
  - 唯一潜在风险是 **CI 构建脚本被误改** 后仍然执行 `git apply`，可能对其他依赖产生意外修改，建议保持审计路径（`SCRIPT_DIR` 与 `/tmp/deps/vllm`）的严格限制。  

**⚠️ 潜在风险**  

1. **版本锁定风险**：若项目后续升级到 vLLM ≥ 0.17.2，当前脚本会因版本不匹配而中止构建，导致 CI 失效。需要在升级后同步删除补丁和对应检查代码。  
2. **双重打补丁**：在源码构建与 wheel 安装同时存在时，可能两个路径都尝试应用同一补丁，若补丁已被应用会产生冲突报错。虽然脚本已通过路径判断避免重复，但在极端环境（例如手动拷贝目录）下仍需留意。  
3. **CI 多 GPU 测试仍关闭**：当前工作流仍把 `run_multi_gpu_tests` 设为 `false`，若后续开启而未同步更新标记（`gpu_2`），可能出现遗漏的测试场景。  
4. **Patch 维护负担**：补丁文件维护在项目内部，若 upstream 修复内容发生变动（例如文件路径重构），本地 patch 可能失效或导致误改。  

**💡 关注建议**  

- **升级准备**：在计划升级至 vLLM 0.17.2 或更高版本前，务必移除 `install_vllm.sh` 中的版本检查与 patch 应用块，同时删除 `multinode‑tp‑init‑order.patch`，防止构建中止。  
- **CI 改进**：待多 GPU 多节点测试标记（`gpu_2`）准备就绪后，打开 `run_multi_gpu_tests` 并删除相关 `TODO` 注释，确保 CI 能覆盖该修复路径。  
- **回归测试**：在多节点（≥ 2 node）TP 环境下执行至少一次完整的启动‑推理‑结束流程，验证不再出现 “_INNER_DP_WORLD 未创建” 的错误，同时监控启动时间的微幅变化。  
- **审计日志**：考虑在 `install_vllm.sh` 中加入日志记录（如 `git log -1` 或 `patch --dry-run`）以供后续审计，帮助快速定位补丁是否成功、是否重复。  
- **文档同步**：在项目的依赖说明或部署指南中注明此补丁的适用范围与迁移步骤，防止用户在自定义镜像时忽视该限制。

### 🔍 技术洞察

#### 1. 架构影响
- **服务解耦**：原本 KV Router 通过本地 radix tree 直接维护缓存索引，现在可以通过 `remote_indexer_component` 将索引工作交给独立的 KV Indexer 服务。这样实现了 **“router ←→ indexer”** 的明确协议边界，利于水平扩展、单独部署和独立升级。
- **运行时统一化**：所有与分布式服务交互（worker 发现、事件平面、请求平面）现在统一走 `DistributedRuntime`，代码复用度提升，原本 mocker 中的手工 `loop` 被 `create_runtime` 替代，降低了运行时初始化错误的概率。
- **新增运行时任务**：  
  - **Discovery watcher**：监控 MDC（Model Discovery Catalog）动态添加/删除的 worker，并在 `WorkerRegistry` 中维护 `discovered_workers`。  
  - **Event subscriber**：基于事件平面（NATS 或 ZMQ）订阅 KV 事件，直接喂给对应的 `Remote` Indexer（不再依赖 ZMQ 监听）。  
  - **Query engine**：在 Runtime 模式下对外提供 `kv_indexer_query` 端点，前端可通过请求平面远程查询 overlap 分数。
- **配置兼容**：通过 `remote_indexer_component` 为实验性功能，默认仍保持原有本地 radix tree 行为，保证向后兼容。

#### 2. 性能影响
| 场景 | 正向影响 | 负向影响 |
|------|----------|----------|
| **本地 Router → 本地 Indexer**（原有模式） | 低延迟（内存直接查询） | 受单节点 CPU、内存限制，若 KV 事件量大可能出现瓶颈 |
| **Router → Remote Indexer（请求平面）** | - 将索引计算迁移到专用服务，可水平扩展<br>- 减轻 Router 进程的 CPU/内存负担<br>- 通过 NATS/HTTP（可压缩）降低网络带宽占用（相比 ZMQ） | - 增加一次网络往返（RTT）+ 序列化/反序列化开销<br>- 依赖请求平面的可靠性，出现网络抖动时查询延迟会增长 |
| **事件平面切换（NATS ↔ ZMQ）** | NATS 提供更可靠的消息持久化、负载均衡，适合大规模集群 | NATS 相比 ZMQ 引入额外的 broker 节点，可能导致额外的网络跳数和分布式调度开销 |

整体来看，**Remote Indexer** 适用于 **大规模集群** 或 **需要专门索引服务的场景**，在单机/小规模部署仍建议保留默认本地索引以获得最小延迟。

#### 3. 安全考虑
- **跨平面通信**：请求平面（NATS/HTTP）默认不带认证，新增的 `kv-indexer` 服务如果暴露在公开网络，需要配合 **Dynamo Runtime 的 ACL / TLS** 配置（当前代码未显式处理），否则可能导致未经授权的查询或 DoS。
- **环境变量泄露**：新加入的 `DYN_EVENT_PLANE`、`DYN_REMOTE_INDEXER_COMPONENT` 等配置通过环境变量传递，运维需注意不要将敏感信息写入日志或共享环境。
- **错误信息**：`RemoteIndexer::find_matches` 将后端错误直接转成 `anyhow::Error` 并记录在日志，建议对错误分类（如超时、服务不可用）做统一监控，以防泄露内部实现细节。

#### 4. 可维护性与可测试性
- **代码分层**：`remote_indexer.rs`、`discovery.rs`、`subscriber.rs` 各自独立，便于单元测试。已经加入了 `tests`（`parse_workers`）作为示例，后续应补齐 **Discovery watcher**、**Event subscriber** 的模拟测试。
- **特性门控**：使用 Cargo feature `indexer-runtime` 隐藏运行时依赖，避免在不需要的场景下拉入 `dynamo-runtime`。但 `standalone-indexer` 默认仍依赖 `metrics`，需要文档明确说明编译方式。
- **配置验证**：`KvRouterConfig` 新增字段已加入 `serde(default)`，但缺少 **互斥校验**（如同时设置 `remote_indexer_component` 与本地 `router_queue_policy` 的冲突），建议在 `validate_kv_router_config` 中加入相应约束。
