### fix(streaming): propagate backend errors in SSE streams (#8674)
**SHA**: `8b55dc3` | 🔗 [查看提交](https://github.com/ai-dynamo/dynamo/commit/8b55dc307460d88781f657b112bd220c8f03960a)

**🎯 变更类型**：Bug修复 / 功能增强  
**⚡ 重要程度**：🔴高  
**📋 变更摘要**：  
- 在 SSE（Server‑Sent Events）流的前端处理器中，引入统一的错误构造与分类工具，实现对后端错误（如图片加载失败）和内部错误的精准传播。  
- Rust 服务端的 metrics 处理逻辑同步升级，优先使用 Dynamo 错误信息，并对注释中的换行符进行安全清理，防止 Axum panic。  
- 新增完整单元测试覆盖错误生成、处理以及 Rust 端的错误转化行为。

**🎯 影响范围**：  
- `components/src/dynamo/frontend/sglang_processor.py`, `vllm_processor.py`（前端流式生成模块）  
- `components/src/dynamo/frontend/utils.py`（错误处理公共函数）  
- `components/src/dynamo/frontend/tests/`（前端错误工具测试）  
- `lib/llm/src/http/service/metrics.rs`（服务端 metrics 事件转换）  

**🔍 技术洞察**  
- **架构影响**：  
  - 前端 SSE 流的错误处理路径被抽象为 `handle_engine_error`、`make_backend_error`、`make_internal_error` 三个函数，提升了代码复用度并统一了错误返回格式。  
  - 服务端 metrics 转换层现在能够直接读取 `DynamoError` 的信息，减少了对 `comment` 字段的依赖，使错误路径更加明确。  
  - 新增的 `try/except` 捕获块保证了意外异常也会以统一的 OpenAI‑style 错误返回，提升了系统的鲁棒性。  

- **性能影响**：  
  - 仅涉及少量字符串拼接与字典构造，时间复杂度保持 O(1)，对高并发 SSE 流的吞吐量影响可忽略。  
  - Rust 端对 `comment` 做 `replace(['\n','\r'], " ")` 的操作同样是线性于注释长度，且注释通常极短，性能损耗极低。  

- **安全考虑**：  
  - 错误信息直接转发给客户端，可能泄露后端实现细节（如具体文件路径、内部异常堆栈）。目前仅转发 `engine_response["message"]`，若该字段包含敏感信息，需要在后端做脱敏。  
  - Rust 端对 `comment` 的换行符进行清理，防止 Axum 在日志/监控系统中因非法字符导致 panic，提升了服务的安全可靠性。  

**⚠️ 潜在风险**  
1. **错误信息泄露**：后端错误的原始 `message` 可能包含业务机密或内部部署信息，暴露给前端用户。  
2. **兼容性**：旧版客户端可能只期望 `error.type == "internal_error"`，引入 `backend_error` 需要确认前端 SDK 对此新类型的兼容处理。  
3. **异常捕获范围**：在 `_generate_and_stream` 中捕获了所有 `Exception`，若未来引入需要特殊处理的异常（如取消、超时），可能被统一包装成 `internal_error`，导致丢失细粒度错误信息。  
4. **依赖循环**：`utils.py` 现在引入了 `logging`，虽然未造成循环，但在极端的 import 组织变化时需留意。  

**💡 关注建议**  
- **错误脱敏**：在后端生成 `backend_error` 前，加入白名单或过滤规则，只保留对用户有意义的错误描述，避免泄露内部实现细节。  
- **文档与 SDK 更新**：更新公开的 API 文档，说明返回的 `error.type` 可能为 `backend_error` 或 `internal_error`，并在各语言 SDK 中加入相应的枚举/异常映射。  
- **细化异常捕获**：考虑区分 `CancelledError`、`TimeoutError` 等可恢复异常，在 `except Exception` 前加入专门的 except 分支，以返回更合适的错误类型（如 `client_error`）。  
- **监控与告警**：利用新增日志信息（`Backend error for request …`），在监控平台中为 `backend_error` 建立独立告警，以便快速定位后端服务异常。  
- **回归测试**：在 CI 中加入对错误类型的兼容性验证，确保旧客户端在收到 `backend_error` 时仍能正常解析并展示。  

通过上述改动，系统在错误可观测性、统一性以及服务端安全性方面获得显著提升，只要注意信息脱敏和兼容性，即可平滑上线。

#### 🟡 中重要度变更 (1)

### fix(sglang): preserve max_new_tokens=None to avoid silent 128-token cap (#8743)
**SHA**: `635eb80` | 🔗 [查看提交](https://github.com/ai-dynamo/dynamo/commit/635eb80ee39a9cce4da615b665041a7a45f2a65e)

**🎯 变更类型**：Bug 修复  
**⚡ 重要程度**：🟡 中  

**📋 变更摘要**  
在 `components/src/dynamo/sglang/request_handlers/llm/decode_handler.py` 中，`_build_sampling_params` 过去会在生成参数字典时直接过滤掉所有值为 `None` 的条目。SGLang 对 `max_new_tokens=None` 有特殊含义（生成至 EOS／上下文长度），但若该键被删除会默认回落到 128 token，导致静默的截断。现在加入 `keep_if_none = {"max_new_tokens"}`，在 `None` 时仍保留该键，以避免意外的 128‑token 上限。

**🎯 影响范围**  
- `components/dynamo/sglang` 相关的请求处理路径（decode_handler）。  
- 所有通过 Dynamo 调用 SGLang 的上层模型/服务，只要在请求中显式设置 `max_new_tokens=None`。  

**💡 关注建议**  
1. **确认下游**：检查 SGLang 客户端的调用链是否能够接受 `max_new_tokens=None`（目前已在代码中使用），避免出现 `None` 被误当作缺省值导致后续错误。  
2. **测试覆盖**：为 `max_new_tokens=None` 场景补充单元测试，验证返回的 token 长度不被 128 限制，并确保其它采样参数仍然按原逻辑过滤 `None`。  
3. **文档更新**：在 Dynamo‑SGLang 接口文档中说明 `max_new_tokens=None` 的行为已被保留，提醒用户如需“无限制”生成请显式传 `null`（即 Python 的 `None`），否则将使用默认 128。  
4. **回滚检查**：若未来在其它语言的实现（例如 Rust‑side）加入了类似的过滤逻辑，需要同步此保留策略，以防出现跨语言行为不一致。  

整体来看，此修复消除了潜在的隐蔽截断错误，对已有功能影响有限，且只涉及参数构建阶段，风险较低。建议尽快在 CI 中加入对应的回归测试，确保上线后行为如预期。
