### fix(tcp): abort writer when reader join fails to avoid monitor hang (#9716)
**SHA**: `8acf173` | 🔗 [查看提交](https://github.com/ai-dynamo/dynamo/commit/8acf1738fc5e1cd4d73f5acdc7a912c4c359890e)

**🎯 变更类型**：Bug修复  
**⚡ 重要程度**：🔴高  
**📋 变更摘要**：在 `wait_for_connection_tasks` 中，原先使用 `tokio::join!` 同时等待 reader 与 writer 两个任务，导致当 reader 任务因 panic 或错误提前退出时，writer 仍可能在 `bytes_rx.recv()` 上阻塞，形成监控（monitor）线程的永久挂起。此次改为**顺序 await**：先等待 reader，若失败立即 **abort** writer 并等待其结束，再统一处理错误。新增单元测试验证在 reader panic 时 writer 能被及时中止，函数不会卡死。  

**🎯 影响范围**：  
- `core` crate 中的 `lib/runtime/src/pipeline/network/tcp/client.rs`（TCP 客户端连接任务调度）。  
- 依赖此模块的上层网络管道、分布式计算调度器以及任何使用 Dynamo TCP 客户端的服务（如参数服务器、模型推理节点）。  

**🔍 技术洞察**  

- **架构影响**  
  - **任务生命周期管理**：将原本的并行 `join!` 改为显式的顺序 `await + abort`，使得连接任务的错误传播更可靠，避免因 writer 隐式阻塞导致上层监控协程卡死。  
  - **错误传播路径**：错误从 reader 直接返回，writer 的错误（若有）也会被捕获并返回；两者错误不再相互掩盖，提升了监控层对异常的可观测性。  
  - **资源回收**：通过 `writer_task.abort()` 强制取消 writer，随后 `await` 其结束，确保 Tokio 运行时能够回收相关 task 本体和底层 socket，防止泄漏。  

- **性能影响**  
  - **无额外运行时开销**：顺序 `await` 与 `join!` 在成功路径上几乎等价（只多了一次 `await` 调用），对正常流量的性能影响可忽略不计。  
  - **避免死锁/阻塞**：在异常路径上显著提升响应速度——原先可能无限等待而导致监控线程卡死，修复后在毫秒级内返回错误，提升系统整体的可用性与恢复时间。  

- **安全考虑**  
  - **panic 处理**：reader panic 现在会被捕获为 `JoinError::panic` 并向上传递，避免了未捕获的 panic 传播到运行时导致进程崩溃。  
  - **数据完整性**：在 abort writer 时，未完成的写入操作会被中止，可能导致部分业务数据未发送。但因为此情形已是读端不可用（reader 已异常），继续写入毫无意义，且不再隐藏错误。总体安全风险不增。  

**⚠️ 潜在风险**  
1. **中止写入导致丢失未发送的数据**：如果 writer 正在发送关键业务 payload，而 reader 因非致命错误（如短暂的解码异常）而退出，abort 会直接丢弃这些数据。业务层如果对“写完即成功”有强一致性要求，需要在更高层实现重试或事务补偿。  
2. **Abort 可能触发 writer 里的 `Drop` 逻辑**：若 writer 在 `Drop` 中执行了重要的清理（如释放锁、发送心跳结束等），强制 abort 可能导致这些清理代码未被执行，需要确认 writer 的 `Drop` 实现是幂等且安全的。  
3. **错误日志重复**：现在 reader 错误会被记录一次，writer 错误（若在 abort 前产生）也会被记录一次，可能导致日志量稍增。  

**💡 关注建议**  
- **业务层容错**：对于关键写入，建议在上层实现 **幂等写入或重试**，因为底层在 reader 失效时会直接中止 writer。  
- **审查 Writer 的 `Drop` 实现**：确保 `abort` 不会导致资源泄漏或未完成的清理操作，如必要可在 `abort` 前发送显式的关闭信号。  
- **监控与告警**：新增的错误日志已区分 reader 与 writer，可在监控系统中分别跟踪两类异常频率，以快速定位是读端还是写端问题。  
- **回归测试**：除了当前新增的 panic 场景，建议补充 **writer 超时/阻塞** 场景的测试，确保在读端正常的情况下 writer 的异常同样能被及时捕获并返回。  
- **文档更新**：更新 `client.rs` 中函数注释，说明函数在读端错误时会主动 abort writer，帮助后续维护者快速理解任务生命周期。

### fix(runtime): encapsulate routing instance snapshots (#9655)
**SHA**: `3974f36` | 🔗 [查看提交](https://github.com/ai-dynamo/dynamo/commit/3974f36c575147b57165e8dd6f3d218e92690c6e)

**🎯 变更类型**：功能增强 / 重构 / 安全修复  
**⚡ 重要程度**：🔴 高  

**📋 变更摘要**  
本次提交对 **kv‑router** 与 **runtime** 两大子系统进行深度改造，核心目标是 **将“繁忙”(busy) 检测升级为“过载”(overload) 检测** 并 **将路由实例快照抽象化**：  
1. 新增 `OverloadedWorkerProvider`，把“哪些 worker 被标记为 overload” 的判断交由调度器统一获取。  
2. 所有调度、路由、选择逻辑从 **`busy` → `overload`** 迁移（包括 KV‑router 的 `ActiveLoad`、`SchedulerQueue`、`WorkerSelector`、`PushRouter`）。  
3. 引入 `RoutingInstanceCounts` 与 `RoutingInstancesState`，统一管理 **已发现、可路由、已 overload、空闲** 四类实例快照，取代原来的 `instance_avail` / `instance_busy` 两条 `ArcSwap`。  
4. 错误模型改为 `KvSchedulerError::AllEligibleWorkersOverloaded` / `PinnedWorkerOverloaded`，并在 `kv_router::find_best_match` 中映射为 `DynamoError(ErrorType::ResourceExhausted)`，实现 **对外统一的资源耗尽错误**。  
5. 相关单测、文档、注释同步更新，加入 **overload‑aware** 的调度与路由单元测试。

## 🔍 技术洞察

### 1. 架构影响
| 维度 | 变更描述 | 正面效应 | 潜在风险 |
|------|----------|----------|----------|
| **路由实例管理** | 引入 `RoutingInstancesState`（immutable快照 + watch 机制）取代两条独立 `ArcSwap`，统一管理 discovered / routable / overloaded / free 集合。 | - 单一来源真相，避免 **instance_avail 与 instance_busy 不一致** 的竞态。<br>- `free = discovered - overloaded` 自动推导，降低手工维护错误。<br>- 更易扩展（如 future 需要 **draining**、**maintenance** 状态）。 | - 迁移期间若有旧代码仍直接访问 `instance_avail`/`instance_busy`（已删除但可能在外部 crate 中仍有引用），会导致编译错误或运行时 panic。<br>- `ArcSwap` 取代为 `RoutingInstancesState`，内部锁 `StdMutex`，在极高并发下可能产生短暂的排队（但锁粒度极小，仅更新快照时）。 |
| **调度器 (KvScheduler)** | 新增 `OverloadedWorkerProvider` 参数，所有调度入口（`LocalScheduler::new_with_overload_provider`、`SchedulerQueue::new_with_overload_provider`、`KvScheduler::start`）接受该闭包。 | - Scheduler 只需要 *读取* overload 信息，无需直接操作 `Client`，实现 **职责分离**。<br>- 业务方（如自定义 selector）可自行提供 overload 策略（如外部熔断、机器学习预测）。 | - 需要确保 provider 的闭包在调用时不产生阻塞或 panic；否则会导致调度器整个卡死。 |
| **Worker 选择器** | `WorkerSelector::select_worker` 增加 `RoutingEligibility` 参数，内部判断 **pinned / allowed / overloaded**、兼容 **taint**。 | - 过载检测与 **taint/allowed** 组合判定统一，避免旧代码中手动过滤导致遗漏。<br>- 对 `pinned` worker 的 overload 判断提升到调度层，错误信息更明确。 | - 旧实现的 `DefaultWorkerSelector` 被扩展，增加代码路径；若自定义实现未同步更新签名会导致编译错误。 |
| **异常映射** | `KvSchedulerError::AllEligibleWorkersOverloaded` → `PipelineError::ServiceOverloaded` → `DynamoError(ErrorType::ResourceExhausted)`。 | - 对外统一 **429/503 → ResourceExhausted** 语义，调用方可以统一捕获并进行重试或降级。<br>- 更明确的错误来源（`overloaded` vs `busy`），便于监控告警。 | - 需要保证所有上层 API（Chat, Completion 等）在错误路径上已处理 `ResourceExhausted`，否则可能出现未捕获的 panic。 |
| **监控与阈值** | `LoadThresholdConfig` 注释、字段改为 “overload detection”。`DecodeBusyLatchState` → `DecodeOverloadLatchState`、对应函数改名。 | - 与业务需求对齐（**阈值触发后不接受新请求**），而非仅做 **忙碌检测**（只用于调度内部）。<br>- 代码可读性提升，避免 “busy” 与 “overload” 混用导致误解。 | - 仍保留 `active_decode_blocks_threshold` 等阈值的 **OR** 逻辑，错误配置仍可能导致所有 worker 永远标记 overloaded（业务不可用）。 |
| **PushRouter** | `client.instance_ids_free()` → `client.routing_instance_counts().free`，并在路由前校验 **selected worker 是否 overloaded**（而不是检查所有 worker 是否 busy）。 | - 只在真正选中的实例上检查 overload，避免 **全局 busy 检查的“全体阻塞”** 误判。<br>- 通过 `routing_instances.is_overloaded` 判定，保持与调度层统一。 | - 需要确保 `fault_detection_enabled` 标记在所有调用点保持一致，否则可能出现 **故障检测关闭** 时仍返回 overload 错误。 |

### 2. 性能影响
| 方面 | 预期影响 | 说明 |
|------|----------|------|
| **快照更新** | **轻微增加**：每次实例列表变化或 overload 状态变化会生成新的 `Arc<RoutingInstances>`（约 2–3 个 `Vec<u64>` + `HashSet<u64>`），并在 `ArcSwap` 中交换。| 该操作频率受实例发现（几秒）和 overload 报告（阈值触发）控制，内存占用极低，且 `ArcSwap` 读取成本为原子指针读取，几乎无竞争。 |
| **调度路径** | **微增**：在 `select_worker` 中额外检查 `overloaded_worker_ids`（HashSet）是否包含 worker。| `HashSet` 查找 O(1) 并在常规调度分支中完成，整体延迟增加 < 1 µs，几乎不影响总体吞吐。 |
| **路由路径 (PushRouter)** | **无显著变化**：只在实例选择后额外做一次 `is_overloaded` 判断。| 同上，开销极小。 |
| **监控任务** | `KvWorkerMonitor` 仍在后台轮询 `ActiveLoad`，但现在更新 **overload_instances** 而非 **busy_instances**。| 逻辑基本相同，额外一次 `client.set_overloaded_instances`（一次 `HashSet` 替换）。 |

### 3. 安全/可靠性考虑
- **错误映射**：将 overload 错误统一为 `ResourceExhausted`（HTTP 429/503），符合 **幂等** 与 **可重试** 语义，减少误报 “内部错误”。  
- **状态一致性**：`RoutingInstancesState` 通过 **单一写锁** (`StdMutex`) 保证快照更新的原子性，避免 **instance_avail 与 instance_busy** 不同步导致的 **请求路由到 down 的实例**。  
- **防止过载蔓延**：`set_overloaded_instances` 仅在实际变化时返回 `true`，从而避免不必要的 watch 通知，降低 **广播风暴**。  
- **闭包安全**：`OverloadedWorkerProvider` 必须是 `Send + Sync + 'static`，且返回 `Option<HashSet<WorkerId>>`

### feat: Add first-class context metadata propagation (#9662)
**SHA**: `2bd2fb0` | 🔗 [查看提交](https://github.com/ai-dynamo/dynamo/commit/2bd2fb04543fce30143d9a5470280f8d4e06bec7)

**🎯 变更类型**：功能增强（首次引入一等的 Context Metadata 传播）  
**⚡ 重要程度**：🔴 高  

**📋 变更摘要**  
1. 为整个 Dynamo 运行时引入 `metadata: BTreeMap<String, String>`，在 `Context`、`GenerateContext`、`AsyncEngineContext`、网络控制消息等结构体中新增字段并提供相应构造函数、访问器与转移/合并逻辑。  
2. 在 Python 绑定层实现了 `ContextMetadata` 类，使用户在 Python 端能够像字典一样读取、写入、删除、迭代、拷贝元数据，并将其在跨语言调用中实时同步。  
3. 修改了请求/响应流的创建、路由、重试、迁移、预填充等代码路径，使子请求在创建时自动继承父请求的 metadata 快照；在 `abort`、`generate` 回调中保留原始快照供业务使用。  
4. 在网络层的控制消息（`RequestControlMessage`）中加入 `metadata` 字段，并对其序列化大小进行上限检查（128 KB）。  
5. 添加了针对元数据传播的单元/集成测试，验证 Python 端的可变映射行为以及在取消、迁移等场景下的元数据保持。  

**🎯 影响范围**  
- `lib/runtime/src/pipeline/context.rs`、`engine.rs`、`adapter.rs`、`migration.rs`、`prefill_router` 等核心后端模块。  
- Python 绑定层：`lib/bindings/python/rust/context.rs`、`backend.rs`、`engine.rs`、`lib.rs`。  
- 网络层控制消息：`runtime/src/pipeline/network/*.rs`（Ingress/ Egress、Addressed Router）。  
- 相关测试：`bindings/python/tests/*`、`runtime/src/pipeline/context.rs` 单元测试。  

**🔍 技术洞察**  

| 维度 | 影响 |
|------|------|
| **架构影响** | - **上下文统一化**：所有请求的 `Context` 现在携带同一套 metadata，实现了 *传递链*（父 → 子 → 重试 → 迁移）的一致快照。<br>- **API 兼容性**：新增 `with_id_and_metadata`、`metadata()`/`metadata_mut()` 等构造函数和 getter，所有调用点已改为新构造函数，保持向后兼容（旧代码仍能使用 `with_id`，但不再自动继承元数据）。<br>- **跨语言边界**：`ContextMetadata` 通过 `Arc<Mutex<_>>` 在 Rust↔Python 之间共享，保证同一个 `Context` 实例在 Python 中的修改即时可见于后端逻辑。 |
| **性能影响** | - **内存开销**：每个 `Context` 额外持有一个 `BTreeMap`（平均键值对数预计在十几条以内），对内存占用几乎可忽略。<br>- **序列化成本**：控制消息在网络层加入了 `metadata`，序列化使用 `serde_json::to_vec`，在大多数请求（metadata < 1 KB）影响不大。为防止异常增长，添加了 128 KB 上限检查。<br>- **锁竞争**：Python 端的 `ContextMetadata` 使用 `Arc<Mutex<_>>`，在高并发请求下可能产生轻微的互斥开销，但每次访问都只锁定极短时间（仅读/写单个键），对整体吞吐影响有限。 |
| **安全考虑** | - **信息泄露风险**：metadata 可任意携带用户自定义键值，若包含敏感信息（tokens、密码）且未加过滤，可能通过日志、trace、或跨服务转发泄露。建议在入口处对敏感前缀（如 `auth_`, `secret_`）进行审计或加密。<br>- **注入与 DoS**：未经限制的键值大小会导致控制消息序列化失败或触发上限检查，可能被恶意客户端利用产生大量错误。已加入大小上限检查，并在错误路径返回明确错误。<br>- **并发安全**：`Arc<Mutex>` 在 Python 端提供了线程安全，但若在 Rust 端的 `Context::metadata_mut` 同时被多个 async 任务持有，仍需遵守 `!Sync` 的使用约束；当前实现仅在单线程/任务链中使用，风险可控。 |
| **可维护性** | - **代码重复度降低**：统一的 `Context::with_id_and_metadata` 与 `metadata()` 方法取代了散落的手动拷贝逻辑。<br>- **测试覆盖**：新增 Python 与 Rust 单元测试覆盖元数据的读写、异常路径以及跨层传递，提升了未来修改的安全网。<br>- **文档更新**：`_core.pyi` 中加入 `ContextMetadata` 接口说明，帮助使用者快速了解新特性。 |

**⚠️ 潜在风险**  
1. **元数据尺寸失控**：若上层业务不限制键值长度，可能触达 128 KB 上限导致请求被拒绝，或引发频繁序列化错误。  
2. **跨服务传播**：在多机部署（NATS、TCP）中，metadata 会随请求在每个节点复制，增加网络流量，且可能在日志收集系统中出现大量冗余字段。  
3. **并发锁竞争**：Python 端的 `ContextMetadata` 使用全局 `Mutex`，在极端高并发（数千并发请求共享同一 Context 对象）下可能出现吞吐瓶颈。  
4. **错误处理链路**：在 `abort` 回调中仍然只能读取原始快照（不可变），如果业务期望在取消后获取最新元数据，可能产生误解。  
5. **向后兼容**：外部插件或第三方 Rust 扩展仍可能调用 `Context::with_id` 并忘记显式传递 metadata，导致元数据在该分支丢失。  

**💡 关注建议**  
- **规范元数据使用**：建议在项目文档中明确元数据的目的（如租户标识、追踪标签），并约定键名、最大长度（如 ≤256 B）。在入口（HTTP、gRPC）做一次白名单过滤或加密处理。  
- **监控与限流**：在 `AddressedRouter` 序列化阶段记录 `metadata.len()` 与序列化大小的 Histogram，监控异常增长并在监控阈值触发时报警。  
- **性能基准**：在高并发基准（≥10k QPS）下进行一次压测，关注 `metadata` 锁竞争、JSON 序列化耗时以及控制消息的网络带宽占比。若发现争用，可考虑使用 `RwLock` 或 `dashmap` 替代。  
- **错误信息统一**：为 `ControlMessage` 序列化错误添加错误码（如 `MetadataTooLarge`），便于上层 SDK 做重试或降级。  
- **迁移路线**：对已存在的第三方插件，提供适配层（例如在 `Context::with_id` 中自动把 `metadata` 设为空）并在发布日志中标注即将废弃。  
- **安全审计**：在 CI 中加入 `cargo audit`、`cargo deny` 检查，并在安全审计阶段重点审查元数据的日志打印路径，避免泄漏。  

> 总结：本次提交为 Dynamo 引入了首级的 **Context Metadata 传播** 能力，提升了跨语言、跨请求链路的可观测性与业务灵活性。实现方式在保持原有功能的同时对结构体、网络协议、Python 绑定均做了统一扩展。只要在使用上遵循元数据大小与敏感信息的最佳实践，风险可控，收益显著。

**SHA**: `828abba` | 🔗 [查看提交](https://github.com/ai-dynamo/dynamo/commit/828abbaca7cfdea133ab9bad58c49191b4f38f46)

**🎯 影响范围**：

- `components/src/dynamo/vllm/multimodal_handlers/encode_worker_handler.py`（多模态编码工作流）

- 相关的测试与示例脚本（若使用 XPU）  

**🔍 技术洞察**  

- **架构影响**  
  - 在 NIXL 接口层引入了统一的 `nixl_mem_type`（`DRAM` / `VRAM`），使得不同设备（CPU、CUDA、XPU）可以共享同一套 descriptor 生成逻辑，提升了代码可维护性。  
  - `DeviceKind` 新增 XPU 并实现 `__str__` 与 `nixl_mem_type`，对上层 device 解析（`validate_device`）做了同步更新，保证配置解析一致性。  
  - 通过在 `EmbeddingTransfer.send_embeddings` 中捕获 NIXL 注册异常并回退到 CPU staging，防止因 XPU 特有的注册缺陷导致整个推理任务崩溃，提升了系统容错性。  

- **性能影响**  
  - 对 XPU 张量在 NIXL 注册前强制 `contiguous()`，避免因非连续内存导致的额外拷贝或注册错误。  
  - 在 XPU 路径加入 `torch.xpu.synchronize()`，确保前置计算 kernel 完成后才进行 NIXL 传输，消除潜在的隐式同步开销，整体延迟预计略有上升（同步本身成本），但能够避免数据竞争导致的错误。  
  - 回退 CPU staging 会产生跨设备拷贝，只有在 NIXL 注册失败的极端情况下触发，对常规路径的性能影响可忽略。  

- **安全考虑**  
  - 新增异常捕获与日志打印，能够在 NIXL 注册失败时记录设备类型与异常信息，便于运维审计。  
  - 由于在异常路径中将张量拷贝到 CPU，避免了在不受信任的 XPU 内存上直接进行跨进程共享，降低了潜在的内存泄露风险。  
  - 代码对 `metadata` 解析加入了 XPU 前缀检查，防止非法字符串进入后续的设备 ID 解析阶段，引入了轻量级的输入校验。  

**⚠️ 潜在风险**  

1. **同步点引入的阻塞**  
   - `torch.xpu.synchronize()` 会导致 CPU 端等待所有 XPU kernel 完成，可能在高并发推理场景下形成瓶颈。建议在性能基准测试中评估同步频率。  

2. **CPU 回退的内存占用**  
   - 当 XPU 注册失败时，张量会被复制到 CPU，临时增加 DRAM 消耗，若在大批量嵌入转移时触发，可能导致 OOM。  

3. **兼容性**  
   - 代码假设 `nixl` 库在 XPU 环境下仍接受 `VRAM` 作为 mem_type；若后续 NIXL 版本更改对应名称，需要同步更新 `nixl_mem_type` 实现。  

4. **连续性强制复制**  
   - 对非连续张量强制 `contiguous()` 可能在极端情况下产生隐式的大块拷贝，需要确认上层调用方对张量布局的预期。  

**💡 关注建议**  

- **性能基准**：在真实 XPU 硬件上跑一次完整的多模态推理基准，比较加入 `synchronize` 前后的端到端延迟，决定是否将同步移到更靠近 NIXL 调用的内部实现。  
- **回退监控**：在生产环境中打开 `logger.warning` 对象的监控报警，及时捕获 NIXL 注册失败的频率，防止回退路径的频繁触发。  
- **内存监控**：监控 CPU DRAM 使用，尤其是在高并发或大 batch 场景下，确保回退不会导致 OOM。必要时可以在异常回退前加上内存检查或提前预分配。  
- **文档更新**：在项目的设备配置文档中加入 `xpu:<device_id>` 示例，并说明 `validate_device` 现在已支持 XPU。  
- **测试覆盖**：添加针对 XPU 的单元测试，验证：  
  1. `DeviceKind.XPU` 的字符串、`nixl_mem_type` 正确；  
  2. `EmbeddingTransfer` 在 XPU 上正常走 NIXL 路径并在异常时回退；  
  3. `torch.xpu.synchronize` 在多线程环境下不产生死锁。

1. **并发 Patch 冲突**：在高频率调和期间，若同一 LWS 同时被多个控制循环 PATCH，可能产生 `ResourceVersion` 冲突导致补丁失败。

3. **外部依赖硬编码名称**：外部监控、日志或脚本硬编码了旧的 LWS 名称（`<dcd‑name>`），升级后需要同步更新，否则监控告警失效。  
4. **Service 名称冲突仍可能出现**：若用户自行在同一命名空间创建了与 LWS 名称相同的 Service，冲突仍会发生（算是使用者误用）。

1. **使用乐观并发控制**：在 Patch 前读取最新的 `ResourceVersion`，并在冲突时重试（如使用 `client.Patch` 的 `PatchOptions{}` 结合 `RetryOnConflict`）。  
2. **升级文档与迁移指南**：在 Release Notes 明确说明 LWS 名称已改为 `<dcd‑name>-0`，并提供脚本示例帮助用户搜索并更新自定义监控/告警配置。

4. **回滚保护**：如果在极少数场景下出现删除误判导致服务不可用，可在 `reconcileLeaderWorkerSetResources` 中加入 **保留阈值**（如仅在 `replicas > 1` 时才删除）或提供 `--disable-legacy-cleanup` 标志供运维手动关闭。  
5. **兼容性测试**：在 CI 中保持对 **旧版（v1.1.x）** 集群的升级路径测试，确保即使 LWS 未携带 `instance-id`，也能顺利迁移或发出警告。

**结论**：本次修改通过统一 LWS 命名和清理旧标签，有效防止了 Operator 自动生成的 Service 与 LWS 之间的命名冲突，提升了系统的稳定性和可维护性。风险主要集中在并发 Patch 与外部依赖硬编码名称上，建议在实际部署后监控相关事件并提供文档迁移指引。

### fix(ci): wait for vCluster pod Ready before port-forward (#9638)
**SHA**: `ed9513a` | 🔗 [查看提交](https://github.com/ai-dynamo/dynamo/commit/ed9513a1d26b3b985eb2867f871c9f251ed995f8)

**🎯 变更类型**：Bug 修复（CI vCluster 连接不可靠）  
**⚡ 重要程度**：🟡 中  
**📋 变更摘要**：  
1. 在 GitHub Action `connect-vcluster` 中，原先直接 `kubectl port‑forward … && sleep 5`，在 vCluster Pod 仍为 Pending 或刚重启时会导致端口转发立即退出，后续 `vcluster connect` 失败。  
2. 新增检测已有 127.0.0.1:8443 转发的逻辑，若已存在则复用；否则先 `kubectl wait --for=condition=ready pod` 等待 Pod Ready，再启动 port‑forward。  
3. 用循环检测本地端口是否真正可达，替代固定 `sleep 5`，并在超时或转发进程异常时直接报错。  
4. `base64 -w 0` 改为通用写法 `base64 | tr -d '\n'`，避免在不同平台（macOS、alpine）上的兼容性问题。  

**🎯 影响范围**  
- `.github/actions/connect-vcluster/action.yml`（CI 步骤脚本）  
- 依赖此 Action 的所有工作流（如 `deploy-operator`、自举脚本）  

**💡 关注建议**  
- **CI 可靠性提升**：等待 Pod Ready 与端口可达后再继续，可显著降低因节点资源紧张或短暂网络波动导致的 flaky。建议在本地或自托管 runner 上跑一次完整工作流，确认 300 s 超时足够。  
- **kubectl 版本**：`kubectl wait --for=create …` 在 1.31 之后才支持组合等待；如果未来的工作流在更老的集群中运行，请保留注释中的备选方案（先 `kubectl get pod` 轮询或 `kubectl rollout status`）。  
- **并发安全**：复用已有 port‑forward 通过 `curl` 检测，避免 “address already in use”。若后续有并行多个 vCluster 实例，需确保端口映射唯一（可改用随机本机端口）。  
- **错误信息**：新增 `::error::` 标记，在 GitHub Actions 日志里更易定位端口转发失败的根因。  
- **兼容性测试**：由于改用了 `base64` 的管道方式，请在 macOS、Ubuntu、Alpine 三类 runner 上分别跑一次，确保没有隐藏的字符换行问题。  

总体来看，此次修复对 CI 稳定性贡献明显，风险主要在对老版 `kubectl` 的兼容性和可能的并发端口冲突，建议在发布前做一次跨平台回归测试。
