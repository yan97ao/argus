### feat: bidirectional request-plane dispatch + runtime-side ingress (#9674)
**SHA**: `0658f05` | 🔗 [查看提交](https://github.com/ai-dynamo/dynamo/commit/0658f05913f40cee2fb4a3913d83ef93ca3d7cac)

**🎯 变更类型**：功能增强（Bidirectional request‑plane dispatch）  
**⚡ 重要程度**：🔴高  

**📋 变更摘要**  
- 引入 **双向流式请求**（ManyIn → ManyOut）能力，支持 OpenAI Realtime API 等需要把客户端输入流持续写入后端的场景。  
- 新增 `RequestStream<T>`（同步化的 `Mutex<Option<DataStream<T>>>`）用于安全地一次性取出输入流，防止多次 `take()` 导致数据泄漏。  
- 扩展协议层：`RequestControlMessage` 现在携带 `request_stream_connection_info`，在 bidirectional 调度时返回 worker 用来回拨的连接信息。  
- 重构 `AsyncEngine`、`AsyncEngineStream`、`ManyIn`、`ManyOut` 的定义，使输入侧与输出侧在类型上对称、更加文档化。  
- 完整实现 **bidirectional** 端到端路径：  
  - `PushRouter` 在调度阶段即可选定实例，随后通过 `AddressedPushRouter::generate_bidirectional` 完成请求‑响应流的建立和转发。  
  - `Ingress` 为 `ManyIn` 添加专门的解析实现，负责解码 header‑only envelope、创建回拨请求流并把收到的帧反序列化后送入 `RequestStream`。  
- 大量错误处理、监控指标、故障检测（overload、超时）迁移到统一函数，避免代码重复。  
- 新增单元与端到端测试（`bidirectional_e2e.rs`），覆盖驱动、路由、流转发、错误路径。  

**🎯 影响范围**  
- `lib/runtime/src/engine.rs`、`pipeline.rs`、`pipeline/network/*`（egress、ingress）  
- `lib/llm/src/engines.rs`、`lib/llm/src/http/service/realtime.rs`（Realtime API）  
- 所有使用 `AsyncEngine<ManyIn<…>, ManyOut<…>>` 的插件/模型（例如 OpenAI Realtime、后续自定义流式模型）  
- 依赖 `ResponseStream` 的外部代码：原本的 `RequestStream` 类型别名已被替换为结构体，可能需要迁移。

### fix(frontend): return HTTP 415 for unsupported image formats instead of 500 (#10281)
**SHA**: `857ad3a` | 🔗 [查看提交](https://github.com/ai-dynamo/dynamo/commit/857ad3a287542a9463087871b201ba737af5ac53)

**🎯 变更类型**：Bug修复  
**⚡ 重要程度**：🔴高  
**📋 变更摘要**：  
1. 在 `ImageLoader` 中捕获 `PIL.Image.UnidentifiedImageError`（以及部分 `ValueError`）并统一抛出 `HttpStatusError(415, "Unsupported Media Type")`，使前端在遇到不支持的图片格式（如 SVG）时返回 415 而不是 500。  
2. 为批量加载 `load_image_batch` 添加对 `HttpStatusError` 的透传逻辑，确保异常聚合后仍能保留客户端错误码。  
3. 新增四个单元测试，验证 URL 与 data‑URL 两种路径以及单张与批量加载时均能得到 415。

**🎯 影响范围**：  
- `components/src/dynamo/common/multimodal/image_loader.py`（核心图片加载模块）  
- 前端调用 `load_image` / `load_image_batch` 的所有服务入口（如文本‑图像多模态 API）  
- 单元测试套件 `components/src/dynamo/common/tests/multimodal/test_image_loader.py`

**🔍 技术洞察**  

- **架构影响**  
  - 通过在 `ImageLoader` 层统一抛出 `HttpStatusError`，错误语义从 “内部服务器错误” 上升到 “客户端错误”。这让上层（API 路由或网关）能够直接转译为正确的 HTTP 状态码，减少了在控制器层进行额外错误映射的需求。  
  - `load_image_batch` 现在在聚合异常时会先检查是否出现 `HttpStatusError`，若有则直接传播该异常，保持 **异常语义的首要性**，不再让后续 `Exception` 包装丢失状态码。  

- **性能影响**  
  - 只在异常分支中额外执行几次 `isinstance` 检查和日志记录，正常路径的性能几乎不受影响。  
  - 批量加载的异常聚合逻辑仍使用 `asyncio.gather(..., return_exceptions=True)`，保持原有并发度；新增的 `status_error` 变量对内存开销可忽略不计。  

- **安全考虑**  
  - 将内部异常转换为明确的 415 状态码，避免因意外的 500 泄露实现细节（如堆栈信息），对安全是一次正向强化。  
  - 错误信息只包含图片 URL，未暴露原始异常类型，符合最小信息披露原则。  

**⚠️ 潜在风险**  

1. **兼容性**：已有调用方（尤其是内部服务）可能仅捕获通用 `Exception`，而不专门处理 `HttpStatusError`。在批量加载时，如果之前依赖统一的 `Exception` 包装来记录错误日志，现改为直接抛出 415 可能导致这些调用方失去对错误的统一处理。  
2. **异常掩盖**：在 `load_image_batch` 中，仅保留 *第一次* 捕获到的 `HttpStatusError`，其余的 415 错误会被忽略并合并进 `collective_exceptions`（如果后面还有其他异常）。这在批量请求混合错误（部分 415、部分其它）时，最终返回的异常可能只反映第一个 415，导致错误信息不完整。  
3. **误判**：对 `ValueError` 的字符串匹配 `"Unsupported image format"` 依赖实现细节，若底层库修改错误消息格式，可能导致该分支失效，仍返回 500。  

**💡 关注建议**  

- **文档/合同**：在模块文档中明确 `load_image` 与 `load_image_batch` 在遇到不支持的媒体类型时会抛出 `HttpStatusError(415)`，并建议调用方捕获该异常以返回对应的 HTTP 响应。  
- **兼容层**：若有旧版服务仍期待通用 `Exception`，可以在调用层做一次适配：`except HttpStatusError as e: raise e`，`except Exception as e: ...`，保持向后兼容。  
- **测试覆盖**：补充 **混合批量** 场景（如 `[支持图片, 不支持图片, 超时]`），验证在出现多种错误时：
  1. 第一个 `HttpStatusError(415)` 被直接抛出，  
  2. 其余错误仍被记录在 `collective_exceptions` 中（或按照业务需求统一包装）。  
- **错误匹配稳健性**：考虑将 `Unsupported image format` 的检测抽象为一个专用异常（例如自定义 `UnsupportedMediaError`），而不是依赖字符串匹配。这样即使底层库更新也能保持行为一致。

| **性能影响** | - **正面**：防止因 oversized 请求阻塞 `max_in_flight` 或 waiter，释放 slot，避免离线/实时 replay 死锁，提升整体吞吐和 latency 稳定性。<br>- **负面**：在 admission gate 增加一次 `to_completion_blocks` 计算（一次除法 + ceil），开销极低；额外的 `rejected` 标记和判断在各 runtime 中几乎为 O(1)。整体性能提升预计 > 95% 的场景中不受影响，且在出现阻塞时能立即恢复。 |
| **安全考虑** | - 仅涉及内部调度与统计，不暴露外部接口。<br>- 新增日志 `rejecting TRT-LLM request whose footprint exceeds the entire KV pool`，便于审计与问题追踪。<br>- 未引入外部资源访问或权限变更，安全风险可视为 **无**。 |
| **可维护性** | - 新增字段 `rejected` 与方法 `to_completion_blocks` 均有完整文档注释，代码路径清晰。<br>- 所有受影响的 runtime 均已加入对应单元测试，降低回归风险。<br>- 通过 `OutputSignal.rejected` 的显式语义，后续若需扩展其他终止原因（如 quota、policy）可直接在该字段或结构体中扩展。 |
| **兼容性** | - 只对 **TRT‑LLM** 引擎生效，其他 engine_type 走旧路径，保持向后兼容。<br>- 老版 `OutputSignal` 在序列化/反序列化时默认 `rejected = false`（`serde(default)`），不会破坏已有离线回放 trace。 |

---

### ⚠️ 潜在风险
1. **误判拒绝**  
   - `to_completion_blocks` 采用 **未折扣的** 总块数（`prompt + max_output`），若调度器后续在 **prefix‑cache** 场景中实现更细粒度的折扣逻辑，可能导致本判断过于保守，拒绝本可运行的请求。需要确保该判断始终基于 **全局物理 footprint**。  

2. **统计偏差**  
   - Rejected 请求已从流量统计 (`traffic.on_request`) 中排除，但如果外部监控仍基于 `num_requests` 与 `completed_requests` 的比值，可能出现 “完成率异常下降” 的误解。建议在监控层面加入 `rejected_requests` 指标。  

3. **遗漏信号传播**  
   - 代码在 `emit_ready_tokens` 之后手动 push `rejected` 信号；若未来在该函数内部再次拆分或重构，可能忘记同步添加该逻辑，导致某些 runtime 未收到终止信号。可考虑将该包装抽象为统一的 `emit_terminal_signal(uuid, rejected)` 函数。  

4. **多worker 环境下的 race**  
   - `rejected_uuids` 在 `execute_pass_internal` 中收集后统一推入 `output_signals`，但在分布式 `disagg` 场景中仍有两阶段的 prefill → decode 路由。需确认 `prefill_router` 不会在 `on_request_completed` 前再次调度同一 UUID，防止 double‑reject。已有测试覆盖，但在高并发/不确定的 worker 数量下仍需持续监控。  

---

### 💡 关注建议
- **监控 & 报警**：在 `trtllm_no_evict` 路径增加 `rejected_requests_total` 计数，配合现有 `vllm_preemptions_total`，便于观察是否出现异常大量拒绝。  
- **文档更新**：说明 `OutputSignal.rejected` 的语义及在 `trace` 文件中的默认值；在 `MockEngineArgs` 示例中标注 “no‑evict policy”。  
- **回归测试**：在 CI 中保留四种 runtime（offline‑agg, offline‑disagg, online, python‑replay）对应的 rejection 场景，防止未来改动破坏终止信号的传递。  
- **对外 API 稳定性**：若项目向外部提供 `OutputSignal`（如 Python binding），应在协议层明确 `rejected` 为 **可选**（默认 false），避免旧客户端解析错误。  
- **后续扩展**：可以把 `rejected` 与 **拒绝原因码**（enum）抽象出来，以支持未来如 “quota‑exceeded”“policy‑blocked”等多种终止原因，保持 API 向前兼容。  

---  

**结论**：本次功能增强在调度层面消除了因 oversized TRT‑LLM 请求导致的离线/实时 replay 死锁，架构改动集中且可追踪，性能与安全影响可忽略不计。唯一需要重点关注的是 **误判** 与 **统计/监控** 的同步更新。建议尽快合并并在生产环境开启 **no‑evict** 模式下的监控，以验证实际拒绝率与业务预期的一致性。

  - **CleanupImage** 必须提供 `/bin/sh` 与 `rm`，默认使用 `busybox:1.36`（官方镜像，已签名）。用户可自行覆盖，若使用非可信镜像可能导致任意文件删除风险。建议在文档中强调使用受信任的镜像或在企业内部镜像仓库中复制。  
  - **Job RBAC**：Controller 为 Job 设置了 OwnerReference，Job 只具备 `Pod` 权限（默认的 ServiceAccount）。若 `cleanupImage` 被恶意替换，可能执行任意 shell 命令，故应通过 Admission 控制或网络策略限制 Job 使用的 ServiceAccount。  
  - **Finalizer 操作**：在删除 checkpoint 前必须成功执行清理 Job，否则会导致资源卡死在 finalizer 状态。错误处理已实现重试机制，但仍需监控 `errCheckpointCleanupPending` 场景。  

- **可维护性**  
  - **CRD 增量**：DeletionPolicy 与 CleanupImage 均标记了 `+kubebuilder:default` 与 `enum`，生成的 OpenAPI schema 完整，向后兼容。  
  - **代码分层清晰**：checkpoint 创建、更新、清理逻辑分别放在 `internal/checkpoint` 与 `controller`，单元测试覆盖率提升（>100% 新增测试）。  
  - **文档同步**：API reference 与 Helm README 已同步，降低用户使用错误概率。  

**⚠️ 潜在风险**  

1. **清理 Job 失败导致资源泄漏**：若 PVC 不可写、镜像拉取失败或节点资源不足，Job 可能卡在 `Pending/Running`，导致 finalizer 永久保留，阻塞 DGD 删除。  
2. **Retention 误用**：Retain 政策会把 checkpoint **永久保留**，若用户忘记手动清理，会占用大量 PVC 空间。  
3. **镜像安全**：`cleanupImage` 可被用户任意覆盖，若误指定恶意镜像，可能在 PVC 中执行任意删除或写入操作。  
4. **OwnerReference 稍后变更**：在 Retain 场景下自动清除 OwnerReference，若后续还有业务依赖该关联（如监控/归档工具），会失去追踪源。  
5. **并发删除冲突**：多个 DGD 同时删除同一 PVC（跨命名空间）时，可能出现同一 checkpoint 的 Cleanup Job 并发执行，需要确保 Job 的 ID 唯一（已通过 checkpoint‑ID 作为 Job 名的一部分保证）。  

**💡 关注建议**  

- **运维监控**：在监控平台上添加 `checkpoint-cleanup-<id>` Job 的状态告警（Pending/Failed），并在 DGD 删除后检查对应 checkpoint 是否仍残留。  
- **清理镜像安全**：建议在 Operator Config 中加入 `cleanupImagePolicy`（如镜像签名校验或固定白名单），或在 Helm Chart 中将默认镜像的签名 digest 锁定。  
- **容量预警**：对使用 Retain 策略的 checkpoint，开启 PVC 使用率告警，防止硬盘被长期占满。  
- **文档强化**：在官方文档中明确“Retain 只能在确认需要长久保存 checkpoint（如 Warm‑up、调试）时使用”，并提供手动删除的 `kubectl delete dynamocheckpoint <name>` 示例。  
- **回滚路径**：若因清理 Job 导致 finalizer 卡死，可提供临时 `kubectl patch dynamocheckpoint <name> -p '{"metadata":{"finalizers":null}}'` 方式手动移除，或在 Operator 中实现 `--force-cleanup` 标志。  
- **测试扩展**：在 CI 中加入跨命名空间、并发 DGD 删除场景的 e2e 测试，验证 Cleanup Job 的幂等性与资源回收完整性。  

总体来看，此次改动为 Dynamo 的 checkpoint 生命周期引入了更细粒度的资源管理能力，提升了可操作性与安全性，但也需要合理的运维监管与镜像安全把控，以避免因自动清理失效或 Retain 资源泄漏带来的成本与风险。

### perf(frontend): profile-guided hot-path optimizations (tokenizer cache, metric handles, log-trace cap) (#10273)
**SHA**: `2aab51b` | 🔗 [查看提交](https://github.com/ai-dynamo/dynamo/commit/2aab51b6a4a7605e420ca9d04076ef49329e30c1)

**📋 变更摘要**：

**🎯 影响范围**：  
- `lib/llm/src/http/service/metrics.rs`（请求响应度量收集器）  
- `lib/tokenizers/src/cache/l1.rs` 与相关的 `mod.rs`（Tokenizer 一级缓存）  
- 所有依赖 `ResponseMetricCollector` 与 `CachedTokenizer` 的前端请求路径，包括模型路由、推理流控制与日志/trace 系统。

**🔍 技术洞察**：

- **架构影响**  
  - **度量收集层**：将原本在 `observe_response` 热路径里每次 `with_label_values` 的标签解析搬到构造函数，保持 `ResponseMetricCollector` 生命周期为单请求。结构上不改变外部接口，仅在内部持有 `IntCounter`、`Histogram`、`Gauge` 的“已解析句柄”。  
  - **Tokenizer 缓存层**：把前缀 token 从 `Vec<u32>` 替换为 `Arc<[u32]>`，实现无复制共享；对外仍返回 `Vec<u32>`（在需要时自行 `to_vec`），对调用方透明。缓存逻辑、插入与失效仍由 moka 负责，架构保持不变。

- **性能影响**  
  - **度量收集**：消除每个 token（甚至每个 chunk）一次标签哈希 + `to_string`（worker id）成本，预计在高吞吐量（>10k token/s）场景下降低 CPU 使用 10%~30%。  
  - **Tokenizer 缓存**：避免前缀 token 的深拷贝与多次 `Vec` 扩容，尤其在长输入（数千 token）时可削减 2‑3 倍的内存复制量，降低 GC‑like 的内存碎片。   
  - **整体**：在 benchmark 中已加入针对缓存句柄和 L1Cache 的回归测试，确认计数、分布与之前一致，同时显著提升响应时延的尾部延迟。

- **安全考虑**  
  - 没有引入外部输入的解析或权限检查，仅是内部结构的复用与缓存。使用 `Arc` 确保跨线程安全，无悬挂指针风险。  
  - 需要注意 `decode_itl_gauge` 的 `set` 调用在并发环境下是线程安全的（Prometheus `Gauge` 本身使用原子操作）。

**⚠️ 潜在风险**：

1. **模型标签不变假设**：`ResponseMetricCollector` 假定同一请求期间模型名不变。若未来出现同请求中切换模型的场景（跨模型代理），缓存句柄会产生误报。  
2. **Gauge 持久化冲突**：`decode_itl_gauge` 使用全局 `WORKER_LAST_INTER_TOKEN_LATENCY_GAUGE`。如果同一 worker ID 在并发请求中被复用，多个请求的 `set` 可能交叉覆盖，导致监控数据出现抖动。  
3. **缓存占用**：`Arc<[TokenIdType]>` 持有的前缀 token 在缓存中保持引用，若缓存未及时淘汰，可能导致内存驻留时间比预期长，尤其在大量独特前缀的场景。  
4. **API 兼容**：部分内部调用仍期望 `Vec<TokenIdType>`，已通过 `.to_vec()` 包装，但若有外部 crate 直接依赖 `longest_prefix_match` 返回的 `Vec`（未更新），将产生编译错误。

**💡 关注建议**：

- **监控验证**：部署后对 Prometheus 指标进行对比验证，关注 `output_tokens_counter`、`time_to_first_token`、`inter_token_latency` 的采样率是否与预期一致。  
- **并发安全审计**：如果业务会在同一 worker 上并行处理多个请求，考虑在 `ResponseMetricCollector::observe_response` 结束后显式 `reset` 或使用 `Gauge::add/sub` 计数而非 `set`，避免跨请求覆盖。

- **文档与测试**：更新 `README` 中的性能说明，补全对 `Arc` 共享语义的解释；将现有单元测试扩展为并发场景（多线程同时创建 `ResponseMetricCollector`），确保没有竞态。
