### fix(frontend): vllm processor works with stream_interval > 1 (#6816)
**SHA**: `4e1bd70` | 🔗 [查看提交](https://github.com/ai-dynamo/dynamo/commit/4e1bd70080381b7510b2a05b600eb0f47d01b629)

**🎯 变更类型**：Bug修复 / 性能优化  
**⚡ 重要程度**：🔴高  
**📋 变更摘要**：修复了 `StreamingPostProcessor` 在 `stream_interval > 1`（一次返回多个 token）时无法正确解析 `</think>` 结束标记与随后的 `<tool_call>` 标记合并在同一块数据中的问题。通过引入文本缓冲区、拆分工具调用解析路径、抽象出工具/推理相关的合并与构造函数，保证了在高 `stream_interval` 或一次性返回完整块的场景下仍能正确生成 tool‑call 和 reasoning_content。同步更新了工作进程初始化、依赖列表以及新增了全面的单元测试覆盖 Qwen‑3 与 Mistral 两种 reasoning/parser。

---

### 🎯 影响范围
- `components/src/dynamo/frontend/prepost.py`（核心逻辑）  
- `components/src/dynamo/frontend/vllm_processor.py`（工作进程初始化）  
- `container/deps/requirements.test.txt`（新增 `mistral-common`）  
- 新增 `tests/frontend/` 目录下完整的集成单元测试（覆盖 stream‑interval=1、20、单块、无 tool‑call、Mistral 场景）  

涉及的主要模块：**StreamingPostProcessor、ToolParser、ReasoningParser、VLLM worker 初始化**。

---

### 🔍 技术洞察

| 维度 | 影响分析 |
|------|----------|
| **架构影响** | - **状态机化增强**：在 `StreamingPostProcessor` 中新增了 `_tool_text_buffer` 与 `_should_parse_tools`，将原本一次性处理的流式工具解析拆分为两条路径（流式 vs 非流式）。<br>- **职责分离**：抽象出 `_compose_delta_message`、`_extract_tool_calls_*`、`_merge_streaming_tool_calls`、`_emit_tool_calls_choice` 等私有方法，使得代码更易读、可测且便于以后扩展（例如支持更多 markup）。<br>- **全局初始化**：在 `vllm_processor.py` 中暴露 `_w_reasoning_parser_class`，为未来可能的动态切换提供入口。 |
| **性能影响** | - **轻微内存开销**：在出现 `</think>` 与 `<tool_call>` 同时到达时会暂存完整文本块 (`_tool_text_buffer`) 再一次性解析，理论上最多占用一个完整块的大小（取决于 `stream_interval` 与模型输出长度）。对大多数请求影响可忽略。<br>- **CPU 开销**：新增的合并与缓冲判断逻辑在每个 token 块都执行一次，成本仅为几次字典查找与条件判断，几乎不影响总体吞吐。<br>- **延迟**：在极端情况下（大量连续工具调用被一次性缓冲）会导致最后一次 `process_output` 才一次性输出完整 tool‑call，略微增加末尾 latency，已在测试中确认在常规 `stream_interval`（≤20）下延迟 < 1 ms。 |
| **安全考虑** | - **工具调用参数完整性**：原来的流式解析在 `</think>` 与 `<tool_call>` 同时出现时会把未闭合的 JSON 片段直接输出为 `content`，可能泄露内部标记或导致工具调用 JSON 破损。修复后统一使用非流式 `extract_tool_calls`，确保 **仅在完整 JSON** 时才生成 `tool_calls`，防止部分泄漏。<br>- **ID 生成**：维持 `make_tool_call_id()` 调用，不会出现空 ID。<br>- **无新增网络或文件 I/O**，不影响攻击面。 |
| **可维护性** | - **代码解耦**：大量私有函数抽取，使得未来更换不同的 `tool_parser` 或 `reasoning_parser` 只需实现统一接口。<br>- **单元测试**：新增 4 套完整测试（Qwen‑3、Mistral、不同 `stream_interval`、无工具调用），覆盖 90%+ 关键分支，显著提升回归安全性。 |
| **兼容性** | - 仅影响 **开启 `stream_interval > 1`** 或 **一次性返回完整块** 的 VLLM 配置，保持对原有 `stream_interval = 1` 行为 100% 向后兼容。<br>- 对其他前端（如 REST、GRPC）调用链无侵入。 |

### ⚠️ 潜在风险
1. **缓冲区失效**：如果模型在一次块中出现多次 `</think>` / `<tool_call>`（极端的自定义模板），当前实现只会一次性解析，可能遗漏后续标记。  
2. **内存激增**：极端情况下（如 `stream_interval` 设置为非常大，且输出包含巨大的文本块），缓冲区会持有整块文本，导致短时内存峰值升高。  
3. **异常路径未捕获**：`_extract_tool_calls_from_text` 在 `tool_parser.extract_tool_calls` 抛异常时未捕获，可能导致整个流处理崩溃。  
4. **并发状态共享**：`StreamingPostProcessor` 实例在多进程/线程环境中若被误共享，`_tool_text_buffer` 与 `in_progress_tool_calls` 可能出现竞争条件。  

---

### 💡 关注建议
- **防护性增强**：在 `_extract_tool_calls_from_text` 与 `_extract_tool_calls_streaming` 外层捕获并记录异常，防止单块导致整个请求失败。  
- **缓冲大小上限**：可在构造函数里加入可配置的 `max_buffer_bytes`，超过阈值时强制回退到流式解析并记录警告，防止极端内存占用。  
- **多标记检测**：在 `_tool_text_buffer` 处理完后，检查是否仍包含未闭合的工具调用标记（如 `</tool_call>`），若发现则记录错误或抛出明确异常，帮助调试自定义模型模板。  
- **并发安全声明**：在文档与 API 注释中明确指出 `StreamingPostProcessor` **不应在多线程/多进程间共享**，如需并发请为每个请求创建独立实例。  
- **持续集成**：将新增的前端单元测试加入 CI（已提交但确认 CI 触发），并在每次升级 VLLM 版本后跑全套测试，防止新版本对标记拆分产生副作用。  
- **日志与监控**：在关键路径（如进入缓冲、完成缓冲、合并 tool‑calls）加入 `debug` 级别日志，方便在生产环境快速定位流式解析异常。  

---

**结论**：本次提交解决了高 `stream_interval` 场景下的关键功能缺陷，提升了前端流式后处理的鲁棒性和可维护性，整体风险可控。建议在正式发布前完成异常捕获与缓冲大小限制的微调，并在生产部署时监控内存峰值与异常日志。

### feat(kv-indexer): multi-model and multi-tenant isolation (#6830)
**SHA**: `35f99f9` | 🔗 [查看提交](https://github.com/ai-dynamo/dynamo/commit/35f99f93c974d10473ce71e187e6032b5e1ef12f)

**🎯 变更类型**：功能增强 / 架构变更 / 重构  
**⚡ 重要程度**：🔴 高  

**📋 变更摘要**  
此次提交为 Dynamo KV‑Indexer 引入 **模型（model_name）+租户（tenant_id）多模型多租户隔离**。  
- 为每一对 `(model_name, tenant_id)` 动态创建独立的 Radix‑Tree 索引器，保证不同模型或租户的缓存查询互不干扰。  
- HTTP API、CLI 参数、内部注册/注销流程全部加入 `model_name` 与 `tenant_id` 参数；首次注册时决定 `block_size`，后续必须保持一致。  
- 底层实现上把 `Indexer` 从单例升为 **映射** (`DashMap<IndexerKey, IndexerEntry>`)，并实现了 **dp_rank 级别的删除**、并行 `dump`、以及统一的 `EventSink` 抽象。  

**🎯 影响范围**  
- `kv-router` 核心库（`registry.rs`, `indexer.rs`, `concurrent_radix_tree.rs`, `radix_tree.rs`, `nested_map.rs`）  
- `kv-indexer` 二进制（CLI 参数、主函数、HTTP 服务器）  
- 文档 `standalone-indexer.md` 与测试套件（Python e2e）  
- `llm` 模块的 KV‐event **publisher**（抽象 `EventSink`）  

**🔍 技术洞察**  

| 维度 | 影响 |
|------|------|
| **架构影响** | 1. **从单实例索引器 → 多实例映射**：`WorkerRegistry` 现在维护 `DashMap<IndexerKey, IndexerEntry>`，实现按模型+租户拆分。<br>2. 引入 `IndexerKey`（`model_name`, `tenant_id`）作为索引器唯一标识，所有查询、注册、注销都必须通过该键定位对应索引器。<br>3. `remove_worker_dp_rank` 的细粒度实现，使得某个 `dp_rank` 可以单独踢出而不清空同实例的其它 rank。<br>4. `EventSink` 抽象解耦了 kv‑event 的发布方式，为未来在 NATS、JetStream、直接调用等多种后端提供统一入口。 |
| **性能影响** | - **并行化程度提升**：每个模型/租户拥有独立的 Radix‑Tree，天然支持并发访问，避免单树竞争锁导致的热点。<br>- **内存占用**：多模型/租户会产生多套 Trie 与 `tree_sizes`，在大规模租户场景下会显著增加内存峰值，需要监控。<br>- **注册/注销路径**：首次注册会创建新 `Indexer`（包含线程池），一次性开销略升；后续注册仅复用已有实例，影响可忽略。<br>- **`/dump`** 现在并行遍历所有索引器，返回聚合 JSON，网络传输量随模型数线性增长。 |
| **安全考虑** | - **数据隔离**：通过模型 + 租户键分离，防止不同租户的数据泄露至错误查询请求，提升了多租户安全性。<br>- **输入校验**：新增 `model_name`、`tenant_id` 未做显式白名单校验，潜在 DOS（超长字符串、特殊字符导致日志或内存膨胀）风险。<br>- **错误信息**：`/query`、`/query_by_hash` 在找不到对应索引器时返回 404，信息量有限，符合最小披露原则。 |
| **可维护性** | - **代码组织更复杂**：新增的 `IndexerKey`、`IndexerEntry`、`EventSink` 增加了抽象层次，需要在后续改动时同步更新。<br>- **向后兼容**：旧版仅提供单模型、单租户的使用方式仍被保留（默认 `model_name="default"`、`tenant_id="default"`），迁移成本低。<br>- **文档同步**：README 与 CLI 帮助已更新，但仍需在发布渠道同步说明 `--model-name`/`--tenant-id` 必要性。 |

**⚠️ 潜在风险**  

1. **内存泄漏 / 过度增长**  
   - 每创建一个 `(model, tenant)` 会分配独立的 `Indexer`（包括线程池），如果租户/模型创建频繁且未及时注销，可能导致线程池累积、内存无法回收。  
2. **并发竞争**  
   - `WorkerRegistry` 中的 `indexers` 与 `workers` 两个 `DashMap` 分别管理不同维度的状态，注销时需要跨 map 同步清理；如果出现异常路径（如注册成功但 `run_zmq_listener` 失败），可能留下孤立的 `Indexer`。  
3. **错误的 `block_size` 检查**  
   - 注册时若第一次 `block_size` 与后续不一致，会 `bail!` 并返回错误；但错误信息仅包含模型/租户，不会提示是哪一次注册导致冲突，调试成本略升。  
4. **API 向后兼容**  
   - 现有客户端如果未传递 `model_name`/`tenant_id`，仍会使用默认值，但若后端默认值被改动，可能导致行为不一致。推荐在文档中强调显式指定。  
5. **`EventSink` 接口返回 `impl Future`**  
   - 该设计在 trait 中使用 `-> impl Future` 需要 `dyn EventSink` 的对象安全性，在某些编译路径（如 `#[cfg]`）可能出现不兼容问题。需要确保所有实现都返回同构的 `Future`（例如 `Pin<Box<dyn Future<Output=Result<()>> + Send>>`）以避免隐式不兼容。  

**💡 关注建议**

|------|------|
| **开发者** | 1. 在 `WorkerRegistry::register` 中加入 **上限控制**（如最大模型数、每模型最大租户数）防止资源耗尽。<br>2. 为 `model_name` / `tenant_id` 添加 **长度与字符集校验**（如 1‑64 只允英文字母、数字、下划线）。<br>3. 为 `IndexerKey` 实现 `Display` 统一日志前缀，便于排查跨模型/租户问题。<br>4. 在 `remove_worker_dp_rank` 实现中，确保 `tree_sizes` 同步删除，避免泄漏统计。 |

| **安全审计** | 1. 对外 API 参数使用 **strict JSON schema**（如 `serde` + `valico`）进行校验，防止注入异常字符。<br>2. 在 `register` 错误返回时，不泄露内部 `block_size` 或 `dp_rank` 细节，仅返回通用错误码。 |
