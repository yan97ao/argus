### feat(mocker): add KV cache transfer latency for disagg serving (#6504)
**SHA**: `5d95862` | 🔗 [查看提交](https://github.com/ai-dynamo/dynamo/commit/5d9586234e73a9981b5355836a5dbc2378a3014b)

**🎯 变更类型**：功能增强  
**⚡ 重要程度**：🔴高  
**📋 变更摘要**：在 Mock‑Engine 中新增 KV 缓存传输延迟模拟，用于离散化（disaggregated）部署场景。新增 CLI 参数 `--kv-transfer-bandwidth`、`--kv-cache-dtype`、`--kv-bytes-per-token`，在 Python 启动脚本中自动计算 KV 每 Token 大小，并将其注入 EngineArgs；Rust 端根据 `kv_bytes_per_token` 与 `kv_transfer_bandwidth` 计算转移时延，并在 Prefill 完成首 Token 后插入精准睡眠，以模拟 KV 从 Prefill Worker 向 Decode Worker 的网络传输时间。文档与 README 同步更新。

**🎯 影响范围**：  
- `components/src/dynamo/mocker/*`（CLI 参数、README、Python 启动逻辑）  
- `components/src/dynamo/mocker/utils/kv_cache.py`（KV 大小、带宽常量计算）  
- `lib/mocker/src/common/protocols.rs`（EngineArgs 新增字段）  
- `lib/mocker/src/common/utils.rs`（精准延迟计算 & sleep 实现）  
- `lib/mocker/src/mocker.rs`（Prefill 工作流中注入 KV 传输延迟）  
- `lib/mocker/src/scheduler/vllm.rs`（改用统一的 `sleep_until_precise`）  
- 文档 `docs/pages/mocker/mocker.md`

---

### 🔍 技术洞察

| 维度 | 影响 |
|------|------|
| **架构影响** | 1. 在 EngineArgs 中新增 `kv_bytes_per_token` 与 `kv_transfer_bandwidth`，使 KV 传输延迟成为 **配置即代码** 的一部分。<br>2. Python CLI 负责在模型预取后基于模型配置自动计算 KV 大小，保持了 **前后端配置统一**。<br>3. Rust 端在 Prefill 完成首 Token 前调用 `compute_kv_transfer_delay` 并使用 `sleep_precise`，为离散化调度路径加入**新同步点**（Prefill → KV Transfer → Decode）。 |
| **性能影响** | - **正向**：提供更真实的端到端时延模型，帮助调度器评估分布式 KV 迁移的瓶颈。<br>- **负向**：引入一次阻塞式精准睡眠，增加 **Prefill** 阶段的额外等待；在高并发、多 Worker 场景下，这会累积到整体吞吐率下降。<br>- **实现细节**：`sleep_precise` 在 Linux 上使用 `timerfd`，误差在毫秒级以内；在非 Linux 平台回退到 `tokio::time::sleep_until`，误差略高。<br>- **启动开销**：`compute_kv_bytes_per_token` 需要 `transformers.AutoConfig` 与 `dynamo.profiler.utils.get_model_info`，在首次启动时会稍微增加 I/O 与 CPU 开销。 |
| **安全考虑** | - 参数均为 **数值**，不涉及外部资源或文件路径，风险极低。<br>- 新增的 `--kv-cache-dtype` 受限于白名单 (`auto`, `bfloat16`, `fp8*`)，防止任意字符串注入。<br>- 仍需注意 **整数溢出**：`kv_bytes_per_token` 与 `num_input_tokens` 的乘积在 `compute_kv_transfer_delay` 采用 `f64` 计算，避免了 Rust `usize` 溢出。 |
| **可维护性** | - 新增的 Python 与 Rust 代码相互独立，模块化良好。<br>- `utils/kv_cache.py` 与 `common/utils.rs` 逻辑对称，便于以后补充其他数据类型或带宽模型。<br>- 文档同步更新，CLI 参数说明完整。 |

### ⚠️ 潜在风险

1. **误配置导致不合理延迟**  
   - 例如把 `--kv-transfer-bandwidth` 设为极小值（<1 GB/s）会产生几秒甚至分钟级的 KV 转移时延，导致测试场景失真。  
2. **阻塞式睡眠对并发度的影响**  
   - `sleep_precise` 在 async 环境下仍会挂起当前 task，若 Prefill Worker 数量较多，累积的延迟可能使 event loop 出现“卡顿”。  
3. **跨平台行为差异**  
   - 非 Linux 系统使用 `tokio::time::sleep_until`，误差比 `timerfd` 大约 10‑20 ms，可能在高精度基准测试中出现偏差。  
4. **模型解析失败**  
   - `compute_kv_bytes_per_token` 捕获异常后返回 `None`，但随后 `engine_args` 仍可能缺少 `kv_bytes_per_token`，导致 Rust 端 `None` → 延迟被禁用，行为与预期不符且缺少显式警告。  
5. **向后兼容性**  
   - 旧版脚本若未传递新参数，默认行为仍是 **带宽 64 GB/s、自动计算 KV 大小**，在大多数场景下影响不大，但如果已有自定义 `kv_bytes_per_token` 的外部工具，需确认其仍能覆盖。  

---

### 💡 关注建议

1. **参数校验与提示**  
   - 在 CLI 中对 `--kv-transfer-bandwidth` 给出警告（如 `<1.0` 或 `>1000`）；在 Rust `MockEngineArgs` 的 `validate` 中加入上限检查，防止异常配置。  
2. **启动时日志**  
   - 当 `kv_bytes_per_token` 自动计算成功或失败时，打印明确信息，帮助使用者快速定位模型解析问题。  
3. **可选的 “dry‑run” 模式**  
   - 提供 `--kv-transfer-dry-run` 标记，仅输出预计的转移时延而不实际睡眠，便于快速评估配置。  
4. **性能回归测试**  
   - 加入基准测试，比较 `kv_transfer_bandwidth=0` 与默认值下的吞吐率、延迟，以验证模拟对整体系统负载的影响在可接受范围。  
5. **跨平台一致性**  
   - 若对延迟精度要求极高，考虑在非 Linux 上引入 `tokio::time::sleep_until` 的高分辨率实现（如 `spin_sleep`）或在文档中明确平台差异。  
6. **文档与示例**  
   - 在 README 与 docs 中提供典型的 `--kv-transfer-bandwidth` 配置示例（InfiniBand vs NVLink），并说明何时应将其设为 `0`（纯聚合或调试） 。  

通过上述措施，可最大化此次 **KV 缓存转移延迟模拟** 的业务价值，同时将潜在的性能回退与使用误区控制在可接受范围。

### fix: vllm launch script errors for disagg and spec decoding (#6562)
**SHA**: `35323da` | 🔗 [查看提交](https://github.com/ai-dynamo/dynamo/commit/35323da20c1537362d7333e85d44bebddbf6a9a8)

**🎯 变更类型**：Bug修复  
**⚡ 重要程度**：🔴高  
**📋 变更摘要**：  
1. 将 **agg_spec_decoding** 示例脚本中的 `method` 参数从 `"eagle"` 修正为 `"eagle3"`，与最新的 VLLM 采样实现保持一致。  
2. 为 **disagg_kvbm_router.sh** 与 **disagg_same_gpu.sh** 脚本新增 `--kv-transfer-config` 参数，显式配置 KV 迁移使用 `NixlConnector` 且角色为 `kv_both`。  
3. 在 `disagg_same_gpu.sh` 中加入 `--max-model-len 16384`，保证在解码/预填阶段使用统一的模型最大序列长度。  

这些改动消除了因脚本参数不匹配导致的启动错误，使分布式 KV（KV‑BM）和投机解码在示例环境下能够顺利运行。

---

**🎯 影响范围**：  
- `examples/backends/vllm/launch/agg_spec_decoding.sh`  
- `examples/backends/vllm/launch/disagg_kvbm_router.sh`  
- `examples/backends/vllm/launch/disagg_same_gpu.sh`  
- 受影响的模块：`dynamo.vllm` CLI、KV‑BM（Key‑Value Buffer Manager）子系统、投机解码（speculative decoding）路径。

---

**🔍 技术洞察**  

- **架构影响**  
  - **更正方法名称**：`eagle3` 是 VLLM 新增的投机解码实现，使用该名称可确保前端解析器加载正确的解码器类，避免运行时 `MethodNotFound` 错误。  
  - **显式 KV 迁移配置**：通过 `--kv-transfer-config` 指定 `kv_connector` 为 `NixlConnector` 并设定 `kv_role=kv_both`，使得 **解码** 与 **预填** 两侧都能共享同一 KV 缓存，实现真正的**双向 KV 同步**，符合 VLLM 的分布式解耦设计。  
  - **统一模型长度**：`--max-model-len 16384` 保证解码和预填工作进程使用相同的上下文上限，防止因长度不一致产生的 **跨进程状态不匹配**。

- **性能影响**  
  - **KV 迁移显式配置**：避免了默认回退到内部硬编码的 `NixlConnector`（若存在）导致的潜在额外序列化/网络开销，提升 **KV 同步** 的吞吐率与延迟稳定性。  
  - **增加 `max-model-len`**：在长上下文场景下可以一次性容纳更大的 token 序列，降低因频繁截断导致的 **重新计算** 开销；但相应的显存占用会随上下文长度线性增长，需要根据 GPU 可用显存进行调优。  

- **安全考虑**  
  - 脚本中传递的 JSON 配置 (`--kv-transfer-config`) 为静态、可信的内部字符串，不涉及外部输入，风险极低。  
  - 如用户自行修改这些参数并指向不受信任的连接器，可能导致 **未授权的 KV 访问**；建议在文档中提醒只使用官方提供的 `NixlConnector`。  

---

**⚠️ 潜在风险**  

1. **版本兼容性**：  
   - 若用户仍在使用旧版 VLLM（不支持 `eagle3` 或 `--kv-transfer-config`），脚本会因参数未知而启动失败。  
2. **显存压力**：  
   - `--max-model-len 16384` 在显存受限的 GPU 上可能触发 OOM，尤其在多卡或多进程部署时。  
3. **Connector 可用性**：  
   - `NixlConnector` 需要相应的插件或库在运行时可用；缺失会导致 KV 同步失效，进而出现 **空缓存** 或 **错误的解码结果**。  

---

**💡 关注建议**  

- **文档同步**：在官方 README 与示例脚本说明中标注 “仅适用于 VLLM ≥ X.Y（支持 `eagle3` 与 `--kv-transfer-config`）”。  
- **回退方案**：提供一个兼容旧版的脚本分支或检测 `dynamo vllm --version` 自动切换 `method` 与 KV 参数。  
- **显存监控**：在启动前加入显存检查（`nvidia-smi`）或在脚本中提示用户根据显卡容量自行调节 `--max-model-len` 与 `--gpu-memory-utilization`。  
- **安全审计**：若未来允许用户自定义 KV connector，务必在 CLI 层加入白名单校验，防止任意可执行代码注入。  

通过上述措施，可确保该修复在提升示例可用性的同时，保持系统的稳健性与安全性。

  - 引入 **流式响应**（`Body::from_stream`）与 **连接监控**（`create_connection_monitor`），使得视频生成引擎的生命周期可以被客户端断开事件及时终止，防止资源泄漏。

  - **CPU**：每帧进行一次 Base64 解码 (`BASE64_STANDARD.decode`) 与一次 `Vec` 拼装，导致额外的 CPU 开销。对高帧率（≥30fps）或大分辨率的流可能产生显著负载。  
  - **内存**：每帧会临时分配 `Vec<header + jpeg + "\r\n">`，随后转为 `Bytes`. 峰值内存主要取决于单帧 JPEG 大小（通常 < 500KB），但在极端情况下（高分辨率或未压缩）可能导致瞬时内存抖动。  
  - **网络**：使用 `multipart/x-mixed-replace`，每帧都有 `Content-Length`，有利于浏览器直接渲染；但若客户端不及时读取，TCP 缓冲区会积压，可能触发背压导致引擎阻塞。  

- **安全考虑**  
  - **输入验证**：后端返回的 `b64_json` 被直接解码后写入响应，没有对 JPEG 内容做额外校验，理论上可能被用于 **压缩炸弹** 或 **恶意 JPEG 解析漏洞**（如 libjpeg 的 CVE）。  
  - **DoS 防御**：流式端点对外部无限制开放，恶意客户端可持续发起请求，占用 `engine`、`inflight`、`http_queue_guard` 等资源；虽有 `inflight` 计数，但缺少超时/速率限制。  
  - **信息泄漏**：`request_id` 通过 `get_or_create_request_id` 传播，若日志未脱敏，可能将内部请求 ID 暴露给外部。  

**⚠️ 潜在风险**  

1. **资源泄漏**：如果 `connection_handle` 未在异常路径（如 `engine.generate` 失败后）及时 `disarm`，可能导致监控 goroutine 持续占用。  
2. **未捕获的错误**：`filter_map` 中对 `base64::decode` 或 `response.data` 的错误仅打印日志并返回 `None`，对客户端表现为流中断而无错误信息，难以定位问题。  
3. **内存峰值**：高分辨率或异常大的 JPEG（未受限）会在单帧 `Vec` 中一次性分配，可能导致 OOM。  
4. **兼容性**：部分旧版浏览器或工具（ffplay 等）对 `multipart/x-mixed-replace` 的实现不一致，可能出现渲染卡顿。  
5. **监控遗漏**：`inflight.mark_ok()` 只在流自然结束时调用，若客户端异常断开导致 `stream_handle.disarm()` 前 `inflight` 未标记为错误，统计上可能误报为成功请求。  

**💡 关注建议**  

- **限流与超时**：在 `video_stream` 入口对单个请求设置最大持续时间（如 5 min）或最大帧数，超出后强制关闭流并返回相应错误码。  
- **帧大小校验**：在解码后检查 `jpeg_bytes.len()` 是否超过合理阈值（例如 2 MiB），超过则直接丢帧并记录警告，防止压缩炸弹。  
- **错误传播**：将解码或数据缺失的错误包装为 `ErrorResponse` 并写入流（例如 `--frame\r\nContent-Type: text/plain\r\n\r\n<error>`），让客户端能够感知异常结束。  
- **监控完善**：在 `filter_map` 的错误路径显式调用 `inflight.mark_error()`，并在 `connection_handle` 的 `drop` 实现中记录 “client disconnect”。  
- **安全审计**：考虑在解码后使用安全的 JPEG 解析库（如 `image` crate）做一次最小化验证，防止恶意 JPEG 触发底层库漏洞。  
- **文档与标记**：在 API 文档显式标注此接口为 **实验性**，并提示用户可能的兼容性与资源消耗问题。提供示例 `curl`/`ffplay` 调用方式。  
- **测试覆盖**：新增单元/集成测试：  
  1. 正常帧流（多帧）能被客户端完整接收。  
  2. 后端返回缺失 `b64_json`、空 `data` 时客户端收到流结束或错误。  
  3. 客户端提前关闭连接时，引擎上下文被成功取消。  

通过上述措施，可在保持新功能的同时，降低资源占用、提升安全性并提供更好的可观测性。

### ⚠️ 潜在风险  

| 风险点 | 说明 | 缓解措施 |
|--------|------|----------|
| **标签基数膨胀** | `error_type` 只包含固定几种值，基数受控；但若后续频繁扩展新错误类型，需评估 Prometheus 标签基数对存储和查询的影响。 | 在设计阶段保持错误类型枚举闭合或使用 `UNKNOWN` 兜底。 |
| **漏报/误报** | 某些异常路径（如 `panic!`、`tokio::task::JoinError`）未显式调用 `mark_error`，仍会在 `Drop` 中记录 `Internal`，可能掩盖实际错误类型。 | 在关键 `await` 块或 `JoinHandle` 结果上统一使用 `map_err` 包装，确保所有错误均调用 `extract_error_type_from_response`。 |

| **错误分类不完整** | 对 4xx 错误统一归为 `Validation`，但某些业务错误（如 401、403）可能更适合单独标签。 | 依据业务需求在 `classify_error_for_metrics` 中细化匹配规则，或在 `ErrorType` 中新增对应枚举。 |
| **并发 Guard 使用** | `InflightGuard` 仍是 `Clone` + `Copy`，但内部状态是 `mut` 并在 `Drop` 时记录。如果同一个请求被多次 `clone` 并在不同线程上使用，可能导致计数重复或误标记。 | 确认所有 `clone` 仅在单线程（同一请求上下文）使用；如有跨线程需求，改为 `Arc<Mutex<...>>` 并在 `Drop` 前统一记录。 |
