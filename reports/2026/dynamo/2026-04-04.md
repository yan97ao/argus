### fix(frontend): Kimi tokenizer fix - use absolute token ID in tiktoken reserved token fallback names (#7886)
**SHA**: `9ad3863` | 🔗 [查看提交](https://github.com/ai-dynamo/dynamo/commit/9ad38639ef380e7f147cfaac77a71c013224fb98)

**🎯 变更类型**：Bug修复  
**⚡ 重要程度**：🔴高  
**📋 变更摘要**：  
- 将 `tiktoken` 的保留特殊 token 名称从相对偏移（`<|reserved_token_{i}|>`）改为使用其绝对 token ID（`<|reserved_token_{id}|>`），解决 Kimi K2.5 模型在对未命名保留 token 解析时的 token 膨胀问题。  
- 同时补充了完整的单元测试，覆盖普通加载、绝对 ID 命名的正确性以及旧命名导致的膨胀回归验证。

**🎯 影响范围**：  
- `core` 模块下的 **TikTokenTokenizer** 实现  
- 依赖该 tokenizer 的所有模型推理入口（如 Kimi、GPT‑系模型）  
- 相关的配置加载逻辑（`tokenizer_config.json`、`config.json`）  

**🔍 技术洞察**：  

| 维度 | 影响 |
|------|------|
| **架构影响** | 仅限于 **tokenizer** 子系统的实现细节，API（`encode`、`decode`）保持不变。<br>修改了 `load_special_tokens` 中保留 token 名称的生成规则，确保在没有显式 `added_tokens_decoder` 时仍能正确映射到绝对 ID。 |
| **性能影响** | - **正向**：避免了因未识别保留 token 而产生的大量 BPE 拆分，实质上降低了编码时的 token 数量和后续模型推理的计算负担。<br>- **负向**：生成名称时仅一次 `format!`，对整体性能影响可忽略不计。 |
| **安全考虑** | 变更不涉及网络、文件 I/O 或外部输入的安全检查，仅是内部映射规则的更正。没有新增的攻击面。<br>唯一需要注意的是确保不在生产环境误用旧的 `bad_special_tokens` 逻辑，以免出现意外的 token 膨胀导致资源耗尽。 |

**⚠️ 潜在风险**：  
1. **兼容性**：若用户自行构造了基于旧相对偏移命名的 `added_tokens_decoder`（极少见），在升级后这些名称将不再匹配，导致 tokenizer 仍会出现膨胀。  
2. **测试依赖**：新增的单元测试使用临时文件和 base64 编码生成 256‑byte 词表，可能在极端的构建环境（如限制 `std::fs` 权限的 sandbox）中失败。  
3. **文档同步**：文档或示例代码中若仍描述旧的 `<|reserved_token_{i}|>` 形式，可能导致使用者误解。  

**💡 关注建议**：  
- **版本说明**：在下一个发布说明中明确指出“保留 token 的回退名称已改为使用绝对 ID”，并提示用户若自行提供 `added_tokens_decoder`，请使用绝对 ID 命名。  
- **回归测试**：在 CI 中保留本次新增的两段测试，以防将来再次回退或误改。  
- **兼容层**：可考虑在 `load_special_tokens` 中加入兼容检测（如检测到 `<|reserved_token_{i}|>` 并给出警告），降低旧配置导致的突发错误。  
- **监控**：在生产环境监控 token 长度异常（如同一请求 token 数异常增长），若出现异常可快速定位是否仍有旧命名残留。  

---  
*该变更通过修正特殊 token 的命名逻辑，消除了 Kimi tokenizer 的严重膨胀缺陷，提升了系统的鲁棒性与推理效率，风险可控，建议尽快合入主分支并在下一次发布中同步文档。*

### fix: honor per-node GPU topology in DGD builder (DYN-2544) (#7868)
**SHA**: `d94c2ea` | 🔗 [查看提交](https://github.com/ai-dynamo/dynamo/commit/d94c2ea2ea35c377b0b6b8859a7e7e03855f53b3)

**🎯 变更类型**：Bug修复  
**⚡ 重要程度**：🔴高  
**📋 变更摘要**：  
本次提交针对 DGD（DynamoGraphDeployment）构建器在多节点 GPU 拓扑下的资源分配缺陷进行修复。新增 `num_gpus_per_node` 参数，在生成 DGD 配置时强制每个 worker 服务的 GPU 限额不超过单节点可用 GPU 数，并在需要跨节点时自动填充 `multinode.nodeCount`。同步更新了单元/集成测试、基准依赖及相关配置文件，以防止小型 MoE 模型产生超出集群 GPU 容量的候选部署。

**🎯 影响范围**：  
- `components/src/dynamo/profiler/utils/config_modifiers/protocol.py`（核心业务逻辑）  
- `components/src/dynamo/profiler/tests/*`（单元、集成测试）  
- `benchmarks/pyproject.toml`、`container/deps/requirements.planner.txt`（依赖版本）  
- 新增 `components/src/dynamo/profiler/tests/data/configs/12_thorough_moe_qwen3_30b_a3b_sglang.yaml`（回归测试用例）  
- 可能波及使用 DGD Builder 的上层调度/规划组件（Planner、Frontend 等）。

**🔍 技术洞察**  
- **架构影响**：  
  - 在 `ConfigModifier.build_dgd_config` 签名中加入 `num_gpus_per_node`，并在内部统一走 `setup_worker_service_resources`，实现 **“per‑node GPU shaping”**。  
  - 为跨节点的 decode/agg worker 自动设置 `multinode.nodeCount`，使 DGD 配置能够正确表达“2 × 8 GPU”节点布局。  
  - 通过新增测试验证了在 `totalGpus=16, numGpusPerNode=8` 场景下，所有候选服务的 GPU 限额 ≤ 8，防止了原先的 “全局 GPU 上限” 误判。  

- **性能影响**：  
  - 仅是配置生成阶段的逻辑改动，对运行时推理性能无直接影响。  
  - 正确的 GPU 资源分配可避免调度器因资源超额而产生的回滚或重调度，从而间接提升集群调度效率。  

- **安全考虑**：  
  - 变更涉及资源配额的约束，未引入新的网络、身份或代码执行风险。  
  - 仍需确保 `num_gpus_per_node` 参数来源可信（通常来源于硬件描述），防止恶意配置导致资源泄露或过度占用。  

**⚠️ 潜在风险**  
1. **向后兼容性**：旧版配置或脚本若未传递 `num_gpus_per_node`，会回退到原有行为（可能仍会产生超额 GPU 请求）。需要在调用方添加显式参数或提供默认值。  
2. **依赖同步**：`aiconfigurator` 依赖更新至新的 commit，若该 commit 引入不兼容的 API 改动，可能导致运行时错误。  
3. **多节点假设**：`setup_worker_service_resources` 目前仅在 `gpus > num_gpus_per_node` 时划分 `multinode`，若出现非均匀 GPU 拓扑（如 6 GPU/节点），可能产生不完整的 `nodeCount` 配置。  
4. **测试覆盖不足**：当前仅覆盖了 MoE “thorough” 场景，其他调度模式（rapid、disagg）未验证跨节点 GPU 形状是否同样受限。  

**💡 关注建议**  
- **文档与 SDK**：在 `ConfigModifier`、`DGD Builder` 的官方文档中加入 `num_gpus_per_node` 参数说明及示例，建议将其设为必填字段或在缺省时自动从硬件描述推导。  
- **回退兼容**：在 `build_dgd_config` 中加入对 `num_gpus_per_node` 为 `None` 的容错路径（保持旧行为），并在日志中给出明确警告。  
- **CI 扩展**：为 `rapid`、`disagg`、`agg` 模式各增添一组跨节点 GPU 资源校验测试，确保所有 worker 类型均受限。  
- **依赖锁定**：在 `requirements.planner.txt` 与 benchmark 项目中同步更新 `aiconfigurator` 依赖版本，并在 CI 中加入 `pip freeze` 检查，以防未来不慎回滚到旧 commit。  
- **监控报警**：在调度平台（Planner）加入 “GPU limit > per‑node capacity” 的校验报警，及时捕获潜在的配置生成错误。  

通过上述改动，系统在多节点 GPU 环境下的资源调度更加可靠，避免了因超额 GPU 请求导致的调度失败或资源浪费。后续可进一步完善跨节点不均匀拓扑的支持以及对其他调度模式的统一约束。

### feat: multimodal request_rate based sweep (#7363)
**SHA**: `5c461d6` | 🔗 [查看提交](https://github.com/ai-dynamo/dynamo/commit/5c461d6117869176b06384a35015c7d10717b74e)

**🎯 变更类型**：功能增强  
**⚡ 重要程度**：🔴高  

**📋 变更摘要**  
本次提交为多模态基准工具引入了 **request‑rate** 维度的 sweep 能力，取代（或补充）原有仅基于并发数的 sweep。通过在 CLI 中新增 `--request-rates` 参数、在配置结构中加入 `request_rates`、以及在 orchestrator 与 runner 中统一抽象 “sweep mode/value”，实现了 **“并发数 ↔ 请求速率”** 的两套互斥运行模式，并相应地完善了结果目录组织、跳过已完成任务及服务器生命周期管理。

**🎯 影响范围**  
- `benchmarks/multimodal/sweep/args.py`（CLI 参数）  
- `benchmarks/multimodal/sweep/config.py`（配置模型）  
- `benchmarks/multimodal/sweep/orchestrator.py`（调度层）  
- `benchmarks/multimodal/sweep/runner.py`（实际执行层）  
- 新增测试 `tests/benchmarks/multimodal/sweep/*`  
- 相关导入与类型 (`BenchmarkConfig`)  

**🔍 技术洞察**  

- **架构影响**  
  - **抽象层提升**：通过 `sweep_mode` / `sweep_values` 把原本硬编码的 “并发数” 变为可扩展的概念，后续若再引入其他维度（如 payload‑size）时，可在同一框架下扩展。  
  - **配置兼容**：YAML 中同时支持 `request_rates` 与 `concurrencies`，且在缺省时自动使用 `_DEFAULT_REQUEST_RATES`，保持对老配置的向后兼容。  
  - **CLI 互斥组**：使用 `argparse` 的 mutually exclusive group，防止用户同时提供两套参数，降低误用风险。  
  - **调度重构**：原先的三层循环（input‑file → config → concurrency）被重新排列为 **config → input‑file → sweep value**，更符合业务需求（每个模型‑配置先完整跑完所有输入，再切换到下一个配置）。  
  - **结果组织**：结果目录从 `c{N}` 更名为 `{mode}{N}`（如 `request_rate8`），直观展示运行维度。

- **性能影响**  
  - **测量维度变化**：引入 request‑rate 使得基准可以在同一并发数下测不同请求速率，能够捕获服务在不同流量下的吞吐与延迟特性。  
  - **启动成本**：`restart_server_every_benchmark` 仍然控制服务器的启动次数；在 `restart_server_every_benchmark=False` 场景下，只会在每个 config 开始时启动一次，和旧实现保持相同的最优启动开销。  
  - **跳过已完成**：在 `orchestrator._run_config` 中先检查目标目录是否已有 `profile_export_aiperf.json`，可显著减少重复运行，提升 CI/本地调试效率。

- **安全考虑**  
  - 变更仅涉及本地 benchmark 逻辑和文件路径，无外部网络交互的新增风险。  
  - 参数解析仍然使用安全的标准库 (`argparse`、`yaml.safe_load`)；未引入额外的命令注入点。  
  - 需留意 `artifact_dir` 的路径拼接在 CI 环境下的写权限，仍然与之前相同。

**⚠️ 潜在风险**  

| 风险点 | 可能后果 | 建议缓解 |
|--------|----------|----------|
| 兼容性 – 老的 YAML 只提供 `concurrencies`，但默认会自动补全 `request_rates` 为 `_DEFAULT_REQUEST_RATES`，可能导致意外的双模式运行 | 产生额外的 benchmark 运行，浪费资源 | 在文档中明确说明若仅想使用并发模式，请在 YAML 中显式设置 `request_rates: null`，或在 CLI 中使用 `--concurrencies` 覆盖默认值。 |
| 参数互斥未在子模块（如 `load_config`）再次验证 | 若外部调用 `load_config` 并手动传入两个字段，可能触发运行时错误 | 在 `SweepConfig.validate` 已加入检查，建议在公共入口前调用 `validate`，或在 `load_config` 中加入早期校验。 |
| 目录命名冲突（`request_rate4` 与未来可能的 `rate4` 等） | 结果目录混淆 | 采用统一前缀 `request_rate`、`concurrency`，如需新增维度请统一遵守 `<mode><value>` 规则。 |
| `restart_server_every_benchmark=False` 时，如果某个 `input_file` 在不同 `sweep_value` 之间留下缓存，可能导致测量不公平 | 基准不准确 | 已在代码注释中说明此模式适用于**不关心缓存影响**的场景，使用前请评估需求。 |
| 测试依赖 `ServerManager` 的 mock 行为未覆盖异常路径（如服务器启动失败） | CI 通过但真实运行在异常情况下未被检测 | 增加异常路径单元测试或在 CI 中加入真实服务器启动的集成测试。 |

**💡 关注建议**  

1. **文档与示例**  
   - 更新 README 与 CLI usage 示例，展示两种模式的调用方式以及相互排斥的说明。  
   - 在 benchmark 配置模板中加入注释，提示用户只能保留 `request_rates` 或 `concurrencies` 之一。

### fix(bench): replay mooncake events offline (#7876)
**SHA**: `c08034e` | 🔗 [查看提交](https://github.com/ai-dynamo/dynamo/commit/c08034ebb2ebc75de9ced34e649d7c9a28376b5b)

**🎯 变更类型**：Bug修复 / 功能增强  
**⚡ 重要程度**：🔴高  
**📋 变更摘要**：  
- 将 Mooncake 基准的离线重放实现从基于实时 `Scheduler` 的异步流改为纯离线、阻塞式的工件生成器，去除了 `EventCollector`、`Scheduler` 以及大量异步调度逻辑。  
- 新增 `ReplayWorkerArtifacts` 系列结构体（`ReplayTimedRequest/OutputSignal/KvEvent`）以及公开的 `generate_trace_worker_artifacts_offline` 接口，支持在磁盘上直接生成完整的请求、输出信号和 KV 缓存事件工件。  
- 为 KV 引擎（VLLM、SGLang）新增 `new_with_kv_capture` 构造函数，实现离线模式下的 KV 事件捕获。  

**🎯 影响范围**：  
- `lib/bench/kv_router`（基准相关）  
- `lib/mocker/src/replay`（离线重放核心）  
- `lib/mocker/src/scheduler`（相关调用被剥离）  
- 对外公开的 `dynamo_mocker::replay::*` API  

**🔍 技术洞察**：

- **架构影响**  
  - **解耦**：离线工件生成不再依赖 `Scheduler`，将 **调度层** 与 **重放层** 完全分离，提升了模块化程度。  
  - **新抽象**：`ReplayWorkerArtifacts` 成为离线重放的统一返回值，后续可能成为其它工具（如分析、可视化）的数据来源。

  - **基准速度提升**：离线模式在基准中采用阻塞 `spawn_blocking`，避免了大量 Tokio 调度开销，且 `speedup_ratio` 默认从 `0.0` 改为 `10.0`（加速 10 倍），显著缩短基准运行时间。

- **架构影响**：

  - 删除操作为 `HashMap::remove`，时间复杂度 O(1)；遍历监听者列表的开销与 dp‑rank 数量成正比，通常为常数级（每个 worker 的 dp‑rank 数量有限）。

- **安全考虑**：  
  - 本次改动不涉及网络、权限或加密，仅是内部状态清理，无直接安全风险。  
  - 避免了因残留 watermark 导致的错误调度或错误的 “已完成” 标记，从间接上提升了系统的可靠性和数据一致性。

1. **并发竞争**：如果在注销期间仍有其他线程持有该 watermark 的 `Arc<AtomicU64>` 并尝试写入，`remove` 可能导致悬空引用。需确保所有对 watermark 的写入在注销前已被取消（已有 `cancel_token.cancel()`），但仍建议审查是否所有路径都能保证此顺序。  
2. **错误的错误处理路径**：在 `#[cfg(feature = "indexer-runtime")]` 分支里未同步加入清理逻辑，若开启该特性且走到该分支，可能仍会残留 watermark。需要确认编译特性覆盖完整。  
3. **测试覆盖不足的边界**：如异常情况下（e.g., `deregister` 失败返回错误）是否已经完成了 watermarks 清理；当前测试仅覆盖成功路径。  

**💡 关注建议**：  
- **代码审查**：确认所有 `deregister*` 路径（包括特性分支）均已调用 `self.watermarks.remove`，防止遗漏。  
- **并发安全**：若有可能在注销后仍保留对 watermark 的引用，考虑使用 `Arc::weak_count` 检查是否为唯一所有者后再删除，或在注销前将 watermark 值置为 `u64::MAX` 并广播取消信号。  
- **监控**：在生产环境加入指标（如 `registry_watermark_entries_total`）并监控其随时间的增长趋势，确保清理生效。

- **文档更新**：在 `kv-router` 的设计文档或 README 中说明 watermark 生命周期已在注销时自动回收，避免后续维护者误以为需要手动清理。  

通过上述改动，系统在 worker 生命周期管理上的资源一致性得到显著提升，降低了长期运行后潜在的状态漂移与内存泄漏风险。
