### fix(operator): gate DCD readiness on observed generation (#9499)
**SHA**: `1f496e4` | 🔗 [查看提交](https://github.com/ai-dynamo/dynamo/commit/1f496e4627a90f7b697640a06ffae4642e494b28)

**🎯 变更类型**：Bug修复  
**⚡ 重要程度**：🔴高  
**📋 变更摘要**：在 `DynamoComponentDeployment`（v1alpha1、v1beta1）实现的 `IsReady()` 方法中加入对 `ObservedGeneration` 与 `Generation` 的比较，只有当控制器已经处理最新的 `spec` 时才返回准备就绪。同步更新了单元测试，覆盖旧副本残留以及 `ObservedGeneration` 落后导致的待定状态。  

**🎯 影响范围**：  
- `deploy/operator/api/v1alpha1/dynamocomponentdeployment_types.go`  
- `deploy/operator/api/v1beta1/dynamocomponentdeployment_types.go`  
- `deploy/operator/internal/controller/dynamocomponentdeployment_controller_test.go`  
- `deploy/operator/internal/controller/dynamographdeployment_controller_test.go`  
- 依赖 `DynamoComponentDeployment` 的所有 Operator 控制循环（尤其是 `DynamoGraphDeployment`）  

**🔍 技术洞察**  

- **架构影响**  
  - **状态一致性强化**：通过显式检查 `ObservedGeneration`，确保 Operator 只在 spec 实际被 reconciled 后才将组件标记为 Ready，防止出现“已就绪但实际未应用最新配置”的错误状态。  
  - **传播链路**：`DynamoGraphDeployment`（DGD）在聚合子组件状态时会依据 `IsReady()` 的返回值决定整体 `Available` 条件和 DGD 状态。此更改直接影响 DGD 从 `Pending` → `Available` 的转移时机，使得状态更可信。  

- **性能影响**  
  - 仅是一次整数比较，几乎不产生额外运行时开销。  
  - 由于 `IsReady()` 现在可能更早返回 `false`，Operator 可能会触发额外的 reconcile 循环，导致极微小的 CPU/内存消耗提升，但对整体吞吐量影响可忽略不计。  

- **安全考虑**  
  - 该改动不涉及网络、身份验证或数据加密等安全面，仅是状态判断的业务逻辑。  
  - 提升了系统对不一致状态的可见性，间接降低因错误就绪导致的业务错误（如流量路由到未更新的服务）所带来的风险。  

**⚠️ 潜在风险**  
1. **就绪迟滞**：如果底层 `Deployment`、`StatefulSet` 等控制器未能及时更新 `status.observedGeneration`（例如因集群负载或 API Server 延迟），DCD 会一直保持 `Pending`，进而导致 DGD 也停留在 `Pending`。  
2. **兼容性**：外部工具或脚本可能直接读取 `DynamoComponentDeployment.IsReady()` 的返回值，原有的 “只看条件” 行为已被改变，可能导致误报。  
3. **测试依赖**：新增的测试用例假设 `ObservedGeneration` 低于 `Generation` 时返回 `false`。如果以后 `IsReady()` 被重构或包装，需同步更新对应的单元测试。  

**💡 关注建议**  
- **监控与告警**：在 Operator 的监控面板上加入 `ObservedGeneration != Generation` 的计数或告警，帮助运维快速定位因控制器延迟导致的卡顿。  
- **文档更新**：在 CRD 文档和使用指南中明确说明 “Ready” 条件仅在 spec 已被处理后才会为 `True`，并解释 `ObservedGeneration` 的含义。  
- **回退方案**：若出现业务因就绪迟滞而受阻的情况，可提供临时参数或注解（如 `dynamo.io/ignore-generation-check=true`）以跳过该检查，但应在发布说明中强调仅用于紧急场景。  
- **回滚验证**：在 CI 中保留旧的 `IsReady()` 行为的对比测试，确保新实现不会误判已就绪的组件。  

---  

此次修复通过对 `generation` 一致性的严格校验，使得 Operator 对组件状态的感知更加精准，提升了整体系统的可靠性。开发者在升级时需关注可能出现的就绪延迟，并据此调整监控或临时绕过策略。

### fix(router): use reservoir sampling for tie-breaks (#9516)
**SHA**: `147dcca` | 🔗 [查看提交](https://github.com/ai-dynamo/dynamo/commit/147dccaa47f1d28f7099d1febc1866465363284f)

**🎯 变更类型**：Bug修复 / 性能优化  
**⚡ 重要程度**：🔴高  
**📋 变更摘要**：  
- 在 `DefaultWorkerSelector`（零温度路径）和 `RoutingOccupancyState::select_exact_min_and_increment` 中，将原先收集所有最小分数/负载工作者再随机挑选的实现，改为使用**reservoir sampling**在遍历时直接完成均匀抽样，消除不必要的内存分配并保证在平分情况下的公平随机选择。  
- 新增对应的单元测试，以验证在多次调度/负载选择时能够随机命中不同的工作者。

**🎯 影响范围**：  
- `lib/kv-router/src/scheduling/selector.rs`（调度器零温度分支）  
- `lib/runtime/src/component/client.rs`（路由占用状态的最小负载抽样）  
- 相关单元测试文件  

**🔍 技术洞察**  

- **架构影响**：  
  - 只涉及调度子系统内部的实现细节，公共接口保持不变。  
  - 通过在遍历过程中直接完成抽样，降低了对 `Vec`/`HashMap` 等临时容器的依赖，提升了调度路径的**空间局部性**，对整体系统架构的耦合度没有负面影响，反而更贴合 “无锁/最小锁” 的设计理念。  

- **性能影响**：  
  - **时间**：原实现在最小分数/负载出现多次时需要 `O(k)` 的额外遍历（`k` 为最小元素数量）以及 `O(k)` 的内存分配/拷贝；现在只保持单个候选和计数，保持 `O(n)`（`n` 为工作者总数）且常数更小。  
  - **空间**：消除 `Vec::new()` / `FxHashMap` 的临时分配，显著降低热路径的堆内存使用，特别是在高并发、大规模工作者列表（数百甚至上千）时收益明显。  
  - **随机性开销**：使用 `rand::rng()`（基于线程局部 `ThreadRng`）的 `random_range`，开销极低，几乎可以忽略不计。  

- **安全考虑**：  
  - 随机数来源为非加密的快速 RNG，适用于调度公平性需求，未涉及安全敏感场景，无新增安全风险。  
  - 代码中未引入 `unsafe`，保持 Rust 安全模型。  

**⚠️ 潜在风险**  

1. **可重复性**：引入随机抽样后，同一请求在相同负载/分数下可能被分配到不同工作者，可能影响调试/日志复现，需要在调试模式或测试中显式控制（如固定种子）来确保可重现。  
2. **统计偏差**：虽然 reservoir sampling 在理论上是均匀的，但极端短跑（如仅 1‑2 次调度）可能仍表现出偏差；若业务对“必选第一个”有隐式假设，需要相应更新文档或兼容层。  
3. **RNG 依赖**：如果项目在未来对随机数源做统一管理（如替换为自定义 RNG），此处直接调用 `rand::rng()` 可能需要适配。  

**💡 关注建议**  

- **可观测性**：在调度日志中加入“抽样模式 / tie‑break random seed”字段，方便运维在出现异常调度时排查。  
- **测试强化**：在 CI 中保持当前的随机化单元测试（多次迭代确保至少出现两种不同的工作者），并考虑加入统计性检验（如卡方检验）以捕获极端偏差。  
- **文档更新**：在 `kv-router` 与 `runtime` 的调度策略说明中标明零温度或最小负载时会进行随机抽样，以避免使用者误以为仍是 “首个最小”。  
- **可配置开关（可选）**：如果业务场景需要完全确定性调度，可提供一个实验性配置 `router_tie_breaker = "deterministic"`，在此模式下回退到原始 `Vec`/`min_by_key` 实现。  

总体而言，此次改动提升了调度路径的 **性能与公平性**，风险可控，建议尽快合并并在生产环境监控抽样行为的分布情况。

### fix(container): backport vllm PR #40932 to unblock Qwen3-VL disagg multimodal (#9491)
**SHA**: `f60d1d8` | 🔗 [查看提交](https://github.com/ai-dynamo/dynamo/commit/f60d1d8e5351ab7fc3979220cbd0c48ae80578eb)

**🎯 变更类型**：Bug修复 / 功能增强  
**⚡ 重要程度**：🔴高  

**📋 变更摘要**  
- 在 `container/deps/vllm/install_vllm.sh` 中新增逻辑，基于 `VLLM_VER` 与本地 `patches/<VLLM_REF>` 目录，自动为低于 0.21.0 的 vLLM 版本打补丁。  
- 新增的补丁 `0001-pr40932-remove-invalid-deepstack-boundary-check-for-qwen3-vl.patch` 移除 Qwen3‑VL 模型中对 deepstack 边界的错误检查，避免在多模态推理时抛出异常，从而解锁 Qwen3‑VL 的分布式多模态能力。  

**🎯 影响范围**  
- `container` 镜像构建流程（vLLM 安装阶段）  
- 依赖 vLLM 0.20.x 系列的模型执行器，尤其是 `qwen3_vl` 与 `qwen3_omni_moe_thinker` 相关代码  
- 所有使用 Qwen3‑VL 多模态模型的 Dynamo 推理服务  

**🔍 技术洞察**  
- **架构影响**：  
  - 通过安装脚本在容器构建时对已安装的 vLLM wheel 进行源码层面的补丁，属于 **运行时二进制补丁**（post‑install patch）。  
  - 该方式保持了对上游 vLLM 版本的兼容性：当使用 ≥ 0.21.0 的 wheel 时自动跳过补丁，避免重复修改。  
  - 引入 `VLLM_SITE_PACKAGES` 解析路径并在根目录 (`cd /`) 执行 patch，确保补丁落在实际运行时加载的 site‑packages 目录，而非源码克隆目录。  

- **性能影响**：  
  - 补丁只在镜像构建阶段执行，运行时不再有额外检查逻辑，实际推理性能几乎不受影响。  
  - 若补丁未正确应用，原来的边界检查会导致异常中断，间接导致性能下降（任务失败）。  

- **安全考虑**：  
  - 脚本在容器内部执行 `patch`，若 `PATCH_DIR` 被恶意篡改，可能注入任意代码。建议在 CI 中对补丁文件做 SHA256 校验或使用只读的补丁目录。  
  - 使用 `--batch --forward` 使得非致命错误（已应用或已逆向的补丁）被静默忽略，降低构建挂起风险，但也可能掩盖真实冲突，需要在 CI 中留意 `rc >= 2` 的退出码。  

**⚠️ 潜在风险**  
1. **版本判断失误**：`VLLM_VER_BASE` 只剔除 `rc|.post|.dev|+`，但若出现非标准标签（如 `v0.20.1-xyz`），可能误判为需打补丁，导致补丁在不兼容的代码上执行。  
2. **补丁冲突**：如果上游已在某个 0.20.x 的发布版中合并该修复，脚本仍会尝试再次打补丁，`patch --forward` 会跳过，但若补丁内容有细微差异，可能产生残余文件或不完整修改。  
3. **路径解析错误**：`VLLM_SITE_PACKAGES` 通过 `import vllm` 获得，若容器中同时存在源码目录和 wheel，可能误指向源码路径，导致补丁未真正作用于运行时库。  
4. **构建时间增加**：额外的补丁遍历和 `patch` 调用会略微延长镜像构建时间，尤其在 CI 并行构建大量镜像时。  

**💡 关注建议**  
- **CI 检查**：在 CI 中加入校验步骤，确认 `VLLM_VER` 与 `VLLM_REF` 的对应关系，并对 `patches/` 目录内每个 `.patch` 计算 SHA256，防止意外篡改。  
- **日志强化**：在脚本中输出实际使用的 `VLLM_SITE_PACKAGES` 路径以及每个补丁的返回码，便于追溯与排障。  
- **版本上限**：当 vLLM ≥ 0.21.0 正式发布后，可移除整个补丁机制，或将其迁移至上游 vLLM 项目。  
- **文档更新**：在项目 README 或容器构建指南中注明：使用低于 0.21.0 的 vLLM 时会自动打补丁，若自行提供自定义 vLLM wheel，请确保兼容性或手动禁用该步骤。  
- **安全审计**：定期审计 `container/deps/vllm/patches/` 目录的内容，确保仅包含可信的 upstream PR 或官方 hotfix。  

---  

通过上述补丁，Dynamo 能够在不等待上游 vLLM 正式发布的情况下，稳妥地使用 Qwen3‑VL 多模态模型，显著提升功能可达性与用户体验，同时保持对后续上游升级的兼容性。

### refactor(llm): Merge chat/completions DeltaGenerator code (#9399)
**SHA**: `0ff011b` | 🔗 [查看提交](https://github.com/ai-dynamo/dynamo/commit/0ff011b02b4a5732ddf6e31207654c2119f98205)

**🎯 变更类型**：重构 / 架构变更  
**⚡ 重要程度**：🔴高  
**📋 变更摘要**  
1. 将 ChatCompletion 与 Completion 两套 *DeltaGenerator* 的实现抽取到 `delta_common.rs`，统一 `DeltaGeneratorOptions`、初始状态创建以及非流式请求的 usage‑enable 逻辑。  
2. `Preprocessor`、`chat_completions::delta.rs`、`completions::delta.rs` 等文件同步改为使用共享模块，删除大量重复结构体与实现，简化 tracker 的返回类型（始终返回 `Arc<RequestTracker>`）。  

**🎯 影响范围**  
- `lib/llm/src/preprocessor.rs`（tracker 包装）  
- `lib/llm/src/protocols/openai/*`（chat_completions、completions）  
- 新增 `lib/llm/src/protocols/openai/delta_common.rs`（公共配置、初始化、usage‑enable）  
- 相关单元测试文件（已同步调用方式）  

**🔍 技术洞察**  

- **架构影响**  
  - **模块化提升**：把两套几乎相同的 *DeltaGenerator* 逻辑抽象为公共模块，降低维护成本，后续新增 OpenAI 相关协议时可直接复用。  
  - **公共 API 变更**：`DeltaGenerator::tracker()` 从 `Option<Arc<_>>` 改为必返回 `Arc<_>`，并在调用方统一使用 `Some(&self.tracker)`。这简化了空指针检查，但对外部库如果直接依赖旧签名会产生编译错误。  
  - **初始化统一**：`delta_common::initial_state()` 把时间、`CompletionUsage`、`RequestTracker` 的创建集中，避免了在不同文件中重复 `SystemTime` → `u32` 转换以及默认 `CompletionUsage` 的手写。  

- **性能影响**  
  - **几乎不变**：抽象层仅涉及结构体实例的创建与复制（`Arc::clone`），没有增加运行时分支或额外锁。  
  - **潜在微利**：统一的 `initial_state` 减少了重复的时间获取与 `Arc::new` 调用，编译器可能会更好地内联，理论上略有提升。

  - **异步完成回调**：通过 `telemetry::stream::notify_on_completion` 返回的 `done_fut`，在流消费完毕后 `tokio::spawn` 异步执行上报，避免阻塞主响应流。  

- **性能影响**  
  - **额外开销**：  
    - `Arc<RequestTracker>` 与 `AgentReplayMetrics` 的创建/克隆带来微小的堆分配。  
    - 每个请求会额外启动一个轻量级 `tokio` 任务用于上报，通常在毫秒级延迟范围内。  
    - `wrap_agent_trace_request_end_stream` 对流进行一次包装，内部仅转发 `Item`，不改变流的背压行为。  
  - **可选性**：当 `is_enabled()` 为 `false` 时，上述路径几乎不产生运行时成本，保持原有性能基准。  

- **安全考虑**  
  - **数据泄露风险**：Trace 记录可能包含完整的 Prompt、模型名称、请求 ID 等敏感信息。需在 `emit_request_end` 或 Telemetry Bus 中确认已有脱敏/过滤机制，防止写入不受信的监控后端。  
  - **上下文注入**：通过 `Context` 读取 `X_REQUEST_ID_CONTEXT_KEY`，如果外部传入的上下文被恶意篡改，可能导致错误的追踪关联。建议在入口对该键值进行校验或签名。  
  - **代码注入**：该改动未引入外部输入的代码执行路径，安全面影响有限。  

**⚠️ 潜在风险**  

1. **异步任务泄漏**：`tokio::spawn` 产生的任务在请求完成后仍会执行上报，即便该请求已被客户端取消。若上报逻辑异常（例如网络不可达），可能导致未捕获的 panic，进而影响运行时。  
2. **上下文获取失败**：若 `X_REQUEST_ID_CONTEXT_KEY` 或分布式追踪上下文不存在，`x_request_id` 为 `None`，仍会产生 partial trace，可能导致监控数据不完整。  
3. **并发竞争**：`AgentTraceRequestEndState` 包含 `Arc<RequestTracker>`，在极高并发场景下频繁克隆可能增加引用计数开销。  
4. **兼容性**：新加入的 `futures::Stream` trait bound 需要 `std` feature `stream` 在旧版编译器或自定义运行时中可用，可能导致编译错误。  

**💡 关注建议**  

- **测试覆盖**：为 `build_agent_trace_request_end_state` 与 `wrap_agent_trace_request_end_stream` 添加单元/集成测试，验证在 trace 已禁用、缺失 `agent_context`、缺失 `request_tracker` 三种情况下行为符合预期。

### feat(global-router): retry failed requests on faster pools (#9460)
**SHA**: `0a9a1b6` | 🔗 [查看提交](https://github.com/ai-dynamo/dynamo/commit/0a9a1b6361b417af893b3068b10fba40c88b1430)

**🎯 变更类型**：功能增强（feat）  

**⚡ 重要程度**：🔴高  

**📋 变更摘要**  
在 Global Router 中新增 “priority retry” 功能：当请求在转发至选定的 pool（prefill / decode / agg）时因网络/服务错误在未产生任何流式输出前失败，系统会按照用户配置的 “池优先级” 依次尝试更快的 pool，直至成功或耗尽可用池。为此引入配置字段 `enable_priority_retry` 与 `*_pool_priorities`，并实现统一的重试调度逻辑 `_forward_with_priority_retry` 与 `get_priority_retry_order`。同时补充大量文档、配置校验和单元测试。

**🎯 影响范围**  
- `components/src/dynamo/global_router/handler.py`（请求转发与重试实现）  
- `components/src/dynamo/global_router/pool_selection.py`（配置结构、优先级校验、重试顺序计算）  
- 相关文档 `README.md`、`global-planner.md`  
- 单元测试：`test_agg_routing.py`、`test_priority_retry.py`、`test_priority_routing.py`  

**🔍 技术洞察**  

- **架构影响**  
  - **路由层引入冗余路径**：原先仅“一次转发 + 错误直接上报”，现在在全局路由层实现了多次转发尝试，保持对后端 pool 的透明度。  
  - **新增配置模型**：`enable_priority_retry` 与 `*_pool_priorities` 被加入 `GlobalRouterConfig`，并在 `validate()` 中统一校验，确保配置一致性。  
  - **统一重试抽象**：`_forward_with_priority_retry` 把 prefill、decode、agg 三类请求的转发与重试抽象为同一实现，降低代码重复度并提升可维护性。  
  - **兼容性**：默认 `enable_priority_retry=False`，且若未配置 `*_pool_priorities`，系统自动使用池顺序作为默认优先级，保持向后兼容。  

- **性能影响**  
  - **成功请求的延迟提升**：在失败后重试至更快池会产生额外的网络往返与请求构造开销，但相较于直接返回错误，可显著降低整体错误率与用户感知的响应时间。  
  - **错误请求的吞吐提升**：对瞬时故障（如节点宕机、网络抖动）可快速切换，减小对整体系统吞吐的冲击。  
  - **资源占用**：每一次重试都会创建新的 `Client.generate` 流，若频繁重试可能导致短暂的连接/线程激增；建议在生产环境开启后监控连接池使用情况。  
  - **日志与监控**：添加了 `logger.debug/warning/error`，便于观察重试路径，但需注意日志量在高流量场景下的增长。  

- **安全考虑**  
  - **不涉及身份或权限变化**：重试仅在同一请求上下文内切换不同的 pool，未改变 JWT/租户信息。  
  - **错误暴露**：在 `yielded_output` 为 True 时（即已经开始流式返回）系统会停止重试并直接抛异常，防止在已产生部分响应后出现不一致的状态。  
  - **后端兼容性**：文档已指出当前 Dynamo 后端不支持 KV‑cache 跨 pool 重用，若后端未来实现缓存迁移，需要在 retry 逻辑中补充相应的清理/迁移步骤。  

**⚠️ 潜在风险**  

1. **重试导致请求幂等性问题**  
   - 对于非幂等的后端（如写入、状态变更的模型），在重试前必须确保后端在错误前未产生副作用。当前仅在 “未产生任何输出” 时进行重试，降低风险，但仍需后端实现幂等或提供幂等 token。  

2. **池优先级配置错误**  
   - 若 `*_pool_priorities` 与实际 `*_pool_dynamo_namespaces` 长度不匹配或出现非整数，会在 `validate()` 抛异常，导致服务启动失败。运维必须在配置变更时使用验证脚本。  

3. **错误传播延迟**  
   - 多次重试会延迟错误返回，可能掩盖后端持续不可用的根本原因，导致监控报警滞后。建议在监控层面对“重试次数>0”进行告警。  

4. **资源耗尽**  
   - 在大规模故障（多池同时不可达）场景，所有请求会依次遍历全部池，产生大量短时连接尝试，可能耗尽客户端连接池或网络带宽。  

5. **日志噪声**  
   - `logger.warning` 在每次重试都会打印，若故障频繁，日志量会激增，影响日志存储与分析。  

**💡 关注建议**  

- **监控 & 报警**  
  - 监控 `global_router.retry_attempts_total`（自增计数）以及 `global_router.retry_success_total`，并对重试率突升设置告警。  
  - 对 “重试后仍失败” 的错误率进行单独监控，及时发现后端整体不可用。  

- **配置管理**  
  - 在 CI/CD 中加入配置校验步骤，确保 `*_pool_priorities` 与命名空间数量一致。  
  - 对常用部署提供模板，默认使用 `enable_priority_retry: false`，由业务团队根据 SLA 决定开启。  

- **幂等性审查**  
  - 若业务使用会产生副作用的模型（e.g., RAG 写库），在启用 priority retry 前评估后端是否支持幂等请求。必要时在请求体中加入 `request_id` 或 `retry_token`。  

- **资源保护**  
  - 为 `Client` 实例配置合理的连接池大小与超时策略，防止在大量重试时触发连接耗尽。  
  - 考虑在 `get_priority_retry_order` 中加入最大重试次数限制（当前已由 pool 列表长度自然限制），必要时提供 `max_retry_attempts` 配置。  

- **文档与培训**  
  - 更新运维手册，明确 `*_pool_priorities` 的语义（数字越小越快）以及默认行为。  
  - 在开发者文档中提示 “若在流式返回后出现异常，请不要依赖自动重试”。  

- **后端适配**  
  - 未来若 Dynamo 后端实现 KV‑cache 跨 pool 迁移，可在 `handle_decode` 的 retry 注释中加入对应的 cache 失效处理逻辑，进一步提升重试成功率。  

通过上述措施，可在提升全局路由容错性的同时，控制潜在的性能波动与运营风险。
