### feat: Make the image field optional in DGDRs (#6557)
**SHA**: `06fc5d5` | 🔗 [查看提交](https://github.com/ai-dynamo/dynamo/commit/06fc5d5ecfd9c65a1e880f02e0ea2b3034ebb393)

**🎯 变更类型**：功能增强 / 重构 / 架构变更  
**⚡ 重要程度**：🔴高  

**📋 变更摘要**  
1. 将 `DynamoGraphDeploymentRequest`（DGDR）中的 `spec.image` 字段设为可选，新增默认化 webhook 在资源 CREATE 时自动填充默认 profiler 镜像。  
2. 迁移 DGDR API 从 `v1alpha1` 到 `v1beta1`，统一使用 `client.Reader`（`APIReader`）以避免缓存不一致。  
3. 调整验证逻辑：去除对 `spec.image` 的必填检查，仅在 webhook 未运行时在控制器层做保护。  
4. 更新 Helm chart、Sidecar 脚本、ConfigMap 生成逻辑以及 observability 计数器，以兼容新模型。  

**🎯 影响范围**  
- **Operator 代码**：`controller`, `webhook/defaulting`, `webhook/validation`, GPU discovery、资源计数器。  
- **API 版本**：`deploy/operator/api/v1beta1`（替代 `v1alpha1`）。  
- **Helm 部署**：Webhook‑CA patch、mutating webhook configuration、annotation 名称变更。  
- **测试套件**：单元测试、controller test、webhook test 全面更新。  
- **用户侧 CRD**：DGDR 资源的 `spec.image` 可省略，默认镜像在创建时注入。  

**🔍 技术洞察**  

- **架构影响**  
  - 引入 **mutating webhook**（默认化）后，Operator 依赖 webhook 链路确保 DGDR 实例在创建时拥有完整 spec，解耦了控制器对必填字段的硬性校验。  
  - 将 `client.Client` 替换为 `client.Reader`（`APIReader`）在 GPU 发现、硬件校验等场景中使用 **非缓存** 读取，提升跨‑namespace/跨‑集群一致性，避免因缓存导致的“image 未设置”假阳性。  
  - 迁移至 `v1beta1` 统一了 API，减少了旧版缓存 informer 的注册成本，改进了资源计数器的统一实现（直接使用 `v1beta1` 列表）。  

- **性能影响**  
  - **Webhook 调用** 增加一次 CREATE 请求的网络往返（≈1‑2 ms），对大规模 DGDR 创建的整体延迟几乎可以忽略。  
  - 使用 `APIReader` 读取节点信息（GPU discovery）可能比缓存读取稍慢，但因为此操作仅在首次创建或硬件信息缺失时执行，影响可接受。  
  - 侧车容器新增 `WorkingDir` 与 `--output-dir` 参数，避免了多余的文件拷贝，略微提升 profiler 生成 ConfigMap 的效率。  

- **安全考虑**  
  - **默认化 webhook** 只在 CREATE 阶段触发，且不对 UPDATE 做任何改动，降低了意外覆盖用户显式配置的风险。  
  - Helm chart 中对 `MutatingWebhookConfiguration` 进行 **双 webhook**（DGDR 和 DGD） patch，确保双方均使用相同的 CA bundle，避免因证书不匹配导致的拒绝服务。  
  - `defaultImageFor` 在 operator 版本未知（本地开发）时返回空字符串，防止自动填充未经过审计的镜像，减少 supply‑chain 风险。  

**⚠️ 潜在风险**  

1. **Webhook 未就绪**：如果默认化 webhook 启动缓慢或被误删，创建 DGDR 时 `spec.image` 仍为空，控制器会因 `imageName == ""` 报错并导致创建失败。  
2. **版本同步**：默认镜像 tag 与 Operator 版本直接绑定，Operator 版本升级后如果镜像未同步发布，可能导致找不到对应 tag 的镜像，导致作业启动失败。  
3. **兼容性**：仍保留旧的 `v1alpha1` CRD 但已不再使用，集群中若同时存在两套 CRD，可能产生混淆或误用。  
4. **APIReader 权限**：`APIReader` 需要 cluster‑wide node read 权限；若 RBAC 策略未同步，GPU discovery 可能因权限不足而报错，导致 validation 失效。  

**💡 关注建议**  

- **部署检查**：在升级 Operator 后，确认 MutatingWebhookConfiguration 中的两个 webhook（`mdynamographdeploymentk...` 与 `mdynamographdeploymentrequestv1beta1...`）均已成功注册，并且 CA bundle 已正确注入。  
- **镜像同步**：发布新 Operator 版本时同步对应的默认 profiler 镜像标签；在 CI 中加入检查，防止 “operator version → 镜像 tag” 不匹配。  
- **RBAC 审计**：确保 Operator ServiceAccount 具备 `get/list/watch` 对所有 Node 资源的权限（或通过 `APIReader`），否则 GPU discovery 将回退并触发 validation 错误。  
- **回滚方案**：若出现因默认镜像不可达导致的作业失败，可临时在 DGDR manifest 中显式声明 `spec.image`，或通过 `kubectl edit` 为已有 DGDR 补充镜像后重试。  
- **文档更新**：在用户手册中明确说明 `spec.image` 为可选字段，默认镜像仅在公开发行版（>=1.0.0）可用，开发者本地构建需要自行指定镜像。  
- **监控告警**：在 observability 中加入 “DGDR 创建失败 – spec.image 为空” 告警，帮助快速定位 webhook 未生效的情况。

### feat: add guided decoding backend config and choice support for TRT-LLM (#5762)
**SHA**: `01ecc8c` | 🔗 [查看提交](https://github.com/ai-dynamo/dynamo/commit/01ecc8c748abe1b86bf75699be6e224197440fcf)

**🎯 变更类型**：功能增强  
**⚡ 重要程度**：🔴高  
**📋 变更摘要**：  
- 在 TRT‑LLM 后端新增 *guided decoding* 配置项 `--guided-decoding-backend`，并限定可选后端为 `xgrammar` 与 `llguidance`。  
- 在请求处理层加入对 `guided_decoding.choice` 列表的兼容转换，自动拼装为安全的正则表达式并填充到 `GuidedDecodingParams.regex`。  
- LLM Worker 启动时将该配置透传给 TRT‑LLM 引擎，确保后端能开启结构化输出特性。  

**🎯 影响范围**：  
- `components/src/dynamo/trtllm/backend_args.py`（参数解析）  
- `components/src/dynamo/trtllm/request_handlers/handler_base.py`（采样参数覆盖）  
- `components/src/dynamo/trtllm/workers/llm_worker.py`（工作进程初始化）  
- 相关单元测试文件  

**🔍 技术洞察**：  
- **架构影响**：  
  - 新增 `guided_decoding_backend` 配置字段，属于 **配置层 → 工作进程 → 引擎层** 的单向传递，对现有架构无破坏性改动。  
  - `HandlerBase._override_sampling_params` 的转换逻辑保持了与 vLLM outlines 后端一致的行为，提升了多后端兼容性。  
- **性能影响**：  
  - 仅在请求到达时做一次 `choice` → 正则的字符串拼装与 `re.escape`，耗时可忽略（O(N) 与选项数量成正比）。  
  - 对后端的实际推理无额外运行时开销，唯一的潜在影响是生成的正则表达式过长导致匹配成本上升，但这受限于用户提供的 `choice` 列表长度，实际风险低。  
- **安全考虑**：  
  - 使用 `re.escape` 对 `choice` 项目进行转义，防止用户注入恶意正则模式，引入 ReDoS 的可能性。  
  - 仍然允许用户直接提供 `regex`，如果输入不受信任，仍可能出现复杂正则导致 CPU 耗尽，需要在上层（API 网关或前端）对 `regex` 长度/复杂度做限制。  
  - `guided_decoding_backend` 受限于有限枚举，避免了任意后端加载的安全风险。  

**⚠️ 潜在风险**：  
1. **后端兼容性**：如果用户开启了 `guided_decoding_backend` 但 TRT‑LLM 发行版不支持对应后端，会在启动时触发运行时错误。  
2. **正则复杂度**：当用户通过 `regex` 字段直接提供复杂表达式时，仍可能导致 ReDoS。  
3. **配置验证缺失**：当前仅在 CLI 参数层做了枚举限制，`DynamoTrtllmConfig` 结构体中未使用 `Literal` 类型，容易在代码路径外（如 JSON 配置）出现非法值。  
4. **导入未使用**：`import re` 在 `handler_base.py` 中仅用于 `re.escape`，如果后续改动移除这行会产生未使用警告。  

**💡 关注建议**：  
- **增强校验**：在 `DynamoTrtllmConfig` 的字段定义中使用 `typing.Literal["xgrammar","llguidance"]` 并在配置加载阶段进行显式校验，防止非法值渗透。  
- **正则安全阈值**：在接受用户自定义 `regex` 前加入长度或回溯深度限制（如 `len(regex) < 256`），并在日志中记录异常正则。  
- **文档更新**：在项目 README / config 文档中补充 `guided_decoding_backend`、`guided_decoding.choice` 与 `regex` 的使用说明、兼容后端列表以及安全最佳实践。  
- **回归测试**：在 CI 中加入对非法后端值、过长正则以及空 `choice` 的负面测试，确保错误路径能够友好提示。  
- **监控告警**：在 `llm_worker` 启动日志中已经输出后端名称，建议在监控平台对该日志做关键字告警，以快速定位配置错误。

  - **正常模式**：对常规推理路径几乎没有额外开销，`checkpoint_restore_engine` 为 `None` 时保持原有路径。

  - **信号处理**：仅在业务进程内部捕获 `SIGUSR1` 与 `SIGCONT`；`SIGKILL` 仍为不可捕获，符合设计预期。若系统误发送其它信号，可能导致进程异常退出。

4. **资源泄漏**：若 checkpoint 失败（Watcher 发送 `SIGKILL`），进程被强制终止，GPU 资源会被系统回收，通常安全，但在某些 GPU 驱动的异常恢复场景下可能留下残留上下文。
