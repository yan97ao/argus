### feat(replay): simulate non-zero worker startup time in offline replay (#8231)
**SHA**: `0bbba98` | 🔗 [查看提交](https://github.com/ai-dynamo/dynamo/commit/0bbba988f27c3fa542f8590ad7c95f1bfe05b625)

**🎯 变更类型**：功能增强  

**⚡ 重要程度**：🔴高  

**📋 变更摘要**  
在离线 Replay（Aggregated 与 Disaggregated 两种模式）中加入对 **Worker 启动延迟** 的模拟。新增 `WorkerReady` 事件、`pending_startup` 集合以及相应的调度、状态转移逻辑，使得新建 worker 只有在延迟结束后才会被注册到 router 并开始处理请求。`is_done` 被改为 `is_workload_done`，确保仅在业务请求全部完成后即返回 “done”，即使仍有启动中的 worker 事件残留。配套单元测试覆盖启动延迟、取消、以及完成判定等场景。

---

### 🎯 影响范围
- `lib/mocker/src/replay/offline/agg.rs`（Aggregated 运行时）  
- `lib/mocker/src/replay/offline/disagg.rs`（Disaggregated 运行时）  
- `lib/mocker/src/replay/offline/components/engine.rs`（EngineComponent）  
- `lib/mocker/src/replay/offline/components/router.rs`（Router 新增 `try_drain_pending`）  
- `lib/mocker/src/replay/offline/events.rs`（新增 `WorkerReady` 事件）  
- `lib/mocker/src/replay/offline/runtime_utils.rs`（新增/扩展 `push_worker_ready`、`pop_ready_worker_ready`）  
- 相关单元测试文件  

---

### 🔍 技术洞察  

| 维度 | 影响分析 |
|------|----------|
| **架构影响** | • 引入 **启动阶段**（`pending_startup`）作为 worker 生命周期的第三状态（Pending → Active → Removal）。<br>• 新的 `WorkerReady` 事件把 “延迟结束” 与 “注册到 router” 解耦，使调度器能够在任意时刻统一处理启动完成逻辑。<br>• `EngineComponent.apply_target_count` 现在依据 `startup_time_ms` 决定是否把新 worker 放入 `pending_startup`，并在 `mark_worker_ready` 中搬迁至活动集合。<br>• `AggRuntime`、`DisaggRuntime` 的 `drain_current_timestamp` 流程新增 `apply_worker_ready_events`，在同一时间戳内完成 **启动完成 → router 注册 → 请求驱逐** 的闭环。 |
| **性能影响** | • 额外的 `WorkerReady` 事件会占用少量堆内存和一次 `BinaryHeap` push/pop，时间复杂度仍为 `O(log N)`，对整体仿真吞吐影响可忽略。<br>• `only_worker_ready_events_remain` 在 `is_workload_done` 中遍历整个事件堆；在极端大量 `WorkerReady` 场景（如成千上万的 worker 启动延迟）可能产生 `O(N)` 检查成本，但在常规仿真规模下仍可接受。 |
| **安全考虑** | - 仅在离线 Replay 环境使用，未涉及网络或外部 I/O，**无安全风险**。<br>- 新增的 `pending_startup` 及 `WorkerReady` 逻辑若误删导致 `worker_id` 泄漏到 `router`，会导致资源泄露或错误路由。代码已通过 `debug_assert` 与单元测试防护。 |
| **可维护性** | • 将启动延迟的处理集中在 `EngineComponent` 与 `runtime_utils`，保持业务代码的可读性。<br>• 新增的 `is_workload_done` 与 `only_worker_ready_events_remain` 语义明确，避免原有 `is_done` 被误用。<br>• 单元测试覆盖 5+ 场景（有/无延迟、启动取消、提前完成等），降低回归风险。 |
| **兼容性** | - 仅在 `offline` replay 中生效；在线路径未受影响。<br>- 若第三方代码仍调用 `AggRuntime.is_done`（已为私有），兼容性无碍；公共 API (`advance_to`) 已改为返回工作负载完成状态。 |

---

### ⚠️ 潜在风险  

1. **遗漏调用 `mark_worker_ready`**  
   - 若未来在别的 runtime（如自定义扩展）中新增 worker 启动延迟但忘记在对应的 `apply_worker_ready_events` 中调用 `engine.mark_worker_ready`，会导致 worker 永远停留在 `pending_startup`，进而使仿真卡死。  
2. **事件堆中仅 `WorkerReady` 的判定 O(N)**  
   - 在极端大规模仿真（数十万 worker）下，`only_worker_ready_events_remain` 每次 `advance_to` 都遍历全部事件，可能成为性能瓶颈。  
3. **`startup_time` 为 `0` 或负数**  
   - 代码已在 `startup_time_ms` 中过滤 `<=0`，但若外部配置产生 `NaN` 或异常值，可能导致 `Some(NaN)` 被当作有效延迟，进而产生无限期的 `WorkerReady`。  
4. **事件序列号冲突**  
   - `push_worker_ready` 与现有 `push_worker_completion` 共用 `next_event_seq`，若未来出现自定义事件在同一时间点加入且未正确维护顺序，可能导致 `BinaryHeap` 失去“先入先出”保证。  

---

### 💡 关注建议  

1. **审计所有调用 `apply_target_count` 的路径**，确保在新增 `EngineComponent` 后仍保持期望的“立即可用”或“需延迟”行为。  
2. **监控 `BinaryHeap` 大小**：在极端规模测试中评估 `only_worker_ready_events_remain` 的耗时，若超过阈值，可考虑维护一个额外计数器在 `push_worker_ready`/`pop_ready_worker_ready` 时同步更新，以 O(1) 判定。  
3. **配置校验**：在 `MockEngineArgs` 解析阶段加入对 `startup_time` 的合法性检查（非负、非 NaN），提前失败。  
4. **文档更新**：在项目的 Replay 使用手册中注明 `startup_time` 对仿真结果的影响，以及 `advance_to` 现在的完成判定语义。  
5. **回归测试**：在 CI 中加入跨模块的集成测试，覆盖：<br>• 多阶段（Aggregated + Disaggregated）并发启动；<br>• 伸缩决策交错（scale‑up → scale‑down 在同一时间窗口）；<br>• 大批量 `WorkerReady` 事件的 `advance_to` 完成判定。  

--- 

**总体结论**：此提交为离线 Replay 引入了重要的 **启动延迟模型**，显著提升仿真对真实系统伸缩行为的逼真度。实现上保持了低耦合、清晰的事件驱动流程，并通过完整的单元测试保障功能正确。风险主要集中在大规模仿真时的判定成本以及配置合法性检查，建议在后续迭代中针对这两点做轻量优化与防御性编程。

### fix(runtime): lock-free TCP connection pool for OOM prevention and inflight/resource-leak fixes (#7806)
**SHA**: `6633be5` | 🔗 [查看提交](https://github.com/ai-dynamo/dynamo/commit/6633be54babe58de75da6f92d3a0bfdb6ea4bf29)

**🎯 变更类型**：Bug修复 / 性能优化 / 重构 / 架构变更  
**⚡ 重要程度**：🔴高  
**📋 变更摘要**：  
1. 在 KV 路由层重构请求/调度清理逻辑，移除 `PendingDispatchGuard`，将资源释放与会话关闭统一交给 `RequestGuard` 的 `Drop` 实现，并新增 `dispatched` 标记防止在调度失败时错误计数与监控。  
2. 为 TCP 客户端实现无锁连接池（使用 `crossbeam-queue`），并在运行时新增 `start_warmup` 接口以提前预热后端连接，避免 OOM 与连接泄漏。  
3. 在多个子 Crate 中加入 `crossbeam-queue` 依赖，实现上述 lock‑free 队列。

**🎯 影响范围**：  
- `lib/llm/src/kv_router/push_router.rs`（请求调度、指标记录）  
- `lib/runtime/src/pipeline/network/egress/tcp_client.rs`（TCP 客户端实现）  
- `lib/runtime/src/pipeline/network/egress/unified_client.rs`（新增 `start_warmup` 默认实现）  
- `runtime` 与所有绑定 Crate（因为新增 `crossbeam-queue` 依赖）  

**🔍 技术洞察**  

- **架构影响**  
  - **请求生命周期统一**：以前的 `PendingDispatchGuard` 负责在调度成功或失败后执行 `free` 与 `SessionCloseAction`，在异常路径下会出现泄漏。现在这些职责统一到 `RequestGuard::Drop`，并在创建 guard 前就准备好，以保证 **无论 `direct()` 成功与否**，资源都会在 guard 被 drop 时被回收。  
  - **连接池抽象**：引入 `crossbeam-queue` 实现的无锁队列，取代以前可能包含全局锁的 TCP 连接池。新增 `start_warmup` 方法，使运行时能够在发现新后端实例后提前建立 TCP 连接，降低首次请求的延迟并平摊连接建立成本。  
  - **依赖层面**：在 `runtime`、`bindings/kvbm`、`bindings/python` 等子项目统一引入 `crossbeam-queue`，保持 Cargo 工作空间的一致性，避免因版本冲突导致的编译差异。  

- **性能影响**  
  - **锁竞争消除**：无锁队列（`crossbeam-queue::SegQueue`/`ArrayQueue`）在高并发请求下的入队/出队开销远低于 `Mutex<Vec<_>>` 之类的实现，尤其在大量短生命周期的 TCP 连接创建/复用场景，可显著降低 CPU 占用。  
  - **OOM 防护**：旧实现在高并发时会因为未及时回收连接而导致 `Vec` 持续增长，触发 OOM。新实现通过固定容量的无锁队列以及主动 `start_warmup`，在连接池满时直接复用而不再无限扩容，从而降低峰值内存。  
  - **监控准确性**：`dispatched` 标记避免在调度失败（如 `direct()` 报错）时错误计数 `requests_total`、OSL 直方图零值以及 `tracker.record_finish()`，提升监控数据的真实性，间接帮助容量规划与自动扩缩容决策。  

- **安全考虑**  
  - **资源泄漏关闭**：所有资源（调度 slot、SessionClose RPC）现在必定在 `Drop` 中执行，降低因异常路径导致的后端卡死或会话残留。  
  - **异步任务调度**：`tokio::runtime::Handle::try_current()` 仍然被用于在 `Drop` 中生成后台任务。如果在非 Tokio 运行时环境下使用（例如单线程测试），会出现警告并 **不执行** 清理逻辑。虽然不会导致内存泄漏（由于 `Drop` 本身仍会执行同步部分），但可能导致后端调度状态未被释放，建议在测试/非运行时场景中显式提供 Runtime。  
  - **跨线程安全**：`crossbeam-queue` 本身对 `Send/Sync` 已经做了严格保证，加入后不应产生新的数据竞争。但需要确保在 `TcpClient` 的其它成员（如 socket句柄）仍然满足 `Send + Sync`，否则会在编译时被捕获。  

**⚠️ 潜在风险**  

1. **Drop 中的异步 spawn 失效**：若 `Drop` 在没有活跃 Tokio 运行时的线程上执行（例如在同步单元测试或在 `Runtime::block_on` 之外），`Handle::try_current()` 会返回错误，导致 clean‑up 任务被跳过，仅留下同步部分。调度 slot 仍会被 `free`（同步），但 `deferred_close.execute` 可能被遗漏，从而导致后端 KV 会话未关闭。  
2. **无锁队列容量误配**：`crossbeam-queue` 默认使用 `SegQueue`（无限增长）或 `ArrayQueue`（固定容量）。如果实现者在 `tcp_client.rs` 选用了 `ArrayQueue` 并设置过小的容量，可能出现“连接池饱和”导致请求阻塞或频繁建立新连接，反而降低性能。  
3. **跨 Crate 依赖冲突**：新增的 `crossbeam-queue` 版本为 `0.3`，如果项目中其他 Crate 仍依赖老版本（如 `0.2`），可能触发 Cargo lock 冲突或二进制兼容性问题。  
4. **监控逻辑回退**：`record_metrics` 现在依赖 `self.dispatched` 标记，若未来某处忘记在成功路径前将 `dispatched` 设为 `true`，会导致合法请求不计入监控，产生 “流量低估” 的假象。  

**💡 关注建议**  

- **测试覆盖**：  
  - 为 `RequestGuard::Drop` 增加单元测试，分别在有/无 Tokio runtime、`direct()` 成功与失败两条路径验证 `chooser.free` 与 `deferred_close.execute` 是否被调用。  
  - 为 TCP 客户端的无锁连接池加入并发压力测试，验证在高并发（≥10k 并发请求）下不会出现内存泄漏或死锁。  
- **运行时检查**：在项目的启动阶段打印或监控 `tokio::runtime::Handle::try_current()` 是否可用，若不可用，可考虑在 `Drop` 中回退到同步执行（例如使用 `tokio::spawn_blocking`）。  
- **配置审计**：在 `runtime` 的配置文件或环境变量中加入 `TCP_CONNECTION_POOL_CAPACITY`（若使用 `ArrayQueue`），并在文档中说明容量调优原则，以防因容量过小导致性能回退。  
- **依赖统一**：在根 `Cargo.toml` 中显式声明 `crossbeam-queue = "0.3"`，并在工作空间的 `Cargo.lock` 中统一锁定版本，避免子 Crate 自动升级产生不一致。  
- **监控仪表盘**：在 Prometheus/Grafana 面板中加入 `requests_dispatched_total`（标记成功调度）与 `requests_failed_before_dispatch_total`（调度前失败）两类指标，以便快速发现新代码是否意外屏蔽了合法请求的计数。

综上，此次提交通过 **统一资源回收、消除锁竞争、提前预热连接** 三大手段，显著提升了系统的 **可靠性、内存使用安全性与高并发性能**。只要在上述风险点上做好测试与配置，即可安心在生产环境部署。

- **性能影响**  
  - 通过 **避免全量重建 `ActiveSequencesMultiWorker`**（仅在 `add_worker` 时插入新的 worker ID），显著降低扩容/缩容时的 CPU 与内存开销，尤其在大规模 worker 集合（数十甚至上百）时。

- **安全考虑**  
  - 变更主要是功能与性能层面，不涉及外部输入验证或加密路径，未引入新的安全风险。  
  - 代码路径中对 `router.as_mut()` 加了 `Option` 检查，避免在无路由器实例时出现空指针 panic，提升了 **鲁棒性**。  

**⚠️ 潜在风险**  
1. **向后兼容性**：`add_worker` 行为从 “重新构建 slots” 改为 “register_external_workers”，若外部代码依赖旧实现的副作用（例如在 `add_worker` 后立即查询 `slots` 的内部 DP‑range）可能出现差异。

### fix(discovery): spawn handle_put tasks concurrently to prevent replay wedging (#7931)
**SHA**: `470bb48` | 🔗 [查看提交](https://github.com/ai-dynamo/dynamo/commit/470bb48f0ec6771c64e8c2a6ee0e33c5176cabf1)

**🎯 变更类型**：Bug修复 / 性能优化  

**⚡ 重要程度**：🔴高  

**📋 变更摘要**  
1. 在模型发现（`ModelWatcher`）流程中，将 `handle_put` 的处理改为并发任务，并使用 `pending_puts` 记录在执行中的任务，以防止单个模型的慢速下载（如 HuggingFace 配置）阻塞后续模型的发现事件。  
2. 引入 `RegistrationGuard` RAII 结构，确保 `registering_worker_sets` 在异常或 panic 时能够及时清理，避免永久性“中毒”。  
3. 对并发注册冲突提供恢复逻辑 `recover_concurrent_registration`，处理已在进行的模型注册竞争并保证校验和一致性。  
4. 调整入口（gRPC、HTTP）和测试代码，使 `ModelWatcher` 通过 `Arc` 共享并可多任务使用。  
5. 将 KV‑store 事件通道容量从 1024 提升至 16384，以降低 KV 重放阶段的发送超时风险。  

**🎯 影响范围**  
- `lib/llm/src/discovery/watcher.rs`：核心模型发现与注册逻辑。  
- `lib/llm/src/entrypoint/input/grpc.rs`、`http.rs`：服务入口对 `ModelWatcher` 的使用方式。  
- `lib/llm/tests/http_metrics.rs`：单元/集成测试。  
- `lib/runtime/src/storage/kv.rs`：KV 事件广播通道容量。  

**🔍 技术洞察**  

| 维度 | 影响 |
|------|------|
| **架构影响** | - 将 `ModelWatcher` 由单实例同步调用转为 **并发任务**，对外保持同一 API（`Arc<Self>`），增强了系统的 **可伸缩性** 与 **容错性**。<br>- `pending_puts`（`DashMap<String, JoinHandle<()>>`）实现了 **写入-删除的同步桥梁**，保证 `Removed` 事件在对应 `Added` 仍在进行时能够等待其完成，避免“悬挂”模型状态。<br>- `RegistrationGuard` 防止在异常路径下忘记清理 `registering_worker_sets`，降低 **资源泄漏 / 死锁** 风险。 |
| **性能影响** | - `handle_put` 不再串行执行，单个模型的慢网络 IO（如大模型配置下载）不再阻塞后续模型的发现，显著提升 **发现吞吐** 与 **启动并发度**。<br>- `pending_puts` 采用 `DashMap`（无锁并发哈希表），插入/删除开销极低，整体开销可忽略。<br>- KV 事件通道扩容至 16384，降低 **发送阻塞** 与 **事件丢失** 的概率，尤其在系统启动或大量 KV 重放时表现更稳健。 |
| **安全考虑** | - 新增的并发任务使用 `tokio::spawn`，默认在同一线程池内运行，没有引入跨进程或跨网络的安全风险。<br>- `recover_concurrent_registration` 对 **校验和不匹配** 的情况进行显式错误日志并返回 `false`，防止出现版本不一致的 WorkerSet，保持 **模型一致性**。<br>- `JoinHandle::abort` 用于取消已过期的旧任务，避免旧任务在完成后误删或误更新 `pending_puts`，防止 **竞态条件** 导致的状态错乱。 |
| **可维护性** | - RAII Guard（`RegistrationGuard`）和专门的恢复函数把 **错误清理** 与 **竞争恢复** 逻辑抽离，代码结构更清晰，后续改动更易定位。<br>- 通过 `Arc<Self>` 公开的接口保持向后兼容，调用方无需修改业务逻辑，仅在入口处包装一次 `Arc`。 |

**⚠️ 潜在风险**  
1. **资源泄漏风险**：如果 `handle_put` 产生的任务长期卡住（例如网络永久超时），`pending_puts` 中对应的 `JoinHandle` 仍会保留，可能导致内存增长。建议对任务设置超时或在 `handle_put` 内部加入超时处理。  
2. **任务竞争**：在极端高并发场景下，多个 `Added` 事件可能几乎同时写入 `pending_puts`，旧任务被 `abort` 后仍有可能在 `await` 前完成，导致 `remove` 调用移除新任务的句柄。当前实现是先 `remove` 再 `abort`，风险可接受，但若出现异常行为，需在 `abort` 前确保已获取旧句柄的唯一所有权。  
3. **通道容量提升**：将 KV 事件通道扩容至 16384 会占用更多内存（每个 `WatchEvent` 复制一次），在极端负载下可能导致内存压力，需要监控实际使用情况。  
4. **错误传播**：`handle_put` 任务内部的 panic 会导致 `JoinHandle` 报错并被忽略（`let _ = handle.await;`），若出现未捕获的 panic，可能导致模型状态不完整。建议在任务内部捕获并记录错误，或在 `await` 时检查 `Result::Err`。  

**💡 关注建议**  
- **监控与超时**：为 `handle_put` 任务加入统一的超时（如 `tokio::time::timeout(Duration::from_secs(30), ...)`），并在超时后记录并清理对应的 `pending_puts` 条目。  
- **审计日志**：在 `recover_concurrent_registration` 中记录每一次竞争的详细信息（模型名、命名空间、冲突次数），帮助运维排查因并发导致的异常。  
- **负载测试**：在 CI 中加入高并发模型发现的压力测试，验证在 10k+ `Added` 事件快速到达时系统仍能保持稳定。  
- **资源限制**：考虑对 `pending_puts` 中的并发任务数量设上限（如使用 `Semaphore`），防止单节点因过多并发下载导致 IO 饱和。  
- **回退方案**：若在生产环境观察到异常增长的 `pending_puts`，可以快速回滚到原来的串行 `handle_put` 逻辑，只需移除 `Arc` 包装和并发调度代码。  

整体来看，此次改动显著提升了 **模型发现的并发能力** 与 **系统鲁棒性**，在保持接口兼容的前提下解决了因单模型下载慢导致的 **发现卡住**（replay wedging）问题，同时引入了必要的竞争恢复与资源清理机制。只要注意上述潜在风险并配合监控/超时措施，即可实现安全、高效的上线。
