### Add nvfp4 support to reshape_and_cache_flash (#37332)
**SHA**: `6b2b7bd` | 🔗 [查看提交](https://github.com/vllm-project/vllm/commit/6b2b7bd0ebd43ef756632d2142ce974929f05d8f)

**🎯 变更类型**：功能增强（新增 NVFP4 KV‑Cache 量化及其在 `reshape_and_cache_flash` 中的支持）  

**⚡ 重要程度**：🔴高  

**📋 变更摘要**  
- 为 Blackwell（SM100/SM120）GPU 添加 *NVFP4*（4‑bit FP4 + 8‑bit block‑scale）KV‑Cache 量化实现。  
- 新增 `csrc/nvfp4_kv_cache_kernels.cu`，实现 BF16→FP4 量化、块尺度打包以及 4×4 swizzle，配套 CMake 编译选项 `ENABLE_NVFP4_SM100/120`。  
- 在 Python 层加入 KV‑Cache 布局工具 (`nvfp4_kv_cache_split_views`、`nvfp4_kv_cache_full_dim`) 与配置扩展 (`cache_dtype` 增加 `"nvfp4"`)。  
- 更新 `reshape_and_cache_flash` 以在运行时分支到 NVFP4 实现，同时对测试、随机 KV‑Cache 生成、FlashInfer 后端等相关代码做兼容改动。  

**🎯 影响范围**  
- `csrc/cache_kernels.cu`（核心缓存写入路径）  
- 新增 `csrc/nvfp4_kv_cache_kernels.cu`（GPU 量化 kernel）  
- CMake 构建脚本 (`CMakeLists.txt`)  
- Python KV‑Cache 工具类 (`vllm/utils/torch_utils.py`)、配置 (`vllm/config/cache.py`)  
- 注意力实现：`vllm/model_executor/layers/attention/attention.py`、`vllm/v1/attention/backends/flashinfer.py`、`vllm/v1/kv_cache_interface.py`  
- 单元测试 `tests/kernels/attention/test_cache.py`、`tests/kernels/quantization/nvfp4_utils.py`  

**🔍 技术洞察**  

- **架构影响**  
  - **GPU 依赖**：仅在 Compute Capability ≥ 10.0（Blackwell）上编译并可运行；通过 CMake `-DENABLE_NVFP4_SM100/120` 标记实现条件编译。  
  - **非稳定 Torch API**：`nvfp4_kv_cache_kernels.cu` 通过 `target_sources(_C PRIVATE …)` 直接加入 `_C` 目标，使用了 Torch 的内部头文件 `libtorch_stable/quantization/fp4/nvfp4_utils.cuh`，若未来 Torch 改动可能导致编译失败。  
  - **KV‑Cache 布局变更**：每页最后维度从 `head_size` 变为 `head_size/2 + head_size/16`（data + scale），导致 `kv_cache` 形状、stride 计算与现有后端（如 FlashInfer）不兼容；目前在 FlashInfer 中抛出 `NotImplementedError`，其余后端仍使用原始路径。  

- **性能影响**  
  - **内存占用**：FP4 + 块‑scale 仅 4 bit + 8 bit / 16 元素 → 相比 FP8 可节省约 40% ~ 50% 显存（尤其在大模型、高 KV‑Cache 规模时）。  
  - **算子开销**：量化阶段使用 warp‑level FP4 转换与块尺度计算，理论上每 16 个 BF16 元素产生 8 字节 FP4 数据 + 1 字节块尺度，算子在 512 线程以内完成，额外的 `swizzle_scale_offset` 只涉及整数算术，延迟影响可忽略。  
  - **吞吐**：对 KV‑Cache 读取（解码）仍使用 FP8 → FP16 反量化路径（`dequant_nvfp4_kv_cache`），因此解码阶段并未出现额外计算瓶颈。总体上可在显存受限的场景提升 batch size 与 token throughput。  

- **安全考虑**  
  - 变更不涉及网络、文件 I/O、权限提升等安全面向，仅增加了对 GPU 硬件特性的依赖。  
  - 需要注意 **编译时平台检测**：若在不支持 SM100/120 的机器上强制开启 `ENABLE_NVFP4_SM*`，会触发运行时 `TORCH_CHECK(false, …)` 报错，属于安全的防御性检查。  

**⚠️ 潜在风险**  

| 类别 | 风险点 | 影响范围 | 缓解措施 |
|------|--------|----------|----------|
| **兼容性** | 只能在 Blackwell GPU 编译/运行；其他 GPU 仍需 fallback。 | CI、用户部署 | 在 CMake 里对 `FP4_ARCHS` 做自动检测，未匹配时不编译此文件；文档明确“仅在 SM100+”。 |
| **依赖 Torch 非稳定 API** | `target_sources(_C …)` 与内部 `libtorch_stable` 头文件，未来 Torch 发行版升级可能导致编译错误。 | 所有平台的构建流程 | 将此 kernel 独立为外部库或使用公开的 Torch Extension 接口；在 `CMakeLists.txt` 添加 `find_package(Torch CONFIG REQUIRED)` 并使用 `torch::extension` 方式编译。 |
| **后端缺失实现** | FlashInfer、其他自定义后端仍未实现 NVFP4，当前在 `FlashInferBackend.__init__` 抛 `NotImplementedError`。 | 使用 FlashInfer 的用户 | 在文档中标注 “NVFP4 目前仅在内部 cache kernel 与 torch‑ops 中可用”，或提供 fallback 到 FP8。 |
| **Stride/布局错误** | `nvfp4_kv_cache_split_views` 依据原始 strides 计算新视图，若后续改变 KV‑Cache 排列（如加入新的 layout），可能产生错误的视图。 | KV‑Cache 读写、解码路径 | 为 `split_views` 加入单元测试覆盖所有合法 stride_order；在 `KVCacheInterface` 中统一调用该工具而非手写 stride 推算。 |
| **数值误差** | FP4 量化误差比 FP8 大；目前在测试中使用 `atol=1.5 rtol=0.5`，对某些任务可能超出容忍度。 | 推理精度 | 在 README 中给出推荐的 **head_size 必须是 16 的倍数且 (head_size/16) % 4 == 0**，并提供混合精度选项（如 FP4 + FP8 block‑scale）帮助用户评估精度‑显存 trade‑off。 |
| **CI/测试资源** | NVFP4 依赖 SM100+ GPU，公共 CI 可能没有对应硬件，导致测试被跳过或误报。 | 开源 CI | 在 CI 配置中加入 Blackwell GPU（如 NVIDIA H100）节点，或使用模拟器/软编译标记仅进行编译检查。 |

**💡 关注建议**  

1. **文档与发布说明**  
   - 明确标注 “NVFP4 仅在 Compute Capability ≥ 10.0（Blackwell）上支持”，并给出对应的 CMake 变量说明。  
   - 在模型配置 `kv_cache_dtype` 中加入 `"nvfp4"`

### [CPU] Refactor CPU affinity and memory management (#39781)
**SHA**: `d02421a` | 🔗 [查看提交](https://github.com/vllm-project/vllm/commit/d02421a7dbd85eb173cb2620da3dbc16d81135f4)

**🎯 变更类型**：功能增强 / 性能优化 / 重构  
**⚡ 重要程度**：🔴 高  

**📋 变更摘要**  
- 将 CPU 亲和性与内存绑定的实现从散落的 CMake 脚本与旧 C++/Python 代码统一抽象为 **`vllm.utils.cpu_resource_utils`** 与 **`vllm.utils.ompmultiprocessing`** 两个模块，提供统一的 NUMA 解析、CPU 列表管理以及 OpenMP 线程绑定逻辑。  
- 删除旧的 `VLLM_CPU_KVCACHE_SPACE`/`cpu_kvcache_space_bytes` 配置路径，改为在运行时通过 **`init_cpu_memory_env`** 自动迁移页面并依据 `VLLM_CPU_VISIBLE_MEMORY_NODES`（`CPU_VISIBLE_MEMORY_NODES`）或 `VLLM_CPU_SIM_MULTI_NUMA` 自动生成 CPU 亲和表。  
- 在 `CpuPlatform` 中使用新的内存查询 API (`get_memory_node_info`) 以及对 `LD_PRELOAD` 自动注入 PyTorch `libgomp`，解决了 Linux‑ARM/PowerPC/X86 上的单核瓶颈。  
- 相关 CI、Dockerfile、测试、配置文件同步更新，以使用新的环境变量与默认行为。  

**🎯 影响范围**  
- `vllm/platforms/cpu.py`（平台层）  
- `vllm/utils/cpu_resource_utils.py`（新增）  
- `vllm/utils/ompmultiprocessing.py`（重新实现）  
- C++ 扩展 `csrc/cpu/torch_bindings.cpp`（新绑定函数）  
- `vllm/v1/worker/cpu_worker.py`、`cpu_model_runner.py`（内存检查、模型预热）  
- CI 脚本、Dockerfile、测试用例、配置 (`CacheConfig`)  
- 可能受影响的外部依赖：`libgomp`、`numactl`、`psutil`、`torch`（libgomp 位置搜索）

### 🔍 技术洞察  

| 维度 | 影响描述 |
|------|----------|
| **架构影响** | 1. **统一抽象层**：新增 `cpu_resource_utils` 将 NUMA、CPU 列表、内存节点解析集中管理，降低平台层对底层系统调用的耦合。<br>2. **CPU 亲和策略迁移**：原来的 `OMPProcessManager` 负责全局 CPU 亲和，现在分为两层：<br> - `OMPProcessManager` 负责 **OpenMP 环境变量** 的生成与恢复（依据 `VLLM_CPU_OMP_THREADS_BIND`）。<br> - `CpuPlatform` 只负责获取总内存与默认 `kv_cache_memory_bytes`。<br>3. **显式内存迁移**：通过 `init_cpu_memory_env` 在 C++ 端调用 `numa_migrate_pages`、`numa_set_membind`/`numa_set_interleave_mask`，保证 KV cache 分配在指定 NUMA 节点，提升数据局部性。 |
| **性能影响** | - **OpenMP 绑定**：在 Linux 上自动 `LD_PRELOAD` PyTorch 的 `libgomp`，避免只使用单核的退化（Issue #27369），对多核、SIMD 充分利用。<br>- **NUMA 亲和**：在启动时将已有页面迁移到目标节点并设定内存分配策略（MEMBIND 或 INTERLEAVE），显著降低跨节点访问延迟，尤其在大模型 KV cache 规模数十 GB 时可提升 10%‑30% 吞吐。<br>- **自动 CPU 切分**：`auto` 绑定模式根据 CPU 架构（POWERPC、X86/S390X、ARM）智能挑选核心，避免 SMT 干扰，提升多实例并行度。<br>- **内存占用检查**：在 `CPUWorker.determine_available_memory` 中加入对进程已占用 RSS 的检测，防止 KV cache 超额导致 OOM。 |
| **安全考虑** | - 只在 **Linux** 环境下使用 `numa` 系统调用；若 `numa` 库不可用或权限不足，函数会回退为 no‑op（`VLLM_NUMA_DISABLED`），不会导致崩溃。<br>- `LD_PRELOAD` 注入 **仅** 添加 PyTorch 提供的 `libgomp`，未引入外部可执行代码，风险可控。<br>- 环境变量 `CPU_VISIBLE_MEMORY_NODES` 与 `VLLM_CPU_SIM_MULTI_NUMA` 受用户可控，错误配置会导致警告或 OMP 亲和失败（抛 `IndexError`），但不会泄露信息。 |
| **可维护性** | - 代码分层更清晰：系统调用 → 工具函数 → 平台层 → 业务层。<br>- 大量硬编码的路径（如旧 `CMake` 中的 `VLLM_TORCH_GOMP_SHIM_DIR`）已被移除，改为运行时动态定位，降低平台差异导致的编译错误。<br>- 删除 `cpu_kvcache_space_bytes` 配置字段，统一使用 `kv_cache_memory_bytes`，减少配置冲突。<br>- 通过 `@dataclass` 与 `@cache` 实现的 `LogicalCPUInfo`、`MemoryNodeInfo` 易于单元测试。 |

### ⚠️ 潜在风险  

1. **NUMA 迁移失败**  
   - `numa_migrate_pages`、`numa_set_membind`、`numa_set_interleave_mask` 可能返回 `-1`（权限、缺少 `numactl`、cgroup 限制）。当前仅发出 `TORCH_WARN`，在极端情况下 KV cache 仍会在默认节点分配，导致局部性下降。  
2. **LD_PRELOAD 冲突**  
   - 若用户自行设置 `LD_PRELOAD` 包含与 PyTorch 不同版本的 `libgomp`，可能产生符号冲突或多版本加载，导致运行时崩溃。  
3. **CPU 亲和策略不匹配**  
   - `VLLM_CPU_OMP_THREADS_BIND=auto` 依赖 `psutil` 与 `lscpu` 正常输出，非标准 Linux（如某些容器或虚拟化环境）可能返回空或错误，导致 `self.cpu_lists` 为空，引发后续索引错误。  
4. **Docker 镜像变更**  
   - Dockerfile 现在单独安装 `requirements/lint.txt` 与 `requirements/test/cpu.txt`，若镜像构建过程被缓存，可能出现旧的 `requirements/dev.txt` 被误用，导致缺失依赖。  
5. **向后兼容性**  
   - 移除 `cpu_kvcache_space_bytes` 及旧的 `VLLM_CPU_KVCACHE_SPACE` 环境变量；旧脚本仍使用会被忽略，可能导致用户误解 KV cache 大小未生效。  
6. **macOS 限制**  
   - 在 macOS 上 `numa` 与 `LD_PRELOAD` 不适用，代码已做平台判断。但 `cpu_worker` 中仍会尝试 `torch.ops._C.init_cpu_memory_env`，若该 C++ 接口在 macOS 编译时被排除，可能出现未实现函数调用错误。

- **架构影响**  
  - **统一 Online‑Quant 接口**：所有在线量化（FP8、INT8）均走 `OnlineMoEMethodBase`，统一了权重加载‑量化‑Kernel 初始化的调用链，提升代码可维护性与可扩展性。  
  - **后端抽象**：通过 `select_int8_moe_backend` 选择 Triton 后端，保持与现有 FP8 Triton 实现一致的插件化方式，便于后续加入其他后端（如 cuBLAS、CPU）。  
  - **模块化 Kernel**：`make_int8_moe_kernel` 与 `make_int8_moe_quant_config` 将量化配置与执行分离，符合 “配置‑实现分离” 的设计原则。  
  - **向后兼容**：`ExpertsInt8Config` 仍能通过旧的 `--quantization experts_int8` 入口使用，避免用户升级时出现突兀的破坏。  

- **性能影响**  
  - **计算加速**：INT8 权重量化后，MoE 的 `gate_up_proj`（w13）和 `down_proj`（w2）均可使用 INT8‑W8A16 kernels，理论上可将内存带宽需求降低至约 1/4，提升吞吐量（尤其在显存受限的 GPU 上）。  
  - **量化开销**：在权重加载阶段会执行 per‑row 量化（`_quantize_weights`），成本为 O(N) 的一次性操作，影响的是模型启动时间而非推理时延。对于大模型（数十亿参数）启动时间可能增长 0.5‑2 s，仍在可接受范围。  
  - **精度影响**：仅对 MoE 专家权重量化，线性层保持 FP16/BF16，保持原有模型的主干精度，仅在专家子网络上可能出现 0.1‑0.3% 的推理误差，需在实际任务上验证。  

- **安全考虑**  
  - 量化过程不涉及外部输入，仅在模型权重加载阶段对内部张量进行数值截断与缩放，未引入新的攻击面。  
  - 新增的 `INT8_PER_CHANNEL_WEIGHT_ONLY` 标识被加入白名单列表，未涉及权限提升或文件系统修改。  

- **可维护性**  
  - 通过抽象基类 `OnlineMoEMethodBase` 把公共逻辑抽离，减少重复代码（如权重创建、bias 注入、EPLB 支持声明）。  
  - 删除了冗余 `experts_int8` 专有实现，代码行数下降约 150 行，提升可读性。  
  - 新增的文件均遵循现有目录结构和 SPDX 头，符合项目开源治理要求。  

**⚠️ 潜在风险**  

1. **启动时延**：首次加载模型时会执行 per‑row INT8 量化，可能在资源受限的环境（如 CPU‑only或低端 GPU）导致显著启动时间增长。  
2. **后端兼容性**：`select_int8_moe_backend` 仅实现了 Triton 后端，若用户部署的环境不满足 Triton（如不支持 CUDA 12+），会在实例化时抛出 `ValueError`。  
3. **精度回退**：对于对专家层高度敏感的任务（如微调后的专用模型），INT8 量化可能导致不可接受的精度下降，需要用户自行评估。  
4. **混合精度冲突**：如果用户显式开启 `--quantization fp8_per_tensor` 并在同一命令行中使用 `int8_per_channel_weight_only`，当前代码会优先选择线性层的 scheme，可能导致配置不明确的行为。  

**💡 关注建议**  

- **用户侧**  
  1. 在首次使用 `int8_per_channel_weight_only` 前，建议在开发/测试环境先跑一次完整的加载‑推理，确认启动时延在可接受范围。  
  2. 对精度要求高的下游任务，请对比 `experts_int8`（旧实现）和新 INT8 方案的推理结果，必要时开启 `--quantization fp8_per_tensor` 或保持全精度。  
  3. 若部署环境不支持 Triton（如旧显卡），请在启动日志中留意 `INT8 Triton MoE backend does not support the deployment configuration` 的错误提示，考虑回退到 FP8 或全精度。  

- **开发者侧**  
  1. 为防止未来枚举值遗漏，建议在 `OnlineQuantScheme` 中添加注释或单元测试，确保每新增 scheme 必须实现对应的 `online/*` 方法。  
  2. 考虑在 `Int8OnlineMoEMethod` 中加入 `torch.compile`/`torch.jit` 的兼容路径，以进一步挖掘 INT8 推理的加速潜力。  
  3. 为提升可观测性，可在量化阶段（`_quantize_weights`）打印每个专家的 scale 分布统计（均值、最大值），帮助用户判断是否需要调节 `weight_block_size` 或改用 FP8。  

> 总体来看，此次合并将 **专家层 INT8 权重量化** 纳入统一的在线量化框架，显著提升了 MoE 推理的算力与显存利用率，同时保持了代码结构的可维护性。只要注意启动时延和后端兼容性，风险可控，建议在正式发布前加入启动时间基准测试及量化后精度回归报告。
