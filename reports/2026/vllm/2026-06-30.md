### fix(security): bound tokenizer work when explicit truncation_side is set (#47007)
**SHA**: `c231d1f` | 🔗 [查看提交](https://github.com/vllm-project/vllm/commit/c231d1f2906ed3f9a995d84a6c7ba67e6914fb16)

**🎯 变更类型**：Bug修复 / 安全修复  
**⚡ 重要程度**：🔴高  
**📋 变更摘要**：本次提交修复了在显式指定 `truncation_side`（左/右）时，Tokenizer 可能出现 **无限制** token 化导致资源耗尽的安全漏洞。核心改动在 `TokenizeParams.get_encode_kwargs` 中关闭 Tokenizer‑level truncation，并在 `_text_len_check` 中加入基于字符数的预裁剪逻辑，以保证在任何 `truncation_side` 下均只对可接受长度的文本进行编码。对应单元测试也同步更新，覆盖了左侧、右侧截断以及预裁剪保护场景。

---

### 🎯 影响范围
- **vllm/renderers/params.py**：`TokenizeParams` 的编码参数生成与文本长度检查逻辑。  
- **tests/renderers/test_completions.py**：新增四个测试用例，验证显式截断侧的行为以及字符数预裁剪的安全保障。  
- **渲染管线**：涉及 `Renderer.render_prompts` → `Renderer.tokenize_prompts` 的完整流程，影响所有使用 `TokenizeParams` 的调用方（包括开放接口和内部批处理）。

---

### 🔍 技术洞察
| 维度 | 影响说明 |
|------|----------|
| **架构影响** | - `TokenizeParams` 现在统一在 **参数层面** 关闭 Tokenizer 的内部截断 (`truncation=False`)，确保后续的 `_token_truncation` 能够基于用户指定的 `truncation_side` 正确裁剪。<br>- 引入字符级预裁剪 (`_text_len_check`) 作为 **防护层**，在进入 Tokenizer 前就把文本长度控制在 `max_input_tokens * tokenizer.max_chars_per_token` 范围内，降低了对 Tokenizer 实现细节的依赖。 |
| **性能影响** | - 额外的字符长度检查和可能的切片操作带来 **微乎其微** 的 CPU 开销（O(1) 字符串切片），相较于防止整段文本被完整 Tokenize 所节省的资源消耗，整体性能是正向的。<br>- 关闭 Tokenizer 的内部截断后，仍会在后期进行一次切片 (`_token_truncation`)，成本与之前相同。 |
| **安全考虑** | - 关键安全漏洞: 当 `truncation_side` 被显式设置且 Tokenizer 默认左侧截断时，原实现会 **忽略** 用户侧截断需求，导致对超长文本的完整 Tokenization，可能触发内存/CPU 耗尽的 DoS。<br>- 通过字符级预裁剪以及强制关闭 Tokenizer 的 truncation，确保即便 Tokenizer 实现有缺陷，也不会对系统造成资源暴露。 |
| **可维护性** | - 将截断策略的防御逻辑集中在 `TokenizeParams`，提升了代码可读性，降低了各调用方自行处理的风险。<br>- 新增的测试覆盖了左、右截断以及未指定截断侧的边界，为后续回归提供保障。 |

---

### ⚠️ 潜在风险
1. **字符到 Token 的比例假设不准确**：`max_chars_per_token` 采用的是 Tokenizer 提供的估算值，若实际比例显著偏离，可能导致合法请求被误拒（触发 `VLLMValidationError`）。  
2. **自定义 Tokenizer 行为差异**：若外部 Tokenizer 实现不遵循 `max_chars_per_token` 或在截断后仍返回超出预期的 token 数，`_token_truncation` 仍会裁剪，但可能出现 **token 丢失** 的业务层面异常。  
3. **截断侧与语言模型期望不匹配**：某些模型在左侧截断时会对系统提示产生影响，强制在字符层面左侧裁剪可能改变提示的语义。需要在文档中明确说明此行为。  

---

### 💡 关注建议
- **文档更新**：在 VLLM 的 Prompt/Tokenizer 章节补充说明：显式 `truncation_side` 时，VLLM 会在字符层面预裁剪并关闭 Tokenizer 自身的 truncation。  
- **参数校验**：对 `max_chars_per_token` 进行容错检查（例如最小阈值 1），防止意外配置导致过度裁剪。  
- **监控告警**：在生产环境加入对 `VLLMValidationError`（尤其是 “input_text” 长度错误）频率的监控，以快速发现潜在的异常请求或不匹配的 Tokenizer 配置。  
- **回归测试**：建议在 CI 中加入对不同模型、不同 `max_chars_per_token` 配置的组合测试，确保字符层预裁剪与实际 token 长度之间的映射保持在可接受误差范围。  
- **安全审计**：在未来的安全评估中，将此类 “输入长度预检查” 设为必审项，防止类似的资源消耗漏洞重复出现。

### [Rust Frontend] Simplify unit tests with shared `TestTokenizer` (#47125)
**SHA**: `7a327f0` | 🔗 [查看提交](https://github.com/vllm-project/vllm/commit/7a327f0b4f8f6886df77996d0e35ce547c9c2bd1)

**🎯 变更类型**：重构 / 测试设施完善  

**⚡ 重要程度**：🔴高  

**📋 变更摘要**：本次提交在 Rust 前端引入了统一的 `TestTokenizer`（位于 `vllm-tokenizer` 的 `test-utils` 特性），取代了大量散落的 ad‑hoc 假实现。所有单元测试改用该共享实现，并相应更新了 Cargo 依赖、错误信息以及部分测试逻辑。核心业务代码未受功能性影响，仅测试层面得到统一与简化。

**🎯 影响范围**：  
- `rust/src/tokenizer`（新增 `test_utils.rs`、`Cargo.toml` feature）  
- 直接依赖 tokenizer 的所有 crate：`chat`, `parser`, `text`, `server`（相应的 `Cargo.toml`）  
- 测试代码：几乎所有模块的单元测试均改为使用 `TestTokenizer`，删除原有 `FakeTokenizer`/`StubTokenizer` 等实现。  

**🔍 技术洞察**  

- **架构影响**  
  - **正向**：  
    - 引入统一的测试工具库，消除重复的 “手写假 tokenizer” 代码，提升代码可维护性。  
    - 通过 Cargo 的 `test-utils` feature 将测试实现与生产实现解耦，防止误用到发行版二进制。  
    - `TestTokenizer` 的实现更贴近真实 tokenizer 的行为（字节回退、最长前缀匹配、特殊 token 过滤），为后续新增测试提供可靠基础设施。  
  - **负向**：  
    - 增加了 `vllm-tokenizer` crate 的可选依赖及编译路径，可能稍微增长编译时间。  
    - 若误将 `test-utils` feature 打包进发布版，会导致二进制体积不必要的膨胀。  

- **性能影响**  
  - 仅限编译期和测试执行时：`TestTokenizer` 的实现相对手写的 `FakeTokenizer` 更加通用，内部使用 `BTreeMap` 查找，理论上略慢于硬编码的 `match`，但对单元测试规模而言可忽略不计。

  - 统一接口后，后续若需要扩展测试 tokenizer（如加入新特殊 token、修改未知 id 处理策略），只需在 `TestTokenizer` 中调整，所有测试自动获益。

| 风险点 | 描述 | 严重度 | 缓解措施 |
|--------|------|--------|----------|

### [FS-Offloading] Batch Lookup in C  (#46713)
**SHA**: `c8fb296` | 🔗 [查看提交](https://github.com/vllm-project/vllm/commit/c8fb2963bd1baebbdd28062097096b59b2ba3189)

**变更概览**  
本次 PR 为 `FileSystemTierManager` 新增了纯 C++ 扩展 `fs_io_C`，实现一次性批量检查文件是否存在 (`batch_lookup`) 并在构建脚本、`setup.py` 中注册，使 Python 侧能够在 GIL 被释放的情况下完成整批 `access(2)` 调用。对应的单元测试覆盖空、单、全、混合、异常输入以及在管理器中是否走 C 扩展路径。

**核心影响**  
- **构建层**：`CMakeLists.txt` 新增 `fs_io_C` 目标，只有 Python ≥ 3.11 时编译，`setup.py` 同步加入 `vllm.fs_io_C.abi3.so`。  
- **运行时**：`vllm/v1/kv_offload/tiering/fs/manager.py` 在 `batch_lookup` 中优先调用 C 扩展；若缺失则回退到生成器逐个 `os.path.exists`。  
- **测试**：在 `tests/v1/kv_offload/tiering/test_fs_tier.py` 中加入对 C 扩展的直接调用及其在 `FileSystemTierManager` 中的调度验证。

**潜在风险 & 建议**  
1. **跨平台兼容**：`access` 只能在 POSIX 环境工作，Windows 编译会失败。若项目需支持 Windows，需改为 `::_access` 或使用 `faccessat` 并加入相应 `#ifdef`。  
2. **GIL 释放安全**：当前在释放 GIL 前仅保存 `const char*` 指针，指向的 Python Unicode 对象在 GIL 释放期间可能被回收，导致悬挂指针。建议在 GIL 持有期间将路径复制为 `std::string`（或 `PyBytes_AsStringAndSize`）后再放入 `paths`，确保线程安全。  
3. **错误信息**：`batch_lookup` 对非 `list`、非 `str` 参数的错误会直接返回 `nullptr`，但未给出自定义异常信息。可在解析阶段加入 `PyErr_SetString(PyExc_TypeError, "...")`，提升调试体验。  
4. **Python 3.11 依赖**：C 扩展仅在 3.11+ 编译，低版本仍走纯 Python 路径，这一点在文档和 `setup.cfg` 中需说明，防止用户误以为所有环境都有该加速。  
5. **包装和发布**：确保 `MANIFEST.in`、`pyproject.toml` 均包含生成的 `.so`，否则 wheel 可能缺失该文件。  

总体而言，此改动提升了文件存在性检查的批量性能，代码结构清晰，测试覆盖完整。按照上述兼容性与安全性检查进行微调后即可合并。
