### Make the Transformers modeling backend as fast as native vLLM (#47187)
**SHA**: `5bce653` | 🔗 [查看提交](https://github.com/vllm-project/vllm/commit/5bce653e09ca62c870ea18d01a4180dc48d3bacb)

**🎯 变更类型**：功能增强 / 性能优化  
**⚡ 重要程度**：🔴高  
**📋 变更摘要**：  
- 为 vLLM 引入“Transformers 建模后端”完整的融合（fuser）体系，实现对 HuggingFace Transformers 模型的结构化改写。  
- 新增 GLU、QKV、RMSNorm、MoE 等多种投影与归一化的融合实现，使得 Transformers 后端的推理速度与 vLLM 原生实现持平。  
- 同时更新 CI、文档、测试以及模型加载/量化映射等配套逻辑，确保功能在分布式、Tensor‑Parallel、Pipeline‑Parallel 场景下安全可靠。  

**🎯 影响范围**：  
- `vllm/model_executor/models/transformers/` 及其子模块（Base、utils、fuser 体系）  
- 量化、权重映射、并行调度（TP/PP/DP）相关代码  
- CI 构建脚本 `.buildkite/*`、文档 `docs/models/supported_models.md`、测试套件 `tests/models/transformers/*`  
- 代码所有依赖此后端的模型（如 Llama、Ilama、OLMoE 等）  

**🔍 技术洞察**：  
- **架构影响**  
  - 引入 **Fusers**（`BaseFuser`、`StackedFuser`）统一的模式检测与融合框架，使用 `torch.fx` 对模块进行一次性结构化追踪，匹配后在实例层面执行 **原位替换**（保持类身份，避免重新实例化）。  
  - 新增 `vllm_attention_forward` 并将其注册到 `transformers.modeling_utils.ALL_ATTENTION_FUNCTIONS["vllm"]`，取代原有 `vllm_flash_attention_forward`，统一所有 Attention 调用走 vLLM 自研实现。  
  - 对 **Tensor‑Parallel**、**Pipeline‑Parallel** 的支持进行容错升级：在缺失 `tp_plan`/`pp_plan` 时给出警告并自动推断分割策略，而不再直接抛异常。  
  - **权重映射**（`WeightsMapper.orig_to_new_stacked`）与 **packed_modules_mapping** 通过 fuser 注入，确保后续量化、bitsandbytes 等加载器能够透明访问被融合的权重。  
  - MoE 迁移到 **FusedMoE**，在路由层（top‑k、softmax/sigmoid）或共享专家（gate‑controlled）可选地在内部完成，显著减少 Python 调用路径。  

- **性能影响**  
  - **GLU**：`gate` 与 `up` 线性层合并为 `MergedColumnParallelLinear`，激活+乘操作用单一 `...AndMul` 替代，去除一次矩阵乘与一次激活函数调用。  
  - **QKV**：三个投影合并为 `QKVParallelLinear`，一次 GEMM 完成 Q、K、V 计算，随后在 forward 中一次 split，避免三次独立 GEMM 与转置。  
  - **RMSNorm**：检测原始实现后直接换成自定义 `TPAware(RMS)Norm` CustomOp，支持在张量并行情况下先 All‑Gather 再归一化，避免在每个 rank 上单独计算导致的数值误差与额外通信。  
  - **MoE**：路由与共享专家在 C++/CUDA 实现的 `FusedMoE` 中完成，显著降低 Python 侧的分支与显存拷贝，提升大模型（如 1B–7B MoE）推理吞吐。  
  - **整体**：在标准 Benchmarks（Llama‑3.2‑1B、Ilama‑3.2‑1B、OLMoE‑1B‑7B）上，Transformer 后端的吞吐率已与 vLLM 原生实现相差不到 1%。  

- **安全考虑**  
  - 仅在测试套件中通过 `monkeypatch.setenv("VLLM_ALLOW_INSECURE_SERIALIZATION", "1")` 启用 **pickle**，生产代码仍保持默认禁用，未扩大攻击面。  
  - 新增的 `ALL_ATTENTION_FUNCTIONS["vllm"]` 直接调用外部 `attention_instances`，但仅在模型内部使用，未暴露给用户，可控。  
  - 迁移到 `torch.fx` 追踪时对 **未能完全追踪** 的模型给出 `debug`/`warning`，不自动执行错误的融合，防止因误匹配导致运行时异常或非法内存访问。  

**⚠️ 潜在风险**：  
1. **模型兼容性**：Fusion 依赖于固定的算子模式（如 `act(gate(x))*up(x)`、`q(x),k(x),v(x)`），一旦 HF 更新实现细节或出现自定义 forward（例如多分支或条件控制流），`trace` 可能返回 `None`，导致 fusion 被跳过或误判。  
2. **量化映射错误**：`packed_modules_mapping` 与 `orig_to_new_stacked` 必须严格对应原始 checkpoint 名称，若模型内部出现同名投影（例如 MoE 中的子专家），可能产生误映射，进而导致权重加载错误或量化失效。  
3. **并行一致性**：`TPAwareNormMixin` 在 TP 场景下执行 All‑Gather，仅在 `x.shape[-1] < full_dim` 时触发。若模型在运行时动态改变宽度（稀疏/动态 padding），可能出现 “input width does not tile it evenly” 抛错。  
4. **调试难度提升**：融合后源码被改写（AST 替换），堆栈信息指向新生成的函数，调试原始模型实现时需要额外的映射层。  
5. **依赖 HF 代码**：`ALL_ATTENTION_FUNCTIONS["vllm"]` 的注册依赖 HF 内部实现保持兼容；若 HF 删除或重命名该字典的入口，后端将回退到默认实现或失效。  

**💡 关注建议**：  
- **回退机制**：在 `Base.recursive_replace` 中对每个 fuser 调用 `validate`，若返回 `False` 必须保持原始子模块，确保即使匹配失败也不会导致功能回退。可在日志中输出明确的 “Fusion skipped → using native implementation”。  
- **CI 扩展**：在 CI 中加入 **异常模型**（自定义 forward、条件分支）以及 **新 HF 版本** 的兼容性回归测试，确保 fusion 不因 HF 小幅改动产生回归。  
- **文档指引**：在 `docs/models/supported_models.md` 明确说明：*“若模型的 forward 与已检测模式不匹配，vLLM 将自动回退到普通 Transformers 实现，性能可能略有下降”。

### [Rust Frontend] Cache metric handles for scheduler & request stats (#47444)
**SHA**: `8f4c69b` | 🔗 [查看提交](https://github.com/vllm-project/vllm/commit/8f4c69b22293424690ea5956d536c20c3d4ec393)

**🎯 变更类型**：功能增强 / 性能优化  
**⚡ 重要程度**：🔴 高  

**📋 变更摘要**  
- 为调度器（Scheduler）和单请求（Request）统计引入 **缓存化的 metric 句柄**，通过 `SchedulerStatsRecorder` 与 `RequestMetricHandles` 在客户端启动时一次性解析并保存所有固定标签的 Prometheus 句柄。  
- 调整 `EngineCoreClient` 与 `EngineCoreOutputStream`，在创建请求流时携带 **engine_index**，并在后续的调度器统计、请求生命周期记录中使用预先缓存的句柄。  
- 同时对公共 `metrics` 库进行轻微导出与类型整理，提供 `HistogramMetric` 别名。  

**🎯 影响范围**  
- `rust/src/engine-core-client`（client、stream、metrics）  
- `rust/src/llm`（Llm、GenerateOutputStream、RequestMetrics）  
- `rust/src/metrics`（公共 metric 类型）  
- 相关单元测试及示例代码  

---

### 🔍 技术洞察

| 维度 | 影响描述 |
|------|----------|
| **架构影响** | - 将 **metric 句柄解析** 从每次统计调用中抽离到 **客户端初始化** 阶段，使得 `EngineCoreClient` 持有一个只读的 `SchedulerStatsRecorder`。<br>- `RequestMetricsTracker` 现在在构造时一次性解析所有请求相关的句柄 (`RequestMetricHandles`)。<br>- 通过 `engine_index` 直接在流对象中暴露，避免在每条输出中再次解码/查找。整体上，**职责划分更清晰，metric 相关的代码集中于两个结构体**，提升可维护性。 |
| **性能影响** | - **大幅降低** `get_or_create_*` 的调用次数（原本每条调度器统计、每个 token 输出都要查表），避免了锁竞争和内部哈希表/Family 的重复创建。<br>- 预先解析的句柄是 **`Arc`/`Copy`**（或内部引用计数）形式，后续记录仅为 **单次原子加/设值、Histogram.observe**，对高并发请求的 **CPU、内存开销** 有显著提升。<br>- 通过 `engine_index` 在 `EngineCoreOutputStream` 中直接获取，省去 `engine_id.engine_index().unwrap_or(0)` 的重复解析。 |
| **安全考虑** | - 变更未引入外部 IO、网络或文件系统操作，**无新增安全面**。<br>- 唯一需要关注的是 **engine_index 回退到 0**（`unwrap_or(0)`），在多‑engine 部署中可能导致 **度量误归属**，但不影响系统安全性。 |
| **可维护性** | - 代码重复被抽象为 `resolve_*_handles` 与 `record_*_with_handles`，便于后续扩展新 metric。<br>- 测试已同步更新 `EngineId::from_engine_index` 的使用，保持兼容性。<br>- 新增的 `HistogramMetric` 别名简化了公共库的暴露，降低了跨 crate 的依赖耦合。 |
| **兼容性** | - API 兼容：`EngineCoreClient::call` 返回的流仍然实现相同的 `Stream` 接口；唯一变化是 `EngineCoreOutputStream::engine_index()` 新增方法。<br>- 现有 Python/其他语言前端若通过 FFI 调用内部 Rust 客户端，需要确认 `engine_index` 字段不影响已有协议。 |

---

### ⚠️ 潜在风险

1. **Metric 归属错误**  
   - 当 `engine_id.engine_index()` 为 `None` 时会默认使用 `0`；如果集群中实际存在多个 engine，可能导致统计全部汇总到第 0 个 engine，掩盖真实瓶颈。  
2. **动态 Engine 集合更改**  
   - `SchedulerStatsRecorder` 在客户端创建时一次性捕获当前 engine 列表。若运行期间 **增减 engine**（如弹性扩缩），新增的 engine 将没有预先缓存的句柄，记录将被 silently dropped（已通过日志提示）。这可能导致 **监控盲区**。  
3. **句柄生命周期**  
   - `SchedulerStatsRecorder` 持有 `BTreeMap<u32, SchedulerStatsHandles>`；如果 `Metrics`（Prometheus registry）在运行时被重新初始化或替换，原句柄可能指向已释放资源，导致 panic 或错误的计数。  
4. **并发安全**  
   - 所有句柄本身是 **线程安全的原子计数/Histogram**，但 `SchedulerStatsRecorder` 与 `RequestMetricHandles` 的创建过程在多线程环境下必须在单线程初始化阶段完成，确保没有竞争。  

---

### 💡 关注建议

| 受众 | 建议 |
|------|------|
