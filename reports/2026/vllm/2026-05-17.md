### Refactor: Pass num_labels explicitly to PoolerClassify instead of reading from global config (#42851)
**SHA**: `1c8e9c0` | 🔗 [查看提交](https://github.com/vllm-project/vllm/commit/1c8e9c0399f6a6a98f406dce5947a2ad318e195a)

**🎯 变更类型**：重构 / 功能增强  
**⚡ 重要程度**：🟡 中  

**📋 变更摘要**  
- `PoolerClassify` 由原先通过 `get_current_vllm_config()` 读取全局 `num_labels`，改为在构造时显式传入 `num_labels`（`int | None`），不再依赖全局配置。  
- `get_act_fn` 与 `resolve_classifier_act_fn` 相应更新，改为从传入的 `config`（或 `static_num_labels` 参数）中读取 `num_labels` 并把它传给 `PoolerClassify`。  
- 测试用例全部改为显式提供 `num_labels`（或 `None`），并去除对 `VllmConfig`、`set_current_vllm_config` 的依赖。  

**🎯 影响范围**  
- `vllm/model_executor/layers/pooler/activations.py`（核心分类池化层实现）  
- 相关测试文件 `tests/model_executor/layers/test_pooler_activations.py`  
- 任何直接实例化 `PoolerClassify`、`get_act_fn`、`resolve_classifier_act_fn` 的业务代码。  

**💡 关注建议**  
1. **调用方检查**：所有自行创建 `PoolerClassify` 的位置需要改为 `PoolerClassify(num_labels=…)`，或在 `ModelConfig` 中确保 `num_labels` 能被正确读取。  
2. **向后兼容**：若仍有代码依赖旧的全局配置读取方式，建议保持一个兼容包装（如 `static_num_labels=True` 的旧构造）或在迁移期提供适配层，以防运行时 `num_labels` 为 `None` 导致逻辑偏差。  
3. **文档更新**：在 `vllm.config` 与池化层使用说明中明确 `num_labels` 为必填（除非明确为 `None` 表示动态推断），并删除关于 `set_current_vllm_config` 的示例。  
4. **测试覆盖**：确认所有模型类型（regression、single/multi‑label、sentence‑transformers、sbert）在新的接口下仍能通过，尤其是 `static_num_labels=False` 场景已经被 `num_labels=None` 替代。  
5. **性能/可靠性**：移除全局配置读取后，实例化过程更轻量，避免了在多线程/多进程环境下共享全局状态的潜在竞争。  

总体而言，此次重构提升了可测性和显式性，只要在项目中统一使用新的构造方式，即可获得更可靠的分类池化行为。

### [KV Connector] Support disk offloading in MooncakeStoreConnector (#42689)
**SHA**: `36e74c9` | 🔗 [查看提交](https://github.com/vllm-project/vllm/commit/36e74c9ea4feb5ade38ffa1ea96f24dd73316e02)

**🎯 变更类型**：功能增强（Mooncake KV‑Cache 连接器新增磁盘 offload 支持）  
**⚡ 重要程度**：🟡 中  
**📋 变更摘要**：在 `MooncakeStoreConnector` 中加入对磁盘 offloading 的完整实现与配置，扩展了 JSON 配置字段（`mode`、`enable_offload`、`preferred_segment` 等），提供了 RDMA 设备解析工具 `rdma_utils`，并在 KV‑Cache 发送/接收线程中加入磁盘预算管理、批次切分、层级日志等逻辑，同时完善了对应单元测试。  

**🎯 影响范围**  
- `vllm/distributed/kv_transfer/kv_connector/v1/mooncake/store/worker.py`（核心业务逻辑）  
- 新增 `rdma_utils.py`（本地 host、RDMA 设备、segment 解析）  
- `vllm/envs.py`（新增环境变量）  
- 文档 `mooncake_store_connector_usage.md`（使用说明）  
- 测试 `tests/v1/kv_connector/unit/test_mooncake_store_worker.py`（覆盖新特性）  

**💡 关注建议**  
1. **配置兼容性**：`MooncakeStoreConfig` 现在默认 `mode=embedded`，而旧版只支持 CPU offload。请确保已有部署的 `mooncake_config.json` 在升级后仍符合新校验（`global_segment_size` 必须在 `embedded` 时非 0，`standalone‑store` 时必须为 0）。  
2. **磁盘预算**：开启 `enable_offload` 时会自动分配 1.28 GiB 的 DirectIO 缓冲区，预算比例受 `VLLM_MOONCAKE_DISK_STAGING_USABLE_RATIO` 控制。大 Prefill 超出预算会触发子批次拆分或直接跳过，务必根据实际 SSD 写入能力调节 `enable_offload` 与 `preferred_segment`。  
3. **环境变量**：`MOONCAKE_PREFERRED_SEGMENT`、`MOONCAKE_REQUESTER_LOCAL_HOSTNAME`、`VLLM_MOONCAKE_STORE_TIER_LOG` 等新变量在多节点部署时非常关键，建议在启动脚本中显式声明，避免意外回退到默认行为。  
4. **RDMA 设备选择**：`rdma_utils.get_configured_worker_rnic` 在未配置 `device_name` 时会记录警告并交由 C++ 自动选择；若集群中存在多 GPU，请使用逗号分隔的设备列表或显式指定 `device_name`，否则可能出现单 NIC 争用。  
5. **性能监控**：开启 `VLLM_MOONCAKE_STORE_TIER_LOG=1` 可观察每批次的 memory/disk 命中率，帮助定位磁盘 offload 是否成为瓶颈。建议在生产环境先开启少量日志进行基准评估。  
6. **回退兼容**：如果已有用户不需要磁盘 offload，只需在配置中保持 `enable_offload: false`（或删除该字段）即可保持原有 CPU‑only 行为。  

**总体建议**：在升级前做好配置备份与兼容性检查，首次在 `standalone‑store` + `enable_offload:true` 模式下运行时，可先在单节点实验环境验证磁盘预算与吞吐，以确保在大规模分布式部署中磁盘 I/O 不会成为新的性能瓶颈。开发者应关注新增的单元测试覆盖范围，确保后续改动不破坏磁盘 offload 的拆分与日志逻辑。
