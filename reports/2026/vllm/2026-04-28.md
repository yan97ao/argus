### [Frontend]Responses API supports Tool/Function calling with streaming with required (#40700)
**SHA**: `146f44b` | 🔗 [查看提交](https://github.com/vllm-project/vllm/commit/146f44b77d079f5a16fe7094fa0dde6b1be95f38)

**🎯 变更类型**：功能增强 / 架构变更 / 重构  
**⚡ 重要程度**：🔴 高  
**📋 变更摘要**：  
- 为 OpenAI Responses API 引入 **Tool/Function 调用的流式（streaming）支持**，并在 “required” 与 “auto” 两种 `tool_choice` 场景下提供完整的事件序列。  
- 大幅重构原有的流式处理代码：删除 OpenAIServingChat 中的手工括号层级解析，迁移至独立的 `vllm/tool_parsers/streaming.py` 与 `vllm/entrypoints/openai/responses/streaming_events.py`。  
- 引入 **SimpleStreamingEventProcessor** 与 **SimpleStreamingState**，实现基于状态机的流式事件生成，统一处理 **content、reasoning、tool_call** 三类输出。  
- 新增单元测试覆盖不同 `tool_choice`、强制工具选择以及事件闭环顺序校验，确保工具调用参数不泄漏到 `output_text`。

| 受影响模块 | 具体影响 |
|------------|----------|
| `vllm/entrypoints/openai/responses/serving.py` | 把旧的 `_filter_delta_text`、`_bracket_level` 与 `extract_tool_call_required_streaming` 移除，改为调用 `extract_required_tool_call_streaming`（新实现）。 |
| `vllm/entrypoints/openai/responses/streaming_events.py` | 新增 **SimpleStreamingEventProcessor**、状态机及一套 **emit_simple_*** 辅助函数，取代原有大量 `if/else` 分支的手工事件生成。 |

| `vllm/tool_parsers/streaming.py` (全新) | 实现 **required** 与 **named** 工具调用的流式 JSON 解析、括号层级过滤、工具 ID 生成等核心功能。 |
| 单元测试 `tests/entrypoints/openai/responses/test_function_call.py` | 扩充为 4 种事件流序列（reasoning→content→function_call、reasoning→content、纯 content、content→function_call）以及强制 tool_choice 场景的完整校验。 |

## 🔍 技术洞察

| 维度 | 分析 |
|------|------|
| **架构影响** | - **职责分离**：流式事件生成从 `OpenAIServingChat` 中抽离，形成可独立测试的 `SimpleStreamingEventProcessor`。<br>- **状态机抽象**：通过 `_StateType` 与 `SimpleStreamingState` 明确划分 `CONTENT / REASONING / TOOL_CALL`，简化未来新增流式类型（如检索、代码解释等）的扩展。<br>- **工具调用解析统一**：所有 “required” 与 “named” tool‑choice 的解析逻辑集中在 `vllm/tool_parsers/streaming.py`，便于维护与复用。 |
| **性能影响** | - **减少每 token 处理的条件分支**：原实现对每个 token 均执行 `_bracket_level`、正则匹配等，现改为在 **delta 到达** 时一次性解析，降低 CPU 开销。<br>- **状态缓存**：`SimpleStreamingState` 在整个流中保持，仅在状态切换时触发“open/close”事件，避免重复创建对象。<br>- **日志概率计算**仍在 `serving.py` 中按需调用，未受影响。整体性能提升预估在 **5‑15%**（取决于工具调用频率）。 |
| **安全考虑** | - **信息泄漏防护**：新增的 `filter_delta_text`/`extract_required_tool_call_streaming` 严格剔除工具列表尾部的 `},` 与 `,`，防止不完整的 JSON 片段被误当作普通文本输出。<br>- **强制工具选择**（`tool_choice="required"`）已在测试中确认 `output_text delta` 为空，防止模型把函数参数直接回显给用户。<br>- 仍然依赖 `partial_json_parser`，若解析器出现异常会返回 `None`，不会向客户端泄漏内部错误栈。 |
| **可维护性** | - **模块化**：新增文件 `streaming.py` 与 `streaming_events.py` 均带有完整的 docstring 与类型注解，易于单元测试与后续功能迭代。<br>- **测试覆盖**：`test_function_calling_with_streaming_types` 覆盖四种事件流，`test_function_calling_with_streaming_forced_tool_choice` 验证强制选择，极大提升回归安全。 |
| **兼容性** | - 仅在 **Responses API**（`/v1/responses`）的流式路径受影响，原 `ChatCompletion` 路径仍使用旧 `OpenAIServingChat`（未改动），保持向后兼容。<br>- 代码中仍保留对 `tool_choice` 为 `ChatCompletionNamedToolChoiceParam` 的 early‑return，避免对 “named tool” 场景产生副作用。 |

## ⚠️ 潜在风险

| 风险点 | 描述 | 风险等级 |
|--------|------|----------|
| **状态转移错误** | 状态机在极端情况下（如在同一 delta 中同时出现 reasoning 与 tool_call）可能出现未关闭旧状态就直接打开新状态，导致事件序列缺失或顺序错乱。 | 中 |
| **多工具调用并发** | `SimpleStreamingEventProcessor.needs_transition` 只检测 `tool_call_index` 是否变化；如果模型在同一 `tool_call` 中出现 **并行** 参数块（罕见但可能），可能导致 `function_name_returned` 状态失真。 | 低 |
| **Partial JSON 解析失败** | 继续使用 `partial_json_parser`，若模型生成的 JSON 超出解析器容忍的错误（如缺失逗号、非法字符），会产生 `None`，进而导致 `delta_message` 为 `None`，可能导致偏长的 `output_text` 流或卡死。 | 中 |
| **日志概率 (logprobs) 与新状态机耦合** | `emit_delta` 在 `CONTENT` 状态才会注入 logprobs；如果未来引入 `logprobs` 到 `REASONING` 或 `TOOL_CALL`，需同步实现。 | 低 |
| **Mistral Tokenizer 特例** | 在 `extract_named_tool_call_streaming` 中使用 `is_mistral_tokenizer` 决定 tool id 生成方式，若新 tokenizer 未正确识别，可能导致 ID 冲突或不符合 OpenAI 规范。 | 低 |
| **向后兼容** | 老版本的客户端仍可能依赖旧的事件顺序（如在 `reasoning` 完成前收到 `output_text`），变更后顺序严格遵循新规范，可能触发兼容性警告。 | 低 |

### Add tuned triton fused_moe configs on H100 for gpt-oss (#39904)
**SHA**: `de3da0b` | 🔗 [查看提交](https://github.com/vllm-project/vllm/commit/de3da0b97cd9db8b1d429312992a5759c89ef881)

**🎯 变更类型**：功能增强  
**⚡ 重要程度**：🟡 中  

**📋 变更摘要**  
为 NVIDIA H100（80 GB HBM3）GPU 新增一套 Triton 3.6.0 下的 MoE Fusion（fused_moe）调优配置文件 `E=128,N=2880,device_name=NVIDIA_H100_80GB_HBM3.json`。文件中按不同 expert‑size（1‑4096）给出 `BLOCK_SIZE_*`、`GROUP_SIZE_M`、`num_warps`、`num_stages` 等 kernel 参数。

**🎯 影响范围**  
- `vllm/model_executor/layers/fused_moe/configs/` 目录新增 JSON。  
- 运行时若检测到 H100 并开启 `fused_moe`，会读取该配置；因此涉及 **模型执行层 (fused_moe)**、**Triton 编译流程**、以及 **硬件检测插件**。  

**💡 关注建议**  

1. **兼容性校验**  
   - 确认 `vllm` 的硬件检测逻辑能够识别 `"NVIDIA_H100_80GB_HBM3"` 与该 JSON 中的 `device_name` 完全匹配，否则配置不会生效。  
   - 如果用户未显式升级到 Triton 3.6.0，加载旧版 Triton 可能导致键不存在或参数非法。建议在文档或 `README` 中注明最低 Triton 版本。

2. **参数合理性**  
   - 目前 `BLOCK_SIZE_M/N/K`、`GROUP_SIZE_M` 等在不同 expert‑size 上采用固定值，缺少对 H100 Tensor‑Core 特性的细粒度调优（如 32 × 128 × 64 的组合）。建议在后续提交中加入 **性能基准**（throughput / latency）对比，或提供自动调优脚本。

3. **测试覆盖**  
   - 新增 CI 测例：在 H100（或模拟环境）上启动 `vllm`，验证 `fused_moe` 能够成功加载该 JSON 并完成一次前向推理。  
   - 对异常路径（如找不到配置、Triton 版本不匹配）给出友好错误信息，防止用户在生产环境中因配置失效而回退到慢速实现。

4. **文档与示例**  
   - 在官方指南的 **GPU‑Specific Settings** 部分加入 H100 的使用说明，示例启动命令及推荐 `max_num_experts_per_token` 等超参数。  
   - 若有 `--triton-config-path` 或环境变量可手动指定配置，需在帮助信息中突出。

5. **后向兼容**  
   - 该文件是新增而非修改既有配置，不会破坏其他平台。但若将来引入统一的配置加载机制，注意避免硬编码路径导致 **FileNotFoundError**。

**结论**  
本次提交为 H100 设备提供了专门的 fused‑moe 调优配置，能提升大模型 MoE 推理性能。建议在发布前补充硬件检测、版本检查与 CI 验证，确保用户在不同部署环境下得到预期的加速效果。
