### [Frontend] Add Streaming Parser Engine and new Qwen3 Parser (#45413)
**SHA**: `c4a3f9d` | 🔗 [查看提交](https://github.com/vllm-project/vllm/commit/c4a3f9d137099023a93fbb2299f3c52477751176)

**🎯 变更类型**：架构变更、功能增强、性能优化、代码重构、Bug 修复、安全修复  
**⚡ 重要程度**：🔴高  

**📋 变更摘要**  
本次提交为 **vLLM** 引入全新的 **Streaming Parser Engine** 框架，实现了基于声明式 `ParserEngineConfig` 的统一解析流水线。核心改动包括：

1. **新增解析引擎模块** (`vllm/parser/engine/*`)：  
   - `ParserEngine`（抽象层），`StreamingParserEngine`（流式核心），`TokenIDScanner`，`IncrementalLexer`，`ParserEngineConfig` 等。  
   - 支持 **终端（terminals）**、**状态机（transitions）**、**内容事件（content_events）**、**参数转换（arg_converter）**、**JSON 流式增量** 等扩展能力。  

2. **适配器层**：`adapters.py` 为旧的 `ReasoningParser` 与 `ToolParser` 提供兼容层，使现有服务代码无需改动即可使用新引擎。  

3. **Qwen3 解析器迁移**：原有 `qwen3_reasoning_parser` 与 `qwen3coder_tool_parser` 被 **废弃**，统一改为 **Qwen3Parser**（基于新引擎）并通过适配器导出 `Qwen3ParserReasoningAdapter`、`Qwen3ParserToolAdapter`。  

4. **测试套件扩展**：新增大量单元、集成、回放与压测测试，覆盖流式、非流式、持回、工具调用、推理、类型强制、空白处理、异常情况等，确保新引擎的可靠性。  

5. **其它改动**：删除冗余属性、清理旧代码、统一日志、更新注册表、工具解析器标记 `engine_based_streaming`、实现对 `strip_trailing_reasoning_whitespace`、`drop_whitespace_only_content_before_tools` 等细粒度配置的支持。

## 🔍 技术洞察

### 1. 架构影响
| 维度 | 正面影响 | 负面影响 |
|------|----------|----------|
| **模块化 & 可扩展性** | 采用 **声明式配置**（`ParserEngineConfig`）+ **状态机**，新模型只需编写配置即可替代手写 parser，显著降低维护成本。适配器层保持了对旧接口的兼容，外部调用无需感知内部改动。 | 引入多层抽象（Adapter → Engine → Lexer/Scanner）导致调用路径变深，调试时需追踪更多层级。 |
| **统一 Streaming 入口** | 所有流式解析统一走 `ParserEngine.parse_delta`，一次性处理 **文本 + token_id**，极大简化 `engine/`、`tool_parsers`、`reasoning` 的实现。 | 需要确保所有已有 Parser（如 KimiK2、Llama3）通过 `make_adapters` 正确注册，否则可能在运行时找不到对应适配器。 |
| **配置驱动** | 通过 `ParserEngineConfig` 定义 **terminals、transitions、content_events**，可灵活支持任意自定义格式（Qwen3、Gemma4、Mistral 等），未来新增模型几乎不需要代码改动。 | 对新手而言，需要理解状态机与终端的关系，错误配置可能导致 **不可达状态** 或 **死循环**。 |
| **引擎与工具/推理解耦** | `ParserEngine` 同时负责推理与工具调用，使用 `engine_based_streaming` 标记决定是否走 **引擎实时解析**（如 Qwen3）或传统基于 **token_id** 的流式处理。 | 中间层的 **_stream_state** 增加了复杂度，若实现不当可能导致 **状态不同步**（如工具调用结束后仍残留 `reasoning_ended` 标记）。 |

### 2. 性能影响
| 维度 | 优化点 | 可能的性能瓶颈 |
|------|--------|----------------|
| **Token‑ID 与文本双通道** | 在 `StreamingParserEngine.feed` 中加入 **快路径**：若无特殊终端且文本中不出现首字符，直接返回 `CONTENT` 事件，避免 lexer & scanner 开销。 | 当模型频繁产生 **特殊 token**（如大量 `<tool_call>`），仍需完整扫描，仍会受 **正则、循环** 所限。 |
| **持回文本恢复** | `TokenIDScanner` 通过 **holdback 重建**（`_recover_holdback_text`）缓存并在下次出现时恢复，避免因 Detokenizer 延迟导致误解析。 | 对极端 **大 holdback**（数百字符）会导致 `rebuild_from_anchors` 线性搜索，最坏 O(N²)（若出现大量重复 literal），但实际情况极少。 |
| **增量 JSON 参数流** | `StreamingParserEngine` 在 `TOOL_ARGS` 状态下实现 **JSON 增量缓冲**（持有未闭合的 `[`、`{`），在 `arg_converter` 中利用 **safe 前缀** 验证 `startswith`，仅在必要时输出增量。 | `arg_converter` 若实现为昂贵的正则或递归解析，可能在每个字符增量时触发，导致 **CPU 超负载**。但默认实现 `_qwen3_arg_converter` 为轻量级正则+字符串拼接。 |
| **缓存** | `TokenIDScanner` 对 `token_id -> text` 结果做缓存，避免每次解码重复调用 tokenizer。| 缓存仅对 **已出现的 token_id** 有效，若模型使用 **大量独立 token**（如大量随机 sub‑word）仍会频繁解码。 |
| **工具调用聚合** | `ParserEngine._coalesce_tool_call_deltas` 合并同一工具索引的多条 delta，降低 `DeltaMessage` 数量，减少网络传输与序列化开销。 | 合并过程为 O(N) 的字典操作，对大批量工具调用仍保持线性，基本无额外开销。 |

### [Bugfix] Two-phase KV allocation for cross-group prefix cache hits (supersedes #33775) (#44409)
**SHA**: `588db18` | 🔗 [查看提交](https://github.com/vllm-project/vllm/commit/588db1836245bb40589d45b3e87048a53a13cfe5)

**🎯 变更类型**：Bug 修复（两阶段 KV 分配）  
**⚡ 重要程度**：🟡 中  

**📋 变更摘要**  
本次提交修复了 #33775 中暴露的跨 KV‑cache 组前缀缓存命中时出现的“双分配”错误。核心改动将原 `allocate_new_computed_blocks` 拆分为  
1. `add_local_computed_blocks` – 先触摸各组本地（缓存命中）块，确保其 ref_cnt 正确；  
2. `allocate_external_computed_blocks` – 在所有本地块已安全 “锁定” 后，统一为每组分配外部（connector）块，防止前面的组在获取新块时意外驱逐后面的缓存块。  

相应地在 `KVCacheCoordinator` 中新增两阶段调度逻辑，并在 `CrossAttentionManager` 中实现空实现以保持接口统一。大量新增单元测试覆盖两组、三组混合模型、预占（preempt）‑重新分配等场景，验证块 ID 唯一性与 ref_cnt 正常。

**🎯 影响范围**  
- `vllm/v1/core/kv_cache_coordinator.py`（调度逻辑）  
- `vllm/v1/core/single_type_kv_cache_manager.py`（核心块分配实现）  
- `vllm/v1/core/cross_attention_manager.py`（接口兼容）  
- 测试模块 `tests/v1/core/test_prefix_caching.py`、`tests/v1/core/test_single_type_kv_cache_manager.py`

**💡 关注建议**  
1. **接口兼容**：`allocate_new_computed_blocks` 已被废弃且在项目内部已改为 `add_local_computed_blocks`，请检查其他模块（如自定义插件或外部调用）是否仍使用旧名，必要时加上兼容包装或 deprecation 警告。  
2. **并发安全**：两阶段分配涉及两轮对 `block_pool` 的 `get_new_blocks` 调用，确保在多线程/多进程环境下仍保持原子性或在上层加锁，以防出现竞争导致 ref_cnt 错误。  
3. **性能评估**：新增一次遍历 `single_type_managers`（先本地块后外部块），对极大模型的影响微乎其微，但建议在高负载基准下跑一次完整的吞吐量/延迟测试，确认不会引入显著回退。  
4. **文档与示例**：更新 `kv_cache_manager` API 文档，说明两阶段分配的调用顺序及何时会触发外部块分配；在 README 或使用示例中加入 “前缀缓存命中 + 外部块” 场景的演示。  
5. **回归测试**：当前已加入覆盖多组、预占‑重新分配的单元测试，建议在 CI 中保持 `--maxfail=1`，并将这些测试标记为 “critical”，防止未来改动再次破坏两阶段逻辑。  

总体来看，此次改动精准定位并彻底解决了跨组缓存命中导致的块双分配与 ref_cnt 错误，风险主要在接口迁移和并发上下文的保持一致性。完成上述检查后即可安全合并。

### Remove redundant Triton KV cache dtype asserts and enforce architectural support (fp8 >= sm89) (#43914)

**⚡ 重要程度**：🟡 中

**🎯 影响范围**  
- Triton 注意力后端 (`vllm/v1/attention/backends/triton_attn.py`)  
- Triton reshape‑and‑cache 实现 (`vllm/v1/attention/ops/triton_reshape_and_cache_flash.py` 与 `…_diffkv.py`)  
- 与 KV‑cache dtype 相关的配置入口 (`--kv-cache-dtype`)

   - 通过 Pydantic `model_validator` 实现 `input → messages` 的自动归一化。  
2. `base/protocol.py` 将原来的 `ChatRequestMixin` 拆分为 **选项** (`ChatRequestOptionsMixin`) 与 **消息体** (`ChatRequestMixin`)，保留 `add_generation_prompt` 等额外字段。  
3. `embed/io_processor.py` 增加 `_pre_process_openai_chat_online` 与 `_batch_render_openai_chat`，在 `pre_process_online` 流程中识别上述四类请求并调度渲染器处理。  
4. 单元测试 `tests/entrypoints/pooling/embed/test_io_processor.py` 扩展，验证：  
   - `input` 为单条/多条聊天消息时能正确解析为对应请求类型；  
   - `messages` 直接传入时仍保持兼容；  
   - `chat_template_kwargs`、`add_generation_prompt`、多模态参数等在渲染调用中被正确转发。  
5. `typing.py` 与 `pooling/typing.py` 更新类型别名，使新请求类型可在服务运行时被识别。  

**🎯 影响范围**  
- **协议层**：`vllm/entrypoints/pooling/embed/protocol.py`、`base/protocol.py`、`typing.py`。  
- **请求处理**：`vllm/entrypoints/pooling/embed/io_processor.py`（新增渲染路径）。  
- **单元测试**：`tests/entrypoints/pooling/embed/test_io_processor.py`。  
- **公共 API**：`/v1/embeddings` 现在接受 `messages`（单/批）以及 `chat_template_kwargs`，保持向后兼容。  

**💡 关注建议**  
1. **向后兼容性**：已保留原 `EmbeddingCompletionRequest`（token‑ids）路径，确保老客户端不受影响。部署前请确认 `OpenAI` 客户端仍使用 `input` 字符串或 token 列表时不会误触新 Chat 解析。  
2. **性能与资源**：批量渲染在 `renderer.render_chat` 中一次性生成所有 Prompt Token IDs，若 Batch 较大可能导致一次性内存峰值增长，建议监控 `max_batch_size` 与 `max_model_len` 的配合。  
3. **多模态扩展**：`prompt_extras` 仍会把 `mm_processor_kwargs`、`cache_salt` 透传给渲染器，确保对应的 `MultimodalConfig` 已在模型配置中声明，否则会出现 `None` 错误。  
4. **测试覆盖**：目前仅覆盖了请求解析与渲染调用验证，建议补充异常路径（如 `messages` 与 `input` 同时出现、`chat_template` 不合法、`chat_template_kwargs` 与模型不匹配）以及大批量（>64）请求的压力测试。  
5. **文档更新**：API 文档须注明 `messages` 与 `chat_template_kwargs` 参数的使用方式、默认行为（`add_generation_prompt=False`），并提供示例。  

总体而言，此次改动为嵌入 API 引入了真实的聊天式输入能力，提升了与 OpenAI 兼容性，同时保持了原有功能。只要在部署前验证多模态配置与批量大小即可平稳上线。
