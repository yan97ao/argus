### [Bugfix] Fix V1 logprobs empty strings for multi-byte UTF-8 tokens when logprobs > 0 (#34875)
**SHA**: `8904fc4` | 🔗 [查看提交](https://github.com/vllm-project/vllm/commit/8904fc4d1942ee0771c094b2b084cd62c55de89d)

**🎯 变更类型**：功能增强（修复 V1 logprobs 在多字节 UTF‑8 token 中出现空字符串的问题）  
**⚡ 重要程度**：🟡 中  

**📋 变更摘要**  
1. 为 `LogprobsProcessor` 引入 `FlatLogprobs`，并实现 **`_get_sampled_context_ids`** 用于从已生成的 logprob（或 prompt logprob）中提取最近 ≤ 4 个采样 token，作为 UTF‑8 修复的上下文。  
2. 重构 `_correct_decoded_token`：签名改为 `(token_id, context_token_ids)`，采用逐步扩展上下文、区分“干净”前缀与字节回退 token，返回修正后的子串或空串。  
3. 更新 `_verify_tokens`、`_update_sample_logprobs`、`_update_prompt_logprobs` 以传递上述上下文。  
4. 大量新增单元测试，覆盖：单 token、跨 token、前后 logprob、无上下文、连续回退、真实 OPT‑125m 场景等。  

**🎯 影响范围**  
- `vllm/v1/engine/logprobs.py`（核心 logprob 处理逻辑）  
- `tests/v1/sample/test_logprobs.py`（新增/重写测试）  

**💡 关注建议**  
- **兼容性**：`_correct_decoded_token` 为内部私有方法，外部未直接调用，当前改动不应破坏已有 API；若将来计划公开，需保持旧签名的兼容层。  
- **性能**：上下文提取在每个需要修复的 token 上执行，最多 4 次 decode。建议在性能关键路径（大 batch）跑基准，确认不会显著拖慢生成速度。  
- **FlatLogprobs 支持**：新增对 `FlatLogprobs` 的 fast‑path 实现，请确保 `vllm.logprobs` 包在所有发行版中同步更新，否则会出现 `ImportError`。  
- **异常安全**：`_get_sampled_context_ids` 目前假设 `logprobs_source` 为 `list[dict]` 或 `FlatLogprobs`；若出现 `None` 或其它实现，应提前返回 `[]`（已做），但建议在未来加上 `isinstance` 检查，以防意外类型。  
- **代码可读性**：函数已添完整 docstring，建议在模块顶部统一导入 `typing`（`list[int] | None`）并开启 `from __future__ import annotations`，提升类型检查体验。  

总体来看，此次改动解决了多字节 UTF‑8 令牌在 `logprobs>0` 场景下产生空字符串的根本原因，测试覆盖全面，风险有限。上线前建议运行完整的性能回归以及对 `FlatLogprobs` 的跨版本兼容性检查。

### [FlashAttention] Symlink FA4 instead of copying when using `VLLM_FLASH_ATTN_SRC_DIR` (#38814)
**SHA**: `308cec5` | 🔗 [查看提交](https://github.com/vllm-project/vllm/commit/308cec5864890f5c0724e1d4531d9fe2ee0a8209)

**🎯 变更类型**：功能增强（FlashAttention 4 集成方式优化）  
**⚡ 重要程度**：🟡 中  
**📋 变更摘要**：在 `VLLM_FLASH_ATTN_SRC_DIR` 指向本地 FlashAttention 源码时，改为在安装目录下创建 `cute/` 的符号链接，而不是复制并改写导入路径；当未指定该变量时仍保留原有复制改写逻辑。为使符号链接下的 `flash_attn.cute.*` 导入生效，在 `vllm/vllm_flash_attn/__init__.py` 中动态注册一个虚拟的 `flash_attn` 包。

**🎯 影响范围**  
- `cmake/external_projects/vllm_flash_attn.cmake`（构建/安装阶段）  
- `vllm/vllm_flash_attn/__init__.py`（运行时模块导入）  
- 依赖 FlashAttention 4 的所有模型推理路径  

**💡 关注建议**  

1. **兼容性**：`file(CREATE_LINK … SYMBOLIC)` 在 Windows 上需要管理员权限或开发者模式，可能导致安装失败。建议在 CMake 中检查 `WIN32` 并回退为 `copy`，或在文档中明确要求。
2. **可重复构建**：使用 symlink 时，源码文件的即时修改会影响已安装的 vLLM 包，这对 CI/CD、二进制分发不利。请在 `setup.py/pyproject.toml` 中说明此模式仅用于开发环境。
3. **模块冲突**：`__init__.py` 直接向 `sys.modules` 注入 `flash_attn` 包，若用户自行导入同名包或已有同名模块，可能产生冲突。建议在注入前检查 `sys.modules.get("flash_attn")` 的 `__file__` 是否指向预期路径，或使用 `importlib.util.module_from_spec` 更安全地加载。
4. **清理与卸载**：`install(CODE … file(REMOVE_RECURSE …))` 在重装时会删除已有目录并创建链接。确认在 `pip uninstall vllm` 时不会残留链接或误删用户目录。
5. **测试覆盖**：新增两个路径分支（symlink 与 copy）都应在 CI 中跑通：  
   - `VLLM_FLASH_ATTN_SRC_DIR` 设置且指向可写目录 → 检查 `vllm_flash_attn.cute` 为 symlink，`import flash_attn.cute.xxx` 正常。  
   - 未设置变量 → 检查文件已被复制且内部 `flash_attn.cute` 已被替换为 `vllm.vllm_flash_attn.cute`。  
6. **文档更新**：在 README/开发指南中说明 `VLLM_FLASH_ATTN_SRC_DIR` 的作用、使用限制以及在 Windows 上的特殊处理。

总体来说，改动提升了本地调试 FlashAttention 4 的便利性，但需注意平台差异、包冲突以及发布环境的可重复性。适当的防御性检查和文档补全即可降低风险。

### [Parser] Migrate response api streaming to unified parser (#38755)
**SHA**: `927975e` | 🔗 [查看提交](https://github.com/vllm-project/vllm/commit/927975ead8ea8f2c818844680fd7b29d0f62d0e9)

**🎯 变更类型**：功能增强（统一流式解析器）  
**⚡ 重要程度**：🟡 中  

**📋 变更摘要**  
1. 引入 `StreamState` dataclass 与 `Parser.parse_delta`，把原先在 `serving._process_simple_streaming_events` 中散落的 reasoning‑/tool‑parser 调用逻辑统一到 `Parser` 实例内。  
2. `serving.py` 只保留 `self.parser(tokenizer, request.tools)` 的一次实例化，后续全部交由 `parser.parse_delta` 处理。  
3. 测试文件通过 `_mock_parser_with_reasoning` 简化了对统一解析器的 mock，删除了对 `reasoning_parser_cls` / `tool_parser_cls` 的直接调用。  

**🎯 影响范围**  
- **核心代码**：`vllm/parser/abstract_parser.py`（新增 `StreamState`、`parse_delta` 实现、部分辅助方法）  
- **入口实现**：`vllm/entrypoints/openai/responses/serving.py`（流式处理改为调用统一 parser）  
- **测试**：`tests/entrypoints/openai/responses/test_serving_responses.py`（mock 调整）  

**💡 关注建议**  

1. **实例生命周期**  
   - 现在 `self.parser(tokenizer, request.tools)` 在每次请求时创建一次 `Parser`，内部状态（`_stream_state`）随流生存。请确认在所有代码路径（包括非‑OpenAI 入口）都仍然使用同一方式实例化 parser，避免出现同一实例被复用导致跨请求状态泄漏。  

2. **向后兼容**  
   - 旧的 `self.parser.reasoning_parser_cls / tool_parser_cls` 接口已被移除。若有外部插件或自定义 parser 仍依赖这些属性，需相应迁移或在 `Parser` 子类中实现 `parse_delta`。建议在 `Parser.__init__` 中保留兼容属性（或在文档中注明已废弃）。  

3. **错误路径**  
   - `parse_delta` 里若 `self._reasoning_parser`/`_tool_parser` 为 None，直接返回 `DeltaMessage(content=delta_text)`。请确保在异常或提前结束（如 `is_reasoning_end` 为 True）时不会产生多余的空 delta，防止客户端收到冗余空事件。  

4. **性能与内存**  
   - `StreamState` 持有 `previous_text` 与 `previous_token_ids`，在长文本流式生成时会逐渐增长。若流非常长（数千 token），可能导致显著内存占用。可以考虑在判断 `reasoning_ended` 并切换到 tool‑parser 后，清空已不再需要的缓存（已在实现中通过 `state.previous_text = ""` 等），但仍要留意极端情形。  

5. **测试覆盖**  
   - 已添加统一的 mock 辅助函数，简化了 “reasoning‑only” 与 “mixed delta” 场景。建议再补充以下情形：  
   a) 仅 tool‑parser（没有 reasoning）  
   b) 直接在 prompt 中已经是 content（`prompt_is_reasoning_end=True`）  
   c) `parse_delta` 抛出异常时的异常链路。  

6. **文档更新**  
   - 统一 parser 的使用方式、`Parser.parse_delta` 参数意义以及 `StreamState` 生命周期应在开发者文档或代码注释中说明，避免新加入的贡献者误用旧的接口。  

总体来看，此次改动将流式解析的状态管理集中化，代码结构更清晰，后续扩展（如加入新的解析阶段）将更容易实现。但需注意实例化方式、跨请求状态隔离以及向后兼容性，适当补齐文档与额外测试即可。
