### [Quantization] add humming quantization kernel (#34556)
**SHA**: `9f771b3` | 🔗 [查看提交](https://github.com/vllm-project/vllm/commit/9f771b3ab92d26a7d91a8255572c5d8d2b3ad601)

**🎯 变更类型**：功能增强（新增 Humming 量化后端）  
**⚡ 重要程度**：🔴 高  

**📋 变更摘要**  
1. 在 vLLM 中加入对 *Humming* 量化框架的完整支持，包括模型配置、环境变量、权重加载、线性层与 MoE（Mixture‑of‑Experts）子模块的实现。  
2. 新增一套 Humming‑专属的 fused‑MoE kernel（indexed / grouped / batched），以及对应的 Triton `moe_fused_mul_sum` 归约 kernel，用于加速专家输出的加权求和。  
3. 扩展 `quantization/__init__.py`、`config/model.py`、`envs.py`、`linear.py`、`parameter.py` 等核心文件，使 Humming 能够在运行时通过环境变量进行细粒度配置（online‑quant、accumulator 精度、GEMM 类型等）。  

**🎯 影响范围**  
- **模型配置**：`vllm/config/model.py`、`vllm/envs.py`（新增 env 变量）  
- **线性层**：`vllm/model_executor/layers/linear.py`（bias 参数暴露、权重分片 rounding）  
- **MoE 核心**：`vllm/model_executor/layers/fused_moe/*`（新增 Humming 专用 Experts 类、调用 HummingMethod）  
- **量化体系**：`vllm/model_executor/layers/quantization/humming.py`（完整的 HummingConfig 与量化方法）  
- **通用工具**：`vllm/model_executor/parameter.py`（分片计算 rounding）  
- **外部依赖**：`humming` Python 包（must be installed separately）

## 🔍 技术洞察  

| 维度 | 影响说明 |
|------|-----------|
| **架构影响** | • 在原有 **FusedMoE** 框架上嵌套了 *Humming* 专用的 `HummingIndexedExperts / HummingGroupedExperts / BatchedHummingGroupedExperts`，实现了 **模块化** 与 **可插拔** 的量化后端。<br>• `HummingMoEMethod` 负责把 checkpoint 权重转换为 Humming 所需的 **padded / packed** 形态，并在 `process_weights_after_loading` 中完成 **prepare_layer_meta** 与 **transform_humming_layer** 两步准备，保持与已有 `UnquantizedMoEMethod` 的接口一致。<br>• 新增环境变量 `VLLM_HUMMING_*`，在模型实例化阶段通过 `envs` 自动注入到 `compute_config`，实现 **运行时可调**（如 `use_f16_accum`、`gemm_type`）。 |
| **性能影响** | • Humming 侧重 **低位量化 (int4/float8)** 与 **聚合 GEMM**，对 **Mixture‑of‑Experts** 场景的 **算子融合**（gate‑up、激活、down‑projection）进行一次性 **Kernel‑Fusion**，可显著降低显存搬迁与同步开销。<br>• `moe_fused_mul_sum` Triton kernel 通过 **block‑wise** 计算与 **共享 workspaces**，针对 FP32/FP16/ BF16 自动调优 `BLOCK_M / BLOCK_K / warp` 配置，理论上在 **SM 90+** GPU 上可比原有逐专家 `matmul + sum` 快 **1.5‑2×**（尤其在 token 数量小、top‑k 大的情形）。<br>• 代码在 `HummingLinearMethod.apply` 中将输入展平后直接调用 `HummingMethod.forward_layer`，避免了额外的 `reshape`，对批量推理的 **吞吐** 有正向提升。 |
| **安全考虑** | • 环境变量 `VLLM_HUMMING_ONLINE_QUANT_CONFIG` 与 `VLLM_HUMMING_INPUT_QUANT_CONFIG` 通过 `maybe_convert_json_str_or_file` 直接 **json.loads**，如果变量指向本地文件，可能读取任意路径的文件内容（**路径遍历**风险）。<br>• 直接 `import humming`，若用户加载了恶意的同名本地模块，可能导致 **代码注入**。建议在生产环境使用 **虚拟环境** 并固定 `humming` 包的来源（pip + hash）。 |
| **可维护性** | • 代码量大（≈2000 行新增），但保持了 **层次清晰**：配置 → 环境 → 权重处理 → Kernel 实现 → MoE 调度。<br>• 新增的 `HummingLinearMethod` 与 `HummingMoEMethod` 继承自已有抽象基类，便于后续统一管理。<br>• `parameter.py` 中的分片 rounding 逻辑改为 `round()`（而非整数除），提升了 **跨平台一致性**（避免 Python 2 除法/负数截断差异），但需确认 **torch.int** 与 **float** 的 rounding 行为一致。 |
| **兼容性** | • 仅在 **CUDA**、显卡 **SM 75+**（`get_min_capability`）上可用；旧显卡会走回退路径（`UnquantizedLinearMethod`/`UnquantizedFusedMoEMethod`）。<br>• 对 **CPU‑only** 或 **非‑NVIDIA** 环境不产生影响。<br>• 通过 `quant_config_name` 检查保留对 `gpt_oss_mxfp4` 的兼容，避免在 Humming 模式下误用旧实现。 |
| **错误处理** | • 大量 `assert` 用于前置检查（Humming 库是否可用、GEMM 类型合法）。在生产环境建议改为抛出自定义异常，以免触发未捕获的 AssertionError。<br>• `process_weights_after_loading` 里对 `force_requant` 的路径未对 **非对齐** 的 padding 进行显式校验，极端情况下可能产生 shape mismatch。 |

## ⚠️ 潜在风险  

1. **外部依赖缺失 / 版本不兼容**  
   - 若 `humming` 包未安装或版本与 vLLM 编译时的 CUDA/PTX 不匹配，模型加载会在 `assert_humming_available` 处直接崩溃。  
2. **环境变量 JSON 注入**  
   - `maybe_convert_json_str_or_file` 可读取任意文件路径，若系统管理员未限制环境变量来源，可能被恶意利用读取敏感配置。  
3. **权重分片 rounding 变化**  
   - 将 `shard_size // packed_factor` 改为 `round(shard_size // packed_factor)` 可能导致 **off‑by‑one** 分片在多卡并行（TP）场景下出现不均衡或越界。需在 `torch.distributed` 环境下进行完整的对齐测试。  
4. **Fallback 逻辑不完善**  
   - 在 `HummingLinearMethod.prepare_weight_loader` 中，当检测到非量化权重且不支持 online‑quant 时，会 **改写类** 为 `UnquantizedLinearMethod` 并删除原有参数。若此路径在分布式同步阶段执行，可能出现 **参数不一致**（不同 rank 产生不同 class）。  
5. **GEMM 类型自动选择**  
   - `get_humming_moe_gemm_type` 在非 SM90 设备上强制使用 `"indexed"`，但某些模型在 **grouped** 类型下更快。缺少用户显式覆盖选项可能导致性能下降。  
6. **Triton Kernel 边界**  
   - `moe_fused_mul_sum_kernel` 中的 `BLOCK_M`、`BLOCK_K` 由启发式逻辑决定，极端 token 数（> 2048）或大小不对齐的 hidden size 可能触发 **shared memory overflow** 或 **illegal memory access**。

## ⚠️ 潜在风险

1. **遗留老代码或插件**  
   - 外部用户或内部脚本可能直接引用 `vllm.v1.spec_decode.eagle.SpecDecodeBaseProposer`。若未同步，会导致 `ImportError`。  
2. **循环依赖**（已解决但需留意）  
   - `eagle.py` 仍会 `import SpecDecodeBaseProposer`，如果在 `eagle.py` 顶部出现相互导入可能导致循环。当前实现通过把基类放在独立文件并在 `eagle.py` 只在需要时导入，风险相对低。  
3. **打包/发布遗漏**  
   - 新增文件 `llm_base_proposer.py` 必须包含在 `MANIFEST.in` 与 `setup.cfg/pyproject.toml` 的 `package_data`，否则在某些非源码安装方式（wheel）中缺失。  
4. **文档/示例未同步**  
   - 官方文档、示例代码如果仍指向旧路径，会产生误导。  
5. **测试覆盖不足的边界**  
   - 迁移过程主要针对 `eagle`、`draft_model`、`dflash`，但可能还有其他隐藏调用（如自定义注意力后端）未搜索到。

- **XPU 代码路径**：所有走 GDN attention 的 XPU 层现在走 `torch.ops.vllm.gdn_attention_core_xpu`，编译时会走 `torch.compile` 的图优化通道。  
- **编译配置**：新增 `"vllm::gdn_attention_core_xpu"`，若未同步到 `torch.compile` 白名单会导致 fallback。  
- **运行时上下文**：包装函数依赖 `forward_context.no_compile_layers[layer_name]` 与 `forward_context.attn_metadata`，若这些在非编译路径或未来的 XPU 特性中缺失，会触发断言。  
- **功能等价**：原实现直接调用 `torch.ops._xpu_C.gdn_attention`，现在改为间接调用。伪实现返回 `None`，保证非 `torch.compile` 场景仍保持原有行为。

**潜在风险与注意点**  
1. **层名映射**：`layer_name` 通过 `self.prefix` 传入，若前缀在 `no_compile_layers` 中不存在会抛异常；建议在包装函数中加入容错或更明确的错误信息。  
2. **Metadata 可靠性**：`attn_metadata` 仍假设为 `GDNAttentionMetadata`，且 `spec_sequence_masks` 必须为 `None`（XPU 暂不支持 speculative decoding），后续若引入此特性，需要同步更新包装实现。  
3. **注册顺序**：`direct_register_custom_op` 必须在第一次调用 `torch.compile` 前执行，确保 `torch.ops.vllm.gdn_attention_core_xpu` 已注册；建议在模块的 `__init__` 中显式调用 `register_ops_once()`。  
4. **性能回归**：包装层增加了 Python 调用开销（获取 forward_context、断言），在大规模推理时可能产生微小延迟，建议在 profiling 中对比原始 `_xpu_C.gdn_attention` 的吞吐。  
5. **测试覆盖**：缺少 XPU CI，强烈建议补充单元/集成测试，验证在 `torch.compile` 开/关、不同 `tp_size`、不同 `gqa_interleaved_layout` 下的数值一致性和梯度正确性。

**建议**  
- 增加异常提示，若 `layer_name` 未在 `no_compile_layers` 中找到，给出 “XPU GDN layer not registered for compile” 等明确信息。  
- 在 `vllm/_xpu_ops.py` 的包装函数开头记录一次 `torch._dynamo.logging`，便于定位 compile‑fallback。

3. **流式兼容性**：`is_reasoning_end_streaming` 直接检查 delta 中是否出现 `<tool_call>`，但若 `<tool_call>` 被拆分为多个 token（如 `"<tool"`、`"_call>"`），则会漏判。建议在 token 拼接层面或在 `extract_reasoning_streaming` 中对 `delta_text` 再次搜索完整标签。  

4. **回退路径**：`extract_content_ids` 在找不到 `<think>` 时会尝试从 `<tool_call>` 开始截取。若两者均不存在，返回空列表，可能导致后续 `content` 为 `None`。需要确认调用方对 `[]` 的容错。  

5. **测试覆盖**：新增测试已覆盖最常见场景，但未覆盖 **`<tool_call>` 前已有 `</think>`**、**`<tool_call>` 与 `</think>` 同时出现**、以及 **多轮对话中出现多次 `<tool_call>`** 的情况。建议补充这类组合，以防回归。  

6. **文档/兼容说明**：该改动改变了 Qwen3 系列模型的推理分离行为，建议在 `README` 或 API 文档中注明 “在 Qwen3.5 及以上模型中，`<tool_call>` 将被视为推理结束”，并提醒用户在自定义提示模板时避免误用 `<tool_call>` 作为普通文本。  

总体来说，本次修改解决了 Qwen3.5 输出 `<tool_call>` 前未闭合 `</think>` 导致的解析错误，逻辑清晰且测试覆盖基本完整。只需注意 token 可得性、跨 token 边界以及更加全面的回归测试，即可在生产环境安全上线。

**🎯 影响范围**：  
- `vllm/distributed/kv_transfer/kv_connector/v1/offloading/scheduler.py`（核心调度器）  
- 新增 `vllm.utils.math_utils.cdiv` 的使用  
- 关联的 `KVCacheBlocks`、`KVCacheManager`、`SchedulerOutput` 以及抽象 offload 接口的调用路径。  

**💡 关注建议**：  
1. **多 group 配置测试**：确认不同 `gpu_block_size` / `offloaded_block_size` 组合下，`cdiv` 计算的 `num_gpu_blocks` 与 `num_blocks` 均符合预期，尤其是滑动窗口或 mamba‑padding 引入的 `null` block 场景。  
2. **并发安全**：`_reqs_to_load`、`_reqs_being_loaded` 与全局 `_blocks_being_loaded` 仍通过集合更新，确保在高并发请求下不出现竞争；如有必要在关键路径加锁或使用线程安全容器。  
3. **回退兼容**：旧的单 group 代码被完全替换，若外部仍假设 `group_sizes` 与 `block_indices` 为单值，需要更新相应的消费方（如日志、监控）。  
4. **性能基准**：因为现在会遍历所有 groups 并多次调用 `cdiv`，建议在大模型/多组配置下跑一次基准，确认调度开销未显著上升。  
5. **异常处理**：当前仍保留若干 `assert`（如 `len(group_blocks) >= num_gpu_blocks`），考虑在生产环境改为抛出可捕获的异常，以免因非法配置导致进程崩溃。  

总体而言，此次改动为 KV‑offload 增加了关键的多‑group 支持，但需要通过针对不同 group 配置的单元/集成测试来确保计算逻辑的正确性与线程安全。
