### [Model] Add HrmTextForCausalLM (Hierarchical Reasoning Model — Text) (#43098)
**SHA**: `a52205b` | 🔗 [查看提交](https://github.com/vllm-project/vllm/commit/a52205bccfc7c270dd445cf253f18deb3778e334)

**🎯 变更类型**：功能增强 / 架构变更  
**⚡ 重要程度**：🔴 高  
**📋 变更摘要**：  
1. 在 vLLM 中加入对 **HRM‑Text**（Hierarchical Reasoning Model – Text）模型的完整支持，新增模型实现 `HrmTextForCausalLM`、对应的注意力层 `PrefillPrefixLMAttention` 以及权重加载映射。  
2. 为非因果（Prefix‑LM）注意力引入 KV‑cache 标记 `non_causal=True`，并在调度器初始化阶段自动关闭 **chunked prefill** 与 **prefix caching**，以避免调度冲突。  

**🎯 影响范围**：  
- `vllm/model_executor/models/hrm_text.py`（模型主体）  
- `vllm/model_executor/layers/attention/prefill_prefix_lm_attention.py`（新注意力实现）  
- `vllm/v1/kv_cache_interface.py`、`vllm/v1/engine/core.py`（KV‑cache 规范与调度逻辑）  
- `vllm/model_executor/models/registry.py`、文档、单元测试注册等  

**🔍 技术洞察**  

| 维度 | 影响描述 |
|------|----------|
| **架构影响** | • 引入 **Prefix‑LM** 注意力模式：在预填阶段（prefill）实现 **双向**（非因果）自注意力，同时保持 `attn_type=DECODER` 以继续使用 KV‑cache。<br>• 每个循环步骤在模型内部生成独立的 `PrefillPrefixLMAttention` 实例，借助 `layer_idx_in_stack` 与全局步骤索引形成唯一的层标识，从而为每一步分配独立的 KV‑slot（满足 HRM‑Text 的 “层级递归” 需求）。<br>• 新增 `non_causal` 标记在 `FullAttentionSpec` 中传播至 `KVCacheSpec`，并在 `engine/core.py` 汇总后统一禁用不兼容的调度特性（chunked prefill、prefix caching）。 |
| **性能影响** | • **正向**：实现了 HRM‑Text 所需的双向预填，使推理结果与原 HF 实现保持一致。<br>• **负向**：启用 `non_causal` 会 **关闭** chunked prefill 与 prefix caching，导致大模型在长提示的预填阶段可能出现 **显著的吞吐下降**（尤其在多用户并发场景下）。<br>• 由于每个递归步骤生成一个独立的注意力对象，模型初始化期间会产生 **O(H·L)** 个注意力层实例，增加显存占用（每层持有 KV‑cache 元数据），但实际运行时仍只占用同样数量的 KV‑slot。<br>• 现有优化（如 sliding‑window、kv‑cache 分页）仍可使用，只要不依赖 chunked prefill。 |
| **安全考虑** | • 新增模型注册时在 `tests/models/registry.py` 为其声明 `min_transformers_version="5.9.0"`，降低了因旧版 Transformers 解析错误导致的潜在安全/稳定风险。<br>• `trust_remote_code=True` 未在 HRM‑Text 条目中使用，保持了 **不自动执行远程代码** 的安全策略。<br>• 代码本身未引入新的 I/O、权限提升或外部依赖。 |
| **可维护性** | • 代码遵循已有的 **layer‑wise** 设计模式（使用 `MergedColumnParallelLinear`、`RMSNorm`、`PrefillPrefixLMAttention`），易于后续迁移至量化或 TP/PP 方案。<br>• `PrefillPrefixLMAttention` 通过复用 `EncoderOnlyAttention` 包装器实现，避免了大量重复实现，但对 `EncoderOnlyAttention` 的实现细节有更高的耦合度。<br>• `non_causal` 标记的加入是一次性改动，后续如果出现其他非因果层（如 RecurrentLM）可直接复用同一机制。 |

**⚠️ 潜在风险**  

1. **调度退化**：非因果层会强制关闭 chunked prefill 与 prefix caching，可能在高并发、长上下文场景下导致显著的 QPS 与吞吐下降。  
2. **层级索引冲突**：`global_idx = step * num_layers_per_stack + layer_idx_in_stack` 依赖 `step` 与 `num_layers_per_stack` 的一致性，若后续模型改变循环步数或层数（例如加入 GQA），需要同步更新此计算。  
3. **TP/PP 兼容性**：当前在 `HrmTextForCausalLM.__init__` 中显式禁止 **pipeline parallelism**；若用户在大规模部署时强行开启，会触发 runtime 错误。<br>   同时，`PrefillPrefixLMAttention` 仍假设 **单一 KV‑cache per layer**，在 **tensor‑parallel** 场景下的跨卡 KV‑cache 合并已通过 `FullAttentionSpec` 的 `non_causal` 传递，但仍需确认 **异构 TP**（如 3‑way）下的正确性。  
4. **权重映射**：`WeightsMapper` 仅做了 `".attn." -> ".self_attn."` 的字符串替换，若 HF 未来对 HRM‑Text 增加新权重前缀，加载将失败。  
5. **测试覆盖不足**：目前只在 `tests/models/registry.py` 增加了注册检查，缺少对 **PrefillPrefixLMAttention** 行为、KV‑cache 非因果标记以及调度禁用逻辑的端到端验证。

### feat: MLA prefill enable FA4 fp8 output (#43050)
**SHA**: `eb04c76` | 🔗 [查看提交](https://github.com/vllm-project/vllm/commit/eb04c769d38d0f39160bd1a2c5279359519de802)

**🎯 变更类型**：功能增强 / 性能优化 / 架构变更  
**⚡ 重要程度**：🔴 高  

**📋 变更摘要**  
- 为 MLA（Multi‑Head Linear Attention）预填（prefill）阶段新增 **FA4‑FP8 融合输出** 功能：在 FlashAttention‑4 中直接写入 FP8（e4m3）量化结果，省去后置量化算子。  
- 扩展 benchmark 脚本、配置与 CLI 参数，支持 `--fp8-output-scale` 与 `--fuse-quant-op`，并在 `mla_fa4_fp8_output.yaml` 中提供示例。  
- 在核心抽象层（`MLAPrefillBackend`、`MLAAttentionImpl.forward_mha`、`flash_attn_varlen_func`）加入 `output_scale` 与 `out` 参数，以及 `supports_quant_output` 能力检查。  
- 新增单元测试，验证能力门与数值等价性（FA4 融合路径 vs bf16‑+‑后置量化）。  

**🎯 影响范围**  
- `benchmarks/attention_benchmarks/`（benchmark 脚本、配置）  
- `vllm/v1/attention/backends/mla/prefill/`（FlashAttention、FlashInfer、TensorSpeed、TRT‑LLM、ROCm_AITER）  
- `vllm/model_executor/layers/attention/mla_attention.py`（融合路径的调度逻辑）  
- `vllm/vllm_flash_attn/flash_attn_interface.py`（FA4 新增 `output_scale` 参数）  
- `tests/v1/attention/test_mla_prefill_quant_output.py`（能力与数值校验）

### 🔍 技术洞察  

| 维度 | 影响描述 |
|------|----------|
| **架构影响** | 1. **抽象签名扩展**：在 `MLAPrefillBackend.run_prefill_new_tokens` 与 `MLAAttentionImpl.forward_mha` 中加入 `out` 与 `output_scale`，所有实现必须保持兼容。<br>2. **能力门**：新增 `MLAPrefillBackend.supports_quant_output(quant_key)`，默认返回 `False`，FlashAttention‑4 实现覆写后返回 `True`（仅在 SM100/110、vLLM‑FA、`kFp8StaticTensorSym`）。<br>3. **路径分支**：`forward_impl` 中通过 `mha_use_quant_output` 判断是走融合路径还是传统 bf16+后置量化路径，保持向后兼容。 |
| **性能影响** | - **省去后置量化 kernel**（`per_token_group_fp8_quant` 或 `QuantFP8`），理论上可减少 0.5–1.5 µs（取决于 batch/seq 长度）并降低显存写入量。<br>- **额外参数检查**（`output_scale`、`out`）开销极低，可忽略。<br>- **硬件限制**：仅在 Blackwell（SM100/110）上可用；在不支持的设备仍回退到原有路径，性能不受负面影响。 |
| **安全考虑** | - 只涉及数值精度的改变，无直接安全风险。<br>- 测试确保 **数值等价**（相对误差 ≤ 12.5 % 以内，且 > 90 % 的元素完全相同），防止因量化误差导致模型输出异常。 |
| **可维护性** | - 抽象层签名改动会波及所有自定义或第三方实现（例如外部插件），需检查是否实现了新增参数。<br>- 增加的 `supports_quant_output` 能力门提供清晰的文档化入口，降低误用风险。<br>- 单元测试覆盖能力判断、签名检查与数值对比，提升回归安全性。 |

   - 任何未更新到新签名的自定义 `MLAPrefillBackend` 实现（第三方或内部 fork）将在运行时因缺少 `out`/`output_scale` 参数而抛异常。  
   - 入口 `MLAAttentionImpl.forward_mha` 的新参数默认 `None`，但若调用方显式传递 `output_scale` 而后端不支持（返回 `False`），仍会进入传统路径，这一点已通过 `supports_quant_output` 检查，但误用仍可能导致意外的 `None` 写入。  

2. **硬件检测**  
   - 依赖 `torch.cuda.get_device_capability()` 判断 SM10/11；在多 GPU 环境或设备能力返回 `None`（如驱动异常）时将默认为不支持，导致基准测试误报。  

3. **性能回归**  
   - 虽然融合路径本身更快，但在极端小 batch（如 `q1`）时启动 CUDA kernel 的固定开销可能抵消收益，需要在 CI 中加入不同规模的基准对比。  

4. **数值一致性**  
   - FP8 静态量化的误差容忍度设置为 0.125（rtol）+ 2*scale（atol），在极端激活分布下可能出现更大偏差，影响模型的推理准确性。  

5. **文档/配置遗漏**  
   - 新增 CLI 参数未在官方文档或 `--help` 中显式标记为实验特性，使用者可能在生产环境误启。

| 维度 | 影响说明 |

| **性能影响** | - **运行时开销**：`CorsLayer` 在每个请求上执行一次头部检查与响应写入，开销极低（几微秒），对高并发的影响可忽略。<br>- **额外拷贝**：`strip_cors_on_no_origin` 会在没有 `Origin` 的请求后再次遍历并删除若干响应头，成本同样在微秒级。整体对吞吐量的影响在 **<1%** 左右。 |
