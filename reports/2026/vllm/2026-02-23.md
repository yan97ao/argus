### [Refactor] Decouple TimingContext from InputProcessingContext (#35083)
**SHA**: `3926454` | 🔗 [查看提交](https://github.com/vllm-project/vllm/commit/392645454b34fec6ff2b690465b275e773e6af09)

**🎯 变更类型**：重构 / 架构变更  
**⚡ 重要程度**：🔴高  

**📋 变更摘要**  
1. 将多模态处理相关的计时功能从 `InputProcessingContext` 中抽离，新增 `TimingContext` 与 `MultiModalTimingRegistry`，实现基于请求 ID 的独立计时记录与聚合。  
2. 统一多模态处理入口，使用全新的 `ProcessorInputs` 数据类代替原先零散的 `prompt、mm_items、mm_uuid_items、hf_processor_mm_kwargs、tokenization_kwargs` 参数，实现 **一次性传参**、更易扩展。  
3. 相应地更新了所有模型执行器、基准脚本、渲染器以及单元测试，使其使用 `processor(inputs, timing_ctx)` 或 `processor(inputs)`（内部创建 disabled `TimingContext`）的新调用方式。  

**🎯 影响范围**  
- `vllm/multimodal/processing/*`（context、dummy_inputs、processor、inputs、__init__）  
- 所有模型实现文件（`clip.py`, `deepseek_vl2.py`, `h2ovl.py`, `llava.py`, `paligemma.py`, `pixtral.py`, `siglip.py`, `terratorch.py`, `transformers/multimodal.py`, `voxtral.py`）  
- 渲染器基类 `vllm/renderers/base.py`（计时注册、MM 处理调用）  
- 基准与统计脚本 `vllm/benchmarks/mm_processor.py`（计时键名、统计逻辑）  
- `vllm/v1/worker/gpu_model_runner.py`（encoder 计时字段改名）  
- `vllm/multimodal/registry.py`（新增 `MultiModalTimingRegistry`，去除对 `ObservabilityConfig` 的直接依赖）  
- 单元测试文件中所有对 `processor.apply(...)` 的调用被改为直接调用 `processor(...)`。  

---

### 🔍 技术洞察  

| 维度 | 影响 |
|------|------|
| **架构影响** | • **职责单一化**：`TimingContext` 只负责计时，`InputProcessingContext` 只保留模型配置、tokenizer 等核心信息。<br>• **解耦请求上下文**：不再使用全局 `contextvars` 保存 request_id，改为在 `Renderer` 层显式获取 `TimingContext`（`MultiModalTimingRegistry.get(request_id)`），提升可测试性与跨线程安全性。<br>• **统一输入结构**：`ProcessorInputs` 把所有参数封装为一个对象，便于未来增加新字段（如自定义 meta、扩展的哈希配置）而无需修改大量函数签名。 |
| **性能影响** | • **计时开销**：使用 `time.perf_counter()` 的上下文管理器实现，开销与原实现基本等价。<br>• **内存/锁**：`MultiModalTimingRegistry` 为每个请求创建 `TimingContext`（字典+锁），在高并发场景下会产生额外的字典条目和锁竞争；但在 `ObservabilityConfig.enable_mm_processor_stats` 关闭时，创建的是 `enabled=False` 的轻量对象，几乎无性能损耗。<br>• **统计聚合**：`collect_mm_processor_stats` 改为直接遍历 `registry.stat()`，省去对 `InputProcessingContext.timing_stats_registry` 的双层锁读取，稍微提升统计阶段的吞吐。 |

| **可维护性** | • 通过 `ProcessorInputs`、`TimingContext` 两个明确的实体，代码可读性提升，API 调用点集中，后续功能（如多阶段计时、跨模型共享计时）更易实现。<br>• 旧的 `apply` 方法已被内部 `__call__` 包装，保持向后兼容（通过 `ProcessorInputs` 自动构造），但外部直接使用 `apply` 的代码必须迁移。 |
| **兼容性** | • 公开 API 已改变：所有模型处理器的 `apply` 现在接受 `(ProcessorInputs, TimingContext)`，而 `__call__` 仍提供兼容包装，确保内部调用（如 `BaseDummyInputsBuilder.get_dummy_processor_inputs`) 正常。<br>• 单元测试已同步更新，说明项目内部已全部迁移；外部使用者需更新调用方式或依赖 `processor(inputs)`（自动创建禁用计时的 `TimingContext`）。 |
| **可观测性** | • 计时键名统一为 `*_secs`，对应 `EncoderTimingStats.encoder_forward_secs`，简化了监控系统的统一单位。<br>• `MultiModalTimingRegistry.stat()` 在每次聚合后会清空内部字典，防止内存泄漏。 |

### ⚠️ 潜在风险  

1. **API 破坏**：外部项目仍在使用 `processor.apply(prompt, mm_items, ...)` 将直接编译错误。需要在文档和迁移指南中明确说明。  
2. **未同步的第三方插件**：如果有人在自定义 `BaseMultiModalProcessor` 子类时仍调用父类的 `apply`（未改为 `ProcessorInputs`），会出现参数不匹配错误。  
3. **计时开关忘记**：在高并发部署中若忘记关闭 `enable_mm_processor_stats`，会为每个请求生成 `TimingContext` 并保留统计字典，可能导致内存峰值上升。  
4. **错误的 `TimingContext` 使用**：如果在 `Renderer` 之外自行创建 `TimingContext(enabled=False)` 并传入，计时会被静默丢失，导致监控数据缺失。  
5. **多线程安全**：`MultiModalTimingRegistry` 使用 `threading.Lock` 包装字典操作；在极端并发（数千请求/秒）下仍可能出现锁竞争导致微小延迟。  
6. **旧字段遗留**：`ProcessorInputs` 中仍保留 `mm_uuid_items`，但部分旧代码可能仍访问 `inputs.mm_items`（已删除），迁移不彻底会导致 `AttributeError`。

### [ModelBash][DSV3] Add TRTLLM DSV3 Router GEMM kernel (6% B1 Speedup) (#34302)
**SHA**: `8435b2e` | 🔗 [查看提交](https://github.com/vllm-project/vllm/commit/8435b2e0492525b2ae7361b7f0081febb483bb34)

**🎯 变更类型**：功能增强 / 性能优化  
**⚡ 重要程度**：🔴高  
**📋 变更摘要**：  
- 为 DeepSeek V3（以及 KIMI‑K2）模型新增专用的 Router‑GEMM CUDA kernel，实现 `mat_a @ mat_b.T`，支持 BF16 输入、Float/BF16 输出，针对 batch ≤ 16、专家数 256/384、隐藏维度 7168 的常用配置做了深度优化。  
- 在 CMake 中加入对 SM90+（Hopper/Blackwell）且 CUDA ≥ 12.0 的编译条件，自动在兼容环境下编译该 kernel。  
- 在 Python 层暴露 `ops.dsv3_router_gemm`，并在 DeepSeek‑V2 MoE 实现中根据硬件与批量大小自动切换到该 kernel，预计可带来约 6%（B1）推理加速。

**🎯 影响范围**：  
- **核心计算模块**：`csrc/moe/*`（新增 3 个 CUDA 源文件 + utils 头文件）  
- **构建系统**：`CMakeLists.txt`（新增编译选项、架构检查）  
- **Python 绑定**：`csrc/moe/torch_bindings.cpp`、`vllm/_custom_ops.py`  
- **模型执行层**：`vllm/model_executor/models/deepseek_v2.py`（新增 `DeepSeekV2Gate` 并接入新 kernel）  
- **公共头文件**：`csrc/moe/moe_ops.h`（声明新 API）  

---

### 🔍 技术洞察

| 维度 | 影响描述 |
|------|----------|
| **架构影响** | - 引入 **DSV3 Router GEMM** 专用 kernel，仅在 **SM90+** GPU（Hopper、Blackwell）以及 **CUDA ≥ 12.0** 环境下编译。<br/>- 通过 `cuda_archs_loose_intersection` 动态筛选兼容的 compute capability，保持向后兼容：不满足条件的环境仍使用原有 MoE 实现。<br/>- 新增 `dsv3_router_gemm` C++/CUDA 接口，并在 Python 层封装，实现 **zero‑overhead** 的后端切换。 |
| **性能影响** | - 采用 **vector‑load（uint4）+ BF16→float 转换**、 **warp‑level butterfly reduction** 与 **grid‑level dependency control**（`griddepcontrol.wait/launch_dependents`）等低层优化，显著降低共享内存占用并提升算子流水线利用率。<br/>- 对 **num_tokens ≤ 16**、**hidden_dim = 7168**、**num_experts = 256/384** 的常见 DeepSeek 配置，实现约 **6% B1 推理加速**（相较于通用 CUDA GEMM）。<br/>- 支持 **PDL（Programmatic Stream Serialization）** 通过环境变量 `TRTLLM_ENABLE_PDL`，在 SM90+ 上可进一步提升并发调度效率。 |
| **安全考虑** | - 仅涉及算子实现与编译条件，没有引入外部库或网络交互，安全面风险几乎为零。<br/>- 新增的环境变量读取逻辑已通过 `std::call_once` 防止竞态，不会产生未定义行为。<br/>- 仍需确保 **CUDA driver 与 runtime 版本匹配**，防止在不兼容的驱动上加载时触发非法指令异常。 |
| **可维护性** | - 代码结构与原 NVIDIA TensorRT‑LLM 实现保持一致，便于后续同步上游改动。<br/>- 使用模板特化统一 `invokeRouterGemm*`，避免代码复制。<br/>- 在 `moe_ops.h` 用 `#ifndef USE_ROCM` 隔离，仅在 CUDA 环境编译，降低跨平台破坏风险。 |
| **兼容性** | - 对老旧 GPU（SM<90）或 CUDA < 12.0 的用户保持 **不构建**、**不导入** 该 kernel，运行时仍会走原有 MoE 路径。<br/>- Python API `ops.dsv3_router_gemm` 采用 **in‑place output**，与现有 API 风格一致。<br/>- 需要在 `torch.ops._moe_C` 中注册新函数，已在 `torch_bindings.cpp` 增加对应实现。 |

### ⚠️ 潜在风险

1. **硬件/软件兼容风险**  
   - 若用户的 GPU 为 **SM<90** 或 CUDA 版本低于 **12.0**，CMake 会跳过编译，但仍会尝试导入 Python API；若包装层未做好 fallback（如 `torch.ops._moe_C.dsv3_router_gemm` 未定义），会导致运行时 `AttributeError`。  
   - 在混合多卡（不同架构）场景下，只有部分卡会装载该 kernel，可能出现 **跨卡不一致** 的性能表现。

2. **维度约束硬编码**  
   - 当前实现强制 `hidden_dim == 7168`、`num_experts ∈ {256,384}`、`num_tokens ≤ 16`。若在未来出现类似模型但维度略有变化，将直接触发 `TORCH_CHECK` 异常，需手动更新 kernel 参数或回退到通用实现。

3. **数值精度**  
   - BF16 → float 转换后再写回 BF16 可能在极端数值范围产生微小误差。虽然对路由 logits 影响有限，但在 **极端梯度放大** 场景下需留意潜在偏差。

4. **环境变量依赖**  
   - `TRTLLM_ENABLE_PDL` 默认关闭，若用户开启但硬件不支持（SM<90），`getEnvEnablePDL` 会返回 `false`，但仍会产生 **属性查询开销**。未严格验证该变量的合法值可能导致误判。

5. **构建复杂度**  
   - 新增的 `cuda_archs_loose_intersection` 逻辑可能在自定义 `CUDA_ARCHS` 时产生误判，导致误编译或遗漏编译。需要在 CI 中加入 **不同 Compute Capability** 的矩阵测试。

### ⚠️ 潜在风险
1. **API 破坏**：外部使用者仍在调用 `get_dummy_mm_data(..., mm_options=None)` 或 `...mm_processor_kwargs=...` 时会报错，导致生产环境服务启动失败。  
2. **默认行为变化**：旧实现在未显式提供 `mm_options` 时会自动使用 `None` → 进而返回默认 `BaseDummyOptions`（即不覆盖任何属性）。新实现如果忘记传入空 dict，则会因缺少必填参数直接报错，而不是回退到默认行为。  
3. **TypedDict 丢失运行时检查**：`TypedDict` 只在静态类型检查阶段有效，运行时仍是普通 dict；若误传入非 `BaseDummyOptions` 对象，仍会在后续调用 `_get_dummy_*` 时触发属性错误。  
4. **配置迁移**：`MultimodalConfig._validate_limit_per_prompt` 现在返回 `MMDummyOptions`，但旧的 `limit_per_prompt` 仍可能保留 `int` 或 `dict[int]` 形式的值，若用户直接手动编辑配置文件而不走 `pydantic` 验证，可能导致运行时异常。

### [Spec Decode] Reduce TP communication for speculative decoding draft token generation (#34049)

- 推理层：`vllm/v1/spec_decode/eagle.py`

5. **性能回退**：在极小 vocab 或 TP=1 时，局部 argmax 并不会带来收益，建议在配置脚本中根据 `vocab_size` 与 `tp_size` 自动禁用。  

总体而言，此改动在保持原有功能的前提下，显著降低了 speculative decoding 的跨卡通信开销。若测试通过，可在大模型（数十‑上百B）和多卡部署场景默认开启。
