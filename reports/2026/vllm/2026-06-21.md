### [Spec Decode] Support mixed KV page sizes for DFlash (#45181)
**SHA**: `2cac89f` | 🔗 [查看提交](https://github.com/vllm-project/vllm/commit/2cac89f9da865dfaceb6d337d97aaff5c9195e48)

**🎯 变更类型**：功能增强 / 重构（引入混合 KV 页面大小的支持，扩展后端能力检测，重构 KV 缓存布局与 reshape 逻辑）

**⚡ 重要程度**：🔴高  

**📋 变更摘要**  
1. 在 `AttentionSpec` 中新增 `indexes_kv_by_block_stride` 标记，并在所有获取 KV 缓存规格的路径里自动填充该标记（依据后端实现）。  
2. `kv_cache_utils.unify_kv_cache_spec_page_size` 现在能够在页面大小不整除时采用 **页填充**（而非仅通过增大 block_size），前提是后端声明支持 `indexes_kv_by_block_stride`。  
3. 统一的 KV 重新 reshape 逻辑抽取为 `_reshape_attention_kv_cache`，实现基于 **stride‑view** 的跨页填充读取，避免额外拷贝。  
4. `Backend.indexes_kv_by_block_stride` 方法实现，统一判断后端是否满足 “块首维” 布局需求。  
5. 相应的单元测试、`kv_connector` 判定、模型运行时统一缓存路径均已更新，以保障新特性在 DFlash、FlashAttention、DiffKV 等后端下正确工作。  

**🎯 影响范围**  
- `vllm/v1/kv_cache_interface.py`（AttentionSpec 新字段、合并/验证逻辑）  
- `vllm/v1/attention/backend.py`（后端 capability 接口）  
- `vllm/v1/core/kv_cache_utils.py`（页面统一算法）  
- `vllm/v1/worker/gpu/attn_utils.py`、`gpu_model_runner.py`（reshape 关键实现）  
- `vllm/v1/worker/kv_connector_model_runner_mixin.py`（统一缓存判定）  
- 测试目录 `tests/v1/...`（新增/修改单元测试）

### 🔍 技术洞察

| 维度 | 影响描述 |
|------|----------|
| **架构** | - **后端能力抽象化**：通过 `Backend.indexes_kv_by_block_stride()` 将“块首维”布局的支持从具体实现中抽离，统一在 `AttentionSpec` 上标记，提升后端可插拔性。<br>- **统一 KV 页面**：`unify_kv_cache_spec_page_size` 现在能在 **非整数倍** 场景下使用 **页填充**，使得不同注意力层（如 Draft‑Attention 与 Target‑Attention）可以共享同一块 KV 缓存，而无需强制统一 block size。<br>- **跨层统一缓存**：`use_uniform_kv_cache` 改为直接检查 `indexes_kv_by_block_stride`，简化判定逻辑，降低对内部 stride‑order 实现细节的耦合。 |
| **性能** | - **内存利用率提升**：页填充避免了因为强行增大 block size 导致的额外内存浪费，尤其在 Draft‑Attention 使用更小的 head size（如 128 vs 192）时可显著降低 KV 内存占用。<br>- **零拷贝视图**：`_reshape_attention_kv_cache` 使用 `torch.as_strided` 对物理页面进行 stride‑view，避免了额外的 `reshape` 或 `copy`，对吞吐量基本无负面影响，仅在页填充时额外一次 stride 计算。<br>- **后端兼容性**：仅在后端明确声明支持 `indexes_kv_by_block_stride` 时才采用填充，避免在不支持的后端产生错误或额外的 stride‑转换开销。 |
| **安全** | - **内存访问安全**：使用 `as_strided` 必须确保 stride 与大小匹配，否则会产生越界读取。代码已在 `assert kv_cache_shape[0] == num_blocks` 前置检查，确保只在 “num‑blocks‑first” 布局下使用。<br>- **后端协议一致性**：新增的 `indexes_kv_by_block_stride` 通过显式后端查询获取，如果后端未实现或返回错误，会回退到旧的 `NotImplementedError`，避免未知行为。 |
| **可维护性** | - **代码复用**：reshape 逻辑抽成 `_reshape_attention_kv_cache`，原本在 `gpu_model_runner` 与 `attn_utils` 中的重复实现被合并，降低维护成本。<br>- **新增字段传播**：`AttentionSpec.merge`、`KVCacheGroupSpec.merge` 等合并路径已同步 `indexes_kv_by_block_stride`，保证在跨层合并时不遗漏该标记。<br>- **单元测试覆盖**：针对 padding、后端不支持情况、跨层统一缓存配置均添加了明确的测试，用例覆盖率提升。 |

### ⚠️ 潜在风险

1. **后端实现遗漏**  
   - 某些自定义或第三方后端可能未实现 `indexes_kv_by_block_stride`，导致在混合页面大小场景下仍抛 `NotImplementedError`。若用户在生产环境开启了混合 KV 页面而未更新后端，会导致启动失败。  
2. **`as_strided` 使用错误**  
   - 对 `page_stride` 的计算依赖 `page_size_bytes` 与 dtype 大小的整数除法。若出现非整除（尤其在自定义 dtype 或量化模式下），可能产生非法 stride，触发运行时异常或错误的内存访问。  
3. **缓存统一判定变化**  
   - `use_uniform_kv_cache` 从检查 stride‑order 改为直接依赖 spec 标记，若后端实现的 `indexes_kv_by_block_stride` 返回不一致（如在不同配置下返回不同值），可能导致同一模型在不同机器上产生不同的 KV 布局，进而影响跨机器 reproducibility。  
4. **交叉层共享的副作用**  
   - 当页面被填充后，不同层共享同一块 KV（`KVCacheTensor` 的 `shared_by` 列表），如果后端在读取时误将 padding 部分当作有效数据，可能导致生成错误或数值漂移。

### [Core] Ensure memory is pinned prior to async h2d copy (#45424)
**SHA**: `7df3d7d` | 🔗 [查看提交](https://github.com/vllm-project/vllm/commit/7df3d7dada840c68b85b26b79de7f59f676d58e3)

**🎯 变更类型**：架构变更 / 性能优化  
**⚡ 重要程度**：🔴 高  
**📋 变更摘要**：本次提交统一了“内存是否需要 pinned” 的判断逻辑，新增全局常量 `PIN_MEMORY`（由 `vllm.utils.platform_utils.is_pin_memory_available()` 提供），并在 `vllm.utils.torch_utils` 中实现 `async_tensor_h2d`、`np_to_pinned_tensor` 等工具函数。所有原先在各模块自行判断 `is_pin_memory_available()` 并在 H2D（host‑to‑device）拷贝前自行 `pin_memory()` 的代码被改写为统一调用这些工具函数，从而 **在真正执行异步拷贝前确保数据已被 pinned**，并消除原实现中因 pin‑memory 与非‑pin‑memory 混用导致的潜在 race‑condition。  
此改动遍及核心层 （device_allocator、model_executor、attention backends、kv‑offload、sampling、structured‑output、GPU/CPU model runner、worker 等），并同步更新了大量单元测试。

### 🎯 影响范围
- **核心运行时**：`vllm/device_allocator/*`, `vllm/model_executor/*`, `vllm/v1/attention/backends/*`, `vllm/v1/worker/*`, `vllm/lora/*`, `vllm/platforms/*`  
- **工具函数**：`vllm/utils/torch_utils.py`（新增 `PIN_MEMORY`, `async_tensor_h2d`, `np_to_pinned_tensor`）  
- **测试套件**：几乎所有涉及 GPU/CPU 输入批处理、模型加载、采样、结构化输出等的测试文件均去掉了 `pin_memory` 参数的显式传递。  
- **平台插件**：`vllm/platforms/xpu.py`, `vllm/platforms/cuda.py` 中对 `supports_xccl`/`supports_xpu_graph` 的调用方式略有改动。  

---

### 🔍 技术洞察

| 维度 | 影响 |
|------|------|
| **架构** | - 将 **pin‑memory 判定** 从散落在多个文件的 `is_pin_memory_available()` 调用，集中到 `vllm.utils.torch_utils.PIN_MEMORY` 常量，降低耦合度。<br>- `async_tensor_h2d` 成为统一的 “CPU→GPU 异步拷贝 + 必要的 pin‑memory” 接口，统一了数据流的入口，便于后续在不同平台（CUDA、XPU、ROCM）上实现特化。 |
| **性能** | - 之前的实现：在某些路径里先 `tensor.pin_memory()`，随后又在拷贝时显式 `pin_memory=False`，导致 **CPU‑GPU 同步**（因为未 pin‑memory）。<br>- 现在 `async_tensor_h2d` **始终在拷贝前确保 pinned**（如果平台支持），配合 `non_blocking=True`，可以让 GPU 流并行进行拷贝和计算，提升高并发推理的吞吐。<br>- 移除 `pin_memory` 参数传递，减少 Python 层的分支判断，使热点路径更轻量。 |
| **安全** | - 改动仅涉及内存管理与拷贝，不改变计算结果的数值或模型权重的完整性。<br>- 通过统一的 `PIN_MEMORY`，避免了在不支持 pinned memory（如部分 WSL 环境）下误用 `torch.tensor(..., pin_memory=True)` 带来的 **非法内存访问**。<br>- 仍然依赖底层 PyTorch 对 `pin_memory` 的实现，若 PyTorch 在未来版本更改其语义，需要再次回归测试。 |
| **可维护性** | - 统一的工具函数让后续**新增拷贝点**只需调用一次 `async_tensor_h2d`，降低代码重复。<br>- 删除了大量 `pin_memory` 参数的传递，使函数签名更简洁。<br>- 仍有少量残余直接引用 `is_pin_memory_available()`（例如 `vllm/v1/attention/backends/flashinfer.py` 已改为 `PIN_MEMORY`，但 `vllm/v1/attention/backends/flex_attention.py` 现在通过 `async_tensor_h2d`），需要一次性搜索确认全部迁移完毕。 |
| **兼容性** | - `async_tensor_h2d` 在接受已有 `torch.Tensor` 时会 `pin_memory()`（若可用）再 `.to(device, non_blocking=True)`，保持向后兼容。<br>- 对 **CPU‑only** 运行（`device="cpu"`）的路径也保持行为一致：返回原始 CPU tensor（不触发拷贝），且 `pin_memory` 参数已被内部统一管理。 |
| **资源占用** | - Pinned内存会占用系统固定内存池，开启后在极端并发（数千个请求）时可能导致 **pinned‑memory pool 用尽**。但该问题在原实现中已经存在，仅是现在统一了开启方式。建议在生产部署中监控 `torch.cuda.memory_reserved()` 与 `torch.cuda.memory_allocated()` 之外的 `pinned` 使用情况。 |

### ⚠️ 潜在风险

1. **平台检测失效**：`PIN_MEMORY = is_pin_memory_available()` 仍旧依赖 `platform_utils` 中的实现。若该函数在某些新平台（如 AMD ROCm、Windows WSL）返回错误值，可能导致 **强制 pin‑memory** 而实际不可用，引发异常。  
2. **内存池耗尽**：统一开启 pin‑memory 会在高并发时快速占满系统的 pinned‑memory pool，导致 `torch.cuda.OutOfMemoryError: cannot allocate pinned memory`。  
3. **类型/设备不匹配**：`async_tensor_h2d` 在接受已是 GPU Tensor 时仍会执行 `data.to(dtype=dtype)`，可能导致意外的 **数据拷贝回 CPU 再前往 GPU**（尤其当 `dtype` 参数为 `None` 并且 `data` 已在 GPU）。  
4. **未清理的旧 API**：部分代码仍保留对 `is_pin_memory_available()` 的直接引用（如 `vllm/platforms/__init__.py` 中的 `supports_xccl` 替换为 `torch.distributed.is_xccl_available()`），如果遗漏会导致 **ImportError** 或 **未使用的标志位**。  
5. **测试覆盖不足**：大量改动集中在内部工具函数，若有特定路径（例如 XPU‑specific图形模式）未走 `async_tensor_h2d`，可能仍保持旧的同步行为，导致 **性能回退**。

### Stop setting CUDA_VISIBLE_DEVICES internally in vLLM, add device_ids arg (#45026)
**SHA**: `ebfbcfe` | 🔗 [查看提交](https://github.com/vllm-project/vllm/commit/ebfbcfe46aa895d428933244908bae08b1ca6397)

**🎯 变更类型**：功能增强 / 架构变更  
**⚡ 重要程度**：🔴高  

**📋 变更摘要**  
本次 PR 彻底去除了 vLLM 在内部自行设置 `CUDA_VISIBLE_DEVICES`（CVD）的逻辑，新增 `--device-ids` 参数，以让用户自行指定 **物理 GPU ID**（整数或 UUID）。为实现该目标，框架在平台层引入了“逻辑‑可见‑物理”三层映射：  
1. **逻辑本地 ID**（vLLM 在内部使用的 rank）  
2. **可见设备序号**（进程 `CUDA_VISIBLE_DEVICES` 中出现的顺序）  
3. **物理设备 ID**（NVML/PCIe 上的唯一编号）  

新增 `assigned_physical_gpu_ids` 配置字段，配合 `set_assigned_physical_gpu_ids()`/`get_assigned_physical_gpu_ids()` 在进程启动阶段完成映射并在后续拓扑查询、P2P 检查、NCCL 初始化等环节使用。

**⚠️ 潜在风险**  

1. **用户迁移错误**  
   - 原有业务可能依赖 vLLM 自动把 `CUDA_VISIBLE_DEVICES` 设为 `[0..N-1]`，迁移后若不显式传递 `--device-ids` 且环境变量被外部工具（如 Slurm、K8s）裁剪，逻辑‑>‑物理映射可能产生 **“可见设备不在物理列表”** 的异常。  
   - 建议在部署脚本中明确使用 `--device-ids`，或确认外部调度器已正确设置 CVD。

2. **混合整数/UUID 参数**  
   - `device_ids` 仍然接受整数索引（相对于 CVD）或 UUID，但不允许混用。若用户在 CLI 中误混，当前实现会抛 `ValueError`，可能导致任务启动失败。需要在文档和错误提示中再次强调。

3. **缓存键不一致**  
   - GPU P2P 检查缓存文件名由 `assigned_physical_gpu_ids` 生成，若同一机器在不同进程使用同一物理集合但顺序不同（如 `[4,5]` 与 `[5,4]`），会产生 **重复计算**。虽然不是错误，但可能浪费磁盘 I/O。可考虑在未来对集合进行排序后再生成键。

4. **Ray 环境变量同步**  
   - 虽然 `RayExecutor` 已不再修改 `CUDA_VISIBLE_DEVICES`，但仍在 `worker_initialize` 时通过 `env_vars` 传递空字典。若用户自行在 Ray Actor 中手动设置 CVD，可能导致 **双重映射**（CVD 中的顺序与 `assigned_physical_gpu_ids` 冲突），触发运行时 `RuntimeError`。需要在使用文档中提醒。

5. **多进程共享全局状态**  
   - `set_assigned_physical_gpu_ids` 在 `vllm/v1/executor/multiproc_executor.py` 中在每个子进程的 `spawn` 前调用，保证了每个进程的全局变量一致。但如果用户在同一进程内多次创建 `VllmConfig`

**SHA**: `c88d3d4` | 🔗 [查看提交](https://github.com/vllm-project/vllm/commit/c88d3d4775c41793543e97f1609d3161a5689905)

**🎯 变更类型**：功能增强  
**⚡ 重要程度**：🟡 中  

**📋 变更摘要**  
本次提交为 SimpleCPUOffload 调度器引入了 **Context‑Parallel（CP）** 支持，新增对 DCP（Decode Context Parallel）和 PCP（Prefill Context Parallel）的感知。核心改动包括：  
1. 在 `SimpleCPUOffloadScheduler` 中计算并保存 `cp_world_size`（`dcp * pcp`），并依据该值对 `block_size`、`hash_block_size` 以及懒加载目标块数进行比例放大。  
2. 调整块大小的计算（`kv_cache_spec.block_size * cp_world_size`），以及在 `update_state_after_alloc` 中的块数推算。  
3. 为测试层新增 3 条 CP 场景的单元测试，覆盖块大小伸缩、同步存取以及懒加载目标块的缩减行为。  

**🎯 影响范围**  
- `vllm/v1/simple_kv_offload/manager.py`（调度器核心逻辑）  
- `tests/v1/simple_kv_offload/test_scheduler.py`（新增 CP 相关测试）  
- 相关配置结构 `VllmConfig.parallel_config`（解码/前置并行尺寸）  

**💡 关注建议**  

1. **兼容性检查**：`cp_world_size` 默认仍为 1，确保未开启 CP 的旧版调用不受影响；建议在文档中明确说明开启 CP 需要的 `parallel_config` 参数。  
2. **性能基准**：块大小随 CP 成倍放大，GPU BlockPool 的 `hash_block_size` 也随之变化，可能导致哈希冲突或内存碎片。建议在不同 `cp_world_size`（1、2、4）下跑一次完整的吞吐量基准，以验证实际收益与额外开销。  
3. **错误处理**：当前在 `update_state_after_alloc` 中直接使用放大后的 `g_block_size`，若 `cp_world_size` 为 0（误配置）会出现除零错误。可以在初始化时加入断言 `assert cp_world_size > 0`。  
4. **测试覆盖**：新增的 3 条测试已覆盖关键路径，但仍缺少对 **多组 KVCache（Mamba/SlidingWindow）** 与 CP 组合的交叉验证，建议补充对应场景。  
5. **文档与示例**：在 README 或使用手册中添加 CP 启动示例及对 CPU 容量、GPU BlockPool 大小的调参建议，帮助用户快速上手。  

总体来看，此次改动为 vllm 在大模型并行推理下的 CPU‑offload 场景提供了必要的伸缩能力，影响范围局限于调度器实现和对应单元测试，风险可控。若遵循上述建议进行充分的回归和文档补全，可安全合并。

### [Perf][KVConnector][Mooncake] Compact chunk-hash keys and zero-copy lookup wire format (#45969)

**🎯 变更类型**：功能增强 / 性能优化  
**⚡ 重要程度**：🟡 中  

**📋 变更摘要**

3. 相关 API 从 `list[BlockHash]` 改为更宽松的 `Sequence[BlockHash]`，统一使用 `chunk_hashes_for_block_size` 生成紧凑哈希序列。

**🎯 影响范围**

- `vllm/v1/core/kv_cache_utils.py`（`BlockHash`、`BlockHashListWithBlockSize` 的使用）  
- 相关单元测试文件。  

**💡 关注建议**  
1. **兼容性**：Wire 协议已从 Msgpack‑list 改为 `[hash_len][blob]`，需确认所有旧版客户端/服务器在升级时能正确回退或同步更新，否则会出现解码错误。  
2. **序列化安全**：`BlobBlockHashes` 依赖 `hash_len` 为正整数且能被 `blob` 长度整除，建议在构造时加入断言或异常提示，以防错传。  
3. **性能验证**：虽然理论上减少拷贝与哈希拼接开销，但实际收益取决于块大小比例。建议在大 batch / 长序列场景下跑基准，对比旧实现的延迟与内存占用。  
4. **代码审查**：`chunk_hashes_for_block_size` 返回 `Sequence[BlockHash]`，其中 `_CompactChunkHashList` 仍继承自已废弃的 `BlockHashListWithBlockSize`，如果后者未来被删除，需迁移到纯抽象接口。  
5. **文档/注释**：更新 README/协议文档，标明新的 “hash_len + raw‑hashes” 格式以及紧凑键的语义（最后子哈希唯一标识整个块）。  

总体来看，此次改动通过压缩键空间和零拷贝传输显著降低了 KV‑cache 查找的内存与网络开销，但需要对兼容性、异常处理以及后续维护做好补充说明。
