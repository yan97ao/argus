### [Attention Backend] TurboQuant: 2-bit KV cache compression with 4x capacity (#38479)
**SHA**: `f4b42df` | 🔗 [查看提交](https://github.com/vllm-project/vllm/commit/f4b42df04847dcbd3247f7f4c56dff45e40bdf0d)

**🎯 变更类型**：功能增强 / 架构变更 / 性能优化  
**⚡ 重要程度**：🔴高  
**📋 变更摘要**：  
本次 PR 为 vLLM 引入了 *TurboQuant* KV‑cache 压缩后端（2‑bit/3‑bit/4‑bit 量化 + 8‑bit FP8 键），实现了 KV‑cache 大约 4× 的容量提升并显著降低显存占用。新增了完整的配置、元数据、后端实现、Trion 量化/解压算子以及配套单元测试、CI、文档与驱动层适配。对 KV‑cache 分配、注意力后端选择以及 Engine 参数做了兼容性改造。

---

## 🎯 影响范围
- **核心模块**：`vllm/v1/attention/backends/`, `vllm/v1/kv_cache_interface.py`, `vllm/v1/core/single_type_kv_cache_manager.py`, `vllm/config/*`。  
- **模型执行层**：`Attention`（新增 `_init_turboquant_buffers`、KV‑cache 更新与前向逻辑）。  
- **平台层**：CUDA/XPU 后端的后端调度 (`platforms/cuda.py`, `platforms/xpu.py`) 现在会直接路由到 TurboQuant。  
- **配置/CLI**：`--kv-cache-dtype turboquant_*`、`tq_max_kv_splits_for_cuda_graph`、边界层自动跳过。  
- **测试/CI**：新增 `tests/quantization/test_turboquant.py`、LM‑Eval 配置、CI 步骤。  
- **文档**：`design/attention_backends.md` 增加 TurboQuant 条目。

## ⚠️ 潜在风险

1. **精度回退**  
   - MSE‑量化的键在 3‑bit/4‑bit 模式下会引入噪声，尤其在深层或对键向量敏感的模型（如多模态/长序列）可能出现明显 PPL 增长。  
   - 目前默认跳过前后 2 层可缓解，但不排除在特定模型（如带有层归一化的跨层残差）仍会出现回退。

2. **硬件兼容**  
   - FP8 key 分支依赖 `float8e4b15`（Ampere/Ada）或 `float8e4nv`（Hopper+）。旧的 GPU（如 Turing、Pascal）不支持 FP8，若用户手动启用 `turboquant_k8v4` 在不支持的设备上会在 kernel 启动时报错。  
   - Triton 代码中对 `FP8_E4B15` 的判断基于 `torch.cuda.get_device_capability`，但在多‑GPU 环境下仅使用了 `device.index or 0`，可能导致非主卡的错误检测。

3. **CUDA‑Graph 稳定性**  
   - 固定 `NUM_KV_SPLITS` 为 32（或配置值）对极端长序列仍足够，但如果用户自定义 `tq_max_kv_splits_for_cuda_graph` 过小，会导致 `split_len` 为 0，从而触发 kernel 错误。需要在 Engine 创建时加入合法性校验。

4. **混合模型不支持**

### [MoE Refactor] Refactor ZeroExpertFusedMoE into new framework (#35549)
**SHA**: `19ec9a0` | 🔗 [查看提交](https://github.com/vllm-project/vllm/commit/19ec9a0a6276d5af449b3011dac2944e8af37ee0)

**🎯 变更类型**：重构  
**⚡ 重要程度**：🔴高  

**📋 变更摘要**  
- 将原有的 `ZeroExpertFusedMoE` 类彻底移除，改为在路由层 (`ZeroExpertRouter`) 中直接完成 zero‑expert（零专家）计算，并在 `MoERunner` 的前向路径中统一合并 zero‑expert 的输出。  
- 新增 `ZeroExpertRouter`，在路由阶段使用完整的 `e_score_correction_bias`，对 zero‑expert 进行 top‑k 选取、计算其 identity（或其它）贡献，并将 zero‑expert 的索引/权重屏蔽，使后续的普通 FusedMoE 计算仍保持不变。  
- `FusedMoE` 构造函数新增 `zero_expert_type` 参数，路由工厂 `create_fused_moe_router` 增加 zero‑expert 参数的优先级判断。  
- 测试覆盖新增 `tests/kernels/moe/test_zero_expert_moe.py`，验证 router 创建、前向形状、输出分解以及 zero‑expert 的 identity 行为。  

**🎯 影响范围**  
- **核心模块**：`vllm/model_executor/layers/fused_moe/`（router、layer、runner）  
- **模型实现**：所有使用 MoE 且启用了 zero‑expert 的模型（如 `longcat_flash.py`）  
- **配置/用户 API**：`FusedMoE` 构造函数签名变化，新增 `zero_expert_type` 参数；旧的 `ZeroExpertFusedMoE` 已被删除。  
- **测试套件**：新增 282 行单元测试，覆盖 zero‑expert 路由与前向逻辑。  

**🔍 技术洞察**  

| 维度 | 影响描述 |
|------|----------|
| **架构影响** | 1. **路由职责统一**：原先 `ZeroExpertFusedMoE` 负责 both routing memoization and zero‑expert 计算，现在把 zero‑expert 计算抽取到路由层，保持 `FusedMoE` 只关注真实专家的计算逻辑，层次更清晰。<br>2. **插件化路由**：`router_factory` 新增 zero‑expert 参数，使其成为路由选择的第一优先级，便于以后扩展更多特殊 router。<br>3. **向后兼容**：通过在 `FusedMoE` 中加入 `zero_expert_type` 参数，保持同一类实例即可兼容 zero‑expert 与普通 MoE，避免在模型代码中拆分两套实现。 |
| **性能影响** | 1. **路由阶段额外计算**：在 `ZeroExpertRouter._compute_routing` 中会额外调用 `zero_experts_compute_triton`，这一步仅在 zero‑expert 被启用时执行，计算量非常小（仅对部分 token × zero‑expert 权重做标量乘），对整体吞吐影响可忽略。<br>2. **内存占用**：`zero_expert_output` 在路由后暂存一次，随后在 `MoERunner._maybe_add_zero_expert_output` 中加回，峰值内存略增（约占输出 Tensor 大小的 1/N），在实际部署中可接受。<br>3. **去除冗余路由**：原 `ZeroExpertFusedMoE` 为 zero‑expert 与真实专家分别走两次路由（一次 memo 化，一次普通），现在只做一次路由，减少了潜在的同步和重复计算，整体 latency 预计下降 5%‑10%。 |
| **安全考虑** | - 路由阶段直接使用 `e_score_correction_bias`（由用户或模型提供）进行 zero‑expert 的 top‑k 选取，未加入额外的安全检查。若 bias 被恶意篡改，可能导致 zero‑expert 被过度选中，从而产生过大的 identity 放大效果（`routed_scaling_factor`）。建议在配置层对 `zero_expert_type` 与 `e_score_correction_bias` 做基本校验（长度、数值范围）。 |
| **可维护性** | - 代码路径从 “ZeroExpertFusedMoE + FusedMoE” 两套实现合并为 “ZeroExpertRouter + FusedMoE”，代码量大幅减少（删除 189 行旧实现）。<br>- 新增的 `ZeroExpertRouter.zero_expert_output` 采用 **一次性读取并清除** 的模式，防止误用残留数据。<br>- 单元测试覆盖路由实例类型、无自定义 routing、前向形状、分解正确性以及 identity 行为，提升回归安全性。 |
| **兼容性** | - 直接删除 `ZeroExpertFusedMoE`，因此外部代码若仍显式导入该类会报错。项目内部已改为 `FusedMoE(..., zero_expert_type=…)`，但使用者需检查自定义模型或插件。<br>- `FusedMoE` 接口向后兼容（新增参数有默认 `None`），不影响不使用 zero‑expert 的旧模型。 |

**⚠️ 潜在风险**  
1. **误用旧类**：外部项目仍依赖 `ZeroExpertFusedMoE` 将导致 ImportError。  
2. **路由参数不匹配**：`create_fused_moe_router` 在 zero‑expert 路径要求 `num_logical_experts` 与 `e_score_correction_bias` 同时提供；若配置不完整会触发 assert，可能在动态模型加载时出现启动失败。  
3. **ZeroExpertRouter 与 EPLB 交叉**：当前实现先计算 zero‑expert 再做 EPLB 掩码，若后续 EPC（expert parallelism）对 zero‑expert 有特殊需求，可能需要额外适配。  
4. **数值精度**：zero‑expert 的 identity 乘以 routing 权重后仍保持 `torch.float16/bfloat16`，在极端高权重（如强制路由至 zero‑expert）时可能出现溢出或 underflow，需要在配置 `routed_scaling_factor` 时保持谨慎。  

**💡 关注建议**  
- **迁移指南**：在项目文档中明确标注 `ZeroExpertFusedMoE` 已被移除，推荐使用 `FusedMoE(..., zero_expert_type="identity")`，并提供示例代码。  
- **参数校验**：在 `FusedMoE.__init__` 或配置解析阶段加入对 `zero_expert_type` 与 `e_score_correction_bias` 长度一致性的检查，给出友好错误信息。  
- **性能监控**：在生产环境开启 `VLLM_MOE_ROUTING_DEBUG`（或类似开关）记录 zero‑expert 的路由比例与计算时间，确保新增路径没有意外的性能回退。  
- **安全审计**：若模型部署在多租户环境，考虑对 `e_score_correction_bias` 做签名或只读限制，防止恶意修改导致 zero‑expert 放大攻击。  
- **回归测试**：将新增的 `tests/kernels/moe/test_zero_expert_moe.py` 集成到 CI，确保在不同硬件（CUDA、CPU 仿真）和不同 dtype（fp16、bf16）下均通过。

### 🔍 技术洞察  

- **架构影响**  
  - **模块化后端抽象**：通过 `MambaSSUBackend` 抽象层，将 Triton 与 FlashInfer 实现解耦，提升后端可插拔性，后续可以平滑加入其他实现（如 CUDA‑C++）。  
  - **配置统一**：新增 `MambaConfig`，将原本分散在 `CacheConfig` 的随机化相关字段迁移至 Mamba 专属配置，防止跨模块职责混淆。  
  - **初始化路径**：在 KV cache 初始化阶段统一调用 `initialize_mamba_ssu_backend`，保证后端在首次使用前已完成一次性创建，避免运行时隐式导入错误。  
  - **CLI 与 Pydantic**：为新配置添加 CLI 参数 (`--mamba-backend`、`--enable-mamba-cache-stochastic-rounding`、`--mamba-cache-philox-rounds`) 并在 `EngineArgs.__post_init__` 中完成字典 → `MambaConfig` 的转换，保持向后兼容。  

- **性能影响**  
  - **FlashInfer 后端**：FlashInfer 在 NVIDIA Blackwell（Compute Capability 10.0）GPU 上提供专用 PTX 指令 `cvt.rs`，能够在 SSU 计算中省去 Triton 产生的额外 launch 开销，理论上可提升单 token 延迟 10%‑30%。  
  - **随机化开销**：`enable_stochastic_rounding` 与 `philox_rounds` 仍由后端实现控制，仅在 FlashInfer 后端通过 `rand_seed` 与 `philox_rounds` 参数实现；Triton 仍使用原有实现。若用户打开随机化，额外的 Philox 轮数会产生可预期的计算开销。  
  - **代码路径简化**：原有 `selective_state_update` 参数中 `D`、`dt_bias` 的可选性被统一为必传，减少运行时分支检查，略微提升内核调用效率。  

- **安全考虑**  
  - **依赖安全**：新引入的 `flashinfer` 是外部的二进制库，若未正确安装会在初始化时报错，且已在测试中使用 `skipif` 防止误报。建议在发行版中对 `flashinfer` 的二进制来源进行签名校验。  
  - **配置校验**：`MambaConfig.__post_init__` 对 `enable_stochastic_rounding` 做了平台、硬件、dtype 的严格检查，防止在不支持的环境下误启用导致崩溃或数值错误。  

- **可维护性**  
  - **统一调度函数**：业务代码只调用 `selective_state_update`（来自 `ssu_dispatch`），后端实现细节隐藏，为后续优化和 bug 修复提供单点入口。  
  - **测试覆盖**：新增单元测试验证默认 Triton、显式 Triton、FlashInfer 初始化、错误路径以及未初始化的异常，提升了回归安全性。  
  - **代码整理**：移除 `CacheConfig` 中与 Mamba 随机化相关的字段，避免配置冲突，提升代码可读性。

### ⚠️ 潜在风险  

1. **兼容性风险**  
   - 旧版模型或脚本若仍依赖 `CacheConfig.enable_mamba_cache_stochastic_rounding` 等字段，升级后会报 `AttributeError`。需要在升级指南中明确迁移路径。  
   - 某些自定义插件可能直接导入 `vllm.model_executor.layers.mamba.ops.mamba_ssm.selective_state_update`；该符号已被保留在 `mamba_ssm.py`（仅包装 Triton），但其签名已变化（`D`、`dt_bias` 必传），可能导致兼容性问题。  

2. **FlashInfer 依赖**  
   - 如果用户在不支持的 GPU（非 Blackwell）上强制启用 `--mamba-backend flashinfer`，会在运行时触发 `ValueError`（由 `MambaConfig.__post_init__` 检查），但仍可能在更深层触发硬件异常。需要在文档中强调平台限制。  

3. **随机化实现差异**  
   - Triton 与 FlashInfer 在随机数生成方式上可能不一致（FlashInfer 使用 `torch.randint` + Philox），导致同一配置下结果略有差异，可能影响复现性。  

4. **初始化顺序**  
   - `initialize_mamba_ssu_backend` 必须在首次调用 `selective_state_update` 前执行。当前已在 KV cache 初始化和模型 runner 初始化处调用，但若有其他路径（如独立测试或自定义模型加载）绕过这些入口，则可能出现 “backend not initialized” 异常。

**💡 关注建议**

3. **向后兼容**：此改动仅在 `sym_shape_indices` 为空时改变行为，对已有动态形状路径没有影响，兼容性良好。  
4. **测试覆盖**：新增的测试依赖 Torch >=2.10，CI 环境需确保相应版本可用；若还有更旧的最低支持版本，考虑在 CI 中加入条件跳过或提供兼容实现。

- **引用管理**：`self._views` 在 `cleanup` 前已 `clear`，但 `self._base` 仍持有 `memoryview`；确保 `torch.frombuffer` 的底层对象在 `cleanup` 前已释放，以免 `mmap.close` 报 `BufferError`。  
- **pin‑memory 条件**：`pin_mmap_region` 只在 CUDA 可用时调用，建议在 `is_pin_memory_available()` 为 `False` 时仍能正常工作（保持非‑pinned 路径）。  
- **性能基准**：新增共享 mmap 与逐块 `torch.zeros` 对比的基准测试可以帮助量化加速幅度，尤其在多 worker 场景。  
- **文档/类型**：为 `SharedOffloadRegion` 增加类级别 docstring，明确 `rank=None` 的含义；在公共 API（`create_next_view`、`pin_mmap_region`）加上返回类型与异常说明。  

总体来说，此次改动将 KV‑offload 的内存布局统一为共享 mmap，显著降低 CPU‑GPU 拷贝的分配开销并提供跨进程零拷贝潜力。若补齐异常清理与跨平台适配，代码将更加稳健且易于维护。
