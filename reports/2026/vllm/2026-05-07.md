#### 🔴 高重要度变更 (2)

### [CPU] Bump up to the latest CPU kernels (#41924)
**SHA**: `b3945cc` | 🔗 [查看提交](https://github.com/vllm-project/vllm/commit/b3945cc3160119111927816841950715c5410708)

## 1.  Overview  

The change set introduces a **complete rewrite of the CPU side of the “SG‑LLaMA” (sgl) kernels** that power vLLM’s mixed‑precision and MoE (Mixture‑of‑Experts) execution.  The most visible results are:

| Area | What changed |
|------|--------------|
| **Kernel core** | New `tinygemm_*` implementations for BF16/FP16/FP8/INT8/INT4, with extensive use of AVX‑512, VNNI, AMX and BRGEMM. |
| **Quantisation** | Support for three 4‑bit methods (`INT8_W8A8`, `FP8_W8A16`, `INT4_W4A8`) and a new enum `CPUQuantMethod`.  Additional enums for the MoE‐specific “AWQ / GPTQ” distinction (`CPUQuantAlgo`). |
| **MoE** | New fused‑MoE kernels for FP8 and INT4 (`fused_experts_fp8_kernel_impl`, `fused_experts_int4_w4a8_kernel_impl`).  The previous `fused_experts_cpu` now dispatches through a thin wrapper that forwards the new arguments. |
| **APIs** | New custom‑op signatures in **torch‑ops registration** and Python‑level bindings (`torch.ops._C.*`).  Old scalar paths have been deliberately disabled. |
| **Utility code** | Refactored vector helpers (`vec.h`, `vec_pack.h`) to expose packed‑VNNI layouts, AVX‑512‑based `transpose_*` and `pack_vnni*` helpers. |
| **Testing** | New unit‑tests for FP8‑MoE, INT4‑MoE, and the AWQ‑to‑INT8 weight‑packing path. |
| **Build / Lint** | `pyproject.toml` excludes the new CPU kernel source tree from the flake/typos infrastructure. |
| **Python layer** | `vllm/_custom_ops.py` now exposes `CPUQuantMethod`/`CPUQuantAlgo` and the new `convert_weight_packed_scale_zp`/`int4_scaled_mm_cpu` helpers.  High‑level MoE layers (`CPUFusedMoE`, `CPUFusedMoEExperts`) have been updated to use them. |

Below we dissect the impact of these changes from **architecture**, **performance**, **security**, **maintainability**, **compatibility** and **testing** viewpoints.

## 2. Architectural Impact  

### 2.1. New Quantisation Stack  

| Component | Role |
|-----------|------|
| `CPUQuantMethod` | Runtime selector for the *type of* weight quantisation used in a MoE layer (UNQUANT, INT8_W8A8, FP8_W8A16, INT4_W4A8). |
| `CPUQuantAlgo` | Distinguishes **AWQ** vs **GPTQ** when converting INT4 weights (different packing/layout). |
| `convert_weight_packed_scale_zp` | Central entry point that receives *raw* 4‑bit weight/zp matrices (int32) and returns three tensors: packed weight (VNNI), packed zero‑points, and packed scales, **already permuted** for the kernel layout. |
| `int4_scaled_mm_cpu` | High‑level BF16/FP16 *mat‑mul* that internally quantises the activation, calls the VNNI GEMM (`tinygemm_kernel`), then de‑quantises the result. |
| `fused_experts_*` | MoE‑specific kernels that combine **two GEMM passes** (gate → expert‑1, expert‑2) with SiLU‑and‑multiply and final reduction.  The FP8 and INT4 versions have their own dedicated implementations. |

The architecture now **decouples** the three phases that existed previously (weight packing, activation quantisation, GEMM) and makes them reusable across MoE and ordinary linear layers.

### 2.2. Kernel Layout & Dataflow  

* **VNNI/AMX‑oriented packing** – The new `pack_vnni`, `pack_vnni2` helpers produce the exact layout expected by the GEMM kernels (e.g. `[K, N/2, 2]` for BF16).  This layout also matches the expectations of Intel AMX `brgemm`.  
* **BRGEMM fallback** – The kernels check `can_use_brgemm<scalar_t>(M)` and, if true, delegate the heavy inner‑product to `at::native::cpublas::brgemm`.  When BRGEMM isn’t available (e.g. small `M` < 5) a hand‑rolled AVX‑512 VNNI kernel runs.  
* **Two‑stage MoE flow** –  
  1. **Gate** (`hidden_states @ w1`) → intermediate buffer (`ic0`).  
  2. **SiLU + Mul** (`ic0` → `ic1`).  
  3. **Second GEMM** (`ic1 @ w2`) → per‑expert output (`ic2`).  
  4. **Top‑k accumulation** (`ic2` → final hidden state).  

The code now **parallelises on the “token‑block × column‑block” grid** (`parallel_2d`) rather than the obsolete `parallel_for`/`data_index_init` pair, improving load‑balancing on many‑core CPUs.

### 2.3. API surface  

| Old signature (simplified) | New signature (vLLM Python) |
|-----------------------------|----------------------------|
| `torch.ops._C.fused_experts_cpu(hidden_states, w1, w2, topk_weights, topk_ids, inplace, use_int8_w8a8, use_fp8_w8a16, w1_scale, w2_scale, block_size, a1_scale, a2_scale, is_vnni)` | `fused_experts_cpu(hidden_states, w1, w2, topk_weights, topk_ids, inplace, moe_comp_method, w1_scale, w2_scale, w1_zero, w2_zero, block_size, is_vnni)` |
| `torch.ops._C.int4_scaled_mm_cpu(x, w, w_zeros, w_scales, bias)` | `int4_scaled_mm_cpu(x, w, w_zeros, w_scales, bias)` (exposed directly from Python) |
| `torch.ops._C.convert_weight_packed_scale_zp(qweight, qzeros, scales)` | `convert_weight_packed_scale_zp(qweight, qzeros, scales, quant_method_4bit)` |

The change **removes unused arguments** (`a1_scale`, `a2_scale`) and **adds explicit enum parameters** for dispatch, thereby making the API self‑documenting.

### 🔍 技术洞察  

| 维度 | 影响说明 |
|------|----------|
| **架构影响** | - 引入 **CPU 自定义算子**（`cpu_gdn_attention_core`、`causal_conv1d_*`、`recurrent_gated_delta_rule`）并通过 `direct_register_custom_op` 注册，形成与 GPU/FlashInfer 统一的注意力后端抽象。<br>- `GDNLinearAttn` 在 `forward` 时根据平台动态选择 `forward_cpu / forward_cuda / forward_xpu`，实现一次代码路径复用。<br>- `CPUAttentionBackend.get_supported_kernel_block_sizes` 返回 `MultipleOf(16)`，迫使混合模型在 CPU 上使用 16‑倍块大小，以匹配新的 GDN 实现。 |
| **性能影响** | - 对 **Qwen 3.5/3.6** 在纯 CPU 环境下可直接使用 GDN 注意力，预计在 **prefill** 场景中比传统 `CPU_ATTN`（基于普通 GEMM）快 1.5‑2×（取决于序列长度与 head‑size），因为 GDN 把递归状态压缩到低维 SSM，减少了 KV 长度的线性增长。<br>- **decode** 仍采用逐 token 更新，算子内部使用 `torch.ops.vllm.cpu_gdn_attention_core`，在没有 JIT 编译的情况下可能略慢于 CUDA 实现；但因为算子是纯 Python+Torch 实现，仍保持可接受的毫秒级延迟。<br>- 新增的 `register_cpu_gdn_attention_ops` 只在 CPU 进程首次调用时触发，对启动时延的影响可忽略。 |
| **安全考虑** | - 代码路径新增 **自定义 Torch 运营**，若实现中出现未检查的张量尺寸或 dtype 转换，可能导致 **内存越界** 或 **数值溢出**（尤其在 `torch.linalg.solve_triangular`）。<br>- 目前没有引入外部依赖或文件系统操作，安全面风险有限。<br>- CI 中加入了 `timeout 600` 与 `set -e`，防止卡死导致资源泄露。 |
| **可维护性** | - 将 GDN 相关逻辑全部集中在 `vllm/model_executor/layers/mamba/ops/cpu/`，代码结构清晰，易于后续升级。<br>- 为 `Linear.create_weights` 添加了维度检查，避免对非 Linear 层误调 `dispatch_cpu_unquantized_gemm`，提升代码健壮性。<br>- 文档同步更新，使用 `%16` 表示块大小必须为 16 的倍数，降低误解。 |

### ⚠️ 潜在风险  

1. **数值一致性**  
   - CPU 与 GPU 上的 GDN 实现采用不同的数值路径（Python vs CUDA kernels），可能导致 **输出略有差异**，在对齐基准测试时需要容忍一定的误差或增加额外的校准步骤。  

2. **内存泄漏 / 越界**  
   - `cpu_gdn_attention_core` 直接操作 KV 缓存张量，若 `state_indices_tensor`、`query_start_loc` 与实际 token 数不匹配，会触发 **索引错误** 或 **未对齐的张量写入**。  

3. **性能回退**  
   - 对不支持 GDN 的模型（如非 Qwen 3.5/3.6）仍会走传统 CPU 注意力；若块大小自动对齐为 16，但模型实际需要更大块，可能导致 **吞吐下降**。  

4. **平台检测误差**  
   - `current_platform.is_cpu()`、`.is_xpu()` 判定在混合环境（CPU+GPU）下可能出现 **误选**，导致 XPU/GPU 仍尝试加载 CPU ops，引发 `RuntimeError: unknown custom op`。  

5. **CI 稳定性**  
   - 新增的 smoke‑test 需要 Docker 镜像中包含对应的模型权重。如果镜像更新或网络不通，CI 可能出现 **假阳性失败**。

### tokenizer: Add fastokens support (#41741)
**SHA**: `2a16ece` | 🔗 [查看提交](https://github.com/vllm-project/vllm/commit/2a16ece2d342c0c154a4949ad317b521f8c04ec4)

**🎯 变更类型**：功能增强（新增 fastokens tokenizer 模式）  
**⚡ 重要程度**：🟡 中  

**📋 变更摘要**  
- 在 `ModelConfig.TokenizerMode` 中加入 `"fastokens"`，并在文档中说明使用方法与适用场景。  
- 新增 `vllm/tokenizers/fastokens.py`：在加载 Hugging‑Face **fast tokenizer** 前调用 `fastokens.patch_transformers()`，实现 Rust BPE 后端替换。  
- 注册表 `registry.py` 增加 `"fastokens"` 对应的实现类。  
- `detokenizer.py` 改为通过 `tokenizers.decoders.DecodeStream` 动态获取类，确保在 `fastokens` 打补丁后能使用替换后的实现。  
- 文档更新，提示用户需自行安装 `fastokens` 包，否则会抛出明确的 `ImportError`。  

**🎯 影响范围**  
- **vllm/config/model.py**、**vllm/tokenizers/**（registry、fastokens、hf）以及 **vllm/v1/engine/detokenizer.py**。  
- 依赖 `fastokens` 的用户需要在运行环境中额外安装该包。  
- 只对 BPE 类 tokenizer（如 Qwen、Llama、DeepSeek）有效，其他自定义 tokenizer 仍走原路径。  

**💡 关注建议**  

1. **兼容性检查**：在 `FastokensTokenizer.from_pretrained` 里可以加入对 tokenizer 类型的检测（`isinstance(tokenizer, PreTrainedTokenizerFast)`），在不兼容时给出友好警告或回退到 `hf`。  
2. **导入顺序**：虽然已改为 `tokenizers.decoders.DecodeStream` 动态查找，但最好在 `vllm/tokenizers/fastokens.py` 中调用 `patch_transformers` 前显式 `import tokenizers.decoders`，防止某些环境下模块提前加载导致补丁失效。  
3. **可选依赖**：在 `setup.cfg/pyproject.toml` 中声明 `fastokens` 为 **optional dependency**，并在 CI 中加入“skip if fastokens not installed” 的测试路径，防止因缺失该包导致全套测试失败。  
4. **单元测试**：添加针对 `TokenizerMode="fastokens"` 的快速测试：  
   - 正常加载（已安装 fastokens） → 检查返回对象类型。  
   - 未安装 fastokens → 捕获并验证 `ImportError` 信息。  
   - 流式解码路径：验证 `Detokenizer` 在 `fastokens` 模式下仍能正常产生 `DecodeStream`。  
5. **文档同步**：在 `README` 或快速入门章节补充 `pip install fastokens` 示例，避免用户在首次尝试时遇到“module not found”。  

综上，此次改动为性能优化提供了可选的 Rust‑级别 tokenizer，影响范围集中在 tokenizer 加载与 detokenizer 实现。只要补丁顺序、依赖声明和兼容性检测做好，就能平滑落地并为大量长文本前缀处理场景带来明显加速。
