### [EPLB] Add alternative communication for EPLB weight exchange (#33176)
**SHA**: `abdbb68` | 🔗 [查看提交](https://github.com/vllm-project/vllm/commit/abdbb683861a2586817ae6836bad308c09919d5c)

**🎯 变更类型**：功能增强 / 重构 / 架构变更  
**⚡ 重要程度**：🔴高  

**📋 变更摘要**  
本次提交为 Expert‑Parallel Load‑Balancing（EPLB）引入可插拔的通信层，新增 `EplbCommunicator` 抽象并实现三种后端（torch‑nccl、torch‑gloo、pynccl），在 `ParallelConfig` 中加入 `communicator` 配置项并实现自动选型。相应地，所有 EPLB 相关的权重搬迁与重排逻辑改为通过该抽象统一调度；测试、异步 worker、Elastic EP 初始化等全部迁移到新接口，并加入跨 rank 验证与完整进程组销毁。  

**🎯 影响范围**  
- `vllm/config/parallel.py`（EPLB 配置扩展）  
- `vllm/distributed/eplb/eplb_communicator.py`（全新实现）  
- `vllm/distributed/eplb/*`（rebalance、async worker、state、utils）  
- `vllm/distributed/elastic_ep/elastic_execute.py`（Elastic EP 启动时创建 communicator）  
- `vllm/distributed/parallel_state.py`（EPLB 组初始化）  
- 测试目录 `tests/distributed/*`（加入多后端参数、跨 rank 验证、进程组清理）  

---

### 🔍 技术洞察  

| 维度 | 影响说明 |
|------|----------|
| **架构影响** | 1. **抽象层**：新增 `EplbCommunicator` 抽象，将权重搬迁的点对点通信从硬编码的 `torch.distributed.isend/irecv` 解耦。<br>2. **后端选择**：通过 `ParallelConfig.eplb_config.communicator` 自动决定使用 NCCL、Gloo（CPU‑staged）或 PyNccl。<br>3. **Elastic EP**：在弹性 EP（stateless）模式下强制使用 `pynccl`，并在 `elastic_execute` 中统一创建 communicator。<br>4. **统一入口**：所有 EPLB 相关入口（`move_to_buffer`、`transfer_layer`、`rearrange_expert_weights_inplace`）现在接受 `communicator` 参数，避免在业务代码中散落大量 `torch.distributed` 调用。 |
| **性能影响** | - **同步 NCCL**：保持原有最快的设备‑间通信路径。<br>- **异步 Gloo**：在 `use_async=True` 时默认使用 Gloo，规避 NCCL 多线程冲突，已在 `override_envs_for_eplb` 中仅在 NCCL‑based communicator 才开启 `NCCL_MAX_CTAS` 限制，避免不必要的性能削减。<br>- **PyNccl**：通过低层 ncclSend/ncclRecv 可进一步降低延迟，尤其在弹性 EP（stateless）场景下。<br>- **流管理**：`set_stream` 统一把 CUDA 流绑定到 communicator，消除原有隐式 stream 切换，提升异步路径的并发度。 |
| **安全考虑** | - 新增的 `communicator` 参数是 **白名单**（Literal）并在配置验证阶段统一检查，未引入任意代码执行风险。<br>- 在 `create_eplb_communicator` 中对设备类型、dtype 支持、pynccl 可用性进行显式校验，防止不可达后端导致进程卡死。<br>- 添加 `atexit` 清理 `torch.distributed.destroy_process_group`，防止残留进程组导致后续测试或服务启动出现 “process group already initialized” 的异常。 |
| **可维护性** | - 代码路径统一后，后期添加新通信实现只需实现抽象类即可，降低改动范围。<br>- 测试框架加入了 **跨 rank 验证**（`assert_verification_synced`），一旦有单点失败即刻可定位，提升回归可靠性。<br>- 通过 `skip_queue` 处理分布式测试的部分跳过情况，避免因单节点缺资源导致全部测试失败。 |

### ⚠️ 潜在风险  

1. **后端兼容性**  
   - `pynccl` 仅在 CUDA 环境且 dtype 被 NCCL 支持时可用；在 CPU‑only或 dtype 不在白名单（如 `torch.int64`）时会抛异常。  
   - 自动选型规则在某些自定义配置（例如 `use_async=False` 同时手动指定 `torch_gloo`）可能导致性能退化或不必要的 CPU‑to‑GPU 拷贝。  

2. **进程组生命周期**  
   - `atexit` 注册的 `destroy_process_group` 可能在异常终止或多轮测试中被多次调用，若有未完成的通信仍在进行，可能出现 “process group not initialized” 或 “transport endpoint is not connected”。  
   - 在弹性 EP（stateless）模式下，`create_eplb_communicator` 强制使用 `pynccl`；若底层库缺失或版本不兼容，整个 Elastic EP 将启动失败。  

3. **流同步**  
   - `EplbCommunicator.set_stream` 只在 async worker 中显式调用，若用户在自定义代码里自行创建 communicator 而忘记设置流，可能导致复制在错误的 CUDA 流上执行，引起数据竞争或不可预期的延迟。  

4. **测试资源需求**  
   - 新增的参数化测试会在 CI 中分别跑三种后端，需要对应的硬件（GPU）和软件（PyNccl）支持；在没有 PyNccl 的机器上会出现报错，导致 CI 失效。

### 💡 关注建议  

1. **文档与使用指南**  
   - 在项目 README 与 `vllm/config/parallel.py` 注释中补充 **communicator 选型矩阵**、何时使用 `torch_gloo`（推荐 async）以及 `pynccl` 的前置条件（CUDA + NCCL‑compatible dtype）。  
   - 为 Elastic EP 文档说明 “stateless 必须使用 pynccl” 的原因，帮助用户在未安装 PyNccl 时及时发现配置问题。  

2. **健壮的后端检测**  
   - 在 `create_eplb_communicator` 中加入 **环境变量或显式 API**（如 `VLLM_EPLB_BACKEND=auto|torch_nccl|torch_gloo|pynccl`）的提示信息，便于调试。  
   - 对 `torch_gloo` 添加 **fallback**：如果 Gloo 二进制缺失或 `torch.distributed.is_gloo_available()` 为 false，自动回退到 NCCL 并给出警告。  

3. **进程组清理策略**  
   - 考虑在 `distributed_run` 中统一调用 `torch.distributed.barrier()` 与 `destroy_process_group`，避免依赖 `atexit` 的不确定执行顺序。  
   - 在异常路径（如子进程抛出）中确保 `destroy_process_group` 仍会被触发，以防残留进程

### [Frontend][3/n] Improve pooling entrypoints | scoring. (#28631)
**SHA**: `d9d21eb` | 🔗 [查看提交](https://github.com/vllm-project/vllm/commit/d9d21eb8e35fa356ebb2a1ffa94da4306905ff0e)

1. **并发/异步安全**：`should_record_tensor` 为全局标量，多个层共享同一块显存，若在异步模式下出现跨 step 的并发写入，需要确认 `torch.fill_` 在同步点前已完成。建议在 async‑mode 里显式同步或在文档中提醒用户 `step_interval` 与 `window_size` 配置需满足 `step_interval >= window_size`。  
2. **Triton kernel 边界**：kernel 对非法 `expert_id` 采用了 0/‑1 填充并做了掩码，确保 `out_size` 与 `logical_to_physical_map` 对齐。生产环境下若出现 `-1`（无效 replica）或 `topk_ids` 超出范围，仍会写入 `out`，建议在调用前加入断言或日志，以便快速定位数据错误。  
3. **兼容性**：CPU fallback 仍返回原始 `topk_ids`，但未对 `record_enabled` 做任何检查；若在 CPU 环境下误打开 `enable_eplb`，会失去统计功能。可以在 fallback 中给出警告。  
4. **配置验证**：`EPLBConfig` 改为 Pydantic `Field`，但其他代码仍直接访问属性，确保所有实例化路径已使用 `EPLBConfig(**kwargs)`。  
5. **测试覆盖**：新增的映射测试覆盖了无冗余、部分冗余、全冗余及不均匀 replica 场景，但未覆盖 `record_enabled=False` 且 `topk_ids` 包含 `-1`（表示未选中专家）的情况，建议再补充此类 edge‑case。  

**总体评价**  
此次改动显著提升了 EPLB 的前置映射与负载统计效率，代码结构更清晰（统一的 `should_record_tensor` 控制），并通过 Triton 实现了 GPU‑native 的原子计数。只要注意上述并发与边界检查，风险可控。建议在 CI 中加入不同 `step_interval` 与 `window_size` 组合的性能基准，以验证预期加速。
