### [Model] Support Hy3 preview (#40681)
**SHA**: `d0009dd` | 🔗 [查看提交](https://github.com/vllm-project/vllm/commit/d0009ddb0b96e95bcfae6038e9b8673bd2263058)

**🎯 变更类型**：功能增强（新增 HYV3（预览）模型及其相关子系统）  
**⚡ 重要程度**：🔴 高  

**📋 变更摘要**  
- 在 `vllm` 中完整加入了 HYV3（腾讯 Hy3‑preview）模型的支持，包括主模型、MTP（多 token 预测）模型、对应的配置类、模型注册、权重映射、推理、工具调用以及推理解析器。  
- 更新文档、模型注册表、推理/工具解析器注册表以及 `speculative` 配置，使 HYV3 能够与现有的 LoRA、PP、Moe、MTP、以及推理/工具解析框架无缝集成。  
- 新增大量单元测试覆盖模型注册、HYV3 推理分割 (`<think>...</think>`) 以及工具调用标记 (`<tool_calls>...</tool_calls>`) 的解析与流式处理。  

**🎯 影响范围**  
- `vllm/model_executor/models/`（`hy_v3.py`、`hy_v3_mtp.py`）  
- `vllm/transformers_utils/configs/`（`hy_v3.py`）  
- `vllm/config/speculative.py`（新增 `hy_v3_mtp` 类型）  
- `vllm/model_executor/weight_utils.py`（新增 HYV3 KV‑scale 重命名规则）  
- `vllm/reasoning/`（新增 `hy_v3_reasoning_parser.py` 并在 `__init__` 中注册）  
- `vllm/tool_parsers/`（新增 `hy_v3_tool_parser.py` 并在 `__init__` 中注册）  
- `vllm/model_executor/models/registry.py`（模型与 MTP 注册）  
- 文档 `docs/models/supported_models.md`、示例测试 `tests/models/registry.py`、推理/工具解析测试套件。

### ⚠️ 潜在风险  

1. **权重映射正则冲突**  
   - 新增的 `\.self_attn\.q\.scale$` 与其他模型的同名字段可能产生误匹配，导致错误的 scale 加载。  
2. **流式工具调用的 JSON 增量构造**  
   - 手动拼接 JSON 片段（尤其是字符串值的转义）如果出现边界错误（如忘记转义 `"`），可能生成非法 JSON，导致上层服务解析失败。  
3. **内存峰值**  
   - `HYV3MoEFused` 在专家并行（EP）下会为每个 GPU 分配 `n_local_physical_experts` 权重，若 EP 配置不当（如专家数远大于 TP），可能导致 OOM。  
4. **依赖 `regex`**  
   - 该库在某些平台的二进制 wheels 可能缺失或版本不兼容，需要在 `setup.cfg/requirements.txt` 中明确声明。  
5. **MTP 与普通解码的交叉使用**  
   - `HYV3MTP` 依赖 `num_nextn_predict_layers` 参数；若用户在配置中错误地同时开启普通解码器与 MTP，可能出现权重重复或层索引冲突。

### [Bugfix] Make Attention Backend Auto-Selection Batch-Invariance-Aware (#40193)

**⚡ 重要程度**：🟡 中  

**📋 变更摘要**  
1. `init_batch_invariance()` 不再接受显式的 `attention_backend` 参数，改为在环境变量 `VLLM_BATCH_INVARIANT` 为真时自动使用当前选中的注意力后端。  
2. 为所有注意力后端 (FlashAttn、TRITON、MLA 系列) 新增 `supports_batch_invariance()` 接口，并在 `AttentionSelectorConfig` 与 `AttentionSelector` 中加入 `use_batch_invariant` 标记，确保在 batch‑invariant 模式下仅使用支持的后端。  
3. 移除旧的 `override_envs_for_invariance(attention_backend)` 检查逻辑，改为统一在 `init_batch_invariance()` 中调用 `override_envs_for_invariance()`（不再校验后端）。  

**🎯 影响范围**  
- `vllm/model_executor/layers/batch_invariant.py`（核心批次不变性初始化）  
- 所有注意力后端实现（`vllm/v1/attention/backends/*`）  
- 注意力选择器 (`selector.py`) 与 GPU worker 初始化 (`gpu_worker.py`)  
- 示例脚本 `examples/rl/rlhf_async_new_apis.py`  

**💡 关注建议**  
- **向后兼容**：`init_batch_invariance` 的签名已变更，旧代码仍传参会报错，建议在 `batch_invariant.py` 添加兼容包装或在文档中标明已弃用。  
- **错误提示**：目前只有 Mamba 后端在 `gpu_worker` 处检测不支持 batch‑invariance，其他后端若不实现 `supports_batch_invariance` 将在运行时产生 `AttributeError`。建议在 `selector` 中统一校验并给出统一异常信息。  
- **文档/示例**：示例已改为不手动传后端，需同步更新 README/使用说明，明确 `VLLM_BATCH_INVARIANT` 与 `--attention-backend` 的关系。  
- **测试覆盖**：补充在 batch‑invariant 环境下切换不同后端的单元测试，确保不支持的后端会抛出预期错误。  

整体改动让 VLLM 在 batch‑invariant 模式下可以自动选取兼容后端，提升易用性；但需注意签名变更的兼容性以及统一的后端支持校验。

2. **异常路径**：`raw_decode` 只能解析第一个 JSON 值，若 `stringified_tool_calls` 含有前置噪声（如 `junk [{...}]`）仍会抛 `JSONDecodeError`，随后进入正则回退。确认正则仍能覆盖所有可能的“前置噪声 + 有效 JSON” 场景。  
3. **缺失 arguments**：已加入 `dict.get(..., {})`，但对应的 `FunctionCall.arguments` 默认值仍是字符串 `"{}`"（在后续 `tool_call.get("arguments", "{}")` 中），保持统一。若业务侧需要区分“空参数” 与 “未提供”，可在 `FunctionCall` 的模型层加注释。  
4. **测试覆盖**：目前仅增加了 `trailing_data_after_json` 用例，建议再补充：  
   - 前置噪声 + 合法 JSON（已覆盖正则回退）  
   - `arguments` 为 `null`、列表或嵌套对象的情况  
   - 大型工具调用数组，确保 `raw_decode` 只返回第一段而不截断中间元素。  
5. **文档更新**：在 `tool_parsers` 章节说明 “模型可能会在 `[TOOL_CALLS]` 后输出额外字符”，以及本解析器已实现的容错策略。  

总体来说，此次改动修复了实际使用中常见的输出噪声问题，提升了 Mistral pre‑v11 的稳定性。但需留意上述兼容性细节，避免因二次序列化导致意外的参数格式。
