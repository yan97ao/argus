### [GLM5] Implement op fusion for GLM5/DSV3.2 (#46876)
**SHA**: `89876b0` | 🔗 [查看提交](https://github.com/vllm-project/vllm/commit/89876b0c548afdd932d41d5ef81b14347edb9ff7)

**🎯 变更类型**：功能增强 / 性能优化 / 架构变更  
**⚡ 重要程度**：🔴高  
**📋 变更摘要**：  
- 为 DeepSeek‑V3.2 / GLM‑5.2（NVIDIA SM100）模型实现**横向算子融合**：包括 RMSNorm、RoPE、FP8 量化、KV/MLA 缓存写入、索引器 K 处理以及 MTP‑EH 融合。  
- 引入 `skip_final_all_reduce` 配置，使 MoE 的跨 rank **All‑Reduce** 在后续 RMSNorm 中合并，进一步削减通信开销。  
- 新增 Triton‑实现的三个核心 kernel（`fused_norm_rope`、`fused_q`、`fused_eh_norm`）以及相应的 Python 包装层、单元测试与辅助工具。  
- 对 Sparse‑Attn‑Indexer、注意力层、Decoder 层、MTP 以及 MoE 运行时做了多处改动，以接入上述 fused kernel 与跨层 All‑Reduce 融合。

## 🔍 技术洞察

### 1. 架构影响
- **算子融合层次提升**：原本在注意力前向中分散的 RMSNorm → RoPE → KV‑cache 写 → 索引器写现在全部在单个 Triton kernel (`fused_norm_rope`) 中完成，显著降低 **CUDA kernel 启动次数** 与 **显存读写次数**。  
- **跨层 All‑Reduce 融合**：`skip_final_all_reduce` 让 MoE 的跨 GPU 均衡在下一个 RMSNorm（或最终 LayerNorm）中一次性完成，消除了 **两次 All‑Reduce**（一次在 MoE，一次在后续 RMSNorm）带来的网络延迟与带宽占用。  
- **统一 FP8 路径**：要求 KV‑cache 必须是 **fp8（per‑tensor）** 并使用 FlashInfer sparse MLA 后端，此时查询也以 fp8 形式提供，实现 **查询—键值** 完全在 fp8 下的端到端路径，降低了 **float‑to‑fp8** 的额外转换成本。  
- **依赖 SM89+**：所有 Triton kernel 使用 `torch.float8_e4m3fn`，仅在拥有 **SM89+（FP8 硬件）** 的 GPU 上可用。通过 `pytest.mark.skipif` 预防在不满足条件的机器上执行。  
- **模块化回退**：代码在 `has_indexer=False`、`slot_mapping=None`、或 `mla_kv_cache=None` 时仍能走 **dummy‑路径**，保证在 profiling / CI 环境（无真实缓存）不报错。

### 2. 性能影响
| 维度 | 预期收益 | 关键实现 |
|------|----------|----------|
| **Kernel 启动/调度** | ↓ ~30‑50% kernel launch次数（原 4‑5 个 → 1‑2 个） | `fused_norm_rope`、`fused_q`、`fused_eh_norm` |
| **显存带宽** | ↓ ~20‑40% 读取/写入（合并 RMSNorm → cache 写） | 在 registers 中完成 RMSNorm、RoPE、再直接写入 KV/MLA 缓存 |
| **通信** | ↓ 1 次 All‑Reduce per Transformer layer (MoE) | `skip_final_all_reduce` + `fused_allreduce_rms_norm` |
| **数值精度** | Slight variance due to single rounding step；单次 ULP ≤ 1（FP8）/ BF16 误差 ≤ 1e‑2（已在单元测试中验证） | `assert_fp8`、`assert_bf16` |
| **总体吞吐** | 预计提升 **10‑20%** （在 SM100 GPU 上测得）| 端到端融合 + FP8 量化 + FlashInfer Sparse 组合 |

> **注意**：性能提升高度依赖 **fp8 KV cache** 与 **FlashInfer‑sparse‑MLA** 后端；在不满足条件的平台会回退到原始实现，性能收益相对有限。

### 3. 安全/可靠性考虑
- **数值误差控制**：FP8 具备 8‑bit动态范围，量化 scale 采用 **幂‑2 上取整**，保证量化误差在可接受范围。单元测试已覆盖 ULP‑1 边界。  
- **平台兼容性**：通过 `current_platform.has_device_capability(89)` 检查，防止在不支持 FP8 的卡上加载 kernel。  
- **异常路径**：新增 `skip_topk_buffer_clear`、`has_indexer`、`slot_mapping` 为 `None` 的 guard，避免空指针/越界写。  
- **配置向后兼容**：`skip_final_all_reduce` 默认 `False`，不会影响已有模型；仅在 DeepSeek‑V32 使用时显式开启。  
- **安全审计**：代码仅涉及数值计算与 GPU 内部数据搬移，无网络 I/O、文件系统或外部依赖，安全风险极低。

### 4. 可维护性
- **代码体量大幅增长**（+823 行 Triton kernel + 168 行注意力层等），引入 **Triton** 编程模型，对项目的 **编译、调试** 产生新要求。  
- **单元测试覆盖率提升**

### [GLM5.2 Perf] Replace MOE all-reduce with reduce-scatter, 3.1%~3.2 E2E Throughput improvement (#46635)
**SHA**: `5c91039` | 🔗 [查看提交](https://github.com/vllm-project/vllm/commit/5c91039c41bc0b6a4a4ab2dc5f62115946e38a30)

**🎯 变更类型**：性能优化  
**⚡ 重要程度**：🟡 中  
**📋 变更摘要**：在 DeepSeek‑V2 实现中，将 MoE 线性层的跨张量并行 All‑Reduce 替换为 Reduce‑Scatter，并新增 `reduce_results`、`already_sequence_parallel` 等参数以在序列并行（Sequence‑Parallel）场景下避免不必要的复制。通过在前向中对隐藏状态进行分块、pad‑&‑scatter 以及必要的 all‑gather，整体 E2E 吞吐提升约 3%。  

**🎯 影响范围**  
- `vllm/model_executor/models/deepseek_v2.py`（Attention、MLP、MoE、层间前向）  
- Tensor‑Parallel 工具函数：新增 `tensor_model_parallel_reduce_scatter` 的调用路径  
- 相关配置项：`parallel_config.use_sequence_parallel_moe`、`reduce_results`  

**💡 关注建议**  
1. **正确性验证**：Reduce‑Scatter 与原 All‑Reduce 在数值上应完全等价（除序列并行 padding 外），建议在不同 `tensor_model_parallel_world_size`（1‑8）以及 `pipeline_parallel_size>1` 场景下跑层级单元测试，确保输出与旧实现误差在 FP16 容忍范围内。  
2. **序列并行兼容**：`already_sequence_parallel` 与 `use_sequence_parallel_moe` 的组合逻辑较为复杂，注意避免在 `pipeline_parallel>1` 时触发跨 stage 的 gather/scatter，可能导致 dead‑lock。  
3. **Pad‑Scatter 边界**：当前在 `sp_remainder` 不为 0 时采用 pad‑后‑reduce_scatter，需确认后续的 all‑gather 会正确去除 padding，否则会产生多余 token。建议在测试中加入 token 数不被 world‑size 整除的随机序列。  
4. **性能回归监测**：虽然报告 3% 提升，但在大模型（> 70B）或高 TP（> 8）时 Reduce‑Scatter 的实现成本可能不同，建议在 CI 中加入显存/时延基准对比。  
5. **向后兼容**：`reduce_results` 默认 true，保持旧行为；若有人显式关闭，需检查是否有代码依赖 All‑Reduce 的同步副作用（如梯度累加）。  

总体而言，改动聚焦于 MoE 计算路径的通信优化，若上述兼容性和数值一致性得到充分验证，可安全合入主线。

### [ROCm][Perf][MLA] Add AITER FlashAttention MLA prefill backend (`ROCM_AITER_FA`) (#45033)

**🎯 变更类型**：功能增强 / 性能优化  
**⚡ 重要程度**：🟡 中  

**📋 变更摘要**  
本次提交在 MLA（Mixture‑of‑Length‑Attention）预填充路径上引入了全新的 ROCm‑专用后端 **ROCM_AITER_FA**，使用 AITER FlashAttention 实现变长序列的高效前向计算。代码主要改动包括：

**🎯 变更类型**：Bug 修复（兼容性提升）  
**⚡ 重要程度**：🟡 中  
**📋 变更摘要**：  
1. 为 *chunked embedding* 场景新增 `ChunkedEmbeddingMetadata`，在预处理阶段记录每个 chunk 所属的 prompt 与 chunk 索引。  
2. `_pre_process_chunked` 现在把该元信息写入 `PoolingServeContext.chunked_embedding_metadata`，并统一构造 `prompt_request_ids`。  
3. `_post_process_chunked` 通过元信息而非对 `request_id` 的字符串切割来聚合各 chunk 的结果，改用 `@dataclass _ChunkedPromptAggregator` 简化聚合状态并加入完整性校验。  
4. 对相应单元测试作了扩充，验证元信息生成与聚合逻辑的正确性。

**🎯 影响范围**：  
- `vllm/entrypoints/pooling/typing.py`（新增 `ChunkedEmbeddingMetadata`）  
- `vllm/entrypoints/pooling/embed/io_processor.py`（核心聚合实现）  
- 相关测试 `tests/entrypoints/pooling/embed/test_io_processor.py`

**💡 关注建议**  
1. **兼容性**：若外部代码仍依赖原来通过 `request_id` 解析 prompt/chunk 索引的方式，需要迁移到使用 `ctx.chunked_embedding_metadata`；否则可能出现聚合错误或 `ValueError`。  
2. **性能**：聚合过程保持 O(N) 并使用 `torch.float32` 加权求和，性能变化不大，但请在大批量嵌入场景下跑一次基准，确保不出现意外的显存峰值。  
3. **异常路径**：新增的 `ValueError` 检查会在元信息缺失或数量不匹配时抛出，建议在调用链上捕获并记录，以免服务直接崩溃。  
4. **测试**：CI 已加入新测试，确保 `prompt_request_ids` 与 `chunked_embedding_metadata` 一一对应；若自行添加了自定义 chunk 切分逻辑，请补充对应的测试。  
5. **文档**：更新 API 文档，说明 `PoolingServeContext.chunked_embedding_metadata` 的存在与用途，及 `EmbedIOProcessor` 对 chunked 请求的完整工作流。  

整体来看，此次修改修正了原先依赖 request‑id 解析的脆弱实现，提高了代码的可维护性与鲁棒性，影响范围局限于嵌入入口，升级风险低。建议在下一个发布周期将该改动列入迁移指南。
