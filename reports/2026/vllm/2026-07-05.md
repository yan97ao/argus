### [MRV2] Enable mm prefix bidi attention support on MRV2 (#46942)
**SHA**: `cc1d020` | 🔗 [查看提交](https://github.com/vllm-project/vllm/commit/cc1d020d01949d11b7ef70dabb0eb196b3f39f53)

**🎯 变更类型**：功能增强（在 MRV2（GPU‑v1）路径上为多模态 Prefix‑LM 引入双向注意力）  
**⚡ 重要程度**：🟡 中  
**📋 变更摘要**  
1. 为模型配置加入 `is_mm_prefix_lm` 缓存属性，以便快速判断模型是否需要多模态前缀‑LM 逻辑。  
2. 在 `vllm/v1/worker/gpu/attn_utils.py` 中新增 `compute_mm_prefix_ranges`，对每个请求的图像/视频 token 计算可双向（bidirectional）注意力的文档区间，并在 `build_attn_metadata` 里把这些区间作为 `mm_req_doc_ranges` 传入。  
3. `default.py` 在准备注意力元数据时调用上述函数并将结果注入 `AttnMetadata`，实现 Prefix‑LM 的跨模态双向上下文。  
4. 新增完整的端到端回归测试 `test_mm_prefix_lm.py`，对 Gemma‑3‑4B‑IT 验证 VLLM 前缀‑LM 的 image‑prefill 隐状态与 HuggingFace 实现的余弦相似度 ≥ 0.9。  
5. CI 配置加入对应测试，同时在 ROCm 环境下暂时跳过（已知精度问题）。  

**🎯 影响范围**  
- `vllm/config/model.py`（模型属性缓存）  
- `vllm/v1/worker/gpu/attn_utils.py`（注意力元数据构造、跨模态区间计算）  
- `vllm/v1/worker/gpu/model_states/default.py`（准备注意力时注入新区间）  
- 测试目录 `tests/models/multimodal/generation/`（新增回归用例）  

**💡 关注建议**  

1. **兼容性**：`is_mm_prefix_lm` 仅在模型 arch config 标记为 `is_mm_prefix_lm=True` 时生效，旧模型不受影响。请检查自定义模型的 `model_arch_config` 是否正确声明。  
2. **性能**：计算跨模态区间本身开销极小，但会导致 Prefix‑LM 的注意力矩阵在图像段落上变为双向，可能增加显存占用（尤其在长滑动窗口下）。建议在生产环境为大图像或长序列限制 `sliding_window`，或使用 `limit_mm_per_prompt` 控制每个提示的图像数。  
3. **ROCm**：当前实现对 ROCm 的注意力实现存在数值差异，CI 已跳过。若在 AMD GPU 上使用，请关注后续修复或手动关闭 Prefix‑LM（`VLLM_ALLOW_INSECURE_SERIALIZATION=1` 不会影响此问题）。  
4. **测试**：新增 `test_mm_prefix_lm.py` 依赖 `google/gemma-3-4b-it` 与 `AutoModelForImageTextToText`，首次运行时会下载模型，建议在 CI 机器上预留足够磁盘和网络带宽。若本地调试，可通过 `--skip-multimodal` 选项暂时跳过。  
5. **代码维护**：`compute_mm_prefix_ranges` 暴露在 `attn_utils.py`，若后续加入新的多模态模态（audio、3D）需在函数中添加对应过滤条件。`mm_req_doc_ranges` 作为可选参数传递，确保旧的 `build_attn_metadata` 调用仍能保持兼容。  

总体来看，此次提交为多模态 Prefix‑LM（MM‑Prefix‑LM）提供了必要的注意力边界信息，使图像前缀在预填阶段能够看到后续文本上下文，显著提升了如 Gemma‑3 这类原生 Prefix‑LM 模型的生成质量。后续关注 ROCm 支持及更大 multimodal 模型的显存表现即可。

### [Feature] Support sequence parallel without the need for DP, 1.9%~5.0% E2E Throughput Improvement (#47070)
**SHA**: `b6cc46e` | 🔗 [查看提交](https://github.com/vllm-project/vllm/commit/b6cc46ec3b903c71405f4355c1e9ecb47ae54bb2)

**🎯 变更类型**：功能增强（为 MoE 引入“序列并行”模式，摆脱对 Data‑Parallel 的依赖）  
**⚡ 重要程度**：🟡 中  

**📋 变更摘要**  
1. 在 `ParallelConfig` 中新增 `use_sequence_parallel_moe` 标志，判定是否启用序列并行；去掉了仅在 DP>1 时才可用的限制。  
2. `DeviceCommunicator` 在初始化 All‑2‑All 管理器时，除了 DP>1 还会在序列并行 EP 场景下打开。  
3. `DPMetadata` 与前向上下文在创建时放宽了对 `data_parallel_size>1` 的断言，允许在仅开启序列并行时仍生成相应 metadata。  
4. `FusedMoE`、`MoERunner`、模型实现（DeepSeek‑V2、GPT‑OSS）均加入对 `is_sequence_parallel`/`use_sequence_parallel_moe` 的判断，处理跨 TP 的 padding、reduce‑scatter 与残差切分逻辑。  

**🎯 影响范围**  
- `vllm/config/parallel.py`  
- `vllm/distributed/device_communicators/*`  
- `vllm/forward_context.py`  
- `vllm/model_executor/layers/fused_moe/*`  
- `vllm/model_executor/models/deepseek_v2.py`、`gpt_oss.py`  

**💡 关注建议**  

1. **功能验证**  
   - 在单卡、DP=1、TP>1 的环境下开启 `use_sequence_parallel_moe=True`，跑完整的 MoE 推理/微调，确认 `DPMetadata` 正确生成且 `num_tokens_across_dp` 为 `[batch]`。  
   - 对比开启 DP+序列并行、仅序列并行、仅 DP 的吞吐率，确保 1.9%~5% 的提升不伴随数值误差。  

2. **兼容性检查**  
   - 目前 `GPTOSS` 将序列并行与 LoRA 互斥（`and vllm_config.lora_config is None`），若后续想同时使用 LoRA，需要额外适配。  
   - 关注 `is_moe_model` 为 `True` 且 `use_sequence_parallel_moe=False` 时的老路径不受影响。  

3. **边界与异常处理**  
   - `hidden_states` 的负数模运算实现了 “填充到最近的 TP 整除”，但在极端序列长度（如 0、非常小于 TP）时需确认不会产生不必要的全零 padding。  
   - `DPMetadata.make` 仍要求 `num_tokens_across_dp` 为张量，确保在仅序列并行时调用方始终提供 `torch.tensor`（已在 `set_forward_context` 中补全）。  

4. **性能监控**  
   - 由于 All‑2‑All 现在在 **DP=1、Seq‑Parallel=True** 场景也会启动，建议在多机多卡环境下监控网络带宽与延迟，防止在高 TP（>8）时出现瓶颈。  

5. **文档与配置**  
   - 在 README、配置示例中补充 `use_sequence_parallel_moe` 的说明及推荐组合（DP = 1 + Seq‑Parallel、DP > 1 + Seq‑Parallel）。  
   - 明确说明该特性不适用于使用 LoRA 的 GPT‑OSS 模型，或提供后续的实现路径。  

总体来看，改动把 MoE 的序列并行从 “DP > 1 + EP” 迁移到 “DP ≥ 1”，降低了对数据并行的硬性依赖，提升了在单节点多 GPU 场景下的吞吐。后续重点关注兼容性测试、极端序列长度的填充正确性以及网络通信开销。

### [Bugfix][Pooling] Forward instruction to Jina reranker scoring prompts (#47590)
**SHA**: `9226613` | 🔗 [查看提交](https://github.com/vllm-project/vllm/commit/922661304357313700340e787cfe19d58b8c0a2f)

**变更概述**  
本次提交在 **Jina Reranker v3** 的 pooling runner 中新增了对 **instruction** 参数的转发与解析，使得 Score / Rerank 接口能够在提示里携带任务指令（如「把体育相关的段落排在前面」），并在生成 prompt 前统一去除模型内部的占位符 `<|embed_token|>`、`<|rerank_token|>`。

**核心实现**  
1. **`JinaRankingIOProcessorMixin.format_docs_prompts_func`**  
   - 将原先的类方法 `sanitize_input` 改为局部函数，避免在类层面留下不必要的静态方法。  
   - 在拼装 prompt 时，若 `instruction` 不为空，会先对其进行同样的 token 清理，再通过 `<instruct>...</instruct>` 包装进系统提示。  
2. **`_pre_process`**  
   - 新增对 `chat_template_kwargs`（尤其是 `instruction`）的提取，将其穿透到 `format_docs_prompts_func`。  
   - 对单条/多条 query‑doc 组合均加入指令字段。  
3. **测试**  
   - `test_jina_reranker_v3.py` 添加了 **instruction** 场景的在线测试，分别验证 Score 与 Rerank 两条链路的 `prompt_tokens` 增加、返回结果长度一致以及分数/相关度的数值相近。  
   - 为了触发指令路径，启动服务器时加入 `--enforce-eager` 参数（强制 eager 模式），防止 lazy‑pipeline 跳过指令处理。  
4. **示例脚本** `examples/pooling/token_embed/jina_reranker_v3_online.py`  
   - 展示了如何在实际请求中携带 `instruction`，并分别调用 `/score` 与 `/rerank`。  

**影响范围**  
- 受影响的模块：`vllm/entrypoints/pooling/scoring/io_processor.py`、`tests/models/language/pooling/test_jina_reranker_v3.py`、示例脚本。  
- 只在 **pooling‑runner**（`--runner pooling`）下生效，不影响文本生成（Chat）路径。  

**兼容性 & 风险**  
- 只在有 `instruction` 参数时增加 token，旧有调用保持原行为，向后兼容。  
- 由于在 `format_docs_prompts_func` 中对 `instruction` 也执行占位符清理，若用户自行在指令中使用这些特殊 token，可能被意外去除；建议在文档中说明该清理规则。  
- `--enforce-eager` 仅在测试中使用，生产环境仍可在 lazy 模式下运行，指令仍会被正常转发。  

**建议**  
1. **文档更新**：在 README / API 文档中明确 `instruction` 支持的场景、清理规则以及对 `prompt_tokens` 的影响。  
2. **类型检查**：为 `chat_template_kwargs` 添加 Pydantic 模型或 TypedDict，防止误传未知键导致 silent‑failure。  
3. **回归测试**：在 CI 中保留 `--enforce-eager` 只针对指令测试，防止懒加载导致测试不稳定。  
4. **性能监控**：指令会导致额外的 token 计数与 prompt 生成，建议在高并发环境下监测 Prompt‑Tokens 的增长幅度，确保不会触发意外的配额或计费。  

整体来看，此次改动为 Jina‑Reranker 引入了更灵活的指令化能力，提升了检索排序的可定制性，代码改动局部且向后兼容，只需注意文档说明与轻量的类型安全即可。

### [Bugfix][Gemma4] Fix FA4 mm_prefix mask: add sliding window and absolute q_idx (#47332)
**SHA**: `34b560b` | 🔗 [查看提交](https://github.com/vllm-project/vllm/commit/34b560b725b401f33477464163244b4744d1c40f)

**🎯 变更类型**：Bug 修复（Gemma‑4 mm_prefix mask）  
**⚡ 重要程度**：🟡 中  

**📋 变更摘要**  
- 为 FlashAttention‑4 中的 `mm_prefix` 掩码加入滑动窗口逻辑，并兼容绝对‑q‑idx 的计算方式。  
- 新增 `sliding_window_left` 参数，区分 **causal + sliding‑window** 与 **bidirectional block** 的处理。  
- 通过 `mm_prefix_clamp_sliding_window` 标记，仅在 Gemma‑4 的局部层上对双向块进行窗口裁剪，保持其它 mm‑prefix 模型行为不变。  

**🎯 影响范围**  
- `vllm/v1/attention/backends/flash_attn.py`（FA4 前向路径与掩码生成）。  
- 依赖 `mm_prefix` 的模型（尤其是 Gemma‑4）在 Prefill 阶段的注意力计算。  

**💡 关注建议**  
1. **模型兼容性**：对使用 `mm_prefix`、但未显式声明 `mm_prefix_clamp_sliding_window` 的模型，行为保持原样；如需在自定义层使用窗口裁剪，务必在层对象上添加该属性。  
2. **性能验证**：新掩码在滑动窗口开启时会额外计算绝对 q‑idx，建议通过大批量 Prefill 测试验证吞吐量是否受影响。  
3. **回归测试**：加入针对 Gemma‑4/FA4 的单元测试，覆盖（a）全局注意力层、（b）滑动窗口层、（c）未开启 `mm_prefix_clamp_sliding_window` 的普通 mm‑prefix 层，确保旧行为未被意外改动。  
4. **文档更新**：在模型配置说明中标记 `mm_prefix_clamp_sliding_window` 参数的意义及默认值，避免使用者误解。  

总体来看，此次修改修正了 Gemma‑4 在预填阶段的注意力遮罩错误，保持了向后兼容性。不过由于引入了绝对位置计算，建议在 CI 中加入性能回归检测，并在发布说明中明确新标记的使用方式。
