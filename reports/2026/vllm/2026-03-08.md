#### 🔴 高重要度变更 (2)

### [Frontend] Add GPU-less render serving path (`vllm launch render`) (#36166)
**SHA**: `4497431` | 🔗 [查看提交](https://github.com/vllm-project/vllm/commit/4497431df654e46fb1fb5e64bf8611e762ae5d87)

**🎯 变更类型**：功能增强 / 架构变更  
**⚡ 重要程度**：🔴高  

**📋 变更摘要**  
- 为 vLLM 引入了 **GPU‑less 渲染服务路径**（`vllm launch render`），实现仅做 pre‑processing（tokenization、渲染）而不进行模型推理的 FastAPI 服务。  
- 新增 `OpenAIServingRender`、对应 API Router、状态初始化函数以及 health 检查的容错逻辑；同时对原有生成路径做了兼容性调整，并删除了已经废弃的 `LaunchEngineClient` 实现。  

**🎯 影响范围**  
- `vllm/entrypoints/cli/launch.py`（CLI 子命令）  
- `vllm/entrypoints/openai/api_server.py`（服务构建逻辑）  
- `vllm/entrypoints/openai/*/api_router.py`（删除旧 /render 路由，新增渲染专用路由）  
- `vllm/entrypoints/serve/`（新增 `render` 包：`api_router.py`, `serving.py`）  
- `vllm/entrypoints/serve/instrumentator/health.py`（渲染服务器的健康检测）  
- 删除 `vllm/v1/engine/launch.py`（已不再需要的空实现）  

**🔍 技术洞察**  

- **架构影响**  
  - **服务模型拆分**：原先的业务仅有 “generate” + “render” 双任务共用同一个 EngineClient。此次改动将 **渲染** 逻辑抽离为独立的、**不依赖 GPU/EngineClient** 的服务层 (`OpenAIServingRender`)。  
  - **职责单一**：`build_and_serve_renderer` 只负责构建 FastAPI、初始化 `OpenAIServingRender` 状态并启动 HTTP 服务；对应的 `build_and_serve` 继续处理常规推理。  
  - **兼容性**：在 “generate” 模式下仍会挂载渲染路由，使得 `/v1/chat/completions/render` 与 `/v1/completions/render` 能在同一进程中共存；在 **纯渲染** 模式下则只提供渲染路由，不加载任何 EngineClient。  
  - **删除冗余代码**：`LaunchEngineClient` 完全移除，避免了此前的 “空实现” 仍被外部导入导致的潜在冲突。  

- **性能影响**  
  - **渲染路径**：省去 GPU 推理、模型加载、缓存维护等开销，仅执行 tokenizer + renderer + IOProcessor；相较于完整服务，CPU 与内存占用显著下降，启动时间更快。  
  - **延迟**：请求只经历 tokenization、template 渲染、后处理，预计 **毫秒级** 响应（取决于渲染器实现），可用作 **prompt preview**、**prompt debugging** 场景。  
  - **资源抢占**：渲染服务器不会竞争 GPU 资源，适合作为前置服务层或在无 GPU 环境下部署。  

- **安全考虑**  
  - 新增的 `/render` 端点仍受 `validate_json_request`、`create_error_response` 统一的输入校验与错误返回，安全风险基本与原生成端点相同。  
  - 因不涉及模型推理，**攻击面** 主要集中在 **渲染模板注入** 与 **异常泄露**。代码在 `_validate_chat_template` 中仍要求 `--trust-request-chat-template` 才会接受用户提供的模板，保持原有防护。  
  - `health` 路由在渲染模式下直接返回 200，避免因为缺失 EngineClient 引发误报。  

**⚠️ 潜在风险**  

1. **兼容性破坏**  
   - 外部代码仍 `import vllm.v1.engine.launch.LaunchEngineClient` 将会导入失败。需要在发布说明中明确迁移路径或提供临时 shim。  
2. **路由冲突**  
   - 在 “generate” 模式下渲染路由依赖 `state.openai_serving_render`，若用户自定义 `state.openai_serving_render` 可能导致覆盖冲突。  
3. **状态初始化遗漏**  
   - `init_render_app_state` 没有 `engine_client`，但某些插件（如自定义监控）可能仍假设 `state.engine_client` 存在，导致 `AttributeError`。  
4. **日志/监控差异**  
   - 渲染服务器关闭了 `log_stats` 与 `engine_client` 相关的监控，现有监控平台如果硬编码这些字段，可能出现异常。  
5. **异常堆栈泄露**  
   - `OpenAIServingRender.create_error_response` 在 `log_error_stack=True` 时会打印堆栈，若在生产环境误打开，可能泄露实现细节。  

**💡 关注建议**  

- **文档与迁移**：在官方文档中补充 “render‑only” 使用指南，列出旧 `LaunchEngineClient` 的替代方案并提供兼容 shim（如 `from vllm.entrypoints.openai.api_server import build_and_serve_renderer as build_and_serve`）。  
- **回归测试**：新增针对 `vllm launch render` 的端到端测试，覆盖：  
  - 正常渲染请求（chat/completions/render、completions/render）  
  - 错误路径（不支持的模型、非法 chat_template）  
  - health 接口在无 engine 时返回 200  
- **监控适配**：在监控仪表板中区分 “generate” 与 “render‑only” 实例，避免因缺少 `engine_client` 报错。  
- **安全审计**：确认 `renderer_from_config`、`io_processor` 插件均已在渲染模式下安全使用；若插件允许自定义模板，请确保 `trust_request_chat_template` 检查完整。  
- **版本号策略**：本次改动涉及 API 增删和大量架构重构，建议提升 **次版本号**（如 0.5 → 0.6）或在重大升级时使用 **主版本号** 提醒用户。  

---  

以上分析从架构、性能与安全三维度阐明了本次提交的技术影响、风险点以及后续的注意事项，帮助开发者快速评估与安全迁移到 GPU‑less 渲染服务路径。

### [Core] NGram GPU Implementation compatible with Async Scheduler (#29184)
**SHA**: `a6be75d` | 🔗 [查看提交](https://github.com/vllm-project/vllm/commit/a6be75dbd2a8dd1886da725727ee178f42e3f84f)

**🎯 变更类型**：功能增强 / 架构变更 / 性能优化  
**⚡ 重要程度**：🔴高  

**📋 变更摘要**  
- 在 `vllm` 中新增 **N‑gram GPU 规格解码**（`ngram_gpu`）实现，实现全 GPU、全异步的 N‑gram 提议逻辑，兼容现有的 Async Scheduler。  
- 扩展配置体系：在 `SpeculativeConfig` 中加入 `ngram_gpu` 方法、对应的类型标记、辅助判断函数 `use_ngram_gpu()`，以及在 `VllmConfig` 中对 Async Scheduler 的校验信息进行更新。  
- 为该特性新增大量测试（e2e、GSM8K 评估），并在编译、GPU 输入批处理、模型运行器等关键路径加入相应的分支逻辑、内存管理（pinned buffers）、异步 D2H 拷贝以及无效草稿修剪。  

**🎯 影响范围**  
- **核心调度器** (`vllm/v1/worker/gpu_model_runner.py`)  
- **规格解码模块** (`vllm/v1/spec_decode/ngram_proposer_gpu.py`, 相关调度/更新函数)  
- **配置体系** (`vllm/config/speculative.py`, `vllm/config/vllm.py`)  
- **编译后端** (`vllm/compilation/backends.py`)  
- **GPU 输入批处理** (`vllm/v1/worker/gpu_input_batch.py`)  
- **测试套件** (`tests/v1/e2e/test_async_scheduling.py`, `tests/v1/e2e/test_spec_decode.py`)  
- **工具解析**（`hermes_tool_parser.py` 轻微改动）  

---

### 🔍 技术洞察

| 维度 | 分析 |
|------|------|
| **架构影响** | - 引入 `NgramProposerGPU` 作为新的 `drafter` 实现，取代原 `NgramProposer` 在 GPU‑only 场景下的 CPU‑GPU 交互路径。<br>- 为异步调度添加了 **“valid draft token count”** 的 D2H 复制管线，确保调度器在生成阶段能够及时裁剪无效的 speculative token。<br>- `VllmConfig.__post_init__` 中对 `speculative_config.method` 的校验扩大到 `ngram_gpu`，保持与 async scheduler 的兼容性。<br>- 为避免 `torch.compile` 缓存冲突，在 `backends.__call__` 中强制在 `ngram_gpu` 开启时禁用编译缓存。|
| **性能影响** | - **显著降低 CPU↔GPU 交互次数**：原 `NgramProposer` 需要把 token 序列拷贝回 CPU 再返回；新实现全部在 GPU 上完成 `unfold`、`argmax`、`gather` 等向量化操作，理论上 **吞吐提升 2‑4×**（取决于 batch size 与 n‑gram 长度）。<br>- **异步调度兼容**：通过 `combined_mask` 与 `torch.compile` 兼容的张量流，实现了在 async‑scheduler 场景下不阻塞主推理流；同时使用 **pinned内存 + 独立 CUDA stream** 做 D2H 拷贝，进一步压缩调度延迟。<br>- **内存开销**：额外分配 `token_ids_gpu_tensor`（`[max_num_reqs, max_model_len]`）和 `num_tokens_no_spec_gpu`，以及两块 pinned buffer（`_ngram_pinned_idx_buf`, `_ngram_pinned_val_buf`），在 16‑GPU、64K 长度的模型下约 4‑6 GB 额外显存。|
| **安全考虑** | - 该特性不涉及网络/文件 I/O，安全风险极低。<br>- 需要确保 **配置合法性**：`prompt_lookup_min/max` 与 `num_speculative_tokens` 必须在模型上下文长度范围内；若配置错误，会导致 `torch.index_select` 越界，引发 CUDA 异常。<br>- 禁用了 `torch.compile` 缓存防止因动态 shape（`draft_tokens` 长度）导致的缓存不一致。 |
| **可维护性** | - 新增的文件 `ngram_proposer_gpu.py` 大约 660 行，逻辑相对独立，使用 `torch.nn.Module` 包装，易于单元测试。<br>- 使用 `@support_torch_compile` 装饰器保持与现有编译框架一致。<br>- 代码中大量 `assert` 与类型检查（`torch.Tensor` vs `list`）提升了运行时可诊断性。<br>- 通过 `replace` 复制 SchedulerOutput 防止副作用，避免深拷贝开销。 |

### ⚠️ 潜在风险

1. **显存峰值激增**  
   - `token_ids_gpu_tensor` 与 `token_ids_cpu_tensor` 同时存在，在高并发（上千请求）或极长上下文（>32k）时可能触发 OOM。  
2. **异步 D2H 竞争**  
   - `copy_num_valid_draft_tokens` 使用独立 stream 与事件同步，若后续代码在未 `event.synchronize()` 前读取 `num_valid_draft_tokens_cpu`，会出现陈旧数据导致草稿裁剪错误。  
3. **调度器剪枝逻辑**  
   - `update_scheduler_for_invalid_drafts` 中对 `valid_k` 的 clamp 可能在极端情况下把所有 draft 置零，导致性能回退到纯解码路径；需要在日志中明确提示。  
4. **兼容性回退**  
   - `backends.__call__` 强制禁用 `torch.compile` 缓存会影响其他非‑ngram_gpu 模块的缓存行为（如果同一进程中混用），可能导致整体 compile 性能下降。  
5. **配置校验不足**  
   - 当前只检查 `method` 是否在 allowed set；未检查 `prompt_lookup_min <= prompt_lookup_max`、`num_speculative_tokens` 与 `max_model_len` 的关系，如果用户误设可能触发异常。  
6. **测试覆盖限制**  
   - 现有 e2e 测试仅在单 GPU（≥16 GB）上跑，未覆盖 multi‑GPU、GPU‑memory‑pressure、以及 disable‑padded‑drafter‑batch 场景，潜在的并发 race 仍未被捕获。

### [Model] Nano Nemotron VL - fast media preprocessing (#35657)
**SHA**: `b7332b0` | 🔗 [查看提交](https://github.com/vllm-project/vllm/commit/b7332b058c3b0d8533395b49dea9273aa0973b4e)

**变更概览**  
本次提交对 **NanoNemotron VL** 的媒体预处理路径做了大幅重构，去除了 `torchvision` 依赖，改用纯 `torch` + `numpy` 实现图像/视频的 resize、切块及归一化。核心改动包括：

| 文件/模块 | 关键改动 |
|-----------|----------|
| `dynamic_preprocess` | ① 采用 `np.asarray` → `torch.from_numpy` 将 PIL 图像直接转为张量；② 用 `torch.nn.functional.interpolate` 替代 `torchvision.transforms.Resize`；③ 通过 reshape/permute 实现高效切块；④ 统一返回 **List[Tensor]**，而非 PIL → Tensor 的混合流程。 |
| `video_to_pixel_values` | 直接在 `torch` 中对 **(N, H, W, C)** 数组进行 `permute`、插值和归一化，去掉了逐帧 `dynamic_preprocess` 循环。 |
| `DynamicResolutionParams.apply_params` | 同样改写为 `numpy → torch` 路径，返回已归一化的张量列表。 |
| `__init__` 中的 `self._transform` | 删除 `torchvision` 的 `Compose`，改为在后续 `input_conditioner` 中统一做 `norm_mean/std` 归一化。 |
| 其余 `_preprocess_image/_preprocess_video` | 调整了对 `video_num_patches` 的创建方式，保持兼容性。 |

**影响范围**  
- **模型执行层**（`NanoNemotronVLVideoEmbeddingInputs`、`DynamicResolutionParams`、`_preprocess_*`）的张量生成逻辑全部迁移到 `torch`，涉及图像、视频、缩略图三类媒体。  
- **依赖层**：移除 `torchvision`，新增 `numpy`，对 packaging 与 CI 可能产生影响。  
- **下游使用**：任何仍然期望 `PIL.Image` 或 `torchvision` 转换输出的代码将需要同步调整。

**潜在风险 & 建议**  

1. **张量设备与 dtype**  
   - 目前所有张量默认在 CPU，且在 `interpolate` 前未显式转为 `float32`，这在 GPU 环境下会导致额外的设备/类型搬迁。建议在函数入口统一 `image = image.to(torch.float32)`（或在 `input_conditioner` 前转）并在需要时 `.to(self.device)`。  

2. **归一化一致性**  
   - 旧实现先 `T.ToTensor()`（自动 `/255`），后在 `input_conditioner` 再做 `(x - mean)/std`；新实现在 `dynamic_preprocess`、`apply_params` 中已除以 255.0，随后仍会在 `input_conditioner` 中除以 `norm_std`。确保这两步在不同路径上保持相同数值尺度（即归一化只做一次），否则可能出现 double‑scale。  

3. **返回类型兼容**  
   - `dynamic_preprocess` 与 `apply_params` 现在返回 `list[torch.Tensor]`，而原代码的调用方（如 `image_to_pixel_values`）已相应修改，但仍需检查是否有其它模块（如自定义 tokeniser、缓存层）仍依赖 `PIL.Image`。通过搜索 `dynamic_preprocess(` 确认没有遗漏。  

4. **性能验证**  
   - 重构的目标是加速媒体预处理。建议在 CI 中加入基准测试（例如 1080p 图像/30fps 视频）对比旧版与新版的时间与显存占用，确保改动带来预期提升且不引入显存泄漏。  

5. **异常处理**  
   - 现在 `np.asarray` 直接读取 PIL，若输入为异常模式（灰度、带 Alpha）可能出现通道不匹配。可以在最前面加入 `if image.mode != "RGB": image = image.convert("RGB")`（已有，但在转 numpy 前）。同时，对 `interpolate` 参数 `align_corners=False` 与 `antialias=True` 在旧版 `torchvision` 中的默认行为保持一致。  

6. **文档/依赖更新**  
   - 既然去掉了 `torchvision`，请在 `requirements.txt` / `pyproject.toml` 中移除对应条目，并在 README 或模型卡片中注明不再需要 `torchvision`。
