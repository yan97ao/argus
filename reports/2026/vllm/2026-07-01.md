### [Spec Decode] DSpark (#46995)
**SHA**: `f5a8d73` | 🔗 [查看提交](https://github.com/vllm-project/vllm/commit/f5a8d73377d0f0a4e00cba172f9fbd0d50471b07)

**🎯 变更类型**：功能增强 / 架构变更  
**⚡ 重要程度**：🔴 高  

**📋 变更摘要**  
本次提交为 vLLM 项目新增 **DSpark**（DeepSeek‑V4 半自回归草稿）推理路径，实现了 DSpark 专用的模型注册、配置选项、非因果稀疏 MLA 计算、上下文 KV 预计算以及专属 **Speculator**。同时配套补充了大量单元/端到端测试、CUDA kernel 实现以及在调度器、模型加载、权重映射等关键模块的适配。DSpark 仅在 CUDA 环境下可用，且在 V2 GPU Model Runner 中强制启用。

**🎯 影响范围**  
- `vllm/model_executor/models/registry.py` – 新增模型别名 `DSparkDraftModel`、`Qwen3DSparkModel`。  
- `vllm/config/speculative.py`、`vllm/config/vllm.py` – 新增 `dspark` 类型、在 V2 runner 中强制开启、调度器 look‑ahead 计数调整、异步调度兼容校验。  
- `vllm/model_executor/models/qwen3_dspark.py`、`vllm/models/deepseek_v4/nvidia/dspark.py` – DSpark backbone 与 Markov 头实现、权重加载逻辑。  
- `vllm/v1/attention/backends/mla/sparse_swa.py` – 非因果索引宽度计算、专用 Triton kernel `_compute_dspark_noncausal_swa_indices_kernel`。  
- `vllm/v1/worker/gpu/spec_decode/*` – DSpark Speculator 实现、CUDA‑graph 捕获、准备输入的 Kernel 参数扩展、采样路径改为 **anchor‑first** 并加入 Markov 采样。  
- `vllm/v1/core/sched/scheduler.py` – `num_lookahead_tokens` 对 DSpark 的特殊处理。  
- 其他：测试覆盖 (`tests/v1/attention/test_dspark_noncausal_sparse_mla.py`、`tests/v1/e2e/spec_decode/test_spec_decode.py`)、平台兼容性标记（ROCm/XPU stub）等。

### [Rust Frontend] Split engine core DTOs into separate modules (#47265)
**SHA**: `c638f92` | 🔗 [查看提交](https://github.com/vllm-project/vllm/commit/c638f9216a08bfb5644d8a266ddd35421e04118d)

**🎯 变更类型**：架构变更 / 重构  
**⚡ 重要程度**：🔴高  
**📋 变更摘要**：  
本次提交将 **vllm‑engine‑core client** 中与 Engine Core 通信的所有传输 DTO（Data Transfer Object）拆分为独立子模块 `dtype、request、sampling、structured_outputs、output`，并相应地调整了项目内所有引用。核心协议实现从单一巨型 `protocol/mod.rs` 中抽离，形成更细粒度的模块边界，提升代码可维护性与可读性。

**🎯 影响范围**：  
- `rust/src/engine-core-client/src/protocol/`  
  - 新增 `request.rs、sampling.rs、structured_outputs.rs、output.rs`，并在 `mod.rs` 中仅导出子模块入口。  
- 所有依赖 Engine Core 协议的 Rust 子项目（chat、engine‑core‑client、llm、parser、server、mock‑engine、text 等）均已更新 `use` 路径。  
- 相关单元/集成测试与示例代码同步改为新的模块路径。

## 🔍 技术洞察

| 维度 | 影响分析 |
|------|----------|
| **架构影响** | - **模块化提升**：协议层从单文件 800 行压缩到 4‑5 个职责明确的文件，降低耦合，便于新增/修改单独 DTO。<br>- **公共 API 变更**：原 `vllm_engine_core_client::protocol::{EngineCoreRequest, EngineCoreSamplingParams,…}` 已不再导出，改为 `protocol::request::EngineCoreRequest`、`protocol::sampling::EngineCoreSamplingParams` 等，属于**向后不兼容**的破坏性变更。<br>- **可维护性**：每个子模块拥有独立的 `serde` 配置、文档注释，未来可单独演进（如增删字段、增加新协议），不必编辑巨型 `mod.rs`。 |
| **编译/性能** | - **编译时间**：拆分后每个文件的增量编译更快，整体增量构建收益显著（尤其在 CI 环境下）。<br>- **运行时性能**：仅修改了 **导入路径** 与 **模块组织**，序列化/反序列化实现未变，更改不会影响 `msgpack` 编码效率或运行时吞吐。 |

- `vllm.distributed.device_communicators.cuda_communicator`（AllGather / ReduceScatter 路由）  
- `vllm.distributed.device_communicators.all_reduce_utils`（开启判断函数）  
- `vllm.distributed.device_communicators.pynccl_allocator`（注册基址管理）  
- 测试套件 `tests/distributed/test_nccl_symm_mem.py`  

**🔍 技术洞察**  

- **架构影响**  
  - **分布式通信层**：在 `CudaCommunicator` 中对 AllGather / ReduceScatter 加入对称‑​内存的选路，保持向后兼容（不满足条件仍走原 ring‑​allgather / reduce_scatter 实现）。  
  - **资源管理**：新增 `_symm_scratch_bufs` 缓存，实现“一次分配、永久注册”模式，避免每次通信重新进入 `nccl_symm_mem_context` 带来的图捕获/同步开销。  
  - **注册粒度**：`pynccl_allocator` 按 NCCL 通信器唯一 ID（`comm_key`）维护 `address → set`，防止跨进程/不同 communicator 重复注册，同步性提升。  

- **性能影响**  
  - **带宽提升**：对称‑​内存通过 NVLink‑SM 直接在 GPU Memory 中共享，可在大规模（≥4 MiB）张量上显著提升 AllGather / ReduceScatter 的吞吐，尤其在 multi‑GPU（TP）设置下，跨 GPU 的数据搬运次数大幅下降。

- **安全考虑**  
  - 未引入外部网络或系统调用，仅在 GPU 内部使用 NCCL 注册内存窗口。唯一风险是 **未对称‑​内存注册**（即 `sizes` 不均匀）会触发 NCCL 的死锁；代码已在 `all_gather`、`reduce_scatter` 前判断并回退到传统路径，避免安全／稳定性隐患。

2. **显存压力**：持久化 scratch 缓冲区在模型并行度高、Batch‑Size 大时可能占用显著显存，导致 OOM。  
3. **不均匀分块**：若上层调度产生非均匀 `sizes`（如变长序列切片），代码会回退到普通实现，但在 `all_gatherv` 分支里仍有 `if sizes is None and should_nccl_symm_mem_ag_rs():` 检查，确保不会误用。若未来加入更多不均匀聚合路径，需要保持此保护。  
4. **注册泄漏**：`_registered_base_addrs` 随每次 `nccl_symm_mem_context` 退出而持久化，长期运行（如服务化部署）中若进程频繁创建/销毁 communicator，可能累积已注册基址集合而不释放。当前实现使用 `comm_key` 进行隔离，但未提供显式清理入口。
