### Refactor AWQ Marlin MoE onto modular WNA16 oracle (#42483)
**SHA**: `78e7a7b` | 🔗 [查看提交](https://github.com/vllm-project/vllm/commit/78e7a7b9b0b9c285bf6978c3fc09eeecea3ff230)

**🎯 变更类型**：功能增强 / 重构  
**⚡ 重要程度**：🔴高  

**📋 变更摘要**  
本次提交将原有的 AWQ‑Marlin MoE 实现迁移到全新的 **WNA16 Oracle** 模块化 Kernel 框架，实现了权重量化、激活全局尺度、零点 (zero‑point) 以及 8‑bit 激活的统一处理。通过在测试层面加入更丰富的参数组合（包括不同数据类型、量化方式和输入 dtype），提升了代码可维护性并为后续的 LoRA、并行 MoE、Expert Parallelism 打下基础。

**🎯 影响范围**  
- `vllm/model_executor/layers/fused_moe/*`（核心 MoE 计算路径）  
- `vllm/model_executor/layers/quantization/awq_marlin.py`（AWQ‑Marlin 量化层）  
- `vllm/model_executor/layers/quantization/auto_gptq.py`（GPTQ 兼容）  
- `tests/kernels/moe/test_moe.py`（单元测试）  

**🔍 技术洞察**  

| 维度 | 影响描述 |
|------|----------|
| **架构影响** | 1. 引入 **WNA16 Oracle**（`oracle/int_wna16.py`）作为统一入口，统一了 GPTQ 与 AWQ 两种量化后处理流程。<br>2. 将权重、尺度、零点的后处理抽象为 `_process_weights_marlin` 与 `_process_awq_weights_marlin`，实现 **模块化、可插拔**。<br>3. 新增 `input_global_scale1/2`、`input_dtype` 参数，支持 8‑bit 激活的全局尺度校正。<br>4. 通过 `select_wna16_moe_backend` 动态选择后端（WNA16‑MoE‑Kernel），在 LoRA 开启时自动创建 `FusedMoEKernel` 实例，取代原 `fused_marlin_moe` 直接调用。 |
| **性能影响** | - **潜在加速**：模块化 Kernel 在内部使用 **batched_fused_marlin_moe**，并针对 `block_size_m` 进行自适应调优，能在 Expert Parallelism 场景下降低 padding 开销。<br>- **内存占用**：新增的排序索引、zero‑point、全局尺度张量会在每个 MoE 层额外分配显存，尤其是使用 8‑bit 激活时的 `input_global_scale`。<br>- **分支路径**：对 `input_dtype == torch.float8_e4m3fn` 的特殊预处理仍保持原有 4‑bit→FP8 转换路径，避免额外的数值误差。 |
| **安全考虑** | - 代码仅在 **CUDA** 环境下执行，未涉及网络或文件 I/O，安全风险极低。<br>- 参数注册/替换使用 `torch.nn.Parameter`，遵循 PyTorch 的梯度保护，未引入新的可执行代码注入点。 |
| **可维护性** | - 参数注册统一到 `_replace_or_register_parameter`，降低了手动 `replace_parameter` 的出错概率。<br>- 将不同量化后处理分别封装，提升了可读性并方便后续扩展（如支持新量化方案）。<br>- 测试用例覆盖了多种 `dtype`、`quant_dtype`、`input_type` 组合，确保回归安全。 |

**⚠️ 潜在风险**  

1. **兼容性回退**：旧版代码直接调用 `fused_marlin_moe`，新实现要求 `input_global_scale*`、`w*_qzeros` 必须在模型加载时显式提供。若第三方模型使用自定义 checkpoint（缺失这些字段），加载会抛异常。  
2. **显存回退**：对大模型（> 70B) 启用 8‑bit 激活时，额外的全局尺度张量可能导致 OOM，尤其在多卡并行且显存紧张的场景。  
3. **数值误差**：`marlin_act_int8_process_scales` 对 int8 激活的缩放因子做了 2‑步处理，若在后续算子中未同步使用同样的全局尺度，可能产生 1e‑3 级别的相对误差（测试已放宽 atol）。  
4. **分支缺失**：`select_gemm_impl` 被硬性设为不可使用（抛异常），若其它路径仍依赖此函数（如老版 LoRA 插件），可能出现运行时错误。  
5. **Backend 选择错误**：`select_wna16_moe_backend` 依据 `kInt4Static` 与 `weight_bits` 选择后端；若未来加入新量化位宽而未同步更新，可能选错后端导致崩溃。  

**💡 关注建议**  

- **兼容层**：在模型加载阶段添加 **回退路径**，如果缺少 `input_global_scale`、`w*_qzeros`，自动填充为 `None` 并使用原始 `fused_marlin_moe` 流程，保证已有 checkpoint 在不升级的情况下仍能运行。  
- **显存监控**：在 `batched_fused_marlin_moe` 中加入 `torch.cuda.max_memory_allocated()` 的日志，仅在调试模式下输出，以便定位因全局尺度导致的 OOM。  
- **单元测试**：补充 **稀疏模型**（`group_size` 与 `num_groups` 不同）以及 **多卡并行** 场景的回归测试，确保 `block_size_m` 动态调节在真实硬件上不出现不合法的 block 大小。  
- **文档更新**：在项目 README 或量化章节中明确说明 **AWQ‑Marlin 8‑bit 激活** 需要额外的 `w13_input_global_scale` / `w2_input_global_scale` 参数，并提供迁移脚本示例。  
- **后端扩展**：为 `select_wna16_moe_backend` 添加 **插件式注册**（如 `backend_registry.register(name, factory)`)），方便未来新增 `kIntXStatic` 或非‑int4 方案时无需改动核心代码。  

总体而言，此次迁移显著提升了 **代码结构的模块化程度**，为后续的 **LoRA、Expert Parallelism** 打下坚实基础，同时引入了新参数和更细粒度的后处理逻辑，需重点关注兼容性与显存开销。

### 🔍 技术洞察

| 维度 | 影响说明 |
|------|----------|
| **架构影响** | - 在 `MambaAttentionMetadata` 中新增 `block_idx_last_scheduled_token_prev_step`，并在 `Mamba2AttentionMetadataBuilder` 中分配持久化缓冲区。<br>- `state_indices_tensor_d` 的列数从 `max_model_len // block_size` 增至 `+ num_speculative_blocks`，使得 **所有** 请求在 spec‑decode 场景下拥有统一的 block 表布局。<br>- `MambaMixer2` 通过 `_decode_state_offsets` 计算 **输入/输出** 的多块索引，实现对 speculative block 的一次性 gather。 |
| **性能影响** | - 额外列和前一步缓冲区带来 **极小的内存增量**（每个请求 `int32` × `num_speculative_blocks`），对吞吐基本无影响。<br>- cudagraph 捕获阶段不再因切片长度错误产生非法内存访问，提升 **稳定性** 与 **复用率**。 |

### ⚠️ 潜在风险

1. **内存占用上升**  
   - `state_indices_tensor_d` 现在会为每个请求多分配 `num_speculative_blocks` 列；在极大 batch size 或极长模型下需留意 GPU memory 边界。  
2. **持久化缓冲区的生命周期**  
   - `block_idx_last_scheduled_token_prev_step` 只在 `mamba_cache_mode="all"` 且 `num_speculative_tokens>0` 时创建，若后续出现配置不一致的路径（如动态调节 `num_speculative_tokens`），可能出现未初始化的访问。  
3. **向后兼容**  
   - 默认 `mamba_cache_mode` 在开启 prefix caching 时仍保持原有行为，但改动了默认值的警告文字；在未显式配置的旧部署上可能导致意外的 `align`→`all` 转变，需要检查部署脚本。  
4. **多 GPU / 分布式**  
   - 持久化缓冲区通过 `CpuGpuBuffer` 在 `gpu_model_runner` 中同步，若跨节点共享，需要确保复制路径正确（当前实现仅在单机 GPU 场景已测）。

---

### 💡 关注建议

- **监控内存使用**：在使用 `mamba_cache_mode="all"` 且启用 spec‑decode 的工作负载下，记录 `state_indices_tensor_d` 与 `block_idx_last_scheduled_token_prev_step` 的显存占用，防止 OOM。  
- **验证配置一致性**：部署前确认 `cache_config.mamba_cache_mode` 与 `num_speculative_tokens` 的组合符合预期，尤其在使用 `prefix_caching` 时。  
- **回归测试**：在 CI 中加入多卡、不同 `max_num_seqs`、`num_speculative_tokens` 组合的压力测试，确保 cudagraph 捕获在所有情况下均能正确填充零并保持指针稳定。  
- **文档更新**：在官方使用指南中补充 “使用 `mamba_cache_mode=all` 时，必须开启 `speculative decoding` 并留意额外内存开销” 的说明。  
- **边界情况审查**：检查 `num_speculative_blocks` 为 0 或 `num_speculative_tokens` 动态变更时对应的缓冲区是否被正确创建/销毁，以避免悬空指针。

### [CPU] Add MXFP4 W4A16 MoE support (#41922)

**🎯 变更类型**：功能增强 / 架构变更 / 重构  
**⚡ 重要程度**：🔴高  
**📋 变更摘要**：    
- 为 CPU 后端新增 MXFP4 W4A16 量化的 MoE（Mixture‑of‑Experts）支持，包括算子、C++ 内核、Python 绑定以及相应的单元测试。    
- 拓展了现有 FP8 W8A16 实现：加入 bias 参数、可选 SwigLU（α、limit）激活以及对统一激活枚举 `CPUAcTMethod` 的抽象。

**🎯 影响范围**：    
- **核心模块**：`csrc/cpu/sgl-kernels/*`（gemm、gemm_fp8、moe、vec）  
- **Python 绑定**：`vllm/_custom_ops.py`、`csrc/cpu/torch_bindings.cpp`  
- **模型执行层**：`vllm/model_executor/layers/fused_moe/*`（CPU‑FP8 与 MXFP4 专家层、后端选择逻辑）  
- **测试套件**：CPU‑FP8 测试迁移、新增 MXFP4 量化测试、CI 配置更新

1. **版本检测**：`is_torch_equal("2.11.0")` 只能捕获完全匹配的 2.11.x，若未来出现 2.11.x 的小版本差异可能失效，建议改为 `is_torch_ge("2.11") and is_torch_lt("2.12")`。  
2. **异常安全**：Monkey‑patch 依赖内部实现，若 PyTorch 改动 `StorageBox` 名称或签名会触发 `AttributeError`，建议在 `try/except` 中捕获并记录警告，以免启动时崩溃。  
3. **文档/标记**：在 README 或 vLLM 配置中注明该补丁仅在 torch‑2.11 使用 `torch.compile` 时启用，且在 torch‑2.12+ 已内置。  
4. **测试覆盖**：加入针对 `torch.compile(..., mode="max-autotune")` 的回归测试，验证 residual 不会被重复计算，并对比 2.11 与 2.12 两个环境的执行时间。  
5. **性能基准**：提供 benchmark 脚本，帮助用户确认在其硬件上是否真的受益，防止在某些 CPU‑GPU 组合下出现逆向效果。  

**结论**：该补丁解决了 vLLM 在 torch‑2.11 全图编译下的热点重复计算问题，提升了推理效率。但因直接修改 PyTorch 私有实现，需做好版本兼容、异常容错和回归测试，以防未来升级导致不可预期的行为。开发者应在发布说明中明确该行为的适用范围，并提供关闭开关供高级用户自行决定是否使用。
