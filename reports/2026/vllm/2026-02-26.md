### [Model] Add nvidia/llama-nemotron-embed-vl-1b-v2 multimodal embedding model (#35297)
**SHA**: `111d869` | 🔗 [查看提交](https://github.com/vllm-project/vllm/commit/111d8690699927af686fa6750cfbbc692a1f8740)

**🎯 变更类型**：功能增强 / 架构变更 / 多模态支持  
**⚡ 重要程度**：🔴 高  
**📋 变更摘要**：在 vLLM 中新增对 `nvidia/llama-nemotron-embed-vl-1b-v2` 多模态 embedding 模型的完整支持。包括文档、模型注册、配置校验、专用 Processor（使用 SigLIP 视觉编码器并加入归一化）、权重映射、池化输出以及对应的单元测试。目标是让用户能够通过 vLLM 的 embeddings 接口对文本、图像或文本+图像进行单向向量嵌入。

**🎯 影响范围**  
- `docs/models/*`（模型列表与使用示例）  
- `vllm/model_executor/models/*`（配置、权重加载、模型实现）  
- `vllm/model_executor/models/registry.py`（模型注册表）  
- `vllm/model_executor/models/nemotron_vl.py`（新增 `LlamaNemotronVLForEmbedding`）  
- `vllm/model_executor/models/config.py`（新增 `LlamaNemotronVLConfig`）  
- `tests/models/multimodal/pooling/`（新增针对该模型的对齐测试）  

---

### 🔍 技术洞察  

| 维度 | 影响分析 |
|------|----------|
| **架构影响** | • 引入 **LlamaNemotronVLForEmbedding**，在原有 `LlamaNemotronVLChatModel` 基础上实现 *Pooling*（embedding）接口。<br>• 新增 `LlamaNemotronVLEmbedProcessor`，专门处理 `<image>` token 替换并使用 **SigLIP** 的归一化 transform。<br>• `LlamaNemotronVLConfig` 在配置阶段强制 `is_causal=False`（双向），并统一池化方式。<br>• 权重映射 (`WeightsMapper`) 重新定义，以适配 SigLIP 与 Llama 的层级命名差异。 |
| **性能影响** | • **正向**：使用 **SigLIP** 视觉编码器相比原 C‑RADIO 在同等分辨率下计算略快（SigLIP 结构更轻），且 `DispatchPooler` 只返回单向量，降低后处理开销。<br>• **负向**：新增视觉前处理（`build_siglip_transform`）以及动态 patch 机制会在大批量图像输入时增加 CPU / GPU 预处理时间和显存占用。（每张图像的 patch 数仍受 `min_dynamic_patch / max_dynamic_patch` 限制）<br>• 默认仅在 `bfloat16` 上测试，未覆盖 `float16`/`int8` 量化路径，可能存在未发现的性能回退。 |
| **安全考虑** | • 文档和示例明确要求 `--trust-remote-code`，这意味着加载模型时会执行远程代码，增加潜在供应链攻击风险。<br>• 新增的 Jinja chat‑template 通过 `messages` 结构注入角色前缀，若服务端未正确校验 `role` 字段，可能导致意外的 prompt 注入（虽然影响有限）。 |
| **可维护性** | • 代码分层清晰：Processor、ProcessingInfo、模型主体、配置验证各自独立，便于后续扩展其它 SigLIP‑based 模型。<br>• 但在 `NemotronVLProcessor` 中对 `image_token_id`、`_replace_image_tokens` 等硬编码字符串仍保留，若未来模型改用不同 token，需同步多处修改。 |
| **兼容性** | • 新模型注册在 `supported_models.md` 和 `registry.py` 中加入，兼容已有的 `pooling` 接口。<br>• 通过 `test_llama_nemotron_vl_embed.py` 与 HF `encode_queries/documents` 对齐，保证功能一致性。<br>• 仍需在 `vllm/model_executor/models/config.py` 中为 `LlamaNemotronVLModel` 添加对应的 `model_type`（若 HF 新增），否则 `verify_and_update_model_config` 可能在未匹配时抛异常。 |

---

### ⚠️ 潜在风险  

1. **远程代码执行**：`trust_remote_code=True` 必须由运维方严格管控，否则恶意模型可能执行任意代码。  
2. **Token 替换冲突**：`_replace_image_tokens` 使用两层占位符 `<NVL_IMG_CONTEXT>`，若未来更改 `IMG_CONTEXT` 文字，可能导致未匹配或多余替换。  
3. **显存泄漏**：`extract_feature` 中对 `vision_model` 输出做 `to(torch.bfloat16)`，但在多卡/分片情况下未显式释放中间 `vit_embeds`，在高并发图像嵌入时可能出现显存峰值。  
4. **配置不完整**：若 HF 模型仓库缺失 `vision_config` 或 `llm_config`，`LlamaNemotronVLConfig` 会直接访问属性，可能抛 `AttributeError`。  
5. **量化路径缺失**：当前实现仅覆盖 `bfloat16`，而 `int8`/`awq` 量化模型在 `weight_mapper` 中未映射 SigLIP 相关层，导致加载失败。  

---

### 💡 关注建议  

- **安全**：在生产环境强制配合内部模型审计，或提供可选的本地化代码包装，尽量避免直接打开 `trust_remote_code`。  
- **测试**：补全 `float16`、`int8`、`awq` 等量化 dtype 的 CI 测试；在大批量图像（>32）场景下加压测，验证显存峰值与吞吐。  
- **文档**：在模型文档明确注明 `image_token_id`、`IMG_CONTEXT` 与 `role` 前缀的映射规则，防止用户自行拼接 prompt 时出现误差。  
- **代码维护**：抽象出 `image_token`、`IMG_START/END/CONTEXT` 常量至基类或配置结构，确保未来模型切换时只改动单点。  
- **错误处理**：在 `LlamaNemotronVLConfig.verify_and_update_model_config` 中加入 `hasattr` 与默认值保护，防止缺失字段导致启动失败。  
- **性能监控**：提供可选的 `--disable-image-preprocess` 标志，允许用户自行在外部完成图像归一化与 patch 化，以降低 CPU 端预处理开销。  

---  

**结论**：此提交为 vLLM 引入了重要的多模态嵌入能力，架构上实现了与语言模型、视觉模型的清晰解耦，并通过池化层提供统一的向量输出。风险主要集中在安全可信执行、显存管理以及对量化模型的兼容性上。通过上述建议可进一步提升可靠性与可维护性。

### [Model] Ring 2.5 (#35102)
**SHA**: `ab87f85` | 🔗 [查看提交](https://github.com/vllm-project/vllm/commit/ab87f85231b07b107f16ddb9e985deb0d83975ae)

**🎯 变更类型**：功能增强 / 重构 / 架构变更  
**⚡ 重要程度**：🔴高  

**📋 变更摘要**  
1. 为 vLLM 项目新增 **BailingMoeV2_5**（Ring‑2.5）模型实现，包含完整的 Linear‑Attention、MoE、以及混合全注意力/Linear‑Attention 的 decoder。  
2. 扩展 `layernorm_guard`，加入 `activation` 参数，使 LayerNorm 在有 gating（z）时支持 **swish / silu / sigmoid** 三种激活路径。  
3. 更新文档、模型注册表、测试以及模型配置转换，使新模型可被自动发现并加载。  

**🎯 影响范围**  
- **模型层**：`vllm/model_executor/layers/fla/ops/layernorm_guard.py`、`vllm/model_executor/layers/mamba/linear_attn.py`、新增 `vllm/model_executor/models/bailing_moe_linear.py`（约 1.2k 行）  
- **模型注册**：`vllm/model_executor/models/registry.py`、`vllm/transformers_utils/model_arch_config_convertor.py`  
- **文档 & 测试**：`docs/models/supported_models.md`、`tests/models/registry.py`

### 🔍 技术洞察  

| 维度 | 影响说明 |
|------|-----------|
| **架构影响** | • 引入 **Hybrid Attention**：每层可根据 `layer_group_size` 动态选择 Linear‑Attention（低延迟）或全注意力（高精度），实现 **层级混合**。 <br>• 新增 **MiniMax Text‑01** 线性注意力实现，使用 Triton kernel `linear_decode_forward_triton` 与 `jit_linear_forward_prefix`，并封装了缓存管理（`clear_linear_attention_cache_for_new_sequences`）。 <br>• 为 MoE 引入 `BailingMoeV25`、`BailingMoeV25MLAAttention`、`BailingMoeV25DecoderLayer`，并在 `BailingMoeV25Model` 中完成 **PP/TP** 的分层组装。 <br>• `layernorm_guard` 现在支持显式激活选择，兼容原 `swish`（默认）并可切换到 `sigmoid`，对已有模型向后兼容无影响（默认保持 `swish`）。 |
| **性能影响** | • **Linear‑Attention** 采用 Minimax/SG‑Lang 思路，理论上在长上下文（>4k）下显著降低 **O(N²)** 计算和显存开销，特别适配 **Ring‑2.5** 大模型（1T 参数）。 <br>• 新增的 `clear_linear_attention_cache_for_new_sequences` 在前缀（prefill）阶段仅清除首次出现的上下文缓存，避免不必要的全缓存清零，提升 decode‑only 场景的吞吐。 <br>• `layernorm_guard` 中激活分支的条件判断在 Triton kernel 中实现，仅在 `HAS_Z` 为真且激活匹配时执行额外乘法，开销可忽略。 <br>• 需要注意 **TP**（张量并行）下的权重切分逻辑（`_weight_loader`、`weight_direct_load`），若 shard 大小不匹配可能导致额外的复制开销。 |
| **安全考虑** | • 未引入新的网络交互，只是本地计算路径，安全风险低。 <br>• `tests/models/registry.py` 增加了对 `BailingMoeV2_5ForCausalLM` 的 `trust_remote_code=True` 标记，提醒使用者该模型加载时仍依赖远程代码执行，保持与已有模型一致的安全警示。 |
| **可维护性** | • 新增文件 `bailing_moe_linear.py` 包含数百行实现，代码结构遵循已有 `bailing_moe` 模块的风格，易于统一维护。 <br>• `layernorm_guard` 参数化后，激活方式统一在函数签名中控制，避免了硬编码，提高可扩展性。 <br>• 权重加载逻辑使用 `AutoWeightsLoader` + `default_weight_loader`，保持与 vLLM 其它模型的统一路径。 |

---

### ⚠️ 潜在风险  

1. **兼容性风险**  
   - 新增的 `activation` 参数默认 `swish`，但如果 downstream 代码误传其他值（如 `"relu"`）将触发未定义分支，导致运行时错误。  
   - `layernorm_guard` 的 Triton kernel签名改变，若旧的自定义编译缓存未重新编译，可能出现 kernel 参数不匹配的异常。  

2. **性能回归**  
   - Linear‑Attention 在 **prefill** 场景仍会遍历所有 prefill blocks；若 `clear_linear_attention_cache_for_new_sequences` 逻辑出错，可能导致缓存未被及时清空，进而出现显存泄漏或不必要的计算。  
   - 对 **TP** 分片的 `weight_loader` 没有显式检查 `shard_size` 能否整除张量维度，极端模型（奇数维）在 `tp_size>1` 时可能出现 shape mismatch。  

3. **测试覆盖不足**  
   - 新增模型的 end‑to‑end 推理/生成未加入 CI 基准，缺少对 **decode‑only** 与 **prefill+decode混合** 两种路径的性能/正确性验证。  
   - Triton kernel 依赖的 `torch.cuda` 环境在 CI（CPU‑only）上不可跑，导致潜在跨平台差异不易捕获。  

4. **权重加载复杂度**  
   - `load_weights` 中对 expert 参数的映射涉及多层 `replace`，若 HF checkpoint 的命名略有变化（如新增前缀）可能导致匹配失败，需手动补充映射。  

---

### 💡 关注建议  

| 对象 | 建议 |

### [torch.compile] Sequence Parallelism threshold compile ranges (#28672)
**SHA**: `9d37941` | 🔗 [查看提交](https://github.com/vllm-project/vllm/commit/9d379410179b649f4e7651940debc35c4ac7c0a5)

**🎯 变更类型**：功能增强 / 架构变更 / 重构  
**⚡ 重要程度**：🔴高  

**📋 变更摘要**  
新增了序列并行（Sequence Parallelism，简称 SP）阈值计算逻辑，并在编译配置、编译范围划分以及相应的测试套件中加入了对该阈值的支撑。主要改动包括：  
1. 在 `vllm/compilation/passes/fusion/sequence_parallelism.py` 中实现 `get_sequence_parallelism_threshold`，依据 GPU Compute Capability、模型 hidden_size、Tensor‑Parallel 大小以及数据类型字节数计算最小 token 数阈值；若模型过小则返回 `None`，自动禁用 SP。  
2. `PassConfig` 增加 `sp_min_token_num` 配置项，并在 `vllm/config/vllm.py`、`_set_compile_ranges` 中提前计算阈值、在必要时强制关闭 SP 与 async‑TP。  
3. `SequenceParallelismPass` 在 `is_applicable_for_range` 中加入阈值检查，仅在编译范围起点 `>= min_token_num` 时才激活 SP。  
4. 大量测试文件（`tests/compile/...`）同步更新，覆盖阈值计算、不同编译范围下的模式匹配以及 SP 与其他 Fusion 之间的优先级关系。  

**🎯 影响范围**  
- `vllm/compilation/passes/fusion/sequence_parallelism.py`（核心阈值实现 & pass 逻辑）  
- `vllm/config/compilation.py`（新增配置字段）  
- `vllm/config/vllm.py`（初始化时阈值计算、自动禁用逻辑）  
- `vllm/_set_compile_ranges`（在编译范围划分阶段插入 SP 阈值点）  
- 相关测试目录 `tests/compile/*`（新增/修改测试以验证新行为）  

**🔍 技术洞察**  

- **架构影响**  
  - 引入了 **设备能力感知** 的阈值判断，使 SP 的启用从“仅 TP>1”升级为“模型足够大且 GPU 支持”。这在运行时不再盲目打开 SP，降低了对不适用场景的错误依赖。  
  - `PassConfig.sp_min_token_num` 与 `VllmConfig` 的初始化过程耦合，使阈值在 **模型加载前** 即可决定是否打开 SP，避免在 Graph 编译后期才进行撤销，保持编译图的一致性。  
  - 编译范围划分 (`_set_compile_ranges`) 现在会在 `min_token_num - 1` 处插入切分点，形成 **“无 SP 区间”** 与 **“有 SP 区间”** 两段编译范围，确保 SP 只在满足阈值的 batch 大小上生效。  

- **性能影响**  
  - **正向**：对大模型（hidden_size ≥ 8192 on H100 等）且 batch 大小超过阈值的场景，SP 能显著降低 All‑Reduce 通信开销，提升吞吐。阈值计算基于每 GPU 最小内存需求（8 MiB），在 TP 增大时自动放宽，使得更大的并行度仍能受益。  
  - **负向**：在 **小模型** 或 **低 batch** 场景下，自动关闭 SP 可避免因切分/拼接带来的额外算子和同步开销，防止性能回退。  
  - **可预测性提升**：阈值决定了编译范围的切分点，用户可通过 `pass_config.sp_min_token_num` 手动覆盖，形成明确的性能调优入口。  

- **安全考虑**  
  - 变更仅影响功能与性能，不涉及外部输入的安全检查。唯一潜在安全关注是 **配置误用**：若用户手动设置极低的 `sp_min_token_num`，可能导致在不满足 SP 前提的情况下强行开启 SP，进而触发 **符号维度分割不支持** 的运行时错误。已通过 `SequenceParallelismPass.is_applicable_for_range` 继续检查 compile_range 是否符合 TP 整除要求，降低此类风险。  

- **可维护性**  
  - 将阈值计算抽离为独立函数 `get_sequence_parallelism_threshold`，并配备完整单元测试，提高了代码可读性和后续扩展（如添加其它 GPU 族阈值）的便利性。  
  - 新增的配置字段 `sp_min_token_num` 与已有的 `enable_sp`/`fuse_gemm_comms` 保持相同的 `PassConfig` 结构，保持一致的配置路径。  
  - 测试覆盖面广（包括阈值边界、不同 TP、不同 element_size 场景），有助于防止回归。

### fix(mxfp4): Disable monolithic path for TRITON backend with EP (#34270)
**SHA**: `c97234c` | 🔗 [查看提交](https://github.com/vllm-project/vllm/commit/c97234c08b42326cf1e5ef024d9ac8441e0848b1)

**🎯 变更类型**：功能增强 / Bug修复  
**⚡ 重要程度**：🔴高  
**📋 变更摘要**：  
1. 修复了在开启专家并行（Expert Parallelism，EP）时，TRITON 后端的 `triton_kernel_moe_forward` 使用全局专家 ID 导致非法内存访问的问题。  
2. 通过在检测到 `expert_map`（EP 场景）时，先执行 `topk` 选取，再用 `expert_map` 将全局 ID 映射为本地 ID，随后构造本地路由结构；`is_monolithic` 对 TRITON 后端始终返回 `True`，因为内部已完成 EP‑aware 的路由重映射。

- `vllm/model_executor/layers/quantization/mxfp4.py`（`Mxfp4MoEMethod.is_monolithic` 判定）  
- 新增测试文件 `tests/kernels/quantization/test_mxfp4_triton_ep.py`（CI 覆盖）  

**🔍 技术洞察**：

- **架构影响**  
  - **专家并行兼容**：原先的 `legacy_routing` 只能处理全局专家 ID，无法在 EP 场景中映射到本地权重张量。此次改动在 `triton_kernel_moe_forward` 中引入了 “分离路由” 流程，使 TRITON 后端在 EP 环境下仍保持单体（monolithic）执行模型，避免了为 EP 专门维护另一套代码路径。  
  - **后端抽象统一**：`is_monolithic` 对 TRITON 后端统一返回 `True`，简化了上层调度逻辑，减少了 backend‑specific 条件分支。  

- **性能影响**  
  - **正向**：通过在 EP 场景下直接在 GPU 上完成 `topk` + `expert_map` 重映射，省去了在 CPU 上额外的映射或复制步骤，理论上可略微降低延迟。  
  - **负向**：新增一次 `topk` 调用和一次 `make_routing_data`，相较于直接走 `legacy_routing` 多了两次 kernel 调度（`topk` 与 `make_routing_data`），在极端小 batch/专家数情况下可能增加开销，但对生产环境中常见的大 batch/专家数影响微乎其微。  

- **安全考虑**  
  - 主要风险在于 **索引越界**：原 bug 正是因全局 ID 与本地权重维度不匹配导致非法访问。新实现通过 `expert_map` 映射后确保所有 ID 均在 `[0, local_num_experts)` 范围内，安全性得到提升。  
  - 代码中新增的 `effective_expert_map` 与 `effective_global_num_experts` 仍保持可为空或默认值的路径，未泄露额外信息。  

**⚠️ 潜在风险**：

| 风险点 | 说明 | 缓解措施 |
|--------|------|----------|
| `expert_map` 为 `None` 时仍走 `legacy_routing` | 与旧行为保持一致，若 `legacy_routing` 本身存在未捕获的异常，则仍会泄露。 | 通过现有单元测试验证两条路径的等价性；建议在 CI 中加入异常注入测试。 |

| `local_num_experts = w1.size(0)` 假设 `w1` 第0维一定是专家维度 | 若后端实现改变维度顺序，可能导致错误。 | 维持现有约定并在文档中明确说明；在未来若改动层实现，需要同步更新此处。 |
| 代码路径分支导致 **is_monolithic** 与实际执行路径不一致的潜在误导 | `is_monolithic` 仍返回 `True`，但内部已经做了 EP‑aware的拆分，理论上仍是单体执行，但在监控/日志层面可能需要更新描述。 | 在日志/监控中记录 `effective_expert_map` 是否为 `None`，便于诊断。 |

**💡 关注建议**：

1. **增加负样本测试**：在 CI 中加入 `expert_map` 包含 `-1`（无效专家）或超出范围的情况，确保代码在异常输入下能安全返回或抛出可控异常。  
2. **文档更新**：在 `Mxfp4MoEMethod.is_monolithic` 与 `triton_kernel_moe_forward` 的 docstring 中说明：“TRITON 后端在 EP 场景下仍视为 monolithic，内部已完成 expert_map 重映射”。  
3. **监控/日志**：在 `triton_kernel_moe_forward` 开头添加轻量日志，记录 `expert_map` 是否被使用以及本地专家数量，以便在生产环境追踪 EP 的实际行为。  
4. **性能基准**：建议在不同 EP 大小（1, 2, 4, 8）以及不同 batch size 下跑一次微基准，确认新增 `topk`+`make_routing_data` 的开销在可接受范围；若发现显著回退，可考虑把两段逻辑合并成单一 kernel（如在 `topk` 完成后直接生成 routing）。
