### [7/n] Migrate pos_encoding and norm kernels to libtorch stable ABI (continued) (#43209)
**SHA**: `a7be0f3` | 🔗 [查看提交](https://github.com/vllm-project/vllm/commit/a7be0f342dd376bf8234eb1b7397be7a79be8f42)

**🎯 变更类型**：功能增强 / 重构 / 架构变更  
**⚡ 重要程度**：🔴 高  

**📋 变更摘要**  
1. 将位置编码（pos_encoding）与层归一化（norm）相关的 CUDA / HIP kernel 移动到 `csrc/libtorch_stable/`，统一使用 **libtorch stable ABI**（`torch::stable::Tensor`、`torch::headeronly::ScalarType`）。  
2. 新增 `dispatch_utils.h` 中的多维度 **宏调度**（量化类型、向量大小、张量秩、组大小），实现无 ATen 依赖的纯 C++ 分支，提升编译兼容性。  
3. 更新 CMakeLists 以编译新文件，移除旧的 `csrc/*.cu` 直接引用。  
4. 扩展 `ops.h` 与 `torch_bindings.cpp`，在稳定 ABI 侧注册新的算子（rms_norm、fused_add_rms_norm、rotary_embedding、fused_qk_norm_rope 等），并在 CUDA/ROCm 分支移除旧实现。  
5. 统一错误检查为 `STD_TORCH_CHECK`，并使用 `torch::stable::accelerator::DeviceGuard` 与 `get_current_cuda_stream` 代替 ATen 版本。

---

### 🎯 影响范围  

| 模块/组件 | 受影响文件 | 说明 |
|----------|-----------|------|
| **核心算子实现** | `csrc/libtorch_stable/*`（pos_encoding、layernorm、layernorm_quant、quantization/fused_kernels） | 迁移至 stable ABI，所有原有 kernel API 改为 `torch::stable::Tensor`。 |
| **构建系统** | `CMakeLists.txt` | 删除旧 `.cu` 文件引用，新增 `csrc/libtorch_stable/...`，确保 libtorch‑stable 目标编译。 |
| **调度工具** | `csrc/libtorch_stable/dispatch_utils.h` | 新增量化、向量、秩、组大小宏，供所有 kernel 使用。 |
| **Python 绑定** | `csrc/torch_bindings.cpp`、`csrc/libtorch_stable/torch_bindings.cpp` | 注册 / 实现 Stable ABI 算子；旧 CUDA‑only 注册被移除。 |
| **公共头文件** | `csrc/type_convert.cuh`、`csrc/quantization/.../quant_conversions.cuh` | 替换为 header‑only half/BFloat16，兼容 ROCm/CUDA。 |
| **CPU 侧** | `csrc/ops.h`、`csrc/torch_bindings.cpp`（CPU 部分） | 保留 `torch::Tensor` 声明，仅在 CPU 编译时使用，保持向后兼容。 |
| **其他** | `csrc/quantization/fused_kernels/fused_silu_mul_block_quant.cu` 等 | 更新 include 路径，确保使用 libtorch‑stable 量化工具。 |

### 🔍 技术洞察  

| 维度 | 影响 |
|------|------|
| **架构影响** | - **ABI 统一**：使用 `torch::stable` 把 CUDA 与 CPU 代码抽象到同一层，实现一次实现、多平台使用。<br>- **模块化**：`dispatch_utils.h` 将原先散落在各 kernel 的 `VLLM_DISPATCH_*` 替换为 `VLLM_STABLE_DISPATCH_*`，统一调度逻辑，降低耦合。<br>- **构建简化**：CMake 通过 `VLLM_STABLE_EXT_SRC` 集中管理 stable kernel，避免旧文件残留导致的二进制冲突。 |
| **性能影响** | - **运行时**：kernel 逻辑未变，仍使用相同的 launch 配置（grid/block、token_heads_per_warp 自动选择），理论性能保持不变。<br>- **编译时间**：`dispatch_utils.h` 中的纯 C++ 分支（如 `VLLM_STABLE_DISPATCH_GROUP_SIZE`、`VLLM_STABLE_DISPATCH_VEC_SIZE`）避免了 ATen 宏展开，显著降低了 CUDA 编译依赖的模板实例化次数，尤其在多平台（CUDA & ROCm）编译时收益明显。<br>- **内存访问**：使用 `mutable_data_ptr` / `const_data_ptr` 替代 `data_ptr`，明确读写意图，帮助编译器更好地执行常量传播与寄存器分配。 |
| **安全考虑** | - **错误检查**：统一为 `STD_TORCH_CHECK`（基于 header‑only 实现），在无 ATen 环境下仍能提供完整的异常信息，未引入新的安全风险。<br>- **设备守卫**：改为 `torch::stable::accelerator::DeviceGuard`，保证在多 GPU 环境下正确切换上下文，降低出现 “wrong stream/device” 的概率。 |
| **可维护性** | - **代码一致性**：全部 kernel 采用同一套宏 (`VLLM_STABLE_DISPATCH_*`)，后期添加新类型或新维度只需在一个文件中实现。<br>- **跨平台**：`#ifdef USE_ROCM` 分支提供 FP8‑e4m3fnuz 支持，兼容 ROCm 7.x+。<br>- **文档/绑定**：`torch_bindings.cpp` 中的 `ops.def` 与 `ops.impl` 同步更新，保持 Python API 与底层实现的一一对应。 |
| **兼容性** | - **CPU 构建**：保留旧 `torch::Tensor` 接口在 `csrc/ops.h`，CPU 仍使用传统 ABI，避免破坏现有纯 CPU 环境。<br>- **旧 ABI**：CMake 已移除旧 kernel 编译路径，若用户自行编译旧代码库会报缺失符号，提示切换到 stable。 |

### ⚠️ 潜在风险  

1. **ABI 不匹配**  
   - 若外部项目仍链接旧 `libvllm`（使用 ATen `Tensor`），在运行时会出现符号冲突或数据布局不匹配。  
   - 建议在发布说明中明确 **从 0.3.0+ 开始所有 CUDA/HIP kernel 必须使用 stable ABI**。  

2. **宏调度错误**  
   - 新的 `VLLM_STABLE_DISPATCH_GROUP_SIZE`、`VLLM_STABLE_DISPATCH_VEC_SIZE` 依赖 **constexpr** 参数。若传入不在 `{2,4,8,16}` 或 `{64,128}` 范围，会走默认路径，可能导致性能下降或错误。需要单元测试覆盖所有分支。  

3. **ROCm 与 FP8 兼容性**  
   - 在 ROCm 上 `Float8_e4m3fnuz` 不可用，宏已做 `#ifdef` 区分，但若用户在 ROCm 环境误传 `Float8_e4m3fnuz`，会触发编译错误。确保 Python API 对应的 dtype 检查在 Python 层完成。  

4. **DeviceGuard 与 Stream**  
   - 新的 `get_current_cuda_stream` 是 libtorch‑stable 的内部工具，若未来 libtorch 版本更改该实现，可能导致 **stream 获取失败**。保持对 libtorch 版本的绑定，并在 CI 中锁定对应 commit。  

5. **异常信息差异**  
   - `STD_TORCH_CHECK` 与原先 `TORCH_CHECK` 的错误信息格式略有不同，可能影响已有监控或日志解析规则。  

6. **CMake 路径变更**  
   - 对应的 `install` 脚本或 packaging (wheel) 可能仍引用旧的 `.cu` 路径，导致 **runtime missing symbol**。需要在 `setup.py` / `CMakeLists` 中同步更新 install 目标。

**⚡ 重要程度**：🔴 高  

**📋 变更摘要**

| **兼容性** | - 新增的 `enable_chunked_prefill=False` 检查在 **vllm/config/vllm.py** 抛异常，保证已有用户在启用该特性时得到明确错误提示。<br>- 旧版测试与示例已同步关闭 `chunked_prefill`，保持 CI 通过。<br>- 对外 API (`output.kv_transfer_params["hidden_states_path"]`) 未变，保持向后兼容。 |

### ⚠️ 潜在风险
1. **锁文件残留**：若写入线程因未捕获异常提前退出，`.lock` 文件可能未被删除，导致后续读取一直阻塞。  
2. **磁盘空间耗尽**：隐藏状态文件默认保存在 `/tmp`（或用户指定目录），大模型、长序列会产生大量 `.safetensors`，需要运维监控磁盘使用。  
3. **线程池饱和**：`num_writer_threads` 过小会成为瓶颈，过大则可能导致过多上下文切换，需根据硬件调优。  
4. **跨进程共享**：若使用 `multiprocessing` 启动多个 vLLM 实例共享同一 `shared_storage_path`，文件锁仍有效但可能出现 **锁竞争**，影响吞吐。  
5. **依赖版本**：`safetensors` 低于 0.8.0 会在 `save_file` 持有 GIL，显著降低写入并行度，文档已给出警告，但仍需用户注意。  

---

### 💡 关注建议
- **监控 & 清理**：在生产部署中加入定时任务清理 `shared_storage_path`，或在 `cleanup_hidden_states` 中根据 `keep_hidden_states=False` 自动删除。  
- **调优线程数**：依据磁盘 I/O 带宽与 CPU 核数，使用 `KVTransferConfig.kv_connector_extra_config["num_writer_threads"]` 进行实验性调优。  
- **使用 RAM‑disk**：在线场景强烈建议将 `shared_storage_path` 指向 `/dev/shm` 或类似内存文件系统，以最大化吞吐并降低延迟。  
- **安全审计**：确认 `shared_storage_path` 目录的访问权限，仅限信任的服务账号；防止恶意用户写入或读取模型内部状态。
