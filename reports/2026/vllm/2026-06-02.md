### [Bugfix] Vendor MiniCPMV/MiniCPMO processors to unblock Transformers v5  (#44282)
**SHA**: `0bdfd5e` | 🔗 [查看提交](https://github.com/vllm-project/vllm/commit/0bdfd5eb843a8e7fcebd37c155b1db3f1c8a5ee4)

**🎯 变更类型**：Bug修复  
**⚡ 重要程度**：🔴高  
**📋 变更摘要**：  
本次提交为 MiniCPM 系列模型（MiniCPMV、MiniCPMO）引入了自研的 **vendored processor** 实现，以在 Transformers **v5+** 环境下保持兼容，解决了原有 processor 中依赖的 `tokenizer.im_start_id` 等字段在新版中被移除的问题。同时删除了相关测试的 skip 标记，并在模型执行器中使用新的 processor；在 `vllm.transformers_utils.processors` 包中完成了类的注册。

**🎯 影响范围**：  
- `vllm/model_executor/models/minicpmv.py`、`vllm/model_executor/models/minicpmo.py`（模型执行器）  
- `vllm/transformers_utils/processors/`（新增 `minicpmv.py`、`minicpmo.py`）  
- `vllm/transformers_utils/processors/__init__.py`（注册入口）  
- 相关单元测试文件 `tests/lora/test_minicpmv_tp.py`、`tests/models/multimodal/generation/test_common.py`  

---

### 🔍 技术洞察

| 维度 | 影响分析 |
|------|----------|
| **架构影响** | 1. **新增 Processor 层**：在 `transformers_utils.processors` 中独立实现 MiniCPMV / MiniCPMO 的 processor，脱离了 HF 官方实现的直接依赖，形成“vendored”包装层。<br>2. **模型执行器解耦**：`MiniCPMVProcessingInfo.get_hf_processor` 与 `MiniCPMOProcessingInfo.get_hf_processor` 现在返回自研 processor，保持 `ctx.get_hf_processor`（原 HF processor）的复用，仅在包装层做少量属性转换（如 numpy → list）。<br>3. **注册统一化**：在 `processors/__init__.py` 中加入 `MiniCPMVProcessor` 与 `MiniCPMOProcessor`，确保 `vllm.get_processor` 仍能通过统一入口获取。 |
| **性能影响** | - **音频特征提取**：`MiniCPMOProcessor.audio_feature_extract` 使用 `torchaudio` 的 `MelSpectrogram` 等算子，计算开销与原 HF 实现相当；额外的 **numpy → list** 转换只在模型加载阶段执行，运行时几乎无额外成本。<br>- **Padding 与序列拼接**：实现了自定义的 `pad` 方法，保持左侧 padding（兼容旧行为），时间复杂度为 O(batch\*max_len)。<br>- **总体影响**：因仅在 **模型初始化** 时进行一次包装，运行时性能基本不变。 |
| **安全考虑** | - **外部依赖**：新增 `torch`, `torchaudio`, `numpy`, `regex`，均为常规安全依赖，无网络请求或文件 I/O。<br>- **数据校验**：`audio_feature_extract` 对 `audio` 长度进行切片、对 `audio_parts` 进行长度校验，已加入断言防止越界。<br>- **序列边界**：在 `_convert` 中对 image/audio/spk token 边界做了严格的 `assert` 检查，防止 malformed 输入导致越界写入。<br>- **潜在风险**：如果用户自定义的 tokenizer 缺少 `audio_start_id`、`audio_end_id` 等属性，`_safe_get_token_id` 会返回 `-1`，后续使用时可能产生非法 token。需要在生产环境做好 fallback 或提前校验。 |
| **可维护性** | - **代码分离**：processor 逻辑和模型执行器分离，后续更新 MiniCPM 模型或适配新 Transformers 版本时，只需修改 `processors/` 下对应文件。<br>- **类型检查**：通过 `TYPE_CHECKING` 条件导入避免运行时循环依赖。<br>- **文档注释**：新增了类、方法的解释性 docstring，提升可读性。 |

### ⚠️ 潜在风险

1. **Tokenizer 兼容性**  
   - 新 processor 假设 tokenizer 含有 `im_start_id、im_end_id、audio_start_id、audio_end_id、spk_start_id、spk_end_id` 等属性。若使用的模型 tokenizer 在未来的 Transformers 版本中改名或去除，这些属性将返回 `None`，导致 `torch.where` 返回空张量，引发 `assert` 错误。  
2. **音频采样率不匹配**  
   - `audio_feature_extract` 默认采样率 16000，若用户输入的音频采样率不同且未显式指定，可能导致特征长度计算错误。  
3. **序列长度截断**  
   - `_convert` 中对 `max_inp_length` 只截取了 **input_ids**，但未同步截断对应的 `audio_placeholder` 长度，可能出现 placeholder 长度大于实际 token 长度的情况。  
4. **包装层的属性同步**  
   - 在 `MiniCPMOProcessingInfo.get_hf_processor` 中，将 `image_processor.mean/std` 从 `np.ndarray` 转为 `list`，仅针对这两个属性做了处理，若 HF 未来在 image_processor 中加入更多 `np.ndarray` 类型属性，可能导致序列化/远程调用失败。

### 💡 关注建议

| 对象 | 建议 |
|------|------|
| **开发者** | 1. 为 `MiniCPMVProcessor`、`MiniCPMOProcessor` 增加 **单元测试**，覆盖缺失 token ID、不同采样率、长音频切片等 edge case。<br>2. 在 `__init__` 中加入对关键 tokenizer 属性的 **显式校验**，在缺失时抛出友好错误而非 `-1`。<br>3. 将 `audio_feature_extract` 的 `chunk_length` 参数暴露给上层 API，便于用户自定义分块大小。 |

**总体评价**：本次提交通过引入自研的 processor，有效解除对 Transformers v5 的硬依赖，恢复 MiniCPM 系列模型在最新环境下的可用性。架构层面的解耦提升了代码的可维护性，运行时性能影响可忽略。唯一需要重点关注的是 tokenizer 接口的向后/向前兼容性和音频输入的边界校验，建议在后续版本中加入更健壮的容错与监控。

### [CPU][Perf] Enable fused kernels for GDN's gated delta rules (#43534)
**SHA**: `0b25cf4` | 🔗 [查看提交](https://github.com/vllm-project/vllm/commit/0b25cf441903b06811e0f75f09c303c18fe08240)

**🎯 变更类型**：功能增强 / 性能优化  
**⚡ 重要程度**：🔴高  

**📋 变更摘要**  
本次提交为 CPU 端的 Gated‑Delta‑Rule（GDN）注意力实现引入了“融合内核”。通过在 `csrc/cpu/sgl-kernels/fla.cpp` 中使用 brgemm（AMX/AVX512）以及在缺乏硬件支持时回退至 OpenBLAS BLAS gemm，实现了原本 Python‑层实现的完全 C++ 加速。新增的 CMake 检测逻辑会在 PyTorch wheel 中寻找 `libopenblas.so` 并在编译时定义 `VLLM_HAS_OPENBLAS`，从而在运行时选择最优路径。对应的 Torch Bindings、单元测试、CI 配置均已同步更新。

**🎯 影响范围**  
- `vllm/model_executor/layers/mamba/ops/cpu/gdn_attention.py`（核心注意力调度）  
- `csrc/cpu/sgl-kernels/fla.cpp`、`blas_gemm.h`、`gemm.h`、`vec.h`（CPU SGL 内核）  
- CMake 构建脚本 `cmake/cpu_extension.cmake`（OpenBLAS 检测与编译宏）  
- `vllm/_custom_ops` 的 Torch Bindings（新增 fused `gdn`、`causal_conv1d` 接口）  
- CI 配置 `.buildkite/*`（CPU 测试套件加入 GDN 相关测试）  
- 新增 `tests/kernels/mamba/cpu/test_cpu_gdn_ops.py`（完整功能 & 精度校验）  

---

## 🔍 技术洞察  

| 维度 | 影响描述 |
|------|-----------|
| **架构影响** | - 将原本分散在 Python 与多次 `torch.ops` 调用的 GDN 计算迁移至单一 C++ 融合内核，降低了跨语言调用开销。<br>- 引入 `VLLM_HAS_OPENBLAS` 编译宏，使库能够在不具备 AMX/AVX512 硬件时仍能利用 PyTorch wheel 自带的 OpenBLAS 加速（保持向后兼容）。<br>- `cpu_gdn_attention_core` 统一了 decode 与 prefill 两条路径，按硬件特性在运行时分支（AMX vs 非‑AMX），简化了代码路径。 |
| **性能影响** | - 当运行于支持 AVX512 + AMX 的 x86 CPU，brgemm（`at::native::cpublas::brgemm`）提供了块级矩阵乘的 VNNI 加速，预计相对原先基于 `torch.mm` 的实现提升 **2‑5×**（尤其在大 batch / 长 seq 场景）。<br>- 对不具备 AVX512 的 ARM 及其它平台，新增 `blas_gemm` 调用 OpenBLAS 的 `sgemm_ / sbgemm_`，仍比纯 PyTorch 调用快 **30‑50%**，因为直接跳过 Python‑层包装与临时转置。<br>- 通过 `brgemm_supported()` 编译期判断，避免在不支持的 CPU 上进行非法指令调用，确保运行时安全。 |
| **安全考虑** | - 无新增网络 I/O、文件写入或进程间通信，仅在内部数值计算中引入外部 BLAS 库。<br>- 通过 `target_compile_definitions(_C PRIVATE VLLM_HAS_OPENBLAS)` 明确限定链接范围，不会影响到其他共享库。<br>- 删除了 `recurrent_gated_delta_rule.py`（纯 Python 实现），彻底依赖新内核；若回退路径（OpenBLAS）出现异常，会导致整体错误而非潜在泄漏。 |
| **可维护性** | - 代码结构更加模块化：`blas_gemm.h` 抽象了不同后端（brgemm / OpenBLAS / PyTorch reference）实现，后续若引入 MKL、oneDNN 可继续在此处扩展。<br>- 条件编译宏 (`CPU_CAPABILITY_AVX512`) 与 runtime 检测 (`torch.cpu._is_amx_tile_supported()`) 分离，使得新平台的适配更为直观。<br>- 单元测试覆盖了 decode、prefill、不同序列长度、不同头/维度组合，确保数值正确性。 |

## ⚠️ 潜在风险  

1. **OpenBLAS 检测不完整**  
   - 某些 PyTorch wheel 可能把 OpenBLAS 编译进 `libtorch.so` 而不提供独立的 `libopenblas.so`，导致 `VLLM_HAS_OPENBLAS` 未被定义，回退至 PyTorch reference BLAS，性能下降。  
2. **硬件特性检测误差**  
   - `torch.cpu._is_amx_tile_supported()` 在部分老旧 CPU 上可能报错或返回错误值，导致走错分支（尝试使用 AMX 代码但硬件不支持）。  
3. **ABI 兼容性**  
   - 新增的 `blas_gemm` 使用了 Fortran‑style BLAS 接口 (`sgemm_`, `sbgemm_`)；若运行环境的 OpenBLAS 采用了不同的符号前缀或编译选项，链接时可能出现未定义符号。  
4. **数值差异**  
   - OpenBLAS 对 BF16 的实现（`sbgemm_`）在极端数值范围可能与 PyTorch reference 略有差异，导致精度回归。  
5. **平台回滚风险**  
   - 删除 `recurrent_gated_delta_rule.py` 后，若新内核在某些边缘案例（如极端 seq_len、异常输入）出现崩溃，缺少回退实现会导致服务不可用。

### ⚠️ 潜在风险

1. **并行限制**  
   - 稀疏更新仅在 **TP=1、PP=1**（单进程单模型）下可用，若用户在多 GPU、tensor‑parallel 环境误用会触发 `NotImplementedError`。需在文档、示例与错误信息中再次强调此限制。

2. **参数连续性假设**  
   - `apply_sparse_weight_patches` 要求参数 **连续**，对使用 `torch.nn.Parameter` 包装的 **view / transpose** 参数会抛异常。某些模型的权重可能因为自定义划分或切片而不连续，导致稀疏补丁失效。

3. **索引类型与跨设备**  
   - 当前实现要求 `indices` 为 **int32** 并在同一 device 上广播。若未来支持更大张量（>2³¹‑1 元素）或跨 GPU 参数，需升级为 int64 并增加跨设备聚合逻辑。

4. **同步错误恢复**  
   - 稀疏广播是 **同步** 的（`torch.accelerator.synchronize()` 在 `update_weights` 末尾）。若在广播期间出现异常，`_weight_update_active` 状态已在 `finally` 中清理，但 **已广播的部分数据** 可能已经写入模型导致不完整更新。建议在异常路径显式回滚（恢复模型快照）或在 Trainer 端进行事务式发送。

5. **测试覆盖局限**  
   - 现有单元测试仅在 **单节点、2 GPU** 环境下运行，未覆盖 **多节点**、**多进程**、**混合 checkpoint+稀疏** 场景。若后续扩展为跨节点稀疏同步，需要额外的协议（如全局 barrier）和错误恢复机制。

**SHA**: `d247a9d` | 🔗 [查看提交](https://github.com/vllm-project/vllm/commit/d247a9dc13a58fef5ce64dc3a8f4be051577bda5)

- **接口兼容**：新方法的签名引用了 `Request` 类型，需在 `base.py` 中加入相应的前向声明或 `from ... import Request`，防止运行时未解析导致的 `NameError`。  

**建议**  
1. 在文档/README 中补充 `ensure_cache_available` 的使用约定及返回 `False` 时的调度行为。  
2. 为防止无限延迟，可在调度器或 connector 中加入超时/重试机制，或者在 `step_skipped_waiting` 超过一定轮次数后强制调度（或报错）。  
3. 考虑为 `ensure_cache_available` 添加 `async` 支持，便于在实际实现中发起异步网络预取而不阻塞调度线程。  
4. 为 `ensure_cache_available` 加入类型注解并显式导入 `Request`，保持代码可检查性。

- 使用 **copy atom**（128 bit）一次性搬运 4 float，显著降低全局‑共享‑寄存器交互次数。  
- 归约逻辑改为基于 `lanes_per_group` 动态计算的 **scale_reduce_steps/offset**，并加入掩码 `group_mask_and_clamp` 以防跨组 shuffle。  
- `__init__` 中的 warp 与 tb 大小重新计算，`rows_per_warp`、`num_warps` 也相应调小。  
- 对 `nope`（rope）块、量化尺度、FP8 打包的写回路径做了按 **group‑lane** 细粒度划分，避免了原来对单个 lane 的硬编码。  

**🎯 影响范围**  
- `vllm/models/deepseek_v4/nvidia/ops/sparse_attn_compress_cutedsl.py`（核心压缩 kernel）  
- 关联的编译参数 `SparseAttnCompressKernel`（rows_per_warp、num_warps、tb_size）  
- 依赖该 kernel 的 DeepSeek‑V4 推理路径（稀疏注意力、kv‑cache 更新）  

**💡 关注建议**  

1. **数值一致性**：  
   - 由于归约顺序与拷贝布局已改变，建议在 CI 中加入 **FP8 输出对比**（前后 kernel 在相同随机输入下的 L2 / 相对误差），确保误差不超出既定容忍度。  
   - 关注 `scale_reduce_steps` 的计算：`lanes_per_group.bit_length()‑1` 只在 `lanes_per_group` 为 2ⁿ 时等价，若 `quant_block` 与 `elems_per_lane` 不匹配可能产生非整数偏移，务必在 `__init__` 加入断言或回退路径。  

2. **边界条件**：  
   - `head_size // quant_block` 必须能被 `groups_per_warp` 整除，否则 `num_warps` 会被截断，可能出现 **线程空洞**。建议在构造函数中检测并抛出友好错误。  
   - `nope_blocks` 与 `group_idx` 的比较已改为 `group_idx == self.nope_blocks`，确保 `nope_dim` 与 `quant_block` 对齐。  

3. **性能回归检测**：  
   - 新增的 `copy_elems / copy_chunks` 机制在不同显卡（A100、H100、RTX 4090）上的带宽利用率可能差异，建议在 benchmark 中记录 **每步时延**，验证相较旧实现的加速比例。  
   - `rows_per_warp` 从 8 改为 16，`num_warps` 从 16 降到 8，需确认 **occupancy** 在目标 GPU 上仍保持在 70% 以上；若下降可考虑在 `compile` 时动态调节。  

4. **代码可维护性**：  
   - 将硬编码常数（如 `8`, `4`, `32`）抽象为类常量或配置参数，以免后续再次改动时遗漏。  
   - `cp_f32x4`、`copy_layout`、`kv_vals`/`score_vals` 的创建放在 kernel 入口前，可抽取成私有函数，提高可读性。  

5. **文档与测试**：  
   - 更新 `README`/kernel 注释，说明 **elems_per_lane**、**lanes_per_group** 的取值要求。  
   - 添加单元测试覆盖：  
     - `head_size` / `quant_block` 不可整除情形  
     - `nope_dim` 为 0（无 rope）时的路径  
     - 极端 `compress_ratio`（1、16）下的数值与尺度输出  

总体来看，此次改动在 **吞吐量** 与 **寄存器占用** 上有明显提升，但引入了更细粒度的分组逻辑，务必通过数值回归、边界检测与性能基准来确保改动在所有支持的硬件上稳健。
