### [Frontend] [Parser] Port DeepSeek V4 to streaming parser engine framework (#45877)
**SHA**: `fb5291b` | 🔗 [查看提交](https://github.com/vllm-project/vllm/commit/fb5291b35b0b1b1ec5840438da0f62f127417710)

**🎯 变更类型**：功能增强 / 重构 / 安全修复  
**⚡ 重要程度**：🔴高  

**📋 变更摘要**  
- 为 DeepSeek 系列新增两套解析器：`DeepSeekV4Parser`（支持 `<think>` 推理标签）和 `DeepSeekV32Parser`（仅工具调用，使用 `function_calls` 包裹）。  
- 在解析引擎层实现 **engine‑adapter**（`DeepSeekV4ParserReasoningAdapter`、`DeepSeekV4ParserToolAdapter`、`DeepSeekV32ParserReasoningAdapter`、`DeepSeekV32ParserToolAdapter`），并在 **注册中心**统一暴露。  
- 统一 **工具解析器** 为 `DeepSeekV4EngineToolParser`、`DeepSeekV32EngineToolParser`，替代旧实现。  
- 改进 **流式解析**：  
  - 正确处理 **EOS / DROP** token（即 token ID 存在但其文本被 detokenizer 剥离）的情况，防止工具调用被意外丢弃。  
  - 在 `ParserEngine` 中加入 `_has_reasoning` 与 `_reasoning_ended` 初始状态逻辑，解决 Reasoning‑End 失效的 bug。  
  - `Adapter.finish_streaming` 现在在跳过工具解析的情形下安全返回 `DeltaMessage`，避免因 `skip_tool_parsing` 失效导致的空返回。  
  - `TokenIDScanner._rebuild_from_anchors` 增加对 `DROP_TERMINAL` 的特殊处理，防止因 EOS 文本缺失导致内容延迟。  
- 引入 **参数包装拆解**（`_unwrap_wrapper_args`），在工具调用中自动把 `{ "arguments": {...} }` 或 `{ "input": {...} }` 展平，兼容 DeepSeek V4/ V3.2 的包装方式。  
- 完善 **类型协同**：当 `string="true"` 与 JSON Schema 冲突时，依据 schema 自动强制转换（如 `"42"` → `42`），并对 `null`、布尔、数组、对象等做统一处理。  
- 大量 **测试** 新增与更新，覆盖非流、流式、并行工具、推理、缺失结束标签、包装参数、EOS‑DROP 场景等，确保新解析器在各种边缘情况下的正确性。  

**🎯 影响范围**  
- `vllm/parser/`（`deepseek_v4.py`、`deepseek_v32.py`、`engine/*`、`reasoning/*`）  
- `vllm/tool_parsers/`（engine 插件包装、注册表）  
- `vllm/reasoning/`（reasoning parser 映射）  
- `vllm/parser/engine/token_id_scanner.py`、`streaming_parser_engine.py`（EOS/DROP 处理）  
- `tests/parser/engine/`、`tests/tool_parsers/`、`tests/reasoning/`（新增/更新）  
- 公开 API `ParserEngine`、`ToolParserManager`、`ReasoningParserManager` 的行为在 DeepSeek V4/ V3.2 场景下被扩展。

### [Bugfix][Structured Output][Spec Decode] Constrain bitmask and trim grammar advance at the reasoning boundary (#44297)
**SHA**: `e7c9df9` | 🔗 [查看提交](https://github.com/vllm-project/vllm/commit/e7c9df944969c251bdceed7b9d7bbcc7fec3639b)

**变更概览**  
本次 PR 主要修复 **Structured Output + Spec‑Decode** 场景下的几个边界错误：  
1. 在 `Scheduler.update_from_output` 中，新增 `trim_reasoning_for_advance`，在调度器直接推进 grammar 前剔除推理阶段的标记及其之前的 token，防止 grammar 误拒绝。  
2. `StructuredOutputManager` 的 `grammar_bitmask` 重构，加入对 **reasoning‑end marker** 的检测与处理：  
   - 当标记出现在窗口内部时，后续草稿仍需生成 bitmask，但不再对标记本身进行 grammar advance。  
   - 通过 `should_fill_bitmask`、`detect_reasoning_end`、`post_reasoning_end_in_window` 等控制路径，实现 **mid‑window reasoning 结束** 的约束。  
3. 新增 `_find_reasoning_end_index` 与 `trim_reasoning_for_advance` 辅助函数，记录并裁剪推理结束位置。  
4. `StructuredOutputRequest` 新增 `reasoning_end_token_index` 用于保存该边界索引。  
5. 大量新增单元测试，覆盖 -1 padding、EOS 中止、mid‑window marker、invalid drafts、bitmask 幂等性等情形。

**影响范围**  
- `vllm/v1/core/sched/scheduler.py`（请求状态更新）  
- `vllm/v1/structured_output/__init__.py`（bitmask 计算、reasoner 接口）  
- `vllm/v1/structured_output/request.py`（数据结构）  
- 相关测试文件（`test_async_scheduler.py`, `test_scheduler.py`, `test_mtp_structured_output.py`）  

**关注建议**  

1. **兼容性**：`reasoning_end_token_index` 默认 `None`，旧代码仍可运行。但若外部自行实现 `StructuredOutputRequest` 子类，需确保该字段被保留，否则 `trim_reasoning_for_advance` 可能返回原始 token，引发重复错误。  

2. **性能**：`grammar_bitmask` 中对每个 token 可能会复制历史 `request.all_token_ids` 并构造 `simulated_buf`，在极大 batch 或高频 speculative decoding 场景下会产生额外的列表拷贝。建议监控该路径的时间开销，必要时改为基于索引的滑动窗口实现。  

3. **错误路径**：若 `grammar.accept_tokens` 在 `advance_grammar` 为 `False` 时仍抛出异常，当前实现会直接 `raise AssertionError`。生产环境可以改为记录日志并安全跳过，以免因单个不合法 draft 使整个请求失败。  

4. **测试覆盖**：新增的 `test_mtp_structured_output.py` 已覆盖主要边界，但仍缺少 **diffusion 模型** 与 **多 token reasoner**（需要组合标记）的回归。建议补充对应组合的测试，确保 `is_reasoning_end_streaming` 在多个 token 才触发时也能正确截断。  

5. **文档**：在 `vllm/docs` 中补充 “Structured Output + Speculative Decoding” 的使用说明，特别是 `enable_in_reasoning` 参数的含义以及何时会触发 `trim_reasoning_for_advance`。  

综上，此次改动针对 #44006、#42452 的回归问题提供了完整的修复，核心逻辑清晰且已通过单元测试验证。注意关注上述性能与兼容性细节，适当补充文档与更广泛的回归测试后即可合并。

### [ROCm][CI][Bugfix] Use VllmRunner for `voxtral_realtime` tests to avoid OOM on AMD GPU (#47536)
**SHA**: `f329ce4` | 🔗 [查看提交](https://github.com/vllm-project/vllm/commit/f329ce405b12623fb8b1cf1830f12e5a712523be)

**🎯 变更类型**：Bug 修复  
**⚡ 重要程度**：🟡 中  

**📋 变更摘要**  
- 将 `voxtral_realtime` 测试从直接使用 `LLM/AsyncLLM` 实例改为通过 `VllmRunner` 启动模型，以在 AMD ROCm 环境下避免 OOM。  
- 新增对 ROCm 平台的显存回收等待 (`wait_for_rocm_memory_to_settle`) 与有条件的 `shutdown` 超时。  
- 对 `engine`/`async_engine` fixture 进行重构，统一使用 `AsyncEngineArgs`、`torch._dynamo.reset`、分布式清理等步骤。  

**🎯 影响范围**  
- **tests/models/multimodal/generation/test_voxtral_realtime.py**（测试逻辑、fixture、清理流程）  
- **vllm/platforms**（`current_platform.is_rocm()` 判定）  
- **vllm/distributed**（`cleanup_dist_env_and_memory`）  
- 可能间接受影响的 CI 脚本（ROCm CI 需要显存回收等待）。  

**💡 关注建议**  
1. **显存回收**：`wait_for_rocm_memory_to_settle` 采用 `threshold_ratio=1.0 - gpu_memory_utilization`，务必确保 `ENGINE_CONFIG` 中的 `gpu_memory_utilization` 与实际卡容量匹配，防止误判导致测试仍然 OOM。  
2. **资源释放**：`llm.shutdown(timeout=…)`、`torch._dynamo.reset()`、`cleanup_dist_env_and_memory()` 必须保持顺序，防止残留进程或显存泄漏。若后续加入更多平台（如 CUDA），请将相同逻辑抽象为统一的 `shutdown_helper`。  
3. **环境变量**：仍保留 `VLLM_ENABLE_V1_MULTIPROCESSING=0`，若以后不再依赖单进程模型，请同步移除或在文档中说明。  
4. **测试兼容性**：当前改动仅在 ROCm 环境下加入等待逻辑，非 ROCm 环境仍使用原有路径，确认 CI 在不同平台均通过。建议在 CI 中加入显存占用阈值的验证，以防回归。  
5. **代码可维护性**：`vllm_runner` 的使用方式已经在其他测试中复用，建议抽取为公共 fixture（如 `vllm_runner`）以统一管理启动/关闭参数，降低重复代码。  

总体而言，此次改动解决了 AMD GPU 上的 OOM 问题，逻辑清晰且对资源回收做了加强。后续关注跨平台统一的显存回收策略即可。

**🎯 影响范围**：`vllm/config/compilation.py`、`vllm/config/vllm.py`、`vllm/v1/worker/gpu/cudagraph_utils.py`、`vllm/v1/worker/gpu/model_runner.py`、文档 `dynamic_speculative_decoding.md`、以及新增的测试文件。
