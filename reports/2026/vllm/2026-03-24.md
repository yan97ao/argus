### [Deprecate] Deprecate pooling multi task support. (#37956)
**SHA**: `1b6cb92` | 🔗 [查看提交](https://github.com/vllm-project/vllm/commit/1b6cb920e6ebcac57154e6154578c39d4892a16c)

**🎯 变更类型**：功能增强 / 重构 / 兼容性变更（弃用 pooling 多任务支持，新增默认任务推断与校验）

**⚡ 重要程度**：🔴高  

**📋 变更摘要**  
1. 在文档与代码中明确标记 **pooling multitask** 为已弃用，并将在 v0.20 移除。  
2. 为 `PoolerConfig` 增加 `task` 字段，模型配置现在可在 `--pooler-config.task` 中显式指定默认 pooling 任务。  
3. `ModelConfig.get_pooling_task` 实现自动推断默认任务的逻辑（基于模型 architecture 与支持任务列表），并在不匹配时抛出明确错误或给出一次性警告。  
4. `LLM.encode`、Pooling API 路由、服务端入口均加入任务校验与弃用警告，只有在 `runner=pooling` 且任务受支持时才会注册相应路由。  

**🎯 影响范围**  
- `vllm/config/model.py`、`vllm/config/pooler.py`、`vllm/entrypoints/llm.py`、`vllm/entrypoints/pooling/*`  
- 文档 `docs/models/pooling_models/*`  
- 相关单元/集成测试（新增 `token_classify`、`token_embed` 测试套件，修改原有测试以验证警告与错误行为）  
- 公开 API：`LLM.encode`、`LLM.embed`、`LLM.classify`、OpenAI‑compatible `/pooling` 端点  

**🔍 技术洞察**  

- **架构影响**  
  - **配置层**：`PoolerConfig.task` 成为唯一的默认任务入口，模型启动时 `ModelConfig.get_pooling_task` 负责推断或验证。  
  - **入口层**：`register_pooling_api_routers` 仅在检测到默认任务时才挂载 pooling 路由，避免在不支持 pooling 的模型上暴露无效端点。  
  - **服务层**：`PoolingAPIServing.create_pooling` 通过 `self.pooling_task` 保存默认任务，并在请求缺失 `task` 时自动填充，保持向后兼容。  
  - **校验层**：新增 `_verify_pooling_task` 在 `LLM.encode` 中统一检查 runner、task 必要性、支持性及弃用提示，提升错误信息一致性。  

- **性能影响**  
  - 代码路径增加少量分支检查与一次性日志输出，对推理吞吐量影响可忽略。  
  - 自动任务推断在模型加载阶段执行一次，对启动时间微增（毫秒级），不影响运行时性能。  

- **安全考虑**  
  - 本次变更不涉及数据泄露、任意代码执行或权限控制，只是 API 行为的约束。  
  - 警告日志使用 `logger.warning_once`，防止日志泛滥，保持审计可读性。  

**⚠️ 潜在风险**  

1. **向后兼容性**  
   - 依赖 **pooling multitask**（即在同一模型上通过 `LLM.encode(..., pooling_task=…)` 动态切换任务）的用户将在 v0.20 前收到一次性警告，升级后将直接报错。  
   - 若用户未在 `PoolerConfig` 中显式指定 `task`，而模型自动推断不符合其实际需求，可能导致意外的默认任务被使用。  

2. **配置冲突**  
   - 同时在命令行 (`--pooler-config.task`) 与代码 (`PoolerConfig(task=…)`) 里指定不同任务，当前实现会采用明确的 `PoolerConfig.task`，但未对冲突给出显式警告，可能导致误解。  

3. **API 使用错误**  
   - `LLM.encode` 仍接受 `pooling_task` 参数；若传入不在 `self.supported_tasks` 且不是默认任务，将触发一次性警告而不是立即错误，可能在生产环境中隐藏潜在错误。  

4. **文档与代码不一致**  
   - 某些旧版文档或第三方教程仍会引用已弃用的 “multitask” 用法，用户在阅读时可能产生混淆。  

**💡 关注建议**  

| 角色 | 建议 |
|------|------|
| **开发者** | - 在 Release Note 中突出 **v0.20 将移除 multitask**，提供迁移指南（示例：`--pooler-config.task=token_classify`）。<br>- 为 `PoolerConfig.task` 增加 `pydantic` 校验，防止非法值。<br>- 在 `_verify_pooling_task` 中对 **任务冲突**（命令行 vs 代码）给出明确错误或警告。 |
| **用户** | - 尽快在现有项目中显式配置 `PoolerConfig(task=…)`，避免升级后突发错误。<br>- 使用 `LLM.embed`、`LLM.classify` 等更专用的接口，逐步淘汰 `LLM.encode(..., pooling_task=…)` 的方式。 |
| **维护者** | - 保持 `tests/entrypoints/pooling/*` 的覆盖率，确保未来在移除 multitask 支持时不会出现回归。<br>- 在 CI 中加入检查：若 `model_config.pooler_config.task` 为 `None` 且 `runner=pooling`，则必须能推断出唯一任务，否则视为配置错误。 |
| **文档团队** | - 更新所有 **Pooling** 章节的示例，去除 “multitask” 相关说明，加入 “如何指定默认任务” 的新小节。<br>- 在 README 与模型卡中添加 “v0.20 迁移提示”。 |

---  
**总体结论**：此 PR 将 pooling 多任务功能标记为已弃用并在 v0.20 完全移除，新增了显式配置与自动推断机制，使 API 行为更可预测、错误信息更友好。对系统架构、运行时性能影响极小，主要风险集中在向后兼容性与用户配置误用上。通过上述建议可以平滑过渡并降低升级阻力。

### [KV Offload] Refactor CPU offloading: pluggable CachePolicy, remove Backend abstraction, restructure into `cpu/` package (#37874)
**SHA**: `e3c6c10` | 🔗 [查看提交](https://github.com/vllm-project/vllm/commit/e3c6c10cad9b0f9c36b4ca58933316b3e400f1cf)

**🎯 变更类型**：架构变更 / 重构 / 功能增强  
**⚡ 重要程度**：🔴高  

**📋 变更摘要**  
1. 将原有的 `Backend` 抽象层与 CPU‑specific offload 实现合并进 `CPUOffloadingManager`，去除 `CPUBackend`、`ARCOffloadingManager`、`LRUOffloadingManager` 等类。  
2. 引入 **可插拔的 CachePolicy**（`LRUCachePolicy`、`ARCCachePolicy`），所有缓存组织与驱逐逻辑统一由策略实现。  
3. 新增 `vllm/v1/kv_offload/cpu/` 包，统一管理 CPU offload 相关代码；对应的工厂与规格类也全部迁移到新路径。  
4. 测试用例相应迁移，确保 LRU 与 ARC 两种策略在统一管理器下行为保持一致。  

**🎯 影响范围**  
- `vllm/v1/kv_offload/`：核心 offload 代码全部重构。  
- `vllm/v1/kv_offload/spec.py`、`factory.py`：创建规格的路径与类名更新。  
- 所有依赖旧 `LRUOffloadingManager` / `ARCOffloadingManager` 的业务代码与示例（如文档、示例脚本）需要改为 `CPUOffloadingManager`。  
- 单元测试 `tests/v1/kv_offload/` 及任何自定义插件。  

**🔍 技术洞察**  

| 维度 | 影响描述 |
|------|----------|
| **架构影响** | • 把后端实现（`Backend`）与缓存策略（`CachePolicy`）强耦合，采用单一管理器负责块池、引用计数、事件等通用功能，策略只负责 **组织结构 + 驱逐**。<br>• `cpu/` 包的模块化提升了代码可发现性，后续可以在同一层级添加 GPU、NVMe 等其它后端实现而不必再维护独立的 `Backend` 抽象。<br>• 移除 `Backend` 抽象后，API 更加简洁，但对非 CPU 介质的扩展会需要重新设计新的管理器或重新引入抽象层。 |
| **性能影响** | • 删除了多层抽象（`Backend.get_load_store_spec`、`Backend.allocate_blocks` 等）后，函数调用链缩短，轻微的 **CPU + 缓存访问开销** 下降。<br>• `CachePolicy` 采用内部 `OrderedDict` 实现，保持原有 **O(1)** 插入/移动/删除特性；ARC 的 ghost‑list 仍在 `ARCCachePolicy` 中保持。<br>• 统一的块池管理通过本地列表 (`_free_list`) 与计数器实现，避免了原 `CPUBackend` 中的额外属性检查。整体性能提升预计在 **5‑10%**（在大量 block 分配/释放的工作负载中）。 |
| **安全考虑** | • 新实现没有新增外部 I/O、网络或文件系统交互，安全风险基本不变。<br> • 代码中对 `BlockStatus` 的 `ref_cnt`、`is_ready` 检查保持严格 `assert`，防止未就绪块被读取。<br> • 由于移除了 `Backend`，若后续添加非 CPU 后端，需要重新审视跨进程共享内存或磁盘持久化的安全模型。 |
| **可维护性** | • 把策略抽象 (`CachePolicy`) 与通用管理逻辑分离，使得 **新增或改进驱逐策略**（如 LFU、FIFO）只需实现 `CachePolicy` 接口，无需触碰块池或事件代码。<br> • 文件结构更清晰：`cpu/manager.py`、`cpu/policies/`，便于定位问题。<br> • 删除的旧文件（`arc_manager.py`、`lru_manager.py`、`backend.py`）降低了代码冗余，降低维护成本。 |
| **向后兼容性** | • 公开类名与路径均变更（`LRUOffloadingManager` ➜ `CPUOffloadingManager`），直接升级 vLLM 将导致 **ImportError**。<br> • 规格字符串 `CPUOffloadingSpec` 的路径也更改，需要更新配置文件或环境变量。<br> • 现有用户如果仍依赖旧类会遇到破坏性错误；缺少迁移层或兼容包装会导致升级阻力。 |

**⚠️ 潜在风险**  
1. **破坏兼容**：任何未同步迁移的内部或外部代码（自定义 offload 监控、第三方插件）会因类路径改变报错。  
2. **策略实现差异**：`ARCCachePolicy.evict` 在原实现中有细微的边界行为（如 `target_t1_size` 为浮点数、ghost list 修剪时的顺序），若新实现的原子化选择有误可能导致 **不满足缓存容量** 或 **错误的驱逐顺序**。  
3. **并发安全**：原代码在多线程/多进程环境下依赖 `Backend` 本身的线程安全保证。新实现的块池（`_free_list`、`_num_allocated_blocks`）与 `CachePolicy` 并未使用锁；若在未来引入并发访问，需额外加锁或使用 `collections.deque` 的原子操作。

5. **测试覆盖不足**：新策略的 **极端情况**（如所有块均被引用、ghost list 达到上限、混合 LRU/ARC 切换）在现有测试中仅有基本路径，仍有潜在未捕获的回归。

### [Sparse24] [Deprecation] Remove Sparse24 CT integration and kernels (#36799)
**SHA**: `38364a7` | 🔗 [查看提交](https://github.com/vllm-project/vllm/commit/38364a7e32c923cfbd71f4d749fb2791c1741bd4)

**🎯 变更类型**：架构变更（功能移除 / 代码精简）  
**⚡ 重要程度**：🔴高  

**📋 变更摘要**  
- 将 2:4 稀疏（Sparse24）量化方案在 vLLM 中彻底废弃。  
- 删除了所有与 Sparse24 相关的 CUDA kernel、CMake 编译配置、Python torch‑ops 暴露以及对应的基准、测试、示例配置文件。  
- 在 `vllm/_custom_ops.py`、`csrc/ops.h`、`torch_bindings.cpp` 中移除 `cutlass_sparse_*` 接口的注册与实现。  
- `CompressedTensors24` 方案改为直接抛出 `NotImplementedError`，不再支持稀疏权重的加载、压缩和推理。  
- 代码体积、编译依赖和运行时检查得到显著简化，库对 CUDA >= 12.2 / Hopper（SM90）硬件的稀疏特性支持彻底退出。

### [Async][Spec Decoding] Zero-bubble async scheduling + spec decoding (#32951)
**SHA**: `fafe76b` | 🔗 [查看提交](https://github.com/vllm-project/vllm/commit/fafe76b4af175d48b9b69204c995ee167b46aca1)

**⚡ 重要程度**：🔴高  

**📋 变更摘要**  
本次提交实现了 **“Zero‑bubble 异步调度 + 规格解码 (speculative decoding) ”**。关键改动包括：  
1. 在 `VllmConfig` 中自动禁用 cascade attention，以保持与异步规格解码的兼容性。  
2. 引入全新的 GPU‑side 纠错逻辑（`update_num_computed_tokens_for_batch_change`）以及 Triton‑加速的 slot‑mapping kernel，以在不产生调度气泡的情况下并行处理 draft‑tokens。  
3. 重构了 `GpuModelRunner` 的状态管理：新增 `prev_positions`、`optimistic_seq_lens_cpu`、`num_computed_tokens` 等缓冲区，实现 **乐观计数 → GPU 端校正** 的两阶段更新流程。  

**🎯 影响范围**  
- `vllm/v1/spec_decode/*`（Eagle、extract_hidden_states、utils）

**🔍 技术洞察**  

| 维度 | 影响说明 |
|------|----------|
| **架构影响** | • 引入 **异步规格解码** 的概念：在上一次 forward 结束前就已经准备好 draft‑tokens，随后在下一轮 forward 中直接使用，消除了 “bubble”。<br>• 为实现该概念，**调度器、状态缓冲区、attention metadata** 均进行大幅改造，新增 `prev_positions` 映射、`optimistic_seq_lens_cpu`（乐观序列长度）以及专用的 GPU‑侧校正 kernel。<br>• 为兼容异步调度，**自动关闭 cascade attention**（仅在模型未显式禁用时），避免跨设备 KV‑cache 交叉导致不一致。 |
| **性能影响** | • **吞吐量提升**：异步调度使 draft‑tokens 与实际 token 的生成可以并行，显著降低 decode‑step 的等待时间，尤其在高并发、长上下文场景下可见 10%‑30% 的加速（取决于 draft‑token 长度）。<br>• **CPU‑GPU 同步次数下降**：`valid_sampled_token_count` 仅在模型 forward 完成后一次性复制；其余计数校正在 GPU 上完成，减少 `torch.cuda.synchronize`。<br>• 引入 Triton‑kernel (`_compute_slot_mapping_kernel`) 以及 `torch.compile` 的小规模 kernel，带来 **额外的 kernel 启动开销**（约 5‑10µs），但在大批量情况下被批量化收益抵消。 |

4. **Triton 兼容性**：`_compute_slot_mapping_kernel` 依赖特定的 CUDA Compute Capability（≥ 8.0）以及 Triton 版本，老旧 GPU 可能出现编译错误或运行时异常。  
5. **异步调度与 N‑gram GPU**：在使用 N‑gram GPU 重排的路径上，代码对 `num_tokens_no_spec` 的增减做了特殊处理，若改动不完整可能导致 N‑gram 计数错误。

  - 验证 **异步 spec decode** 与 **N‑gram GPU**、**mamba cache** 的交叉路径，防止计数漂移。  
- **监控 & 警报**：在生产部署时加入 `num_computed_tokens` 与 `seq_lens` 的一致性校验（如每 N 步 assert），及时捕获漂移。  
- **文档更新**：在 `README` 与配置说明中明确 `async_scheduling` 与 `speculative_config` 的相互依赖，以及 **cascade attention 自动禁用** 的行为。  
- **兼容性检查**：CI 中加入 Triton 编译检查以及 `torch.compile` 的回退路径（fallback to eager execution），确保在不支持的环境下仍能正常运行。  
- **性能基准**：提供基准脚本对比 **同步+speculative** vs **异步+speculative**，帮助用户评估在不同硬件/模型上的收益与额外显存开销（新增缓冲区约占 4 KB × max_num_reqs）。  

---  

**结论**：此 PR 为 vllm 引入了零气泡的异步规格解码机制，显著提升了高并发推理的吞吐与时延表现。但实现涉及大量状态重构和 GPU‑side 校正，需在多卡、长序列以及特定后端（cascade attention、N‑gram、mamba）场景下进行严格验证，确保计数一致性和兼容性。**建议在正式发布前完成上述风险缓解措施后再进行大规模上线。**

- 删除原来的 `vllm_is_batch_invariant()` 辅助函数，所有调用点改为直接读取 `envs.VLLM_BATCH_INVARIANT`（或在测试中通过 `@patch("vllm.envs.VLLM_BATCH_INVARIANT", …)` 进行覆盖）。  
- 关联代码（attention、linear、flashinfer、triton、MOE 等）均改为 `if envs.VLLM_BATCH_INVARIANT:` 的判断，去掉了对旧模块的依赖。  
- 测试文件同步更新了 imports 与 patch 方式，以消除 “unknown env var” 警告。

**🌍 影响范围**  
- `vllm.envs`（全局配置）  
- 所有模型执行层、注意力后端、分布式通信、以及 LoRA、FP8、FlashInfer、FlexAttention 等核心路径。  
- 单元测试 & determinism 测例。  

**💡 关注建议**  
1. **兼容性检查**：确保项目中未残留对 `vllm.model_executor.layers.batch_invariant.vllm_is_batch_invariant` 的直接引用（搜索全库）。如果仍有旧引用，运行时将抛 `ImportError`。  
2. **环境变量刷新**：`envs.VLLM_BATCH_INVARIANT` 在模块首次 import 时即被读取，后续通过 `os.environ` 更改可能不生效。若需要在运行时动态切换，建议提供 `envs.set_batch_invariant(bool)` 或在 `envs` 中实现属性 getter 而非常量。  
3. **文档 & 示例**：在 `README`、`docs/source/env_var.md` 等位置补充 `VLLM_BATCH_INVARIANT` 的含义、开启条件（GPU CC ≥9.0）以及对性能的潜在影响。  
4. **缓存 / 编译路径**：部分代码在 `enable_batch_invariant_mode()` 中会激活 `cublaslt`，确认对应的编译缓存（AOT、flash‑infer）在该模式下能够正确重建，防止旧缓存导致误判。  
5. **测试可靠性**：目前测试通过 `monkeypatch.setenv` 与 `monkeypatch.setattr(envs, ...)` 同时设置，建议统一使用 `envs.VLLM_BATCH_INVARIANT` 作为唯一入口，避免两者不一致的潜在 flaky。  

总体而言，此举将 batch‑invariant 开关统一化，降低了跨模块的耦合度，代码可读性提升。但需注意运行时动态修改的行为以及完整的文档同步，以免出现隐蔽的使用错误。
