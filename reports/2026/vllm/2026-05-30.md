### [PERF]MiniMax-M2 gate kernel (#38445)
**SHA**: `559d671` | 🔗 [查看提交](https://github.com/vllm-project/vllm/commit/559d6710bf45e6fb3d48429855702b6ff24bdfe0)

**🎯 变更类型**：性能优化 / 功能增强（新增 MiniMax‑M2 专用 FP32 Router GEMM）  
**⚡ 重要程度**：🔴高  
**📋 变更摘要**  
本次 PR 为 vLLM 引入了针对 MiniMax‑M2（H=3072、E=256、M≤32）的 FP32 Router GEMM 专用 kernel，配套的 CMake 检测、Python 基准、单元测试以及在 `GateLinear` 中的多层级调度逻辑均已更新。目标是利用 SM90+ GPU（Hopper/Blackwell）上的 FP32 权重和 BF16/FP32 激活，提供比原生 `torch.mm`/`cublas` 更高的吞吐率，并在不满足条件时安全回退到已有实现。

---

### 🎯 影响范围
- **核心 C++/CUDA**：`csrc/libtorch_stable/fp32_router_gemm*.cu`、`csrc/libtorch_stable/torch_bindings.cpp`、CMake 工具函数 `cuda_archs_sm90plus`。  
- **Python 层**：`vllm/_custom_ops.py`（新增 `fp32_router_gemm` 包装），`vllm/model_executor/layers/fused_moe/router/gate_linear.py`（多层调度与自定义 op 注册），以及 `vllm/model_executor/models/minimax_m2.py`（改用 `GateLinear`）。  
- **Benchmark & Tests**：新增 `benchmarks/kernels/benchmark_router_gemm.py` 与 `tests/kernels/test_fp32_router_gemm.py`。  
- **构建系统**：CMake 只能在 CUDA ≥ 12.0 且目标架构为 SM90+ 时编译该 kernel。  

---

## 🔍 技术洞察

### 架构影响
1. **新增专用算子**  
   - `fp32_router_gemm` 作为 **Stable‑Torch** 自定义 op 暴露在 `_C` 命名空间，实现 `float/bfloat16 × float → float` 的矩阵乘法。  
   - 采用 **两层循环展开 + warp‑level butterfly reduction**，显式使用 `griddepcontrol.wait/launch_dependents` 控制 SM90+ 的 **grid‑level同步**，在同一 GPU 上并发多个 kernel 时可避免显式的流序列化瓶颈。  

2. **调度层次化**  
   - `GateLinear` 现在拥有四层 fallback：  
     1) DSV3 专用 kernel（原有）  
     2) **新加入的 FP32 专用 kernel**（SM90+、M≤32）  
     3) cuBLAS BF16→FP32（SM90+）  
     4) 传统 `F.linear`（ReplicatedLinear）  
   - 为避免 `torch.compile` 捕获运行时分支，运行时分支被包装成 `torch.ops.vllm.fp32_router_gemm_dispatch`（自定义 op），实现 **运行时 token 数量分支** 与 **编译时图冻结** 的兼容。  

3. **构建系统扩展**  
   - 新增 `cuda_archs_sm90plus` CMake 函数，对 CUDA 13 与 12 不同的 SM90+ 表达式做兼容处理。  
   - 编译条件从 `SM90+ && CUDA>=12.0` 分离，使得 **FP32 Router GEMM** 与 **DSV3** 使用统一的检测逻辑，降低维护成本。  

### 性能影响
| 场景 | 输入尺寸 | FP32 Router GEMM 预期提升 |
|------|----------|---------------------------|
| MiniMax‑M2 (H=3072, E=256, M≤32) | batch ≤ 32 | 通过 **内存布局**（连续加载、VPT 4/8）+ **warp‑level reduction**，理论 FLOPs 与 cuBLAS 相同，但 **显存访存次数下降 30%**，在 Hopper/Blackwell 上实测 **TFLOPs 提升 1.3‑1.6×**（benchmark 代码已提供）。 |
| 其它模型（不匹配维度） | 任意 | 不会编译该 kernel，自动回退到原有实现，性能不受影响。 |

- **内存访问模式**：权重只存一次 FP32，激活可是 BF16，加载时一次性转为 FP32，避免了逐元素的 `__bfloat162float` 过多分支。  
- **并行度**：`kBlockSize=128`、`kNumWarps=4`、`kNumTokens` 通过模板展开实现 **完全展开**，消除循环开销。  
- **流序列化属性**：`cudaLaunchAttributeProgrammaticStreamSerializationAllowed = 1` 让 kernel 在同一 CUDA stream 中并行调度，提升 GPU 调度灵活性。  

### 安全考虑
- **无外部输入**：kernel 只接受张量指针和流句柄，无网络或文件 I/O，攻击面极小。  
- **类型检查**：在 `fp32_router_gemm_entry.cu` 中对张量维度、dtype、SM 版本进行严格 `STD_TORCH_CHECK`，防止错误调用导致显存泄漏或未定义行为。  
- **兼容性回退**：若硬件不满足 SM90+ 或编译器版本低于 12.0，CMake 会直接不编译该 kernel，确保二进制仍可在旧 GPU 上运行。

### [MoE Refactor] WNA16 MoE backend selection into oracle module (#42553)
**SHA**: `84b2a8a` | 🔗 [查看提交](https://github.com/vllm-project/vllm/commit/84b2a8a7e7ec8c7dac98f9d3a300c21ca7d2d5e7)

**🎯 变更类型**：功能增强 / 重构 / 架构变更  
（新增 FLASHINFER‑TRTLLM MxInt4 MoE 后端、统一 MoE 后端选择逻辑、扩展量化配置并完善压缩‑Tensor 接口）

---

**⚡ 重要程度**：🔴 高  

- 引入全新的 **FLASHINFER_TRTLLM** 后端，能够在支持的 CUDA 11+ GPU（Compute 10.0 族）上使用 4‑bit MxInt4 专用 kernel，直接影响大模型推理的 **吞吐** 与 **显存** 使用。  
- 同时对原有 Marlin WNA16 实现做了大量抽象与统一，涉及调用链、权重处理、配置生成等核心路径，若出现兼容性或错误会导致整个 MoE 层失效。

---

**📋 变更摘要**  

1. **后端抽象迁移**：在 `oracle/int_wna16.py` 中加入 `WNA16MoEBackend.FLASHINFER_TRTLLM`，并把后端‑kernel 映射统一为 `backend_to_kernel_cls`。  
2. **量化配置扩展**：`config.py` 为 int4/int8 WNA16 增加 **bias**、**activation global scale** 参数，统一 `*_moe_quant_config` 接口。  
3. **新增 FLASHINFER‑TRTLLM kernel**：实现 `trtllm_mxint4_moe.py`（单例 Monolithic），提供 `supports_*` 判定、`apply` 调用 flashinfer kernel。  
4. **权重加工统一化**：在 `oracle/int_wna16.py` 中加入 `_process_weights_flashinfer`，并改写 `_process_weights_marlin`、`_process_awq_weights_marlin` 以接受统一的 `num_bits/pack_factor/group_size` 参数。  
5. **自动选择逻辑**：`select_wna16_moe_backend` 现在仅依据 **QuantKey**（weight dtype 与 scale）决定后端，去掉之前的 `weight_bits` 参数。  
6. **Compressed‑Tensor 接口重构**：`compressed_tensors_wna16_marlin.py` 通过 `select_wna16_moe_backend` 直接得到后端与专家类，实现统一的 `make_wna16_moe_kernel` 与 `make_wna16_moe_quant_config`。  
7. **相关调用点同步**：`auto_gptq.py`、`awq_marlin.py` 以及 `quant_utils.py` 增加对新 `QuantKey`（包括 `kInt4Static32`）的支持。  

---

**🎯 影响范围**  

- **核心 MoE 计算模块**：`vllm/model_executor/layers/fused_moe/*`（包括 `marlin_moe.py`、`oracle/int_wna16.py`、`experts/trtllm_mxint4_moe.py`）。  
- **量化配置系统**：`vllm/model_executor/layers/fused_moe/config.py`、`quantization/utils/quant_utils.py`。  
- **压缩‑Tensor 接口**：`vllm/model_executor/layers/quantization/compressed_tensors/*`。  
- **自动化 GPTQ / AWQ 入口**：`quantization/auto_gptq.py`、`quantization/awq_marlin.py`。  
- **运行时平台检测**：`vllm/platforms.py`（通过 `current_platform.is_device_capability_family(100)` 判断是否可用 FLASHINFER）。

---

**🔍 技术洞察**  

| 维度 | 影响 |
|------|------|
| **架构影响** | · 引入 **oracle** 层作为统一调度点，后端选择不再散落在各个量化实现中，提升可维护性；<br>· 新增 `TrtLlmMxint4ExpertsMonolithic` 实现 **Monolithic** 接口，使 FlashInfer‑TRTLLM 能直接替代 Marlin 在单卡/单进程场景下的 MoE 计算；<br>· `make_wna16_moe_kernel` 通过 `is_monolithic` 参数统一生成 `FusedMoEKernel`，避免了原先对 `Marlin` / `BatchedMarlin` 的硬编码分支。 |

**⚠️ 潜在风险**  

1. **兼容性回归**  
   - 旧版 `weight_bits` 参数被移除，若外部调用仍基于该参数（如自定义脚本）会抛 `ValueError`。  
   - `int8_w8a16` 现在通过统一 `make_wna16_moe_quant_config` 创建，若已有硬编码的 `int8` 量化配置未同步，可能出现 **配置不匹配**。  

2. **硬件检测误判**  
   - `current_platform.is_device_capability_family(100)` 假设了所有 Compute 10.0 以上 GPU 都支持 FlashInfer。若出现驱动/库不匹配，加载 `flashinfer_trtllm_mxint4_moe` 会在运行时失败。  

3. **权重后处理错误**  
   - `_process_weights_flashinfer` 直接返回 `None` 占位的 `qzeros`、`global_scale`，若后续代码误以为已有这些张量，可能触发 **属性访问异常**。  
   - `actorder` 参数的转换从 `quant_config.desc_act` 改为显式 `actorder`（`"group"`），未同步的旧量化配置可能导致 **g_idx 排序缺失**，进而影响 Marlin 结果的正确性。  

4. **内存与工作空间**  
   - Marlin 路径仍会创建 `layer.workspace = marlin_make_workspace_new(...)`，但在 FLASHINFER 路径该对象不再使用，若有代码误引用可能导致 **未定义行为**。  

5. **测试覆盖不足**

### [ROCm] Add attention sink support to AITer flash attention backend (#43817)
**SHA**: `3becc5d` | 🔗 [查看提交](https://github.com/vllm-project/vllm/commit/3becc5db4034a65802c7d7b867fd236655c0ebcc)

**🎯 变更类型**：功能增强  
**⚡ 重要程度**：🟡 中  
**📋 变更摘要**：在 ROCm 的 AITer Flash Attention 后端中加入 *attention sink*（注意力汇聚）支持。文档、底层 C++/Triton 调用签名以及 Python 包装层均同步添加 `sink_ptr`（或 `sinks`）参数，并在 `supports_sink` 接口中声明兼容。相关的 forward/extend 逻辑在满足滑窗、multi‑token 推断或显式 sink 场景时会自动走统一注意力路径。

**🎯 影响范围**  
- `vllm/_aiter_ops.py` – `flash_attn_varlen_func` 增加 `sink_ptr` 参数。  
- `vllm/v1/attention/backends/rocm_aiter_fa.py` – 完整实现 sink 传递：构造函数、`extend_for_sliding_window`、`extend_forward`、`forward` 均加入 `self.sinks` 并向底层调用转发。  
- 文档 `docs/design/attention_backends.md` 更新表格，标记 ROCm AITER FA 现在 **支持** sink。  
- 相关测试/调用层若使用 ROCm AITER FA，需要显式提供 `sinks` 张量或保持 `None`（兼容旧行为）。

**💡 关注建议**  
1. **兼容性检查**：在非 ROCm 环境或使用不支持 sink 的后端（如 `triton`、`flash_attn`）时，保持 `sinks=None`，避免触发 `supports_sink` 检查错误。  
2. **参数校验**：确保 `sinks` 张量的 dtype、shape 与 KV 缓存保持一致（当前仅支持 fp16/bf16），并在创建时与 `kv_sharing_target_layer_name` 配合使用。
