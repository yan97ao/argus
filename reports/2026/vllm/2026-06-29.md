### [MoE Refactor] Standardize Humming MoE experts + utilities (#43373)
**SHA**: `bc8481a` | 🔗 [查看提交](https://github.com/vllm-project/vllm/commit/bc8481af09cd4c7f7272ba7bc1913f1051649813)

**🎯 变更类型**：功能增强 / 架构变更  
**⚡ 重要程度**：🔴高  
**📋 变更摘要**  
- 将 MoE（Mixture‑of‑Experts）实现统一为 Humming 后端的标准化专家模块，新增 *Indexed / Grouped / Batched* 三类 Humming 专家实现。  
- 重构_quantization_ 与 _humming_ 的交叉逻辑：提供统一的 `make_humming_moe_quant_config`、`select_humming_moe_experts`、`convert_to_humming_moe_kernel_format` 等工具函数，把权重/激活的 schema 转换、强制 requant、元信息准备等全部集中在一处。  
- 删除旧的 `prepare_humming_moe_layer`，改为在 `HummingQuantization` 中直接选取专家类并在 `apply()` 中调用统一的 `HummingMethod`，避免双重量化。  
- 更新 MXFP4/FP8/Quark/Compressed‑Tensors 量化路径，使其能够使用统一的 Humming 量化描述。  
- 新增对多种 Humming schema（AWQ、GPTQ、Compressed‑Tensors、Modelopt 等）的 `QuantKey` 映射，实现 **“checkpoint‑format → Humming‑format → QuantKey”** 的全链路转换。  
- 对工作空间尺寸、缓冲区形状以及锁 buffer 的创建方式进行细化，使用 `element_size` 计算更精确的内存需求。  

**🎯 影响范围**  
- `vllm/model_executor/layers/fused_moe/*`（所有 MoE expert 实现、模块化 kernel、调度逻辑）  
- `vllm/model_executor/layers/quantization/*`（MXFP4、FP8、Quark、Compressed‑Tensors 等）  
- `vllm/model_executor/layers/quantization/utils/humming_utils.py`（schema‑>QuantKey 映射、权重转换、专家选择）  
- `vllm/utils/humming.py`（对外导出的 Humming 类型与常量）  
- 相关的工作空间与缓冲区管理代码  
- 环境变量 `VLLM_HUMMING_MOE_GEMM_TYPE`、`VLLM_HUMMING_INPUT_QUANT_CONFIG` 的解析路径  

**🔍 技术洞察**  

| 维度 | 影响描述 |
|------|----------|
| **架构影响** | 1. **统一专家抽象**：原有 `HummingIndexedExperts` / `HummingGroupedExperts` / `BatchedHummingGroupedExperts` 通过统一的 `FusedMoEExperts` 接口实现，实现了「模块化‑插件化」的后端切换。<br>2. **全新 Kernel 构造链路**：`make_humming_moe_kernel` 负责 `PrepareFinalize → Experts → Kernel` 的完整装配，取代之前在 `HummingQuantization.apply` 里硬编码的 expert 创建。<br>3. **强解耦**：量化配置、权重转换、专家选择、kernel 组装分别在独立的 util 中完成，降低了各模块之间的耦合度，后续新增后端或新量化方案时只需实现对应的 `QuantKey` 映射与 `select_humming_moe_experts` 逻辑。<br>4. **向后兼容**：当 `has_humming()` 为 `False` 时，相关路径会安全回退（例如 `select_humming_moe_experts` 返回 `None`），不影响已有非 Humming 的 MoE 实现。 |
| **性能影响** | 1. **GPU 内核提升**：Humming 的 `indexed`、`grouped_contiguous`、`grouped_masked` GEMM 在 N‑Vidia Ampere+ GPU 上具备更佳的内存访问模式和矩阵分块调度，预计可提升 15%~30%（取决于 token‑expert 分布）。<br>2. **消除双重量化**：通过 `expects_unquantized_inputs=True`，`apply()` 直接把原始 `hidden_states` 交给 Humming 方法处理，省去一次 CPU‑side的 `quantize_input`，降低了 latency 与内存拷贝。<br>3. **工作空间计算更精确**：改为使用 `params_dtype.itemsize` 来计算 workspace 大小，避免了「按字节/按元素」混用导致的过度分配或不足分配，提升显存利用率。<br>4. **缓存层元信息（`humming_metas`）一次性准备**，减少运行时重复初始化的开销。 |
| **安全考虑** | - **可选依赖**：所有 Humming 相关对象通过 `import_utils.has_humming()` 延迟导入，若环境中缺失 Humming 库将抛出明确的 `ImportError`，不会导致未定义行为。<br>- **输入校验**：在 `convert_to_humming_moe_kernel_format` 中对 `Int4` 打包维度做了 `last_dim % 2 == 0` 检查，防止非法 checkpoint 触发内存越界。<br>- **量化映射**：`weight_schema_to_quant_key`、`input_schema_to_quant_key` 对所有已知 schema 做了显式分支，若出现未知 schema 将直接 `raise TypeError`，避免隐式错误传播。<br>- **日志与异常**：大多数关键路径使用 `logger.info_once / logger.debug_once`，在生产环境中不会泄露敏感信息。<br>- **并发锁**：`locks` buffer 仍保持原来的 `torch.int32` 形式，未改变其线程安全模型。总体上未引入新的安全漏洞，但需要确保外部传入的 schema 不被恶意构造（目前已通过类型检查进行防护）。 |

### [Frontend] Consolidate scale out entrypoints (#44512)
**SHA**: `3483240` | 🔗 [查看提交](https://github.com/vllm-project/vllm/commit/3483240b7ea3d4372b6c79369ea36617f8b1fbb2)

**🎯 变更类型**：架构变更 / 功能增强  
**⚡ 重要程度**：🔴高  
**📋 变更摘要**：  
1. 将原有的 **Disaggregated (serve.disagg)** 入口点统一迁移至新的 **scale_out** 包，创建 `vllm.entrypoints.scale_out` 子模块，并在 CI、文档、示例中全部改为使用该路径。  
2. 新增 **Derender**（聊天/补全）API 路由及对应 `ServingDerender` 实现，原有 `ServingRender` 中的 Derender 逻辑抽离为独立服务。  
3. 引入 `scale_out.factories` 用于统一初始化渲染、反渲染、Token‑In‑Token‑Out（generate）等状态，并在 `openai/api_server.py` 中统一注册对应路由。  

---

### 🎯 影响范围
- **核心入口点**：`vllm.entrypoints.scale_out.*`（包括 `render`, `derender`, `token_in_token_out`）  
- **服务器初始化**：`vllm.entrypoints.openai.api_server`、`vllm.entrypoints.generate.api_router`、`vllm.entrypoints.serve.elastic_ep.api_router`  
- **CI 配置**：`.buildkite/*.yaml` 以及相应的测试目录  
- **文档/示例**：`docs/*`、`examples/scale_out`  
- **内部依赖**：`vllm.renderers.online_derenderer`、`vllm.entrypoints.serve.utils.*`（导入路径全部改为 `scale_out`）  

---

### 🔍 技术洞察  

| 维度 | 影响分析 |
|------|----------|
| **架构影响** | <ul><li>引入全新 **scale_out** 包，职责更加清晰：<br>• `render` – 正向生成并返回 token ID / 文本<br>• `derender` – 从 `GenerateResponse` 逆向构造 OpenAI‑compatible 完整响应（聊天/补全）<br>• `token_in_token_out` – 旧的 “disagg” 生成逻辑（Tokens‑In / Tokens‑Out）</li><li>`scale_out.factories` 成为所有子模块的统一入口，避免在 `api_server` 中散落大量 `import` 与实例化代码，提高可维护性。</li><li>原 `vllm.entrypoints.serve.disagg` 完全被软删除，所有内部引用已改为新路径，形成一次 **breaking change**。</li></ul> |
| **性能影响** | <ul><li>功能上仅是 **代码组织迁移**，核心算法、渲染/反渲染实现未变，预计 **无显著性能回退**。</li><li>新增 `ServingDerender` 类仅在 **非流式** 场景被调用，额外一次 `online_derenderer.derender_*` 调用，延迟极小（微秒级）。</li><li>`register_scale_out_api_routers` 中的路由注册顺序保持不变，不影响请求路由匹配性能。</li></ul> |
| **安全考虑** | <ul><li>新增两条 **Derender** POST 接口（`/v1/chat/completions/derender`、`/v1/completions/derender`），同样使用 FastAPI 的 **JSON schema 校验** 与 **统一错误模型**，未引入额外安全风险。</li><li>Derender 逻辑只依赖内部 `online_derenderer`，不访问外部资源，属于 **数据处理层**，安全面影响有限。</li><li>迁移过程中未改动身份验证/访问控制，保持原有 **ScalingMiddleware** 及 **Elastic EP** 中的安全策略。</li></ul> |
| **可维护性** | <ul><li>通过 `scale_out.factories` 抽象出 **state 初始化** 与 **路由注册**，后续若再加入新的子模块（例如新型推理模式），只需在 factories 中添加即可。</li><li>旧路径的全部删除降低了 **模块耦合**，但也意味着 **向后兼容性** 被打破，需要在发布说明中明确告知用户。</li><li>测试目录同步迁移，CI 新增 `scale_out` 用例，覆盖率提升。</li></ul> |
| **兼容性** | <ul><li>**破坏性变更**：所有直接 import `vllm.entrypoints.serve.disagg.*` 的用户将出现 `ImportError`。</li><li>建议在下一个次要版本提供 **兼容 shim**（如 `vllm.entrypoints.serve.disagg = vllm.entrypoints.scale_out`）或在 **release notes** 中明确迁移步骤。</li></ul> |

### ⚠️ 潜在风险
1. **向后兼容性破坏**  
   - 第三方项目、旧示例或内部脚本仍使用 `serve.disagg` 导入会在运行时失败。  
2. **初始化顺序错误**  
   - `init_render_state` 与 `init_scale_out_state` 必须在 `state` 中先后设置 `serving_render`、`serving_derender`、`serving_tokens`；若其他代码在 `api_server` 初始化之前访问这些属性，会出现 `AttributeError`。  
3. **路由重复注册**  
   - `api_server` 仍保持对旧 `attach_disagg_router` 的注释删除，若未来误删 `register_scale_out_api_routers` 调用，将导致 **generate** 接口缺失。  
4. **Derender 依赖的 `OnlineDerenderer` 未在 `ServingDerender` 初始化时检查**  
   - 若 `state.online_derenderer` 为 `None`（例如在仅启用 `render` 场景），调用 Derender API 会抛出未实现异常。  
5. **文档/示例未同步**  
   - 部分用户可能仍看到旧 `disaggregated` 示例，导致混淆。

### Remove unnecessary `load_weights` methods (#44589)
**SHA**: `5051698` | 🔗 [查看提交](https://github.com/vllm-project/vllm/commit/5051698e41b7dc3da421f1c50bfe178a92dc7881)

**🎯 变更类型**：重构 / 架构变更  

**⚡ 重要程度**：🔴高  

**📋 变更摘要**  
本次提交在整个 `vllm/model_executor` 代码树中统一移除了各模型实现的手写 `load_weights` 方法，改为依赖 `AutoWeightsLoader` 与全新的 `WeightsMapper` 完成权重映射与加载。新增了对 KV‑cache 量化 scale 的统一正则映射、对堆叠（fused）权重的拆分映射、对 shard‑id 的携带与校验，并在量化配置中引入统一的 “忽略后缀” 列表。模型层级的 LoRA、PP、Quant 等特性在加载时均通过统一的映射器实现组合 (`mapper |= cache_scale_mapper`)。

**🎯 影响范围**  
- `vllm/model_executor/layers/*`（线性层、QKVParallelLinear、MinimaxM3 等）  
- 所有模型实现文件 `vllm/model_executor/models/*.py`（Arcee、ChatGLM、Gemma、Llama、Qwen2、等 30+）  
- 量化配置基类 `vllm/model_executor/layers/quantization/base_config.py` 与子类 `fp8.py`, `quark.py`  
- 权重加载器相关工具 `vllm/model_executor/model_loader/*`（bitsandbytes_loader、reload/layerwise、utils）  
- `vllm/lora/worker_manager.py` 与少量其他辅助模块
