### [6/n] Migrate activation kernels, gptq, gguf, non cutlass w8a8 to libtorch stable ABI (continued) (#42663)
**SHA**: `07aeaf9` | 🔗 [查看提交](https://github.com/vllm-project/vllm/commit/07aeaf9d4df870a76d5a0dc19d6a7e74b4be5d3b)

**🎯 变更类型**：架构变更 / 功能增强  
**⚡ 重要程度**：🔴高  

**📋 变更摘要**  
- 将 activation、GPTQ、GGUF、非‑CUTLASS W8A8 量化等核心 CUDA/ROCm kernel 从原先的 **unstable ABI**迁移至 **libtorch stable ABI**，新增 `_C_stable_libtorch` 扩展目标。  
- 重构文件组织与头文件引用，统一使用 `torch::stable::Tensor` 与 stable‑ABI 相关 API（`torch::stable::accelerator::DeviceGuard`、`get_current_cuda_stream()`、`VLLM_STABLE_DISPATCH_*` 等）。  
- 为 ROCm 添加显式的 `amdhip64` 链接与设备 guard 逻辑，解决多 GPU 环境下的 HIP 运行时冲突。  
- 在 CMake 与 Python `setup.py` 中加入对 CUDA 与 HIP 双语言的条件编译，使两套扩展在同一次构建中均可生成。  
- 为新 stable‑ABI 目标实现完整的 op 注册（激活、int8/fp8 量化、GPTQ、GGML），并在 `vllm.platforms.rocm` 中尝试导入 `_C_stable_libtorch`。  

**🎯 影响范围**  
- `csrc/activation/*`、`csrc/quantization/*`（GPTQ、GGUF、W8A8）以及对应的头文件。  
- CMake 构建系统及 Python 包的编译逻辑。  
- 运行时，所有依赖这些 kernel 的模型推理路径（包括 SwiGLU、GeGLU、FATReLU、GELU、int8/fp8 量化、GPTQ‑accelerated GEMM、GGML‑related 计算）。  
- 支持平台：CUDA 11+（包括 Hopper/SM9.0）以及 ROCm 5+（HIP）。  

**🔍 技术洞察**  

| 维度 | 影响说明 |
|------|----------|
| **架构影响** | - 引入 **stable ABI**（`torch::stable::*`）后，扩展不再依赖 PyTorch 的内部实现，能够在 PyTorch 2.1 及以上的二进制 wheel 中直接使用，提升二进制兼容性。<br>- 新的 `_C_stable_libtorch` 与原 `_C` 共存，保持向后兼容；但若两者同时被加载，可能产生 **op 重复注册**（同名 op 在两个库中实现），需要确保只加载一套。 |
| **性能影响** | - 代码层面仅替换了 `at::cuda::*` 到 stable 对应的轻量包装，运行时开销基本持平。<br>- 对 HIP 运行时加入了手动链接 `amdhip64`，避免二次初始化导致的 **DeviceGuard 错误** 与 **core dump**，提升多 GPU 稳定性。<br>- `VLLM_STABLE_DISPATCH_*` 宏保持原有模板特化逻辑，算子调度不变，性能不受负面影响。 |
| **安全考虑** | - 新增的动态链接 `amdhip64` 若系统中存在多个版本，可能出现加载错误；但已通过 `find_library` 限定在 PyTorch 安装目录搜索，风险可控。<br>- 改用 `STD_TORCH_CHECK`（header‑only 实现）后，异常信息仍保持原有行为，无新增安全漏洞。 |
| **可维护性** | - 统一使用 `torch_utils.h` 中的 `TORCH_UTILS_CHECK`，简化跨 ABI 检查代码。<br>- 将大量 `#include <torch/...>` 替换为相对路径，便于文件移动后的 **git rename detection**，有助于历史追踪。<br>- 新增 `torch_utils.h` 与 `torch_utils.cpp`（仅头文件）解耦检查逻辑，后续扩展更容易。 |
| **兼容性** | - 只在 **PyTorch ≥ 2.1**（stable ABI 引入版本）上可用；在旧版 PyTorch（仅提供 unstable ABI）时，`_C_stable_libtorch` 将编译失败，回退到仅使用 `_C`。<br>- 对于非官方构建的 PyTorch（自行编译的 libtorch），需确保 `TORCH_TARGET_VERSION` 宏定义，以匹配 stable ABI。 |
| **部署影响** | - `setup.py` 现在会在检测到 CUDA **或** HIP 环境时额外编译 `_C_stable_libtorch`，导致 **wheel 大小** 约提升 5‑10 MiB。<br>- 部署脚本需确保 `torch` 包的版本满足 stable ABI 要求，否则会出现 **ImportError**（已在 `vllm/platforms/rocm.py` 捕获并记录警告）。 |

**⚠️ 潜在风险**  

1. **ABI 不匹配**：如果用户使用的 PyTorch 版本低于 2.1，`torch::stable` 命名空间不存在，加载 `_C_stable_libtorch` 会立刻报错。  
2. **双库冲突**：在同一进程中同时导入 `_C` 与 `_C_stable_libtorch`，相同 op 可能被注册两次，引发 `RuntimeError: Operator ... is already registered`。  
3. **HIP 链接不确定性**：`find_library(_STABLE_TORCH_AMDHIP64 ...)` 只在 PyTorch 打包了 `libamdhip64.so` 时生效；若用户自行编译 PyTorch 并未携带该库，将出现未链接的符号，导致加载失败。  
4. **编译路径错误**：大量相对路径的头文件引用（如 `"../../../quantization/gguf/ggml-common.h"`）在 CI 环境外可能失效，尤其在 `pip install` 时的源码目录结构变化。  
5. **宏迁移遗漏**：部分 kernel 仍残留 `VLLM_DISPATCH_*`（未改为 `VLLM_STABLE_DISPATCH_*`），在 stable ABI 环境下仍会调用老的宏，可能导致编译警告或行为不一致。  
6. **Meta‑tensor 处理**：在 `gptq_gemm` 中对 `b_g_idx` 的 meta‑tensor 检查改为 `device().type() == torch::stable::DeviceType::Meta`，若旧版 PyTorch 没有该枚举，可能编译错误。

### [MISC] Fix symm_mem cap-equal gate; log AR backend selection (#42993)

**🎯 变更类型**：Bug 修复 / 小幅功能增强  
**⚡ 重要程度**：🟡 中  

**📋 变更摘要**  
1. 修正了 `symm_mem` 全局归约中 “size ≤ max_size” 的判断，从原来的 `<` 改为 `<=`，防止恰好等于阈值的张量被错误地排除。  
2. 在 `CudaCommunicator` 中新增 `_log_all_reduce_backend_selection`，在实例化时记录当前进程组实际可用的 all‑reduce 后端及其调度顺序，便于调试和性能分析。  

**🎯 影响范围**  
- `vllm/distributed/device_communicators/cuda_communicator.py`（实例化即日志、后端可用性判断）  
- `vllm/distributed/device_communicators/symm_mem.py`（size gate 修正）  

**💡 关注建议**  
1. **功能验证**：针对 `symm_mem` 的 `max_size` 边界（等于阈值）编写单元测试，确保此前因 `<` 导致的回退路径被正确触发。  
2. **日志噪声**：`logger.info_once` 已防重复，但仍会在每个进程组首次创建时输出。建议在 CI/生产环境通过 `VLLM_LOG_LEVEL` 或 `scope="global"` 控制，可在文档中说明如何关闭。  
3. **后端选择一致性**：新增日志依赖 `NCCL_SYMM_MEM_ALL_REDUCE_CONFIG` 中的 `min_world_size`、`custom_ar_preferred_ranges` 等字段。若后续配置改变，请同步更新日志描述，防止误导。  
4. **兼容性检查**：确认 `should_nccl_symm_mem_allreduce` 的内部大小判定仍然使用 `<=`，避免出现“日志显示可用但实际不可用”的不一致状态。  
5. **性能回归**：在多节点（world_size ≥ 2）场景下跑一次全模型推理基准，确保新增日志和轻量级判断不会显著影响启动时间。  

总体来看，本次修改修复了一个潜在的错误边界并提升了可观测性，对整体功能稳定性有正面作用，风险有限。建议通过上述测试与配置检查后合并。

**🎯 变更类型**：Bugfix / 功能增强  
**⚡ 重要程度**：🟡 中  
**📋 变更摘要**：  
1. 将 `DeepSeekV32ToolParser` 中的参数类型转换逻辑抽离为共享工具 `coerce_to_schema_type`、`extract_types_from_schema`、`find_tool_properties`，去除原有的 `_convert_param_value*` 实现。  
2. 新增/完善了大量单元测试，覆盖对象/数组、数值、复合类型（`["integer","null"]`）以及缺失 schema 时的默认行为，确保流式与非流式解析的一致性。

**🎯 影响范围**：  
- `vllm/tool_parsers/deepseekv32_tool_parser.py`（核心解析实现）  
- `vllm/tool_parsers/utils.py`（新增/修改的公共函数）  
- 测试套件 `tests/tool_parsers/test_deepseekv32_tool_parser.py`（大量新增案例）  
- 可能影响其他使用相同工具解析器的模块（如 `vllm/engine` 中的工具调用包装）。

**💡 关注建议**：  
- **兼容性检查**：确认 `coerce_to_schema_type` 在旧版 schema（仅单一 `type` 字段）上仍能得到预期结果，避免因类型列表处理改变导致回归。  
- **异常处理**：当前实现对 JSON 解析异常直接抛出，建议在 `coerce_to_schema_type` 中保留“回退为原字符串”机制，以免因 malformed JSON 使整条 tool call 失效。  
- **性能评估**：抽象为公共函数会增加一次函数调用和一次字典遍历（`find_tool_properties`），但影响微乎其微；仍建议在大规模并发推理场景下跑基准测试。  
- **文档同步**：更新 `DeepSeekV32ToolParser` 的使用说明，明确 `"string=\"false\""` 参数将在 schema 存在时被强制类型转换。  
- **回归测试**：运行完整的 CI（包括其它模型的 tool parser）确保新工具函数未意外改变其他解析器的行为。  

总体而言，此次改动通过统一类型转换逻辑提升了可靠性和可维护性，新增的测试大幅覆盖边缘情形。只要在发布前确认上述兼容性与异常回退策略即可安全合并。

2. **参数一致性**：`top_k` 与 `num_experts` 必须在模型的所有 MoE 层保持同值；若未来计划支持异构层，需另行实现兼容逻辑。  
3. **资源回收**：`cleanup()` 在每次重新分配前调用，确认底层 CUDA/CPU 缓冲已被释放，防止显存泄漏。  
4. **文档/示例**：更新使用 FlashInfer One‑Sided 的部署文档，说明“同一进程内多次 `initialize` 可安全增大 workspace，但不能缩小”。  
5. **兼容性测试**：在多节点、多 GPU（>2）以及混合精度（NVFP4、BF16、FP8）组合下跑一遍 CI，确保 barrier 改动不会在大型集群中引入死锁。  

总体来看，此次改动解决了异构 MoE 层在 FlashInfer 单侧通信中的崩溃问题，提升了框架对混合量化模型的支持度，只要在上述注意点上做好验证，即可安全合入主分支。

### [feat] Add FP8 per-tensor Q scale support to Triton attention backend (#42080)
**SHA**: `d247a93` | 🔗 [查看提交](https://github.com/vllm-project/vllm/commit/d247a931cc25e7253feccbd6260d48216ff5c081)

**变更类型**：功能增强（为 Triton 注意力后端加入 FP8 per‑tensor Q‑scale 支持）

**核心改动**  
1. **统一注意力接口 (`unified_attention`)**  
   - 新增 `q_descale` 参数，默认 `None`，并在调用 Triton kernel 时传递。  
   - 在调用 `kernel_unified_attention` 时加入 `q_scale`、`Q_IS_FP8` 常量，以在 Q 也是 FP8 时完成 **scale‑folding**（将 Q、K 的比例折叠进 softmax 分数，V 的比例折叠进输出累加器）。  
   - 移除原先的断言 `assert q_descale is None`，允许 Q 端的去标量化。

2. **Triton kernel (`kernel_unified_attention`)**  
   - 新增 `USE_FP8_Q_DESCALE` 标志：当 `KV_QUANT_MODE == 1`（FP8 per‑tensor）且查询张量本身是 FP8 时生效。  
   - 计算 `score_scale = scale * q_scale * k_scale`、`value_scale = v_scale`，并在 softmax 计算和 epilogue 中分别使用这些折叠后的比例。  
   - 通过 `Q_IS_FP8` 让编译时常量化判断是否需要该路径。

3. **后端封装 (`triton_attn.py`)**  
   - 在 `forward` 中根据 `self._kv_quant_mode` 判断是否要把 Q‑scale 传递给 kernel。  
   - 兼容原有 per‑token‑head 以及 per‑tensor 路径，保持 `k_descale/v_descale` 的扩展方式不变。

4. **测试**  
   - 扩展 `test_triton_unified_attention`，在全量 FP8 场景下显式使用非 1.0 的 Q/K/V scale，验证去标量化路径；  
   - 新增 *bf16‑query + FP8‑kv* 场景，检查闭环数值误差（`atol/rtol = 1.5e‑1`），确保折叠比例的正确性。

**影响范围**  
- **Attention 前向**：`vllm/v1/attention/backends/triton_attn.py`、`vllm/v1/attention/ops/triton_unified_attention.py`；  
- **KV 缓存接口**：`KVQuantMode.FP8_PER_TENSOR` 与 `kv_cache_uses_per_token_head_scales` 逻辑；  
- **单元测试**：`tests/kernels/attention/test_triton_unified_attention.py`。  
其他后端（torch、flash‑attn）暂未受此改动影响。

**风险与注意事项**  
- 仅在 **查询张量为 FP8** 且 **KV 为 per‑tensor FP8** 时会走 `USE_FP8_Q_DESCALE` 分支，若 Q 为 bf16 而 KV 为 FP8，仍走原有去标量化路径，不会产生错误。  
- `q_descale` 仍在 Python 层保留为可选，调用方必须在使用 FP8 Q 时显式提供对应的 `torch.float32` 标量或张量，否则会得到默认 `None`（相当于 1.0）。  
- 通过 `q_scale * k_scale` 折叠到软最大得分后，数值范围会扩大，需要确保 `softmax_scale` 与 `output_scale` 的取值仍在 FP8 动态范围内；否则可能触发裁剪导致精度下降。  

**建议**  
- **开发者**：在新增模型或实验中，若使用 FP8 Q，请在 `unified_attention` 调用时传入 `q_descale`（或让 `KVQuantMode.FP8_PER_TENSOR` 自动填充），并在配置文件中打开 `kv_quant_mode = "fp8_per_tensor"`。  
- **用户**：检测 GPU 是否支持 FP8（H100、Ada 等），并在 `vllm` 启动脚本中使用 `--kv-dtype fp8`、`--dtype fp8` 组合，观察推理吞吐率和数值误差；若出现异常，请确认 `q_descale` 与 `k_descale/v_descale` 是否匹配。  
- **文档**：补充 FP8‑Q‑scale 折叠的使用说明，说明何时需要显式提供 `q_descale`，以及 `atol/rtol` 的推荐阈值。  
- **后续**：考虑在 `torch` 与 `flash-attn` 后端同步实现同等折叠逻辑，以保持量化路径的一致性。
