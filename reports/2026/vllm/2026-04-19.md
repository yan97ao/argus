### Optimize nemotron VL image/video preprocessing (#40283)
**SHA**: `982beae` | 🔗 [查看提交](https://github.com/vllm-project/vllm/commit/982beae809b1690f745d9d52616de4d1da2c5855)

**🎯 变更类型**：功能增强 / 性能优化  
**⚡ 重要程度**：🟡 中  

**📋 变更摘要**  
- 将 Nemotron‑VL 的图片/视频前处理从 NumPy → torch 的逐步路径，改为一次性 **torch.compile** 的 `bicubic` resize+归一化算子，实现数据排布、插值、类型转换的融合。  
- 删除冗余 `warnings` 捕获，统一使用 NHWC→NCHW 入口函数 `_pil_to_nhwc_tensor`。  
- 对图片、视频以及多图 Tile 的预处理统一注入 `norm_mean / norm_std / dtype`，并在 `DynamicResolutionParams.apply_params` 中直接返回已归一化的 Tensor。  
- 视频帧分隔符批量 token‑化，利用 HF tokenizer 的 Rust 后端并行，减少循环调用。  

**🎯 影响范围**  
- `vllm/transformers_utils/processors/nano_nemotron_vl.py`（核心前处理实现）  
- `DynamicResolutionImageTiler` 与 `DynamicResolutionParams`（图像 Tile 逻辑）  
- 视频预处理入口 `video_to_pixel_values`、`_videos_to_pixel_values_lst`  
- `NemotronVLProcessor` 中的 `norm_mean / norm_std` 传递及 `dtype` 支持  
- `get_video_repl` 中的 frame‑separator token 化路径  

**💡 关注建议**  
1. **兼容性**：`torch.compile` 仅在 PyTorch≥2.0 并开启 Dynamo/Inductor 时生效，使用者需要在部署脚本里确认 `torch.backends.cuda.enable_mem_efficient_spline=True`（如有需要）或提供回退路径。  
2. **配置校验**：新增 `dtype` 参数默认 `torch.float32`，但模型可能要求 `bfloat16`/`float16`；建议在 `PretrainedConfig` 中显式声明 `dtype`，否则可能出现精度不一致。  
3. **归一化数值**：`norm_mean`、`norm_std` 仍为 `torch.Tensor`，调用方要确保它们与图像 dtype 匹配（如 `float32`），否则会触发隐式 cast。  
4. **性能验证**：在 CPU、单 GPU、多个 GPU（torchrun）三种环境下跑一次完整的文本‑图像‑视频推理基准，比较 `torch.compile` 前后的 latency 与显存占用，确认无额外的 CUDA kernel 编译开销。  
5. **单帧/单视频路径**：代码已加入 “若仅有一个视频/图像直接返回” 的 fast‑path，建议补充单元测试覆盖 `len(pixel_values_lst) == 1` 的分支，防止遗漏 .contiguous() 调用导致 stride 异常。  

总体来看，此次提交通过算子融合、批量 token‑化以及 `torch.compile` 的 JIT 编译，显著降低了图像/视频前处理的 Python‑to‑C 边界次数和拷贝成本，预计在大批量多模态推理场景下可获得 15%‑30% 的吞吐提升。开发者在升级时注意 PyTorch 版本及配置即可，用户侧无需改动调用接口。

### [FEAT] [Perf] [Gemma4] Fused Gemma4 Routing Function Triton (#39083)
**SHA**: `45232a4` | 🔗 [查看提交](https://github.com/vllm-project/vllm/commit/45232a454e4c57191633fefda5baa9fa1f5dc8fd)

**🎯 变更类型**：功能增强 / 性能优化  
**⚡ 重要程度**：🟡 中  
**📋 变更摘要**：为 Gemma‑4 MoE 路由实现了基于 Triton 的全新 fused kernel，使用位排序 + 向量化计算一次性得到 top‑K 权重和专家 ID，显著减少循环与跨 lane 归约；在 `routing_function` 中根据平台自动切换到 Triton 实现；并补充了对应的单元测试。  

**🎯 影响范围**  
- `vllm/model_executor/models/gemma4.py`（新增 Triton kernel、包装函数、平台检测逻辑）  
- `vllm/platforms/current_platform`（用于判断是否使用 CUDA/XPU）  
- 测试目录 `tests/kernels/moe/test_gemma4router.py`（验证数值一致性）  

**💡 关注建议**  
1. **兼容性**：确保 `current_platform` 正确识别所有 GPU、XPU 环境，防止在不支持 Triton 的平台上仍调用该 kernel。  
2. **数值误差**：测试容忍 1e‑2 的相对/绝对误差已通过，但在极端 logits 分布或更高精度需求（e.g. FP64）时需额外验证。  
3. **内存使用**：kernel 采用 `BLOCK_E = next_power_of_2(E)`，对极大专家数（>64K）可能导致显存浪费，建议在文档中注明最优 E 范围。  
4. **调参**：`num_warps` 默认 1，针对不同 GPU（A100、H100）可提供调优指南，以获得最佳吞吐。  
5. **回归测试**：CI 需在 CUDA 与 XPU 两类机器上运行新测试，防止平台判断错误导致功能回退。  

总体来说，此次改动显著提升 Gemma‑4 的 MoE 路由吞吐，并保持与原实现的数值一致性，风险主要在平台检测与极端规模下的显存开销，建议在 release notes 中标明使用前提和调优参数。

## 3. 核心风险 & 验证要点
1. **文字层面的解析 vs token‑ID 层面**  
   之前的实现依赖 tokenizer 的 token‑ID 来定位标记，新的实现直接在 `current_text` 上做字符串匹配。若底层 tokenizer 对特殊标记做了 **子词拆分**（如 BPE 将 `<|tool_calls_section_begin|>` 拆成多个 token），在极端情况下可能出现 **未对齐的标记**，导致 `_extract_content` 误判截断位置。建议在 CI 中加入对 **不同 tokenizer（Llama‑3, Mistral, Gemma）** 的跨块标记拆分测试。

2. **`partial_tag_overlap` 的边界**  
   该函数仅返回公共前缀长度，若标记被 **多字符**（如 Unicode 表情）跨块截断，仍能正确计算。但若标记出现 **重复子串**（如 `<|tool_call_begin|>` 包含 `<|tool_`），可能产生误判。建议对标准标记集合跑一次“标记内部子串”冲突检查。

3. **异常容错**  
   - 解析非法 JSON 时现在 **仍返回 ToolCall**（arguments 保持原始字符串），但 `json.loads` 在后续使用时会抛异常。需要确认调用方在渲染或执行工具前已进行 `json.loads` 包装，或在 `run_tool_extraction` 中已经捕获。
   - 对 **malformed ID**（缺少 `:digit`）直接跳过，这与旧实现保持一致，但应留意在 `tool_calls` 列表中出现 “空洞” 的情况。

4. **状态清理**  
   `reset_streaming_state` 被移除，改为在每次 `extract_tool_calls_streaming` 开始时依据 `self._sent_content_idx` 自动回收。若同一个 parser 实例在 **多轮对话** 中复用，可能残留上一轮的 `prev_tool_call_arr`、`streamed_args_for_tool`。建议在 `ChatCompletionRequest` 的 `tool_choice="none"` 或对话结束时显式调用 `self.__init__(self.tokenizer, self.tools)` 进行 “软重置”，或在 `adjust_request` 中检测 `request.tools` 变化。

---

## 4. 给开发者的建议
1. **增加跨 tokenizer 的兼容性测试**  
   - 用 `get_tokenizer` 分别加载两三种常用模型（Llama‑3‑8B, Mistral‑7B, Gemma‑2B），模拟标记被拆分成多个 token 的情形，确保 `extract_tool_calls_streaming` 仍能正确恢复完整标记。

2. **回归旧实现的行为**  
   - 运行一遍旧的 **非流式** `extract_tool_calls` 与新实现在 `streaming=False` 时的输出对比，确保两者在正常、异常、无工具三类场景下保持一致。

3. **文档和示例同步更新**  
   - 在 `docs/source/` 中补充 “Kimi‑K2 流式工具调用” 使用示例，强调 `skip_special_tokens=False` 的必要性，以及怎样通过 `run_tool_extraction_streaming` 获取 `DeltaMessage`。

4. **状态重置说明**  
   - 在 `ToolParser` 基类或 `KimiK2ToolParser` 中加入 `reset()` 方法的注释，提醒使用者在每轮对话结束后手动调用，以防意外状态泄漏。

5. **监控关键日志**  
   - 通过 `logger.debug` 已保留多处关键路径日志（如 “Entering tool section”, “Deferring section exit” 等）。在产品环境启用 `INFO` 级别时可以监控是否出现 “section overflow” 警告，及时捕获异常模型输出。

---

## 5. 小结
本次提交对 **Kimi‑K2** 的工具调用流式解析做了 **一次彻底的重构**：从基于 token‑ID 的状态机转向基于纯文本切片的轻量实现，核心漏洞（标记泄漏、参数截断、内容丢失）得到根本性修复。改动主要集中在 `kimi_k2_tool_parser.py` 与相应的测试文件，影响范围主要是该 parser 以及使用它的流式聊天路径

### [KV Offload] Pass request context (#39185)
**SHA**: `4353c9c` | 🔗 [查看提交](https://github.com/vllm-project/vllm/commit/4353c9cb4ac8d9818303c2e61282062aff4074c3)

**🎯 变更类型**：功能增强（在 KV offload 流程中引入 per‑request 上下文 `ReqContext`）  
**⚡ 重要程度**：🟡 中  
**📋 变更摘要**：  
- 新增 `ReqContext` 数据结构，用于携带请求级 KV‑transfer 参数。  
- `OffloadingManager` 接口及其实现（CPU、Reuse、Scheduler）统统改为 `lookup / prepare_load / prepare_store` 三个方法接受 `req_context` 参数。  
- 相应单元测试和 mock 对象全部更新，以传递该上下文。  

**🎯 影响范围**：  
- `vllm/v1/kv_offload/abstract.py`（接口定义）  
- `vllm/v1/kv_offload/cpu/manager.py`、`reuse_manager.py`（实现）  
- `vllm/distributed/kv_transfer/kv_connector/v1/offloading/scheduler.py`（调度层）  
- 所有 KV‑offload 相关测试文件  

**💡 关注建议**  

1. **兼容性**：`OffloadingManager` 的公开 API 已改变，外部插件或自定义 manager 需同步更新签名，否则会在运行时出现 `TypeError`。建议在 `abstract.py` 中保留旧签名的兼容包装（例如 `def lookup(self, keys, req_context=None)`) 并在文档中标记已废弃，以避免突发错误。  

2. **默认 Context**：目前测试里使用 `_EMPTY_REQ_CTX`，但实际入口（如 `RequestOffloadState.__post_init__`) 直接把 `self.req.kv_transfer_params` 包装成 `ReqContext`。请确保在所有非‑KV‑transfer 场景下 `req_context` 不为 `None`，否则可能导致属性访问异常。可以在 `ReqContext` 中提供 `__post_init__` 或 `default_factory`。  

3. **实现细节**：CPU manager 在 `lookup`、`prepare_load`、`prepare_store` 中仍未使用 `req_context`（仅作占位）。若将来需要依据 `kv_transfer_params` 做分层或策略选择，请在实现里加上相应的分支逻辑，避免出现“参数未使用”导致的代码审计警告。  

4. **性能影响**：每次调度都要创建 `ReqContext` 实例（轻量），但在高并发推理场景下仍会产生微量 GC。建议在 `RequestOffloadState` 中缓存该对象（已实现）并在后续路径复用，确保不在热点循环中重复实例化。  

5. **测试覆盖**：目前仅修改了现有单元测试的 mock 调用签名，未新增针对 `req_context` 内容变化的校验。建议补充测试：① 不同 `kv_transfer_params` 影响 `prepare_store` 的返回；② 传入 `None` 或空 dict 时保持旧行为。  

6. **文档与示例**：更新 README / API 文档，说明新 `ReqContext` 的结构与使用场景，提供示例代码，帮助开发者快速迁移。  

总体而言，此次改动为后续实现请求感知的 KV‑offload 打下基础，代码改动保持一致且测试通过。但需注意向后兼容、上下文实际使用以及文档同步，以免在生产环境中出现意外。

### [Frontend] Preserve structured output special tokens in offline LLM.chat (#39352)
**SHA**: `38907e4` | 🔗 [查看提交](https://github.com/vllm-project/vllm/commit/38907e4391c29465745f0a9e11dd7e3bae4b30a0)

**🎯 变更类型**：功能增强  
**⚡ 重要程度**：🟡 中  
**📋 变更摘要**：在 `vllm/entrypoints/llm.py` 的离线聊天路径中，新增对结构化输出（思考块、工具调用）所使用的特殊 token 的保留逻辑。当启用思考模式或提供工具列表且模型（目前硬编码为 Gemma4 系列）将这些标记注册为 *special tokens* 时，自动把 `skip_special_tokens` 设为 `False`，确保这些 token 不会在 `output.text` 中被剔除，便于后续解析。  

**🎯 影响范围**：  
- `vllm.entrypoints.llm.LLM._run_chat`  
- 新增私有方法 `_adjust_params_for_parsing`  
- 与渲染器 (`self.renderer`)、模型配置 (`self.model_config`) 以及 `SamplingParams` 的交互  

**💡 关注建议**  

1. **硬编码检测**：目前仅通过 `architectures` 中包含 “Gemma4” 来决定是否开启保留，这会在后续加入其他同样使用特殊 token 的模型时失效。建议抽象为配置项或在 tokenizer 上直接检查 `all_special_tokens` 是否包含目标 token。  
2. **兼容性**：对所有 `SamplingParams` 均强制修改 `skip_special_tokens`，如果用户显式在别处设为 `True` 可能导致意外行为。可在文档中明确说明此行为，或提供 `preserve_structured_tokens` 参数让调用方自行控制。  
3. **单元/集成测试**：补充针对 Gemma4（或模拟 tokenizer）和非 Gemma4 两类模型的测试，验证在开启/关闭 thinking、提供/不提供 tools 时 `output.text` 是否正确保留或移除特殊 token。  
4. **性能与错误处理**：`self.renderer.get_tokenizer()` 每次调用都会获取 tokenizer，确认其内部实现是轻量的；若有异常（如 tokenizer 未初始化）应捕获并给出友好错误。  
5. **文档更新**：在 `LLM.chat` 的参数说明中加入 `enable_thinking`、`tools` 与 `skip_special_tokens` 的交互说明，提醒用户此自动修改的前置条件。
