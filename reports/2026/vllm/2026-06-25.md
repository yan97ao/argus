### [Rust Frontend] Make `ToolParserOutput` a seq of `ToolParserEvent` to preserve order (#46584)
**SHA**: `4d3b4b9` | 🔗 [查看提交](https://github.com/vllm-project/vllm/commit/4d3b4b9b01efbca77872e3d4a568b273c7a245a7)

**🎯 变更类型**：功能增强 / 重构  
**⚡ 重要程度**：🔴高  

**📋 变更摘要**  
此次提交对 **ToolParser** 的输出模型进行根本性改造：原有的 `ToolParserOutput { normal_text, calls }` 被替换为 **有序事件流** `events: Vec<ToolParserEvent>`，其中 `ToolParserEvent` 可是普通文本、推理文本或工具调用。为此新增 `push_text / push_reasoning / push_call` 接口、`normal_text()` 与 `calls()` 访问器以及 `coalesce()` 合并实现。全部工具解析器、统一解析器、相关测试以及 Python 包装层同步改写，确保解析过程能够 **保留文本与工具调用的原始顺序**（例如在 `ToolCallArguments` 之后仍能出现普通文本）。

---  

**🎯 影响范围**  

| 受影响模块 | 说明 |
|------------|------|
| `rust/src/parser/src/tool/*` | 所有具体模型的解析器（DeepSeek、Gemma4、Granite4、Llama、Qwen 等）全部改为使用 `output.push_text`、`output.push_call`。 |
| `rust/src/parser/src/unified/*` | 统一解析器以及组合解析器改为使用 `UnifiedParserOutput::events`，新增对 `Reasoning` 事件的支持。 |
| `rust/src/parser/src/tool/mod.rs` | 引入 `ToolParserEvent` 枚举、`ToolParserOutput` 结构及其新方法。 |
| `rust/src/parser/src/tool/tests.rs`、各子模块 test 文件 | 大量断言从直接访问 `normal_text` / `calls` 改为通过 accessor。 |
| `rust/src/parser/src/benches/utils/mod.rs` | 调用从 `result.normal_text`、`result.calls` 改为 `result.normal_text()`、`result.calls()`。 |
| Python 包装层 `vllm/tool_parsers/rust_tool_parser.py` 与对应测试 | `coalesce_calls()` 改为 `coalesce()`，保持返回的 `ToolParserOutput` 接口一致。 |
| 相关文档/示例（`tests/tool_parsers/test_rust_tool_parser.py`） | 同步更新调用方式。 |

**🔍 技术洞察**  

| 维度 | 影响分析 |
|------|----------|
| **架构影响** | - **从属性模型到事件流模型**：`ToolParserOutput` 由两块平行数据 (`normal_text`、`calls`) 变为单一有序 `events` 列表，统一了内部数据组织，便于以后扩展其他事件类型（例如推理、系统消息）。<br>- **向后兼容性**：所有原有直接字段访问被移除，需通过 `normal_text()/calls()` 访问器。若外部 crate（或自定义解析器）仍依赖结构体字段，将导致编译错误。<br>- **统一解析器增强**：新增 `Reasoning` 事件，能够区分模型的内部推理文本与真实输出，对后续“思考过程”可视化或日志记录提供基础。 |
| **性能影响** | - **轻微额外拷贝**：`normal_text()` 和 `calls()` 现在在每次调用时会重新聚合字符串或遍历事件；但在实际使用场景（一次性完成解析后调用一次）影响可忽略。<br>- **内存布局**：`ToolParserEvent` 为 `enum`，每个事件携带 `String`/`ToolCallDelta`，与原来的 `Vec<String>` + `Vec<ToolCallDelta>` 类似，整体内存占用基本持平。<br>- **合并过程**：`coalesce()` 仍基于 `BTreeMap` 按 `tool_index` 合并，时间复杂度 O(n log n)，与原实现相同。 |
| **安全考虑** | - 代码全部使用安全 Rust，无 `unsafe`。<br>- 解析逻辑未改变，仅迁移了数据写入方式；因此 **不引入新的安全风险**（如缓冲区溢出、未检查的索引等）。 |
| **可维护性** | - **接口统一**：通过 `push_text`、`push_call`、`push_reasoning`，各解析器实现更简洁，可避免手动维护 `normal_text` 与 `calls` 的同步。<br>- **易于扩展**：新增事件类型只需在 `ToolParserEvent` 中加枚举成员并在 `append_tool_output` 中分支，现有代码无需大幅改动。<br>- **测试覆盖**：几乎所有 parser 的单元测试已更新，确保新行为正确。<br>- **向后兼容层**：若需要平滑迁移，可在 `ToolParserOutput` 中保留已废弃的 `normal_text`/`calls` 字段并标记 `#[deprecated]`，但当前提交已直接移除。 |
| **对外 SDK** | - Python 侧的 `RustToolParser` 对应方法改为 `coalesce()`，调用方需要同步更新。<br>- 其他语言 Bindings（如 Java、Node）若直接映射 `ToolParserOutput.normal_text` 字段，也需同步改动。 |

### [Rust Frontend] Introduce unified parser interface & combined parser (#46583)
**SHA**: `76c3c4f` | 🔗 [查看提交](https://github.com/vllm-project/vllm/commit/76c3c4ff63d6f8ffb27a4a6895f97bca2c60ec87)

**🎯 变更类型**：架构变更 / 功能增强  
**⚡ 重要程度**：🔴 高  

**📋 变更摘要**  
1. 将原有的 *reasoning‑parser* 与 *tool‑parser* 合并为统一的 **vllm‑parser**（原 `vllm‑tool‑parser`），并在 Cargo 工作区中重新组织模块路径。  
2. 引入 **UnifiedParser** 接口以及 **CombinedParser** 适配器，使一次流式解析即可同时产出 *普通文本*、*推理文本* 与 *工具调用* 的事件。  
3. `DefaultChatOutputProcessor` 由两段（reasoning → tool）改为单段（统一解析 → 结构化组装），并相应删减了 `reasoning.rs`、`tool.rs` 两大模块。  
4. 更新所有依赖（Cargo.toml、Cargo.lock）以及 Python bindings（`rust/src/parser/python`），并统一导入路径 `vllm_parser::...`。  

**🎯 影响范围**  
- `rust/src/chat`：输出流水线、解析工厂、请求结构体、单元测试。  
- `rust/src/parser`：新建 `unified` 子模块、改写 `reasoning`、`tool` 子模块的 `pub use` 与错误路径。  
- `rust/src/reasoning-parser`：整包被删除。  
- `tools/build_rust.py`、Python 绑定：路径从 `tool-parser` 调整为 `parser`.  
- 项目整体的 Cargo 工作区成员列表、依赖图、发布脚本。

## 🔍 技术洞察  

| 维度 | 影响分析 |
|------|-----------|
| **架构** | • 统一 **UnifiedParser** 抽象，摆脱原来 `ReasoningParser → ContentEvent → ToolParser` 的两层包装，简化了流式处理逻辑。<br>• **CombinedParser** 充当适配层，同时持有可选的推理解析器和工具解析器，实现“先推理再工具”或仅文本/仅工具的组合模式。<br>• 删除了 `ContentEvent` ⇢ `AssistantEvent` 的 `subenum` 机制，统一为 `AssistantEvent`，降低了 enum 嵌套的复杂度。<br>• 通过 `UnifiedParser::preserve_special_tokens` 与 `structural_tag_model`，仍能向下兼容原有模型对特殊 token 与结构化标签的需求。 |
| **性能** | • **单遍** 解析避免了两次遍历 decoded 文本的开销，理论上降低 CPU 使用和内存拷贝（原先每段 delta 先走 reasoning，再走 tool）。<br>• 实现中仍会在 `parse_into` 中分别调用推理解析器的 `push` 与工具解析器的 `parse_into`，但两者共享同一 `output` 对象，减少了中间集合的分配。<br>• 初始基准（`benches/*.rs`）已迁移至新命名空间，需重新跑一次基准以确认无回退。 |
| **安全** | • 统一解析器仍然通过原工具解析器的 **StructuralTagModel** 对工具调用进行严格检测，未削弱已有的安全检查。<br>• 错误回退策略保持：解析错误时会 **fallback to plain text**，并记录 `warn!`，防止异常数据泄露。<br>• `UnifiedParser::preserve_special_tokens` 仍可在需要保留 `<|assistant|>`、`<think>` 等特殊标记时自动关闭 `skip_special_tokens`。 |
| **可维护性** | • 代码路径统一，新增/修改 parser 时只需实现 `UnifiedParser`（或单独的 `ReasoningParser` / `ToolParser` 并交给 `CombinedParser` 包装），降低了维护成本。<br>• 迁移过程引入的大量 `pub use` 重命名，若未在外部 crate（如用户自定义插件）同步更新，会导致编译错误。提供 **crate alias**（例如在 `Cargo.toml` 中 `[patch]`）或保留旧 crate 的 `pub use` 兼容层可缓解。 |
| **兼容性** | • 公开 API 改动：<br> - `vllm_tool_parser::Tool` → `vllm_parser::tool::Tool` <br> - `vllm_tool_parser::Result` → `vllm_parser::tool::Result` <br> - 原 `vllm-reasoning-parser` 包被废弃。<br> - Python 包名已从 `_rust_tool_parser` 改为 `_rust_parser`（路径更新），导致 `pip install vllm` 的二进制 wheel 名称变化。<br>• 需要在 **发行说明** 中给出迁移指南，并考虑在 **v0.XX+1** 版本中提供向后兼容的 *re‑export*（如 `pub use vllm_parser as vllm_tool_parser`). |
| **测试** | • 新增 `unified_event_stream` 单元测试、`combined.rs` 测试，覆盖了推理、工具、混合、错误回退等场景。<br>• 旧的 `reasoning_event_stream` 与 `tool_event_stream` 已被删除，相关测试也同步迁移。<br>• 建议保留 **回归测试**（完整聊天流、结构化输出、并行 tool 调用）以捕获迁移遗漏。 |
| **CI/构建** | • `Cargo.lock` 中的依赖项从 `vllm‑tool‑parser`、`vllm‑reasoning‑parser` 替换为 `vllm‑parser`，所有工作区成员均需要更新。<br>• `tools/build_rust.py` 已同步路径，但若还有自定义脚本或第三方 CI 仍引用旧路径，会导致构建失败。 |

3. **对称/非对称**：新增的位宽仅在对称量化下受支持，代码已在构造函数中抛异常。请在上层 API（如模型加载）提供明确错误信息，防止用户在配置 `symmetric=False` 时陷入隐式错误。  
4. **单元测试**：补充针对 2‑7 位 pack‑quantized 权重的推理路径测试，特别是：
   - `PackedvLLMParameter` 的形状是否与 `torch.nn.Linear` 等价；
   - 零点张量的尺寸计算 (`packed_output_dim`) 与实际 kernel 对齐；
   - 在 `group_size==-1` 且 `strategy!='channel'` 时抛出期望错误。  
5. **文档**：更新 README/quantization 章节，列出新支持的位宽、对应的 `scalar_type` 名称以及使用限制（必须有 group‑或 channel‑wise 量化、只能对称）。  
6. **性能回归监测**：由于 pack‑factor 变为分数，内部循环可能出现额外的整数除法或 `ceil`，建议在 CI 中加入基准测试，确保 2/3 位等低位宽的推理速度仍优于原始 4/8 位实现。  

总体来看，新增位宽显著提升了低位量化的灵活性，但需在尺寸计算、错误提示和测试覆盖方面加以完善，以避免在实际部署时出现维度不匹配或不支持的配置错误。
