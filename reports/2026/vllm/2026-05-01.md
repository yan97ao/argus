#### 🔴 高重要度变更 (4)

### [kv_offload+HMA][13/N]: Enable HMA support (#41445)
**SHA**: `2fa1f8e` | 🔗 [查看提交](https://github.com/vllm-project/vllm/commit/2fa1f8ec00cf85b15422cb4c0e8eb3632ee13ea8)

**🎯 变更类型**：功能增强 / 架构变更  
**⚡ 重要程度**：🔴高  

**📋 变更摘要**  
1. 为 OffloadingConnector 引入 `SupportsHMA` 标记，使其能够在 KV‑offload 连接器层面声明并使用 Hybrid Memory Access（HMA）特性。  
2. 重构测试框架 `RequestRunner`，加入 `block_size_factor`、`GPUBlock` 抽象以及对多 KV‑group（全注意力、滑动窗口、不同 block 大小） 的支持。  
3. 调整 `KVTransferConfig`、`VllmConfig` 与调度器接口，以便在 HMA 场景下正确传递 `block_size`、`disable_hybrid_kv_cache_manager` 等配置，并去除 `request_finished` 中不再需要的 `block_ids` 参数。  
4. 扩展单元/端到端测试，覆盖多模型、多注意力后端以及 HMA 场景（Gemma‑3、Mamba‑130m、Falcon‑H1 等），并加入对 KV‑event 兼容性的条件处理。  

**🎯 影响范围**  
- **核心模块**：`vllm.distributed.kv_transfer.kv_connector.v1.offloading`（connector、scheduler、spec）  
- **配置层**：`vllm.distributed.kv_transfer.kv_connector.v1.base.SUPPORTS_HMA`、`vllm.distributed.kv_transfer.kv_connector.v1.offloading_connector`、`vllm.distributed.kv_transfer.kv_connector.v1.offloading.scheduler`  
- **测试套件**：`tests/v1/kv_connector/unit/offloading_connector/*`（新增多组 KV‑cache、HMA 场景）  
- **工具函数**：`tests/v1/kv_connector/unit/utils.py`（RequestRunner 重构）  
- **配置生成**：`tests/v1/kv_connector/unit/utils.py#create_vllm_config`（新增 `disable_hybrid_kv_cache_manager` 参数）  

**🔍 技术洞察**  

| 维度 | 影响描述 |
|------|----------|
| **架构影响** | • 引入 `SupportsHMA` 接口后，OffloadingConnector 需要在调度器层面感知 HMA，调度器新增 `request_finished_all_groups` 以一次性返回所有 KV‑group 的块列表。<br>• `RequestRunner` 通过 `GPUBlock`（group_idx + request_block_offset）抽象，使调度器、offload spec、KV‑cache manager 能统一处理跨组块映射，支撑不同 block 大小、Sliding‑Window 等混合场景。<br>• `KVTransferConfig` 现在能够显式关闭混合 KV‑cache manager (`disable_hybrid_kv_cache_manager`)；在 HMA 场景下默认保持开启，确保多组协同工作。 |
| **性能影响** | • HMA 通过在 CPU 与 GPU 之间共享同一块内存映射（或更高效的访问路径），可显著降低跨设备拷贝延迟，尤其在大模型、长上下文（如 Gemma‑3、Mamba）下提升前缀缓存命中率。<br>• 引入 `block_size_factor` 让 offloaded 块大小可调，适配 GPU block 大小的整数倍，减少因对齐导致的额外拷贝。<br>• 多 KV‑group 需要分别触发 `touch`、`prepare_store`，在极端高并发下可能产生轻微调度开销，但整体受益仍然为正向。 |
| **安全考虑** | • 变更仅涉及内部数据结构的扩展和调度流程，未引入外部输入或网络交互的新路径。<br>• 新增的 KV‑event 开关在 HMA 场景下被默认关闭，以避免 HMA 与事件发布冲突；若误开启可能导致事件数据不一致。<br>• 需要确保 `block_size_factor` 与实际 KV‑group 的 `block_size` 对齐，否则可能出现越界写入或内存泄漏。 |
| **可维护性** | • 通过 `GPUBlock` 抽象，将原先仅使用 “GPU block index” 的实现升级为“组 + 偏移”，代码可读性提升，但也增加了对 `num_kv_groups` 正确传播的依赖。<br>• 删除 `request_finished` 参数 `block_ids` 可能破坏旧调用方；建议在 `OffloadingConnector` 添加兼容层并记录 deprecation 警告。<br>• 单元测试大幅扩展，覆盖了多模型、多后端、多块大小的组合，提升回归安全性。 |

**⚠️ 潜在风险**  

1. **向后兼容性**  
   - 其他代码（例如自定义 connector 实现或第三方插件）如果仍旧调用 `request_finished(request, block_ids)`，会因签名不匹配抛异常。  
   - `KVTransferConfig` 中的 `disable_hybrid_kv_cache_manager` 默认 `None`，若未显式设置，某些旧版本可能继续使用旧的 KV‑cache manager，导致行为不一致。  

2. **块映射错误**  
   - `GPUBlock` 的生成依赖 `block_size_factor` 与每个 KV‑group 的 `block_size` 完全匹配；若模型配置或测试中出现不匹配，将导致 `self.offloaded` 键查找失效、断言触发或错误的加载/存储路径。  

3. **并发调度开销**  
   - 在 `request_finished_all_groups` 中一次性返回所有组的块列表，若请求数量极大（数千并发）可能导致单次调度返回的数据结构体积骤增，触发 GIL 或序列化瓶颈。  

4. **KV‑event 与 HMA 的冲突**  
   - 代码已在测试层面禁止在 HMA 场景启用 KV‑event；若在生产配置中误打开，两者的内部状态同步可能出现 race，导致缓存不一致或事件丢失。  

5. **测试依赖平台**  
   - 多模型测试在 CUDA 上才完整运行（HMA 支持限定），在 ROCm 上将跳过部分模型，可能掩盖 ROCm 下的兼容性缺陷。  

**💡 关注建议**  

| 方向 | 建议 |
|------|------|
| **兼容性** | - 为 `OffloadingConnector.request_finished` 添加兼容包装：若检测到旧版 `block_ids` 参数（通过 *args 长度或关键字），抛出明确的 `DeprecationWarning` 并内部转调用新 `request_finished_all_groups`。<br> - 在文档和 Release Note 中声明 `SupportsHMA` 为必选实现，并注明 `disable_hybrid_kv_cache_manager` 默认行为。 |
| **配置校验** | - 在 `RequestRunner.__init__` 或 `KVConnectorScheduler` 初始化阶段加入断言：`offloaded_block_size == gpu_block_size * block_size_factor`，并对每个 KV‑group 做同样校验，提前捕获不一致。 |
| **性能监控** | - 增加 HMA 场景下的专属指标（如 `offload_bytes_per_sec`, `hma_touch_latency`），通过现有 `kv_connector_metrics` 报表进行监控，确保未出现意外回退。 |
| **测试覆盖** | - 增加 ROCm + HMA（即使现在不支持）的负测试，确保在不支持的平台上能够优雅报错。<br> -

### Add nvfp4 kv cache support (#40177)
**SHA**: `947138b` | 🔗 [查看提交](https://github.com/vllm-project/vllm/commit/947138b6c22f9a4751b63b9aa75a2bc4b42835e9)

**🎯 变更类型**：功能增强  

**⚡ 重要程度**：🔴高  

**📋 变更摘要**  
本次提交为 vLLM 引入了 **NVFP4 KV Cache** 支持，扩展了 FlashInfer/TRT‑LLM 后端的 KV‑cache 量化路径。核心实现包括：  
1. 在文档、测试、代码生成脚本中加入 `nvfp4` 关键字；  
2. 实现 BF16 → NVFP4 量化、解量化工具及相应的缓存布局 (`nvfp4_kv_cache_full_dim`)；  
3. 在 `FlashInferBackend`、`FlashInferImpl`、`V1Attention` 等层级通过 `kv_cache_dtype` 选择对应的 kernel（TRTLLM‑gen）并提供 FP8‑output 缓冲区；  
4. 在配置模型时加入对 MLA 与 NVFP4 不兼容的校验；  
5. 更新 ModelOpt 量化方法列表，统一 KV‑cache 量化抽象。  

**🎯 影响范围**  
- `docs/design/attention_backends.md`（文档矩阵）  
- `tests/kernels/attention/*`、`tests/v1/attention/*`（单元/集成测试）  
- `vllm/config/vllm.py`（配置校验）  
- `vllm/model_executor/layers/quantization/modelopt.py`（量化注册）  
- `vllm/v1/attention/backends/flashinfer.py`、`vllm/v1/attention/flashinfer_impl.py`（后端实现）  
- `vllm/v1/kv_cache_interface.py`（KV‑cache 接口尺寸计算）  
- 代码生成脚本 `tools/pre_commit/generate_attention_backend_docs.py`（文档生成）  

---

### 🔍 技术洞察  

| 维度 | 影响描述 |
|------|----------|
| **架构影响** | • 新增 `nvfp4` 作为 **KV‑cache 量化 dtype**，在 `FlashInferBackend.get_dtype_for_flashinfer` 中映射为 `torch.uint8`。<br>• 增加 `KVQuantMode.NVFP4` 标记，`FullAttentionSpec` 现在携带 `kv_quant_mode`。<br>• 关键路径（prefill / decode）在检测到 `is_kvcache_nvfp4` 时，强制使用 `trtllm‑gen` 后端，因为 FA2/FA3 不支持该布局。<br>• 为 NVFP4 添加专用 **FP8 output缓存** (`_nvfp4_fp8_out`) 以匹配 kernel 只能输出 FP8 的限制。 |
| **性能影响** | • **内存**：NVFP4 使用 2‑bit（FP4）+ 8‑bit块尺度，较 FP8（8 bit）节省约 **75 %** 的 KV‑cache 带宽/显存，能显著提升大模型、长上下文的可容纳 token 数。<br>• **算子延迟**：额外的 **reshape_and_cache_flash** 量化步骤以及后续的 **FP8→bfloat16** 解量化会增加少量 CPU‑GPU 同步开销，但在大批量推理场景中相对显存节约的收益更显著。<br>• **量化误差**：测试中设定 `rtol≈1.0, atol≈1.0`，说明 NVFP4 的相对误差比 FP8‑FP4 组合高数倍，可能导致生成质量轻度下降。 |
| **安全考虑** | • 变更仅涉及数值压缩与 kernel 参数，不引入外部 I/O、文件或网络交互，安全风险极低。<br>• 需要确保 **scale 值**（amax/448）在 `float32` 范围内，防止溢出导致非法 NaN/Inf；现有实现使用 `torch.tensor(..., dtype=torch.float32)`，已规避。 |
| **可维护性** | • 新增多个辅助函数（`nvfp4_kv_cache_full_dim`, `build_paged_kv_metadata`, `make_nvfp4_kv_cache` 等），代码路径集中在 `tests/...` 与 `vllm/v1/attention`，模块化程度良好。<br>• 通过统一的 `KVQuantMode` 与 `KV_CACHE_QUANT_ALGOS`，后续若要加入其他 KV‑cache 量化方式（如 INT4）可复用现有结构。<br>• 文档生成脚本已同步排除 `nvfp4` 在非‑SM100 兼容表格中，降低误导用户的概率。 |

---

### ⚠️ 潜在风险  

1. **兼容性风险**  
   - `nvfp4` 只能在 **SM100（Blackwell）** GPU 上运行；如果用户在较旧硬件上仍通过 `--kv-cache-dtype nvfp4` 启动，后端会在运行时抛出 `NotImplementedError` 或 CK kernel 找不到的错误。  
   - 与 **MLA（Multi‑head Latent Attention）** 后端不兼容，已在 `VllmConfig.validate_nvfp4_kv_cache_with_mla` 做检查，但如果用户自行修改配置或通过旧版 CLI 参数跳过验证，仍可能触发运行时错误。  

2. **量化误差导致生成质量下降**  
   - 由于 FP4 仅 2 bit精度，量化误差相对较大（rtol/atol≈1），在对数值敏感的任务（如数值推理）中可能产生可感知的质量回退。  

3. **后端选择错误**  
   - 当 `kv_cache_dtype="nvfp4"`，`FlashInferBackend._get_prefill_wrapper` & `_get_decode_wrapper` 会强制 `backend="trtllm-gen"`。如果未来加入新的非‑TRTLLM 实现而忘记更新这些分支，可能导致未使用最优实现或出现不支持的错误。  

4. **尺度（scale）管理**  
   - `make_nvfp4_kv_cache` 通过全局 `kv_scale_val = amax/448` 计算标量，若 `amax` 为 0（极端全零 KV），会产生 `scale=0` 导致后续除以 0 的情况。虽然概率极低，但缺少显式保护。  

5. **多线程/并发**  
   - `_nvfp4_fp8_out` 为每个 `FlashInferImpl` 实例共享的临时缓冲区。若同一实例被多线程并发调用（例如在分布式推理的 DCP 场景），可能出现写冲突。当前实现基于单 GPU 单实例的假设，需文档明确。
