### [CI/Build] Skip terratorch + torchgeo while PyPI has lightning quarantined (#41377)
**SHA**: `10558f5` | 🔗 [查看提交](https://github.com/vllm-project/vllm/commit/10558f5f4608f825d06018def1317e7d3a96d6fe)

**🎯 变更类型**：架构变更  
**⚡ 重要程度**：🔴高  

**📋 变更摘要**  
- 由于 PyPI 对 `lightning` 包实施了 **quarantine**，导致所有间接依赖 `lightning` 的项目（如 `terratorch` 与 `torchgeo`）在依赖解析时失败。  
- 本次提交在 `requirements/test/*.in/.txt` 中 **注释掉** `terratorch` 与 `torchgeo`，并在相应测试文件里加入 **条件跳过**（`importorskip` / `skipif`）逻辑，以避免 CI 失效。  
- 目标是保持 CI 正常运行，同时保留代码路径，待 PyPI 恢复正常后可一键恢复这些依赖。

**🎯 影响范围**  
- **依赖管理**：`requirements/test/cuda.in`, `requirements/test/rocm.in` 以及生成的 `*.txt`。  
- **测试套件**：`tests/entrypoints/openai/chat_completion/test_vision_embeds.py`, `tests/models/multimodal/pooling/test_prithvi_mae.py`, `tests/models/test_terratorch.py`, `tests/plugins_tests/test_terratorch_io_processor_plugins.py`.  
- **可选特性**：Prithvi 系列模型及基于 `torchgeo` 的地理空间任务的测试与 CI 验证。

**🔍 技术洞察**  

- **架构影响**  
  - 暂时 **剥离** 了对 `terratorch`、`torchgeo`（以及它们的链式依赖 `lightning`）的硬性依赖，使项目的 **核心库** 与这些可选模块解耦。  
  - 代码本身未改动，仅在 CI 环境和测试入口处加入运行时检查，保持功能完整性，防止因缺失依赖导致 **ImportError**。  
  - 这种做法在长期看来提示了项目可以将此类 **非核心依赖** 移入 **可选 extras**（如 `vllm[geospatial]`），从而在正式发布中避免类似阻断。

- **性能影响**  
  - 对 **运行时性能** 没有直接影响，因为 `terratorch`/`torchgeo` 本身是 **可选** 的模型路径。  
  - CI **构建时间** 与 **依赖解析时间** 将显著下降，省去因 `lightning` quarantine 导致的解析错误与重试。

- **安全考虑**  
  - 移除 `lightning`（已被 PyPI 标记为 quarantine）可间接降低对 **潜在安全风险**（如受影响的旧版本）暴露的机会。  
  - 未引入新代码或新依赖，安全面没有新增攻击面。

**⚠️ 潜在风险**  

1. **功能退化**：用户在本地或 CI 环境未显式安装 `terratorch`/`torchgeo` 时，会失去对 Prithvi 等地理空间模型的测试覆盖，潜在回归风险不易捕获。  
2. **文档/期望不一致**：如果 README 或模型卡仍声称支持 Prithvi，而实际 CI 已跳过，可能导致用户困惑。  
3. **忘记恢复**：后续如果 `lightning` 问题自行解决，维护者可能忘记撤销注释，导致长期缺失功能。  
4. **依赖漂移**：手动注释的方式比使用 `extras_require` 更容易在以后合并冲突时产生不一致。

**💡 关注建议**  

- **文档更新**：在项目的 **Installation** 与 **Supported Models** 页面添加脚注，说明 “在 PyPI `lightning` quarantine 时，Prithvi/torchgeo 相关功能暂不可用”。  
- **使用可选 extras**：将 `terratorch` 与 `torchgeo` 移入 `vllm[geospatial]`（或类似）extras，配合 `pip install "vllm[geospatial]"`，可在正常情况下自动拉取，在 quarantine 时通过 `--no-deps` 或手动省略。  
- **自动化提醒**：在 CI 中加入脚本检测 `requirements/test/*.in` 中的 `# terratorch` 注释是否仍然存在，若检测到 `lightning` 已恢复（如 `pip install lightning` 成功），则 **fail** CI 并提示恢复依赖。  
- **测试覆盖**：在本地开发环境中仍保留完整测试套件，确保 PR 合并前通过本地完整 run，防止因 CI 跳过导致回归。  
- **追踪 Issue**：持续关注 Issue #41376，待 PyPI 状态恢复后，及时解除注释并删除对应的 `importorskip` 逻辑，保持代码库的整洁。  

---  

*此分析聚焦于本次提交对项目架构、CI 稳定性及潜在维护风险的影响，帮助维护者在恢复依赖前作出明智决策。*

### [Refactor][kv_offload] KV Offloading maintainability improvements (#40538)
**SHA**: `c42981d` | 🔗 [查看提交](https://github.com/vllm-project/vllm/commit/c42981d034713da814913ebd2e53269346f3ecea)

**🎯 变更类型**：重构 / 可维护性提升  
**⚡ 重要程度**：🔴 高  
**📋 变更摘要**：  
本次提交将 KV Offload 相关的抽象接口与规格（`abstract.py`、`mediums.py`、`spec.py`）统一迁移至新模块 `vllm/v1/kv_offload/base.py`，并相应地对所有内部引用、实现类、单元测试做了批量更新。删除旧文件后，引入 `cpu/common.py` 作为 CPU 端 `LoadStoreSpec` 的实现，统一使用 `BlockIDsLoadStoreSpec` 基类。整体目标是提升代码组织结构、降低耦合度、便于后续特性扩展。

---

### 🎯 影响范围
| 受影响模块/组件 | 主要改动 |
|----------------|----------|
| **核心 KV Offload 抽象** (`abstract.py`, `mediums.py`, `spec.py`) | 删除并迁移至 `base.py`，统一导出 `OffloadKey、ReqContext、LoadStoreSpec、OffloadingManager、OffloadingSpec、GPULoadStoreSpec、CPULoadStoreSpec` 等 |
| **CPU Offload 实现** (`cpu/*`) | 更新 import 为 `base`，引入 `cpu/common.py`，部分策略文件（ARC、LRU、base）改用新的 `OffloadKey` 定义 |
| **调度器与工作者** (`distributed/kv_transfer/*`, `kv_offload/worker/*`) | 所有 `from …abstract`、`from …mediums`、`from …spec` 改为 `from …base` |
| **单元测试** (`tests/v1/kv_offload/*`, `tests/v1/kv_connector/*`) | 同步修改 import 路径，确保测试覆盖迁移后的接口 |
| **工厂入口** (`kv_offload/factory.py`) | 适配 `OffloadingSpec` 的新导入路径 |
| **文档/示例**（未在本提交中出现） | 需要同步更新引用路径 |

---

### 🔍 技术洞察

| 维度 | 影响分析 |
|------|----------|
| **架构影响** | - 将散落的抽象统一到 `base.py`，形成“KV Offload 核心层”。<br>- 新增 `cpu/common.py` 让 CPU 端规格实现与 GPU 端实现对称，提升模块化。<br>- 删除的旧文件之前在多个子包中分别被引用，迁移后形成单一来源，降低循环依赖风险。 |
| **性能影响** | - 迁移仅涉及 import 路径和类组织，运行时行为未变更。<br>- 新的 `BlockIDsLoadStoreSpec` 仍使用 `numpy.ndarray` 保存 block id，保持原有内存布局和访问速度。<br>- 由于 `OffloadingManager`、`OffloadingSpec` 接口实现未改动，实际 KV offload 的吞吐和延迟保持不变。 |
| **安全考虑** | - 代码变更未涉及权限检查、数据加密或外部交互，安全风险基本为 **0**。<br>- 但迁移导致的导入路径变化可能导致 **运行时 ImportError**，如果用户在项目外部直接引用旧路径（如 `vllm.v1.kv_offload.abstract`），将触发 **ModuleNotFoundError**，这会在生产环境中导致服务不可用。 |
| **可维护性** | - 所有 KV Offload 相关抽象集中管理，后续新增 medium（如 NVMe、RDMA）时只需在 `base.py` 添加实现即可。<br>- 对外统一的 `LoadStoreSpec` 基类消除了 `mediums.py` 与 `spec.py` 的重复实现。<br>- 测试同步更新，确保迁移后行为保持一致。 |
| **兼容性** | - 该改动是 **非兼容的 API 变更**（breaking change）。<br>- 旧的 import 路径在本次提交后已被删除，未提供兼容 shim。<br>- 若有外部项目或内部脚本仍使用 `vllm.v1.kv_offload.abstract`、`vllm.v1.kv_offload.mediums`、`vllm.v1.kv_offload.spec`，会直接报错。 |

---

### ⚠️ 潜在风险

1. **向后兼容性缺失**  
   - 直接使用旧模块路径的代码会在运行时崩溃。  
   - 影响范围包括外部插件、用户自定义 offloading 实现、以及历史的 CI 脚本。

2. **遗漏的 import**  
   - 大量改动后，某些未被测试覆盖的子包（如实验性或临时脚本）可能仍残留旧 import，导致隐藏的 `ImportError`。

3. **文档/示例不同步**  
   - 官方文档、README、示例代码如果仍引用旧路径，会误导使用者。

4. **包装层次变化**  
   - `OffloadingSpec` 初始化时的日志 `warning` 仍保留，未改变；但如果有代码依赖于该 warning 的捕获，行为会产生微小差异。

**总结**  

此次提交通过把 KV Offload 的抽象层统一到 `base.py`，显著提升了代码组织的清晰度和后续扩展的便利性，是一次高价值的结构性重构。唯一需要重点关注的是 **向后兼容性**：当前直接删除旧模块会对已有集成产生破坏。建议在正式发布前加入短期的兼容 shim 与 deprecation 警告，并完善文档与升级指南，以降低升级阻力。整体风险可控，收益远超潜在成本。

### [ROCm] ROCm DeepEP API updated to latest (#39721)
**SHA**: `121dbe7` | 🔗 [查看提交](https://github.com/vllm-project/vllm/commit/121dbe7a221d2bf6415caea0acb812c666a7665e)

**🎯 变更类型**：功能增强（ROCm DeepEP API 更新）  
**⚡ 重要程度**：🟡 中  
**📋 变更摘要**：  
1. 在 `docker/Dockerfile.rocm` 中更新了 ROCshmem 与 DeepEP 的分支，改用更高的 commit，新增对 `--rocm‑explicit‑ctx` 的构建标志，并在需要时自行编译最新 `rdma‑core` 以兼容新 ROCshmem。  
2. `vllm/distributed/device_communicators/all2all.py` 移除对 `current_platform.is_rocm()` 的条件判断，统一在所有平台上使用 `explicitly_destroy=True`（以及低延迟模式下的 NVLink/MNNVL 选项），简化平台分支。  
3. `vllm/model_executor/layers/fused_moe/prepare_finalize/deepep_ll.py` 同样去除对平台的分支，直接走统一的 `low_latency_dispatch` 参数路径，保持代码一致性。

**🎯 影响范围**：  
- Docker 镜像构建流程（`docker/Dockerfile.rocm`）  
- All‑2‑All 通信实现（`vllm/distributed/device_communicators/all2all.py`）  
- DeepEP‑MOE 调度与执行（`vllm/model_executor/layers/fused_moe/prepare_finalize/deepep_ll.py`）  

**💡 关注建议**

### [BUG] Two phase pause to prevent deadlock (#39366)
**SHA**: `0335316` | 🔗 [查看提交](https://github.com/vllm-project/vllm/commit/0335316a9ba245e5e82a20ef1b53ba3da108afd5)

**🎯 变更类型**：Bug 修复（两阶段 pause 的 deadlock）  
**⚡ 重要程度**：🟡 中（仅在使用 Data‑Parallel (DP) 且开启 Expert‑Parallel 时才会触发）  

**📋 变更摘要**  
1. 为 DP 引入 **两阶段 pause 协议**：  
   - `ParallelConfig.sync_dp_state` 用一次 `all_reduce` 同时同步「是否还有未完成请求」与「本轮是否已发出 pause」；  
   - `EngineCoreProc` 新增 `pending_pause`、`ignore_start_dp_wave` 等标记，只有在所有 rank 达成 pause 共识后才真正停机，并屏蔽后续的 `START_DP_WAVE`。  
2. `engine_idle_callback` 与 `resume_scheduler`、`_handle_client_request` 等入口改为依据新状态决定是否直接返回或注册 idle‑callback。  
3. 测试 **test_dp_pause_barrier_request_deadlock** 新增，模拟在 pause‑keep 状态下对单个 rank 发送 barrier、再向另一 rank 发送请求的竞争情形，验证死锁已被消除。  

**🎯 影响范围**  
- **核心调度层**：`vllm/v1/engine/core.py`（EngineCoreProc 的 pause/resume、idle‑callback、START_DP_WAVE 处理）。  
- **并行配置**：`vllm/config/parallel.py`（新增 sync_dp_state）。  
- **测试套件**：`tests/v1/distributed/test_async_llm_dp.py`（两段新测试）。  
- **只在 DP + Expert‑Parallel 环境下生效**，其他模式不受影响。

**💡 关注建议**  

1. **部署注意**  
   - 若使用 DP（尤其 2‑rank）并开启 `expert_parallel=True`，升级后 **pause_generation(mode="keep")** 将不再出现 “START_DP_WAVE 反复唤醒 → deadlock”。  
   - 仍需确认 `torch.distributed` 进程组的 `size` 与 `dp_group` 正确初始化，否则 `sync_dp_state` 仍会报错。  

2. **回退方案**  
   - 若出现与旧版不兼容的异常（如 `ignore_start_dp_wave` 误判导致请求被丢），可在启动时通过 `VLLM_CONFIG.pause_two_phase=False`（若实现）回退到旧逻辑，或临时关闭 `expert_parallel`。  

3. **性能监控**  
   - 两阶段协议在每 32 步会触发一次 `all_reduce`，对极端的大规模 DP 可能增加微小同步开销。建议在生产监控中观察 `engine.step_counter` 与 `dp_group` 的网络延迟。  

4. **代码审阅**  
   - `EngineCoreProc._pause_complete` 现在直接返回 `False` 并把 `pending_pause` 设为 `True`，确保 idle‑callback 必定被注册；若后续有自定义 pause 实现，需要保持该行为一致。  
   - `resume_scheduler` 中的 `if self.pending_pause or (self.engines_running and self.ignore_start_dp_wave): raise RuntimeError` 可能在异常路径抛出，建议在外层捕获并给出可恢复的错误信息。  

总体而言，此次提交通过统一的 DP 状态同步解决了 “在 pause‑keep 状态下发 barrier → START_DP_WAVE 重新唤醒 → deadlock” 的根本原因，提升了分布式推理的可靠性。对使用 DP 的用户建议尽快升级并在 CI 中跑新增的 deadlock 测试以确认环境兼容。
