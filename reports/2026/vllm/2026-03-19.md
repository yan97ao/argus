### [Misc] Cleanup more configs and processors (#37560)
**SHA**: `657855a` | 🔗 [查看提交](https://github.com/vllm-project/vllm/commit/657855ab417988834a4d0ff99de27fd66c6a6b3c)

**🎯 变更类型**：功能增强 / 重构 / 架构变更  
**⚡ 重要程度**：🔴高  
**📋 变更摘要**：  
- 将大量模型专属的配置、处理器实现抽离到统一的 `vllm.transformers_utils.configs/*` 与 `vllm.transformers_utils.processors/*` 包中，统一入口并精简 `model_executor/models/*.py` 中的导入路径。  
- 删除了多处旧的 **builder**（`_build_*_info`、`_build_*_processor`）函数，改为在多模态注册表直接注册 **Processor 类** 与相应的 **ProcessingInfo**。  
- 为 **Isaac**、**Kimi‑K2.5**、**Step‑3 VL** 等模型重新实现了完整的 `Processor`（包含图像预处理、patch 化、Token 替换等），并把大量图像处理逻辑从模型文件迁移到独立的 processor 模块。  
- 统一了 `SpeculatorsConfig` 的导出路径，完善了 `__all__`。  
- 通过 `cached_tokenizer_from_config` 替代手动 `get_tokenizer` → `cached_tokenizer` 流程，简化 token‑id 解析。

## 🔍 技术洞察

| 维度 | 影响 |
|------|------|
| **架构影响** | - **模块化提升**：所有模型专属的配置/处理器现在都有清晰的子包，易于维护、复用以及单独升级。<br>- **注册表简化**：`MULTIMODAL_REGISTRY.register_processor` 直接使用 `Processor` 类和 `ProcessingInfo`，去掉了冗余的 builder wrapper，降低了代码耦合度。<br>- **统一入口**：`vllm.transformers_utils.configs.__init__` 只做路径映射，外部代码仍然可以通过统一的 import (`from vllm.transformers_utils.configs import …`) 访问。 |
| **性能影响** | - **运行时**：迁移的代码（图像归一化、patchify、尺寸搜索）保持原有实现，未引入额外算子，性能基本持平。<br>- **启动时**：删除了大量冗余函数（`_make_writeable`, `extract_image_pil`, …）的重复定义，模块体积缩小，导入时间稍有提升。<br>- **缓存**：`cached_tokenizer_from_config` 直接返回已经缓存的 tokenizer，少了一次 `get_tokenizer` 调用，微幅加速。 |
| **安全考虑** | - 新增 `numpy`, `PIL`, `torchvision` 等依赖在处理器模块中，若运行环境中这些库的二进制组件被篡改，可能导致 **代码执行** 或 **内存泄漏**。但这些库本身为官方维护，风险与原实现相当。<br>- 所有外部文件读取均通过 `PIL.Image` → `np.asarray` → `torch.from_numpy`，已保留对 **只读 buffer** 的写入标记处理，避免 PyTorch 警告和潜在未定义行为。 |
| **可维护性** | - 代码行数大幅下降（如 `isaac.py` 从 480 行删至 ~30 行），逻辑更易追踪。<br>- `ProcessorMixin` 子类的 `attributes`、`tokenizer_class` 统一声明，降低了未来新增模型时的样板代码。<br>- `__all__` 明确导出，避免意外的内部符号泄漏。 |
| **兼容性** | - 公开的 import 路径保持不变（`vllm.transformers_utils.configs.*`），外部使用者不需要改动。<br>- 但 **内部** 依赖 `InputProcessingContext`、`BaseDummyInputsBuilder` 的旧 builder 已被移除；若有第三方插件仍调用这些私有函数，将导致 `ImportError`。<br>- `IsaacVisionEmbedding` 等后续代码已改用 `cached_tokenizer_from_config`，行为上保持一致，但在 **无缓存** 情况下第一次调用会返回 `None`，导致后续 `encode` 报错，需要确保 tokenizer 已被提前缓存。 |

## ⚠️ 潜在风险

1. **注册表行为改变**  
   - 原来的 `_build_*_processor` 包装会在首次调用时 **创建** `BaseMultiModalProcessor` 并可能携带 `cache` 参数。新注册方式直接实例化 `Processor`，如果有自定义 `cache` 实例化路径（如外部项目使用 `BaseMultiModalProcessorCache`），可能失效。  
   - 建议在项目内部所有 `MULTIMODAL_REGISTRY.register_processor` 调用后，跑一次 **集成测试**，确保 `processor` 实例化不依赖旧 wrapper 参数。

2. **Tokenizer 缓存依赖**  
   - `IsaacProcessingInfo._resolve_vision_token_id` 现在使用 `cached_tokenizer_from_config`，若 `model_config` **未** 触发 tokenizer 缓存（如在特殊的 `trust_remote_code=False` 环境下），会返回 `None` 并导致 `TypeError`。需要在模型加载前显式调用一次 `cached_tokenizer_from_config` 或在 `IsaacProcessingInfo.get_hf_processor` 前检查返回值。

3. **外部插件/脚本仍引用已删除 Symbol**  
   - 如有第三方代码依赖 `vllm.model_executor.models.isaac._make_writeable`、`extract_image_pil`、`_build_*_info` 等私有函数，会立即报 `AttributeError`。提供迁移文档或保留向后兼容的 shim（仅导入并抛出明确错误信息）会降低升级阻力。

4. **新增依赖版本冲突**  
   - `torchvision` 被引入到 `step3_vl` 处理器。若部署环境的 `torchvision` 与 `torch` 版本不匹配，可能出现 **ABI** 错误。确保 CI 环境锁定兼容版本或在 `requirements.txt` 中声明。

5. **资源使用**  
   - `IsaacImageProcessor.preprocess` 在大图（>60M 像素）会一次性读取为 `torch.Tensor`，仍保持原有上限。若用户传入更大的图片，会抛 `ValueError`，但错误信息仍保持原有行为。

### [Misc] Clean up processing logic (#37541)

- 在 `SkyworkR1V` 中直接复用 `BaseInternVLMultiModalProcessor` 与 `BaseInternVLDummyInputsBuilder`，删除了冗余的图像 token 计数、尺寸计算等实现。  
- 其它细微改动：删除未使用的 `Mapping, Sequence`，统一 `list(range(...))` 的写法，确保 `video_num_frames` 与元数据一致，改进了 `apply` 方法中对 `use_audio_in_video` 参数的处理逻辑。

- 为 `swap_states` 与 `condense` 增加覆盖不同 token 长度（including 0） 的单元测试。  
- 在高并发/多卡环境下跑一次性能基准，确认 swap 开销下降幅度符合预期。
