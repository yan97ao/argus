### [MoE Refactor] FusedMoE/MoERunner inversion refactor (#41184)
**SHA**: `dc68bd8` | 🔗 [查看提交](https://github.com/vllm-project/vllm/commit/dc68bd8c4199b00631fe71eb37313f406cc66ac1)

**🎯 变更类型**：🛠️ **架构重构 / 功能增强**  
（将原先的 `FusedMoE` 拆分成 **`RoutedExperts`（权重容器）** 与 **`MoERunner`（执行器）**，并同步更新所有模型、LoRA、量化、EPLB、路由等子模块的实现与调用）

---

## 📋 变更摘要
- 新增 `RoutedExperts`：专职保存 MoE 权重、提供统一的 `weight_loader`、负责 expert‑map、EP/EPLB 相关状态。  
- 新增 `MoERunner`：负责路由、调度、共享专家、流‑同步、模块间协同（前向、内部/外部路由、DPO‑overlap 等）。  
- `FusedMoE` 完全废除（仅保留导入别名），所有模型、LoRA、量化实现、测试、插件统一迁移到 `RoutedExperts`/`MoERunner`。  
- 大量 **路由类**（`BaseRouter`, `FusedTopKRouter`, `GroupedTopKRouter`, …）的构造签名去掉 `indices_type_getter`，改为在 `select_experts` 中直接接收 `topk_indices_dtype`。  
- `WeightLoader` 迁移到 `RoutedExperts`，并加入 **专家映射**、**padding‑compatible**、**BitsAndBytes**、**Compressed‑Tensors** 等复杂逻辑。  
- 为 **EPLB**、**Zero‑Expert**, **SharedExperts**、**LoRA** 添加兼容层/新属性，以兼容原有 `FusedMoE` 行为。  
- 多处 **模型文件**（Gemma‑4, Aria, Dbrx, Ernie45‑MoE, GLM4‑Moe, etc.）全部改为使用 `MoERunner`（类型 `MoERunner`）并在需要时显式指定 `routed_experts_cls`/`routed_experts_args`。  
- 新增 **权重重映射工具** `remap_moe_expert_weights`，解决旧 checkpoint 中 `experts.w13_weight` → `experts.routed_experts.w13_weight` 的兼容性。  
- 相关 **自定义 Op**、**Cudagraph 捕获**、**Warm‑up**、**DeepGemm**、**量化插件**等均同步改为基于 `MoERunner`/`RoutedExperts`。  
- **大量单元测试**也随之更新，确保功能等价。

### [Connector] Remove `P2pNcclConnector` (#44854)
**SHA**: `5add018` | 🔗 [查看提交](https://github.com/vllm-project/vllm/commit/5add018bebbd54471af32094923205fe4c9b7740)

**🎯 变更类型**：功能移除 / 架构变更  
**⚡ 重要程度**：🔴高  

**📋 变更摘要**  
- 从代码库中彻底删除了 **P2pNcclConnector** 及其完整实现（包括引擎、内存池、示例脚本、基准脚本以及对应的文档）。  
- 同时在 `KVTransferConfig` 中去掉了对 `kv_parallel_size` 必须为 2 的说明，并将其描述为普通并行实例数量。  
- 在 `KVConnectorFactory` 中注销了 `P2pNcclConnector` 的注册，使得该 connector 不再能够通过配置被创建。  

**🎯 影响范围**  
- **vllm/distributed/kv_transfer/kv_connector/**：所有与 `p2p`（P2pNccl）相关的文件（`__init__.py`, `p2p_nccl_connector.py`, `p2p_nccl_engine.py`, `tensor_memory_pool.py`）全部删除。  
- **benchmarks/disagg_benchmarks/**：两套用于评估 P2p NCCL 的基准脚本全部删除。  
- **examples/disaggregated/**：包括 `disaggregated_prefill.py/sh`, `p2p_nccl_xpyd/*`‑相关脚本全部删除。  
- **docs/**：`design/p2p_nccl_connector.md` 以及 `features/disagg_prefill.md` 中的 P2pNccl 章节被删减。  
- **vllm/config/kv_transfer.py**：对 `kv_parallel_size` 文档的细节更改。  
- **vllm/distributed/kv_transfer/kv_connector/factory.py**：注销注册。  

---

### 🔍 技术洞察

| 维度 | 影响分析 |
|------|----------|
| **架构影响** | - **KV 传输层**：P2pNcclConnector 是唯一实现 “点对点 NCCL + ZMQ” 方式的 connector，负责 1P1D（或更复杂的 XpYd）跨实例 KV‑cache 迁移。其删除意味着 vLLM 代码库不再提供基于 NCCL 的 KV‑cache 直接发送/接收能力。<br>- **插件化**：`KVConnectorFactory` 失去一个可选实现，其他组件（如 `LLM`、`KVTransferConfig`）仍然保持抽象，仍能通过其它已实现的 connector（如 `LMCacheConnectorV1`、`MooncakeConnector`、`MoRIIOConnector`）工作。<br>- **服务发现/代理**：`disagg_proxy_p2p_nccl_xpyd.py` 与配套的 round‑robin、prefill‑proxy 代码全部删除，相关的 ZMQ 注册/心跳机制不再需要维护。 |
| **性能影响** | - **正面**：删除了一个实验性、实现复杂且在大规模部署（如 96P+144D）上已知会产生显著 GPU memory overhead（每个 NCCL group ~50‑100 MiB）的问题，降低了库的运行时内存占用和启动开销。<br>- **负面**：对于依赖此 connector 的用户（主要是内部/实验项目），失去了在单 GPU 上实现 **prefill‑decode 分离** 并利用 NCCL 进行 KV‑cache 直接点对点传输的能力，进而失去潜在的 **PUT_ASYNC → PUT → GET** 三种传输模式的性能对比与最高吞吐/低延迟选项。 |
| **安全考虑** | - 删除了大量使用 **ZeroMQ** 与 **NCCL** 的网络/进程间通信代码，降低了因 ZMQ 消息注入或 NCCL 连接泄漏导致的潜在攻击面。<br>- 仍然保留 `MoRIIOConnector`（ROCm 专用）中对 `request_id` 中嵌入 ZMQ 地址的解析逻辑，已做好注释，安全影响有限。 |
| **可维护性** | - 大幅简化了代码基线：约 3000 行 Bash/Python 示例、文档以及 1 KB‑级的实现代码全部去除，代码库更易阅读、构建和 CI。<br>- 失去的功能被标记为 “experimental” 并在 PR 标题里已经说明，是一次 **技术债务清理**，后续不必维护这套高度耦合的实现。 |
| **兼容性** | - **向后兼容性破坏**：任何在生产或实验环境下显式配置 `kv_connector="P2pNcclConnector"`（包括通过 `--kv-transfer-config`）的启动方式将立刻报错 `KVConnectorFactory` 未注册。<br>- 框架本身仍然可以在不指定该 connector 的情况下正常运行，默认回退到 `LMCacheConnectorV1` 或其它已注册的实现。 |
| **资源与部署** | - 删除了对 **NCCL** 动态加载 (`pynccl_wrapper`)、`msgpack`、`zmq` 的运行时依赖，降低了部署包体积和外部库冲突概率。<br>- 受影响的 CI 测试和镜像构建脚本需相应去除 `p2p` 相关的安装步骤（如 `apt-get install libzmq3-dev`、`pip install zmq` 等）。 |

### ⚠️ 潜在风险

1. **用户升级破坏**  
   - 已经在生产或实验环境使用 `P2pNcclConnector` 的用户在升级到此提交后会遇到 `KVConnectorFactory` 未找到对应实现的错误，导致服务启动失败。  
   - 需要在发布说明中明确告警，并提供迁移指南（切换到 `MooncakeConnector` / `LMCacheConnectorV1` 或自行实现自定义 connector）。  

2. **文档残留**  
   - `docs/features/disagg_prefill.md` 中仍保留对 P2pNcclConnector 的链接（虽然已删除），可能导致阅读者点击失效的 URL。必须同步清理或在文档中加上 “已移除” 的标注。  

3. **内部依赖遗漏**  
   - `vllm/distributed/kv_transfer/kv_connector/v1/moriio/moriio_common.py` 中的注释提到 “router embeds both zmq_addresses in the request_id (similar to P2pNcclConnector)”。虽然只是注释，但如果有逻辑依赖该格式的解析，需确认其他 connector（如 MoRIIO） 已经实现兼容或更新对应文档。  

4. **残余的测试用例**  
   - 项目自带的单元测试或集成测试中若仍有针对 `P2pNcclConnector` 的测试文件，CI 将报错。需要同步删除或标记 `xfail`.  

5. **二进制兼容性**  
   - 由于删除了 `p2p_nccl_engine` 中对 NCCL 的 `ncclCommInitRank`、`ncclSend/Recv` 调用，编译时不再需要链接 NCCL 动态库。如果有第三方插件自行调用这些内部函数，可能会出现 `ImportError`。

| **架构影响** | - 引入 XPU‑specific 代码路径，使 DeepSeek‑V4 在 XPU 上拥有独立的模型类、注意力实现和 MTP（多令牌预测）实现。<br>- 通过平台检测 (`current_platform.is_xpu()`) 在运行时动态切换实现，保持了统一的模型注册接口。<br>- 新增的 `deepseek_v4_xpu_attention` 继承自抽象的 `DeepseekV4Attention`，但在构造函数中临时把 `torch.cuda.Event` 替换为 `torch.xpu.Event`，保证父类的 Event 创建不报错。<br>- MegaMoE 权重在 XPU 上仍通过第三方 `deep_gemm` 完成转换；但 `DeepseekV4MegaMoEExperts._check_runtime_supported` 直接抛出 `NotImplementedError`，表明当前 XPU 并不支持该路径，仍保留接口以防以后实现。 |

**⚠️ 潜在风险**  

1. **性能退化**：XPU 解码路径依赖即时解量化，可能在高并发、长上下文场景下出现显著吞吐下降。  
2. **内存峰值**：`workspace` 缓冲区（`num_tokens * K_total * 512 *2B`）在批量极大时可能导致显存 OOM。  
3. **不完整特性**：`DeepseekV4MegaMoEExperts._check_runtime_supported` 暂未实现，若用户开启 `--kernel-config moe_backend=deep_gemm_mega_moe` 在 XPU 上会直接崩溃。

5. **权重尺度转换**：如果未来出现除 `e8m0` 之外的其他 float8 布局，转换代码可能遗漏，导致稀疏注意力进入错误路径并产生 NaN。  
6. **自定义 OP 注册**：`deepseek_v4_mega_moe_experts` 的 fake 实现返回 `None`，在 `torch.compile` 场景下若不正确剔除，可能导致未定义行为。  

**💡 关注建议**

3. **特性开关**：对未实现的 MegaMoE，提供明确的配置错误提示或在 CLI 中隐藏对应选项，避免用户误用导致崩溃。

**SHA**: `ac3409d` | 🔗 [查看提交](https://github.com/vllm-project/vllm/commit/ac3409d162a92c4f57ccfd6ed37b7304b1e5747f)

**⚡ 重要程度**：🟡 中  

**📋 变更摘要**

2. **网络开销**：对每个请求会额外进行一次 `/tokenize`（以及必要时的 `/detokenize`）调用，最高并发受 `Semaphore(64)` 限制。对于大规模基准（数十万条请求）可能显著拖慢准备阶段，建议在 CI/本地运行时提供 `--skip-tokenizer-align` 之类的开关。

4. **SSL 与鉴权**：函数接受 `ssl_context`，但未传递 `headers`（如 token），若服务端需要鉴权会请求失败。若以后需要在 `/tokenize`/`/detokenize` 添加认证，请在函数签名中加入 `headers` 参数。  
5. **性能**：使用 `asyncio.gather` 并发执行对齐，返回时保留异常对应的原请求，保证不会因单条失败导致整体崩溃。可以进一步通过 `batch_tokenize`（一次请求多条）来减少 HTTP 调用次数。  
6. **测试**：新增单元/集成测试，验证：  
   - 本地 tokenizer 与服务端一致时，函数直接返回原列表（无额外请求）。  
   - 不一致时，返回的 `prompt` 已被截断并能够通过 `len(tokenize(prompt)) == prompt_len`。  
   - 在 `/tokenize` 不可达、/detokenize 报错、或 `prompt` 非字符串时的容错路径。  

总体来看，此改动提升了随机数据集基准的可靠性，避免因 client‑server tokenizer 不匹配导致的统计偏差，但也引入了额外网络开销和潜在的鉴权兼容问题，建议在文档中提供使用说明并考虑添加可选开关。
