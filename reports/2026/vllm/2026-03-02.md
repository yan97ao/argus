### [Kernel] Integrate SM100 MXFP8 blockscaled grouped MM and quant kernels (#34448)
**SHA**: `cb21972` | 🔗 [查看提交](https://github.com/vllm-project/vllm/commit/cb21972a976b01e5ee1d899ab744207826435684)

**🎯 变更类型**：功能增强 / 性能优化 / 架构变更  
**⚡ 重要程度**：🔴高  

**📋 变更摘要**  
本次提交在 vLLM 中为 SM100（Hopper 及后续 GPU）实现了 **MXFP8（float‑8）块级缩放的分组矩阵乘（Grouped MM）** 与 **专家专化量化（Experts Quant）** 两套 CUDA CUTLASS kernel，并在 CMake、torch‑bindings、Python API 以及单元测试层面完成了全链路集成。该特性面向 MoE（Mixture‑of‑Experts）场景，能够在保持模型精度的前提下显著降低算力与显存需求。

---

### 🎯 影响范围
| 受影响模块 | 具体文件/入口 |
|------------|----------------|
| **构建系统** | `CMakeLists.txt` – 追加针对 CUDA ≥ 12.8、SM100+ 的 `-DENABLE_ES_MXFP8_GROUPED_MM_SM100` 编译宏；新增源文件列表。 |
| **核心 CUDA kernel** | `csrc/moe/mxfp8_moe/*.cu`、`*.cuh`（共 5 + 4 个文件）实现量化、预计算、Grouped MM 以及 Traits。 |
| **Torch C++ 接口** | `csrc/torch_bindings.cpp` – 注册 `mxfp8_experts_quant` 与 `cutlass_mxfp8_grouped_mm`。 |
| **Python 包装层** | `vllm/_custom_ops.py` – 新增 `mxfp8_experts_quant`、`cutlass_mxfp8_grouped_mm` 的 Python 调用与假实现（用于 CI）。 |
| **测试** | `tests/kernels/moe/test_cutlass_mxfp8_grouped_mm.py` – 完整端到端功能与数值误差验证。 |
| **可能受影响的上层 MoE 代码** | 任何调用 `ops.cutlass_mxfp8_grouped_mm` / `ops.mxfp8_experts_quant` 的 MoE 路由实现。 |

---

### 🔍 技术洞察

#### 架构影响
- **新增算子层**：在 MoE 框架中加入 FP8‑blockscaled GEMM 与对应的量化路径，形成“专家‑专化”流水线，替换原有 FP16/BFloat16 GEMM。
- **编译分支**：仅在 **CUDA ≥ 12.8 且检测到 SM100 (10.0 f, 11.0 f)** 的目标架构时才会编译；否则保持原有路径不变，保持向后兼容。
- **CUTLASS 依赖**：利用 CUTLASS SM100 Block‑Scaled Tensor‑Op (`cutlass::arch::OpClassBlockScaledTensorOp`) 与新加入的 `float_e4m3_t`（FP8）类型，扩展了 `cutlass::gemm::device::GemmUniversalAdapter` 的特化。
- **内存布局**：引入 `expert_specialization::CutlassMxfp8GroupedMmOffsetFunctor`、`LayoutFunctor`、`StrideFunctor`，在 kernel 启动前完成 **per‑expert 指针、stride、layout** 的一次性预计算，降低 kernel 内部分支与地址计算开销。
- **TMA + shared‑memory pipeline**：量化 kernel (`mxfp8_experts_quant_kernel`) 使用 CUDA PTX 异步拷贝 (`cp_async_bulk_*`) 与共享内存做 **scale‑factor** 传递，提升隐藏延迟。

#### 性能影响
| 场景 | 预计收益 | 关键实现 |
|------|----------|----------|
| **MoE 前向推理（大量小专家）** | **算力降低 2‑3×**（FP8 对比 FP16）<br>**显存占用降低 50%‑75%**（每个元素 8 bit） | - 使用 **block‑scaled** FP8（每 128 元素统一缩放）避免 per‑tensor 量化误差累计。<br>- 分组 GEMM 统一调度，减少 kernel 启动开销。 |
| **量化阶段** | **一次性量化**（GPU 内核完成）→ **CPU <-> GPU 互传次数下降** | - `mxfp8_experts_quant` 将 FP16/BF16 → FP8 并输出 blockscale，全部在 GPU 上完成。 |
| **编译/执行开销** | **编译时间略增**（CUTLASS 模板实例化较多）<br>**运行时 negligible**（仅在 SM100+ 可用） | - 通过 `-DENABLE_ES_MXFP8_GROUPED_MM_SM100` 控制编译，仅在支持的硬件上开启。 |

#### 安全考虑
- **输入校验**：新增 kernel 在入口处使用 `TORCH_CHECK` 验证张量维度、dtype、对齐 (k、n 必须 128 整除) 与 stride（确保 Row‑major / Column‑major），防止非法指针导致 GPU Crash。  
- **无外部依赖**：全部实现基于 CUDA intrinsics 与 CUTLASS，无网络或文件 I/O，安全风险极低。  
- **数值溢出**：FP8 量化采用 `__nv_cvt_float_to_e8m0` 并在 `reciprocal_approximate_ftz` 中使用 `ftz`（flush‑to‑zero）防止 NaN/Inf 传播；仍需关注极端数值（如 0 → Scale = 0）对下游 GEMM 的影响。

---

### ⚠️ 潜在风险
1. **硬件/驱动限制**  
   - 只在 **SM100 (CUDA ≥ 12.8)** 上可用，若用户使用 Hopper‑前代 GPU（如 SM80、SM90），编译会自动跳过，但运行时若误调用会触发 `torch.ops._C` 未注册的错误。
2. **对齐要求**  
   - `k` 与 `n` 必须对齐 128，`m` 对齐要求在量化阶段通过 `align` 处理，若上层代码提供不满足对齐的张量会直接报错，可能导致兼容性回退失败。  
3. **数值误差**  
   - FP8 采用 **块级缩放**，当单个块内数值分布极不均匀时，量化误差可能放大，进而影响 MoE 的路由与输出精度。当前测试仅在随机正态分布下通过，建议在实际模型（尤其是稀疏/长尾分布）上做更严格的回归。  
4. **编译依赖**  
   - 新增的 CUTLASS 头文件与模板实例化会显著增大编译时间和生成的二进制体积，CI 可能需要更大缓存或更长的构建时长。  
5. **维护成本**  
   - 代码高度依赖 CUTLASS SM100 特性（`OpClassBlockScaledTensorOp`、`float_e4m3_t`），CUTLASS 升级时可能出现兼容性破坏，需要同步更新 `*_traits.cuh` 与 `*_functor.cuh`。

### [Attention] FA4 integration (#32974)
**SHA**: `8b5014d` | 🔗 [查看提交](https://github.com/vllm-project/vllm/commit/8b5014d3dd343736ccf3e26cd44a0bb7700d205c)

**🎯 变更类型**：功能增强 / 架构变更 / 安全修复  
**⚡ 重要程度**：🔴高  

**📋 变更摘要**  
- 为 vLLM 引入 **FlashAttention 4 (FA4)** 的 CuteDSL 支持，包括新增 CMake 组件、Python 包装层、文档、依赖声明以及运行时选择逻辑。  
- 在 `get_flash_attn_version` 中加入对 Blackwell（SM100+）GPU 自动使用 FA4 的判断，并在不满足头维限制或与 ALiBi 不兼容时安全回退到 FA2。  
- 对 CI、文档、依赖、`setup.py`、`vllm` 代码多处改动，以让新组件可选、向后兼容并在缺失时给出明确警告。

---

**🎯 影响范围**  
- **核心模块**：`vllm/v1/attention/backends/*`（FlashAttention、MLA、FA utils）  
- **配置模块**：`vllm/config/attention.py`  
- **构建系统**：`cmake/external_projects/vllm_flash_attn.cmake`、`setup.py`、`vllm/vllm_flash_attn/*`  
- **文档/CI**：`docs/design/attention_backends.md`、`.buildkite/test_areas/misc.yaml`、启动脚本 `tools/pre_commit/generate_attention_backend_docs.py`  
- **运行时依赖**：`nvidia-cutlass-dsl`、`quack-kernels`

---

**🔍 技术洞察**  

| 维度 | 影响分析 |
|------|----------|
| **架构影响** | 1. 新增 **FA4 CuteDSL** 作为 *Python‑only* CMake 组件 (`_vllm_fa4_cutedsl_C`)；依赖 `add_custom_target` 与 `install(CODE …)` 实现源码级导入路径转换。<br>2. 将 FA2/FA3 安装前缀重写为 `…/vllm/`，并在安装后恢复，防止组件跨前缀混淆。<br>3. `vllm/vllm_flash_attn` 包现在统一导出 FA2/FA3/FA4 可用性检查，保持向后兼容。<br>4. `get_flash_attn_version` 接口签名扩展，引入 `head_size` 参数，所有调用点已同步更新。 |
| **性能影响** | - 在 **Blackwell (SM100+)** GPU 上，默认使用 FA4，预计比 FA3/FA2 提供 **~10‑20%** 的吞吐提升（取决于模型 head‑size）。<br>- FA4 对 **head size > 128** 存在 TMEM 限制，自动降级至 FA2，避免运行时崩溃但会失去性能提升。<br>- 对 **ALiBi** 不再支持 FA4，亦会回退到 FA2，保持功能正确性。<br>- 通过 `VLLM_WORKER_MULTIPROC_METHOD=spawn` 在 CI 中切换多进程启动方式，减轻因共享 CUDA 上下文导致的资源竞争。 |
| **安全考虑** | - 新增的 **外部 PyPI 依赖** `nvidia-cutlass-dsl>=4.4.0.dev1` 与 `quack-kernels>=0.2.7`，需审计其供应链（签名、维护者可信度）。<br>- `install(CODE …)` 脚本在安装阶段执行字符串替换 (`flash_attn.cute` → `vllm.vllm_flash_attn.cute`)；若源码被篡改，可能植入恶意代码。建议在发布渠道使用签名校验或 SHA256 校验。<br>- `fa_version_unsupported_reason` 等函数在缺失组件时抛出 `ImportError`，已在 `__init__` 中统一处理，避免因未安装 FA4 导致隐藏的 `ImportError`。 |
| **可维护性** | - 通过 `CMakeExtension(..., optional=True)` 使 FA4 成为 **可选** 组件，降低对不支持环境的强制依赖。<br>- 文档、脚本、配置均已同步更新，保持 **单一来源**（`generate_attention_backend_docs.py`）生成的注意力后端表格，降低手动错误概率。<br>- 代码中大量 `logger.warning_once` 与 `logger.error` 用于在不兼容场景下给出明确提示，提升可调试性。 |

---

**⚠️ 潜在风险**  
1. **构建复杂度提升**：FA4 需要 `nvidia-cutlass-dsl` 与 `quack-kernels`；在离线或受限环境下可能导致构建失败。  
2. **运行时回退不透明**：当 head size > 128 或使用 ALiBi 时，自动降级至 FA2；如果用户未关注日志，可能误以为在使用 FA4，导致性能预期偏差。  
3. **兼容性破坏**：`get_flash_attn_version` 的新参数 `head_size` 可能被外部插件或自定义代码以旧签名调用，引发 `TypeError`。  
4. **供应链风险**：`nvidia-cutlass-dsl` 为 **dev** 预发布版本，API 可能不稳定；升级时可能导致二进制不兼容。  
5. **安装路径混淆**：多次修改 `CMAKE_INSTALL_PREFIX`，若用户自行 `cmake --install` 某子组件，可能出现 **路径冲突** 或 **文件覆盖**。  
6. **文档/CLI 不一致**：`--attention-config.flash_attn_version` 默认值已改为 “FA4 on SM100+”，但旧版二进制仍可能返回 FA3；需要确保二进制与文档同步发布。

---

**💡 关注建议**  
- **完善测试矩阵**：在 CI 中加入 Blackwell (SM100+)、Hopper (SM90) 与旧系列 GPU 的全套回归，特别验证 head size > 128 与 ALiBi 场景的回退路径。  
- **发布说明**：在 Release Notes 中明确 “FA4 仅在 Blackwell 及以上且 head size ≤ 128 时可用，使用 ALiBi 将自动回退”。并提供 `vllm --check-float-attn` 类的诊断工具。  
- **依赖锁定**：为 `nvidia-cutlass-dsl` 与 `quack-kernels` 添加 **exact version**（或 `>= x, < y`）的 `requirements.txt` 约束，防止未来不兼容升级。  
- **安全审计**：对 `install(CODE …)` 中的字符串替换进行 SHA256 校验或使用 CMake `file(DOWNLOAD … VERIFYSHA256 …)` 机制，确保源码未被篡改。  
- **向后兼容包装**：在公共 API（如 `vllm.get_flash_attn_version`) 中保留旧签名兼容层，抛出友好错误信息或自动填充 `head_size=None`。  
- **用户可选关闭**：提供环境变量
