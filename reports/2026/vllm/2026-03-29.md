### [Transformers v5] fix missing pixtral/voxtral multimodal dispatch (#38410)
**SHA**: `8c0b626` | 🔗 [查看提交](https://github.com/vllm-project/vllm/commit/8c0b6267d7fa5c8a07e318809180fc021a0afbf2)

**🎯 变更类型**：功能增强（兼容 Transformers v5，修复 Pixtral/Voxtral 多模态调度缺失）  
**⚡ 重要程度**：🟡 中  

**📋 变更摘要**  
1. 为 Pixtral 与 Voxtral 模型分别新增 `get_image_processor` / `get_feature_extractor`，在 `get_hf_processor` 中显式注入这些组件。  
2. 调整 `MistralCommonPixtralProcessor` 与 `MistralCommonVoxtralProcessor` 构造函数签名，改为接受外部创建好的 `image_processor` / `feature_extractor`，去除内部硬编码。  
3. 相关代码路径（如 `get_image_size_with_most_features`、`get_data_parser`、`get_dummy_processor_inputs`、静态 `get_num_audio_tokens`）同步改用新接口。  

**🎯 影响范围**  
- `vllm/model_executor/models/pixtral.py`、`voxtral.py`（模型执行层）  
- `vllm/transformers_utils/processors/pixtral.py`、`voxtral.py`（处理器实现）  
- 可能影响所有直接调用 `MistralCommonPixtralProcessor`、`MistralCommonVoxtralProcessor` 构造函数的自定义插件或测试。  

**💡 关注建议**  
1. **兼容性检查**：项目中仍有 `ctx.init_processor(... )` 的调用方式。若其他模型仍依赖该入口，需要保持 `init_processor` 的兼容实现或在 `init_processor` 中适配新参数。  
2. **依赖更新**：确保 `transformers` 升级到 v5 时 `mm_encoder`、`audio_encoder` 接口保持不变；若未来再次改动，建议在 `MistralCommon*Processor` 中加入容错逻辑（如属性检查或回退）。  
3. **单元测试**：补充 Pixtral/Voxtral 的多模态输入测试，验证图像/音频处理器被正确注入，且 `get_num_*_tokens` 与 `get_image_size_with_most_features` 返回预期值。  
4. **文档与示例**：更新模型文档，说明新获取处理器的方式（`model.get_image_processor()` / `model.get_feature_extractor()`），并演示如何在自定义推理管线中手动构造处理器。  

总体来说，此次改动解决了 v5 中缺失的多模态调度问题，结构更清晰但需要留意向后兼容和相应测试的覆盖。

### [ROCm][CI] Fix cross-attention dispatch for encoder-decoder models (#38450)
**SHA**: `43cc513` | 🔗 [查看提交](https://github.com/vllm-project/vllm/commit/43cc5138e5145752413235a2a8aa303886077327)

**🎯 变更类型**：Bug 修复 / 功能增强  
**⚡ 重要程度**：🟡 中  

**📋 变更摘要**  
1. 修正了 ROCm 上 Encoder‑Decoder（跨注意力）模型的注意力分发逻辑：`ROCM_AITER_FA` 与 `ROCM_ATTN` 均声明不支持 `ENCODER_DECODER`，防止错误的 cross‑attention 计算。  
2. 增加 CI 测例，在 ROCm 环境下分别使用默认、`TRITON_ATTN` 与 `ROCM_AITER_UNIFIED_ATTN`（仅在 MI3xx GPU）进行验证。  
3. 细化日志输出，统一在 `--attention-backend` 显式指定时仅记录一次，并在后台被覆盖时给出详细原因。  
4. 文档生成脚本与注意力后端表格同步更新。

**🎯 影响范围**  
- `vllm/v1/attention/backends/rocm_aiter_fa.py`、`vllm/v1/attention/backends/rocm_attn.py`（注意力后端实现）  
- `vllm/platforms/rocm.py`（后端选择与日志）  
- CI 测试 `tests/entrypoints/openai/speech_to_text/test_transcription_validation_whisper.py`  
- 文档 `docs/design/attention_backends.md`、生成脚本 `tools/pre_commit/generate_attention_backend_docs.py`

**💡 关注建议**  
- **用户**：在 ROCm 环境下，若使用 Encoder‑Decoder 模型，请避免 `--attention-backend ROCM_ATTN` 或 `ROCM_AITER_FA`，改用 `TRITON_ATTN`（通用）或在 MI3xx GPU 上使用 `ROCM_AITER_UNIFIED_ATTN`。  
- **部署**：启用 AITER 后端需设置环境变量 `VLLM_ROCM_USE_AITER=1` 与 `VLLM_ROCM_USE_AITER_MHA=1`（后者仅对 `ROCM_AITER_FA` 必要）。  
- **开发者**：注意日志已改为 `info_once`，在多次选择冲突时会额外打印被剔除的后端列表，便于调试。文档生成脚本已修正集合比较逻辑，确保注意力后端表格正确。  
- **回归**：运行完整的 CI（尤其是 ROCm CI）以确认跨注意力的输出一致性；在非 ROCm 平台仍保持默认行为不变。

### [Feature] Add Qwen3-ForcedAligner support via token classification pooling (#35367)
**SHA**: `d39b8da` | 🔗 [查看提交](https://github.com/vllm-project/vllm/commit/d39b8daf5f6d220c68bb402806ad11e7be748883)

**🎯 变更类型**：功能增强  
**⚡ 重要程度**：🟡 中  

**📋 变更摘要**  
- 为 Qwen3‑ForcedAligner（`Qwen/Qwen3-ForcedAligner-0.6B`）新增 **token‑classification pooling** 支持，实现音频‑文本强制对齐功能。  
- 添加了示例脚本、对应单元测试、文档章节以及模型注册、权重映射、配置字段等全链路实现。  

**🎯 影响范围**  
- `vllm/model_executor/models/qwen3_asr_forced_aligner.py`（新模型类）  
- `vllm/model_executor/models/registry.py`、`vllm/transformers_utils/configs/qwen3_asr.py`（模型/配置注册）  
- `vllm/model_executor/layers/pooler/tokwise.py`（使用 `pooler_for_token_classify`）  
- 文档 `docs/models/pooling_models/token_classify.md`、示例 `examples/.../forced_alignment_offline.py`  
- 测试 `tests/models/multimodal/pooling/test_qwen3_asr_forced_aligner.py`  

**💡 关注建议**  

1. **模型加载**：使用时必须在 `--hf-overrides` 中显式声明 `architectures=["Qwen3ASRForcedAlignerForTokenClassification"]`，否则会走普通生成路径。  
2. **配置兼容**：新增 `classify_num`、`timestamp_token_id` 与 `timestamp_segment_time` 必须随 HF 版本 ≥ 4.57 提供；CI 中应校验这三个字段存在。  
3. **权重映射**：`hf_to_vllm_mapper` 将 `thinker.lm_head` 替换为 `classifier`，若未来模型结构变化，需要同步更新映射规则。  
4. **推理模式**：强制对齐仅在 **pooling‑runner** 且 `pooling_task="token_classify"` 时可用，确保 `enforce_eager=True` 防止异步导致的 token‑id 对齐错误。  
5. **测试覆盖**：当前仅验证 logits 维度与时间戳数量，建议补充对 **timestamp_segment_time** 乘积结果的数值检查（如 0‑5 s 区间）。  
6. **文档同步**：注意文档中 “Pooling multitask 支持已在 v0.20 移除” 的提示，后续若迁移到单任务接口，请及时更新示例和说明。  

**对用户**：在使用强制对齐时准备 16 kHz 单通道音频（或自行实现多模态），并按示例构造 `<timestamp>` 包围的 prompt；若发现时间戳异常，请核对模型配置中的 `timestamp_token_id` 与 HF 版本。

### [BugFix][Frontend] apply task instruction as system prompt in cohere v2/embed  (#38362)
**SHA**: `fafca38` | 🔗 [查看提交](https://github.com/vllm-project/vllm/commit/fafca38adc1ce65d2c9e2857138c3c0d65b0905e)

**🎯 变更类型**：Bug 修复 / 功能增强  
**⚡ 重要程度**：🟡 中  
**📋 变更摘要**：  
1. 在 Cohere v2 Embed 接口中，将任务指令（task instruction）改为 **系统提示（system role）** 而非简单文本前缀，统一在有 chat 模板时使用 system‑prompt、无模板时仍保持前缀行为。  
2. 为此实现了 `_has_chat_template`、`_preprocess_cohere_text_completion` 等内部工具，并对 `EmbedIOProcessor._pre_process_cohere_online` 逻辑进行重构。  
3. 新增单条、批量以及输入类型的单元测试，并在 parity 测试中改用余弦相似度容忍 BF16 轻微数值漂移。  

**🎯 影响范围**：  
- `vllm/entrypoints/pooling/embed/io_processor.py`（核心处理逻辑）  
- `vllm/entrypoints/pooling/embed/protocol.py`（新增 `CohereEmbedContent`、`CohereEmbedInput` 引入）  
- `vllm/entrypoints/pooling/typing.py`（`PoolingServeContext` 仍被使用）  
- 新增/修改的测试文件 `tests/entrypoints/pooling/embed/*`  

**💡 关注建议**：  
- **向后兼容**：确认老版本模型在无 chat‑template 时仍能接受任务前缀，否则可能出现意外行为。  
- **性能**：`_has_chat_template` 每次调用会解析模板，若频繁调用建议缓存结果。  
- **测试**：CI 需要加入 BF16 环境的覆盖，以防止数值漂移导致偶发失败。  
- **文档**：更新 Cohere v2 Embed 接口说明，明确任务指令现以 system‑prompt 形式发送。  

总体而言，此次改动提升了多模态/指令嵌入的一致性与可维护性，但需关注模板解析的开销以及保持老模型的兼容性。
