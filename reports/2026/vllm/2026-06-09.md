### [Rust Frontend] Add /tokenize and /detokenize endpoints (#44222)
**SHA**: `69fdaff` | 🔗 [查看提交](https://github.com/vllm-project/vllm/commit/69fdaffbcd8cb406a9750685370a831fe8978a0a)

**🎯 变更类型**：功能增强、API 扩展、代码重构  

**⚡ 重要程度**：🔴高  

**📋 变更摘要**  
- 在 Rust 前端新增 `/tokenize` 与 `/detokenize` 两个 HTTP 端点，实现对提示词的 **编码/解码**（tokenization）功能，保持与 Python 端接口一致。  
- 为该功能添加完整的请求体结构体、校验逻辑、错误映射以及单元测试，覆盖普通完成、Chat 模板、特殊标记、模型校验等多种场景。  
- 通过 `ChatLlm::tokenize_chat` 将 chat‑template 渲染 → token 化的路径抽离到公共实现，同时将消息校验抽象为共享函数，提升代码复用度。  

**🎯 影响范围**  
- `rust/src/server/src/routes.rs`、`routes/tokenize.rs`、`routes/tokenize/types.rs`（新增端点及类型）  
- `rust/src/chat/src/lib.rs`（新增 `tokenize_chat` 方法）  
- `rust/src/chat/src/error.rs`（新增 Tokenizer 错误包装）  
- `rust/src/server/src/routes/openai/chat_completions/*`（公开 `validate_messages` 供 tokenize 使用）  
- 单元测试文件 `rust/src/server/src/routes/tests.rs`（大量 tokenization/ detokenization 场景）  
- 相关依赖模块：`vllm_text::tokenizer`、`vllm_chat` 渲染流水线  

---

### 🔍 技术洞察  

| 维度 | 影响分析 |
|------|----------|
| **架构影响** | - 新增 **独立的路由模块** `tokenize.rs`，遵循已有的 “ValidatedJson → ApiError → IntoResponse” 处理链路，保持与现有 OpenAI 兼容层的统一风格。<br>- `ChatLlm::tokenize_chat` 复用了 chat 渲染、multimodal 处理、模型 dtype 读取等核心路径，只在**生成前**截断，确保 token ID 与实际生成保持一致，避免对已有推理路径的侵入式改动。<br>- 将模型名称合法性检查抽象为 `check_model`，统一错误返回（404 + `model_not_found`），与其他路由保持一致。 |
| **性能影响** | - **无引擎调用**，仅在进程内使用 `DynTokenizer` 进行编码/解码，延迟极低（类似 Python 本地实现），对整体吞吐基本没有负面影响。<br>- 对 `tokenize_chat`，仍会执行模板渲染和 multimodal 特征解析，成本与实际 chat 生成相同，但因不运行模型推理，仍显著更快。<br>- 新增的 `token_strs` 选项会遍历返回的 token ID 并查询词表，复杂度 O(N) 且词表查询为 O(1) 哈希/直接映射，影响可忽略。 |
| **安全考虑** | - **模型校验**：在 `/tokenize` 与 `/detokenize` 中通过 `check_model` 防止对未加载模型的非法请求，避免潜在的资源泄露或异常路径。<br>- **输入校验**：共享的 `validate_messages` 对用户消息进行非空、内容合法性检查，防止空消息导致渲染异常。<br>- **错误映射**：所有错误均包装为 `ApiError`，统一返回 JSON，避免泄露内部堆栈信息。<br>- **特殊 token**：`add_special_tokens` 标记由调用方决定，未强制添加 BOS/EOS，降低意外触发模型特殊行为的风险。 |
| **可维护性** | - 将消息校验抽离为公共函数，减少代码重复，提升后续修改的一致性。<br>- 新增的 `tokenize`、`detokenize` 路由与其他路由结构相同，便于未来统一扩展（如新增 tokenizer 参数）。<br>- 单元测试覆盖 **正向、负向、边界** 场景，提供回归保障。<br>- `TokenizeRequest::Validate` 实现确保在反序列化后统一校验，保持接口安全性。 |
| **兼容性** | - 与 Python 端相同的根路径 `/tokenize`、`/detokenize`，确保已有使用 Python SDK 的用户无需迁移。<br>- 仍保持对旧的 OpenAI 兼容路由不受影响，增量功能不破坏现有业务。 |

### ⚠️ 潜在风险  

1. **模型未加载时的 404**  
   - 依赖 `AppState.served_model_names()` 检查模型是否已加载。若在运行时动态加载模型（如 LoRA 更新）而未同步到该列表，可能导致误报 404。  
2. **Tokenizer 实现差异**  
   - `DynTokenizer::encode` 与 Python tokenizer 可能在特殊 token（BOS/EOS、空格处理）上出现细微差异，导致两端 token ID 不完全一致，特别是在 `add_special_tokens = true` 场景。  
3. **错误信息泄露**  
   - `server_error!` 使用 `to_report_string()`，若底层错误携带敏感路径或模型名称，可能在日志或返回的错误消息中泄露。  
4. **并发状态**  
   - `tokenize_chat` 调用了 `self.backend.chat_renderer()` 与 `self.text.tokenizer()`，这些内部对象需保证 `Sync + Send`，否则在高并发请求下可能出现竞态或锁竞争。当前代码未新增额外锁，但需确认底层实现的线程安全。  
5. **API 版本冲突**  
   - 新增的根路径 `/tokenize`、`/detokenize` 与已有的 `/inference/v1/generate` 等路径冲突可能导致代理或负载均衡的路由规则误匹配。需在部署文档明确路径声明。

### [WideEP] Integrate DeepEP v2 (#41183)
**SHA**: `e2f993d` | 🔗 [查看提交](https://github.com/vllm-project/vllm/commit/e2f993dc4116ba96ea10ec40b7bd2ee45b27b2b8)

**🎯 变更类型**：功能增强 / 架构变更 / 性能优化  
**⚡ 重要程度**：🔴高  

**📋 变更摘要**  
本次提交在 vLLM 中新增对 DeepEP v2（ElasticBuffer）接口的完整支撑，完成了从检测、配置、通信层到 MoE 调度与算子实现的全链路集成。主要工作包括：  
1. 在 `import_utils` 中加入 `has_deep_ep_v2()`，并在运行时校验 NCCL ≥ 2.30.4。  
2. 扩展并行配置 `parallel_config`，加入 `deepep_v2` 选项。  
3. 实现 `DeepEPV2All2AllManager` 与 `DeepEPV2PrepareAndFinalize`，提供统一的 All‑2‑All 与 ElasticBuffer 生命周期管理。  
4. 为 FP8‑TrtLLM、FlashInfer 等后端添加对 DeepEP v2 的兼容路径，优化权重转换与专家切片逻辑。  
5. 新增大量单元测试（包括 manager 生命周期、完整 MoE 前向与 CUDAGraph 场景），并在 CI 中加入对应 test 入口。  
6. 添加环境变量、构建脚本改动以及文档层面的后端注册。  

**🎯 影响范围**  
- `vllm/distributed/device_communicators/*`（All2All 管理）  
- `vllm/model_executor/layers/fused_moe/*`（准备/完成、专家实现、配置）  
- `vllm/config/parallel.py`（新增 `deepep_v2` 后端标识）  
- `vllm/envs.py`（新增三个 DeepEP v2 相关调优开关）  
- `vllm/utils/import_utils.py`（运行时 NCCL 检查）  
- 多个测试目录（`tests/kernels/moe/*`、`tests/distributed/*`）  
- 构建脚本 `tools/ep_kernels/install_python_libraries.sh`（DeepEP commit 更新）  

**🔍 技术洞察**  

- **架构影响**  
  - 引入 ElasticBuffer 统一 API，使原有 DeepEP LL（低延迟/高吞吐）实现统一为 “deepep_v2”。  
  - 通过 `All2AllManagerBase` 子类实现，保持与已有 `MoriAll2AllManager`、`NixlEPAll2AllManager` 接口兼容，降低对上层调度器改动的侵入度。  
  - `prepare_finalize` 层对不同后端统一抽象，新增 `use_deepep_v2_kernels` 标志，使模型并行配置可以透明切换。  
  - 通过 `DeepEPV2PrepareAndFinalize` 实现两种运行模式（decode‑cudagraph 与 prefill‑非cudagraph），在同一类中统一调度逻辑，简化代码路径。  

- **性能影响**  
  - **通信**：ElasticBuffer 使用 NCCL GIN 后端，可在 NVLink+RDMA 环境下自动开启混合模式（可通过 `VLLM_DEEPEP_V2_ALLOW_HYBRID_MODE` 调整），对跨节点的大规模 MoE 任务显著降低延迟与带宽占用。  
  - **计算**：在 decode 场景下 `do_expand=False` + `do_cpu_sync=False` 允许完整 CUDAGraph 捕获，减少 host‑device 同步；在 prefill 场景下 `do_expand=True` 通过精准内存分配削减显存峰值。  
  - **量化**：对 FP8‑TrtLLM 后端优先使用，利用 TrtLLM 的 tile‑level 跳过技术避免对 padding 行进行冗余计算，配合 `VLLM_DEEPEP_V2_ALLOW_MULTIPLE_REDUCTION` 可在高压缩率下保持吞吐。  
  - **实验结果**：新增测试通过多个随机 `MNK`、`dtype`、`world_size` 参数组合，对比纯 PyTorch 实现误差在 `6e‑2` 以内，说明数值一致性已得到保障。  

- **安全考虑**  
  - 只在运行时检测到符合版本的 NCCL 后才启用 DeepEP v2，避免因库不匹配导致的段错误或未定义行为。  
  - `has_deep_ep_v2()` 实现对 NCCL 版本的安全解析，若获取失败则安全回退至原有后端。  
  - 所有新增的 C++/Python 调用均保持异常捕获（如在 `tests/distributed` 中的 `requires_deep_ep_v2` 标记），防止在缺失或不兼容的环境中挂起 CI。  

**⚠️ 潜在风险**  
1. **运行时依赖不确定**：DeepEP v2 需要 NCCL ≥ 2.30.4，若集群使用的是老版本 NCCL，后端会被隐藏，但用户若显式强制 `--backend deepep_v2` 仍可能触发运行时错误。  
2. **显存峰值波动**：在 decode‑cudagraph 模式下会进行最坏情况的张量预分配，可能导致显存占用瞬间激增，特别是在极大 `max_tokens_per_rank` 场景下。  
3. **多进程句柄泄漏**：`DeepEPV2All2AllManager.destroy()` 在异常路径未必被调用，若进程异常退出可能留下 NCCL/Ep 句柄，影响后续作业。  
4. **量化路径兼容**：FP8‑TrtLLM 依赖 `convert_to_fp8_moe_kernel_format`，若未来 FlashInfer/TrtLLM 后端更新其内部布局，可能导致权重转换失效。  
5. **调度冲突**：`use_deepep_v2_kernels` 与 `enable_eplb` 同时开启时会被 `TrtLlmFp8ExpertsModular._supports_parallel_config` 拒绝，需要用户明确配置。  

**💡 关注建议**  
- **CI 与部署**：在 CI 中继续保留 `has_deep_ep_v2()` 的 skip 判定，防止因环境差异导致 CI 失效。生产集群上线前，务必验证 NCCL 版本（`torch.cuda.nccl.version()` 与实际运行时库）满足要求。  
- **显存监控**：建议对大型推理任务开启 `VLLM_DEEPEP_V2_PREFER_OVERLAP=False`（默认），在显存紧张时手动切换到 `prefill` 模式或降低 `max_tokens_per_rank` 参数。  
- **资源回收**：在多实验脚本或长时间服务中显式调用 `all2all_manager.destroy()`，或使用 `with` 上下文包装 `DeepEPV2All2AllManager`，确保句柄及时释放。  
- **文档与用户指引**：在 vLLM 官方文档中补充 “DeepEP v2 使用指南”，包括环境变量、兼容后端、显存估算公式以及常见错误排查（如 NCCL 版本不匹配）。  
- **后向兼容**：保持 `deepep` 与 `deepep_v2` 两套实现并存，避免一次性迁移导致现有生产环境不可用；在未来版本中可考虑将 `deepep` 标记为 deprecated。

**🎯 变更类型**：Bug 修复 / 功能增强  
**⚡ 重要程度**：🟡 中  
**📋 变更摘要**：在 Model Runner V2 中加入对多模态（MM）LoRA 的激活逻辑。新增 `set_active_mm_loras` 实现对 **tower** 与 **connector** 两类 LoRA 的映射构建，并在 `GPUModelRunner.execute_model` 前调用；相应单元测试验证了映射结构和 `WorkerLoRAManager.set_active_adapters` 的调用次数。  

**🎯 影响范围**  
- `vllm/v1/worker/gpu/mm/lora.py`（核心实现）  
- `vllm/v1/worker/gpu/model_runner.py`（调用点）  
- `tests/v1/worker/test_gpu_model_runner.py`（新增测试）  
- 相关 LoRA、Multi‑Modal、EncoderCache 类型的导入路径  

**💡 关注建议**  

1. **异常安全**：`set_active_mm_loras` 直接通过 `encoder_cache.mm_features[req_id]` 访问，若 `req_id` 在 cache 中不存在会抛 `KeyError`。建议在遍历前加 `if req_id not in encoder_cache.mm_features: continue`，与已有的 `req_idx is None` 检查保持一致。  
2. **类型契约**：`model.get_mm_mapping()` 可能返回 `None`，已做判断，但 `model.get_num_mm_encoder_tokens` 与 `model.get_num_mm_connector_tokens` 的返回值必须是 `int`。若未来模型实现改为返回 `torch.Tensor`，会导致 `np.repeat` 报错。可在调用前显式 `int()` 转换。  
3. **性能与内存**：`np.repeat` 在大批量视觉 token 场景下会产生临时数组。若预期单次调度的 token 数量可达数万，考虑改为分块拼接或使用 `torch.repeat_interleave`（避免 Python → NumPy 切换）。  
4. **测试覆盖**：当前测试覆盖了两条请求（有 LoRA / 无 LoRA）以及缺失请求的跳过。建议再加入：  
   - `encoder_cache` 为 `None` 的路径，确保函数直接返回；  
   - `lora_manager.supports_tower_connector_lora()` 为 `False` 的分支；  
   - `model.get_mm_mapping().connector = False` 的情形。  
5. **文档**：在 `vllm/v1/worker/gpu/mm/lora.py` 顶部补充简要使用说明，说明何时应在 `execute_model` 前调用以及参数意义，便于后续维护。  

总体来看，改动解决了 MRV2 中多模态 LoRA 未正确激活的问题，逻辑清晰并通过单元测试验证。只需轻微加强异常处理与覆盖边缘分支，即可在生产环境安全使用。
