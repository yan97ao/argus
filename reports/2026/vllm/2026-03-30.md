### [ROCm][Bugfix] fix exception related to trust_remote_code for MiniMax-M2.1-MXFP4 (#37698)
**SHA**: `dbdd9ae` | 🔗 [查看提交](https://github.com/vllm-project/vllm/commit/dbdd9ae06738bb8dc5b3fe6c99b3dd5bcd5d5526)

**🎯 变更类型**：Bugfix（同时伴随接口统一）  
**⚡ 重要程度**：🟡 中  
**📋 变更摘要**：  
1. 为所有量化配置的基类 `QuantizationConfig.maybe_update_config` 增加 `hf_config` 与 `revision` 参数，并在子类实现中相应签名统一。  
2. `QuarkConfig.maybe_update_config` 由原来自行调用 `get_config` 改为接受外部 `hf_config`，仅在模型 **model_type** 属于 DeepSeek‑V3 系列且 `quantization_config` 中的 `weight.dtype` 为 **fp4** 时打开 `dynamic_mxfp4_quant`。  
3. `vllm/config/vllm.py` 在创建量化配置时传入 `model_config.hf_config`，确保上层已经获取到的 HuggingFace 配置能够直接复用。  
4. 为其它量化实现（AWQ、GPTQ、CPU‑WNA16 等）补齐同样的签名，保持接口一致。  
5. 新增单元测试 `tests/quantization/test_quark_maybe_update_config.py` 验证：非 DeepSeek‑V3 模型不启用 flag、DeepSeek‑V3‑fp4 启用、缺失 hf_config 时保持默认。

**🎯 影响范围**：  
- `vllm/model_executor/layers/quantization/*`（所有量化子类）  
- `vllm/config/vllm.py`（量化配置创建路径）  
- 新增测试文件，对 CI 有直接影响。  

**💡 关注建议**：  
1. **兼容性检查**：签名变更后，除 `vllm.py` 外的所有调用点必须显式传入 `hf_config`（或保持默认 `None`），否则会回退到原实现逻辑。建议在项目其余内部调用（如自定义模型加载脚本）做一次搜索确认。  
2. **异常处理**：`QuarkConfig.maybe_update_config` 现在依赖外部 `hf_config`；若调用方忘记传递且 `trust_remote_code=False`，会导致 `AttributeError`（`NoneType` 没有 `model_type`），建议在函数入口添加 `if hf_config is None: return` 或记录警告，以免因网络不可达导致构建失败。  
3. **文档同步**：更新 `vllm/model_executor/layers/quantization/base_config.py` 中的接口说明，明确 `revision` 暂未使用但保留向后兼容。  
4. **测试可靠性**：新测试依赖远程 HuggingFace 服务器和 `trust_remote_code=True`，在离线 CI 环境可能失效。可以考虑在 CI 中加入网络重试或使用 `skipif` 机制。  
5. **性能影响**：仅在 DeepSeek‑V3‑fp4 场景下打开 `dynamic_mxfp4_quant`，不改动其它模型的执行路径，预计对整体吞吐无负面影响。

总体来看，此次改动定位明确、实现简洁，解决了 MiniMax‑M2.1‑MXFP4 在 `trust_remote_code=False` 环境下抛异常的问题，同时为未来量化配置的自动化适配奠定了统一接口。建议完成上述兼容性与测试细节的检查后合并。

### [Core][CI] Add opt-in media URL caching via VLLM_MEDIA_CACHE (#37123)
**SHA**: `677424c` | 🔗 [查看提交](https://github.com/vllm-project/vllm/commit/677424c7acd9fb7477294017c99f798588002d4f)

**变更类型**：功能增强（可选媒体文件本地缓存）  
**重要程度**：🟡 中  

### 变更摘要  
1. 在 `vllm.envs` 中新增三项 env‑var：`VLLM_MEDIA_CACHE`（缓存根目录），`VLLM_MEDIA_CACHE_MAX_SIZE_MB`（容量上限，默认 5 GB）和 `VLLM_MEDIA_CACHE_TTL_HOURS`（TTL，默认 24 h）。  
2. `MediaConnector` 新增缓存相关成员并在构造函数里依据 env‑var 启用缓存，加入写入日志的初始化。  
3. 实现 `_media_cache_path`（URL→哈希文件名）、`_get_cached_bytes`、`_put_cached_bytes`、`_maybe_evict`（TTL 清理 + LRU 限制）以及相应的同步/异步读取路径。  
4. 读取 HTTP URL 前先尝试命中缓存，命中则直接返回；未命中则下载后写入缓存。  
5. 新增一组单元测试覆盖缓存写入、TTL 失效、LRU 驱逐、目录失效、并发删除等边界情况。  

### 影响范围  
- **核心模块**：`vllm/multimodal/media/connector.py`（缓存实现、下载路径改动）  
- **配置模块**：`vllm/envs.py`（新增 env 参数、编译列表）  
- **测试套件**：`tests/multimodal/media/test_connector.py`（新增 112 行）  

### 关注建议  

| 角色 | 建议 |
|------|------|
| **开发者** | 1. 目前缓存的 LRU/TTL 判断在单进程内完成，跨进程（多节点）下可能出现竞争或冗余文件，建议后续考虑文件锁或统一的缓存守护进程。<br>2. `_maybe_evict` 在遍历目录时未加锁，若同一目录被多个进程并发写入，可能导致误删或缓存膨胀。<br>3. 读取时 `cache_path.touch()` 用于更新 mtime，若文件系统不支持高精度时间或在只读挂载下会抛异常，建议捕获并记录。<br>4. 为避免异常掩埋，建议在 `load_from_url`/`load_from_url_async` 中对 `_put_cached_bytes` 的返回值或异常做更细粒度的日志。 |
| **用户** | 1. 若希望缓存，请在启动脚本中设置 `VLLM_MEDIA_CACHE=/path/to/cache` 并保证该目录可写且有足够磁盘空间。<br>2. 如不想使用缓存，保持默认空字符串或显式 `VLLM_MEDIA_CACHE=` 即可，系统会透明回退到原始网络请求。<br>3. 对于大模型推理的高并发场景，监控磁盘使用量与 TTL 设置，以防磁盘被缓存填满导致 I/O 错误。 |
| **测试维护** | 现有缓存相关单元测试依赖 `TemporaryDirectory`，确保 CI 环境的 `/tmp` 有写权限；如迁移至只读容器，需要添加 `VLLM_MEDIA_CACHE=` 环境变量以禁用缓存。 |

总体而言，此次改动为 VLLM 引入了可选的本地媒体缓存，能显著降低重复下载开销，但在多进程/多节点部署下仍需注意并发安全和磁盘容量管理。后续可考虑加入锁机制、Prometheus 指标以及更细致的错误上报。

### Fix ambiguous num_blocks for hybrid attn mamba (#37236)
**SHA**: `1031c84` | 🔗 [查看提交](https://github.com/vllm-project/vllm/commit/1031c84c360874ddc37589f21b15686788ac142e)

**变更类型**：Bug 修复  
**重要程度**：🟡 中  
**变更摘要**：在混合注意力/​Mamba 场景下，KV 缓存的维度 `(2, num_blocks, …)` 与 `(num_blocks, 2, …)` 在 `num_blocks == 2` 时无法区分，导致布局错误。新增对 `AttentionSpec` 的显式检查，并通过后端 `get_kv_cache_block_dim` 判定当前布局，仅在需要时对张量重新 stride。对应单元测试也加入了 “num_blocks=2” 的歧义情形验证。  

**影响范围**  
- `vllm/v1/worker/gpu_model_runner.py`（核心布局逻辑）  
- `tests/v1/worker/test_gpu_model_runner.py`（新增测试）  

**关注建议**  
1. **后端实现**：确保所有注意力后端（FlashAttention、其他）实现 `get_kv_cache_block_dim`，并在 `kernel_block_sizes` 与 `kv_cache_spec` 对应时返回正确的维度标记。  
2. **向后兼容**：在旧版本后端未实现该方法时提供安全回退（如默认按现有维度处理），避免运行时 `AttributeError`。  
3. **性能验证**：虽然 `as_strided_` 只改动 stride，不涉及复制，但仍建议在大模型下跑一次完整的推理基准，确认没有意外的内存布局回退。  
4. **测试覆盖**：当前仅在 `num_blocks=2` 场景新增测试，建议再加入 `num_blocks>2` 与 `num_blocks=1` 的路径，以防止误判。  
5. **文档更新**：在 KV‑Cache 章节说明 “混合注意力/​Mamba 布局的自动纠正” 以及 `kernel_block_sizes` 与 `AttentionGroup` 的关系，帮助使用者理解新参数的意义。  

总体来看，此次改动修复了一个细粒度但潜在致命的布局错误，影响范围限定在混合注意力实现，不会对其它模型路径产生副作用。确保后端实现同步更新即可安全合并。

整体来看，此次改动提升了对损坏图像的容错能力，避免了内部未捕获的 `500` 错误，对前端用户体验有正面影响。请在 CI 中跑完整的多模态测试以确认所有路径均能正确映射到 400 返回。
