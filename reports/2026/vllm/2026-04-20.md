### nixl refactor [2/N]: unify TpKVTopology + HeteroTPTransferConfig into TransferTopology (#39529)
**SHA**: `cc3993b` | 🔗 [查看提交](https://github.com/vllm-project/vllm/commit/cc3993b05d008be370778466865cce1959ac85a3)

**🎯 变更类型**：重构 / 架构变更  

**⚡ 重要程度**：🔴高  

**📋 变更摘要**  
本次提交将原有的 `TpKVTopology`（负责本地/远端 TP 拓扑信息）与 `HeteroTPTransferConfig`（负责异构 TP 场景下的 FlashAttention / Mamba 传输计划）合并为统一的 **`TransferTopology`**。  
- 引入 `EngineTransferInfo` 与 `MambaEngineTransferInfo`，统一管理每个远端 engine 的 TP 大小、块大小、物理块倍率等元数据。  
- 所有使用旧类的测试和核心代码均改为使用 `TransferTopology`，包括 TP‑ratio、块大小比、KV‑replication 判定、目标远端 rank 计算、Mamba‑HMA（Hybrid‑Mamba‑Attention）专属逻辑等。  
- 通过 `register_remote_engine` 在握手阶段一次性完成远端信息的注册，消除原来依赖多个共享字典（`_tp_size`, `_block_size`）的散乱状态。  

**🎯 影响范围**  
- `vllm/distributed/kv_transfer/kv_connector/utils.py`（核心拓扑类实现）  
- `vllm/distributed/kv_transfer/kv_connector/v1/*`（Nixl、Mooncake、tests 等所有 KV‑connector 实现）  
- 相关单元测试文件（`test_mooncake_connector.py`, `test_nixl_connector.py`, `test_nixl_connector_hma.py`）  
- `vllm/distributed/kv_transfer/kv_connector/v1/ssm_conv_transfer_utils.py`（Mamba 物理块倍率函数改名）  

---

### 🔍 技术洞察

| 维度 | 影响说明 |
|------|----------|
| **架构影响** | 1. **单一真相来源**：`TransferTopology` 统一管理本地 TP 信息、远端 engine 元数据以及 Mamba‑HMA 传输计划，避免了 `TpKVTopology` 与 `HeteroTPTransferConfig` 两套并行维护的风险。<br>2. **更清晰的层次**：`EngineTransferInfo` 与 `MambaEngineTransferInfo` 用 `@dataclass(frozen=True)` 表示不可变，提升代码可读性并帮助 IDE/静态检查发现误用。<br>3. **注册流程集中**：通过 `register_remote_engine` 完成一次性注册，原先在 `NixlWorker.add_remote_agent` 中散落的 `_tp_size`/`_block_size` 更新被废弃，降低了状态同步错误概率。 |
| **性能影响** | 1. **运行时开销**：新增了若干 `@property` 计算（如 `tp_ratio`、`block_size_ratio`）以及在每次握手时创建 `EngineTransferInfo` 对象，所需的 CPU/内存开销非常低（仅几百字节对象），对整体吞吐几乎无影响。<br>2. **缓存局部性**：原来分散的字典查询被合并为一次对象属性访问，缓存命中率提升，可略微加速热点路径（如 `handshake_target_ranks`、`target_remote_ranks`）。<br>3. **Mamba‑HMA 计算**：`_build_mamba_info` 在握手阶段一次性完成所有派生数据（FA source ranks、所有 source ranks、FA descriptor 大小等），消除了后续分散的 `if‑else` 逻辑，能够在多 TP 场景下保持 O(1) 的访问。 |
| **安全考虑** | 本次改动不涉及网络协议、序列化格式或权限检查，只是内部数据结构的重构。<br>唯一需关注的是 **序列化兼容性**：如果外部系统（如监控或日志）依赖旧的 `TpKVTopology` 对象序列化，则在升级后需同步更新或保持向后兼容的桥接层。 |
| **可维护性** | 1. **命名统一**：`TransferTopology` 直观表达“传输拓扑”，摆脱了 “TpKV” 与 “HeteroTP” 的混用。<br>2. **代码复用**：原本分散在 `TpKVTopology` 与 `HeteroTPTransferConfig` 的方法（如 `tp_ratio`, `target_remote_ranks`, `fa_head_slot`）均迁移至 `TransferTopology`，避免重复实现和潜在不一致。<br>3. **测试覆盖**：所有相关测试已全面迁移，保持原有覆盖率，帮助捕获迁移过程的功能回归。 |
| **潜在风险** | - **迁移遗漏**：若还有未被搜索到的业务代码直接访问 `worker.kv_topo`、`worker._tp_size`、`worker._block_size`，在运行时会触发 `AttributeError`。<br>- **Mamba‑HMA 逻辑差异**：`_build_mamba_info` 中的 `fa_source_ranks` 与旧 `HeteroTPTransferConfig.fa_read_targets` 的生成细节需保持一致，任何细微差别都会导致 remote‑FA descriptor 偏移错误，进而出现 KV‑read 数据错位或崩溃。<br>- **序列化/持久化**：部分场景（如 checkpoint、进程恢复）可能会序列化 `kv_topo`；新实现未提供 `__getstate__/__setstate__`，需确认不被外部依赖。 |
| **兼容性** | - 对外 API 已全部改为 `TransferTopology`，**向后不兼容**（旧类已删除）。升级 vLLM 需要同步更新自定义插件或第三方代码。<br>- 通过 `compute_physical_blocks_per_logical` 替代原 `compute_mamba_phys_ratio`，函数签名保持一致，仅名称变化，影响最小。 |

### ⚠️ 潜在风险

1. **遗漏旧属性**  
   - `worker.kv_topo`、`worker._tp_size`、`worker._block_size` 在生产环境中可能被外部插件或监控脚本引用。若未同步改写，会导致运行时异常。  
2. **Mamba‑HMA 计算差异**  
   - `fa_head_slot`、`fa_rank_offset` 等方法的实现细节改动（从 `range` 查找改为使用 `fa_source_indices`）在极端 GQA、replicated KV 场景下需要严格回归验证。  
3. **序列化/持久化**  
   - 如果进程在异常恢复时依赖 `pickle` 保存的 `kv_topo`，新类未实现旧的 `__dict__` 结构，可能出现 `pickle.UnpicklingError`。  
4. **线程安全**  
   - `register_remote_engine` 目前在 `TransferTopology` 上没有显式锁，若多个线程/协程并发调用（如在多进程 `NixlWorker` 初始化阶段），有极小概率出现 race 条件。  
5. **日志/监控兼容**  
   - 旧日志格式中会出现 `TpKVTopology` 的字段名，如 `remote_tp_size`、`remote_block_size`，升级后日志解析脚本需要适配新的 `TransferTopology.describe()` 输出。

### [EPLB] Add nixl-based eplb communicator (#36276)
**SHA**: `50dd4cb` | 🔗 [查看提交](https://github.com/vllm-project/vllm/commit/50dd4cb42726777635acda9ed4f5440ae4a2e281)

**🎯 变更类型**：功能增强（为 EPL （Expert Parallel Load‑Balancing）新增 NIXL/RIXL 后端）  
**⚡ 重要程度**：🔴 高  

**📋 变更摘要**  
- 在 EP Load‑Balancing（EPLB）实现中加入 **NIXL/RIXL** 作为新的通信后端，支持基于 *READ* 的零拷贝专家权重传输。  
- 扩展配置、文档、单元测试及工厂函数，使其能够检测、创建并在运行时选择该后端；同时提供 `has_nixl()` 检查函数。  
- 为 NIXL 相关的环境变量、懒加载及共享工具抽离到 `vllm/distributed/nixl_utils.py`，并在其它模块统一引用。  
- 调整 `rearrange_expert_weights_inplace` 的 profiling 路径，使不需要额外的 NCCL 缓冲区预留，从而避免在使用 NIXL 时的额外内存峰值。  

**🎯 影响范围**  
- **核心模块**：`vllm/distributed/eplb/*`（communicator、factory、配置）  
- **KV‑Transfer 子系统**：NIXL 统计与 utils  
- **模型加载/运行时**：`vllm/v1/worker/gpu_model_runner.py`（EPLB 启动与异步循环）  
- **文档**：新增 `communicator` 配置字段说明  
- **测试**：`tests/distributed/test_eplb_execute.py` 增加 NIXL 参数化

### 🔍 技术洞察  

| 维度 | 影响描述 |
|------|----------|
| **架构影响** | - 引入 **`NixlEplbCommunicator`**，实现了基于 NIXL/ RIXL 的 “receiver‑initiated READ” 传输模型。<br>- 新增 **`nixl_utils.py`** 进行懒加载、环境变量预置（`UCX_RCACHE_MAX_UNRELEASED`）以及统一的可用性检测 (`has_nixl`)。<br>- `EPLBCommunicatorBackend` 类型从 3 项扩展至 4 项，后端选择逻辑在 `create_eplb_communicator` 中统一。<br>- 通过 `needs_profile_buffer_reservation` 抽象，使不同后端在 profiling 阶段拥有不同的缓冲区预留策略。 |
| **性能影响** | - **正向**：NIXL 使用 **GPU‑direct RDMA**（或 RO‑GPU on ROCm）实现零拷贝读取，理论上比 NCCL/Gloo 的复制路径降低延迟与带宽占用，尤其在大模型、跨节点的 Expert Weight 迁移场景。<br>- **负向**：`_init_registered_buffers` 为每个 rank 预分配 `world_size * partition_bytes` 的发送缓冲区，若 Expert Weight 体积大，可能导致显存 OOM（代码已有 `TODO` 注释说明）。<br>- Profiling 时不再强制执行 dummy all‑gather，避免额外 **NCCL** 缓冲区分配，减小 profiling 期间的显存峰值。 |
| **安全考虑** | - 仅涉及 **可选的外部依赖**（`nixl` / `rixl`）和环境变量，无直接安全风险。<br>- 通过 `has_nixl()` 检查并在未安装时抛出明确异常，防止因隐藏的 ImportError 产生不可预期的运行时错误。 |
| **可维护性** | - NIXL 相关代码被抽离到独立文件 `nixl_utils.py`，降低对主体代码的侵入性，便于后续独立升级或替换实现。<br>- 通过 `@property needs_profile_buffer_reservation` 统一接口，后端实现只需覆盖一次即可。<br>- 新增大量注释和错误信息，提升可读性。 |
| **兼容性** | - 仅在 **CUDA‑like** 设备上可用（`current_platform.is_cuda_alike()`），CPU、ROCm（RIXL）路径已做区分。<br>- 对 **elastic / stateless EP** 组的限制保持不变，仅支持 `torch_nccl` / `pynccl`。<br>- 当 NIXL 不可用时会回退到原有后端或抛异常，保持向后兼容。 |
| **测试覆盖** | - 扩展 `test_eplb_execute.py` 参数化列表加入 `"nixl"`，并在缺失 NIXL 时自动跳过，确保 CI 不因缺少依赖失败。 |

### ⚠️ 潜在风险  

1. **显存 OOM**  
   - 发送缓冲区大小为 `world_size * partition_bytes`（每种 dtype 的最大需要量），在大模型或高 `world_size` 场景可能超过 GPU 容量。  
2. **依赖可用性**  
   - NIXL / RIXL 是 **可选的闭源/商业** 包，若用户未安装或授权失效，会导致 `RuntimeError`。需要在部署文档中明确说明安装方式及兼容的硬件/驱动要求。  
3. **平台差异**  
   - 对 ROCm（使用 RIXL）和 CUDA 的代码路径略有不同；若平台检测错误（`current_platform.is_cuda_alike()`）可能误报或误选后端。  
4. **调试复杂度**  
   - NIXL 的错误信息主要来自其 C‑API（如 `check_xfer_state`），异常会被包装为 Python `RuntimeError`，调试时需要额外查阅底层日志。  
5. **Profiling 逻辑变化**  
   - `needs_profile_buffer_reservation` 影响 profiling 的 buffer 预留；若其他自定义后端忘记覆盖该属性，可能出现意外的显存峰值或 profiling 失效。

### [CPU][RISC-V] Support multiple RVV VLEN targets via compile-time dispatch (#39478)
**SHA**: `ec7aafc` | 🔗 [查看提交](https://github.com/vllm-project/vllm/commit/ec7aafc02ae86c41edf2f245df793a37bd097fc4)

**🎯 变更类型**：功能增强 / 重构  

**⚡ 重要程度**：🟡 中  

**📋 变更摘要**  
1. **CMake 增加 `VLLM_RVV_VLEN` 选项**：在检测到 RISC‑V RVV 时，尝试从 `/proc/cpuinfo` 自动解析最大 `zvl<N>b`，若检测失败则强制用户通过 `-DVLLM_RVV_VLEN=128|256|0` 指定 VLEN。  
2. **CPU 类型实现彻底重构**：原来的 `cpu_types_riscv.hpp` 被拆分为  
   - `cpu_types_riscv_defs.hpp`（VLEN→LMUL、mask、accumulator 的宏映射）  
   - `cpu_types_riscv_impl.hpp`（所有向量包装类的实现）  
   - `cpu_types_riscv.hpp` 仅保留宏判断与头文件引入。  
   通过宏 `RVVTYPE / RVVI / RVVI3 / RVVIB` 将 **编译时 VLEN** 透明映射到对应的 RVV 类型，实现一次源码即可在 VLEN=128、256、512、1024 四种硬件上编译。

**🎯 影响范围**  
- **编译系统**：`cmake/cpu_extension.cmake` 中的 `MARCH_FLAGS` 生成逻辑。  
- **向量库**：`csrc/cpu/cpu_types_riscv.hpp/defs.hpp/impl.hpp`，所有使用 `vec_op::*` 的 CPU kernel（例如自注意力、layer‑norm、gelu 等）现在会根据 `VLLM_RVV_VLEN` 编译不同的实例。  
- **宏/编译选项**：新增 `RISCV_BF16_SUPPORT`、`VLLM_RVV_VLEN`，并在 `cpu_types_riscv.cpp`（若有）中加入 `#if __riscv_v_min_vlen` 检查。  

**💡 关注建议**  

| 关注点 | 建议 |
|--------|------|
| **自动检测可靠性** | `/proc/cpuinfo` 只在 Linux 且拥有权限时可用。建议在文档里明确说明在交叉编译或非 Linux 环境必须手动 `-DVLLM_RVV_VLEN=…`，并在 CMake 中加入 `CACHE STRING` 以便用户查询当前值。 |
| **编译器兼容性** | `__riscv_v_min_vlen` 需要编译器在 `-march=…zvl<N>b` 下定义。对旧版 GCC/Clang 可能没有该宏，导致 `#error`。可以在 CMake 中先检测宏是否存在（`check_cxx_source_compiles`），若不存在给出友好的错误提示。 |
| **宏膨胀与可读性** | `RVVTYPE / RVVI / RVVI3 / RVVIB` 的拼接层数较多，调试时容易出现 “cannot find symbol” 的错误。建议在 `cpu_types_riscv_defs.hpp` 中添加 `#pragma message` 或 `static_assert`，在不匹配的 VLEN 时给出清晰报错。 |
| **二进制体积** | 每个 VLEN 产生一套模板实例，可能导致对象文件膨胀。若目标平台只支持一种 VLEN，使用 `-DVLLM_RVV_VLEN=0`（强制标量）或在 CMake 中 `set(CMAKE_CXX_FLAGS "${CMAKE_CXX_FLAGS} -fno-rtti")` 来控制实例化。 |
| **BF16/F16 切换** | `RISCV_BF16_SUPPORT` 与 `RVV_FP16_FOUND` 的组合仍保留旧的 “BF16 不可用 -> warning”。建议在 `cpu_types_riscv_defs.hpp` 中加入 `#define VLLM_HAS_BF16 (defined(RISCV_BF16_SUPPORT) && VLLM_RVV_VLEN>0)`，统一判断。 |
| **测试覆盖** | 由于向量宽度变化会影响 `reduce_sum`、`exp`、`tanh` 等实现，务必在 CI 中加入至少两种 VLEN（128 与 256）交叉编译运行的单元测试，确保数值误差满足既有阈值。 |
| **文档与示例** | 在 README 或 `docs/riscv.md` 中提供 **完整编译示例**：`cmake -DVLLM_RVV_VLEN=256 -DRISCV_BF16_SUPPORT=ON ..`，并说明在 Spacemit X100（VLEN=256）应使用该参数。 |
| **向后兼容** | 其他平台（x86、ARM）仍继续使用原来的 `cpu_types_xxx.hpp`，确保 `#include "cpu_types_riscv.hpp"` 不会在非‑RVV 环境中产生语法错误（已有 `#error "__riscv_vector"` 检查）。 |

**结论**  
本次改动实现了 **多 VLEN 编译时调度**，极大提升了 vLLM 在不同 RISC‑V 硬件上的可移植性。关键在于 **编译选项的明确化** 与 **宏映射的可靠性**，建议在 CI 中加入各 VLEN 的交叉编译测试，并在文档里给出手动指定 VLEN 的使用指南，以防出现自动检测失效或旧版编译器不支持的情况。整体风险可控，代码结构更清晰，后续如需新增更大 VLEN（如 2048）只需在 `defs.hpp` 增加对应 `LMUL` 映射即可。
