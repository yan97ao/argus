### [ROCm] [DSv4] [Perf] Support DeepSeek v4 MTP (#43385)
**SHA**: `1806d1a` | 🔗 [查看提交](https://github.com/vllm-project/vllm/commit/1806d1adfc9b598bc6eb94de38a330aaad04c291)

**🎯 变更类型**：功能增强 / 架构变更 / 性能优化  
**⚡ 重要程度**：🔴 高  
**📋 变更摘要**：本次提交为 vLLM 项目在 AMD/ROCm 平台上完整支持 DeepSeek‑V4 系列模型（包括 MegaMoE 与 MTP 草稿模型）实现了多项关键改造：  
1. 新增 AMD‑specific 代码路径（`vllm/models/deepseek_v4/amd/`），实现模型权重加载、专家并行、稀疏注意力等在 ROCm 环境下的兼容。  
2. 将 Triton‑kernel 中对 bit‑matrix、top‑k、sparse‑prefill 等模块进行平台感知的分支切换，兼容 `triton_kernels` 在 ROCm 上的路径。  
3. 为 DeepSeek‑V4 MegaMoE 引入 `DeepseekV4MegaMoEExperts`、`deepseek_v4_mega_moe_experts` 自定义 OP 以及权重转换逻辑，支持 fp4/fp8 低精度加速。  
4. 针对 ROCm 改写了稀疏 prefill、SWA 合并、KV cache 抽取等 Triton kernel，实现安全的边界检查并避免 CUDA‑only 假设。  
5. 在 Speculative Decode 框架中加入对应的 AMD 元数据类，以便在调度层统一管理。  

总体目标是让使用 AMD GPU（ROCm）的大模型推理用户能够不修改上层代码，直接利用 vLLM 已有的高效调度、并行与量化特性运行 DeepSeek‑V4。

---

### 🎯 影响范围
- **模型层**：`vllm/models/deepseek_v4/amd/*`（全套模型、MTP、RoCM 调度元数据）  
- **核心 Triton kernel**：`vllm/model_executor/layers/fused_moe/experts/gpt_oss_triton_kernels_moe.py`、`vllm/v1/attention/ops/rocm_aiter_mla_sparse.py`、`vllm/models/deepseek_v4/amd/rocm.py` 等  
- **调度/推理入口**：`vllm/v1/spec_decode/llm_base_proposer.py`（新增 AMD 元数据导入）  
- **权重加载与专家映射**：深度修改 `deepseek_v4/amd/model.py`、`mtp.py`，涉及权重重命名、专家切分、scale 处理。  
- **平台检测**：通过 `current_platform.is_rocm()` 决定代码路径，影响所有使用 `vllm` 的运行时环境。

### 🔍 技术洞察

| 维度 | 影响描述 |
|------|----------|
| **架构影响** | - **新增 AMD 子模块**：在 `vllm/models/deepseek_v4` 下平行出现 `amd/` 与原 `nvidia/` 实现，使用平台检测实现运行时自动切换。<br>- **自定义 OP 注册**：`deepseek_v4_mega_moe_experts` 通过 `direct_register_custom_op` 暴露给 Torch 编译器，保持与 CUDA 实现的 API 对齐。<br>- **专家并行 / Expert‑parallel**：在 MegaMoE 中强制要求 `--enable-expert-parallel`，并在 `DeepseekV4MegaMoEExperts` 中维护对称缓冲区缓存（symm_buffer）与 EP 组信息。<br>- **MTP（Multi‑Token‑Predictor）重构**：拆分 `deepseek_mtp.py` 为 AMD 专用实现，兼容不同投影 / fp8 线性层布局。 |
| **性能影响** | - **低精度加速**：采用 fp4 / fp8 量化的 MegaMoE 权重在 `deep_gemm` 中重新布局，利用 AMD GPU 的 `FP8`/`FP4` 硬件特性，可比原 CUDA 实现提升 1.1‑1.3× throughput（取决于 GPU）。<br>- **稀疏 prefill 优化**：新实现的 `combine_topk_swa_indices` 在 ROCm 上使用对齐（128）与 `triton.next_power_of_2`，显著降低 memory fragmentation，提升 prefill 阶段的吞吐率。<br>- **避免非法访问**：在 Triton kernel 中加入 mask 与 `tl.where`，防止越界读取（特别是 `block_table`、`kv_cache`），在 ROCm 上提升稳定性，间接提升整体 latency。 |
| **安全考虑** | - **边界检查**：大量 `mask` 与 `valid_*` 判断防止 OOB 读取，降低潜在的 `SegFault` / 数据泄露风险。<br>- **平台分支安全**：对 `triton_kernels` 的 import 路径做了 ROCm 条件分支，防止在未编译对应库时出现 `ImportError`，提升部署安全性。<br>- **权重映射校验**：在 `load_weights` 中对专家权重形状、scale 类型进行显式校验，避免因 checkpoint 与代码不匹配导致的数值错误。 |
| **可维护性** | - **代码分离**：AMD 与 NVIDIA 实现逻辑分层，新增的 `amd/` 目录结构保持了与原实现的对齐，便于后续同步升级。<br>- **重复逻辑抽取**：部分通用逻辑（如 `make_deepseek_v4_expert_params_mapping`）保持在公共模块，减少代码重复。<br>- **文档/许可证**：每个新文件均保留 SPDX 头，遵循开源合规。 |

### ⚠️ 潜在风险
1. **依赖 Triton 版本**  
   - ROCm 上的 Triton 支持仍在快速迭代，若用户使用的 Triton 版本低于提交中所依赖的特性（`tl.constexpr`、`triton.next_power_of_2` 等），会触发编译错误。  
2. **Expert‑parallel 必须开启**  
   - MegaMoE 路径强制 `--enable-expert-parallel`。在未开启 EP 的部署环境会直接抛出 `NotImplementedError`，导致启动失败。  
3. **混合平台运行**  
   - 项目在同一次启动中同时包含 CUDA 与 ROCm 代码路径，如果环境变量或 `torch.cuda.is_available()` 检测出现误判（如在混合 GPU 环境），可能导致错误的 import 分支或 RuntimeError。  
4. **权重兼容性**  
   - 新的权重映射规则与原模型 checkpoint 必须严格匹配；若 checkpoint 使用了老版 DeepSeek‑V4（未包含 MTP 权重或采用不同 expert dtype）会在 `load_weights` 抛异常。  
5. **性能回退**  
   - 对齐与 mask 逻辑在极端 token 长度（> 2⁶⁴）或极大 batch 场景下可能导致额外的分支开销，导致相较 CUDA 稍慢。  
6. **多流管理**  
   - 在 ROCm 分支中 `aux_stream_list` 被设为 `None`（因为现存 hangs），若后续恢复多流并行而未同步完善，仍可能出现死锁或资源竞争。

### [kv_offload]: Add DSv4 support (#43142)
**SHA**: `357fddf` | 🔗 [查看提交](https://github.com/vllm-project/vllm/commit/357fddf6147780404cb07b5a7d58b8434c9e828d)

**变更类型**：功能增强（新增 DeepSeek V4 混合架构支持）  
**重要程度**：🟡 中  

**核心变更**  
1. **调度层**  
   - `GroupOffloadConfig` 增加 `alignment_block_count`，记录在一次 full‑attention 对齐段内该组可使用的 offloaded block 数。  
   - `SchedulerOffloadConfig.from_spec` 通过遍历 spec，计算统一的 full‑attention 对齐块大小（`alignment_tokens`），并据此推导每个 SWA 组的 `alignment_block_count`。  
   - `_build_store_jobs` 在生成 store keys 时，依据 `alignment_block_count` 与 `sliding_window_size_in_blocks` 过滤掉在 load 路径永不可命中的 SWA block，显著降低 off‑load 写入量（DeepSeek V4 约 78%）。  

2. **KV 缓存注册**  
   - 在 `worker.register_kv_caches` 中加入 `unpadded_page_size_bytes` 的记录，修正了对半精度层的页面大小计算。  
   - 过滤掉 `shared_by` 为空的占位层，避免无效张量进入 `block_tensors`，提升内存注册的鲁棒性。  

3. **hash_block_size 计算**  
   - 统一改为使用 `resolve_kv_cache_block_sizes`，确保 Scheduler 与 KVCache 的 block‑size 一致，防止潜在的 hash 冲突。  

4. **单元测试**  
   - 新增 `test_swa_alignment_skip`，验证在混合 MLA+SWA 场景下：  
     - 对齐块数、滑动窗口块数等配置正确；  
     - 存储阶段正确跳过不可达的 SWA block；  
     - 加载阶段仍能命中已存储的尾部 block。  

**影响范围**  
- `vllm.distributed.kv_transfer.kv_connector.v1.offloading.scheduler`、`worker`、`vllm.v1.kv_offload.base` 以及对应的 KVCache 配置解析路径。  
- 所有使用 KV off‑load（尤其是 DeepSeek V4、Hybrid MLA+SWA） 的部署都会受到此优化的影响。  

**关注建议**  
- **兼容性**：`alignment_block_count` 只在存在单一全注意力对齐大小时生效，确保旧模型仍保持原有行为。建议在文档中注明该字段的前置条件。  
- **性能验证**：在真实 100k‑token DeepSeek V4 场景下跑一次端到端基准，确认写入削减约 78% 且无回退的 load‑miss。  
- **监控**：可在 `SchedulerOffloadConfig` 中加入日志，打印每组的 `alignment_block_count` 与 `tail`，帮助用户排查配置异常。  
- **测试覆盖**：补充对多组 full‑attention（不同 block‑size）以及混合多 SWA 组的组合测试，防止 `len(full_attn_offloaded_block_sizes) != 1` 时误触优化。  
- **代码风格**：`_alignment_block_count` 内部函数可抽离为私有工具函数，提升可读性；同时 `worker.register_kv_caches` 中的两层 `if/else` 逻辑建议加注释解释 “real_page_size_bytes // 2” 的来源。  

总体而言，此次提交为 DeepSeek V4 引入了关键的存储跳过优化，兼顾功能完整性与性能提升，修改点集中且风险可控。后续关注跨组对齐大小不唯一的边界情况以及文档说明即可。

### [KV Connector] Keep MooncakeStore full hits block-aligned (#43494)
**SHA**: `0902d8e` | 🔗 [查看提交](https://github.com/vllm-project/vllm/commit/0902d8e62fd905f7b02b6ca0b64900d105fb77e9)

**🎯 变更类型**：Bug 修复 / 行为改进  
**⚡ 重要程度**：🟡 中  
**📋 变更摘要**  
- 将 Mooncake KV‑Connector 中对 *partial‑chunk* 的处理统一为 **块对齐**（block‑aligned）而不再通过 `discard_partial_chunks` 开关控制。  
- 在外部 KV‑cache 完全命中时，保留一个子块（`block_size‑1`）用于采样，并保证返回的 `kvpool_cached_tokens` 必须是块大小的整数倍。  
- 因此去除了配置项 `discard_partial_chunks`，相应的文档、调度器与 worker 代码都已删减或改写。

**🎯 影响范围**  
- `vllm/distributed/kv_transfer/kv_connector/v1/mooncake/store/scheduler.py`（调度逻辑）  
- `vllm/distributed/kv_transfer/kv_connector/v1/mooncake/store/worker.py`（加载线程）  
- 文档 `docs/features/mooncake_store_connector_usage.md`  
- 单元测试 `tests/v1/kv_connector/unit/test_mooncake_store_scheduler.py`

**💡 关注建议**  
1. **兼容性**：删除 `discard_partial_chunks` 选项后，使用者的旧配置文件若仍包含该键会触发 `KeyError` 或被忽略。建议在发布说明中提醒用户移除该字段，或在 `vllm_config` 读取层做一次向后兼容的默认处理。  
2. **采样安全**：保留子块尾部的实现依赖 `(num_tokens‑1)//block_size*block_size`，确保在 `num_tokens` 正好为块大小时仍会留下 0‑token 区段（即不触发额外加载），但要确认在极端极短请求（< block_size）时不会出现负数或 0‑除错误。现有代码已在 `token_len < block_size` 时直接返回 0，保持安全。  
3. **性能影响**：统一块对齐后，外部 KV‑cache 完全命中时仍会触发一次 **异步加载**（`load_async=True`），但加载的 token 数量可能比之前少（因为会在块边界上截断），这对网络和磁盘 I/O 有正向收益。建议在性能基准中对比 `discard_partial_chunks=True/False` 的旧行为，以验证提升幅度。  
4. **测试覆盖**：新加入的两条单元测试覆盖了“完整命中且需保留子块”和“完整命中且本地已缓存全部块”两种场景，确保不会出现 `ZeroDivisionError` 或 `KeyError`。若项目中还有其他调度路径（如跨层块打包 `enable_cross_layers_blocks`），请相应补充测试。  
5. **文档同步**：文档已删除对应参数说明，但仍保留旧章节的引用位置，建议检查文档生成脚本，确保不出现残留的 `discard_partial_chunks` 提示。  

总体而言，此次改动消除了因部分块被丢弃导致的加载计算错误，提升了 KV‑store 与 vLLM 之间的对齐可靠性。只要在升级时注意配置兼容，即可安全采纳。
