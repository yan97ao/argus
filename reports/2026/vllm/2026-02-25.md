### [Model][Spec Decode] Nemotron-H MTP and Mamba Speculative Decoding Support (#33726)
**SHA**: `f5972a8` | 🔗 [查看提交](https://github.com/vllm-project/vllm/commit/f5972a872fa3fabd94b7a6c6f031f4b5bcee2b2d)

**🎯 变更类型**：功能增强 / 架构变更 / 性能优化  
**⚡ 重要程度**：🔴高  

**📋 变更摘要**  
此次提交为 **vLLM** 增加了 Nemotron‑H 多步预测（MTP）模型以及对 **Mamba** 系列模型的 **Speculative Decoding**（多 token 推理）支持。关键改动包括：

1. 在模型注册表中加入 `NemotronHMTPModel`，并在 `tests/models/registry.py` 添加占位示例。  
2. 拓展 `SpeculativeConfig` 与 `VllmConfig`，引入 `num_speculative_tokens` 属性以统一获取 speculative token 数目。  
3. 在 `nemotron_h` 基础模型中实现 **MTP 层**（`NemotronHMTPAttentionDecoderLayer`、`NemotronHMTPMoEDecoderLayer`），并新增完整的 **MTP 模型包装** `NemotronHMTP`.  
4. 大幅修改 Mamba 相关层（`abstract.py`、`mamba_mixer.py`、`mamba_mixer2.py` 等），加入对 `state_indices_tensor_p/d`、`query_start_loc_d`、`num_accepted_tokens` 等 speculative‑decoding 元数据的处理。  
5. 更新 `mamba_attn` 与 `mamba2_attn` 元数据构造器，使其在有 `num_speculative_tokens>0` 时走 **speculative decode** 路径，并在 CUDA graph 捕获阶段使用持久化 buffer。  
6. 相应地在 `vllm/config/vllm.py`、`vllm/config/speculative.py`、`vllm/transformers_utils/configs/nemotron_h.py` 等处加入配置字段 `nemotron_h_mtp`、`mtp_hybrid_override_pattern`、`num_speculative_tokens`。  

**🎯 影响范围**  
- **模型层**：`vllm/model_executor/models/nemotron_h.py`、`vllm/model_executor/models/nemotron_h_mtp.py`、`vllm/model_executor/layers/mamba/*`。  
- **配置层**：`vllm/config/speculative.py`、`vllm/config/vllm.py`、`vllm/transformers_utils/configs/nemotron_h.py`。  
- **调度/运行时**：`vllm/v1/worker/gpu_model_runner.py`、`vllm/v1/attention/backends/mamba_attn.py`、`vllm/v1/attention/backends/mamba2_attn.py`。  
- **注册表与测试**：`vllm/model_executor/models/registry.py`、`tests/models/registry.py`、`tests/v1/attention/test_mamba_update_block_table.py`。  

**🔍 技术洞察**  

| 维度 | 影响 |
|------|------|
| **架构影响** | ① 新增 **MTP 预测模型**（`NemotronHMTP`），在原 `NemotronH` 基础上分离出专门的 draft（MTP）网络，使用 `ModuleDict` 动态构建层；<br>② 在 `SpeculativeConfig` 与 `VllmConfig` 中统一 `num_speculative_tokens`，所有层通过 `vllm_config.num_speculative_tokens` 读取，减少了散落的 “speculative_token” 参数；<br>③ Mamba 系列层的元数据结构从单一 `state_indices_tensor` 扩展为 **prefill / decode 双路** (`state_indices_tensor_p/d`)，并加入 `query_start_loc_d` 与 `num_accepted_tokens`，实现**并行预填 + 多 token 推理**。 |
| **性能影响** | - **吞吐提升**：在支持 speculative decoding 的模型上，最多可一次性前向 `num_speculative_tokens`（默认 0 → 1）个 token，理论上显著降低 decode‑only 场景的调度与 kernel 启动开销。<br>- **额外复制成本**：在 CUDA graph 捕获阶段，为 decode 引入了持久化 `state_indices_tensor_d`、`decode_num_accepted_tokens` 等 buffer，增加了少量拷贝（`torch.copy_`），对极端高并发的 capture 场景可能产生微小延迟。<br>- **内存占用**：MTP 层会额外创建 `embed_tokens`、`lm_head`（权重共享），以及 `num_speculative_tokens` 大小的 `conv_state` padding（`conv_state_shape` 通过 `+ num_spec` 扩展），导致显存略增。 |
| **安全考虑** | - 代码改动均在内部计算路径，无对外 API、网络或文件系统的新增访问，安全风险极低。<br>- **权重加载逻辑**：`NemotronHMTP.load_weights` 对名称进行多层映射与过滤，若映射错误可能导致未加载的权重被误当作已加载，从而产生未初始化的参数；但这属于功能/完整性风险而非安全泄露。 |
| **可维护性** | - **代码复杂度提升**：新增的 `*_p` / `*_d` 双向变量、多个 `if self.use_spec_decode` 分支，导致相关层的逻辑更难追踪。<br>- **后向兼容性**：在 `nemotron_h` 中加入 `is_spec_layer` 检查并提前 `continue`，避免旧模型在加载时误触 MTP 权重，基本保持向后兼容。<br>- **测试覆盖**：仅增加了 `test_mamba_update_block_table` 的少量改动，针对 MTP 的专门测试仍不足。 |

**⚠️ 潜在风险**  

1. **元数据不匹配**  
   - `state_indices_tensor_p/d`、`query_start_loc_d`、`num_accepted_tokens` 的 shape/切分逻辑极其细致，任何 off‑by‑one 或维度不匹配都会在 decode 时导致 **CUDA illegal memory access**。  
2. **CUDA Graph 捕获失效**  
   - 对 `mamba_cache_mode == "all"` 的特殊处理依赖持久化 buffer；在 speculative 场景下若 `num_speculative_tokens` 与实际 capture batch 大小不匹配，可能导致 **graph replay 错误**。  
3. **显存激增**  
   - `conv_state_shape` 在 `mamba_utils.mamba2_state_shape` 中通过 `+ num_spec` 扩展，若 `num_speculative_tokens` 被误设为大值（如 32），每层的 conv cache 将膨胀 32 倍，导致 OOM。  
4. **权重加载遗漏**  
   - `NemotronHMTP.load_weights` 过滤 `if not name.startswith("mtp.")`，若未来模型的 MTP 参数不遵循 `mtp.` 前缀，将导致权重未被加载。  
5. **API 兼容性**  
   - `VllmConfig.num_speculative_tokens` 新增属性；旧版调用 `VllmConfig` 的外部项目若直接访问属性字典而不使用属性访问，可能出现 `AttributeError`。

### [Misc][LoRA] Increase max vocab size limit to 258048 in logits processor (#34773)
**SHA**: `42489e4` | 🔗 [查看提交](https://github.com/vllm-project/vllm/commit/42489e43c2718674828ece00eefc0f11088e801d)

**🎯 变更类型**：功能增强 / 兼容性修复  
**⚡ 重要程度**：🟡 中  

**📋 变更摘要**  
- 将 LoRA 相关的 logits processor 对 vocab‑size 的限制从原来的 “≤ 257 024” 放宽至 “≤ 258 048”，并要求 > 32 000。  
- 相应地在测试中加入了更大 vocab‑size（258 048）以及非法值（258 049、300 000）的边界检查。  
- 调整了测试用的 dummy model、fixture 中的 `unpaded_vocab_size` 与 `ParallelLMHead`、`LogitsProcessor` 的初始化尺寸，以匹配新上限。  

**🎯 影响范围**  
- `vllm/lora/layers/logits_processor.py`：核心校验逻辑。  
- `tests/lora/*`：新增/修改的参数化测试，包括非法 vocab‑size 检验。  
- 相关的 dummy model 构造（`tests/lora/conftest.py`）也同步更新。  

**💡 关注建议**  
1. **文档同步**：README、API 文档或 LoRA 使用说明中仍可能保留旧的 “≤ 257 024” 表述，务必更新为 “> 32 000 且 ≤ 258 048”。  
2. **边界一致性**：代码中出现硬编码的 “32000” 与 “258048”，如果后续继续放宽，需要将这些阈值抽取为常量或配置项，避免再次出现遗漏。  
3. **兼容性检查**：在非 CUDA 环境下 `LogitsProcessorWithLoRA.create_lora_weights` 仍会抛 `ValueError`，请确保上层调用在捕获异常后给出明确错误信息或 fallback。  
4. **性能影响**：放宽 vocab‑size 本身不改变计算路径，但会导致 LoRA 权重矩阵在极大 vocab 时占用更多显存，建议在发布说明中提示用户显存需求的潜在增长。  

总体来看，此次改动合理扩大了 LoRA 的适用范围，并通过新增测试验证了边界行为，风险有限。后续只需关注文档同步和阈值抽象即可。

### [Bugfix] Fix step3p5 reasoning with interleaved thinking (#34211)
**SHA**: `af5e6af` | 🔗 [查看提交](https://github.com/vllm-project/vllm/commit/af5e6afa0af28b75409104906fe47ac9e8c03cf1)

**🎯 变更类型**：Bug 修复  
**⚡ 重要程度**：🟡 中（影响模型推理的流式输出，若不修复会导致推理结束检测延迟）  

**📋 变更摘要**  
- 重新实现 `step3p5_reasoning_parser` 中对 `</think>` 结束标记的检测逻辑。  
- 删除原先通过 `end_offset` 人为延迟一次检测的做法，改为在解析器内部维护 `_end_token_pending` 状态，依据 **最近一次出现的 `<think>` / `</think>`** 以及后续是否出现新 token 来决定是否结束。  
- 新增大量单元测试 `tests/reasoning/test_step3p5_reasoning_parser.py`，覆盖普通、流式、换行、跨 turn 等边界情况，确保 `is_reasoning_end`、`extract_content_ids` 等方法在各种输入下返回正确结果。  

**🎯 影响范围**  
- **核心模块**：`vllm/reasoning/step3p5_reasoning_parser.py`（解析器实现）  
- **测试模块**：`tests/reasoning/`（新增 340 行测试）  
- 可能连带影响使用该 parser 的推理调度路径（如 `ReasoningParserManager` 调用处），但接口未变，兼容性保持。  

**💡 关注建议**  
1. **状态复位**：`_end_token_pending` 在每次完整推理结束后应显式复位，防止残留影响后续请求（当前在 `is_reasoning_end`/`_is_reasoning_end_from_ids` 中已处理，但建议在 `extract_reasoning` 等入口显式清零）。  
2. **性能评估**：新实现在每次调用时遍历 `input_ids` 逆序查找最近的特殊标记，复杂度为 O(k)（k 为最近特殊标记到序列尾部的距离），通常很小；建议跑一次基准对比，确保不会引入明显的延迟。  
3. **文档更新**：`ReasoningParser` 的注释或 README 中应说明 “在流式模式下，结束标记需要后随至少一个 token” 的新规则，以免使用者产生误解。  
4. **兼容性回归**：虽然单元测试已覆盖多数场景，仍建议在真实模型（`stepfun-ai/Step-3.5-Flash`）上执行一次完整的推理/流式对话脚本，验证新 parser 与实际 token 流匹配。  

整体来看，此次改动解决了 **“</think> 后换行导致结束检测延迟一轮”** 的 bug，逻辑更加直观且通过大量测试验证，风险可控。只需关注状态清理与文档同步即可。

- 调整了宏 `WVSPLITKQ` 参数，使 N=1‑4 时均使用 16‑thread‑block 配置（`WVSPLITKQ(16,…)`），为新 MFMA 形状匹配。  

**🎯 影响范围**：

**💡 关注建议**  
1. **功能验证**：在支持 16×16 MFMA（MI4xx）和不支持的老 GPU（如 MI200）上分别跑单元测试，确认数值误差在 FP8 规范内且与旧实现一致。  
2. **性能基准**：对比旧 32×32 MFMA 与新 16×16 MFMA 在不同 batch/seq 长度下的吞吐，确保改动带来预期提升。  
3. **回退路径**：当前代码未检测硬件特性，若在不支持 16×16 MFMA 的机器上运行会触发未定义行为，建议在 kernel launch 前添加 `hipDeviceGetAttribute` 检查并回退到原实现。  
4. **代码可维护性**：`scalar8` 与 `floatx16` 的定义应保持同步，防止未来对 `bigType` 结构的改动遗漏更新。  

总体来看，这是一项面向新硬件的性能迁移，改动明确且影响范围局限在 ROCm 瓶颈 GEMM。只要做好兼容性检测和数值回归，就能安全合入。

### [Perf] Optimize Python Slice for Structured Output using `islice` instead of [:] (#33593)
**SHA**: `9ce8fad` | 🔗 [查看提交](https://github.com/vllm-project/vllm/commit/9ce8fad2a9fe010ede46b085f6cb4099c8ec402d)

**🎯 变更类型**：性能优化  
**⚡ 重要程度**：🟡 中  
**📋 变更摘要**：将结构化输出推理过程中的切片操作从 `list[:]` 改为惰性迭代 `itertools.islice`，并把参数类型从 `Sequence[int]` 放宽为 `Iterable[int]`，以避免不必要的复制和临时列表，提升流式推理的运行时性能。  

**🎯 影响范围**  
- `vllm/reasoning/*` 各种 ReasoningParser 实现（basic、step3、step3p5、deepseek_v3、identity、mistral）

**💡 关注建议**  
1. **接口兼容**：由于 `is_reasoning_end_streaming` 参数改为 `Iterable[int]`，请确认所有内部或外部调用仍能提供可迭代对象（如 `list`、`torch.Tensor` 的 `tolist()`），避免出现 `TypeError`。  
2. **性能验证**：建议在大批量流式推理的基准测试中对比改动前后的延迟与内存占用，特别是长序列（>10k tokens）场景，确保 `islice` 确实带来显著改进。  
3. **代码可读性**：`islice(input_ids, 0, max(0, len(input_ids)-1))` 与 `input_ids[:-1]` 语义等价，建议在关键位置添加注释，帮助后续维护者快速理解“避免复制”。  
4. **边界情况**：检查 `delta_from` 为负数时的 `start` 计算是否与旧实现保持一致，防止在异常回滚或重试时出现 off‑by‑one 错误。  
5. **单元测试**：补充针对 `Iterable` 参数的测试，尤其是使用生成器或 `itertools.chain` 之类的惰性序列，以验证逻辑在非 `list` 场景下仍然正确。  

总体来看，此次改动通过惰性切片降低了内存开销，属于安全的局部性能优化，但需注意兼容性与边界条件的完整测试。
