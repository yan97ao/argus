### [New Model][ROCm] Add AMD support for DeepSeek V4 (#40871)
**SHA**: `628c436` | 🔗 [查看提交](https://github.com/vllm-project/vllm/commit/628c43630155b9fa8a8ec8871eb90c06bf8b36e6)

**🎯 变更类型**：功能增强（为 DeepSeek V4 引入 AMD ROCm 支持）  

**⚡ 重要程度**：🔴 高  

**📋 变更摘要**  
1. 在 CMake 构建链中加入 `fused_deepseek_v4_qnorm_rope_kv_insert_kernel.cu` 并对 CUDA/ROCm 条件编译进行统一处理。  
2. 将多个底层 CUDA kernel（Top‑K‑softplus‑sqrt、MoE、FP8 量化等）迁移为兼容 ROCm 的实现，新增 `USE_ROCM` 宏、FP8 E8M0 解码、以及基于 HIP/ROCm 的自定义 op。  
3. 在 Python 层引入 ROCm 路径的 fallback 与平台检测：关闭 CUDA‑专属特性（PDL、aux‑stream、skip‑kv‑cache‑insert），添加 ROCm‑专属实现（`rocm_*` 系列函数），并在配置/平台枚举中加入 `deepseek_v4_fp8`、`triton_unfused` 等标识。  

**🎯 影响范围**  

| 受影响模块/文件 | 关键改动 |
|----------------|----------|
| `CMakeLists.txt` | 新增 ROCm 兼容 kernel 源文件，调整 CUDA 条件链接 |
| `csrc/*.cu` | 条件编译 `USE_ROCM`，实现 HIP‑compatible FP8、mask、final‑mask、warp‑reduce 等 |
| `csrc/moe/*.cu` | 移除 CUDA‑专属宏，改用宽度可配置的 `VLLM_SHFL_XOR_SYNC_WIDTH` 以兼容 64‑lane warp |
| `vllm/model_executor/*` | 多处平台分支：禁用 aux‑stream、PDL、skip‑k‑cache‑insert；新增 `rocm_*` 参考实现和 fallback；FP8‑block‑scaled‑mm 在 ROCm 上进行 E8M0 → FP32 解码 |
| `vllm/platforms/rocm.py` | 新增模型特性标记 `deepseek_v4_fp8` |
| `vllm/config/kernel.py` | 新增 `triton_unfused` 选项，用于 ROCm 上的 MXFP4 路径 |
| `vllm/v1/attention/ops/*` | `rocm_aiter_mla_sparse.py` 添加完整的 ROCm native sparse‑attention 实现、top‑k 采样 torch fallback、块尺度展开等 |
| `requirements/rocm.txt` | 增加 `tilelang==0.1.9` 依赖 |
| 测试 `tests/kernels/moe/test_topk_softplus_sqrt.py` | 使用 `is_cuda_alike()` 兼容 ROCm |  

整体影响遍及编译系统、底层 CUDA/HIP kernel、量化路径、MoE、稀疏注意力以及模型执行层。  

**🔍 技术洞察**  

- **架构影响**  
  - **平台抽象层统一**：通过 `current_platform.is_rocm()` 将原本只针对 CUDA 的代码路径分离，形成双后端（CUDA / ROCm）结构。  
  - **自定义 Op 增量**：新增多个 `torch.ops.vllm.*` HIP‑ops（`rocm_aiter_sparse_attn_indexer`、`indexer_k_quant_and_cache` 等），以及对应的 C++/HIP 实现，形成与 CUDA 对等的功能集合。  
  - **后端特性降级**：在 ROCm 环境下禁用多流 GEMM、PDL（Program‑Driven Launch）以及 `skip_k_cache_insert`，避免已知的硬件/驱动挂起问题。代码路径仍保留这些特性，仅在 CUDA 上激活。  
  - **量化统一**：对 FP8 E8M0 的解码在 ROCm 上统一在 Python 层完成，防止 HIP‑kernel 直接处理不受支持的 `__nv_fp8_*` 类型。  

- **性能影响**  
  - **正向**：新增的 `rocm_aiter_*` 实现使用 AMD A‑ITer（CK）后端或 Triton‑unfused 路径，在 GFX 942（MI250）等新硬件上可发挥对应算子（如 `deepgemm_fp8_paged_mqa_logits`）的高吞吐。  
  - **负向**：  
    - 关闭 aux‑stream 与 PDL 会导致原本在 CUDA 上的 **并行 GEMM + KV‑insert** 工作在单流串行执行，短期内会降低吞吐，尤其在多‑token decode 场景。  
    - 部分 kernel（如 Top‑K‑softplus‑sqrt）在 ROCm 上采用宽度可调的手写 reduction，可能略低于 NVIDIA 原生 warp‑shuffle 优化。  
    - FP8 E8M0 的上采样在 Python 层完成，会产生额外的拷贝与显存占用，可能在大批量推理时形成瓶颈。  
  - **内存**：`rocm_dequantize_blocked_k_cache` 通过一次性展开块尺度，避免在 GPU 上额外分配中间 buffer，但在 ROCm 上仍会产生额外的 BFloat16 临时张量。  

- **安全考虑**  
  - **新 C/C++/HIP 代码**：大量新增的 HIP kernel 与自定义 op 未经过长期社区审计，潜在的 **缓冲区越界、未对齐访问** 风险仍需通过 GPU‑GPU 细致的单元测试来排除。  
  - **依赖升级**：`tilelang==0.1.9` 与 `amd-quark` 可能引入供应链风险；确保使用受信任的 PyPI 包，并对其二进制轮子进行 hash 验证。  
  - **平台检测**：`is_cuda_alike()` 与 `is_rocm()` 的实现若出现误判，可能在不支持的硬件上调用 ROCm‑only代码，导致 runtime 错误或崩溃。建议在入口处添加明确的 **异常信息**。  

**⚠️ 潜在风险**  

| 风险点 | 可能后果 | 缓解措施 |
|--------|----------|----------|
| ROCm 下 aux‑stream 为 `None` 但仍有代码尝试访问 `self.aux_stream_list[:3]` | `AttributeError` / 运行时崩溃 | 在所有使用前统一 `aux_streams = self.aux_stream_list or []`，并在调用前检查长度 |
| FP8 E8M0 上采样在大规模推理时显存翻倍 | OOM 错误 | 提供 `--no-rocm-fp8-upcast` 选项或在 kernel 中直接支持 E8M0 |
| `topk_softplus_sqrt` 迁移后 `FINAL_MASK` 变为 64‑bit，若在旧 ROCm 驱动上不兼容可能导致 illegal instruction |

### [ROCm][Quantization][2/N] Refactor quark_moe w4a8 w/ oracle  (#39136)
**SHA**: `1e95004` | 🔗 [查看提交](https://github.com/vllm-project/vllm/commit/1e9500410a21782847ae86561b4de7f3aa69f0bc)

**🎯 变更类型**：功能增强 / 重构 / 性能优化  
**⚡ 重要程度**：🔴高  

**📋 变更摘要**  
1. 为 ROCm 平台实现了全新的 **MXFP4 W4A8**（权重量化 MXFP4 + 静态 FP8 激活）后端，并在 `aiter` 框架上提供了 Triton‑GPU Kernels。  
2. 重构了量化后端选择/映射逻辑：新增 `AITER_MXFP4_BF16`、`AITER_MXFP4_FP8` 两个枚举值、对应的 kernel 类 (`AiterExperts`、`AiterW4A8ExpertsMonolithic`) 以及统一的 **oracle** 接口 (`convert_to_mxfp4_moe_kernel_format`、`make_mxfp4_moe_quant_config`、`make_mxfp4_moe_kernel`)。  
3. 扩展了激活函数实现：新增 `swigluoai`（交错布局）和对 `reference_moe` 的参数化，使得 **SWIGLUOAI**、**SILU** 与 **RELU2** 三种激活均可在 chunked 与 interleaved 布局下切换。  
4. 在测试层面加入 **ROCm‑Oracle** 端到端精度/功能验证，包括多种后端配置、权重/尺度转换、kernel 创建、输出形状/数值检查。  
5. 清理并统一了 `select_mxfp4_moe_backend` 与 `select_deepseek_v4_mxfp4_moe_backend` 的实现，让激活量化键（`kFp8StaticTensorSym`）能够驱动 W4A8 路径的自动选择。  

---  

**🎯 影响范围**  
- **核心组件**：`vllm/model_executor/layers/fused_moe/*`（oracle、backend映射、kernel抽象）  
- **量化子模块**：`vllm/model_executor/layers/quantization/mxfp4.py`、`quark/quark_moe.py`（后端选择、权重处理）  
- **平台层**：`vllm/platforms/rocm`（GFX950 检测）  
- **测试套件**：`tests/kernels/moe/test_ocp_mx_moe.py`（新加入的 ROCm‑Oracle 测试）  
- **依赖**：新增对 `aiter`（ROCm‑AITER）和 Triton‑Kernels 的可选 import，新增 `kFp8StaticTensorSym` 用于激活量化键。

### 🔍 技术洞察  

| 维度 | 影响分析 |
|------|----------|
| **架构影响** | • 引入了 **oracle** 机制，将权重转换、量化配置、kernel 创建抽象为统一函数，提升了不同后端（TRITON、MARLIN、AITER、EMULATION）的可组合性。<br>• `AiterW4A8ExpertsMonolithic` 作为 **Monolithic** 实现，遵循 `mk.FusedMoEExpertsMonolithic` 接口，使得后端切换不影响上层 MoE 调度逻辑。<br>• 新增 `select_deepseek_v4_mxfp4_moe_backend`，在 DeepSeek‑V4 场景中自动依据激活量化键选取 BF16（W4A16）或 FP8（W4A8）后端，降低了模型配置的显式硬编码。 |
| **性能影响** | • **W4A8**（MXFP4 + 静态 FP8）在 ROCm GFX950 上使用 **aiter.triton.moe_gemm_a8w4**，理论上比 BF16（W4A16）拥有约 **2×** 的算子吞吐（FP8 × 4 bits），并且保持了 `swiglu` 交叉‐gate 的激活计算在 `float8_e4m3fn` 中完成，显著降低内存带宽。<br>• 通过 `downcast_to_static_fp8` 在 kernel 前对输入进行一次 static‑FP8 量化，避免了运行时逐 token 动态量化的开销。<br>• 代码在 `apply` 与 `apply_monolithic` 中统一使用 `aiter_triton_kernel_w4a8_moe_forward`，避免了在 `triton_kernel_moe_forward` 中的分支路径，从而降低了分支预测失误。<br>• 可能的 **热点**：`moe_gemm_a8w4` 中的 `gather_idx` / `scatter_idx` 以及 `downcast_to_static_fp8`，在极大 batch/ token 数时仍受限于显存规模与 `torch.float8` 的硬件支持。 |

| 方向 | 建议 |
|------|------|

| **回退兼容** | 对于已存在的 `copy.deepcopy(tokenizer)` 调用（如老版插件），仍可工作，但会产生额外拷贝。建议在迁移期添加兼容性警告，鼓励统一使用 `maybe_make_thread_pool`。 |
| **测试** | 添加并发调用 tokenizer 的单元测试，覆盖 `encode/decode`、`apply_chat_template`，并在异常路径（`queue.Empty`）下验证会自动创建临时拷贝。 |

总体而言，此次改动通过“池化深拷贝”彻底解决了 Rust tokenizer 的并发抢占问题，消除了原有的“重试‑睡眠”方案的非确定性。后续需要关注内存占用和序列化在分布式环境中的表现，并补全相应的测试和文档。

- **并发安全**：`current_draft_step` 在多请求批次间共享，`fill_` 与 `store` 在 Triton kernel 中同步，理论安全；但若在同一 batch 中出现不同 step 的请求（例如因 `skip_attn` 产生的早停），仍会把同一 step 写入所有请求，需要在上层确保 batch 中的所有请求步号统一。  

**建议**  
1. 为 `gumbel_sample`、`prepare_eagle_inputs` 添加单元测试，覆盖 `processed_logits_out=None`、`processed_logits_out` 与 `current_draft_step` 同时为非空的场景。  
2. 在 benchmark 中对比 `num_speculative_steps=1/2/4` 时的吞吐量与显存占用，确认多步 metadata 重建的开销在可接受范围。  
3. 增加 `assert` 检查：`output_processed_logits.shape[1] == vocab_size`、`output_processed_logits_col.shape == (1,)`，提前捕获维度错误。  
4. 更新文档，说明在使用 Eagle‑speculative 时必须通过 `ModelRunnerV2`（或相应 API）开启 `rebuild_attn_metadata=True`，否则会回退到旧的单步实现。  

总体来看，此次改动实现了在多步 draft 期间及时刷新注意力元数据，提升了全图 CUDAGraph 的正确性；若严格遵循新增接口并做好回归测试，风险可控。
