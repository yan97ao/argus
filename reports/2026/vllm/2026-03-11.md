### Kimi k2.5 MLA based eagle3 (#36361)
**SHA**: `5573894` | 🔗 [查看提交](https://github.com/vllm-project/vllm/commit/557389473755bff50b6d00c03ca5c68e5c37c9a0)

**🎯 变更类型**：功能增强 / 架构变更 / 新增模型支持  
**⚡ 重要程度**：🔴高  
**📋 变更摘要**  
- 在 vLLM 中新增对 **Eagle3** 规格解码的支持，首次引入 **Deepseek‑V2/V3** 与 **Kimi‑K2.5**（aka K2.5）模型的 Eagle3 实现。  
- 新增 `SupportsEagle3` 接口、对应模型实现 `deepseek_eagle3.py`、`kimi_k25.py` 的 Eagle3 适配层，以及注册表、配置与 speculative 参数的扩展。  
- 实现了 **auxiliary hidden‑state** 抽取、draft‑to‑target 词表映射、以及权重加载的细粒度处理，配合原有 Eagle2 机制实现更高的推理并行度。

---

### 🎯 影响范围
- **模型执行层**：`vllm/model_executor/models/deepseek_v2.py`、`vllm/model_executor/models/kimi_k25.py`、`vllm/model_executor/models/deepseek_eagle3.py`。  
- **模型注册与调度**：`vllm/model_executor/models/registry.py`、`vllm/v1/spec_decode/eagle.py`。  
- **配置体系**：`vllm/config/speculative.py`、`vllm/transformers_utils/config.py`。  
- **公共接口**：`vllm/model_executor/models/interfaces.py`（新增 `SupportsEagle3`），以及相关单元测试 (`tests/models/registry.py`)。  
- **权重加载逻辑**：`vllm/model_executor/models/utils.py` 中的 `process_eagle_weight`、`AutoWeightsLoader` 调用被扩展。

---

### 🔍 技术洞察

| 维度 | 影响说明 |
|------|----------|
| **架构影响** | 1. **新增模型类** `DeepseekV2Eagle3Model`、`Eagle3DeepseekV2ForCausalLM`（兼容 V3）。<br>2. 引入 **aux hidden‑state layers**（在 `DeepseekV2ForCausalLM` 中通过 `aux_hidden_state_layers` 控制），实现 Eagle3 所需的多尺度隐藏状态合并。<br>3. `SupportsEagle3` 接口使得调度器能够统一处理 Eagle3 与 Eagle2，实现 **插件化** 的解码策略。<br>4. 通过 `speculative_config.draft_model_config` 把 Draft（草稿）模型与 Target（主模型）解耦，使得 **多模型混合** 成为可能。 |
| **性能影响** | - **吞吐提升**：Eagle3 通过提前生成多个 token（草稿），在 LLM 的自回归环节可并行化，理论上适用于 **Deepseek‑V2/V3** 与 **Kimi‑K2.5** 的 8B~13B 参数规模，预计提升 1.5‑2.5×（取决于 GPU 规格、batch size）。<br>- **内存占用**：新增 `aux_hidden_state_layers`、`fc` 投影层以及 Draft‑to‑Target 词表映射会额外占用 **≈ 0.2‑0.4 GB**（取决于模型大小），在多卡部署时需留意显存裕度。<br>- **编译兼容**：`@support_torch_compile` 装饰器已加入，允许 Torch‑Dynamo 编译，进一步降低 CPU‑GPU 交互开销。 |
| **安全考虑** | - 仅在模型注册表中添加了 `trust_remote_code=True` 的条目（Deepseek‑V2/V3、Kimi‑K2.5），仍依赖 **HF** 的远程代码执行，需在生产环境显式审计模型仓库。<br>- 新增的 `draft_id_to_target_id` 映射不涉及外部 I/O，未引入额外的安全漏洞。 |
| **可维护性** | - 代码结构与已有 Eagle2 实现保持高度相似，复用了 `DeepseekV2MLAAttention`、`DeepseekV2MLP`，降低学习成本。<br>- 通过 `process_eagle_weight` 抽象权重处理，避免在每个模型中重复实现。<br>- `aux_hidden_state_layers` 的默认实现 (`(2, N//2, N-3)`) 集中在 `DeepseekV2ForCausalLM`，易于统一调优。 |

### ⚠️ 潜在风险
1. **权重加载兼容性**  
   - `load_weights` 中对 `"midlayer."`、`"scale"`、以及 **sharded** 参数的特殊处理可能在未来的 HF 权重结构变化时失效。  
   - 若模型仓库未提供 `draft_id_to_target_id` 映射，会导致 `skip_substrs` 过滤掉关键参数，进而产生 **logits shape mismatch** 错误。  

2. **模型注册冲突**  
   - 同时在 `registry.py` 注册了 `"Eagle3DeepseekV2ForCausalLM"` 与 `"Eagle3DeepseekV3ForCausalLM"`（V3 直接指向 V2 实现），如果后续加入真正的 V3‑专属实现，可能导致 **意外共享状态**。  

3. **Eagle3 业务场景误用**  
   - Eagle3 需要 **draft model** 与 **target model** 参数匹配（tokenizer、vocab size、位置编码等）。不匹配会导致 **hidden state 对齐错误**，尤其在多模态（image/video）模型中。  

4. **性能回退**  
   - 对于 **小 batch** 或 **GPU 显存紧张** 的部署，Eagle3 的额外 `fc` 投影和 aux hidden‑state 存储可能导致 **吞吐下降**，而且 `torch.compile` 对某些自定义 ops 支持仍不完善。  

5. **测试覆盖不足**  
   - 目前仅在 `tests/models/registry.py` 添加了注册验证，缺少 **端到端推理**、**权重加载**、**speculative decoding 正确性** 的单元/集成测试，可能隐藏隐藏状态拼接或 logits 归一化错误。

### Add XPU MLA Sparse backend for DeepSeek v3.2 (#33230)
**SHA**: `e584dce` | 🔗 [查看提交](https://github.com/vllm-project/vllm/commit/e584dce52b9584ffb0fc4a1a4cd31163d4257a41)

**🎯 变更类型**：功能增强 / 架构变更 / 性能优化  
**⚡ 重要程度**：🔴高  
**📋 变更摘要**：  
- 为 Intel XPU（GPU‑like 加速器）新增 **MLA（Multi‑query LLaMA）稀疏注意力** 后端，实现 DeepSeek v3.2‑compatible 的 FlashMLA Sparse kernel。  
- 新增平台检测、后端注册、稀疏索引器以及 Triton bf16 稀疏实现，并配套单元测试。  
- 同时在 XPU 上统一了 `top_k_per_row_*` 计数逻辑，使调度与 CUDA/ROCm 保持一致。

**🎯 影响范围**：  
- `vllm/_xpu_ops.py`（量化、缓存写入、top‑k 采样）  
- `vllm/platforms/xpu.py`（后端选择）  
- `vllm/v1/attention/backends/registry.py`（枚举新后端）  
- `vllm/v1/attention/backends/mla/xpu_mla_sparse.py`（核心实现）  
- `vllm/v1/attention/ops/xpu_mla_sparse.py`（Triton kernel）  
- `vllm/model_executor/layers/sparse_attn_indexer.py`（调度路径）  
- 文档、测试文件（`tests/kernels/attention/test_xpu_mla_sparse.py`）

| **安全考虑** | - 新增代码均在用户可控的 Torch/XPU 环境下执行，未引入网络 I/O、文件系统写入等危险操作。<br>- 关键点在 **索引合法性检查**：`group_quant_torch` 对 `group_size`、`stride` 做了断言；稀疏索引器在 kernel 前对越界 `-1` 做了 mask 处理，防止 OOB 读取。<br>- 仍需关注 **torch.xpu.is_available()** 判断失效导致回退到未实现路径的异常抛出。 |
| **可维护性** | - 代码遵循 vLLM 现有的后端抽象（`AttentionBackend`、`MetadataBuilder`），易于后续扩展（例如支持 fp8 kv）。<br>- Triton kernel 使用了常量 `LOG2E / LOGE2` 以及块大小参数，在不同硬件上调参仍较为直观。<br>- 新增的 `group_quant_torch` 实现在 `_xpu_ops.py` 较为冗长，建议抽离为独立模块或复用已有量化库。 |
| **兼容性** | - 对已有 CUDA/ROCm 路径 **不产生影响**，仅在检测到 XPU 且 `use_sparse=True` 时激活。<br>- 文档已同步更新 `attention_backends.md`，但运行时仍依赖 `torch-xpu` 版本 ≥ 2.3（包含 `torch.xpu` 命名空间）。 |

## ⚠️ 潜在风险

1. **平台检测失误**  
   - `current_platform.is_xpu()` 只检查 `torch.xpu.is_available()`。在 XPU 驱动/库不匹配的环境下，代码会走到 `raise NotImplementedError`，导致服务启动失败。  

2. **索引越界/非法值**  
   - 稀疏索引 `indices` 若全部为 `-1`（或超过 `seq_kv`）会在 kernel 中产生 `-inf` 的 logits，随后 Softmax 会产生 `nan`（因为 `logsumexp` 计算 `-inf`）。虽然已有 mask 处理，但在极端全‑invalid 场景仍需特殊检查。  

3. **量化路径兼容性**  
   - `group_quant_torch` 中强制使用 `current_platform.fp8_dtype()`，但在稀疏路径仍未支持 FP8 KV，若用户自行开启 `kv_cache_dtype="fp8"` 会触发 `NotImplementedError`。  

4. **内存占用**  
   - 稀疏 KV 缓存每 token 仍保留完整的 `head_size`（576）+ 4 bytes scale，且 `topk_indices_buffer` 按 `max_num_batched_tokens × topk` 预分配，长序列或大 batch 时可能导致 O(GB) 的额外显存占用。  

5. **测试覆盖不足**  
   - 仅提供了 prefill（单 token）单元测试，缺少 **decode 多 token**、**不同 batch size**、**mixed‑precision** 场景的验证。

### 🔍 技术洞察  

#### 架构影响  
- **新增目标架构**：**RISC‑V (rv64gc) + RVV**。通过 `-march=rv64gcv_zvfh…` 启用向量指令，后端代码在编译期会直接生成 RVV 汇编。  
- **模块化向量抽象**：`vec_op` 命名空间提供统一的 `Vec<T>` 接口，CPU 其它平台（x86/ARM）已有对应实现，新增实现保持同一接口，降低上层代码改动。  
- **宏控制**：`RISCV_BF16_SUPPORT` 根据检测到的 BF16 扩展条件编译 BF16 向量类型，保持向后兼容（在仅有 FP16 的硬件上仍能编译）。  

#### 性能影响  
- **向量化加速**：RVV 的 128‑bit（或更宽）向量宽度对应 8‑16‑32 元素的 FP16/FP32/BF16 操作，理论上可比现有 scalar 路径提升 **4‑8×**（取决于硬件的 `VLEN` 与 `LMUL` 配置）。  
- **激活函数与指数实现**：在 `FP32Vec8/16` 中手写了 `exp、tanh、erf` 的多项式近似，并使用向量化 `vfmul、vfredusum` 等指令实现，避免了标量循环的函数调用开销。  
- **回退路径**：当编译器未检测到 `zvfbfmin`（BF16）时，仍提供基于 FP32 仿真的 BF16 向量实现，保证功能完整性，只是性能会回落到标量/FP32 模式。  

#### 安全考虑  
- **非法指令风险**：若编译标志误认为目标硬件支持 `zvfhmin/zvfbfmin`，但运行时硬件缺失对应扩展，将触发 **非法指令（SIGILL）**。目前依赖 CMake 的 `find_isa` 检测和编译器宏 `__riscv_v`，但在交叉编译或运行时迁移到功能更弱的硬件时仍有风险。  
- **内存访问**：向量 load/store 使用 `__riscv_vle*` 与 `__riscv_vse*`，若传入的指针未对齐至向量宽度或越界，将触发硬件异常。代码中未加入显式对齐检查，仅假设上层调度已确保安全。  
- **浮点异常**：自实现 `exp/tanh/erf` 使用多项式近似，未经过 IEEE‑754 严格的舍入控制，极端输入（如 NaN、Inf）在向量化路径可能产生非预期结果。

- **数据解析**：实现 `KimiAudioMultiModalDataParser` 与 `KimiAudioMultiModalProcessor`，负责音频特征抽取、字段映射 (`whisper_input_features`/`feature_attention_mask`) 以及 Prompt 中占位符的替换逻辑。  
- **权重加载**：通过 `AutoWeightsLoader` 与 `WeightsMapper` 两阶段加载：① LLM 主体和投影层；② Whisper 子模型的 safetensors 权重（含 Q/K/V 融合、fc→mlp 重映射）。  
- **注册中心**：在模型、渲染器、Tokenizer 注册表中分别加入 `"MoonshotKimiaForCausalLM"`、`"kimi_audio"`、`"kimi_audio"` 条目，使得配置文件只需指定模型名称即可自动走到全链路。  
- **Tokenizer Mode**：在 `renderer_from_config` 中强制覆盖 `tokenizer_mode` 为 `kimi_audio`，防止因默认 `auto` 走到不兼容的 HF Tokenizer。  

#### 性能影响
- **前置计算成本**：音频特征抽取利用 Whisper 的卷积层，计算量大约 **O(L·C)**（L 为音频帧数，C≈5120），并在 `embed_multimodal` 后将特征 **4×下采样 + 投影**，整体算子数与已有的 Whisper‑ASR 模型相当。  
- **显存占用**：除 LLM 参数外，还需加载 Whisper 编码器（≈1‑2 GB）以及 VQ‑Adaptor（≈200 MB），在 7B 参数模型上总体显存提升约 **2‑3 GB**。  
- **吞吐量**：在单卡推理时，音频特征抽取是瓶颈；在流水线并行（PP）场景下，`embed_input_ids` 中的 `is_multimodal` 掩码处理已做兼容，但仍需额外的同步步骤，可能导致 **~10‑15%** 吞吐下降。  
- **推理停顿**：在示例中增加了 `stop_token_ids=[151644]`（EOS），防止生成循环。  

#### 安全考虑
- **trust_remote_code**：新增模型默认开启 `trust_remote_code=True`，同其他 Moonshot 系列模型一样，需确认模型仓库可信。  
- **Tokenizer 自实现**：自行解析 `tiktoken.model`，不依赖 HF 官方 Tokenizer，降低了代码执行外部代码的风险。  
- **权重过滤**：`load_weights` 中显式跳过 TTS/MIMO 层，避免加载不必要的、潜在的未审计权重。

### ⚠️ 潜在风险

| 风险点 | 描述 | 可能后果 |
|--------|------|----------|
| **音频特征对齐错误** | `KimiAudioMultiModalProcessor._call_hf_processor` 手动把 `audios` → `audio`，若输入结构不符合（例如多声道或采样率异常）会导致特征抽取失败。 | 推理异常、报错。 |
| **PP 多模态掩码不完整** | `embed_input_ids` 中的 `is_multimodal` 仅在非‑PP 场景经过充分测试，PP 环境下 token 位置可能错位。 | 生成语义错乱或音频特征被错误覆盖。 |

| **测试环境缺少 SpeechToTextConfig** | 单元测试中对 `MoonshotKimiaForCausalLM` 直接 `skip`，意味着真实部署后缺乏 CI 覆盖。 | 隐蔽 bug 难以及时发现。 |

### [Perf] Optimize scheduler overhead for PD disaggregation, around 5% E2E perf improvement (#35781)
**SHA**: `a8ff2cc` | 🔗 [查看提交](https://github.com/vllm-project/vllm/commit/a8ff2cca92807f1b15b9b8d21135784298ad7814)

**变更概览**  
本次 PR 在调度器实现上引入了 **`skipped_waiting`** 队列，用来暂存因异步 KV 加载、结构化输出（FSM）或流式输入等依赖未满足而被跳过的请求。原来的 `waiting` 队列仅保存可直接调度的请求，导致在每轮调度时必须频繁在 `waiting` 中来回搬移、判断状态，产生额外开销。新的实现：

1. **双队列结构**：`waiting`（可直接调度） + `skipped_waiting`（阻塞状态）。  
2. **统一计数**：`_num_waiting_requests`、`get_request_counts`、统计 API 都把两个队列的长度相加。  
3. **调度路径**：在 `schedule()` 中先遍历 `skipped_waiting`（FCFS）或在 PRIORITY 模式下比较两头元素，再决定调度哪个请求。  
4. **状态提升**：新增 `_is_blocked_waiting_status` 与 `_try_promote_blocked_waiting_request`，在遍历时尝试把阻塞请求升为 `WAITING`，成功后直接调度，否则重新放回 `skipped_waiting`。  
5. **入口统一**：新增 `_enqueue_waiting_request` 与 `_select_waiting_queue_for_scheduling`，所有对 `waiting` 的写入统一走此路径，保证阻塞请求被放入 `skipped_waiting`。  
6. **其它调整**：`finish_requests`、`get_num_unfinished_requests`、`make_stats` 等统计函数同步考虑 `skipped_waiting`；测试代码相应改为使用 `_num_waiting_requests`。

**影响范围**  
- **核心调度器（scheduler.py）**：逻辑改动最多，涉及请求入队、出队、状态提升、统计等多处。  
- **请求队列实现**：仍使用统一 `RequestQueue`，但新增 `skipped_waiting` 实例。  
- **KVConnector 交互**：`_update_waiting_for_remote_kv` 被拆分为检查+提升两步，调用点相应更新。  
- **单元及集成测试**：大量 `len(scheduler.waiting)` 判断改为 `_num_waiting_requests`，并显式验证 `skipped_waiting` 的顺序与内容。

整体来看，此次重构通过把阻塞请求与可调度请求分离，显著降低了调度循环中的状态检查与队列搬迁成本，是一次 **调度层面的性能优化**，对现有功能兼容性影响有限，且已通过大量单元测试验证。后续关注并发安全与文档同步即可。

4. **测试覆盖**：新加入的 `padded_a` 参数仅在 ROCm 测试里打开，建议在 CPU/mock‑CUDA 环境下也跑一次以验证填充不影响逻辑。  
5. **性能基准**：`fits_wvsplitkrc` 条件改为 `N_p2 * m * ((k+511)/512) <= 128*1024*12`，该阈值与硬件实际 LDS/SM 配置强关联，建议在不同 GPU（如 gfx1030、gfx1100）上跑基准，确认阈值是否过于保守或激进。  
6. **异常处理**：`throw std::runtime_error("Invalid wvSplitKrc size")` 在 CUDA kernel 调用前直接抛异常，若在多线程环境下调用可能导致进程异常退出，建议改为返回错误码或使用 `AT_ERROR` 统一报错。  

总体而言，此次提交在原子计数与写回路径上做了细致的优化，预计可在大 batch、宽矩阵情况下显著降低原子冲突，提升 ROCm skinny‑gemm 的吞吐。但需关注同步语义、计数清零以及跨硬件的阈值设置，以防出现隐蔽的数值误差或崩溃。
