### [Frontend][1/n] Improve pooling entrypoints | classify. (#35604)
**SHA**: `ea46397` | 🔗 [查看提交](https://github.com/vllm-project/vllm/commit/ea463978bb987a4c15c9b51c0013d620a722aa67)

**🎯 变更类型**：功能增强  

**⚡ 重要程度**：🔴高  

**📋 变更摘要**  
本次提交在 vLLM 中为 *Pooling* 场景（classification、embedding、score 等）引入了统一的 IO 处理抽象和通用的 Serving 基类。新增 `ChatTemplateConfig` 数据类、`PoolingIOProcessor`、`PoolingServing`、IO 处理工厂以及多处调用方的适配（`LLM`、OpenAI serving、SageMaker 等），并对错误统一化处理。整体目标是提升多模态/分类等 pooling 接口的可扩展性、一致性以及安全性。

**🎯 影响范围**  
- `vllm/entrypoints/chat_utils.py` – 新增 `ChatTemplateConfig`。  
- `vllm/entrypoints/llm.py` – `LLM.__init__` 增加 `chat_template` 参数，加载并保存模板配置；引入 `init_pooling_io_processors`。  
- `vllm/entrypoints/openai/engine/serving.py` – 移除 classification 相关路由、更新错误提示。  
- `vllm/entrypoints/pooling/base/io_processor.py` – 核心抽象基类，实现在线/离线预处理、后处理公共逻辑。  
- `vllm/entrypoints/pooling/base/serving.py` – 新增通用 `PoolingServing` 与 `PoolingServeContext`，封装请求生命周期。  
- `vllm/entrypoints/pooling/classify/*` – 分类任务专属 IOProcessor 与 Serving 实现。  
- `vllm/entrypoints/pooling/io_processor_factories.py` – 根据任务动态创建对应 IOProcessor。  
- `vllm/entrypoints/pooling/typing.py` – 为所有 pooling 请求/响应统一类型别名。  
- 其它入口（SageMaker、utils）同步更新以兼容新基类与错误构造函数。

**🔍 技术洞察**  

- **架构影响**  
  - 引入 **IOProcessor** 抽象层，使得不同 pooling 任务（classification、embedding、score 等）只需实现少量专用前后处理逻辑，即可复用统一的渲染、分词、并发等基础设施。  
  - `PoolingServing` 抽象出 **请求预处理 → 生成器准备 → 批收集 → 响应构造** 的完整流水线，取代之前在各任务下散落的实现，提升代码复用度并降低新任务的接入成本。  
  - `ChatTemplateConfig` 将聊天模板相关配置集中管理，配合 `trust_request_chat_template` 标记，实现安全的模板校验。  
  - `init_pooling_io_processors` 工厂函数根据 `SupportedTask` 动态注入对应的 IOProcessor，实现 **插件化**，未来新增任务仅需新增对应 Processor 即可。  

- **性能影响**  
  - `PoolingIOProcessor` 在构造时创建了 `ThreadPoolExecutor(max_workers=1)`，用于在需要时异步执行分词等 CPU 密集操作，避免阻塞主事件循环，提升并发吞吐。  
  - 对于 **offline** 场景（一次性批量推理），提前完成 token 化，减少运行时开销。  
  - 新增的 `merge_async_iterators` 在 `PoolingServing._prepare_generators` 中并行拉取多个引擎生成器的结果，保持原有的 **多流合并** 能力，理论上不会导致显著延迟。  
  - 可能的轻微开销在于额外的对象创建与一次性线程池启动，建议在高并发部署时通过 `max_workers` 调优或复用全局线程池。  

- **安全考虑**  
  - `ChatTemplateConfig.trust_request_chat_template` 搭配 `_validate_chat_template`，在未显式信任时拒绝请求中携带的自定义模板，防止模板注入或路径遍历攻击。  
  - `create_error_response` 统一错误生成逻辑，增加对 `VLLMValidationError`、Jinja2 `TemplateError` 等异常的映射，保证错误信息不泄露内部实现细节。  
  - 将异常栈打印（可通过 `log_error_stack` 控制）从原来的 `print` 改为安全的 `traceback`，避免在生产环境意外泄漏。  

**⚠️ 潜在风险**  

1. **向后兼容性**  
   - `LLM.__init__` 新增 `chat_template` 参数，原有调用若使用位置参数可能产生 TypeError。需更新文档并提供兼容包装。  
   - 移除 OpenAI serving 中的 classification 路由，外部直接依赖 `/classify` 的旧实现会返回 400 错误。  

2. **未实现的 IOProcessor**  
   - `PoolingServing.init_io_processor` 在基类中抛 `NotImplementedError`。如果未来添加新 pooling 任务但未在 `init_pooling_io_processors` 中注册对应 Processor，将导致运行时错误。  

3. **线程池资源泄露**  
   - 每个 `PoolingIOProcessor` 实例创建独立的 `ThreadPoolExecutor`；在高并发部署中可能导致大量线程池实例，进而耗尽系统资源。建议在实例复用场景（如单模型多任务）共享同一线程池或改为全局池。  

4. **错误统一化后的兼容**  
   - 旧代码可能捕获 `OpenAIServing.create_error_response` 返回的 `ErrorResponse`，但现在改为 `utils.create_error_response`，行为保持基本一致但返回对象路径变更，需检查导入路径。  

**💡 关注建议**  

- **升级指南**：在项目升级说明中明确 `LLM` 构造函数签名变化、classification API 已迁移至 pooling 路径，并提供迁移示例。  
- **单元/集成测试**：新增针对 `ClassifyIOProcessor`、`PoolingServing._preprocess`、`_prepare_generators`、`_collect_batch` 的覆盖，确保 offline/online 两条路径都能正确生成 `ProcessorInputs`。  
- **线程池调优**：评估模型部署规模后，可将 `PoolingIOProcessor` 的线程池改为共享全局实例或使用 `concurrent.futures.ProcessPoolExecutor` 以提升 CPU 利用率。  
- **安全审计**：重点审查 `_validate_chat_template` 的边界条件（例如模板文件路径解析、文件读取权限），防止特权提升。  
- **监控与日志**：利用 `request_logger` 记录每个 pooling 请求的 `prompt_token_cnt`、`model_token_cnt` 与 `latency`，帮助捕捉因新 IOProcessor 引入的潜在性能回退。  
- **文档同步**：在官方文档中补充 **Pooling API** 使用说明、`ChatTemplateConfig` 参数意义及 `trust_request_chat_template` 的安全建议。

### 🔍 技术洞察

| 维度 | 分析 |
|------|------|
| **架构影响** | - 引入 `SupportsCrossEncoding` 接口，使模型能够在 **Score / Rerank** 场景下执行跨模态全序列前向，复用 `DispatchPooler` 进行 pooled 表征后接线性分类层。<br>- 权重映射 `WeightsMapper(orig_to_new_prefix={"model.": ""})` 兼容 checkpoint 中 `model.*` 前缀，确保迁移无缝。<br>- 对缺失的内部 `language_model.score.*` 参数进行随机初始化，防止 `nn.Module` 参数不完整导致的 runtime 错误。 |
| **性能影响** | - **计算成本**：跨模态 **Score** 需对每个 query‑document 对执行完整的 transformer 前向，显著高于仅使用向量检索的 **Embedding**（O(N·M) 而非 O(N+M)）。<br>- **内存占用**：在 `DispatchPooler.for_seq_cls` 中保留所有 token 表征直至 pooling，批量较大时 GPU 内存压力会提升。<br>- **初始化**：额外的线性层和随机初始化几乎不增加启动时间，权重映射过程保持 O(参数数)。 |
| **安全考虑** | - 依赖 `--trust-remote-code`（与其他远程模型保持一致），未引入新代码执行路径。<br>- 权重加载时若 checkpoint 缺失 `language_model.score.*`，会使用 **Kaiming/Zero** 初始化；该过程不涉及外部输入，安全风险极低。 |
| **可维护性** | - 新类复用 `LlamaNemotronVLForEmbedding` 基础实现，代码复用率高。<br>- 权重映射与缺失权重填充逻辑集中在 `load_weights`，易于后续 checkpoint 结构变更时修改。<br>- 文档、示例、模板同步更新，使用者可直接参考，无需自行编写跨模态 Prompt。 |

---

### ⚠️ 潜在风险

1. **缺失权重的随机初始化**  
   - 若 checkpoint 实际上应该提供 `language_model.score.*`（例如自定义 head）而未被映射，随机初始化可能导致推理分数偏差，且难以追溯。  
2. **跨模态 Score 计算成本**  
   - 对大量文档进行一次性 `rerank`（如上千条）会导致显存 O(文档数) 爆炸，易触发 OOM。  
3. **模板兼容性**  
   - 该模型强制使用自定义 Jinja 模板；若用户忘记覆盖 `--chat-template`，默认模板会产生错误的输入格式。  
4. **注册冲突**  
   - `LlamaNemotronVLForSequenceClassification` 与已有 `LlamaNemotronVLModel` 同名前缀可能在自定义配置文件里产生歧义（尤其在 `model_type` 自动推断时）。

**🔍 技术洞察**  

| 维度 | 影响分析 |
|------|----------|
| **架构影响** | • 通过 **SpeculativeConfig** 注册新方法，将 hidden‑state 提取视作一种“猜测”路径，保持了现有 spec‑decode 抽象（Proposer、DraftModel）。<br>• 新的 **CacheOnlyAttentionBackend** 与 **CacheOnlyAttentionLayer** 把隐藏状态直接写入 KV‑Cache，避免了注意力矩阵计算，极大简化了算子图。<br>• **ExampleHiddenStatesConnector** 与 KV‑Transfer 子系统解耦：只负责读取 KV‑Cache、写入磁盘，既可以与分布式 KV‑Transfer 组合，也可以在单机上使用。<br>• 引入 **KVConnectorOutput.merge**，为多 connector（如原有和新 connector 并存）提供统一聚合路径。 |
| **性能影响** | • **正向**：因为不进行注意力乘法，隐藏状态写入的计算成本极低，CPU‑GPU 侧基本只做一次 tensor copy。<br>• **负向**：每个请求在生成结束后会将 `[num_layers, prompt_len, hidden_size]` 大小的 tensor 写入磁盘（safetensors），I/O 成本随模型规模呈线性增长。若显存/磁盘带宽不足，可能成为瓶颈。<br>• **内存**：`ExtractHiddenStatesProposer` 为每个批次预分配 `hidden_states` 缓冲区，最大尺寸为 `max_num_tokens × num_hidden_layers × hidden_size`，对长序列或大模型（如 8B+）会占用大量显存。<br>• **并行**：KV‑Connector 仍然遵循原有的跨进程/跨节点 KV‑Cache 组约束，仅在 `extract_hidden_states` 场景下强制要求所有 draft 层属于同一 KV‑Cache 组（已在 `validate_same_kv_cache_group` 中校验）。 |
| **安全考虑** | • 隐藏状态本质上是模型内部激活，可能泄露模型权重特征或用户提示信息。<br>• 新 connector 将文件写入 `shared_storage_path`（默认 `/tmp`），若未显式限制目录权限，可能导致敏感信息被其他进程读取。<br>• 使用 `safetensors.safe_open` 读取时已具备安全校验（防止任意代码执行），但写入侧仍需确保目录安全（建议使用 `chmod 700` 或专属临时目录）。 |
| **可维护性** | • 新增的多个文件（约 1500 行）在代码结构上保持了 vLLM 的插件化风格，易于后期扩展。<br>• `CacheOnlyAttentionBackend` 与现有 AttentionBackend 接口保持一致，后续若需加入更多 “只写” 类算子，可直接复用。<br>• 测试覆盖率高（单元 + 集成），但建议在多节点、多 GPU、不同 KV‑Cache 布局（NHD / HND）下补充兼容性测试。 |

**⚠️ 潜在风险**  

1. **磁盘 I/O 与磁盘空间耗尽**  
   - 大模型、长提示或高并发时，每个请求会产生数百 MB‑GB 的 safetensors 文件，若共享存储路径未预留足够空间，会导致写入失败或磁盘被写满，引起后续请求卡死。  
2. **显存峰值超限**  
   - `hidden_states` 缓冲区在 `max_num_tokens` 基础上乘以 `num_hidden_layers` 与 `hidden_size`，在 `max_num_tokens` 本身已较大（如 2048）且层数为 24、隐藏维度 4096 时，显存占用可达数十 GB。  
3. **跨进程/跨节点 KV‑Cache 组不一致**  
   - `validate_same_kv_cache_group` 只检查层数相同，未对 KV‑Cache 布局（NHD vs HND）强制统一，若用户自行修改 `kv_cache_layout`，可能导致 data race 或错误的 slot mapping。  
4. **安全泄漏**  
   - `shared_storage_path` 默认 `/tmp`，在共享机器上可能被其他用户读取，导致潜在的模型激活泄露。  
5. **兼容性回退**  
   - 旧版本的 `SpeculativeConfig`（不含 `extract_hidden_states`）在升级后仍能正常工作，但如果用户自行修改 `draft_model_config` 而不提供 `eagle_aux_hidden_state_layer_ids`，会触发 `ValueError`，导致启动失败。  
6. **KVConnectorOutput.merge** 实现的集合合并逻辑在极端并发（多个 connector 同时返回）时可能产生未预期的 `invalid_block_ids` 合并错误。

|------|------|
