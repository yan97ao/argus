### [Kernel] Add gpt-oss Router GEMM kernel (#37205)
**SHA**: `b1169d7` | 🔗 [查看提交](https://github.com/vllm-project/vllm/commit/b1169d7be8add20ab1db4bc93c2b5c6336ef9754)

**🎯 变更类型**：功能增强 / 性能优化  
**⚡ 重要程度**：🔴高  

**📋 变更摘要**  
本次提交为 vLLM 项目新增针对 GPT‑Oss 模型的专用 Router GEMM CUDA kernel（`gpt_oss_router_gemm`），并在 CMake、Python 绑定、自动调度逻辑、单元测试、基准脚本以及 LoRA 相关的层注册中完成全链路集成。该 kernel 针对 Hopper/Blackwell（SM90+）GPU、批大小 ≤ 128、专家数 32/128、隐藏维度 2880 的常见配置做了高度流水线化实现，以显著提升路由阶段的算子吞吐与时延。

**🎯 影响范围**  
- `csrc/moe/`：新增 `gpt_oss_router_gemm.cu/.cuh`，修改 `moe_ops.h` 与 `torch_bindings.cpp`  
- `vllm/_custom_ops.py`：公开 Python 接口 `gpt_oss_router_gemm`  
- `vllm/model_executor/layers/fused_moe/router/gate_linear.py`：调度层加入新 kernel 判定与调用  
- `vllm/lora/layers/`：新增 `gate_linear.py` 并在 LoRA 注册表中加入  
- `tests/kernels/moe/test_router_gemm.py`、`benchmarks/kernels/benchmark_router_gemm.py`：对应单元测试与基准脚本  
- `CMakeLists.txt`：构建系统加入新 `.cu` 文件  

**🔍 技术洞察**  

- **架构影响**  
  - **模块化**：通过 `GateLinear` 层的 pluggable 机制把新 kernel 纳入统一调度路径，保持原有三层（DSV3 → cuBLAS → fallback）结构，只是将 `gpt_oss_router_gemm` 插入为第 2层，使整体架构向下兼容且易于后续扩展。  
  - **构建**：在 CMake 中显式添加 `.cu` 源文件，确保在 CUDA 编译链路中生成对应 PTX，未对其它平台产生副作用。  
  - **LoRA**：新增 `GateLinearWithLoRA`，保证在 LoRA 场景下仍可利用该专用 kernel，且实现 `can_replace_layer` 通过 `maybe_get_oot_by_class` 保持 OOT（out‑of‑tree）兼容性。  

- **性能影响**  
  - **算子实现**：基于 NVIDIA TensorRT‑LLM TinyGEMM2 代码改写，使用 `cuTensorMap` + `cp.async` + 双缓冲流水线（STAGES=16）实现 16‑stage 共享内存预取，极大降低全局内存访问延迟。  
  - **并行度**：每块 384 线程（12 warps），每个 warp 负责 16 × 8 输出子块，配合 `ldmatrix` 与 `mma.sync` 指令实现 BF16→BF16→FP32 的矩阵乘法，单块可并行计算 16 × 8 × K。  
  - **调度**：在 `GateLinear.forward` 中，仅当 `batch_size ≤ 128` 且满足维度匹配时走此 kernel，否则回退到 cuBLAS 或 `F.linear`，避免在不适用场景下出现性能倒退。  
  - **基准**：新增 `benchmark_router_gemm.py` 可对不同 batch、expert、hidden size 进行 TFLOPs 对比，方便快速评估改动收益。  

- **安全考虑**  
  - **类型检查**：在 C++ 接口层对输入/weight/bias 的维度、dtype 进行严格 `TORCH_CHECK`，防止因维度不匹配导致未定义行为。  
  - **异常处理**：`gpuAssert` 在 CUDA 错误码非 SUCCESS 时抛异常，确保 kernel 启动失败能够及时曝光。  
  - **平台限制**：所有调用均被 `current_platform.is_device_capability(90)` 或 `family(100)` 检测包裹，避免在不支持的 GPU（如 A100）上触发非法指令。  

**⚠️ 潜在风险**  
1. **兼容性风险**：kernel 依赖 SM90+ 的 Tensor Core 与 `cp.async`/`cuTensorMap`，在 SM89（如 A100）或更旧的卡上会直接报错或被跳过。若用户未显式检查平台，可能在运行时出现 `ValueError: Unsupported router gemm`。  
2. **共享内存上限**：`smem_size` 计算为 `STAGES * STAGE_UNROLL * (TILE_M*TILE_K + TILE_N*TILE_K)`，在某些 GPU 的每块共享内存上限（64 KB / 96 KB）可能被超出；虽然 `cudaFuncSetAttribute` 已设定，但在极端 batch/专家组合下仍需监控。  
3. **数值精度**：kernel 采用 BF16 输入、BF16 输出（内部 FP32 accumulate），与 cuBLAS BF16→FP32 的累加方式略有差异，可能出现微小数值偏差。测试已设定 `atol=1e-2`，但在极端量化模型上需额外验证。  
4. **多线程竞争**：使用 `mbarrier` 与 `cudaLaunchAttributeProgrammaticStreamSerialization` 的同步模型对 CUDA driver 版本敏感，老版本 driver 可能不支持 `programmaticStreamSerializationAllowed = 1`，导致潜在的 deadlock。  

**💡 关注建议**  

- **平台检测**：在高层 API（如模型加载）加入明确的 GPU 能力检查，若不满足 SM90+ 自动回退到 DSV3 或 cuBLAS，避免抛异常。  
- **共享内存监控**：在 CI 中加入针对不同 `STAGES`、`TILE_*` 参数的 `cudaFuncGetAttributes` 检查，确保不超过目标 GPU 的 `maxDynamicSharedMemorySize`。  
- **数值回归**：在模型量化路径（如 INT8/FP8）加入对比基准，确保 BF16→BF16 路径的累计误差在业务可接受范围。  
- **文档与示例**：更新 README 与 `benchmark_router_gemm.py` 的使用说明，明确 “仅在 Hopper/Blackwell 上可得最大性能提升”。  
- **回滚路径**：保持 `torch.ops._moe_C.gpt_oss_router_gemm` 的注册，即使 kernel 编译失败（如用户自行编译），仍应提供 fallback 的 Python 包装层，以保证库的可用性。  

通过上述措施，可在保证安全与兼容性的前提下，充分释放 Hopper/Blackwell GPU 的性能潜力，提升 GPT‑Oss 与其它小 expert‑数模型的路由阶段吞吐。

### [2/3] Refactor InternVL-based processors (#37324)
**SHA**: `99267c2` | 🔗 [查看提交](https://github.com/vllm-project/vllm/commit/99267c23ca51ef9b4486fecaf7d9ec25475f9894)

**🎯 变更类型**：重构 / 架构变更  
**⚡ 重要程度**：🔴 高  

**📋 变更摘要**  
1. 将所有基于 InternVL 系列模型的图像/视频处理器统一抽象为 `InternVLImageProcessor`、`InternVLVideoProcessor` 与 `InternVLProcessor`，剥离了原有的 `*_Processor`（如 `Eagle2_5_VLProcessor`、`SkyworkR1VProcessor` 等），并在 `__init__` 中通过 `ProcessingInfo` 动态构造。  
2. 引入 `cached_property`、统一的 token 常量（`<img>` / `</img>` / `<IMG_CONTEXT>` 等）以及对 **视频 token** 的可选支持，简化了多模态占位符的生成与替换逻辑。  
3. 更新了所有受影响模型（eagle2_5_vl、glm4v、h2ovl、internvl、nemotron_vl、nvlm_d、qwen_vl、skyworkr1v、nemotron_vl‑embed、nano_nemotron_vl 等）的 `ProcessingInfo`，统一使用新的 Image/Video Processor API。  
4. 移除旧的 processor 实现文件（`eagle2_5_vl.py`、`skyworkr1v.py`），并在 `processors/__init__.py` 中剔除对应导出。  
5. 测试层面同步更新了对占位符 token 的硬编码（`IMG_START`、`IMG_END`、`IMG_CONTEXT`），确保新实现能够通过现有 multimodal 测试用例。

### 🔍 技术洞察

| 维度 | 影响分析 |
|------|----------|
| **架构影响** | - **统一抽象层**：所有 InternVL‑系模型的图像/视频前处理集中在 `transformers_utils/processors/internvl.py`，极大降低了代码重复度。<br>- **ProcessingInfo 重构**：每个模型仅需要实现 `get_image_processor`（可选 `get_video_processor`）和 `get_hf_processor`，其它细节统一由基类 `InternVLProcessor` 完成。<br>- **Token ID 改名**：原来使用 `image_token_id` / `video_token_id` 改为 `ctx_image_token_id` / `ctx_video_token_id`，并通过 `cached_property` 懒加载，提升首次访问的性能。 |
| **性能影响** | - **正向**：去除旧的 `*_Processor` 复制逻辑，减少了模块导入次数、对象创建以及重复的 `resolve_min_max_num` 调用。<br>- **负向**：`InternVLProcessor.__call__` 中对文本进行 **多轮占位符替换**（先遍历图片，再遍历视频）实现为 Python 循环，可能在极端高并发、长文本（> 10 k tokens）场景下产生微小的 CPU 开销。总体预计仍优于旧实现的多层继承与重复转化。 |
| **安全考虑** | - **输入校验**：新实现仍然直接把 `images`/`videos` 交给 `PIL` / `torchvision` 进行转换，未加入额外的文件大小或 MIME 检查，安全风险与之前相同。<br>- **Token 注入**：占位符替换使用固定字符串 `<placeholder>` 进行中间占位，避免了原来 `replace("<image>", ...)` 直接多次替换导致的意外递归替换风险。<br>- **并发安全**：`cached_property` 在 CPython 中是线程安全的（首次计算锁），对多线程推理不会产生竞争。 |
| **可维护性** | - **代码量大幅下降**：`internvl.py` 由 ~500 行压缩至 ~250 行，删除 2 份冗余 processor 文件（共约 470 行）。<br>- **统一接口**：所有模型只需关心 `image_processor`、`video_processor` 参数，降低新模型接入门槛。<br>- **文档/测试同步**：更新了 `tests/models/multimodal/generation/vlm_utils/model_utils.py` 以及 `registry.py`，确保新常量被正确导入。 |

### [Perf] Optimize top-k search in apply_top_k_top_p_triton sampler (#37225)
**SHA**: `51b2333` | 🔗 [查看提交](https://github.com/vllm-project/vllm/commit/51b2333be19000db7d03b76ccf1b842972c98541)

- 代码路径分别在 **buffer‑search** 与 **全 vocab 搜索** 两端统一，删除了冗余 `min`/`sum` 循环，减小了内存访问次数和 kernel 启动开销。

总体来看，此次改动通过一次遍历显著降低了 Triton kernel 的内存访问次数，预计在大 vocab 场景下可提升 5‑15% 的采样速度。但需关注数值精度与跨平台兼容性，建议在 CI 中加入更多的随机种子与多卡验证。
