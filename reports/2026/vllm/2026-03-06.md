### [Core][KVConnector] Support HMA+NixlConnector (#35758)
**SHA**: `5b3ba94` | 🔗 [查看提交](https://github.com/vllm-project/vllm/commit/5b3ba94ab4bd9da739bcc27cdd05505467fa499e)

**🎯 变更类型**：功能增强 / 架构变更 / 性能优化  
**⚡ 重要程度**：🔴高  

**📋 变更摘要**  
- 为 KV‑Connector 引入对 **Hybrid KV‑Cache Manager (HMA)** 的支持，新增 `SupportsHMA` 接口并在 `NixlConnector`、`NixlConnectorScheduler`、`NixlConnectorWorker` 中加入 HMA 相关的逻辑。  
- 扩展内部块 ID 表达方式为 **BlockIds**（支持多组块），并实现块映射、滑动窗口裁剪、逻辑‑>‑物理块转换等工具函数。  
- 更新大量单元测试与集成脚本，使其能够在 HMA 开/关两种模式下运行，并加入对新模型（如 `google/gemma-3-4b-it`）的验证。  
- 兼容性：在计算 NIXL 兼容性哈希时加入 `is_hma_enabled` 标识；对 Mamba 模型给出明确不支持的报错。  

**🎯 影响范围**  
- `vllm/distributed/kv_transfer/kv_connector/v1/nixl_connector.py`（核心实现）  
- `vllm/distributed/kv_transfer/kv_connector/utils.py`（BlockIds 类型）  
- `vllm/v1/core/kv_cache_manager.py`（新增 `get_unhashed_block_ids_all_groups`）  
- 新增/修改的单元测试、集成脚本位于 `tests/v1/kv_connector/` 目录。  
- 相关的配置、环境变量 (`ENABLE_HMA_FLAG`、`VLLM_SERVE_EXTRA_ARGS`) 也被同步更新。  

**🔍 技术洞察**  

| 维度 | 影响描述 |
|------|----------|
| **架构影响** | - 引入 **Hybrid KV Cache Manager** 需要在调度器、工作节点、元数据层之间传递多组块 ID (FA + SW 等)。<br>- `SupportsHMA` 接口把 HMA 相关能力抽象出来，保持与原有单组 KV‑Cache 代码的兼容。<br>- `NixlConnectorScheduler` 现在会根据 KV‑Cache 配置计算每个组的 **Sliding Window 大小（blocks_per_sw）**，并在 `request_finished`、`build_connector_meta` 等关键路径对块进行裁剪。<br>- `NixlConnectorWorker` 按组循环执行 `copy_blocks`，避免在一次拷贝中跨组混用，提升可维护性。 |
| **性能影响** | - **正向**：开启 HMA 后，`FullAttention` 与 `SlidingWindow` 两类缓存可并行使用，显著降低长序列（尤其超过滑动窗口）时的远程传输带宽与内存占用。<br>- **负向**：引入跨组块映射与裁剪逻辑会在调度路径多一次 Python 循环与 `cdiv` 计算，极端情况下（大量 KV‑Cache 组）可能产生微小的 CPU 开销。<br>- 通过 `BlockIds` 结构的不可变/可变属性，可在内部做就地扩容，减少列表拷贝。 |
| **安全考虑** | - 新增的环境变量 `ENABLE_HMA_FLAG` 与 `VLLM_SERVE_EXTRA_ARGS` 均为 **可选**，不会直接影响外部攻击面。<br>- 代码在计算兼容性哈希时加入 `is_hma_enabled`，防止在 HMA 状态不匹配时误连通导致数据泄漏或未授权访问。<br>- 对不支持的模型（如 Mamba）抛出明确异常，避免在运行时触发未定义行为。 |
| **可维护性** | - 引入 `BlockIds = tuple[list[int], ...] | list[list[int]]` 统一了块 ID 表达，避免散落的 `list[int]` 兼容代码。<br>- 将 HMA 相关的判断抽象为 `_is_hma_required`，集中在调度器与 worker 中，便于后续特性扩展（如支持更多 KV‑Cache spec）。<br>- 大量单元测试覆盖 HMA 开/关、不同模型、不同滑动窗口大小，提升回归安全性。 |

**⚠️ 潜在风险**  
1. **API 兼容性破坏**：`request_finished` 参数从 `list[int]` 改为 `tuple[list[int], ...]`，若外部插件或自定义 KV‑Connector 仍使用旧签名会导致运行时 `TypeError`。  
2. **块映射错误**：`_logical_to_kernel_block_ids` 现在返回多组列表，若后续代码仍假设单列表，可能出现索引越界或错误的物理块映射。  
3. **滑动窗口裁剪**：`get_sw_clipped_blocks` 在 HMA 开启时强制裁剪块数量，若模型的 `sliding_window` 配置不一致（比如自定义 `SlidingWindowSpec` 与实际序列长度不匹配），可能导致 **请求提前结束** 或 **额外的远程传输**。  
4. **并发资源泄漏**：在 HMA 场景下 `self._invalid_block_ids.update(meta.local_block_ids[0])` 只处理第一组块，若出现跨组错误（例如某组块失效而另一组正常），可能留下未释放的块，导致内存泄漏。  
5. **测试覆盖盲区**：当前测试主要在 CPU 环境下跑，未覆盖 GPU/UCX 异构 TP 场景，可能隐藏在高性能部署下的 race condition。  

**💡 关注建议**  
- **文档升级**：在项目 README 与 `vllm.config` 中明确标注 `disable_hybrid_kv_cache_manager`、`ENABLE_HMA_FLAG` 的默认行为、支持的 KV‑Cache spec（FullAttention + SlidingWindow），以及不支持的 `MambaSpec`。  
- **向后兼容桥接**：为旧插件提供适配层，例如在 `NixlConnectorBase` 中保留 `request_finished`（单组）包装实现，内部调用新 `request_finished_all_groups`，以免突发性破坏。  
- **健壮性检查**：在 `NixlConnectorWorker._read_blocks` 入口加入断言，确保 `len(local_block_ids) == len(remote_block_ids) == len(kv_cache_groups)`，并在异常路径提供清晰的错误日志。  
- **资源回收**：在 `NixlConnectorWorker._handle_failed_transfer` 中若 `_is_hma_required` 为 `True`，遍历所有组的 `local_block_ids` 进行失效块回收，防止潜在泄漏。  
- **扩展测试**：补充 GPU+UCX、混合 TP（prefill‑TP > decode‑TP）以及多机环境下的 HMA 场景，确保跨进程、跨机器的块映射与通知逻辑在高并发下保持正确。  
- **性能基准**：提供一套对比基准（开启/关闭 HMA、不同 `sw_size`、不同模型），帮助用户评估在实际工作负载下的内存占用与网络吞吐提升幅度。  

---  

*整体来看，此次提交为 vLLM 引入了 **Hybrid KV‑Cache Manager** 的完整支撑，极大提升了长序列推理时的内存利用率与跨节点传输效率。但需注意 API 兼容、跨组错误处理及在异构硬件上的更严格测试，以确保在生产环境中平稳落地。*

### [Frontend][Core] Add shutdown timeout - allowing in-flight requests to finish (#34730)
**SHA**: `27066d1` | 🔗 [查看提交](https://github.com/vllm-project/vllm/commit/27066d1b2bd0dea89d617afa24da611d9a32e36a)

**🎯 变更类型**：功能增强（新增 Shutdown Timeout 机制）  
**⚡ 重要程度**：🔴高  

**📋 变更摘要**  
1. 在 `VllmConfig` 中加入 `shutdown_timeout` 参数，支持在收到 SIGTERM 时指定一个宽限期，让正在执行的请求有机会完成，超时后强制终止未完成请求。  
2. 为引擎、进程管理器、协调进程、API 服务器等核心组件统一实现 `shutdown(timeout: float|None)` 接口，取代原来的 `close()`，并在 `serve` 与 `launcher` 的信号处理逻辑中使用该接口。  
3. 引入 `EngineShutdownState` 状态机、`SignalCallback` 线程安全信号触发、`WAKEUP` 请求等，实现了在信号到达时能够安全唤醒阻塞的 `input_queue`，完成请求驱赶或等待。  
4. 大量新增与修改集成测试，验证 **等待（wait）超时**、**立即中止（abort）**、**多 API 进程**、**子进程清理** 等场景的正确性。

---

- **引擎核心**：`vllm/v1/engine/core.py`、`vllm/v1/engine/core_client.py`、`vllm/v1/engine/protocol.py`、`vllm/v1/engine/async_llm.py`、`vllm/v1/engine/coordinator.py`、`vllm/v1/engine/launch.py`、`vllm/v1/engine/utils.py`

- **入口脚本**：`vllm/entrypoints/cli/serve.py`、`vllm/entrypoints/launcher.py`  
- **测试**：`tests/entrypoints/openai/test_shutdown.py`、`tests/entrypoints/test_api_server_process_manager.py`

### 🔍 技术洞察

| 维度 | 影响分析 |
|------|----------|
| **架构影响** | - **状态机加入**：`EngineShutdownState` 将引擎从 “运行中” → “已请求关闭” → “关闭中”。<br>- **统一 shutdown 接口**：所有管理对象（EngineManager、Coordinator、APIServerProcessManager、ProcessGroup）实现 `shutdown(timeout)`，统一资源回收路径，避免散落的 `close()` 调用导致遗漏。<br>- **信号安全**：新增 `SignalCallback` 通过后台线程触发回调，避免在信号处理器中直接操作非可重入对象（如 `queue.mutex`），提升可靠性。<br>- **WAKEUP 请求**：在收到 SIGTERM 时向 `input_queue` 注入特殊请求，确保即使引擎处于空闲等待也能被唤醒，顺利进入关闭流程。 |
| **性能影响** | - **等待超时**：当 `shutdown_timeout > 0` 时，主进程会阻塞最多 `timeout` 秒以让在-flight 请求完成；这会在正常关闭时产生可控延迟（可配置），但避免了因强制终止导致的请求重试。<br>- **abort（timeout=0）**：保持原有快速退出行为，无额外开销。<br>- **子进程清理**：`shutdown` 使用统一 `shutdown(procs, timeout)`，在默认 `timeout`≥5 s 的基础上保证子进程有足够时间退出，防止僵尸进程累积。 |
| **安全考虑** | - **信号处理安全**：通过 `SignalCallback` 将信号上下文与实际业务逻辑解耦，防止在信号处理器中执行不安全的 Python 代码，降低崩溃或资源泄漏的风险。<br>- **不暴露新外部接口**：`shutdown_timeout` 仅是配置项，未改变网络协议或鉴权逻辑，安全面影响极小。 |
| **可维护性** | - **统一 API**：`shutdown` 成为所有资源的标准销毁方法，未来可以在同一位置加入统一日志、监控或资源审计。<br>- **向后兼容**：仍保留旧的 `close()` 调用点（已改为 `shutdown`），但未提供别名，外部直接调用 `close()` 将导致 `AttributeError`。建议添加兼容层或文档提示。<br>- **代码复杂度**：引入状态机和额外信号回调，核心 `EngineCoreProc` 增加约 200 行逻辑，需要细致的单元/集成测试来防止回归。 |
| **测试覆盖** | - 新增 **OpenAI API** 级别的集成测试，覆盖：<br>  - 正常 shutdown（timeout>0）时在飞请求完成并返回 503/500。<br>  - abort（timeout=0）立即终止请求。<br>  - 多 API 进程下的信号传播。<br>  - 子进程清理验证。<br>  - `APIServerProcessManager`、`EngineManager`、`Coordinator` 的 `shutdown` 路径。 |
