### [1/2] Move InternVL-based processors (#37260)
**SHA**: `f340324` | 🔗 [查看提交](https://github.com/vllm-project/vllm/commit/f34032433573cda9bc495cf02e783c8b0d99d20d)

**🎯 变更类型**：功能增强 / 架构重构  
**⚡ 重要程度**：🔴高  

**📋 变更摘要**  
- 将所有 **InternVL 系列**（InternVL、H2OVL、Eagle2.5‑VL、Nano‑Nemotron‑VL、Nemotron‑Parse、Nemotron‑VL、NVLM‑D、SkyworkR1V）在模型层的自定义 **HF Processor** 迁移到 `vllm/transformers_utils/processors/` 目录下，形成统一、可复用的处理器模块。  
- 在模型实现 (`vllm/model_executor/models/*.py`) 中删除原有的冗余实现，改为从新模块导入对应 `*Processor`。  
- 更新 `vllm/transformers_utils/processors/__init__.py` 注册新类，使外部 `cached_image_processor_from_config`、`init_processor` 等统一入口能够发现它们。  
- 为每个模型新增专属处理器文件（`eagle2_5_vl.py、h2ovl.py、internvl.py、nano_nemotron_vl.py、nemotron_parse.py、nemotron_vl.py、nvlm_d.py、skyworkr1v.py`），保留原有图像/视频/音频预处理逻辑，并对共有函数（如 `build_transform`、`find_closest_aspect_ratio`、`calculate_*_targets` 等）做了抽象复用。  
- 对若干测试文件的 import 路径做相应修改。  

**🎯 影响范围**  
- 核心模型文件：`eagle2_5_vl.py、h2ovl.py、internvl.py、nano_nemotron_vl.py、nemotron_parse.py、nemotron_vl.py、nvlm_d.py、skyworkr1v.py`（约 7k 行代码）。  
- 处理器注册：`vllm/transformers_utils/processors/__init__.py`。  
- 相关测试：`tests/models/multimodal/processing/*`。  
- 依赖库：新增对 `albumentations`, `einops`, `regex` 的显式导入；原有 `PIL.Image.MAX_IMAGE_PIXELS` 被全局关闭。

## 🔍 技术洞察

| 维度 | 影响描述 |
|------|----------|
| **架构** | - **职责分离**：将所有多模态前处理（图像/视频/音频）从模型执行层抽离到 `transformers_utils.processors`，形成 **Processor‑Library**，模型层仅负责调度。<br>- **复用与统一**：共用 `BaseInternVLProcessor`、`BaseNanoNemotronVLProcessor` 等抽象基类，避免代码复制，提升可维护性。<br>- **模块化加载**：`cached_image_processor_from_config` 通过 `__all__` 与 `_CLASS_TO_MODULE` 动态导入，降低模型包的导入开销。 |
| **性能** | - **运行时开销**：处理器本身的计算（图像 resize、分块、归一化）保持不变；仅因 **import 路径变短**、 **移除冗余实现**，可能略微提升 Python 模块加载速度和内存占用（约几 MB）。<br>- **并行/批处理**：`DynamicResolutionImageTiler` 等仍保留原有批量处理逻辑，未影响已有的高效 `torch.cat`、`einops.rearrange` 实现。 |
| **安全** | - **大图安全**：在 `nemotron_vl.py` 与 `skyworkr1v.py` 中执行 `Image.MAX_IMAGE_PIXELS = None`（关闭 Pillow 的 DecompressionBomb 警告）。这使得 **恶意超大图片** 可能导致 **DoS（内存/磁盘消耗）**，如果上层未对输入尺寸做额外检查。<br>- **依赖安全**：引入 `albumentations`, `einops`, `regex`，需要在发布镜像中确保这些库的安全版本（无已知高危 CVE）。 |
| **可维护性** | - **代码结构**：所有模型共享的函数（`build_transform`, `find_closest_aspect_ratio`, `calculate_*_targets`, `dynamic_preprocess_*`）统一放在对应处理器文件，避免了在每个模型文件中重复定义，后续 **修复 bug / 调整预处理** 只需改动一处。<br>- **文档/测试**：测试只需修改 import 路径，原有行为保持不变；新增的处理器文件均自带完整实现，便于单元测试覆盖。 |
| **兼容性** | - 对外 API **保持**：每个新处理器依旧实现 `__call__(self, text, images, …) -> BatchFeature` 与原模型层的 Processor 接口兼容。<br>- **Tokenizer 依赖**：`image_token_id`、`video_token_id` 等属性仍通过 `tokenizer.get_vocab()` 访问；若模型的 tokenizer 配置变化（例如 token 名称改动），仍会抛出 `ValueError`，行为与旧实现一致。 |
| **风险点** | 1. **全局关闭 Pillow 限制** 可能被攻击者利用制造极大图像导致 OOM。<br>2. **新依赖缺失**：`albumentations` / `einops` / `regex` 若未在部署环境安装，将导致模型启动失败。<br>3. **动态导入映射错误**：`_CLASS_TO_MODULE` 中的类名若拼写错误或文件路径变动，会在运行时抛出 `ImportError`，影响模型注册。<br>4. **缓存特性**：`DynamicResolutionImageTiler.feature_size_cache` 依赖 `id(image)`，在并发环境下若同一 Image 对象被多线程同时使用，可能出现 **KeyError**（缓存已被删）。 |

## ⚠️ 潜在风险

| 风险 | 可能导致 | 影响范围 | 缓解建议 |
|------|----------|----------|----------|
| **DoS via 超大图片** (`Image.MAX_IMAGE_PIXELS = None`) | 进程占用巨量内存/CPU 进行 resize/分块 | 所有使用 Pillow 的处理器（InternVL 系列、Nemotron‑VL、SkyworkR1V） | - 在 HF Processor 前加入尺寸上限检查（如 `if image.width * image.height > 1e8: raise ValueError`）<br>- 或在部署层限制上传文件大小。 |
| **缺失第三方库** (`albumentations`, `einops`, `regex`) | `ImportError` 导致模型加载失败 | 仅在使用 Nemotron‑Parse、Nano‑Nemotron‑VL 等模块时 | - 在 `setup.cfg` / `requirements.txt` 中明确列出这些依赖并 pin 版本。 |
| **映射/类名不一致** | `cached_image_processor_from_config` 找不到对应 Processor | 所有模型在 `init_processor` 时 | - 在 CI 中加入自动检查，确保 `_CLASS_TO_MODULE` 包含所有 `*Processor`。 |
| **并发缓存冲突** (`DynamicResolutionImageTiler.feature_size_cache`) | `KeyError` 或错误的特征尺寸 | 使用动态分辨率的 Nano‑Nemotron‑VL 场景（多图并发） | - 将缓存改为局部变量或使用 `WeakKeyDictionary`，或在并发前 lock。 |
| **Tokenizer token 不存在** (`image_token_id` 等) | `KeyError` / `ValueError` | 所有处理器在首次调用时 | - 使用 `dict.get(..., default_id)` 并在模型配置里提供后备 ID。 |

### [Feature][Frontend] add support for Cohere Embed v2 API (#37074)
**SHA**: `061980c` | 🔗 [查看提交](https://github.com/vllm-project/vllm/commit/061980c36a7b78e5d8ea96893b79fd0b9c11a20e)

**🎯 变更类型**：功能增强（为 vLLM 添加 Cohere Embed v2 API 兼容层）  

**⚡ 重要程度**：🔴 高  

**📋 变更摘要**  
1. 在 OpenAI‑兼容服务器中新增 `POST /v2/embed` 路由，实现 Cohere v2 Embed API（支持 `texts`、`images`、`inputs`、`input_type`、`truncate`、`output_dimension`、多种 `embedding_types` 等）  
2. 扩展协议层，新增 `CohereEmbedRequest/Response`、类型转换工具（二进制、无符号二进制、base64）以及任务指令加载逻辑，使模型能够根据 `input_type` 自动前缀化 Prompt。  
3. 在 `EmbedIOProcessor` 中实现 Cohere‑专用的前置/后置处理、左/右截断语义、Chunked 长文本分块，以及对多模态（base64 图片）输入的统一渲染。  
4. 通过 `TokenizeParams.truncation_side` 将左侧截断语义注入到底层 Tokenizer，确保在 `truncate=START` 时保留输入尾部。  
5. 完备单元与集成测试，覆盖文本、图片、混合输入、嵌入类型、截断行为、OpenAI‑Cohere 嵌入等 300+ 用例。  

**🎯 影响范围**  
- `vllm/entrypoints/pooling/embed/*`（协议、IOProcessor、Serving、API Router）  
- `vllm/renderers/params.py`（新增 `truncation_side` 参数）  
- `vllm/entrypoints/pooling/base/protocol.py`、`classify/protocol.py`、`score/protocol.py`（同步新增 `truncation_side`）  
- `vllm/entrypoints/pooling/typing.py`（Request Union 增加 `CohereEmbedRequest`）  
- 文档 `docs/serving/openai_compatible_server.md`  
- 新增 4 套测试文件（约 1 700 行）  
- 相关依赖（`openai` 类型、`ImageURL`、`ChatCompletionContentPart*`）

## ⚠️ 潜在风险  

1. **异构 TP 兼容性**  
   - `register_cross_layers_kv_caches` 目前假设 **跨层 KV‑Cache** 只能在 **同构 TP** 场景下使用（`prefer_cross_layer_blocks` 在 Mamba 组返回 `False`）。若用户在混合 TP 环境下开启 HMA，可能触发 **RuntimeError** 或不一致的块映射。  

2. **块大小比例 (`_physical_blocks_per_logical_kv_block`)**  
   - 当 ratio > 1（kernel block 与模型逻辑块不同）时，所有相关计算（如 `block_len_per_layer`、`_get_block_descs_ids`）依赖该比例的正确性。若后端实现未遵循相同的比例或出现 float 除法，可能导致 **误配**（如页大小不整除），进而导致 **段错误** 或 **数据错位**。  

3. **Mamba 与 FA 同时注册的内存 region**  
   - 对于 **Hybrid SSM‑FA**，`register_kv_caches` 对同一底层 tensor 会在 `self.num_regions` 上乘 2，再额外为 Mamba 再次复制一次 region。若底层 tensor 实际只提供单一物理页面（如 FP8），可能出现 **duplicate registration** 报错（NIXL 不允许运行时重复注册相同地址）。  

4. **跨进程元数据同步**  
   - Handshake payload 现在携带 `ssm_sizes`，但 `NixlHandshakePayload` 在解码时对该字段的校验仍然较宽松（只检查是否存在），若两端 `ssm_sizes` 不匹配，会导致 **tensor shape 不一致** 错误，且错误信息不易定位。  

5. **测试与实际部署差异**  
   - CI 中的 `test_get_block_descs_ids_*` 用的是 **CPU** 环境的 `NixlConnectorWorker`，而真实部署将走 **UCX**/GPU 实现。GPU 驱动或 UCX 版本差异可能导致 **硬件层面的对齐/stride** 行为不一致。

5. **测试覆盖**：现有测试已覆盖基本、无参数、复杂、混合内容以及跨步 JSON 场景。建议再补充：  
   - 非法 JSON（缺失右括号、额外字符） → 应回退为普通文本。  
   - 多 tool‑call 连续出现的情况。  
   - `message_sep` 与 `role_sep` 组合出现顺序异常的容错。  

整体来看，新增的 Gigachat 3.1 兼容层与流式解析的修复提升了 `gigachat3` 系列模型的可用性，但正则的贪婪匹配和内部临时字段的持久化需要进一步梳理，以防在生产环境出现隐藏的解析错误或数据结构冲突。
