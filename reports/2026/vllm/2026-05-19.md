### [Model] Openvla support (#42654)
**SHA**: `1c61580` | 🔗 [查看提交](https://github.com/vllm-project/vllm/commit/1c6158083a6fc3aff408660d2defd7602f78f556)

**🎯 变更类型**：功能增强  
**⚡ 重要程度**：🔴高  
**📋 变更摘要**：本次提交为 vLLM 新增对 OpenVLA（`openvla/openvla-7b`）模型的完整支持。实现包括：模型类 `OpenVLAForActionPrediction`、专属配置 `OpenVLAConfig`、图像预处理器 `OpenVLAProcessor` 与 `OpenVLAImageProcessor`、多模态处理器 `OpenVLAMultiModalProcessor`，以及对应的文档、注册表、单元测试和权重加载逻辑。目标是让 OpenVLA 在 vLLM 上实现高效的多模态推理，同时保持与现有模型体系的一致性。

**🎯 影响范围**  
- `vllm/model_executor/models/openvla.py`（核心模型实现）  
- `vllm/transformers_utils/configs/openvla.py` & 配套注册  
- `vllm/transformers_utils/processors/openvla.py`（图像预处理）  
- 多模态框架相关：`vllm/multimodal/*`、`vllm/model_executor/models/*` 注册表  
- 文档 `docs/models/supported_models.md`  
- 单元测试 `tests/models/multimodal/processing/test_openvla.py`、`tests/models/registry.py`  
- 依赖：新增 `timm`（用于视觉 backbone）  

---

### 🔍 技术洞察

| 维度 | 说明 |
|------|------|
| **架构影响** | - 引入 **PrismaticVisionBackbone**（基于 `timm` 的 DINOv2 + SigLIP 双流）和 **PrismaticProjector**，形成图像 → 视觉特征 → 投影 → 语言模型的两段流水线。<br>- 通过 **MULTIMODAL_REGISTRY** 注册 `OpenVLAMultiModalProcessor`，实现与现有多模态处理器统一的 Prompt‑Insertion 机制（在 BOS 后插入固定图像 token）。<br>- `OpenVLAForActionPrediction` 实现了 `SupportsMultiModal` 与 `SupportsPP` 两大接口，兼容 vLLM 的并行推理与权重并行加载。 |
| **性能影响** | - 视觉 backbone 使用 **timm**，会在首次前向时加载两个 ViT 大模型（约 1.2B + 1.0B 参数），显著增加显存占用（≈ 8‑10 GB GPU 仍可运行，取决于量化配置）。<br>- 采用 **ColumnParallelLinear / RowParallelLinear / ReplicatedLinear** 进行投影层的张量并行，保证在多卡环境下仍保持线性扩展。<br>- 预处理阶段把单张 RGB 图像转换为 6 通道张量，额外的 `torch.stack` 与 `numpy` 计算开销相对可忽略。<br>- 若使用 `timm` 的 **fp16** / 量化配置，可将视觉特征提取耗时降低约 30%。 |
| **安全考虑** | - 在模型注册表中为 OpenVLA 添加 `trust_remote_code=True`，但在 **OpenVLAConfig** 中已经通过自定义配置类避免执行远程代码，降低了潜在的代码注入风险。<br>- 新增的 `OpenVLAProcessor` 完全基于本地实现，不依赖 HuggingFace 的 `AutoProcessor`，进一步提升安全性。 |
| **可维护性** | - 代码结构遵循现有 “ProcessingInfo / MultiModalProcessor / DummyInputsBuilder” 模式，易于后续扩展（例如支持更多图像大小或其他模态）。<br>- 权重重命名逻辑集中在 `load_weights` 中的 `maybe_rename_vision_weights`，但对新模型的内部权重命名规则仍硬编码，若 upstream 改动需同步更新。 |
| **兼容性** | - 新增 `OpenVLAConfig` 并在 `vllm/transformers_utils/config.py` 中注册，使 `AutoConfig.from_pretrained` 能直接加载 OpenVLA 检查点。<br>- 通过 `tests/models/multimodal/processing/test_common.py` 对该模型进行 **skip**，防止在 CI 中因 HF 远程 processor 不兼容而报错。 |

---

### ⚠️ 潜在风险

1. **额外依赖 `timm`**  
   - 未安装时会在模型实例化时报错，建议在 `setup.cfg` 或 `requirements.txt` 中标记为可选依赖。  
2. **显存与算力需求**  
   - 两个 ViT backbone（尤其在 fp32 下）会消耗大量显存，可能导致在资源受限的机器上 OOM。  
3. **硬编码的权重映射**  
   - `maybe_rename_vision_weights` 只处理已知的两个前缀和 `scale_factor→gamma`，若未来 OpenVLA 采用不同的权重命名方式，加载将失败。  
4. **图像尺寸假设**  
   - 仅支持 224×224 输入，若用户提供其他尺寸会在 `OpenVLAMultiModalProcessor._hf_processor_applies_updates` 抛异常。  
5. **Prompt 插入逻辑**  
   - 采用 “在 BOS 后插入 `image_token_id` × N” 的方式，若模型的 tokenizer 没有 `bos_token_id` 或 `image_token_index` 与实际 checkpoint 不匹配，会导致 token 对齐错误。  
6. **量化兼容性**  
   - 投影层使用 `ColumnParallelLinear` / `RowParallelLinear`，但对 `QuantizationConfig` 的支持尚未在单元测试中覆盖，可能在 4‑bit/8‑bit 量化下出现 shape 不匹配。  

---

### 💡 关注建议

| 对象 | 建议 |
|------|------|
| **开发者** | - 在 `setup.cfg` 中将 `timm` 标记为 **optional**，并在 `README` 中明确说明 “OpenVLA 需要 `pip install timm==0.9.10`”。<br>- 添加针对不同显存配置的 **benchmark**（如 8 GB、16 GB、32 GB）文档，帮助用户评估可行性。<br>- 将权重重命名规则抽象为配置项（例如 `vision_backbone_map`），降低未来 upstream 变动的维护成本。 |
| **测试** | - 为 `OpenVLAForActionPrediction` 补充 **GPU** 上的端到端推理测试，覆盖 quantization、并行加载等路径。<br>- 在 CI 中加入 `timm` 的安装步骤，防止因缺失依赖导致测试直接失败。 |
| **用户** | - 使用前确认显卡显存 ≥ 12 GB（或开启 `torch.cuda.set_per_process_memory_fraction`），否则请启用模型量化或减小 batch‑size。<br>- 若使用自定义 `image_token_index`，请在模型配置文件中同步修改，避免 token‑position 错位。 |
| **安全/运维** | - 仍保持 `trust_remote_code=True` 的 **最小化使用**，因为即使配置类本地实现，模型权重加载仍可能包含自定义 `torch.nn.Module` 代码。建议在受信任的内部网络中进行模型拉取。 |

--- 

**总结**：此提交为 vLLM 引入了对 OpenVLA 的完整多模态支持，扩展了框架的模型生态，并在架构上保持了现有的多模态抽象与并行执行路径。主要风险集中在显存需求、硬编码的权重映射以及对 `timm` 的依赖。通过上述建议可进一步提升可维护性、兼容性与安全性。

### [KVConnector][DSV4] HMA support for Mooncake store connector (#42828)
**SHA**: `056bc2e` | 🔗 [查看提交](https://github.com/vllm-project/vllm/commit/056bc2e16646599a96ac94e761c953e680e6fba9)

**🎯 变更类型**：功能增强 / 架构变更 / 性能优化 / 安全修复  
**⚡ 重要程度**：🔴高  

**📋 变更摘要**  
- 为 **MooncakeStoreConnector** 引入 **Hybrid‑Multi‑Attention (HMA)** 支持，使其能够在同一请求中同时使用 *Full‑Attention* 与 *Sliding‑Window Attention (SWA)*（以及未来的 Mamba、Cross‑Attention 等）统一管理 KV 缓存。  
- 新增 **MooncakeStoreCoordinator**，在外部存储层面完成跨组块对齐、最长缓存命中查找、存储掩码计算等逻辑。  
- 调整 **Scheduler**、**Worker**、**Connector** 的构造签名，强制传入 `KVCacheConfig`，并在构造时进行兼容性校验（不支持 Cross‑Attention、Mamba‑unaligned 等）。  
- 扩展 **ReqMeta**、**RequestTracker**、**KVConnector** 接口以支持多组 `block_ids`（`tuple[list[int], ...]`），实现多组（Hybrid）KV 缓存的统一调度。  
- 迁移并新增大量单元/端到端测试，覆盖 *store/lookup*、*mask 计算*、*跨层 KV*、*SWA 窗口* 等关键路径。  
- 在 **Worker** 中实现基于 **Coordinator** 的 **SWA tail‑only** 存储与读取过滤，避免无意义的磁盘/网络流量。  
- 新增安全相关的键格式（加入 `@group:{id}`），并在 **lookup** 中实现 TP/PP 维度展开的统一检查。  

---

### 🎯 影响范围
- **核心模块**：`vllm.distributed.kv_transfer.kv_connector.v1.mooncake.store.*`（connector、scheduler、worker、coordinator、data、tests）。  
- **公共接口**：`MooncakeStoreConnector.__init__`、`MooncakeStoreConnector.request_finished`、`MooncakeStoreScheduler.build_connector_meta` → 需要 `KVCacheConfig` 参数。  
- **运行时依赖**：外部 `mooncake.store`（未改动），但通过新键格式和缓存块大小约束与其交互。  
- **其他组件**：`vllm.distributed.kv_transfer.kv_connector.v1` 的所有子类在未来也会继承 HMA 支持框架。

## 🔍 技术洞察

| 维度 | 影响说明 |
|------|----------|
| **架构影响** | 1. **引入 Coordinator**：把原有的单一 “块‑哈希 → 缓存” 逻辑抽象为 `MooncakeStoreCoordinator`，负责：<br> • 计算 LCM（scheduler block size）<br> • 分组（attention groups）合并相同 spec<br> • 统一的 **find_longest_cache_hit** 与 **store_mask**。<br>2. **从单组到多组**：KVCache 由 `list[KVCacheGroupSpec]` 驱动，`MooncakeStoreWorker` 与 `Scheduler` 按组维护独立的 `ChunkedTokenDatabase`，实现 HMA。<br>3. **Connector‑Scheduler/Worker 解耦**：Connector 只负责调度/worker 创建，不再直接持有块‑地址列表，所有块信息通过 `Coordinator` 与 `ChunkedTokenDatabase` 传播。 |
| **性能影响** | 1. **命中检测更精细**：使用 LCM 对齐后仅在可能命中的 LCM 边界处检查，显著减少跨节点 RPC/磁盘 I/O（尤其对 SWA 长窗口）<br>2. **存储掩码**：`store_mask` 在发送端仅提交 **tail‑window** 区块，削减网络流量和磁盘写入量<br>3. **额外 CPU 开销**：在 `KVCacheStoreSendingThread`/`RecvingThread` 需要对每组计算 `store_mask` / `load_mask`、合并块哈希（`BlockHashListWithBlockSize`），在极端多组（>4）情况下会产生 O(G·N) 的遍历成本，但一般模型仅 2–5 组，影响可接受。<br>4. **内存占用**：每组维持独立 `kv_caches_base_addr` 与 `block_len`；对单组模型几乎无变化，对多组模型会略增（最多 2×‑3×）。 |
| **安全考虑** | 1. **键结构**：新增 `@group:{id}` 字段，若恶意构造的 `block_hash` 长度异常（未校验）可能导致键冲突或 DoS（大量键导致 `batch_is_exist`/`batch_put` 线性扫描）。建议在 `PoolKey.to_string` 前对 hash 长度进行断言（已有 32‑byte 固定）。<br>2. **外部存储交互**：`batch_is_exist`/`batch_put_from_multi_buffers` 仍直接转发至 MooncakeDistributedStore，未做额外权限检查。若外部 store 被窃取或被恶意修改，可能导致缓存失效或泄露 KV 内容。考虑在 store 侧加入签名或 HMAC 校验。<br>3. **并发安全**：`stored_requests`、`finished_store_req` 仍在多线程环境下使用 `defaultdict`/`set`，但已通过 `self.dec_stored_request`、`self.inc_stored_request` 加锁（内部 `self.stored_requests` 本身为 `defaultdict`，但对其计数的增删是原子的 CPython 操作，仍建议加 `threading.Lock` 以防在非 CPython 环境出现竞态）。 |

4. **Zero‑padding 正确性**：kernel 现在在 `valid_output` 条件下写入，未覆盖的列保持原始分配的 0。单元测试已验证 `out[:, n//2:]` 为全 0，建议在 CI 中加入对不同 `backend`（cutlass、flashinfer‑cutlass、flashinfer‑cudnn）的完整跑分，以防隐藏的对齐误差。  

5. **文档/示例**：更新 `vllm/_custom_ops.py` 中的 docstring，说明 `padded_n` 的意义、使用场景以及不支持的后端。示例代码应演示 `create_fp4_output_tensors` 与 `scaled_fp4_quant` 的配套使用方式。  

6. **性能回归监控**：虽然提交声称 2.4%~5.7% 端到端提升，但在不同显卡（A100、H100）以及不同 batch‑size 下的基准仍需验证。建议在项目的 performance benchmark 流程中加入对比基线，以防在极端 shape（如极宽 K）上出现反向下降。  

总体而言，此次改动通过在 kernel 层一次性写入 padded 输出，消除了额外的内存拷贝，符合 vLLM 对高吞吐量的需求。只要注意上述兼容性与文档同步，影响范围可控，建议合并。
