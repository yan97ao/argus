### [Bugfix]fix output Nan/Inf in marlin if dtype=float16 (#33972)
**SHA**: `148a5c1` | 🔗 [查看提交](https://github.com/vllm-project/vllm/commit/148a5c1226f8668cb52c4900b5ff2c80344e78f2)

**🎯 变更类型**：Bug 修复  
**⚡ 重要程度**：🟡 中  
**📋 变更摘要**：本次提交解决了在 Marlin MOE WNA16 路径下使用 `dtype=float16` 时出现的 NaN/Inf 输出问题。核心做法是将原先以 **uint16** 存储的 *global_scale* 改为 **float**，并在 CUDA kernel 中以 `float` 进行乘法，而不是仅在 `__hmul2`（FP16 × FP16）中完成。与此同时，对 `use_fp16_accum` 的判定逻辑做了细化，避免在 NVFP4（4‑bit）和 `group_blocks==-1`（即不分组）时误用 FP16 累加。Python 侧的 fp4‑Marlin 工具函数也同步改为接受/返回 `float32`，并在必要时根据输入 dtype（half / bfloat16）选择合适的 scale factor。

**🎯 影响范围**  
- `csrc/moe/marlin_moe_wna16/*`：kernel 参数、内部 scaling 变量、写回路径。  
- `csrc/quantization/marlin/*`：marlin gemm 接口对 `global_scale` 类型的检查与默认值。  
- `vllm/model_executor/layers/quantization/utils/marlin_utils_fp4.py`：scale‑factor 计算、`nvfp4_marlin_process_global_scale`、`prepare_*_for_marlin` 等高层准备逻辑。  
- 其他 `marlin_template.h` 与 `ops.cu` 的宏定义与条件编译。

**💡 关注建议**  

1. **精度回归**：在 FP16 + NVFP4 组合上跑完整的推理/生成基准，核对数值与旧版（使用 `uint16`）的差异，确保不再出现 NaN/Inf 且误差在可接受范围。  
2. **兼容性检查**：`global_scale` 参数类型已从 `uint16` 变为 `float`，使用旧模型权重文件时可能需要迁移或自动转型；建议在加载阶段加入类型兼容警告。  
3. **CUDA 架构限制**：`marlin_gemm` 的错误提示从 “CUDA_ARCH >= 8.0” 改为 “>= 7.5”，确保在 Ampere/前代（如 Turing）上的行为已通过测试。  
4. **文档与示例**：更新 README / API 文档，说明 `global_scale` 必须为 `float32`，以及在 `dtype=torch.float16` 时 `scale_factor` 会被强制为 1.0。  
5. **单元测试**：新增对 `nvfp4_marlin_process_global_scale`、`_nvfp4_compute_scale_factor` 在 `a_dtype==torch.half` 情况下返回 1.0 的覆盖；以及对 `use_fp16_accum` 判定的边界（group_size==-1、num_bits==4）进行验证。  

整体来看，此次改动在保证数值稳定性的同时，保持了原有的性能路径，但需注意模型兼容与文档同步。

### [Perf] Use torch compile to fuse pack topk in trtllm moe (#37695)
**SHA**: `b69bf2f` | 🔗 [查看提交](https://github.com/vllm-project/vllm/commit/b69bf2f0b170ac5b43f72f4dd4139c5388fa5de8)

**🎯 变更类型**：性能优化  
**⚡ 重要程度**：🟡 中  
**📋 变更摘要**：引入 `torch.compile` 对 TRT‑LLM MoE 中的 “pack topk ids & weights” 过程进行静态图融合，以取代手写的位运算实现。新工具函数 `trtllm_moe_pack_topk_ids_weights` 被加入 `utils.py`，并在 FP8‑MoE 与 NVFP4‑MoE 两个专家实现中统一调用。  

**🎯 影响范围**：  
- `vllm/model_executor/layers/fused_moe/experts/trtllm_fp8_moe.py`  
- `vllm/model_executor/layers/fused_moe/experts/trtllm_nvfp4_moe.py`  
- `vllm/model_executor/layers/fused_moe/utils.py`（新增函数）  

**💡 关注建议**  

1. **兼容性**  
   - `torch.compile` 需要 PyTorch ≥ 2.0，且在 `utils.py` 使用 `current_platform.simple_compile_backend`。请确认在所有部署环境（尤其是旧版 CUDA/CPU）下该 backend 存在，否则会在 import 时抛异常。可以考虑在函数内部捕获 `RuntimeError` 并回退到原生实现。  
   - `disable_inplace` 的缓存判断仍基于 `torch >= 2.9`，但本次 `torch.compile` 已在 2.0 引入，建议同步更新注释或阈值，避免误判。

2. **功能等价性**  
   - 新实现的位运算与原代码完全相同，但添加了 `to(torch.int32)` 与 `to(torch.bfloat16).view(torch.int16)` 的显式转换。请确保 `topk_ids` 永远是非负且小于 `2^16`，否则左移可能溢出。建议在函数开头加 `assert topk_ids.max() < (1 << 16)` 之类的安全检查。  

3. **性能验证**  
   - `torch.compile` 对单次调用的编译开销不容忽视。请在 CI 中加入基准测试，比较首次调用与后续调用的延时，确保整体加速大于手写实现。若编译时间显著，应考虑使用 `torch._dynamo.optimize` 按需开启。  

4. **测试覆盖**  
   - 添加单元测试，验证不同 dtype（int64、int32）与不同设备（cpu、cuda）下的打包结果与手写实现一致。  
   - 对 `torch.compile` 失败的路径也要有测试，确保 fallback 正常工作。  

5. **文档与可维护性**  
   - 在 `utils.py` 增加函数的详细说明（输入形状、dtype、返回值的意义），并在 README 或 `model_executor` 对象的注释中说明该函数已使用 Torch 2.x 编译特性。  

总体来看，此次改动通过 Torch 2.x 的编译特性将位运算融合进图内部，可望减少 Python‑C++ 边界开销，提升 MoE 路由的吞吐。但需在兼容性、错误回退和测试三方面做好保障，以免在旧环境或异常输入下导致运行时错误。

5. **回归测试**：重点验证在多 GPU DP 环境下的同步行为、FULL‑graph 直接执行路径以及旧的 piecewise 路径的等价性，防止因 `num_tokens_across_dp` 处理不当导致输出偏移。  

总体而言，改动提升了代码可读性和调度抽象，若上述细节得到验证，风险可控。

5. **文档/使用说明**：补充 “online quantization + reload” 的配置示例及注意事项，尤其 `quantization=fp8` 需要平台 FP8 支持。  

整体来看，本次改动为 vLLM 引入了在线量化后重载的关键特性，代码结构已抽象为通用层级处理，若做好兼容性验证与文档补全，可显著提升模型热更新的灵活性。

**🎯 影响范围**  
- `vllm/config/vllm.py`（fusion 开关逻辑）

**💡 关注建议**  
1. **跨平台兼容性**：删除 `triton` 依赖后，其他平台（尤其是 CUDA）仍然使用原来的对齐（64/128），请确认没有遗漏对 `get_padding_alignment` 的调用；全库搜索确认已全部替换。  
2. **模型配置检查**：`enable_norm_pad_fusion` 现在仅在 hidden size = 2880 且 RMSNorm 开启时返回 true，若未来出现非 GPT‑OSS 但仍需要该融合，需手动扩展判断。建议在文档或 README 中标注此限制。  
3. **对齐硬编码风险**：将 ROCm 对齐固定为 256 能满足当前 `gfx950` 等 GPU，但若后续支持其他 AMD 架构（如 CDNA3）可能需要重新评估。可考虑保留可配置常量或在运行时检测硬件特性。  
4. **回归测试**：重点运行 ROCm‑only 的 MOE‑MXFP4 测例，验证 `hidden_size` 与 `intermediate_size` 的对齐结果以及推理数值一致性；同时跑一次 CUDA 测试确保未受此改动影响。  

总体来看，此次修改简化了 ROCm 路径的实现，消除了对 Triton runtime 的不必要依赖，风险主要在于硬编码对齐值的未来可扩展性和对非‑GPT‑OSS 模型的兼容性。建议在下一轮发布前补充对应的测试覆盖并在文档中明确限制条件。
