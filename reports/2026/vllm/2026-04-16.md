### [Frontend][last/5] Improve pooling entrypoints | clean up. (#39675)
**SHA**: `4e8c3f1` | 🔗 [查看提交](https://github.com/vllm-project/vllm/commit/4e8c3f1c197952454f244e5585a5de5990865939)

**🎯 变更类型**：重构 / 架构变更  
**⚡ 重要程度**：🔴高  

**📋 变更摘要**  
- 将 Pooling（embedding、classification、scoring 等）相关的工厂、路由、IO 处理器统一迁移至 `vllm/entrypoints/pooling/factories.py`，并删除旧的 `io_processor_factories.py`。  
- 为 OpenAI **生成**（`generate`）任务新增 `vllm/entrypoints/openai/generate/factories.py`，提供 `get_generate_invocation_types`。  
- `vllm/entrypoints/sagemaker/api_router.py`、`vllm/entrypoints/openai/api_server.py` 等入口文件改为使用新的工厂函数统一获取 **invocation types**，实现生成任务与 pooling 任务的解耦。  
- 代码层面统一了相对导入路径（ `..`），删除了大量冗余/循环导入，并对多处 `io_processor`、`serving`、`typing` 的导入做了整理。  
- 文档补充了 **Pooling Types** 表格，更新了 scoring、token‑classify、OpenAI 兼容服务器等章节的链接与描述。  

**🎯 影响范围**  
- `vllm/entrypoints/pooling/*`（包括 `base`, `embed`, `classify`, `scoring`, `pooling` 子模块）  
- `vllm/entrypoints/openai/*`（`api_server.py`, `generate/factories.py`, `generative_scoring/api_router.py`）  
- `vllm/entrypoints/sagemaker/api_router.py`  
- `vllm/entrypoints/openai/openai_compatible_server.md` 以及其他文档文件  

---

### 🔍 技术洞察  

| 维度 | 影响说明 |
|------|----------|
| **架构影响** | 1. **模块化增强**：所有与任务类型相关的「入口点」统一到 `factories.py`，使得 `register_*_api_routers` 与 `init_*_state` 的实现更具可插拔性。<br>2. **解耦生成 & pooling**：生成任务的路由/调用类型不再与 pooling 代码混杂，`get_generate_invocation_types` 与 `get_pooling_invocation_types` 分别负责各自职责，降低耦合度。<br>3. **导入路径统一**：通过相对 `..` 导入消除循环依赖，提升 IDE/类型检查的可读性。 |
| **性能影响** | - 代码改动主要是 **导入路径和工厂创建**，运行时几乎无额外开销。<br>- `init_pooling_io_processors` 依旧在服务启动阶段一次性实例化，每种任务只创建对应的 `PoolingIOProcessor`，与旧实现相当。 |
| **安全考虑** | - 变更未引入新的网络、文件或外部依赖，安全面没有新增风险。<br>- 由于路由注册顺序更明确，理论上可避免旧实现中因模块加载顺序导致的未捕获异常。 |
| **可维护性** | - 统一的工厂文件简化了后续新增任务（如自定义 pooling 类型）时的代码改动点。<br>- 文档同步更新，提升了使用者对不同 **Pooling Types**（CLS、LAST、MEAN、ALL、STEP）的认知。 |
| **向后兼容** | - 旧的 public import 路径（如 `vllm.entrypoints.pooling.io_processor_factories`）已被删除，直接使用会导致 `ImportError`。<br>- 若外部项目依赖这些内部路径，需要在迁移指南中提供临时兼容 shim 或在下一个 major 版本中正式删除。 |

### ⚠️ 潜在风险  

1. **导入路径破坏**：外部代码或内部测试仍引用已删除的 `vllm.entrypoints.pooling.io_processor_factories`，会在运行时抛出 `ModuleNotFoundError`。  
2. **工厂返回的 Processor 实例缺失属性**：在 `PoolingServeContext` 中原有的 `intermediates` 被改为 `original_engine_inputs`，若存在旧代码直接访问 `intermediates` 会报错。  
3. **路由顺序微调**：`sagemaker/api_router.py` 现在先拼装生成任务的 `invocation types`，再拼装 pooling；若有依赖生成任务路由先于 pooling（如同一 URL 受不同任务影响）的场景，需要确认行为保持一致。  
4. **文档链接变更**：原先指向 `score-api` 的链接已被重写，若用户使用旧链接可能 404。  
5. **测试覆盖不足**：大量文件移动后，原有的单元/集成测试可能未覆盖所有新路径，导致隐藏的运行时错误。  

---

### 💡 关注建议  

1. **兼容层**：在 `vllm/entrypoints/pooling/__init__.py`（或单独的 shim 模块）添加向后兼容的导入提示，暂时保留 `io_processor_factories` 的导出并记录 *DeprecationWarning*。  
2. **更新依赖方**：在项目的 **CHANGELOG** 中明确说明 “移除 `vllm.entrypoints.pooling.io_processor_factories`，请改用 `vllm.entrypoints.pooling.factories`”。并在文档的 **升级指南** 中提供示例迁移代码。  
3. **完善测试**：新增针对 `init_pooling_io_processors`、`get_pooling_invocation_types`、`get_generate_invocation_types` 的单元测试，验证在不同 `supported_tasks` 组合下返回的 router 与处理器数量、类型均符合预期。  
4. **CI 检查**：在 CI 中加入对 **ImportError** 的全局搜索，确保没有遗漏的旧路径引用。  
5. **性能基准**：虽然理论上无影响，仍建议在关键服务启动（OpenAI API Server、SageMaker）上跑一次启动时间基准，确认额外的工厂调用未引入显著延迟。  
6. **文档同步**：确认所有 README / MD 中的内部链接 (`../models/pooling_models/...`) 已指向新位置，防止用户在阅读时出现 404。  

--- 

**结论**：此次提交通过 **工厂化** 与 **路径重构** 大幅提升了代码的组织结构和可扩展性，对运行时性能基本无影响，但在向后兼容性和测试覆盖方面需要额外关注与验证。若在上线前完成上述建议的兼容 shim 与测试补全，可将风险降至可接受水平。

### [Mistral Grammar] Fix tool and reasoning parsing (#39217)
**SHA**: `c0722f2` | 🔗 [查看提交](https://github.com/vllm-project/vllm/commit/c0722f22de7159c7ed91fa8268bcecd87b35fe9e)

**🎯 变更类型**：功能增强 / 重构 / 架构变更  
**⚡ 重要程度**：🔴高  
**📋 变更摘要**  
- 为 Mistral 系列模型引入 **Grammar‑Driven Tool Call 与 Reasoning 解析**，新增 `MistralToolParser` 的完整流式与非流式处理路径。  
- 在 `ChatCompletionRequest` 中加入私有属性 `_grammar_from_tool_parser`，用于标记请求已由 Grammar 注入结构化输出。  
- 改写 `vllm` 的多处核心入口（`chat_completion/serving.py`、`engine/serving.py`、`render/serving.py`）以在 **Mistral Grammar** 路径下统一走*推理 → 工具调用*的流式解析分支。  
- 对 Mistral tokenizer 与 `mistral-common` 的兼容层做了大量清理与适配（字段过滤、默认值填充、工具对象适配函数 `adapt_inplace_to_mistral_tool`）。  
- 新增/扩展测试覆盖：工具调用解析、流式解析、并行工具调用、错误路径、预‑v11 与 v11+ 不同分支的行为校验。  

**🎯 影响范围**  
- **Tokenizer**：`vllm/tokenizers/mistral.py`  
- **协议层**：`vllm/entrypoints/openai/chat_completion/protocol.py`（新增 PrivateAttr）  
- **服务层**：`vllm/entrypoints/openai/chat_completion/serving.py`、`engine/serving.py`、`render/serving.py`  
- **Tool Parser**：`vllm/tool_parsers/mistral_tool_parser.py`（核心实现）  
- **采样参数**：`vllm/sampling_params.py`（结构化输出验证改动）  
- **测试套件**：`tests/tool_parsers/*`、`tests/tool_use/*`  

---  

### 🔍 技术洞察  

| 维度 | 影响描述 |
|------|----------|
| **架构影响** | 1. **新增 Grammar 路径**：通过 `ChatCompletionRequest._grammar_from_tool_parser` 标记，使得后续的流式/非流式处理能够 **一次性** 完成 *推理 + 工具调用* 的解析，避免了原来在 `chat_completion_stream_generator` 中分别走两条分支的复杂度。<br>2. **ToolParser 接口扩展**：`MistralToolParser` 现在提供 `extract_maybe_reasoning_and_tool_streaming` 与 `build_non_streaming_tool_calls` 两个公开入口，进一步解耦了推理解析 (`ReasoningParser`) 与工具解析 (`ToolParser`) 的耦合。<br>3. **Tokenizer 与 Grammar 统一**：Mistral tokenizer 新增 `supports_grammar` 检查，只有满足条件才会在 `preprocess_chat` 中强制调用 `adjust_request`，确保 grammar 能在 **tool_choice=none** 场景下仍然生效，防止“工具泄漏”。 |
| **性能影响** | 1. **一次性流式解析**：原本在流式路径里会在 *推理结束后* 再次遍历已生成的 token，导致 **两遍解码**；新实现在同一次迭代中完成推理与工具解析，理论上降低 **CPU 与内存拷贝** 开销。<br>2. **正则回退 + JSON 解析**：仍保留对非标准输出的正则回退路径，但加入了 `ValidationError` 捕获，避免因错误的 JSON 结构导致异常崩溃。<br>3. **Tokenizer 适配层**：`adapt_inplace_to_mistral_tool` 在工具列表上做一次遍历并原地修改，成本极低（O(N)），对整体吞吐几乎无感。 |
| **安全考虑** | 1. **JSON 解析安全**：使用 `json.loads` 直接解析模型输出，需要确保模型输出的 JSON 结构不会被恶意注入，例如嵌入 `__proto__`、`eval` 等属性。当前使用 `json` 标准库，已足够安全，但仍建议在上层对 `function.arguments` 进行 **schema 验证**（如 Pydantic）后再执行。<br>2. **工具字段过滤**：通过 `_pop_unallowed_keys_and_warn` 移除 `mistral-common` 不支持的字段，防止意外传递不受信任的键值到下游系统。<br>3. **Tool Call ID 长度**：在 pre‑v11 与 v11+ 使用不同长度的随机 ID（9 vs 30），需确保 ID 生成不产生碰撞或泄漏模型内部信息。 |
| **可维护性** | 1. **代码分层更清晰**：推理/工具解析的混合逻辑抽到 `MistralToolParser`，服务层仅做路由判断，后续若引入新语言模型或新 Grammar，只需实现对应的 `ToolParser` 即可。<br>2. **类型标注**：大量使用 `typing`（`Any`, `TYPE_CHECKING`）以及 `msgspec.Struct`，提升 IDE 与静态检查的友好度。<br>3. **测试覆盖**：新增约 **2000 行**的单元/集成测试，覆盖前向、回退、异常、并行调用等多场景，降低回归风险。 |

### ⚠️ 潜在风险  

| 风险点 | 说明 | 严重度 |
|--------|------|--------|
| **兼容性回退** | 旧版（pre‑v11）模型仍使用 `extract_tool_calls` 的正则回退路径，若模型输出不符合 `[TOOL_CALLS]` 的严格 JSON 格式，可能触发 `ValidationError`（已在测试中捕获）。生产环境中若有老模型，需监控此类错误并可能提供兼容开关。 | 中 |

2. **后端列表完整性**：`get_current_attn_backends` 必须在所有可能的执行路径返回非空列表，否则 `select_common_block_size` 可能返回 `None`。可加入容错日志。  
3. **性能影响**：统一使用字节长度避免了对每层形状的额外检查，理论上略有提升；但 `select_common_block_size` 每次实例化时都会遍历后端列表，若后端查询开销较大，可考虑缓存。  
4. **测试覆盖**：现有新增测试覆盖了 MLA+Eagle 同时存在的场景，建议再补充跨 GPU/CPU、不同 `block_size` 配置下的集成测试，确保在真实推理流水线中 `self.block_len_per_layer` 与实际块大小保持一致。  
5. **文档更新**：在 KV‑connector 使用说明中补充“支持混合 MLA 与 Eagle（或其他 speculative decoding）布局”，并说明块大小会自动同步为所有后端的最小公共值。  

总体来看，此次改动解决了混合布局下的 KV‑cache 注册错误，提升了 MooncakeConnector 对多后台 speculative decoding 场景的鲁棒性，只需关注上述兼容性与文档同步即可安全上线。
