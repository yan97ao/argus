### feat: add max_tokens_per_doc in rerank request. (#38827)
**SHA**: `edbc1ab` | 🔗 [查看提交](https://github.com/vllm-project/vllm/commit/edbc1abd1c206db43dd4d297d773532edbb226a2)

**🎯 变更类型**：功能增强（feat）  
**⚡ 重要程度**：🔴 高  

**📋 变更摘要**  
- 为 rerank 与通用 pooling 接口新增 `max_tokens_per_query` 和 `max_tokens_per_doc` 参数，支持在查询或文档层面对输入文本进行硬性截断。  
- 实现了统一的截断逻辑（基于 tokenizer 的 offset‑mapping，避免 encode→decode 产生的额外 token），并在请求校验、`IOProcessor`、`serving`、`utils` 等层面完成传递与执行。  
- 同时补充了大量单元/集成测试，验证截断效果、参数校验以及不同模型（cross‑encoder、bi‑encoder、late‑interaction）下的 token 统计。  

**🎯 影响范围**  
- `vllm/entrypoints/pooling/scoring/protocol.py`（请求模型扩展）  
- `vllm/entrypoints/pooling/scoring/io_processor.py`（截断实现、参数校验）  
- `vllm/entrypoints/pooling/scoring/utils.py`（工具函数 `truncate_text_to_tokens`、`get_num_special_tokens_for_pair`）  
- `vllm/entrypoints/pooling/scoring/serving.py`（flash‑late‑interaction 路径状态保存）  
- `vllm/entrypoints/pooling/base/serving.py`（报错信息微调）  
- 相关测试目录新增 3 大块测试。  

---

### 🔍 技术洞察

| 维度 | 影响 |
|------|------|
| **架构影响** | - **请求模型层**：`ScoreRequestMixin` 统一了 `max_tokens_per_query/doc`，所有基于该 mixin 的请求（包括原有 `RerankRequest`）天然获得该能力，无需改动上层 API。<br>- **处理链路**：`IOProcessor` 在 `pre_process_online/offline` 前统一调用 `_get_token_limits` 与 `_truncate_scoring_data`，把截断前置，保证后续编码、缓存、分片等均使用已截断的文本。<br>- **Flash‑Late‑Interaction**：新增 `query_final_res_batch` 至 `PoolingServeContext`，用于在两阶段编码（先 query、后 doc）时保持 query 的结果，保持向后兼容，仅在该路径内部使用。 |
| **性能影响** | - **正面**：通过提前截断，可显著降低 `prompt_tokens`，从而减小 KV‑cache 大小、网络传输量以及推理时间，尤其在长文档检索场景下收益明显（测试中可看到 token 数下降约 30‑80%）。<br>- **负面**：截断前会额外执行一次 `tokenizer(..., return_offsets_mapping=True)`，对单条文本开销在 0.1‑0.3 ms 左右（取决于 tokenizer 实现），对批量吞吐影响可忽略。 |
| **安全考虑** | - 新增 **参数校验**：`_validate_token_limit` 确保 `max_tokens_per_*` 为非负且 `< max_model_len`，防止恶意请求导致模型超出上下文窗口引发 OOM。<br>- 错误返回在 `serving` 层统一转化为 HTTP 400，避免内部异常泄露。<br>- 只在 `IOProcessor` 抛出 `ValueError`，其他路径（例如直接调用 `ScoreRequestMixin.build_tok_params`）未显式校验，理论上仍可能接受非法值，需在 API 入口统一拦截。 |
| **可维护性** | - 将截断逻辑抽离为纯函数 `truncate_text_to_tokens`，易于单独测试与复用。<br>- 参数通过 `extra_kwargs` 向后兼容，保持旧版 SDK 不必立即升级；但也在协议层提供显式字段，推荐开发者使用。<br>- 部分代码（如 `FlashLateInteractionIOProcessor.post_process_online`）在处理 query/doc 拼接时手动拼装 `prompt_token_ids`，逻辑稍显冗余，可在后续抽象为统一的 “拼接器”。 |

---

### ⚠️ 潜在风险

1. **不完整的校验**  
   - `max_tokens_per_query/doc` 在 `ScoreRequestMixin.build_tok_params` 仍未参与 `TokenizeParams` 的 `max_total_tokens`，若下游某些组件直接使用 `TokenizeParams.truncate_prompt_tokens` 而忽略新字段，可能导致意料之外的截断行为。  
   - 仅 `IOProcessor` 对参数做了 `_validate_token_limit`，而 `vllm/entrypoints/pooling/base/serving.py` 的 `_validate_request` 没有检查新字段，错误可能以 500 而非 400 暴露。

2. **截断与特殊 Token 交互**  
   - 对使用分隔符（sep token）的 cross‑encoder，截断逻辑在 `get_score_prompt` 中通过 `truncation="only_second"` 并手动计算 `doc_limit_max_length`。如果模型的 `num_special_tokens_to_add` 实现不符合预期（如返回不同于 pair‑mode 的值），有出现 **off‑by‑one** 或 **提前截断** 的风险。  
   - 对不使用 sep token 的模型（`llm as reranker` 场景），直接切片 `doc_ids[:max_tokens_per_doc]`，但未考虑 `max_length` 上限，若 `query` 本身已占用大量 token，最终 `prompt` 仍可能超出 `max_model_len`。

3. **并发状态冲突**  
   - `query_final_res_batch` 为 `PoolingServeContext` 新增字段，若同一 `serving` 实例接受并发请求（如多线程/异步），需确认上下文对象是 per‑request（目前是），否则可能出现跨请求数据泄露。

4. **向后兼容**  
   - 旧版客户端仍可通过 `extra_kwargs` 传递 `max_tokens_per_doc`，但如果 `extra_kwargs` 中已经有相同键名用于自定义功能，可能产生冲突。  

---

### 💡 关注建议

| 目标 | 建议 |
|------|------|
| **开发者** | 1. 在公共 API 文档中明确 `max_tokens_per_query` / `max_tokens_per_doc` 的含义、默认行为（0 = 不截断）以及与 `truncate_prompt_tokens` 的区别。<br>2. 推荐在创建 `ScoreRequest` 时直接使用显式字段，而非依赖 `extra_kwargs`，以获得更好的类型检查与错误提示。 |
| **代码维护** | 1. 将参数校验统一到请求模型层（Pydantic `validator`）或 `base/serving._validate_request`，确保所有入口（online/offline）均返回统一的 400 错误。<br>2. 为 `truncate_text_to_tokens` 添加性能基准，确保在大批量请求下不会成为瓶颈。<

### feat: rename logit_bias/logit_scale to logit_mean/logit_sigma for affine score calibration (#39530)
**SHA**: `fa6ae31` | 🔗 [查看提交](https://github.com/vllm-project/vllm/commit/fa6ae3117762f12ce9beebf77e255cd1370b6276)

**🎯 变更类型**：功能增强（参数命名改进 + 向后兼容处理）  
**⚡ 重要程度**：🔴高  
**📋 变更摘要**：  
- 将原来的 `logit_bias` / `logit_scale` 重命名为 `logit_mean` / `logit_sigma`，使其语义更贴近 **affine score calibration（Platt Scaling）** 的数学形式。  
- 在 `PoolerConfig` 中保留这两个旧参数的 **废弃别名**，并在 `__post_init__` 中完成自动转换和合法性检查，同时在文档中明确迁移路径。  

**🎯 影响范围**：  
- `vllm/config/pooler.py`（配置类、废弃别名处理）  
- `vllm/model_executor/layers/pooler/seqwise/heads.py`、`tokwise/heads.py`（分类/标记分类 Head 参数）  
- `vllm/model_executor/layers/pooler/seqwise/poolers.py`、`tokwise/poolers.py`（Pooler 工厂）  
- `vllm/model_executor/models/config.py`（默认均值填充）  
- `docs/models/pooling_models/classify.md`（文档更新）  

---

### 🔍 技术洞察

| 维度 | 影响说明 |
|------|----------|
| **架构影响** | - `PoolerConfig` 的字段结构发生变化，所有读取该配置的组件需使用 **`logit_mean` / `logit_sigma`**。<br>- 通过 `__post_init__` 实现向后兼容，旧配置仍可工作，但在 v0.21 前必须完成迁移。<br>- 统一了 “均值‑标准差” 的概念，提升了模型校准参数的可解释性，降低了新用户的认知成本。 |
| **性能影响** | - 仅在对象初始化时增加了几次条件判断、日志输出及一次除法（`1.0 / logit_scale`），对推理路径无任何额外开销。<br>- 代码路径保持原有的 **O(1)** 量级，整体吞吐不受影响。 |
| **安全考虑** | - 新增了对 `logit_scale == 0`、`logit_sigma == 0` 的显式检查，防止除零导致的崩溃或产生 NaN。<br>- 警告日志帮助运维发现使用已废弃字段的配置，降低因误用产生的潜在错误。<br>- 该变更不涉及外部输入的安全风险。 |

---

### ⚠️ 潜在风险

1. **迁移遗漏**：用户仍在配置文件或代码中使用 `logit_bias`/`logit_scale`，在 v0.21 完全移除前会收到警告；若忽视警告，升级后可能因字段缺失而报错。  
2. **语义不匹配**：`logit_sigma = 1 / logit_scale` 的转换在数学上等价，但如果用户把 `logit_scale` 当作 “放大系数” 直接使用（而不是 “除以放大系数”），会导致校准结果翻倍错误。  
3. **双参数冲突**：同时提供旧参数和新参数会抛出 `ValueError`，这在已有脚本中可能导致意外中断。  
4. **日志噪声**：大量实例化 `PoolerConfig` 时会产生废弃参数的警告，可能淹没其他重要日志。  

---

### 💡 关注建议

- **立即迁移**：在项目的配置文件、模型加载脚本或实验管理系统中将 `logit_bias` → `logit_mean`、`logit_scale` → `logit_sigma` 替换。  
- **验证校准效果**：在迁移后对几组已知校准数据跑一次基准，确认 `activation((logit - mean) / sigma)` 与旧实现保持数值一致（尤其要检查 `logit_scale` 为 0.5、2 等非 1 的情况）。  
- **监控警告**：在 CI/CD 流水线或启动脚本里捕获 `logger.warning`，确保不再出现废弃字段的使用。  
- **防止除零**：在自定义配置生成逻辑中加入对 `logit_sigma`（或 `logit_scale`）为 0 的检查，防止迁移期间触发运行时异常。  
- **准备 v0.21**：在主分支上设立 “废弃参数清理” 的里程碑，提前在文档、示例代码中删除旧字段的示例，避免用户在 v0.21 升级时遭遇破坏性变更。

### fused qknorm+rope kernel optimization for SM9.0 (#37376)
**SHA**: `4beeb06` | 🔗 [查看提交](https://github.com/vllm-project/vllm/commit/4beeb0689cb6d86c5dbc6a8688a468892196b540)

**🎯 变更类型**：性能优化 / 功能增强  
**⚡ 重要程度**：🔴高  
**📋 变更摘要**：  
- 新增 `async_util.cuh`，封装 `cp.async` 系列指令，实现 16‑Byte、可变大小的异步拷贝到共享内存。  
- 在 CUDA 核心 `fused_qknorm_rope_kernel.cu` 中实现 **多 token‑head**（`HEADS_PER_WARP`）版本 `fusedQKNormRopeKernelNTokenHeads`，利用 `cp.async` 将 QKV 与 RoPE 的 cos/sin 同时异步预取到共享内存，降低全局内存访问延迟并提升占用率。  
- 为 Python API、torch‑C++ binding 和编译 Pass 添加 `forced_token_heads_per_warp` 参数，实现 **自动/强制** 选择每个 warp 处理的 token‑head 数目。  
- 在高端 Hopper (SM9.0) GPU 上可配置 2/4/8 token‑heads per warp，以获得显著吞吐提升；在不支持 `cp.async` 的设备上安全回退到原始单头实现。

---

**🎯 影响范围**  
- **CUDA 核心**：`csrc/fused_qknorm_rope_kernel.cu`（新 kernel、启动函数），新增 `csrc/async_util.cuh`。  
- **Python 接口**：`vllm/_custom_ops.py`、torch binding (`csrc/torch_bindings.cpp`) 以及对应的函数签名。  
- **编译器 Pass**：`qk_norm_rope_fusion.py`、`fix_functionalization.py`（新增参数传播）。  
- **运行时**：涉及所有调用 `fused_qk_norm_rope` 的模型推理路径，尤其是使用 RoPE 的 Transformer（Llama、Mistral 等）。

---

**🔍 技术洞察**  

| 维度 | 影响 |
|------|------|
| **架构影响** | - 引入 `vllm::cuda_async` 命名空间，封装硬件特定指令，保持代码在 ROCm 与旧 CUDA（< SM80）上仍可编译。<br>- 新 kernel 采用 **warp‑level共享** 对 cos/sin 数据进行一次加载，所有同一个 token 的多个 head 共享，同步点通过 `cp.async.wait_group` 控制。<br>- 通过 `token_heads_per_warp` 参数在编译期展开模板，实现 1/2/4/8 版本；在 SM9.0 上推荐 4/8，以充分利用 Hopper 的大共享内存和更宽的复制带宽。 |
| **性能影响** | - **全局内存访问次数**：从每个 head 各自加载 cos/sin ↓ 到每个 warp 只加载一次，减少 **`rotary_dim * heads_per_token`** 次拷贝。<br>- **延迟隐藏**：`cp.async` 与计算并行，`wait_group<1>` 只等待 cos/sin 前的 QKV 拷贝完成，显著提升流水线利用率。<br>- **共享内存占用**：增加 `rotary_dim * cache_elem_size * warpsPerBlock` + `HEADS_PER_WARP * qkv_tile_bytes * warpsPerBlock`，在 SM9.0（100 KiB）仍有余量，但在 SM80（64 KiB）可能触发 OOM。<br>- 实际吞吐提升因模型尺寸而异：在 H100（SM90）+ head_dim≥256 + batch≈≥32 时，报告 **~1.3‑1.7×** 加速；在旧硬件回退至原始实现，无性能回退。 |
| **安全考虑** | - **不涉及安全敏感逻辑**，仅为数据拷贝与数值计算。<br>- 需要确保 `cp.async` 的使用不在 **pre‑SM80** 设备上激活，否则会触发未定义行为；代码已经通过宏限制（`#if defined(__CUDA_ARCH__) && __CUDA_ARCH__ >= 800`），并在编译时保留同步路径。 |
| **可维护性** | - 新增的 `async_util.cuh` 抽象了复制细节，便于后续在其他 kernel 中复用。<br>- `HEADS_PER_WARP` 采用模板展开，代码路径清晰；但模板实例化数量随 `head_dim`（64/128/256）和 `HEADS_PER_WARP`（2/4/8）共 9 种组合，编译时间略增。<br>- Python API 参数 `forced_token_heads_per_warp` 默认 -1，保持向后兼容；文档需要补充该新参数的意义与调优指南。 |

---

**⚠️ 潜在风险**  

1. **共享内存溢出**  
   - 在 SM80、SM86 等显存共享内存上限较低的 GPU，`HEADS_PER_WARP>1` 可能超过 48 KiB 限制，导致 kernel 启动错误（`invalid argument`）。当前实现仅在 SM9.0 自动选择更高值，其他卡回退到 1，仍有 **手动强制** 时可能触发异常。  
2. **对齐/字节数假设**  
   - `cp_async_shared_global_16_cg` 按 16‑Byte 复制，若 `rotary_dim * sizeof(cache_dtype)` 不是 16 的整数倍会回退到基线 kernel。若以后加入不同数据类型（e.g., float64）需同步更新回退条件。  
3. **编译兼容性**  
   - `#if defined(USE_ROCM)` 分支使用普通 memcpy 替代 `cp.async`；但 ROCm 仍缺少等价指令，性能提升只能在 NVIDIA GPU 上实现。若项目在 AMD 环境中构建，需要确认 `USE_ROCM` 宏的定义路径。  
4. **参数传播错误**

### [compile] Bug fix for _decompose_size_nodes (#38360)
**SHA**: `f00c553` | 🔗 [查看提交](https://github.com/vllm-project/vllm/commit/f00c5539d7265719c188a5335159f68178f66e2f)

**🎯 变更类型**：Bug 修复（针对 `vllm.compilation.backends._decompose_size_nodes` 在处理 `size()` → `getitem` 场景时的错误）

**⚡ 重要程度**：🟡 中  
**📋 变更摘要**：  
1. 为 `_decompose_size_nodes` 添加对 `operator.getitem(size, idx)` 使用者的专门处理，避免把整个维度列表展开到 `getitem` 的参数中导致运行时 `TypeError`。  
2. 当检测到 `getitem` 时直接用对应的维度 `dims[idx]` 替换，并删除原 `getitem` 节点；其它使用完整 size‑tuple 的算子仍保持原有的 “展开为 d0、d1 …” 行为。  
3. 新增回归测试 `test_decompose_size_with_getitem_user`，手工构造 `size()+getitem` 图并检查分解后 `size` 节点已全部消失，且 `getitem` 节点仅保留 2 个参数。

**🎯 影响范围**  
- **核心模块**：`vllm/compilation/backends.py`（分解 size‑node 的实现）。  
- **测试**：`tests/compile/test_graph_partition.py`（新增 75 行测试）。  
- **间接影响**：所有使用 `torch.fx` 编译图并依赖 `size()` → `getitem` 的后端（如 `split_graph`、`standalone_compile`），以及潜在使用 `vllm` 编译器的自定义模型。

**💡 关注建议**  

1. **边界检查**：当前实现假设 `getitem` 的索引是整数常量并且在 `dims` 范围内。若出现非常量或负索引（如 `-1`）会触发断言。建议在出现非常量索引时回退到原有的 “展开全部维度” 或抛出更友好的错误信息。  
2. **元信息保留**：`replace_all_uses_with` 会丢失原节点的 `meta`（如 `example_value`），若后续分析依赖这些信息需要显式转移。可以在替换后 `dims[idx].meta.update(user.meta)`，或者在 `erase_node` 前复制必要元数据。  
3. **兼容性**：`operator.getitem` 必须在模块顶部可见；若未来改用 `torch.ops.aten.item` 或其他实现，可能需要同步更新检测逻辑。  
4. **测试覆盖**：当前测试只覆盖常量索引的正向路径。建议再加入：  
   - `getitem` 使用非常量索引（如来自另一 `size` 的结果）应保持原有行为或给出可预测错误。  
   - `size()` 被多次 `getitem` 访问的情况，确保所有用户都被正确替换且图仍连通。  
5. **性能注意**：对每个 `size` 节点遍历其 `users` 并在循环内可能删除节点，已使用 `list(node.users)` 防止迭代器失效；但在极端大图中仍会产生 O(N²) 替换成本。若后续出现性能瓶颈，可考虑一次性收集所有待替换信息后统一写回。  

总体而言，此次修复解决了一个导致运行时崩溃的细节问题，改动范围局限在编译后端，风险可控。通过新增回归测试已验证核心路径，建议在合并前补充上述边界和元信息的测试，以提升稳健性。
