### [Model Runner V2] Add probabilistic rejection sampling for spec decoding (#35461)
**SHA**: `c77181e` | 🔗 [查看提交](https://github.com/vllm-project/vllm/commit/c77181e534597f7347fc03b7d26600fb3cea9981)

**🎯 变更类型**：功能增强 / 性能优化  
**⚡ 重要程度**：🔴高  
**📋 变更摘要**：在模型运行时引入**概率拒绝采样（probabilistic rejection sampling）**作为 Speculative Decoding（草稿模型）的可选策略。为此新增 `rejection_sample_method` 配置项、`RejectionSampler` 类以及大量内部调用改造，同时提供对草稿 logits 的缓存以支撑概率采样。  
**🎯 影响范围**：  
- `vllm/config/speculative.py`（配置层）  
- `vllm/v1/worker/gpu/model_runner.py`（模型调度层）  
- `vllm/v1/worker/gpu/spec_decode/*`（草稿生成与采样层）  
- `vllm/v1/worker/gpu/sample/*`（采样器抽象层）  
- `vllm/v1/worker/gpu/states.py`（状态管理层）  

**🔍 技术洞察**  

- **架构影响**  
  - 引入 `RejectionSampler`，在 `ModelRunner` 中将原本的 `rejection_sample` 替换为基于 `Sampler` 的统一调用路径。  
  - `SpeculativeConfig` 新增 `rejection_sample_method`，向上兼容默认 `strict`（原有行为），并提供 `probabilistic` 选项。  
  - `WorkerState` 增加 `draft_logits` 缓存（可选），为概率采样保存已处理的草稿 logits，改变了状态对象的内存布局。  
  - `Sampler.__call__` 参数从散列表形式改为直接接受 `InputBatch`，统一了 API，简化了调用链。  
  - 删除旧的 `vllm/v1/worker/gpu/spec_decode/rejection_sample.py`，全部迁移到 `rejection_sampler.py`，实现两套采样路径（strict 与 probabilistic）并在运行时动态切换。  

- **性能影响**  
  - **接受率提升**：概率拒绝采样避免了在草稿与目标模型完全匹配时强制停止，可在同等 speculative 步数下获得更多有效 token，提升吞吐量。  
  - **额外计算**：需要对草稿 logits 进行一次 `softmax`（转概率）以及残差 logits 计算和一次额外的 Gumbel 采样，增加了 GPU 运算成本。  
  - **内存开销**：`draft_logits` 缓存占用 `max_num_reqs × num_speculative_steps × vocab_size × dtype`。在大 vocab（如 32k）和高 speculative 步数时可能导致显著显存增长。默认在 `strict` 模式下不缓存，以控制资源。  
  - **Kernel 调度**：新增多段 Triton kernel（严格采样、概率采样、残差 logits 计算），每次推理会多次 launch，潜在的调度开销在小 batch 场景下可能抵消接受率的提升。  

- **安全考虑**  
  - 变更仅涉及内部计算路径，没有对外接口或网络交互的改变，不会直接引入安全漏洞。  
  - 新增的 `model_dtype` 参数在状态初始化时传递，若上层误传错误 dtype（如 `torch.float64`），可能导致显存异常或数值精度问题，需要在配置层做好校验。  

**⚠️ 潜在风险**  

1. **显存 OOM**  
   - `draft_logits` 缓存的大小随 `vocab_size`、`num_speculative_steps`、`max_num_reqs` 线性增长。对大模型（30B+）或高并发（上千请求）使用 `probabilistic` 时极易触发 OOM。  

2. **数值差异 / 采样偏差**  
   - 概率拒绝采样的实现依赖软max、残差概率、随机数生成的顺序。若 Triton kernel 的随机数产生方式和原 Python 实现不一致，可能导致生成的 token 分布与预期略有偏差，进而影响生成质量。  

3. **向后兼容性**  
   - `Sampler.__call__` 参数签名变化可能导致自定义插件或外部代码（如用户自行实现的 `Sampler`）编译错误。  

4. **调度开销**  
   - 对于短序列、低 speculative 步数的场景，新增的 kernel（尤其是残差 logits 计算）可能带来不必要的 GPU 吞吐下降。  

**💡 关注建议**  

- **资源监控**：在使用 `probabilistic` 采样前，建议通过 `vllm --max-model-len` 与 `--max-num-reqs` 调整，或在配置中显式关闭 `cache_draft_logits`（保持 `strict`），以免显存激增。  

- **渐进式开启**：默认保持 `rejection_sample_method="strict"`，在性能基准测试确认接受率提升与显存消耗比例合适后，再切换到 `probabilistic`。  

- **单元/集成测试**：  
  - 验证在不同 `num_speculative_steps`、`vocab_size`、`batch_size` 组合下，`num_sampled` 与 `num_rejected` 与理论值保持一致。  
  - 对比 `strict` 与 `probabilistic` 在同一随机种子下的生成质量（BLEU、ROUGE 等），确保没有引入明显偏差。  

- **文档与使用指引**：  
  - 在官方文档中明确 `rejection_sample_method` 的含义、内存开销估算公式以及何种场景适合开启概率采样。  
  - 为 `ModelRunner` 添加警告日志，当 `probabilistic` 开启且 `draft_logits` 为 `None` 时，提示用户显存可能不足。  

- **类型安全**：在 `SpeculativeConfig` 初始化时加入 `assert isinstance(..., torch.dtype)` 检查，防止误传 `float64` 等不支持的 dtype。  

- **回滚路径**：保留 `strict` 采样实现（已实现），确保在出现显存或数值异常时能够快速回退，不影响生产服务的可用性。

### [Kernel] Add FP8 KV cache support to Triton MLA decode attention (#34597)
**SHA**: `a1257fd` | 🔗 [查看提交](https://github.com/vllm-project/vllm/commit/a1257fd1ea93da6e27b31e4739ac2707781d8ba7)

**🛠️ 变更类型**：功能增强  
**⚡ 重要程度**：🟡 中  

**📋 变更摘要**  
1. 在 Triton MLA 实现中加入对 FP8（e4m3）KV‑cache 的支持，新增 `fp8`/`fp8_e4m3` 选项并在 kernel 中完成 FP8→BF16 的解量化。  
2. 通过 `k_scale` / `v_scale` 参数把量化比例从 Python 层传入 kernel，默认使用 1.0 以保持向后兼容。  
3. 更新文档、测试用例以及 `decode_attention_fwd` 接口，以验证 FP8 路径的数值误差（rtol 1e‑2、atol 5e‑1）在合理范围内。  

**🎯 影响范围**  
- `vllm/v1/attention/backends/mla/triton_mla.py`（后端类）  
- `vllm/v1/attention/ops/triton_decode_attention.py`（kernel 调用 & 参数签名）  
- `tests/kernels/attention/test_triton_decode_attention.py`（新增 FP8 单元测试）  
- `docs/design/attention_backends.md`（文档列出新 dtype）  

**💡 关注建议**  
1. **参数校验**：在 `decode_attention_fwd` 入口处检查 `k_scale`、`v_scale` 与 KV‑cache dtype 是否匹配，防止意外传入 BF16‑scale 导致数值偏差。  
2. **跨平台兼容**：当前实现仅在 CUDA 上加载 `torch.float8_e4m3fn`，建议在 `current_platform` 检查时给出明确的 “FP8 not supported on ROCm” 警告，防止 CI 在非 CUDA 环境报错。  
3. **性能基准**：FP8 量化会影响内存带宽与算子吞吐，建议在 CI 或内部 benchmark 中加入 FP8 与 BF16 的对比，以验证 “更低显存占用 ≈ 更高吞吐” 的预期。  
4. **量化粒度**：目前使用 **per‑tensor** 缩放，未来若出现需求可考虑实现 per‑token/segment 缩放，并在接口中保留向后兼容的默认行为。  
5. **文档同步**：确保 README / config 示例中说明 `TRITON_MLA` 现在接受 `fp8`、`fp8_e4m3`，并提醒用户在 `model_config.json` 中显式指定 `kv_cache_dtype: "fp8"`。  

总体而言，此次改动为 FP8 KV‑cache 打通了完整路径，代码结构保持向后兼容，核心风险集中在尺度参数的正确传递和跨平台支持上。适当增加校验与基准即可平滑落地。

### [Hardware] Replace torch.cuda.device_count/current_device/set_device API (#36145)

**🎯 变更类型**：功能增强（硬件抽象统一）  
**⚡ 重要程度**：🟡 中  
**📋 变更摘要**：本次 PR 用 `torch.accelerator.*` 系列 API 替换了项目中几乎所有直接调用的 `torch.cuda.*`（`device_count、current_device、set_device、synchronize、empty_cache` 等），并在文档、示例、benchmark、test 以及核心分布式/模型执行代码中同步更新。  

**🎯 影响范围**  
- **核心库**：`vllm/distributed/…、vllm/model_executor/…、vllm/v1/worker/…` 等涉及设备初始化、GPU 计数、流同步的模块。  
- **工具链**：`tools/pre_commit/check_torch_cuda.py`（检测规则同步）  
- **文档/示例**：所有 CUDA 示例、配置文档、troubleshooting 章节改为使用 `torch.accelerator`。  
- **测试套件**：几百条 `torch.cuda.*` 调用全部迁移，确保 CI 在多 GPU、CUDA/ROCm 环境下仍能通过。  
- **Benchmarks**：attention、kernel 基准均改为新 API，避免 fork‑subprocess 重复初始化错误。  

**💡 关注建议**  
1. **兼容性检查**：`torch.accelerator` 在非 CUDA 平台（ROCm、CPU）仍会返回合理值，确保所有 `if current_platform.is_cuda_alike()` 的分支保持原语义。  
2. **残余调用**：在代码库搜索 `torch.cuda.`，确认没有遗漏的隐式调用（如 `torch.cuda.Stream`、`torch.cuda.CUDAGraph` 已保留但需配合 `accelerator.current_device_index()` 使用）。

### [Perf] add packed recurrent fast path for decode (#36596)
**SHA**: `9e19f83` | 🔗 [查看提交](https://github.com/vllm-project/vllm/commit/9e19f8338b4098047175ca3119d5ae0368bcf24a)

**变更概览**  
- 新增 **packed recurrent decode** 快路径，实现了在解码阶段直接对 `mixed_qkv`（Q、K、V 打包）进行门控‑递归计算，省去显式的 `torch.split/reshape` 与 `fused_recurrent_gated_delta_rule` 两次内存搬运。  
- 引入 **环境变量** `VLLM_ENABLE_FLA_PACKED_RECURRENT_DECODE`（默认开启），在 `Qwen3NextAttention` 中检测到纯解码（`num_prefills==0、num_decodes>0、无 spec mask`）时走该路径。  
- 增加 Triton kernel `fused_recurrent_gated_delta_rule_packed_decode_kernel` 与对应 Python 包装函数，完成输入检查、维度推导、grid/warp 配置。  
- 更新 `__init__.py` 导出新算子，并在 `tests/kernels` 中加入对比参考实现的单元测试，覆盖不同 dtype、stride‑mixed‑qkv 场景。  

**影响范围**  
- **核心算子层**：`vllm/model_executor/layers/fla/ops/fused_recurrent.py`（新 kernel & 包装函数）。  
- **模型实现**：`vllm/model_executor/models/qwen3_next.py`（新增 `enable_packed_recurrent_decode` 标志及 `_forward_core_decode_non_spec` 分支）。  
- **环境配置**：`vllm/envs.py`（新增 env 读取），以及公共导出 `vllm/model_executor/layers/fla/ops/__init__.py`。  
- **测试**：`tests/kernels/test_fused_recurrent_packed_decode.py`。  

**关键实现细节 & 限制**  
1. **维度/Stride 检查**：要求 `mixed_qkv` 为 2‑D、最后一维连续；`a/b` 为 `[B, HV]` 且 contiguous；`initial_state` 为 `[B, HV, V, K]`，且 `K` 必须满足 `NK=1`（即 `K` 能被 `triton.next_power_of_2` 整除且 `cdiv(K, BK)==1`），`V` 通过 `BV=min(next_pow2(V),32)` 划分。  
2. **软阈值**：使用 `SOFTPLUS_THRESHOLD=20.0` 防止数值溢出，和原实现保持一致。  
3. **L2‑norm 选项**：默认 `use_qk_l2norm_in_kernel=True`（在 Qwen3‑Next 中硬编码），若关闭则需显式传参。  
4. **状态索引**：`ssm_state_indices` 必须是 1‑D，负值（`-1`）表示 PAD，kernel 会直接写 0 到输出。  
5. **并行布局**：grid 为 `(NV, B*HV)`，每个 block 处理单个 `v` 维度块，warps=1、stages=3，适合小 `K`（如 128）的大 batch 场景。  

**可能的风险点**  
- **兼容性**：仅在 **纯解码且无 spec mask** 的情况下使用；若模型或调度在同一前向中混用 prefill‑decode，仍会走旧路径，不会出现错误，只是性能不提升。  
- **维度不匹配**：如果用户自行更改 `HV、V、K` 配置或使用非 2‑D `mixed_qkv`（例如在自定义模型里做了额外的 padding），会触发显式 `ValueError`，需要确保在配置层统一。  
- **NK=1 限制**：当前实现只支持 `K` 能在一次 block 中完成（`cdiv(K, BK)==1`），对更大 `K`（如 256+）会直接报错，后续若模型升级需要扩大此限制，需要改写 kernel 为循环或多 block 版本。  
- **数值误差**：对 `float16/bfloat16` 使用相对宽松的 `rtol/atol`（2e‑2 / 1e‑2），在极端极小梯度下可能出现轻微偏差，需在上层对结果容忍度进行评估。  

**建议**  

*开发者*  
- 在 `vllm` 文档或模型说明中标注 **“仅限非 spec‑mask 解码”** 并说明 `VLLM_ENABLE_FLA_PACKED_RECURRENT_DECODE` 的作用与关闭方式。  
- 为防止意外触发，建议在 `Qwen3NextAttention._forward_core` 前加入 **日志**（如 `logger.debug`)），打印是否走了 fast‑path。  
- 考虑在未来的 kernel 中加入 **多‑block K 循环** 支持，以覆盖更大头维度。  
- 拓展单元测试覆盖 **异常路径**（stride 不连续、错误 batch 大小、负 `ssm_state_indices` 长度不匹配等），保证报错信息清晰。  

*用户*  
- 若使用 **Qwen‑3‑Next** 或类似模型，默认已开启该加速。若出现异常（如报错 “Invalid packed `mixed_qkv`”），可通过 `export VLLM_ENABLE_FLA_PACKED_RECURRENT_DECODE=0` 回退至老实现。  
- 在自定义模型或改动 `mixed_qkv` 结构时，务必保持 **2‑D、最后一维连续**；如需要对 `mixed_qkv` 做切片或填充，请使用 `torch.view`/`torch.as_strided` 保证 stride 正确。  
- 对于 **float16/bfloat16** 部分，若对数值精度极其敏感，可在测试环境下对比 `out_ref` 与 `out_packed` 的误差，确认在业务容忍范围内。  

总体来看，此次提交在解码阶段通过 **一次 Triton kernel** 完成门控‑递归与输出写回，显著减少了中间张量的创建与拷贝，预计在大 batch、长序列的推理场景下可带来 10%‑20% 左右的 latency 提升。同时，完善的输入校验与单元测试降低了回归风险，只要遵守上述使用前提，安全性与兼容性均可得到保证。
