### [Quantization] add humming mxfp4 moe backend (#41083)
**SHA**: `08834cc` | 🔗 [查看提交](https://github.com/vllm-project/vllm/commit/08834cc3ceb86d77231666a5e07847f741637355)

**🎯 变更类型**：功能增强  
**⚡ 重要程度**：🔴高  
**📋 变更摘要**：  
- 为 vLLM 引入 Humming Mixed‑Precision 后端，支持 MXFP4 量化的 MoE（Mixture‑of‑Experts）算子。  
- 新增/修改配置选项、环境变量、专家类实现以及量化/权重处理工具，使 Humming 可以在标准、Grouped、Batched 三种 GEMM 类型下工作。  
- 删除旧的 Humming MoE 对齐实现，统一使用 `humming` 库提供的元数据转换逻辑。  

**🎯 影响范围**：  
- `vllm/config/kernel.py`、`vllm/envs.py`（新增 kernel 名称、环境变量读取）  
- `vllm/model_executor/layers/fused_moe/*`（专家基类、各实现、后端映射、权重加载）  
- `vllm/model_executor/layers/quantization/humming.py`（量化配置、权重处理）  
- `vllm/model_executor/layers/quantization/mxfp4.py`（后端创建流程）  
- 新增 `vllm/model_executor/layers/quantization/utils/humming_utils.py`（权重/输入 schema 转换）  

**🔍 技术洞察**：

- **架构影响**  
  - **后端抽象扩展**：`Mxfp4MoeBackend` 增加 `HUMMING`，并在 `backend_to_kernel_cls` 中返回 Humming 相关专家实现 (`BatchedHummingGroupedExperts`、`HummingGroupedExperts`、`HummingIndexedExperts`)。这把 Humming 作为 MXFP4 族的一员，统一到 `make_mxfp4_moe_kernel` 的创建路径。  
  - **专家基类重构**：`HummingExpertsBase` 将 `humming_gemm_type` 从实例属性改为 `@staticmethod`，`is_batched` 改为类方法，以便在未实例化时即可查询激活格式/GEMM 类型。构造函数现在直接接受 `moe_config`、`quant_config`，解除对 `quant_method` 的硬耦合，提升模块化程度。  
  - **量化配置统一**：`humming_utils.get_humming_moe_quant_config` 替代原先在 `HummingConfig` 中手写的 `FusedMoEQuantDesc` 构造，统一了权重尺度、零点、偏置等信息的生成方式，降低重复代码。  
  - **环境变量简化**：`VLLM_HUMMING_MOE_GEMM_TYPE` 直接读取字符串，不再进行布尔转换，避免误判。  

- **性能影响**  
  - **混合精度 GEMM**：Humming 提供的 `GROUPED_CONTIGUOUS` 与 `GROUPED_MASKED` 两种实现相较于原先的 `INDEXED` 在支持更大 Token 批次及显存利用率上更优，尤其在 Blackwell 及以上 GPU 上可以使用 TMA（Tensor Memory Access）带来的带宽提升。  
  - **权重预处理**：`prepare_humming_layer` / `prepare_humming_moe_layer` 通过一次性转换并注册压缩权重，避免在推理过程中重复格式化，有助于降低 per‑token 延迟。  
  - **潜在开销**：引入 `regex` 用于层级过滤（`humming_is_layer_skipped`），在模型加载阶段会进行一次正则匹配，开销极小且仅在加载时触发。  

- **安全考虑**  
  - 代码中对外部输入（如 `envs.VLLM_HUMMING_INPUT_QUANT_CONFIG`、用户提供的 `quant_config`）仅做键查找和正则匹配，没有执行代码或网络请求，安全风险低。  
  - 仍需确保 `regex` 库的版本安全，避免潜在的 ReDoS（正则拒绝服务）攻击；当前使用的匹配逻辑简单，风险可接受。  

- **可维护性**  
  - 将 Humming 相关逻辑抽离到 `humming_utils.py`，降低核心 MOE 实现的耦合度，后续若更换 Humming 版本或增加新量化方案，只需修改该工具文件。  
  - 使用 `@staticmethod` / `@classmethod` 统一查询方式，避免因实例化顺序导致的属性未定义异常。  

**⚠️ 潜在风险**：

1. **兼容性回退**  
   - 默认 GEMM 类型仍为 `indexed`，但在支持 TMA 的 GPU 上会自动切换为 `grouped_contiguous`。若用户显式设定了不兼容的类型（如在非 Blackwell GPU 上使用 `grouped_contiguous`），可能触发运行时错误或性能下降。  

2. **环境变量解析变更**  
   - 之前对 `VLLM_HUMMING_MOE_GEMM_TYPE` 进行 `maybe_convert_bool`，现在直接返回字符串。若旧的部署脚本仍传入 `0/1` 布尔值，`get_humming_moe_gemm_type` 会把它当作非法字符串并回退到 `indexed`，导致意外的后端选择。  

3. **权重加载路径改动**  
   - `weight_loader` 中移除了对 `quant_config_name == "humming"` 的特殊处理，依赖 `HummingConfig` 的 `quant_method` 统一走 `prepare_humming_*`。如果外部仍通过旧的 `quant_config_name` 传递，可能导致权重未被正确转换。  

4. **激活格式判断**  
   - `is_batched` 现在是类方法，子类若仍实现为属性会产生 AttributeError。必须确保所有自定义 MoE 实现同步更新。  

5. **缺乏单元测试**  
   - 大量新路径（`HummingGroupedExperts`、`BatchedHummingGroupedExperts`、`HummingIndexedExperts`）在现有 CI 中可能未覆盖，潜在回归风险。  

**💡 关注建议**：

- **兼容性层**：在 `get_humming_moe_gemm_type` 中加入对 `"0"` / `"1"` 的向后兼容转换或给出明确的警告日志，以免误配。  
- **文档与示例**：更新 README / docs，明确说明：
  - 如何通过 `--moe-backend humming` 启用该后端；
  - 何时需要手动设置 `VLLM_HUMMING_MOE_GEMM_TYPE`（例如在不支持 TMA 的机器上）。  
- **回归测试**：在 CI 中加入以下矩阵测试：
  - 不同 GPU 架构（Blackwell vs. pre‑Blackwell）下的 GEMM 选择；
  - `indexed`、`grouped_contiguous`、`grouped_masked` 三种实现的功能验证；
  - 与现有 `marlin`、`flashinfer` 等后端的兼容性对比。  
- **安全审计**：对 `regex` 匹配的正则表达式来源进行审计，确保不接受用户可控的恶意模式。  
- **监控性能**：在生产环境加入对 `humming` 后端的 latency / throughput 监控，确保新实现带来预期的加速。  

通过上述措施，可最大化地发挥 Humming 后端的性能优势，同时将兼容性和维护风险降至可接受范围。
