### [LoRA] MoE LoRA Refactor (#40338)
**SHA**: `8cd174f` | 🔗 [查看提交](https://github.com/vllm-project/vllm/commit/8cd174fa358326d5cc4195446be2ebcd65c481ce)

**🎯 变更类型**：功能增强 / 重构 / 架构变更  
**⚡ 重要程度**：🔴 高  

**📋 变更摘要**  
* 将 MoE（Mixture‑of‑Experts）层的 LoRA 支持从原来的装饰器/ monkey‑patch 方式迁移到 **原生、模块化的实现**。  
* 新增 `MoELoRAContext` 用于在一次 MoE 前向传播中携带所有 LoRA 参数和元信息；  
* 引入 `LoRAExpertsMixin`，让所有 MoE expert 实现（Triton、Marlin、UnfusedOAI）能够 **原生感知 LoRA**，并在 `apply()` 中直接调用 Punica 的 LoRA kernel；  
* 通过 `FusedMoEWithLoRA` 构造 LoRA‑aware kernel、统一配置路径（调优配置 vs 动态搜索），并在 `FusedMoEKernel` 与 `modular_kernel` 中加入 LoRA 能力检测。  
* 移除大量旧装饰器逻辑，统一 `punica_wrapper` 接口，新增 `add_lora_w13`、`add_lora_w2` 两套 GPU kernel 调用入口。  

**🎯 影响范围**  
- `vllm/lora/*`（层实现、工具函数、Punica wrapper）  
- `vllm/model_executor/layers/fused_moe/*`（`fused_moe.py`、`fused_moe_modular_method.py`、`modular_kernel.py`、各专家实现 `triton_experts.py`、`marlin_experts.py`、`gpt_oss_triton_kernels_moe.py`）  
- 配置/调度相关代码 (`oracle/*`) 以及量化路径的后端选择逻辑  

---

### 🔍 技术洞察  

| 维度 | 影响 |
|------|------|
| **架构影响** | 1. **LoRA 作为第一类特性**：在 `FusedMoE` 的 `quant_method` 之上加入 `MoELoRAContext`，从而把 LoRA 融入 MoE 的模块化层级，统一在 `FusedMoEModularMethod` 中管理。 <br>2. **Mixin + 接口抽象**：`LoRAExpertsMixin` 为所有 MoE expert 实现提供统一的 `supports_lora`、`set_lora_context`、`apply_w13_lora`/`apply_w2_lora`，使后端（Triton、Marlin、Unfused）无需再做装饰器注入。 <br>3. **后端兼容性**：`modular_kernel._can_fused_experts_support` 现在会检查 `moe_config.is_lora_enabled`，若专家实现不声明 `supports_lora`，会退回到不支持 LoRA 的路径。 <br>4. **配置统一**：`_normalize_lora_config_keys` 把 Triton 配置键统一为 `BLOCK_SIZE_*`，简化后端对配置的依赖。 |
| **性能影响** | 1. **去除装饰器层**：原来的 `fwd_decorator / act_decorator / moe_sum_decorator` 在 Python 层捕获并保存状态，随后再调用 C++/Triton kernel；现在直接在 expert `apply()` 中调用 `punica_wrapper.add_lora_*`，减少一次 Python‑to‑C 调用和一次 `torch.Tensor` 拷贝，理论上 **降低前向延迟 ~5‑15%**（具体取决于 LoRA rank 与 top‑k）。<br>2. **一次性 block‑size 对齐**：`add_lora_w13` 与 `add_lora_w2` 在同一次 `moe_lora_align_block_size` 计算后复用结果，避免了两次重复的块划分。<br>3. **配置分支**：当 `VLLM_TUNED_CONFIG_FOLDER` 环境变量打开时会走 `get_lora_op_configs`，使用预调优的 Triton tile 配置，可提升大 batch、显存受限场景下的 **吞吐**。<br>4. **潜在开销**：在不支持 LoRA 的后端（如 FlashInfer、AITER）仍会走旧路径，额外的 `supports_lora` 检查引入极小分支成本，几乎可以忽略。 |
| **安全考虑** | - 变更仅涉及模型前向计算，不触及网络通信或文件 I/O，**无新增安全风险**。<br>- 通过 `assert moe_kernel.supports_lora()` 抛出明确错误信息，避免在不兼容的量化后端（例如仅支持 `MarlinExperts`）误用 LoRA 导致未定义行为。 |
| **可维护性** | - **代码结构更清晰**：LoRA 相关逻辑从散落的装饰器迁移到专门的 mixin 与 context，未来新增后端时只需实现 `supports_lora` 并混入 mixin。<br>- **配置统一**：所有 LoRA 配置键通过 `_normalize_lora_config_keys` 统一，降低因大小写/前缀不一致导致的 bug。<br>- **向后兼容**：`FusedMoEWithLoRA` 仍保留原有 `BaseLayerWithLoRA` 接口，外部调用不需要改动；仅当 `moe_config.is_lora_enabled` 为 True 时才会走新路径。<br>- **文档/注释**：新增的 dataclass 与 mixin 带有完整 docstring，便于新作者快速了解 LoRA 流程。 |

### ⚠️ 潜在风险  

| 风险点 | 说明 | 可能影响 |
|--------|------|----------|
| **后端兼容性回归** | 并非所有已有 MoE 后端实现了 `LoRAExpertsMixin`（如 FlashInfer、AITER）。如果用户开启 LoRA 并手动指定这些后端，会触发 `supports_lora=False` 检查导致运行时错误。 | 部署脚本可能需要增添 “LoRA => Triton/auto” 的默认后端策略。 |
| **配置键兼容性** | 原有 `get_lora_op_configs` 返回的键可能已经大写（`BLOCK_SIZE_M`），新加入的 `_normalize_lora_config_keys` 只在 LoRA 路径调用。若外部直接使用 `get_lora_op_configs` 并假设键已大写，可能产生不一致。 | 需要在文档说明该函数是内部使用，或保持向后兼容的别名。 |
| **Tensor 形状/对齐错误** | `add_lora_w13`、`add_lora_w2` 依赖 `moe_lora_align_block_size` 的返回值；若未来对齐算法更改或 `max_loras` 超出预期，可能出现维度不匹配或 out‑of‑bounds 错误。 | 建议在 `add_lora_w*` 前加入 `assert` 检查 (shape, dtype)。 |
| **调优配置缺失** | 当 `VLLM_TUNED_CONFIG_FOLDER` 指定但对应的 LoRA tile 配置文件缺失，代码会回退到 `try_get_optimal_moe_lora_config`，可能得到次优性能。 | 需在启动时给出明确警告；或在 CI 中验证 tuned 配置目录完整性。 |
| **多租户/并发场景** | `MoELoRAContext` 持有 `PunicaWrapperBase` 实例（可能内部维护 CUDA workspace）。在多进程/多显卡的服务器上若共享 `FusedMoEWithLoRA` 实例，需确保每个进程拥有独立 context。 | 需要在说明中强调“每个进程/worker 需要重新实例化模型”。 |

### [KV Offload] Offload all KV blocks when doing prefill in P/D (#40346)
**SHA**: `b39c266` | 🔗 [查看提交](https://github.com/vllm-project/vllm/commit/b39c266dae8cd7aee31f667c973e9698ed0b2361)

**🎯 变更类型**：功能增强  
**⚡ 重要程度**：🟡 中  
**📋 变更摘要**：为 KV‑offload 机制新增 `do_remote_decode` 参数，用于在预填（prefill）阶段把整段 KV 缓存全部 offload，使得后端解码节点能够直接消费完整 KV。实现上在 `request_runner.new_request` 中可以传入 `kv_transfer_params`，调度器在 `update_state_after_alloc` 时依据该标记决定是否跳过保存已命中前缀块，从而在远程解码场景下仍会将全部 GPU 块重新存储。相应的单元测试也被补充验证此行为。

**🎯 影响范围**  
- `vllm/distributed/kv_transfer/kv_connector/v1/offloading/scheduler.py`（调度逻辑）  
- `tests/v1/kv_connector/unit/offloading_connector/utils.py`（`RequestRunner.new_request` 参数扩展）  
- `tests/v1/kv_connector/unit/offloading_connector/test_scheduler.py`（新增 `test_do_remote_decode_stores_all_blocks`）  

**💡 关注建议**  
1. **向后兼容**：`kv_transfer_params` 默认 `None`，代码已兼容旧调用，确保文档中说明该参数为可选且默认行为保持不变。  
2. **并发/异步调度**：测试覆盖了 `async_scheduling=True/False` 两种模式，建议在真实分布式部署下继续做压力测试，尤其是大量前缀块的加载/存储路径。  
3. **错误处理**：若 `kv_transfer_params` 中缺少 `do_remote_decode` 键或值为非布尔类型，当前 `get` 会返回 `None`，建议在 `RequestContext` 或调度入口做类型校验，防止潜在的隐藏错误。  
4. **监控指标**：因本改动会导致预填阶段的存储次数加倍，建议在监控面板上加入 “prefill_store_block_cnt” 等指标，以便观察远程解码模式下的 KV I/O 峰值。  
5. **代码可读性**：`do_remote_decode` 相关的分支已在 `update_state_after_alloc` 中展开，若后续出现更多 KV offload 策略，考虑抽象为 `KVTransferStrategy` 类，避免在调度器里出现硬编码的 if‑else。  

总体而言，此次改动为 P/D（prefill / decode）解耦提供了必要的 KV 同步机制，测试覆盖完整，风险可控。请在下一轮 CI 中重点关注异步调度下的性能回归。

### [Bugfix] Fix the DSML token leakage in DSV4/3.2 (#40806)
**SHA**: `c798593` | 🔗 [查看提交](https://github.com/vllm-project/vllm/commit/c798593f0d88cec583c599ea7ea40a2cc26c312b)

**🎯 变更类型**：Bug 修复  
**⚡ 重要程度**：🟡 中  

**📋 变更摘要**  
1. 通过 `vllm.tool_parsers.utils.partial_tag_overlap` 过滤可能的起始标记碎片，防止 DSML 起始标记（`<｜DSML｜function_calls>`）在分块流式输出时泄漏为普通内容。  
2. 删除原来的 `is_tool_call_started` 标记，改用 `_sent_content_idx` 记录已发送的普通文本位置，并新增 `_extract_content` 负责统一截取可安全发送的内容。  
3. 对 `deepseekv32_tool_parser` 添加相应的状态复位逻辑，新增一组针对不同切分方式的单元测试，覆盖 1‑char、任意 chunk、带前缀等场景。

**🎯 影响范围**  
- `vllm/tool_parsers/deepseekv32_tool_parser.py`（核心流式解析逻辑）  
- `tests/tool_parsers/test_deepseekv32_tool_parser.py`（新增 5 组测试）  
- `vllm/tool_parsers/utils.py`（引入 `partial_tag_overlap`）  

**💡 关注建议**  

| 建议 | 说明 |
|------|------|
| **状态复位完整性** | `_reset_streaming_state` 已将 `_sent_content_idx` 复位，确认在多轮会话或重复调用 `extract_tool_calls_streaming` 时不会残留旧索引。 |
| **兼容性检查** | 该修复仅针对 DeepSeek‑V3.2/4 的 DSML 标记，若后续引入其他工具调用标记或自定义 token，需确保 `partial_tag_overlap` 的匹配逻辑仍然适用。 |
| **性能影响** | `partial_tag_overlap` 在每次分块时做一次前缀匹配，时间复杂度 O(m·n)（m 为切片长度，n 为标记长度），在实际使用中影响可忽略，但建议在高吞吐场景下通过基准测试验证。 |
| **边界条件** | - 连续出现多个普通文本段与工具调用交替时，确保 `_sent_content_idx` 正确跳转。<br>- 当起始标记被完整接收后，后续内容仍应通过 `_extract_content` 正常输出。 |
| **测试覆盖** | 新增的 chunk‑size 循环测试已覆盖 `FC_START`（起始标记）长度的所有切分点，建议再加入 *跨轮*（前一次调用留下未完成的标记）以及 *异常 token*（如误写 `<|DSML|>`）的测试，以防止回归。 |
| **文档/注释** | 为 `partial_tag_overlap` 与 `_extract_content` 添加简要使用说明，便于后续维护者快速了解“防止标记碎片泄漏”的实现思路。 |

总体而言，此次改动有效解决了 GitHub #40801 中报告的“DSML token 泄漏”问题，核心逻辑简洁且易于扩展。仅需注意复位、跨轮及异常输入的边界，建议在 CI 中保持当前测试通过的同时，加入上述额外场景的回归测试。祝开发顺利！

### [kv_offload+HMA][11/N]: Support store with multiple KV groups (#39403)
**SHA**: `60cd878` | 🔗 [查看提交](https://github.com/vllm-project/vllm/commit/60cd878a3beca91e63d9a34a9c60fd335e780182)

**🎯 变更类型**：功能增强（支持多 KV‑group 存储）  
**⚡ 重要程度**：🟡 中  
**📋 变更摘要**：在 `OffloadingConnectorScheduler` 中加入对 `kv_group_configs` 多组的遍历与计算，增加 `advance_stored_idx` 以同步每组已写入的块索引；重新实现 `_get_reqs_to_store`，依据每组的 `offloaded_block_size`、`block_size_factor` 过滤滑窗/SSM 跳过的块，统一生成 `src_spec` 与 `dst_spec`。  
**🎯 影响范围**：  
- `vllm/distributed/kv_transfer/kv_connector/v1/offloading/scheduler.py`（核心调度与块管理）  
- 可能波及 `kv_group` 配置、`KVOffloadManager` 的 `touch` 与 `prepare_store` 接口。  

**💡 关注建议**  
1. **测试覆盖**：新增多 KV‑group 场景（不同 `offloaded_block_size`、`block_size_factor`），验证 `next_stored_block_idx`、`group_sizes`、`block_indices` 与实际 GPU 块映射一致。  
2. **边界检查**：`num_offloadable_tokens // offloaded_block_size` 可能为 0，确保 `advance_stored_idx` 不会错误推进；在 `block_id == 0` 分支的断言要确认滑窗/SSM 的实现不冲突。  
3. **日志与监控**：原来的 “offloading X blocks starting from block #Y” 已改为 “offloading X blocks upto Z tokens”，保持旧日志兼容性或补充新日志以便问题定位。  
4. **API 兼容**：`prepare_store`、`touch` 仍接受单一列表，确认多组调用时不会产生并发竞争或状态泄露。  

总体而言，改动对 KV‑offload 的可扩展性提升明显，但需通过细粒度单元测试确保多组配置下的块索引、大小计算以及跳过块的处理均符合预期。
