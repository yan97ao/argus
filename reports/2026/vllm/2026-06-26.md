### [API] Add token offsets to render endpoints (/v1/.../render) (#44226)
**SHA**: `950ee4c` | 🔗 [查看提交](https://github.com/vllm-project/vllm/commit/950ee4c2e48fd462a03cebb3283ac9fdcb27b2e4)

**🎯 变更类型**：功能增强  
**⚡ 重要程度**：🔴高  
**📋 变更摘要**：在 OpenAI 兼容的 `/v1/completions/render` 与 `/v1/chat/completions/render` 接口中新增 `return_token_offsets` 标记，支持在渲染响应中返回每个 token 在原始文本中的字符区间 `(start, end)`。为此在请求/响应协议、内部 `TokenizeParams`、渲染器、Engine 输入结构以及序列化/反序列化路径全链路添加了对应字段和逻辑，并补充了大量单元/集成测试。

**🎯 影响范围**  
- `vllm.entrypoints.openai.completion.protocol.CompletionRequest`、`ChatCompletionRequest`（新增字段、构建 `TokenizeParams`）  
- `vllm.entrypoints.serve.disagg.protocol.GenerateRequest`（新增 `token_offsets`）  
- `vllm.renderers.params.TokenizeParams`（新增 `return_token_offsets`）  
- `vllm.renderers.base.BaseRenderer`、`vllm.renderers.hf.HfRenderer`（新增偏移提取、能力检测）  
- `vllm.inputs.engine.TokensInput`、`vllm.inputs.llm.TokensPrompt`（新增 `prompt_token_offsets`）  
- 渲染入口 `vllm.entrypoints.serve.render.serving.py`（将偏移信息写入 Engine 输入）  
- 测试目录下新增/扩展的 10+ 个测试文件。  

---

### 🔍 技术洞察  

| 维度 | 影响说明 |
|------|----------|
| **架构影响** | - **协议层**：在 OpenAI 兼容协议中加入 `return_token_offsets`（请求） 与 `token_offsets`（响应）字段，保持向后兼容（默认 `False` / `null`）。<br>- **渲染路径**：`BaseRenderer._tokenize_prompt` 现在会根据 `<renderer>._can_produce_offsets()` 与请求 flag 决定是否调用 tokenizer 的 `return_offsets_mapping` 并把结果装进 `TokensPrompt` → `EngineInput` → `GenerateRequest`。<br>- **能力检测**：通过 `BaseRenderer._can_produce_offsets`（默认 `False`）—> `HfRenderer._can_produce_offsets`（仅 fast tokenizer）实现 **可插件化**，未来其他渲染器可自行实现。<br>- **多模态保护**：在存在 `multi_modal_data`/`multi_modal_uuids` 时强制不返回 offsets，以避免语义歧义。 |
| **性能影响** | - 仅在 **Fast (Rust‑backed) tokenizer** 且 `return_token_offsets=True` 时会多一次 `offset_mapping` 计算。该路径本身在 tokenizer 实现中已是 O(N) 与普通 `input_ids` 计算同量级，额外的少量 Python 列表转化开销可忽略。<br>- 异步路径仍通过线程池 `make_async(self._tokenize_prompt)` 执行，保持对主事件循环的非阻塞特性。 |
| **安全考虑** | - 新增字段均为 **只读**，不影响模型执行或资源调度。<br>- `token_offsets` 本质是字符下标，无泄露模型内部状态的风险。<br>- 仍需确保在 **JSON 反序列化** 时对 `token_offsets` 做类型校验（已由 Pydantic `list[tuple[int, int]] | None` 完成），防止恶意构造异常数据导致服务崩溃。 |
| **可维护性** | - 代码路径清晰：请求 → `TokenizeParams` → `BaseRenderer._tokenize_prompt` → `TokensPrompt` → `EngineInput` → `GenerateRequest`。<br>- 新增的测试覆盖 **单元**（协议字段传递）、**渲染器层**（offset 生成、能力分支、异步/同步一致性）以及 **端点集成**（真实 HTTP 调用），为后续重构提供安全网。<br>- 通过 `BaseRenderer._can_produce_offsets` 的抽象，避免在基类中硬编码 fast‑tokenizer 检查，保持对其他渲染实现的可扩展性。 |
| **兼容性** | - 默认值 `False` / `null` 保持现有客户端行为不变。<br>- `GenerateRequest.token_offsets` 默认 `None`，即使旧版本的渲染器不产出 offsets，序列化仍兼容。<br>- 对 **慢速 tokenizer**（`is_fast=False`）或 **多模态输入** 自动降级为 `null`，防止意外返回错误长度。 |

### ⚠️ 潜在风险  

1. **长度不匹配**：若 tokenizer 的 `offset_mapping` 长度与 `input_ids` 不一致（极少数自定义 tokenizer），可能导致 `prompt_token_offsets` 与 `token_ids` 不匹配，引发客户端解析错误。  
2. **多模态混合场景**：当前实现直接抛弃 offsets 当出现 `multi_modal_*` 键，若后续业务需要在多模态情况下仍提供文本部分的 offsets，需要额外的 **partial‑offset** 方案。  
3. **异步/同步不一致**：虽然已在代码中统一，未来对 `_tokenize_prompt` 的修改若忘记同步更新 `*_async` 包装，可能出现两条路径返回不同的 offsets。  
4. **序列化兼容**：`GenerateRequest.model_dump()` 仍返回 `token_offsets` 列表；若 downstream 系统未更新 schema，可能出现 **未知字段** 警告。  
5. **性能峰值**：在高并发、大批量 `return_token_offsets=True` 场景，额外的 offset 计算与内存拷贝会增加每请求的 CPU/内存占用，需要监控线程池任务排队情况。

---

### 💡 关注建议  

| 对象 | 建议 |
|------|------|
| **开发者** | - 在文档中明确 `return_token_offsets` 仅在 **Fast** tokenizer、**纯文本**、**无多模态** 时生效，推荐在客户端根据模型的 tokenizer 类型自行判断是否开启。<br>- 若项目使用自定义 tokenizer，务必实现 `offset_mapping` 并保证其与 `input_ids` 同步长度。 |
| **测试/CI** | - 保持现有 10+ 单元/集成测试，并在 CI 中加入 **慢速 tokenizer**（`use_fast=False`）的兼容性回归，防止误将 `return_token_offsets` 当成必选。 |
| **运维** | - 监控渲染服务的线程池排队时长；在 `return_token_offsets=True` 的请求占比显著提升时，可考虑适当提升 `renderer_num_workers`。 |
| **后续迭代** | - 如需在 **多模态** 场景返回文本部分的 offsets，可在 `BaseRenderer._wants_offsets` 中加入 “只对 text 部分” 的细粒度检测。<br>- 考虑提供 **offset 精度选项**（如仅返回 token 起始位置），进一步降低带宽开销。 |

--- 

**总体结论**：本次改动为 vLLM 增添了一个高价值的可选特性（字符级 token 边界），实现方式遵循了“向后兼容 + 能力检测” 的最佳实践，对系统架构的侵入最小，性能开销可控，安全风险几乎为零。只需在发布说明中明确使用前提并在 CI 中保持对慢速 tokenizer 与多模态路径的覆盖，即可安全上线。

### Remove grok model arch from vllm (#46706)
**SHA**: `c7645bc` | 🔗 [查看提交](https://github.com/vllm-project/vllm/commit/c7645bce044be01c3c30ceead99388f01aa6d496)

**🎯 变更类型**：重构 / 架构变更  
**⚡ 重要程度**：🔴高  

**📋 变更摘要**  
本次提交彻底移除了对 Grok（Grok‑1 / Grok‑2）模型及其专属 tokenizer、renderer 的实现。相关代码（模型实现、权重加载、专有渲染器、tokenizer、文档表格、测试用例）均被删除，同时在配置、模型注册以及 tokenizers/registry 中去掉了 Grok 相关的映射和默认 tokenizer‑mode。目标是缩减项目体积、降低维护成本，并统一模型实现路径（仅保留通用 Transformers 后端对 Grok 的支持）。

**🎯 影响范围**  
- `vllm/model_executor/models/grok1.py`、`vllm/renderers/grok2.py`、`vllm/tokenizers/grok2.py` 完全删除。  
- `vllm/model_executor/models/registry.py` 与 `vllm/tokenizers/registry.py` 的 Grok 映射被撤销。  
- `vllm/config/model.py` 中对 `arch == "Grok1ForCausalLM"` 的 tokenizer‑mode 自动切换被去除。  
- 文档 `docs/models/supported_models.md`、`docs/configuration/optimization.md` 相应更新。  
- 测试 `tests/models/language/generation/test_grok.py`、相关注册测试被删除。  
- 依赖 Grok 的用户代码（如显式使用 `--tokenizer-mode grok2`、手动导入 `Grok2Tokenizer`、`Grok2Renderer`）将失效。

---

### 🔍 技术洞察  

| 维度 | 影响描述 |
|------|----------|
| **架构影响** | - 移除了专有的 `Grok1Model`/`Grok2Model` 与 `GrokBaseForCausalLM` 体系，所有对 Grok 的入口均回退到通用 **Transformers** 后端（通过 `model_executor/models/transformers`）。<br>- `model_config` 解析时不再自动切换 `tokenizer_mode` 为 `grok2`，统一使用 `auto`/`hf` 路径。<br>- 迁移后，模型注册表 `SUPPORTED_MODELS` 仍保留了占位条目（`"Grok1ModelForCausalLM": "0.24.0"`），但实际实现已经不存在，属于历史兼容标记。 |
| **性能影响** | - 删除约 800 行 Grok 专属模型代码及 450 行 tokenizer 实现，导致二进制体积与启动时的 Python 模块解析时间略有下降。<br>- 运行时不再有 Grok‑专属的 **MoE** 实现与自定义 `Attention`，全部走通用实现，性能波动主要取决于 Transformer's 官方实现，与之前的自研实现大致持平。 |
| **安全考虑** | - 删除了专有 tokenizer（依赖 `tiktoken`）与 renderer，降低了外部依赖泄漏或潜在解析漏洞的风险。<br>- 统一使用 HuggingFace 官方 tokenizer，已在社区得到广泛审计，安全性提升。 |
| **可维护性** | - 代码基线从 1400+ 行削减至约 600 行，维护负担显著降低。<br>- 移除了大量专有测试和文档描述，避免未来因模型细节改动导致的破碎测试。<br>- 统一模型加载路径，后续加入新模型只需在 Transformers 后端实现，复用现有层级结构。 |

---

### ⚠️ 潜在风险  

1. **向后兼容性破坏**  
   - 使用显式 `--model grok-2` 或 `--tokenizer-mode grok2` 的老用户将在启动时抛 `ImportError`（模块不存在）。  
   - 代码层面仍保留 `vllm/model_executor/models/registry.py` 中的占位条目，若用户通过 `vllm.get_model_class("Grok1ForCausalLM")` 仍会得到 `None`，导致运行时错误。  

2. **文档/版本号不一致**  
   - 虽然 `SUPPORTED_MODELS` 中仍列出 `"Grok1ModelForCausalLM": "0.24.0"`，但实际实现已被删除，若没有同步更新 Release Note，用户可能误以为该版本仍支持 Grok。  

3. **遗漏的隐式引用**  
   - 可能还有散落在其它文件（如 `examples/`, `scripts/`, 第三方插件）中对 `Grok2Tokenizer`、`Grok2Renderer` 的硬编码，导致导入错误。  

4. **权重加载兼容性**  
   - 仅保留了 **Transformers** 后端对 Grok 的支持（通过 `trust_remote_code=True`），但某些 Grok‑2 checkpoint（如使用 `tokenizer.tok.json`）的权重仍需要专有 tokenizer；在未安装 `tiktoken` 时会报错。
