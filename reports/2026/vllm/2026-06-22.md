### [Attention] Re-enable cross-layer KV cache layout for MLA via stride-aware kernels (#45111)
**SHA**: `aa4990a` | 🔗 [查看提交](https://github.com/vllm-project/vllm/commit/aa4990a9a2024b3f93f1f26f828931f7301daa15)

**🎯 变更类型**：功能增强 / 重构 / 架构变更  
**⚡ 重要程度**：🔴高  
**📋 变更摘要**：  
- 重新启用 MLA（Multi‑Layer‑Attention）在 *跨层*（cross‑layer）KV Cache 布局下的运行能力。  
- 通过在 CUDA、Triton 与 PyTorch 侧显式读取 **page‑dim stride**，让所有已验证的 MLA 后端（Cutlass、FlashMLA、FlashInfer、Triton）能够直接在 **block‑major（层间页交叉）** 的缓存视图上执行，而不再强制每层缓存必须是连续的。  
- 新增大量单元测试，验证在跨层视图下的 **decode**、**write**、**indexer**、**FP8‑sparse** 等路径的位相等性，并对后端 `kv_cache_stride_order` 接口实现了统一的非‑identity 置换 `(1,0,2,3)`。  

---

### 🎯 影响范围
- **核心计算库**：`csrc/libtorch_stable/*.cu`, `csrc/libtorch_stable/cache_kernels.cu`，以及 Triton attention 实现。  
- **后端抽象层**：`vllm/v1/attention/backends/mla/*Backend`（Cutlass、FlashMLA、FlashInfer、Triton）。  
- **KV Cache布局**：`vllm/model_executor/layers/attention/mla_attention.py` 的 `get_kv_cache_stride_order`。  
- **测试套件**：新增 5+ 组跨层等价性测试，扩展原有 decode/attention 单元测试。  

---

### 🔍 技术洞察

| 维度 | 影响说明 |
|------|----------|
| **架构影响** | 1. **跨层 KV Cache** 从 “每层独立、页面紧凑” → “层间页面块统一分配（block‑major）”。  <br>2. 通过 `get_kv_cache_stride_order(include_num_layers_dimension=True)` 返回 **(1,0,2,3)**，让调度器在分配物理块时把 **num_blocks** 放在最前面，从而支持跨层视图。 <br>3. 所有已验证的 MLA 后端实现了该查询，默认实现仍返回 **(0,1,2,3)** 表示不支持，保持向后兼容。 |
| **性能影响** | - **读‑写路径** 现在使用 **真实 stride** 而非硬编码 `PAGE_SIZE * token_stride`，对跨层视图 **没有额外复制**，因此 **内存访问次数不变，带宽保持**。 <br>- 计算上新增一次 **stride‑乘法**（如 `kv_page_number * stride_buf_kpbs + kv_in_page * stride_buf_kbs`），在 CUDA/Triton 中算术开销极低 (<0.1 %)。 <br>- 对于仍然使用连续缓存的场景，代码路径保持不变，性能不受影响。 |
| **安全考虑** | - 读取 **tensor.stride** 需要保证对应维度在 `int64` 范围内，已使用 `int64_t` 类型谨慎转型，避免溢出。 <br>- 所有内核仍然依据 **bounds mask**（`mask = offs_n < split_kv_end`）进行访问屏蔽，跨层 stride 误算不会导致越界写。 <br>- 新增的 `cache_block_stride` 参数在 Python 层已经通过 `tensor.stride(0)` 直接获取，避免手动传递错误的 stride。 |
| **可维护性** | - 所有后端显式实现 `get_kv_cache_stride_order`，代码自说明，后续新增后端只需按同样模式返回。 <br>- Triton kernel 中抽取的 `_page_stride` 统一了 3D/4D 缓冲区的 stride 读取逻辑，避免散落的硬编码。 <br>- 单元测试覆盖 **每个后端 + 每种数据格式（bfloat16 / fp8）** 的跨层等价性，提升回归安全性。 |
| **兼容性** | - 未实现 `get_kv_cache_stride_order` 的旧后端仍返回 **(0,1,2)**，在 `MLACommonBackend` 中默认 **拒绝跨层 KV Cache**（保持原有行为）。 <br>- 对外 API（`ops.sm100_cutlass_mla_decode`、`ops.concat_and_cache_mla`、`ops.indexer_k_quant_and_cache` 等）参数保持不变，仅内部使用了真实 stride。 |

---

### ⚠️ 潜在风险
1. **Stride 计算错误**  
   - 若某后端在未来更改内部 tensor 维度顺序（如增加额外维度），需要同步更新对应的 `stride_buf_*` 读取逻辑，否则会出现页面错位或越界。  
2. **老旧 GPU/驱动兼容**  
   - `sm100`（Hopper）专属的 CUTLASS/FlashInfer 路径在旧 GPU 上会自动 fallback；若 fallback 仍使用 **identity stride**，会导致跨层视图在这些硬件上错误。需要 CI 在不同 compute capability 上验证 fallback 行为。  
3. **整数溢出**  
   - `int64_t` 已足够大，但在极端模型（超大 `num_blocks * page_size * head_dim`）下仍有理论溢出风险。建议在 API 前加 `assert kv_c_and_k_pe_cache.numel() < 2**63`。  
4. **测试依赖**  
   - 新增的跨层等价性测试需要 **CUDA 11.8+**、**FlashInfer**、**FlashMLA** 等可选依赖，CI 需要确保这些可选依赖在所有 runner 中可用，否则可能出现假阴性/假阳性。  

---

### 💡 关注建议
| 对象 | 建议 |
|------|------|
| **核心开发者** | - 将 `get_kv_cache_stride_order` 作为 **后端必须实现的抽象方法**（抽象基类可加入 `@abstractmethod`），防止忘记在新后端实现。 <br>- 在提交新 CUDA/Triton kernel 时，显式使用 `*_stride` 参数而不是硬编码常数，保持代码对跨层视图的透明度。 |
| **CI/测试维护者** | - 为每个可选后端（FlashInfer、FlashMLA）配置独立的 GPU runner，确保跨层等价性测试在真实硬件上通过。 <br>- 在 CI 中加入 **stress test**：极端 `num_layers=8、page_size=128、block_size=256`，检查 `int64` stride 计算是否仍安全。 |
| **文档/使用者** | - 在 vLLM 文档中添加 *“跨层 KV 缓存布局（block‑major）”* 的概念说明，展示何时可开启、何时会被后端自动回退。 <br>- 示例代码中展示 `torch.as_strided(..., stride=(n_layers*..., ...))` 的创建方式，帮助用户自行构造跨层视图。 |
| **性能分析师** | - 通过 `nsight`/`rocm-smi` 监控 **L2/DRAM 带宽** 与 **SM 活跃度**，确认跨层视图没有引入额外 cache miss（理论上应相同）。 <br>- 对比 **连续 vs 跨层** 两种布局在大 batch/长 seq 场景的吞吐，记录是否出现微小的延迟提升（主要来自 stride 乘法），若有则考虑在 kernel 中做一次 `const` cache。 |
| **安全审计** | - 检查所有 kernel 中的 `mask` 条件，确保跨层 stride 变化后仍覆盖所有合法元素。 <br>- 在 Python 包装层增加 **参数校验**：`assert cache_stride == cache.stride(0)`，防止用户手动传错 stride 参数。 |

---

### [feature][kv_offload] Self-describing KV events for OffloadingConnector (#43468)
**SHA**: `a9f7b2d` | 🔗 [查看提交](https://github.com/vllm-project/vllm/commit/a9f7b2d41c5e92f0d60ebdd3dbe04e627019c179)

**🎯 变更类型**：功能增强（Self‑describing KV 事件）  
**⚡ 重要程度**：🔴高  
**📋 变更摘要**：  
为 OffloadingConnector 引入可选的 *self‑describing KV events*，在 KV 缓存事件开启且 `self_describing_kv_events=True` 时，OffloadingConnector 会在 `BlockStored`/`BlockRemoved` 事件中携带完整的块级元数据（块哈希列表、完整 token‑id 序列、块大小、父块哈希、LoRA 信息以及 KV‑group 规格）。默认保持向后兼容的占位符 payload，且在 TieringOffloadingSpec 中明确不支持此特性。新增大量单元测试并对 Scheduler、Manager、Spec 等核心路径进行改动，以实现元数据捕获、缓存与清理。

**🎯 影响范围**：  
- `vllm/distributed/kv_transfer/kv_connector/v1/offloading/events.py`（全新实现）  
- `vllm/distributed/kv_transfer/kv_connector/v1/offloading/scheduler.py`（事件追踪器接入）  
- `vllm/v1/kv_offload/base.py`（新增 `OffloadingKVEventsConfig`）  
- `vllm/v1/kv_offload/cpu/spec.py`、`vllm/v1/kv_offload/tiering/spec.py`（事件开关传递、Tiering 中的限制）  
- 文档 `docs/features/kv_offloading_usage.md`（新增配置信项）  
- 多套单元测试（`tests/v1/kv_connector/unit/offloading_connector/*`）  

**🔍 技术洞察**：

| 维度 | 影响分析 |
|------|----------|
| **架构影响** | - 在 OffloadingConnector Scheduler 与 Manager 之间新增 `OffloadingEventsTracker`，形成“事件侧表”层，负责在块存储时快照完整元数据，直至对应的 eviction（`BlockRemoved`）时使用。<br>- 通过 `OffloadingKVEventsConfig` 将全局 KV 事件开关与 connector‑level 自描述开关解耦，使配置层次更清晰。<br>- 对 TieringOffloadingSpec 明确抛出 `ValueError`，防止在多层 tier 场景下产生不匹配的元数据，保证架构一致性。 |
| **性能影响** | - **正面**：对外部消费者而言，事件携带完整信息后可直接索引，无需二次查询或自行重建 token‑id 映射，降低下游处理延迟。<br>- **负面**：在每一次块 offload 时需要遍历块哈希、切片 token‑id、捕获 LoRA 元信息，导致 CPU 端少量额外算力和内存占用（每个 offload chunk 增加一次 `tuple`/`list` 拷贝）。在高吞吐、极大 batch 场景下可能略微增大 offload 时延。<br>- 通过 `self_describing_enabled` 仅在 KV 事件开启时生效，未开启时保持原有轻量路径。 |
| **安全考虑** | - 事件 payload 中加入 `lora_id`、`lora_name`、`extra_keys` 等字段，可能暴露内部模型适配器信息。若使用公网 KV 事件发布器，需要在部署文档中提醒对这些字段进行脱敏或限制访问。<br>- `ValueError` 在 TieringSpec 中阻止不受支持的配置，避免因错误配置导致不可预期的内存泄漏或错误的事件流。 |
| **可维护性** | - 新增的 `OffloadingEventsTracker` 将状态维护局部化，代码职责单一，便于单元测试（已覆盖）。<br>- 为向后兼容保留占位符路径，且在 `logger.warning_once` 中提供明确提示，降低调试成本。<br>- 引入 `OffloadingKVEventsConfig` dataclass，统一配置来源，减少散落的 `dict.get` 调用。 |
| **兼容性** | - 默认 `self_describing_kv_events=False`，保持对现有用户的兼容。<br>- 当用户显式开启且使用 `TieringOffloadingSpec` 时会抛异常，防止静默错误。<br>- 现有 `tests/v1/kv_connector/unit/offloading_connector/test_scheduler.py` 已去除对旧占位符事件的硬编码检查，确保新旧代码路径均可通过。 |

**⚠️ 潜在风险**：

1. **内存峰值增加**  
   - 每个 offload chunk 在 `record_store` 中创建 `_OffloadEventMetadata`，在大量并发请求（尤其是长上下文）下会暂时占用额外内存，直至对应 `BlockRemoved` 触发。若 `max_tracker_size` 设置过小，可能导致频繁的元数据回收与重复构造，影响性能。

2. **事件顺序误差**  
   - `take_events` 依赖 `self._pending_event_metadata` 与 `OffloadingEvent` 流的对应关系。若外部 `manager.take_events` 产生的事件顺序与 `record_store` 捕获的顺序不一致（例如多线程调度导致提前 eviction），可能出现 “placeholder” 警告或缺失的完整 payload。

3. **不支持的 KV 类型**  
   - 仅针对 **Full‑Attention** 组实现自描述；Sliding‑Window、SSM 仍使用占位符，若用户误以为所有组都支持自描述，可能在下游消费时出现缺失字段。文档需要明确说明。

4. **跨进程/跨节点事件消费**

总体来看，改动实现了对标签化指标的完整支持，逻辑清晰且兼容旧行为。只需注意上表中的细节，确保在高并发或多标签环境下的鲁棒性与性能即可。
