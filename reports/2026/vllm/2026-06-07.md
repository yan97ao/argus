### [DSV4] Decouple DS V4 Sparse MLA Metadata from DS V3.2 (#44699)
**SHA**: `2a983c7` | 🔗 [查看提交](https://github.com/vllm-project/vllm/commit/2a983c79acdbe78fa62e67eaa87ed4bbb7c8257e)

**🎯 变更类型**：架构变更 / 重构  
**⚡ 重要程度**：🔴高  
**📋 变更摘要**：将 DeepSeek‑V4 稀疏‑MLA（DS V4 Sparse MLA）所使用的元数据与 DS V3.2 的实现解耦，新增 `vllm/models/deepseek_v4/sparse_mla.py` 提供独立的 Backend、Metadata 与 Builder；相应地在 NVIDIA、AMD、FlashInfer 等子模块以及后端注册表中改为引用新类，删除原先在 `flashmla_sparse` 中的 DS V3.2 相关实现。  

**🎯 影响范围**：  
- `vllm/models/deepseek_v4/nvidia/flashmla.py`、`flashinfer_sparse.py`、`amd/rocm.py`  
- `vllm/v1/attention/backends/mla/flashmla_sparse.py`（元数据构建逻辑被移除）  
- `vllm/v1/attention/backends/registry.py`（后端导入路径更新）  
- 文档 `docs/design/attention_backends.md`  
- 依赖于 DeepSeek‑V4 稀疏 MLA 的所有模型入口（Deepseek‑V4 系列）  

**🔍 技术洞察**  

- **架构影响**  
  - **解耦**：原先的 `FlashMLASparseMetadata` / `FlashMLASparseBackend` 同时服务 DS V3.2 与 DS V4，导致代码交叉、难以单独演进。新增 `DeepseekV4FlashMLABackend`、`DeepseekV4FlashMLAMetadata` 与 `DeepseekV4FlashMLAMetadataBuilder`，形成 **模型驱动的专属后端**，明确职责，便于后续仅针对 V4 进行功能扩展或修复。  
  - **后端抽象**：新版后端直接继承自 `AttentionBackend`（而非 V3.2 的 `FlashMLASparseBackend`），表明 **V4 不再走通用 FlashMLA 实现路径**，而是使用 DeepSeek‑V4 自己的 `DeepseekV4Attention` 实现。  

- **性能影响**  
  - **保持原有预分配**：压缩比 128 时的 C128A‑topk 缓冲区、压缩‑slot‑mapping 缓冲区仍在 `MetadataBuilder` 中预先分配，以保证 CUDA‑graph 的地址稳定性，未因抽象层的提升而引入额外的运行时拷贝。  
  - **元数据构建路径精简**：V4 现在只走 `DeepseekV4FlashMLAMetadataBuilder`，去除了 V3.2 相关的 `c128a_*` 判断与无用字段，理论上可以略微降低构建时间和显存占用。  
  - **后端实现无变化**：真正的前向计算仍由 `DeepseekV4Attention` 调用原有的 `flash_mla_sparse_fwd`，因此 **算子性能保持不变**。  

- **安全考虑**  
  - 变更仅涉及内部数据结构与导入路径，没有新增外部接口或网络交互，安全面风险极低。  
  - 需要确保 **序列化/反序列化**（如 checkpoint 恢复）仍兼容新旧元数据名称，防止因类路径改变导致加载异常。  

**⚠️ 潜在风险**  

1. **导入路径破坏**：外部项目或自定义插件仍可能引用旧路径  
   `vllm.models.deepseek_v4.nvidia.flashmla.DeepseekV4FlashMLASparseBackend`，会在运行时抛 `ImportError`。  
2. **元数据字段不匹配**：旧的 `FlashMLASparseMetadata` 包含 `fp8_extra_metadata`、`c128a_*` 等字段；新 `DeepseekV4FlashMLAMetadata` 直接复用了这些字段，但如果后续有后端仍期待 `FlashMLASparseMetadata` 的子类，可能出现 **类型检查失败**。  
3. **CUDA‑graph 地址稳定性**：新 Builder 仍依赖预分配缓冲区，但若 `compress_ratio` 或 `topk_tokens` 配置在运行时被修改，可能导致缓冲区大小不匹配，引发 **CUDA kernel 写越界**。  
4. **文档/注册表同步**：`AttentionBackendEnum` 已更新，但旧的字符串路径仍可能在配置文件或环境变量中出现，导致启动错误。  

**💡 关注建议**  

- **兼容层**：在迁移期间提供一个轻量的别名模块，例如在 `vllm/models/deepseek_v4/nvidia/flashmla.py` 中导出旧类名指向新实现，帮助已有用户平滑升级。  
- **测试覆盖**：  
  - 增加针对 **MetaDataBuilder** 的单元测试，覆盖 `compress_ratio=1、2、128` 三种路径，检查 `slot_mapping`、`c128a_*` 缓冲区内容是否符合预期。  
  - 对 **CUDA‑graph** 场景执行长时间跑批，确认地址稳定性没有回归。  
- **序列化兼容**：在 `AttentionMetadata` 的 `__getstate__` / `__setstate__` 中加入旧类名的兼容映射，以免 checkpoint 在升级后无法加载。  
- **文档与示例更新**：在 `docs/design/attention_backends.md` 明确说明 V4 使用的后端路径以及已废弃的 V3.2 相关 API，提供迁移指引。  
- **CI 检查**：在 CI 中加入对 `vllm.registry` 的路径解析测试，确保所有后端字符串均可成功 `importlib.import_module`。  

整体来看，此次 **解耦** 为 DeepSeek‑V4 稀疏‑MLA 的后续演进奠定了清晰的架构基础，若按上述建议完成兼容与测试，风险可控且对性能、可维护性均有正面提升。

- **模型执行层**：`vllm/model_executor/layers/quantization/compressed_tensors/compressed_tensors_moe/*`（新增 RDNA3 MoE 方法），`vllm/model_executor/layers/fused_moe/layer.py`（权重加载分支），`vllm/_custom_ops.py`（对外 API）。  
- **测试套件**：`tests/kernels/quantization/test_rdna3_compile_guards.py`、`tests/kernels/quantization/test_rdna3_moe_w4a16.py`（功能、精度、编译守护）。

## 🔍 技术洞察

### 1. 架构影响
| 维度 | 影响说明 |
|------|----------|
| **后端平台** | 通过 `VLLM_ROCM_HAS_GFX1100` 与 `#ifdef VLLM_ROCM_GFX1100` 完全隔离 RDNA3 代码。非 gfx1100（如 CDNA）仍然走原有 Triton/Marlin 路径，保持向后兼容。 |
| **算子调度** | `moe_gptq_gemm_rdna3` 采用 **单 kernel** 完成路由 -> dequant -> dot -> 原子累加，极大减少 kernel 启动与全局内存写入次数。 `output_topk` 参数进一步让第二层 GEMM 直接写入最终 `[M, hidden]`，省去 `moe_sum` 额外的 reduce 步骤。 |
| **数据布局** | 与 dense RDNA3 kernel 共享相同的 **shuffled int32** 权重量化格式 (`[E, K/8, N]`) 与 **packed zero‑points**。这避免了在 MoE 场景下对每个专家分别转置/重新打包的开销。 |
| **API 扩展** | 在 `vllm._custom_ops` 中增加 `moe_gptq_gemm_rdna3` 包装，并通过 `hasattr` guard 自动在没有该 op 的环境中注入 no‑op 假实现，保证 import 时不抛异常。 |
| **层级融合** | `CompressedTensorsWNA16RDNA3MoEMethod` 直接在 `apply()` 中调用两次 fused kernel，实现 **w1 → activation → w2** 的完整 MoE 前向流，且内部使用了预分配的 buffer，进一步降低显存碎片和 CUDA/HIP malloc 开销。 |

### 2. 性能影响
| 方面 | 预计提升 |
|------|-----------|
| **Kernel 启动次数** | 原先 MoE 需要 **3‑4** kernel launches（路由、dense‑gemm‑1、MoE‑sum、dense‑gemm‑2），现在 **2**（w1 + w2，且 w2 直接完成 `output_topk` reduce）。 |
| **内存访问** | 所有专家的 weight‑scale‑zero‑point 通过 **只读常量** 直接在 kernel 中读取，激活的 `block_a` 采用 LDS（共享内存）缓存，避免了多次全局读写。 |
| **原子累加策略** | 使用 **64‑bit packed CAS** 将 4 个 FP16/BF16 结果一次写回，减少冲突与缓存行争用。 |
| **算子吞吐** | 通过 `THREADS_X=256`、`BLOCK_KN_SIZE=256` 的配置，满足 RDNA3 **8 wave** 并行执行，理论上可达到 **~1.8‑2.0×** 于原 dense‑MoE 实现的算子峰值（在官方 benchmark 中已报告 1.6‑1.9× 的提升）。 |
| **显存占用** | 仅额外保留一次 `w1_out` 与 `act_out` 的预分配 buffer（≈ `M*top_k*N_gate_up`），对比原先 `moe_sum` 需要的中间 buffer，显存消耗略有降低。 |
| **数值误差** | 采用 **split‑K 原子累加**，与 dense‑gemm 在 FP16/BF16 下的误差保持在 **0.1‑0.5** 的绝对容差（已在测试中验证），对模型质量影响可忽略。 |

### 3. 安全考虑
| 风险点 | 说明 |
|--------|------|
| **未编译 guard 失效** | 若 `VLLM_ROCM_HAS_GFX1100` 判定错误，会导致在 CDNA 硬件上编译出 RDNA3 kernel，运行时 `__builtin_amdgcn_fdot2` 等指令不可用，出现 illegal instruction。已通过 **编译守护测试** 检测此类错误。 |
| **原子累加溢出** | 4×FP16/BF16 的累加使用 **64‑bit CAS**，若单个 token 的累计值超过 FP16/BF16 表示范围（≈ 6e4），理论上会出现溢出。实际模型的激活值远低于此阈值，风险极低。 |
| **Zero‑point 合成错误** | GPTQ‑v1 的 `zero_offset=1` 处理已在 kernel 中硬编码，若未来改为不同版本（v2）需同步更新，否则会产生系统性偏差。 |
| **输错参数导致未定义行为** | `block_size_m` 必须是 1、2、4、8 中的合法值，且 `output_topk` 必须为 0 或 `top_k`。库在 C++ 端使用 `TORCH_CHECK` 保护，Python 包装亦未在源码层做额外检查，建议在高层 API 中添加参数校验。 |
| **异常路径** | 当 `expert_id == -1`（专家并行模式）时 kernel 直接 `return`，确保不会写入错误的输出位置；对应的 unit test 已覆盖。 |

## ⚠️ 潜在风险

1. **平台检测失效**  
   - `on_gfx1100()` 依赖 `rocminfo`/`hipDeviceGetAttribute`，在某些容器中可能返回 `False` 而实际硬件支持，导致失去最佳实现。需要在 CI 中显式检测并给出 **fallback warning**。  

2. **Mix‑precision 隐式转换**  
   - 在 `ops.moe_gptq_gemm_rdna3` 调用时，`topk_weights` 被强制 `float32`，但激活可能是 BF16。若未来出现 **AMP**（自动混合精度）策略，需要确认 `float32`→target dtype 的转换不会产生额外的同步点。  

3. **内存对齐**  
   - `BLOCK_KN_SIZE` 与 `THREADS_X` 必须严格相等，且 `block_a` 使用了 `LDS_PAD = 8` 防止 bank 冲突。若未来改动 `THREADS_X`（例如调低至 128 以适配低功耗卡），必须同步更新 `LDS_PAD` 与 `static_assert`，否则可能出现共享内存越界。

### [1/N][KV-Cache Layout Refactor] Refactor DSV4 KV cache config construction (#44454)
**SHA**: `4dcd10e` | 🔗 [查看提交](https://github.com/vllm-project/vllm/commit/4dcd10eb0d223a3ec4b2c96deaf3a48a96c8dcaa)

**🎯 变更类型**：重构（KV‑Cache 布局）  
**⚡ 重要程度**：🟡 中  

**📋 变更摘要**  
- 将 DeepSeek‑V4 KV‑Cache 布局的构建逻辑从「以最大 bucket 大小」的整体计算，改为先把所有层按 page‑size 分桶，再对每个 bucket 的 slot（同一页尺寸下的第 i 个层集合）生成共享的 KV‑CacheTensor。  
- 引入 `_bucket_layers_by_page_size` 辅助函数，实现“页尺寸 → slot → layer‑list”的映射。  
- 重新计算每块所需字节数与块数，去掉了对 `UniformTypeKVCacheSpecs` 的强制假设，提升对混合‑MLA / SWA 组的兼容性。  

**🎯 影响范围**  
- `vllm/v1/core/kv_cache_utils.py`（核心 KV‑Cache 配置路径）。  
- 相关的 `KVCacheGroupSpec`、`UniformTypeKVCacheSpecs`、`KVCacheTensor` 创建流程。  

**💡 关注建议**  
1. **功能完整性**：  
   - 确认 `_bucket_layers_by_page_size` 对所有可能的 `kv_cache_spec`（统一页大小、非统一）均能正确分桶，尤其是含有自定义 `page_size_bytes` 的子组。  
   - 验证 `shared_by` 列表的顺序与原实现保持一致，因为后续可能依赖层次顺序进行块表索引。  

2. **性能与内存**：  
   - 新的块数计算 `available_memory // total_num_bytes_per_block` 与旧版 `layer_tuple_page_bytes * num_layer_tuples` 等价性需在边界条件（如单页大小、极大层数）下做基准测试，防止因整数除法产生的显著块数变化。  
   - `defaultdict` 的使用在大模型多层情形下可能产生额外的临时列表，建议在性能敏感的启动路径加上 `@functools.lru_cache`（若参数可哈希）或显式预分配。  

3. **回归测试**：  
   - 在 CI 中加入 DeepSeek‑V4、SWA、混合‑MLA 场景的 KV‑Cache 配置对比测试，确保 `num_blocks` 与 `KVCacheTensor.size` 与历史行为在相同可用内存下保持预期比例。  
   - 检查 `may_override_num_blocks` 仍能覆盖新计算的 `num_blocks`。  

4. **文档/注释**：  
   - 对 `_bucket_layers_by_page_size` 的返回结构给出更明确的示例，便于后续维护者快速理解 “slot_idx → layer_names” 的含义。  

总体而言，此次重构把 KV‑Cache 布局抽象为更通用的分桶模型，提升了代码可读性与对多种模型布局的适配能力。但需通过充分的单元/集成测试验证在极端配置下的块数与共享张量行为，以防出现意外的内存/性能回退。

3. 调度路径在判断是否可以接受 async KV‑load 时，使用 `reserved_blocks` 进行额外检查，防止新请求抢占正在预填请求所需的块。  
4. 相关单元/端到端测试新增，验证：① 当块资源不足时，后续 async 加载被阻塞而不是导致 wedge；② 资源足够时两个 async 加载均可被接受；③ 预填完成后相应记录被及时清理。  

**🎯 影响范围**  
- `vllm/v1/core/kv_cache_manager.py`、`vllm/v1/core/sched/scheduler.py`：调度与块分配逻辑。  
- 相关测试套件：`tests/v1/core/test_scheduler.py`、`tests/v1/e2e/general/test_mamba_prefix_cache.py`、`tests/v1/kv_connector/unit/test_remote_prefill_lifecycle.py`。  

**💡 关注建议**  
- **开发者**：在新增或修改 async KV‑connector 相关逻辑时，务必通过 `reserved_blocks` 参数保持对在‑flight 预填的感知，防止再次出现资源竞争。  
- **用户**：升级后可在高并发、跨请求共享前缀的场景下获得更稳定的吞吐，若遇到请求频繁被卡在 `WAITING_FOR_REMOTE_KVS`，请检查块池大小是否满足 `max_num_seqs * block_size` 的需求。  

整体而言，此次改动在不改变现有 API 的前提下，通过内部预留机制避免了 deadlock，提升了 async KV‑load 的鲁棒性。
