### [Rust Frontend] Improve scheduler stats logging parity (#47435)
**SHA**: `ec0ffaa` | 🔗 [查看提交](https://github.com/vllm-project/vllm/commit/ec0ffaacc8a6437681b5743cfdcc7841ada6b4d7)

**🎯 变更类型**：功能增强 / 架构变更 / 性能优化  
**⚡ 重要程度**：🔴 高  

**📋 变更摘要**  
1. 为 `EngineCoreClient` 新增 `engine_indices()` 接口，使 Rust 前端能够获取每个 engine 在服务器端的索引，取代原先使用 `engine_count` 的假设。  
2. 完善 `SchedulerStats` 与 `CudagraphStats` 的序列化结构，引入 `CudagraphStats`（原 `CudagraphStat`）并在日志记录路径中加入 CUDA‑graph 统计。  
3. 在 `StatsLogger` 中扩展指标集合：外部前缀缓存、抢占次数、Spec‑Decoding 相关计数、MFU（每 GPU FLOPS/GBps）以及“deferred” 等待原因。  
4. 引入 **非 Prometheus** 的 `SchedulerLogStatsAccumulator`，在日志间隔内聚合原始 DTO（每位置接受率、CUDA‑graph 样本），并在 `run_stats_logger` 中统一抽取、汇总后输出人类可读的日志行。  
5. 在多处使用 `engine_indices` 取代 `engine_count`，确保日志在多‑engine（非连续 IDs）部署场景下保持正确。

**🎯 影响范围**  
- `rust/src/engine-core-client/*`（客户端与协议层）  
- `rust/src/metrics/*`（调度器度量、Accumulator）  
- `rust/src/llm/*`（StatsLogger 启动、日志输出逻辑）  
- 公开 API：`EngineCoreClient::engine_indices`（可能被外部库调用）  
- 依赖该库的 Python 前端（通过 PyO3）在采集统计信息时会获得更丰富的指标。

**🔍 技术洞察**  

| 维度 | 影响 |
|------|------|
| **架构影响** | - 引入 `SchedulerLogStatsAccumulator`（`Arc<Mutex<>>`）在每个 engine 上维护独立的间隔聚合，保持 **锁粒度** 仅在记录时（极短）且在日志消费时统一 drain，基本不破坏现有无锁 Prometheus 家族设计。<br>- `engine_indices` 替代 `engine_count` 使得多‑engine 部署（engine IDs 非 0..N‑1）能够正确映射，提升跨语言（Python ↔ Rust）统计一致性。 |
| **性能影响** | - 记录阶段新增数个原子计数器（`U64Counter`）以及一次 `Mutex` 加锁（`observe_*`），开销极低（计数器已是 `AtomicU64`，锁仅在每次统计点一次，间隔为 10 s）。<br>- 计算 MFU、Spec‑Decoding 统计在日志线程中完成，对主请求路径无影响。<br>- 通过 `log_stats.drain()` 在后台一次性复制数据，避免累加期间的竞争。 |
| **安全考虑** | - 新增 `Mutex` 可能在极端高并发下出现 **poisoning**，代码已使用 `expect` 抛出明确错误，避免沉默错误。<br>- 所有新增结构均实现 `Serialize/Deserialize`，保持与 Python 端相同的 JSON schema，未引入新的外部可控输入。<br>- `engine_indices()` 中 `expect("engine id must encode as u16")` 假设 engine ID 必须可以映射为 `u16`，若未来出现更大 ID 会导致 panic，建议在未来改为安全返回 `Option<u32>`。 |
| **可维护性** | - 新增的统计逻辑与已有 Prometheus 度量分离，代码职责清晰：Prometheus 负责监控系统；Accumulator 负责人类可读日志。<br>- 引入大量常量（如 `WAITING_REASON_DEFERRED`）以及辅助函数（`cache_hit_rate`、`format_position_rates`）提升可读性。<br>- 单元测试覆盖 `cache_hit_rate`、Spec‑Decoding、MFU 以及 `SchedulerLogStatsAccumulator`，降低回归风险。 |
| **兼容性** | - 对外 API 变化：`EngineCoreClient::engine_indices` 新增；原有 `engine_count()` 仍保留，兼容旧代码。<br>- Prometheus metric 名称未变，但新增了多项计数器和 `log_stats` family；旧的监控仪表盘若未更新不会受影响。 |
| **监控价值** | - 现在可以监控 **Spec‑Decoding** 的接受率、草稿吞吐、每位置接受率，帮助研发评估 Spec‑Decoding 效果。<br>- **MFU**（TFLOPS/GBps）提供硬件利用率直接视图，便于调优 GPU 配置。<br>- **Deferred** 等待原因区分，让运维更快定位调度瓶颈。 |

**⚠️ 潜在风险**  
1. **锁竞争**：`SchedulerLogStatsAccumulator.observe_*` 在高频率（每 token）调用时会产生 `Mutex` 锁，极端情况下可能成为瓶颈。虽然目前每次调用均在 CPU‑only 的统计路径，风险低，但在极端高并发（数千并发请求）下需要监控。  
2. **panic risk**：`engine_indices()` 中的 `expect` 假设 engine ID 可映射为 `u16`，若未来扩展到 >65535 的 engine ID，会导致进程崩溃。  
3. **指标膨胀**：新增的 Prometheus 计数器会在长期运行的服务中产生大量时间序列（每个 engine 产生一套），在大规模部署（数百 engine）时需要确保监控后端的压缩/保留策略。  
4. **日志噪声**：在空闲期间仍会输出 DEBUG 日志，若日志级别配置不当，可能淹没关键日志。  
5. **序列化兼容**：`CudagraphStats` 名称变更可能影响依赖旧 JSON schema 的外部工具，需同步文档或保持向后兼容的别名。  

**💡 关注建议**  
1. **监控锁延迟**：在关键路径加入 `tracing::instrument` 或 `metrics::Histogram` 记录 `Mutex` 持有时间，确保不会在生产环境出现热点。  
2. **安全化 engine_indices**：考虑把 `engine_indices()` 改为返回 `Result<Vec<u32>, EngineIdError>`，或在 `expect` 前添加断言并给出友好错误信息。  
3. **仪表盘更新**：在 Prometheus/Grafana 中加入新计数器（`external_prefix_cache_*`、`num_preemptions`、`spec_decode_*`、`estimated_*_per_gpu`）以及 `log_stats` 的聚合指标，确保运维能看到完整视图。  
4. **日志级别策略**：建议默认把 `log_stats_line!` 使用 `info!` 输出，仅在 `debug` 模式下开启详细 Spec‑Decoding/ MFU 行，防止生产日志过大。  
5. **回滚兼容**：若业务方仍使用 Python 前端统计，需要确认 `CudagraphStat` → `CudagraphStats` 的名称变化不会破坏已有解析脚本；可在序列化层面添加 `#[serde(rename = "CudagraphStat")]` 以保持兼容。  
6. **负载测试**：在多‑engine（如 64、128）环境下跑压测，关注 `log_stats` 相关锁的延迟和 Prometheus 报表的渲染时间。  
7. **文档同步**：更新 README / API 文档，说明新暴露的 `engine_indices` 方法及新增的统计指标含义和单位。

### [Rust Frontend] Use enum-backed domain types for engine outputs and structured outputs (#47283)

**🎯 变更类型**：功能增强 / 重构 / 架构变更  
**⚡ 重要程度**：🔴高  
**📋 变更摘要**：  
- 将 *engine‑core 输出* 与 *structured‑outputs 参数* 从原有的宽松 product‑shaped结构体，改为 **枚举 + 结构体** 的领域模型。

- 更新所有使用到这些类型的调用点：核心 `client`、`coordinator`、`mock engine`、单元测试、gRPC/HTTP 路由、结构化输出转换等。  
- 增加错误类型 `InvalidStructuredOutputsParams`，在结构化输出约束不唯一或非法时提前报错。

## 🔍 技术洞察

### 1. 架构影响
| 维度 | 影响 |
|------|------|
| **类型安全** | 通过枚举（`EngineCoreOutputs`、`StructuredOutputConstraint`）强制“一次只能有一种约束”，在编译期即捕获错误，避免运行时的 `assert!` 检查或潜在的混合形状导致的未定义行为。 |
| **分层职责** | 引入 **wire‑type** (`WireEngineCoreOutputs`、`WireStructuredOutputsParams`) 与 **domain‑type** (`EngineCoreOutputs`、`StructuredOutputsParams`) 的双层结构，保持外部协议不变的同时，让业务代码只操作语义化模型。 |
| **错误可定位性** | 新增 `InvalidStructuredOutputsParams`，在反序列化阶段即中止错误请求，错误信息更加明确（“multiple constraints specified …”, “json_object must be true” 等）。 |
| **代码可维护性** | 统一了“从原始 `EngineCoreOutputs` 到语义化枚举”的转换逻辑，消除了原本的 `ClassifiedEngineCoreOutputs::Other` 分支，降低了以后新增输出族的实现成本。 |
| **向后兼容** | 实现 `Serialize`/`Deserialize` for the domain types delegating到内部 wire 类型，保持与已有 Python 前端、msgpack 编码的 1:1 对应，故部署升级不会破坏已有客户端。 |

### 2. 性能影响
| 项目 | 评估 |
|------|------|
| **序列化/反序列化** | 增加了 **一次 `From/TryFrom`** 的转换（在反序列化时执行），但这仅是 **O(1)** 的结构拷贝，几乎可以忽略。真实的网络/模型推理成本远大于此。 |
| **运行时分支** | `EngineCoreOutputs::classify` 现在是 `TryFrom`，在成功路径下直接返回对应枚举，避免了原来 `match` 中的 `Other` 检查。微小的分支代价降低。 |
| **内存布局** | 新增的包装结构仍然使用原有的 `Vec`、`BTreeSet` 等容器，无额外堆分配，整体内存占用不变。 |
| **编译优化** | 由于使用了 `#[derive(EnumAsInner)]` 与 `#[derive(Default)]`，编译器能够更好地进行内联与单态化，潜在的微小提升。 |

### 3. 安全考虑
| 场景 | 改进 |
|------|------|
| **结构化输出参数校验** | 原本在 Python 层通过 validation，Rust 端现在在 `TryFrom<WireStructuredOutputsParams>` 中**强制只能出现单一约束**，防止恶意或错误客户端发送混合约束导致解析错误或后端生成错误的 grammar。 |
| **错误信息泄露** | 新错误仅包含 “invalid structured outputs params: …”，不泄露内部实现细节，安全等级保持。 |
| **异常路径检测** | `EngineCoreOutputs::classify` 现在返回 `Result<Self, Error>`，在出现非法 wire shape 时会返回 `Error::Decode`，调用方统一使用 `?` 传播，避免了未捕获的 `panic!`。 |
| **兼容性风险** | 若外部（Python）客户端仍然发送 **非法混合形状**，现在会在 **Rust 客户端** 报错并关闭连接，防止潜在的安全漏洞传播到后端模型服务。 |

### 4. 可维护性与可扩展性
- **新增约束**（如 future `xml_schema`）只需在 `StructuredOutputConstraint` 中添加枚举变体，并在 `WireStructuredOutputsParams -> StructuredOutputsParams` 的 `try_from` 中映射，无需改动业务代码的核心路径。  
- **新增输出族**（比如 `Metrics`）同理，只需在 `EngineCoreOutputs` 枚举中加入对应结构体并实现 `From`/`Into`。  
- **测试覆盖**：所有涉及 `EngineCoreOutputs` 的测试已更新为使用新包装，并继续通过 **msgpack round‑trip** 验证兼容性，保证未来改动不会破坏协议。

## ⚠️ 潜在风险

| 风险点 | 说明 | 缓解措施 |
|--------|------|----------|
| **API 破坏** | `EngineCoreOutputs` 从结构体变为枚举，外部直接构造该类型的代码（如果有）将编译失败。 | 更新所有内部调用（已在本次 PR 中完成），若外部库自行使用此结构体，需要升级至新版本并使用 `.into()` 或对应构造函数。 |
| **序列化不匹配** | 只要 `Serialize`/`Deserialize` 实现正确，外部 Python 仍使用相同 msgpack 结构，但若实现细节（字段顺序）修改，可能导致不兼容。 | 通过单元测试 (`python_msgpack_fixtures_match_rust_encoding`) 验证，确保字段顺序与 Python 完全一致。 |
| **错误传播** | `TryFrom` 在解析结构化输出时返回 `InvalidStructuredOutputsParams`，若上层未 `?` 处理，将产生 `Result::Err` 丢失。 | 所有调用点已改为 `?` 传播；审查新增路径确保错误被上抛并在 `client` 层转为 `Error::Decode`。 |

### [Bugfix] Prevent padding placeholders from reaching embeddings (#47029)
**SHA**: `00eb7ce` | 🔗 [查看提交](https://github.com/vllm-project/vllm/commit/00eb7cefa31e32585fb419db5bb945f6a42480fe)

**🎯 变更类型**：Bug 修复  
**⚡ 重要程度**：🟡 中  

**📋 变更摘要**  
本次提交修复了在 Speculative Scheduler（投机调度）路径中，`-1` 作为占位符（padding placeholder）被错误地直接送入词嵌入层导致的异常。新增两条单元测试验证：  
1. `InputBatch` 在 `update_req_spec_token_ids` 后保持 `-1` 原样写入 `spec_token_ids` 与 CPU token 缓冲区。  
2. `GPUModelRunner._calc_spec_decode_metadata` 在计算 draft‑token 元数据时保留 `-1`，并在嵌入查找前统一 clamp 为 `0`。  

核心改动位于 **vllm/v1/worker/gpu_model_runner.py**，在 `_preprocess` 中加入：

```python
if self.speculative_config is not None:
    self.input_ids.gpu[:num_input_tokens].clamp_(min=0)
```

**🎯 影响范围**  
- `GPUModelRunner._preprocess`（embedding 前的输入预处理）  
- Speculative decode 相关的 `InputBatch`、`_calc_spec_decode_metadata`  
- 受 `speculative_config` 启用时影响的所有模型推理路径（GPU/CPU）  

**💡 关注建议**  
1. **文档/注释**：在 `GPUModelRunner._preprocess` 处补充注释，说明为何仅在此处 clamp，防止后续误将占位符视为合法 token。  
2. **兼容性检查**：确认其它使用 `self.input_ids`（如 KV‑cache 填充、跨‑GPU 复制）未依赖 `-1` 的原始值，避免出现意外行为。  
3. **性能评估**：clamp 操作成本极低，但建议在大批量推理时跑一次基准，确保不会引入不可感知的开销。  
4. **完整回归**：执行全套 vllm 测试（包括 multi‑modal、tensor‑parallel、pipeline‑parallel 场景），确保占位符在所有并行配置下仍被正确处理。  
5. **后续优化**：若未来计划在 embedding 前对占位符做更细粒度的掩码处理，可考虑在 `Embedding` 层内部直接支持负索引，而非在前置 clamp。  

总体而言，此次修复消除了投机调度期间因占位符导致的异常，影响范围局限在投机解码路径，风险低，建议合并后立即跑全量 CI。
