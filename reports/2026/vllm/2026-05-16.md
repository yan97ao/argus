#### 🔴 高重要度变更 (2)

### [Experimental] Breakable CUDA graph (#42304)
**SHA**: `8a56da3` | 🔗 [查看提交](https://github.com/vllm-project/vllm/commit/8a56da3845270837424ef4b7ee83ca97a7883025)

**🎯 变更类型**：功能增强 / 架构变更 / 实验性安全修正  
**⚡ 重要程度**：🔴 高  
**📋 变更摘要**  
- 引入 **Breakable CUDA Graph** 实验特性：在运行时使用 CUDA stream capture 并在自定义算子处“打断”捕获，转为 eager 执行后继续捕获，实现 **分段图**（graph + eager segment）机制。  
- 新增环境变量 `VLLM_USE_BREAKABLE_CUDAGRAPH`，开启后会 **关闭 torch.compile 编译链**（等价于 `-cc.mode=none`），并在多处层实现（MLA、DeepSeek v4、Sparse Attn Indexer、ROCM aiter）上通过 `@eager_break_during_capture` 装饰器自动完成 “eager break”。  
- 实现 `BreakableCUDAGraphCapture`（线程局部、支持嵌套检测、异常安全）与 `BreakableCUDAGraphWrapper`（兼容原 `CUDAGraphWrapper` 接口），并在模型加载、profile、清理等路径中加入对应的 wrapper。  
- 新增完备的单元测试覆盖捕获、重放、嵌套装饰器、异常清理等场景。  

---

### 🎯 影响范围
- **核心运行时**：`vllm.compilation.breakable_cudagraph`、`vllm.config.vllm`、`vllm.envs`、`vllm/v1/worker/gpu_model_runner.py`、`vllm/v1/cudagraph_dispatcher.py`。  
- **模型层实现**：`mla_attention.py`、`deepseek_v4_attention.py`、`sparse_attn_indexer.py`、`rocm_aiter_mla_sparse.py`（所有被标记为可中断的自定义算子）。  
- **测试套件**：新增 `tests/v1/cudagraph/test_breakable_cudagraph.py`。  

---

### 🔍 技术洞察

| 维度 | 影响分析 |
|------|----------|
| **架构影响** | - **新抽象层**：`BreakableCUDAGraphCapture` 负责捕获/分段管理，`BreakableCUDAGraphWrapper` 替代原 `CUDAGraphWrapper`，实现 **运行时分段图**。<br>- **线程局部**：通过 `threading.local()` 实现每线程独立的 capture，上层代码（如 `gpu_model_runner`）只需感知 wrapper，保持向后兼容。<br>- **配置分支**：`VLLM_USE_BREAKABLE_CUDAGRAPH` 在 `VllmConfig.__post_init__` 里强行关闭 torch.compile，防止两套编译路径冲突。 |
| **性能影响** | - **正向**：避免了一次性的大规模 `torch.compile` 编译开销，捕获过程只在第一次 forward 时完成，后续调用仅执行 `CUDAGraph.replay` + 若干 eager segments（函数调用开销极低）。<br>- **负向**：每次 **eager break** 会在捕获流上执行一次额外的 kernel 调用（如 attention、kv‑cache 处理），并在 capture 期间引入 `graph.begin/end`、同步 (`offloader.sync_prev_onload`) 以及 weak‑ref 包装，可能带来微秒级额外延迟。<br>- **内存**：使用 **全局 cuGraph pool**，同一 batch‑descriptor 的多段图会占用多块 pool 内存；在 DEBUG 模式下会检查 tensor address 稳定性，防止因地址变化导致图失效。|
| **安全/可靠性** | - **异常安全**：`BreakableCUDAGraphCapture.__exit__` 在异常情况下仍会调用 `_end_segment` 并恢复 thread‑local 状态，防止 “残留 capture”。<br>- **嵌套捕获检测**：在 `__enter__` 抛 `RuntimeError`，防止不支持的嵌套导致图状态破坏。<br>- **弱引用机制**：通过 `weak_ref_tensor(s)` 防止捕获段持有强引用导致 pool 内存无法回收。<br>- **环境变量**：仅在显式开启时生效，默认 `False`，避免生产环境意外开启。 |
| **可维护性** | - **代码量**：新增约 800 行（核心实现 + 测试），保持单文件实现，易于审查。<br>- **装饰器使用**：要求 **外层** 装饰 `@eager_break_during_capture`，文档已说明；若误放在内部装饰器内，可能把副作用写入图中，增加调试成本。<br>- **兼容性**：`BreakableCUDAGraphWrapper` 仍实现 `__getattr__` 透明转发，原有 `CUDAGraphWrapper` 调用路径不受影响。 |
| **平台/依赖** | - 仅在 **CUDA** 环境下可用（测试 `torch.cuda.is_available()`），在 CPU/ROCm 上不触发。<br>- 与 **torch.compile** 完全解耦，故在不支持 `torch.compile` 的旧版 PyTorch 上也能使用（前提 CUDA 可用）。 |

---

### ⚠️ 潜在风险

1. **性能回退风险**  
   - 在高度算子密集的模型（如大模型的多层 attention）中，频繁的 eager break 可能导致整体 latency 增加（每段图的切换、同步、函数调用成本累计）。  
   - 对于 **ubatching** 场景（`parallel_config.use_ubatching=True`），当前实现 **不** 启用 breakable 图，可能导致用户在开启该实验特性后仍回退到传统路径，产生预期不一致。

2. **线程局部泄漏**  
   - 若用户在捕获上下文外手动修改 `BreakableCUDAGraphCapture._tls.active`（极端情况），或在捕获期间创建子线程并尝试使用同一 capture，可能导致 “active capture not cleared” 或 “nested capture” 错误。

3. **弱引用失效**  
   - 捕获期间返回的 tensors 被弱引用包装；如果用户在 capture 完成后仍保有对原张量的强引用，弱引用可能在 `replay` 时指向已被回收的内存，出现 `NoneType` 错误。

4. **兼容性冲突**  
   - `VLLM_USE_BREAKABLE_CUDAGRAPH=1` 会强制 `compilation_config.mode = NONE`，这会关闭所有基于 `torch.compile` 的优化，可能对已有依赖 torch.compile 的插件或自定义编译器产生意外行为。

5. **调试难度**  
   - `DEBUG` 模式下的地址检查会在每次 `replay` 抛 AssertionError；在生产环境若误开启 `VLLM_LOGGING_LEVEL=DEBUG`，可能导致异常中断推理。

6. **资源回收**  
   - `profile_cudagraph_memory` 现在需要遍历两套 wrapper 集合；若未来再添加第三套实现，可能忘记同步更新导致统计不完整或内存泄漏。

### fix: add API key authorization to /v2 endpoints (#42594)
**SHA**: `39c67d7` | 🔗 [查看提交](https://github.com/vllm-project/vllm/commit/39c67d714ef091df1533181bdc3df82dc9ac3e07)

**🎯 变更类型**：安全修复 / 功能增强  
**⚡ 重要程度**：🔴高  

**📋 变更摘要**  
- 为 vLLM 的 HTTP 服务新增对 `/v2`（Cohere Embed / Rerank）以及 `/inference` 前缀的 API‑Key 鉴权，原本仅在 `/v1` 前缀下生效。  
- 通过在 `AuthenticationMiddleware` 中引入 `GUARDED_PREFIX = ("/v1", "/v2", "/inference")`，统一在这些路径上执行 token 校验。  
- 文档与单元测试同步更新，确保新行为得到验证并在运行时返回 401/200 状态码。  

**🎯 影响范围**  
- `vllm/entrypoints/openai/server_utils.py`（AuthenticationMiddleware）  
- `docs/usage/security.md`（安全指南）  
- `tests/entrypoints/serve/instrumentator/test_optional_middleware.py`（新增针对 `/v2` 的鉴权测试）  
- 所有基于 vLLM HTTP 接口的部署与客户端代码，尤其是使用 `/v2/*` 与 `/inference/*` 端点的用户。  

**🔍 技术洞察**  

- **架构影响**  
  - 鉴权中间件的过滤规则从单一的 `"/v1"` 扩展为元组 `("/v1", "/v2", "/inference")`，对路由层面无侵入式改动，仅在 `__call__` 方法的路径判断处加入一次 `startswith` 检查。  
  - 统一了不同前缀的安全策略，减少了因遗漏某路由而导致的未授权风险。  

- **性能影响**  
  - `startswith` 对元组的实现与单字符串几乎等价，开销在微秒级别，对高并发请求的总体吞吐几乎不产生可感知的影响。  
  - 仅在受保护的路径（约占全部请求的少数）触发 token 验证，额外的 header 读取与比较对 CPU、内存占用均可忽略不计。  

- **安全考虑**  
  - 将 `/v2/*` 与 `/inference/*` 列入必鉴权范围，显著降低了未授权访问嵌入、重排、生成等敏感功能的风险。  
  - 仍保留对 `OPTIONS`、`/health`、`/metrics` 等健康检查类路径的免鉴权，以免影响监控与弹性探针。  
  - 未改动错误返回体，仅统一返回 `{"error":"Unauthorized"}`，符合现有 API 错误格式。  

**⚠️ 潜在风险**  

1. **兼容性破坏**  
   - 已有使用 `/v2` 或 `/inference` 端点且未提供 `--api-key` 或 `Authorization` 头的部署将在升级后收到 401，导致业务直接不可用。  
2. **路径误匹配**  
   - `startswith(GUARDED_PREFIX)` 会对所有以 `/v2`、`/v1`、`/inference` 为前缀的路径生效，若未来引入类似 `/v2beta`、`/v1extra` 等新路由，可能无意中被纳入鉴权范围，需在后续扩展时审慎检查。  
3. **文档/代码同步**  
   - 部分文档仍保留 “Unprotected Endpoints” 列表，若未同步更新可能导致用户误解。  
4. **中间件顺序依赖**  
   - 该中间件假设在路由解析之前执行，若在用户自定义的中间件链中被错误放置，可能导致鉴权失效。  

**💡 关注建议**  

- **升级指引**：在发布说明中明确提醒使用 `/v2` / `/inference` 的用户需配置 `--api-key` 或相应环境变量，否则请求会被拒绝。  
- **回退方案**：如果业务确实需要临时关闭鉴权，可在启动脚本中移除 `--api-key` 参数或在代码层面提供可选的 `--disable-v2-auth` 标志（可考虑后续实现）。  
- **路径审计**：在下一次路由扩展时，审计所有新加入的路由是否应被纳入 `GUARDED_PREFIX`，避免意外拦截。  
- **监控**：在监控平台为 `/v2/*`、`/inference/*` 的 401 响应添加告警，以快速捕获因缺失 token 导致的业务中断。  
- **文档完善**：确保 `docs/usage/security.md` 中的 “Unprotected Endpoints” 章节已剔除所有已受保护的路径，保持与实现一致。  
- **测试覆盖**：新增对误匹配路径（如 `/v2beta/status`）

### [ROCm][CI] Removed problematic command override mechanism (#42807)
**SHA**: `4db300e` | 🔗 [查看提交](https://github.com/vllm-project/vllm/commit/4db300e95fd29f5b1a4a7c34f4fbe91b7e9abb24)

**🎯 变更类型**：其它（CI 脚本重构）  
**⚡ 重要程度**：🟡 中  

**📋 变更摘要**  
此次提交在 `.buildkite/scripts/hardware_ci/run-amd-test.sh` 中删除了 `apply_rocm_test_overrides` 及其完整的 ROCm 专属 pytest 指令重写逻辑。原本通过在 CI 脚本里追加 `--ignore`、`-k` 等参数，屏蔽在 AMD/ROCm 硬件上尚未支持或表现异常的测试用例；现在这些覆盖被完全移除，CI 将直接执行全部原始 pytest 命令。

**🎯 影响范围**  
- **CI 流水线**：AMD/ROCm 设备的持续集成测试。  
- **测试套件**：`models/`, `kernels/*`, `entrypoints/*` 等目录下的众多测试用例。  
- **开发者本地调试**：若依赖 `run-amd-test.sh` 进行本地 ROCm 测试，同样会受到影响。

**💡 关注建议**  
1. **监测 CI 结果**：合并后首批 AMD CI 运行可能出现大量失败，需及时捕获并评估哪些测试真的不兼容 ROCm。  
2. **逐步迁移**：建议将不兼容的测试标记为 `@pytest.mark.skipif` 或使用 `-k`/`--ignore` 在 pytest 配置文件中声明，而非通过脚本硬编码，便于维护和跨平台统一。  
3. **文档更新**：README/CI 文档应说明 ROCm 环境现在会运行全部测试，若仍需临时屏蔽，请在 `run-amd-test.sh` 中自行添加相应覆盖。  
4. **回滚机制**：若部分关键测试仍无法在 ROCm 上通过，可考虑在脚本中保留可选的 `apply_rocm_test_overrides` 开关（通过环境变量控制），避免因误删导致 CI 长时间阻塞。  

总体而言，此改动简化了脚本逻辑，去除了潜在的双引号/转义问题，但也把先前的兼容性保护全部剥离，需在后续确保相应测试在 ROCm 上能够通过或采用更可维护的排除方式。

### [LoRA][Bugfix] Dedup LoRA wrapping for modules referenced from multiple attribute paths (MoE gate) (#42757)
**SHA**: `32b7177` | 🔗 [查看提交](https://github.com/vllm-project/vllm/commit/32b7177909d1c9928bcedd81de7de5a1fa21d2b3)

**🎯 变更类型**：Bug 修复（LoRA 包装去重）  
**⚡ 重要程度**：🟡 中  

**📋 变更摘要**  
1. 在 `LoRAModelManager` 中新增 `wrapped_by_id` 用于记录已经被 LoRA 包装的底层模块实例。  
2. 当同一模块在模型中出现多条属性路径（如 MoE gate 在 `moe.gate` 与 `moe.runner.gate`），会复用已有的 wrapper 并把别名属性指向同一实例，避免二次包装。  
3. 对 `lm_head` 做特殊排除，防止其因别名（如 `embed_tokens`）而被误删，保证 logits‑processor 仍能被正确包装。  
4. 新增对应的单元测试，验证去重行为以及 `lm_head` 的例外逻辑。

**🎯 影响范围**  
- `vllm/lora/model_manager.py`（核心包装逻辑）  
- `vllm/lora/layers.py` 中的 `BaseLayerWithLoRA`、`LogitsProcessorWithLoRA`  
- 相关测试文件 `tests/lora/test_lora_manager.py`  

**💡 关注建议**  
1. **代码可读性**：`wrapped_by_id` 与 `existing_wrapper` 的注释已较完整，可再将 “lm_head 例外” 的判断抽成专门函数，提高可维护性。  
2. **性能**：`wrapped_by_id` 使用 `id` 作为键，查找 O(1)；但在极大型模型中仍会遍历所有子模块，若未来加入增量加载，考虑在首次包装后缓存路径映射。
