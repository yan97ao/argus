### [Kernel][ROCm] Native W4A16 kernel for AMD RDNA3 (gfx1100) — fp16 + bf16 (#41394)
**SHA**: `0cff074` | 🔗 [查看提交](https://github.com/vllm-project/vllm/commit/0cff0741ff8853beb2cae862c0daa8133cb1aa43)

**🎯 变更类型**：功能增强 / 性能优化 / 重构 / 架构变更  
**⚡ 重要程度**：🔴高  

**📋 变更摘要**  
- 为 AMD RDNA3（gfx1100/1101/1102）GPU 新增原生 **W4A16 GPTQ**（4‑bit 权重 + fp16/bf16 激活）算子。  
- 在 CMake 中检测 `gfx1100` 并编译 `q_gemm_rdna3.cu`、`q_gemm_rdna3_wmma.cu` 两套实现（标量+WMMA）。  
- 通过 `torch.ops._rocm_C.gptq_gemm_rdna3` / `gptq_gemm_rdna3_wmma` 向 Python 暴露。  
- 在 kernel 选择链路中把 **RDNA3W4A16LinearKernel** 注册为 ROCm‑优先实现，若平台不满足则回退到 Triton 实现。  
- 新增 dequant 化工具 `qdq_4_rdna3.cuh`、原子‑CAS‑packed 写回、fp16 精确版、bf16‑to‑fp32 dequant、wmma‑路径的 K‑split、双缓冲等多项微调。  
- 完整单元测试覆盖功能正确性、bias、zero‑point 合成、以及 kernel 选择逻辑。  

**🎯 影响范围**  
- **核心模块**：`csrc/rocm/*`（新 CUDA/HIP kernel）、`vllm/_custom_ops.py`、`vllm/model_executor/kernels/linear/*`（kernel 注册与调度）。  
- **平台**：仅在 ROCm 环境且 GPU 架构包含 `gfx1100`（RDNA3）时生效；其它 AMD GPU（CDNA、MI300）会自动回落到原有 Triton 实现。  
- **使用路径**：所有使用 **4‑bit 量化 + bf16/fp16** 推理的模型（如 Qwen、Llama‑GPTQ）将在满足条件的机器上走新路径。

## 🔍 技术洞察

### 1. 架构影响
| 维度 | 说明 |
|------|------|
| **硬件绑定** | 新 kernel 使用 RDNA3 专有指令 `v_dot2_f32_f16`、`v_wmma_f32_16x16x16_{f16,bf16}_w32` 以及 `global_atomic_cmpswap_b64`，只能在 `gfx1100` 系列上编译通过。 |
| **代码组织** | 为避免 HIP 编译器跨 TU 优化冲突，将 WMMA 实现单独放在 `q_gemm_rdna3_wmma.cu`。同时保留空 stub 以确保在非‑RDNA3 编译时符号可链接。 |
| **库依赖** | 通过 CMake 条件 `VLLM_ROCM_HAS_GFX1100` 向 `VLLM_GPU_ARCHES` 中匹配 `gfx1100` 时自动加入源码，其他 ROCm 架构不受影响。 |
| **参数布局** | 采用与 Exllama 相同的 **4‑bit 行/列打包**（K/8 × N）并保持 `qzeros`、`scales` 与激活 dtype 对齐，保证与已有 checkpoint 格式兼容。 |
| **注册/选择** | 在 `vllm/model_executor/kernels/linear/__init__.py` 中把 `RDNA3W4A16LinearKernel` 插入 ROCm 列表的首位，使其优先于 Triton、Marlin 等实现。 |

### 2. 性能影响
| 场景 | 关键优化 | 预期收益 |
|------|----------|----------|
| **fp16 标量路径** | - 直接写入 `half/ bf16` 输出 via 64‑bit CAS（原子‑加）<br>- 使用 `v_dot2_f32_f16` 进行 FP32 累加，避免 4‑bit → FP32 中间缓冲与 `memset` | 减少 **M×N×4 B** 的临时 fp32 缓冲，降低内存带宽占用，提升 1–2×（取决于 K 大小）。 |

| **整体** | 通过去除 fp32‑scratch、原子‑CAS 合并、wmma 加速，预计在 RDNA3 GPU 上 **整体推理 latency** 可降低 **30‑45 %**（视模型大小而定）。 |

### 3. 安全考虑
| 项目 | 说明 |
|------|------|
| **内存访问** | 所有 kernel 在入口处均做了 `if (n >= size_n) return;`、`if (out_m >= size_m) continue;` 等边界检查；原子‑CAS 仅在 8‑字节对齐的地址上执行，符合 `global_atomic_cmpswap_b64` 要求。 |
| **类型转化** | 使用 `__builtin_memcpy` 实现 bit‑cast，避免未定义的行为；`f32_to_bf16_no_canon` 手动截断，防止 HIP 编译器在 NaN‑canon 化路径插入额外指令导致数值差异。 |
| **未定义行为** | 依赖 `__float_as_uint`、`__uint_as_float` 对齐转换，只要输入合法（fp16/bf16、int32）即安全。若用户误传非 8‑倍对齐的 `N`，`can_implement` 会拒绝，防止运行时越界。 |
| **并发原子** | K‑split 采用 32‑bit CAS（两列共享），在同一 wave 内通过 `shfl_xor` 配对，避免同 block 内的冲突；跨 block 的 CAS 只在 `gridDim.z ≤ 4` 时出现，冲突概率极低且符合模型正确性（加法是可交换的）。 |

### [9/n] Migrate attention and cache kernels to torch stable ABI (continued)  (#43717)
**SHA**: `22a5864` | 🔗 [查看提交](https://github.com/vllm-project/vllm/commit/22a58640b4563f5945aa2052e9e61d425351588d)

**🎯 变更类型**：架构变更 / 重构 / 安全修复  
**⚡ 重要程度**：🔴高  

**📋 变更摘要**  
本次提交把注意力（paged_attention）以及缓存（cache）相关的 CUDA/Kernels 从使用 PyTorch 内部非公开 API 的实现迁移到 **torch stable ABI**（`torch::stable::Tensor`、`torch::stable::accelerator::DeviceGuard`、`get_current_cuda_stream` 等）。为此：
- 将源码移动至 `csrc/libtorch_stable/`，并统一使用相对路径的头文件。  
- 替换所有 `torch::Tensor`、`ATen`/`c10` 相关的内部调用为对应的 Stable API。  
- 更新 CMakeLists 将这些文件加入 `VLLM_STABLE_EXT_SRC`，并以 `_C_stable_libtorch` 重新注册扩展。  
- 为缓存相关功能单独声明了 `_C_cache_ops` 库，并把 CPU‑only 批量 swap 实现迁移到 CPU 实现。  
- 将原有的 `TORCH_CHECK` 替换为 `STD_TORCH_CHECK`（Stable 版），并使用 `mutable_data_ptr`、`const_data_ptr` 取代 `data_ptr`，以兼容只读/可写指针的区别。  

**🎯 影响范围**  
- **attention子模块**：`csrc/libtorch_stable/attention/*`（`paged_attention_v1/v2`、`merge_attn_states`、`attention_kernels`、`attention_utils`）  
- **cache子模块**：`csrc/libtorch_stable/cache_kernels.*`, `cache_kernels_fused.*`，以及相关工具文件（`torch_utils.h`、`dispatch_utils.h`、`torch_vec_utils.cuh`）  
- **quantization子模块**：FP8、FP4 量化工具的头文件路径同步更新。  
- **构建系统**：`CMakeLists.txt` 中 `VLLM_STABLE_EXT_SRC` 与 `VLLM_EXT_SRC` 的划分。  
- **Python层**：新增/修改 `_C_stable_libtorch` 与 `_C_cache_ops` 两个 TorchScript 库的注册入口。

## 🔍 技术洞察  

### 1. 架构影响  
| 维度 | 具体变化 | 影响 |
|------|----------|------|
| **API层次** | 从 `torch::Tensor`（内部实现）切换到 `torch::stable::Tensor`（公开、跨版本兼容） | 兼容性大幅提升，后续升级 PyTorch（≥2.4）时不易因内部结构变更导致编译/运行时错误。 |
| **模块划分** | 将所有与缓存、注意力相关的 CUDA kernel 放入 `libtorch_stable`，并通过 `_C_stable_libtorch` 统一导出；缓存单独用 `_C_cache_ops` | 代码组织更清晰，CPU‑only 实现（如 `swap_blocks_batch`）可以独立加载，降低了 CUDA 依赖的耦合度。 |
| **CMake构建** | 原 `csrc/*.cu` 从 `VLLM_EXT_SRC` 移动到 `VLLM_STABLE_EXT_SRC`；新增 `list(APPEND VLLM_STABLE_EXT_SRC ...)` | 只在启用 Stable ABI 时编译对应文件，避免在不需要的构建配置中拉入不兼容代码。 |
| **符号冲突** | 旧实现（仍保留在原路径）与新实现共存 | 需要确保 `torch::stable::` 与原 `torch::` 命名空间不发生链接冲突；若两套实现同时被加载，可能出现二义性调用。 |
| **注册入口** | `TORCH_LIBRARY` 改为 `STABLE_TORCH_LIBRARY`、`STABLE_TORCH_LIBRARY_IMPL`，并使用 `REGISTER_EXTENSION(_C_stable_libtorch)` | 通过 Stable ABI 专属的 registration 宏，保证仅在对应库被加载时才注册符号。 |

### 2. 性能影响  
| 方面 | 可能的正向影响 | 潜在负向影响 |
|------|----------------|--------------|
| **DeviceGuard / Stream 获取** | 使用 `torch::stable::accelerator::DeviceGuard` 与 `get_current_cuda_stream` 与原实现等价，几乎无额外开销。 | `mutable_data_ptr`、`const_data_ptr` 的额外检查（`STD_TORCH_CHECK`）在 Release 模式下会被编译掉，影响可忽略。 |
| **内存访问** | 代码改为显式使用 `mutable_data_ptr`/`const_data_ptr`，可避免不必要的 `const_cast`，提升编译器优化机会。 | 若误将只读指针用于写操作（如 `reinterpret_cast<T*>(tensor.data_ptr())` 替换为 `mutable_data_ptr`）导致未对齐写入，可能出现 subtle bug。 |
| **批量 memcpy** (`swap_blocks_batch`) | 迁移到 CPU 实现后仍调用底层 `cuMemcpyBatchAsync`/`hipMemcpyBatchAsync`，不受 Stable ABI 影响。 | 若在 CUDA 运行时未能正确检测到 `USE_ROCM` 宏导致走到错误分支，可能失去批量提交的性能优势。 |
| **分支判断** | `DISPATCH_BY_KV_CACHE_DTYPE` 仍基于 `scalar_type()`，保持原有 JIT‑dispatch 结构。 | 重新包装后可能出现模板实例化遗漏（未覆盖全部 `scalar_type`），导致在少数 datatype（如 `Float8_e5m2`）下回退到慢路径。 |

### 3. 安全考虑  
- **稳定 ABI** 本身即是安全强化：避免依赖内部实现细节，降低因 PyTorch 版本升级导致的 **未定义行为**（UB）或 **内存越界**。  
- 替换 `TORCH_CHECK` 为 `STD_TORCH_CHECK`：这些宏在 **Release** 编译时会被剔除，必须保证所有关键前置条件在 Debug/测试阶段能被捕获。  
- 使用 `reinterpret_cast` 直接操作原始指针仍然是 **unsafe**（对齐、类型相容性），但这在原代码中已经存在，迁移未改变该风险。  
- 新增 `std::optional<torch::stable::Tensor>` 参数时，仍旧使用 `has_value()` 再 `value()` 访问，需要在 Release 编译时确认不产生未检查的空值访问。
