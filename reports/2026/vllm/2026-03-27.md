### fix(security): Add VLLM_MAX_N_SEQUENCES environment variable and enforce limit (#37952)
**SHA**: `b111f8a` | 🔗 [查看提交](https://github.com/vllm-project/vllm/commit/b111f8a61f100fdca08706f41f29ef3548de7380)

**🎯 变更类型**：安全修复 / 功能增强  
**⚡ 重要程度**：🔴 高  

**📋 变更摘要**  
此提交在 vLLM 中新增 `VLLM_MAX_N_SEQUENCES` 环境变量（默认 `16384`），并在 `SamplingParams` 与 OpenAI 接口请求解析时强制限制 `n`（每次请求生成的独立输出序列数）。当 `n` 超过该阈值时，抛出 `ValueError` 并拒绝请求，从而防止恶意或误配导致的内存、CPU、GPU 资源耗尽。文档、单元测试同步更新，确保行为可验证。

**🎯 影响范围**  
- `vllm.envs`：新增环境变量定义与默认值获取。  
- `vllm.sampling_params`：在 `SamplingParams._verify_args` 中加入上限检查。  
- OpenAI 兼容层（`ChatCompletionRequest` 等）：`to_sampling_params` 依赖的校验逻辑受到新限制的约束。  
- 文档 `docs/usage/security.md`：加入资源限制章节。  
- 测试套件：新增针对默认上限、定制上限、异常大值的完整覆盖。  

**🔍 技术洞察**  

- **架构影响**  
  - 环境变量的统一管理仍通过 `vllm.envs`，保持现有配置读取模式，无需对启动流程或插件系统做额外改动。  
  - 只在 `SamplingParams` 层做校验，属于业务层防护，未侵入底层调度、执行器或 GPU 资源管理，保持模块解耦。  

- **性能影响**  
  - 读取一次环境变量并在对象创建时比较整数，开销极低（纳秒级），对高并发路径几乎无影响。  
  - 通过提前拒绝超大 `n` 请求，显著降低潜在的内存分配、显存占用和 CPU 计算，从而提升整体系统的稳健性。  

- **安全考虑**  
  - 直接防御了 CVE‑related 的 DoS 攻击向量：攻击者可在 `/v1/completions` / `/v1/chat/completions` 通过极大 `n` 消耗服务器资源。  
  - 环境变量提供可操作的防护阈值，使运维人员能够根据实际硬件容量与业务需求灵活调节，符合“深度防御”原则。  

**⚠️ 潜在风险**  

1. **兼容性冲突**  
   - 部分已有使用场景（内部基准测试、研究实验）可能依赖 `n` 超过 16384；部署后会立即报错，需要手动调高阈值或改造代码。  
2. **环境变量解析错误**  
   - 若用户误将 `VLLM_MAX_N_SEQUENCES` 设置为非整数字符串，`int()` 转换将在 `envs` 初始化阶段抛出 `ValueError`，导致服务启动失败。  
3. **缓存失效**  
   - `vllm.envs` 使用 `functools.lru_cache`（或类似）缓存属性值；在运行时更改 env 变量后若未清除缓存，旧值仍会生效。测试已手动 `cache_clear`，但生产环境若依赖热重载，需要注意。  
4. **默认阈值偏大**  
   - 16384 对于显存 8 GB 以下的机器仍可能触发 OOM，尤其在多用户共享场景下。默认值虽然兼容历史行为，但在公共云部署时建议显式调低。  

**💡 关注建议**  

- **运维**  
  - 在生产环境（尤其是公开 API）显式设置 `VLLM_MAX_N_SEQUENCES`（如 `64`、`128`），并配合反向代理的请求体大小与速率限制。  
  - 在容器或系统启动脚本中加入环境变量检查，确保其为合法正整数。  

- **开发者**  
  - 若业务确实需要大于默认上限的 `n`，请在启动脚本中提前设置 `VLLM_MAX_N_SEQUENCES`，并在代码中捕获 `ValueError` 给用户友好的错误提示。  
  - 注意在单元测试或本地调试时，使用 `monkeypatch`/`patch.dict` 清理或重设环境变量，并在修改后调用 `envs.__getattr__.cache_clear()`（或提供统一的 `reload_envs()` 接口）防止缓存残留。  

- **监控**  
  - 增加 `n` 参数的监控指标（如请求分布直方图），配合阈值报警，及时发现异常激增的请求。  

- **文档与社区**  
  - 将本次安全修复写入发行说明（Release Notes），并在 FAQ 中提醒用户在公开部署时务必配置该阈值。  

通过上述措施，可在最小化兼容性影响的同时，显著提升 vLLM 对资源滥用的防御能力。

### [kv_offload+HMA][7/N]: Support register_kv_caches for hybrid models (#37853)
**SHA**: `7cc302d` | 🔗 [查看提交](https://github.com/vllm-project/vllm/commit/7cc302dd87cba546e1cf9b9967317437142438cb)

**🎯 变更类型**：功能增强 / 架构变更 / 重构  
**⚡ 重要程度**：🔴高  

**📋 变更摘要**  
- 为 **Hybrid（混合）模型** 引入 `register_kv_caches` 接口，使 KV‑Cache 能够在不同注意力子后端（FullAttention、MLA、Mamba 等）之间共享并统一管理。  
- 新增 **CanonicalKVCacheTensor / CanonicalKVCacheRef / CanonicalKVCaches** 三个数据结构，实现 KV‑Cache 的**块级标准化**表示，解决不同后端在张量布局（如 FlashAttention 的 `(2, num_blocks, …)`）上的差异。  
- `OffloadingConnectorWorker` 与 `OffloadingSpec` 的签名全部改为接受 **CanonicalKVCaches**，并在内部完成 **层‑张量映射、页面大小计算、跨层缓存注册** 等逻辑。  
- 大量单元测试覆盖 **统计、调度、并发、预取/预写、abort、统一类型规格**，以及对 **CPU↔GPU 迁移** 的端到端验证。  
- 其他配套改动：`set_kv_cache_layout` 现在会清除 LRU 缓存；测试工具 `utils` 重构；`vllm/v1/kv_offload/spec.py` 接口更新；`vllm/v1/kv_offload/cpu/spec.py` 适配新规范。  

**🎯 影响范围**  
- `vllm/distributed/kv_transfer/kv_connector/v1/offloading/worker.py`（核心实现）  
- `vllm/v1/kv_offload/spec.py`（抽象层）  
- `vllm/v1/kv_offload/cpu/spec.py`（CPU‑offload 入口）  
- `vllm/v1/kv_offload/worker/cpu_gpu.py`（GPU↔CPU 迁移实现）  
- 相关测试目录 `tests/v1/kv_connector/unit/offloading_connector/*`  
- 其它辅助模块：`vllm/v1/attention/backends/utils.py`、`vllm/v1/kv_offload/spec.py`、`vllm/platforms`（布局清理）

## 🔍 技术洞察

### 架构影响
| 维度 | 变化 | 价值/风险 |
|------|------|-----------|
| **统一的 KV‑Cache 表示** | 引入 `CanonicalKVCaches`，把所有层的 KV‑Cache 拉平为 **块‑页** 结构，且每块使用 `int8` 存储。 | ✅ 统一不同后端的布局，消除 “FlashAttention‑K/V 分离” 与 “Mamba‑state 多张量” 的差异。<br>⚠️ 需要确保 `CanonicalKVCacheRef` 与实际后端的 `page_size_bytes` 对齐，否则会出现写越界。 |
| **注册流程迁移** | `register_kv_caches` 由原来的 “层→张量 & 后端” 两个字典，改为一次性接受 **CanonicalKVCaches**。 | ✅ 简化调用方（测试、模型初始化）只需提供已标准化的缓存即可，降低出错概率。<br>⚠️ 旧插件/第三方代码仍使用旧签名会编译错误，需要迁移。 |
| **跨层缓存（Cross‑layer KV）** | 新增 `register_cross_layers_kv_cache`，并在内部检查 `num_blocks` 必须位于物理第 0 维，支持 **All‑Layers** 统一缓存布局。 | ✅ 为稀疏/共享 KV 场景提供统一入口。<br>⚠️ 仅在 `kv_cache_config.kv_cache_groups` 长度为 1 时生效，误用可能导致 `AssertionError`。 |
| **处理块大小因子** | `SingleDirectionOffloadingHandler` 现在计算 `src_block_size_factor / dst_block_size_factor` 基于 **GPU→CPU** 或 **CPU→GPU** 的方向，而不是直接使用传入的因子。 | ✅ 正确处理 `block_size_factor`（CPU 页面 = GPU 页面 × factor），兼容所有后端。 |
| **Handler 注册** | `OffloadingSpec.get_handlers` 只接受 `CanonicalKVCaches`，返回 `src_cls, dst_cls, handler`。 | ✅ 解耦了 `kv_caches` 与 `attn_backends`，避免在 `worker` 中再次遍历两遍字典。 |

### 性能影响
| 场景 | 正向影响 | 负向影响 |
|------|----------|----------|
| **KV‑Cache 分配/reshape** | 使用 **GPUModelRunner.initialize_kv_cache_tensors** 进行真实分配，保证布局与模型运行时完全一致。 | 额外的 **canonical 化** 步骤（遍历层、拆分 FlashAttention、构造 `CanonicalKVCacheTensor`）在模型启动阶段引入 O(#layers) 的 CPU 开销。通常在秒级范围，可接受。 |
| **迁移（swap）路径** | `SingleDirectionOffloadingHandler.transfer_async` 直接使用 `ops.swap_blocks`，`num_bytes` 计算改为 **group_block_size**，避免了对每个张量累加的冗余。 | 若 **group_data_refs** 较多（大量 KV‑Cache 组），`group_block_size_in_bytes` 计算会在每次迁移前进行列表遍历，略增 CPU 开销。可通过缓存 `group_block_size_in_bytes`（已实现）进一步降低。 |
| **内存占用** | 所有 KV‑Cache 均转为 **int8**，页面大小统一后 **GPU/CPU** 端占用与原实现相同。 | `CanonicalKVCacheTensor` 持有额外的 Python 对象包装层，略增 Python 层内存占用（几个十 KB），对整体影响微乎其微。 |
| **并发/预取** | 新增 **prefix cache** 与 **transfer job deduplication** 测试，确保多请求共享同一加载任务，不会重复 I/O。 | 代码路径增加了对 `TransferSpec` 列表的管理，需要确保锁/原子操作的正确性，否则可能出现 **竞态**（已在 `OffloadingConnectorWorker` 中使用 `self._transfer_events` 进行同步）。 |

### 安全考虑
- **内存越界**：`CanonicalKVCacheRef.page_size_bytes` 与实际张量 `page_size_bytes` 必须保持一致。若出现不匹配，`ops.swap_blocks` 可能在 GPU 内核层触发 **segfault**。已通过断言在 `register_kv_caches` 中进行校验。  
- **设备一致性**：`register_cross_layers_kv_cache` 中强制检查 `num_blocks` 位于物理第 0 维，防止 **错误的 stride** 产生跨设备读写。  
- **并发安全**：`_transfer_events` 使用 `torch.Event` 进行同步，且在 `wait` 阶段使用循环等待，避免了 Python 线程锁的复杂性。若未来引入多线程调度，需要再次审计此处的竞争风险。  
- **配置泄漏**：`set_kv_cache_layout` 现在清除 LRU 缓存，以防在测试或多实例环境中残留布局信息导致 **跨实例污染**。

总体而言，此次改动把索引生成抽离为独立后置步骤，提升了构建的并行度与可维护性。只要在并发情况下做好锁或同步，替换 `regex` 的 hack 能够在正式合并前清理干净，CI 将更加可靠。

### [Bugfix] Offload blocking tokenizer ops to shared thread pool to unblock event loop (#34789)
**SHA**: `999dfc1` | 🔗 [查看提交](https://github.com/vllm-project/vllm/commit/999dfc1622d042dffbe4e273056fa4554453c8e7)

**变更概览**  
- 新增 `renderer_num_workers` 配置并在 CLI 中提供 `--renderer-num-workers` 参数。  
- `BaseRenderer` 引入共享 `ThreadPoolExecutor`，把 **Tokenizer**、**Multimodal 预处理**、以及 **clear_mm_cache** 等阻塞操作全部 offload 到该线程池。  
- 为 tokenization、multimodal 处理、engine 输入构造等关键路径实现 `*_async` 版，并在 `render_cmpl_async`、`render_chat_async` 中并行调度。  
- 具体渲染器（DeepSeek、Grok2、HF、Mistral）改为通过 `make_async` 调用 `apply_chat_template`，不再在主线程阻塞。  
- `AsyncMicrobatchTokenizer` 可接受外部 executor，保持与渲染器共享同一线程池。  
- `AsyncLLMEngine.reset_mm_cache` 改为 await `renderer.clear_mm_cache_async()`。  
- 测试 MockModelConfig 补全 `renderer_num_workers`，确保新字段不破坏旧行为。

**影响范围**  
- `vllm/config/*`, `engine/arg_utils.py`, `renderers/*`, `utils/async_utils.py`, `v1/engine/async_llm.py` 以及相关测试文件。  
- 运行时：所有使用 `Renderer` 的入口（OpenAI API、vLLM 内部调用）均会在后台线程池执行 tokenization 与多模态前处理。

**关注点 & 建议**  

1. **线程池生命周期**  
   - `BaseRenderer.shutdown()` 已显式 `shutdown(wait=False)`，但 `AsyncLLMEngine` 仍可能在未调用 `renderer.shutdown()` 前析构，引发悬挂任务。建议在 engine 停机流程末统一调用 `renderer.shutdown()`，并在异常路径确保执行。  

2. **并发竞争**  
   - `clear_mm_cache` 与 `process_inputs` 仍共享同一 executor；若 `clear_mm_cache` 较慢，可能阻塞后续 tokenization。可考虑为两类任务提供独立的 `max_workers`（例如 tokenizer 2, mm 1），或在 `clear_mm_cache_async` 中使用 `await asyncio.sleep(0)` 让调度器先行。  

3. **默认 worker 数**  
   - 当前默认 `1` 可能仍成为瓶颈，尤其在高并发请求下。建议在文档中说明调优方法，并在 CI 中加入基准测试，帮助用户决定合适的 `--renderer-num-workers`。  

4. **错误传播**  
   - 异步包装 (`make_async`) 会捕获异常并在调用处重新抛出，但若线程池已关闭，后续 `await` 可能得到 `RuntimeError: cannot schedule new futures`. 在 `shutdown` 前应先等待所有挂起的 async 任务完成（`executor.shutdown(wait=True)`），或在 `clear_mm_cache_async` 等入口加防护。  

5. **兼容性**  
   - 新增字段在 `ModelConfig`、`EngineArgs` 中已有默认值，向后兼容无问题。但确保所有手动构造 `ModelConfig`（如自定义脚本）也更新该字段，否则可能出现 `AttributeError`。可在 `ModelConfig.__post_init__` 中加入 `self.renderer_num_workers = getattr(self, "renderer_num_workers", 1)` 作为容错。  

6. **测试覆盖**  
   - 当前仅添加字段默认，尚未验证并发执行的正确性。建议补充 `test_renderer_threadpool_concurrency`：在高并发下调用 `render_chat_async`，检查返回顺序、无 deadlock、cache 清理成功。  

**结论**  
此次改动通过统一线程池有效降低了 event‑loop 被阻塞的风险，提升了多模态/高并发场景的响应性。但需要注意线程池的生命周期管理、潜在的任务竞争以及对外文档/默认值的说明，以防在生产环境出现资源泄漏或性能回退。适当增补并发测试与调优指南，可让该特性更加稳健。
