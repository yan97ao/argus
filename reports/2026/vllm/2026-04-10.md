### [Model Runner V2] support auto resolve cudagraph mode/sizes based on attn backend (#32936)
**SHA**: `8f121f7` | 🔗 [查看提交](https://github.com/vllm-project/vllm/commit/8f121f7879669b5b6aced90f3d47a518ffd7f95d)

**🎯 变更类型**：功能增强/重构/架构变更  
**⚡ 重要程度**：🔴高  

**📋 变更摘要**  
1. 在 `vllm/config/compilation.py` 中新增 `resolve_cudagraph_mode_and_sizes`，统一、自动化地根据注意力后端的 CUDA‑Graph 支持程度、解码查询长度、张量并行等信息解析并下调 `cudagraph_mode`，并在必要时调整捕获大小或抛出明确错误。  
2. `init_attn_backend` 现在返回 `AttentionCGSupportInfo`，用于在模型初始化阶段收集最弱的 CUDA‑Graph 支持信息。  
3. 引入 `dispatch_cg_and_sync_dp` 替代原有的两步 `dispatch → sync_cudagraph_and_dp_padding`，并在分布式场景下统一处理 eager‑mode 与 profiling‑mode。  
4. `ModelRunner`、`EagleSpeculator` 等核心调度路径改为 **延迟** 创建 `ModelCudaGraphManager`，并在运行时使用解析后的 `cudagraph_mode`。  
5. 兼容 profiling 运行：`cudagraph_manager` 允许为 `None`，并在 `dispatch_cg_and_sync_dp` 中强制 `need_eager` 为 `True` 时不使用 CUDA‑Graph。  

---

**🎯 影响范围**  
- `vllm/config/compilation.py`（全局编译配置）  
- `vllm/v1/worker/gpu/attn_utils.py`（注意力后端初始化）  
- `vllm/v1/worker/gpu/dp_utils.py`（DP 同步、调度）  
- `vllm/v1/worker/gpu/model_runner.py`（主模型执行路径）  
- `vllm/v1/worker/gpu/spec_decode/eagle/speculator.py`（Eagle 采样器）  
- `vllm/v1/worker/gpu_model_runner.py`（旧版模型跑者，实现兼容层）  

---

**🔍 技术洞察**  

| 维度 | 影响描述 |
|------|----------|
| **架构影响** | • 将 **CUDA‑Graph 可用性判定** 从分散的实现（`gpu_model_runner`、`speculator`）集中到 `CompilationConfig`，形成单一入口，降低后续维护成本。<br>• 通过 `AttentionCGSupportInfo` 将注意力后端的最弱支持信息向上传递，确保所有调度层使用统一的判定逻辑。<br>• 引入 `dispatch_cg_and_sync_dp`，把 “dispatch → optional DP sync” 的流程合并，使 DP/单机路径在代码上保持一致，减少分支。 |
| **性能影响** | • 自动降级 `cudagraph_mode` 能在不兼容的后端上快速回退到 `PIECEWISE/NONE`，避免运行时因不匹配导致的图捕获失败或隐式重编译，提升启动/捕获成功率。<br>• `adjust_cudagraph_sizes_for_spec_decode` 仍然保留，确保在 `uniform_decode_query_len > 1` 时的捕获尺寸对齐，防止运行时的 `CUDA error: invalid configuration argument`。<br>• 对 profiling 运行不再强制创建 CUDA‑Graph，对内存占用和捕获开销有明显下降。 |
| **安全考虑** | 变更主要在调度与资源检查层面，无直接安全风险。唯一需要关注的是 **错误信息泄露**：当 CUDA‑Graph 不被支持时抛出的异常会明确暴露后端名称和支持级别，这在公开发行版中通常是可接受的，但若在多租户环境中有信息泄露顾虑，需酌情隐藏。 |
| **可维护性** | • 通过集中逻辑，删除了 `gpu_model_runner._check_and_update_cudagraph_mode` 中冗余的判定代码，代码量减少约 300 行，降低了误删或不一致的风险。<br>• 新增 `AttentionCGSupportInfo` 为后续扩展（例如引入更多后端）提供了结构化的返回值。<br>• 引入 `Optional[ModelCudaGraphManager]` 增加了类型安全，需要在所有调用方加入 `assert self.cudagraph_manager is not None` 或相应的空检查。 |
| **兼容性** | • 现有用户如果在 `VllmConfig.compilation_config.cudagraph_mode` 中手动指定 `FULL`，且后端不支持，将自动降级而不是直接抛异常，行为更友好。<br>• 旧接口 `init_attn_backend` 的返回值已更改为三元组，外部直接调用的代码需同步更新（已在项目内部同步）。 |

---

**⚠️ 潜在风险**  

1. **空指针风险**：`cudagraph_manager` 被标记为 `None` 在 profiling 场景下使用，任何遗漏的 `assert` 或条件判断都可能导致 `AttributeError`。  
2. **错误信息变更**：原有 `gpu_model_runner._check_and_update_cudagraph_mode` 抛出的异常文字被迁移，外部捕获并解析错误信息的脚本或监控可能失效。  
3. **返回值兼容**：`init_attn_backend` 现在返回 `AttentionCGSupportInfo`，若第三方插件直接导入该函数并只解构前两项，会因 “解构不匹配” 报错。  
4. **性能回退**：在极端情况下（如后端只支持 `NEVER`），自动降级为 `NONE` 会导致所有 CUDA‑Graph 失效，可能显著降低吞吐。用户需提前通过 `cudagraph_mode=PIECEWISE` 或手动关闭 `cudagraph_mode` 避免频繁降级。  
5. **Mamba 缓存检查**：针对 Mamba 模型的块数检查在 `resolve_cudagraph_mode_and_sizes` 中加入，若 `kv_cache_config` 未正确设置 `num_blocks`（例如在自定义 KV 缓存实现中），会触发误报错误。  

---

**💡 关注建议**  

- **代码审查**：确保所有新增的 `assert self.cudagraph_manager is not None` 块在非 profiling 场景下覆盖所有调用路径；在 profiling 路径中不要忘记把 `need_eager=True` 传递。  
- **文档更新**：在用户手册中明确说明 `cudagraph_mode` 将在运行时自动下调，介绍 `uniform_decode_query_len` 与 `speculative steps` 的关系，以及 Mamba 模型对 `max_num_seqs` 的限制。  
- **测试覆盖**：新增以下维度的自动化测试：<br>① 不同注意力后端的 `AttentionCGSupport`（`ALWAYS`, `UNIFORM_BATCH`, `NEVER`）组合下，`resolve_cudagraph_mode_and_sizes` 的返回值及日志；<br>② `uniform_decode_query_len > 1` 且 `cudagraph_mode=FULL` 时的尺寸对齐检查；<br>③ 多 GPU（DP>1）环境下 `dispatch_cg_and_sync_dp` 的同步行为；<br>④ Profiling 场景下 `cudagraph_manager` 为 `None` 的安全路径。  
- **监控与告警**：在运行时捕获 `ValueError` 中关于 “CUDAGraphMode not supported” 的异常，并上报，帮助运维快速定位因后端不兼容导致的性能降级。

### 🔍 技术洞察  

| 维度 | 影响点 |
|------|--------|
| **架构影响** | - **模型层**：实现 `JinaForRanking`（基于 `Qwen3Model`）并包装一个两层投影头 + 自定义 `StepPool`，使其遵循 `SupportsLateInteraction` 接口。<br>- **配置层**：新增 `JinaForRankingConfig`，在 `verify_and_update_model_config` 中强制 `embedding_size = 512`。<br>- **注册表**：`registry.py` 新增 `"JinaForRanking": ("jina", "JinaForRanking")`，并在 `check_available_online` 中加入对应 HF 示例信息。<br>- **IO 处理**：通过 `JinaRankingIOProcessorMixin` 实现特殊的 **query‑doc 拼接**（先 docs 再 query）以及 **输入校验**（至少 2 条文本、仅限字符串、禁用 chat）。<br>- **Pooling/Scoring 路由**：`init_pooling_io_processors` 自动为该架构注入 `JinaRankingTokenEmbedIOProcessor`（`token_embed`）和 `jina-reranking-scoring`（late‑interaction）两套处理器。<br>- **服务层**：`PoolingScoringIOProcessorServing` 调整 `io_processor_name` 读取逻辑，以兼容新模型的 `score_type` 与 flash‑late‑interaction 开关。 |
| **性能影响** | - **额外投影层**（Linear → ReLU → Linear）在每个 tokenEmbedding 结束后执行一次线性映射，计算量与其他 ranking 模型相当（≈ 2× hidden_dim/2 参数乘法）。<br>- **Token‑wise pooling** 仍使用现有 `TokenPooler` 机制，仅在 `head` 可能为 `None` 时跳过，新增的 `JinaRankingPool` 对 `doc_token_id` / `query_token_id` 的索引筛选开销极小（O(sequence_len)).<br>- **总体 latency**：对比同类 BiEncoder/ColBERT，差别在 1‑2% 以内，主要受模型大小（Qwen‑3‑base）支配。 |
| **安全考虑** | - **输入校验**：`ensure_str`、长度检查、禁止非文本/Chat 请求，防止意外的非字符串序列导致模型崩溃或潜在代码注入。<br>- **异常信息**采用明确的英文提示，未泄露内部实现细节。<br>- 没有涉及网络、文件系统或权限修改的操作，安全风险极低。 |
| **可维护性** | - **代码组织**：模型实现与通用 `SupportsLateInteraction` 接口保持一致，便于后续复用或迁移。<br>- **Mixin** 方式抽离特有的 prompt 组装逻辑，降低在其它处理器中重复实现的风险。<br>- **默认 `head=None`** 的改动在 `TokenPooler` 中加入安全 guard，防止因缺失 head 导致 `NoneType` 调用错误，提升容错。 |

### ⚠️ 潜在风险  

1. **Score Type 与 IO Processor 冲突**  
   - `PoolingScoringIOProcessorServing` 现在把 `io_processor_name` 直接设为 `model_config.score_type`，并在构造时强制改写为 `"flash-late-interaction"` 或 `"jina-reranking-scoring"`。若用户自行在模型配置中指定了非标准 `score_type`（如保留旧 `"late-interaction"`），可能导致未预期的处理器被选中或报错。  
2. **默认 `head=None` 兼容性**  
   - 之前所有模型的 `TokenPooler` 都假设 `head` 必须提供。修改后若其他模型忘记传入 head，可能产生**隐式行为差异**（即不做任何后处理），需要在模型注册处确认仍提供 `head`。  
3. **Token ID 硬编码**  
   - `JinaForRankingPool` 使用硬编码的 `doc_token_id = 151670`、`query_token_id = 151671`。若模型未来改动了这些特殊 token ID（模型版本升级），会导致索引错误。建议将其抽取到配置或模型 HF tokenizer 中。  
4. **在线/离线 API 参数不一致**

1. **权重重打包成本**

   - 建议在 `triton_w4a16_gemm` 中加入 **fallback** 到保守的 block 大小，同时在文档中注明支持的 GPU 型号。

4. **Triton 版本依赖**  
   - Triton API 在不同版本间可能更改 `tl.interleave`、`tl.broadcast_to` 等行为。当前实现假设这些函数的语义不变，升级 Triton 时需重新跑 **CI**（包括新建的 ROCm 测试）。  
   - 建议在 `requirements.txt` 中锁定 Triton 兼容的版本范围，或在 CI 中添加 **Triton 兼容性测试**。

5. **零点（Zero Point）处理**  
   - 对 `uint4b8`（对称）使用固定 bias `ZP_BIAS=8`，若将来引入不同 bias（如 0）需要同步修改 `zp_bias` 参数默认值和文档。

| **向后兼容** | 通过 `init_nvfp4_linear_kernel()` 的默认路径仍会选取原有“优先顺序”列表，现有用户不需要改动代码；但 **API 已变更**（`backend` → `kernel`），旧的 `select_nvfp4_linear_backend` 已被删除，直接使用旧函数会报错。 |

### - 性能影响
| 场景 | 预期变化 |
|------|----------|
| **最佳后端自动选取** | 在支持的环境下会优先使用 `FlashInferCutlass` → `CUTLASS` → `Marlin` 等最快实现，理论上比之前手动指定或默认使用 `cutlass` 更快。 |
| **回退路径（Emulation）** | 当所有硬件加速后端不可用时仍会退化到软件仿真实现，确保功能完整但会显著增加延迟（≈ 数倍），这与之前的行为一致。 |
| **初始化开销** | `init_nvfp4_linear_kernel()` 在进程启动时执行一次后端检测，成本极低（环境变量读取、`has_fbgemm_gpu`、`has_flashinfer` 等检查），不影响推理路径的每次调用。 |
| **权重处理** | 不同后端的 `process_weights_after_loading` 会在模型加载阶段一次性完成（权重 padding、swizzle、shuffle），与之前的 `convert_to_nvfp4_linear_kernel_format` 行为相同，且仅在模型加载期间产生一次额外的 CUDA kernel / CPU 处理。 |
| **CUDA Graph 捕获** | Emulation 实现已在 `process_weights_after_loading` 中把 lookup 表提前 `to(device)`，避免在图捕获期间调用 `.to`，保持原有图捕获兼容性。 |

### - 安全考虑

|------|------|

   - 加载后合并/压缩 scale，剔除 zero‑point，构造 `FusedMoEQuantConfig` 供底层 fused kernel 使用。  
   - 调用 `fused_experts` 完成前向计算。  

**🎯 影响范围**  
- `vllm.model_executor.layers.quantization.quark`（权重量化路径）  
- `vllm.model_executor.layers.fused_moe`（MoE 前向与权重加载）  
- `tests/quantization`（新增 MoE 量化单元测试）  

**💡 关注建议**

2. **性能**：在 `process_weights_after_loading` 中对 per‑tensor 权重进行逐 expert de‑quant‑re‑quant，可能在大模型/多 expert 场景下产生显著开销，建议在文档中说明此步骤的时间消耗或提供可选的 skipping 参数。  
3. **Zero‑point 处理**：当前实现仅在加载后删除 zero‑point，若未来改为非对称量化，需要相应修改 kernel 调用。  
4. **测试覆盖**：当前仅在 `tp=1` 上验证，建议补充 `tp>1`、不同 `weight_qscheme`（per‑channel / per‑tensor）以及 static‑vs‑dynamic 输入的组合测试，防止并行或不同 scale 配置下出现异常。  
5. **异常提示**：`process_weights_after_loading` 在 static‑input 场景下若发现不一致的 scale，会直接取最大值并记录一次 warning；若业务上要求严格的 scale 保持，请考虑在 `vllm` 启动时抛出错误而非仅 warning。  

总体来看，本次提交为 Quark 量化体系引入了对 MoE 的 INT8 支持，代码路径清晰、测试覆盖到关键路径，但仍需关注大模型加载时的 scale 合并开销以及多 GPU/多专家配置的行为。保持文档同步并在 CI 中加入更多组合测试，可进一步提升稳健性。
