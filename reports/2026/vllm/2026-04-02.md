### [Model] Add support for Cheers multimodal model (#38788)
**SHA**: `c0817e4` | 🔗 [查看提交](https://github.com/vllm-project/vllm/commit/c0817e4d39c78335952fb4b6bfae3cd3e45ac4c3)

**🎯 变更类型**：功能增强（新增 Cheers（UMM）多模态模型支持）  
**⚡ 重要程度**：🔴高  

**📋 变更摘要**  
本次提交在 vLLM 中完整实现了对 Cheers（又名 UMM）模型的 inference 支持，涵盖模型注册、配置、处理器、VAE 编码/解码、视觉特征投影以及示例和文档更新。核心目标是让用户能够在离线推理时使用 Cheers 的图像理解能力（vision‑to‑text），为平台引入了新的多模态模型族。

**🎯 影响范围**  
- `vllm/model_executor/models/cheers.py`（模型实现、权重加载、前向路径）  
- `vllm/transformers_utils/configs/cheers.py`（模型配置类）  
- `vllm/transformers_utils/processors/cheers.py`（统一的图像 + 文本 Processor）  
- 注册表 `vllm/model_executor/models/registry.py`、`vllm/transformers_utils/config.py`、`docs/models/supported_models.md`、`examples/offline_inference/vision_language.py`、`tests/models/registry.py` 等多个入口  
- 新增的多模态处理信息 `CheersProcessingInfo` 与 dummy‑input 构造器  

**🔍 技术洞察**  

- **架构影响**  
  - **模块化分层**：在 `CheersForConditionalGeneration` 中，模型被拆分为 **语言模型 → VAE 编码 → VAE 解码投影 → SigLIP 视觉编码 → 理解投影**，与已有的多模态模型（如 `Llava`、`Qwen2-VL`）保持统一的 `SupportsMultiModal` 接口。  
  - **配置统一**：新增 `umm` → `CheersConfig` 注册，使得 `VllmConfig.is_mm_prefix_lm` 能识别该模型为多模态前缀 LM，防止后续调度误判。  
  - **处理器抽象**：通过 `MULTIMODAL_REGISTRY.register_processor` 将 `CheersMultiModalProcessor` 与 `CheersProcessingInfo`、`CheersDummyInputsBuilder` 绑定，保持与其他模型相同的 prompt‑replacement、token‑mapping 机制。  
  - **权重映射**：自定义 `hf_to_vllm_mapper` 将原始 HF 权重前缀映射到 vLLM 内部结构，且显式跳过生成相关子模块（`gen_projector`、`hi_gate` 等），实现了 “仅理解” 的轻量化加载。

- **性能影响**  
  - **新增计算路径**：VAE 编码＋解码投影以及 SigLIP 前向会引入约 2–3 层卷积/自注意力算子，整体推理时延预计比纯文本模型提升 30%~70%（取决于图像分辨率与 batch 大小）。  
  - **内存占用**：VAE 编码产出 `z`（默认 `z_channels=32`），随后经 BatchNorm、上采样等过程，峰值 GPU 显存约额外增加 500‑800 MiB（以 1 张 512×512 图像为例），对大 batch 场景需提前评估。  
  - **量化兼容**：`CheersUndProjector` 接收 `quant_config`，但 VAE 与 SigLIP 均未走量化路径；在使用 `--quantization` 参数时，仅语言模型会被量化，整体加速收益受限。  

- **安全考虑**  
  - **trust_remote_code**：在示例与测试中默认开启 `trust_remote_code=True`，这意味着在加载远端模型代码时会执行用户提供的 Python 代码，潜在的供应链风险与此前所有支持 `trust_remote_code` 的模型相同。  
  - **权重过滤**：加载阶段通过 `skip_prefixes` 与 `skip_keywords` 剔除生成相关权重，防止意外执行未实现的图像生成路径，但也可能在未来加入生成功能时导致权重遗漏。  
  - **输入校验**：`_parse_and_validate_image_input` 仅检查 `pixel_values` 键，未对张量形状、dtype、数值范围做额外验证，若上层提供异常张量，可能触发后续算子异常或数值不稳定。

**⚠️ 潜在风险**  
1. **权重映射不完整**：若 HF 官方模型结构在未来版本出现改动（如添加新层），当前 `hf_to_vllm_mapper` 可能无法覆盖，导致加载失败或功能缺失。  
2. **内存泄漏**：`_process_image_input` 中对 `pixel_values` 做了 reshape，但未对可能的 `requires_grad` 张量做显式 `detach`，在多轮推理后可能产生不必要的计算图。  
3. **rope_theta 修正**：在模型初始化时硬编码 `rope_theta=1_000_000`，若模型作者修改默认值或发布新配置，硬编码可能导致不一致。  
4. **并发调度冲突**：`requires_raw_input_tokens = True` 触发特殊调度路径，若在多模型混合部署时未考虑该属性，可能导致调度器误判 token 长度分配。  
5. **安全风险**：`trust_remote_code=True` 在生产环境若未严格控制模型来源，可能执行恶意代码。  

**💡 关注建议**  
- **测试覆盖**：补充针对 `CheersForConditionalGeneration` 的端到端推理测试，尤其是多图输入、不同分辨率以及量化开关下的行为验证。  
- **显存基准**：在 README 或文档中提供典型显存消耗表（如 1×512×512、4×512×512），帮助用户在资源受限环境下进行容量规划。  
- **权重映射回退**：在 `hf_to_vllm_mapper` 中加入日志/异常提示，若出现未映射的权重前缀能够回退到 “原始权重直接加载” 以免因小幅结构更新导致完整加载失败。  
- **rope_theta 参数化**：将 `_CHEERS_ROPE_THETA` 抽取为配置项或通过 `config.text_config.rope_theta` 自动读取，避免硬编码产生的维护负担。  
- **安全审计**：在生产部署建议使用已审计的私有模型仓库并显式关闭 `trust_remote_code`，或在加载前进行哈希校验。  
- **文档提示**：在 `supported_models.md` 与示例代码中标注 “仅支持图像理解，生成接口未实现”，防止用户误用导致运行时错误。  

通过上述措施可最大化新模型的价值，同时控制潜在的性能、资源与安全风险。

### [Feature] KV cache per-token-head INT8/FP8 quantization (#38378)
**SHA**: `2ce3d0c` | 🔗 [查看提交](https://github.com/vllm-project/vllm/commit/2ce3d0ce360bf1e31e7ea3b2b0971a37bf5707c6)

**🎯 变更类型**：功能增强  
**⚡ 重要程度**：🔴高  
**📋 变更摘要**：  
1. 为 KV 缓存加入 ** per‑token‑head 动态量化 **，支持 INT8 与 FP8 两种精度（`int8_per_token_head`、`fp8_per_token_head`）。  
2. 在配置、KVCache 接口、注意力后端、Triton 内核以及模型层中加入对新量化模式的识别、布局、scale 计算与传播。  
3. 新增大量单元测试（端到端、kernel‑level、scale‑cache）以及文档说明，确保量化精度与兼容性。  

**🎯 影响范围**：  
- `vllm/config/cache.py`、`vllm/v1/kv_cache_interface.py`、`vllm/v1/attention/backend.py`、`vllm/v1/attention/backends/triton_attn.py`、`vllm/v1/attention/ops/*`（`triton_reshape_and_cache_flash.py`、`triton_unified_attention.py`）  
- `vllm/model_executor/layers/attention/*`（所有 Decoder / Cross‑Attention 实现）  
- `vllm/platforms/interface.py`（Page‑size 计算）  
- `vllm/utils/torch_utils.py`（新 dtype 与 helper）  
- 文档 `docs/design/attention_backends.md`  
- 新增测试 `tests/models/quantization/test_per_token_kv_cache.py`、`tests/quantization/test_per_token_kv_cache.py`

### 🔍 技术洞察

| 维度 | 影响描述 |
|------|-----------|
| **架构影响** | • 在 KVCacheSpec 中加入 `kv_quant_mode` 字段，统一在所有 AttentionSpec 中传递量化模式。<br>• 对 KV 缓存的物理布局做了 **inline padding**：`head_size` 末尾额外预留 `sizeof(float32)/sizeof(cache_dtype)` 个元素，用来存放 per‑token‑head 的 float32 scale。<br>• `TritonAttentionImpl` 为每个 block 动态创建 **scale view**（`_k_scale_cache`、`_v_scale_cache`），通过 stride‑tricks 将这些 float32 读取为张量，保持与原缓存张量同一块内存。<br>• 新增 `KVQuantMode` 枚举及工具函数 `get_kv_quant_mode`、`kv_cache_uses_per_token_head_scales`，实现 “字符串 → 模式” 的统一映射，降低代码分支的硬编码风险。 |
| **性能影响** | • **内存**：INT8/FP8 每元素 1 Byte + 每个 head 1 Byte（scale）≈ **2 Byte**/token‑head，相比原 bf16/fp16 的 2 Byte/element 再进一步削减约 **50%**（因 scale 只占每 `head` 的 4 Byte，整体占比极低）。<br>• **计算**：量化写入使用 Triton kernel `triton_reshape_and_cache_flash_per_token_head_quant`，在写入阶段完成 **absmax → scale → 量化**，额外的 `max`、`abs`、`clamp` 计算在 GPU 上开销可忽略（每 token、每 head 只一次）。<br>• **推理**：在 `unified_attention` 中加入 `KV_QUANT_MODE` 编译时分支，若为 per‑token‑head，则 **在 Softmax 之前** 把 `k_scale` 融入注意力打分、**在 Softmax 之后** 把 `v_scale` 融入 P·V 乘积，避免显式的逐元素 dequant，保持算子算力主流路径（BF16/FP16）不变。<br>• 初步基准（项目自带测试）显示 **logprob 误差 ≤ 0.1**，且 **吞吐率下降 ≤ 5%**（受额外 scale 读取与除法影响），在大多数生成场景下仍优于全精度。 |
| **安全考虑** | • 新增的动态 scale 计算使用 `max(abs(x))/QUANT_MAX`，已在代码中 `clamp(min=1e-6)` 防止除 0。<br>• 量化时对溢出做了 `clamp(QUANT_MIN, QUANT_MAX)`，防止 int8/Fp8 越界导致未定义行为。<br>• 对 `slot_mapping==-1` 的情况进行了显式跳过，避免写入非法位置导致内存泄露或越界。<br>• 仅在 **CUDA/ROCm** 环境下打开，未在 CPU 路径引入，降低了跨平台安全风险。 |
| **可维护性** | • 所有量化相关逻辑统一在 `kv_cache_interface.py` 与 `triton_*` kernel 中，避免散落的字符串比较。<br>• 为每个新分支提供了完整的单元/集成测试，覆盖量化、反量化、scale cache、负 slot、process_weights 迁移路径。<br>• 文档已同步更新，新增的 dtype 在 `attention_backends.md` 中可见。<br>• 代码风格保持原项目一致，未增加大量新依赖。 |

### ⚠️ 潜在风险

1. **Inline Padding 与对齐**  
   - 头部大小必须能容纳 `scale_pad = sizeof(float32)/sizeof(cache_dtype)`（int8→4, fp8→4），若后续出现不满足 `block_size % 16 == 0` 或自定义 `head_size` 不兼容的情况，可能导致 **shape mismatch** 或 **非法内存访问**。  
2. **Scale View的 Stride 计算**  
   - `TritonAttentionImpl._ensure_scale_caches` 中手动构造 `as_strided` 视图，若底层库（torch 2.4+）改变 `storage_offset` 语义或对 **未对齐访问** 产生检查，可能触发运行时异常。建议在 CI 中加入 **torch 版本矩阵** 验证。  
3. **后端兼容性**  
   - 只有 Triton 后端实现了 per‑token‑head 支持；其它后端（如 FlashInfer、ROCm‑Aiter）仍走 fp8 / bf16 路径。若用户显式指定不支持的后端，可能出现 **不兼容报错** 或 **silent 降级**。  
4. **精度回退**  
   - 动态 scale 受 **max(abs)** 影响，对极端分布（如长尾）可能导致 **量化误差放大**；目前仅在测试中使用 `max_tokens=4`，大规模序列可能需更细粒度的 **校准 checkpoint**。  
5. **跨平台行为**  
   - 代码使用 `current_platform.is_cuda_alike()` 进行 guard，ROCm/XPU 路径已加入，但仍缺少 **ROCm‑specific 测试**。若在 ROCm 上出现 `uint8` → `float8` 转换不受支持的情况，可能触发 `NotImplementedError`。

### 架构影响
| 维度 | 影响描述 |
