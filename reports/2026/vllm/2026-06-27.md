### fix: Correct reasoning-end detection for prompt history (#44551)
**SHA**: `d706dec` | 🔗 [查看提交](https://github.com/vllm-project/vllm/commit/d706dec904e89e4067efd9535b96705aa61f3935)

**🎯 变更类型**：Bug修复  

**⚡ 重要程度**：🔴高  

**📋 变更摘要**：  
- 修正了 Cohere 系列模型在推理阶段（reasoning）结束检测的逻辑，使其在存在历史对话或 `<|CHATBOT_TOKEN|>` 标记时能够正确识别 `<|END_THINKING|>` 为本轮推理的结束标记。  
- 为对应的单元测试新增了大量覆盖，包括流式解码、工具调用、结构化输出及响应格式的各种组合，确保新实现的稳定性。  
- 同时在 CI 依赖中加入 `cohere-melody`，为新测试提供所需的解析器实现。  

**🎯 影响范围**：  
- `vllm/reasoning/cohere_command_reasoning_parser.py`（核心推理解析器）  
- 新增 `tests/reasoning/test_cohere_command_reasoning_parser.py`（覆盖多种使用场景）  
- CI 依赖文件 `requirements/test/cuda.in`、`requirements/test/cuda.txt`（引入 `cohere-melody`）  

**🔍 技术洞察**：  
- **架构影响**  
  - 该修改位于 **ReasoningParser** 层，属于输入 token 序列分析的底层实现，对上层的 `ChatCompletion`、`Responses` 接口行为没有结构性破坏，只是改进了 **状态机** 的判定规则。  
  - 引入了 `self.chatbot_token_id`、`self.start_token_id`、`self.end_token_id` 三个成员，使判定逻辑更加显式，避免了历史 prompt 中残余的 `<|START_THINKING|>`/`<|END_THINKING|>` 干扰。  

- **性能影响**  
  - 检测函数 `is_reasoning_end` 仍是 **O(N)**（遍历 token 序列），但只在逆序遍历一次并提前返回，实际开销与原实现相当，几乎对吞吐量无负面影响。  
  - 新增的单元测试会在 CI 中执行，略微增加测试时间，但对运行时性能不产生影响。  

- **安全考虑**  
  - 仅涉及 token ID 判定，无外部数据依赖或资源访问，安全风险极低。  
  - 新增的 `cohere-melody` 依赖属于开源库，已在 `requirements` 中锁定具体版本（0.9.0），风险可控。  

**⚠️ 潜在风险**：  
1. **Token ID 映射假设**：该实现假设 Cohere tokenizer 对特殊标记返回唯一负数 ID（-1、-2、-3）。若未来 tokenizer 实现改动（如使用正数 ID），此判断将失效。  
2. **跨模型兼容性**：`CohereCommand3ReasoningParser` 与 `CohereCommand4ReasoningParser` 共用该基类逻辑，若其他模型复用此基类但不使用 `<|CHATBOT_TOKEN|>`，可能出现误判。  
3. **测试依赖膨胀**：引入 `cohere-melody` 作为测试依赖，若该库的内部实现出现破坏性更新，可能导致 CI 失效。  
4. **历史对话边界**：当前实现仅在逆序遍历时遇到 `<|CHATBOT_TOKEN|>` 即返回 `False`，这意味着在同一次生成中出现多个 `<|CHATBOT_TOKEN|>`（极端情况）仍会按最新的标记判断，需确认这一行为符合业务预期。  

**💡 关注建议**：  
- **文档更新**：在项目文档或 parser 注释中说明 “推理结束检测” 的新规则及对特殊 token ID 的依赖，方便后续维护者了解实现细节。  
- **兼容性测试**：加入针对非 Cohere 模型的负面测试，验证在没有 `<|CHATBOT_TOKEN|>` 标记的情况下仍保持原有行为，避免潜在回归。  
- **Token ID 抽象**：可考虑在 tokenizer 接口中提供 `get_special_token_id(name: str) -> int` 方法，避免硬编码负数 ID，提升适配性。  
- **CI 稳定性**：为 `cohere-melody` 添加 pin 至特定 commit（而非仅版本号），防止因上游 minor/patch 变动导致依赖冲突。  
- **边界案例**：补充测试场景，如 “prompt 中出现多段 `<|START_THINKING|>`、`<|END_THINKING|>` 混合”，确保逆序遍历逻辑在所有可能的历史对话布局下都能正确返回。  

此轮修复提升了 Cohere 系列模型在多轮对话、历史 prompt 环境下的推理解析准确性，是对 vLLM 推理框架关键路径的质量改进。

### fix(docker): eliminate race conditions in shared buildkit cache mounts (#44984)
**SHA**: `2e05885` | 🔗 [查看提交](https://github.com/vllm-project/vllm/commit/2e058851d39448bf282e64a6aac04466968622e7)

**🎯 变更类型**：Bug修复  

**⚡ 重要程度**：🔴高  

**📋 变更摘要**：  
在 `docker/Dockerfile` 系列（标准、nightly‑torch、XPU）中为 Cargo 缓存挂载添加了 `sharing=locked` 参数，以消除 BuildKit 在并发构建时对同一缓存目录的竞争导致的 race condition。此举确保了在多实例并行构建镜像时，Cargo 的 registry 与 git 缓存能够安全、可靠地共享。

**🎯 影响范围**：  
- Docker 镜像的构建阶段（Rust 编译相关）  
- 使用 BuildKit 并开启缓存的 CI/CD 流水线  
- 本地开发者在使用 `docker buildx` 并行构建时的体验  

**🔍 技术洞察**  

- **架构影响**：  
  - 此变更仅影响 Dockerfile 的构建指令层面，不会触及项目运行时的 Python/Rust 代码。  
  - 通过 `sharing=locked`，BuildKit 会对同一路径的缓存进行排他锁定，避免多个并发构建进程同时写入导致的元数据损坏。  
  - 对现有的多阶段构建流程（Rust → CSRC → WHEEL）保持兼容，仅在缓存挂载语法上做了微调。

- **性能影响**：  
  - **正向**：缓存仍然被复用，编译时间不受影响；`sharing=locked` 只在缓存被竞争时增加极短的等待时间，整体构建吞吐基本不变。  
  - **负向**：在极端并发（例如 CI 中一次性启动大量并行 `docker buildx`）时，锁竞争可能导致轻微的排队延迟，但远低于因为缓存损坏导致的全量重新编译成本。

- **安全考虑**：  
  - 通过排他锁定，防止了因并发写入导致的文件系统状态不一致，间接提升了构建过程的完整性与可追溯性。  
  - 该改动不涉及运行时代码或网络配置，无直接安全风险。唯一需要关注的是使用的 Docker/BuildKit 版本必须支持 `sharing=locked`（Docker 20.10+），旧版可能会报错或忽略该属性。

**⚠️ 潜在风险**：  
1. **兼容性**：老旧的 Docker Engine 或 BuildKit 版本（< 20.10）不识别 `sharing=locked`，会导致构建失败或回退为默认共享行为。  
2. **CI 环境锁竞争**：在极端并发的 CI 任务中，锁竞争可能导致短暂的构建阻塞，若超时设置过低可能出现构建失败。  
3. **缓存一致性假设**：如果项目在构建前手动修改了 Cargo 缓存目录内容（极少见），锁定可能阻止预期的覆盖，需确保缓存只由 BuildKit 管理。

**💡 关注建议**：  
- **升级检查**：确认 CI / 本地机器使用的 Docker Engine ≥ 20.10，或在构建前通过 `docker version` 校验。  
- **监控锁延迟**：在高并发 CI 场景下，可在构建脚本中记录 `BUILD_START` 与 `BUILD_END` 时间，监测是否出现异常等待。若有，需要考虑降低并行构建度或使用专属缓存卷。  
- **回滚方案**：若在特定旧版环境出现构建错误，可临时回退到原始 Dockerfile（去掉 `sharing=locked`），或在 CI 中显式禁用 BuildKit 并使用普通 `docker build`。  
- **文档更新**：在项目的 Docker 构建指南中标注 “需要 BuildKit 20.10+ 并支持 `sharing=locked`”，并说明该参数的目的与可能的兼容性限制。  

--- 

通过上述改动，vLLM 项目的 Docker 镜像构建过程在并发场景下更加可靠，避免了因 Cargo 缓存竞争导致的构建不确定性，为 CI/CD 的高效稳定奠定了基础。

### Vram semaphore infra (#44465)
**SHA**: `af16446` | 🔗 [查看提交](https://github.com/vllm-project/vllm/commit/af16446bf39de047ab57649c933063cf1cbf1e50)

**🎯 变更类型**：功能增强 / 架构变更 / 性能优化  
**⚡ 重要程度**：🔴高  

**📋 变更摘要**  
1. 引入 **前端 GPU 多模态内存限流机制**：新增 `MultiModalGPUMemoryPool` 与全局单例，用于在 API‑server（前端）进程中对硬件视频解码占用的 GPU 内存进行预算控制。  
2. 在模型配置 (`ModelConfig`、`MultiModalConfig`、`EngineArgs`) 中加入 `mm_ipc_gpu_memory_gb` 参数，允许用户配置前端 GPU 解码的 GiB 预算。  
3. 在 `vllm.renderers.base.Renderer` 初始化时根据配置创建全局内存池；在 `GpuWorker` 中根据预算以及是否使用 **PyNvVideoCodec** 后端预留相应的解码器持久化显存与 CUDA 上下文占用。  
4. 为 **PyNvVideoCodec** 添加完整的 GPU‑side 视频解码实现，包括解码器复用槽、CUDA 流管理、原始帧字节计数获取、与新内存池的集成。  
5. 新增 `requirements/cuda.txt` 依赖 `PyNvVideoCodec==2.0.4`，并提供大量单元测试覆盖内存池、并发 acquire、解码后端行为以及与 API 进程数的交叉验证。

### 🔍 技术洞察

| 维度 | 影响评估 |
|------|----------|
| **架构影响** | • 在 API‑server 进程中新建了 **全局单例内存池**，实现了前端与后端之间的显存预算隔离，避免硬件视频解码抢占 KV‑cache。<br>• 通过 `maybe_init_mm_gpu_ipc_pool` 按 `api_process_count` 均分预算，兼容多 API‑server 进程的部署模型。<br>• `VideoBackend` 通过后端标识（`pynvvideocodec`）自动走 GPU 解码路径；其它后端保持不变，向后兼容。 |
| **性能影响** | • **正面**：前端解码被显式限流，防止因大量并发视频请求导致显存 OOM，从而提升整体稳定性。<br>• **负面**：在使用 `PyNvVideoCodec` 时，解码前需要 **acquire** 预算，会导致请求在显存紧张时被阻塞；在高并发、GPU 显存充足的场景下，阻塞时间可忽略。<br>• 解码器复用槽 (`PyNvVideoCodecDecoderSlot`) 限制为 1（当前实现），显著降低每帧解码的 CUVID/NVDEC 初始化开销，提升吞吐。 |
| **安全考虑** | • 新增的全局池使用 `threading.Condition` 实现，只有在同一进程内部共享，未引入跨进程共享，因此不存在额外的共享内存安全风险。<br>• 对外暴露的 API 仅是 `acquire`/`release`，并对负数或超限请求抛 `ValueError`，防止恶意或错误的显存预占。<br>• 引入 `PyNvVideoCodec` 依赖，需要确保该库在生产环境符合许可证（Apache‑2.0）并且不带有未审计的本地二进制。 |

### ⚠️ 潜在风险

1. **显存预留不足**  
   - 若用户配置的 `mm_ipc_gpu_memory_gb` 加上解码器/CUDA‑context 预留总和大于实际可用显存，`GpuWorker._reserve_mm_ipc_gpu_memory` 会抛 `ValueError` 并导致服务启动失败。需要在部署文档中明确显存预算计算方法。

2. **单解码器槽限制**  
   - 当前 `PYNVVIDEOCODEC_MAX_RETAINED_DECODERS = 1` 可能成为并发视频请求的瓶颈，导致大量线程在 `acquire` 后仍被 `borrow_decoder_slot` 阻塞。高并发视频工作负载下，需要通过调参或实现多槽策略来提升吞吐。

3. **跨进程显存使用不一致**  
   - 预算在每个 API‑server 进程均分，但实际使用的显存（decoder surfaces + CUDA context）是 **每进程** 固定的。如果不同进程的 GPU 负载不均衡，仍可能出现单进程 OOM。

4. **依赖兼容性**  
   - `PyNvVideoCodec` 需要 NVIDIA 驱动与对应的 CUDA 版本匹配；在某些驱动/CUDA 组合上可能出现 ABI 不兼容或运行时错误。需要在 CI 中加入针对不同驱动环境的兼容性测试。

5. **内存泄漏风险**  
   - `MultiModalGPUMemoryLease.release` 在异常路径中是幂等的，但如果用户自行保存 lease 对象而未进入上下文管理，且忘记调用 `release`，可能导致显存泄漏。建议在文档中强调 **使用 `with pool.acquire(...)`**。

6. **线程死锁**  
   - `acquire` 使用 `Condition.wait()`，若出现异常导致 `release` 未被调用，可能导致其他线程永久阻塞。虽然当前实现对负数/超限请求抛异常，但仍需确保所有代码路径（包括解码异常）都能执行 `lease.release()`。
