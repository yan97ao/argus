### Add tensor IPC transfer mechanism for multimodal data (#32104)
**SHA**: `80b7088` | 🔗 [查看提交](https://github.com/vllm-project/vllm/commit/80b70884ebd6b5a297d4b87e05651ea2d7c9d116)

**🎯 变更类型**：功能增强 / 架构变更 / 性能优化  

**⚡ 重要程度**：🔴高  

**📋 变更摘要**  
本次提交为 vLLM 引入了面向多模态张量的 **IPC（进程间通信）传输机制**，实现了基于 `torch.multiprocessing.Queue` 的零拷贝共享内存（CPU / CUDA）传递。新增 `MMTensorIPC` 配置项（`direct_rpc`、`torch_shm`），在模型配置、命令行参数、Engine 启动流程、序列化层（MsgpackEncoder/Decoder）以及 Core‑Client、Core‑Proc 中加入对应的发送方/接收方实现。同步测试覆盖了单/多发送者、CPU 与 CUDA 张量、空值、并发缓冲等场景，验证了在多 API‑Server 与单 Engine Core 拓扑下的正确性。

---

### 🎯 影响范围
- **核心模块**：`vllm/v1/engine/tensor_ipc.py`（全新实现）  
- **序列化**：`vllm/v1/serial_utils.py`（Encoder/Decoder 支持 OOB 张量消费/提供）  
- **配置系统**：`vllm/config/multimodal.py`、`vllm/config/model.py`、`vllm/config/vllm.py`（新增 `mm_tensor_ipc` 选项及默认值）  
- **启动入口**：`vllm/entrypoints/cli/serve.py`、`vllm/v1/engine/utils.py`、`vllm/v1/engine/core.py`、`vllm/v1/engine/core_client.py`（传递 `tensor_queue`，在 Engine 与 API‑Server 之间共享）  
- **测试**：`tests/v1/test_serial_utils.py`、`tests/v1/test_tensor_ipc_queue.py`（覆盖 IPC 正确性、并发、CPU/GPU、缓冲、错误回退等）  

---

### 🔍 技术洞察  

| 维度 | 影响分析 |
|------|----------|
| **架构影响** | - 引入 **TensorIpcSender / TensorIpcReceiver**，将张量的传输从传统的 **msgpack 体内序列化**（会产生拷贝）切换为 **共享内存队列**，在多模态推理（图片、视频、音频）中可显著降低跨进程传输开销。<br>- 在 `launch_core_engines` 中统一创建 **单一 `tensor_queue`**，所有 API‑Server 使用同一队列，`EngineCore` 通过 `TensorIpcReceiver` 按需缓冲和匹配张量句柄，实现 *N 发送者 → 1 接收者* 的流控模型。<br>- 新增的 `MMTensorIPC` 配置让用户在 **`direct_rpc`（默认）** 与 **`torch_shm`** 两种传输方式间切换，保持向后兼容。<br>- 对 `EngineCore` 与 `EngineCoreClient` 的 IO 线程做了统一的 **OOB Tensor Provider** 注入，避免在普通请求路径中出现不必要的序列化路径分支。 |
| **性能影响** | - 对 **CUDA / CPU 多模态张量** 采用 **共享内存零拷贝**（`share_memory_`），消除了 `msgpack` 中的 **大块拷贝** 与 **pickle** 回退路径，理论上可将张量传输成本从 O(N·size) 降至 O(N)（仅句柄传递）。<br>- `TensorIpcReceiver` 实现了 **drain‑and‑buffer** 策略，能够在高并发（多 API‑Server）情况下保持顺序一致性，避免因生产者乱序导致的阻塞。<br>- 引入的 `size_threshold` 仍保留对小张量的内联序列化，避免对极小张量创建共享内存的额外开销。<br>- 额外的 **队列同步**（`queue.get(timeout=10)`）会在极端情况下增加延迟，但在正常负载下延迟可忽略。 |
| **安全考虑** | - 通过 `OOBTensorConsumer`/`OOBTensorProvider` 的抽象层，张量句柄的生成与解析均在受控代码路径中完成，未向外部暴露句柄结构，安全风险较低。<br>- `torch_shm` 模式在 **DP>1**、**TP>1**、**PP>1** 场景下被显式禁止，防止跨 rank 的句柄冲突或共享内存泄漏。<br>- 增加了对 **`VLLM_WORKER_MULTIPROC_METHOD=spawn`** 环境变量的检测，确保在 `fork` 模式下不会因共享内存句柄失效导致错误或安全隐患。 |
| **可维护性** | - 新增的 `tensor_ipc.py` 代码量适中（≈180 行），遵循明确的 **接口分离**（发送/接收），易于单元测试与后续扩展。<br>- 序列化层的 `OOBTensorConsumer` 与 `OOBTensorProvider` 采用抽象基类，保持了向后兼容的同时为未来自定义传输方式提供了扩展点。<br>- 配置项的加入保持了 **TypedDict** 与 **field_validator** 机制，默认值仍为 `direct_rpc`，已有用户无需改动。<br>- 在 `EngineCore`、`CoreClient` 中对 `tensor_ipc_receiver`/`sender` 的实例化做了 `None` 检查，降低空指针风险。 |

---

### ⚠️ 潜在风险  
1. **多进程共享内存竞争**：在极端并发（大量 API‑Server）下，`torch.multiprocessing.Queue` 的内部锁可能成为瓶颈；如果队列积压导致 `put` 超时，当前实现会回退到普通序列化，产生突发的复制开销。  
2. **进程退出顺序**：发送进程在 `TensorIpcSender` 发送完张量后需要保持存活直到接收方取走所有共享张量。若进程过早退出（异常、超时），共享内存句柄可能失效，导致 `receiver.get` 报错或返回错误张量。已有测试通过 `retrieval_done` 事件同步，但生产环境需确保对应守护进程的生命周期管理。  
3. **跨设备混用限制**：`torch_shm` 仅在 **DP=1、TP=1、PP=1** 时可用；如果用户在配置文件中手动开启 `mm_tensor_ipc=torch_shm` 而开启了 DP/TP，启动时会抛出 `ValueError`。文档需明确提示。  
4. **GPU 共享内存可见性**：在部分平台（尤其是 Windows）上，CUDA 共享内存的句柄跨进程可能受到驱动限制，导致 `torch.cuda.is_available()` 为 True 但实际 IPC 失败。当前实现仅在 `torch.cuda.is_available()` 下运行相应测试，仍需在生产环境进行验证。  
5. **资源泄漏**：`TensorIpcReceiver` 中 `_tensor_buffers` 在处理完所有 tensors 后不会自动清理 `sender_id` 对应的字典，可能在长期运行的服务中残留无用的键值对。虽然影响不大，但可考虑在 `tensor_id` 递增到一定阈值后进行显式清理。  

---

### 💡 关注建议  
- **监控与告警**：在服务启动阶段检测 `mm_tensor_ipc` 与 `parallel_config` 的兼容性，若不匹配立即抛出可读错误；运行时监控 `tensor_queue.qsize()`，当队列长度长期偏高时预警可能的吞吐瓶颈。  
- **优雅停机**：在 API‑Server 与 Engine 进程的销毁流程中，明确调用 `tensor_queue.close()` + `tensor_queue.join_thread()`，并确保所有发送端在关闭前已发送完毕（可通过 `retrieval_done` 事件或类似机制）。  
- **文档与示例**：在官方文档中补充 `--mm-tensor-ipc torch_shm` 的使用前提（DP/TP 必须为 1、需使用 `spawn` 启动方式），并提供一段最小化的 “API‑Server ↔ Engine” 示例代码，帮助用户快速验证本地环境是否支持。

总体而言，改动集中在把缓存的多模态向量并入 Draft‑Model 前向，逻辑清晰且保持了原有接口。只要做好兼容性检查、显存控制以及测试覆盖，即可安全合并。

4. **回归测试**：运行完整的 MoE 单元测试，尤其是 `test_deepep_*`、`test_flashinfer_*`，确认在不同平台（CUDA、Blackwell、NIXL）下仍能正确检测可选依赖并完成动态导入。  
5. **代码审计**：检查是否还有残留的旧路径引用（如在其他非测试代码、CI 脚本或示例 notebooks），及时清理。  

总体来看，此次重构仅是文件层级的整理，对运行时行为无直接影响，但对 import 路径依赖较强的外部使用者需做好迁移准备。若加入兼容层，可在不破坏现有生态的前提下完成结构优化。
