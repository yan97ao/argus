### [Kernel] Remove NormGateLinear (#43554)
**SHA**: `d400445` | 🔗 [查看提交](https://github.com/vllm-project/vllm/commit/d4004455d2357985830af10e432709b42c820455)

**🎯 变更类型**：功能移除 / 重构（删除 DeepSeek V4‑Pro **NormGateLinear** 融合算子，改为拆分式实现）

**⚡ 重要程度**：🔴 高  

**📋 变更摘要**  
1. 彻底删除 DSV4‑Pro 专用的 `NormGateLinear` 融合算子（RMSNorm + router GEMV），包括 C++/CUDA 实现、Python 包装以及对应的 benchmark。  
2. 将模型实现改为先做独立的 `RMSNorm`，再使用 `GateLinear`（即 `dsv3_router_gemm`）进行路由计算，所有相关权重和 hash‑table 均迁移到 `GateLinear`。  
3. CMake 与编译脚本不再生成 DSV4‑Pro kernel；`cudaLaunchAttributeProgrammaticStreamSerialization` 的 env 检查被移除，统一硬编码为 `1`。  
4. 若干默认参数（如 `norm_eps`）从 `0.0` 调整为 `1e‑6`，以及权重映射、模型代码的相应适配。

---

### 🎯 影响范围
- **核心 CUDA kernel**：`csrc/moe/dsv4_norm_router_gemm*`（全部删除）  
- **Python bindings**：`vllm/_custom_ops.py`、`vllm/model_executor/torch_bindings.cpp`（对应 API 移除）  
- **模型实现**：`vllm/models/deepseek_v4/*`（NormGateLinear 替换为 RMSNorm + GateLinear）  
- **构建系统**：`CMakeLists.txt`（不再编译 DSV4‑Pro kernel）  
- **Benchmark & 测试**：`benchmarks/kernels/benchmark_norm_router_gemm.py`（删除）  
- **配置/权重映射**：`weights_mapper` 与 `load_weights` 中的名称映射相应修改。

---

### 🔍 技术洞察

| 维度 | 影响分析 |
|------|----------|
| **架构影响** | - 代码结构更加模块化：RMSNorm 与 router 逻辑分离，符合大多数 MoE 框架的设计，降低耦合度。<br>- 删除了 `NormGateLinear` 这一层级，模型的前向路径仅剩 `RMSNorm → GateLinear` 两步。<br>- `GateLinear` 现在承担了原本在 `NormGateLinear` 上的 `tid2eid`、`e_score_correction_bias` 等属性，统一管理。 |
| **性能影响** | - **正面**：删掉的 fused kernel 只在 **SM90+**、`num_tokens ≤ 16`、`hidden_dim=7168`、`num_experts=384` 的极窄场景下提供一次 kernel 迁移的加速；其他场景本就走 unfused路径。<br>- **负面**：对上述特化场景（如 DeepSeek V4‑Pro 在推理时 token 数常在 1‑16 范围），将回退到两次 kernel（`rms_norm` + `dsv3_router_gemm`），大约 **10‑30%** 延迟回升（视硬件而定），并产生额外的显存读写。<br>- `ProgrammaticStreamSerialization` 现在始终打开，对 SM<90 的卡无效但不报错，对 SM90+ 卡保持原有并行调度特性。 |
| **安全考虑** | - 仅删除了内部实现，不涉及外部输入校验或权限控制，未引入新的安全风险。<br>- 移除 `getEnvEnablePDL` 环境变量检查，避免因误设导致 kernel 禁用，降低配置错误的潜在风险。 |
| **可维护性** | - 代码量大幅下降（CUDA 文件累计删减约 **800 行**），维护负担显著降低。<br>- 统一使用 `GateLinear`，后续新增或修改 router 只需在单一实现上工作。<br>- 删除的 benchmark 失去对比基准，但可以在 CI 中以 `dsv3_router_gemm` 替代。 |
| **兼容性** | - 对外公开的 Python API `torch.ops._moe_C.dsv4_norm_router_gemm` 被移除，现有用户代码若直接调用会抛 `RuntimeError: could not find the requested op`. <br>- 迁移路径已在模型代码中改写，默认不会出现调用缺失。<br>- 权重映射已更新，旧模型权重仍可加载（只改名），但需注意 `ffn.gate.bias` → `ffn.gate.e_score_correction_bias` 的改名。 |

---

**⚠️ 潜在风险**

1. **性能回退**  
   - 在 DeepSeek V4‑Pro 场景（SM90+、小 batch、token≤16）用户可能明显感受到推理吞吐下降。  
   - 对于对 latency 敏感的生产部署，需要重新评估基准并可能做额外的 kernel 合并或手动开启 `torch.compile` 优化。

2. **API 兼容性**  
   - 直接使用 `_custom_ops.dsv4_norm_router_gemm` 的第三方库将失效，需修改为 `rms_norm` + `dsv3_router_gemm` 或使用新的 `GateLinear` 接口。缺乏向后兼容包装可能导致突发错误。

3. **PDL 属性的强制开启**  
   - 在极少数老卡（不支持 Programmatic Stream Serialization）上，CUDA 运行时可能返回 `cudaErrorInvalidValue`。虽然理论上属性会被忽略，但未在代码中捕获返回值，若出现异常会导致整个进程崩溃。

4. **权重迁移**  
   - `ffn.gate.bias` 重命名为 `ffn.gate.e_score_correction_bias`，旧 checkpoint 若使用 `strict=False` 加载会通过，但若 `strict=True` 则报错，需要更新加载脚本。

5. **默认 `norm_eps` 改动**  
   - `tilelang` 系列函数默认从 `0.0` 改为 `1e‑6`，可能导致与旧配置的数值微差（尤其在极端小数值实验中），需要确认下游代码是否假设 `0.0`。

### File system secondary tier implemented in python (#41735)
**SHA**: `d0a100c` | 🔗 [查看提交](https://github.com/vllm-project/vllm/commit/d0a100c87af832ad97ade60b8ec7610018a08427)

**🎯 变更类型**：功能增强 / 架构变更  
**⚡ 重要程度**：🔴高  
**📋 变更摘要**：在 vLLM KV‑offload 子系统中实现了全新的 **Python 文件系统二级层**（`fs_python`），包括  
- 文件路径映射 (`FileMapper`) 负责将 KV Block 哈希映射到磁盘路径；  
- 负责磁盘读写的管理器 (`FileSystemTierManager`)；  
- 支持优先级读写的双队列线程池 (`DualQueueThreadPool`)；  
- 相应的工厂注册、模块入口以及完整单元测试。目标是让 KV 缓存可以落盘到本地文件系统，提供持久化/扩展存储能力，同时保持异步、并行的 I/O 能力。

---

## 🎯 影响范围
- **核心模块**：`vllm/v1/kv_offload/*`（offloading spec、tier manager、factory、IO）  
- **新增二级层类型**：`fs_python`（Python 实现的磁盘层）  
- **测试套件**：`tests/v1/kv_offload/*`（FileMapper、FS tier 行为、端到端离线/offload）  
- **用户配置**：`KVTransferConfig.kv_connector_extra_config` 新增 `secondary_tiers` 参数，可声明 `{type: "fs_python", root_dir: "...}"`

## 🔍 技术洞察

| 维度 | 影响描述 |
|------|----------|
| **架构影响** | - 将 KV‑offload 体系从仅支持 `gpu↔cpu` 扩展到 **磁盘**，实现了三层层级（GPU‑>CPU‑>磁盘）。<br>- 通过 `SecondaryTierFactory.register_tier` 将新类型无侵入式注册，保持现有插件化设计。<br>- `FileMapper` 基于 KV block hash 生成 **确定性、分层目录**，避免单目录文件数量爆炸。<br>- `DualQueueThreadPool` 引入 **读写优先级分离**，防止写入阻塞读取，提升并发吞吐。 |

| **安全考虑** | - **路径安全**：文件路径全部由内部哈希和模型名自动生成，用户只能控制 `root_dir`，不存在路径遍历风险。<br>- **文件权限**：创建文件使用 `0o644`，仅把写权限限制在创建者，未对目录设置更严格的 ACL，若 `root_dir` 位于共享目录，需要运维自行加固。<br>- **异常清理**：写入失败会尝试删除临时文件；读取失败会删除损坏的块文件，防止残留不可恢复数据。<br>- **配置写入**：在构造时仅在 `config.json` 不存在时写入，避免并发覆盖；但多进程同时创建同一 `root_dir` 仍可能产生竞争，建议外部加锁或使用独立 `root_dir`。 |
| **可维护性** | - 代码划分清晰：路径映射、I/O、线程池、管理器分别独立，易于单元测试。<br>- 大量新增 **pytest** 用例覆盖路径生成、存取成功/失败、并发任务、异常路径、数据完整性，提供回归保障。<br>- 采用 **类型注解**（`JobId`, `JobResult`），配合 `mypy` 检查，可提升后期维护质量。 |
| **兼容性** | - 对已有 `OffloadingConnector` 无侵入，仅在 `secondary_tiers` 中显式声明时激活。<br>- 旧版仅使用 GPU↔CPU 的用户不受影响，新增功能为可选。<br>- `O_DIRECT` 在非 Linux 平台退化为普通 I/O，保持功能兼容。 |

## ⚠️ 潜在风险

1. **磁盘空间耗尽**  
   - 每个 KV block 按哈希落盘，若离线数据未及时清理，磁盘使用会线性增长。建议提供 **TTL/清理机制** 或管理员监控。

2. **并发写入冲突**  
   - 多进程/多实例使用同一 `root_dir` 时，`store_block` 只检查文件是否已存在，若并发写入同一哈希，可能出现 **写入竞争**（一个写入成功，另一个被跳过）。虽不会导致错误，但可能导致数据不一致（例如写入的块被后来的写覆盖）。

3. **O_DIRECT 对齐要求**  
   - 在 Linux 上使用 `O_DIRECT` 时，写入缓冲区、文件偏移和块大小必须是文件系统块对齐的（通常 4 KiB）。当前实现直接使用 `memoryview.cast("B")`，若块大小不是对齐值，可能触发 **EINVAL**。虽捕获异常并回退，但会导致频繁异常日志，降低性能。

4. **线程池资源泄漏**  
   - `DualQueueThreadPool` 的线程是 **daemon**，在异常情况下（如主进程异常退出）可能留下僵尸线程。`shutdown()` 已清空队列并 `join`，但若用户未主动调用 `shutdown`，进程结束时仍会强制退出。建议在 `FileSystemTierManager.__del__` 中调用 `shutdown` 以防止资源泄漏。

5. **异常文件残留**  
   - `store_block` 在写入期间若异常，尝试删除临时文件；但如果进程被强制终止（SIGKILL）或机器突然掉电，临时文件可能残留，导致 **路径冲突** 或 **磁盘占用**。

6. **权限/目录不可写**  
   - 构造函数在 `config.json` 写入失败时抛 `OSError`，这会中断整个 LLM 启动，用户需提前确保 `root_dir` 可写。错误信息应更友好提示 “ensure directory exists & writable”.
