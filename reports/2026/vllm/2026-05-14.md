### [RFC] Replace shared-memory routed experts with ModelRunnerOutput transfer and HTTP support (#39568)
**SHA**: `c7560af` | 🔗 [查看提交](https://github.com/vllm-project/vllm/commit/c7560af42487b1570c4e6f4cea5df1605a4d59fc)

**🎯 变更类型**：功能增强 / 架构变更 / 性能优化  
**⚡ 重要程度**：🔴 高  

**📋 变更摘要**  
1. 将原先基于进程共享内存的 *routed‑experts* 捕获/读取机制彻底移除，改为在 **GPUModelRunner** 中直接把专家路由信息随 `ModelRunnerOutput` 传输到调度器（CPU）侧。  
2. 新增 `RoutedExpertsCapturer`（仅负责 GPU 端捕获）、`RoutedExpertsManager`（调度器侧基于 KV‑cache slot 的 CPU 缓冲区）以及对应的 `RoutedExpertsTensors / RoutedExpertsLists` 数据结构。  
3. 在 `vllm/config/vllm.py` 中加入配置校验，禁止在开启 `--enable-return-routed-experts` 时使用 Pipeline Parallelism 或任意 KV‑connector。  
4. HTTP 接口（OpenAI‑compatible）现在通过 Base64 编码的 **.npy** 字节返回每个 token 的路由决策 (`routed_experts` 字段)。  
5. 相关测试、采样参数、调度、输出处理等代码同步更新，以支持新机制并保持向后兼容。  

**🎯 影响范围**  
- `vllm/model_executor/layers/fused_moe/routed_experts_capturer.py`（GPU 捕获实现）  
- `vllm/v1/core/sched/scheduler.py`、`vllm/v1/engine/output_processor.py`、`vllm/v1/outputs.py`（调度器与输出路径）  
- `vllm/entrypoints/serve/disagg/serving.py`、`vllm/entrypoints/serve/disagg/protocol.py`（API 层）  
- `vllm/config/vllm.py`（配置校验）  
- `vllm/sampling_params.py`（新增 `routed_experts_prompt_start` 参数）  
- 单元测试文件 `tests/model_executor/test_routed_experts_capture.py`、`tests/v1/core/test_async_scheduler.py`、`tests/v1/core/test_scheduler.py`

## 🔍 技术洞察  

### 架构影响
| 维度 | 影响描述 |
|------|----------|
| **捕获路径** | 原来的 **共享内存 + 文件锁** 实现被废除，消除了跨进程同步（`fcntl`、`shared_memory`、锁文件）以及资源泄漏风险。现在捕获仅在同一进程内完成，数据通过 `ModelRunnerOutput` 结构在 GPU → CPU（复制流）后交给调度器。 |
| **调度器侧缓冲** | 新增 `RoutedExpertsManager`，使用 **CPU 大块 NumPy 缓冲区**（按 KV‑cache slot 索引）持久保存路由信息，避免频繁跨进程共享内存映射。缓冲区大小等于 `num_blocks * block_size * num_layers * top_k * dtype`，在大模型（数十 GB KV）下仍在几 GB 量级，可通过日志监控。 |
| **数据流** | - **GPU 端**：`RoutedExpertsCapturer` 将 `topk_ids` 写入 `device_buffer`；在 `GPUModelRunner` 的 `execute_model` 结束时 `clear_buffer` 并 **clone** 出当前 step 的子张量。<br>- **复制流**：`AsyncGPUModelRunnerOutput` 在独立 CUDA stream 上执行非阻塞 D2H（`to_cpu_nonblocking`），随后包装为 `RoutedExpertsLists`。<br>- **CPU 端**：`Scheduler.update_from_output` 调用 `RoutedExpertsManager.store_batch` 把 `routing_data` 按 `slot_mapping` 写入全局 slot 缓冲区；随后在请求结束时 `RoutedExpertsManager.get` 根据块 ID 读取对应路由切片。 |
| **并发模型** | 为 **单 DP / 单 PP** 环境设计（配置校验已阻断 `pipeline_parallel_size > 1`），并且不兼容任意 **KV‑connector**（PD‑disaggregation、offload）。这保证了路由信息始终能在本地 GPU 内存与 CPU 同步，否则会出现 *slot‑index* 与实际 KV 块不匹配的错误。 |
| **API** | HTTP JSON 响应中新增 `routed_experts`（Base64‑encoded `.npy`），并在 `GenerateResponseChoice` 中对文档进行说明。客户端需要自行 `base64 → np.load` 解码。|

### 性能影响
| 维度 | 正面影响 | 负面/注意点 |
|------|----------|--------------|
| **共享内存锁** | 完全移除文件锁和跨进程共享内存的创建/销毁开销，尤其在高并发部署时可显著降低 I/O 与系统调用次数。 | 无 |
| **GPU→CPU 复制** | 使用 **pinned memory** + **独立 copy stream** 实现非阻塞 D2H，理论上复制延迟被隐藏在后续调度/采样计算中。<br>对同步调度路径仍保持 `copy_` + `non_blocking=True`，但复制仍会在默认流完成后才真正开始。 | 复制流与默认流的同步点（`event.synchronize()`）会在 `_to_list` 中出现，若复制量大（数十 MB）可能成为瓶颈。可以通过调大 `pin_memory` 或在 GPU 侧压缩 dtype（已使用 `int32`）来缓解。 |
| **CPU 端存储** | 统一使用 **NumPy** 大块数组进行一次性 **fancy‑index** 写入，相比之前逐个进程写共享内存的锁竞争提升了几倍吞吐。 | 内存占用随 `num_blocks` 成比例增长，极端配置（如 1M KV‑blocks）可能导致几 GB 甚至十几 GB RAM 使用，需要在部署时监控。 |
| **网络传输** | 路由信息通过 **Base64 + .npy** 发送，避免了大量 JSON 字段拼接的开销且可一次性压缩。 | Base64 会把二进制体积膨胀约 33%，对大批量 token（如 2k token）返回 `routed_experts` 可能显著增加响应体积，建议仅在需要时开启（`enable_return_routed_experts`）。 |

### 安全考虑
- **配置校验**：在 `VllmConfig.__post_init__` 中加入明确的 `ValueError` 检查，防止在不兼容的并行模式下误开启功能，从而避免 **数据错位**、**内存越界** 或 **信息泄露**。  
- **HTTP 编码**：使用 **Base64** 编码后再放入 JSON，防止 JSON 注入或非法字符导致解析错误。仍需注意 **payload size**，避免 DoS 攻击（可在 API 层加 `max_response_body_size` 限制）。  
- **共享内存删除**：已彻底去除共享内存，消除了跨进程残留文件/内存泄漏的潜在攻击面。

### 🔍 技术洞察

| 维度 | 影响说明 |
|------|----------|
| **架构影响** | - 将 **RMSNorm** 与 **Router GEMV** 融合为单一 kernel，消除两次显式 kernel launch 与中间全局 memory 写/读。<br>- 只在 **SM90+（CUDA ≥ 12）** GPU 上编译，使用 **PDL（Pipeline‑Level Dataflow）** 与 **warp‑butterfly reduction**，保持与已有 DSV3 router GEMM 实现一致的计算图。<br>- 新增 `NormGateLinear` 层，取代原 `GateLinear + RMSNorm` 的拼装方式，保持 API 向后兼容（在 token 数 > 16 时自动回退到 unfused 路径）。<br>- 权重量映射在 `DeepseekV4` 及其 MTP 版本中同步迁移至 `norm_gate`，减少模型权重文件的结构改动。 |
| **性能影响** | - **延迟**：在常见的低‑token‑数推理场景（batch‑wise token 数 ≤ 16），预计 **单次 kernel launch** 可比原两次 launch（RMSNorm + dsv3_router_gemm）快 **30‑45%**（受限于内存访次数减少与共享 L2 使用提升）。<br>- **吞吐**：对大 batch、`num_tokens` > 16 仍走 unfused路径，性能不受负面影响。<br>- **资源占用**：kernel 采用 128 线程 block、使用 **shared memory** 进行跨 warp 归约；对 SM90+ 设备占用约 **30KB** SMEM，仍在安全范围内。<br>- **兼容性**：仅在 **SM90–SM103**（A100‑H100‑等）上可用，其他 GPU 自动回退至原实现，保持功能完整。 |
| **安全考虑** | - 新增的 C++/CUDA 代码没有外部输入解析，仅接受张量指针与标量 `eps`，风险主要来自 **类型/shape 检查**。代码已在入口处使用 `TORCH_CHECK` 强制验证维度、dtype、contiguity，从而防止 **非法内存访问**。<br>- 由于 kernel 只在 **SM90+** 编译，老旧显卡无法加载，避免潜在的 **硬件不兼容崩溃**。<br>- 没有引入网络或文件 I/O，安全风险极低。 |

### ⚠️ 潜在风险

1. **硬件门禁**  
   - 仅在 **SM90+（A100, H100, etc.）** 上编译。若用户在低端 GPU（如 T4、RTX 3080）上运行，库会回退到 unfused 路径，但如果 CMake 未检测到合适 arch，可能导致 **编译失败**。  
   - 解决方案：在 CI 中加入对 “unsupported arch” 的警告测试，确保 `CMakeLists.txt` 的 `if(VLLM_GPU_LANG STREQUAL "CUDA")` 分支在不满足条件时明确提示。

2. **硬编码形状限制**  
   - Kernel 只支持 `num_tokens ∈ [1,16]`、`hidden_dim = 7168`、`num_experts ∈ {256,384}`（Pro/Flash）。如果模型配置或权重文件出现不同形状，会直接触发 `TORCH_CHECK` 抛异常。  
   - 运营时必须在模型加载前 **验证配置一致性**（已有检查在 `NormGateLinear.__init__` 中），但仍建议在模型构造阶段给出友好错误信息。

3. **数值一致性**  
   - 融合实现使用 `rsqrtf(s_sum / H + eps)`（除法而非乘以倒数）以匹配参考 RMSNorm，实现细节差异导致 **1‑2 ULP** 误差。基准已设定容差 `~1 bf16 ULP`，但在极端数值（超小/大）下可能出现 **可感知偏差**。  
   - 建议在正式发布前加入 **大规模随机数值回归测试**（不同种子、不同 eps）以及 **FP16 vs BF16** 对照。

4. **共享显存竞争**  
   - `normed_x` 与 `logits` 同时写入全局内存，若后续代码在同一流中对 `normed_x` 进行立即使用，可能出现 **隐式同步** 问题。当前实现使用默认 CUDA stream，确保顺序执行；若未来引入异步 pipeline，需要确认 `normed_x` 的可见性。

5. **代码重复/维护成本**  
   - 与 DSV3 router 代码的实现高度相似，若后续对 GEMV 部分进行优化，需要同步更新两个实现（容易遗漏）。可考虑抽象公共部分到 `dsv_common` 库，以降低维护负担。

整体来看，改动实现了灵活的插件式草稿生成，代码结构清晰、异常处理到位，唯一需要关注的是对实验性特性的使用约束与后向兼容性。
