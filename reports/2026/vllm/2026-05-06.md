### [Spec Decode] Add Gemma4 MTP speculative decoding support (#41745)
**SHA**: `27e0057` | 🔗 [查看提交](https://github.com/vllm-project/vllm/commit/27e0057aeda6bc443069c20fdf2f3cc95ed892f3)

**🎯 变更类型**：功能增强 / 架构变更 / 性能优化  
**⚡ 重要程度**：🔴 高  
**📋 变更摘要**  
- 为 **vLLM** 新增对 **Gemma‑4 MTP（Multi‑Token Prediction）** 规范解码的完整支持，包括模型实现、提议器、注册表、配置和测试。  
- 实现了 **KV‑cache 共享**、**常量草稿位置**、**多组 KV‑cache（全局+滑动）** 以及 **稀疏 logits（centroids‑masking）**，并在 GPU‑Runner 中接入新的 **Gemma4Proposer**。  

**🎯 影响范围**  
- `vllm/model_executor/models/gemma4_mtp.py`（全新模型实现）  
- `vllm/v1/spec_decode/gemma4.py`（全新提议器）  
- `vllm/config/speculative.py`（新增 `gemma4_mtp`、`use_gemma4_mtp` 判定）  
- `vllm/model_executor/models/registry.py`（模型注册）  
- `vllm/transformers_utils/model_arch_config_convertor.py`（模型架构转换）  
- `vllm/v1/worker/gpu_model_runner.py`（Runner 对接）  
- `vllm/v1/spec_decode/llm_base_proposer.py`（常量草稿位置、元数据更新抽离）  
- 测试扩展 `tests/v1/e2e/spec_decode/test_spec_decode.py`、`tests/models/registry.py`

## 🔍 技术洞察

| 维度 | 影响描述 |
|------|----------|
| **架构影响** | - 新增 **Gemma4MTPModel** 类型，模型内部由 **draft（assistant）** 与 **backbone（target）** 两套网络构成，draft 只保留 Q‑projection，**KV 通过 `kv_sharing_target_layer_name` 与 target 共享**。<br>- 为多组 KV‑cache（滑动注意力 + 全局注意力）提供 **`_per_group_block_tables`**，在 `GPUModelRunner` 中收集并注入到提议器。<br>- `SpeculativeConfig.use_gemma4_mtp()` 判定草稿模型是否为 `gemma4_mtp`，并在 `SpeculativeConfig.max_num_new_slots_for_drafting` 中加入特化逻辑。<br>- `SpeculativeBaseProposer` 新增 `constant_draft_positions`，用于 **所有草稿步骤使用同一位置**（Q‑only attention 必须如此），从而避免位置递增导致 KV 不匹配。 |
| **性能影响** | - **MTP** 能一次性预测多个 token（默认 1，assistant‑model 时 2），显著降低 **target‑model 前向次数**，在大模型上预期提升 30%‑60%（测试中 gemma4‑e4b 示例约 62%）。<br>- **KV‑sharing** 消除了 draft‑model 的 KV 存储，节省显存约 `draft_hidden_size * num_layers`（例如 256 × 24）。<br>- **Centroids‑masking**（可选）通过稀疏 logits 将全 vocab 计算缩减至 **≈ top‑k × (vocab/num_centroids)**，在大 vocab (≈ 32k) 场景下降低算子负载。<br>- 为 centroids 计算捕获 **CUDA Graph**（1‑64 token）以进一步削减 kernel 启动开销。<br>- 由于 draft‑model 采用 **TRITON_ATTN**（保持与 target‑model 一致），避免了 `FLASH_ATTN` 在 KV‑shared 场景下的兼容性问题。 |
| **安全考虑** | - 注册表为 `Gemma4MTPModel` 设置 `trust_remote_code=True`，仍然依赖 **HuggingFace remote code**；若用户在不受信环境下启用此模型，仍存在执行恶意代码的风险。<br>- 新增对 **Transformers ≥ 5.8.0** 的版本检查，若低于此版本直接跳过测试，避免因旧版本缺失字段导致崩溃。<br>- 代码路径中未出现显式的内存泄漏或未检查的外部输入。 |
| **可维护性** | - 通过 **`WeightsMapper`** 与 **`AutoWeightsLoader`** 实现权重映射，保持与其它模型一致的加载框架。<br>- `Gemma4MTPMaskedEmbedder` 独立封装稀疏 logits，逻辑清晰且可在未来复用。<br>- 在 `SpecDecodeBaseProposer` 中抽离的 `_update_positions_dependent_metadata` 使得位置更新逻辑统一，便于后续新增 “常量位置” 模型。<br>- 代码量较大（603 行新模型），但遵循现有 vLLM 模块划分，易于定位。 |

## ⚠️ 潜在风险

1. **KV‑共享映射错误**  
   - `Gemma4Proposer._setup_gemma4_kv_sharing` 通过层类型匹配最后一个非 KV‑shared 层。若目标模型层类型或 `num_kv_shared_layers` 改变，映射可能失配，导致生成错误或段错误。  
2. **常量草稿位置副作用**  
   - `self.constant_draft_positions` 在 `SpecDecodeBaseProposer` 中全局开启，若未来引入其他不需要该特性的草稿模型（但未显式关闭），会错误复用同一位置。  
3. **多组 KV‑cache 元数据同步**  
   - `GPUModelRunner` 在 `_build_attn_group_metadata` 中把每组 block table 注入 propos­er；若未来出现 **更多 KV‑cache 组**（例如新增局部‑KV 方案），仍需手动扩展 `Gemma4Proposer._setup_gemma4_kv_sharing`。  
4. **CUDA Graph 捕获限制**  
   - 捕获的图只覆盖 1‑64 token 大小，超过该范围将回退到普通执行，可能出现 **性能突变**。若模型在异常负载下频繁跨范围，图的回收/重新捕获成本未实现。  
5. **依赖 Transformers 5.8+**  
   - 在低版本环境会直接 `pytest.skip`，但在生产部署若未显式检查版本，加载模型时会抛出 `AttributeError`（如 `model_type` 字段缺失）。  
6. **内存峰值**  
   - Draft‑model 仍保留完整的 **embedding、MLP、Attention**，虽然 KV‑shared 减少缓存，但在 **TP（tensor parallel）** 场景下仍会占用一套模型参数。大规模部署时需评估显存足够性。

### fix(openai): tolerate empty content in forced tool choice (#40148)
**SHA**: `6467213` | 🔗 [查看提交](https://github.com/vllm-project/vllm/commit/6467213a9f118fcf5ebf65d5e1ee8d64d2e101e4)

**🎯 变更类型**：Bug修复  
**⚡ 重要程度**：🔴高  
**📋 变更摘要**：本次提交修复了在 OpenAI 兼容层强制工具选择（`tool_choice="required"` 或命名函数）时，若模型返回的 `content` 为 `None`（即空内容）导致的断言异常。通过放宽对 `content` 与 `tool_calls` 的非空检查，使得在 `content` 为 `None` 时能够安全返回空的 `tool_calls` 列表和 `None` 内容，保持与官方 OpenAI 行为一致。新增相应单元测试验证此行为。  

**🎯 影响范围**：  
- `vllm/entrypoints/openai/chat_completion/serving.py`（生成响应时的工具调用处理）  
- `vllm/entrypoints/openai/engine/serving.py`（`_parse_tool_calls_from_content` 逻辑）  
- `vllm/parser/abstract_parser.py`（抽象解析器的 `_parse_tool_calls` 实现）  
- 新增/修改的测试文件位于 `tests/entrypoints/openai/` 目录，覆盖 ChatCompletion 与 Responses 两条 API 路径  

**🔍 技术洞察**  
- **架构影响**：  
  - 变更局限在 OpenAI 接口层与抽象解析器内部的工具调用解析路径，未触及底层调度或模型执行模块，整体架构保持不变。  
  - 通过在强制工具调用分支加入 `content is None` 的 guard，使得工具调用解析更为宽容，提升了 API 兼容性。  

- **性能影响**：  
  - 仅在极少数异常路径（`content` 为 `None`）增加了两次 `if` 检查，CPU 与内存开销可以忽略不计。  
  - 由于避免了断言触发的异常抛出，极端情况下可减少异常捕获导致的额外栈展开开销。  

- **安全考虑**：  
  - 该修改不涉及权限、数据泄露或执行外部代码，仅是对空值的容错，未引入新的安全风险。  
  - 需要关注后续如果在其他模块仍然使用 `assert content is not None`，可能出现未捕获的异常，建议统一使用显式的空值检查。  

**⚠️ 潜在风险**  
1. **下游假设破裂**：如果其他代码（如自定义插件或第三方集成）仍然基于 `content` 必然非空进行业务处理，可能出现 `NoneType` 错误。  
2. **工具调用输出为空时的业务语义**：返回空 `tool_calls` 与 `None` 内容在某些业务场景下可能被误解为“未生成任何响应”，需确保调用方能够正确区分“长度截断”与“正常空响应”。  
3. **测试覆盖不足**：目前仅覆盖了强制工具选择的两条 API（ChatCompletion 与 Responses），若未来出现其他入口（如流式接口）也需同步容错。  

**💡 关注建议**  
- **文档与使用说明**：在项目的 OpenAI 接口文档中标注 “当 `tool_choice` 为强制调用且返回 `content` 为 `null` 时，返回空 `tool_calls`，与官方 OpenAI 行为保持一致”。  
- **统一空值检查**：在所有解析器实现中将 `assert content is not None` 替换为显式的 `if content is None: return [], None`，防止未来出现类似断言导致的崩溃。  
- **回归测试**：在 CI 中加入流式 `chat/completions` 与 `responses` 的空内容场景测试，确保该容错在所有路径均生效。  
- **监控与告警**：在生产环境对 `tool_calls` 为 `[]` 且 `content` 为 `None` 的响应事件进行统计，若比例异常升高，可快速排查模型生成异常。

### nixl refactor: new transfer design (#40731)
**SHA**: `df8e63f` | 🔗 [查看提交](https://github.com/vllm-project/vllm/commit/df8e63f4edabc06e51c10fe479fc9b95c303542f)

**🎯 变更类型**：重构 / 架构变更  
**⚡ 重要程度**：🔴高  
**📋 变更摘要**：本次提交对 vLLM 的 NIXL KV‑cache 传输层做了大幅重构，核心改动包括  
1. 引入 **TPMapping** 统一计算本地‑远程 TP（tensor‑parallel）映射，取代原先散落在 `TransferTopology` 与 `MambaEngineTransferInfo` 中的复杂逻辑。  
2. 将 `NIXL_CONNECTOR_VERSION` 升级至 3，并在 `NixlAgentMetadata` 中加入 `physical_blocks_per_logical_kv_block` 字段，以统一物理/逻辑块比例的概念。  
3. 合并并精简描述符 ID、split‑handle 以及本地/远程 block 描述的生成过程，删除了大量专门针对 Mamba‑HMA 的分支代码。  
4. 大量单元测试随之更新，覆盖了新的 TP 映射、描述符计算以及兼容性校验。  

**🎯 影响范围**：  
- `vllm/distributed/kv_transfer/kv_connector/v1/nixl/` 目录下的 **所有核心组件**（`worker.py、metadata.py、tp_mapping.py、utils.py`）。  
- 依赖 NIXL KV‑cache 传输的 **模型推理路径**（包括 FlashAttention、Sliding‑Window、Mamba）。  
- 与旧版 hand‑shake 协议兼容的 **远程引擎**（通过 `compatibility_hash` 校验）。

### 🔍 技术洞察

| 维度 | 影响分析 |
|------|----------|
| **架构影响** | • **统一抽象**：`TPMapping` 把原本分散在 `TransferTopology` 与 `MambaEngineTransferInfo` 的“源 rank / 读数 / head‑slot” 逻辑集中到一个数据类，降低了模块耦合度。<br>• **职责划分更清晰**：`TransferTopology` 只负责 TP 比例、远端信息登记与基本查询；`TPMapping` 负责真实的映射计算，包括 attention‑group 与 SSM‑group 的不同处理。<br>• **元数据演进**：`NixlAgentMetadata` 通过新字段 `physical_blocks_per_logical_kv_block` 让物理块比例在握手阶段就确定，避免后期在 `Worker` 中再进行推导。 |
| **性能影响** | • **描述符 ID 计算向量化**：`_compute_desc_ids` 使用 NumPy 全向量运算，取代了之前多层循环，理论上可降低 CPU 端的描述符准备时间，尤其在大模型（>50B）上块数上万时收益明显。<br>• **分片句柄构建**：`_build_local_splits_from_plan` 通过一次遍历生成所有 split‑handle，去除了旧实现中对 Mamba‑HMA 需要两套不同 split‑handle 的繁琐路径，减少了内存复制和 Python 调用开销。<br>• **去除冗余分支**：大量 `if self._has_mamba` / `if tp_ratio < 0` 逻辑被简化，分支预测和解释器开销相应下降。<br>• **潜在回退**：在同构 TP（`tp_ratio == 1`）情况下仍会走 4‑region 方案（FA+Mamba），未实现“单 descriptor”快速路，仍有进一步优化空间。 |
| **安全考虑** | • **兼容性校验**：`NIXL_CONNECTOR_VERSION` 升级后，远端仍通过 `compatibility_hash` 做防篡改校验，未出现新增安全漏洞。<br>• **信息泄露**：新增 `physical_blocks_per_logical_kv_block` 直接暴露了模型内部块划分细节，但该信息本来在 `handshake` 中已通过 `block_size`、`block_len_per_layer` 传递，风险可接受。<br>• **错误处理**：新增的 `TPMapping` 计算函数对异常情况（如无合法 source ranks）使用 `assert`，在生产环境中可能抛出 `AssertionError` 导致 worker 崩溃；建议改为抛出自定义异常并记录错误。 |
| **可维护性** | • **代码量大幅压缩**：`utils.py` 中约 375 行被删除，`worker.py` 中约 417 行删除，整体文件行数下降约 25%。<br>• **单元测试覆盖**：新增 `tests/v1/kv_connector/unit/test_tp_mapping.py` 对映射逻辑进行详细断言，降低回归风险。<br>• **模块化**：`tp_mapping.py` 成为独立的公共库，后续如果引入新的 KV‑cache 规格（如 Mix‑Attention）可以在此层扩展，而不必改动 `TransferTopology`。 |

### ⚠️ 潜在风险

1. **映射计算错误**  
   - `compute_tp_mapping` 对 **GQA（grouped‑query attention）** 场景的 “head dedup” 采用 `np.unique`，如果 `total_num_kv_heads` 与 `remote_tp_size` 之间存在非整除关系，可能导致 **遗漏** 某些 remote ranks。未在测试中覆盖的极端比例（如 7 heads / 8 remote TP）需要额外验证。  

2. **分割句柄边界**  
   - `fa_num_splits` 与 `ssm_num_splits` 基于 `source_ranks_per_group` 长度计算，若某组是空列表（例如不存在 SSM）则 `ssm_num_splits` 为 0，后续 `chunk = local_len // ssm_num_splits` 会触发 ZeroDivisionError。当前代码在 `has_ssm_descs` 判断后才使用，但仍建议在 `plan` 生成阶段确保每组都有至少一个 source rank。  

3. **异常路径缺失**  
   - 多处仍保留 `assert`（如 `assert self.transfer_topo is not None`、`assert len(local_block_descs_ids) == len(remote_block_descs_ids)`），在异常环境下会导致 worker 被杀。生产环境建议改为可恢复的错误报告。  

4. **性能回退**  
   - 对 **同构 TP**（`tp_size == remote_tp_size`）场景，仍会走 **4‑region**（FA+Mamba）路径，导致不必要的 descriptor 数量提升，可能在极端带宽受限的网络上产生轻微回退。  

5. **兼容性冲突**  
   - 旧版远端进程在 handshake 时若未升级，会因为 `NIXL_CONNECTOR_VERSION` 不匹配而直接 abort，可能导致集群中混合版本的滚动升级受阻。建议在部署脚本中加入版本检查或回退兼容层。
