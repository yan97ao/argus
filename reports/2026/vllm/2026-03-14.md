### fix: resolve chat template names before kwargs detection (#36937)
**SHA**: `b41aa26` | 🔗 [查看提交](https://github.com/vllm-project/vllm/commit/b41aa264f9ec0f3d2d47ec8e0a136305dafbbe4a)

**🎯 变更类型**：Bug修复  
**⚡ 重要程度**：🔴高  
**📋 变更摘要**：  
1. 在 `resolve_chat_template` 中加入对 **模板名称**（如 `"tool_use"`）的解析，改为调用 `tokenizer.get_chat_template` 返回实际的 Jinja 模板文本。  
2. 为该行为新增单元测试，验证名称解析以及在已解析为 Jinja 内容后，`resolve_chat_template_kwargs` 能正确保留合法的模板变量而过滤未知参数。  

**🎯 影响范围**：  
- `vllm/renderers/hf.py` 中的 `resolve_chat_template` 与 `resolve_chat_template_kwargs`  
- 依赖这些函数的所有聊天模板渲染路径（如 `apply_chat_template`、`ChatTemplateRenderer` 等）  
- 相关单元测试模块 `tests/renderers/test_hf.py`  

**🔍 技术洞察**：  
- **架构影响**：  
  - 通过引入 `Tokenizer.get_chat_template` 的调用，使渲染层与 **Tokenizer 配置** 更紧密耦合。模板名称的解析职责从调用方转移到渲染器内部，统一了 “名称 → Jinja 内容” 的转换路径，提升了 API 的一致性。  
  - 只影响 Python 层的渲染逻辑，不会波及底层模型推理或分布式调度模块。  

- **性能影响**：  
  - 新增一次函数调用（`tokenizer.get_chat_template`），在正常情况下只涉及一次字典查找或文件读取，成本极低。  
  - 对批量调用的影响可以忽略；若模板名称每次都需要从磁盘加载，建议在 `Tokenizer` 实现中做缓存。  

- **安全考虑**：  
  - 解析过程仍然受限于 `Tokenizer` 的实现，模板内容来源于可信的 tokenizer 配置或用户显式提供的文件。  
  - 若攻击者能够控制 `chat_template` 参数并让 `Tokenizer` 加载任意文件，可能导致模板注入攻击。**建议**在 `Tokenizer.get_chat_template` 实现中加入白名单检查或路径限制。  

**⚠️ 潜在风险**：  
1. **向后兼容性**：旧代码若依赖 `resolve_chat_template` 直接返回传入的字符串（不做任何检查），现在会额外调用 `Tokenizer.get_chat_template`，可能在不存在该方法的自定义 tokenizer 上抛出 `AttributeError`。  
2. **错误传播**：当提供的模板名称在 tokenizer 配置中不存在时，`get_chat_template` 可能抛出异常或返回 `None`，导致后续的模板变量检测失效。  
3. **缓存缺失**：如果 `Tokenizer.get_chat_template` 每次都重新读取文件，频繁调用会增加 I/O 开销。  

**💡 关注建议**：  
- **兼容性检查**：在调用前使用 `hasattr(tokenizer, "get_chat_template")` 或在 `Tokenizer` 基类中提供默认实现返回原始输入，以避免因缺失方法导致崩溃。  
- **异常处理**：在 `resolve_chat_template` 中捕获 `KeyError`/`FileNotFoundError` 并回退到原始 `chat_template`，并记录警告日志，保证高可用性。  
- **缓存策略**：若仍未实现，建议在 `Tokenizer` 内部对已解析的模板名称进行 LRU 缓存，以降低磁盘 I/O。  
- **安全审计**：审视 `Tokenizer.get_chat_template` 的实现，确保只能加载 tokenizer 发行版自带的模板或用户经过验证的自定义模板，防止模板注入。  
- **文档更新**：在 vLLM 文档中明确说明 `resolve_chat_template` 现在会自动解析模板名称，并展示如何在自定义 tokenizer 中实现 `get_chat_template`。  

这样既修复了原有的变量检测失效问题，又确保在生产环境中保持稳定、可维护和安全。

### [Frontend][Core] Re-add shutdown timeout - allowing in-flight requests to finish (#36666)
**SHA**: `7afe0fa` | 🔗 [查看提交](https://github.com/vllm-project/vllm/commit/7afe0faab1eb2ab84cda5cab29b24046e516f7b8)

**🎯 变更类型**：功能增强 / 架构变更  
**⚡ 重要程度**：🔴 高  

**📋 变更摘要**  
1. 为 vLLM 引入 *shutdown timeout*（`--shutdown-timeout`），支持两种关机策略：  
   - `0` 秒 → 立即中止所有在途请求（abort）。  
   - `>0` 秒 → 等待最多 N 秒，允许已提交请求完成后再强制退出。  
2. 为整个服务栈（Engine、API‑Server 进程管理、CLI、Launcher、内部进程通信）实现统一的 graceful‑shutdown 接口 `shutdown(timeout: float|None)`，并在 signal 处理器中触发。  
3. 新增大量集成测试，验证在单 / 多 API‑Server、不同 timeout、SIGTERM/SIGINT 场景下的请求完成、拒绝、子进程回收等行为。  
4. 将原有的 `close()` 接口改名为 `shutdown()` 并加入可选超时参数，统一资源回收路径。  

**🎯 影响范围**  
- `vllm/config/vllm.py`（配置字段）  
- `vllm/engine/arg_utils.py`（CLI 参数解析）  
- `vllm/engine/protocol.py`（新增 `shutdown` 抽象）  
- `vllm/entrypoints/cli/serve.py`、`vllm/entrypoints/launcher.py`（信号处理、超时等待）  
- `vllm/v1/engine/*`（核心引擎进程、调度器、请求拦截、信号回调、WAKEUP 请求）  
- `vllm/v1/engine/utils.py`（进程关闭、SignalCallback）  
- `vllm/v1/engine/core_client.py`（客户端 shutdown 接口）  
- `tests/entrypoints/*`（全套 shutdown 行为测试）  
- 相关的进程管理类 (`APIServerProcessManager`、`EngineProcessManager`、`Coordinator`)  

**🔍 技术洞察**  

- **架构影响**  
  - **统一 shutdown 抽象**：在 `EngineProtocol` 中加入 `shutdown(timeout)`，所有实现（本地进程、远程进程、multiprocessing、Ray）必须实现该方法，提升了资源回收的一致性。  
  - **信号驱动闭环**：通过 `SignalCallback` 将 signal‑handler 中的“唤醒”操作转到安全线程，避免在非可重入上下文中直接操作 `queue`/`mutex`，降低了死锁风险。  
  - **状态机**：新增 `EngineShutdownState`（RUNNING → REQUESTED → SHUTTING_DOWN），在 `EngineCoreProc.run_busy_loop` 中检查并按状态执行 **abort** 或 **drain** 逻辑，实现了 *in‑flight request* 的可控退出。  
  - **子进程清理**：`_assert_children_cleaned_up` 与统一的 `shutdown(procs, timeout)` 能够确保 forked 子进程（GPU worker、API server）在主进程退出后被彻底回收，避免僵尸进程。  
  - **向后兼容**：所有原有 `close()` 调用已改为 `shutdown()`，但未保留别名，可能导致使用旧 API 的第三方代码在运行时抛 `AttributeError`。  

- **性能影响**  
  - **正常请求路径**：新增的状态检查 (`self.shutdown_state`) 与 `has_work` 判定在每次循环中执行，开销极小（常数时间的整数比较）。  
  - **关机期间**：在 `REQUESTED` 状态下，若 timeout>0，调度器仍会继续调度已提交的请求直至超时或完成，导致短暂的 CPU/内存占用提升，但这正是预期的“等待完成”。  
  - **SignalCallback**：在信号触发时通过轻量级 `threading.Event` 唤醒专用线程，几乎零额外延迟。  

- **安全考虑**  
  - **信号处理安全**：原先的直接 `signal.signal(..., handler)` 中对队列进行 `put_nowait` 的做法可能在 CPython 的内部锁持有时导致死锁；现在改为 `SignalCallback` 把实际写队列的工作交给独立线程，符合 POSIX 信号安全要求。  
  - **资源泄漏**：统一的 `shutdown(procs, timeout)` 强制在超时后 `terminate`，降低因异常退出导致的资源残留风险。  
  - **非授权外部请求**：在 shutdown 阶段，新增对新请求的直接拒绝（返回 503/500/connection error），防止恶意客户端在服务器即将退出时占用资源。  

**⚠️ 潜在风险**  

1. **向后兼容性**：外部项目仍调用 `EngineProcessManager.close()`、`Coordinator.close()`、`APIServerProcessManager.close()` 会抛异常。需在迁移指南中说明或保留兼容包装。  
2. **超时配置误用**：`shutdown_timeout` 默认值 0（立即 abort），若用户未显式设置，业务可能在 SIGTERM 时出现请求被意外中止，导致不符合业务期望的错误率上升。  
3. **死锁风险**：虽然 `SignalCallback` 已经很安全，但在 Windows 平台（没有 `signal.SIGTERM`）或在多线程环境下仍可能出现信号未正确转发的情况，需要在 CI 中覆盖。  
4. **进程间通信延迟**：在 `REQUESTED` → `SHUTTING_DOWN` 期间，若子进程未及时收到 `WAKEUP` 消息，可能导致 `has_work()` 永远返回 `False` 而提前退出，导致未完成的请求被误 abort。  
5. **测试依赖**：新增的测试使用 `psutil`、`httpx` 等第三方库，若在某些 CI 环境缺失这些依赖，测试会失败。  

**💡 关注建议**  

1. **兼容层**：在 `EngineProcessManager`、`APIServerProcessManager`、`Coordinator` 中提供一个薄包装的 `close()` 方法，内部调用 `shutdown()`，以免破坏现有生态。  
2. **文档与默认值**：在官方文档明确说明 `--shutdown-timeout` 的默认行为（abort）以及何时应当显式设置正值，例如在生产环境的滚动升级或容器编排中。  
3. **监控与报警**：在 `EngineCoreProc._handle_shutdown` 中记录 "shutdown initiated", "abort X requests", "drain completed" 等日志，并提供对应的 Prometheus 指标，以便运维观察 shutdown 阶段的健康状态。  
4. **跨平台测试**：补充 Windows（或不支持 SIGTERM）下的行为测试，确保 `signal.signal` 替代方案（如 `event.set()`）能够正常触发 shutdown。  
5. **超时容忍**：考虑在 `shutdown(procs, timeout)` 中加入一个更大 “hard‑kill” 阶段（如 `kill()`），防止进程在极端情况下卡死不退出。  
6. **回归测试**：在 CI 中保留现有的 `test_shutdown*` 套件，并在每次引入新请求类型（utility、streaming）时验证其在 shutdown 期间的正确拒绝/abort 行为。

### Fp8 lora dense kernel (#35242)

**🎯 变更类型**：功能增强 / 性能优化 / 架构变更  
**⚡ 重要程度**：🔴 高  
**📋 变更摘要**  
- 为 LoRA（Low‑Rank Adaptation）在 vLLM 中新增 **FP8 量化** 支持：实现了 `lora_shrink_fp8` 与 `lora_expand_fp8` 两套 Triton kernel。  
- 新增 **FP8 量化/反量化工具**（`fp8_kernel_utils.py`）以及 **针对不同量化方式**（per‑tensor、per‑channel、block‑wise）的测试套件。  
- 在 `triton_ops/__init__.py` 中对外暴露新算子，并将默认 block‑K 从 16 调整为 32，以更好匹配 FP8 计算块大小。

## 🔍 技术洞察

| 维度 | 影响描述 |
|------|----------|
| **架构影响** | • 在 LoRA 的 **shrink / expand** 流程中加入了两条全新执行路径（FP8‑w8a8 与纯 FP8），同时保持原有 BF16/FP16 路径不变。<br>• 新增 `fp8_kernel_utils.py` 作为 **量化/反量化** 与 **block‑wise scaling** 的公共层，复用了 Triton 的 `tl` 与 `triton` 编译器特性。<br>• 通过全局指针字典 (`_EXPAND_LORA_SCALE_PTR_DICT`, `_SHRINK_LORA_SCALE_PTR_DICT`) 实现 **多切片**（slice）场景下的指针间接访问，兼容已有的 LoRA 调度逻辑。 |
| **性能影响** | • FP8 数据宽度 **4 bit**（e4m3 / e5m2）相比 BF16/FP16 减少 **50%–75%** 带宽需求，显著提升 **GPU 内存占用** 与 **缓存命中率**。<br>• 通过 **block‑wise 量化**（可选 `group_k / group_n`），在 K 维和 N 维上实现局部尺度恢复，降低因统一尺度导致的数值溢出/下溢。<br>• Triton kernel 采用 **SPLIT‑K**、**GROUP_SIZE_M**、**GDC**（GPU‑direct‑cache）等调度技巧，理论上在 **BATCH ≥ 4**、**hidden ≥ 1024** 时可获得 **1.3‑2.0×** 的算子吞吐提升（实际提升受硬件 FP8 支持、PCIe/NVLink 带宽限制）。<br>• 对 `utils.get_lora_op_configs` 中 `block_k` 调整为 32，使得 **K‑axis** 更好匹配 FP8 的 8‑bit 乘法对齐，减少循环迭代次数。 |
| **数值/精度影响** | • 通过 **“量化 → 反量化 → BF16 基准”** 的对比方式，测试将 FP8 误差与原始 BF16 误差分离，容忍度设为 `rtol = 0.1, atol = 0.25`（shrink）/`0.15`（expand），表明在 **LoRA 乘加** 场景下误差仍在可接受范围。<br>• Block‑wise量化可以进一步降低误差峰值（局部尺度更贴近真实幅度），但会引入 **scale‑lookup** 开销。 |
| **安全/可靠性** | • 纯数值计算路径，没有外部 I/O，未引入新攻击面。<br>• 依赖 **CUDA FP8 硬件指令**（Tensor‑Core 支持），在不具备该硬件的 GPU 上会触发 **`torch.cuda.UnsupportedError`**。因此在生产部署前必须做好 **硬件兼容检测** 与 **fallback**。<br>• 全局指针字典使用 `Lock` 防止并发写冲突，已基本安全；但在长时间运行的服务中若出现 **指针泄露**（如忘记 `clear`），会导致显存占用增长。 |
| **可维护性** | • 新增 3 k 行 Triton kernel 代码，逻辑相对复杂（多分支的尺度加载、块级循环、GDC、PDL）。<br>• 代码与现有 `lora_shrink` / `lora_expand` 保持相同 API，便于迁移。<br>• 单元测试覆盖 **所有量化模式**、**不同 slice 数**、**多 batch/loras**，为后期改动提供回归保障。 |
| **兼容性** | • `torch.ops.vllm.lora_shrink_fp8` 与 `..._expand_fp8` 在 `triton_ops.__init__` 中公开，未破坏旧的 `lora_shrink` / `lora_expand` 调用方式。<br>• 仅在 **CUDA ≥ 12.1**、**torch >= 2.3**（支持 `torch.float8_*`）的环境下可用；如果项目的 CI 环境或用户机器不满足，导入会抛错。建议在 `setup.py`/`pyproject.toml` 中声明 **optional** 依赖。 |

## ⚠️ 潜在风险

| 风险点 | 可能后果 | 缓解建议 |
|--------|----------|----------|
| **硬件不支持 FP8** | 程序在启动时崩溃，或在运行时出现 `illegal memory access`。 | 在 `torch.cuda.is_available()` 与 `torch.cuda.get_device_capability()` 检查 `fp8` 能力（如 `torch.cuda.get_device_capability() >= (9, 0)`），没有时回退到 BF16 路径。 |
| **尺度 overflow/underflow** | 块尺度不恰当时会导致 **数值失真**，影响 LoRA 效果。 | 在量化函数里加入 **clamp** 与 **最小 scale**（已实现 `1e‑12`），并在测试中加入 **极端取值**（大/小激活）用例。 |
| **显存泄漏**（指针字典未清理） | 长时间服务后显存占用逐渐上升，可能 OOM。 | 在每次 `profile_run` 结束后显式 `clear()`，或使用 **weakref** 自动回收。 |
| **Triton 版本差异** | 不同 Triton 发行版的 **GDC/PDL** 行为可能变化，导致性能回退或错误。 | 在 `requirements` 中固定 Triton 版本（如 `triton==3.1.0`），并在 CI 中加入 **kernel‑compilation test**。 |
| **多线程竞争** | 多请求并发时 `Lock` 仍可能出现 **死锁**（尤其在异常路径）。 | 将锁粒度缩小，仅在字典创建阶段加锁；在异常捕获块中 `finally` 释放锁。 |
| **数值回归** | FP8 引入的近似误差在特定模型（如 7B‑70B）上可能对生成质量产生可感知影响。 | 在实际推理基准上加入 **BLEU/Perplexity** 对比，与 BF16 baseline 做回归验证。 |
| **API 混淆** | 两套 `lora_shrink`/`lora_shrink_fp8` 同名但功能不同，用户可能误用。 | 在文档与函数 docstring 中明确标注 “FP8 version – only for GPUs with native FP8 support”。 |

总体来看，此次改动提升了 vLLM 对最新 Transformers 版本的兼容性，统一了权重映射逻辑并简化了量化配置路径，风险集中在自定义映射与旧版 Transformers 的兼容性，按上述建议检查即可平滑迁移。

**SHA**: `6341d43` | 🔗 [查看提交](https://github.com/vllm-project/vllm/commit/6341d43043517a431d49b9c571fc5fa8afe182cb)

**🎯 变更类型**：功能增强（新增 W4A8 MXFP4+FP8 量化方案）  
**⚡ 重要程度**：🟡 中  

**📋 变更摘要**  
1. 在 `vllm._aiter_ops` 中实现并注册了 `rocm_aiter_gemm_a8wfp4` 的真实与占位实现，用于 ROCm 上的 AITER Triton GEMM。  
2. 为 Quark 量化体系新增 `QuarkW4A8_MXFP4_FP8` 类，支持权重量化为 MXFP4（每 32 元素 E8M0 缩放）且激活为 static‑FP8；在支持的 ROCm GPU（gfx950 以上）上走 AITER 内核，否则回退到 Python 解码实现。

**🎯 影响范围**  
- `vllm._aiter_ops`、`vllm.model_executor.layers.quantization.quark.*`（新增 scheme、注册 op）  
- 依赖 AITER 的线性层（LinearLayer），特别是使用 ROCm 的部署环境。  

**💡 关注建议**  
1. **平台兼容性**：当前实现仅在 ROCm 且 `on_gfx950()` 为真时使用内核；在其他平台会走 emulation。建议在文档或检查点明确提示 “仅在 ROCm GPU 上获得加速”。  
2. **动态 FP8 支持**：目前强制要求 static FP8 scales，抛出 `NotImplementedError`。若用户误配 `is_dynamic=True`，会在模型加载时报错，建议在配置检查阶段给出更友好的错误信息。  
3. **尺度广播**：`x_scales = input_scale.expand(M, 1)` 假设 `input_scale` 为标量张量；若 `input_scale` 为多元素张量（例如 QKV 融合后），会产生广播错误。可以在 `_apply_aiter_kernel` 前做 `input_scale = input_scale.squeeze()` 或显式检查维度。  
4. **权重/尺度持久化**：`create_weights` 中对 `input_scale` 预填充了 `torch.finfo(torch.float32).min`，防止 NaN，但在实际加载后未显式恢复真实值，务必确认 checkpoint 加载逻辑能覆盖该初始化。  
5. **测试覆盖**：新增单元测试，验证：  
   - 在支持的 ROCm 环境下 `use_aiter_kernel=True`，输出与 emulation 实现一致。  
   - 非 ROCm（如 CUDA）路径不会误注册或调用该 op。  
   - 错误配置（动态 FP8、错误 group_size）抛出预期异常。  
6. **日志与警告**：已有 `logger.warning_once`，建议在 fallback 时记录使用的 fallback 类型（emulation），方便用户排查性能差异。  

总体而言，此次 PR 为 ROCm 平台提供了高效的 W4A8 量化路径，代码结构清晰。只要做好平台检测、异常提示及测试验证，即可安全合入主线。
