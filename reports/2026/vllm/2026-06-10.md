### [MRV2][Spec Decode] DFlash (#44586)
**SHA**: `0bae1d3` | 🔗 [查看提交](https://github.com/vllm-project/vllm/commit/0bae1d38480374365ad77bbea50be225237572ea)

**🎯 变更类型**：功能增强 / 架构变更  
**⚡ 重要程度**：🔴高  

**📋 变更摘要**  
- 为 vLLM 引入全新的 *DFlash*（Block Diffusion）并行草稿（parallel drafting）推理路径，实现了在 V2 Model Runner 下的 Speculative Decoding。  
- 新增 `dflash` 子包：Speculator、CUDA‑Graph 管理器、准备 kernels 与工具函数，并在配置、初始化、采样等层面完成与已有 Eagle/Eagle3/MT‑P 路径的统一。  
- 测试用例通过 `VLLM_USE_V2_MODEL_RUNNER` 环境变量在 V1 与 V2 两条路径上进行参数化，确保兼容性。  

**🎯 影响范围**  
- `vllm/config/vllm.py`（模型运行器特性检测）  
- `vllm/v1/worker/gpu/model_runner.py`（Speculative 方法分支）  
- `vllm/v1/worker/gpu/attn_utils.py`（`causal` 参数向下传递）  
- `vllm/v1/worker/gpu/spec_decode/*`（新增 DFlash 实现，改动基类、gumbel 采样等）  
- `tests/v1/e2e/spec_decode/test_spec_decode.py`（MRV2 参数化）

### 🔍 技术洞察

| 维度 | 影响描述 |
|------|----------|
| **架构影响** | • 新增 `vllm.v1.worker.gpu.spec_decode.dflash` 包，实现 **DFlashSpeculator**，通过 `init_speculator` 动态返回对应实现。<br>• 在 V2 Model Runner 中，`_get_v2_model_runner_unsupported_features` 将 `"dflash"` 纳入白名单，允许并行草稿，仅在 `parallel_drafting` 为 `True` 时排除 EAGLE 场景。<br>• `DFlashCudaGraphManager` 负责捕获 **全图**（FULL）CUDA Graph，内部自行构造注意力元数据，避免依赖外层 `attn_metadata`。<br>• `load_dflash_model` 在运行时复用目标模型的 `embed_tokens` 与 `lm_head`（若符合共享条件），保持参数共享，降低显存占用。 |
| **性能影响** | • **并行草稿**：一次前向同时生成 *N*（`num_speculative_steps`）个草稿 token，理论上可把采样/注意力开销降低到原来的 `1/N`。<br>• **CUDA‑Graph 捕获**：将 `generate_draft` 前向全流程捕获为一次图执行，显著降低 GPU‑CPU 同步与 kernel launch 开销。<br>• `gumbel_sample` 添加 `PER_TOKEN_COL` 标志，仅在有 per‑token logits 列的情况下读取对应位置，避免不必要的缓存访问。<br>• 通过 `prepare_dflash_inputs` kernel 直接在 GPU 上并行填充 `input_ids`、`positions`、slot‑mapping 等，进一步提升批处理规模。 |

### ⚠️ 潜在风险

1. **CUDA‑Graph 稳定性**  
   - 捕获的图假设输入张量尺寸不变（通过对齐到 `max_num_reqs` / `max_num_tokens`），若运行时出现超出上限的请求会导致 OOB 或图重捕获。  
   - 需要确保 `max_num_reqs`、`max_num_tokens` 在部署时足够大，并在异常路径（如 `dummy_run`）做好安全防护。

2. **并行草稿 token ID 依赖**  
   - `parallel_drafting_token_id` 由模型配置提供，错误的 ID 会导致生成无意义 token，进而影响接受率与准确率。  
   - 建议在模型加载阶段验证该 ID 是否在词表范围内。

3. **与 Pipeline Parallel (PP) 的兼容**  
   - `model_runner` 中仍然在 `pipeline parallel` 场景下直接抛 `ValueError`，因此在使用 PP 时必须禁用 DFlash。文档需明确该限制。

4. **内存峰值**  
   - DFlash 需要额外的 `context_positions`、`sample_*` 缓冲区以及 **两套** KV‑Cache（目标 + 草稿），显存需求约增加 1–2 倍。对大模型（如 70B）可能触发 OOM。

5. **注意力后端匹配**  
   - `load_dflash_model` 会根据 `dflash_config.causal` 自动切换 `use_non_causal`，但若后端不支持非因果注意力（如 Flash‑Attention 只实现因果），运行时会出错。

### KV-Cache multi-tier offloading async batched lookup (#44193)
**SHA**: `af65e08` | 🔗 [查看提交](https://github.com/vllm-project/vllm/commit/af65e08fc5e42a909e66be34ff9f35e5307fc6ab)

**🎯 变更类型**：功能增强 / 架构变更 / 性能优化  
**⚡ 重要程度**：🔴 高  
**📋 变更摘要**：  
1. 为 KV‑offload 二级层实现 **AsyncLookupManager**，在后台线程中批量执行 `batch_lookup`，实现异步、批量的键存在性检查。  
2. 在 **FileSystemTierManager** 与 **ObjectStoreSecondaryTierManager** 中引入该管理器，替换原同步 `lookup` 实现，新增 `on_schedule_end`、`on_request_finished` 与 `cleanup` 接口。  
3. 新增完整的单元测试覆盖 **AsyncLookupManager** 与两大 tier 的异步查找行为，提升代码可靠性。  

**🎯 影响范围**：  
- `vllm/v1/kv_offload/tiering/async_lookup.py`（全新基类）  
- `vllm/v1/kv_offload/tiering/fs/manager.py`（文件系统 tier）  
- `vllm/v1/kv_offload/tiering/obj/manager.py`（对象存储 tier）  
- 相关测试目录 `tests/v1/kv_offload/tiering/*`  

**🔍 技术洞察**  

- **架构影响**  
  - **职责分离**：lookup 的调度、批处理、结果合并统一交由 `AsyncLookupManager`，二级层只需实现 `batch_lookup`，大幅简化 tier 实现并提升可复用性。  
  - **线程模型**：新增后台 *worker* 线程负责实际磁盘或对象存储查询，调度线程只做状态记录与结果消费，避免在模型推理路径上出现阻塞。  
  - **状态管理**：`LookupState` 持有 `result` 与 `request_ids`，并通过 `_req_keys` 进行逆向索引，`cleanup` 能在请求结束时安全移除不再需要的条目。  
  - **接口扩展**：tier 现在必须在每个调度周期调用 `on_schedule_end()`（flush）以及在请求结束时调用 `on_request_finished()`（cleanup），这对已有调用方是向后兼容的（已有实现默认空实现）。  

- **性能影响**  
  - **批量 I/O**：文件系统层使用 `os.path.exists` 对整批路径一次性检查，对象存储层一次性调用 `query_memory`，显著降低每个键的系统调用/网络往返次数。  
  - **并行度提升**：后台线程可以在模型推理的空闲时间运行，提升整体吞吐。实验性测评预计在大规模 KV‑offload 场景（数万块）下的查找延迟从 **O(N)** 降至 **O(1) per‑step**（批次大小为调度步长）。  
  - **额外开销**：每个 tier 多创建一个长期运行的线程以及若干 `queue.Queue`/`SimpleQueue`，在极端请求并发（成千上万）时需关注线程上下文切换与队列内存占用。  

- **安全考虑**  
  - **线程安全**：依赖 Python 标准库 `queue.Queue`/`SimpleQueue` 实现的内部锁，无显式锁，设计上避免死锁风险。唯一需要关注的是 `shutdown()` 的调用顺序——必须在 tier 的 `shutdown()` 之前执行，以保证后台线程已安全退出。  
  - **错误传播**：`batch_lookup` 中若抛异常会被捕获并记录为全批次 “未找到”，防止异常泄漏到调度线程；但这也可能掩盖真实错误，建议在日志级别或监控中报警。  
  - **资源泄漏**：`cleanup` 只在 request 完成时调用，若上层忘记调用 `on_request_finished`，对应的 `LookupState` 会永久保留，可能导致内存增长。  

**⚠️ 潜在风险**  

| 风险点 | 说明 | 严重度 |
|--------|------|--------|
| **线程泄漏 / 未关闭** | 若用户在自定义 tier 中忘记在 `shutdown()` 中调用 `_lookup_manager.shutdown()`，后台线程将永远存活。 | 中 |
| **请求清理不完整** | 上层未正确触发 `on_request_finished`，导致 `_lookup_state` 永久占用内存。 | 中 |
| **结果竞争** | `lookup()` 在同一步骤内多次调用同一键会产生重复条目，但实现已通过 `len(_lookup_batch)` 检查避免。若在极端并发中 `_lookup_batch` 被其他线程误写（理论上不可能）会导致错误。 | 低 |
| **异常隐藏** | `batch_lookup` 异常被捕获后全部视为 “未找到”，可能隐藏磁盘/网络故障，影响业务判断。 | 中 |
| **兼容性** | 新增 `req_context` 为必需参数，旧调用方如果仍使用 `lookup(key)`（无 `req_context`）会在运行时抛 TypeError。 | 低‑中（需要检查所有内部调用） |
| **性能回退** | 对于极小批次（如每步仅 1 条 key），额外的线程切换与队列开销可能略高于直接同步 `os.path.exists`。 | 低 |

**💡 关注建议**  

1. **文档与示例**  
   - 在官方文档中补充 **AsyncLookupManager** 的使用约束：必须在每个调度周期调用 `on_schedule_end`、请求结束时调用 `on_request_finished`，并在进程退出前调用 `shutdown`。  
   - 给出最小化 **lookup** 调用次数的最佳实践（如在同一个请求中复用同一 `OffloadKey`），避免不必要的批次增长。  

2. **监控 & 报警**  
   - 为 `batch_lookup` 中捕获的异常添加统计计数或监控指标（如 `kv_offload.lookup.batch_error`），便于在生产环境快速发现磁盘或网络故障。  
   - 暴露后台线程的队列深度（`_lookup_queue.qsize()`、`_pending_results.qsize()`）做为性能健康检查。  

3. **资源回收**  
   - 在 `AsyncLookupManager.shutdown()` 前加入 `self._lookup_queue.join()`（或手动清空队列），确保所有挂起的 batch 都已处理完毕，避免潜在的数据丢失。  
   - 考虑在 `lookup()` 中加入 TTL/过期策略，防止长时间未被清理的 `LookupState` 占用内存。  

4. **容错策略**  
   - 将 “异常全视为未找到” 的默认行为改为 **可配置**（例如 `self._treat_error_as_miss = True`），让上层根据业务需求决定是否返回 `None` 或抛异常。  
   - 在对象存储实现中，若 `query_memory` 返回异常或超时，适当重试或回退到同步查询，以提升可靠性。  

5. **性能基准**  
   - 添加微基准测试（benchmark）对比同步 vs 异步 lookup 在不同批次大小、并发请求数下的延迟与吞吐，帮助调优 `max_workers`、`max_queue_size`（若后续引入）等参数。  

6. **兼容性回滚**  
   - 若已有插件或自定义 tier 直接依赖旧同步 `lookup` 实现，可在 `BaseTier` 中保留向后兼容的包装方法：`def lookup_sync(...): return self._lookup_manager.lookup(... )`，并在文档中标记为 deprecated。  

---  

**结论**：本次引入的 **AsyncLookupManager** 为 KV‑offload 二级层提供了干净的异步批量查找能力，能够显著降低磁盘/网络查询开销，提升高并发推理场景的整体吞吐。但引入新的后台线程和状态清理机制也带来一定的资源管理与异常处理复杂度。建议在正式发布前完成上述监控、文档与回滚措施，以确保在生产环境中能够安全、平稳运行。

2. `vllm/multimodal/media/audio.py` 为 `load_audio_pyav` 与 `load_audio_soundfile` 增加 `max_duration_s` 参数，实现基于元数据和实时采样计数的时长上限检查，超限直接抛 `ValueError`，从而在解码前阻止内存爆炸。  
3. 相应单元测试加入 “合法时长通过” 与 “超限被拒” 两个用例。  

**🎯 影响范围**  
- `vllm/envs.py`（新增配置项）  
- `vllm/entrypoints/speech_to_text/base/serving.py`（使用新配置）  
- `vllm/multimodal/media/audio.py`（核心音频加载逻辑）  
- 相关测试 `tests/multimodal/media/test_audio.py`  

**💡 关注建议**  
- 生产环境可通过 `VLLM_MAX_AUDIO_DECODE_DURATION_S` 调整上限，防止合法长音频被误拦截。  
- 部署时确保已安装 `vllm[audio]`（PyAV）以使用新检查；若仅依赖 `soundfile`，仍能获益。  
- 旧版本客户端若自行调用 `load_audio`，需同步更新签名（加入 `max_duration_s` 参数）或保持兼容包装。  
- 监控日志中 “Audio exceeds maximum allowed duration” 信息，以快速定位潜在 DoS 攻击。
