### feat: add TxtSlicesDataset to allow sampling slices from txt file for benchmarking (#30156)
**SHA**: `893b2af` | 🔗 [查看提交](https://github.com/vllm-project/vllm/commit/893b2affff548bf0f54978ca2ba4545c5b4d5fbe)

**🎯 变更类型**：功能增强 / 重构 / 架构变更  
**⚡ 重要程度**：🔴高  

**📋 变更摘要**  
1. 引入 **TxtSlicesDataset**：通过 `create_txt_slices_dataset.py` 将任意文本文件（本地或 URL）切分为随机 token‑slice，生成符合 `CustomDataset` 规范的 JSONL 数据，用于基准测试。  
2. 将原来散落在 `vllm.benchmarks.datasets.datasets` 中的采样逻辑抽取到 `vllm/benchmarks/datasets/utils.py`，统一为 `get_sampling_params` 并支持 **RangeRatio**（单 float 或 `{"input":…, "output":…}`）。  
3. 迁移多处代码使用新的 `get_sampling_params`，改用 `dataclasses.replace` 替代 `deepcopy`，并把 `SampleRequest.expected_output_len` 改为 `int | None`（兼容动态长度）。  
4. 新增单元测试 `tests/benchmarks/*`，覆盖采样参数函数、JSONL 生成以及 `CustomDataset` 加载。  

**🎯 影响范围**  
- `vllm/benchmarks/datasets/__init__.py`、`datasets.py`（所有数据集实现）  
- 新增模块 `benchmarks/datasets/utils.py`、`benchmarks/datasets/create_txt_slices_dataset.py`  
- 测试目录 `tests/benchmarks/`  
- 依赖 `TokenizerLike`、`numpy`、`transformers` 以及标准库 `urllib.request`  

**🔍 技术洞察**  

| 维度 | 影响描述 |
|------|----------|
| **架构影响** | • 采样逻辑抽象为公共工具 `get_sampling_params`，消除了各数据集中重复实现，提升代码可维护性和一致性。<br>• `TxtSlicesDataset` 通过 **CLI** 入口提供了新的数据生成方式，使得 benchmark 能直接使用任意文本资源，属于 **扩展点**，不影响原有 `CustomDataset` 接口。<br>• `SampleRequest` 字段类型放宽，允许 `expected_output_len=None`（动态长度），对上层调度器兼容性略有影响，需要检查 downstream 是否依赖强制 int。 |
| **性能影响** | • `get_sampling_params` 仍使用 NumPy 的向量化 `integers`，时间复杂度 O(num_requests) 与原实现相同。<br>• `create_txt_slices_jsonl` 会一次性 **完整 token 化** 输入文本，可能在超大文件（≥ GB）时导致显存/内存峰值升高。对基准生成阶段影响有限，但建议在生产流水线中控制文件大小或采用分块 token 化。<br>• 使用 `random.Random` 处理起始位置避免了 NumPy 的 `integers` 在 Python 循环里的额外开销，整体开销几乎可以忽略不计。 |
| **安全考虑** | • 新增 `load_text` 支持 **http/https** 下载，未对 URL 做白名单或大小限制，可能被恶意 URL 用来引入大文件或恶意内容。<br>• `json.loads` 直接解析 `--range-ratio`，若外部传入的字符串包含恶意 JSON（如注入代码的 `__proto__`），在 Python 中不会执行，但仍建议对输入进行 schema 校验。<br>• 其余路径（本地文件读取、tokenizer 调用）保持原有安全模型，未新增特权操作。 |
| **可维护性** | • 统一的 `utils` 模块让后续修改采样策略只需改动单点，降低维护成本。<br>• 将 `deepcopy` 替换为 `replace` 可避免不必要的深拷贝，提高可读性并减少潜在的 copy‑side‑effects。<br>• 新增测试覆盖了正向、边界、异常路径，显著提升回归安全性。 |

**⚠️ 潜在风险**  

1. **内存峰值**：对极大文本文件执行 `tokenizer(text).input_ids` 会一次性加载整个 token 列表，可能触发 OOM。  
2. **CLI 兼容性**：`--random-range-ratio` 参数类型改为 `str` 并在内部解析为 float 或 dict，旧脚本若直接以 `float` 传递仍可工作，但自定义解析器（未更新的第三方包装）可能因类型冲突报错。  
3. **`SampleRequest.expected_output_len` 可为空**：部分代码路径（如自定义调度器或监控统计）可能假设该字段为 `int`，导致 `TypeError` 或 `None` 传递给模型推理阶段。  
4. **远程 URL 下载**：未对 URL 大小、证书验证或重定向做限制，可能被用于 DoS 或加载恶意内容。  
5. **RangeRatio 错误输入**：若用户提供非法 JSON（缺少键或非数值），会抛出 `ValueError`，但错误信息可能不够友好；在 CI 中可能导致测试失败。  

**💡 关注建议**  

| 建议 | 说明 |
|-----|------|
| **限制文本大小** | 在 `load_text` / `create_txt_slices_jsonl` 中加入大小阈值（例如 500 MiB）或分块 token 化，以防 OOM。 |
| **增强 URL 校验** | 只允许 `https`，并可选加入 `--allow-insecure` 标记；对响应的 `Content-Type` 进行基本检查，防止下载二进制文件。 |
| **保持向后兼容** | 在 `add_random_dataset_base_args` 中保留 `type=float` 的别名或接受 `float` 直接，不强制转换为 `str`；提供 deprecation 警告。 |
| **明确 `expected_output_len` 使用** | 在所有调用 `SampleRequest` 的地方检查 `is None` 并使用默认长度或抛出明确异常，防止隐式 `None` 传递到模型层。 |
| **改进错误信息** | 在 `_parse_range_ratio` 捕获 `JSONDecodeError` 并提示用户正确的格式（float 或 `{"input":…, "output":…}`）。 |
| **文档同步** | 更新 README / benchmark 文档，说明新工具 `create_txt_slices_dataset.py` 用法、参数意义以及 `--range-ratio` 的两种写法。 |
| **性能基准** | 新增基准报告（例如 1 GB 文本切片的内存/时间消耗），帮助用户评估生成过程的资源需求。 |
| **测试扩展** | 增加对极大文件（使用 mock）和异常 URL（网络错误）场景的单元测试，提升鲁棒性。 |

总体来看，此次提交为 **vLLM** 的基准数据准备提供了强大的、可配置的文本切片功能，同时通过抽取公共采样实现提升了代码质量。只要在生产环境对文件大小、远程下载以及 `SampleRequest` 可空字段做好防护，即可安全上线。

### [MoE Refactor] Remove MoE DP chunking (#39107)
**SHA**: `e1e318a` | 🔗 [查看提交](https://github.com/vllm-project/vllm/commit/e1e318af010b4f92d39324d03231cfd409766bf9)

**🎯 变更类型**：功能增强（MoE DP Chunking 重构）  
**⚡ 重要程度**：🟡 中  

**📋 变更摘要**  
1. 完全移除原先通过环境变量 `VLLM_MOE_DP_CHUNK_SIZE / VLLM_ENABLE_MOE_DP_CHUNK` 实现的 MoE DP Chunking 机制。  
2. 引入统一的 `max_num_tokens` 参数，由 `SchedulerConfig.DEFAULT_MAX_NUM_BATCHED_TOKENS_FOR_BATCHED_DP`（默认 256）统一控制；`FusedMoEConfig.max_num_tokens`、`FusedMoELayer`、测试用例等均改用该值。  
3. 在 `ParallelConfig` 中新增 `use_batched_dp_moe` 判断，用于在 DeepSpeed‑LL、NIXL‑EP 等后端启用 *batched* DP MoE。  
4. 删除 `chunking_moe_runner` 及其所有调度逻辑，`moe_runner_factory` 直接返回 `DefaultMoERunner`。  
5. 迁移 `next_power_of_2` 到公共工具 `vllm.utils.math_utils`，并在多处测试和配置调用以保证 token 数对齐。  
6. 测试层面去掉显式设置环境变量 `VLLM_MOE_DP_CHUNK_SIZE=128`，改为通过 `SchedulerConfig.max_num_batched_tokens` 自动计算。  

**🎯 影响范围**  
- **配置系统**：`vllm/config/parallel.py`、`vllm/config/scheduler.py`、`vllm/engine/arg_utils.py`、`vllm/model_executor/layers/fused_moe/config.py`、`vllm/model_executor/layers/fused_moe/layer.py`。  
- **运行时**：MoE DP Chunking 相关的 `forward_context.DPMetadata`、`chunked_sizes`、`ChunkingMoERunner` 均被删除，代码路径简化。  
- **测试**：所有涉及 `VLLM_MOE_DP_CHUNK_SIZE` 的单元测试改用 `next_power_of_2`、`SchedulerConfig`，并删除环境变量依赖。  
- **用户 API**：不再需要在 ``.env`` 或启动脚本中手动设置 `VLLM_MOE_DP_CHUNK_SIZE`，改为通过 `SchedulerConfig.max_num_batched_tokens`（或 `SchedulerConfig.max_num_batched_tokens_for_batched_dp`）进行调优。  

**💡 关注建议**  

1. **代码迁移**：如果项目或自定义脚本中仍显式读取 `VLLM_MOE_DP_CHUNK_SIZE` 或 `VLLM_ENABLE_MOE_DP_CHUNK`，需要改为读取 `vllm_config.scheduler_config.max_num_batched_tokens`（或 `parallel_config.use_batched_dp_moe`）来决定是否开启批量 DP MoE。  

2. **调参指导**：  
   - 对于使用 DeepSpeed‑LL、NIXL‑EP 后端且 Data‑Parallel > 1 的场景，`max_num_batched_tokens` 默认 256 可能较小；依据显存与吞吐需求，可在 `VllmConfig` 中显式覆盖 `scheduler_config.max_num_batched_tokens`。  
   - `next_power_of_2` 已统一为公共工具，建议在自定义的 MoE‑相关代码中使用，以避免冲突的本地实现。  

3. **兼容性验证**：  
   - 运行完整的 `pytest -m moe` 测例，确保所有 `max_num_tokens` 参数在新默认值下仍满足模型尺寸（尤其是 `k >= 2048` 的高维情形）。  
   - 对比旧版与新版的性能曲线，确认批量 DP MoE 的吞吐提升或退化，必要时通过 `SchedulerConfig.DEFAULT_MAX_NUM_BATCHED_TOKENS_FOR_BATCHED_DP` 调整。  

4. **文档更新**：

3. **性能与一致性**：`psutil.virtual_memory()` 每次调用都会遍历系统状态，若在高频监控场景下可能带来微小开销。建议在 `MemorySnapshot` 中缓存 `psutil` 结果或在上层做一次性获取。  
4. **测试覆盖**：新增的单元测试使用 `unittest.mock` 替换底层平台对象，确保逻辑分支完整。但仍缺少在真实 UMA 机器上的集成测试，建议在 CI 中加入对应硬件标签或使用容器模拟。  
5. **文档更新**：README/开发手册中应同步说明 UMA GPU 的内存报告差异及本次修复的原理，防止使用者误以为所有 GPU 均使用 `cudaMemGetInfo`。  

总体而言，此次修改解决了 UMA GPU 上内存报告偏低的问题，对项目的调度与负载均衡功能提升明显；只要在其他平台补齐实现并注意依赖与性能，即可安全合并。

5. **性能影响**：此改动主要是抽象层面的重命名与路径统一，对运行时性能几乎无影响。但因为权重加载时会走统一的 `_is_mxfp4` 判断，建议在热点路径上进行一次基准，以确保没有因额外字符串匹配导致微秒级开销。

总体来看，此次提交为 GPT‑OSS MXFP4 量化提供了专门的实现路径，提升了代码可维护性并避免了后续对 “mxfp4” 统一处理的歧义。但在全局调用链和文档层面仍需同步更新，以确保用户和内部模块都能正确识别新名字。
