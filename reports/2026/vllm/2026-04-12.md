### fix(moe): fix RoutedExpertsCapturer assertion failure with DP>1 and MK path (#37879)
**SHA**: `21fab0a` | 🔗 [查看提交](https://github.com/vllm-project/vllm/commit/21fab0a3db091173444112be8caf6b8248ed5043)

**🎯 变更类型**：Bug修复  
**⚡ 重要程度**：🔴高  
**📋 变更摘要**：修复 `RoutedExpertsCapturer` 在多数据并行（DP > 1）且使用 MK（Modular‑Kernel）路径时出现的断言错误。通过显式区分 **Naive DP concatenated** 与 **Modular‑Kernel** 两种 `topk_ids` 形状，添加相应的索引计算逻辑并在不匹配时抛出明确的 AssertionError。同步新增单元测试覆盖四种关键情形，确保行为可靠。  

**🎯 影响范围**：  
- `vllm/model_executor/layers/fused_moe/routed_experts_capturer.py`（核心 MoE 路由捕获逻辑）  
- `tests/model_executor/test_routed_experts_capture.py`（新增/扩展的单元测试）  
- 受影响的上层模块包括 MoE 相关的 `fused_moe`、`model_executor` 以及分布式训练/推理的 DP 代码路径。  

**🔍 技术洞察**：  
- **架构影响**：  
  - `RoutedExpertsCapturer` 负责在模型前向阶段捕获每层 token 对应的 top‑k expert IDs，供后续统计或调度。该修复提升了在 **多 DP** 场景下的鲁棒性，防止因不同数据并行实现（Naive Concat vs Modular‑Kernel）产生的索引错位。  
  - 对外部接口保持不变，仍然接受 `layer_id` 与 `topk_ids`；仅在内部根据 `dp_metadata` 动态决定切片范围。  

- **性能影响**：  
  - 新增的分支判断、`torch.cumsum` 与整数转换开销极低，属于 O(1) 常量时间，对整体前向吞吐基本无感。  
  - 通过提前捕获形状不匹配的错误，避免后续错误传播导致的潜在计算浪费。  

- **安全考虑**：  
  - 代码仅在内部做形状校验与索引计算，不涉及外部输入的直接执行或内存写越界。  
  - 新增的 AssertionError 提供明确错误信息，利于快速定位问题，提升系统可观测性。  

**⚠️ 潜在风险**：  
1. **DP 元数据不一致**：如果 `dp_metadata.num_tokens_across_dp_cpu` 与实际 `topk_ids` 行数不匹配，仍会触发 AssertionError，导致服务中断。需要确保上层 DP 调度逻辑始终同步元数据。  
2. **类型兼容性**：`num_tokens_across_dp_cpu` 目前假设为 `torch.Tensor`（整型），若未来改为 Python `list` 或 `np.ndarray`，直接的 `torch.cumsum` 与 `.item()` 调用会报错。  
3. **测试覆盖局限**：单元测试使用了 `SimpleNamespace` 与 `patch` 替代真实上下文，真实分布式环境中可能出现未覆盖的边界情况（如 DP = 0、异常负数等）。  

**💡 关注建议**：  
- **集成验证**：在多节点、多卡的真实 DP 环境下跑一次完整的 MoE 推理/训练流水线，确认 `RoutedExpertsCapturer` 的捕获数据与预期一致。  
- **日志与监控**：在捕获路径加入适度的 DEBUG 级别日志，记录 `dp_rank`、`token_num_per_dp`、`total`、`n` 等关键变量，以便在异常触发 Assertion 时快速定位根因。  
- **兼容性防护**：考虑在代码中增加对 `num_tokens_across_dp_cpu` 类型的容错（如 `torch.as_tensor`），防止未来元数据结构变更导致硬崩。  
- **文档更新**：在 MoE 模块的部署指南中说明 `RoutedExpertsCapturer` 对 DP 布局的假设，以及在使用 Modular‑Kernel 路径时必须保证 `topk_ids` 只包含本 rank 的 token。  

> 综合来看，此次修复解决了一个在多 DP 环境下可能导致隐藏错误的关键断言，提升了 vLLM MoE 路由捕获的可靠性，风险可控且对性能影响可忽略。建议尽快合并并在 CI 中加入多 DP 场景的集成测试，以防止回归。

## ⚠️ 潜在风险  

1. **向后兼容破坏**  
   - 外部项目仍使用旧模块路径会立即报错。若这些项目在生产环境中未同步升级，会导致服务不可用。  
2. **环境变量冲突**  
   - `utils.py` 自动写入 `UCX_RCACHE_MAX_UNRELEASED=1024`。在极少数需要自定义该变量的部署（例如使用自研 UCX 参数）可能被意外覆盖。虽然会打印 warning，但仍可能导致运行时行为变化。  
3. **动态库加载顺序**  
   - 通过 `utils.py` 的懒加载，`NixlWrapper` 可能在多线程环境中首次被引用时触发 import（内部可能进行一次全局初始化），如果在多个进程/线程同时首次 import，可能出现竞争导致 `ImportError` 或状态不一致。  
4. **包装层的属性/方法泄露**  
   - `NixlConnector` 仍通过 `self.connector_scheduler` / `self.connector_worker` 访问内部实现，若外部直接访问 `NixlConnector.connector_scheduler`（原来不暴露），可能产生不稳定的依赖。  
5. **打包/发布遗漏**  
   - 新增的子目录文件必须在 `setup.cfg` / `MANIFEST.in` 中被正确包含，否则在 pip 安装或 CI 中缺失导致运行时错误。

### [Core][Metrics] Remove `vllm:prompt_tokens_recomputed` metric (#38709)
**SHA**: `72ff142` | 🔗 [查看提交](https://github.com/vllm-project/vllm/commit/72ff142c37564790a11a186e64bbcaa1d4441142)

**🎯 变更类型**：功能增强 / 代码清理  
**⚡ 重要程度**：🟡 中  

**📋 变更摘要**  
本次提交在 `vllm` 的指标体系中删除了 `vllm:prompt_tokens_recomputed` 及其对应的统计字段 `recomputed_tokens`。  
- `PromptTokenStats` 的不变量由 “computed + local_cache_hit + external_kv_transfer – recomputed = total” 简化为 “computed + local_cache_hit + external_kv_transfer = total”。  
- 相关计数器、日志记录以及单元测试均同步去掉了对该字段的引用，改为直接使用 `local_cache_hit` 与 `external_kv_transfer` 的差值。

**🎯 影响范围**  
- `vllm/v1/metrics`（日志记录、计数器创建）  
- `vllm/v1/metrics/stats.py`（PromptTokenStats 数据结构与更新逻辑）  
- `tests/v1/metrics`（单元测试校验）  

**💡 关注建议**  
1. **兼容性检查**：确认项目其它模块（如监控面板、外部插件）没有仍在读取 `vllm:prompt_tokens_recomputed`，若有需同步更新或提供迁移警告。  
2. **负数缓存命中**：代码注释已提示 `local_cache_hit` 可能因抢占而出现负值，建议在上报或可视化时做好防护，避免误报。  
3. **文档同步**：更新官方文档、Prometheus 指标说明及代码注释，说明 `recomputed_tokens` 已被合并入 `prompt_tokens_cached` 的计算方式。  
4. **回归测试**：运行全套 CI，尤其是多 GPU 与分布式调度路径，确保在全部缓存（全局 prefix cache）场景下仍能得到正确的 `computed`、`local_cache_hit` 与 `external_kv_transfer`。  

总体来看，此次简化消除了冗余指标，提升了统计模型的可读性和实现的直观性，只要完成上述兼容性与文档工作即可安全上线。

### [HMA] [KVEvent] Enable GPU-side KV events for HMA (#37688)
**SHA**: `cc07dad` | 🔗 [查看提交](https://github.com/vllm-project/vllm/commit/cc07dad789e9ae3f02ab6a4794800b8c1cc240c6)

**🎯 变更类型**：功能增强（GPU‑side KV 事件加入 `group_idx`）  
**⚡ 重要程度**：🟡 中  
**📋 变更摘要**：为 `BlockStored`、`BlockRemoved` 事件新增 `group_idx` 字段，并在 `__hash__` 中加入该字段，使不同 HMA 组产生不同的 hash。`BlockPool.cache_full_blocks` 与 `_maybe_evict_cached_block` 现会把组号写入事件；`VllmConfig` 不再因启用 KV 事件而强制关闭混合 KV 缓存管理器；新增一系列单元测试覆盖默认值、哈希行为、以及跨组的事件生成。  

**🎯 影响范围**  
- `vllm/distributed/kv_events.py`（事件结构、哈希）  
- `vllm/v1/core/block_pool.py`（事件产生处）  
- `vllm/config/vllm.py`（禁用 HMA 的逻辑）  
- 示例 `kv_events_subscriber.py`（展示 `group_idx`）  
- 多处测试文件以及 `engine` 测试用例。  

**💡 关注建议**  
1. **向后兼容**：`group_idx` 设为 `None` 的默认值，保证已有消费者仍能正常解析；如有外部序列化（JSON、protobuf）请检查是否需要迁移。  
2. **序列化/网络层**：`KVCacheEvent` 通过 `Publisher` 传输，确认 `group_idx` 已加入对应的 schema（尤其是 `null` 情况），防止协议不匹配。  
3. **性能**：哈希中加入整数字段成本极低，但频繁创建事件仍会带来微小开销，建议在生产环境打开 `enable_kv_cache_events` 前评估事件消费端的吞吐。  
4. **文档**：在 `config.kv_events.KVEventsConfig`、`BlockPool` 与示例中补充 `group_idx` 说明，帮助用户理解 HMA 组号的意义。  
5. **测试覆盖**：新增的单元测试已覆盖默认值、哈希唯一性及跨组行为，建议在 CI 中保留这些测试，并在未来加入对序列化/反序列化的端到端校验。  

总体而言，此次改动为 HMA 场景下的 KV 事件提供了必要的分组信息，提升了前缀缓存路由的灵活性，只要注意上述兼容和文档细节即可安全上线。

### fix(kimi_k25): resolve media_placeholder_token_id from tokenizer (#39344)

**🎯 影响范围**

- `vllm/model_executor/info`（间接使用 `media_token_id`）

### [Bugfix] Runtime driver check for cuMemcpyBatchAsync in swap_blocks_batch (#38919)
**SHA**: `bd8bd52` | 🔗 [查看提交](https://github.com/vllm-project/vllm/commit/bd8bd52308ea40c0d534d17969110b03e935a382)

**⚡ 重要程度**：🟡 中

   - 对 **ROCm** 编译保持不变，确认 `#if !defined(USE_ROCM)` 条件仍然有效。  

2. **线程安全**  
   - `static BatchFn batch_fn` 在首次调用时会执行一次 `cuGetProcAddress`，该初始化在多线程环境下是隐式的（C++ 静态局部变量）且是线程安全的（自 C++11 起），但仍建议在文档中说明此前置条件，以防未来改为手动初始化导致竞态。  

3. **错误处理**  
   - 当 `batch_fn` 为 `nullptr` 时直接走回退路径，已避免崩溃；若解析成功但返回错误码（如不支持的属性），仍会触发 `TORCH_CHECK`。可考虑在错误信息中加入 `cudaGetErrorString` 或 `cuGetErrorString` 以提供可读的错误原因。  

4. **性能回退**  
   - 单块 `cudaMemcpyAsync` 的循环在大量块时会显著降低吞吐量。建议在回退路径上加入 **批量拷贝的手动聚合**（如合并相邻块）或在文档中提示用户在旧驱动上预期性能下降。  

5. **编译依赖**  
   - 确认 `#include <cuda.h>` 已经提供 `cuGetProcAddress` 与 `CU_GET_PROC_ADDRESS_DEFAULT`，并且 `CMakeLists.txt` 中的 `CUDA_NVCC_FLAGS` 不会因为缺少 `-lcuda` 而导致链接错误。  

6. **回归检查**  
   - 运行已有的 `vllm` benchmark，比较开启/关闭 `cuMemcpyBatchAsync`（可通过环境变量或编译宏强制走回退）下的延迟与吞吐，确保功能正确且性能提升符合预期。  

总体而言，此改动提升了 `vllm` 对老旧驱动的鲁棒性，逻辑清晰，回退实现安全。完成上述兼容性与性能回归验证后即可合入主线。
