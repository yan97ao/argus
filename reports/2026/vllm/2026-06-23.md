### [Frontend] Split ServingRender into renderer and entrypoint. (#44285)
**SHA**: `31ca950` | 🔗 [查看提交](https://github.com/vllm-project/vllm/commit/31ca9504b14c328291fda1ad1a2659180ae8eb34)

**🎯 变更类型**：功能增强 / 重构  
**⚡ 重要程度**：🔴 高  

**📋 变更摘要**  
- 将原有 `OpenAIServingRender` 拆分为 **渲染器**（`OnlineRenderer`）和 **入口层**（`ServingRender`），后者在 `vllm/entrypoints/serve/render/serving.py` 中实现，仅负责把模型、渲染器、反渲染器组装进服务层。  
- 新增 `OnlineDerenderer` 用于把生成的 token 序列转化为 OpenAI‑compatible Chat/Completion 响应（包括 logprob 解析、tool‑call 解析等），并移动了之前 `OpenAIServingRender` 中的 “解码/反渲染” 逻辑。  
- 大量改动测试用例以及各入口文件的导入路径，将 `OpenAIServingRender` 替换为 `OnlineRenderer`（渲染）或 `ServingRender`（入口）。  
- `OnlineRenderer` 仍保留原 `OpenAIServingRender` 中的 **chat‑template、tool‑parsing、harmony‑兼容** 等核心功能，并对外提供 `render_chat`、`render_completion`、`preprocess_*` 等 API，供上层 `OpenAIServingChat/Completion` 调用。

| **性能影响** | - **渲染路径** 与之前基本相同（仍调用底层 `BaseRenderer` 的异步渲染），拆分本身不引入额外计算开销。 <br>- **反渲染分离** 可在仅需要 token 级返回（如 `/v1/completions/render`）时避免加载不必要的解析器，潜在 **IO/CPU** 节约。 <br>- 由于 `ServingRender` 现在只持有引用而不再实现大量业务逻辑，实例化成本略有下降。 |

| **可维护性** | - 代码 **重复度降低**（原 `OpenAIServingRender` 中的 600+ 行被拆分成两块，分别专注渲染/反渲染）。 <br>- 测试文件均已迁移至新类，覆盖面保持。 <br>- 新增的 `OnlineDerenderer` 与 `OnlineRenderer` 将来可以单独发布或替换（例如支持不同的 tokenizer 实现），提升演进灵活性。 |
| **兼容性** | - 所有入口（API routers、`OpenAIServingChat/Completion/Responses`）已改为使用 `online_renderer` 参数，运行时行为保持不变。 <br>- 仍保留 `OnlineRenderer` 的 `model_config`、`renderer`、`chat_template*` 等属性，同旧实现兼容。 <br>- 迁移期间，需要 **更新所有内部 import**（已在提交中完成），外部用户如果直接引用 `vllm.entrypoints.serve.render.serving.OpenAIServingRender` 将会失效，文档需同步提醒。 |

## ⚠️ 潜在风险

1. **向后兼容破坏**  
   - 直接引用旧类名 `OpenAIServingRender` 的第三方代码会立即失效。建议在 `vllm/entrypoints/serve/render/__init__.py` 中保留一个兼容 shim（例如 `OpenAIServingRender = OnlineRenderer`）或在发行说明中明确迁移步骤。  

2. **功能不一致**  
   - `OnlineDerenderer` 中的日志/工具解析实现与原 `OpenAIServingRender` 必须保持完全一致。任何细微的差异（如 `default_chat_template_kwargs` 的合并顺序）都可能导致用户看到不同的 `tool_calls` 或 `logprobs`。需要在 CI 中加入 **对比测试**（同一请求分别走旧渲染和新渲染，断言结果相等）。  

3. **线程安全/资源泄漏**  
   - `ServingRender` 持有两个渲染器实例；如果其中任意实例在 `__del__` 或 `close` 时需要释放资源，而未在 `ServingRender` 的生命周期里显式调用，可能导致内存/文件句柄泄漏。  

4. **错误堆栈信息丢失**  
   - `log_error_stack` 参数只在 `OnlineRenderer` 中使用，`OnlineDerenderer` 的错误包装未传递该标记，导致调试信息不完整。

**🎯 影响范围**  
- `vllm/distributed/kv_transfer/kv_connector/v1/moriio/`（`moriio_common.py`, `moriio_connector.py`, `moriio_engine.py`）  
- 相关单元测试 `tests/v1/kv_connector/unit/`（新增 `test_moriio_tp_ack.py`，以及对已有测试的适配）

| **架构影响** | - **ACK 结构化**：从原来的字符串 `transfer_id` 升级为 `NamedTuple` `MoRIIOTransferAck(transfer_id, consumer_tp_size)`，使 ACK 能携带解码 TP 大小信息，兼容旧协议（通过 `isinstance(str)` 兼容）。<br>- **异构 TP 映射**：`get_moriio_remote_tp_rank` 为 `prefill‑TP ↔ decode‑TP` 提供映射，支持 `remote_tp_size` 与本地 `tp_size` 不同的情形（必须是整倍数），并在握手阶段使用。<br>- **Producer‑side Fan‑in**：在 `MoRIIOConnectorWorker.get_finished` 中聚合多个解码 rank 发来的 ACK，只有在收到 **所有期望的 ACK**（`get_moriio_expected_ack_count`）后才将 transfer 标记为完成，防止块提前回收。<br>- **状态维护**：新增 `_consumer_notification_counts` 与 `_completed_consumer_notifications`，确保 ACK 计数在多线程/多进程环境下保持一致。 |
| **性能影响** | - **额外计数开销**：每条 ACK 需要一次字典查找与计数更新，复杂度 `O(1)`，对高吞吐量的影响可以忽略。<br>- **Fan‑in 延迟**：当 decode TP > prefill TP 时，需要等待多个 ACK 才能释放块，最坏情况下延迟会增加到 `consumer_tp_size / producer_tp_size` 倍（已在 `get_moriio_expected_ack_count` 中做出预期）。<br>- **网络负载**：`send_notify` 现在会在通知中携带 `consumer_tp_size`，略微增大消息体，但仍然是几百字节以内，对网络带宽影响极小。 |
| **安全考虑** | - **输入校验**：在 `get_moriio_remote_tp_rank`、`validate_moriio_heterogeneous_tp_kv_heads`、`_handle_release_message` 中加入了正数、倍数、范围检查，防止恶意/错误的 `remote_tp_size`、`consumer_tp_size` 导致异常映射或整数除零错误。<br>- **兼容性防护**：`resolve_moriio_transfer_ack` 对于非活跃 `transfer_id`、已完成的 ACK 均直接返回 `None`，避免重复处理可能导致的资源泄漏或状态不一致。 |

### ⚠️ 潜在风险  

1. **ACK 丢失或顺序错乱**  
   - 依赖 `pop_finished_req_ids` 返回所有未处理的 ACK。若底层 MoRIIO 实现出现消息丢失（网络抖动），可能导致 `resolve_moriio_transfer_ack` 永远等待，导致块回收卡死。  
2. **不匹配的 TP 大小**  
   - `remote_tp_size` 必须是本地 TP 大小的倍数或其因子。若上游调度系统配置错误，`ValueError` 会在运行时抛出，可能导致整个 KV‑transfer 过程中止。  
3. **兼容性回退**  
   - 老版 MoRIIO（不发送 `consumer_tp_size`）仍能工作，但旧版的 `send_notify` 在处理结构化 ACK 时会收到 `None`，仍然返回 `TransferId`，兼容性已测试。若未来删除兼容路径，需同步升级所有节点。  
4. **并发竞争**  
   - `_consumer_notification_counts` 与 `_completed_consumer_notifications` 通过 `self.lock`（在 `MoRIIOEngine` 中）保护，但 `MoRIIOConnectorWorker.get_finished` 读取这些结构时未加锁。若 `get_finished` 与 ACK 处理在不同线程并行，可能出现短暂的竞争条件（计数少加一次）。
