### [Frontend][4/n] Improve pooling entrypoints | pooling. (#39153)
**SHA**: `66c079a` | 🔗 [查看提交](https://github.com/vllm-project/vllm/commit/66c079ae83f5d5200306da28acb8a0949a6a9a45)

**🎯 变更类型**：功能增强 / 重构 / 架构变更  
**⚡ 重要程度**：🔴 高  
**📋 变更摘要**：本次提交对 vLLM 的 **Pooling**（embedding / classification / token‑classify / scoring）入口进行大幅重构，核心改动包括  
1. **彻底移除 EngineClient 中的 `io_processor` 成员**，改为在 Pooling 层通过 `init_pooling_io_processors` 按需实例化插件化 IOProcessor。  
2. 引入统一的 **`PoolingIOProcessor`** 基类，并实现 `PluginWithIOProcessorPlugins`、`PluginWithoutIOProcessorPlugins` 两种实现，统一线上/线下前后处理逻辑。  
3. 新增 **`ServingPooling`**（原 `OpenAIServingPooling`）及其抽象基类 `PoolingServingBase`，把请求的初始化、生成、后处理完整封装为统一的调用链。  
4. 更新所有测试、路由、渲染等相关代码以适配新结构，去掉大量 `engine.io_processor` 的 mock 与传参。  
5. 新增 `has_io_processor` 辅助函数用于检测是否配置了插件，防止在未装插件时产生运行时错误。  

预期效果是：**插件化 IOProcessor 更加灵活**（支持无插件、单插件、不同任务的插件），**代码结构更清晰**（Pooling 与 OpenAI 业务解耦），**维护成本降低**，并为后续在 Pooling 场景引入更多自定义前后处理提供统一入口。

---

### 🎯 影响范围
- **核心模块**：`vllm/entrypoints/pooling/*`（IOProcessor、serving、factory、api_router）  
- **Engine 客户端**：`vllm/engine/protocol.py`、`vllm/v1/engine/*`（去除 `io_processor`）  
- **OpenAI 入口**：`vllm/entrypoints/openai/*`（渲染、模型、服务初始化）  
- **测试套件**：全部 `tests/entrypoints/*/pooling*`、`tests/entrypoints/openai/*` 相关文件  
- **插件子系统**：`vllm/plugins/io_processors/*`（新增检测函数）

### 🔍 技术洞察

| 维度 | 影响描述 |
|------|----------|
| **架构影响** | 1. **职责分离**：原本 EngineClient 持有 `io_processor`，导致模型层与前后处理耦合。现在 IOProcessor 与 Engine 完全解耦，由 Pooling 层自行加载，符合单一职责原则。<br>2. **插件化路径**：通过 `init_pooling_io_processors` 根据 `SupportedTask` 动态构建对应的 `PoolingIOProcessor`，并在 `ServingPooling` 中统一调用。<br>3. **统一请求流**：`PoolingServingBase` / `PoolingServing` 把 “初始化 ctx → 前处理 → 生成 → 后处理 → 构造响应” 整合为一次 `__call__`，去除了之前散落在 `OpenAIServingPooling.create_pooling` 中的多分支代码。 |
| **性能影响** | - **正向影响**：省去每次请求都在 EngineClient 上检查 `io_processor`（属性访问、空值判断）以及不再在 `OpenAIServingRender` 中持有多余的 `io_processor` 实例，略微降低内存占用。<br>- **潜在开销**：`init_pooling_io_processors` 需要在服务启动时实例化所有任务对应的 IOProcessor；若插件实现内部资源占用大（如加载外部模型或大文件），会导致启动时间增加。整体对推理吞吐量影响有限，主要在请求调度前的前处理阶段。 |
| **安全考虑** | - 新增的 `has_io_processor` 通过 `VllmConfig` 与模型 HF 配置双向检验插件是否存在，避免在未配置插件时误调用导致未捕获异常。<br>- `PluginWithIOProcessorPlugins.pre_process_online` 中对插件返回的 `prompt` 进行显式类型检查，确保只接受 `bytes` 或可解析为 `ModelPrompt`，降低代码注入风险。<br>- 所有错误路径统一使用 `ErrorResponse`，不再泄露内部实现细节。 |
| **可维护性** | - 删除了大量在测试和业务代码中对 `engine.io_processor` 的手工 mock，降低了维护成本。<br>- 通过抽象基类 `PoolingServingBase` 和统一的 `PoolingIOProcessor` 接口，今后新增任务（如 `rerank`）仅需实现对应的 IOProcessor，无需侵入核心调度逻辑。<br>- 代码量虽然在迁移阶段增加（新类/工厂），但每个文件职责更单一，阅读与定位 BUG 更加直接。 |
| **向后兼容** | - 公开的 OpenAI API（`/v1/embeddings`、`/v1/classifications` 等）行为保持不变；仅在错误信息中加入 “No IOProcessor plugin installed.” 的提示。<br>- 仍支持旧的 “plugin” 任务，仅当配置了插件时才会生效，否则抛出明确错误。 |

### ⚠️ 潜在风险
1. **启动时插件实例化失败**：如果插件内部依赖外部资源（如模型权重、网络服务），启动期间抛异常会导致整个 vLLM 实例不可用。建议在插件实现中捕获并包装异常，给出友好提示。  
2. **请求路径中遗漏 `pooling_params`**：`PoolingServeContext.pooling_params` 在新实现里必须在 `pre_process_online_async` 前被填充，若某自定义 IOProcessor 不返回 `PoolingParams`，后续会触发 `assert ctx.pooling_params is not None`。需确保所有插件实现 `create_pooling_params` 或在前处理阶段显式设置。  
3. **旧代码仍引用已删除属性**：如 `engine_client.io_processor`、`OpenAIServingRender.io_processor` 等未全部清理完成，可能在第三方扩展或用户自定义代码中产生 AttributeError。  
4. **并发场景下的插件状态**：`PluginWithIOProcessorPlugins` 在每个 `ServingPooling` 实例中持有单例插件对象，若插件内部不是线程安全的（例如使用全局变量），并发请求可能出现竞态。需要在插件实现中明确文档化线程安全要求或自行实例化。  
5. **测试覆盖不足**：虽然大量测试已更新，但仍缺少对 **异常路径**（如插件加载失败、`pre_process_offline` 参数不匹配）的集成测试，可能导致上线后隐藏 bug。

### 💡 关注建议
1. **插件健康检查**  
   - 在服务启动后执行一次 `has_io_processor` 并尝试实例化插件，若失败记录告警并提供可选的 “fallback to dummy processor”。  
2. **文档同步**  
   - 更新官方文档说明 **Pooling** API 已不再需要在模型配置中手动传递 `io_processor`，改为 `--pooler-config.task` 或通过 `VllmConfig.model_config.io_processor_plugin` 指定。  
   - 为插件开发者提供 `PoolingIOProcessor` 接口的实现示例，明确 `pre_process_online`、`post_process_online`、`pre_process_offline` 等必实现方法。  
3. **线程安全审计**  
   - 对已内置的 `ScoringIOProcessor`、`ClassifyIOProcessor`、`EmbedIOProcessor` 等进行并发安全评估，必要时在构造时使用 `threading.Lock` 或 `asyncio.Lock` 包装共享资源。  
4. **回滚路径**  
   - 若用户仍需旧行为，可通过环境变量或配置项保留 `engine_client.io_processor`（保持兼容），并在代码中加入 `DeprecationWarning`。  
5. **性能基准**  
   - 建议在生产环境跑一次端到端基准（包括前处理、生成、后处理），对比插件开启/关闭两种情形，确保额外的前处理开销在可接受范围内。

### [W8A8 Block Linear Refactor][2/N] Remove W8A8Fp8BlockLinearOp and adopt Fp8 block linear kernel selections. (#33892)
**SHA**: `2e9034c` | 🔗 [查看提交](https://github.com/vllm-project/vllm/commit/2e9034c998e231634a75ab74c5812e0dba2cf3a1)

**🎯 变更类型**：功能增强 / 重构 / 架构变更  
**⚡ 重要程度**：🔴 高  

---

## 📋 变更摘要
1. **移除 `W8A8BlockFp8LinearOp`**，引入统一的 **FP8 Linear Kernel 选型** 接口 `init_fp8_linear_kernel`。  
2. **新增抽象基类 `MMLinearKernel`/`MMLinearLayerConfig`** 与对应的 **Params**（`FP8Params`、`Int8Params`），统一量化层参数访问与 kernel 实现方式。  
3. **实现块级 FP8 线性算子抽象 `Fp8BlockScaledMMLinearKernel`**，并提供 **动态调度子类**（FlashInfer ↔ DeepGemm、Cutlass ↔ Triton ↔ AIter），形成 **可插拔的优先级列表**（`_POSSIBLE_FP8_BLOCK_KERNELS`）。  
4. **扩展 `init_fp8_linear_kernel`** 来同时支持 **标量/块级** 以及 **输入 dtype**，并在各量化实现（CompressedTensors、ModelOpt、Online FP8、Quark、FBGEMM、MXFP8 等）中统一使用。  
5. **大量测试、基准、工具链同步**（benchmark、compile passes、utils、conftest）以适配新的 kernel 初始化与参数结构。  
6. **删除/精简老旧实现**（`W8A8BlockFp8LinearOp`、相关处理函数、flashinfer/aiter/torch 直接调用代码）并迁移至新抽象。  

> 这些改动把 “FP8 线性算子” 从 **硬编码的单一实现** 转变为 **可组合、可扩展、平台感知的 kernel 选择层**，为后续新 kernel（如 Future‑FP8、NVFuser）提供统一入口。

## 🔍 技术洞察

### 1. 架构影响
| 维度 | 说明 |
|------|------|
| **统一抽象层** | `MMLinearKernel` 抽象了 **配置 → 参数 → 实际算子** 的完整链路。所有 FP8 相关 kernel 必须实现统一的 `apply_weights` 与 `process_weights_after_loading`，极大提升可维护性和可扩展性。 |
| **插件式 kernel 选择** | 新增 `_POSSIBLE_FP8_BLOCK_KERNELS` 与 `_POSSIBLE_KERNELS`，依据 **平台、算子优先级、配置兼容性** 动态挑选实现，避免硬编码 `if/else`，并在 `init_fp8_linear_kernel` 中统一日志记录。 |
| **块级支持** | 引入 `Fp8BlockScaledMMLinearKernel` 与 `Fp8BlockScaledDynamicMMLinearKernel`，将 **块级量化（group shape）** 与 **是否需要输入量化**（`apply_input_quant`）抽象出来，降低每个子类的重复实现量。 |

### [kv_offload+HMA][5/N]: Track group block hashes and block IDs (#37109)
**SHA**: `512c5eb` | 🔗 [查看提交](https://github.com/vllm-project/vllm/commit/512c5eb455594b7366b6ae5c27e934bdcc5eed6e)

**📋 变更摘要**：本次提交在 KV‑offload 子系统中引入了 **OffloadKey**（原 BlockHash 的演进），将 **块哈希 + KV‑group 索引** 打包为单一的 bytes 标识，并在调度器、CPU offload manager、缓存策略以及过滤复用管理器等所有核心路径全面改用 OffloadKey。除此之外，调度器新增了对 KV‑group 配置的抽象，支持多组 KV‑cache（HMA）以及更细粒度的 block‑ID 与 offload‑key 对齐。单元测试同步改写以验证新接口。

## 🔍 技术洞察

| 维度 | 影响描述 |
| ---- | -------- |
| **架构影响** | <ul><li>**统一标识**：OffloadKey 将 block hash 与所属 KV‑group 索引合并，消除了跨模块对两套数据结构的同步需求，调度器、OffloadingManager、缓存策略只需处理单一类型。</li><li>**多 KV‑group 支持**：`SchedulerOffloadConfig`、`GroupOffloadConfig`、`RequestGroupState` 等新类抽象了每个 cache group 的 `gpu_block_size`、`offloaded_block_size` 与 hash‑block‑size‑factor，调度器现在可在 **单次调度步骤** 处理多个 KV‑group（为 HMA 提前铺路）。</li><li>**向后兼容**：旧的 `BlockHash` 仍在代码库的内部实现中保留（仅用于类型别名），但外部 API 已全部迁移到 OffloadKey，需在使用方做好迁移适配。</li></ul> |
