### [Feature] Add Azure Blob Storage support for RunAI Model Streamer (#34614)
**SHA**: `a3e2e25` | 🔗 [查看提交](https://github.com/vllm-project/vllm/commit/a3e2e250f09d7a347cfdccfe2f7b593edd1b7bce)

**🎯 变更类型**：功能增强  
**⚡ 重要程度**：🟡 中  
**📋 变更摘要**：为 RunAI Model Streamer 增加 Azure Blob Storage（az://）支持，更新依赖、文档、DSL 检测以及错误提示。  
**🎯 影响范围**：  
- `vllm/transformers_utils/runai_utils.py`、`vllm/transformers_utils/utils.py`（URI 判定）  
- `vllm/model_executor/model_loader/runai_streamer_loader.py`（加载说明）  
- 配置验证 `vllm/config/vllm.py`（错误信息）  
- Docker 镜像、requirements、setup 中的依赖版本升级  
- 文档、测试用例以及 CI 依赖（Azure SDK、身份认证）  

**💡 关注建议**  
1. **兼容性**：新增 `az://` 方案后，`SUPPORTED_SCHEMES` 与 `is_cloud_storage` 必须保持同步，确保旧代码在未安装 Azure SDK 时仍能正常导入。可考虑在 `runai_utils` 中捕获 `ImportError` 并回退。  
2. **安全凭证**：加载 Azure Blob 依赖 `DefaultAzureCredential`，运行环境若缺少相关环境变量或登录状态会报错。建议在 README 中明确说明必须 `az login` 或提供服务主体凭证。  
3. **测试覆盖**：已加入 `is_azure` 与 URI 判定单元测试，建议补充对实际 Azure Blob 读取的集成测试（可使用本地 Azurite）避免在 CI 环境缺少网络资源导致 flaky。  
4. **错误提示**：`vllm/config/vllm.py` 中的提示文字已统一为 “object storage (S3/GCS/Azure)”，但仍仅在 `load_format` 为 `runai_streamer[_sharded]` 时检查。若后续扩展其他加载方式，需要对应更新。  
5. **依赖冲突**：升级 `runai-model-streamer` 到 0.15.7 并显式加入 `azure-identity`, `azure-storage-blob`，可能与用户已有的 Azure SDK 版本产生冲突。建议在 `setup.cfg` 中使用宽松的版本范围（如 `>=1.25,<2`），并在发布说明中提醒。  

总体而言，改动集中在 Azure 支持层面，影响范围局限于模型加载路径和依赖升级，风险较低。若遵循上述建议，可平滑上线而不影响已有 S3/GCS 工作流。

### [Misc] Add online audio_in_video test (#36775)
**SHA**: `143e4dc` | 🔗 [查看提交](https://github.com/vllm-project/vllm/commit/143e4dccdfd8293c70c76f8d32a60ce23ecc23ea)

**🎯 变更类型**：功能增强 / Bug 修复  
**⚡ 重要程度**：🟡 中  

**📋 变更摘要**  
1. 为 `audio_in_video` 场景新增在线端到端测试 `test_audio_in_video.py`，验证模型能在视频中提取音频并参与推理。  
2. 在 `requirements/test` 中加入 `av==16.1.0`，并在 `tests/multimodal/media/test_audio.py` 中使用 `librosa` 读取视频音轨进行对比验证。  
3. `vllm/entrypoints/serve/render/serving.py` 的 `_preprocess_chat` 逻辑扩展：从模型配置读取 `multimodal_config`，并将 `media_io_kwargs` 与 `mm_processor_kwargs` 通过 `default_*` 参数注入渲染链路，实现对 `use_audio_in_video` 的统一传递。  

**🎯 影响范围**  
- **测试套件**：新增 `tests/entrypoints/openai/test_audio_in_video.py`、改动 `tests/multimodal/media/test_audio.py`。  
- **依赖**：`av` 与 `librosa` 成为测试运行的必备库，可能牵涉到系统层面的 FFmpeg 支持。  
- **服务端渲染层**：`serving.py` 中的聊天前处理现在依赖 `model_config.multimodal_config`，所有调用该入口的代码路径会感受到额外的默认参数注入。  

**💡 关注建议**  
1. **兼容性**：`av` 依赖 FFmpeg，建议在 CI 中加入平台检查或使用 `pytest.importorskip`，防止在缺少编解码器的环境中导致测试报错。  
2. **后向兼容**：`mm_config` 可能为 `None`，代码已做防护，但所有自定义 `model_config` 必须保证 `multimodal_config` 字段存在（即使为空），否则可能触发属性错误。  
3. **性能**：视频解码与音轨抽取在测试中会占用显著资源，建议在非必需的 CI 环境中对该测试设 `slow` 标记或提供 `--skip-audio-in-video` 开关。  
4. **文档与示例**：更新 README / API 文档，说明 `use_audio_in_video` 参数的作用及依赖的 `av`/`librosa` 安装方式。  
5. **异常处理**：`AudioMediaIO.load_bytes` 现在直接使用 `librosa.load`，若视频格式不被支持，异常会向上抛出。建议捕获并包装为统一的 `MultimodalError`，避免服务器端意外崩溃。  

整体来看，此次提交为多模态音视频处理提供了关键功能验证，但引入的系统级依赖与渲染路径的默认参数传播需要在文档、CI 与异常处理上做相应补强。

### [Frontend] Remove `torchcodec` from audio dependency (#37061)
**SHA**: `6590a3e` | 🔗 [查看提交](https://github.com/vllm-project/vllm/commit/6590a3ecdafdc001f29c3820b80dd14d994e640c)

**🎯 变更类型**：重构 / 依赖去除  
**⚡ 重要程度**：🟡 中  
**📋 变更摘要**：本次提交在前端 Speech‑to‑Text 路径中移除对 `torchcodec`（via torchaudio）的依赖，改为仅使用 `librosa` + `soundfile`，并在解析失败时回退到基于 `pyav`（`extract_audio_from_video_bytes`）的解码实现。为此删掉了 `vllm/entrypoints/openai/speech_to_text/utils.py`，并在入口文件中内联了音频加载逻辑，同时在 `setup.py` 中删除了 `torchcodec` 依赖声明。

**🎯 影响范围**  

| 模块 | 影响 | 说明 |
|------|------|------|
| `setup.py` | 依赖列表变更 | `torchcodec` 已不再列入 `audio` 需求，安装体积与编译链会减少。 |
| `vllm/entrypoints/openai/speech_to_text/speech_to_text.py` | 业务逻辑重写 | 删除 `load_audio_bytes` 调用，改为直接使用 `librosa.load`，并在 `soundfile` 报错时调用 `extract_audio_from_video_bytes`。 |
| `vllm/multimodal/media/audio.py`（间接） | 新增 `extract_audio_from_video_bytes` 调用 | 依赖 `pyav`，若未安装会在运行时报错。 |
| 已删除 `vllm/entrypoints/openai/speech_to_text/utils.py` | 代码体积下降 | 该文件原本封装了 torchaudio 的回退路径，现已废弃。 |

**💡 关注建议**  

1. **运行时依赖**：确保 `pyav`（或 `av`）在生产环境中可用，否则在 `soundfile` 无法解码的容器（如 MP4/M4A）时会直接抛出 `ValueError`。建议在文档或 `requirements.txt` 中说明此软依赖。  
2. **异常路径测试**：新增单元测试覆盖以下场景：  
   - 支持的 WAV/FLAC 等直接由 `librosa` 解码的文件。  
   - 不被 `soundfile` 支持的 MP4/M4A，验证 `pyav` 回退成功。  
   - 完全损坏或未知格式，确保抛出统一的 `ValueError`。  
3. **性能评估**：`pyav` 解码涉及子进程或内部 FFmpeg 调用，可能比原先的 torchaudio（基于 TorchCodec）慢。对常见容器格式做基准测试，确认延迟在可接受范围。  
4. **兼容性**：`torchcodec` 原本是可选依赖，移除后如果用户仍在环境中安装了 torchaudio，代码不会使用它，避免潜在的冲突。若用户依赖 torchaudio 提供的 GPU 加速功能，需自行在上层实现。  
5. **日志信息**：当前仅在 `debug` 级别记录回退过程，建议在 `info` 级别添加一次性提示，帮助运维快速定位因缺少 `pyav` 而导致的解码失败。  

总体来看，此次改动简化了音频解码链，降低了构建复杂度，但引入了对 `pyav` 的软依赖并改变了错误处理路径，需在部署文档、测试以及性能监控方面做相应补充。

### [Feature] Add InstantTensor weight loader (#36139)
**SHA**: `8c29042` | 🔗 [查看提交](https://github.com/vllm-project/vllm/commit/8c29042bb98e79546576ff1a46c9def863046258)

**🎯 变更类型**：功能增强（新增 InstantTensor 权重加载器）  
**⚡ 重要程度**：🟡 中  
**📋 变更摘要**：  
- 在 `LoadConfig` 中加入 `instanttensor` 选项，并在模型加载器的注册表里映射到 `DefaultModelLoader`。  
- 实现 `instanttensor_weights_iterator`，利用 `instanttensor.safe_open` 在 CUDA 环境下以分布式、流水线方式读取 Safetensors。  
- 更新依赖、Dockerfile、文档及测试，用 `--load-format instanttensor` 支持此方式加载模型。  

**🎯 影响范围**：  
- `vllm/config/load.py`、`vllm/model_executor/model_loader/__init__.py`、`vllm/model_executor/model_loader/default_loader.py`、`vllm/model_executor/model_loader/weight_utils.py`（核心加载逻辑）。  
- `requirements/*.txt`、`setup.py`（增加 `instanttensor` 依赖）。  
- `docs/models/extensions/instanttensor.md`（使用说明）。  
- 新增的单元测试 `tests/model_executor/model_loader/instanttensor_loader/*`。  

**💡 关注建议**  
1. **兼容性**：`instanttensor` 仅在 CUDA 环境工作，建议在 `LoadConfig` 中加入校验或 fallback，防止在 CPU / 非 NVIDIA 环境因 `--load-format instanttensor` 抛异常。  
2. **进程组获取**：当前在单元测试中捕获 `AssertionError` 来处理未初始化的进程组，建议使用更明确的 API（如 `is_distributed_initialized()`）以免误判。  
3. **错误信息**：当 `instanttensor` 未安装或 CUDA 不可用时抛出的 `ImportError/ValueError` 应在用户层统一包装为 `VLLMException`，保持异常风格一致。  
4. **性能基准**：文档给出加速比例，建议在 CI 中加入对比基准（safetensors vs instanttensor）并在 PR 注释中展示，以防回归。  
5. **依赖锁定**：`instanttensor>=0.1.5` 可能随时间产生不兼容，考虑在 `requirements` 中 pin 到具体小版本或在 `setup.cfg` 中添加兼容性说明。  

总体而言，此次改动为 vLLM 引入了高效的权重加载路径，代码结构清晰，测试覆盖基本功能。只需在异常处理和兼容检查上再做少量强化，即可在生产环境安全使用。

### [Model Runner V2] Add Support for XD-RoPE (#36817)
**SHA**: `3ed46f3` | 🔗 [查看提交](https://github.com/vllm-project/vllm/commit/3ed46f374b17d98ca6f098e74cb7c5fd4146179c)

**🎯 变更类型**：功能增强（为 Model Runner V2 添加 XD‑RoPE 支持）  
**⚡ 重要程度**：🟡 中  
**📋 变更摘要**：  
1. 删除旧的 `mrope_utils.py`，新建统一的 `rope.py`，用 `RopeState` 同时管理 M‑RoPE（含 delta）和 XD‑RoPE（无 delta）。  
2. `ModelRunner`、`ModelState`、`CUDAGraph` 等核心路径改为通过 `rope_state` 生成位置张量；在 CUDA‑graph 前向函数中占位 `intermediate_tensors`（后续用于 PP‑CUDA‑graph）。  
3. 相关代码路径（`prepare_inputs`、`prepare_dummy_inputs`、`add_request`、`apply_staged_writes`）均改为使用 `RopeState`，并在不存在多维 RoPE 时保持原有 1‑D 行为。  

**🎯 影响范围**：  
- `vllm/v1/worker/gpu/rope.py`（新实现）  
- `vllm/v1/worker/gpu/mm/mrope_utils.py`（删除）  
- `vllm/v1/worker/gpu/model_states/default.py`（创建/使用 `rope_state`）  
- `vllm/v1/worker/gpu/model_runner.py`、`cudagraph_utils.py`（加入 `intermediate_tensors` 参数、断言）  
- 相关接口 `SupportsMRoPE`、`SupportsXDRoPE` 以及 `ModelConfig` 中的 `uses_mrope`、`uses_xdrope_dim` 标记。

**💡 关注建议**  
1. **模型实现**：使用 XD‑RoPE 的模型必须实现 `SupportsXDRoPE.get_xdrope_input_positions`；使用 M‑RoPE 的模型仍需实现 `SupportsMRoPE`。未实现的模型在加载时会触发断言。  
2. **CUDA‑graph 兼容**：当前在 `cudagraph_utils.forward_fn` 中仍传 `intermediate_tensors=None`，后续 PR（#35162）会加入实际张量。若提前开启 PP‑CUDA‑graph，需确保该字段被正确填充，否则会触发运行时错误。  
3. **内存与性能**：`RopeState` 通过 `StagedWriteTensor` + UVA 仍保持原有的预写/拷贝策略，已兼容大 batch 场景。新增维度不会显著增加 GPU 内存，只会在 CPU‑UV‑A 区占用更多页面。  
4. **回归测试**：验证以下场景：  
   - 只使用 1‑D RoPE（默认） → 位置张量与旧实现一致。  
   - 启用 M‑RoPE → 位置张量维度 3，decode 阶段 delta 正确加入。  
   - 启用 XD‑RoPE（dim=3/4） → 位置张量维度对应，decode 时 delta 为 0。  
   - 在 `CUDAGraphMode.FULL` 下运行，确保 `intermediate_tensors` 为非空时模型仍能正常前向。  
5. **文档与配置**：更新 `ModelConfig` 注释，说明 `uses_xdrope_dim` 与 `uses_mrope` 的互斥关系；在 README/CHANGELOG 中补充 “Model Runner V2 – XD‑RoPE” 的使用说明。  

总体而言，此次改动为多维 RoPE 引入统一抽象，增强了模型兼容性，影响范围集中在 GPU worker 层，建议在 CI 中加入对新接口的覆盖测试后再发布。
