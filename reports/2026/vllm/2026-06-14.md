### [V1][Spec Decode] Add Dynamic SD (#32374)
**SHA**: `4ef4492` | 🔗 [查看提交](https://github.com/vllm-project/vllm/commit/4ef4492e9b7a5a7ba295da783d456d45db5eb9d6)

**🎯 变更类型**：功能增强（Dynamic Speculative Decoding）  
**⚡ 重要程度**：🔴高  
**📋 变更摘要**：  
本次提交在 vLLM 中引入 *Dynamic Speculative Decoding*（DSD），通过在运行时根据当前 batch‑size 动态调节 speculative token 数 (K)。新增配置字段 `num_speculative_tokens_per_batch_size`，实现了批量‑size → K 的密集查表，并在调度器、各 proposer、GPU model runner 以及统计模块中加入对可变 K 的支持。文档与大量回归/单元测试同步更新。

**🎯 影响范围**：  
- `vllm/config/speculative.py`、`vllm/config/vllm.py`（配置、CUDA‑graph 模式自动切换）  
- 调度器核心 `vllm/v1/core/sched/scheduler.py`、`output.py`、`async_scheduler.py`（动态 K 查表、占位符）  
- 所有 Speculative Decoding proposer (`llm_base_proposer.py`, `ngram_proposer*.py`, `suffix_decoding.py`, `medusa.py`, `extract_hidden_states.py`, `step3p5.py`)  
- `vllm/v1/worker/gpu_model_runner.py`（变量 draft token 长度的复制与维度管理）  
- 统计 `vllm/v1/spec_decode/metrics.py`（新增 draft‑tokens‑per‑pos 统计）  
- 文档 `docs/features/speculative_decoding/*`、`dynamic_speculative_decoding.md`  
- 新增工具 `vllm/v1/spec_decode/dynamic/utils.py` 与对应测试 `tests/v1/spec_decode/test_dynamic_sd.py`  

**🔍 技术洞察**  

- **架构影响**  
  - 在 `SpeculativeConfig` 中加入 `num_speculative_tokens_per_batch_size`，实现了 **配置‑驱动的运行时调度**。  
  - `Scheduler` 在初始化时依据该字段生成 **密集查表** (`dynamic_sd_lookup`)；调度阶段依据已调度请求数 (`len(num_scheduled_tokens)`) 取出当前 K，并通过 `SchedulerOutput.num_spec_tokens_to_schedule` 传递。  
  - 所有 proposer 接口统一新增 `num_speculative_tokens` 参数，支持 **K=0** 的“关闭 draft”路径，确保在动态 K 为 0 时仍能正常走 prefill/kv 同步。  
  - `GPUModelRunner` 通过 `prev_num_spec_tokens` 记录上一步的 K，动态调整 draft‑token Tensor 的切片复制逻辑，防止维度不匹配。  
  - 为兼容性，`VllmConfig._maybe_override_dynamic_sd_cudagraph_mode` 在启用 DSD 且使用 **全图** CUDA‑graph 时强制切换为 **PIECEWISE** 模式，避免因目标 verification 长度在运行时变化导致图失效。  

- **性能影响**  
  - **正向**：在并发变化明显的场景（RL rollout、QPS 波动）可自动降低 K，防止 **BS × K** 超出硬件瓶颈，从而提升 *tokens‑per‑second*（TPOT）。  
  - **负向**：在每个调度周期进行一次查表并在多个模块传播 K，带来极小的 CPU 开销（O(1) 查表）。  
  - **GPU 端**：切换到 Piecewise CUDA‑graph 可能略微降低 kernel‑launch 隐式优化，但相较于因 K 失配导致的图失效，整体收益更大。  
  - **内存**：`draft_token_ids` Tensor 维度随 K 变化，代码已在复制、CPU‑sync 处使用 `prev_num_spec_tokens` 防止悬空内存写，理论上不会产生额外碎片。  

- **安全考虑**  
  - 变更仅涉及调度与内部数据路径，无新增外部 I/O、权限或加密逻辑。  
  - 配置解析增加了严格的类型与范围校验（正数、非重叠、必须以 1 起始），防止恶意或错误配置导致异常崩溃。  
  - 自动强制切换 CUDA‑graph 模式也属于安全防护，避免因不兼容导致的未定义行为。  

**⚠️ 潜在风险**  

| 风险点 | 说明 | 可能后果 |
|--------|------|----------|
| **K=0 时的路径完整性** | 仅部分 proposer 添加了 `if self.num_speculative_tokens == 0` early‑return（如 `llm_base_proposer`），其他路径若仍假设 K>0 可能触发 shape mismatch。 | 运行时 AssertionError / CUDA kernel 错误，导致服务崩溃。 |
| **CUDA‑graph 切换** | 自动降级为 `PIECEWISE` 依赖 `cudagraph_mode.has_full_cudagraphs()` 检测，若未来新增模式未覆盖该判断，可能仍维持全图导致错误。 | 图失效、性能回退、潜在内存泄漏。 |
| **动态维度同步** | `GPUModelRunner` 中的 `prev_num_spec_tokens` 更新依赖 `_draft_token_ids` 为 Tensor；若出现 `None`（如未生成 draft）可能导致 stale `prev_num_spec_tokens`。 | `IndexError` 或越界复制。 |
| **配置误配** | `num_speculative_tokens_per_batch_size` 必须覆盖所有可能 batch‑size，若用户误删范围导致查表返回 `0`（默认），可能在非预期情况下关闭 SD。 | 性能下降、用户不易定位原因。 |
| **兼容性** | 目前仅在 Model Runner V1、非 full‑cuda‑graph 环境下验证，V2、MRv2 上仍标记为不支持。 | 误用导致启动失败或运行时异常。 |
| **测试覆盖不足** | 只针对调度与 lookup 写了单元测试，未覆盖全部 proposer 在 K=0 情形的行为。 | 隐蔽 bug 随版本发布。 |

**💡 关注建议**  

1. **完善 K=0 的兼容实现**  
   - 检查所有 proposer (`medusa`, `ngram`, `suffix`, `extract_hidden_states`, `step3p5`) 是否已经在头部对 `num_speculative_tokens == 0` 做提前返回或空 Tensor 生成。  
   - 若尚未覆盖，添加对应 guard 并补充单元测试。  

2. **配置验证提前化**  
   - 在 `VllmConfig` 加载阶段即调用 `validate_and_normalize_dynamic_sd_schedule`，并在日志中打印最终 dense lookup 长度，帮助用户快速发现遗漏。  

3. **监控与指标**  
   - 利用新增 `SpecDecodingStats.num_draft_tokens_per_pos` 持续监控实际 K 与配置期望的差异，检测是否因调度误判导致 “K 被卡住”。  
   - 在 Prometheus / metrics 输出中加入 `dynamic_sd.active_k`。  

4. **文档与示例**  
   - 在官方 README 与部署指南明确标注：DSD 仅在 **Model Runner V1** 且 **不使用全图 CUDA‑graph** 时可用；推荐在高并发 / RL 场景下使用。  
   - 提供一个 “禁用 DSD 自动切换” 的 env/CLI 参数，以便在调优阶段快速回退。  

5. **性能回归基准**  
   - 在 CI 中加入带有不同 batch‑size

### [BugFix] Fix prompt_embeds for multimodal models (#45383)
**SHA**: `c621af1` | 🔗 [查看提交](https://github.com/vllm-project/vllm/commit/c621af16908f05270e033afd4237509902b7ba4d)

**🎯 变更类型**：Bug 修复 / 功能增强  
**⚡ 重要程度**：🟡 中  
**📋 变更摘要**：  
1. 在 `VLLMConfig` 初始化时增加检查，若模型开启 `prompt_embeds` 且为多模态模型，禁止使用异步调度，并在配置冲突时抛异常或给出一次性警告并自动关闭 async‑scheduling。  
2. `GPUModelRunner._preprocess` 中的嵌入生成逻辑改为区分 “token‑id” 与 “prompt‑embed” 两类位置：仅对 token‑id 位置重新嵌入，随后使用 `torch.where` 将新嵌入写回而不覆盖已有的 prompt‑embeds。  

**🎯 影响范围**  
- `vllm/config/vllm.py`：调度器初始化、异步调度开关。  
- `vllm/v1/worker/gpu_model_runner.py`：多模态模型的输入预处理（embedding）路径。  
- 受 `enable_prompt_embeds`、`is_multimodal_model` 标志控制的所有模型实例。

**💡 关注建议**  
1. **兼容性**：开启 `prompt_embeds` 的多模态模型现在只能走同步调度，务必在文档或示例中提醒用户；若已有部署使用 async，升级后会触发异常或警告。  
2. **功能验证**：新增单元/集成测试应覆盖以下场景：  
   - 多模态模型 + `enable_prompt_embeds=True` → 异步调度被禁用并记录警告。  
   - 多模态模型 + `enable_prompt_embeds=False` → 仍支持 async。  
   - 混合输入（部分 token‑ids、部分 prompt‑embeds）在 `GPUModelRunner` 中是否保持原始 prompt‑embeds 不被覆盖，且形状与 dtype 正确。  
3. **性能**：当前仍保留一次 `torch.where` + `copy_`，仍有额外的显存拷贝。后续可考虑在 `embed_input_ids` 时直接接受掩码，或在 `inputs_embeds` 上原位写入，减少拷贝开销。  
4. **错误路径**：`self.is_token_ids` 的布尔张量在 GPU 上的形状必须与 `self.input_ids` 对齐，否则 `torch.where` 会报错；建议在 `_preprocess` 开头添加断言或容错校验。  
5. **日志与监控**：`logger.warning_once` 已经避免重复噪声，建议在用户配置中提供显式的 `disable_async_for_prompt_embeds` 开关，以便在需要调度性能时自行决定。  

总体来看，此次改动修复了多模态模型在使用 prompt‑embeds 时的错误路径，并防止了不受支持的异步调度导致的潜在崩溃，影响范围局限于开启相应特性的模型，风险可通过上述测试与文档提醒进行控制。

### [ROCm][Perf] Enable W4A16 FlyDSL MoE (#44400)
**SHA**: `725c3bc` | 🔗 [查看提交](https://github.com/vllm-project/vllm/commit/725c3bc808c6eb5a572bdb37ed8a84bd11aad24a)

**🎯 变更类型**：功能增强（新增 AMD FlyDSL W4A16 MoE 后端）  
**⚡ 重要程度**：🟡 中  

**📋 变更摘要**  
- 在 `vllm.model_executor.layers.fused_moe` 中实现了全新的 FlyDSL MoE kernel（`fused_flydsl_moe.py`），并提供了缓存、自动调参、配置文件加载等完整流水线。  
- 新增压缩‑Tensor W4A16 FlyDSL 量化实现 `compressed_tensors_moe_w4a16_flydsl.py`，改写权重解压、分块、scale 处理逻辑，并在 `get_moe_method` 中根据平台、group‑size、LoRA 等条件自动选择。  
- 新增 ROCm‑only 单元测试 `test_flydsl_moe.py` 与调参脚本 `benchmark_flydsl_moe_w4a16.py`。  
- 将 FlyDSL 方案加入 `KernelConfig` 可选后端列表并提供默认/调优 JSON 配置文件。  

**🎯 影响范围**  
- **模型执行层**：`fused_moe/*`（新增 `fused_flydsl_moe.py`，修改 `routed_experts.py`）。  
- **量化子系统**：`quantization/compressed_tensors/compressed_tensors_moe/*`（新增 `compressed_tensors_moe_w4a16_flydsl.py`，在 `get_moe_method` 中挂载）。  
- **配置体系**：`vllm/config/kernel.py`（新增 `flydsl` 后端选项），以及若干 JSON 调优文件。  
- **测试与基准**：`tests/kernels/moe/test_flydsl_moe.py`、`benchmarks/kernels/benchmark_flydsl_moe_w4a16.py`。  

**💡 关注建议**  

1. **平台检测**  
   - 代码在 `KernelConfig` 中默认加入 `flydsl`，但实际实现仅在 ROCm (gfx950+) 可用。建议在 `KernelConfig` 文档或 CLI 中提示 “仅在 ROCm 上可选”，并在 `fused_flydsl_moe_impl` 加入显式平台检查，防止在 CUDA 环境意外加载导致未定义错误。  

2. **依赖完整性**  
   - 该后端依赖 `aiter`、`flydsl` 与 `compressed_tensors`。在 CI 中确保这三个库在 ROCm 镜像中预装，否则测试会被跳过而隐藏潜在加载失败。  

3. **权重转换路径**  
   - `process_weights_after_loading` 直接覆盖 `layer.w13_weight_packed` / `w2_weight_packed` 为 1‑D Tensor，可能破坏原有的 `is_transposed` 元信息。确认其他后端（如 Triton）在同一模型加载过程中不会重新访问旧形状。必要时在 `set_weight_attrs` 中重新写入 `is_transposed`、`quant_method`。  

4. **JSON 配置文件路径**  
   - `try_get_optimal_config` 采用相对路径 `configs/<json>`，在打包或 pip 安装后可能失效。建议使用 `importlib.resources` 或 `pkgutil.get_data` 读取，以保证在 wheel 包中也能找到。  

5. **性能回退**  
   - 当调优文件缺失时会使用 `_FLYDSL_MOE_DEFAULT_CONFIG`，但默认配置可能在部分显卡上显著下降。建议在日志中给出 “可通过 benchmark 脚本生成调优文件” 的提示，或在首次运行时自动触发一次小规模搜索。  

6. **测试覆盖**  
   - 现有单元测试只覆盖 `num_experts=384, hidden=7168` 的两种 `inter_dim`。如果后端计划支持不同专家数或不同 `top‑k`，请在 CI 中加入参数化组合，以防止维度不匹配的断言。  

7. **文档更新**  
   - 在官方 README/文档中加入 “FlyDSL MoE 后端” 使用说明、支持的硬件、必要的环境变量以及调优流程（run benchmark → 生成 json → 通过 `--kernel-config moe_backend=flydsl` 启用）。  

**总体结论**  
此次提交为 vLLM 引入了基于 FlyDSL 的 W4A16 MoE 加速路径，涉及核心执行、量化、配置与测试全链路改动。若在 ROCm 环境完整安装依赖，功能可正常跑通；在非‑ROCm 环境需确保后端被适当禁用或回退到已有实现。后续关注配置文件的加载可靠性和权重转换的兼容性即可。

### [Build] Fix CUDA arch build coverage gaps (#45277)

- **第三方依赖**：QUTLASS、CUTLASS MLA、MARLIN‑FP8 等的 arch 选择逻辑。  

**💡 关注建议**  
1. **CI/发布**：确认所有 CUDA 12.8‑13.0 / 12.9‑13.1 组合都能成功编译；尤其注意 `+PTX` 已不在全局列表，若有 kernel 需要 PTX，请在对应 `CMake` `arch` 列表手动加上。  
2. **用户构建**：使用旧版 CUDA（< 12.9）或不在 `CUDA_ARCH_X86(_AARCH64)_CU129` 列表的设备时，需要自行编译或切换到 CUDA 13 wheel，否则可能缺失 SM12x、SM100 等新架构。  
3. **MXFP4 Experts Quant**：仅在 CUDA ≥ 12.9、SM10x/11x（< 12.0）上可用；在运行时通过 `ops.mxfp4_experts_quant_supported(capability)` 判定，避免 `AttributeError`。  
4. **文档**：更新 Release Notes，说明 “+PTX 只能在 kernel‑level 指定” 以及 MXFP4 Quant 支持的 CUDA/SM 条件。  
5. **性能回归**：对新增的 SM12x、SM100、SM120 kernel（scaled mm、NVFP4、MLA、MARLIN FP8 等）跑一次基准测试，确保编译选项未引入额外开销。  

总体而言，此次提交提升了跨 CUDA 版本的构建兼容性并为 FP4/MXFP4 Experts Quant 提供了可选功能，需在 CI 与文档层面同步更新以防用户在旧环境或误用 +PTX 导致编译/运行错误。

### [Perf] SM90 cutlass fp8 mm supports odd M by swap_ab, 180~290% kernel performance improvement (#44572)
**SHA**: `71b961d` | 🔗 [查看提交](https://github.com/vllm-project/vllm/commit/71b961dd356a399150d25738c175c71859aa1301)

**变更概览**  
本次 PR 为 SM‑90（Hopper）上 Cutlass fp8 block‑wise GEMM 增加了对 **M 为奇数** 的支持。核心做法是在 `cutlass_3x_gemm_fp8_blockwise` 中加入模板布尔 `swap_ab_`，在编译期根据该标志切换 A/B 以及相应 Scale、Layout 与主循环的构造；运行时根据 `m % 4 != 0` 决定是否启用 swap‑ab，进而避免原先对 `m` 必须是 4 的倍数的硬性限制。对应的 Python 包装层 `cutlass_gemm_blockwise_sm90_fp8_dispatch` 也同步实现了该逻辑，并删除了原先的 “padded_cutlass / dynamic_padded_cutlass” 预填充实现。

**影响范围**  
- `csrc/libtorch_stable/quantization/w8a8/cutlass/.../scaled_mm_blockwise_sm90_fp8_dispatch.cuh`（核心 kernel 定义）  
- `vllm/model_executor/kernels/linear/scaled_mm/cutlass.py`（Python 调用层，去除动态 padding 逻辑）  
- 单元测试 `tests/kernels/quantization/test_cutlass_scaled_mm.py`（去掉对 `m%4!=0` 的早退）  

**关注要点 & 建议**  
1. **正确性**：`swap_ab` 会把 A↔B、Scale↔Scale、Layout↔Layout 以及 stride 重新排列，确保后续的 `CollectiveEpilogue` 仍使用转置后的布局。建议在 Hopper 之外的设备（SM<90）使用默认 `swap_ab=false`，防止误触发不支持的布局。  
2. **对齐与共享内存**：虽然转置后仍使用原来的 `AlignmentA/B/D`（128‑bit），但在 `M` 为奇数时可能出现未对齐的访问。务必在 CI 中加入奇数‑M、奇数‑K 的组合测试，验证不出现 UB。  
3. **构建兼容性**：新增模板默认参数 `swap_ab_ = false`，保持现有编译路径不受影响。但请确认所有显式实例化（如 `cutlass_gemm_blockwise_sm90_fp8_dispatch`）已同步传入该参数，否则可能导致编译错误或默认走旧路径。  
4. **Python 层**：删除 `direct_register_custom_op` 与预填充实现后，`CutlassFp8BlockScaledMMKernel.apply_block_scaled_mm` 直接走 `ops.cutlass_scaled_mm`。若仍有旧模型依赖 `dynamic_padded_cutlass`，需要在文档中说明已不再提供。  
5. **性能验证**：提交声称 180~290% 加速，建议在真实工作负载（不同 batch‑size、seq‑len）上跑基准，确认在奇数‑M 场景下的提升与预期一致。  
6. **回退路径**：在 `cutlass_gemm_blockwise_sm90_fp8_dispatch` 中已经写了 “TODO: better heuristics”，后续可根据实际 shape（如 `k%128`、`n%8`）完善调度，以免在某些尺寸下仍出现性能回退。  

总体来看，此改动显著放宽了 Hopper 上 fp8 GEMM 的尺寸限制，代码结构清晰且已去除冗余 padding 实现。只要通过上述对齐、兼容性及性能回归检查，即可安全合入。
