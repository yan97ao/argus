### [Rust Frontend] Align tool parser fallback behavior between streaming & non-streaming paths (#43662)
**SHA**: `396c8fe` | 🔗 [查看提交](https://github.com/vllm-project/vllm/commit/396c8fee5071117914104244cc962db805ab0af0)

**🎯 变更类型**：架构变更 / 重构 / 功能增强  
**⚡ 重要程度**：🔴 高  
**📋 变更摘要**：  
- 将原有的 `ToolParseResult`（一次 `push` 的完整返回值）统一改为 **增量输出** `ToolParserOutput`，并在 `ToolParser` trait 中引入 `parse_into(&mut self, chunk: &str, output: &mut ToolParserOutput) -> Result<()>` 接口。  
- 为每个具体的 tool‑parser（DeepSeek‑DSML、DeepSeek‑JSON、Gemma4、GLM‑XML、Kimi‑K2、MiniMax‑M2、Qwen‑Coder 等）迁移实现，所有 parser 现在在 **增量模式** 与 **一次性 `parse_complete`** 两条路径上共享同一套核心状态机。  
- 新增 `reset(&mut self) -> String` 用于在解析错误后恢复未消费的缓冲文本。  
- 引入 `easy-ext` 依赖，为 `ToolParser` 增添 `ToolParserTestExt` 便利测试辅助（`parse_chunk`、`parse_complete` 默认实现）。  
- 统一错误回退行为：在流式路径解析失败后仍会 **保留已解析的部分 output**，随后关闭当前 tool 调用并把缓冲文本作为普通文本返回；在非流式路径 `parse_complete` 中同样会得到一致的回退结果。

## 🔍 技术洞察

| 维度 | 影响 |
|------|------|
| **架构** | - 抽象层次提升：`ToolParser` 由“一次性返回 `ToolParseResult`”变为“**增量写入** `ToolParserOutput`”。<br>- 所有实现统一使用 `parse_into`，`push` 已被废弃。<br>- `ToolParserOutput` 同时承担 **普通文本** 与 **tool 调用增量**，促进 **流式** 与 **非流式** 两条路径共享同一状态机，避免代码重复。<br>- 引入 `ToolParserTestExt` 通过 `easy-ext` 提供默认 `parse_chunk` 与 `parse_complete` 实现，降低测试代码量。 |
| **性能** | - 逻辑上 **无额外拷贝**：`parse_into` 直接把增量写入已有 `ToolParserOutput`，而原实现每次 `push` 都返回新对象后再 `append`，产生一次 `Vec` 拼接和 `String` 复制。<br>- `reset` 直接 `std::mem::take`，避免 `clear` 产生的重新分配。<br>- 统一错误回退路径，避免在 `parse_complete` 中二次解析，从而略微提升整体解析吞吐。 |
| **内存** | - `ToolParserOutput` 持有的 `String` 与 `Vec<ToolCallDelta>` 与旧结构相同，**峰值内存保持不变**。<br>- `reset` 直接把内部缓冲抽走，交给调用者一次性输出，可能稍微降低短期占用。 |
| **安全** | - 统一错误处理：在解析错误后只要调用 `reset` 即可得到残余文本，防止错误时丢失已缓存内容。<br>- 新增 `reset` 方法返回被抽走的 `String`，确保 **未消费的缓冲区** 不被意外泄露或再次被误用。<br>- 通过 `ToolParserTestExt` 的默认实现，避免在业务代码中误用已经被标记为 **已废弃** 的 `push`/`finish`。 |
| **可维护性** | - **显著降低代码重复**：所有 parser 只需实现 `parse_into`、`finish`、`reset` 三个核心方法，原有的 `push`/`finish`/`reset` 各自复制的逻辑被统一抽象。<br>- 新增的 `ToolParserTestExt` 为单元测试提供统一入口，减少测试文件中的冗余包装代码。<br>- `Result`、防错宏 `parsing_failed!` 仍保持一致，迁移成本低。 |
| **兼容性** | - **破坏性更改**：`ToolParser` trait 的签名改变、`ToolParseResult` 被废除，所有外部使用 `ToolParser`（包括 vLLM Rust 前端、插件、用户自定义 parser）需要迁移至新接口。<br>- Cargo 添加 `easy-ext` 依赖，可能导致 **编译冲突**（尤其在 `no‑std` 环境或在已有 `easy-ext` 版本不同的项目中）。<br>- Rust 代码中仍保留 `ToolParser::push`、`finish` 的旧签名的 **默认实现**（已被删除），因此 **编译失败** 而不是运行时错误，迁移必须全量更新。 |
| **测试** | - 大量单元测试已同步到 `ToolParserOutput`，并使用新的 `collect_stream` 辅助。<br>- 通过 `expect_test` 验证 `Debug` 输出一致性，确保行为未因结构变更而产生差异。 |

### [KVConnector][Mooncake] Wire reset_cache cascade end-to-end (#42694)
**SHA**: `1fc2cee` | 🔗 [查看提交](https://github.com/vllm-project/vllm/commit/1fc2cee50a09a094b9f2bbdfcb0ab0cadb536712)

**🎯 变更类型**：功能增强 / Bug 修复 / 架构变更  
**⚡ 重要程度**：🔴高  

**📋 变更摘要**  
1. 为 Mooncake KV‑Connector 引入显式的 “reset_cache” 流程，新增 ZMQ 协议层（`protocol.py`）以及 Scheduler、Worker、Client 的实现，使得在前缀缓存硬重置时能够同步清除外部 KV 存储。  
2. 修复 Scheduler 在未配置 KV‑Connector 时 `reset_prefix_cache(reset_connector=True)` 返回 `False` 的 bug，改为返回 `True`（视为无操作成功），并相应地在 `engine.core._reset_caches` 中默认开启外部缓存清理。  
3. 补充大量单元测试，覆盖无 connector 情况、调度器/worker 角色的 reset 行为、协议标签唯一性、服务端 drain‑queue‑then‑remove‑all 逻辑等，提升可靠性。

---

### 🎯 影响范围
- **核心调度器**：`vllm/v1/core/sched/scheduler.py`、`vllm/v1/engine/core.py`  
- **KV‑Connector Mooncake Store**：`connector.py`、`scheduler.py`、`worker.py`、新建 `protocol.py`  
- **测试套件**：`tests/v1/core/test_scheduler.py`、`tests/v1/kv_connector/unit/test_mooncake_store_connector.py`  

---

### 🔍 技术洞察

| 维度 | 影响说明 |
|------|----------|
| **架构影响** | - **新增协议层** (`protocol.py`) 将请求类型抽象为显式的字节标签，统一了 Scheduler ↔ Worker 的 admin 通信，提升可扩展性。<br>- **Connector 责任分离**：`MooncakeStoreConnector.reset_cache()` 在 Scheduler 端清理本地状态（`load_specs`、`_kv_cache_events`），随后委托给 `MooncakeStoreScheduler.reset_store()` 通过 ZMQ 触发 Worker 的 `remove_all`。<br>- **统一错误语义**：返回 `True/False/None` 明确区分成功、失败、以及 Worker 角色的 no‑op。 |
| **性能影响** | - **额外一次 ZMQ RPC** 在硬重置时会增加微秒级延迟（网络 IPC），对整体吞吐影响极小，因为该路径只在权重更新后、生成暂停期间使用。<br>- **发送队列 drain** 确保在 `remove_all` 前完成所有未发出的 KV put，防止“脏数据”重新写入，避免后续查询产生不一致，整体不会产生显著性能回退。 |
| **安全考虑** | - **协议标签唯一且非空** 的单元测试确保不会出现冲突或误解析，降低因协议误判导致的非法操作风险。<br>- **异常捕获**：`reset_store` 与 Worker `RESET_MSG` 处理均捕获异常并返回明确的 NACK (`RESP_ERR`) 而不是抛异常，防止异常泄露到上层导致服务崩溃。 |
| **可维护性** | - 通过 `protocol.py` 集中管理所有 admin 消息常量，后续扩展（如增加 `FLUSH`, `STATS`）仅需在一处定义。<br>- 完备的单元测试（共计 > 300 行新增）覆盖各种角色、成功/失败路径，降低回归风险。 |
| **业务价值** | - 对使用 Mooncake KV 存储的 RL 或多模型推理场景，能够在权重更新后强制清空外部 KV，保证新权重下的 KV 不被旧权重的散列键污染，提升模型行为的一致性与可预测性。<br>- 使得 `reset_prefix_cache(..., reset_connector=True)` 成为可靠的“一键清理”操作，提升用户体验。 |

---

### ⚠️ 潜在风险
1. **同步假设失效**：`reset_store` 依赖调用方在调用前已经 **暂停生成**，若外部调用者忘记或误用（例如在高并发推理中直接调用 `reset_prefix_cache` 而未暂停），仍可能出现“在 reset 期间仍有 puts 进入队列”的竞态，导致残留 KV 重新写入。  
2. **ZMQ 队列阻塞**：`request_queue.join()` 会阻塞直至所有待发送的任务完成，若发送线程出现卡死（网络故障、内部异常），reset 将被无限阻塞。当前实现没有超时机制。  
3. **兼容性**：对已有用户，`reset_connector_cache` 现在在没有 connector 时返回 `True`（原来返回 `False`），可能影响依赖返回值判定的自定义逻辑。大多数场景应受益，但需在发行说明中明确。  
4. **跨进程资源泄露**：`LookupKeyServer.close()` 仍保持旧的 `socket.close(linger=0)` 行为，若 reset 时异常导致线程提前退出，可能留下未关闭的 ZMQ 句柄。

---

### 💡 关注建议
1. **文档和示例**：在 vLLM 文档中明确说明 **使用 `reset_prefix_cache(reset_connector=True)` 前必须先 `pause_generation`（或等价的生成暂停）**，并给出典型的 RL 权重更新流程示例。  
2. **超时与监控**：考虑在 `reset_store` 或 Worker `RESET_MSG` 处理里加入 **可配置的超时**（如 `join(timeout=5)`），并在日志中记录超时事件，以防止因卡死导致的系统不可用。  
3. **返回值兼容层**：若有外部插件依赖旧的 `False` 行为，可提供一个 **兼容选项**（如 `Scheduler.reset_connector_cache(allow_no_op_failure=False)`)**，保持向后兼容。  
4. **测试覆盖**：新增 **并发压力测试**，模拟高并发 `lookup` 与 `reset` 同时进行的情景，验证 `join` 的安全性与 `remove_all` 的幂等性。  
5. **安全审计**：虽然当前协议仅在内部进程间使用 IPC，但建议在生产环境开启 ZMQ 的 **身份验证**（CURVE）或使用 **Unix domain socket 权限**，防止外部进程伪造 `RESET_MSG` 造成意外数据清除。

### [DSv4] Refactor compressor & Fix ROCm compatibility (#43710)
**SHA**: `adaa5e4` | 🔗 [查看提交](https://github.com/vllm-project/vllm/commit/adaa5e455ad89525e8afa69b0633ec9c4b720f7e)

**🎯 变更类型**：功能增强 / 重构 / 性能优化 / 平台兼容性修复  
**⚡ 重要程度**：🔴高  
**📋 变更摘要**：

- 调整了模块导入层级、删除了不再使用的 cutedsl 包装函数，统一了 `compress_norm_rope_store_*` 接口，实现了跨平台（CUDA / ROCm）兼容。

  - **CUDA（head=512）**：仍保留 cutedsl 融合 kernel `compress_norm_rope_store_cutedsl`，在 C4A 场景下比两步 Triton 方案快约 10‑15%（依据 upstream 基准）。

  - **工作区开销**：额外分配一个固定大小的工作区 (`torch.Tensor`) 用于 fused kernel，内存占用与 `max_tokens * (q_size+kv_size)*dtype` 成正比，规模在 2048 token 以下，影响可接受。  

- **安全考虑**  
  - **自定义 CUDA Kernel**：新的 kernel `minimax_allreduce_rms_qk` 通过 `torch.ops._C` 暴露，已在项目的 C++/CUDA 代码中实现。只在检测到该符号时使用，若缺失则自动回退，无安全回退风险。  
  - **输入校验**：`forward_qkv` 中对维度和 epsilon 一致性做了断言，防止异常张量导致未定义行为。  
  - **兼容性**：通过 `logger.warning_once` 明示废弃标志，不会导致意外开启/关闭融合。  

**⚠️ 潜在风险**  
1. **Kernel 不可用**：在未编译或未加载对应的 `minimax_allreduce_rms_qk` 二进制时，`MiniMaxText01RMSNormTP` 会回退到 eager 实现，仍能工作但失去加速；需要确保 CI/发行版包含该 CUDA kernel。  
2. **工作区大小错误**：如果 `MAX_TOKEN_NUM` 常量在未来被修改，而工作区分配未同步，可能导致 kernel 越界。  
3. **TP 环境不一致**：在单卡（tp_world=1）或非 CUDA 环境下仍会创建 `MiniMaxText01RMSNormTP`，但不使用 fused kernel；若误将 `fuse_minimax_qk_norm` 标志留在配置中，用户会收到废弃警告，可能产生误解。  
4. **版本兼容**：老的 vLLM 代码仍可能引用已删除 `minimax_qk_norm_fusion` pass，升级时需确保所有插件或自定义 Pass 已同步删除。  

**💡 关注建议**  

- **发布前**：在 CI 中加入对 `torch.ops._C.minimax_allreduce_rms_qk` 可用性的检测测试，确保所有官方二进制发行版均包含该 kernel。  
- **文档**：在 README 与配置说明中明确 “MiniMax Q/K Norm 融合已自动在运行时开启，`fuse_minimax_qk_norm` 标志已废弃”。  
- **监控**：在生产部署中通过日志捕获 `logger.warning_once`（废弃标志）和 `MiniMaxQKNormPass disabled` 类信息，确保没有意外的回退。  
- **内存评估**：对使用大量 TP（如 8‑way）且 decode‑size 批次接近阈值的场景，测量工作区占用，必要时提供 `MINIMAX_QK_NORM_MAX_TOKEN_NUM` 可配置的环境变量。  
- **回滚路径**：若出现 CUDA kernel 兼容性问题，可通过显式禁用 `--compilation-config '{"mode":3,"pass_config":{"fuse_minimax_qk_norm":false}}'`（虽然已废弃）保持旧实现，避免业务中断。

### [Bugfix][V1] Fix TOCTOU race causing intermittent `EADDRINUSE` on multi-API-server DP startup (#42585)
**SHA**: `812e7e7` | 🔗 [查看提交](https://github.com/vllm-project/vllm/commit/812e7e73648573142808fdeea0ffc30adb247ba3)

**🎯 变更类型**：Bugfix（修复 TOCTOU 竞态导致的间歇性 `EADDRINUSE`）  
**⚡ 重要程度**：🟡 中  

**📋 变更摘要**  
- 通过在 API‑Server 子进程绑定 ZMQ `tcp://host:0`（让内核自动分配端口）后，用 `getsockopt(zmq.LAST_ENDPOINT)` 把真实地址回传给父进程，避免父进程在子进程实际 bind 前就尝试连接导致的端口冲突（TOCTOU）  
- 为此新增 `APIServerProcessManager.gather_actual_addresses`、`actual_address_pipe` 通道、以及在 `AsyncMPClient`/`EngineZmqAddresses` 中对占位地址的解析逻辑。  
- `serve.py` 在启动子进程后立即收集子进程实际绑定的输入/输出地址并写回 `EngineZmqAddresses`，确保后续引擎握手使用正确端口。  
- 对 `get_engine_client_zmq_addr`、`get_engine_zmq_addresses`、`launch_core_engines` 等网络工具函数做了细微重构，使占位端口（`:0`）在需要时才解析。  

**🎯 影响范围**  
- **vllm/v1/utils.py**、**vllm/v1/engine/utils.py**、**vllm/v1/engine/coordinator.py**：网络地址生成/绑定逻辑更改  
- **vllm/v1/engine/core_client.py**：新增向父进程回报实际绑定地址的代码  
- **vllm/entrypoints/cli/serve.py**：启动多 API‑Server 时开启 `defer_api_server_ports` 并收集实际地址  
- **tests/entrypoints/test_api_server_process_manager.py**：增加两条端到端测试，验证地址回报与子进程异常处理  

**💡 关注建议**  
1. **向后兼容**：Rust 前端在 `serve.py` 中通过 `defer_api_server_ports=False` 保持原有行为，确认文档已说明该选项的意义。  
2. **资源清理**：`APIServerProcessManager.shutdown` 现在会关闭所有管道，确保进程退出时不残留文件描述符。  
3. **异常路径**：`gather_actual_addresses` 会在子进程提前退出或管道 EOF 时抛 `RuntimeError`，调用方（如 `serve.py`）需捕获并做适当清理，防止进程泄漏。  
4. **测试覆盖**：新加入的测试覆盖了正常回报和子进程崩溃两种情况，建议在 CI 中保持 `timeout` 足够，以免因宿主机器慢导致误报超时。  
5. **性能影响**：新增管道通信和一次 `getsockopt` 调用，开销极小，仅在进程启动阶段产生，几乎不影响运行时吞吐。  

总体来看，此次改动成功消除了多 API‑Server 启动时的端口争用问题，提升了启动可靠性，对现有功能影响有限且向后兼容性已得到保障。请在后续版本更新日志中明确 “多 API‑Server 启动的 TOCTOU 修复” 以及 `--defer-api-server-ports`（内部标记） 的说明。
