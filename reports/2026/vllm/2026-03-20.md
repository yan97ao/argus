### fix: disambiguate multimodal prefix cache keys (#36708)
**SHA**: `269bf46` | 🔗 [查看提交](https://github.com/vllm-project/vllm/commit/269bf46d99f1df74e4d779f9c52c74002e057a17)

**🎯 变更类型**：Bug修复  
**⚡ 重要程度**：🔴高  
**📋 变更摘要**：在多模态（MM）前缀缓存的哈希键生成逻辑中，加入了 **相对偏移量** 信息，使得相同的 MM 标识符在不同块位置能够产生不同的缓存键。相应地，测试用例已更新以验证新键的结构和行为。  

**🎯 影响范围**：  
- `vllm/v1/core/kv_cache_utils.py`（核心哈希键生成函数）  
- `tests/v1/core/*`（KV 缓存和前缀缓存相关单元测试）  
- 任何依赖 **KV 缓存前缀键** 进行缓存命中/失效判断的模块（如调度器、块分配器）。

---

**🔍 技术洞察**  

- **架构影响**  
  - **键唯一性提升**：原实现仅使用 MM 标识符 (`identifier`) 作为额外哈希键，导致不同位置出现相同标识符时产生冲突，进而错误地命中前缀缓存。新增的 `(identifier, relative_offset)` 元组在结构上保持向后兼容（仍是 hashable），但在语义上确保了 **同一标识符在不同块起始位置的前缀缓存键是唯一的**。  
  - **模块耦合**：该改动仅限于哈希键生成函数 `_gen_mm_extra_hash_keys` 与 `generate_block_hash_extra_keys`，不影响其它系统组件的调用约定。若有外部插件直接依赖 `extra_keys` 为纯字符串，需要相应适配。  

- **性能影响**  
  - **计算开销**：在每次块哈希生成时多做一次整数减法（`offset - start_token_idx`）并生成一个二元元组，开销几乎可以忽略不计。  
  - **缓存命中率**：通过避免误命中，实际运行时会 **降低错误的缓存命中**，进而可能提升推理正确性和整体吞吐（因为错误命中会导致需要回滚或重新计算）。  
  - **内存占用**：每个额外键多占用一个整数对象，整体内存增长极小（相对于 KV 缓存本身的规模）。  

- **安全考虑**  
  - 此次改动不涉及网络、权限或输入验证，安全风险基本为 **零**。唯一需要注意的是：哈希键的结构变化可能导致缓存持久化格式不兼容（如果用户自行持久化 KV 缓存），但在常规使用场景下并无安全隐患。  

**⚠️ 潜在风险**  

1. **向后兼容性**  
   - 任何直接检查 `extra_keys` 内容并假设是纯字符串的第三方代码（例如自定义缓存插件、实验性调度器）会在运行时抛出类型错误。  
2. **序列化/持久化**  
   - 若项目在未来实现 KV 缓存持久化，需更新序列化方案以支持新的二元元组结构。  
3. **测试覆盖不足**  
   - 目前仅覆盖了常见的单块、多块、跨块等场景，极端情况下（如多个 MM 项重叠、位置跨度跨越多个块）仍需手动验证。  

**💡 关注建议**  

- **代码审查**：确认所有使用 `extra_keys` 的位置（包括未在本次提交中出现的内部工具）均已适配新元组结构。  
- **文档更新**：在 `kv_cache_utils` 模块的函数说明中标注 `extra_keys` 现在返回 `Tuple[Tuple[str, int], ...]`（标识符 + 相对偏移）。  
- **回归测试**：增加针对 **重叠 MM 输入**、**跨块多 MM 输入** 的单元测试，确保偏移计算在所有边界情况均正确。  
- **兼容层**：如果计划对外保持旧接口，可以提供一个包装函数 `normalize_extra_keys`，在内部统一转换为新结构，外部仍可接受旧的纯字符串列表。  
- **性能监控**：在生产环境开启前缀缓存命中率统计，验证误命中率是否真的下降，若有异常回滚到旧实现。  

--- 

此修复通过在前缀缓存键中加入相对位置信息，根本解决了多模态前缀缓存键冲突的问题，提升系统的 **正确性** 与 **可预测性**，对整体架构影响正面且风险可控。请在合并前完成上述兼容性检查与补充测试。

### [ROCm] Enable wvSplitK skinny GEMM kernel for RDNA4/gfx1x decode (#34709)
**SHA**: `1779c09` | 🔗 [查看提交](https://github.com/vllm-project/vllm/commit/1779c09898e091167b5d29cb8931ce1f5dea9a47)

**🎯 变更类型**：功能增强（为 RDNA4 / gfx1x GPU 添加 skinny GEMM 支持）  
**⚡ 重要程度**：🟡 中（涉及底层 Hip‑CUDA kernel 与平台检测，若出现编译/运行时错误会影响 ROCm 上的 GPU 推理）  

**📋 变更摘要**  
1. **平台检测**  
   - 新增宏 `__HIP__GFX1X__`、`__HIP__GFX12__`，并在 `csrc/rocm/skinny_gemms.cu` 中实现运行时函数 `on_gfx1x()`、`on_gfx12()`，用于在代码路径上区分 gfx11/12 与旧架构。  
2. **指令重构**  
   - `DOT2C` 改为在 gfx9 使用 `v_dot2c_f32_f16`，在 gfx1x 使用 `v_dot2_f32_f16`（VOP3‑P dot10）。  
   - 对行级归约从 AMD DPP（0x142/0x143）切换为 `__shfl_xor`，并在 gfx1x / gfx12 上统一为 `THRDS‑1` 线程负责写回。  
3. **线程块尺寸 & wave‑size**  
   - 传统 64 threads 固定改为可配置的 `THRDS`（64 → 32 当 `on_gfx1x()` 为真），并相应调整 `WvPrGrp*THRDS` 的 launch‑bounds。  
   - 对 FP8 路径（MI300X / gfx12）加入 `__HIP__GFX12__` 分支，使用 `v_dot4_f32_fp8_fp8` 并在波宽为 32 时手动实现行‑shrink + cross‑row shuffle。  
4. **上层调用**  
   - `rocm_unquantized_gemm_impl` 现在在 `on_gfx9()` **或** `on_gfx1x()` 时开启 skinny GEMM。  
5. **测试**  
   - 新增 `test_rocm_unquantized_gemm.py` 检验在 gfx1x、gfx950、以及 N > 4 的回退情况。  
   - 调整 `test_rocm_wvsplitk_kernel` 的容差公式（`atol = eps*sqrt(K)）`，更符合 fp16 累加误差的理论上界。  
   - 为 `wvSplitKrc` 增加 bias‑mode 3（`(1, M)`）的覆盖。  

**🎯 影响范围**  
- **核心算子**：`csrc/rocm/skinny_gemms.cu`（所有 `wvSplitK*`、`wvSplitKrc*`、FP8 kernel）  
- **平台检测**：`vllm/platforms/rocm.py`（新增 `on_gfx1x`、`on_gfx12`）  
- **高层接口**：`vllm/model_executor/layers/utils.py`（skinny GEMM 开关）  
- **测试套件**：`tests/kernels/quantization/test_rocm_skinny_gemms.py`、`tests/model_executor/layers/test_rocm_unquantized_gemm.py`

**💡 关注建议**  

1. **编译配置**  
   - 确认 ROCm 编译命令行已显式传入 `-march=gfx1100`/`gfx1200`（或通过 `HIPCC` 的 `--amdgpu-target=`），否则宏 `__gfx1100__`/`__gfx1200__` 不会被定义，导致 `__HIP__GFX1X__`/`__HIP__GFX12__` 未生效，代码会走 gfx9 路径甚至产生不兼容指令。  
2. **运行时检测**  
   - `on_gfx1x()` 与 `on_gfx12()` 使用 `at::cuda::getCurrentDeviceProperties()`，在多‑GPU 环境下请确保在每个线程/进程中调用前已切换到目标设备，否则可能缓存错误的架构信息。  
3. **写回线程**  
   - 更改为 `threadIdx.x == (THRDS‑1)` 对 gfx9 仍等价（63），但对未来如果调整 `THRDS` 为非 64 的值（如 32），请确认其他依赖 “第 63 线程” 的 host‑side调度或统计代码同步更新。  
4. **FP8 路径**  
   - `__HIP__GFX12__` 条件仅在 `__HIP__MI3XX__`（gfx942/950）或显式目标为 gfx12 时编译，若新增其他 MI3XX 变体需同步更新宏列表。  
5. **测试覆盖**  
   - 新增的 bias‑mode 3 仅在 `wvSplitKrc` 使用；若以后在 `wvSplitK` 也支持同类广播，需要同步相应的单元测试。  
6. **回退逻辑**  
   - `rocm_unquantized_gemm_impl` 仍在 `use_skinny` 为 False 时走标准 `torch.nn.functional.linear`，请在非‑skinny 路径加入 `torch.cuda.synchronize()`（或在调用者侧）以避免潜在的异步错误。

**总体建议**：本次改动对核心功能几乎没有影响，只是清理了已不再维护的实验选项。确保升级文档同步更新，并在 CI 中加入一次对 `--disable-frontend-multiprocessing` 的负向测试（确认报错），以防未来误回引入。这样既保持了代码整洁，又给使用者留足够的迁移提示。

### 2️⃣ 受影响的主要模块  
- **DeepSeekV32ToolParser**：核心解析器实现变更。  
- **工具调用相关的流式输出**：依赖 `DeltaMessage`、`DeltaToolCall` 的上层服务（如 API 服务器、OpenAI 兼容层）会收到不同的增量结构。  
- **测试套件**：新增大量测试文件，导致 CI 运行时间增长。

### 3️⃣ 潜在风险 & 注意点  
1. **向后兼容性**  
   - 旧代码可能仍在使用已删除的属性 (`self.dsml_token`、`self.invoke_start_prefix` 等)。如果外部插件或自定义派生类直接访问这些私有属性，将触发 `AttributeError`。建议在文档或迁移指南中说明已移除的 API。  
2. **流式分块边界**  
   - 新实现在 `_extract_delta_tool_calls` 中一次性解析 **完整的 `<invoke>`** 块，依赖正则 `self.invoke_complete_regex` 能在当前累计文本中捕获完整块。若 `stream_interval` 极大导致 `<invoke>` 的开始与结束跨多次回调且中间出现其它非工具内容（如普通文本） ，当前实现仍能正常工作，但需确保 `current_text` 总是累计全部已经收到的字符。  
3. **工具 schema 类型转换**  
   - `_convert_params_with_schema` 只在工具列表（`request.tools`）中查找匹配的函数名并读取 `properties`。如果工具使用 `anyOf`、`oneOf` 或嵌套对象，类型转换可能不完整。可在后续迭代中加入对更复杂 JSON Schema 的支持。  
4. **并发或复用实例**  
   - 解析器实例在一次请求结束后会在下次请求的第一帧 (`previous_text == ""`) 调用 `_reset_streaming_state`。若同一实例被并发多个请求共享（例如在全局单例中），状态会互相污染。建议在服务器层确保每个请求独享解析器实例，或在 `extract_tool_calls_streaming` 开头加入请求标识检查。  
5. **测试运行成本**  
   - 476 行新增测试在 CI 中会显著增加执行时长。可考虑将部分极端流式块划分为参数化测试或使用 `pytest.mark.timeout` 防止因意外死循环卡住。  

### 4️⃣ 建议的后续行动  
- **文档更新**：在 `tool_parsers` 章节补充 “DeepSeekV32ToolParser 迁移指南”，标明已删除的 DSML 常量以及新的流式使用方式。  
- **向后兼容包装**：如果预计外部仍会引用旧属性，可提供兼容属性（例如 `self.dsml_token = "｜DSML｜"`）并在内部标记为已废弃。  
- **专项压测**：在真实流式请求（不同 `stream_interval`，并发）下跑一次基准，确认增量返回的频率与体积符合预期，防止因过度拆分导致大量小 `DeltaMessage` 增加网络开销。  
- **增加异常保护**：在 `_extract_delta_tool_calls` 中对 `json.dumps`、`_convert_param_value` 加上 `try/except`，防止单个参数错误导致整条流被中断。  
- **CI 优化**：将部分冗长的字符流测试拆分为 `pytest.mark.parametrize`，或使用更小的 `chunk_size` 随机抽样，以维持测试覆盖度的同时控制执行时间。  

总体来看，此次改动通过 **简化流式状态管理、统一参数类型转换** 解决了 `stream_interval > 1` 时工具调用解析错误的问题，并配套了完整的单元测试，提升了可靠性。只要在上述兼容性与并发使用场景上再做一次验证，即可安全合并。

总体来看，此次改动简化了多模态子模块的编译流程，降低了遗漏 `set_model_tag` 导致缓存冲突的风险。但因为涉及缓存路径和编译范围的底层细节，务必通过上述检查确保新机制在所有支持的模型上都能稳定工作。
