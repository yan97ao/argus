### [Bugfix] Fix unstreamed tool call args dropped in Responses API streaming (#44348)
**SHA**: `209709a` | 🔗 [查看提交](https://github.com/vllm-project/vllm/commit/209709a8c131f15ed5c1ea231f4365f61a05d009)

**🎯 变更类型**：Bug 修复  
**⚡ 重要程度**：🟡 中  

**📋 变更摘要**  
本次提交修复了 **Responses API** 在流式返回时，工具调用参数被意外丢失的问题。核心做法是为 `Parser.parse_delta` 增加 *finished* 标记，让解析器在流结束时能够正确把未完成的工具调用参数 flush 出来，同时在 `serving.py` 中传入该标记并相应更新测试用例。

**🎯 影响范围**  
- `vllm/parser/abstract_parser.py`（`parse_delta` 接口签名及内部实现）  
- `vllm/entrypoints/openai/responses/serving.py`（流式输出构造）  
- `tests/parser/test_streaming.py`（新增 `finished=False` 参数）  

**💡 关注建议**  
1. **兼容性**：`parse_delta` 现在强制要求 `finished` 参数且为关键字传递，若项目中还有未更新的调用点会导致运行时错误。建议在函数签名上保留默认值（`finished: bool = False`），或在迁移期间提供包装层。  
2. **业务正确性**：确认 `finished=True` 时的分支能够完整输出工具调用的参数，尤其是多轮 tool‑call 场景。可以新增单元测试，验证 `output.finish_reason` 为 `null` 与非 `null` 的两条路径。  
3. **文档与类型**：在函数 docstring 中说明 `finished` 的意义和使用时机，并在 `serving.py` 添加注释解释 `output.finish_reason` 与流结束的关联。  
4. **回归检测**：运行全仓库的测试套件，特别是 `ChatCompletion` 相关的流式用例，确保新参数未对其他流式路径产生副作用。  

总体而言，此改动直接解决了工具调用参数丢失的关键缺陷，影响范围局限于解析器和响应层，若做好兼容处理和充分回归测试，可安全上线。

### [Bugfix] Update TrtLLM MoE routing methods (#44347)
**SHA**: `ace95c9` | 🔗 [查看提交](https://github.com/vllm-project/vllm/commit/ace95c9cf8307301fbcdffacc4ab44e4d5e6829a)

**变更类型**：Bug 修复 / 功能增强  
**重要程度**：🟡 中  
**变更摘要**：在 `fused_moe` 路由配置中加入了 **Sigmoid** 路由类型并引入 `routed_scaling_factor` 参数，以兼容 TrtLLM FP8/NVFP4 MOE 对 `sigmoid`‑>`TopK`（无归一化）和 MiniMax2（可配置 scale）的需求；同时放宽了对路由 bias 的 dtype 要求，去掉了强制转为 `bfloat16` 的限制。

**影响范围**  
- `vllm/model_executor/layers/fused_moe/config.py`：枚举扩展、`get_routing_method_type` 参数签名与判定逻辑改动。  
- 路由实现 (`fused_topk_bias_router.py`, `grouped_topk_router.py`, `zero_expert_router.py`)：调用时传递 `routed_scaling_factor`。  
- FP8/NVFP4 MOE 实现 (`trtllm_fp8_moe.py`, `trtllm_nvfp4_moe.py`)：支持 `RoutingMethodType.Sigmoid`，路由 bias dtypes 由 “仅 bfloat16” 改为 “bfloat16 或 float32”。  

**关注建议**  

1. **向前兼容**：`get_routing_method_type` 新增可选参数，所有内部调用需显式提供默认值 `1.0`，防止因位置参数错位导致错误。  
2. **路由判定**：`has_e_score_bias` 与 `scoring_func=="sigmoid"` 的组合现在可能返回 `Unspecified`（当 `renormalize=False` 或 `routed_scaling_factor!=1.0`）。请确认调用方在这些分支下的行为已经覆盖（如报错或回退）。  
3. **dtype 检查**：`_supports_router_logits_dtype` 现在接受 `torch.float32` 与 `torch.bfloat16`，但实际 kernel 仍可能要求 `bfloat16`（FlashInfer 议题 #2909）。建议在文档或日志中明确说明在 `float32` 情况下的性能或精度影响。  
4. **单元测试**：新增 `Sigmoid` 与 `MiniMax2`（含非默认 scale）两条路由路径的覆盖，验证在 FP8/NVFP4 条件下的正确性与数值稳定性。  
5. **性能回归**：`Sigmoid` 路由去除归一化后可能导致 `router_logits` 溢出，建议在关键路径加入阈值检查或用户可选的 `clip` 参数。  

综上，本次改动提升了 TrtLLM 融合 MoE 的路由灵活性，但需在调用方、文档和测试层面确保新枚举与参数的正确使用，防止因未传递 `routed_scaling_factor` 或误判 route 类型导致运行时错误或数值异常。

### [Rust Frontend] Fix several hf chat template rendering issues (#44311)
**SHA**: `449be4f` | 🔗 [查看提交](https://github.com/vllm-project/vllm/commit/449be4f93484d2b8b9ae06bb2f86c6e42c6760d7)

**🎯 变更类型**：功能增强（修复 HF chat template 渲染缺陷）  
**⚡ 重要程度**：🟡 中  
**📋 变更摘要**：在 Rust 前端加入 `indexmap`，去除 `serde_json` 的 `arbitrary_precision` 特性，实现 `TemplateValue` 包装，将 JSON → MiniJinja 值的转换与字典方法冲突分离，并对 `tojson` 过滤器的数字表现进行统一。新增针对工具调用参数、`items()` 方法以及数值保真度的单元测试，同时更新 InternLM2 工具解析器的文档说明。  

**🎯 影响范围**：  
- `rust/src/chat`（模板渲染、`hf` 渲染器、`tojson`、新 `value.rs`）  
- `rust/src/tool-parser`（InternLM2 说明、参数处理）  
- `Cargo.toml` 与工作区依赖（新增 `indexmap`，修改 `serde_json` 特性）  

**💡 关注建议**：  
1. 由于关闭 `serde_json` 的 `arbitrary_precision`，渲染后数值将统一为标准的 `1.0` 表示，若业务依赖精确的字符串形式，需要在上层自行保留或使用自定义过滤器。  
2. `TemplateValue` 通过 `IndexMap` 保持键顺序并始终返回 `UnknownMethod`，可避免 HF 模板中 `dict.items()` 与同名字段冲突，但注意此实现可能影响已有模板对字段方法的自定义实现。
