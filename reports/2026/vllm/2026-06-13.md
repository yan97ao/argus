### [Render] Add `/derender` endpoints for disaggregated postprocessing (#43606)
**SHA**: `43f0e02` | 🔗 [查看提交](https://github.com/vllm-project/vllm/commit/43f0e024bcc304e88b5be47555daabf795582354)

**🎯 变更类型**：功能增强、架构变更、API 扩展、测试覆盖  

**⚡ 重要程度**：🔴高  

**📋 变更摘要**：  
- 新增 `/v1/chat/completions/derender` 与 `/v1/completions/derender` 两个 **后处理（Derender）** 接口，用于在不依赖 GPU 的情况下将 *render* 阶段得到的 `GenerateResponse`（或 `GenerateResponses`）转化为符合 OpenAI Chat/Completion 响应格式的完整结果。  
- 引入 `DerenderChatRequest`、`DerenderCompletionRequest` 数据模型以及对应的请求/响应序列化逻辑。  
- 抽象出 `format_token_id_placeholder` 与 `resolve_token_id_placeholder` 用于在 **logprob** 中对 `token_id:N` 占位符进行统一编码/解码，避免在多个模块中散布硬编码。  
- 为两套 Derender 接口实现了 **detokenize、logprob 解析、usage 统计、kv_transfer 参数传递** 等完整流程。  
- 新增大量单元测试覆盖正向、异常、边界等场景，提升代码可靠性。

**🎯 影响范围**：  
- `vllm.entrypoints.serve.render`（serving、api_router）  
- `vllm.entrypoints.openai.*`（chat_completion、completion、engine）  
- `vllm.entrypoints.serve.disagg.protocol`（Derender 请求模型）  
- `vllm.tokenizers`（占位符解析）  
- 相关的测试目录 `tests/entrypoints/serve/render`  

**🔍 技术洞察**  

- **架构影响**  
  - **服务层解耦**：原本 `render` 与 `postprocess` 紧耦合在同一个请求路径（`/render` → 立即返回完整响应），现在拆分为 **两步**：① `render` 只返回 token ID 列表 + 元信息；② `derender` 完成解码与结构化。这使得 **离线推理、分布式流水线、GPU‑CPU 分离** 场景更加友好。  
  - **协议扩展**：在 `disagg.protocol` 中加入 Derender 请求模型，保持了 **前后端协议的一致性**，并通过 Pydantic `model_validator` 进行跨字段校验，提升数据有效性。  
  - **公共占位符函数**：`format_token_id_placeholder` 与 `resolve_token_id_placeholder` 把占位符逻辑抽象为单点，实现 **代码复用**，并为将来可能的占位符格式变更提供单一修改入口。  

- **性能影响**  
  - **CPU 端解码成本**：`derender` 需要在 CPU 上完成 `tokenizer.decode` 与 `convert_ids_to_tokens`，对单次请求的时延会有轻微提升（典型几毫秒），但相较于 GPU 推理成本，这一开销可以接受。  
  - **批量处理**：`derender_completion_response` 在多 Prompt 场景下采用循环逐条解码，未做并行优化。若一次性处理大量响应（>100），可能出现 CPU 瓶颈，可在未来通过异步批量解码或向 tokenizer 添加批量接口进行优化。  
  - **内存占用**：新增的 `logprobs` 解析会生成临时字符串及字节列表，峰值内存随 `logprobs` 大小线性增长，仍在可接受范围（几 MB 级）。  

- **安全考虑**  
  - **输入校验**：Derender 端对 `model`、`token_ids`、`prompt_tokens` 长度等进行显式校验，返回 400/404 错误，防止非法请求导致服务异常。  
  - **占位符解析安全性**：`resolve_token_id_placeholder` 在解析 `token_id` 时捕获 `ValueError`，并在词表缺失时返回空字符串，避免抛异常泄漏内部实现细节。  
  - **日志信息**：仅记录请求 ID、模型名、choice 数、token 数量，不泄露原始用户提示内容，符合最小化日志原则。  

**⚠️ 潜在风险**  

1. **CPU 负载激增**：大规模并发的 Derender 请求（尤其是带大量 `logprobs`）会在 CPU 上产生高解码负载，可能触发 CPU 饱和，导致响应延迟增加。  
2. **占位符冲突**：如果未来引入其他占位符前缀，`resolve_token_id_placeholder` 仍会只处理 `token_id:`，可能导致未被识别的占位符直接返回原始字符串，出现业务异常。  
3. **KV 参数不一致**：`derender_completion_response` 对 `kv_transfer_params` 进行统一检查并在不一致时置为 `None`，但没有向用户明确返回警告，可能让调用方误以为所有响应共享同一 KV 参数。  
4. **向后兼容性**：新增的路由仅在模型实现 `render` 能力时可用，若已有部署未实现 `render`（如仅支持同步推理），调用 Derender 会抛 `NotImplementedError`，需在文档中明确兼容范围。  

**💡 关注建议**  

- **监控与限流**：在生产环境对 `/derender` 路由加入 **CPU 使用率监控** 与 **请求速率限流**，防止突发流量导致节点过载。  
- **批量解码优化**：考虑在 `TokenizerLike` 接口实现 `decode_batch`，或在 `derender_*` 中使用线程池/进程池对大批量 `token_ids` 进行并行解码，以降低单线程 CPU 峰值。  
- **错误信息统一**：为 `kv_transfer_params` 不一致的情况返回明确的 **warning 字段** 或在 `UsageInfo` 中加入 `warnings` 列表，让上层调用方能够感知潜在信息不一致。  
- **文档补全**：在 OpenAI 接口兼容层的 README 中加入 Derender 接口的使用场景、限制（如必须先通过 `/render` 获取 token IDs）以及 **兼容模型列表**。  
- **安全审计**：对 `resolve_token_id_placeholder` 添加 **输入长度限制**（防止极端长 token_id 导致内存占用）以及对 `tokenizer.convert_ids_to_tokens` 的异常捕获，确保不会因词表异常导致服务崩溃。  

总体来看，此次改动为 vLLM 增加了 **“渲染‑后处理”** 的完整工作流，提升了可扩展性与灵活性，风险主要集中在 CPU 端的解码成本与占位符处理的边界情况，建议通过监控、文档和适度的性能优化来降低风险。

### [Kernel] Consolidate Marlin thread-tile padding across all dense Marlin paths (#45295)
**SHA**: `9eaacb2` | 🔗 [查看提交](https://github.com/vllm-project/vllm/commit/9eaacb23ec1826ddac31657e0eab699de6de3c59)

**🎯 变更类型**：功能增强 / 重构 / 性能优化  
**⚡ 重要程度**：🔴高  

**📋 变更摘要**  
1. 为 Marlin 量化路径统一了 **线程‑Tile 填充（tile‑padding）** 逻辑，新增 `marlin_padded_nk`、`marlin_pad_*`、`marlin_unpad_output` 等工具函数。  
2. `LinearBase.can_implement` 现在在 **allow_tile_padding=True** 时接受不满足 64×128（或 128×64）对齐的权重形状，只要量化组大小不跨越 TP 分片即可。  
3. 权重加载、scale、bias 等在加载后统一执行 **零填充**，并在前向结束时 **去除填充**，确保计算结果与原始模型一致。  
4. 对 **GPTQ、AWQ、FP4、FP8、MXFP8** 等所有密集 Marlin 量化实现做了相应的适配修改。  
5. 新增 470 行测试，覆盖所有量化路径的填充‑去填充 round‑trip 与数值精度验证。  

**🎯 影响范围**  
- `vllm/model_executor/kernels/linear/mixed_precision/marlin.py`（Marlin kernel 调度层）  
- `vllm/model_executor/layers/quantization/utils/marlin_utils*.py`（核心填充/对齐/检查逻辑）  
- 各量化实现：`awq_marlin.py`, `modelopt.py`, `marlin_utils_fp4.py`, `marlin_utils_fp8.py`  
- 相关单元测试 `tests/kernels/quantization/test_marlin_tile_padding.py`  
- 对项目中所有使用 Marlin 进行 **密集量化（GPTQ/FP4/FP8/MXFP8/AWQ）** 的层产生影响。  

---

## 🔍 技术洞察  

| 维度 | 影响描述 |
|------|----------|
| **架构影响** | - 引入统一的填充抽象，使得 **TP‑Shard**（模型并行）下的非对齐权重能够自动适配 Marlin。<br>- `check_marlin_supports_layer` 新增 `allow_tile_padding` 参数，层级检查从“严格对齐”放宽到“可填充”。<br>- 对 **act‑order（g_idx）** 的层仍保持严格检查，因为填充会破坏分组顺序。<br>- 现有的 `MarlinLinearKernel` 调用路径未改变，仅在 **权重准备阶段**（`process_weights_after_loading`、`prepare_*_layer_for_marlin`）插入填充步骤。 |
| **性能影响** | - **正面**：模型可以在更多硬件/分片配置下运行，不会因形状不匹配直接回退到慢速非‑Marlin 实现。<br>- **负面**：填充会导致 **额外的内存占用**（最多 `(padded_n·padded_k - n·k)` 元素）以及 **一次额外的拷贝**（`torch.nn.functional.pad`），在极大模型（≥ 120B）上会产生数百 MB‑GB 的临时开销。<br>- 对于 **已对齐** 的层，填充路径几乎为零开销（快速路径）。<br>- `marlin_unpad_output` 在前向结束后做一次 slice拷贝，增加微小的显存/时间成本。<br>- 由于填充会导致 **K** 与 **N** 维度增大，`should_use_atomic_add_reduce` 的阈值判断也会使用更大的 `k`/`n`，可能触发 **atomic‑add** 方案，而非更高效的 **reduction‑in‑register**，在极端情况下略微降低吞吐。 |
| **安全考虑** | - 仅在内部张量之间进行零填充、切片，没有外部输入解析或文件 I/O，未引入新攻击面。<br>- 通过 `torch.nn.functional.pad` 与 `torch.nn.functional.unpad`（切片）实现，保持 **torch.autograd** 正确性，防止梯度泄漏。<br>- 代码在 `check_marlin_supports_layer` 中对 `group_size` 跨 TP 边界的情况仍保持拒绝，防止 **错误的零点/scale** 逻辑导致数值溢出或未定义行为。 |
| **可维护性** | - 新增的 **公共填充/去填充函数**（`marlin_pad_*`、`marlin_unpad_output`）在多个文件中统一使用，降低了重复实现的风险。<br>- `marlin_padded_nk` 的实现使用 `round_up` 与 `math.lcm`，逻辑清晰且易于单元测试。<br>- 通过 `allow_tile_padding` 参数，未来若要在稀疏/MoE 场景中加入填充，只需在对应检查处传递该标记即可，扩展性好。<br>- 增加的测试覆盖率高（每条量化路径都在不同形状、组大小、bias 情况下验证），有助于防止回归。 |

---

## ⚠️ 潜在风险  

1. **显存激增**  
   - 对于大模型（比如 Nemotron‑Super‑120B），`padded_n * padded_k` 可能比原始矩阵大 10%‑30%，导致 **GPU 显存占用显著上升**，在显存紧张的部署场景下可能触发 OOM。  
2. **性能回退**  
   - 填充后 `size_k`/`size_n` 增大，若增幅导致进入 **atomic‑add** 路径，实际算子吞吐可能下降 5%‑15%。  
3. **稀疏/MoE 兼容性**  
   - 当前只在 **dense** 路径实现填充，MoE 专用的 `prepare_nvfp4_moe_layer_for_marlin` 只对 `w13`/`w2` 进行手动 Pad，仍有 **未覆盖** 的稀疏/专家切分场景。若用户自行在 MoE 上使用 `allow_tile_padding=True`，会触发 **未实现的错误**。  
4. **组大小跨 TP 边界的误判**  
   - `can_implement` 中对 “group straddling TP shards” 的判断仅基于 `partition_weight_shape[0] % group_size != 0`，但在实际 `group_size` 为 -1（无分组）时仍返回 `True`。若未来出现 **自定义 group 抽象**（例如多维分组），可能产生错误的填充。  
5. **量化误差扩大**  
   - 对 **INT‑4/INT‑8**（GPTQ、AWQ）路径，填充部分的 **scale/zero‑point** 被强制为 0，理论上不影响结果，但在极端数值范围或极端低‑precision 场景下，可能出现 **数值漂移**（尤其是 `zero_points` 参与计算的路径）。

**📋 变更摘要**：在 Rust 前端中新增对 Granite 4 模型的工具调用解析器 `Granite4ToolParser`，并完成注册、模式匹配、单元测试以及相关导出。该解析器支持 `<tool_call>…</tool_call>` 包裹的 JSON 调用，能够处理对象形式与转义字符串形式的 `arguments`，并兼容流式分块输入。

**结论**：本次改动显著提升了 benchmark 的可配置性（多参数 sweep、value‑head、split‑K、NCU 直接捕获），并统一了计时实现，使结果更可靠。但为了平滑迁移，需要对向后兼容、参数校验和文档进行补充，并加入相应的单元/集成测试。整体方向正确，建议在上述细节上做少量修补后合并。
