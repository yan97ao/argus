### [1/N] Elastic EP Milestone 2 (#34861)
**SHA**: `dea2683` | 🔗 [查看提交](https://github.com/vllm-project/vllm/commit/dea268336fb51b5bc342dca29189f3d4440ca2a0)

**🎯 变更类型**：功能增强 / 大幅架构重构（Elastic Expert Parallelism + Stateless 分布式组）  

**⚡ 重要程度**：🔴 高  

**📋 变更摘要**  
- 为 vLLM 引入 **Elastic Expert Parallelism (Elastic EP)**，支持在运行时弹性扩缩 **Data‑Parallel (DP)** 与 **Expert‑Parallel (EP)** 节点，无需重新启动服务。  
- 全新实现 **StatelessGroupCoordinator**，基于独立的 `TCPStore` 与自行创建的 `ProcessGroup`（CPU、GPU），实现 **DP/EP/EPLB/World** 的动态组创建与销毁，避免对全局 `torch.distributed` 状态产生副作用。  
- 新增 **ElasticEPScalingExecutor** 与 **ElasticEPScalingState**，负责 **创建 standby 组 → 权重/映射广播 → KV‑Cache 同步 → 组切换 → EPLB 重排** 的完整状态机。  
- 改写 **EngineCore、EngineCoreProc、DPLBAsyncMPClient、DP‑Coordinator、模型加载、编译、工作空间** 等关键路径，以适配弹性扩缩需求。  
- 新增 **CLI 参数** `--enable-elastic-ep`、对应配置字段 `ParallelConfig.enable_elastic_ep`，以及 **环境变量** `VLLM_ELASTIC_EP_SCALE_UP_LAUNCH / VLLM_ELASTIC_EP_DRAIN_REQUESTS` 用于内部协调。  
- 对 **All2All**、**CUDA 通讯**、**PyNCCL**、**DeepEP**、**FlashInfer** 等底层通信层加入 **stateless** 支持，统一在 `base_device_communicator` 中处理 `stateless` 与普通组两套路径。  
- 对 **EPLB**（Expert Load Balancer）做了大量改造，使其在弹性 EP 场景下能够 **暂停/恢复**、**禁止重排**、**通过广播/点对点传输** 迁移权重与专家映射。  
- 增加 **工作空间解锁** 接口以支持在弹性缩放期间动态扩容。  

**🎯 影响范围**  
- `vllm/distributed/*`（并行状态、stateless 组、通信层）  
- `vllm/distributed/elastic_ep/*`（核心弹性逻辑）  
- `vllm/v1/engine/*`（EngineCore、Coordinator、Client、CoreProc）  
- `vllm/v1/worker/*`（GPU/CPU ModelRunner、权重迁移、EPLB）  
- `vllm/v1/executor/*`（Ray、Multiproc、Uniproc）  
- `vllm/compilation/*`（CUDA‑Graph、torch.compile 重置）  
- `tests/*`（大量测试适配 `ensure_current_vllm_config`）  
- CLI 与配置层（`engine/arg_utils.py`, `entrypoints/cli/serve.py`）

## 🔍 技术洞察

| 维度 | 影响与细节 |
|------|-----------|
| **架构** | 1. **StatelessGroupCoordinator** 通过 `TCPStore` 与自定义 `ProcessGroup` 实现 **可随时创建/销毁** 的通信组，突破了传统 `torch.distributed` 只能在进程启动时一次性初始化的限制。<br>2. **ElasticEPScalingState** 将弹性缩放拆分为 **状态机**（CreateStby → Transfer → Switch → EPLB 重排），保证每一步都有 **双阶段 TCPStore barrier** 保障所有节点同步，防止出现半边切换导致死锁或错误。<br>3. **ElasticEPScalingExecutor** 负责 **权重批量点对点传输**（`batch_transfer_weights`）以及 **专家映射广播**，使用 `P2POp` 与自研的 `batch_isend_irecv`（在 stateless 组下走 `CudaCommunicator`）实现高效迁移。<br>4. **EngineCoreProc** 在 `__init__` 中根据 `VLLM_ELASTIC_EP_SCALE_UP_LAUNCH` 在弹性扩容阶段提前 **发送 NEW_CORE_ENGINES_INIT_READY** 通知，确保旧节点已进入等待状态。<br>5. **DP‑Coordinator** 增加对新发动机的 **READY** 消息处理，用于在弹性扩容时配合 `EngineCoreClient` 完成订阅‑确认流程。 |
| **性能** | - **弹性扩容/缩容仅在对应步骤进行通信**（权重、映射、KV‑Cache 大小同步），不需要全局重启，极大降低扩容 latency（秒级 vs minutes）。<br>- **All2All & P2P** 在 stateless 组下直接走 `CudaCommunicator.batch_isend_irecv`，避免 `torch.distributed` 额外的序列化/聚合开销。<br>- **编译缓存 & CUDA‑Graph** 在 `ElasticEPScalingExecutor.switch_and_prepare` 中被强制 **reset**（`reset_compile_wrapper`），防止旧的编译常量导致错误，同时通过 `torch.compiler.reset()` 重新触发编译。<br>- **KV‑Cache** 大小同步通过 `ParallelConfig.sync_kv_cache_memory_size` 在扩容后一次性广播，避免每个请求单独检查。<br>- **工作空间** 在扩容期间可 **unlock**，防止 OOM，随后再次 `lock` 保证后续执行的显存安全。 |
| **安全** | - 新增 **双阶段 TCPStore barrier**（`_staged_barrier`）在首次等待出现 timeout 时，会让已进入工作循环的进程继续运行，避免因超时导致强制退出，引入 **可恢复** 的安全策略。<br>- 所有 **stateless** 组在创建时使用 **独立端口**，在 `ParallelConfig.allocate_elastic_ep_ports` 中统一申请，避免端口冲突。<br>- `StatelessProcessGroup` 对外仅提供 **广播、发送/接收对象**，不暴露底层 `torch.distributed` 全局状态，降低误用风险。<br>- 对 **EPLB 重排** 加入 **is_profile** 与 **execute_shuffle** 参数，保证在 `profile_run` 中不触发实际参数移动，防止意外泄露或破坏模型状态。 |
| **可维护性** | - **代码分层清晰**：弹性 EP 逻辑集中在 `elastic_ep/` 目录，原有路径仅需少量适配。<br>- 引入 **contextmanager `ensure_current_vllm_config`** 解决大量测试及部分运行时对全局 config 的依赖问题，提升测试可复用性。<br>- 通过 **统一的 `set_current_vllm_config`** 与 `reset_compile_wrapper`，保持编译状态的一致性，降低因旧编译缓存导致的难以排查问题。<br>- 但是 **新增的状态机、双阶段 barrier、stateless 组** 交叉使用，代码路径增多，文档与注释必须同步更新，防止维护者误用。 |
| **兼容性** | - 原有 **非弹性** 场景保持不变；如果 `enable_elastic_ep=False`，所有新建的 `StatelessGroupCoordinator` 不会被触发，仍走原有 `torch.distributed` 组。<br>- **CPU ModelRunner** 明确不支持弹性 EP（在 `load_model` 中抛异常），因此在混合 CPU‑GPU 环境下需禁用此特性。<br>- `torch.compile` 与 `stock_torch_compile` 在弹性 EP 中仍可使用，但在扩容阶段会强制 **重新编译**，可能出现编译时间波动。 |
| **测试覆盖** | - 大量 `tests/*` 已加入 `ensure_current_vllm_config`，确保分支覆盖 `init_distributed_environment` 与 `initialize_model_parallel` 中对 `VllmConfig` 的依赖。<br>- 新增 **Elastic EP Scaling Test** (`tests/distributed/test_elastic_ep.py`) 包括 **scale‑up**、**uneven scale‑up**、**scale‑down** 场景，验证 GSM‑8K 精度不下降，确保业务正确性。 |

### [Refactor][Kernel] Add global helper to deduplicate vectorized memory ops (#35105)
**SHA**: `a201ad7` | 🔗 [查看提交](https://github.com/vllm-project/vllm/commit/a201ad72d87eaaa1fe20e2f42378be4ddbc867f4)

**🎯 变更类型**：功能增强 / 重构 / 架构变更  
**⚡ 重要程度**：🔴 高  

**📋 变更摘要**  
1. 新增 `csrc/cuda_vec_utils.cuh`，统一管理 **向量化内存操作**（128/256 bit 加载、存储、对齐检查、类型转换）以及 **CUDA 类型/packed 类型映射**。  
2. 通过 `VecTraits`、`PackedTypeConverter`、`CUDATypeConverter`、`PackedVec` 等模板元编程结构，实现 **编译期自动选择 128‑bit 或 256‑bit 向量**，并消除各 kernel 中的重复实现。  
3. 所有激活、量化、专家混合等 CUDA kernel 改为使用新工具函数（`ld128`、`ld256`、`st128`、`st256`、`is_16byte_aligned`、`is_32byte_aligned`），并把原有的手写对齐检测、手写向量结构 (`u32x8_t`、`PackedVec`) 删除或迁移。  
4. 启动宏 `LAUNCH_ACTIVATION_GATE_KERNEL` 等也改为基于 `VecTraits` 计算 **最大向量尺寸**，并在 CUDA 12.9+ + SM100+ 环境下开启 256‑bit PTX 指令（`VLLM_256B_PTX_ENABLED`）。

---

## 🎯 影响范围
- **核心 CUDA 计算模块**：`activation_kernels.cu`、所有 **fp4 量化** 相关 kernel（`activation_nvfp4_quant_fusion_kernels.cu`、`nvfp4_experts_quant.cu`、`nvfp4_quant_kernels.cu`）以及对应的 host‑side 调用入口。  
- **公共工具库**：新增的 `cuda_vec_utils.cuh` 将被多个 `.cu` 文件 `#include`，成为向量化内存访问的唯一实现点。  
- **编译配置**：对 `CUDA_VERSION`、`SM` 号的检测逻辑产生变动，影响 **所有支持的 GPU 架构**（尤其是 SM<100） 的编译路径。

## 🔍 技术洞察

| 维度 | 影响说明 |
|------|----------|
| **架构影响** | • 引入 **256‑bit PTX（v8.u32）** 指令，仅在 **CUDA 12.9+ 且 SM 100+**（A100‑后代、H100、Ada 等）可用。<br>• 通过 `VecTraits<support_256>` 在编译期决定向量宽度，保持向后兼容：旧显卡仍走 128‑bit (`int4`) 路径。<br>• 新增 `is_16byte_aligned / is_32byte_aligned` 作为统一对齐检查，取代 kernel 内部散乱的手写位运算。 |
| **性能影响** | • 256‑bit 加载/存储一次搬运 8×`uint32_t`，相较原先 128‑bit（4×）可提升 **≈2× 内存吞吐**（在支持的硬件上）。<br>• 统一使用 `__ldg`（只读缓存）或 `.cg/.cs`（全局/缓存流）实现，消除冗余的 `if constexpr` 分支，减小寄存器压力。<br>• 由于 `PackedVec` 按向量大小对齐，编译器更易生成高效访存指令，理论上 **占用的寄存器/共享内存不变**，实际占用略增（额外的模板实例化），但对吞吐提升的抵消可忽略。 |
| **安全考虑** | • 代码仅在设备侧执行，未涉及外部输入。<br>• 通过 `assert(false && "...")` 在不满足 SM100+ + CUDA 12.9+ 条件时导致运行时错误，防止错误指令执行；对生产环境来说相当于 **安全失效**（快速定位不兼容硬件），但可能在 CI 中导致异常退出，需要在 CI 中覆盖旧 GPU 测试。 |
| **可维护性** | • 大幅 **消除重复实现**：所有向量加载/存储、对齐检测、类型映射集中在 `cuda_vec_utils.cuh`，后续修改只需改动单文件。<br>• 更清晰的类型抽象（`CUDATypeConverter`、`PackedTypeConverter`）提升代码自解释性，降低新开发者的学习成本。<br>• 通过 `constexpr` 与 `static_assert` 提前捕获不匹配的向量尺寸，编译期错误代替运行时错误。 |
