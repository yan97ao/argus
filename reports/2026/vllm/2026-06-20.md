### [Perf] Optimize Qwen3-VL multi-video prompt processing (#46026)
**SHA**: `d272418` | 🔗 [查看提交](https://github.com/vllm-project/vllm/commit/d272418f459a82e1012b60116ac00659a7017cde)

**🎯 变更类型**：性能优化（Perf）  
**⚡ 重要程度**：🟡 中  

**📋 变更摘要**  
- 为 Qwen3‑VL 多视频提示的处理路径新增基于 token‑level 的占位替换逻辑，避免了原先对完整文本进行逐段展开导致的二次遍历（quadratic）开销。  
- 新增 `_replace_video_token_placeholders` 实现一次遍历完成三标记占位（\<|vision_start|>\<|video_pad|>\<|vision_end|>）的批量替换，并在 `Qwen3VLMultiModalProcessor._call_hf_processor` 中调用。  
- 对视频处理过程中的 EVS（effective video sampling）计数、token‑id 选取等保持不变，只是把 HF 处理得到的 `input_ids` 替换为预先生成的 `video_repl.full` 序列。  
- 相应单元测试 `test_processor_multi_video` 扩展到 2、4 条视频，校验占位数量、长度一致性以及不重叠。

**🎯 影响范围**  
- `vllm/model_executor/models/qwen3_vl.py`（核心多模态处理器）  
- `tests/models/multimodal/processing/test_qwen3_vl.py`（新增/扩展的多视频测试）  

**💡 关注建议**  
1. **兼容性**：新逻辑依赖占位顺序严格对应 `video_input_ids_lst`，若未来在 prompt 中出现非连续或自定义占位（如手动插入文字）需确保 `replacements` 顺序仍然对应。建议在文档或注释中明确“占位必须保持出现顺序”。  
2. **异常路径**：`_replace_video_token_placeholders` 对占位数量不匹配会触发 `assert`，在生产环境中可能抛异常。可考虑改为抛出自定义异常或回退到原始文本展开路径，以免因用户误漏占位导致服务崩溃。  
3. **性能验证**：虽然单元测试覆盖了 2、4 视频的占位替换，但未测量实际加速幅度。建议在 CI 中加入基准测试（如 8、16 视频）并对比旧实现的耗时，以确保优化效果在大规模 prompts 下显著。  
4. **EVS 逻辑保持**：视频 token 数量仍通过 `compute_retained_tokens_count` 计算，确保 `select_token_id` 的传递未被误改；建议在注释中说明 “此处不再使用 HF 的 input_ids，仅保留 token‑level 替换”。  

总体而言，此次改动通过一次线性遍历实现多视频占位的高效展开，符合性能优化目标，风险主要在占位顺序和异常处理上，建议适当加强容错和文档说明。

### [Bugfix] Fix gridDim.y overflow for large row counts (#45255)
**SHA**: `93bad11` | 🔗 [查看提交](https://github.com/vllm-project/vllm/commit/93bad119120d0f9bff707dcbf5af5c029158b969)

**🎯 变更类型**：Bug修复（gridDim.y 溢出导致大行数启动失败）  

**⚡ 重要程度**：🟡 中  

**📋 变更摘要**  
1. 修正 `per_token_group_quant_8bit_packed_register_kernel` 中对 CUDA grid 索引的错误映射——原先把 `blockIdx.x` 当作行索引、`blockIdx.y` 当作列索引，导致在行数（mn）超过 2^16‑1 时 `grid.y` 超界。改为 `sf_k_idx = blockIdx.y * kGroupsPerBlockX + sf_k_local`、`mn_idx = blockIdx.x * kRowsPerBlock + row_local`。  
2. 在调度函数中把原来的 `blocks_x / blocks_y` 变量分别改为 `row_blocks`（对应 `grid.x`）和 `sf_k_blocks`（对应 `grid.y`），并加入对 `grid.y` 上限 65535 的显式检查。  
3. 添加单元测试 `test_per_token_group_quant_fp8_packed_large_mn`，在 `mn = 65537`（超出旧实现的 `grid.y` 限制）上跑 FP8 per‑token‑group 量化，确认修复后能够成功启动并得到与参考实现相同的结果。  

**🎯 影响范围**  
- 核心 CUDA 量化实现：`csrc/libtorch_stable/quantization/w8a8/fp8/per_token_group_quant.cu`。  
- Python 调用层：`vllm.quantization.fp8_utils`（通过 `per_token_group_quant_fp8_packed_for_deepgemm`）的行为不变，但在极大 token 数时不再抛出 “invalid argument”。  
- 测试套件：新增 `tests/kernels/quantization/test_per_token_group_quant.py`，仅在 CUDA‑like 环境下执行。  

**💡 关注建议**  
- **开发者**：审阅新加入的 `kMaxGridDimYZ` 常量和边界检查，确保与 CUDA 运行时文档保持一致；若后续支持新 GPU（如拥有更大 `grid.y`），记得同步更新该阈值。  
- **用户**：在使用 `fp8`/`int8` per‑token‑group 量化时，可放心处理超过 65 535 行的张量（例如超大 batch 或长上下文）。若仍遇到 “CUDA error: invalid argument”，请检查是否还有其他 kernel（如 Triton 实现）受相同限制。  
- **CI/性能**：新测试会在 GPU 环境下跑约数十秒，确保 CI 配置的 GPU 具备足够显存；若出现超时，可考虑对大规模测试做分片或仅在 nightly 分支执行。  

总体而言，此次改动解决了罕见但关键的启动溢出 Bug，提升了库在大规模推理场景下的鲁棒性，风险有限且兼容性良好。

### [Bugfix][Parser] Fix U+FFFD leak at reasoning-to-content transition in engine parsers (#46159)
**SHA**: `859e4d4` | 🔗 [查看提交](https://github.com/vllm-project/vllm/commit/859e4d436ba0fb0da8a655a80d5c4fab12adc82e)

**🎯 变更类型**：Bug 修复  
**⚡ 重要程度**：🟡 中  
**📋 变更摘要**：修复在 reasoning→content 转换时出现的 U+FFFD（替换字符）泄漏。通过在 `AbstractParser.parse_delta` 中显式调用推理解析器的 `finish_streaming`，并在 `EngineParser.finish_streaming` 里将延迟的 content（`_deferred_content`）一起打包返回，确保残余的字节‑fallback 令牌被正确冲刷。新增回归测试 `test_ufffd_reasoning_transition.py`，覆盖多种 chunk 大小以及多 token 边界情形。  

**🎯 影响范围**  
- `vllm/parser/abstract_parser.py`（核心流控制逻辑）  
- `vllm/parser/engine/parser_engine.py`（`finish_streaming` 返回策略）  
- 测试目录 `tests/parser/engine/`（新增、微调）  

**💡 关注建议**  
1. **逻辑完整性**：`parse_delta` 现在在 `self._engine_based` 分支里直接使用 `reasoning_parser.finish_streaming()`，若 `reasoning_parser` 已经结束或返回 `None`，代码仍会安全拼接；但需确认其它非 engine‑based 解析路径不受影响。  
2. **返回一致性**：`EngineParser.finish_streaming` 现在在有 `_deferred_content` 时强制返回 `DeltaMessage`（`finished=True`），请检查调用方是否已准备好处理 `finished=True` 的额外字段，防止旧实现假设 `None`。  
3. **测试覆盖**：新增的 `CHUNK_SIZES` 常量在两个测试文件中复用，确保在未来增删测试时保持同步。  
4. **性能与副作用**：`finish_streaming` 结束后会清空引擎状态，确认在流式解码的多轮交互场景中不会提前重置导致 token 丢失。  
5. **文档与类型**：建议在 `finish_streaming`、`parse_delta` 添加详细注释，说明“冲刷推理 lexer” 的意图，并为返回值加上 `Optional[DeltaMessage]` 的显式类型提示，以提升可维护性。  

总体来看，此次改动定位准确、测试充分，风险主要在于修改了流控制的返回行为，建议在全套 CI 中跑一遍现有的流式解析用例，确认未引入其他隐藏的回归。

### [DSv4] Pack KV caches into contiguous per-block allocations for DeepSeek V4 (#44577)
**SHA**: `0119213` | 🔗 [查看提交](https://github.com/vllm-project/vllm/commit/01192139bf022bec84e2cca3a3e36e8bb5293b5c)

**🎯 变更类型**：功能增强（DeepSeek V4 KV Cache 连续块打包）  
**⚡ 重要程度**：🟡 中  
**📋 变更摘要**：为 DeepSeek V4 模型实现每块连续的 KV Cache 分配：在 `kv_cache_utils` 中生成统一大小、统一 stride 的 `KVCacheTensor`，在 GPU/CPU 侧统一只分配一次 backing storage 并通过 `offset / block_stride` 的 strided view 进行层间映射；NIXL 连接器新增对打包存储的注册路径；配套单元测试验证 stride、offset、视图独立性等属性。  

**🎯 影响范围**  
- `vllm/v1/core/kv_cache_utils.py`（核心配置生成）  
- `vllm/v1/kv_cache_interface.py`（`KVCacheTensor` 新增字段）  
- GPU 工作线程 `gpu/attn_utils.py`、`gpu_model_runner.py`（统一 backing、视图创建）  
- NIXL KV Cache 连接器 `kv_connector/v1/nixl/base_worker.py`（打包注册）  
- Offloading 代码 `kv_connector/v1/offloading/worker.py`（视图创建）  
- 新增测试 `tests/v1/core/test_contiguous_kv_packing.py`  

**💡 关注建议**  

1. **兼容性检查**  
   - 确认所有非 DeepSeek V4（或不满足 `block_stride>0`）的模型路径仍走原有分配逻辑；当前实现通过 `len(kv_caches)>1 and not self._has_mamba` 检测，需防止误判（如多层共享但未使用 DSv4 打包）。  
   - `KVCacheTensor.offset` 与 `block_stride` 默认 0，确保在序列化/复制时不丢失，避免旧代码读取到不完整结构。

2. **视图正确性**  
   - `prod(kv_cache_shape[1:]) * get_dtype_size(dtype)` 计算的 `page_bytes` 依赖 `kv_cache_shape` 已经 permuted，建议在注释或代码中明确前置条件（`inv_order[0]==0`），防止未来改动导致错误。  
   - 在 `gpu/attn_utils.py` 与 `gpu_model_runner.py` 两处实现几乎相同，考虑抽取公共函数，避免一致性风险。

3. **内存对齐与跨设备**  
   - `block_stride = total_size // self.num_blocks` 假设 total_size 能被块数整除；若 `available_memory` 不是块大小的整数倍，后续可能出现 `block_stride` 非整数或碎片。建议在 `may_override_num_blocks` 前加入对齐检查并在日志中提示。  
   - NIXL 注册时使用 `storage.device.index`，确保在多 GPU 场景下 `device_id` 一致，避免在跨节点迁移时产生错误。

4. **测试覆盖**  
   - 当前测试覆盖了基本 stride、offset、独立视图，但未涉及带 `page_size_padded` 的层或 Mamba‑Spec。建议补充相应场景的单元测试。  
   - 添加对 `kv_cache_config.block_stride == 0`（即未打包）路径的回归测试，防止意外回退。

5. **文档与使用说明**  
   - 在 README 或模型配置文档中说明何时会触发 “contiguous KV packing”，以及对显存占用、性能的预期提升。  
   - 若用户自行指定 `num_gpu_blocks_override`，确认该参数仍能正确影响 `num_blocks` 计算，而不会破坏 `block_stride` 的计算。

整体来看，此次改动为 DeepSeek V4 引入了显著的显存布局优化，代码结构清晰且已配套单元测试。只需在兼容性、对齐检查以及文档层面做好补充，即可稳妥上线。
