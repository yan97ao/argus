### Revert "[MoE Kernel] Flashinfer nvfp4 cutedsl moe kernel integration" (#38050) (#38169)
**SHA**: `be1a85b` | 🔗 [查看提交](https://github.com/vllm-project/vllm/commit/be1a85b7a2929f25c93d469fdd733a3576609e70)

**🎯 变更类型**：重构 / 架构变更  
**⚡ 重要程度**：🔴高  
**📋 变更摘要**：本次提交回滚了原先对 FlashInfer nvfp4 CuteDSL MoE kernel 的批量（BatchedExperts）集成，同时对 `flashinfer_cutedsl_moe.py` 实现进行合并、参数和工作区计算方式的统一，删除了对应的批量实现文件及相关导入/选项。代码路径从 “Flashinfer CuteDSL Batched” 迁回 “Flashinfer CuteDSL Standard”，并调整了后端映射、量化配置以及依赖检查。

**🎯 影响范围**  
- `vllm/model_executor/layers/fused_moe/experts/flashinfer_cutedsl_moe.py`（核心 MoE 实现）  
- `vllm/model_executor/layers/fused_moe/experts/flashinfer_cutedsl_batched_moe.py`（被删除）  
- `vllm/model_executor/layers/fused_moe/oracle/nvfp4.py`（后端选择逻辑）  
- `vllm/model_executor/layers/quantization/utils/flashinfer_fp4_moe.py`（去除批量专用准备函数）  
- `vllm/utils/flashinfer.py`（懒加载包装、特性检测）  
- 相关单元测试 `tests/kernels/moe/test_cutedsl_moe.py`  

**🔍 技术洞察**  

- **架构影响**  
  - 移除 `FLASHINFER_CUTEDSL_BATCHED` 后端，统一 MoE 激活格式为 `BatchedExperts`（原实现中实际采用的是 `Standard`，新版显式返回 `BatchedExperts`），简化了后端映射表和 `backend_to_kernel_cls` 分支。  
  - `FlashInferCuteDSLExperts` 现在直接使用 `flashinfer_cutedsl_moe_masked`（基于 `flashinfer_cutedsl_grouped_gemm_nt_masked`）进行两阶段 GEMM + SiLU‑Quant，取代之前的 `flashinfer_cute_dsl_fused_moe_nvfp4` 调用。  
  - 工作区形状计算统一为 `(local_num_experts, M, K_dim)` 与 `(local_num_experts, M, N)`，与批量实现保持兼容，但不再依赖 `workspace13`（已删除），降低了内部状态管理复杂度。  

- **性能影响**  
  - **正向**：去掉了批量实现的额外代码路径，减少了 import/函数调用层级，可能略微降低启动时的模块加载开销。  
  - **负向**：原批量实现利用 `flashinfer_cutedsl_batched_moe` 进行一次性 GEMM + 激活，理论上在大 token‑batch 场景下拥有更好的显存复用和算子融合。回滚后改为两次 `flashinfer_cutedsl_grouped_gemm_nt_masked`（GEMM1 + GEMM2）并在中间显式 SiLU+Quant，可能导致 **额外一次工作区拷贝** 与 **两次 kernel 启动**，在极端高吞吐场景下有轻微性能下降。  
  - 通过 `workspace_shapes` 的统一，仍然支持 “activation chunking”，所以在切片（chunk）模式下的性能损失可被最小化。  

- **安全考虑**  
  - 变更仅涉及内部算子调用与量化比例的处理，没有引入网络 I/O、权限检查或外部依赖，安全风险极低。  
  - 需要关注 **量化尺度的正确性**：`prepare_static_weights_for_trtllm_fp4_moe` 中的 `is_nvfp4_scale_swizzled` 条件已被简化，只在 `FLASHINFER_TRTLLM` 时关闭 swizzle，确保其他后端仍保持原有 swizzle 行为，防止数值误差导致的模型崩溃。  

**⚠️ 潜在风险**  

| 风险点 | 可能后果 | 触发条件 |
|--------|----------|----------|
| 删除 `flashinfer_cutedsl_batched_moe.py` 后，仍有用户代码显式导入该模块 | ImportError，启动失败 | 用户自定义插件或旧版脚本硬编码路径 |
| `backend_to_kernel_cls` 未同步更新所有引用，导致 `NvFp4MoeBackend.FLASHINFER_CUTEDSL_BATCHED` 仍保留在配置文件中 | 运行时抛出 “Unsupported backend” 错误 | `runner_backend` 参数设为 `flashinfer_cutedsl_batched` |
| 工作区形状计算变化未同步到 `PrepareAndFinalize` 流程 | Tensor shape mismatch、CUDA 错误 | 某些自定义 `MoEParallelConfig` 改动导致 `local_num_experts` 与 `global_num_experts` 计算不同 |
| `quant_config.is_nvfp4_scale_swizzled` 逻辑变化可能影响已有 `VLLM_CUTLASS` 或 `FLASHINFER_CUTLASS` 的量化比例 | 量化误差、模型精度下降 | 使用这些后端且依赖原 swizzle 行为的模型 |
| `flashinfer_cutedsl_moe_masked` 对 `masked_m`（token index）假设 `max(masked_m) <= m`，未做显式校验 | 越界访问导致 CUDA Illegal memory access | 极端异常路由分配（如 token‑to‑expert 超过分配块） |

2. **性能基准**：对比回滚前后在典型的 8‑B, 64‑B、256‑B token‑batch 场景中 MoE 前向吞吐（tokens/s）与显存占用，确认是否出现显著回退；若回退超过 5% ~ 10%，考虑保留批量实现作为可选后端。

### [CI] Reorganize scoring tests (#38207)
**SHA**: `dcdc145` | 🔗 [查看提交](https://github.com/vllm-project/vllm/commit/dcdc1458931a1a57cbcadb92b2dfc10d225d675f)

**🎯 变更类型**：重构 / 功能增强（测试体系重构并新增大量评分相关用例）  

**⚡ 重要程度**：🔴高  

**📋 变更摘要**  
1. 将原有的 `pooling/score` 测试迁移并重新组织，统一放在 `tests/entrypoints/pooling/scoring/` 下，新增离线/在线的 **Bi‑Encoder、Cross‑Encoder、Late‑Interaction** 三类完整测试套件。  
2. 删除已废弃的旧测试文件，减少冗余代码。  
3. 将错误提示文字从 *“Please, select a smaller truncation size.”* 调整为 *“Please request a smaller truncation size.”*，并同步更新所有相关断言。  

**🎯 影响范围**  
- `vllm/entrypoints/openai/engine/serving.py`  
- `vllm/entrypoints/pooling/base/serving.py`（错误信息文字修改）  
- 项目测试目录（新增约 1.5k 行测试代码，删除约 1k 行旧测试）  
- 依赖：新增对 `transformers`, `huggingface_hub`, `safetensors` 的运行时依赖（用于离线/在线模型调用的测试）  

**🔍 技术洞察**  
- **架构影响**：仅测试层面的重构，对运行时服务逻辑、模型调度、渠道层（OpenAI、Pooling）保持不变，系统架构不受影响。  
- **性能影响**：新增的在线测试会在 CI 中实际启动 RemoteOpenAIServer 并请求模型推理，导致 **CI 运行时长显著增长**（尤其在 GPU/CPU 资源受限的环境下），但不影响生产环境的吞吐或延迟。  
- **安全考虑**：无安全相关改动。唯一需要注意的是，测试阶段会通过网络拉取 HF 模型，若 CI 环境对外网访问受限可能导致测试失败。  

**⚠️ 潜在风险**  
1. **API 向后兼容性**：错误信息文字的微调会导致依赖错误文本进行匹配的第三方客户端或脚本失效。  
2. **CI 稳定性**：大量模型下载、GPU/CPU 资源竞争可能导致超时或 flaky 测试，影响合并速度。  
3. **依赖膨胀**：新增的 `transformers`、`safetensors`、`huggingface_hub` 可能在某些轻量化部署环境引入不必要的包。  

**💡 关注建议**  
- **文档同步**：在 Release Note 与 API 文档中声明错误提示文字已更新，建议用户改为检测错误码或使用更宽松的匹配方式。  
- **兼容层**：若担心破坏现有用户，可在错误返回中保留旧文本（如同时返回两种提示），或提供 `error_message` 与 `error_code` 双字段。  
- **CI 优化**：  
  - 将大型模型的下载缓存到 CI 持久化目录，避免每次运行都重新拉取。  
  - 为新增的在线测试设置专用的 `pytest` 标记（例如 `@pytest.mark.slow`），在常规 CI 中可选择性跳过。  
  - 监控测试时长并在必要时拆分为多阶段 CI。  
- **依赖审计**：确保新增的第三方库在许可证、二进制兼容性上与项目现有政策保持一致；在最小化发行版中可以通过 `extras_require` 将这些测试依赖标记为可选。  

---  

> 综上，代码改动本身对产品功能无直接影响，主要是提升了 **评分功能的测试覆盖率** 与 **错误信息统一性**。唯一需要重点关注的是 **向后兼容的错误信息变更** 与 **CI 稳定性**，在正式发布前建议做好文档说明并在 CI 中做好缓存与分组执行策略。

| **性能影响** | - **并发度提升**：一次请求内部生成 *N* 个 `engine_client.generate` 异步生成器，通过 `merge_async_iterators` 并行收集结果。理论上在同等硬件下批量请求的吞吐量会提升（多请求共享同一轮询循环），但单个对话的响应延迟仍受模型生成时间支配。<br>- **内存占用**：每个子请求会持有独立的 `EngineInput`、`SamplingParams` 与 token‑id 缓冲，批量大小过大可能导致显存或 CPU 内存压力。建议在部署层面设定 `max_batch_size`（未实现）或通过前端限流。 |

### ⚠️ 潜在风险  

1. **资源耗尽**：批量大小未受限，极端情况下会导致显存/CPU 爆炸。  
2. **错误传播**：若某个子请求在生成阶段抛出异常（例如模型内部错误），`merge_async_iterators` 会中止所有批次，返回统一错误，可能影响其他已完成的子请求。

4. **取消处理**：已在 `chat_completion_full_generator_batch` 捕获 `CancelledError`，但若客户端在部分子请求完成后中止，仍会导致所有子请求被中断，未返回已完成的结果。  
5. **工具/流式支持缺失**：文档已声明不支持 streaming、tools、beam search，若用户在 batch 请求中误传这些字段，会被 `model_validator` 抛错；若后续实现其中任意功能，需要同步更新 validator 与业务逻辑。

- **文档同步**：更新 OpenAPI schema、README 与示例，明确 batch 端点的限制（不支持 streaming、tools、beam search、`n>1`），并在自动生成的 Swagger 中展示。  
- **安全加固**：在服务入口（如 `uvicorn` 启动参数或前置网关）限制请求体大小，或在路由内部对 `messages` 总长度进行阈值检查。  
- **回归测试**：补充极端批量（如 100 条对话）和错误场景（包含 `stream=True`、`tools`、`n=2`）的测试，确保 validator 可靠且不会导致服务器异常。
