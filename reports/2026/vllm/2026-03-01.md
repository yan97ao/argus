### fix(mxfp4): return is_monolithic=False when LoRA is enabled for Triton backend (#35382)
**SHA**: `5a43550` | 🔗 [查看提交](https://github.com/vllm-project/vllm/commit/5a435507d877f4eb16802095037d5c56e767c589)

**🎯 变更类型**：Bug 修复  
**⚡ 重要程度**：🟡 中  
**📋 变更摘要**：在 `vllm/model_executor/layers/quantization/mxfp4.py` 中的 `is_monolithic` 属性新增了对 LoRA（Low‑Rank Adaptation）的检测：当模型开启 LoRA（`self.moe.is_lora_enabled` 为 `True`）时，强制返回 `False`，从而阻止 Triton 后端误将 LoRA 模型当作单体（monolithic）实现执行。  

**🎯 影响范围**  
- `vllm/model_executor/layers/quantization/mxfp4.py`（核心量化实现）  
- 依赖 `is_monolithic` 判断的调度、kernel 选择逻辑，尤其是 Triton 后端的 GEMM 实现分支  
- 使用 LoRA 并选择 mxfp4 量化的模型推理路径（包括 CPU/GPU 以及多卡部署）  

**💡 关注建议**  
1. **回归测试**：对开启 LoRA 的 mxfp4‑Triton 组合跑完整套推理/生成测试，确保不再出现因为误判为 monolithic 而导致的 kernel 加载错误或性能下降。  
2. **性能监控**：`is_monolithic=False` 将使模型走 “拆分” 路径，可能带来额外的 kernel 调用开销。建议在常规基准（FP16/FP8 等）上对比前后延迟/吞吐，确认影响在可接受范围。  
3. **文档更新**：在 LoRA 与量化兼容性章节补充说明：当使用 Triton 后端且启用 LoRA 时，系统会自动禁用 monolithic 模式。  
4. **代码审查**：确认 `self.moe.is_lora_enabled` 在所有可能的 MoE/LoRA 组合中都有可靠的值（避免因属性缺失导致 AttributeError），必要时加上容错或类型注解。  
5. **CI 扩展**：在 CI 中加入 LoRA+MXFP4（Triton） 的矩阵乘法单元测试，防止未来改动再次回退此逻辑。  

总体来看，此次修改修复了 LoRA 与 Triton 后端的兼容性问题，影响范围主要局限在量化层和调度路径，风险相对低。只要做好上述回归与性能验证，即可安全合并。

### [Mamba1] - Kernel Level Chunk Alignment for Prefix Caching (#34798)
**SHA**: `bbf81f9` | 🔗 [查看提交](https://github.com/vllm-project/vllm/commit/bbf81f9a9284d572b69db2c4fb002c2a8a80d507)

**🎯 变更类型**：功能增强（Kernel Level Chunk Alignment for Prefix Caching）  
**⚡ 重要程度**：🟡 中  
**📋 变更摘要**：  
- 在 `SSMParamsBase` 中新增 `cu_chunk_seqlen_ptr`、`last_chunk_indices_ptr` 以在 CUDA 层携带 **chunk‑level** 序列元信息。  
- `selective_scan_fwd_kernel` 重写 chunk 划分逻辑：若提供上述元数据，使用 **对齐块（APC）** 而非固定 `block_size`，并据此计算 `chunk_tokens`、`current_position`、缓存写入位置等。  
- 统一了 API：`selective_scan_fwd`、`ops.h`、torch binding、Python 包装层全部加入对应可选张量参数。  
- 在注意力元数据构建 (`mamba_attn.py`、`mamba1_attn.py`) 中实现统一的 **chunk metadata 计算并输出张量**，并在 `Mamba2` 中移除冗余实现。  
- 测试、调用方代码同步更新，以支持新参数。

**🎯 影响范围**：  
- `csrc/mamba/mamba_ssm/*`（CUDA kernel 与参数结构）  
- `csrc/ops.h`、`csrc/torch_bindings.cpp`（C++ 接口）  
- `vllm/_custom_ops.py`、`vllm/model_executor/layers/mamba/*`（Python 包装 & 调用）  
- `vllm/v1/attention/backends/mamba*_attn.py`（注意力元数据构建）  
- 单元测试 `tests/kernels/mamba/test_mamba_ssm.py`

**💡 关注建议**  
1. **兼容性检查**：在非 APC 场景（`cache_enabled=False` 或未传 `cu_chunk_seqlen`）仍使用旧的 `block_size` 分块，确保默认值不导致越界。  
2. **边界条件**：首块可能为不完整块，代码已加 `chunk_start_offset`，建议加入针对 `seqlen < block_size`、`last_chunk_tokens = 0` 的单元测试。  
3. **性能验证**：对比开启/关闭 APC 时的吞吐与显存占用，确认对齐块确实提升缓存状态写入频率。  
4. **文档/注释**：在 `selective_scan_fwd` 的 Python/CPP 文档中明确 `cu_chunk_seqlen` 与 `last_chunk_indices` 的尺寸与含义，防止用户误传。  
5. **回滚路径**：若出现轨迹错误，可临时将 `params.cu_chunk_seqlen_ptr` 设为 `nullptr`，回退到旧的块划分逻辑，便于快速定位问题。

总体而言，此次改动为 Mamba 前缀缓存提供了 **块对齐** 支持，提升了在大 batch 前置（prefill）场景下的状态复用潜力，但需在实际部署中验证边界正确性与性能收益。

### Add TMA support to fused_moe_lora kernel (#32195)
**SHA**: `3ecd0bf` | 🔗 [查看提交](https://github.com/vllm-project/vllm/commit/3ecd0bf9fccc425c015f7723b6a7730c0dda2970)

**🎯 变更类型**：功能增强（为 fused_moe_lora 内核加入 TMA（Tensor‑Map‑Accelerator）支持）  
**⚡ 重要程度**：🟡 中 – 主要面向高端（SM90+）GPU，提升 LoRA 计算效率，但不影响已有功能。  

**📋 变更摘要**  
1. 在 `vllm/lora/ops/triton_ops/fused_moe_lora_op.py` 中为 LoRA expand / shrink 核心 Triton kernel 添加 TMA 路径：引入 `a_desc / b_desc`、`USE_TMA`、`sort_c` 参数，并实现基于 TMA 的加载/写回逻辑。  
2. 在 `utils.py` 增加 `supports_tma` 检测函数；在 `allocation.py` 新增 `set_triton_allocator`，在需要时为 TMA 分配器提供固定‑device Tensor。  
3. 测试层面扩展 `tests/lora/test_fused_moe_lora_kernel.py`、`test_fused_moe_lora_kernel_naive_block_assignment.py`，加入 `num_slices` 参数，验证 TMA 与普通路径在 1‑slice 与 2‑slice 场景下的数值等价。  
4. 新增 `tests/lora/test_olmoe_tp.py`，通过随机 LoRA 权重检查多卡 TP（tensor‑parallel）下的混合 LoRA 逻辑。  

**🎯 影响范围**  
- **vllm.lora.ops.triton_ops**：核心 kernel、调度封装以及 TMA 兼容性判断。  
- **vllm.triton_utils**：新增分配器工具 `set_triton_allocator`。  
- **测试套件**：LoRA 相关单元测试全部更新，覆盖 `num_slices` 与 TMA 分支。  

**💡 关注建议**  
1. **设备兼容**：`supports_tma` 只在 SM90+（A100 80GB、H100 等）上返回 true，运行在低于此级别的 GPU 时会回退到已有路径；确保 CI 环境能覆盖两种情况。  
2. **Fully‑sharded 约束**：代码显式禁用了 TMA 与 fully‑sharded 组合，使用时需在配置中关闭 `fully_sharded_loras`，或在未来实现排序一致性。  
3. **Allocator 生命周期**：`set_triton_allocator` 每次在 `use_tma` 为 true 时调用，确保在多进程/多卡环境下不会产生跨设备泄漏或竞争。建议在 `vllm` 初始化阶段统一调用一次。  
4. **内存布局**：使用 `sort_c` 需要在 `shrink` 阶段将中间缓存按 `sorted_token_ids` 排序；若用户自行提供 `sorted_token_ids` 且关闭 TMA，仍应保证 `sort_c=False` 以免写入错误位置。  
5. **回归测试**：跑完整套 `pytest -m lora`，特别是 `num_slices=2` 场景和多卡 TP（`test_olmoe_lora_tp2`），确认数值误差在 1e‑2 以内且没有 CUDA illegal‑access。  

总体而言，此次改动为高端 GPU 带来了显著的 LoRA 计算加速潜力，代码路径清晰、兼容性检查完善，只要在非 SM90 环境以及 fully‑sharded 场景下确认回退正确，即可安全发布。

总体而言，此次修复提升了 Anthropic‑OpenAI 转换的完整性和鲁棒性，影响主要限于消息转换路径，风险可控。后续如有新增的图片源类型或更复杂的 `tool_result` 结构，请同步更新对应的转换逻辑并补充测试。

**影响范围**

- **Pooling 子模块**：`vllm/entrypoints/pooling/*/protocol.py`。  
- **多模态**：`vllm/multimodal/utils.py`、`vllm/multimodal/processing/processor.py`。  
- **测试**：相应的在线嵌入测试已更新为 `use_activation`。  

**关注建议**  

1. **代码迁移**：所有调用 `LLM.encode(..., truncate_prompt_tokens=…)`、`LLM.embed(..., truncate_prompt_tokens=…)`、`SamplingParams.truncate_prompt_tokens` 的位置必须改为 `tokenization_kwargs={"truncate_prompt_tokens": …}`；同理在 gRPC/HTTP 请求体中使用新字段 `tokenization_kwargs`。  
2. **参数兼容**：如果仍需保留旧行为，可在业务层自行包装 `tokenization_kwargs`，但请尽快清理旧参数，以免在 0.17+ 版本出现 `AttributeError`。

总体而言，此次改动主要是清理已废弃的字段并统一 tokenization 参数入口，对功能本身无行为改变，除非旧参数仍在使用，否则升级后即保持原有行为。完成上述迁移即可安全升级到 v0.17+。
