### [NVFP4] NVFP4 MOE emulation fallback for H100/MI300/MI350, standardize `TritonExperts` usage for OCP MX emulation (#35737)
**SHA**: `d622e27` | 🔗 [查看提交](https://github.com/vllm-project/vllm/commit/d622e27d2be9cd4321d75073e4b7ef522204853d)

**🎯 变更类型**：功能增强 / 重构 / 架构变更 / 性能优化（实质为功能增强，提供了新后端）  

**⚡ 重要程度**：🔴高  

**📋 变更摘要**  
本次提交为 vLLM 引入了 NVFP4 与 OCP MX 量化模式在 MoE（Mixture‑of‑Experts）场景下的**软硬件无关仿真后端**。新增 `emulation` 选项到 `KernelConfig`，实现了两套 `TritonExperts` 子类（`Nvfp4QuantizationEmulationTritonExperts`、`OCP_MXQuantizationEmulationTritonExperts`），在不具备原生 NVFP4 / OCP MX 硬件（如 AMD Instinct MI3xx）时仍可使用 BF16/FP16 跑模型。相应地，后端选择逻辑、权重转换、量化/反量化工具、以及单元测试均做了兼容性扩展。

**🎯 影响范围**  
- **核心层**：`vllm/config/kernel.py`、`vllm/model_executor/layers/fused_moe/*`（experts、oracle、utils、fused_moe、quark_moe）。  
- **量化工具**：`vllm/model_executor/layers/quantization/utils/nvfp4_emulation_utils.py`、相关 MXFP4/MXFP6 utils。  
- **后端选择**：`oracle/mxfp4.py`、`oracle/nvfp4.py` 增加 EMULATION 枚举并映射到新专家实现。  
- **测试**：`tests/models/quantization/test_nvfp4.py`、配置文件 `tests/evals/gsm8k/configs/models-mi3xx.txt`。  
- **环境变量**：`VLLM_NVFP4_GEMM_BACKEND` 现在支持 `emulation`。  

---

### 🔍 技术洞察  

#### 架构影响  
1. **模块化后端抽象**  
   - 通过在 `KernelConfig` 中加入 `"emulation"`，统一了后端声明方式，使得后续还能继续向其他软实现（如 CPU、XPU）扩展。  
2. **专家类层次结构**  
   - 新增 `Nvfp4QuantizationEmulationTritonExperts` 与 `OCP_MXQuantizationEmulationTritonExperts` 均继承自 `TritonExperts`，只在 `apply` 前对权重进行 **即时解量化**，并在 `moe_kernel_quantize_input` 中触发激活 QDQ。  
   - 采用 **“量化仿真标记”** (`self.quantization_emulation = True`) 让上层 `FusedMoE` 在调用时能够识别并传递 `quantization_emulation` 标志。  
3. **后端选择路径**  
   - `oracle/mxfp4.py` 与 `oracle/nvfp4.py` 在枚举 `EMULATION` 后统一返回对应的仿真专家类，保持了原有 `select_*_backend` 逻辑不变。  
   - `quark/quark_moe.py` 中对 `self.emulate` 标志的处理，使得在 **没有原生支持** 时仍能自动切换到仿真后端，而不需要手动改代码。  
4. **函数/接口兼容**  
   - `fused_experts_impl` 中对 `ocp_mx_scheme` 参数的使用被标记为 **已废弃**（抛异常），强制使用专门的专家实现，降低了函数签名的歧义。  
   - `moe_kernel_quantize_input` 新增 `quantization_emulation` 参数，只有在仿真路径才会走 `ref_nvfp4_quant_dequant` 等参考实现，避免在原生路径误用。

#### 性能影响  
- **计算开销**  
  - **即时解量化** 会在每次 forward 前将 `uint8` 权重量化为 `bf16/fp16`，相当于一次额外的 `break_fp4_bytes → float` 与 Scale‑multiply 操作，显著增加算子 FLOPs 与内存带宽使用。  
  - 激活 QDQ（`ref_nvfp4_quant_dequant`）同样会产生两次额外的算子（量化 + 立即反量化），在大 batch / 长序列情况下可能导致 **GPU/CPU 侧的内存占用 >2×**。  
- **实际运行速度**  
  - 预计比原生 NVFP4 / OCP MX（硬件原生 FP4/FP6）慢 **2‑5 倍**，在极端 MoE 配置（大量专家）下甚至更高。  
  - 对于 **ROCm** 环境（AMD Instinct），目前仍使用 `flashinfer` 进行调用（在 `trtllm_nvfp4_moe.py` 中动态 import），因此在 ROCm 平台上可能出现额外的 **库加载延迟**。  
- **内存**  
  - 由于权重在 FP4 表示后仍保留在显存中（`uint8`），解量化产生的中间 FP16/BF16 张量占用额外显存，尤其是 **3‑D MoE 权重**（`[num_experts, M, K/2]`）会产生 `num_experts * M * K` 的临时副本。  

#### 安全考虑  
- **环境变量**：`VLLM_NVFP4_GEMM_BACKEND` 直接决定后端，无校验地传递给内部代码。若在生产环境误设为 `emulation`，将导致意外的性能退化。建议在代码层面加入 **值合法性检查**（仅允许 `flashinfer`, `aiter`, `emulation`）并记录审计日志。  
- **数值安全**：仿真路径使用 `ref_nvfp4_quant_dequant`，在 **极端 scale**（非常小/大的 Global Scale）时可能产生 **数值溢出或下溢**，尤其是对 FP8/FP6 方案的 `block_size` 固定为 16，若模型的激活范围超过 FP8/E4M3 的表示范围，会产生 `inf/NaN`。  
- **代码路径分支**：大量 `if … emulation` 分支若未覆盖所有异常路径，可能导致 **未初始化的变量** 被传递给 Triton kernel，触发运行时错误。

### [MoE Refactor] Combine MoERunnerBase + DefaultMoERunner (#40560)
**SHA**: `809d83c` | 🔗 [查看提交](https://github.com/vllm-project/vllm/commit/809d83c2dc2f74d52fe852b09d0e38c91c7d8334)

**🎯 变更类型**：重构 / 架构变更  
**⚡ 重要程度**：🔴高  

**📋 变更摘要**  
- 将 `MoERunnerBase` 与 `DefaultMoERunner` 合并为单一实现 `MoERunner`，并通过实现新抽象 `MoERunnerInterface` 直接提供 MoE 运行时功能。  
- 删除 `moe_runner_factory` 与 `default_moe_runner.py`，所有创建入口改为直接实例化 `MoERunner`。  
- 对 `layer.py` 的导入路径进行更新，统一使用 `MoERunner` 与 `MoERunnerInterface`。  
- 迁移并保留原有的自定义算子（`moe_forward`、`moe_forward_shared`）注册逻辑与层名解析代码，使其在新类中依旧可用。  

**🎯 影响范围**  
- **核心层**：`vllm/model_executor/layers/fused_moe/runner/*`（`moe_runner.py`、`moe_runner_interface.py`）  
- **MoE 层实现**：`vllm/model_executor/layers/fused_moe/layer.py`（runner 实例化）  
- **工厂相关**：`moe_runner_factory.py`（删除）  
- **依赖方**：任何直接 import `DefaultMoERunner`、`create_moe_runner` 或 `MoERunnerBase` 的内部模块或第三方插件。

## 🔍 技术洞察

### 架构影响
| 维度 | 说明 |
|------|------|
| **类层次** | 通过消除抽象基类 `MoERunnerBase` 与具体实现 `DefaultMoERunner` 的两层结构，直接让 `MoERunner` 继承 `MoERunnerInterface`。<br>简化了继承链，降低了类之间的耦合度，易于维护与扩展（如以后可以在同一文件中实现 `ChunkingMoERunner` 等特化）。 |
| **API 兼容性** | 删除了 `DefaultMoERunner` 与 `create_moe_runner`，对外暴露的构造方式从工厂函数改为 `MoERunner(...)`。这属于 **破坏性变更**，需要迁移代码或提供兼容包装。 |
| **模块依赖** | `layer.py` 现在直接引用 `MoERunner` 与 `MoERunnerInterface`，并删除了原先的 `moe_runner_factory` 导入。<br>其他模块（如调度器、测试）若仍使用旧路径会报 `ImportError`。 |
| **抽象接口** | 新增 `MoERunnerInterface`，保留了 `forward`、`is_internal_router`、`shared_experts` 与 `_replace_quant_method` 抽象方法，为后续不同 runner 实现（Chunking、Hybrid）提供统一签名。 |

### 性能影响
| 维度 | 说明 |
|------|------|
| **运行时开销** | 合并后没有额外的函数层级，实例化与调用路径略有简化（去掉一次 `super().__init__` 的间接层），对性能影响可以忽略不计（< 0.1 %）。 |
| **编译缓存/torch.compile** | 自定义算子注册保持不变；`_select_forward` 仍在 `MoERunner.__init__` 根据平台返回对应的 C++/Python 实现，兼容性保持。 |
| **内存占用** | 类定义本身的内存占用基本不变；删除 `MoERunnerBase` 的类对象可以略微降低模块加载时的元数据开销。 |
| **分布式行为** | 迁移过程中没有改动核心的 `tensor_model_parallel_all_reduce`、`dispatch_router_logits` 等分布式调用，分布式性能保持原状。 |

### 安全考虑
| 维度 | 说明 |
|------|------|
| **攻击面** | 移除抽象基类后对外暴露的公共成员数下降，潜在攻击面小幅收敛。 |
| **自定义算子安全** | `direct_register_custom_op` 的调用方式未变，仍然通过 `torch.compiler.assume_constant_result` 声明结果不随输入变化；不会引入新的安全漏洞。 |
| **输入校验** | 类内部对 `layer_name` 的解析仍保持原来的断言检查 (`assert`)，未降低安全性。 |
| **兼容性风险** | 若外部插件仍旧使用已删除的工厂函数，可能在运行时触发 `ImportError`，导致服务不可用。此类错误容易在启动阶段被捕获，属于兼容性风险而非安全漏洞。 |

## ⚠️ 潜在风险
1. **破坏性 API 变更**  
   - `DefaultMoERunner`、`create_moe_runner`、`MoERunnerBase` 已被删除。任何直接引用这些名称的代码（包括内部单元测试、第三方插件）都会在导入阶段失败。  
2. **导入循环或未更新的引用**  
   - 仍有可能残留 `from ...moe_runner_factory import create_moe_runner` 的旧导入路径，在 CI/部署时触发 `ModuleNotFoundError`。  
3. **类型提示/静态检查失效**  
   - 之前依赖 `MoERunnerBase` 进行 `isinstance(..., MoERunnerBase)` 的判断将不再成立，需要改为 `MoERunnerInterface` 或具体实现。  
4. **层名解析兼容性**  
   - 新实现中 `_resolve_layer_name` 与 `LayerName` 逻辑复制于旧基类，若未来平台对 `LayerName` 做出改变，需同步更新该函数。  
5. **文档/示例未同步**  
   - 官方文档、示例代码中仍可能出现已删除的类/函数，导致用户在阅读文档后遇到运行错误。
