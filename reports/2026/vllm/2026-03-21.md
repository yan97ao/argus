### [MoE Refactor] Mxfp4 oracle rebased (#37128)
**SHA**: `87bd918` | 🔗 [查看提交](https://github.com/vllm-project/vllm/commit/87bd91892f8c63ed9aeb2ad2e701472ead8be84c)

**🎯 变更类型**：功能增强 / 重构（新增 MXFP4 MoE Oracle 与多后端支持，重构原有 MXFP4 实现）  
**⚡ 重要程度**：🔴高  
**📋 变更摘要**  
- 引入 **`oracle/mxfp4.py`**，统一管理 MXFP4 MoE 后端的映射、优先级选择、量化/核函数构造等逻辑，取代原来分散在 `quantization/mxfp4.py` 与 `trtllm_moe.py` 的实现。  
- 新增 **FlashInfer‑TRTLLM、FlashInfer‑CUTLASS、Triton、Marlin、CK (ROCm AITER)** 等五大后端实现，并提供对应 **Monolithic / Modular** Expert 类（`trtllm_mxfp4_moe.py`、`gpt_oss_triton_kernels_moe.py` 中的 MXFP4 专家）。  
- 对 **FusedMoE** 的尺寸对齐、权重格式转换、缓存‑permutations、以及 `skip_forward_padding` 逻辑进行统一处理；在 `layer.py` 中去除旧的 `maybe_roundup_hidden_size` 参数 `is_mxfp4_quant`，改为在 **Oracle** 中统一完成。  
- 移除已废弃的 `trtllm_moe.py`（184 行删除），并相应更新文档、测试、以及 `compressed_tensors` 的量化路径。  
- 兼容性改动：原来的 `Mxfp4Backend` 枚举被 **`Mxfp4MoeBackend`** 替代；`Mxfp4MoEMethod` 改为在构造函数中调用 `select_mxfp4_moe_backend`，并在 `process_weights_after_loading` 中统一完成权重布局转换。

### [ROCm] Enable DeepEP ROCm as all2allbackend for AMD GPUs.  (#34692)
**SHA**: `3982bc2` | 🔗 [查看提交](https://github.com/vllm-project/vllm/commit/3982bc2cd0bd9d633060b22e9ff683d8316a0f82)

**🎯 变更类型**：功能增强（为 AMD ROCm GPU 引入 DeepEP 作为 all‑to‑all 后端）  
**⚡ 重要程度**：🟡 中  
**📋 变更摘要**：  
本次提交在 `vllm` 中加入对 ROCm 平台的专属适配。主要工作包括：  
1. Docker 镜像新增 `liblzma-dev`、`pkg‑config` 依赖，以满足 ROCm 编译需求。  
2. 在 all‑to‑all 通信实现里通过 `vllm.platforms.current_platform` 判断平台，针对 ROCm 移除 `explicitly_destroy`、`allow_nvlink_for_low_latency_mode` 等仅在 CUDA 上生效的参数。  
3. 将所有与 FP8 相关的硬编码 dtype（`torch.float8_e4m3fn`）统一为 `current_platform.fp8_dtype()`，并在 ROCm 上支持 `torch.float8_e4m3fnuz`。  
4. `deepep_ll_prepare_finalize.py` 中对 ROCm 做了参数简化，只保留必需的调用签名。

**🎯 影响范围**  
- `vllm/distributed/device_communicators/all2all.py`（通信层）  
- `vllm/model_executor/layers/fused_moe/*`（MoE 量化、调度）  
- Docker 基础镜像 `docker/Dockerfile.rocm_base`  
- `vllm/platforms`（平台抽象层）  

**💡 关注建议**  
1. **平台检测可靠性**：`current_platform.is_rocm()` 现在决定关键参数是否传递，请确保该函数在非 ROCm 环境始终返回 `False`，避免意外隐藏参数。  
2. **FP8 dtype 兼容**：ROCm 上使用 `float8_e4m3fnuz`，而 CUDA 仍是 `float8_e4m3fn`。建议在文档或 README 中明确说明两者差异及使用限制。  
3. **测试覆盖**：在 AMD GPU（MI250、MI100 等）上跑完整的 unit‑test + inference benchmark，验证 all‑to‑all、low‑latency 模式以及 fp8‑moe 的数值正确性。  
4. **后向兼容**：已有用户可能在 CUDA 环境下依赖 `explicitly_destroy=True` 等参数，确保在 ROCm 之外保持原有行为不变。  
5. **依赖升级**：新增的系统库可能影响镜像体积和构建时间，建议在 CI 中加入镜像大小检查，防止无意膨胀。  

总体来看，改动为 AMD 用户打开了关键的 DeepEP 支持路径，逻辑上通过平台抽象层实现了最小侵入式改写。但仍需在实际硬件上进行充分验证，并在文档中补充 ROCm‑specific 使用说明，以降低迁移风险。

### [Responses API] Add kv_transfer_params for PD disaggregation (#37424)

**🎯 变更类型**：功能增强  
**⚡ 重要程度**：🟡 中  

**📋 变更摘要**  
本次提交在 *Responses* API 中新增 `kv_transfer_params` 字段，用于在 PD（Parameter‑Distributed）拆分部署场景下透传 KV‑Transfer 参数。相关上下文类、请求/响应模型以及生成器均加入该字段的保存、传递与序列化逻辑，并在 `to_sampling_params` 中把它并入 `extra_args`。  

**🎯 影响范围**  
- `vllm/entrypoints/openai/responses/context.py`：所有 `*Context` 类新增 `kv_transfer_params` 成员并在 `append_output` 时写入。  
- `vllm/entrypoints/openai/responses/protocol.py`：`ResponsesRequest` 与 `ResponsesResponse` 增加该字段；`to_sampling_params` 合并到 `extra_args`。  
- `vllm/entrypoints/openai/responses/serving.py`：在返回 `ResponsesResponse` 时把上下文中记录的参数写回。  

**💡 关注建议**  

1. **兼容性**：字段默认 `None`，对老客户端无影响，但序列化时会出现额外字段，建议在 OpenAPI/JSON schema 中标记为可选，防止严格模式报错。  
2. **测试**：补充单元测试，验证在普通请求与 PD‑disaggregation 场景下 `kv_transfer_params` 能完整从 `RequestOutput` 流向最终响应。  
3. **文档**：在 README/API 文档中说明该字段的用途、取值格式以及何时需要使用。  
4. **代码审视**：检查其他路径（如错误分支、流式 `cancel`）是否仍能安全忽略 `kv_transfer_params`，防止空指针或意外覆盖。  
5. **性能**：该字段只在对象之间拷贝，影响可以忽略，但请确认 `extra_args` 合并不会导致不必要的深拷贝。  

整体变更聚焦于参数透传，影响范围局限于 Responses API 的内部数据流，若配套测试和文档到位，可安全上线。

5. **测试**：`tests/test_config.py` 已改为显式调用 `backend.configure_post_pass()`，确保 `repr(config)` 不受后置 Pass 副作用影响。请运行完整测试套件，确认未引入隐藏回归。  

总体而言，此回退使后置 Pass 的初始化回归到原始的“编译时”时点，降低了实例化时的副作用，但需要注意一次性资源的重复配置及并发场景的安全性。

本次提交在前端音频依赖上完成了 **“移除 librosa → 使用 PyAV + resampy”** 的迁移。核心改动包括：

5. **测试更新**：去除对 `librosa.load` 的 mock，改为直接验证返回的 ndarray；新增对 PyAV 重采样的行为检查，调整基准音频 base64 数据等。
