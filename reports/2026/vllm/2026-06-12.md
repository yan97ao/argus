### [KV Connector]: Support KV push from Prefill to Decode node using Nixl KV Connector (#35264)
**SHA**: `88ed636` | 🔗 [查看提交](https://github.com/vllm-project/vllm/commit/88ed63621866d1e4bdaacc560c911f7b8859c53d)

**🎯 变更类型**：功能增强 / 架构变更（新增 KV‑push 模式）  
**⚡ 重要程度**：🔴 高  
**📋 变更摘要**：  
- 为 NIXL KV‑cache 迁移实现 **push‑mode**（WRITE‑based）方案，新增 `NixlPushConnectorScheduler`、`NixlPushConnectorWorker` 以及对应的调度器/工作线程逻辑。  
- 在 connector 层引入 `has_pending_push_work` 接口，使调度器在所有请求结束后仍能驱动引擎循环，等待 Push‑WRITE 完成。  
- 完整的文档、示例、单元测试以及 factory 注册，使得用户可以通过 `KVConnectorFactory` 选择 `NixlPushConnector`。

## 🔍 技术洞察

### 1. 架构影响
| 维度 | 说明 |
|------|------|
| **模块划分** | 通过 **pull** 与 **push** 两套 Scheduler/Worker 实现，实现了“业务层+通信层”解耦：调度器只负责元数据准备，实际 NIXL 读/写都交给专属 Worker。 |
| **线程模型** | 为 Push‑mode 引入 **`nixl-push-writer`** 背景线程，专职处理：<br>① 发送 `PUSH_REG` 通知（D → P）<br>② 匹配 D 注册 ↔ P 已完成块<br>③ 发起 WRITE 传输<br>④ 轮询 `nixl_wrapper.get_new_notifs()` 以转发 HB 与完成通知。<br>这与原来的单线程（worker 与 engine 同步）模型不同，增加了跨线程队列 (`queue.Queue`) 与 `Event` 协调。 |
| **调度器 ↔ Worker 接口** | `build_connector_meta` 现在会填充 `push_registrations`（D→P）和 `push_finished_blocks`（P→D）两个新字段，Worker 通过 `start_load_kv` 把这些信息搬入写线程的 inbox。 |
| **向后兼容** | `NixlConnector` 仍指向 **pull** 实现，保持默认行为不变；新 connector 必须显式配置 `NixlPushConnector`。 |
| **统一状态查询** | 新增 `has_pending_push_work` 在 `KVConnectorBase_V1`、`MultiConnector`、调度器 `has_requests` 中传播，确保引擎在所有请求完成后仍保持运行直至 push 完成。 |

### 2. 性能影响
| 场景 | 预期收益 | 可能的开销 |
|------|-----------|------------|
| **Prefill → Decode** 采用 **push**：P 端在完成 prefill 后直接 `WRITE` KV 到 D，省去 D 端 `READ` 阶段的往返（一次网络 RTT）以及额外的 `nixl_wrapper.send_notif`（`PUSH_REG`）+ `GET_META` 步骤。| **降低端到端延迟**（尤其在跨机房/跨节点的场景）。| 额外的 **writer 线程自旋**（1 ms 轮询）以及 **消息序列化**（msgpack）带来微弱 CPU/内存占用。|
| **并发 TP 规模不匹配**（如 D TP > P TP）：push‑mode 通过 `tp_ratio` 逻辑在写端进行块映射，避免了原先多次 `READ` 的复杂调度。| **提升吞吐**（写一次即可满足多个 D‑TP）。| **映射计算**（`np.asarray` → `get_mapped_blocks`）在极端大块数时可能增加 CPU 负载。|
| **心跳/租约**：push‑mode 同样使用原有 lease 机制，额外的 **PUSH_REG** 需要一次 `send_notif`，成本与现有 `HB` 相当。| - | - |

> 实际基准请在大量并发、不同 TP 配置下对比 *pull* 与 *push* 两种 connector（尤其关注 1 ms 自旋的 CPU 占用）。

### 3. 安全考虑
| 项目 | 潜在风险 | 当前防护 |
|------|----------|----------|
| **跨节点通知** (`PUSH_REG`) 包含 `request_id`、`decode_engine_id`、`decode_host/port`、`remote_*` 信息。若未做身份校验，恶意节点可伪造通知导致错误的 KV 写入或资源泄露。 | 代码中未实现任何 **鉴权/签名** 检查，依赖底层 NIXL 通道的网络安全（通常是内部可信网络）。 | 文档明确该通道为内部 side‑channel，仅在受信任环境部署。若在不可信网络使用，需要在部署层面加固 NIXL 端口防火墙或使用 VPN。 |
| **请求 ID 随机后缀**：匹配时会去掉 8‑hex 随机后缀。若出现异常 `request_id` 格式（如未包含后缀）可能导致误匹配。 | 正则匹配 `-`+8hex 仅在后缀出现时剔除，默认保留原 ID。误匹配概率极低，但在手工构造异常 ID 时可能产生误匹配。 | 使用 `get_base_request_id` 并在匹配前进行严格类型检查（`isinstance(request_id, str)`），并在日志中记录不匹配情况。 |

### [11b/n] Migrate Machete kernels to torch stable ABI (#45304)
**SHA**: `a014ddd` | 🔗 [查看提交](https://github.com/vllm-project/vllm/commit/a014dddbaa67661236a8a7d0dc3d5773d4e0f60a)

**🎯 变更类型**：重构 / 架构变更  
**⚡ 重要程度**：🔴高  

**📋 变更摘要**  
本次提交将 Machete（面向 Hopper GPU 的混合精度 GEMM）相关的 CUDA 核心代码迁移到 PyTorch **stable ABI**（`torch::stable` 与 `torch::headeronly`）实现。主要工作包括：  
1. 使用 `torch::stable::Tensor`、`torch::headeronly::ScalarType` 替换原先的 `at::Tensor`/`at::ScalarType`，并改用 `torch::stable` 的异常检查宏（`STD_TORCH_CHECK*`）。  
2. 将 Machete 生成脚本、编译规则统一移至 `libtorch_stable` 目录，使其仅在满足 CUDA 12 + sm90a 条件时编译。  
3. 基于 stable ABI 重写了 launcher、prepack、kernel wrapper 等文件，并在 `torch_bindings.cpp` 中改为条件编译的 `STABLE_TORCH_LIBRARY_IMPL` 注册。  

预期效果是 **提升库对不同 PyTorch 发行版的兼容性**，并避免因 ATen/CUDA 头文件变动导致的二进制破坏（ABI break），为后续升级与维护提供更稳固的基础。

---

### 🎯 影响范围
- **csrc/libtorch_stable/quantization/machete/** 相关源码（kernel、launcher、prepack、binding）  
- **CMakeLists.txt** 中的 Machete 编译路径与生成脚本逻辑  
- **csrc/cutlass_extensions/vllm_cutlass_library_extension.py**（标记数据类型映射）  
- **torch_bindings.cpp / libtorch_stable/torch_bindings.cpp**（算子注册）  

---

### 🔍 技术洞察

| 维度 | 影响分析 |
|------|----------|
| **架构影响** | - 引入 **stable ABI** 将 Machete 完全脱离 ATen/CUDA 细节，能够在不重新编译 PyTorch 本身的前提下使用二进制兼容的扩展。<br>- 通过 `STABLE_TORCH_LIBRARY_IMPL` 实现的算子注册，只在满足 CUDA 12 + sm90a 时生效，保持主库的轻量化。 |
| **性能影响** | - 代码功能未改变，核心 kernel（Cutlass + Machete）保持原有实现，性能基本不受影响。<br>- 使用 `torch::stable::Tensor` 通过 header‑only 实现的内存分配与 `torch::stable::accelerator::DeviceGuard`，在运行时开销与原 ATen 实现相当，甚至因避免多余的 ATen 依赖而略有提升。 |
| **安全考虑** | - 所有 `TORCH_CHECK` 替换为 `STD_TORCH_CHECK*`，统一异常抛出方式，避免因宏改动导致的未捕获异常。<br>- 迁移后不再直接引用 `ATen/cuda` 头文件，降低因 CUDA runtime ABI 变化产生的安全风险。 |
| **可维护性** | - 统一使用 `torch::stable` 与 `torch::headeronly` 命名空间，代码更具可读性，后续迁移到更高版本 PyTorch 时冲突概率下降。<br>- 生成脚本的 hash 检测逻辑保持不变，避免重复生成，提高构建效率。 |
| **兼容性** | - 需要 **PyTorch ≥ 2.2**（stable ABI 引入的最低版本）和 **CUDA 12.0+**，在低版本环境下会自动跳过 Machete 编译，保持向后兼容。<br>- 对已有使用 `torch.ops._C.machete_*` 的用户，调用签名未变，只是底层实现切换。 |

---

### ⚠️ 潜在风险
1. **版本约束**：若用户使用的 PyTorch 低于 stable ABI 所需最低版本，Machete 将被静默禁用，可能导致预期功能缺失。  
2. **宏替换遗漏**：部分 `TORCH_CHECK` 仍残留在旧文件（如 `csrc/cutlass_extensions/...`），在异常路径上可能出现不一致的行为。  
3. **编译环境差异**：`torch::stable` 依赖 header‑only 实现，某些自定义编译选项（如 `-D_GLIBCXX_USE_CXX11_ABI`）若与 PyTorch 本身不一致，仍可能产生二进制不兼容。  
4. **生成脚本路径变更**：`generate.py` 迁移至 `libtorch_stable`，若外部工具硬编码旧路径，构建会失败。  
5. **运行时类型转换**：`torch::headeronly::toString` 与旧 `at::ScalarType` 的字符串化实现略有差异，可能影响日志或错误信息的下游解析。

---

### 💡 关注建议
- **版本检测**：在项目文档与 CMake 配置中明确声明 “Machete 需要 PyTorch ≥ 2.2、CUDA ≥ 12、GPU ≥ Hopper”。提供回退提示，防止用户在不支持环境下误以为已启用。  
- **宏统一**：跑一次全局搜索，确保所有 `TORCH_CHECK*` 已全部替换为对应的 `STD_TORCH_CHECK*`，避免潜在的异常行为差异。

## ⚠️ 潜在风险

1. **平台兼容性**  
   - `fused_experts_cpu` 依赖 AVX‑512 VNNI（`is_vnni=True`），在不支持 VNNI 的 CPU（如老旧的 Xeon 处理器）会回退到非 VNNI 路径，可能导致性能降至约 30‑50% 的预期。文档需明确最低硬件要求。  

2. **零点处理**  
   - 对称量化时使用固定 `0x77777777` 零点，若第三方模型在导出时使用非对称 zero‑point（如实际 AWQ 零点），但在 CPU 端未正确传递 `w13_zeros / w2_zeros`，会导致显著的数值偏差。当前实现已把 `w13_qzeros`、`w2_qzeros` 传递给 `prepare_int4_moe_layer_for_cpu`，但在某些路径（如 `compressed_tensors`）仍可能漏传，需要在模型转换脚本中加以验证。  

3. **内存占用**  
   - CPU 端保持 **packed** int4 (`int32`) 权重，但在 `fused_experts_cpu` 计算前会进行内部 unpack 到 int8 并再乘以 scale，理论上会在每个 expert 产生临时 `int8` 缓冲区。大模型（E > 64）在高并发情况下可能出现瞬时显存/内存峰值，需在生产环境监控。  

4. **测试覆盖不足**  
   - 单元测试仅覆盖了 4 组维度组合与两种 quant_algo，未覆盖 **group_size ≠ 128**、**非 2‑倍 N**（如奇数维度）以及 **混合精度 bias** 场景。若未来出现非标准模型，可能触发断言或数值错误。  

5. **接口向后兼容**  
   - `apply_monolithic` 新增到 `AutoGPTQConfig`、`AWQMarlinConfig` 中，若外部代码直接调用旧的 `apply`（非 monolithic）而未检测 `is_monolithic`，可能出现 `NotImplementedError`。需要在文档或代码中明确两者的调用约束。

### ⚠️ 潜在风险  

1. **ABI 迁移遗漏**  
   - 某些非 `libtorch_stable` 代码仍可能直接包含旧路径（如 `quantization/marlin/*.cu`），在混编时可能出现链接冲突。   
2. **宏替换差异**  
   - `STD_TORCH_CHECK` 与 `TORCH_CHECK` 在异常信息包装上略有差异，极端情况下可能影响调试信息的可读性。  
3. **CUDA 编译器版本兼容**  
   - 生成脚本根据 `CMAKE_CUDA_COMPILER_VERSION` 判断 arch 列表；若未来 CUDA 13.x 或更高版本的版本号规则变化，可能导致 **ARCH 过滤失效**，导致 kernel 未被编译。  
4. **运行时流/设备管理**  
   - 新增的 `DeviceGuard` 与 `get_current_cuda_stream` 假设每个 Tensor 的 `device_index` 与 `get_device_index()` 同步；若出现跨设备 Tensor（如在同一函数里混用了不同 GPU 的 Tensor），可能导致 **stream 错误**。  
5. **Python 接口兼容**  
   - 虽然 ops 名称未变，但 `torch::stable::Tensor` 与旧 `torch::Tensor` 在 Python 层的包装方式有细微差异，某些用户自定义 C++ 扩展可能仍使用旧类型导致 **类型不匹配**。

总体来看，本次修改有效解决了工具调用参数被截断的致命 bug，且对现有接口的向后兼容性影响有限。仅需留意消费端对新增 `text` 内容块的兼容性以及潜在的异常安全性。
