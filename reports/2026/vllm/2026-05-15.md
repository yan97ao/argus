### [Feat][RL] IPC weight sync optimizations: multigpu support and chunked packed tensors (#37476)
**SHA**: `e0a45f1` | 🔗 [查看提交](https://github.com/vllm-project/vllm/commit/e0a45f1455ce27693f7cbcac262900589e77e89f)

**🎯 变更类型**：功能增强 / 架构变更 / 性能优化 / 安全修复  

**⚡ 重要程度**：🔴高  

**📋 变更摘要**  
- 为 CUDA IPC 权重同步引入 **多 GPU 支持** 与 **打包（chunked）传输**，实现跨 GPU、跨进程的大模型权重同步而不出现完整模型的双倍显存占用。  
- 新增 `IPCTrainerSendWeightsArgs.send_mode`（支持自定义 callable）、`packed` 与 `packed_buffer_size_bytes` 参数；扩展 `IPCWeightTransferUpdateInfo` 支持 `packed`、`tensor_sizes` 与统一的 `ipc_handles` 结构。  
- 实现 **统一的打包/解包工具**（`pack_tensors`、`unpack_tensor`、`packed_ipc_producer`、`packed_ipc_consumer`）并在 NCCL 与 IPC 引擎中切换使用。  
- 文档、示例与测试同步更新，覆盖多 GPU + FSDP、HTTP + Ray、非连续张量等场景。  

**🎯 影响范围**  
- `vllm/distributed/weight_transfer/ipc_engine.py` （核心权重同步实现）  
- `vllm/distributed/weight_transfer/nccl_engine.py`（NCCL 端打包广播）  
- `vllm/distributed/weight_transfer/packed_tensor.py`（全新打包/解包库）  
- 示例：`examples/rl/rlhf_ipc_fsdp_ep.py`、`examples/rl/rlhf_ipc.py`、`examples/rl/rlhf_http_ipc.py`  
- 文档：`docs/training/weight_transfer/ipc.md`  
- 单元/集成测试：`tests/distributed/test_packed_tensor.py`、`tests/distributed/test_weight_transfer.py`  
- 预提交检查脚本 `tools/pre_commit/check_torch_cuda.py`（加入新文件的 CUDA 检查）

### 🔍 技术洞察  

| 维度 | 影响阐述 |
|------|----------|
| **架构影响** | 1. **多 GPU 协同**：在多 GPU（如 FSDP）环境下，所有 Trainer 进程均调用 `trainer_send_weights`，通过一次 **all‑gather** 合并每个 GPU 对应的 IPC handle，确保每个推理进程仅获取本地 GPU 的句柄。<br>2. **打包层**：新层 `PackedChunk`、`PackedIpcChunk` 将若干权重序列化为单一 `uint8` 缓冲区，配合 `packed=true` 可在 **单次 API 调用** 内完成分块传输，显著降低峰值显存。<br>3. **统一 Transport**：`send_mode` 兼容字符串（`ray`/`http`）与自定义 Callable，极大提升在非标准部署（如自研 RPC、消息队列）时的可扩展性。 |
| **性能影响** | - **显存峰值**：原始实现需在 Trainer 与 Inference 两端同时保有全模型，打包 + 分块可把显存需求降至 `packed_buffer_size_bytes`（默认 1 GB）左右。<br>- **吞吐量**：分块后每块均通过 NCCL / IPC 进行并行广播，整体传输时间趋近于 **带宽受限** 而非 **内存复制**。<br>- **额外代价**：All‑gather 所带来的 O(N) 通信（N = GPU 数）以及 `torch.cuda.ipc_collect()` 的同步开销，在单 GPU 场景几乎不变，且可通过 `torch.distributed.is_initialized()` 检测后跳过。 |

| **可维护性** | - 代码划分更清晰：`packed_tensor.py` 完全负责打包/解包；`ipc_engine.py` 与 `nccl_engine.py` 只负责调度。<br>- 添加 `PackedChunk`、`PackedIpcChunk` 数据结构，使函数签名更具表达力，降低后续改动的认知负担。<br>- 通过 `*_all_gather_and_merge_handles` 抽象分布式聚合逻辑，避免在不同发送路径重复实现。 |

### ⚠️ 潜在风险  

1. **Chunk 同步失误**  
   - 在 `packed_ipc_producer` 中必须确保 **消费者已完成拷贝**（如示例中使用 ack 队列）后才继续生成下一个块。若用户直接在 `trainer_send_weights` 里使用 `packed=True` 而没有外部同步机制，可能出现 **数据覆盖**（消费者读取到后续块的内容），导致权重不完整或模型崩溃。  
2. **不匹配的 `packed_buffer_size_bytes`**  
   - 当单个张量大小 > `packed_buffer_size_bytes` 时会抛 `ValueError`。若用户在配置中误设太小，会导致 **完整性错误** 并阻塞训练/推理。  
3. **All‑gather 规模**  
   - 多 GPU 场景下每个 rank 会 `all_gather_object` 完整的句柄列表，句柄数等于 **参数数 × GPU 数**。极端模型（数十万参数）在 64 GPU 环境下有可能产生 **显存/网络负载** 较大，虽在大模型实验中仍可接受，但需监控。  
4. **向后兼容**  
   - 直接构造 `IPCWeightTransferUpdateInfo`（如第三方插件）需要提供 `packed`、`tensor_sizes` 或使用新版字段，否则会触发 `ValueError`。未更新的代码库可能在升级后崩溃。  
5. **安全泄漏**  
   - HTTP 仍需显式开启 `VLLM_ALLOW_INSECURE_SERIALIZATION`，若在生产环境误开启，外部攻击者可能利用 pickle 进行任意代码执行。

### fix: propagate revision/code_revision pins to all artifact boundaries (#42616)
**SHA**: `d26a28a` | 🔗 [查看提交](https://github.com/vllm-project/vllm/commit/d26a28ab033697f55a1414b5b0435de7cd6045b6)

**🎯 变更类型**：Bug修复  
**⚡ 重要程度**：🔴高  
**📋 变更摘要**：修复了模型加载过程中未将 `revision` 与 `code_revision` 传递到所有下载/解析入口的问题。现在无论是 GGUF、Whisper、Kimi‑Audio、Kimi‑K25、Roberta 还是通用 Transformers 解析，都统一使用 `ModelConfig` 中的版本信息，从而保证下载的模型文件、代码和 tokenizer 与用户指定的版本保持一致。  

**🎯 影响范围**：  
- `vllm/model_executor/model_loader/gguf_loader.py`（GGUF 权重下载）  
- `vllm/model_executor/models/kimi_audio.py`（Whisper 子模块）  
- `vllm/model_executor/models/kimi_k25.py`（图片处理器获取）  
- `vllm/model_executor/models/registry.py`（通用 Transformers 解析）  
- `vllm/model_executor/models/roberta.py`（二次权重加载）  
- 单元测试 `tests/models/test_gguf_download.py`  

**🔍 技术洞察**  
- **架构影响**：  
  - 将 `ModelConfig.revision` 与 `ModelConfig.code_revision` 作为统一的版本标识，贯穿模型、权重、tokenizer、image processor 等所有“artifact boundary”。  
  - 通过 `hf_hub_download`、`cached_get_image_processor`、`_try_resolve_transformers` 等统一入口传递，提升了配置的集中管理和可追溯性。  
- **性能影响**：  
  - 仅在下载函数中多传递两个可选参数，CPU/内存开销可忽略。  
  - 对已缓存的文件不会产生额外网络请求，缓存命中率保持不变。  
- **安全考虑**：  
  - 明确的 `revision`/`code_revision` 锁定可以防止因默认使用 `main` 分支导致的供应链攻击或意外升级。  
  - 在使用 `trust_remote_code` 时仍需用户自行评估远程代码的风险，但版本固定有助于审计与复现。  

**⚠️ 潜在风险**  
1. **向后兼容性**：如果调用方显式传入 `None` 并依赖默认行为，新的 `revision=None` 仍兼容，但某些旧版代码可能对 `revision` 参数的存在做了额外检查，导致 `TypeError`。  
2. **网络错误传播**：错误的 `revision`（如不存在的 commit）会导致 `hf_hub_download` 抛出异常，之前可能被隐藏在默认 `None` 情况下的下载成功。需要确保调用方能够捕获并处理此类异常。  
3. **缓存目录变更**：在 GGUF 加载中加入了 `cache_dir=self.load_config.download_dir`，若用户未显式配置 `download_dir`，可能导致缓存落在不同位置，引发磁盘占用或权限问题。  

**💡 关注建议**  
- **开发者**：在新增或修改 `ModelConfig` 时务必同步更新文档，明确 `revision` 与 `code_revision` 的默认值行为。添加单元测试覆盖 `revision=None`、`revision` 为合法 commit、以及不存在的 commit 三种场景。  
- **用户**：升级后请检查配置文件中是否已经填入期望的 `revision`/`code_revision`，尤其在使用 `trust_remote_code=True` 时，更应指定明确的代码版本以降低供应链风险。  
- **CI/测试**：在 CI 中加入对不同模型仓库（公开、私有）以及不同分支/标签的下载兼容性验证，确保新参数不会破坏已有缓存机制。

### [CPU][RISC-V] Add RVV-optimized attention kernels for RISC-V Vector Extension (#40119)
**SHA**: `f351455` | 🔗 [查看提交](https://github.com/vllm-project/vllm/commit/f351455f0f06c17c723341bc6d78c3d4470b5b09)

**🎯 变更类型**：功能增强（新增 RISC‑V Vector Extension（RVV）专用注意力算子）  
**⚡ 重要程度**：🔴 高  

**📋 变更摘要**  
本次提交在 `vllm` 项目中为 CPU 后端实现了基于 RISC‑V Vector Extension（RVV）的注意力计算路径。核心工作包括：  
1. 新增 `cpu_attn_rvv.hpp` 实现 8×8 tile‑gemm 微内核，硬编码 VLEN=128（对应 `vl=8`），并在 C++ 侧加入 `ISA::RVV`。  
2. 扩展平台检测、CMake 编译选项以及 Python 调度逻辑，使得在检测到真实的 RVV‑VL128 硬件时自动选择 `rvv` ISA。  
3. 更新测试套件和基准脚本，加入 `--isa rvv` 参数和对应的参数化测试。  
4. 对相关宏、类型定义、向量实现做了兼容性补丁（如 BF16 宏名、FP8 stub），保证在非 RVV 环境下安全回退到原有 `vec/vec16` 实现。

**🎯 影响范围**  
- **CPU 注意力后端**：`csrc/cpu/*`（`cpu_attn.cpp/impl.hpp`，新增 `cpu_attn_rvv.hpp`），`cpu_arch_macros.h`  
- **构建系统**：`cmake/cpu_extension.cmake`（RVV VLEN 检测与编译宏）  
- **Python 调度层**：`vllm/v1/attention/backends/cpu_attn.py`（ISA 选择、RVV 支持检测）  
- **测试与基准**：`tests/kernels/attention/test_cpu_attn.py`、`benchmarks/kernels/cpu/benchmark_cpu_attn.py`  
- **平台枚举**：`CpuArchEnum.RISCV`（在 `cpu_attn.py` 中加入）

### 🔍 技术洞察

| 维度 | 影响分析 |
|------|----------|
| **架构影响** | - 在 CPU 注意力调度器中加入 `ISA::RVV`，实现了 RISC‑V 向量指令路径的完整闭环。<br>- 通过 CMake 条件 (`__riscv_v_min_vlen == 128`) 将 RVV 源码仅在编译主机支持 VLEN=128 时加入，防止在非 RVV 或 VLEN>128 环境出现编译错误。<br>- `cpu_attn_rvv.hpp` 使用 `riscv_vector.h` 的固定‑VL typedefs，保持与现有 `cpu_attn_vec.hpp` 接口兼容，统一 `TileGemm` 抽象层。 |

1. **VLEN 硬编码风险**  
   - 只在 VLEN=128 时编译 RVV 路径；如果用户在 VLEN=256（或更高）硬件上运行且 `cpu_attn_rvv.hpp` 被排除，调度器仍可能尝试选择 `rvv`（若 Python 判定错误），导致 `TORCH_CHECK` 失败。

5. **浮点精度差异**  
   - 对于 BF16 与 FP16，使用了 `vfwcvt` 或手动位移实现，兼容性已在代码中处理，但在不支持 `zvfh`/`zvfbfmin` 的 RVV 实现上会走标量路径，可能出现性能回退且不易在基准中直接对比。
