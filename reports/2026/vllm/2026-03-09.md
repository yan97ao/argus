### Reapply [Attention] Refactor `check_and_update_config` (#35122)
**SHA**: `77a7345` | 🔗 [查看提交](https://github.com/vllm-project/vllm/commit/77a73458e3ae8b5b7a2a13f78d3a6b4d39b1414d)

**🎯 变更类型**：重构 / 架构变更  

**⚡ 重要程度**：🔴高  

**📋 变更摘要**  
- 将 **KV‑cache block size** 的默认和校验逻辑从各注意力实现、平台 (`cpu/cuda/rocm/xpu`) 中抽离，统一到 `CacheConfig` 与平台统一的 `update_block_size_for_backend` 接口。  
- `CacheConfig` 新增 `DEFAULT_BLOCK_SIZE`、`user_specified_block_size`、`_block_size_resolved` 标记，并在模型加载后统一完成默认值填充。  
- `get_attn_backend`、`AttentionBackend` 接口签名去掉了 `block_size` 参数，改为在平台层根据后端自行决定。  
- 新增 `VllmConfig.validate_block_size` 与平台 `update_block_size_for_backend`，在模型加载、执行器初始化、以及多进程/Ray 启动路径中统一调用。  
- 大量测试代码同步改为显式构造 `CacheConfig(block_size=…)`，确保覆盖新路径。  

**🎯 影响范围**  
- `vllm/config/cache.py`、`vllm/config/vllm.py`（配置与校验）  
- 各平台实现：`cpu.py`、`cuda.py`、`rocm.py`、`xpu.py`（平台级 block size 处理）  
- 注意力后端选择：`vllm/v1/attention/selector.py`、`backend.py` 以及所有 `*attention*.py` 实现（去除 block size 参数）  
- 执行器入口：`vllm/v1/executor/*_executor.py`、`vllm/v1/worker/gpu_model_runner.py`（首次加载后重新校准 block size）  
- 关键运行时：`vllm/v1/engine/core.py`（KV‑cache 初始化后校验）  
- 单元测试文件：`tests/kernels/attention/*`、`tests/models/multimodal/*`、`tests/v1/spec_decode/*` 等  

---

### 🔍 技术洞察  

| 维度 | 影响分析 |
|------|----------|
| **架构影响** | 1. **配置统一化**：`CacheConfig` 成为唯一负责 block size 的来源，避免各后端、平台散落的硬编码。<br>2. **后端自适应**：后端实现现在通过 `AttentionBackend.get_preferred_block_size` 返回兼容的大小，平台在 `update_block_size_for_backend` 中统一写入，提升可扩展性（新增后端只需实现该方法）。<br>3. **验证入口**：`VllmConfig.validate_block_size` 在 KV‑cache 初始化后统一检查 DCP 与 Mamba 约束，防止在模型加载后出现不兼容的 block size。 |
| **性能影响** | - **KV‑cache block size** 直接决定每次 KV‑cache 分配的粒度，进而影响显存利用率与缓存命中率。新逻辑在未显式指定 `--block-size` 时，会自动选取后端推荐的最小兼容尺寸（如 FlashMLA → 64、CUTLASS_MLA → 128），确保后端能够发挥最佳吞吐。<br>- 对已有显式 `--block-size` 的用户，仍保留原值，但在不兼容的情况下会通过 `validate_block_size` 抛异常或在平台层打印警告，防止性能退化。 |
| **安全考虑** | 变更仅涉及配置与性能调优，无外部输入直接导致代码执行路径改变，未引入新安全风险。但需确保 **默认值写入** 过程不被恶意配置覆盖（如 `CacheConfig.block_size` 通过环境变量或 CLI），仍然保持 **Pydantic** 的类型校验。 |
| **可维护性** | - 参数列表简化（`get_attn_backend` 不再接受 `block_size`），降低函数签名分散的风险。<br>- 所有对 `block_size` 的修改集中在 `CacheConfig` 与平台的 `update_block_size_for_backend`，新增后端或平台时只需关注这两处。<br>- 引入 `user_specified_block_size` 标记，防止平台误覆盖用户显式提供的值，提升可读性与调试体验。 |
| **兼容性** | - 对外 CLI `--block-size` 行为保持不变（仍能覆盖），只在内部统一使用 `CacheConfig.block_size`。<br>- 旧代码仍可在未升级的分支中工作（因为 `CacheConfig.block_size` 仍可为 `None`，平台会补齐默认），但在新版中强制调用 `validate_block_size`，有可能在旧模型/自定义后端上触发断言，需要更新自定义实现以实现 `get_preferred_block_size` 或 `supports_block_size`。 |

---

### ⚠️ 潜在风险  

1. **未覆盖的自定义后端**  
   - 若用户自行实现 `AttentionBackend` 而未实现 `get_preferred_block_size`，平台可能仍使用默认 `16`，导致不兼容的 **block size**（如需要 32‑aligned）。会在 `validate_block_size` 触发断言。  

2. **并发更新**  
   - `update_block_size_for_backend` 在多进程/Ray/Multiproc 启动路径多次调用。虽然只写一次最终值，但若不同 worker 依据不同后端获得不同推荐尺寸，可能出现 **不一致的 block size** 导致 KV‑cache 组划分错误。  

3. **DCP 与 Mamba 约束**  
   - `validate_block_size` 现在在模型加载后统一检查，但原来在 `cpu.check_and_update_config` 中已有类似检查。若平台在 `check_and_update_config` 里提前写入不兼容的默认值（例如 `cpu` 仍在 `check_and_update_config` 中设为 `128`），后续 `validate_block_size` 仍会报错。需要确保各平台只在 `update_block_size_for_backend` 中写入，或在 `validate_block_size` 前清除冲突。  

4. **测试覆盖不足**  
   - 目前只在单元测试中补齐了 `CacheConfig` 的创建，缺少 **多 KV‑cache 组**、**DCP**、**Mamba align** 场景的集成测试，可能在真实分布式部署中暴露错误。  

5. **日志噪音**  
   - 新增的 `logger.warning`（当用户指定 block size 导致降级后端）在大规模集群中可能产生大量重复日志，需确保 `logger.warning_once` 或相似机制被使用。  

---

### 💡 关注建议  

1. **完善后端实现**  
   - 为所有官方后端实现 `AttentionBackend.get_preferred_block_size`，并在文档中注明返回值的含义。对外部自定义后端提供基类实现的最小范例。  

2. **统一调用点**  
   - 确保所有平台的 `check_and_update_config` **不再直接写入** `cache_config.block_size`，而是仅调用 `update_block_size_for_backend`。可以在 `cpu/cuda/rocm/xpu` 中加入 `# TODO: 已迁移` 注释并删除残余赋值。  

3. **同步测试**  
   - 增加以下测试：  
   - 多 KV‑cache 组下的 `block_size` 自动取最小值。  
   - DCP

### [Frontend][2/n] Improve pooling entrypoints | embed. (#36110)
**SHA**: `fff3711` | 🔗 [查看提交](https://github.com/vllm-project/vllm/commit/fff3711a244dd9e2915323e31c20768d922e90b5)

**🎯 变更类型**：功能增强 / 重构 / 架构变更  
**⚡ 重要程度**：🔴 高  

**📋 变更摘要**  
- 将 `openai_serving_*` 状态对象统一改为 `serving_*`，并在 `init_pooling_state` 中统一初始化，消除旧的 OpenAI‑specific 命名。  
- 重构 `PoolingIOProcessor` 基类，明确在线/离线前后处理接口，去除不再使用的线程池，新增 `name` 字段用于任务标识。  
- 为 **Embedding** 任务实现全新 `EmbedIOProcessor`，加入 **长文本分块聚合（chunked processing）**，在在线路径上实现分块编码、加权平均聚合，显著降低单次超长输入的显存占用。  
- 精简 OpenAI‑style serving 类：`vllm.entrypoints.openai.engine.serving` 中的 embedding 逻辑被迁移至独立的 `PoolingServing` 实现，返回统一的 `Response`（JSON/Streaming），并使用 `create_error_response` 统一错误处理。  
- 新增 `get_json_response_cls` 缓存函数，优先使用 `orjson` 加速响应。  
- 扩展 `PoolingServeContext` 为通用 dataclass，加入 `engine_prompts`、`prompt_request_ids`、`intermediates` 等用于分块处理的状态字段。  
- 统一错误类型，引入 `VLLMNotFoundError`，在 `utils.create_error_response` 中映射为 HTTP 404。  

**🎯 影响范围**  
- **vllm/entrypoints/pooling/**：所有 pooling 子模块（embed、classify、score、pooling 本身）以及公共 IOProcessor、ServeContext、typing、utils。  
- **vllm/entrypoints/openai/**：`engine/serving.py`、`run_batch.py`、`openai/api_router` 等涉及 embedding / score / rerank 的路由与调度。  
- **tests/**：仅涉及 `tests/entrypoints/pooling/score/test_online_score_vision.py`（调整后端列表）。  
- **核心模型交互层**：`EngineClient.encode` 调用路径受到新分块 ID 与聚合逻辑的影响。  

**🔍 技术洞察**  

| 维度 | 影响 |
|------|------|
| **架构影响** | 1. **统一状态命名**：`serving_*` 取代 `openai_serving_*`，降低模块耦合度，后续新增非‑OpenAI 入口（如自研 HTTP、gRPC）更为简洁。<br>2. **抽象化 IOProcessor**：基类只保留通用前后处理框架，子类通过 `name` 标识任务，插件式扩展更易维护。<br>3. **PoolingServeContext** 从单一 `ServeContext` 演进为支持 `engine_prompts`、`prompt_request_ids`、`intermediates`，为分块聚合提供必要上下文。 |
| **性能影响** | 1. **分块嵌入**：长文本切分为模型最大长度块，逐块推理后加权平均，显著降低单次显存峰值，允许超过 `max_model_len` 的输入。<br>2. **在线聚合**：在 `post_process_online` 中完成聚合，避免把全部块结果全部留在内存中再统一处理，提升吞吐和响应延迟。<br>3. **去除 ThreadPoolExecutor**：IOProcessor 不再在后台线程中执行 tokenization，改为同步调用（tokenizer 本身已经是 C++ 实现），降低线程切换开销。<br>4. **JSON 序列化加速**：`get_json_response_cls` 引入 `orjson`（若可用），可把原本 2–3 ms 的响应序列化降至 <1 ms。 |
| **安全考虑** | - 新增 `VLLMNotFoundError` 并在统一错误生成器中映射为 404，避免原来返回 200+错误信息泄露模型内部状态。<br>- `EmbeddingRequest` 仍然通过统一的 `create_error_response` 进行异常捕获，确保异常不泄露堆栈。<br>- 代码路径对 `trust_request_chat_template` 仍保持原有检查，无新增安全风险。 |
| **可维护性** | - 统一命名和模块划分后，代码搜索/审计更直接，新增任务只需实现对应 `IOProcessor` 并在 `init_pooling_io_processors` 注册即可。<br>- `PoolingServeContext` 使用 `pydantic.ConfigDict(arbitrary_types_allowed=True)`，兼容 FastAPI 自动验证，降低手动类型检查工作。 |
| **兼容性** | - 对外 API（/v1/embeddings、/v1/score 等）保持不变；仅内部实现和状态字段调整。<br>- 仍然兼容旧的 `EmbeddingChatRequest`/`EmbeddingCompletionRequest`，但在 `tests` 中把 `ATTN_BACKENDS` 默认改为 `["auto"]`，防止在非 ROCm 环境意外报错。 |

**⚠️ 潜在风险**  
1. **分块聚合逻辑错误**：加权平均的实现依赖 `prompt_token_ids` 正确返回；若某些模型返回 `None`（如 Encoder‑Decoder），会触发 `NotImplementedError`，导致 API 500。  
2. **请求 ID 冲突**：分块时生成的 `prompt-request-id` 使用 `"{req_id}-prompt-{i}-chunk-{c}"`，如果业务侧自行拼接了相同前缀可能出现冲突，影响日志追踪。  
3. **并发处理**：`engine_prompts` 与 `prompt_request_ids` 在 `PoolingServing.__call__` 中是列表共享，同步修改（如在 `pre_process_online` 中替换为分块列表）需确保没有并发竞争；FastAPI 默认单请求单协程，风险不大，但在自定义并发执行器时需注意。  
4. **错误包装**：`run_batch.py` 中对 `serving_engine_func` 包装改为 `create_error_response(e)`，如果原函数已经返回 `ErrorResponse`（自定义错误码），会被再次包装导致错误结构不一致。  
5. **移除 `ThreadPoolExecutor`**：如果未来开启了自定义的异步 tokenizer（如 torch‑async），缺少线程池可能出现阻塞。  
6. **JSON 响应类缓存**：`orjson` 不可用时仍会返回普通 `JSONResponse`，但原有的 `logger.warning_once` 只在首次触发，若后续切换环境安装了 `orjson`，缓存仍返回旧的 `JSONResponse`，需要重启进程才能生效。  

**💡 关注建议**  
- **分块处理回归**：编写专门的单元/集成测试覆盖 `EmbedIOProcessor` 的分块路径，包括极长文本、空文本、以及 `prompt_token_ids` 为 `None` 的模型。  
- **日志审计**：在 `post_process_online` 完成聚合后，记录每个原始请求的 `prompt_token_ids` 总长度、块数、聚合耗时，以便后续性能监控。  
- **异常统一**：在 `run_batch.py` 中区分 `ErrorResponse` 与普通异常，避免二次包装；可改为 `if isinstance(e, ErrorResponse): response = e else: response = create_error_response(e)`。  
- **文档更新**：在 API 文档中说明 `embedding` 支持 `enable_chunked_processing`（默认从模型配置读取），并标明最大输入长度仍受 `max_embed_len` 或 `max_model_len` 限制。

### feat(attention): extract KV-cache update from FlexAttention backend (#36263)
**SHA**: `7474310` | 🔗 [查看提交](https://github.com/vllm-project/vllm/commit/747431044df6b15c7b359b5720cc7368c662c232)

**🎯 变更类型**：功能增强  

**⚡ 重要程度**：🔴高  

**📋 变更摘要**：  
- 在 `FlexAttentionBackend` 中将 KV‑cache 的写入逻辑抽取为独立的 `do_kv_cache_update` 方法，并通过 `forward_includes_kv_cache_update = False` 表明 `forward` 不再负责缓存写入。  
- `forward` 只保留注意力计算及缓存读取的流程，KV‑cache 写入仅在显式调用 `do_kv_cache_update` 时执行。  

**🎯 影响范围**：  
- `vllm/v1/attention/backends/flex_attention.py`（核心注意力后端）  
- 可能波及使用 `FlexAttentionBackend.forward` 的 decoder 代码路径（如 `vllm/v1/modeling/xxx.py` 中的层实现）以及测试套件。  

**🔍 技术洞察**：  
- **架构影响**：  
  - 将“计算”和“缓存写入”职责分离，使后端实现更符合单一职责原则。  
  - 引入 `forward_includes_kv_cache_update` 标记，为未来支持 **只读** 前向或 **异步缓存更新** 提供统一入口。  
  - 需要在上层调用栈（例如 `TransformerLayer`）显式调用 `do_kv_cache_update`，否则 KV‑cache 将不再被更新。  

- **性能影响**：  
  - **正面**：在只需要 “查询” 而不写入 KV‑cache（如在推理中做 prefix‑cache 复用、实验性解耦或梯度检查）时，可省去 `reshape_and_cache_flash` 调用，降低 CUDA kernel 调用次数，提升吞吐。  
  - **潜在负面**：若上层忘记调用 `do_kv_cache_update`，会导致 KV‑cache 不被刷新，进而引起重复计算或生成质量下降，间接影响整体性能。  

- **安全考虑**：  
  - 代码路径变更后，KV‑cache 写入的完整性依赖调用者的正确性，属于 **功能完整性风险**（而非安全漏洞）。  
  - `torch.ops._C_cache_ops.reshape_and_cache_flash` 本身仍然保持原有的安全检查。  

**⚠️ 潜在风险**：  
1. **功能回退**：上层代码若未同步更新以调用 `do_kv_cache_update`，decoder 的 KV‑cache 将不再更新，导致生成错误或显著性能下降。  
2. **兼容性**：已有的自定义后端或插件可能假设 `forward` 包含缓存写入，升级后会出现不兼容行为。  
3. **并发同步**：如果 KV‑cache 更新被拆分到不同的调用点，需要确保 **slot_mapping** 与缓存的时序一致，防止 race 条件（尤其在多线程/多进程推理场景）。  

**💡 关注建议**：  
- **代码审查**：在所有使用 `FlexAttentionBackend` 的地方（Transformer 层、pipeline parallel 等）确认已经在 forward 之后或适当位置调用 `do_kv_cache_update`。  
- **单元/集成测试**：新增或更新测试，验证在 decoder 模式下 KV‑cache 正确写入，并对比旧实现的输出一致性。  
- **文档/注释**：在 `FlexAttentionBackend` 类和 `forward` 方法的 docstring 中明确说明 `forward_includes_kv_cache_update = False` 以及何时应调用 `do_kv_cache_update`。  
- **回退机制**：可考虑在 `forward` 中保留可选开关（如 `include_kv_cache_update: bool = True`），以免在迁移期间出现突发错误。  
- **监控报警**：在运行时加入 KV‑cache “未更新” 的监控（例如统计 `slot_mapping` 是否被消费），及时捕获漏调用导致的异常。

- **核心层**：`vllm.model_executor.layers.fla.ops` 新增 API，原有 `fused_recurrent_gated_delta_rule` 仍保留。

**关键实现细节**  
1. **Heuristics 与特性开关**：通过 `@triton.heuristics` 自动推导是否使用初始状态、变长序列、连续批处理、Spec‑Decoding 等，保持与旧 kernel 接口一致。

**潜在风险**  
- **数值误差**：kernel 中对 `softplus` 采用近似分支，测试容忍 `1e‑2` 的 atol/rtol；在极端 `beta`、`x` 范围下可能出现更大误差。建议在生产中对关键模型进行额外的 FP16/ BF16 验证。

- **API 变更**：`fused_sigmoid_gating_delta_rule_update` 替换了原 kernel 的调用方式，外部如果直接引用旧函数并手动传入 `g、beta` 会失编译。文档需明确新旧接口差异及迁移指南。  
- **状态索引**：在 Spec‑Decoding 场景通过 `num_accepted_tokens` 计算 `i_t`，但若 `num_accepted_tokens` 为 0 会产生负索引；目前代码在 `if IS_SPEC_DECODING` 分支里已减 1，需确保 `num_accepted_tokens >= 1`（已有断言）。  

**建议**  
1. **完善文档**：在 `fla/ops/README` 中说明新 kernel 的参数意义、默认值以及何时应使用。  
2. **回退路径**：提供一个轻量的包装函数，当 Triton 编译失败或 GPU 不满足要求时自动回退到 `fused_recurrent_gated_delta_rule`。  
3. **数值容差**：在 CI 中加入对 BF16/FP16 精度的 stricter 对比（如 `rtol=5e-3`），防止回归。  
4. **性能基准**：加入 micro‑benchmark（decode、spec‑decode 两种场景）比较新旧 kernel 的吞吐和延迟，确保新实现的 “fuse” 真正带来提升。  

总体而言，此次改动在保持功能等价的前提下，将 gating 与递归更新合并为单个 Triton kernel，能够显著减少显存搬迁和 kernel 调用开销，对 Qwen‑3‑Next 的解码性能预计有正面提升。只要做好兼容性检查与数值验证，风险可控。

1. **兼容性**：合并空分区后，原有的 “空子模块” 可能被外部代码通过子模块名称访问，需确认不存在此类硬编码依赖。

4. **文档更新**：在开发者文档或 inline comment 中说明 “空分配子图会被自动合并”，帮助使用者理解划分结果的变化。  

总体而言，此改动稳妥地解决了 CUDA‑Graph 警告，影响范围局限在图划分实现，风险较低，仅需关注向后兼容与文档同步。
