### Refactor Arctic loading to use AutoWeightsLoader (#38955)
**SHA**: `93726b2` | 🔗 [查看提交](https://github.com/vllm-project/vllm/commit/93726b2a1c766a27f10ad7c3f709ffbacb94bcf9)

**🎯 变更类型**：重构（引入 `AutoWeightsLoader` 统一权重加载逻辑）  
**⚡ 重要程度**：🟡 中  

**📋 变更摘要**  
- 将 Arctic 模型的权重加载实现从手写的 `load_weights` 重构为使用新抽象 `AutoWeightsLoader`，并在构造函数中加入 `self.config`、去除冗余日志。  
- 同时把原本在文件底部的 `ArcticForCausalLM` 类重新组织，保留原有前向、logits 以及嵌入接口，只是把实际的权重量化/切片工作交由 `AutoWeightsLoader` 处理。  
- 对 MoE 与 Residual‑MLP 的层划分逻辑作了更清晰的判断（依据 `moe_layer_frequency` 与 `use_residual`），并统一了 `mlp_params_mapping` 与 `expert_params_mapping` 的生成方式。  

**🎯 影响范围**  
- `vllm/model_executor/models/arctic.py`（核心模型实现、权重加载、`ArcticForCausalLM`）  
- 依赖 `AutoWeightsLoader` 的通用权重加载框架（间接影响所有使用该基类的模型）  

**💡 关注建议**  
1. **兼容性**：`AutoWeightsLoader` 会自动跳过 `lm_head`（在 tie‑word‑embeddings 情况下），确保旧版 checkpoint 仍可加载。若有自定义前缀或特殊权重命名，建议在调用方显式提供 `skip_prefixes`。  
2. **性能**：新加载路径保留了原来的分片‑加载逻辑，理论上不应出现性能回退；但建议在首次大模型（16‑bit）加载时对比前后耗时，确认日志信息移除未导致误解。  
3. **测试**：新增类后，需要在模型单元测试中覆盖 `load_weights` 的两种分支：MoE 层与 Residual‑MLP 层。特别关注 `self.config.moe_layer_frequency`、`self.config.use_residual` 的组合。  
4. **文档**：更新模型文档，说明权重加载已由 `AutoWeightsLoader` 完成，用户如需自定义权重切片方式，可参考 `vllm/model_executor/utils.py` 中的实现。  

总体来看，此次重构提升了代码可维护性与一致性，对现有功能无功能性回退风险。只要在 CI 中加入权重加载的覆盖测试，即可放心合并。

### [Bugfix] Fix DSV32 weight loading (#38870)
**SHA**: `8617f86` | 🔗 [查看提交](https://github.com/vllm-project/vllm/commit/8617f8676b5ae936382ea00fa92693f59fbb9d69)

**🎯 变更类型**：Bug 修复  
**⚡ 重要程度**：🟡 中  

**📋 变更摘要**  
本次提交针对 DeepSeek（MTP 与 V2）模型在使用 ModelOpt‑FP4 量化权重（DSV32）时的加载和前向逻辑做了修正。  
1. 在模型构造阶段记录 `quant_config` 并通过 `is_fp4_ckpt` 标记是否为 FP4 检查点。  
2. 对于 FP4 检查点，恢复原有的 **wk + weights_proj 融合层**（`MergedColumnParallelLinear`），并在权重映射表中加入对应的 fused‑mapping。  
3. 非 FP4 场景则继续使用拆分的 `ReplicatedLinear` 实现，以保持原有量化路径。  

**🎯 影响范围**  
- `vllm/model_executor/models/deepseek_mtp.py`  
- `vllm/model_executor/models/deepseek_v2.py`  
- 相关的权重加载逻辑与前向计算（`load_weights`、`forward`）  

**💡 关注建议**  
- **测试覆盖**：确保在含 FP4 权重量化的 DeepSeek checkpoint 与普通 checkpoint 上分别跑通权重加载、推理以及多卡并行（TP）场景。  
- **兼容性检查**：其他量化方案（如 GPTQ、AWQ）仍使用拆分实现，需验证新分支不影响它们的行为。  
- **文档更新**：说明在 `quant_config.get_name() == "modelopt_fp4"` 时模型会自动切换到 fused wk/weights_proj，用户可不必手动指定。  
- **性能监控**：对比 FP4 fused 版与非 fused 版的吞吐与显存占用，确认优化效果符合预期。  

总体而言，此次修改恢复了 FP4 权重的正确加载与推理，风险集中在条件判断错误或未覆盖的量化组合上，建议在 CI 中加入 FP4 检查点的回归测试。

### Remove MQ multi-node tests (#38934)
**SHA**: `ab79863` | 🔗 [查看提交](https://github.com/vllm-project/vllm/commit/ab79863e6c4f4df652328af6901be2ee208dacec)

**🎯 变更类型**：其他（测试/CI 调整）  
**⚡ 重要程度**：🟡 中  

**📋 变更摘要**  
- 在 CI 配置 `.buildkite/test_areas/distributed.yaml` 中移除了 “MessageQueue TCP Multi‑Node” 步骤。  
- 同时删除了对应的多节点集成测试 `tests/distributed/test_mq_tcp_multinode.py`（119 行）。  

**🎯 影响范围**  
- `vllm/distributed/device_communicators/shm_broadcast.py`、`vllm/distributed/parallel_state.py`（间接受影响的核心模块）。  
- CI Pipeline 中的分布式多节点覆盖率下降。  

**💡 关注建议**  
1. **测试覆盖**：该测试验证了跨节点时 MessageQueue 自动回退到 ZMQ‑TCP 的行为以及大数据块的切分与传输。删除后，相关回退路径将失去 CI 保障，建议在其他位置补充单元测试或在本地手动验证。  
2. **文档同步**：若此特性已不再维护或已被其他实现取代，需要在 README / docs 中更新说明，避免用户误以为多节点 MQ 已经被 CI 验证。  
3. **回归风险**：后续对 `MessageQueue`、`in_the_same_node_as` 或进程组初始化的改动，可能重新引入跨节点错误而不被检测。可以在现有的 `distributed/test_multi_node_assignment.py` 或 `test_pipeline_parallel.py` 中加入简化的回退检查，以保持最低的保护。  
4. **CI 维护**：确保删除的步骤在 Buildkite UI 中不再显示，避免误报“可选”但缺失的步骤。  

总体来说，此次改动仅影响测试层面，不会直接修改运行时逻辑。但削减了对跨节点 TCP 回退路径的自动验证，建议通过补充更轻量的单元/集成测试或在文档中明确说明，以降低未来回归的风险。
