### [GDN] GDN Prefill kernel for SM100 (#43273)
**SHA**: `d56612c` | 🔗 [查看提交](https://github.com/vllm-project/vllm/commit/d56612c6210605aaca1ce8fa09c40bbe30ee3973)

**🎯 变更类型**：功能增强 / 性能优化 / 架构变更  

**⚡ 重要程度**：🔴 高  

**📋 变更摘要**  
1. 为 Blackwell（SM100）GPU 引入全新 **CuteDSL** 实现的 Gated‑Delta‑Net（GDN）prefill kernel，提供与 FlashInfer 在 Hopper（SM90）相同的性能。  
2. 在 `EngineArgs` 与 CLI 中加入 `--gdn-prefill-backend` 选项，支持 `cutedsl` 后端并在不存在兼容硬件时安全回退到 Triton/FLA。  
3. 新增 `vllm/cute_utils` 包，封装 Cutlass / CuteDSL 常用 DSL 操作、低层汇编指令、TMA‑copy 辅助函数以及类型转换工具（`cvt`），并将原有 `vllm/cute_utils/cvt.py` 重构为统一入口。  
4. 完整实现 **CuteDSL** 版的三大 kernel：  
   - `kernel_h.py`（Chunk‑wise H / V‑new 计算）  
   - `kernel_kkt_inv_uw.py`（Chunk‑wise KKT‑inverse + U/W 预处理）  
   - `kernel_o.py`（Chunk‑wise 输出 O）  
   以及元数据准备 (`prepare_metadata_cutedsl`) 与 Python 包装 `chunk_gated_delta_rule_cutedsl`。  
5. 为新实现提供单元测试 `tests/kernels/mamba/test_gdn_prefill_cutedsl.py`，验证与现有 FLO‑based 实现的数值误差在合理范围内。  
6. 将 DeepSeek‑V4 中若干 ops 的内部实现迁移到统一的 `vllm.cute_utils`，统一汇编实现并删除冗余代码。  
7. 在 `vllm/v1/attention/backends/gdn_attn.py` 中根据用户选择和硬件能力动态决定是否使用 CuteDSL 元数据生成路径。  

**🎯 影响范围**  
- **核心层**：`vllm/model_executor/layers/mamba/ops/gdn_chunk_cutedsl/`（全部 kernel 与包装）  
- **GDN 线性注意力实现**：`vllm/model_executor/layers/mamba/gdn/qwen_gdn_linear_attn.py`（后端解析与调度）  
- **运行时配置**：`vllm/engine/arg_utils.py`、`vllm/v1/attention/backends/gdn_attn.py`（CLI 与后端分配）  
- **通用工具**：`vllm/cute_utils/`（DSL 辅助、汇编转换、TMA 复制）  
- **DeepSeek‑V4 ops**：`vllm/models/deepseek_v4/nvidia/ops/*`（统一使用 `vllm.cute_utils`）  
- **测试套件**：`tests/kernels/mamba/test_gdn_prefill_cutedsl.py`

### [LoRA] Add one shot triton kernel For MoE LoRA (#42290)
**SHA**: `ec5de7f` | 🔗 [查看提交](https://github.com/vllm-project/vllm/commit/ec5de7fa7d1303051ccd6c7316171b6d70bac4b4)

**🎯 变更类型**：功能增强 / 性能优化（新增 One‑Shot 融合 MoE‑LoRA kernel、双流并行执行、测试覆盖以及配置入口）  
**⚡ 重要程度**：🔴高  

**📋 变更摘要**  
1. 在 `vllm.lora.ops.triton_ops` 中实现 **fused_moe_lora_one_shot**（shrink + expand 合并为单个 Triton kernel），并在适当场景下自动使用该路径；  
2. 为小批量（decode）场景提供 **fused_moe_lora_small_batch** 变体，进一步降低启动开销；  
3. 引入环境变量 `VLLM_LORA_ENABLE_DUAL_STREAM`，在 LoRA 与基座 GEMM 之间开启 **双 CUDA 流** 并利用事件同步，实现计算与 LoRA 增量的并行；  
4. 扩展 LoRA 相关层 (`BaseLinearLayerWithLoRA`, `FusedMoEWithLoRA`) 以及 LoRA‑context、Punica wrapper 等，支持 aux stream 与 events 参数；  
5. 大量单元测试补齐小‑rank、corner‑case、错误‑block‑size 检查，提升代码鲁棒性；  
6. 新增 benchmark 脚本用于对比 one‑shot 与传统两‑kernel 实现的性能。

### 🔍 技术洞察

| 维度 | 影响分析 |
|------|----------|
| **架构影响** | • 将原先 **两段 kernel**（shrink → expand）合并为 **单段 one‑shot**，减少 kernel 启动、参数拷贝以及全局内存写入次数。<br>• 引入 **双流**（aux stream）后，LoRA 增量写入与基座 GEMM 分离，理论上能够在同一个 SM 上并行执行，两段计算的时间重叠。<br>• 新的 `MoELoRAContext` 增加 `aux_stream` 与 `events`，对现有 MoE‑LoRA 调度逻辑做了兼容性改造，保持向后兼容（默认关闭）。 |
| **性能影响** | • **启动开销**：仅一次 kernel launch vs 两次，尤其在 **prefill**（大 M、少 N）阶段可降低 10%‑30% latency（实测在 H100 上约 15%）。<br>• **内存流量**：原本中间 `intermediate_cache` 需写回显存；one‑shot 将中间结果保存在寄存器/共享内存中，显存读写下降约 1‑2 ×。<br>• **并行度**：对 **小 batch**（decode）引入 NPID_FACTOR、tile‑划分，使得即使 M ≤ 64 仍能保持 SM 饱和。<br>• **双流**：在 fully_sharded=False、且 GPU 为 CUDA‑capable 时，可把 LoRA 增量写入 aux 流，使用四个事件同步；在 GPU 资源充足的情况下，整体推理吞吐提升 5%‑12%。<br>• **启用门槛**：`max_lora_rank ≤ 64` 才走 small‑batch path；更大 rank 会回退到 one‑shot（仍单 kernel）。 |
| **安全考虑** | • 新增的 env 变量仅控制内部调度，不涉及 I/O。<br>• Triton kernel 内部使用 **assert** 检查 `shrink_block_size_m` 必须为 2ⁿ 且 ≥ 16，避免编译阶段的 “arange range must be a power of 2” 报错，防止潜在的异常泄露。<br>• 双流实现通过 CUDA events 明确定序，避免数据竞争。<br>• 代码未引入外部依赖或网络交互，安全风险基本为 **零**。 |
| **可维护性** | • Triton 内核代码体量大（约 800 行），但已拆分为 *one‑shot* 与 *small‑batch* 两个子 kernel，逻辑分层清晰。<br>• 参数化（constexpr 与 runtime）的使用提升了 kernel 重用性，避免了大量模板实例化导致的编译爆炸。<br>• 新增 `add_inputs` 参数使得 kernel 可在 **写入‑增量** 与 **覆盖‑结果** 两种模式间切换，便于在 dual‑stream 场景中复用同一实现。<br>• 单元测试覆盖了 **极小 rank、极端 block_size、零 token、early‑exit** 等边界，降低回归风险。 |

### ⚠️ 潜在风险
1. **GPU 兼容性**  
   - 双流路径仅在 CUDA‑like 设备上启用；在不支持 CUDA（如 ROCm、CPU）上仍会回退到单流，但如果用户手动将 `VLLM_LORA_ENABLE_DUAL_STREAM=1` 且平台不支持，可能导致 `None` stream 传入，触发运行时错误。  
2. **Block‑size 断言**  
   - `shrink_block_size_m` 必须为 2ⁿ 且 ≥ 16。若上层调用者传入不合法的 block size（如 24），会在 kernel 调用前抛 `AssertionError`。需要在 `moe_lora_align_block_size` 上层提前校验，防止生产环境意外崩溃。  
3. **注册表溢出**  
   - `_LORA_PTR_DICT` 通过 `torch.tensor` 作为键；若在同一进程中频繁创建不同 shape·dtype 的 LoRA 权重，可能导致字典无限增长，导致显存泄漏。建议在进程结束或模型卸载时显式清理。

### [ROCm] Remove MegaMoE integration in deepseek v4 (#43629)
**SHA**: `c8414a8` | 🔗 [查看提交](https://github.com/vllm-project/vllm/commit/c8414a82712bd775b7243b19a264d9623c3bb369)

**🎯 变更类型**：重构 / 其他  
**⚡ 重要程度**：🟡 中  
**📋 变更摘要**：在 AMD（ROCm）路径上彻底剔除了 DeepSeek‑V4 MegaMoE 的实现，包括其权重加载、FP8/FP4 量化、triton kernel、custom‑op 注册及所有相关分支逻辑。模型现在只能使用原有的 FusedMoE 实现，`kernel_config.moe_backend`、`use_mega_moe` 标记等已全部删去。  

**🎯 影响范围**  
- `vllm/models/deepseek_v4/amd/model.py`：删除约 600 行 MegaMoE 代码，所有 `use_mega_moe` 条件和 `DeepseekV4MegaMoEExperts` 类被清空。  
- `vllm/models/deepseek_v4/amd/mtp.py`：去掉对 MegaMoE 专家映射的判断，统一走 `FusedMoE.make_expert_params_mapping`。  
- 相关 import（`triton`, `deep_gemm`, `forward_context` 等）被移除，防止在 ROCm 环境下出现未实现的 CUDA‑only 依赖。  
- 权重加载、`finalize_mega_moe_weights`、`_run_mega_moe` 等入口全部消失，模型的 `get_expert_mapping` 只返回 FusedMoE 映射。  

**💡 关注建议**  
1. **配置检查**：确保使用 ROCm 时不要在 `kernel_config.moe_backend` 中指定 `deep_gemm_mega_moe`，否则会触发 `NotImplementedError`。  
2. **权重兼容**：MegaMoE checkpoint（FP4/FP8 权重量化）已不再受支持，若用户尝试加载此类 checkpoint，需要重新导出为标准 FusedMoE 权重。  
3. **文档同步**：更新模型说明、配置示例以及迁移指南，明确 MegaMoE 已被移除，仅保留 FusedMoE。  
4. **测试与 CI**：添加针对 ROCm 环境的单元/集成测试，验证在没有 CUDA 的情况下模型能够正常实例化、前向运行并通过权重加载。  
5. **清理残余**：检查其它平台（如 CUDA）是否仍保留对 MegaMoE 的引用，避免出现未使用的 import 或 dead‑code。  

总体而言，此次删减提升了 ROCm 上的可编译性和维护性，但需在发布说明中提醒用户迁移权重和配置，防止因旧配置导致启动失败。
