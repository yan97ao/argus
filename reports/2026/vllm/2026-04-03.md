### [Frontend] new online quantization frontend (#38138)
**SHA**: `7b1a742` | 🔗 [查看提交](https://github.com/vllm-project/vllm/commit/7b1a7423bea1705bd51d838e34bef99e8a01cbbd)

**🎯 变更类型**：功能增强 / 架构变更 / 安全修复  
**⚡ 重要程度**：🔴高  

**📋 变更摘要**  
本次提交为 vLLM 引入 **在线量化（Online Quantization）前端**，实现了在模型加载阶段对 FP8 进行即时量化的能力。新增了枚举 `OnlineQuantScheme`、配置类 `OnlineQuantizationConfigArgs` 与 `OnlineQuantizationConfig`，并在 `EngineArgs`、`ModelConfig`、`LLM` 构造函数中加入 `quantization_config` 参数的解析与传递。实现了针对 Linear 与 MoE 层的在线 FP8 量化方法（Tensor‑wise 与 Block‑wise），并在模型权重加载流程中加入 meta‑device 延迟 materialize、按需量化、忽略层等控制。配套测试验证了配置选择、层级忽略以及加载峰值显存。

---

### 🎯 影响范围
- **模型配置层**：`vllm/config/model.py`, `vllm/config/quantization.py`, `vllm/config/vllm.py`  
- **引擎参数解析**：`vllm/engine/arg_utils.py`  
- **入口构造**：`vllm/entrypoints/llm.py`  
- **量化方法注册**：`vllm/model_executor/layers/quantization/__init__.py`  
- **在线量化实现**：`vllm/model_executor/layers/quantization/online/base.py`, `vllm/model_executor/layers/quantization/online/fp8.py`（Linear + MoE）  
- **权重加载逻辑**：`vllm/model_executor/model_loader/weight_utils.py`（冲突检测）  
- **测试套件**：`tests/quantization/test_online.py`, `tests/quantization/utils.py`

### 🔍 技术洞察

| 维度 | 影响说明 |
|------|-----------|
| **架构影响** | • 新增 **OnlineQuantizationConfig** 作为模型级量化配置，统一了 *online* 与 *fp8_per_tensor / fp8_per_block* 的快捷用法。<br>• `EngineArgs.__post_init__` 通过 `resolve_online_quant_config` 将用户输入的 `quantization` 与 `quantization_config` 合并并校验，形成统一的配置对象。<br>• 量化方法注册表 `QUANTIZATION_METHODS` 中加入 `"online"` 与两种 FP8 scheme 的字符串 alias，保持向后兼容且不产生循环依赖。<br>• 通过 `LinearBase` 与 `FusedMoE` 的 **online** 子类，实现 **meta‑device** 延迟加载：权重先保存在 `meta`，随后在 `process_weights_after_loading` 中原位量化为 FP8。 |
| **性能影响** | • **存储**：无需预先生成 FP8 checkpoint，显著降低模型磁盘占用（示例模型 12 GiB → ~6‑7 GiB）。<br>• **加载时延**：在 weight loading 阶段额外执行一次 `scaled_fp8_quant` 或 `per_block_cast_to_fp8`，导致加载时间约增加 10‑20%（取决于块大小、Cutlass 支持情况）。<br>• **显存峰值**：加载阶段需同时保留原始 BF16/FP16 权重和临时 FP8 + scale 张量；测试中对 `allenai/OLMoE-1B-7B-0125-Instruct` 的峰值设定为 1.4×模型实际占用，已通过 CI 断言。<br>• **运行时推理**：量化后使用专门的 FP8 kernel（Cutlass / AItER / DeepGEMM），在支持的硬件上可获得与离线量化相近的吞吐率，且在不支持的 GPU（如旧版 CUDA）会回退至 BF16。 |
| **安全考虑** | • 防止 **配置冲突**：`weight_utils.get_quant_config` 检测到 checkpoint 已携带 `quantization_config` 时，如果用户再次提供 `quantization_config`（在线量化）会抛出明确异常，避免不一致的量化策略导致未定义行为。<br>• 仅在单元测试中打开了 `VLLM_ALLOW_INSECURE_SERIALIZATION` 环境变量，生产代码未受影响。<br>• 新增的正则层过滤 `ignore` 使用标准 `re:` 前缀，避免潜在的路径遍历。 |
| **可维护性** | • 使用 **Pydantic** (`@config`) 进行参数校验，统一了字段默认值与类型检查。<br>• 代码结构保持 “online‑*” 与原有离线量化实现平行，后续新增 scheme（如 `mxfp8`）只需在 `OnlineQuantScheme` 中添加枚举值并实现对应 Method 类。<br>• `OnlineQuantizationConfig.from_config` 明确声明不支持从 checkpoint 加载，降低误用风险。<br>• `tests/quantization` 提供了覆盖不同 scheme、override、ignore 组合的端到端验证。 |

### ⚠️ 潜在风险
1. **显存峰值超出预期**  
   - 在线量化在加载阶段会保留原始 BF16/FP16 权重直至量化完成，若模型极大或块大小设置不当，峰值可能超过 GPU 可用显存，导致 OOM。  
2. **硬件兼容性**  
   - 代码在 `cutlass_fp8_supported`、`cutlass_block_fp8_supported`、`rocm_aiter_ops.is_linear_fp8_enabled` 等函数上做条件分支，未明确记录不支持的 GPU 灰名单，可能在某些老旧驱动上触发未捕获的 `RuntimeError`。  
3. **忽略层正则匹配误差**  
   - `ignore` 支持 `re:` 前缀，但在 `should_ignore_layer` 中的实现若正则写法不严谨，可能误过滤掉必须量化的层，导致推理精度下降。  
4. **重复调用 `process_weights_after_loading`**  
   - 对于 weight reload 场景，若外部逻辑在 `process_weights_after_loading` 前未重置标记，可能导致权重未被重新量化或残留旧 scale。  
5. **向后兼容性**  
   - 将 `"fp8_per_tensor"` 与 `"fp8_per_block"` 直接映射到 `OnlineQuantizationConfig`，在已有基于离线量化的脚本中可能不再使用 `quantization_config`，导致意外切换至 online 量化。

### [CPU] Replace OMP initialization (#36487)
**SHA**: `abebd93` | 🔗 [查看提交](https://github.com/vllm-project/vllm/commit/abebd9323d7cf8abefd07f976be3be356d873075)

**🎯 变更类型**：架构变更 / 重构 / 性能优化  

**⚡ 重要程度**：🔴高  

**📋 变更摘要**  
- 在 C++ 侧删除了原有的 `init_cpu_threads_env`（基于 `libnuma`、`sched_setaffinity` 的 OMP 线程绑定）实现，改为全部在 Python 层通过新的 **OMPProcessManager** 完成 OMP 进程/线程亲和度的配置。  
- `CpuPlatform` 迁入统一的 OMP 资源管理，新增对模拟 NUMA、SMT、全局 CPU mask 的支持，并强制 CPU 端使用 **MP**（多进程）执行器。  
- CI 脚本中对 DP+TP 场景的测试被注释掉，防止在不支持 OMP 绑定的环境中跑出错。  

**🎯 影响范围**  
- `csrc/cpu/*`（原 OMP 初始化函数及其绑定实现）  
- `vllm/platforms/cpu.py`（CPU 平台配置、NUMA/OMP 管理）  
- `vllm/utils/ompmultiprocessing.py`（全新 OMP 进程包装）  
- `vllm/v1/executor/multiproc_executor.py`（worker 进程启动改为经过 OMP manager）  
- `vllm/v1/worker/cpu_worker.py`（去除旧的线程绑定逻辑）  
- CI 脚本 `run‑cpu‑distributed‑smoke‑test.sh`（关闭 DP+TP 自动化测试）  

---

## 🔍 技术洞察

| 维度 | 分析 |
|------|------|
| **架构影响** | 1. **从 C++ 到 Python**：把 CPU 亲和度、NUMA 绑定的实现从原生 C++ / libnuma 移动到纯 Python，统一在进程启动阶段通过环境变量 (`OMP_PLACES`, `OMP_NUM_THREADS`, `OMP_PROC_BIND`) 完成。<br>2. **新增 OMPProcessManager**：负责解析 `lscpu -J` 输出、构造 OMP “places”，并在 `multiproc_executor` 中为每个 worker 进程分配唯一的 CPU 集合。<br>3. **平台抽象统一**：`CpuPlatform` 现在持有 `omp_process_manager`、`global_cpu_mask`、`simulate_numa`，并提供 `reserve_cpus()` / `get_global_cpu_mask()` 接口，后续可在其他模块直接查询。<br>4. **执行器约束**：CPU 端强制使用 `mp`（多进程）后端，`uni`（单进程）已被禁用，以保证 OMP 环境能够被正确设置。 |
| **性能影响** | - **正面**：<br>  • 通过 `OMP_PLACES` + `OMP_PROC_BIND=TRUE` 明确指定每个进程使用的核心集合，避免了之前因 `torch.set_num_threads` 与系统调度器不一致导致的 **CPU 争用**。<br>  • 采用 `nodes`（默认）策略，可让每个 worker 只使用其所属 NUMA 节点上的核心，提升 **内存局部性**。<br>  • 通过 `VLLM_CPU_OMP_THREADS_BIND=nobind` 可以关闭绑定，兼容用户自行调优。<br>- **潜在负面**：<br>  • 依赖 `lscpu` 命令与 JSON 解析，启动时会有一次子进程调用（开销极小，但在极端的启动‑latency 场景下会被放大）。<br>  • 绑定逻辑不再使用 `sched_setaffinity` 的系统调用，而是交由 OpenMP 运行时，这在某些平台上（如老旧的 libgomp）可能不如手工设置精细。 |
| **安全考虑** | - 去除 C++ 中直接调用 `numa_*`、`sched_setaffinity`，降低了 **系统调用层面的攻击面**。<br> - 新增的 Python 解析 `CPU_VISIBLE_MEMORY_NODES`、`VLLM_CPU_OMP_THREADS_BIND` 环境变量，若输入恶意或格式错误，仅会导致 **绑定失效**（抛出异常或回退到默认），不会触发提升权限的行为。<br> - 仍然依赖 `lscpu` 可执行文件的可访问性，在受限容器中可能被禁用；若被篡改返回错误 JSON，程序会 `json.loads` 失败并抛异常，属于可控的 **拒绝服务**（DoS）风险。 |
| **可维护性** | - **代码体量大幅缩减**（C++ 侧仅剩 3 行），降低了平台特异性代码的维护成本。<br> - Python 实现更易于单元测试和 CI 验证，新增 `ompmultiprocessing.py` 提供明确的入口函数 `run`，便于以后扩展（例如支持自定义策略）。<br> - 删除了大量 `#ifdef VLLM_NUMA_DISABLED` 条件编译代码，使主线更清晰。 |
| **兼容性** | - **向后兼容**：通过环境变量 `VLLM_CPU_OMP_THREADS_BIND` 可以仍然手动指定 CPU mask；若用户显式设为 `nobind`，行为与旧实现相同（不绑定）。<br> - **破坏性变更**：不再支持 `uni` 后端；使用 `cpu` 平台且仍保持 `uni` 会自动降级为 `mp` 并打印警告。<br> - 旧的 C++ 绑定 API (`torch.ops._C.init_cpu_threads_env`) 已被删除，任何外部插件直接调用该 API 将出现 `AttributeError`。 |

## ⚠️ 潜在风险

| 风险点 | 说明 | 严重度 |
|--------|------|--------|
| **依赖 `lscpu`** | 在极端容器或 minimal OS 中可能缺失，导致 OMP 资源发现失败，进而抛异常或全部 worker 使用相同 CPU 集合。 | 中 |
| **NUMA 模拟逻辑错误** | `simulate_numa` 通过拆分现有 OMP `places` 来假装多 NUMA，若原始拓扑已经很细碎（如每个 core 为独立 place），拆分可能失败并抛 `ValueError`。 | 低‑中 |
| **默认 `OMP_NUM_THREADS=1`** | `set_multiprocessing_worker_envs` 中对非‑CPU 平台（GPU）仍会在未设置 `OMP_NUM_THREADS` 时

### [Kernel] Fuse FP8 output quantization into merge_attn_states (#36518)
**SHA**: `3bc2734` | 🔗 [查看提交](https://github.com/vllm-project/vllm/commit/3bc2734dd03d2073e1ba066bda042b4b085037bd)

**🎯 变更类型**：功能增强 & 性能优化  
**⚡ 重要程度**：🔴 高

### 🔍 技术洞察  

| 维度 | 影响说明 |
|------|----------|
| **架构影响** | - 将 **FP8 量化** 直接嵌入 `merge_attn_states`，消除原本的两段式（merge → torch.compile‑quant）路径，降低 kernel 调度次数。<br>- 新增 `output_scale` 作为可选张量，统一了 CUDA 与 Triton 两条实现路径的 API。<br>- 通过 `VLLM_DISPATCH_FP8_TYPES` 与 `USE_FP8` 编译期开关，实现对 `float8_e4m3fn` 与 `float8_e4m3fnuz` 两种 FP8 类型的透明调度，不影响现有 BF16/FP16/FP32 流程。 |

### ⚠️ 潜在风险  

1. **参数错误**：如果调用方在 FP8 输出时忘记传 `output_scale`，会触发断言或产生未量化的随机值。

3. **平台兼容性**：当前实现依赖 `current_platform.fp8_dtype()`，在非 CUDA（如 ROCm）或旧显卡不支持 FP8 时仍会走 unfused路径，若未在构建脚本中明确声明可能导致意外 fallback。  
4. **线程块配置**：`head_size` 必须是 `pack_size` 的整数倍；对某些自定义 head‑size（非 8/4 整倍）仍会回退至 Triton 实现，可能出现性能不一致。

**SHA**: `4a06e12` | 🔗 [查看提交](https://github.com/vllm-project/vllm/commit/4a06e1246e30214d644a1732d6853cfe2b935a75)

5️⃣ **性能基准**：建议添加微基准测试（不同块数、不同块大小、不同 CUDA 版本），对比原循环版本与批量版本的提交延迟，以验证实际收益。  

6️⃣ **文档与 API**：`swap_blocks_batch` 在 Python docstring 已说明需求（CPU int64 张量）。确保在发布说明中提示：*仅在 CUDA 12.8+ 能获得一次性提交的加速*，并说明在旧 CUDA 环境下仍会工作但不一定更快。  

7️⃣ **资源泄漏**：`cpu_gpu.py` 中新建的 `batch_src/batch_dst/batch_sizes` 为临时 `torch.Tensor`，生命周期短，无需额外管理。若在高并发环境下频繁创建，建议复用已有张量或使用 `torch.empty_like` 以降低分配开销。  

总体来看，改动聚焦在降低 KV‑cache 迁移的驱动调用开销，逻辑清晰且保留了向后兼容路径。重点检查跨 CUDA 版本的回退行为、指针尺寸假设以及错误信息的可诊断性即可确保安全上线。

**💡 关注建议**

总体来说，此次改动为 Mamba 系列提供了更灵活的布局选项，代码路径已全面适配，唯一需注意的是 DS 布局在 speculative decoding 场景的未实现路径及潜在的 stride‑相关假设。建议在 CI 中保留两种布局的回归测试，并在正式发行前提供使用指南。

### [Frontend] Re-enable running MaxSim on GPU  (#38620)
**SHA**: `a9b4f07` | 🔗 [查看提交](https://github.com/vllm-project/vllm/commit/a9b4f07ba259f6fa95b61a6aa64ea80ea7681493)

**🔧 变更概览**  
本次提交在 **pooling → scoring** 流程中引入 “flash‑late‑interaction” 模式，使 MaxSim 计算可以在工作节点（GPU）上完成，从而提升 late‑interaction 评分的吞吐与延迟。核心实现包括：

| 位置 | 关键改动 |
|------|----------|
| `vllm/entrypoints/openai/cli_args.py` | 新增 `enable_flash_late_interaction`（默认 `True`）的 CLI 参数。 |
| `vllm/entrypoints/pooling/__init__.py` | 将该标志从 `args` 中读取并传递给 `init_pooling_state`。 |
| `vllm/entrypoints/pooling/base/serving.py` | 拆分 `__call__` 为 `ctx` 初始化 + 两阶段处理；新增 `flash_late_interaction` 流程。 |
| `vllm/entrypoints/pooling/scoring/io_processor.py` | 用 `n_queries` 替代原来的 `offset`，新增 `FlashLateInteractionIOProcessor`（直接返回 worker‑侧的分数），并统一 `ScoringIOProcessors` 为 `dict[str, type]`。 |
| `vllm/entrypoints/pooling/scoring/serving.py` | 根据 `enable_flash_late_interaction` 自动切换 `score_type` 为 `flash-late-interaction`，并实现查询‑文档两阶段 GPU 编码/打分逻辑。 |
| `vllm/v1/pool/late_interaction.py` | 将原 `compute_maxsim_score` 抽离为 `compute_maxsim_score_batched`，供 worker 批量计算。 |
| 其它（tests、utils、typing） | 相应地把 `offset` 更名为 `n_queries`、添加类型字段、修正导入、更新测试用例。 |

**📦 影响范围**  
- **API 服务器**：新增 `--enable-flash-late-interaction/--no-enable-flash-late-interaction` 参数；默认开启。  
- **模型配置**：`ModelConfig.score_type` 为 `"late-interaction"` 时会自动切换为 `"flash-late-interaction"`。  
- **IO Processor / Serving**：所有涉及 late‑interaction 的路径（`BiEncoderIOProcessor`、`LateInteractionIOProcessor`、`FlashLateInteractionIOProcessor`）均受到影响。  
- **离线/在线评分**：离线 `score` 接口已改用 `n_queries` 参数；在线请求走两阶段 GPU 编码。  
- **测试**：`test_late_interaction_online` 通过参数化同时验证开启/关闭两种行为。

**💡 关注建议**  
1. **兼容性检查**：确认老版客户端或自定义插件未直接访问已删除的 `offset` 字段；如有，需迁移到 `n_queries`。  
2. **资源消耗**：`flash_late_interaction` 会为每个查询生成缓存 token（`late_interaction_params`），在高并发下可能导致显存峰值升高，建议监控 GPU memory 并提供 `--max-num-batches` 等限制手段。  
3. **日志与监控**：在 `ServingScores.__init__` 中记录 `enable_flash_late_interaction` 状态，便于排障。  
4. **错误路径**：若 worker 端返回 `None` 或分数不匹配，`FlashLateInteractionIOProcessor` 直接返回 `outputs`，请确保 upstream 对 `final_res_batch` 的非空断言不会误触。  
5. **性能回归**：在 CI 中加入对比基准（开启/关闭 flash）确保数值一致且有预期加速。  

总体而言，此改动为 late‑interaction 评分提供了显著的 GPU 加速路径，但需要在生产环境中关注显存占用及向后兼容性。
