### [MoE Refactor] Make SharedExperts class for use with DefaultMoERunner (#35153)
**SHA**: `7cf56a5` | 🔗 [查看提交](https://github.com/vllm-project/vllm/commit/7cf56a59a267d2f32860c96a601e56bb29f6627a)

**🎯 变更类型**：架构变更 / 重构 / 功能增强  
**⚡ 重要程度**：🔴 高  

**📋 变更摘要**  
本次提交对 VLLM 中 MoE（Mixture‑of‑Experts）实现做了大幅重构，新增 `SharedExperts` 类统一管理 **共享专家（shared experts）** 的调度、流同步、跨 DP/EP 的归约等逻辑；相应地在 `FusedMoE`、`MoERunner`、各量化实现以及 kernel 构造入口等多处删减/替换了原有的 `shared_experts` 参数与属性。实现目标是：  
1. 把共享专家的生命周期交给专门类，降低 `FusedMoE` 与 `MoERunner` 的耦合度。  
2. 为支持异步（async）执行、DP‑Chunking 与多流重叠提供统一的调度入口。  
3. 简化自定义 Op（`moe_forward` / `moe_forward_shared`）的签名，统一为 `forward_dispatch`。  

**🎯 影响范围**  
- `vllm/model_executor/layers/fused_moe/`（层实现、runner、kernel、router）  
- `vllm/model_executor/layers/quantization/`（所有量化方法的 MoE 调用）  
- `vllm/distributed/elastic_ep/elastic_execute.py`（量化方法切换）  
- `tests/kernels/moe/utils.py`、`tools/ep_kernels/install_python_libraries.sh`（测试与构建脚本）  
- 相关的配置类 `FusedMoEParallelConfig`、`FusedMoEConfig`  

---

### 🔍 技术洞察  

| 维度 | 影响概述 |
|------|----------|
| **架构影响** | • 引入 `SharedExperts` 将共享专家的创建、流管理、DP/EP 归约、DBO 缓冲等职责抽离出来，`FusedMoE` 只保留专家路由、输入变换等核心功能。<br>• `MoERunner` 现在通过 `forward_dispatch` 统一调度：先同步流、执行 gate（若内部持有）、再根据 `SharedExpertsOrder` 调用共享专家，实现 **内部/外部**、**无重叠/多流重叠** 四种执行模式。<br>• `FusedMoEKernel` 的 `owns_shared_experts` 接口用于判断 runner 是否拥有共享专家实例，取代原来的 `shared_experts is not None` 检查。 |
| **性能影响** | • 对 **DP‑Chunking** 场景：`SharedExperts._determine_shared_experts_order` 会返回 `MK_INTERNAL_OVERLAPPED`，共享专家直接在 MK（matrix kernel）内部并行执行，避免额外流切换。<br>• 对 **CUDA‑stream 重叠** 场景：当 token 数低于 `VLLM_SHARED_EXPERTS_STREAM_TOKEN_THRESHOLD` 并且未使用 DP‑Chunking，`SharedExperts` 会在独立 aux‑stream 上运行，实现 **gate+router 与共享专家并行**，理论上提升吞吐。<br>• 统一 `apply` 接口后，可在 `forward_impl` 中省去旧的 `run_shared_experts_before`、`use_shared_experts_stream` 判断，降低分支开销。<br>• 迁移后仍保留旧的 “外部调用” 路径（`EXTERNAL`），兼容不支持 async 的 kernel。 |
| **安全考虑** | - 代码未引入外部文件 I/O、网络请求或解释执行等新攻击面。<br>- 仅涉及 CUDA 流同步和跨进程的 Tensor 归约，若流同步错误可能导致 **GPU 数据竞争**（但不属于安全漏洞）。<br>- 环境变量 `VLLM_DISABLE_SHARED_EXPERTS_STREAM`、`VLLM_ENABLE_MOE_DP_CHUNK` 仍控制行为，建议在生产环境显式设定，防止意外开启未充分验证的路径。 |
| **可维护性** | • `SharedExperts` 将大量分散的共享专家处理代码集中，便于以后加入更多调度策略（如 NCCL‑graph、异步流水线）。<br>• 通过 `SharedExpertsOrder` 枚举明确了四种调度模式，代码可读性提升。<br>• 大量 `torch.nn.Module | None` 的参数被改为 `SharedExperts | None`，类型更精准，IDE/静态检查更友好。<br>• 移除 `layer._shared_experts`、`layer._gate` 等私有属性，降低了层内部的耦合度。 |
| **兼容性** | • 公开的 `FusedMoE.shared_experts` 仍返回 `SharedExperts | None`，但返回类型已变更，使用旧代码直接访问内部属性会报错。<br>• 部分 kernel 构造函数仍接受 `shared_experts=None`，但在使用 `mk`（modular kernel）时会自动过滤掉不支持 async 的情况。<br>• TPU/CPU 路径仍走原有 `_moe_forward` / `_moe_forward_shared` 调用，不受新流逻辑影响。 |

### ⚠️ 潜在风险  

1. **运行时同步错误**  
   - `SharedExperts.maybe_sync_shared_experts_stream` 与 `runner._maybe_sync_shared_experts_stream` 的顺序极为关键；若在非 CUDA 环境或 `aux_stream()` 返回 `None` 时仍尝试 stream.wait，可能导致 **CUDA illegal memory access**。  
2. **DP‑Chunking 与外部共享专家冲突**  
   - `SharedExperts._has_external_experts` 在 DP‑Chunking 场景返回 `False`，但如果用户手动强制传入 `shared_experts`（可能在自定义模型中），仍可能触发不被支持的路径，引发 AssertionError。  
3. **DBO（Dynamic Batch Offloading）交叉使用**  
   - `SharedExperts` 用 `dbo_current_ubatch_id()` 进行索引，如果 DBO 开关不一致或 ubatch ID 计数错误，可能出现 **tensor泄漏或错误覆盖**。  
4. **自定义 MoE kernel 与新接口不匹配**  
   - 第三方插件若实现 `FusedMoEPrepareAndFinalizeModular` 并仍假设 `shared_experts` 直接作为 `torch.nn.Module` 传递，将在运行时触发类型错误。  
5. **测试覆盖不足**  
   - 许多分支（e.g., `EXTERNAL`、`NO_OVERLAP`、`MULTI_STREAM_OVERLAPPED`）在现有 CI 中的覆盖率未知，可能在特定 TP/DP/EP 组合下出现未捕获的回归。  
6. **环境变量遗漏**  
   - 生产环境若未显式设置 `VLLM_ENABLE_MOE_DP_CHUNK`，默认行为可能与期望不符，导致共享专家未被并行执行，出现 **性能回退**。

   - 增加矩阵并行 (TP)、数据并行 (DP)、专家并行 (EP) 以及混合模式下的 **共享专家**

### [Misc] Add 20 regression tests for 11 tool parser bug fixes (#38172)
**SHA**: `cb0b443` | 🔗 [查看提交](https://github.com/vllm-project/vllm/commit/cb0b4432746f8a7aaeeba9a8068e8fd4baf84a0a)

**🎯 变更类型**：测试增强（Regression Tests）  
**⚡ 重要程度**：🔴高  
**📋 变更摘要**：为 VLLM 项目中 11 种不同的 Tool Parser（DeepSeek‑V32、GLM‑4‑MoE、Kimi‑K2、Minimax‑M2、Mistral、Qwen‑3‑Coder、Step‑3.5）新增 20+ 份回归测试，覆盖了工具调用解析的边界条件、零参数调用、特殊字符、Unicode、anyOf nullable 参数、Fast Detokenization、原生 ID 传递、流式分块等多个历史 bug 场景，确保这些修复不再 regression。  

**🎯 影响范围**：  
- `tests/tool_parsers/` 目录下的所有新增/修改的测试文件。  
- 间接影响到 `vllm/tool_parsers/` 实现的兼容性与稳定性（通过 CI 触发）。  

**🔍 技术洞察**：

- **架构影响**  
  - 仅为测试层改动，不涉及生产代码的架构或接口变更。  
  - 通过 **`run_tool_extraction_streaming`** 工具函数对多种流式解析路径进行统一驱动，强化了对不同 Parser 实现的抽象统一性，间接提升了未来新增 Parser 时的可测试性。  

- **性能影响**  
  - 新增约 700 行测试，执行时间预计在 **30‑90 秒**（取决于 CI 机器），对整体 CI 耗时有一定增加。  
  - 测试采用 **MockTokenizer**、纯文本切片等轻量方式，未对模型推理路径产生实际负载，故对运行时性能无影响。  

- **安全考虑**  
  - 添加了对 **Unicode、非 ASCII、特殊字符** 以及 **anyOf nullable 参数** 的解析测试，确保解析器不会错误地对输入进行 **Unicode 转义** 或 **JSON double‑encoding**，降低了潜在的 **注入/字符误解析** 风险。  
  - 测试中对 **`skip_special_tokens`** 参数的检查确保在开启工具调用时不意外隐藏特殊 token，防止信息泄露或误判。  

**⚠️ 潜在风险**：

1. **测试 flakiness**  
   - 部分测试依赖于 **`run_tool_extraction_streaming`** 进行 “逐块” 与 “一次性” 两种流式解析对比，若底层 Tokenizer 实现或 `extract_tool_calls_streaming` 行为在不同 Transformers 版本之间出现细微差异，可能导致间歇性失败。  

2. **环境依赖**  
   - 新增的 `tests/tool_parsers/utils` 及 `MagicMock` 的 `tokenize` 方法假设 `MOCK_TOKENIZER.tokenize` 存在；在老版本的测试框架或自定义 tokenizers 中若缺失该属性，可能抛出属性错误。  

3. **CI 资源消耗**  
   - 约 20 份新测试对 CI 报告的整体执行时间有正向叠加，可能导致 **GitHub Actions** 超时或并发限制。  

4. **误报/误判**  
   - 部分 “异常” 输入（如缺失 `>` 的 XML）在当前实现下被宽容处理并返回空列表；若未来更严格的解析逻辑被引入，这些测试可能需要同步更新。  

**💡 关注建议**：

- **CI 稳定性**：在 CI 中为 `tests/tool_parsers/` 添加 **单独的 job**，并配置 **重试** 或 **flaky** 标记，以防偶发性失败影响主分支合并。  
- **依赖锁定**：确保 `requirements.txt` 中锁定的 *transformers*、*pytest* 版本与本地开发环境保持一致，避免因上游库改动导致流式 token id 判断差异。  
- **监控测试时长**：通过 GitHub Actions 的 `jobs.<job>.timeout-minutes` 设置上限，必要时对耗时最长的测试（如 MTP 大块流式）进行拆分或使用 **parameterized** 方式降低单次运行负载。  
- **文档同步**：在 `docs` 或 `CHANGELOG.md` 中记录对应的 **bug ID（PR #xxxx）** 与覆盖的场景，帮助后续维护者快速定位回归点。  
- **未来扩展**：将公共的 “流式工具调用重建” 逻辑抽象为 `tests/tool_parsers/utils.py` 中的可复用函数（已部分实现），以便新 Parser 增加时直接复用，保持测试代码 DRY。

### [vLLM IR] 1/N Implement IR skeleton and rms_norm op (#33825)
**SHA**: `40bb175` | 🔗 [查看提交](https://github.com/vllm-project/vllm/commit/40bb17502736a9b6127cefc535303285b13e0630)

**🎯 变更类型**：功能增强 / 架构变更 / 重构  
**⚡ 重要程度**：🔴高  

**📋 变更摘要**  
1. 在 `vllm/ir` 目录新增了完整的 **vLLM IR（中间表示）** 框架：`IrOp` 注册、实现调度、优先级配置、torch 包装开关、UUID 哈希等核心模块。  
2. 实现 **rms_norm** 这一根均值归一化算子在 IR 中的抽象，并在 `vllm/kernels` 中提供多种后端实现（`vllm_c`、`aiter`、`oink`、`xpu_kernels`），以及默认的 native 实现。  
3. 新增 **IR 降低 Pass**（`VllmIRLoweringPass`），在编译 graph 时把 IR 调用替换为选中的实现，实现与 `torch.compile`、Inductor 的无缝集成。

### ⚠️ 潜在风险

1. **并发批量大小极端情况**  
   - 若单请求的 `label_token_ids` 列表极长（如上千），`logprob_token_ids` 会导致 `token_ids_tensor` 在 `Sampler.gather_specific_token_logprobs` 中膨胀到 `batch × (max_len+1)`，可能触发显存 OOM。

2. **与 `logprobs` 参数冲突**  
   - 当前实现若同时提供 `num_logprobs`（>0）和 `logprob_token_ids`，后者会覆盖前者。若用户误认 `num_logprobs` 仍会返回完整 vocab，则会得到意外的截断。

3. **跨模型兼容**  
   - `logprob_token_ids` 依赖模型在 `generate` 阶段返回 `logprobs` 信息。若后端模型在某些实现上未实现 `logprobs`（如自定义外挂模型），会返回 `None` 并触发错误。  
   - **建议**：在 `create_generative_scoring` 中提前检查 `engine_client` 是否支持 `logprobs`，并给出清晰错误。

4. **Trace Header 传递**  
   - 新增的 `trace_headers` 均通过 `_get_trace_headers` 传播，若 tracing 未开启会打印警告。大量调用可能产生日志噪声。  
   - **建议**：在生产环境可通过环境变量关闭 `log_tracing_disabled_warning`。

5. **测试资源需求**  
   - E2E 测试会启动真实模型实例（`Qwen3‑0.6B`），在 CI 环境可能导致资源超限。  
   - **建议**：在 CI 只运行单元测试，或提供轻量级模型（如 `facebook/opt-125m`）的可选标记。

**🎯 变更类型**：功能增强（feat）  
**⚡ 重要程度**：🔴高  
**📋 变更摘要**：在 vLLM 的 gRPC 入口点中新增周期性统计信息日志的功能。默认情况下，服务器会每 `envs.VLLM_LOG_STATS_INTERVAL` 秒调用一次 `AsyncLLM.do_log_stats()`，并在关闭时安全取消该后台任务；可通过 `--disable-log-stats` 参数关闭此特性。  

**🎯 影响范围**：  
- `vllm/entrypoints/grpc_server.py`（gRPC 服务器启动与关闭流程）  
- `vllm/envs.py`（提供统计间隔的环境变量）  
- 与统计日志相关的内部模块（如 `AsyncLLM.do_log_stats`）  

**🔍 技术洞察**：  
- **架构影响**：  
  - 引入了一个独立的 `asyncio` 长生命周期任务 (`_force_log`) 作为服务器生命周期的一部分，保持与 HTTP 服务器相同的日志统计机制，提升了跨入口点的一致性。  
  - 通过 `args.disable_log_stats` 与环境变量 `VLLM_LOG_STATS_INTERVAL` 实现可配置性，符合“配置优先”设计原则。  
- **性能影响**：  
  - 统计日志本身的开销已有实现（`do_log_stats`），额外的周期性调用会在每个间隔产生一次额外的 CPU/IO 负载，通常极小，但在高并发场景或极短间隔（如低于 1 s）时可能对延迟产生可感知影响。  
  - 任务取消采用 `Task.cancel()`，若 `do_log_stats` 包含阻塞 I/O，取消可能被延迟，需要在实现中确保对 `CancelledError` 的合适处理。  
- **安全考虑**：  
  - 统计信息可能包含模型加载路径、GPU 使用情况等内部运行时细节。如果服务器部署在受限网络环境，暴露此类信息可能成为信息泄露风险。建议在生产环境通过 `--disable-log-stats` 或将 `VLLM_LOG_STATS_INTERVAL` 设为 `0` 来关闭。  

**⚠️ 潜在风险**：  
1. **环境变量缺失**：如果 `envs.VLLM_LOG_STATS_INTERVAL` 未定义或类型不为数值，启动时会抛出 `AttributeError` 或 `TypeError`，导致服务器无法启动。  
2. **任务取消不彻底**：在 `finally` 块中仅调用 `stats_task.cancel()`，但未 `await` 任务完成，可能留下未结束的协程，引发 “Task was destroyed but it is pending!” 警告。  
3. **频繁日志导致磁盘/网络压力**：在日志后端（如 ELK、Datadog）未做限流的情况下，高频率的统计日志可能导致写入激增。  
4. **兼容性**：新增 `args.disable_log_stats` 参数需要在 CLI 参数解析处对应的 `argparse` 配置，否则会出现 “unknown argument” 错误。  

**💡 关注建议**：  
- **默认值与容错**：在 `vllm/envs.py` 为 `VLLM_LOG_STATS_INTERVAL` 提供安全的默认值（如 `60` 秒）并在读取前做类型校验。  
- **任务清理**：在关闭阶段 `await stats_task`（捕获 `CancelledError`）以确保协程完整结束，防止资源泄漏。  
- **日志频率限制**：在 `do_log_stats` 内部加入最小间隔检查或在配置中加入上限阈值，防止意外设置极小间隔。  
- **文档与示例**：在 README 或 CLI 手册中说明 `--disable-log-stats` 与环境变量的使用场景，特别是生产环境的安全建议。  
- **测试覆盖**：补充单元/集成测试：  
  - 验证在启用/禁用状态下 server 启动不会抛异常。  
  - 确认 `stats_task` 能在 `shutdown` 时被正确取消并不会导致未完成的任务。  
  - 模拟极短间隔，确保不会出现明显的性能回退。  

通过以上措施，可在提升可观测性的同时将风险控制在可接受范围。
