### [2/N] Elastic EP Milestone 2: Integrating NIXL-EP (#35627)
**SHA**: `d5af196` | 🔗 [查看提交](https://github.com/vllm-project/vllm/commit/d5af196c183bef2e886c7ec12db63b6161cacfde)

**🎯 变更类型**：功能增强 / 架构变更 / 安全修复（引入可弹性扩缩的 NIXL‑EP All2All 后端）  
**⚡ 重要程度**：🔴高  
**📋 变更摘要**：  
本次提交为 vLLM 添加了对 **NIXL‑EP**（Elastic EP）All‑All 通信后端的完整支持，包括配置项、环境变量、专用 `All2AllManager` 实现、调度器与 `cuda_communicator` 的适配，以及在 CoreClient 中启用 ZMQ ROUTER HANDOVER 以实现弹性 EP 的 “失联‑接管” 功能。目标是让 EP（Expert Parallel）在节点增删或故障恢复时能够 **动态连接/断开**，并利用 NIXL‑EP 通过 RDMA 零 SM 方式提升跨节点通信效率。

**🎯 影响范围**：  
- `vllm/config/parallel.py`（新增 `nixl_ep` 选项）  
- `vllm/distributed/device_communicators/*`（All2All manager、CUDA communicator）  
- `vllm/model_executor/layers/fused_moe/*`（配置、prepare/finalize、量化路径）  
- `vllm/utils/import_utils.py`（`has_nixl_ep` 检测）  
- `vllm/v1/engine/core_client.py`（ZMQ ROUTER HANDOVER）  
- `vllm/envs.py`（`VLLM_NIXL_EP_MAX_NUM_RANKS` env）  
- 相关单元测试及若干辅助文件  

---

### 🔍 技术洞察

| 维度 | 影响描述 |
|------|----------|
| **架构影响** | 1. **新增全局单例 Buffer**：`NixlEPAll2AllManager` 通过类变量 `_buffer` 持有跨进程共享的 `nixl_ep.Buffer`，并在 rank 规模变化时通过 `connect_ranks` / `disconnect_ranks` 动态更新。<br>2. **弹性 EP 路由**：`ParallelConfig.enable_elastic_ep` 与 CoreClient 的 `router_handover` 结合，实现 “失联‑接管”，使得同一身份的 Rank 可在故障后重新加入。<br>3. **配置链路**：`all2all_backend="nixl_ep"` → `ParallelConfig.use_nixl_ep_kernels` → `FusedMoEPrepareAndFinalize` → `NixlEPPrepareAndFinalize`，完整闭环。 |
| **性能影响** | - **通信层**：NIXL‑EP 使用 RDMA，`max_sms_used` 返回 `0`，可显著降低 GPU SM 占用并减少通信延迟。<br>- **调度开销**：首次初始化 Buffer 需要一次 `Buffer.update_memory_buffers` 与 `connect_ranks`，后续弹性扩缩仅做增删操作，成本较低（锁保护的单例）。<br>- **量化路径**：对 FP8 调度提供 `use_fp8_dispatch` 与 `ue8m0` 支持，避免额外的显存拷贝。整体预期在多 EP 场景下 **带宽利用率提升**，但在单机或未启用 NIXL‑EP 时保持原有性能。 |
| **安全考虑** | - 引入了对外部可选依赖 `nixl_ep` 的动态加载，若库被篡改或版本不兼容，可能导致 **运行时异常或内存错误**。<br>- ZMQ ROUTER HANDOVER 会接受来自同一 identity 的新连接，若部署环境未对身份验证做限制，潜在 **网络接入攻击** 风险（虽然 VLLM 仍在内部受信网络中运行）。<br>- 环境变量 `VLLM_NIXL_EP_MAX_NUM_RANKS` 默认 32，如实际 EP 数超过该值而未手动调高，将导致 **缓冲区溢出或连接失败**。 |

---

### ⚠️ 潜在风险

1. **库可用性**：`import nixl_ep` 仅在可选依赖存在时成功，缺失时会抛 `ImportError`。当前实现未在 `has_nixl_ep()` 为 False 时提供后备路径，可能导致启动失败。  
2. **单例 Buffer 线程安全**：虽然使用 `threading.Lock` 包裹 `get_handle`，但 `destroy` 与 `_update_buffer` 之间的状态切换未加锁，极端并发下可能出现 **竞争条件**。  
3. **弹性 EP 边界**：`VLLM_NIXL_EP_MAX_NUM_RANKS` 受限于环境变量，动态扩容超过上限会触发 `assert`，导致进程异常退出。  
4. **兼容性回归**：新增 `use_nixl_ep_kernels` 判定导致部分已有代码路径（如 `determine_expert_placement_strategy`）出现新警告；若用户误配置 `all2all_backend=nixl_ep` 而未装 `nixl_ep`，会在运行时触发 `NotImplementedError`（`dispatch`/`combine` 未实现）。  
5. **ZMQ HANDOVER**：开启后 `router_handover=True` 会让旧连接的未完成请求被新连接直接接管，若上层业务对请求顺序有强依赖，可能出现 **请求顺序错乱**。  

---

| **性能评估** | - 在真实弹性 EP 场景（如容器编排的 pod 横向扩缩）下进行 **基准对比**：NIXL‑EP vs Mori vs FlashInfer，关注 **吞吐/延迟、SM 占用、RDMA 资源使用**。 |

### [Model] Add ColPali late interaction model for multi-modal retrieval (#36818)
**SHA**: `10f08de` | 🔗 [查看提交](https://github.com/vllm-project/vllm/commit/10f08dedfa1636dd477f1ac970827c64f234aad4)

**🎯 变更类型**：功能增强（新增 ColPali 多模态检索模型，支持 Late Interaction）  
**⚡ 重要程度**：🔴 高  

**📋 变更摘要**  
- 在 vLLM 中加入 ColPali 模型实现，基于 PaliGemma 骨干并添加 ColBERT‑style 的线性投影层，实现 **token‑wise** 嵌入与 MaxSim 迟交互评分，可用于多模态检索与重排。  
- 为该模型补齐配置类、处理器、权重映射、模型注册以及文档、测试等支撑代码，使其在 `pooling` 路径下可直接调用 `token_embed` 与 `score` 接口。  

**🎯 影响范围**  
- **模型执行层**：`vllm/model_executor/models/colpali.py`、`registry.py`、`interfaces`、`pooler`。  
- **配置系统**：`transformers_utils/config.py`、`configs/colpali.py` 新增 `ColPaliConfig`。  
- **多模态处理**：`transformers_utils/chat_templates/registry.py`、`multimodal` 注册表。  
- **文档**：`docs/models/supported_models.md`。  
- **测试**：`tests/models/multimodal/pooling/test_colpali.py`、`tests/models/registry.py`。  
- **运行时入口**：通过 `vllm.entrypoints.pooling` 统一调度。  

---

### 🔍 技术洞察  

| 维度 | 影响分析 |
|------|----------|
| **架构影响** | - 新增 **ColPaliModel**，继承自 `PaliGemmaForConditionalGeneration` 并混入 `SupportsLateInteraction`，实现 **Late Interaction** 功能。<br>- 通过 `ColPaliProcessingInfo` 与 `ColPaliMultiModalProcessor` 将自定义 `colpali` 配置映射到原生 `PaliGemmaProcessor`，保持与已有多模态流水线兼容。<br>- 在模型注册表 (`registry.py`) 中加入 `"ColPaliForRetrieval": ("colpali", "ColPaliModel")`，实现统一调度。<br>- `WeightsMapper` 处理多种权重命名（HF checkpoint、colpali‑engine），降低模型迁移成本。 |
| **性能影响** | - 额外的线性投影层 **custom_text_proj**（hidden → embed_dim）在前向仅做一次矩阵乘，耗时可忽略。<br>- **Late Interaction** 采用 MaxSim，对每对查询‑文档的 token 序列进行两两相似度计算，复杂度 O(N·M)。在长文档或大批量检索时 GPU 计算量显著上升，需注意 `gpu_memory_utilization` 与 `max_model_len` 设置。<br>- 输出的 token‑wise 嵌入在 `token_embed` 任务中返回，内存占用与文档长度呈线性关系。<br>- 通过 `torch.norm` 实现 L2‑归一，有助于后续相似度计算的数值稳定性。 |
| **安全考虑** | - 仅添加模型与配置映射，无新增网络请求或外部依赖。<br>- `ColPaliProcessingInfo.get_hf_processor` 强制使用 `PaliGemmaProcessor`，避免 `trust_remote_code=True` 带来的潜在执行风险。<br>- 权重加载采用统一的 `AutoWeightsLoader` 与显式的 `default_weight_loader`，不涉及不安全的 `torch.load` 参数。 |
| **可维护性** | - 代码风格遵循 vLLM 现有模式，使用装饰器 `@default_pooling_type` 与 `@MULTIMODAL_REGISTRY.register_processor` 实现可插拔。<br>- 配置类 `ColPaliConfig` 把所有可能的嵌入维度字段统一暴露，降低因字段命名差异导致的加载错误。<br>- 测试覆盖 **token embed**、**late interaction 评分**、**多模态混合** 三大路径，确保功能正确性。 |

---

### ⚠️ 潜在风险  

1. **投影层初始化失败**  
   - 若 checkpoint 中缺失投影权重或 `embedding_dim` 未能被解析，`custom_text_proj` 会在 `load_weights` 时延迟创建，可能出现 `NoneType` 使用错误。  
2. **内存/算力峰值**  
   - MaxSim 需要在 GPU 上持有完整的查询与文档 token 矩阵，长文本或大批量检索可能触发 OOM。  
3. **兼容性冲突**  
   - 新增的 `colpali` 条目在 `transformers_utils/configs/__init__.py` 与 `config.py` 中的顺序可能与未来新增模型冲突，需要保持唯一 `model_type`。  
4. **测试依赖硬件**  
   - 测例强制 `dtype="half"` 与 `enforce_eager=True`，在仅支持 fp32 的机器上会失败。  
5. **Late Interaction 与现有 Pooler 逻辑耦合**  
   - `pooler.head.projector` 被动态替换，若后续改动改动 `pooler` 接口，可能导致投影层未被正确挂载。  

---

### 💡 关注建议  

- **强化初始化校验**：在 `__init__` 或 `load_weights` 结束后，显式检查 `self.embed_dim` 与 `self.custom_text_proj` 已正确实例化，若未满足抛出友好错误提示。  
- **提供文档/示例**：在官方文档或 README 中补充 **ColPali** 使用场景、**token 长度限制**、**GPU 内存建议** 与 **MaxSim 计算成本** 示例。  
- **添加可选分块评分**：考虑实现分块（sharding）或近似 MaxSim（如 Top‑k）以降低 O(N·M) 计算，在大规模检索时可显著提升吞吐。  
- **测试多硬件兼容**：在 CI 中加入 **fp32** 与 **CPU** 运行路径，确保在不支持半精度的环境下仍能通过。  
- **监控权重映射**：记录 `hf_to_vllm_mapper` 实际匹配的键值对，便于后续模型升级或新增 checkpoint 时快速定位映射缺失。  
- **安全审计**：虽然目前未使用 `trust_remote_code`，仍建议在模型仓库页面注明 “仅在受信任环境下使用”，防止误用自定义代码。  

--- 

**结论**：本次提交为 vLLM 引入了重要的多模态检索模型 **ColPali**，在架构与功能上实现了与现有 PaliGemma 系统的无缝衔接，并提供了完整的测试与配置支撑。风险主要集中在投影层的动态创建与迟交互计算的资源消耗上，建议通过严格的初始化检查、分块评分以及硬件兼容性测试来进一步提升可靠性。整体来看，此功能提升了 vLLM 在跨模态检索领域的竞争力，值得在正式发布前进行上述补强。

### 🔍 技术洞察

| 维度 | 影响分析 |
|------|----------|
| **架构影响** | - 统一了 **FusedMoE** 的执行模型：所有实现现在只需实现 `supports_expert_map` 与 `supports_packed_ue8m0_act_scales`，不再需要区分「支持 Chunk」或「不支持 Chunk」的实现。<br>- 移除 `supports_chunking` 抽象后，`FusedMoEExpertsModular` 基类不再依赖 `envs`，简化了对象模型和配置路径。 |
| **性能影响** | - **正向**：删除了分块循环后，单次调度的 kernel 启动次数大幅下降（原本每 `CHUNK_SIZE` 会启动一次 kernel），在 GPU 上可显著提升吞吐，尤其在 `M`（token 数）远小于原 `CHUNK_SIZE` 时。<br>- **负向**：全量执行需要一次性分配足够的工作空间，峰值显存将随 `M` 线性增长。对非常大的 batch（如 `M` > 64k）可能导致 OOM，需要在上层 `vllm` 调度层做显存预估或分批推理。<br>- **缓存/流水线**：原先的 chunk 机制可以让 GPU 在每块之间更好地重叠计算与通信，删除后需依赖更大的内部 workspace；在多 GPU（DP+EP）场景下，若 `M` 很大且不适配单卡显存，仍可能出现性能回退。 |
| **安全/可靠性** | - 移除 `envs.VLLM_ENABLE_FUSED_MOE_ACTIVATION_CHUNKING` 规避了过去 **torch.compile** 与 Chunk 逻辑产生的 **IMA（Invalid Memory Access）** 问题（#19631），提升了在 JIT/Compiled 环境下的稳定性。<br>- 统一使用 **int64** 计数（`num_tokens * top_k * SPARSITY_FACTOR`），防止了 `int32` 溢出导致的结果错误（已在 `test_fused_moe_int64_overflow` 中验证）。 |
| **可维护性** | - 删除了大量冗余分支（`if self.fused_experts.enable_chunking()`），代码路径简化约 30% 行数，易于阅读、调试。<br>- 不再需要在 `mk_objects.py` 中维护 `supports_chunking` 标记，减少了专家实现注册的复杂度。 |
| **兼容性** | - **向后兼容**：任何依赖 `VLLM_FUSED_MOE_CHUNK_SIZE` 环境变量或 `--fused-moe-chunk-size` CLI 参数的外部脚本将失效，需要更新为 **不使用 Chunk**。<br>- **API 破坏**：`Config.is_fe_supports_chunking()` 已删除，外部代码若仍调用会抛 `AttributeError`。<br>- **测试**：所有受影响的单元测试已相应改动；如果用户自定义测试仍保留旧参数，需要手动迁移。 |
| **部署/运维** | - 环境变量清理后，部署脚本可以去掉 `VLLM_FUSED_MOE_CHUNK_SIZE` 的默认设置，降低配置错误概率。<br>- 需要在生产监控中关注 **显存峰值**，尤其在大 batch 推理或多卡并行时。 |

### ⚠️ 潜在风险

1. **显存峰值激增**  
   - 全量执行会一次性分配 `M * N`（或 `M * K`）大小的工作空间，若 `M` 超过先前 Chunk 限制（默认 16 KB），可能触发 OOM。  
   - 需要在上层 `vllm` 调度器对 `M` 做显存预估，或限制每次推理的 `max_batch_size`。

2. **多卡通信路径的性能回退**  
   - 对于 DP+EP 场景，Chunk 机制可以在每块之间隐藏 All‑to‑All 延迟。删除后，All‑to‑All 与 MoE 计算在同一次 kernel 中完成，可能导致网络瓶颈显现。  
   - 建议在大规模集群中进行基准测试，确认是否需要在 `FusedMoEParallelConfig` 中加入 **内部分块**（而不是外部 Chunk）。

3. **旧版插件或自定义 MoE 实现**  
   - 如果用户自行基于 `FusedMoEExpertsModular` 实现了 `supports_chunking` 并在外部依赖该标识，编译会失败。需更新自定义实现为 **不实现** 该方法，或迁移至新版基类。

4. **测试覆盖差异**  
   - 部分原本依赖 Chunk 的边界用例（如极小 `M`、极大 `M`）已被删除，可能导致回归未被检测。建议补充 **显存极限** 和 **大 token** 场景的专门测试。

### [ROCm][CI] Preparing gfx90a mirroring (#36210)
**SHA**: `cc8f1f4` | 🔗 [查看提交](https://github.com/vllm-project/vllm/commit/cc8f1f47644868869d5a7fb4c55cebbf91fb9943)

- **核心改动**  
  1. **Docker 镜像构建**：在 `.buildkite/hardware_tests/amd.yaml` 中的 `ARG_PYTORCH_ROCM_ARCH` 从 `gfx942;gfx950` 扩展为 `gfx90a;gfx942;gfx950`，因此镜像将在构建时包含对 gfx90a 的 PyTorch ROCm 支持。  
  2. **CI 测试流水线**：在 `.buildkite/test-amd.yaml` 中一次性新增 **上千行**的步骤，几乎把原有的全部 AMD CI 流程复制为 “mirrored” 版本，针对新硬件标签 `amdgfx90a`（对应 MI250 GPU），并为每一步指定 `mirror_hardwares: [amdexperimental, amdproduction, amdgfx90a]`。这些步骤覆盖：依赖检查、单元/集成测试、分布式/多节点测试、模型/多模态/LoRA/插件/量化/基准/LM‑Eval 等几乎所有已有的 AMD 测试场景。

**影响范围**  
| 受影响模块 | 具体影响 |
|------------|----------|
| `docker/Dockerfile.rocm` | 构建镜像时会额外安装 PyTorch 对 gfx90a 的支持；若该架构的二进制包在 ROCm 5.6+ 中缺失，构建会失败。 |
| Buildkite CI 代理 | 需要 **`mi250_1 / mi250_2 / mi250_4 / mi250_8`** 这四类 GPU‑agent 池以及 **`amdgfx90a`** 硬件标签。如果这些标签/池未在 Buildkite 配置中声明，所有新增步骤会被标记为 “skipped” 或直接报错。 |
| 测试脚本/依赖 | 大量步骤复用了已有的 `source_file_dependencies`，但在 AMD 环境下会额外触发 `torch_nightly`、`torch_nightly`、`no_gpu`、`optional` 等标记。依赖的 Python 包（如 `torchcodec`, `helion`, `mamba` 等）必须在 ROCm 镜像中可用，否则会在执行阶段报错。 |
| 资源消耗 | 新增的 30++ 条测试大幅提升 CI 时长和 GPU 资源占用，可能导致排队时间增长，尤其是 `optional: true` 的步骤如果被打开会消耗大量机器。 |

**潜在风险**  
1. **构建失败** – `ARG_PYTORCH_ROCM_ARCH='gfx90a;...'` 需要对应的 PyTorch 预编译包。如果官方尚未发布或镜像仓库不存在该包，`docker build` 会中断。  
2. **Agent/标签缺失** – Buildkite 上若未提前创建 `amdgfx90a` 硬件标签或对应 `mi250_*` 池，所有步骤会报 “No matching agent found”。  
3. **依赖冲突** – 某些测试在 ROCm 环境下使用了 `torch_nightly`、`torchcodec`、`mamba` 等外部仓库的特定分支，若这些分支不兼容当前 ROCm 版号，可能导致运行时错误。  
4. **超时/资源争用** – 部分步骤（如分布式 8‑GPU、全图编译、LM‑Eval 大模型）本就接近 90 min 超时阈值，复制后在 GPU 资源紧张时容易因排队或超时导致 CI 失效。  

**建议**  
- **镜像验证**：在正式 CI 之前，单独跑一次 `docker build`（含 `gfx90a`）并确认 `torch`、`torchvision`、`torchtext` 对该架构的 wheel 能成功下载并安装。  
- **添加硬件标签**：在 Buildkite UI 或配置文件中确保 `amdgfx90a` 与相应 `mi250_*` 代理池已注册；若使用自定义硬件映射，需同步到所有 `mirror_hardwares` 列表。  
- **分阶段开启**：先将关键的 “mirrored” 步骤（如依赖检查、基本正确性、Engine Test）设为默认运行，其余 `optional: true` 的大规模分布式/量化/LM‑Eval 步骤可在需求突增时手动打开，以免占满 GPU 资源。  
- **监控构建时间**：在 Buildkite 中开启步骤时间阈值报警，关注新增的长跑任务（>60 min）是否经常触发超时，必要时拆分为更细的子任务或调低 `timeout_in_minutes`。  
- **依赖锁定**：对 `torch_nightly`、`mamba`、`torchcodec` 等 nightly 或特定 commit 的依赖使用固定的 SHA/标签，防止上游突变导致 CI 非预期失败。  
- **文档更新**：在项目 README/CI 章节添加 “AMD gfx90a 支持” 的说明，列出需要的机器规格、镜像标签以及如何在本地复现（例如 `docker pull rocm/vllm-ci:<commit>`）。  

**结论**  
本次提交是对 CI 系统的 **功能增强**，旨在把整个 AMD 测试套件完整迁移到新的 `gfx90a`（MI250）平台。代码本身未改动，风险主要位于构建镜像和 CI 环境的硬件/标签准备。若上述硬件标签、镜像包及依赖均已就绪，改动能够提升对最新 AMD GPU 的覆盖率；否则可能导致 CI 构建或测试阶段的批量失败，建议在正式合入前在内部预跑一次完整的 “mirrored” 流水线。
