### [vLLM IR] 2/N fused_add_rms_norm and maybe_inplace overload (#36823)
**SHA**: `d58c42e` | 🔗 [查看提交](https://github.com/vllm-project/vllm/commit/d58c42e19cb792e24eb335b75164356a4f71bff0)

**🎯 变更类型**：功能增强 / 架构变更  
**⚡ 重要程度**：🔴高  

**📋 变更摘要**  
本次 PR 通过在 `vllm.ir` 中引入 **Functional Intermediate Representation (IR)**，为运算符提供语义层与实现层的分离。  
- 新增 `@register_op(..., allow_inplace=True)` 与 `maybe_inplace` overload，实现 **捐赠（donated）激活** 的零拷贝路径。  
- 在编译前加入 **VllmIRInplaceFunctionalizationPass**：把 `maybe_inplace` 调用转换为默认 overload 并检测激活是否被后续再次使用。  
- 实现 **VllmIRLoweringPass**：依据 `ir_op_priority` 配置在编译期选取具体实现（native、vllm_c、aiter、oink 等），并在 `replace_by_example` 时保留原始实现函数，以便在原位实现时插入 clone。  
- 新增 **UnsafeCloneEliminationPass**：在 lowering 之后安全地删除冗余 `torch.ops.aten.clone.default` 节点，利用 `PassContext.donated_input_ids` 判断图输入是否可被捐赠。  
- 将原来在 `layernorm.py`、`_aiter_ops.py`、`_oink_ops.py` 中的 RMSNorm/ fused‑add‑RMSNorm 实现迁移至 IR，原有 AITER RMSNorm 代码被移除或禁用。  
- 更新配置 (`IrOpPriorityConfig`) 与平台默认（CUDA、ROCM、XPU）以支持 `fused_add_rms_norm` 的优先级列表。  
- 大量测试覆盖：IR 注册、dispatch、图追踪、功能化、clone‑消除、端到端编译、分段编译、Triton 交叉、错误检测等。  

**🎯 影响范围**  
- **核心 IR 代码**：`vllm/ir/op.py`, `vllm/ir/ops/*`（layernorm、fused_add_rms_norm）  
- **编译 Pass**：`vllm/compilation/passes/ir/*`（functionalization、lowering、clone_elimination、utils）  
- **后端管理**：`vllm/compilation/backends.py`（新增 pre‑grad pass 注入）  
- **Fusion Passes**（AllReduce、RMS‑Quant、ROCM‑Aiter、SequenceParallelism 等）全部改为直接调用 `vllm.ir.ops.fused_add_rms_norm` 代替原 RMSNorm custom‑op。  
- **平台配置**：`vllm/platforms/cuda.py`, `vllm/platforms/rocm.py`（默认优先级扩展为 `fused_add_rms_norm`）  
- **模型层**：`vllm/model_executor/layers/layernorm.py`（改为通过 IR 调用，去除 Oink 检测逻辑）  
- **文档**：新增 IR 设计文档与调试指南。  
- **测试**：新增 400+ 行 IR/clone/functionalization 单元测试、若干集成测试、对现有融合/功能化测试的适配。  

**🔍 技术洞察**  

| 维度 | 影响 |
|------|------|
| **架构** | - **IR 层** 把语义 (rms_norm、fused_add_rms_norm) 与实现 (native、vllm_c、aiter、oink…) 解耦，提供统一的 **priority 配置**。<br>- **编译管线** 现在在 **pre‑grad** 阶段执行 `VllmIRInplaceFunctionalizationPass`，随后 AOTAutograd → functionalization → **IR fusion** → **IR lowering** → **clone‑elimination** → Inductor。<br>- 引入 **PassContext.donated_input_ids**，在整个编译阶段共享捐赠信息，避免在后置 Pass 中再次分析。 |
| **性能** | - **maybe_inplace** 允许底层实现直接复用激活内存（`x`、`residual`），在 LLM 的残差路径上可实现 **0 额外分配**，显著降低显存占用与拷贝带宽。<br>- **CloneEliminationPass** 删除因 functionalization 引入的多余 `clone`，进一步削减显存与算子调度开销。<br>- 统一的 **kernel priority** 与 **IR‑level fusion** 较原先的 CustomOp + 手写 Fusion 更易实现跨平台统一优化，理论上提高整体吞吐。<br>- 对不支持 in‑place 实现的 provider（如 native）仍保持功能完整，无性能倒退。 |
| **安全 / 稳定性** | - 新增 **donated‑input 检查**：若在 `maybe_inplace` 调用后仍使用捐赠张量，将在编译阶段抛 `ValueError`（包装为 `BackendCompilerFailed`），防止**未定义行为**导致错误结果或显存泄漏。<br>- `UnsafeCloneEliminationPass` 采用保守写检测（最高写检测、HOP 保守假设），在当前已知的 vLLM 用例下安全；但若未来出现更复杂 alias（如 view、切片）仍需增强。<br>- 移除 AITER RMSNorm 代码，避免在不满足 SM100 条件下误用 AITER 实现，提升跨平台正确性。 |
| **可维护性** | - **统一注册 API** (`@register_op`, `register_impl`) 取代了历史的 `CustomOp` 类层级，降低新运算符上线的学习成本。<br>- 所有实现均集中在 `vllm.ir.ops`，实现分散、查询方便。<br>- 通过 **priority 列表** 支持用户在命令行/Python 中灵活覆盖，兼容外部插件（AITER、Oink、Triton）。<br>- 测试覆盖率显著提升（包括 clone、donated‑input、错误路径），有助于后续迭代。 |
| **兼容性** | - 默认优先级仍是 `["vllm_c", "native"]`（CUDA/XPU），保持现有行为不变。<br>- 对 ROCm，默认仍 `["native"]`，但若用户显式开启 `VLLM_USE_AITER_RMSNORM`，会在 `IrOpPriorityConfig` 中加入 `"aiter"`。<br>- 现有 `RMSNorm.enabled()`（用于 ROCm AITER）已在代码中删除，外部插件需通过 IR 调用或自行配置优先级。 |
| **调试/可观测性** | - `VllmIRInplaceFunctionalizationPass`、`UnsafeCloneEliminationPass` 在 `VLLM_LOGGING_LEVEL=DEBUG` 时会打印捐赠 ID 与删除的 clone 数量，便于定位显存异常。<br>- 文档新增 `debug_vllm_compile.md` 与 `vllm_ir.md`，提供 **torch.compile 日志获取** 与 **IR 设计** 指南。 |

**⚠️ 潜在风险**

### [Attention] Abstract the MLA prefill backends and eliminate cuDNN (#32623)
**SHA**: `f3fef12` | 🔗 [查看提交](https://github.com/vllm-project/vllm/commit/f3fef123504db07b3ac83ad4ef677915b53e8386)

**🎯 变更类型**：架构变更 + 功能增强 + 重构  
（为 MLA（Multi‑Linear‑Attention）引入统一的 prefill 后端抽象，废除 cuDNN 实现，新增 `mla_prefill_backend` 配置项并实现自动/显式后端选择逻辑）

---

**⚡ 重要程度**：🔴 高  
- 影响 **attention** 关键路径，涉及模型推理期间的大量算子调用。  
- 迁移旧标志（`use_cudnn_prefill`、`disable_flashinfer_prefill`、`use_trtllm_ragged_deepseek_prefill`）为新字段，若迁移不彻底会导致运行时错误。  
- 新增的后端选择机制直接决定推理性能与内存占用。

---

**📋 变更摘要**  
1. **文档**：更新注意力后端表格，说明 `-ac.mla_prefill_backend=<BACKEND>` 用法；删除 cuDNN 相关说明。  
2. **配置**：`AttentionConfig` 增加 `mla_prefill_backend`（枚举），并在 `__post_init__` 中迁移旧标志，抛出 cuDNN 已被移除的错误。  
3. **后端抽象**：新增 `vllm.v1.attention.backends.mla.prefill` 包，包含  
   - `MLAPrefillBackend` 抽象基类  
   - 三实现：`FlashAttnPrefillBackend`、`FlashInferPrefillBackend`、`TrtllmRaggedPrefillBackend`  
   - `MLAPrefillBackendEnum` 注册表与懒加载机制。  
4. **后端选择器**：`selector.py` 根据硬件 Compute Capability、模型维度（DeepSeek‑R1 兼容性）以及用户显式指定，自动挑选可用后端；实现缓存 (`functools.cache`) 以避免重复计算。  
5. **核心实现**：`mla_attention.py` 重构为通过统一的 `MLAPrefillBackend` 实例 (`self._prefill_backend`) 完成 prefill 前向；移除原先的多分支 `use_*_prefill` 逻辑。  
6. **测试**：新增后端选择单元测试、更新已有测试以去除 cuDNN 标志、覆盖显式/自动选择路径。  
7. **其它**：`DeviceCapability` 加入 `__hash__` 以支持缓存键；移除不再使用的 `CUDNN_WORKSPACE_SIZE` 相关代码。  

---

**🎯 影响范围**  
- **attention/backends/mla**：所有 MLA 相关实现（prefill 及 decode）  
- **config/attention.py**：配置结构、字段解析、校验逻辑  
- **platforms/interface.py**：`DeviceCapability` 可哈希化（被后端选择缓存使用）  
- **文档** `docs/design/attention_backends.md`  
- **测试套件** `tests/v1/attention/*`、`tests/engine/test_arg_utils.py`  
- **代码生成脚本** `tools/pre_commit/generate_attention_backend_docs.py`（生成文档的 AST 解析逻辑）

**🔍 技术洞察**  

| 维度 | 影响说明 |
|------|----------|
| **架构影响** | 通过 `MLAPrefillBackend` 抽象，将原本散落在 `mla_attention.py` 的多分支实现统一到插件式后端。后端通过统一接口 (`run_prefill_new_tokens`、`run_prefill_context_chunk`) 与元数据 (`prepare_metadata`) 交互，极大提升可维护性与可扩展性（后续可无侵入式添加新后端）。 |

### [Eval][CI] Add basic mrcr eval to tests/evals/ (#40164)
**SHA**: `3ccc1ff` | 🔗 [查看提交](https://github.com/vllm-project/vllm/commit/3ccc1ff4958dd07dbffeaa1c48463325c892b518)

**🎯 变更类型**：功能增强（新增 MRCR 长上下文评估及对应 CI 步骤）  
**⚡ 重要程度**：🔴中  
**📋 变更摘要**：在 `tests/evals/mrcr` 中加入了对 OpenAI MRCR 数据集的长上下文准确性评估脚本、配置与 pytest 用例，并在 Buildkite CI 中新增了 “MRCR Eval Small Models” 步骤，使得小模型的长上下文回归能够在 CI 中被捕获。  

**🎯 影响范围**：  
- CI 配置文件 `.buildkite/test_areas/lm_eval.yaml`  
- 新增测试模块 `tests/evals/mrcr/`（README、配置、conftest、评估脚本、pytest 用例）  
- 依赖 `datasets` 包（通过 `uv pip install --system 'gpt-oss[eval]==0.0.5'` 已在 CI 中包含）  
- 与现有评估框架的交互（`RemoteOpenAIServer`、`pytest` 参数化）  

**🔍 技术洞察**：

- **架构影响**  
  - 采用 **黑盒** 调用方式：通过 HTTP `/v1/models`、`/tokenize` 与正在运行的 vLLM 服务交互，未侵入服务器内部代码，保持了现有架构的解耦。  
  - 通过 `tests/evals/mrcr/conftest.py` 实现基于配置文件的参数化，实现了 **可插拔** 的评估插件化模式，与项目已有的评估插件体系保持一致。  
  - 新增的 `mrcr_eval.py` 使用 `aiohttp` 异步并发请求，遵循项目已经采用的 **asyncio + tqdm** 进度条风格，保持代码风格统一。  

- **性能影响**  
  - 评估过程会 **并发 (默认 8)** 调用服务器的 `/v1/chat/completions`，在 CI 环境下可能导致显著的网络 I/O 与服务器负载，已通过 `timeout: 30 min` 控制最大耗时。  
  - 预筛选逻辑（字符数 × 4 ≤ max_prompt_tokens、二次 `/tokenize` 校验）有效避免了超长请求造成的服务器 OOM。  
  - 评估脚本统计了 `tokens_per_second`，为后续性能基准提供了可观测指标。  

- **安全考虑**  
  - 依赖外部 HuggingFace 数据集 `openai/mrcr`，在 CI 环境中通过 `datasets` 动态下载，属于 **受信数据**；若在生产环境使用，需要审计数据来源。  
  - `RemoteOpenAIServer` 启动时加入了 `--trust-remote-code`，允许模型自带的自定义代码执行，已在 CI 中显式声明，需注意在非受信环境中禁用。  
  - 评估请求中默认加入 `extra_body = {"chat_template_kwargs": {"enable_thinking": False}}`，防止推理模型的 chain‑of‑thought 影响评分，降低了因模板泄露导致的评分偏差风险。  

**⚠️ 潜在风险**  

1. **网络/数据集不稳定**：CI 依赖实时下载 MRCR parquet 文件，网络抖动或 HuggingFace 服务不可用会导致 CI 失效。  
2. **服务器启动超时**：长上下文模型（如 `--max-model-len 128K`）启动耗时较长，若 `startup_max_wait_seconds` 配置不足会误报失败。  
3. **资源竞争**：并发请求数默认 8，若 CI 机器资源受限（CPU、内存或网络带宽），可能出现超时或 OOM。  
4. **依赖漂移**：`datasets` 包未在项目根 `requirements.txt` 中锁定版本，未来更新可能引入兼容性问题。  
5. **误判阈值**：不同模型的长上下文表现差异大，单一阈值（如 `match_ratio_threshold: 0.99`）可能导致频繁回归，需根据模型调校。  

**💡 关注建议**  

- **CI 稳定性**：在 CI 中加入缓存层（如 `cache: pip`、`cache: datasets`）或将 MRCR shard 预先打包到构件，以降低网络波动影响。  
- **资源配置**：根据机器规格适当调低 `concurrency`（如 4），或在 CI 步骤中显式限制 `--max-model-len` 与 `max_prompt_tokens`，防止 OOM。  
- **依赖锁定**：在 `requirements.txt` 或 `uv.lock` 中添加 `datasets==2.16.*`（或当前兼容版本），避免因上游升级导致突发错误。  
- **阈值管理**：将阈值配置抽离为单独文件，并在 PR 中提供更新日志，便于模型迭代时快速调整。  
- **安全审计**：在非 CI 环境使用该评估时，务必关闭 `--trust-remote-code`，并确认下载的 `mrcr` 数据集满足内部合规要求。  

通过上述措施，可在保持长上下文回归检测能力的同时，最大限度降低 CI 失误与生产环境的安全风险。

### [Model Runner V2] Add `logprob_token_ids` support (#40559)
**SHA**: `5129579` | 🔗 [查看提交](https://github.com/vllm-project/vllm/commit/51295793a2eed0eefc7505cb9a7d5f96effd7773)

**🎯 变更类型**：功能增强（为 `SamplingParams` 增加 `logprob_token_ids` 支持）  
**⚡ 重要程度**：🟡 中  

**📋 变更摘要**  
1. 在 `SamplingParams` 中新增常量 `MAX_LOGPROB_TOKEN_IDS`（上限 128）以及属性 `num_logprobs`，使 `logprobs` 与 `logprob_token_ids` 能统一计数。  
2. 增加对 `logprob_token_ids` 长度的校验。  
3. 调度器、Logprobs 处理器以及 GPU 采样路径改为使用 `num_logprobs`，并在产生 logprob 时加入自定义 token‑id 列表的逻辑。  
4. 新增 `LogprobTokenIdsState`（管理每个请求的自定义 token‑id），以及相应的 Triton 填充 kernel `_fill_logprob_token_ids_kernel`，实现批量、零拷贝的 token‑id 合并。  

**🎯 影响范围**  
- `vllm/sampling_params.py`（参数校验、属性实现）  
- `vllm/v1/core/sched/scheduler.py`、`vllm/v1/engine/logprobs.py`（调度与后处理）  
- `vllm/v1/worker/gpu/sample/logprob.py`、`sampler.py`（GPU 采样与 logprob 计算）  
- 新增 `vllm/v1/worker/gpu/sample/logprob.py` 中的 `LogprobTokenIdsState` 与 Triton kernel  
- 相关的 buffer 工具 `StagedWriteTensor`、`UvaBackedTensor` 的使用  

**💡 关注建议**  
1. **兼容性**：若业务仅使用 `logprobs` 而不提供 `logprob_token_ids`，行为保持不变；但代码中仍应改为通过 `sampling_params.num_logprobs` 判断。  
2. **性能**：自定义 token‑id 仅在存在时触发额外的 Triton 填充与 mask 计算，开销与 `max_per_req_token_ids` 成正比，请在生产环境评估其对吞吐的影响。  
3. **限制**：当前最大长度 128 如有需求请在 `MAX_LOGPROB_TOKEN_IDS` 中调整并同步 GPU‑side 缓冲区大小。  
4. **测试**：加入单元/集成测试，覆盖以下情形：  
   - `logprobs` 与 `logprob_token_ids` 同时为空  
   - 仅 `logprobs` 为正数  
   - 仅 `logprob_token_ids`（不同长度）  
   - 同时使用两者（`logprobs` 为 `None`）  
5. **监控**：可在运行时打印 `sampling_params.num_logprobs` 与 `max_per_req_token_ids`，确保请求的自定义列表未超过上限。  

整体来看，此次改动为用户提供了更细粒度的 logprob 查询能力，对核心调度与 GPU 采样路径做了必要适配，维持向后兼容的同时略增 GPU 计算与内存占用。建议在更新后验证所有涉及 `logprobs` 的下游模块（如结果解析、削峰）能够正确处理可能出现的 `-inf` 掩码值。
