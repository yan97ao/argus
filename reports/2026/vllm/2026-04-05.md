### [Perf] Change Trtllm fp8 MoE to use Shuffled Weights and BlockMajorK Layout (#38993)
**SHA**: `1af6f78` | 🔗 [查看提交](https://github.com/vllm-project/vllm/commit/1af6f78ae5a1bd3d70f32d47fe4901bda5c97fdd)

**🎯 变更类型**：性能优化 / 功能增强  
**⚡ 重要程度**：🟡 中  

**📋 变更摘要**  
1. 为 TRT‑LLM 的 FP8 MoE 增加 **Shuffled Weights + BlockMajorK** 布局的支持，新增 `WeightLayout` 参数并在 `moe_problem_size` 中对 4‑D 权重 (E, K/bk, Mn, bk) 进行尺寸解析。  
2. 在 `flashinfer_utils.py` 中实现 ` _shuffle_deepseek_fp8_moe_weights`，将 DeepSeek‑FP8 权重经 `shuffle_matrix_a` → `convert_to_block_layout` 处理后转为 BlockMajorK 形式。  
3. 关联代码在 `trtllm_fp8_moe.apply`、`_apply_block_scale` 中传递新的 `weight_layout`，并统一 `use_shuffled_weight=True`（原 DeepSeek 分支之前是 `False`）。  
4. `deep_gemm_warmup.py` 的检测逻辑改为通过 `moe_kernel` 判断是否使用 DeepGemm，实现方式更稳健。  

**🎯 影响范围**  
- `vllm/model_executor/layers/fused_moe/experts/trtllm_fp8_moe.py`（核心 MoE 调度）  
- `vllm/model_executor/layers/quantization/utils/flashinfer_utils.py`（权重预处理）  
- `vllm/model_executor/warmup/deep_gemm_warmup.py`（热身路径判定）  
- 依赖 `flashinfer` 1.0+ 版本的 `WeightLayout` 枚举和 `shuffle_matrix_a / convert_to_block_layout` 接口。  

**💡 关注建议**  
1. **兼容性检查**：确保运行环境的 FlashInfer 版本提供 `WeightLayout.BlockMajorK` 与 `WeightLayout.MajorK`，否则会在导入时抛异常。  
2. **回归测试**：新增对 DeepSeek‑FP8（非 MXFP8）路径的单元/集成测试，验证 `use_shuffled_weight=True` 与 `weight_layout=BlockMajorK` 在不同 expert 数量、batch 大小下的数值正确性与吞吐提升。  
3. **性能基准**：对比旧版 DeepSeek‑FP8（不 shuffle）与新版（shuffle+BlockMajorK）在实际推理场景（e.g., 80B 模型）下的 latency / throughput，确保改动带来预期提升且不会显著增加显存占用。  
4. **文档同步**：在 vLLM 的 Quantization / TRT‑LLM 使用说明中加入 “DeepSeek‑FP8 需要开启 shuffled weights” 的提示，并说明何时使用 `WeightLayout.MajorK`（MXFP8） vs `BlockMajorK`（DeepSeek‑FP8）。  
5. **异常监控**：在 `apply` / `_apply_block_scale` 中加入断言或日志，检查 `weight_layout` 与实际权重维度匹配，防止因旧模型权重未经过 shuffle 导致错误。  

总体来看，此次改动为 TRT‑LLM FP8 MoE 引入了更高效的权重布局，预计在 DeepSeek‑FP8 场景下可提升算子利用率和吞吐。务必在多卡、不同 batch‑size、不同 expert‑count 的实际部署中完成验证，以防出现布局不匹配或显存回收不当的问题。

### [Bugfix][Spec Decode] Fix extract_hidden_states for VLM models (#38987)
**SHA**: `9a52826` | 🔗 [查看提交](https://github.com/vllm-project/vllm/commit/9a528260ef648500262709550807c292098a70c0)

**🎯 变更类型**：Bug 修复（Spec Decode）  
**⚡ 重要程度**：🟡 中  

**📋 变更摘要**  
- `ExtractHiddenStatesConfig` 在接收包含嵌套 `text_config`（如 LLaVA、Kimi‑K2.5） 的 VLM 配置时，会把 `text_config` 通过 `to_dict()` 展平为普通 `dict`，导致后续 `get_hf_text_config()` 无法直接访问模型属性。  
- 新增逻辑：在构造阶段保存原始 `PretrainedConfig` 对象（`source_text_config`），并在合并后重新写回 `combined["text_config"]`，确保序列化仍为 `dict`，但内部保留对象供属性查找。  
- 对应测试用例覆盖了：① 纯文本模型保持兼容；② VLM 配置正确保留 `text_config`；③ 丢失必需属性时仍抛出 `ValueError`。  

**🎯 影响范围**  
- `vllm/transformers_utils/configs/extract_hidden_states.py`（核心配置类）  
- `vllm/v1/spec_decode/extract_hidden_states.py`（使用该配置的推理路径）  
- 相关单元测试 `tests/v1/spec_decode/test_extract_hidden_states.py`  

**💡 关注建议**  
1. **兼容性检查**：确认项目中其他读取 `hf_config` 或 `hf_text_config` 的代码仍能接受 `text_config` 为 `PretrainedConfig`（而非纯 `dict`），尤其是序列化/反序列化路径。  
2. **文档更新**：在 `ExtractHiddenStatesConfig` 的使用说明中标明对 VLM（复合）模型的特殊处理，解释 `to_dict()` 会自动展平的行为。  
3. **序列化一致性**：虽然对象在内部被保留，`to_dict()` 与 `to_json_string()` 仍应输出平铺的字典结构，防止 downstream 工具误以为仍是对象。已有测试验证，但建议在 CI 中加入对 `json.dump` 的兼容性检查。  
4. **异常路径**：`get_hf_text_config` 现在会在缺失 `num_attention_heads` 等关键属性时抛错，确保错误信息明确，避免静默失败。可考虑在 `ExtractHiddenStatesConfig.__init__` 里提前校验，以更早发现配置错误。  

总体来看，此次修复修正了 VLM 场景下的隐藏状态提取错误，保持了对传统文本模型的向后兼容，测试覆盖充分，风险有限。后续关注上述兼容性与文档即可。

**🎯 变更类型**：功能移除 / 兼容性调整  
**⚡ 重要程度**：🟡 中（只影响使用 Petit NVFP4 量化的少数用户）  
**📋 变更摘要**：本次 PR 将 vLLM 中对 AMD GPU 的 “Petit NVFP4” 量化后端全部删除。相关的依赖声明、枚举、配置验证、实现代码以及 `rocm` 平台的特征列表均被剔除。  

**🎯 影响范围**

- `vllm/config/model.py`、`vllm/model_executor/layers/linear.py`、`vllm/model_executor/layers/quantization/__init__.py`：删除对 `petit_nvfp4` 的枚举与导入。  
- `vllm/model_executor/layers/quantization/petit.py`、`utils/petit_utils.py`：完整删除实现文件（约 440 行）。  
- `vllm/platforms/rocm.py`：在 ROCm 平台特性列表中去掉 `petit_nvfp4`。  

**💡 关注建议**  

1. **使用者**：若项目曾显式指定 `quantization="petit_nvfp4"`，或在 `hf_quant_config.json` 中使用 `NVFP4`，将会在启动时抛出 “unsupported quantization” 错误。请改为使用已保留的 AMD 相关后端，如 `modelopt_fp4`、`mxfp4`，或迁移到 NVIDIA GPU 支持的 FP8、AWQ 等。  

2. **文档/示例**：检查并更新所有文档、示例脚本以及 CI 配置中出现的 “Petit” 关键字，防止用户误导。  

3. **兼容性**：代码已删除对 `petit-kernel` 的可选依赖声明，pip 安装时不再拉取该包，减小体积并避免不必要的 ImportError。  

4. **回归测试**：新增或加强针对 AMD GPU（ROCm） 的量化路径测试，确认除 `petit` 外的其他量化后端仍能正常加载。  

5. **后续维护**：若未来计划重新引入 AMD 专属量化，请在 `setup.py` 中恢复 extras，并在平台特性列表中加入相应标记，避免再次出现残留引用导致的导入错误。

### Revert "[vLLM IR] gemma_rms_norm" (#38998)
**SHA**: `7d266ab` | 🔗 [查看提交](https://github.com/vllm-project/vllm/commit/7d266abb2254a5f8afad0901be11284b7c922e84)

**🎯 变更类型**：Bug 修复 / 重构（恢复 Gemma‑RMS‑Norm 的原始行为）  
**⚡ 重要程度**：🟡 中  
**📋 变更摘要**：  
1. 将 `ir.ops.rms_norm` 中的 dtype 处理恢复为最初的实现：在归一化后先恢复原始 dtype，再在需要时乘以 `weight`（不再强制把 `weight` 转为 `x` 的 dtype）。  
2. 简化各后端 kernel（AITer、vLLM‑C、XPU）对 `rms_norm` 的支持判断，只保留 “不支持 `variance_size`” 的限制，去掉对 weight dtype 必须相同的限制。  
3. 在 `model_executor.layers.layernorm` 中新增纯 PyTorch 实现 `_forward_static_no_residual` / `_forward_static_with_residual`，并在 `forward_cuda` 中对这些实现进行 `torch.compile`（仅在编译模式下使用 native 实现），以保证在没有自定义 kernel 时仍能得到与 Gemma‑RMS‑Norm 相同的数值结果。  

**🎯 影响范围**：  
- `vllm/ir/ops/layernorm.py`（IR 层）：dtype 处理细节改变。  
- `vllm/kernels/aiter_ops.py`、`vllm/kernels/vllm_c.py`、`vllm/kernels/xpu_ops.py`（后端 kernel 条件判定）。  
- `vllm/model_executor/layers/layernorm.py`（模型执行层）：增加静态实现并在 CUDA 路径上使用 `torch.compile`。  

**💡 关注建议**：  
1. **数值验证**：在混合精度（fp16/bf16）和不同模型（Gemma、Llama）上跑完整的推理基准，确认 `forward_native` 与原 kernel 的输出误差在可接受范围内。  
2. **编译兼容性**：`torch.compile` 只在编译模式下生效，确保 CI 中包含 `TORCH_COMPILE=1` 或相应的测试场景，避免在未编译时仍走旧实现导致潜在回归。  
3. **后端清理**：既然已经放宽了 weight dtype 的限制，后续可考虑删除过时的 `weight.dtype` 检查注释，保持代码简洁。  
4. **文档更新**：说明 `rms_norm` 现在不再要求 weight 与输入 dtype 匹配，以及 `forward_cuda` 在编译模式下的行为。  

总体来看，此次回滚恢复了 Gemma‑RMS‑Norm 的数值兼容性，并通过纯 PyTorch 实现提供了安全的后备路径，影响主要集中在层归一化相关模块，风险可通过上述测试与文档同步降低。

### [vLLM IR] gemma_rms_norm (#38780)
**SHA**: `156405d` | 🔗 [查看提交](https://github.com/vllm-project/vllm/commit/156405d243924fbede2d4360494a81dea7203334)

**🎯 变更类型**：功能增强 / 重构  
**⚡ 重要程度**：🟡 中  

**📋 变更摘要**  
- 将 RMS‑Norm 实现统一为 `ir.ops.rms_norm`，在 `LayerNorm` 中去除手写的 FP16/FP32 统计路径及 Torch‑compile 的缓存逻辑。  
- 为 CPU/GPU/XPU/AITER 三类内核统一加入 *weight dtype 必须与输入 dtype 相同* 的检查，并在 IR 实现中先将 `x` 转回原始 dtype，再在需要时提升到 `weight` 的 dtype 进行乘法。  
- 对 `gemma` 系列模型的权重做了 “+1.0” 预处理，使其兼容原始实现的 `1 + gamma` 形式。

**🎯 影响范围**  
- `vllm/ir/ops/layernorm.py`（RMS‑Norm IR）  
- `vllm/kernels/*_ops.py`（AITer、CUDA、XPU 三套 kernel 注册）  
- `vllm/model_executor/layers/layernorm.py`（高层 LayerNorm 调用路径）  

**💡 关注建议**  
1. **dtype 对齐**：新加入的 `weight.dtype == x.dtype` 检查会在混合精度场景（如 fp16 输入、fp32 权重）抛出错误，确保模型权重在加载时已转换为相同 dtype，或在 `LayerNorm` 中使用 `weight.data.float() + 1.0` 的方式统一 dtype。  
2. **残差路径**：残差加法现已在 `forward_native` 中先做 dtype‑提升，随后统一调用 `ir.ops.rms_norm`，请验证 FP16‑only 推理时的数值误差是否仍在可接受范围。  
3. **后端回退**：若某平台不满足 “weight dtype 与 x 相同” 的约束，内核会回退到 Python 实现。请在 CI 中加入不同 dtype 组合的覆盖测试，确保回退路径维持原有行为。  
4. **性能验证**：去掉 `torch.compile` 的缓存后，理论上调用一次 `ir.ops.rms_norm` 即可得到同等性能。建议在真实推理工作流中对比前后速度，确保没有意外的编译开销。  

总体来看，本次改动统一了 RMS‑Norm 的实现逻辑，提升了代码可维护性并对多后端做了更严格的 dtype 检查。只要在模型加载和推理前保证权重 dtype 与输入一致，即可平滑迁移。
