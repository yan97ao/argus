### [Mistral Grammar] Support Grammar Factory (#38150)
**SHA**: `fef56c1` | 🔗 [查看提交](https://github.com/vllm-project/vllm/commit/fef56c18555e881c671acf654630732b7271c14f)

**🎯 变更类型**：功能增强/架构变更  
**⚡ 重要程度**：🔴高  
**📋 变更摘要**：  
- 引入 **Mistral Grammar Factory**，为 Tekken 版 Mistral 分词器提供 `grammar_factory` 与 `llg_tokenizer`，使其能够直接生成 Lark 语法并与 Guidance 后端配合使用。  
- 在 `SamplingParams` 中加入对非‑Tekken Mistral 分词器的校验，防止在 Guidance 后端使用不受支持的 tokenizer。  
- 重构 `MistralToolParser.adjust_request`：在支持的情形下自动生成结构化输出语法，统一工具选择、JSON‑schema、并加入缓存机制。  
- 更新依赖版本 (`mistral-common>=1.11.0`) 并添加大量单元测试，覆盖 GrammarFactory、LLTokenizer、工具解析以及后端行为。  

**🎯 影响范围**：  
- `vllm/tokenizers/mistral.py`（新增属性 & 缓存）  
- `vllm/sampling_params.py`（结构化输出校验）  
- `vllm/tool_parsers/mistral_tool_parser.py`（请求调整与语法生成）  
- `vllm/v1/structured_output/backend_guidance.py`（LLTokenizer 获取路径）  
- 测试目录 `tests/*`（新增 600+ 行测试）  
- `requirements/*`（mistral-common 版本升级）  

---

### 🔍 技术洞察

| 维度 | 影响 |
|------|------|
| **架构影响** | - **模块解耦**：GrammarFactory 与 tokenizer 分离，形成 “tokenizer ↔ grammar_factory ↔ llguidance” 三层链路，提升可组合性。 <br>- **向后兼容**：仅在 `is_tekken` 且 `mistral >= v11` 的 tokenizer 上暴露新属性；其它 tokenizer 行为保持不变，仅在 `adjust_request` 中做降级处理。 <br>- **统一 Grammar 生成**：MistralToolParser 现在统一走 `grammar_factory`，避免在不同代码路径自行拼接 Jinja 模板或 JSON‑schema，简化维护。 |
| **性能影响** | - **缓存**：`grammar_factory` 与 `llg_tokenizer` 使用 `@cached_property`，首次创建后复用，同一请求序列化多次时省去重复实例化开销。 <br>- **轻微额外开销**：首次调用 `grammar_factory.get_lark_*` 会触发 Jinja 渲染与 Lark 编译，成本与之前在 `MistralToolParser` 手工渲染相当，但已被缓存，整体影响可忽略。 |
| **安全考虑** | - **输入校验**：在 `SamplingParams._validate_structured_outputs` 中加入针对非‑Tekken Mistral tokenizer 的显式 `ValueError`，防止用户在不受支持的环境下误用 Guidance，避免潜在的未定义行为或拒绝服务。 <br>- **依赖升级**：`mistral-common` 由 1.10 → 1.11，须审查新版本是否引入安全修复或行为变更（已在官方 changelog 中确认无破坏性 API 变动）。 |
| **可维护性** | - **代码集中**：Grammar 相关逻辑从 `MistralToolParser` 移至 `mistral-common` 的 `GrammarFactory`，后者由 Mistral 官方维护，降低 vLLM 侧维护负担。 <br>- **测试覆盖**：新增 600+ 行测试，覆盖属性缓存、错误路径、各种 `tool_choice`、结构化输出来源（json_object、json_schema、结构化标签），提升回归安全性。 |
| **兼容性** | - 仅对 **Tekken v11+** 的 Mistral tokenizer 生效，旧模型（如 < v11）仍保持原有行为，且在 `adjust_request` 中会自动回退到旧路径或返回 `skip_special_tokens`。 <br>- 对非 Mistral tokenizer（HF、Llama 等）保持原有 “Guidance 不支持” 报错逻辑。 |

---

### ⚠️ 潜在风险

1. **误判 Tekken 版本**：`supports_grammar` 通过 `GrammarFactory.is_supported(self.mistral)` 判断。如果未来 Mistral 改动导致版本检测失效，可能在不受支持的 tokenizer 上错误暴露 `grammar_factory`，导致运行时错误。  
2. **缓存一致性**：`cached_property` 只在进程生命周期内保持。如果在同一进程里动态切换 tokenizer 配置（极少见），缓存的 `GrammarFactory` 可能使用旧的内部状态。  
3. **依赖升级冲突**：`mistral-common>=1.11.0` 可能与用户自行锁定的旧版产生冲突，需要在 `requirements` 中明确上限或在文档中提示。  
4. **结构化输出回退路径**：在 `adjust_request` 中若检测到不支持的属性，会直接调用 `super().adjust_request` 并返回。若上层逻辑仍假设 `structured_outputs` 已被填充，可能导致空值错误。需要确保调用方对 `None` 做防护。  

---

### 💡 关注建议

- **对开发者**  
  1. 在使用 Mistral tokenizer 前检查 `tokenizer.is_tekken` 与 `tokenizer.supports_grammar`，确保仅在兼容模型上启用 Guidance。  
  2. 若需要在非‑Tekken Mistral 上使用结构化输出，请切换 `structured_outputs_config.backend` 为 `xgrammar`、`outlines` 或改用 `tokenizer_mode='hf'`。  
  3. 关注 `mistral-common` 的版本更新日志，尤其是 GrammarFactory 的 API 兼容性变动。  

- **对用户/运维**  
  1. 在部署脚本中将 `mistral-common>=1.11.0` 明确写入依赖，防止自动回滚至不支持的旧版。  
  2. 若在容器化环境里使用多模型（包括旧版 Mistral），确保每个模型对应的 tokenizer 实例独立，避免共享缓存导致 cross‑model 语法混用。  
  3. 监控启动日志中 `supports_grammar` 的判定信息，快速发现因模型版本不匹配导致的功能降级。  

- **后续优化**  
  1. 将 `model_can_reason`（当前硬编码为 `False`) 通过模型元信息或配置自动推断，减少手动维护。  
  2. 考虑在 `SamplingParams` 中直接提供 `is_supported_structured_output_backend(tokenizer, backend)` 辅助函数，以统一所有后端的兼容性检查。

### [Gemma4] Enable Fast Prefill Optimization (#38879)
**SHA**: `47e6050` | 🔗 [查看提交](https://github.com/vllm-project/vllm/commit/47e605092b7fce3d64264b34250b1a286f344633)

**🎯 变更类型**：功能增强（Fast‑Prefill YOCO 优化）  
**⚡ 重要程度**：🟡 中  

**📋 变更摘要**  
1. 为 Gemma‑4 引入 “You Only Cache Once”(YOCO) 划分：前 K 层作为自回归 decoder（self‑decoder），后续层作为 KV‑共享 cross‑decoder。  
2. 在 `kv_sharing_fast_prefill` 开启时，新增 `Gemma4SelfDecoderLayers`、`Gemma4CrossDecoderLayers` 两个编译包装类，并在模型初始化时拆分原始 `self.layers`。  
3. 为 CUDAGraph‑快速预填充分配静态 `positions`、`hidden_states`、`per_layer_inputs` 缓冲区，实现 `fast_prefill_forward`，在 `forward` 中自动走该路径。  
4. 引入 `KVSharingFastPrefillMetadata` 读取 logits‑indices，完成对 batch‑size‑reduced的图调度。  
5. 将原有的 embed / per‑layer 输入、投影逻辑迁移到 `Gemma4SelfDecoderLayers`，保持普通路径不变。  

**🎯 影响范围**  
- `vllm/model_executor/models/gemma4.py`（核心模型实现）  
- 编译后端 `vllm/compilation/backends`（模型标签 `self_decoder`、`cross_decoder`）  
- 前向上下文 `vllm/forward_context.py`（新增 `attn_metadata` 读取）  
- 缓存配置 `cache_config.kv_sharing_fast_prefill`（新开关）  

**💡 关注建议**  

| 开发者 | 建议 |
|-------|------|
| **功能验证** | 运行包含大量短序列的预填充基准，比较开启/关闭 `kv_sharing_fast_prefill` 时的吞吐与显存占用，确保结果与原实现一致。 |
| **回退路径** | 保持 `Gemma4Model` 仍可在 `kv_sharing_fast_prefill=False` 下正常工作；单元测试应覆盖两条分支的 `forward`。 |
| **CUDA‑Graph** | 静态缓冲区大小受 `scheduler_config.max_num_batched_tokens` 限制，检查在极端 batch‑size 场景下不会出现 overflow。 |
| **编译兼容** | `@support_torch_compile` 条件已加入 `enable_if`，确认在不支持 torch.compile 的环境仍能正常实例化。 |
| **元数据一致性** | `KVSharingFastPrefillMetadata` 必须在 attention backend 正确填充；建议在 `AttnMetadata` 创建处加断言，以免因类型错误导致隐蔽 bug。 |
| **文档/配置** | 在 README/CHANGELOG 中标明新配置项 `kv_sharing_fast_prefill` 的默认值及适用硬件（需要 CUDA Graph 与 torch.compile）。 |
| **性能监控** | 添加 telemetry（如 `prefill_fast_path_time`) 以便用户在生产环境评估收益。 |

**结论**  
本次改动在模型层面实现了显著的预填充加速（特别是长短混合的推理场景），但引入了对编译、CUDA‑Graph 与元数据的依赖。建议在 CI 中加入对两种路径的完整回归，并提供清晰的配置说明，以降低用户迁移成本。

### [Feat][Core] safely abort requests when FSM fails to advance (#38663)
**SHA**: `e69a265` | 🔗 [查看提交](https://github.com/vllm-project/vllm/commit/e69a265135ef48312d78130f64b7bfce4cd81a37)

**🎯 变更类型**：功能增强（安全终止结构化输出请求）  
**⚡ 重要程度**：🟡 中  

**📋 变更摘要**  
- 在 `Scheduler.update_from_output` 中加入对结构化输出 FSM（finite‑state‑machine）无法前进的检测。若 `grammar.accept_tokens` 返回 `False`，立即把请求标记为 `FINISHED_ERROR`、`resumable=False` 并从调度器的 `running`/`requests` 中移除。  
- 对相同逻辑在 `AsyncScheduler` 的单元测试里也做了验证。  
- 将原本的 `logger.warning` 改为 `logger.error`，并统一在同一代码块内完成状态更新与 `stopped` 标记。  

**🎯 影响范围**  
- `vllm/v1/core/sched/scheduler.py`（核心调度器）  
- `vllm/v1/core/sched/async_scheduler.py`（异步调度器的测试）  
- 相关测试文件 `tests/v1/core/test_scheduler.py`、`tests/v1/core/test_async_scheduler.py`  

**💡 关注建议**  

1. **错误路径完整性**  
   - 确认 `structured_output_manager.should_advance` 为 `True` 时才会触发 `grammar.accept_tokens` 调用；若返回 `False`（比如内部已经决定不继续），当前实现保持原有行为，这是合理的。  
   - `request.resumable` 明确设为 `False`，避免后续恢复尝试。请检查在 `Engine` 层对 `FINISHED_ERROR` 的统一处理是否已覆盖此新入口（如状态转译、统计上报）。  

2. **日志与监控**  
   - 现在使用 `logger.error`，请在监控平台确认相应的错误级别不会导致误报。若业务侧仅希望看到警告，考虑保留 `logger.warning` 并在文档说明错误级别的区别。  

3. **测试覆盖**  
   - 新增的单元测试覆盖了同步与异步调度器两条路径，建议在 CI 中加入 **完整回归**（包括多 turn、并发调度等场景），确保在高并发下仍能安全释放 KV 缓存。  
   - 检查 `FinishReason.ERROR` 的传播链路：`engine_core_outputs[0].outputs[0].finish_reason` 必须在所有调用方（如 `Engine.step`）里被正确解释。  

4. **资源回收**  
   - `_free_request` 在本次改动中仍然通过 `side_effect` 调用，确保真实实现中对 KV 块的释放、内部计数的清理都保持幂等。建议在 `Scheduler._free_request` 中添加对异常的捕获，防止因提前终止导致资源泄漏。  

5. **文档同步**  
   - 结构化输出的 FSM 可能因语法不匹配而中止，这一行为在用户文档或 API 说明中应明确标注，以免使用者误以为模型会自行纠正。  

**总体评估**  
此次改动在 **安全性** 与 **可观测性** 上提供了必要的保护，防止因结构化语法不接受生成 token 而导致调度器进入不可恢复状态。实现简洁、与现有代码路径保持一致，但需确保异常路径的资源清理与上层错误处理同步更新，避免引入潜在的内存泄漏或统计不准。建议在合并前跑一遍全套集成测试并检查日志级别配置。
