### [CPU][Zen] Route W8A8 and W4A16 linear inference through zentorch on AMD Zen CPUs (#41813)
**SHA**: `3fd9d2d` | 🔗 [查看提交](https://github.com/vllm-project/vllm/commit/3fd9d2d35714e80b4cb3fcd3c408a0398fa2525f)

**🎯 变更类型**：功能增强 – 为 AMD Zen CPU 引入 Zentorch 后端的 W8A8 / W4A16 线性算子。  
**⚡ 重要程度**：🟡 中（提升特定硬件性能，但不影响已有 CPU/GPU 路径）。  

**📋 变更摘要**  
- `setup.py` 将 Zen 可选依赖从每日构建版切换为稳定版 `zentorch==2.11.0.0`。  
- 在 `vllm/model_executor/kernels/linear` 中新增 `ZentorchWNA16LinearKernel` 与 `ZentorchInt8ScaledMMLinearKernel`，并把它们加入 CPU 平台的优先 kernel 列表。  
- 新增 `zentorch_utils.py` 用于检查平台与 op 注册情况；实现了权重重新打包、scale 转 bf16 等专用处理逻辑。  
- 相应的 `__init__` 与注册表 (`register_linear_kernel`) 也同步更新。  

**🎯 影响范围**  
- `vllm/model_executor/kernels/linear/*`（mixed_precision、scaled_mm）  
- `setup.py` 的依赖声明  
- 运行时平台检测 (`vllm.platforms.current_platform`)  

**💡 关注建议**  
1. **依赖**：在非 Zen 环境安装时请确保 `zen` extra 不被误触发，或在 `requirements.txt` 中锁定 `zentorch==2.11.0.0`。  
2. **兼容性**：代码已在 `can_implement`/`is_supported` 中对平台、op、g_idx 等做防御检查，仍建议在 CI 中加入 Zen‑CPU（或模拟）与普通 CPU 的双套测试，确保回退到 `CPUWNA16LinearKernel` / `CPUInt8ScaledMMLinearKernel` 不会产生副作用。  
3. **加载后权重处理**：首次使用 Zen 路径会在 `process_weights_after_loading` 中置空原始参数，若后续出现 `torch.nn.Module._parameters` 为 `[]` 的异常，请检查模型保存/加载顺序。  
4. **文档**：更新 README/使用手册，说明只在 AMD Zen CPU 并安装 `zen` extra 时会自动切换；提供 `torch.ops.zentorch` 注册检查示例。  

总体而言，此次改动为 AMD Zen CPU 带来显著推理加速，风险主要在依赖与平台检测。确保测试覆盖和文档说明后即可安全合并。

### [MRV2] Support breakable CUDA graph (#44050)
**SHA**: `27fa5aa` | 🔗 [查看提交](https://github.com/vllm-project/vllm/commit/27fa5aa3b952a6108de127423397e50364a95fcb)

**🎯 变更类型**：功能增强（在已有的 Piecewise CUDA‑Graph 基础上加入可“中断”/可拆分的 CUDA Graph 实现）  
**⚡ 重要程度**：🟡 中  
**📋 变更摘要**：  
1. 新增 `BreakableCUDAGraphWrapper` 与开关 `is_breakable_cudagraph_enabled`，在 `CudaGraphManager` 中根据配置决定使用原生 Torch‑compiled piecewise graph 还是可中断的 CUDA Graph。  
2. `ModelCudaGraphManager` 在捕获阶段初始化 `breakable_cg_runner` 并在 `run_pw_graph` 中统一调度。  
3. `model_runner.py` 与 Eagle‑speculator 中的前向调用改为走 `run_pw_graph`（若开启）或直接调用模型，实现对两种执行路径的透明切换。  

**🎯 影响范围**：  
- `vllm/v1/worker/gpu/cudagraph_utils.py`（核心 CUDA‑Graph 管理）  
- `vllm/v1/worker/gpu/model_runner.py`（推理调度入口）  
- `vllm/v1/worker/gpu/spec_decode/eagle/speculator.py`（Eagle‑Speculative decode）  

**💡 关注建议**  
1. **开关验证**：确认 `is_breakable_cudagraph_enabled` 的环境变量/配置正确，防止在不支持的硬件上意外启用导致图破裂异常。  
2. **内存与图池**：`breakable_cg_runner.graph_pool = self.pool` 共享图池，需检查多卡/多进程场景下是否会出现竞争或泄漏。  
3. **回退路径**：确保 `use_breakable_cg` 为 `False` 时仍走原有 `torch.compile` 的 piecewise 路径，保持兼容性。  
4. **测试覆盖**：新增单元/集成测试应覆盖：① 仅开启 breakable，② 关闭 breakable，③ 多 batch‑size、不同 token‑counts 的捕获与执行，尤其是 `prefill_cudagraph_manager` 的 `capture` 与 `run_pw_graph`。  
5. **错误信息**：当前使用 `assert` 检查 `breakable_cg_runner` 是否已初始化，建议改为更友好的异常提示，以免在生产环境因断言被关闭而产生不明错误。  
6. **文档与示例**：在 README/配置说明中补充如何启用 breakable CUDA Graph（环境变量、编译选项），并给出性能对比示例，帮助用户判断是否值得开启。  

总体来看，此次改动为 vLLM 引入了在不依赖 `torch.compile` 的情况下使用可拆分 CUDA Graph 的能力，提升了在不满足 Torch‑compile 条件的机器上的性能潜力。但需重点关注兼容性、资源共享以及异常处理，以免在分布式推理或不同硬件上出现回退失效或资源竞争。
