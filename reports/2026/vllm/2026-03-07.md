### feat: expose media_io_kwargs at runtime (#34778)
**SHA**: `755356b` | 🔗 [查看提交](https://github.com/vllm-project/vllm/commit/755356b3d18d8079b1b115dfd2111dc1accdb764)

**🎯 变更类型**：功能增强  

**⚡ 重要程度**：🔴高  

**📋 变更摘要**  
- 在 OpenAI‑兼容入口及内部协议中新增 `media_io_kwargs` 字段，允许用户在单次请求层面覆盖媒体 I/O（图片、视频、音频）的加载/解码参数。  
- 实现了默认配置（`--media-io-kwargs`）与运行时覆盖的合并逻辑，新增跨模块传递、合并、以及对 `VideoMediaIO` 的特殊处理（fps 与 num_frames 互斥）。  
- 增加相应单元测试，验证覆盖生效、错误处理及后续请求的恢复能力。

**🎯 影响范围**  
- `vllm/entrypoints/openai/*`（请求/响应协议、服务层）  
- `vllm/renderers/*`（所有聊天渲染器）  
- `vllm/renderers/params.py`（`ChatParams` 扩展）  
- `vllm/multimodal/media/*`（`MediaIO` 基类、各 modality 实现、`MediaConnector`）  
- 相关测试：`tests/entrypoints/openai/test_video.py`、`tests/entrypoints/pooling/embed/test_online_vision.py`  

---

### 🔍 技术洞察  

| 维度 | 分析 |
|------|------|
| **架构影响** | 1. 通过 `media_io_kwargs` 从协议层向下渗透至 `ChatParams → MultiModalItemTracker → MediaConnector → MediaIO`，形成统一的 “全局默认 + 请求覆盖” 配置模型。<br>2. 引入 `merge_media_io_kwargs` 与 `MediaIO.merge_kwargs`，在 modality 维度实现可插拔的合并策略，保持已有组件的最小侵入。<br>3. `VideoMediaIO` 覆盖逻辑避免 `fps` 与 `num_frames` 冲突，提升 API 的可预测性。 |
| **性能影响** | - 正常情况下仅涉及一次浅拷贝字典，开销可忽略。<br>- 通过请求层面降低 `num_frames`（如测试中从 32→4）可显著减少视频解码与特征提取成本，提升吞吐量。<br>- 额外的合并与校验在高并发场景下仍是 O(1) 操作，对整体 latency 影响可接受。 |
| **安全考虑** | - `media_io_kwargs` 直接进入 `MediaIO` 实例，若不做严格校验，用户可能注入恶意路径、外部 URL、或自定义加载器导致代码执行或资源滥用。<br>- 已在 `MediaIO`、子类构造函数中加入 **“用户提供的配置应经过验证和错误处理”** 的注释，但实际验证实现仍缺失，需要在入口层（pydantic model）或 `MediaConnector` 中加入白名单/类型检查。 |
| **可维护性** | - 通过统一的 `merge_media_io_kwargs` 与 `MediaIO.merge_kwargs`，后续若新增 modality（如 `depth`）只需在 `MODALITY_IO_MAP` 中登记即可获得自动合并支持。<br>- 新增的字段在多个协议文件中重复定义，易受忘记同步的风险，建议抽取公共基类或 schema。 |
| **兼容性** | - 老版本客户端若不提供 `media_io_kwargs`，行为保持不变（默认 `None`），向后兼容。<br>- 新增的 CLI 参数 `--media-io-kwargs` 与 runtime 字段均使用相同结构，保持一致性。 |

---

### ⚠️ 潜在风险  

1. **参数校验不足**  
   - 目前仅在文档层面提示“确保正确验证”，实际未实现具体检查。恶意或错误的键值（如 `{"video": {"num_frames": -1}}`、路径注入）可能导致异常或资源泄漏。  

2. **资源耗尽**  
   - 改动允许用户在请求中提升 `num_frames`、`fps` 等，若未设上限，可能导致单请求消耗大量 GPU/CPU、内存，触发 DoS。  

3. **不一致的默认行为**  
   - `VideoMediaIO` 在覆盖 `num_frames` 时自动清除 `fps`（反之亦然），但其他 modality 仍保留所有默认键，可能出现隐藏冲突。  

4. **错误恢复路径**  
   - 测试覆盖了一次失败后的恢复，但若异常在 `MediaConnector` 初始化阶段抛出，后续请求仍会复用同一 `MediaConnector` 实例，可能留下错误状态。  

5. **文档/SDK 同步**  
   - OpenAI SDK、vllm Python 包的类型提示需同步更新，否则用户在 IDE 中看不到新字段。

| **运维/安全团队** | - 为 `media_io_kwargs` 设置 **资源上限**（如 `max_num_frames`, `max_fps`），并在启动参数或配置文件中提供可调阈值。<br>- 在日志中记录每次请求的实际 merged `media_io_kwargs`，便于审计异常行为。 |
| **用户** | - 如需降低视频处理开销，可在请求体 (`extra_body.media_io_kwargs`) 中显式设置 `num_frames`，但请确保不超过模型文档推荐范围。<br>- 若使用 `--media-io-kwargs`，请使用合法的 JSON 并遵循以下结构：`{ "modality": { "key": value, … } }`。 |

| **未来扩展** | - 将 `media_io_kwargs` 抽象为单独的 **配置对象**，在 `ModelConfig` 中统一管理，减少协议层的重复定义。<br>- 考虑在 `MediaConnector` 支持 **插件化** 的自定义 loader，配合 `media_io_kwargs` 实现更灵活的媒体处理管线。 |

### [V0 Deprecation] Remove unused swap_space parameter (#36216)
**SHA**: `00b814b` | 🔗 [查看提交](https://github.com/vllm-project/vllm/commit/00b814ba5a4139910c0824619a8dc6af547e178a)

**🎯 变更类型**：功能废弃 / 代码清理  
**⚡ 重要程度**：🟡 中  
**📋 变更摘要**  
本次提交在 V1 体系中彻底移除 `swap_space` 参数。相关的 CLI 选项、配置字段、验证逻辑以及文档说明均被删除；同时在 `LLM` 构造函数中加入了对旧参数的兼容警告（`DeprecationWarning`）。所有测试、基准以及示例配置文件中出现的 `swap_space` 也同步删除。

**🎯 影响范围**  
- **核心配置**：`vllm/config/cache.py`、`vllm/config/vllm.py`、`vllm/engine/arg_utils.py`、`vllm/entrypoints/llm.py`  
- **CLI**：`--swap-space` 参数不再接受，相关帮助文本被去除。  
- **文档**：`docs/design/metrics.md`、`docs/serving/integrations/llamaindex.md` 更新说明。  
- **测试/基准**：大量测试文件、benchmark 脚本中删除 `swap_space` 参数。  
- **用户代码**：任何显式传入 `swap_space`（如 `LLM(..., swap_space=…)`、`vllm_kwargs={"swap_space": …}`）将触发 `DeprecationWarning`，但仍能正常运行。

**💡 关注建议**  
1. **迁移路径**：使用 `gpu_memory_utilization` 或 `kv_cache_memory_bytes`（若需固定 KV 缓存大小）代替 `swap_space` 的场景。`swap_space` 之前主要用于 KV 缓存迁出 CPU，V1 已经不再支持此特性。  
2. **代码审查**：确认项目中是否还有对 `swap_space` 的显式引用（如自定义脚本、第三方集成），全部改为不使用或直接删除。  
3. **兼容性测试**：在升级到包含此提交的版本后，运行完整测试套件，尤其是分布式/多 GPU 场景，确保 `kv_cache_memory_bytes` 与 `gpu_memory_utilization` 的组合仍能满足内存需求。  
4. **警告处理**：在生产环境中关闭 `DeprecationWarning`（或捕获并记录），防止日志噪声；但建议在代码中彻底移除 `swap_space` 参数以避免未来硬拆。  
5. **文档同步**：确认所有用户手册、示例 notebook、CI 配置中已不再出现 `swap_space`，防止误导新手。  

总体来看，此次改动仅是清理过时的功能，对现有功能不产生行为变更，除非用户仍依赖 `swap_space`，否则升级风险极低。祝迁移顺利！

### [BugFix] Avoid ignored trust_remote_code warnings (#36290)
**SHA**: `c7f32e0` | 🔗 [查看提交](https://github.com/vllm-project/vllm/commit/c7f32e08c2e49665621be72f8e83d6433b2564d1)

**🎯 变更类型**：Bug 修复  
**⚡ 重要程度**：🟡 中  
**📋 变更摘要**：  
1. 新增 `without_trust_remote_code` 工具函数，用于在不修改原始 `kwargs` 的前提下剔除 `trust_remote_code` 参数。  
2. 在各类 `Config` 的加载入口（`vllm/transformers_utils/config.py` 以及若干子模块）中，对传入的 `kwargs` 统一调用该函数，防止 `trust_remote_code` 被不期望地传递给 `transformers` 的 `PretrainedConfig.get_config_dict`，从而避免 “ignored trust_remote_code warnings”。  
3. 对 `parse` 中的 `trust_remote_code` 处理逻辑作了微调：先合并外部 `trust_remote_code` 再清除对应键，以保持行为不变但不触发警告。

**🎯 影响范围**  
- `vllm/transformers_utils/config.py`（核心配置解析）  
- `vllm/transformers_utils/configs/*`（EAGLE、ExtractHiddenStates、Medusa、Speculators 等子配置）  
- `vllm/transformers_utils/utils.py`（新增函数）  

**💡 关注建议**  
1. **功能完整性**：`trust_remote_code` 仍然通过局部变量 `trust_remote_code` 传递给需要的分支（如 `parse` 中的 `trust_remote_code |= kwargs.get(... )`），请确认在离线模式下或本地文件路径时，这一标记的行为与之前保持一致。  
2. **API 兼容性**：`without_trust_remote_code` 返回的是新的字典，原始 `kwargs` 不会被修改，这避免了副作用。但如果外部代码依赖 `kwargs` 在后续被再次使用（例如多次调用不同的加载函数），应确保它们仍能获取原始 `trust_remote_code`。建议在文档中说明该函数的“只读”特性。  
3. **测试覆盖**：新增单元测试验证：  
   - `without_trust_remote_code` 在包含/不包含 `trust_remote_code` 时的返回值。  
   - 各子配置的 `from_pretrained` 在传入 `trust_remote_code=True` 时不抛出警告且仍能正常加载（尤其是自定义模型仓库）。  
4. **潜在风险**：如果未来在其他位置（如 `pipeline`、`scheduler`）仍直接把 `kwargs` 传给 `transformers`，可能会再次触发相同警告。建议在项目的所有 `PretrainedConfig.get_config_dict` 调用处统一使用 `without_trust_remote_code`，或在更高层封装统一入口。  
5. **代码可读性**：可以考虑将 `trust_remote_code` 的合并与 `without_trust_remote_code` 的调用封装为一个小工具（例如 `prepare_hf_kwargs(kwargs, trust_remote_code_flag)），以降低重复代码并提升可维护性。

总体而言，此次改动解决了实用的警告噪声问题，对核心配置加载逻辑的影响局限且明确。只要补齐相应的测试与文档说明，风险可控。

5. **测试覆盖**：加入单元测试验证 `log_enabled_passes` 能正确列出开启的 fusion，尤其在不同 `optimization_level`、不同硬件模拟（SM90/SM100）下的组合。  

总体而言，此次改动提升了用户对 vLLM 编译优化流水线的可见性与可调性，对功能本身无风险，关注日志实现细节即可顺利发布。
