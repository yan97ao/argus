### [kv_offload] Add multi-tier KV cache offloading framework (#40020)
**SHA**: `11f6b54` | 🔗 [查看提交](https://github.com/vllm-project/vllm/commit/11f6b545d41a42422410bb529466463f466cafd9)

**🎯 变更类型**：功能增强（新增多层 KV 缓存 offloading 框架）  
**⚡ 重要程度**：🔴高  

**📋 变更摘要**  
- 在 `vllm.v1.kv_offload` 包中引入 **TieringOffloadingSpec**、**TieringOffloadingManager**、以及一套二级层（Secondary Tier）抽象与实现，实现 “CPU → 多级存储” 的层次化 KV 缓存 offloading。  
- 新增 API：二级层管理器接口、二级层工厂、示例二级层（内存 LRU），以及对 `SharedOffloadRegion` 的零拷贝 memoryview 暴露，以便二级层直接访问 CPU KV 缓冲区。  
- 相应单元测试覆盖基础存取、级联写入、提升读取、引用计数保护、跨层淘汰等关键路径。  

**🎯 影响范围**  
- `vllm/v1/kv_offload/*`（核心 offload 框架）  
- 新增 `vllm/v1/kv_offload/tiering/*`（二级层抽象、实现、工厂、管理器）  
- `vllm/v1/kv_offload/cpu/*`（CPU offload 适配）  
- `vllm/v1/kv_offload/factory.py`（注册新 Spec）  
- 新增/修改单元测试文件 `tests/v1/kv_offload/*`  

**🔍 技术洞察**  

| 维度 | 影响说明 |
|------|----------|
| **架构影响** | 1. **层次化管理器**：`TieringOffloadingManager` 负责在 **CPU 主层** 与 **任意数量的二级层** 之间调度，保持原有 GPU ↔ CPU 接口不变，保证向后兼容。<br>2. **二级层抽象** (`SecondaryTierManager`) 明确定义 `lookup / submit_store / submit_load / get_finished / touch / shutdown`，为存储、网络、持久化等各种后端提供统一入口。<br>3. **零拷贝共享内存**：`SharedOffloadRegion.create_kv_memoryview()` 暴露 `memoryview`，二级层直接读写 CPU KV 缓冲区，避免额外 memcpy。<br>4. **引用计数 (`ref_cnt`)** 机制延伸到跨层传输：在 cascade/store 与 promotion 期间，主层块的 `ref_cnt` 被临时提升，防止淘汰导致数据不一致。 |
| **性能影响** | - **存储路径**：GPU → CPU → *所有* 二级层（级联）会产生额外的异步拷贝，增大写入延迟，但通过 **异步 job** 与 **批量 submit_load**（按请求聚合）可在 CPU 与二级层之间并行化，降低峰值阻塞。<br>- **读取路径**：二级层命中时触发 **promotion**（CPU ← 二级层 ← GPU），在同一步内通过 `prepare_write` 预分配主层槽位，随后异步 `submit_load` 完成。首次 miss 仍需一次延迟，但后续同请求的重复访问会直接命中主层。<br>- **引用计数保护** 防止在 promotion 期间的淘汰，保证正确性，代价是额外的计数更新（微不足道）。<br>- **内存开销**：为每个 worker 维持 `SharedOffloadRegion`（CPU page‑size × world_size × num_blocks），以及二级层自己的块容器（示例实现为 `OrderedDict`).<br>- **吞吐量**：在二级层采用真正的持久化（如 SSD、网络）时，写放大和网络带宽将成为瓶颈；框架已留出 `JobMetadata` 与异步完成回调，可在实现层面加入流水线与批量压缩。 |
| **安全考虑** | 1. **Zero‑copy memoryview**：`create_kv_memoryview` 通过 `numpy()` 转换后检查 `ctypes.data` 与原始指针相同，防止不经意的复制泄露实际内存句柄。二级层实现必须确保不对该 view 做写越界或持久化到不受信任的进程。<br>2. **异步作业调度**：`JobResult.success` 由二级层返回，若实现层出现异常并错误返回 `success=True`，可能导致缓存不一致；建议二级层在异常捕获后返回 `success=False` 并记录日志。<br>3. **资源泄露**：`CpuGpuOffloadingHandler.shutdown` 现在会自动调用 `mmap_region.cleanup()`，确保进程退出时清理文件映射。<br>4. **配置注入**：`TieringOffloadingSpec` 通过 `kv_connector_extra_config` 接收二级层列表，若用户传入恶意配置（如非常大的 `max_blocks`），可能导致 OOM；框架已在创建时未强制上限，建议在生产环境加入配额检查。 |
| **可维护性** | - **模块化**：二级层抽象、工厂、管理器分别放置在独立目录，便于新增自定义存储后端（如 Redis、Ceph）。<br>- **测试覆盖**：新增 1.900+ 行单元测试，覆盖层级写入、自动淘汰、引用计数、并发 promotion、错误路径等，提升回归安全性。<br>- **向后兼容**：`TieringOffloadingManager` 在 `secondary_tiers=[]` 的情况下行为完全等同于原 `CPUOffloadingManager`，已有部署无需改动。<br>- **文档/日志**：在 `TieringOffloadingSpec` 中加入 `logger.info` 与错误日志，便于运维排查。 |

**⚠️ 潜在风险**  
1. **复杂的异步状态机**：`ref_cnt`、`_pending_load_submissions`、`_processed_jobs_this_step` 的交叉使用若出现顺序错误，可能导致：<br>   - 永久阻塞的 promotion（ref_cnt 永不归零）<br>   - 重复提交同一 promotion（浪费 IO）<br>   - 失去对已完成 job 的清理（内存泄漏）。  
2. **二级层实现不一致**：示例实现是同步的 `OrderedDict`，真实存储（磁盘、网络）可能出现 **慢回调**、**部分失败**（部分块写入成功），目前框架仅接受整体 `JobResult.success`，缺乏细粒度错误处理。  
3. **资源竞争**：多个请求在同一步内对同一块发起 promotion 时，`_initiate_promotion` 会在 `primary_tier.prepare_write` 中预分配 slot 并将 `ref_cnt=-1`，若大量并发请求导致 slot 预分配失效（主层满），返回 `False` 并导致请求直接失效。  
4. **内存映射泄露**：`SharedOffloadRegion.cleanup` 只在 `CpuGpuOffloadingHandler.shutdown` 与 `CPUPrimaryTierOffloadingManager.shutdown` 调用，若进程异常退出未走到 `shutdown`（如 SIGKILL），映射文件可能残留。  
5. **兼容性**：新 spec `TieringOffloadingSpec` 复用了 `CPUOffloadingSpec` 的内部字段（如 `cpu_page_size_per_worker`），但在不同的 `parallel_config.world_size` 与 `rank` 场景下仍需验证每个 worker 的 mmap 大小是否正确分配。

| 建议类别 | 具体措施 |
|----------|----------|

## 📋 变更摘要
1. **在 ROCm 环境下为 DeepSeek‑V4 引入 AITer 与 TileLang 两套后端实现**，并把原来的 CUDA‑only 实现拆分为统一的 *custom‑op* 框架。

> **目标**：让 vLLM 在 AMD GPU（ROCm）上能够使用高度融合的 MHC 前/后块实现，从而在 DeepSeek‑V4、DeepSeek‑V4‑MTP 等模型上获得 **显著的算子融合加速**，同时保持在 CUDA 环境的兼容性。

## 🔍 技术洞察

### 1. 架构影响
| 维度 | 说明 |
|------|------|
| **模块化算子层** | 通过 `CustomOp` 把每一种实现（CUDA‑tilelang、ROCm‑aiter、torch fallback、triton）抽象为同一接口，使得 **模型层不需要感知底层实现细节**。这提升了代码可维护性，且便于以后继续添加例如 XPU、CPU‑only 实现。 |
| **平台分支统一** | `layers/mhc.py` 采用 `current_platform.is_rocm()` 与 `hidden_size % 256` 判断来选择 AITer（ROCm）或 fallback。这样在同一二进制里可以兼容 **CUDA 与 ROCm**，避免了发布两套代码的复杂度。 |
| **算子融合** | `mhc_fused_post_pre_tilelang` 将 **MHC post** + **MHC pre** 两块融合为一次 GEMM + 多阶段聚合，极大降低全局内存读写次数。相当于在 CUDA 端已有 `mhc_fused_post_pre` 的功能，在 ROCm 上得到等价实现（基于 AITer）。 |
| **依赖抽象** | `requirements/rocm.txt` 删除了 `tilelang==0.1.9`（仅用于 CUDA），而在代码里使用 `has_tilelang()` 动态检查，**降低了 ROCm 镜像的体积**。|

### 2. 性能影响
| 场景 | 关键改动 | 预期性能提升 |
|------|----------|---------------|
| **DeepSeek‑V4（HC 前置）** | 使用 `mhc_pre_tilelang`（CUDA）或 `mhc_pre_aiter`（ROCm）取代原先的纯 PyTorch 实现 | **~1.5‑2×** 速率提升（GPU‑算子融合 + TF32‐accelerated GEMM） |

1. **ABI 跨版本不匹配**
