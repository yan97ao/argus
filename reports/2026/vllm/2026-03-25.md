### [Feature] limit thinking tokens (hard limit) (#20859)
**SHA**: `4731884` | 🔗 [查看提交](https://github.com/vllm-project/vllm/commit/4731884796d43bd8493769dbfb0e199a2a542092)

**🎯 变更类型**：功能增强（引入思考预算控制）  
**⚡ 重要程度**：🔴高  
**📋 变更摘要**：本次提交为 vLLM 引入 *thinking token budget* 功能，实现对具备思考块（reasoning）的模型（如 Qwen3、DeepSeek、Nemotron3）在推理阶段使用的 token 数量进行硬性限制。新增 `ReasoningConfig`、CLI 参数、请求字段 `thinking_token_budget`，以及 `ThinkingTokenBudgetLogitsProcessor` 在 logits 生成阶段强制模型在预算耗尽后输出 `think_end_str`。配套文档、E2E 测试以及相应的配置、采样参数、输入校验均已同步更新。

---

### 🎯 影响范围
- **核心模块**：`vllm/config/*`、`vllm/engine/arg_utils.py`、`vllm/v1/engine/input_processor.py`  
- **采样/ logits 处理**：`vllm/v1/sample/logits_processor/builtin.py`（新增 `ThinkingTokenBudgetLogitsProcessor`）  
- **OpenAI 接口层**：`vllm/entrypoints/openai/chat_completion/protocol.py`（新增 `thinking_token_budget` 字段）  
- **CLI/启动脚本**：`vllm/engine/arg_utils.py` 增加 `--reasoning-config` 参数  
- **文档**：`docs/features/reasoning_outputs.md`（使用说明）  
- **测试**：`tests/v1/entrypoints/openai/test_thinking_token_budget.py`、`tests/v1/logits_processors/test_correctness.py`（针对新 processor 的单元与端到端验证）

### 🔍 技术洞察

| 维度 | 分析 |
|------|------|
| **架构影响** | <ul><li>新增 `ReasoningConfig`，在 `VllmConfig` 中以可选字段注入，统一在 `VllmConfig.finalize()` 时调用 `initialize_token_ids`，实现 token‑id 的懒初始化。</li><li>`ThinkingTokenBudgetLogitsProcessor` 作为自定义 logits processor 参与 `LogitsProcessor` 链路，使用 `VllmConfig.reasoning_config` 判断是否启用。</li><li>在 `EngineArgs` 中将 `reasoning_config` 暴露给 CLI，保持配置路径一致。</li><li>请求层面在 `ChatCompletionRequest` 中加入 `thinking_token_budget`，并在 `to_sampling_params` 中映射到 `SamplingParams.thinking_token_budget`。</li><li>`_validate_params` 中对 `thinking_token_budget` 与 `reasoning_config` 的耦合关系做显式校验，防止配置缺失导致运行时错误。</li></ul> |
| **性能影响** | <ul><li>每个活跃请求在 `ThinkingTokenBudgetLogitsProcessor` 中维护一个小的状态字典（约 10‑12 项）以及已生成 token 列表，内存开销与 request 数线性增长。</li><li>状态更新在每一步仅对新增 token 进行增量扫描，最长需要检查 `max(think_start_len, think_end_len)` 个 token，开销极低。</li><li>当预算耗尽并进入 “force‑end” 模式时，仅在涉及的 batch（`mask[i]=True`）上进行一次 GPU‑side强制 logits 填充（`logits[indices, token_id]=1e9`），对整体吞吐影响可以忽略。</li><li>在高并发、极短的 think 块（如 1‑2 token）场景下，状态切换频繁，可能导致少量额外的同步/CPU‑GPU 交互，但仍在可接受范围。</li></ul> |
| **安全考虑** | <ul><li>`reasoning_config` 通过 CLI JSON 注入，现已在 `ReasoningConfig.initialize_token_ids` 中对 tokenizer 编码结果做空值校验，防止异常字符串导致 `ValueError`。</li><li>`thinking_token_budget` 为整数，未做上限校验；若设定异常极大值，会导致长时间占用 `in_think` 状态，但不影响系统安全。</li><li>日志/异常信息未泄露模型内部 token ID，安全风险低。</li></ul> |

### ⚠️ 潜在风险
1. **Token‑ID 不匹配**  
   - 若模型词表不包含 `think_start_str` / `think_end_str`，`ReasoningConfig.initialize_token_ids` 会抛异常，导致服务启动失败。  
2. **状态同步错误**  
   - 在异步调度或 batch‑reordering (`batch_update.moved`) 场景下，如果 `_state` 未能正确迁移，可能出现预算计数错位，导致提前/延迟触发结束 token。  
3. **边界序列冲突**  
   - 当 `think_start_str` 与 `think_end_str` 有重叠 Token 序列（如子串相同）时，`_find_last_sequence_index` 的定位逻辑可能误判，进而导致错误的 `in_think` 标记。  
4. **异常请求混用**  
   - 同 batch 中既有带 `thinking_token_budget` 的请求，也有不带的请求时，processor 必须保证对无预算请求不产生副作用（已通过 `if state and state["in_end"]` 过滤，但仍需防止因旧 state 残留导致误mask）。  
5. **资源泄漏**  
   - `output_tok_ids` 持续累计所有已生成 token，若请求极长（数千 token）或大量并发，可能导致单请求内存占用增长。  

---

### 💡 关注建议
- **配置校验**  
  - 在 CLI 解析阶段加入 schema 检查（如 `jsonschema`），确保 `think_start_str` / `think_end_str` 为非空字符串，且在模型词表中成功映射。  
- **上限与默认**  
  - 为 `thinking_token_budget` 添加合理的上限（如 1024），防止异常输入导致不必要的状态维护。  
- **日志与监控**  
  - 在 `ThinkingTokenBudgetLogitsProcessor` 的关键转折点（进入/退出 think、强制 end）加入 DEBUG 级别日志，便于运维排查。  
- **单元测试扩展**  
  - 增加以下边界测试：<br>① `think_start_str` 与 `think_end_str` 相同的情况；<br>② `thinking_token_budget=0`（应直接强制输出 `think_end_str`）；<br>③ 大 batch 中随机迁移请求，验证 `_state` 正确搬迁。  
- **内存优化**  
  - 若 `output_tok_ids` 仅用于查找最近的 start/end 序列，可改为仅记录最近 N（max(start_len, end_len)）个 token，降低长期内存占用。  
- **文档同步**  
  - 确保用户手册明确说明：必须同时配置 `--reasoning-config` 与 `thinking_token_budget`，否则会抛出配置错误。

### [Core] add option to schedule requests based on full ISL (#37307)
**SHA**: `dc78c2c` | 🔗 [查看提交](https://github.com/vllm-project/vllm/commit/dc78c2c933ac8903d7e34bff7b5161eae70595fc)

**🎯 变更类型**：功能增强  
**⚡ 重要程度**：🟡 中  

**📋 变更摘要**  
- 在 `SchedulerConfig` 中新增 `scheduler_reserve_full_isl`（默认 `True`），用于在调度新请求时检查 KV‑cache 是否能容纳整个输入序列，而非仅首个 chunk。  
- 通过 `EngineArgs`、CLI 与 `EngineConfig` 将该 flag 向下传递。  
- `KVCacheManager` 新增 `can_fit_full_sequence` 方法实现完整序列占用检查。  
- `Scheduler` 在分配 KV‑slots 前调用该检查，若不满足则提前终止本轮调度（并在有 Encoder 输入时释放对应缓存）。  

**🎯 影响范围**  
- **调度器** (`vllm/v1/core/sched/scheduler.py`)：新增 `scheduler_reserve_full_isl` 成员并在 `schedule` 中使用。  
- **KV 缓存管理** (`vllm/v1/core/kv_cache_manager.py`)：实现完整序列适配检查逻辑。  
- **配置/CLI**：`SchedulerConfig`, `EngineArgs`, `add_cli_args`, `create_engine_config` 均加入对应字段。  
- **相关文档/测试**：若有自动化测试或文档涉及调度行为，需要同步更新。  

**💡 关注建议**  

1. **向后兼容性**：默认开启可能导致现有部署在极端长序列下出现调度延迟或吞吐下降。建议在发布日志中明确说明，或提供快速回退方式（如 `--scheduler-reserve-full-isl=False`）。  
2. **性能评估**：`can_fit_full_sequence` 额外调用 `Coordinator.get_num_blocks_to_allocate`，在高并发场景下可能增加调度开销。应在 CI 中加入基准测试，比较开启/关闭该选项的吞吐与延迟。  
3. **错误路径**：当检查不通过时直接 `break` 当前循环，可能导致后续请求被遗漏。确认调度循环的外层是否会在后续 tick 重新尝试，以防出现“卡住”现象。  
4. **Encoder 输入的清理**：已在不满足条件时调用 `encoder_cache_manager.free(request)`，但若请求同时拥有 decoder KV，需确认对应资源也已及时释放，避免内存泄漏。  
5. **文档与示例**：在 README / CLI 帮助中添加 `--scheduler-reserve-full-isl` 的说明，解释其对 chunked‑prefill 场景的意义。  
6. **测试覆盖**：新增单元测试，覆盖：① 完整序列能够适配时正常调度；② 不能适配时调度提前中止并释放 encoder 缓存；③ 关闭 flag 时行为保持老实现。  

整体来看，此改动通过在调度前做更严格的 KV‑cache 检查，能够有效防止因 chunked‑prefill 导致的 KV‑cache 抖动和 over‑admission，提升长期稳定性。但需要注意额外的调度成本及对现有工作负载的潜在影响，建议在正式发布前完成上述验证与文档更新。
