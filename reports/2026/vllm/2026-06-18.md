### [ROCm][DSv4] Functional fixes for DeepSeek V4 on MI300X/MI325X (#45681)
**SHA**: `afdcbd5` | 🔗 [查看提交](https://github.com/vllm-project/vllm/commit/afdcbd5d39eaf2b37b616c8ee8aabc51e15e70ef)

**🎯 变更类型**：功能增强 / Bug修复 / 性能优化 / 重构  
**⚡ 重要程度**：🔴 高  

**📋 变更摘要**  
1. 为 DeepSeek‑V4 在 ROCm（MI300X/MI325X）上提供功能性修复，主要解决 FP8 编码在不同硬件（FNUZ vs OCP）上的不一致问题。  
2. 在 C++/CUDA、Triton、Python 三级实现中加入 `use_fnuz` 参数与平台检测，确保在 gfx942（MI300X）使用 ROCm FNUZ 编码，其余平台使用 OCP E4M3 编码。  
3. 新增针对 gfx942 的 AITER FP8 MQA logits fallback kernel，避免因 AITER 旧版缺陷导致 LDS 超限或错误解码。  
4. 更新单元测试，使量化‑解码路径在 NoPE 区域保持位相等，在 RoPE 区域容忍 ≤ 1 bf16/fp8 ULP 误差，提升跨平台数值一致性。  

**🎯 影响范围**  
- `csrc/libtorch_stable/fused_deepseek_v4_qnorm_rope_kv_insert_kernel.cu`（FP8 转换）  
- `vllm/models/deepseek_v4/*`（缓存编码、前向路径、ops）  
- `vllm/model_executor/layers/quantization/utils/fp8_utils.py`（权重归一化）  
- Triton kernel文件 `cache_utils.py`、`rocm_aiter_mla_sparse.py`（decode kernel）  
- 新增 `vllm/v1/attention/ops/triton_fp8_mqa_logits.py`（gfx942 fallback）  
- 测试 `tests/kernels/test_fused_deepseek_v4_qnorm_rope_kv_insert.py`

### 🔍 技术洞察  

| 维度 | 影响说明 |
|------|----------|
| **架构影响** | • 引入 `current_platform.is_fp8_fnuz()` 判断，将 FP8 编码从固定的 OCP (`float8_e4m3fn`) 切换为平台感知的 FNUZ (`float8_e4m3fnuz`) <br>• 在 C++ kernel 中针对 gfx942 使用 `__hip_fp8_e4m3fnuz`，其它 ROCm 目标使用 `__hip_fp8_e4m3fn`，保持与 Triton‑encoded 缓存的编码一致性 <br>• `quantize_and_insert_k_cache`、`dequantize_and_gather_k_cache` 统一加入 `use_fnuz` 参数，调用方必须显式匹配缓存的实际编码（compressed‑cache 为 OCP，SWA‑cache 依据平台）。 |
| **性能影响** | • 仅在分支选择时增加极小的运行时判断（`if use_fnuz`），对热点路径几乎没有影响。 <br>• 对 FP8→bf16/float32 的转换仍使用 `to(tl.float8e4nv)` 或 `to(tl.float8e4b8)`，两者在硬件上均为单指令 FP8→FP32，性能相同。 <br>• 新增 gfx942‑专用 AITER fallback：在 LDS <64 KiB 限制下自动切换 `BLOCK_KV`/`num_stages`，避免 JIT 因 LDS 超限报错，确保在 MI300X 上能够运行而不牺牲显著吞吐。 <br>• 单元测试中允许 1 ULP 误差，防止因极限 rounding 引入不必要的回退路径，保持最高吞吐。 |

| **可维护性** | • 通过 `use_fnuz` 参数显式传递编码信息，避免了原有“暗码”依赖平台宏的隐蔽实现，提升代码可读性。 <br>• `get_fp8_min_max()` 抽象了 `FP8_MAX`，统一在 Python 与 C++/CUDA 中使用同一常量，降低硬编码错误概率。 <br>• 增加的大量单元测试覆盖了 NoPE、RoPE、KV、Full‑cache 三条路径，且对 ULP 误差作了明确阈值，便于后续回归。 |

### ⚠️ 潜在风险  

1. **误用 `use_fnuz`**：若上层调用者忘记传递正确的 flag（例如在非 gfx942 上强制 `use_fnuz=True`），将导致解码时使用错误的比例因子，产生显著数值偏差。  
2. **平台检测不充分**：当前仅检查 `__gfx942__` 与 `current_platform.is_fp8_fnuz()`，新出现的 ROCm GPU（如未来的 MI4xx）若采用不同 FP8 编码，需同步更新条件。  
3. **ULP 容忍阈值**：测试容忍 ≤ 1 ULP 可能掩盖某些极端输入下的系统性偏差，尤其在长序列、低‑precision微调时，累计误差可能放大。  
4. **AITER fallback 兼容性**：该模块在 `rocm_aiter_ops.is_enabled()` 为 True 时加载，若 AITER 版本升级后内部实现改变，可能导致重复实现冲突或性能倒退。  
5. **权重归一化路径**：在 `process_fp8_weight_block_strategy` 中对 `float8_e8m0fnu` 的手动 exponent‑+1 操作假设 NaN sentinel 为 `-128`（int8）。若未来 ROCm 改变 NaN 表示，代码将产生错误。

### fix(quantization): Fix AWQ dequantize on Intel XPU and refactor AutoAWQ config (#42727)
**SHA**: `8d4f549` | 🔗 [查看提交](https://github.com/vllm-project/vllm/commit/8d4f54966cdb8f1d0768fbe5319e400047877a3d)

**🎯 变更类型**：功能增强 / 重构 / 安全修复  
（统一 AWQ 与 Auto‑AWQ 配置、修复 Intel XPU 上的 dequantize、去除冗余 `awq` 模块）

**⚡ 重要程度**：🔴高  

**📋 变更摘要**  
1. 将原有的 `awq` 与 `awq_marlin` 代码合并进全新 `AutoAWQConfig`，对外统一为 **`auto_awq`** 量化方法；  
2. 在 `override_quantization_method` 中加入对 **CPU** 平台的保护，防止在 CPU 上错误覆盖为 Auto‑AWQ；  
3. 新增 **XPU** 线性实现 `AutoAWQXPULinearMethod`，并在 `auto_awq.py` 中完成权重重排与 `int4_gemm_w4a16` 调用，解决了 Intel XPU 上的 dequantize 错误；  
4. 大规模更新所有引用（`awq` → `auto_awq`），包括模型加载、MoE、Inc、ROCm 支持等；  
5. 删除旧的 `vllm/model_executor/layers/quantization/awq.py`（286 行），并补充 231 行单元测试，覆盖配置转换、CPU/XPU 行为、MoE 兼容性等。

**🎯 影响范围**  
- **量化子系统**：`vllm.model_executor.layers.quantization.*`（AutoAWQ、AutoGPTQ、Inc、Compressed‑Tensors 等）  
- **模型加载与推理**：所有使用 AWQ 权重量化的模型（如 `TheBloke/OpenHermes‑2.5‑Mistral‑7B‑AWQ`）  
- **多专家 MoE**：`fused_moe` 与 `moe_wna16` 代码路径  
- **平台适配**：CUDA、CPU、Intel XPU、ROCm（`supported_quantization` 列表）  
- **文档与示例**：`docs/features/quantization/auto_awq.md` 与示例脚本  
- **测试套件**：`tests/quantization/*`（新增 231 行）  

**🔍 技术洞察**  

- **架构影响**  
  - **统一配置层**：`AutoAWQConfig` 取代 `AWQConfig` 与 `AWQMarlinConfig`，从而在同一类中根据运行时平台自动选择 Triton、Marlin 或 XPU 实现，简化了 `get_quantization_config` 与 `override_quantization_method` 的分支逻辑。  
  - **平台分派**：`override_quantization_method` 现在在 CPU 上直接返回 `None`，防止误用 Auto‑AWQ；在 CUDA 环境下仍根据 `envs.VLLM_BATCH_INVARIANT` 与 `check_marlin_supported` 决定是否走 Marlin。  
  - **XPU 支持**：新增 `AutoAWQXPULinearMethod`，在权重加载后调用 `vllm_xpu_kernels` 中的 `AWQUtils.repack` 与 `int4_gemm_w4a16`，实现了 XPU 的原生 INT4 GEMM，彻底修复了原先 `awq_dequantize` 在 XPU 上的崩溃。  
  - **MoE 兼容**：`MoE` 相关代码改为检测 `AutoAWQConfig`，并在不支持 Marlin 时回退至 `MoeWNA16Config`，保持原有 **awq → awq** 与 **auto_awq → auto_awq** 的一致性。  
  - **后向兼容**：在 `vllm/config/model.py` 将 `"awq"`、`"awq_marlin"`、`"auto_awq"` 全部映射到 `AutoAWQConfig`，保障老用户的配置仍能工作（仅在文档中建议迁移到 `auto_awq`）。  

- **性能影响**  
  - **XPU**：通过 XPU 专用 `int4_gemm_w4a16` 替代原先的 CPU‑fallback `awq_dequantize`，大幅提升在 Intel Arc / Xe‑HPC 上的吞吐量（理论上比 CPU‑MATMUL 快 2‑4×），并消除了之前的运行时错误。  
  - **CUDA**：保持原有 Marlin 与 Triton 两条路径；在 `envs.VLLM_BATCH_INVARIANT` 为 `True` 时仍走 `torch.matmul`（兼容批量不变模式），其性能与旧实现相当。  
  - **CPU**：仍使用 Marlin‑CPU 实现 (`CPUWNA16LinearKernel`)；由于 `override_quantization_method` 不再在 CPU 上强制使用 Auto‑AWQ，CPU 端行为保持不变。  
  - **整体**：统一入口避免了二次解析配置的开销，且 `from_config` 复制 `full_config` 并强制 `quant_method="awq"`，保证 MoE fallback 不会因 `"auto_awq"` 失效而产生异常。  

- **安全考虑**  
  - 代码仅在量化路径中添加了对平台检查与异常提示，没有引入外部网络、文件写入或加密相关逻辑。  
  - 对 XPU 的依赖通过 `import vllm_xpu_kernels` 实现，若未安装则抛出明确的 `ImportError`，避免在不受支持的环境中产生未捕获的低级错误。  
  - 迁移过程中删除了 `awq.py`，若有第三方插件仍显式导入 `vllm.model_executor.layers.quantization.awq` 将触发 `ModuleNotFoundError`，建议在发行说明中给出兼容性提示。  

**⚠️ 潜在风险**  

| 风险点 | 可能的后果 | 缓解措施 |
|--------|-----------|----------|
| **旧版代码仍引用 `AWQConfig`** | 运行时 `ImportError` 或配置不匹配。 | 检查所有自定义插件、脚本，改为 `auto_awq`；提供迁移文档和软链接（如在 `__init__.py` 中保留别名）。 |
