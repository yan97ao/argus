### [Core] Add All-to-All communication backend for DCP  (#34883)
**SHA**: `6cb9010` | 🔗 [查看提交](https://github.com/vllm-project/vllm/commit/6cb901093f3df8e26cbc0a8a0e1a884f4dbaa5ea)

**🎯 变更类型**：功能增强 / 性能优化  
**⚡ 重要程度**：🔴高  

**📋 变更摘要**  
本次提交为 Decode Context Parallel（DCP）引入了全新的 All‑to‑All（A2A）通信后端 `a2a`，通过在各 DCP rank 之间直接交换局部注意力输出及其 LSE（Log‑Sum‑Exp），随后使用 Triton 实现的 LSE‑加权合并 kernel 完成结果聚合。相较于原先的 AllGather + ReduceScatter（`ag_rs`），A2A 将每层的 NCCL 调用从 3 次降至 2 次，可显著降低长上下文解码阶段的通信延迟。同时新增了配置选项、参数校验、CLI 参数、以及对应的单元测试。

---

### 🎯 影响范围
- **核心模块**：`vllm/config/parallel.py`, `vllm/engine/arg_utils.py`, `vllm/model_executor/layers/attention/mla_attention.py`, `vllm/v1/attention/backends/flash_attn.py`, `vllm/v1/attention/backends/flashinfer.py`  
- **新后端实现**：`vllm/v1/attention/ops/dcp_alltoall.py`（Triton kernel + CPU reference）  
- **测试覆盖**：`tests/distributed/test_dcp_a2a.py`（配置校验、LSE 加权组合的正确性）  

---

### 🔍 技术洞察

| 维度 | 影响分析 |
|------|----------|
| **架构影响** | 1. 在 ParallelConfig 中新增 `dcp_comm_backend`（`"ag_rs"` / `"a2a"`），并将其向上层（CLI、EngineArgs、VllmConfig）传播。<br>2. 通过 `get_current_vllm_config_or_none()` 动态判断运行时是否启用 A2A，避免在不支持的环境中触发异常。<br>3. 新增 `dcp_a2a` 标记并在 MLA、FlashAttention、FlashInfer 等注意力实现中分支选择 `dcp_a2a_lse_reduce` 或原 `cp_lse_ag_out_rs`。|
| **性能影响** | - **通信次数**：原 `ag_rs` 需要 3 次 NCCL（AllGather Q、AllGather K‑meta、ReduceScatter 输出），A2A 只需 2 次（一次 All‑to‑All 输出，一次 All‑to‑All LSE），在跨节点、GPU 数量较多的场景下降低了通信启动开销与链路占用。<br>- **计算开销**：Triton 实现的 LSE 加权合并在 GPU 上 O(N·B·H·D) 的计算量基本保持不变，但利用了 Tensor‑core 并行，实际吞吐率可比 CPU 实现提升 2‑4×。<br>- **内存占用**：需要额外的 `recv_output`、`recv_lse` 缓冲区，等价于原始输出大小的 2 倍（但在同一次 All‑to‑All 中可重用通信缓冲），对显存需求略有提升。|
| **安全考虑** | - 新增的 All‑to‑All 使用 `torch.distributed.all_to_all_single`，若各 rank 的 `world_size`、`tensor_shape` 不匹配会导致死锁或数据错位，已在 `ParallelConfig._validate_parallel_config` 中加入 `dcp_comm_backend="a2a"` 时的大小校验。<br>- Triton kernel 对 `nan/inf` LSE 做了显式过滤，防止数值异常传播。<br>- 代码路径保持向后兼容（默认仍是 `ag_rs`），不影响已有部署的安全性。|
| **可维护性** | - 新增的 Triton kernel 与 CPU 参考实现分离，单元测试覆盖了多个数值边界（相同 LSE、极端 LSE、返回全局 LSE），有助于后续维护。<br>- 通过 `get_current_vllm_config_or_none()` 控制分支，避免在未完成配置加载时导入不必要的依赖，降低了模块耦合度。|
| **可扩展性** | 可在未来进一步支持 **混合后端**（如在部分层使用 A2A、部分层使用 AG‑RS），或在多机多卡场景下调度不同的通信实现。Triton kernel 已参数化 `IS_BASE_E`、`RETURN_LSE`，便于适配不同 LSE 基底或返回需求。|

---

### ⚠️ 潜在风险
1. **跨节点不一致**：若某些节点仍使用默认 `ag_rs`，而其他节点配置为 `a2a`，会导致 All‑to‑All 参数不匹配产生死锁。需在部署脚本或启动包装器中确保全局统一。  
2. **显存压力**：在极大模型（如 130B）且 DCP 并行度较高时，额外的 `recv_output` 与 `recv_lse` 缓冲可能接近显存上限，建议在 `torch.cuda.memory_summary()` 中监控。  
3. **Triton 兼容性**：Triton kernel 依赖特定的 CUDA/Compute Capability（>=7.0）以及对应的 `triton` 版本，老旧环境可能无法编译；需要在 `requirements.txt` 或文档中明确最低版本。  
4. **数值精度**：在极端 LSE（如 -1e9）情况下，`exp(lse - max)` 仍会触发下溢为 0，权重归一化后可能出现全部为 0 的情况，导致输出全为 0。当前实现对 `-inf` 处理为 0，仍需在极端训练情形下关注梯度传播。  
5. **调试难度**：All‑to‑All 的错误往往表现为“非对称结果”而非显式异常，调试需要对每个 rank 的 `send_output`/`recv_output` 进行打印或使用 NCCL 检查工具。

---

### 💡 关注建议
- **部署前检查**：在多机启动脚本中加入 `--dcp-comm-backend a2a` 参数统一检查，确保所有节点使用相同的 `dcp_comm_backend`。  
- **显存监控**：在开启 A2A 前后对 `torch.cuda.max_memory_allocated()` 进行对比，评估显存增长是否在可接受范围。  
- **兼容性测试**：在 CI 中加入跨机器（2‑4 节点）带 GPU 的集成测试，验证 All‑to‑All 的正确性与性能提升。  
- **文档更新**：在官方文档的 *Parallelism* 章节补充 A2A 的使用说明、适用场景（长上下文解码、GPU 数量 ≥ 8）以及已知限制。  
- **回退机制**：建议在运行时检测 NCCL/torch.distributed 支持的 All‑to‑All（`dist.is_nccl_available()` 且 `torch.distributed.is_initialized()`），若不满足直接回退到 `ag_rs`，并给出警告日志。  
- **数值安全**：在极端 LSE 场景下，可考虑在 `dcp_lse_combine_triton` 前对 LSE 做 `torch.clamp(min=-MAX_LOG)`，防止 `exp` 产生 NaN。  

---

**总体结论**  
此次引入的 DCP All‑to‑All 后端是一次**高价值的功能增强与性能优化**。它在保持原有行为的同时，为大规模解码场景提供了显著的通信延迟削减，且通过严格的配置校验与单元测试降低了风险。只要在生产环境中做好统一配置、显存和兼容性检查，就能安全地获得性能收益。持续关注跨节点测试及数值极端情况的稳健性，将有助于该特性的平滑落地。

| **可维护性** | - 代码量约 800 行，新模型实现遵循 vLLM 现有抽象（`Interfaces`, `BaseMultiModalProcessor`），整体结构清晰。<br>- 采用 `ReplicatedLinear` 替代普通 `nn.Linear`，保持与多卡并行加载的一致性。<br>- 处理音频的 **dummy inputs** 与 **prompt 替换** 逻辑较为复杂，建议后续抽离为公共工具函数，以降低重复代码。<br>- `FireRedASR2FeatureExtractor` 参数列表较长，且有很多未使用的超参（如 `max_length`），可考虑精简或在 `__init__` 中标记为 `**kwargs`。 |

## ⚠️ 潜在风险

1. **依赖安全** – `kaldi-native-fbank` 为 C++ 扩展，若使用不受信任的 wheel（如内部镜像）可能引入供应链风险。  
2. **显存泄漏** – `Conv2dSubsampling`、`ConformerEncoder` 在 `forward` 中使用 `torch.nn.functional.pad` 与 `torch.masked_fill_`，在长序列或大 batch 时可能产生未释放的临时张量，导致 OOM。  
3. **权重映射不完整** – `hf_to_vllm_mapper` 中未覆盖所有 encoder 层（仅映射了 `pre_layer_norm`、`linear_expand`、`linear_project`），若模型结构细节变更（例如新增层），加载会失败或产生维度不匹配。  
4. **假 token 长度计算** – `fake_token_lengths` 使用 `torch.clamp(..., min=1)`，在极短音频（< frame）时可能产生误差，导致 `<|AUDIO|>` token 数目与实际特征帧不匹配，进而出现 token 对齐错误。  
5. **API 兼容性** – `FireRedASR2Processor.__call__` 必须同时提供 `audio` 与带有相同数量 `<|AUDIO|>` 标记的 `text`。若用户忘记提供 `text`（仅音频）或提供不匹配的 token 数，当前实现会抛异常，这在高层 API（如 OpenAI‑style）中可能导致不友好的错误信息。  
6. **文档/依赖同步** – `requirements/common.txt` 加入 `kaldi-native-fbank >=1.18.7`，但未在 `setup.cfg` 中列出，可导致某些构建方式（例如 `pip install .[all]`）缺失该依赖。

5. **深度堆叠（deepstack）模式的兼容**

### [Refactor] Clean up processor kwargs extraction (#35872)
**SHA**: `e379396` | 🔗 [查看提交](https://github.com/vllm-project/vllm/commit/e3793961674af8bf01208b2216542ad00ae325e6)

**变更概况**  
本次提交对 *processor* 中 kwargs 的提取逻辑做了结构性重构：原来的 `get_processor_kwargs_from_processor`（返回键集合）被拆分为  
1. `get_processor_kwargs_type` – 负责从 Processor 实例中找出对应的 `ProcessingKwargs`（或 `ProcessorKwargs`）类；  
2. `get_processor_kwargs_keys` – 根据该类解析出所有动态键（包括 `text_kwargs、images_kwargs、videos_kwargs、audio_kwargs`）。  

相应测试也改为先获取类型再取键，`cached_get_processor_without_dynamic_kwargs` 也同步使用新 API。

**核心影响**  
- **API 变动**：原 `get_processor_kwargs_from_processor` 被移除（仅在内部使用），外部代码若仍依赖该函数将报错。  
- **键收集方式**：原先通过 `_collect_dynamic_keys_from_processing_kwargs` 对每个 `*ProcessingKwargs` 进行累加并合并，现在只在找到首个 `*ProcessorKwargs`（或显式的 `Unpack` 注解）后返回该类，再统一由 `get_processor_kwargs_keys` 解析。若同一模块定义多个 `*ProcessorKwargs`，仅会取第一个，行为与旧实现略有差异。  
- **异常容错**：若任何步骤抛异常，统一回退到 `processing_utils.ProcessingKwargs` 并只返回四个 modality 键，确保不会中断模型加载。  

**潜在风险**  
1. **向后兼容**：库外部或文档中仍可能引用已删除的 `get_processor_kwargs_from_processor`，建议保留一个薄包装的兼容层或在发布说明中明确迁移路径。  
2. **多 Kwargs 类**：如果某些 Processor 模块里同时存在 `*ProcessingKwargs` 与 `*ProcessorKwargs`，或出现多个 `*ProcessorKwargs`，新的实现只会使用第一次匹配的类，可能导致遗漏键。可以在遍历时合并所有匹配的类以保持旧行为。  
3. **类型解析**：`Unpack[SomeKwargs]` 仍通过 `get_args` 取得内部类型，但若出现 `Union[SomeKwargs, Other]` 或 `typing_extensions.Annotated`，会返回非预期的对象。可以在 `get_processor_kwargs_type` 加入额外的类型检查或回退逻辑。  

**建议**  
- 在 `processor.py` 中保留 `def get_processor_kwargs_from_processor(...): return get_processor_kwargs_keys(get_processor_kwargs_type(...))` 的兼容实现，避免突发破坏。  
- 完善函数 docstring，说明两步获取的意图以及在异常时的默认返回。  
- 在单元测试里加入「同一模块出现多个 ProcessorKwargs」的情形，确保新实现不遗漏键。  
- 监控 CI 中是否还有旧 API 的引用，及时清理或更新。  

总体来说，重构提升了代码可读性与缓存粒度，潜在兼容性影响可通过上述措施降低。
