#### 🔴 高重要度变更 (6)

### [Models] Cohere MoE (#40817)
**SHA**: `91a2d39` | 🔗 [查看提交](https://github.com/vllm-project/vllm/commit/91a2d3901416fcff11e192f32683ca963726989b)

**🎯 变更类型**：功能增强  
**⚡ 重要程度**：🔴高  
**📋 变更摘要**：本次提交为 vLLM 项目新增对 Cohere 系列 MoE（Mixture‑of‑Experts）模型的原生支持，实现了 `CohereMoeForCausalLM` 及其底层组件（注意力、MLP、路由函数），并在模型注册、文档、测试以及路由层识别中完成了相应的集成。目标是让用户能够直接使用 Command‑R（MoE 版）模型进行高效推理。  

**🎯 影响范围**：  
- `vllm/model_executor/models/cohere_moe.py`（核心实现）  
- `vllm/model_executor/models/registry.py`（模型注册表）  
- `docs/models/supported_models.md`（文档）  
- `tests/models/registry.py`（测试用例）  
- `vllm/model_executor/layers/fused_moe/router/custom_routing_router.py`（路由方法识别）  
- `vllm/model_executor/layers/fused_moe/experts/trtllm_nvfp4_moe.py`（路由方法列表）  

**🔍 技术洞察**：

- **架构影响**  
  - 引入 **CohereMoeModel** 作为全新 Transformer 解码器，复用了已有的 `FusedMoE`、`Attention`、`VocabParallelEmbedding` 等组件。  
  - 新增 **自定义路由函数** `token_choice_with_bias`（Sigmoid → Top‑K → (可选) Renorm），并在 `custom_routing_router` 中映射到 `RoutingMethodType.SigmoidRenorm`，实现对 MoE 专家的动态选择。  
  - 通过 `SupportsPP` 与 `SupportsQuant` 接口，保持与现有流水线并行（PP）和量化框架的一致性。  
  - 在 `CohereMoe` 中加入 **共享专家**（shared_experts）支持，允许配置 `num_shared_experts` 与组合策略（sum / average），提升模型在稀疏专家场景下的灵活性。  

- **性能影响**  
  - MoE 本身通过稀疏激活显著降低每张卡的计算量，理论上在大规模专家数下可提升吞吐量。  
  - 自定义路由 `token_choice_with_bias` 使用 `torch.topk` 与归一化，额外的算子开销相对轻微，但在极高并行度（如 TP=8）下需要关注 GPU‑Kernel 启动成本。  
  - `FusedMoE` 已支持 TOP‑K、归一化等特性，新增的路由方式在 `trtllm_nvfp4_moe.py` 中被列入支持列表，意味着在 TensorRT‑LLM 后端也能利用加速。  
  - 量化路径保持兼容，`QuantizationConfig` 直接传递至所有 Linear 层，避免额外性能回退。  

- **安全考虑**  
  - 仅在文档和测试中声明 “trust_remote_code=True”，与其他 Cohere 系列模型保持一致，无额外安全风险。  
  - 代码未引入外部依赖或网络交互，安全面影响有限。  

**⚠️ 潜在风险**  

1. **路由方法重复列入**：`trtllm_nvfp4_moe.py` 中将 `RoutingMethodType.SigmoidRenorm` 添加了两次，可能导致配置解析或日志输出的歧义。  
2. **自定义路由函数兼容性**：`custom_routing_router` 只在 `token_choice_with_bias` 匹配时返回 `SigmoidRenorm`；若后续模型改为其他路由函数，需要同步更新识别逻辑。  
3. **权重加载复杂度**：`CohereMoeForCausalLM.load_weights` 中涉及多层映射、专家切分、共享专家合并，容易产生参数名错配或缺失，尤其在使用 `trust_remote_code=False` 时。  
4. **TP 与专家数不匹配**：构造时已检查 `tp_size > num_experts` 抛异常，但在分布式环境下若配置错误可能导致进程卡死。  
5. **缺失在线可用性检测**：在 `tests/models/registry.py` 将该模型标记为 `is_available_online=False`，但未提供离线检查脚本，可能导致 CI 误报。  

**💡 关注建议**  

- **代码清理**：删除 `trtllm_nvfp4_moe.py` 中的重复 `RoutingMethodType.SigmoidRenorm` 条目，保持列表唯一。  
- **单元/集成测试**：新增对 `token_choice_with_bias` 的数值正确性、TOP‑K 计数、归一化行为的回归测试；验证在不同 `tp_size` 与 `num_experts` 配置下的 forward 路径是否保持一致。  
- **权重检查**：在 `load_weights` 完成后加入日志或断言，确保所有预期的专家参数都已被加载（尤其是共享专家相关的 `gate_up_proj` 与 `down_proj`）。  
- **文档同步**：在 `supported_models.md` 中补充对 MoE 特有的配置项说明（如 `num_shared_experts`、`shared_expert_combination_strategy`、`expert_selection_fn`），帮助用户正确构造模型配置。  
- **异常处理**：为 `tp_size > num_experts` 场景提供更友好的错误信息或自动降级提示，以免在大规模集群部署时出现非预期的启动失败。  
- **性能基准**：在 CI 或内部 benchmark 中加入 MoE 与非 MoE 同尺寸模型的吞吐、显存对比，明确新模型的收益范围，指导用户在实际生产环境做选型。

### [Feat] CPU fp8 attn for AMX/AVX-512 (#39445)
**SHA**: `22524f7` | 🔗 [查看提交](https://github.com/vllm-project/vllm/commit/22524f7a92b71c8e65eade20ef274fa3b4006d3e)

**🎯 变更类型**：功能增强（CPU FP8 KV‑Cache 支持，覆盖 AMX 与 AVX‑512 路径）  
**⚡ 重要程度**：🔴 高  

**📋 变更摘要**  
- 在 CPU 注意力实现中加入对 8 位浮点（FP8）键值缓存的完整支持，新增量化/反量化、比例缩放以及对应的解码逻辑。  
- 扩展所有 `AttentionImpl` 模板以接受 `kv_cache_scalar_t`，并在 AMX、VEC（AVX‑512/AVX2）路径上实现 FP8 → BF16/FP32 的高效解码。

## 🔍 技术洞察

| 维度 | 影响描述 |
|------|----------|
| **架构影响** | 1. **模板参数化**：`AttentionImpl` 现在接受四个模板参数 `<ISA, scalar_t, head_dim, kv_cache_scalar_t>`，实现对不同 KV‑Cache 表现的静态分发。<br>2. **新枚举** `Fp8KVCacheDataType` 与 `kv_cache_dtype` 参数，统一在 C++ 与 Python 层的表示。<br>3. **调度宏扩展**：`CPU_ATTN_DISPATCH` 现在接受 `KV_CACHE_IDX`，在编译时根据 head‑dim、ISA 与 KV‑Cache 类型挑选特化实现。<br>4. **平台约束**：仅在检测到 AMX（Intel BF16）或 AVX‑512（x86）时生成 FP8 分支；其它平台恢复原 `auto` 行为。 |
| **性能影响** | - **内存带宽**：KV‑Cache 从 16 B（BF16）或 32 B（FP32）降低至 8 B，实现 2×‑4× 的缓存占用缩减，显著提升大模型推理时的 L2/L3 命中率。<br>- **量化/反量化开销**：在 `reshape_and_cache_fp8_*` 中使用向量化 (AVX‑512/AVX2) 或 AMX Tile‑Level 解码，将每 32 B FP8 直接展开为 BF16，无需中间 FP32 循环，理论上成本 ≤ 5% 的额外算子时间。<br>- **计算路径**：在 QK 与 PV 阶段的 `execute_attention` 中通过 `scale *= k_scale * bias` 将缩放提前，消除额外乘法。<br>- **实际加速**：取决于模型大小、块大小和 head‑dim（需 32 整除），预计在 90%+ KV‑Cache 场景下整体推理时间下降 10‑20%。 |
| **安全考虑** | - **输入校验**：新增 `TORCH_CHECK` 对 KV‑Cache `torch.uint8` 类型、`k_scale/v_scale > 0`、ISA 兼容性进行严格校验，防止恶意或错误参数导致内存泄漏/越界。<br>- **数值范围**：量化函数对输入做了 `[-fp8_max, fp8_max]` 裁剪，避免 NaN/Inf 产生。<br>- **平台特化**：在非 x86 平台或不支持 AMX/AVX‑512 时，调度宏会直接报错，避免在不安全的路径上执行未验证的 FP8 解码。 |
| **可维护性** | - **代码分离**：FP8 量化、解码封装在 `cpu_attn_fp8.hpp`，与原有 BF16/FP32 路径解耦，后续可单独演进。<br>- **模板膨胀**：`AttentionImpl` 与 `TileGemm*` 增加模板特化，编译时间略增，但因仅在特定 ISA/kv 类型组合生成代码，整体可接受。<br>- **API 向后兼容**：在 Python 层提供默认参数 `k_scale=1.0, v_scale=1.0, kv_cache_dtype="auto"`，旧代码仍可运行，仅在显式使用 FP8 时才会触发新行为。 |

## ⚠️ 潜在风险

1. **平台兼容错误**  
   - 若用户在不支持 AMX/AVX‑512 的机器上强制使用 `kv_cache_dtype="fp8"`，会触发运行时 `TORCH_CHECK`，但在多进程或分布式启动时可能出现 “partial launch” 的不一致错误。  
2. **数值误差**  
   - FP8 E4M3 最大相对误差约 12.5%，E5M2 约 25%。在对 KV‑Cache 进行多次乘加后累计误差可能导致注意力分布偏差，尤其在深度层数很多时。测试中使用了 `atol/rtol` 放宽，但在极端模型（如 GPT‑NeoX‑20B）可能出现可感知质量下降。  
3. **量化尺度不匹配**  
   - `k_scale`、`v_scale` 需要与模型训练时使用的尺度保持一致。错误的尺度会导致系统性偏差或数值溢出。当前只检查正数，未做范围校验。  
4. **编译器/CPU 特性差异**  
   - `cpu_attn_fp8.hpp` 中的 AVX‑512/AVX2 BF16Vec32 构造使用 `_mm512_cvtepu8_epi16` 等指令，老旧的 GCC/Clang 可能不完整支持或产生错误代码。  
5. **调度宏膨胀**  
   - `generate_cpu_attn_dispatch.py` 现在为每种 `kv_cache` 生成多倍的 case（≈3 倍），导致编译期 CPU 与二进制体积增大。对极端 head‑dim/ISA 组合可能触碰编译器 switch‑case 限制。  
6. **多线程安全**

|------|----------|
| **架构影响** | 1. **元数据演进**：`OffloadingConnectorMetadata` 由 `reqs_to_load/store` → `load_jobs/store_jobs`（job‑id 为键），并新增 `jobs_to_flush`。<br>2. **调度器 ↔ 工作节点**：使用 `OffloadingWorkerMetadata.completed_jobs` 汇报 **每个 job 完成的计数**，调度器在 `update_connector_output` 中聚合并在计数归零时触发 `complete_load/store`。<br>3. **Fence 机制**：新增 `_block_id_to_pending_jobs`，在 `request_finished` 时记录仍在执行的 store，后续 block 再被分配时会自动 Flush（通过 `jobs_to_flush`）。<br>4. **状态拆分**：每个请求的 `transfer_jobs` 集合维护其在-flight job‑ids，避免 load 与 store 同时出现冲突。 |

1. **计数不一致**：如果某个 job 在多个 worker 上完成计数出现偏差（比如网络异常导致重复报告），`pending_count` 可能出现负值，导致 `complete_store/load` 未被调用。<br>**缓解**：在 `update_connector_output` 中加入 `assert job_status.pending_count >= 0` 并在异常路径记录错误日志。

4. **性能回退**：在极小的 batch（如单 token 推理）场景下，额外的 per‑job bookkeeping可能导致微小的 latency 上升。<br>**缓解**：可在 `SchedulerOffloadConfig` 中加入开关 `enable_per_job_store`，在 batch<阈值时回退到旧的 “全请求 store 完成后再 flush” 逻辑。

- **监控 & 统计**：在 `OffloadingConnectorStats` 中加入 `store_job_latency`、`store_job_pending_counts` 等指标，帮助运维判断 fence 触发频率与延迟。  
- **回归测试**：加入 **多 worker**（`world_size>1`）的集成测试，验证 `completed_jobs` 在分布式环境下正确聚合。  
- **文档更新**：在 `KVConnector` 文档中注明 **Per‑Job Store Completion** 机制、`jobs_to_flush` 与 `_block_id_to_pending_jobs` 的语义，提醒插件作者避免直接使用已废弃字段。  
- **异常处理**：在 `worker.get_finished` 捕获异常（例如网络超时）时，仍应记录失败的 `job_id`，并在调度器侧对未完成的 job 进行超时重试或强制 flush，以防永久阻塞 fence。

**结论**：本次提交通过引入 per‑job store 完成与 fence 机制，显著提升了 KV offloading 的并发利用率和块复用安全性。虽然代码复杂度有所提升，但通过详细的单元测试与清晰的元数据抽象，风险可控。建议在生产环境开启前完成多节点压测，并在监控中关注 `store_job` 的完成计数与 fence flush 行为。
