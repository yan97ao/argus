### [quant][autoround]Refactor INC quantization into package with INCScheme orchestrator (#40601)
**SHA**: `bcb518a` | 🔗 [查看提交](https://github.com/vllm-project/vllm/commit/bcb518ad7a2138ad51b5f9912c07c63af35cee86)

**🎯 变更类型**：功能增强 / 重构 / 架构变更  

**⚡ 重要程度**：🔴高  

**📋 变更摘要**  
- 将原本在 `vllm/model_executor/layers/quantization/inc.py` 中实现的 *Intel Neural Compressor (INC)* 量化逻辑拆分为独立的 `inc/` 包，新增 `INCConfigParser`、方案（scheme）工厂以及对线性层/ MoE 层的统一调度入口。  
- 引入 **INCScheme orchestrator**：通过 `resolve_scheme` 根据 `INCLayerConfig` 动态选择具体实现（如 `INCWna16Scheme`、`INCWna16LinearScheme`、`INCARKLinearMethod` 等）。  
- 添加大量单元测试，覆盖配置解析、融合键（fusion key）误匹配修复、不同平台（CPU/XPU）下的后备路径以及 MoE 量化分支。  
- 在 CI 中加入 XPU 量化测试步骤。  
- 更新 `requirements/xpu.txt` 与 `test-intel.yaml`，确保 CI 能拉取最新 `auto_round_lib`。  

**🎯 影响范围**  
- `vllm/model_executor/layers/quantization/inc/` 包及其子模块（config_parser、schemes、inc、inc_linear）。  
- 量化配置解析入口 `INCConfig.get_layer_config`、`INCConfig.get_quant_method`。  
- 依赖 `auto_round_kernel`（ARK） 的 XPU/CPU 4‑bit 量化路径。  
- CI 流程（Buildkite Intel jobs）。

### 🔍 技术洞察

| 维度 | 影响分析 |
|------|----------|
| **架构影响** | - **模块化**：原本 800 行的单体实现被拆分为 *配置解析 → Scheme 选择 → Linear/MoE 实现* 三层结构，提升可维护性和可扩展性。<br>- **工厂模式**：`resolve_scheme` 让以后新增量化方案（如 FP8、MX‑FP）只需实现 `INCScheme` 接口并注册，避免在核心 `INCConfig` 中散布大量 `if/elif`。<br>- **向后兼容**：`INCConfig.get_quant_method` 仍保留原有 “model.” 前缀的早退出逻辑，已有模型不受影响。 |
| **性能影响** | - **运行时开销**：额外的层级调度（`resolve_scheme` → `scheme.get_*_method`) 带来极小的函数调用开销，几乎可忽略。<br>- **实际算子**：未改变底层算子实现（仍使用 Marlin、ARK、XPU int4 kernel），因此推理吞吐和 latency 与旧实现保持一致。<br>- **错误路径**：在 ARK 不可用时会回退到 XPU/CPU 自实现，这可能稍慢，但已有相同回退逻辑；新实现把回退代码集中在 `inc_wna16_scheme.py`，更易调优。 |
| **安全考虑** | - **动态导入**：`get_ark_state` 使用 `import auto_round_kernel` 并捕获 `ImportError`，在缺失库时返回 `False`，不抛异常，安全性保持。<br>- **正则解析**：新增 `regex`（而非标准 `re`）处理用户提供的 `extra_config` 模式。`regex` 支持更强的表达式，但也可能带来 **ReDoS**（正则拒绝服务）风险；代码在解析时捕获 `re.error` 并忽略非法模式，降低风险。<br>- **外部依赖**：`auto_round_lib>=0.13.3` 升级，确保安全补丁得以更新。 |
| **可维护性** | - **代码拆分**：每个文件职责单一（配置解析、线性实现、方案工厂），代码阅读与审查成本下降。<br>- **单元测试**：新增 1.5k 行测试，覆盖关键分支（正则匹配、融合键、ARK 可用性、平台分支），提升回归安全。<br>- **文档/导出**：`__all__` 明确导出 `INCConfig`，防止误用内部实现。 |
| **兼容性** | - 对外 API 仍为 `INCConfig`，`from_config`、`get_layer_config`、`get_quant_method` 保持签名不变。<br>- 迁移时未删除旧 `inc.py`（已抽离为新包），因此老版本的 `import vllm.model_executor.layers.quantization.inc` 仍可工作。<br>- 需要确保 **torchscript**、**pickle** 等序列化方式在新类层级下仍能恢复；目前未显式测试，需要在后续验证。 |

### ⚠️ 潜在风险

1. **正则表达式误用**  
   - `extra_config` 中的正则如果写得过于宽泛或嵌套捕获组，可能匹配大量子层导致 **配置膨胀**，进而触发不必要的 FLOP 增加或意外量化。  
   - 解决方案：在文档中明确推荐使用锚点 (`^`/`$`) 或限定层级的正则，并在 CI 加入正则复杂度检查（如 `re2` 的 DFA 大小限制）。

2. **平台判定分支遗漏**  
   - `get_quant_method` 的早退出只检查 `bits >= 16`，若用户在 `extra_config` 中手动指定 `bits=32`（非量化）但 `quantized=False` 仍会走 `scheme` 路径，可能返回不必要的 `LinearMethod`。  
   - 建议在 `INCConfigParser._resolve_raw` 中加入 `bits >= 16 → quantized=False` 的统一判断。

3. **序列化兼容**  
   - `INCConfig` 现在持有 `config_parser` 实例（不可 picklable），旧的 `torch.save(state_dict)` 后再 `torch.load` 时可能报错。  
   - 可在 `__getstate__/__setstate__` 中排除该属性，或在 `from_config` 时重新构造。

4. **ARK 后端加载失败的日志噪声**  
   - 当 ARK 不可用时会在每个 4‑bit 层打印 `debug`，在大模型上可能产生大量日志。可改为 **一次性** 警告或在首次失败后缓存结果。

5. **依赖冲突**  
   - 新增 `regex` 库可能与项目中已有的 `re` 使用产生版本冲突（尤其在某些 Linux 发行版的 `regex` 包）。应在 `requirements.txt` 中显式声明版本范围（如 `regex>=2023.12.25`）避免意外回滚。

### 💡 关注建议

| 对象 | 建议 |
|------|------|
| **开发者** | - 在新增或修改量化配置时，优先使用 **exact key** 而非正则，除非确有跨层统一需求。<br>- 如果需要在 `extra_config` 中对融合键（如 `qkv_proj`）进行配置，确保对应的子键在 `extra_config` 中实际出现；否则旧的误匹配已被修复，测试用例 `test_false_substring_match_does_not_override` 会帮助验证。 |
| **平台维护者** | - 在 XPU CI 中加入 **ARK 可用性检查**（`auto_round_kernel` 是否能成功加载），避免因库缺失导致 CI 失真。<br>- 对 CPU/CPU‑GPU 混用场景，确认 `current_platform.is_cpu()` 与 `is_xpu()` 判定逻辑在多卡环境下的行为一致。 |
| **用户** | - 迁移到新版本时 **无需修改代码**，但若自行编写 `extra_config`，请参考新版文档

**⚡ 重要程度**：🔴高  

**📋 变更摘要**  
- 为 MiniMax M2 大模型新增完整的 **Streaming Parser Engine**，实现 XML‑style `<invoke>` / `<parameter>` 工具调用的解析。  
- 在 parser‑engine 框架中加入 **MinimaxM2Parser**、对应的 Reasoning/Tool 适配器、注册入口以及测试套件。

| **性能影响** | • `incremental_lexer._drain` 现在在 **非终止**（`final=False`）时检测是否还可能出现更长的 literal 前缀，避免在流式输入中误提前切分，例如 `<invoke name=` 前缀被错误输出为单独 token。此改动降低了 **无效 token 生成** 与 **后续重解析** 的开销，对所有使用增量词法的模型都有正面影响。<br>• MiniMax M2 解析使用 **一次性参数转换** (`arg_converter`) 而非逐字符累积，保持与其他模型相同的 O(N) 复杂度。 |
| **安全考虑** | • `ParserEngine._is_valid_tool_name` 结合 `tool_parsers.utils.find_tool_name`，在 **工具调用** 生成前检查名称是否在请求中声明的 `tools` 列表里。<br>• 只在 MiniMax M2 的配置里默认开启 `validate_tool_names=True`，避免对已有模型产生意外拒绝。<br>• 通过统一的 `find_tool_name` 实现，避免手动字符串比较带来的误判。 |
| **可维护性** | • 将 MiniMax M2 的解析规则抽象为 **声明式配置**，后续如果 MiniMax M2 更新其 XML 语法，只需修改 `ParserEngineConfig` 而不必重写大量状态机代码。<br>• 测试覆盖面广（非流式、流式、分块、错误路径），并移除了原先的 **内部状态** (`prev_tool_call_arr`) 测试，防止实现细节泄漏。 |
| **兼容性** | • 新增的 `ParserEngineConfig.validate_tool_names` 默认值为 `False`，保持对已有模型的兼容性。<br>• 旧的 `MinimaxM2ToolParser` 被裁剪为适配器子类，外部 API（类名、构造函数签名）保持不变，现有调用代码无需修改。 |

**⚠️ 潜在风险**  
1. **工具列表缺失导致误拒**  
   - 对 MiniMax M2，`validate_tool_names` 默认开启。如果调用方未在 `request.tools` 中提供对应函数，工具调用将被 **全部丢弃**（`tools_called=False`），可能导致业务层误以为模型未产生工具调用。  
2. **增量词法改动对其他模型**  
   - `incremental_lexer._drain` 的新前缀检测逻辑在极端情况下可能导致 **漏切**（某些合法短 token 被误认为是更长 literal 的前缀），虽风险低，但需在高并发/多模型环境下进行回归验证。  
3. **EOS/结束标记处理**  
   - 原先的 EOS 处理代码（在 `tool_parser` 中对 `EOS_ID` 的特殊处理）被移除。若底层模型在工具调用块后仅返回一个 **空 token**，可能导致 **未发送空 content**，从而使流式响应在客户端卡住。  
4. **配置不一致**  
   - `ParserEngineConfig.validate_tool_names` 的默认 `False` 与 MiniMax M2 配置的 `True` 形成差异，若新加入的其他模型误使用了 MiniMax M2 的配置对象，可能产生意外拒绝。  
5. **测试依赖的 MockTokenizer**  
   - 部分测试硬编码了 token ID（如 `THINK_END: 99`），如果实际 tokenizer vocab 变化，测试会失效，导致 CI 误报。  

**💡 关注建议**  
- **上线前验证**：在上线环境对 MiniMax M2 接口进行 **端到端** 流式调用测试，特别是：<br>  a) 请求中未提供 `tools` 时模型返回的 `tools_called` 是否符合预期；<br>  b) 连续多块数据（`stream_interval>1`）下的 `<invoke>` 前缀拆分是否总能正确解析。  
- **监控指标**：新增两项监控：<br>  1) `parser.tool_name_rejected_total`（被验证拒绝的工具名称计数）；<br>  2) `lexer.premature_token_emit_total`（因前缀检测失误而提前输出的 token）。  
- **文档更新**：在官方文档中说明 MiniMax M2 需要在请求 `tools` 中明确列出可调用函数，否则调用会被 silently dropped。提供示例 `tool_choice: "auto"` 与 `tool_choice: "none"` 的交互说明。  
- **回退方案**：保持 `validate_tool_names=False` 的全局默认，若出现业务侧大量“工具被误拒”投诉，可通过环境变量或配置文件临时关闭该特性。  
- **代码审查**：确认所有新增适配器 (`ReasoningAdapter`、`ToolAdapter`) 正确实现 `ParserEngine` 的抽象方法，避免未来在新模型中忘记实现 `extract_reasoning` 或 `extract_tool_calls`。  

--- 

*总体结论*：此次提交在 **功能扩展**（加入 MiniMax M2 支持）与 **安全强化**（工具名校验）上贡献显著，同时通过声明式配置提升了 **可维护性**。唯一需要重点关注的点是 **工具列表缺失时的拒绝行为** 与 **增量词法的全局影响**，建议在生产环境做充分的回归与监控。

### [KV Connector][Offloading] Avoid blocking the engine to flush offloads on idle (#45595)
**SHA**: `6d8fff5` | 🔗 [查看提交](https://github.com/vllm-project/vllm/commit/6d8fff5698aea7225792b37c3b075cc64c4a0178)

总体来看，此次改动消除了引擎在空闲时的阻塞点，提升了吞吐和响应延迟；只要确保所有 offload tier 正确实现 `has_pending_work`，即可安全部署。
