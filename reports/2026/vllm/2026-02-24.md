### [Refactor] [1/N] Reorganize kernel abstraction directory (#34055)
**SHA**: `6af03f2` | 🔗 [查看提交](https://github.com/vllm-project/vllm/commit/6af03f2394b54e95440945d2659a8120288b21ea)

**🎯 变更类型**：重构 / 架构变更  
**⚡ 重要程度**：🔴高  
**📋 变更摘要**：此次 PR 将原先分散在 `vllm.model_executor.layers.quantization.kernels.*` 下的所有线性算子（scaled‑mm、mixed‑precision 等）迁移至全新的 `vllm.model_executor.kernels.linear` 包，并在 `__init__.py` 中统一导出。与此同时，引入 `MPLinearKernel` 与 `choose_mp_linear_kernel` 统一的混合精度 kernel 选型逻辑，完善了 import 稳定性，为后续按 provider（cutlass、flashinfer、rocm 等）重构奠定基础。  

**🎯 影响范围**：  
- `vllm/model_executor/kernels/linear/**`（全新目录）  
- 所有依赖原 `layers.quantization.kernels.*` 的测试、模型实现、量化方案等文件（约 30+）  
- 运行时 kernel 选择入口 `choose_mp_linear_kernel`、`init_fp8_linear_kernel`、`init_int8_linear_kernel`  
- 与混合精度相关的压缩张量方案、GPTQ、Marlin、AllSpark 等子模块  

**🔍 技术洞察**  

- **架构影响**  
  - **模块边界清晰**：把“kernel”从层（layer）抽离成独立子包，遵循单一职责原则，便于后续按硬件/提供者拆分。  
  - **导出统一**：通过 `vllm.model_executor.kernels.linear.__init__` 进行一次性 re‑export，降低了跨模块 import 的耦合度，旧代码在迁移后仍能保持兼容（只需修改 import 路径）。  
  - **选型集中**：`choose_mp_linear_kernel` 将混合精度 kernel 的可实现性与平台能力检测集中在一处，统一管理 `VLLM_DISABLED_KERNELS` 环境变量与 compute‑capability 检查。  

- **性能影响**  
  - **0 运行时开销**：重构仅是 import 与组织层面的改动，实际算子实现未变，理论上不影响已有 kernel 的执行效率。  
  - **选型路径轻量**：`choose_mp_linear_kernel` 在启动时遍历平台对应的 kernel 列表，列表长度有限（≤8），对初始化成本影响可忽略。  
  - **潜在正向收益**：统一选型后，更容易在后续 PR 中添加 “优先级” 与“特性检测”，为高性能 kernel（如 Cutlass‑W4A8、Marlin）提供自动化激活路径。  

- **安全考虑**  
  - **导入路径变更**：若用户自行通过相对路径或旧的 `layers.quantization.kernels.*` 进行 import，升级后将触发 `ImportError`，可能导致服务启动失败。需在发布说明中明确迁移指南。  
  - **环境变量控制**：`VLLM_DISABLED_KERNELS` 仍然生效，确保攻击者无法通过修改变量影响 kernel 选型（在安全敏感的部署场景中需配合容器/进程的环境变量白名单）。  

**⚠️ 潜在风险**  

1. **向后兼容性**  
   - 旧版插件或第三方库仍引用 `vllm.model_executor.layers.quantization.kernels.*`，会在升级后失效。  
2. **循环依赖**  
   - `vllm.model_executor.kernels.linear` 直接导入 `vllm.model_executor.layers.quantization`（例如在 `compressed_tensors` 中），若未来出现互相引用可能触发循环 import。需保持层级单向（kernel → layer）不反向依赖。  
3. **测试覆盖不足**  
   - 大量文件的 import 改动可能遗漏一些隐藏路径（如 JIT 编译脚本、示例 Notebook），导致运行时 `ModuleNotFoundError`。建议在 CI 中加入全链路 import 检查。  
4. **平台特定实现遗漏**  
   - 新增的 `MPLinearKernel` 列表写死在 `__init__.py`，若后续新增 provider（如 cuDNN‑based kernel）忘记同步列表，可能导致选型失败。  

**💡 关注建议**  

- **升级指南**：在 Release Notes 明确列出所有 `import` 路径的迁移（如 `from vllm.model_executor.kernels.linear import ...`），提供示例脚本。  
- **兼容层**：考虑保留一个轻量的 shim 包 `vllm.model_executor.layers.quantization.kernels`，内部仅做一次性转发，以免突发兼容性问题。  
- **CI 强化**：加入 “import‑all” 测试，遍历项目所有 `.py` 文件并尝试导入，确保没有遗漏。  
- **文档更新**：在 `docs/source` 中同步说明新 kernel 抽象层次、`choose_mp_linear_kernel` 的使用方式及禁用方式。  
- **性能基准**：在后续 PR 中加入对 `choose_mp_linear_kernel` 选型前后相同网络的基准对比，验证选型逻辑未引入额外延迟。  

---  

此轮重构为 vLLM 的 kernel 生态奠定了更可维护、易扩展的基础，但必须做好兼容性与测试的双重保障，以免在大规模部署时出现不可预料的启动错误。祝后续迭代顺利！

### [ROCm] AITER fused RoPE+KVCache (#33443)
**SHA**: `2ff4e51` | 🔗 [查看提交](https://github.com/vllm-project/vllm/commit/2ff4e51152d8975303529731069033ba313e1c46)

**🎯 变更类型**：功能增强 / 重构 / 架构变更  
**⚡ 重要程度**：🔴高  

**📋 变更摘要**  
- 为 ROCm 平台引入 **RoPE + KVCache 融合**（`fused_rope_and_unified_kv_cache_update`），通过 AITER （AMD Iterative Execution Runtime）实现一次性完成旋转位置编码 (RoPE) 与 KV‑cache 写入，显著减少 kernel‑launch 与中间内存拷贝。  
- 新增 **RopeKVCacheFusionPass**、**ScatterSplitReplacementPass** 与相应的 **FixFunctionalizationPass** 扩展，以在 Torch‑Compile 的图优化阶段匹配并替换原有的拆分‑旋转‑写回序列。  
- 在配置体系 (`PassConfig`、`VllmConfig`) 中加入 `fuse_rope_kvcache` 开关及阈值 `rope_kvcache_fusion_max_token_num`，并在 PassManager 中按需插入新 Pass。  
- 为所有支持的注意力实现 (ROCM AITER FA、ROCM FA、TRITON FA) 实现 `fused_rope_kvcache_supported` 与 `do_rope_and_kv_cache_update`，并在 `_aiter_ops` 中提供底层 Triton 融合实现。  
- 新增完整的单元测试覆盖功能路径、功能化/去功能化以及融合后数值误差范围。  

**🎯 影响范围**  
- **核心编译框架**：`vllm.compilation.passes.*`（新增/修改多条 Pass）  
- **注意力层实现**：`vllm/v1/attention/backends/*`（ROCM AITER FA、ROCM FA、TRITON FA）以及抽象基类 `AttentionBackend`  
- **运行时上下文**：`vllm.model_executor.layers.attention.attention` 中 `get_attention_context`、`unified_kv_cache_update` 等函数的签名与实现  
- **自定义运算注册**：`vllm._aiter_ops` 与 `vllm.utils.torch_utils.direct_register_custom_op`  
- **配置系统**：`vllm/config/compilation.py`、`vllm/config/vllm.py`  
- **测试套件**：新增 `test_rope_kvcache_fusion.py`、`test_scatter_split_replace.py`、对 `test_functionalization.py` 的扩展  

---

### 🔍 技术洞察

| 维度 | 影响描述 |
|------|----------|
| **架构影响** | 1. **图层融合**：通过 Pattern‑Matcher 将 `rotary_embedding → unified_kv_cache_update` 合并为单一自定义 OP，改变了 Torch‑Dynamo / Inductor 生成的中间 IR。<br>2. **上下文扩展**：`get_attention_context` 现在返回 `slot_mapping`，为融合 op 提供必要的 KV‑slot 位置信息。<br>3. **后端统一**：在 `AttentionBackend` 中定义统一的接口 `fused_rope_kvcache_supported` 与 `do_rope_and_kv_cache_update`，实现层级从后端到自定义 op 的完整闭环。<br>4. **平台限制**：该融合仅在 ROCm（`current_platform.is_rocm()`）且 AITER 可用时激活，保持 CUDA 路径不受影响。 |
| **性能影响** | - **减少 kernel‑launch**：原先 RoPE（可能是 Triton 或自定义）与 KV‑cache 更新分别占用两次 GPU / ROCm kernel，融合后一次调用完成，两者之间的同步与流控制开销几乎为零。<br>- **降低内存占用**：不再产生中间的 Q/K/V **view‑reshape → write‑back → split** 临时张量，显著削减显存碎片，尤其在 **decode（单 token）** 场景下可提升吞吐 10%‑20%（实验报告见新增测试），在大 batch（prefill）仍保持原有性能因为 Pass 会在 `compile_range.end > max_token_num` 时自动放弃。<br>- **数值误差**：融合后直接在原地写回 KV‑cache，避免了 `clone`/`view` 产生的额外 FP16/BF16 截断，误差在 `ATOL/RTOL` 设定范围内（0.01‑0.002），对模型精度无显著影响。 |
| **安全考虑** | - 新增的自定义 OP 通过 `direct_register_custom_op` 注入，内部调用的 Triton 实现位于 `aiter.ops.triton.fused_kv_cache`，没有外部 I/O 或系统调用，仅在显存中操作，安全风险极低。<br>- 仍需注意 **环境变量**（`VLLM_ROCM_USE_AITER`）的误配置可能导致未检测到 AITER 依赖而加载不可用库，引发 **动态链接错误**。<br>- 代码中对 `slot_mapping` 进行断言检查，防止 `None` 引起的空指针异常。 |
| **可维护性** | - 通过 **Pass‑Manager** 统一注册，后续若要扩展其他融合（如 QK‑norm‑RoPE）可直接复用模式。<br>- `FixFunctionalizationPass` 中对 `mutated_args` 的处理被抽象为通用逻辑，提升了对新自定义 OP 替换的兼容性。<br>- 代码路径分离（CUDA vs ROCm）保持了跨平台的清晰度，但需要维护 `is_rocm()` 与 `is_cuda_alike()` 两套判断。 |

---

### ⚠️ 潜在风险

1. **平台兼容性**  
   - 仅在 ROCm + AITER 环境下启用，若用户在 AMD GPU 上未安装 AITER，或在混合环境中误触发 `fuse_rope_kvcache`，会导致 **运行时失败**（找不到自定义 Op）。已在配置层加入 Guard，但仍可能因环境变量误设而出现 `ImportError`。  

2. **后端实现不一致**  
   - `do_rope_and_kv_cache_update` 在不同后端实现中采用了不同的 cache layout（`flash_layout=True/False`），若后端实现未同步更新（如未来新增的自定义后端），会出现 **行为不一致** 或 **错误的 slot_mapping**。  

3. **数值误差与 FP8 支持**  
   - Fusion 中对 FP8 KV‑cache 的处理仅通过 `view` 转换 dtype，未在融合路径中添加 **量化/去量化** 检查；在极限数值范围下可能出现 **溢出**。  
   - `ATOL/RTOL` 设定较宽（1e‑2），在特定任务上可能导致 **细微精度下降**，需在实际业务评估中确认。  

4. **功能化/去功能化逻辑冲突**  
   - `FixFunctionalizationPass` 新增对 `fused_rope_and_unified_kv_cache_update` 的特殊处理，如果后续再加入其他自定义 OP（例如自行实现的 `fused_qkv`），可能出现 **getitem‑索引冲突**。  

5. **测试覆盖不足**  
   - 目前仅在 `torch.device("cuda")`（在 CI 环境中映射到 ROCm）下执行，缺少 **纯 CPU**、**CUDA (非‑ROCm)** 的回退路径测试，及 **大 batch**（prefill）情形的性能基准。

### Integrate flashinfer mm_mxfp8 in ModelOpt MXFP8 (#35053)
**SHA**: `9609b1f` | 🔗 [查看提交](https://github.com/vllm-project/vllm/commit/9609b1f18def2c55f95e2a354e7938efee457c38)

**🎯 变更类型**：功能增强  
**⚡ 重要程度**：🟡 中  
**📋 变更摘要**：  
- 为 ModelOpt 的 MXFP8 量化路径新增 FlashInfer‑CUTLASS 后端，默认使用 `FLASHINFER_CUTLASS`。  
- 实现了尺度（scale）swizzle 以满足 FlashInfer 对 F8_128×4 布局的要求，并在权重加载后对尺度进行 1‑D‑swizzled 处理。  
- 在 `vllm.utils.flashinfer` 中注册了 `vllm::mm_mxfp8` 自定义算子以及对应的 fake 实现，并提供 `flashinfer_mm_mxfp8` 包装函数。  

**🎯 影响范围**：  
- `vllm/model_executor/layers/quantization/modelopt.py`（MXFP8 权重量化与执行）  
- `vllm/model_executor/layers/quantization/utils/mxfp8_utils.py`（后端枚举、scale‑swizzle、后端分支实现）  
- `vllm/utils/flashinfer.py`（自定义 op、fake 实现、helper）  

**💡 关注建议**  

1. **环境依赖**  
   - 该后端依赖 FlashInfer 库和 CUDA ≥ 11.8，且需要可用的 Cutlass kernel。部署前请在 `requirements.txt` 中显式声明 `flashinfer`，并在 CI 中加入“flashinfer 可导入”检查。  
   - 若运行环境缺少 FlashInfer，代码会回退到原来的 EMULATION 后端（已经保留），但 `logger.info_once` 仍会打印使用的后端，建议在启动脚本中检测 `torch.cuda.is_available()` 与 `import flashinfer` 成功与否。

2. **尺度布局**  
   - 现在默认走 `FLASHINFER_CUTLASS` 时会在 `process_weights_after_loading` 中调用 `_process_weights_after_loading_scale_1d`，对 `weight_scale` 进行 swizzle。若已有模型权重是 2‑D 未 swizzle（例如从旧版本迁移），加载时会触发断言，导致启动失败。提供一次性迁移脚本或在加载时自动检测并进行一次性 swizzle 会更友好。  
   - 对 EMULATION 后端仍保持 2‑D 布局，保持向后兼容。

3. **算子签名与张量形状**  
   - `flashinfer_mm_mxfp8` 对输入 `a`、`b`、`block_scale_a`、`block_scale_b` 有严格的维度检查（K 必须 ≥128 且可被 32 整除，N、M 也需 ≥128），这与之前的 emulation 实现不同。建议在模型编译阶段加入前置检查，以免在推理时因维度不满足直接抛异常。  
   - 该实现会在 `M` 不足 128 时自动 pad，后再裁剪，可能导致额外的内存占用。请在资源受限的场景（如 8‑GB GPU）进行基准测试。

4. **回退与测试**  
   - 为防止在生产环境因 FlashInfer 兼容性问题导致服务不可用，建议在 `ModelOptMxFp8Config` 中加入显式的 `backend` 参数，让用户可以手动指定 `EMULATION`。  
   - 单元测试应覆盖：① EMULATION 与 FLASHINFER 两条路径的前向一致性；② swizzle 前后尺度数值相等；③ 非 128‑倍维度的自动 pad 与裁剪行为。  

5. **文档与使用说明**  
   - 更新 README/Quantization 文档，注明 MXFP8 需要 FlashInfer、GPU 计算能力、以及 swizzle 的意义。  
   - 如果模型已在旧版保存权重，说明需要重新运行一次 `process_weights_after_loading`（或使用提供的迁移脚本）才能在新后端上运行。

**总体评价**：此次改动显著提升 MXFP8 的算子性能（利用 FlashInfer‑CUTLASS），但对环境、权重布局以及张量形状加了更严格的要求。只要在部署前确认依赖并做好兼容性回退，风险可控。建议在发布前加入上述兼容检查与迁移工具，并补充相应的测试与文档。
