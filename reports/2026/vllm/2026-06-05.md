### Add objectstore as a secondary tier to multi-tier kv cache offloading (#41968)
**SHA**: `6a89457` | 🔗 [查看提交](https://github.com/vllm-project/vllm/commit/6a894574bf7b78fe639cb0c5ddf3eed911ea9011)

**🎯 变更类型**：功能增强、架构变更、测试覆盖  
**⚡ 重要程度**：🔴高  
**📋 变更摘要**：  
为 vLLM KV‑Cache 多层 offload 引入了 “obj” 二级层，实现基于 S3 兼容对象存储的持久化。新增 `ObjectStoreSecondaryTierManager`、对应配置结构 `ObjStoreConfig`，在工厂 `SecondaryTierFactory` 中注册新层类型，并补充了完整的单元测试（使用内存 Mock 替代真实对象存储）。同时对错误日志做了轻微改进，使其在多层配置场景下更易定位。

**🎯 影响范围**：  
- `vllm/v1/kv_offload/tiering/*`（新 `obj` 包含 manager、config、初始化）  
- `vllm/v1/kv_offload/tiering/factory.py`（注册新 tier）  
- `vllm/v1/kv_offload/tiering/spec.py`（错误信息微调）  
- 新增测试 `tests/v1/kv_offload/tiering/test_obj_tier.py`  
- 任何使用 `SecondaryTierFactory.create_secondary_tier` 并传入 `"obj"` 类型的调用方  

---

### 🔍 技术洞察  

| 维度 | 影响描述 |
|------|----------|
| **架构影响** | 1. **新增二级存储层**：在原有 `DRAM → FS`（文件系统）路径之外，加入 `DRAM → OBJ`（对象存储）路径，使 KV‑Cache 可在三层之间弹性迁移。<br>2. **NIXL 集成**：通过 `nixl_agent` 与对象存储后端交互，实现异步的 `WRITE`/`READ`。<br>3. **FileMapper 复用**：统一使用 `FileMapper` 生成对象键名，保持与文件系统层的路径约定一致。<br>4. **工厂模式扩展**：`SecondaryTierFactory` 现在支持 `"obj"`，对外保持统一创建入口。 |
| **性能影响** | - **网络 I/O 进入关键路径**：写入/读取 KV‑Cache 块需经历网络往返（对象存储）以及 NIXL 的异步调度，潜在增加请求延迟。<br>- **并发度**：`ObjectStoreSecondaryTierManager` 默认使用 4 条 I/O 线程，可通过 `io_threads` 参数调优；对吞吐量有正向提升空间（可并行上传多个 block）。<br>- **内存占用**：在提交 transfer 前会在 DRAM 中保持 `primary_kv_view` 的完整映射，无额外拷贝；但每个 job 会创建 NIXL 描述符与临时句柄，轻微占用 Python 对象。<br>- **失败速率**：网络不稳定、对象存储访问错误会导致 transfer 直接标记失败，已在 `get_finished_jobs` 中统一收敛。 |
| **安全考虑** | - **凭证硬编码风险**：新增 `ObjStoreConfig` 包含 `access_key`、`secret_key`，若配置文件未加密或误提交到代码库，会泄露云资源凭证。<br>- **传输安全**：默认 `scheme` 为 `http`，在生产环境建议使用 `https` 并配置 `ca_bundle` 进行服务器证书校验。<br>- **异常捕获**：在 `_probe_connectivity`、`lookup`、`_exists` 中统一捕获异常并记录，避免凭证错误直接导致进程崩溃。 |
| **可维护性** | - **代码分层清晰**：对象存储实现放在独立 `obj` 包，便于后续切换底层 SDK（如 AWS SDK、MinIO）而不影响上层逻辑。<br>- **单元测试覆盖**：使用 `MockNixlAgent` 完全脱离真实对象存储，确保 core logic（state machine、错误路径）可在 CI 中快速验证。<br>- **错误日志改进**：在 `spec.py` 中将错误信息改为 “config index”，便于定位多 tier 配置错误。 |

---

### ⚠️ 潜在风险  

1. **网络/对象存储可用性**  
   - 对象存储不可达或延迟过高会导致 KV‑Cache 再次命中失效，进而影响整体推理吞吐。  
2. **凭证泄露**  
   - 配置文件若未走 secret 管理（如环境变量、Vault），可能在 CI/CD 或镜像中泄露。  
3. **资源泄漏**  
   - `shutdown` 已遍历 `_transfers` 释放句柄，但若在异常路径（如 `register_memory` 失败）未正确清理 `files_desc`，可能残留 NIXL 句柄。  
4. **并发冲突**  
   - `self._next_obj_dev_id` 在多线程/多进程环境下未加锁，若同一 `ObjectStoreSecondaryTierManager` 被多个进程共享（unlikely，但在分布式部署中可能出现），会产生 dev_id 冲突。  
5. **数据一致性**  
   - 当前实现没有对写入成功后做校验（如 MD5），若对象存储出现部分写入错误，后续读取可能返回损坏块。

| 对象 | 建议 |
|------|------|

| **运维/部署** | - 在部署脚本中加入对象存储连通性探测（已在 `_probe_connectivity`），并在探测失败时阻止服务启动。<br>- 监控 `ObjectStoreSecondaryTierManager.get_finished_jobs` 输出的 `JobResult.success`，及时告警 transfer 失败率升高。 |
| **测试** | - 增加对网络超时、HTTP 5xx、对象键冲突等异常场景的 mock 测试，以覆盖更加真实的失败路径。<br>- 考虑在 CI 中加入集成测试（使用 MinIO 本地模拟），验证真实 NIXL 与对象存储交互的端到端行为。 |
| **代码维护** | - 将 `ObjectStoreSecondaryTierManager` 中的硬编码常量（如 `"PROC"`、`"DONE"`）统一抽取为 `Enum`，提升可读性并降低拼写错误风险。<br>- 在 `lookup` 返回 `bool | None` 时，文档明确说明 `None` 语义（当前实际返回 `False`），或直接统一为 `bool`。 |

**结论**：此次提交为 vLLM 引入了重要的持久化层，使 KV‑Cache 可在对象存储上弹性扩展，提升了大模型部署在云环境下的可伸缩性和成本控制能力。但随之而来的网络、凭证和并发风险需在部署和代码层面适当加固，建议在正式发布前完成安全审计与性能基准测试。

### Remove KV cache scale boilerplate from model weight loading methods (#43167)
**SHA**: `62215e7` | 🔗 [查看提交](https://github.com/vllm-project/vllm/commit/62215e72c6ff4979009c27aa95f7449213007f8f)

**🎯 变更类型**：功能增强 / 重构  
**⚡ 重要程度**：🔴高  

**📋 变更摘要**  
- 将原先散落在各模型 `load_weights` 实现中的 KV‑cache 缩放参数加载逻辑抽象为统一的 **`WeightsMapper`** 与 **`QuantizationConfig.get_cache_scale_mapper()`** 接口。  
- 新增 `KVCacheScaleParameter` 统一管理 KV‑cache 缩放标量，提供专用 `weight_loader` 检查形状。  
- 大幅删减 30+ 个模型文件中的重复 KV‑cache 处理代码，统一在 `AutoWeightsLoader` 中完成映射与加载。  
- 相应单元测试简化，验证 `WeightsMapper` 与 `maybe_remap_kv_scale_name` 的行为。  

**🎯 影响范围**  
- **核心层**：`vllm/model_executor/layers/quantization/*`（fp8、kv_cache、quark）  
- **模型基类**：所有具体模型实现（`apertus`, `arcee`, `gemma*`, `llama*`, `qwen*`, 等约 40+ 文件）  
- **工具函数**：`vllm/model_executor/model_loader/weight_utils.py`, `vllm/model_executor/models/utils.py`  
- **测试**：`tests/model_executor/test_eagle_quantization.py`  

**🔍 技术洞察**  

- **架构影响**  
  - **去中心化 → 中心化**：KV‑cache 缩放的名称映射与加载从每个模型的 `load_weights` 中抽离，统一交由 `AutoWeightsLoader`（通过 `WeightsMapper`）处理，降低代码重复度。  
  - **接口扩展**：`QuantizationConfig` 新增 `get_cache_scale_mapper` 抽象，支持任意量化实现一次性声明映射规则，模型层无需感知细节。  
  - **参数统一**：`KVCacheScaleParameter` 为标量缩放提供统一构造与校验，避免手动 `torch.nn.Parameter(torch.tensor(-1.0))` 的错误写法。  

- **性能影响**  
  - **加载阶段**：引入 `WeightsMapper` 合并 (`__or__`) 以及一次性 `apply(weights)`，相较于原先在每个模型内部逐条判断的分支，整体遍历次数相同或更少，CPU‑side开销几乎可以忽略。  
  - **运行时**：KV‑cache 仍然使用标量 `torch.nn.Parameter`，未改变其推理路径，故推理性能保持不变。  

- **安全考虑**  
  - **数据完整性校验**：`KVCacheScaleParameter.weight_loader` 增加 `numel()==1` 检查，防止误加载多维张量导致数值溢出或未捕获异常。  
  - **向后兼容**：保留 `maybe_remap_kv_scale_name` 的旧逻辑，并在新 `maybe_remap_kv_scale_name` 中加入 “已在 `params_dict` 中则直接返回” 的保护，防止因 double‑remap 产生错误。  
  - **未发现引入新的网络或文件访问风险**。  

**⚠️ 潜在风险**  

1. **兼容性**  
   - 老的 checkpoint 可能仍使用 `kv_scale` 名称（已在 `maybe_remap_kv_scale_name` 中给出 **DEPRECATED** 警告），但如果用户自行跳过 `WeightsMapper`，这些旧名称可能不被映射，导致 KV‑cache 缩放缺失，进而影响量化模型的数值稳定性。  
2. **量化配置实现**  
   - 新增的 `get_cache_scale_mapper` 必须在自定义 `QuantizationConfig` 中实现返回 `WeightsMapper`；若返回 `None`，旧的 `get_cache_scale` 已被删除，可能导致 `AttributeError`。  
3. **错误的映射规则**  
   - `WeightsMapper` 使用后缀/前缀等简单字符串替换，若模型中出现同样后缀的非 KV‑cache 参数，可能被误映射。  
4. **测试覆盖缺失**  
   - 只保留了部分映射的单元测试，复杂模型（如 MoE、专家模型）未显式验证 KV‑scale 加载路径，需在 CI 中确保覆盖。  

**💡 关注建议**  

- **向后兼容**：在发布说明中提醒用户如果仍使用旧版 checkpoint，最好在加载前手动运行 `maybe_remap_kv_scale_name` 或保持 `WeightsMapper` 的默认行为。  
- **文档完善**：在 QuantizationConfig 文档中明确 `get_cache_scale_mapper` 的返回类型与示例，防止自定义实现遗漏。  
- **测试扩展**：为几类典型模型（MoE、混合专家、Multi‑TP）新增包含 KV‑cache 缩放的 checkpoint 加载测试，验证 `WeightsMapper` 与 `KVCacheScaleParameter` 的协同工作。  
- **监控警告**：利用现有 `logger.warning_once` 输出一次性 “DEPRECATED. Found kv_scale in the checkpoint.”，确保在生产环境中可快速定位未被映射的旧名称。  
- **代码审查**：检查所有自定义量化实现（如 `fp8`, `quark`, 未来可能加入的 `bitsandbytes`）是否已经迁移至 `get_cache_scale_mapper`，防止出现运行时 `AttributeError`。  

---  

总体而言，此次提交通过 **统一抽象 + 删除冗余代码** 大幅提升了代码可维护性和可扩展性，几乎不影响已有的加载性能或运行时行为。唯一需要关注的是 **向后兼容与自定义量化实现的完整性**，建议在后续版本中提供迁移指南并加大测试覆盖。

### ⚠️ 潜在风险  

1. **硬件/驱动兼容性**：`hipb_mm` 只在 **MI3xx** 系列 GPU（`on_mi3xx()`）上启用，若在不满足条件的硬件上误开启 `VLLM_ROCM_USE_AITER_LINEAR_HIPBMM`，会在运行时抛出 `hipblasLtMatmulAlgoGetHeuristic found 0 valid solutions`。已在测试中通过 `skip_if_no_hipb_mm_solution` 处理，但生产环境仍需确保 flag 与硬件对应。  
2. **启用在线调优的副作用**：在某些 ROCm 版本（尤其是早期 5.4‑5.6）`HIP_ONLINE_TUNING=1` 可能导致 **超时** 或 **内存泄漏**，需要在部署前对目标驱动版本做回归验证。  
3. **环境变量冲突**：`VLLM_ROCM_USE_AITER_LINEAR` 默认 `True`，而 `VLLM_ROCM_USE_AITER_LINEAR_HIPBMM` 默认 `False`。如果用户手动修改前者而忘记同步后者，可能导致 `is_linear_hipbmm_enabled` 为 `False`，出现意外的 fallback 行为。  
4. **权重布局假设**：`process_weights_after_loading` 假设原始权重已是 `[K, N]` 的 fp8 tensor。如果未来出现不同的权重布局（如从其他框架直接导入），可能产生维度不匹配的错误。  
5. **测试依赖 aiter**：单元测试在缺失 `aiter` 时会全部跳过，导致 CI 在非 ROCm 环境下无法覆盖这段代码，潜在回归风险。
