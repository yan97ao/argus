### [PERF] [Qwen3.5] Split mixed prefill+decode batches: route decodes to the recurrent kernel (#44700)
**SHA**: `fa27d4e` | 🔗 [查看提交](https://github.com/vllm-project/vllm/commit/fa27d4e9cf3c8d8a5a143f38c346b27c02b2c2e3)

**🎯 变更类型**：性能优化（拆分 mixed prefill+decode 批次，改走递归 kernel）  
**⚡ 重要程度**：🟡 中  

**📋 变更摘要**  
- 在 `QwenGatedDeltaNetAttention._forward_core` 中加入对 **mixed non‑spec**（即同时包含 1‑token decode 与长 prefills）的检测，将 decode 部分提前剥离，交由原有的递归更新 kernel（`fused_sigmoid_gating_delta_rule_update`）处理，而只对剩余的 prefill 部分走 chunk kernel。  
- 为此在 `GDNAttentionMetadata` 中新增 `prefill_query_start_loc`、`prefill_state_indices`、`prefill_has_initial_state` 三个字段，并在 builder 中根据是否存在 decode‑prefill 混合批次进行相应的 rebasing。  
- ROCm 侧的 fast‑path 参数名从 `fast_kernel` 改为 `use_aiter`，并统一调用 `_forward_core_decode_aiter`。  
- 新增集成测试 `test_gdn_forward_core_split.py`，比较 split 与 unified 两条路径在 conv‑state、ssm‑state 与输出上的数值一致性。  
- 为 Qwen3.5‑397B‑NVFP4‑MTP 添加了 GSM8K 评估配置。  

**🎯 影响范围**  
- `vllm/model_executor/layers/mamba/gdn/qwen_gdn_linear_attn.py`（核心 forward 逻辑）  
- `vllm/v1/attention/backends/gdn_attn.py`（metadata 构造）  
- 相关的 ROCm/AITER 调用签名  
- 测试目录 `tests/kernels/mamba/` 与 `tests/evals/gsm8k/configs/`  

**💡 关注建议**  

1. **兼容性检查**：  
   - 现有 `spec_sequence_masks` 为 `None` 时才会触发 split，确认在未来可能加入 spec‑mask 的场景（如混合 spec+non‑spec）不会误走该分支。  
   - 对 CPU / 老版 CUDA（<SM10x）保持原有路径不变，防止因 `prefill_query_start_loc` 为 `None` 触发 runtime 错误。  

2. **元数据一致性**：  
   - `prefill_*` 字段在 `GDNAttentionMetadata.build` 中的 rebasing 逻辑已假设 `num_decode_tokens` 与 `non_spec_query_start_loc` 同步，建议在 `build` 前加入断言或日志，防止因为未来改动导致不匹配。  

3. **测试覆盖**：  
   - 当前测试仅覆盖 CUDA SM10x 环境，建议在 CI 中添加对应的 ROCm/AITer 运行验证（即使用 `use_aiter=True` 的路径）。  
   - 考虑补充更大 batch、不同 `fresh_prefill` 场景以及 edge case（如 decode 为 0、prefill 为 0）以提升鲁棒性。  

4. **文档/参数**：  
   - 将 `use_aiter` 参数写入公开 API 文档，说明仅在 ROCm 环境下有效。  
   - 在 `QwenGatedDeltaNetAttention` 类的注释中补充 “mixed batch split” 的原理说明，便于后续维护。  

5. **性能监控**：  
   - 该改动在实际推理负载下的加速比例取决于 decode 与 prefill 的比例，建议在 benchmark 脚本中加入对比（split vs. unified）统计，以便用户根据实际使用场景评估收益。  

总体来看，改动实现了在非 spec‑batch 中将 decode 部分单独走递归 kernel，能够显著降低 chunk kernel 的调度开销，同时保持数值一致性。只要在上述兼容性与文档方面做好收尾，这一性能提升将对 Qwen‑3.5 系列的大模型推理产生积极影响。

### [Misc] Replaced asserts with proper exceptions to improve UX for pooling (#44593)

**🎯 变更类型**：功能增强 / 可用性改进  
**⚡ 重要程度**：🟡 中  
**📋 变更摘要**：将代码中大量 `assert` 检查改为显式抛出 `ValueError` / `RuntimeError`，在模型配置缺失、长度不匹配、或出现 partial‑prefill 等非法状态时提供可捕获且语义清晰的异常。相应单元测试也从 `AssertionError` 改为 `RuntimeError`。  

**🎯 影响范围**  
- `vllm/config/pooler.py`：`get_seq_pooling_type`、`get_tok_pooling_type` 的空值检查。  
- `vllm/model_executor/layers/pooler/*`：所有序列/token pooler 的长度校验、partial‑prefill 检查、以及 `pooler_for_classify`、`pooler_for_token_classify` 中对 `model_config.pooler_config` 的必填验证。  
- 测试文件 `tests/model_executor/layers/test_pooler_methods.py` 牵涉到异常类型的更新。  

**💡 关注建议**  
1. **异常语义统一**：目前使用 `ValueError`（参数缺失/不匹配）和 `RuntimeError`（partial‑prefill）两类异常，建议在文档或注释中说明何种错误对应何种异常，以免调用方捕获错误时产生混淆。  
2. **向后兼容**：若已有外部代码依赖 `assert` 抛 `AssertionError`，切换到 `RuntimeError` 可能导致失效。考虑在 `__init__` 时提供开关或在 `except AssertionError` 的旧代码中加入兼容层。  
3. **日志可观测性**：`assert` 在 release‑mode会被剔除，这次改为显式异常后，异常信息会在生产环境传播。建议在异常前加入 `logger.error`，便于排障。  
4. **测试覆盖**：确保所有新异常路径都有对应的单元测试，尤其是 `pooler_for_*` 中 `model_config.pooler_config` 为 `None` 的路径。  
5. **文档同步**：更新用户手册中关于 “pooler 配置必须显式提供” 与 “partial prefill 不支持”的说明，防止使用者在配置文件中遗漏导致运行时错误。  

整体来看，此次改动提升了库的可用性和错误可调试性，只要做好文档、兼容性和日志的配套工作即可安全发布。

5. **文档**：更新工具调用相关的用户手册，说明 `phi4_mini_json` 的使用方式、对应的模板文件（`tool_chat_template_phi4_mini.jinja`）以及参数键的两种可能名称 (`arguments` / `parameters`)。  

整体来看，此次改动为 Phi‑4 Mini 模型提供了完整的工具调用支持，保持与现有 JSON 解析器一致的行为模型，影响范围局限在解析器注册与导出层面，风险低。开发者在发布新版本前，请验证 CLI 参数文档以及跨语言（Python）对应的 `--tool-call-parser` 选项已同步。

- 新增 `RopeStaticQQuantKVCachePattern`，在原有 RoPE‑KVCache 合并模式基础上，匹配 “RoPE → static‑scaled‑fp8‑quant → KV‑cache update” 的子图并替换为一次性 fused‑op，同时保留 KV‑cache 的依赖顺序。  
- 在 `RopeKVCacheFusionPass` 初始化时根据平台是否提供 `static_scaled_fp8_quant` 与 FP8 dtype 动态决定是否注册该新模式。  
- 测试层面：  
  - `test_rope_kvcache_fusion_default_keeps_large_ranges_unfused` 验证默认范围规则不受影响。  
  - `test_rope_static_qquant_kvcache_fusion` 对 ROCm AITer、TRITON‑ROPE、不同 KV‑cache dtype（auto / fp8）以及 NEOX 与非‑NEOX 场景进行端到端编译、前向和数值对比，确保融合后结果与未融合保持在合理误差范围内。  

**🎯 影响范围**  
- **核心模块**：`vllm/compilation/passes/fusion/rope_kvcache_fusion.py`（新增 matcher 与辅助函数），`vllm/compilation/passes/fusion/rope_kvcache_fusion_test.py`（大量参数化测试），以及相关配置/工具 (`vllm/config/utils.py`、`vllm/_aiter_ops.py`)。  
- **运行时**：仅在 ROCm 环境且 `static_scaled_fp8_quant` 可用时激活；其它平台仍使用原有 RoPE‑KVCache 融合逻辑，不受影响。  

**💡 关注建议**  
1. **平台检测**：确认 `current_platform.fp8_dtype()` 与 `torch.ops._C.static_scaled_fp8_quant` 在用户机器上均可正常 import，避免在不支持的 GPU 上出现 `None` 导致编译失败。  
2. **兼容性回退**：如果上述检查返回 `None`，fusion_pass 会自动不注册新模式；但仍要在 CI 中跑一次 “unsupported” 场景，确保不出现未捕获的异常。  
3. **数值误差**：FP8 量化本身会引入误差，测试已给出 `ATOL/RTOL` 阈值。若业务对数值精度更敏感，建议在配置中显式关闭 `fuse_rope_kvcache` 或在 `kv_cache_dtype` 设为 `"auto"`。  
4. **性能评估**：融合后理论上可以省一次独立 FP8 量化及一次 KV‑cache 插入的显存搬移，建议在实际推理基准（吞吐/延迟）上验证收益，尤其是在大 batch / 长序列场景。  
5. **文档/使用说明**：更新 README 或 API 文档，明确该特性仅在 ROCm + AITER 环境且编译模式为 `VLLM_COMPILE` 时可用，且依赖 `VLLM_ROCM_USE_AITER` 与 `VLLM_ROCM_USE_AITER_TRITON_ROPE` 环境变量。  

总体来看，此次改动为 ROCm‑FP8 场景提供了关键的算子融合，提升了性能潜力且对主线代码影响最小，建议在 ROCm CI 中加入持续回归测试。
