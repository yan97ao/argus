### [ROCm][Engine] Fix GPU memory leaks in engine shutdown and test workaround for async KV prefix cache reset (#38503)
**SHA**: `95995bb` | 🔗 [查看提交](https://github.com/vllm-project/vllm/commit/95995bbef81292e3ee1ef0df5ca3989bb481bdd5)

**变更类型**：Bug 修复 / 性能优化  
**重要程度**：🟡 中  

**核心改动**  
1. **Dockerfile**：更新 RIXL/UCX 代码分支，新增 `HSA_ENABLE_IPC_MODE_LEGACY=1` 环境变量以规避 UCX rocm_ipc 的 GPU 内存 pin 问题。  
2. **内存泄漏修复**：在 `engine/core.py`、`worker/gpu_model_runner.py`、`worker/gpu_worker.py` 中加入 `gc.unfreeze()` 与 `model_runner.shutdown()`，在进程内关闭时显式释放模型权重、KV 缓存、工作空间等 GPU 张量。  
3. **utils/func_utils.py**：将 `supports_kw` 的实际实现抽离为内部 `_supports_kw` 并加 LRU 缓存，防止对绑定方法的缓存导致对象（及其 GPU 张量）被永久引用。  
4. **测试**：新增 RIXL GPU 内存诊断测试以及对 `test_shutdown` 的容忍度微调；修改前缀缓存重置测试，使其在等待期间触发一次 dummy 生成，确保异步 offload 进度被调度器拉取。  

**影响范围**  
- ROCm 环境下的容器镜像构建与运行。  
- 引擎、GPU worker、模型 runner 的 shutdown 逻辑。  
- `supports_kw` 在任何需要动态检查关键字参数的调用处。  

**关注建议**  
- 确认 `gc.unfreeze()` 仅在 Python ≥3.12（或已移植的实现）可用，必要时加版本检测或回退。  
- `model_runner.shutdown()` 需保持幂等，防止二次调用触发异常。  
- 在 CI 中加入 ROCm 运行器，确保新测试不因硬件缺失被误报。  
- 文档补充 “Legacy IPC mode” 说明及内存泄漏的已知触发条件，帮助用户在自定义镜像时不遗漏该环境变量。  

整体来看，此次提交针对 ROCm 上 UCX 产生的 GPU 内存永久 pin 问题提供了多层防护，并通过单元测试验证，风险可控。

### [Feature] Warm up readonly multimodal processor during renderer startup (#40797)
**SHA**: `07351e0` | 🔗 [查看提交](https://github.com/vllm-project/vllm/commit/07351e0883470724dd5a7e9730ed10e01fc99d08)

**🎯 变更类型**：功能增强（新增只读多模态处理器的预热并细化缓存控制）  
**⚡ 重要程度**：🟡 中  

**📋 变更摘要**  
1. 为 OpenAI‑Chat、Completion、Tokenize、Disaggregated‑Serve 等入口添加了 *skip_mm_cache* 参数，使渲染路径在 **“仅渲染器”** 场景下能够跳过多模态缓存，而在 **“引擎执行”** 场景下保持缓存。  
2. 在 `BaseRenderer` 中抽取了 `_warmup_mm_processor`、`_clear_processor_cache` 与 `clear_mm_cache`，并在 `warmup` 流程中分别对 **可写** 与 **只读** 多模态处理器进行预热、日志记录和缓存清理。  
3. 新增 `BaseRenderer._readonly_mm_processor` 的使用路径，并在 `warmup` 完成后通过 `_clear_processor_cache` 清理只读处理器缓存。  
4. 大量单元测试覆盖：  
   - `skip_mm_cache` 在不同调用链（engine‑side vs renderer‑only）下的取值。  
   - 只读多模态处理器的预热是否被调用以及缓存是否被清除。  
   - `serve_tokens`、`tokenize`、`render_chat/completion` 等入口的行为。  

**🎯 影响范围**  
- **渲染层**：`vllm/entrypoints/serve/render/serving.py`（新增 `skip_mm_cache` 参数并向下传递），`vllm/renderers/base.py`（预热与缓存清理抽取）。  
- **多模态处理**：只读处理器的初始化与清理路径。  
- **测试**：新增/修改 8+ 个测试文件，覆盖新特性。  
- **调用方**：内部调用 `render_chat` / `render_completion` 的代码现在必须接受可选关键字 `skip_mm_cache`（已有默认值，向后兼容）。  

**💡 关注建议**  

| 方向 | 建议 |
|------|------|
| **兼容性** | 确认所有自定义渲染器或第三方插件在调用 `render_chat/completion` 时未显式传入位置参数，避免因新关键字导致 `TypeError`。如有，需要改为关键字调用或保持默认。 |
| **只读处理器** | `BaseRenderer._readonly_mm_processor` 现在在 `warmup` 时会被访问。请确保在创建 Renderer 时已正确设定该属性（或保持为 `None`），否则会出现属性未定义的潜在错误。 |
| **缓存一致性** | `clear_mm_cache` 仍然在渲染结束后统一调用，但只读处理器使用 `_clear_processor_cache`。若未来加入异步并发的预热或缓存访问，请审查是否需要对只读缓存也做异步序列化，以防 race。 |
| **性能** | 预热过程已抽取为 `_warmup_mm_processor`，但仍会在每次 `Renderer.warmup` 时执行两次（写 + 只读）。在大模型启动时请监控额外的初始化耗时，尤其在只读处理器资源占用较高时。 |
| **测试** | 新增的测试使用了 `MagicMock`、`AsyncMock`，请在 CI 环境确保 `pytest-asyncio` 已安装，并且所有 mock 的属性（如 `mm_processor.info.allowed_mm_limits`）与真实实现保持一致，以防假阳性。 |
| **日志与可观测性** | 预热完成后会打印 `"Multi-modal warmup completed"` 与 `"Readonly multi-modal warmup completed"`，建议在生产部署中监控这些日志，确认两次预热均成功。 |
| **文档** | 在渲染器和 OpenAI‑compatible 接口的文档中加入 `skip_mm_cache` 参数说明，解释何时使用 **engine‑side**（缓存保留）与 **renderer‑only**（缓存跳过）路径。 |

---  
**总体评价**：本次 PR 在保证已有功能不受影响的前提下，提供了更细粒度的多模态缓存管理，并为只读多模态处理器添加了安全的预热与缓存清理路径，提升了首次请求的响应一致性。实现方式清晰、单元测试覆盖全面。后续关注属性初始化、并发安全以及文档同步即可顺利上线。

### [EPLB] Fix replica selection bias in fused_moe router (#40810)
**SHA**: `333529d` | 🔗 [查看提交](https://github.com/vllm-project/vllm/commit/333529deae59cd4100df540f225470c9bc539bee)

**🎯 变更类型**：Bug 修复（修正 fused‑moe 路由器的副本选择偏差）  

**⚡ 重要程度**：🟡 中  

**📋 变更摘要**  
1. 在 `router/base_router.py` 中，原先使用 `offs % replica_count` 直接取余的方式会在 `top_k` 为 `R` 的整数倍时把同一逻辑专家的所有 token 固定到同一副本，导致负载不均。  
2. 新增基于 Knuth 乘数的 32‑bit 乘法哈希 `hashed = (token_idx * 2654435769) & 0xffffffff` ，再对 `replica_count` 取模，实现伪随机的副本选择。  
3. 为哈希计算传入 `num_active_experts`（即 `top_k`），并在 Triton‑kernel 与 Python 包装层同步修改签名。  
4. 补充单元测试 `test_eplb_map_hot_expert_replica_balance` 验证在 `top_k` 为 `R` 的倍数时热专家副本负载均衡，并更新原有测试的注释与预期输出。  

**🎯 影响范围**  
- `vllm/model_executor/layers/fused_moe/router/base_router.py`（核心路由 Triton kernel）。  
- 直接调用 `eplb_map_to_physical_and_record` 的所有位置（目前仅在 MOE 相关层）。  
- 测试目录 `tests/kernels/moe/test_routing.py`。  

**💡 关注建议**  
1. **接口兼容**：`_eplb_map_and_record_i32_kernel` 与 `_eplb_map_and_record_triton` 新增 `num_active_experts` 参数，确认项目其他模块（如自定义编译路径）已同步更新，否则会出现调用签名不匹配。  
2. **性能评估**：哈希计算引入一次 64‑bit 乘法和位掩码，理论开销极小，但建议在大模型/高并发场景下跑一次基准，确保不出现意外回退。  
3. **数值安全**：`token_idx = (offs // num_active_experts).to(tl.int64)` 假设 `offs` 为 `int32`，除法结果仍在 32‑bit 范围；若 `num_active_experts` 为 0（异常情况）会触发除零错误，建议在调用前做显式断言。  
4. **文档/注释**：在函数头部加入简要说明，解释 Knuth 哈希的来源与目的，便于后续维护者了解为何不再使用直接取余。  
5. **回归测试**：确保 CI 仍运行所有 MoE 相关的性能/准确性基准，防止新哈希在极端负载（如极高 `top_k`）下产生不均衡的副本分配。  

总体来看，此次修改有效消除了副本选择的系统性偏差，提升了热专家的负载均衡，风险主要在签名兼容和极端输入的鲁棒性，需要在上线前完成上述检查。
