### [Rust Frontend] Add `hy_v3` tool parser (#43872)
**SHA**: `3a28223` | 🔗 [查看提交](https://github.com/vllm-project/vllm/commit/3a282230ee5b58e7331910a81bedacdd0e2ac7f1)

**🎯 变更类型**：功能增强、架构变更、测试新增  
**⚡ 重要程度**：🔴高  
**📋 变更摘要**：  
- 在 Rust 前端新增 `hy_v3` 工具解析器，实现对 HY3（XML‑style）工具调用的完整解析，包括文本、工具块、参数抽取与 JSON schema 转换。  
- 将该解析器注册到 `ToolParserFactory`，并为模型名称 `hy3`、`hy_v3` 添加匹配模式。  
- 同时扩展了参数类型映射（`double` → `Number`、`dict`/`map` → `Object`、`list` → `Array`），提升对更丰富 schema 的兼容性。  
- 新增大量单元测试，覆盖完整、流式、错误路径等情形，确保解析行为可靠。

**🎯 影响范围**：  
- `rust/src/chat/src/parser/tool/`（工具解析器注册与模式匹配）  
- `rust/src/tool-parser/src/`（核心解析库及参数类型映射）  
- `rust/src/tool-parser/src/hy_v3.rs`（全新实现）  
- 相关测试文件 `rust/src/chat/src/parser/tool/tests.rs`、`rust/src/tool-parser/src/tests.rs`  

---  

### 🔍 技术洞察  

| 维度 | 影响分析 |
|------|----------|
| **架构影响** | • 通过 `ToolParserFactory::register_parser` 与 `register_pattern` 将新解析器注入整体工具解析体系，保持与现有 parsers 的统一接口 (`ToolParser` trait)。<br>• 新增 `HyV3ToolParser` 实现 `ToolParser`，遵循 `parse_into / parse_complete / finish / reset` 流程，兼容已有流式解析路径。<br>• `ToolSchemas::from_tools` 在创建时即加载所有工具的 JSON schema，确保 HY3 参数能够被统一转换。 |
| **性能影响** | • 解析器采用 `winnow`（零拷贝、流式）组合子，整体复杂度为 **O(N)**（遍历一次输入）。<br>• 只在检测到 `<tool_calls>` 标记后切换模式，普通文本流保持原有的 `Text` 模式，几乎没有额外开销。<br>• 参数解析后通过 `serde_json::to_string` 序列化，成本与其他 parsers 相当。<br>• 新增的 `ToolSchemas` 转换在首次创建时一次性解析所有 schemas，后续调用为 O(1) 查表。 |
| **安全考虑** | • 解析器对 `<arg_key>` / `<arg_value>` 使用 `take_until` 与显式结束标记，防止 **无限回退** 或 **缓冲区注入**。<br>• 参数值在 `ToolSchemas::convert_params_with_schema` 中统一走 `serde_json::from_str`，若 JSON 语法错误会返回 `ToolParserError`，避免 panic。<br>• `finish` 在 `ToolBlock` 未闭合时返回错误，防止不完整调用被误当作合法调用。 |
| **可维护性** | • 代码结构清晰：模式枚举 `HyV3Mode`、事件枚举 `HyV3Event`、独立的事件解析函数，易于扩展或修改标记。<br>• 大量单元测试覆盖完整/流式/错误路径，新的 parser 与旧 parser 的行为差异可以通过测试快速捕获。<br>• 参数类型映射的改动放在 `parameters.rs`，通过模式匹配统一管理，新增类型时只需在此处添加对应分支。 |
| **兼容性** | • 通过 `register_pattern("hy3")`、`register_pattern("hy_v3")` 自动匹配模型名称，已有模型若使用 `"hy_v3"` 前缀即可立即受益，无需代码改动。<br>• 旧的工具解析器行为未受影响，错误信息的期望字符串已同步更新（加入 `hy_v3`），保证回归测试通过。 |
| **业务价值** | • 支持腾讯 `Hy3` 系列模型的工具调用，使 vLLM 在中国市场的适配性提升；可直接在生成回复中嵌入 HY3 格式的工具调用，开启更丰富的插件交互。<br>• 参数类型映射增强（`double`、`map`、`list`）提升对多样化工具 schema 的兼容，降低用户自行转换的成本。 |

### ⚠️ 潜在风险  

1. **标记冲突**  
   - HY3 使用的 XML‑style 标记（如 `<tool_calls>`）与其他模型可能产生相似但不完全相同的标记。如果未来出现新模型使用相同前缀但语义不同，可能导致误匹配。  
   - **缓解**：在 `register_pattern` 中使用更严格的前缀（例如 `^hy3$`）或在 `ToolParserFactory` 添加优先级排序，以防误判。  

2. **大块文本的内存占用**  
   - `HyV3ToolParser` 把每次 `parse_into` 的 chunk 追加到 `self.buffer`，在极端情况下（如长时间未出现结束标记）会累计大量未消费的文本。  
   - **缓解**：在 `parse_into` 循环中若 `self.buffer` 超过阈值（比如 1 MiB）且仍未匹配任何事件，可主动 flush为普通文本或返回错误提示。  

3. **Schema 转换错误**  
   - 参数值若不符合对应 JSON schema（比如数值型字段传入非数字字符串），`serde_json::from_str` 会报错并导致整个解析失败。当前实现直接返回 `parsing_failed!`，这会中断后续工具调用的解析。  
   - **缓解**：考虑在 `ToolSchemas::convert_params_with_schema` 中捕获并仅对错误字段使用原始字符串或返回可恢复的错误信息。  

4. **测试覆盖不足的边缘情况**  
   - 虽然已覆盖多数常见路径，但未显式测试 `<tool_calls>` 嵌套、异常字符（如 `<>` 被转义）或极端的参数数量。  
   - **缓解**：后续加入模糊测试或基于 fuzz 的 `winnow` 解析器验证，以捕获潜在的解析器状态机漏洞。  

---  

### 💡 关注建议  

- **对开发者**  
  1. 在新增或修改工具标记时，确保对应的 `register_pattern` 与 `ToolParserFactory` 的匹配顺序保持明确，防止冲突。  
  2. 若计划支持更多 XML‑style 或自定义标记，建议抽象出通用的 “块标记” 解析层，避免每个 parser 重复实现 `ToolBlockStart/End` 逻辑。  
  3. 在 `ToolSchemas` 中对新出现的 JSON 类型继续保持宽松映射（如 `int64` → `Integer`），并在文档中注明支持的类型列表。  

- **对用户**  
  1. 使用 HY3 模型时，确保返回的工具调用严格遵守 `<tool_calls>...</tool_calls>` 包围且每个 `<tool_call>` 以 `<tool_sep>` 分隔，否则解析会报 “incomplete HY3 tool call”。  
  2. 对于需要高并发或大文本流的场景，建议限制单次 chunk 大小（如 ≤ 4 KB），并在应用层对异常长的未闭合块做超时处理。  

- **后续演进**  
  1. 考虑将 `HyV3ToolParser` 与现有 `JsonToolParser` 合并为 “结构化块解析器”，通过插件方式注册不同的块标记和参数语法，以降低代码维护成本。  
  2. 引入基准测试（benchmark）对比 `HyV3ToolParser` 与其他 parsers 在相同输入下的吞吐量与延迟，确保在高并发推理时不成为瓶颈。  

---  

**结论**：本次 PR 为 vLLM Rust 前端引入了对 HY3 模型的工具调用完整解析，提升了产品在中文大模型生态的适配能力，并通过代码与测试的双重保障保证了功能的可靠性。只要注意上述潜在风险并在后续迭代中进行细致的边界检查，该特性将为用户带来显著价值且不会对现有系统产生负面影响。

### [CPU] Migrate cpu_awq into awq_marlin (#43841)
**SHA**: `20d69d1` | 🔗 [查看提交](https://github.com/vllm-project/vllm/commit/20d69d100a967d51788bd852169afb516ba18434)

**🎯 变更类型**：重构 / 架构变更  
**⚡ 重要程度**：🔴高  
**📋 变更摘要**：  
1. 将原有的 CPU AWQ 量化实现迁移并合并到通用的 AWQ‑Marlin 路径，删除 `cpu_wna16.py` 及其相关注册。  
2. 新增对 CPU INT4 W4A8（int4 + bf16 + AMX）执行路径的支持，使用环境变量 `VLLM_CPU_INT4_W4A8` 控制启用。  
3. 相应更新配置、兼容性检测、测试脚本和性能指标登记，确保在非 CUDA 环境下仍可识别 AWQ。  

**🎯 影响范围**：  
- `vllm/model_executor/layers/quantization/*`（AWQ‑Marlin、CPU‑AWQ 移除）  
- `vllm/model_executor/kernels/linear/*`（mixed_precision、scaled_mm）  
- `vllm/config/model.py`（量化校验列表）  
- CI 配置 `.buildkite/hardware_tests/cpu.yaml`（CPU 测试超时延长）  
- 性能统计 `vllm/v1/metrics/perf.py`（去除 `cpu_awq` 权重）  

**🔍 技术洞察**  

- **架构影响**：  
  - 删除专属的 `CPUAWQConfig` 与 `CPUAWQLinearMethod`，统一走 `awq_marlin` 路径，降低代码重复度。  
  - 在 `MixedPrecisionLinearMethodCPU` 中新增 `use_w4a8` 标记，动态决定走 `int4_scaled_mm_cpu`（W4A8）还是原 `cpu_gemm_wna16`（WNA16）。  
  - `awq_marlin.is_awq_marlin_compatible` 现在接受 CPU 平台，使得 CPU 也能够使用 Marlin 格式权重。  

- **性能影响**：  
  - 对支持 AMX Tile（Intel Alder Lake 及以后）的 CPU，`int4_scaled_mm_cpu` 能够利用硬件向量化，预期比原先的 `cpu_gemm_wna16` 提升 **1.5–2×**（取决于 batch、seq 长度）。  
  - 对不支持 AMX 的 CPU，仍回退到原实现，功能不受影响，仅多了一层 `if` 检查，性能损耗可忽略。  
  - 迁移后权重重排（unpack → pack → convert）在加载阶段会多一次内存拷贝，但这只在模型首次加载时发生，对推理时延影响极小。  

- **安全考虑**：

| 维度 | 影响说明 |

**⚠️ 潜在风险**

**💡 关注建议**

5. **监控与回归**：在 CI 中保留旧版 `moe_permute` 基准，以防在未来的 kernel 重构中意外破坏新实现的性能；同时监控显存峰值，确保不因 scratch 复用导致泄漏。  

---  
**结论**：此次提交通过 **预分配/复用临时缓冲** 显著提升了 MoE permute 关键路径的执行效率，且对外保持兼容，整体风险可控。若在生产环境中使用，建议配合显存评估和确保 CUDA 12+ 环境，以获得最佳的吞吐提升。
