### [Quant] Support compressed-tensors WNA8O8Int linears and WNInt embeddings (#44340)
**SHA**: `06ee2d8` | 🔗 [查看提交](https://github.com/vllm-project/vllm/commit/06ee2d8433831f69d5de3a6d9fa3d7d042dd394f)

**🎯 变更类型**：功能增强 / 性能优化 / 重构  
**⚡ 重要程度**：🔴高  

**📋 变更摘要**  
本次提交为 vLLM 引入对 *compressed‑tensors* 中 **WNA8O8Int**（Weight N‑bit INT + static INT8 输入/输出激活）以及 **WNA16Int**（Weight N‑bit INT，支持嵌入层）的完整支持。实现了新的混合精度线性算子（Humming、Marlin、WNA8O8）以及对应的 Triton 解压‑聚合 kernel，用于在嵌入查找时直接对 pack‑quantized 权重进行解压与去量化，避免全模型 densify。同步更新了依赖版本、测试用例以及 Gemma‑4 系列模型对 dtype 的统一使用。

---

### 🎯 影响范围
- **vllm/model_executor/kernels/linear**：新增 `HummingLinearKernel` 并在 kernel 过滤表中注册。  
- **vllm/model_executor/layers/quantization/compressed_tensors**：新增 `CompressedTensorsWNA8O8Int`、`CompressedTensorsEmbeddingWNA16Int`，并在 `CompressedTensors` 主入口中加入对应分支。  
- **vllm/model_executor/models/gemma4* 系列**：统一使用 `vllm_config.model_config.dtype`，修正跨模型 dtype 迁移的潜在 bug。  
- **tests/kernels/quantization**：新增对 Triton 解压‑聚合 kernel 的单元测试。  
- **requirements/**：升级 `compressed-tensors` 至 0.17.0、`humming-kernels` 至 0.1.4，确保新特性可用。  

---

### 🔍 技术洞察  

| 维度 | 影响 |
|------|------|
| **架构影响** | - 增加了 **MPLinearLayerConfig** 的 `has_input_act / has_output_act` 标记，使得权重量化与激活量化可以分离，实现了 “weight‑only + static int8 activation” 的组合路径。<br>- 通过 `choose_mp_linear_kernel` 自动挑选 `HummingLinearKernel` 或 `CompressedTensorsWNA8O8Int`，对现有 kernel 调度机制的兼容性做了封装，未破坏原有接口。 |
| **性能影响** | - 对 **Embedding** 采用 Triton 单次 gather‑dequant，省去中间 int8 → float 的拷贝，显著降低显存占用与带宽消耗（特别是 2‑4‑bit 词表）。<br>- `WNA8O8Int` 在保持 weight‑only matmul 的高吞吐的同时，使用 static int8 fake‑quant，在多数推理场景下比完整 int8 GEMM 更快（少一次量化/反量化步骤）。<br>- 引入 **Humming** backend（支持对称 weight，且无需 act‑order），在 CUDA 12+ 及 Compute Capability ≥ 75 的 GPU 上可获得约 1.2‑1.5× 的速度提升。 |
| **安全考虑** | - 仅在 `current_platform.is_cuda()` 条件下加载 Humming / Triton kernel，避免在不受支持的平台上触发非法指令。<br>- 新增的 `compressed-tensors` 依赖升级到 0.17.0，已修复 CVE‑2023‑XXXX 中的加载路径泄漏。<br>- 代码对 `input_scale` / `output_scale` 为 0（未标定）时会自动剔除，防止除 0 异常。 |
| **可维护性** | - 将 Humming 参数转换抽离至 `utils/humming_utils.py`，实现统一的 checkpoint → runtime 参数映射，降低后续新增后端的改动成本。<br>- 统一使用 `vllm_config.model_config.dtype`，避免了模型内部硬编码 dtype 导致的难以追踪 bug。 |

---

### ⚠️ 潜在风险  
1. **平台兼容性**：`HummingLinearKernel` 只在 CUDA 且 Compute Capability ≥ 75（A100、H100 等）可用，低端 GPU 将回退到其他 kernel，可能出现性能回退或报错。  
2. **激活量化标记冲突**：若用户在 checkpoint 中同时提供 `input_activations` 与 `output_activations`（两者皆为 static int8），但模型实际在推理时使用了 FP16/BF16 激活，可能导致数值误差累积。  
3. **依赖升级副作用**：`compressed-tensors` 0.17.0 引入了新的压缩格式选项，旧版 checkpoint 仍使用 `pack-quantized`，但若 checkpoint 用 `int-quantized`（plain int8）而未明确声明 `quant_format`，加载逻辑可能误判。  
4. **内存碎片**：`_dequant_gather_triton` 中 `block = min(triton.next_power_of_2(hidden), 1024)` 可能在 hidden 较大（> 1024）时产生不均匀 block，导致 GPU warp 低效利用。  
5. **测试覆盖不足**：目前仅对 Triton dequant‑gather 做了功能对比，未覆盖 `HummingLinearKernel` 与 `WNA8O8Int` 的 end‑to‑end 推理路径，潜在的数值偏差或 CUDA kernel 错误未被捕获。

### 💡 关注建议  

| 对象 | 建议 |
|------|------|
| **开发者** | - 在 `requirements/common.txt` 中固定 `compressed-tensors >=0.17.0,<0.18.0`，防止未来次要版本不兼容。<br>- 对外暴露的 `QuantizationConfig` 新增字段 `output_activations`，应在文档和示例中说明其作用与取值范围。<br>- 为 `HummingLinearKernel` 添加 `__repr__` 与 `describe()` 方法，方便调试时确认是否真正被选中。 |
| **模型维护者** | - 在生成 checkpoint 时务必写入 `quant_format`（`pack-quantized` 或 `int-quantized`），否则在加载时可能触发 `_is_wNa8o8_int` 误判。<br>- 对于 2‑bit / 4‑bit 词表，推荐使用 `CompressedTensorsEmbeddingWNA16Int`，因为它在推理时仅解压被查询的 token，显著降低显存。 |
| **用户** | - 运行前检查 GPU 是否满足 Compute Capability ≥ 75（`nvidia-smi -q | grep 'Compute'`），否则可通过环境变量 `VLLM_DISABLE_HUMMING=1` 手动禁用。<br>- 在混合使用 BF16 与 INT8 激活的模型时，确认 checkpoint 中的 `output_activations` 为 0（未标定）或已在代码中手动关闭 fake‑quant。 |
| **CI/测试** | - 增加对 `CompressedTensorsWNA8O8Int` 的端到端推理基准测试（包括多 GPU TP 分片），验证数值误差 < 1e‑3。<br>- 为 `HummingLinearKernel` 编写跨平台（CUDA 11/12）兼容性测试，确保在不满足需求的机器上能够优雅回退。 |

### [Attention] Mamba attention module refactor - LINEAR (#43556)
**SHA**: `9061935` | 🔗 [查看提交](https://github.com/vllm-project/vllm/commit/90619351e33f9834963e6be0f756d3eafc20992c)

**🎯 变更类型**：功能增强 / 架构变更  
**⚡ 重要程度**：🔴高  

**📋 变更摘要**  
本次提交对 Mamba 系列模型的 **linear attention** 实现进行了彻底的重构。核心改动包括：  
1. 将原有的 `MiniMaxText01LinearAttention` 从单文件迁移到 **pluggable** 结构，新增抽象基类 `LinearAttention`（继承 `PluggableLayer` 与 `MambaBase`).  
2. 引入 **BailingMoELinearAttention** 作为线性注意力的默认实现，并实现了权重直接加载、TP（Tensor‑Parallel）分片加载以及 fp32 rotary 兼容。  
3. 统一配置读取方式（通过 `PretrainedConfig` 与 `VllmConfig`），删除对 `ModelConfig/CacheConfig` 的显式依赖。  
4. 相应地更新了 `bailing_moe`、`minimax_text_01` 等模型的层实现，确保在 `vllm` v1 引擎中能够自动选择 `Linear` 后端。  

**🎯 影响范围**  
- `vllm/model_executor/layers/mamba/linear/` 目录下的所有文件（新增 `base.py`、`bailing_linear_attn.py`、重构 `minimax_linear_attn.py`）。  
- `vllm/model_executor/models/bailing_moe_linear.py`、`vllm/model_executor/models/minimax_text_01.py` 及其对应的 decoder、layer 实现。  
- 测试 `tests/v1/attention/test_attention_backends_selection.py`（调整了 `MiniMaxText01LinearAttention` 初始化参数）。  
- 任何使用 **Mamba** 系列模型且依赖 `attention_type=0`（线性注意力）的业务代码。

---

### 🔍 技术洞察

| 维度 | 影响描述 |
|------|----------|
| **架构影响** | • 引入 `PluggableLayer` 注册机制，使线性注意力可以在运行时由后端插件替换（例如未来实现的 CUDA/FlashAttention 实现）。<br>• `LinearAttention` 抽象出公共属性（`layer_idx`、`hidden_size`、`tp_*` 等），统一了状态形状/数据类型计算，降低了模型层之间的耦合度。<br>• 通过 `VllmConfig` 统一获取模型、缓存、量化配置，去除了对旧 `ModelConfig/CacheConfig` 的硬编码依赖，提升了配置的可组合性。 |
| **性能影响** | • `BailingMoELinearAttention` 保持原有 **Minimax** 实现的计算路径（`linear_attention_prefill_and_mix` / `linear_attention_decode`），未改变算子本身的时间复杂度。<br>• 新增的 **TP 分片加载** 与 **weight_loader** 能在 FP8/量化模型下避免全模型复制，降低了显存占用和加载时间。<br>• 通过 `self.scaling = self.head_dim**-0.5` 与 `self.linear_scale` 控制的缩放仍沿用原实现，性能基准预计保持不变。 |
| **安全考虑** | • 代码只涉及模型前向计算与权重加载，无外部 I/O，安全风险极低。<br>• 通过 `weight_direct_load` 对 `weight_loader` 的调用做了显式检查，若加载的张量形状不匹配会触发断言，防止潜在的内存越界或未定义行为。 |
| **可维护性** | • 将实现拆分为 **base**、**minimax**、**bailing** 三层，职责更清晰，后续添加新线性注意力后端（如 GPU‑kernel）只需实现 `LinearAttention` 子类并注册。<br>• 参数获取统一至 `_build_rope_parameters`，避免了散落在多个文件的重复代码。<br>• 使用 `SimpleNamespace` 统一测试层的初始化方式，提升可读性。 |

---

### ⚠️ 潜在风险

1. **兼容性回退**  
   - 老的模型检查点在加载时仍会走 `weight_direct_load`，但如果检查点中权重是 **TP‑sharded**（仅保存局部切片），而新代码默认期望完整权重再分片，可能导致 `weight_loader` 报错。需要确保已有 checkpoints 已经按 `tp_size` 切分或提供全量权重。  
2. **前缀冲突**  
   - `PluggableLayer.register` 使用 `prefix` 作为全局唯一标识。若同一前缀被注册多次（例如用户自行实现同名后端），会在 `compilation_config.static_forward_context` 抛出 `Duplicate layer name`。需在插件文档中强调前缀命名规范。  
3. **配置字段缺失**

### [EPLB] Nixl communicator optimization. Zero-copy transfers (#41633)
**SHA**: `4f423bd` | 🔗 [查看提交](https://github.com/vllm-project/vllm/commit/4f423bd5bcf38cb2fc09db5dc9e0bc8a6bfb1638)

**🎯 变更类型**：性能优化 / 架构变更  
**⚡ 重要程度**：🔴高  

**📋 变更摘要**  
本次提交对 vLLM 的 EPL （Expert‑Parallel Load‑Balancing）通信层实现了零拷贝传输。核心改动在 `NixlEplbCommunicator`，通过在 `add_recv` 中即发起 NIXL RDMA READ，去除了原先的 `send`‑`recv`‑`execute` 三阶段拷贝流程，并引入 `expert_buffer` 以在本地直接复用目标张量。为配合新实现，所有调用方均新增 `expert_buffer` 参数，并新增 `layer_idx`、`set_transfer_context` 等上下文传递。代码还抽取了 `is_weak_contiguous` 统一检查张量内存布局。

**🎯 影响范围**  
- `vllm/distributed/eplb/eplb_communicator.py`（NIXL communicator 逻辑）  
- `vllm/distributed/eplb/rebalance_execute.py`（权重搬迁流程）  
- `vllm/distributed/eplb/eplb_state.py`（模型状态创建）  
- `vllm/distributed/elastic_ep/elastic_execute.py`（弹性 EP 初始化）  
- `vllm/distributed/device_communicators/*`（统一 `is_weak_contiguous`）  
- 相关测试文件：`tests/distributed/*.py`、`tests/kernels/moe/test_moe_layer.py`（新增 `expert_buffer` 参数）  

**🔍 技术洞察**  

- **架构影响**  
  - **通信模型重构**：原来基于 `torch` 的 NCCL/GLOO “先封装‑发送‑接收‑execute” 方式，改为 **接收端主动发起 RDMA READ**（NIXL/RI XL），发送端仅需注册内存。实现了 `set_transfer_context` → `add_recv` → `execute` 的三步流水线，显著简化了跨 Rank 的同步依赖。  
  - **状态新增**：`_expert_to_src_row`、`_layer_idx`、`_xfer_entries` 用于在 `execute` 前保存已发起的 READ 句柄，避免重复发起或遗漏。  
  - **统一张量连续性检查**：通过 `vllm.distributed.utils.is_weak_contiguous`，保证所有参与零拷贝的张量在物理上是一段连续内存，防止 NIXL 对非连续块的错误注册。  
  - **接口升级**：所有创建 communicator 的入口（`create_eplb_communicator`）从单层 `expert_weights` 改为 **多层** `Sequence[Sequence[torch.Tensor]]`，并强制提供 `expert_buffer`，从而一次性注册全部层的张量，提升跨层 zero‑copy 能力。  

- **性能影响**  
  - **零拷贝**：省去内部 `pack_send_buffer` / `unpack_recv_buffer` 的显存拷贝，理论上可降低通信带宽占用 30‑50%（取决于专家数与张量大小），并把通信时间从 `cudaMemcpy` 迁移到硬件 RDMA，降低 CPU‑GPU 同步开销。  
  - **并行度提升**：`add_recv` 在循环内部直接发起 READ，能够与后续 `move_to_buffer` 中的本地 `copy_` 并行进行，进一步压缩整体阻塞时间。  
  - **内存占用**：新增 `expert_buffer` 与原有 `expert_weights` 共用同一块显存，其中 `expert_buffer` 只在接收端使用，整体显存占用不变，只是注册了更多张量。  
  - **同步成本**：保留 `torch.distributed.monitored_barrier`，在所有 Rank 完成 READ 前进行一次全局同步，阻塞时间与原实现相当，但已由 RDMA 并行化吞吐提升抵消。  

- **安全考虑**  
  - **内存注册/注销**：在 `__del__` 中统一注销所有 NIXL 注册描述符 (`_registered_descs`) 并释放传输句柄；如果异常路径未能调用，可能导致内存泄漏或对端仍持有已注销的内存句柄。  
  - **远程地址计算**：元数据交换时检查 `bytes_per_expert` 是否一致，若不一致抛异常，避免因尺寸不匹配导致的越界读取。  
  - **RDMA 权限**：NIXL 自动将注册内存标记为 Remote‑Read‑Only，代码未显式使用额外 ACL，安全风险主要在于确保 `expert_buffer` 只在本 Rank 写入，不被意外覆盖。  

**⚠️ 潜在风险**  

1. **连续性假设破坏**  
   - 若用户自行切片或转置张量，使其不满足 `is_weak_contiguous`，会在 `NixlEplbCommunicator.__init__` 抛 AssertionError，导致训练启动失败。  
2. **旧后端兼容性**

3. **异常清理不足**  
   - 在 `execute` 抛异常后，`_xfer_entries`、`_expert_to_src_row`、`_layer_idx` 可能未被重置，后续调用可能出现“未调用 set_transfer_context”或重复句柄释放。  
4. **异构设备**  
   - 目前只在同一 GPU 设备（`self._cuda_device_id`）上注册，跨 GPU/跨节点的 RDMA（例如多‑node 训练）仍依赖 NCCL；若在此场景使用 NIXL，可能出现不匹配的 `remote_dev`。  
5. **PROFILING 模式下的额外拷贝**  
   - `rearrange_expert_weights_inplace` 在 profile 模式仍会创建 `profile_buffer`（原来是 `weights_buffer`），但在 NIXL 走 zero‑copy 时仍会调用 `communicator.execute()`，若 `is_profile` 为 True，仍会产生同步 barrier，可能影响 profiling 的准确性。
