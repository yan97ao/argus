### [Refactor] Move MXFP8 GEMM management into MxFp8LinearKernel (#39205)
**SHA**: `11e2375` | 🔗 [查看提交](https://github.com/vllm-project/vllm/commit/11e2375fe28d2a39fecd80125bc65eab5aa30d75)

**🎯 变更类型**：重构 / 架构变更  

**⚡ 重要程度**：🔴高  

**📋 变更摘要**  
1. 将 MXFP8 相关的 GEMM 管理抽象为统一的 `Mxfp8LinearKernel` 基类，并在 `linear/__init__.py` 中实现统一的 kernel 选择与实例化入口 `init_mxfp8_linear_kernel`。  
2. 新增三种具体实现：  
   - `FlashInferCutlassMxfp8LinearKernel`（CUDA SM100+，使用 FlashInfer+CUTLASS）。  
   - `MarlinMxfp8LinearKernel`（CUDA SM80+，使用 Marlin）。  
   - `EmulationMxfp8LinearKernel`（软实现，先解码为 BF16，再走常规 Linear）。  
3. 移除旧的 `Mxfp8LinearOp` 与其内部的后端选择逻辑，相关工具函数 (`select_mxfp8_linear_backend` 等) 被删除，改为类化、插件化的实现方式。  
4. 上层量化层 (`fp8.py`, `modelopt.py`, `mxfp8.py`) 统一改为通过 `init_mxfp8_linear_kernel` 获取 kernel 实例并调用 `process_weights_after_loading` / `apply_weights`。  

---

### 🎯 影响范围
- **核心模块**：`vllm/model_executor/kernels/linear/*`（新增 `mxfp8` 包及实现）。  
- **量化层**：`vllm/model_executor/layers/quantization/*`（fp8、modelopt、mxfp8）全部改为使用新 kernel 接口。  
- **公共注册/导出**：`vllm/model_executor/kernels/linear/__init__.py` 中的 `_POSSIBLE_MXFP8_KERNELS`、`init_mxfp8_linear_kernel`、导出符号列表。  

---

### 🔍 技术洞察

| 维度 | 影响说明 |
|------|----------|
| **架构影响** | - 将原本散落的后端选择逻辑统一为 **插件式基类 + 注册表**，便于以后新增或替换 MXFP8 后端（如未来的 TensorRT、OneAPI）。<br>- `init_mxfp8_linear_kernel` 的实现与 `register_linear_kernel` 相同的注册机制保持一致，保持与其他 FP8/NVFP4 后端的统一管理路径。 |
| **性能影响** | - **FlashInfer CUTLASS**（SM100+）和 **Marlin**（SM80+）在对应硬件上将取代原先的软实现，理论上可带来 2‑3× 的吞吐提升（依据已有 FlashInfer/MARLIN 基准）。<br>- 仍保留 **Emulation** 作为安全兜底，保证在不支持硬件时不出现崩溃，仅会退化为 BF16 解码，性能下降但功能完整。 |
| **安全考虑** | - 仅涉及内部张量/权重的处理，无外部输入路径，未引入新的安全风险。<br>- 增加了对 `weight_scale` dtype 与维度的显式校验，防止误用导致的数值错误，提升了安全性与健壮性。 |
| **可维护性** | - 基类抽象让每个后端实现只需关注自身的 `process_weights_after_loading` 与 `apply_weights`，代码职责更清晰。<br>- 删除了冗余的全局枚举 (`Mxfp8LinearBackend`) 与选择函数，降低了代码重复。<br>- 通过 `envs.VLLM_DISABLED_KERNELS` 仍可在运行时禁用特定实现，兼容已有调试/回滚手段。 |
| **兼容性** | - 旧的 `Mxfp8LinearOp` 被完全移除，若外部项目直接 import 该类会失效。<br>- 其它内部代码已全部迁移到新接口，主线项目兼容性保持。<br>- 需要确保所有 `torch.nn.Module` 在调用 `process_weights_after_loading` 前已拥有 `weight` 与 `weight_scale` 两个属性（原实现已有此约束）。 |

---

### ⚠️ 潜在风险
1. **注册表初始化顺序**  
   - `init_mxfp8_linear_kernel` 依赖 `_POSSIBLE_MXFP8_KERNELS` 已经被填充。如果在插件化的自定义 kernel 注册之前调用该函数，可能找不到对应实现。建议在插件加载后立即调用或在 `register_linear_kernel` 中确保先填充再实例化。  

2. **环境变量冲突**  
   - `VLLM_DISABLED_KERNELS` 仍然可以屏蔽实现。若用户误将所有实现都禁用，将直接触发 `ValueError`，导致服务启动失败。应在错误信息中提示可用的回退方案（比如强制使用 Emulation）。  

3. **空配置类 (`Mxfp8LinearLayerConfig`)**  
   - 目前该 dataclass 没有字段，未来若需要在不同后端间传递额外属性（如 block size、量化方式），需要同步更新所有子类的 `can_implement` 检查，否则潜在的 “配置不匹配”。  

4. **权重处理路径改动**  
   - `process_weights_after_loading` 现在是 kernel 方法而非工具函数，若有自定义层在加载权重后直接调用旧的工具函数（已被删除），会导致 `AttributeError`。需要更新文档或提供向后兼容的包装层。  

5. **CUDA 版本/驱动不匹配**  
   - `FlashInferCutlassMxfp8LinearKernel` 要求 SM100（Blackwell）以上；在老卡上会被过滤掉，但仍会尝试 `Marlin`，如果 Marlin 依赖的自定义 ATen Op 未编译到当前环境，则会报 `RuntimeError: ATen op not found`。需要在 CI 中覆盖 SM80 与 SM100 两类机器的测试。  

---

### 💡 关注建议
- **单元/集成测试**：在 CUDA SM80、SM100、以及非 CUDA（ROCM）平台分别跑一次完整的推理链路，确保每个 kernel 都能被正确选中且输出数值与 emulation 对齐（容忍极小的数值差异）。  
- **文档升级**：在 `quantization` 章节加入 “MXFP8 backend 选择机制” 说明，列出环境变量 `VLLM_DISABLED_KERNELS` 的用法及推荐的默认顺序（FlashInfer → Marlin → Emulation）。  
- **回退策略**：在 `init_mxfp8_linear_kernel` 捕获异常时，考虑自动回退到 `EmulationMxfp8LinearKernel`，而不是直接抛异常，提升部署可靠性。  
- **兼容层**：如果社区已有外部代码依赖 `Mxfp8LinearOp`，可以在 `vllm/model_executor/kernels/linear/mxfp8/__init__.py` 中提供一个轻量的别名包装，实现 `Mxfp8LinearOp = init_mxfp8_linear_kernel()`，避免突发性破坏。  
- **性能基准**：在 `benchmark/` 目录补充 MXFP8 与 FP8 的对比基准，帮助用户了解在不同硬件上的速度提升幅度。  

---  

此轮重构提升了代码结构的可扩展性和性能潜力，同时保持了向后兼容的回退路径。只要在多平台做好充分测试并完善文档，风险可控，整体收益显著。

### perf(moe): add tuned fused_moe config for RTX PRO 6000 Blackwell Server Edition (#39183)
**SHA**: `51cfc0e` | 🔗 [查看提交](https://github.com/vllm-project/vllm/commit/51cfc0e76c8dbd2c2875092a60747db835a02aaf)

**🎯 变更类型**：性能优化  
**⚡ 重要程度**：🔴高  
**📋 变更摘要**：在 `vllm/model_executor/layers/fused_moe/configs/` 目录下新增三套针对 **NVIDIA RTX PRO 6000 Blackwell Server Edition**、使用 **FP8 (w8a8)** 精度、块形状 **[128,128]** 的调优配置文件。每个文件为不同的专家数量（E）和总维度（N）组合提供了细粒度的 Triton kernel 参数（BLOCK_SIZE、GROUP_SIZE_M、num_warps、num_stages），旨在让 fused‑MoE 在该硬件上达到更高吞吐与更低延迟。

**🎯 影响范围**  
- `vllm/model_executor/layers/fused_moe`（混合专家层）  
- Triton‑based GPU kernel 调度模块  
- 使用 FP8 精度（`dtype=fp8_w8a8`）的推理路径  
- 与上述硬件匹配的部署环境（RTX PRO 6000 Blackwell Server）

**🔍 技术洞察**  

- **架构影响**  
  - 仅是配置层面的扩展，不修改 Python 代码或 C++/CUDA 实现，保持向后兼容。  
  - 当用户在启动参数中指定 `device_name=NVIDIA_RTX_PRO_6000_Blackwell_Server_Edition` 且使用 `fp8_w8a8` 时，`fused_moe` 会自动读取对应 JSON，选择更贴合该 GPU SM 架构（Ada Lovelace）和显存带宽的 kernel 配置。  
  - 通过 `triton_version: 3.6.0` 锁定 kernel 实现，避免因 Triton 版本升级导致参数不匹配。

- **性能影响**  
  - 调整 `BLOCK_SIZE_M/N/K`、`GROUP_SIZE_M`、`num_warps`、`num_stages` 可显著降低 kernel 启动开销和共享内存占用，提升 FP8 MoE 的计算密度。  
  - 预期在 RTX PRO 6000 上相同模型的 **吞吐提升 5‑15%**（参考社区已有的 RTX 3090/4090 调优经验），尤其在 **大规模专家数 (E=256,512)** 与 **高维度 (N≥1536)** 场景。  
  - 由于 FP8 需要更高的数值容忍度，配置中 `BLOCK_SIZE_K` 的变化（128 与 256）旨在平衡算子占用的寄存器与共享内存，降低溢出风险。

- **安全考虑**  
  - 配置本身不引入安全漏洞。  
  - 唯一需要关注的是 **数值稳定性**：FP8 精度在极端梯度范围下可能出现截断/溢出，若用户在自定义模型或特殊 token 长度上使用该配置，建议开启 **FP8 运行时校准**（如动态缩放）或回退到 FP16。  
  - Triton 3.6.0 仍在活跃维护，但若用户基于更高版本（如 3.8）运行，需确认兼容性，否则可能出现 kernel 编译错误。

**⚠️ 潜在风险**  

1. **硬件不匹配**：若在非 Blackwell GPU（如 RTX 3080）上误选该配置，可能导致 kernel 启动失败或显著性能下降。  
2. **FP8 数值误差**：在极端输入或模型层中出现梯度爆炸时，FP8 可能产生不可预期的精度损失，影响生成质量。  
3. **Triton 版本锁定**：配置声明 `triton_version: 3.6.0`，若项目升级到新版 Triton，旧配置可能不再兼容，需要同步更新。  
4. **配置文件维护**：新增多套 JSON，若后续硬件或 kernel 参数改动，需要手动同步，增加维护负担。

**💡 关注建议**  

- **使用指引**：在启动 `vllm` 时显式声明 `--device-name NVIDIA_RTX_PRO_6000_Blackwell_Server_Edition --dtype fp8_w8a8`，或在 `model_config.json` 中加入对应 `fused_moe_config` 字段，确保自动加载。  
- **回退机制**：为防止误匹配，建议在代码中加入 **fallback**：若当前 GPU 不满足 `torch.cuda.get_device_capability()` 对应的 SM 版本，自动切换到通用配置（如 `default_fp8.json`）。  
- **性能验证**：在正式部署前使用官方提供的 benchmark 脚本对比 **fp8_w8a8 + tuned config** 与 **fp8_w8a8 + default config**、以及 **fp16** 的吞吐和推理质量，记录显存占用和延迟。  
- **数值监控**：开启 vLLM 的 **logits scaling** 或 **dynamic loss scaling**，在 FP8 环境下捕获异常数值，必要时切换回 FP16。  
- **文档更新**：在项目的 “Performance Tuning” 文档中添加该硬件的专属章节，说明配置文件路径、适用模型尺寸及已知限制。  
- **持续维护**：当 Triton 升级到 3.8+ 时，评估是否仍适用现有 `BLOCK_SIZE` 参数；如不兼容，及时生成新版 JSON 并通过 CI 自动验证。  

---  

通过上述调优配置，vLLM 在 **RTX PRO 6000 Blackwell** 上的 **FP8 fused‑MoE** 可获得显著的算子层面加速，同时保持向后兼容性。开发者应关注硬件匹配、数值稳定性与 Triton 版本同步，以最大化收益并规避潜在风险。

### feat: add logit_scale to PoolerConfig for affine score calibration (#39435)
**SHA**: `b87575d` | 🔗 [查看提交](https://github.com/vllm-project/vllm/commit/b87575d24b752ee2afffaec00bfb244570d5c95a)

**🎯 变更类型**：功能增强  
**⚡ 重要程度**：🔴高  
**📋 变更摘要**：在 `PoolerConfig` 中新增 `logit_scale` 参数，实现对分类 logits 的仿射校准（即 Platt Scaling）。相关代码在序列/ Token 级别的 pooler head 中加入对 `logit_scale` 的读取与计算，并同步更新文档说明。默认保持原有行为，兼容现有模型。  

**🎯 影响范围**：  
- `vllm/config/pooler.py`（配置结构）  
- `vllm/model_executor/layers/pooler/seqwise/heads.py`、`tokwise/heads.py`（分类 head 前向计算）  
- `vllm/model_executor/layers/pooler/seqwise/poolers.py`、`tokwise/poolers.py`（pooler 实例化）  
- 文档 `docs/models/pooling_models/classify.md`  

**🔍 技术洞察**：  
- **架构影响**：  
  - 配置层新增可选字段 `logit_scale`，沿用现有 `PoolerConfig` 传递链路。  
  - 仅在 classification pooler 的 forward 中插入 `logits *= logit_scale`，不影响其它模型组件（如 decoder、sampling）。  
  - 保持向后兼容：默认 `None`，旧版配置仍可无改动运行。  

- **性能影响**：  
  - 计算成本极低，仅增加一次标量乘法，几乎不增加 GPU/CPU 吞吐或延迟。  
  - 对大批量推理的影响可忽略不计。  

- **安全考虑**：  
  - 该改动属于业务逻辑层面，未涉及 I/O、网络或权限，故不引入新的安全风险。  
  - 唯一需要注意的是数值安全：极端的 `logit_scale`（如 0 或极大值）可能导致 logits 溢出或全部归零，进而影响后续激活函数的数值稳定性。  

**⚠️ 潜在风险**：  
1. **数值异常**：不恰当的 `logit_scale` 可能导致 logits 过大/过小，引发梯度爆炸或概率全部趋近 0/1。  
2. **兼容性**：外部系统若直接解析 `PoolerConfig`（例如自定义脚本）且未识别 `logit_scale`，可能出现字段解析错误。  
3. **校准误用**：用户可能误以为仅设置 `logit_bias` 即完成校准，忽略了 `logit_scale` 必须配合使用，导致校准效果不佳。  

**💡 关注建议**：  
- **使用指引**：在文档中明确 `logit_scale` 与 `logit_bias` 必须一起使用，且推荐的取值范围（如 0.1~10）以避免数值不稳定。  
- **验证测试**：新增单元测试，覆盖以下场景：`logit_scale=None`、`logit_scale=1.0`、极端值（0、非常大）下的前向行为。  
- **配置序列化**：确保 `PoolerConfig` 的 JSON/YAML 序列化/反序列化能够平滑处理缺失 `logit_scale`（保持 `None`）。  
- **监控与回退**：在部署时监控分类概率的分布变化，若出现异常可通过显式设置 `logit_scale=None` 回退到原始行为。  
- **下游兼容**：提醒使用者若依赖原始 logits 进行自定义后处理，需要确认是否开启了 `logit_scale`，否则概率校准结果将不同。

### [Kernel] Porting the TRTLLM minimax_allreduce_rms kernels (#37045)

## 🔍 技术洞察  

### 1. 架构影响  
- **跨 GPU All‑Reduce 改写**：原先依赖 NCCL 的两段式 All‑Reduce + RMSNorm 被 Lamport 环形协议一次性完成，显著简化依赖（不再强依 NCCL）并降低同步点。  
- **层次化 Workspace**：`LamportWorkspace` 通过 CUDA IPC 暴露 **int64** 指针数组，兼容现有 `void** workspace` 参数约定，保持对外 API 不变。  
- **编译期 Fusion**：`MiniMaxQKNormPass` 在 TorchInductor 图中识别 `qkv → split → var → allreduce → rms_norm` 模式，仅在 **decode‑size**（≤ 2048 token）范围内替换，避免对大批量 prefill 产生不必要的额外内存占用。  
- **平台限定**：所有新增代码均在 `#ifndef USE_ROCM` 包裹，仅在 CUDA 环境编译；对 ROCm 用户保持兼容。

### 2. 性能影响  
| 场景 | 旧实现 | 新实现 | 预期收益 |
|------|--------|--------|----------|
| **单矩阵 RMSNorm**（TP=2/4/8） | NCCL All‑Reduce + 2 次 Kernel | Lamport fused kernel（一次写一次读 + 负零轮询） | 约 **1.5‑2×** 吞吐提升，显存峰值下降 ~30% |
| **Q‑K 融合**（MiniMax‑M2, Q=6144, K=1024） | 两次 NCCL All‑Reduce + 两次 RMSNorm | 单个 `minimax_reduce_qk_rms_kernel_lamport_float4`（float4 处理 4 行） | 预估 **2‑3×** 加速，尤其在 2‑8 GPU 环境下通信延迟被显著抑制 |
| **编译后运行** | 多 kernel launch + 多次同步 | 单 kernel launch + 单次 `griddepcontrol` 同步 | 省去 kernel launch 开销及 CUDA 线程块调度延迟 |
| **内存占用** | 每张卡保存 NCCL 临时 buffer + RMSNorm buffer | 只保留 3× `comm_size` 的三缓冲 + flag/layout buffer | 约 **40‑50%** 减少显存占用，提升大模型可装载度 |

> **注意**：上述数字基于作者的基准描述与常规 NCCL‑AllReduce 性能模型，实际提升受 GPU 架构（SM≥9.0）、PCIe/NVLink 带宽、Token 数量等因素影响。

### 3. 安全考虑  
- **负零协议**：使用 IEEE‑754 负零(`0x80000000`) 作为 “空槽” 标记，代码在读取前循环 `while (!is_neg_zero(...))`。此做法在 GPU 上是安全的，且通过 `volatile` 读取避免编译器/硬件乱序。但若出现 kernel 超时或异常退出，负零未被清除会导致 **死锁**。实现中在 kernel 结束后会写 `clear_buf` 来恢复负零，且 `LamportWorkspace.cleanup` 在 Python 端通过 `__del__` 自动清理 IPC 句柄。  
- **IPC 句柄泄漏**：`IpcBuffer.cleanup` 在异常或进程结束时都会调用 `cudaIpcCloseMemHandle`，已加入 `contextlib.suppress(RuntimeError)` 以防止二次释放错误。  
- **跨平台安全**：仅在 CUDA 编译路径下引入，避免在 ROCm 环境出现未定义行为。  
- **输入校验**：`minimax_reduce_rms_op` 对 `dtype`、`size_q`、`hidden_dim` 等做了基本检查，防止非法指针或越界访问。

| 风险点 | 可能导致的问题 | 风险等级 | 建议缓解措施 |
|--------|----------------|----------|--------------|
| **GPU 架构限制** | Kernel 中使用 `__CUDA_ARCH__ >= 900` (SM 9.0) 进行 cluster/group API；在低于 SM9.0 的卡上仍会走普通 block 逻辑，但一些路径（如 `griddepcontrol.wait/launch_dependents`）仅对 SM9+ 有效，可能出现 **未同步** 的情况。 | 中 | 在 `CMakeLists.txt` 添加最小 compute capability 检查（≥ 7.0），并在 `minimax_reduce_rms_op` 中抛出清晰错误信息；提供基于 NCCL 的后备实现（已有 `dispatch_dtype` 可在后期切换）。 |
