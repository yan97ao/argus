### [Frontend][RFC] Rust front-end integration (#40848)
**SHA**: `f2ace1d` | 🔗 [查看提交](https://github.com/vllm-project/vllm/commit/f2ace1d57d28df8d4c5e973dd62d87f47d628cb3)

**🎯 变更类型**：功能增强 / 架构变更 / 重构  
**⚡ 重要程度**：🔴 高  

**📋 变更摘要**  
此次提交在 vLLM 项目中引入了 Rust 前端（`vllm-rs`）的完整集成：  
1. 将 Rust 前端代码作为 Git 子模块引入，新增 CI/Docker 构建阶段用于独立编译生成 `vllm-rs` 可执行文件。  
2. 在 `setup.py` 中使用 `setuptools‑rust`，实现对 Rust 二进制的打包、预编译与本地编译的统一管理。  
3. 新增环境变量 `VLLM_USE_RUST_FRONTEND`, `VLLM_RUST_FRONTEND_PATH`, `VLLM_USE_PRECOMPILED_RUST` 等，以实现二者的灵活开关。  
4. 在服务启动路径（`serve.py`、`api_server.py`、`launch.py`）加入对 Rust 前端进程的调度逻辑，原有 Python API Server 仍作为后备。  
5. 完善 CI 测试、文档、构建脚本以及 `.gitignore/.dockerignore`，确保子模块、编译产物不泄漏。  

**🎯 影响范围**  
- Docker 镜像构建系统（所有 `Dockerfile*`）  
- CI 流水线（Buildkite 脚本、测试 YAML）  
- Python 包打包与安装流程（`setup.py`、`pyproject.toml`、`requirements/*`）  
- 运行时入口：`vllm/entrypoints/cli/serve.py`、`launch.py`、`api_server.py`  
- 环境变量与配置管理（`vllm/envs.py`）  
- 相关单元/集成测试  
- 文档与贡献指南

| **可维护性** | - Rust 前端代码独立在 `rust/` 子模块，符合 **单一职责**；Python 侧仅负责进程包装与参数传递，代码改动量小。<br>- `setup.py` 增加了 `precompiled_build_rust`、`RustExtension`，仍保持向后兼容；未开启 Rust 时完全不触发 Cargo。<br>- 新增的 `jsonify_non_default_args` 与日志改造，使调试信息更 JSON‑friendly，利于未来的日志聚合与监控。 |

### ⚠️ 潜在风险  

1. **子模块初始化遗漏**  
   - 构建镜像或本地开发如果忘记执行 `git submodule update --init --recursive`，`cargo` 会报错。Dockerfile 已加检查，但本地 `python -m pip install -e .` 仍需手动同步。  
2. **跨平台可执行文件分发**  
   - 预编译的 `vllm-rs` 仅在 Linux x86_64（Docker 基础镜像）生成，Windows/macOS 用户仍需自行编译，若 `VLLM_USE_PRECOMPILED_RUST` 被误设，可能导致启动错误。  
3. **信号处理冲突**  
   - `launch.py` 与 `api_server.py` 为防止 SIGTERM 在 `uvicorn` 安装前导致僵尸进程，分别注册了自定义 SIGTERM 处理。若外部系统（Kubernetes、systemd）发送其他信号（如 SIGINT），仍会被默认转发，可能出现未捕获的异常。  
4. **运行时二进制路径解析**  
   - `VLLM_RUST_FRONTEND_PATH=auto` 依赖二进制在 `vllm/vllm-rs` 位置且具可执行权限。对非标准安装方式（如 `conda`、`pipx`）的路径解析可能失败。  
5. **资源竞争**  
   - Rust 前端在同一进程内使用多线程，默认 CPU 亲和策略未显式设置，可能与 Engine 的多进程/多线程争夺 CPU 核心，导致性能抖动。  
6. **CI 缓存失效**  
   - `--mount=type=cache` 用于 Cargo 编译缓存，若缓存目录路径在不同平台之间不兼容（e.g., macOS），会导致缓存失效，构建时间回升。

### Integrate flashinfer b12x MoE and FP4 GEMM kernels for SM120/121 (#40082)
**SHA**: `363fc84` | 🔗 [查看提交](https://github.com/vllm-project/vllm/commit/363fc84407f8c966c1cee6786e45e9e6ab289684)

**🎯 变更类型**：功能增强 / 性能优化 / 架构变更  
**⚡ 重要程度**：🔴高  

**📋 变更摘要**  
本次提交为 vLLM 项目集成了 FlashInfer 在 NVIDIA SM12x（Compute Capability 12.0/12.1）上提供的 **b12x** 系列 FP4 GEMM 与 **融合 MoE**（MoE + SwiGLU）内核。新增了对应的 Python 包装、后端选择逻辑、环境变量、配置选项以及完整单元测试，支持在 RTX Pro 6000 / DGX Spark 等硬件上以 FP4 量化实现更高吞吐的 MoE 推理。

### 🔍 技术洞察  

| 维度 | 影响说明 |
|------|----------|
| **架构影响** | - 引入新 **FlashInfer CuteDSL** 后端，仅在 **SM12x** 设备可用，保持原有后端（cutlass、cudnn、trtllm）不变。<br>- `FlashInferB12xExperts` 采用 **FusedMoEExpertsModular** 接口，内部一次调用 `flashinfer_b12x_fused_moe` 完成 **token dispatch → W1 GEMM → SwiGLU → W2 GEMM → top‑k reduction**，显著简化执行图，减少 kernel launch overhead。<br>- 为了兼容已有的 `nvfp4` 量化流程，增加了 **scale 处理**（`process_weights_after_loading`）以及 **MMA 布局转换**（`flashinfer_convert_sf_to_mma_layout`），确保在加载时即完成一次性布局变换，运行时仅做一次矩阵乘。 |
| **性能影响** | - **显存占用**：一次性融合后期望比原先的两次 GEMM + 中间激活缓存省去约 50% 激活暂存显存。<br>- **吞吐**：FlashInfer b12x 使用 **CUTLASS SM12x mma** 以及 **FP4 动态量化**，在 SM12x 上可比 Cutlass‑FP4 提升 1.3‑1.8×，尤其在 **expert‑per‑token ≥ 4**、**expert 数量≥16** 时收益更明显。<br>- **启动延迟**：因 kernel 规模变大，单次 launch 的启动时间略增，但整体推理时延仍下降（因为跨 kernel 同步开销被消除）。<br>- **自适应选择**：默认自动选择仍排除 b12x，防止在未明确支持的平台上出现 fallback。用户需要显式设置 `MOE_BACKEND=flashinfer_b12x` 或 `VLLM_NVFP4_GEMM_BACKEND=flashinfer-b12x` 才会启用；这保证了对老硬件的向后兼容。 |
| **安全考虑** | - 仅在 **CUDA** 环境下执行，所有新增的 `has_flashinfer_*` 检查都基于 **模块属性存在**，不会引入未定义符号。<br>- 新增的 **环境变量** 与 **配置选项** 均为字符串常量，未涉及动态代码执行。<br>- 单元测试使用 `pytest.skip` 机制确保在不满足 SM12x 条件或 FlashInfer 版本不匹配时不运行，避免因缺失依赖导致的崩溃。 |
| **可维护性** | - 代码遵循现有 **Modular‑MoE** 框架抽象，新增文件与类位置清晰，易于后续扩展（例如支持 NVFP8、不同激活函数）。<br>- 通过 **懒加载** (`_lazy_import_wrapper`) 与 **functools.cache** 实现插件可选加载，保持主仓库对 FlashInfer 版本的松耦合。<br>- 单元测试覆盖了多维度（不同 `m,n,k`、专家数、top‑k、dtype）以及量化尺度路径，提升回归安全性。<br>- 文档/注释已经在代码中说明了 **scale‑convention** 与 **weight reorder** 的细节，降低新成员理解门槛。 |

### ⚠️ 潜在风险  

1. **硬件依赖**：仅在 SM12x（≥12.0）上可用，若用户误设 `flashinfer_b12x` 而运行在旧卡（如 A100，SM80）会触发 `is_supported` 返回 `False` 并报错，需确保文档与 CLI 参数对硬件需求有足够提示。
