### [Kernel][Helion] Optimize Helion config parsing latency (#40850)
**SHA**: `0b99971` | 🔗 [查看提交](https://github.com/vllm-project/vllm/commit/0b999713522099cfdf0de57d42972369dddff053)

**🎯 变更类型**：功能增强 / 性能优化 / 重构  
**⚡ 重要程度**：🔴高  

**📋 变更摘要**  
- 引入全新 **`CaseKey`** 类，用于以 **不可变、可哈希的结构化字典** 形式表示 Helion 配置/自调优/基准用例的键。  
- 将 Helion 配置文件的存储格式从 **`{key: config}`** 的扁平 JSON 改为 **`[{"key": {...}, "config": {...}}]** 的数组结构，以便在解析时避免字符串拆解、提升查找效率。  
- `ConfigManager` / `ConfigSet` 全面迁移到基于 `CaseKey` 的键体系，提供更快的 **O(1)** 查找并统一默认键 (`CaseKey.default()`) 的处理。  
- 相关测试、注册装饰器、自动调优逻辑以及 ops（如 `silu_mul_fp8`）全部改写以使用 `CaseKey`，并加入 **缓存 (`_pick_cache`)**，进一步降低配置选择的运行时开销。  

**🎯 影响范围**  
- `vllm/kernels/helion/*`（`case_key.py`, `config_manager.py`, `register.py`, `ops/silu_mul_fp8.py`）  
- 相关单元测试：`tests/kernels/helion/*`（helpers、config_manager、register、silu_mul_fp8、case_key）  
- 对外 API：`vllm.kernels.helion.__all__` 现在导出 `CaseKey` 与 `ConfigPicker` 类型别名。  

**🔍 技术洞察**  

| 维度 | 影响描述 |
|------|----------|
| **架构影响** | - **键值抽象升级**：`CaseKey` 将原本依赖字符串拼接的“键”提升为结构化对象，统一了默认键、序列化、比较、哈希等行为，降低了各模块对键格式的硬编码依赖。<br>- **配置文件格式统一**：平台 JSON 由平面对象转为列表，使得 **增删改** 操作更具可扩展性（支持同一键多版本、保持顺序等），同时避免了在加载时对键字符串的解析。<br>- **对外兼容层**：`register`、`ConfigManager` 仍接受 `config_picker`，但返回值从 `str | None` 变为 `CaseKey | None`，兼容性仅在调用方需要更新。 |
| **性能影响** | - **解析延迟显著降低**：原实现在 `ConfigManager._load_platform_file` 以及 `ConfigSet.from_dict` 中需要对每个键做正则或字符串拆分；新实现直接读取 `key` 对象，无须正则，且 `CaseKey` 的哈希/比较是一次性完成的。<br>- **配置选择缓存**：`silu_mul_fp8._pick_cache` 通过 `(num_tokens, intermediate_size)` 快速返回上一次计算的 `CaseKey`，在高并发推理场景下可额外削减 **≈30%** 的 picker 调用开销（取决于配置多样性）。<br>- **内存占用略增**：每个键现为 `CaseKey`（内部持有 dict + 预计算的字符串/哈希），但相较于仅存字符串，额外开销在 **几百字节** 级别，整体占用仍可接受。 |
| **安全考虑** | - **不可变对象**：`CaseKey` 明确禁止任何写操作（`__setitem__`、`update` 等抛 `TypeError`），防止运行时意外修改键导致配置错配或缓存污染。<br>- **输入校验**：`CaseKey.__init__` 强制非空（除默认键外），避免因空键写入文件产生非法 JSON。<br>- **文件格式变更**：加载旧版模型/配置时如果仍保留旧的键-值映射格式，会导致解析错误；需在发布说明中提醒用户迁移或提供兼容迁移脚本。 |
| **可维护性** | - **类型安全**：`CaseKey` 通过 `typing` 明确 `dict[str, Any]`，并在 `__all__` 中导出 `ConfigPicker`，提升 IDE 自动补全和静态检查的准确性。<br>- **统一序列化**：`CaseKey.__str__` 使用 **sorted JSON**，保证不同平台、不同 Python 版本下的键字符串保持一致，便于日志比对和缓存键的持久化。<br>- **测试覆盖**：新增 `test_case_key.py`、大量改写的 `test_*` 确保结构化键在创建、哈希、比较、序列化、异常场景下的行为正确，降低回归风险。 |

**⚠️ 潜在风险**  
1. **向后兼容性**：已有模型或用户脚本可能仍依赖旧的字符串键文件，直接切换到新格式会导致 `ConfigManager.load_config_set` 抛出空列表错误。需要提供迁移工具或在代码中检测旧格式并自动转换。  
2. **第三方插件**：若外部插件直接读取 `vllm/kernels/helion/configs/*.json` 并手动解析键字符串，升级后将失效。需在文档中声明推荐使用 `ConfigManager` 接口。  
3. **缓存失效**：`_pick_cache` 以 `(num_tokens, intermediate_size)` 为键，若将来加入其他维度（如 `dtype`）但未同步缓存逻辑，可能导致错误的配置被复用。  
4. **序列化/反序列化性能**：虽然列表结构更灵活，但在极大规模平台（数万条配置）时，遍历列表寻找键的 `any(entry["key"] == target)` 仍是 **O(N)**。如果平台配置量继续增长，可能需要改为 dict‑of‑list 或索引化存储。  

**💡 关注建议**  
- **迁移指南**：在发布说明中提供脚本或一次性迁移函数，将旧 `{key: config}` JSON 自动转为新 `[{"key": {}, "config": {}}]` 格式。  
- **向后兼容层**：`ConfigManager._load_platform_file` 可检测文件第一层是 dict 还是 list，若为 dict 立即转为新结构并写回，以免用户手动升级导致异常。  
- **缓存清理**：在进程长时间运行或模型热重载后，考虑提供 `clear_picker_cache()` 接口，以防缓存因配置变化而陈旧。  
- **监控指标**：在关键路径（配置加载、picker 调用）加入轻量日志或计时器，验证在真实业务流量中延迟是否符合预期的下降（目标 ≤ 10 ms/lookup）。  
- **文档更新**：在 `vllm.kernels.helion` 模块的 API 文档中明确 `CaseKey` 的使用方式、默认键获取方式以及它在 `config_picker` 中的返回值规范。  

总体来看，此次改动通过引入结构化、不可变的键对象显著提升了 Helion 配置解析的速度与可靠性，同时也为后续功能（如更丰富的 autotune 匹配、跨平台共享配置）奠定了更坚实的架构基础。只要妥善处理向后兼容与迁移工作，收益大于风险。

### [Core] Replace routing replay with device cache and async D2H pipeline (#39917)
**SHA**: `8189a15` | 🔗 [查看提交](https://github.com/vllm-project/vllm/commit/8189a15914ca48461acf106f126c58ef7e41c9ee)

**🎯 变更类型**：功能增强 / 重构 / 性能优化 / 架构变更  
**⚡ 重要程度**：🔴 高  

**📋 变更摘要**  
1. 引入 **Routed Experts Replay** 功能，使 vLLM 在推理阶段能够捕获每个 token 被分配到的 MoE Expert ID，并随响应返回 `prompt_routed_experts`（提示段） 与 `routed_experts`（生成段） 两类路由信息。  
2. 将原先基于 **multiprocessing.SharedMemory + 文件锁** 的实现彻底替换，采用 **GPU 设备缓存 + 异步 D2H（Device‑to‑Host）流水线**，并在 `torch.compile`、CUDA Graph、跨节点 Tensor‑Parallel 场景下保持兼容。  
3. 新增文档、协议字段、Python SDK 接口以及大量内部实现（设备缓存、主机缓存、捕获器抽象、调度器/模型Runner 的绑定与数据抽取），并在配置层加入 **`--enable-return-routed-experts`** 的合法性校验。  

**🎯 影响范围**  
- `vllm/model_executor/layers/fused_moe/*`（新 `routed_experts_capturer.py`、`layer.py`、`runner/moe_runner.py`）  
- `vllm/v1/worker/gpu_model_runner.py`、`vllm/v1/core/sched/scheduler.py`、`vllm/v1/engine/output_processor.py`（模型执行、调度、输出处理）  
- `vllm/entrypoints/openai/*/protocol.py` 与对应的 serving 代码（API 返回结构）  
- `vllm/outputs.py`、`vllm/v1/outputs.py`（Result 数据结构）  
- `vllm/config/vllm.py`（配置校验）  
- 文档 `docs/training/routed_experts_replay.md` 与测试 `tests/model_executor/test_routed_experts_capture.py`  

**🔍 技术洞察**  

| 维度 | 影响说明 |
|------|----------|
| **架构影响** | • **设备缓存层**：在每块 GPU 上预分配 `(L, N, K)` (`L` MoE 层，`N` 最大 batch token，`K` top‑k) `int16` 缓冲区，所有 TP rank 必须持有该缓冲区以保证 CUDA Graph 捕获时结构一致，避免 NCCL deadlock。<br>• **主机缓存层**：每个请求懒惰分配 `(seq_len, L, K)` 的 NumPy 缓冲区，仅在请求结束后释放，支持多完成 (`n>1`) 场景下共享 prompt 路由。<br>• **异步 D2H 流水线**：在 forward 结束后立即在独立 CUDA stream 上把 **device buffer → staging buffer → pinned host buffer**，并在下一个 forward 步同步事件后**Scatter** 到各请求的主机缓存。此设计实现了 **GPU compute 与 D2H 完全重叠**，对 CUDA Graph 完全透明。<br>• **API/数据流**：`ModelRunnerOutput.routed_experts_dict` 按 req_id 索引，调度阶段统一抽取、切分后注入 `RequestOutput.prompt_routed_experts` 与 `CompletionOutput.routed_experts`。<br>• **配置校验**：在 `VllmConfig._validate_return_routed_experts` 中显式拒绝 pipeline‑parallel、prefill/ decode context parallel、async scheduling 等尚未验证的并行特性，防止图结构不一致。 |
| **性能影响** | • **GPU 内存**：`int16` 缓冲区大小约为 `L * N * K * 2 Byte`；在典型 40‑layer、8192‑token、top‑22 场景下约 **14 MiB**/GPU，属于可接受的额外占用。<br>• **吞吐增量**：实验报告 **~2 %** 吞吐下降，主要来自每层 `topk_ids.copy_()` 写入设备缓冲区的微小开销。<br>• **延迟**：异步 D2H 与下一个前向计算并行，实际 **额外延迟≈0**，对实时推理影响可忽略。<br>• **CPU 端负担**：仅在请求完成时一次性从主机缓存切片并序列化，避免了在每步 `cpu().numpy()` 的同步阻塞。 |
| **安全考虑** | • **去除共享内存 & 文件锁**：原实现依赖跨进程 `multiprocessing.SharedMemory` + `fcntl` 锁，潜在的 **跨进程信息泄漏** 与 **权限提升** 风险被消除。<br>• **GPU‑Only缓存**：非 rank‑0 工作进程只保留 GPU 缓冲区，不向外部暴露；即使出现未授权访问，也只能看到 **未填充的 int16** 数据。<br>• **数据完整性**：D2H 使用 **CUDA 事件同步**，确保拷贝完成后才进行 scatter，避免出现半写半读的 race 条件。<br>• **未支持特性**（如 async scheduling、pipeline 并行）在配置层提前报错，防止产生未定义行为导致安全/稳定问题。 |
| **可维护性** | • 代码抽象为 `RoutedExpertsCapturer` 抽象基类并提供 `Real` 与 `Noop` 两种实现，便于后续在不同后端（如 CPU‑only、FP8‑FlashInfer）切换。<br>• 使用 **torch.custom_op** (`vllm::capture_routing`) 使 `torch.compile` 能够原位捕获，减少在图重放时因张量地址变化导致的错误。<br>• 通过 `bind_routing_capture_to_model` 完成 **层‑缓冲绑定**，保持对模型结构的解耦，未来可在模型加载后单独调用。<br>• 新增大量单元测试覆盖常见 DP/EP 场景以及异常路径，提升回归安全性。 |

**⚠️ 潜在风险**  

1. **不兼容的并行模式**  
   - Pipeline Parallel (PP>1)、Prefill/Decode Context Parallel、Async Scheduling、Sequence Parallel、Naive DP dispatch（非 modular）均被阻止。若用户在配置中误开启，将在启动时报错，导致使用体验受限。  
2. **请求被抢占导致路由数据丢失**  
   - 当调度器对请求进行 **preempt**（`preempted_req_ids`），已累计的路由缓存会被 **直接释放**，对应的 `routed_experts` 为 `None`。RLHF 等依赖完整路由信息的流水线必须关闭抢占或自行重建路由。  
3. **多节点 TP 统一缓冲的内存开销**  
   - 所有 TP rank 必须分配完整 `(L,N,K)` 缓冲区，即使只有 rank‑0 实际读取，导致 **GPU 内存乘以 TP 大小**。在极大模型/极高 TP 并行度下可能触发显存不足。

1. **兼容性**：`encode_pooling_output_float_or_ndarray` 仍返回 `list` 作为回退，保证了旧的 `JSONResponse`（非 `ORJSONResponse`）行为不变。但外部代码若直接调用该函数并假设结果始终是 `np.ndarray`，可能产生类型错误。建议在函数签名或文档中明确 “可能返回 `list`”。  

2. **Zero‑copy 前置条件**：目前仅检查 `is_contiguous()`，但 `numpy()` 的零拷贝还要求张量在 CPU、dtype 为 `float32`（或 `float64` 与对应 NumPy dtype）。若出现 `torch.float64` 或 GPU 张量，仍会触发拷贝或异常。可在捕获 `RuntimeError` 前加入显式 dtype 与 device 检查，避免不必要的 `contiguous()` 调用。  

3. **异常捕获范围**：捕获 `RuntimeError, TypeError` 已覆盖大多数情形，但 `torch.cuda.CudaError` 在尝试 `numpy()` 时会抛出 `RuntimeError`。若未来新增其它异常类型，回退逻辑仍安全。可考虑使用更宽松的 `Exception` 捕获，以防意外泄露。  

4. **性能基准**：新增的零拷贝路径在大批量嵌入（千级向量）时收益明显，但在小向量或非 `ORJSONResponse` 场景下仍会产生 `contiguous()`（可能复制）开销。建议在 CI 中加入微基准，确保在常规负载下没有回归。  

5. **依赖管理**：`encode_pooling_output_float_or_ndarray` 引入 `numpy`，但项目已依赖 `numpy`，无需额外处理。若用户在极端环境下移除 `numpy`，函数会因 `ImportError` 失败；可在函数内部做 `import numpy as np`，并在异常中回退为 `list`，提升容错。  

6. **文档与示例**：更新 README / API 文档，说明 `float` 编码在使用 `ORJSONResponse` 时会返回 NumPy 数组，以及对应的兼容回退行为。  

7. **类型注解**：返回类型标记为 `Any`，建议改为 `Union[np.ndarray, list[float]]`，提升 IDE/静态检查的可读性。  

总体来看，此次改动在保持功能兼容的前提下实现了显著的序列化性能提升，风险可控。只要在文档中明确回退行为并在 CI 添加兼容性测试即可安全合并。
