### feat(kv-events): emit KV cache metadata (#40984)
**SHA**: `bcb9c13` | 🔗 [查看提交](https://github.com/vllm-project/vllm/commit/bcb9c133bafed161b3f4d0ea391c985465429680)

**🎯 变更类型**：功能增强  
**⚡ 重要程度**：🔴高  
**📋 变更摘要**：  
本次提交为 KV 缓存事件体系引入了 **缓存规格元数据**。新增 `KVCacheSpecKind` 枚举、`get_kv_cache_spec_kind` 与 `get_kv_cache_spec_sliding_window` 辅助函数，用于在 `BlockStored` 事件中携带缓存类型（full‑attention、sliding‑window 等）以及 sliding‑window 大小。`KVCacheManager.take_events` 在返回前为每个 `BlockStored` 事件注入上述字段，并在 `KVCacheManager` 构造时预先缓存这些元数据。相应的测试、日志和 Engine API 亦同步更新，实现了对外可序列化的 KV‑cache 分组元数据查询。

**🎯 影响范围**：  
- `vllm/distributed/kv_events.py`（`BlockStored` 结构扩展、hash 计算）  
- `vllm/v1/core/kv_cache_manager.py`（事件注入元数据）  
- `vllm/v1/core/kv_cache_coordinator.py`（构造参数对齐）  
- `vllm/v1/engine/core.py`（提供 `get_kv_cache_group_metadata` 接口）  
- `vllm/v1/kv_cache_interface.py`（新增 `KVCacheSpecKind` 枚举与查询函数）  
- 单元测试文件（覆盖新功能）  

**🔍 技术洞察**：  
- **架构影响**：  
  - **职责分离**：原本 `BlockPool` 只负责结构化事件（存储/删除），现在 `KVCacheManager` 负责为事件注入语义层面的 KV‑cache 规格信息，保持 `BlockPool` 的简洁性。  
  - **可扩展性**：通过 `KVCacheSpecKind` 枚举和统一的查询函数，后续新增注意力子类（例如新型稀疏注意力）只需在枚举与 `get_kv_cache_spec_kind` 中添加映射，即可自动被事件携带。  
  - **API 公开**：`EngineCore.get_kv_cache_group_metadata` 为调度器外部提供统一、msgspec‑可序列化的元数据，便于监控、可观测平台或自定义调度策略使用。  

- **性能影响**：  
  - **新增字段**：`BlockStored` 现在多两个可选字段（字符串、整数），对单个事件的内存占用微增（≈数十字节），对整体 KV‑cache 规模的影响可忽略不计。  
  - **运行时开销**：在 `KVCacheManager.take_events` 中遍历一次事件列表并做字典查找和赋值，时间复杂度 O(N)（N 为当批次产生的事件数），典型批次几百条，额外耗时在毫秒级以下。  
  - **哈希行为变更**：`BlockStored.__hash__` 现在包含新字段，导致相同结构但不同规格的事件哈希不再相同，这在缓存或去重逻辑上是预期行为，未见性能倒退。  

- **安全考虑**：  
  - 本次改动仅是 **只读元数据** 的曝光，没有引入外部输入或可执行代码，无直接安全风险。  
  - 若外部系统将这些元数据用于访问控制或审计，需要确保枚举值的可信来源（本项目内部生成），仍属安全可信范围。  

**⚠️ 潜在风险**：  
1. **兼容性**：旧版消费者（如第三方监控）在解析 `BlockStored` 时可能会因未知字段报错，尤其在使用严格的 msgspec/JSON schema 时。建议更新 schema 为可选字段。  
2. **哈希语义变更**：引入 `kv_cache_spec_kind`、`kv_cache_spec_sliding_window` 后，`hash(event_a) != hash(event_b)` 即使其它字段相同，这可能影响已有的基于哈希的去重/缓存逻辑，需要审查是否有代码依赖旧哈希行为。  
3. **日志/警告**：当 `group_idx` 超出已配置的 KV‑cache 分组数，会产生警告日志。如果系统频繁出现此类情况，可能掩盖真实错误。  
4. **错误的元数据注入**：若 `kv_cache_event_metadata` 与实际 `kv_cache_config` 不一致（例如在运行时动态修改配置），注入的元数据将不匹配，建议在配置变更后重新实例化 `KVCacheManager`。  

**💡 关注建议**：  
- **文档与 SDK**：在 vLLM 官方文档中补充 `KVCacheSpecKind` 与新事件字段的说明，提供示例代码展示如何过滤特定规格的块事件。  
- **版号升级**：此为向后兼容的功能增加，建议在 `CHANGELOG` 中标记为 “新增 KV‑cache 事件元数据” 并在主版本号后递增次版本（如 `0.3.1 → 0.4.0`）。  
- **向后兼容性测试**：在 CI 中加入兼容性验证，确保使用旧版 `BlockStored` 结构的代码能够在新版库中顺利运行（字段默认 `None`）。  
- **性能基准**：对高并发推理场景跑一次 `take_events` 的微基准，确认额外遍历不会在极端负载下产生瓶颈。  
- **监控与告警**：若启用了 KV‑cache 事件监控，更新监控规则以捕获 `kv_cache_spec_kind` 与 `kv_cache_spec_sliding_window`，并在异常值（如未知 `UNKNOWN`）时触发告警。  

---  
此变更显著提升了 KV‑cache 事件的可观测性和可扩展性，对系统架构是一次正向的解耦改造，风险可控，只需做好兼容性和文档同步即可安全上线。

### [MoE Refactor] Move expert map related code into ExpertMapManager class (#41046)
**SHA**: `206eaed` | 🔗 [查看提交](https://github.com/vllm-project/vllm/commit/206eaed08d25fb26f0ffb4de26fbadcbeec3dc7b)

**🎯 变更类型**：重构 / 架构变更  
**⚡ 重要程度**：🔴高  
**📋 变更摘要**：  
1. 将 MoE（Mixture‑of‑Experts）层中所有与 *expert map*、*expert placement*、*routing table* 相关的逻辑抽离到新模块 `vllm/model_executor/layers/fused_moe/expert_map_manager.py`，实现 `ExpertMapManager` 类统一管理。  
2. `FusedMoE` 及其所有量化/压缩实现改为通过 `ExpertMapManager` 获取 `expert_map`、`expert_mask`、`routing_tables`，并删除原有的 `determine_expert_map` 与 `determine_expert_placement_strategy` 函数。  
3. 相应测试、内核调用以及若干辅助函数更新引用，统一使用 `layer._expert_routing_tables()`（内部包装）而非旧的 `_maybe_init_expert_routing_tables()`。

---

## 🎯 影响范围
- **核心 MoE 层**：`vllm/model_executor/layers/fused_moe/layer.py`（几乎全部 expert‑map 相关代码被迁移）。  
- **新模块**：`expert_map_manager.py`（约 500 行实现）。  
- **量化/压缩子模块**：所有 `compressed_tensors_moe/*`、`quantization/*` 以及 `unquantized_fused_moe_method.py` 中的路由表获取方式均改为 `layer._expert_routing_tables()`。  
- **测试**：`tests/distributed/test_expert_placement.py`、`tests/kernels/moe/*` 更新 import。  
- **内部工具**：`_expert_routing_tables`、`update_expert_map_info`、`update_expert_map` 等新方法在 `FusedMoE` 中提供。  

---

## 🔍 技术洞察

| 维度 | 影响分析 |
|------|----------|
| **架构影响** | - **职责分离**：`ExpertMapManager` 将 expert‑map、placement 策略、routing‑table 生成与 EP（Expert Parallelism）配置耦合的逻辑集中，`FusedMoE` 只负责层本身的 forward/backward 与 kernel 调度，降低类的复杂度，利于后续功能扩展（例如动态弹性 EP、跨节点迁移）。<br>- **可复用性**：其它 MoE 变体（如 `fused_moe_topk`、`mixed_dtypes`）可直接实例化 `ExpertMapManager`，避免重复实现。<br>- **单一入口**：所有 expert‑placement 决策（linear / round_robin）统一校验，日志、警告以及回退策略集中管理，减少分散的 edge‑case 处理。 |
| **性能影响** | - **初始化成本**：创建 `ExpertMapManager` 会在构造阶段一次性计算 `expert_map`、`expert_mask` 与（可选）路由表；相较于原来在 `FusedMoE.__init__` 中分散调用，整体开销略有增加，但仅在模型加载阶段完成，对运行时推理吞吐几乎无影响。<br>- **内存布局**：`ExpertMapManager` 使用 `torch.full` 与 `torch.arange` 生成 `expert_map`，保持与旧实现相同的 Tensor 大小和 dtype（int32），不产生额外拷贝。<br>- **路由表缓存**：`_expert_routing_tables()` 在首次请求时返回已注册的 buffer，后续调用成为 O(1) 读取，保持原有低延迟。 |
| **安全考虑** | - **输入校验**：`determine_expert_map` 仍然断言 `ep_size > 0`，并在 `determine_expert_placement_strategy` 中对不支持的 round‑robin 场景给出明确 warning，防止误用导致错误路由表。<br>- **日志记录**：`ExpertMapManager` 在首次初始化时通过 `logger.info_once` 输出 EP 配置，避免因隐式默认导致安全/资源误判。<br>- **无新外部依赖**：仅使用标准 `torch` 与项目内部配置，未引入额外的安全风险。 |
| **可维护性** | - **代码量缩减**：`layer.py` 删除了约 300 行与 expert‑map 直接相关的实现，文件从 543 行降至 248 行，使核心逻辑更易阅读。<br>- **统一测试点**：所有对 expert placement 的单元测试仍然指向同一公有函数 `determine_expert_map`（仍保留在新模块），避免因路径改动导致测试失效。<br>- **API 稳定**：对外唯一公开的函数仍为 `determine_expert_map`（现在位于 `expert_map_manager.py`），兼容已有用户代码。 |
| **兼容性** | - **向后兼容**：旧的 `determine_expert_map` 仍可通过新模块导入，未破坏外部 import。<br>- **序列化**：`ExpertMapManager` 本身不持有任何 `nn.Parameter`，只注册普通 `torch.Tensor` 为 buffer，模型 `state_dict` 仍保持原有键名（`_expert_map`、`expert_mask`、`expert_global_to_physical` 等），不影响 checkpoint 加载。 |

## ⚠️ 潜在风险
1. **动态 EP 变更未充分测试**  
   - `ExpertMapManager.update()` 支持在运行时更改 `moe_parallel_config`（如弹性扩缩容），但现有代码路径（如 `FusedMoE.update_expert_map`) 只在模型启动时调用一次。若未来加入弹性 EP，需确保在所有使用 `expert_map` 的 kernel 前调用 `update_expert_map_info()`，否则可能出现 stale 的路由表。  
2. **注册 Buffer 的顺序**  
   - `FusedMoE.__init__` 现在在 `ExpertMapManager` 创建后立即调用 `self.update_expert_map_info()`，该过程注册 `_expert_map`、`expert_mask`、路由表为 buffers。若后续在子类或插件中再次调用 `self.register_buffer` 同名属性，可能触发 PyTorch 警告或覆盖。  
3. **GPU/CPU 设备不匹配**  
   - `ExpertMapManager` 在 `determine_expert_map` 中未显式指定 device；返回的 Tensor 默认位于 CPU。随后 `FusedMoE.register_buffer` 会把它迁移到当前模块的设备（由 `self.register_buffer` 自动完成），但若在多进程（DDP）场景下提前使用了这些 Tensor（如在自定义日志或调度逻辑中），可能出现 CPU‑GPU 不匹配。  
4. **错误的 `rocm_aiter_enabled` 组合**

3. **INT32_MAX 检查**：对 `blocks_x`、`blocks_y` 加入上限检查是合理的，但若输入极端大（如 2⁶⁰ 行），仍可能在 `dim3` 构造时溢出。可考虑将检查信息改为 “grid dimensions exceed CUDA limits”。  
4. **兼容性**：模板参数从 `per_token_group_quant_8bit_packed_register_kernel<T,DST,GROUP_SIZE>` 变为 `..., kGroupsPerBlockX, kRowsPerBlock>`。若其他文件显式特化或手动实例化该 kernel，需要同步更新。

总的来说，本次改动通过 2‑D grid 消除了热点除取模，代码可读性和执行效率都有提升。但需确保上述边界处理和 `kx` 限制在所有生产数据上都成立，并补充相应的单元/性能测试以防回归。
