### [Feature] ViT Full CUDA Graph (#35963)
**SHA**: `f85e479` | 🔗 [查看提交](https://github.com/vllm-project/vllm/commit/f85e479e66167532feddc9aafd56562402def9d9)

**🎯 变更类型**：功能增强 / 重构 / 架构变更  
**⚡ 重要程度**：🔴高  
**📋 变更摘要**：  
本次提交为 vLLM 引入了视觉编码器（ViT）全链路 CUDA Graph 支持。新增编译配置开关 `cudagraph_mm_encoder` 以及预算相关参数；在模型层面通过 `SupportsEncoderCudaGraph` 协议抽象出 Vision Encoder 所需的元数据、输入切片、缓冲区准备与前向接口；实现 `EncoderCudaGraphManager` 完成预算‑批次（budget‑batch）CUDA Graph 捕获、贪心打包、数据并行（DP）分片与全局聚合；并在 `GPUModelRunner` 中接入该管理器，实现自动 fallback 到 eager 前向。配套单元测试覆盖预算生成、预算查找、统计、GPU 捕获/回放、fallback 与分块等关键路径。

---

### 🎯 影响范围
- **配置层**：`vllm/config/compilation.py`（新增 CUDA Graph 开关及预算参数）  
- **模型接口**：`vllm/model_executor/models/interfaces.py`（新增 `SupportsEncoderCudaGraph` 协议及 `supports_encoder_cudagraph` 判断函数）  
- **Vision 模型实现**：`vllm/model_executor/models/qwen3_vl.py`（实现协议方法、预算范围、捕获/回放准备等）  
- **CUDA Graph 管理**：`vllm/v1/worker/gpu/mm/encoder_cudagraph.py` 与对应的 DTO `encoder_cudagraph_defs.py`（核心实现）  
- **运行时调度**：`vllm/v1/worker/gpu_model_runner.py`（在模型捕获阶段实例化并使用 `EncoderCudaGraphManager`）  
- **测试**：`tests/v1/cudagraph/test_encoder_cudagraph.py`（完整单元测试）  
- **其它**：`vllm/v1/worker/gpu_model_runner.py` 中的少量变量新增与日志输出。

---

### 🔍 技术洞察

#### 架构影响
1. **模块化协议层** – 通过 `SupportsEncoderCudaGraph` 将 Vision Encoder 与 CUDA Graph 管理解耦，其他模型只要实现协议即可复用，无需改动管理器内部逻辑。  
2. **预算‑批次划分** – 引入 **Token Budget** 概念，实现 *greedy smallest‑first* 打包算法，保证在给定 token 上限下尽可能多图像复用同一 Graph，减少 eager fallback。  
3. **数据并行 (DP) 支持** – 在 `EncoderCudaGraphManager._dp_shard` / `_dp_gather` 中利用现有的 **tensor model parallel** 通道，实现跨 TP 设备的负载均衡与全局聚合，保持与 LLM 主体的并行策略一致。  
4. **捕获/回放分离** – `prepare_encoder_cudagraph_capture_inputs`/`prepare_encoder_cudagraph_replay_buffers` 把 **dummy** 捕获输入与 **真实** 回放缓冲区解耦，避免在图捕获阶段依赖实际 batch 内容。  
5. **统计/日志** – 新增 `graph_hits/misses` 计数及周期性日志，便于运维监控 CUDA Graph 利用率。  

#### 性能影响
| 场景 | 预期提升点 | 说明 |
|------|------------|------|
| 大批量小图像（如 4‑8 张 224×224） | **显著降低 kernel launch 开销**（一次 Graph 复用多张图） | 预算‑batch 使得所有图像在同一 Graph 内完成前向，避免 per‑image 启动开销。 |
| 中等 batch、混合尺寸 | **较小的回放开销 + 只对符合预算的子批次使用 Graph** | 贪心打包 + fallback 保证大尺寸图像仍可通过 eager 方式执行，性能退化可控。 |
| 多卡（TP>1） | **DP 负载均衡** + **All‑Gather** 后一次性拼接输出 | 通过 `get_load_balance_assignment` 将 patch 数据均匀分摊至各 TP，保持图捕获期间的 batch 大小固定（max_batch_size），提升跨卡并行度。 |
| 常规 LLM 推理 | **基本无影响** | 仅在 `supports_mm_inputs` 为真且模型实现协议时才激活，默认 `cudagraph_mm_encoder=False`，不影响现有路径。 |

> 由于 Graph 捕获在 **CUDA inference mode** 中进行，图内所有张量都保持 **同一设备、同一 dtype**，因此可以避开跨设备同步，进一步提升吞吐。

#### 安全考虑
- **代码执行安全**：新增的协议方法均在模型内部实现，且在图捕获/回放阶段只执行 **Tensor 操作**（`torch.nn.Linear`、`torch.mean`、张量切片），不涉及外部脚本或系统调用，没有新增的外部依赖。  
- **内存安全**：捕获时使用 **dummy** 输入并在回放时 `zero_()` 再 slice‑copy，防止泄露前一次 batch 的残余数据；但若用户提供的 `token_budget` 超出显存容量，可能导致 OOM，需在配置层或模型层做显存预估检查。  
- **并发安全**：`EncoderCudaGraphManager` 在捕获阶段仅在单进程、单 GPU 上运行；回放阶段通过 `torch.cuda.CUDAGraph.replay()` 是线程安全的（官方已保证），但在多线程环境下仍建议对 `execute` 做外层序列化或使用 `torch.cuda.graph` 自身的锁机制。  

---

### ⚠️ 潜在风险
| 风险 | 描述 | 可能影响 |
|------|------|----------|
| **显存占用激增** | 每个 token budget 捕获一次完整前向 Graph，且每个 Graph 需要 **输入缓冲区 + 多个 metadata 缓冲区 + 输出缓冲区**（按 `max_batch_size` 预留）。在预算数较多或 `max_batch_size` 较大时，显存需求可能超出卡容量。 |
| **预算不匹配导致频繁 fallback** | 若实际图像 token 远大于最高预算（例如极高分辨率图像），会触发 eager 前向，失去 Graph 的优势；统计显示 `graph_misses` 会快速增长，性能收益被抵消。 |
| **DP 分片不一致** | `get_load_balance_assignment` 依据 *patch数* 做负载均衡，若模型内部还有 **额外内部状态**（如可变长度的 KV cache）未同步，可能出现跨 TP 输出错位。 |
| **捕获阶段的模型副作用** | 某些模型在 forward 时有 **随机数** 或 **层内部状态更新**（如 dropout、BN），捕获后固定了这些随机状态，导致回放时输出不符合期望。当前实现使用 `torch.inference_mode()` 并没有显式禁用 dropout，需确保模型在捕获前已设为 eval 模式。 |
| **兼容性回归** | 新增 `cudagraph_mm_encoder` 配置默认 `False`，但在某些自定义脚本中可能直接打开而未提供合适的 `token_budgets` 或 `max_images_per_batch`，导致启动时报错或误捕获。 |
| **统计/日志频率** | `log_stats_interval` 固定为 100，若单轮请求量极低（如调试时）可能导致日志噪声或误判；若请求量极高，打印频率仍足够，但需注意开启日志后

### [Test] Consolidate tool parser unit tests to tests/tool_parsers (#37834)
**SHA**: `3bbe2e1` | 🔗 [查看提交](https://github.com/vllm-project/vllm/commit/3bbe2e1e6e36c87b49eff3919c6bb1fa93df2a4d)

**🎯 变更类型**：重构（测试结构统一）  
**⚡ 重要程度**：🟢低  

**📋 变更摘要**：  
1. 将分散在 `tests/entrypoints/openai/tool_parsers/` 的工具解析器单元测试统一迁移到新的统一目录 `tests/tool_parsers/`，并删除冗余的 `conftest.py`。  
2. 新增、重命名了一批针对 **Granite‑4** 与 **Hermes‑2Pro** 解析器的测试文件，补全对流式与非流式解析行为的覆盖。  
3. 为 **Gigachat‑3** 等特定 parser 添加了 `default_tokenizer` fixture，避免跨测试用例的 tokenizer 状态相互污染。  

**🎯 影响范围**：  
- `tests/tool_parsers/` 目录下的所有测试文件  
- CI/CD 测试流水线（下载模型/Tokenizer 产生的网络 I/O）  
- 项目文档中关于测试目录结构的说明（若有）  

**🔍 技术洞察**：  
- **架构影响**：仅限测试层面，未触及库的运行时核心代码。统一测试目录有助于维护一致的 fixture 约定，提升可读性与可维护性。  
- **性能影响**：  
  - **正面**：删除了全局 `conftest.py`，减小了不必要的 fixture 加载范围；每个测试文件自行声明 tokenizer，避免不必要的重复实例化。  
  - **负面**：新增的测试在运行时会动态下载 `gpt2`、`ibm-granite/granite-4.0-h-tiny`、`Qwen/Qwen3-32B` 等模型，可能导致 CI 时间延长或出现网络波动。  
- **安全考虑**：无直接安全改动，唯一需关注的是模型下载过程的网络安全（使用官方模型仓库，风险极低）。  

**⚠️ 潜在风险**：  
1. **CI 依赖网络**：首次运行或缓存失效时，会从 HuggingFace 拉取模型，可能导致构建超时或因网络限制失败。  
2. **fixture 冲突**：虽然新增了 `default_tokenizer` 覆盖，但若其它测试仍引用已删除的全局 fixture，可能出现 `NameError`。  
3. **随机性**：`random_chunks` 使用 `random.randint`，若未固定随机种子，理论上可能出现偶发性失败；当前代码未显式设种子。  
4. **依赖版本**：测试直接导入 `transformers.AutoTokenizer`，若项目的 `transformers` 版本升级导致 API 变更，测试可能失效。  

**💡 关注建议**：  
- **CI 缓存**：在 CI 配置中开启模型缓存（如使用 `actions/cache`），确保模型只下载一次，降低网络波动风险。  
- **种子固定**：在使用 `random` 的测试文件顶部加入 `random.seed(0)`，确保随机切分行为可复现。  
- **依赖管理**：将 `transformers` 与 `huggingface_hub` 的最小兼容版本写入 `requirements-dev.txt`，防止上游升级导致突发性失败。  
- **文档更新**：在项目的贡献指南或测试说明中注明新的测试目录结构与 fixture 使用方式，帮助贡献者快速定位相关测试。  
- **监控 CI 时长**：观察合并后 CI 总时长是否出现显著增长，必要时考虑对大模型使用轻量化的 mock tokenizer 或预先打包好的 tokenizer 文件。

| **可维护性** | • **代码复用**：MusicFlamingo 通过继承与封装，减少重复实现，后续若 AudioFlamingo3 再有改动，MusicFlamingo 自动受益。<br>• **明确常量来源**：`MAX_AUDIO_LEN` 被移除，统一使用 `processor.max_audio_len`，降低硬编码风险。<br>• **测试覆盖率**：新增 150+ 条测试，覆盖占位符展开、RoTE、批量/单条一致性，降低回归风险。 |

### ⚠️ 潜在风险
1. **Transformer 兼容性**：最低版本要求提升至 5.3.0，使用低于该版本的环境会导致模型加载失败。建议在 CI/发布脚本中显式检查。  
2. **RoTE 时间戳维度**：`rote_timestamps` 必须与音频特征序列对齐，若用户自定义 `MultiModalKwargs` 丢失此字段，将抛出 `ValueError`。文档需明确说明。  
3. **内存峰值**：`MusicFlamingo` 的 rotary 时间嵌入在 `seq_len` 维度上会产生额外的 `cos/sin` 张量（大小 ≈ `batch * seq_len * head_dim`），在超长音频（>5 min）或大 batch 时可能导致 OOM。  
4. **Chunk 计数逻辑**：`_count_audio_tokens_from_mask` 依赖 `chunk_counts` 与 `feature_attention_mask` 的同步，如果用户自行构造 `mm_data` 而未提供 `chunk_counts`，会回退到旧的计数方式，可能产生不一致的 token 长度。  
5. **多模型并行（PP）**：虽然 `SupportsPP` 仍然声明，但新引入的 `pos_emb` 需在 pipeline 并行的每个阶段保持同步，若跨节点切分可能出现状态不一致。

- **并行子系统**：pipeline parallel（PP）路径、data parallel（DP）保持兼容。  
- **CUDA Graph 管理**：分段图 (piecewise) 在 PP 场景下可用。  

**💡 关注建议**  
1. **运行时验证**：在多卡/多节点环境开启 PP，分别测试 `is_first_pp_rank` 与 `is_last_pp_rank` 分支，确认 `intermediate_tensors` 正确复制并进入后续层。  
2. **内存管理**：持久化的 `intermediate_tensors` 会占用与最大捕获批次相同的显存，监控 `max_num_tokens` 设置，避免 OOM。  
3. **向后兼容**：旧版代码仍可在单卡或仅 DP 场景下运行，但请确保 `CUDAGraphMode.PIECEWISE` 与 `use_pp=False` 的路径保持一致。  
4. **调试日志**：建议在 `cudagraph_utils` 中加入 `logger.debug`，输出当前 PP rank、是否首/尾以及捕获的 tensor 名称，便于定位图捕获失败的原因。  
5. **测试覆盖**：新增单元测试应覆盖：① 首 PP rank 完整输出；② 中间 PP rank 只返回 `IntermediateTensors`；③ 最后 PP rank 合并输出并触发 `kv_connector`。  

总体来说，此次改动为 VLLM 引入了在流水线并行下使用分段 CUDA Graph 的能力，提升了多卡推理的吞吐与延迟，但也对显存和调试复杂度提出了新的要求。
