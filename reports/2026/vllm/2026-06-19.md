### fix(anthropic): auto-detect template support for mid-conversation system messages (#46025)
**SHA**: `4ce2d01` | 🔗 [查看提交](https://github.com/vllm-project/vllm/commit/4ce2d0145312809ef6122ccb7be8ae7cafa462a9)

**🎯 变更类型**：Bug修复 / 功能增强  
**⚡ 重要程度**：🔴高  
**📋 变更摘要**：  
- 为 Anthropic 接口新增自动检测聊天模板是否强制系统消息必须位于对话开头的逻辑。  
- 当检测到此类限制时，运行时会将所有 **inline**（非首位）系统消息合并到统一的 `system` 块，以避免模板渲染错误。  
- 添加对应单元测试，验证三种场景（需要合并、无需合并、无模板默认合并）的行为。

**🎯 影响范围**：  
- `vllm/entrypoints/anthropic/serving.py`（核心请求转换与模板检测）  
- `tests/entrypoints/anthropic/test_anthropic_messages_conversion.py`（新增测试）  

---

**🔍 技术洞察**  

- **架构影响**  
  - **模块耦合度**：新增 `_detect_merge_inline_system` 静态方法，并在 `AnthropicServingMessages` 实例化时决定 `self._merge_inline_system` 标志，保持原有调用链不变，仅在构造阶段做一次判断，架构侵入度低。  
  - **职责划分**：检测逻辑放在服务层（entrypoint），而非模板层，符合“运行时自适应”原则，避免在用户侧强制配置。  
  - **向后兼容**：默认行为保持不变（若未提供模板或检测异常则保守合并），对已有部署的影响最小。  

- **性能影响**  
  - **初始化成本**：首次创建 `AnthropicServingMessages` 时会渲染一次模板（四条消息），使用 `jinja2.sandbox.ImmutableSandboxedEnvironment`，开销极低（毫秒级）。  
  - **运行时成本**：合并系统消息仅在检测到需合并时执行一次列表遍历，时间复杂度 O(N)（N 为消息数），对大批量请求影响可忽略。  
  - **缓存/缓存失效**：检测成功后 `self._merge_inline_system` 固定为布尔值，后续请求不再触发模板渲染，保持原有缓存友好性。  

- **安全考虑**  
  - 使用 **Jinja2 沙盒环境**（`ImmutableSandboxedEnvironment` + `loopcontrols` 扩展），避免模板代码执行任意 Python，符合安全最佳实践。  
  - 捕获 `jinja2.TemplateError`，在任何渲染异常时回退为保守合并策略，防止因模板错误导致服务崩溃。  

- **可维护性**  
  - 检测逻辑封装为纯函数，易于单元测试；新增的测试覆盖了核心分支，提升了代码可验证性。  
  - 代码改动局部，未大幅改动公共 API，文档更新需求仅在说明 “系统消息可能会被自动合并”。  

---

**⚠️ 潜在风险**  

1. **误判风险**  
   - 某些模板可能在特定内容下才抛出异常（例如条件判断依赖于 message 内容），导致检测结果与实际运行时不一致，进而错误地合并或不合并系统消息。  
2. **模板渲染副作用**  
   - 虽然使用沙盒，但若模板中引入自定义过滤器或全局函数（在用户自定义模板场景），可能在检测阶段触发未预期的异常。  
3. **计数/token 差异**  
   - 合并系统消息后，`system` 块的内容被拼接，可能在某些模型的 token 计数上与原始 “inline system” 行为产生细微差异，进而影响费用或上下文长度限制。  
4. **向后兼容性**  
   - 对于依赖 “mid‑conversation system” 能够直接出现的旧模型/部署，若检测误判为不需要合并，可能导致模板渲染错误并返回 500。  

---

**💡 关注建议**  

- **监控与告警**：在生产环境加入对 `self._merge_inline_system` 的日志曝光（info 级别），以及捕获 `jinja2.TemplateError` 的异常计数，快速定位误判问题。  
- **模板白名单**：对于已知需要强制系统首位的模型（如 Qwen），可在部署文档或配置文件中显式声明 `merge_inline_system=True`，避免检测过程的额外渲染。  
- **回归测试**：在 CI 中加入对不同模型（OpenAI、Claude、Qwen 等）的真实模板渲染测试，确保检测逻辑在真实模板下表现一致。  
- **文档更新**：在 Anthropic 接口使用说明中注明：“在使用自定义 chat_template 时，系统消息可能会被自动合并。若希望保留原始顺序，可通过 `merge_inline_system=False` 手动关闭检测”。  
- **安全审计**：定期审计自定义模板是否引入了非沙盒安全风险，尤其是用户可上传模板的场景。

### [Frontend] Add Streaming Parser Engine and new GLM4.7/GLM5.1/GLM5.2 Parser (#45915)
**SHA**: `6c379b9` | 🔗 [查看提交](https://github.com/vllm-project/vllm/commit/6c379b9e5439ae305913e4a87ebf2b2e816072b4)

**🎯 变更类型**：功能增强 / 重构 / 架构变更  
**⚡ 重要程度**：🔴 高  

**📋 变更摘要**  
1. 为 GLM‑4.7、GLM‑5.1、GLM‑5.2 系列模型新增 **Streaming Parser Engine**，实现统一的 XML‑style tool‑call 解析，支持增量流式输出。  
2. 新增 `glm47_moe` 解析器实现（`vllm/parser/glm47_moe.py`），并在 **ParserEngine** 框架下注册对应的 Reasoning 与 Tool 适配器。  
3. 将原先的 `glm4_moe_tool_parser`（基于手写正则和差分逻辑）删除，改为继承自 `Glm47MoeParserToolAdapter`，实现同一套引擎代码的复用。  
4. 更新注册表、工具解析器映射表、推理适配表以及大量单元测试，使 GLM‑4.5 也复用 GLM‑4.7 的解析实现，实现向后兼容。  

**🎯 影响范围**  
- **parser/engine**：新增 `glm47_moe` 适配器、配置及解析实现。  
- **parser**：`registered_adapters.py`、`reasoning/__init__.py`、`tool_parsers/__init__.py`、`tool_parsers/glm47_moe_tool_parser.py` 等核心入口被更新。  
- **tests**：大量测试文件迁移到新 parser，实现对 stream‑arg‑deltas 的覆盖。  
- **用户代码**：使用 `tool_parser="glm45"` / `glm47` 的调用不再需要改动，内部统一走 `Glm47MoeParser`。  

---

## 🔍 技术洞察

| 维度 | 影响分析 |
|------|----------|
| **架构影响** | • 引入 **ParserEngine** 的声明式配置 (`glm47_moe_config`) ，将原本散落在 `Glm4MoeModelToolParser` 的状态机、正则、差分逻辑统一到引擎内部。<br>• 通过 `make_adapters` 自动生成 *Reasoning* 与 *Tool* 适配器，降低手写 Boilerplate，提升可维护性。<br>• 原有 `Glm4MoeModelToolParser` 被废弃，统一使用 `Glm47MoeParserToolAdapter`，实现 **模型复用**（GLM‑4.5 → GLM‑4.7 parser）。 |
| **性能影响** | • 新引擎在每个流式 `delta` 时 **重新解析** 当前完整文本，避免手动维护大量内部缓冲状态（`prev_tool_call_arr`、`streamed_args_for_tool` 等），代码更简洁。<br>• `stream_arg_deltas=True` 使引擎只返回 **增量** JSON 片段，理论上与旧实现的差分相当，但实际性能取决于正则匹配与状态机切换的开销。<br>• 由于使用 `regex`（而非 Python 标准 `re`），匹配速度略有下降，但库对 Unicode、嵌套等更稳健，整体延迟仍在毫秒级。 |
| **安全考虑** | • 解析器仍基于 **正则** 来抓取 `<arg_key>` / `<arg_value>`，如果攻击者发送极长且构造恶意嵌套的 XML（如 `"<arg_key>" * 10⁶`），可能触发 **ReDoS**（正则回溯）风险。`regex` 库对回溯有一定保护，但仍建议在生产环境对输入长度做上限。<br>• 解析后直接 `json.dumps`，不执行 `eval`，避免代码注入。<br>• 对 `tool_name` 与 `arg_key` 采用 `strip()`，防止潜在的空白注入。 |
| **可维护性** | • 通过配置化的 `ParserEngineConfig`，新增模型只需定义 **terminals**、**transitions** 与 **arg_converter**，极大降低后续模型适配工作量。<br>• 测试覆盖从 **reasoning**、**tool extraction**、**streaming** 多维度，保证改动不易回归。<br>• 旧实现的 500 行代码被删除，代码基线更小、更易审计。 |
| **兼容性** | • `ToolParserManager` 中已将 `"glm45"` 与 `"glm47"` 映射到同一实现，保证现有使用 GLM‑4.5 的用户无需迁移。<br>• `is_reasoning_end`、`extract_content_ids` 为兼容老模型提供了 **思考模式开关**，旧模型如果关闭 `thinking` 仍保持原有行为。 |

## ⚠️ 潜在风险

1. **正则回溯攻击**  
   - `_ARG_RE` 与 `_PARTIAL_ARG_RE` 对输入中大量嵌套 `<arg_key>`、`<arg_value>` 仍会进行全局搜索。若外部系统允许用户直接控制模型输出（如攻击者注入恶意 prompts），可能导致 CPU 资源耗尽。  
2. **流式增量失效**  
   - `stream_arg_deltas=True` 依赖 `arg_converter` 对 **部分** XML 进行正确的 JSON 片段生成。如果 `<arg_value>` 中出现未闭合的引号或特殊字符，增量 JSON 可能变得**不可解析**，导致前端渲染错误。  
3. **思考(TINKING) 模式开关不一致**  
   - `thinking_enabled` 默认取自 chat_template 参数。若下游产品在运行时修改模板而未同步 `chat_template_kwargs`，可能出现 **reasoning 输出被错误截断** 的情况。  
4. **工具 schema 不匹配**  
   - `arg_converter` 只负责把 `<arg_key>`/`<arg_value>` 组合成字典，不对 **工具定义的 JSON schema** 做验证。若模型输出的键名与注册工具不匹配，调用方仍会收到错误的 tool call，需在上层做容错。  
5. **多模型共用同一 Parser 实例**  
   - 在同一个进程中若并发请求使用不同模型（GLM‑4.5 vs GLM‑5.2）但共用同一 `Glm47MoeParser` 实例，内部的 `thinking_enabled` 状态可能被意外覆盖。当前实现通过实例化时捕获 `thinking`，但若实例被复用，需要确认 **实例生命周期** 与请求绑定。

### [12/n]  final _C library kernel migration (#45415)
**SHA**: `b9a7cd4` | 🔗 [查看提交](https://github.com/vllm-project/vllm/commit/b9a7cd464c9ae9b1b450f8982b76d7be4de73724)

**⚡ 重要程度**：🟡 中  

**📋 变更摘要**  
- 将大量 CUDA/HIP kernel 从旧的 `_C` 扩展迁移到 `_C_stable_libtorch`（基于 Torch‑stable ABI），并把仅 ROCm‑only 的代码保留在 legacy `_C`。  
- 新增可选扩展 **`_qutlass_C`**，负责 QUTLASS 相关的注册逻辑。  
- 统一头文件路径（`libtorch_stable/...`），并在 CMake 中引入 CUTLASS v4.4.2 的获取/使用。  
- 在 Python 层 `vllm.platforms.cuda` 中加入对 `_qutlass_C`、`_C_stable_libtorch`、`_moe_C_stable_libtorch` 的安全导入。  
- 新增 `weak_ref_tensor`、`persistent_masked_m_silu_mul_quant`、`silu_and_mul_quant` 等 stable‑ABI 接口，并在 `torch_bindings.cpp` 中重新注册。  

**🎯 影响范围**  
- **构建系统**：`CMakeLists.txt`、`cmake/external_projects/qutlass.cmake`、`setup.py`。  
- **核心 C++ 实现**：`csrc/libtorch_stable/*`（quantization、moe、fp4、cutlass 扩展等）。  
- **Python 平台层**：`vllm/platforms/cuda.py`。  
- **对外 API**：新增/迁移的 Torch ops (`silu_and_mul_quant`, `persistent_masked_m_silu_mul_quant`, `weak_ref_tensor`)。  

**💡 关注建议**
