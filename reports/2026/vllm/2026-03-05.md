### [Kernel][Mamba] Optimize Mamba2 SSD prefill Triton kernels (#35397)
**SHA**: `7faba50` | 🔗 [查看提交](https://github.com/vllm-project/vllm/commit/7faba503c403bc8c562888df3a841b6df104d042)

**🎯 变更类型**：性能优化 / 重构  
**⚡ 重要程度**：🔴高  
**📋 变更摘要**：  
1. 新增 `triton_helpers.fast_exp`，用硬件 `exp2` 替代 `tl.exp`，降低函数调用开销。  
2. 在 Mamba‑SSD 相关 Triton kernels（`mamba_ssm`, `ssd_chunk_scan`, `ssd_chunk_state`, `ssd_state_passing`）统一改用 `fast_exp`。  
3. 为 `ssd_chunk_scan`、`ssd_chunk_state` 等 kernel 重新设计 autotune 配置，加入更多 warp‑数、块大小及 `num_stages` 组合，以提升不同 `hdim/dstate` 场景的并行度与寄存器压力平衡。  
4. 重构状态传递 kernel：去除对 `seq_idx`、`cu_chunk_seqlens` 的依赖，改为基于 `last_chunk_indices` 的直接范围定位，实现 **branch‑less** 的序列级并行。  
5. 删除了旧的变长 `chunk_state_varlen` 实现（已注释），降低代码维护成本。  

---

### 🎯 影响范围
- **核心计算模块**：`vllm/model_executor/layers/mamba/ops/*`（Mamba SSD 前缀的所有 Triton kernels）。  
- **辅助工具**：新增 `triton_helpers.py`。  
- **上层调用**：`ssd_combined.py` 中的状态传递逻辑也随之更新。  

---

### 🔍 技术洞察

| 维度 | 影响描述 |
|------|----------|
| **架构影响** | - 通过 `fast_exp` 减少了 Triton 对 libdevice `__nv_expf` 的调用路径，保持在同一 kernel 中完成指数运算，降低了 kernel 启动与函数调用开销。<br>- 重构后的 state‑passing kernel 直接依据 `last_chunk_indices` 计算每条序列的 chunk 范围，实现 **每条序列独立的并行**，不再需要跨序列的条件分支（`seq_idx` 判断），简化了数据流。 |
| **性能影响** | - `fast_exp` 只需一次 `exp2` PTX 指令，理论上比 `tl.exp` 快 **≈2–3×**（省去 libdevice 包装层）。<br>- 新增的 autotune 配置覆盖了 **小 dstate / small head_dim** 场景（8‑16 warps、低寄存器压力）以及 **大 dstate** 场景（单 stage、低 warp），提升了在不同硬件（A100、H100、RTX 4090 等）上的吞吐与延迟隐藏。<br>- 删除 `chunk_state_varlen`（未被外部调用）并把相关逻辑合并到主 kernel，降低了代码体积与编译时间。 |
| **安全考虑** | - 只涉及数值计算的内部实现，无外部 I/O，未引入新安全威胁。<br>- 需关注 `exp2` 在极端输入（大负数/大正数）时的溢出/下溢行为，与 `tl.exp` 是否保持相同的 NaN/inf 语义。若差异出现，可能导致后续数值不稳定。 |
| **可维护性** | - `fast_exp` 抽象为单独函数后，后续若需改为更精确或带限幅的实现，只需在一处修改。<br>- `state_passing` 的参数从 `seq_idx / cu_chunk_seqlens` 简化为 `last_chunk_indices`，接口更直观，降低了调用方的准备工作。<br>- 大量自动调优配置写在同一文件，易于根据硬件特性裁剪或新增。 |

---

### ⚠️ 潜在风险

1. **数值精度差异**  
   - `exp2(x·log2(e))` 与 `exp(x)` 在 **极端** `x`（>~88 或 <-~87）会产生不同的上溢/下溢行为。若模型在这些范围上有敏感的梯度或状态累积，可能出现 **inf / 0** 与原实现不一致的情况。  
2. **硬件兼容性**  
   - `tl.math.exp2` 依赖 PTX `ex2.approx.f32`，在极旧的 GPU（如 compute capability < 7.0）上可能不支持或表现为低精度实现。  
3. **Autotune 配置体积**  
   - 新增的 12+ 配置会导致 **编译时间** 与 **二进制体积** 增大，尤其在 CI 环境或低内存 GPU 上可能触发 OOM。  
4. **删除的 `chunk_state_varlen`**  
   - 虽然当前代码库未直接调用该函数，但外部插件或自定义扩展可能仍依赖它，导致隐藏的兼容性回归。  

---

### 💡 关注建议

1. **全面数值回归**  
   - 在单元测试中加入 **极端 dt / A** 输入（正负 100 以上）以及 **梯度检查**，确保 `fast_exp` 与 `tl.exp` 在 FP32 范围内误差在可接受阈值（如 < 1e‑6）。  
   - 若发现溢出差异，可在 `fast_exp` 中加入 **范围裁剪**（如 `tl.where(x > MAX, INF, ...)`），保持行为一致。  

2. **硬件兼容性检测**  
   - 在初始化阶段检测 `torch.cuda.get_device_capability()`，如果低于 7.0，回退到 `tl.exp`（可在 `fast_exp` 内部实现条件分支）。  

3. **调优配置管理**  
   - 对常用硬件（A100, H100, RTX 4090）进行 **基准测试**，挑选最优的几组配置并在 `triton.autotune` 中保留，以避免不必要的配置膨胀。可考虑使用 `triton.ConfigPriority` 或手动剔除低效组合。  

4. **文档与 API 兼容**  
   - 在 `ssd_state_passing` 的函数注释中明确说明 **`last_chunk_indices`** 替代 **`seq_idx` / `cu_chunk_seqlens`**，并提供迁移示例。  
   - 为防止外部插件失效，建议在 `__init__.py` 中保留一个 **轻量包装**（调用内部实现），或在 `vllm` 的 changelog 中标记为 **破坏性变更**。  

5. **监控生产环境**  
   - 部署后通过监控指标（GPU 利用率、Kernel 执行时间、错误率）对比改动前后，确保 **latency** 与 **throughput** 有实质提升且 **数值异常**（如 NaN、inf）未出现。  

---

**结论**：本次提交通过引入硬件加速的指数函数、细化 autotune 配置以及重构状态传递逻辑，显著提升了 Mamba‑SSD 前缀的执行效率并简化了代码路径。只要在极端数值范围、旧硬件兼容性以及配置管理方面做好验证与防护，整体风险可控，预计对大模型推理吞吐有 **显著正向** 影响。

**⚡ 重要程度**：🔴 高

- 新增 `vllm launch render` 子命令，实现 **GPU‑less** 的渲染（tokenization / prompt 渲染 / detoken化）服务，专用于前置/后置处理而不进行模型推理。

- `vllm.entrypoints.cli.launch`（全新模块）

- `vllm/v1/engine/launch.py`（新 EngineClient 实现）  
- 测试套件：`tests/entrypoints/openai/*`、`tests/entrypoints/test_launch_cli.py`、`tests/utils.py`  
- CLI 主入口 `vllm.entrypoints.cli.main`（注册 launch 模块）

1. **一次性编译**：`PiecewiseCompileInterpreter` 只负责运行 split‑graph，随后 `PiecewiseBackend.__init__` 直接调用 `compile_all_ranges()`（或在 warm‑start 时调用 `load_all_ranges()`），把所有 `RangeEntry` 在模型首次调用前全部编译完成，去除了原来的 `to_be_compiled_ranges` 及编译完成回调。  
2. **缓存保存**：在所有子图编译完后立即把 `compiler_manager` 缓存落盘，并在耗时 > 1 s 时打印日志。  
3. **监控信息**：`monitor.end_monitoring_torch_compile` 的日志文字改为同时包含 profiling 阶段。  
4. **去除 contextvar 回调**、`enable_python_dispatcher` 等旧实现，改为在 `decorators.py` 中直接调用 `save_aot_compiled_function()`。  
5. 测试相应更新：编译次数从 5→6、结构化日志 artifact 数量从 2→4。

**可能风险**  
- **启动时编译开销**：一次性编译所有范围会在首次调用前产生一次性的长时间阻塞，可能导致服务器启动延迟。  
- **内存占用**：一次性生成所有子图的 compiled runnable，尤其在大模型或大量 compile‑size 时，可能显著增加峰值内存。  
- **错误恢复**：若某个 range 编译失败，当前实现会在 `__init__` 抛异常，导致整个模型无法使用；缺少局部回滚或降级路径。  
- **并发安全**：`compiler_manager.save_to_file()` 在多实例同时创建时可能出现竞争，需要确认文件锁或单例使用。

**建议**  
1. 为一次性编译提供可选开关（如 `vllm_config.compile_lazy=False`），在资源受限的环境下仍可保持原有 lazy 行为。  
2. 在 `compile_all_ranges` 前后记录时间、内存，便于监控启动成本。  
3. 对每个 range 编译加入异常捕获，记录失败并回退到 lazy 编译或使用已有缓存。  
4. 在保存缓存前加入文件锁或使用 `tempfile`‑rename 再写入的原子操作，防止并发写冲突。  
5. 更新文档，说明新行为对启动时延的影响以及如何通过配置调节。  

总体来看，改动实现了“停止懒编译”，让编译过程更可预测，但需要关注启动时资源消耗和异常处理，以免在生产环境引入不可接受的延迟或崩溃。
