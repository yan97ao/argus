### [Model] Add HyperCLOVAX-SEED-Think-14B language model support (#37107)
**SHA**: `2390d44` | 🔗 [查看提交](https://github.com/vllm-project/vllm/commit/2390d44209d0dc8d9c52c5e05e9d57407d57b1d6)

**🎯 变更类型**：功能增强  

**⚡ 重要程度**：🔴高  

**📋 变更摘要**  
1. 在 vLLM 中新增对 HyperCLOVAX‑SEED‑Think‑14B 模型的完整支持，包括模型实现、配置、注册以及文档说明。  
2. 完善对应的单元测试，使 CI 能够在具备 32 GB GPU 的环境下验证模型的加载与推理。  
3. 更新模型注册表和 `supported_models.md`，确保用户能够在 `vllm` 命令行/API 中直接选用该模型。  

**🎯 影响范围**  
- `vllm/model_executor/models/hyperclovax.py`（模型实现）  
- `vllm/transformers_utils/configs/hyperclovax.py`（模型配置）  
- `vllm/model_executor/models/registry.py`（模型注册映射）  
- `vllm/transformers_utils/configs/__init__.py`（配置导入）  
- `docs/models/supported_models.md`（文档）  
- `tests/models/language/generation/test_common.py`、`tests/models/registry.py`（CI 测试）  

**🔍 技术洞察**  

- **架构影响**  
  - 新增 `HyperCLOVAXForCausalLM`，实现了 `SupportsLoRA` 与 `SupportsPP` 接口，完整兼容 vLLM 的 LoRA、流水线并行（PP）以及张量并行（TP）机制。  
  - 采用 **MuP（μ‑P）** 方案对 embedding、残差、logits 进行倍率缩放，保持与原始 HyperCLOVAX 训练尺度一致。  
  - 引入可选的 **后置归一化（post‑norm / Peri‑LN）**，通过 `use_post_norm` 控制，兼容原模型的双层归一化设计。  
  - 通过 `make_layers`、`make_empty_intermediate_tensors_factory` 等通用工具实现层级切分，保持与其它模型（Llama、InternLM 等）一致的分布式执行路径。  

- **性能影响**  
  - 结构上与 Llama 系列相近，核心算子（QKV、GEMM、RMSNorm）均走张量并行路径，理论上不引入额外的计算瓶颈。  
  - 额外的 **post‑norm** 会产生一次额外的 RMSNorm，但其开销相对于整体自注意力/MLP 前向仍然有限。  
  - `attention_multiplier`、`residual_multiplier` 的 MuP 缩放对数值稳定性有正面作用，可能略微提升收敛速度和推理数值精度。  
  - 若开启 **dual‑chunk attention**（`dual_chunk_attention_config`），会在长序列（>8192）场景下提升显存利用率；否则保持原有 RoPE 实现。  

- **安全考虑**  
  - 权重加载逻辑同其他模型保持一致：  
    - 跳过 `rotary_emb.inv_freq`、`rotary_emb.cos_cached`、`rotary_emb.sin_cached`（防止不兼容的 ColossalAI 检查点泄漏）。  
    - 对量化尺度（`scale`、`zero_point`）做名称重映射，防止误加载。  
    - 使用 `trust_remote_code=True` 与模型注册保持一致，仍需依赖 HF 官方的安全审计。  
  - 未引入新的网络请求或代码执行路径，安全风险可视为与已有模型相同。  

**⚠️ 潜在风险**  

1. **权重映射兼容性**  
   - `stacked_params_mapping` 只覆盖了 `q/k/v` 与 `gate/up` 两类堆叠权重。若模型 checkpoint 中出现额外的 bias 或自定义层（例如 GPTQ 专用 bias），可能被误跳过或导致加载失败。  

2. **分布式切分边界**  
   - 当 `num_key_value_heads` 与 TP 规模不匹配时，代码在 `HyperCLOVAXAttention` 中进行 **复制** 或 **跨 GPU 分片**，但未显式验证 `tp_size % num_kv_heads == 0` 场景的边界条件，可能在极端 TP 配置下触发断言或不均衡的显存分配。  

3. **默认配置不一致**  
   - `HyperCLOVAXConfig` 默认 `max_position_embeddings=2048`，而模型实现若未显式覆盖会使用 `8192`（在 `HyperCLOVAXAttention` 中硬编码），可能导致文档与实际行为不匹配。  

4. **测试覆盖不足**  
   - 新增的测试仅在 GPU（≥32 GB）上运行，缺少 CPU/单卡、TP/PP 多节点的回归验证，导致在不同硬件环境下潜在的兼容性问题未被捕获。  

**💡 关注建议**  

1. **权重加载验证**  
   - 在本地或 CI 中使用官方 `naver-hyperclovax/HyperCLOVAX-SEED-Think-14B` checkpoint 完整跑一遍 `vllm` 的 weight‑loading 流程，并对比与原 HuggingFace `transformers` 实现的输出差异（如 L2 范数）。  

2. **加强分布式容错**  
   - 为 `HyperCLOVAXAttention` 添加明确的错误提示（如 “num_key_value_heads 与 TP 不兼容”），并在 `make_layers` 之前加入配置校验函数，以免在生产环境中因断言导致作业挂掉。  

3. **统一配置默认值**  
   - 将 `HyperCLOVAXConfig.max_position_embeddings` 的默认值提升至 `8192`，或在 `HyperCLOVAXAttention` 中使用 `config.max_position_embeddings`，避免硬编码产生的隐式不一致。  

4. **扩展测试矩阵**  
   - 增加 CPU、单卡（<32 GB）以及多节点 TP/PP 场景的单元测试；同时加入对 `use_post_norm=False`、`dual_chunk_attention_config` 等可选特性的回归检查。  

5. **文档同步**  
   - 在 `docs/models/supported_models.md` 中注明该模型的 **MuP** 参数需求（embedding_multiplier、logits_scaling、attention_multiplier、residual_multiplier）以及 **post‑norm** 的默认开启情况，帮助用户在自行微调或部署时避免误用。  

6. **监控显存与吞吐**  
   - 建议在正式部署前通过 `vllm.run_benchmarks` 对

总体来看，此次改动仅是清理未使用的参数，风险低，完成后编译日志将更干净。若全部调用已同步，直接合并即可。

2. **向后兼容**  
   - `current_platform.is_zen_cpu()` 在基类默认返回 `False`，但在 `ZenCpuPlatform` 中覆盖为 `True`。若外部代码直接实例化 `CpuPlatform`（例如单元测试），会得到错误的 `False`。可以在 `Platform` 基类实现 `is_zen_cpu` 为 `False` 并在 `__init__` 中根据检测动态切换类，或在 `cpu_platform_plugin` 直接返回 `ZenCpuPlatform` 实例。  

3. **权重预打包的安全性**  
   - `VLLM_ZENTORCH_WEIGHT_PREPACK` 默认 `True`，但在没有 `zentorch_weight_prepack_for_linear` 的情况下仍会尝试调用，导致 `AttributeError`。代码已做 `hasattr` 检查，但后续 `layer.weight` 被替换为预打包张量后，若用户后续手动修改权重（微调、LoRA）可能导致不一致。建议在 `dispatch_cpu_unquantized_gemm` 返回后记录 `layer._zen_prepacked = True`，并在权重被重新赋值时清除该标记或重新预打包。  

4. **Docker 镜像尺寸**  
   - `vllm-openai-zen` 通过 `uv pip install "vllm[zen]"` 拉入 `zentorch`，该包体积相对较大。若仅在 AMD CPU 环境使用，建议在 CI 中提供 `--platform=linux/amd64` 检查，或在文档中提醒用户根据实际需求选择 `vllm-openai` 或 `vllm-openai-zen`。  

5. **测试覆盖**  
   - 目前仅测试了平台检测与未量化 GEMM 调度。建议补充：  
     - 有 `zentorch` 实际可用时的端到端推理基准。  
     - `VLLM_ZENTORCH_WEIGHT_PREPACK=0` 时权重不被预打包的行为。  
     - `remove_weight=True` 后对 `layer.weight` 进行后续访问的异常路径。  

6. **文档更新**  
   - 需要在 README / docs 中加入 “Zen CPU 支持” 部分，说明安装方式、环境变量、已知限制（仅在 `amd64` 平台、需要 AVX‑512、`zentorch` 发行版兼容性）。  

总体来看，此次提交为 AMD Zen CPU 引入了完整的后端路径，代码结构清晰且有基本测试。只要在平台检测、预打包状态管理以及文档/镜像体积方面做好细节把控，即可平稳上线。
