### [Model] Add MiMo-V2.5 support (#40967)
**SHA**: `c245d35` | 🔗 [查看提交](https://github.com/vllm-project/vllm/commit/c245d35ff467bb3e9a73fcb3c4b02e6c7a3d2964)

**🎯 变更类型**：功能增强 / 新增模型 / 多模态支持  
**⚡ 重要程度**：🔴高  

**📋 变更摘要**  
- 为 vLLM 新增对 MiMo‑V2.5 系列模型的完整支持，包含 **MiMoV2Pro、MiMoV2‑Omni** 两大架构以及对应的 **MTP（Multi‑Token Prediction）** 推理实现。  
- 新增 **MiMo 音频 Tokenizer / 音频 Encoder**（代码量 ~1 400 行），实现音频的 VQ 编码、向量量化、时序对齐并可与视觉特征融合。  
- 引入 **MiMo‑Omni 视觉 Transformer**，实现 2D‑Rope、窗口注意力、空间‑时间 Patch‑Merge、SW‑A 重排等专属算子。  
- 在模型注册、配置转换、KV‑Cache 规格、推理调度、编译装饰等中心模块中加入对应条目，使新模型能够在现有 vLLM 框架下使用 **TP、PP、量化、Speculative Decoding** 等特性。  
- 新增 **MiMo‑Omni Processor**（≈1 300 行）实现从 HF‑style `Processor` 到 vLLM `ProcessorMixin` 的兼容层，支持图片、视频、音频以及 Video‑Audio 混合输入的统一预处理、时间戳插入、占位符替换等逻辑。  

**🎯 影响范围**  
- **模型实现**：`vllm/model_executor/models/mimo_v2.py`、`mimo_v2_mtp.py`、`mimo_v2_omni.py`、`mimo_audio.py`。  
- **注册表**：`vllm/model_executor/models/registry.py`、`vllm/model_executor/models/__init__.py`。  
- **配置转换**：`vllm/transformers_utils/model_arch_config_convertor.py`（新增 `MimoV2*` Convertor）。  
- **KV‑Cache**：`vllm/v1/core/kv_cache_utils.py`（加入 `head_size_v`、`kv_quant_mode`）。  
- **多模态 Processor**：`vllm/transformers_utils/processors/mimo_v2_omni.py`、`__init__.py`。  
- **Speculative Decoding**：`vllm/v1/spec_decode/llm_base_proposer.py`（加入 Omni 支持）。  
- **文档、测试、示例**：`docs/models/supported_models.md`、`tests/models/registry.py`。

### [Perf] FP8 FlashInfer Attn for ViT (#38065)
**SHA**: `5d5c776` | 🔗 [查看提交](https://github.com/vllm-project/vllm/commit/5d5c7764446da0d4888add9a060604e376e4e856)

**🎯 变更类型**：功能增强 / 性能优化 / 架构变更  
**⚡ 重要程度**：🔴 高  

**📋 变更摘要**  
本次提交为 vLLM 引入了 **FP8 量化的 ViT 编码器注意力**（仅在 Qwen3‑VL 系列模型上可用），通过 FlashInfer cuDNN 后端在前向计算时将 Q/K/V 实时转为 FP8 后再进入 cuDNN 核心注意力，实现 **1.1‑1.4×** 的算子加速。实现包括：  
1. 配置层新增 FP8 相关选项、校准/保存 scale 文件的 CLI 参数。  
2. 对 `MMEncoderAttention` 添加 FP8 动态/静态标定逻辑、量化历史缓存、自动保存。  
3. 新增 **Trition** 量化 kernel（支持 head_dim 按 16 对齐的 stride‑aware 量化），并在 BF16 路径保持兼容。  
4. `vit_flashinfer_wrapper` 接口扩展以接受 Q/K/V 的 scale 与输出 dtype。  
5. 完整的单元/集成测试、基准脚本、文档以及安全检查。  

**🎯 影响范围**  
- `vllm/config/model.py`、`vllm/config/multimodal.py`、`vllm/engine/arg_utils.py`（新增 CLI 参数）。  
- `vllm/model_executor/layers/attention/mm_encoder_attention.py`（FP8 状态、scale 读取、动态标定、量化路径）。  
- `vllm/model_executor/models/vision.py`、`vllm/model_executor/models/qwen3_vl.py`（FP8‑padded hidden size 计算、cu_seqlens 逻辑分支）。  
- `vllm/utils/flashinfer.py`（平台/版本检测函数）。  
- `vllm/v1/attention/ops/vit_attn_wrappers.py`（wrapper 参数扩展）。  
- 新增 Triton 量化 kernel `vllm/kernels/triton/qkv_padded_fp8_quant.py`。  
- 文档、基准、单元测试、CI 覆盖。

## 🔍 技术洞察  

### 1. 架构影响  
| 维度 | 具体改变 | 影响评估 |
|------|----------|----------|
| **模块耦合** | `MMEncoderAttention` 现在依赖 `MultiModalConfig`、`QuantFP8`、Triton kernel以及全局自动保存状态。 | 增加了跨模块的隐式依赖（如全局 `_fp8_scale_save_path`），需要确保在多进程/多卡场景下的初始化顺序。 |
| **配置扩展** | 新增 5 个 multimodal 配置字段（dtype、scale_path、save_path、save_margin）以及对应 CLI 参数。 | 对现有 `EngineArgs`、`ModelConfig` 的向后兼容性良好，只要不显式传入新参数，行为保持不变。 |
| **后端切换** | 只有在 `AttentionBackendEnum.FLASHINFER` 且 `mm_encoder_attn_dtype="fp8"` 时才走 FP8 路径。 | 通过 `backend_patch`（单元测/基准）确保不影响其他后端（FA、FA‑V2 等）。 |
| **CUDA Graph** | 动态标定需要在每次前向读取 Tensor 最大值，**冲突于完整 CUDA‑Graph 捕获**（文档已提示）。 | 在使用全图 CUDA‑Graph 的生产部署时必须采用静态 scale。 |
| **全局状态** | 自动保存使用模块级全局变量 `_fp8_scale_save_path`、`_fp8_saved_scale_refs`。 | 在单进程单卡使用安全；在多进程（e.g., `torch.distributed`）或多实例时可能产生竞争条件，需要额外锁或进程内唯一路径。 |
| **兼容性** | 仅在具备原生 FP8（SM90）且 cuDNN≥9.17.1 的平台开放；对不满足条件的机器直接回退 BF16。 | 对现有用户无破坏，仅在满足硬件/软件前提时提供加速。 |

### 2. 性能影响  
| 场景 | 关键改动 | 预期收益 | 风险/开销 |
|------|----------|----------|-----------|
| **大分辨率图像（QHD/4K）** | FP8 量化 + cuDNN FlashInfer FP8 kernel（Tensor‑Core） | 1.08‑1.18×（端到端）<br>1.12‑1.42×（核心 kernel） | 量化 kernel 需额外 3 次 kernel launch（Q/K/V）+ pad/unpad，故在**小 seq_len**（HD、FullHD）可能出现 0.9‑1.0× 效率。 |
| **动态标定** | 每次前向 compute `amax` → 更新 3×scale（GPU 内存写） | 额外 0.1‑0.3 µs（几乎可忽略） | 在 CUDA‑Graph 完全捕获时不允许，且会导致 **额外显存占用**（3×`_FP8_AMAX_HISTORY_LEN` × 4 B）。 |
| **静态标定** | 省去动态 `amax` 计算，直接读取 pre‑calibrated scales | 与动态标定相同的算子时间，且无额外 CPU‑GPU 同步 | 需要用户提前跑 `bench mm-processor` 生成 scale 文件。 |
| **Memory** | FP8 输出 tensor 使用 **padded head_dim**（向上对齐至 16），额外 `head_dim_padding/16 * num_heads` 元素。 | 对 `head_dim=72` (padding to 80) 增加约 11% 的显存占用；对 `head_dim=128` 无惩罚。 | 大 batch/长 seq 时需评估显存预算。 |
| **Triton 量化** | Stride‑aware kernel 直接在原始 Q/K/V 视图上量化，无中间 copy。 | 量化时间约 0.5‑1.0 µs（相较于 `torch.to(fp8)`） | 需要 Triton 环境；若 Triton 不可用则回退到 `QuantFP8`（不做 pad）。 |

- **文件路径验证**：`MultiModalConfig` 现在在初始化阶段检查 `mm_encoder_fp8_scale_path` 是否存在、`mm_encoder_fp8_scale_save_path`

### [Bugfix] Fix k_norm weight sharding in MiniMaxM2Attention when total_num_kv_heads < tp_size (#38191)
**SHA**: `f8ac0c7` | 🔗 [查看提交](https://github.com/vllm-project/vllm/commit/f8ac0c7cf0e3d4ac8894346005bdffe3bd7bd378)

**🎯 变更类型**：Bugfix  
**⚡ 重要程度**：🟡 中  

**📋 变更摘要**  
1. 为 `MiniMaxText01RMSNormTP` 增加了可选的 `weight_shard_world_size` / `weight_shard_rank` 参数，并使用 `functools.partial` 把它们传递到 `weight_loader`，使权重切分能够脱离默认的 TP 世界大小。  
2. 在 `MiniMaxM2Attention` 中修正了 `k_norm` 的切分逻辑：当 `total_num_kv_heads < tp_size` 时，KV‑head 在 TP 维度上是复制的，原先按照 `tp_size` 切分会导致权重缺失。现在改为按实际 KV‑head 数量切分，并利用 `tp_rank // replicas` 计算对应的子块。  

**🎯 影响范围**  
- `vllm/model_executor/layers/mamba/linear_attn.py`（RMSNorm 自定义算子）  
- `vllm/model_executor/models/minimax_m2.py`（MiniMaxM2Attention 初始化）  
- 依赖 `MiniMaxText01RMSNormTP` 的其他模型（若有）也会受到权重加载行为的变化。  

**💡 关注建议**  
- **测试覆盖**：在不同的 Tensor Parallel (TP) 大小（如 1、2、4、8）和不同 `total_num_kv_heads` 组合下跑完整的推理/生成，确保 `k_norm` 权重不再出现 NaN 或维度不匹配。  
- **权重兼容**：旧模型权重仍使用默认 TP 切分方式，新增参数对向后兼容不产生影响；若自行调用 `MiniMaxText01RMSNormTP`，请显式传入 `weight_shard_world_size` 与 `weight_shard_rank`。  
- **文档更新**：在模型配置说明中标注 KV‑head 小于 TP 时的特殊切分行为，提醒用户无需手动修改权重文件。  
- **性能检查**：权重切分仅在加载阶段进行，推理路径未受影响，但请确认 `partial` 包装不会引入额外的函数调用开销。  

总体而言，此次修复解决了在高并行度下 MiniMax‑M2 模型因 KV‑head 复制导致的权重加载错误，提升了模型在多卡环境中的可靠性。后续可关注是否还有其他层的类似切分逻辑需要同步调整。

### Add system_fingerprint field to OpenAI-compatible API responses (#40537)

**🎯 变更类型**：功能增强  
**⚡ 重要程度**：🟡 中  

**📋 变更摘要**

### [torch.compile]: Disable Sequence Parallelism (SP) for piecewise compilation (#38373)
**SHA**: `32e4563` | 🔗 [查看提交](https://github.com/vllm-project/vllm/commit/32e45636e3d7e02615facc8c63645ce4ac1d7e11)

**🎯 变更类型**：功能增强 / Bug 修复  
**⚡ 重要程度**：🟡 中  

**📋 变更摘要**  
本次提交在 `torch.compile` 的全图与分块编译路径上做了统一约束：当 **不使用 Inductor 图分区（`use_inductor_graph_partition=False`）** 时，**序列并行（Sequence Parallelism，SP）以及基于它的 AsyncTP** 均被强制关闭。为防止错误使用，新增了断言与警告，并在配置初始化时自动清空 `splitting_ops`、切换 `cudagraph_mode` 为 `FULL`。相应的单元测试也加入了对该限制的验证。

**🎯 影响范围**  
- `vllm/config/compilation.py`：新增 SP 兼容检查、自动调整 `splitting_ops` 与 `cudagraph_mode`。  
- `vllm/config/vllm.py`、`vllm/v1/worker/utils.py`：在 SP 启用路径加入断言/警告，统一正向逻辑。  
- 编译 Pass：`sequence_parallelism.py`、`collective_fusion.py`、`async_tp.py`（间接）改为在全图编译下才可生效，未满足条件时直接 `assert`。  
- 测试：`tests/compile/*` 增加了对 “必须全图编译” 的异常检查，并相应修改了 `test_should_split`、`test_cudagraph_sizes_post_init` 等期望。  

**💡 关注建议**  
1. **使用者**：在开启 `pass_config.enable_sp`（或 `fuse_gemm_comms`）时，请确保 `compilation_config.use_inductor_graph_partition=True` 或显式将 `splitting_ops=[]`。否则配置会被自动降级为全图编译并关闭 SP，可能导致性能回退。  
2. **开发者**：若后续需要在分块编译下保留 SP，需要在 `SequenceParallelismPass` 中实现子图间残差张量的跨 TP 对齐，或者提供新的自定义 ops。当前实现已通过断言防止误用。  
3. **维护者**：关注日志中的两条一次性警告：① “Sequence parallelism requires full-graph compilation …”，② “Sequence parallelism is incompatible with piecewise cudagraph …”。它们帮助快速定位配置冲突。  
4. **回归测试**：新增的 `test_async_tp_pass_requires_full_graph_compilation`、`test_sequence_parallelism_pass_requires_full_graph_compilation` 已覆盖核心断言，保持现有 CI。若未来调整 `use_inductor_graph_partition` 的默认行为，请同步更新这些测试。  

总体而言，此次改动提升了 **配置一致性**，防止了在不兼容的分块编译模式下产生的图形错误或性能异常，对使用 SP/AsyncTP 的用户影响可控。建议在升级后检查配置文件或启动脚本，确保 `use_inductor_graph_partition` 与 SP 选项保持一致。
