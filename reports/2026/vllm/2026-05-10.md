### Add NVFP4 all-gather GEMM fusion for AsyncTP (#41882)
**SHA**: `bc5fdc1` | 🔗 [查看提交](https://github.com/vllm-project/vllm/commit/bc5fdc1e6a718aec4b76b47b69ec5e7cb3c2412a)

**🎯 变更类型**：功能增强 / 架构变更  
**⚡ 重要程度**：🔴高  

**📋 变更摘要**  
本次提交为 Blackwell GPU（SM‑100）上的 NVFP4 模型引入 **All‑Gather + GEMM 融合**，即在 AsyncTP（异步张量并行）与 Sequence Parallel（序列并行）路径中使用 FlashInfer‑FP4 实现的自定义算子 `fused_all_gather_flashinfer_fp4_matmul`。通过在编译阶段注册相应的 Fusion Pattern，使得原先的 All‑Gather → GEMM → Reduce‑Scatter 三段通信‑计算链路被一次性完成，从而显著降低通信开销并提升 BF16/FP4 混合精度推理的吞吐。  

**🎯 影响范围**  
- `vllm.compilation.passes.fusion.collective_fusion`（新增 FP4 融合算子及匹配模式）  
- `vllm.compilation.passes.fusion.sequence_parallelism`（新增针对 NVFP4 的 RMSNorm‑Quant 融合）  
- `vllm.utils.flashinfer`（新增 `flashinfer_scaled_fp4_mm_out` 对外 API）  
- 相关测试文件：`tests/compile/e2e/...`、`tests/compile/fusions_e2e/...`，以及 `test_sequence_parallel.py`  
- 运行时环境变量 `VLLM_NVFP4_GEMM_BACKEND`（控制后端）  

**🔍 技术洞察**  

- **架构影响**  
  - 引入 **自定义算子** `fused_all_gather_flashinfer_fp4_matmul` 并在 `direct_register_custom_op` 中注册，使得 Torch‑Inductor 在图优化阶段能够识别并替换相应的 All‑Gather + GEMM 组合。  
  - 在 `AsyncTPPass` 与 `SequenceParallelismPass` 中额外检测 `torch.ops.vllm.flashinfer_mm_fp4`，并依据后端（cutlass、cudnn、trtllm）和 Scale‑Factor 布局（8x4）注册多套 Pattern。  
  - 通过 `auto_functionalized` 与 `scaled_fp4_quant` 实现 **RMSNorm → FP4 量化 → All‑Gather** 融合，保持序列并行时的激活尺度同步，仅在 NVFP4 场景下开启。  
  - 仅在 **Blackwell（SM‑100）** 及拥有 FlashInfer 的环境下激活，使用 `current_platform.is_device_capability_family(100)` 与 `has_flashinfer()` 进行前置检查，防止在不兼容硬件上触发未实现路径。  

- **性能影响**  
  - **通信压缩**：All‑Gather 与 GEMM 合并后，通信只需一次 All‑Gather，后续 GEMM 直接在合并后的完整矩阵上完成，省去 `Reduce‑Scatter` 的二次同步。  
  - **算子调度**：使用 FlashInfer‑FP4 的专属后端（cutlass/cudnn/trtllm）可在张量形状较小（单卡 8‑16 维）时实现 **低延迟、低显存** 的矩阵乘法，尤其在 `tp_size=2`、`hidden_size=512` 场景下预计提升 1.2‑1.5×吞吐。  
  - **量化开销**：通过 `scaled_fp4_quant` 的 **视图转换**（float8_uint8、uint8）避免额外拷贝，进一步降低量化阶段的显存占用。  
  - **编译缓存**：为确保计数统计准确，`test_toy_llama.py` 中显式导入 `compilation_counter`，帮助 CI 监控因新 Fusion 带来的额外 Inductor 编译次数。  

- **安全考虑**  
  - 代码路径仅在检测到 FlashInfer 与相应硬件后才会运行，未满足条件时直接 `pytest.skip`，不影响已有用户的安全性。  
  - 通过环境变量 `VLLM_NVFP4_GEMM_BACKEND` 控制后端选择，若设置不当可能导致调用不存在的实现（如 typo），会抛出明确的 RuntimeError，未出现隐式安全风险。  
  - 仍依赖 FlashInfer 的内部实现，需关注其对显存溢出或数值异常的错误处理（暂未在本 PR 中加入额外校验）。  

**⚠️ 潜在风险**  
1. **硬件限制**：仅支持 Blackwell（SM‑100）及以上 GPU，若用户在旧 GPU（如 RTX 4090）上误设环境变量，可能出现 `Unsupported device` 错误。  
2. **后端兼容性**：`backend` 参数接受 `"cutlass"`、`"cudnn"`、`"trtllm"`，但不同后端对 Scale‑Factor 布局的实现细节不同，若 FlashInfer 升级导致接口变更，当前包装层可能失效。  
3. **量化尺度同步**：NVFP4 目前只在 All‑Gather 路径实现了尺度融合，Reduce‑Scatter 仍未实现对应 FP4 版本，若未来开启 **Reduce‑Scatter** 可能出现尺度不一致的隐蔽错误。  
4. **测试覆盖不足**：CI 只在多 GPU 环境（2 GPUs）下跑该测试，单卡或大规模（≥4 GPUs）场景的行为缺少验证。  
5. **代码路径分支**：新增 `SCALED_FP4_QUANT_OUT_OVERLOAD` 与 `SCALED_FP4_QUANT_DEFAULT_OVERLOAD` 的动态导入，若未来 PyTorch 删除这些符号会导致 ImportError。  

**💡 关注建议**  
- **文档与使用指南**：在 README 或 vLLM 文档中明确标注 “NVFP4 仅在 Blackwell GPU + FlashInfer 环境下可用”，并提供检查指令 (`torch.cuda.get_device_capability`) 与环境变量配置示例。  
- **回退策略**：在 `VLLM_NVFP4_GEMM_BACKEND` 默认值设为 `cutlass`，并在代码中捕获后端初始化异常，自动回退到普通 AsyncTP（不使用 FP4 融合），避免因后端不兼容导致整个推理失败。  
- **CI 扩展**：添加 **单卡**、**4 GPU**、以及 **不同后端组合**（cutlass / cudnn / trtllm） 的矩阵乘法对比基准，以验证在更大规模下的可扩展性和数值稳定性。  
- **监控显存与数值**：在 Fusion 实现中加入 `torch.cuda.memory_allocated` 与 `torch.allclose` 的可选 debug 打印，帮助定位潜在的显存泄漏或量化误差。  
- **兼容性检查**：在 `flashinfer_scaled_fp4_mm_out` 中加入对 `out_dtype` 与 `out.dtype` 不匹配的断言，防止用户误传 `torch.float32` 导致隐式类型转化。  

通过上述措施，可确保新加入的 NVFP4 All‑Gather + GEMM 融合在提升性能的同时保持稳定、可回退，并在后续迭代中逐步扩展至 Reduce‑Scatter 与更高规模的并行场景。

### [Performance] Make safetensors checkpoint prefetch settings configurable (#41499)
**SHA**: `21943d4` | 🔗 [查看提交](https://github.com/vllm-project/vllm/commit/21943d4c258983c4b8eb56d50029aca4f18e4629)

**🎯 变更类型**：性能优化  
**⚡ 重要程度**：🟡 中  

**📋 变更摘要**  
为 safetensors 检查点的预取机制新增可配置参数：`safetensors_prefetch_num_threads`（预取线程数）和 `safetensors_prefetch_block_size`（每次读取块大小）。这些参数在 `LoadConfig` 中加入，并通过 CLI、`EngineArgs` 与模型加载路径向下传递。实现上把原先硬编码的 8 线程、16 MiB 块改为可调，并使用 `ThreadPoolExecutor` 并行读取，以提升不同硬件/网络环境下的加载速度和内存占用可控性。  

**🎯 影响范围**  
- **vllm/config/load.py**：新增默认常量与两个字段。  
- **vllm/engine/arg_utils.py**：CLI 与 `EngineArgs` 同步新增参数。  
- **vllm/model_executor/model_loader/default_loader.py**：向 `safetensors_weights_iterator` 传递新配置。  
- **vllm/model_executor/model_loader/weight_utils.py**：实现可调的预取函数、线程池并加入参数校验及日志。  

**💡 关注建议**  
1. **兼容性**：默认值保持原行为，旧版脚本无需改动；若用户自行修改，请确保 `num_threads≥1`、`block_size≥1`。  
2. **资源监控**：在 I/O‑受限或内存紧张的机器上，适当降低线程数或块大小，以防止系统缓存压力过大。  
3. **文档更新**：在使用手册和 `--help` 输出中补充说明这两个新选项的意义与取值范围。  
4. **测试**：验证分布式环境下每个 rank 正确分配文件并在不同 `num_threads` 下的并发行为，防止潜在的 race 条件或线程泄漏。  

总体而言，此改动提升了 safetensors 加载的灵活性和可调性能，对用户体验和大模型部署都有正面影响。

### [DSV4] Add PP support for deepseek-v4 (#41694)
**SHA**: `f396bee` | 🔗 [查看提交](https://github.com/vllm-project/vllm/commit/f396bee56fb516195a3669e5499129021ced4443)

**🎯 变更类型**：功能增强（为 DeepSeek‑V4 添加 Pipeline‑Parallel (PP) 支持）  
**⚡ 重要程度**：🟡 中  

**📋 变更摘要**  
- 在 `deepseek_v4.py` 中引入 `SupportsPP` 接口，并在模型各层加入 PP 判断。  
- 首层（rank 0）负责词表嵌入，末层（rank‑1）负责 RMSNorm 与 lm_head；中间 rank 使用 `PPMissingLayer` 占位。  
- 为 PP 流添加 `make_empty_intermediate_tensors`，在非首 rank 通过 `intermediate_tensors["hidden_states"]` 传递隐藏状态。  
- 权重加载时通过 `is_pp_missing_parameter` 与 `PPMissingLayer` 跳过不存在的参数，避免跨 rank 的非法访问。  
- 文档表格对应项更新为 “✅︎” 表示已支持 PP。  

**🎯 影响范围**  
- `vllm/model_executor/models/deepseek_v4.py`（模型定义、前向、权重加载）  
- `vllm/model_executor/models/interfaces.py`（新增 `SupportsPP`）  
- `vllm/model_executor/models/utils.py`（新增 `PPMissingLayer`、`is_pp_missing_parameter`）  
- 文档 `supported_models.md`  

**💡 关注建议**  
1. **兼容性**：在单机/单卡运行时 `get_pp_group()` 仍会返回默认组，确保 `is_first_rank/last_rank` 正确为 True，防止 `PPMissingLayer` 被错误创建。  
2. **权重加载**：`is_pp_missing_parameter` 只在当前 rank 缺少的参数上 `break/continue`，需要确认所有被跳过的参数在其他 rank 能够完整加载，否则会出现不匹配。建议在加载完毕后加入完整性校验（如检查 `loaded_params` 与期望集合的差异）。  
3. **中间张量**：`make_empty_intermediate_tensors` 只在最后 PP rank 分配缓冲区，其他 rank 返回 `None`。调用方（调度器）需要在 PP 场景下显式传递 `intermediate_tensors`，否则会触发断言。建议在 `vllm/engine` 相关代码中加入相应的 guard。  
4. **测试**：新增 PP‑V4 测试用例，覆盖：单卡、二卡、四卡等不同 PP 拓扑；验证嵌入、norm、lm_head 只在对应 rank 实例化且前向输出一致。  
5. **文档**：说明在非 PP 环境时模型行为保持不变，并提供启动 PP 的命令行示例（`--tensor-parallel-size`、`--pipeline-parallel-size`）。  

整体来看，改动实现了 DeepSeek‑V4 的切片并兼容现有权重格式，关键在于确保跨 rank 参数分配与中间张量传递的正确性。后续通过充分的多卡 CI 与运行时检查即可稳固此功能。

**🎯 影响范围**

**💡 关注建议**  
1. **实现检查**：若项目另有自定义 `KVOffloadManager` 实现，需要同步添加 `req_context` 参数并在内部使用或安全忽略，防止运行时 `TypeError`。  
2. **上下文使用**：当前实现仅转发参数，后续可能在 `touch`、`complete_load/store` 中依据 `kv_transfer_params` 决定优先级或日志，建议提前在代码中保留 `req_context` 的引用，以便后续扩展。  
3. **测试覆盖**：已更新核心 CPU 测试，建议在多机分布式环境下跑一次完整的集成测试，确保调度器在 `update_connector_output` 中正确匹配 `req_status`。  
4. **性能评估**：`ReqContext` 为轻量对象，额外的参数传递对性能影响可忽略，但若在实现中加入 heavy‑logic，需监控 LRU 触发路径的时延。  

总体而言，此次改动为后续的请求级 KV 迁移策略提供支撑，只要各实现同步签名即可平滑升级。
