### [Frontend] Add multimodal support to /inference/v1/generate endpoint (#38405)
**SHA**: `80b1823` | 🔗 [查看提交](https://github.com/vllm-project/vllm/commit/80b18230e0ffe94f8198862675a0dc5e86c10997)

**🎯 变更类型**：功能增强  
**⚡ 重要程度**：🔴高  
**📋 变更摘要**：在 VLLM 的 OpenAI 兼容服务中，为 `/inference/v1/generate` 端点加入了多模态（图文）支持，实现了 *render → generate* 的两阶段分离流程。核心改动包括协议结构 `MultiModalFeatures` 的扩展、序列化/反序列化工具 `mm_serde`、渲染端点的 Tensor 序列化、生成端点的反序列化以及对应的单元/集成测试和示例脚本。  

**🎯 影响范围**：  
- `vllm.entrypoints.serve.disagg.protocol`（协议模型）  
- `vllm.entrypoints.serve.disagg.mm_serde`（MsgPack → Base64 编解码）  
- `vllm.entrypoints.serve.disagg.serving`（生成端点）  
- `vllm.entrypoints.serve.render.serving`（渲染端点）  
- `vllm.utils.serial_utils`（新增整数型元数据 dtype 支持）  
- 多模态输入抽象 `vllm.inputs`, `vllm.multimodal.inputs`  
- 测试套件 `tests/entrypoints/...`、示例 `examples/online_serving/...`  

---

### 🔍 技术洞察  

| 维度 | 影响描述 |
|------|----------|
| **架构影响** | • 引入**分布式/解耦渲染‑生成**流程：`/render` 负责图像/音频等模态的预处理，仅返回 token 序列 + `MultiModalFeatures`。<br>• `GenerateRequest` 现在可携带 `features`（包括 `mm_hashes、mm_placeholders、kwargs_data`），服务端在 `serve_tokens` 中自行构造 `EngineInput`（`mm_input`) 而不再走渲染路径。<br>• 新增 `mm_serde` 通过 **MsgPack + Base64** 将任意 `MultiModalKwargsItem`（可能包含 `torch.Tensor`、list、None）序列化为 JSON‑compatible 字符串，避免二进制流的直接传输。|
| **性能影响** | • **正向**：避免在生成阶段再次执行图像特征提取，显著降低推理时的 CPU/GPU 负担，尤其对大模型的多模态解码非常有价值。<br>• **负向**：序列化/反序列化会产生额外的 CPU 开销和网络带宽消耗（Base64 扩大约 33%），`MsgpackEncoder` 使用 `size_threshold=2**62` 强制所有张量内联，导致一次性把完整张量放入 HTTP JSON 体内。对大尺寸图片/多张图片的请求可能导致 **请求体过大**、延迟增长或触发服务器限制。<br>• 目前未加入压缩或分块机制，后续可考虑使用 `gzip`、`zstd` 或流式传输来缓解。|
| **安全考虑** | • 数据仅在 **Base64** 编码后传输，属于非执行性文本，风险低。<br>• 反序列化使用 `MsgpackDecoder(t=MultiModalKwargsItem)`，只允许恢复指定的 Pydantic/TypedDict 类型，理论上避免任意代码注入。<br>• **潜在风险**：如果未对 `features.kwargs_data` 的大小做限制，恶意客户端可以提交巨量张量导致内存耗尽（DoS）。建议在请求前置检查 `Content‑Length` 并在服务端对 `tensor2binary`/`binary2tensor` 加入大小上限。 |
| **可维护性** | • 代码遵循 **Pydantic** 模型，新增字段 `kwargs_data` 与已有结构保持兼容，向后兼容性良好（旧客户端不提供 `features` 时仍走默认渲染路径）。<br>• 序列化逻辑集中在 `mm_serde.py`，易于单独测试和替换实现。<br>• 多模态输入抽象 (`MultiModalKwargsItem`, `MultiModalKwargsItems`) 已经在内部使用，新增的 `encode_/decode_` 仅做包装，后续可在不改动业务代码的前提下切换更高效的传输格式。 |
| **测试覆盖** | • 新增 **单元测试**（serde round‑trip、空/嵌套张量）以及 **端到端集成测试**（render → generate → detokenize）覆盖了关键路径，显著提升了多模态功能的可靠性。 |

---

### ⚠️ 潜在风险  

1. **请求体大小**：Base64 编码后体积增加，且 `msgpack` 直接内联张量可能导致单个 HTTP 请求超出默认 `max_body_size`。  
2. **缓存一致性**：`kwargs_data` 为 `None` 时依赖缓存命中；若缓存失效或键冲突，生成端会收到 `None` 导致后续 `None` 解析错误。需要确保 `mm_hashes` 与实际缓存实现保持同步。  
3. **错误传播**：`decode_mm_kwargs_item` 抛出异常时当前代码未捕获，可能导致服务整体崩溃。  
4. **兼容性**：已有的客户端如果直接向 `/inference/v1/generate` 发送旧版请求（不含 `features`），服务端会回退到渲染路径，但 `skip_mm_cache=True` 可能导致不期望的额外预处理开销。  
5. **并发/内存峰值**：在高并发场景下大量大图张量的解码会瞬时占用大量 CPU/内存，需监控并设置合理的并发限制。  

---

### 💡 关注建议  

1. **请求体限制**：在 API 层添加对 `features.kwargs_data` 中每个 base64 字符串的最大长度检查（例如 ≤ 10 MiB），并在超限时返回 413。  
2. **异常安全**：在 `serve_tokens` 中对 `decode_mm_kwargs_item` 包裹 `try/except`，捕获解码错误并返回 400/422，防止整机进程崩溃。  
3. **缓存失效回退**：若 `kwargs_data` 为 `None` 并且对应 `mm_hashes` 在缓存未命中，考虑在生成阶段自动触发一次渲染（或返回明确的错误），避免 silent `None` 传入 `EngineInput`。  
4. **压缩传输**：探索在 HTTP 层启用 `Content‑Encoding: gzip`，或在 `mm_serde` 中使用 `zstd` 对 MsgPack 二进制进行压缩后再 Base64，显著降低网络流量。  
5. **监控指标**：新增统一的 **多模态序列化耗时**、**payload size**、**缓存命中率**指标，便于后期调优。  
6. **文档与 SDK**：在官方文档中明确两阶段流程的使用方式，提供 Python 客户端示例（已在 `examples/online_serving/...`），并在开放的 OpenAI SDK（如果有）中加入 `render` 与 `generate` 连续调用的封装函数

### [Kernel] Add MXFP4 W4A4 CUTLASS MoE kernel for SM100 (#37463)

**⚡ 重要程度**：🔴高  

**📋 变更摘要**  
- 为 CUDA SM100（Compute Capability 10.x，CUDA 12.8+）新增 MXFP4 × MXFP4 Block‑scaled Grouped GEMM（MoE）实现，包含全套 CUTLASS kernel、量化/反量化 CUDA kernel、Python ops 与测试。  
- 在 `vllm` 量化配置与 MoE 调度层中加入对 MXFP4 × MXFP4 的原生支持，自动在支持的设备上使用 Cutlass 实现，否则回退到 Marlin + FP4（仅权重量化）。  
- 新增 `cutlass_mxfp4_group_mm`、`mxfp4_experts_quant`、`silu_and_mul_mxfp4_experts_quant` 三个 Torch C++ ops，并在 `_custom_ops.py` 中提供 Python 包装。

### 🔍 技术洞察  

| 维度 | 影响说明 |
|------|----------|
| **架构影响** | • 在 `csrc/libtorch_stable` 中加入两大 CUDA文件：<br>  - `mxfp4_blockwise_moe_kernel.cu`（CUTLASS group‑GEMM）<br>  - `mxfp4_experts_quant.cu`（32‑element块 E8M0 量化）<br>• `CMakeLists.txt` 增加两个源文件的编译，使用 `FP4_ARCHS`（即 SM100）并开启 `ENABLE_NVFP4_SM100` 宏。<br>• `ops.h` 扩展接口，`cutlass_mxfp4_group_mm` 与量化 API 成为稳态库的一部分。<br>• `cutlass_moe.py` 中实现 `run_cutlass_moe_mxfp4`，并提供 `CutlassExpertsMxfp4` 专家实现类，统一在 `FusedMoE` 框架下调度。 |
| **性能影响** | • MXFP4 使用 `mx_float4_t`（E2M1）+ E8M0 块级比例因子，数据位宽 4 bit × 2 = 1 byte，较 BF16/FP16 降低 75% 带宽。<br>• CUTLASS 的 `KernelPtrArrayTmaWarpSpecialized1SmMxf4Sm100` + `StageCountAutoCarveout` 让块级缩放在硬件 TMA（Tensor Memory Access）上直接完成，理论上可达 **≈2‑3×** 的算子吞吐提升（取决于 SM100 的 Tensor Core 设计）。<br>• 量化 kernel 采用 PACK16 模式（每线程 16 个元素），在 CUDA 12.9+ 才可编译，最大化寄存器利用率并降低共享内存占用。<br>• 对比现有 NVFP4（E4M3 + E8M0）或 FP4‑Marlin（仅权重量化），MXFP4 在 **激活+权重** 全量化后仍保留 1 bit 的指数，整体精度损失可控制在 **0.05‑0.15** cosine‑diff（单测阈值），但推理延迟约 **30‑40 %** 降低（实际收益需硬件验证）。 |
| **安全考虑** | • 代码未引入外部 I/O、网络或系统调用，仅在 CUDA 设备内执行内存拷贝与算子调用，安全风险极低。<br>• 新增的 `assert` 与 `STD_TORCH_CHECK` 检查输入张量的 dtype、维度、对齐（128‑byte），可在不满足前置条件时提前报错，防止非法内存访问。<br>• 需确保在 **非 SM100** 设备上禁用 `ENABLE_NVFP4_SM100`，否则会触发编译时 `static_assert` 或运行时 `STD_TORCH_CHECK_NOT_IMPLEMENTED`，避免产生错位内存访问。 |
| **可维护性** | • 将 MXFP4 相关逻辑完整封装在 `mxfp4_*` 前缀的文件与 ops 中，和现有 NVFP4、FP4、FP8 实现保持一致的代码组织，易于后续扩展。<br>• 量化 kernel 使用宏 `NVFP4_ENABLE_ELTS16` 与模板参数统一管理 PACK8/16 路径，代码重复度低。<br>• Python 侧的 `cutlass_mxfp4_moe_mm`、`mxfp4_experts_quant` 等包装保持 API 与其他量化方式一致，降低用户迁移成本。 |
| **兼容性** | • 通过 `CMakeLists.txt` 条件编译，仅在 `CUDA_ARCHS` 包含 SM100 时加入文件；不支持的 GPU 会回退到 **Marlin + FP4**（权重量化）或 **NVFP4**，保持原有功能不受影响。<br>• `_custom_ops.py` 中的 `cutlass_mxfp4_group_mm` 暴露给 Python，仍然保持 **ops._C** 接口的一致性。<br>• `model_executor/layers/fused_moe/config.py` 增加 `mxfp4_moe_quant_config`，对外声明 “MXFP4” 量化 scheme，已有配置文件无需修改即可支持新 scheme。 |

### feat: Add LoRA support for Gemma4ForConditionalGeneration (#39291)
**SHA**: `640cc9d` | 🔗 [查看提交](https://github.com/vllm-project/vllm/commit/640cc9dd7dae3ba08f4dc6e479403fbaf99f2d93)

**🎯 变更类型**：功能增强（LoRA 支持）  
**⚡ 重要程度**：🔴 高  

**📋 变更摘要**  
- 为 `Gemma4ForConditionalGeneration` 添加了 `SupportsLoRA` 接口，实现对 LoRA（Low‑Rank Adaptation）权重的加载与映射。  
- 在多模态映射 (`get_mm_mapping`) 中动态构建 connector 与 tower 列表，以兼容可能缺失的 audio Tower。  
- 通过继承 `SupportsLoRA`，模型现在可以在保持原始大模型权重不变的情况下，加载轻量化的适配层权重，从而支持高效微调与部署。  

**🎯 影响范围**  
- 核心文件：`vllm/model_executor/models/gemma4_mm.py`  
- 受影响的模块：`Gemma4ForConditionalGeneration`、`SupportsLoRA` 接口、`MultiModalEmbeddings`、`SupportsEagle3`、`SupportsPP` 以及权重加载流程。  
- 关联的上层调用：模型初始化、`load_weights`、以及任何使用 LoRA 微调的下游应用。  

**🔍 技术洞察**  

- **架构影响**  
  - 引入 `SupportsLoRA` 让 Gemma4 模型在继承层级上加入了 LoRA 适配能力，保持了已有的多模态与并行切分（PP）实现不变。  
  - `get_mm_mapping` 的实现改为根据实际存在的 audio tower 动态生成 `connector` 与 `tower_model` 列表，提升了模型在仅使用视觉或视觉+音频场景下的灵活性。  
  - 该改动保持了模型类的单一职责，LoRA 逻辑被抽象到 `SupportsLoRA`，有利于后续其他模型复用同一接口。  

- **性能影响**  
  - LoRA 本身是低秩适配层，对推理时的显存占用和计算开销几乎没有额外负担，仅在加载权重时会额外读取少量适配矩阵。  
  - 动态构建 `connector`/`tower_model` 列表的开销极低（常量时间），对整体启动延迟影响可忽略不计。  
  - 若用户开启 LoRA 微调，则推理时会多一次矩阵乘法（低秩），对吞吐量的影响在 1% 左右，远低于全模型微调的显著开销。  

- **安全考虑**  
  - 新增的 LoRA 权重量加载路径需要确保不接受未验证的外部文件，以防止恶意注入异常张量。现有 `load_weights` 仍通过 `torch.load` 的安全校验流程，未出现显式安全漏洞。  
  - 需要关注 LoRA 权重文件的来源与完整性，建议在生产环境加入 checksum / 签名校验。  

**⚠️ 潜在风险**  
1. **兼容性**：以前依赖 `Gemma4ForConditionalGeneration` 的代码若未实现 `SupportsLoRA`，在类型检查或运行时可能出现 `AttributeError`（如调用 `load_lora_weights`）。  
2. **权重冲突**：如果用户同时加载全模型权重与 LoRA 权重，可能出现同名参数被覆盖的情况，需要在文档或实现中明确加载顺序。  
3. **多模态路径**：`audio_tower` 为 `None` 时，仅保留 `embed_vision`；若后续代码仍硬编码 `embed_audio`，可能触发 `KeyError`。  
4. **测试覆盖不足**：本次提交仅修改了类定义与映射函数，缺少针对 LoRA 权重加载的单元测试，可能掩盖边缘 case（如空 LoRA、尺寸不匹配）。  

**💡 关注建议**
