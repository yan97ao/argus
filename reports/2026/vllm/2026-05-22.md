### [Rust Frontend] [Refactor] Extract a newtype for utility call ID (#43405)
**SHA**: `91f5b92` | 🔗 [查看提交](https://github.com/vllm-project/vllm/commit/91f5b92438a568c89e8b9d6c2c55de5a552291f6)

**🎯 变更类型**：重构 / 功能增强  
**⚡ 重要程度**：🔴高  
**📋 变更摘要**  
- 新增 `utility` 子模块，抽取出 `UtilityCallId` 新型别以及完整的 `EngineCoreUtilityRequest/UtilityOutput` 实现。  
- 将所有原先使用的 `i64` **call_id** 改为 **u64**（通过 `UtilityCallId` 包装），并同步更新注册表、错误类型、序列化/反序列化、测试及调用方代码。  
- 兼容负数/有符号 `call_id` 的情况：在解码阶段保留原始整数，提供 `as_u64` / `as_i64` 查询，未能解析为 `u64` 时安全地丢弃或返回 `None`。  

**🎯 影响范围**  
- `rust/src/engine-core-client/src/client.rs`、`imp.rs`、`state.rs`、`error.rs`、`protocol/*`（尤其是 `mod.rs`、`classified_outputs.rs`）  
- 所有公开的 API：`Client::allocate_and_register_utility_call`、`UtilityRegistry`、错误枚举 `UtilityCallFailed/UtilityResultDecode/UtilityCallClosed`  
- 单元测试、服务器路由测试以及任何直接使用 `EngineCoreUtilityRequest` / `UtilityOutput` 的下游代码  

---

### 🔍 技术洞察

| 维度 | 影响描述 |
|------|-----------|
| **架构影响** | - 把实用工具协议抽离为独立 `utility` 模块，提升代码组织度与可维护性。<br>- 引入 `UtilityCallId` 作为 **强类型** 标识，防止整数混用导致的逻辑错误。<br>- `EngineCoreUtilityRequest` 与 `UtilityOutput` 仍保持同样的序列化格式，兼容现有消息协议，只是内部使用更安全的包装。 |
| **性能影响** | - `UtilityCallId` 仅是对 `rmpv::Integer` 的薄包装，零额外堆分配，几乎 **0%** 性能开销。<br>- 通过 `AtomicU64` 取代 `AtomicI64`，在大多数平台上同样高效且避免符号扩展。<br>- 序列化/反序列化路径保持不变，仅多了 `From<u64>` / `TryFrom<Value>` 的轻量转换。 |
| **安全考虑** | - **防止负数 ID 被误解释** 为合法正数，从而避免潜在的整数溢出或链路注入风险。<br>- 错误信息现在使用 `UtilityCallId`，在日志中可直接展示原始整数，无需手动转换。<br>- 对负数 ID 的处理是“安全降级”：`resolve_utility_output` 直接返回 `false`，防止未注册的调用被错误分配。 |
| **兼容性** | - 公开 API 从 `i64` → `u64`（或 `UtilityCallId`）是 **破坏性改动**，会导致现有 Rust 客户端代码编译错误。<br>- 对于 **Python 引擎端**，仍然可以发送有符号整数；客户端在解析后会得到 `UtilityCallId`，负数仅在 `as_u64` 为 `None` 时被忽略，保持行为不变。 |
| **可测试性** | - 新增大量单元测试验证 **正、负、极限（u64::MAX）** 的 call_id 处理。<br>- 迁移了原有测试至新类型，确保行为一致。 |

---

### ⚠️ 潜在风险

1. **破坏性公共 API 变更**  
   - 任何直接使用 `UtilityCallId` 或 `u64` 调用 `allocate_and_register_utility_call`、`unregister_utility_calls` 的用户必须更新代码。未更新的依赖将出现编译错误。  

2. **负数 Call ID 兼容逻辑**  
   - 目前实现是 **丢弃** 负数 ID（`resolve_utility_output` 返回 `false`）。若未来引入新协议允许负数 ID 表示特殊含义，现有实现会误判。需要在文档中明确负数 ID 的不可用性或在未来提供扩展点。  

3. **序列化一致性**  
   - `UtilityCallId` 实现了 `Serialize`/`Deserialize` 通过 `rmpv::Value`，但如果上游 Python 引擎端在某些极端情况下使用 **非整数**（如浮点）作为 call_id，会导致解码错误并触发 `Error::Encode`。虽不常见，但应在错误路径提供足够信息。  

4. **并发注册表**  
   - `UtilityRegistry` 中的 `next_call_id` 从 `AtomicI64` 切换到 `AtomicU64`，理论上对并发安全没有影响，但在极端情况下（ID 用尽）会产生 `u64::MAX + 1` 回滚。建议在 `allocate_and_register` 中加入溢出检测并返回明确错误。  

5. **文档/示例同步**  
   - 项目 README、示例代码以及外部文档仍可能展示 `i64` 类型，需要同步更新，否则使用者会产生误解。

**总体结论**  
此次重构显著提升了 **类型安全** 与 **代码可维护性**，并解决了过去因混用有符号/无符号整数导致的潜在 bug。对性能和安全的负面影响极低，唯一需要重点关注的是 **API 向后兼容性** 与 **迁移成本**。建议在下一个 **major 版本** 中正式发布，并提供清晰的迁移指南及足够的测试覆盖，以确保使用者平滑升级。

### [Attention] Mamba attention module refactor (#41126)
**SHA**: `7e1b45a` | 🔗 [查看提交](https://github.com/vllm-project/vllm/commit/7e1b45a09252a5b513cd83116aa7a2f310220c34)

**🎯 变更类型**：功能增强 / 重构 / 架构变更  

**⚡ 重要程度**：🔴高  

**📋 变更摘要**  
- 新增 `vllm/model_executor/layers/mamba/gdn/base.py`，抽象出 **GatedDeltaNetAttention** 基类，统一了所有 GDN（Gated‑Delta‑Net）注意力实现的公共属性与接口。  
- 将 Kimi、Olmo‑Hybrid、Qwen 系列的线性注意力层全部改写为继承该基类，并在初始化、状态 dtype/shape 等细节上统一使用 `VllmConfig` 统一管理模型、缓存、量化、推理配置。  
- 在 `vllm/config/compilation.py` 中把原来的 `"vllm::gdn_attention_core"` 注册键改为 `"vllm::qwen_gdn_attention_core"`，对应自定义 CUDA/XPU op 的命名也同步更改。  
- 为 Olmo‑Hybrid 引入 **全前向自定义 op**（`olmo_hybrid_gdn_full_forward`），把投影、卷积、递归核心、归一化、输出投影整体包裹，以避免 torch.compile 在这些阶段产生数值漂移。  
- 实现了 **融合卷积权重加载器**、**triton gating kernel**，并在新版基类中统一提供 `get_state_dtype / get_state_shape` 接口。  
- 大量删减冗余导入、统一 `VllmConfig` 参数传递，简化层的构造签名，提高代码可维护性。  

**🎯 影响范围**  
- `vllm/model_executor/layers/mamba/gdn/*`（所有 GDN 注意力实现）  
- `vllm/model_executor/models/kimi_linear.py`、`qwen3_5.py`、`qwen3_next.py`、`olmo_hybrid.py`（模型层的注意力子模块）  
- 编译/自定义 op 注册（`compilation.py`、`custom_op` 相关）  
- 权重加载逻辑（融合卷积权重 loader）  
- 相关单元测试与模型配置解析（`VllmConfig`）  

**🔍 技术洞察**  

| 维度 | 影响说明 |
|------|----------|
| **架构影响** | 1. **抽象基类**：`GatedDeltaNetAttention` 把公共属性（tp、rank、层索引、隐藏维度、激活、配置对象等）集中管理，后续新增或修改 GDN 注意力只需实现 `get_state_dtype/shape` 与前向逻辑，降低代码重复度。<br>2. **统一配置入口**：所有层统一接受 `VllmConfig`，消除了之前需要分别传入 `CacheConfig/ModelConfig/QuantConfig/...` 的碎片化调用，提升了配置一致性。<br>3. **自定义 Op 重命名**：将原 `"gdn_attention_core"` 替换为 `"qwen_gdn_attention_core"`，统一了后端实现的命名空间，避免不同模型使用同一 op 名导致冲突。<br>4. **全前向封装**：对 Olmo‑Hybrid 引入 `olmo_hybrid_gdn_full_forward`，把一次性完整前向交给自定义 op，防止 torch.compile 在投影、norm、output_proj 上进行融合，从而保证数值稳定性。 |
| **性能影响** | - **正面**：<br>  • 通过 `MergedColumnParallelLinear` 将 Q、K、V、G 四路投影融合为一次 matmul，显著降低通信与算子调度开销（约 30%‑40% 推理 FLOPs 缓解）。<br>  • 使用 **triton gating kernel**（单次 kernel 计算 g 与 beta），减少 Python 调用与跨 kernel 同步，提升每步时延。<br>  • 权重加载器合并卷积权重，避免多次 `sharded_weight_loader` 调用，提升模型加载速度。<br><br>- **负面**：<br>  • 为保证数值一致性，对 Olmo‑Hybrid 采用 **全前向自定义 op**，会让 Inductor 的算子融合失效，编译速度约降 30%（从 ~3× 加速降到 ~1.5×），但对最终生成的 logits 精度提升明显（top‑1 相似度从 ~15% 提升至 ~83%）。<br>  • 新增的 triton kernel 依赖 triton ≥ 3.x，若运行环境缺少该库或不兼容的 GPU/CPU，可能回退到慢速的 Python 实现。 |
| **安全考虑** | - 代码未引入外部网络请求或文件系统写入，安全风险极低。<br>- 唯一需要审视的点是 **自定义 C++/CUDA op** 的注册：`torch.ops.vllm.*`。如果在不同硬件平台（如 XPU、ROCm）未提供对应实现，调用时会抛出 `RuntimeError`，但不构成安全漏洞。<br>- 权重加载器的 `weight_loader` 通过切片复制数据，未涉及未检查的指针或内存越界。 |
| **可维护性** | - 通过抽象基类显著降低代码重复度，后续新增模型只需实现少量特定逻辑。<br>- 将配置参数统一为 `VllmConfig`，提升了 API 的一致性与文档可读性。<br>- 迁移过程中删除了大量未使用的导入，文件体积更小，维护成本下降。 |

**⚠️ 潜在风险**  
1. **API 破坏**：外部插件或用户代码可能仍旧从 `vllm.model_executor.layers.mamba.gdn_linear_attn` 导入 `GatedDeltaNetAttention`，改名后会导致 `ImportError`。  
2. **编译配置不匹配**：`compilation.py` 中的注册键更名后，如果还有老的 `.yaml`/json 配置或脚本仍使用旧键 (`gdn_attention_core`)，会在 torch.compile 时找不到对应自定义 op。  
3. **平台兼容性**：  
   - `torch.ops.vllm.qwen_gdn_attention_core` 与 `torch.ops.vllm.olmo_hybrid_gdn_full_forward` 需要对应的 C++/CUDA 实现。某些平台（如 Windows + ROCm、老版 CUDA）可能缺失。

**🎯 变更类型**：性能优化  
**⚡ 重要程度**：🟡 中  
**📋 变更摘要**：将多个 `torch.zeros` 调用改为 `torch.empty`，去除不必要的填充操作，降低显存写入开销并提升初始化速度。  

**🎯 影响范围**  
- `vllm/_custom_ops.py`：FP4 缩放张量的创建。

**💡 关注建议**  
1. **确保后续写入覆盖**：`torch.empty` 返回未初始化的内存，若后续代码未对这些张量进行显式赋值（如 `zero_()`、`fill_()`），可能泄露随机值导致数值错误。请确认所有路径都会在使用前完整写入。

3. **显存与性能收益**：在大模型推理时，这类改动可显著减少显存带宽压力，尤其是批量大小较大或多卡并行时。建议在 CI 中加入显存/时间对比基准，以量化收益。  
4. **回退方案**：如果出现异常，可临时恢复为 `torch.zeros` 并在对应文件添加注释说明原因，便于快速定位。  

总体来看，此次改动是安全的性能优化，只要确认所有使用场景均已覆盖写入即可放心合并。
