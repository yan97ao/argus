### Add Granite 4.1 Vision as built-in multimodal model (#40282)
**SHA**: `d249a9e` | 🔗 [查看提交](https://github.com/vllm-project/vllm/commit/d249a9e90eda9c8588b4f093293be6c08f3bf9ec)

**🎯 变更类型**：功能增强  
**⚡ 重要程度**：🔴高  

**📋 变更摘要**  
本次 PR 为 vLLM 项目新增对 **Granite 4 Vision** 多模态模型的原生支持。核心工作包括：  
1. 在模型注册表、配置映射、处理器等框架层面加入 `granite4_vision` 类型的入口。  
2. 实现全新的模型类 `Granite4VisionForConditionalGeneration`（≈930 行），在语言模型（Granite）之上接入 SigLIP 视觉塔、Window‑QFormer Downsampler、深度堆叠（deep‑stack）特征注入以及可选空间采样。  
3. 为示例脚本、文档、CI 测试以及权重加载机制同步更新，确保用户可以直接通过 `ibm-granite/granite-vision-4.1-4b` 进行离线推理。  

**🎯 影响范围**  
- `vllm/model_executor/models/`（新增 `granite4_vision.py`）  
- `vllm/model_executor/models/registry.py`（模型注册）  
- `vllm/transformers_utils/config.py`、`configs/`、`processors/`（新增/扩展配置与处理器）  
- `examples/offline_inference/vision_language_multi_image.py`（示例代码）  
- `docs/models/supported_models.md`（文档）  
- `tests/models/multimodal/generation/`（新增针对 Granite 4 Vision 的单元测试）

### 🔍 技术洞察  

| 维度 | 影响分析 |
|------|----------|
| **架构影响** | - **模型层次**：在原有 Granite CausalLM 基础上，额外挂载 **SigLIP** 视觉编码器 + **WindowQFormerDownsampler**（两类投影器）形成 *tower‑model* 与 *connector*（layerwise‑projectors / spatial‑projectors）的多模态融合结构。<br>- **深度堆叠（deep‑stack）**：通过 `deepstack_layer_map` 将视觉层特征注入任意 LLM 层，实现跨模态信息流的细粒度交互。<br>- **管线并行（PP）**：模型实现遵循 vLLM 的 PP 约定，`Granite4VisionLLMModel` 在各 PP 阶段显式处理 `deepstack_input_embeds`，保持图捕获 (CUDA‑graph) 兼容。<br>- **LoRA**：在模型构造时继承 `SupportsLoRA`，支持两种 LoRA 使用方式（merge‑time HF overrides 与运行时 `--enable-lora --default-mm-loras`）。 |
| **性能影响** | - **计算开销**：新增视觉塔（SigLIP）+两套投影器（layerwise + spatial）比纯文本模型多约 2‑3 倍 FLOPs，尤其在高分辨率多图输入时更为明显。<br>- **内存占用**：为每个 deep‑stack 层预分配的 GPU 缓冲区 (`_ds_buffers`) 长度等于 `max_num_batched_tokens × hidden_dim`，在 1‑GPU 场景下额外消耗约 `num_levels × hidden_dim × 4B`（如 8 layers × 4096 dim ≈ 128 MiB）。<br>- **吞吐/延迟**：在预填（prefill）阶段，插入深度堆叠特征的拷贝操作在 CUDA‑graph 中已捕获，理论上对 latency 影响可忽略；但在推理阶段每一步都要执行投影与特征注入，可能导致每 token 的额外 0.5‑1 ms 延迟（实际取决于 GPU 型号）。<br>- **并行**：支持 PP 与 TP，且深度堆叠特征在非首 PP rank 通过 `IntermediateTensors` 传播，保持分布式训练/推理的可伸缩性。 |
| **安全考虑** | - 本次改动仅涉及模型结构与权重加载，不引入外部网络请求或系统调用，安全风险极低。<br>- 通过自定义 `Granite4VisionConfig` 避免 `trust_remote_code=True`，防止潜在的恶意代码执行。<br>- 仍然依赖用户自行提供模型权重；若权重来源不可信，仍可能带来模型层面的安全隐患（如隐藏后门），但这属于使用层面的常规风险。 |
| **可维护性** | - 代码量大（≈ 930 行），逻辑较为复杂（多模态输入解析、特征打包/拆分、deep‑stack 缓冲管理）。<br>- 与其他多模态实现（如 `llava_next`）保持高度复用，使用统一的 `BaseLlavaNextMultiModalProcessor`，有助于后期统一维护。<br>- 新增的配置、处理器文件使得模型在 **transformers** 低版本中仍能正常注册，降低依赖升级风险。<br>- 单元测试覆盖 `embed_multimodal`、`embed_input_ids`、`forward` 等关键路径，提升回归安全性。 |

### ⚠️ 潜在风险  

1. **CUDA Graph 捕获不完整**  
   - `embed_input_ids` 中的 `_ds_buffers` 在首次调用时会 `to(device)`，若在 CUDA‑graph 捕获前出现第一次调用导致图重新捕获，可能出现“graph already captured”错误。  
2. **层索引不一致**  
   - `deepstack_layer_map` 在配置文件与模型实现之间的负索引（如 `-19`）必须与实际视觉层数匹配；若 HF checkpoint 改动导致视觉层数变化，注入层会错位，导致特征维度不匹配或性能下降。  
3. **内存泄漏**  
   - `forward` 完成后仅在 `inputs_embeds` 非 `None` 时清零 buffers；若走 `intermediate_tensors` 路径（pipeline 并行的后续阶段）且未触发 `inputs_embeds`，残余数据可能残留在 GPU，导致显存漂移。  
4. **权重映射错误**  
   - `hf_to_vllm_mapper` 通过前缀替换实现映射，若未来 HF 官方模型结构微调（如新增子模块）而未同步更新映射表，加载将报错或产生未初始化参数。  
5. **图像 token ID 硬编码**  
   - 测试中硬编码 `mm_token_id = 100352`、`eos_token_id = 100257`，若模型的 tokenizer 采用不同的 special token ID，这段后处理会产生错误输出。  
6. **兼容性**  
   - `Granite4VisionConfig` 与未来 Transformers 版本的原生实现冲突（同名注册），可能导致配置重复或覆盖，需要在后续版本中移除自定义实现。

### [Frontend] Remove frontend pooling multi task support.  (#37861)
**SHA**: `d2e2e85` | 🔗 [查看提交](https://github.com/vllm-project/vllm/commit/d2e2e856ad4daf4b36bb37acaac06f3f61ac84ce)

**🎯 变更类型**：架构变更 / 功能削减  
**⚡ 重要程度**：🔴高  
**📋 变更摘要**：  
此次提交正式移除前端对 **Pooling 多任务**（`pooling_task` 参数）的支持，所有任务需通过 `PoolerConfig(task=…)` 明确定义。文档、示例、单元测试和核心入口逻辑相应改写，废弃的 `score` 任务已在 v0.21 前删除，相关警告改为直接抛异常。

**🎯 影响范围**：  
- `vllm/entrypoints/llm.py`、`vllm/entrypoints/pooling/*`（serving、factories、utils）  
- `vllm/tasks.py`（新增 `SCORE_TYPE_MAP`）  
- `vllm/pooling_params.py`（删除对 `score` 任务的兼容）  
- 示例脚本 `examples/pooling/*`  
- 文档 `docs/models/pooling_models/README.md`  
- 大量测试文件（offline/online、embed、classify、token_classify）  

**🔍 技术洞察**  

- **架构影响**  
  - 将任务决定权从运行时 API 参数 (`pooling_task`) 迁移到模型配置层 (`PoolerConfig.task`)。  
  - `init_pooling_io_processors`、`init_pooling_state`、`get_pooling_invocation_types` 等核心工厂函数在加载时只会实例化 **当前配置任务** 对应的 IOProcessor，简化了多任务分支和日志/错误处理路径。  
  - `SCORE_TYPE_MAP` 新增，用于在 `score` 接口内部统一映射 `pooling_task → score_type`，避免硬编码 `model_config.score_type`。  

- **性能影响**  
  - 移除对 `pooling_task` 参数的运行时检查与警告日志（`logger.warning_once` → `ValueError`），可略微降低请求路径的 CPU 开销。  
  - 由于不再需要在每次请求中重新解析并匹配多任务列表，内部调度表结构更紧凑，潜在提升高并发场景下的吞吐量（预计 < 2% 改进，实际取决于模型大小）。  

- **安全考虑**  
  - 变更不涉及新 I/O、网络或权限控制，安全面影响极低。  
  - 通过抛异常而非仅警告，防止用户在生产环境中误用已废弃任务导致不可预期行为，从而提升**使用安全性**。  

**⚠️ 潜在风险**  

1. **向后兼容性**  
   - 仍在代码中使用 `llm.encode(..., pooling_task="…")` 或调用 `/pooling` 接口时传递 `task` 参数的用户将遇到 `ValueError`，服务会返回 `BadRequestError`。  
   - 部分旧版脚本、CI/CD 流水线、第三方库（如 LangChain、OpenAI 兼容层）可能未更新，导致运行时错误。  

2. **文档/示例同步**  
   - 若外部文档或博客仍引用 `pooling_task` 参数，可能误导新手。  

3. **测试覆盖缺失**  
   - 迁移后未覆盖到的隐藏路径（如 `plugin` 任务与自定义 IOProcessor）可能出现未预料的 `KeyError`。  

**💡 关注建议**  

- **迁移指南**：在发行说明中提供明确步骤：  
  ```bash
  # 旧写法
  llm.encode(prompts, pooling_task="embed")
  # 新写法
  llm = LLM(..., pooler_config=PoolerConfig(task="embed"))
  llm.encode(prompts)
  ```  
  同时在 OpenAI 接口层示例中使用 `--pooler-config.task <task>`。  

- **版本号提升**：此为破坏性改动，建议在 `v0.21.0` 及以后版本中标记为 **Breaking Change**，并在 `CHANGELOG` 中列出。  

- **自动化检查**：在 CI 添加 **兼容性验证**（例如，使用旧参数的测试必须显式捕获 `ValueError`），防止回归。  

- **用户教育**：在文档首页与示例代码的头部添加 “此版本已不再支持 `pooling_task` 参数” 的显眼提醒。  

- **监控与回滚**：若生产环境出现大量 `BadRequestError`，可暂时通过环境变量或配置回退到旧实现（保留一段兼容层代码），待用户完成迁移后再彻底移除。  

- **审计**：确认所有 `Score` 相关路径均已使用 `SCORE_TYPE_MAP`，避免因 `model_config.score_type` 失效导致的隐藏错误。  

总体而言，此次改动通过统一配置入口提升了系统的可维护性与一致性，风险主要集中在向后兼容性和用户迁移成本，需要在文档、发布说明以及 CI 流程中给予充分支撑。

### feat(multimodal): support externally processed mm_kwargs with cache injection (#39502)

**⚡ 重要程度**：🔴高  

**📋 变更摘要**  
- 在 `vllm/v1/engine/input_processor.py` 中新增 `inject_into_mm_cache()` 接口，允许运行在前端的多模态处理器（如 HuggingFace Processor）预先生成的 `mm_kwargs` 直接写入后端的 MM 缓存，从而避免在后端再次重复处理。

### ⚠️ 潜在风险
1. **输入验证不足**  
   - 若前端注入的 `mm_kwargs` 包含不符合内部预期的张量（dtype、shape、设备），可能导致后续推理步骤报错或产生未定义行为。  
2. **缓存竞争**  
   - 多线程或多进程并发注入同一 `mm_hash` 时，`cache.get_and_update_item()` 的内部锁机制需要保证线程安全（现有实现已使用锁，但需在高并发场景下做压测）。  
3. **内存泄漏**  
   - 注入的对象若未被后续请求正确引用或清除，可能在 SHM（共享内存）缓存中残留，导致长期内存占用。  
4. **监控偏差**  
   - `update_mm_cache_stats()` 只在成功注入后调用，若注入因异常被跳过，统计可能出现轻微低估。  
5. **API 滥用**  
   - 开发者误将未处理的原始图片路径或 URL 直接注入，导致缓存中保存不可用对象，进而触发下游错误。

### 💡 关注建议
1. **加强输入校验**  
   - 在 `inject_into_mm_cache()` 中添加 **张量类型、设备、shape** 检查，或使用统一的 **schema**（如 `pydantic`）防止非法数据写入缓存。  
2. **文档与示例**  
   - 在官方文档中明确 “外部处理的 mm_kwargs 注入” 场景的前置条件（必须经过 HF processor、保持相同 `mm_hash` 生成算法），并提供示例代码。  
3. **并发压测**  
   - 在 CI 或专门的性能基准中加入 **并发注入** 场景（多线程/多进程），验证锁/共享内存实现的可伸缩性。  
4. **监控告警**  
   - 为 `vllm:mm_cache_injection_failures` 增加计数器，便于监控异常注入次数，及时发现异常前端行为。  
5. **回退机制**  
   - 若注入失败（例如缓存已满或异常），可以在后端 **fallback** 到标准的 `process_mm_kwargs` 流程，确保功能完整性。  
6. **安全审计**  
   - 在安全审计时关注注入路径，确保不易被利用进行 **DoS**（比如注入极大张量占满 SHM）或 **信息泄露**（注入含有敏感信息的张量）。  

--- 

**结论**：此次提交在保持现有架构不变的前提下，提供了一个高价值的功能点——**外部预处理的多模态数据直接注入缓存**。它能显著降低后端的重复计算开销，提升吞吐与响应速度，同时保持监控数据的准确性。风险主要集中在输入合法性和高并发下的缓存一致性，建议在后续迭代中加入更严密的校验与监控，以确保该功能在生产环境中的安全可靠。

### fix: clamp NaN/Inf in topk_softmax to prevent duplicate expert IDs (#39391)
**SHA**: `28c2221` | 🔗 [查看提交](https://github.com/vllm-project/vllm/commit/28c222157bbf54881385aae4a102fd04178312be)

**🎯 变更类型**：Bug 修复  

**⚡ 重要程度**：🔴 高  

**📋 变更摘要**：  
在 MoE（Mixture‑of‑Experts） gating 的 CUDA 实现 `topk_softmax_kernels.cu` 中，加入对 Softmax、Sigmoid 以及后续 argmax 计算产生的 NaN / Inf 值的强制 clamp（置零）逻辑，防止在异常隐藏状态（如 CUDA graph padding）导致的全 NaN/Inf 输出进而产生重复的 expert ID。对应新增单元测试验证 NaN/Inf 场景下 top‑k 结果的唯一性与数值有限性。

**🎯 影响范围**：  
- `csrc/moe/topk_softmax_kernels.cu`（CUDA 核心实现）  
- `tests/kernels/moe/test_fused_topk.py`（新增回归测试）  
- 依赖该 kernel 的上层 Python 入口 `vllm.model_executor.layers.moe`（以及所有使用 `fused_topk` 的模型）  

**🔍 技术洞察**  

- **架构影响**  
  - 该修改位于 **MoE 迁移路径的最底层**（CUDA kernel），对整个系统的专家路由逻辑产生直接影响。  
  - 通过在算子内部消除异常数值，避免了**下游 FlashInfer 三步 MoE 排序**（`three‑step MoE sort`）在遇到全 NaN/Inf 时崩溃的情况，提升了整体 **系统稳健性** 与 **容错能力**。  
  - 对上层业务逻辑（如 Transformer 解码）无侵入性改变，仅在极端异常数据时改变专家选择策略（从“全部选 0”变为基于索引的均匀分配），因此 **兼容性影响极小**。  

- **性能影响**  
  - 额外的 `isnan` / `isinf` 判定和一次写回（`softmax_val = 0.f`）在每个线程中执行一次，开销可忽略（单次浮点比较在 CUDA 中几乎为零时延）。  
  - 通过避免后续的异常异常路径（异常的专家 ID 会导致排序 kernel 重复计算并触发错误），**整体吞吐量在出现异常时会显著提升**。在正常数据流下性能基本持平。  

- **安全考虑**  
  - 该改动属于 **安全/鲁棒性修复**：防止因 NaN/Inf 传播导致的内存越界或未定义行为（如 argmax 循环在 NaN 环境下的无限循环或异常分支），降低了 **潜在的崩溃风险**。  
  - 并未引入外部输入校验或显式异常抛出，保持了现有的安全模型。  

**⚠️ 潜在风险**  

1. **语义改变**：在极端输入（本应全部为 NaN/Inf）的场景下，原实现会返回全 0 的 expert ID（全部指向 expert 0），而现在会返回 `[0,1,…,k‑1]`。虽然这并不会导致错误崩溃，但在极少数依赖“全 0”作为错误信号的上层逻辑里可能产生行为差异。  
2. **数值截断**：把 NaN/Inf 直接置零可能掩盖了上层应当捕获的异常（例如模型训练阶段的数值失稳）。如果用户希望监控此类异常，需要额外的日志或检测手段。  
3. **GPU 兼容性**：`isnan`、`isinf` 在不同 Compute Capability 下实现略有差异，虽然 NVCC 已提供统一实现，但在非常老的显卡（< SM 5.0）上可能出现未预期行为。  

**💡 关注建议**  

- **回归验证**：在 CI 中保留本次新增的 `test_fused_topk_nan_inf_clamp`，并在每次发布前运行全套 GPU 单元测试，以防止误改导致其他 kernel 行为异常。  
- **监控异常**：建议在上层 `fused_topk` 包装层添加可选的 **数值异常计数**（如 NaN/Inf 被 clamp 的次数），在训练/推理日志中给出警告，帮助用户定位根本原因。  
- **文档更新**：在 MoE gating 文档中注明：**“在出现 NaN/Inf gating logits 时将自动 clamp 为 0，保证 Top‑K 结果唯一且数值有限”。**  
- **针对老显卡做兼容性测试**（如 Tesla K40、GTX 1080），确认 `isnan/isinf` 行为一致。若发现问题，可在 kernel 中使用 `__isnanf` / `__isinff` 替代。  
- **安全审计**：虽然本次改动已解决崩溃风险，仍建议在后续的安全审计中检查所有使用 `argmax`、`topk` 的路径，确保类似的 NaN/Inf 传播不会导致未定义行为。  

---  

此 PR 通过在关键路径加入数值保护，有效提升了 vLLM 在 GPU 推理时的鲁棒性，风险可控，推荐合并至主干。

**🔍 技术洞察**  

| 维度 | 影响 |
|------|------|
| **架构影响** | • 将原有的**双向锁+标志位**同步方案简化为 **单向事件‑结果** 模式，降低状态管理复杂度。<br>• 引入 `CpuGpuEvent` 作为跨 CPU 线程 + GPU 流的同步基元，明确了 “生产者‑消费者” 角色（主线程记录、异步线程等待）。<br>• `EplbModelState` 从多个原子布尔/整数字段合并为 `pending_result`，使得状态在 GIL 下安全可见，降低并发竞争。<br>• `AsyncEplbLayerResult` 将所有层级转移的元数据聚合，一次性交付给主线程，提升代码可读性并为后续扩展（比如批量层转移）提供结构化入口。 |
| **性能影响** | • **减少锁竞争**：原实现在每层转移前需要 `buffer_lock.acquire(blocking=False)` 循环等待，可能导致自旋开销。新方案仅在 `pending_result` 产生时进行一次 CPU‑GPU 事件同步，显著降低自旋次数。<br>• **流水线化改进**：异步线程在 `transfer_layer` 完成后立刻 `wait()` `consumed_event`，而主线程在 `move_from_buffer` 完成后 `record()`，两者形成紧耦合的生产者‑消费者流水线，理论上可实现更高的重叠度（GPU 计算 vs. 数据搬迁）。<br>• **额外开销**：每层转移引入一次 `torch.cuda.Event`（在 `CpuGpuEvent` 内部）和一次 CPU `threading.Event`，但这些都是轻量级、一次性使用的同步原语，对整体吞吐影响极小。 |
| **安全考虑** | • 该改动不涉及网络通信或加密，仅影响内部线程同步，**安全风险几乎为零**。<br>• 需要确保 `CpuGpuEvent.record()` 与 `wait()` 的调用配对严格遵循生产者‑消费者模型，防止 **未记录即等待** 的情况导致死锁或错误的 GPU 同步。<br>• 由于 `CpuGpuEvent` 只支持 **两线程** 使用，若未来误将其用于多消费者场景会产生未定义行为，需在文档中明确限制。 |
| **可维护性** | • 代码结构更清晰：状态字段大幅削减，核心同步逻辑集中在 `CpuGpuEvent` 与 `AsyncEplbLayerResult`。<br>• 删除了大量注释/dead‑code（如 `window_ready_event`、`ep_buffer_ready`），降低了技术债务。<br>• 新增的单元测试专门验证 `CpuGpuEvent` 的阻塞行为，提升了回归安全性。 |

**⚠️ 潜在风险**  
1. **跨线程事件使用错误**：`CpuGpuEvent` 假设恰好两线程配对使用，若出现异常路径（异常抛出、线程提前退出）导致 `record()` 未被调用，异步线程将永远阻塞在 `wait()`。  
2. **GIL 同步假设**：`pending_result`、`rebalanced` 等布尔状态依赖 Python GIL 保证可见性。如果在极端的 C‑扩展或 `torch.jit` 环境下出现 GIL 释放，可能出现可见性竞态。  
3. **流（CUDA stream）管理错误**：`run_rebalance_experts` 现在接受 `cuda_stream` 参数并在 `with torch.cuda.stream(cuda_stream):` 中执行 `cpu()` 复制；若外部调用传入错误的流或在同一流上并发使用同一事件，可能导致 **流顺序错误**。  
4. **一次性事件重复使用**：`CpuGpuEvent.record()` 在内部会重新创建 `torch.cuda.Event`，但若用户在异常后再次调用 `record()` 而未先 `wait()`，会触发 `RuntimeError`（已有检测），但这可能导致整体 EPLB 失效。  
5. **向后兼容**：某些外部插件可能仍依赖已删除字段（如 `window_ready_event`），升级后可能出现 AttributeError。  

**💡 关注建议**  
- **回归测试**：在多机多卡（CPU + GPU 混合）环境下运行完整的 Distributed EPLB 测试套件，验证每层转移的正确性与整体吞吐。  
- **异常路径防护**：在 `async_worker.transfer_run_periodically` 的 `await state.rearrange_event.wait(stream=cuda_stream)` 前后加入超时/异常捕获，防止因主线程崩溃导致异步线程永久阻塞。  
- **文档更新**：明确 `CpuGpuEvent` 只能在 **单生产者‑单消费者** 场景使用；注明 `record()` 必须在主线程且在 `move_from_buffer` 完成后调用，否则会产生死锁。  
- **监控指标**：新增关键指标（例如 `async_worker_wait_time_ms`, `event_queue_depth`）以便在生产环境快速定位因事件未触发导致的卡顿。  
- **兼容层**：考虑在 `EplbModelState.__post_init__` 中为已删除字段提供 *兼容属性*（抛出友好提示），帮助第三方代码平滑迁移。  
- **性能基准**：对比重构前后的 **Lock‑contended time**、**GPU Utilization** 与 **overall inference latency**，确保改动带来的性能提升大于事件开销。  

通过上述检查与建议，团队可以在保持功能正确性的同时，充分利用新的事件‑结果模型提升 Async EPLB 的并发效率与代码可维护性。

### 🔍 技术洞察  

#### 架构影响  
1. **统一化 MoE 前向路径**  
   - 原先每个模型在调用 `FusedMoE` 后自行处理 shared‑expert 合并、`reduce_results`、FP16 overflow、路由尺度缩放等逻辑。现在这些职责统一交给 `MoERunnerBase`，模型只需提供 `hidden_states` 与 `router_logits`。

     - 对 fused 输出进行 optional all‑reduce / truncation

- **正向路径合并**：All‑reduce、截断、缩放等操作统一在 runner 中完成，避免在每个模型中重复调用 `tensor_model_parallel_all_reduce`，理论上可减少一次 Python‑C 边界切换和 kernel 调用的开销。
