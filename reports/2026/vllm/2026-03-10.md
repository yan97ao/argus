### feat(kv-offload): Strategy A — StoreReusedOffloadingManager gates CPU stores on reuse frequency (#35342)
**SHA**: `106ff69` | 🔗 [查看提交](https://github.com/vllm-project/vllm/commit/106ff69c4eb4921d33341a96b9c3d6db9d12ba76)

**🎯 变更类型**：功能增强  
**⚡ 重要程度**：🔴高  
**📋 变更摘要**：新增 `FilterReusedOffloadingManager`，在 CPU KV‑cache 离线存储前依据块的“重用频次”进行过滤；仅当同一块在 `lookup()` 中出现次数达到 `store_threshold` 才会被真正写入 CPU。实现通过装饰器模式包装原有的 `OffloadingManager`，并在 `vllm/v1/kv_offload/cpu.py` 中根据配置动态启用。对应单元测试亦同步补充，验证过滤逻辑、LRU 追踪器的容量限制以及 evict 行为。

**🎯 影响范围**：  
- `vllm/v1/kv_offload/cpu.py`（管理器构建入口）  
- 新增模块 `vllm/v1/kv_offload/reuse_manager.py`（核心过滤实现）  
- 测试 `tests/v1/kv_offload/test_cpu_manager.py`（覆盖新特性）  
- 与 KV‑cache 相关的离线存取路径（CPU 后端）  

**🔍 技术洞察**：  
- **架构影响**：  
  - 引入 **装饰器（Decorator）** 结构，`FilterReusedOffloadingManager` 在运行时包裹任意 `OffloadingManager`（LRU / ARC），保持原有接口不变，增强可插拔性。  
  - 新增 **配置入口** `store_threshold`、`max_tracker_size`，通过 `extra_config` 传递，使功能可在运行时开启/关闭，保持向后兼容。  
  - 通过 `OrderedDict` 实现 **内部 LRU 计数表**，在 `lookup` 时更新计数并在超出 `max_tracker_size` 时逐出最久未使用的条目。  

- **性能影响**：  
  - **正向**：降低 CPU offload 存储频率，尤其在大量一次性访问的块（如一次性推理）时，可显著减小 CPU memory 使用与 I/O 开销。  
  - **负向**：每次 `lookup` 都要在 `OrderedDict` 做一次 `move_to_end` / 插入操作，时间复杂度为 O(1)，但会产生额外的 Python 对象访问和内存占用（计数表大小上限 `max_tracker_size`），在极高并发或超大模型下可能成为微小的热点。  
  - **整体**：对 CPU offload 的 **吞吐量** 可能略有下降（因为过滤后仍会走完整的 `prepare_store` 流程），但对 **内存占用** 与 **磁盘/网络 I/O** 有明显收益。  

- **安全考虑**：  
  - 代码对输入 `store_threshold`、`max_tracker_size` 添加了显式校验（`>=2` 与 `>=1`），防止异常配置导致运行时异常或资源耗尽。  
  - 过滤逻辑本身不涉及跨进程/跨机器的数据泄露，未引入新的攻击面。  
  - 唯一需要关注的是 **异常路径**：`prepare_store` 可能返回 `None`（如底层 manager 不支持空列表），调用方需确保对该返回值进行容错处理，避免因 `None` 引发的后续 `AttributeError`。  

**⚠️ 潜在风险**：  
1. **配置误用**：若 `store_threshold` 设置过高，块可能永远不被 offload，导致 CPU cache 失效、频繁回滚到 GPU，反而降低性能。  
2. **计数器驱逐**：`max_tracker_size` 过小会导致频繁的 LRU 驱逐，使得同一块的计数被重置，可能出现 “本应已达阈值却被过滤” 的情况。  
3. **线程安全**：当前实现使用普通 `OrderedDict`，在多线程/多进程共享同一 `OffloadingManager` 实例时可能出现竞争条件（计数不一致、驱逐错乱）。  
4. **兼容性**：`prepare_store` 在过滤后可能得到空列表。如果底层 manager 在某些实现（如自定义插件）未正确处理空列表，可能抛异常。  
5. **测试覆盖不足**：现有单元测试仅覆盖 LRU 后端的基本路径，未覆盖 ARC、GPU‑CPU 混合、异常 `prepare_store` 返回 `None` 等场景。  

**💡 关注建议**：  
- **配置建议**：在生产环境首次启用时，建议 `store_threshold=2`、`max_tracker_size` 按 KV‑cache 大小的 5%~10% 进行调优，监控 `cpu_offload_store_events` 与 `cpu_offload_hit_rate` 指标，逐步提升阈值。  
- **监控与报警**：新增 Prometheus/metrics 采集点：  
  - `vllm_offload_filtered_store_total`（被过滤的块数）  
  - `vllm_offload_store_total`（实际写入的块数）  
  - `vllm_offload_tracker_size`（当前计数表大小）  
- **线程安全**：如果计划在多线程环境中共享同一 manager，考虑使用 `threading.Lock` 包装对 `self.counts` 的读写，或改用 `collections.defaultdict` + `collections.deque` 实现的无锁 LRU。  
- **异常容错**：在调用链的上层（如 `CpuGpuOffloadingHandlers`）对 `prepare_store` 的返回值做 `is None` 判空，确保在极端情况下仍能安全回退。  
- **扩展测试**：补充以下场景的单元测试：  
  - ARC 后端与过滤器的组合；  
  - `store_threshold` 为 0 / 1 的配置路径（确保不启用过滤器）；  
  - 并发 `lookup`/`prepare_store` 的 race 条件模拟；  
  - 极端 `max_tracker_size`（如 1）下的 evict 行为。  
- **文档更新**：在 `README` 或配置文档中加入 `store_threshold`、`max_tracker_size` 的说明、取值范围以及推荐调优指南，避免用户误配。  

通过上述措施，可最大化该功能的收益（降低 CPU offload 开销），并将潜在风险控制在可接受范围内。

### [Frontend][Core] Revert "Add shutdown timeout" (#34730 and #36270) (#36628)
**SHA**: `2348603` | 🔗 [查看提交](https://github.com/vllm-project/vllm/commit/234860399b9d390bf59bfe1f19c2e2304ac5c806)

**🎯 变更类型**：Bug修复 / 重构（撤销 “shutdown timeout” 功能）  
**⚡ 重要程度**：🔴高  
**📋 变更摘要**：本次提交回滚了此前在 vLLM 中引入的 **shutdown‑timeout** 配置及相关实现。所有涉及该超时的 CLI 参数、配置字段、信号处理回调和测试用例均被移除，改为在收到 SIGTERM/SIGINT 时直接抛 `SystemExit` 并立即关闭资源（`close()` 而非 `shutdown(timeout=…)`）。同时清理了 `SignalCallback`、`WAKEUP` 消息等辅助代码。

**🎯 影响范围**  
- **核心引擎**：`vllm/v1/engine/core.py`、`engine/protocol.py`、`engine/arg_utils.py`、`engine/utils.py`  
- **配置模型**：`vllm/config/vllm.py`（删除 `shutdown_timeout` 字段）  
- **入口脚本**：`vllm/entrypoints/cli/serve.py`、`vllm/entrypoints/launcher.py`（删掉超时逻辑）  
- **进程管理**：`vllm/v1/engine/core_client.py`、`vllm/v1/utils.py`、`tests/entrypoints/*`（改为 `close()`）  
- **多 API Server**：`vllm/entrypoints/api_server_process_manager.py`（改为 `close()`）  
- **测试**：大量原有关于 shutdown‑timeout 的集成测试被删除。

### 🔍 技术洞察

| 维度 | 影响描述 |
|------|----------|
| **架构影响** | - 移除了 `VllmConfig.shutdown_timeout`，简化了配置模型。<br>- `EngineCore` 现在仅依赖系统信号触发 `SystemExit`，不再需要额外的 “WAKEUP” 消息或 `SignalCallback` 线程。<br>- `APIServerProcessManager`、`EngineManager`、`Coordinator` 统一使用 `close()` 接口，删除了超时参数的传递链路，降低了跨模块耦合。 |
| **性能影响** | - 省去在 shutdown 时的 `time.sleep`/`await` 等等待逻辑，理论上 **关闭更快**，尤其在高并发或大量子进程的场景下可显著缩短退出时间。<br>- 失去“等待在处理中的请求完成” 的机会，可能导致请求被强制中止，进而触发上层重试，整体吞吐量在极端 shutdown 场景下会略有下降。 |
| **安全考虑** | - 无直接安全风险（仅是功能回滚）。<br>- 需要确保 **资源清理**（socket、文件句柄、GPU 内存）在 `close()` 中仍然可靠，否则可能导致泄露或残留进程被恶意利用。 |
| **可维护性** | - 删除了冗余的信号回调实现（`SignalCallback`、`WAKEUP`），代码体积降低 ~200 行，阅读和调试成本下降。<br>- `close()` 统一了资源释放入口，避免了不同模块对 `shutdown(timeout=…)` 的不一致实现。<br>- 仍保留了 `shutdown()` 方法的实现（但内部调用 `close()`），保持向后兼容性。 |

### ⚠️ 潜在风险

1. **业务兼容性**  
   - 使用 `--shutdown-timeout`（或在配置文件里设置 `shutdown_timeout`）的用户将收到 **未知参数错误** 或直接忽略该配置，可能导致异常退出或不符合预期的快速终止。  
   - 依赖 **平滑关闭**（等待在处理请求完成）的部署场景（如容器编排的滚动升级）会失去该保障，可能产生请求丢失或不完整的响应。

2. **进程残留风险**  
   - `close()` 直接调用 `finalizer()`，若子进程在接收到 `SIGTERM` 后未能及时自行退出，`shutdown` 逻辑（带超时等待）不再存在，可能导致 **僵尸进程** 短时间残留。  
   - 相关测试已被删除，缺少对 “子进程清理” 的回归覆盖，未来可能出现资源泄露而不易察觉。

3. **异常处置差异**  
   - 之前在超时期间会显式 **abort** 未完成的请求并发送 `FinishReason.ABORT`，现在改为直接 `SystemExit`，若上层客户端未捕获异常，可能出现 **未定义的错误码**（如 `-15`）或不友好的错误信息。

4. **文档/示例不一致**  
   - 项目文档、CI 示例或用户社区中仍可能提及 `--shutdown-timeout`；未同步更新会导致使用者困惑。

### [Perf] Compute maxsim in worker side, reducing redundant copies, 2.7% E2E throughput improvement (#36159)
**SHA**: `7279374` | 🔗 [查看提交](https://github.com/vllm-project/vllm/commit/7279374f9108652296a8f38b6f9c7f0585a0cda4)

**⚡ 重要程度**：🔴高  
**📋 变更摘要**：  
1. 将 MaxSim（Late‑Interaction）得分的计算搬到 worker 进程，减少在 engine ↔ worker 之间的张量拷贝，提升端到端吞吐约 2.7%。  
2. 新增 `LateInteractionParams`、`LateInteractionRunner`、`vllm/v1/pool/late_interaction.py` 等模块，实现查询向量缓存、文档向量得分以及请求的“粘性”路由（基于 query_key CRC32）

- **测试**：`tests/v1/engine/test_engine_core_client.py`、`tests/v1/worker/test_late_interaction_runner.py`  

**🔍 技术洞察**  

- **架构影响**：  
  - 引入 **Late‑Interaction 元数据** (`LateInteractionParams`) 到 `PoolingParams`，在请求层面携带查询/文档的关联信息。  
  - **粘性路由**：`get_late_interaction_engine_index` 使用 CRC32 将同一 query_key 的请求强制调度到同一 worker，避免跨进程缓存失效。  
  - **Worker‑Side 缓存**：`LateInteractionRunner` 在每个 worker 维护 query‑embedding 缓存 (`_query_cache`) 与使用计数 (`_query_uses`)，实现一次编码多次评分的 1‑N 场景。  
  - 现有的 `EngineCoreClient` 仍保持原有负载均衡路径，但在 `data_parallel_rank` 为 `None` 时会尝试通过 Late‑Interaction 判定固定引擎。  

- **性能影响**：  
  - **拷贝削减**：原本在 engine 侧收集所有 token‑embeddings 再统一计算 MaxSim，导致大量跨进程 GPU→CPU→GPU 拷贝；改为 worker 本地直接 `torch.matmul` 成本仅一次。  
  - **吞吐提升**：官方测得 **2.7%** E2E 吞吐提升，受益于减少同步阻塞和显存/带宽压力。  
  - **缓存命中**：对 1‑N 查询-文档对（如 RERANK）可显著提升延迟，尤其在 query 复用多次的场景。  
  - **额外开销**：每次查询请求会在 worker 内部复制一次 embedding（`output.clone()`），在极端高并发、查询向量巨大时可能占用额外显存。  

- **安全考虑**：  
  - 新增的 `query_key` 通过 CRC32 分配引擎，无加密或签名校验，理论上 **不涉及安全敏感信息**，但若业务使用可控的 query_key（如用户 ID）作为路由标识，可能导致 **请求聚合泄露**（同一 key 的请求被强制到同一进程）。  
  - `LateInteractionRunner` 对异常 `query_uses`、非法 `query_key` 均抛出 `ValueError`，防止缓存错乱。  
  - 代码路径未引入外部 I/O 或网络调用，攻击面保持不变。  

**⚠️ 潜在风险**  

1. **缓存泄漏 / OOM**：如果 `query_uses` 参数未正确设置（如远大于实际文档数），缓存不会及时释放，可能导致显存泄漏。  
2. **路由不均衡**：CRC32 哈希在少量 engine 时可能导致热点（多个不同 query_key 映射到同一 engine），进而出现负载不均。  
3. **兼容性**：删除 `--use-gpu-for-pooling-score` 参数后，旧版脚本仍可能引用导致启动失败；同时 `EngineCoreClient` 的 `data_parallel_rank` 为 `None` 时新旧路由逻辑共存，需确保 `late_interaction_params` 正确传递。  
4. **测试覆盖不足**：当前仅新增单元测试覆盖了粘性路由与 runner 基本路径，缺少多进程、显存回收、大规模 1‑N 场景的集成测试。  

**💡 关注建议**  

- **显存监控**：在生产环境开启显存监控报警，特别是对 `query_uses` 大于 1 的查询。建议在 `LateInteractionRunner` 中加入显存占用日志或 LRU 清理策略。  
- **路由均衡**：可考虑在 `get_late_interaction_engine_index` 上加入 **负载感知**（如基于当前 engine 的请求数）或使用一致性哈希来缓解热点。

- **回滚路径**：保留 `--use-gpu-for-pooling-score` 的兼容标志（可通过环境变量）以便在出现不可预期的回归时快速回滚。  
- **扩展测试**：加入多进程、跨 GPU、不同 `api_server_count` 场景的集成测试，验证粘性路由的正确性与显存回收。

**SHA**: `483463f` | 🔗 [查看提交](https://github.com/vllm-project/vllm/commit/483463f735c41c36a41431044fa537dc4c81fc3c)

**🎯 变更类型**：功能增强 / 重构 / 架构变更  
**⚡ 重要程度**：🔴高  

**📋 变更摘要**  
本次提交对 CUDA‑Graph（CG） 的调度与执行机制进行了全方位的重构：  
1. 引入 **`BatchExecutionDescriptor`**，统一描述一次批处理的 token 数、请求数、CG 模式以及是否为统一‑token 批。  
2. 将原有的 `CudaGraphManager` 拆分为 **`CudaGraphManager`（基类）** 与 **`ModelCudaGraphManager`（模型专用实现）**，实现了基于描述符的图捕获、调度与复用。  
3. 通过 **`sync_cudagraph_and_dp_padding`** 实现跨数据并行（DP）进程的图模式、token 数、请求数同步，确保所有 DP rank 在同一图上执行。  
4. 对 `block_table`、`slot_mappings` 的 Triton kernel 与 Python 调用加入 **padding‑aware** 参数，实现零化填充行的融合，避免额外的后处理。  
5. 统一了 `model_runner`、`spec_decode/eagle`、`input_batch` 等关键路径的调用方式，使其基于 `BatchExecutionDescriptor` 进行前置准备、图捕获与图 replay。  

预期效果：支持更灵活的 **PIECEWISE / FULL / FULL_DECODE_ONLY** 组合，提升多卡 DP 场景下的图捕获成功率与执行效率，降低因不同 rank token 数不一致导致的 fallback 到 eager 执行的频率。

- **核心模块**：`vllm/config/compilation.py`、`vllm/v1/worker/gpu/*`（block_table、cudagraph_utils、dp_utils、input_batch、model_runner、model_states、spec_decode/eagle）  
- **CUDA‑Graph 捕获/调度**：全局图池、图描述符、DP 同步逻辑  
- **Triton kernel**：`_gather_block_tables_kernel` 增加 padding 零化分支

### 🔍 技术洞察

| 维度 | 影响分析 |
|------|----------|
| **架构影响** | 1️⃣ 引入 `BatchExecutionDescriptor` 将 “token‑size ↔ CG‑mode ↔ request‑padding” 的映射抽象化，解耦了 capture 与 runtime 决策逻辑。<br>2️⃣ `CudaGraphManager` 现在只负责图的 **capture / replay**，而调度（如 DP 同步、模式选择）由 `ModelCudaGraphManager` 与外部 `sync_cudagraph_and_dp_padding` 完成，职责更清晰。<br>3️⃣ 通过 **全局图池**（`current_platform.get_global_graph_pool()`）统一管理不同模型/不同图的内存，避免了过去在每个 manager 内部自行创建 pool 而导致的资源碎片。 |

| **安全考虑** | - 代码仅在内部 tensor 维度、DP 同步上做了更严格的校验（`assert`），未引入外部输入的代码执行或内存泄漏风险。<br>- 新增的 `__bool__` 实现对 `CUDAGraphMode` 类的布尔化，若已有代码误用 `if cudagraph_mode:` 而原来期望 `Enum` 的 *truthiness*（始终 True），可能导致逻辑分支变化，需在全库搜索确认。<br>- 通过 `torch.inference_mode()` 保证 capture / replay 期间不产生梯度，安全性不受影响。 |

### ⚠️ 潜在风险
1. **描述符兼容性错误**：若 `dispatch` 选出的 descriptor 与实际运行的请求/token 数不匹配（例如 DP 同步后 `num_reqs` 仍不足），可能触发 **out‑of‑bounds** 或 **未填充的 padding 行**，导致模型产生错误输出。  
2. **旧代码兼容**：`CUDAGraphMode.__bool__` 改动后，所有原本依赖 `if cudagraph_mode:`（期望只检查 `!= NONE`）的地方行为保持，但若有代码显式比较 `bool(cudagraph_mode)` 以决定 “是否使用 CG”，旧行为仍相同；若有 `cudagraph_mode == True` 之类的比较，可能出现意料之外的结果。  
3. **DP 同步死锁**：`sync_cudagraph_and_dp_padding` 在跨进程 `all_reduce` 之后返回的 `BatchExecutionDescriptor` 被用于后续图 replay。如果某个 rank 在捕获阶段因为 `cudagraph_mode == NONE` 而没有创建对应图，但其他 rank 已经捕获并尝试 replay，可能导致 **图不存在** 的异常。当前实现已在 `dispatch` 中回退到 `NONE`，但仍需在全局启动阶段确保所有 rank 都已完成 capture。  
4. **图池冲突**：Eagle 采用 **独立图池**（在 `EagleCudaGraphManager` 中仍会创建 `torch.cuda.graph_pool_handle()`），若与主模型共享同一 `global_graph_pool`，可能出现 **内存竞争** 或 **图复用错误**。已在代码中注释说明使用独立池，但若上层仍误传 `None`，会产生冲突。  
5. **性能回归风险**：在极端 **小 batch（num_tokens < 256）** 场景下，捕获/回放的开销可能超过直接 eager 执行。虽然 `dispatch` 会回退到 `NONE`，但捕获阶段仍会执行一次 warm‑up，导致额外的 GPU 同步。  
6. **测试覆盖不足**：新路径包括 **FULL_DECODE_ONLY**、**PIECEWISE** 与 **uniform token count** 三种混合模式，需要在 DP=1

2. **性能验证**：空子图不再进入 splitting‑op 子图，避免不必要的跨子图同步，建议在大模型/多 GPU 场景跑一次基准，观察 `split_graph` 的子图数量与执行时间是否符合预期。

总体而言，此次修复提升了图划分的语义正确性，风险主要在签名变更和对 `split_op_graphs` 的构造正确性上，保持现有调用路径不变即可安全迁移。

2. **缓存配置扩展**：`KVCacheConfig` 新增 `has_mamba_layers` 与 `needs_kv_cache_zeroing` 两个属性，统一判断是否需要零化。  
3. **后端维度发现**：`AttentionBackend.get_kv_cache_block_dim` 用于自动定位 KV‑cache 中块索引所在的维度，以兼容不同后端布局。  
4. **单类型 KV‑Cache 管理器**：在 `SingleTypeKVCacheManager` 中记录每次分配的 block_id（仅 FullAttentionSpec），并提供 `take_new_block_ids` 接口供上层抽取。  
5. **实现零化 kernel**：在 `worker/utils.py` 中实现 Triton 零化 kernel `_zero_kv_blocks_kernel` 与封装类 `KVBlockZeroer`，在模型初始化时预计算段地址、页面大小等元数据，运行时只需拷贝 block‑id 列表并一次 launch 完成全部段的清零。  
6. **其他配套改动**：`math_utils.largest_power_of_2_divisor`、`KVCacheBackend.get_kv_cache_block_dim`、调度器、GPU runner、GPU worker 等位置加入相应调用。

### 影响范围  
- **核心缓存管理**：`vllm/v1/core/*`（`kv_cache_manager`, `single_type_kv_cache_manager`, `sched/*`）  
- **后端实现**：`vllm/v1/attention/backend.py`  
- **GPU 运行时**：`vllm/v1/worker/gpu_model_runner.py`, `gpu_worker.py`  
- **工具函数**：`vllm/v1/worker/utils.py`（Triton kernel）  
- **配置层**：`vllm/v1/kv_cache_interface.py`

### 关注建议  

| 关注点 | 建议 |
|--------|------|
| **新代码路径的正确性** | 单元测试需覆盖 `KVBlockZeroer.init_meta`、`zero_block_ids`，以及 `SchedulerOutput.new_block_ids_to_zero` 在没有 Mamba 层时是否保持 `None`，防止误零化。 |
| **性能** | 零化 kernel 会在每个调度步调用，建议在大模型、重复分配场景下对 `grid` 大小、`blk_size`（取 2^k ≤1024）进行基准，确保不成为显存/调度的瓶颈。 |
| **后端兼容性** | `get_kv_cache_block_dim` 采用占位值 `_S` 推断维度，需在所有已实现后端（FlashAttention、Torch‑scaled‑dot、Mamba‑SSM）上跑通；若新增后端，需要相应实现此方法。 |
| **内存泄漏/清理** | `KVBlockZeroer._ids_pinned` 与 `_ids_gpu` 会在零化时重新分配，确认在进程结束或模型卸载时能被回收。 |
| **异常路径** | `KVBlockZeroer.init_meta` 在未检测到任何段时返回 `None`，后续调用必须检查 `_meta`，当前实现已做 guard，保持不变。 |

总体而言，此次改动通过在块首次使用前显式清零解决了 **Mamba/SSM** 层在多请求并发时出现的 *NaN / 残留数据* 问题，改动范围局限在 KV‑cache 生命周期管理与 GPU 运行时，风险主要在新 Triton kernel 的正确性和性能上。建议在 CI 中加入针对含 Mamba 层模型的长序列、并发调度的回归测试。
