### [Kernel][Helion][1/N] Add Helion kernel for per_token_group_fp8_quant (#36902)
**SHA**: `2ec6594` | 🔗 [查看提交](https://github.com/vllm-project/vllm/commit/2ec6594db9c2397cc3c315ff3ce3b38e0d40e176)

**🎯 变更类型**：功能增强、轻量重构  
**⚡ 重要程度**：🔴高  
**📋 变更摘要**：  
- 为 vLLM 引入全新 **Helion** 计算内核 *per_token_group_fp8_quant*，实现基于 FP8 的 per‑token 量化，并加入自动调优配置选择逻辑。  
- 同步 CI/依赖：Helion 版本从 **1.0.0** 升至 **1.1.0**，并在 ``setup.py``、Buildkite 脚本中统一更新。  
- 新增大量配置 JSON（NVIDIA B200 / H100）、单元测试、工具函数，以及 ``register.py`` 中对 ``mutates_args`` 参数的显式支持。  

---

### 🎯 影响范围
- **核心计算层**：`vllm/kernels/helion/ops/per_token_group_fp8_quant.py`、`vllm/kernels/helion/register.py`  
- **配置管理**：新增 `vllm/kernels/helion/configs/per_token_group_fp8_quant/*.json`（B200、H100）  
- **构建与发布**：`.buildkite/*.yaml`、`setup.py`（Helion 依赖升级）  
- **测试套件**：`tests/kernels/helion/test_per_token_group_fp8_quant.py`、`tests/kernels/helion/utils.py`、`tests/kernels/helion/test_register.py`（对 ``mutates_args`` 的断言）  
- **用户可选依赖**：`vllm[helion]` 包装器  

---

### 🔍 技术洞察

| 维度 | 影响说明 |
|------|----------|
| **架构影响** | 1. **Helion 集成**：通过 ``register_kernel`` 将 Helion 生成的自定义操作包装为 vLLM 可直接调用的 Python 接口。<br>2. **配置选择器**：实现 ``pick_config``，在运行时根据 ``hidden_size``, ``group_size``, ``num_tokens`` 自动挑选最匹配的预调配置（JSON），并使用 ``_pick_cache`` 缓存提升选择速度。<br>3. **注册机制扩展**：``HelionKernelWrapper`` 新增 ``_mutates_args`` 字段，允许显式声明哪些输出参数会被原位修改，便于 IR/调度层做依赖分析。 |
| **性能影响** | 1. **FP8 量化**：FP8（8‑bit）相比 BF16/FP16 可显著降低显存带宽和存储需求，提升推理吞吐。<br>2. **自动调优**：Helion 的 autotune 基线函数 ``baseline`` 与选定配置比对后，可在首次运行时挑选最优的 block‑size/warp‑配置，避免手动调参。<br>3. **缓存**：``_pick_cache`` 将 ``(num_tokens, group_size, hidden_size)`` 三元组映射到 ``CaseKey``，在同一批次的多次调用中几乎零开销。 |
| **安全考虑** | - **依赖升级**：Helion 版本从 1.0.0 → 1.1.0，引入外部二进制库，潜在供应链风险。<br>- **输入校验**：新 kernel 对 ``input`` 维度、dtype、对齐方式做了断言，降低异常触发的概率。<br>- **异常处理**：在未安装 Helion 时会抛出明确的 ``ImportError``，并在测试层跳过。 |
| **可维护性** | - **配置分离**：所有硬件/模型规模相关的调优参数存放在独立 JSON，便于针对新 GPU（如 H100）添加或更新而不改动代码。<br>- **测试覆盖**：新增 **完整的正确性对比**（基准 vs Helion 实现 + 1 ULP 误差容忍）以及 **注册/伪实现** 测试，提升后续回归安全性。<br>- **代码分层**：生成输入、配置挑选、伪实现、真实实现各自独立，易于单元测试和后续扩展。 |

### ⚠️ 潜在风险

1. **Helion 兼容性**  
   - 不同平台（CUDA 11.x/12.x、ROCm）可能在底层实现上有差异，若用户未使用 NVIDIA B200/H100，可能找不到对应配置 → ``skip_if_platform_unsupported`` 会直接跳过测试，但在实际使用中会回退到基准实现（torch.ops._C），导致性能不升。  
   - Helion 1.1.0 与旧版的 API 兼容性仍需关注；若上游发生破坏性改动，vLLM 需要同步更新。

2. **配置覆盖不足**  
   - ``pick_config`` 只在 ``config_keys`` 中找最近的 ``hidden_size``、``group_size``、``num_tokens``。若新模型尺寸（如 6144）未在 JSON 中出现，可能选到次优配置或回退到最大值，导致性能不如预期。  
   - 目前的 fallback 策略是“取最大可用 num_tokens”，在极大 batch 场景可能导致显存超限。

3. **GPU 资源争用**  
   - 该 kernel 使用 ``hl.tile`` 循环，块大小依赖配置，如果 ``num_warps``、``num_stages`` 与实际 GPU SM 资源不匹配，可能出现 occupancy 低下或共享内存溢出。  
   - 自动调优基线函数未显式验证资源约束，需要在生产环境持续监控。

4. **测试/CI 依赖**  
   - 新增的 Buildkite 步骤仅在 CI 中安装 Helion 1.1.0，若本地开发者忘记 ``pip install vllm[helion]``，运行测试会被 skip，可能误以为功能正常。  
   - CI 仍保留对 ``tests/kernels/helion/test_register.py`` 的 mock‑注册路径，若该文件因重构而忘记更新 ``mutates_args``，会导致注册信息不完整。

5. **数值误差**  
   - 只允许 **1 ULP** 差异；在极端输入（极大/极小值）或不同 GPU 上的 FP8 rounding行为可能超过此阈值，导致测试失败或推理结果偏差。

### [Attention] add triton diff-kv backend for mimo (#41797)
**SHA**: `f81daf8` | 🔗 [查看提交](https://github.com/vllm-project/vllm/commit/f81daf8880632eea46590a8222c082a1e27fd11f)

**🎯 变更类型**：功能增强、架构变更  
**⚡ 重要程度**：🔴高  

**📋 变更摘要**  
1. 为 MiMo‑V2 系列模型在 KV 缓存的 **K 与 V 维度不一致（DiffKV）** 场景新增 Triton 实现，代码路径 `TRITON_ATTN_DIFFKV`。

4. 补齐文档、CI 配置和单元测试，验证 Triton‑DiffKV 与 FlashAttention‑DiffKV（FA3/FA4）输出在多种序列、头数、滑动窗口等配置下的数值误差在 2e‑2 范围内。

|------|----------|
| **架构影响** | - 在 `SpeechToTextParams` 中加入 `request_prompt`（即 `prompt`）字段，`Qwen3ASR.get_generation_prompt` 现在会在需要时生成系统（system）角色的 Turn，从而保持与官方 Qwen3‑ASR SDK 的对齐。<br>- 新增 `_sanitize_transcription_user_text`，在构造 ChatML Prompt 前统一清理用户文本，形成了一个 **安全边界层**，对所有使用该函数的模型统一生效。 |
| **性能影响** | - 正则 `re.compile(r"<\|[^|]+\|>")` 在模块加载时一次性编译，对后续调用开销极小。<br>- 清洗函数采用循环至不变点的 “fixpoint” 方案，最坏情况（极端嵌套攻击）仍在几次迭代后收敛，时间复杂度近似 O(n)（n 为字符串长度），对转写吞吐影响可以忽略。 |
| **安全考虑** | - 直接把用户 `prompt` 拼入 ChatML 模板会导致 **模型指令注入**（prompt injection）风险。清洗函数剥离 `<|...|>` 控制标记以及 `<asr_text>` 分隔符，且采用多轮替换保证嵌套构造攻击无效，提升了系统的 **隔离性** 与 **可信执行**。<br>- 新增测试覆盖了多种攻击样例，确保函数具备 **幂等性**（再次调用不再修改已清洗的字符串）。 |

1. **误删合法内容**：极端情况下用户的文本可能本身包含形似 `<|token|>` 的业务数据（例如 XML/HTML 片段），会被误删。

**💡 关注建议**
