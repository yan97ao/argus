### [Feature] Add support for JetBrains' Mellum v2 code generation model (#43992)
**SHA**: `023808c` | 🔗 [查看提交](https://github.com/vllm-project/vllm/commit/023808c23d234387298732ebc942fff5939dbd8b)

**🎯 变更类型**：功能增强（新增 Mellum 2 模型支持）  
**⚡ 重要程度**：🟡 中  

**📋 变更摘要**  
- 在 `vllm/model_executor/models` 新增 `mellum.py`，实现 `MellumForCausalLM` 以及对应的 `MellumModel/DecoderLayer/Attention`，复用 Qwen3‑MoE 的实现并加入对 per‑layer sliding‑window 与交错 SWA 的支持。  
- 在模型注册表、配置映射、文档、测试中分别加入 `MellumForCausalLM`，并新增 `MellumConfig`（继承自 `Qwen3MoeConfig`）。  

**🎯 影响范围**  
- **模型执行层**：`vllm/model_executor/models/*`（新增 `mellum.py`、注册表更新）  
- **配置体系**：`vllm/transformers_utils/config.py`、`configs/__init__.py`、`configs/mellum.py`  
- **文档与测试**：`docs/models/supported_models.md`、`tests/models/registry.py`  

**💡 关注建议**  

1. **兼容性检查**：`MellumConfig` 直接继承 `Qwen3MoeConfig`，若 JetBrains‑Mellum 在未来加入专有字段，需要同步更新此配置类，否则在加载时可能出现 `KeyError`。建议在 `MellumConfig` 中显式声明所有模型特有的属性（如 `dual_chunk_attention_config`、`sliding_window`）并提供默认值。  

2. **分布式与 TP**：`MellumAttention`、`MellumDecoderLayer` 均使用 `get_tensor_model_parallel_world_size()`，但未对 `per_layer_sliding_window` 在跨卡缓存上做额外校验。运行大规模 TP 时应在 CI 中加入随机层级的 sliding‑window 配置进行回归，防止缓存对齐错误。  

3. **测试覆盖**：目前仅在 `tests/models/registry.py` 添加了注册检查，建议补充以下测试：  
   - 端到端推理（tiny‑random checkpoint）验证输出形状与 logits 处理器兼容。  
   - `torch.compile` 下的兼容性（已装饰 `MellumModel`），可加一个简易 `torch.compile` 调用确保不抛 `NotImplementedError`。  
   - 多模态（SWA/滑窗）参数组合的性能回归，以免新特性影响已有模型的吞吐。  

4. **文档更新**：在模型列表中已经列出，但对 Mellum 2 的新特性（交错 SWA、层级滑窗）缺少说明。建议在模型文档页补充这些特性及对应的 `transformers` 参数含义，帮助用户在 HF Hub 上正确配置。  

5. **发布注意**：新增模型会在 `vllm` 包的 `__all__` 中暴露，需要在 `setup.cfg`/`pyproject.toml` 的 `package_data` 中确认 `vllm/transformers_utils/configs/mellum.py` 被打包，否则用户在 `from vllm import MellumConfig` 时会报 `ModuleNotFoundError`。  

**总结**：本次提交为 Mellum 2 提供了完整的执行路径，核心实现复用 Qwen3‑MoE，代码结构清晰。开发者在合并前重点检查配置字段完整性、分布式缓存对齐以及新增单元测试，以确保该模型在多卡、`torch.compile` 与常规推理场景下均能可靠运行。用户则可直接通过 `MellumForCausalLM` 加载 JetBrains 提供的 checkpoint，享受滑窗与交错 SWA 带来的更长上下文支持。

### [Perf] Optimize cutlass fp8 scaled mm bypassing padding, 20% kernel performance improvement (#43706)
**SHA**: `985c97a` | 🔗 [查看提交](https://github.com/vllm-project/vllm/commit/985c97a6a884f1a29e7584c05e11214f7fb96dbf)

**🎯 变更类型**：性能优化  
**⚡ 重要程度**：🟡 中  

**📋 变更摘要**  
为 Hopper（FP8）路径加入 `dynamic_padded_cutlass`，在编译期或运行时根据 batch 第一维是否能被 4 整除自动切换到有填充的 `padded_cutlass` 或直接调用 `cutlass_scaled_mm`，避免在已对齐的矩阵上额外的 padding，提升约 20% 的 kernel 吞吐。

**🎯 影响范围**  
- `vllm/model_executor/kernels/linear/scaled_mm/cutlass.py`（核心算子注册与调用）  
- 通过 `apply_block_scaled_mm` 间接影响所有使用 FP8 线性层的模型推理路径（尤其是 Hopper GPU）  

**💡 关注建议**  
- **兼容性**：新注册的 `dynamic_padded_cutlass` 与原 `padded_cutlass` 同共享假实现，保持 CI/torch‑script 的可运行性。若项目中已有自定义 `torch.compile` 脚本，请确认不受 `torch.cond` 控制流影响。  
- **回退**：若出现对齐判断错误导致性能回退，可通过显式设置 `torch.ops.vllm.padded_cutlass` 进行手动控制。  
- **测试**：建议在不同 batch 大小（4 的倍数与非倍数）上跑一次性能基准，确保条件分支在编译期正确触发。  
- **文档**：更新对应算子说明，提醒用户“在 Hopper 上 FP8 乘法会自动跳过不必要的 padding”。  

总体来看，此改动在保持原有功能的前提下，显著提升了 FP8 矩阵乘的效率，风险较低，建议尽快合入主分支。

### [Rust Frontend] Add InternLM2 tool parser (#43481)

**⚡ 重要程度**：🟡 中  

**📋 变更摘要**  
- 为 Rust 前端新增 **InternLM2** 工具调用解析器，支持 `<|action_start|><|plugin|>…<|action_end|>` 包裹的 JSON。  
- 在 `ToolParserFactory` 中注册对应 parser 并加入模型名称匹配模式（仅匹配 `internlm2` 子串），防止误匹配到其他 InternLM 系列模型。  
- 将 JSON 解析核心的 `arguments_key` 改为 **&[&str]**，以兼容 InternLM2 同时接受 `parameters` 与 `arguments`，并在所有现有 JSON‑based parser（Hermes、Llama、Mistral、Qwen）同步改为切片形式。  
- 新增大量单元测试，包括正负样例、流式解析、特殊 token 保留、错误路径等。  

**🎯 影响范围**  
- `rust/src/chat/src/parser/tool/*`（注册、模式匹配）  
- `rust/src/tool-parser/src/json/*`（核心配置、解析实现）  
- `rust/src/tool-parser/src/lib.rs`（对外导出）  
- 相关测试文件  

**💡 关注建议**  
1. **兼容性**：`arguments_key` 参数类型变更后，所有自定义 parser 必须更新为切片形式，防止编译错误。  
2. **模型匹配**：`internlm2` 匹配规则仅针对 `internlm2` 子串，若未来出现新模型命名（如 `internlm2x`）需相应扩展。  
3. **错误信息**：新增的错误路径（如未知参数键、截断）与 Python 实现不完全一致，文档和用户提示需同步说明。  
4. **性能**：多键匹配在 `json_arguments_key` 中使用 `candidates.contains`，开销极小，但仍建议在高并发场景下跑基准测试。  
5. **测试覆盖**：已加入丰富的单元测试，建议在 CI 中跑全部平台（Linux/macOS/Windows）以及 `--release` 模式，确保特化 token（`<|action_start|>` 等）在不同 tokenizer 实现下仍被保留。  
6. **文档**：更新 CLI `--tool-call-parser` 帮助信息，列出 `internlm`（映射到 InternLM2）以及新模型匹配规则。  

开发者在添加后续模型 parser 时，可复用 `json_arguments_key` 逻辑，只需提供 `arguments_key` 切片即可；用户在使用 `internlm` 参数时已无需改动，仅需确保模型名符合 `internlm2*` 形式。

### [Frontend] Resettle generative scoring entrypoint. (#44153)
**SHA**: `0910f7e` | 🔗 [查看提交](https://github.com/vllm-project/vllm/commit/0910f7e0e1f5ee9f2e5a6f76d0d09e68fddadc01)

**📝 变更概述**  
本次提交将 **generative‑scoring** 的实现从 `vllm.entrypoints.openai` 移动到 `vllm.entrypoints.generate`，并对相关入口、路由、状态初始化以及测试文件做了同步更新。核心类由 `OpenAIServingGenerativeScoring` 重命名为 `ServingGenerativeScoring`，所有 import、工厂函数以及 CI 配置也随之改写。

**🔍 主要影响模块**  
- `vllm/entrypoints/generate/*`（新代码所在）  
- `vllm/entrypoints/openai/*`（删除对应实现，修改 `api_server.py`）  
- `vllm/entrypoints/openai/generative_scoring/*`（已废弃）  
- CI 配置 `.buildkite/*.yaml`（加入 `tests/entrypoints/generate`，并在部分步骤中排除）  
- 单元/集成测试 `tests/entrypoints/generate/*`

**⚠️ 潜在风险**  
1. **向后兼容**：外部代码仍可能通过 `from vllm.entrypoints.openai.generative_scoring import OpenAIServingGenerativeScoring` 导入旧类，当前提交已移除该路径，导致 ImportError。  
2. **文档/示例**：官方文档、README 或示例脚本中若引用旧路径，需要同步更新，否则用户在升级后会遇到运行错误。  
3. **状态初始化**：`init_generate_state` 现在同时创建 `state.serving_generative_scoring`，但 `api_server.py` 中的旧 `init_generative_scoring_state` 已被删除，若有人在自定义入口中手动调用旧函数会报错。  
4. **路由注册**：`register_generate_api_routers` 现在额外调用 `register_generative_scoring_api_router`；若自定义 `FastAPI` 实例只引入 `generate` 的子路由而未执行 `register_generate_api_routers`，则 generative‑scoring 接口可能未暴露。

**✅ 建议措施**  
- **提供兼容别名**：在 `vllm/entrypoints/openai/generative_scoring/__init__.py` 中保留 `OpenAIServingGenerativeScoring = ServingGenerativeScoring`，并在迁移指南中标注即将废弃。  
- **更新文档**：搜索项目所有 `.md`、`.rst`、示例脚本，统一改为 `vllm.entrypoints.generate.generative_scoring`。  
- **CI 保障**：确保 `tests/entrypoints/generate` 在所有平台（AMD、ROCm、CPU）均执行；在 `entrypoints` 统一测试中加入 `--ignore=tests/entrypoints/generate` 的相反逻辑，防止误屏蔽。  
- **发布备注**：在 CHANGELOG 中注明 “generative‑scoring 移动至 generate 包，旧路径在 X.Y 版本后移除”，并给出迁移指引。  
- **回归测试**：运行完整的 `pytest -m "not gpu"`（所有平台）以及性能基准，确认新路由不影响已有 `openai` 接口的延迟与吞吐。  

总体来看，改动是一次结构性整理，提升了 `generate` 子包的完整性。只要妥善处理向后兼容和文档同步，风险可控。

### fix: glm5.1 pp model loading (#42944)
**SHA**: `1f6048a` | 🔗 [查看提交](https://github.com/vllm-project/vllm/commit/1f6048abe57511ed789deb8f9db4760546bcf4f5)

**🎯 变更类型**：Bug 修复  

**⚡ 重要程度**：🟡 中  

**📋 变更摘要**  
本次提交在 DeepSeek 系列模型（`deepseek_mtp.py`、`deepseek_v2.py`）的权重加载路径中加入了对 **pipeline‑parallel（PP）缺失层** 的处理。通过 `get_pp_missing_layer_names` 获取在 PP 场景下被省略的层名，并在 `_try_load_fp8_indexer_wk` 中提前返回，避免在缺失层上尝试加载 FP8‑WK 权重导致的 KeyError 或不匹配错误。

**🎯 影响范围**  
- `vllm/model_executor/models/deepseek_mtp.py`  
- `vllm/model_executor/models/deepseek_v2.py`  
- 相关工具函数 `vllm/model_executor/models/utils.py`（新增 `get_pp_missing_layer_names`）  

**💡 关注建议**  
1. **测试覆盖**：在多机多卡、不同 PP 配置下跑通 DeepSeek‑5.1、V2 的完整加载和推理，用 CI 增加对应的权重加载单元测试。  
2. **向后兼容**：`_try_load_fp8_indexer_wk` 参数签名变更，若有其他模型或插件直接调用该私有函数，请同步改动或保持兼容包装。  
3. **日志与警告**：当前对缺失层直接 `return True`，可能掩盖真实缺失的权重。建议在调试模式下打印层名，帮助使用者定位模型拆分策略。  
4. **文档更新**：在模型加载章节说明 PP 环境下的“missing layer”概念及如何检查 `get_pp_missing_layer_names` 的返回值。  
5. **性能检查**：新逻辑只增加一次集合遍历，对加载时间影响微乎其微，但仍建议在大模型（数百层）下跑一次基准，确认无额外开销。  

总体而言，此修复解决了 DeepSeek 系列在 PP 场景下因缺失层导致的加载异常，提升了 vLLM 在分布式推理中的鲁棒性。后续关注兼容性和监控信息即可安全上线。

**🎯 变更类型**：功能增强（为 RISC‑V RVV 添加缺失的 WNA16 辅助）

**🎯 影响范围**  
- `csrc/cpu/cpu_types_riscv_impl.hpp`：新增 `FP16Vec16(const c10::Half)`, `BF16Vec16(const c10::BFloat16)`，以及基于 lookup‑table 的 `FP32Vec16(int64_t, const FP32Vec16&)` 实现。

**💡 关注建议**

2. **对齐与类型安全**：`interleave_save_16b` 使用 `alignas(64)` 的局部数组，确认目标平台支持 64‑byte对齐，否则会产生未定义行为。  
3. **性能验证**：新增的 `FP32Vec16(int64_t, const FP32Vec16&)` 采用多次 RVV 指令实现查表，建议在实际模型跑通后对比原有实现（x86）性能，避免因指令组合导致流水线阻塞。  
4. **单元测试**：补齐的构造函数和 `interleave_save` 在现有 test suite 中可能缺失，建议新增针对 `c10::Half`、`c10::BFloat16` 的向量加载/保存以及交错写回的 RVV‑only 测试，确保数值精度与端序正确。  
5. **后向兼容**：在非 RVV 编译路径下仍需提供占位实现（如空函数或 static_assert），防止链接错误。  

整体来看，改动为 RISC‑V 上的 WNA16 支持奠定了基础，若按上述建议做好宏、对齐及测试，风险可控。
