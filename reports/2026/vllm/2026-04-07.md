### [KVConnector] Support 3FS KVConnector (#37636)
**SHA**: `96b5004` | 🔗 [查看提交](https://github.com/vllm-project/vllm/commit/96b5004b71105b7b191a55de710736eea315f871)

**🎯 变更类型**：功能增强（新增 HF3FS KV‑Connector）  
**⚡ 重要程度**：🔴 高  

**📋 变更摘要**  
- 在 `vllm.distributed.kv_transfer.kv_connector` 中新增 **HF3FSKVConnector**，实现基于 3FS（hf3fs）文件系统的 KV‐cache 持久化与共享。  
- 引入 **AsyncOperationManager**、**Metadata Server**、**Mock Client**、以及若干底层工具（triton kernels、共享内存拷贝、缓冲区池）。  
- 在工厂类中注册新 connector 并在 `setup.py` 中把相关 C++ 源文件加入打包。  
- 新增完整的单元测试覆盖资源管理、客户端行为、metadata 逻辑以及统计/指标收集。  

**🎯 影响范围**  
- `vllm.distributed.kv_transfer.kv_connector`（所有 v1 connector 实现）  
- Scheduler 与 Worker 之间的 KV‑cache 调度路径  
- `vllm.config.VllmConfig.kv_transfer_config`（新增 hf3fs 配置项）  
- 依赖 `hf3fs_fuse`、CUDA、triton、torch C++ 扩展的运行时环境  
- CI / packaging（`setup.py`、文档）

## 🔍 技术洞察

### 1. 架构影响
| 维度 | 影响描述 |
|------|----------|
| **模块划分** | 新增 `hf3fs` 包含四个子模块：<br>• `hf3fs_client`（真实或 mock 实现）<br>• `hf3fs_connector`（业务层）<br>• `hf3fs_metadata_server`（全局元数据管理）<br>• `utils`（并发计数器、缓冲区池、triton gather/scatter）<br>保持原有 **Base**、**Factory**、**Metrics** 接口不变，且通过 `KVConnectorFactory.register_connector` 完全解耦。 |
| **同步/异步** | `AsyncOperationManager` 采用两条后台线程 + `ThreadPoolExecutor` 完成并发 I/O，使用轮转 `AtomicCounter` 均衡 client 实例。<br>所有写入在 **CUDA stream → event → async write** 之间同步，读写完成后通过 `Future` 通知 Scheduler。 |
| **调度兼容** | 在 Scheduler 侧，新增 `HF3FSRequestMetadata`、`RequestSchedulingState`，并在 `build_connector_meta`、`get_num_new_matched_tokens`、`update_state_after_alloc` 中完成 cache‑hit、写入、加载的统一调度。<br>对已有 `FlexKVConnectorV1`、`SimpleCPUOffloadConnector` 不产生破坏性依赖。 |
| **元数据服务** | `GlobalMetadataState`（内存版） + `Hf3fsMetadataServer`（FastAPI）提供 **页分配 / 确认 / 查询** API，支持多 Rank、TP‑world 完整性检查。<br>客户端通过 `Hf3fsGlobalMetadataClient` 实现 **重试**、**JSON/ORJSON** 编解码兼容。 |
| **可插拔性** | `HF3FS_AVAILABLE` 标记与 `try/except` 导入，若机器上没有 `hf3fs_fuse` 会自动回退到 **Mock Client**，保证库在 CI 环境仍可运行。 |

### 2. 性能影响
| 维度 | 正向增益 | 可能的瓶颈 |
|------|----------|------------|
| **I/O 并发** | 通过 `entries=DEFAULT_MAX_IO_ENTRIES`（8）+ `numjobs`（默认 16）实现多路 I/O，理想情况下可将单文件的磁盘吞吐提升到 10‑20 GB/s（取决于底层 3FS）。 |
| **CPU‑GPU 协同** | 使用 **共享内存 + pin‑memory** + **CUDA stream/event** 实现零拷贝路径（GPU → pinned → shm → FD → 3FS）。在高并发写入时可显著降低迁移开销。 |
| **triton gather/scatter** | 自研 `kv_cache_gather_kernel` / `kv_cache_scatter_kernel` 把 KV‑cache 按 token‑index 直接搬运到/从 3FS 缓冲区，避免 Python 循环，带来数倍的 CPU → GPU 拷贝提升。 |
| **线程 & Future 开销** | 每个请求产生两组 `Future`（save / load），在极高请求数（>10k）时可能产生大量 `Future` 对象与锁竞争。 |
| **网络 RTT** | 元数据查询（`batch_key_exists`、`get_key_locations`）为同步 HTTP 调用，单次往返约 1‑2 ms（本地），在大批量请求时累计可达数百毫秒，需要 batch‑size 调整或异步 HTTP 客户端。 |
| **内存占用** | 每个 client 持有 `entries * bytes_per_page * 2` 的 pinned buffer + shm + torch tensors，默认 8 entries × 1 MiB ≈ 16 MiB / client；`numjobs=16` → ~256 MiB，仍在可接受范围，但在多 GPU/节点下需关注总体占用。 |
| **CPU 端资源** | `AtomicCounter`、线程锁、`ThreadPoolExecutor` 产生轻度 CPU 争用，建议在 CPU 负载高的机器上监控。 |

### 3. 安全考虑
| 项目 | 说明 |
|------|------|
| **外部依赖** | `hf3fs_fuse.io` 是 C++‑extension，未签名的二进制可能带来 **供应链风险**。建议在 `setup.py` 中声明 `hf3fs_fuse` 为 **optional**，并在发布说明中提供安全哈希校验。 |
| **文件路径** | `hf3fs_storage_path` 直接拼接到磁盘路径，若配置错误可能写入非受信目录；应在初始化时校验路径是否在安全的挂载点（如 `/mnt/hf3fs`），并限制用户只能写入该目录。 |
| **网络交互** | Metadata Server 暴露 CRUD API，默认未做身份验证；在多租户环境下需要 **认证/ACL** 或通过内部网络隔离。 |
| **并发资源释放** | `Hf3fsClient._release_resources` 使用 `del` 与 `close`，但在异常路径若 `register_fd` 成功而 `make_ioring` 失败，仍会调用 `deregister_fd`，已做容错；仍建议在 `close` 中使用 `finally` 包裹全部释放，以防止资源泄漏。 |
| **异常传播** | 大量 `raise RuntimeError` 在后台线程中会导致 **线程默默终止**，但 `Future` 仍返回 `None`。当前实现把错误记录日志并返回 `False`，符合容错需求。 |
| **数据完整性** | 写入成功后才执行 `confirm_write_for_keys`；若写入成功却未能成功提交元数据（网络错误），页面会被 **release** 回池，导致后续读取缺失。建议在写成功后使用 **两阶段提交**（先写入 → ack → confirm），或在确认失败时进行 **重试**。 |

**📋 变更摘要**：  
本次提交将原本 2500 行的单体文件 `compressed_tensors_moe.py` 拆分为一个包 (`compressed_tensors_moe/`) 并在其中实现多个独立的 MoE 量化方法文件（如 `compressed_tensors_moe_w4a4_mxfp4.py`、`compressed_tensors_moe_w4a4_nvfp4.py` 等），同时新增 `__init__.py` 只暴露 `CompressedTensorsMoEMethod` 基类。文档中的类路径也同步更新。此举提升了代码的模块化、可读性与后期扩展性。

**🎯 影响范围**：  
- `vllm/model_executor/layers/quantization/compressed_tensors/compressed_tensors_moe/` 目录下全部文件  
- 依赖 `CompressedTensorsMoEMethod` 的层级（如 `FusedMoE`）的加载与权重处理逻辑  
- 文档 `docs/design/moe_kernel_features.md` 中的类引用  
- 任何外部项目直接 `import vllm.model_executor.layers.quantization.compressed_tensors.compressed_tensors_moe` 的旧路径（已被删除）

**🔍 技术洞察**  

| 维度 | 分析 |
|------|------|
| **架构影响** | - **模块化**：每种量化方案独立文件，职责单一，后续新增/维护新方案时不必在同一巨文件中滚动修改。<br>- **层级关系**：`compressed_tensors_moe/__init__.py` 只导出 `CompressedTensorsMoEMethod`，内部各子文件通过 `from .compressed_tensors_moe_w4a4_mxfp4 import …` 动态引入，实现了“基类‑实现分离”。<br>- **循环依赖**：基类文件 `compressed_tensors_moe.py` 不再引用具体实现，避免了原来文件内部的相互导入风险。 |
| **性能影响** | - **运行时**：核心计算路径（权重创建、后处理、kernel 调用）保持不变，拆分文件本身对推理性能几乎无影响。<br>- **导入开销**：启动时需要逐个加载更多 Python 模块，理论上会略增启动时间（毫秒级），但对长期运行的服务器影响可忽略。<br>- **缓存友好**：文件更小，Python 编译缓存（`.pyc`）更易命中，潜在的微小加速。 |
| **安全考虑** | - **兼容性**：删除旧的单体文件会导致外部直接 import 失效；若未更新依赖会抛 `ModuleNotFoundError`，属于功能回退风险。<br>- **代码审计面**：拆分后每个文件的代码行数显著降低，审计、静态检查（如 Bandit、pyright）更易覆盖。<br>- **状态字典（state_dict）**：权重命名仍保持旧有 `w13_weight_packed → w13_weight`、`w2_weight_packed → w2_weight` 的转换逻辑，未改变外部可序列化的键名，兼容性基本保持。 |
| **可维护性** | - **可读性**：单个文件平均 < 200 行，函数/类注释完整，便于新人快速定位实现细节。<br>- **扩展性**：新增量化方案仅需在 `compressed_tensors_moe/` 新增文件并在基类 `get_moe_method` 中加入对应分支，无需修改已有实现。<br>- **测试粒度**：可以针对每种实现编写独立单元测试，提升覆盖率。 |

**⚠️ 潜在风险**  

1. **向后兼容破坏**  
   - 外部用户或历史脚本仍可能使用 `vllm.model_executor.layers.quantization.compressed_tensors.compressed_tensors_moe`（旧路径）进行导入或序列化检查，现已被删除。  
2. **导入顺序错误**  
   - 子模块在导入基类前使用相对导入（`from .compressed_tensors_moe import …`），若包结构在某些运行环境下被错误地加入 `PYTHONPATH`，可能出现循环导入或模块解析失败。  
3. **权重加载兼容性细节**

4. **文档/示例同步遗漏**  
   - 除了 `moe_kernel_features.md`，仓库其他使用旧路径的示例或 README 可能仍指向旧文件，导致用户困惑。

- **持续集成**：在 CI 中加入针对所有量化配置的端到端加载和推理测试，确保每个新子模块在模型权重加载后能够成功初始化并完成一次前向。

### [Quantization] Add FlashInfer CuteDSL batched experts backend for NVFP4 MoE (#38251)
**SHA**: `e8ebbdd` | 🔗 [查看提交](https://github.com/vllm-project/vllm/commit/e8ebbdde8304a8cf89bbd4e101ebdfc25118b125)

**🎯 变更类型**：功能增强 / 架构变更 / 性能优化  
**⚡ 重要程度**：🔴 高  

**📋 变更摘要**  
1. 新增 **FlashInfer CuteDSL Batched Experts** 实现 (`flashinfer_cutedsl_batched_moe.py`)，在 NVFP4 量化的 MoE 场景下使用 FlashInfer 的 `cute_dsl_fused_moe_nvfp4` 一体化 kernel，采用 **BatchedExperts** 激活格式（即一次性对所有专家的输出做激活与归约），大幅简化原有两阶段 GEMM‑Activation‑GEMM 流程。  
2. 对原有 `flashinfer_cutedsl_moe.py` 进行裁剪，改为 **Standard** 激活格式，直接调用 FlashInfer 的函数式 API，去除自行实现的 `flashinfer_cutedsl_moe_masked`。  
3. 扩展 MoE 后端枚举、选择逻辑以及量化准备函数，以支持 **Batched** 变体（`FLASHINFER_CUTEDSL_BATCHED`）。  
4. 新增权重准备工具 `prepare_nvfp4_moe_layer_for_flashinfer_cutedsl`，实现 **gate/linear 行交叉**、**scale swizzle → MMA layout** 转换。  
5. 在 `utils/flashinfer.py` 中懒加载新 kernel 与辅助函数，并提供检测 `has_flashinfer_cutedsl_moe_nvfp4`。

总体来看，此次改动为 Mamba‑HMA 在异构 TP 环境下提供了完整的 descriptor 规划和过滤逻辑，代码结构清晰、抽象到位。只要在非‑Mamba 场景保持向后兼容、补齐性能基准与异常检测，即可安全合入主线。
